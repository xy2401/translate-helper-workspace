Chapter 9. 已知的问题

本部分介绍了Red Hat Enterprise Linux 8中的已知问题。

Web控制台

无法使用session_recording shell登录RHEL Web控制台

目前，对于启用tlog记录的用户，RHEL Web控制台登录将失败。RHEL Web控制台要求用户的shell存在于/etc/shells目录中以允许成功登录。但是，如果将tlog-rec-session添加到/etc/shells ，则记录的用户可以通过使用“chsh”实用程序将shell从tlog-rec-session更改为/etc/shells另一个shell来禁用记录。出于这个原因，Red Hat不建议将tlog-rec-session添加到/etc/shells 。

（BZ＃1631905）

安装程序和映像创建

auth和authconfig Kickstart命令需要AppStream存储库

安装期间auth和authconfig Kickstart命令需要authselect-compat包。如果没有此软件包，则在使用auth或authconfig安装将失败。但是，根据设计， authselect-compat包仅在AppStream存储库中可用。

要解决此问题，请验证安装程序是否可以使用BaseOS和AppStream存储库，或者在安装期间使用authselect Kickstart命令。

（BZ＃1640697）

默认情况下不安装xorg-x11-drv-fbdev ， xorg-x11-drv-vesa和xorg-x11-drv-vmware视频驱动程序

在安装RHEL 8.0 Server后，具有特定型号的NVIDIA显卡和具有特定AMD加速处理单元的工作站的工作站将不会显示图形登录窗口。

要解决此问题，请在工作站计算机上执行RHEL 8.0 Workstation安装。如果工作站上需要安装RHEL 8.0 Server ，请在安装后通过运行yum -y groupinstall base-x命令手动安装base-x软件包组。

此外，依赖EFI进行图形支持的虚拟机（如Hyper-V）也会受到影响。如果在Hyper-V上选择了Server with GUI基本环境的Server with GUI由于重新启动时显示黑屏，您可能无法登录。要在Hyper-v上解决此问题，请使用以下步骤启用多用户或单用户模式：

  1. 重新启动虚拟机。
  2. 在引导过程中，使用键盘上的向上和向下箭头键选择所需的内核。
  3. 按键盘上的e键编辑内核命令行。
  4. 将systemd.unit=multi-user.target添加到GRUB中的内核命令行。
  5. 按Ctrl-X启动虚拟机。
  6. 登录后，运行yum -y groupinstall base-x命令。
  7. 重新引导虚拟机以访问图形模式。

（BZ＃1687489）

使用reboot --kexec命令时安装失败

使用包含reboot --kexec命令的Kickstart文件时，RHEL 8安装失败。要避免此问题， reboot --kexec在Kickstart文件中使用reboot命令而不是reboot --kexec 。

Anaconda安装包括最低限度的资源设置要求

Anaconda在系统上启动安装，只需要最少的资源设置，并且不提供有关成功执行安装所需资源的先前消息警告。因此，安装可能会失败，并且输出错误不会为可能的调试和恢复提供明确的消息。若要解决此问题，请确保系统具有安装所需的最小资源设置：PPC64（LE）上的2GB内存和x86_64上的1GB内存。因此，应该可以成功安装。

（BZ＃1696609）

reboot --kexec和inst.kexec命令不提供可预测的系统状态

请注意，不推荐使用kexec功能，将在未来的Red Hat Enterprise Linux版本中删除。

核心

i40iw模块在启动时不会自动加载

由于许多i40e网卡不支持iWarp和i40iw模块不完全支持挂起/恢复，默认情况下不会自动加载此模块以确保挂起/恢复正常工作。要解决此问题，请手动编辑/lib/udev/rules.d/90-rdma-hw-modules.rules文件以启用i40iw的自动加载。

另请注意，如果在同一台计算机上安装了另一台带有i40e设备的RDMA设备，则非i40e RDMA设备会触发rdma服务，该服务会加载所有已启用的RDMA堆栈模块，包括i40iw模块。

（BZ＃1623712）

连接许多设备时，系统有时会无响应

当Red Hat Enterprise Linux 8配置大量设备时，系统控制台上会出现大量控制台消息。例如，当存在大量逻辑单元号（LUN）时，会发生这种情况，每个LUN都有多条路径。除了内核正在进行的其他工作之外，大量的控制台消息可能会导致内核监视程序强制内核出现紧急情况，因为内核似乎已挂起。

由于扫描在引导周期的早期发生，因此在连接许多设备时系统无响应。这通常发生在启动时。

如果在引导后的设备扫描事件期间在您的计算机上启用了kdump ，则硬锁定将导致捕获vmcore映像。

若要解决此问题，请增加看门狗锁定计时器。为此，请将watchdog_thresh= N选项添加到内核命令行。将N替换为秒数：

  • 如果您的设备少于一千台，请使用30 。
  • 如果您有超过一千台设备，请使用60 。

对于存储，设备数是指所有LUN的路径数：通常是/dev/sd*设备的数量。

应用解决方法后，系统在配置大量设备时不再无响应。

（BZ＃1598448）

KSM有时会忽略NUMA内存策略

当使用merge_across_nodes=1参数启用内核共享内存（KSM）功能时，KSM忽略由mbind（）函数设置的内存策略，并且可能将页面从某些内存区域合并到非统一内存访问（NUMA）节点不符合政策。

要解决此问题，请禁用KSM或将merge_across_nodes参数设置为0如果使用与QEMU绑定的NUMA内存）。因此，为KVM VM配置的NUMA内存策略将按预期工作。

（BZ＃1153521）

qede驱动程序挂起NIC并使其无法使用

由于存在错误，41000和45000 QLogic系列NIC的qede驱动程序可能导致固件升级和调试数据收集操作失败并使NIC无法使用或处于挂起状态，直到主机重新启动（PCI重置）使NIC再次运行。

在以下所有方案中都检测到此问题：

  • 使用收件箱驱动程序升级NIC的固件时
  • 收集运行ethtool -d ethx命令的调试数据时
  • 运行sosreport命令，因为它包含ethtool -d ethx.
  • 当收件箱驱动程序启动自动调试数据收集时，例如IO超时，邮箱命令超时和硬件注意。

Red Hat的未来错误将通过Red Hat Bug Advisory（RHBA）发布以解决此问题。要解决此问题，请在https://access.redhat.com/support中创建一个案例，以便在发布RHBA之前请求针对该问题的受支持修复程序。

（BZ＃1697310）

在Red Hat Enterprise Linux 8中，检查点和用户间还原（CRIU）包的版本已过时。因此，CRIU不支持容器检查点和还原功能，并且podman实用程序无法检查点容器。运行podman container checkpoint命令时，将显示以下错误消息：'checkpointing a container至少需要CRIU 31100'

（BZ＃1689746）

early-kdump和标准kdump如果失败add_dracutmodules+=earlykdump选项中使用dracut.conf

目前，为early-kdump安装的内核版本与为其生成的内核版本initramfs之间存在不一致。因此，在启用early-kdump启动， early-kdump失败。此外，如果early-kdump检测到它被包含在标准的kdump initramfs映像中，则会强制退出。因此，如果将early-kdump添加为默认的dracut模块，则在尝试重建kdump initramfs时，标准kdump服务也会失败。因此， early-kdump和标准kdump都失败了。要解决此问题，请不要在dracut.conf文件中添加add_dracutmodules+=earlykdump或任何等效配置。因此，默认情况下， dracut不包含early-kdump ，这可以防止问题的发生。但是，如果需要early-kdump图像，则必须手动创建。

（BZ＃1662911）

调试内核无法在RHEL 8中的崩溃捕获环境中启动

由于调试内核的内存要求很高，因此在使用调试内核并触发内核崩溃时会出现问题。因此，调试内核无法作为捕获内核引导，而是生成堆栈跟踪。若要解决此问题，请相应地增加崩溃内核内存。因此，调试内核在崩溃捕获环境中成功引导。

（BZ＃1659609）

软件管理

更新libdnf软件包后，在非root用户下运行yum list命令时， YUM可能会意外终止。如果您遇到此错误，请在root下运行yum list以解决问题。因此，后续尝试在非root用户下运行yum list不再导致YUM崩溃。

（BZ＃1642458）

默认情况下， YUM v4会跳过不可用的存储库

YUM v4默认为所有存储库的“skip_if_unavailable = True”设置。因此，如果所需的存储库不可用，则在安装，搜索或更新操作中不会考虑存储库中的包。随后，即使存在不可用的存储库，一些yum命令和基于yum的脚本也会以退出代码0成功。

目前，除了更新libdnf软件包之外，没有其他可用的解决方法。

（ BZ＃1679509 ）

基建服务

nslookup和host实用程序忽略名称服务器的回复，其递归不可用

如果配置了更多名称服务器并且名称服务器没有递归，则nslookup和host实用程序将忽略来自此名称服务器的回复，除非它是上次配置的回复。在最后配置的名称服务器的情况下，即使没有recursion available标志，也接受答案。但是，如果最后配置的名称服务器没有响应或无法访问，则名称解析将失败。

要解决这个问题：

  • 确保已配置的名称服务器始终使用recursion available标志设置进行回复。
  • 允许所有内部客户端的递归。

要解决该问题，您还可以使用dig实用程序来检测递归是否可用。

（BZ＃1599459）

硬件启用

使用ARP链接监视器时，备份从站MII状态不起作用

默认情况下，由i40e驱动程序管理的设备执行源修剪，这会丢弃具有与其中一个接收过滤器匹配的源媒体访问控制（MAC）地址的数据包。因此，在通道绑定中使用地址解析协议（ARP）监视时，备份从属媒体独立接口（MII）状态不起作用。若要解决此问题，请通过以下命令禁用源修剪：

# ethtool --set-priv-flags <ethX> disable-source-pruning on

因此，备份从属MII状态将按预期工作。

（BZ＃1645433）

身份管理

KCM凭证缓存不适用于单个凭证缓存中的大量凭据

Kerberos凭据管理器（KCM）可以处理高达64 kB的ccache大小。如果它包含太多凭据，则由于用于在sssd-kcm组件和底层数据库之间传输数据的缓冲区的硬编码限制，Kerberos操作（如kinit ）会失败。

要解决此问题，请在/etc/sssd/sssd.conf文件的kcm部分中添加ccache_storage = memory选项。这指示kcm响应者仅将凭证高速缓存存储在内存中，而不是持久存储。如果这样做，重新启动系统或sssd-kcm将清除凭据缓存。

（BZ＃1448094）

对/etc/nsswitch.conf文件的任何更改（例如，运行authselect select profile_id命令）都需要重新启动系统，以便所有相关进程都使用/etc/nsswitch.conf文件的更新版本。如果无法重新启动系统，请重新启动将系统加入Active Directory的服务，即Active Directory System Security Services Daemon （SSSD）或winbind 。

（ BZ＃1657295 ）

冲突的超时值会阻止SSSD连接到服务器

与系统安全服务守护程序（SSSD）使用的故障转移操作相关的某些默认超时值是冲突的。因此，为SSSD保留与单个服务器通信的超时值可防止SSSD在连接操作之前尝试其他服务器作为整个超时。要解决此问题，设置的值ldap_opt_timeout比价值更高的超时参数dns_resolver_timeout参数，并且设定的值dns_resolver_timeout参数比的值越高dns_resolver_op_timeout参数。

（BZ＃1447945）

SSSD为本地用户返回不正确的LDAP组成员身份

如果系统安全服务后台驻留程序（SSSD）从本地文件为用户提供服务，则文件提供程序不包含来自其他域的组成员身份。因此，如果本地用户是LDAP组的成员，则id local_user命令不会返回用户的LDAP组成员身份。要解决此问题，请还原系统查找/etc/nsswitch.conf文件中用户的组成员身份的数据库的顺序，将sss files替换为files sss ，或通过添加禁用隐式files域

enable_files_domain=False

到/etc/sssd/sssd.conf文件中的[sssd]部分。

因此， id local_user为本地用户返回正确的LDAP组成员身份。
