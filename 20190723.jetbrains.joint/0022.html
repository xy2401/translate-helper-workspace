通过神器生成签名版本APK

要在物理设备上部署和运行Android应用程序，您需要以数字方式对应用程序进行签名 。使用IntelliJ IDEA，您可以在包提取时使用现有发行密钥签署Android应用程序包（ .apk文件）。IntelliJ IDEA还包含一个可在打包过程中调用的释放密钥生成工具。生成的密钥保存在密钥库二进制文件中。

您可以根据需要拥有任意数量的密钥库文件和密钥，并使用现有密钥，或在现有密钥库中创建新密钥，甚至创建新的密钥库。

除了使用Generate Signed APK向导 ，您还可以通过创建Android应用程序工件定义将.apk文件配置为工件 。当IntelliJ IDEA按照此定义构建程序包时，程序包将自动签名。

使用工件定义提取和签署版本的Android应用程序包

  1. 选择文件|从主菜单中选择项目结构 ，然后单击左窗格中的Artifacts 。

  2. 单击“ 新建”按钮并从上下文菜单中选择Android Application 。

  3. 在弹出菜单中，通过选择以下选项之一来定义工件内容：

    • 要创建空布局定义，请选择“ 空” 。

    • 要在工件中包含模块数据，请选择From module <module name>

    新工件的常规设置将显示在右侧的“ 工件布局”窗格中。

  4. 指定工件的常规设置 。在“ 输出目录”文本框中，指定目标包.apk文件的位置。

  5. 使用以下步骤完成工件定义：

    • 配置工件结构 。

    • 向工件添加资源 。

    • 排列工件中包含的元素 。

    • 如有必要，请在“ 预处理”和“ 后处理”选项卡中指定在构建工件之前和之后要执行的其他活动。

  6. 切换到Android选项卡选项卡，然后从“ 类型”下拉列表中选择“ 发布已签名 ”。

  7. 指定要使用的发行密钥以及包含它的密钥库文件。执行以下操作之一：
    • 使用现有密钥库文件中的密钥对包进行签名：

      1. 在密钥库路径文本框中指定文件位置。手动键入路径或单击“ 选择现有”按钮，然后在打开的对话框中选择该文件。在密钥库密码文本框中，键入所选密钥库的密码。

      2. 指定密钥别名并输入密码以访问密钥。

    • 要在现有密钥库中生成新密钥：

      1. 在密钥库路径文本框中指定文件位置。手动键入路径或单击“ 选择现有”按钮，然后在打开的对话框中选择该文件。在密钥库密码文本框中，键入所选密钥库的密码。

      2. 单击“ 创建新”按钮，然后通过填写打开的“ 新建密钥库”对话框中的数据来配置要生成的版本密钥 。

    • 要使用新密钥生成新的密钥库文件：

      1. 单击“ 创建新”按钮。在打开的“ 新建密钥库”对话框中，在“ 密钥库路径”文本框中指定要生成的文件的位置。手动键入路径或单击“ 浏览” ，然后选择父文件夹并指定文件的名称。

      2. 指定并确认密码以访问密钥库。

      3. 通过填写Key区域中的数据来配置新的释放密钥 。

    单击下一步 。

  8. 要让IntelliJ IDEA在打包过程中对应用程序进行模糊处理 ，请选中Run ProGuard复选框并指定proguard.txt配置文件的位置。该文件在项目创建时生成，并存储在项目根目录中。IntelliJ IDEA在“ 配置文件路径”文本框中建议此默认位置。单击“ 浏览”接受建议或指定自定义配置文件并在打开的对话框中选择所需的文件。

更新IntelliJ IDEA

默认情况下，IntelliJ IDEA配置为自动检查更新，并在新版本可用时通知您。更新是基于补丁的 ：它们应用于现有安装，只需要重新启动IDE。

工具箱应用程序

如果您使用Toolbox App安装了IntelliJ IDEA ，它将建议在新版本可用时进行更新。如果您希望Toolbox App自动更新所有托管工具：

  1. 打开工具箱应用程序，然后单击右上角的螺母图标以打开设置。

  2. 选择自动更新所有工具 。

您还可以单独为每个受管IDE实例配置更新设置：

  1. 打开工具箱应用程序，单击必要实例的螺母图标，然后选择设置 。

  2. 选择是否要保留当前主要版本，自动更新实例以及要使用的更新通道：

    • 发布 ：仅更新为推荐用于生产的稳定版本。

    • 发布和EAP ：包括测试版，候选版本和早期访问计划 （EAP）版本的更新，这些版本不建议用于生产并包括功能预览。

    • 发布，EAP和夜晚 ：包括高度不稳定的夜间构建。

独立实例

如果手动安装了IntelliJ IDEA ，则更新由独立实例本身管理。它将在新版本可用时通知您。您可以选择更新当前实例，下载并安装新版本作为单独的实例，推迟通知或完全忽略更新。

您可以在“ 设置/首选项”对话框中的“ 系统设置”下的“ 更新”页面上管理IntelliJ IDEA更新（ Ctrl + Alt + S ）：

手动检查更新

如果禁用了自动更新检查，请单击“ 立即检查”按钮以查看是否有更新的版本。

或者，您可以从主菜单中检查更新：

  • 对于Windows和Linux： 帮助|检查更新

  • 对于macOS： IntelliJ IDEA |检查更新

选择更新频道

使用更新通道列表来限制您要使用的版本类型。

只有在使用稳定版本时才能选择频道。对于EAP构建，通道始终设置为Early Access Program 。

在编辑器语法中探索HTTP请求

要在WebStorm代码编辑器中撰写HTTP请求，请使用以下常规语法：

方法Request-URI HTTP-Version标头字段：标头值Request-Body

编辑器格式的HTTP请求引入了其他功能，如以下示例所示。有关使用HTTP请求的详细信息，请参阅WebStorm代码编辑器中的HTTP客户端 。

在HTTP请求中使用注释

  • 在请求中，使用//或#开始任何行以使其成为注释行。

    //基本请求GET http://example.com/a/

使用简短形式进行GET请求

  • 对于GET请求，您可以省略请求方法并仅指定URI。

    //基本请求http://example.com/a/

在单个文件中撰写多个请求

  1. 通过在其下方键入###分隔符来标记请求的结尾。

    //一个基本请求http://example.com/a/ ###

  2. 在分隔符下面撰写另一个请求。

    //一个基本请求http://example.com/a/ ### //使用GET方法的第二请求http://example.com:8080/api/html/get?id = 123＆ 值 =内容

将长请求分成几行

  • 缩进所有查询字符串行，但第一个。

    //使用缩进的换行符GET http://example.com:8080 / api / html / get？id = 123 ＆ value = content

使用身份验证访问Web服务

  • 根据您访问的Web服务，提供基本或摘要 Authorization标头。

    //基本身份验证GET http://example.com授权：基本用户名密码### //摘要身份验证GET http://example.com授权：摘要用户名密码

与其他HTTP请求的元素类似，可以通过环境变量参数化提供的username和password 。

提供请求消息正文

  • 在请求中，在请求正文前面加一个空行并执行以下操作之一：

    • 输入请求正文：

      //提供请求正文POST http://example.com:8080/api/html/post HTTP / 1.1 Content-Type：application / json Cookie： key = first-value { “key” ： “value” ， “列表” ： [ 1,2,3 ] }

      如果将Content-Type头字段值设置为WebStorm支持的某种语言 ，则相应的语言片段将自动注入HTTP请求消息体。如果未指定Content-Type则可以手动注入语言片段。有关更多信息，请参阅语言注入 。

    • 要从文件中读取请求正文，请键入<符号，然后键入文件的路径。

      //从文件POST中读取请求正文http://example.com:8080/api/html/post Content-Type：application / json <./input.json

使用multipart / form-data内容类型

  • 将请求的Content-Type为multipart/form-data 。要将文件作为multipart/form-data消息的一部分发送，请在Content-Disposition标头中包含filename参数。

    POST http://example.com/api/upload HTTP / 1.1 Content-Type：multipart / form-data ; boundary = boundary --boundary Content-Disposition：form-data ; name = “first” ; filename = “input.txt” // 'input.txt'文件将被上传<./input.txt --boundary Content-Disposition：form-data ; name = “second” ; filename = “input-second.txt” //将创建一个带有“Text”内容的临时“input-second.txt”文件并上传Text --boundary Content-Disposition：form-data ; name = “third” ; //'input.txt'文件内容将以纯文本形式发送。<./input.txt --boundary--

要加快创建multipart / form-data请求，请使用mptr live模板 。

启用或禁用以下重定向

根据您访问的Web服务，您可能希望HTTP请求遵循重定向或不遵循重定向。当遵循重定向时，返回重定向的页面响应;否则，返回实际的重定向响应头（例如301或302 ）。

  • 在请求之前，添加带有@no-redirect标记的注释行。

    // @ no-redirect example.com/status/301

启用或禁用将请求保存到请求历史记录

如有必要，可以阻止将请求保存到请求历史记录中 。如果请求包含一些敏感数据，并且您不想记录它，这可能会有所帮助。

  • 在请求之前，添加带有@no-log标记的注释行。

    // @ no-log GET example.com/api

验证Web内容文件

PyCharm执行两种不同的有效性检查：

  • 动态验证可用于所有Web内容文件，并在编辑文件时执行。PyCharm检查格式良好，即检测各种违反语法要求的行为，例如未关闭的标记，错误的结束标记名称，重复标记，未解析的链接等。所有遇到的错误都会在编辑器中突出显示。

    但是，这种形式的代码验证相当软 ，即并非所有要求都被考虑在内。

  • 除了良好形成检查之外， 完全验证还涉及结构验证。对与XSD（XML架构定义）架构关联的文件或包含数据类型定义（DTD）的文件可以进行完全验证。PyCharm检查XML文件的结构是否符合相应DTD或Schema中定义的结构。

    完整验证的结果以消息视图的形式提供。

选择默认的HTML语言级别

通常，HTML或XHTML文件具有<!DOCTYPE>声明，声明源文件中使用的语言级别 。此语言级别用作验证文件内容的标准。如果HTML或XHTML文件没有<!DOCTYPE>声明，将根据默认标准（模式）验证文件的内容。

取消更改

还原未提交的更改

在提交之前，您始终可以撤消在本地完成的更改：

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 本地更改”选项卡。

  2. 在活动更改列表中，选择要还原的一个或多个文件，然后从上下文菜单中选择“ 还原 ”，或按Ctrl + Alt + Z.自上次提交以来对所选文件所做的所有更改都将被撤消，并且它们将从“ 本地更改”视图中消失。

撤消上次提交

IntelliJ IDEA允许您撤消当前分支中的最后一次提交（即HEAD）：

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 日志”选项卡。

  2. 选择当前分支中的最后一个提交，然后从上下文菜单中选择“ 撤消提交 ”。

  3. 在打开的对话框中，选择一个更改列表，其中将移动要放弃的更改。您可以从“ 名称”下拉列表中选择现有的更改列表，也可以指定新更改列表的名称（默认情况下使用提交消息）。

  4. 如果要使用要放弃活动更改列表的更改来创建更改列表，请选择“ 设置活动”选项。

  5. 如果希望IntelliJ IDEA记住您的上下文并在此更改列表变为活动状态时在编辑器中重新加载当前打开的文件，请选择“ 跟踪上下文”选项。

如果将提交推送到受保护的分支，即不允许使用force --push的分支（在Version Control | Git下的Settings / Preferences对话框（ Ctrl + Alt + S ）中配置受保护的分支），则无法撤消提交。

还原推送的提交

如果您发现已推送的特定提交中存在错误，则可以还原该提交。此操作将导致新提交，该提交会反转您要撤消的提交的效果。因此，保留项目历史记录，因为原始提交保持不变。

  1. 在“ 日志”视图中找到要还原的提交，右键单击它，然后从上下文菜单中选择“ 还原 ”。“ 提交更改”对话框将打开，并显示自动生成的提交消息。

  2. 如果所选提交包含多个文件，并且您只需要还原其中一些文件，请取消选择您不想触摸的文件。

  3. 单击“ 提交”以提交一个更改集，该更改集将更改还原到此特定提交中的选定文件。

将分支重置为特定提交

如果您发现一组最近提交中的错误并想要重做该部分，则可以将存储库回滚到特定状态。这是通过将当前分支HEAD重置为指定的提交（如果您不希望在历史记录中反映撤消，则可选地重置索引和工作树）来完成的。

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 日志”选项卡。

  2. 选择要将HEAD移动到其上的提交，然后从上下文菜单中选择“将当前分支重置为此处 ”。

  3. 在打开的Git Reset对话框中，选择您希望如何更新工作树和索引，然后单击Reset ：

    • Soft ：所有提交之后提交的所有更改都将被暂存（即，它们将被移动到Local Changes视图，以便您可以查看它们并在必要时稍后提交）。

    • 混合 ：将保留所选提交后所做的更改，但不会为提交暂存。

    • 硬 ：在所选提交之后进行的所有更改都将被丢弃（分阶段和提交）。

    • 保持 ：在所选提交之后进行的已提交更改将被丢弃，但本地更改将保持不变。

获取文件的先前版本

如果您需要还原单个文件而不是丢弃包含对多个文件的更改的整个提交，则可以返回到该文件的特定版本：

  1. 在任何视图中选择所需文件（在“ 项目”工具窗口中，在编辑器中，在“ 版本控制”工具窗口的“ 本地更改”选项卡中，等等）。

  2. 选择Git |从主VCS菜单或选择的上下文菜单中显示历史记录 。“ 历史记录”选项卡将添加到“ 版本控制”工具窗口，其中显示所选文件的历史记录，并允许您查看和比较其修订版本。

  3. 当您确定要回滚的修订版本后，在列表中选择它，然后从上下文菜单中选择“ 获取 ”。

IntelliJ IDEA 2019.1帮助

当地历史

IntelliJ IDEA使用本地历史记录不断跟踪对项目所做的所有更改。与显示特定提交之间差异的版本控制系统不同， 本地历史记录会自动维护所有有意义事件的修订。这会生成项目结构和源代码更改的详细时间表，并能够在必要时回滚到任何点。

本地历史不能取代适当的源代码控制。

安装新版本的IntelliJ IDEA或使缓存无效时，它会被清除。它具有保留期和最大大小，因此不能保证修订永久存在。

要查看本地历史记录：

  • 在VCS菜单上，指向“ 本地历史记录” ，然后单击“ 显示历史记录” 。

本地历史记录包含修订列表，并显示它们之间的差异。查看目录的本地历史记录时 ，修订版仅显示目录中已更改的文件的名称。特定文件的修订包含对文件内容的更改。

本地历史记录不适用于二进制文件的内容。

要恢复到特定修订：

  • 从列表中选择它，然后单击工具栏上的“ 还原 ”。

这会为本地历史记录添加新修订。

查看源代码元素的本地历史记录

在某些情况下，您可能不希望查看整个文件的本地历史记录 ，但仅针对特定的源代码元素：类，其成员（字段和方法）或任何选定的文本片段。

要查看班级的本地历史记录：

要查看类方法的本地历史记录：

  • 右键单击编辑器中的方法名称，指向“ 本地历史记录” ，然后单击“ 显示方法的历史记录” 。

要查看源代码片段的本地历史记录：

  • 在编辑器中右键单击所选文本，指向“ 本地历史记录” ，然后单击“ 显示选择历史记录” 。

要为当前修订添加标签：

  1. 在VCS菜单上，指向“ 本地历史记录” ，然后单击“ 放置标签” 。

  2. 在“ 放置标签”对话框中，键入标签名称，然后单击“ 确定” 。

  • 在Local History中选择相关修订，然后单击工具栏上的Create Patch 。

要查看最近的更改：

  • 在视图菜单上，单击最近的更改 。

您可以选择您感兴趣的更改以查看差异，并在必要时还原它。

本地历史的保留期限

  3. 修改localHistory.daysToKeep参数的值。

  4. 单击“ 关闭”并重新启动IntelliJ IDEA以使更改生效。

撤消更改 - 帮助| PyCharm

发送反馈

PyCharm 2019.1帮助

取消更改

还原未提交的更改

在提交之前，您始终可以撤消在本地完成的更改：

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 本地更改”选项卡。

  2. 在活动更改列表中，选择要还原的一个或多个文件，然后从上下文菜单中选择“ 还原 ”，或按Ctrl + Alt + Z.自上次提交以来对所选文件所做的所有更改都将被撤消，并且它们将从“ 本地更改”视图中消失。

撤消上次提交

PyCharm允许您撤消当前分支中的最后一次提交（即HEAD）：

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 日志”选项卡。

  2. 选择当前分支中的最后一个提交，然后从上下文菜单中选择“ 撤消提交 ”。

  3. 在打开的对话框中，选择一个更改列表，其中将移动要放弃的更改。您可以从“ 名称”下拉列表中选择现有的更改列表，也可以指定新更改列表的名称（默认情况下使用提交消息）。

  4. 如果要使用要放弃活动更改列表的更改来创建更改列表，请选择“ 设置活动”选项。

  5. 如果希望PyCharm记住您的上下文并在此更改列表变为活动状态时在编辑器中重新加载当前打开的文件，请选择“ 跟踪上下文”选项。

如果将提交推送到受保护的分支，即不允许使用force --push的分支（在Version Control | Git下的Settings / Preferences对话框（ Ctrl + Alt + S ）中配置受保护的分支），则无法撤消提交。

还原推送的提交

如果您发现已推送的特定提交中存在错误，则可以还原该提交。此操作将导致新提交，该提交会反转您要撤消的提交的效果。因此，保留项目历史记录，因为原始提交保持不变。

  1. 在“ 日志”视图中找到要还原的提交，右键单击它，然后从上下文菜单中选择“ 还原 ”。“ 提交更改”对话框将打开，并显示自动生成的提交消息。

  2. 如果所选提交包含多个文件，并且您只需要还原其中一些文件，请取消选择您不想触摸的文件。

  3. 单击“ 提交”以提交一个更改集，该更改集将更改还原到此特定提交中的选定文件。

将分支重置为特定提交

如果您发现一组最近提交中的错误并想要重做该部分，则可以将存储库回滚到特定状态。这是通过将当前分支HEAD重置为指定的提交（如果您不希望在历史记录中反映撤消，则可选地重置索引和工作树）来完成的。

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 日志”选项卡。

  2. 选择要将HEAD移动到其上的提交，然后从上下文菜单中选择“将当前分支重置为此处 ”。

  3. 在打开的Git Reset对话框中，选择您希望如何更新工作树和索引，然后单击Reset ：

    • Soft ：所有提交之后提交的所有更改都将被暂存（即，它们将被移动到Local Changes视图，以便您可以查看它们并在必要时稍后提交）。

    • 混合 ：将保留所选提交后所做的更改，但不会为提交暂存。

    • 硬 ：在所选提交之后进行的所有更改都将被丢弃（分阶段和提交）。

    • 保持 ：在所选提交之后进行的已提交更改将被丢弃，但本地更改将保持不变。

获取文件的先前版本

如果您需要还原单个文件而不是丢弃包含对多个文件的更改的整个提交，则可以返回到该文件的特定版本：

  1. 在任何视图中选择所需文件（在“ 项目”工具窗口中，在编辑器中，在“ 版本控制”工具窗口的“ 本地更改”选项卡中，等等）。

  2. 选择Git |从主VCS菜单或选择的上下文菜单中显示历史记录 。“ 历史记录”选项卡将添加到“ 版本控制”工具窗口，其中显示所选文件的历史记录，并允许您查看和比较其修订版本。

  3. 当您确定要回滚的修订版本后，在列表中选择它，然后从上下文菜单中选择“ 获取 ”。

验证Web内容文件

IntelliJ IDEA执行两种不同的有效性检查：

  • 动态验证可用于所有Web内容文件，并在编辑文件时执行。IntelliJ IDEA检查格式良好，即检测各种违反语法要求的行为，例如未关闭的标记，错误的结束标记名称，重复标记，未解析的链接等。所有遇到的错误都会在编辑器中突出显示。

    但是，这种形式的代码验证相当软 ，即并非所有要求都被考虑在内。

  • 除了良好形成检查之外， 完全验证还涉及结构验证。对与XSD（XML架构定义）架构关联的文件或包含数据类型定义（DTD）的文件可以进行完全验证。IntelliJ IDEA检查XML文件的结构是否符合相应DTD或Schema中定义的结构。

    完整验证的结果以消息视图的形式提供。

选择默认的HTML语言级别

通常，HTML或XHTML文件具有<!DOCTYPE>声明，声明源文件中使用的语言级别 。此语言级别用作验证文件内容的标准。如果HTML或XHTML文件没有<!DOCTYPE>声明，将根据默认标准（模式）验证文件的内容。

  1. 在“ 设置/首选项”对话框（ Ctrl + Alt + S ）中，单击“ 语言和框架”下的“ 架构和DTD ”，然后单击“ 默认XML架构” 。将打开“ 默认XML架构”页面。

  2. 在“ 默认HTML语言级别”区域中，选择默认架构以验证HTML和XHTML文件，而不使用<!DOCTYPE>声明。可用选项包括：

    • HTML 4或HTML 5 ：选择其中一个选项，将文件视为HTML 4或HTML 5，并根据其中一个标准进行验证。

    • 其他doctype ：选择此选项可使HTML文件默认再次验证自定义DTD或模式，并指定要使用的DTD或模式的URL。

      请注意，此字段中提供了代码完成功能：按Ctrl + Space可查看建议的URL列表。

  3. 选择XSD（XML架构定义）架构以验证XML文件。可用选项包括：

    • XML Schema 1.1请参阅W3C XML模式定义语言（XSD）1.1第1部分：结构和W3C XML模式定义语言（XSD）1.1第2部分：数据类型以获取详细信息。

    • XML Schema 1.0有关详细信息，请参阅XML架构第1部分：结构第二版和XML架构第2部分：数据类型第二版 。

选择默认架构以验证XML文件

  1. 在“ 设置/首选项”对话框（ Ctrl + Alt + S ）中，单击“ 语言和框架”下的“ 架构和DTD ”，然后单击“ 默认XML架构” 。将打开“ 默认XML架构”页面。

  2. 在“ 默认XML模式版本”区域中，选择XSD（XML模式定义）模式以验证XML文件。可用选项包括：

    • XML Schema 1.1请参阅W3C XML模式定义语言（XSD）1.1第1部分：结构和W3C XML模式定义语言（XSD）1.1第2部分：数据类型以获取详细信息。

    • XML Schema 1.0有关详细信息，请参阅XML架构第1部分：结构第二版和XML架构第2部分：数据类型第二版 。

对XML文件运行完全验证

  1. 在编辑器中打开所需的XML文件，或者只在“ 项目”工具窗口中选择它。

  2. 从上下文菜单中，选择“ 验证” 。

与远程存储库同步

在通过将更改推送到上游来共享工作结果之前，需要与远程存储库同步以确保项目的本地副本是最新的。您可以通过以下方式之一执行此操作： 获取更改 ， 拉 取 更改 ， 更新项目 。

获取更改

从上游获取更改时，自上次与远程存储库同步后提交的所有新数据都将下载到本地副本中。此新数据未集成到本地文件中，更改不会应用于您的代码。

提取的更改将存储为远程分支，这使您有机会在将它们与文件合并之前对其进行检查。由于fetch不会影响您的本地开发环境，因此这是一种更新远程存储库所有更改的安全方法。

要获取更改，请从主菜单中选择VCS | Git |抓取 。

拉变化

从远程仓库拉变化为方便快捷方式获取并随后合并的变化。当您拉动时 ，您不仅可以下载新数据，还可以将其集成到项目的本地工作副本中。

要进行更改，请执行以下操作：

  1. 从主菜单中，选择VCS | Git |拉 。“ 拉动更改”对话框打开。

  2. 如果您的项目有多个Git存储库，请从Git Root下拉列表中选择要更新的本地存储库的路径。

  3. 如果为存储库配置了多个远程数据库，请在“ 远程”下拉列表中选择要从中提取数据的远程数据库的URL。

  4. 选择要从中获取更改的分支，然后合并到当前签出的分支中。

  5. 从“ 策略”下拉列表中，选择将用于解决合并期间发生的冲突的合并策略 。

  6. 如有必要，请选择以下内容

    • 无提交 ：如果您不希望WebStorm自动提交合并结果，请选择此选项。在这种情况下，您可以检查它们并在必要时进行调整。

    • 无快进 ：选择此选项可生成合并提交，即使合并已解析为快进（即仅更新了分支指针）。

    • Squash commit ：选择此选项可在当前分支之上创建单个提交，而不是合并一个或多个分支。它产生工作树和索引状态，就好像真正的合并发生一样，但它实际上并没有提交或移动HEAD。

    • 添加日志信息 ：如果希望WebStorm使用除分支名称之外的正在合并的实际提交中的单行描述填充日志消息，请选择此选项。

  7. 单击“ 拉”以从所选远程存储库中获取并应用更改。

已预先选择已跟踪的远程分支。

更新您的项目

如果您有多个项目根，或者每次与远程存储库同步时想要从所有分支获取更改，您可能会发现更新项目是一个更方便的选项。

执行更新操作时，WebStorm从所有项目根和分支中获取更改，并将跟踪的远程分支合并到本地工作副本（相当于pull ）。

如果您的项目包含子模块 ，并且它们位于分支上，则它们也将自动更新。

如果子模块处于分离的HEAD状态，WebStorm将调用git submodule update ，该git submodule update将检查根存储库中引用的提交。这意味着仅当子模块中的子模块引用更改或添加了新的子模块时才会执行更新。

要更新项目，请执行以下操作：

  1. 从主菜单中，选择VCS |更新项目或按Ctrl + T. “ 更新项目”对话框打开。

  2. 选择更新类型（此策略将应用于Git版本控制下的所有根目录）：

    • 合并 ：选择此选项可在更新期间执行合并 。这相当于运行git fetch然后运行git fetch git merge或git pull --no-rebase 。

    • Rebase ：选择此选项可在更新期间执行rebase 。这相当于运行git fetch然后运行git fetch git rebase ，或者git pull --rebase （所有本地提交都将放在更新的上游头之上）。

    • 分支默认值 ：如果要为不同分支应用不同的更新策略，请选择此选项。您可以为.git/config配置文件的branch.<name>部分中的每个分支指定默认更新类型。

  3. 指定在更新之前清理工作副本时用于保存更改的方法，以便在更新完成后还原未提交的更改：

    • 使用Stash ：选择此选项可在git存储中保存本地更改。如果您需要在WebStorm之外应用具有隐藏更改的修补程序，这非常有用，因为它们是由Git本身生成的。

    • 使用Shelve ：选择此选项可将本地更改放入工具架 。搁置由WebStorm完成，搁置更改生成的补丁通常应用于WebStorm中。

如果您以后选择不显示“ 更新项目”对话框，然后想要稍后修改默认更新策略，请在“ 设置/首选项”对话框（ Ctrl + Alt + S ）中选择“ 版本控制”|“控制”。确认 ，在调用这些命令时选择“ 显示选项”对话框下的“更新”，并在下次执行更新时修改更新策略。

提取方法

Extract Method重构允许您获取可以组合在一起的代码片段，将其移动到单独的方法中，并通过调用方法替换旧代码。

提取方法时，需要检查变量。如果有一个输出变量，则将其用作提取方法的返回值。如果存在多个输出变量，则可能不会应用“ 提取方法”重构，并显示错误消息。

在这种情况下，有几种解决方法允许Extract Method工作。例如，您可以引入包含所有输出值的特殊数据类。

Extract Method重构具有以下限制：

  • 重构不适用于自动模式下的多个输出值。在应用重构之前，您必须更改代码。

  • 重构不适用于有条件地从包含方法返回的代码片段，并且不会放在它的末尾。

提取方法：

  1. 选择要提取到方法的代码片段。

  2. 按Ctrl + Alt + M或在主菜单上选择Refactor |提取|方法 。

  3. 在打开的对话框中，配置方法选项，例如可见性，参数等。如果需要，您还可以更改方法的名称。

    单击确定 。

    准备好后，单击“ 执行重构”以完成重构。

    请注意，如果它只是一个副本，IntelliJ IDEA将静默替换它。

    最后，如果IntelliJ IDEA检测到仅部分重复的代码，则建议您提取参数以继续重构。

  1. 在“ 设置/首选项”对话框（ Ctrl + Alt + S ）中，单击“代码样式” ，然后单击该语言。

  2. 选择代码样式方案，以用作所选语言的自定义编码样式的基础。

  3. 浏览所选语言页面的选项卡，并为其配置代码样式首选项。

仅显示这些语言的Set from链接，其中可以使用其他语言定义设置。

从其他语言复制代码样式设置

对于大多数受支持的语言，您可以从其他语言或框架复制代码样式设置。

  1. 在“ 设置/首选项”对话框（ Ctrl + Alt + S ）中，单击“代码样式” ，然后单击该语言。

  2. 单击右上角的“ 设置自 ”。

  3. 从显示的列表中，选择要从中复制代码样式的语言。

应用特定于框架的预配置编码标准

对于PHP文件，您可以应用特定于框架的预配置编码标准。

  1. 在“ 设置/首选项”对话框（ Ctrl + Alt + S ）中，单击“代码样式” ，然后单击该语言。

  2. 单击Set from链接，选择Predefined ，然后选择相关的预配置标准。

使用EditorConfig配置项目的代码样式

在开始使用EditorConfig之前，请确保已启用EditorConfig插件。该插件与IntelliJ IDEA捆绑在一起，默认情况下处于激活状态。如果未激活插件 ，请在“ 管理插件 ”中所述的“设置/首选项”对话框的“ 插件”页面上启用它。

  1. 在“ 设置/首选项”对话框（ Ctrl + Alt + S ）中，转到“ 编辑器”|代码样式并确保选中 “ 启用EditorConfig支持”复选框 。

  2. 在项目树中，右键单击要在其中创建EditorConfig设置文件的目录，然后选择“ 新建”|档案 。

  3. 在打开的对话框中，输入.editorconfig并单击“ 确定” 。

    IntelliJ IDEA创建EditorConfig设置文件并在弹出窗口中显示通知 。

    每次打开文件时，EditorConfig插件.editorconfig在打开的文件的目录和每个父目录中查找名为.editorconfig的文件。如果到达根文件路径或找到root = true的EditorConfig文件，将停止搜索.editorconfig文件。因此，如果要使用IDE设置而不是EditorConfig设置，请清除编辑器中的“ 启用EditConfig支持”复选框。默认选择的代码样式 。

  4. 定义代码样式设置并保存.editorconfig （ Ctrl + S ）以立即将设置应用于项目。EditorConfig代码样式配置会覆盖IntelliJ IDEA设置中的代码样式配置。
