运行|编辑配置| | NodeUnit

在此对话框中，创建配置以运行Node.js应用程序的单元测试。

在你开始之前

  1. 下载并安装Node.js.

  2. 使用以下方法之一安装Nodeunit测试框架：

    • 从https://github.com/caolan/nodeunit下载框架并按照官方说明进行安装。

    • 使用节点包管理器（NPM）安装nodeunit包，如npm和Yarn中所述 。

  3. 在插件页面上安装Node.js插件，如管理插件中所述 。

NodeUnit特定的配置设置

项目

描述

节点解释器

在此字段中，指定要使用的Node.js解释器。这可以是Windows子系统Linux上的本地Node.js解释器或Node.js。

工作目录

在此文本框中，指定要在其下查找测试的文件夹。这可以是项目根文件夹或test文件夹的父目录。手动键入路径或单击“ 浏览” 并在对话框中选择打开的位置。

环境变量

在此字段中，指定Node.js可执行文件的环境变量 （如果适用）。单击“ 浏览” 在该字段的右侧并在“ 环境变量”对话框中配置变量列表，该对话框将打开：

  • 要定义新变量，请单击“ 添加” 并指定变量名称和值。

  • 要放弃变量定义，请在列表中选择它，然后单击“ 删除” 。

  • 准备好后，单击“ 确定”

变量的定义显示在“ 环境变量只读”字段中，并以分号作为分隔符。可接受的变量是：

  • NODE_PATH ：A :以模块搜索路径为前缀的NODE_PATH目录列表。

  • NODE_MODULE_CONTEXTS ：设置为1以在其自己的全局上下文中加载模块。

  • NODE_DISABLE_COLORS ：设置为1以禁用REPL中的颜色。

Nodeunit模块

在此文本框中，指定Nodeunit框架的安装文件夹。手动键入路径或单击并在打开的对话框中选择文件夹。

跑

从此下拉列表中，选择要执行的测试范围。可用选项包括：

  • 目录中的所有测试文件：选择此选项可运行文件夹中的所有测试文件，并指定此文件夹相对于工作目录的路径。

  • 测试文件：选择此选项以从特定文件执行测试，并键入相对于工作目录的此文件的路径。

常见选项

编辑运行配置（但不是运行配置模板）时，可以为其指定以下选项：

项目

描述

名称

在此文本框中，指定运行/调试配置的名称。当您选择稍后编辑或调用时，该名称将帮助您识别创建的配置，例如，从“ 运行”弹出窗口（ Shift + Alt + F10 ）。

分享

选中此复选框可使运行/调试配置可供其他团队成员使用。

如果使用基于目录的项目格式 ，则运行/调试配置的设置将存储在.idea\runConfigurations文件夹中的单独.xml文件中（如果共享运行/调试配置），或存储在.idea\workspace.xml否则为.idea\workspace.xml文件。

如果使用基于文件的格式 ，则设置将存储在.ipr文件中以用于共享配置，否则存储在.iws文件中。

允许并行运行

禁用时，每次启动新的运行/调试配置时，IntelliJ IDEA都会检查是否存在相同运行/调试配置的其他实例，并显示确认对话框。如果在确认对话框中单击“ 确定 ”，则将停止第一个运行器实例，下一个实例将取代它。

当某些资源的使用可能导致冲突时，或者当启动两个相同类型的运行/调试配置时，会消耗太多的CPU和内存资源，这是有意义的。

如果选中此复选框，则可以根据需要启动任意数量的运行/调试配置实例。这样，每个运行器将在Run Tool Window或Debug Tool Window的自己的选项卡中启动 。

角

Angular是构建跨平台应用程序的流行框架。WebStorm为Angular提供支持，并帮助您完成开发过程的每一步 - 从创建新的Angular应用程序和处理组件到调试和测试它。

在开始之前，请确保您的计算机上有Node.js.

创建一个新的Angular应用程序

您也可以在Node.js和NPM页面上安装@angular/cli包，如npm和Yarn中所述 。

您可以使用Angular CLI或创建一个空的WebStorm项目并在其中安装Angular。

Angular CLI是开始构建新Angular应用程序的推荐方法。

全局安装Angular CLI

  • 在嵌入式终端 （ Alt + F12 ）中，键入：

    npm install -g @ angular / cli

创建应用程序

  1. 选择文件|新的|项目主菜单上，或单击欢迎屏幕上创建新项目 。将打开“ 创建新项目”对话框 。

  2. 在左侧窗格中，选择Angular CLI 。

  3. 在右侧窗格中：

    1. 指定将存储与项目相关的文件的文件夹的路径。

    2. 在Node Interpreter字段中，指定要使用的Node.js解释器。从列表中选择已配置的解释器，或选择“ 添加”以配置新的解释器

    3. 在Angular CLI字段中，指定@angular/cli包的路径。

    4. （可选 ）在“ 附加参数”字段中，指定要传递给Angular CLI的额外ng新选项 。此字段中提供了代码完成功能：当您开始键入选项名称或按Ctrl + Space时 ，WebStorm会显示可用选项及其说明。

  4. 单击“ 创建”时 ，WebStorm将生成一个包含所有必需配置文件的Angular特定项目。

创建一个空的WebStorm项目

  1. 选择文件|新的|项目主菜单上，或单击欢迎屏幕上创建新项目 。将打开“ 创建新项目”对话框 。

  2. 在左侧窗格中，选择Empty Project 。

  3. 指定将存储与项目相关的文件的文件夹的路径。

  4. 单击“ 创建”时 ，WebStorm将创建并打开一个空项目。

在空项目中安装Angular

  1. 打开您将使用Angular的空项目。

  2. 在嵌入式终端 （ Alt + F12 ）中，键入：

    npm install @ angular / core

    这将使用框架的关键运行时部分安装核心Angular包。

    您可能还需要安装其他作为Angular部分的软件包，请参阅软件包列表 。

从现有的Angular应用程序开始

启动新Angular应用程序的推荐方法是Angular CLI 。只有在这种情况下，您的应用程序才会使用即时使用的TypeScript和Webpack配置进行自举。

要继续开发现有的Angular应用程序，请在WebStorm中打开它并下载所需的依赖项。

打开计算机上已有的应用程序源

  • 单击欢迎屏幕上的打开或选择文件|在主菜单上打开 。在打开的对话框中，选择存储源的文件夹。

从版本控制中检出应用程序源

  1. 单击“ 欢迎”屏幕上的“ 从版本控制中签出”或选择“ VCS”从主菜单上的版本控制中查看 。

  2. 从列表中选择您的版本控制系统。

  3. 在打开的特定于VCS的对话框中，键入您的凭据和存储库以检出应用程序源。

运行和调试Angular应用程序

此外，对于使用Angular CLI创建的应用程序，WebStorm会使用默认设置生成两个运行/调试配置：

  • 具有默认名称Angular CLI Server的npm配置 。此配置运行ng serve命令，该命令启动开发服务器并在开发模式下启动应用程序。

  • 具有默认名称Angular Application的 JavaScript Debug配置。此配置启动调试会话。

运行Angular应用程序

  1. 对于Angular CLI应用程序，从工具栏上的列表中选择Angular CLI Server运行配置，然后单击在列表旁边。

    或者，在终端中运行npm start （ Alt + F12 ）或双击npm工具窗口中的start任务（ View | Tool Windows | npm ）。

  2. 等到编译应用程序并且Webpack开发服务器准备就绪。在http://localhost:4200/打开浏览器以查看应用程序。

当开发服务器运行时，只要更改任何源文件并保存更新，就会自动重新加载应用程序。

调试Angular应用程序

  1. 在代码中设置断点。

  2. 如上所述在开发模式下启动应用程序，并等待应用程序编译并且Webpack开发服务器准备就绪。

  3. 要查看应用程序，请在“ 运行”工具窗口中显示的URL处打开浏览器，默认情况下为http://localhost:4200/ 。

    如果您使用的是其他端口，或者您的应用程序未使用Angular CLI创建，请复制实际运行应用程序的URL地址，稍后您将在调试配置中指定此URL。

  4. 如果使用Angular CLI创建应用程序并且它在默认端口4200上运行，则WebStorm会生成默认的Angular Application配置。

    否则，创建一个新的JavaScript调试配置：转到Run |编辑配置 ，单击 ，并从列表中选择JavaScript Debug 。在“ 运行/调试配置：JavaScript调试”对话框中，将保存的URL粘贴到URL字段中并保存配置。

  5. 从配置列表中，选择自动生成的Angular Application配置或新创建的自定义配置，然后单击启动调试会话在列表旁边。

  6. 当第一个断点被点击时，切换到调试工具窗口并照常进行： 逐步执行程序 ， 停止并恢复程序执行， 暂停时检查它 ，浏览调用堆栈和变量，设置监视，评估变量， 查看实际HTML DOM ，等等。

请参阅“准备远程调试”中的过程说明。

从此只读字段复制import语句，并将其粘贴到本地脚本中：

导入pydevd

添加以下命令以连接到调试服务器

复制下面的命令行语句，并将其粘贴到本地脚本。

pydevd.settrace（<主机名>，端口= <端口号>）

哪里

  • <host name>取自此调试配置的Local host name字段。

  • <port number>是从此调试配置的端口字段中获取的数字，或者，如果尚未指定，则会将一些随机数写入控制台。

请注意，此命令的参数取决于此页面中指定的设置。使用默认设置的命令是：

pydev.pydevd.settrace（'localhost'，port = $ SERVER_PORT，stdoutToServer = True，stderrToServer = True）

对应主机名'localhost'，端口号0，选中复选框将输出重定向到控制台 ， 连接后挂起 。

本地主机名

指定运行IDE的计算机的IP地址，通过该计算机可以访问远程调试服务器。该主机名将自动替换为命令行。默认情况下，使用localhost 。

港口

指定端口号，该端口号将自动替换为命令行。如果使用默认的帖子编号（0），则IntelliJ IDEA会在每次启动此调试配置时将任意数字替换为命令行;如果您指定任何其他值，它将永久使用。

路径映射

使用此字段可在本地和远程路径之间创建映射。单击“浏览”按钮打开“ 编辑路径映射”对话框，您可以在其中添加新路径映射，并删除选定的路径映射。本地路径是项目的绝对路径，例如/Users/jetbrains/PyCharmProjects/MyDeployment 。远程路径相对于SFTP服务器根路径，例如/tmp/pycharm_project_923 。

将输出重定向到控制台

如果选中此复选框，输出和错误流将重定向到IntelliJ IDEA控制台，命令行将补充stdoutToServer=True, stderrToServer=True

连接后暂停

如果选中此复选框，调试器将在连接到IDE之后立即挂起，在settrace调用之后的下一行。
如果未选中此复选框，则调试器仅在遇到断点或单击时暂停 ，命令行补充了suspend=False

同时处理多个功能

有时你需要在未完成的事情之间切换不同的任务，然后返回给他们。WebStorm为您提供了几种方便地处理多种不同功能而不会丢失工作的方法：

  • 您可以存储或搁置挂起的更改。

    存储变化与搁置非常相似。唯一的区别在于生成和应用补丁的方式。Stas由Git生成，可以在WebStorm内部或外部应用。具有搁置更改的修补程序由WebStorm生成，也可通过IDE应用。此外，存储涉及所有未提交的更改，而当您将更改添加到工具架时，您可以选择一些本地更改，而不是将它们全部搁置。

  • 您可以在不同的更改列表中保留与不同任务或功能相关的更改 。

  • 您可以创建分支以处理不同的不相关功能 。

搁置变化

搁置暂时存储您尚未提交的挂起更改。这很有用，例如，如果您需要切换到另一个高优先级任务，并且您希望将更改设置为稍后处理它们。

使用WebStorm，您可以搁置单独的文件和整个更改列表。

搁置后，可以根据需要多次进行更改，通过取消搁置并随后将其恢复到货架上。

将更改放到架子上

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 本地更改”选项卡。

  2. 右键单击要放入工具架的文件或更改列表，然后从上下文菜单中选择“ 搁置更改 ”。

  3. 在“ 搁置更改”对话框中，查看已修改文件的列表。

  4. 在“ 提交消息”字段中，输入要创建的货架的名称，然后单击“货架更改”按钮。

您还可以静默搁置更改，而不显示“ 搁置更改”对话框。要执行此操作，请选择要搁置的文件或更改列表，然后单击Shelve Silently图标在工具栏上，或按Ctrl + Alt + H.包含要搁置的更改的更改列表的名称将用作货架名称。

为避免出现大量具有相同名称的货架（例如Default ），您只需将文件或更改列表从“ 本地更改”选项卡拖放到“ 货架”选项卡，等待一秒钟直到它被激活，并在释放鼠标按钮时即时编辑新的货架名称。

您可以通过选择Changelist下拉列表从Shelve Changes对话框切换到其他更改列表。

取消改变

令人不安的是将推迟的更改从一个架子转移到一个待定的更改列表。可以从视图中过滤掉未删除的更改或从工具架中删除。

  1. 版本控制工具窗口的保存选项卡，选择您要取消搁置一个变更或文件。

  2. 按Ctrl + Shift + U或从选择的上下文菜单中选择Unshelve 。

  3. 在打开的“ 取消保护更改”对话框中，在“ 名称”字段中指定要将未经保留的更改还原到的更改列表。您可以从列表中选择现有的更改列表，也可以键入要创建的包含未更改的更改的新更改列表的名称。您可以在“ 注释”字段中输入新更改列表的说明（可选）。

    如果要激活新的更改列表，请选择“ 设置活动”选项 。否则，当前活动的更改列表仍保持活动状态。

  4. 如果希望WebStorm在其停用时保留与新更改列表关联的任务的上下文并恢复上下文，则更改列表将变为活动状态，请选择“ 跟踪上下文”选项（有关详细信息，请参阅任务和上下文 ）。

  5. 如果要删除要撤消的更改，请选择“ 从架子中删除已成功应用的文件”选项。未加载的文件将从此工具架中删除，并添加到另一个更改列表并标记为已应用。在通过单击明确删除之前，不会完全删除它们在工具栏上，或从上下文菜单中选择Clean Already Unshelved 。

    如果您意外删除了未受控制的文件，则可以从“ 最近删除”节点查看和还原它们。

  6. 单击确定 。如果修补版本与当前版本之间发生冲突，请按照解决冲突中的说明解决它们。

您还可以静默取消更改，而不显示“ 取消保护更改”对话框。要做到这一点，选择文件或要取消搁置一个变更，并单击取消搁置默默图标在工具栏上，或按Ctrl + Alt + U.未被调用的文件将被移动到活动的挂起chdrag

将组更改为不同的更改列表

当您处理多个相关功能时，您可能会发现将更改分组到不同的更改列表中非常方便。这种方法有其优点和缺点，而不是使用功能分支来处理多个任务。

优点 ：

  • 您可以轻松地在不同的逻辑变更集之间切换，并将它们彼此分开提交。

  • 与为相同目的使用分支不同，您可以手头进行所有更改，而无需在分支之间切换，如果您的项目非常大，可能需要一段时间。

  • 测试不同功能如何协同工作很方便。

  • 您可以在构建服务器上远程运行更改列表。

缺点 ：

  • 虽然使用更改列表可能看起来比分支更轻量级，但它并不安全，因为在您提交并推送它们之前，您的更改没有备份。如果您的本地工作副本出现问题，您的所有更改都将丢失，因为它们不属于Git项目历史记录。

  • 不可能对特征进行原子测试。

  • 不可能在同一功能上进行协作。此外，除非您通过电子邮件发送包含更改的修补程序，否则您无法从不同的计算机中做出贡献，这可能不太方便。

所有更改列表都显示在“ 版本控制”工具窗口的“ 本地更改”选项卡中。所有已修改的文件都会自动放入活动更改列表中，这是默认更改列表，除非您创建了另一个并使其处于活动状态。

要创建新的更改列表，请单击在工具栏上。

要激活非默认更改列表，请右键单击它并从上下文菜单中选择“ 设置活动更改列表”。

在更改列表之间移动更改

  1. 选择要移动的文件，然后从上下文菜单中选择“ 移动到另一个更改列表”。

  2. 如果要使用要放弃活动更改列表的更改来创建更改列表，请选择“ 设置活动”选项。

  3. 如果希望WebStorm记住您的上下文并在此更改列表变为活动状态时在编辑器中重新加载当前打开的文件，请选择“ 跟踪上下文”选项。

您还可以在更改列表之间拖动文件。

使用功能分支

Git中的一个分支代表了一个独立的开发线，所以如果您正在开发一个单独的功能，您希望在准备好共享工作结果并将它们集成到master之前完成并进行测试，那么在功能分支中执行它是最好的解决方案。这样，您可以确保不会将不稳定的代码提交到项目的主代码库，并且可以在必要时轻松切换到其他任务。

优点 ：

  • 与使用更改列表进行组更改相反，使用功能分支是安全的。在您向Git提交更改后，它们将成为Git项目历史记录的一部分，因此即使您损坏了工作树，也可以通过Git reflog恢复提交。推送更改后，它们将被备份。

  • 您可以开发并行的非相关功能并以原子方式对其进行测试。

  • 当您在分支中完成开发后，您可以重新排序或压缩提交 ，以便您的历史记录线性且干净。

  • 您可以轻松地在功能上进行协作，或者从不同的计算机上进行开发。

缺点 ：

  • 在真正大型项目上切换分支可能需要一些时间。

  • 一起测试相关功能并不是很方便。

  • 您必须学习使用功能分支并将更改集成到主代码库中的工作流程 。

使用功能分支并将更改集成到主代码库中有两种主要方法：

  • 合并选项

  • rebase选项

使用merge来集成功能分支中的更改

合并选项的主要好处是完全可跟踪性，因为提交合并到主代码库中保留其原始哈希和作者，并且属于一个功能的所有提交可以组合在一起。

此工作流适用于对主代码库提交更改涉及拉取请求或分层审批过程的项目，因为现有分支不会以任何方式更改。

这种方法的主要缺点是每次需要合并更改时都会创建无关的合并提交，这会严重污染项目历史记录并使其难以阅读。

  1. 为您的独立开发线创建一个分支 。

  2. 在开发过程中提交您的更改。

  3. 将分支推送到远程存储库。这应该用于备份，以便您可以在不同的计算机上进行协作或工作。

  4. 当您需要执行与您的功能无关的工作时，请切换到其他分支。

  5. 对您的功能进行审核和测试，并进行必要的修复。

  6. 当您准备将工作结果集成到主分支（例如master ）时，请执行以下操作：

    • 将您的功能分支合并到主代码库中。

    • 删除功能分支 。

    • 推。

使用rebase集成功能分支中的更改

此选项的主要好处是，您可以获得清晰的项目历史记录，以便其他人阅读和理解。您的日志不包含merge操作产生的不必要的合并提交，并且您可以获得易于导航和搜索的线性历史记录。

在决定采用此工作流程时，您应该记住， rebase重写项目历史记录，因为它会为原始要素分支中的每个提交创建新提交，因此它们将具有不同的哈希值，这会阻碍可跟踪性。

  1. 为您的独立开发线创建一个分支 。

  2. 在开发过程中经常提交更改。

  3. 将分支推送到远程存储库。这应该用于备份，以便您可以在不同的计算机上进行协作或工作。

  4. 衍合你的特性分支到master不时。如果您的功能分支很长，那么这样做才有意义。这对以下内容很有用：

    • 确保你的功能分支和master不会分开太远。

    • 当您最终将更改集成到主代码库中时，避免解决大量冲突。当你经常变换时，你可以迭代地解决冲突，并且不会因为长时间的差异而挣扎。

    • 加快检查分支，因为分支之间的切换一旦充分发散就会变慢。

    重新定位涉及以下步骤：

    • 从远程获取更改，或将更改提取到master分支。

    • 衍合你的分支到master 。

    • 强制将 rebase操作的结果推送到功能分支。

  5. 当您需要执行与您的功能无关的工作时，切换到master 。当您返回功能分支时，执行Checkout和Rebase to Current 。

  6. 对您的功能进行审核和测试，并进行必要的修复。

  7. 功能完成后执行Interactive Rebase 。这允许您重新排序和压缩提交，以使您的功能分支历史看起来漂亮和干净。

  8. 当您准备将工作结果集成到主分支（例如master ）时，请执行以下操作：

    • 签出master分支。

    • 将您的分支与master 合并 。由于master没有发散，Git只会将指针移动到功能分支的最新提交，而不是创建新的合并提交（这称为快进合并）。

    • 删除功能分支 。

    • 推 。

使用首字母或昵称（如果它很短）作为功能分支名称的前缀是有意义的。这样，您可以使用“ 分支”菜单中的速度搜索轻松找到所有分支。

V8 CPU和内存分析

使用WebStorm，您可以使用V8基于样本的分析器捕获和分析Node.js应用程序的CPU配置文件和堆快照。

您还可以打开并浏览Google Chrome DevTools中捕获的配置文件和快照，以获取客户端代码。

在你开始之前

确保您的计算机上有Node.js.

CPU分析

CPU分析可帮助您更好地了解代码的哪些部分占用大部分CPU时间，以及V8 JavaScript引擎如何执行和优化代码。

Node.js WebStorm中的CPU分析基于V8内置CPU分析器，它提供有关代码执行和JavaScript引擎本身行为的信息，包括垃圾收集周期，编译和重新编译以及代码优化。

探查器以特定间隔拍摄快照，这些间隔称为刻度线。测量不仅针对代码的工作，还针对引擎本身执行的活动，例如编译，系统库调用，优化和垃圾收集。

启用CPU分析

要在应用程序启动时调用V8 CPU分析，您需要在Node.js运行配置中指定其他设置。

  1. 在主菜单上，选择“运行”|编辑配置 。点击在工具栏上，从列表中选择Node.js.

  2. 从列表中，选择Node.js运行配置以激活CPU分析或创建新配置，如运行和调试Node.js中所述 。

  3. 切换到V8 Profiling选项卡，然后选中Record CPU profiling info复选框。在“ 日志文件夹”字段中，指定将存储记录日志的文件夹的路径，日志文件名为isolate-<session number> 。

收集CPU分析信息

  1. 从主工具栏上的列表中选择运行配置，然后单击或选择运行|从主菜单中运行<配置名称> 。

  2. 执行需要配置的方案时，单击以停止该过程在工具栏上。

分析CPU分析日志

停止应用程序时，WebStorm会自动打开V8性能分析工具窗口，并在其中显示收集的性能分析数据。如果窗口已打开并显示另一个会话的收集数据，WebStorm将打开一个新选项卡。自动打开的选项卡以控制应用程序执行和收集分析数据的运行配置命名。

要打开并分析以前保存的某些性能分析数据，请转到主菜单上的V8性能分析 - 分析V8性能分析日志 ，然后选择相关的isolate-<session number> 。WebStorm创建一个单独的选项卡，其中包含所选日志文件的名称。

根据收集的分析数据，WebStorm构建三个调用树并在单独的窗格中显示每个调用树。使用这些调用树，您可以从两个不同的角度分析应用程序执行：一方面，哪些调用是耗时的（“重”），另一方面，“谁叫谁”。

了解调用树中的度量标准

调用树使用Total和Self度量标准，这些度量标准显示函数中的滴答数或其与总执行时间的比率：

  • Total显示在函数内部花费了多少时间以及它调用的函数。

  • Self表示仅在函数内部花费了多少时间而没有考虑其子节点。

Of Parent度量标准显示函数的纯执行时间与调用它的函数（ Parent ）的执行时间之比。

V8优化器

在某些情况下，V8可以优化您的代码，有关详细信息，请参阅优化V8 ：

  • 函数名称前面的星号（ * ）表示该函数已经过优化。

  • 波浪号（ ~ ）表示该功能可能需要优化但由于某些原因尚未优化。如果代码短时间运行，引擎可以推迟优化或跳过它，但是代码可以重写的位置处的代数可以获得更好的性能。

“ 热门呼叫”窗格按自行度量标准按降序排列已执行活动。对于每个活动，将显示其总计百分比 ， 总百分比和自我百分比指标。对于每个函数调用，WebStorm显示文件的名称，行和定义函数的列。

“ 概述”窗格中的图表显示Self％指标高于1％的通话的自我时间分布。

自下而上的树

如果您遇到繁重的功能并且想要找出它的调用位置，则自下而上的树很有用。

“ 自下而上”窗格还列出了自我指标按降序排序的已执行活动。与“ 热门呼叫”窗格不同，“ 自下而上”窗格仅显示总百分比度量值大于2的活动以及调用它们的函数。

  • 对于每个活动，将显示其在刻度线中的执行时间和Of Parent指标。

内存分析

内存分析使您可以检测内存泄漏和动态内存问题，并找到导致它们的代码片段。

启用内存分析

要在应用程序启动时调用内存快照，您需要在Node.js运行配置中指定其他设置。

  1. 在主菜单上，选择“运行”|编辑配置 。点击在工具栏上，从列表中选择Node.js.

  2. 从列表中，选择Node.js运行配置以激活CPU分析或创建新配置，如运行和调试Node.js中所述 。

  3. 切换到V8 Profiling选项卡，然后选中允许获取堆快照复选框。

收集内存分析信息

  1. 从主工具栏上的列表中选择运行配置，然后单击或选择运行|从主菜单中运行<配置名称> 。

  2. 在应用程序执行期间的任何时候，单击在“ 运行”工具窗口的工具栏上。

  3. 在打开的对话框中，指定快照的名称以及存储它的文件夹的路径。要立即开始分析快照，请选中“ 打开快照”复选框。

分析内存快照

拍摄快照并选择对其进行分析时，WebStorm会打开包含所收集数据的V8堆工具窗口。如果窗口已打开并显示另一个会话的收集数据，WebStorm将打开一个新选项卡。

要打开并分析以前保存的一些内存分析数据，请转到主菜单上的V8性能分析 - 分析V8堆快照 ，然后选择相关的.heapsnapshot文件。WebStorm创建一个单独的选项卡，其中包含所选文件的名称。

工具窗口有三个选项卡，用于显示来自不同视点的收集信息。

  • 在遏制选项卡显示在几个顶级项分组应用程序中的对象：DOMWindow对象 ， 本地浏览器的对象 ，GC和根 ，这是根的垃圾收集器实际使用。有关详细信息，请参阅遏制视图 。

    对于每个对象，选项卡显示其与GC根的距离 ，即对象和GC根之间节点的最短简单路径，对象的浅大小以及对象的保留大小 。除了对象大小的绝对值外，WebStorm还显示了对象占用的内存百分比。

  • “ 最大对象”选项卡显示按保留大小排序的内存最多的对象。在此选项卡中，您可以发现通过在某个全局对象中累积数据而引发的内存泄漏。

  • “ 摘要”选项卡显示应用程序中按类型分组的对象。该选项卡显示每种类型的对象数，它们的大小以及它们占用的内存百分比。该信息可能是存储器状态的线索。

每个选项卡都有一个“ 详细信息”窗格，该窗格显示GC根目录中当前所选对象的路径以及对象的保留器列表，即保留指向所选对象的链接的对象。每个堆快照都有许多“后退”引用和循环，因此每个对象总是有许多保留器。

使用文本标签标记对象

标签可帮助您区分对象并在不丢失上下文的情况下从一个对象移动到另一个对象。

  • 要为对象设置标签，请选择该对象并单击在工具栏上或从上下文菜单中选择标记 。然后在打开的对话框中指定文本标签。

浏览快照

  • 要导航到与对象对应的函数或变量，请选择此对象并单击在工具栏上或从上下文菜单中选择“ 编辑源 ”。如果禁用了按钮和菜单选项，则表示WebStorm未找到与所选对象对应的任何函数或变量。

    如果找到多个函数或变量，WebStorm会在建议列表中显示它们。

  • 为了帮助您调查但从围堵点对象，集中对象之间的联系，WebStorm让您从一个对象跳的最大对象或摘要选项卡或纪录 ，以期遏制该选项卡中的同一个对象。

    为此，选择对象并单击在工具栏上，或从上下文菜单中选择“ 在主树中导航” 。

搜索快照

  1. 在遏制选项卡，单击在工具栏上。

  2. 在打开的V8堆搜索对话框中，指定要搜索的搜索模式和范围。可用范围是：

    • 无处不在：选中此复选框可在所有范围内搜索。选中此复选框后，将禁用所有其他搜索类型。

    • 链接名称：选中此复选框以在调用C ++运行 时时搜索V8创建的对象名称。

      在V8堆工具窗口中，链接名称标有%字符（ %<link name> ）。

    • 类名：选中此复选框可在函数构造函数中进行搜索。

    • 文本字符串：选中此复选框可在对象内容中执行文本搜索。

    • 快照对象ID：选中此复选框可在对象的唯一标识符中进行搜索。在创建对象时， V8会以格式为每个对象分配这样一个唯一标识符，并保留它直到对象被销毁。这意味着您可以在同一会话中拍摄的多个快照中查找和比较相同的对象。

      在V8堆工具窗口中，对象ID用@字符标记（ @<object id> ）。

    • 标记：选中此复选框可通过单击手动搜索您手动设置的标签在遏制选项卡的工具栏上。

搜索结果显示在“ 详细信息”窗格中，在“<搜索模式>”视图的单独出现中 。要使搜索结果按照您指定的搜索范围分组，请按工具栏上的“ 按类别分组”切换按钮。

下次打开对话框时，它将显示上一次搜索的设置。

打字稿

WebStorm支持开发，运行和调试TypeScript源代码。WebStorm可识别*.ts文件，并提供全方位的编码帮助，无需您执行任何其他步骤即可对其进行编辑。TypeScript文件标有图标。

TypeScript感知编码辅助包括完成关键字，标签，变量，参数和函数，错误和语法突出显示，格式化，大量代码检查和快速修复，以及常见和特定于TypeScript的重构。WebStorm还可以动态地验证TypeScript并将其编译为JavaScript。

验证TypeScript并将其编译为JavaScript

默认情况下， Compile All代表整个项目中的编译所有TypeScript文件 。要更改此行为，请在TypeScript页面的“ 编译范围”字段中定义另一个默认范围。

WebStorm主要基于TypeScript语言服务中的数据来验证TypeScript代码，该服务还将TypeScript编译为JavaScript。

在TypeScript工具窗口的“ 错误和编译错误”选项卡中报告所有检测到的语法和编译错误。对于每个错误，WebStorm提供了有关其发生的行号的简要描述和信息。

“ 控制台”选项卡显示自打开工具窗口以来TypeScript语言服务的日志。

    • 捆绑：选择此选项使用typescript是自带WebStorm捆扎包装，而不试图找到另一个。

    • 选择：选择此选项可使用自定义typescript包而不是与WebStorm捆绑的打包。在打开的对话框中选择相关包的路径。

  4. 确保选中TypeScript Language Service复选框。

  5. 使用下面的控件配置TypeScript语言服务的行为。

自动完成代码和路径

代码完成是WebStorm编辑器的基本功能之一。为了向您显示完成建议，WebStorm会使用源代码浏览项目文件，并将其中定义的类，方法，函数和变量添加到特殊的内部索引中。来自JSDoc注释， TypeScript类型定义等的附加信息也可以显着提高完成率。

本节介绍了各种上下文感知代码完成技术，可以加快编码过程。

完成也适用于来自第三方代码的符号。在大多数情况下，您只需要将所需的文件添加到项目中，有关详细信息，请参阅配置JavaScript库 。

基本完成

基本代码完成可帮助您完成可见范围内的类，方法，字段和关键字的名称。当您调用代码完成时，WebStorm会分析上下文并建议可从当前插入符号位置访问的选项（建议还包括实时模板 ）。

调用基本完成

  1. 开始输入名称。默认情况下，WebStorm会在您键入时自动显示代码完成弹出窗口。

    或者，按Ctrl + Space或选择Code |完成|主菜单中的基本菜单。

  2. 要获得更多建议，请再次按Ctrl + Space （或按Ctrl + Alt + Space ）。

  3. 要缩小建议列表，请键入预期名称或关键字的任何部分（即使接受中间某处的字符），也可以在点分隔符后调用代码完成。WebStorm显示包含任何位置的键入字符的建议。

通过以下方式之一接受列表中的建议

  • 按Enter键或双击相关列表项将其插入插入符号的左侧。

  • 按Tab键从插入符号右侧替换字符。

  • 使用Ctrl + Shift + Enter使当前代码构造在语法上正确（平衡括号，添加缺少的大括号和分号，等等）。

  • 使用特定语言和上下文相关键来插入所选的完成建议，请参阅配置完成设置以获取详细信息。

    在此字段中，指定其他Mocha命令行选项 。例如，添加一个--watch标志来打开类似自动测试的跑步者 。因此，当前运行配置中的任何测试都会在更改相关源代码时自动重新启动，而无需单击“ 重新运行”按钮 。
