调查变化

在IntelliJ IDEA中，您可以追溯项目中的所有更改。这有助于您找到任何更改的作者 ， 查看文件版本之间的差异 ，并在必要时安全回滚和撤消更改。

查看项目历史记录

您可以查看对与指定过滤器匹配的项目源所做的所有更改。要查看项目历史记录，请打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 日志”选项卡。它显示了提交给所有分支和远程存储库的所有更改：

在多存储库项目中，左侧的彩色条带指示所选提交所属的根（每个根都标记有自己的颜色）。将鼠标光标悬停在彩色条纹上以调用显示根路径的提示：

搜索项目历史

  • 通过输入完整提交名称或消息或其片段，修订号或正则表达式来搜索提交列表。

  • 按分支或收藏分支 ，用户，日期和文件夹（或多根项目的根目录和文件夹）筛选提交。

  • 单击Go to Hash / Branch / Tag 工具栏上的图标或按Ctrl + F并指定提交哈希， 标记或要跳转到的分支的名称（您将进入该分支中的最新提交）。

  • 单击箭头以跳转到长分支中的下一个提交：

  • 按向左和向右键跳转到父/子提交。如果您在“ 日志”视图中提交了不同的存储库并且多个分支都已混合，则此功能尤其有用。

按Ctrl + L将焦点切换到搜索字段。

要避免来回设置过滤器，请单击在工具栏上打开与您的过滤器匹配的新标签页。

有关项目日志视图的更多详细信息，请参阅日志选项卡 。

单击Windows / Linux上的Ctrl + A或MacOS上的⌘+ A ，根据您在“ 提交详细信息”窗格中单击的位置选择提交消息，作者或标记。

在特定修订版本中查看项目的快照

IntelliJ IDEA允许您在选定的版本中查看项目的状态：

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 日志”选项卡。

  2. 选择一个提交，然后从上下文菜单中选择“ 修订时浏览存储库” 。

将打开“ 存储库”工具窗口，其中包含所选修订版本中项目的快照。

查看文件历史记录

您可以查看对特定文件所做的所有更改，并确定每个修订版中的确切修改内容。

  1. 在任何视图中选择所需文件（在“ 项目”工具窗口中，在编辑器中，在“ 版本控制”工具窗口的“ 本地更改”选项卡中，依此类推）。

  2. 选择Git |从上下文菜单中显示历史记录 。“ 历史记录”选项卡将添加到“ 版本控制”工具窗口，其中显示所选文件的历史记录，并允许您查看和比较其修订版本。

  3. 要确定特定修订中引入了哪些更改，请在列表中选择它并按Ctrl + D或单击工具栏上的按钮。差异查看器将打开，显示此文件修订版中的更改内容。

查看本地版本和已提交版本之间的差异

您可以检查已提交的文件修订版本与其本地版本的不同之处：

  1. 打开“ 版本控制”工具窗口（ Alt + 9 ）并切换到“ 日志”选项卡。

  2. 选择您感兴趣的提交，然后在右侧窗格中选择该文件。

  3. 点击工具栏上的按钮。

查看如何合并更改

IntelliJ IDEA的允许您查看如何变化合并从一个分支到另一个 ，以及如何准确冲突（如果有的话）是解决在合并过程中：

  • 在“ 日志”视图中，选择您感兴趣的合并提交。
    • 如果在合并期间未检测到并解决冲突，IntelliJ IDEA将在“ 更改的文件”窗格中显示相应的消息，并建议您查看源自父母双方的更改：

      从其中一个节点中选择所需文件，然后单击“ 显示差异” 工具栏上的图标或按Ctrl + D.差异查看器将显示双面板差异，允许您将当前版本与所选父项进行比较。

    • 如果在合并期间发生冲突，“ 更改的文件”窗格将显示与冲突合并的文件列表。

      选择所需文件，然后单击“ 显示差异” 工具栏上的图标或按Ctrl + D.差异查看器将显示一个三面板差异，允许您将当前版本与其每个父项进行比较，并查看冲突是如何解决的。

找到代码作者（注释/责备）

您可以通过使用VCS注释找出谁引入了对文件的更改（对应于git-blame ）。带注释的视图显示了每行代码的详细信息::

在当前版本中修改的行的注释用粗体和星号标记。

启用注释

  • 在编辑器或差异查看器中右键单击左侧装订线，然后从上下文菜单中选择“ 注释 ”。

    您可以为Annotate命令指定自定义快捷方式：在Settings / Preferences对话框中（ Ctrl + Alt + S ）选择左侧的Keymap并转到Version Control | Git |注释 。

配置注释中显示的信息量

您可以在注释视图中选择要查看的信息量。

  • 右键单击注释装订线，选择“ 查看”并选择要查看的信息类型，包括发生此更改的版本，日期，不同格式的作者姓名以及提交编号。

    您还可以在“ 颜色”下设置突出显示。

配置注释选项

  • 右键单击注释装订线，然后从上下文菜单中选择“ 选项 ”：

    • 忽略空格 ：空格将被忽略（git blame -w ）。这意味着注释将指向先前有意义的提交。

    • 检测文件中的移动：当提交移动或复制同一文件中的行时，将忽略此类更改（git blame -M ）。这意味着注释将指向先前有意义的提交。

    • 检测跨文件的移动：当提交移动或复制在同一提交中修改的其他文件中的行时，将忽略此类更改（git blame -C ）。这意味着注释将指向先前有意义的提交。

您可以从注释视图跳转：

  • 到日志中的相应提交：使用Select in Git log context-menu命令

  • 到https://github.com上的相应提交：使用Open on GitHub context-menu命令

注释先前的修订版

IntelliJ IDEA不仅可以注释当前文件修订版，还可以注释其以前的修订版。注释装订线的上下文菜单中提供了以下选项：

  • 注释修订 ：如果要在提交特定更改后检查文件的外观，此选项很有用。

  • 注释先前版本 ：如果您发现自己处于特定行中的最后一次更改毫无意义的情况下，此选项很有用，例如，如果所有更改都是代码格式。在这种情况下，您可以检查文件的先前版本是什么样的。

您还可以从“ 历史记录”视图中注释特定文件。在“ 历史记录”选项卡中，选择要查看的文件版本，右键单击相应的行，然后从上下文菜单中选择“ 注释 ”。

测试RESTful Web服务 - 帮助| WebStorm

WebStorm 2019.1帮助

测试RESTful Web服务

介绍

使用WebStorm，您可以测试RESTful Web服务。WebStorm提供了通过编写和运行请求来模拟与WebService交互的工具，就好像您是实际调用该服务的应用程序一样。除此之外，您还可以创建，保存，编辑和删除Cookie，这些Cookie都是通过响应接收并手动创建的。如果尚未达到到期日期 ，则cookie的名称和值将自动包含在与为cookie指定的域和路径匹配的URL地址的每个请求中。

通过HTTP Client捆绑插件支持测试RESTful Web服务，默认情况下启用该插件。如果没有，请在“ 设置”对话框的“ 插件”页面中将其激活。

当您需要撰写并运行对RESTful Web服务的请求时，有两个主要用例：

  • 当您开发和部署RESTful Web服务并希望确保它按预期工作时：它可以按照规范访问，并且可以正确响应。

  • 在开发解决RESTful Web服务的应用程序时。在这种情况下，在开始开发之前调查对服务的访问和所需的输入数据是有帮助的。在开发过程中，您还可以从应用程序外部调用Web服务。这可能有助于在应用程序导致意外输出时找到错误，同时在代码中未检测到逻辑错误，并且您怀疑瓶颈是与Web服务的交互。

测试RESTful Web服务包括以下检查：

  • 根据服务部署端点和方法注释正确构成URL地址。

  • 生成的服务器请求调用相应的方法。

  • 该方法返回可接受的数据。

通过WebStorm，您可以通过编写请求并将请求提交到服务器，查看和分析服务器响应 ，从REST客户端工具窗口运行这些检查。

如有必要，请在“设置”对话框的“ HTTP代理”页面上配置“代理”设置。

编写测试请求并将其提交给Web服务方法

  1. 如果要测试自己的Web服务，请确保已部署并运行。

  2. 选择工具| HTTP客户端|测试RESTful Web服务 。将打开REST客户端专用工具窗口 。

  3. 要让WebStorm生成将在基本身份验证中使用的身份验证标头 ，请单击“ 生成授权标头”按钮，然后在打开的对话框中，指定用于访问目标RESTful Web服务的用户名和密码。基于这些凭据，WebStorm将生成将在基本身份验证中使用的身份验证标头 。欲了解更多信息， 请访问http://www.ics.uci.edu/~fielding/pubs/dissertation/rest_arch_style.htm 。

  4. 从HTTP方法下拉列表中选择测试请求方法 。可用选项包括：

    • 得到

    • POST

    • 放

    • 补丁

    • 删除

    • 头

    • OPTIONS

  5. 提供数据以计算目标方法的URL地址：

    1. 在“ 主机/端口”文本框中，键入部署Web服务的主机的URL地址。

    2. 在“ 路径”字段中，指定要调用的方法的相对路径。

    您可以在主机/端口字段中输入要测试的方法的完整URL地址。无论选择哪种HTTP方法 ，按下时都是如此 WebStorm会将URL地址拆分为主机/端口和方法的路径。提取的相对路径将显示在“ 路径”字段中，提取的参数将添加到“ 请求”选项卡的“ 请求参数”窗格中的列表中。

  6. 在“ 标头数据”窗格中，指定请求标头中包含的技术数据。这些数据通过头字段传递，并定义输入参数（ 接受字段）的格式，响应格式（ 内容类型字段），缓存机制（ 缓存控制字段）等。

    要向列表中添加字段，请单击“ 添加” ，然后在“ 名称”文本框中指定字段名称，并在“ 值”下拉列表中指定字段值。

    字段集及其值应符合Web服务API。换句话说，指定的输入格式应该是Web服务所期望的格式，并且预期的响应格式应该是服务返回的格式。

    对于accept ， content-type和其他一些字段，WebStorm提供了建议值列表。从“ 值”下拉列表中选择相关的格式类型。

  7. 创建一组要传递给目标方法的参数并指定它们的值。根据所选的请求方法，您可以通过两种方式创建参数列表：
    • 对于GET请求，请指定要作为URL内的查询字符串传递的参数。使用“ 请求参数”窗格。默认情况下，窗格显示带有一行的空列表。

      • 要添加参数，请单击“ 添加” ，然后在“ 名称”文本框中指定参数的名称，并在“ 值”下拉列表中指定参数的值 。

      • 要从列表中删除参数，请选择该参数，然后单击“ 删除” 。

远程服务器是在另一台计算机（远程主机）上运行的服务器。可以通过FTP / SFTP / FTPS协议访问服务器上的文件。

要在此设置中配置对服务器的访问，您需要指定以下内容：

  1. 连接设置：服务器主机，端口和用户凭据。

  2. 服务器配置根文件夹和访问它的URL地址。

  3. 项目根文件夹 ，服务器上用于将数据从项目根文件夹复制到的文件夹 ，以及用于访问服务器上复制数据的URL地址之间的对应关系。这种对应关系称为映射 。

指定服务器配置的名称，类型和可见性

  1. 通过执行以下操作之一打开“ 部署”页面：

    • 在设置/首选项（Ctrl + Alt + S），选择下生成，执行，部署 部署 。

    • 选择工具|部署|在主菜单上配置 。

  2. 在列出所有现有服务器配置的左侧窗格中，单击“ 添加” 并根据您将用于与服务器交换数据的协议选择服务器配置类型。

    • FTP：选择此选项可让IntelliJ IDEA通过FTP 文件传输协议访问服务器。

    • SFTP：选择此选项可让IntelliJ IDEA通过SFTP文件传输协议访问服务器。

    • FTPS：选择此选项可让IntelliJ IDEA通过SSL上的FTP文件传输协议（ FTPS扩展）访问服务器。

  3. 在打开的“ 创建新服务器”对话框中，键入要创建的服务器的名称，然后单击“ 确定” 。将关闭“ 创建新服务器”对话框，并返回“ 部署”节点的“ 连接”选项卡。

    单击“ 用作默认值”工具栏按钮要使IntelliJ IDEA在以下情况下以静默方式应用当前配置：

    • 自动上传已更改的文件 。

    • 无需选择目标主机即可手动上载和下载文件。

    • 编辑远程主机上的单个文件 。

如果IntelliJ IDEA无法自动检测配置文件，您可以手动添加构面：

  1. 按Ctrl + Shift + Alt + S打开“项目结构”对话框。

  2. 从左侧列表中，选择模块 。

  3. 选择所需的模块，然后单击中间部分的按钮。

  4. 从列表中选择Spring 。

  5. 添加构面时可能需要设置库。在这种情况下，单击警告消息旁边窗口底部的“ 修复” 。

    如果您已有Spring库，则可以按原样使用它，也可以使用计算机上的JAR文件创建新库。在这种情况下，请选择“ 使用库”选项。

    如果您没有库，请选择下载 。

如果Spring facet配置正确，您将看到标记每个组件，bean等的Spring gutter图标。要配置装订线图标，请按Ctrl + Alt + S并转到编辑器|一般|天沟图标 。

创建应用程序上下文

Spring应用程序上下文是一种在IntelliJ IDEA中对配置文件进行分组的方法。创建上下文时，您可以让IntelliJ IDEA了解此上下文中包含的配置文件之间的关系。

您可以根据需要创建任意数量的应用程序上下文;任何配置文件都可以包含在多个上下文中。

  1. 按Ctrl + Shift + Alt + S并选择Facets 。

  2. 从列表中选择Spring facet，然后单击右侧部分的按钮。

  3. 在“ 新建应用程序上下文”对话框中，输入名称，然后选择要包含在上下文中的文件。

在某些情况下，您不需要配置上下文。例如，Spring MVC Web应用程序对其配置有严格的规定。IntelliJ IDEA中的Spring支持可以推导出它们并为您创建自动检测的应用程序上下文，以及设置Web构面。

配置父上下文

IntelliJ IDEA允许您配置上下文之间的父子关系。来自父上下文的Bean对于子上下文中的bean是可见的，但反之亦然。因此，来自子上下文的bean可以使用父上下文中的配置。

例如，Spring MVC应用程序通常有两个上下文。一个上下文属于Web层bean，另一个上下文用于服务和存储库。在这种情况下，Web层上下文将是子上下文，因为您需要将服务注入控制器，而不是其他方式。

要配置父上下文，请使用“ 新建应用程序上下文”对话框。

“ 多个上下文”面板显示在编辑器顶部，用于包含在两个或多个应用程序上下文中的文件。您可以使用此面板选择另一个活动上下文，例如，如果要使用不同配置运行应用程序，并更改突出显示。

要禁用面板，请单击按钮，并清除“ 显示多个上下文”面板复选框。

自动导入

如果您正在使用尚未导入的类，静态方法或静态字段，IDE将显示一个弹出窗口，建议您添加缺少的导入语句，以便您不必手动添加它。按Alt + Enter接受建议。

代码分析基础知识

PyCharm具有强大，快速和灵活的静态代码分析功能。它检测语言和运行时错误，建议更正和改进。

PyCharm通过对代码应用检查来执行代码分析。代码检查适用于Python和其他支持的语言。

检查不仅检测编译错误，还检测不同的代码低效率。每当你有一些无法访问的代码，未使用的代码，非本地化的字符串，未解决的方法，内存泄漏甚至拼写问题时 - 你会很快找到它。

PyCharm的代码分析是可配置的。您可以启用或禁用每个代码检查并更改其严重性 ，使用自定义检查集创建配置文件 ， 在不同范围内应用不同的检查， 抑制特定代码段中的检查等。

分析可以通过以下几种方式进行：

  • 默认情况下，PyCharm会分析所有打开的文件，并在编辑器中突出显示所有检测到的代码问题。在编辑器的右侧，您可以看到整个文件的分析状态（右上角的图标）。

    严重性级别如下：错误（ ）， 一个警告 （ ），没有发现问题（ ）。

  • 或者，您可以在指定范围的批量模式下运行代码分析，该范围可以与整个项目一样大。

  • 如有必要，您可以在特定范围内应用单个代码检查 。

对于大多数检测到的代码问题，PyCharm提供快速修复建议 。通过按F2 Shift + F2，可以快速查看文件中的错误，方法是从一个突出显示的行导航到另一个。

有关更多信息和过程描述，请参阅配置检查严重性 。

检查概况

当您检查代码时，您可以告诉PyCharm您想要搜索哪些类型的问题并获取报告。这种配置可以保留为检查配置文件 。

检查配置文件定义了要寻找的问题类型，即启用/禁用哪些代码检查以及这些检查的严重性 。配置文件可在“ 检查设置”页面中进行配置。

要设置当前检查配置文件（用于在编辑器中进行动态代码分析的配置文件），请在“ 检查设置”页面中选择它并应用更改。执行代码分析或执行单个检查时 ，可以指定每次运行使用的配置文件。

检查配置文件可适用于整个IDE或特定项目：

  • 项目配置文件可通过VCS共享和访问团队成员。它们存储在项目目录中： <project>/.idea/inspectionProfiles 。

  • IDE配置文件仅供个人使用，并本地存储在USER_HOME/.<PyCharm version>/config/inspection目录下的XML文件中。

PyCharm带有以下预定义的检查配置文件：

  • 默认值 ：此本地（IDE级别）配置文件供个人使用，适用于所有项目，并存储在本地存储在USER_HOME/.<PyCharm version>/config/inspection目录下的Default.xml文件中。

  • 项目默认值 ：创建新项目时，将从模板项目的设置中复制项目默认配置文件。此配置文件是共享的，适用于当前项目。
    创建项目后，对项目默认配置文件的任何修改都将被忽略不计地传递给任何其他项目。
    在“ 模板项目”设置中修改“ 项目默认”配置文件的设置时 ，更改的配置文件将应用于所有新创建的项目，但现有项目不会受到影响，因为它们已拥有此配置文件的副本。
    项目默认配置文件存储在位于<project>/.idea/inspectionProfiles目录中的Project_Default.xml文件中。

可以根据需要拥有尽可能多的检查配置文件。有两种创建新配置文件的方法：您可以添加新配置文件作为项目默认配置文件的副本或复制当前选定的配置文件。新创建的配置文件存储在XML文件中，具体取决于基本配置文件的类型 。

只要完成并应用对配置文件的某些更改，就会显示表示检查配置文件的<profile_name>.xml文件。这些文件仅存储与默认配置文件的差异。

有关详细信息 ，请参阅自定义配置文件部分。

检查严重程度

检查严重性表示检测到的代码问题对项目的影响程度，并确定如何在编辑器中突出显示检测到的问题。默认情况下，每次检查都具有以下严重性级别之一：

  • 服务器问题

  • 错字

  • 没有突出显示，只有修复

  • 弱警告

  • 警告

  • 错误

您可以增加或减少每次检查的严重性级别。例如，您可以强制PyCharm显示一些警告，如错误或弱警告。以类似的方式，最初被认为是弱警告的内容可以显示为警告，错误或仅作为参考信息。

您还可以配置用于突出显示每个严重性级别的颜色和字体样式。此外，您可以创建自定义严重性级别并将其设置为特定检查。

如有必要，您可以为不同范围内的相同检查设置不同的严重性级别。

上述检查的所有修改都保存在检查设置中当前选择的检查配置文件中，并在使用此配置文件时应用。

检查范围

默认情况下，所有启用的代码检查都适用于所有项目文件。如有必要，您可以为不同的范围单独配置每个代码检查（ 启用或禁用 ， 更改其严重性级别和选项）。与任何其他检查设置一样，此类配置将作为特定配置文件的一部分进行保存和应用。

当检查具有与不同范围相关联的不同配置时，可能存在复杂的情况。当在属于某些或所有这些范围的文件中执行此类检查时，将应用优先级最高的范围特定配置的设置。优先级由检查设置中检查范围特定配置的相对位置定义：最高配置具有最高优先级。Everywhere else配置始终具有最低优先级。

有关更多信息和过程描述，请参阅配置不同范围的检查 。

当您运行push ，如果远程存储库包含您缺少的更改并且您要使用存储库的本地副本覆盖，Git将拒绝完成操作。通常，在使用更改进行更新之前，需要执行pull以与远程同步。

--force push命令禁用此检查，并允许您覆盖远程存储库，从而删除其历史记录并导致数据丢失。

当您可能仍需要执行--force push时，可能会出现这样的情况：当您重新--force push推送的分支然后想要将其推送到远程服务器时。在这种情况下，当您尝试推送时，Git将拒绝您的更改，因为远程引用不是本地引用的祖先。如果你在这种情况下执行pull ，你将得到两个你需要合并的分支副本。

如果您决定强制推送重新分支的分支并且您正在团队中工作，请确保：

  • 没有人拉你的分支并对它进行了一些局部更改

    字段集及其值应符合Web服务API。换句话说，指定的输入格式应该是Web服务所期望的格式，并且预期的响应格式应该是服务返回的格式。

    对于accept ， content-type和其他一些字段，PyCharm提供了一个建议值列表。从“ 值”下拉列表中选择相关的格式类型。

  7. 创建一组要传递给目标方法的参数并指定它们的值。根据所选的请求方法，您可以通过两种方式创建参数列表：
    • 对于GET请求，请指定要作为URL内的查询字符串传递的参数。使用“ 请求参数”窗格。默认情况下，窗格显示带有一行的空列表。

      • 要添加参数，请单击“ 添加” ，然后在“ 名称”文本框中指定参数的名称，并在“ 值”下拉列表中指定参数的值 。

      • 要从列表中删除参数，请选择该参数，然后单击“ 删除” 。

      • 要禁止发送指定的查询字符串参数并禁用“ 请求参数”窗格中的控件， 请按“不发送任何内容”切换按钮 。

    • 要将参数传递给请求消息正文中的目标方法，请使用“ 请求 正文”窗格或将它们插入到本地文件的请求中。选择GET ， DELETE ， HEAD或OPTIONS请求方法时，将禁用“ 请求正文”窗格。

      • 要显式指定参数，请选择“ 文本”选项，然后在文本框中键入参数和值。

      • 要从文本文件中插入参数，请选择“ 文件内容”选项，然后在“ 要发送的文件”字段中指定文件位置。

      • 要在请求中转换和发送二进制文件，请选择“ 文件上载（multipart / form-data）”选项，并在“ 要发送的文件”字段中指定文件位置。

    在任何一种情况下，参数集及其类型都应符合Web服务API，特别是它们应与目标方法的输入参数完全相同。
  8. 要向服务器提交请求 ，请单击“ 提交请求”按钮 。

    请注意，服务器可能缺少证书，或者不受信任。

    如果服务器不受信任，PyCharm会显示一个建议接受服务器的对话框，或拒绝它。如果您将服务器接受为受信任，则PyCharm会将其证书写入信任存储区。在下一次连接到服务器时，将不会显示此对话框。

将IDE连接到跟踪器

    通常，您必须指定服务器URL和连接凭据。

    选择“ 共享URL”选项以允许您的团队中的其他成员访问服务器。

    对于某些跟踪器（例如，对于不支持开箱即用的跟踪器），您还必须配置服务器参数。指定用于访问任务的URL和请求类型，选择跟踪服务器响应的格式 - 用于XPath的XML，用于JSONPath的 JSON或用于正则表达式的文本。您还可以使用选择器表来指定要从服务器获取的任务的详细信息。例如，这可以是创建任务的日期或其URL。请注意，您可以在此窗口中使用代码完成。

  2. 在“ 加载上下文”弹出窗口中，单击向右箭头并选择“ 删除” 。

  • 在编辑器中打开文件，然后选择Go To |测试或转到|从上下文菜单中测试主题 ，或者只需按Ctrl + Shift + T.

  • 要从测试结果跳转到测试，请在“ 测试运行器”选项卡中选择测试名称，然后从上下文菜单中选择“ 跳转到源 ”。测试文件在编辑器中打开，光标位于测试定义中。

  • 对于失败的测试，WebStorm会从堆栈跟踪中引导您进入测试中的故障线。如果确切的行不在堆栈跟踪中，您将进入测试定义。

测试文件应遵循流行的命名约定，例如，具有.test. ， .spec.或_spec.后缀，应位于源文件旁边或test文件夹中。

调试测试

使用WebStorm，您可以直接从编辑器快速开始调试单个Karma测试，或创建运行/调试配置来调试部分或全部测试。

从编辑器开始调试单个测试

  • 点击要么在左侧排水沟中，从列表中选择Debug <test_name> 。

通过运行/调试配置启动测试调试

  1. 如上所述创建Karma运行/调试配置。

  2. 从主工具栏上的列表中选择Karma运行/调试配置，然后单击在列表的右侧。

  3. 在打开的调试工具窗口中，像往常一样继续： 逐步执行测试 ， 停止并恢复测试执行， 暂停时检查测试 ，等等。

监控代码覆盖率

使用WebStorm，您还可以监控Karma测试涵盖的代码量。WebStorm在专用工具窗口中显示此统计信息，并在编辑器中直观地标记已覆盖和未覆盖的行。要监控覆盖范围，您需要安装karma-coverage软件包并更新karma.conf.js 。

安装业力报道

  • 在嵌入式终端 （ Alt + F12 ）中，键入：

    npm install --save-dev karma-coverage

将karma-coverage定义添加到配置文件中

  1. 在编辑器中打开karma.conf.js 。

  2. 找到reporters定义并将coverage添加到以下格式的值列表中：

    记者 ： [ '进展' ， '覆盖' ]

  3. 添加preprocessors定义并以以下格式指定coverage范围：

    预处理器 ： { '** / * .js' ： [ 'coverage' ]}

启动覆盖范围的测试

  1. 如上所述创建Karma运行/调试配置。

  2. 从主工具栏上的列表中选择Karma运行/调试配置，然后单击在列表的右侧。

    或者，使用编辑器中的测试图标快速运行特定套件或覆盖测试。

  3. 在Coverage工具窗口中监视代码覆盖率。

    每次运行Karma测试时，都会在磁盘上生成覆盖率报告。可以在配置文件中配置覆盖率报告的格式，例如：

    // karma.conf.js 模块 。 exports = function （ config ） { config 。 集 （{ ... //可选，配置报告者 coverageReporter： { type ： ' html ' ， dir ： ' coverage / ' } ... }）;};

    可接受以下type值：

    • html生成一堆带有注释源代码的HTML文件。

    • lcovonly生成一个lcov.info文件。

    • lcov生成HTML + .lcov文件。默认情况下应用此格式。

    • cobertura生成一个cobertura-coverage.xml文件，以便于Hudson集成。

    • text-summary生成一个紧凑的文本摘要，通常是控制台。

    • text生成详细的文本表，其中包含所有文件的覆盖范围。
