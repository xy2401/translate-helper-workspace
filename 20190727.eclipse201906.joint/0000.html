<？xml version =“1.0”encoding =“iso-8859-1”？>

<meta http-equiv =“Content-Type”content =“text / html; charset = utf-8”/>

在导入WAR文件之前，应首先确定WAR文件是否包含所需的Java™源文件。将WAR文件导入现有Web项目时，导入的Web部署描述符文件不会被导入的WAR文件中包含的Web部署描述符文件更改或覆盖，具体取决于您对提供的提示的响应。在这两种情况下，这个动作并不代表两套部署描述符的合并。

  2. 在Import对话框中，选择WAR file ，然后单击Next 。

  4. 该向导假定您要创建一个与WAR文件同名的新Web项目。如果接受此选择，则将使用与WAR文件指定的相同servlet版本并在同一位置创建项目。如果要覆盖这些设置，可以单击“ 新建”并在“动态Web项目”向导中指定新设置。

Java EE模块之间的周期性依赖关系

当从Workbench外部导入项目时，企业应用程序中两个或多个模块之间的循环依赖关系最常发生。

当企业应用程序中的两个或多个模块之间存在循环时，Java™构建器无法准确计算项目的构建顺序。完全构建在这些条件下失败，或者需要多次调用。

因此，最佳做法是对您的项目或模块进行组件化。这允许您将模块依赖项用作树而不是循环图。这种做法具有产生更好的分解和分层应用程序的额外好处。

导入EAR后更正周期性依赖关系

导入EAR后，您可以解决周期性依赖关系。

当从工作台外部导入项目时，企业应用程序中两个或多个模块之间的循环依赖关系最常发生。当企业应用程序中的两个或多个模块之间存在循环时，Java™构建器无法准确计算项目的构建顺序。完全构建在这些条件下失败，或者需要多次调用。

因此，最佳做法是将项目或模块组织到组件中。这允许您的模块依赖项充当树而不是循环图。这种做法具有产生更好的分解和分层应用程序的额外好处。

要重新组织项目以更正周期性依赖项，请完成以下步骤：

  1. 识别具有循环依赖性的JAR文件中的所有类，并将这些类移动到公共Java项目或JAR文件中。
  2. 使用企业应用程序编辑器将实用程序JAR文件映射到公共项目。
  3. 对于应用程序中JAR的每个模块，使用JAR依赖关系编辑器或属性页面仅将依赖关系设置为真正需要的JAR文件。

JavaServer Pages（JSP）技术

JavaServer Pages（JSP）技术

JavaServer Pages技术使您能够生成动态Web内容，例如HTML，DHTML，XHTML和XML文件，以包含在Web应用程序中。JSP文件是实现服务器端动态页面内容的一种方法。JSP文件允许Web服务器（如Apache Tomcat）在将HTML页面发送到请求的浏览器之前动态地将内容添加到HTML页面。

将JSP文件部署到提供servlet引擎的Web服务器时，会将其预处理到在Web服务器上运行的servlet中。这与在浏览器中运行的客户端JavaScript™（在<SCRIPT>标记内）形成对比。JSP页面非常适合更适合在服务器上执行的任务，例如访问数据库或调用Enterprise Java™bean。

您可以在HTML编辑器中创建和编辑JSP文件，方法是使用HTML，JSP标记或JavaScript（包括scriptlet标记内的Java源代码）添加自己的文本和图像。通常，JSP文件的文件扩展名为.jsp。此外，JSP规范建议JSP片段文件应具有文件扩展名.jspf。如果未遵循此约定，则JSP验证程序会将JSP片段视为常规独立JSP文件，并且可能会报告编译错误。

Sun Microsystems JSP 1.2规范提供了创建自定义JSP标记的功能。自定义标签简化了复杂的操作，使开发人员能够更好地控制页面内容自定义标签被收集到库（taglib）中。标记库描述符文件（taglib.tld）是一个XML文档，它提供有关标记库的信息，包括taglib短名称，库描述和标记描述。有关更多详细信息，请参阅Sun Microsystems JSP 1.2规范 。

要使用JSP 1.2自定义标记库，可以将标记库.tld和.jar文件导入到项目中以使用它们，或将它们关联为Web库项目。您还可以使用URI引用TLD文件。

Servlet的

Servlet是服务器端Java™程序，它使用Sun Microsystems Java Servlet API及其相关的类和方法，如Sun Microsystems Java Servlet 2.3规范中所定义。这些Java程序通过生成动态内容和响应Web客户端请求来扩展Web服务器的功能。当浏览器向服务器发送请求时，服务器可以将请求信息发送到servlet，以便servlet可以构造发送回浏览器的响应。

就像applet在Web浏览器上运行并扩展浏览器的功能一样，servlet在支持Java的Web服务器上运行并扩展服务器的功能。由于其灵活性和可伸缩性，servlet通常用于使企业能够将数据库连接到Web。

尽管servlet可以是一个完全独立的程序，但您可以将应用程序开发分为两部分：
  • 业务逻辑（内容生成），它控制输入，处理和输出之间的关系
  • 表示逻辑（内容表示或图形设计规则），用于确定如何向用户呈现信息
使用此范例，您可以选择由Java bean处理业务逻辑，由JavaServer Pages（JSP）或HTML文件处理的表示逻辑以及由servlet处理的HTTP协议。
注意： JSP文件可用于管理Web应用程序的表示和业务逻辑。JSP文件使用结构化标记进行表示，并在运行时提供servlet模型行为。

您可以开发，调试和部署servlet，在servlet对象中设置断点，并逐步执行代码以进行动态折叠到正在运行的服务器上正在运行的servlet中的更改，而无需每次都重新启动。

有关servlet的更多信息，请参阅java.sun.com/products/servlet/download.html上的Sun Microsystems Java Servlet 2.3规范 。

注释是一个修饰符或元数据标记，它为Java类，接口，构造函数，方法，字段，参数和局部变量提供附加数据。注释替换了样板代码，某些应用程序所需的通用代码。例如，注释可以替换Web服务所需的配对接口和实现。注释还可以替换程序所需的其他文件，这些文件是单独维护的。通过使用注释，不再需要此单独的文件。例如，注释可以取代对JavaBeans™单独维护的部署描述符的需求。

注释
  • 为大多数目的替换描述符
  • 不需要标记接口（如java.rmi。远程）
  • 允许应用程序设置在它们影响的组件中可见
Java EE 5为以下任务提供注释，其中包括：
  • 开发Enterprise JavaBean应用程序
  • 定义和使用Web服务
  • 将Java技术类映射到XML
  • 将Java技术类映射到数据库
  • 将方法映射到操作
  • 指定外部依赖项
  • 指定部署信息，包括安全属性
Java EE 5定义了许多可以注入源代码的注释。要声明注释，只需在关键字前面加上“at”符号（@）即可。
包com.ibm.counter; import javax.ejb。无状态; @Stateless公共类CounterBean {}

J2EE开发工具的局限性

J2EE开发工具的局限性

本主题概述了当前已知的J2EE工具限制和限制。

企业应用程序中的JAR URI不支持的空间

企业应用程序中的模块或实用程序JAR文件的URI不支持空格。JAR文件或模块中MANIFEST.MF文件的“Class-Path：”属性是企业应用程序中以空格分隔的相对路径列表。如果引用的JAR文件的URI包含空格，则JAR文件将无法引用EAR中的另一个JAR文件。

企业应用程序项目名称不应包含DBCS字符

创建企业应用程序项目时，建议您不要为其指定包含双字节字符集（DBCS）字符的名称。

删除对Utility JAR文件的依赖关系时，Java™构建路径更新

删除对Utility JAR的依赖关系时，仅当依赖JAR仍由EAR项目引用时，才会从Java构建路径中删除相应的Java项目。例如，假设您创建了一个J2EE 1.3 Web项目和EAR以及JUnit Java Example项目。接下来，将JUnit项目添加为EAR中的Utility JAR，然后将JUnit添加为Web项目的Java JAR依赖项。如果您希望删除JUnit和Web项目之间的依赖关系，请首先从Web项目中删除Java JAR依赖项，然后从EAR中删除Utility JAR。按照此顺序确保此操作正常。

Java JAR Dependencies页面无法更新Java构建路径

Java JAR依赖项页面未与项目属性对话框中的Java构建路径页面同步。因此，在一个中应用的更改可能不会在同一对话会话中反映在另一个中。还有一些情况，当单击“ 确定”按钮或在“ 确定”按钮之前单击“ 应用”按钮时，在页面之间来回切换将导致从一个更新以从另一个更新取消更新。通常，这看起来好像添加了JAR依赖项，但项目未添加到Java构建路径中。解决方法是重新打开属性对话框，切换到JAR依赖关系页面，清除并重新选择依赖的JAR文件，然后单击“ 确定” 。

为新J2EE项目使用非默认项目位置时出现“项目描述无效”错误

创建新的J2EE项目（包括Java，企业应用程序，动态Web，EJB，应用程序客户端和连接器项目）时，不能使用工作台中另一个项目已使用的项目位置。如果选择另一个项目使用的项目位置，向导将显示“无效的项目描述”错误对话框或消息。如果在收到此消息后，通过单击“浏览”按钮选择有效的项目位置，项目创建仍将无法完成。解决方法是单击“取消”并重新打开项目创建向导。

定义自己的注释

使用@Interface批注定义自己的批注定义：
  • 注释定义类似于接口定义
  • 注释方法声明既没有参数也没有throws子句，并返回以下元素之一：
    • 原语
    • 串
    • 类
    • 枚举
    • 以上类型的数组
  • 方法可能具有默认值
public @interface CreatedBy {String name（）;字符串日期（）; boolean contractor（）默认为false; }
@CreatedBy（name =“Mary Smith”，date =“02/02/2008”）;公共类MyClass {....}

  • @目标
    • 限制注释的使用
    • 单个参数必须来自Enum ElementType
      • {TYPE，FIELD，METHOD，PARAMETER，CONSTRUCTOR，LOCAL_VARIABLE，ANNOTATION_TYPE}
  • @保留
    • 指示将保留注释信息的位置
    • 单个参数必须来自Enum RetentionPolicy
      • {SOURCE，CLASS，RUNTIME}
  • @Documented
    • 标记应该包含在Javadoc中的注释
  • @遗传
    • 类型注释的标记，由子类型继承

    • 应用于一种方法
    • 指示在外部使用方法时编译器应生成警告
  • @SuppressWarnings

静态Web项目

如果要创建不包含任何动态内容的基于内容的Web应用程序（例如servlet，JSP文件，过滤器和关联的元数据），则可能更喜欢创建静态Web项目，而不是动态Web项目 。

静态Web项目具有以下特征：
  • 所有可发布资源的Web内容文件夹（称为WebContent），您可以从项目的弹出菜单中更改此文件夹的名称。
  • 主题文件夹，用于存储级联样式表和其他与样式相关的对象的建议目录。
  • 能够在Web内容文件夹之外定义文件夹，用于存储中间文件，例如MIF文件
  • 从静态Web项目到动态Web项目的转换路径。如果您决定转换项目，它将是一个完全有效的动态Web项目。
此外，您的项目仍将具有以下功能（静态和动态Web项目共有）：
  • HTML语法验证
  • 一个破碎的链接修复向导
  • 网站导航工具
  • 一种新的服务器类型，静态Web服务器，可以轻松发布静态Web项目

发布静态Web项目的文件夹是可修改的，因此当您为静态项目设置发布“root”值（称为上下文根 ）（例如/ web1 ）时，Web内容文件夹中的所有内容都将被发布到Web服务器的doc根目录下的web1文件夹。这使您可以在Web服务器上将Web资源分组到与工作台中的Web项目对应的文件夹中。当以这种方式定义的项目准备好生产时，您可以通过将值更改为/来将特定项目直接发布到doc根目录，并且所有发布，链接修复和浏览都将自动更新。

Java EE 5：概述

使用Java™Platform，Enterprise Edition（Java EE）体系结构，您可以构建分布式Web和企业应用程序。此体系结构可帮助您专注于演示和应用程序问题，而不是系统问题。

  • 缩短开发时间

  • 带注释的内联配置，使部署描述符现在可选

  • Java持久性API（JPA）允许在没有显式SQL或JDBC的情况下进行数据管理
  • 将普通旧Java对象（POJO）用于Enterprise JavaBeans和Web服务

过滤器向导可帮助您创建Java™servlet过滤器，方法是引导您完成创建过程，并为您提供可以使用的输出文件，或者您可以修改以用于Web应用程序的输出文件。过滤器可以在符合Java EE的Web服务器上运行。

  1. 在Java EE透视图中，在Project Explorer视图中展开动态项目 。
  2. 右键单击Filter图标，然后从弹出菜单中选择New > Filter 。 出现“ 创建过滤器”向导。
  3. 按照项目向导提示进行操作。

一般信息

修饰符
Sun Microsystems Java Servlet 2.3规范声明Servlet类必须是公共的而不是抽象的 。因此，您无法更改这些修饰符。唯一可用于更改的是最终修饰符。
接口
过滤器类必须实现一个强制接口： javax.servlet。过滤 。该接口由javax.servlet包提供，用于表示过滤器的生命周期。这个接口有三个方法：init，doFilter和destroy。
  • init方法由servlet容器调用一次，当它完成实例化过滤器。
  • doFilter方法是执行过滤的地方，每次用户请求过滤器映射到的资源（如servlet）时都会调用该方法。
  • servlet容器调用destroy方法告诉过滤器它将被取消服务。
虽然，那是javax.servlet。默认情况下，在“ 接口”字段中提供过滤器接口，您可以使用“ 添加”按钮添加要实现的其他接口。
界面选择对话框
如果选择将接口添加到过滤器，则会出现此对话框。在键入要添加的接口的名称时，“ 匹配项目”列表框中列出的可用接口列表会动态更新，以仅显示与该模式匹配的接口。选择一个界面以查看限定符，然后在完成后单击“ 确定” 。
过滤映射
要使过滤器拦截对servlet的请求，您必须：
  1. 在部署描述符中使用<filter>元素声明过滤器
  2. 使用<filter-mapping>元素将过滤器映射到servlet。
有时您希望过滤器可以处理多个servlet。您可以通过将过滤器映射到URL模式来执行此操作，以便过滤与该URL模式匹配的任何请求。如果您使用“ 过滤器映射”字段，则会自动进行所有这些操作。
方法存根
因为每个过滤器都必须实现javax.servlet。过滤器接口，始终选中“ 继承的抽象方法”选项，并且无法更改。

调整验证器

调整验证器

验证器是否验证特定资源取决于该验证器的适当过滤器。

首次开发验证器时，验证器的实现者会定义一组默认的过滤器。这些过滤器可能基于：
  • 文件扩展名
  • 文件名文件夹
  • 项目性质
  • 项目方面
  • 内容类型
通过“验证过滤器”对话框，您可以进一步调整这些设置。通常，您只需保留默认值，但是您可能需要调整验证的两个原因是：
  • 性能：如果您拥有非常大的工作空间，则可以减少验证量。
  • 非标准约定：如果使用非标准命名约定（例如，将XML存储在扩展名为.acme-xml的文件中），您仍可以启用相应的验证程序来对这些文件运行。
您可以通过单击窗口 > 首选项 > 验证 ，然后单击每个验证器旁边的设置来访问此对话框。

过滤器存储在组中。有两种类型的组：包括组和排除组。您可以拥有任意数量的Include组。Include组内的过滤器会导致资源得到验证。如果任何规则匹配，则整个组匹配。在组内部，过滤器规则一起进行OR运算。但是，个别包含组是AND并在一起。您可以拥有一个排除组。如果其任何过滤规则匹配，则排除该资源。排除优先于包含。

这个假设的例子说明了这些规则：

  • 如果资源位于已禁用的文件夹中，则会将其排除，因为排除优先于其他所有内容。
  • 如果资源没有JSP源内容类型，并且它没有JSP片段源内容类型，并且它没有.jsp或.jspf的文件扩展名，那么它将被排除，因为它没有任何规则第一组匹配。
  • 如果项目没有模块核心性质，那么它将被排除，因为第二组中的单个规则不匹配。
  • 否则，该特定验证器将验证资源。
要将规则添加到组，请选择左侧的组，然后单击“ 添加规则” 。

项目方面

项目方面

Facets定义Java EE项目的特征和要求，并用作运行时配置的一部分。

向项目添加构面时，该项目配置为执行特定任务，满足特定要求或具有某些特征。例如，EAR facet通过添加部署描述符和设置项目的类路径来设置项目以充当企业应用程序。

您只能将facet添加到Java EE项目和基于J2EE项目的其他类型的项目，例如企业应用程序项目，动态Web项目和EJB项目。例如，您无法将facet添加到Java™项目或插件项目中。通常，启用构面的项目在创建时至少有一个构面，允许您在必要时添加更多构面。例如，一个新的EJB项目具有EJB Module facet。然后，您可以像EJBDoclet（XDoclet）构面一样向此项目添加其他构面。要向项目添加构面 ，请参阅将构面添加到Java EE项目 。

一些方面需要其他方面作为先决条件。其他方面不能在同一个项目中。例如，您无法将动态Web模块构面添加到EJB项目，因为EJB项目已具有EJB Module构面。某些方面可以从项目中删除，而其他方面则不能。

Facets也有版本号。只要您保持在构面的要求范围内，您就可以更改构面的版本号。要更改构面的版本号，请参阅更改构面的版本 。

Java™2平台企业版（J2EE）为开发多层企业应用程序提供了标准。

当今的经济和技术加剧了对更快，更高效和更大规模的信息管理解决方案的需求。J2EE规范通过提供可提高开发效率的编程模型，标准化托管企业应用程序的平台，并通过广泛的测试套件确保开发应用程序的可移植性来满足这些挑战。

J2EE体系结构支持基于组件的多层企业应用程序开发。J2EE应用程序系统通常包括以下层：

客户层
在客户端层中，Web组件（如Servlet和JavaServer Pages（JSP））或独立Java应用程序为中间层提供动态接口。
中间层
在服务器层或中间层，企业bean和Web服务封装应用程序的可重用，可分发的业务逻辑。这些服务器层组件包含在J2EE应用服务器上，该服务器为这些组件提供执行操作和存储数据的平台。
企业数据层
在数据层中，企业的数据通常存储在关系数据库中并保存。

J2EE应用程序由组件，容器和服务组成。组件是应用程序级组件。Web组件（如Servlet和JSP）为来自Web页面的请求提供动态响应。EJB组件包含企业应用程序的服务器端业务逻辑。Web和EJB组件容器承载支持Web和EJB模块的服务。

有关J2EE体系结构及其隐式技术的更多信息，请下载并阅读J2EE 1.4规范 。

Web服务描述语言（WSDL）参考

Web服务描述语言（WSDL）参考

WSDL文档将服务定义为网络端点或端口的集合。在WSDL中，端点和消息的抽象定义与其具体的网络部署或数据格式绑定分开。这允许重用抽象定义：消息，它们是被交换数据的抽象描述，以及作为抽象操作集合的端口类型。

特定端口类型的具体协议和数据格式规范构成可重用绑定。通过将网络地址与可重用绑定相关联来定义端口，并且端口集合定义服务。因此，WSDL文档在网络服务的定义中使用以下元素：

  • 类型 ：使用某种类型系统（例如XSD）的数据类型定义的容器。
  • 消息 ：正在传递的数据的抽象类型定义。
  • 操作 ：服务支持的操作的抽象描述。
  • 端口类型 ：一个或多个端点支持的一组抽象操作。
  • 绑定 ：特定端口类型的具体协议和数据格式规范。绑定通常是SOAP，使用的编码和数据格式规则（也称为样式）通常是文字（包括文档/文字，有时是rpc / literal）。
  • 端口 ：定义为绑定和网络地址组合的单个端点。
  • 服务 ：相关端点的集合。

有关WSDL的更多信息，请参阅

www.w3.org/TR/wsdl

各种WSDL元素可以以多种方式构建。它可以位于单个WSDL文件中，也可以位于多个WSDL文件中。

应用客户端项目

应用程序客户端项目包含在联网客户端系统上运行的程序，因此项目可以从服务器的工具中受益。

应用程序客户端项目包含应用程序客户端模块所需的资应用程序客户端模块用于包含连接到并使用服务器中定义的Java EE资源的全功能客户端Java™应用程序（非基于Web）。将客户端代码放在应用程序客户端模块而不是简单的JAR文件中时，应用程序客户端可以从服务器的资源中受益（它不需要重新指定Java EE和服务器JAR文件的类路径）以及更简单的JNDI查找（客户端容器填充初始上下文和其他参数）。应用程序客户端项目允许您像在Java项目中创建独立Java应用程序一样工作。

应用程序客户端项目使您可以执行以下操作：

  • 开发实现客户端模块的Java类
  • 设置应用程序客户端部署描述符
  • 测试应用程序客户端

与Java项目一样，应用程序客户端项目包含应用程序客户端所需的资源，包括Java类文件。创建新的应用程序客户端项目时，将为Java开发设置环境。Java构建器与项目关联，因此Java源可以在更新时进行增量编译。应用程序客户机项目包含有关类型层次结构和Java元素的信息。在进行更改时，此信息将保持最新，并且Java构建器将在更新资源时逐步编译这些项目中的资源。

在工作台中，应用程序客户端项目始终由企业应用程序（EAR）项目引用。创建应用程序客户机项目时，指定应用程序客户机项目所属的企业应用程序项目。模块元素会自动添加到EAR项目的application.xml部署描述符中。

应用程序客户端项目部署为JAR文件。此应用程序客户机JAR文件包含应用程序所需的资源，包括Java类文件，部署描述符信息以及任何元数据扩展和绑定文件。

应用程序客户端项目通常在连接到Java EE（EJB）服务器的联网客户端系统上运行。应用程序客户机的入口点是Java主类，它只是一个包含静态main方法的Java类。该类在客户端模块的清单文件中声明。

Java EE应用程序客户机容器提供对Java EE服务（JNDI命名服务，部署服务，事务服务和安全服务）和通信API（Internet协议，远程方法调用协议，对象管理组协议，消息传递协议和数据）的访问格式）。

默认情况下，应用程序客户端项目包含一个名为appClientModule的文件夹，其中包含Java源代码和已编译的.class文件，以及META-INF子文件夹中的所有元数据文件。

使用调试器可以检测和诊断应用程序中的错误。它允许您通过设置断点，挂起线程，单步执行代码以及检查变量的内容来控制程序的执行。您可以在服务器上调试servlet，而不会丢失应用程序的状态。

  1. 在Project Explorer视图中，打开您的servlet。 在Web项目中，servlet位于JavaSource文件夹中。servlet在编辑器中打开。

  3. 确保项目与服务器上运行相关联。
  4. 选择servlet，从Project Explorer视图的servlet上下文菜单中，单击Debug As> Debug on Server 。 打开“服务器”视图以查看服务器是否已启动。

  9. 如果启用了自动类重新加载（默认设置），请单击Web浏览器中的“ 刷新”图标以更新更改。 您不会丢失应用程序的状态。
  10. 如果未启用自动类重新加载，则会丢失应用程序的状态。 要更新更改，请重新启动服务器。

结果是服务器识别您的更改。

JAX-RPC

JAX-RPC

JAX-RPC代表基于XML的RPC的Java™API，也称为JSR 101。它是一个描述Java应用程序编程接口（API）的规范，用于构建使用远程过程调用（RPC）和XML的Web服务和Web服务客户端。它将Java到WSDL和WSDL标准化为Java映射，并提供用于在Java平台上开发Web服务和Web服务客户端的核心API。RPC机制通常用于分布式客户端/服务器模型，使客户端能够在其他系统上执行过程。

当前版本的JAX-RPC需要SOAP over HTTP支持以实现互操作性。SOAP规范定义了在RPC编程模型中交换信息的消息结构，编码规则和约定。这些调用和响应通过HTTP作为SOAP消息传输。在此版本中，JAX-RPC支持SOAP 1.1和HTTP 1.1。有关SOAP的更多信息，请参阅SOAP 。

虽然底层运行时机制（例如，Java到WSDL序列化，WSDL到Java反序列化，协议和传输）非常复杂，但API隐藏了应用程序开发人员的这种复杂性。在服务器端，开发人员可以提供服务端点接口（SEI）;或者，可以使用Web服务wizrds创建SEI。SEI是用Java编程语言编写的接口，它指定Web服务的远程过程。开发人员还以一个或多个Java类的形式提供Web服务的实现，这些Java类实现与SEI上的签名相同的签名方法。客户端程序也很容易编码。客户端创建代理，表示服务实现和SEI的本地对象，然后只调用代理上的方法。

JAX-RPC具有高度的互操作性：JAX-RPC客户端可以访问未在Java平台上运行的Web服务，反之亦然。这种灵活性是可能的，因为JAX-RPC使用SOAP over HTTP和Web服务描述语言（WSDL）。JAX-RPC旨在支持WS-I。

有关JAX-RPC的更多信息，请参阅官方JSR 101规范 。

Web服务描述语言（WSDL）

Web服务描述语言（WSDL）

Web服务描述语言（WSDL）是用于描述基于XML的联网服务的标准规范。它为服务提供者提供了一种简单的方法来描述对其系统的基本请求格式，而不管底层的运行时实现如何。

WSDL定义了一种XML格式，用于将网络服务描述为一组端点 ，这些端点对包含面向文档或面向过程的信息的消息进行操作。首先抽象地描述操作和消息，然后将其绑定到具体的网络协议和消息格式以便定义端点。相关的具体端点组合成抽象端点（服务）。无论使用哪种消息格式或网络协议进行通信，WSDL都是可扩展的，以允许描述端点及其消息。这意味着使用XML模式抽象地定义接口，然后将其绑定到适合协议的具体表示。

WSDL允许服务提供者指定Web服务的以下特征：

  • Web服务的名称和寻址信息
  • 访问Web服务的公共操作时要使用的协议和编码样式
  • 包含Web服务接口的操作，参数和数据类型等类型信息

WSDL文档允许开发人员将其应用程序公开为Internet上的网络可访问服务。通过UDDI和WSIL，其他应用程序可以发现WSDL文档并与它们绑定以执行事务或执行其他业务流程。

该开发平台支持并鼓励开发和使用符合WS-I的WSDL。业务服务提供商可以将Java™bean和企业bean部署为Web服务，并生成描述该服务的WSDL文档。它们还可以从现有的WSDL文件生成Java和企业bean骨架。业务服务客户端可以从WSDL文档生成Java代理，从而为Web服务提供易于使用的Java接口。Java界面隐藏了客户端的网络通信详细信息，使业务服务提供商能够专注于应用程序的业务和流程部分。

除了提供创建Web服务的工具之外，工作台还提供了一个WSDL编辑器，允许您使用图形界面创建WSDL文件，WSDL验证器可确保您的WSDL文件在语义上正确并可选择检查WS-I合规性，以及Web Services Explorer，允许您动态测试Web服务，而无需生成代理。

工作台支持WSDL 1.1。有关WSDL规范的更多信息，请访问www.w3.org/TR/wsdl

设置Web项目属性

您可以设置许多Web项目属性，这些属性会影响项目生命周期中的阶段及其对其他Web资源和Web开发过程的依赖性。在更新现有项目或将Web资源导入项目后，您可能需要验证或更新这些属性。

企业应用项目

企业应用程序项目将部署J2EE企业应用程序所需的资源联系在一起。

企业应用程序项目包含一组对其他J2EE模块和Java™项目的引用，这些项目组合在一起构成一个EAR文件。这些项目可以是Web模块，EJB模块，应用程序客户端模块，连接器模块，通用实用程序Java JAR文件和EJB客户端JAR文件。在工作台中创建的企业应用程序项目包括部署描述符，以及部署描述符中定义的所有J2EE模块通用的文件。

创建J2EE模块项目时，它可以与企业应用程序项目相关联。项目向导通过允许您指定新的或现有的企业应用程序项目来帮助实现此目的。企业应用程序项目导出为EAR（企业归档）文件，其中包括企业应用程序项目中定义的所有文件以及部署描述符中定义的每个J2EE模块或实用程序JAR项目的相应归档文件，例如Web归档（WAR）文件和EJB JAR文件。

企业应用程序可以包含要包含的模块使用的实用程序JAR文件。这允许多个Web，EJB或应用程序客户端模块在应用程序级别共享代码。这些JAR文件通常称为实用程序JAR文件。为企业应用程序项目定义的实用程序JAR文件可以是项目中的实际JAR文件，也可以包括在组装和部署期间指定为实用程序JAR文件的实用程序Java项目。

要开始开发J2EE应用程序，通常首先要创建一个企业应用程序项目，以将Web，EJB和应用程序客户端模块连接在一起。企业应用程序项目用于从各个模块组成整个应用程序。由于没有源代码直接构建到企业应用程序中，因此这些项目不是Java项目，并且它们不是由Java构建器编译的。

此文件是J2EE规范中定义的企业应用程序的部署描述符，负责将J2EE模块与特定EAR文件相关联。此文件在META-INF文件夹中创建。

为J2EE项目指定目标服务器

开发J2EE应用程序时，可以为J2EE项目指定服务器运行时环境。目标服务器在项目创建和导入期间指定，可以在项目属性中更改。目标服务器设置是用于设置J2EE项目的类路径的默认机制。

为了支持对其Java™运行时环境（JRE）使用不同JDK级别的不同应用程序服务器，工作台会提示您为每个J2EE项目设置目标服务器。例如，如果要利用JDK 1.4.2的功能，则应用程序需要的类路径条目与先前版本的工作台中使用的条目路径条目不同。通过提示您指定目标服务器，工作台强制添加适当的条目以便在您选择的服务器上运行。

您还可以为项目添加多个目标服务器。在这种情况下，工作台会阻止您添加所有目标服务器不支持的任何方面。如果添加多个目标服务器，请选择将作为主服务器的项目类路径的服务器。

创建项目时，将使用两个类路径容器更新项目的类路径。一个容器是JDK容器，另一个是服务器容器。JDK容器指向包含支持JDK版本所必需的JAR文件的目录。服务器容器指向包含所选服务器中可用的多个公共JAR文件的目录。然后，项目将根据位于这些文件夹中的所需JAR文件进行编译，您无需担心在开发期间从服务器添加其他JAR文件。编译项目时，JAR文件包含在类路径中。您仍然可以将自己的JAR文件添加到类路径中。

目标运行时环境在项目的.settings文件夹中的org.eclipse.wst.common.project.facet.core.xml文件中指定。您不应手动编辑此文件;相反，请使用本主题中描述的属性窗口。

所有J2EE项目创建和导入向导都会提示您为生成的项目指定目标服务器。您可以选择的目标服务器列表将根据已安装的运行时，应用程序的J2EE级别和J2EE模块类型进行过滤。例如，对于EJB项目，仅显示支持Enterprise JavaBeans™的应用程序服务器。单个EAR文件中的所有项目必须定位到同一服务器。如果在创建期间创建新项目并将其添加到现有EAR项目，则项目将继承EAR项目的目标服务器设置。

注意：添加到应用程序的实用程序Java项目的目标是与应用程序相同的目标服务器。添加到Web项目的Web库项目将定向到与Web项目相同的目标服务器。

要修改现有项目的目标运行时和默认服务器，请完成以下步骤：

  1. 在Java EE透视图的Project Explorer视图中，右键单击企业应用程序或模块项目，然后选择Properties 。
  2. 在“属性”对话框中选择“ 目标运行时”页面。
  3. 在“ 目标运行时”列表中，选中要为其开发项目的每个运行时旁边的复选框。如果您没有看到要使用的运行时，则需要将其添加到工作台中的运行时。有关更多信息，请参阅定义已安装的服务器运行时环境 。
    注意：仅显示与项目构面兼容的运行时。您可以选中“ 显示所有运行时”复选框，以显示与项目当前构面配置不兼容的运行时。这些运行时显示为灰色。
  4. 要选择主运行时，请单击运行时，然后单击“生成主要”按钮。 如果为项目选择任何运行时，则必须将其中一个运行时作为项目的主运行时。如果从列表中仅选择一个运行时，则该运行时将自动成为主运行时。主运行时以粗体文本显示。
  5. 单击确定 。

UDDI和WSIL之间的关系

UDDI和WSIL之间的关系

Web服务检查语言（WSIL）和通用描述，发现和集成（UDDI）规范都解决了与Web服务发现相关的问题。但是，每个规范采用不同的方法来发现Web服务。这两种规格可以单独使用或联合使用;使用WSIL搜索可以导致在UDDI中查找项目。

UDDI规范通过使用集中模型来解决Web服务发现问题。创建一个或多个存储库以容纳有关企业及其提供的服务的信息。有关服务和业务相关信息的请求和更新将直接针对存储库发出。另外，UDDI规定了存储的描述信息的一部分的特定格式，并且为了便于高级（聚焦）搜索，假设其他描述信息也将被存储并在系统内注册。

UDDI系统有助于集中发现模式，因为这些系统基于提供高级搜索功能的有组织的存储库。这有助于请求者找到潜在的通信伙伴。但是，为了提供高级功能，UDDI需要部署和维护一定数量的基础架构，从而增加了操作成本。此外，除非服务描述仅存储在UDDI中，否则与保持不同版本同步相关的成本。

WSIL规范依赖于完全分布式模型来提供与服务相关的信息。服务描述可以存储在任何位置，并且检索信息的请求通常直接提供给提供服务的实体。WSIL规范没有规定服务信息的任何特定格式。它依赖于其他标准（包括UDDI）来定义描述格式。WSIL规范还依赖于现有的Web技术和基础结构来提供发布和检索其文档的机制。

WSIL提供了直接从提供服务的点通过现有协议传播服务相关信息的能力。这使得能够在单个目标上执行集中发现;但是，由于其分散和分布式模型，如果通信伙伴未知，WSIL不是执行集中发现的良好机制。

UDDI和WSIL规范应视为互补技术，可根据具体情况一起使用或单独使用。例如，可以根据执行搜索WSIL文档时找到的结果填充UDDI存储库。同样，当请求者检索引用存储库中的条目的WSIL文档时，本身可能会发现UDDI存储库。在不需要UDDI提供的高级功能并且约束不允许其部署的环境中，WSIL机制可以提供所需的所有功能。在需要集中管理数据的情况下，单独使用UDDI解决方案可以提供最佳匹配。不应将UDDI和WSIL规范视为互斥。

注意： UDDI和WSIL都只包含对WSDL的引用 - 它们本身不包含WSDL文档。

发布涉及将文件（项目，资源文件和服务器配置）复制到正确的位置，以便服务器查找和使用它们。在测试环境中，这些文件可能已位于正确的位置。在某些情况下，可能会创建服务器配置的临时副本。您可以自动或手动发布应用程序。

如果选中“服务器”首选项页面（“ 窗口” > “首选项” >“ 服务器” >“ 启动” ）上的“ 启动服务器时自动发布”复选框，则工作台将检查服务器上的项目和文件是否已同步。如果不是，则在启动或重新启动服务器时，项目和文件会自动更新。

要手动发布应用程序，您可以在Servers视图中完成以下任一操作：

在工作台中，您可以为发布设置选择多个选项。您可以通过进入Servers视图设置这些发布设置，右键单击服务器并选择Open 。服务器编辑器打开。在服务器编辑器的“ 概述”页面中，在“ 发布”设置下，您将找到以下设置：
  • 永远不会自动发布 ：指定工作台永远不应将文件发布到服务器。
  • 资源更改时自动发布 ：指定在保存与服务器关联的文件上的更改并在“ 发布间隔”设置中通过完整时间间隔后发出发布的工作台。
  • 在构建事件之后自动发布 ：指定在对需要构建并与服务器关联的文件进行更改之后发布发布的工作台，并在发布间隔设置中传递完整时间间隔。
  • 发布间隔（以秒为单位） ：指定在工作台调用服务器上发布的发布之前需要经过的秒数 。但是，如果在此时间间隔完成之前对文件进行后续更改，则会在重置计时器时延迟发布。工作台仅在完整时间间隔过后才发布到服务器。如果将发布间隔设置为0秒，则应在保存文件更改后立即发布。

在工作台中，默认设置是启用资源更改时自动发布选项，并且在发布间隔中设置了值。

此外，在“ 选择发布操作”列表下，选择要在发布期间运行的任务。

手动发布到服务器

如果您不想等待自动发布间隔通过，则可以随时手动请求工作台向服务器发出发布命令。每个手动发布命令都会向服务器发出单个发布请求。要手动发布应用程序，可以在Servers视图中完成以下操作之一：

  • 选择服务器，然后单击“ 发布到服务器” 位于工具栏上的图标。
  • 右键单击服务器，然后选择“ 发布” 。

发布服务器配置中定义的与所选服务器关联的所有项目。这些项目中的资源将传输到服务器定义的位置。

Web服务检查语言（WSIL）

Web服务检查语言（WSIL）是一种服务发现机制，它是UDDI的替代方案，也是UDDI的补充。当您使用UDDI发现Web服务时，您将转到集中式注册表。WSIL是Web服务发现的替代方法。WSIL允许您直接转到服务提供商并询问其提供的服务。

IBM®和Microsoft针对WSIL规范的提议是围绕基于XML的模型设计的，该模型用于构建对使用标准Web服务器技术公开的现有Web服务描述的引用的聚合。

WSIL提供了一种分布式服务发现方法，通过指定如何检查网站是否有可用的Web服务，在服务提供者的提供点提供对服务描述的引用。WSIL规范定义了Web站点上可以查找Web服务描述的位置。

由于WSIL专注于分布式服务发现，因此WSIL规范通过促进发现可能尚未在UDDI注册中心中列出的Web站点上的服务来补充UDDI。本文档中的单独主题讨论了UDDI和WSIL之间的关系 。

WSIL规范没有定义服务描述语言。WSIL文档提供了一种聚合不同类型的服务描述的方法。在WSIL文档中，单个服务可以具有多个对服务描述的引用。例如，单个Web服务可能在WSIL文档中被引用两次：一次直接通过其WSDL引用，再一次通过UDDI注册中心中的businessService条目引用。应将这两个服务描述的引用放入WSIL文档中。如果有多个引用可用，则将所有引用放在WSIL文档中是有益的，这样使用该文档的应用程序可以选择与该应用程序兼容和首选的服务描述类型。

WSIL规范有两个主要功能：

  • WSIL定义了一种XML格式，用于列出对现有服务描述的引用。可以使用任何格式定义这些服务描述，例如WSDL，UDDI或纯HTML。WSIL文档通常在文档中引用的服务的提供点处可用。WSIL文档可以包含对服务描述的引用列表，以及对其他WSIL文档的引用。

    将WSIL文档链接到一个或多个不同的WSIL文档的功能允许您通过将服务描述引用分组到不同的文档中来构建服务描述引用，并构建WSIL文档的层次结构。例如，可以为不同类别的服务创建单独的WSIL文档，并且一个主WSIL文档可以将所有这些文档链接在一起。

  • WSIL定义了一组约定，以便于查找其他WSIL文档。WSIL规范不限制可以引用的服务描述的类型。WSIL规范为WSDL和UDDI定义了一组标准可扩展性元素。WSIL规范是可以访问WSIL文档的位置的定义。
  • 两个约定使WSIL文档的位置和检索变得容易：
    • 固定名称的WSIL文档。 WSIL文档的固定名称是inspection.wsil。inspection.wsil文件放在网站的公共入口点。例如，如果公共入口点是http://entrypoint.com，那么WSIL文档的位置将是http://entrypoint.com/inspection.wsil
    • 链接的WSIL文档。 对WSIL文档的引用也可以出现在不同的内容文档中，例如HTML页面。

有关Web服务检查语言规范的更多信息，请访问www.ibm.com/developerworks/webservices/library/ws-wsilspec.html

  • UDDI和WSIL之间的关系
    Web服务检查语言（WSIL）和通用描述，发现和集成（UDDI）规范都解决了与Web服务发现相关的问题。但是，每个规范采用不同的方法来发现Web服务。这两种规格可以单独使用或联合使用;使用WSIL搜索可以导致在UDDI中查找项目。

肥皂

SOAP（以前称为简单对象访问协议）是一种用于在分散的分布式环境中交换信息的轻量级协议。SOAP消息是从发送方到接收方的信息传输。可以组合SOAP消息以执行请求/响应模式。

SOAP与传输无关，但最常通过HTTP传输，以便与现有的Internet基础结构一起运行。SOAP通过定义路由消息的消息路径来启用已发现Web服务的绑定和使用。SOAP用于查询UDDI以获取Web服务。工作台支持SOAP 1.1。

SOAP是一种基于XML的协议，它为每条消息定义了三个部分：

  • 信封。  信封定义了一个框架，用于描述消息中的内容以及如何处理它。SOAP消息是一个包含零个或多个标题且只包含一个主体的信封。信封是XML文档的顶级元素，为控件信息，消息地址和消息本身提供容器。标头传输任何控制信息，例如服务质量属性。正文包含消息标识及其参数。标题和正文都是信封的子元素。
  • 编码规则。  编码规则集表示应用程序定义的数据类型的实例。编码规则定义了一个序列化机制，可用于交换应用程序定义的数据类型的实例。SOAP定义了一种独立于编程语言的数据类型方案，该方案基于XSD以及针对根据此模型定义的所有数据类型的编码规则。SOAP编码不符合WS-I，因此建议可互操作的Web服务使用Literal（无编码），并且符合WS-I标准。
  • 沟通风格。 通信可以遵循远程过程调用（RPC）或面向消息（Document）格式。这些将在下面讨论。

装订风格

SOAP支持两种不同的通信方式：

远程过程调用（RPC）：调用返回结果的操作。通常与SOAP编码一起使用，不符合WS-I。

文档样式：也称为面向文档或面向消息的样式。这种风格提供了较低的抽象层，需要更多的编程工作。

编码样式

在分布式计算环境中，编码样式定义如何将应用程序中定义的数据值转换为特定协议格式和从特定协议格式转换。翻译过程称为序列化和反序列化。

SOAP规范定义了SOAP编码样式：

SOAP编码： SOAP编码样式允许您从SOAP数据模型序列化/反序列化数据类型的值。此编码样式在SOAP 1.1标准中定义，并且不符合WS-I。

WSDL定义了Literal XML编码样式：

Literal XML： Literal指的是文档应该按原样读取或未编码。该文档被序列化为XMI，这意味着消息XML符合WSDL中的Schema。使用Literal编码时，每个消息部分都引用一个具体的模式定义。文字编码符合WS-I标准。

数据模型

SOAP数据模型的目的是为常见编程语言类型使用的数据类型提供与语言无关的抽象。它包括：

  • 简单的XSD类型。  例如int，string和date。
  • 复合类型。  有两种复合类型， 结构和数组 。结构被命名为聚合类型，其中每个元素都具有唯一的名称或XML标记。数组具有按位置标识的元素，而不是名称。

包含SOAP数据模型的所有元素和标识符都在名称空间URI中定义。SOAP标准定义了如何构造数据类型的规则。项目特定的XML模式必须定义实际的数据类型。SOAP规范的元素在http://schemas.xmlsoap.org/soap/envelope/和http://schemas.xmlsoap.org/soap/encoding/中定义。

SOAP实现

今天可以使用SOAP协议的不同实现。例如，Apache基金会提供的Apache SOAP，它的前身是被称为SOAP4J，以及Apache Axis和是IBM WebSphere®运行时环境中的IBM®项目。

映射

映射定义了限定的XML元素名称，Java™类名称和编码样式之间的关联。映射指定在给定编码下如何将具有完全限定名称的传入XML元素转换为Java类，反之亦然。

有关Apache SOAP的更多信息，请参阅xml.apache.org/soap有关SOAP的更多信息，请参阅www.w3.org/TR/SOAP

动态Web项目和应用程序

Web项目有两种类型：动态和静态 。除了静态资源（如图像和HTML文件）之外，动态Web项目还可以包含动态Java EE资源，如servlet，JSP文件，过滤器和关联的元数据。静态Web项目仅包含静态资源。创建Web项目时，可以包括级联样式表和JSP标记库（用于动态Web项目），以便您可以使用更丰富的项目资源开始开发。

动态Web项目始终嵌入在企业应用程序项目中。用于创建动态Web项目的向导还将创建企业应用程序（EAR）项目（如果该项目尚不存在）。该向导还将更新指定的Enterprise Application项目的application.xml部署描述符，以将Web项目定义为模块元素。如果要导入WAR文件而不是创建新的动态Web项目，则WAR导入向导要求您指定一个已经需要EAR项目的Web项目。

如果您只想创建一个静态的，基于内容的Web应用程序，它不包含动态文件（例如JSP文件或servlet），那么Java EE约定可能会带来额外的开销。在这种情况下，当您只需要最基本的Web项目时，您可能希望使用静态 Web项目类型（请参阅静态Web项目 ）。请注意，通过从“项目”菜单中选择“ 转换为动态Web项目” ，可以将静态Web项目转换为动态Web项目。

HTML内容辅助

HTML是灵活的，因为一些HTML元素允许可选地省略结束标记，例如P ， DT ， DD ， LI ， THEAD ， TR ， TD ， TH等。定义为没有内容的其他HTML元素可能要求始终省略结束标记，例如BR ， HR ， LINK ， META和IMG 。这种灵活性使得HTML源页面编辑器中的内容辅助功能不如使用更严格约束的标记语言那样精确。

当您需要完成标记名称，在开始标记中添加属性名称 - 值对或从枚举的属性值列表中进行选择时，HTML内容辅助最有用。

Web服务概述

Web服务概述

Web服务是一组相关的应用程序函数，可以通过Internet以编程方式调用。企业可以动态地混合和匹配Web服务，以最少的编程执行复杂的事务。Web服务允许世界各地的买家和卖家相互发现，动态连接，并以最少的人工交互实时执行交易。

Web服务是独立的，自描述的模块化应用程序，可以在Web上发布，定位和调用。

Web服务是独立的。 在客户端，不需要其他软件。具有XML和HTTP客户端支持的编程语言足以让您入门。在服务器端，需要Web服务器和servlet引擎。客户端和服务器可以在不同的环境中实现。Web服务可以在不编写任何代码的情况下启用现有应用程序。

Web服务是自我描述的。 客户端和服务器只需要识别请求和响应消息的格式和内容。消息格式的定义与消息一起传播;不需要外部元数据存储库或代码生成工具。

Web服务是模块化的。 通过使用工作流技术或从Web服务实现调用较低层Web服务，可以聚合简单Web服务以形成更复杂的Web服务。

Web服务与平台无关。 Web服务基于一套简明的基于XML的开放标准，旨在促进Web服务与客户之间在各种计算平台和编程语言之间的互操作性。

Web服务可能是任何内容，例如，剧院评论文章，天气预报，信用检查，股票报价，旅行建议或航空旅行预订流程。这些独立的业务服务中的每一个都是可以轻松地与来自相同或不同公司的其他服务集成的应用程序，以创建完整的业务流程。这种互操作性允许企业通过Internet动态发布，发现和绑定一系列Web服务。

Web服务的类别

Web服务可以分为三类：

商业信息。 企业与消费者或其他企业共享信息。在这种情况下，企业正在使用Web服务来扩展其范围。业务信息Web服务的示例是新闻流，天气报告或股票报价。

业务整合。 企业向其客户提供交易性“收费”服务。在这种情况下，企业成为可用于开展商业的全球增值供应商网络的一部分。业务集成Web服务的示例包括出价和拍卖电子市场，预订系统和信用检查。

业务流程外部化。 通过创建全球价值链，企业可以从竞争中脱颖而出。在这种情况下，企业使用Web服务动态集成其流程。业务流程外部化Web服务的一个示例是不同公司之间的关联，以组合特定产品的制造，组装，批发分销和零售。

服务角色和互动

Web服务体系结构中的网络组件可以扮演一个或多个基本角色：服务提供者，服务代理和服务客户端。

  • 服务提供者创建和部署他们的Web服务，并可以通过服务注册表（例如UDDI Business Registry）发布其WSDL描述的服务的可用性。
  • 服务经纪人注册并对已发布的服务进行分类并提供搜索服务。例如，UDDI充当WSDL描述的Web服务的服务代理。
  • 服务客户端使用代理服务（如UDDI业务注册表）来发现所需的WSDL描述的服务，然后绑定并调用服务提供者。

绑定涉及建立成功完成服务所必需的所有环境先决条件。环境先决条件的示例包括安全性，事务监视和HTTP可用性。这些角色之间的关系如图1所示。

使用Servers视图执行以下任务：

  • 创建服务器
  • 编辑服务器
  • 删除服务器
  • 以调试模式启动服务器
  • 启动服务器
  • 以调试模式重新启动服务器
  • 重新启动服务器
  • 停止服务器
  • 发布您的申请
  • 监控服务器端口
  • 从工作台显示或隐藏服务器的元数据
  • 从服务器添加和删除项目

Web服务标准

Internet标准的一个关键属性是它们专注于协议而不是实现。Internet由异构技术组成，通过共享协议成功实现互操作。这可以防止个别供应商在互联网上强加标准。开源软件开发在保持供应商标准实施的互操作性方面发挥着至关重要的作用。

以下标准在Web服务中发挥关键作用：通用描述，发现和集成（UDDI），Web服务描述语言（WSDL），Web服务检查语言（WSIL），SOAP和Web服务互操作性（WS-I）。这些标准之间的关系如图2所示。

UDDI规范定义了开放的，独立于平台的标准，使企业能够在全局业务注册中共享信息，发现注册表上的服务，并定义它们在Internet上的交互方式。有关UDDI的更多信息，请访问www.uddi.org

WSIL是一种基于XML的开放式规范，它定义了一种分布式服务发现方法，该方法通过指定如何检查Web站点以获取可用的Web服务，从而在服务提供者的提供点提供对服务描述的引用。WSIL文档定义了Web站点上可以查找Web服务描述的位置。由于WSIL专注于分布式服务发现，因此WSIL规范通过促进发现可能尚未在UDDI注册中心中列出的Web站点上的服务来补充UDDI。本文档中的单独主题讨论了UDDI和WSIL之间的关系 。有关WSIL的更多信息，请访问www.ibm.com/developerworks/webservices/library/ws-wsilspec.html

WSDL是一种基于XML的开放式规范，描述了网络上Web服务的接口和实例。它是可扩展的，因此无论用于通信的消息格式或网络协议如何，都可以描述端点。企业可以通过UDDI，WSIL或通过电子邮件或网站将URL广播到其WSDL，从而为其Web服务提供WSDL文档。WSDL在本文档中被描述为一个单独的主题。有关WSDL的更多信息，请访问www.w3.org/TR/wsdl

SOAP是一种基于XML的标准，用于通过HTTP和其他Internet协议进行消息传递。它是一种用于在分散的分布式环境中交换信息的轻量级协议。它基于XML，由三部分组成：
  • 一个信封，用于定义用于描述消息内容以及如何处理消息的框架。
  • 一组用于表示应用程序定义的数据类型实例的编码规则。
  • 用于表示远程过程调用和响应的约定。
SOAP通过定义路由消息的消息路径来启用已发现Web服务的绑定和使用。SOAP可用于查询UDDI以获取Web服务。有关SOAP 1.1的更多信息（Web服务工具不支持SOAP 1.2），请参阅www.w3.org/TR/SOAP

服务提供者托管Web服务，并使用SOAP / HTTP或SOAP / JMS等协议对其进行访问。 Web服务由WSDL文档描述，该文档存储在提供程序的服务器或特殊存储库中。UDDI业务注册表和WSIL文档可以引用WSDL文档。这些包含指向Web服务的WSDL文件的指针。

WS-I简单SOAP绑定配置文件和WS-I附件配置文件是WSDL和Web服务协议（SOAP / HTTP）流量必须遵守的要求的概要，以声明WS-I一致性。Web服务WS-I验证工具目前支持WS-I Simple SOAP Binding Profile 1.0和Attachment Profile 1.0。要查看规范，请参阅WS-I Web站点，在Resources下选择Documentation ： http ： //www.ws-i.org

    Web服务描述语言（WSDL）是用于描述基于XML的联网服务的标准规范。它为服务提供者提供了一种简单的方法来描述对其系统的基本请求格式，而不管底层的运行时实现如何。

  • JAX-RPC
    JAX-RPC代表基于XML的RPC的Java API，也称为JSR 101。它是一个描述Java应用程序编程接口（API）的规范，用于构建使用远程过程调用（RPC）和XML的Web服务和Web服务客户端。它将Java到WSDL和WSDL标准化为Java映射，并提供用于在Java平台上开发Web服务和Web服务客户端的核心API。RPC机制通常用于分布式客户端/服务器模型，使客户端能够在其他系统上执行过程。

  • Web服务互操作性（WS-I）
    WS-I是一个旨在促进跨平台，操作系统和编程语言的Web服务互操作性的组织。
