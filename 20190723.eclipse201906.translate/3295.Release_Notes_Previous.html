Sirius 5.0.1的变化

Sirius 5.0.1是一个维护版本，只对错误修正和对播出编辑器的UI进行小调整。

Sirius 5.0.0的变化

用户可见的更改

  • 添加 .aird文件中提供了一个新的aird编辑器 。它提供了一种替代的基于表单的GUI，用于可视化和操作sirius会话，这些会话与Modeling透视图或Model Explorer视图无关。有关更多详细信息，请参阅编辑器的文档 。
  • 改性从Palette注释的标签对准已从朱诺修饰以露娜或者Eclipse的较新版本。标签对齐已从TOP ＆ CENTER变为MIDDLE ＆ LEFT （分别）。初始对齐（ TOP ＆ CENTER ）已恢复。这是通过自动迁移 aird文件来完成的。
  • 修改 “ 导出图表为图像”操作现在根据图表大小调整输出分辨率。这样可以更好地解决中小图（以及更大的文件）以及在图表非常大的情况下不再出现内存错误的操作。
  • 修改 “拉直”动作以前有重叠检测的限制。这已不再是这种情况。先前已知的限制已从文档中删除（ 在多选的情况下，重叠检查在现有边界节点的当前上下文中完成，而不是在每个拉直边缘之后的边界节点的位置的上下文中完成 ）。
  • 图元素上的修改装饰器绘制在其他图形之上。如果图A的装饰器与图A上的另一个图B之间存在重叠，则当用户点击装饰器时，现在选择图B（在它是父图A之前）。

开发人员可见的变化

Specifier-Visible Changes

  • 添加现在可以在组中自动创建一个或多个预先配置的窗口小部件，如果已经指定了域类，并且已经指定了包含它的属性视图的元模型。创建的窗口小部件将来自域类中的功能（例如，将为布尔属性创建正确配置的Checkbox窗口小部件）。
  • 添加的属性视图定义现在支持扩展和覆盖属性视图描述，而无需修改原始属性。有关详细信息，请参阅文档
  • 添加了新模型操作。现在，您可以利用属性视图语言进行新的专用模型操作，从而打开对话框或向导 。您还可以使用新的Let模型操作创建新变量，有关详细信息，请参阅其文档 。
  • 添加了在属性视图定义中，您现在可以在页面和组上创建工具栏操作 ，以便在其工具栏中定义小按钮。
  • 由于新的Image表达式， 添加了 Buttons和Widget Actions现在可以使用图像和标签。
  • 添加现在可以在最终用户点击Reference Widget时执行某些行为，这要归功于添加了初始操作 。
  • 已添加您可以指示Dynamic Mapping For应强制使用“ 强制刷新”属性刷新用户界面。
  • 添加了 类别以便组织属性 DSL的页面和组。
  • 修改在Sirius 5.0中引入了瞬态层的概念。（请参阅Diagram章节中的Sirius Specifier和Sirius用户文档）。瞬态层是仅包含装饰或工具的附加层。如果您的VSM中有此类图层，则用户可能会遇到一些意外行为。如果默认情况下图层未处于活动状态，则其激活状态将不会在会话结束时保留。换句话说，如果在图中启用了瞬态层，然后会话关闭，则在下一个图表打开时，该图层将被禁用。如果用户具有使用以前版本制作的现有表示文件，并且已激活此类图层，则在Sirius 5.0.0中的下一次打开时将不再启用它们。如果图层默认处于活动状态，则在打开图表时将自动应用该图层。我们将此行为视为已知限制，并且可以在进一步的版本中进行增强。与此同时，有一个解决方法，包括添加假映射（即任何映射，其字段为空），以便该层不会被视为瞬态。
  • 已修改用于配置“属性”视图描述所使用的元模型的选项卡现在使用与用于配置表示的元模型选项卡相同的界面。
  • 修改现在完全支持VSM中的域类和类型名称的package::Class语法，并且是新的默认值。这意味着Sirius会解释此语法，并在输入第一个:字符时为其提供完成。遗留语法package.仍然支持Class兼容性。
  • 已修改在属性 DSL设计中的上下文操作的组织已得到改进。
  • 修改 属性 DSL的大多数概念现在可以有标识符，标签和一些文档。

迁移

  • 添加此版本中添加了迁移，以修复注释中文本的对齐方式（如用户可见更改中所述）。存储在视点的属性版本中的相应版本：aird文件的DAnalysis是12.0.0.201704070000 。

  • 修改后的Acceleo查询语言 （AQL）解释器得到了改进。AQL及其在Sirius中的支持不再被认为是实验性的，而是现在推荐用于新VSM的查询语言。改进主要涉及更好的完成和验证，以及更精确的错误报告。特别是AQL表达式现在能够在Interpreter视图中显示警告和错误。评估后在Interpreter视图中显示的消息也得到了改进，以显示计算值类型的限定名称（ ePackageName::eClassName ）。单击下面的图像可以查看实现的改进（单击可放大）。

  • 修改与前一点相关， Viewpoint规范项目向导创建的项目现在配置为默认使用AQL（而不是之前的Acceleo 3 / MTL）。它们依赖于org.eclipse.sirius.common.acceleo.aql而不是org.eclipse.sirius.common.acceleo.mtl ，并且不再具有Acceleo特性（AQL不需要它）。如果要在查询中使用其他语言，请确保设置适当的依赖关系，以便在运行时出现Sirius对该语言的支持以解释查询。
  • 修改在Sirius 3.0.0中引入了对图表上具有垂直或水平隔室的容器的支持作为实验特征。围绕该特征的行为已经进行了若干改进和校正，这不再被认为是实验性的。VSM编辑器中的验证规则和工具提示可帮助说明符创建一致的映射结构。
  • 修改 实验对图上的递归分区的新实验支持： Container Mapping的子表示属性，定义为一个分区 （其父ContainerMapping具有VerticalStack或HorizontalStack 子代表示 ），现在也可以设置为Vertical Stack and Horizontal堆栈 。它允许定义混合两个堆叠方向的复杂堆叠的隔间 。虽然工作可靠，但这种支持在Sirius 3.1.0中标记为实验性，因为仍有一些行为不符合预期。如果您使用此功能，请注意在将来的版本中可能仍会更改某些详细信息，包括3.1.x维护版本。
  • 已修改您现在可以将BorderSizeComputationExpression设置为0以用于ContainerMapping样式。在Sirius 3.1.0之前，大小为0在运行时被解释为1。出于兼容性原因，对于ContainerMapping样式，所有现有BorderSizeComputationExpression等于0都自动设置为1（对于容器元素的表示文件，此表达式的结果也是如此）。对于ContainerMapping ，新样式的BorderSizeComputationExpression的默认值现在为1，但Worskpace Image的值为0除外。

Sirius 3.0.0的变化

用户可见的更改

  • 添加了重新引入打印表格表示的功能。
  • 新增快速轮廓特征已经增加了对树和表格编辑器，详细信息请参见文档。
  • 添加现在，用户可以从大纲，标签栏或上下文菜单中隐藏列表或容器的标签。
  • 添加在外观选项卡中，除了粗体和斜体之外 ，现在可以将字体格式样式设置为下划线和打击 （或这些的任意组合）。
  • 图上添加的 Container元素现在支持垂直和水平隔离专区布局（仅当建模者配置使用此新功能时才可见）。
  • 添加在表格和树形编辑器中，现在可以通过按图表中的F5来启动刷新 （以前唯一的方法是单击“ 刷新”图标或菜单项）。
  • 添加在表格编辑器中，点击Del键现在相当于在上下文菜单中选择“ 删除行 ”。
  • 添加在图表上，现在可以在列表容器上设置自定义背景图像。
  • 已修改在“ 视点选择”对话框和向导页面中，将删除从插件加载的视点的“插件”装饰器，并添加了从当前工作空间加载的视点的“文件夹”装饰器。这使得图标在最常见的视图中更加明显，其中视点来自已安装的插件。

Specifier-Visible Changes

  • 添加了 实验可选的支持实验性加速查询语言 （AQL）已添加到编写解释表达式。它由org.eclipse.sirius.common.acceleo.aql （评估表达式所需）和org.eclipse.sirius.common.acceleo.aql.ide （可选，仅需要提供表达式等时间特性完成）插件。AQL类似于Acceleo 3 / MTL，但更简单，并且在Sirius环境中提供了更好的性能。如果您使用此功能，请注意在将来的版本中可能仍会更改某些详细信息，包括3.0.x维护版本。有关语言本身的更多详细信息，请参阅AQL文档 。
  • 添加了对图表上具有垂直或水平隔间的容器的实验性新实验支持： Container Mapping的children presentation属性有两个新值： Vertical Stack和Horizontal Stack 。它们表明容器将其子容器呈现为垂直/水平堆叠的隔间。虽然工作可靠，但这种支持在Sirius 3.0.0中标记为实验性，因为仍然存在一些不符合预期的映射，样式和行为的组合。如果您使用此功能，请注意在将来的版本中可能仍会更改某些详细信息，包括3.0.x维护版本。
  • 添加无处不在VSM中可以配置图像路径，文本字段右侧会出现一个新按钮。它会打开一个对话框，从工作区或已安装的插件中选择图像。
  • 添加 Everywhere可以在VSM中配置解释表达式，文本字段右侧会出现一个新按钮。它打开一个多行对话框，可以更轻松地编辑长表达式。多行对话框内支持自动完成。
  • 现在支持添加压缩SVG图像（ *.svgz ），支持普通SVG图像。
  • 添加在“实时模式”下开发建模器时，VSM与使用它的示例会话位于同一工作空间中，使用service:或aql: VSM表达式现在可以调用工作空间中定义的Java服务（无需启动单独的运行时）。
  • 添加现在可以使用快捷键Ctrl+o在VSM编辑器中访问快速概述。它支持对元素的名称和属性进行过滤，以便在复杂的VSM中进行快速，仅键盘导航。
  • 添加 “默认功能隐藏标签”已添加到容器样式中。

Sirius 2.0.0的变化

用户可见的更改

  • 现在可以选择与选择矩形相交的元素，而不是选择矩形完全包含的元素。当用户从右到左选择元素时，将启用此新行为。当用户从左到右选择元素时，保持正常模式（先前模式）。
  • 现在，当其中一个肢体移动时，边缘外观尽可能地保持。肢体的移动应仅移动边缘的最近部分。
  • 移动节点（容器或不移动）时，边缘外观尽可能保持。现在，节点的移动仅影响链接边缘的最近部分。

  • 当节点，容器或非容器调整到左侧，上方或两者时，其子节点（容器和边界节点内的元素）的位置不会更改。在调整大小期间按F3功能键可以检索先前的行为。

  • 现在默认启用“捕捉形状”（参见Sirius> Sirius Diagram> Rulers and Grid首选项页面）。这仅适用于新图表。现有图表不受影响。
  • 两个顶级菜单更改了“导航”顶级上下文菜单条目，其中包含用于创建新表示和打开现有表示的混合操作：
    • 一个名为“新建”，仅列出在所选元素上创建新表示的可用操作。
    • 一个名为“Open”，仅列出所选元素的现有表示。
  • 当调整形状的大小（无论方向）时，保持边缘（朝向或来自该边缘）的连接位置。在此之前，边缘连接根据调整大小的比例移动。

这个里程碑（Sirius 1.0M5）引入了几个新的插件，统称为“库扩展”。它们都被命名为org.eclipse.sirius.ext.LIB ，其中LIB是Sirius所依赖的库或组件的名称。这些插件用于隔离通用代码，这些代码使用Sirius所需的功能和帮助程序扩展现有库，但不依赖于Sirius本身。Sirius中相对较大的代码部分属于这种类型：可以/应该在EMF，GMF，Eclipse本身等中的东西，它们不是Sirius特有的，但它们使构建Sirius本身变得更容易。Sirius 1.0M5引入了其中几个库扩展，并开始用现有的类填充它们，这些类在Sirius中“隐藏”，现在在库扩展中公开。大多数情况下，使用Sirius API的现有代码只需要：

  1. 添加必要的Import-Package （推荐）或Require-Bundle以依赖他们现在使用的代码所在的库扩展;
  2. 执行一个简单的“组织导入”操作，该操作应该在新位置找到类和接口。

从Obeo Designer或Viewpoint迁移到Sirius

如果您在使用此技术成为Eclipse Sirius并希望迁移到Sirius之前有现有项目，则必须：

  1. 首先将您的项目和模型迁移到最新的Sirius版本的Viewpoint或Obeo Designer（Viewpoint是以前的Sirius名称）;
  2. 然后将它们迁移到Sirius 0.9。

Obeo Designer版本中提供的最新版本的Viewpoint是在Obeo Designer 6.2中发布的Viewpoint 6.8。Sirius 0.9基于Viewpoint 6.10，未公开发布。

如果您使用的是Obeo Designer，则必须首先将项目和模型迁移到Obeo Designer 6.2，然后查阅Viewpoint 6.9和6.10的发行说明 ，其中列出了从Viewpoint 6.8到6.10的API更改。

一旦您的项目与您可用的最新Viewpoint兼容，您就可以迁移到Sirius，首先更新您的Viewpoint规范项目和使用Viewpoint API的插件，然后更新任何建模项目和表示文件（ *.aird ） 。

迁移Viewpoint规范项目

  • 必须将对Viewpoint插件的任何插件依赖项更新为相应的Sirius插件和版本号。请参阅下面的对应表，了解要使用的插件名称。
  • 如果您的插件使用了Viewpoint中定义的任何扩展点，则需要更新plugin.xml文件以引用正确的扩展点标识符。特别是，使用Viewpoint创建的所有Viewpoint规范项目都使用其plugin.xml的fr.obeo.dsl.viewpoint.componentization来注册它们定义的VSM。应将其更改为使用等效的org.eclipse.sirius.componentization扩展点ID。其他扩展点的标识符更改遵循与定义插件的名称更改相同的规则。
  • VSM（ *.odesign文件）的迁移是自动的：只需在VSM编辑器中打开一次文件，执行一些无操作更改以使编辑器“脏”并保存。保存的版本将迁移到Sirius。
  • 如果您使用的是使用Viewpoint API的Java代码，则需要更新它以使用Sirius API。源文件夹上的简单“Source> Organize Imports”应该找到您在新命名空间中使用的大多数类。Viewpoint代码库和API使用术语“Viewpoint”作为产品名称和概念名称。当它引用产品的名称时，名称被更改为使用“Sirius”，因此您可能需要调整代码中的某些名称，而不仅仅是更新导入（例如， ViewpointControlCommand现在是SiriusControlCommand ）。

请注意，如果您使用了Obeo Designer，则可能已将旧版Acceleo 2.x语言用于*.odesign文件中的计算表达式（如果表达式的格式为<%something%>则使用旧语言）。在Sirius 0.9中不再支持该语言，因此您必须迁移表达式以使用Acceleo 3和/或专用解释器（ var: ， feature:和service: ,请参阅文档）。

迁移建模项目和表示文件

  • 首先迁移建模项目和表示文件使用的任何建模器定义（请参阅上面的部分）。
  • 使用Viewpoint创建的建模项目将不会被Sirius直接识别，因为自然识别符已更改。必须使用项目上下文菜单中的“配置/转换为建模项目”操作，使其成为与Sirius兼容的建模项目（有关更改的技术详细信息，请参阅下面的部分）。
  • 表示文件（ *.aird ）的迁移在内存中加载时会自动执行。它将在您第一次保存时成为永久性的（您可以通过打开表示，进行任意更改并保存会话来强制执行此操作）。

如果您使用Viewpoint创建了建模项目，则必须启动“配置/转换为建模项目”操作以用新的替换旧的自然ID（ fr.obeo.dsl.viewpoint.nature.modelingproject ）。
如果您有许多建模项目要“迁移”，您可以轻松地创建一个脚本，在每个建模项目的所有.project文件中用org.eclipse.sirius.nature.modelingproject替换fr.obeo.dsl.viewpoint.nature.modelingproject 。

其他API更改

本节仅列出Sirius 0.9.0中的API更改与基于它的Viewpoint的最新版本（即Viewpoint 6.10）相比较。

已添加API

  • 添加IMigrationParticipant.getPackage()以返回用于给定命名空间的EPackage。此更改涉及我们的内部迁移框架。这对于处理我们的命名空间更改很有用（从Viewpoint到Sirius）。

API删除

  • 删除了org.eclipse.sirius.common.tools.api.interpreter.IAcceleoInterpreterVariables ：此接口仅定义了一个常量，该常量IAcceleoInterpreterVariables了Acceleo语言的过时版本的语法。
  • 删除了org.eclipse.sirius.common.tools.api.resource.IFileContributorExtension ：此接口仅定义了一些相对于不再存在的扩展点的常量。
  • 删除了org.eclipse.sirius.reconciler扩展点定义（以前在org.eclipse.sirius插件中定义）：此扩展点不再受支持（并且已经很长时间没有了）。
  • 从Session删除了所有已弃用的方法，并将内部调用替换为具有IProgressMonitors的等效版本。删除的方法是：
void open();
void addSemanticResource(Resource newResource, boolean addCrossReferencedResources);
void addSemanticResource(URI semanticModelURI, boolean addCrossReferencedResources);
void addSemanticResource(URI semanticModelURI, boolean addCrossReferencedResources, IProgressMonitor monitor);
void removeSemanticResource(Resource resource, boolean removeCrossReferencedResources);
void removeSemanticResource(Resource semanticResource);
void save();
void close();
Collection<Viewpoint> getSelectedViewpoints();
void createView(Viewpoint viewpoint, Collection<EObject> semantics);
void createView(Viewpoint viewpoint, Collection<EObject> semantics, boolean createNewRepresentations);
void addSelectedView(DView view) throws IllegalArgumentException;
void removeSelectedView(DView view);

Specifier-Visible Changes

sizeComputationExpression有新变量可用于计算EdgeStyleDescription和BracketEdgeStyleDescription的大小：

  • 图 ：当前的DDiagram
  • 观点 ：目前的DEdge
  • sourceView ：当前边缘的源视图。
  • targetView ：当前边缘的目标视图。

用户可见的更改

  • 由于外部依赖性问题，已针对0.9版本禁用了打印表格表示的功能（有关详细信息，请参阅错误＃422223 ）。它应该在1.0中重新引入。
