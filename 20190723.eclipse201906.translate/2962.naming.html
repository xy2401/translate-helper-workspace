命名约定

最后修订于2012年8月15日

Eclipse平台的命名约定和指南：

  • Java包
  • 类和接口
  • 方法
  • 变量
  • 插件和扩展点

Java包

Eclipse Platform包含一系列Java包。包命名空间的管理符合Java的包命名准则 ;未经包子树所有者事先批准，不得创建子包。Eclipse平台的包都是子包org.eclipse 。org.eclipse之后的第一个包名称组件称为主要包名称。在Eclipse 4.3发行版中分配了以下主要的org.eclipse包：
org.eclipse.ant [。*] - Ant支持
org.eclipse.compare [。*] - 比较支持
org.eclipse.core [。*] - 平台核心
org.eclipse.debug [。*] - 调试
org.eclipse.equinox [。*] - Equinox子项目
org.eclipse.help [。*] - 帮助支持
org.eclipse.jdi [。*] - Java调试接口（JDI）的Eclipse实现
org.eclipse.jdt [。*] - Java开发工具
org.eclipse.jface [。*] - JFace
org.eclipse.jsch [。*] - Jsch SSH库支持
org.eclipse.ltk [。*] - 通用语言工具基础结构
org.eclipse.osgi [。*] - 用于与OSGi交互的Eclipse API
org.eclipse.pde [。*] - 插件开发环境
org.eclipse.search [。*] - 搜索支持
org.eclipse.swt [。*] - 标准窗口小部件工具包
org.eclipse.team [。*] - 团队支持以及版本和配置管理
org.eclipse.text [。*] - 文本编辑器框架
org.eclipse.ui [。*] - 工作台
的org.eclipse.update [。*] - 更新/安装
保留以下包名称段：
internal - 表示不包含API的内部实现包
tests - 表示仅包含测试套件的非API包
examples - 表示仅包含示例的非API包
这些名称用作限定符，并且只能出现在主要包名称之后。例如，
org.eclipse.core.internal.resources - 正确用法
org.eclipse.internal.core.resources - 不正确。 内部先于主要包名。
org.eclipse.core.resources.internal - 不正确。 内部不会立即遵循主要包名称。

除了Eclipse平台的结构之外：Eclipse平台分为Core和UI 。归类为核心的任何东西都独立于窗口系统;依赖于Core而不是UI的应用程序和插件可以无头运行。Core和UI之间的区别与API和非API之间的区别不一致; Core和UI都包含API。 Eclipse Platform的UI部分称为Workbench。Workbench是一个高级UI框架，用于构建具有由可插入组件构建的复杂UI的产品。Workbench构建于JFace，SWT和Platform Core之上。SWT（标准窗口小部件工具包）是一种低级，独立于操作系统平台的与本机窗口系统通信的方式。JFace是一个中级UI框架，可用于构建复杂的UI部分，例如属性查看器。根据定义，SWT和JFace是UI。Java工具是在工作台上构建的Java IDE。放在一边。

API包 API包是包含必须可供ISV使用的类和接口的包。 API包的名称需要对ISV有意义。 ISV需要记住的不同软件包的数量应该很少，因为大量的API软件包可能使ISV难以知道他们需要导入哪些软件包。在API包中，所有公共类和接口都被视为API。 API包的名称不应包含内部 ， 测试或示例，以避免与命名非API包的方案混淆。

内部实现包作为平台实现的一部分但不包含应暴露给ISV的API的所有包都被视为内部实现包。所有实现包都应标记为内部 ，标记出现在主要包名之后。ISV将被告知所有标记为内部的包都是不受限制的。（简单的文本搜索“ .internal。 ”检测源文件中的可疑引用;同样，“/ internal / ”在.class文件中是可疑的）。

测试套件包所有包含测试套件的软件包都应标记为测试 ，标签出现在主要软件包名称之后。全自动测试是常态;因此，例如，org.eclipse.core.tests.resources将包含在org.eclipse.core.resources API自动化测试。交互式测试（需要动手测试的测试）应标记为交互式作为最后一个包名称段;因此，例如， org.eclipse.core.tests.resources.interactive将包含相应的交互式测试。

示例包所有包含发送到ISV的示例的包都应标记为示例 ，标记位于主要包名之后。例如， org.eclipse.swt.examples将包含有关如何使用SWT API的示例。

附加规则：

  • 包名称应仅包含小写ASCII字母数字，并避免使用下划线_或美元符号$字符。

类和接口

Java的命名指南陈述

类名应该是名词，大小写混合，每个内部单词的首字母大写。尽量保持您的类名简单和描述性。使用整个单词 - 避免使用缩写词和缩写词（除非缩写词比长词形式使用得更广泛，例如URL或HTML）。
 
例子：
类栅格;
class ImageSprite;
 
接口名称应该像类名一样大写。

对于接口名称，我们遵循“I”-for-interface约定：所有接口名称都以“ I ”为前缀。例如，“ IWorkspace ”或“ IIndex ”。该约定通过使接口名称更容易识别来帮助代码可读性。（Microsoft COM接口订阅此约定）。

附加规则：

  • 异常类（ Exception的子类）的名称应遵循以“ Exception ”结尾的常见做法。

方法

Java的命名指南陈述

方法应该是动词，在第一个字母小写的大小写混合的情况下，每个内部单词的首字母大写。
 
例子：
跑（）;
快跑（）;
的getBackground（）;
附加规则：
  • 命名方法应该遵循命名getter（ get X （） ），setters（ set X （） ）和谓词（ 是X （） ， 有X （） ）的通用做法。

变量

Java的命名指南陈述

除变量外，所有实例，类和类常量都是小写的第一个字母。内部单词以大写字母开头。变量名称不应以下划线_或美元符号$字符开头，即使两者都允许。
 
变量名称应该简短而有意义。变量名的选择应该是助记符 - 也就是说，旨在向不经意的观察者表明其使用的意图。除临时“一次性”变量外，应避免使用单字符变量名。临时变量的通用名称是整数的i ， j ， k ， m和n ; c ， d和e表示字符。
 
例子：
int i;
char c;
float myWidth;

（注意：我们不再遵循使用“ f ”作为前缀非常量字段名称的约定，例如“ fWidget ”。）

常量

Java的命名指南说明

声明类常量和ANSI常量的变量名称应全部为大写，并用下划线（“_”）分隔。
 
例子：
static final int MIN_WIDTH = 4;
static final int MAX_WIDTH = 999;
static final int GET_THE_CPU = 1;

插件和扩展点

所有插件（包括属于Eclipse平台的插件，如Resources和Workbench插件）必须具有与Java包相同的命名模式的唯一标识符。例如，工作台插件名为org.eclipse.ui [。*] 。

插件命名空间是分层管理的;未经封闭命名空间所有者事先批准，不要创建插件。

期望多个扩展的扩展点应具有多个名称。例如，“ 建设者 ”而不是“ 建设者 ”。
