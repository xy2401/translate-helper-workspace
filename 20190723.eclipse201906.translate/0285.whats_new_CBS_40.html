2.1标准和管理构建系统的合并

在CDT 4.0中，标准和托管构建系统被整合到一个CDT构建系统中。这允许所有标准制作功能用于托管构建项目，反之亦然，并且还为构建设置提供一个通用API和UI界面。

CDT构建系统是基于托管构建系统功能（org.eclipse.cdt.managedbuilder.core和org.eclipse.cdt.managedbuilder.ui插件）创建的。因此，用于CDT 3.x中的托管构建项目的所有API对4.0中的Makefile项目都有效，即

org.eclipse.cdt.managedbuilder.core.buildDefinitions扩展点充当工具集成到Build System的入口点。

org.eclipse.cdt.managedbuilder.core。ManagedBuildManager类充当访问/操作构建设置信息的入口点。

从API的角度来看，Makefile和Managed Build Projects之间没有主要区别。从Build System的角度来看，“Makefile”和“Managed”模式之间的区别在于，在Managed构建的情况下执行makefile生成（或使用Internal Builder），而对于Makefile Build，没有makefile生成完成，因此预期makefile由用户提供。在Managed和Makefile模式之间切换实际上是打开或关闭Makefile生成。

工具链，工具，构建器定义现在具有“supportsManagedBuild”属性，用于指定是否支持给定对象的托管构建。

2.2新的新项目向导

CDT 4.0提供了一个新的New Project Wizard。本节主要关注和描述工具集成商如何影响他/她的工具链和项目类型在第一个向导页面上的向导中的显示方式。有关“新建项目向导”UI的更多详细信息，请参阅“新建项目向导”用户说明。

第一个向导页面显示向导允许用户选择要与项目类型一起使用的项目类型和工具链。

 

在“新建项目向导”中显示项目类型和工具链

工具集成器有两个选项，可以在向导中显示他的项目类型。

  1. 定义新的自定义项目类型条目。
  2. 使用常规项目类型条目机制。

注意：新的“新建项目向导”现在使用工具链进行操作，允许选择要在项目创建时使用的工具链。因此，要求代表不同工具链/工具/构建器的所有toolChain / tool / builder构建定义必须具有不同的名称以及代表同一个工具链/工具/构建器必须具有的toolChain / tool / builder构建定义。相同的名字。

示例：为了说明上述要求，这里是如何在gnu工具链定义中处理它：

gnu插件包含Linux上的gcc链接器工具定义为

<工具
natureFilter = “cnature”
NAME = “％ToolName.linker.gnu.c”
outputFlag = “ - ○”
命令= “GCC”
ID = “cdt.managedbuild.tool.gnu.c.linker”

...

   

同时，gnu工具链定义通过将新工具定义为“cdt.managedbuild.tool.gnu.c.linker”工具的超类来引用gcc链接器

<工具
ID = “cdt.managedbuild.tool.gnu.c.linker.base”
超类= “cdt.managedbuild.tool.gnu.c.linker”>

...

上面列出的两个工具定义实际上都被构建系统视为两个不同的工具，而它们都引用了同一个“gcc”可执行文件。为了使构建系统知道两个工具定义都引用同一个工具/可执行文件，需要两个工具定义指定同一个名称。在上面的示例中，id =“cdt.managedbuild.tool.gnu.c.linker.base”的工具没有指定任何名称，因此使名称从超类工具继承，因此两个工具都具有相同的名称。

另一方面，cygwin gcc链接器定义为

<工具
ID = “cdt.managedbuild.tool.gnu.c.linker.cygwin”
NAME = “％ToolName.linker.cygwin.gnu.c”
超类= “cdt.managedbuild.tool.gnu.c.linker”>
...

虽然工具定义被定义为linux gcc链接器的超类，但它指的是与Linux链接器定义不同的工具（可执行文件）。cygwin链接器定义指定name =“％ToolName.linker.cygwin.gnu.c”，它与Linux gcc链接器定义的名称不同。

定义新的项目类型条目

如果工具集成商愿意将他/她的项目类型显示为具有自定义名称的单独条目，则他的项目类型定义必须为项目类型指定“名称”属性，例如

在向导中选择项目类型条目后，“工具链：”窗格将显示已定义/与项目类型关联的工具链列表

使用常规项目类型条目

“通用项目类型”机制允许在一个项目类型条目下对多个项目类型/工具链进行分组，从而确保项目类型信息的紧凑性，并确保跨不同工具链和集成的共同用户体验。选择常规项目类型条目后，“ToolChains：”窗格将列出由不同项目类型提供的所有工具链，允许用户选择要与给定项目类型一起使用的工具链。

一般项目类型条目是什么？

一般项目类型条目机制是基于新CDT构建系统中引入的新构建属性机制而制定的。每个常规项目类型条目都是“buildArtefactType”属性的值，表示构建工件类型。New Project向导搜索支持每个已定义构建工件类型的工具链，并在每个构建工件类型的“ToolChains：”窗格中显示它们。

CDT预定义了构建工件类型属性的以下值：

“org.eclipse.cdt.build.core.buildArtefactType.exe” - 表示可执行文件

“org.eclipse.cdt.build.core.buildArtefactType.staticLib” - 表示静态库

“org.eclipse.cdt.build.core.buildArtefactType.sharedLib” - 表示共享库

ISV可以通过贡献org.eclipse.cdt.managedbuilder.core.buildProperties扩展点来定义自己的自定义构建工件值。

有关“构建属性”机制的更多详细信息，请参阅“构建属性”部分。

贡献一般项目类型条目

指定应使用常规项目类型条目所需的最小步骤，项目类型定义应指定“buildArtefact”属性并将其分配给buildArtefactType构建属性的值之一，例如

<项目类型
buildArtefactType = “org.eclipse.cdt.build.core.buildArtefactType.exe”
ID = “cdt.managedbuild.target.gnu.exe”>

...

 

2.3 makefile项目的多配置支持

新的CDT Build System支持多种配置设置。这适用于Managed和Makefile项目。与托管项目一样，org.eclipse.cdt.managedbuilder.core。IConfiguration接口用作配置设置的持有者。

 

2.4 makefile项目的工具链支持

新的CDT构建系统提供程序是工具链/工具/构建器定义的概念。工具链用于Makefile项目，作为包含/分组适用于给定工具链的不同设置的持有者/配置文件，从而允许根据所使用的工具链自动配置项目seyttings。这包括调整错误解析器，二进制解析器设置，扫描程序发现配置文件设置等。

2.5每个文件夹设置

现在可以在每个文件夹级别指定自定义设置。也可以从构建中排除文件夹。将显示一个新的接口folderInfo架构元素。有关更多详细信息，请参阅上面的buildDefinitions模型。

2.6内部构建器

现在可以在工具链定义中使用内部构建器。内部构建器被构建系统视为常规构建器。要将内部构建器与工具链相关联，工具链应定义构建器，通过构建器#superClass属性将内部构建器指定为其超类。内部构建器ID为“org.eclipse.cdt.build.core.internal.builder”。例如

    </助洗剂>

2.7自定义配置构建

现在可以启动项目构建，该构建将使用与当前活动配置不同的自定义设置。

支持以下模式：

  1. 使用一个构建请求构建任意数量的项目构建配置

  2. 使用自定义构建器设置构建任意数量的配置

可以通过org.eclipse.cdt.managedbuilder.core以编程方式使用此功能。ManagedBuildManager.buildConfigurations（...）方法。

它也可以通过“Build Configurations” - >“Build”项目上下文菜单在UI中访问。

 

注意：对于托管构建（自动生成makefile），此功能仅在构建文件生成器实现org.eclipse.cdt.managedbuilder.makegen的情况下有效。IManagedBuilderMakefileGenerator2。此接口通过定义一个新的initialize（）方法来扩展旧的IManagedBuilderMakefileGenerator，该方法接受IConfiguration和IBuilder而不是IManagedBuildInfo，从而消除了只有活动配置可用于构建的假设。

随CDT提供的默认GnuMakefileGenerator现在实现此接口，因此，如果构建器使用此默认实现，则不需要进行任何更改。

 

2.8为托管构建项目创建目标构建

结合旧Managed和Standard Build系统的功能，make目标构建现在也可用于Managed Projects。

在内部，它实际上被视为自定义配置构建的特殊情况（请参阅“自定义配置构建”部分）。

  1. 指定工具链/工具转换器

2.10构建属性机制

构建属性机制允许定义一组属性以及每个属性的可能属性值。

可以将一组属性与配置/项目类型相关联。

允许工具链和工具指定这些属性支持的属性集及其支持的值。

 

鉴于这些功能，Build Properties机制用于促进以下功能：

  1. 现在，New Project Wizard现在根据“org.eclipse.cdt.build.core.buildArtefactType”构建属性的可用值创建一组“常规”项目类型条目，并允许将工具链与这些项目类型条目相关联给出有关工具链支持并为项目类型定义的构建工件类型值的信息。

  2. 通过工具链修改功能， 工具链，工具和构建器可以使用构建属性机制来使用启用表达式机制来调整其设置（例如，工具的选项值等）。这样可以轻松地进行工具链/工具/构建器切换，而无需实现转换器。使用此机制可以定义工具，其默认选项值将取决于与配置关联的设置构建属性。因此，可以有一个工具链定义，其值将根据工具链是否用于创建可执行文件或库等自动调整，或者取决于工具链是否在“发布”中使用或“调试”构建等

  3. 现在可以通过在工具链支持的情况下更改buildArtefactType属性的值来更改已创建项目的构建工件类型。

使用“构建属性”自动调整工具设置

 

工具链ant工具定义可以指定启用表达式，以根据Build Properties，其他选项，字符串等的值自动调整其默认设置。

下面是gcc编译器工具定义的片段，它根据“buildType”属性值指定动态选项值调整。

 

   <选项

      name = “％Option。POSIX。DEBUGLEVEL”

            ...

      valueType = “enumerated” >

...

      <启用

           type = “CONTAINER_ATTRIBUTE” < - 指定启用适用于option属性

           attribute = “value” < - 启用适用的属性名称

           value = “gnu.c.debugging.level.none” < - 在启用表达式为true的情况下分配给属性的值

           extensionAdjustment = “false” > < - 指定启用适用于非扩展（项目）元素

           <checkBuildProperty < - 计算构建属性值。如果属性值等于value属性中定义的值，则视为true

                property = “org.eclipse.cdt.build.core.buildType” < - 要检查的属性的属性id

                value = “org.eclipse.cdt.build.core.buildType.release” /> < - 期望的属性值id

      </ enablement> < - 如果buildType属性值为“release”，则调试级别设置为“none”

       <启用

           type = “CONTAINER_ATTRIBUTE”

           attribute = “value”

           value = “gnu.c.debugging.level.max”

           extensionAdjustment = “false” >

           <checkBuildProperty

                property = “org.eclipse.cdt.build.core.buildType”

      value = “org.eclipse.cdt.build.core.buildType.debug” />
