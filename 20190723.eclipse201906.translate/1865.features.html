编译器

Object Teams Compiler是一个基于Eclipse Compiler for Java的增量编译器。从版本2.5开始，编译器支持Java™8的所有功能。
语言定义
编译器实现OT / J 语言定义版本1.3.1（OTJLD）中定义的语言 。语言定义不断被修改以精确指定语言，包括各种角落案例和特征组合。
语言实施
编译器支持OT / J的所有功能 OTJLD V1.3.1 §§1-7,9。支持连接点子语言（ OTJLD§8 ）不包括在内。

可以使用全局Preferences或项目特定Properties 配置编译器。
  • 将OT / J特定的编译器诊断配置为“忽略”，“信息”，“警告”或“错误”。
    使用@SuppressWarnings批注在源代码中也可以抑制在此属性表上配置的任何警告。
  • 范围关键字 OTJLD§A.0.1可以被禁用（默认启用）。
  • 可以在项目创建期间选择目标编织方案 。

奇才

  • 在Eclipse中使用Object Teams开发的第一步是创建一个 Object Teams Project 。向导将指导您完成该过程。它还会将对象团队运行时环境（ otre.jar ）添加到项目的类路径中。
  • 完成后，您可以开始添加类 - 无论是常规类，还是团队或角色类 - 向导将帮助您正确创建类。

编辑

  • 大纲视图支持编辑对象团队源代码，显示已编辑源文件的结构。这包括对象团队特定的构造， 团队 （ ）， 角色 （ ）和方法绑定 （ ， ， ， ）。
  • 编辑器突出显示像“ team ”这样的对象团队关键字，与突出显示标准Java关键字的方式相同。
  • 引用其他代码元素的所有构造都是可导航的。例如，您可以导航super（） ， tsuper（） ， base（）调用，以及callin和callout绑定 。team package声明中的名称允许您从角色文件导航（ OTJLD§1.2.5 ）给其附属团队。
  • 使用playedBy绑定一个或多个角色的类使用playedBy进行注释（ ），允许导航到角色声明。类似地，基类中与callin绑定绑定的方法使用标记进行注释（ ），允许导航到方法绑定声明。
    由于此功能在大型工作空间中可能消耗资源，因此全局对象团队首选项页面允许完全禁用callin标记的生成。
  • 完全支持语义突出显示（复选框“标记出现”）。
  • 代码完成（Ctrl-space）得到广泛支持。例如，可以使用完成来创建标注绑定。
  • 内容辅助为许多OT / J特定错误和警告提供快速修复（Ctrl-1）。此外，还提供了一些特定OT / J语言结构的模板 - 即with{}和within ( Expression ){} 。

绑定编辑器

  • 支持使用基于表的图形用户界面轻松创建连接器定义。在此上下文中， 连接器是指从其超级团队继承结构（角色）和功能（方法）的团队，其主要目的是使用playedBy声明和callin / callout方法绑定绑定继承的结构和功能。 。
  • 可以通过使用类型选择对话框选择提供的角色类型和所需的基本类型来定义类型绑定（ playedBy ）。
  • 可以通过分别从提供的角色和基本方法列表中选择角色方法和基本方法来建立方法绑定（callin或callout）。
    • 在相应的上下文（callin或callout）中只能选择适当的绑定种类。
    • 除了提供的角色方法之外，还可以通过选择提供的基本方法来实现新的角色方法，从而创建标注绑定。
  • 通过在参数映射配置选项卡的相应文本字段中键入表达式，可以定义参数和结果映射。
可以从新团队向导或使用package explore的上下文菜单打开绑定编辑器（ Open Binding Editor ）。

重构

在支持OT / J代码的自动重构方面已经做了大量工作。以下重构考虑了特定于对象团队的关系（隐式角色继承，团队嵌套，角色基础绑定和方法绑定）。
  • 提取方法
  • 移动方法
  • 拉起适用于方法或字段
  • 按下适用于方法或字段
  • 重命名适用于：项目，源文件夹，包，类型，方法，字段。
    尝试重命名团队包时，系统会要求您重命名团队类。
此外，正在开发OT / J的特定重构。目前这些已实施：
  • 提取Callin
  • 内联Callin
  • 转到角色文件

结构观察者

包资源管理器
  • 提供对所有项目工件及其结构的访问，即源文件，使用的库（ jar文件）和其他文档。请注意，团队和角色类使用不同的图标进行注释（ 对于团队和对于角色）。（只要团队或角色可见，整个IDE中都会使用相同的图标。）
  • 如果在常规首选项/标签装饰下启用了“Java类型指示器”，则特定于OT的装饰也将应用于编译单元（文件）和团队包。
  • 通过打开树枝，您可以窥视例如类的结构。
  • 对于团队类，对角色文件有特殊支持（ OTJLD§1.2.5 ）：
    角色文件是单独的文件，可以在物理上显示（ ）作为（单独）团队包中的文件或逻辑上 （ ）作为封闭团队的成员类型。
  • 此外，Package Explorer为其他操作提供当前上下文。例如，在Package Explorer中选择团队时，Team-and Role-Wizards有一个预先设置的源文件夹，包和封闭类。
层次结构视图
  • 对象团队介绍新的类型层次结构。除了普通（“ extends ”）继承之外，还有一个额外的“ 隐式 ”角色继承，由团队层次结构中的角色名称相等性建立。与团队嵌套角色一起支持受控形式的多重继承。
  • 新的隐式继承已完全集成到标准JDT 层次结构视图中 。
Java结构比较
  • 在比较OT / J文件的版本时，Java Structure Compare也将正确呈现Object Teams元素。
  • 当选择callin替换绑定时，可以通过上下文菜单使用“Compare With> Bound Base Method”操作，这可以让您轻松比较角色和基础的方法实现。

搜索

搜索参考
  • Java搜索功能已扩展为包括对象团队代码的搜索结果，即callin和callout绑定以及playingBy-relationship。
调用层次结构
  • 此外，调用层次结构已被扩展，用于浏览通过方法绑定（callin和callout）的控制流。
  • OTDT引入了对字段访问（读，写，任何）和类实例化的调用层次结构的支持。从版本3.4开始，Eclipse已采用此功能。
  • 由于提升而可能导致角色创建的OT / J结构被调用层次结构视为调用角色的提升构造函数。

执行环境

方面编织
  • 所有面向方面的语言都有某种编织过程 ，其中方面代码与其他（ 基本 ）代码相结合。
  • 对象团队程序在应用程序启动时执行编织，即此时程序启动。为了做到这一点，有一个所谓的对象团队运行时环境（OTRE） ，它挂钩到类加载过程。
  • 所有这些都是使用标准的Eclipse“Run”功能透明地处理的。运行对象团队项目的任何主类都会自动包含OTRE（此功能已在OTDT 1.2.2中更改 ）。
编织方案
  • 从版本2.3开始，OTDT附带两个不同的编织器，具有下面提到的属性。由于OT / J编译器为相应的编织器执行一些准备，因此必须根据目标编织方案生成不同的字节代码格式。因此，从一种编织方案切换到另一种编织方案需要对所涉及的所有OT / J代码进行完全重新编译。
    • OTRE ：这是传统的加载时织布工。这个编织器已经成熟多年，但不幸的是，底层库BCEL不再被维护，因此无法处理来自最近Java版本的类文件。最新的Java 7可以解决这个问题，但Java 8类文件包含这个基于BCEL的编织器无法处理的几个元素。
    • OTDRE ：新的对象团队动态运行时环境已经开发，甚至可以实现运行时编织 。为了解决JVM的限制，这个weaver必须在每个weavable类中插入一些通用钩子。这个编织器对编织成Java 8类文件也很有意思，因为它基于库ASM ，已经存在支持Java-8的版本。
Aspect部署/激活
  • “运行配置”中的新“团队激活”选项卡允许实例化和激活团队，而无需修改程序的源代码。
  • 将方面代码编织到应用程序中的团队可以添加到程序中
    • 通过程序源代码中的显式引用或
    • 通过配置文件，作为VM或的参数提到
    • 通过前面提到的“团队激活”选项卡添加它们。
    编译时不需要这样的配置。
使用的技术
  • 将weaver集成到JVM中利用了Java 5的Java代理概念（ JPLIS ）。

OT /春分

从OTDT版本0.9.9（2006）开始，可以使用OT / J作为实现语言来开发Eclipse插件。
  • 使用插件org.eclipse.objectteams.otequinox ，插件（“aspect plugin”）可以声明它适应来自指定的其他插件（“基本插件”）的类。
  • 通过将Object Teams Runtime Environment与Equinox实现中的钩子集成，来自aspect插件的team类可以在加载时触发字节码方面编织到其基本插件中。因此，方面插件确实可以适应另一个插件的行为。这些绑定是使用上述插件org.eclipse.objectteams.otequinox的扩展点aspectBindings声明的。
  • OT / Equinox技术的初始客户端是一个小插件，它通过另外显示哪些插件正在由哪个方面插件进行调整来扩展Eclipse的“插件”对话框。此信息位于上述对话框的版本列中。这样，OTDT的用户可以始终检查给定插件是否按出厂时执行，或者是否已由另一个插件进行调整。
  • OTDT使用OT / Equinox的功能来实现自己的功能。该技术主要用于使JDT-UI识别对象团队，而无需复制源代码或修改JDT-UI的源代码。
  • 一个新的项目性质 Object Teams Plugin Project ，支持用OT / J编写的插件的开发。
  • 所有与Equinox相关的启动都会在运行/调试对话框中添加对OT / Equinox的支持。
  • 如果需要，可以通过在-vmargs行之后-vmargs下行添加到文件eclipse.ini来禁用加载时织-vmargs ：
    -Dot.equinox = FALSE
    。

调试

调试器已得到增强，可支持在源代码级别调试OT / J程序。可以设置断点并逐步执行对象团队代码。
单步执行代码
以下语言功能生成字节代码，标准Java调试器无法显示适当的源位置：
  • 角色文件（ OTJLD§1.2.5 ）
  • 隐性遗传（ OTJLD§1.3.1 ）
  • 标注绑定（ OTJLD§3 ）（包括标注覆盖（ OTJLD§3.1（e） ））
  • Callin绑定（ OTJLD§4 ）
OTDT-Debugger将所有这些结构生成的字节代码重新映射到适当的源位置。因此，单步执行程序的这些部分会完全隐藏用户的内部翻译。

调试器尚未完全支持以下功能：
  • 角色构造函数（参见 OTJLD§2.4 ）
  • 声明提升的参数（ OTJLD§2.3.2 - 提升代码尚未过滤掉）
设置断点
使用双击标尺或上下文菜单Toggle Breakpoint断点在OT / J-Code中设置断点。也支持在角色文件中设置断点。
新观点：“团队监控”
团队监督（ ）视图有助于调试与团队激活相关的问题（ OTJLD§5 ）。为此，它显示所有已知的团队实例，并将其激活状态显示为其中一个
  • 待用 ，
  • 活性 （全局或在调试视图中选择的线程），或
  • 含蓄地活跃 （暂时处于活动状态，因为团队或其中一个角色当前正在执行代码）。
此外，可以通过上下文菜单以交互方式（de）激活所选团队。
过滤变量
OT / J编译器和加载时织入器都会向代码添加一些内部变量。默认情况下，此类内部变量隐藏在Team Monitor以及标准变量视图中。如果需要，两个视图都可以配置为显示这些内部变量。
已知问题和限制
  • 尚不支持动态代码评估，请务必清除表达式视图。
  • 在编织方案OTDRE下逐步调试callin绑定还不能始终显示正确的源代码位置，并通过生成的调度代码显示更多的跳数。
