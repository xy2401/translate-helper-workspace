菜单操作

  • 加
    • 将工作树中存在的更改添加到git索引，也称为分段更改。
    • 将新创建的资源放在git版本控制之下（Git不会自动启动跟踪资源）。
    • 解决冲突 。
  • 应用补丁 - 应用补丁。
  • 假设不变 - 资源可以标记为“假设不变”。这意味着Git停止检查工作树文件是否有可能的修改，因此您需要手动取消设置该位以在更改工作树文件时告诉Git。菜单操作Team> Assume不变 ，可以打开此设置，并在菜单操作Team> No Assume不变的情况下切换回来。
  • 分支 ， 创建分支 - 签出分支或创建分支。
  • 更改凭据 - 更改获取或推送规范的登录凭据，凭据在Eclipse Secure Store中按URL存储。
  • 结帐 - 签出分支，标记 ， 提交或参考。
  • Cherry-pick - Cherry-选择一个提交到当前签出分支的顶端。
  • 清除凭据 - 清除提取或推送规范的登录凭据，凭据在Eclipse Secure Store中按URL存储。
  • 提交 - 提交更改。
  • 删除提取 - 删除提取规范。
  • 删除推送 - 删除推送规范。
  • 配置提取 - 配置提取规范。
  • 配置推送 - 配置推送规范。
  • 删除分支 - 删除分支。
  • 删除存储库 - 删除存储库。
  • 断开连接 - 从此项目中断开附加的Git Team Provider。git存储库仍然存在，但不再与Eclipse集成。
  • 忽略 - 将文件添加到.gitignore，以便git忽略它们。
  • 导入项目 - 将项目导入Eclipse工作台。
  • 合并 - 合并分支。
  • 合并工具 - 使用合并工具解决冲突。
  • 打开属性视图 - 查看和编辑存储库配置。
  • 拉 - 拉当前检出的本地分支跟踪的远程分支的更改。
  • Remote> Fetch From - 从远程存储库获取更改
  • 远程> 从Gerrit中获取 - 从Gerrit Code Review Server更改了Fetch
  • 远程> 推送 - 将更改推送到其他存储库
  • 远程> 配置从上游获取 - 配置上游以进行自动提取
  • 远程> 配置推送到上游 - 配置上游以进行自动推送
  • Rebase - 将分支重新定位到另一个分支。
  • 删除存储库 - 从存储库视图中删除存储库。
  • 重命名分支 - 重命名分支。
  • 重置 - 重置当前的HEAD，索引或工作树。
  • 在历史记录中显示 - 在历史记录视图中显示所选资源。
  • 在存储库视图中显示 - 在存储库视图中显示所选资源。
  • 切换到... - 切换到（也称为结帐）另一个分支或标记。
  • 同步 - 使本地和远程分支彼此同步。
  • 标记 - 创建，删除标记。
  • Untrack - 从git版本控制中删除资源。如果要从工作树中删除资源，请单击资源上下文菜单中的“ 删除 ”。

历史观

概观

Git版本控制下的资源历史视图是给定存储库中资源的以提交为中心的视图。它可用于执行以下任务：

  • 在Git版本控制下检查给定文件的更改历史记录（查看和比较存储库中此类文件的版本）
  • 使用不同的搜索条件搜索某个提交
  • 签出某个提交
  • 基于特定提交创建分支和标记
  • 根据特定提交中的更改创建修补程序
  • 将完整的存储库重置为某个提交
  • 设置quickdiff基线并将其重置为某个提交

  • 在资源管理器中的Git版本控制下的任何资源上右键单击团队>在历史记录中显示

打开视图后，您可以激活“ 使用选择链接”按钮，以使视图的输入自动与资源管理器中的选择保持同步。

使用历史记录视图

检查提交图

“提交图”区域是“历史记录视图”的主要部分。默认情况下，它显示当前已检出的提交及其所有祖先，即列表中的第一个条目是已检出的提交。以下图片用于解释历史视图的一些功能：

提交图中的每一行对应一个提交。分支，标签和HEAD可视化如下：

  • 局部分支的尖端显示为绿色矩形
  • 远程分支的提示显示为灰色矩形
  • 局部HEAD显示为白色矩形
  • 标签显示为黄色矩形

（我们的示例没有远程分支）。

左侧的行是实际的提交图，它显示了列表中提交的父子关系（每个提交至少有一个父级，但存储库中的第一次提交除外）。可以存在对应于分支操作的分支和连接，其对应于合并操作。在我们的示例中，在使用分支“beforeSplit”提交之后创建了一个分支“实验”，并且在“master”和“experimental”分支中同时更改了相同的文件。最后一次提交是合并提交，其中“实验”分支的内容与“主”分支合并。

可以通过标记提交并查看“修订注释”区域来检查确切的更改。在“修订注释”区域中向下滚动时，将显示更改的文本差异，在我们的示例中，它表示Project1 / f1 / file1.txt的内容已从“已修改”更改为“在主文件中修改”。当选择下一个提交（对应于“实验”分支）时，将显示类似的差异，表示该文件的内容从“已修改”更改为“在实验中修改”。最新的提交是将“experimental”合并为“master”的结果。因此，新提交有两个祖先，并且“主”和“实验”行再次连接。

显示和比较文件的版本

如果当前输入已经是一个文件，用鼠标右键单击打开上提交将打开对应于当前选择的提交文件内容的编辑器。如果所选提交中不存在该文件，则会显示错误消息。单击与工作树比较将打开比较编辑器，将当前所选提交的文件内容与工作空间中的文件内容进行比较。

通过双击提交也可以执行打开和比较工作树操作：如果“比较模式”工具栏按钮（见下文）已关闭，将执行与工作树的比较 ，否则打开 。

通过选择两个提交并右键单击相互比较，可以比较当前输入过滤的两个提交的内容。如果当前输入不是文件，则在树中有一个额外的菜单操作相互比较 。第一个操作打开Eclipse比较编辑器，第二个操作打开Git树比较视图 。

此外，可以选择任意数量的提交并右键单击“ 打开”以查看与所选提交相对应的文件的所有版本（每个版本将打开一个编辑器）。

如果当前输入不是文件，则Open不会有菜单操作。但是，可以双击“修订详细信息”区域中的条目。如果比较模式处于活动状态，将打开比较编辑器，显示在当前所选提交中双击文件的更改（即当前所选提交中文件内容与此提交的祖先的文件内容的差异）。如果比较模式未激活，则显示具有与当前所选提交相对应的文件内容的编辑器。

使用过滤器设置

可以使用相应的工具栏操作更改过滤器设置（请参见下文）。默认情况下，“资源”设置处于活动状态，即列表中仅显示包含当前输入更改的提交。如果当前输入不是文件，则显示所有提交，其中包含当前输入的任何子项的更改。

如果过滤器设置为“Resource”且当前输入是文件，则提交列表仅包含那些包含该文件更改的提交。这在分析该文件的历史记录时很有用。但是，在某些情况下，还可以看到其他不会更改实际文件的提交。例如，可能有趣的是查看文件中的给定更改是在其他提交之前或之后是否更改该文件本身。在我们的示例中，我们可能想知道给定的更改是在“之前”还是“之后”提交标记为“Project1”。通过将过滤器设置从“资源”更改为“存储库”，可以轻松完成：

视图工具栏中的下四个切换按钮控制如何根据当前输入过滤显示的提交：

与工作树比较

仅当当前输入是文件并且选择了单个提交时，此操作才可用。它将打开一个比较编辑器，将所选提交的文件内容与工作树中的文件内容进行比较。

相互比较

仅当当前输入是文件并且选择了两个提交时，此操作才可用。它将打开一个比较编辑器，将所选提交的文件内容相互比较。

打开

仅当当前输入是文件时，此操作才可用。它将为每个选定的提交打开一个编辑器，显示给定提交的文件内容。

查看

这将检出当前选定的提交。如果此提交存在分支，则检出分支，如果此提交存在多个分支，则将显示一个对话框，询问应检出哪个分支。如果提交没有分支，则将检出提交并且HEAD将分离 。

创建分支......

在当前选定的提交上创建分支。将显示一个对话框，询问分支名称以及是否应检出新创建的分支。

删除分支

如果当前所选提交存在一个未签出的分支，则将启用此操作。如果此提交中有一个分支（未检出），则此操作将立即删除此分支。如果存在多个这样的分支，将显示一个对话框，询问应删除哪些分支。如果“删除分支”上的提交变得无法访问，将显示一个确认对话框，以防止意外无法提交提交。

创建标签...

在当前选定的提交上创建标记。将显示一个对话框，询问标签名称和标签消息。

创建补丁......

在第一次提交存储库时，此操作不可用。它将创建一个补丁，其中包含与该提交的前任相比当前所选提交的更改。将显示一个对话框，询问是应将补丁创建为文件还是剪贴板，以及是否使用通用补丁格式的Git补丁格式。

樱桃挑选

在当前签出的提交之上应用所选提交引入的更改。

还原提交

通过在当前检出的提交之上创建新提交来还原所选提交引入的更改。

合并

将所选提交合并到当前签出的分支中。

重新基础

在所选提交的基础上重新绑定当前签出的分支。

重置>软/混合/硬

此操作会重置包含当前所选提交的当前输入的存储库。根据子菜单的选择，将执行软复位或混合复位。

Quickdiff>将Quickdiff Basline重置为HEAD
Quickdiff>将Quickdiff Basline重置为HEAD的第一个父级

这两个操作将存储库的quickdiff基线设置为HEAD或HEAD的父级。即使选择了多个提交，这些操作也始终可用。

Quickdiff>设为基线

仅当选择了单个提交时，此操作才可用;它会将存储库的quickdiff基线引导到选定的提交。

复制

将当前所选提交的ID或提交复制到剪贴板。

显示修订评论

切换“修订注释”区域的可见性。

显示修订详情

切换“修订详细信息”区域的可见性。

Git Staging View

此视图提供了git status的等效项，显示了在工作树中所做的更改。尚未传输到git索引的未分级更改显示在“ Unstaged Changes”窗格中，已经“添加”（暂存）到Git索引的更改显示在“ Staged Changes”窗格中。默认情况下，这些窗格以行布局显示，可以通过“ 列布局”选项将其更改为列布局。默认情况下，Staged-Unstaged Changes窗格显示文件的完整路径。可以通过Show File Names First选项配置它们，以首先显示文件名，然后显示文件所在的目录。

双击修改的文件以打开比较视图。如果从“未暂存的”窗格触发，则比较视图将显示尚未暂存的更改。从“暂存”窗格中触发时，它将显示已经分阶段的更改。要在编辑器中打开文件，请在文件的上下文菜单中使用“ 打开工作区版本”操作。

要暂存文件，请将其从“ Unstaged Changes”窗格拖到“ Staged Pages”窗格。或者，在“ Unstaged Changes”窗格中的文件上下文菜单中使用“ 添加到Git索引”操作。Git Index操作中的替换为文件将替换工作树中的选定文件。如果文件未暂存，则会重置该文件。如果它是暂存的，则工作树版本将替换为Git索引中的暂存版本。

要取消暂存文件，请将其从“ Staged Changes”窗格拖到“ Unstaged Changes”窗格。或者，使用文件上下文菜单中的“ 从Git索引中删除”操作。

提交操作将仅提交暂存的更改 - 类似于git commit在本机git中执行的操作。集成的提交消息编辑器允许编辑提交的提交消息。与提交对话框相反，登台视图可以在进行更改时保持打开状态。这允许递增地写入提交消息以及更改。正在编辑的提交消息与存储库相关联，暂存视图与之链接。它不会持久存储，如果暂存视图或Eclipse关闭，它将会丢失。

要提交，请在提交消息文本字段中按Ctrl + Enter （在Mac OS X上输入 Command + Enter ），或单击Commit或Commit and Push按钮。

Git网址

Git URL通常包括传输协议方案，远程服务器的地址和远程服务器内的存储库路径，对于某些验证协议，还包括用户ID。

EGit支持以下协议

  • file - 对存储库的直接文件系统访问。
  • git - 最有效的内置git协议（默认端口9418）。该协议不提供身份验证。通常用于对存储库的匿名读取访问。
  • ssh - Git over secure shell（SSH）协议。通常用于对存储库进行身份验证的写访问。
  • http - 超文本传输协议可以通过防火墙进行隧道传输。
  • https - 超文本传输协议安全可以通过防火墙进行隧道传输。
  • ftp - 文件传输协议
  • sftp - SSH文件传输协议

Git URL用于

  • 克隆存储库
  • 从另一个存储库中获取更改
  • 将更改推送到另一个存储库


Git参考

Git References也很快被称为Refs 。
他们包括

  • 分支机构
  • 远程跟踪分支机构
  • 标签

它们都以路径命名，使用'/'作为路径分隔符，并以“refs”开头。

  • 本地分支机构以“refs / heads /”开头
  • 远程跟踪分支以“refs / remotes /”开头。远程跟踪分支位于远程存储库中的代理分支，以便在没有与存储库的连接可用时（离线）也可以查询其在上次传输操作时的状态。
  • 标签以“refs / tags /”开头

只要缩写形式是唯一的，就可以缩写参考名称。
例如

  • “master”是“refs / heads / master”的缩写
  • “origin / master”是“refs / remotes / origin / master”的缩写
  • “v1.0.1”是“refs / tags / v1.0.1”的缩写

Refs还有一些“保留”名称，对某些场景很有用：

参考名称 备注
头 指向当前结帐提交
FETCH_HEAD 指向上次获取操作的结果
ORIG_HEAD 指向在合并或rebase操作开始之前签出的提交

有关Ref名称的完整列表以及如果多个引用具有相同的简写形式的优先顺序，请参阅git rev-parse的 “指定修订”部分。

Refspecs

获取和推送操作使用“refspec”来描述远程Ref和本地Ref之间的映射。从语义上讲，它们定义了本地分支或标记如何映射到远程存储库中的分支或标记。在本机git中，它们以<src>：<dst>格式的冒号组合，前面加上可选的加号，+表示强制更新。在EGit中，它们可以在Push Ref规范和Fetch Ref规范以及其他对话框中以表格形式显示和编辑。

RefSpec的“左手”侧称为源，“右侧”称为目的地。根据RefSpec是用于获取还是用于推送，源和目标的语义不同：对于Push RefSpec，源表示源存储库中的Ref，目标表示目标存储库中的Ref。

推Refspecs

Push RefSpec的典型示例可能是

HEAD：参考文献/头/主

这意味着当前签出的分支（由HEAD参考指示，请参阅Git引用 ）将被推送到远程存储库的主分支中。

获取Refspecs

Fetch RefSpec的典型示例可能是

裁判/头/ *：参/遥控器/产地/ *

这意味着远程存储库中的所有分支都将被提取到本地存储库的相应远程跟踪分支中。

遥控器

遥控器用于管理您从存储库中跟踪其分支的存储库（“遥控器”）。

在EGit遥控器定义时

  • 从另一个存储库克隆存储库，按照惯例，已创建新克隆的存储库的存储库名为“origin”。如果您更喜欢其他名称，则克隆向导允许指定该名称。
  • 在存储库视图中定义远程数据库

远程首先定义了您跟踪其分支的存储库的名称 ，这很重要，因为您可能希望跟踪来自不同存储库的分支，因此该名称有助于了解某个操作正在处理的存储库。此外，为给定的Remote指定的Refspecs定义了本地存储库中分支和标记到远程存储库中的分支和标记的映射 。您可能希望对入站或出站传输操作使用不同的映射，因此有编辑器可以定义EGit中可用的提取和推送配置。

Git Ignore

位于工作树中的.gitignore文件指定git故意不应跟踪的文件。它们只涉及git尚未跟踪的文件。为了忽略已经跟踪的文件中未提交的更改，请参阅假设未更改的操作 。

.gitignore文件中的每一行都定义了一个模式。Git检查忽略工作树层次结构中从最高到最低的模式。在较高级别.gitignore文件中定义的模式将被较低级别中定义的模式覆盖。对于在给定项目上工作的所有人都应忽略的文件通常包含在项目的存储库中，以便在团队中轻松共享。

模式格式定义：

  • 空行被忽略
  • 以＃开头的行作为注释
  • 可选的前缀！ 否定了这种模式。由匹配的先前模式排除的文件将再次包含在内。以斜杠结尾的模式仅匹配目录，但不匹配文件或符号链接。
  • 不包含斜杠的模式被视为与相对于.gitignore文件位置的路径匹配的shell glob模式
  • git将模式视为fnmatch（3）中定义的shell globs
  • 在模式中通配符不匹配/路径名
  • 前导斜杠与路径名的开头匹配

EGit Ignore 菜单操作将所选资源添加到资源的父目录中的.gitignore文件中。要输入其他忽略模式，请使用文本编辑器。

用于PDE构建的Git Fetch Factory

作为EGit的PDE工具的一部分， org.eclipse.egit.fetchfactory插件中包含了一个用于Git的PDE Build fetch工厂。

映射文件的文件格式： type @ id，[version] = GIT，args

其中args是以逗号分隔的键值对列表。

接受的参数包括：

  • tag * - 强制Git标记
  • repo * - 强制回购地点
  • path - 相对于repo的可选路径，指向元素（否则假设元素位于存储库根目录）
  • prebuilt - 可选的布尔值，指示路径指向存储库中的预构建包

获取是通过三个步骤实现的：

  • 存储库被克隆到本地光盘。如果它已经存在，则假定它先前已克隆，并且仅提取新的提交
  • 指定的标记将在本地克隆中签出
  • 路径的内容将复制到最终的构建位置
