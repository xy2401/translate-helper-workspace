构建嵌套的文档结构

由于插件为平台提供了功能，因此添加描述新功能的文档是很常见的。如何构建这些文档，以便用户看到一套连贯而完整的文档而不是许多个人贡献？目录定义提供了以自上而下和自下而上的方式构建文档的机制。

自上而下的筑巢

自上而下的嵌套是指定义主内容表的技术，该内容表引用所有其他包含的toc。自上而下的嵌套是将已知内容分解为更小块的便捷方法。使用自上而下的嵌套， link属性在内容表定义中用于引用链接的toc而不是提供href 。

<toc label =“在线帮助示例”topic =“html / book.html”> <topic label =“Concepts”> <link toc =“toc_Concepts.xml”/> </ topic> <topic label =“Tasks”> <link toc =“toc_Tasks.xml”/> </ topic> <topic label =“Reference”> <link toc =“toc_Ref.xml”/> </ topic> </ toc>

基本结构保持不变（概念，任务，参考），但各个toc可以自由发展。他们反过来可能会链接到其他子集。

自下而上的组合

自下而上的组合更灵活，因为它允许新的插件决定文档在toc结构中应该存在的位置。使用锚属性完成自下而上的组合。toc定义了命名的锚点，其他插件可以提供文档。在我们的示例中，我们可以添加锚点，以便插件可以在概念，任务和参考部分之间提供额外的材料。

<toc label =“在线帮助示例”topic =“html / book.html”> <topic label =“Concepts”> <link toc =“toc_Concepts.xml”/> < anchor id =“postConcepts”/> </ topic > <topic label =“Tasks”> <link toc =“toc_Tasks.xml”/> < anchor id =“postTasks”/> </ topic> <topic label =“Reference”> <link toc =“toc_Ref.xml” /> < anchor id =“postReference”/> </ topic> </ toc>

然后，其他插件可以从插件中为锚点做出贡献。这在定义toc时使用link_to属性完成。

<toc link_to =“../ com.example.helpexample / toc.xml＃postConcepts” label =“关于概念的最新信息”> <topic> ...</ topic> </ toc>
