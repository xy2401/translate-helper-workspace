团队支持逻辑模型集成

在描述团队对逻辑模型集成的支持时，有两个感兴趣的观点：

  • 存储库提供程序 ：存储库提供程序是本地工作空间与远程存储库之间的连接。从存储库提供程序的角度来看，逻辑模型支持的详细信息可以在逻辑模型集成的存储库路线图中找到。
  • 模型提供者 ：模型提供者是允许用户使用存储在本地工作空间中的资源中的模型元素的工具。有关模型提供者如何利用此支持的详细信息，请参阅逻辑模型集成的模型路线图 。

以下几点总结了Team逻辑模型支持所涵盖的功能。

  • 维护工作区一致性：直接对资源执行的操作可能会对持久存在或与这些资源相关联的模型元素产生不良副作用。客户端可以使用ResourceChangeValidator验证对资源的更改不会对模型产生不良副作用，而模型可以实现ModelProvider＃validateChange方法来验证资源更改。
  • 团队操作和装饰：通过使模型元素适应相应的IResource，始终可以在具有一对一关系的模型元素上显示团队操作和装饰。现在，通过使模型元素适应ResourceMapping，可以在具有更复杂的资源关系的模型元素上显示操作和装饰。
  • 模型元素的语义合并：如果模型元素和资源之间存在一对一的对应关系，则模型提供者可以通过将IStorageMerger与特定文件类型相关联来参与无头合并。对于更复杂的关系，模型提供程序可以将其ModelProvider调整为IResourceMappingMerger，以访问合并操作的完整内容。
  • 模型参与团队查看者：团队视图现在使用Common Navigator框架。通过扩展Common Navigator扩展点和Team扩展点，并提供内容提供者和标签提供者，模型提供者可以出现在团队视图中。通过一些额外的步骤，还可以为模型提供合并预览支持。
  • 远程发现：模型提供者可以通过使用Team ProjectSetCapability类参与远程发现，以从项目集条目中获取URI。然后，可以将此URI与Eclipse File System API一起使用以访问远程内容。
  • 模型历史记录：模型提供程序可以通过FileHistory API访问单个文件历史记录，并在自定义历史记录页面中显示模型历史记录，该历史记录页面将显示在历史记录视图中。
