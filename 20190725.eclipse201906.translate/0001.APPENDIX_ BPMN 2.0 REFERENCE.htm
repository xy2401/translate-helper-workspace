附录：BPMN 2.0参考

本附录包含BPMN2 Modeler中所有BPMN 2.0元素及其可视化表示的简要说明。还有对支持的图表类型的详细讨论。

连接

协会

关联用于将信息和工件与BPMN图形元素链接。文本注释和其他工件可以与图形元素相关联。关联时的箭头指示适当时的流动方向。

会话链接

会话链接用于连接与参与者 （ 池 ）的对话

数据协会

数据关联显示来自或来自活动的数据流。箭头用于指示数据流的方向。

数据关联用于在“ 数据对象” ，“过程变量”以及“ 活动” ，“ 进程 ”和“ 全局任务”的输入和输出之间移动数据。流程执行不会沿着数据关联流动，因此它们对流程的流程没有直接影响。从数据对象或过程数据输入中检索数据的目的是填充活动输入，然后将执行活动的输出值推回到数据对象或过程数据输出中 。

消息流

消息流用于显示两个参与者之间的消息流。协作图中的池用于表示两个参与者 。

顺序流程

次序流 用于显示其中 活动 将在过程或编排被执行 的顺序 。次序流 可任选地定义条件 表达 ，这表明控制将被向下传递的 顺序流 只有当 表达式 求值为真。当 序列流 的源 是 网关 或 活动 时，通常使用 此 表达式 。具有独占，包含或 复杂网关 或 活动 作为其源 的 序列流 也可以定义为“默认”。这样的 序列流 将具有标记以显示它是默认流。仅当 来自 活动 或 网关的 所有其他传出 顺序流 无效时（即，它们的条件 表达式 为假） ， 才会 使用默认 顺序流 。

活动

边界事件

边界 的 事件 被连接到 活动 的边界 并用于处理 可能的 活动 的执行期间已经导致 条件（ 事件定义 ）。

结束事件

顾名思义， 结束事件 表示进程将结束的位置。在 序列流方面 ， 结束事件 结束了流程的流程，因此，没有任何传出的 序列流， 并且没有 序列流 可以从 End Even t 连接 。结束事件 可以具有一个或多个触发器（ 事件定义 ），其被传递回一个调用或含有处理（如果有的话）。

中级捕获事件

的 中间钩事件 被用来处理某种条件（ 事件定义 ） 的 已经在过程中或在外部进程发生。

中级投掷赛事

中间体 throw事件 被用于某种条件（ 事件定义 ） 报告 给调用或含过程。接收过程的设计应使其准备好处理事件，无论是 启动事件，中间捕获事件 还是 边界事件。

开始活动

顾名思义， Start Event 指示特定进程的起始位置。在 序列流方面 ， 启动事件 启动流程的流程，因此，将不会有任何传入的 序列流， 并且没有 序列流 可以连接到 启动偶数 t。一个 启动事件 可具有一个或多个事件触发（ 事件定义 ），其导致要发起的进程。

事件定义

事件定义确定活动的行为。事件可能包含零个或多个事件定义 。

取消

仅在事务子流程中使用时才允许此类型的事件定义 。它用于回滚交易的影响。

补偿

补偿类似于取消事件 ，因为它用于逆转一个或多个活动的影响 。在这种情况下， 活动可能不包含在交易子流程中，因此需要单独补偿每个活动 。

条件

当条件变为真时，将触发此类事件 。事件定义包含条件表达式 。

错误

错误事件定义用于抛出或捕获错误 。Error有效负载与Event拥有的变量相关联。有关详细信息，请参阅下面的错误 。

升级

升级事件定义用于抛出或捕获升级 。Escalation有效负载与Event拥有的变量相关联。有关详细信息，请参阅下面的升级 。

链接

链接是一种连接流程的两个部分的机制。链接事件可用于创建循环情况或避免长序列流线。链接事件使用仅限于单个流程级别（即，它们无法将父流程与子流程链接 ）。配对中间事件也可用作“关闭页面连接器”，用于跨多个页面打印流程。它们还可以用作Process级别中的通用�GoTo�对象。

信息

消息 事件定义可用于发送或接收消息 。Message有效负载与Event拥有的变量相关联。有关详细信息 ，请参阅下面的消息 。

信号

信号事件定义用于投掷或捕获信号 。有关详细信息，请参阅下面的信号 。

终止

此类结束事件表示应立即结束流程中的所有活动 。这包括多实例的所有实例。流程在没有补偿或事件处理的情况下结束。

计时器

定时器事件定义充当基于特定时间日期或特定周期（例如，每周一上午9点）的延迟机制，可以设置将触发事件 。

网关

网关必须具有多个传入或多个传出序列流 （即，它必须合并或拆分流程）。Gateway Direction属性确定其行为;此属性可以是以下之一

         未指定 � 网关可能同时具有多个传入和传出的序列流

         混合 � 网关 必须同时具有多个传入和传出的序列流

         收敛 � 网关 必须具有多个传入的序列流 ，但可能只有一个传出的序列流

         发散 网关只能有一个进入顺序流 ，但必须有多个外出顺序流

复杂网关

复杂网关可用于建模复杂的同步行为。表达式用于描述精确的行为。例如，此Expression可以指定激活网关需要五个传入的序列流中的三个。网关采用的传出路径由传出顺序流上的条件确定，如包含网关的拆分行为。

独家门户

分散的专用网关 （决策）用于在流程流中创建备用路径。这基本上是一个过程的“道路转移点”。对于进程的给定实例，只能采用其中一个路径。可以将决策视为在流程中的特定点提出的问题。这个问题有一套明确的备选答案。每个答案都与条件表达式相关联，该表达式与网关的传出序列流相关联。

可以选择标识默认路径，以便在没有条件表达式求值为true的情况下进行。如果未指定缺省路径并且执行Process使得没有条件表达式计算为true，则会发生运行时异常。

融合专用 网关用于合并备用路径。

基于事件的网关

基于事件的网关表示流程中的分支点，其中网关之后的备用路径基于发生的事件，而不是使用流程数据评估表达式（与独占或包含网关一样 ）。特定事件（通常是收到消息 ）确定将采用的路径。基本上，该决定是由另一个参与者根据流程不可见的数据做出的，因此需要使用基于事件的网关 。

包容性网关

可以使用不同的包含网关 （包含决策）在流程流中创建替代路径以及并行路径。与Exclusive Gateway不同，将评估所有条件表达式。对一个条件表达式的真实评估不排除对其他条件表达式的评估。将遍历具有真实评估的所有序列流 。由于每个路径被认为是独立的，因此可以从零到全部采用路径的所有组合。但是，它应该设计成至少采用一条路径。

可以选择标识默认路径，以便在没有条件表达式求值为true的情况下进行。如果未指定缺省路径并且执行Process使得没有条件表达式计算为true，则会发生运行时异常。

融合的包含 网关用于合并替代路径和并行路径的组合。

并行网关

并行网关用于同步（组合）并行流并创建并行流。并行网关创建并行路径而不检查任何条件;每个传出的序列流在执行此网关时传递控制。对于传入流， 并行网关将在通过其传出序列流触发流之前等待所有传入流 。

活动

业务规则任务

业务规则任务为流程提供了一种机制，以便为业务规则引擎提供输入并获取业务规则引擎可能提供的计算结果。

编舞任务

编排任务是编排过程中的原子活动 。它代表一个交互，可能是两个参与者之间的一个或两个消息交换。

 

 

 

手动任务

手动任务是不由任何业务流程引擎管理的任务。在业务流程引擎不跟踪此类任务的开始和完成的意义上，它可以被视为非托管任务。这种情况的一个例子可以是基于纸张的指令，用于电话技术人员在客户位置安装电话。

接收任务

接收任务是一项简单的任务，旨在等待消息从外部参与者 （相对于进程）到达。收到消息后 ，任务完成。

脚本任务

脚本任务由业务流程引擎执行建模器或实现者使用引擎可以解释的语言定义脚本当任务准备好启动时，引擎将执行脚本当脚本完成时，任务也将是完成。

发送任务

发送任务是一项简单的任务，旨在向外部参与者 （相对于流程）发送消息 。一旦发送消息 ，任务就完成了。

 

服务任务

服务任务是使用某种服务的任务，可以是Web服务或自动化应用程序。

服务任务定义了一个实现 ，它是用于实现服务的基础技术。有效值为“## unspecified”用于保持实现技术开放，“## WebService”用于Web服务技术或URI标识任何其他技术或协调协议。

任务

任务是包含在进程中的原子活动 。当流程中的工作未细分为更精细的流程细节时，将使用任务 。

 

用户任务

用户任务是典型的工作流任务，其中人类演员在软件应用程序的帮助下执行任务。任务的生命周期由软件组件（称为“任务管理器”）管理，通常在进程的上下文中执行。可以使用Implementation属性指定的不同技术来实现用户任务 。除了Web服务技术，可以使用任何技术。例如，可以使用WSHumanTask实现User Task ，方法是将Implementation设置为：http：//docs.oasis-open.org/ns/bpel4people/ws-humantask/protocol/200803�。

数据元素

本节描述了可视化和非可视化过程中活动可以操作的所有可能数据元素。

BPMN 2.0规范还提供了“数据状态”，它是数据元素中包含的数据的状态。这些状态的定义（例如，可能的值）和任何特定的语义都超出了BPMN 2.0规范的范围。因此，BPMN采用者可以使用此元素和BPMN可扩展性功能来定义自己的状态。

� 数据输入/输出

活动和流程通常需要数据才能执行。此外，它们可能在执行期间或作为执行结果生成数据。数据要求被捕获为数据输入和数据输出 。

 

输入和输出集

可以设计活动或过程 ，使得它可以用不同的输入数据集执行。这使得流程架构师在设计流程时具有更大的灵活性，因为在执行Activity时并非所有数据都可用，甚至不需要。例如，如果没有在运行时提供给Activity ，则可以导出或查找一些信息（例如城市名称）（例如通过城市邮政编码）。这类似于Java或C ++中方法重载的概念。

BPMN 2.0引入了输入集和输出集的概念来支持这一概念。基本上， 数据输入在组合在一起时（� 输入集 �），为活动定义了一组有效的数据输入。 组合在一起的数据输出 （� 输出集 �）定义活动完成时生成的数据项集。这些输入和输出集还允许您指定哪些数据项在输入时可选，或者可选择在输出中创建，以及哪些项是必需的或必需的。

输入和输出集也可以定义相互依赖关系; 输入集可以指示生成哪些输出集 ， 输出集可以指示生成输出数据所需的输入集 。

数据对象

数据对象是在流程流中建模数据的主要构造。数据对象具有明确定义的生命周期和结构。数据对象可以在流程图中多次出现，每个流程图都引用相同的数据对象实例。这些引用用于简化图表连接。

数据存储

数据存储为活动提供了一种机制，用于检索或更新将超出流程范围的存储信息。可以通过数据存储引用在流程中的一个或多个位置显示相同的数据存储 。

 

信息

消息表示两个参与者之间的通信内容。

 

错误

错误表示错误事件的内容或失败操作的错误 。

升级

Escalation标识了Process可能需要响应的业务情况。

信号

信号是在池中生成的触发器。它们通常用于进程内，跨进程池和进程图之间的广播通信。

资源

资源是参与活动的参与者或责任方。例如，这可以是特定个人，团体，组织角色或职位，或组织。在流程执行期间将资源分配给活动。

资源可能需要其他信息才能完成任务。例如，这可以是客户的联系信息，订单请求，发票等。该信息表示为资源参数 。在可执行的流程中 ， 资源参数必须采用可在拥有资源的活动范围内访问的数据项的形式。

变量（属性）

与数据对象一样 ， 属性是数据容器。但是，与数据对象不同，它们不会在图表上直观显示。只有进程 ， 活动和事件可能包含属性 。在编程语言的上下文中， 属性类似于“变量”，因为它们用于在数据移动过程中存储，转换和传送数据。属性与数据对象具有相同的属性，因为它们具有明确定义的结构，种类和基数。

BPMN2 Modeler使用术语�Variable�而不是�Property�，因为它对软件开发人员来说似乎更直观，尽管它们指的是相同的BPMN2模型元素。此外，因为术语�Property�无处不在，所以在讨论财产的 � 属性时有时会引起混淆。

数据类型（ItemDefinition）

BPMN元素（如数据对象和消息 ）表示在流程流中处理的项目。描述这些项的一组特征在BPMN 2.0中称为项定义 。但是，由于BPMN2 Modeler主要是为软件工程师设计的，因此它使用更直观的名称�DataType�。

项目定义可以是“物理”，例如车辆的机械部件，也可以是“信息”，例如车辆机械部件的目录。这被称为�ItemCond�。如果项目是“信息”，则还必须在“ 项目定义”中提供其数据结构。

也可以通过设置�iscollection�标志来定义项目集合。没有关于集合中项目的排序的假设。

集装箱

特设子流程

Ad-Hoc子流程包含任意数量的嵌入式内部活动，并且与典型的流程路由相比，旨在以更灵活的顺序执行。包含的活动按顺序或并行执行，它们可以按照仅通过指定的顺序流 ， 网关和数据连接进行约束的顺序执行多次。

通话活动

“ 调用活动”标识进程中使用全局进程或全局任务的点。Call Activity充当在执行中调用全局Process或Global Task的包装器。激活调用活动会导致将控制权转移到被调用的全局流程或全局任务 。

呼叫编排

Call Choreography识别Process中使用全局编排或全局编排任务的点。Call Choreography作为一个占位符，用于包含它正在调用的Choreography元素。

 

车道

Lane是进程内的子分区（通常在池中 ），用于组织池中的 活动并对其进行分类。车道通常用于内部角色（例如，经理，员工），系统（例如，企业应用），内部部门（例如，运输，财务）等。此外， 车道可以嵌套或定义在矩阵。例如，公司部门可能有一组外部车道 ，然后是每个部门内部角色的内部车道 。

池

池是协作或编排中的参与者的图形表示，并且可以是特定的合作伙伴实体 （例如，公司）或可以是更一般的合作伙伴角色 （例如，买方，卖方或制造商）。池可以引用一个过程，但不是必需的，即它可以是一个黑框。

子编排

Sub-Choreography是一个复合活动 ，因为它具有被定义为其他活动流的细节，在本例中是一个编排。每个子编排涉及两个或更多参与者 。Sub-Choreography的名称和每个参与者都显示在构成图形符号的不同波段中。有两个或更多参与者乐队和一个子流程名称乐队。

子流程

子流程是一个活动，其内部详细信息已使用“ 活动” ，“ 网关” ，“ 事件 ”和“ 序列流”建模。子流程定义了一个上下文范围，可用于属性可见性，事务范围，异常处理，事件处理或补偿。子流程可以处于隐藏其详细信息的折叠视图中，也可以位于展开视图中，该视图在包含它的流程视图中显示其详细信息。

如果将Sub-Process声明为事件处理程序（设置了“由Event�标记触发”标志），则它不是正常处理流程的一部分，必须按如下方式配置：

         它可能没有任何传入或传出的序列流 。

         它必须只有一个Start Event触发器，它具有以下一个或多个事件定义 ：

Ø     信息

Ø     错误

Ø     升级

Ø     赔偿金

Ø     条件

Ø     信号

触发事件子流程时，父流程可能会产生两种后果：

1。     父进程可以被中断

2。     父进程可以继续工作（不中断）

这取决于所使用的启动事件是否设置了“中断”标志。

交易

事务是一个以原子方式执行的专用子流程 ，也就是说，它所包含的所有活动要么成功执行完成，要么回滚它们的组合效果（主要是数据）。

交易必须指定交易 协议和用于提交或取消交易的方法 。应将协议设置为特定于技术的URI，例如，http：//schemas.xmlsoap.org/ws/2004/10/wsat for WSAtomicTransaction，或http://docs.oasis-open.org/ws-tx/ wsba / 2006/06 /适用于WS-BusinessActivity的AtomicOutcome。

文物

会话

对话是对话（协作）图的原子元素。它表示基于概念和/或相关键组合在一起的一组消息 流 。对话将涉及两个或更多参与者 。

组

Group对象是一个Artifact ，它提供了一种可视化机制，可以非正式地对图表元素进行分组。Group不是Activity或任何Flow对象 ，因此无法连接到Sequence Flows或Message Flows 。此外， 组不被池和泳道的限制约束。这意味着组可以跨越池的边界延伸以包围图元素，通常用于标识分布式企业对企业事务中存在的活动 。组通常用于突出显示图表的某些部分，不会影响流程的流程。

图表类型

作为为您的组织或客户实施BPM系统的软件开发人员，您可能主要关注于定义用户角色，工作活动，流程逻辑，异常处理，确保所有数据结构都已正确定义，并且所有端点是正确的，依此类推。有时感觉你太靠近森林看树了。

构成OMG的聪明人理解业务流程可能非常复杂，并且没有任何一种类型的图表可以捕获完全理解大型企业内部工作所需的所有细节，更不用说一个组织与其他组织交互的方式。商业世界。这就是为什么规范定义了几种类型的图表，它们呈现了业务流程的不同视图。本节讨论这些图表类型以及它们的使用方式。

流程图

这是“框和箭头”流程图类型的图表，定义了它们的排序（“箭头”），决策分支（“钻石”）等活动（“框”）。此类图表通常表示组织的私有过程，即组织内部如何工作的描述。虽然该图表可能显示来自或离开本组织的外部世界的信息（以“消息”的形式），但此类图表并非旨在显示不同组织之间的交互。

协作图

此类图表显示了两个或多个流程之间的交互，通常由不同方或组织拥有。这些流程由“池”表示，并且与流程图一样，每个池可以包含一个或多个流程。协作图与Process Diagrams类似，因为它们描述了组织内部活动的流程;不同之处在于，Process Diagram用于描述单个流程，Collaboration Diagrams显示多个流程以及它们之间的接口点。

编排图

编排图主要关注业务流程中的参与者（“池”）以及它们之间交换的信息，而不是正在执行的工作的编排。编排图可以被视为两个或更多组织之间的商业合同。

这是一个编排图，显示了买家和在线零售商之间的互动。每个交换都由一个圆角矩形表示，称为编排任务;它们之间交换的信息显示为表示从发起参与者发送的消息的信封。
