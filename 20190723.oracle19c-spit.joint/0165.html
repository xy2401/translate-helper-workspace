3.14 SET

目的

使用SET命令控制作业或会话中的RMAN行为。使用CONFIGURE配置跨会话保持的选项。

先决条件

您可以在RMAN提示符下或在RUN块中使用SET命令。在RMAN提示符下使用时， SET所做的更改将一直持续到您退出RMAN客户端（请参阅setRmanOption ）。在RUN块内部使用时， SET所做的更改将持续到RUN块结束或下一个更改相同属性值的SET命令（请参阅setRunOption ）。

setRmanOption

语法元素 描述

备份文件

指定Data Guard环境中备份的可见性。默认情况下，磁盘备份仅对与备份关联的数据库可见。关联由数据库的DB_UNIQUE_NAME确定。

使在主数据库或备用数据库上创建的磁盘备份仅对创建这些备份的数据库可用。

可以访问

使主数据库或备用数据库上创建的备份可供DB_UNIQUE_NAME与创建备份的数据库不同的数据库访问。

COMPRESSION ALGORITHM ' algorithm_name '

指定备份集的压缩算法。此命令将覆盖当前RMAN会话的当前CONFIGURE COMPRESSION ALGORITHM设置。

您可以通过为algorithm_name指定BASIC来配置不需要高级压缩选项的基本压缩级别。

如果您已启用高级压缩选项，则可以从以下压缩级别中进行选择：

  • HIGH -最适合在较慢的网络上进行备份，其中限制因素是网络速度

  • MEDIUM -适用于大多数环境。压缩比和速度的良好组合

  • LOW - 对备份吞吐量的影响最小，适用于CPU资源是限制因素的环境。

注意：压缩比通常从增加LOW至HIGH ，具有潜在的消耗更多的CPU资源的折衷。

由于各种压缩级别的性能取决于数据库中的数据性质，网络配置，系统资源以及计算机系统的类型及其功能，因此无法获得普遍适用的性能统计信息。在确定哪个级别最佳时，必须考虑系统与CPU的带宽和CPU的实际速度之间的平衡程度。 Oracle建议您对环境中的数据运行具有不同压缩级别的测试。根据您的环境，网络流量特征（工作负载）和数据集选择压缩级别。这是确保备份集压缩级别能够满足组织的性能要求和任何适用的服务级别协议的唯一方法。

注意： V$RMAN_COMPRESSION_ALGORITHM描述了支持的算法。

另请参阅： $RMAN_COMPRESSION_ALGORITHM Oracle数据库参考条目。
OPTIMIZE FOR LOAD {TRUE|FALSE}

指定Oracle在请求压缩备份时是否执行预压缩块处理。TRUE是默认值， FALSE启用预压缩处理。默认行为是不执行预压缩块处理。这样的处理可能消耗额外的CPU资源，并且对于包含所有最初加载的数据的块不需要，并且从未成为单行插入和删除的主题。指定FALSE使用额外的CPU资源来执行预压缩块处理，该处理由内部块清理和碎片整理组成，可以提高二进制压缩级别。

另请参阅： Oracle数据库备份和恢复用户指南，以了解有关此选项的更多信息。

AS OF RELEASE 'version number'

注意：如果从使用不同密码创建的一组备份进行还原， 请在SET DECRYPTION命令中指定所有必需的密码。RMAN会自动为每个备份集使用正确的密码。

另请参阅： “ 备份集的加密 ”

ECHO {OFF |上}

控制RMAN命令是否出现在消息日志中。从命令文件中读取命令时，RMAN会自动将它们回显到消息日志中。从标准输入读取命令时，默认情况下RMAN不会将这些命令回显到消息日志。要强制RMAN回显命令，请在运行命令文件之前运行SET ECHO ON命令。运行SET ECHO OFF以禁用对命令日志的回显。

重定向stdin和stdout时，该命令很有用。例如，在UNIX中，您可以通过以下方式重定向RMAN的输入和输出：

％rman TARGET / <in_file> out_file

通过包括SET ECHO ON在in_file ，您启用包含在命令in_file是可见out_file 。

ARCHIVELOG DESTINATION TO ' log_archive_dest '

覆盖了在后续RESTORE和RECOVER命令期间为已还原的归档重做日志文件形成名称时，目标数据库中的LOG_ARCHIVE_DEST_1初始化参数。RMAN将日志还原到' log_archive_dest '指定的目标。

BACKUP COPIES integer

指定信道创建的每个备份片的副本的数目： 1 ， 2 ， 3 ，或4 （参见实施例3-58 ）。

RMAN可以将备份双工复制到磁盘或磁带，但不能同时将备份复制到磁带和磁盘。备份到磁带时，请确保副本数量不超过可用磁带设备的数量。此外，如果BACKUP COPIES大于1，则必须在目标数据库上启用BACKUP_TAPE_IO_SLAVES初始化参数。

SET BACKUP COPIES命令影响在SET BACKUP COPIES （但不是之前）之后发出的RUN块中的所有BACKUP命令，并且在明确禁用或更改之前有效。SET BACKUP COPIES命令仅影响BACKUP命令，但不适用于BACKUP AS COPY命令。

SET BACKUP COPIES命令影响会话中分配的所有通道。优先顺序如下，列表中的设置更高，覆盖列表中较低的设置：

  1. 备份副本

  2. SET BACKUP COPIES

  3. CONFIGURE ...备份副本

备份片段的名称取决于BACKUP命令中的FORMAT子句。您最多可以指定四个FORMAT字符串。仅当BACKUP COPIES ， SET BACKUP COPIES或CONFIGURE ...时，RMAN才使用第二个，第三个和第四个值CONFIGURE ...BACKUP COPIES生效。选择用于每个备份片段的格式时，RMAN使用副本1的第一个格式值，副本2的第二个格式值，依此类推。如果格式值的数量超过副本数，则不使用额外的格式。如果格式值的数量小于副本的数量，RMAN将重用格式值，从第一个开始。

注意：在快速恢复区域中创建文件时， BACKUP COPIES选项无效。备份到快速恢复区域不能双工。

注意：磁盘上的控制文件自动备份是一种特殊情况，从不双工：RMAN始终只写一个副本。

MAXCORRUPT FOR DATAFILE datafileSpec TO integer

设置数据库在指定数据文件或数据文件组中允许的先前未检测到的块损坏的数量限制。默认限制为零，这意味着RMAN不会容忍任何损坏的块。

SET MAXCORRUPT命令指定备份作业期间数据文件中允许的物理和逻辑损坏的总数。如果为数据文件检测到的物理和逻辑损坏总和不超过其MAXCORRUPT设置，则BACKUP命令完成。如果存在超过MAXCORRUPT损坏的块，则RMAN终止而不创建输出文件。

无论是否超出MAXCORRUPT限制，RMAN都会使用它找到的任何损坏块范围填充V$DATABASE_BLOCK_CORRUPTION视图。但是，在MAXCORRUPT+1损坏块后终止备份或恢复作业，因此在这种情况下，RMAN仅记录MAXCORRUPT+1损坏。不记录超出备份作业终止点的任何块损坏。

注意：如果指定CHECK LOGICAL ，则MAXCORRUPT限制适用于检测到的逻辑和物理损坏的总和。否则， MAXCORRUPT仅适用于物理块损坏的数量。

13.1什么是动态SQL？

大多数数据库应用程序执行特定的工例如，一个简单的程序可能会提示用户输入员工编号，然后更新EMP和DEPT表中的行。在这种情况下，您知道在预编译时UPDATE语句的组成。也就是说，您知道可能更改哪些表，为每个表和列定义的约束，可能更新的列以及每列的数据类型。

但是，某些应用程序必须在运行时接受（或构建）和处理各种SQL语句。例如，通用报表编写者必须为其生成的各种报表构建不同的SELECT语句。在这种情况下，语句的构成在运行时才会被识别。这些陈述可以并且可能会从执行变为执行。它们恰当地称为动态 SQL语句。

与静态SQL语句不同，动态SQL语句未嵌入源程序中。相反，它们存储在运行时输入或由程序构建的字符串中。它们可以交互式输入或从文件中读取。

13.2动态SQL的优缺点

接受和处理动态定义的SQL语句的主机程序比普通的嵌入式SQL程序更通用。可以使用对SQL知之甚少或根本不了解的用户的输入以交互方式构建动态SQL语句。

例如，您的程序可能只是提示用户在SELECT，UPDATE或DELETE语句的WHERE子句中使用搜索条件。更复杂的程序可能允许用户从列出SQL操作，表和视图名称，列名称等的菜单中进行选择。因此，动态SQL允许您编写高度灵活的应用程序。

但是，一些动态查询需要复杂的编码，特殊数据结构的使用以及更多的运行时处理。虽然您可能没有注意到增加的处理时间，但除非您完全理解动态SQL概念和方法，否则您可能会发现编码很困难。

13.3何时使用动态SQL

实际上，静态SQL几乎可以满足您的所有编程需求。仅在需要开放式灵活性时才使用动态SQL。在预编译时未知以下项目之一时建议使用它：

  • SQL语句的文本（命令，子句等）

  • 主机变量的数量

  • 主机变量的数据类型

  • 对数据库对象的引用，例如列，索引，序列，表，用户名和视图

13.4动态SQL语句的要求

要表示动态SQL语句，字符串必须包含有效SQL语句的文本，但不包含EXEC SQL子句，语句终止符或以下任何嵌入式SQL命令：

  • 分配

  • 关

  • 宣布

  • 描述

  • 执行

  • 取

  • 自由

  • 得到

  • 包括

  • 打开

  • 准备

  • 组

  • 每当

在大多数情况下，字符串可以包含虚拟主机变量。它们在SQL语句中保存实际主机变量的位置。因为虚拟主机变量只是占位符，所以不要声明它们，并且可以将它们命名为任何您喜欢的名称。例如，Oracle不区分以下两个字符串： 

'从EMP删除MGR =：mgr_number AND JOB =：job_title''从EMP删除MGR =：m AND JOB =：j'

13.5如何处理动态SQL语句

通常，应用程序会提示用户输入SQL语句的文本以及语句中使用的主机变量的值。然后Oracle解析SQL语句以确保它符合语法规则。

接下来，Oracle 将主机变量绑定到SQL语句。也就是说，Oracle获取主机变量的地址，以便它可以读取或写入它们的值。

然后Oracle 执行 SQL语句。也就是说，Oracle执行SQL语句请求的内容，例如从表中删除行。

可以使用主机变量的新值重复执行SQL语句。

此方法允许程序接受或构建动态SQL语句，然后使用PREPARE和EXECUTE命令对其进行处理。SQL语句不能是查询。必须在预编译时知道输入主机变量的占位符数和输入主机变量的数据类型。例如，以下主机字符串属于以下类别：

此方法允许程序接受或构建动态查询，然后使用带有DECLARE，OPEN，FETCH和CLOSE游标命令的PREPARE命令处理它。必须在预编译时知道选择列表项的数量，输入主机变量的占位符数以及输入主机变量的数据类型。例如，以下主机字符串符合条件：

13.6.4方法4

此方法允许程序接受或构建动态SQL语句，然后使用描述符对其进行处理。在运行时之前，选择列表项的数量，输入主机变量的占位符数以及输入主机变量的数据类型可能是未知的。例如，以下主机字符串属于以下类别： 

'INSERT INTO EMP（<unknown>）VALUES（<unknown>）''SELECT <unknown> FROM EMP WHERE DEPTNO = 20'

13.6.5指南

使用所有四种方法，必须将动态SQL语句存储在字符串中，该字符串必须是主变量或带引号的文字。将SQL语句存储在字符串中时，省略关键字EXEC SQL和';'声明终结者。

每个后续方法对应用程序施加的约束较少，但编码起来更加困难。通常，使用最简单的方法。但是，如果方法1将重复执行动态SQL语句，请使用方法2来避免每次执行的重新分析。

方法4提供了最大的灵活性，但需要复杂的编码和对动态SQL概念的充分理解。通常，仅当您不能使用方法1,2或3时才使用方法4。

13.6.5.1关于避免常见错误

如果使用命令行选项DBMS = V6_CHAR进行预编译，请在存储SQL语句之前对数组进行空白填充。这样，你就清除了无关紧要的角色。当您将数组重用于不同的SQL语句时，这一点尤为重要。通常，在存储SQL语句之前始终初始化（或重新初始化）主机字符串。不要空终止主机字符串。Oracle不会将null终止符识别为字符串结尾的标记。相反，Oracle将其视为SQL语句的一部分。

如果使用命令行选项DBMS = V8进行预编译，请确保在执行PREPARE或EXECUTE IMMEDIATE语句之前字符串为空终止。

无论DBMS的值如何，如果使用VARCHAR变量来存储动态SQL语句，请确保在执行PREPARE或EXECUTE IMMEDIATE语句之前正确设置（或重置）VARCHAR的长度。

13.8使用方法2

方法1在一步中做了什么，方法2做了两个。动态SQL语句不能是查询，首先是PREPAREd（命名和解析），然后是EXECUTEd。

使用方法2，SQL语句可以包含输入主机变量和指示符变量的占位符。您可以一次PREPARE SQL语句，然后使用不同的主机变量值重复执行它。此外，你不必 rePREPARE后COMMIT或ROLLBACK（除非你注销并重新连接）的SQL语句。

您可以使用方法4将EXECUTE用于非查询。

PREPARE语句的语法如下：

EXEC SQL PREPARE statement_name FROM {：host_string |字符串字面量 };

PREPARE解析SQL语句并为其命名。

statement_name是预编译器使用的标识符， 而不是主机或程序变量，不应在Declare Section中声明。它只是指定要执行的PREPAREd语句。

EXECUTE语句的语法是

EXEC SQL EXECUTE statement_name [USING host_variable_list];

其中host_variable_list代表以下语法：

：host_variable1 [：indicator1] [，host_variable2 [：indicator2]，...]

EXECUTE使用为每个输入主机变量提供的值执行已解析的SQL语句。

13.8.1使用条款

当SQL语句为EXECUTEd时，USING子句中的输入主机变量将替换PREPAREd动态SQL语句中的相应占位符。

PREPAREd动态SQL语句中的每个占位符都必须与USING子句中的不同主机变量相对应。因此，如果相同的占位符在PREPAREd语句中出现两次或多次，则每个外观必须对应于USING子句中的主变量。

占位符的名称不需要与主变量的名称匹配。但是，PREPAREd动态SQL语句中占位符的顺序必须与USING子句中相应主机变量的顺序匹配。

如果USING子句中的一个主变量是一个数组，则所有主变量都必须是数组。

要指定NULL，可以将指示符变量与USING子句中的主变量相关联。

13.9使用方法3

方法3类似于方法2，但将PREPARE语句与定义和操作游标所需的语句组合在一起。这允许您的程序接受和处理查询。实际上，如果动态SQL语句是查询，则必须使用方法3或4。

对于方法3，必须在预编译时知道查询选择列表中的列数和输入主机变量的占位符数。但是，在运行时之前不需要指定数据库对象（如表和列）的名称。数据库对象的名称不能是主机变量。还可以在运行时指定限制，分组和排序查询结果（例如WHERE，GROUP BY和ORDER BY）的子句。

使用方法3，您可以使用以下嵌入式SQL语句序列： 

PREPARE statement_name FROM {：host_string |字符串字面量 }; DECLARE cursor_name CURSOR FOR statement_name; OPEN cursor_name [USING host_variable_list]; FETCH cursor_name INTO host_variable_list; CLOSE cursor_name;

可滚动游标也可以与方法3一起使用。必须将以下嵌入式SQL语句序列用于可滚动游标。

PREPARE statement_name FROM {：host_string |字符串字面量 }; DECLARE cursor_name SCROLL CURSOR FOR statement_name; OPEN cursor_name [USING host_variable_list]; FETCH [FIRST | PRIOR | NEXT | LAST | CURRENT | RELATIVE fetch_offset | ABSOLUTE fetch_offset] cursor_name INTO host_variable_list; CLOSE cursor_name;

现在我们来看看每个陈述的作用。

13.9.1 PREPARE（动态SQL）

PREPARE解析动态SQL语句并为其命名。在以下示例中，PREPARE解析存储在字符串select_stmt中的查询，并为其指定名称sql_stmt ： 

char select_stmt [132] =“SELECT MGR，JOB FROM EMP WHERE SAL <：salary”; EXEC SQL PREPARE sql_stmt FROM：select_stmt;

通常，查询WHERE子句在运行时从终端输入或由应用程序生成。

标识符sql_stmt 不是主机或程序变量，但必须是唯一的。它指定一个特定的动态SQL语句。

以下陈述也是正确的：

EXEC SQL PREPARE sql_stmt FROM SELECT MGR，JOB FROM EMP WHERE SAL <：salary;

以下使用'％'通配符的prepare语句也是正确的：

EXEC SQL PREPARE S FROM select ename FROM test WHERE ename LIKE'SMIT％';

13.9.2 DECLARE（动态SQL）

DECLARE通过为游标指定名称并将其与特定查询相关联来定义游标。继续我们的示例，DECLARE定义了一个名为emp_cursor的游标并将其与sql_stmt相关联，如下所示： 

执行SQL DECLARE emp_cursor CURSOR FOR sql_stmt;

标识符sql_stmt和emp_cursor 不是主机或程序变量，但必须是唯一的。如果使用相同的语句名称声明两个游标，则预编译器会将两个游标名称视为同义。

我们可以定义一个名为emp_cursor的可滚动游标，并将其与sql_stmt关联，如下所示：

执行SQL DECLARE emp_cursor用于sql_stmt的SCROLL CURSOR;

例如，如果执行语句

EXEC SQL PREPARE sql_stmt FROM：select_stmt; EXEC SQL DECLARE emp_cursor FOR sql_stmt; EXEC SQL PREPARE sql_stmt FROM：delete_stmt; EXEC SQL DECLARE dept_cursor FOR sql_stmt;

当您打开emp_cursor时 ，您将处理存储在delete_stmt中的动态SQL语句，而不是存储在select_stmt中的动态SQL语句。

13.9.3 OPEN（动态SQL）

OPEN分配Oracle游标，绑定输入主机变量，并执行查询，识别其活动集。OPEN还将光标定位在活动集中的第一行，并将SQLCA中sqlerrd的第三个元素保留的行处理计数归零 。 USING子句中的输入主机变量替换PREPAREd动态SQL语句中的相应占位符。

在我们的示例中，OPEN分配emp_cursor并将主机变量salary分配给WHERE子句，如下所示：

EXEC SQL OPEN emp_cursor使用：salary;

13.9.4 FETCH（动态SQL）

FETCH从活动集返回一行，将选择列表中的列值分配给INTO子句中的相应主机变量，并将光标前进到下一行。如果没有更多行，FETCH将“无数据找到”Oracle错误代码返回给sqlca.sqlcode 。

在我们的示例中，FETCH从活动集返回一行，并将列MGR和JOB的值分配给主变量mgr_number和job_title ，如下所示：

EXEC SQL FETCH emp_cursor INTO：mgr_number，：job_title;

如果光标在SCROLL模式下声明，则可以使用各种FETCH方向模式随机访问结果集。

13.9.5 CLOSE（动态SQL）

CLOSE禁用光标。关闭游标后，您将无法再使用它。

在我们的示例中，CLOSE禁用emp_cursor ，如下所示：

EXEC SQL CLOSE emp_cursor;

13.10使用方法4

本节概述了Oracle Dynamic SQL方法4。Oracle Dynamic SQL方法4不支持对象类型，结果集，结构数组或LOB。

ANSI SQL支持所有数据类型。对所有新应用程序使用ANSI SQL。

您的程序无法使用方法3处理某种动态SQL语句。如果输入主机变量的选择列表项或占位符的数量在运行时之前是未知的，则程序必须使用描述符。描述符是程序和Oracle用于在动态SQL语句中保存变量的完整描述的内存区域。

回想一下，对于多行查询，您将FETCH选定的列值INTO到已声明的输出主机变量列表中。如果选择列表未知，则INTO子句无法在预编译时建立主机变量列表。例如，您知道以下查询返回两个列值：

SELECT ename，empno FROM emp WHERE deptno =：dept_number;

但是，如果您让用户定义选择列表，您可能不知道查询将返回多少列值。

13.10.1需要SQLDA

要处理这种动态查询，您的程序必须发出DESCRIBE SELECT LIST命令并声明一个名为SQL Descriptor Area（SQLDA）的数据结构。因为它包含查询选择列表中列的描述，所以此结构也称为选择描述符 。

同样，如果动态SQL语句包含输入主机变量的未知数量的占位符，则USING子句无法在预编译时建立主机变量列表。

要处理动态SQL语句，程序必须发出DESCRIBE BIND VARIABLES命令并声明另一种称为绑定描述符的SQLDA，以保存输入主机变量占位符的描述。（输入主变量也称为绑定变量 。）

如果您的程序有多个活动的SQL语句（例如，它可能打开了两个或更多个游标），则每个语句都必须有自己的SQLDA。但是，非并发游标可以重用SQLDA。程序中的SQLDA数量没有设置限制。

13.10.2 DESCRIBE声明

DESCRIBE初始化描述符以保存选择列表项或输入主变量的描述。

如果提供选择描述符，则DESCRIBE SELECT LIST语句将检查PREPAREd动态查询中的每个选择列表项，以确定其名称，数据类型，约束，长度，比例和精度。然后它将此信息存储在选择描述符中。

13.12关于使用PL / SQL

Pro * C / C ++预编译器将PL / SQL块视为单个SQL语句。因此，像SQL语句一样，PL / SQL块可以存储在字符串主变量或文字中。将PL / SQL块存储在字符串中时，省略关键字EXEC SQL EXECUTE，关键字END-EXEC和';'声明终结者。

但是，预编译器处理SQL和PL / SQL的方式有两点不同：

  • 预编译器将所有PL / SQL主变量视为输入主变量，无论它们是PL / SQL块内的输入还是输出主变量（或两者）。

  • 您无法从PL / SQL块进行FETCH，因为它可能包含任意数量的SQL语句。

13.12.1使用方法1

如果PL / SQL块不包含主机变量，则可以使用方法1以常规方式执行PL / SQL字符串。

13.12.2使用方法2

如果PL / SQL块包含已知数量的输入和输出主机变量，则可以使用方法2以常规方式PREPARE和EXECUTE PL / SQL字符串。

您必须将所有主变量放在USING子句中。当PL / SQL字符串为EXECUTEd时，USING子句中的主变量将替换PREPAREd字符串中的相应占位符。虽然预编译器将所有PL / SQL主机变量视为输入主机变量，但是值是正确分配的。输入（程序）值分配给输入主变量，输出（列）值分配给输出主变量。

PREPAREd PL / SQL字符串中的每个占位符都必须与USING子句中的主机变量相对应。因此，如果相同的占位符在PREPAREd字符串中出现两次或多次，则每个外观必须对应于USING子句中的主机变量。

13.12.3使用方法3

除了方法3允许FETCHing之外，方法2和3是相同的。由于您无法从PL / SQL块进行FETCH，因此只需使用方法2即可。

13.12.4使用Oracle方法4

如果PL / SQL块包含未知数量的输入或输出主机变量，则必须使用方法4。

要使用方法4，请为所有输入和输出主机变量设置一个绑定描述符。执行DESCRIBE BIND VARIABLES将有关输入和输出主机变量的信息存储在绑定描述符中。由于预编译器将所有PL / SQL主机变量视为输入主机变量，因此执行DESCRIBE SELECT LIST无效。

警告：

在动态SQL方法4中，您无法使用类型为“table”的参数将主机数组绑定到PL / SQL过程。

警告：

不要在将动态处理的PL / SQL块中使用ANSI样式的注释（ - - ），因为忽略了行尾字符。因此，ANSI样式的注释扩展到块的末尾，而不仅仅是行的末尾。相反，使用C风格的评论（/ * ...* /）。

15.1.1.2了解服务器结果缓存的工作原理

执行查询时，数据库将搜索高速缓存以确定结果是否存在于结果高速缓存中。如果结果存在，则数据库从内存中检索结果，而不是执行查询。如果未缓存结果，则数据库执行查询，将结果作为输出返回，并将结果存储在结果缓存中。

当用户重复执行查询和函数时，数据库将从缓存中检索行，从而缩短响应时间。当修改依赖数据库对象中的数据时，缓存结果将变为无效。

15.1.2客户端结果缓存概念

Oracle调用接口（OCI）客户端结果缓存是客户端进程内部的一个内存区域，用于缓存OCI应用程序的SQL查询结果集。此客户端缓存存在于每个客户端进程，并由进程内的所有会话共享。Oracle建议对只读或主要表的查询进行客户端结果缓存。

注意：

客户端结果缓存与服务器结果缓存不同，后者驻留在SGA中。启用客户端结果缓存后，可以将查询结果集缓存在客户端，服务器或两者上。即使禁用了服务器结果缓存，也可以启用客户端缓存。

15.1.2.1使用客户端结果缓存的好处

OCI驱动程序（如OCCI，JDBC OCI驱动程序和ODP.NET）支持客户端结果缓存。使用客户端结果缓存的性能优势包括：

  • 减少查询响应时间

    当重复执行查询时，应用程序直接从客户端缓存内存中检索结果，从而加快查询响应时间。

  • 更有效地使用数据库资源

    服务器往返的减少可以显着节省服务器资源的性能，例如服务器CPU和I / O.这些资源可以用于其他任务，从而使服务器更具可伸缩性。

  • 降低内存成本

    结果缓存使用客户端内存，这可能比服务器内存便宜。

15.1.2.2了解客户端结果缓存的工作原理

客户端结果缓存存储最外层查询的结果，这些查询是OCI应用程序定义的列。不缓存子查询和查询块。

  • 自动共享内存管理

    如果使用SGA_TARGET初始化参数管理共享池的大小，Oracle数据库SGA_TARGET参数值的0.50％分配给结果缓存。

  • 手动共享内存管理

    如果使用SHARED_POOL_SIZE初始化参数管理共享池的大小，则Oracle数据库会将1％的共享池大小分配给结果缓存。

注意：

Oracle数据库不会将超过75％的共享池分配给服务器结果缓存。

服务器结果缓存的大小会增加，直到达到最大大小。不缓存大于缓存中可用空间的查询结果。数据库采用最近最少使用（LRU）算法来老化缓存结果，但不会自动从服务器结果缓存中释放内存。

15.2.3配置结果缓存模式

结果缓存模式是一种数据库设置，用于确定哪些查询有资格在服务器和客户端结果缓存中存储结果集。如果查询符合高速缓存条件，则应用程序会检查结果高速缓存以确定查询结果集是否存在于高速缓存中。如果存在，则直接从结果缓存中检索结果。否则，数据库执行查询并将结果作为输出返回并将其存储在结果缓存中。Oracle建议对只读或主要读取数据库对象的查询进行结果缓存。

启用结果缓存后，数据库还会缓存调用非确定性PL / SQL函数的查询。当缓存调用此类函数的SELECT语句时，结果缓存会跟踪PL / SQL函数和数据库对象的数据依赖性。但是，如果函数使用未跟踪的数据（例如序列， SYSDATE ， SYS_CONTEXT和包变量），则在调用此函数的查询上使用结果缓存可能会产生过时的结果。在这方面，结果缓存的行为与缓存PL / SQL函数相同。因此，在选择启用结果缓存时，请始终考虑数据准确性和性能。

要设置结果缓存模式：

  • 设置值 RESULT_CACHE_MODE初始化参数，用于确定结果缓存的行为。

    您可以为实例（ ALTER SYSTEM ），会话（ ALTER SESSION ）或服务器参数文件设置此参数。

    表15-3说明了此参数的值。

表15-3 RESULT_CACHE_MODE参数的值

15.2.4结果缓存的要求

启用结果缓存并不能保证特定结果集将包含在服务器或客户端结果缓存中。要缓存结果，必须满足以下要求：

  • 阅读一致性要求

  • 查询参数要求

  • 结果缓存的限制

15.2.4.1读取一致性要求

要使快照可重用，它必须具有读取一致性。要使结果集符合缓存条件，必须至少满足下列条件之一：

  • 用于构建结果的读一致性快照必须检索数据的最新，已提交状态。

  • 查询指向使用闪回查询的显式时间点。

如果当前会话具有引用查询中的对象的活动事务，则此查询的结果不适用于缓存。

15.2.4.2查询参数要求

如果在查询等效且参数值相同时使用变量值对缓存结果进行参数化，则可以重用缓存结果。不同的值或绑定变量名称可能会导致缓存未命中。如果在查询中使用以下任何构造，则会对结果进行参数化：

  • 绑定变量

  • SQL函数DBTIMEZONE ， SESSIONTIMEZONE ， USERENV/SYS_CONTEXT （带常量变量）， UID和USER

  • NLS参数

15.2.4.3结果缓存的限制

当查询中包含以下对象或函数时，无法缓存结果：

  • SYS或SYSTEM模式中的临时表和表

  • 序列CURRVAL和NEXTVAL伪列

  • SQL函数CURRENT_DATE ， CURRENT_TIMESTAMP ， LOCAL_TIMESTAMP ， USERENV/SYS_CONTEXT （带有非常量变量）， SYS_GUID ， SYSDATE和SYS_TIMESTAMP

客户端结果缓存对结果缓存有其他限制。

注意：

结果缓存对以只读模式打开的Active Data Guard备用数据库不起作用。

15.3.1.1使用RESULT_CACHE提示

当结果缓存模式为MANUAL ， /*+ RESULT_CACHE */ hint指示数据库缓存查询块的结果，并在将来的执行中使用缓存的结果。

例15-3显示了使用RESULT_CACHE提示的查询。

例15-3使用RESULT_CACHE提示

SELECT / * + RESULT_CACHE * / prod_id，SUM（amount_sold）FROM sales GROUP BY prod_id ORDER BY prod_id;

在此示例中，查询指示数据库缓存sales表的查询行。

15.3.1.2使用NO_RESULT_CACHE提示

/*+ NO_RESULT_CACHE */ hint指示数据库不要将结果缓存在服务器或客户端结果缓存中。

示例15-4显示了使用NO_RESULT_CACHE提示的查询。

示例15-4使用NO_RESULT_CACHE提示

SELECT / * + NO_RESULT_CACHE * / prod_id，SUM（amount_sold）FROM sales GROUP BY prod_id ORDER BY prod_id;

在此示例中，查询指示数据库不要为sales表的查询缓存行。

15.3.1.3在视图中使用RESULT_CACHE提示

RESULT_CACHE提示仅适用于指定提示的查询块。如果仅在视图中指定提示，则仅缓存这些结果。视图缓存具有以下特征：

  • 视图必须是以下类型之一：

    • 标准视图（使用CREATE ...创建的视图CREATE ...VIEW声明）

    • SELECT语句的FROM子句中指定的内联视图

    • WITH子句创建的内联视图

  • 无法缓存具有相关列（对外部查询块的引用）的视图查询的结果。

  • 查询结果存储在服务器结果缓存中，而不是客户端结果缓存中。

  • 缓存视图未合并到其外部（或引用）查询块中。

    将RESULT_CACHE提示添加到内联视图会禁用外部查询和内联视图之间的优化，以最大限度地提高缓存结果的可重用性。

以下示例显示了内联视图view1的查询。 

SELECT * FROM（SELECT / * + RESULT_CACHE * / department_id，manager_id，count（*）count FROM hr.employees GROUP BY department_id，manager_id）view1 WHERE department_id = 30;

在此示例中，来自view1的SELECT语句是外部块，而来自employees的SELECT语句是内部块。由于RESULT_CACHE提示仅在内部块中指定，因此内部查询的结果存储在服务器结果缓存中，但不会缓存外部查询的结果。

假设同一会话运行视图view2的查询，如以下示例所示。 

with view2 AS（SELECT / * + RESULT_CACHE * / department_id，manager_id，count（*）count FROM hr.employees GROUP BY department_id，manager_id）SELECT * FROM view2 WHERE count BETWEEN 1和5;

在此示例中，因为RESULT_CACHE提示仅在WITH子句的查询块中指定，所以employees查询的结果有资格被缓存。因为这些结果是从第一个示例中的查询缓存的，所以第二个示例中WITH子句中的SELECT语句可以检索缓存的行。

  • 数据编码

  • 错误消息语言

  • 低级分配回调

通过使用符合条件的上下文使得统一成为可能。符合上下文意味着返回的上下文必须以xmlctx ;它可能在标准头之后有任何其他特定于实现的部分。

应用程序获取xmlctx ，它使用统一的DOM调用，所有这些调用都将xmlctx作为第一个参数。

全球化支持XDK for C组件

XDK for C解析器支持超过300个IANA字符集。

这些字符集包括由XDK for C支持的字符集中列出的字符集 。

使用字符集时的注意事项：

  • Oracle建议您使用Internet Assigned Numbers Authority（IANA）字符集名称与其他XML解析器进行互操作。

  • XML解析器只需要支持UTF-8和UTF-16，因此这些字符集更可取。

  • 默认输入编码（“incoding”）是UTF-8。如果输入文档的编码不是不言自明的（通过HTTP字符集，字节顺序标记（BOM），XMLDecl等），则假定默认输入编码。如果仅使用单字节字符集（如US-ASCII或任何ISO-8859字符集），Oracle建议您显式设置默认编码，因为单字节性能最快。标志XML_FLAG_FORCE_INCODING指定默认输入编码始终应用于输入文档，忽略任何BOM或XMLDecl。然而，诸如HTTP字符集之类的协议声明总是受到尊重。

  • 请仔细选择DOM和SAX的数据编码（“outcoding”）。单字节编码是最快的，但只能代表一组非常有限的字符。下一个最快的是Unicode（UTF-16），最慢的是多字节编码，如UTF-8。如果输入数据无法在没有丢失的情况下转换为输出，则会发生错误。为了获得最大效用，请使用基于Unicode的输出，因为Unicode可以表示任何字符。如果未指定outcoding，则默认为解析的第一个文档的编码。

6使用非结构化数据类型

某些非Oracle数据库本机的数据类型可以在Oracle数据库中表示为大对象（LOB）。您可以在PL / SQL和OCI for Data Cartridges中实现它们。

6.1磁带盒和非结构化数据类型概述

某些数据磁带必须处理大量oMultif原始二进制数据（如图形图像或声音波形）或字符数据（如文本或数字流）。Oracle支持大对象LOB来处理这些类型的数据。

  • 内部LOB 以优化空间和提供高效访问的方式存储在数据库表空间中。内部LOB参与服务器的事务模型。

    内部LOBs可以存储二进制数据（ BLOB ），单字节字符数据（ CLOB ）或固定宽度单字节或多字节字符数据（ NCLOB ）。NCLOB由对应于为Oracle数据库定义的国家字符集的字符数据组成。Oracle不支持变化的宽度字符数据。

  • 外部 LOBs作为BFILE的二进制数据存储在数据库表空间外的操作系统文件中。他们不能参与交易。

内部LOB和BFILE在处理大量数据方面提供了相当大的灵活性。

存储在LOB数据称为LOB的值 。对于Oracle服务器， LOB的值是非结构化的，无法查询。您必须以墨盒特定的方式解压缩和解释LOB的值。

可以使用Oracle调用接口，OCI或PL / SQL DBMS_LOB包来操作LOB 。您可以编写函数，包括可以包含LOB的对象类型的方法，来操作LOB的部分。

6.8 LOB和触发器

您无法在任何类型的触发器中写入LOB （ :old或:new值）。

在常规触发器中，您可以读取:old值，但无法读取:new值。在INSTEAD OF触发器中，您可以读取:old和:new值。

您不能在OF子句中指定LOB类型列，因为可以更新BFILE类型而无需更新定义触发器的基础表。

使用OCI函数或DBMS_LOB包更新对象列的LOB值或LOB属性不会触发在包含列或属性的表上定义的触发器。

6.9使用打开/关闭作为包围操作以获得高效性能

使用“ Open/Close功能可以指示一系列LOB操作的开始和结束，以便在调用“ Close功能时可以执行大规模操作，例如更新索引。这意味着当进行Open调用时，每次修改LOB时都不会更新索引，并且直到Close调用才会恢复此类更新。

您不必在Open/Close操作中包装所有LOB操作，但由于以下原因，代码块可能非常有价值：

  • 如果未将LOB操作包装在Open/Close调用中，则对LOB每次修改都会隐式打开和关闭LOB,从而触发所有触发器。如果你确实将LOB操作包装在一对Open...Close操作，然后不会为每个LOB修改触发触发器。而是在进行Close调用时触发一个触发器。同样，在Close调用之前，不会更新可扩展索引。这意味着LOB上的任何可扩展索引在Open...之间无效Open...Close电话。

  • 您必须仔细应用此技术，因为反映LOB更改的状态不会保存在“ Open和“ Close操作之间。当您调用Open ，Oracle不再跟踪LOB值的哪些部分被修改，也不再跟踪任何修改产生的LOB的旧值和新值。对于每个OCILob*或DBMS_LOB操作， LOB值仍然直接更新，并且通常的读取一致性机制仍然存在。您可能还希望更新LOB上的可扩展索引，因为LOB修改是因为可扩展LOB索引始终有效并且可以随时使用。

  • API使您可以确定LOB是否已打开。在所有情况下，开放性与LOB相关联，而不是定位器。定位器不保存任何状态信息。

6.9.1关于开/关操作的错误和限制

在关闭所有先前打开的LOB之前提交事务是错误的。在事务回滚时，所有仍处于打开状态的LOB都将被丢弃，这意味着它们不会被关闭，从而触发触发器。

Open/Close相同的LOB两次，使用不同的定位器或使用相同的定位器是错误的。关闭尚未打开的LOB是错误的。

例6-11假定loc1引用了一个打开的LOB ，并被赋值给loc2 。如果随后使用loc2修改LOB值，则修改将与loc1的修改分组。这意味着LOB管理器的状态中只有一个条目，而不是每个定位器的一个条目。当LOB关闭时，无论是通过loc1还是loc2 ，都会触发触发器，因此通过任一定位器对LOB进行的所有更新都将被提交。在LOB关闭后，如果用户尝试使用任一定位器修改LOB ，则操作执行隐式Open()和Close() ，作为Open() ... operation ...Close() 。请注意，仍然为每个定位器保持一致的读取。请记住，打开和关闭的是LOB ，而不是定位器。无论定位器的副本数量是多少， LOB的触发器在第一次Close()调用时只触发一次。

使用共享服务器时，客户端不直接与数据库服务器进程通信，数据库服务器进程是代表数据库处理客户端请求的数据库进程。相反，客户端请求被路由到一个或多个调度程序 。调度程序将客户端请求放在公共队列中。来自共享服务器进程池的空闲共享服务器从队列中获取并处理请求。这意味着一小部分服务器进程可以为大量客户端提供服务。

1.1.4.2数据传输优化的会话数据单元大小

在通过网络发送数据之前，Oracle Net会将数据缓冲并封装到会话数据单元（SDU）中 。当缓冲区已满，刷新或数据库服务器尝试读取数据时，Oracle Net会发送存储在此缓冲区中的数据。当传输大量数据或消息大小一致时，调整SDU缓冲区的大小可以提高性能，网络利用率或内存消耗。您可以在客户端，应用程序Web服务器和数据库上部署SDU。

调整应用程序以减少通过网络的往返次数是提高网络性能的最佳方法。如果这样做，那么也可以进行优化通过调整SDU的大小来传输数据。

修改SDU大小的注意事项

在以下情况下修改SDU大小：

  • 从服务器返回的数据被分段为单独的数据包。

  • 您所在的广域网（WAN）有很长的延迟。

  • 数据包大小始终相同。

  • 返回大量数据。

在以下情况下，请勿修改SDU大小：

  • 可以调整应用程序以避免相邻列中列出的延迟。

  • 您有一个高速网络，数据传输的影响可以忽略不计。

  • 您的请求会从服务器返回少量数据。

注意：

从Oracle Database 11g开始 ，Oracle Net Services优化了某些组件的批量数据传输，例如Oracle SecureFiles LOB和Oracle Data Guard重做传输服务。网络参数文件中指定的SDU大小限制不适用于这些批量数据传输。启用ASO选项或使用TLS传输时，批量数据传输优化不适用。

1.1.4.4套接字直接协议

Oracle Net Services为InfiniBand高速网络提供支持。InfiniBand是一种高带宽I / O架构，旨在提高CPU，服务器端设备和网络子系统之间的通信速度。Oracle Net Services提供对套接字直接协议（ SDP ）的支持。SDP是一种行业标准的有线协议，旨在用于InfiniBand网络对等体之间。

SDP通过消除数据的中间复制并将大部分消息传递负担从CPU转移到网络硬件上来减少TCP / IP的开销。结果是低延迟 ，增加带宽，高吞吐量连接，减少了专用于网络处理的CPU周期。

客户端（包括Oracle WebLogic Server或任何其他第三方中间层客户端）与Oracle Database 12 c之间的通信可以利用高速互连优势。Oracle WebLogic Server在其安装过程中包含Oracle TCP / IP支持。

安装在Oracle WebLogic Server服务器上的驱动程序透明地将TCP / IP支持转换为SDP支持。然后，SDP请求将发送到InfiniBand交换机，该交换机处理从Oracle WebLogic Server服务器到数据库服务器的请求并将其转发。

A.59 set_runtime_exec_credentials

PROCEDURE set_runtime_exec_credentials（dbuser VARCHAR2，osuser VARCHAR2，ospass VARCHAR2）

其中， dbuser是数据库用户的名称或模式名称和osuser ， ospass是OS帐户凭据。

将数据库用户/模式dbuser与osuser/ospass操作系统（OS）凭据对osuser/ospass 。此关联已加密并存储在SYS用户拥有的表中。一旦建立了新的有效关联，每个新的OS进程都由java.lang.分叉java.lang.由dbuser调用以运行OS命令的Runtime.exec方法或每个ProceessBuilder作为UID osuser运行，而不是作为Oracle进程的OS ID运行。也就是说，分叉进程的UID位设置为UID osuser 。

注意：

DBA和安全管理员可以使用此过程来加强部署到Oracle数据库的Java应用程序的安全性。通过指定权限较低的帐户，DBA可以根据需要限制生成的进程的权限和访问权限。您必须是SYS用户才能使用set_runtime_exec_credentials过程，否则会引发ORA-01031: insufficient privileges错误。创建新进程时，使用无效的帐户凭据会导致IOException 。

以下示例显示了如何使用此过程：

例1

以下命令将用户/架构DBUSER到凭证osuser / ospass ：

dbms_java.set_runtime_exec_credentials（'DBUSER'，'osuser'，'ospass'）;

例2

以下任一命令都DBUSER绑定DBUSER和凭证osuser / ospass ：

dbms_java.set_runtime_exec_credentials（'DBUSER'，''，''）;
dbms_java.set_runtime_exec_credentials（'DBUSER'，null，null）;

注意：

要使用set_runtime_exec_credentials过程，必须在oracle产品安装期间将Oracle jssu工具配置为setuid root ，否则process spawn via jssu failed...的process spawn via jssu failed... 在创建进程时可能会引发IOException。
