为独立服务器配置Oracle Grid Infrastructure的软件二进制文件

使用这些步骤为Oracle Restart的独立服务器安装配置和激活仅限软件的Oracle Grid Infrastructure。

使用Oracle自动存储管理进行配置

  1. 安装软件二进制文件。请参阅“ Oracle数据库安装指南 ”中的“为独立服务器安装Oracle Grid Infrastructure的软件二进制文件” 。

  2. 运行gridSetup.sh以启动Oracle Grid Infrastructure安装向导。

    请参阅“ Oracle数据库安装指南 ”中的“使用新数据库安装为独立服务器安装Oracle Grid Infrastructure ” 。

配置不使用Oracle自动存储管理

  1. 以root身份登录并使用以下语法运行位于Grid_home路径中的roothas.sh脚本：

    #cd Grid_home / crs / install＃。/ roothas.sh

    其中， Grid_home是Oracle Grid Infrastructure主目录。

    例如：

    #cd /u01/app/oracle/product/19.0.0/grid/crs/install#./roothas.sh

  2. 将目录更改为路径Grid_home/ oui/bin 。

  3. 以Oracle Restart软件所有者用户身份登录，并使用以下命令语法，其中Grid_home是独立服务器主目录的Oracle Grid Infrastructure的路径。

    $ ./runInstaller -updateNodeList ORACLE_HOME = Grid_home -defaultHomeName CLUSTER_NODES = CRS = TRUE

    例如：

    $ ./runInstaller -updateNodeList ORACLE_HOME = / u01 / app / oracle / product / 19.0.0 / grid -defaultHomeName CLUSTER_NODES = CRS = TRUE

  4. 使用SRVCTL实用程序和Oracle Network Configuration Assistant将侦听器添加到Oracle Restart配置。

注意：

此过程不适用于以前版本的Oracle Restart升级。

在其他群集节点上创建相同的数据库用户和组

Oracle软件所有者用户和Oracle Inventory，OSDBA和OSOPER组必须存在且在所有群集节点上都相同。

要在其他群集节点上创建用户和组，请在每个节点上重复以下过程：
只有在使用本地用户和组时，才必须完成以下过程。如果使用目录服务（如NIS）中定义的用户和组，则它们在每个群集节点上已相同。
  1. 以root身份登录节点。
  2. 输入类似于以下内容的命令以创建asmadmin ， asmdba ， backupdba ， dgdba ， kmdba ， asmoper ， racdba和oper组，如果未由Oracle预安装RPM或先前安装配置，则输入oinstall和dba组。

    使用-g选项为每个组指定正确的组ID。 

    #groupadd -g 54421 oinstall #groupadd -g 54322 dba #groupadd -g 54323 oper #groupadd -g 54324 backupdba #groupadd -g 54325 dgdba #groupadd -g 54326 kmdba #groupadd -g 54327 asmdba #groupadd -g 54328 asmoper #groupadd -g 54329 asmadmin＃groupadd -g 54330 racdba

    注意：

    在此示例中，您不需要使用UID和GID。如果组已存在，则使用groupmod命令在必要时进行修改。如果无法对节点上的特定组使用相同的组ID，请查看所有节点上的/etc/group文件，以标识每个节点上可用的组ID。然后，您必须将所有节点上的组ID更改为相同的组ID。
  3. 要创建Oracle Grid Infrastructure（ grid ）用户，请输入类似于以下内容的命令：
    #useradd -u 54322 -g oinstall -G asmadmin，asmdba，racdba grid

    • -u选项指定用户标识，该标识必须是您之前标识的用户标识。

    • -g选项指定Grid用户的主要组，该用户必须是Oracle Inventory组（OINSTALL），它授予OINSTALL系统特权。在此示例中，OINSTALL组是oinstall 。

    • -G选项指定辅助组。Grid用户必须是OSASM组（ asmadmin ）和ASM组的OSDBA（ asmdba ）的成员。

    注意：

    如果用户已存在，则使用usermod命令在必要时进行修改。如果您无法在每个节点上为该用户使用相同的用户ID，请查看所有节点上的/etc/passwd文件，以标识每个节点上可用的用户ID。然后，您必须在所有节点上为该用户指定该ID。
  4. 设置用户的密码。

    例如：

    #passwd grid
  5. 为每个用户完成用户环境配置任务。

配置Oracle自动存储管理群集文件系统

查看此信息以为Oracle RAC Oracle数据库主目录配置Oracle ACFS。

Oracle ACFS作为Oracle Grid Infrastructure 19c安装的一部分安装。

您还可以使用ASMCA创建ACFS的通用文件系统配置。

要为Oracle RAC数据库的Oracle数据库主目录配置Oracle ACFS，请执行以下操作：

  1. 为群集安装Oracle Grid Infrastructure。

  2. 将目录更改为Oracle Grid Infrastructure主目录。例如：

    $ cd /u01/app/19.0.0/grid

  3. 确保Oracle Grid Infrastructure安装所有者对您要使用的存储装入点具有读写权限。例如，如果要使用mountpoint /u02/acfsmounts/ ：

    $ ls -l / u02 / acfsmounts

  4. 以网格安装所有者身份启动Oracle ASM Configuration Assistant。例如：

    ./asmca

  5. “配置ASM：ASM磁盘组”页面显示您在安装期间创建的Oracle ASM磁盘组。单击“ ASM群集文件系统”选项卡。

  6. 在ASM Cluster File Systems页面上，右键单击Data disk，然后选择Create ACFS for Database Use 。

  7. 在“创建ACFS for Database”窗口中，输入以下信息：

    • 卷名称 ：输入数据库主目录的名称。该名称在您的企业中必须是唯一的。例如： dbase_01

    • 挂载点 ：输入挂载点的目录路径。例如： /u02/acfsmounts/dbase_01

      记下此挂载点以供将来参考。

    • 大小（GB） ：以千兆字节为单位输入您希望数据库所在的大小。默认值为12 GB，建议的最小大小。

    • 所有者名称 ：输入您计划用于安装数据库的Oracle数据库安装所有者的名称。例如： oracle1

    • 所有者组 ：输入您计划在安装数据库时提供其成员的OSDBA组。该组的成员将获得对数据库的SYSDBA权限的操作系统身份验证。例如： dba1

    选择“ 自动运行配置命令”以自动运行ASMCA配置命令。要使用此选项，您必须在ASMCA设置页面上提供根凭据。

    完成输入后单击“ 确定” 。

  8. 如果未选择自动运行配置命令，则以特权用户（ root ）身份运行Oracle ASM Configuration Assistant生成的脚本。在Oracle Clusterware环境中，该脚本将ACFS注册为Oracle Clusterware管理的资源。将ACFS作为资源注册有助于Oracle Clusterware在将ACFS用于Oracle RAC Oracle数据库主目录时以适当的顺序自动挂载ACFS。

  9. 在Oracle RAC安装期间，请确保您或安装Oracle RAC的DBA为Oracle主目录选择您在Mount Point字段中提供的安装点 （在前面的示例中为/u02/acfsmounts/dbase_01 ）。

将所有密钥添加到公共authorized_keys文件

要配置SSH，请将所有群集节点成员上生成的所有密钥复制到每个节点上相同的授权密钥文件中。

完成以下步骤：
  1. 在本地节点上，将目录更改为Oracle Grid Infrastructure所有者主目录中的.ssh目录（通常为grid或oracle ）。然后，使用以下命令将DSA密钥添加到authorized_keys文件：
    $ cat id_dsa.pub >> authorized_keys $ ls
    在.ssh目录中，您应该看到已创建的id_dsa.pub键和文件authorized_keys 。
  2. 在本地节点上，使用SCP（安全复制）或SFTP（安全FTP）将authorized_keys文件复制到远程节点上的oracle用户.ssh目录。以下示例是SCP在一个名为node2的节点上，其中包含Oracle Grid Infrastructure所有者grid ，其中grid用户路径为/home/grid ：
    [grid @ node1 .ssh] $ scp authorized_keys node2：/home/grid/.ssh/
    1. 系统会提示您接受DSA密钥。输入Yes，您会看到要复制到的节点已添加到known_hosts文件中。
    2. 出现提示时，提供网格用户的密码，该密码在群集中的所有节点上应该相同。authorized_keys文件被复制到远程节点。
    您的输出应类似于以下内容，其中xxx表示有效IP地址的一部分：
    [grid @ node1 .ssh] $ scp authorized_keys node2：/home/grid/.ssh/无法建立主机'node2（xxx.xxx.173.152）的真实性。DSA密钥指纹是7e：60：60：ae：40：40：d1：a6：f7：4e：zz：me：a7：48：ae：f6：7e。您确定要继续连接（是/否）吗？是警告：永久添加'node1，xxx.xxx.173.152'（dsa）到已知主机列表grid @ node2的密码：authorized_keys 100％828 7.5MB / s 00:00
  3. 使用SSH登录到您复制authorized_keys文件的节点。然后切换到.ssh目录，并使用cat命令将第二个节点的DSA密钥添加到authorized_keys文件，在提示输入密码时单击Enter ，以便设置无密码SSH：
    [grid @ node1 .ssh] $ ssh node2 [grid @ node2 grid] $ cd .ssh [grid @ node2 ssh] $ cat id_dsa.pub >> authorized_keys

4.6.1 UCP中语句缓存概述

语句缓存使得使用语句更有效。语句缓存通过缓存重复使用的可执行语句来提高性能，并使程序员无需显式重用预准备语句。语句缓存消除了由于重复创建游标，重复语句解析和创建而导致的开销，并减少了应用程序与数据库之间通信的开销。语句缓存和重用对应用程序是透明的。每个语句高速缓存都与物理连接相关联。也就是说，每个物理连接都有自己的语句缓存。

缓存语句的匹配条件如下：

  • 语句中的SQL字符串必须与缓存中的SQL字符串相同（区分大小写）。

  • 语句类型必须与缓存中的语句类型相同（ prepared或callable ）。

  • 语句生成的可滚动类型的结果集必须与缓存中的结果集相同（ forward-only或scrollable ）。

根据JDBC驱动程序供应商的不同，实现和启用语句缓存。本节中的说明特定于Oracle的JDBC驱动程序。可以通过在连接工厂上设置连接属性来配置对其他供应商的驱动程序的语句缓存。请参阅JDBC供应商的文档以确定是否支持语句高速缓存以及是否可以将其设置为连接属性。如果JDBC供应商支持，UCP确实支持JDBC 4.0（JDK16）API以启用语句池。

4.6.2在UCP中启用语句缓存

最大语句数属性指定要为每个连接缓存的语句数。该属性仅适用于Oracle JDBC驱动程序。如果未设置该属性，或者该属性设置为0 ，则禁用语句缓存。默认情况下，禁用语句缓存。启用语句高速缓存时，语句高速缓存与连接池维护的每个物理连接相关联。所有物理连接都不共享单个语句高速缓存。

以下示例演示了启用语句缓存：

pds.setMaxStatements（10）;

确定语句高速缓存大小

缓存大小应设置为应用程序向数据库发出的不同语句的数量。如果应用程序向数据库发出的语句数未知，请使用JDBC性能指标来帮助确定语句高速缓存大小。

语句缓存大小资源问题

每个连接都与其自己的语句缓存相关联。连接的语句缓存中保存的语句可能会保留在数据库资源上。打开的连接数可能与每个连接的高速缓存语句数相结合，可能超过数据库允许的打开游标数限制。可以通过减少缓存中允许的语句数量，或通过增加数据库允许的打开游标限制来避免此问题。

语义

身体

关于Oracle Inventory目录和安装

您指定为Oracle Inventory目录（oraInventory）的目录存储系统上安装的所有软件的清单。

服务器上的所有Oracle软件安装所有者都被授予OINSTALL权限以读取和写入此目录。如果您在服务器上安装了以前的Oracle软件，则其他Oracle软件安装会从/var/opt/oracle/oraInst.loc文件中检测此目录，并继续使用该Oracle Inventory。确保指定为OINSTALL组的组可用作所有计划的Oracle软件安装所有者的主要组。

如果您是第一次安装Oracle软件，则OUI会创建Oracle基础和中央库存，并使用以下优先级的信息创建Oracle库存：

  • 在为安装所有者用户帐户设置的ORACLE_BASE环境变量中指示的路径中

  • 在最佳灵活体系结构（OFA）路径中（u [01-99] /app/ owner ，其中owner是运行安装的用户帐户的名称），并且该用户帐户具有写入该路径的权限

  • 在用户主目录中的路径/app/ owner ， owner是运行安装的用户帐户的名称

例如：

如果您正在执行Oracle数据库安装，并且在安装之前将ORACLE_BASE for user oracle设置为路径/u01/app/oracle ，并为该路径授予oracle 755权限，那么Oracle Universal Installer会创建一个以上级别的Oracle Inventory目录路径ORACLE_BASE/../oraInventory中的ORACLE_BASE/../oraInventory ，因此Oracle Inventory路径为/u01/app/oraInventory 。Oracle Universal Installer在ORACLE_BASE路径中安装该软件。如果要为群集安装执行Oracle Grid Infrastructure，则在安装后将Grid安装路径更改为root所有权，并且Grid home软件位置应与Grid用户Oracle base位于不同的路径中。

如果创建OFA路径/u01 ，并授予oracle 755写入该路径的权限，则会在路径/u01/app/oraInventory创建Oracle Inventory目录，Oracle Universal Installer会创建路径/u01/app/oracle ，并将Oracle用户的ORACLE_BASE环境变量配置到该路径。如果您正在执行Oracle数据库安装，那么Oracle主目录将安装在Oracle基础之下。但是，如果要为群集安装Oracle Grid Infrastructure，请注意在安装后Grid Home的路径的所有权将更改为root ，并且Grid base和Grid home应位于不同的位置，例如/u01/app/19.0.0/grid网格主目录路径为/u01/app/19.0.0/grid ，网格基础为/u01/app/grid 。例如：

/u01/app/oracle ，由oracle:oinstall拥有oracle:oinstall
 /u01/app/oracle/product/19.0.0/dbhome_1/ ，归o racle:oinistall
 /u01/app/grid ，由grid:oinstall拥有
 /u01/app/19.0.0/grid ，由root拥有

如果既没有设置ORACLE_BASE，也没有创建符合OFA的路径，则Oracle Inventory目录放在执行安装的用户的主目录中，Oracle软件安装在路径/app/ owner ，其中owner是Oracle软件安装所有者。例如：

目的

TO_YMINTERVAL将其参数转换为INTERVAL MONTH TO YEAR数据类型的值。

对于参数，您可以指定任何计算结果为CHAR ， VARCHAR2 ， NCHAR或NVARCHAR2数据类型的字符串的表达式。

TO_YMINTERVAL接受以下两种格式之一的参数：

  • SQL间隔格式与SQL标准兼容（ISO / IEC 9075）

  • ISO持续时间格式与ISO 8601：2004标准兼容

在SQL格式中， years是0到999999999之间的整数， months是0到11之间的整数。格式元素之间允许有额外的空格。

在ISO格式中，年和月是0到999999999之间的整数。天， hours ， minutes ， seconds和frac_secs是非负整数，如果指定则被忽略。值中不允许空格。如果指定T ，则必须至少指定一个hours ， minutes或seconds值。

可选的DEFAULT return_value ON CONVERSION ERROR子句允许您指定在将参数转换为INTERVAL MONTH TO YEAR类型时发生错误时此函数返回的值。如果在计算参数时发生错误，则此子句无效。return_value可以是表达式或绑定变量，它必须求值为CHAR ， VARCHAR2 ， NCHAR或NVARCHAR2数据类型的字符串。它可以是SQL格式或ISO格式，也不必与函数参数格式相同。如果return_value无法转换为INTERVAL MONTH TO YEAR类型，则该函数返回错误。

跨平台传输数据

您可以跨平台传输数据。

跨平台传输数据的功能可用于：

  • 使数据库可以从一个平台迁移到另一个平台。

  • 为内容提供商提供更简单，更有效的方法来发布结构化数据并将其分发给在不同平台上运行Oracle数据库的客户。

  • 简化从数据仓库环境到数据集市的数据分发，数据集市通常在较小的平台上运行。

  • 假设您可以从这些平台访问存储系统并且平台都具有相同的字节顺序，从而在不同操作系统或平台上的Oracle数据库安装之间共享只读表空间，如以下各节所述。

跨平台数据传输支持许多（但不是全部）平台。您可以查询V$TRANSPORTABLE_PLATFORM视图以查看支持的平台，并确定每个平台的endian格式（字节顺序）。以下查询显示支持跨平台数据传输的平台：

如果源平台和目标平台具有相同的字节顺序，则数据将从源平台传输到目标平台，而无需任何数据转换。

如果源平台和目标平台具有不同的字节序，则必须将传输的数据转换为目标平台格式。您可以使用以下方法之一转换数据：

  • DBMS_FILE_TRANSFER包中的GET_FILE或PUT_FILE过程

    当您使用这些过程之一在源平台和目标平台之间移动数据文件时，每个数据文件中的每个块都将转换为目标平台的字节顺序。转换发生在目标平台上。

  • RMAN CONVERT命令

    在源或目标平台上运行RMAN CONVERT命令。此命令将传输的数据转换为目标平台格式。

    注意：

    具有还原段的数据文件不支持在不同的endian格式之间转换数据文件。

在将数据文件中的数据传输到不同平台之前，数据文件头必须标识它所属的平台。要在不同平台上的Oracle数据库安装之间传输只读表空间，请至少对数据文件进行一次读/写操作。

AutoUpgrade和Microsoft Windows ACL和CLI

在Microsoft Windows系统上运行AutoUpgrade时，Oracle建议使用访问控制列表（ACL）和命令行界面（CLI）的其他最佳实践。

自动升级和访问控制列表（ACL）

在Windows系统上使用AutoUpgrade时，设置自动化工具以使用Windows访问控制列表时会遇到困难。Oracle强烈建议您完成以下最佳实践过程：

  • 查看每个目标数据库的权限，以及这些权限如何与这些数据库的目录相关联，例如Oracle基本目录和oraInventory文件。

  • 有关NTFS文件系统的安装后配置任务，请参阅Oracle数据库管理员参考部分。

  • 查看Microsoft有关Windows PowerShell的文档。

  • 查看组ORA_DBA ， ORA_HOME_USERS和ORA_ASM组的权限。ORA_DBA组仅为Oracle数据库提供SYSDBA权限。Oracle ASM管理权限由组ASM DISKGROUPS成员控制。

  • 与POSIX系统上可用的权限相比， ORA_DBA组成员执行许多管理任务的权限有限。要使AutoUpgrade按预期运行，Oracle建议Microsoft Windows服务器上具有管理员权限的用户帐户还管理Oracle基本目录元素，例如跟踪，侦听器和配置。

  • 必须以Windows管理员用户身份运行AutoUpgrade。

  • 请参阅My Oracle Support说明1529702.1和1595375.1。

AutoUpgrade对数据库上运行的服务使用以下过程：

  • AutoUpgrade停止源数据库上的服务，并在目标数据库Oracle主目录上创建临时服务。

    如果需要还原，则会删除目标中的服务，并重新启动源中的服务。

  • 部署模式处理成功完成后，将删除目标中的服务。此时，DBA负责升级的Oracle数据库使用ORADIM创建服务。需要手动创建此服务，因为AutoUpgrade没有密码来获取创建ORADIM服务的权限。

AutoUpgrade和Windows命令行界面

使用Windows上的命令行界面，应用程序可以在等待将返回字符发送到控制台时停止响应。此行为可能会影响AutoUpgrade实用程序。原因是与QEM（快速编辑模式）相关的众所周知的Microsoft Windows控制台窗口特征。即使您禁用了“快速编辑”控制台模式，应用程序仍然可能会遇到此行为。

要避免程序等待响应，请在应用程序启动后按几次Enter键。这样做提供了在不等待终端响应的情况下帮助应用程序继续所需的终端输入。
