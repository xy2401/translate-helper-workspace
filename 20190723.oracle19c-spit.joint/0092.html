显式游标是指向私有SQL区域的命名指针，该区域存储用于处理特定查询或DML语句的信息 - 通常是返回或影响多行的语句。

在使用显式游标之前，必须声明并定义它。您可以先声明它（使用cursor_declaration ），然后在同一个块，子程序或包中使用cursor_definition定义它，或者同时声明和定义它（使用cursor_definition ）。

显式游标声明和定义也分别称为游标规范和游标体 。

注意：

包规范中声明的显式游标受包的AUTHID子句的影响。有关更多信息，请参阅“ CREATE PACKAGE语句 ” 。

5.2 MARK_FAILED

使用MARK_FAILED过程将索引的状态从LOADING更改为FAILED 。

在极少数情况下，如果CREATE INDEX或ALTER INDEX失败，索引可能会保留状态LOADING 。当索引处于LOADING状态时，将阻止使用RESUME INDEX进行恢复的任何尝试。对于这种情况，使用CTX_ADM.MARK_FAILED强制将状态从LOADING更改为FAILED以便您可以使用RESUME INDEX 。

您必须以CTXSYS身份登录才能运行CTX_ADM.MARK_FAILED 。

警告：

请谨慎使用CTX_ADM.MARK_FAILED 。它只应作为最后的手段使用，并且只有在没有其他会话触及索引时才会使用。通常，如果另一个会话正在使用CREATE或ALTER INDEX主动构建索引，则CTX_ADM.MARK_FAILED不会成功。但是，索引创建或更改可能包括CTX_ADM.MARK_FAILED可以成功的时间窗口，即使正在由另一个会话构建索引，也会将索引标记为失败。

CTX_ADM.MARK_FAILED适用于本地分区索引。但是，它会将所有分区的状态更改为FAILED 。因此，在使用CTX_ADM.MARK_FAILED后，应使用ALTER INDEX REBUILD PARTITION PARAMETERS ('RESUME') CTX_ADM.MARK_FAILED ALTER INDEX REBUILD PARTITION PARAMETERS ('RESUME')重建所有索引分区。如果在此操作之后运行ALTER INDEX PARAMETER ('RESUME') ，则Oracle会将索引分区状态重置为有效。Oracle不会重建在MARK_FAILED操作之前成功构建的索引分区。

使用时间有效性

时间有效性使您可以跟踪现实世界有效性的时间段。用户和应用程序可以为数据设置有效时间，并且可以按指定的有效时间或有效时间范围选择数据。

应用程序通常会记录数据库中记录的事实的有效性（或有效性），其中包含与业务管理相关的日期或时间戳。例如，人力资源（HR）应用中雇员的雇用日期（其确定保险行业的有效保险日期）是有效日期。此日期与员工记录在数据库中输入的日期或时间形成对比。前一时间属性（雇用日期）称为有效时间（VT），而后者（输入数据库的日期）称为交易时间（TT）。虽然有效时间通常由用户控制，但交易时间是系统管理的。

对于ILM，有效时间属性可以表示事实在商业世界中何时有效以及何时不存在。使用有效的时间属性，查询可以只显示当前有效的行，而不显示包含当前无效的事实的行，例如已关闭的订单或将来的租用。

有效时间时间建模不可或缺的概念包括：

  • 有效时间

    这是用户定义的时间表示。有效时间的示例包括项目开始和结束日期，以及员工雇用和终止日期。

  • 具有有效时间语义的表

    这些表具有用户定义时间的一个或多个维度，每个维度具有开始和结束。

  • 有效时间闪回查询

    这是使用有效时间维度进行as-of和版本查询的能力。

有效时间段由表定义中指定的两个日期时间列组成。您可以通过显式添加列来添加有效时间段，也可以自动创建列。可以在create table或alter table进程中添加有效时间段。

为了支持临时表查询的会话级可见性控制， DBMS_FLASHBACK_ARCHIVE PL / SQL包提供了ENABLE_AT_VALID_TIME过程。要执行该过程，您需要所需的系统和对象权限。

以下PL / SQL过程设置给定时间的有效时间可见性。

SQL> EXECUTE DBMS_FLASHBACK_ARCHIVE.enable_at_valid_time（'ASOF'，'31 -DEC-12 12.00.01 PM'）;

以下PL / SQL过程在会话级别的有效时间段内将时态数据的可见性设置为当前有效数据。

SQL> EXECUTE DBMS_FLASHBACK_ARCHIVE.enable_at_valid_time（'CURRENT'）;

以下过程将时态数据的可见性设置为完整表，这是默认的临时表可见性。

SQL> EXECUTE DBMS_FLASHBACK_ARCHIVE.enable_at_valid_time（'ALL'）;

ORA-12315：数据库链接类型对ALTER DATABASE语句无效

原因：您在ALTER DATABASE语句中指定的数据库链接名称不是ROM：链接。使用ALTER DATABASE语句装入或打开辅助数据库时，必须指定ROM：链接。

操作：使用有效的ROM重新发出ALTER DATABASE语句：指向要装入或打开的数据库的链接。如果不存在有效的ROM：链接，请使用CREATE DATABASE LINK命令创建一个。有关使用ROM：链接类型创建数据库链接的更多信息，请参阅“受信任的ORACLE RDBMS安全功能指南”。

ORA-12317：登录到数据库（链接名称string ）被拒绝

原因：此错误可能有多种原因。首先，如果辅助数据库中的用户名（和密码，如果使用数据库而不是操作系统身份验证）与主数据库中的用户名（和密码）不同，则会出现此错误。其次，如果辅助数据库中的用户名无效（尚未创建），则可能会出现此错误。第三，如果在数据库链接定义的连接字符串中指定的用户名/密码组合无效（未创建或密码无效），则可能会出现此错误。

操作：在第一种情况下，请确保辅助数据库包含与您在主数据库中使用的用户名（和密码，如果使用数据库身份验证）相同的用户名。通常，您应始终在Trusted ORACLE中使用操作系统身份验证（有关OS身份验证的优点的详细信息，请参阅“受信任的ORACLE RDBMS安全功能指南”）。在第二种情况下，请确保已创建辅助数据库中的用户名。在第三种情况下，请确保已在辅助数据库中创建了连接字符串中指定的用户名。

操作：您采取的操作取决于消息的原因：在第一种情况下，请确保在辅助数据库中的用户名被授予CREATE SESSION系统特权。在第二种情况下，请确保在辅助数据库中为数据库链接定义的连接字符串中指定的用户名授予CREATE SESSION系统特权。

5.2多级集合类型

多级集合类型是集合类型，其元素本身是直接或间接的另一种集合类型。

可能的多级集合类型是：

  • 嵌套表类型的嵌套表

  • varray类型的嵌套表

  • 嵌套表类型的Varray

  • Varray类型的Varray

  • 嵌套表或用户定义类型的varray，其属性是嵌套表或varray类型

与单级集合类型一样，多级集合类型：

  • 可以用作关系表中的列，也可以用作对象表中的对象属性。

  • 要求源和目标都具有相同的已声明数据类型以进行分配。

5.2.1多级集合类型的嵌套表存储表

要使用嵌套表的多级嵌套表集合，必须指定嵌套表存储子句。

嵌套表类型列或对象表属性需要存储表来存储其所有嵌套表的行，如“ 存储嵌套表的元素 ”中所述 。

使用嵌套表的多级嵌套表集合，必须为内部集合和外部嵌套表集指定嵌套表存储子句（ STORE AS ）。您必须拥有与集合中嵌套表级别一样多的嵌套表存储子句。

每个嵌套表存储表都包含一个可由NESTED_TABLE_ID引用的NESTED_TABLE_ID ，该列将存储表中的行NESTED_TABLE_ID父表中的关联行。父表本身就是一个嵌套表，它有两个系统提供的ID列：

  • 系统提供的ID列，可由NESTED_TABLE_ID引用，该列将其行键回到其父表中的行。

  • 系统提供的ID列，由嵌套表子NESTED_TABLE_ID中的NESTED_TABLE_ID列隐藏和引用。

如果未指定具有NESTED_TABLE_ID列的主键，则数据库会自动在NESTED_TABLE_ID列上创建b树索引以获得更好的性能。

17.4使用ALTER SESSION SET CONTAINER对OCI呼叫的限制

               

                  
描述具体的限制。

                  
ALTER SESSION SET CONTAINER语句可用于将OCI连接从一个可插拔数据库切换到另一个可插拔数据库。但是，使用ALTER SESSION SET CONTAINER语句在可插拔数据库之间切换的应用程序需要确保它们的使用与下面描述的OCI限制一致。
                  

                  
    
                     
  • 
                        
    对于OCI可迁移会话（例如在登录期间使用OCI_MIGRATE模式创建的会话），不允许使用ALTER SESSION SET CONTAINER语句，并且该组合会导致ORA-65135错误。
                        
    
                     
    • 
                     
  • 
                        
    OCI连接池（旧的OCI连接池API）不支持ALTER SESSION SET CONTAINER语句，并且该组合导致ORA-65135错误。
                        
    
                     
    • 
                     
  • 
                        
    与OCI会话切换（其中多个OCI用户句柄共享相同的OCI服务器句柄）不支持ALTER SESSION SET CONTAINER语句。
                        
    
                     
    • 
                     
  • 
                        
    如果客户端最初连接到具有EXTENDED MAX_STRING_SIZE设置的容器，然后在同一会话中切换到具有STANDARD MAX_STRING_SIZE设置的容器（使用ALTER SESSION SET CONTAINER语句），则后续的OCIStmtExecute()调用将导致如果尝试使用大小超过4000字节的任何绑定变量，则ORA-14697错误。

17.5使用ALTER SESSION SWITCH CONTAINER SWITCH SERVICE对OCI呼叫的限制

               

                  
描述使用ALTER SESSION SWITCH CONTAINER SWITCH SERVICE语句对OCI调用的限制。
                  

                  
从Oracle Database 12c第2版（12.2）开始，添加了ALTER SESSION SWITCH CONTAINER SWITCH SERVICE语句，如果新服务的配置与连接到原始服务时的配置不同，则可能导致服务属性发生更改。但是，OCI不会根据此声明的新设置更改其处理;相反，它将继续使用原始设置，如连接到原始服务时。例如，FAN开/关，TAF开/关设置等将保持与首次连接到原始服务时相同。这被认为是切换后的期望行为，因为ALTER SESSION SWITCH CONTAINER SWITCH SERVICE语句的正常使用情况是在多租户环境中，在切换后使用相同的应用程序。

4.2使用API创建更改前的SQL试用版

               

                  
本节介绍如何使用DBMS_SQLPA包构建更改前的性能数据。
                  

                  
在创建更改前的SQL试验之前，您需要创建SQL Performance Analyzer任务，如创建分析任务中所述 。
                  

                  
                  

                     
要创建更改前的SQL试用版：

                     
    
                        
  • 
                           
    打电话给使用以下参数执行EXECUTE_ANALYSIS_TASK过程：
    
                           
      
                              
    • 
                                 
      将task_name参数设置为要执行的SQL Performance Analyzer任务的名称。
                                 
      
                              
      • 
                              
    • 
                                 
      使用以下方法之一设置execution_type参数：
      
                                 
        
                                    
      • 
                                       
        设置为EXPLAIN PLAN ，为SQL调优集中的所有SQL语句生成执行计划，而不执行它们。
                                       
        
                                    
        • 
                                    
      • 
                                       
        设置为TEST EXECUTE （推荐）以执行SQL调优集中的所有语句并生成其执行计划和统计信息。指定TEST EXECUTE ，该过程将生成执行计划和执行统计信息。执行统计信息使SQL Performance Analyzer能够识别已改进或已退化的SQL语句。除了生成执行计划之外，收集执行统计信息还可以提高性能分析的准确性，但需要更长的时间。
                                       
        
                                    
        • 
                                    
      • 
                                       
        设置为CONVERT SQLSET以引用SQL调试集，以用于SQL试验的执行统计信息和计划。还应指定执行参数SQLSET_NAME和SQLSET_OWNER值。
                                       
        
                                    
        • 
                                 
      
                              
      • 
                              
    • 
                                 
      使用execution_name参数指定名称以标识execution_name 。如果未指定，则SQL Performance Analyzer会自动为任务执行生成名称。
                                 
      
                              
      • 
                              
    • 
                                 
      使用execution_params参数指定execution_params参数。execution_params参数被指定为指定执行的（ 名称 ， 值 ）对。例如，您可以设置以下执行参数：
      
                                 
        
                                    
      • 
                                       
        time_limit参数指定在超时之前处理SQL调优集中所有SQL语句的全局时间限制。
                                       
        
                                    
        • 
                                    
      • 
                                       
        local_time_limit参数指定在超时之前处理SQL调优集中的每个SQL语句的时间限制。
                                       
        
                                    
        • 
                                    
      • 
                                       
        要执行远程测试执行，请将DATABASE_LINK任务参数设置为连接到具有DBMS_SQLPA程序包的EXECUTE特权和测试系统的ADVISOR特权的用户的公共数据库链接的全局名称。
                                       
        
                                    
        • 
                                    
      • 
                                       
        要完全执行DML语句（包括获取行锁和修改行设置），请将EXECUTE_FULLDML参数设置为TRUE 。 SQL Performance Analyzer将在执行DML语句后发出回滚，以防止发生持久更改。此参数的默认值为FALSE ，它仅执行DML语句的查询部分而不修改数据。
                                       
        
                                    
        • 
                                    
      • 
                                       
        要在测试执行期间恢复相关捕获的init.ora设置，请将APPLY_CAPTURED_COMPILENV参数设置为TRUE 。这不是默认行为，因为通常在运行SQL试验以在更改环境时测试更改。但是，此方法可用于未更改init.ora设置的情况（例如创建索引）。远程SQL试验不支持此方法。
                                       
        
                                    
        • 
                                 
      
                              
      • 
                           
    
                           
    以下示例说明了在系统更改之前进行的函数调用：
    EXEC DBMS_SQLPA.EXECUTE_ANALYSIS_TASK（task_name =>'my_spa_task'， -  execution_type =>'TEST EXECUTE'， -  execution_name =>'my_exec_BEFORE_change'）;

3.21开关

         

         

            

               

                  
目的

                  
使用SWITCH命令执行以下任一操作：

                  
    
                     
  • 
                        
    将数据库，表空间或数据文件的文件名更新为可用于指定文件的最新映像副本
    
                     
    • 
                     
  • 
                        
    更新已发出SET NEWNAME命令的数据文件和临时文件的文件名
    
                     
    • 
                  

                  
SWITCH等同于SQL语句ALTER DATABASE RENAME FILE ：更新RMAN存储库中文件的名称，但数据库不会在操作系统级别重命名文件。
                  

               

               
               

                  
先决条件

                  
RMAN必须连接到目标数据库。切换表空间，数据文件或临时文件时，文件必须处于脱机状态。切换整个数据库时，不能打开数据库。

               

               
               

                  
使用说明

                  
SWITCH命令从恢复目录中删除数据文件副本的RMAN存储库记录，并将控制文件记录更新为状态DELETED 。

                  
如果RMAN连接到恢复目录，并且数据库正在使用从备份还原的控制文件，则SWITCH将使用恢复目录中已知但从控制文件中丢失的任何数据文件的记录更新控制文件。
                  

                  
执行SWITCH ...仅在RMAN提示下TO COPY 。仅在RUN块内使用不带TO COPY SWITCH 。
                  

               

               
               

                  
句法

                  
 switch :: =

                  

说明switch.eps的说明

                  
                  
（ datafileSpec :: = ）

                  
 switchFile :: =

                  

说明switchfile.eps的说明

                  
                  
（ datafileSpec :: = ， tempfileSpec :: = ）

               

               
               

                  
语义

                  
开关

                  
本子条款将数据库，表空间或数据文件的文件名切换为可用于指定文件的最新映像副本。通过执行此命令，可以避免从备份还原数据文件。执行SWITCH ...仅在RMAN提示下TO COPY 。

9.16 V $ SESS_TIME_MODEL

         

         

            

               
V$SESS_TIME_MODEL显示各种操作的会话累计时间。 报告的时间是经过的总时间或CPU时间（以微秒为单位）。任何定时操作都将缓冲最多5秒的时间数据。具体而言，这意味着如果定时操作（例如SQL执行）需要很长时间才能执行，则发布到此视图的数据最多会缺少5秒的操作累计时间。
               

               

                  
时间值是8字节整数，因此可以在换行前保存大约580,000年的时间。除非统计信息专门用于后台进程，否则后台处理时间不包括在统计值中。

能够为每个可插拔数据库创建密钥库

                  

                     
每个可插拔数据库（PDB）现在都可以拥有自己的密钥库，而不是只有一个密钥库用于整个容器数据库（CDB）。

                     
在以前的版本中，PDB与整个容器数据库（CDB）共享密钥库，其中包括CDB根和所有PDB。此密钥库存储了CDB的主加密密钥以及所有PDB。在此版本中，PDB可以像以前一样继续与整个CDB共享同一个密钥库，或者拥有单独的密钥库。这种设计在PDB之间提供了更大的隔离，因为每个独立的密钥库都可以独立管理。例如，每个密钥库都可以使用不同的密码进行保护。

                     
此功能的另一个优点是，它可以在多租户环境中由每个租户（PDB）执行独立的密钥管理操作，而不必在CDB根级别共享密钥库。此功能对多租户和非多租户环境都有好处，因为它提供了便于配置密钥库位置和密钥库类型的参数，从而无需编辑sqlnet.ora文件。
                     

                     
此功能提供以下新功能：

                     
    
                        
  • 
                           
    对于多租户环境，有以下两种模式：
    
                           
      
                              
    • 
                                 
      联合模式，其中密钥库和主加密密钥主要从CDB根管理，并且可以从联合模式PDB访问。在PDB中，可以仅为该PDB打开和关闭密钥库。您还可以为此密钥库创建特定于PDB的主加密密钥。
      
                              
      • 
                              
    • 
                                 
      隔离模式，其中密钥库和加密密钥在单个PDB中管理。这样，每个PDB可以独立配置自己的密钥库类型，并在配置后创建和管理该密钥库。
      
                              
      • 
                           
    
                           
    您可以混合使用这两种模式。例如，假设您有1个CDB和10个PDB。您可以在联合模式下运行其中3个PDB，其余7个在隔离模式下运行。根据您的环境和场地要求，此设计提供最高的灵活性。

      •

此功能特别有利于Oracle Fusion Applications软件即服务部署。您可以在自己的环境中创建这些密钥。如果要插入您信任的加密密钥，但稍后决定必须替换密钥，则可以在没有云提供程序干预的情况下插入另一个密钥。

从PGQL结果集创建顶点/边集

resultSet = g.queryPgql（“SELECT x MATCH（x）WHERE x.age> 24”）vertexSet = g.getVertices（new ResultSetVertexFilter（resultSet，“x”））
当在客户端 - 服务器模式下运行时，此操作在PGX服务器上执行，并且无需将结果集提取到客户端以创建集合。这为任何大小的结果集带来了性能优势，并且能够处理在客户端计算机上处理可能很麻烦的非常大的结果集。

OraclePgqlResultSet的弃用

                  

                     
从19c开始，是oracle.pg.rdbms.不推荐使用OraclePgqlResultSet接口，可能会在将来的发行版中将其删除。而是使用标准化的接口oracle.pgql.lang.ResultSet从PGQL结果集中检索值。
                     

                     
这两个接口之间的一个区别是oracle.pgql.lang.ResultSet不提供API来检索顶点和边缘对象。使用这些接口的现有代码应更改为项目ID，而不是OracleVertex和OracleEdge对象。
                     

                     
例如，更改投影顶点和边的代码片段：
OraclePropertyGraph opg = ...; OraclePgqlStatement ops = OraclePgqlExecutionFactory.createStatement（opg）; OraclePgqlResultSet rs = ops.executeQuery（“SELECT v，e MATCH（v） -  [e]  - >（）WHERE ...”，“”）; Iterator <OraclePgqlResult> itr = rs.getResults（）。iterator（）; OraclePgqlResult opr = itr.next（）; OracleVertex v = opr.getVertex（1）; OracleEdge e = opr.getEdge（2）;
以下内容：
OraclePropertyGraph opg = ...; OraclePgqlStatement ops = OraclePgqlExecutionFactory.createStatement（opg）; ResultSet rs = ops.executeQuery（“SELECT id（v）AS vid，id（e）AS eid MATCH（v） -  [e]  - >（）WHERE ...”，“”）; rs.next（）; OracleVertex v = opg.getVertex（rs.getLong（“vid”））; OracleEdge e = opg.getEdge（rs.getLong（“eid”））;
仅使用ResultSet API将确保在删除OraclePgqlResultSet时应用程序不会中断。

8.2.1升级前步骤

               

                  

                     
之前升级时，必须删除在其中创建的任何数据挖掘模型 Java和在中创建的任何挖掘活动 Oracle Data Miner Classic（早期版本的 Oracle Data Miner）。
                     

                     

                        
警告：

                        
在Oracle Database 12c中 ，Oracle Data Mining不支持Java API，Oracle Data Miner Classic无法针对Oracle Database 12 c运行。
                        

                     

                  

                  
               
8.2.1.1删除用Java创建的模型

                  

                     

                        
如果10 g或11 g数据库包含使用Java创建的模型，请在升级数据库之前使用DBMS_DATA_MINING.DROP_MODEL例程删除模型。
                        

                     

                     
                  

               
8.2.1.2删除Oracle Data Miner Classic中创建的挖掘活动

                  

                     
如果您的数据库包含挖掘活动在Oracle Data Miner Classic中，删除挖掘活动并在升级数据库之前删除存储库。跟着这些步骤：

                     
    
                        
  1. 
                           
    使用Data Miner Classic用户界面删除挖掘活动。
    
                        
    2. 
                        
  2. 
                           
    在SQL * Plus或SQL Developer中，删除这些表：
    DM4J $ ACTIVITIES DM4J $结果DM4J $ TRANSFORMS
    以及这些观点：
    DM4J $ MODEL_RESULTS_V DM4J $ RESULTS_STATE_V
    3. 
                     

                     
完成这些步骤后，数据库中的任何模式中都必须没有前缀为DM4J$表或视图。
                     

                  

               

            
8.2.2升级Oracle Data Mining

               

                  
了解如何升级Oracle Data Mining。

                  

                     
在你之后完成“升级前步骤” ，无论您是从11 g还是从10 g版本升级 ，所有模型和挖掘元数据都与Oracle数据库升级过程完全集成。
                     

                     
升级的模型继续像以前的版本一样工作。您在升级后的环境中创建的升级模型和新模型都可以使用新版本中引入的新挖掘功能。

                     
要升级数据库，可以使用数据库升级助手（DBUA），也可以使用导出/导入实用程序执行手动升级。

8.2.2.1.1从版本10 g升级

                     

                        

                           
在 Oracle Data Mining 10g，数据挖掘元数据和PL / SQL包存储在DMSYS模式中。在Oracle Data Mining 11 g和12 c中 ， DMSYS不再存在;数据挖掘元数据对象存储在SYS 。

NOAUDIT（传统审计）

         

         

            

               

                  
本节介绍传统 审计的NOAUDIT语句，该语句与早于Oracle Database 12 c的发行版中使用的审计功能相同。
                  

                  
Oracle数据库12c中开始，Oracle引入统一 审计 ，它提供了一整套增强审核功能。为了向后兼容，仍然支持传统审计。但是，Oracle建议您计划将现有审核设置迁移到新的统一审核策略语法。对于新的审计要求，Oracle建议您使用新的统一审计。在未来的主要版本中可能不支持传统审计。
                  

                  

                     
也可以看看：

                     
NOAUDIT（统一审计），用于描述统一审计的NOAUDIT语句

                  

               

               
               

                  
目的

                  
使用NOAUDIT语句停止以前由AUDIT语句启用的审核操作。
                  

                  
NOAUDIT语句必须具有与上一个AUDIT语句相同的语法。此外，它仅反转了该特定陈述的影响。例如，假设一个AUDIT语句A启用特定用户的审核。第二个语句B启用所有用户的审核。NOAUDIT语句C禁用所有用户的审计反转语句B.但是，语句C使语句A生效并继续审计用户指定的语句A.

审计（传统审计）

                  

               

               
               

                  
先决条件

                  
要停止审计SQL语句，您必须具有AUDIT SYSTEM系统特权。
                  

                  
要停止审核架构对象，您必须是停止审核的对象的所有者，或者您必须具有AUDIT ANY系统特权。此外，如果您选择用于审计的对象是目录，那么即使您创建了该目录，也必须具有AUDIT ANY系统特权。
                  

                  
要指定CONTAINER子句，必须连接到多租户容器数据库（CDB）。要指定CONTAINER = ALL ，当前容器必须是根，并且您必须具有通常授予的AUDIT SYSTEM权限才能停止审计SQL语句的发布，或者通常授予的AUDIT ANY权限以便停止审计对架构对象的操作。要指定CONTAINER = CURRENT ，当前容器必须是可插拔数据库（PDB），并且您必须具有本地授予的AUDIT SYSTEM权限才能停止审核SQL语句的颁发或本地授予的AUDIT ANY权限以便停止对架构对象的审计操作。

语义

               

               
               

                  
audit_operation_clause

                  
使用audit_operation_clause停止审计特定SQL语句。
                  

                  
statement_option

                  
对于sql_statement_shortcut ，请指定要停止审计的SQL语句的快捷方式。有关SQL语句快捷方式列表和它们审计的SQL语句，请参阅表12-1和表12-2 。
                  

                  
所有

                  
指定ALL以停止审核由于较早的AUDIT ALL ...语句而当前正在审计的所有语句选项。您不能使用此子句来反转早期的AUDIT ALL STATEMENTS ...语句。
                  

                  
所有声明

                  
指定ALL STATEMENTS以反转早期的AUDIT ALL STATEMENTS ...语句。您不能使用此子句来反转早期的AUDIT ALL ...语句。
                  

                  
system_privilege

                  
对于system_privilege ，请指定要停止审计的系统特权。有关系统权限及其授权的语句的列表，请参阅表18-1 。
                  

                  
所有特权

                  
指定ALL PRIVILEGES以停止审核当前正在审核的所有系统权限。
                  

               

               
               

                  
auditing_by_clause

                  
使用auditing_by_clause只能由指定的用户在他们的后续会议印发的SQL语句停止审核。如果省略此子句，则Oracle数据库将停止对所有用户语句的审核，但WHENEVER SUCCESSFUL描述的情况除外。

               

               
               

                  
audit_schema_object_clause

                  
使用audit_schema_object_clause停止审计特定数据库对象。
                  

                  
sql_operation

                  
对于sql_operation ，请指定要在ON子句中指定的对象上停止审计的操作类型。有关这些选项的列表，请参阅表12-3 。
                  

                  
所有

                  
将ALL指定为等效于指定适用于对象类型的所有SQL操作的快捷方式。
                  

               

               
               

                  
auditing_on_clause

                  
auditing_on_clause允许您指定要停止审计的特定架构对象。
                  

                  
    
                     
  • 
                        
    对于object，请指定表，视图，序列，存储过程，函数或包，物化视图或库的对象名称。如果不使用schema限定object ，则Oracle数据库假定该对象位于您自己的模式中。有关审核特定架构对象的信息，请参阅AUDIT（传统审核） 。
                        
    
                     
    • 
                     
  • 
                        
    DIRECTORY子句允许您指定要停止审计的目录的名称。
                        
    
                     
    • 
                     
  • 
                        
    SQL TRANSLATION PROFILE子句允许您指定要停止审计的SQL转换配置文件。
                        
    
                     
    • 
                     
  • 
                        
    指定DEFAULT以删除指定的对象选项作为后续创建的对象的默认对象选项。
                        
    
                     
    • 
                  

                  
网络

                  
使用此子句可以停止审核数据库链接使用和登录。

                  
DIRECT_PATH负载

                  
使用此子句可以停止审计SQL * Loader直接路径加载。

                  
在[NOT]成功的时候

                  
指定WHENEVER SUCCESSFUL仅停止对成功完成的架构对象上的SQL语句和操作的审计。
                  

                  
指定WHENEVER NOT SUCCESSFUL仅停止审计导致Oracle数据库错误的SQL语句和操作。
                  

                  
如果省略此子句，则无论成功与否，数据库都会停止对所有语句或操作进行审计。

               

               
               

                  
集装箱条款

                  
使用CONTAINER子句指定NOAUDIT命令的范围。
                  

                  
    
                     
  • 
                        
    指定CONTAINER = CURRENT以停止您所连接的PDB中的审核。如果指定auditing_by_clause ，则user必须是当前PDB中的普通用户或本地用户。如果指定auditing_on_clause ，则对象必须是当前PDB中的本地对象。
    
                     
    • 
                     
  • 
                        
    指定CONTAINER = ALL以停止整个CDB的审核。如果指定auditing_by_clause ，则user必须是普通用户。如果未指定auditing_by_clause ，则会停止所有公共用户和每个PDB中的所有本地用户的审核。如果指定auditing_on_clause ，则对象必须是公共对象。
                        
    
                     
    • 
                  

                  
如果省略此子句，则默认为CONTAINER = CURRENT 。
