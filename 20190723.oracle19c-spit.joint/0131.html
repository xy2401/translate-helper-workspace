2.4.1关于Instant Client的优点

  • 安装只涉及复制四个文件。

  • 客户端的存储空间要求很低

  • 部署的应用程序不会丢失任何功能或性能

  • 使用ISV应用程序简化包装

OCCI Instant Client功能简化了OCCI的安装。尽管OCCI是独立的ORACLE_HOME的即时客户端模式的设置，依赖于应用程序ORACLE_HOME设置，可以将其设置为适当的值继续操作。Instant Client模式的激活仅取决于加载Instant Client数据共享库的能力。特别是，此功能允许与使用ORACLE_HOME作为其数据的Oracle应用程序进行互操作，但使用较新版本的Oracle Client。

2.4.2关于安装Instant Client

OCCI只需要由操作系统的动态加载程序加载四个共享库（或动态链接库，因为它们在某些操作系统上被调用）。使用Oracle Database 12 c第1版（12.1）库名称;库名称的数字部分更改为与将来的版本号保持一致。

为了清晰和易于开发，从Oracle Database 12 c第1版（12.1）开始更改库结构。客户端共享库libclntsh.so.12.1依赖于libclntshcore.so.12.1 。libclntshcore.so.12.1库包含NLS和CORE功能。这两个库必须安装在同一目录中。

  • 客户端代码库 （Linux和UNIX上的libclntsh.so.11.1和Windows上的oci.dll ）

  • 安全库 （ libnnz11.so在Linux和UNIX和orannzsbb11.dll在Windows上）

  • OCCI库 （ libocci.so.11.1在Linux和UNIX和oraocci11.dll在Windows上）

2.4.2.1.1关于Instant Client SDK

Instant Client也可以作为SDK包下载。SDK包含所有必需的头文件和用于在Instant Client环境中开发OCCI应用程序的makefile。一旦开发，这些应用程序可以部署在任何客户端环境中。SDK具有以下附加功能：

  • 它包含C ++演示程序。

  • 它包括在Windows上链接应用程序所需的库，并提供Make.bat文件来构建演示。

  • Makefile demo.mk用于构建Linux和UNIX的演示。在链接应用程序之前， instantclient_12_1目录必须位于LD_LIBRARY_PATH 。这些程序需要在instantclient_12_1目录中分别为客户端代码库和OCCI库libclntsh.so.12.1和libocci.so.12.1符号链接。演示Makefile， demo.mk ，在链接步骤之前生成这些。这些符号链接也可以在shell脚本中创建：

    cd instantclient_12_1 ln -s libclntsh.so.11.1 libclntsh.so ln -s libocci.so.11.1 libocci.so

  • SDK还包含对象类型转换器（OTT）实用程序及其生成应用程序头文件的类。

2.4.2.2关于完整客户端安装

如果通过选择“ 管理”选项执行完整的客户端安装，

  • 在Linux或UNIX平台上，可以从$ ORACLE_HOME /instantclient目录中复制libociei.so库。可以在完整的Oracle安装中从$ ORACLE_HOME /lib目录复制所有其他库。

  • 在Windows上，可以从ORACLE_HOME \instantclient目录复制oraociei11.dll库。可以从ORACLE_HOME \bin目录复制所有其他Windows库。

2.4.2.3运行Oracle Universal Installer

如果未安装数据库，则可以通过从Oracle Universal Installer中选择Instant Client选项来安装这些库。完成这些步骤后，您就可以开始运行OCCI应用程序了。

  1. 将Instant Client共享库安装到诸如instantclient_12_1的目录中。
  2. 将操作系统共享库路径环境变量设置为步骤1中的目录。例如，在Linux或UNIX上，将LD_LIBRARY_PATH设置为instantclient_12_1 。在Windows上，设置PATH以找到instantclient_12_1目录。
2.4.2.4关于Instant Client CD

您还可以从Instant Client CD安装Instant Client。您必须在空目录或其他系统上安装Instant Client。

操作系统Library Path变量上应该只有一组Oracle库;如果您有多个Instant Client库的目录或副本，则操作系统库路径中只应有一个目录。

同样，如果您在同一系统的ORACLE_HOME上也有安装，请不要将ORACLE_HOME /lib和Instant Client目录放在操作系统库路径上，而不管它们在库路径中的显示顺序如何。ORACLE_HOME /lib目录（用于非Instant Client操作）或Instant Client目录（用于Instant Client操作）中只应有一个位于操作系统Library Path变量上。

2.4.3关于使用Instant Client

Instant Client功能专为运行生产应用程序而设计。对于开发，请使用Instant Client SDK或完整安装来访问OCCI头文件，makefile，演示程序等。

2.4.4在UNIX上修补Instant Client共享库

Windows平台上不提供此功能。

由于Instant Client主要是部署功能，因此其设计目标之一是减少必要文件的数量和大小。因此，Instant Client部署不包括用于修补共享库的所有文件。您应该在基于ORACLE_HOME的OPATCH上使用OPATCH实用程序来修补Instant Client共享库。OPATCH实用程序将用于Linux和UNIX的ORACLE_HOME安装的修补信息存储在libclntsh.so.11.1中。可以使用genezi实用程序检索此信息：

genezi -v

如果未在部署Instant Client的系统上安装genezi实用程序，则可以从ORACLE_HOME系统的ORACLE_HOME /bin目录中复制它。

在ORACLE_HOME环境中应用修补程序后，将“ 关于安装Instant Client ”中列出的文件复制到Instant Client目录。您可以生成Instant Client *.zip文件，而不是复制单个文件，如“ 重新生成数据共享库和Zip文件 ”中所述 。然后，您可以将zip文件复制到目标系统并解压缩，而不是复制单个文件。

2.4.7设置OCCI Instant Client的环境变量

ORACLE_HOME环境变量不再确定全球化支持，CORE和错误消息文件的位置。仅OCCI应用程序不应要求设置ORACLE_HOME 。但是，如果已设置，则不会对OCCI的操作产生影响。OCCI始终从数据共享库获取其数据。如果数据共享库不可用，则仅使用ORACLE_HOME并假定ORACLE_HOME安装。设置后， ORACLE_HOME应该是标识目录的有效操作系统路径名。

Instant Client模式ORA_NLS忽略环境变量ORA_NLS33 ， ORA_NLS32和ORA_NLS 。

在Instant Client模式下，如果未设置ORA_TZFILE变量，则使用数据共享库中较大的默认timezlrg_ n .dat文件（其中n是文件的版本号）。如果使用数据共享库中较小的timezone _ n .dat文件，则将ORA_TZFILE环境变量设置为文件名，而不包含任何绝对路径名或相对路径名，如例2-3所示。

如果OCCI未在Instant Client模式下运行，因为数据共享库不可用，则ORA_TZFILE变量（如果已设置）将命名完整的路径名。

如果使用TNSNAMES条目，则TNS_ADMIN目录必须包含TNSNAMES配置文件。如果未设置TNS_ADMIN ，则ORACLE_HOME /network/admin目录必须包含Oracle Net Services配置文件。

示例2-3如何设置ORA_TZFILE环境变量

在Linux和UNIX上：

setenv ORA_TZFILE timezone_ n .dat

在Windows上：

设置ORA_TZFILE timezone_ n .dat

分区修剪

分区修剪是数据仓库的基本性能特征。

在分区修剪中，优化器分析SQL语句中的FROM和WHERE子句，以在构建分区访问列表时消除不需要的分区。此功能使Oracle数据库仅对与SQL语句相关的分区执行操作。

分区修剪的好处

分区修剪大大减少了从磁盘检索的数据量并缩短了处理时间，从而提高了查询性能并优化了资源利用率。

如果在不同的列（使用全局分区索引）上对索引和表进行分区，则即使无法消除基础表的分区，分区修剪也会消除索引分区。

根据实际的SQL语句，Oracle数据库可能使用静态或动态修剪。静态修剪在编译时发生，其中包含有关事先访问的分区的信息。动态修剪在运行时发生，这意味着事先不知道语句要访问的确切分区。静态修剪的示例方案是包含WHERE条件的SQL语句，该条件在分区键列上具有常量文字。动态修剪的一个示例是在WHERE条件中使用运算符或函数。

分区修剪会影响修剪发生的对象的统计信息，还会影响语句的执行计划。

可用于分区修剪的信息

可以对分区列执行分区修剪。

当您在范围或列表分区列上使用范围， LIKE ，相等和IN -list谓词时，以及在散列分区列上使用相等和IN -list谓词时，Oracle数据库会修剪分区。

在复合分区对象上，Oracle数据库可以使用相关谓词在两个级别进行修剪。例如，请参阅表sales_range_hash ，该表由s_saledate列上的范围分区，并在示例3-1中的列s_productid上通过哈希进行子分区。

Oracle使用分区列上的谓词来执行分区修剪，如下所示：

  • 使用范围分区时，Oracle仅访问分区sal99q2和sal99q3 ，代表1999年第三和第四季度的分区。

  • 使用哈希子分区时，Oracle仅访问存储s_productid=1200的行的每个分区中的一个子分区。子分区和谓词之间的映射是基于Oracle的内部哈希分布函数计算的。

引用分区表可以通过与引用表的连接来利用分区修剪。基于虚拟列的分区表受益于对SQL语句中使用虚拟列定义表达式的语句进行分区修剪。

静态分区修剪

Oracle主要根据静态谓词确定何时使用静态修剪。

在许多情况下，Oracle确定在编译时要访问的分区。如果使用静态谓词，则会发生静态分区修剪，但以下情况除外：

  • 使用子查询的结果进行分区修剪。

  • 优化程序使用星形转换重写查询，并在星形转换后进行修剪。

  • 最有效的执行计划是嵌套循环。

这三种情况导致使用动态修剪。

分区修剪技巧

本主题介绍了分区修剪的技巧。

使用分区修剪时，应考虑以下事项：

  • 数据类型转换

  • 函数调用

  • 收集表

数据类型转换

要从分区修剪中获得最大的性能优势，应避免使用需要数据库转换指定数据类型的构造。

数据类型转换通常会导致动态修剪，否则可能会进行静态修剪。受益于静态修剪的SQL语句比受益于动态修剪的语句执行得更好。

使用Oracle DATE数据类型时会发生数据类型转换的常见情况。Oracle DATE数据类型不是字符串，但在查询数据库时仅表示为该字符串;表示的格式由实例或会话的NLS设置定义。因此，在将数据插入DATE字段或在此类字段上指定谓词时，必须进行相同的反向转换。

通过指定TO_DATE转换，可以隐式或显式地进行转换。只有正确应用的TO_DATE函数才能保证数据库可以唯一地确定日期值并将其用于静态修剪，这对单个分区访问尤其有用。

6.1.1工作状态

描述工作状态。

有一个与工作的每个阶段相关的状态，如下所示：

  • 未定义 - 在创建句柄之前

  • 定义 - 首次创建句柄时

  • 执行 - 当DBMS_DATAPUMP 。执行START_JOB过程

  • 完成 - 当作业完成工作并且数据泵流程结束时

  • 完成 - 工作完成时

  • 停止等待 - 当请求有序的作业关闭时

  • 停止 - 当工作停止时

  • 空闲 - 执行DBMS_DATAPUMP.ATTACH以附加到已停止作业的时间与执行DBMS_DATAPUMP.START_JOB以重新启动该作业的时间之间的时间段

  • 未运行 - 当存在未运行的作业的主表时（没有与之关联的数据泵进程）

执行DBMS_DATAPUMP 。处于空闲状态的作业的START_JOB将使其返回执行状态。

如果所有用户都执行DBMS_DATAPUMP 。 DETACH从定义状态的作业中分离，然后该作业将完全从数据库中删除。

当作业异常终止或运行作业的实例关闭时，如果作业先前正在执行或空闲，则作业将处于“未运行”状态。然后，用户可以重新启动它。

主控制过程在定义，空转，执行，停止，停止等待和完成状态中处于活动状态。它在Stopped和Completed状态下也会短暂激活。作业的主表存在于除未定义状态之外的所有状态。工作进程仅在执行和停止挂起状态中处于活动状态，并且仅在导入作业的定义状态中处于活动状态。

作业处于执行状态时分离不会暂停作业，您可以随时重新连接到正在执行的作业以恢复获取有关作业的状态信息。

执行DBMS_DATAPUMP.DETACH过程时，可以显式发生分离，或者当数据泵API会话运行时，Data Pump API无法与数据泵作业或DBMS_DATAPUMP.STOP_JOB通信时，可以隐式发生DBMS_DATAPUMP.STOP_JOB执行DBMS_DATAPUMP.STOP_JOB过程。

“未运行”状态表示主表存在于正在执行的作业的上下文之外。如果作业已停止（可能稍后重新启动）或作业异常终止，则会发生这种情况。在作业开始时的作业状态转换期间以及在删除主表之前的作业结束时，也可以暂时看到此状态。需要注意的是，非运行状态显示仅在DBA_DATAPUMP_JOBS视图和USER_DATAPUMP_JOBS视图。它永远不会被GET_STATUS程序返回。

远程插入Rowsource

远程插入行源功能允许远程插入，要求本地Oracle数据通过Oracle数据库和Oracle数据库网关工作。此功能要求Oracle数据库和Oracle Database Gateway为12.2或更高版本。

通过Oracle数据库设计，必须在数据库链接站点执行某些分布式语句。但在某些情况下，执行这些查询所需的数据必须从原始Oracle数据库中获取。在同构连接下，远程Oracle数据库将回调源Oracle数据库以获取此类数据。但是在异构连接中，这是不可行的，因为这意味着外部数据存储必须查询只能由发出查询的Oracle实例提供的回调函数或数据。通常，这些类型的语句不能通过Oracle数据库网关支持。

以下类别的SQL语句会导致回调：

  • 任何带有子选择的DML，它指的是Oracle数据库中的一个表。

  • 任何DELETE ， INSERT ， UPDATE或“ SELECT...FOR UPDATE..."包含需要在原始Oracle数据库中执行的SQL函数或语句的SQL语句。

    这些SQL函数包括USER ， USERENV和SYSDATE ;并涉及从原始Oracle数据库中选择数据。

  • 任何涉及Oracle数据库中的表以及远程表中的LONG或LOB列的SQL语句。

可以通过远程插入行源功能工作的远程INSERT语句的示例如下：

INSERT INTO gateway_table @gateway_link select * from local_table ;

区分大小写

Informix处理字母大小写与Oracle不同。Informix使用以下规则：

  • 表所有者名称默认为大写字母，除非名称由双引号字符包围

  • 列名，表名，视图名等始终被视为小写字母

除非您使用双引号字符包围标识符，否则Oracle数据库默认为大写。例如，要引用名为emp的Informix表，请输入带双引号字符的名称，如下所示：

SQL> SELECT * FROM“emp”@IFMX;

但是，要从Oracle应用程序引用SCOTT拥有的名为emp的Informix表，请输入以下内容：

SQL> SELECT * FROM“Scott”。“emp”@IFMX;

如果名为emp的Informix表由SCOTT拥有（表所有者名称为大写字母），则可以输入不带双引号字符的所有者名称，如下所示：

SQL> SELECT * FROM SCOTT。“emp”@IFMX;

要么

SQL> SELECT * FROM scott。“emp”@IFMX;

Oracle建议您使用双引号字符包围所有Informix对象名称，并使用Informix数据字典中显示的对象名称的确切字母大小写。参照中列出的支持的Oracle数据字典的表或视图时，则不需要此约定数据字典 。

如果无法根据这些约定更改现有应用程序，请在Oracle中创建视图以将Informix名称与正确的字母大小写相关联。例如，要通过仅使用大写名称从现有Oracle应用程序引用Informix表emp ，请定义以下视图：

SQL> CREATE VIEW EMP（EMPNO，ENAME，SAL，HIREDATE）AS SELECT“empno”，“ename”，“sal”，“hiredate”FROM“emp”@IFMX;

使用此视图，应用程序可以发出如下语句：

SQL> SELECT EMPNO，ENAME FROM EMP;

使用视图是一种解决方案，可复制源自Informix数据字典的数据字典信息。每当在Informix数据库中更改相应表的数据定义时，您必须准备好更新Oracle视图定义。

二进制，字节和文本字面符号

Oracle SQL使用由单引号括起的十六进制数字来表示要比较或插入到定义为数据类型RAW列中的文字值。

此表示法未转换为与Informix BINARY ， BYTE和TEXT数据类型兼容的语法（0x后跟十六进制数字，由单引号括起）。

例如，不支持以下语句：

SQL> INSERT INTO BYTE_TAB @ IFMX VALUES（'Oxff'）;

其中BYTE_TAB包含数据类型为BINARY ， BYTE或TEXT 。插入或更新BINARY ， BYTE或TEXT数据类型时使用绑定变量。

数据类型转换

Informix不支持隐式日期转换。此类转换必须明确。

例如，网关为以下SELECT语句发出错误：

SQL> SELECT DATE_COL FROM TEST @ IFMX WHERE DATE_COL =“1-JAN-2001”;

要避免隐式转换出现问题，请添加显式转换，如下所示：

SQL> SELECT DATE_COL FROM TEST @ IFMX WHERE DATE_COL = TO_DATE（“1-JAN-2001”）;

查询

查询问题包括以下内容：

  • 行选择

  • 空字符串

行选择

在返回任何行之前，Informix会评估所有选定行的查询条件。如果一个或多个行的评估过程中存在错误，则即使其余行满足条件，也不会返回任何行。

Oracle逐行评估查询条件，并在评估成功时返回一行。返回行直到一行未通过评估。

空字符串

Oracle将SQL语句中的空字符串作为空值处理。Informix将空字符串作为空字符串处理。

比较空字符串

网关将文字空字符串传递给Informix数据库而不进行任何转换。如果您希望空字符串表示空值，则Informix不会以此方式处理该语句;它使用空字符串。

您可以通过在SQL语句中使用NULL或IS NULL而不是空字符串语法来避免此问题，如下例所示：

SELECT * from“emp”@IFMX where“ename”IS NULL;

选择一个空字符串

对于VARCHAR列，网关将空字符串作为NULL值返回到Oracle数据库。

对于CHAR列，网关返回列的完整大小，每个字符为空格（''）。

空绑定变量

对于VARCHAR绑定变量，网关将空绑定变量作为NULL值传递给Informix数据库。

166.1 DBMS_STORAGE_MAP概述

此术语和说明将帮助您了解DBMS_STORAGE_MAP API。

  • 映射库

    映射库可帮助您映射I / O处理堆栈元素的组件。I / O处理组件的示例包括文件，逻辑卷和存储阵列I / O目标。映射库在filemap.ora中标识。

  • 映射文件

    映射文件是描述文件的映射结构。它提供了一组属性，包括文件大小，文件组成的扩展区数和文件类型。

  • 映射元素和子元素

    映射元素是描述I / O堆栈中的存储组件的抽象映射结构。元素的示例包括镜像，条带，分区，raid5，连接元素和作为映射构建块的磁盘结构。映射子元素描述元素与I / O映射堆栈中的下一个元素之间的链接

  • 映射文件范围

    映射文件范围描述了驻留在一个元素上的连续块块。这包括设备偏移量，范围大小，文件偏移量，类型（数据或奇偶校验）以及范围所在元素的名称。对于原始设备或卷，该文件仅由一个文件扩展区组件组成。映射文件范围与Oracle范围不同。

125.1 DBMS_PRIVILEGE_CAPTURE概述

通过数据库特权分析，您可以创建一个策略，记录已授予用户的系统和对象特权的使用情况。然后，您可以确定用户正在使用而不使用的权限。从那里，您可以撤消任何未使用的权限，从而减少用户的超额权限授予数量。

通过分析用户执行特定任务所必需的权限，权限分析策略可帮助您为用户实现最小权限模型。

125.2 DBMS_PRIVILEGE_CAPTURE安全模型

默认情况下，权限分析管理员角色CAPTURE_ADMIN,被授予DBMS_PRIVILEGE_CAPTURE包的EXECUTE权限。

在数据库安装期间， CAPTURE_ADMIN角色被授予DBA角色。

创建DISKGROUP

注意：

仅当您使用Oracle ASM并且已启动Oracle ASM实例时，此SQL语句才有效。您必须从Oracle ASM实例中发出此语句，而不是从普通数据库实例发出。有关启动Oracle ASM实例的信息，请参阅“ Oracle自动存储管理管理员指南” 。

目的

使用CREATE DISKGROUP子句命名一组磁盘，并指定Oracle数据库应该为您管理该组。Oracle数据库将磁盘组作为逻辑单元进行管理，并在磁盘上均匀分布每个文件以平衡I / O. Oracle数据库还会自动在磁盘组中的所有可用磁盘上分发数据库文件，并在存储配置更改时自动重新平衡存储。

此语句创建磁盘组，将一个或多个磁盘分配给磁盘组，并首次装入磁盘组。请注意， CREATE DISKGROUP仅在本地节点上安装磁盘组。如果希望Oracle ASM在后续实例中自动装入磁盘组，则必须将磁盘组名称添加到初始化参数文件中ASM_DISKGROUPS初始化参数的值。如果使用SPFILE ，则磁盘组会自动添加到初始化参数中。

  • “Oracle自动存储管理管理员指南” ，提供有关Oracle ASM和使用磁盘组以简化数据库管理的信息

  • 有关将磁盘组名称添加到初始化参数文件的详细信息，请ASM_DISKGROUPS

REDUNDANCY条款

REDUNDANCY子句允许您指定磁盘组的冗余级别。

  • NORMAL REDUNDANCY要求至少存在两个故障组（参见FAILGROUP子句）。Oracle ASM根据磁盘组模板中指定的属性为磁盘组中的所有文件提供冗余。NORMAL REDUNDANCY磁盘组可以容忍丢失一个组。有关磁盘组模板的更多信息，请参阅ALTER DISKGROUP ... diskgroup_template_clauses 。

    如果省略REDUNDANCY子句，则NORMAL REDUNDANCY是默认值。因此，如果省略此子句，则必须至少创建两个故障组，否则create操作将失败。

  • HIGH REDUNDANCY要求至少存在三个故障组。Oracle ASM修复了3向镜像的镜像，每个扩展区获得两个镜像副本。HIGH REDUNDANCY磁盘组可以容忍两个故障组的丢失。

  • FLEX REDUNDANCY是一种磁盘组，允许数据库在创建磁盘组后指定自己的冗余。文件的冗余也可以在创建后更改。此类型的磁盘组支持Oracle ASM文件组和配额组。Flex磁盘组需要至少存在三个故障组。如果一个Flex磁盘组的故障组少于五个，那么它可以容忍丢失一个;否则，它可以容忍两个故障组的丢失。要创建Flex磁盘组，请使用COMPATIBLE 。 ASM和COMPATIBLE 。 RDBMS磁盘组属性必须设置为12 。 2或更高。

  • EXTENDED REDUNDANCY是一个磁盘组，除了在扩展群集环境中具有高可用性之外，还具有Flex磁盘组的所有功能。群集包含跨多个物理上分离的站点的节点。

    您可以使用SITE关键字为每个站点指定扩展磁盘组中的文件和文件组的冗余，而不是为每个磁盘组指定冗余。

  • EXTERNAL REDUNDANCY表示Oracle ASM不为磁盘组提供任何冗余。磁盘组中的磁盘必须提供冗余（例如，使用存储阵列），或者如果磁盘发生故障（例如，在测试环境中），您必须愿意容忍丢失磁盘组。如果指定EXTERNAL REDUNDANCY则无法指定FAILGROUP子句。

创建磁盘组后无法更改冗余级别，但以下情况除外：您可以将普通或高冗余磁盘组转换为Flex磁盘组。有关更多信息，请参阅ALTER DISKGROUP的convert_redundancy_clause 。

  • REGULAR磁盘或非仲裁失败组中的磁盘可以包含任何文件。

  • QUORUM磁盘或仲裁失败组中的磁盘不能包含任何数据库文件，Oracle Cluster Registry（OCR）或动态卷。但是， QUORUM磁盘可以包含群集同步服务（CSS）的投票文件。Oracle ASM尽可能使用仲裁故障组中的仲裁磁盘或磁盘来投票文件。

    在确定冗余要求时，不考虑仲裁失败组中的磁盘。

关于咨询锁定

顾问锁定是一种文件锁定机制，可锁定单个进程的文件。

文件锁定机制不能独立地强制执行任何形式的锁定，并且需要来自参与进程的支持。例如，如果进程P1对文件F1具有write锁定，则锁定API或操作系统不执行任何动作以防止任何其他进程P2对文件F1发出read或write系统调用。文件锁定机制的这种行为也适用于其他文件系统操作。涉及的进程（在文件锁定机制中）必须遵循用户级库以合适的API形式提供的锁定或解锁协议。保证文件锁定语义可用，这些过程包含锁定协议的推荐用法并尊重API调用的结果。

关于强制锁定

强制锁定是一种文件锁定机制，可以从参与的进程获得支持。

强制锁定是强制锁定方案，其不依赖于参与的进程来协作和/或遵循锁定API。例如，如果一个进程P1已经采取了write锁定在文件F1和如果一个不同的过程P2试图发出一个read / write系统调用（或任何其他的文件系统操作）上的文件F1 ，该请求被阻止，因为所关注的文件由进程P1独占锁定。

文件锁定支持

启用文件锁定机制可帮助应用程序阻止文件以进行各种文件系统操作。

UNIX和LINUX的fcntl() ， lockf()和flock()系统调用提供文件锁定支持。这些系统调用使应用程序能够通过dbfs_client-FUSE回调接口使用文件锁定工具。fcntl()提供的文件锁被广泛称为POSIX文件锁， flock()提供的文件锁称为BSD文件锁。POSIX和BSD文件锁的语义和行为彼此不同。通过fcntl()和flock()放置在同一文件上的锁彼此正交。在DBFS中设计和实现的文件锁定功能的语义类似于POSIX文件锁。在DBFS中，通过flock()系统调用放置的文件锁的语义类似于POSIX文件锁（例如fcntl() ）而不是BSD文件锁。 lockf()是一个库调用，它在大多数UNIX系统上作为fcntl()系统调用的包装器实现，因此，它提供了POSIX文件锁定语义。在DBFS中，通过fcntl() ， flock()和lockf()系统调用放置的文件锁提供了POSIX文件锁的相同类型的行为和语义。

文件锁定行为

DBFS文件锁定功能表现出以下行为：

  • DBFS中的文件锁使用幂等函数实现。如果进程在同一文件上发出“N” read或write锁定调用，则只有第一次调用才会生效，后续的“N-1”调用将被视为冗余并返回“无操作”（NOOP）。

  • 文件只能解锁一次。如果进程在同一文件上发出“N” unlock呼叫，则只有第一个呼叫才会生效，后续的“N-1”呼叫将被视为冗余并返回NOOP。

  • 仅从read到write支持锁定转换。如果进程P在文件F上保持read锁定（并且P是保持read锁定的唯一进程），则文件F上的P的write锁定请求将把read锁转换为exclusive / write锁。

公平调度

这种调度技术是基于每个文件使用排队机制实现的。例如，如果文件F被进程P1读取锁定，并且进程P2和P3在文件F上提交阻塞写锁定请求，则将阻止这两个进程（使用自旋锁的形式）并等待获取锁。请求将按DBFS客户端收到的顺序排队。如果进程P4在文件F上提交读锁定请求（阻塞调用或非阻塞调用），则即使可以向该进程授予读锁定，该请求也将排队。

DBFS客户端确保在P1释放其读锁之后，锁定请求的顺序是P2-> P3-> P4。

这意味着P2将是第一个获得锁定的人。一旦P2释放其锁定， P3将获得锁定，依此类推。

8.5关于SQL * Plus脚本调整

大多数性能优势来自调优脚本中执行的SQL查询。这是通过SQL * Plus的AUTOTRACE命令等工具完成的。它涉及重构查询以充分利用Oracle数据库SQL优化器。有关调整SQL语句的信息，请参阅“ Oracle数据库性能调整指南” 。

调整SQL * Plus特定命令所带来的性能提升较小，但对某些应用程序可能很重要。下列系统变量和命令可以影响SQL * Plus的性能。

8.5.1列NOPRINT

COLUMN NOPRINT 关闭屏幕输出和打印列标题和为列选择的所有值。

最好从SELECT中删除不需要的列，然后使用COLUMN NOPRINT来停止显示。从查询中删除列意味着SQL引擎不需要处理它，或者需要将列数据传输回SQL * Plus。

8.5.2设置APPINFO关闭

设置自动通过注册脚本 DBMS_APPLICATION_INFO包。将APPINFO设置为OFF可防止管理员监视脚本的性能和资源使用情况。

如果正在调用许多SQL脚本，则关闭APPINFO会停止对数据库DBMS_APPLICATION_INFO程序包的内部SQL * Plus调用。

8.5.3设置ARRAYSIZE

设定数量 SQL * Plus将一次从数据库中获取的行。有效值为1到5000。

设置ARRAYSIZE的有效性取决于Oracle数据库填充网络数据包的程度以及网络延迟和吞吐量。在最新版本的SQL * Plus和Oracle数据库中，ARRAYSIZE可能收效甚微。超大尺寸可以轻松占用更多SQL * Plus内存，这可能会降低整体性能。

8.5.4设置定义关闭

SET DEFINE OFF禁用解析要替换的命令替换变量及其值。

8.5.5设置FLUSH OFF

设置冲洗使操作系统能够缓冲输出。ON禁用缓冲和刷新输出到屏幕。将FLUSH设置为ON或OFF的任何好处取决于您的操作系统和数据。收益可能是微不足道的。

仅在运行不需要用户交互的脚本时才使用OFF，并且在脚本完成运行之前不需要查看其输出。

8.5.6设置LINESIZE

SET LINESIZE设置总数在开始换行之前，SQL * Plus在一行上显示的字符。

保持LINESIZE尽可能小，以避免额外的内存分配和内存复制。

但是，如果LINESIZE太小，则不能彼此相邻的列将放在单独的行上。这可能会显着降低性能。

8.5.7设置LONGCHUNKSIZE

SET LONGCHUNKSIZE套装 SQL * Plus用于检索BLOB的增量大小 ，BFILE，CLOB ， 长 ，NCLOB 或XMLType 值。

如果正在获取LONGS或LOB，请尝试不同的大小。

8.5.8 SET PAGESIZE

集每页上的行数输出

增加PAGESIZE以避免频繁打印标题，或将其设置为0以防止显示标题。

8.5.9 SET SERVEROUTPUT

SET SERVEROUTPUT OFF抑制显示输出（ DBMS_OUTPUT.PUT_LINE）SQL * Plus中的存储过程或PL / SQL块。

将SERVEROUTPUT设置为OFF将停止对用户SQL语句之后完成的DBMS_OUTPUT包的内部SQL * Plus调用。

8.5.10设置SQLPROMPT

设置SQL * Plus命令提示符。

使用默认提示“SQL>”可在每次显示提示时停止发生变量替换。

8.5.11 SET TAB

确定SQL * Plus如何格式化终端输出中的空白区域。

设置TAB ON会导致在终端输出中压缩多个空格。除非这显着减少了书面数据，否则所需的处理可能会略微超过任何好处。

8.5.12 SET TERMOUT

组 TERMOUT OFF会禁止显示，以便您可以从脚本中输出假脱机，而无需在屏幕上看到它。

如果不需要假脱机到文件和写入终端，请在SQL脚本中使用SET TERMOUT OFF来禁用终端输出。

8.5.13设置TRIMSPOOL ON时设置TRIMOUT

组 TRIMOUT ON或SET TRIMSPOOL ON删除每个显示或假脱机行末尾的尾随空白。

将这些变量设置为ON可以减少写入的数据量。但是，如果LINESIZE是最佳的，则将变量设置为OFF可能会更快。 SQL * Plus输出行在整个查询处理例程中都是空白的，因此删除空格可能需要额外的工作。

8.5.14未定

删除显式（使用DEFINE命令）或隐式（使用START命令的参数或COLUMN NEW_VAL | OLD_VAL）定义的替换变量。

使用UNDEFINE命令删除不必要的替换变量。这可以减少使用'＆'，new_value或old_value变量的任何操作所花费的时间。
