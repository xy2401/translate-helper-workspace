26.1关于奇异值分解

奇异值分解（SVD）与密切相关主成分分析（PCA）是完善的特征提取方法，具有广泛的应用。Oracle Data Mining将SVD实现为特征提取算法，将PCA实现为SVD模型的特殊评分方法。

SVD和PCA是正交线性变换，在捕获数据的基础方差时是最佳的。此属性对于减少高维数据的维度和支持有意义的数据可视化非常有用。

除降维外，SVD和PCA还有许多重要的应用。这些包括矩阵求逆，数据压缩和未知数据值的插补。

26.1.1矩阵操作

奇异值分解（SVD）是一种分解方法，它将矩形矩阵X分解为三个矩阵的乘积：

图26-1矩阵操作


“图26-1矩阵操作”的描述
  • U矩阵由一组“左”正交基础组成
  • S矩阵是对角矩阵
  • V矩阵由一组“正确的”正交基础组成

S中的值称为奇异值。它们是非负的，它们的大小表明相应碱基（组分）的重要性。奇异值反映了基数捕获的数据方差的数量。第一个基础（具有最大奇异值的基础）位于最大数据方差的方向上。第二个基础捕获具有第二个最大方差的正交方向，依此类推。

SVD基本上执行坐标旋转，其将变换的轴与数据中的最大方差的方向对齐。在假设观测数据具有高信噪比并且大方差对应于感兴趣的数据内容而较低方差对应于噪声的情况下，这是有用的过程。

SVD假设基础数据是高斯分布的，并且可以用均值和协方差来很好地描述。

26.1.2低秩分解

为了降低维数，奇异值分解（SVD）保持低阶基（具有最大奇异值的基）并忽略高阶基（具有最小奇异值的基）。该策略背后的基本原理是，低阶基数保留了对其方差贡献最大的数据特征，并且可能捕获数据的最重要方面。

给定数据集X （ n × m ），其中n是行数， m是属性的数量，低秩SVD仅使用k个分量（ k <= min （ m ， n ））。在SVD的典型实现中， k的值需要目视检查与各个组件相关联的排序奇异值。在Oracle Data Mining中，SVD自动估算截止点，这对应于解释方差的显着下降。

25.1关于奇异值分解

奇异值分解（SVD）与密切相关主成分分析（PCA）是完善的特征提取方法，具有广泛的应用。Oracle Data Mining将SVD实现为特征提取算法，将PCA实现为SVD模型的特殊评分方法。

SVD和PCA是正交线性变换，在捕获数据的基础方差时是最佳的。此属性对于减少高维数据的维度和支持有意义的数据可视化非常有用。

除降维外，SVD和PCA还有许多重要的应用。这些包括矩阵求逆，数据压缩和未知数据值的插补。

SVD假设基础数据是高斯分布的，并且可以用均值和协方差来很好地描述。

  • 如果在输入有关其中一个安装屏幕的错误信息后单击“ 下一步” ，则单击“上一步”返回到屏幕并更正信息。

SYSTEM_DRIVE：\ Program Files \ Oracle \ Inventory \ logs

installActions date_time .log

installActions2009-05-14_09-00-56-am.log

目的

PREDICTION_COST返回选择中每行的成本。成本是指最低成本类或指定class 。成本以BINARY_DOUBLE形式返回。

PREDICTION_COST可以执行分类或异常检测。对于分类，返回的成本是指预测的目标类别。对于异常检测，返回的成本是指1 （对于典型行）或0 （对于异常行）的分类。

您可以将PREDICTION_COST与PREDICTION函数结合使用，以获得预测和预测成本。

cost_matrix_clause

成本是最小化最有害的错误分类的偏向因素。例如，误报可能被认为比假阴性更昂贵。成本在成本矩阵中指定，该矩阵可与模型关联或在VALUES子句中内联定义。所有分类算法都可以使用成本来影响评分。

决策树是唯一可以使用成本来影响模型构建的算法。用于构建决策树模型的成本矩阵也是模型的默认评分成本矩阵。

以下成本矩阵表指定1的错误分类比错误分类0成本高五倍。 

ACTUAL_TARGET_VALUE PREDICTED_TARGET_VALUE COST ------------------- ---------------------- ------ ---- 0 0 0 0 1 1 1 0 5 1 1 0

在cost_matrix_clause ：

  • COST MODEL表明应该通过考虑与模型相关联的评分成本矩阵来执行评分。如果成本矩阵不存在，则该函数返回错误。

  • COST MODEL AUTO表示存在成本矩阵未知。如果存在成本矩阵，则该函数使用它来返回最低成本预测。否则，该函数返回最高概率预测。

  • VALUES子句为class_value指定内联成本矩阵。例如，您可以指定1的错误分类比错误分类0成本高5倍，如下所示：

    预测（nb_model成本（0,1）值（（0,1），（1,5））使用*）

    如果使用内联成本矩阵调用具有评分成本矩阵的模型，则使用内联成本。

语法选择

PREDICTION_COST可以通过以下两种方式之一对数据进行评分：它可以将挖掘模型对象应用于数据，也可以通过执行构建和应用一个或多个瞬态挖掘模型的分析子句来动态挖掘数据。选择语法或分析语法 ：

  • 语法 - 使用第一种语法使用预定义的模型对数据进行评分。提供执行分类或异常检测的模型的名称。

  • 分析语法 - 使用分析语法在没有预定义模型的情况下对数据进行评分。分析语法使用mining_analytic_clause ，它指定是否应为多个模型构建分区数据。mining_analytic_clause支持query_partition_clause和order_by_clause 。（参见“ analytic_clause :: = ” 。）

    • 对于分类，请指定FOR expr ，其中expr是一个表达式，用于标识具有字符数据类型的目标列。

    • 对于异常检测，请指定关键字OF ANOMALY 。

在对分区模型进行评分时， PREDICTION_COST函数的语法可以使用可选的GROUPING提示。请参阅GROUPING提示 。

4.4.1关于Oracle调用接口（OCI）

OCI支持传统的3GL和面向对象的数据库访问技术，如以下部分所述。

OCI的一个重要组件是一组管理称为对象缓存的工作空间的调用。该对象缓存是客户端上的一个内存块，它允许程序存储整个对象并在它们之间导航，而无需额外的往返服务器。

对象缓存完全受使用它的应用程序的控制和管理。Oracle服务器无法访问它。使用它的应用程序必须保持与服务器的数据一致性，并保护工作空间免受同时冲突的访问。

OCI提供功能

  • 使用访问服务器上的对象 SQL。

  • 通过遍历指针或REF访问，操作和管理对象缓存中的对象。

  • 将Oracle日期，字符串和数字转换为C数据类型。

  • 管理对象缓存内存的大小。

OCI提高了并发性允许锁定单个对象。它通过支持复杂的对象检索来提高性能。

OCI开发人员可以使用对象类型转换器生成与Oracle对象类型对应的C数据类型。

4.4.2 OCI程序中的关联访问

OCI程序支持使用可以操作对象数据的API对对象的关联访问。

传统上，3GL程序通过执行SQL语句和PL / SQL过程来操纵存储在关系数据库中的数据。通常在服务器上操纵数据而不会产生将数据传输到客户端的成本。OCI通过提供用于执行操作对象数据的SQL语句的API来支持对对象的关联访问。具体而言，OCI使您能够：

  • 执行操作对象数据和对象类型架构信息的SQL语句

  • 将对象实例，对象引用（ REF ）和集合作为输入变量传递给SQL语句

  • 返回对象实例， REF和集合作为SQL语句提取的输出

  • 描述返回对象实例， REF和集合的SQL语句的属性

  • 使用对象参数或结果描述和执行PL / SQL过程或函数

  • 通过增强的提交和回滚功能同步对象和关系功能

4.4.3 OCI程序中的导航访问

OCI程序通过API提供导航访问。

在面向对象的编程范例中，应用程序将它们的真实世界实体建模为一组形成对象图的相互关联的对象。对象之间的关系实现为引用。应用程序通过从一些初始对象集开始处理对象，使用这些初始对象中的引用遍历其余对象，并对每个对象执行计算。OCI为这种对象访问提供了一种API，称为导航访问。具体而言，OCI使您能够：

  • 缓存客户端计算机内存中的对象

  • 取消引用对象引用并固定对象缓存中的相应对象。固定对象透明地映射到宿主语言表示中。

  • 当不再需要固定对象时通知缓存

  • 在一次调用中从数据库中获取相关对象的图形到客户端缓存中

  • 锁定对象

  • 在缓存中创建，更新和删除对象

  • 将对客户端缓存中的对象所做的更改刷新到数据库

请参阅“ Pro * C / C ++中的导航访问 ” 。

4.4.4对象缓存

为了支持对象的高性能导航访问，OCI运行时提供了一个对象缓存，用于缓存内存中的对象。

对象缓存支持对对象缓存中的数据库对象的引用（ REF ），数据库对象可以通过它们的引用来标识（即固定）。当数据库对象加载到缓存中时，应用程序不需要分配或释放内存，因为对象缓存为数据库对象提供透明和有效的内存管理。

此外，当数据库对象加载到缓存中时，它们会透明地映射到宿主语言表示中。例如，在C编程语言中，数据库对象被映射到其对应的C结构。对象缓存维护缓存中的对象副本与相应的数据库对象之间的关联。在事务提交时，对缓存中的对象副本所做的更改会自动传播到数据库。

对象缓存维护一个快速查找表，用于将REF映射到对象。当应用程序取消引用REF并且相应的对象尚未缓存在对象缓存中时，对象缓存会自动向服务器发送请求以从数据库中获取对象并将其加载到对象缓存中。相同REF后续解引用速度更快，因为它们成为本地缓存访问并且不会导致网络往返。

要通知对象缓存应用程序正在访问缓存中的对象，应用程序将对象固定;当它完成对象时，它会取消它。对象缓存维护缓存中每个对象的引脚数。计数在引脚调用时递增，在取消固定调用时递减。当引脚数变为零时，表示应用程序不再需要该对象。

对象缓存使用最近最少使用（LRU）算法来管理缓存的大小。当高速缓存达到最大大小时，LRU算法释放引脚数为零的候选对象。

结果集支持

各种关系数据库使存储过程能够返回结果集（一组或多组行）。

传统上，数据库存储过程与任何高级编程语言中的过程完全相同。它们具有固定数量的参数，可以是IN ， OUT或IN OUT 。如果一个过程有n参数，它最多可以返回n值作为结果。但是，假设您希望存储过程执行SELECT * FROM emp类的查询并返回结果。emp表可能具有固定数量的列，但是在过程创建时无法告知它具有的行数。因此，不能创建返回此类查询结果的传统存储过程。因此，一些关系数据库供应商添加了从存储过程返回结果集的功能，但每个关系数据库都以不同方式返回存储过程的结果集。

Oracle有一种称为REF CURSOR的数据类型。与所有其他Oracle数据类型一样，存储过程可以将此数据类型作为IN或OUT参数。对于Oracle数据库，存储过程必须具有REF CURSOR类型的输出参数。然后它打开一个SQL语句的游标，并在该输出参数中放置该游标的句柄。然后，调用者可以从REF CURSOR检索与从任何其他游标相同的方式。

Oracle数据库可以做很多事情而不是返回结果集。REF CURSOR数据类型可以作为输入参数传递给PL / SQL例程，以便在客户端程序和PL / SQL例程之间来回传递，或者作为几个PL / SQL例程之间的输入参数。

  • 结果集支持非Oracle系统
    一些非Oracle系统允许存储过程返回结果集，但它们以不同的方式执行。
  • 结果集的异构服务支持
    结果集支持以不同形式存在于非Oracle数据库中。所有这些必须映射到Oracle REF CURSOR模型。

结果集支持非Oracle系统

一些非Oracle系统允许存储过程返回结果集，但它们以不同的方式执行。

非Oracle数据库的结果集支持通常基于以下两个模型之一。

  • 模型1：结果集支持

    创建存储过程时，可以显式指定该存储过程可以返回的最大结果集数。执行时，存储过程可以从零开始到指定的最大结果集数。执行存储过程后，客户端程序通过使用嵌入式SQL指令或调用客户端库函数来获取这些结果集的句柄。之后，客户端程序可以从与典型游标相同的方式从结果集中检索。

  • 模型2：结果集支持

    在此模型中，存储过程可以返回的结果集数没有指定的限制。Model 1和Oracle Database都有一个限制。对于Oracle数据库，存储过程返回的结果集的数量最多可以是REF CURSOR OUT参数的数量。对于模型1，使用存储过程语言中的指令指定上限。Model 2与Oracle Database和Model 1的另一种不同之处在于它们不返回结果集的句柄。相反，它们在从存储过程返回时将整个结果集放在线上。对于Oracle数据库，句柄是REF CURSOR OUT参数。对于模型1，它是在执行存储过程之后单独获得的。对于Oracle数据库和模型1，在获得句柄之后，通过对句柄进行提取来获得结果集中的数据;有几个游标打开，获取可以按任何顺序。但是，在Model 2的情况下，所有数据都已经在线上，结果集按照存储过程确定的顺序和最后程序的输出参数。必须检索整个第一个结果集，然后检索整个第二个结果集，直到检索到所有结果。最后，检索存储过程OUT参数。

父主题： 结果集支持

结果集的异构服务支持

结果集支持以不同形式存在于非Oracle数据库中。所有这些必须映射到Oracle REF CURSOR模型。

由于非Oracle系统之间的行为差异，异构服务结果集支持以两种不同的方式之一运行，具体取决于它所连接的非Oracle系统。

请注意以下有关异构服务结果集支持的信息：

  • 结果集支持是异构服务通用代码的一部分，但是要使该功能在网关中工作，驱动程序必须实现它。并非所有驱动程序都实现了结果集支持，您必须验证是否支持您的网关。

  • 异构服务支持存储过程中的REF CURSOR OUT参数。不支持IN和IN OUT参数。

  • REF CURSOR OUT参数都是匿名引用游标。由异构服务返回的REF CURSOR没有类型。

  • 结果集：光标模式
    由Oracle驱动程序将非Oracle系统存储过程返回的每个结果集映射到REF CURSOR类型的OUT参数。
  • 结果集：顺序模式
    特定存储过程可以返回最大数量的结果集。返回的结果集数最多为存储过程的REF CURSOR OUT参数数。它可以返回更少的结果集，但它永远不会返回更多。

父主题： 结果集支持

结果集：光标模式

由Oracle驱动程序将非Oracle系统存储过程返回的每个结果集映射到REF CURSOR类型的OUT参数。

客户端程序检测具有REF CURSOR类型的多个OUT参数的存储过程。执行存储过程后，客户端程序可以从获取REF CURSOR的方式，因为它会从同一个REF CURSOR的Oracle存储过程返回。当按照第3.4.1.1节所述连接到网关时，异构服务将处于光标模式。

结果集：顺序模式

特定存储过程可以返回最大数量的结果集。返回的结果集数最多为存储过程的REF CURSOR OUT参数数。它可以返回更少的结果集，但它永远不会返回更多。

对于3.4.1.2节中描述的系统，没有可返回的最大结果集数。对于模型1（在第3.4.1.1节中），过程可以返回的结果集的最大数量是已知的，并且驱动程序可以返回到异构服务，在存储过程中通过REF CURSOR OUT的数量指定REF CURSOR OUT参数。如果在执行存储过程时返回的结果集少于最大值，则其他REF CURSOR OUT参数将设置为NULL 。

Model 2数据库服务器的另一个问题是必须按照数据库将它们放在线上的顺序检索结果集。这可以防止异构服务在连接到这些数据库时以游标模式运行。要访问这些存储过程返回的结果集，异构服务必须处于顺序模式。

在顺序模式下，驱动程序返回的过程描述包含以下内容：

使用自动数据准备（ADP）了解数据转换。

大多数算法都需要某种形式的数据转换。在模型构建过程中，Oracle Data Mining可以自动执行算法所需的转换。您可以选择使用自己的其他转换来补充自动转换，也可以选择自己管理所有转换。

在计算自动转换时，Oracle Data Mining使用启发式方法来解决给定算法的常见要求。在大多数情况下，该过程导致合理的模型质量。

分箱，归一化和异常处理是数据挖掘算法通常需要的转换。

4.3.1分箱

分档也称为离散化，是一种降低连续和离散数据基数的技术。分箱将相关值组合在一起以减少不同值的数量。

分档可以显着提高资源利用率和模型构建响应时间，而不会显着降低模型质量。分箱可以通过加强属性之间的关系来提高模型质量。

监督分箱是智能分箱的一种形式，其中数据的重要特征用于确定箱边界。在有监督的分箱中，箱边界由单预测器决策树识别，该决策树考虑与目标的联合分布。有监督的分箱可用于数字和分类属性。

4.3.2标准化

归一化是减少数值数据范围的最常用技术。大多数规范化方法将单个变量的范围映射到另一个范围（通常为0,1）。

4.3.3异常值处理

如果值与列中的大多数其他值明显不同，则该值被视为异常值。异常值的存在可能对数据产生偏移影响，并且可能干扰转换的有效性，例如归一化或分级。

异常处理方法，如修剪或可以实现裁剪以最小化异常值的影响。

异常值表示有问题的数据，例如，由于仪器的异常情况导致的读数错误。但是，在某些情况下，特别是在商业领域，异常值是完全有效的。例如，在人口普查数据中，一些最富有的人的收入可能与一般人群有很大差异。不要将此信息视为异常值，因为它是数据的重要组成部分。您需要领域知识来确定异常值处理。
