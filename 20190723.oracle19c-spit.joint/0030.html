关于orabasetab

orabasetab文件用于根据$ ORACLE_HOME定义基本目录：ORACLE_BASE，ORACLE_BASE_HOME和ORACLE_BASE_CONFIG。

orabasetab文件驻留在ORACLE_HOME/install/orabasetab ，可用于确定ORACLE_HOME是只读还是读/写。它还定义了Oracle主目录的ORACLE_BASE和HOME_NAME。HOME_NAME是ORACLE_HOME的内部名称。

orabasetab文件中的最后一行以$ ORACLE_HOME开头，它定义了$ ORACLE_HOME的目录。最后一行包含四个字段，每个字段用冒号分隔符(:).分隔(:).

  1. 第一个字段匹配当前的$ ORACLE_HOME。

  2. 第二个字段定义当前ORACLE_HOME的ORACLE_BASE。

  3. 第三个字段定义了HOME_NAME，用于在只读ORACLE_HOME中构造ORACLE_BASE_HOME路径。

  4. 第四个字段在读/写ORACLE_HOME中显示N ，在只读ORACLE_HOME中显示Y

在只读ORACLE_HOME中，ORACLE_BASE_HOME路径为ORACLE_BASE/homes/HOME_NAME ，ORACLE_BASE_CONFIG与ORACLE_BASE相同。

在读/写ORACLE_HOME中，ORACLE_HOME，ORACLE_BASE_HOME和ORACLE_BASE_CONFIG都是相同的。

  1. 以Oracle安装所有者用户帐户（ oracle ）身份登录。

    $ cat orabasetab $ /u01/app/oracle/product/19.0.0/dbhome_1:/u01/app/oracle:OraDB19Home1：Y：

    在此示例中，行末尾第四个字段中的Y表示您具有只读Oracle主目录。

加载源文件会创建或更新Java源模式对象，并使先前从源派生的类模式对象无效。如果类模式对象不存在，则loadjava工具会创建它们。loadjava工具使旧的类架构对象无效，因为它们不是从新加载的源编译的。新加载的源（例如，类A编译由以下任何条件自动触发：

32.1关于评分和部署

评分是将模型应用于新数据。在Oracle Data Mining中，评分由SQL语言函数执行。

预测函数执行分类，回归或异常检测。群集功能将行分配给群集。特征提取功能将输入数据转换为一组高阶预测变量。DBMS_DATA_MINING PL / SQL包中也提供了评分过程。

部署是指在目标环境中使用模型。构建模型后，面临的挑战就是部署模型以获得最佳结果，并将其维护在生产环境中。 部署可以是以下任何一种：

  • 为批次或分数评分数据实时结果。分数可以包括预测，概率，规则和其他统计数据。

  • 提取模型详细信息以生成报告。例如：聚类规则，决策树规则或属性重要性模型中的属性排名。

  • 通过将挖掘结果合并到应用程序或操作系统中来扩展数据仓库的商业智能基础架构。

  • 将模型从构建它的数据库移动到用于评分的数据库（导出/导入）

Oracle Data Mining支持所有这些部署方案。

注意：

Oracle Data Mining评分操作支持并行执行。什么时候启用并行执行，将多个CPU和I / O资源应用于单个数据库操作的执行。

并行执行提供显着的性能改进，尤其是涉及复杂查询的操作以及通常与决策支持系统（DSS）和数据仓库相关联的大型数据库。

HugePages和透明HugePages配置的限制

查看本节中讨论的HugePages和Transparent HugePages指南。

Oracle建议您禁用Transparent HugePages，因为它们可能会导致访问内存的延迟，从而导致Oracle RAC环境中的节点重新启动，或Oracle Database单实例的性能问题或延迟。Oracle继续建议使用标准的HugePages for Linux。

Red Hat Enterprise Linux 7，SUSE 12和Oracle Linux 7默认启用透明HugePages内存，早期版本的Oracle Linux具有Unbreakable Enterprise Kernel 2（UEK2）内核。默认情况下，在UEK2及更高版本的UEK内核的后续版本中禁用透明HugePages内存。

注意：

虽然在UEK2和更高版本的UEK内核上禁用了透明HugePages，但默认情况下可以在Linux系统上启用Transparent HugePages。

透明Hugepages类似于标准HugePages。但是，虽然标准HugePages在启动时分配内存，但Transparent Hugepages内存使用内核中的khugepaged线程在运行时使用可交换的HugePages动态分配内存。

HugePages使用内存映射文件为大页表分配不可交换的内存。默认情况下不启用HugePages。如果启用HugePages，则应在计算交换空间之前从可用RAM中扣除分配给HugePages的内存。有关更多信息，请参阅您的分发文档以及Oracle Technology Network和My Oracle Support。

在Oracle Grid Infrastructure安装期间，Grid Infrastructure Management Repository（GIMR）配置为使用HugePages。由于Grid Infrastructure Management Repository数据库在群集上安装的所有其他数据库之前启动，如果分配给HugePages的空间不足，则一个或多个数据库的系统全局区域（SGA）可能会映射到常规页面，而不是Hugepages，这会对性能产生不利影响。将HugePages内存分配配置为足够大的大小，以容纳要在群集上安装的所有数据库的SGA大小的总和，以及Grid Infrastructure Management Repository。

ORA_ROWSCN伪列

ORA_ROWSCN反映最近一行更改的系统更改编号（SCN）。此更改可以是块（粗略）或行级（细粒度）的级别。后者由行级依赖性跟踪提供。参考CREATE TABLE ...NOROWDEPENDENCIES |有关行级依赖关系跟踪的更多信息，请参阅ROWDEPENDENCIES 。在没有行级依赖关系的情况下， ORA_ROWSCN反映了块级依赖关系。

无论是在块级还是在行级， ORA_ROWSCN都不应被视为精确的SCN。例如，如果事务更改了块中的行R并在SCN 10处ORA_ROWSCN ，则该行的ORA_ROWSCN将返回10并不总是如此。虽然永远不会返回小于10的值，但可以返回大于或等于10的任何值。也就是说，行的ORA_ROWSCN并不总是保证是最后修改该行的事务的确切提交SCN。但是，使用细粒度的ORA_ROWSCN ，如果两个事务T1和T2一个接一个地修改了同一行R并且已提交，则在提交T1之后对行R的ORA_ROWSCN进行查询将返回低于之后返回的值的值。提交T2。如果一个块被查询两次，那么ORA_ROWSCN的值可能会在查询之间发生变化，即使在查询之间的时间内没有更新行。唯一的保证是两个查询中ORA_ROWSCN的值都大于上次修改该行的事务的提交SCN。

您不能在查询中使用ORA_ROWSCN伪列。但是，您可以在创建视图时使用它来引用基础表。您还可以在UPDATE或DELETE语句的WHERE子句中使用此伪列。

闪回查询不支持ORA_ROWSCN 。而是使用版本查询伪列，它是为闪回查询显式提供的。有关这些伪列的其他信息，请参阅SELECT ... flashback_query_clause以获取有关闪回查询和版本查询伪列的信息。

对ORA_ROWSCN的限制：外部表不支持此伪列。

服务器池和基于策略的管理

通过基于策略的管理，数据库管理员可以指定运行数据库资源的服务器池（不包括Generic或Free）。

基于政策的管理：

  • 在需要时根据您使用策略设置的优先级提供服务器容量，从而启用动态容量分配

  • 允许按重要性分配资源，以便应用程序尽可能获得所需的最小资源，以便较低优先级的应用程序不从更重要的应用程序中获取资源

  • 确保在必要时进行隔离，以便您可以在群集中为应用程序和数据库提供专用服务器

  • 允许将策略配置为根据业务需求或应用程序需求更改池，以便池在适当的时间提供正确的服务

在服务器池中运行的应用程序和数据库不共享资源。由于服务器池不共享资源，因此它们会在必要时隔离资源，但会根据需要启用动态容量分配。与角色分离管理一起，此功能可满足具有标准化群集环境的组织的需求，但允许多个管理员组共享公共群集基础结构。

Oracle Clusterware有效地在集群中分配不同的资源。您只需提供资源可以运行的最小和最大节点数，并结合这些节点上运行的每个资源的重要性级别。

4.3关于使用静态连接池优化实际性能

Real-World Performance组调查的大多数联机事务处理（OLTP）性能问题与应用程序使用的连接策略有关。因此，设计良好的连接策略对于系统性能至关重要，尤其是在必须扩展以满足不断增长的需求的企业环境中。

大多数应用程序使用与数据库的动态连接池，使用最少数量的连接进行配置以在数据库上保持打开状态以及可以对数据库建立的最大连接数。当应用程序需要连接到数据库时，它会从池中请求一个。如果没有可用的连接，则应用程序会创建新连接（如果尚未达到最大连接数）。如果连接未在指定的持续时间内使用，则应用程序将关闭连接（如果有超过最小可用连接数）。

此配置仅通过维护应用程序主动需要的连接数来节省系统资源。在现实世界中，此配置可实现连接风暴和数据库系统CPU超额预订，从而快速破坏系统稳定性。当应用程序服务器上有许多需要数据库连接的活动时，可能会发生连接风暴。如果没有足够的数据库连接来提供所有请求，则应用程序服务器将打开新连接。创建与数据库的新连接是一项资源密集型活动，当在很短的时间内建立大量连接时，它可能会占用数据库系统上的CPU资源。

因此，要创建静态连接池，与数据库系统的连接数必须基于系统上可用的CPU核心。Oracle建议每个CPU核心使用1-10个连接。理想数量取决于应用程序和系统硬件。但是，该值在该范围内。 Real-World Performance组建议通过将最小和最大连接数设置为相同的值来创建与数据库的静态连接池。这可以通过将数据库连接数保持为预定义值来防止连接风暴。

例如，如果数据库服务器有2个CPU，每个CPU 12个核心，每个CPU 2个线程，则有24个可用核心，与数据库的连接数应在12到120之间。不考虑线程数，因为只有CPU内核能够退出指令。如果系统上有多个数据库，则此数字对于连接到系统和所有数据库的所有应用程序都是累积的。如果有两个应用程序服务器，则应在它们之间划分最大连接数（例如，在这种情况下为120）。如果系统上运行了两个数据库，那么最大连接数即120个连接需要在它们之间进行划分。

目的

XMLForest将其每个参数参数转换为XML，然后返回一个XML片段，该片段是这些转换参数的串联。

  • 如果value_expr是标量表达式，则可以省略AS子句，Oracle Database使用列名作为元素名称。

  • 如果value_expr是对象类型或集合，则AS子句是必需的，Oracle使用指定的表达式作为封闭标记。

    您可以通过指定c_alias （字符串文字）或指定EVALNAME value_expr 。在后一种情况下，将计算值表达式，并将结果（必须是字符串文字）用作标识符。标识符不必是列名或列引用。它不能是表达式或null。如果初始化参数MAX_STRING_SIZE = STANDARD ，则最多可包含4000个字符;如果MAX_STRING_SIZE = EXTENDED则MAX_STRING_SIZE 32767个字符。有关详细信息，请参阅“ 扩展数据类型 ” 。

  • 如果value_expr为null，则不为该value_expr创建任何元素。

5.3使用群集资源活动日志监视群集资源故障

群集资源活动日志提供有关资源故障的精确和特定信息，与诊断日志分开。

如果Oracle Clusterware管理的资源出现故障，则Oracle Clusterware会在Grid Infrastructure Management资源库中的群集资源活动日志中记录有关故障的消息。由于资源，托管节点或网络出现问题，可能会发生故障。群集资源活动日志提供资源故障原因的统一视图。

对集群资源活动日志的写入使用活动ID进行标记，并且任何相关数据都将获得相同的父活动ID，并嵌套在父数据下。例如，如果Oracle Clusterware正在运行并且您运行crsctl stop clusterware -all命令，则所有活动都会获取活动ID，相关活动将使用相同的父活动ID进行标记。在每个节点上，该命令在父ID下创建子ID，并使用相应的活动ID标记每个相应的活动。此外，各个节点上的每个资源基于父ID创建子ID，从而创建活动ID的层次结构。通过活动ID层次结构，您可以分析数据以查找特定活动。

例如，您可能拥有许多资源，这些资源之间具有复杂的依赖关系，并且具有数据库服务。星期五，您看到所有资源都在一个节点上运行，但是当您在星期一返回时，每个资源都在不同的节点上，并且您想知道原因。使用crsctl query calog命令，可以查询群集资源活动日志，以查找涉及这些资源和数据库服务的所有活动。输出提供了完整的流，您可以查询父服务故障转移ID中的每个子ID，并具体查看发生的情况和原因。

您可以使用筛选器查询群集资源活动日志中的任意数量的字段。例如，您可以查询特定操作系统用户（如root所写的所有活动。crsctl query calog命令生成的输出可以以表格格式或XML格式显示。

群集资源活动日志是当前Oracle Clusterware日志记录和警报日志消息的附件。

注意：

Oracle Clusterware不会将包含安全相关信息（如登录凭据）的消息写入群集活动日志。

Oracle RAC数据库和服务器池

Oracle RAC数据库支持两种类型的服务器池管理样式和部署模型。

  • 策略管理 ：部署基于服务器池，其中数据库服务在服务器池中作为单例或统一在服务器池中的所有服务器上运行。数据库部署在一个或多个服务器池中，服务器池的大小决定了部署中的数据库实例数。策略管理允许群集和数据库随着需求的变化而扩展或缩小。

    策略管理的数据库由基数定义， 基数是您希望在正常操作期间运行的数据库实例的数量。策略管理的数据库在群集管理员在群集中创建的一个或多个数据库服务器池中运行，并且可以在不同的时间在不同的服务器上运行。数据库实例在为数据库定义的服务器池中的所有服务器上启动。

    客户端可以使用相同的基于SCAN的连接字符串连接到策略管理的数据库，无论他们当时正在运行哪些服务器。

  • 管理员管理 ：部署基于Oracle数据库11g第2版（11.2）之前存在的Oracle RAC部署类型，并要求您静态配置每个数据库实例以在群集中的特定节点上运行，并将数据库服务配置为使用首选和可用的名称在属于某个数据库的特定实例上运行。

    查看管理员管理的数据库的数据库资源时，您会看到定义了与Oracle数据库同名的服务器池。此服务器池是Oracle定义的特殊服务器池（称为Generic）的一部分。Oracle RAC管理通用服务器池以支持管理员管理的数据库。使用SRVCTL或DBCA添加或删除管理员管理的数据库时，Oracle RAC会创建或删除作为Generic成员的服务器池。

完成这些过程以将Oracle Grid Infrastructure用于独立服务器，其中包括Oracle自动存储管理（Oracle ASM）。

用于独立服务器的Oracle Grid Infrastructure（也称为Oracle Restart）为单实例Oracle数据库提供系统支持。Oracle ASM是Oracle数据库文件的卷管理器和文件系统，支持单实例Oracle数据库和Oracle Real Application Clusters（Oracle RAC）配置。Oracle自动存储管理还支持满足您的应用程序需求的通用文件系统，包括Oracle数据库二进制文件。Oracle自动存储管理是Oracle推荐的存储管理解决方案。它提供了传统卷管理器和文件系统的替代方案。

注意：

  • 如果要使用Oracle ASM或Oracle Restart，则必须在安装和创建数据库之前为独立服务器安装Oracle Grid Infrastructure。否则，您必须使用Oracle Restart手动注册数据库。

  • 您既不能在Oracle Grid Infrastructure集群成员节点上安装Oracle Restart，也不能将Oracle Restart服务器添加到Oracle Grid Infrastructure集群成员节点。Oracle Restart支持在一台服务器上支持单实例数据库，而Oracle Grid Infrastructure for Cluster支持群集上的单实例或Oracle RAC数据库。

    您可以将直接连接存储（DAS）和存储区域网络（SAN）磁盘与Oracle ASM一起使用。

从版本10g之前的客户端连接到Oracle数据库失败，错误ORA-28040: No matching authentication protocol 。

通常，安装程序以交互模式运行，这意味着它会提示您在图形用户界面（GUI）屏幕中提供信息。使用响应文件提供此信息时，可以使用以下任一模式从命令提示符运行安装程序：

验证Linux上的磁盘I / O调度程序

为了获得Oracle ASM的最佳性能，Oracle建议您使用Deadline I / O Scheduler。

磁盘I / O调度程序对磁盘I / O的请求进行重新排序，延迟或合并，以实现更好的吞吐量和更低的延迟。Linux有多个磁盘I / O调度程序可用，包括Deadline，Noop，Anticipatory和Completely Fair Queuing（CFQ）。
在每个群集节点上，输入以下命令以验证是否已配置Deadline磁盘I / O调度程序以供使用：
#cat / sys / block / $ {ASM_DISK} / queue / scheduler noop [deadline] cfq

在此示例中，默认磁盘I / O调度程序为Deadline， ASM_DISK是Oracle自动存储管理（Oracle ASM）磁盘设备。

在某些虚拟环境（VM）和特殊设备（如快速存储设备）上，上述命令的输出可能为none 。操作系统或VM绕过内核I / O调度并将所有I / O请求直接提交给设备。请勿在此类环境中更改I / O计划程序设置。

如果默认磁盘I / O调度程序不是Deadline，则使用rules文件设置它：

  1. 使用文本编辑器为Oracle ASM设备创建UDEV规则文件：

    #vi /etc/udev/rules.d/60-oracle-schedulers.rules

  2. 将以下行添加到规则文件并保存：

    ACTION ==“add | change”，KERNEL ==“sd [az]”，ATTR {queue / rotating} ==“0”，ATTR {queue / scheduler} =“deadline”

  3. 在群集系统上，将规则文件复制到群集上的所有其他节点。例如：

    $ scp 60-oracle-schedulers.rules root @ node2：/etc/udev/rules.d/

  4. 加载规则文件并重新启动UDEV服务。例如：

    1. Oracle Linux和Red Hat Enterprise Linux

      #udevadm control --reload-rules

    2. SUSE Linux Enterprise Server

      #etet/init.d boot.udev restart

  5. 验证磁盘I / O调度程序是否设置为截止日期。

关于安装修复脚本

Oracle Universal Installer检测何时不满足安装的最低要求，并创建称为修复脚本的shell脚本，以完成不完整的系统配置步骤。

Fixup脚本执行以下任务：

  • 如有必要，将内核参数设置为成功安装所需的值，包括：

    • 共享内存参数。

    • 打开文件描述符和UDP发送/接收参数。

  • 在Oracle Inventory（中央库存）目录上创建和设置权限。

  • 如有必要，为Oracle Inventory目录和操作系统特权组创建或重新配置安装所有者的主要和辅助组成员身份。

  • 如有必要，将shell限制设置为所需的值。

注意：

使用修复脚本无法确保满足安装Oracle数据库的所有先决条件。您仍必须验证是否满足所有预安装要求以确保安装成功。

句法

目的

XMLColAttVal创建一个XML片段，然后展开生成的XML，以便每个XML片段都具有带有属性name的name column 。

您可以使用AS子句将name属性的值更改为列名以外的值。您可以通过指定c_alias （字符串文字）或指定EVALNAME value_expr 。在后一种情况下，将计算值表达式，并将结果（必须是字符串文字）用作别名。如果初始化参数MAX_STRING_SIZE = STANDARD ，则别名最多为4000个字符;如果MAX_STRING_SIZE = EXTENDED则MAX_STRING_SIZE = 32767个字符。有关详细信息，请参阅“ 扩展数据类型 ” 。

您必须为value_expr指定一个值。如果value_expr为null，则不返回任何元素。

对XMLColAttVal的限制

您不能为value_expr指定对象类型列。
