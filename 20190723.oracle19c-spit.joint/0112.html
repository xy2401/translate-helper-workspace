30.1 DBMS_AUTO_REPORT概述

该软件包提供了一个界面，用于查看已捕获到AWR中的SQL监控和实时ADDM数据。它还提供子程序来控制如何将这些数据捕获到AWR的行为。捕获的数据存储在AWR中并通过2个视图公开： DBA_HIST_REPORTS和DBA_HIST_REPORTS_DETAILS 。

30.2 DBMS_AUTO_REPORT安全模型

该程序包可供PUBLIC使用，并执行自己的安全检查。

30.3.1 FINISH_REPORT_CAPTURE程序

此过程结束完全捕获使用START_REPORT_CAPTURE过程启动的SQL监视器数据。

调用此子程序后，每分钟都会继续捕获数据，除了没有为所有活动的SQL捕获数据，但仅针对那些被认为重要的SQL，即前5个SQL（经过时间，或PQ时的经过时间* DOP）监视已经完成了。

句法

DBMS_AUTO_REPORT.FINISH_REPORT_CAPTURE;

1.1全球化支持架构

Oracle数据库中的全球化支持使您能够以本机语言存储，处理和检索数据。它确保数据库实用程序，错误消息，排序顺序以及日期，时间，货币，数字和日历约定自动适应任何本地语言和区域设置。

过去，Oracle将全球化支持功能称为国家语言支持（NLS）功能。NLS实际上是全球化支持的一个子集。NLS能够选择国家语言并将数据存储在特定字符集中。全球化支持使您能够开发可以在世界任何地方同时访问和运行的多语言应用程序和软件产品。应用程序可以呈现用户界面的内容，并以本机用户的语言和区域设置首选项处理数据。

1.1.1区域设置数据点播

Oracle数据库全球化支持通过Oracle NLS运行时库（NLSRTL）实现。NLS RTL提供了一整套独立于语言的函数，可以执行正确的文本和字符处理以及语言约定操作。这些函数对特定语言和区域的行为由一组在运行时标识和加载的特定于语言环境的数据控制。

特定于语言环境的数据被构建为Oracle数据库支持的每个语言环境的独立数据集。可以独立于其他区域设置数据加载特定区域设置的数据。

这种设计的优点如下：

  • 您可以通过选择所需的语言环境集来管理内存消耗。

  • 您可以添加和自定义特定区域设置的区域设置数据，而不会影响其他区域设置。

1.1.2支持多语言应用的架构

Oracle数据库使多层应用程序和客户端/服务器应用程序能够支持配置数据库的语言。

依赖于语言环境的操作由客户端和数据库服务器上的多个参数和环境变量控制。在数据库服务器上，代表客户端启动的每个会话可以在与其他会话相同或不同的语言环境中运行，并且可以指定相同或不同的语言要求。

Oracle数据库具有一组与会话无关的NLS参数，这些参数在创建数据库时指定。其中两个参数指定数据库字符集和国家字符集，这是可以为NCHAR ， NVARCHAR2和NCLOB数据指定的备用Unicode字符集。参数指定用于在数据库中存储文本数据的字符集。其他参数（如语言和区域）用于评估和检查约束。

如果客户端会话和数据库服务器指定了不同的字符集，则数据库会自动转换字符集字符串。

从全球化支持的角度来看，所有应用程序都被视为客户端，即使它们与Oracle数据库实例在同一台物理计算机上运行。例如，当SQL用户从安装了RDBMS软件的Oracle主目录中拥有Oracle数据库软件的UNIX用户启动时，SQL * Plus通过指定ORACLE_SID参数通过适配器连接到数据库，SQL * Plus被视为客户。它的行为由客户端NLS参数决定。

1.1.3在多语言数据库中使用Unicode

Unicode是通用编码字符集，使您可以使用单个字符集以任何语言存储信息。Unicode为每个角色提供唯一的代码值，无论平台，程序或语言如何。Oracle建议使用AL32UTF8作为数据库字符集。AL32UTF8是Unicode标准的UTF-8编码形式的正确实现。

注意：

从Oracle Database 12 c第2版开始，如果使用Oracle Universal Installer（OUI）或Oracle Database Configuration Assistant（DBCA）创建数据库，则使用的默认数据库字符集是Unicode字符集AL32UTF8 。

Unicode具有以下优点：

  • 简化字符集转换和语言排序功能。

  • 与本机多字节字符集相比，提高了性能。

  • 支持基于Unicode标准的Unicode数据类型。

  • 它支持选择性地转换数据。

  • 它提供进度监控。

4用于DRDA的Oracle数据库提供程序的SQL转换和示例

SQL Translation是Oracle Database 12 c第2版（12.2）中的一项新功能。在早期版本的Oracle数据库中，可以在没有SQL Translation的情况下使用Oracle Provider for DRDA。

有关SQL Translation的更多信息，请参阅Oracle®数据库迁移指南 。

注意：

SQL Translation是一个可选功能，建议在应用程序中使用的SQL更多以DB2为中心的语法中。

SQL转换过程概述

Oracle Database Release 12 c引入了SQL Translation的概念。此功能允许将“外部”SQL语句（例如DB2）转换为Oracle数据库可以正确使用的SQL语法。SQL Translation本身是通过SQL转换器实现的，该转换器通常由翻译的第三方提供。SQL Translator检查输入SQL，有时会将其更改为遵循Oracle SQL语法。通过SQL转换程序接口程序包指定的SQL转换配置文件指定使用的SQL转换程序。

SQL Translation Profile是SQL TRANSLATION PROFILE类型的模式级对象。它通过ATTR_TRANSLATOR属性引用PL / SQL包;这个包称为SQL转换器接口包。该程序包指定在SQL转换配置文件处于活动状态时执行SQL转换的第三方SQL转换程序。一次只能启用一个SQL转换配置文件。

SQL翻译配置文件可以在用户之间共享。通常，所有用户共享一组包的相同单个SQL翻译配置文件，但这不是必需的。

请注意，每个DB2包可以通过为该包保留的属性与SQL Translation配置文件相关联。在为DB2包准备SQL语句时，将使用与为DB2包指定的SQL转换配置文件关联的SQL转换程序。

实现SQL转换

要进行翻译，必须执行以下事件序列：

  1. 获得SQL转换器。
  2. 创建引用该转换程序的SQL接口包。
  3. 创建引用SQL接口包的SQL转换配置文件。

    该步骤可以在实例的生命周期中仅执行一次。但是，必须至少执行一次才能使用SQL Translation。

    在具有多个翻译器的情况下，或者需要不同的SQL翻译配置文件的情况下，可以重复该过程。

  4. DB2包与SQL转换配置文件的关联。

    必须为创建的每个包完成此步骤。

    请注意，在与转换配置文件关联之前，不必创建包;只需要包的名称。此步骤不会验证特定包是否已存在。

  5. 在执行时，用户通过包将SQL文本传递给Oracle数据库提供程序以获取DRDA。
  6. 当Oracle数据库提供程序用于DRDA获取SQL文本时，它会检查该程序包是否与SQL转换配置文件关联，然后将该SQL转换配置文件设置为在解析和执行SQL文本期间生效。
  7. 在Oracle数据库提供程序for DRDA准备SQL文本以供执行之后，Oracle数据库使用当前的SQL转换配置文件来转换SQL语句，然后执行它们。

SQL转换的要求

成功的SQL转换只能在以下情况下发生：

  • 必须通过以下命令为会话启用SQL转换配置文件：

    ALTER SESSION SET SQL_TRANSLATION_PROFILE

  • 该进程必须指定传入的SQL语句使用外部语法，或者使用非Oracle SQL方言。在这里讨论的所有情况中，这些方言都是DB2 SQL的变体。

对于DRDA产品的Oracle数据库提供程序，前两个条件是耦合的;如果DB2包与SQL转换配置文件相关联，那么SQL语句应该是外部语法，并且调用与SQL转换配置文件关联的SQL转换器来转换该包中的任何SQL。

翻译特定于DB2的SQL语法的细节

虽然可以直接执行客户端应用程序提交给Oracle数据库提供程序以获取DRDA的大多数SQL构造，但Oracle无法识别某些DB2 SQL构造。考虑在转换由重新配置为使用Oracle数据库实例的应用程序发出的DB2 SQL语句时发生的已知问题。

如果存在SQL转换配置文件，则可以将与该配置文件关联的SQL转换程序设计为更改这些SQL语句，以便应用程序在Oracle SQL中执行等效或类似操作，并返回预期结果。

SQL转换器接口包

SQL转换配置文件是通过DBMS_SQL_TRANSLATOR.CREATE_PROFILE()过程创建的SQL TRANSLATION PROFILE类型的模式级对象。SQL转换器接口包是某种格式的PL / SQL包;它引用第三方提供的转换器对象，并且本身由SQL转换配置文件引用。因此，SQL Translator Interface Package连接SQL Translation Profile和第三方提供的SQL转换器对象。

关于SQL转换器接口包

SQL转换配置文件引用具有固定格式的PL / SQL包装程序包，即SQL转换程序接口程序包 。当会话设置SQL TRANSLATION PROFILE ，它指定所有SQL都由与SQL转换器接口包关联的第三方SQL转换器进行转换。SQL Translator Interface Package的过程translate_sql()执行转换。

注意，Oracle 不提供SQL转换。相反，SQL转换器必须从第三方供应商处获得，或在内部开发。Oracle提供了各种管理脚本，用于创建和管理SQL转换配置文件。

验证SQL转换程序配置文件

以下步骤验证SQL转换配置文件是否已正确安装并完全启用。

要验证SQL转换程序配置文件：

  1. 使用SYSDBA权限登录Oracle数据库
  2. 检查转换程序配置文件是否已加载到Oracle数据库中。
    SELECT * FROM ALL_SQL_TRANSLATION_PROFILES;
  3. 使用具有DRDAAS_USER_ROLE权限的ID登录。
  4. 确保设置角色：
    SET ROLE DRDAAS_USER_ROLE;
  5. 将会话的SQL转换配置文件设置为创建SQL转换配置文件时指定的值。
    ALTER SESSION SET SQL_TRANSLATION_PROFILE = DRDAAS_TRANS_ADMIN.MY_PROFILE;
  6. 尝试以下命令：
    ALTER SESSION SET EVENTS ='10601永远跟踪名称上下文，等级32';来自DUAL的SELECT CAST（1234567 AS BIGINT）; ALTER SESSION SET EVENTS ='10601 trace name context off';来自DUAL的SELECT CAST（1234567 AS BIGINT）;

    • 外来语法（永远跟踪名称上下文，级别32）

    • 本机Oracle语法（跟踪名称上下文关闭）

    这仅在使用SQL * Plus时有效。

更改SQL转换配置文件

有时，有必要完全更改SQL转换配置文件，并使DB2包的SQL TRANSLATION PROFILE属性引用新的SQL转换配置文件。

DB2包通常以集合形式出现，DB2包的名称由客户端确定。但是，如果客户端使用ODBC访问Oracle数据库提供程序以获取DRDA，则ODBC驱动程序将确定程序包的名称。

Oracle提供了两个脚本，可用于为一组包设置SQL Translation Profile属性。

  • 如果DataDirect ODBC驱动程序访问用于DRDA的Oracle数据库提供程序，请使用drdaas/admin目录中的drdasqt_set_profile_dd.sql脚本。

  • 如果IBM ODBC驱动程序访问用于DRDA的Oracle数据库提供程序，请使用drdaas/admin目录中的drdasqt_set_profile_ibm.sql脚本。本机客户端应用程序也可以使用此脚本，但可能必须进行扩展。

可以复制和更改这些脚本以与其他DB2包集合一起使用。

此外，具有DRDAAS_ADMIN_ROLE权限的用户必须在SQL * Plus中运行这些脚本中的每一个。该脚本会提示输入各种包引用的配置文件的限定名称（例如DRDAAS_TRANS_ADMIN.MY_PROFILE ）。它还会提示输入默认的Package Collection架构，该架构通常为NULLID 。

5.1集合数据类型

Oracle支持varray和嵌套表集合数据类型。

  • varray是有序的元素集合。

  • 嵌套表可以包含任意数量的元素，并且是无序的。

如果您只需要存储固定数量的项目，或者按顺序遍历元素，或者经常需要检索和操作整个集合作为值，那么请使用varray。

如果需要对集合运行高效查询，处理任意数量的元素，或执行批量插入，更新或删除操作，则使用嵌套表。

  • 您不能使用SELECT语句将远程临时LOB定位器选择到局部变量中。例如，

    从lob_tab @ db2中选择substr（c2,3,1），其中c1 = 1

  • 索引组织表（IOT）不支持远程lob功能。尝试从远程IOT表获取定位器将导致错误。

  • 本地数据库和远程数据库都必须是Database 12.2或更高版本。

  • 对于分布式LOB功能， from子句或where子句中提到的表应该在同一个数据库中并置。如果远程定位器在where子句中用作绑定变量，则它们应属于同一个远程数据库。您不能将DB1中的一个定位器和DB2中的另一个定位器用作绑定变量。

新的数据库角色强制执行职责分离

Oracle Database Vault配置实现了职责分离的概念，因此您可以满足法规，隐私和其他合规性要求。

Oracle Database Vault明确区分了数据库内的帐户管理职责，数据安全责任和数据库管理职责。这意味着超级特权用户（例如， DBA ）的概念分为几个新的数据库角色，以确保没有一个用户可以完全控制系统的数据和配置。Oracle Database Vault可以防止特权用户（具有DBA和其他特权角色和系统特权的用户）访问名为realms的数据库的指定保护区域。它还引入了称为Oracle Database Vault Owner（ DV_OWNER ）和Oracle Database Vault Account Manager（ DV_ACCTMGR ）的新数据库角色。这些新的数据库角色将数据安全性和帐户管理与传统的DBA角色分开。您应该将这些角色映射到组织内的不同安全专业人员。

PGA_CAPABILITY参数注意事项

选择设置时 PGA_CAPABILITY参数，请注意确保根据RTP将要执行的操作使用正确的设置。

该应该始终使用READ_ONLY设置 RTP是只读的，也就是说，当RTP不执行数据库更新时。当RTP执行数据库更新时，永远不应该使用READ_ONLY 。例如，如果选择了READ_ONLY设置，并且如果网关调用的RTP执行对外部数据库的更新，则 Oracle数据库不提供这些更新任何完整性保护。此外， READ_ONLY模式允许网关事务成为可能更新多个其他数据库的分布式事务的一部分。如果网关调用在此情况下执行更新的RTP，并且如果发生故障，则RTP更新的数据库与其他数据库不同步。

在RTP对外部数据库执行更新的情况下，有 PGA_CAPABILITY两个选项：

  • SINGLE_SITE

  • COMMIT_CONFIRM

这些选项中的每一个都允许通过许可来防止数据完整性问题 COMMIT和 ROLLBACK请求转发到RTP并通知 Oracle数据库，关于网关的分布式更新和恢复功能。选择的特定选项取决于RTP的设计以及RTP的能力 OLTP执行的位置。

如果OLTP具有LU6.2 SYNCLEVEL 1或2支持，那么 COMMIT_CONFIRM功能提供了有限的两阶段提交 Oracle数据库和OLTP，没有其他限制 commit-confirm站点（网关或Oracle）可以是分布式事务的一部分。如果无法使用COMMIT_CONFIRM ，那么SINGLE_SITE能力提供了Oracle数据库和更新之间的能力 OLTP，限制只有OLTP可以执行更新，并且Oracle端不会发生更新。

用于更新控制的每个PGA_CAPABILITY选项都对RTP和OLTP施加了特定要求。对于COMMIT_CONFIRM功能，这些要求将在Oracle Database Gateway for APPC用户指南的第5章“实现提交确认”中详细讨论。另请参阅本指南中的配置OLTP以进行提交确认 配置OLTP以进行提交确认 。对于SINGLE_SITE功能，RTP负责执行适当的任务以响应代表网关从网关收到的COMMIT和ROLLBACK请求。 Oracle数据库。网关使用APPC CONFIRM和SEND_ERR请求分别实现COMMIT和ROLLBACK 。收到CONFIRM命令后，RTP必须执行COMMIT处理，然后使用APPC CONFIRMED响应响应网关。收到SEND_ERR命令后，RTP必须执行ROLLBACK处理。

因为分布式事务的能力 Oracle数据库受网关使用的PGA_CAPABILITY选项的影响，希望通过为每个应用程序使用不同的网关实例来分离查询和更新应用程序。一个网关可以被定义PGA_CAPABILITY设置为READ_ONLY和其他人PGA_CAPABILITY设置为SINGLE_SITE或COMMIT_CONFIRM 。

这允许只读事务程序在Oracle数据库的控制下参与分布式事务。例如，来自DB2的数据可以通过READ_ONLY网关通过仅查询RTP检索，然后可以在一个Oracle事务中用作Oracle数据库上数据库更新的输入。甲SINGLE_SITE网关只能用于访问其中的Oracle数据库控制的范围之外执行更新国外数据库的RTP一起使用。可以从Oracle数据库可访问的任何数据库中读取数据，并且该数据可用于通过网关执行更新。

PGA_CONFIRM参数注意事项

在决定设置时 PGA_CONFIRM 参数，了解每个设置的效果很重要。首先，请记住，此参数仅影响在SYNCLEVEL 1运行的那些会话。默认设置PGA_CONFIRM=REJECT适用于大多数应用程序。使用此设置，网关生成一个如果从远程事务程序收到CONFIRM请求，则会出错。如果你有一个使用远程交易CONFIRM以验证数据是由网关接收，则必须使用PGA_CONFIRM=ACCEPT ，让通往这些传入响应CONFIRM与请求CONFIRMED响应。您必须知道，当Oracle应用程序发出COMMIT请求时，网关会向远程事务发送CONFIRM请求。为了使COMMIT处理正常工作，必须编写远程事务，以便在从网关收到CONFIRM请求时执行其本地提交处理，并在提交处理成功完成后使用CONFIRMED响应网关。如果在提交处理期间发生错误，然后远程事务必须使用SEND_ERR响应网关以指示提交失败。

使用PGA_CONFIRM=ACCEPT一个特例是IMS / TM版本7。使用IMS / TM版本7提供的隐含APPC支持时，在SYNCLEVEL 1运行的会话的处理方式与在SYNCLEVEL 0.运行的会话的处理方式不同SYNCLEVEL 0. 当会话处于SYNCLEVEL 1时，IMS / TM会在每次APPC SEND后自动生成CONFIRM请求。 在网关侧，如果未指定PGA_CONFIRM=ACCEPT ，则IMS / TM发送的CONFIRM请求将导致网关生成错误。使用PGA_CONFIRM=ACCEPT缓解这一问题，使网关进入的回应CONFIRM与请求CONFIRMED响应。以这种方式运行的唯一限制是IMS提供的隐含APPC支持在从网关收到CONFIRM请求时不通知应用程序。这意味着网关不能使用CONFIRM来实现COMMIT ，从而禁止使用COMMIT/ROLLBACK来控制对话的IMS端的更新。

如果在将expr转换为type_name时发生错误，则此子句允许您指定此函数返回的值。如果在计算expr时发生错误，则此子句无效。

如果expr计算为CHAR ， VARCHAR2 ， NCHAR或NVARCHAR2类型的字符串，并且type_name为BINARY_DOUBLE ， BINARY_FLOAT ， DATE ， INTERVAL DAY TO SECOND ， INTERVAL YEAR TO MONTH ， NUMBER ， TIMESTAMP ， TIMESTAMP WITH TIME ZONE ，则此子句有效或TIMESTAMP WITH LOCAL TIME ZONE 。

return_value可以是字符串文字，null，常量表达式或绑定变量，并且必须求值为null或CHAR ， VARCHAR2 ， NCHAR或NVARCHAR2类型的字符串。如果return_value无法转换为type_name ，则该函数返回错误。

fmt和nlsparam

fmt参数允许您指定格式模型，而nlsparam参数允许您指定NLS参数。如果指定这些参数，则在将expr和return_value （如果指定）转换为type_name时将应用它们。

如果type_name是以下数据类型之一，则可以指定fmt和nlsparam ：

  • BINARY_DOUBLE

    如果指定BINARY_DOUBLE ，则可选的fmt和nlsparam参数与TO_BINARY_DOUBLE函数的用途相同。有关更多信息，请参阅TO_BINARY_DOUBLE 。

  • BINARY_FLOAT

    如果指定BINARY_FLOAT ，则可选的fmt和nlsparam参数与TO_BINARY_FLOAT函数的用途相同。有关更多信息，请参阅TO_BINARY_FLOAT 。

  • 日期

    如果指定DATE ，则可选的fmt和nlsparam参数与TO_DATE函数的用途相同。有关更多信息，请参阅TO_DATE 。

  • 数

    如果指定NUMBER ，则可选的fmt和nlsparam参数与TO_NUMBER函数的用途相同。有关更多信息，请参阅TO_NUMBER 。

  • TIMESTAMP

    如果指定TIMESTAMP ，则可选的fmt和nlsparam参数用于与TO_TIMESTAMP函数相同的目的。如果省略fmt ，则expr必须采用TIMESTAMP数据类型的缺省格式，该格式由NLS_TIMESTAMP_FORMAT参数显式确定，或由NLS_TERRITORY参数隐式确定。有关更多信息，请参阅TO_TIMESTAMP 。

  • TIMESTAMP WITH TIME ZONE

    如果指定TIMESTAMP WITH TIME ZONE ，则可选的fmt和nlsparam参数与TO_TIMESTAMP_TZ函数的用途相同。如果省略fmt ，则expr必须采用TIMESTAMP WITH TIME ZONE数据类型的默认格式，该格式由NLS_TIMESTAMP_TZ_FORMAT参数显式确定，或由NLS_TERRITORY参数隐式确定。有关更多信息，请参阅TO_TIMESTAMP_TZ 。

  • TIMESTAMP WITH LOCAL TIME ZONE

    如果指定TIMESTAMP WITH LOCAL TIME ZONE则可选的fmt和nlsparam参数与TO_TIMESTAMP函数的用途相同。如果省略fmt ，则expr必须采用TIMESTAMP数据类型的缺省格式，该格式由NLS_TIMESTAMP_FORMAT参数显式确定，或由NLS_TERRITORY参数隐式确定。有关更多信息，请参阅TO_TIMESTAMP 。
