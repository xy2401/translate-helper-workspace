2.12创建目录

目的

使用CREATE CATALOG命令创建基本恢复目录。将目录权限授予虚拟专用目录所有者时，将自动创建虚拟专用目录。

  • 基本恢复目录是包含一组目标数据库的RMAN元数据的数据库模式。

  • 虚拟专用目录是一组安全策略，用于限制用户对基本恢复目录子集的访问。

也可以看看：

  • Oracle数据库备份和恢复用户指南 ，了解如何创建恢复目录

  • “ RMAN兼容性 ” ，了解恢复目录与RMAN环境的其他组件的兼容性要求

先决条件

仅在RMAN提示符下执行此命令。必须通过CATALOG命令行选项或CONNECT CATALOG命令将RMAN连接到恢复目录数据库，并且必须打开目录数据库。不需要与目标数据库的连接。

必须为基本恢复目录的恢复目录所有者授予RECOVERY_CATALOG_OWNER角色。恢复目录在恢复目录所有者的默认表空间中创建。

注意：

从Oracle Database 12c第1版（12.1.0.2）开始，恢复目录数据库必须使用Enterprise Edition。

拥有虚拟专用目录的数据库用户必须被授予CREATE SESSION特权。从Oracle Database 12 c第1版（12.1.0.2）开始，只有在使用Oracle数据库企业版时才能创建虚拟专用目录。

如果要创建虚拟专用目录，则基本恢复目录所有者必须使用RMAN GRANT命令授予其中一个 CATALOG或REGISTER权限（参见例2-71 ）。

有关RMAN客户端来自发行版Oracle数据库10g或更早版本的RMAN客户端与虚拟目录的连接的限制，请参阅CONNECT CATALOG说明。

使用说明

通常，您在专门为此目的创建的数据库中创建恢复目录。不要在特权架构（如SYS或SYSBACKUP创建恢复目录。

最佳做法是创建一个恢复目录，作为许多数据库的中央RMAN存储库。因此，它被称为基本恢复目录 。

基本恢复目录的拥有者可以GRANT或REVOKE到类别到其他数据库用户的受限接入。每个受限用户都对其自己的元数据具有完全读/写访问权限，称为虚拟专用目录 。RMAN元数据存储在虚拟专用目录所有者的模式中。基本恢复目录的所有者控制每个虚拟目录用户可以访问的内容。

要删除文件，您必须具有相应诊断目标的操作系统权限。

    $ tfactl managelogs -purge -older 30d -dryrun $ tfactl managelogs -purge -older 30d

20.2管理磁盘使用快照

使用tfactl命令管理Oracle跟踪文件分析器磁盘使用情况快照。

Oracle跟踪文件分析器自动监视磁盘使用情况，记录快照并将快照存储在tfa_install_dir /tfa/repository/suptools/node/managelogs/usage_snapshot/

管理磁盘使用情况快照：

  1. 要更改快照的默认时间间隔：
    $ tfactl set diskUsageMonInterval = minutes

    其中minutes是快照之间的分钟数。

  2. 要打开或关闭磁盘使用情况监视器：
    $ tfactl set diskUsageMon = ON | OFF

只读

                  
READ ONLY子句将当前事务建立为只读事务。该子句建立了事务级读取一致性 。
                  

                  
该事务中的所有后续查询仅查看在事务开始之前提交的更改。只读事务对于针对一个或多个表运行多个查询的报表很有用，而其他用户更新这些相同的表。

                  
用户SYS不支持此子句。即使SYS已将事务设置为READ ONLY ， SYS查询也将返回事务期间所做的更改。

7.3编写顶级呼叫规范

         

         

            

               

                  
在SQL * Plus中，您可以使用以下语法以交互方式定义顶级调用规范：
CREATE [OR REPLACE] {PROCEDURE procedure_name [（param [，param] ...）]|功能function_name [（param [，param] ...）]RETURN sql_type} [AUTHID {DEFINER | CURRENT_USER}] [PARALLEL_ENABLE] [确定性] {IS | AS} LANGUAGE JAVA NAME'method_fullname（java_type_fullname [，java_type_fullname] ...）[return java_type_fullname]';
其中param由以下语法表示：
parameter_name [IN | OUT | IN OUT] sql_type
该 AUTHID子句确定以下内容：

               

               
               

                  
    
                     
  • 
                        
    存储过程是否以其定义者（ AUTHID DEFINER ）或调用者（ AUTHID CURRENT_USER ）的特权运行
    
                     
    • 
                     
  • 
                        
    是否在定义者或调用者的模式中解析了对模式对象的非限定引用
    
                     
    • 
                  

                  
如果未指定AUTHID ，则默认行为为DEFINER ，即存储过程以其定义者的权限运行。您可以通过将AUTHID指定为CURRENT_USER来覆盖默认行为。但是，您不能通过指定CURRENT_USER来覆盖loadjava选项-definer 。

                  
该 PARALLEL_ENABLE选项声明可以在并行DML评估的从属会话中安全地使用存储的函数。永远不会与从会话共享主会话的状态。每个从属会话都有自己的状态，在会话开始时初始化。函数结果不应该取决于会话变量的状态。否则，结果可能因会话而异。
                  

                  
 DETERMINISTIC选项可帮助优化器避免冗余函数调用。如果先前使用相同的参数调用了存储函数，则优化器可以决定使用以前的结果。函数结果不应取决于会话变量或模式对象的状态。否则，结果可能会因呼叫而异。只能从基于函数的索引或启用了查询重写的物化视图中调用DETERMINISTIC函数。
                  

                  
中的字符串 NAME子句唯一标识Java方法。完全限定的Java名称和按位置映射的调用规范参数必须对应。但是，此规则不适用于main()方法。如果Java方法不接受任何参数，则为其写入空参数列表，但不为函数或过程写入。
                  

                  
使用点表示法编写完全限定的Java名称。以下示例显示完全限定名称可以在点边界处的线上划分：

查找和重置使用10G密码版本的用户密码

选择数据库名称

         

         

            

               
数据库名称由各种字符串组成，并且只能包含允许的字符。选择数据库名称时，请查看以下准则。

               
数据库名称输入字段设置以下Oracle初始化参数值：

               
    
                  
  • 
                     
    DB_NAME
    
                  
    • 
                  
  • 
                     
    DB_UNIQUE_NAME
    
                  
    • 
                  
  • 
                     
    DB_DOMAIN
    
                  
    • 
               

               
在Oracle RAC环境中，数据库名称（ DB_UNIQUE_NAME ）部分是一个不超过30个字符的字符串，可以包含字母数字，下划线（_），美元（$）和磅（＃）字符，但必须以字母字符开头。数据库名称中不允许使用其他特殊字符。数据库的DB_NAME参数设置为数据库名称的前8个字符。
               

               
全局数据库名称（ DB_DOMAIN ）的域部分不能超过128个字符。不允许使用下划线（_）的域名。DB_UNIQUE_NAME.DB_DOMAIN的整体值必须在企业内是唯一的。
               

               

                  
注意：
对于Oracle Real Applications Cluster（Oracle RAC）数据库，可插拔数据库（PDB）名称在群集中必须是唯一的。
               

               

                  
数据库名称和ORACLE_SID

                  
Oracle服务标识符（SID）前缀是数据库名称的前8个字符。SID前缀只能包含字符az，AZ和0-9。SID前缀不能包含操作系统特殊字符，因此如果在数据库名称的前8个字符中使用特殊字符，则在SID前缀中省略这些特殊字符。每个数据库都有一个SID前缀。数据库的SID前缀在群集中必须是唯一的。

                  
对于Oracle RAC数据库，每个实例都有一个唯一标识符ORACLE_SID ，它由SID前缀和实例编号组成。ORACLE_SID前缀最多可包含12个字符。Oracle RAC数据库实例的ORACLE_SID生成方式不同，具体取决于您选择如何管理数据库。如果选择策略管理的数据库，则Oracle将以name_#格式生成SID，其中name是DB_UNIQUE_NAME的前八个字母数字字符，而#是实例编号。如果选择管理员管理的数据库，则Oracle Database Configuration Assistant将使用格式name#生成实例名称的默认SID，其中name是DB_UNIQUE_NAME的前八个字母数字字符，而#是实例编号。但是，在安装或数据库创建期间，您可以为SID指定非默认值。对于每个实例，实例编号会自动添加到此字符串的末尾。
                  

                  
对于Oracle RAC One Node数据库，实例名称为ORACLE_SID_1 ，它由附加到SID前缀的_1组成。在线重定位期间，将启动第二个实例ORACLE_SID_2 ，它将成为重定位完成后的唯一实例。下一次在线重定位使用ORACLE_SID_1作为新实例。

2 SQL转换框架概述

         

         

            

               
旨在与非Oracle数据库一起使用的各种客户端应用程序无法在没有重大更改的情况下与Oracle数据库一起使用。这是因为SQL方言因数据库技术供应商而异，不同的供应商使用不同的语法来表达SQL查询和语句。

               
从Oracle Database 12 c开始 ，有一种称为SQL Translation Framework的新机制。它将客户端程序的SQL语句从外部（非Oracle）SQL方言转换为Oracle数据库SQL编译器使用的SQL方言。
               

               
除了翻译非Oracle SQL语句之外，SQL Translation Framework还可用于将Oracle SQL语句替换为另一个Oracle语句，以解决语义或性能问题。通过这种方式，您可以解决应用程序问题，而无需修补客户端应用程序。

               
SQL转换框架包含两个基本组件：SQL转换程序和SQL转换配置文件。

               

                  
SQL转换器

                  
SQL转换程序是由Oracle或第三方供应商提供的软件组件，可以安装在Oracle数据库中。它在Oracle Database SQL编译器处理客户端程序之前对其进行转换。如果由转换的SQL语句执行导致错误，则Oracle Database SQL编译器会生成Oracle错误消息。

                  
SQL Translator自动将非Oracle SQL转换为Oracle SQL，从而使现有的客户端应用程序代码能够在很大程度上保持对Oracle数据库的运行。这大大降低了迁移到Oracle数据库存储的成本。作为推论，翻译特征可以用在其他场景中，其中可以有利地介入客户端提交的原始SQL语句与其实际执行之间。

               

               
               

                  
SQL翻译配置文件

                  
SQL Translation Profile是一个数据库对象，包含一组捕获的非Oracle SQL语句及其翻译或转换错误。SQL Translation Profile用于审阅，批准和修改翻译。配置文件与单个翻译器相关联。但是，翻译器可以用在一个或多个SQL翻译配置文件中。通常，每个应用程序都有一个SQL Translation Profile，否则应用程序可以共享已翻译的查询。您可以在各种数据库中导出配置文件

如何使用SQL Translation Framework

               

                  

                     
执行以下步骤以使用SQL Translation Framework：

                  

                  
                  
    
                     
  1. 在Oracle数据库中安装Oracle转换程序，可以是Oracle或第三方供应商。
    2. 
                     
  2. 创建SQL转换配置文件并使用配置文件注册SQL转换器。
    3. 
                     
  3. 创建数据库服务并将SQL转换配置文件指定为应用程序可以连接的服务属性。
    
                           
    请注意，在服务级别设置SQL Translation Profile可确保自动翻译通过该侦听器服务运行的所有内容。
    
                           
    也可以使用ALTER SESSION语句或LOGON触发器在连接级别激活转换程序。
                           
    
                        
    
                     
    4. 
                     
  4. 将应用程序与Oracle驱动程序链接，以将应用程序连接到Oracle数据库。您还必须更改连接设置以使用SQL转换配置文件连接到数据库服务。
    5. 
                     
  5. 根据Oracle数据库测试应用程序的所有功能。在应用程序运行时，SQL Translation Profile将应用程序的SQL语句从第三方SQL方言转换为语义等效的Oracle语法，并在配置文件中注册它们。
    
                           
    如果翻译器没有特定SQL语句或错误的翻译，那么您可以注册自己的翻译来填补其位置。
    
                        
    
                     
    6. 
                     
  6. 验证自定义翻译并根据需要进行编辑。或者，注册新的应用程序以确保应用程序按预期执行，直到测试完成。
    
                           
    Oracle建议建立一个测试环境并严格测试应用程序，最好是通过回归测试套件。
    
                        
    
                     
    7. 
                     
  7. 在生产Oracle数据库中设置服务器端应用程序对象和数据，以便部署到生产环境。
    8. 
                     
  8. 创建数据库服务，将概要文件设置为服务属性，并更改应用程序的连接设置，以便它连接到生产数据库中的数据库服务。该应用程序预计将按测试运行。
    9. 
                  

                  

                     
Oracle建议监视应用程序，以防止由于任何SQL语句的转换不可用而导致错误的可能性。您必须首先禁用配置文件中新的和未看到的SQL语句的自动转换;遇到任何此类语句时，会引发记录的错误。如果出现错误翻译警报，您必须对配置文件进行调整。

何时使用SQL Translation Framework

               

                  
使用SQL Translation迁移使用具有供应商专有SQL语法的SQL语句的客户端应用程序。

                  
目前，SQL Translators仅适用于Sybase和SQL Server，并且对DB2的支持有限。

                  
SQL Translation Framework设计用于开放API应用程序（如ODBC或JDBC），以及使用可能转换为语义等效Oracle语法的SQL语句的应用程序。这些应用程序必须重新链接到Oracle ODBC或JDBC驱动程序，然后通过转换服务执行。

                  
以下是连接机制的可能方案：

                  
    
                     
  • 
                        
    如果应用程序使用ODBC，JDBC，OLE DB或.NET驱动程序或数据提供程序连接到数据库，则必须替换Oracle的驱动程序或数据提供程序。

22.1关于NMF

               

                  
非负矩阵分解是最先进的特征提取算法。当存在许多属性且属性不明确或具有弱可预测性时，NMF非常有用。通过组合属性，NMF可以生成有意义的模式，主题或主题。

                  
NMF创建的每个要素都是原始属性集的线性组合。每个要素都有一组系数，这些系数衡量要素上每个属性的权重。有一个单独的每个数字属性的系数和每个分类属性的每个不同值。系数都是非负的。
                  

               
22.1.1矩阵分解

                  

                     
非负矩阵分解使用来自多变量分析和线性代数的技术。它将数据作为矩阵M分解为两个较低等级矩阵W和H的乘积。子矩阵W包含NMF基础;子矩阵H包含相关系数（权重）。
                     

                     
该算法迭代地修改W和H的值，使得它们的乘积接近M.该技术保留了原始数据的大部分结构，并保证基础和权重都是非负的。当近似误差收敛或达到指定的迭代次数时，算法终止。
                     

                     
必须使用种子初始化NMF算法以指示迭代的起始点。由于处理空间的高维度以及没有全局最小化算法的事实，适当的初始化对于获得有意义的结果是至关重要的。Oracle Data Mining使用随机种子，该种子基于均匀分布初始化W和H的值。这种方法在大多数情况下都很有效。

                  

               
22.1.2使用NMF进行评分

                  

                     
了解使用非负矩阵分解（NMF）进行评分。

                     
NMF可用作分类，回归，聚类和其他挖掘任务中降维的预处理步骤。对NMF模型进行评分会在新要素空间中生成数据投影。投影的大小表示记录映射到要素的强度。

                     
用于特征提取的SQL评分函数支持NMF模型。使用分析语法调用函数时，函数将构建并应用瞬态NMF模型。特征提取功能包括： FEATURE_DETAILS ， FEATURE_ID ， FEATURE_SET和FEATURE_VALUE 。

22.3 NMF的数据准备

               

                  
了解如何准备非负矩阵分解（NMF）的日期。

                  
自动数据准备规范化NMF的数字属性。

                  
当具有简单数据类型（非嵌套）的列中缺少值时，NMF会将它们随机解释为缺失。该算法用模式替换缺失的分类值，用平均值替换缺失的数值。

                  
当嵌套列中缺少值时，NMF会将它们解释为稀疏列。该算法用零和矢量稀疏分类数据替换稀疏数值数据。

                  
如果您选择管理自己的数据准备，请记住异常值会对NMF产生重大影响。之前使用剪辑变换分级或标准化。NMF通常受益于标准化。然而，具有最小 - 最大归一化的异常值导致差的矩阵分解。要改进矩阵分解，需要降低误差容差。这反过来会导致更长的构建时间。

21.1关于NMF

               

                  
非负矩阵分解是最先进的特征提取算法。当存在许多属性且属性不明确或具有弱可预测性时，NMF非常有用。通过组合属性，NMF可以生成有意义的模式，主题或主题。

                  
NMF创建的每个要素都是原始属性集的线性组合。每个要素都有一组系数，这些系数衡量要素上每个属性的权重。有一个单独的每个数字属性的系数和每个分类属性的每个不同值。系数都是非负的。
                  

               
21.1.1矩阵分解

                  

                     
非负矩阵分解使用来自多变量分析和线性代数的技术。它将数据作为矩阵M分解为两个较低等级矩阵W和H的乘积。子矩阵W包含NMF基础;子矩阵H包含相关系数（权重）。
                     

                     
该算法迭代地修改W和H的值，使得它们的乘积接近M.该技术保留了原始数据的大部分结构，并保证基础和权重都是非负的。当近似误差收敛或达到指定的迭代次数时，算法终止。
                     

                     
必须使用种子初始化NMF算法以指示迭代的起始点。由于处理空间的高维度以及没有全局最小化算法的事实，适当的初始化对于获得有意义的结果是至关重要的。Oracle Data Mining使用随机种子，该种子基于均匀分布初始化W和H的值。这种方法在大多数情况下都很有效。

                  

               
21.1.2使用NMF进行评分

                  

                     
了解使用非负矩阵分解（NMF）进行评分。

                     
NMF可用作分类，回归，聚类和其他挖掘任务中降维的预处理步骤。对NMF模型进行评分会在新要素空间中生成数据投影。投影的大小表示记录映射到要素的强度。

                     
用于特征提取的SQL评分函数支持NMF模型。使用分析语法调用函数时，函数将构建并应用瞬态NMF模型。特征提取功能包括： FEATURE_DETAILS ， FEATURE_ID ， FEATURE_SET和FEATURE_VALUE 。

21.3 NMF的数据准备

               

                  
了解如何准备非负矩阵分解（NMF）的日期。

                  
自动数据准备规范化NMF的数字属性。

                  
当具有简单数据类型（非嵌套）的列中缺少值时，NMF会将它们随机解释为缺失。该算法用模式替换缺失的分类值，用平均值替换缺失的数值。

                  
当嵌套列中缺少值时，NMF会将它们解释为稀疏列。该算法用零和矢量稀疏分类数据替换稀疏数值数据。

                  
如果您选择管理自己的数据准备，请记住异常值会对NMF产生重大影响。之前使用剪辑变换分级或标准化。NMF通常受益于标准化。然而，具有最小 - 最大归一化的异常值导致差的矩阵分解。要改进矩阵分解，需要降低误差容差。这反过来会导致更长的构建时间。

说明correlation.eps的说明

                  
               

               
               

                  
目的

               

               
               

                  
CORR函数（参见CORR ）计算Pearson的相关系数，并要求数值表达式作为输入。CORR_ *函数支持非参数或秩相关。它们允许您查找按顺序缩放的表达式之间的相关性（可以对值进行排序）。相关系数取值范围从-1到1，其中1表示完美关系，-1表示完美的反向关系（当一个变量随着另一个变量减小时增加），并且接近0的值表示没有关系。
                  

                  
这些函数将任何数值数据类型或任何非数字数据类型作为参数，可以隐式转换为数字数据类型。Oracle数据库确定具有最高数字优先级的参数，隐式将剩余参数转换为该数据类型，进行计算并返回NUMBER 。

CORR_S

               

                  

                     
CORR_S计算Spearman的rho相关系数。输入表达式应该是一组（x i ，y i ）个观察对。该函数首先用等级替换每个值。x i的每个值被其在样本中的所有其他x i s中的等级替换，并且y i的每个值被其在所有其他y i s中的等级替换。因此，每个x i和y i取1到n的值，其中n是值对的总数。如果他们的价值略有不同，则会为他们分配平均排名。然后该函数计算等级的线性相关系数。
                     

                  

                  
                  

                     
CORR_S示例

                  

                  
                  

                     
使用Spearman的rho相关系数，以下示例为两个不同的比较中的每一个得出相关系数 - salary和commission_pct ，以及salary和employee_id ：
SELECT COUNT（*）count，CORR_S（salary，commission_pct）佣金，CORR_S（薪水，employee_id）empid FROM employees; COUNT COMMISSION EMPID ---------- ---------- ---------- 107 .735837022 -.04473016

                  
               

            
CORR_K

               

                  

                     
CORR_K计算肯德尔的tau-b相关系数。对于CORR_S ，输入表达式是一组（x i ，y i ）个观察对。为了计算系数，该函数计算一致和不一致对的数量。如果具有较大x的观察值也具有较大的y值，则一对观察结果是一致的。如果使用较大x的观察具有较小的y，则一对观察是不一致的。
                     

                     
tau-b的重要性是由tau-b指示的相关性是由偶然性引起的概率 - 值为0到1。较小的值表示tau-b的正值（或tau-b的负值的反相关）的显着相关性。

                  

                  
                  

                     
CORR_K示例

                  

                  
                  

                     
使用Kendall的tau-b相关系数，以下示例确定员工的工资和佣金百分比之间是否存在相关性：
SELECT CORR_K（salary，commission_pct，'COEFFICIENT'）系数，CORR_K（salary，commission_pct，'TWO_SIDED_SIG'）two_sided_p_value FROM employees; COEFFICIENT TWO_SIDED_P_VALUE ----------- ----------------- .603079768 3.4702E-07

目的

                  
使用ALTER VIEW语句显式重新编译无效的视图或修改视图约束。显式重新编译可让您在运行时之前找到重新编译错误。您可能希望在更改其一个基表后显式重新编译视图，以确保更改不会影响视图或依赖于它的其他对象。
                  

                  
您还可以使用ALTER VIEW来定义，修改或删除视图约束。
                  

                  
您不能使用此语句来更改现有视图的定义。此外，如果DDL更改为视图的基表使视图无效，则无法使用此语句编译无效视图。在这些情况下，您必须使用带有OR REPLACE关键字的CREATE VIEW重新定义视图。
                  

                  
发出ALTER VIEW语句时，Oracle数据库会重新编译视图，无论视图是有效还是无效。数据库还会使依赖于视图的任何本地对象无效。
                  

                  
如果更改由一个或多个实例化视图引用的视图，则这些实例化视图将失效。查询重写无法使用无效的实例化视图，因此无法刷新。

36.1 JDBC DMS度量标准概述

               

                  
DMS指标使应用程序和系统开发人员能够测量和导出特定软件组件的自定义性能指标。以下启用DMS的JAR文件中提供了所有DMS度量标准：

                  
    
                     
  • 
                        
    ojdbc6dms.jar
    
                     
    • 
                     
  • 
                        
    ojdbc6dms_g.jar
    
                     
    • 
                     
  • 
                        
    ojdbc7dms.jar
    
                     
    • 
                     
  • 
                        
    ojdbc7dms_g.jar
    
                     
    • 
                  

                  
任何其他JDBC JAR文件都不会生成任何DMS度量标准。Oracle Database 12 c第1版（12.1）中生成的度量标准与10.2，10.1,9.2和早期版本的Oracle JDBC不同，因为它不会尝试保留与早期版本的兼容性。也没有兼容模式。在处理Oracle JDBC 12 c生成的度量标准时，依赖于早期版本的JDBC生成的DMS度量标准的详细信息的系统可能会出现意外行为。这是设计使然无法更改的。
                  

                  
可以针对连接中的所有语句或针对每个语句单独报告语句度量标准。除了与单个语句相关的DMS指标外，所有DMS指标始终处于启用状态。

                  

                     
注意：

                     
您可以启用或禁用SQLText语句指标。默认情况下禁用它。如果启用，则为所有语句启用它。
                     

                  

               

            
36.2关于确定要生成的度量类型

               

                  
要确定是使用统一指标还是单个指标，JDBC会检查DMSConsole传感器权重。如果传感器重量小于或等于DMSConsole.NORMAL ，然后JDBC生成合并语句指标。如果传感器重量大于DMSConsole.NORMAL ，然后JDBC生成单独的语句指标。
                  

                  
JDBC在创建Prepared或Callable语句时检查DMSConsole传感器权重，并根据创建语句时的传感器权重生成度量标准。创建语句后，更改传感器权重的值不会导致语句在统一和单个度量标准之间切换。

                  

                     
注意：

                     
在存在语句缓存的情况下，可能看起来改变传感器权重没有影响，因为从缓存中检索语句而不是重新创建语句。

                  

                  
只为统一和单个语句度量标准生成了一个语句度量标准列表。这两个列表之间的唯一区别是语句的聚合。生成单个语句度量标准时，将为JDBC驱动程序创建的每个不同语句对象生成一组度量标准。另一方面，当生成合并语句度量标准时，给定连接创建的所有语句都使用相同的语句度量标准集。

                  
例如，考虑一个“ execute ”阶段事件。如果使用单个语句度量标准，则创建的每个语句都将具有不同的“ execute ”阶段事件。因此，从两个这样的语句中，可以区分两个单独语句的执行统计信息。如果一个执行时间为1秒而另一个执行时间为3秒，则会有两个不同的“执行”阶段事件，一个具有总时间，平均为1秒，另一个具有总时间和平均值3秒但是，如果使用合并语句度量标准，则所有语句都将使用连接常用的单个“ execute ”阶段事件。因此，从同一连接创建的两个这样的语句中，将无法区分两个语句的执行统计信息。如果一个执行时间为1秒而另一个执行时间为3秒，那么常见的“ execute ”阶段事件将报告总执行时间为4秒，平均为2秒。
                  

               

            
36.3关于生成SQLText指标

               

                  
根据DMS的版本，有两种机制可用于确定SQLText语句度量标准的生成：

                  
    
                     
  • 
                        
    如果classpath环境变量中存在12 c版本的DMS JAR文件，则JDBC将检查DMS更新SQL文本标志。如果此标志为true ，则更新SQLText指标。
                        
    
                     
    • 
                     
  • 
                        
    如果classpath环境变量中不存在12 c版本的DMS JAR文件，则JDBC使用DMSStatementMetrics连接属性的值。如果此statement属性为true ，则更新SQLText指标。此连接属性的默认值为false 。
                        
    
                     
    • 
                  

                  
是否生成SQLText度量标准与使用的语句度量标准类型（即单个语句度量标准或统一语句度量标准）无关。

型号功能：

                              
    
                                 
  • 
                                    
    association - 关联是一种描述性挖掘功能。关联模型识别关系及其在数据集中出现的概率。
                                    
    
                                 
    • 
                                 
  • 
                                    
    attribute_importance - 属性重要性是一种预测挖掘功能。属性重要性模型识别属性在预测给定结果中的相对重要性。
                                    
    
                                 
    • 
                                 
  • 
                                    
    classification - 分类是一种预测挖掘功能。分类模型使用历史数据来预测新的离散或分类数据。
                                    
    
                                    
    classification功能也可用于异常检测。在这种情况下，使用具有空目标的SVM算法（One-Class SVM）。
                                    
    
                                 
    • 
                                 
  • 
                                    
    clustering - 群集是一种描述性挖掘功能。聚类模型识别数据集内的自然分组。
                                    
    
                                 
    • 
                                 
  • 
                                    
    feature_extraction - 特征提取是一种描述性挖掘功能。特征提取模型创建基于模型的优化数据集。
                                    
    
                                 
    • 
                                 
  • 
                                    
    regression - 回归是一种预测性挖掘函数。回归模型使用历史数据来预测新的连续数字数据。
