了解Oracle数据库兼容性

如果新功能与早期版本不兼容，则数据库兼容性可能会导致问题。

如果Oracle Database数据库软件的数据库支持相同的功能，并且这些功能的执行方式相同，则它们是兼容的。升级到新版本的Oracle数据库时，某些新功能可能会使您的数据库与早期版本不兼容。

在以下情况下，升级的数据库与早期版本不兼容：

  • 新功能将任何无法使用早期版本处理的磁盘上的数据（包括数据字典更改）存储起来。

  • 与旧环境相比，现有功能在新环境中的行为不同。

Oracle数据库使您可以控制数据库与COMPATIBLE初始化参数的COMPATIBLE 。

了解COMPATIBLE初始化参数

在Oracle Database 19c中，如果未在参数文件中设置COMPATIBLE初始化参数，则COMPATIBLE参数值默认为19.0.0如果未将COMPATIBLE初始化参数设置为19.0.0 ，则无法使用新的Oracle Database 19c功能，因为已升级的数据库未在Oracle Database 19c功能所需的COMPATIBILITY设置中运行。

当Oracle Database COMPATIBLE参数增加到19.0.0 ，对数据库的第一次Java调用将启动“名称转换”操作。此操作可能需要几分钟才能完成。在增加兼容性参数后，第一次对数据库进行Java调用时，应该会出现此延迟。执行名称转换的初始延迟仅在初始Java调用期间发生。

注意：

  • 在升级到Oracle Database 19c之前，必须将COMPATIBLE初始化参数设置为至少11.2.0 ，这是Oracle Database 19c的最低设置。

  • 兼容参数必须至少为3个十进制数，以句点分隔。例如：

    SQL> ALTER SYSTEM SET COMPATIBLE = '12 .1.0.2'SCOPE = SPFILE;

  • Oracle建议您在对已升级的数据库进行全面测试后才提高COMPATIBLE参数。

  • 增加COMPATIBLE参数后，无法降级数据库。

警告：

如果要从Oracle Database 11.2升级，则必须将兼容值设置为至少11.2.0。您必须在升级时执行此操作。在准备升级之前不要进行此更改，因为在提高COMPATIBLE初始化参数值后，无法降级到较早的兼容级别。

COMPATIBLE初始化参数如何在Oracle数据库中运行

COMPATIBLE初始化参数根据版本兼容性启用或禁用Oracle数据库功能

COMPATIBLE初始化参数按以下方式操作：

  • COMPATIBLE初始化参数启用或禁用功能，以帮助保护现有应用程序对数据的使用。

    如果运行COMPATIBLE初始化参数设置为11.2.0的Oracle Database 12 c数据库，则数据库软件将在磁盘上生成与Oracle Database Release 11 g第 2版（11.2）兼容的数据库结构。如果您尝试使用属于Oracle数据库的更高版本的功能，并使数据库与COMPATIBLE初始化参数不COMPATIBLE ，则会发生错误。但是，启用了新功能，这些功能不会在磁盘上创建与Oracle Database Release 11 g发行版2不兼容的更改。

  • 如果对数据库进行更改以使数据库与要使用的COMPATIBLE初始化参数设置不COMPATIBLE ，则数据库不会启动，并且初始化会以错误结束。如果发生这种情况，则必须将COMPATIBLE初始化参数设置为数据库的适当值。

1.1什么是Pro * COBOL？

要访问Oracle数据库，请使用称为结构化查询语言（SQL）的高级查询语言。您经常通过交互式界面使用SQL，例如SQL * Plus。

Pro * COBOL是一种编程工具，可以将SQL语句嵌入到COBOL程序中。Pro * COBOL预编译器将COBOL程序中的SQL语句转换为标准的Oracle运行时库调用。然后可以以通常的方式编译，链接和运行生成的输出文件。

当快速开发和与其他系统兼容是您的首要任务时，请使用Pro * COBOL预编译器。

1.2 Pro * COBOL预编译器

Pro * COBOL预编译器是一种编程工具，使您可以在主机COBOL程序中嵌入SQL语句。如图1-1所示，预编译器接受主机程序作为输入，将嵌入式SQL语句转换为标准的Oracle运行时库调用，并生成可以通常方式编译，链接和执行的源程序。

1.3 Pro * COBOL预编译器的优点

Pro * COBOL预编译器使您可以将SQL的强大功能和灵活性打包到应用程序中。您可以在COBOL中嵌入SQL语句。方便易用的界面使您的应用程序可以直接访问Oracle。

与许多应用程序开发工具不同，Pro * COBOL允许您创建高度自定义的应用程序。例如，您可以创建包含最新窗口和鼠标技术的用户界面。您还可以创建在后台运行的应用程序，而无需用户交互。

此外，使用Pro * COBOL，您可以微调您的应用程序。它可以密切监视资源使用情况，SQL语句执行和各种运行时指示器。有了这些信息，您可以调整程序参数以获得最佳性能。

1.4 SQL语言

如果要访问和操作Oracle数据，则需要SQL。交互式使用SQL还是嵌入应用程序中取决于手头的工作。如果作业需要COBOL的程序处理能力，或者必须定期完成，请使用嵌入式SQL。

SQL已成为首选的数据库语言，因为它灵活，功能强大且易于学习。它是非过程性的，它允许您指定您想要的内容，而无需指定如何操作。一些类似英语的语句可以轻松地一次操作一行或多行Oracle数据。

您可以从应用程序执行任何SQL（而不是SQL * Plus）语句。例如，您可以：

  • 动态创建，更改和DROP数据库表。

  • SELECT，INSERT，UPDATE和DELETE数据行。

  • COMMIT或ROLLBACK事务。

在将SQL语句嵌入应用程序之前，可以使用SQL * Plus以交互方式测试它们。通常，从交互式SQL切换到嵌入式SQL只需要进行微小的更改。

1.5 PL / SQL语言

作为SQL的扩展，PL / SQL是一种事务处理语言，支持过程构造，变量声明和强大的错误处理。在同一个PL / SQL块中，您可以使用SQL和所有PL / SQL扩展。

嵌入式PL / SQL的主要优点是性能更好。与SQL不同，PL / SQL允许您以逻辑方式对SQL语句进行分组，并将其以块的形式发送到Oracle，而不是逐个发送到Oracle。这减少了网络流量和处理开销。

214.1 OWA_CUSTOM概述

Cookie是发送到浏览器的不透明字符串，用于维护HTTP调用之间的状态。可以在整个客户的会话期间维护状态，如果包含截止日期，则可以更长。系统日期是参考OWA_CUSTOM包中指定的信息计算的。

214.2 OWA_COOKIE类型

此数据类型包含cookie名称 - 值对。

由于HTTP标准允许cookie名称被重载（即，多个值可以与相同的cookie名称相关联），因此有一个PL / SQL RECORD保存与给定cookie名称相关联的所有值。 

TYPE vc_arr是由BINARY_INTEGER表示的VARCHAR2（4000）索引。 TYCO COOKIE是RECORD（名称VARCHAR2（4000），vals vc_arr，num_vals INTEGER）;

所有HTTP标头必须是英文和ASCII字符集。如果标头是从数据库生成的，请验证它们是否以英语创建。

继承是面向对象编程语言的一个重要特性。它使类能够包含其他类的属性。继承属性的类称为子类或子类，从中继承属性的类称为父类或超类。此功能还有助于重用已定义的代码。

1.1.1.4接口

Java仅支持单继承，也就是说，每个类只能继承一个类的字段和方法。如果需要从多个源继承属性，那么Java提供了接口的概念，这是一种多重继承的形式。接口类与类相似。但是，它们只定义方法的签名而不是它们的实现。接口中声明的方法在类中实现。当类实现多个接口时，会发生多重继承。

1.1.1.5封装

封装描述了对象隐藏其数据和方法与世界其他地方的能力，并且是面向对象编程的基本原则之一。在Java中，类封装了保存对象状态的字段，以及定义对象操作的方法。封装使您可以编写可重用的程序。它还允许您仅限制对已声明为public的对象的那些功能的访问。所有其他字段和方法都是private ，可用于内部对象处理。

13.11收集变量声明

集合变量是一个复合变量，其内部组件（称为元素）具有相同的数据类型。

集合变量的值及其元素的值可以更改。

您可以通过名称引用整个集合。您使用语法collection ( index )引用集合元素。

PL / SQL有三种集合类型：

  • 关联数组（以前称为PL / SQL表或索引表 ）

  • 可变大小的数组（varray）

  • 嵌套表

关联数组可以由字符串类型或PLS_INTEGER索引。 Varrays和嵌套表由整数索引。

您可以使用以下任一方式创建集合变量：

  • 定义集合类型，然后声明该类型的变量。

  • 使用%TYPE声明与先前声明的集合变量相同类型的集合变量。

注意：

本主题适用于您在PL / SQL块或包内定义的集合类型，这些集合类型与使用“ CREATE TYPE语句 ”创建的独立集合类型不同。

在PL / SQL块或包中，您可以定义所有三种集合类型。使用CREATE TYPE语句，您可以创建嵌套表类型和VARRAY类型，但不能创建关联数组类型。

1.1什么是数据仓库？

数据 仓库是一个数据库，旨在实现商业智能活动：它的存在是为了帮助用户理解和增强其组织的性能。它专为查询和分析而不是事务处理而设计，通常包含从事务数据派生的历史数据，但可以包含来自其他来源的数据。数据仓库将分析工作负载与事务工作负载分开，并使组织能够合并来自多个源的数据。这有助于：

  • 维护历史记录

  • 分析数据以更好地了解业务并改善业务

除了关系数据库之外，数据仓库环境还可以包括提取，传输，转换和加载（ETL）解决方案，统计分析，报告，数据挖掘功能，客户端分析工具以及管理数据收集过程的其他应用程序。 ，将其转化为有用的，可操作的信息，并将其交付给业务用户。

为了实现增强商业智能的目标，数据仓库使用从多个源收集的数据。源数据可能来自内部开发的系统，购买的应用程序，第三方数据聚合器和其他来源。它可能涉及交易，生产，营销，人力资源等。在当今的大数据世界中，数据可能是网站上数十亿次点击，或者来自内置于复杂机械中的传感器的大量数据流。

数据仓库与在线事务处理（OLTP）系统不同。使用数据仓库可以将分析工作负载与事务工作负载分开因此，数据仓库是非常面向读取的系统。与写入和更新相比，它们具有更高的数据读取量。这样可以实现更好的分析性能并避免影响您的事务系统。可以优化数据仓库系统，以整合来自多个来源的数据，以实现关键目标：它成为您组织的“单一事实来源”。拥有所有用户可以关注的一致数据源是非常有价值的;它可以防止许多争议并提高决策效率。

数据仓库通常存储数月或数年的数据以支持历史分析。数据仓库中的数据通常通过来自多个数据源的提取，转换和加载（ETL）过程加载。现代数据仓库正在向提取，加载，转换（ELT）架构发展，其中所有或大多数数据转换都在托管数据仓库的数据库上执行。值得注意的是，定义ETL过程是数据仓库设计工作的一个重要部分。同样，ETL操作的速度和可靠性是数据仓库启动和运行后的基础。

数据仓库的用户执行通常与时间相关的数据分析。例如，包括去年的销售数据，库存分析以及按产品和客户分类的利润。但是，时间关注与否，用户希望“切分”并切分他们的数据，但是他们认为合适，并且设计良好的数据仓库将足够灵活以满足这些需求。用户有时需要高度聚合的数据，有时他们需要深入了解详细信息。更复杂的分析包括趋势分析和数据挖掘，它们使用现有数据来预测趋势或预测未来。数据仓库充当中间件商业智能环境使用的底层引擎，为最终用户提供报告，仪表板和其他界面。

虽然上面的讨论集中在术语“数据仓库”上，但还有两个重要的术语需要提及。这些是数据集市和操作数据存储（ODS）。

数据集市与数据仓库具有相同的作用，但其范围有限。它可以服务于某个特定部门或业务线。数据集市与数据仓库的优势在于，由于覆盖范围有限，因此可以更快地创建数据集市。但是，数据集市也会产生不一致的问题。在数据集市中保持数据和计算定义一致需要严格的纪律。这个问题已被广泛认可，因此数据集市有两种风格。独立数据集市是直接从源数据馈送的数据集市。他们可以变成信息不一致的岛屿。从属数据集市从现有数据仓库提供。依赖数据集市可以避免不一致的问题，但它们要求已经存在企业级数据仓库。

存在操作数据存储以支持日常操作。ODS数据经过清理和验证，但历史上并不深入：它可能只是当天的数据。ODS不是支持数据仓库可以处理的历史丰富的查询，而是为数据仓库提供了访问最新数据的位置，这些数据尚未加载到数据仓库中。ODS还可以用作加载数据仓库的源。随着数据仓库加载技术变得更加先进，数据仓库可能不再需要ODS作为加载数据的源。相反，恒定的涓流进给系统可以近乎实时地加载数据仓库。

引入数据仓库的常用方法是参考William Inmon提出的数据仓库的特征：

  • 主题导向

  • 集成

  • 非易失

  • 时间变化

主题导向

数据仓库旨在帮助您分析数据。例如，要了解有关贵公司销售数据的更多信息，您可以构建专注于销售的数据仓库。使用此数据仓库，您可以回答诸如“去年谁是我们这个项目的最佳客户？”之类的问题？“或”明年谁可能成为我们最好的客户？“这种通过主题定义数据仓库的能力，在这种情况下是销售，使数据仓库面向主题。

集成

整合与学科定位密切相关。数据仓库必须将来自不同来源的数据放入一致的格式。他们必须解决诸如命名冲突和计量单位之间的不一致之类的问题。当他们实现这一目标时，他们被认为是整合的。

非易失

非易失性意味着， 一旦进入数据仓库，数据就不应该改变。这是合乎逻辑的，因为数据仓库的目的是使您能够分析发生的情况。

时间变化

数据仓库专注于随时间的变化是术语时变的意思。为了发现趋势并识别业务中隐藏的模式和关系，分析师需要大量数据。这与在线事务处理（OLTP）系统形成鲜明对比，后者的性能要求要求将历史数据移动到存档。

1.1.1数据仓库的关键特征

数据仓库的特性如下：

  • 数据的结构简化了访问和高速查询性能。

  • 最终用户是时间敏感的，并且渴望思考速度的响应时间。

  • 使用大量历史数据。

  • 查询通常会检索大量数据，可能是数千行。

  • 预定义和即席查询都很常见。

  • 数据加载涉及多个源和转换。

通常，具有高数据吞吐量的快速查询性能是成功数据仓库的关键。

1.2对比OLTP和数据仓库环境

OLTP系统和数据仓库之间存在重要差异。系统类型之间的一个主要区别是数据仓库不仅仅是第三范式（3NF） ，这是OLTP环境中常见的一种数据规范化。

数据仓库和OLTP系统有着非常不同的要求。以下是典型数据仓库和OLTP系统之间差异的一些示例：

  • 工作量

    数据仓库旨在适应即席查询和数据分析。您可能事先不知道数据仓库的工作负载，因此应优化数据仓库以便在各种可能的查询和分析操作中正常运行。

    OLTP系统仅支持预定义的操作。您的应用程序可能经过专门调整或设计为仅支持这些操作。

  • 数据修改

    ETL过程（每晚或每周运行）使用批量数据修改技术定期更新数据仓库。数据仓库的最终用户不会直接更新数据仓库，除非使用分析工具（如数据挖掘）来预测相关概率，将客户分配到细分市场并开发客户档案。

    在OLTP系统中，最终用户会定期向数据库发出单独的数据修改语句。OLTP数据库始终是最新的，并反映每个业务事务的当前状态。

  • 架构设计

    数据仓库通常使用部分非规范化模式来优化查询和分析性能。

    OLTP系统通常使用完全规范化的模式来优化更新/插入/删除性能，并保证数据的一致性。

  • 典型的操作

    典型的数据仓库查询扫描数千或数百万行。例如，“查找上个月所有客户的总销售额”。

    典型的OLTP操作只访问少量记录。例如，“检索此客户的当前订单”。

  • 历史数据

    数据仓库通常存储数月或数年的数据。这是为了支持历史分析和报告。

    OLTP系统通常只存储几周或几个月的数据。OLTP系统仅根据需要存储历史数据，以成功满足当前事务的要求。

269.1 UTL_URL概述

统一资源定位符（URL）是标识Web资源（例如页面或图片）的字符串。使用URL通过超文本传输协议（HTTP）访问此类资源。

例如，Oracle网站的URL是：

http://www.oracle.com

通常，URL包含英文字母字符，数字和标点符号。这些字符称为无保留字符 。URL中的任何其他字符（包括多字节字符或二进制八位字节代码）都必须进行转义，以便Web浏览器或Web服务器进行准确处理。一些标点符号（如美元符号（ $) ，问号（ ?) ，冒号（ :)和等号（ =)将保留为URL中的分隔符。它们被称为保留字符 。要从字面上处理这些字符，而不是将它们视为分隔符，它们必须被转义。

未保留的字符是：

  • A到Z, a到z,和0到9

  • 连字符（ - ），下划线（ _ ），句点（ . ），感叹号（ ! ），波浪号（ ~ ），星号（ * ），重音符号（ ' ），左括号（ ( ），右括号（ ) ）

保留的字符是：

  • 分号（ ; ）斜杠（ / ），问号（ ? ，结肠）（ ： :符号（ @ ），符号（ & ），等号（ = ），加号（ + ），美元符号（ $ ） ，百分号（ % ）和逗号（ , ）

UTL_URL包有两个函数，为URL字符提供转义和unescape机制。在使用URL之前使用escape函数转义URL以通过UTL_HTTP包获取Web页面。在从URL中提取信息之前，使用unescape函数来转义转义的URL。

有关更多信息，请参阅Request For Comments（RFC）文档RFC2396。请注意，此URL转义和unescape机制与HTML规范中描述的x-www-form-urlencoded编码机制不同：

http://www.w3.org/TR/html

安全要求概述

之前实施您的安全方案，您必须了解您环境中的现有安全要求和期望。由于您要启用对不同系统上不同数据库的应用程序访问，因此您可能需要合并多个安全文化。当您将多个不同的系统连接到一个操作整体时，具有最严格安全要求的系统通常会指示其他系统能够做什么和不能做什么。

网关安全包括两个主要问题：

  • 允许访问特定网关实例和OLTP的用户和应用程序

  • 用户和应用程序能够执行的OLTP事务

您可以在网关体系结构中的多个点控制访问。主要选项将在以下部分中讨论。每个OLTP都提供对远程事务程序访问的控制，具有基于用户ID的本机授权机制。 OLTP的产品文档中描述了这些功能。安全要求中的信息包括网关设施如何确定对特定OLTP连接有效的用户ID。

当网关涉及RPC请求时，安全机制对网关遇到的每个系统组件都有效。遇到的第一个系统组件是应用程序工具或第三代语言（3GL）程序。遇到的最后一个系统组件是OLTP。

以下每个部分都标识了该组件中可用的组件和安全性处理类型。每个部分都提供了主要功能和参数的摘要。有关Oracle和非Oracle产品的非网关组件的详细信息，请参阅产品特定的文档。

验证应用程序登录

应用程序必须连接到使用网关之前的Oracle数据库。您使用的登录身份验证类型决定了生成的Oracle用户ID，并且可能会影响网关操作。

可以使用以下类型的身份验证：

  • Oracle身份验证

    使用Oracle身份验证，每个Oracle用户标识都有一个Oracle已知的关联密码。当应用程序连接到服务器时，它会提供用户标识和密码。Oracle确认存在用户标识，并且密码与存储在数据库中的密码相匹配。

  • 操作系统认证

    通过操作系统身份验证，底层服务器操作系统负责身份验证。使用操作系统身份验证访问使用IDENTIFIED EXTERNALLY属性（而不是密码）创建的Oracle用户标识。要使用此类用户标识登录，应用程序会为用户标识提供正斜杠（ / ），并且不提供密码。

    要执行操作系统身份验证，服务器会确定请求者操作系统用户ID，可选地为其添加固定前缀，并将结果用作Oracle用户ID。服务器确认用户ID存在且IDENTIFIED EXTERNALLY已IDENTIFIED EXTERNALLY ，但未进行密码检查。基本假设是用户在登录操作系统时进行身份验证。

    操作系统身份验证并非在所有平台上都可用，并且在某些平台上不可用 Oracle Net（客户端 - 服务器）和多线程服务器配置。请参阅特定于平台的Oracle数据库文档和“ Oracle数据库网络服务管理员指南”，以确定配置中此功能的可用性。

有关验证应用程序登录的详细信息，请参阅“ Oracle数据库管理员指南” 。

定义和控制数据库链接

以下部分讨论了使用TCP / IP或SNA通信协议的网关用户的数据库链接。

链接可访问性

数据库链接的第一个控制点是，如果给定用户可以访问它。一个公共数据库链接可以由任何用户ID使用。私有数据库链接只能由创建它的用户使用。数据库链接可用性取决于其打开网关会话的能力。 Oracle数据库不区分使用类型（例如只读与更新或写入）或可以访问哪些远程对象。这些区别是访问的OLTP的责任。

链接和CONNECT子句

该 CONNECT子句是另一个与安全相关的属性数据库链接。您可以使用CONNECT子句指定显式用户标识和密码，该标识可能与Oracle用户标识和密码不同。首次打开数据库链接连接时，此CONNECT用户标识和密码组合将发送到网关。根据特定于网关的选项，网关可能会将该用户标识和密码发送到要验证的OLTP。

如果使用Oracle身份验证创建了没有CONNECT子句的数据库链接，则在打开连接时会将用户的Oracle用户ID和密码发送到网关。如果用户登录到 Oracle数据库具有操作系统身份验证，然后网关不会从Oracle数据库接收用户ID或密码。经过系统身份验证的Oracle用户不可能使用没有CONNECT子句定义的网关数据库链接。但是，如果您的OLTP提供基于网关的u ser ID映射工具用户连接的LU名称，如果同一网关实例上的所有用户都可以使用相同的OLTP用户ID，则可以进行此类连接。

有关数据库链接的更多信息，请参阅“ Oracle数据库管理员指南” 。

使用SNA安全验证

2.3 JSON与XML相比

JSON和XML（可扩展标记语言）通常用作数据交换语言。他们的主要区别列在这里。

JSON对于简单的结构化数据最有用。XML对结构化和半结构化数据都很有用。JSON通常以数据为中心，而不是以文档为中心; XML也可以。JSON不是标记语言;它仅用于数据表示。XML既是文档标记语言又是数据表示语言。

  • JSON数据类型很少并且是预定义的。XML数据可以是无类型的，也可以基于XML模式或文档类型定义（DTD）。

  • JSON具有简单的结构定义和文档组合结构：它缺少属性，命名空间，继承和替换。

  • JavaScript对象文字成员的顺序无关紧要。通常，订单在XML文档中很重要。

  • JSON缺少等效的XML文本节点（XPath节点测试text() ）。特别是，这意味着没有混合内容（这是说JSON不是标记语言的另一种方式）。

  • JSON没有日期数据类型（与XML和JavaScript不同）。使用可用的数据类型（如字符串）以JSON表示日期。在日期和JSON字符串之间进行转换有一些事实上的标准。但是，使用JSON的程序必须以某种方式处理日期表示转换。

由于其简单的定义和功能，JSON数据通常比XML数据更容易生成，解析和处理。涉及组合不同数据源的用例通常很适合使用XML，因为它提供了名称空间和其他构造，有助于模块化和继承。

模式匹配条件比较字符数据。

LIKE条件指定涉及模式匹配的测试。等式运算符（=）将一个字符值与另一个字符值完全匹配，而LIKE条件通过搜索第二个值指定的模式的第一个值来匹配一个字符值的一部分。LIKE使用输入字符集定义的字符计算字符串。LIKEC使用Unicode完整字符。LIKE2使用UCS2代码点。LIKE4使用UCS4代码点。

在这个语法中：

  • char1是一个字符表达式，例如字符列，称为搜索值 。

  • char2是一个字符表达式，通常是文字，称为模式 。

  • esc_char是一个字符表达式，通常是文字，称为转义字符 。

LIKE条件是几乎所有情况下的最佳选择。使用以下指南确定是否有任何变体对您的环境有所帮助：

  • 使用LIKE2使用UCS-2语义处理字符串。LIKE2将Unicode补充字符视为两个字符。

  • 使用LIKE4使用UCS-4语义处理字符串。LIKE4将Unicode补充字符视为一个字符。

  • 使用LIKEC使用Unicode完整字符语义处理字符串。LIKEC将复合字符视为一个字符。

如果未指定esc_char ，则没有默认转义字符。如果char1 ， char2或esc_char任何一个为null，则结果未知。否则，转义字符（如果指定）必须是长度为1的字符串。

所有字符表达式（ char1 ， char2和esc_char ）可以是任何数据类型CHAR ， VARCHAR2 ， NCHAR或NVARCHAR2 。如果它们不同，则Oracle将所有这些转换为char1的数据类型。

该模式可以包含特殊的模式匹配字符：

  • 模式中的下划线（_）恰好匹配值中的一个字符（而不是多字节字符集中的一个字节）。

  • 模式中的百分号（％）可以匹配值中的零个或多个字符（而不是多字节字符集中的字节）。模式'％'不能与null匹配。

您可以使用ESCAPE子句在模式中包含实际字符%或_ ，该子句标识转义字符。如果转义字符位于模式中的字符%或_之前，则Oracle会在模式中按字面解释此字符，而不是作为特殊的模式匹配字符。您还可以通过重复搜索转义字符本身。例如，如果@是转义字符，那么您可以使用@@来搜索@。

注意：

只有ASCII等效的下划线（_）和百分号（％）字符被识别为模式匹配字符。它们的全宽变体以东亚字符集和Unicode表示，被视为普通字符。

为了处理LIKE条件，Oracle将模式划分为子模式，每个子模式由一个或两个字符组成。双字符子模式以转义字符开头，另一个字符是％，或_或转义字符。

设P 1 ，P 2 ，...，P n为这些子图案。如果有一种方法可以将搜索值划分为子串S 1 ，S 2 ，...，S n，那么对于1和n之间的所有i ，类似的条件都是正确的：

  • 如果P i是_，则S i是单个字符。

  • 如果P i是％，则S i是任何字符串。

  • 如果P i是以转义字符开头的两个字符，则S i是P i的第二个字符。

  • 否则，P i = S i 。

使用LIKE条件，您可以将值与模式进行比较，而不是与常量进行比较。该模式必须出现在LIKE关键字之后。例如，您可以发出以下查询以查找名称以R开头的所有员工的工资：

SELECT salary FROM employees WHERE last_name LIKE'R％'ORDER BY salary;

以下查询使用=运算符而不是LIKE条件来查找名为“R％”的所有员工的工资：

SELECT salary FROM employees WHERE last_name ='R％'ORDER BY salary;

以下查询查找名为“SM％”的所有员工的工资。Oracle将“SM％”解释为文本文字而不是模式，因为它在LIKE关键字之前：

SELECT salary FROM employees WHERE'SM％'LIKE last_name ORDER BY salary;

ESCAPE子句将反斜杠（\）标识为转义字符。在模式中，转义字符位于下划线（_）之前。这导致Oracle按字面解释下划线，而不是作为特殊模式匹配字符。

REGEXP_LIKE条件

REGEXP_LIKE类似于LIKE条件，除了REGEXP_LIKE执行正则表达式匹配而不是LIKE执行的简单模式匹配。此条件使用输入字符集定义的字符计算字符串。

此条件符合POSIX正则表达式标准和Unicode正则表达式指南。有关更多信息，请参阅Oracle正则表达式支持 。
