设置输出中列之间的空格数。n的最大值为10。

C.13 SET TRU [NCATE] {ON | OFF}（已废弃）

控制SQL * Plus是否截断或包装对于当前行宽而言太长的数据项。

ON的功能与SET WRAP OFF相同，反之亦然。您可能更喜欢使用WRAP，因为SHOW命令识别WRAP并且无法识别TRUNCATE。

C.14 TTI [TLE] 文本 （过时的旧表格）

在每个报告页面的顶部显示标题。

旧形式的TTITLE提供的格式化功能比新表单的格式更有限，但提供了与UFI（SQL * Plus的前身）的兼容性。旧表单将顶部标题定义为日期左对齐且页码右对齐的行，后跟带有居中文本的行，然后是空行。

您输入的文本定义了TTITLE显示的标题。

SQL * Plus基于SET LINESIZE确定的行大小来居中文本。分隔符（|）开始一个新行;一行中的两个行分隔符（||）插入一个空行。您可以使用SET HEADSEP更改行分隔符。

您可以通过定义名为“_page”的变量来控制旧形式的TTITLE和BTITLE中页码的格式。_page的默认值是格式化字符串“page＆P4”。要更改格式，可以使用新的格式字符串DEFINE _page，如下所示：

SET ESCAPE / SQL> DEFINE _page ='Page /＆P2'

此格式化字符串将使用首字母大写字母打印单词“page”，并将页码格式设置为宽度为2。您可以将任何文本替换为“页面”，将任何数字替换为宽度。必须设置escape，以便SQL * Plus不会将＆符号（＆）解释为替换变量。请参见SET ERRORL [OGGING] {ON | OFF} [TABLE [schema。] tablename] [TRUNCATE] [IDENTIFIER identifier] ，了解有关设置转义字符的更多信息。

如果有效的new-form子句不紧跟命令名，SQL * Plus会以旧格式解释TTITLE。

如果你想使用带有TTITLE的CENTER并在一行上放置多个单词，你应该使用新形式的TTITLE。有关更多信息，请参阅TTITLE命令。

9.2关于创建和控制角色

您可以使用SQL命令创建和控制对角色的访问，以便为数据库表提供安全性。通过创建角色然后控制谁有权访问它，您可以确保只有特定用户才能访问特定的数据库权限。

创建角色并与SQL CREATE，GRANT和SET命令一起使用：

  • 要创建角色，请使用CREATE命令。您可以创建带或不带密码的角色。

  • 要授予对角色的访问权限，请使用GRANT命令。通过这种方式，您可以控制谁有权访问与该角色关联的权限。

  • 要访问角色，请使用SET ROLE命令。如果您使用密码创建了角色，则用户必须知道密码才能访问该角色。

9.2.1关于禁用SET ROLE

从SQL * Plus，用户可以提交任何SQL命令。在某些情况下，这可能会导致安全问题。除非采取适当的预防措施，否则用户可以使用SET ROLE访问通过应用程序角色获得的权限。使用这些权限，他们可能会从SQL * Plus发出可能错误地更改数据库表的SQL语句。

要防止应用程序用户访问SQL * Plus中的应用程序角色，可以使用PUP表禁用SET ROLE命令。您还需要禁用BEGIN和SQL * Plus EXECUTE命令，以防止应用程序用户通过PL / SQL块设置应用程序角色。这为SQL * Plus用户提供了与启动SQL * Plus时启用的角色关联的权限。有关用户角色的创建和使用的更多信息，请参见“ Oracle数据库SQL语言参考”和“ Oracle数据库管理员指南” 。

9.2.2关于禁用用户角色

至禁用给定用户的角色，在PUP表中插入一行，其中包含Userid列中的用户用户名，Attribute列中的“ROLES”和Char_Value列中的角色名称。

注意：

为Userid列输入“PUBLIC”或“％”时，将禁用所有用户的角色。您只应将“％”或“PUBLIC”用于授予“PUBLIC”的角色。如果尝试禁用尚未授予用户的角色，则不会禁用该用户的任何角色。

Scope，Numeric_Value和Date_Value列应包含NULL。例如：

PRODUCT USERID属性范围数字日期长期价值价值------- ------ --------- ----- -------- --- --- ----- ----- SQL * Plus HR ROLES ROLE1 SQL * Plus PUBLIC ROLES ROLE2

在登录期间，这些表行将转换为命令

SET ROLE ALL除了ROLE1，ROLE2

要确保用户在登录后不使用SET ROLE命令更改其角色，可以禁用SET ROLE命令。

至重新启用角色，删除包含限制的行。

有关更多信息，请参见关于禁用SET ROLE 。

9.4关于程序参数安全性

某些操作系统允许任何用户查看正在运行的程序。如果显示屏还显示命令行参数，则可以查看其他SQL * Plus用户的用户名和密码。

例如，在许多UNIX或Linux系统上，ps命令显示程序参数。停止显示密码取决于您使用SQL * Plus的方式。

  • 要以交互方式运行SQL * Plus，请始终等待SQL * Plus提示输入连接信息，尤其是密码。

  • 要从UNIX shell脚本运行批处理SQL脚本，请将环境变量MYUSERNAME和MYPASSWORD设置为适当的值。运行shell脚本，其中包含：

    sqlplus / nolog << EOF connect $ MYUSERNAME / $ MYPASSWORD选择...EOF

  • 要运行批处理SQL脚本，请将用户名和密码硬编码为SQL脚本的第一行。然后调用脚本：

    sqlplus @ myscript.sql

当像这样启动SQL * Plus时，它使用脚本的第一行作为用户名/密码@ connection_identifier字符串。

避免将用户名和密码存储在文件或脚本中。如果您确实将用户名和密码存储在文件或脚本中，请确保文件或脚本受到非授权访问的保护。

创建序列

目的

使用CREATE SEQUENCE语句创建一个序列 ，该序列是一个数据库对象，多个用户可以从中生成唯一的整数。您可以使用序列自动生成主键值。

生成序列号时，序列会递增，与事务提交或回滚无关。如果两个用户同时递增相同的序列，则每个用户获取的序列号可能有间隙，因为序列号是由另一个用户生成的。一个用户永远无法获取另一个用户生成的序列号。在由一个用户生成序列值之后，该用户可以继续访问该值，而不管该序列是否由另一个用户递增。

序列号是独立于表生成的，因此相同的序列可用于一个或多个表。单个序列号可能会被跳过，因为它们是在最终回滚的事务中生成和使用的。另外，单个用户可能没有意识到其他用户正在从相同的序列中绘制。

创建序列后，可以使用CURRVAL伪列访问SQL语句中的值，该伪列返回序列的当前值，或NEXTVAL伪列，它会递增序列并返回新值。

也可以看看：

  • 有关使用CURRVAL和NEXTVAL更多信息的Pseudocolumns

  • “ 如何使用序列值 ”获取有关使用序列的信息

  • 有关修改或删除序列的信息，请参见 ALTER SEQUENCE或DROP SEQUENCE

先决条件

要在自己的模式中创建序列，您必须具有CREATE SEQUENCE系统特权。

要在另一个用户的模式中创建序列，您必须具有CREATE ANY SEQUENCE系统特权。

句法

语义

模式

指定要包含序列的模式。如果省略schema ，则Oracle数据库会在您自己的模式中创建序列。

序列

指定要创建的序列的名称。该名称必须满足“ 数据库对象命名规则 ”中列出的要求。

如果没有通过GLOBAL指定INCREMENT BY子句，则创建一个以1开头的递增序列，并且在没有上限的情况下增加1。仅指定INCREMENT BY -1会创建一个以-1开头的降序序列，并且在没有下限的情况下减少。

  • 创建一个序列 无限制的增量，对于升序序列，省略MAXVALUE参数或指定NOMAXVALUE 。对于降序序列，请省略MINVALUE参数或指定NOMINVALUE 。

  • 创建一个序列 停止在预定义的限制，对于升序序列，指定MAXVALUE参数的值。对于降序，请为MINVALUE参数指定值。同时指定NOCYCLE 。一旦序列达到其极限，任何尝试生成序列号都会导致错误。

  • 创建一个序列 达到预定义限制后重新启动，指定MAXVALUE和MINVALUE参数的值。同时指定CYCLE 。

共享

仅当在应用程序根目录中创建序列时，此子句才适用。这种类型的序列称为应用程序公共对象，它可以与属于应用程序根目录的应用程序PDB共享。要确定序列的共享方式，请指定以下共享属性之一：

  • METADATA - 元数据链接共享序列的元数据，但其数据对每个容器都是唯一的。这种类型的序列称为元数据链接的应用程序公共对象 。

  • DATA - 数据链接共享序列，其数据对于应用程序容器中的所有容器都是相同的。其数据仅存储在应用程序根目录中。这种类型的序列称为数据链接的应用程序公共对象 。

  • NONE - 不共享序列。

如果省略此子句，则数据库使用DEFAULT_SHARING初始化参数的值来确定序列的共享属性。如果DEFAULT_SHARING初始化参数没有值，则默认值为METADATA 。

创建序列后，无法更改序列的共享属性。

也可以看看：

  • 有关DEFAULT_SHARING初始化参数的更多信息，请参见Oracle数据库参考

  • “Oracle数据库管理员指南” ，提供有关创建应用程序公共对象的完整信

增量

指定序列号之间的间隔。此整数值可以是任何正整数或负整数，但不能为0。对于升序序列，此值可以包含28个或更少的数字，对于降序序列，该值可以包含27个或更少的数字。该值的绝对值必须小于MAXVALUE和MINVALUE的差值。如果此值为负，则序列下降。如果值为正，则序列上升。如果省略此子句，则间隔默认为1。

从...开始

指定要生成的第一个序列号。使用此子句以大于其最小值的值开始升序，或者以小于其最大值的值开始降序。对于升序序列，默认值是序列的最小值。对于降序序列，默认值是序列的最大值。对于正值，此整数值可以包含28个或更少的数字，对于负值，该数字可以包含27个或更少的数字

注意：

该值不一定是上升或下降循环序列分别在达到其最大值或最小值之后循环的值。

MAXVALUE

指定序列可以生成的最大值。对于正值，此整数值可以包含28个或更少的数字，对于负值，该数字可以包含27个或更少的数字MAXVALUE必须等于或大于START WITH且必须大于MINVALUE 。

NOMAXVALUE

指定NOMAXVALUE以指示升序序列的最大值10 28 -1或降序序列的最大值-1。这是默认值。

MINVALUE

5.5.1关于共享服务器进程

共享服务器进程在共享服务器体系结构中使用， 如图5-4所示 。使用共享服务器体系结构，客户端进程最终连接到调度程序。LREG进程使用侦听器注册调度程序的位置和负载，使侦听器能够将请求转发给负载最少的调度程序。该注册过程未在图中示出。

一个调度程序可以同时支持多个客户端连接。每个客户端连接都绑定到一个 虚电路 。虚拟电路是调度程序用于客户端数据库连接请求和回复的共享内存。当请求到达时，调度程序将虚拟电路放置在公共请求队列上。空闲的共享服务器从中获取虚拟电路请求队列，为请求提供服务，并在尝试从请求队列中检索另一个虚拟电路之前放弃虚拟电路。共享服务器将所有已完成的请求放入调度程序的响应队列中。每个调度员都有自己的调度员 SGA（系统全局区域）中的响应队列。这种方法使一小部分服务器进程能够为大量客户端提供服务。

5.6了解Oracle Connection Manager体系结构

Oracle Connection Manager是一个网关，客户端连接请求通过该网关发送到下一个跃点或直接发送到数据库服务器。通过Oracle Connection Manager中继连接请求的客户端可以利用Oracle Connection Manager上配置的会话多路复用和访问控制功能。在LREG进程注册其服务之前，它不携带任何服务信息。

Oracle Connection Manager由三个组件组成：

  • 倾听者

  • CMGW（Oracle连接管理器网关）

  • CMADMIN（Oracle连接管理器管理）

侦听器接收客户端连接并根据一组规则评估是否拒绝或允许访问。如果它允许访问，则侦听器将请求转发到网关进程，选择具有最少连接的进程。反过来，CMGW进程将请求转发到另一个Oracle Connection Manager或直接转发到数据库服务器，中继数据直到连接终止。如果已存在与服务器的连接，则网关通过现有连接多路复用或汇集其连接。CMADMIN监视网关进程和侦听器的状态，根据需要关闭或启动进程。此外，它还会向侦听器注册网关进程的位置和负载，并响应来自Oracle Connection Manager Control实用程序的请求。

支持单客户端访问名称（SCAN）

Oracle Database 11g第2版及更高版本的客户端使用支持单客户端访问名称（SCAN）连接到数据库。它为连接到Oracle RAC的客户端提供单一名称，即使您在群集中添加或删除节点，该名称也不会在群集的整个生命周期中发生更改。与SCAN连接的客户端可以使用简单的连接字符串，例如瘦JDBC URL或EZConnect，并实现负载平衡和客户端连接故障转移。

除了三个SCAN侦听器（每个虚拟IP地址一个）之外，在托管数据库实例的每个节点上都有一个节点侦听器。使用两层侦听器（SCAN侦听器和节点侦听器）的目的是分离Oracle RAC中侦听器的两个功能，首先是负载平衡连接，其次是产生和遗留会话。SCAN侦听器将接收来自客户端的连接请求，由GNS（网格命名服务）随机分发。然后，SCAN侦听器将使用负载平衡度量标准将请求智能地重定向到最能够提供所请求服务的节点上的节点侦听器。数据库实例向SCAN侦听器注册为远程侦听器，并将节点侦听器注册为本地侦听器。

无法使用SCAN配置Oracle数据库网关，SCAN是数据库连接到网关的单个名称。有两个原因。网关不适用于远程侦听器。与您可以指定REMOTE_LISTENER以设置为SCAN侦听器的数据库不同，Oracle Database Gateway中不支持它。这对于SCAN侦听器将连接路由到节点侦听器至关重要。其次，网关不向群集注册，以便将其作为群集资源进行管理。

对于网关，当Oracle数据库和Oracle数据库网关在同一群集上运行时，SCAN不是很有用。可以在安装数据库的每个节点上安装和配置Oracle数据库网关，并且可以配置数据库，使每个实例连接到在同一节点上运行的网关。

本地监听器

应使用本地侦听器配置Oracle Database Gateway服务。它是本地监听器产生网关进程。这意味着监听器应该知道要生成哪个网关进程。使用Grid infrastructure home中的listener.ora添加Gateway SID。如果数据库的SCAN侦听器已在该节点上运行，则可以使用相同的listener.ora文件来配置本地侦听器。

负载平衡和透明应用程序故障转移（TAF）

Oracle Database Gateway本身不支持客户端（使用数据库主目录中的tnsnames）或服务器端（使用SCAN）负载平衡。网关级别的负载平衡不适用，因为网关进程当前专用于单个会话。但是，通过为每个数据库实例关联不同的Oracle Database Gateway实例，可以实现节点级负载平衡，即通过数据库上的负载平衡选择负载最小的节点。

Oracle Database Gateway支持连接故障转移功能。如果为Gateway配置客户端连接负载平衡，则其工作方式与故障转移类似。

每当发生数据库故障转移时，即会话从一个数据库实例迁移到另一个数据库实例（在新节点上），迁移的会话将使用该新节点中的网关实例。

默认情况下，Oracle Call Interface（OCI）应用程序可以使用三种类型的Oracle Net故障转移功能：

  • session ：设置为故障转移会话。如果用户连接丢失，则会自动为备份上的用户创建新会话。此类故障转移不会尝试恢复选择操作。

  • select ：设置为允许具有打开游标的用户在失败后继续获取它们。但是，此模式涉及正常选择操作中客户端的开销。

  • none ：这是默认值。没有使用故障转移功能。这也可以明确指定，以防止发生故障转移。

要使故障转移起作用，需要使用多个侦听器地址配置Database home中的tnsnames.ora 。

如果实例在连接后失败，则TAF应用程序将故障转移到另一个节点的侦听器，并保留正在进行的任何SELECT语句。

在以下用于负载平衡的tnsnames.ora示例中，仅用作故障转移，仅当gateway1-server上的网关不可用时，数据库才连接到主机gateway2-server的网关：

dg4sybs.us.example.com =（DESCRIPTION =（LOAD_BALANCE = on）（FAILOVER = on）（ADDRESS =（PROTOCOL = tcp）（HOST = gateway1-server）（PORT = 1521））（ADDRESS =（PROTOCOL = tcp） （HOST = gateway2-server）（PORT = 1521））（CONNECT_DATA =（SERVICE_NAME = dg4sybs.us.example.com）（HS = OK）（FAILOVER_MODE =（TYPE = select）（METHOD = basic））））

126.1 DBMS_PROCESS概述

默认情况下，Oracle数据库可以预先设置前台进程以提高客户端连接的性能。

预先确定的流程是指已生成但尚未召开会话的流程。当用户连接到数据库或需要服务进程时，该进程根据需要执行进一步的初始化。

要管理前台进程，请使用DBMS_PROCESS包。此程序包中的过程配置连接池的前台进程数，启动连接池和停止连接池。

也可以看看：

数据字典SYS.LINK$和SYS.SCHEDULER$_CREDENTIAL系统表存储敏感凭证数据，例如用户密码。

SYS.LINK$表存储有关数据库链接的信息。SYS.SCHEDULER$_CREDENTIAL存储有关Oracle Scheduler事件的信息。默认情况下，存储在这些表中的敏感凭据数据会被混淆。

您可以使用ALTER DATABASE DICTIONARY语句手动加密存储在SYS.LINK$和SYS.SCHEDULER$_CREDENTIAL表中的数据。虽然此功能使用透明数据加密（TDE），但您无需具有高级安全选项许可证即可执行加密，但您必须具有SYSKM管理权限。TDE使用AES256（高级加密标准）算法执行加密。加密遵循与使用TDE加密的其他数据相同的行为。

作为最佳安全实践，Oracle建议您加密此敏感凭据数据。要检查数据字典凭据的状态，可以查询DICTIONARY_CREDENTIALS_ENCRYPT数据字典视图。

D.2.2数据保护模式

                  

                     
如果Oracle RAC主数据库的任何实例失去与备用数据库的连接，则所有其他主数据库实例将停止向备用数据库发送重做达LOG_ARCHIVE_DEST_ n REOPEN属性上指定的秒数，之后所有主数据库实例都尝试重新连接到备用数据库。
                     

                     
以下列表描述了Oracle RAC环境中保护模式的行为：

                     
    
                        
  • 
                           
    最大保护配置
    
                           
    如果丢失的目标是最后一个参与的SYNC目标，则该实例将失去连接并被关闭。Oracle RAC配置中仍与备用目标连接的其他实例会恢复丢失的实例并继续将重做发送到备用目标。仅当Oracle RAC配置中的每个实例都失去与最后一个备用目标的连接时，主数据库才会关闭。
                           
    
                        
    • 
                        
  • 
                           
    最高可用性和最高性能配置
    
                           
    Oracle RAC配置中仍与备用目标连接的其他实例将恢复丢失的实例并继续将重做发送到其备用目标。当Oracle RAC配置中的每个实例都失去与备用目标的连接时，主数据库将继续以最高性能模式运行。最大性能模式确保非常小的数据丢失，除非整个备用数据库失败。
    
                           
    最大可用性保护模式可确保零数据丢失，但某些双重故障除外，例如所有备用数据库发生故障后主数据库出现故障。

217.1 OWA_OPT_LOCK概述

               

                  
OWA_OPT_LOCK包中包含强制执行锁定策略的子程序，以防止丢失更新。

                  

                     
它检查用户是否有兴趣更新的行是否已被其他人同时更改。  
                     

                     
PL / SQL网关不能使用传统的数据库锁定方案，因为HTTP是无状态协议。OWA_OPT_LOCK包为您提供了两种处理丢失更新问题的方法：

                     
    
                        
  • 
                           
    隐藏字段方法将以前的值存储在HTML页面的隐藏字段中。当用户请求更新时，PL / SQL Gateway会根据数据库的当前状态检查这些值。仅在值匹配时才执行更新操作。要使用此方法，请调用owa_opt_lock.store_values过程。
    
                        
    • 
                        
  • 
                           
    校验和方法存储校验和而不是值本身。要使用此方法，请调用owa_opt_lock.checksum函数。
    
                        
    • 
                     

                     
这些方法很乐观。它们不会阻止其他用户执行更新，但如果发生了干预更新，它们会拒绝当前更新。

217.3.1 CHECKSUM函数

                  

                     
此函数返回指定字符串或表中行的checksum值。对于表中的行，该函数根据行中列的值计算checksum值。此功能有两个版本。

    3. 
                     
  3. 单击“ 添加策略”按钮以创建性能策略。将出现“添加策略”页面。
    
                           

    说明cc12102_add_perf_policy.png的说明
    
                           
                           
    对于每个性能策略，您必须指定唯一的名称。您还可以提供策略及其意图的说明。然后，您必须为策略配置性能类。

      • 
                              
    • 
                                 
      指定性能目标的值。

34.1 SDO_TUNE.AVERAGE_MBR

               

                  

                     
格式
SDO_TUNE.AVERAGE_MBR（table_name IN VARCHAR2，column_name IN VARCHAR2，width OUT NUMBER，height OUT NUMBER）;

                  
                  

                     
描述

                     
计算图层中几何图形的平均最小边界矩形（MBR）。

                  

                  
                  

                     
参数

                     

                        
 table_name

                        

                           
空间几何表。

                        

                        
 column_name

                        

                           
要计算平均最小边界矩形的几何列。

                        

                        
宽度

                        

                           
平均最小边界矩形的宽度。

                        

                        
身高

                        

                           
平均最小边界矩形的高度。

                        

                     

                  

                  
                  

                     
使用说明

                     
此过程计算并存储空间几何表中所有几何的平均最小边界矩形的宽度和高度。它通过跟踪空间几何表中所有几何的最大和最小X和Y值来计算平均MBR。

                  

                  
                  

                     
例子

                     
以下示例计算COLA_MARKETS表的SHAPE列的最小边界矩形。
DECLARE table_name VARCHAR2（32）：='COLA_MARKETS'; column_name VARCHAR2（32）：='SHAPE';宽度NUMBER;身高NUMBER; BEGIN SDO_TUNE.AVERAGE_MBR（table_name，column_name，width，height）; DBMS_OUTPUT.PUT_LINE（'Width ='|| width）; DBMS_OUTPUT.PUT_LINE（'Height ='|| height）;结束; /宽度= 3.5高度= 4.5

34.2 SDO_TUNE.ESTIMATE_RTREE_INDEX_SIZE

               

                  

                     
格式
SDO_TUNE.ESTIMATE_RTREE_INDEX_SIZE（schemaname IN VARCHAR2，tabname IN VARCHAR2，colname IN VARCHAR2，partname IN VARCHAR2 DEFAULT NULL）RETURN NUMBER;
要么
SDO_TUNE.ESTIMATE_RTREE_INDEX_SIZE（number_of_geoms IN INTEGER，db_block_size IN INTEGER，sdo_rtr_pctfree IN INTEGER DEFAULT 10，num_dimensions IN INTEGER DEFAULT 2，is_geodetic IN INTEGER DEFAULT 0）RETURN NUMBER;

                  
                  

                     
描述

                     
估计R树空间索引表所需的最大兆字节数。

                  

                  
                  

                     
参数

                     

                        
 schemaname

                        

                           
拥有空间几何表的模式。

                        

                        
 tabname

                        

                           
空间几何表名称。

                        

                        
 colname

                        

                           
几何列名称。

                        

                        

                        

                           
包含colname几何的分区的名称。如果指定此参数，则函数返回的值是该分区中几何图形上的R树索引表的估计大小。如果未指定此参数，则该值是colname所有几何的R树索引表的估计大小。
                           

                        

                        
 number_of_geoms

                        

                           
空间几何表中的近似几何数。

                        

                        
 db_block_size

                        

                           
数据库块大小（以字节为单位）。

                        

                        
 sdo_rtr_pctfree

                        

                           
创建索引时，每个索引树节点中的最小槽位百分比为空。当新数据插入表中时，可以稍后填充留空的插槽。该值的范围为0到50。默认值（10）最适合大多数应用程序;但是，如果不对几何列执行任何更新，则建议值为0。

                        

                        
 num_dimensions

                        

                           
要编制索引的维数。默认值为2。如果计划在CREATE INDEX语句中指定sdo_indx_dims参数，则num_dimensions值应与sdo_indx_dims值匹配。
                           

                        

                        
 is_geodetic

                        

                           
指示空间索引是否为大地测量索引的值：1表示大地测量索引，或0（默认值）表示非大地测量索引。

                        

                     

                  

                  
                  

                     
使用说明

                     
该函数返回要创建的R树空间索引的空间索引表（在空间索引表定义中描述）所需的估计最大兆字节数。返回的值是索引创建后所需的最大兆字节数。在创建索引期间，表空间中将需要此值兆字节的大约三倍，以确保在创建索引时有足够的空间用于临时表。
                     

                     
此功能有两种格式：

                     
    
                        
  • 
                           
    在空间几何表已经存在的情况下，在大多数情况下使用带有字符串参数的格式（ schemaname ， tabname ， colname和可选的partname ），您不打算在创建索引之前向其添加更多的几何，并且您打算使用默认的R树索引参数。
                           
    
                        
    • 
                        
  • 
                           
    在以下任何情况下，使用带整数参数的格式（ number_of_geoms ， db_block_size ， sdo_rtr_pctfree ， num_dimensions ， is_geodetic ）：空间几何表不存在;存在空间几何表，但您计划在创建索引之前向其添加更多几何图形;或者对于非大地测量数据， num_dimensions值不为2，对于大地测量数据，则为3，并且将使用CREATE INDEX语句中的sdo_indx_dims参数指定非默认值。
                           
    
                        
    •

34.3 SDO_TUNE.EXTENT_OF

               

                  

                     
格式
SDO_TUNE.EXTENT_OF（table_name IN VARCHAR2，column_name IN VARCHAR2，use_index VARCHAR2默认为'TRUE'）RETURN SDO_GEOMETRY;

                  
                  

                     
描述

                     
返回图层中所有几何的最小边界矩形（MBR）。
                     

                  

                  
                  

                     
参数

                     

                        
 table_name

                        

                           
空间几何表或基于空间几何表的视图。

                        

                        
 column_name

                        

                           
要返回最小边界矩形的几何列。

                        

                        
 use_index

                        

                           
字符串TRUE （默认值）导致表中的空间索引（如果存在）用于计算。字符串FALSE会导致忽略任何空间索引。（有关此参数的详细信息，请参阅使用说明。）
                           

                        

                     

                  

                  
                  

                     
使用说明

                     
空间聚合函数中记录的SDO_AGGR_MBR函数也返回几何的MBR。如果在几何列上定义了空间索引，则SDO_TUNE.EXTENT_OF函数的性能优于SDO_AGGR_MBR函数;但是，SDO_TUNE.EXTENT_OF函数仅限于二维几何，而SDO_AGGR_MBR函数则不是。此外，SDO_TUNE.EXTENT_OF函数计算表中所有几何的范围;相反， SDO_AGGR_MBR函数可以对行的子集进行操作。
                     

                     
如果使用R树空间索引，则此函数可以返回包含索引中存储的最大数据范围的近似MBR，即使随后删除了数据。

                     
在大多数情况下， use_index的默认值TRUE最好，但在某些情况下，您可能需要指定FALSE 。空间索引有时可能具有大于实际数据范围的范围，因为DML操作可能会删除边缘周围的数据，从而导致实际范围变小，尽管此类删除不会减少索引范围。在这种情况下，如果您想要实际的数据范围，请指定use_index=>'FALSE' 。但请注意，指定FALSE会对函数的性能产生显着的负面影响，因为它需要全表扫描。
                     

                  

                  
                  

                     
例子

                     
以下示例计算COLA_MARKETS表的SHAPE列中对象的最小边界矩形。

3.1 Oracle性能改进方法

               

                  
Oracle性能方法可帮助您识别Oracle数据库中的性能问题。这涉及识别bo ttlenecks并修复它们。建议仅在确认存在瓶颈后才对系统进行更改。
                  

                  
从性质上来说，性能改进是迭代的。因此，消除第一个瓶颈可能不会立即导致性能提升，因为可能会出现另一个瓶颈。此外，在某些情况下，如果序列化点转移到效率更低的共享机制，那么性能可能会降低。凭借经验，并通过严格的瓶颈消除方法，可以调试应用程序并使其可扩展。

                  
性能问题通常是由于缺乏吞吐量，不可接受的用户/作业响应时间或两者兼而有之。问题可能是应用程序模块之间的本地化，也可能是整个系统的问题。

                  
在查看任何数据库或操作系统统计信息之前，从系统的最重要组件获取反馈至关重要：系统用户和最终为应用程序付费的人员。典型的用户反馈包括以下语句：

                  
    
                     
  • 
                        
    “在线表现非常糟糕，以至于无法阻止我的员工完成工作。”
    
                     
    • 
                     
  • 
                        
    “结算运行时间过长。”
    
                     
    • 
                     
  • 
                        
    “当我遇到大量网络流量时，响应时间变得不可接受，而我正在失去客户。”
    
                     
    • 
                     
  • 
                        
    “我目前每天进行5000次交易，系统最大化。下个月，我们向所有用户推出，预计交易数量将翻两番。“
    
                     
    • 
                  

                  
通过坦诚的反馈，很容易为任何绩效工作设定关键的成功因素。确定性能目标和性能工程师的退出标准使得在各个级别管理性能过程变得更加简单和成功。这些关键的成功因素在实际业务目标而非系统统计方面更好地定义。

                  
这些典型用户语句的一些实际业务目标可能是：

                  
    
                     
  • 
                        
    “计费运行必须在三个小时的时间内处理1,000,000个帐户。”
    
                     
    • 
                     
  • 
                        
    “在网站的高峰期，页面刷新的响应时间不得超过五秒。”
    
                     
    • 
                     
  • 
                        
    “该系统必须能够在8小时的时间内处理25,000笔交易。”
    
                     
    • 
                  

                  
成功的最终衡量标准是用户对系统性能的感知。性能工程师的职责是消除任何降低性能的瓶颈。这些瓶颈可能是由于有限共享资源的低效使用或滥用共享资源造成的，导致序列化。由于所有共享资源都是有限的，因此性能工程师的目标是通过有效使用共享资源来最大化业务操作的数量。在非常高的级别，整个数据库服务器可以被视为共享资源。相反，在较低级别，单个CPU或磁盘可视为共享资源。

                  
您可以应用Oracle性能改进方法，直到达到或认为不可能实现性能目标。这个过程是高度迭代的。不可避免地，一些调查可能对数据库性能影响很小或没有影响。时间和经验对于培养准确快速查明关键瓶颈的技能是必要的。但是，之前的经验有时会对有经验的工程师起作用，他们忽略了使用可用的数据和统计数据。这种行为鼓励通过神话和民间传说进行数据库调整。这是一种非常危险，昂贵且不太可能成功的数据库调优方法。

                  
自动数据库诊断监视器（ADDM）实现了部分性能改进方法并分析统计信息，以提供主要性能问题的自动诊断。使用ADDM可以显着缩短提高系统性能所需的时间。

                  
系统是如此不同和复杂，以至于难以实现严格的性能分析规则。实质上，Oracle性能改进方法定义了一种工作方式，但不是一组确定的规则。有了瓶颈检测，唯一的规则就是没有规则！最佳性能工程师使用提供的数据并横向思考以确定性能问题。

               
3.1.1 Oracle性能改进方法中的步骤

                  

                     

                        
    
                           
  1. 
                              
    执行以下初始标准检查：
    
                              
      
                                 
    1. 
                                    
      获得用户的坦诚反馈。确定绩效项目的范围和后续绩效目标，以及未来的绩效目标。这个过程是未来容量规划的关键。
      
                                 
      2. 
                                 
    2. 
                                    
      当性能好坏时，从系统获取一整套操作系统，数据库和应用程序统计信息。如果这些不可用，那么获取任何可用的东西。缺少统计数据类似于在犯罪现场缺少证据：它们使侦探工作更加困难，而且更耗时。
      
                                 
      3. 
                                 
    3. 
                                    
      完整性 - 检查涉及用户性能的所有计算机的操作系统。通过完整性检查操作系统，您可以查找充分利用的硬件或操作系统资源。将任何过度使用的资源列为症状以便稍后进行分析。此外，检查所有硬件是否显示错误或诊断。
      
                                 
      4. 
                              
    
                           
    2. 
                           
  2. 
                              
    检查Oracle数据库中最常见的十大错误，并确定是否存在任何问题。将这些列为症状以供以后分析。包含这些是因为它们代表了最可能出现的问题。ADDM会自动检测并报告这十大问题中的九个。
    
                           
    3. 
                           
  3. 
                              
    使用症状作为线索来构建系统上发生的事情的概念模型，以了解导致性能问题的原因。请参阅“ 性能概念建模的示例决策过程 ” 。
                              
    
                           
    4. 
                           
  4. 
                              
    向系统提出一系列补救措施和预期行为，然后按照最有利于应用程序的顺序应用它们。ADDM提出的建议均具有预期收益。性能工作的一个黄金法则是，您一次只能改变一件事，然后测量差异。不幸的是，系统停机要求可能会禁止这种严格的调查方法。如果同时应用多个更改，请尝试确保它们是隔离的，以便可以独立验证每个更改的效果。
    
                           
    5. 
                           
  5. 
                              
    验证所做的更改是否具有所需的效果，并查看用户对性能的感知是否有所改善。否则，寻找更多瓶颈，并继续完善概念模型，直到您对应用程序的理解变得更加准确。
    
                           
    6. 
                           
  6. 
                              
    重复最后三个步骤，直到达到性能目标或由于其他约束而变得不可能。
    
                           
    7. 
                        

                        
此方法确定了最大的瓶颈，并使用客观的方法来提高性能。重点是通过提高应用程序效率和消除资源短缺和瓶颈来大幅提高性能。在此过程中，预计实例调整可以获得最小（低于10％）的性能提升，并且可以通过隔离应用程序的低效率获得大量增益（100％+）。

                     

                     
                  

               
3.1.2性能概念建模的样本决策过程

                  

                     
概念建模几乎是确定性的。但是，随着您获得性能调优的经验，您开始意识到不存在真正的规则。需要灵活的单挑方法来解释统计数据并做出正确的决策。
                     

                     
要快速轻松地进行性能调整，请使用ADDM。 ADDM会自动监视您的Oracle系统，并提供解决性能问题的建议。例如，假设DBA收到来自抱怨系统运行缓慢的用户的呼叫。DBA只是检查最新的ADDM报告，以了解应该实施哪些建议来解决问题。

                     
以下步骤说明了性能工程师如何在不使用自动诊断功能的情况下查找瓶颈。这些步骤仅作为手动过程的指导。凭借经验，性能工程师可以添加相关步骤。此分析假定已收集操作系统和数据库的统计信息。

                     
    
                        
  1. 
                           
    空载或轻载计算机上的单个用户的响应时间/批处理运行时间是否可接受？
    
                           
    如果不可接受，那么应用程序可能没有被编码或设计得最佳，并且在共享系统资源时在多用户情况下永远不会被接受。在这种情况下，获取应用程序内部统计信息，并获取SQL跟踪和SQL计划信息。与开发人员一起调查数据，索引，事务SQL设计中的问题以及可能将工作推迟到批处理和后台处理。
    
                        
    2. 
                        
  2. 
                           
    是否正在使用所有CPU？
    
                           
    如果内核利用率超过40％，则调查操作系统以进行网络传输，分页，交换或处理颠簸。继续检查用户空间中的CPU利用率，以验证系统上是否存在消耗CPU的非数据库作业，从而限制共享CPU资源的数量，例如备份，文件转换，打印队列等。在确定数据库正在使用大部分CPU之后，按CPU利用率调查顶级SQL。这些陈述构成了所有未来分析的基础。检查SQL和提交SQL的事务以获得最佳执行效果。Oracle数据库在V$SQL和V$SQLSTATS提供CPU统计信息。

    有关V$SQL和V$SQLSTATS更多信息，请参见Oracle数据库参考
    
                           
    
                           
    如果应用程序是最佳的并且SQL执行中不存在低效率，则考虑将某些工作重新安排到非高峰时段或使用更大的计算机。
    
                        
    3. 
                        
  3. 
                           
    此时，系统性能不令人满意，但CPU资源未得到充分利用。
    
                           
    在这种情况下，您在服务器中具有序列化和不可扩展的行为。从服务器获取WAIT_EVENTS统计信息，并确定最大的序列化点。如果没有序列化点，则问题很可能在数据库之外，这应该是调查的重点。消除WAIT_EVENTS涉及修改应用程序SQL和调整数据库参数。此过程非常迭代，需要能够系统地向下钻取WAIT_EVENTS以消除序列化点。

3.1.3 Oracle Systems中发现的十大错误

                  

                     
本节列出了Oracle数据库中最常见的错误。通过遵循Oracle性能改进方法，您应该能够完全避免这些错误。如果您在系统中发现了这些错误，那么请重新设计应用程序，以确保您的性能。

                     
    
                        
  1. 
                           
    连接管理不良
    
                           
    应用程序连接和断开每个数据库交互。此问题在应用程序服务器中的无状态中间件中很常见。它对性能有超过两个数量级的影响，并且完全不可扩展。
    
                        
    2. 
                        
  2. 
                           
    游标和共享池的使用不当
    
                           
    不使用游标会导致重复解析。如果未使用绑定变量，则会对所有SQL语句进行硬解析。这对性能有一个数量级的影响，并且完全不可扩展。使用带有绑定变量的游标，打开游标并多次执行。怀疑生成动态SQL的应用程序。
    
                        
    3. 
                        
  3. 
                           
    糟糕的SQL
    
                           
    错误的SQL是使用比适用于应用程序要求的资源更多的SQL。这可以是运行超过24小时的决策支持系统（DSS）查询，也可以是需要超过一分钟的在线应用程序查询。您应该研究消耗大量系统资源以进行潜在改进的SQL。ADDM识别高负载SQL。 SQL Tuning Advisor可以提供改进建议。
    
                        
    4. 
                        
  4. 
                           
    使用非标准初始化参数
    
                           
    这些可能是基于糟糕的建议或不正确的假设而实施的。大多数数据库仅使用一组基本参数提供可接受的性能。特别是，与锁存器上的SPIN_COUNT和未记录的优化器功能相关联的参数可能会导致大量问题，需要进行大量调查。
                           
    
                           
    同样，初始化参数文件中设置的优化器参数可以覆盖经过验证的最佳执行计划。出于这些原因，应将模式，模式统计信息和优化程序设置作为一个组进行管理，以确保性能的一致性。
    
                           
    
                              
    也可以看看：
    
                              
      
                                 
    • 
                                    
      有关初始化参数和数据库创建的信息，请参见Oracle数据库管理员指南
      
                                 
      • 
                                 
    • 
                                    
       Oracle Database Reference有关初始化参数的详细信息
      
                                 
      • 
                              
    
                           
    
                        
    5. 
                        
  5. 
                           
    获取数据库I / O错误
    
                           
    许多站点在可用磁盘上布置数据库的能力很差。其他站点错误地指定了磁盘数，因为它们按磁盘空间而不是I / O带宽配置磁盘。
    
                        
    6. 
                        
  6. 
                           
    在线重做日志设置问题
    
                           
    许多网站运行的在线重做日志文件太少，文件太小。小型重做日志文件会导致系统检查点不断对缓冲区缓存和I / O系统施加高负荷。如果存在太少的重做日志文件，则存档无法跟上，数据库必须等待存档器赶上。
    
                        
    7. 
                        
  7. 
                           
    由于缺少空闲列表，空闲列表组，事务槽（ INITRANS ）或回滚段不足而导致缓冲区高速缓存中的数据块序列化。
                           
    
                           
    这在INSERT -heavy应用程序，块大小超过8K的应用程序或具有大量活动用户和少量回滚段的应用程序中尤为常见。使用自动段空间管理（ASSM）和自动撤消管理来解决此问题。
                           
    
                        
    8. 
                        
  8. 
                           
    长时间全表扫描
    
                           
    对于大量或交互式在线操作的长全表扫描可能表明交易设计不佳，索引丢失或SQL优化不佳。长桌扫描本质上是I / O密集型和不可扩展的。
    
                        
    9. 
                        
  9. 
                           
    大量的递归（ SYS ）SQL
    
                           
    SYS执行的大量递归SQL可能表明发生了空间管理活动，例如扩展区分配。这是不可扩展的，会影响用户响应时间。使用本地管理的表空间来减少由于扩展区分配而导致的递归SQL。在另一个用户ID下执行的递归SQL可能是SQL和PL / SQL，这不是问题。
                           
    
                        
    10. 
                        
  10. 
                           
    部署和迁移错误
    
                           
    在许多情况下，应用程序使用太多资源，因为拥有表的模式尚未从开发环境或旧实现成功迁移。例如缺少索引或不正确的统计信息。这些错误可能导致次优执行计划和较差的交互式用户性能。迁移已知性能的应用程序时，请使用DBMS_STATS程序包导出架构统计信息以维护计划稳定性。
                           
    
                           
    虽然ADDM不会直接检测到这些错误，但ADDM会突出显示由此产生的高负载SQL。

3.2紧急性能方法

               

                  
本节提供了处理的技巧表现紧急情况。您可能有一种建立和改进应用程序性能的方法。但是，在紧急情况下，系统的一个组件已经改变，将其从可靠，可预测的系统转换为不可预测且不满足用户请求的系统。
                  

                  
在这种情况下，性能工程师必须快速确定已更改的内容并采取适当的操作以尽快恢复正常服务。在许多情况下，有必要立即采取行动，严格的绩效改进项目是不现实的。

                  
在解决了即时性能问题之后，性能工程师必须收集足够的调试信息，以便更好地清楚性能问题，或者至少确保不会再次发生。

                  
调试紧急性能问题的方法与本书前面的性能改进方法中描述的方法相同。但是，由于问题的及时性，快捷方式分为不同阶段。随着调试过程的进展，保留详细的事实记录和事实记录对于后续分析和任何补救措施的理由至关重要。这类似于医生保留良好的患者注意事项以供将来参考。

               
3.2.1紧急性能方法的步骤

                  

                     

                        
该紧急性能方法如下：

                     

                     
                     
    
                        
  1. 调查性能问题并收集性能问题的症状。此过程应包括以下内容：
    
                              
      
                                 
    • 
                                    
      用户对系统如何表现不佳的反馈。是问题吞吐量还是响应时间？
      
                                 
      • 
                                 
    • 
                                    
      问一个问题，“自从我们上次表现良好以来发生了什么变化？“这个答案可以提供问题的线索。但是，在升级的情况下获得无偏见的答案可能很困难。尝试找到问题之前和之后拍摄的一些参考点，例如收集的统计信息或日志文件。
      
                                 
      • 
                                 
    • 
                                    
      使用自动调整功能来诊断和监控问题。此外，您还可以使用Oracle Enterprise Manager云控制（云控制）性能功能来识别顶级SQL和会话。
      
                                 
      • 
                              
    
                           
    
                        
    2. 
                        
  2. 完整性 - 检查应用程序系统的所有组件的硬件利用率。检查CPU利用率最高的位置，并检查所有系统组件上的磁盘，内存使用情况和网络性能。此快速过程可识别导致问题的层。如果问题出在应用程序中，则将分析转移到应用程序调试。否则，继续进行数据库服务器分析。
    3. 
                        
  3. 确定数据库服务器是否受CPU限制，或者是否花费时间等待等待事件。如果数据库服务器受CPU限制，请调查以下内容：
    
                              
      
                                 
    • 
                                    
      在操作系统级别和数据库中消耗大量CPU的会话;检查V$SESS_TIME_MODEL以了解数据库CPU使用情况
      
                                 
      • 
                                 
    • 
                                    
      执行许多缓冲区的会话或语句在数据库级别获得;检查V$SESSTAT和V$SQLSTATS
      
                                 
      • 
                                 
    • 
                                    
      执行计划更改导致次优SQL执行;这些可能很难找到
      
                                 
      • 
                                 
    • 
                                    
      初始化参数设置不正确
      
                                 
      • 
                                 
    • 
                                    
      代码更改或所有组件升级导致的算法问题
      
                                 
      •

261.1 UTL_MATCH概述

               

                  
在确定匹配时， UTL_MATCH可以使用编辑距离算法或Jaro-Winkler算法。
                  

                  

                     
编辑距离，也称为Levenshtein距离（以俄罗斯科学家Vladimir Levenshtein命名，他于1965年设计算法），是两个字符串s1和s2之间相似性的度量。距离是将s1变换为s2所需的插入，删除或替换的数量。
                     

                     
字符串shack le ford和shack el ford之间的编辑距离= 2。
                     

                     
“Jaro-Winkler算法”是计算两个字符串之间的编辑距离的另一种方法。在美国人口普查中开发的这种方法是一种字符串比较器度量，它给出了两个字符串之间部分一致的值。字符串比较器考虑字符串的长度，并部分地解释了在字母数字字符串中产生的典型人为错误。
