10.1关于模板对象

Template类是Oracle OLAP Java API非常强大功能的基础。您使用Template对象来创建可修改的Source对象。使用这些Source对象，您可以创建可以响应最终用户选择而更改的动态查询。Template对象还为您提供了一种将用户界面元素转换为OLAP Java API操作和对象的便捷方式。

有关用于更改动态Source的Transaction对象以及保存或放弃这些更改的信息，请参阅使用TransactionProvider 。

10.1.1关于创建动态源

Template的主要特征是能够生成动态Source 。该功能基于Template使用的两个其他对象： DynamicDefinition和MetadataState类的实例。

创建Source ，Oracle OLAP会自动将SourceDefinition与其关联。SourceDefinition包含有关Source信息。一旦创建， Source和相关的SourceDefinition地关联起来。SourceDefinition的getSource方法返回与之关联的Source 。

DynamicDefinition是SourceDefinition的子类。甲Template创建DynamicDefinition ，其充当代理SourceDefinition所述的Source通过所产生的Template 。这意味着， DynamicDefinition的getSource方法不是始终获得相同的不可变关联的Source ，而是获取Template当前生成的任何Source 。即使它获得的Source不同， DynamicDefinition的实例也不会更改。

Template生成的Source可以更改，因为Template用于创建Source的值（包括其他Source对象）可以更改。Template将这些值存储在MetadataState 。Template提供了获取MetadataState的当前状态，获取或设置值以及设置状态的方法。您可以使用这些方法更改MetadataState存储的数据值。

您使用DynamicDefinition来获取Template生成的Source 。如果您的应用程序更改了Template用于创建Source的值的状态，例如，为了响应最终用户选择，则应用程序使用相同的DynamicDefinition再次获取Source ，即使新Source定义了结果设置与上一个Source不同。

Template生成的Source可以是一系列创建其他Source对象的Source操作的结果，例如一系列选择，排序，计算和连接。您将这些操作的代码放在Template的SourceGenerator的generateSource方法中。该方法返回Template生成的Source 。操作使用存储在MetadataState中的MetadataState 。

您可能构建一个极其复杂的查询，该查询涉及由许多不同的Template对象生成的动态Source对象的交互。查询构建的最终结果是定义整个复杂查询的Source 。如果更改用于创建最终Source的任何一个Template对象的状态，则最终Source表示的结果集与前一个Source的结果集不同。因此，您可以修改最终查询，而无需重现定义查询所涉及的所有操作。

10.1.2关于将用户界面元素转换为OLAP Java API对象

您设计Template对象以表示应用程序的用户界面的元素。您的Template对象将最终用户所做的选择转换为生成Source OLAP Java API查询构建操作。然后创建一个Cursor ，从Oracle OLAP获取Source定义的结果集。您从Cursor获取值并将其显示给最终用户。当最终用户更改选择时，您可以更改Template的状态。然后，您将获得Template生成的Source ，创建一个新的Cursor ，获取新值并显示它们。

10.2模板和相关类概述

在OLAP Java API中，几个类一起工作以生成动态Source 。在设计Template ，您必须实现或扩展以下内容：

  • Template抽象类

  • MetadataState接口

  • SourceGenerator接口

这三个类的实例以及DataProvider和DynamicDefinition类的实例一起工作以生成Template定义的Source 。

10.2.1生成动态源的类之间的关系是什么？

生成动态Source的类一起工作如下：

  • Template具有创建DynamicDefinition并获取和设置MetadataState的当前状态的方法。Template抽象类的扩展添加了获取和设置MetadataState上字段值的方法。

  • MetadataState实现具有用于存储用于生成Template Source的数据的字段。创建新Template ，将MetadataState传递给Template的构造函数。当您调用DynamicDefinition的getSource方法时， MetadataState将传递给SourceGenerator的generateSource方法。

  • DataProvider用于创建Template ，由SourceGenerator用于创建新的Source对象。

  • SourceGenerator实现有一个generateSource方法，该方法使用MetadataState数据的当前状态来生成Template的Source 。您将SourceGenerator传递给Template的createDynamicDefinition方法以创建DynamicDefinition 。

  • 所述DynamicDefinition具有getSource方法，其获取Source所生产SourceGenerator 。DynamicDefinition充当SourceDefinition的代理， SourceDefinition与Source SourceDefinition关联。

10.2.2模板类

您使用Template生成可修改的Source 。Template具有创建DynamicDefinition以及获取和设置Template当前状态的方法。在扩展Template类时，您添加的方法可以访问Template的MetadataState上的字段。Template创建一个DynamicDefinition ，用于获取SourceGenerator为Template生成的Source 。

有关Template实现的示例，请参见示例10-1 。

10.2.3 MetadataState接口

MetadataState接口的实现存储Template的值的当前状态。MetadataState必须包含一个clone方法，该方法可以创建当前状态的副本。

实例化新Template ，将MetadataState传递给Template构造函数。Template具有获取和设置MetadataState存储的值的方法。所述generateSource所述的方法SourceGenerator为Template使用MetadataState当该方法产生一个Source的Template 。

有关MetadataState实现的示例，请参见示例10-2 。

10.2.4 SourceGenerator接口

SourceGenerator的实现必须包含generateSource方法，该方法为Template生成Source 。SourceGenerator必须只生成一种类型的Source ，例如BooleanSource ， NumberSource或StringSource 。在生成Source ， generateSource方法使用MetadataState为Template表示的数据的当前状态。

要获得Source所生产generateSource方法，您可以创建一个DynamicDefinition通过将SourceGenerator到createDynamicDefinition的方法Template 。然后通过调用DynamicDefinition的getSource方法获取Source 。

Template可以创建多个DynamicDefinition ，每个都有一个不同的SourceGenerator 。不同SourceGenerator对象的generateSource方法使用相同的数据（由Template的MetadataState的当前状态定义）来生成定义不同查询的Source对象。

有关SourceGenerator实现的示例，请参见示例10-3 。

10.2.5 DynamicDefinition类

DynamicDefinition是SourceDefinition的子类。您可以通过调用Template的createDynamicDefinition方法并将其传递给SourceGenerator创建DynamicDefinition 。您可以通过调用DynamicDefinition的getSource方法获取SourceGenerator生成的Source 。

甲DynamicDefinition通过创建Template是用于代理SourceDefinition所述的Source所生产SourceGenerator 。SourceDefinition与Source SourceDefinition相关联。如果的状态Template的变化，则该Source所生产SourceGenerator是不同的。由于DynamicDefinition是一个代理，因此即使Source具有不同的SourceDefinition ，您也可以使用相同的DynamicDefinition来获取新的Source 。

DynamicDefinition的getCurrent方法返回与generateSource方法当前返回的Source不可变地关联的SourceDefinition 。有关使用DynamicDefinition 示例 ，请参见示例10-4 。

10.3设计和实现模板

Template的设计反映了应用程序用户界面的查询构建元素。例如，假设您要开发一个应用程序，该应用程序允许最终用户创建一个查询，该查询从值列表的顶部或底部请求许多值。值来自度量的一个维度。该度量的其他维度仅限于单个值。

应用程序的用户界面有一个对话框，允许最终用户执行以下操作：

  • 选择一个单选按钮，指定数据值是应该从值范围的顶部还是底部。

  • 从度量下拉列表中选择度量。

  • 从字段中选择一个数字。该数字指定要显示的数据值的数量。

  • 选择度量的某个维度作为要显示的数据值的基础。例如，如果用户选择产品维度，则查询会从产品列表的顶部或底部指定一定数量的产品。该列表由度量和其他维度的选定值确定。

  • 单击按钮以显示一个对话框，最终用户通过该对话框为所选度量的其他维度选择单个值。选择尺寸值后，最终用户单击第二个对话框上的“确定”按钮并返回到第一个对话框。

  • 单击“确定”按钮以生成查询。查询结果出现。

要生成表示最终用户在第一个对话框中创建的查询的Source ，请设计名为TopBottomTemplate的Template 。您还设计了第二个名为SingleSelectionTemplate Template ，以创建一个Source ，该Source表示最终用户对基本维度以外的维度的单个值的选择。Template对象的设计反映了对话框的用户界面元素。

在设计TopBottomTemplate及其MetadataState和SourceGenerator ，您将执行以下操作：

  • 创建一个名为TopBottomTemplate的类来扩展Template 。在类中，添加获取Template当前状态的方法，设置用户指定的值，然后设置Template的当前状态。

  • 创建一个名为TopBottomTemplateState的类，该类实现MetadataState 。您在类上提供字段以存储SourceGenerator值，以用于生成Template生成的Source 。这些值由TopBottomTemplate的方法设置。

  • 创建一个名为TopBottomTemplateGenerator的类，它实现SourceGenerator 。在类的generateSource方法中，您提供了创建最终用户选择指定的Source的操作。

使用您的应用程序，最终用户在第一个对话框中选择作为度量销售的单位和产品作为基本维度。最终用户还选择2001年第一季度的亚太地区和直接销售渠道作为每个剩余维度的单一值。

最终用户创建的查询要求在2001日历年期间通过直销渠道向亚太地区的客户销售的单位总数最高的十种产品。

对于的实施方式的示例TopBottomTemplate ， TopBottomTemplateState ，和TopBottomTemplateGenerator类，以及使用它们的应用程序的一个例子，请参见实施例10-1 ， 实施例10-2 ， 实施例10-3 和 实施例10-4 。TopBottomTemplateState和TopBottomTemplateGenerator类是作为TopBottomTemplate外部类的内部类实现的。

18.1了解操作系统性能问题

操作系统性能问题通常涉及进程管理，内存管理和调度。如果您已调整Oracle数据库实例并仍需要提高性能，请验证您的工作或尝试缩短系统时间。确保有足够的I / O带宽，CPU功率和交换空间。但是，不要指望进一步调整操作系统会对应用程序性能产生重大影响。与简单地调整操作系统相比，Oracle数据库配置或应用程序中的更改可能会导致操作系统效率的显着差异。

例如，如果应用程序遇到过多的缓冲区忙等待，则系统调用的数量会增加。如果通过调整应用程序来减少缓冲区忙等待，则系统调用的数量会减少。

  • 内存使用情况

18.1.1使用操作系统缓存

操作系统和设备控制器提供不与Oracle数据库缓存管理直接冲突的数据缓存。尽管如此，这些结构可以消耗资源，同时提供很少或没有性能优势。当数据库文件存储在Linux或UNIX文件系统中时，这种情况最为明显。默认情况下，所有数据库I / O都通过文件系统缓存。

在某些Linux和UNIX系统上，文件存储可以使用直接I / O.这种安排允许在文件系统内访问数据库文件，绕过文件系统缓存。直接I / O可节省CPU资源，并允许文件系统缓存专用于非数据库活动，例如程序文本和假脱机文件。

注意：

在Windows上不会发生此问题。数据库的所有文件请求都会绕过文件系统中的缓存。

虽然操作系统缓存通常是多余的，因为Oracle数据库缓冲区缓存缓冲区块，但在某些情况下，数据库不使用数据库缓冲区缓存。在这些情况下，使用直接I / O或原始设备可能会比使用操作系统缓冲产生更差的性能。例子包括：

  • 读取或写入TEMP表空间

  • 数据存储在NOCACHE LOB中

  • 并行执行服务器读取数据

    注意：

    在某些情况下，数据库可以将并行查询数据缓存在数据库缓冲区缓存中，而不是直接从磁盘读取到PGA中。当数据库服务器具有大量内存时，此配置可能是适当的。请参阅Oracle数据库VLDB和分区指南以了解有关使用并行执行的更多信息。

您可能希望缓存但不是操作系统级别的所有文件。

18.1.1.1异步I / O.

对于同步I / O，当I / O请求提交给操作系统时，写入过程将一直阻塞，直到写入被确认为完成。然后它可以继续处理。使用异步I / O，在提交和处理I / O请求时继续处理。尽可能使用异步I / O以避免瓶颈。

某些平台默认支持异步I / O，其他平台需要特殊配置，有些平台仅支持某些底层文件系统类型的异步I / O.

18.1.1.2 FILESYSTEMIO_OPTIONS初始化参数

您可以使用FILESYSTEMIO_OPTIONS初始化参数来启用或禁用文件系统文件上的异步I / O或直接I / O.此参数是特定于平台的，并且具有最适合特定平台的默认值。

FILESYSTEMIO_OPTIONS可以设置为以下值之一：

  • ASYNCH:在文件系统文件上启用异步I / O，对传输没有时间要求。

  • DIRECTIO:在文件系统文件上启用直接I / O，绕过缓冲区缓存。

  • SETALL:在文件系统文件上启用异步和直接I / O.

  • NONE:禁用文件系统文件上的异步和直接I / O.

18.1.1.3限制NFS服务器环境中的异步I / O.

在某些网络文件存储（NFS）服务器环境中，如果在短时间内发出大量异步I / O请求，性能可能会受到影响。在这种情况下，通过限制Oracle进程发出的I / O数量，使用DNFS_BATCH_SIZE初始化参数来提高系统性能并提高系统稳定性。

DNFS_BATCH_SIZE初始化参数控制启用Direct NFS客户端时Oracle前台进程可排队的异步I / O数。在NFS服务器无法处理大量未完成的异步I / O请求的环境中，Oracle建议将此参数设置为128。然后，您可以根据NFS服务器的性能增加或减少其值。

注意：

DNFS_BATCH_SIZE初始化参数的默认设置为4096。建议值128仅适用于NFS服务器无法处理大量异步I / O请求且检测到严重延迟的系统。

18.1.1.4使用Direct NFS Client提高I / O性能

Direct NFS Client直接在Oracle数据库中集成NFS客户端功能。由于Direct NFS Client是Oracle数据库的专用NFS客户端，因此它经过高度优化。与传统的操作系统NFS客户端相比，Direct NFS Client大大提高了NFS上的数据库性能。

并行NFS是直接NFS客户端的可选功能，它在NFS版本4.1中引入，并受Oracle Database 12 c第2版（12.2）及更高版本支持。并行NFS是一种高度可扩展的分布式存储协议，客户端，服务器和存储设备负责管理文件访问。在4.1之前的NFS版本中，只有服务器负责管理文件访问。因此，Parallel NFS支持高度可扩展的分布式NAS存储，以实现更好的I / O性能。

从Oracle Database 12 c第2版（12.2）开始，您还可以使用Direct NFS Client的Direct NFS调度程序功能。Direct NFS调度程序将从数据库实例创建的TCP连接合并到NFS服务器。在大型数据库部署中，使用Direct NFS调度程序可提高可伸缩性和网络性能。因此，对于大量TCP连接，Oracle建议使用Direct NFS调度程序和Parallel NFS进行直接NFS客户端部署。

18.1.2内存使用情况

内存使用受缓冲区缓存限制和初始化参数的影响。

18.1.2.1缓冲区高速缓存限制

UNIX缓冲区高速缓存消耗操作系统内存资源。虽然在某些版本的UNIX中，UNIX缓冲区高速缓存可以分配一定量的内存，但是现在通常使用更复杂的内存管理机制。通常，这些将允许使用空闲内存页来缓存I / O.在这样的系统中，操作系统报告工具通常显示没有空闲存储器，这通常不是问题。如果进程需要更多内存，则通常会释放内存缓存I / O数据以允许分配进程内存。

18.1.2.2影响内存使用的参数

任何一个Oracle数据库会话所需的内存取决于许多因素。通常，主要因素是：

  • 打开游标的数量

  • PL / SQL使用的内存，例如PL / SQL表

  • SORT_AREA_SIZE初始化参数

在Oracle数据库中， PGA_AGGREGATE_TARGET初始化参数可以更好地控制会话的内存使用情况。

18.1.3使用操作系统资源管理器

某些平台提供操作系统资源管理器。这些旨在通过优先访问系统资源来减少峰值负载使用模式的影响。它们通常实现管理策略，用于管理用户可以访问的资源以及允许每个用户使用的资源量。

操作系统资源管理器与域或其他类似设施不同。域在一个系统内提供一个或多个完全分离的环境。磁盘，CPU，内存和所有其他资源专用于每个域，无法从任何其他域访问。其他类似的设施将系统资源的一部分完全分离到不同的区域，通常是单独的CPU或内存区域。与域一样，单独的资源区域仅专用于分配给该区域的处理;流程无法跨越边界迁移。与域不同，系统上的所有分区都可以访问所有其他资源（通常是磁盘）。

只要分区内存（RAM）资源的分配是固定的，而不是动态的，Oracle数据库就可以在域内运行，也可以在其他不太完整的分区结构中运行。

操作系统资源管理器优先考虑全局资源池（通常是域或整个系统）内的资源分配。将进程分配给组，这些组又在资源池中的任何位置分配资源。

注意：

  • 如果节点上有多个实例，并且您希望在它们之间分配资源，则应将每个实例分配给专用的操作系统资源管理器组或管理实体。要在管理实体中运行多个实例，请使用实例锁定来管理如何在实例之间分配管理实体中的CPU资源。当Oracle数据库资源管理器管理CPU资源时，它需要为该实例提供固定数量的CPU资源。如果没有实例囚禁，它期望可用的CPU资源等于被管实体中的CPU数量。使用实例caging，它期望可用的CPU资源等于CPU_COUNT初始化参数的值。如果CPU资源少于预期，则Oracle数据库资源管理器在执行资源计划中的资源分配方面效果不佳。

  • 不支持将Oracle数据库用于任何UNIX操作系统资源管理器的内存管理和分配工具。Oracle数据库资源管理器在Oracle数据库实例中提供资源分配功能，不能与任何操作系统资源管理器一起使用。

    有关与Oracle数据库和Oracle数据库资源管理器一起使用的操作系统资源管理以及资源分配和释放功能的完整列表，请咨询您的系统供应商和Oracle代表。Oracle不会证明这些系统功能与特定版本级别的兼容性。

18.2.1基于UNIX的系统的性能提示

上 UNIX系统尝试在操作系统完成系统调用和执行进程调度所花费的时间与应用程序运行的时间之间建立良好的比率。目标应该是在应用程序模式（也称为用户模式）而不是系统模式下运行大部分时间。

每种模式花费的时间比例只是潜在问题的症状，可能涉及以下方面：

  • 分页或交换

  • 执行太多操作系统调用

  • 运行太多进程

如果存在这样的条件，那么应用程序运行的可用时间就会减少。从操作系统端释放的时间越长，应用程序可以执行的事务就越多。

18.2.2 Windows系统上的性能提示

在Windows系统上，与基于UNIX的系统一样，在应用程序模式下的时间与系统模式下的时间之间建立适当的比率。您可以使用Windows管理性能工具轻松监控许多因素：CPU，网络，I / O和内存都显示在同一图表中，以帮助您避免任何这些区域中的瓶颈。

18.2.3 HP OpenVMS系统的性能提示

考虑大型机上的分页参数，并记住Oracle数据库可以利用非常大的工作集。

HP OpenVMS环境中的可用内存实际上是未映射到任何操作系统进程的内存。在繁忙的系统上，可用内存可能包含属于一个或多个当前活动进程的页面。发生该访问时，会发生soft page fault ，并且该页面将包含在该进程的工作集中。如果进程无法扩展其工作集，则必须将进程当前映射的其中一个页面移动到空闲集。

任意数量的进程可能在其工作集中具有共享内存页面。因此，工作集的大小总和可以显着超过可用存储器。当Oracle服务器运行时，SGA，Oracle数据库内核代码和Oracle Forms运行时可执行文件通常都是可共享的，占访问页面的80％或90％。

18.3了解CPU

要解决CPU问题，请首先针对系统应使用的CPU资源量建立适当的期望。然后，确定是否有足够的CPU资源可用，并识别系统何时消耗过多资源。首先确定Oracle数据库实例在以下三种情况下对系统使用的CPU资源量：

  • 当存在少量Oracle数据库和非Oracle活动时，系统处于空闲状态

  • 系统平均工作负载

  • 高峰工作负载下的系统

您可以使用Automatic Workload Repository，Statspack或UTLBSTAT / UTLESTAT实用程序捕获各种工作负载快照。操作系统实用程序（如UNIX上的vmstat ， sar和iostat以及Windows上的管理性能监视工具）可以与自动工作负载存储库，Statspack或同一时间间隔内的V$OSSTAT或V$SYSMETRIC_HISTORY视图一起使用。 UTLBSTAT / UTLESTAT提供整体统计数据的免费视图。

在评估系统的CPU利用率时，工作负载是一个重要因素。在高峰工作负载时间内，90％的CPU利用率，10％的空闲和等待时间是可以接受的。在低工作负载时，即使30％的利用率也是可以理解的。但是，如果您的系统在正常工作负载下显示高利用率，则没有高峰工作负载的空间。例如，下图说明了在上午10:00和下午2:00具有高峰期的应用程序随时间的工作负载。

此示例应用程序有100个用户每天工作8小时。每5分钟输入一个事务的每个用户每天转换为9,600个事务。在8小时内，系统必须每小时支持1,200笔交易，平均每分钟交易20笔。如果需求率保持不变，那么您可以构建一个系统来满足此平均工作负载。

但是，使用模式不是恒定的，在这种情况下，每分钟20个事务可以理解为仅仅是最低要求。如果您需要达到的峰值速率是每分钟120个事务，那么您必须配置一个可以支持此峰值工作负载的系统。

20 tpm / 120 tpm * 90％=可用CPU资源的15％

其中tpm是一分钟的交易。

如果系统需要50％的CPU资源来达到20 tpm，那么就存在一个问题：使用90％的CPU，系统每分钟无法实现120个事务。但是，如果您调整此系统使其仅使用15％的CPU达到20 tpm，那么，假设线性可扩展性，系统可能使用90％的CPU资源每分钟实现120个事务。

随着用户被添加到应用程序中，工作负载可能会升至先前的峰值水平。然后，没有更多的CPU容量可用于新的峰值速率，实际上高于之前的峰值速率。

18.4解决CPU问题

  • 检测并解决过度消耗中的CPU问题，如“ 查找和调整CPU利用率 ”中所述 。

18.4.1.4.1调度和切换

18.4.1.4.2上下文切换

由于操作系统特定的特性，您的系统可能会花费大量时间上下文切换。上下文切换可能很昂贵，尤其是对于大型SGA。在Windows上，上下文切换不是问题，每个实例只有一个进程。所有线程共享相同的页表。

Oracle数据库具有多种上下文切换功能：

  • 等待后的司机

    Oracle进程必须能够发布另一个Oracle进程（给它一条消息），并且还必须能够等待发布。例如，前台进程可能需要发布LGWR以告知它写出到达给定点的所有块，以便它可以确认提交。

    这种后等待机制通常是通过UNIX信号量实现的，但这些可能是资源密集型的。因此，某些平台提供后等待驱动程序，通常是内核设备驱动程序，它是实现后等待接口的轻量级方法。

  • 内存映射系统计时器

    Oracle数据库通常需要查询系统时间以获取时序信息。这可能涉及操作系统调用，这会引起相对昂贵的上下文切换。某些平台实现了内存映射计时器，该计时器使用进程虚拟地址空间中的地址来包含当前时间信息。从这个内存映射定时器读取时间比系统调用的上下文切换开销要便宜。

  • 列出I / O接口以在One Call中提交多个异步I / O.

    List I / O是一个应用程序编程接口，允许在单个系统调用中提交多个异步I / O请求，而不是通过单独的系统调用提交多个I / O请求。此功能的主要好处是减少所需的上下文切换次数。

18.4.1.4.3启动新的操作系统进程

启动新的操作系统进程的成本很高。开发人员通常会创建一个单一用途的流程，退出流程，然后创建一个新流程。此技术每次都会重新创建和销毁进程，从而消耗过多的CPU，尤其是在具有大型SGA的应用程序中。每次都需要CPU来构建页表。当您锁定或锁定共享内存时，问题会更加严重，因为您必须访问每个页面。

例如，如果您有1千兆字节的SGA，那么您可能每4 KB有一个页表条目，而页表条目可能是8个字节。您最终可能会得到（1 GB / 4 KB）* 8字节条目。这变得很昂贵，因为您需要不断确保加载页表。

18.4.2使用Oracle数据库资源管理器管理CPU资源

神谕 Database Resource Manager通过以下方式在数据库用户和应用程序之间分配和管理CPU资源：

  • 防止CPU饱和

    如果CPU以100％运行，则可以使用Oracle数据库资源管理器为每个使用者组中的会话分配最大量的CPU。此功能可确保高优先级会话可立即运行并降低低优先级会话的CPU消耗。

  • 限制使用者组的CPU使用率

    您可以使用资源管理器指令max_utilization_limit对使用者组可以使用的CPU百分比设置硬限制。此功能限制低优先级会话的CPU消耗，并有助于为使用者组中的工作负载提供更一致的性能。

  • 限制失控查询造成的损害

    从Oracle Database 11g第2版（11.2.0.2）开始，Oracle数据库资源管理器可以通过限制呼叫的最长执行时间或将长时间运行的查询移动到较低优先级的使用者组来限制失控查询造成的损害。

  • 限制使用者组的并行语句活动

    从Oracle Database 11g第2版（11.2.0.2）开始，您可以使用资源管理器指令parallel_target_percentage来阻止一个使用者组独占所有并行服务器。如果数据库导致超出此限制，则会对并行语句进行排队。

性能设计

数据库和应用程序性能的关键是设计，而不是调优。虽然调整非常有价值，但它无法弥补糟糕的设计。您的设计必须从高效的数据模型，明确定义的性能目标和指标以及合理的基准测试策略开始。否则，您将在实施过程中遇到问题，因为没有多少调整会产生您可以通过良好设计获得的结果。您可能不得不稍后重新设计系统，尽管已经调整了原始的糟糕设计。

可扩展性设计

可扩展性是指数据库或数据库应用程序适应未来增长的难易程度。数据库或应用程序的可扩展性越高，就越容易添加或更改功能，而对现有功能的影响最小。

注意：

可扩展性与前向兼容性 ，应用程序从其自身的未来版本接受数据的能力以及仅使用其旨在接受的数据的能力不同。

例如，假设应用程序的早期版本仅处理文本，同一应用程序的更高版本处理文本和图形。如果早期版本可以接受文本和图形，并忽略图形和处理文本，那么它是向前兼容的。如果可以升级早期版本以处理文本和图形，那么它是可扩展的。升级应用程序越容易，它就越具有可扩展性。

为了最大限度地提高可扩展性，您必须将其设计到数据库和应用程序中，方法是包含允许在不对基础结构进行重大更改的情早期版本的数据库或应用程序可能不会使用这些机制，也许没有版本会使用所有这些机制，但它们对于易于维护和避免早期过时至关重要。

数据磁带

数据盒式磁带利用Oracle可扩展性架构框架扩展了Oracle数据库服务器的功能。通过此框架，您可以捕获与用户定义的数据类型中的专用或特定于域的数据相关联的业务逻辑和流程。在不使用其他属性的情况下提供新行为的数据盒式磁带可以使用包而不是用户定义的数据类型来实现。使用用户定义的类型或包，可以确定服务器如何解释，存储，检索和索引应用程序数据。数据盒式磁带打包此功能，创建插入服务器的软件组件，并将其功能扩展到新域，使数据库本身可扩展。

您可以自定义可扩展数据库管理系统的索引和查询优化机制，并为用户定义的业务对象和丰富类型提供专门的服务或更有效的处理。通过可扩展性接口向服务器注册实现时，可以指示服务器实现自定义处理指令，而不是自己的默认进程。

便携性设计

虽然PL / SQL不是为Oracle数据库和第三方数据库之间的可移植性而设计的，但它在操作系统和语言之间具有高度可移植性。大多数编程语言都可以调用PL / SQL，并且PL / SQL在支持Oracle数据库的每个平台上都是一致的，包括Macintosh，Linux和Windows。如果您在一个平台上开发PL / SQL应用程序，您可以非常确信它们将在所有其他平台上一致地工作。

PL / SQL存储过程提供跨多个数据库的一些应用程序可移植性。虽然使用以给定供应商的语言编写的存储过程可能在某种程度上将您与该供应商联系起来，但存储过程使应用程序的可视组件（用户界面）和应用程序逻辑可移植。数据逻辑针对运行应用程序的数据库进行了最佳编码。由于数据逻辑隐藏在存储过程中，因此您可以使用供应商的扩展和功能来优化数据层。

可诊断性设计

Oracle数据库包括用于预防，检测，诊断和解决数据库问题的故障诊断基础结构。问题包括严重错误，例如代码错误，元数据损坏和客户数据损坏。可诊断性基础架构的目标是主动检测问题，在检测到问题后限制损坏和中断，减少诊断和解决问题所需的时间，并简化与Oracle支持的任何可能的交互。

数据仓库

数据仓库是一个关系数据库，专为查询和分析而不是事务处理而设计。它通常包含从事务数据派生的历史数据，但可以包含来自其他来源的数据。数据仓库将分析工作负载与事务工作负载分开，并使组织能够合并来自多个源的数据。此策略可帮助组织维护历史记录并分析数据，以便更好地了解和改进其业务。

除了关系数据库之外，数据仓库环境还可以包括：

  • 提取，运输，转化和装载（ETL）解决方案

  • 统计分析

  • 报告

  • 数据挖掘功能

  • 客户分析工具

  • 管理收集数据过程的应用程序;将其转化为有用的，可操作的信息;并将其交付给业务用户

数据仓库系统通常：

  • 使用许多索引

  • 使用一些（但不是很多）连接

  • 使用非规范化或部分非规范化模式（例如星型模式）来优化查询和分析性能

  • 使用派生数据和聚合

  • 具有旨在适应即席查询和数据分析的工作负载

    由于您可能事先不知道数据仓库的工作负载，因此必须优化数据仓库以便在各种可能的查询和分析操作中正常运行。

  • 使用批量数据修改技术，通过ETL过程定期（每晚或每周）更新

    数据仓库的最终用户不会直接更新数据仓库，除非使用分析工具（如数据挖掘）进行相关概率的预测，将客户分配到细分市场并开发客户档案。

在线事务处理（OLTP）

在线事务处理（OLTP）系统经过优化，可实现快速可靠的事务处理。与数据仓库系统相比，大多数OLTP交互涉及相对较少的行，但涉及较大的一组表。在OLTP系统中，性能要求要求将历史数据频繁移动到存档。

OLTP系统通常：

  • 使用少量索引。

  • 使用许多连接。

  • 使用完全规范化的模式来优化更新，插入和删除性能，并保证数据的一致性。

  • 很少使用派生数据和聚合。

  • 具有由预定义操作组成的工作负载。

  • 让用户定期向数据库发出单独的数据修改语句，以便OLTP数据库始终反映每个事务的当前状态。

SQL分析函数

SQL分析函数根据一组行计算聚合值。SQL分析函数与聚合函数的不同之处在于它为每个组返回多行。对于每一行，定义一个行窗口。该窗口确定用于执行当前行计算的行范围。窗口大小可以基于物理行数或逻辑间隔（例如时间）。

SQL分析函数使面向集合的SQL能够在结果集上使用数组语义。它们提供编码效率，因为它们能够实现简单，直接的逻辑编码，否则会很麻烦或不可能。它们还提供了处理效率，因为它们是Oracle数据库不可或缺的，并使用内部优化。

分析函数的典型用途是检索表中的最新信息。例如，以下表单的查询将返回具有最新更新时间的行中的信息，其中每个客户的记录都在表中：

选择 ...FROM my_table t1 WHERE upd_time =（SELECT MAX（UPD _TIME） FROM my_table t2 WHERE t2.cust_id = t1.cust_id）;

前面的查询使用相关子查询来查找cust _id的MAX(UPD _TIME) ，逐个记录。因此，可以针对表中的每一行评估相关子查询一次。如果表记录很少，那么性能可能就足够了;如果该表具有数万条记录，则重复执行相关子查询的累积成本很高。

物化观点

物化视图是已作为模式对象存储（“物化”）的查询结果。与表和视图一样，物化视图可以出现在查询的FROM子句中。

物化视图用于汇总，计算，复制和分发数据。它们对预先回答一般问题类很有用 - 用户可以查询物化视图，而不是单独聚合详细记录。物化视图有用的一些环境是数据仓库，复制和移动计算。

物化视图需要时间来创建和更新，以及用于存储的磁盘空间，但这些成本可以通过显着更快的查询来抵消。在这些方面，物化视图就像索引，它们被称为“数据仓库的索引”。与索引不同，可以直接查询实例化视图（使用SELECT语句），有时使用DML语句进行更新（具体取决于所需的更新类型）。

创建和维护物化视图的一个主要好处是能够利用查询重写 ，该查询重写将表或视图表示的SQL语句转换为访问详细信息表中定义的一个或多个物化视图的语句。转换对最终用户或应用程序是透明的，不需要干预，也不需要引用SQL语句中的物化视图。由于查询重写是透明的，因此可以像索引一样添加或删除实例化视图，而不会使应用程序代码中的SQL无效。

以下语句基于SH示例模式中的三个主表创建并填充实例化聚合视图：

CREATE MATERIALIZED VIEW sales_mv AS SELECT t.calendar_year，p.prod_id，SUM（s.amount_sold）AS sum_sales FROM times t，products p，sales s WHERE t.time_id = s.time_id AND p.prod_id = s.prod_id GROUP BY t .calendar_year，p.prod_id;

分区

分区是数据库将非常大的表，索引或物化视图物理地分解为可以独立管理的较小块的能力。分区类似于并行处理，它将大型过程分解为可以独立处理的较小块。

每个分区都是一个独立的对象，它有自己的名称，也可以选择自己的存储特性。分区对许多不同类型的数据库应用程序很有用，特别是那些管理大量数据的应用程序。优点包括提高可用性，更轻松地管理模式对象，减少OLTP系统中共享资源的争用，以及增强数据仓库中的查询性能。

要对表进行PARTITION ，请在CREATE TABLE语句中指定PARTITION BY子句。SELECT和DML语句不需要特殊的语法来从分区中受益。

常见的策略是按日期范围划分记录。以下语句创建了四个分区，一个用于四年销售数据（2008年至2011年）中的每一个的记录：

时间有效性支持

时间有效性支持允许您将一个或多个有效时间维度与表关联，并使数据可见，具体取决于其基于时间的有效性，由给定记录被视为有效的期间的开始和结束日期或时间戳确定。基于时间的有效性的示例是人力资源应用程序中员工的雇用和终止日期，保险单的保险生效日期以及客户或客户的地址变更生效日期。

对于在AS OF和VERSIONS BETWEEN子句中指定有效时间段的查询，时间有效性支持通常与Oracle闪回技术一起使用。您还可以使用DBMS_FLASHBACK_ARCHIVE.ENABLE_AT_VALID_TIME过程为表数据的可见性指定选项：所有表数据（默认值），在指定时间有效的数据，或会话级有效时间段内的当前有效数据。

时间有效性支持在信息生命周期管理（ILM）和任何其他应用程序中非常有用，在这些应用程序中，重要的是要知道某些数据何时变为有效（从应用程序的角度来看）以及何时变为无效（如果有的话）。

注意：

使用时间有效性支持创建和使用具有有效时间支持和更改数据的表假定用户具有创建表并对其执行数据操作语言（DML）和（数据定义语言）DDL操作的权限。

3.3 SQL多媒体类型的容差值

由于SQL多媒体标准未定义如何将容差与ST_ xxx一起使用，因此Spatial和Graph在ST_GEOMETRY类型的所有成员方法中使用默认值0.005。

如果要指定与ST_GEOMETRY成员函数一起使用的其他容差值，请通过将所需值插入SDO_ST_TOLERANCE表来覆盖缺省值。

SDO_ST_TOLERANCE表是一个全局临时表，应该有一行指定要与ST_GEOMETRY成员方法一起使用的容差。此表有一列，定义为(tolerance NUMBER) 。

对于使用空间索引的所有空间运算符，空间和图形使用在USER_SDO_GEOM_METADATA视图中为空间列指定的容差值。

3.4避免名称冲突

为避免第三方名称与Oracle提供的名称之间可能发生冲突，Oracle上任何第三方ST_ xxx类型的实现都应使用模式前缀。

一些第三方供应商在Oracle上支持他们自己的ST_ xxx类型。例如，供应商可能会创建自己的ST_GEOMETRY类型版本。

为避免可能的冲突，该供应商在Oracle上实现ST_ xxx类型应使用模式前缀。这将确保如果有人将列类型指定为ST_GEOMETRY，则将使用ST_GEOMETRY类型的Oracle实现创建该列。

174.1 DBMS_TSDP_MANAGE概述

DBMS_TSDP_MANAGE包使您可以管理Oracle数据库中的敏感列和敏感类型。

已识别的敏感列根据敏感类型进行分类。通过使用DBMS_TSDP_PROTECT程序包创建一个策略，该策略基于列类型而不是数据本身来保护给定类的数据，然后您可以统一管理这些类型的安全性并修改设置以适应不断变化的合规性规则。

使用Data Pump执行完全导出时，也可以将策略导出到其他数据库。您无法导出策略本身，但数据库的导出将包括TSDP策略。

174.2 DBMS_TSDP_MANAGE安全模型

所有程序都以调用者的权利执行。DBMS_TSDP_MANAGE包由SYS拥有。

应酌情授予此程序包的EXECUTE特权。通常，应为应用程序数据库管理员授予此程序包的EXECUTE特权，而DBMS_TSDP_PROTECT程序包将由安全管理员管理。

5.1.3使用DBMS_INMEMORY_ADMIN.POPULATE_WAIT的人口

DBMS_INMEMORY_ADMIN.POPULATE_WAIT函数启动优先级大于或等于指定优先级的所有INMEMORY对象的填充，并返回填充的状态值。用户指定的时间间隔指定函数在将值返回给调用者之前等待的最长时间。

确保填充对象的示例用例包括：

  • 关闭数据库时，使用STARTUP RESTRICT打开数据库，以便只有管理员才能访问数据库，然后使用所需的超时设置执行POPULATE_WAIT 。当函数返回0 ，禁用受限会话，以便非管理用户可以查询数据库。

  • 使用服务或应用程序层技术阻止数据库连接。如果不存在分析索引，并且当应用程序依赖于IM列存储来提供合理的性能时，这些技术可以防止失控查询。

POPULATE_WAIT函数不接受表名作为输入。相反，该函数处理所有 INMEMORY对象，其PRIORITY设置大于或等于指定的priority （默认值为LOW ）。如果priority为NONE ，则该函数处理所有INMEMORY对象。请注意， POPULATE_WAIT不适用于没有优先级设置的外部表。

该函数接受总体百分比作为输入，默认为100 ，以及超时间隔，默认为99999999秒（115.74天）。执行该函数时，数据库会尝试在超时间隔内填充满足指定PRIORITY条件的对象，然后返回一个指示填充状态的值。

下表描述了POPULATE_WAIT的可能返回值。该函数返回值0 ， 1 ， 2 ，和3仅当条件被指定的时间间隔结束之前遇到timeout 。例如，如果timeout为600 ，则仅当在600秒之前发生内存不足错误时，该函数才返回1 。在数据库完成请求的操作之前，超时间隔结束时，函数返回-1 。

25.1 Oracle XML DB资源库版本控制概述

您可以创建和管理存储库资源的不同版本。保留对受版本控制的资源的所有更改的记录。更新版本控制的资源时，更新前的内容将存储为单独的资源版本 - 历史记录的快照。

版本控制功能包括以下内容：

  • 资源的版本控制。

    您可以为Oracle XML DB资源库资源打开或关闭版本控制。

    您可以像访问任何其他资源一样访问版本控制的资源。

  • 访问资源版本。

    要访问特定版本的资源，请使用该版本的资源ID。资源ID可以从资源版本历史记录或从版本控制的资源本身获得。请参阅资源版本和资源ID 。

25.3资源版本和资源ID

资源对象ID或资源ID是资源的唯一，常量，系统生成的标识符。每个资源都有一个资源ID。这包括版本资源 ，它是系统生成的资源，没有任何路径名。资源ID有时称为RESID 。

您可以使用PL / SQL包DBMS_XDB_VERSION将资源置于版本控制之下并管理其不同版本。某些DBMS_XDB_VERSION例程接受版本控制资源的路径名作为参数，并返回相关版本资源的资源ID。

例如，您使用函数DBMS_XDB_VERSION.makeVersioned将资源置于版本控制之下，即将其转换为版本控制的资源。它接受资源的存储库路径作为参数。

当您对其执行各种版本控制操作时，不需要为给定的版本控制资源使用相同的路径名，但您使用的路径名必须全部引用相同的资源。

每当路径名称作为表示版本控制资源的参数传递时，它就是所使用资源的最新版本（即当前版本）。路径名始终代表最新版本。 您可以引用当前版本以外的版本的唯一方法是使用其资源ID。

给定版本的资源ID是常量。请记住，版本本身就是一种资源，资源的资源ID永远不会改变。

每次签入版本控制的资源时，Oracle XML DB都会创建新的版本资源。版本资源是资源（其内容和元数据）的快照以及资源ID。给定版本控制资源的版本资源集合构成先前版本的历史序列， 版本系列或资源的历史。

当您签入具有资源ID R的版本控制资源时，Oracle XML DB会创建一个新资源ID P ，它引用资源的快照（包括内容和元数据），就像上次检出之前一样。 。快照是在签出之前创建的，但关联的版本资源（及其资源ID P ）是在签入时创建的。因此，新资源ID P和它所引用的快照一起表示资源的先前版本而不是当前版本。资源ID R继续引用当前版本。

换句话说，当您签入版本控制的资源时，会创建一个版本资源，表示版本控制资源的先前状态。与任何新资源一样，此新版本资源被分配新资源ID（ P ）。

您可以考虑以编写文件备份副本的方式创建版本资源（签入）：就像您为备份文件指定新名称一样，以便为资源的先前版本快照提供一个新资源ID。当前资源保留原始资源ID，就像您的工作文件保留其原始名称一样。

这意味着当您签入资源时，为了“创建新版本”，真正新的是版本资源（资源ID P和它引用的快照），它代表旧的 （以前的）版本。资源（ R ）的最新版本或最新版本实际上只是当前版本。请记住：新版本资源=旧（先前）版本的资源内容和元数据。

资源ID R指的是版本控制资源在其生命周期内的当前版本，从版本控制到删除之前的那一刻。您始终可以使用其原始资源ID访问资源的最新版本。

当您需要引用以前版本的资源时，必须使用其资源ID来引用它。您不能使用路径名。您可以使用函数DBMS_XDB_VERSION.getPredsByRESID来获取给定资源的先前版本的资源ID。

注意：

如果删除资源，则对资源ID或路径名的任何后续引用都会引发错误（通常为ORA-31001: Invalid resource handle or path name ）。您无法访问已删除的任何版本的版本控制资源。

25.4资源版本和ACL

版本资源是不可变的。它是资源内容和元数据的快照，加上资源ID，快照和ID都是静态的。同样，不能更改版本资源的访问控制列表（ACL）。

您可以修改已签出的版本控制资源的ACL。签入时，修改后的ACL将继续与当前（最新）版本的资源相关联，并且先前版本（即新创建的版本资源）在修改之前与ACL关联。也就是说，先前版本与先前的ACL相关联，并且当前版本与更新的ACL相关联。

请记住，重要的是：

  • 不同版本的资源可以具有与之关联的不同ACL。

  • 签出资源后，您可以修改与当前版本关联的ACL。

  • 签入将ACL在签出之前与新创建的版本资源相关联，即与先前版本的资源相关联。

  • 与给定版本关联的ACL保持不变。

并行执行概念

并行执行使多个CPU和I / O资源的应用能够执行单个SQL语句。

并行执行大大减少了通常与决策支持系统（DSS）和数据仓库相关联的大型数据库上的数据密集型操作的响应时间。您还可以在联机事务处理（OLTP）系统上实现并行执行，以进行批处理或模式维护操作，例如索引创建。

并行执行有时称为并行性。并行性是分解任务的想法，因此，不是一个进程在查询中完成所有工作，而是许多进程同时完成部分工作。这方面的一个例子是当四个流程结合起来计算一年的总销售额时，每个流程处理一年中的一个季度，而不是单独处理所有四个季度的流程。性能的提高可能非常显着。

并行执行改进了以下处理：

  • 查询需要大型表扫描，联接或分区索引扫描

  • 创建大型索引

  • 创建大表，包括物化视图

  • 批量插入，更新，合并和删除

何时实施并行执行

并行执行用于通过利用硬件中的CPU和I / O功能来减少查询的执行时间。

在以下情况下，并行执行比串行执行更好：

  • 该查询引用了一个大型数据集。

  • 并发性低。

  • 经过的时间很重要。

并行执行使许多进程协同工作以执行单个操作，例如SQL查询。并行执行有益于具有以下所有特征的系统：

  • 对称多处理器（SMP），集群或大规模并行系统

  • 足够的I / O带宽

  • 未充分利用或间歇使用的CPU（例如，CPU使用率通常低于30％的系统）

  • 足够的内存来支持额外的内存密集型进程，例如排序，散列和I / O缓冲区

如果您的系统缺少任何这些特性，并行执行可能不会显着提高性能。事实上，并行执行可能会降低过度使用系统或I / O带宽较小的系统的系统性能。

在DSS和数据仓库环境中可以观察到并行执行的好处。OLTP系统还可以在批处理期间和模式维护操作（如创建索引）期间从并行执行中受益。表征OLTP应用程序的平均简单DML或SELECT语句不会受到并行执行的任何好处。

何时不实施并行执行

串行执行与并行执行不同，因为只有一个进程执行单个数据库操作，例如SQL查询。

在以下情况下，串行执行是比并行执行更好的选择

  • 该查询引用一个小数据集。

  • 有很高的并发性。

  • 效率很重要。

并行执行通常不适用于：

  • 典型查询或事务非常短（几秒或更短）的环境。

    这包括大多数在线交易系统。并行执行在这些环境中没有用，因为协调并行执行服务器会产生相关成本;对于短期交易，这种协调的成本可能超过并行性的好处。

  • 大量使用CPU，内存或I / O资源的环境。

    并行执行旨在利用额外的可用硬件资源;如果没有这样的资源，那么并行执行不会产生任何好处，实际上可能对性能有害。

基本硬件要求

并行执行旨在有效地使用多个CPU和磁盘来快速回答查询。

它本质上是I / O密集型的。为了获得最佳性能，硬件配置中的每个组件的大小必须保持相同的吞吐量水平：从计算节点中的CPU和主机总线适配器（HBA）到交换机，再到I / O子系统，包括存储控制器和物理磁盘。如果系统是Oracle Real Application Clusters（Oracle RAC）系统，则还必须适当调整互连大小。最薄弱的环节将限制配置中操作的性能和可伸缩性。

建议在没有Oracle数据库的情况下测量硬件配置可以实现的最大I / O性能。您可以将此度量用作未来系统性能评估的基准。请记住，并行执行不可能实现比底层硬件可以承受的更好的I / O吞吐量。Oracle数据库提供了一个名为Orion的免费校准工具，旨在通过模拟Oracle I / O工作负载来测量系统的I / O性能。并行执行通常执行大型随机I / O.

并行执行SQL语句

每个SQL语句在解析时都会经历优化和并行化过程。

如果确定该语句是并行执行的，那么执行计划中会出现以下步骤：

  1. 用户会话或影子进程承担协调器的角色，通常称为查询协调器（QC）或并行执行（PX）协调器。QC是启动并行SQL语句的会话。

  2. PX协调器获得称为并行执行（PX）服务器的必要数量的进程。PX服务器是代表发起会话并行执行工作的各个进程。

  3. SQL语句作为一系列操作执行，例如全表扫描或ORDER BY子句。如果可能，每个操作并行执行。

  4. 当PX服务器完成执行语句时，PX协调器执行无法并行执行的任何工作部分。例如，具有SUM()操作的并行查询需要添加由每个PX服务器计算的各个小计。

  5. 最后，PX协调器将结果返回给用户。

制片人/消费者模型

并行执行使用生产者/消费者模型。

并行执行计划作为一系列生产者/消费者操作来执行。为后续操作生成数据的并行执行（PX）服务器称为生产者，需要输出其他操作的PX服务器称为消费者。每个生产者或消费者并行操作由称为PX服务器集的一组PX服务器执行。PX服务器集中的PX服务器数称为并行度（DOP）。PX服务器集的基本工作单元称为数据流操作（DFO）。

PX协调器可以具有多级生产者/消费者操作（多个DFO），但PX协调器的PX服务器集数量限制为两个。因此，在一个时间点，PX协调器只能有两个PX服务器组处于活动状态。结果，DFO中的操作和DFO之间存在并行性。单个DFO的并行性称为操作内并行性，DFO之间的并行性称为操作间并行性。为了说明操作内和操作间并行性，请考虑以下语句：

SELECT * FROM employees ORDER BY last_name;

执行计划实现了employees表的完整扫描。此操作之后是基于last_name列的值对检索到的行进行排序。为了这个例子，假设last_name列没有被索引。还假设查询的DOP设置为4 ，这意味着对于任何给定的操作，四个并行执行服务器可以是活动的。

平行的颗粒

并行性的基本工作单元称为粒子。

Oracle数据库将并行执行的操作（例如表扫描或索引创建）划分为粒度。并行执行（PX）服务器一次执行一个granule操作。颗粒的数量及其尺寸与平行度（DOP）相关。颗粒数量也会影响PX服务器之间的工作平衡。

块范围颗粒

块范围颗粒是大多数并行操作的基本单元，即使在分区表上也是如此。从Oracle数据库的角度来看，并行度与分区数无关。

块范围颗粒是表中的物理块范围。Oracle数据库在运行时计算粒度的数量和大小，以优化和平衡所有受影响的并行执行（PX）服务器的工作分配。颗粒的数量和大小取决于物体的大小和DOP。块范围颗粒不依赖于表或索引的静态预分配。在计算粒度期间，Oracle数据库会考虑DOP并尝试将来自不同数据文件的粒度分配给每个PX服务器，以尽可能避免争用。此外，Oracle数据库考虑了粒度在大规模并行处理（MPP）系统上的磁盘亲和力，以利用PX服务器和磁盘之间的物理距离。

分配颗粒

使用分区分块时，并行执行（PX）服务器可以在表或索引的整个分区或子分区上运行。

由于在创建表或索引时，分区粒度是由表或索引的结构静态确定的，因此分区粒度不能让您灵活地执行块粒度操作的并行操作。最大允许并行度（DOP）是分区数。这可能会限制系统的利用率和PX服务器之间的负载平衡。

当分区粒度用于并行访问表或索引时，应使用相对大量的分区，理想情况下是DOP的三倍，以便Oracle数据库可以有效地平衡PX服务器之间的工作。

分区粒度是并行索引范围扫描的基本单元，两个均分表之间的连接，其中查询优化器选择使用分区连接，以及修改分区对象的多个分区的并行操作。这些操作包括并行创建分区索引以及并行创建分区表。

生产者和消费者之间的分配方法

分发方法是将数据从一个并行执行（PX）服务器集发送（或重新分发）到另一个的方法。

以下是并行执行中最常用的分发方法。

  • 哈希分布

    散列分配方法在行中的一个或多个列上使用散列函数，然后确定生产者应该将行发送到哪个消费者。该分发试图基于散列值在消费者之间平均分配工作。

  • 广播发布

    在广播分发方法中，每个生产者将所有行发送给所有消费者。当连接操作中左侧的结果集很小并且广播所有行的成本不高时，使用该方法。在这种情况下，不需要分配连接右侧的结果集;分配给连接操作的使用者PX服务器可以扫描右侧并执行连接。

  • 范围分布

    范围分布主要用于并行排序操作。在此方法中，每个生产者将具有一系列值的行发送给同一个使用者。这是图8-1中使用的方法。

  • 混合哈希分布

    混合散列是一种在连接操作中使用的自适应分配方法。实际的分配方法由优化器在运行时决定，具体取决于连接左侧结果集的大小。计算从左侧返回的行数并根据阈值进行检查。当行数小于或等于阈值时，广播分布用于连接的左侧，而右侧不分发，因为分配给连接操作的相同的消费者PX服务器扫描右侧和执行连接。当从左侧返回的行数高于阈值时，散列分布用于连接的两侧。

并行执行服务器如何通信

为了并行执行查询，Oracle数据库通常会创建一组生产者并行执行服务器和一组使用者并行执行服务器。

生产者服务器从表中检索行，并且使用者服务器在这些行上执行诸如连接，排序，DML和DDL之类的操作。生产者集中的每个服务器都与消费者集中的每个服务器建立了连接。并行执行服务器之间的虚拟连接数随着并行度的平方而增加。

每个通信信道至少有一个，有时最多四个内存缓冲区，它们是从共享池中分配的。多个内存缓冲区便于并行执行服务器之间的异步通信。

单实例环境对每个通信通道最多使用三个缓冲区。Oracle Real Application Clusters环境最多为每个通道使用四个缓冲区。图8-3说明了消息缓冲区以及生产者并行执行服务器如何连接到客户并行执行服务器。

当连接位于同一实例上的两个进程之间时，服务器通过在内存中（在共享池中）来回传递缓冲区进行通信。当连接在不同实例中的进程之间时，将通过互连使用外部高速网络协议发送消息。在图8-3中 ，DOP等于并行执行服务器的数量，在本例中为n。图8-3未显示并行执行协调器。每个并行执行服务器实际上都有一个到并行执行协调器的附加连接。在使用并行执行时，充分调整共享池的大小非常重要。如果共享池中没有足够的可用空间来为并行服务器分配必要的内存缓冲区，则无法启动。

并行执行服务器池

实例启动时，Oracle数据库会创建一个并行执行服务器池，可用于任何并行操作。

初始化参数PARALLEL_MIN_SERVERS指定Oracle数据库在实例启动时创建的并行执行服务器的数量。

当执行并行操作时，并行执行协调器从池中获取并行执行服务器并将它们分配给操作。如有必要，Oracle数据库可以为该操作创建其他并行执行服务器。这些并行执行服务器在整个执行期间保持运行。处理完该语句后，并行执行服务器将返回池。

如果并行操作的数量增加，Oracle数据库将创建其他并行执行服务器来处理传入请求。但是，Oracle数据库从不为实例创建比初始化参数PARALLEL_MAX_SERVERS指定的值更多的并行执行服务器。

如果并行操作数减少，Oracle数据库将终止任何空闲阈值间隔的并行执行服务器。无论并行执行服务器空闲多长时间，Oracle数据库都不会将池的大小减小到小于PARALLEL_MIN_SERVERS的值。

没有足够的并行执行服务器的处理

Oracle数据库可以处理少于请求的进程数的并行操作。

如果池中的所有并行执行服务器都已被占用并且已启动并行执行服务器的最大数量，则并行执行协调器将切换到串行处理。

平衡工作负载以优化性能

为了优化性能，所有并行执行服务器应具有相同的工作负载。

对于按块范围或并行执行服务器并行运行的SQL语句，工作负载在并行执行服务器之间动态分配。这可以最大限度地减少工作负载偏差，这种情况发生在某些并行执行服务器执行的工作量远远超过其他进

对于由分区并行执行的相对较少的SQL语句，如果工作负载均匀分布在分区中，则可以通过将并行执行服务器的数量与分区数相匹配或通过选择分区数量的DOP来优化性能是进程数的倍数。这适用于在Oracle9 i之前创建的表上的分区连接和并行DML。有关更多信息，请参阅并行度限制 。

例如，假设一个表有16个分区，并行操作在它们之间平均分配工作。您可以使用16个并行执行服务器（DOP等于16）来完成工作，大约是一个进程所需的时间的十分之一。您可能还会在五分之一的时间内使用五个流程来完成工作，或者在一半的时间内使用两个流程来完成工作。

但是，如果您使用15个进程来处理16个分区，那么在一个分区上完成其工作的第一个进程将开始在第16个分区上工作;随着其他流程完成工作，他们就会闲着。当工作在分区之间平均分配时，此配置无法提供良好的性能。当工作分工不均时，性能会根据最后留下的分区是否比其他分区更多或更少而有所不同。

同样，假设您使用六个进程处理16个分区，并且工作均分。在这种情况下，每个进程在完成其第一个分区后在第二个分区上工作，但只有四个进程在第三个分区上工作，而其他两个进程保持空闲。

一般情况下，您不能假设在给定数量的并行执行服务器（P）上对给定数量的分区（N）执行并行操作所花费的时间等于N除以P.此公式不考虑某些可能性进程可能必须等待其他人完成最后一个分区的工作。但是，通过选择适当的DOP，可以最大限度地减少工作负载偏差并优化性能。

多个并行化程序

执行计划中的每个并行执行（PX）协调器称为并行程序。

SQL语句使用的PX服务器数由语句并行度（DOP）和并行程序数决定。由于并行程序的PX服务器集数量限制为2，因此大多数语句的PX服务器数量为DOP * 2。一些语句可以有多个并行化程序。由于每个并行程序可以使用两个PX服务器集，因此这些语句的PX服务器数量可能超过DOP * 2。您可以通过查看EXPLAIN PLAN来识别这些语句。如果计划有多个PX协调器，则表示该语句具有多个并行程序。

SQL语句使用多个并行化程序的一些示例情况是子查询因子分组，分组集，星型查询，内存中聚合和非相关子查询。

SQL语句中的多个并行程序可以同时处于活动状态，也可以依次处于活动状态，具体取决于执行计划。

具有单个并行程序的语句在执行开始时分配所需数量的PX服务器，并保留这些已分配的PX服务器，直到语句完成为止。这可确保整个执行过程中的PX服务器数量不变。具有多个并行程序的语句不同，因为它们在每个并行程序启动时分配PX服务器。由于并行程序可以在执行期间的不同时间启动，因此每个并行程序可以基于系统中的可用进程数运行不同数量的PX服务器。

如果同时执行多个并行程序，则该语句可以使用比DOP * 2更多的PX服务器。

视图V$PQ_SESSTAT显示STATISTIC列中的并行程序数。数据流操作统计信息DFO Trees显示并行化程序的数量。“ Server Threads统计信息显示SQL语句并发使用的最大PX服务器数。

Oracle RAC上的并行执行

默认情况下，在Oracle RAC环境中，并行执行的SQL语句可以在群集中的所有节点上运行。

要执行此跨节点或节点间并行执行，必须适当调整Oracle RAC环境中的互连，因为节点间并行执行可能会导致大量互连流量。节点间并行执行不会与尺寸较小的互连一起扩展。

限制可用实例数

在Oracle RAC环境中，您可以使用服务来限制参与并行SQL语句执行的实例数。默认服务包括所有可用实例。您可以创建任意数量的服务，每个服务由一个或多个实例组成。当用户使用服务连接到数据库时，只有作为服务成员的实例上的PX服务器才能参与并行语句的执行。

要将并行执行限制为单个节点，可以将PARALLEL_FORCE_LOCAL初始化参数设置为TRUE 。在这种情况下，只有会话连接到的实例上的PX服务器用于执行该会话的并行语句。请注意，当此参数设置为TRUE ，在该实例上运行的所有并行语句都在本地执行，无论会话是直接连接到实例还是使用服务连接。

Flex集群上的并行执行

在flex集群上执行的并行语句可以使用hub和leaf节点。由于用户会话仅允许连接到中心节点，因此协调器进程（查询协调器或PX协调器）驻留在中心节点上，并且可以使用来自群集中任何节点的PX服务器进程。对于并行查询，任何节点上的任何PX服务器都可以参与语句的执行。对于并行DML操作，只有中心节点上的PX服务器可以参与语句的DML部分的执行，因为只允许中心节点执行DML操作。

123 DBMS_PREDICTIVE_ANALYTICS

数据挖掘可以发现埋藏在大量数据中的有用信息。但是，通常情况下，获取这些结果所需的编程接口和数据挖掘专业知识过于复杂，无法被广泛的受众使用，这些受众可以从使用Oracle数据挖掘中获益。

DBMS_PREDICTIVE_ANALYTICS包通过自动化整个数据挖掘过程（从数据预处理到模型构建到评分新数据）来解决这两个复杂问题。该软件包提供了一个重要的工具，使广泛的用户，特别是业务分析师可以进行数据挖掘。

本章包含以下主题：

  • 概观

  • 安全模型

  • DBMS_PREDICTIVE_ANALYTICS子程序摘要

123.1 DBMS_PREDICTIVE_ANALYTICS概述

DBMS_PREDICTIVE_ANALYTICS自动化部分数据挖掘过程。

根据常用的流程模型，数据挖掘需要以下步骤：

  1. 了解业务问题。

  2. 理解数据。

  3. 准备挖掘数据。

  4. 使用准备好的数据创建模型。

  5. 评估模型。

  6. 部署并使用该模型对新数据进行评分。

DBMS_PREDICTIVE_ANALYTICS自动执行此过程的步骤3 - 5的部分内容。

预测分析程序分析和准备输入数据，使用输入数据创建和测试挖掘模型，然后使用输入数据进行评分。评分结果将返回给用户。操作完成后，不会保留模型和支持对象。

123.2 DBMS_PREDICTIVE_ANALYTICS安全模型

DBMS_PREDICTIVE_ANALYTICS包由用户SYS拥有，并作为数据库安装的一部分安装。包的执行权限授予公开。程序包中的例程使用调用者的权限运行（使用当前用户的权限运行）。

DBMS_PREDICTIVE_ANALYTICS包公开了Oracle数据挖掘选项所利用的API。希望调用此程序包中的过程的用户需要CREATE MINING MODEL系统特权（以及CREATE TABLE和CREATE VIEW系统特权）。
