9.1文档分类概述

每个主题都是单个单词，单个短语或父主题的分层列表。

要筛选大量文档，您可以使用关键字搜索引擎。但是，关键字搜索有局限性。一个主要缺点是关键字搜索不能通过上下文区分。在许多语言中，单词或短语可能具有多种含义，因此搜索可能会导致许多与特定主题无关的匹配。例如，关于短语河岸的查询可能会返回有关哈德逊河岸和信托公司的文件，因为单词bank有两个含义。

或者，您可以对文档进行排序并按内容对其进行分类。这种方法对于大量文档是不可行的。

Oracle Text提供了各种文档分类方法。在基于规则的分类 （有时称为简单分类 ）下，您自己编写分类规则。通过监督分类 ，Oracle Text基于您预分类的一组示例文档创建分类规则。最后，通过无监督分类 （也称为聚类 ），Oracle Text将执行从编写分类规则到为文档分类的所有步骤。

9.3分类解决方案

Oracle Text使您可以通过以下方式对文档进行分类：

  • 基于规则的分类 。对于此解决方案，您可以对文档进行分组，选择类别，并制定定义这些类别的规则;这些规则实际上是查询短语。然后，您可以索引规则并使用MATCHES运算符对文档进行分类。

    优点：此解决方案对于小型文档集非常准确。结果总是基于您定义的内容，因为您编写规则。

    缺点：对于具有许多类别的大型文档集，定义规则可能很繁琐。随着文档集的增长，您可能需要编写相应的更多规则。

  • 监督分类 。此解决方案类似于基于规则的分类，但使用CTX_CLS.TRAIN.自动执行规则编写步骤CTX_CLS.TRAIN.此过程根据您提供的一组预先分类的文档制定一组分类规则。与基于规则的分类一样，您可以使用MATCHES运算符对文档进行分类。

    Oracle Text提供两种版本的监督分类，一种使用RULE_CLASSIFIER首选项，另一种使用SVM_CLASSIFIER首选项。这些偏好在“ 监督分类 ”中讨论。

    优点 ：自动为您编写规则。此方法对大型文档集很有用。

    缺点：在生成规则之前，必须将文档分配给类别。规则可能不像您自己编写的那样具体或准确。

  • 无监督分类（聚类） 。从分组文档到编写类别规则的所有步骤都通过CTX_CLS.CLUSTERING.自动CTX_CLS.CLUSTERING. Oracle Text统计分析您的文档集，并根据内容将它们与集群相关联。

    好处：

    • 您无需提供分类规则或示例文档作为训练集。

    • 此解决方案有助于发现文档集中忽略的模式和内容相似性。

      实际上，当您对规则或分类没有明确的概念时，可以使用此解决方案。例如，使用它来提供一组初始类别，并通过监督分类建立类别。

    缺点：

    • 群集基于内部解决方案。它可能会导致意外分组，因为群集操作不是用户定义的。

    • 您没有看到创建群集的规则。

    • 群集操作是CPU密集型的，并且至少可以与索引相同。

9.5.1决策树监督分类

要使用决策树分类，请将CTX_CLS.TRAIN的首选参数设置为RULE_CLASSIFIER.

这种分类形式使用决策树算法来创建规则。一般而言，决策树是一种决定两种（或更多，但通常是两种）选择的方法。在文档分类中，选项是“文档与训练集匹配”或“文档与训练集不匹配”。

决策树具有一组可以测试的属性。在这种情况下，属性包括：

  • 文件中的文字

  • 来自文档的单词的词干（例如， 运行的词干运行 ）

  • 文档中的主题（如果使用的语言支持主题）

Oracle Text中的学习算法为训练集中提供的每个类别构建一个或多个决策树。然后将这些决策树编码成适合CTXRULE索引使用的查询。例如，一个类别具有“日本甲虫”的培训文档，另一个类别具有“日本货币”的文档。该算法可以基于“日语”，“甲虫”和“货币”创建决策树，然后相应地对文档进行分类。

决策树包括置信度的概念。 在给定当前训练集的情况下，为每个生成的规则分配一个百分比值，该百分比值表示规则的准确性。在简单的例子中，准确度几乎总是100％，但这个百分比仅代表训练集的局限性。同样，从普通训练集生成的规则可能看起来比您预期的要少，但它们足以区分当前训练集中的不同类别。

决策树分类的优点是它可以生成用户可以轻松检查和修改的规则。当您希望计算机生成大量规则时，决策树分类是有意义的，但您希望之后通过编辑规则集对它们进行微调。

9.5.3基于SVM的监督分类

您可以用于培训目的的第二种方法是支持向量机（SVM）分类。SVM是一种源自统计学习理论的机器学习算法。SVM分类的一个属性是从非常小的样本集中学习的能力。

使用SVM分类器与使用决策树分类器非常相似，但以下区别除外：

  • 在对CTX_CLS.TRAIN,的调用中CTX_CLS.TRAIN,使用SVM_CLASSIFIER首选项而不是RULE_CLASSIFIER首选项。（如果您不想修改任何属性，请使用预定义的CTXSYS.SVM_CLASSIFIER首选项。）

  • 如果您不想在表上填充CONTEXT索引，请使用NOPOPULATE关键字。分类器仅使用它来通过数据存储区和过滤器首选项查找文本的来源，并确定如何通过词法分析器和分段器首选项处理文本。

  • 在生成的规则表中，至少使用以下列：

    cat_id number，type number，rule blob;

如您所见，生成的规则将写入BLOB列。因此，它对用户不透明，与决策树分类规则不同，它无法编辑或修改。这里的权衡是，使用SVM比使用决策树分类通常可以获得更高的准确性。

通过SVM分类，分配的内存必须足够大才能加载SVM模型;否则，在SVM上构建的应用程序会导致内存不足错误。以下是如何计算内存分配：

最小内存请求（以字节为单位） = 唯一类别 数 x 特征数量示例:( MAX_FEATURES属性值 ）x 8

如有必要满足最低内存要求，请增加以下内存之一：

  • SGA（如果处于共享服务器模式）

  • PGA（如果在专用服务器模式下）

例如，假设您有大量关于动物的文档集合。CTX_CLS.CLUSTERING创建了一个关于狗的叶簇，另一个关于猫，另一个关于鱼，第四个关于熊。（前三个可能在一个关于宠物的节点集群下分组。）进一步假设你有一个关于一种狗的文件，如吉娃娃。CTX_CLS.CLUSTERING将CTX_CLS.CLUSTERING分配给具有非常高的相关性得分的文档，而猫群被分配较低的分数，并且鱼和熊群仍然被分配较低的分数。在将所有群集的分数分配给所有文档之后，应用程序可以基于分数采取行动。

如“ 决策树监督分类 ”中所述 ，用于确定聚类的属性可以包括简单的单词（或标记），词干和主题（在支持的情况下）。

  • 文档分配表，显示文档与每个叶簇的相似性。该信息采用文档识别，群集识别以及文档与群集之间的相似性得分的形式。

  • 包含有关生成的集群的信息的集群描述表。此表包含群集标识，群集描述文本，建议的群集标签以及群集的质量分数。

171.5 DBMS_TRACE操作说明

某些操作说明适用于DBMS_TRACE 。

这些在以下部分中描述：

  • 控制数据量

  • 创建数据库表以收集DBMS_TRACE输出

  • 收集跟踪数据

  • 收集的数据

控制数据量

分析大型应用程序可能会产生大量数据。您可以通过启用特定程序单元来跟踪数据收集来控制收集的数据量。

您可以通过编译它来启用程序单元。这可以通过以下两种方式之一完成：

alter session set plsql_debug = true;创建或替换... / *创建库单元 - 将生成调试信息* /

要么：

/ *使用调试选项重新编译特定的库单元* / alter [PROCEDURE |功能| PACKAGE BODY] <libunit-name>编译调试;

注意：

您不能将第二种方法用于匿名块。

您可以通过在SET_PLSQL_TRACE过程的TRACE_LEVEL参数中包含TRACE_LIMIT来仅保留最近的8,192条记录（大约）来限制数据库中使用的存储量。

创建数据库表以收集DBMS_TRACE输出

你必须创建DBMS_TRACE包写入输出的数据库表。否则，不会收集数据。要创建这些表，请运行脚本TRACETAB.SQL 。此脚本创建的表由SYS拥有。

收集跟踪数据

您可以跟踪的PL / SQL功能在脚本DBMSPBT.SQL中描述。一些关键的跟踪功能是：

  • 追踪电话

  • 跟踪异常

  • 跟踪SQL

  • 追踪线

DBMS_TRACE的其他功能还允许暂停和恢复跟踪，并限制输出。

追踪电话

有两种级别的呼叫跟踪：

  • 级别1：跟踪所有呼叫。这对应于常量TRACE_ALL_CALLS 。

  • 级别2：仅对已启用的程序单元进行跟踪调用。这对应于常量TRACE_ENABLED_CALLS 。

无法检测远程过程调用（RPC）的启用情况;因此，RPC仅跟踪级别1。

跟踪异常

可以使用两级异常跟踪：

  • 第1级：跟踪所有异常。这对应于TRACE_ALL_EXCEPTIONS 。

  • 级别2：仅在启用的程序单元中引发的跟踪异常。这对应于TRACE_ENABLED_EXCEPTIONS 。

跟踪SQL

有两种级别的SQL跟踪：

  • 级别1：跟踪所有SQL。这对应于常量TRACE_ALL_SQL 。

  • 级别2：仅在启用的程序单元中跟踪SQL。这对应于常量TRACE_ENABLED_SQL 。

追踪线

有两种级别的线追踪：

  • 第1级：跟踪所有行。这对应于常量TRACE_ALL_LINES 。

  • 级别2：仅在启用的程序单元中跟踪行。这对应于常量TRACE_ENABLED_LINES 。

在跟踪行时，Oracle会在每次行号更改时向数据库添加记录。这包括由于过程调用和返回而导致的行号更改。

注意：

对于所有类型的跟踪，级别1将覆盖级别2。例如，如果启用了级别1和级别2，则级别1优先。

收集的数据

如果仅对启用的程序单元请求跟踪，并且如果未启用当前程序单元，则不会写入跟踪数据。

跟踪呼叫时，将跟踪呼叫和返回。如果被调用的例程或调用例程被“启用”，则检查是否“已启用”跟踪。

呼叫跟踪将始终输出呼叫者和被呼叫者的程序单元类型，程序单元名称和行号。它将输出调用者的堆栈深度。如果启用了呼叫者，则还将输出呼叫者的姓名。如果启用了被调用者，则还将输出被调用者的名称。

异常跟踪会写出行号。引发异常会显示有关异常是用户定义还是预定义的信息。它还显示了预定义异常情况下的异常编号。引发异常的位置和跟踪它们的处理程序。“启用”跟踪检查是针对引发异常的位置和处理异常的位置独立完成的。启用NO_TRACE_HANDLED_EXCEPTIONS会将数据收集限制为未处理的异常

对DBMS_TRACE.SET_PLSQL_TRACE和DBMS_TRACE.CLEAR_PLSQL_TRACE所有调用DBMS_TRACE.SET_PLSQL_TRACE在数据库中放置一条特殊的跟踪记录。因此，始终可以确定何时更改跟踪设置。

跟踪控制

除了确定收集哪些项目之外，您还可以暂停和恢复跟踪过程。在跟踪暂停的时间和恢复时间之间不收集任何信息。常量TRACE_PAUSE和TRACE_RESUME用于实现此目的。生成跟踪记录以指示跟踪已暂停/恢复。

通过使用常量TRACE_LIMIT也可以仅保留运行的最后8,192个跟踪事件。这允许在不填满数据库的情况下打开跟踪。当跟踪停止时，将保存最后的8,192条记录。限制是近似值，因为未在每个跟踪记录中检查。至少会生成所请求的跟踪记录数;最多可生成1,000条记录。至少会生成所请求的跟踪记录数;最多可生成1,000条记录。可以更改8,192记录限制。将事件10940设置为级别n会将记录限制更改为1024 * n 。

启用NO_TRACE_ADMINISTRATIVE可防止生成PL/SQL Trace Tool started ， Trace flags changed ， PL/SQL Virtual Machine started以及PL/SQL Virtual Machine stopped等此类管理事件记录。

使用SNA安全验证

当网关收到启动远程事务程序（RTP）的远程过程调用（RPC）请求时，网关尝试启动与OLTP的APPC对话。在开始对话之前，必须在Windows逻辑单元之间启动会话 （ LU）和OLTP LU。

SNA及其各种访问方法实现（包括HIS和IBM Communication Server和VTAM）在会话启动时提供安全验证，允许每个LU对其伙伴进行身份验证。在网关和OLTP应用程序开始对话并处理对话级安全数据之前，此验证完全由网络软件执行。如果使用会话级安全性，则必须在Windows SNA Server定义中以及要访问的OLTP中的类似参数结构中建立正确的密码信息。有关此主题的详细信息，请参阅合适的通信软件产品文档。

独立对抗的LU

Oracle建议为Oracle数据库网关APPC提供独立的LU，因为它们支持多个并行会话或会话。这意味着多个Oracle客户端应用程序可以通过独立LU与同一OLTP同时处于活动状态。

从属LU仅支持单个活动会话。CP（节点的控制点，在这种情况下是Windows的SNA Server）在已经活动的对话后面从网关服务器排队其他对话请求。换句话说，对话LU是单线程的。

如果正确定义了从属LU，则不需要对Oracle数据库网关进行APPC配置的更改，也不需要对主机事务或OLTP如何启动进行任何更改。

依赖LU的操作影响是第一个客户端应用程序可以通过数据库网关与OLTP发起对话。当该事务处于活动状态时（可能是几秒到几分钟到几小时，具体取决于客户端应用程序和事务的设计方式），启动与同一OLTP实例的对话的任何其他客户端应用程序在等待前一个对话后都会挂起。

如果生产应用程序在任何时候确实只使用一个会话或事务，那么应该没有任何影响。

但是，测试或其他应用程序开发可能需要额外的并发对话或事务。每个都要求在远程主机上定义其他从属LU，以及其他SNA Server配置条目，这些条目定义Windows系统上的其他从属LU。该启动对话的提示必须通过不同的侧面信息配置文件或通过覆盖LU名称来指定不同的合作伙伴LU。请参阅Oracle Database Gateway for APPC用户指南的第2章“过程网关管理实用程序”中的PGAU DEFINE TRANSACTION ， SIDEPROFILE和LUNAME参数。

配置IBM Communications Server

如果您使用IBM Communications Server作为SNA Server，请阅读以下部分以了解如何配置它以与Oracle Database Gateway for APPC进行通信。

独立和依赖的LU

Oracle建议为Oracle数据库网关APPC提供独立的LU，因为它们支持多个并行会话或会话。这意味着多个Oracle客户端应用程序可以通过独立LU与同一OLTP同时处于活动状态。

从属LU仅支持单个活动会话。CP（节点的控制点，在这种情况下是IBM Communications Server）将来自网关服务器的其他会话请求排队到已经活动的会话之后。换句话说，对话LU是单线程的。

如果正确定义了网关LU，则不需要对Oracle Database Gateway进行APPC配置的更改，也不需要对网关服务器进行任何更改。

依赖LU的操作影响是第一个客户端应用程序可以通过数据库网关与网关服务器发起对话。当该事务处于活动状态时（可能是几秒到几分钟到几小时，具体取决于客户端应用程序和事务的设计方式），启动与同一网关服务器的会话的任何其他客户端应用程序在等待上一个会话后都会挂起。

如果生产应用程序在任何时候确实只使用一个对话，那么应该没有任何影响。

但是，测试或其他应用程序开发可能需要额外的并发对话。每个都要求在远程主机上定义其他从属LU，以及其他IBM Communication Server配置条目，这些条目定义主机Windows系统上的其他从属LU。

应定义附加的辅助信息配置文件以使用新的从属LU。应创建新网关实例并将其配置为使用这些新的辅助信息配置文件。

请参阅Oracle Database Gateway for APPC用户指南的第2章“过程网关管理实用程序”中的PGAU DEFINE TRANSACTION，SIDEPROFILE和LUNAME参数。

4.1创建和映射元数据概述

OLAP Java API提供了创建持久元数据对象的功能。顶级元数据对象存在于Oracle数据库实例的数据字典中。API还提供了创建仅在会话期间存在的临时元数据对象的功能。应用程序可以使用两种类型的元数据对象来创建检索或以其他方式使用数据存储中的数据的查询。

在OLAP Java API应用程序可以创建元数据对象之前，数据库管理员必须已准备好Oracle数据库实例。DBA必须在数据库中设置永久和临时表空间以支持Oracle OLAP元数据对象的创建，并且必须已授予允许会话用户创建和管理对象的权限。维度元数据模型通常包括“ 了解OLAP Java API元数据”中描述的对象。

您可以通过创建OLAP Java API元数据对象来实现维模型。您可以使用oracle.olapi.metadata.mapping包中的类将元数据对象映射到关系源对象并构建分析工作区。您可以使用oracle.olapi.syntax包中的类来指定在映射元数据时使用的Expression对象。您可以使用oracle.olapi.metadata.deployment包中的类在分析工作区或关系数据库（ROLAP）组织中部署元数据对象。

在分析工作区中将维模型实现为OLAP Java API对象的基本步骤如下：

  1. 创建一个AW对象和MdmPrimaryDimension和MdmCube对象。

  2. 将MdmPrimaryDimension和MdmCube对象部署到AW 。

  6. 通过构建分析工作区将数据从关系源加载到对象中。

关于导出和导入加密数据

您可以使用Oracle Data Pump导出和导入具有加密列的表。

对于软件和硬件密钥库，必须导出包含加密列的表时，以下几点非常重要：

  • 在运输过程中，敏感数据应该仍然难以理解。

  • 授权用户应该能够在目标导入数据后解密数据。

使用Oracle Data Pump导出和导入包含加密列的表时，它使用ENCRYPTION参数来启用转储文件集中数据的加密。ENCRYPTION参数允许以下值：

  • ENCRYPTED_COLUMNS_ONLY ：将加密列写入以加密格式设置的转储文件

  • DATA_ONLY ：将所有数据写入以加密格式设置的转储文件

  • METADATA_ONLY ：将所有元数据写入以加密格式设置的转储文件

  • ALL ：将所有数据和元数据写入以加密格式设置的转储文件

  • NONE ：不对转储文件集使用加密

导出和导入带有加密列的表

您可以使用ENCRYPTION=ENCRYPTED_COLUMNS_ONLY设置导出和导入带有加密列的表。

  1. 在尝试导出包含加密列的表之前，请确保密钥库已打开。

    在多租户环境中，如果要在可插拔数据库（PDB）中导出数据，请确保在PDB中打开钱包。如果要导出到根目录，请确保钱包在根目录中打开。

    要查找密钥库是否已打开，请查询V$ENCRYPTION_WALLET视图的STATUS列。如果必须打开密钥库，则运行以下SQL语句：

    ADMINISTER密钥管理设置KEYSTORE打开由software_keystore_password识别[CONTAINER = ALL |当前];

    software_keystore_password设置是密钥库的密码。密钥库必须打开，因为必须使用TDE表密钥对加密列进行解密，这需要访问TDE主加密密钥。在导出列之前，使用密码重新加密列。

  2. 运行EXPDP命令，使用ENCRYPTION_PASSWORD参数指定用于加密导出转储文件集中的列数据的密码。

    以下示例导出employee_data表。使用ENCRYPTION_PWD_PROMPT = YES设置可以交互式提示输入密码，这是一种推荐的安全措施。 

    expdp hr TABLES = employee_data DIRECTORY = dpump_dir DUMPFILE = dpcd2be1.dmp ENCRYPTION = ENCRYPTED_COLUMNS_ONLY ENCRYPTION_PWD_PROMPT = YES密码： password_for_hr

  3. 要将导出的数据导入目标数据库，请确保指定用于导出操作的密码，该密码由ENCRYPTION_PASSWORD参数设置。

    密码用于解密数据。使用目标数据库中生成的新TDE表密钥重新加密数据。目标数据库必须打开密钥库才能访问TDE主加密密钥。以下示例导入employee_data表：

    impdp hr TABLES = employee_data DIRECTORY = dpump_dir DUMPFILE = dpcd2be1.dmp ENCRYPTION_PWD_PROMPT = YES密码： password_for_hr

使用Oracle Data Pump加密整个转储集

Oracle Data Pump可以加密整个转储集，而不仅仅是透明数据加密列。

导入时，您可以使用密码或密钥库TDE主加密密钥来解密数据。如果未提供密码，则密钥库中的TDE主加密密钥用于解密数据。密钥库必须存在并在目标数据库中打开。open keystore还需要重新加密目标数据库中的列加密数据。

您可以使用ENCRYPTION_MODE=TRANSPARENT设置透明地加密存储在密钥库中的TDE主加密密钥的转储文件集。在这种情况下，不需要密码。密钥库必须存在并在目标数据库中打开，并且必须包含 源数据库中的TDE主加密密钥才能在导入操作期间成功解密列加密元数据。

open keystore还需要在目标数据库中重新加密列加密元数据。如果密钥存储在目标数据库中已经存在，那么你可以从源数据库的密钥库导出当前TDE主加密密钥并将其导入到目标数据库的密钥库。

  • 使用ENCRYPTION_MODE参数指定加密模式。ENCRYPTION_MODE=DUAL使用存储在密钥库中的TDE主加密密钥和提供的密码对转储集进行加密。

例如，要使用双加密模式导出加密数据：

expdp hr DIRECTORY = dpump_dir1 DUMPFILE = hr_enc.dmp ENCRYPTION = all ENCRYPTION_PASSWORD = encryption_password ENCRYPTION_PWD_PROMPT = yes ENCRYPTION_ALGORITHM = AES256 ENCRYPTION_MODE = dual密码： password_for_hr加密密码： password_for_encryption

将Oracle Data Pump与加密数据字典数据结合使用

Oracle Data Pump操作为加密密码和其他加密数据提供保护。

在源数据库中启用固定用户数据库密码加密后，Oracle Data Pump导出操作转储将为数据库链接密码存储已知的无效密码。此密码已到位，而不是导出操作从数据库中提取的加密密码。ORA-39395: Warning: object <database link name> requires password reset after import警告消息ORA-39395: Warning: object <database link name> requires password reset after import显示ORA-39395: Warning: object <database link name> requires password reset after import 。如果将数据导入Oracle Database 18c或更高版本的数据库，则在目标数据库中创建具有无效密码的数据库链接对象时，将显示相同的警告。发生这种情况时，您必须重置数据库链接密码，如下所示：

ALTER DATABASE LINK database_link_name CONNECT TO schema_name IDENTIFIED BY password ;

要查找有关数据库链接的信息，可以查询V$DBLINK动态视图。

如果在源数据库中禁用了固定用户数据库密码的加密，则Data Pump不会发生任何更改。与先前版本一样，导出和导入模糊的数据库链接密码。

在这种情况下，Oracle建议如下：

  • 在expdp命令上设置ENCRYPTION_PASSWORD参数，以便进一步保护模糊的数据库链接密码。

  • 将ENCRYPTION_PWD_PROMPT参数设置为YES以便可以从提示中以交互方式输入密码，而不是在屏幕上回显。

ENCRYPTION_PASSWORD和ENCRYPTION_PWD_PROMPT参数在导入操作中都可用。ENCRYPTION_PWD_PROMPT仅适用于expdp和impdp命令行客户端，而ENCRYPTION_PASSWORD在命令行客户端和DBMS_DATAPUMP PL / SQL包中DBMS_DATAPUMP 。

配置透明数据加密以在多数据库环境中工作

同一服务器上的每个Oracle数据库（例如共享相同Oracle二进制文件但使用不同数据文件的数据库）必须访问其自己的TDE密钥库。

密钥库不是为在数据库之间共享而设计的。按照设计，每个数据库必须有一个密钥库。您不能将同一个密钥库用于多个数据库。

  • 要在多数据库环境中配置密钥库的使用，请使用以下选项之一：

    • 选项1：通过为每个CDB（或独立数据库）单独设置WALLET_ROOT静态初始化参数和TDE_CONFIGURATION动态初始化参数（其KEYSTORE_CONFIGURATION属性设置为FILE ）来指定密钥库位置。必须将KEYSTORE_CONFIGURATION属性设置为FILE才能使WALLET_ROOT参数起作用。

      例如：

      WALLET_ROOT = $ ORACLE_BASE / admin / db_unique_name TDE_CONFIGURATION =“KEYSTORE_CONFIGURATION = FILE”

    • 选项2：如果未设置WALLET_ROOT和TDE_CONFIGURATION ，并且数据库共享同一Oracle主目录，则确保未设置sqlnet.ora中的WALLET_LOCATION和ENCRYPTION_WALLET_LOCATION参数。默认情况下， sqlnet.ora位于$ORACLE_BASE/network/admin目录中（如果设置了$ORACLE_BASE ）或位于$ORACLE_HOME/network/admin目录中。

      这使Oracle数据库能够使用位于$ORACLE_BASE/admin/db_unique_name/wallet的密钥库（假设已设置$ORACLE_BASE ）或$ORACLE_HOME/admin/db_unique_name/wallet目录。

    • 选项3：如果选项1和2不可行，则使用单独的sqlnet.ora文件，每个数据库一个。确保正确设置TNS_ADMIN环境变量以指向正确的数据库配置。但是，请注意，从版本19c开始，不推荐使用sqlnet.ora中的ENCRYPTION_WALLET_LOCATION参数，而是支持WALLET_ROOT和TDE_CONFIGURATION初始化参数。

警告：

使用其他数据库中的密钥库可能会导致部分或完全数据丢失。

B网关RPC接口

要使用Oracle Database Gateway for APPC执行远程事务程序，必须执行PL / SQL程序以调用网关功能， 远程过程调用（RPC）。网关功能处理与远程事务程序的网关会话的启动，数据交换和终止。

用于APPC的Oracle数据库网关包括一个工具PGAU，用于生成 PL / SQL包（TIP）根据您以COBOL记录布局和形式提供的定义自动生成 PGDL（程序网关定义语言）。

该网关功能全部通过执行远程过程调用（RPC）。函数从中调用 PL / SQL代码如下：

功能@ DBLINK（PARM1，parm2，...，parmn）;
