类：MapMarker

OM 。 MapMarker

此类表示作为点要素的地图标记。地图标记可以具有顺序标签
这使它们易于链接到详细视图中的条目。例如，地图标记可以指示
POI（例如酒店）在地图上，而相应的详细信息显示在页面上的单独视图中。

新的MapMarker（mObj）

参数：
名称 类型 描述
MOBJ 宾语

一个指定标记属性的对象。它可能包含以下属性。


  • id {String}：地图标记的唯一ID。

  • label {String}：可选。标记的标签文本。

  • labelStyle {OM.style。文字}：可选。标记的标签样式。

  • markerText {String}：可选。标记上的文字。

  • markerTextStyle {OM.style。文字}：可选。标记上的文字样式。

  • draggable {Boolean}：可选。如果设置为true，则标记可拖动。

  • renderingStyle {OM.style。标记}：可选。标记的渲染样式。
    如果未指定渲染样式，则将使用默认标记样式。

  • position {Object}：指定标记点位置的对象。它可能包含以下属性。


    • x {Number}：标记的x坐标

    • y {Number}：标记的y坐标

    • srid {Integer}：可选。上述x，y值的坐标系。默认值为8307


  • dragStart {Function}：可选。拖动启动事件监听器功能。

  • 拖动{功能}：可选。拖动事件监听器功能。

  • dragEnd {Function}：可选。拖动结束事件监听器功能。

5用C，C ++和Java实现数据磁带

您可以使用C，C ++和Java来实现数据盒式磁带方法。方法是定义使用数据盒定义的数据所允许的操作的过程和函数。重点是与开发和调试外部程序有关的问题。

5.1共享库

共享库是一个操作系统文件，例如Windows DLL或Solaris共享对象，它存储外部过程的编码实现。您可以使用别名库从Oracle访问共享库，该别名库是表示PL / SQL中的库的架构对象。出于安全原因，您需要DBA权限才能创建别名库。

也可以看看：

  • 有关使用专用外部过程代理的详细信息，请参见Oracle Database PL / SQL语言参考

  • 有关库的目录对象支持的更多信息，请参见“Oracle数据库安全指南

5.2外部程序

PL / SQL是一种用于数据库编程的强大语言，但有些方法过于复杂，无法在PL / SQL中进行最佳编码。例如，如果在C而不是PL / SQL中实现，则执行数字集成的例程可能运行得更快。

为了支持这种特殊目的的处理，PL / SQL提供了一个用于调用用其他语言编写的例程的接口。这使得3G的优势和功能（如C）可通过数据库服务器的调用获得。这种3GL程序称为外部程序 ;它存储在一个共享库中，使用PL / SQL注册，并在运行时从PL / SQL调用。

外部过程是数据盒式开发人员的重要工具。它们不仅可用于为盒式磁带编写快速，高效，计算密集的例程，还可用于将现有代码与数据库集成为数据盒式磁带。来自其他语言的现有共享库（例如具有用于执行音频文件格式转换的C例程的Windows NT DLL）可以直接从音频盒式磁带实现的类型中的方法调用。同样，您可以使用外部过程处理信号，驱动设备，分析数据流，渲染图形或处理数字数据。

5.3 PL / SQL如何调用外部过程

要调用外部过程，PL / SQL必须知道过程所在的DLL或共享库。PL / SQL在注册外部过程的子程序的EXTERNAL子句中查找别名库。数据字典用于确定操作系统共享库或DLL的实际路径。

PL / SQL警告侦听器进程，该进程又启动特定于会话的代理程序。除非在过程指定库的CREATE LIBRARY语句中或在CREATE PROCEDURE语句的agent参数中指定了某个其他特定代理程序，否则将启动默认代理程序extproc 。监听器将与代理的连接移交给代理。PL / SQL向代理传递DLL的名称，外部过程的名称以及调用者传入的任何参数。此帐户的其余部分假定启动的代理是默认代理extproc 。

在收到DLL的名称和外部过程后， extproc加载DLL并运行外部过程。此外， extproc处理服务调用，例如引发异常，以及回调到Oracle服务器。最后， extproc将外部过程返回的任何值传递给PL / SQL。图5-1显示了控制流程。

5.4.6使用WITH CONTEXT子句

启动时，外部过程必须访问数据库。例如， DS_Findmin不会将整个CLOB数据复制到c_findmin ，因为这样做会大大增加C例程所需的堆栈数量。相反，PL / SQL函数只是将LOB定位器传递给C例程，目的是再次从C访问数据库以读取实际的LOB数据。

当C例程读取数据时，它可以使用与LOB关联的OCI缓冲和流接口，因此只需要增量堆栈。从外部过程重新访问数据库称为回调 。

为了能够回调数据库，必须使用WITH CONTEXT子句为外部过程提供对数据库环境，服务和错误句柄的访问。使用WITH CONTEXT调用外部过程时，相应的C例程会自动获取类型为OCIExtProcContext *的参数作为其第一个参数。可以使用PARAMETERS子句更改参数的顺序。您可以使用此上下文指针使用OCIExtProcGetEnv调用获取句柄，然后回调数据库。该过程如例5-7所示。

5.8使用数据磁带外部程序的指南

确保编写线程安全的外部过程。特别是，避免使用静态变量，这些变量可以由在不同线程中运行的例程共享。

有关创建动态链接库的帮助，请查看RDBMS子目录/public ，其中可以找到模板makefile 。

调用外部过程时，切勿写入IN参数或溢出OUT参数的容量。PL / SQL不对这些错误条件进行运行时检查。同样，永远不要读取OUT参数或功能结果。此外，始终为IN OUT和OUT参数以及功能结果分配值。否则，您的外部过程不会成功返回。

如果包含WITH CONTEXT和PARAMETERS子句，则必须指定参数CONTEXT ，该参数显示参数列表中上下文指针的位置。如果省略PARAMETERS子句，则上下文指针是传递给外部过程的第一个参数。

如果包含PARAMETERS子句且外部过程是函数，则必须在最后一个位置指定参数RETURN （不是RETURN property ）。

对于每个形式参数， PARAMETERS子句中必须有相应的PARAMETERS 。此外，请确保PARAMETERS子句中的PARAMETERS数据类型与C原型中的PARAMETERS数据类型兼容，因为不会进行隐式转换。

为其指定INDICATOR或LENGTH的参数与相应的形式参数具有相同的参数模式。但是，即使您还指定了BY REFERENCE ，指定MAXLEN ， CHARSETID或CHARSETFORM的参数也始终被视为IN参数。

对于CHAR ， LONG RAW ， RAW或VARCHAR2类型的参数，必须使用属性LENGTH 。此外，如果该参数为IN OUT或OUT且为null，则必须将相应C参数的长度设置为零。

E.1关于使用Oracle Secure Backup云模块在云上进行备份

Oracle Secure Backup云模块是Oracle Secure Backup产品系列的一部分，可以灵活地将数据库备份到Amazon S3 Cloud和磁带。借助此云产品，本地磁盘备份可直接发送到Amazon S3以进行异地存储，并与Recovery Manager（RMAN）功能完全集成。

Oracle Secure Backup云模块有效地处理Oracle数据库备份到S3存储。您可以从Oracle Database 9 i Release 2或更高版本开始备份Oracle数据库。此外，Oracle Secure Backup云模块备份可与Oracle Enterprise Manager和您自定义的RMAN脚本等工具配合使用。Oracle Secure Backup云模块不备份操作系统文件。

Oracle Secure Backup云模块使用RMAN SBT（系统备份到磁带）界面扩展Amazon S3功能以进行Oracle备份操作。Oracle安全备份云模块提供易于管理，经济高效且可扩展的替代方案，可用于维护内部数据存储和管理本地完全配置的备份基础架构。

与传统的基于磁带的异地备份相比，Oracle Secure Backup云模块具有以下几个优势：

  • 连续可访问性

    始终可以访问存储在Amazon S3存储上的Oracle Secure Backup云模块备份。云存储服务可用性和访问模型可帮助组织简化恢复操作。例如，在执行还原之前无需发送或加载磁带。您仍然可以使用熟悉的标准工具（如Enterprise Manager），并且您组织的当前脚本将继续执行备份和还原任务。由于能够持续轻松地访问备份，因此可以大大减少恢复备份所花费的时间。

  • 提高可靠性

    由于S3存储是基于磁盘的，因此它本身比磁带介质更可靠。互联网存储服务提供商会保留数据的多个冗余副本，以实现可用性和可扩展性，并且这种做法可以为您的组织和数据带来更高的可靠性。

物理存储结构简介

RDBMS的一个特征是逻辑数据结构（例如表，视图和索引）与物理存储结构的独立性。

由于物理和逻辑结构是分开的，因此您可以管理数据的物理存储，而不会影响对逻辑结构的访问。例如，重命名数据库文件不会重命名存储在其中的表。

Oracle数据库是一组将Oracle数据存储在永久磁盘存储中的文件。本节讨论发出CREATE DATABASE语句时生成的数据库文件：

  • 数据文件和临时文件

    数据文件是磁盘上的物理文件，由Oracle数据库创建，包含表和索引等数据结构。临时文件是属于临时表空间的数据文件。数据库以Oracle专有格式将数据写入这些文件，其他程序无法读取。

  • 控制文件

    控制文件是跟踪数据库物理组件的根文件。

  • 在线重做日志文件

    联机重做日志是一组文件，包含对数据所做更改的记录。

存储数据库文件的机制

有几种机制可用于分配和管理这些文件的存储。

最常见的机制包括：

  • Oracle自动存储管理（Oracle ASM）

    Oracle ASM包含专为Oracle数据库使用而设计的文件系统。

  • 操作系统文件系统

    大多数Oracle数据库将文件存储在文件系统中 ，该文件系统是在连续磁盘地址空间内构建的数据结构。所有操作系统都有文件管理器，可以将磁盘空间分配和释放到文件系统中的文件中。

    文件系统允许将磁盘空间分配给许多文件。每个文件都有一个名称，并使其显示为Oracle数据库等应用程序的连续地址空间。数据库可以创建，读取，写入，调整大小和删除文件。

    文件系统通常构建在由称为逻辑卷管理器（LVM）的软件包构建的逻辑卷之上。LVM可以将多个物理磁盘组合成一个连续的地址空间，该地址空间作为一个磁盘显示在更高层的软件上。

  • 集群文件系统

    集群文件系统是一个分布式文件系统，它是一组服务器，它们协作以向其客户端提供高性能服务。在Oracle RAC环境中，群集文件系统使共享存储显示为群集环境中许多计算机共享的文件系统。使用群集文件系统时，群集中计算机的故障不会使文件系统不可用。但是，在操作系统文件系统中，如果通过NFS或其他方式共享文件的计算机出现故障，则文件系统不可用。

数据库采用前述存储机制的组合。例如，数据库可以将控制文件和联机重做日志文件存储在传统文件系统中，原始分区上的一些用户数据文件，Oracle ASM中的剩余数据文件以及将重做日志文件存档到群集文件系统。

Oracle自动存储管理（Oracle ASM）

Oracle ASM是一种针对Oracle数据库文件的高性能，易于管理的存储解决方案。Oracle ASM是一个卷管理器，它提供专门供数据库使用的文件系统。

与传统文件系统和存储管理器相比，Oracle ASM具有以下优势：

  • 简化与存储相关的任务，例如创建和布局数据库以及管理磁盘空间

  • 跨物理磁盘分配数据以消除热点并在磁盘上提供统一的性能

  • 存储配置更改后自动重新平衡数据

要使用Oracle ASM，可以为Oracle数据库分配带有条带化和镜像首选项的分区磁盘。Oracle ASM管理磁盘空间，在所有可用资源之间分配I / O负载，以优化性能，同时无需手动I / O调整。例如，您可以增加数据库的磁盘大小，或者将数据库的某些部分移动到新设备，而无需关闭数据库。

    注意：

    Oracle ASM范围与用于在段中存储数据的范围不同。

  • Oracle ASM分配单元

    Oracle ASM分配单元是磁盘组中的基本分配单元 。分配单元是Oracle ASM分配的最小连续磁盘空间。一个或多个分配单元形成Oracle ASM范围。

Oracle托管文件和用户管理文件

Oracle Managed Files是一种文件命名策略，使您可以根据数据库对象而不是文件名来指定操作。例如，您可以创建表空间而不指定其数据文件的名称。

Oracle Managed Files使管理员无需直接管理数据库中的操作系统文件。Oracle ASM需要Oracle Managed Files。

注意：

此功能不会影响管理文件（如跟踪文件，审核文件和警报日志）的创建或命名。

使用用户管理的文件，您可以直接管理数据库中的操作系统文件。您可以做出有关文件结构和命名的决定。例如，在创建表空间时，可以设置表空间数据文件的名称和路径。

通过初始化参数，可以为特定类型的文件指定文件系统目录。Oracle Managed Files功能可确保数据库创建唯一文件，并在不再需要时将其删除。数据库在内部使用标准文件系统接口来创建和删除存储在快速恢复区域中的数据文件和临时文件，控制文件以及与恢复相关的文件的文件。

Oracle Managed Files不会消除现有功能。您可以在手动管理旧文件时创建新文件。因此，数据库可以混合使用Oracle托管文件和用户管理的文件。

数据文件概述

在操作系统级别，Oracle数据库将数据库数据存储在称为数据文件的结构中。每个Oracle数据库必须至少有一个数据文件。

使用数据文件

Oracle数据库将表空间数据物理存储在数据文件中。

每个非分区的模式对象和对象的每个分区都存储在它自己的段中 ，该段只属于一个表空间。例如，非分区表的数据存储在单个段中，而该段又存储在一个表空间中。表空间和数据文件密切相关，但有重要区别：

  • 每个表空间由一个或多个数据文件组成，这些文件符合运行Oracle数据库的操作系统。

  • 数据库的数据共同存储在位于数据库的每个表空间中的数据文件中。

  • 段可以跨越一个或多个数据文件，但不能跨越多个表空间。

  • 数据库必须具有SYSTEM和SYSAUX表空间。在数据库创建期间，Oracle数据库会自动为SYSTEM表空间分配任何数据库的第一个数据文件。

    SYSTEM表空间包含数据字典 ，一组包含数据库元数据的表。通常，数据库还具有撤消表空间和临时表空间（通常名为TEMP ）。

永久和临时数据文件

永久表空间包含持久模式对象。永久表空间中的对象存储在数据文件中。

临时表空间仅在会话期间包含架构对象。本地管理的临时表空间具有临时文件（临时文件），这些文件是用于在哈希，排序和其他操作中存储数据的特殊文件。当内存中存在的空间不足时，临时文件还会存储结果集数据。

临时文件类似于永久数据文件，但以下情况除外：

  • 永久数据库对象（如表）永远不会存储在临时文件中。

  • 临时文件始终设置为NOLOGGING模式，这意味着它们永远不会为它们生成重做。介质恢复无法识别临时文件。

  • 您无法将临时文件设置为只读。

  • 您无法使用ALTER DATABASE语句创建临时文件。

  • 创建或调整临时文件大小时，并不总是保证为指定的文件大小分配磁盘空间。在Linux和UNIX等文件系统上，临时文件创建为稀疏文件 。在这种情况下，磁盘块不是在创建文件或调整大小时分配的，而是在第一次访问块时分配。

    注意：

    稀疏文件可以创建快速临时文件和调整大小;但是，访问临时文件后，磁盘可能会耗尽空间。

  • 临时文件的信息被示出在数据字典视图DBA_TEMP_FILES和动态性能视图V$TEMPFILE ，但不是在DBA_DATA_FILES或V$DATAFILE视图。

在线和离线数据文件

每个数据文件都是在线（可用）或脱机（不可用）。

您可以通过使其脱机或使其联机来更改单个数据文件或临时文件的可用性。数据库在联机之前无法访问脱机数据文件。

您可能出于多种原因使数据文件脱机，包括执行脱机备份或阻止损坏。如果数据库无法写入数据库，则数据库会自动使数据文件脱机。

与数据文件一样，表空间本身也是脱机或联机的。在联机表空间中使数据文件脱机时，表空间本身仍保持联机状态。您可以通过使表空间本身脱机来使表空间的所有数据文件暂时不可用。

从Oracle Database 12c开始 ，您可以使用ALTER DATABASE MOVE DATAFILE语句在数据库打开并访问该文件时将在线数据文件从一个物理文件移动到另一个物理文件。您可以使用此技术来实现以下目标：

  • 将表空间从一种存储移动到另一种存储

  • 将不经常访问的数据文件移动到较低成本的存储

  • 将表空间设置为只读并将其数据文件移动到一次写入存储，例如一次写入多次读取（WORM）驱动器

  • 将数据库移动到Oracle ASM

控制文件概述

数据库控制文件是仅与一个数据库关联的小型二进制文件。尽管允许多个相同的副本，但每个数据库都有一个唯一的控制文件。

使用控制文件

Oracle数据库使用控制文件来定位数据库文件并通常管理数据库的状态。

控制文件包含以下信息：

  • 数据库名称和数据库唯一标识符（DBID）

  • 数据库创建的时间戳

  • 有关数据文件，联机重做日志文件和归档重做日志文件的信息

  • 表空间信息

  • RMAN备份

控制文件用于以下目的：

  • 它包含有关打开数据库所需的数据文件，联机重做日志文件等信息。

    控制文件跟踪数据库的结构更改。例如，当管理员添加，重命名或删除数据文件或联机重做日志文件时，数据库会更新控制文件以反映此更改。

  • 它包含在数据库未打开时必须可访问的元数据。

    例如，控制文件包含恢复数据库所需的信息，包括检查点。检查点指示重做流中的SCN ，其中需要开始实例恢复。检查点SCN之前的每个已提交的更改都保证保存在数据文件的磁盘上。检查点进程至少每三秒在控制文件中记录有关联机重做日志中检查点位置的信息。

Oracle数据库在数据库使用期间不断读取和写入控制文件，并且只要数据库处于打开状态，就必须可以写入。例如，恢复数据库涉及从控制文件中读取数据库中包含的所有数据文件的名称。其他操作，例如添加数据文件，更新存储在控制文件中的信息。

多个控制文件

Oracle数据库允许多个相同的控制文件同时打开并写入同一数据库。通过在不同磁盘上复用控制文件，数据库可以实现冗余，从而避免单点故障。

注意：

Oracle建议您在不同的磁盘上维护多个控制文件副本。

如果控制文件变得不可用，则数据库实例在尝试访问损坏的控制文件时会失败。当存在其他当前控制文件副本时，您可以重新装入数据库并在没有介质恢复的情况下打开它。但是，如果数据库的所有控制文件都丢失，则数据库实例将失败并需要进行介质恢复。如果必须使用较旧的控制文件备份，因为当前副本不可用，则介质恢复并不简单。

控制文件结构

有关数据库的信息存储在控制文件的不同部分中。每个部分都是一组关于数据库方面的记录。

例如，控制文件中的一个部分跟踪数据文件并包含一组记录，每个记录对应一个数据文件。每个部分都存储在多个逻辑控制文件块中。记录可以跨越一个部分中的块。

控制文件包含以下类型的记录：

  • 循环重用记录

    循环重用记录包含非关键信息，如果需要，可以覆盖这些信息。当所有可用记录槽已满时，数据库会扩展控制文件以为新记录腾出空间或覆盖最旧记录。示例包括有关归档重做日志文件和RMAN备份的记录。

  • 非圆重用记录

    非圆形重用记录包含不会经常更改且无法覆盖的关键信息。信息示例包括表空间，数据文件，联机重做日志文件和重做线程。除非从表空间中删除相应的对象，否则Oracle数据库永远不会重用这些记录。

您可以查询动态性能视图（也称为V$视图）以查看存储在控制文件中的信息。例如，您可以查询V$DATABASE以获取数据库名称和DBID。但是，只有数据库可以修改控制文件中的信息。

读取和写入控制文件块不同于读取和写入数据块。对于控制文件，Oracle数据库直接从磁盘读取和写入程序全局区域（PGA）。每个进程为控制文件块分配一定量的PGA内存。

在线重做日志概述

最重要的恢复结构是联机重做日志 ，它由两个或多个预分配文件组成，这些文件在数据库发生变化时存储更改。联机重做日志记录对数据文件的更改。

使用在线重做日志

数据库维护联机重做日志文件以防止数据丢失。具体而言，在实例发生故障后，联机重做日志文件使Oracle数据库能够恢复尚未写入数据文件的已提交数据。

服务器进程将每个事务同步写入重做日志缓冲区 ，然后LGWR进程将其写入联机重做日志。联机重做日志的内容包括未提交的事务以及架构和对象管理语句。

当数据库对还原段进行更改时，数据库还会将这些更改写入联机重做日志。因此，联机重做日志始终包含永久对象的撤消数据。您可以将数据库配置为在临时还原段中存储临时对象的所有还原数据，从而节省空间并提高性能，或允许数据库在联机重做日志中存储永久和临时撤消数据。

Oracle数据库仅使用联机重做日志进行恢复。但是，管理员可以通过Oracle LogMiner实用程序中的SQL界面查询联机重做日志文件（请参阅“ Oracle LogMiner ” ）。重做日志文件是有关数据库活动的有用历史信息源。

Oracle数据库如何写入联机重做日志

数据库实例的联机重做日志称为重做线程 。

在单实例配置中，只有一个实例访问数据库，因此只存在一个重做线程。但是，在Oracle Real Application Clusters（Oracle RAC）配置中，多个实例同时访问数据库，每个实例都有自己的重做线程。每个实例的单独重做线程可避免争用一组联机重做日志文件。

联机重做日志包含两个或多个联机重做日志文件。Oracle数据库至少需要两个文件，以保证一个文件始终可用于写入，以防另一个文件正在被清除或存档。

日志写入器进程（LGWR）进程正在主动写入的联机重做日志文件称为当前联机重做日志文件 。

  • 恢复数据库备份

联机重做日志文件包含重做记录。

  • SCN和更改的时间戳

  • 进行更改的操作类型

  • 修改后的数据段的名称和类型

参数

trgthndlp （IN）

要检索的属性类型。句柄类型及其可读属性在Handle和Descriptor Attributes中列出。

14.1 SEM_RDFSA.APPLY_OLS_POLICY

格式

SEM_RDFSA.APPLY_OLS_POLICY（policy_name IN VARCHAR2，rdfsa_options IN NUMBER DEFAULT SEM_RDFSA.SECURE_SUBJECT，table_options IN VARCHAR2 DEFAULT'ALL_CONTROL'，label_function IN VARCHAR2 DEFAULT NULL，谓词IN VARCHAR2 DEFAULT NULL，network_owner IN VARCHAR2 DEFAULT'MDSYS'，network_name IN VARCHAR2 DEFAULT NULL） ;

描述

将OLS策略应用于语义数据存储。

参数

policy_name

现有OLS策略的名称。

rdfsa_options

指定要为RDF数据启用的细粒度访问控制模式的选项。保护RDF数据的默认选项包括为出现三元组主题位置的资源分配敏感度标签。您可以使用rdfsa_options参数覆盖默认值，并指定使用说明中表14-1中定义的常量之一。

table_options

政策执行选项。默认值（ ALL_CONTROL ）是此过程唯一支持的值。

label_function

一个字符串，用于调用函数以返回标签值以用作默认值。

谓词

与基于标签的谓词结合使用的附加谓词。

network_owner

语义网络的所有者。（ 见表1-1 ）

network_name

语义网络的名称。（ 见表1-1 ）

使用说明

必须使用CTXT1作为列名创建使用此过程指定的OLS策略，并且它应使用默认策略选项。有关策略选项的信息，请参阅“ Oracle Label Security管理员指南” 。

此过程在MDSYS模式中定义的多个表上调用sa_policy_admin.apply_table_policy过程。SEM_RDFSA.APPLY_OLS_POLICY过程的参数table_options ， label_function和predicate与sa_policy_admin.apply_table_policy过程中具有相同名称的参数具有相同的语义。

对于rdfsa_options参数，可以为所需选项指定包常量。表14-1列出了这些常量及其说明。

表14-1 rdfsa_options参数的SEM_RDFSA包常量

不变 描述

SEM_RDFSA.SECURE_SUBJECT

为出现三元组主题位置的资源指定敏感度标签。

SEM_RDFSA.SECURE_PREDICATE

为出现三元组谓词位置的资源指定敏感度标签。

SEM_RDFSA.SECURE_OBJECT

为出现三元组对象位置的资源指定敏感度标签。

SEM_RDFSA.TRIPLE_LEVEL_ONLY

应用三级安全性。提供良好的性能，无需为各个资源分配标签。（需要安装My Oracle Support提供的Patch 9819833。）

SEM_RDFSA.OPT_DEFINE_BEFORE_USE

在为资源定义灵敏度标签之前，限制在三元组中使用RDF资源。如果未指定此选项，则用户的初始行标签将在首次使用时用作资源的默认标签。

SEM_RDFSA.OPT_RELAX_TRIPLE_LABEL

放宽三重标签与与其所有组件相关的标签之间存在的主导关系。使用此选项，如果用户具有对所有三元组件的READ访问权限，并且三元标签可能与组件标签没有任何关系，则可以定义三元组。如果没有此选项，三重标签应至少覆盖其所有组件的标签。

您可以在label_function参数中指定一个函数，以便为新插入的三元组生成自定义标签。label函数与MDSYS.RDF_LINK $表关联，此表中的列可以配置为label函数的参数，如以下示例所示：

fgac_admin.new_triple_label（：new.model_id，：new.start_node_id，：new.p_value_id，：new.canon_end_node_id）'

由于OLS策略应用于具有不同结构的多个表，因此分配给predicate参数的任何谓词中唯一有效的列引用是标签列：CTXT1。如果为具有现有数据的语义数据存储启用OLS，则可以指定'OR CTXT1 is null'形式的谓词，以便能够继续使用此数据而不受访问限制。

支持OLS的语义数据存储使用针对在多个模型中组织的所有RDF三元组的敏感性标签。用户通过模型视图和SEM_MATCH查询访问此类三元组受OLS策略的限制。此外，独立于拥有应用程序表的用户，对表中的三列（类型为SDO_RDF_TRIPLE_S）的访问仅限于具有OLS策略的FULL访问权限的用户。

使用相应应用程序表上的INSERT权限将三元组插入到特定的RDF模型中。使用用户的会话上下文（初始行标签）或标签函数生成新三元组的敏感度标签。使用与三元组件关联的标签验证三元组是否存在任何RDF策略违规。虽然可能无法通过应用程序表访问三重信息，但可以查询模型视图以访问三元组，同时强制执行OLS策略限制。如果您具有必要的策略权限（例如writeup，writeacross），则可以更新模型视图中的CTXT1列以重置分配给该三元组的标签。对于涉及三重组件的任何RDF策略违规，新标签会自动进行验证。模型视图的CTXT1列的更新权限被授予模型的所有者，并且此用户可以选择性地将此权限授予其他用户。

如果RDF模型是在具有FULL访问权限的用户之外的模式中创建的，则对模型对象具有必要的权限 - 特别是对应用程序表的读/写访问权限，对模型视图的读访问权以及对CTXT1列的写访问权。模型视图 - 可以授予此类用户进行维护操作。这些操作包括批量加载到模型中，重置分配给三元组的任何敏感标签，以及使用模型创建蕴涵。

要禁用OLS策略，请使用SEM_RDFSA.DISABLE_OLS_POLICY过程。

有关OLS支持的信息，请参阅RDF数据的细粒度访问控制 。

有关语义网络类型和选项的信息，请参阅语义网络 。

例子

以下示例使用安全主题和安全谓词选项启用对RDF数据的安全访问。

开始sem_rdfsa.apply_ols_policy（policy_name =>'defense'，rdfsa_options => sem_rdfsa。SECURE_SUBJECT + sem_rdfsa。SECURE_PREDICATE）;结束; /

以下示例通过指定“使用前定义”选项扩展了前面的示例，该选项允许用户仅在使用关联的敏感度标签预定义三重组件（主题，谓词或对象）时才定义三元组。如果插入三元组的用户没有SEM_RDFSA包的执行权限，则此配置有效。

开始sem_rdfsa.apply_ols_policy（policy_name =>'defense'，rdfsa_options => sem_rdfsa。SECURE_SUBJECT + sem_rdfsa。SECURE_PREDICATE + sem_rdfsa。OPT_DEFINE_BEFORE_USE）;结束; /

11.1什么是线程？

多线程应用程序在共享地址空间中执行多个线程。线程是在进程内执行的“轻量级”子进程。它们共享代码和数据段，但有自己的程序计数器，机器寄存器和堆栈。全局变量和静态变量对于所有线程都是通用的，并且通常需要互斥机制来管理来自应用程序内多个线程的这些变量的访问。 互斥锁是确保保持数据完整性的同步机制。

有关互斥锁的进一步讨论，请参阅有关多线程的文本。有关多线程应用程序的更多详细信息，请参阅线程函数的文档。

11.2 Pro * C / C ++中的运行时上下文

为了松散地耦合线程和连接，Pro * C / C ++引入了运行时上下文的概念。运行时上下文包括以下资源及其当前状态：

  • 与一个或多个Oracle服务器的零个或多个连接

  • 用于服务器连接的零个或多个游标

  • 内联选项，例如MODE，HOLD_CURSOR，RELEASE_CURSOR和SELECT_ERROR

Pro * C / C ++不是简单地支持线程和连接之间的松散耦合，而是提供了将线程与运行时上下文松散耦合的能力。Pro * C / C ++允许您的应用程序定义运行时上下文的句柄，并将该句柄从一个线程传递到另一个线程。

例如，交互式应用程序生成一个线程T1，以执行查询并将前10行返回给应用程序。然后T1终止。在获得必要的用户输入之后，产生另一个线程T2（或者使用现有线程），并且T1的运行时上下文被传递给T2，因此它可以通过处理相同的光标来获取接下来的10行。见图11-1 。

11.4.4编程注意事项

虽然Oracle确保SQLLIB代码是线程安全的，但您有责任确保Pro * C / C ++源代码能够与线程一起正常工作;例如，仔细考虑使用静态和全局变量。

此外，多线程应用程序需要有关以下内容的设计决策：

  • 将SQLCA声明为线程安全结构，通常是一个自动变量，每个运行时上下文一个

  • 声明SQLDA是一个线程安全的结构，就像SQLCA一样，通常是一个自动变量，每个运行时上下文都有一个

  • 以线程安全的方式声明主机变量，换句话说，仔细考虑使用静态和全局主机变量。

  • 避免在多个线程中同时使用运行时上下文

  • 是否使用默认数据库连接或使用AT子句显式定义它们

此外，在给定时间，在运行时上下文中，只有一个可执行的嵌入式SQL语句（例如，EXEC SQL UPDATE）可能是未完成的。

11.6连接池

连接池是一组到数据库的物理连接，可以由多个连接重用。连接池功能的目标是通过避免每个连接使用专用连接来提高性能并减少资源使用。

图11-4说明了连接池功能的功能。在此示例中，应用程序的四个线程使用连接池与数据库交互。连接池有两个物理连接。连接池句柄由四个使用不同运行时上下文的线程使用。

图11-4连接池

thread1（）{EXEC SQL CONTEXT ALLOCATE：ctx1; EXEC SQL语境使用：ctxl; EXEC SQL CONNECT：uid AT：TC1 USING：db_string; ...} thread2（）{EXEC SQL CONTEXT ALLOCATE：ctx2; EXEC SQL CONNECT：uid AT：TC2使用：db_string; ...} thread3（）{EXEC SQL CONTEXT ALLOCATE：ctx3; EXEC SQL CONNECT：uid AT：TC3 USING：db_string; EXEC SQL AT：TC3 SELECT count（*）into：count FROM emp; ...} thread4（）{EXEC SQL CONTEXT ALLOCATE：ctx4; EXEC SQL CONNECT：uid AT：TC4使用：db_string; ...}

在该示例中，四个命名连接TC1，TC2，TC3和TC4分别是由线程T1，T2，T3和T4创建的虚拟连接。来自不同运行时上下文的命名连接TC1，TC2，TC3和TC4共享相同的连接池，并共享连接池中可用的物理数据库连接。两个物理连接C1和C2提供四个命名连接并连接到同一个数据库。

当接收到来自线程T1的第一个连接请求TC1时，SQLLIB创建一个连接池，其中一个物理连接C1到数据库。当来自线程T2的另一个连接请求TC2被发送到同一数据库时，C1向数据库提供TC2请求（如果它是空闲的）。否则，创建新的物理连接C2以服务该请求。如果来自名为TC3的线程T3的另一个连接请求进入，则TC3要么等待指定的时间，要么返回错误消息，如果物理连接C1和C2都忙。

当线程T2需要使用TC2命名连接选择数据时，它获取任何空闲物理连接C1或C2。在提供请求后，所选连接将再次在连接池中可用，以便另一个命名或虚拟连接可以使用相同的物理连接。

25.3 XStream Out和多租户环境

多租户环境使数据库能够包含一组可移植的模式，对象和相关结构，这些模式在逻辑上作为单独的数据库出现在应用程序中。

这个自包含的集合称为可插拔数据库（PDB）。多租户容器数据库（CDB）包含PDB。在CDB中，XStream Out的功能与非CDB中的功能大致相同。

CDB还可以包含应用程序容器。应用程序容器是CDB的可选组件，它由应用程序根目录和与之关联的所有应用程序PDB组成。应用程序容器存储一个或多个应用程序的数据。应用程序容器共享应用程序元数据和公共数据在CDB中，以下各项都是容器：CDB根，每个PDB，每个应用程序根和每个应用程序PDB。

XBream Out在CDB和非CDB中的主要区别在于：

  • 必须仅在CDB根目录中配置XStream Out。

  • XStream Out可以查看对CDB中任何容器所做的更改。

  • XStream Out捕获规则可以将LCR限制为客户端应用程序所需的LCR。系统生成的捕获规则根据传递给DBMS_XSTREAM_ADM包中的ADD_OUTBOUND和CREATE_OUTBOUND过程的参数选择适当的LCR。您可以在同一个包中使用ADD_*_RULES过程，以便对XStream Out组件使用的规则进行更细粒度的控制。

  • 执行XStream Out任务的用户必须是普通用户。

在XStream环境中拔下插头和插件操作

当XBream Out涉及的PDB，应用程序根或应用程序PDB从其CDB中拔出并插入另一个CDB时，任何捕获进程或出站服务器都不被视为容器的一部分。您必须在另一个CDB中再次配置捕获进程和出站服务器。

如果出站服务器配置在与捕获过程不同的数据库中，则拔出和插入操作还有其他注意事项。

对于此示例，假设以下内容：

  • 名为CDB1的CDB包含PDB PDB1 。

  • 捕获过程在CDB1配置，它将LCR从PDB1发送到名为CDB2的CDB中的出站服务器。

  • 您从CDB1拔出PDB1 ，然后将其插入名为CDB3的CDB。

要继续从提供的LCR PDB1在出站服务器CDB2 ，必须配置一个新的捕获过程CDB3捕获和发送的LCR到CDB2 。

XStream环境中的应用程序容器

当CDB具有一个或多个应用程序容器时，必须在CDB根目录中配置XStream Out，并且XStream Out可以捕获在CDB中的任何容器中所做的更改，包括应用程序根和应用程序PDB。可以将应用程序容器中捕获的更改发送到任何类型的容器，包括PDB，应用程序根和应用程序PDB。

将更改从一个应用程序根复制到另一个应用程序根时，XStream可以复制ALTER PLUGGABLE DATABASE APPLICATION语句。为避免错误，应用语句的目标应用程序根目录必须与源应用程序根目录安装相同的应用程序，并且应用程序名称在两个应用程序根目录中必须相同。

为了避免在将更改从应用程序根目录复制到非应用程序根目录的容器时出错，必须确保不复制ALTER PLUGGABLE DATABASE APPLICATION语句。

使用XStream OCI API，您可以控制是否使用OCIXStreamOutAttach函数和OCILCRHeaderGet函数复制ALTER PLUGGABLE DATABASE APPLICATION语句。使用XStream Java API，您可以使用XStreamOut.attach方法中的mode参数控制此行为。

25.5 XStream In和多租户环境

多租户环境使Oracle数据库能够包含一组可移植的模式，对象和相关结构，这些模式在逻辑上作为单独的数据库出现在应用程序中。

这个自包含的集合称为可插拔数据库（PDB）。多租户容器数据库（CDB）包含PDB。它还可以包含应用程序容器。应用程序容器是CDB的可选组件，它由应用程序根目录和与之关联的所有应用程序PDB组成。应用程序容器存储一个或多个应用程序的数据。应用程序容器共享应用程序元数据和公共数据在CDB中，以下各项都是容器：CDB根，每个PDB，每个应用程序根和每个应用程序PDB。

在CDB中，入站服务器仅限于从一个源数据库接收LCR，并且仅在当前容器中执行更改（一个PDB，一个应用程序根，一个应用程序PDB或CDB根）。单个入站服务器无法将更改应用于CDB中的多个容器。

当入站服务器位于CDB根目录中时，应用用户必须是普通用户。当入站服务器位于应用程序根目录中时，应用用户必须是普通用户或应用程序公共用户。当入站服务器位于PDB或应用程序PDB中时，应用用户可以是普通用户或本地用户。

注意：

XStream不会同步应用程序根容器中所做的更改。不要使用XStream In复制来复制在应用程序根容器中完成的操作。您可以在目标中的应用程序根容器中手动应用这些更改。请注意，仍可以复制在PDB中完成的操作。
