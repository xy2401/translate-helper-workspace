NLSSORT

句法


说明nlssort.eps的描述

目的

NLSSORT返回字符值char的排序规则键以及显式或隐式指定的排序规则。排序规则键是一个字节字符串，用于根据指定的排序规则对char进行排序。核对键的属性是当根据它们的二进制顺序进行比较时为给定的核对生成的两个这样的键的相互排序与根据给定的核对进行比较时的源字符值的相互排序相同。

char和' nlsparam '都可以是CHAR ， VARCHAR2 ， NCHAR或NVARCHAR2任何数据类型。

' nlsparam '的值必须具有该形式

'NLS_SORT = 整理 '

collation是语言排序规则或BINARY的名称。 NLSSORT使用指定的排序规则生成排序规则键。如果省略' nlsparam '，则此函数使用参数char的派生归类。如果指定BINARY ，则此函数返回char值本身强制转换为RAW并可能截断，如下所述。

如果指定' nlsparam '，则可以在语言排序规则名称后附加后缀_ai以请求不区分重音的排序规则，或者_ci请求不区分大小写的排序规则。有关重音和不区分大小写排序的详细信息，请参阅“ Oracle数据库全球化支持指南” 。不建议对ORDER BY查询子句使用不区分重音或不区分大小写的排序规则，因为它会导致不确定的排序顺序。

返回的归类键是RAW数据类型。由给定排序规则的给定char值产生的排序规则键的长度可能超过NLSSORT返回的RAW值的最大长度。在这种情况下， NLSSORT的行为取决于初始化参数MAX_STRING_SIZE的值。如果MAX_STRING_SIZE = EXTENDED ，则返回值的最大长度为32767字节。如果排序规则键超出此限制，则该功能将失败，并显示错误“ORA-12742：无法创建排序规则键”。如果短输入字符串包含高百分比的Unicode字符且分解率非常高，则也可能会报告此错误。

操作系统资源是任何计算机上的有限商品。由于Java的目标是提供计算平台和编程语言，因此它包含与平台无关的类和框架，用于访问特定于平台的资源。 Java类方法通过JVM访问操作系统资源。 Java有这个模型的潜在问题，因为当垃圾收集器管理的所有内容都是Java对象而不是Java对象所持有的操作系统资源时，程序员依赖垃圾收集器来管理所有资源。

2.9.2垃圾收集和操作系统资源

想象一下，内存分为两个领域：Java对象内存和操作系统结构。Java对象内存领域包含所有对象和变量。操作系统构造包括操作系统在询问时分配给对象的资源。这些资源包括文件，套接字等。

基本编程规则要求您关闭所有内存，包括Java对象和操作系统构造。Java程序员错误地认为垃圾收集器释放了内存。该创建垃圾收集器以收集所有未使用的Java对象内存。但是，它不会关闭操作系统结构。在Java对象被垃圾回收之前，程序必须关闭所有操作系统构造。

例如，只要对象打开文件，操作系统就会创建文件并为对象提供文件句柄。如果文件未关闭，则操作系统成立文件句柄构造打开，直到调用结束或JVM退出。这可能会导致您在必要时提前用完这些构造。每个操作系统中都有有限数量的句柄。为了保证您没有用完句柄，请在退出方法之前关闭资源。这包括在关闭套接字之前关闭连接到套接字的流。

出于性能原因，垃圾收集器无法检查每个对象以查看它是否包含句柄。因此，垃圾收集器会收集Java对象和变量，但不会发出适当的操作系统方法来释放任何句柄。

UNION [ALL]，INTERSECT，MINUS Operators

您可以使用集合运算符UNION ， UNION ALL ， INTERSECT和MINUS组合多个查询。所有集合运算符具有相同的优先级。如果SQL语句包含多个集合运算符，则Oracle数据库将从左到右对它们进行求值，除非括号明确指定另一个顺序。

复合查询的组件查询的选择列表中的相应表达式必须在数量上匹配，并且必须位于相同的数据类型组（例如数字或字符）中。

如果组件查询选择字符数据，则返回值的数据类型确定如下：

  • 如果两个查询都选择相等长度的数据类型CHAR的值，则返回的值具有该长度的数据类型CHAR 。如果查询选择具有不同长度的CHAR值，则返回的值为VARCHAR2 ，其长度为较大的CHAR值。

  • 如果其中一个或两个查询选择数据类型VARCHAR2的值，则返回的值具有数据类型VARCHAR2 。

如果组件查询选择数字数据，则返回值的数据类型由数字优先级确定：

  • 如果任何查询选择BINARY_DOUBLE类型的值，则返回的值具有数据类型BINARY_DOUBLE 。

  • 如果没有查询选择BINARY_DOUBLE类型的值，但任何查询选择BINARY_FLOAT值，则返回的值具有数据类型BINARY_FLOAT 。

  • 如果所有查询都选择NUMBER类型的值，则返回的值的数据类型为NUMBER 。

在使用set运算符的查询中，Oracle不会跨数据类型组执行隐式转换。因此，如果组件查询的相应表达式同时解析为字符数据和数字数据，则Oracle会返回错误。

集合运算符的限制

集合运算符受以下限制：

  • set运算符在BLOB ， CLOB ， BFILE ， VARRAY或嵌套表类型的列上无效。

  • UNION ， INTERSECT和MINUS运算符在LONG列上无效。

  • 如果set运算符前面的选择列表包含表达式，则必须为表达式提供列别名，以便在order_by_clause引用它。

  • 您也不能使用set运算符指定for_update_clause 。

  • 您不能在这些运算符的subquery中指定order_by_clause 。

  • 您不能在包含TABLE集合表达式的SELECT语句中使用这些运算符。

注意：

为了符合新兴的SQL标准，Oracle的未来版本将使INTERSECT运算符比其他集合运算符具有更高的优先级。因此，您应该使用括号来指定使用INTERSECT运算符和其他集合运算符的查询中的评估顺序。

DBMS_DEBUG_JDWP_CUSTOM包为数据库用户提供了一种使用Java调试线协议（JDWP）对调试器执行数据库会话的调试连接请求的自定义处理的方法。

56.1 DBMS_DEBUG_JDWP_CUSTOM概述

当您尝试使用Java调试线协议（JDWP）将数据库会话连接到调试器时，将调用DBMS_DEBUG_JDWP_CUSTOM包。在连接到数据库之前，必须在客户端OCI应用程序中设置ORA_DEBUG_JDWP环境变量。通过在模式中定义此程序包，可以在将数据库会话连接到调试程序之前执行自定义操作。

56.2 DBMS_DEBUG_JDWP_CUSTOM安全模型

DBMS_DEBUG_JDWP_CUSTOM包的此默认实现不执行任何其他安全检查，仅调用DBMS_DEBUG_JDWP包进行调试连接。调用DBMS_DEBUG_JDWP程序包时，它会在建立连接之前检查必需的DEBUG CONNECT系统或用户权限。

想要执行其他自定义安全检查的数据库用户必须执行以下步骤来覆盖此默认实现：

  • 在用户自己的架构中定义DBMS_DEBUG_JDWP_CUSTOM包。

  • 在用户的本地包副本中调用DBMS_DEBUG_JDWP包之前实现检查。

PGA_AGGREGATE_LIMIT指定实例消耗的聚合PGA内存的限制。

属性 描述

默认值

如果MEMORY_TARGET设置，然后PGA_AGGREGATE_LIMIT默认为MEMORY_MAX_TARGET值。

如果MEMORY_TARGET没有设置，然后PGA_AGGREGATE_LIMIT默认为200％ PGA_AGGREGATE_TARGET 。

如果未设置MEMORY_TARGET ，并且PGA_AGGREGATE_TARGET显式设置为0，则PGA_AGGREGATE_LIMIT的值设置为物理内存大小的90％减去总SGA大小。

对于PDB，默认值与CDB的默认值相同。

D句法和语义检查

通过检查嵌入式SQL语句和PL / SQL块的语法和语义，Oracle预编译器可帮助您快速查找和修复编码错误。本附录向您介绍如何使用SQLCHECK选项来控制检查的类型和范围。

主题是：

  • 句法和语义检查基础

  • 控制检查的类型和范围

  • 指定SQLCHECK = SEMANTICS

D.1句法和语义检查基础

语法规则指定语言元素如何排序以形成有效语句。因此， 语法检查会验证关键字，对象名称，运算符，分隔符等是否正确放置在SQL语句中。它也适用于从PL / SQL块调用的过程和函数。例如，以下嵌入式SQL语句包含语法错误：

*  - 拼写错误的关键字WHERE EXEC SQL DELETE FROM EMP WERE DEPTNO = 20 END-EXEC。 *  - 列名称缺少括号COMM和SAL EXEC SQL INSERT INTO EMP COMM，SAL VALUES（NULL，1500）END-EXEC。

语义规则指定如何进行有效的外部引用。因此， 语义检查验证对数据库对象和主机变量的引用是否有效以及主机变量数据类型是否正确。例如，以下嵌入式SQL语句包含语义错误：

*  - 不存在的表，EMPP EXEC SQL DELETE FROM EMPP WHERE DEPTNO = 20 END-EXEC。 *  - 未声明的主机变量，EMP-NAME EXEC SQL SELECT * FROM EMP WHERE ENAME =：EMP-NAME END-EXEC。

SQL语法和语义的规则在Oracle数据库SQL语言参考中定义。

D.2控制检查的类型和范围

您可以通过在命令行上指定SQLCHECK选项来控制检查的类型和范围。使用SQLCHECK，检查的类型可以是语法，也可以是语法和语义。检查的范围可以包括数据操作语句和PL / SQL块。但是，SQLCHECK无法检查动态SQL语句，因为它们在运行时才完全定义。

您可以为SQLCHECK指定以下值：

  • 语义学|充分

  • 语法|有限

SEMANTICS和FULL的值是等价的，SYNTAX和LIMITED的值也是等价的。默认值为SYNTAX。

D.3指定SQLCHECK = SEMANTICS

当SQLCHECK = SEMANTICS时，预编译器检查语法和语义

  • INSERT和UPDATE等数据操作语句。

  • PL / SQL块。

但是，只对使用AT db_name子句的数据操作语句进行语法检查。

预编译器从嵌入式DECLARE TABLE语句中获取语义检查的信息，或者，如果指定选项USERID，则通过连接到数据库并访问数据字典来获取。

如果连接到数据库但在数据字典中找不到某些表信息，则必须使用DECLARE TABLE语句提供缺少的信息。如果数据字典定义发生冲突，则DECLARE TABLE定义将覆盖它。

在检查数据操作语句时，预编译器使用Oracle数据库SQL语言参考中的Oracle语法规则集，但使用更严格的语义规则集。因此，当SQLCHECK = SEMANTICS时，为早期版本的Oracle编写的现有应用程序可能无法成功预编译。

预编译新程序时指定SQLCHECK = SEMANTICS。如果在主机程序中嵌入PL / SQL块，则必须指定SQLCHECK = SEMANTICS。

D.3.1启用语义检查

当SQLCHECK = SEMANTICS时，预编译器可以通过以下任一方式获取语义检查所需的信息：

  • 连接到Oracle并访问数据字典

  • 使用嵌入式DECLARE TABLE语句

D.3.1.1连接到Oracle

要进行语义检查，预编译器可以连接到数据库，该数据库维护主机程序中引用的表和视图的定义。连接后，预编译器访问数据字典以获取所需信息。数据字典存储表和列名称，表和列约束，列长度，列数据类型等。

如果在数据字典中找不到某些所需信息（例如，因为您的程序引用了尚未创建的表），则必须使用DECLARE TABLE语句提供缺少的信息。

D.3.1.2使用DECLARE TABLE

只要程序不从匿名PL / SQL块调用任何存储过程或函数，预编译器就可以在不连接数据库的情况下进行语义检查。要进行检查，预编译器必须从嵌入式DECLARE TABLE指令获取有关表和视图的信息。因此，必须在DECLARE TABLE语句中定义数据操作语句或PL / SQL块中引用的每个表。

DECLARE TABLE语句的语法是

EXEC SQL DECLARE table_name TABLE（col_name col_datatype [DEFAULT expr] [NULL | NOT NULL]，...）END-EXEC。

其中expr是可以在CREATE TABLE语句中用作默认列值的任何表达式。 col_datatype是Oracle列声明。只能使用整数，而不能使用表达式。请参阅“ DECLARE TABLE（Oracle嵌入式SQL指令） ” 。

如果使用DECLARE TABLE定义已存在的数据库表，则预编译器将使用您的定义，忽略数据字典中的定义。

磁盘组的性能和可伸缩性注意事项

本节讨论评估磁盘组性能的以下注意事项：

  • 确定磁盘组数量

  • 分组磁盘时的性能特征

  • Oracle ASM存储限制

确定磁盘组数量

使用以下条件确定要创建的磁盘组数：

  • 给定磁盘组中的磁盘应具有相似的大小和性能特征。如果在大小和性能方面有多种不同类型的磁盘，则创建包含类似特征的多个磁盘组。

  • 为数据库文件创建单独的磁盘组，为备份文件创建快速恢复区。如果发生磁盘组故障，此配置允许快速恢复。

有关为新安装创建磁盘组的信息，请参阅为新Oracle安装创建磁盘组 。

分组磁盘时的性能特征

Oracle ASM通过在磁盘组中的所有磁盘上均匀分布文件扩展区来平衡文件活动。为使此技术有效，磁盘组中的磁盘具有相似的性能特征非常重要。例如，最新和最快的磁盘可能驻留在为数据库工作区保留的磁盘组中，较慢的驱动器可能驻留在为快速恢复区保留的磁盘组中。

在某些情况下，可以接受暂时使磁盘组中存在不同大小和性能特征的磁盘共存。从一组旧磁盘迁移到一组新磁盘时就是这种情况。将添加新磁盘并删除旧磁盘。删除旧磁盘后，当磁盘组联机时，它们的存储将迁移到新磁盘。

Oracle ASM存储限制

Oracle ASM为未来增长提供了几乎无限的容量，但确实存在一些存储限制。

Oracle ASM对磁盘组，磁盘和文件的数量有以下限制：

  • 适用于Oracle Database 12 c第1版或更高版本的存储系统中的511个磁盘组

  • 磁盘组中的10,000个Oracle ASM磁盘

  • 存储系统中的65530个Oracle ASM磁盘

  • 每个磁盘组有100万个文件

如果COMPATIBLE.ASM或COMPATIBLE.RDBMS磁盘组属性设置为小于12.1则在没有任何Oracle Exadata存储的情况下，Oracle ASM具有以下存储限制：

  • 每个Oracle ASM磁盘的最大存储容量为2 TB

  • 存储系统最大20 PB（PB）

如果COMPATIBLE.ASM和COMPATIBLE.RDBMS磁盘组属性设置为12.1或更高，则在没有任何Oracle Exadata存储的情况下，Oracle ASM具有以下存储限制：

  • 每个Oracle ASM磁盘的4 PB最大存储空间，分配单元（AU）大小等于1 MB

  • 每个Oracle ASM磁盘的8 PB最大存储空间，AU大小等于2 MB

  • 每个Oracle ASM磁盘的16 PB最大存储空间，AU大小等于4 MB

  • 每个Oracle ASM磁盘的32 PB最大存储空间，AU大小等于8 MB

  • 存储系统最大320艾字节（EB）

对于所有Oracle Exadata存储，Oracle ASM具有以下存储限制：

  • 每个Oracle ASM磁盘的4 PB最大存储空间，AU大小等于1 MB

  • 每个Oracle ASM磁盘的8 PB最大存储空间，AU大小等于2 MB

  • 每个Oracle ASM磁盘的16 PB最大存储空间，AU大小等于4 MB

  • 每个Oracle ASM磁盘的32 PB最大存储空间，AU大小等于8 MB

  • 存储系统最大320 EB

磁盘组的最大大小限制等于最大磁盘大小乘以磁盘组中的最大磁盘数（10,000）。

所有磁盘组中的最大磁盘数为10,000。10,000个磁盘可以位于一个磁盘组中，也可以分布在最多511个磁盘组中。这是对Oracle ASM磁盘数量的限制，不一定是主轴数量。存储阵列可以将多个心轴组合到一个用作单个Oracle ASM磁盘的LUN中。

文件大小限制取决于磁盘组兼容性属性的值。当COMPATIBLE.RDBMS磁盘组属性设置为大于10.1时，Oracle ASM在任何冗余模式下都支持大于128 TB的文件大小。

如果COMPATIBLE.RDBMS设置为10.1 ，则文件大小限制较小。例如， COMPATIBLE.RDBMS等于10.1且AU大小等于1 MB，Oracle ASM文件大小限制为：

  • 外部冗余：16 TB

  • 正常冗余：5.8 TB

  • 高冗余：3.9 TB

注意：

Oracle数据库支持最大128 TB的数据文件大小，具体取决于文件系统。此外，Oracle数据库的文件大小限制取决于DB_BLOCK_SIZE初始化参数。

管理磁盘组中的容量

当Oracle ASM提供冗余时，例如创建具有NORMAL或HIGH冗余的磁盘组时，每个磁盘组中必须具有足够的容量来管理在一个或两个故障组发生故障后丢失的数据的重新创建。在一个或多个磁盘发生故障后，恢复所有数据的冗余的过程需要磁盘组中正常运行的磁盘的空间。如果剩余的空间不足，则某些文件最终可能会减少冗余。

冗余减少意味着文件中的一个或多个扩展区未在预期级别进行镜像。例如，高冗余磁盘组中的减少冗余文件具有至少一个文件范围，其具有两个或更少的范围的总副本而不是三个。对于不受保护的文件，数据范围可能完全丢失。冗余文件减少的其他原因是磁盘空间不足或故障组数量不足。

以下准则有助于确保在一个或多个磁盘发生故障后，您有足够的空间来恢复所有磁盘组数据的完全冗余。

  • 正常冗余磁盘组 - 最好在磁盘组中有足够的可用空间以容忍丢失一个故障组中的所有磁盘。可用空间量应等于最大故障组的大小。

  • 高冗余磁盘组 - 最好有足够的可用空间来应对两个故障组中所有磁盘的丢失。可用空间量应等于两个最大故障组的大小总和。

    注意：

    当您从多个故障组中丢失多个磁盘时，您可能会丢失数据的主副本和冗余副本。此外，如果您没有足够的容量来恢复冗余，那么Oracle ASM可以继续运行。但是，如果另一个磁盘发生故障，则系统可能无法容忍其他故障。

USABLE_FILE_MB的负值

由于FREE_MB ， REQUIRED_MIRROR_FREE_MB和USABLE_FILE_MB之间的关系， USABLE_FILE_MB可能变为负数。虽然这不一定是一个危急情况，但它确实意味着：

  • 根据FREE_MB的值，您可能无法创建新文件。

  • 下一次失败可能会导致冗余减少的文件。

如果USABLE_FILE_MB变为负数，强烈建议您尽快为磁盘组添加更多空间。

SQL方言

14.4更改程序声明

ALTER PROCEDURE语句显式重新编译一个独立的过程。

显式重新编译消除了对隐式运行时重新编译的需要，并防止了相关的运行时编译错误和性能开销。

要重新编译作为包的一部分的过程，请使用“ ALTER PACKAGE语句 ”重新编译整个包。

注意：

本声明不会更改现有过程的声明或定义。要重新声明或重新定义独立过程，请使用带有OR REPLACE子句的“ CREATE PROCEDURE语句 ” 。

ALTER PROCEDURE语句与ALTER FUNCTION语句非常相似。有关更多信息，请参见“ ALTER FUNCTION语句 ” 。

传输数据的一般限制

传输数据存在一般限制。还存在特定于完全可传输导出/导入，可传输表空间或可传输表的限制。

您计划传输数据时请注意以下一般限制：

  • 源数据库和目标数据库必须使用兼容的数据库字符集。具体而言，必须满足以下条件之一：

    • 源数据库和目标数据库的数据库字符集是相同的。

    • 源数据库字符集是目标数据库字符集的严格（二进制）子集，并且满足以下三个条件：

      • 源数据库是Oracle数据库10g第1版（10.1.0.3）或更高版本。

      • 要传输的表空间不包含具有字符长度语义的表列，或者源数据库字符集和目标数据库字符集中的最大字符宽度相同。

      • 要传输的数据不包含具有CLOB数据类型的列，或者源数据库字符集和目标数据库字符集都是单字节或两者都是多字节的。

    • 源数据库字符集是目标数据库字符集的严格（二进制）子集，并且满足以下两个条件：

      • 源数据库位于Oracle Database 10g第1版（10.1.0.3）之前。

      • 源数据库字符集和目标数据库字符集中的最大字符宽度相同。

    注意：

    Oracle数据库全球化支持指南中介绍了Oracle数据库识别的字符集之间的子集 - 超集关系。

  • 源数据库和目标数据库必须使用兼容的国家字符集。具体而言，必须满足以下条件之一：

    • 源数据库和目标数据库的国家字符集是相同的。

    • 源数据库是Oracle数据库10g第1版（10.1.0.3）或更高版本，要传输的表空间不包含具有NCHAR ， NVARCHAR2或NCLOB数据类型的列。

  • 运行可传输导出操作时，以下限制适用：

    • 执行导出的用户的默认表空间不能是正在传输的表空间之一。

    • 执行导出的用户的默认表空间必须是可写的。

  • 在非CDB中，您无法将表空间传输到包含同名表空间的目标数据库。

    在CDB中，您无法将表空间传输到包含同名表空间的目标容器。但是，不同的容器可以具有相同名称的表空间。

    您可以使用REMAP_TABLESPACE导入参数将数据库对象导入到其他表空间中。或者，在传输操作之前，您可以重命名要传输的表空间或目标表空间。

    从Oracle Database 12 c第2版（12.2）开始，Recovery Manager（RMAN） RECOVER命令可以在重新映射表空间时将表移动到其他模式。有关更多信息，请参见“ Oracle数据库备份和恢复用户指南” 。

  • 在CDB中，默认的数据泵目录对象DATA_PUMP_DIR不适用于PDB。您必须在PDB中使用Data Pump导出/导入定义显式目录对象。

  • 使用XMLTypes传输数据具有以下限制：

    • 目标数据库必须安装XML DB。

    • XMLType表引用的模式不能是XML DB标准模式。

    • 如果目标数据库中不存在传输的XMLType表的模式，则会导入并注册该模式。如果目标数据库中已存在架构，则在导入期间将显示一条消息。

    • 您必须仅使用Data Pump导出和导入包含XMLTypes的数据的元数据。

    以下查询返回包含XMLTypes的表空间列表：

    从dba_tablespaces p中选择distinct p.tablespace_name，dba_xml_tables x，dba_users u，all_all_tables t其中t.table_name = x.table_name和t.tablespace_name = p.tablespace_name和x.owner = u.username;

    有关XMLTypes的信息，请参阅Oracle XML DB Developer's Guide 。

  • 可以传输其解释是特定于应用程序且对数据库不透明的类型（例如RAW ， BFILE和AnyTypes），但它们不会作为跨平台传输操作的一部分进行转换。它们的实际结构仅为应用程序所知，因此应用程序必须在将这些类型移动到新平台后解决任何字节序问题。直接或间接使用这些不透明类型的类型和对象也受此限制。

控制阵列提取的重新阻塞

默认情况下，代理从非Oracle系统获取数据，直到它检索到足够的数据以发送回系统。它一直持续到从非Oracle系统获取的字节数等于或高于HS_RPC_FETCH_SIZE初始化参数的值。代理程序以HS_RPC_FETCH_SIZE初始化参数的值定义的大小重新锁定代理程序和Oracle数据库之间的数据。

什么时候非Oracle系统支持数组提取，您可以通过数组提取立即将从非Oracle系统获取的数据发送到Oracle数据库，而无需等到达到HS_RPC_FETCH_SIZE初始化参数的确切值。您可以将数据从非Oracle系统流式传输到Oracle数据库，并通过将HS_RPC_FETCH_REBLOCKING初始化参数的值设置为OFF来禁用重新HS_RPC_FETCH_REBLOCKING 。

例如，假设您将HS_RPC_FETCH_SIZE设置为64千字节（KB），将HS_FDS_FETCH_ROWS为100行。还假设每行大小约为600字节，因此100行大约为60 KB。当HS_RPC_FETCH_REBLOCKING初始化参数设置为ON ，代理程序开始从非Oracle系统获取100行。

由于代理中只有60 KB的数据，因此代理不会将数据发送回Oracle数据库。相反，代理从非Oracle系统中获取接下来的100行。由于代理中现在有120 KB的数据，因此可以将前64 KB的数据发送回Oracle数据库。

现在代理中剩下56 KB的数据。在将下一个64 KB数据发送到Oracle数据库之前，代理会从非Oracle系统中获取另外100行。通过将HS_RPC_FETCH_REBLOCKING初始化参数设置为OFF ，前100行立即被发送回Oracle数据库服务器。

自动升级作业概述

自动升级作业是与升级相关联的工作单元，由作业标识符（ jobid ）标识。

作业表示AutoUpgrade执行的一组操作。每项工作都经历一系列阶段以实现其目的。作业由唯一的正整数标识，称为jobid 。每个新的AutoUpgrade作业都会为AutoUpgrade实用程序的配置文件中的每个数据库生成一个新的作业ID（ jobid ）。如果AutoUpgrade检测到您的系统上存在的具有您之前启动的jobid的数据库不完整，则AutoUpgrade会将此现有jobid标识为恢复操作。在恢复操作中，由在前一次AutoUpgrade运行期间未完成的jobid标识的作业的阶段从它们被停止的点继续。

父主题：自动 升级消息和过程描述术语

自动升级阶段概述

AutoUpgrade实用程序作业通过一系列阶段（称为阶段） ，在此阶段执行特定操作。

在阶段期间发生的操作由您为自动升级选择的处理模式定义： 分析 ， 修复 ， 部署和升级 。

AutoUpgrade具有以下阶段：

  • SETUP ：AutoUpgrade实用程序作业管理器在启动作业的准备过程中创建的初始阶段。
  • PREUPGRADE ：AutoUpgrade根据您当前的系统配置执行系统检查以确定其升级准备情况的阶段，例如检查以确定您是否有足够的可用磁盘空间。
  • PRECHECKS ：AutoUpgrade分析源Oracle数据库以确定数据库是否满足升级要求的阶段。
  • PREFIXUPS ：AutoUpgrade在开始升级之前执行预升级修正的阶段。例如，这是AutoUpgrade在源Oracle数据库上收集字典统计信息的阶段。
  • DRAIN ：AutoUpgrade从源Oracle数据库实例中排出数据库会话的阶段，为准备关闭侦听器和数据库做准备。此阶段适用于Windows系统以及Linux和Unix系统。
  • DBUPGRADE ：AutoUpgrade执行升级的阶段，并编译升级完成后找到的所有无效对象。
  • POSTCHECKS ：AutoUpgrade在启动postupgrade修正之前对目标Oracle数据库（升级的Oracle数据库）执行检查的阶段。
  • POSTFIXUPS ：AutoUpgrade执行升级后修正的处理阶段，例如升级时区。
  • POSTUPGRADE ：AutoUpgrade将源Oracle数据库配置文件移动到目标Oracle数据库配置的阶段，并在新的Oracle主目录中启动升级的Oracle数据库。

AutoUpgrade阶段操作和状态概述

要识别阶段消息的状态，请查看操作和状态的消息说明。

了解操作消息

操作是描述AutoUpgrade状态期间发生的事情的内部阶段消息。有两个操作消息。

准备 ：正在创建，初始化或调用AutoUpgrade实例，以准备完成AutoUpgrade阶段。这是一条信息消息。您无需执行任何操作。

执行 ：AutoUpgrade正在执行阶段的主要工作流程。这是一条信息消息。您无需执行任何操作。

状态指示显示消息的阶段的当前工作流的状态。有四种状态消息：

  • 已中止 ：AutoUpgrade根据用户请求停止执行阶段工作流程。

  • 错误 ：执行阶段工作流时遇到错误。查看错误原因。

  • 表面处理 ：自动升级成功地完成了阶段的工作流程。

  • RUNNING ：AutoUpgrade正在执行舞台工作流程。
