2.1.1嵌入式SQL语句

术语嵌入式SQL是指放置在应用程序中的SQL语句。因为它包含SQL语句，所以应用程序被称为宿主程序 ，其编写语言称为宿主语言 。例如，Pro * C / C ++提供了在C或C ++主机程序中嵌入某些SQL语句的功能。

要操作和查询Oracle数据，请使用INSERT，UPDATE，DELETE和SELECT语句。INSERT向数据库表添加数据行，UPDATE修改行，DELETE删除不需要的行，SELECT检索符合搜索条件的行。

功能强大的SET ROLE语句允许您动态管理数据库权限。角色是相关系统和对象特权的命名组，或授予用户或其他角色的相关系统或对象特权的命名组。角色定义存储在Oracle数据字典中。您的应用程序可以使用SET ROLE语句根据需要启用和禁用角色。

只有SQL语句 - 而不是SQL * Plus语句 - 在应用程序中才有效。（SQL * Plus具有用于设置环境参数，编辑和报告格式的其他语句。）

2.1.1.1可执行语句和指令

嵌入式SQL包括所有交互式SQL语句以及允许您在Oracle和宿主程序之间传输数据的其他语句。嵌入式SQL语句有两种类型： 可执行语句和指令 。可执行语句导致对运行时库SQLLIB的调用。您可以使用它们连接到Oracle，定义，查询和操作Oracle数据，控制对Oracle数据的访问以及处理事务。它们可以放在可以放置C或C ++语言可执行语句的任何位置。

另一方面，指令不会导致对SQLLIB的调用，也不会对Oracle数据进行操作。您可以使用它们来声明Oracle对象，通信区域和SQL变量。它们可以放在可以放置C或C ++变量声明的任何位置。

2.1.2嵌入式SQL语法

在您的应用程序中，您可以自由地将完整的SQL语句与完整的C语句混合，并在SQL语句中使用C变量或结构。将SQL语句构建到宿主程序中的唯一特殊要求是，您可以使用关键字EXEC SQL开始它们，并以分号结束它们。Pro * C / C ++将所有EXEC SQL语句转换为对运行时库SQLLIB的调用。

许多嵌入式SQL语句与其交互式对应语句的区别仅在于添加了新子句或使用了程序变量。以下示例比较了交互式和嵌入式ROLLBACK语句： 

ROLLBACK WORK： - 交互式EXEC SQL ROLLBACK WORK; - 嵌入式

这些语句具有相同的效果，但您可以在交互式SQL环境中使用第一个语句（例如在运行SQL * Plus时），在Pro * C / C ++程序中使用第二个语句。

2.1.3静态与动态SQL语句

大多数应用程序旨在处理静态SQL语句和固定事务。在这种情况下，您可以在运行时知道每个SQL语句和事务的组成;也就是说，您知道将发出哪些SQL命令，哪些数据库表可能会更改，哪些列将更新，等等。

但是，某些应用程序可能需要在运行时接受和处理任何有效的SQL语句。因此，您可能直到运行时才知道所涉及的所有SQL命令，数据库表和列。

动态SQL是一种高级编程技术，它允许程序在运行时接受或构建SQL语句，并对数据类型转换进行显式控制。

2.1.4嵌入式PL / SQL块

Pro * C / C ++将PL / SQL块视为单个嵌入式SQL语句。您可以将PL / SQL块放在可以放置SQL语句的应用程序中的任何位置。要在主机程序中嵌入PL / SQL，只需声明要与PL / SQL共享的变量，并将PL / SQL块与关键字EXEC SQL EXECUTE和END-EXEC括起来。

从嵌入式PL / SQL块，您可以灵活，安全地操作Oracle数据，因为PL / SQL支持所有SQL数据操作和事务处理命令。

2.1.5主机和指标变量

主机变量是Oracle与程序之间通信的关键。主变量是在C中声明并与Oracle共享的标量或聚合变量，这意味着您的程序和Oracle都可以引用其值。

您的程序使用输入主机变量将数据传递给Oracle。Oracle使用输出主机变量将数据和状态信息传递给您的程序。程序为输入主变量赋值; Oracle将值分配给输出主机变量。

主机变量可以在任何可以使用SQL表达式的地方使用。在SQL语句中，主机变量必须以冒号（:)为前缀，以使它们与SQL关键字区分开来。

您还可以使用C 结构来包含许多主变量。当您在嵌入式SQL语句中命名结构（以冒号为前缀）时，Oracle将结构的每个组件用作主机变量。

您可以将任何主机变量与可选的指示符变量相关联。指示符变量是一个短整数变量，用于“指示”其主变量的值或条件。您可以使用指示符变量将NULL分配给输入主变量，并检测输出主变量中的NULL或截断值。NULL是缺失，未知或不适用的值。

在SQL语句中，指示符变量必须以冒号为前缀，并紧跟其关联的主变量。关键字INDICATOR可以放在主变量和其指示符之间，以进一步清晰。

如果主变量打包在结构中，并且您想要使用指示符变量，则只需创建一个结构，该结构具有主机结构中每个主机变量的指示符变量，并在SQL语句中命名指示符结构，紧跟在宿主变量struct，以冒号为前缀。您还可以使用INDICATOR关键字分隔主机结构及其关联的指标结构。

2.1.6 Oracle数据类型

通常，主机程序将数据输入Oracle，Oracle将数据输出到程序。Oracle将输入数据存储在数据库表中，并将输出数据存储在程序主机变量中。要存储数据项，Oracle必须知道其数据类型 ，该数据类型指定存储格式和有效值范围。

Oracle识别两种数据类型： 内部和外部 。内部数据类型指定Oracle如何在数据库列中存储数据。Oracle还使用内部数据类型来表示数据库伪列，它返回特定数据项但不是表中的实际列。

外部数据类型指定数据在主机变量中的存储方式。当主机程序向Oracle输入数据时，如有必要，Oracle会在输入主机变量的外部数据类型与目标数据库列的内部数据类型之间进行转换。当Oracle将数据输出到主机程序时，如有必要，Oracle会在源数据库列的内部数据类型和输出主机变量的外部数据类型之间进行转换。

2.1.7数组

Pro * C / C ++允许您定义数组主机变量（称为主机数组）和结构数组，并使用单个SQL语句对它们进行操作。使用数组SELECT，FETCH，DELETE，INSERT和UPDATE语句，您可以轻松查询和操作大量数据。您还可以在宿主变量struct中使用宿主数组。

2.1.8数据类型等价

Pro * C / C ++通过让您使用等价数据类型为您的应用程序增加了灵活性。这意味着您可以自定义Oracle解释输入数据和格式化输出数据的方式。

在逐个变量的基础上，您可以将支持的C数据类型等效于Oracle外部数据类型。您还可以将用户定义的数据类型等同于Oracle外部数据类型。

2.1.9私有SQL区域，游标和活动集

为了处理SQL语句，Oracle打开了一个名为私有SQL区域的工作区 。私有SQL区域存储执行SQL语句所需的信息。称为游标的标识符允许您命名SQL语句，访问其私有SQL区域中的信息，并在某种程度上控制其处理。

对于静态SQL语句，有两种类型的游标： 隐式和显式 。Oracle隐式声明所有数据定义和数据操作语句的游标，包括只返回一行的SELECT语句（查询）。但是，对于返回多行的查询，要处理超出第一行，必须显式声明游标（或使用主机数组）。

返回的行集称为活动集 ;它的大小取决于符合查询搜索条件的行数。您使用显式游标来标识当前正在处理的行，称为当前行 。

想象一下，这组行被返回到终端屏幕。屏幕光标可以指向要处理的第一行，然后指向下一行，依此类推。同样，显式游标“指向”活动集中的当前行。这允许您的程序一次处理一行。

2.1.10交易

事务是一系列逻辑上相关的SQL语句（例如，两个UPDATE，它们将一个银行帐户记入贷方并借记另一个银行帐户），Oracle将其视为一个单元，以便语句带来的所有更改都是永久性的或同时撤消的。

自上次执行数据定义，COMMIT或ROLLBACK语句以来执行的所有数据操作语句构成当前事务。

为了帮助确保数据库的一致性，Pro * C / C ++允许您使用COMMIT，ROLLBACK和SAVEPOINT语句定义事务。

COMMIT使当前事务期间所做的任何更改成为永久性的。ROLLBACK结束当前事务并撤消自事务开始以来所做的任何更改。SAVEPOINT标志着交易处理的当前点;与ROLLBACK一起使用，它撤消了部分交易。

2.1.11错误和警告

执行嵌入式SQL语句时，它会成功或失败，并可能导致错误或警告。您需要一种方法来处理这些结果。Pro * C / C ++提供了两种错误处理机制：SQL通信区域（SQLCA）和WHENEVER语句。

SQLCA是您在主机程序中包含（或硬编码）的数据结构。它定义了Oracle用于将运行时状态信息传递给程序的程序变量。使用SQLCA，您可以根据Oracle对刚尝试的工作的反馈采取不同的操作。例如，您可以检查DELETE语句是否成功，如果是，则删除了多少行。

使用WHENEVER语句，您可以指定Oracle检测到错误或警告条件时要自动执行的操作。这些操作是：继续下一个语句，调用函数，分支到带标签的语句或停止。

2.1.12 SQL99语法支持

SQL标准支持SQL应用程序在所有符合标准的软件产品中的可移植性。Oracle功能符合ANSI / ISO SQL99标准，包括符合ANSI标准的连接。Pro * C / C ++支持Oracle数据库支持的所有SQL99功能，这意味着支持SELECT，INSERT，DELETE和UPDATE语句的SQL99语法以及DECLARE CURSOR语句中的游标主体。

2.5.1指南

以下指南将帮助您避免一些常见问题。

2.5.1.1引用游标

游标名称是SQL标识符，其范围是预编译单元。因此，游标操作不能跨越预编译单元（文件）。也就是说，你不能在一个文件中DECLARE游标，而在另一个文件中将它从OPEN或FETCH中删除。因此，在进行单独的预编译时，请确保对给定游标的所有定义和引用都在一个文件中。

2.5.1.3使用单个SQLCA

如果只想使用一个SQLCA，则必须在其中一个程序模块中将其声明为全局，而在其他模块中将其声明为外部。使用extern 存储类，以及您的代码中的以下定义：

#define SQLCA_STORAGE_CLASS extern

注意：

应用程序中的所有源文件必须唯一命名，否则将生成错误。

3.2使用Directory Server进行集中管理

3.2.1.1域组件命名空间的完全限定名称

对于域组件名称空间，为客户端定义的默认目录条目必须采用以下格式之一：

DC [特区] [...]OU，DC [特区] [...]

在前面的语法中，[dc]表示可选的域组件，[...]表示其他域组件条目，ou表示组织单元条目。

客户端在连接标识符中使用的完全限定名称必须采用以下格式之一：

cn.dc [.DC] [...]CN [.ou] @dc [.DC] [...]

在前面的语法中，[cn]表示Oracle Net条目。

示例3-1使用具有组织单位的完全限定名称

考虑一个目录服务器，其中包含数据库对象销售条目，其DN为cn=sales,cn=OracleContext,dc=jp,dc=example,dc=com 。在此示例中，客户端需要sales.jp.example.com的连接标识符。

考虑一个包含数据库对象销售的类似条目，其DN为cn=sales,cn=OracleContext, ou=mktg ,dc=jp,dc=example,dc=com 。

由于域组件必须与组织单元分开，因此客户端必须使用cn.ou@dc.dc.dc格式。客户端要求连接标识符为sales.mktg@jp.example.com 。

3.2.6.1性能考虑因素

连接标识符存储在目录服务器中，供所有客户端访问。根据客户端的数量，目录服务器上可能会有很大的负载。

您可以通过更改网络拓扑或实施复制来解决性能问题。

3.2.6.2.1认证方法

客户端使用不同的身份验证方法，具体取决于要执行的任务。

  • 对目录服务器中的信息执行查找的客户端通常使用匿名身份验证。在Oracle Database 12c中 ，可以将客户端配置为在名称查找期间对其LDAP绑定进行身份验证。需要保护其网络服务数据或禁用匿名绑定到目录的站点必须将其客户端配置为在名称查找期间使用钱包进行身份验证。这些客户端需要在sqlnet.ora文件中设置以下两个参数：

    names.ldap_authenticate_bind = TRUE wallet_location = location_value

  • 管理目录服务器条目的客户端使用目录服务器进行身份验证。Database Configuration Assistant或Oracle Net Manager可用于添加或修改条目。只有具有适当权限的已认证用户才能修改条目。使用以下身份验证方法之一：

    • 简单认证

      客户端通过DN和密码向目录服务器标识自己。服务器验证客户端发送的DN和密码是否与目录服务器中存储的DN和密码匹配。

    • 强认证

      客户端通过安全套接字层（SSL）提供的公钥加密向目录服务器标识自己。在公钥加密中，消息的发送者使用接收者的公钥加密消息。在传递时，接收者使用接收者的私钥解密该消息。

GOST定义了256位的密钥大小。在Oracle数据库中，使用外部密码块链接（CBC）模式。

种子

Oracle数据库支持韩国信息安全局（KISA）加密算法SEED。

SEED定义了128位的密钥大小。标准的扩展定义了192位和256位的其他密钥大小，但Oracle数据库不支持这些扩展。在Oracle数据库中，SEED以外部密码块链接（CBC）模式运行。

三重DES支持

Oracle数据库支持三重DES加密（3DES），它通过三次DES算法加密消息数据。

3DES提供了高度的消息安全性，但性能下降。性能损失的大小取决于执行加密的处理器的速度。与标准DES算法相比，3DES加密数据块通常需要三倍的时间。

3DES有两键和三键版本，有效密钥长度分别为112位和168位。两个版本都在外部密码块链接（CBC）模式下运行。

Oracle数据库和安全网络服务提供的DES40算法是DES的变体，其中秘密密钥被预处理以提供40个有效密钥位。它旨在为美国和加拿大以外的客户提供基于DES的加密，此时美国的出口法律更具限制性。目前DES40，DES和3DES都可以出口。仍然支持DES40为国际客户提供向后兼容性。

基于Diffie-Hellman的密钥协商

您可以使用Diffie-Hellman密钥协商算法来保护多用户环境中的数据。

在多用户环境中，安全密钥分发很困难。Oracle数据库使用众所周知的Diffie-Hellman密钥协商算法来执行加密和数据完整性的安全密钥分发。

使用加密来保护加密数据的安全性时，必须经常更改密钥以最大限度地减少受损密钥的影响。因此，Oracle数据库密钥管理功能会在每个会话中更改会话密钥。

您可以使用Authentication Key Fold-in来解决Diffie-Hellman密钥协商算法密钥协商中可能存在的第三方攻击（历史上称为中间人攻击 ）。它通过将仅由客户端和服务器知道的共享秘密与由Diffie-Hellman协商的原始会话密钥相结合，显着增强了会话密钥。

客户端和服务器使用Diffie-Hellman生成的会话密钥开始通信。当客户端向服务器进行身份验证时，它们会建立一个只有双方都知道的共享密钥。Oracle数据库结合了共享密钥和Diffie-Hellman会话密钥，以生成更强大的会话密钥，旨在抵御中间人攻击。

身份验证密钥折叠功能是Oracle数据库的嵌入功能，无需系统或网络管理员进行配置。

关于同时为不同用户启用Oracle本机加密和SSL身份验证

                        

                           
默认情况下，Oracle数据库不允许同时为不同用户提供Oracle本机加密和安全套接字层（SSL）身份验证。

                           
同时使用Oracle本机加密（也称为高级网络选项（ANO）加密）和SSL身份验证称为双重加密。假设客户端将SQLNET.ENCRYPTION_CLIENT参数设置为required ，并且服务器将SQLNET.ENCRYPTION_SERVER参数设置为required 。如果使用TCPS侦听器，则拒绝连接并且ORA-12696: Double Encryption Turned On, login disallowed将返回ORA-12696: Double Encryption Turned On, login disallowed错误。
                           

                           
但是，在某些情况下，必须同时配置TCP和TCPS侦听器，以便某些用户可以使用用户名和密码连接到服务器，而其他用户可以使用SSL证书对服务器进行验证。在这些情况下，您必须配置基于密码的身份验证和SSL身份验证。在以前的版本中一种解决方法是设置SQLNET.ENCRYPTION_SERVER参数requested 。如果您的要求是将SQLNET.ENCRYPTION_SERVER设置为required ，则可以将SQLNET.ENCRYPTION_CLIENT和SQLNET.ENCRYPTION_SERVER的IGNORE_ANO_ENCRYPTION_FOR_TCPS参数设置为TRUE 。默认情况下，它设置为FALSE 。

                           
将IGNORE_ANO_ENCRYPTION_FOR_TCPS设置为TRUE强制客户端忽略为所有传出TCPS连接的SQLNET.ENCRYPTION_CLIENT参数设置的值。在配置TCPS和ANO时，不是返回ORA-12696错误， SQLNET.ENCRYPTION_SERVER当使用TCPS客户端和这两个参数之间存在冲突时，此参数允许数据库忽略SQLNET.ENCRYPTION_CLIENT或SQLNET.ENCRYPTION_SERVER设置。设置为required 。

要编辑特定数据库表或视图列中的数据，必须创建Oracle数据编辑策略。使用Oracle Data Redaction策略指定的编校格式编辑数据。要编辑数据，您可以使用任何Oracle提供的编校格式，或创建和使用自定义编校格式。如果包含要编辑的数据的表或视图列映射到敏感列类型，Oracle将使用该映射为数据推荐合适的编校格式。因此，Oracle Data Redaction策略封装了数据库模式，数据库表和视图列，敏感列类型和Oracle Data Redaction格式。

使用Enterprise Manager创建Oracle数据编校策略

                  

                     
您可以使用Enterprise Manager Cloud Control创建Oracle Data Redaction策略。

                     
    
                        
  1. 以用户SYSTEM或SYSMAN身份登录Oracle Enterprise Manager云控制。
    URL如下：
    
                           
     https:// host : port /em 
                           
    
                        
    2. 
                        
  2. 从“ 目标”菜单中，选择“ 数据库” 。
    3. 
                        
  3. 选择“ 搜索列表” ，然后单击要为其创建Oracle Data Redaction策略的数据库目标的名称。
    4. 
                        
  4. 在数据库目标的主页上，从“ 安全性”菜单中选择“ 数据编辑” 。
    5. 
                        
  5. 如果系统提示您，请登录数据库。
    确保以具有DBMS_REDACT PL / SQL包的EXECUTE特权的用户身份登录数据库。
                           
    
                        
    6. 
                        
  6. 在“策略”选项卡的“策略”部分中，选择“ 创建” 。
    
                              
    如果这是您第一次创建数据编辑策略，则会出现数据编辑：设置以启用列敏感类型发现对话框。此功能允许对数据编辑策略使用列敏感类型发现。
    
                              
    为此，Enterprise Manager在DBSNMP架构中创建GET_COL_DATA_SENSITIVE_TYPES过程。要在创建Oracle数据编校策略时对选定列执行敏感类型发现，用户必须具有DBSNMP.GET_COL_DATA_SENSITIVE_TYPES过程的EXECUTE特权。如果数据库受Oracle Database Vault保护，则确保必须创建数据编辑策略的任何用户都是保护DBSNMP架构的域的DBSNMP 。
                              
    
                           
    
                        
    7. 
                        
  7. 如果出现“数据编辑：设置为启用列敏感类型发现”对话框，并且当前登录用户没有正确的要求，请选择对DBSNMP.GET_COL_DATA_SENSITIVE_TYPES具有EXECUTE特权的用户的凭据。然后单击OK 。
    8. 
                        
  8. 在“创建数据编辑策略”页面上，输入以下信息：
      
                              
    • 
                                 
      架构 ：输入（或搜索）包含要编辑的数据的架构的名称。
                                 
      
                              
      • 
                              
    • 
                                 
      表/视图 ：输入（或搜索）包含要编辑的列的表或字段。
                                 
      
                              
      • 
                              
    • 
                                 
      策略名称 ：为策略输入a，例如emp_wages_pol 。
                                 
      
                              
      • 
                              
    • 
                                 
      默认表达式 ：输入默认表达式。默认设置为1 = 1，这意味着将始终强制执行策略。如果您不熟悉策略表达式的组件，请单击“ 策略表达式”字段旁边的铅笔图标以使用“策略表达式构建器”。选择“ 策略”时 ，选择所需条件，然后单击“ 添加” 。如果要手动编辑策略表达式，请单击“ 编辑” 。构建所需的策略表达式后，单击“ 确定” 。策略表达式生成器如下所示：

      插图说明em_red_pol_exp_bldr.png
      • 
                           
    
                        
    9. 
                        
  9. 在“对象列”部分中，单击“ 添加”以将表或视图列添加到编校策略。
    
                              
    出现类似于以下内容的对话框：

    em_red_pol_attr.png插图的描述
    编校策略仅应用于添加到其中的表或视图列。
    
                           
    
                        
    10. 
                        
  10. 从“ 列”菜单中，选择要应用编校策略的表或视图列。
    “ 列”菜单右侧是一个图标，您可以单击该图标以查看所选列的内容。
                           
    
                           
    如果列包含敏感数据并已映射到敏感列类型，则从“ 敏感列类型”菜单中选择已映射到的敏感列类型。如果“ 敏感列类型”菜单中的搜索模式匹配，则默认情况下会选择敏感列类型。例如，对于列出信用卡号的列，如果匹配，则菜单将列出Undefined和CREDIT_CARD_TYPE 。如果未创建敏感列类型，则默认的“ 敏感列类型”菜单列表仅为“ 未定义” 。
                           
    
                        
    11. 
                        
  11. 从“ 编辑格式”菜单中，选择要使用的编校格式。
    下拉列表中填充了Oracle数据库提供的编校格式，以及您创建和保存的自定义编校格式。
                              
    如果您不想使用预定义的编校格式（即Oracle数据库提供的编校格式或您创建的自定义编校格式），而是希望在创建编校策略时指定编校细节，选择CUSTOM for Redaction Format 。
                              
    
                              
    “添加”对话框会调整以适应您选择的编校格式和功能的类型。例如，如果选择CUSTOM编校格式和REGEX编校功能，则会在对话框中显示“功能属性”区域。
                              
    
                           
    
                        
    12. 
                        
  12. 从“ 编辑功能”菜单中，选择要用于编辑列数据的功能。
    如果要编辑整个列数据，请选择FULL;如果要仅编辑部分列数据，请选择PARTIAL;如果要根据正则表达式编辑列数据，请选择REGEX;如果要编辑列数据，请选择RANDOM以随机方式，使用随机生成的值，如果您只想测试编校策略的定义，则为NONE ，而不是编辑任何列数据。请注意，所有编校功能可能不适用于特定的编校格式。下拉列表仅显示适用于所选编校格式的编校功能。
                           
    
                           
    如果在上一步中选择了CUSTOM作为编校格式 ，而在编辑功能中选择了 PARTIAL或REGEX ，请确保指定了功能属性。
                           
    
                        
    13. 
                        
  13. 单击确定 。
    14. 
                        
  14. 对于要添加到编校策略的所有列，从步骤8开始重复这些步骤。
    15. 
                        
  15. 在“创建数据编校策略”页面上，单击“ 确定”以创建数据编校策略。
    
                              
    创建Oracle Data Redaction策略时，默认情况下会启用该策略。
    
                           
    
                        
    16.

7为XML C API打包SAX

         

         

            

               
SAX是基于事件的XML解析的标准接口，由XML-DEV邮件列表的成员协作开发。要使用SAX， xmlsaxcb结构将使用函数指针进行初始化，并传递给其中一个XmlLoadSax调用。还提供了指向用户定义的上下文结构的指针，并将其传递给每个SAX函数。
               

               
对于基于事件的模式验证API，请参阅C的Package Event API 。

12.1数据库内存和Oracle RAC概述

               

                  
每个Oracle RAC节点都有自己的内存（IM）列存储。默认情况下，填充的对象分布在群集中的所有IM列存储中。

                  
Oracle建议您在每个Oracle RAC节点上均等地调整IM列存储的大小。如果Oracle RAC节点不需要IM列存储，则将INMEMORY_SIZE参数设置为0 。
                  

                  
可以在每个节点上填充完全不同的对象，或者在集群中的所有IM列存储上分布更大的对象。在Oracle集成系统上，同一对象也可以出现在每个节点上的IM列存储中。跨集群中IM列存储的对象分布由INMEMORY属性的两个子句控制： DISTRIBUTE和DUPLICATE 。

                  
在Oracle RAC环境中，仅指定了INMEMORY属性的对象将自动分布在群集中的IM列存储中。您可以使用DISTRIBUTE子句指定对象在群集中的分布方式。默认情况下，使用的分区类型（如果有）确定对象的分布方式。如果对象未分区，则按rowid范围分发。或者，您可以指定DISTRIBUTE子句以覆盖默认行为。
                  

                  
在Oracle集成系统上，您可以跨群集中的IM列存储复制或镜像填充的对象。该技术提供最高级别的冗余。DUPLICATE子句控制对象的复制方式。如果仅指定DUPLICATE ，则数据的一个镜像副本将分布在群集中的IM列存储中。要复制每个IM列存储中的整个对象，请指定DUPLICATE ALL 。

                  

                     
注意：

                     
在非工程系统上部署Oracle RAC时， DUPLICATE子句将被视为NO DUPLICATE 。

                  

               

               

                  
    
                     
  • 多个IM列存储
    在Oracle RAC中，每个数据库实例都有自己的IM列存储。
                     
    • 
                     
  • Oracle RAC中列式数据的分发和复制
    指定INMEMORY ， DISTRIBUTE和DUPLICATE关键字控制对象的分布。
                     
    • 
                     
  • Oracle RAC中的并行性
    数据库实例必须访问它们所在的IM列存储中的IMCU。Oracle RAC中IM列存储的填充和访问必须并行发生，以便可以从任何实例访问所有IM列存储。
                     
    • 
                     
  • Oracle RAC中的FastStart区域
    FastStart区域在所有Oracle RAC节点之间共享。此功能可实现群集中的最大共享和可重用性。

12.1.2.1 Oracle RAC中列式数据的分布

设置默认选项AUTO ，Oracle RAC实例会自动分发数据。在填充段时，空间管理从属进程（W nnn ）进程尝试在每个实例上放置相同数量的数据。分布取决于访问模式和对象大小。或者，您可以手动指定数据库必须如何跨实例分发分区，子分区或rowid范围。
                        

                        
平等的数据分配对性能很重要。目标是并行查询过程在相同的数据集大小上工作，以便它们都在最短的时间内完成。如果数据分布有偏差，则长时间运行的进程会延迟查询的完成。

DBA_TABLES.INMEMORY_DISTRIBUTE列指示IMCU的分发方式。设置AUTO选项后，列值为AUTO-DISTRIBUTE 。

12.1.2.2.2在Oracle RAC中重复所有子句

                        

                           
DUPLICATE ALL子句指定在每个数据库实例上镜像每个In-Memory对象。
                           

                           
此设置提供最高级别的冗余并提供线性可伸缩性，因为查询可以在单个节点内完全执行。例如， sales表的每个IMCU都填充在实例1，实例2和实例3中的IM列存储中。因此，任何数据库实例都可以检索sales查询所请求的数据。
                           

                           
DUPLICATE ALL子句的结果是DISTRIBUTE子条款没有应用程序，因为该对象的所有IMCU都是分布式的。您在对象级别指定复制，这意味着IM列存储中的所有对象都不需要DUPLICATE ALL子句。
                           

                           
DUPLICATE ALL技术的主要优点是：

                           
    
                              
  • 
                                 
    高可用性
    
                                 
    对所有内存中对象使用DUPLICATE ALL子句时，具有n个实例的Oracle RAC数据库可以承受n -Oracle RAC实例故障。如果需要将一个数据库实例停止服务以进行维护，则至少有一个IM列存储中可以使用关键数据。所有数据都不可访问的唯一情况是群集中所有数据库实例都出现故障。
                                 
    
                              
    • 
                              
  • 
                                 
    明星查询的表现
    
                                 
    如果查询将较小的维度表连接到较大的分区事实表，则可以使用DUPLICATE ALL在每个Oracle RAC实例中镜像维度表。事实表由分区或子分区分发。在此策略中，每个数据库实例中的IM列存储都具有星型连接所需的数据。此技术类似于分区连接，因为整个维度表都填充在每个IM列存储中。

12.1.3.1 Oracle RAC中的串行和并行查询

                     

                        
Oracle RAC中的数据库内存是一种无共享架构。除非每个活动实例上至少运行一个并行服务器进程，否则无法保证查询从IM列存储中访问所有必需的数据。

                        

                           
串行查询

                           
在Oracle RAC数据库中的一个节点上运行的串行查询无法访问其他IM列存储中的IMCU。例如，在实例1上运行的串行查询请求sales的完整扫描。某些sales分区填充在实例1中的IM列存储中，而其他sales分区则填充在实例2中的IM列存储中。查询只能访问实例1上IM列存储中的IMCU：其余数据必须来自磁盘存储。
                           

                        

                        
                        

                           
并行查询

                           
启用并行执行但PARALLEL_DEGREE_POLICY初始化参数未设置为AUTO ，情况类似于串行查询情况。查询协调器在执行查询的数据库实例上运行。PQ进程将数据发送给协调器。在这种情况下，数据库启动多个PQ进程。但是，除非DOP大于或等于包含为查询中引用的对象填充的IMCU的IM列存储的数量，否则不能从IM列存储中访问所有数据。如果未启用“自动DOP”，请确保DOP至少与使用IMCU的IM列存储在查询中的填充对象一样大。
                           

                        

                        
                        

                           
内存中动态扫描

                           
串行和并行查询都可以执行内存中动态扫描（ IM动态扫描 ）并使用轻量级线程基础结构。并行执行基础结构与新的线程基础结构共存，该基础结构由Oracle数据库资源管理器（资源管理器）动态管理。当INMEMORY_SIZE大于0时，默认情况下启用资源管理器。
                           

                           
表扫描过程可以是串行查询中的前台进程，也可以是并行查询中的并行服务器进程。当并行查询执行IM动态扫描时，每个表扫描进程都可以拥有一个线程池。

12.1.3.2 Oracle RAC中的自动DOP

                     

                        
通过自动并行度（Auto DOP），优化器执行基于成本的计算，以确定SQL语句的并行度。

                        
通过将PARALLEL_DEGREE_POLICY初始化参数设置为AUTO来启用自动DOP。当优化器解析SQL语句时，它会估计执行时间。它根据PARALLEL_MIN_TIME_THRESHOLD初始化参数的设置检查此估计值，该参数在启用IM列存储时自动设置。然后，优化器会做出以下基于成本的决策：

                        
    
                           
  • 
                              
    如果估计时间小于PARALLEL_MIN_TIME_THRESHOLD ，则语句将按顺序执行。
                              
    
                           
    • 
                           
  • 
                              
    如果估计的时间大于PARALLEL_MIN_TIME_THRESHOLD ，则语句并行执行。
                              
    
                              
    优化器根据资源需求计算并行度。计算受PARALLEL_DEGREE_LIMIT初始化参数的限制，如果已配置，则受数据库资源管理器的限制。
                              
    
                           
    • 
                        

                        
在Oracle RAC环境中使用IM列存储时，目标是避免磁盘或缓冲区缓存访问。为此，您必须保证每个活动数据库实例上至少运行一个并行服务器进程。Auto DOP是实现此目标的推荐方式。
                        

                        

                           
注意：

                           
如果不使用自动DOP，则必须确保DOP大于或等于包含查询所需的IMCU的IM列存储的数量。

                        

                        
Auto DOP保证了足够的进程分配，因为每个共享池都存储元数据，指示所有IMCU的位置，它们的大小等等。每个共享池中都有相同的映射。无论查询源自集群，并行查询协调器都知道IMCU的归属位置 （驻留的实例）。
                        

                        
例如，PQ协调器知道2016的销售分区在实例1中，而2015的分区在实例2中。如果在实例1上运行的查询同时请求2015和2016分区，则查询协调器使用归属位置来确定要访问的IM列存储。如果DOP设置得足够高，则协调器会自动启动两个实例上的PQ进程，并且进程将请求的数据发送回查询协调器。

12.1.4 Oracle RAC中的FastStart区域

                  

                     
FastStart区域在所有Oracle RAC节点之间共享。此功能可实现群集中的最大共享和可重用性。

                     
只有一个IMCU副本位于FastStart区域。例如，如果为四节点群集中的对象指定了DUPLICATE ALL ，则IM列列存储中将存在该对象的四个副本。但是，数据库仅将一个副本保存到FastStart区域。
                     

                     
Oracle RAC集群中的任何数据库实例都可以使用FastStart区域中的IMCU。此功能可提高Oracle RAC环境中实例重新启动的性能。

                     
例如， sales表可能有三个分区： sales_2014 ， sales_2015和sales_2016 ，每个分区都填充在不同的实例中。发生实例故障，一个实例无法重新启动。如果IM列存储中有足够的可用空间，则幸存的实例可以读取先前在不可访问的实例中填充的IMCU。因此，所有三个sales表分区都可供应用程序使用。

12.2.2对象级服务控制

                  

                     
对于单个对象， INMEMORY ...DISTRIBUTE子句有一个FOR SERVICE子句，它将群体限制为可以运行此服务的数据库实例。
                     

                     
PARALLEL_INSTANCE_GROUP初始化参数控制服务级别的段，其中服务表示一个或多个实例。相比之下， INMEMORY ...DISTRIBUTE FOR SERVICE控制分段级别的分配。例如，您可以将INMEMORY对象配置为仅在实例1上的IM列存储中填充，或仅在实例2上或在两个实例中填充。
                     

                     
DISTRIBUTE FOR SERVICE选项包括：

                     
    
                        
  • 
                           
    DEFAULT - 如果设置了PARALLEL_INSTANCE_GROUP ，则会在PARALLEL_INSTANCE_GROUP指定了IM列存储的所有数据库实例中填充该对象。如果未设置PARALLEL_INSTANCE_GROUP ，则会在具有IM列存储的所有实例中填充该对象。
                           
    
                           
    指定FOR SERVICE等同于指定FOR SERVICE DEFAULT 。
    
                        
    • 
                        
  • 
                           
    ALL - 数据库填充具有IM列存储的所有实例中的对象。
                           
    
                           
    
                              
    注意：
    
                              
    如果未设置PARALLEL_INSTANCE_GROUP ，则DEFAULT和ALL在功能上是等效的。
                              
    
                           
    
                        
    • 
                        
  • 
                           
    service_name - 作为其职责的一部分，IMCO触发从分配给先前服务的数据库实例中删除对象，并将其填充到分配给新服务的实例中。
                           
    
                           
    重新分发段时，数据库会执行必要的最少工作。例如，服务dbmperf被分配给实例1和实例2。sales分区在实例1和实例2之间均匀分布。您将实例3添加到此服务。数据库仅填充实例3中的IMCU，然后在必要时从实例1或实例2中删除它们以进行均匀分发。一些IMCU保留在原始位置。
                           
    
                        
    • 
                        
  • 
                           
    NONE -  IMCO从当前指定服务的IM列存储中删除对象。
                           
    
                        
    • 
                     

                     
如果对象的PRIORITY值不是NONE ，则在DDL执行或服务启动后，W nnn进程在下一个IMCO循环期间填充对象。但是，如果对象的PRIORITY设置为NONE ，则仅在完整表扫描期间填充对象。扫描会触发表的指定服务处于活动状态且未阻止的所有数据库实例上的内存容量。请注意，此服务可能与发布服务的扫描不同。
                     

                     
如果用于内存中填充的服务停止，则数据库将从此服务表示的IM列存储中删除该段。在这方面，停止服务就像关闭实例一样。此对象的INMEMORY属性不会更改。如果服务再次启动，则数据库将根据其INMEMORY属性填充对象。要从IM列存储中删除对象， NO INMEMORY在DDL语句中指定NO INMEMORY 。
                     

                     
您可以将DUPLICATE与DISTRIBUTE FOR SERVICE结合使用。例如，您可以指定对象对服务dbmperf使用DUPLICATE ALL ，该服务分配给四个节点中的三个节点。在这种情况下，这三个节点中的每一个上的IM列存储都具有该对象的副本。

12.2.3 Oracle RAC中数据库内存服务的优点

                  

                     
服务和DUPLICATE的组合使您可以控制节点访问和内存中的人口。
                     

                     
服务的好处包括以下内容：

                     
    
                        
  • 
                           
    滚动补丁和升级
    
                           
    假设您设置Oracle RAC服务以将客户端查询定向到包含IM列存储的实例。如果使用DUPLICATE子句，则可以有选择地删除实例，而不会影响查询响应时间。此方法假定服务中的其他实例上存在足够的资源来处理已删除实例的工作负载。
                           
    
                           
    例如，在四节点群集中，您可以依次删除每个节点，对其进行修补，然后再次使其可用。暂时无法访问的节点的IMCU至少在另一个节点上可用，具体取决于您是使用DUPLICATE还是DUPLICATE ALL子句。因此，对列式数据的应用程序访问保持不间断。
                           
    
                        
    • 
                        
  • 
                           
    应用亲和力
    
                           
    您可以根据服务名称限制对单个节点的应用程序访问。例如，服务dbmperf1仅限于节点1，服务dbmperf2仅限于节点2，依此类推。当应用程序连接到特定服务并提交并行查询时，查询将使用属于同一服务的节点上的进程。例如，连接到服务dbmperf1的应用程序仅使用节点1上的进程。
                           
    
                           
    应用程序可以独立地共存在Oracle RAC数据库中并访问列式数据。可以在每个节点中填充完全不同的对象。例如，您可以指示HR应用程序为dbmperf1提供服务，并将销售历史记录应用程序指向服务dbmperf2 。
