17 DBMS_ALERT

DBMS_ALERT支持数据库事件（警报）的异步通知。通过适当使用此包和数据库触发器，应用程序可以在数据库中感兴趣的值发生更改时通知自身。

17.1 DBMS_ALERT概述

此方案描述了DBMS_ALERT包的可能用法。

假设图形工具正在显示数据库表中某些数据的图形。在读取和绘制数据之后，图形工具可以等待覆盖刚刚读取的数据的数据库警报（ WAITONE ）。当任何其他用户更改数据时，该工具会自动唤醒。所需要的只是在数据库表上放置一个触发器，每当触发器触发时它就会执行一个信号（ SIGNAL ）。

17.2 DBMS_ALERT安全模型

可以通过将此程序包上的EXECUTE授予选定的用户或角色来控制此程序包的安全性。您可能希望在此之上编写一个封面包，用于限制所使用的警报名称。然后可以授予此封装包的EXECUTE特权，而不是此封装。

由于数据库警报器会发出提交，因此它们不能与Oracle Forms一起使用。有关在Oracle Forms处于活动状态时调用存储过程的限制的更多信息，请参阅Oracle Forms文档。

17.6 DBMS_ALERT操作说明

本主题列出了与常规和特定应用程序相关的注释。此外，还提供了DBMS_ALERT错误消息列表。

  • 警报是基于事务的。这意味着在发出警报提交的事务之前，不会警告等待会话。给定警报可以有任意数量的并发信号，并且在给定警报上可以有任意数量的并发服务员。

  • 等待的应用程序在数据库中被阻止，无法执行任何其他工作。

  • 应用程序可以注册多个事件，然后可以使用WAITANY过程等待其中任何事件发生。

  • 应用程序还可以为WAITONE或WAITANY过程提供可选的timeout参数。如果没有待处理警报，则timeout 0立即返回。

  • 信令会话可以可选地传递由等待会话接收的消息。

  • 警报可以比相应的应用程序等待调用更频繁地发出信号。在这种情况下，旧的警报将被丢弃。应用程序始终获取最新警报（基于事务提交时间）。

  • 如果应用程序不需要基于事务的警报，则DBMS_PIPE包可能提供有用的替代方法。

    也可以看看：

    DBMS_PIPE

  • 如果在调用SIGNAL之后回滚事务，则不会发出警报。

  • 可以接收警报，读取数据，并发现没有数据发生变化。这是因为在先前警报之后但在为先前警报读取数据之前数据已更改。

  • 通常，Oracle是事件驱动的;这意味着没有轮询循环。有两种情况可能发生轮询循环：

    • 共享模式。如果数据库以共享模式运行，则需要轮询循环来检查来自其他实例的警报。轮询循环默认为一秒，可以通过SET_DEFAULTS过程设置。

    • WAITANY程序。如果使用WAITANY过程，并且信令会话发出信号但在信号的一秒内未提交，则需要轮询循环，以便此未提交的警报不会伪装其他警报。轮询循环以一秒的间隔开始，并以指数方式退回到30秒的间隔。

17.7 DBMS_ALERT示例

在此示例中，假设您要按部门绘制所有员工的平均工资。每当EMP发生变化时，您的应用程序都需要知道。

您的应用程序看起来与此代码类似：

DBMS_ALERT.REGISTER（ 'emp_table_alert'）; << readagain >>：/ * ...读取emp表并将其图形化为* / DBMS_ALERT.WAITONE（'emp_table_alert'，：message，：status）;如果status = 0则转到<< readagain >>; else / * ...错误条件* /

EMP表将具有类似于此的触发器：

在INS BEGIN DBMS_ALERT.SIGNAL（'emp_table_alert'，'message_text'）上插入或更新或删除后创建TRIGGER emptrig;结束;

当应用程序不再对警报感兴趣时，它会发出以下请求：

DBMS_ALERT.REMOVE（ 'emp_table_alert'）;

这减少了警报信号器所需的工作量。如果会话在存在已注册的警报时退出（或死亡），则此程序包的未来用户最终会清除警报。

该示例保证应用程序始终可以看到最新数据，尽管它可能看不到每个中间值。

删除警报非常重要，因为它可以减少警报信号员完成的工作量。如果会话在没有删除警报的情况下死亡，则最终（但不是立即）清除该警报。

17.8.3 REMOVEALL程序

此过程从注册列表中删除此会话的所有警报。当会话不再对任何警报感兴趣时，您应该这样做。

在会话期间首次引用此程序包时会自动调用此过程。因此，来自先前会话的可能已异常终止的警报可能会影响此会话。

此过程始终执行提交。

句法

DBMS_ALERT.REMOVEALL;

17.8.5信号程序

此过程表示警报。SIGNAL调用的效果仅在其提交的事务提交时发生。如果事务回滚，则SIGNAL无效。

所有已注册对此警报感兴趣的会话都会收到通知。如果感兴趣的会话当前正在等待，他们会被唤醒。如果感兴趣的会话当前没有等待，则下次他们进行等待呼叫时会通知他们。

多个会话可以在同一警报上同时执行信号。每个会话在发出警报信号时都会阻止所有其他并发会话，直到它提交为止。这具有序列化事务的效果。

句法

DBMS_ALERT.SIGNAL（名称IN VARCHAR2，消息IN VARCHAR2）;

参数

表17-7 SIGNAL过程参数

参数 描述

名称

要发出警报的名称。

信息

与此警报关联的消息，1800字节或更少。

此消息将传递给等待会话。通过使用消息中的信息，等待会话可能能够避免在警报发生后读取数据库。

55.1 DBMS_DEBUG_JDWP概述

Oracle支持通过JDWP调试PL / SQL存储过程和Java存储过程。使用DBMS_DEBUG_JDWP包，您可以：

  • 检索当前会话的会话ID和序列号

  • 通过JDWP将数据库会话连接到调试器

  • 设置NLS参数以更改通过JDWP表示PL / SQL程序值的格式

55.2 DBMS_DEBUG_JDWP安全模型

调用DBMS_DEBUG_JDWP子程序的调试用户需要具有将目标数据库会话连接到调试器的权限。

1.3.3 PROCESSOR_GROUP_NAME初始化参数

PROCESSOR_GROUP_NAME指定运行实例的处理器组的名称。此参数指示Oracle数据库仅在作为指定操作系统处理器组的一部分的处理器上运行。对于NUMA系统，从与该处理器组中的CPU相关联的NUMA节点分配所有系统全局区域（SGA）和程序全局区域（PGA）。

只有Linux x86-64和Oracle Solaris 11 SRU 4及更高版本支持PROCESSOR_GROUP_NAME参数。

在Linux x86-64上，命名的CPU子集是通过称为控制组（cgroups）的Linux功能创建的。Cgroups在Linux内核版本2.6.24中引入。它是通过为组指定名称和一组CPU来创建的。当进程映射到cgroup时，它仅使用与cgroup关联的CPU。

在Oracle Solaris 11 SRU 4上，通过称为资源池的功能创建指定的CPU子集。每个资源池由一个名称和一组CPU组成。当进程映射到资源池时，它使用与资源池关联的CPU。

注意：

6.1关于触发器

触发器是存储在数据库中的PL / SQL单元，并且（如果它处于启用状态）自动执行（“触发”）以响应指定的事件。

触发器具有以下结构：

TRIGGER trigger_name triggering_event [ trigger_restriction ] BEGIN triggered_action ;结束;

对于架构中的触发器， trigger_name必须是唯一的。触发器可以与架构中的另一种对象具有相同的名称（例如，表）;但是，Oracle建议使用避免混淆的命名约定。

如果触发器处于启用状态，则triggering_event会在trigger_restriction为TRUE或省略时使数据库执行triggered_action 。triggering_event与表，视图，架构或数据库相关联，它是以下之一：

  • DML语句（在“ 关于数据操作语言（DML）语句 ”中描述 ）

  • DDL语句（在“ 关于数据定义语言（DDL）语句 ”中描述 ）

  • 数据库操作（ SERVERERROR ， LOGON ， LOGOFF ， STARTUP或SHUTDOWN ）

如果触发器处于禁用状态，则triggering_event不会导致数据库执行triggered_action ，即使trigger_restriction为TRUE或被省略。

默认情况下，会在启用状态下创建触发器。您可以禁用已启用的触发器，并启用已禁用的触发器。

与子程序不同，不能直接调用触发器。触发器仅由其触发事件调用，该触发事件可由任何用户或应用程序引起。您可能不知道触发器正在执行，除非它导致错误处理不正确。

一个简单的触发器可以恰好触发其中一个定时点 ：

  • 在执行触发事件之前（ 语句级别BEFORE触发器 ）

  • 触发事件执行后（ 语句级AFTER触发器 ）

  • 在事件影响的每一行之前（ 行级别BEFORE触发器 ）

  • 在事件影响的每一行之后（ 行级AFTER触发器 ）

复合触发器可以在多个定时点触发 。有关复合触发器的信息，请参阅Oracle Database PL / SQL语言参考 。

在视图上定义了一个INSTEAD OF触发器 ，其触发事件是一个DML语句。Oracle数据库不执行DML语句，而是执行INSTEAD OF触发器。有关更多信息，请参阅“ 创建INSTEAD OF触发器 ” 。

系统触发器在架构或数据库上定义。在架构上定义的触发器会触发与架构所有者（当前用户）关联的每个事件。在数据库上定义的触发器将触发与所有用户关联的每个事件。

触发器的一个用途是实施适用于所有客户端应用程序的业务规则。例如，假设添加到EMPLOYEES表的数据必须具有某种格式，并且许多客户端应用程序可以向此表添加数据。表上的触发器可以确保添加到其中的所有数据的正确格式。因为只要任何客户端向表添加数据，就会执行触发器，没有客户端可以规避规则，并且只能在触发器中而不是在每个客户端应用程序中存储和维护强制执行规则的代码。有关触发器的其他用法，请参阅Oracle Database PL / SQL语言参考 。

6.5关于触发器编译和依赖关系

编译的触发器取决于定义它们的模式对象。如果触发器所依赖的对象被丢弃或更改，使得触发器和对象之间不匹配，则触发器无效。

SELECT * FROM USER_ERRORS WHERE TYPE ='TRIGGER';

1.4.2使用响应文件静默安装Oracle数据库示例

静默安装在后台运行，不需要用户输入。通常不会显示用户通常看到的交互式对话框。如果必须在多台计算机上多次安装Oracle产品，则无提示安装非常有用。

要执行Oracle数据库示例的静默安装，必须以静默或响应文件模式启动Oracle Universal Installer。输入类似于以下内容的命令：
  1. 在UNIX平台上：
    $ / directory_path / runInstaller [-silent] -responseFile responsefilename

    注意：

    不要指定响应文件的相对路径。如果指定相对路径，则Oracle Universal Installer会失败。

    哪里：

    • directory_path是DVD或硬盘驱动器上数据库目录的路径。
    • -silent以静默方式运行Oracle Universal Installer。
    • responsefilename是您配置的安装响应文件的完整路径和文件名。
  2. 在Windows平台上：
    DRIVE_LETTER：\ setup.exe_location setup [-silent]“variable = setting”[-nowelcome] [-noconfig] [-nowait] -responseFile

    哪里：

    • DRIVE_LETTER ：标识响应文件的完整路径。
    • setup.exe_location ：指示位置setup.exe 。
    • "variable=setting"是指响应文件中您可能更喜欢在命令行运行而不是在响应文件中设置的变量。将变量及其设置括在引号中。
    • -noconfig ：禁止在安装期间运行配置助手。
    • -nowait ：静默安装完成后关闭控制台窗口。

252.4.2 LZ_COMPRESS函数和过程

这些函数和过程使用Lempel-Ziv压缩算法压缩数据。

句法

此函数接受RAW作为输入，压缩它并返回压缩的RAW结果和元数据：

UTL_COMPRESS.LZ_COMPRESS（src IN RAW，质量IN BINARY_INTEGER DEFAULT 6）返回原始状态;

此函数接受BLOB作为输入，压缩它并返回压缩数据的临时BLOB：

UTL_COMPRESS.LZ_COMPRESS（src IN BLOB，quality IN BINARY_INTEGER DEFAULT 6）RETURN BLOB;

此过程将压缩数据返回到现有BLOB（dst），该BLOB被修剪为压缩数据大小：

UTL_COMPRESS.LZ_COMPRESS（src IN BLOB，dst IN OUT NOCOPY BLOB，quality IN BINARY_INTEGER DEFAULT 6）;

此函数返回压缩数据的临时BLOB ：

UTL_COMPRESS.LZ_COMPRESS（src IN BFILE，quality IN BINARY_INTEGER DEFAULT 6）RETURN BLOB;

此过程将压缩数据返回到现有BLOB（dst），该BLOB被修剪为压缩数据大小：

UTL_COMPRESS.LZ_COMPRESS（src IN BFILE，dst IN OUT NOCOPY BLOB，quality IN BINARY_INTEGER DEFAULT 6）;

参数

表252-4 LZ_COMPRESS函数和过程参数

参数 描述

SRC

要压缩的数据（ RAW ， BLOB or BFILE ）。

DST

压缩数据的目标

质量

1到9之间的整数，1 =快速压缩，9 =最佳压缩，默认= 6

使用说明

  • quality是可选的压缩调整值。它允许UTL_COMPRESS用户在速度和压缩质量之间进行选择，这意味着减小尺寸的百分比。更快的压缩速度将导致更少的数据压缩。较慢的压缩速度将导致更多的数据压缩。有效值为[1..9]，1 =最快，9 =最慢。默认的“质量”值为6。

252.4.6 LZ_UNCOMPRESS函数和过程

此过程接受RAW ， BLOB或BFILE压缩字符串作为输入，验证它是有效的压缩值，使用Lempel-Ziv压缩算法解压缩它，并返回未压缩的RAW或BLOB结果。

句法

此函数将未压缩的数据作为RAW返回：

UTL_COMPRESS.LZ_UNCOMPRESS（src IN RAW）返回原始状态;

此函数将未压缩的数据作为临时BLOB返回：

UTL_COMPRESS.LZ_UNCOMPRESS（src IN BLOB）返回BLOB;

此过程将未压缩的数据返回到现有的BLOB （ dst ），该BLOB将被修剪为未压缩的数据大小：

UTL_COMPRESS.LZ_UNCOMPRESS（src IN BLOB，dst IN OUT NOCOPY BLOB）;

此函数返回未压缩数据的临时BLOB：

UTL_COMPRESS.LZ_UNCOMPRESS（src IN BFILE）返回BLOB;

此过程将未压缩的数据返回到现有的BLOB （ dst ）。原始dst数据将被覆盖。 

UTL_COMPRESS.LZ_UNCOMPRESS（src IN BFILE，dst IN OUT NOCOPY BLOB）;

参数

表252-8 LZ_UNCOMPRESS函数和过程参数

参数 描述

    用于数据库查询的Oracle XML DB Web服务位于URL http:// host : port / orawsv ，其中host和port是数据库主机和HTTP（S）端口。它有一个关联的WSDL，它使用XML Schema指定传入和传出文档的格式。此WSDL位于URL http:// host : port / orawsv?wsdl 。

    用于访问PL / SQL存储函数和过程的Oracle XML DB Web服务位于URL http:// host : port / orawsv / dbschema / package / fn_or_proc或者用于独立的函数或过程（不在包中） ， http:// host : port / orawsv / dbschema / fn_or_proc 。

  • bindXML - XMLType绑定变量值。

33.4使用Web服务访问PL / SQL存储过程

用于访问PL / SQL存储函数和过程的Oracle XML DB Web服务位于URL http:// host : port / orawsv / dbschema / package / fn_or_proc或者用于独立的函数或过程（不在包中） ， http:// host : port / orawsv / dbschema / fn_or_proc 。

这里， host和port是数据库主机和HTTP（S）端口， fn_or_proc是存储的函数或过程名称， package是其包， dbschema是拥有该包的数据库模式。

输入XML文档必须包含函数或过程所需的输入。输出XML文档包含返回值和所有OUT变量的值。

输入和输出文档中的XML元素的名称对应于函数或过程的变量名称。生成的WSDL文档向您显示确切的XML元素名称。这是使用的命名约定：

  • 将输入引入PL / SQL函数的XML元素名为function-name Input ，其中function-name是function-name （大写）。

  • 引入函数输入参数的XML元素名为param-name - param-type - io-mode ，其中param-name是param-name （大写）， param-type是其SQL数据类型，以及io-mode是它的输入输出模式，如下所示：

    • IN - IN模式

    • OUT - OUT模式

    • INOUT - IN OUT模式

  • 引入PL / SQL函数输出的XML元素名为S return-type - function-name Output ，其中return-type是返回值（大写）的SQL数据类型， function-name是功能（大写）。

  • 引入函数输出参数的XML元素的名称与输出参数本身（大写）相同。引入返回值的元素名为RETURN 。

函数的返回值位于输出文档的RETURN元素中，该元素始终是文档中的第一个元素。此返回值位置将其与任何可能命名为“ RETURN ”的OUT参数消除歧义。

每个存储的函数或过程都与一个单独的动态Web服务相关联，该Web服务具有自己的生成的WSDL文档。此WSDL文档位于URL http:// host : port / orawsv / dbschema / package / fn_or_proc ?wsdl或http:// host : port / orawsv / dbschema / fn_or_proc ?wsdl 。此外，您可以选择生成单个WSDL文档，以用于给定包中的所有存储函数和过程。该WSDL文档的URL是http:// host : port / orawsv / dbschema / package ?wsdl 。

18.1 JDBC RowSet概述

一个 RowSet是一个对象，它封装来自Java数据库连接（JDBC）结果集或表格数据源的一组行。RowSet支持基于组件的开发模型，如JavaBeans，具有一组标准属性和事件通知机制。

RowSets是通过可选包在JDBC 2.0中引入的。但是，RowSets的实现在JDBC RowSet实现规范（JSR-114）中标准化，该规范从Java平台标准版（Java SE）5.0开始作为非可选包提供。Java SE 6.0 RowSets包含更多支持RowId，National Language Charactersets等功能的API。Java SE Javadocs提供有关JDBC RowSet实现的标准接口和基类的信息。

JSR-114规范包括五种RowSet的实现细节：

  • CachedRowSet的

  • 的JdbcRowSet

  • WebRowSet的

  • FilteredRowSet的

  • JoinRowSet中

Oracle JDBC通过oracle.jdbc.rowset包中提供的接口和类支持所有五种类型的RowSet。自Oracle Database 11g第1版以来，已在服务器端驱动程序中添加了RowSet支持。因此，从Oracle Database 11g第1版开始，RowSets支持在所有Oracle JDBC驱动程序类型中是统一的。标准Oracle JDBC Java Archive（JAR）文件（例如， ojdbc6.jar和ojdbc7.jar包含oracle.jdbc.rowset包。

18.1.2事件和事件监听器

RowSet支持JavaBeans事件。RowSet接口支持以下类型的事件：

  • cursorMoved

    只要有光标移动，就会生成此事件。例如，当调用next或previous方法时。

  • rowChanged

    从RowSet插入，更新或删除行时会生成此事件。

  • rowSetChanged

    创建或更改整个RowSet时会生成此事件。例如，调用execute方法时。

应用程序组件可以实现RowSet侦听器以侦听这些RowSet事件，并在事件发生时执行所需的操作。对这些事件感兴趣的应用程序组件必须实现标准的javax.sql.RowSetListener接口并使用RowSet对象注册此类侦听器对象。收听者可使用被注册RowSet.addRowSetListener方法使用和未注册RowSet.removeRowSetListener方法。可以使用相同的RowSet对象注册多个侦听器。

18.1.4关于遍历RowSet

javax.sql.RowSet接口扩展了java.sql.ResultSet接口。因此， RowSet接口提供了从ResultSet接口继承的游标移动和定位方法，用于遍历RowSet对象中的数据。一些继承的方法是absolute ， beforeFirst ， afterLast ， next和previous 。

22.1关于集群

集群提供了一种存储表数据的可选方法。集群由一组共享相同数据块的表组成。这些表组合在一起，因为它们共享公共列并且经常一起使用。

例如， emp和dept表共享deptno列。当您对emp和dept表进行集群时（参见图22-1 ），Oracle数据库实际存储了相同数据块中emp和dept表中每个部门的所有行。

由于集群将相关的不同表行存储在同一数据块中，因此正确使用的集群提供了两个主要好处：

  • 磁盘I / O减少，并且对于群集表的连接，访问时间得到改善。

  • 群集键是群集表共有的列或列组。您在创建群集时指定群集键的列。随后在创建添加到群集的每个表时指定相同的列。每个群集密钥值仅在群集和群集索引中存储一次，无论多少行的不同表包含该值。

    因此，在群集中存储相关表和索引数据可能需要的存储空间少于非群集表格式所需的存储空间。例如，在图22-1中 ，注意每个簇密钥（每个deptno ）如何只存储一次，对于emp和dept表中包含相同值的许多行。

创建集群后，您可以在集群中创建表。但是，在将任何行插入到集群表之前，必须创建集群索引。使用群集不会影响在群集表上创建其他索引;它们可以照常创建和删除。

不应将群集用于经常单独访问的表。

22.2.2为群集密钥选择适当的列

仔细选择群集键列。如果在连接表的查询中使用多个列，请将群集键设置为组合键。通常，表示良好聚类索引的特征与任何索引的特征相同。

有关良好索引特征的信息，请参阅“ 索引管理指南 ” 。

良好的簇密钥具有足够的唯一值，使得与每个键值对应的行组填充大约一个数据块。每个群集密钥值的行数太少会浪费空间并导致可忽略的性能提升。除非在群集创建时指定了小SIZE ，否则只有少数几个行共享公共值的群集密钥才会导致块中的空间浪费（请参阅“ 指定平均群集密钥及其关联行所需的空间 ” ） 。

每个群集键值的行太多可能导致额外搜索以查找该键的行。过于笼统的值（例如， male和female ）上的群集键导致过度搜索，并且可能导致比没有群集时更差的性能。

群集索引不能是唯一的，也不能包含定义为long的列。

22.2.3指定平均群集密钥及其关联行所需的空间

CREATE CLUSTER语句有一个可选子句SIZE ，它是平均簇密钥及其关联行所需的估计字节数。

执行以下任务时，数据库使用SIZE参数：

  • 估计可以适合群集数据块的群集密钥（和关联的行）的数量

  • 限制放置在集群数据块中的集群键的数量。这最大化了群集中密钥的存储效率。

SIZE不限制给定群集密钥可以使用的空间。例如，如果设置SIZE使得两个群集密钥可以适合一个数据块，则任何一个群集密钥仍然可以使用任何数量的可用数据块空间。

默认情况下，数据库仅在群集数据段的每个数据块中存储一个群集密钥及其关联的行。虽然块大小可能因操作系统而异，但每个块的一个键的规则将保持不变，因为群集表将导入到其他系统上的其他数据库。

如果给定群集键值的所有行都不能适合一个块，则将这些块链接在一起以加速使用给定键访问所有值。集群索引指向块链的开头，每个块包含集群键值和关联的行。如果群集SIZE使得多个键适合块，则块可以属于多个链。

22.3.1创建集群

关于使用OCI检索列授权指示器

Oracle调用接口（OCI）应用程序可以访问已启用数据安全策略的数据库表，然后测试列以获取授权指示符。

  • 如果确定该列对用户是未授权的，则向用户返回空列值，其中指示符“未授权”。

  • 如果无法确定列授权，则评估的列（或列表达式）值将与指示符“unknown”一起返回给用户。如果顶部列表达式评估中涉及的任何基础表列未经授权，则授权指示符可以是“未知”，并且空值将用作表达式评估的基础列值。

  • 如果该列被确定为对用户授权，则评估的列值和指示符将在没有授权指示符的情况下返回给用户。

OCI返回代码用于将列授权信息传递给用户。要获取列的授权信息，必须在绑定或定义列缓冲区时提供返回代码缓冲区。将列数据返回到用户缓冲区后，您可以检查与该列关联的返回代码以获取授权信息。列授权指示符适用于定义应用程序定义的变量或输出绑定变量。如果应用程序未检索列授权指示符，则不必提供返回码缓冲区。

要访问具有列安全性的表，至少应在绑定或定义列时访问返回码。如果未访问返回代码，则应用程序需要知道列值是否已通过其他方式授权，以便它可以正确解释指标和值。

您还应该提供绑定指示器或定义是否启用了列安全性。如果未提供指示符且列值未授权或未知，则Oracle数据库将返回错误ORA-1405。

如果列值授权未知，则授权指示符（对于ORA-24531）将优先于可能以其他方式返回给用户的常规返回代码。例如，当返回非空列值以及未知授权指示符时，列空值提取（ORA-1405）和列截断（ORA-1406）可能同时发生。在这种情况下，应用程序将ORA-24531作为此列的返回码，而不是获取ORA-1405或ORA-1406。因此，应用程序不应依赖列返回码ORA-1405或ORA-1406来查找空取或截取的确切列。

关于未知授权指示器的警告

如果为任何列返回未知授权指示符（ORA-24531），则会向应用程序返回OCI警告，即OCI函数状态将为OCI_SUCCESS_WITH_INFO ，而不是OCI_SUCCESS 。同时，ORA-24536将在返回给应用程序的错误句柄中设置。您必须检查此警告，检查正在执行的SQL，并采取适当的操作。错误ORA-24536优先于授权列或未启用列安全性时返回的错误。

如果列值未经授权或授权，则不会更改OCI功能状态代码。

默认情况下，为未知授权启用列授权警告。应用程序应设计为处理错误。如果应用程序已准备好列安全性并且想要忽略任何未知授权指示符，则可以通过在获取列值之前在OCI语句句柄中将OCI属性OCI_ATTR_NO_COLUMN_AUTH_WARNING设置为TRUE来关闭OCI警告。

关于检查表列的用户授权

该 oracle.jdbc. getAuthorizationIndicator方法oracle.jdbc.OracleResultSet接口使您可以检查列的AuthorizationIndicator属性。AuthorizationIndicator属性具有以下定义：

public static enum AuthorizationIndicator {NONE，UNAUTHORIZED，UNKNOWN; }

AuthorizationIndicator可以具有以下值：

  • NONE意味着访问列数据被授权。用户可能具有显式授权，或者列可能缺少安全属性。

  • UNAUTHORIZED意味着未授权访问列数据。

    检索列值时，将根据列的已启用列约束策略评估授权指示符。如果用户未被授权访问列值，则会将NULL值与身份验证指示符AuthorizationIndicator.一起返回给应用程序AuthorizationIndicator.UNAUTHORIZED 。

    如果存在涉及未授权基本列的列表达式，则将评估值与AuthorizationIndicator.一起返回给应用程序AuthorizationIndicator.UNAUTHORIZED指标。应用程序应在解释返回的数据之前检查授权指示符。

  • UNKNOWN意味着授权指标无法确定。

    有时，由于功能限制或性能限制，服务器无法确定SELECT项的授权指示符。例如，如果查询涉及列表达式，并且服务器无法计算顶级运算符是否应该被授权，则会发生这种情况。在这种情况下，服务器返回授权指示符AuthorizationIndicator.UNKNOWN到应用程序。返回值可以是NULL或非NULL具体取决于列表达式如何对基础列值进行操作。

    如果应用程序看到UNKNOWN授权指示符，则应确定是否应访问返回的值。如果查询及其列表达式旨在处理来自基础列的未授权NULL值，则应用程序可以使用返回的值。否则，应用程序可能必须对返回的值采取适当的操作。
