6在数据仓库环境中使用分区

分区功能可以提高数据仓库环境的性能。

本章介绍了可显着增强数据访问并提高整体应用程序性能的分区功能。对于访问具有数百万行和数千兆字节数据的表和索引的应用程序，分区的改进尤其如此，如数据仓库环境中所示。数据仓库通常包含大型表，并且需要用于管理这些大型表的技术以及跨这些大型表提供良好的查询性能。

什么是数据仓库？

数据仓库是一个关系数据库，专为查询和分析而不是事务处理而设计。

数据仓库通常包含从事务数据派生的历史数据，但可以包含来自其他来源的数据。数据仓库将分析工作负载与事务工作负载分开，并使组织能够合并来自多个源的数据。

除了关系数据库之外，数据仓库环境还可以包括提取，转换和加载（ETL）解决方案，分析处理和数据挖掘功能，客户端分析工具以及管理收集数据并将其传递到的过程的其他应用程序。商业用户。

更大的数据库

将大型数据库对象拆分为较小的部分的能力透明地简化了对非常大的数据库的有效管理。

您可以识别和操作单个分区和子分区以管理大型数据库对象。考虑分区对象的以下优点：

  • 可以在较低粒度级别上执行备份和恢复，以管理数据库的大小。

  • 数据库对象的一部分可以放在压缩存储中，而其他部分可以保持未压缩。

  • 分区可以将数据透明地存储在不同的存储层上，以降低保留大量数据的成本。有关更多信息，请参阅管理和维护基于时间的信息 。

更大的单独表格：表格中的更多行

扫描大表所需的时间比扫描小表所需的时间长。对分区表的查询可以访问与表的总大小相比较小的一个或多个分区。

同样，查询可以利用索引上的分区消除。从磁盘读取较小部分索引比读取整个索引花费的时间更少。可以在逐个分区的基础上访问和维护与表共享分区策略的索引结构，例如本地分区索引。

如果使用并行执行来加速查询，DML和DDL语句，则数据库可以利用不同分区中的不同数据集。各个并行执行服务器可以处理由分区边界标识的自己的数据集。

更多用户查询系统

通过分区，用户更有可能查询隔离的和较小的数据集。

因此，与所有用户查询相同且更大的数据集相比，数据库可以更快地返回结果。数据争用的可能性较小。

更复杂的查询

您可以使用较小的数据集更快地执行复杂查询。

如果正在访问较小的数据集，则更有可能在内存中处理复杂的计算，这从性能角度来看是有益的，并降低了应用程序的I / O要求。可能必须将更大的数据集写入临时表空间以完成查询，在这种情况下，会发生针对数据库存储的附加I / O操作。

数据仓库中的分区修剪

分区修剪是数据仓库的基本性能特征。

在分区修剪中，优化器分析SQL语句中的FROM和WHERE子句，以在构建分区访问列表时消除不需要的分区。因此，Oracle数据库仅对与SQL语句相关的分区执行操作。

分区修剪大大减少了从磁盘检索的数据量并缩短了处理时间，从而提高了查询性能并优化了资源利用率。

SELECT SUM（amount_sold）day_sales FROM sales WHERE time_id = TO_DATE（'02 -JAN-1998'，'DD-MON-YYYY'）;

高级分区修剪技术

Oracle数据库修剪功能有效地处理涉及分区表的更复杂的谓词或SQL语句。

常见的情况是分区表连接到另一个表的子集，受WHERE条件的限制。例如，请考虑以下查询：

部分分区 - 明智的联接

Oracle数据库只能并行执行部分分区连接。

与完全分区连接不同，部分分区连接要求您只对连接键上的一个表进行分区，而不是两个表。分区表称为引用表。另一个表可能已分区，也可能未分区。部分分区连接比完全分区连接更常见。

为了执行部分分区连接，数据库根据引用表的分区动态分区或重新分区另一个表。在重新分区另一个表之后，执行类似于完全分区连接。

减少通信开销

并行执行时，分区连接可减少通信开销。

在默认情况下，这种减少的发生是因为一组并行执行服务器并行执行连接操作需要将连接列上的每个表重新分配为不相交的行子集。然后，这些不相交的行子集由单个并行执行服务器成对地连接。

数据库可以避免重新分配分区，因为这两个表在连接列上进行了分区。此功能使每个并行执行服务器能够加入一对匹配的分区。在使用节点并行执行的Oracle Real Application Clusters配置中，使用并行执行的这种改进性能更加明显。

分区智能连接可显着减少互连流量。使用此功能是使用Oracle Real Application Clusters的大型决策支持系统（DSS）配置的关键。目前，大多数Oracle Real Application Clusters平台，如大规模并行处理（MPP）和对称多处理（SMP）集群，与其处理能力相比，提供有限的互连带宽。理想情况下，互连带宽应与磁盘带宽相当，但这种情况很少发生。因此，Oracle Real Application Clusters中的大多数连接操作都会遇到高互连延迟，而无需并行执行分区连接。

减少内存要求

分区连接所需的内存少于要连接的表的完整数据集的等效连接操作。

对于串行连接，连接在一对匹配的分区上同时执行。如果数据在分区之间均匀分布，则内存需求除以分区数，并且并行服务器之间的数据分布没有偏差。

对于并行连接，内存要求取决于并行连接的分区对的数量。例如，如果并行度为20并且分区数为100，则需要5倍的内存，因为每个同时执行两个分区的20个连接。分区连接需要较少内存的事实对性能有直接的有益影响。例如，连接可能不需要在散列连接的构建阶段将块写入磁盘。

并行分区 - 明智联接的性能考虑因素

在决定是否使用分区智能联接时，优化程序会权衡优缺点。

优化器根据以下内容选择是否使用分区智能联接：

  • 在分区大小不同的范围分区中，数据偏差会增加响应时间;某些并行执行服务器需要比其他服务器更长的时间来完成连接。Oracle建议使用散列分区和子分区来启用分区智能连接，因为散列分区（如果分区数是2的幂）可以限制倾斜的风险。理想情况下，散列分区键是唯一的，以最大限度地降低倾斜的风险。

  • 如果可能，用于分区连接的分区数应该是查询服务器数量的倍数。例如，在16的并行度下，您可以拥有16个，32个甚至64个分区。如果存在奇数个分区，则使用的某些并行执行服务器少于其他服务器。例如，如果有17个均匀分布的分区对，则只有一对在最后一个连接上工作，而另一对必须等待。这是因为，在执行开始时，每个并行执行服务器在不同的分区对上工作。在第一阶段之后，只剩下一对。因此，单个并行执行服务器加入该剩余对，而所有其他并行执行服务器都空闲。

在某些情况下，并行连接可能会导致远程I / O操作。例如，在MPP配置上运行的Oracle Real Application Clusters环境中，如果一对匹配的分区未在同一节点上并置，则由于远程I / O，分区连接需要额外的节点间通信。这是因为Oracle数据库必须将至少一个分区传输到执行连接的节点。在这种情况下，最好显式重新分配数据，而不是使用分区连接。

本地分区索引

在本地索引中，特定索引分区中的所有键仅指存储在单个基础表分区中的行。

本地索引与底层表格相同。Oracle数据库在与基础表相同的列上对索引进行分区，创建相同数量的分区或子分区，并为它们提供与基础表的相应分区相同的分区边界。

当添加，删除，合并或拆分基础表中的分区时，或者添加或合并散列分区或子分区时，Oracle数据库还会自动维护索引分区。这可确保索引保持与表格相同。

对于数据仓库应用程序，本地非前缀索引可以提高性能，因为可以通过索引键上的范围查询并行扫描许多索引分区。以下示例在分区的customers_dw表上创建本地B树索引：

CREATE INDEX cust_last_name_ix ON customers_dw（last_name）LOCAL PARALLEL NOLOGGING;

非分区索引

您可以在非分区表和分区表上创建非分区索引。

非分区索引主要用于数据仓库环境中的非分区表，并且通常在需要在数据仓库环境中强制执行唯一约束的状态时强制实施唯一性。您可以在分区表上使用非分区全局索引来强制实施主键或唯一键。非分区（全局）索引对于通常根据分区键中未包含的列或列集上的等式谓词或IN -list检索极少行的查询非常有用。在这些情况下，扫描单个索引比扫描多个索引分区以查找所有匹配行更快。

除分区列以外的列上的唯一索引必须是全局的，因为不支持其键不包含分区键的唯一本地非前缀索引。由于受控数据加载过程和强制执行唯一约束的性能成本，因此并非总是在数据仓库中强制使用唯一键。对于具有数十亿行的表，全局索引可能会变得非常大。

以下示例在sales表上创建全局唯一索引：

CREATE UNIQUE INDEX sales_unique_ix ON sales（cust_id，prod_id，promo_id，channel_id，time_id）PARALLEL NOLOGGING;

很少有查询从此索引中受益。在具有非常有限的数据加载窗口的系统中，请考虑不创建和维护它。

全局分区索引

您可以在非分区表和分区表上创建全局分区索引。

在全局分区索引中，特定索引分区中的键可以指存储在多个基础表分区或子分区中的行。全局索引可以是范围或散列分区，但可以在任何类型的分区表上定义。

通过指定GLOBAL属性创建全局索引。数据库管理员负责在创建时定义全局索引的初始分区以及随时间维护分区。可以根据需要合并或拆分索引分区。

如果存在一类使用表的访问路径通过索引检索几行的查询，并且通过对索引进行分区，则可以消除大部分查询的大部分索引，全局索引可能很有用。在分区表上，如果实现分区修剪所包含的一列或多列不包含表分区键，则会考虑全局分区索引。

分区交换负载

可以使用分区交换负载（PEL）添加分区。

使用PEL时，您将创建一个看起来与单个分区完全相同的单独表，包括相同的索引和约束（如果有）。如果使用复合分区表，则单独的表必须使用与复合分区表的子分区策略匹配的分区策略。然后，您可以使用此单独的表交换现有的表分区。在数据加载方案中，可以将数据加载到单独的表中。在不影响表用户查询的情况下，在单独的表上构建索引并实现约束。然后执行PEL，与数据加载相比，这是一个影响非常小的事务。在数据仓库环境中，日常负载（按天使用范围分区策略）很常见。

以下示例显示sales表的分区交换负载：

ALTER TABLE sales ADD PARTITION p_sales_jun_2007 VALUE少于（TO_DATE（'01-FEB-2007'，'dd-MON-yyyy'））; CREATE TABLE sales_jun_2007 COMPRESS FOR OLTP AS SELECT * FROM sales WHERE 1 = 0;

接下来，使用2007年6月的销售数量填充表sales_jun_2007 ，并创建已在sales表上实现的等效位图索引和约束：

CREATE BITMAP INDEX time_id_jun_2007_bix ON sales_jun_2007（time_id）NOLOGGING; CREATE BITMAP INDEX cust_id_jun_2007_bix ON sales_jun_2007（cust_id）NOLOGGING; CREATE BITMAP INDEX prod_id_jun_2007_bix ON sales_jun_2007（prod_id）NOLOGGING; CREATE BITMAP INDEX promo_id_jun_2007_bix ON sales_jun_2007（promo_id）NOLOGGING; CREATE BITMAP INDEX channel_id_jun_2007_bix ON sales_jun_2007（channel_id）NOLOGGING; ALTER TABLE sales_jun_2007 ADD CONSTRAINT prod_id_fk FOREIGN KEY（prod_id）REFERENCES products（prod_id）; ALTER TABLE sales_jun_2007 ADD CONSTRAINT cust_id_fk FOREIGN KEY（cust_id）REFERENCES customers（cust_id）; ALTER TABLE sales_jun_2007 ADD CONSTRAINT promo_id_fk FOREIGN KEY（promo_id）参考促销（promo_id）; ALTER TABLE sales_jun_2007 ADD CONSTRAINT time_id_fk FOREIGN KEY（time_id）REFERENCES times（time_id）; ALTER TABLE sales_jun_2007 ADD CONSTRAINT channel_id_fk FOREIGN KEY（channel_id）REFERENCES channels（channel_id）;

分区维护操作最容易在本地索引上执行。

在进行分区管理时，本地索引不会使全局索引无效。在PEL语句中使用INCLUDING INDEXES将本地索引与单独表上的等效索引进行交换，以便不会使索引分区失效。对于PEL，您可以将全局索引更新为负载的一部分。使用PEL语句的UPDATE GLOBAL INDEXES扩展。如果索引需要更新，那么PEL需要更长的时间。

从表中删除数据

数据仓库通常保留数据的时间窗口。例如，存储三年的历史数据。

分区使得从表中清除数据变得非常容易。您可以使用DROP PARTITION或TRUNCATE PARTITION语句清除数据。常见策略还包括使用分区交换负载从表中卸载数据，并在删除分区之前用空表替换分区。归档您在清空或删除之前交换的单独表。

drop或truncate操作会使全局索引或全局分区索引无效。本地索引仍然有效。删除表分区时，将删除本地索引分区。

分区和数据压缩

可以逐个分区地压缩分区表中的数据。

对于不经常更改的数据，使用压缩数据最有效。通常的数据仓库方案通常会在数据老化时看到很少的数据更改，而其他方案仅插入数据使用分区管理功能，您可以逐个分区地压缩数据。尽管Oracle数据库支持对所有DML操作进行压缩，但在非压缩表中修改数据仍然更有效。

更改分区以启用压缩仅适用于要插入分区的未来数据。要压缩分区中的现有数据，必须移动分区。可以在单个操作中完成启用压缩和移动分区。

要对具有位图索引的分区表使用表压缩，必须在首次引入压缩属性之前执行以下操作：

  1. 标记位图索引UNUSABLE 。

  2. 设置压缩属性。

  3. 重建索引。

第一次将压缩分区作为现有的完全未压缩分区表的一部分时，必须先删除所有现有位图索引，或者在添加压缩分区之前将其标记为UNUSABLE 。无论任何分区是否包含数据，都必须执行此操作。它还与导致一个或多个压缩分区成为表的一部分的操作无关。这不适用于仅具有B树索引的分区表。

操作：以其他用户身份登录并重试。

PROT-00311：未获得足够的许可权

原因：用户不是特权用户。

操作：以root或管理员用户身份登录，然后重新发出该命令。

PROT-00312：指定的文件' string '格式无效。

原因： ocrdump命令的file参数指定的文件不是有效的Oracle Cluster Registry备份文件。

操作：使用'ocrconfig -showbackup'命令列出所有可用备份。重新发出ocrdump命令，指定ocrconfig命令列出的有效Oracle Cluster Registry备份文件。

PROT-00313：Oracle高可用性服务未在本地节点上运行。

原因：当Oracle高可用性服务未在本地节点上运行时，尝试执行'ocrdump'命令。

操作：运行'crsctl start crs'命令以在本地节点上启动Oracle高可用性服务并重新发出该命令。

PROT-00601：初始化ocrcheck失败

原因：无法初始化并启动ocrcheck。

操作：检查日志文件以获取有关失败的详细信息。

PROT-00602：无法从群集注册表中检索数据

原因：无法从群集注册表中检索必要的信息。

操作：从日志文件中检查群集注册表的可用性以及故障的详细信息。

PROT-00603：Oracle Cluster Registry在逻辑上已损坏

原因： Oracle Cluster Registry中检测到逻辑损坏。

操作：检查日志文件以获取更多信息。使用'ocrconfig -showbackup'命令检查备份的可用性。恢复兼容的备份，并使用ocrcheck进行验证。

PROT-00604：无法检索群集注册表的已配置位置。

原因：无法检索群集注册表的已配置位置。

操作：检查日志文件以获取有关失败的详细信息。

PROT-00605：Leaf节点不支持'ocrcheck'命令。

原因：尝试从Leaf节点执行'ocrcheck'命令，该节点不受支持。

操作：从Hub节点执行'ocrcheck'命令。

PROT-00606：未获得足够的许可权

原因：用户不是特权用户。

操作：以root或管理员用户身份登录，然后重新发出该命令。

PROT-00607：检测到多个OCR存储类型。

原因：检测到多个Oracle Cluster Registry（OCR）存储类型。不支持同时在不同存储类型上存储OCR。

操作：使用相同的存储类型配置OCR。

26.1为重复数据库文件指定备用名称

如果源数据库和重复数据库不对数据库文件使用相同的名称，则必须为重复文件选择备用命名策略。

根据源和重复数据库是使用Oracle托管文件（OMF）还是Oracle自动存储管理（ASM），请使用以下策略之一：

  • 为重复数据库文件指定非OMF或非ASM备用名称
  • 使用CONFIGURE AUXNAME命令命名文件系统数据文件和OMF或ASM目标数据文件

    如果源数据文件使用OMF，则无法使用DB_FILE_NAME_CONVERT初始化参数重命名它们。使用非ASM存储可以讨论OMF管理的数据文件的详细信息和选项。

26.1.1为重复数据库文件指定非OMF或非ASM备用名称

如果源数据库和重复数据库使用不同的目录结构或使用相同的结构但您希望以不同方式命名重复文件，则必须指定必须如何命名重复的数据库文件。

  1. 确保满足所选复制技术的先决条件。

    请参阅“ 复制数据库的先决条件 ” 。

  2. 在开始数据库复制之前，请完成所需的规划任务。

    请参阅“ 计划复制数据库 ” 。

  3. 准备创建重复数据库时使用的辅助实例。

    复制Oracle Real Application Clusters（Oracle RAC）数据库时，请将辅助数据库上的CLUSTER_DATABASE初始化参数设置为FALSE 。复制过程完成后，此参数可以重置为TRUE 。

    请参阅“ 准备辅助实例 ” 。

  4. 启动RMAN并连接到所需的数据库。根据您的复制技术，您可能需要连接到以下一个或多个目标：目标数据库，辅助实例或恢复目录。

    请参阅“ 启动RMAN并连接到数据库 ” 。

  5. 如有必要，将源数据库置于适当的状态。

    请参阅“ 将源数据库置于适当的状态 ” 。

  6. （可选）配置RMAN通道以提高复制性能。通道执行复制数据库的主要任务。

    请参阅“ 配置RMAN通道以用于复制 ” 。

26.2在没有共享磁盘的情况下使磁盘备份可访问

您可以使用共享磁盘技术使备份可用于辅助实例。当NFS或共享磁盘不是一个选项，然后存储该备份必须存在于源主机和目标主机，除非路径BACKUP LOCATION选项用于DUPLICATE命令没有目标或恢复目录的连接。

假设您维护两个主机， srchost和dsthost 。srchost上的数据库是srcdb 。srcdb的RMAN备份位于主机srchost上的/dsk1/bkp目录中。目录主机上正在使用目录/dsk1/bkp ，因此您打算将备份存储在目标主机上的/dsk2/dup目录中。

要将备份从源主机传输到目标主机：

  1. 在源主机中创建一个目录，该目录与包含备份的目标主机上的目录同名。

    例如，如果你打算到RMAN备份在存储/dsk2/dup目录中的目标主机上，然后创建/dsk2/dup源主机上目录。

  2. 在源主机上，将备份复制到上一步中创建的目录，然后对备份进行编目。您可以使用以下任一方法：
    • 将RMAN作为TARGET连接到源数据库，并使用BACKUP命令备份备份。例如，使用BACKUP COPY OF DATABASE命令将源主机上/dsk2/dup /dsk1/bkp的备份复制到源主机上的/dsk2/dup 。在这种情况下，RMAN会自动对新位置中的备份进行编目。如果要复制PDB，请使用BACKUP COPY OF命令的PLUGGABLE DATABASE语法仅复制PDB的备份。
    • 使用操作系统实用程序将源主机上/dsk2/dup /dsk1/bkp目录中的备份复制到源主机上的/dsk2/dup目录。然后，将RMAN作为TARGET连接到源数据库，并使用CATALOG命令使用手动传输的备份的位置更新源控制文件。

  3. 手动将源主机上新目录中的备份传输到目标主机上具有相同名称的目录。

    例如，使用FTP将源主机上/dsk2/dup目录中的备份传输到目标主机上的/dsk2/dup目录。

辅助通道可以在目标主机上的/dsk2/dup目录中搜索备份并还原它们。

3.1登录用户名和密码

当您启动SQL * Plus时，您需要一个用户名和密码以登录Oracle数据库架构。您的用户名和密码将您标识为Oracle数据库架构的授权用户。

数据库管理员（DBA）负责使用必要的权限创建数据库帐户，并为您提供用于访问帐户的用户名和密码。

将创建默认登录，并在Oracle数据库安装期间提示您输入关联的密码。创建的一些默认登录用户名是：

  • SYS

  • 系统

  • HR

在Oracle数据库安装期间，将在消息中创建并显示登录。

有关默认登录的详细信息，请参阅Oracle数据库用户类型 。

登录后，您可以使用其他用户名进行连接使用CONNECT 命令。用户名和密码必须对数据库有效。例如，要使用密码FOX将用户名TODD连接到默认数据库，您可以输入

连接TODD

系统会提示您输入密码FOX。

在命令行界面中，如果省略用户名和密码，SQL * Plus会提示您输入用户名和密码。由于CONNECT首先将您与当前数据库断开连接，因此如果在CONNECT命令中使用无效的用户名和密码，则将保持未连接到任何数据库。

如果以帐户已过期的用户身份登录或连接，系统将提示您更改密码，然后才能进行连接。

如果帐户被锁定，则会显示一条消息，并且在您的DBA解锁帐户之前，不允许以该用户身份进行连接。

您可以使用DISCONNECT 命令断开与数据库的连接没有离开在SQL * Plus。

3.1.1安全外部密码存储

作为应用程序使用密码凭据连接到数据库的大规模部署的命令行替代方法，可以将此类凭据存储在客户端Oracle钱包中。Oracle钱包是一个安全的软件容器，用于存储身份验证和签名凭据。

在客户端Oracle钱包中存储数据库密码凭证，无需在应用程序代码中嵌入用户名和密码， 批量作业，或脚本。这样可以降低在脚本和应用程序代码中以明文形式泄露密码的风险，并简化维护，因为每次用户名和密码更改时都不需要更改代码。此外，无需更改应用程序代码也可以更轻松地为这些用户帐户实施密码管理策略。

将客户端配置为使用外部密码存储时，应用程序可以使用以下语法连接到使用密码身份验证的数据库：

CONNECT / @ database_alias

请注意，您无需在此CONNECT语句中指定数据库登录凭据。相反，您的系统会在客户端钱包中查找数据库登录凭据。

1.2.1关于模式对象

    您可以创建一个视图，将来自多个不同表的信息组合到一个演示文稿中。视图可以依赖来自表和其他视图的信息。

  • 序列

    当表的所有记录必须是不同的时，您可以使用序列为数字列生成唯一整数的序列列表，每个整数表示一个记录的ID。

该网关系统标识符 （SID）是标识网关实例的字母数字字符串。对于要访问的每个Sybase数据库，您需要一个网关实例，因此需要一个网关SID。SID用作初始化参数文件的文件名的一部分。默认SID是 dg4sybs 。

您可以定义网关SID，但使用默认的dg4sybs更容易，因为您不需要更改初始化参数文件名。但是，如果要访问两个Sybase数据库，则需要两个网关SID，每个网关用于网关的每个实例。如果您只有一个Sybase数据库并且希望有时使用一组网关参数设置来访问它，而其他时候使用不同的网关参数设置，那么您将需要多个网关SID用于单个Sybase数据库。

自定义初始化参数文件

 $ ORACLE_HOME / dg4sybs /管理/ initdg4sybs.ora

21.1.1更新批处理概述

通过将多个UPDATE ， DELETE或INSERT语句分组到一个批处理中，并将整个批处理发送到数据库并在一次行程中处理，您可以减少到数据库的往返次数，从而提高应用程序性能。这被称为“更新批处理”。当您使用不同的绑定变量重复相同的语句时，这对预准备语句特别有用。

注意：

  • JDBC 2.0规范将“更新批处理”称为“批量更新”。

  • 为了遵守JDBC 2.0标准，标准更新批处理的Oracle实现支持不带OUT参数，泛型语句和预准备语句的可调用语句。您可以毫不费力地将标准更新批处理迁移到Oracle JDBC应用程序中。但是，标准更新批处理的Oracle实现不会为通用语句和可调用语句实现真正的批处理，您将看到仅对PreparedStatement对象的性能改进。

21.1.2标准更新批处理

JDBC标准更新批处理依赖于使用addBatch方法向批处理显式添加语句，并使用executeBatch方法显式处理批处理。

注意：

使用更新批处理时禁用自动提交模式。如果在处理批处理时发生错误，则可以选择提交或回滚在错误之前成功运行的操作。

21.1.2.1 Oracle标准批处理实现的局限性

本节讨论有关Oracle标准更新批处理实现的限制和实现细节。

在Oracle JDBC应用程序中，更新批处理旨在与使用不同绑定值集重复处理的预准备语句一起使用。

标准更新批处理的Oracle实现不会为通用语句和可调用语句实现真正的批处理。尽管Oracle JDBC支持对Statement和CallableStatement对象使用标准批处理，但您不太可能看到性能提升。

21.1.2.2关于向批处理添加操作

首次创建任何语句对象时，其语句批处理为空。使用标准的addBatch方法将操作添加到语句批处理中。此方法在标准java.sql.指定java.sql.Statement ， PreparedStatement和CallableStatement接口，由oracle.jdbc.实现oracle.jdbc.OracleStatement ， OraclePreparedStatement和OracleCallableStatement接口，分别。

对于Statement对象， addBatch方法将带有SQL操作的Java String作为输入。例如：

...语句stmt = conn.createStatement（）; stmt.addBatch（“INSERT INTO emp VALUES（1000，'Joe Jones'）”）; stmt.addBatch（“INSERT INTO dept VALUES（260，'Sales'）”）; stmt.addBatch（“INSERT INTO emp_dept VALUES（1000,260）”）; ...

此时，批处理中有三个操作。

对于预准备语句，更新批处理用于批处理具有不同绑定参数集的同一语句的多次运行。对于PreparedStatement或OraclePreparedStatement对象， addBatch方法不接受任何输入。它只是使用最后由相应的set XXX方法设置的绑定参数将操作添加到批处理中。对于CallableStatement或OracleCallableStatement对象也是如此，但请记住，在标准更新批处理的Oracle实现中，您可能会看到在批处理可调用语句时没有性能改进。

21.1.2.3关于处理批处理

要处理当前批处理操作，请使用语句对象的executeBatch方法。此方法在标准Statement接口中指定，该接口由标准PreparedStatement和CallableStatement接口扩展。

注意：

21.2.1关于预取LOB数据

对于Oracle Database 11g第2版JDBC驱动程序之前的JDBC驱动程序，如果要在一次往返中检索LOB数据，则必须以VARCHAR2类型获取数据，即必须使用OracleTypes.VARCHAR或OracleTypes.使用JDBC defineColumnType方法的LONGVARCHAR 。这种方法的局限性在于，当LOB数据被提取为CHAR类型时，无法与数据一起提取定位器。因此，如果应用程序想要在稍后的时间点获取LOB数据，或者应用程序想要执行其他LOB操作，则需要再往返一次以获取LOB定位器，因为LOB定位器不可用于应用。

注意：

JDBC API不支持LOB定位器上的数组操作。

从Oracle Database 11g第2版JDBC驱动程序开始，通过预取经常使用的元数据（例如LOB长度和块大小）以及常规获取操作期间LOB数据的开头以及定位器，可以减少往返次数。 。对于小LOB ，数据可以在一次往返中完全预取，即，在一次往返中发生select解析，执行和提取，并且性能得到极大改善。对于大于预取大小的5倍的大型LOB ，性能改进不是很显着，因为不需要用于检索块大小的往返行程。

defaultLobPrefetchSize连接属性

从Oracle Database 11g第2版开始，有一个新的连接属性oracle.jdbc.defaultLobPrefetchSize ，可用于设置连接的默认LOB预取大小。此连接属性定义为以下常量： OracleConnection.CONNECTION_PROPERTY_DEFAULT_LOB_PREFETCH_SIZE 。此属性的值用作当前连接的默认LOB预取大小。此连接属性的默认值为4000。如果要在语句级别更改默认值，请使用oracle.jdbc.定义的setLobPrefetchSize方法oracle.jdbc.OracleStatement接口。您可以将默认值更改为：

  • -1禁用当前连接的LOB预取

  • 0仅为元数据启用LOB预取

  • 任何大于0的值，用于指定BLOB的字节数以及在获取操作期间与定位器一起预取的CLOB的字符数

21.2.2 Oracle行预取限制

没有最大预取设置。默认值为10。根据查询所需的行数和列数，可能更大或更小的值。您可以使用Properties对象设置默认连接行预取值。

创建语句对象时，它将从关联的连接接收默认的行预取设置。对默认连接行预取设置的后续更改将不会对语句行预取设置产生任何影响。

如果结果集的列的数据类型为LONG ， LONG RAW或LOB通过数据接口返回，即流类型，则JDBC会将语句行预取设置更改为1，即使您实际上从未读过这两种类型的价值。

设置预取大小可能会影响应用程序的性能。增加预取大小将减少获取所有数据所需的往返次数，但会增加内存使用量。这取决于查询中列的数量和大小以及预期返回的行数。它还取决于JDBC客户端计算机的内存和CPU负载。独立客户端应用程序的最佳选择将与负载很重的应用程序服务器不同。还应考虑网络连接的速度和延迟。

注意：

如果要将应用程序从早期版本的Oracle JDBC驱动程序迁移到10g第1版（10.1）或更高版本的Oracle JDBC驱动程序，那么您应该重新审视之前已完成的优化，因为内存使用和性能特征可能具有变化很大。

您可能遇到的一种常见情况是，您有一个选择唯一键的查询。查询将仅返回零行或一行。将预取大小设置为1将降低内存和CPU使用率，并且不能增加往返次数。但是，您必须小心避免通过写入while(rs.next())而不是if(rs.next())来请求额外提取的错误。

如果您使用的是JDBC Thin驱动程序，则使用useFetchSizeWithLongColumn连接属性，因为它将在一次往返中执行PARSE ， EXECUTE和FETCH 。

21.2.3关于定义列类型

注意：

从Oracle Database 12 c第1版（12.1）开始，不推荐使用defineColumnType方法。

自Oracle Database 10g以来， defineColumnType的实现发生了重大变化。以前， defineColumnType既用作性能优化又用于强制数据类型转换。在以前的版本中，所有驱动程序都受益于对defineColumnType调用。从Oracle数据库10g开始 ，JDBC Thin驱动程序不再需要提供的信息。JDBC Thin驱动程序无需调用defineColumnType即可实现最高性能。当应用程序使用defineColumnType时，JDBC Oracle调用接口（OCI）和服务器端内部驱动程序仍可获得更好的性能。

如果您的代码与JDBC Thin和OCI驱动程序一起使用，则可以在使用Thin驱动程序时通过将connection属性disableDefineColumnType设置为true来禁用defineColumnType方法。这样做会使defineColumnType无效。使用JDBC OCI或服务器端内部驱动程序时，请勿将此连接属性设置为true 。

您还可以使用defineColumnType来控制客户端分配的内存量或限制可变长度数据的大小。

请按照以下常规步骤为查询定义列类型：

  1. 如果有必要，投你的Statement对象OracleStatement ， OraclePreparedStatement ，或OracleCallableStatement （如适用）。

  2. 如有必要，请使用Statement对象的clearDefines方法清除此Statement对象的所有clearDefines定义。

  3. 在每一列上，调用Statement对象的defineColumnType方法，并将这些参数传递给它：

    • 列索引（整数）

    • 类型代码（整数）

      使用java.sql.的static常量java.sql.Types类或oracle.jdbc.OracleTypes类，例如Types.INTEGER ， Types.FLOAT ， Types.VARCHAR ， OracleTypes.VARCHAR和OracleTypes.ROWID 。标准类型的类型代码在这两个类中是相同的。

    • 输入名称（字符串）

      对于结构化对象，对象引用和数组，还必须指定类型名称。例如， Employee ， EmployeeRef或EmployeeArray 。

    • 最大字段大小（整数）

      （可选）为此列指定最大数据长度。

      如果要为结构化对象，对象引用或数组定义列类型，则不能指定最大字段大小参数。如果您尝试包含此参数，则会将其忽略。

    • 使用形式（短）

      （可选）指定列的使用形式。这可以是OraclePreparedStatement.FORM_CHAR使用数据库字符集或OraclePreparedStatement.FORM_NCHAR使用国家字符集。如果省略此参数，则默认为FORM_CHAR 。

    例如，假设stmt是Oracle语句，请使用：

    stmt.defineColumnType（ column_index ， typeCode ）;

    如果列是VARCHAR或等效的，并且您知道长度限制：

    stmt.defineColumnType（ column_index ， typeCode ， max_size ）;

    对于需要原始最大长度并请求转换为数据库字符集的NVARCHAR列：

    stmt.defineColumnType（ column_index ， typeCode ，0，OraclePreparedStatement。FORM_CHAR）;

    对于结构化对象，对象引用和数组列：

    stmt.defineColumnType（ column_index ， typeCode ， typeName ）;

    如果您不想接收数据的完整默认长度，请设置最大字段大小。调用标准JDBC Statement类的setMaxFieldSize方法会对返回的数据量设置限制。具体而言，返回的数据大小将是以下最小值：

    • defineColumnType设置的最大字段大小

    • setMaxFieldSize设置的最大字段大小

    • 数据类型的自然最大大小

getProcedures和getProcedureColumns方法的注意事项

根据JDBC版本1.1和1.2，方法getProcedures和getProcedureColumns以相同的方式处理catalog ， schemaPattern ， columnNamePattern和procedureNamePattern参数。在这些方法的Oracle定义中，参数的处理方式不同：

  • 目录

    Oracle没有多个目录，但它确实有包。因此， catalog参数被视为包名称。这适用于输入（作为catalog参数）和输出（它是返回的ResultSet的catalog列）。在输入时，构造" "是一个空字符串，它在没有包的情况下检索过程和参数，即独立对象。null值意味着从选择标准中删除，即返回有关独立对象和打包对象的信息。也就是说，它与传递百分号（ % ）具有相同的效果。否则，如果需要， catalog参数应该是包名称模式，带有SQL通配符。

  • schemaPattern

    Oracle数据库中的所有对象都必须具有模式，因此在没有模式的情况下返回这些对象的信息是没有意义的。因此，构造" " （它是一个空字符串）在输入上被解释为表示当前模式中的对象，即您当前连接的对象。为了与catalog参数的行为保持一致， null被解释为从选择条件中删除模式。也就是说，它与传入%具有相同的效果。它也可以用作SQL通配符的模式。

  • procedureNamePattern和columnNamePattern

    空字符串（“”）对任一参数都没有意义，因为所有过程和参数都必须具有名称。因此，构造" "将引发异常。为了与其他参数的行为保持一致， null与传入百分号（ % ）具有相同的效果。

95.5 DBMS_LOCK操作说明

用户锁永远不会与Oracle锁冲突，因为它们用前缀“UL”标识。您可以使用企业管理器锁定监视器屏幕或相应的固定视图查看这些锁定。

会话终止时会自动释放用户锁定。锁标识符是0到1073741823范围内的数字。

由于保留的用户锁与Oracle锁相同，因此它具有Oracle锁的所有功能，例如死锁检测。确保在COMMIT上释放分布式事务中使用的任何用户锁，否则可能会发生未检测到的死锁。

DBMS_LOCK最有效，每个会话限制几百个锁。Oracle强烈建议您开发使用这些锁的标准约定，以避免尝试使用相同锁的过程之间发生冲突。例如，将您的公司名称包含在锁定名称中。

95.6.1 ALLOCATE_UNIQUE过程

此过程为指定的锁名称分配唯一的锁标识符（在1073741824到1999999999范围内）。锁标识符用于使应用程序能够协调其锁的使用。这是因为应用程序可能更容易根据锁名称而不是锁定号来协调它们对锁的使用。

句法

参数

要为其生成唯一ID的锁的名称。

不要使用以ORA$开头的锁名称;这些保留用于Oracle提供的产品。

锁定名称与最后一次调用具有指定锁定名称的ALLOCATE_UNIQUE expiration_secs （默认为10天）的返回锁定ID相关联。在此之后，可以删除此锁定名称的dbms_lock_allocated表中的行以恢复空间。ALLOCATE_UNIQUE执行提交。
