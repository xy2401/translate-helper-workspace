网络附加存储和NFS协议

网络附加存储（NAS）系统使用网络文件系统（NFS）协议来访问网络上的文件，这使客户端服务器能够像直接连接到服务器的存储设备一样轻松地通过网络访问文件。您可以在受支持的NFS系统上存储数据文件。NFS是一种共享文件系统协议，因此NFS可以支持单实例和Oracle Real Application Clusters数据库。

注意：

存储在NAS设备上的Oracle软件和数据库的性能取决于服务器与网络连接存储设备之间的网络连接性能。为了获得更好的性能，Oracle建议您使用专用的专用网络连接将服务器连接到NAS设备。NFS网络连接应使用千兆以太网或更好。

请参阅供应商文档以完成NFS配置和安装。

信息集成挑战

信息集成是一项影响许多组织的挑战，因为它们可能运行多个不同的数据库系统。

这些系统中的每一个都存储数据，并具有一组针对数据运行的应用程序。这些数据只是文件系统上的位和字节 - 只有数据库才能将数据的位和字节转换为业务信息。集成和整合所有业务信息使组织能够利用业务信息中固有的协同作用。

将所有数据合并到一个数据库系统中通常很困难。这主要是因为许多针对一个数据库运行的应用程序可能没有与另一个数据库运行的等效应用程序。在将数据迁移到一个统一数据库系统之前，异构数据库系统必须协同工作。

在可以实现互操作性之前，有几个问题需要克服。可以使用不同的接口，不同的数据类型，不同的功能以及处理错误的不同方式来访问数据库系统。即使一个关系数据库试图访问另一个关系数据库，差异也很大。在这种情况下，数据库的共同特征包括通过SQL进行数据访问，两阶段提交协议和类似的数据类型。

也存在显着差异。SQL方言可以不同，事务语义也可以。一个数据库中可能存在一些在另一个数据库中不存在的数据类型。最重要的区别在于两个数据库的数据字典。大多数数据字典包含类似的信息，但是以不同的方式为每个数据字典构建信息。有几种方法可以克服这个问题。本指南介绍了Oracle同步访问多个来源信息的方法。

回顾HugePages内存分配

如果您的操作系统启用了HugePages，请查看此信息。

在Linux平台安装中，Oracle建议您使用HugePages获取Oracle数据库的最佳性能。在启用了HugePages的服务器上升级Oracle Grid Infrastructure和Oracle数据库时，Oracle建议您查看HugePages内存分配要求。

GIMR和HugePages记忆

Oracle域服务集群安装包括Grid Infrastructure Management Repository（GIMR），但GIMR配置对于Oracle Standalone Cluster安装是可选的。在群集成员节点上配置HugePages时，GIMR系统全局区域（SGA）将安装到HugePages内存中。GIMR SGA占用高达1 GB的HugePages内存。Oracle Grid Infrastructure在群集上安装Oracle数据库之前启动。

如果群集成员节点操作系统内存分配到HugePages不足以满足群集上所有Oracle数据库实例的SGA大小，那么您可能会发现一个或多个Oracle数据库SGA映射到常规页面，而不是的巨大页面，降低了预期的性能。为避免此问题，在计划升级时，请确保为HugePages保留的内存足够大，以满足您的内存要求。

将内存分配给HugePages足够大，以便计划在群集上运行SGA的所有数据库，并为Grid Infrastructure Management Repository提供SGA。

关于自动内存管理安装选项

确定是否要在安装期间配置自动内存管理。

在典型安装期间，使用Database Configuration Assistant（DBCA）创建数据库，并启用自动内存管理。如果选择高级安装，则可以手动指定内存分配，也可以启用自动内存管理。

通过自动内存管理，Oracle数据库实例可以自动管理和调整内存。通过自动内存管理，您可以选择内存目标，实例会自动在系统全局区域（SGA）和实例程序全局区域（实例PGA）之间分配内存。随着内存需求的变化，实例动态地在SGA和实例PGA之间重新分配内存。

您可以在数据库安装期间或之后启用自动内存管理。安装后启用自动内存管理涉及关闭并重新启动数据库。

从12 c开始，Oracle数据库在Oracle Solaris 10 1/13或更高版本以及Oracle Solaris 11 SRU 7.5或更高版本系统上使用Oracle Solaris的优化共享内存（OSM）模型来实现自动内存管理。但是，Oracle Database 12 c继续在OSM不可用的系统上使用私密共享内存（ISM）或动态内部共享内存（DISM）。

嵌套子查询的解除

当子查询出现在父语句的WHERE子句中时，它们将嵌套 。当Oracle数据库使用嵌套子查询评估语句时，它必须多次评估子查询部分，并且可能会忽略一些有效的访问路径或连接。

子查询不需要的东西，并将子查询的主体合并到包含它的语句的主体中，允许优化器在评估访问路径和连接时将它们一起考虑。除了一些例外，优化器可以取消大多数子查询。这些异常包括分层子查询和包含ROWNUM伪列的子查询，集合运算符之一，嵌套聚合函数或对查询块的相关引用，该查询块不是子查询的直接外部查询块。

假设不存在任何限制，优化器会自动删除以下嵌套子查询中的一些（但不是全部）：

  • 不相关的IN子查询

  • IN和EXISTS相关子查询，只要它们不包含聚合函数或GROUP BY子句

您可以通过指示优化器不再使用其他类型的子查询来启用扩展子查询的取消 ：

  • 您可以通过IN子查询中指定HASH_AJ或MERGE_AJ提示来HASH_AJ不相关的NOT IN子查询。

  • 您可以通过在子查询中指定UNNEST提示来UNNEST其他子查询。

您必须确认非联网计算机可以连接到自身，以确保您可以在安装后配置客户端网络分辨率。非联网计算机是没有固定网络地址的计算机，例如使用DHCP的计算机。

您可以在非联网计算机上安装Oracle数据库。如果计算机（如笔记本电脑）配置为DHCP并且您计划在安装Oracle数据库后将计算机连接到网络，则在要安装数据库的计算机上使用ping命令检查计算机可以连接到自己。首先只使用主机名，然后使用完全限定名称执行此步骤，该名称应位于/etc/hosts文件中。

关于Oracle Base Homes

在只读ORACLE_HOME和读/写ORACLE_HOME中，用户特定文件，特定于实例的文件和日志文件都位于称为ORACLE_BASE_HOME的位置。

在读/写ORACLE_HOME中，ORACLE_BASE_HOME路径与ORACLE_HOME目录相同。但是，在只读ORACLE_HOME中，ORACLE_BASE_HOME目录与ORACLE_HOME不在同一位置，而是位于ORACLE_BASE / homes / HOME_NAME。

其中，HOME_NAME是ORACLE_HOME的内部名称。

例如，网络目录network/admin ， network/trace和network/log位于ORACLE_BASE_HOME目录中。在读/写ORACLE_HOME中，网络目录似乎位于ORACLE_HOME中，因为ORACLE_BASE_HOME与ORACLE_HOME位于同一位置，而在只读ORACLE_HOME中，网络目录位于ORACLE_BASE / homes / HOME_NAME中。

要打印ORACLE_BASE_HOME路径，请从$ORACLE_HOME/bin目录运行orabasehome命令：
$ setenv ORACLE_HOME /u01/app/oracle/product/19.0.0/dbhome_1 $ cd $ ORACLE_HOME / bin $ ./orabasehome

例如：

$ ./orabasehome $ / u01 / app / oracle / homes / OraDB19Home1

其中， /u01/app/oracle是ORACLE_BASE， OraDB19Home1是HOME_NAME

优化分布式查询的性能

您可以优化分布式查询的性能。

你可以提高性能使用以下策略分发查询：

  • 选择最好的SQL语句。

    在许多情况下，有几个SQL语句可以实现相同的结果。如果所有表都在同一个数据库上，那么这些SQL语句之间的性能差异可能很小。如果表位于不同的数据库上，则性能差异可能更大。另请注意，最佳SQL语句可能会从一个发行版更改为下一个发行版。

  • 使用查询优化器。

    查询优化器使用远程表上的索引，考虑比基于规则的优化器更多的执行计划，并且通常会提供更好的结果。使用查询优化器，分布式查询的性能通常令人满意。只有在极少数情况下才需要更改SQL语句，创建视图或使用过程代码。

  • 使用视图。

    在某些情况下，视图可用于提高分布式查询的性能。例如：

    • 加入远程数据库上的几个远程表

    • 通过网络发送不同的表

    • 并行检索远程表中的数据

  • 使用程序代码。

    在极少数情况下，通过过程代码（例如PL / SQL过程或预编译器程序）替换分布式查询可能更有效。

检查直接NFS客户端的TCP网络协议缓冲区

检查TCP网络缓冲区大小，确保它足以满足服务器的速度。

默认情况下，TCP的网络缓冲区大小设置为1 MB，UDP的网络缓冲区大小设置为2 MB。 TCP缓冲区大小可以设置文件传输限制，这可能会对Direct NFS客户端用户的性能产生负面影响。

要检查Oracle Solaris 11上的当前TCP缓冲区大小：

#ipad show-prop -p max_buf tcp

Oracle建议您根据服务器的链接速度设置值。例如：

在Oracle Solaris 11上：

#ipad set-prop -p max_buf = 1048576 tcp

此外，请检查TCP发送窗口大小和TCP接收窗口大小，以确保它们足以满足服务器的速度。

要在Oracle Solaris 11上检查当前TCP发送窗口大小和TCP接收窗口大小：

#ipad show-prop -p send_buf tcp #ipadm show-prop -p recv_buf tcp

Oracle建议您根据服务器的链接速度设置值。例如：

在Oracle Solaris 11上：

#ipad set-prop -p send_buf = 1056768 tcp #ipadm set-prop -p recv_buf = 1056768 tcp

Oracle大约每三个月发布一个新版本的Oracle Trace File Analyzer。

应用标准版本更新修订（RUR）会自动更新Oracle跟踪文件分析器。但是，发布更新修订（RUR）不包含其余的Oracle数据库支持工具包更新。从My Oracle Support说明1513912.1下载最新版本的Oracle跟踪文件分析器和Oracle数据库支持工具包。

升级与首次安装类似。以root installTFA platform使用installTFA platform脚本。如果已安装Oracle跟踪文件分析器，则安装程序将更新现有安装。已安装时，群集升级不需要SSH。群集升级使用主机之间的现有守护程序安全套接字通信。 

$ ./installTFA platform

如果您无法以root身份安装，请将Oracle Trace File Analyzer安装为Oracle主人。使用-extractto和–javahome选项：

$ ./installTFA platform jre_home dir -javahome jre_home
