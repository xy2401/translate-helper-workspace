12.1 Oracle Locale Builder Utility概述

Oracle Locale Builder提供了一种简单有效的方法来自定义区域设置数据。它提供了一个图形用户界面，通过它您可以轻松查看，修改和定义特定于语言环境的数据。它从文本中提取数据和二进制定义文件，并以可读格式显示它们，以便您可以处理信息，而无需担心这些文件中使用的格式。

Oracle Locale Builder管理四种类型的区域设置定义：语言，区域，字符集和语言排序。它还支持用户定义的字符和自定义语言规则。您可以查看现有文本和二进制定义文件中的定义并对其进行更改，也可以创建自己的定义。

12.1.1为Oracle Locale Builder配置Unicode字体

Oracle Locale Builder在其许多功能中使用Unicode字符。例如，它显示了本地字符代码点到Unicode代码点的映射。Oracle Locale Builder依赖于在Java Runtime中配置的逻辑字体Serif和SansSerif来显示字符。如果无法使用配置的字体呈现字符，则通常将其显示为矩形框。如果在Oracle Locale Builder用户界面中无法正确查看某些字符，则可能需要重新配置逻辑字体以包含支持缺少字符的其他物理字体。

注意：

Oracle Locale Builder使用的Java Runtime位于Oracle Home目录的jdk / jre子目录中。

12.5.3 Unicode专用区

在Unicode中，代码点E000-F8FF的范围保留用于专用区（PUA）。PUA适用于最终用户或供应商的私人使用字符定义。

通过使用Unicode PUA作为中间形式，可以在两个Oracle数据库字符集之间转换用户定义的字符，这与标准字符相同。

12.5.4字符集之间的用户定义字符交叉引用

在操作系统中注册用户定义的字符时，需要在不同字符集之间进行交叉引用。交叉引用确保在映射到Unicode PUA值时，可以跨不同的字符集正确转换用户定义的字符。

例如，在日语Shift-JIS操作系统和日语IBM主机操作系统上注册用户定义字符时，您可能希望在Shift-JIS操作系统上分配F040代码点，并在其上分配6941代码点。用于此角色的IBM Host操作系统。这样Oracle数据库就可以在字符集JA16SJIS和JA16DBCS之间正确映射此字符。

通过使用Oracle Locale Builder查看字符集定义，可以找到用户定义的字符交叉引用信息。例如，您可以确定Shift-JIS UDC值F040和IBM Host UDC值6941都映射到相同的Unicode PUA值E000。

12.5.5从现有字符集创建新字符集的准则

默认情况下 Oracle Locale Builder为您生成下一个可用的字符集ID。您也可以选择自己的字符集ID。使用以下格式命名字符集定义NLT文件：

lx2 dddd .nlt

dddd是十六进制的4位字符集ID。

修改字符集时，请遵循以下准则：

  • 不要重新映射现有字符。

  • 所有字符映射必须是唯一的。

  • 应将新字符映射到e000-f4ff的Unicode专用范围。

    注意：

    实际的Unicode私有使用范围是e000-f8ff 。但是，Oracle数据库保留f500-f8ff用于自己的私人用途。

  • 字符集定义文件中的任何行都不能超过80个字符。

    注意：

    从现有字符集创建新的多字节字符集时，请使用8位或多字节字符集作为原始字符集。

如果从现有Oracle数据库字符集派生新字符集，则Oracle建议使用以下字符集命名约定：

< Oracle_character_set_name > < organization_name > EXT < version >

例如，如果Sun Microsystems等公司将用户定义的字符添加到JA16EUC字符集，则以下字符集名称是合适的：

JA16EUCSUNWEXT1

字符集名称包含以下部分：

  • JA16EUC是Oracle数据库定义的字符集名称

  • SUNW代表组织名称（Sun Microsystems的公司股票交易缩写）

  • EXT指定此字符集是JA16EUC字符集的扩展

12.8从早期版本的Oracle数据库升级自定义NLB文件

要将语言环境自定义文件从当前版本的数据库迁移到新版本或补丁集，请执行以下步骤：

  1. 创建一个目录并在那里复制您自定义的NLB或NLT文件。

  3. 使用新的或更新的Oracle Home中的Oracle Locale Builder打开在步骤（1）中复制的每个文件，并将它们以NLT格式保存到源目录。
  4. 按照“ 生成和安装NLB文件 ”一节中的说明重复NLB生成和安装步骤，仍然使用新版本的Oracle Locale Builder和相同的源目录。

请注意，Oracle Locale Builder可以读取和处理以前版本的NLT和NLB文件，以及从不同平台读取和处理这些文件。但是，Oracle Locale Builder始终保存NLT文件，并以最新格式生成用于已安装的Oracle数据库的NLB文件。

12.10将自定义NLB文件部署到另一个平台上的Oracle安装

NLB文件依赖于版本，与平台无关。在NLB加载期间，透明地处理二进制格式的平台相关差异（例如32位与64位，大端与小端，ASCII与EBCDIC）。因此，在将区域设置自定义文件部署到使用相同Oracle数据库版本和修补程序配置运行的其他Oracle数据库安装中时，但在不同的操作系统平台下，您可以选择以下两个选项之一：

  1. 复制自定义.NLT文件到新平台并重复NLB生成和安装步骤，如“ 生成和安装NLB文件 ”中所述 。

  2. 复制整套.NLB文件（Oracle提供的NLB文件和自定义NLB文件）到您的新平台。

注意，由于需要透明的平台处理，选项（2）可能在NLB加载时引入一些开销。但是，此开销应该可以忽略不计，因为每个NLB文件通常仅在Oracle数据库实例或Oracle客户端应用程序启动后加载一次，并且在实例或应用程序关闭之前进行缓存。此外，NLB文件按需加载。因此，在大多数安装中，只有所有可用NLB文件的一小部分被加载到内存中。

选项（2）对于为不支持Oracle Locale Builder的平台自定义文件特别有用。

  1. 使用目标Oracle Home关闭所有Oracle数据库实例和Oracle客户端应用程序。

  2. 作为拥有文件的用户（通常是用户oracle ），将所有NLB文件从目标Oracle Home的ORA_NLS10目录移动到备份目录。在UNIX平台上，输入类似于以下示例的命令：

  4. 根据需要重新启动数据库实例和/或应用程序。

13优化数据访问路径

为了实现数据密集型查询的最佳性能，物化视图和索引对于SQL语句至关重要。但是，实现这些对象并非没有代价。创建和维护这些对象可能非常耗时。空间要求可能很重要。SQL Access Advisor使您可以通过推荐实体化视图并查看特定工作负载的日志，索引，SQL配置文件和分区来优化查询访问路径。

物化视图通过将查询结果存储在单独的模式对象中来提供对表数据的访问。与不占用存储空间或包含数据的普通视图不同，实体化视图包含来自一个或多个基表或视图的查询的行。物化视图日志是一个模式对象，它记录对主表数据的更改，以便可以逐步刷新在主表上定义的物化视图。SQL Access Advisor建议如何优化实体化视图，以便快速刷新它们并使用查询重写功能。

SQL Access Advisor还建议使用位图，基于函数和B树索引。位图索引缩短了许多类型的即席查询的响应时间，并且与其他索引相比也可以减少存储空间。基于函数的索引计算涉及一列或多列的函数或表达式的值，并将其存储在索引中。B树索引通常用于索引唯一或接近唯一的密钥。

3创建分析任务

一旦捕获了要分析到SQL调优集（STS）的SQL工作负载，就可以运行SQL Performance Analyzer来分析系统更改对SQL性能的影响。 要运行SQL Performance Analyzer，必须首先创建SQL Performance Analyzer任务。任务是一个容器，它封装有关完整SQL Performance Analyzer分析的所有数据。SQL性能分析器分析包含至少两个SQL试验和比较。SQL试用版捕获特定环境条件下SQL调优集的执行性能，可以使用SQL Performance Analyzer通过以下方法之一自动生成：

  • 测试执行SQL语句

  • 生成SQL语句的执行计划

  • 引用SQL调优集中捕获的执行统计信息和计划

创建SQL Performance Analyzer任务时，您需要选择一个 SQL调优设置为其输入源。SQL调优集将用作测试执行的源或生成SQL试验的执行计划。因此，试验之间的性能差异是由环境差异引起的。

  • 使用Enterprise Manager创建分析任务

  • 使用API创建分析任务

  • 使用API配置分析任务

3.1.3使用Exadata Simulation工作流程

通过Exadata仿真工作流，您可以模拟Exadata存储服务器安装对SQL工作负载性能的影响。

Oracle Exadata提供了极大的I / O带宽，并能够将SQL处理从数据库卸载到存储。这使Oracle数据库可以显着减少通过I / O互连发送的数据量，同时将CPU资源卸载到Exadata存储单元。

SQL Performance Analyzer可以通过模拟Exadata存储服务器安装并测量SQL工作负载的I / O互连使用量的减少来分析Exadata SQL卸载处理的有效性。

运行Exadata模拟不需要对系统进行任何硬件或配置更改。选择SQL调优集后，SQL Performance Analyzer将创建一个任务并在禁用Exadata存储服务器模拟的情况下执行初始试用。然后，SQL Performance Analyzer会在启用Exadata存储服务器模拟的情况下执行第二次试用。然后，SQL Performance Analyzer使用I / O互连字节比较度量标准对两个试验进行比较，并生成SQL性能分析器报告，该报告估计如果Oracle Exadata是，则不需要从Exadata存储单元发送到数据库的数据量。正在使用。在两个SQL试验中，SQL语句执行完成，I / O互连字节测量分别作为第一次和第二次试验的实际和模拟Exadata值。测量的I / O互连字节的变化可以很好地估计Exadata存储单元中可以执行的过滤量，以及通常用于处理此数据的CPU量，但现在可以从数据库。

注意：

使用Exadata模拟不会导致任何计划更改。Exadata存储服务器安装中的执行计划不会更改，因为模拟侧重于测量I / O互连使用的改进。此外，I / O互连字节不会增加，除非使用数据压缩（请参阅下一个注释），因为Oracle Exadata只会减少发送到数据库的数据量。

注意：

由于I / O互连字节是用于衡量使用Exadata存储服务器安装对性能更改影响的唯一指标，因此如果Oracle Exadata与数据压缩一起使用，则无法正常工作。由于Exadata存储单元也对数据进行解压缩，因此SQL语句的Oracle Exadata（或第二个SQL试用版）的I / O互连字节可能大于没有Oracle Exadata（或第一个SQL试用版）的I / O互连字节。数据被压缩。这种比较会产生误导，因为SQL语句将被报告为回归;事实上，事实并非如此。

注意：

Exadata仿真工作流用于模拟非Exadata硬件上的Exadata Storage Server安装。要测试Exadata硬件上的更改，请使用标准SQL Performance Analyzer工作流。

3.3.3使用API重新分析分析任务中的多租户容器数据库标识符

13.33时间戳类

此类支持SQL标准TIMESTAMP WITH TIME ZONE和TIMESTAMP WITHOUT TIME ZONE类型，并适用于所有数据库TIMESTAMP类型： TIMESTAMP ， TIMESTAMP WITH TIME ZONE和TIMESTAMP WITH LOCAL TIME ZONE 。

Timestamp时间组件（例如小时，分钟，秒和小数部分）位于为Timestamp指定的时区中。这是10 克释放的新行为;以前的版本支持时间组件的GMT值。时间组件仅在存储在数据库中时转换为Timestamp指定的时区。例如，以下Timestamp（）调用在时区+5:30构造Timestamp值13-Nov 2003 17:24:30.0 。 

时间戳ts（env，2003,11,13,17,24,30,0,5,30）;

此调用在先前版本中的行为会将时间戳组件解释为GMT，从而导致时区值为13-Nov 2003 11:54:30.0 ，时区为+5:30 。在调用构造函数之前，用户被迫将时间戳转换为GMT。请注意，对于GMT时区，小时和分钟均等于0 。

此行为更改也适用于setDate（）和setTime（）方法。

10.1.3将大数据文件的备份划分为多个部分

如果在BACKUP命令上指定SECTION SIZE参数，则RMAN会创建一个备份集，其中每个备份片段包含来自一个文件节的块。

文件部分是文件中连续的块范围。这种类型的备份称为多节备份 。

注意：

您不能使用MAXPIECESIZE指定SECTION SIZE 。

多节备份的目的是使RMAN通道能够并行备份单个大文件。RMAN在多个通道之间划分工作，每个通道备份文件中的一个文件段。在单独的部分中备份文件可以提高大型数据文件备份的性能。

如果多分区备份成功完成，则备份期间生成的备份集不包含部分数据文件。如果多节备份不成功，则RMAN元数据可以包含部分备份集的记录。RMAN不考虑部分备份以进行还原和恢复。必须使用DELETE命令删除部分备份集。

如果指定的段大小大于文件大小，则RMAN不会对文件使用多段备份。如果指定的小部分大小将产生超过256个部分，则RMAN会将部分大小增加到一个导致恰好256个部分的值。

10.4双工备份集

RMAN最多可以同时备份四个备份集，每个副本与其他备份集完全相同。

双面备份集的副本是备份集中每个备份片的副本，每个副本都会获得唯一的副本号（例如， 0tcm8u2s_1_1和0tcm8u2s_1_2 ）。无法将备份集双工复制到快速恢复区域 。

你可以使用BACKUP ... COPIES或CONFIGURE ... BACKUP COPIES到双工备份集。RMAN可以将备份双工复制到磁盘或磁带，但不能同时将备份复制到磁带和磁盘。对于DISK通道，请在FORMAT选项中指定多个值，以将多个副本定向到不同的物理磁盘。对于SBT通道，如果使用支持SBT API第2版的介质管理器，则介质管理器会自动将每个副本写入单独的介质（例如，单独的磁带）。备份到磁带时，请确保副本数量不超过可用磁带设备的数量。

双面打印仅适用于备份集，而不适用于映像副本。指定BACKUP...是错误的BACKUP...COPIES创建映像副本备份，而当CONFIGURE...映像副本备份将忽略BACKUP COPIES设置。

10.5使用RMAN进行分割镜像备份

许多站点保留存储在磁盘上的数据库的备份，以防主数据库发生介质故障或不正确的用户操作需要时间点恢复。磁盘上的数据文件备份简化了恢复的恢复步骤，使恢复更加快速和可靠。

警告：

切勿对联机重做日志进行备份，分割镜像或其他方式。恢复联机重做日志备份可以创建两个具有相同序列号但内容不同的归档日志。此外，最好使用BACKUP CONTROLFILE命令而不是分割镜像来进行控制文件备份。

在磁盘上创建数据文件备份的一种方法是使用磁盘镜像 。例如，操作系统可以在数据库中维护每个文件的三个相同副本。在此配置中，您可以拆分数据库的镜像副本以用作备份。

RMAN不会自动拆分镜像，但可以在备份和恢复中使用分割镜像。例如，RMAN可以将数据文件的分割镜像视为数据文件副本，也可以将此副本备份到磁盘或磁带。本节中的过程说明如何使用ALTER SYSTEM SUSPEND / RESUME功能进行分割镜像备份 。

某些镜像技术不需要Oracle数据库在镜像可以分离并用作备份之前挂起所有I / O.有关是否必须从数据库实例挂起I / O的信息，请参阅存储管理器，卷管理器或文件系统文档。

使用SUSPEND / RESUME创建表空间的分割镜像备份：

  1. 启动RMAN，然后使用ALTER TABLESPACE ...将表空间放回备份模式ALTER TABLESPACE ...BEGIN BACKUP声明。（要将所有表空间置于备份模式，您可以使用ALTER DATABASE BEGIN BACKUP 。）

    例如，要将表空间users置于备份模式，请将RMAN连接到目标数据库并运行以下SQL命令：

    ALTER TABLESPACE用户BEGIN BACKUP;
  2. 如果您的镜像软件或硬件需要I / O，请暂停I / O.例如，在RMAN中输入以下命令：
    ALTER SYSTEM SUSPEND;
  3. 拆分镜像以查找这些表空间中包含的基础数据文件。
  4. 将数据库从暂停状态中取出。例如，在RMAN中输入以下命令：
    ALTER SYSTEM RESUME;
  5. 将表空间从备份模式中删除。例如，输入：
    ALTER TABLESPACE用户END BACKUP;

    您还可以使用ALTER DATABASE END BACKUP将所有表空间从备份模式中删除。

  6. 使用CATALOG命令将用户管理的镜像副本编目为数据文件副本。例如，输入：
    CATALOG DATAFILECOPY'/dk2/oradata/trgt/users01.dbf'; ＃catalog split mirror
  7. 备份数据文件副本。例如，在提示符处运行BACKUP DATAFILECOPY命令：
    BACKUP DATAFILECOPY'/dk2/oradata/trgt/users01.dbf';
  8. 当您准备好重新分割镜像时 ，首先使用CHANGE ... UNCATALOG命令用于取消您在步骤6中编目的数据文件副本的编目。例如，输入：
    CHANGE DATAFILECOPY'/dk2/oradata/trgt/users01.dbf'UNCATALOG;
  9. 为受影响的数据文件重新分配镜像。

数据库为每个加密备份使用新的加密密钥。然后，使用密码，数据库主密钥或两者对加密密钥进行加密，具体取决于所选的加密模式。单个备份加密密钥或密码永远不会以明文形式存储。

单个还原操作可以处理以不同模式加密的备份。对于要还原的每个备份片段，RMAN会检查它是否已加密。如果Oracle密钥库已打开且可用，则透明加密的备份无需干预。

如果检测到密码加密，则RMAN将在SET DECRYPTION命令中输入的密码列表中搜索匹配的密钥。如果RMAN找到可用密钥，则恢复操作继续。否则，RMAN将在Oracle密钥库中搜索密钥。如果RMAN找到可用密钥，则恢复操作继续进行;否则，RMAN发出无法解密备份片段的错误信号。

注意：

如果RMAN恢复使用不同密码创建的一组备份，则必须在SET DECRYPTION包含所有必需的密码。

RMAN加密是一项CPU密集型操作，可能会影响备份性能。加密期间的实际CPU利用率取决于磁盘和SBT的输入和输出设备是否比CPU可以加密数据更快地生成和使用数据。以下是管理和尝试最大化CPU性能的一些准则：

  • 由于加密备份比未加密备份消耗更多CPU资源，因此可以使用更多RMAN通道提高加密备份到磁盘的性能。一般规则是使用与系统中CPU核心数相同的通道数。例如，对于双核处理器使用两个通道。

  • 如果磁盘子系统和SBT子系统都很快，则可以预期CPU利用率非常高。您可能需要考虑通过设置RMAN READRATE参数来降低备份速率。例如，您可以设置块读取的上限，以便RMAN不会消耗过多的磁盘带宽，从而降低在线性能。

10.7.1关于可重新启动的备份

可重启性的最小单位是数据文件。但是，如果备份集包含一个备份片，并且如果该片包含来自多个数据文件的块，则可重新启动单元是备份片。图像副本的可重启性单位是数据文件。

可重新启动备份的好处是，如果备份生成多个备份集，则无需重新运行成功完成的备份集。但是，如果将整个数据库写入一个备份集，并且如果备份中途失败，则必须重新启动整个备份。

RMAN在读取文件或写入备份片或映像副本时遇到的任何I / O错误都会导致RMAN终止正在进行的备份作业。例如，如果RMAN尝试备份数据文件但数据文件不在磁盘上，则RMAN会终止备份。但是，如果正在使用多个通道或正在创建备份的冗余副本，则RMAN可以在没有用户干预的情况下继续备份。

RMAN只能备份自指定日期以来尚未备份的文件。备份无法备份失败备份错过的数据库部分后使用此功能。

您可以通过在BACKUP命令上指定SINCE TIME子句来重新启动备份。如果SINCE TIME晚于完成时间，则RMAN备份文件。如果在没有SINCE TIME参数的SINCE下使用BACKUP DATABASE NOT BACKED UP ，则RMAN仅备份从未备份过的文件。

10.8.1关于备份Windows

备份窗口是备份必须完成的一段时间。例如，您可能希望将数据库备份限制在系统上的用户活动较少时的时间窗口，例如凌晨2:00到6:00之间

RMAN首先备份最近最少备份的文件。默认情况下，RMAN以最大可能的速度备份文件。指定窗口并不意味着RMAN比正常情况更快地备份数据，以确保备份在窗口结束之前完成。

默认情况下，如果备份未在DURATION时间内完成，则RMAN会中断备份并报告错误。如果BACKUP命令在RUN命令中，则RUN命令终止。即使整个备份未完成，也会保留任何已完成的备份集并可用于还原操作。因此，如果重试在可用持续时间到期时中断的作业，则每次连续尝试都会覆盖更多需要备份的文件。丢弃任何不完整的备份集。

在此方案中，SAT是逻辑备用数据库，NYC是主数据库。

  1. 在SAT数据库上，发出以下语句以配置FAL_SERVER参数以启用日志文件的自动恢复。

15.2.3将逻辑备用数据库闪回到特定的应用SCN

                  

                     
备用数据库的一个好处是可以在备用数据库上执行闪回数据库，而不会影响主数据库服务。将数据库闪回到特定时间点是一项简单的任务，但是在逻辑备用数据库上，您可能希望闪回到提交已知事务之前的时间。在故障转移后使用新的主数据库配置逻辑备用数据库时可能会出现这种需求。这些步骤描述了如何使用闪回数据库和SQL应用程序恢复到已知应用的SCN。

    SQL> SELECT DBMS_LOGSTDBY.MAP_PRIMARY_SCN（PRIMARY_SCN => APPLIED_SCN） - > AS TARGET_SCN FROM DUAL;
    
                        
    2. 
                        
  2. 发出以下SQL语句以将逻辑备用数据库闪回到指定的SCN，并使用RESETLOGS选项打开逻辑备用数据库：
    SQL> SHUTDOWN; SQL> STARTUP MOUNT EXCLUSIVE; SQL> FLASHBACK DATABASE TO SCN <TARGET_SCN>; SQL> ALTER DATABASE OPEN RESETLOGS;
    
                        
    3. 
                        
  3. 发出以下查询以确认SQL Apply已应用小于或高达APPLIED_SCN 。
    SQL> SELECT APPLIED_SCN FROM V $ LOGSTDBY_PROGRESS;

    SQL> FLASHBACK STANDBY DATABASE到SCN resetlogs_change＃-2;

15.4指定NOLOGGING条款后恢复

               

                  
某些SQL语句允许您指定NOLOGGING子句，以便在联机重做日志文件中不记录该操作。
                  

                  
实际上，重做记录仍会写入联机重做日志文件，但没有与该记录关联的数据。这可能导致备用站点上的日志应用程序或数据访问错误，并且可能需要手动恢复才能恢复应用日志文件。根据您是具有逻辑备用还是物理备用，可以通过执行以下操作来避免这些错误：

                  
    
                     
  • 
                        
    逻辑备用数据库 - 在CREATE DATABASE或ALTER DATABASE语句中指定FORCE LOGGING子句。
                        
    
                     
    • 
                     
  • 
                        
    物理备用数据库 - 指定适合您计划使用Data Guard配置的方式的日志记录模式，如启用适当的日志记录模式中所述 。
                        
    
                     
    • 
                  

                  
您可以在V$DATABASE.FORCE_LOGGING列（对于CDB）或DBA_PDBS.FORCE_LOGGING列（对于PDB）中查看当前日志记录模式。

15.4.2物理备用数据库的恢复步骤

                  

                     

                        
当重做应用于物理备用数据库时，数据文件的一部分被标记为不可恢复。当您故障转移到物理备用数据库，或打开备用数据库以进行只读访问，并尝试读取标记为UNRECOVERABLE的块范围时，您会看到类似于以下内容的错误消息：
ORA-01578：ORACLE数据块损坏（文件＃1，块＃2521）ORA-01110：数据文件1：'/ oracle / dbs / stdby / tbs_1.dbf'ORA-26040：使用NOLOGGING选项加载数据块
当您使用STANDBY NOLOGGING FOR LOAD PERFORMANCE模式时，在物理备用数据库上执行的查询仍然可以报告由于nologging操作而导致的损坏块。但是，当作为Active Data Guard配置的一部分在该备用服务器上运行托管恢复时，它会自动从主服务器提取替换块。（使用ALTER DATABASE RECOVER MANAGED STANDBY DATABASE DISCONNECT语句启动托管恢复。）托管恢复尝试在首次创建后立即获取所有此类块，并继续尝试修复它们直到成功为止。每当启动托管恢复时，它都会再次开始修复早期恢复会话中的所有未完成的块。此自动恢复仅适用于由启用STANDBY NOLOGGING FOR LOAD PERFORMANCE模式生成的重做引起的损坏块。由于传统的非记录操作而损坏的块必须使用以下过程。这些步骤描述了一种恢复所有未记录块的简单方法。（有关其他方法的信息，请参阅后续部分，例如确定在不可恢复的操作之后是否需要备份，以及恢复物理备用数据库的某些部分。）
                        

                        
    
                           
  1.

15.4.4部分物理备用数据库的恢复步骤

                  

                     
RMAN RECOVER ...NONLOGGED BLOCK命令可用于恢复属于一组数据文件或一组表空间或仅一个可插拔数据库（PDB）以及多租户容器数据库（CDB）的块。
                     

                     
如果可以接受在某些数据文件中保留未记录的块，则此功能可能很有用，例如，因为明确知道非记录块是由于将行加载到现在已被删除的对象中而创建的。

                     
视图V$NONLOGGED_BLOCK通常列出每个数据文件的已知无效块的范围，并且条目作为介质恢复的一部分进行维护。但是，有时信息不完整。通常，这是在从Oracle Database 12 c第1版（12.1）之前的版本升级之后或在还原数据文件的操作系统备份之后。下次运行介质恢复时，将删除过时条目，并记录任何新的无效块，但任何先前的无效块都没有V$NONLOGGED_BLOCK条目。V$DATAFILE视图中的FIRST_NONLOGGED_SCN列仍可用于查看数据文件中至少有一个无效块，即使数据文件没有V$NONLOGGED_BLOCK条目也是如此。
                     

                     
RMAN命令VALIDATE ...NONLOGGED BLOCK可用于将V$NONLOGGED_BLOCK的条目恢复为与数据文件同步。它通过确定现有范围是否完整来执行此操作，如果不是，则会扫描必要的数据文件以识别任何无效的块，并确保它们被V$NONLOGGED_BLOCK的条目捕获。有效VALIDATE ...NONLOGGED BLOCK命令与RECOVER ...具有相同的选项RECOVER ...NONLOGGED BLOCK命令仅用于验证一组数据文件或一组表空间或PDB以及CDB。

                     

                        
注意：
V$NONLOGGED_BLOCK视图中的条目表示无效块的超集，并且可以包括一些接近无效块的正常块。例如，如果存在从块100开始并且在其中具有50个块的文件7的条目，那么这50个块中的任何一个，一些或全部都是无效的。运行VALIDATE命令后，没有范围中没有无效块，范围的第一个和最后一个块也是无效块。但是，控制文件中可以保留的范围总数存在限制，因此有时可能需要合并同一文件的范围，从而导致常规块包含在范围内。
                     

                     
如果仅验证或恢复脱机数据文件，则可以在运行RMAN命令时打开它们所属的数据库。

15.6从主数据库上的丢失写入错误中恢复

               

                  
在Oracle Data Guard配置中进行介质恢复期间，可以使用物理备用数据库检测主数据库上的丢失写入数据损坏错误。

                  

                     
这是通过将存储在主数据库上的重做日志中的块的SCN与物理备用数据库上的块的SCN进行比较来完成的。如果主数据库上的块的SCN低于备用数据库上的SCN，则主数据库上存在丢失写入错误。

                     
在这种情况下，如果在主数据库和备用数据库都启用了丢失写入检测（使用DB_LOST_WRITE_PROTECT初始化参数设置），则备用数据库的恢复尝试会导致ORA-752错误。如果未启用丢失写入检测，则恢复尝试会导致ORA-600 [3020]错误。但是，并非所有ORA-600 [3020]错误都是由于主数据库丢失造成的。因此，在遵循本节中给出的准则之前，请与Oracle支持代表合作，以确定ORA-600 [3020]错误的根本原因是否确实是在主数据库上发生的丢失写入。另请参阅http://support.oracle.com的My Oracle Support说明1265884.1中的“在待机恢复期间解析ORA-752或ORA-600 [3020]”。

                     

                        
注意：

                        
因为只有当主节点将块读入高速缓存并且稍后将相应的重做与备用数据块上的块进行比较时才会检测到丢失写入错误，所以在主数据库和备用数据库上可能还有未检测到的过时块尚未检测到已被阅读和验证。这些过时的块不会影响当前数据库的操作，因为在读取这些块之前，备用数据库会验证已用于备用数据库上当前应用的重做的SCN以进行查询或更新的所有块。

                     

                     
在备用数据库上检测到主要丢失写入错误时，将在备用数据库的警报文件中打印与每个过时块类似的以下一个或多个块错误消息：
2006年12月12日星期二19:09:48 STANDBY REDO应用程序检测到主要数据库丢失了26块的磁盘写入，文件7在SCN 389667之后或之后没有重做可以用于恢复。 。。。

然后，警报文件显示备用数据库上引发了ORA-00752错误，并取消了托管恢复：
Slave退出ORA-752异常文件中的错误/oracle/log/diag/rdbms/dgstwrite2/stwrite2/trace/stwrite2_pr00_23532.trc：ORA-00752：恢复检测到数据块丢失写入ORA-10567：重做不一致数据块（文件＃7，块＃26）ORA-10564：表空间TBS_2 ORA-01110：数据文件7：'/ oracle / dbs / btbs_21.f'ORA-10561：块类型'TRANSACTION MANAGED DATA BLOCK'，数据对象＃57503。。。

然后，在警报文件中打印的SCN上，备用数据库将恢复到一致状态，而不会由此错误导致其数据文件损坏：
恢复中断！在更改389569处将恢复的数据文件恢复为一致状态
最后一条消息可能会在警报文件中显着出现，并且可能具有比块错误消息更低的SCN。此外，主数据库可以在没有可见错误的情况下运行，即使其数据文件可能已经被破坏。

  1. 
                           
    关闭主数据库。在块错误消息中打印的SCN处或之后的所有数据都将丢失。
    
                        
    2. 
                        
  2. 
                           
    在备用数据库上发出以下SQL语句以将其转换为主数据库：
    SQL> ALTER DATABASE ACTIVATE STANDBY DATABASE;数据库改变了2006年12月12日星期二19:15:23更改数据库激活备用数据库ALTER DATABASE激活[PHYSICAL] STANDBY DATABASE（stwrite2）RESETLOGS恢复不完整后UNTIL CHANGE 389569重置resetlogs激活ID 612657558（0x24846996）在线日志/oracle/dbs/bt_log1.f ：线程1组1先前被清除在线日志/oracle/dbs/bt_log2.f：线程1组2先前被清除待机成为主要SCN：389567 Tue Dec 12 19:15:23 2006将恢复目标化身设置为3转换待机安装到主要安装。ACTIVATE STANDBY：完成 - 作为主数据库挂载的数据库（stwrite2）已完成：alter database activate standby database

15.7使用DBCOMP程序检测丢失的写入和其他不一致

               

                  
您可以使用PL / SQL过程DBMS_DBCOMP.DBCOMP来检测丢失的写入，还可以检测主数据库和物理备用数据库之间的不一致。
                  

                  
DBMS_DBCOMP.DBCOMP过程比较主数据库和物理备用数据库上的相同数据块。该程序可以随时执行。（不要求启用DB_LOST_WRITE_PROTECT初始化参数。）
                     
您可以通过查询V$SESSION_LONGOPS视图来监视正在进行的块比较操作的进度。
                     

                     

                        
注意：
逻辑备用数据库，远程同步实例和级联备用数据库不能是DBMS_DBCOMP.DBCOMP过程的目标数据库。
                     

                  

                  
                  
DBMS_DBCOMP.DBCOMP过程假定有一个主数据库和一个或多个物理备用数据库。在块比较之前，应至少安装数据库。
                  

                  
                  

                     
在以下示例情况中，假设存在具有唯一名称dgmain的主数据库，并且该物理备用数据库名为dgmainb ， dgmainc ， dgmaind等。
                     

                  

                  
                  

                     
示例15-1主服务器和所有备用服务器已安装或打开，并且从主服务器执行DBCOMP

                     
在这种情况下，当从主数据库执行DBCOMP过程时，将在主数据库和每个物理备用数据库之间逐块比较指定的数据文件。例如，假设您执行以下查询：
SQL> exec sys.dbms_dbcomp.dbcomp（'1'，'BlockCompare'，：retval）;
生成的输出文件是BlockCompare_dgmainb_1和BlockCompare_dgmainc_d_1 。
                     

                  

                  
                  

                     
示例15-2主要和所有备用数据库已安装或打开，并且从备用数据库执行DBCOMP

                     
在这种情况下，当从其中一个备用数据库（例如， dgmainb ）执行DBCOMP过程时，仅在主数据库和特定备用数据库之间比较指定的数据文件。不考虑其他备用数据库。例如，假设您执行以下查询：
SQL> exec sys.dbms_dbcomp.dbcomp（'1'，'BlockCompare'，：retval）;
生成的输出文件是BlockCompare_dgmain_1 。

15.9在不重新创建物理备用数据库的情况下更改主要字符集

               

                  
Oracle Data Guard允许您更改主数据库的数据库字符集和国家字符集，而无需在配置中重新创建任何物理备用数据库。

                  
在执行主数据库的字符集转换时，您可以继续使用物理备用数据库，同时将中断降至最低。

                  
字符集迁移过程包括准备步骤，例如扫描可能的问题和确定解决方法。在执行这些准备步骤期间，Oracle Data Guard配置可以保持不变，并且不需要额外的步骤来维护物理备用。在准备步骤完成之后，执行实际转换，其可能涉及系统数据（元数据）和用户数据的改变。必须在转换过程中运行Oracle Data Guard特有的几个过程。运行这些过程的步骤散布着由Database Migration Assistant for Unicode（DMU）或其他适当的字符集迁移工具执行的步骤。

                  
有关此过程中涉及的步骤的详细说明，请参阅http://support.oracle.com My Oracle Support说明1124165.1。

               

            
15.10在主服务器上的PDB PITR或PDB闪回后待机所需的操作

               

                  
在主服务器上执行PDB PITR或PDB闪回后，您可以还原PDB或闪回备用服务器上的PDB以使备用服务器跟随主服务器。

                  

                     

                     
使用Oracle Database Release 19c，如果备用数据库处于装载模式并且启用了闪回数据库，则可能不需要执行任何操作。在常见情况下，如果有足够的闪回数据，则备用数据库会自动闪回，以便备用数据库继续跟随主数据库。

                     
在所有其他情况下，当在主服务器上执行PDB PITR或PDB闪回，并且第一次遇到操作开始的重做时，备用服务器上的MRP终止，错误ORA-39874 ，然后是补充错误ORA-39873 。以下是警报日志中可能出现的消息示例：
由于可插拔数据库打开resetlog标记，可插拔数据库PDB1的恢复已中止。要继续恢复，请将此PDB的所有数据文件恢复到低于1437261的检查点SCN或11/15/2012 16:38:49之前的时间戳，然后重新启动恢复MRP0：后台介质恢复已终止，错误为39874 ORA-39874：可插拔数据库PDB1恢复暂停ORA-39873：将所有数据文件恢复到低于1437261的检查点SCN。


                     
使用Oracle Database Release 19c，如果备用数据库已打开且启用了闪回数据库，则可以将备用数据库置于装载模式并启动备用数据库恢复。MRP试图允许备用数据库跟随主数据库，触发备用数据库的自动闪回。如果成功，如警报日志中所报告的那样，您只需要跟进打开Active Data Guard实例。如果未触发自动闪回，或者自动闪回未导致主数据库之后的备用数据库，则可以执行本节中描述的手动步骤。

                     
在备用数据库上的介质恢复可以继续进行之前，必须还原该PDB的所有数据文件。您可以通过两种方式执行此操作：

                     
    
                        
  • 
                           
    如果在备用数据库上启用了闪回，则可以在备用数据库上使用PDB闪回，然后重新启动备用数据库管理恢复。请参阅下面的“启用闪回时执行恢复”。
    
                        
    • 
                        
  • 
                           
    如果在备用数据库上未启用闪回，则可以从在主数据库上恢复PDB的时间点之前的备份进行恢复。请参阅下面的“未启用闪回时执行恢复”。
    
                        
    • 
                     

                     
无论采用哪种方法，已经在主服务器上运行PDB PITR或闪回的PDB都无法在备用数据库上打开，直到它赶上备用数据库的其余部分。

                  

                  
                  

                     
启用闪回时执行恢复

                     
如果在备用数据库上启用了闪回，则可以在备用数据库上闪回PDB，然后重新启动备用管理恢复。

                     
    
                        
  1. 
                           
    确定受影响的PDB和PITR SCN。

    SQL> ALTER DATABASE CLOSE;
    3.

    SQL> FLASHBACK PLUGGABLE DATABASE pdb1 TO SCN 1437260;
