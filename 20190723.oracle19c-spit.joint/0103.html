JSON_query_wrapper_clause

使用此子句可以控制此函数是否将路径表达式匹配的值包装在数组包装器中 - 即将值序列括在方括号（ [] ）中。

  • 指定WITHOUT WRAPPER以省略数组包装器。仅当路径表达式与单个JSON对象或JSON数组匹配时，才能指定此子句。这是默认值。

  • 指定WITH WRAPPER以包含数组包装器。如果路径表达式匹配单个标量值（不是JSON对象或JSON数组的值）或任何类型的多个值，则必须指定此子句。

  • 指定WITH UNCONDITIONAL WRAPPER子句等同于指定WITH WRAPPER子句。提供UNCONDITIONAL关键字是为了语义清晰。

  • 仅当路径表达式匹配单个标量值或任何类型的多个值时，才指定WITH CONDITIONAL WRAPPER以包括数组包装器。如果路径表达式与单个JSON对象或JSON数组匹配，则省略数组包装器。

ARRAY关键字是可选的，提供语义清晰度。

如果函数返回单个标量值或任何类型的多个值，并且未指定WITH [UNCONDITIONAL | CONDITIONAL] WRAPPER ，然后该函数默认返回null。您可以使用JSON_query_on_error_clause覆盖此默认行为。请参阅JSON_query_on_error_clause 。

JSON_query_on_error_clause

发生以下错误时，使用此子句指定此函数返回的值：

  • 使用严格或宽松的JSON语法， expr不是格式良好的JSON数据

  • 使用SQL / JSON路径表达式评估JSON数据时找不到匹配项。您可以通过指定JSON_query_on_empty_clause来覆盖此类错误的行为。

  • 返回值数据类型不足以容纳返回字符串

  • 该函数匹配单个标量值或任何类型的多个值，以及WITH [UNCONDITIONAL | CONDITIONAL]未指定WRAPPER子句

您可以指定以下子句：

  • NULL ON ERROR - 发生错误时返回null。这是默认值。

  • ERROR ON ERROR - 发生错误时返回相应的Oracle错误。

  • EMPTY ON ERROR - 指定此子句相当于指定EMPTY ARRAY ON ERROR 。

  • EMPTY ARRAY ON ERROR - 发生错误时返回空的JSON数组（ [] ）。

  • EMPTY OBJECT ON ERROR - 发生错误时返回空JSON对象（ {} ）。

JSON_query_on_empty_clause

如果在使用SQL / JSON路径表达式计算JSON数据时未找到匹配项，请使用此子句指定此函数返回的值。此子句允许您为此类错误指定与使用JSON_query_on_error_clause指定的结果不同的结果。

您可以指定以下子句：

  • NULL ON EMPTY - 未找到匹配项时返回null。

  • ERROR ON EMPTY - 未找到匹配项时返回相应的Oracle错误。

  • EMPTY ON EMPTY -指定该子句相当于指定EMPTY ARRAY ON EMPTY 。

  • EMPTY ARRAY ON EMPTY - 未找到匹配项时返回空JSON数组（ [] ）。

  • EMPTY OBJECT ON EMPTY -返回一个空的JSON对象（ {}时，没有发现匹配。

如果省略此子句，则JSON_query_on_error_clause确定未找到匹配时返回的值。

例子

以下查询返回上下文项或指定的JSON数据字符串。路径表达式匹配单个JSON对象，该对象不需要数组包装器。请注意，JSON数据在返回值中转换为严格的JSON语法 - 也就是说，对象属性名称用双引号括起来。

SELECT JSON_QUERY（'{a：100，b：200，c：300}'，'$'）AS值FROM DUAL; VALUE ------------------------------------------------- ------------------------------- {“a”：100，“b”：200，“c”：300}

以下查询返回具有属性名称a的成员的值。路径表达式匹配标量值，该值必须包含在数组包装器中。因此，指定了WITH WRAPPER子句。 

SELECT JSON_QUERY（'{a：100，b：200，c：300}'，'$ .a'WITH WRAPPER）AS FROM FROM DUAL; VALUE ------------------------------------------------- ------------------------------- [100]

以下查询返回所有对象成员的值。路径表达式匹配多个值，这些值必须一起包含在数组包装器中。因此，指定了WITH WRAPPER子句。 

SELECT JSON_QUERY（'{a：100，b：200，c：300}'，'$。*'WITH WRAPPER）AS FROM FROM DUAL; VALUE ------------------------------------------------- ------------------------------- [100,200,300]

以下查询返回上下文项或指定的JSON数据字符串。路径表达式匹配单个JSON数组，该数组不需要数组包装器。

SELECT JSON_QUERY（'[0,1,2,3,4]'，'$'）AS值FROM DUAL; VALUE ------------------------------------------------- ------------------------------- [0,1,2,3,4]

以下查询与上一个查询类似，但指定了WITH WRAPPER子句。因此，JSON数组包装在数组包装器中。 

SELECT JSON_QUERY（'[0,1,2,3,4]'，'$'WITH WRAPPER）AS值FROM DUAL; VALUE ------------------------------------------------- ------------------------------- [[0,1,2,3,4]]

以下查询返回JSON数组中的所有元素。路径表达式匹配多个值，这些值必须一起包含在数组包装器中。因此，指定了WITH WRAPPER子句。 

SELECT JSON_QUERY（'[0,1,2,3,4]'，'$ [*]'WITH WRAPPER）AS值FROM DUAL; VALUE ------------------------------------------------- ------------------------------- [0,1,2,3,4]

以下查询返回JSON数组中索引0,3到5和7处的元素。路径表达式匹配多个值，这些值必须一起包含在数组包装器中。因此，指定了WITH WRAPPER子句。 

SELECT JSON_QUERY（'[0,1,2,3,5,6,7,8]'，'$ [0,3到5,7]'WITH WRAPPER）AS值FROM DUAL; VALUE ------------------------------------------------- ------------------------------- [0,3,4,5,7]

以下查询返回JSON数组中的第四个元素。路径表达式匹配标量值，该值必须包含在数组包装器中。因此，指定了WITH WRAPPER子句。 

SELECT JSON_QUERY（'[0,1,2,3,4]'，'$ [3]'WITH WRAPPER）AS值FROM DUAL; VALUE ------------------------------------------------- ------------------------------- [3]

以下查询返回JSON数组中的第一个元素。指定了WITH CONDITIONAL WRAPPER子句，并且路径表达式与单个JSON对象匹配。因此，返回的值不会包含在数组中。请注意，JSON数据在返回值中转换为严格的JSON语法 - 也就是说，对象属性名称用双引号括起来。 

SELECT JSON_QUERY（'[{a：100}，{b：200}，{c：300}]'，'$ [0]'WITH CONDITIONAL WRAPPER）AS FROM FROM DUAL; VALUE ------------------------------------------------- ------------------------------- {“a”：100}

以下查询返回JSON数组中的所有元素。指定了WITH CONDITIONAL WRAPPER子句，路径表达式与多个JSON对象匹配。因此，返回的值将包装在一个数组中。 

SELECT JSON_QUERY（'[{“a”：100}，{“b”：200}，{“c”：300}]'，'$ [*]'WITH CONDITIONAL WRAPPER）AS FROM FROM DUAL; VALUE ------------------------------------------------- ------------------------------- [{“a”：100}，{“b”：200}，{“ C“：300}]

以下查询与上一个查询类似，只是返回的值是数据类型VARCHAR2(100) 。 

SELECT JSON_QUERY（'[{“a”：100}，{“b”：200}，{“c”：300}]'，'$ [*]'RETURNING VARCHAR2（100）WITH CONDITIONAL WRAPPER）AS FROM FROM DUAL ; VALUE ------------------------------------------------- ------------------------------- [{“a”：100}，{“b”：200}，{“ C“：300}]

以下查询返回JSON数组中的第四个元素。但是，提供的JSON数组不包含第四个元素，这会导致错误。指定了EMPTY ON ERROR子句。因此，查询返回一个空的JSON数组。 

SELECT JSON_QUERY（'[{“a”：100}，{“b”：200}，{“c”：300}]'，'$ [3]'EMPTY ON ERROR）AS FROM FROM DUAL; VALUE ------------------------------------------------- ------------------------------- []

25.1 JDBC用户的数据库分片概述

现代Web应用程序面临着大量数据的新可扩展性挑战。这个问题的一个普遍接受的解决方案是分片。Sharding是一种数据层架构，其中数据跨独立数据库进行水平分区。在这种配置中每个数据库被称为碎片 。所有分片一起组成一个逻辑数据库，称为分片数据库 （ SDB ）。Sharding是一种无共享数据库体系结构，因为分片不共享物理资源，如CPU，内存或存储设备。

Sharding使用全局数据服务（GDS），其中GDS根据可用性，负载，网络延迟和复制滞后等参数将客户端请求路由到适当的数据库。GDS池是一组提供相同全局服务的复制数据库。GDS池中的数据库可以位于不同区域的多个数据中心中。分片GDS池包含分片数据库及其副本的所有分片，并显示为数据库客户端的单个分片数据库。

从Oracle Database 12 c第2版（12.2.0.1）开始，Oracle JDBC支持数据库分片。JDBC驱动程序识别指定的分片键和超级分片键，并连接到包含数据的相关分片。一旦与分片建立连接，就会以通常的方式支持和执行任何数据库操作，例如DML，SQL查询等。以下部分描述了本指南中使用的分片术语：

分片，碎片和分片数据库

Sharding是一种数据层架构，其中数据在独立数据库之间进行水平分区。在这种配置中的每个数据库称为碎片 。所有分片一起组成一个逻辑数据库，称为分片数据库（SDB） 。

分片键，复合分片键和超级分片键

分片键是按范围，列表或一致哈希在单级分片中使用的分区键。所有分片键一起称为复合分片键。超分片键是复合分片中使用的分区键，用于按范围或列表进行顶级分片。分片键和超分片键都可以包含一个或多个列，用于确定存储每行的分片。分片键可以是VARCHAR2，CHAR，DATE，NUMBER，TIMESTAMP等类型。

多个碎片查询

多分片查询支持路由和处理访问存储在多个分片上的数据的查询和事务。在没有分片键的情况下执行多分片查询。多分片操作用于简单的数据聚合和跨分片报告。

碎片目录

Shard Catalog是一个特殊的数据库，用于存储分片数据库和支持多分片查询。它还有助于集中管理分片数据库。

碎片总监

分片导向器是全局服务管理器（GSM）的特定实现，其充当连接到SDB并维护SDB的当前拓扑图的客户端的区域监听器。根据连接请求期间传递的分片密钥，它将连接路由到适当的分片。

块

块是表系列的每个表中的单个分区。它是分片之间数据迁移的单位。

块拆分

Chunk Split是一个过程，当块变得太大或者只有一部分块需要迁移到另一个碎片时才需要。

块迁移

块迁移是将块从一个分片移动到另一个分片的过程，当数据或工作负载偏差发生而分片数量没有任何变化时。它由DBA发起以消除热点。

Resharding

重新分片是在分片数量变化触发的分片之间重新分配数据的过程。块之间在碎片之间移动，以便在碎片上均匀分布块。但是，块的内容不会改变，也就是说，在Resharding期间不会发生重新散列。

Oracle数据库提供业界第一个也是唯一一个嵌入式OLAP服务器。在分析多个维度的数据时，Oracle OLAP提供本机多维存储和思考速度响应时间。该数据库为分析提供了丰富的支持，例如时间序列计算，预测，使用加法和非加法运算符的高级聚合以及分配运算符。这些功能使Oracle数据库成为一个完整的分析平台，能够支持整个商业智能和高级分析应用程序。

1.1.1多维技术的完全整合

通过将多维对象和分析集成到数据库中，Oracle提供了两全其美的优势：多维分析的强大功能以及Oracle数据库的可靠性，可用性，安全性和可伸缩性。

Oracle OLAP完全集成到Oracle数据库中。在技术层面，这意味着：

  • 多维数据集和其他维对象是Oracle数据字典中表示的第一类数据对象。

  • CREATE，ALTER，DROP和SELECT语句中的标准SQL语法支持多维数据集和其他维对象。

  • OLAP引擎在Oracle数据库的内核中运行。

  • 维度对象以其本机多维格式存储在Oracle数据库中。

  • 通过向Oracle数据库用户和角色授予和撤消权限，以标准方式管理数据安全性。

您组织的好处是巨大的。Oracle OLAP提供简单的功能：一个数据库，标准管理和安全性，标准接口和开发工具。

1.1.2易于应用开发

Oracle OLAP使您可以轻松地使用有趣的分析内容丰富数据库和应用程序。与提供专有接口的系统相比，对Oracle多维对象和计算的本机SQL访问极大地简化了开发任何类型的仪表板，报告，商业智能（BI）和分析应用程序的任务。此外，SQL访问意味着任何数据库应用程序都可以使用Oracle OLAP分析的强大功能，而不仅仅是传统的，有限的OLAP应用程序集合。

1.1.3易于管理

由于Oracle OLAP完全嵌入在Oracle数据库中，因此没有通常与独立OLAP服务器相关联的管理学习曲线。您可以利用现有的DBA员工，而不是投资专业的管理技能。

Oracle嵌入式OLAP技术的主要管理优势是自动化多维数据集维护。对于独立的OLAP服务器，刷新多维数据集的负担完全由管理员负责。这可能是一项复杂且可能容易出错的工作。您必须创建过程以从关系源中提取已更改的数据，将数据从源系统移动到运行独立OLAP服务器的系统，然后加载并重建多维数据集。在此过程中，您还必须对增量（更改的值）的安全性负责。

相比之下，使用Oracle OLAP，多维数据集刷新完全由Oracle数据库处理。数据库跟踪维对象的陈旧性，自动跟踪源表中的增量，并在刷新过程中自动仅应用更改的值。您只需按适当的时间间隔安排刷新，Oracle数据库将负责其他所有操作。

1.1.4安全

使用Oracle OLAP，标准Oracle数据库安全功能用于保护您的多维数据。

相反，对于独立的OLAP服务器，管理员必须管理两次安全性：一次在关系源系统上，一次在OLAP服务器系统上。此外，他们还必须管理从关系系统到独立OLAP系统的传输数据的安全性。

1.1.5无与伦比的性能和可扩展性

商业智能和分析应用程序由诸如上下层次结构和比较总体价值（如期间，父母份额，对未来时间段的预测以及无数类似计算）等行为主导。通常，这些操作在潜在的分层聚合的整个空间中基本上是随机的。由于Oracle OLAP预先计算或有效地计算定义的多维空间中的所有聚合，因此它为典型的商业智能应用程序提供了无与伦比的性能。

Oracle OLAP查询利用Oracle共享游标，大大降低了内存需求并提高了性能。

当Oracle数据库与Real Application Clusters（Oracle RAC）一起安装时，OLAP应用程序在性能，可伸缩性，故障转移和负载平衡方面可以获得与任何其他应用程序相同的优势。

所有这些功能都可以降低成本。由于可以利用现有的人员技能，因此降低了管理成本。此外，Oracle数据库可以管理维度对象的刷新，这是一项留给其他系统管理员的复杂任务。标准安全性也降低了管理成本。由于具有SQL知识的大量应用程序开发人员的可用性以及大量基于SQL的开发工具集合，因此可以更快地开发和部署应用程序，从而降低了应用程序开发成本。任何基于SQL的开发工具都可以利用Oracle OLAP。 Oracle OLAP有效管理聚合，使用共享游标和Oracle RAC，从而降低了硬件成本，从而可以从低成本的商品组件构建高度可扩展的系统。

1.3维度数据模型概述

维度对象是OLAP的组成部分。由于OLAP处于联机状态，因此必须快速提供答案;分析师在交互式会话期间提出迭代查询，而不是在隔夜运行的批处理作业中。而且因为OLAP也是分析的，所以查询很复杂。维度对象和OLAP引擎旨在实时解决复杂查询。

维对象包括多维数据集，度量，维度，属性，级别和层次结构。模型的简单性是固有的，因为它定义了代表真实世界业务实体的对象。分析师知道他们有兴趣检查哪些业务度量，哪些维度和属性使数据有意义，以及业务维度如何组织成级别和层次结构。

维度数据模型是高度结构化的。结构意味着管理数据之间关系的规则，并控制如何查询数据。多维数据集是维度模型的物理实现，因此针对维度查询进行了高度优化。OLAP引擎利用这种先天维度来执行行间计算的高效跨多维数据集连接，时间序列分析的外部联接和索引。尺寸预先连接到措施。作为立方体基础的技术基于索引的多维数组模型，该模型提供直接的小区访问。

维度模型使Oracle OLAP能够支持高端商业智能工具和应用程序 OracleBI Discoverer Plus OLAP， OracleBI电子表格插件， OracleBI Suite企业版， BusinessObjects Enterprise，和 Cognos ReportNet。

1.3.1立方体

多维数据集提供了一种组织具有相同形状的度量的方法，即它们具有完全相同的尺寸。可以轻松地分析和显示同一个多维数据集中的度量。

多维数据集通常对应于单个事实表或视图。

1.3.2措施

度量使用收集的有关业务操作的事实填充多维数据集的单元格。度量按维度组织，通常包括时间维度。

分析数据库包含历史数据的快照，这些快照源自事务数据库，遗留系统，联合源或其他数据源中的数据。通常认为三年的历史数据适用于分析应用。

在分析师使用这些措施为其决策提供信息时，措施是静态和一致的。它们定期在批处理窗口中更新：每周，每天或每天定期更新。某些管理员通过将时段添加到度量的时间维度来刷新其数据，并且还可以滚动相同数量的最旧时段。每次更新都会为该时间间隔提供特定业务活动的固定历史记录。其他管理员完全重建其数据，而不是执行增量更新。

定义度量的关键决策是最低级别的细节。用户可能永远不会查看此详细数据，但它确定了可以执行的分析类型。例如，市场分析师（不同于订单输入人员）不需要知道密歇根州安娜堡的Beth Miller在2006年7月6日下午2:34订购了10号蓝色波尔卡圆点连衣裙。但是他们可能想知道2006年夏天在美国中西部地区哪种颜色的服装最受欢迎。

基准水平决定了分析师是否可以得到这个问题的答案。对于这个特定问题，Time可以累积到几个月，Customer维度可以汇总到区域，Product维度可以汇总到具有颜色属性的项目（例如连衣裙）。但是，这一级别的汇总数据无法回答这个问题：女性一天中什么时候最有可能下订单？一个重要的决定是数据在加载到数据仓库之前的聚合程度。

计算度量返回在运行时根据存储在一个或多个度量中的数据计算的值。与关系视图一样，计算的度量将查询存储在存储在其他对象中的数据中。由于计算的度量不存储数据，因此可以在不增加数据库大小的情况下创建数十个。您可以将它们用作定义其他计算度量的基础，从而为您可以创建的计算类型增加深度。

1.3.3尺寸

维度包含一组用于标识和分类数据的唯一值。它们形成立方体的边缘，因此形成立方体内的度量。由于度量通常是多维的，因此度量中的单个值必须由每个维度的成员限定才有意义。例如，Sales度量有四个维度：时间，客户，产品和渠道。特定销售价值（43,413.50）仅在特定时间段（2006年2月），客户（Warren Systems），产品（便携式PC）和渠道（目录）合格时才有意义。

基本维度值对应于事实表的唯一键。

度量维度是将度量作为维度成员的维度。使用度量维度，您可以同时为多维数据集中的所有度量生成计算度量。此外，您不必为添加到多维数据集的每个度量创建一组新的计算度量。现有的计算度量适用于度量维度中的新度量。如果您经常创建新度量，这尤其有用。

1.3.4层次结构和级别

层次结构是一种在不同聚合级别组织数据的方法。在查看数据时，分析师使用维度层次结构来识别一个级别的趋势，深入到较低级别以确定这些趋势的原因，并汇总到更高级别以查看这些趋势对业务的更大部分有何影响。

1.3.4.1基于级别的层次结构

每个级别代表层次结构中的位置。基础（或最详细）级别之上的每个级别都包含其下级别的聚合值。不同级别的成员具有一对多的亲子关系 。例如， Q1-05和Q2-05是孩子2005 ，因此2005是父Q1-05和Q2-05 。

假设数据仓库包含每天三次（即每8小时）采集数据的快照。分析师通常可能更愿意查看已汇总为数天，数周，季度或年的数据。因此，时间维度需要具有至少五个级别的层次结构。

同样，具有特定目标的销售经理可能希望在其领土内的销售代表中分配该目标金额;分配需要维度层次结构，其中各个销售代表是特定区域的子值。

层次结构和级别具有多对多关系。层次结构通常包含多个级别，单个级别可以包含在多个层次结构中。

每个级别通常对应于维度表或视图中的列。基本级别是主键。

1.3.4.2基于值的层次结构

虽然层次结构通常由命名级别组成，但它们并非必须如此。维度成员之间的父子关系可能无法定义有意义的级别。例如，在员工维度中，每个经理都有一个或多个报表，这些报表形成父子关系。基于这些关系创建级别（例如个人贡献者，一级管理者，二级管理者等）可能对分析没有意义。同样，财务数据的订单项维度没有级别。这种类型的层次结构称为基于值的层次结构。

1.3.5属性

属性提供有关数据的其他信息。某些属性用于显示。例如，您可能有一个产品维度使用库存保留单位（SKU）作为维度成员。SKU是识别数千种产品的绝佳方式，但如果用于在报告或图表中标记数据，则对大多数人来说毫无意义。您可以定义描述性标签的属性。

您可能还有颜色，味道或大小等属性。这种类型的属性可用于数据选择和回答问题，例如：2005年夏季哪些颜色最受女性服装欢迎？这与前一个夏天相比如何？

时间属性可以提供有关时间维度的信息，这些信息在某些类型的分析中可能很有用，例如识别每个时间段的最后一天或天数。

每个属性通常对应于维度表或视图中的列。

一个用户管理本地域索引

在引入系统管理分区的Oracle Database 11g第1版之前，用户管理的分区域索引方法是唯一可用的方法。用户管理的方法有三个重要的局限性：

  • 由于可扩展索引框架不存储有关内核中与域索引相关的对象的信息，因此必须通过调用用户提供的例程来维护表和分区。

  • 由于内核不支持均分表，因此每个分区都必须具有一组表和从属模式对象，这些对象必须在用户管理的索引代码中以编程方式进行管理。

    随着分区数量的增加，域索引存储对象的激增可能成为高效操作的障碍。要使用包含图像并具有1,000个分区的表，在其存储表上创建64位图索引的索引模式（在扩展为支持本地域索引之后）将需要创建和管理1,000个域索引存储表和64,000个位图索引。

  • 在使用本地域索引进行DML和查询处理期间，每个分区都需要一组单独的游标;这是必需的，因为每个分区都有自己的一组表。因此，使用大量分区并同时需要访问多个分区的应用程序必须在运行时编译新的SQL游标，这会影响性能。

Oracle建议您使用系统管理的方法，如构建域索引中所述 。

Oracle计划在将来的版本中弃用用户管理的方法。本附录中提供的信息记录了用户管理的进程和API之间的特定差异。

A.1比较用户管理和系统管理的域索引

另一种方法是使用系统管理的域索引。它解决了这些限制，具有以下优点：

  • 由于内核代表用户执行了更多维护任务，因此不需要对表和分区维护操作进行编程支持。可以通过在服务器中执行操作并使用极少量的接口例程来实现这些操作。然后，盒代码可能相对不知道分区问题。

  • 必须管理以支持本地分区域索引的对象数与非分区域索引的数量相同。对于本地分区索引，域索引存储表是针对基表进行分配的。因此，域索引存储表的数量不会随着分区数量的增加而增加。

  • 现在，一组查询和DML语句可以访问和操作系统分区存储表，从而促进游标共享和增强性能。

注意：

包含包装的PL / SQL源文本的文件称为包装文件 。可以通过SQL * Plus或导入和导出实用程序移动，备份或处理包装文件。

要生成包装文件，请使用PL / SQL Wrapper实用程序或DBMS_DDL子程序。PL / SQL Wrapper实用程序包装由指定SQL文件创建的每个可包装PL / SQL单元的源文本。DBMS_DDL子程序包装单个动态生成的可包装PL / SQL单元的源文本。

PL / SQL Wrapper实用程序和DBMS_DDL子程序都检测标记化错误（例如，失控字符串），但不检测语法或语义错误（例如，不存在的表或视图）。

A.1 PL / SQL源文本环绕限制

  • Oracle数据库版本之间的包装文件不向下兼容。

    例如，您无法将版本为 .1 PL / SQL Wrapper实用程序生成的文件加载到版本（ n -1）.2 Oracle数据库中。您也可以加载版本N 0.2 PL / SQL包装实用为版本n 0.1 Oracle数据库生成的文件。包装文件在补丁集中向上和向下兼容。

  • 包装PL / SQL源文本不是隐藏密码或表名的安全方法。

    要获得高可靠性安全性，请使用Oracle Database Vault管理员指南中所述的Oracle Database Vault 。

  • 您无法包装触发器的PL / SQL源文本。

    要隐藏触发器的实现细节，请将它们放在存储的子程序中，包装子程序，并写入调用子程序的单行触发器。

A.2 PL / SQL源文本环绕指南

  • 仅包装包装或类型的主体，而不是规格。

    将规范解包后，其他开发人员可以查看使用该包或类型所需的信息（参见示例A-5 ）。包裹身体可以防止他们看到包或类型实现。

  • 仅在完成编辑后包装文件。

    您无法编辑包装文件。如果包装文件需要更改，则必须编辑原始的未包装文件，然后将其包装。

  • 在分发包装文件之前，请在文本编辑器中查看它，并确保包装所有重要部分。

A.3使用PL / SQL包装器实用程序包装PL / SQL源文本

PL / SQL Wrapper实用程序获取单个SQL文件（例如SQL * Plus脚本）并生成一个等效的文本文件，其中包含每个可包装的PL / SQL单元的PL / SQL源文本。

注意：

Oracle建议使用PL / SQL Wrapper Utility版本10或更高版本。

有关可换行PL / SQL单元的列表，请参阅“ PL / SQL源文本换行 ”简介 。

PL / SQL Wrapper实用程序无法连接到Oracle数据库。要运行PL / SQL Wrapper实用程序，请在操作系统提示符下输入此命令（等号周围没有空格）：

wrap iname = input_file [oname = output_file ] [keep_comments = yes]

input_file是包含任何SQL语句组合的现有文件的名称。 output_file是PL / SQL Wrapper实用程序创建的文件的名称 - 包装文件。

注意：

input_file不能包含使用SQL * Plus DEFINE表示法指定的替换变量，因为output_file由PL / SQL编译器解析，而不是由SQL * Plus解析。

除非指定了keep_comments=yes否则PL / SQL Wrapper实用程序将删除包装文件中的所有注释。当指定keep_comments=yes ，仅保留源外的注释。

注意：

如果input_file是包装文件，则input_file和output_file具有相同的内容。

input_file的默认文件扩展名是sql 。output_file的默认名称是input_file .plb 。因此，这些命令是等效的：

wrap iname = / mydir / myfile wrap iname = / mydir / myfile.sql oname = / mydir / myfile.plb

此示例为input_file指定了不同的文件扩展名，并为output_file指定了不同的名称：

wrap iname = / mydir / myfile .src oname = /yourdir/yourfile.out keep_comments = yes

您可以在SQL * Plus中将output_file作为脚本运行。例如：

SQL> @ myfile.plb;

A.4使用DBMS_DDL子程序包装PL / SQL源文本

DBMS_DDL包提供WRAP函数和CREATE_WRAPPED过程，每个过程包装单个动态生成的可装入PL / SQL单元的PL / SQL源文本。DBMS_DDL包还提供异常MALFORMED_WRAP_INPUT （ORA-24230），如果WRAP或CREATE_WRAPPED的输入不是有效的可装入PL / SQL单元，则会引发该异常。（有关可换行PL / SQL单元的列表，请参阅“ PL / SQL源文本换行 ”简介 。）

每个WRAP函数都将一个CREATE语句作为输入，该语句创建一个可装入的PL / SQL单元并返回一个等效的CREATE语句，其中包含PL / SQL源文本。有关WRAP函数的更多信息，请参见Oracle Database PL / SQL包和类型参考 。

警告：

如果传递DBMS_DDL语句。 WRAP返回DBMS_SQL 。 PARSE过程的形式参数statement数据类型为VARCHAR2A ，那么必须设置DBMS_SQL的lfflg参数。 PARSE到FALSE 。否则， DBMS_SQL 。 PARSE将行添加到包装的PL / SQL单元中，从而破坏它。（有关语法DBMS_SQL 。 PARSE ，请参阅Oracle数据库PL / SQL包和类型参考 。）

每个CREATE_WRAPPED过程执行其相应的WRAP函数所执行的操作，然后运行返回的CREATE语句，从而创建指定的PL / SQL单元。有关CREATE_WRAPPED过程的详细信息，请参阅Oracle Database PL / SQL包和类型参考 。

小费：

当调用一个DBMS_DDL子程序，使用完全合格的包名， SYS.DBMS_DDL ，以避免名称冲突，如果有人创建了一个名为本地包DBMS_DDL或定义了公共同义词DBMS_DDL 。

注意：

输入到WRAP函数或CREATE_WRAPPED过程的CREATE语句以调用子程序的用户的权限运行。

2.2.1关于对象方法

对象方法（也称为子程序）是可以在对象类型定义中声明的函数或过程，以实现您希望该类型的对象执行的行为。应用程序调用子程序来调用该行为。

子程序可以用PL / SQL或几乎任何其他编程语言编写。用PL / SQL或Java编写的方法存储在数据库中。用其他语言编写的方法（如C）存储在外部。

注意：

SQL需要括号用于所有子程序调用，甚至是那些没有参数的子程序。PL / SQL不是这样。

成员方法为应用程序提供对对象实例数据的访问。

2.2.2.2比较对象的成员方法

要比较和排序对象类型的变量，必须指定比较它们的基础。

标量数据类型（如CHAR或REAL具有预定义的顺序，允许对它们进行比较。但是对象类型（例如person_typ ）可以具有各种数据类型的多个属性，没有预定义的比较轴。您可以选择定义用于比较对象的map方法或order方法，但不能同时定义两者。

map方法将对象返回值映射到标量值，并可以通过它们在标量轴上的位置对多个值进行排序。订单方法直接比较两个特定对象的值。

2.2.2.2.1关于地图方法

2.2.2.2.4订购方法

订单方法可以直接进行一对一的对象比较。

与map方法不同，order方法无法确定多个对象的顺序。根据使用的标准，它们只是告诉您当前对象小于，等于或大于与其进行比较的对象。

order方法是对象（ SELF ）的函数，其中一个声明的参数是相同类型的对象。该方法必须返回负数，零或正数。此值表示对象（隐式未声明的SELF参数）小于，等于或大于声明的参数对象。

与map方法一样，只要需要比较该类型的两个对象，就会自动调用order方法（如果已定义）。

在比较语义可能过于复杂而无法使用map方法的情况下，Order方法很有用。

2.2.2.2.5比较方法指南

您可以声明map方法或order方法，但不能同时声明两者。

对于地图和订单类型方法，您可以使用SQL语句和PL / SQL过程语句来比较对象。但是，如果不声明这些方法之一，则只能比较SQL语句中的对象，并且只能比较相等或不等。只有当相应属性的值相等时，才认为两个相同类型的对象是相等的。

在排序或合并大量对象时，使用map方法将所有对象映射到标量中，然后对标量进行排序。订单方法的效率较低，因为必须重复调用它（它一次只能比较两个对象）。

2.2.2.2.6类型层次结构中的比较方法

在类型层次结构中，如果根类型（超类型）未指定映射或订单方法，则子类型也不能。

  • 类型层次结构中的映射方法

    如果根类型指定了map方法，则其任何子类型都可以覆盖它。如果根类型未指定map方法，则任何子类型都不能指定一个。

  • 类型层次结构中的顺序方法

    只有根类型才能定义订单方法。如果根类型没有定义一个，则其子类型不能添加一个。

2.2.3声明和调用静态方法

COPY显示的前三条消息显示影响COPY的SET命令变量的值操作。最重要的是长 ，它限制了LONG列值的长度。（LONG是一种数据类型，类似于CHAR。）如果源表包含LONG列，则COPY会将该列中的值截断为系统变量LONG指定的长度。

变量 ARRAYSIZE 范围 SQL * Plus一次从数据库中提取的行数。这个行数组成一个批次。变量 COPYCOMMIT 套 COPY提交更改到数据库之后的批次数。（如果将COPYCOMMIT设置为零，则COPY仅在复制所有批次后提交更改。）有关SET变量的更多信息，包括如何更改其设置，请参阅SET命令。

列出三个系统变量及其值后，COPY会告诉您在复制期间是否删除，创建或更新了表。然后COPY列出选择，插入和提交的行数。

B.2.4指定另一个用户的表

您可以参考其他用户的表格在COPY中命令通过使用用户名限定表名，就像在本地数据库中或在具有数据库链接的查询中一样。

例如，要在与Oracle Net连接标识符BOSTONDB关联的数据库上创建由用户名ADAMS拥有的名为DEPARTMENT的表的本地副本，您将输入

从HR @ BOSTONDB复制 - 创建员工_COPY2  - 使用SELECT *来自ADAMS.DEPARTMENT

当然，您可以通过指示COPY以ADAMS身份登录到远程数据库来获得相同的结果。但是，除非您知道与用户名ADAMS关联的密码，否则不能这样做。

B.3关于在一个数据库的表之间复制数据

你可以复制数据从一张桌子在一个数据库中的另一个 （本地或远程）。要在本地数据库中的表之间进行复制，请指定自己的表用户名和服务名称对于FROM或TO子句中的本地数据库（省略其他子句）：

复制HR @ MYDATABASE  -  INSERT EMPLOYEE_COPY2  - 使用SELECT * from EMPLOYEE_COPY

要在远程数据库的表之间进行复制，请在FROM和TO子句中包含相同的用户名和服务名：

从HR @ BOSTONDB复制 - 到HR @ BOSTONDB  -  INSERT EMPLOYEE_COPY2  - 使用SELECT * from EMPLOYEE_COPY
