31.2.4关于分区模型

引入分区模型来组织和表示多个模型。

Oracle Data Mining支持构建持久的Oracle Data Mining分区模型。分区模型将单个模型组织并表示为单个模型实体中的分区，使用户能够轻松构建和管理针对独立数据片段定制的模型。持久意味着分区模型具有磁盘表示。该产品管理分区模型的组织，并简化了对分区模型进行评分的过程。在评分时，您必须将分区列包含在USING子句中。

分区名称，键值和分区模型的结构在ALL_MINING_MODEL_PARTITIONS视图中可见。

31.2.4.1分区模型构建过程

要构建分区模型，Oracle Data Mining需要分区键。分区键通过设置表中的构建设置进行设置。

分区键是来自输入数据集的一个或多个列（最多16个）的逗号分隔列表。分区键基于分区键的离散值水平地对输入数据进行切片。也就是说，分区是作为列表值执行的，而不是针对连续值的范围分区。分区键仅支持数据类型为NUMBER和VARCHAR2 。

在构建过程中，输入数据集根据指定键的不同值进行分区。每个数据切片（唯一键值）都会产生自己的模型分区。此结果模型分区不是独立的，并且作为独立模型不可见。分区模型的最大分区数的默认值为1000分区。您还可以设置不同的最大分区值。如果输入数据集中的分区数超过定义的最大值，Oracle Data Mining将引发异常。

分区模型组织所有分区和分区特定功能共有的功能。常见功能包括以下元数据：

  • 型号名称

  • 挖掘功能

  • 挖掘算法

  • 所有分区引用的所有挖掘模型属性的超集（签名）

  • 一组通用的用户定义列转换

  • 任何用户指定的或默认的构建设置，都被解释为全局。例如，自动数据准备（ADP）设置

31.2.4.2分区模型中的DDL

通过以下DDL操作维护分区模型：

  • 删除模型或删除分区

  • 添加分区

31.2.4.2.1丢弃模型或丢弃分区

Oracle Data Mining支持为给定的分区名称删除单个模型分区。

如果只剩下一个分区，则无法显式删除该分区。相反，您必须在删除分区之前添加其他分区，或者您可以选择删除模型本身。删除分区模型时，将在单个原子操作中删除所有分区。从性能角度来看，由于DROP_PARTITION选项的有效行为，Oracle建议使用DROP_PARTITION后跟ADD_PARTITION而不是利用REPLACE选项。

31.2.4.2.2添加分区

Oracle Data Mining支持将单个分区或多个分区添加到现有分区模型。

添加基于输入数据集和现有分区模型的名称进行。该操作将输入数据集和现有分区模型作为参数。从输入数据集中提取分区键，并根据输入数据集构建模型分区。这些分区将添加到分区模型中。如果新分区的分区键与模型中的现有分区冲突，则可以从以下三种方法中进行选择以解决冲突：

  • ERROR ：终止ADD操作而不添加任何分区。

  • REPLACE ：替换找到冲突键的现有分区。

  • IGNORE ：消除具有冲突密钥的行。

如果输入数据集包含多个键，则操作将创建多个分区。如果模型中的分区总数增加到超过创建模型时指定的用户定义的最大值，则会出现错误。分区数的默认阈值为1000 。

31.2.4.3分区模型评分

了解分区模型的评分。

分区模型的评分与非分区模型的评分相同。数据挖掘功能的语法保持不变，但扩展为您提供可选提示。可选提示可能会影响查询的性能，该查询涉及对分区模型进行评分。

对于对分区模型进行评分，在构建分区键期间使用的签名列必须存在于评分数据集中。这些列组合在一起形成一个唯一的分区键。然后将唯一键映射到特定的底层模型分区，并使用标识的模型分区对该行进行评分。

评分所需的分区对象在查询执行期间按需加载，并根据系统全局区域（SGA）内存进行老化。

14.1关于软件管理和补丁版本

软件管理涉及使用最新的产品修复程序使Oracle数据库软件保持最新。 发现产品缺陷或错误时 ，会创建一个修补程序来解决问题。补丁修正了已安装软件中的单个缺陷。单个补丁（也称为临时补丁）可供客户使用，这些客户出于业务原因不能等到下一个补丁集接收产品修复程序。

Oracle以补丁集的形式定期发布其软件的维护版本。补丁集是在维护版本发布之前一直发布的产品修订集合。补丁集是经过全面测试和集成的产品修复。补丁集中的所有产品修复程序均已经过测试，并且经过认证可以相互协作。

每个补丁或补丁集都与错误号相关联，以便识别。补丁集也与版本号相关联。例如，如果使用Oracle Database 12 c第1版（12.1.0.1），则可用的修补程序集为12.1.0.2。如果应用临时补丁，则修补软件的版本号不会更改。

每个补丁都有一个相关的README文件，描述了它如何修复软件。README文件还包含应用补丁的说明。

每个补丁集都附带一个Patch Set Notes文件，其中包含安装说明以及有关补丁集中包含的产品修订的信息。将修补程序集应用于Oracle软件时，可以更改已安装软件的维护版本号。应用修补程序集仅影响驻留在Oracle主目录中的软件，而不会更改数据库中的数据。

您可以使用Oracle OPatch和Oracle Universal Installer（OUI）分别应用修补程序和修补程序集。或者，您可以使用Oracle Enterprise Manager Cloud Control自动执行这些修补程序的应用程序。使用这些产品超出了本文档的范围。

B Oracle XML DB限制

此处列出了与Oracle XML DB相关的限制。

  • 某些 XMLType 函数 不支持精简JDBC驱动程序 - XMLType方法transform()仅适用于OCI驱动程序。不是所有的oracle.xdb.瘦JDBC驱动程序支持XMLType函数。如果你不使用oracle.xdb.XMLType类和OCI驱动程序，可以失去性能优势。

  • 不支持NCHAR，NVARCHAR2和NCLOB - Oracle XML DB不支持将SQL数据类型NCHAR ， NVARCHAR2和NCLOB用于以下任何方面：

    • 使用XML模式中的SQLType注释将XML元素或属性映射到这些数据类型

    • 使用SQL / XML函数XMLElement ， XMLAttributes和XMLForest从这些数据类型生成XML数据

    • 在SQL / XML函数XMLQuery和XMLTable ，使用XQuery函数ora:view （ desupported ）， fn:doc和fn:collection包含包含这些数据类型的列的表

    为了处理，存储或生成包含多字节字符的XML数据，Oracle强烈建议您使用AL32UTF8作为数据库字符集。

  • XML标识符长度限制 - Oracle XML DB仅支持最大32767字节或4000字节的XML标识符，具体取决于初始化参数MAX_STRING_SIZE的值。请参见Oracle数据库SQL语言参考 。

  • 存储库文件大小限制 - Oracle XML DB存储库中文件的最大大小为4千兆字节。这意味着对不同类型的文件数据有以下限制：

    • 任何LOB为4千兆字节，这意味着存储在数据库字符集AL32UTF8中的CLOB 2个千兆字节。

    • 用于二进制XML编码数据的4千兆字节，通常意味着在编码之前超过4千兆字节的外部XML数据。

    • 对于以对象方式存储的XML数据不确定。

  • 存储库范围的资源配置文件限制 - 您无法为存储库范围的配置创建超过125个资源配置文件。

  • 递归文件夹删除 - 使用递归删除选项无法删除超过50级嵌套文件夹。

  • 否列级加密 XMLType不支持列级加密- XMLType 。所有XMLType存储模型都支持表空间级加密。

  • 没有 XMLType复合分区 - XMLType表或列不支持复合分区（无论XMLType存储模型如何）。

  • 没有为分层启用的表分区 - 您无法对启用了层次结构的表进行分区。（有关启用层次结构的表的信息，请参阅存储库资源和数据库表安全性 。）

  • 没有针对 XMLType Oracle Real Application Testing（RAT） - XMLType不支持Oracle Real Application Testing（RAT）。

  • 无 数据库链接的 XMLType 访问 - 不支持访问远程XMLType表或列。

  • 某些功能需要Oracle JVM - 通常，Oracle XML DB的行为不依赖于您是否安装了Oracle JVM（Java虚拟机）。但是，如果使用Java servlet，则必须安装Oracle JVM。

  • 编辑视图与 XMLType 不兼容 - 编辑视图与以对象关系存储的XMLType数据不兼容。无法在包含持久对象类型的数据库模式中启用它们。

  • 可传输表空间和数据库整合 - 如果Oracle Database 11g第2版（11.2）中的Oracle XML DB存储库具有现有数据，则无法使用可传输表空间将该数据库直接插入容器数据库（CDB）。相反，将11.2数据库升级到12.1，拔掉它，然后将其插入。

  • 没有 XMLType 的对象关系存储虚拟列XMLType -你无法在虚拟添加XMLType列的对象关系XMLType表或到有一个对象-关系表中XMLType列。

  • XMLType或存储库没有分片 - XMLType数据或Oracle XML DB存储库不支持分片 。

  • 使用堆表而不是索引组织表（IOT） - 为了获得最佳结果，Oracle建议您使用堆表而不是IOT与Oracle XML DB。

  • XML架构URL唯一忽略协议部分 -属性schemaurl用于注册的XML模式必须是唯一的距协议的一部分。在测试唯一性时忽略协议部分（例如， http或https ）。

4.8 fileNameConversionSpec

目的

使用fileNameConversionSpec子句指定一个或多个模式，用于根据旧文件名生成新文件名。用于 BACKUP ， CONVERT ，和 DUPLICATE是生成输出文件名的一种方法。

使用说明

字符串模式的规则以及它们如何影响文件命名等于初始化参数的规则 DB_FILE_NAME_CONVERT 。在括号中，提供偶数个字符串模式。

当RMAN基于旧文件名生成新文件名时，它会将原始文件名与每对字符串模式的第一个成员进行比较。RMAN第一次找到作为原始文件名的子字符串的模式时，RMAN通过将该对的第二个成员替换为匹配的子字符串来生成新文件名。

模式不必在文件名的开头匹配。以下命令将数据文件/disk1/dbs/users.dbf的映像副本创建为/disk1/newdbs/users.dbf ：

备份为COPY DB_FILE_NAME_CONVERT =（'dbs'，'newdbs'）; TABLESPACE用户;

当存在转换的给定文件名的多个可能匹配时，RMAN使用模式列表中的第一个匹配来生成新文件名。以下命令与前一个示例具有相同的效果，因为模式dbs与文件名匹配，因此文件名永远不会与第二个模式/disk1 ：

备份为COPY DB_FILE_NAME_CONVERT =（'dbs'，'newdbs'，'/ disk1'，'/ newdisk'）TABLESPACE用户;

对于CONVERT TABLESPACE ， CONVERT DATABASE和BACKUP AS COPY命令，如果这些操作的源文件是Oracle管理的文件，则不能使用fileNameConversionSpec将源文件名转换为新的输出文件名。对于Oracle管理的文件，无论是在自动存储管理（ASM）还是在普通文件系统存储中，都必须允许数据库生成输出文件的文件名。例如，非ASM存储中的数据文件的OMF文件名可能具有以下形式：

/private/boston/datafile/01_mf_system_ab12554_.dbf

一个 ASM存储中的OMF文件名可能具有以下形式：

+ DISK /波士顿/数据文件/ system.256.4543080

只有数据库才能生成和管理OMF文件名。通常，将不同磁盘组或不同OMF位置的名称替换为OMF文件名不会在新目标中生成有效的文件名。要转换OMF文件名以便存储在另一个OMF位置，请使用带有这些命令的替代方法（如FORMAT子句）来指定新的输出位置，并允许数据库管理特定的输出文件名。

14.1更改功能声明

ALTER FUNCTION语句显式重新编译独立函数。

显式重新编译消除了对隐式运行时重新编译的需要，并防止了相关的运行时编译错误和性能开销。

注意：

此声明不会更改现有函数的声明或定义。要重新声明或重新定义独立函数，请使用带有OR REPLACE子句的“ CREATE FUNCTION语句 ” 。

例

例14-1重新编译函数

要显式重新编译示例用户oe拥有的函数get_bal ，请发出以下语句：

ALTER FUNCTION oe.get_bal COMPILE;

如果数据库在重新编译get_bal时没有遇到编译错误，则get_bal变为有效。数据库随后可以运行它，而无需在运行时重新编译它。如果重新编译get_bal导致编译错误，那么数据库将返回错误，并且get_bal仍然无效。

该数据库还使依赖于get_bal所有对象无效。如果您随后引用其中一个对象而未先显式重新编译它，则数据库会在运行时隐式重新编译它。
