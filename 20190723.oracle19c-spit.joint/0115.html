1.1关于Oracle Data Mining

了解Oracle Data Mining的用法并了解不同的挖掘技术。

Oracle Data Mining在Oracle数据库中提供了强大的，最先进的数据挖掘功能。您可以使用Oracle Data Mining构建和部署预测性和描述性数据挖掘应用程序，为现有应用程序添加智能功能，以及为数据探索生成预测性查询。

Oracle Data Mining提供了一套全面的数据库内算法，用于执行各种挖掘任务，例如分类，回归，异常检测，特征提取，聚类和市场购物篮分析。算法可以处理标准案例数据，事务数据，星型模式以及文本和其他形式的非结构化数据。Oracle Data Mining非常适合挖掘超大型数据集。

Oracle Data Mining是Oracle数据库企业版Oracle高级分析选件的两个组件之一。另一个组件是Oracle R Enterprise，它将开源统计环境R与Oracle数据库集成在一起。Oracle Data Mining和Oracle R Enterprise共同构成了一个全面的高级数据分析分析平台。

1.2数据库内核中的数据挖掘

了解数据挖掘的实现。

Oracle Data Mining在Oracle数据库中实现核心。数据挖掘模型是一流的数据库对象。Oracle数据挖掘过程使用Oracle数据库的内置功能来最大化可扩展性并有效利用系统资源。

Oracle数据库中的数据挖掘提供了许多优势：

  • 无数据移动：某些数据挖掘产品要求从公司数据库导出数据并将其转换为专门的挖掘格式。使用Oracle Data Mining，无需数据移动或转换。这使得整个挖掘过程不那么复杂，耗时且容易出错，并且允许挖掘非常大的数据集。

  • 安全性：您的数据受Oracle数据库广泛的安全机制保护。此外，不同的数据挖掘活动需要特定的数据库权限。只有具有相应权限的用户才能定义，操作或应用挖掘模型对象。

  • 数据准备和管理：大多数数据必须在开采之前以各种方式进行清理，过滤，标准化，采样和转换。数据挖掘项目中高达80％的工作量通常用于数据准备。Oracle Data Mining可以自动管理数据准备过程中的关键步骤。此外，Oracle数据库还提供了用于准备和管理数据的广泛管理工具。

  • 易于数据刷新：Oracle数据库中的挖掘过程可以随时访问刷新的数据。Oracle Data Mining可以根据当前数据轻松提供挖掘结果，从而最大限度地提高其及时性和相关性。

  • Oracle数据库分析：Oracle数据库为高级分析和商业智能提供了许多功能。Oracle Data Mining可以轻松地与数据库的其他分析功能集成，例如统计分析和OLAP。

  • Oracle技术栈：您可以利用Oracle技术堆栈的各个方面，将数据挖掘集成到更大的商业智能或科学查询框架中。

  • 域环境：必须在适当的应用程序域环境中构建，测试，验证，管理和部署数据挖掘模型。数据挖掘结果可能需要作为域特定计算的一部分进行后处理（例如，计算估计的风险和响应概率），然后存储到永久存储库或数据仓库中。使用Oracle Data Mining，采矿前后的活动都可以在同一环境中完成。

  • 应用程序编程接口：PL / SQL API和SQL语言运算符提供对Oracle数据库中Oracle数据挖掘功能的直接访问。

1.3 Oracle Exadata中的数据挖掘

了解Oracle Exadata中的评分。

评分是指将数据挖掘模型应用于数据以生成预测的过程。该评分过程可能需要大量的系统资源。可能涉及大量数据，并且算法处理可能非常复杂。

借助Oracle Data Mining，可以将评分卸载到处理效率极高的智能Oracle Exadata存储服务器。

Oracle Exadata存储服务器结合了Oracle的智能存储软件和Oracle的行业标准Sun硬件，可提供业界最高的数据库存储性能。有关Oracle Exadata的更多信息，请访问Oracle技术网。

1.4关于分区模型

引入分区模型来组织和表示多个模型。

Oracle Data Mining支持构建持久的Oracle Data Mining分区模型。分区模型将单个模型组织并表示为单个模型实体中的分区，使用户能够轻松构建和管理针对独立数据片段定制的模型。持久意味着分区模型具有磁盘表示。该产品管理分区模型的组织，并简化了对分区模型进行评分的过程。在评分时，您必须将分区列包含在USING子句中。

1.5.4预测分析

预测分析是一种在简单例程中捕获数据挖掘过程的技术。

有时被称为“一键式数据挖掘” 预测分析简化并自动化数据挖掘过程。

预测分析使用数据挖掘技术，但使用预测分析不需要数据挖掘知识。您只需指定要对数据执行的操作即可使用预测分析。您无需创建或使用挖掘模型或了解“Oracle数据挖掘基础知识”中汇总的挖掘函数和算法。

Oracle数据库如何解析模式对象引用

当您在SQL语句中引用对象时，Oracle会考虑SQL语句的上下文并将对象定位在相应的命名空间中。找到对象后，Oracle会执行该对象上的语句指定的操作。如果在相应的命名空间中找不到命名对象，则Oracle会返回错误。

以下示例说明了Oracle如何解析对SQL语句中对象的引用。考虑这个语句，它将一行数据添加到由名称departments标识的表中：

插入部门VALUES（280，'ENTERTAINMENT_CLERK'，206,1700）;

根据声明的上下文，Oracle确定departments可以是：

  • 您自己的架构中的表

  • 您自己的架构中的视图

  • 表或视图的私有同义词

  • 公共同义词

在考虑模式之外的名称空间之前，Oracle总是尝试在您自己的模式中解析名称空间内的对象引用。在此示例中，Oracle尝试按如下方式解析名称departments ：

  1. 首先，Oracle尝试在包含表，视图和私有同义词的自己的模式中定位名称空间中的对象。如果对象是私有同义词，则Oracle将找到同义词所代表的对象。此对象可以位于您自己的架构，另一个架构或另一个数据库中。该对象也可以是另一个同义词，在这种情况下，Oracle会找到此同义词所代表的对象。

  2. 如果对象位于命名空间中，则Oracle会尝试对该对象执行该语句。在此示例中，Oracle尝试将数据行添加到departments 。如果对象的语句类型不正确，则Oracle会返回错误。在此示例中， departments必须是表，视图或解析为表或视图的专用同义词。如果departments是序列，则Oracle返回错误。

  3. 如果对象不在目前为止搜索的任何名称空间中，则Oracle搜索包含公共同义词的名称空间。如果对象位于该命名空间中，则Oracle会尝试对其执行语句。如果对象的语句类型不正确，则Oracle会返回错误。在此示例中，如果departments是序列的公共同义词，则Oracle会返回错误。

如果公共同义词具有任何依赖表或用户定义类型，则无法在与依赖对象相同的模式中创建与同义词同名的对象。

如果同义词没有任何依赖表或用户定义类型，则可以在与依赖对象相同的模式中创建具有相同名称的对象。Oracle使任何依赖对象无效，并在下次访问时尝试重新验证它们。

  • 如何创建数据库链接

对数据库链接的引用

只有在使用Oracle分布式功能时，才能使用数据库链接。发出包含数据库链接的SQL语句时，可以使用以下其中一种形式指定数据库链接名称：

  • 存储在数据字典中的完整数据库链接名称，包括database ， domain和可选的connection_qualifier组件。

  • partial数据库链接名称是database和可选的connection_qualifier组件，但不是domain组件。

Oracle在连接到远程数据库之前执行以下任务：

  1. 如果语句中指定的数据库链接名称是partial，则Oracle会扩展名称以包含在数据字典中存储的全局数据库名称中找到的本地数据库的域。（您可以在GLOBAL_NAME数据字典视图中查看当前的全局数据库名称。）

  2. Oracle首先在您自己的模式中搜索私有数据库链接，其名称与语句中的数据库链接相同。然后，如有必要，它将搜索具有相同名称的公共数据库链接。

    • Oracle始终从第一个匹配的数据库链接（私有或公共）确定用户名和密码。如果第一个匹配的数据库链接具有关联的用户名和密码，则Oracle使用它。如果它没有关联的用户名和密码，则Oracle会使用您当前的用户名和密码。

    • 如果第一个匹配的数据库链接具有关联的数据库字符串，则Oracle使用它。否则，Oracle会搜索下一个匹配（公共）数据库链接。如果未找到匹配的数据库链接，或者没有匹配的链接具有关联的数据库字符串，则Oracle将返回错误。

  3. Oracle使用数据库字符串来访问远程数据库。访问远程数据库后，如果GLOBAL_NAMES参数的值为true ，则Oracle验证数据库链接名称的database.domain部分是否与远程数据库的完整全局名称匹配。如果此条件为真，则Oracle使用在步骤2中选择的用户名和密码继续连接。如果没有，Oracle会返回错误。

  4. 如果使用数据库字符串，用户名和密码的连接成功，则Oracle会尝试使用解析对象引用和引用本节前面讨论的其他模式中的对象的规则来访问远程数据库上的指定对象。

通过将初始化参数GLOBAL_NAMES或ALTER SYSTEM或ALTER SESSION语句的GLOBAL_NAMES参数设置为FALSE ，可以禁用数据库链接名称的database.domain部分必须与远程数据库的完整全局名称匹配的要求。

对分区表和索引的引用

可以对表和索引进行分区。分区时，这些模式对象由许多称为分区的部分组成，所有部分都具有相同的逻辑属性。例如，表中的所有分区共享相同的列和约束定义，并且索引中的所有分区共享相同的索引列。

分区扩展和子分区扩展名称允许您在一个分区或子分区上执行一些分区级和子分区级操作，例如从分区或子分区中删除所有行。如果没有扩展名，则此类操作将要求您指定谓词（ WHERE子句）。对于范围和列表分区表，尝试使用谓词表示分区级操作可能很麻烦，尤其是当范围分区键使用多个列时。对于散列分区和子分区，使用谓词仍然比较困难，因为这些分区和子分区基于系统定义的散列函数。

分区扩展名称允许您像使用表一样使用分区。此方法的一个优点是，对范围分区表最有用的是，您可以通过向这些视图（或来自）其他用户或角色授予（或撤销）特权来构建分区级访问控制机制。要将分区用作表，请通过从单个分区中选择数据来创建视图，然后将该视图用作表。

扩展名称的限制

目前，分区扩展和子分区扩展表名的使用具有以下限制：

  • 无远程表：分区扩展或子分区扩展表名称不能包含数据库链接（dblink）或转换为具有dblink的表的同义词。要使用远程分区和子分区，请在远程站点上创建使用扩展表名语法的视图，然后引用远程视图。

  • 无同义词：必须使用基表指定分区或子分区扩展名。您不能使用同义词，视图或任何其他对象。

  • PARTITION FOR和SUBPARTITION FOR子句对视图的DDL操作无效。

  • 在PARTITION FOR和SUBPARTITION FOR子句中，不能为partition_key_value或subpartition_key_value指定关键字DEFAULT或MAXVALUE或绑定变量。

  • 在PARTITION和SUBPARTITION子句中，您不能为分区或子分区名称指定绑定变量。

36.1 SDO_WCS.CreateTempTable

格式

SDO_WCS.CreateTempTable（schema_name IN VARCHAR2 DEFAULT USER，tablespace_name IN VARCHAR2 DEFAULT NULL）;

描述

在涉及重新投影或转换时，在GetCoverage处理期间创建必要的临时表。

参数

架构名称

将在其中创建临时表的模式的名称。如果未指定，则在调用用户架构时创建表。

tablespace_name

临时表的表空间的名称。如果未指定，则使用用户的默认表空间。

使用说明

每个临时GeoRaster对象都存储在与原始GeoRaster对象相同的模式中。将GetCoverage操作（WCS）响应发送到客户端后，将删除在处理请求期间创建的任何临时GeoRaster。

必须为其GeoRaster对象将作为coverage发布的所有模式调用此过程。

有关WCS支持的信息，请参阅Web Coverage Service（WCS）支持 。

例子

以下示例在调用程序的架构中创建WCS临时表。

CALL SDO_WCS.createTempTable（）;

以下示例在SCOTT模式中创建WCS临时表。调用用户必须具有在SCOTT模式中创建表的权限。

CALL SDO_WCS.createTempTable（'SCOTT'）;

21.1.1关于在多租户环境中使用数据泵

通常，将数据泵与PDB一起使用与使用非CDB的数据泵相同。

多租户容器数据库（CDB）是一个Oracle数据库，包括零个，一个或多个用户创建的可插拔数据库（PDB）。PDB是一组可移植的模式，模式对象和非模式对象，它们在Oracle Net客户端中显示为非CDB。非CDB是不是CDB的Oracle数据库。

在以下方案中，您可以使用Data Pump迁移全部或部分数据库：

  • 从非CDB到PDB

  • 在相同或不同CDB内的PDB之间

  • 从PDB到非CDB

注意：

Data Pump不支持任何CDB范围的操作。如果您连接到CDB的根数据库或种子数据库，则Data Pump会发出以下警告：

ORA-39357：警告：连接到容器数据库的根目录或种子时，通常不需要Oracle Data Pump操作。

父主题： 在CDB中导入和导出数据

21.1.2使用数据泵将数据移入CDB

创建空PDB后，可以使用Oracle Data Pump全模式导出和导入操作将数据移动到PDB中。

您可以使用或不使用transportable选项导入数据。如果在完整模式导出或导入时使用可传输选项，则将其称为完全可传输导出/导入。

使用可传输选项时，导出和导入使用可传输的表空间数据移动和传统的数据移动;后者用于驻留在非可传输表空间（如SYSTEM和SYSAUX那些表。使用transportable选项可以减少导出时间，尤其是导入时间，因为不需要卸载和重新加载表数据，也不需要重新创建用户表空间中的索引结构。

使用Data Pump将数据移动到CDB时，请注意以下要求：

  • 要管理多租户环境，您必须具有CDB_DBA角色。

  • 可以将Oracle Database 11.2.0.2及更早版本的完整数据库导出导入Oracle Database 12c（CDB或非CDB）。但是，Oracle建议您首先将源数据库升级到Oracle Database 11g第2版（11.2.0.3或更高版本），以便导出中包含有关已注册选项和组件的信息。

  • 使用完整数据库导出或完全可传输数据库导出将Oracle Database 11g第2版（11.2.0.3或更高版本）迁移到CDB（或非CDB）时，必须将数据泵导出参数VERSION=12为生成一个准备导入Oracle Database 12c的转储文件。如果未设置VERSION=12 ，则生成的导出文件不包含有关已注册数据库选项和组件的完整信息。

  • 基于网络的完全可传输导入需要使用FULL=YES ， TRANSPORTABLE=ALWAYS和TRANSPORT_DATAFILES= datafile_name参数。当源数据库是Oracle Database 11g 11.2.0.3或更高版本，但早于Oracle Database 12c第1版（12.1）时，还需要VERSION=12参数。

  • 基于文件的完全可传输导入仅需要使用TRANSPORT_DATAFILES= datafile_name参数。数据泵导入推断出TRANSPORTABLE=ALWAYS和FULL=YES参数的存在。

  • 从Oracle Database 12c第2版（12.2）开始，在多租户容器数据库（CDB）环境中，默认数据泵目录对象DATA_PUMP_DIR被定义为CDB中每个PDB的唯一路径。定义此唯一路径是定义CREATE PLUGGABLE DATABASE语句的PATH_PREFIX子句还是未定义相对路径。

  • 从Oracle Database 19c开始， impdp的credential参数指定凭证对象的名称，该凭证对象包含访问对象存储桶所需的用户名和密码。您还可以使用名为DEFAULT_CREDENTIAL的PDB属性指定默认凭据。使用当时的默认凭据运行impdb时，使用DEFAULT_CREDENTIAL:为转储文件名添加前缀DEFAULT_CREDENTIAL:并且不指定credential参数。

21.3 CDB和PDB的DBNEWID注意事项

DBNEWID参数PDB允许您更改可插拔数据库（PDB）上的DBID。

默认情况下，在容器数据库（CDB）上运行DBNEWID实用程序时，它仅更改CDB的DBID;包含CDB的可插拔数据库（PDB）的DBID保持不变。在某些情况下，例如克隆CDB时，这可能会导致PDB的重复DBID出现问题。

从Oracle Database 12 c第2版（12.2）开始，您可以使用新的DBNEWID PDB参数更改PDB上的DBID。您无法指定特定的PDB;要么全部，要么都没有新的DBID。PDB参数仅适用于CDB环境。它具有以下格式：

PDB = [ALL |没有]

  • 如果指定ALL ，则除了CDB更改的DBID之外，还会更改包含CDB的所有PDB的DBID。

  • 即使CDB DBID已更改，指定NONE （默认值）也会使PDB DBID保持不变。

Oracle建议您使用PDB=ALL ，但出于向后兼容性原因， PDB=NONE是默认值。

Oracle RAC中的单节点故障

Oracle RAC中的实例恢复不包括恢复在失败实例上运行的应用程序。Oracle Clusterware会自动重新启动实例。

使用RMAN与群集实例建立通道连接

使用通道配置定义的连接字符串确定与实例的通道连接。例如，在以下配置中，使用dbauser/pwd@ service_name分配了三个通道。如果配置启用了负载平衡的SQL Net服务名称，则会根据负载平衡算法在节点上分配通道。

但是，如果连接字符串中使用的服务名称不用于负载平衡，则可以使用单独的连接字符串为每个通道配置控制在哪个实例上分配通道，如下所示：

配置设备类型sbt PARALLELISM 3;配置频道1 ..CONNECT'dbauser / pwd @ mydb_1';配置频道2 ..CONNECT'dbauser / pwd @ mydb_2';配置频道3 ..CONNECT'dbauser / pwd @ mydb_3';

在前面的示例中，假设mydb_1 ， mydb_2和mydb_3是连接到Oracle RAC环境中预定义节点的SQL * Net服务名称。或者，您也可以使用手动分配的通道来备份数据库文件。例如，以下命令备份SPFILE，控制文件，数据文件和归档重做日志：

运行{ALLOCATE CHANNEL CH1 CONNECT'dbauser / pwd @ mydb_1'; ALLOCATE CHANNEL CH2 CONNECT'dbauser / pwd @ mydb_2'; ALLOCATE CHANNEL CH3 CONNECT'dbauser / pwd @ mydb_3'; BACKUP DATABASE PLUS ARCHIVED LOG; }

在备份操作期间，如果分配的至少一个通道可以访问存档日志，则RMAN会自动计划该通道上特定日志的备份。由于控制文件，SPFILE和数据文件可由任何通道访问，因此这些文件的备份操作将在分配的通道中分布。

对于本地归档方案，必须至少有一个通道分配给写入其本地归档日志的所有节点。对于群集文件系统归档方案，如果每个节点将归档日志写入同一群集文件系统，则归档日志的备份操作将在分配的通道中分布。

在备份期间，通道连接的实例必须全部安装或全部打开。例如，如果node1上的实例已安装数据库，而node2和node3上的实例已打开数据库，则备份将失败。

节点关联性快速连接意识

在某些群集数据库配置中，群集的某些节点可以更快地访问某些数据文件而不是其他数据文件。RMAN自动检测到这种情况，这被称为节点亲和感知。在决定使用哪个通道备份特定数据文件时，RMAN会优先选择能够更快访问要备份的数据文件的节点。例如，如果您有一个三节点集群，并且如果node1对数据文件7,8和9的读/写访问速度比其他节点更快，那么node1与那些文件的节点关联性要大于node2和node3 。

成功备份后删除存档的重做日志

如果已配置“使用RMAN将通道连接到群集实例”部分中定义的自动通道，则可以使用以下示例删除已备份n次的已归档日志。设备类型可以是DISK或SBT ：

DELETE ARCHIVELOG备份的所有n TIMES设备类型device_type ;

PGA_CAPABILITY参数注意事项

选择设置时 PGA_CAPABILITY参数，请注意确保根据远程事务程序将要执行的操作使用正确的设置。

该应该始终使用READ_ONLY设置远程事务程序是只读的，即远程事务程序不执行数据库更新时。远程事务程序执行数据库更新时，永远不应使用READ_ONLY 。例如，如果选择了READ_ONLY设置，并且如果网关调用的远程事务程序对外部数据库执行更新，则 Oracle数据库不为这些更新提供任何完整性保护。此外， READ_ONLY模式使网关事务成为可能更新其他几个数据库的分布式事务的一部分。如果网关调用在这种情况下执行更新的远程事务程序，并且如果发生故障，则远程事务程序更新的数据库不会与其他数据库同步。

在远程事务程序对外部数据库执行更新的情况下，存在 PGA_CAPABILITY值的两个选项：

  • SINGLE_SITE

  • COMMIT_CONFIRM

通过允许，每个选项都可以防止数据完整性问题 COMMIT和 ROLLBACK请求转发到远程事务程序并通过通知 Oracle数据库，关于网关的分布式更新和恢复功能。特定选项取决于远程事务程序的设计和功能 OLTP（在线事务处理器）执行的位置。

如果OLTP具有LU6.2 SYNCLEVEL 1或2支持，那么 COMMIT_CONFIRM功能提供了有限的两阶段提交 Oracle数据库和OLTP，没有其他限制 commit-confirm站点（网关或Oracle）可以是分布式事务的一部分。如果无法使用COMMIT_CONFIRM ，那么SINGLE_SITE能力提供了Oracle数据库和更新之间的能力 OLTP，限制条件是只有OLTP可以执行更新，并且Oracle端不会发生更新。

用于更新控制的每个PGA_CAPABILITY选项都对远程事务程序和OLTP施加了特定要求。对于COMMIT_CONFIRM功能，这些要求将在Oracle Database Gateway for APPC用户指南的第5章“实现提交确认”中详细讨论。另请参阅本指南中的配置OLTP以进行提交确认 。对于SINGLE_SITE功能，远程事务程序负责执行所需的任务以响应代表网关从网关接收的COMMIT和ROLLBACK请求。 Oracle数据库。网关使用APPC CONFIRM和SEND_ERR请求分别实现COMMIT和ROLLBACK 。收到CONFIRM ，远程事务程序必须执行COMMIT处理，然后使用APPC CONFIRMED响应响应网关。收到SEND_ERR ，远程事务程序必须执行ROLLBACK处理。

因为分布式事务的能力 Oracle数据库受网关使用的PGA_CAPABILITY选项的影响，希望通过为每个应用程序使用不同的网关实例来分离查询和更新应用程序。一个网关可以被定义PGA_CAPABILITY设置为READ_ONLY和其他人PGA_CAPABILITY设置为SINGLE_SITE或COMMIT_CONFIRM 。

这允许只读事务程序在Oracle数据库的控制下参与分布式事务。例如，来自DB2的数据可以通过READ_ONLY网关由只查询的远程事务程序检索，然后可以在一个Oracle事务中用作Oracle数据库上数据库更新的输入。SINGLE_SITE网关只能用于访问远程事务程序，这些程序对Oracle数据库控件范围之外的外部数据库执行更新。可以从Oracle数据库可访问的任何数据库中读取数据，并且该数据可用于通过网关执行更新。

当需要更新Oracle端和OLTP端的资源时，可以使用COMMIT_CONFIRM网关，前提是OLTP和远程事务程序设置为实现commit-confirm。

PGA_CONFIRM参数注意事项

在决定设置时 PGA_CONFIRM 参数，了解每个设置的效果很重要。首先，请记住，此参数仅影响在SYNCLEVEL 1运行的那些会话。默认设置PGA_CONFIRM = REJECT对大多数应用程序都是正确的。使用此设置，网关生成一个如果从远程事务程序收到CONFIRM请求，则会出错。如果你有一个使用远程交易CONFIRM以验证数据是由网关接收，则必须使用PGA_CONFIRM = ACCEPT ，以允许网关向这些传入响应CONFIRM与请求CONFIRMED响应。您必须知道，当Oracle应用程序发送了COMMIT请求时，网关会向远程事务发送CONFIRM请求。为使COMMIT处理正常工作，必须编写远程事务，以便在从网关收到CONFIRM请求时执行其本地提交处理，并在提交处理成功完成后使用CONFIRMED响应网关。如果在提交处理期间发生错误，然后远程事务必须使用SEND_ERR响应网关以指示提交失败。

使用PGA_CONFIRM = ACCEPT一个特例是IMS / TM版本7。使用IMS / TM版本7提供的“隐含APPC”支持时，在SYNCLEVEL 1上运行的会话的处理方式与在SYNCLEVEL 0上运行的会话的处理方式不同。当会话处于SYNCLEVEL 1时，IMS / TM会在每次APPC SEND后自动生成CONFIRM请求。 在网关侧，如果未指定PGA_CONFIRM = ACCEPT ，则IMS / TM发送的CONFIRM请求将导致网关生成错误。使用PGA_CONFIRM = ACCEPT缓解这一问题，使网关进入的回应CONFIRM与请求CONFIRMED响应。以这种方式运行的唯一限制是IMS提供的隐含APPC支持在从网关收到CONFIRM请求时不通知应用程序。这意味着网关不能使用CONFIRM来实现COMMIT ，从而禁止使用COMMIT和ROLLBACK来控制对话的IMS端的更新。

16.2使用时移

本节介绍如何使用Consolidated Database Replay进行时移，并假设您希望使用时移来协调从三个应用程序捕获的工作负载峰值并同时重放它们。 该场景演示了如何使用时移进行压力测试。有关时移的更多信息，请参阅“ 关于时间转换 ” 。

此方案使用以下假设：

  • 从Sales应用程序捕获第一个工作负载。

  • 第二个工作负载从CRM应用程序捕获，其高峰时间发生在Sales工作负载之前1小时。

  • 第三个工作负载从DW应用程序捕获，其高峰时间出现在Sales工作负载之前30分钟。

  • 为了协调这些工作负载的峰值，通过向CRM工作负载添加一小时的延迟并在重放期间向DW工作负载延迟30分钟来执行时移。

要在此方案中执行时移：

  1. 在将要进行压力测试的重放系统上，为存储捕获的工作负载的根目录创建目录对象：

    CREATE [OR REPLACE] DIRECTORY cons_dir AS'/ u01 / test / cons_dir';

  2. 将各个工作负载捕获预处理到单独的目录中：

    • 对于销售工作量：

      1. 创建目录对象：

        创建或更换DIRECTORY销售AS'/ u01 / test / cons_dir / cap_sales';

      2. 确保Sales应用程序中捕获的工作负载存储在此目录中。

      3. 预处理工作量：

        EXEC DBMS_WORKLOAD_REPLAY.PROCESS_CAPTURE（'SALES'）;

    • 对于CRM工作负载：

      1. 创建目录对象：

        创建或更换目录crm AS'/ u01 / test / cons_dir / cap_crm';

      2. 确保从CRM应用程序捕获的工作负载存储在此目录中。

      3. 预处理工作量：

        EXEC DBMS_WORKLOAD_REPLAY.PROCESS_CAPTURE（'CRM'）;

    • 对于DW工作负载：

      1. 创建目录对象：

        创建或更换DIRECTORY DW AS'/ u01 / test / cons_dir / cap_dw';

      2. 确保从DW应用程序捕获的工作负载存储在此目录中。

      3. 预处理工作量：

        EXEC DBMS_WORKLOAD_REPLAY.PROCESS_CAPTURE（'DW'）;

  3. 将replay目录设置为根目录：

    EXEC DBMS_WORKLOAD_REPLAY.SET_REPLAY_DIRECTORY（'CONS_DIR'）;

  4. 创建重播计划并添加工作负载捕获：

    EXEC DBMS_WORKLOAD_REPLAY.BEGIN_REPLAY_SCHEDULE（'align_peaks_schedule'）; SELECT DBMS_WORKLOAD_REPLAY.ADD_CAPTURE（'SALES'）FROM dual; SELECT DBMS_WORKLOAD_REPLAY.ADD_CAPTURE（'CRM'，3600）FROM dual; SELECT DBMS_WORKLOAD_REPLAY.ADD_CAPTURE（'DW'，1800）FROM dual; EXEC DBMS_WORKLOAD_REPLAY.END_REPLAY_SCHEDULE;

    请注意，CRM工作负载会添加3,600秒（或1小时）的延迟，并且DW工作负载会延迟1,800秒（或30分钟）。

  5. 初始化整合重播：

    EXEC DBMS_WORKLOAD_REPLAY.INITIALIZE_CONSOLIDATED_REPLAY（'align_peaks_replay'，'align_peaks_schedule'）;

  6. 重新映射连接：

    1. 在DBA_WORKLOAD_CONNECTION_MAP视图中查询连接映射信息：

      SELECT schedule_cap_id，conn_id，capture_conn，replay_conn FROM dba_workload_connection_map;

    2. 重新映射连接：

      EXEC DBMS_WORKLOAD_REPLAY.REMAP_CONNECTION（schedule_cap_id => 1，conn_id => 1，replay_connection =>'inst1'）; EXEC DBMS_WORKLOAD_REPLAY.REMAP_CONNECTION（schedule_cap_id => 1，conn_id => 2，replay_connection =>'inst1'）; EXEC DBMS_WORKLOAD_REPLAY.REMAP_CONNECTION（schedule_cap_id => 2，conn_id => 1，replay_connection =>'inst2'）; EXEC DBMS_WORKLOAD_REPLAY.REMAP_CONNECTION（schedule_cap_id => 2，conn_id => 2，replay_connection =>'inst2'）; EXEC DBMS_WORKLOAD_REPLAY.REMAP_CONNECTION（schedule_cap_id => 3，conn_id => 1，replay_connection =>'inst3'）; EXEC DBMS_WORKLOAD_REPLAY.REMAP_CONNECTION（schedule_cap_id => 3，conn_id => 2，replay_connection =>'inst3'）;

      replay_connection参数表示在测试系统上定义的服务。

    3. 验证连接重映射：

      SELECT schedule_cap_id，conn_id，capture_conn，replay_conn FROM dba_workload_connection_map;

  7. 准备整合重播：

    EXEC DBMS_WORKLOAD_REPLAY.PREPARE_CONSOLIDATED_REPLAY;

  8. 开始重播客户：

    1. 估计所需的重播客户端数量：

      wrc mode = calibrate replaydir = / u01 / test / cons_dir / cap_sales wrc mode = calibrate replaydir = / u01 / test / cons_dir / cap_crm wrc mode = calibrate replaydir = / u01 / test / cons_dir / cap_dw

    2. 添加输出以确定所需的重播客户端数。

      您需要在整合工作负载中包含的每个工作负载捕获中至少启动一个重播客户端。

    3. 重复此命令启动所需数量的重播客户端：

      wrc 用户名 / 密码模式=重播replaydir = / u01 / test / cons_dir

      replaydir参数设置为存储工作负载捕获的根目录。

  9. 开始整合重播：

    EXEC DBMS_WORKLOAD_REPLAY.START_CONSOLIDATED_REPLAY;

3.3关于重定向服务器上的输出

您可以将Java输出传递给SQL语句，以便对Oracle JVM的输出目标进行更广泛的控制。通过添加以下新功能增强了PL / SQL包DBMS_JAVA ，这些新功能提供了以前只能通过DBMS_JAVA.SET_OUTPUT过程使用的扩展功能：

  • remove_output_to_sql

  • enable_output_to_sql

  • disable_output_to_sql

  • query_output_to_sql

set_output_to_sql

set_output_to_sql定义一个命名输出规范，该规范构成执行SQL语句的指令，每当输出到默认的System.out和System.err流时。规范定义为当前会话的持续时间，或直到使用其ID调用remove_output_to_sql函数。只要有Java输出，就会发生规范规定的SQL操作。这可以通过分别调用disable_output_to_sql和enable_output_to_sql函数来停止和启动。成功时此函数的返回值为null，否则为错误消息。 

FUNCTION set_output_to_sql（id VARCHAR2，stmt VARCHAR2，bindings VARCHAR2，no_newline_stmt VARCHAR2 default null，no_newline_bindings VARCHAR2 default null，newline_only_nwts VARCHAR2 default null，newline_only_bindings VARCHAR2 default null，maximum_line_segment_length NUMBER default 0，allow_replace NUMBER default 1，from_stdout NUMBER default 1，from_stderr NUMBER default 1，include_newlines NUMBER默认为0，急切NUMBER默认为0）返回VARCHAR2;

表3-2 set_output_to_sql参数摘要

争论 描述

ID

规范的名称。多个规范可能存在于同一会话中，但每个规范必须具有不同的ID。该ID用于标识函数remove_output_to_sql, enable_output_to_sql, disable_output_to_sql,和query_output_to_sql.

语句

发生Java输出时要执行的默认SQL语句。

绑定

包含来自集合ID，TEXT，LENGTH，LINENO，SEGNO，NL和ERROUT的标记的字符串。此字符串定义SQL语句stmt的绑定方式。绑定字符串中的标记位置对应于SQL语句中的绑定位置。令牌的含义是：

  • ID是规范的ID。它被绑定为VARCHAR2。

  • TEXT是要输出的文本。它被绑定为VARCHAR2。

  • LENGTH是文本的长度。它被绑定为NUMBER。

  • LINENO是会话输出开始以来的行号。它被绑定为NUMBER。

  • SEGNO是一行中的段号，它以多个片段输出。它被绑定为NUMBER。

  • NL是一个布尔值，表示文本是否被视为换行终止。它被绑定为NUMBER。新行可能会或可能不会实际包含在文本中，具体取决于include_newlines参数的值。

  • ERROUT是一个布尔值，指示输出是来自System.out还是System.err. 它被绑定为NUMBER。如果输出来自System.out. ，则值为0 System.out.

no_newline_stmt

当输出不是换行符时，要执行的可选备用SQL语句。

no_newline_bindings

一个字符串，其语法与前面讨论的bindings参数相同，描述了no_newline_stmt的绑定方式。

newline_only_stmt

输出为单个换行符时要执行的可选备用SQL语句。

newline_only_bindings

一个字符串，其语法与前面讨论的bindings参数相同，描述了newline_only_stmt的绑定方式。

maximum_line_segment_length

在给定的SQL语句执行中绑定的最大字符数。较长的输出序列被分解为具有不同SEGNO值的单独调用。值0表示no maximum.

allow_replace

当存在具有相同ID的先前定义的规范时，控制行为。值为1表示替换旧规范。 0表示返回错误消息而不修改旧规范。

控制System.out输出是否导致执行规范规定的SQL语句。值为0表示如果输出来自System.out,则即使启用了规范，也不会执行该语句。

from_stderr

控制System.err输出是否导致执行规范规定的SQL语句。值为0表示如果输出来自System.err,则即使启用了规范，也不会执行该语句。

include_newlines

控制绑定到文本时换行符是否保留在输出中。值0表示不包括新行。但是新绑定的存在仍然由NL绑定和no_newline_stmt的使用no_newline_stmt 。

急于

控制输出是否未被换行符导致每次接收时都会导致执行SQL语句，或者在收到换行符之前累积此类输出。值0表示累积未终止的输出。

remove_output_to_sql

remove_output_to_sql删除由set_output_to_sql.创建的规范set_output_to_sql. 如果不存在此类规范，则返回错误消息。

139 DBMS_RESUMABLE

使用DBMS_RESUMABLE程序包，您可以在执行很长时间后暂停空间不足或达到空间限制的大型操作，修复问题并使语句恢复执行。通过这种方式，您可以编写应用程序而无需担心遇到与空间相关的错误。

139.1 DBMS_RESUMABLE操作说明

暂停语句时，应将暂停记录在警报日志中。您还应该在语句暂停时注册要执行的过程。使用视图，您可以监视语句的进度并指示语句当前是在执行还是暂停。

挂起语句会自动导致暂停事务。因此，所有事务资源都在语句暂停和恢复期间保留。当错误条件消失时，挂起的语句将自动恢复执行。可恢复的空间分配可以在执行期间暂停和恢复多次。

暂停超时间隔与可恢复空间分配相关联。在超时间隔（默认为两小时）暂停的可恢复空间分配唤醒并向用户返回异常。可以使用DBMS_RESUMABLE.ABORT()过程强制挂起的语句抛出异常。

20使用Java确定XML差异

解释了如何使用Oracle XML Developer's Kit（XDK）中包含的Java库来确定两个可扩展标记语言（XML）输入之间的差异。

Java的XML差异实用程序概述

在Java XML版本比较库包括版本比较，散列和用于在类XML输入平等-比较方法XmlUtils包的oracle.xml.diff 。

oracle.xml.diff包中的Options类提供了一些选项，使用户可以控制XmlUtils类中的方法处理输入的方式（请参阅Java XML Diffing Library的用户选项 ）。其中一个受支持的选项是空白区域标准化，默认情况下已启用。

XML diffing方法使用的算法专门针对在几秒钟内查找两个大型XML文档（5 MB或更多）之间差异的用例，其中不需要最小差异。最小差异是最小的可能变化集，当应用于第一个XML输入时，产生与第二个XML输入等效的输出（相同）。已知的最小差异算法需要非常大量的内存和时间来处理多兆字节输入。在XML输入中没有重复出现的相同子树的情况下，XML diff方法中使用的算法可以产生最佳质量（尽可能接近最小）差异。

Java XML diffing库为每种方法提供了几种等效变体，以允许不同形式的XML输入，包括文档对象模型（DOM）节点，文件和输入流。在内部，差异，散列和相等比较在DOM树上运行。不是DOM树形式的输入在内部转换为DOM树。为了减少计算开销，Oracle建议尽可能直接传入DOM。

Java XML diffing库包括将diff输出作为DOM文档或作为对象列表返回的方法，每个对象都表示diff操作。使用第二个选项，可以避免生成XML文档的开销。使用第一个选项，生成的文档符合Diff Output Schema中描述的XML模式。第一个选项很有用，例如，如果必须将diff输出存储为日志以供将来参考。

Java XML diffing库提供的哈希方法计算XML输入的哈希值。如果两个XML输入的哈希值相等，则它们的概率非常高。

Java XML diffing库中提供的相等方法比较两个输入是否相等。

要使用Java XML差异库，您的应用程序必须在Java 1.6或更高版本上运行，并且可以使用任何DOM实现。

注意：

本章中描述的应用程序编程接口（API）组件包含在Java包oracle.xml.diff 。为简洁起见，仅在必要时使用完全限定名称以避免混淆。

有关oracle.xml.diff包的更多信息，请参见Oracle数据库XML Java API参考 。

Java XML Diffing Library的用户选项

Java XML diffing库支持两个选项，您可以使用oracle.xml.diff包的Options类中的方法设置这两个选项。Options对象在每次调用时直接传递给diff，hash和equal方法。

  • 文本节点规范化（默认启用）

    文本节点在DOM树中标准化，diff树，hash和equal方法在DOM树中运行。文本节点规范化涉及合并相邻的文本节点，然后从合并的节点中剥离前导和尾随空白。单个文本节点的前导和尾随空格被剥离。仅消除空白文本节点。

    规范化在库中执行，具有最小的额外空间，并且无需修改提供的XML输入。

    要在将DOM输入传递给库之前对DOM输入执行自己的规范化，必须在Options对象上调用normalizeTextNodes(false)方法以关闭默认规范化。

    Oracle不建议在不执行某些类型的规范化（默认或自己的规范化）的情况下调用diff方法。差异质量在存在相同的空白文本节点时受到影响，这些节点通常出现在XML文档中。

  • 忽略命名空间前缀差异（默认情况下启用）

    diff，hash和equal方法忽略XML名称空间前缀差异。例如，如果两个DOM节点相同，则除了具有不同的前缀之外（即使两个不同的前缀映射到相同命名空间的统一资源标识符（URI）），它们被认为是相等的。要将库配置为将不同的名称空间前缀视为真正不同，即使它们映射到同一URI，您也可以在Options对象上调用ignorePrefixDifferences(false)方法来关闭默认名称空间前缀行为。

使用Java XML Diffing方法查找差异

Java的XML dffing库提供各种diff和diffToDoc的方法XmlUtils类的oracle.xml.diff包。您可以使用这些方法比较两个XML输入，以确定它们之间是否存在任何差异。

diffToDoc方法将输出作为DOM文档返回，该文档符合Diff Output Schema中描述的模式 。Java XML diffing库包含这些方法的几个等效变体，它们接受不同形式的输入（DOM节点，文件和其他形式）。

Java XML diffing库包含一组等效的diff方法，使您可以处理作为diff操作对象列表返回的diff输出。

因为不需要构造表示diff的DOM文档，所以使用diff方法比使用difftoDoc方法更有效。只要不需要XML格式的diff表示，就应该考虑使用这些方法。要使用diff方法，必须创建DiffOpReceiver接口的实现，然后将其作为参数传递给diff方法。DiffOpReceiver.receiveDiff方法接收diff作为DiffOp对象的列表。

diff结果，无论是作为DOM文档还是作为DiffOps对象列表DiffOps ，都可以理解为一系列diff操作。可能的差异操作是：

  • 追加节点

  • 插入节点，前

  • 删除节点

在第一个DOM树上应用diff操作序列会生成一个等同于第二个DOM树的树。例如，使用这两个XML输入：

第一个输入： <a><b/></a>

第二个输入： <a><c/></a>

比较第一个和第二个输入的差异结果是一个列表，具有这两个差异操作：

delete-node / a [1] / b [1] append-node <c /> to / a [1]

在第一个输入中删除由XPath表达式/a/b表示的节点，然后将<c/>附加到第一个输入中由XPath表达式/a表示的节点，将生成结果<a><c/></a> ，相当于第二个输入。

当diff操作通过domToDoc(…)方法输出到DOM文档时，它们依赖于XPath表达式来指示节点位置。这些XPath位置指的是原始第一个输入中的节点位置。它们不反映应用的差异操作。

注意：

Java XML diffing库不支持属性节点的append-node，insert-node-before和delete-node操作。因此，当更改节点的任何属性时，更改将显示为整个节点的删除，后跟插入或具有更改的属性的新节点的附加。

例如，对于这两个输入：

第一个输入： <a attr1="val1"><b/></a>

第二个输入： <a attr2="val2"><b/</a>

差异由这两个差异操作组成：

在/ [1]删除/ a [1]之前插入<a attr2="val2"> <b /> </a>

本节使用XML文档输出来描述每个diff操作。虽然这里没有描述它们，但是以编程方式返回的diff操作结果是等效的。

CHAR语义学

此功能允许网关可选地在CHAR语义模式下运行。而不是总是将UDB CHAR列描述为CHAR(n BYTE) ，此功能将它们描述为CHAR(n CHAR)和VARCHAR(n CHAR) 。该概念类似于Oracle数据库CHAR Semantics。您需要指定HS_NLS_LENGTH_SEMANTICS=CHAR网关参数以激活此选项。有关更多详细信息，请参阅初始化参数 。

多字节字符集比率抑制

此功能可选地抑制从UDB数据库到涉及多字节字符集的Oracle数据库的比率扩展。默认情况下，Oracle数据库网关DRDA采用最差比率来防止数据被截断或缓冲区大小不足。但是，如果您具有UDB数据库的特定知识并且不希望进行扩展，则可以指定HS_KEEP_REMOTE_COLUMN_SIZE参数以禁止扩展。有关更多详细信息，请参阅初始化参数 。

IPv6支持

除了Oracle数据库和网关之间的完全IPv6支持外，此网关和UDB数据库之间也支持IPv6。有关更多详细信息，请参阅初始化参数中的HS_FDS_CONNECT_INFO参数。

网关会话IDLE超时

您可以选择使用网关参数HS_IDLE_TIMEOUT自动终止长空闲网关会话。具体来说，当网关会话空闲超过指定的时间限制时，网关会话将终止，并回滚任何挂起的更新

两阶段提交处理

要完全参与两阶段提交事务，服务器必须支持 PREPARE TRANSACTION语句。PREPARE TRANSACTION语句确保所有参与的数据库都准备好COMMIT或ROLLBACK特定的工作单元。

Oracle数据库支持PREPARE TRANSACTION语句。任意数量的Oracle数据库都可以参与分布式两阶段提交事务。当应用程序显式发出COMMIT时，或者在应用程序的正常结束时隐式执行PREPARE TRANSACTION语句。

网关不支持 PREPARE TRANSACTION语句。这限制了网关参与分布式事务时的两阶段提交协议。网关成为分布式事务的提交焦点站点。网关配置为commit / confirm，因此无论提交点强度设置如何，它始终是提交点站点。在验证事务中的所有Oracle数据库是否已成功提交其工作后，网关将提交工作单元。网关必须协调分布式事务，以便只有一个网关实例可以参与两阶段提交事务。

两阶段提交事务记录在HS_TRANSACTION_LOG表中（请参阅初始化参数HS_FDS_TRANSACTION_LOG ），该表在安装期间创建。这个表是在创建的时候创建的 o2pc.sql脚本运行。该表的所有者也拥有该包。有关POWER系统（64位）上的IBM AIX的Oracle Database Gateway安装和配置指南，Linux x86-64，SPARC上的Oracle Solaris（64位），x86-64上的Oracle Solaris，请参阅“DRDA网关程序包绑定注意事项”。 适用于Microsoft Windows的 （64位）和HP-UX Itanium或Oracle Database Gateway安装和配置指南 ，具体取决于您的平台，以获取更多信息。

跟踪SQL语句

3.1从逻辑设计转向物理设计

逻辑设计是什么在构建数据仓库之前，使用笔和纸或使用Oracle Designer等工具进行设计。物理设计是使用SQL语句创建数据库。

要将逻辑设计转换为物理设计，必须创建以下部分或全部结构：表空间，表，表上的分区或索引组织表，索引（包括分区索引），视图，完整性约束，物化视图和维度。

3.2.1.1关于数据仓库中的表空间

表空间由一个或多个数据文件组成，这些文件是您正在使用的操作系统中的物理结构。数据文件仅与一个表空间相关联。从设计角度来看，表空间是物理设计结构的容器。

表空间需要按差异分隔。例如，表应与索引分开，小表应与大表分开。如果可能，表空间还应表示逻辑业务单位。由于表空间是备份和恢复的最粗粒度或可传输表空间机制，因此逻辑业务设计会影响可用性和维护操作。

你现在可以使用了超大型数据文件，是超大型数据库的重大改进。

3.2.1.2关于数据仓库中的分区

Oracle分区是数据仓库非常重要的功能，可提高可管理性，性能和可用性。本节介绍了为数据仓库注意特殊值的分区的关键概念和优点。

分区允许将表，索引或索引组织表细分为更小的部分。每个数据库对象都称为分区。每个分区都有自己的名称，并且可以选择具有自己的存储特性。从数据库管理员的角度来看，分区对象具有多个可以集体或单独管理的部分。这为管理员提供了管理分区对象的极大灵活性。但是，从用户的角度来看，分区表与非分区表相同;使用SQL DML命令访问分区表时不需要进行任何修改。

数据库对象 - 表，索引和索引组织表 - 使用分区键进行分区，分区键是一组列，用于确定给定行将驻留在哪个分区中。例如，使用每月分区策略在销售日期划分的销售表;该表在任何应用程序中都显示为单个普通表。但是，DBA可以单独管理和存储每个月分区，可能使用不同的存储层，将表压缩应用于旧数据，或者将完整范围的旧数据存储在只读表空间中。

3.2.1.2.1数据仓库中使用的基本分区策略

Oracle分区提供三种基本数据分发方法，用于控制数据实际放置到各个单独分区的方式，即：

  • 范围

    数据是根据分区键的一系列值进行分配的（对于作为分区键的日期列，'January-2012'分区包含分区键值在'01 -JAN-2012'和'31之间的行 - JAN-2012' ）。数据分布是没有任何孔的连续体，并且范围的下边界由前一范围的上边界自动定义。

  • 名单

    数据分布由分区键的值列表定义（对于作为分区键的区域列， North_America分区可以包含值Canada ， USA和Mexico ）。可以定义一个特殊的DEFAULT分区来捕获未由任何列表显式定义的分区键的所有值。

  • 哈希

    散列算法应用于分区键以确定给定行的分区。与其他两种数据分发方法不同，哈希不提供数据与任何分区之间的任何逻辑映射。

除了这些基本方法，Oracle数据库还提供了以下几种：

  • 区间分区

    范围分区的扩展，增强了可管理性。分区由间隔定义，提供等宽范围。除第一个分区外，所有分区都是在匹配数据到达时按需自动创建的。

  • 按引用分区

    子表的分区是通过主键 - 外键关系从父表继承的。简化了分区维护并启用了分区智能连接。

  • 基于虚拟列的分区

    由上述分区技术之一定义，分区键基于虚拟列。虚拟列不存储在磁盘上，仅作为元数据存在。这种方法可以更灵活，更全面地匹配业务需求。

使用上述数据分发方法，可以将表分区为单个或复合分区表：

  • 单（一级）分区

    通过使用一个或多个列作为分区键指定一种数据分布方法来定义表。例如，考虑一个带有数字列作为分区键的表和两个分区less_than_five_hundred和less_than_thousand ， less_than_thousand分区包含满足以下条件的行： 500 <= Partitioning key <1000 。

    您可以指定范围，列表和散列分区表。

  • 复合分区

3.2.1.3数据仓库中的索引分区

无论选择的索引分区策略如何，索引都与基础表的分区策略耦合或解耦。根据业务需求选择适当的索引分区策略，使分区非常适合支持任何类型的应用程序。Oracle Database 12 c区分了三种类型的分区索引。

  • 本地索引

    本地索引是分区表上的索引，它与底层分区表耦合，从表中“继承”分区策略。因此，本地索引的每个分区对应于基础表的一个且仅一个分区。耦合可实现优化的分区维护;例如，当删除表分区时，Oracle数据库也必须删除相应的索引分区。无需昂贵的索引维护。本地索引在数据仓库环境中最常见。

  • 全局分区索引

    全局分区索引是分区或非分区表的索引，该表使用与表不同的分区键或分区策略进行分区。全局分区索引可以使用范围或散列分区进行分区，并与基础表分离。例如，表可以按月进行范围分区并具有12个分区，而该表上的索引可以使用不同的分区键进行散列分区，并具有不同数量的分区。对于OLTP，全局分区索引比数据仓库环境更常见。

  • 全球非分区指数

    全局非分区索引与非分区表上的索引基本相同。索引结构未与基础表分区和解耦。在数据仓库环境中，全局非分区索引的最常见用法是强制执行主键约束。

3.2.1.4关于可管理性的分区

用于可管理性的分区的典型用法是支持数据仓库中的“滚动窗口”加载过程。假设DBA每天将新数据加载到表中。该表可以进行范围分区，以便每个分区包含一天的数据。加载过程只是添加了一个新分区。添加单个分区比修改整个表更有效，因为DBA不需要修改任何其他分区。使用分区的另一个优点是何时需要删除数据。在这种情况下，与单独删除每一行相比，可以删除整个分区，这非常有效且快速。

3.2.1.5关于性能分区

通过限制要检查或操作的数据量，分区提供了许多性能优势。特别值得注意的两个特点是：

  • 分区修剪：分区修剪是使用分区提高性能的最简单也是最实质的方法。分区修剪通常可以将查询性能提高几个数量级。例如，假设一个应用程序包含一个包含订单历史记录的ORDERS表，并且该表已按日分区。请求单周订单的查询只能访问ORDERS表的七个分区。如果表具有两年的历史数据，则此查询将访问七个分区而不是730个分区。由于分区修剪，此查询可能会执行速度提高100倍。分区修剪与Oracle的所有其他性能功能一起使用。Oracle数据库将结合使用任何索引技术，连接技术或并行访问方法来进行分区修剪。

  • 分区连接：分区还可以通过使用称为分区连接的技术来提高多表连接的性能。当两个表连接在一起时，可以应用分区智能连接，并且这些表中的至少一个在连接键上进行分区。分区连接将大型连接分解为连接表的“相同”数据集的较小连接。这里的“相同”被定义为在连接的两侧覆盖完全相同的分区键值集，从而确保只有这些“相同”数据集的连接才能产生结果而其他数据集不必是考虑。Oracle数据库使用已经（物理）等分区表的事实进行连接，或者在运行时透明地重新分配（“重新分区”）一个表以创建与另一个表的分区匹配的均分数据集，从而完成整个连接更短的时间。这为串行和并行执行提供了显着的性能优势。

3.2.1.6关于可用性的分区

分区数据库对象提供分区独立性。分区独立性的这一特性可以成为高可用性策略的重要组成部分。例如，如果分区表的一个分区不可用，则表的所有其他分区保持联机且可用。应用程序可以继续对此分区表执行查询和事务，如果这些数据库操作不需要访问不可用的分区，则这些操作将成功运行。数据库管理员可以指定将每个分区存储在单独的表空间中;这将允许管理员在单个分区或多个分区集上执行备份和恢复操作（通过分区到表空间映射），独立于表中的其他分区。因此，在发生灾难的情况下，可以仅使用包括活动数据的分区来恢复数据库，然后可以在方便的时间恢复其他分区中的非活动数据，从而减少系统停机时间。鉴于可管理性，性能和可用性优势，它应该是每个数据仓库的一部分。

3.2.6关于数据仓库中的维度

维度是一种结构，通常由一个或多个层次结构组成，用于对数据进行分类。维度属性有助于描述维度值。它们通常是描述性的文本值。几个不同的维度与事实相结合，使您能够回答业务问题。常用的尺寸是客户，产品和时间。

维度架构对象定义列或列集之间的层次关系。层次关系是从层次结构的一个级别到下一个层次结构的功能依赖性。维度对象是逻辑关系的容器，不需要数据库中的任何空间。典型的维度是城市，州（或省），地区和国家。

通常在最低级别收集维度数据详细信息然后汇总到更高级别的总计，这对分析更有用。维表中的这些自然汇总或聚合称为层次结构。

3.2.6.1关于维度层次结构

层次结构是使用有序级别来组织数据的逻辑结构。层次结构可用于定义数据聚合。例如，在时间维度中，层次结构可能会将月份级别到季度级别的数据聚合到年级别。层次结构还可用于定义导航钻取路径和建立族结构。

在层次结构中，每个级别在逻辑上连接到它上面和下面的级别。较低级别的数据值汇总到较高级别的数据值。维度可以由多个层次结构组成。例如，在产品维度中，可能有两个层次结构 - 一个用于产品类别，另一个用于产品供应商。

维度层次结构还将组级别从一般到粒度。Q uery工具使用层次结构使您能够深入查看数据以查看不同级别的粒度。这是数据仓库的主要优势之一。

在设计层次结构时，必须考虑业务结构中的关系。例如，分部多级销售组织可能具有复杂的结构。

层次结构在维度值上强加了族结构。对于特定级别值，下一个较高级别的值是其父级，下一个较低级别的值是其子级。这些家庭关系使分析师能够快速访问数据。
