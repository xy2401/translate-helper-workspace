124.1 DBMS_PREPROCESSOR概述

首先要了解子程序的三种样式，以了解DBMS_PREPROCESSOR工作原理。

以下是三种子程序样式：

  1. 采用模式名称，单元类型名称和单元名称的子程序。

  2. 采用VARCHAR2字符串的子程序，其中包含任意PL / SQL编译单元的源文本。

  3. 采用VARCHAR2索引表的子程序，该表包含任意PL / SQL编译单元的分段源文本。

第一种样式的子程序用于打印或检索存储的PL / SQL单元的后处理源文本。用户必须具有查看本机原始源文本所需的权限。用户还必须指定定义单元的模式，单元类型和单元名称。如果架构为null，则使用当前用户架构。如果存储单元的状态为VALID且用户具有所需的权限，则保证后处理的源文本与上次编译时的单元的文本相同。

第二或第三种样式的子程序用于在当前用户模式中生成后处理的源文本。源文本作为第二个样式中的单个VARCHAR2字符串传递，或者作为第三个样式中的VARCHAR2索引表传递。源文本可以表示任意PL / SQL编译单元。典型用法是传递匿名块的源文本，并在当前用户模式中生成其后处理的源文本。当源文本超出VARCHAR2长度限制时，第三种样式可能很有用。

124.2 DBMS_PREPROCESSOR操作说明

这些注释解释了DBMS_PREPROCESSOR如何使用三种子程序样式。

  • 对于第一种样式的子程序，存储的PL / SQL单元的状态不需要是VALID 。同样，作为VARCHAR2字符串或VARCHAR2索引表传入的源文本可能包含编译时错误。如果在生成后处理源时发现错误，则错误消息文本也将出现在经过后处理的源文本的末尾。在某些情况下，可以因错误而中止预处理。发生这种情况时，后处理的源文本将显示为不完整，并且相关的错误消息可以帮助指示在预处理期间发生了错误。

  • 对于第二或第三种样式的子程序，源文本可以表示任意PL / SQL编译单元。但是，有效PL / SQL编译单元的源文本不能包含常用的前缀，例如CREATE OR REPLACE 。通常，输入源应该以一种从ALL_SOURCE视图获得的方式进行语法准备。以下列表给出了一些PL / SQL编译单元的有效初始语法示例。 

    匿名块（BEGIN | DECLARE）......包PACKAGE <名称> ...包体PACKAGE BODY <名称> ...程序程序<名称> ...功能功能<名称> ...输入TYPE <name> ...type body TYPE BODY <name> ...触发器（BEGIN | DECLARE）......

    如果源文本表示有效的命名PL / SQL单元，则在生成后处理源文本后将不会创建该单元。

  • 如果从ALL_SOURCE视图获取包装的PL / SQL单元的文本，则关键字WRAPPED始终紧跟单元的名称，如下例所示：

    程序“一些程序”包装了a000000 b2 ......

    如果这样的源文本被提交给一个GET_POST_PROCESSED_SOURCE功能或到一个PRINT_POST_PROCESSED_SOURCE程序 ，例外DBMS_PREPROCESSOR 。 WRAPPED_INPUT被提出。

15.2.3 DECLARE_REWRITE_EQUIVALENCE过程

只要等效声明保持启用，并且destination_stmt在性能方面更有利，此过程将创建一个声明，指示source_stmt在功能上等同于destination_stmt 。

声明的范围是系统范围的。查询重写引擎使用此类声明在QUERY_REWRITE_INTEGRITY = trusted和stale_tolerated模式下执行重写转换。

由于查询重写引擎无法强制执行源语句和目标语句之间的基础等效，因此只能在trusted和stale_tolerated完整性模式下重写查询。

句法

DBMS_ADVANCED_REWRITE.DECLARE_REWRITE_EQUIVALENCE（名称VARCHAR2，source_stmt VARCHAR2，destination_stmt VARCHAR2，validate BOOLEAN：= TRUE，rewrite_mode VARCHAR2：='TEXT_MATCH'）; DBMS_ADVANCED_REWRITE.DECLARE_REWRITE_EQUIVALENCE（名称VARCHAR2，source_stmt CLOB，destination_stmt CLOB，验证BOOLEAN：= TRUE，rewrite_mode VARCHAR2：='TEXT_MATCH'）;

13.1.1 Oracle Active Data Guard中IM列存储的用途

您只能在主数据库上配置IM列存储，仅在备用数据库上配置IM列存储，或在主数据库和备用数据库上配置IM列存储。

如果为两个数据库配置IM列存储，则可以在这两个实例上填充相同或不同的对象集。该技术有效地增加了IM列存储大小。

本节包含以下主题：

  • 主数据库和备用数据库中的相同IM列存储
    在最简单的方案中，主数据库和备用数据库都包含具有相同大小的IM列存储（这不是必需的）。IM列存储包含相同的对象。
  • IM列存储仅在备用数据库中
    在此方案中，IM列存储存在于备用数据库中，但不存在于主数据库中。
  • 主要和备用IM列存储中的不同对象
    最灵活的方案是单独为主数据库和备用数据库配置IM列存储。

13.1.1.1主数据库和备用数据库中的相同IM列存储

在最简单的方案中，主数据库和备用数据库都包含具有相同大小的IM列存储（这不是必需的）。IM列存储包含相同的对象。

此方案的优点是分析查询可以访问任一数据库上的IM列存储。因此，您可以将分析查询定向到备用数据库，而不是消耗主数据库上的资源。因此，主数据库可以支持事务工作负载，而备用数据库则支持分析工作负载。

主要任务如下：

  • 在主数据库实例和备用数据库实例上设置INMEMORY_SIZE初始化参数。

  • 确保备用数据库实例上的INMEMORY_ADG_ENABLED初始化参数设置为TRUE （缺省值）。

  • 在要在两个IM列存储中填充的所有对象上设置INMEMORY属性。

如果更改对象的INMEMORY属性，则主数据库INMEMORY更改传播到备用数据库。例如，如果在sales表上设置NO INMEMORY属性，则两个IM列存储都会逐出sales 。

在主数据库上，您可以将表的列的子集启用到IM列存储中。您还可以为不同的列指定不同的压缩级别。启用特定列涉及字典更改。主数据库上的DDL将传播到Oracle Active Data Guard数据库。

13.1.1.2 IM列仅存储在备用数据库中

在此方案中，IM列存储存在于备用数据库中，但不存在于主数据库中。

在此方案中，主数据库可以用作纯OLTP数据库。IM列存储的主数据库中不需要额外的内存。您可以将分析报告应用程序定向到备用数据库，而不会牺牲主数据库的性能或消耗资源。

主要任务如下：

  • 在备用数据库实例中将INMEMORY_SIZE初始化参数设置为非零值，并在主数据库实例中将其设置为0 。

  • 确保备用数据库实例上的INMEMORY_ADG_ENABLED初始化参数设置为TRUE （缺省值）。

  • 在要备份数据库的IM列存储中填充的所有对象上，使用DISTRIBUTE FOR SERVICE子句设置INMEMORY属性。

    例如，如果您登录到主数据库，并且在sh.sales表上设置INMEMORY属性，则此表将不会填充在主数据库上的IM列存储中 - 因为不存在IM列存储这个数据库。然而，备用数据库将继承INMEMORY的属性sh.sales表。该表将填充在备用数据库的IM列存储中。

13.1.1.3主要和备用IM列存储中的不同对象

最灵活的方案是单独为主数据库和备用数据库配置IM列存储。

此方案的优点是您可以在每个数据库中运行不同的工作负载。例如，HR应用程序在主数据库中运行报表，而销售历史记录应用程序在备用数据库中运行报表。因此，两个数据库都不承担分析报告的全部负担。

主要任务如下：

  • 在备用数据库和主数据库实例INMEMORY_SIZE初始化参数设置为非零值。值不需要相同。

  • 确保备用数据库实例上的INMEMORY_ADG_ENABLED初始化参数设置为TRUE （缺省值）。

  • 设置INMEMORY ...对两个IM列存储中要填充的所有对象的DISTRIBUTE FOR SERVICE子句。该服务指定填充对象的实例。

三服务配置

在典型配置中，您可以创建三种服务：仅待机，仅限主要和主要和备用。例如，您可能需要主实例中的最新月份的sales事实表数据，但备用实例中的先前sales数据。您希望在两个实例中填充维度表。对于每个sales分区，您使用INMEMORY ...DISTRIBUTE FOR SERVICE指定备用或主要服务。对于每个维度表，指定包含主数据库实例和备用数据库实例的服务。

注意：

只要为主实例和备用实例定义了服务名称，就可以在DISTRIBUTE FOR SERVICE指定相同的服务名称，以填充主数据库和备用数据库中的相同表。

Oracle RAC和Oracle Active Data Guard

在Oracle RAC中，您可以将FOR SERVICE子句与DISTRIBUTE AUTO或DISTRIBUTE BY子句结合使用， FOR SERVICE子句指定了population的实例， DISTRIBUTE AUTO或DISTRIBUTE BY子句控制IMCU的分布。但是，在Oracle Active Data Guard中， FOR SERVICE子句指定要在其中填充指定对象的主实例或备用实例：不能使用DISTRIBUTE AUTO或DISTRIBUTE BY在主实例和备用实例之间分发IMCU。例如，您无法在主实例和备用实例之间划分sales表的填充，因此一半的IMCU位于主实例中，一半的IMCU位于备用实例中。

在Oracle Active Data Guard环境中，对象级PRIORITY属性控制填充。仅在服务处于活动状态的数据库实例中填充对象。

13.2在Oracle Active Data Guard环境中配置IM列存储

在Oracle Active Data Guard中配置IM列存储需要设置INMEMORY_SIZE ，并为要填充的对象设置适当的INMEMORY属性。

此任务假定您了解Oracle Active Data Guard概念和过程。

先决条件

您必须满足以下要求：

  • 备用数据库必须在Oracle集成系统或Oracle Cloud Platform即服务中运行。

  • COMPATIBLE初始化参数必须为12.2.0或更高。

  • 要在每个数据库中填充不同的对象，请配置相应的服务。

要在Oracle Active Data Guard环境中配置IM列存储，请执行以下操作：

  1. 在将包含IM列存储的数据库实例上设置INMEMORY_SIZE初始化参数。

    请遵循以下准则：

    • 要在主数据库和备用数据库上配置IM列存储， INMEMORY_SIZE在两个数据库实例上设置INMEMORY_SIZE 。

    • 要仅在备用数据库上配置IM列存储， INMEMORY_SIZE在备用数据库实例上设置INMEMORY_SIZE 。

  2. 确保备用数据库实例上的INMEMORY_ADG_ENABLED初始化参数设置为TRUE （缺省值）。

  3. （可选）如果要使用IM列存储启用多实例重做应用，请将ENABLE_IMC_WITH_MIRA初始化参数设置为TRUE 。

  4. 在主数据库上，使用INMEMORY属性执行DDL语句。

    任务取决于IM列存储的位置，以及是否将在每个IM列存储中填充不同的对象：

    • 如果两个数据库中都存在IM列存储，则连接到主数据库，并在没有DISTRIBUTE FOR SERVICE子句的情况下设置INMEMORY属性。例如，将INMEMORY属性应用于sh.sales表。

      根据标准规则在每个数据库上发生人口。例如，如果备用数据库上的sales优先级为NONE ，则触发完整sales扫描的备用数据库查询会在备用IM列存储中填充此表。

      注意：

      备用数据库上的完整sales扫描不会在主数据库的IM列存储中填充此表。

    • 如果仅在备用数据库中存在IM列存储，则登录到主数据库，并在没有DISTRIBUTE FOR SERVICE子句的情况下设置INMEMORY属性。

      在重做传输期间，备用数据库从主数据库接收此DDL语句。群体以正常方式发生在备用数据库上。例如，如果sales具有INMEMORY属性且优先级为NONE ，则表必须进行全面扫描才能进行填充。

    • 如果两个数据库中都存在IM列存储，并且您希望这些IM列存储包含不同的对象，则登录到主数据库，然后设置INMEMORY ...根据每个对象DISTRIBUTE FOR SERVICE 。

      在每个DDL语句中，服务指定应在其中填充对象的实例。例如，要仅在备用数据库中启用填充sales ，请在DDL语句中指定仅备用服务。要在两个数据库中启用填充products ，请在DDL语句中指定备用和主要服务。

转换列表是转换记录的集合。添加新转换记录时，它会附加到转换列表的顶部。您可以使用以下任何方法来构建转换列表：

  • 该GET_MODEL_TRANSFORMATIONS和GET_TRANSFORM_LIST函数DBMS_DATA_MINING

30.4.2.1 SET_TRANSFORM

SET_TRANSFORM过程将单个转换记录添加到转换列表。 

DBMS_DATA_MINING_TRANSFORM.SET_TRANSFORM（xtype_list IN OUT NOCOPY TRANSFORM_LIST，attribute_name VARCHAR2，attribute_subname VARCHAR2，expression VARCHAR2，reverse_expression VARCHAR2，attribute_spec VARCHAR2 DEFAULT NULL）;

使用SET_TRANSFORM指定的SQL表达式必须适合VARCHAR2 。要指定更长的表达式，可以使用SET_EXPRESSION过程，该过程通过将行附加到VARCHAR2数组来构建表达式。

30.4.2.2堆栈接口

STACK接口从转换指令表创建转换记录，并将它们添加到转换列表中。

STACK接口指定必须以相同的方式转换给定类型的全部或部分属性。例如， STACK_BIN_CAT将分类属性的分箱指令附加到变换列表。STACK接口包含三个步骤：

  1. CREATE过程创建转换定义表。例如， CREATE_BIN_CAT创建一个表来保存分类分箱指令。该表包含用于存储属性名称，属性值以及值的bin分配的列。

  2. INSERT过程计算一个或多个属性的bin边界并填充定义表。例如， INSERT_BIN_CAT_FREQ对数据源中的部分或全部分类属性执行基于频率的分级，并填充CREATE_BIN_CAT创建的表。

  3. STACK过程根据定义表中的信息创建转换记录，并将转换记录附加到转换列表。例如，STACK_BIN_CAT为存储在分类分箱定义表中的信息创建转换记录，并将转换记录附加到转换列表。

决定分箱策略的因素有很多。具有较少的值通常会导致更紧凑的模型和更快构建的模型，但它也会导致精度的某些损失。

30.4.4.3异常值处理的例程

异常值是极值，通常是与平均值的几个标准偏差。为了最大限度地减少异常值的影响，您可以对数据进行Winsorize或trim。

Winsorizing涉及将属性的尾部值设置为某个指定值。例如，对于90％Winsorization，最低5％的值设置为等于第5百分位数中的最小值，而值的高5％设置为等于第95百分位数中的最大值。

修剪将尾部值设置为NULL。该算法将它们视为缺失值。

C.1性能不佳的原因是什么？

性能不佳的一个原因是Oracle通信开销很高。Oracle必须一次处理一个SQL语句。因此，每个语句都会导致对Oracle的另一次调用以及更高的开销。在网络环境中，必须通过网络发送SQL语句，从而增加网络流量。繁重的网络流量会显着降低您的应用程序速度

性能不佳的另一个原因是效率低下的SQL语句。因为SQL非常灵活，所以可以使用两个不同的语句获得相同的结果，但是一个语句可能效率较低。例如，以下两个SELECT语句返回相同的行（每个部门的名称和编号至少有一名员工）：

EXEC SQL SELECT DNAME，DEPTNO FROM DEPT OF DEPTNO IN（从EMP中选择DEPTNO）; EXEC SQL SELECT DNAME，来自DEPT的DEPTNO EXERE EXTS（从DEPE.DEPTNO = EMP.DEPTNO的EMP中选择DEPTNO）;

但是，第一个语句较慢，因为它对DEPT表中的每个部门号执行耗时的EMP表全扫描。即使EMP的DEPTNO列已编制索引，也不会使用索引，因为子查询缺少一个命名DEPTNO的WHERE子句。

性能不佳的第三个原因是不必要的解析和绑定。回想一下，在执行SQL语句之前，Oracle必须解析并绑定它。解析意味着检查SQL语句以确保它遵循语法规则并引用有效的数据库对象。绑定意味着将SQL语句中的主机变量与其地址相关联，以便Oracle可以读取或写入其值。

许多应用程序管理游标很差。这导致不必要的解析和绑定，这显着增加了处理开销。

C.3使用主机阵列

主机阵列可以提高性能，因为它们允许您使用单个SQL语句操作整个数据集合。例如，假设您要将300名员工的工资插入EMP表。如果没有阵列，您的程序必须执行300次单独插入 - 每个员工一次。对于数组，只需要一个INSERT 。请考虑以下声明：

EXEC SQL INSERT INTO EMP（SAL）VALUES（：薪水）;

如果salary是一个简单的宿主变量，Oracle会执行一次INSERT语句，将一行插入EMP表。在该行中， SAL列具有salary的值。要以这种方式插入300行，必须执行300次INSERT语句。

但是，如果salary是大小为300的主机数组，Oracle会立即将所有300行插入到EMP表中。在每一行中， SAL列都具有salary数组中元素的值。

有关更多信息，请参阅使用主机阵列

C.7利用行级锁定

您可以指定表级锁定，但会降低事务处理选项的有效性。有关表锁定的详细信息，请参阅“ 关于使用LOCK TABLE语句 ” 。

C.8.1关于处理显式游标

回想一下，有两种类型的游标：隐式和显式。Oracle隐式声明了所有数据定义和数据操作语句的游标。但是，对于返回多行的查询，必须显式声明游标（或使用主机数组）。您使用DECLARE CURSOR语句来声明显式游标。如何处理显式游标的打开和关闭会影响性能。

如果需要重新评估活动集，只需重新打开游标即可。OPEN语句将使用任何新的主机变量值。如果不先关闭光标，可以节省处理时间。

为了使性能调优更容易，预编译器允许您重新打开已打开的游标。但是，这是ANSI / ISO嵌入式SQL标准的Oracle扩展。因此，当MODE=ANSI ，必须在重新打开之前关闭游标。

只有在想要释放通过打开光标获取的资源（内存和锁）时才CLOSE光标。例如，您的程序应该在退出之前关闭所有游标。

C.8.2光标控制

通常，有三种方法可以控制显式声明的游标：

  • 使用DECLARE ， OPEN和CLOSE语句

  • 使用PREPARE ， DECLARE ， OPEN和CLOSE语句

  • 当MODE=ANSI时， COMMIT关闭光标

用第一种方法，注意不必要的解析。OPEN语句执行解析，但前提是解析后的语句不可用，因为游标已关闭或从未打开过。您的程序应该DECLARE游标，每次主机变量的值更改时重新打开它，并且只有在不再需要SQL语句时才CLOSE它。

使用第二种方式（动态SQL方法3和4）， PREPARE语句执行解析，并且解析的语句可用，直到执行CLOSE语句。您的程序应准备SQL语句并DECLARE游标，每次主机变量的值更改时重新打开游标，如果SQL语句更改，则重新打开SQL语句并重新打开游标，并且仅在SQL语句为no时CLOSE游标需要更久。

如果可能，避免将OPEN和CLOSE语句放在循环中;这是不必要的重新解析SQL语句的潜在原因。在下一个示例中， OPEN和CLOSE语句都在外部while循环内。当MODE=ANSI ， CLOSE语句必须如图所示定位，因为ANSI要求在重新打开之前关闭游标。 

EXEC SQL DECLARE emp_cursor CURSOR FOR SELECT ename，sal from emp where sal>：salary and sal <=：salary + 1000;薪水= 0; while（salary <5000）{EXEC SQL OPEN emp_cursor; while（SQLCODE == 0）{EXEC SQL FETCH emp_cursor INTO .......} salary + = 1000; EXEC SQL CLOSE emp_cursor; }

但是，使用MODE=ORACLE ，可以在不打开游标的情况下执行CLOSE语句。通过将CLOSE语句放在外部while循环之外，可以避免在OPEN语句的每次迭代时进行可能的重新分析。 

...while（salary <5000）{EXEC SQL OPEN emp_cursor; while（sqlca.sqlcode == 0）{EXEC SQL FETCH emp_cursor INTO .......} salary + = 1000; } EXEC SQL CLOSE emp_cursor;

C.8.3关于使用光标管理选项

除非您更改其构成，否则只需解析一次SQL语句。例如，通过向其选择列表或WHERE子句添加列来更改查询的组成。HOLD_CURSOR ， RELEASE_CURSOR和MAXOPENCURSORS选项使您可以控制Oracle如何管理SQL语句的解析和重新分析。声明显式游标可以最大程度地控制解析。

C.8.4私有SQL区域和游标缓存

执行数据操作语句时，其关联的游标将链接到游标缓存中的条目。游标缓存是用于游标管理的不断更新的内存区域。游标缓存条目又链接到私有SQL区域。

该私有SQL区域是Oracle在运行时动态创建的工作区，包含已解析的SQL语句，主机变量的地址以及处理该语句所需的其他信息。显式游标允许您命名SQL语句，访问其私有SQL区域中的信息，并在某种程度上控制其处理。

C.8.5资源使用

每个用户会话中的最大打开游标数由Oracle初始化参数OPEN_CURSORS 。

MAXOPENCURSORS指定游标缓存的初始大小。如果需要新的游标且没有可用的缓存条目，Oracle会尝试重用一个条目。它的成功取决于HOLD_CURSOR和RELEASE_CURSOR的值，对于显式游标，它取决于游标本身的状态。

如果MAXOPENCURSORS的值小于实际需要的高速缓存条目数，则Oracle使用标记为可重用的第一个高速缓存条目。例如，假设INSERT语句的高速缓存条目E （1）被标记为可重用，并且高速缓存条目的数量已经等于MAXOPENCURSORS 。如果程序执行新语句，则可以将高速缓存条目E （1）及其私有SQL区域重新分配给新语句。要重新执行INSERT语句，Oracle必须重新解析它并重新分配另一个缓存条目。

如果找不到要重用的缓存条目，Oracle会分配一个额外的缓存条目。例如，如果MAXOPENCURSORS=8并且所有八个条目都处于活动状态，则会创建第九个条目。如有必要，Oracle会继续分配其他缓存条目，直到内存不足或达到OPEN_CURSORS设置的限制为止。这种动态分配增加了处理开销。

因此，为MAXOPENCURSORS指定一个较低的值可以节省内存，但会导致新的缓存条目的动态分配和解除分配。为MAXOPENCURSORS指定一个高值可确保快速执行但使用更多内存。

C.8.6不经常执行

有时， 不经常执行的SQL语句与其私有SQL区域之间的链接应该是临时的。

当HOLD_CURSOR=NO （默认值）时，在Oracle执行SQL语句并关闭游标后，预编译器HOLD_CURSOR=NO游标和游标缓存之间的链接标记为可重用。只要另一个SQL语句需要它指向的游标缓存条目，就会重用该链接。这将释放分配给私有SQL区域的内存并释放解析锁。但是，由于准备好的游标必须保持活动状态，因此即使HOLD_CURSOR=NO ，也会保持其链接。

当RELEASE_CURSOR=YES ，在Oracle执行SQL语句并且游标关闭后，将自动释放私有SQL区域并丢失解析的语句。例如，如果MAXOPENCURSORS在您的站点设置为低以节省内存，则可能需MAXOPENCURSORS 。

如果数据操作语句在数据定义语句之前并且它们引用相同的表，则为数据操作语句指定RELEASE_CURSOR=YES 。这避免了数据操作语句获得的解析锁与数据定义语句所需的排它锁之间的冲突。

当RELEASE_CURSOR=YES ，将立即删除私有SQL区域和高速缓存条目之间的链接，并释放私有SQL区域。即使您指定HOLD_CURSOR=YES ，Oracle仍必须为私有SQL区域重新分配内存并在执行之前重新分析SQL语句，因为RELEASE_CURSOR=YES会覆盖HOLD_CURSOR=YES 。

尽管如此，当RELEASE_CURSOR=YES ，重新分析可能不需要额外的处理，因为Oracle会在其共享SQL缓存中缓存 SQL语句和PL / SQL块的已解析表示。即使其光标关闭，解析后的表示仍然可用，直到它从缓存中老化。

C.8.7频繁执行

应保持频繁执行的SQL语句与其私有SQL区域之间的链接，因为私有SQL区域包含执行该语句所需的所有信息。维护对此信息的访问使得后续执行语句的速度更快。

当HOLD_CURSOR=YES ，Oracle执行SQL语句后，将保持游标和游标缓存之间的链接。因此，解析的语句和分配的内存仍然可用。这对于您希望保持活动状态的SQL语句非常有用，因为它可以避免不必要的重新分析。

当HOLD_CURSOR=YES且RELEASE_CURSOR=NO （缺省值）时，在Oracle执行SQL语句后，将保留高速缓存条目与专用SQL区域之间的链接，除非打开游标的数量超过MAXOPENCURSORS的值，否则不会重复使用。这对于经常执行的SQL语句很有用，因为解析的语句和分配的内存仍然可用。

使用默认值HOLD_CURSOR=YES和RELEASE_CURSOR=NO ，在执行具有早期Oracle版本的SQL语句后，其解析的表示仍然可用。对于Oracle数据库版本7，在类似条件下，解析后的表示只有在老化出共享SQL高速缓存之后才可用。通常，这不是问题，但如果在重新分析SQL语句之前引用对象的定义发生更改，则可能会出现意外结果。

3.1全球数据服务管理概述

全局数据服务由全球数据服务管理员管理，其职责包括以下任务：

  • 安装和升级全局服务管理器软件

  • 创建和维护全球数据服务目录

  • 启动，停止和配置全局服务管理器

  • 创建和管理全球数据服务区域和池

  • 管理全球服务

  • 监控全球数据服务框架组件

每个全局数据服务配置至少需要一个全局数据服务管理员。一个小配置可以由执行所有管理职责的一个人管理。对于具有许多区域和池的大型配置，可能需要一组共享职责的全局数据服务管理员。所有全局数据服务管理员都有权执行给定全局数据服务配置的所有列出的管理任务。

应在全局服务管理器运行的所有计算机上为全局数据服务管理员提供操作系统帐户。帐户用户应具有安装和运行全局服务管理器软件的权限。只应授予全局数据服务管理员这些权限。

还必须将全局数据服务管理员作为用户添加到全局数据服务目录数据库，并授予GSMADMIN_ROLE角色。全局数据服务管理员的数据库帐户应由目录数据库的数据库管理员创建。如果全局数据服务管理员恰好拥有此数据库的本地数据库管理员权限，则可以自行创建此帐户。

如果全局数据服务配置包含多个池，则除了管理整个配置的全局数据服务管理员之外，每个池还可以有一个或多个全局数据服务池管理员。池管理员的职责仅限于特定池的管理，并包括以下任务：

  • 添加和删除池中的数据库

  • 管理池中的全球服务

要执行这些任务，全局数据服务池管理员必须是具有适当权限的全局数据服务目录数据库的用户。使用-USER选项创建全局数据服务池时，将自动为池管理员创建数据库用户并授予权限。池管理员也可以在创建后使用gdsctl modify gdspool命令添加到池中。全局数据服务管理员始终具有管理数据库配置中任何池的权限。

应使用适当的GDSCTL命令执行所有管理操作。执行大多数GDSCTL命令需要访问全局数据服务目录。对于此类命令，必须使用适当的命令选项指定目录数据库的凭据。

许多管理操作（例如将数据库添加到全局数据服务池或启用全局服务）不仅需要更改全局数据服务目录，还需要更改全局数据服务配置中的数据库。此类命令的通用工作流程如下：

  • GDSCTL使用管理员提供的凭据连接到目录数据库，并对目录进行适当的更改。

  • 目录数据库通知全局数据服务配置中有关更改的所有全局服务管理器。使用每个全局服务管理器与目录数据库维护的Oracle Net Services连接发送通知。

  • 收到通知后，其中一个全局服务管理器连接到需要配置的配置数据库并进行适当的更改。

要支持此工作流，全局服务管理器应该能够连接到目录和配置数据库。使用GSMCATUSER帐户建立与目录数据库的连接，该帐户在数据库安装期间默认在任何Oracle数据库上创建。该帐户必须由目录数据库的数据库管理员和提供给Global Data Services管理员的密码解锁。每当将新的全局服务管理器添加到GDS配置时，全局数据服务管理员都必须指定GSMCATUSER帐户的密码。然后将密码加密并存储在全局服务管理器钱包中，以供将来全局服务管理器使用。

全局服务管理器使用GSMUSER帐户连接到池数据库，该帐户默认情况下也存在于任何Oracle数据库中。数据库安装后，帐户被锁定。在将数据库添加到全局数据服务池之前，应由本地数据库管理员解锁。GSMUSER帐户的密码将提供给池或全局数据服务管理员，后者将数据库添加到全局数据服务池，并且必须在gdsctl add database命令中指定。密码存储在全局数据服务目录中，供将来所有全局服务管理员使用。

GSMUSR密码使用PKCS 1加密/解密模式以加密形式存储在GDS目录中。您可以通过执行modify catalog命令，使用新生成的密钥加密存储在GDS目录中的GSMUSR密码。例如：

GDSCTL>修改目录-newkeys

GSMCATUSER和GSMUSER帐户由全球数据服务框架中的所有全球服务经理共享，并用于全球服务经理执行的所有管理操作，包括服务故障转移等自动操作。人类用户永远不应该使用这些帐户连接到数据库。

除GSMCATUSER和GSMUSER帐户外，GSMADMIN_INTERNAL帐户还用于目录和池数据库中的GDS配置。此帐户的唯一目的是拥有支持GDS安装所需的表，包和其他对象。它永远不应该被解锁，分配密码或用于交互式登录。

3.3.2启动全球服务

全局服务在创建后立即自动启用。术语启用意味着如果数据库符合运行服务的条件，即满足以下条件时，可以在数据库上启动服务：

  • 数据库已打开并向全局服务经理注册。

  • 该数据库尚未禁用该服务。

  • 数据库角色与服务的角色属性匹配。

  • 数据库上的复制滞后不会超过为服务指定的最大值。

  • 该服务尚未达到由首选数据库数量定义的基数。

  • 池中没有其他数据库是启动服务的更好候选者，例如，只有在没有合格的首选数据库时才能在可用数据库上启动服务。

在用户执行start service命令之前，新创建的全局服务永远不会自动start service 。这使池管理员可以控制初始服务启动，这对于将多个服务添加到池并且需要一定数量的服务启动时的情况可能很重要。

必须首先通过执行不带-database选项的start service命令来start service具有自动管理策略（默认选项）的start service 。此命令不仅在池中的所有符合条件的数据库上启动服务，还在以下情况下启用自动服务启动：

  • 在有资格运行它的数据库上自动创建服务之后。（上一节列出了两种自动服务创建案例。）

  • 已关闭的数据库将重新启动并符合该服务的条件。

  • 数据库有资格运行该服务。例如，这可能发生，因为数据库上的复制滞后已降低到可接受的值，或者用户已增加服务基数。

带有-database选项的start service命令可用于在特定数据库上启动具有自动管理策略的服务（如果服务已通过停止全局服务中所述的stop service命令关闭）。

必须在每个单独的数据库上手动启动具有手动策略的服务，包括何时重新启动数据库或有资格运行该服务。使用手动策略对start service执行时，不带-database选项的start service命令将在池中当前存在的所有符合条件的数据库上启动服务。如果与-database选项一起使用，则该命令仅在指定的数据库上启动该服务（如果它们有资格运行它）。

注意：

本节中列出的自动服务启动的情况仅描述使用自动管理策略对服务执行start service命令时发生的情况。它们不包括由于从另一个数据库进行故障转移而在数据库上自动启动服务的情况。服务故障转移与start service命令无关，其行为与具有自动和手动管理策略的服务相同。

可以通过执行start service命令手动重新启动已停止的服务。

5.2.3提供转换

您可以选择将xform_list参数中的构建数据转换为CREATE_MODEL 。转换指令嵌入在模型中，并在模型应用于新数据时重新应用。

5.2.3.1创建转换列表

以下是创建转换列表的方法：

  • DBMS_DATA_MINING_TRANSFORM的STACK接口。

    STACK接口提供一组预定义的转换，您可以将这些转换应用于属性或属性组。例如，您可以为所有分类属性指定受监督的分箱。

  • 该SET_TRANSFORM手续DBMS_DATA_MINING_TRANSFORM 。

    SET_TRANSFORM过程将指定的SQL表达式应用于指定的属性。例如，以下语句将country_id的转换指令附加到名为my_xforms的转换列表中。在算法处理开始之前，转换指令将country_id除以10。反向转换将country_id乘以10。 

    dbms_data_mining_transform。SET_TRANSFORM（my_xforms，'country_id'，NULL，'country_id / 10'，'country_id * 10'）;

    反向转换应用于模型详细信息中。如果country_id是受监督模型的目标，则反向变换也将应用于评分目标。

5.2.3.2转换列表和自动数据准备

了解转换列表和自动数据准备（ADP）之间的交互。

CREATE_MODEL的转换列表参数与控制的PREP_AUTO设置进行交互 ADP：

  • 启用ADP并指定转换列表后，您的转换将应用自动转换并嵌入到模型中。您指定的转换将在自动转换之前执行。

  • 当ADP关闭并指定转换列表时，您的转换将应用并嵌入到模型中，但不会执行系统生成的转换。

  • 启用ADP并且未指定转换列表时，将应用系统生成的转换并将其嵌入到模型中。

  • 当ADP关闭且您未指定转换列表时，模型中不会嵌入转换;您必须单独准备用于构建，测试和评分模型的数据集。

B.6使用Tools Instant Client Package连接到数据库

SQLLDR小时/ your_password @ \ “MYDOMAIN // mymachine上 ： 端口 / MYDB \”。

sqlldr hr / your_password @MYDB

14.1工作负载智能概述

工作负载捕获会生成许多二进制文件（称为捕获文件），其中包含有关捕获的工作负载的相关信息。存储在捕获文件中的信息使工作负载重放能够在以后实际再现捕获的工作负载。 对于由工作负载捕获记录的每个客户端请求，捕获的信息包括SQL文本，绑定值，事务信息，计时数据，访问对象的标识符以及有关请求的其他信息。

Workload Intelligence使您能够以其他方式使用存储在捕获文件中的信息，包括：

  • 分析并模拟工作量

  • 发现工作量中的重要模式和趋势

  • 在工作负载捕获期间可视化生产系统上运行的内容

本节介绍Workload Intelligence并包含以下主题：

  • 工作负载智能的用例

  • 使用工作负载智能的要求

也可以看看：

  • 有关工作负载捕获的信息，请参阅“ Workload Capture ”

  • 有关工作负载重放的信息，请参阅“ 工作负载重播 ”

14.1.1关于Workload Intelligence

Workload Intelligence包含一套Java程序，使您可以分析存储在捕获的工作负载中的数据。这些Java程序对工作负载捕获期间记录的数据进行操作，以创建描述工作负载的模型。 此模型可以帮助您识别作为工作负载一部分执行的重要模板模式。

一个 template表示只读SQL语句，或由一个或多个SQL语句组成的整个事务。如果两个SQL语句（或事务）表现出显着的相似性，则它们由相同的模板表示。

Workload Intelligence使您能够通过探索模板模式和相应的SQL语句，更好地可视化捕获的工作负载的外观。对于每一个您可以查看重要的统计信息，例如给定模式的执行次数以及模式在执行期间所消耗的数据库时间。

14.1.2工作负载智能的用例

您可以使用Workload Intelligence发现捕获的工作负载中的重要模式。

在生产系统中执行的SQL语句通常不是由用户手动输入的，而是来自在连接到数据库服务器的应用程序服务器上运行的一个或多个应用程序。应用程序中通常存在有限数量的此类SQL语句。即使在特定语句的每次执行中使用不同的绑定值，其SQL文本基本上保持不变。

根据用户对应用程序的输入，执行代码路径，该代码路径包括按应用程序代码定义的给定顺序提交给数据库的一个或多个SQL语句。频繁的用户操作对应于定期执行的应用程序代码路径。这种频繁执行的代码路径生成由数据库以给定顺序执行的SQL语句的频繁模式。通过分析捕获的工作负载，Workload Intelligence发现了这些模式并将它们与相关的执行统计信息相关联。换句话说，Workload Intelligence使用存储在捕获文件中的信息来发现在工作负载捕获期间由生产系统中运行的应用程序的重要代码路径生成的模式。Workload Intelligence执行此操作时无需任何有关应用程序的信息。

使用Workload Intelligence发现重要模式：

  • 使您能够在工作负载捕获期间更好地可视化数据库中正在运行的内容。

  • 提供可用于优化的更多信息。

  • 提供更好的上下文，因为SQL语句不是孤立的，而是组合在一起的。

14.1.3使用工作负载智能的要求

Workload Intelligence使用存储在捕获文件中的信息，并且不需要使用工作负载重放来执行工作负载。此外，Workload Intelligence不需要任何用户架构，用户数据或与生产系统的连接。 为了避免生产系统中的任何开销，建议在已复制捕获文件的测试系统上使用Workload Intelligence，尤其是对于大型捕获的工作负载，因为运行Workload Intelligence可能会占用大量资源。

用于调用包含Workload Intelligence的Java程序的必要Java类打包在$ORACLE_HOME/rdbms/jlib/dbrintelligence.jar 。必须在类路径中包含另外两个jar文件： $ORACLE_HOME/rdbms/jlib/dbrparser.jar和$ORACLE_HOME/jdbc/lib/ojdbc6.jar 。

Workload Intelligence还在内部使用一些SYS表和视图。

31.2.1选择挖掘函数

解释为CREATE_MODEL提供挖掘功能。

挖掘函数是CREATE_MODEL过程的必需参数。数据挖掘功能指定了可以建模和解决的一类问题。

数据挖掘功能实现有监督或无监督学习。监督学习使用一组独立属性来预测依赖属性或目标的值。无监督学习不区分依赖属性和独立属性。监督功能是预测性的。无监督功能是描述性的。

注意：

在数据中挖掘术语， 函数是通过给定的数据挖掘方法要解决的一般类型的问题。在SQL语言术语中， 函数是返回值的运算符。

  • DBMS_DATA_MINING_TRANSFORM的STACK接口。

    STACK接口提供一组预定义的转换，您可以将这些转换应用于属性或属性组。例如，您可以为所有分类属性指定受监督的分箱。

  • 该SET_TRANSFORM手续DBMS_DATA_MINING_TRANSFORM 。

    dbms_data_mining_transform。SET_TRANSFORM（my_xforms，'country_id'，NULL，'country_id / 10'，'country_id * 10'）;

31.2.3.2转换列表和自动数据准备

了解转换列表和自动数据准备（ADP）之间的交互。

CREATE_MODEL的转换列表参数与控制的PREP_AUTO设置进行交互 ADP：

  • 启用ADP并指定转换列表后，您的转换将应用自动转换并嵌入到模型中。您指定的转换将在自动转换之前执行。

  • 当ADP关闭并指定转换列表时，您的转换将应用并嵌入到模型中，但不会执行系统生成的转换。

  • 启用ADP并且未指定转换列表时，将应用系统生成的转换并将其嵌入到模型中。

  • 当ADP关闭且您未指定转换列表时，模型中不会嵌入转换;您必须单独准备用于构建，测试和评分模型的数据集。

2.13.1呼叫结束迁移

在共享服务器模式下，Oracle数据库通过将可从static变量可访问的所有对象迁移到调用结束时的会话空间来保留调用之间Java程序的状态。会话空间存在于客户端的会话中，用于存储调用之间存在的static变量和对象。Oracle JVM在每次调用结束时自动执行此迁移操作。

此迁移操作是内存和性能考虑因素。因此，您应该了解调用之间存在的内容，并将static变量和对象保持在最低限度。如果不必要地将对象存储在static变量中，则会给内存管理器带来不必要的负担，以执行迁移并消耗每个会话资源。通过将static变量限制为仅需要，您可以帮助内存管理器并提高服务器的性能。

为了最大限度地同时运行Java程序的用户数量，最小化会话的占用空间非常重要。特别是，为了实现最大的可伸缩性，非活动会话应占用尽可能少的内存空间。最小化占用空间的一种简单技术是在每次调用结束时释放大型数据结构。当您在另一个调用中再次需要时，可以懒洋洋地重新创建许多数据结构。因此，Oracle JVM具有在会话即将变为非活动状态时（例如在调用结束时）调用指定Java方法的机制。

此机制是EndOfCallRegistry通知。它使您能够在调用结束时清除static变量，并在下一个调用进入时使用延迟初始化技术重新初始化变量。只有当您担心需要内存管理器在调用之间存储的存储量时，才应运行此命令。只有您在Java中实现的复杂有状态服务器应用程序才会成为问题。

是否在呼叫结束时清空数据结构然后为每个新呼叫重新创建它们的决定是典型的时间和空间权衡。在重新创建结构时花费了一些额外的时间，但是您可以通过不保持调用之间的结构来节省大量空间。此外，还需要考虑时间，因为对象（尤其是大型对象）在迁移到会话空间后访问起来会更加昂贵。惩罚来自会话表示的差异，而不是基于呼叫空间的对象。

适合此类优化的数据结构示例包括：

  • 缓冲区或缓存。

  • 一旦初始化的静态字段（例如数组）在程序过程中可以保持不变。

  • 任何动态构建的数据结构，可以在调用之间具有节省空间的表示，并且在调用期间具有更高速的表示。这可能很棘手，可能会使代码复杂化，使其难以维护。因此，只有在证明节省的空间值得付出努力之后，才应考虑这样做。

2.13.2针对呼叫结束优化的Oracle特定支持

在创建缓冲区，字段或数据结构时，您可以注册在调用结束时要清除的static变量。在oracle.aurora.memoryManager.EndOfCallRegistry类， registerCallback()方法接受一个实现Callback对象的对象。registerCallback()方法将此对象存储到调用结束。在调用结束时，Oracle JVM调用了所有已注册的Callback对象中的act()方法。实现Callback对象中的act()方法以清除用户定义的缓冲区，字段或数据结构。清除后，将从注册表中删除Callback对象。

注意：

如果呼叫结束也是会话结束，则不会启动回调，因为会话空间无论如何都会被清除。

弱表保存了调用结束回调的注册表。如果无法从Java程序访问Callback对象或值，则将从表中删除对象和值。使用弱表来保存回调也意味着注册回调不会阻止垃圾收集器回收该对象。因此，如果需要，您必须自己保持回调，并且不能依赖于将其保留的表。

68.1 DBMS_FILE_TRANSFER概述

DBMS_FILE_TRANSFER包提供了复制数据库中的二进制文件或在数据库之间传输二进制文件的过程。

目标数据库在从具有不同字节顺序的平台接收文件时转换每个块。数据文件在作为可传输操作的一部分移动到目标数据库之后可以导入，而无需RMAN转换。GET和PUT操作都将在目标上跨平台差异转换文件。但是， COPY是本地操作，因此不需要转换。

68.2 DBMS_FILE_TRANSFER安全模型

必须在SYS （ CONNECT INTERNAL ）下创建DBMS_FILE_TRANSFER包。此程序包提供的操作在当前调用用户下执行，而不是在程序包所有者（ SYS ）下执行。

要使用此界面，以下用户必须具有以下权限：

  • 本地数据库中的当前用户必须对source_directory_object参数中指定的目录对象具有READ权限。

  • 目标数据库中的已连接用户必须具有对destination_directory_object参数中指定的目录对象的WRITE特权。

68.3 DBMS_FILE_TRANSFER操作说明

DBMS_FILE_TRANSFER支持在线备份。因此，您应该小心复制或传输数据库正在修改的文件，因为这可能导致文件不一致，并且需要恢复。为了保证一致性，请在使用数据库时使文件脱机。

如果要使用DBMS_FILE_TRANSFER执行备份，请注意您正在实施自我管理的备份，因此应将文件置于热备份模式。

16 SQL / JSON函数JSON_VALUE

SQL / JSON函数json_value选择JSON数据并返回SQL标量或用户定义的SQL对象类型或SQL集合类型（varray，嵌套表）的实例。

  • 如果json_value以单个标量 JSON值为目标，则返回标量SQL值。您可以为返回的标量值指定SQL数据类型。默认情况下，它是VARCHAR2(4000) 。

  • 如果json_value以JSON 数组为目标，并指定SQL 集合类型 （varray或嵌套表）作为返回类型，则json_value将返回该集合类型的实例。

    目标JSON数组的元素提供返回的集合类型实例的元素。标量JSON数组元素在返回的集合实例中生成标量SQL值（请参阅上一页）。作为对象的JSON数组元素（参见下一个）或数组是递归处理的。

  • 如果json_value以JSON 对象为目标 ，并且您将用户定义的SQL 对象类型指定为返回类型，则json_value将返回该对象类型的实例。

    目标JSON对象的字段值提供返回的对象类型实例的属性值。将目标JSON对象的字段名称与SQL对象属性的SQL名称进行比较。标量字段值在返回的对象类型实例中生成标量SQL值（请参见上文）。以递归方式处理作为数组（参见上一个）或对象的字段值，

    最终，它是带有标量值的JSON字段的名称，这些标量值与标量SQL对象属性的名称进行比较。如果名称不完全匹配（区分大小写），则在查询编译时会引发不匹配错误 。

您还可以使用json_value创建基于函数的B树索引以与JSON数据一起使用 - 请参阅JSON数据的索引 。

函数json_value有两个必需参数，它接受可选的返回和错误子句。

json_value的第一个参数是一个SQL表达式，它返回标量SQL数据类型或用户定义的SQL对象类型的实例。标量返回值可以是数据类型VARCHAR2 ， BLOB or CLOB 。

第一个参数可以是表或视图列值，PL / SQL变量或具有适当强制转换的绑定变量。评估SQL表达式的结果用作评估路径表达式的上下文项 。

json_value的第二个参数是SQL / JSON路径表达式，后跟可选子句RETURNING ， ON ERROR和ON EMPTY 。路径表达式必须以单个标量值为目标，否则会发生错误。

默认的错误处理行为是NULL ON ERROR ，这意味着如果发生错误则不返回任何值 - 不会引发错误。特别是，如果路径表达式以非标量值（如数组）为目标，则默认情况下不会引发错误。要确保引发错误，请使用ERROR ON ERROR 。

给定JSON对象中的每个字段名称不一定是唯一的;可以重复相同的字段名称。Oracle数据库采用的流式评估始终只使用具有给定字段名称的一个对象成员;具有相同字段名称的任何其他成员将被忽略。未指定使用多个这样的成员中的哪一个。

JSON的布尔值为true和false 。当SQL / JSON函数json_value计算SQL / JSON路径表达式并且结果为JSON true或false ，它可以作为BOOLEAN值返回到PL / SQL，或者可以作为VARCHAR2值'true'返回到SQL或'false' 。

支持的SQL语句

除了少数例外，网关提供对Oracle DELETE ， INSERT ， SELECT和UPDATE语句的完全支持。

网关不支持Oracle 数据定义语言（DDL）语句。没有形式的Oracle ALTER ， CREATE ， DROP ， GRANT ，或可以使用TRUNCATE语句。相反，对于ALTER ， CREATE ， DROP和GRANT语句，如果需要对Sybase数据库使用DDL语句，请使用网关的传递功能。

注意：

TRUNCATE不能用于传递语句。

在网关将其转换为Sybase SQL后，Sybase数据库将评估所有函数。

Sybase不支持的功能

不支持在Sybase中没有等效功能的Oracle SQL函数 DELETE ， INSERT ，或 UPDATE语句，但如果语句是SELECT语句，则由Oracle数据库评估。也就是说，Oracle数据库执行发送到网关的SELECT语句的后处理。

如果在DELETE ， INSERT或UPDATE语句中使用了不受支持的函数，则会发生以下Oracle错误：

ORA-02070：数据库db_link_name在此上下文中不支持函数
