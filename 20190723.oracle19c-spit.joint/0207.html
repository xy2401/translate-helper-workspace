51.1 DBMS_DBFS_HS概述

所述DBMS_DBFS_HS包是下一个服务提供商DBMS_DBFS_CONTENT包使得能够使用磁带或Amazon S3 Web服务作为存储用于数据的。

磁带或Amazon S3 Web服务上的数据是Oracle数据库的一部分，可以通过所有标准接口访问，但只能通过数据库访问。该软件包允许用户在对其内容进行信息生命周期管理时使用磁带或Amazon S3 Web服务作为存储层。

该包最初将所有内容文件存储在一级缓存中。当第1级缓存填满时，内容文件将移至第2级缓存，然后使用批量写入移至外部存储设备。

51.5.12 STOREPUSH程序

此过程将本地暂存的数据推送到远程存储。

句法

DBMS_DBFS_HS.STOREPUSH（store_name IN VARCHAR2，路径IN VARCHAR2 DEFAULT NULL）;

参数

表51-17 STOREPUSH程序参数

参数 描述

STORE_NAME

使用说明

  • Hierarchical Store将内容文件本地缓存在数据库表中。如果在缓存中积累了足够的内容以使其有效写入外部存储设备（或缓存完全填满），则分层存储会从本地内容创建tarball并将这些tarball作为文件写入外部设备。由分层存储创建的tarball的大小由商店属性PROPNAME_OPTTARBALLSIZE 。

  • 当缓存中的可用空间量使得内容文件的缓存将推送cache_size上方使用的空间时，Hierarchical Store将在内部调用STOREPUSH 。 STOREPUSH Procedure从缓存中的现有脏或修改的内容文件中创建tarball，并将它们写出到外部设备。STOREPUSH调用不保证将所有脏内容从本地缓存写入外部存储，因为某些文件可能被其他会话锁定。

  • STOREPUSH具有内置功能，可以自动恢复操作。如果STOREPUSH呼叫在将一些tarball传输到外部设备后中断（例如网络中断），则可以在中断后重新启动它，然后从中断点继续传输数据。换句话说，在停电之前完成的工作不会丢失。STOREPUSH可以安全地重新启动，效果就是如果中断从未发生过。

  • 如果此方法成功执行，则用户无法回滚其操作。

  • 默认情况下，当path为NULL ，商店中的所有文件都是STOREPUSH候选STOREPUSH 。如果path具有有效的输入值，则将给定路径的命名空间下的所有文件从本地缓存写入外部存储。如果给定路径是现有文件，则会将其再次推送到远程存储。

C.3支持透明数据加密（TDE）

Oracle Data Guard SQL Apply可用于为启用了透明数据加密（TDE）的主数据库提供数据保护。

使用逻辑备用数据库为具有高级安全要求的应用程序提供数据保护时，请考虑以下事项：

  • 当主数据库和备用数据库都以11.1或更高的兼容级别运行时，使用服务器持有密钥的透明数据加密表将复制到逻辑备用数据库上。

  • Oracle Database 11g第2版（11.2）及更高版本中的逻辑备用数据库支持硬件安全模块上下文中的透明数据加密。

在逻辑备用数据库的上下文中，您希望复制具有加密列的表时，必须考虑以下限制：

  1. 要转换加密的重做记录，SQL Apply必须能够访问包含透明数据加密密钥的打开的钱包。因此，在创建密钥后，必须将包含密钥的钱包从主数据库复制到备用数据库。

  2. 每次更改主密钥时，必须将钱包从主数据库复制到逻辑备用数据库。

  3. Oracle建议您不要在逻辑备用数据库上重新生成主密钥，而逻辑备用数据库正在从主数据库复制加密表。这样做可能会导致SQL Apply在遇到加密的重做记录时停止。

  4. 您可以在逻辑备用数据库上重新生成复制表的加密密钥。这要求在发出重定密钥命令之前将保护设置降低到NONE 。

  5. 复制的加密表可以使用与主数据库中使用的列不同的加密方案。例如，如果HR.EMPLOYEES表的SALARY列使用AES192加密算法在主数据库中加密，则可以使用AES256加密算法在逻辑备用数据库中对其进行加密。或者， SALARY列可以在逻辑备用数据库中保持未加密状态。

在这种情况下， 支持透明数据加密（TDE）中列出的限制1,2和3适用。

C.5支持行级安全性和细粒度审计

从Oracle Database 11g开始 ，逻辑备用数据库可以自动复制通过DBMS_RLS和DBMS_FGA PL / SQL包提供的安全环境。

由于安全环境是透明维护的，因此当服务器故障转移到备用数据库时，此支持简化了安全性考虑因素的管理。它还确保应用于主数据的访问控制策略可以自动转发到备用数据库，并且备用数据可以透明地提供相同级别的保护。如果使用11g新创建备用服务器，则默认启用此复制;否则必须在适当的时候由DBA启用。

支持复制这些PL / SQL程序包要求主数据库和备用数据库都以11.1或更高的兼容性设置运行。

它还要求引用的表是逻辑备用维护对象。例如，具有rowid列的表没有由Logical Standby维护其数据，因此不会维护引用该表的DBMS_RLS和DBMS_FGA调用。

C.5.1行级安全性

行级安全性（也称为虚拟专用数据库（VPD））是一种在访问表，视图或同义词时以精细粒度级别强制实施安全性的功能。

当用户直接或间接访问受VPD策略保护的表，视图或同义词时，服务器会动态修改用户的SQL语句。修改创建由实现安全策略的函数返回的WHERE条件（称为谓词）。使用可以在函数中表达或返回的任何条件，动态地，对用户透明地修改语句。VPD策略可以应用于SELECT ， INSERT ， UPDATE ， INDEX和DELETE语句。通过使用DBMS_RLS包来应用安全策略来实现VPD。

当在主服务器上执行DBMS_RLS过程时，在重做中捕获附加信息，该重做允许在备用数据库上逻辑重建和执行过程调用。Logical Standby支持复制VPD的辅助对象，例如上下文，数据库登录触发器及其支持包。您必须确保将这些对象放在维护的架构中，并且没有配置任何会停止复制的DDL跳过。

C.5.2细粒度审计

细粒度审计提供了一种审计选择语句的方法。

DBMS_FGA程序包允许捕获访问表的所有select语句以及访问的数据。FGA策略可以应用于特定列，甚至只应用于返回指定谓词返回TRUE行的那些select语句。

当在主DBMS_FGA上执行DBMS_FGA过程时， DBMS_FGA附加信息捕获到重做，以允许在备用数据库上逻辑重建和执行过程调用。

C.5.3跳过和启用PL / SQL复制

PL / SQL可以使用skip和skip_error规则配置为与DDL语句完全相同，只是不支持在包和过程上使用通配符。

例如，要跳过VPD的所有方面，请执行以下操作：

DBMS_LOGSTDBY.Skip（stmt =>'PL / SQL'，schema_name =>'SYS'，object_name =>'DBMS_RLS'，use_like => FALSE）;

指定的模式是定义包的模式。要跳过包中的单个过程，语法如下：

DBMS_LOGSTDBY.Skip（stmt =>'PL / SQL'，schema_name =>'SYS'，object_name =>'DBMS_RLS.ADD_POLICY'，use_like => FALSE）;

要在某些模式或表上跳过VPD，必须使用跳过过程。skip过程传递要执行的完全限定的PL / SQL语句，例如：

DBMS_RLS.DROP_POLICY（object_schema =>'SCOTT，object_name =>'EMP'，policy_name =>'MYPOLICY'）;

然后，该过程可以解析语句以决定是跳过它，应用它还是停止应用并让DBA采取补偿措施。

与DDL不同，PL / SQL上的跳过过程不支持返回替换语句。

您还可以在逻辑备用数据库中创建作业。只要逻辑备用数据库维持备用角色，这些作业就会运行。

C.11.2.2支持DBMS_SCHEDULER

已提供对DBMS_SCHEDULER特定支持，以允许作业在备用数据库上运行。

调度工作的一个新的属性在Oracle数据库11克称为创建database_role其内容符合database_role的属性V$DATABASE 。创建调度程序作业时，它默认为本地角色，因此在备用database_role上创建的作业默认为LOGICAL STANDBY的database_role 。作业计划程序仅执行特定于当前角色的作业。在切换或故障转移时，调度程序会自动切换到特定于新角色的正在运行的作业。

除了使用DBMS_ROLLING PL / SQL包完成滚动升级的上下文之外，调度程序作业不会复制到备用数据库。但是，可以使用DBMS_SCHEDULER.Set_Attribute过程在新角色下激活现有作业。或者，可以克隆应该在两个角色中运行的作业，并使副本特定于另一个角色。DBA_SCHEDULER_JOB_ROLES视图显示哪些作业特定于哪个角色。

调度程序作业在逻辑备用数据库上运行时遵循数据库防护。因此，要运行需要修改未维护表的作业，请将数据库防护设置为STANDBY 。 （无法在PL / SQL块中使用ALTER SESSION DISABLE GUARD语句并使其生效。）

C.11.3在逻辑备用中处理XML和XDB PL / SQL包

Logical Standby支持所有存储模型的XMLType数据，并具有一些兼容性要求。

  • 以CLOB格式存储的XMLType要求主数据库以11.0或更高的兼容性设置运行（从Oracle Database 12 c第1版（12.1）开始，不推荐使用存储为CLOB XMLType 。）

在创建逻辑备用数据库之前，必须在主数据库上标识不受支持的数据库对象。

从逻辑备用支持的角度来看，数据库上有三种类型的对象：

  • 由SQL Apply显式维护的对象

  • 由SQL Apply隐式维护的对象

  • 不由SQL Apply维护的对象

SQL Apply可能无法正确应用引用数据库链接的DDL语句。

这种陈述的一个例子如下：

这是因为逻辑备用数据库中的dblink可能不指向与主数据库相同的数据库。如果SQL Apply在执行此类DDL语句时失败，则对正在创建的表使用DBMS_LOGSTDBY.INSTANTIATE_TABLE过程，然后重新启动SQL APPLY操作。

C.14.2逻辑备用数据库中AUD $和FGA_LOG $的复制

逻辑备用数据库支持审计和细粒度审计。

对主数据库的AUD$和FGA_AUD$表所做的更改将在逻辑备用数据库中进行复制。

AUD$表和FGA_AUD$表都有一个DBID列。如果DBID值是主数据库的值，则根据主数据库上的活动将该行复制到逻辑备用数据库。如果DBID值是逻辑备用数据库的值，则由于逻辑备用数据库中的本地活动而插入该行。

之后，逻辑备用数据库承担的角色转换的结果主要角色（无论是切换或故障），则AUD$和FGA_AUD$在新的主 （最初的逻辑备用），并在新的逻辑备用表（原primary）不一定是同步的。因此，新主逻辑数据库中的AUD$或FGA_AUD$表中的所有行都不可能出现在新的逻辑备用数据库中。但是，在数据库处于主要角色时插入的所有AUD$和FGA_LOG$将被复制并存在于逻辑备用数据库中。

C.15分布式事务和XA支持

您可以使用几种不同的方法执行分布式事务。

  • 使用数据库链接以协调的方式修改多个数据库中的表。

  • 使用XA接口，由提供的PL / SQL包中的DBMS_XA包或通过OCI或JDBC库公开。XA接口实现X / Open分布式事务处理（DTP）架构。

使用这两种方法之一在分布式事务期间对主数据库所做的更改将复制到逻辑备用数据库。

但是，不会复制分布式事务状态。逻辑备用数据库不会继承此类事务的不确定或准备状态，并且不会使用在主数据库中为XA事务使用的相同全局事务标识符来复制更改。因此，如果在提交分布式事务之前故障转移到逻辑备用数据库，则更改将在逻辑备用数据库中回滚。即使主数据库上的分布式事务处于准备状态并且已成功完成两阶段提交协议的第一阶段，也会发生此回滚。切换操作等待所有活动的分布式事务完成，并且不受此限制的影响。

XA事务可以通过两种方式执行：

  • 紧密耦合，不同的XA分支共享锁

  • 松散耦合，不同的XA分支不共享锁

无论COMPATIBLE参数值如何，都支持复制由松散耦合的XA分支所做的更改。只有COMPATIBLE=11.2或更高版本才支持复制由Oracle RAC主服务器（在11g第1版中引入）中的紧密耦合分支所做的更改。

C.16支持SecureFiles LOB

当数据库兼容级别设置为11.2或更高时，支持SecureFiles LOB。

可以在主数据库的SecureFiles LOB列上启用透明数据加密和数据压缩。

C.17支持数据库文件系统（DBFS）

数据库文件系统（DBFS）在存储在数据库表中的文件和目录之上创建标准文件系统接口，这使您可以更轻松地访问和管理存储在数据库中的文件。

10使用可扩展优化器

您可以使用Oracle Database可扩展优化器来优化SQL语句执行。描述了优化概念，统计，选择性，成本分析，谓词排序以及优化器的依赖模型。

10.1查询优化概述

查询优化是选择执行SQL语句的最有效方法的过程。定义运算符中讨论的可扩展索引功能引入了用户定义的访问方法。

可扩展优化器功能允许用户定义函数和索引的作者创建统计信息收集，选择性和成本函数，优化程序在选择查询计划时使用这些函数。优化器成本模型扩展为集成用户提供的信息以评估CPU和I / O成本，其中CPU成本是使用的机器指令数，I / O成本是获取的数据块数。

具体来说，您可以执行以下计算。

  • 将成本函数和默认成本与域索引（分区或非分区），索引类型，包和独立函数相关联。优化器可以获得扫描域索引，多个域索引分区或整个索引的单个分区的成本。

  • 将选择性函数和缺省选择性与对象类型，包函数和独立函数的方法相关联。优化程序可以估计单个分区，多个分区或查询中涉及的整个表的用户定义的选择性。

  • 将统计信息收集功能与域索引和表列相关联。优化器可以在域索引或表的分区级别和对象级别收集用户定义的统计信息。

  • 使用基于成本的函数对谓词进行排序。

  • 根据访问开销为表选择用户定义的访问方法（域索引）。

  • 使用DBMS_STATS包调用用户定义的统计信息收集和删除功能。

  • 使用新数据字典视图包含有关与列，域索引，索引类型或函数关联的统计信息收集，成本或选择性函数的信息。

  • 添加提示以保留函数谓词的评估顺序。

请注意，只有基于成本的优化器已得到增强; Oracle没有改变基于规则的优化器的操作。

优化器为SQL查询和DML语句SELECT ， INSERT ， UPDATE或DELETE生成执行计划。为简单起见，我们描述了根据SELECT语句生成执行计划，但DML语句的过程类似。

执行计划包括用于在每个表的访问方法FROM子句中的表中，和一个排序，叫做连接顺序， FROM子句。系统定义的访问方法包括索引，散列簇和表扫描。优化器通过生成一组连接顺序或排列来选择计划，方法是计算每个连接顺序或排列，然后选择成本最低的过程。对于连接顺序中的每个表，优化程序计算每种可能的访问方法和连接方法的开销，并选择成本最低的方法。连接顺序的成本是访问方法和连接方法成本的总和。使用包含成本模型的算法计算成本。成本模型包括有关执行查询的物理环境的不同详细程度。

优化程序使用有关查询中引用的对象的统计信息来计算选择性和成本。使用DBMS_STATS包收集统计信息。谓词的选择性是谓词选择的表中行的比例，它是介于0和1之间的数字。

可扩展索引功能允许用户定义新的运算符，索引类型和域索引。对于用户定义的运算符和域索引，可扩展优化器功能使您可以控制优化程序使用的三个主要组件，以选择执行计划统计信息，选择性和成本。

10.1.1统计

可以使用DBMS_STATS包生成表和索引的统计信息。通常，统计信息越准确，优化程序生成的执行计划就越好。

10.1.1.1用户定义的统计信息

Extensible Optimizer功能允许您为域索引，索引类型，数据类型，单个表列和分区定义统计信息收集功能。这意味着无论何时分析域索引，都会调用用户指定的统计信息收集功能。数据库不知道用户收集的统计信息的表示和含义。

除了域索引之外，Oracle还支持表的各个列以及用户定义的数据类型的用户定义的统计信息收集功能。在前一种情况下，无论何时分析列，除了数据库收集的任何标准统计信息之外，还会调用用户定义的统计信息收集功能来收集统计信息。如果存在数据类型的统计信息收集功能，则会为正在分析的具有所需类型的表的每列调用该函数。

评估用户定义函数的成本取决于算法及其参数的统计属性。为所有可能用作所有函数的参数的列的所有可能组合存储统计信息是不切实际的。因此，Oracle仅维护各列的统计信息。功能成本也可能取决于每个参数的不同统计特性。每列可能需要对每个适用函数的每个参数位置进行统计。Oracle不支持这种统计和成本函数的扩散，因为它会降低性能。

只要存在用户定义的统计信息收集功能，就需要用户定义的丢弃统计信息的功能。

10.1.1.2分区对象的用户定义统计信息

使用系统管理的本地域索引时，必须实现ODCIStats接口的两个方法： ODCIStatsExchangePartition（）和ODCIStatsUpdPartStatistics（） 。

10.1.2选择性

优化程序使用统计信息来计算谓词的选择性。选择性是谓词选择的表或分区中行的比例。它是介于0和1之间的数字。谓词的选择性用于估计特定访问方法的成本;它还用于确定最佳连接顺序。优化程序对连接顺序的不良选择可能导致非常昂贵的执行计划。

目前，优化器使用标准算法来估计选择和连接谓词的选择性。但是，在谓词包含函数或类型方法的情况下，该算法并不总是有效。此外，谓词可以包含优化程序没有任何信息的用户定义的运算符。在这种情况下，优化器无法计算准确的选择性。

10.1.3成本

优化器估计选择最佳计划的各种访问路径的成本。例如，它计算使用索引和全表扫描的CPU和I / O成本，以便在两者之间进行选择。但是，优化器不知道域索引的内部存储结构，因此无法计算域索引成本的良好估计。

10.1.3.1用户定义的成本

为了获得更大的灵活性，已扩展成本模型，以便为域索引，索引分区和用户定义的独立函数，包函数和类型方法定义成本。用户定义的成本可以是优化程序查找的默认成本的形式，也可以是优化程序调用以计算成本的完整成本函数。

与用户定义的选择性统计信息一样，用户定义的成本统计信息是可选如果没有可用的用户定义成本，则优化程序使用启发式算法来计算估计值。然而，在缺乏关于用户定义的域索引和函数中的存储结构的足够有用信息的情况下，这样的估计可能非常不准确并且导致选择次优执行计划。

仅当域索引是用户定义的运算符的有效访问路径时，优化程序才会调用用户定义的域索引的成本函数。仅当谓词具有示例10-1中概述的形式之一时，才会调用用户定义的函数，方法和域索引的成本函数，这与用户定义的选择性函数的条件相同。

用户定义的成本函数可以返回三个成本值，每个值表示单个执行函数或域索引实现的成本：

  • CPU - 函数或域索引实现执行的机器周期数。这不包括调用函数的开销。

  • I/O - 函数或域索引实现读取的数据块数。对于域索引，这不包括对Oracle表的访问。多块I / O因子不会传递给用户定义的成本函数。

  • NETWORK - 传输的数据块数。这对分布式查询，函数和域索引实现有效。对于Oracle，此成本组件未使用，因此被忽略;但是，用户需要规定一个值，以确保向后兼容性。

优化程序根据这些成本值计算复合成本。

DBMS_ODCI包包含一个函数estimate_cpu_units以帮助从输入中获取CPU和I / O开销，其中包括用户函数的已用时间。 estimate_cpu_units通过将经过的时间乘以机器的处理器速度来测量CPU单位，并返回与用户函数关联的近似CPU指令数。对于多处理器机器， estimate_cpu_units考虑单个处理器的速度。

查询的成本是成本值的函数。优化程序初始化参数的设置确定要最小化的成本。如果optimizer_mode是first_rows ，则返回单行的资源成本最小化，优化程序模式将传递给用户定义的成本函数。否则，返回所有行的资源成本最小化。

10.2定义统计，选择性和成本函数

您可以为域索引和列计算和存储用户定义的统计信息。用户定义的函数和域索引的选择性和成本函数可以在计算中使用标准和用户定义的统计信息。Oracle无需了解这些统计信息的内部表示，但您必须提供其收集方法。您自行负责定义此类统计信息的表示以及维护这些统计信息。请注意，用户收集的统计信息仅由用户定义的选择性和成本函数使用;优化程序仅使用其标准统计信息。

10.6限制和建议

统计类型是普通对象类型。由于对象类型必须至少具有一个属性，因此必须具有统计类型。但是，因为永远不会访问或设置它，所以这是一个虚拟属性。

10.6.1分布式执行

Oracle的分布式实现不支持向远程功能列表添加功能。引用远程表的所有函数都作为过滤器执行。过滤器的位置发生在优化程序之外。成本模型反映了此实现，并不尝试优化这些谓词的放置。

由于谓词未发送到远程站点，因此无法在远程表上使用域索引。因此， DESCRIBE协议保持不变，并且从本地站点看不到远程域索引。

10.6.2系统管理的存储表和ASSOCIATE STATISTICS

如果要创建具有WITH SYSTEM MANAGED STORAGE TABLES SYSTEMED WITH SYSTEM MANAGED STORAGE TABLES的索引类型，则还应创建其关联的统计类型WITH SYSTEM MANAGED STORAGE TABLES 。如果使用系统分区表收集有关本地索引列的统计信息，则Oracle服务器会在分区维护操作期间为它们维护系统分区统计信息表。当indextype与统计类型相关联时，您只能使用WITH SYSTEM MANAGED STORAGE TABLES选项;否则系统会引发错误。

10.6.3聚合对象级统计

使用本地索引时，维护分区级别和聚合对象级别统计信息可能很有用。在分区维护操作期间，将删除分区级别统计信息，同时调整聚合对象级别统计信息以反映操作，或者“按原样”保留以便以后重新计算。

调整或重新计算聚合统计信息的决定是基于服务器中的_minimal_stats_aggregation参数进行的。如果参数为FALSE ，则重新计算聚合统计信息。如果参数为TRUE ，则不会重新计算统计信息。

系统管理的域索引方法支持与索引类型相关联的系统管理的统计信息; indextype本身也应该是系统管理的。

注意：

当消息同时入队时，使用FIRST_MESSAGE导航选项可能更有效。如果未指定FIRST_MESSAGE选项，则Oracle数据库高级队列会在第一个出列命令时不断生成快照，从而导致性能下降。如果指定了FIRST_MESSAGE选项，则Oracle Database Advanced Queuing会为每个dequeue命令使用新快照。

将同一事务中排队的消息放入已启用消息分组的队列中将形成一个组。如果事务中只有一条消息入队，那么这将有效地形成一组消息。可以在单个事务中分组的消息数没有上限。

在尚未启用消息分组的队列中， LOCKED或REMOVE模式下的出队仅锁定单个消息。相反，试图使作为组的一部分的消息出列的出列操作将锁定整个组。当组中的所有消息都需要作为原子单元处理时，这很有用。

当组中的所有消息都已出列时，出队返回一个错误，指示已处理该组中的所有消息。然后，应用程序可以使用NEXT_TRANSACTION开始从下一个可用组中出列消息。如果没有可用的组，则在指定的WAIT时间段之后出队将超时。

2.1本体

本体论是特定领域中知识的共享概念化。

它由一组类，属性和可选实例组成。类通常与类层次结构（子类/超类关系）相关。类似地，属性可以通过属性层次结构（子属性/超级属性关系）来关联。属性可以是对称的或可传递的，或两者兼而有之。属性还可以具有为其指定的域，范围和基数约束。

基于RDFS的本体仅允许指定类层次结构，属性层次结构， instanceOf关系以及域和属性范围。

OWL本体建立在基于RDFS的本体上，另外允许指定属性特征。OWL本体可以进一步分类为OWL-Lite，OWL-DL和OWL Full。OWL-Lite将基数最小值和最大值限制为0或1。OWL-DL通过允许最小值和最大值来放宽此限制。OWL Full允许实例也被定义为类，OWL-DL和OWL-Lite本体不允许这样做。

支持的OWL子集描述了语义数据支持和不支持的OWL功能。

2.1.2支持的OWL子集

本节介绍支持的OWL词汇表子集。

Oracle数据库支持RDFS ++，OWLSIF和OWLPrime词汇表，它们具有增强的表达能力，以及OWL 2 RL。每个支持的词汇表都有相应的规则库;但是，这些规则库不需要填充，因为这三个词汇表的基本蕴涵规则是内部实现的。支持的词汇表如下：

  • RDFS ++：RDFS的最小扩展;这是RDFS加owl:sameAs和owl:InverseFunctionalProperty 。

  • OWLSIF：具有IF语义的OWL，具有完整性中的 pD *语义的词汇和语义，RDF模式的蕴涵的复杂性和复杂性以及涉及OWL词汇的语义扩展 ，作者：HJ Horst，Web Semantics 3,2（2005） ），79-115。

  • OWLPrime：以下OWL功能：

    • 基础知识：类，子类，属性，子属性，域，范围，类型

    • 性质特征：传递性，对称性，功能性，反函数性，逆性

    • 类比较：等价，不相交

    • 财产比较：等价

    • 个人比较：相同，不同

    • 类表达式：补码

    • 属性限制： hasValue ， someValuesFrom ， allValuesFrom

      与pD *一样，这些值限制支持的语义只是内涵（IF语义）。

  • OWL 2 RL：在W3C OWL 2 Web Ontology Language Profiles推荐（ http://www.w3.org/TR/owl2-profiles/#OWL_2_RL ）的“OWL 2 RL”部分中描述为：“OWL 2 RL profile适用于需要可扩展推理而不会牺牲太多表现力的应用程序。它旨在容纳OWL 2应用程序，这些应用程序可以交换语言的完整表现力以提高效率，以及RDF（S）应用程序需要OWL 2的一些额外表现力。“

    系统定义的规则库OWL2RL支持为OWL 2 RL定义的所有标准生产规则。与OWLPRIME ，用户不会在此OWL2RL规则库中看到任何规则。如果规则库OWL2RL尚不存在，则会自动创建。

    请注意，当使用OWL2RL规则库时，推理相关的优化（例如并行推理和RAW8）都适用。

  • OWL 2 EL：在W3C OWL 2 Web本体语言配置文件的推荐（ http://www.w3.org/TR/owl2-profiles/#OWL_2_EL ）的“OWL 2 EL”部分中描述为：“OWL 2 EL profile被设计为OWL 2的子集

    • 特别适用于使用定义了大量类和/或属性的本体的应用程序，

    • 捕获许多此类本体所使用的表达能力，以及

    • 可以在多项式时间内决定本体一致性，类表达式包含和实例检查。“

对于性能，默认情况下，推理引擎执行每个用户规则，而不检查推断的三元组的语法合法性（例如，文字值作为主题，空白节点作为谓词），直到最后一轮蕴涵之后。在完成最后一次恳求轮次后，推理引擎会删除所有语法上非法的三元组，而不会为这些三元组丢失任何错误。但是，由于在多轮推理期间可能存在具有非法语法的三元组，因此规则可以将这些三元组用作其前因的一部分。例如，请考虑以下用户定义的规则：

  • 规则1：

    （？s：帐户？y）（？s：国家：西班牙） - >（？y rdf：type：SpanishAccount）

  • 规则2：

    （？s：帐户？y）（？y rdf：type：SpanishAccount） - >（？s：语言“es_ES”）

规则1查找所有西班牙语用户并将其帐户指定为西班牙语帐户。规则2将具有西班牙语帐户的所有用户的语言设置为es_ES （西班牙语）。请考虑以Turtle格式显示的以下数据：

：Juan：帐号“123ABC4Z”：国家：西班牙：Alejandro：帐号“5678DEF9Y”：国家：西班牙

应用规则1和规则2会产生以下推断的三元组：

2.2.8优化owl：sameAs推理

您可以通过在执行OWLPrime蕴涵时在options参数中指定'OPT_SAMEAS=T'来优化大型owl:sameAs cliques的推理性能。（ clique是一个图表，其中每个节点都是双向连接到同一图表中的每个其他节点。）

根据OWL语义， owl:sameAs构造被视为等价关系，因此它是自反的，对称的和传递的。因此，在推理期间， owl:sameAs相关蕴涵的完全实现可以显着增加推断图的大小。请考虑以下示例三元组：

：John owl：sameAs：John1。：John owl：sameAs：John2。：John2：hasAge“32”。

将OWLPrime推断（指定了SAM组件）应用于此集合将生成以下新三元组：

：John1 owl：sameAs：John。：John2 owl：sameAs：John。：John1 owl：sameAs：John2。：John2 owl：sameAs：John1。：John owl：sameAs：John。：John1 owl：sameAs：John1。：John2 owl：sameAs：John2。：John：hasAge“32”。：John1：hasAge“32”。

在前面的示例中， :John ， :John1和:John2使用owl:sameAs关系相互连接;也就是说，他们是owl:sameAs成员owl:sameAs clique 。要为大型owl:sameAs cliques提供优化推理，您可以通过在执行OWLPrime蕴涵时在options参数中指定'OPT_SAMEAS=T'来合并owl:sameAs三元组而不会牺牲正确性。例如：

EXECUTE sem_apis.create_entailment（'M_IDX'，sem_models（'M'），sem_rulebases（'OWLPRIME'），null，null， 'OPT_SAMEAS = T' ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

当您指定此选项时，对于每个owl:sameAs clique，将选择来自clique的一个资源作为规范代表，并且该clique的所有推断都围绕该资源进行合并。使用前面的示例，如果:John1是clique代表，在合并之后，推断的图形将仅包含以下三元组：

：John1 owl：sameAs：John1。：John1：hasAge“32”。

owl:sameAs合并会产生一些开销。在推理期间，所有断言的模型都被复制到推理分区中，在那里它们与推断的三元组合并在一起。此外，对于非常大的断言图，合并和删除重复三元组会导致较大的运行时开销，因此建议仅对具有大量owl:sameAs关系和大团体大小的本体使用OPT_SAMEAS=T选项。

在OPT_SAMEAS = T选项被用于蕴涵之后， SEM_APIS.CREATE_ENTAILMENT对该蕴涵的所有后续使用也必须使用OPT_SAMEAS=T ，否则将报告错误。要禁用优化的sameAs处理，必须首先删除sameAs 。

2.2.9执行增量推理

增量推断可用于在三次添加之后有效地更新蕴涵（规则索引）。有两种方法可以为蕴涵启用增量推理：

  • 创建蕴涵时指定options参数值INC=T例如：

    EXECUTE sem_apis.create_entailment（'M_IDX'，sem_models（'M'），sem_rulebases（'OWLPRIME'），null，null， 'INC = T' ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

  • 使用SEM_APIS.ENABLE_INC_INFERENCE过程。

    如果使用此过程，则必须具有VALID状态。在调用该过程之前，如果您不拥有该蕴涵中涉及的模型，则必须确保相应的模型所有者已使用SEM_APIS.ENABLE_CHANGE_TRACKING过程为这些模型启用更改跟踪。

当为包含启用增量推断时，必须在调用该蕴涵的SEM_APIS.CREATE_ENTAILMENT过程时指定参数INC=T

蕴涵的增量推断取决于创建蕴涵所涉及的模型的应用表的触发器。这意味着增量推断仅在使用常规路径加载在所需的应用程序表中插入三元组时起作用，除非您在调用SEM_APIS.CREATE_ENTAILMENT过程中使用delta_in参数指定三元组，如下例所示将模型M_NEW的三元组添加到模型M ，并使用新的推论更新蕴涵M_IDX ：

EXECUTE sem_apis.create_entailment（'M_IDX'，sem_models（'M'），sem_rulebases（'OWLPRIME''），SEM_APIS.REACH_CLOSURE，null，null， sem_models（'M_NEW'） ，network_owner =>'RDFUSER'，network_name =>' NET1' ）;

如果增量推理调用中涉及多个模型， DEST_MODEL=指定要添加delta_in模型的目标模型，请在options参数中指定DEST_MODEL= <model_name> DEST_MODEL= <model_name> 。例如，以下原因导致模型M_NEW的语义数据被添加到模型M2 ：

执行sem_apis.create_entailment（'M_IDX'，sem_models（'M1'，'M2'，'M3'），sem_rulebases（'OWLPRIME''），SEM_APIS.REACH_CLOSURE，null，'DEST_MODEL = M2'， sem_models（'M_NEW'） ），network_owner =>'RDFUSER'，network_name =>'NET1'）;

使用增量推理时绕过传统路径加载要求的另一种方法是设置UNDO_RETENTION参数以覆盖执行批量加载时的蕴涵间隔。例如，如果最后一次蕴涵是在6小时前创建的，则UNDO_RETENTION值应设置为大于6小时;如果它小于那个，那么（考虑到繁重的工作量和有限的撤销空间），不能保证保留所有相关的撤销信息以进行增量推理。在这种情况下， SEM_APIS.CREATE_ENTAILMENT过程将回退到常规（非增量）推理。

要检查是否在模型上启用了更改跟踪，请使用SEM_APIS.GET_CHANGE_TRACKING_INFO过程。要获取有关蕴涵的增量推断的其他信息，请使用SEM_APIS.GET_INC_INF_INFO过程。

  • 它不适用于优化的owl:sameAs处理（ OPT_SAMEAS ），用户定义的规则，启用VPD的模型或启用版本的模型。

  • 它仅支持添加三元组。通过更新或删除，将完全重建蕴涵。

  • 它取决于应用程序表上的触发器。

  • 增量推断中使用的列类型（RAW8或NUMBER）必须一致。例如，如果最初使用RAW8=T来构建蕴涵，那么对于每个后续的SEM_APIS.CREATE_ENTAILMENT调用，必须使用相同的选项。要将列类型更改为NUMBER，您必须删除并重建所需的内容。

2.2.10使用并行推理

并行推理可以通过利用多核或多CPU架构的功能来提高推理性能。要使用并行推理，请在使用SEM_APIS.CREATE_ENTAILMENT过程时指定DOP （并行度）关键字和适当的值。例如：

执行sem_apis.create_entailment（'M_IDX'，sem_models（'M'），sem_rulebases（'OWLPRIME'），sem_apis。REACH_CLOSURE，null， 'DOP = 4' ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

指定DOP关键字会导致为Oracle选择的推理组件集启用并行执行

并行推断的成功在很大程度上取决于运行数据库的系统的良好硬件配置。关键是要有一个“平衡”系统，实现数据库性能调优和Oracle SQL并行执行的最佳实践。例如，不要将单个1 TB磁盘用于800 GB数据库，因为在单个物理磁盘上并行执行SQL语句甚至比在串行模式下执行SQL语句要慢。并行推理需要充足的内存;对于每个CPU核心，您应该至少有4 GB的内存。

并行推理最适合大型本体;但是，对于小型本体，推理性能也可以提高。

使用并行推断会产生一些临时存储开销。并行推理构建一个源表，其中包含基于所有源RDF / OWL模型和现有推断图的所有三元组。与存储数据和源模型索引所需的空间相比，此表可能使用额外10％到30％的存储空间。

2.2.11使用基于命名图的推理（全局和本地）

Oracle数据库中的默认推理从所提供的所有源模型或模型中获取所有断言的三元组，并在所有断言的三元组之上应用语义规则，直到达到推理闭包。即使给定的源模型包含一个或多个多个命名图形，它也没有区别，因为所有断言（无论是否是命名图形的一部分）都被视为来自单个图形的相同。（有关RDF语义图中命名图支持的介绍，请参阅命名图 。）

此默认推理可以被视为完全“全局”，因为它根本不考虑命名图。

但是，如果使用命名图，则可以覆盖默认推理，并使用以下任一功能考虑命名图：

  • 基于命名图的全局推理（NGGI），它将所有指定的命名图视为统一图。NGGI允许您缩小三元组的范围，同时实现极大的灵活性;它在基于命名图的全局推理（NGGI）中进行了解释 。

  • 基于命名图的局部推理（NGLI），它将每个指定的命名图视为一个单独的实体。NGLI在基于命名图的局部推断（NGLI）中进行了解释 。

对于使用NGGI和NGLI在一起，看到一个推荐的使用流量使用NGGI和NGLI一起 。

在创建蕴涵（规则索引）时，通过SEM_APIS.CREATE_ENTAILMENT过程的某些参数和选项指定NGGI或NGLI。

  • 基于命名图的全局推理（NGGI）
  • 基于命名图的局部推理（NGLI）
  • 一起使用NGGI和NGLI

父主题： 使用OWL推理

2.2.11.1基于命名图的全局推理（NGGI）

基于命名图的全局推理（NGGI）使您能够缩小在命名图级别（与模型级别相对）进行推理的三元组范围。它还可以灵活地选择范围;例如，您可以包含零个或多个命名图形和/或默认图形中的三元组，并且可以包含指定模型中具有空图形名称的所有三元组。

例如，在医院应用程序中，您可能只想对描述特定医院患者的一组命名图中包含的所有信息应用推理规则。如果与患者相关的命名图仅包含与实例相关的断言（ABox），则可以指定一个或多个其他模式相关模型（TBox），如例2-6所示 。

例2-6基于命名图的全局推理

执行sem_apis.create_entailment（'patients_inf'，models_in => sem_models（ '病人'，'hospital_ontology' ），rulebases_in => sem_rulebases（'owl2rl'），传递=> SEM_APIS.REACH_CLOSURE，inf_components_in => null，options =>'DOP = 4，RAW8 = T'， include_default_g => sem_models（'hospital_ontology'）， include_named_g => sem_graphs（'<urn：hospital1_patient1>'，'<urn：hospital1_patient2>'）， inf_ng_name =>'<urn：inf_graph_for_hospital1>' ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

在例2-6中 ：

  • 涉及两个模型： patients包含一组命名图，其中每个命名图包含与特定患者相关的三元组，并且hospital_ontology包含描述为医院定义的概念和关系的模式信息。这两个模型一起是源模型，它们为推理建立了整体范围。

  • include_default_g参数使指定模型中具有NULL图名的所有三元组都参与NGGI。在此示例中，模型hospital_ontology具有NULL图名称的所有三元组将包含在NGGI中。

  • include_named_g参数使得来自指定命名图（所有源模型）的所有三元组都参与NGGI。在此示例中，来自命名图形<urn:hospital1_patient1>和<urn:hospital1_patient2>三元组将包含在NGGI中。

  • inf_ng_name参数将图形名称<urn:inf_graph_for_hospital1>分配给<urn:inf_graph_for_hospital1>推断的所有新三元组。

2.2.11.2基于命名图的局部推理（NGLI）

基于命名图的局部推理（NGLI）将每个命名图视为单独的实体，而不是将图形视为单个统一图。推理逻辑在每个实体的边界内执行。您可以在默认图中指定与模式相关的断言（TBox），该默认图将参与每个命名图的推断。例如，基于具有名称G1的图形的推断三元组将在推断数据分区中被分配相同的图形名称G1 。

来自任何两个单独的命名图的断言将永远不会共同产生任何新的断言。

例如，假设以下内容：

  • 图G1包括以下断言：

    ：John：hasBirthMother：Mary。

  • 图G2包括以下断言：

    ：John：hasBirthMother：Bella。

  • 默认图表包含以下断言:hasBirthMother是一个owl:FunctionalProperty 。（此断言具有空图形名称。）

在这个例子中，基于命名图的局部推理（NGLI） 不会推断出:Mary是owl:sameAs :Bella因为这两个断言来自两个不同的图， G1和G2 。相比之下，基于命名图的全局推理（NGGI）包括G1 ， G2和功能属性定义将能够推断:Mary是owl:sameAs :Bella 。

NGLI目前不与证明生成，用户定义规则，优化owl:sameAs处理或增量推理一起使用。

例2-7基于命名图的局部推理

例2-7显示了NGLI。

执行sem_apis.create_entailment（'patients_inf'，models_in => sem_models（ 'patients'，'hospital_ontology' ），rulebases_in => sem_rulebases（'owl2rl'），传递=> SEM_APIS.REACH_CLOSURE，inf_components_in => null， options =>'LOCAL_NG_INF = T' ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

在例2-7中 ：

  • 两个模型患者和医院 - 病理学一起是源模型，并且它们建立了推断的总体范围，类似于示例2-6中的全局推断的情况。具有空图形名称的所有三元组都被视为公共模式（TBox）的一部分。推理是在每个命名图的边界内与公共模式结合进行的。

  • 然后， options参数关键字 - 值对LOCAL_NG_INF=T指定要执行基于命名图的局部推断（NGLI）。

请注意，根据设计，NGLI不适用于默认图表本身。但是，您可以在默认图上轻松应用基于命名图的全局推理（NGGI），并将inf_ng_name参数设置为null。通过这种方式，TBox推断得到了预先计算，从而提高了整体性能和存储消耗。

NGLI不允许以下内容：

  • 基于多个命名图形的三元组混合推断新关系

  • 仅使用默认图表中的三元组来推断新关系。

要获得您通常期望的推断，您应该将模式断言和实例断言分开。模式断言（例如:A rdfs:subClassOf :B和:p1 rdfs:subPropertyOf :p2 ）应作为未命名的三元组存储在默认图形中（带有空图形名称）。相反，实例断言（例如， :X :friendOf :Y ）应存储在其中一个命名图中。

有关使用NGLI以语义索引文档执行以文档为中心的推理的讨论和示例，请参阅执行以文档为中心的推理 。

2.2.11.3使用NGGI和NGLI一起

以下是一起使用NGGI和NGLI的推荐使用流程。它假设TBox和ABox存储在两个单独的模型中，TBox包含模式定义，TBox中的所有三元组都具有空图形名称，但ABox由一组描述实例相关数据的命名图组成。

  1. 在TBox上自行调用NGGI。例如：
    执行sem_apis.create_entailment（'TEST_INF'，sem_models（'abox'，'tbox'），sem_rulebases（'owl2rl'），SEM_APIS.REACH_CLOSURE， include_default_g => sem_models（'tbox'） ，network_owner =>'RDFUSER'，network_name = >'NET1'）;
  2. 为所有命名图调用NGLI。例如：
    执行sem_apis.create_entailment（'TEST_INF'，sem_models（'abox'，'tbox'），sem_rulebases（'owl2rl'），SEM_APIS.REACH_CLOSURE， options =>'LOCAL_NG_INF = T，ENTAIL_ANYWAY = T' ，network_owner =>'RDFUSER' ，network_name =>'NET1'）;

    指定ENTAIL_ANYWAY=T是因为步骤1中的NGGI调用将设置推断图形的状态为VALID ，并且步骤2中的SEM_APIS.CREATE_ENTAILMENT过程调用将立即退出，除非指定了ENTAIL_ANYWAY=T

2.3使用语义运算符查询关系数据

您可以使用语义运算符基于表列中的数据与本体中的术语之间的语义关系，以本体辅助的方式查询关系数据。

SEM_RELATED语义运算符基于语义相关性检索行。SEM_DISTANCE语义运算符返回语义相关性的距离度量，以便SEM_RELATED运算符返回的行可以使用距离度量进行排序或限制。索引类型MDSYS.SEM_INDEXTYPE允许有效执行此类查询，从而实现大型数据集的可伸缩性能。

注意：

模式专用语义网络不支持SEM_RELATED和SEM_DISTANCE。

  • 使用SEM_RELATED运算符
  • 使用SEM_DISTANCE辅助操作员
  • 创建MDSYS.SEM_INDEXTYPE类型的语义索引
  • 当索引列不是第一个参数时，使用SEM_RELATED和SEM_DISTANCE
  • 当值未存储为URI时使用URIPREFIX

2.3.1使用SEM_RELATED运算符

参考Example：Disease Ontology中的本体示例，考虑以下需要语义匹配的查询： 查找诊断类型为“Immune_System_Disorder”的所有患者。 涉及语法匹配的PATIENTS表（在示例：疾病本体中描述）的典型数据库查询将不返回任何行，因为没有行具有包含精确值Immune_System_Disorder的DIAGNOSIS列。例如，以下查询不会返回任何行：

SELECT诊断FROM患者WHERE诊断='Immune_System_Disorder';

实施例2-9 SEM_RELATED操作员

但是，患者数据表中的许多行都是相关的，因为它们的诊断属于此类。示例2-9使用SEM_RELATED运算符（而不是词法相等）从患者数据表中检索所有相关行。（在此示例中，术语Immune_System_Disorder以名称空间为前缀，默认假设是表列中的值也具有名称空间前缀。但是，情况可能并非总是如此，如使用URIPREFIX时，如果值未存储为URI，则说明 。） 

选择诊断来自患者WHERE SEM_RELATED（诊断，'<http://www.w3.org/2000/01/rdf-schema#subClassOf>'，'<http://www.example.org/medical_terms/Immune_System_Disorder>' ，sem_models（'medical_ontology'），sem_rulebases（'owlprime'））= 1;

SEM_RELATED运算符具有以下属性：

SEM_RELATED（sub VARCHAR2，predExpr VARCHAR2，obj VARCHAR2，ontologyName SEM_MODELS，ruleBases SEM_RULEBASES，index_status VARCHAR2，lower_bound INTEGER，upper_bound INTEGER）RETURN INTEGER;

sub属性是要搜索的表列的名称。表格列中的术语通常是<subject，predicate，object>三元模式中的主题。

predExpr属性表示谓词，该谓词可以显示为从主题节点到对象节点的路径上的边的标签。

obj属性表示本体中的术语，必须在表中找到相关术语（由predExpr属性相关）（在sub属性指定的列中）。该术语通常是<subject，predicate，object>三重模式中的对象。（在使用相等运算符的查询中，这将是查询项。）

ontologyName属性是包含术语之间关系的本体的名称。

rulebases属性标识一个或多个规则库，其规则已应用于本体以推断新关系。当指定此属性时，将根据本体关系和推断的新关系来回答查询。

即使相关的蕴涵（在指定的规则库应用于本体时创建）没有有效状态， index_status可选属性也允许您查询数据。如果此属性为null，则如果蕴涵没有有效状态，则查询将返回错误。如果此属性不为null，则必须为字符串VALID ， INCOMPLETE或INVALID ，以指定查询成功所需的最小状态。由于OWL不保证单调性，因此在指定OWL规则库时不应使用值INCOMPLETE 。

lower_bound和upper_bound可选属性允许您指定相关术语之间关系的距离度量的界限。有关距离测量的说明，请参阅使用SEM_DISTANCE辅助操作员 。

如果两个输入项与本predExpr指定的predExpr关系相关，则SEM_RELATED运算符返回1，如果两个输入项不相关，则返回0。如果指定了下限和上限，则如果两个输入项与大于或等于lower_bound且小于或等于upper_bound的距离度量相关，则返回1。

2.3.2.1距离信息的计算

在推理（蕴涵）期间为以下属性生成距离：定义为传递属性的OWL属性，以及RDFS subClassOf和RDFS subPropertyOf属性。通过这些属性链接的两个术语之间的距离被计算为它们在分层类结构中的最短距离。通过其他属性链接的两个术语的距离是未定义的，因此设置为null。

原始模型中的每个传递属性链接（被视为分层类结构）具有1的距离，并且根据两个术语之间的链接的数量生成推断的三元组的距离。请考虑以下假设的示例场景：

  • 如果原始图包含C1 rdfs:subClassOf C2和C2 rdfs:subClassOf C3 ，则将导出C1 rdfs:subClassof of C3 。在这种情况下：

    • C1 rdfs:subClassOf C2 ：distance = 1，因为它存在于模型中。

    • C2 rdfs:subClassOf C3 ：distance = 1，因为它存在于模型中。

    • C1 rdfs:subClassOf C3 ：distance = 2，因为它是在推理过程中生成的。

  • 如果原始图包含P1 rdfs:subPropertyOf P2和P2 rdfs:subPropertyOf P3 ，则将导出P1 rdfs：subPropertyOf P3。在这种情况下：

    • P1 rdfs:subPropertyOf P2 ：distance = 1，因为它存在于模型中。

    • P2 rdfs:subPropertyOf P3 ：distance = 1，因为它存在于模型中。

    • P1 rdfs:subPropertyOf P3 ：distance = 2，因为它是在推理过程中生成的。

  • 如果原始图包含C1 owl:equivalentClass C2和C2 owl:equivalentClass C3 ，那么将导出C1 owl:equivalentClass C3 。在这种情况下：

    • C1 owl:equivalentClass C2 ：distance = 1，因为它存在于模型中。

    • C2 owl:equivalentClass C3 ：distance = 1，因为它存在于模型中。

    • C1 owl:equivalentClass C3 ：distance = 2，因为它是在推理期间生成的。

SEM_RELATED运算符使用用户定义的规则库。但是，尚不支持将SEM_DISTANCE运算符与用户定义的规则库一起使用，并将引发错误。

2.3.3创建MDSYS.SEM_INDEXTYPE类型的语义索引

使用SEM_RELATED运算符时，可以在包含本体术语的列上创建MDSYS.SEM_INDEXTYPE类型的语义索引。创建这样的索引将导致更有效地执行查询。CREATE INDEX语句必须包含INDEXTYPE IS MDSYS.SEM_INDEXTYPE子句，以指定要创建的索引的类型。

例2-12创建语义索引

例2-12使用Example：Disease Ontology中的本体，在PATIENTS表的DIAGNOSIS列上创建一个名为DIAGNOSIS_SEM_IDX的语义索引。 

CREATE INDEX diagnosis_sem_idx ON患者（诊断）INDEXTYPE IS MDSYS.SEM_INDEXTYPE;

构建索引的列（ 示例2-12中的 DIAGNOSIS）必须是SEM_RELATED运算符的第一个参数，以便使用索引。如果它不是第一个参数，则在执行查询期间不使用索引。

示例2-13创建指定模型和规则库的语义索引

要提高某些语义查询的性能，可以通过在创建索引时指定一个或多个模型和规则库来为语义索引生成统计信息。示例2-13创建了一个索引，该索引还将生成指定模型和规则库的统计信息。在查询期间，索引可以与其他模型和规则库一起使用，但只有在创建索引期间指定的模型和规则库与查询中指定的模型和规则库相同时，才会使用统计信息。 

CREATE INDEX diagnosis_sem_idx ON患者（诊断）INDEXTYPE是MDSYS.SEM_INDEXTYPE（'ONTOLOGY_MODEL（medical_ontology）， RULEBASE（OWLPrime）'） ;

例2-14查询受益于统计信息的生成

统计信息对于返回按语义距离排序的top-k结果的查询非常有用。例2-14显示了这样的查询。 

SELECT / * + FIRST_ROWS * /诊断来自患者WHERE SEM_RELATED（诊断，'<http://www.w3.org/2000/01/rdf-schema#subClassOf>'，'<http://www.example.org / medical_terms / Immune_System_Disorder>'，sem_models（'medical_ontology'），sem_rulebases（'owlprime'），123）= 1 ORDER BY SEM_DISTANCE（123）;

2.3.4当索引列不是第一个参数时，使用SEM_RELATED和SEM_DISTANCE

如果已在作为SEM_RELATED运算符的第一个参数的表列上创建了MDSYS.SEM_INDEXTYPE类型的索引，则将使用该索引。例如，以下查询检索DIAGNOSIS列中具有值的所有行，该列是（ rdfs:subClassOf ） Immune_System_Disorder的子类。 

选择诊断来自患者WHERE SEM_RELATED（诊断，'<http://www.w3.org/2000/01/rdf-schema#subClassOf>'，'<http://www.example.org/medical_terms/Immune_System_Disorder>' ，sem_models（'medical_ontology'），sem_rulebases（'owlprime'））= 1;

但是，假设此查询需要检索在Immune_System_Disorder是子类的DIAGNOSIS列中具有值的所有行。您可以按如下方式重写查询：

选择诊断来自患者WHERE SEM_RELATED（'<http://www.example.org/medical_terms/Immune_System_Disorder>'，'<http://www.w3.org/2000/01/rdf-schema#subClassOf>'，诊断，sem_models（'medical_ontology'），sem_rulebases（'owlprime'））= 1;

但是，在这种情况下，不会使用DIAGNOSIS列上的语义索引，因为它不是SEM_RELATED运算符的第一个参数。要使用索引，可以更改前面的查询以使用inverseOf关键字，如下所示：

选择诊断来自患者WHERE SEM_RELATED（诊断，'inverseOf（http://www.w3.org/2000/01/rdf-schema#subClassOf）'，'<http://www.example.org/medical_terms/Immune_System_Disorder> '，sem_models（'medical_ontology'），sem_rulebases（'owlprime'））= 1;

此表单使表格列（构建索引的列）成为SEM_RELATED运算符的第一个参数，并且它检索DIIGNOSIS列中具有值的所有行，其中Immune_System_Disorder是子类。

2.3.5当值不存储为URI时使用URIPREFIX

默认情况下，语义运算符支持假定表中存储的值是URI。这些URI可以来自不同的名称空间。但是，如果表中的值没有URI，则可以在创建语义索引时使用URIPREFIX关键字指定URI。在这种情况下，指定的URI以表中的值为前缀并存储在索引结构中。（一个含义是不能使用多个URI）。

例2-15创建了一个使用URI前缀的语义索引。

示例2-15在语义索引创建期间指定URI前缀

CREATE INDEX diagnosis_sem_idx ON患者（诊断）INDEXTYPE是MDSYS.SEM_INDEXTYPE参数（'URIPREFIX（<http://www.example.org/medical/>）'）;

URI末尾的斜杠（/）字符很重要，因为URI没有任何解析，而是以表值（在索引结构中）作为前缀。
