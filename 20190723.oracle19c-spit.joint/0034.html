关于安装期间的字符集选择

在创建数据库之前，请确定要使用的字符集。

创建数据库后，更改其字符集通常在时间和资源方面非常昂贵。此类操作可能需要通过导出整个数据库并将其导回来转换所有字符数据。因此，在安装时仔细选择数据库字符集非常重要。

Oracle数据库使用以下字符集：

  • 存储在SQL字符数据类型（ CHAR ， VARCHAR2 ， CLOB和LONG ）中的数据。

  • 表名，列名和PL / SQL变量等标识符。

  • 存储的SQL和PL / SQL源代码，包括嵌入在此代码中的文本文字。

从Oracle Database 12c第2版（12.2）开始，从通用/事务处理或数据仓库模板创建的数据库的缺省数据库字符集是Unicode AL32UTF8 。

Unicode是通用字符集，支持世界上大多数当前使用的语言。它还支持许多历史脚本（字母表）。Unicode是许多技术的本机编码，包括Java，XML，XHTML，ECMAScript和LDAP。 Unicode非常适合支持互联网和全球经济的数据库。

由于AL32UTF8是一个多字节字符集，因此与单字节数据库字符集（如WE8ISO8859P1或WE8MSWIN1252相比，对字符数据的数据库操作可能会稍微慢WE8MSWIN1252 。与支持该语言的传统字符集相比， AL32UTF8中使用ASCII AL32UTF8集以外的字符的大多数语言的文本的存储空间要求更高。仅当存储在CLOB （字符大对象）列中时，英语数据可能需要更多空间。非字符数据类型（如NUMBER或DATE ）的存储不依赖于字符集。Unicode的普遍性和灵活性通常超过了这些额外的成本。

仅当数据库需要支持单组语言时才考虑使用旧版字符集，并且使用旧版字符集对于满足兼容性，存储或性能要求至关重要。在这种情况下要选择的数据库字符集是连接到此数据库的大多数客户端的字符集。

多租户容器数据库（CDB）的数据库字符集确定以后可以插入哪些数据库。确保为CDB选择的字符集与要插入此CDB的数据库的数据库字符集兼容。如果使用Unicode AL32UTF8作为CDB字符集，则可以在Oracle数据库支持的任何数据库字符集中插入可插拔数据库（PDB）（基于EBCDIC的字符集除外）。

所有Java类都包含对其他类的引用。传统的JVM搜索CLASSPATH命名的目录，ZIP文件和JAR文件中的类。相反，Oracle JVM在模式中搜索类模式对象。数据库中的每个类都有一个解析器规范，它与CLASSPATH Oracle数据库相对应。例如，类的解析器规范alpha列出了用于搜索alpha使用的类的模式。请注意，解析器规范是每个类，而在经典JVM中， CLASSPATH对所有类都是全局的。

除了解析器规范之外，每个类模式对象都有一个类间引用绑定列表。每个引用列表项都包含对另一个类的引用，并且包含以下内容之一：

  • 类使用引用时要调用的类模式对象的名称

  • 指示引用是否不满足的代码，即，引用模式对象是否已知

Oracle数据库工具称为 解析器维护参考列表。对于类中的每个类间引用，解析器搜索由类的解析器规范指定的模式以获取有效满足引用的类模式对象。如果所有引用都已解析，则解析程序将该类标记为有效。从未解决过或已解决失败的类被标记为无效。依赖于变为无效的模式对象的类在第一个类被标记为无效时也标记为无效。换句话说，失效从一个类向上级联到使用它的类和使用这些类的类，依此类推。解析依赖于无效类的类时，解析程序首先尝试解析引用的类，因为它可能只是因为它从未被解析而被标记为无效。解析程序不解析标记为有效的类。

开发人员可以指示loadjava工具来解析类，或者可以将解析延迟到运行时。当类尝试加载标记为无效的类时，解析程序会自动运行。最好在运行时间之前解决，以便尽早了解缺失的课程。运行时不成功的解决方案会产生ClassNotFound异常。此外，运行时解析可能会失败，原因如下：

  • 如果类树非常大，则缺少数据库资源

  • 由于循环依赖性导致的死锁

该 loadjava工具有两种解析模式：

  • 负载和决心

    -resolve选项加载您在命令行上指定的所有类，将它们标记为无效，然后解析它们。在最初加载彼此引用的类时使用此模式，并且通常在重新加载隔离类时也是如此。通过加载所有类然后解析它们，此模式可以避免在类引用稍后将在执行命令时加载的类时发生的错误消息。

  • 负载当时的解决

    此模式在运行时解析每个类。未指定-resolve选项。

    注意：

    与Java编译器一样， loadjava工具解析对类的引用，但不解析对资源的引用。确保正确加载类所需的资源文件。

如果可以，请推迟解决，直到所有类都已加载。这避免了解析器标记类无效的情况，因为它所使用的类尚未加载。

  2. 在源数据库上运行preupgrade_fixups.sql 。

6.1什么是Oracle数据库QoS管理？

Oracle数据库QoS管理是一种基于策略的自动化产品，可监控整个系统的工作负载请求。

许多公司正在整合和标准化他们的数据中心计算机系统。它们不是为每个应用程序使用单独的服务器，而是在集群数据库上运行多个应用程此外，应用程序向Internet的迁移引入了管理开放工作负载的问题。开放式工作负载受需求激增的影响，这可能使系统过载，从而导致无法完全预期或计划的新型应用程序故障。要在此类环境中保持应用程序可用并在其目标服务级别内执行，您必须：

  • 池资源。

  • 拥有可实时检测性能瓶颈的管理工具。

  • 重新分配资源以满足需求变化。

Oracle数据库QoS管理是一种基于策略的自动化产品，可监控整个系统的工作负载请求。Oracle数据库QoS管理管理跨应用程序共享的资源，并调整系统配置以使应用程序以业务所需的性能级别运行。Oracle数据库QoS管理可以优雅地响应系统配置和需求的变化，从而避免应用程序性能级别的额外振荡。

Oracle数据库QoS管理监视目标系统上每个工作请求的性能。Oracle数据库QoS管理从工作请求使用数据库服务请求连接到数据库时开始跟踪工作请求。完成工作请求所需的时间或响应时间（也称为端到端响应时间或往返时间）是从启动数据请求到数据时的时间请求已完成。通过准确测量响应时间的两个组成部分，即使用资源所花费的时间和等待使用资源所花费的时间，Oracle数据库QoS管理可以快速检测系统中的瓶颈。然后，Oracle数据库QoS管理提出重新分配资源以缓解瓶颈的建议 ，从而保留或恢复服务级别。

Oracle数据库QoS管理管理系统上的资源，以便：

  • 当有足够的资源来满足需求时，即使工作负载发生变化，也可以满足应用程序的业务级性能要求。

  • 当无法获得足够的资源来满足需求时，Oracle数据库QoS管理会尝试以较少业务关键型工作负载为代价来满足更多业务关键型工作负载的性能要求。

关于Grid Infrastructure Management Repository

每个Oracle域服务集群都包含一个Grid Infrastructure Management Repository（GIMR），但GIMR配置对于Oracle Standalone Cluster是可选的。

网格基础结构管理存储库（GIMR）或管理数据库（MGMTDB）是一个多租户数据库，具有用于每个集群的GIMR的可插拔数据库（PDB）。GIMR存储有关群集的以下信息：

  • Cluster Health Monitor收集的实时性能数据

  • Cluster Health Advisor收集的故障，诊断和指标数据

  • 有关Oracle Clusterware收集的所有资源的群集范围事件

  • 服务质量管理（QoS）收集的工作负载性能和CPU架构数据

  • Oracle Fleet补丁和配置所需的元数据

从Oracle Grid Infrastructure 19c开始，为Oracle Standalone Cluster部署配置GIMR是可选的。Oracle独立群集在Oracle ASM磁盘组或共享文件系统上本地托管GIMR;这个GIMR是一个带有单个可插拔数据库（PDB）的多租户数据库。

全局GIMR在Oracle域服务集群中运行。Oracle Domain Services Cluster在单独的Oracle ASM磁盘组中本地托管GIMR。Oracle成员集群数据库使用位于Oracle域服务集群上的远程GIMR。在远程群集上托管GIMR可减少在群集上运行额外基础架构存储库的开销。Oracle域服务集群的GIMR是一个多租户数据库，其中包含一个PDB，并为添加的每个成员集群添加了额外的PDB。

配置Oracle域服务群集时，安装程序会提示为GIMR配置单独的Oracle ASM磁盘组，默认名称为MGMT 。

目的

NTILE是一种分析功能。它将有序数据集划分为expr指示的多个存储桶，并为每一行分配相应的存储桶编号。桶的编号为1到expr 。expr值必须解析为每个分区的正常量。Oracle数据库需要一个整数，如果expr是非整数常量，那么Oracle会将该值截断为整数。返回值为NUMBER 。

存储桶中的行数最多可相差1。剩余值（剩余的行数除以桶）从每个桶分配一个，从桶1开始。

如果expr大于行数，则将填充等于行数的桶数，剩余的桶将为空。

您不能使用NTILE或expr任何其他分析函数来嵌套分析函数。但是，您可以为expr使用其他内置函数表达式。

关于Grid Infrastructure Management Repository

每个Oracle域服务集群都包含一个Grid Infrastructure Management Repository（GIMR），但GIMR配置对于Oracle Standalone Cluster是可选的。

网格基础结构管理存储库（GIMR）或管理数据库（MGMTDB）是一个多租户数据库，具有用于每个集群的GIMR的可插拔数据库（PDB）。GIMR存储有关群集的以下信息：

  • Cluster Health Monitor收集的实时性能数据

  • Cluster Health Advisor收集的故障，诊断和指标数据

  • 有关Oracle Clusterware收集的所有资源的群集范围事件

  • 服务质量管理（QoS）收集的工作负载性能和CPU架构数据

  • Oracle Fleet补丁和配置所需的元数据

从Oracle Grid Infrastructure 19c开始，为Oracle Standalone Cluster部署配置GIMR是可选的。Oracle独立群集在Oracle ASM磁盘组或共享文件系统上本地托管GIMR;这个GIMR是一个带有单个可插拔数据库（PDB）的多租户数据库。

全局GIMR在Oracle域服务集群中运行。Oracle Domain Services Cluster在单独的Oracle ASM磁盘组中本地托管GIMR。Oracle成员集群数据库使用位于Oracle域服务集群上的远程GIMR。在远程群集上托管GIMR可减少在群集上运行额外基础架构存储库的开销。Oracle域服务集群的GIMR是一个多租户数据库，其中包含一个PDB，并为添加的每个成员集群添加了额外的PDB。

配置Oracle域服务群集时，安装程序会提示为GIMR配置单独的Oracle ASM磁盘组，默认名称为MGMT 。

为Oracle XML DB配置FTP和HTTP端口以及HTTP身份验证

Oracle数据库配置助手（DBCA）不会对Oracle数据库12c中配置的Oracle XML DB港口和以后升级版本使用摘要身份验证。

Oracle建议您在配置端口时，还要配置HTTP身份验证以访问Oracle XML DB存储库，以利用改进的安全功能。

与Oracle数据库12c的开始，Oracle通过支持摘要式身份验证增强数据库的安全性。摘要式身份验证是一种行业标准协议，通常与HTTP协议一起使用。大多数HTTP客户端都支持它。摘要式身份验证可确保始终以安全的方式传输密码，即使未使用加密（HTTPS）连接也是如此。支持摘要式身份验证使组织能够部署使用Oracle XML DB HTTP的应用程序，而无需担心密码被泄露。Oracle XML DB中的摘要式身份验证支持还可确保Oracle XML DB HTTP服务器与Microsoft Web Folders WebDAV客户端保持兼容。

安装或升级新版本后，必须手动为Oracle XML DB配置FTP和HTTP端口，如下所示：

  1. 使用DBMS_XDB_CONFIG.setHTTPPort( HTTP_port_number )为Oracle XML DB设置HTTP端口：

    SQL> exec DBMS_XDB_CONFIG.setHTTPPort（ port_number ）;

  2. 使用DBMS_XDB_CONFIG.setFTPPort( FTP_port_number )设置Oracle XML DB的FTP端口：

    SQL> exec DBMS_XDB_CONFIG.setFTPPort（ FTP_port_number ）;

    注意：

    您可以分别使用DBMS_XDB_CONFIG.getFTPPort和DBMS_XDB_CONFIG.getHTTPPort查询过程中用于FTP和HTTP的端口号。

  3. 要查看所有使用的端口号，请查询DBMS_XDB_CONFIG.usedport 。

创建操作系统权限组

以下部分介绍如何为Oracle Grid Infrastructure和Oracle数据库创建操作系统组：

  • 创建OSASM组
    如果OSASM组不存在，或者您需要新的OSASM组，则创建它。
  • 为ASM Group创建OSDBA
    您必须在安装期间将组指定为ASM（ asmdba ）组的OSDBA。该组的成员被授予对Oracle自动存储管理的访问权限。
  • 为ASM Group创建OSOPER
    您可以选择在安装期间将组指定为ASM组的OSOPER（ asmoper ）。该组的成员被授予Oracle自动存储管理的启动和关闭权限。
  • 为数据库安装创建OSDBA组
    每个Oracle数据库都需要将操作系统组指定为OSDBA组。该组的成员被授予SYSDBA系统特权以管理数据库。
  • 为数据库安装创建OSOPER组
    仅当您要使用一组有限的数据库管理权限（SYSOPER操作员权限）标识一组操作系统用户时，才创建OSOPER组。
  • 为数据库安装创建OSBACKUPDBA组
    您必须在安装期间将组指定为OSBACKUPDBA组。该组的成员被授予SYSBACKUP权限，以使用RMAN或SQL * Plus执行备份和恢复操作。
  • 为数据库安装创建OSDGDBA组
    您必须在安装期间将组指定为OSDGDBA组。该组的成员被授予SYSDG权限以执行Data Guard操作。
  • 为数据库安装创建OSKMDBA组
    您必须在安装期间将组指定为OSKMDBA组。该组的成员被授予SYSKM权限以执行透明数据加密密钥库操作。
  • 为数据库安装创建OSRACDBA组
    您必须在数据库安装期间将组指定为OSRACDBA组。该组的成员被授予SYSRAC权限，以在Oracle RAC集群上执行Oracle数据库的日常管理。

使用响应文件运行Net Configuration Assistant

您可以在静默模式下运行Net Configuration Assistant，以在系统上配置和启动Oracle Net Listener，配置命名方法以及配置Oracle Net服务名称。

要以静默方式运行Net Configuration Assistant，您必须复制和编辑响应文件模板。Oracle在$ORACLE_HOME/assistants/netca目录中提供名为netca.rsp的响应文件模板。

要使用响应文件运行Net Configuration Assistant：

  1. 将netca.rsp响应文件模板从响应文件目录复制到系统上的目录：
    $ cp / directory_path /assistants/netca/netca.rsp local_directory

    在此示例中， directory_path是您复制安装二进制文件的目录的路径。

  2. 在文本编辑器中打开响应文件：
    $ vi / local_dir /netca.rsp
  3. 按照文件中的说明进行编辑。

    注意：

    如果未正确配置响应文件，Net Configuration Assistant将失败。

  4. 以Oracle软件所有者用户身份登录，并设置ORACLE_HOME环境变量以指定正确的Oracle主目录。
  5. 输入类似于以下内容的命令以静默方式运行Net Configuration Assistant：
    $ $ ORACLE_HOME / bin / netca / silent / responsefile / local_dir /netca.rsp

    在此命令中：

    • /silent选项表示以静默方式运行Net Configuration Assistant。

    • local_dir是您复制netca.rsp响应文件模板的目录的完整路径。

DDL操作中的自动锁定

数据字典（DDL）锁保护模式对象的定义，同时它由正在进行的DDL操作操作或引用。例如，当用户创建过程时，Oracle数据库会自动为过程定义中引用的所有模式对象获取DDL锁。DDL锁可防止在过程编译完成之前更改或删除这些对象。

Oracle数据库代表需要它的任何DDL事务自动获取DDL锁。用户无法显式请求DDL锁定。在DDL操作期间，仅锁定已修改或引用的各个模式对象。整个数据字典永远不会被锁定。

DDL操作还会在要修改的架构对象上获取DML锁。

独家DDL锁

独占DDL锁可防止其他会话获取DDL或DML锁。

大多数DDL操作都需要独占的DDL锁，以防止对可能修改或引用同一模式对象的其他DDL操作造成破坏性干扰。例如，当ALTER TABLE操作向其添加列时，不允许DROP TABLE操作删除表，反之亦然。但是，不会阻止对表的查询。

独占DDL锁定在DDL语句执行和自动提交期间持续存在。在获取专用DDL锁定期间，如果另一个操作已经在架构对象上保留了另一个DDL锁定，则获取将等待，直到释放旧的DDL锁定然后继续。

分享DDL锁

资源的共享DDL锁可防止对冲突的DDL操作的破坏性干扰，但允许类似DDL操作的数据并发。

例如，运行CREATE PROCEDURE语句时，包含的事务将获取所有引用表的共享DDL锁。其他事务可以同时创建引用相同表并在同一个表上获取并发共享DDL锁的过程，但是没有事务可以在任何引用的表上获取独占DDL锁。

共享DDL锁定持续DDL语句执行和自动提交的持续时间。因此，保证共享DDL锁定的事务在引用的持续时间内引用的模式对象的定义是恒定的。

易碎的解锁

关于在安装期间分配全局数据库名称

数据库名称输入字段用于设置DB_NAME ， DB_UNIQUE_NAME和DB_DOMAIN Oracle初始化参数值。

Oracle数据库软件通过其全局数据库名称标识数据库。全局数据库名称由数据库名称和数据库域组成。通常，数据库域与网络域相同，但不一定如此。全局数据库名称唯一地区分数据库与同一网络中的任何其他数据库。在安装期间或使用Oracle Database Configuration Assistant创建数据库时，可以指定全局数据库名称。

例如：
sales.us.example.com

这里：

  • sales.us是数据库的名称。数据库名称DB_UNIQUE_NAME ，部分是一个不超过30个字符的字符串，可以包含字母数字字符，下划线（_），美元符号（$）和井号（＃），但必须以字母字符开头。数据库名称中不允许使用其他特殊字符。

  • sales.us也是DB_NAME 。 DB_NAME初始化参数指定最多八个字符的数据库标识符。

  • example.com是数据库所在的数据库域。在此示例中，数据库域等于网络域。数据库名称和数据库域一起使全局数据库名称唯一。域部分是一个不超过128个字符的字符串，可以包含字母数字字符，下划线（_）和井号（＃）。DB_DOMAIN初始化参数指定数据库域名。

但是， DB_NAME参数不一定是DB_UNIQUE_NAME的前八个字符。

DB_UNIQUE_NAME参数和DB_DOMAIN名称参数组合在一起，以创建分配给初始化参数文件中的SERVICE_NAMES参数的全局数据库名称值。

系统标识符（SID）标识特定的数据库实例。SID唯一地将实例与同一计算机上的任何其他实例区分开来。每个数据库实例都需要唯一的SID和数据库名称。在大多数情况下，SID等于全局数据库名称的数据库名称部分。

关于Oracle数据库AutoUpgrade

AutoUpgrade实用程序可在升级之前识别问题，部署升级，执行升级后操作，并启动升级后的Oracle数据库。

AutoUpgrade实用程序旨在在升级之前，升级部署期间以及升级后检查和配置迁移期间自动执行升级过程。下载新版Oracle数据库二进制文件后，使用AutoUpgrade，并设置新版Oracle Home。

自2019年1月发布的最新更新（ DBJAN2019RU ）起，Auto Database支持可用于Oracle Database 12c第2版（12.2）和Oracle Database 18c（18.5）目标主目录。对于Oracle Database 12c第2版（12.2）和Oracle Database 18c（18.5）目标主目录，必须从My Oracle Support Document 2485457.1下载AutoUpgrade工具包。

在Oracle Database 19c（19.3）和更高版本的Oracle家庭目标中，默认情况下存在autoupgrade.jar文件。但是，在使用AutoUpgrade之前，Oracle建议您下载最新版本，该版本可从My Oracle Support Document 2485457.1中获得。

注意：

AutoUpgrade适用于Oracle数据库企业版和Oracle数据库标准版。它不适用于Oracle Database Express Edition。

预防问题：分析和修复模式模式

在升级之前，在分析模式下，AutoUpgrade实用程序会在升级之前对数据库执行只读分析，以便它可以识别需要修复的问题。您可以在正常的数据库操作期间运行该实用程序，而不会影在Fixup Mode中，AutoUpgrade实用程序检测并识别需要手动干预的两个修补程序，以及AutoUpgrade实用程序在升级部署阶段可以执行的修复。

简化升级：部署和升级模式

在“部署阶段”中，“自动升级”实用程序会修改数据库。它使您可以在升级期间调用自己的自定义脚本来配置数据库。在许多情况下，AutoUpgrade实用程序可以在升级过程中对数据库执行自动修复，而无需手动干预。

部署和升级升级后检查和修复

使用“部署”或“升级”模式完成升级后，AutoUpgrade将执行升级后检查。它提供了一个过程，您可以在其中启用自定义脚本以在升级的数据库上运行，并且还可以在升级后过程中运行自动升级后修订。在“部署”模式下，“自动升级”还确认升级已成功，并将数据库文件（如sqlnet.ora ， tnsname.ora和listener.ora从源主目录移动到目标主目录。完成这些操作后，升级后的Oracle数据库版本将在新的Oracle主目录中显示。

自动撤消管理是默认的撤消空间管理模式。UNDO_MANAGEMENT初始化参数指定系统应使用的撤消空间管理模式：

  • 如果UNDO_MANAGEMENT设置为AUTO （或未设置UNDO_MANAGEMENT ），则数据库实例将以自动撤消管理模式启动。

    null UNDO_MANAGEMENT初始化参数默认为Oracle Database 11g第1版（11.1）及更高版本中的自动撤消管理模式。在早期版本中，默认为手动撤消管理模式。升级早期版本时请小心。

  • 如果UNDO_MANAGEMENT设置为MANUAL ，则撤消空间在外部分配为回滚段。

  1. 将UNDO_MANAGEMENT参数设置为UNDO_MANAGEMENT=MANUAL 。
  2. 再次启动实例并运行标准业务周期以获取代表性工作负载。评估工作负载，并计算自动撤消管理所需的撤消表空间的大小。
  3. 标准业务周期完成后，运行以下函数来收集撤消表空间大小，并帮助调整撤消表空间的大小。您需要SYSDBA权限才能运行此功能。

Oracle RAC数据库的本地监听器

安装了Oracle Grid Infrastructure后，本地侦听器或默认侦听器位于Grid主目录中。listener.ora文件位于Grid_home /network/admin目录中。

如果需要，您可以编辑Grid主侦听器的listener.ora文件，以定义节点和SCAN侦听器的侦听器参数。请勿修改端点，因为这些端点由侦听器代理自动管理。

在Oracle数据库创建期间， LOCAL_LISTENER参数自动配置为指向数据库的本地侦听器。您可以手动为LOCAL_LISTENER设置值。如果修改LOCAL_LISTENER参数的值，则数据库代理进程不会自动更新此值。Oracle建议您保留参数unset，以便Database Agent进程可以自动维护它。如果未设置LOCAL_LISTENER ，则数据库代理进程会自动更新与Grid主目录中的本地侦听器关联的数据库，即使该侦听器的端口或IP地址已更改也是如此。

创建操作系统权限组

以下部分介绍如何为Oracle Grid Infrastructure和Oracle数据库创建操作系统组：

  • 创建OSASM组
    您必须在安装期间将组指定为OSASM组（ asmadmin ）。该组的成员被授予SYSASM系统权限以管理存储。
  • 为ASM Group创建OSDBA
    您必须在安装期间将组指定为ASM（ asmdba ）组的OSDBA。该组的成员被授予对Oracle自动存储管理的访问权限。
  • 为ASM Group创建OSOPER
    您可以选择在安装期间将组指定为ASM组的OSOPER（ asmoper ）。该组的成员被授予Oracle自动存储管理的启动和关闭权限。
  • 为数据库安装创建OSDBA组
    每个Oracle数据库都需要将操作系统组指定为OSDBA组。该组的成员被授予SYSDBA系统特权以管理数据库。
  • 为数据库安装创建OSOPER组
    仅当您要使用一组有限的数据库管理权限（SYSOPER操作员权限）标识一组操作系统用户时，才创建OSOPER组。
  • 为数据库安装创建OSBACKUPDBA组
    您必须在安装期间将组指定为OSBACKUPDBA组。该组的成员被授予SYSBACKUP权限，以使用RMAN或SQL * Plus执行备份和恢复操作。
  • 为数据库安装创建OSDGDBA组
    您必须在安装期间将组指定为OSDGDBA组。该组的成员被授予SYSDG权限以执行Data Guard操作。
  • 为数据库安装创建OSKMDBA组
    您必须在安装期间将组指定为OSKMDBA组。该组的成员被授予SYSKM权限以执行透明数据加密密钥库操作。
  • 为数据库安装创建OSRACDBA组
    您必须在数据库安装期间将组指定为OSRACDBA组。该组的成员被授予SYSRAC权限，以在Oracle RAC集群上执行Oracle数据库的日常管理。

关于在安装期间分配全局数据库名称

数据库名称输入字段用于设置DB_NAME ， DB_UNIQUE_NAME和DB_DOMAIN Oracle初始化参数值。

Oracle数据库软件通过其全局数据库名称标识数据库。全局数据库名称由数据库名称和数据库域组成。通常，数据库域与网络域相同，但不一定如此。全局数据库名称唯一地区分数据库与同一网络中的任何其他数据库。在安装期间或使用Oracle Database Configuration Assistant创建数据库时，可以指定全局数据库名称。

例如：
sales.us.example.com

这里：

  • sales.us是数据库的名称。数据库名称DB_UNIQUE_NAME ，部分是一个不超过30个字符的字符串，可以包含字母数字字符，下划线（_），美元符号（$）和井号（＃），但必须以字母字符开头。数据库名称中不允许使用其他特殊字符。

  • sales.us也是DB_NAME 。 DB_NAME初始化参数指定最多八个字符的数据库标识符。

  • example.com是数据库所在的数据库域。在此示例中，数据库域等于网络域。数据库名称和数据库域一起使全局数据库名称唯一。域部分是一个不超过128个字符的字符串，可以包含字母数字字符，下划线（_）和井号（＃）。DB_DOMAIN初始化参数指定数据库域名。

但是， DB_NAME参数不一定是DB_UNIQUE_NAME的前八个字符。

DB_UNIQUE_NAME参数和DB_DOMAIN名称参数组合在一起，以创建分配给初始化参数文件中的SERVICE_NAMES参数的全局数据库名称值。

系统标识符（SID）标识特定的数据库实例。SID唯一地将实例与同一计算机上的任何其他实例区分开来。每个数据库实例都需要唯一的SID和数据库名称。在大多数情况下，SID等于全局数据库名称的数据库名称部分。

为Oracle XML DB配置FTP和HTTP端口以及HTTP身份验证

Oracle数据库配置助手（DBCA）不会对Oracle数据库12c中配置的Oracle XML DB港口和以后升级版本使用摘要身份验证。

Oracle建议您在配置端口时，还要配置HTTP身份验证以访问Oracle XML DB存储库，以利用改进的安全功能。

与Oracle数据库12c的开始，Oracle通过支持摘要式身份验证增强数据库的安全性。摘要式身份验证是一种行业标准协议，通常与HTTP协议一起使用。大多数HTTP客户端都支持它。摘要式身份验证可确保始终以安全的方式传输密码，即使未使用加密（HTTPS）连接也是如此。支持摘要式身份验证使组织能够部署使用Oracle XML DB HTTP的应用程序，而无需担心密码被泄露。Oracle XML DB中的摘要式身份验证支持还可确保Oracle XML DB HTTP服务器与Microsoft Web Folders WebDAV客户端保持兼容。

安装或升级新版本后，必须手动为Oracle XML DB配置FTP和HTTP端口，如下所示：

  1. 使用DBMS_XDB_CONFIG.setHTTPPort( HTTP_port_number )为Oracle XML DB设置HTTP端口：

    SQL> exec DBMS_XDB_CONFIG.setHTTPPort（ port_number ）;

  2. 使用DBMS_XDB_CONFIG.setFTPPort( FTP_port_number )设置Oracle XML DB的FTP端口：

    SQL> exec DBMS_XDB_CONFIG.setFTPPort（ FTP_port_number ）;

    注意：

    您可以分别使用DBMS_XDB_CONFIG.getFTPPort和DBMS_XDB_CONFIG.getHTTPPort查询过程中用于FTP和HTTP的端口号。

  3. 要查看所有使用的端口号，请查询DBMS_XDB_CONFIG.usedport 。
