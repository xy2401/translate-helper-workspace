创造角色

目的

使用CREATE ROLE语句创建角色 ，该角色是可以授予用户或其他角色的一组权限。您可以使用角色来管理数据库权限。您可以为角色添加权限，然后将该角色授予用户。然后，用户可以启用角色并行使角色授予的权限。

角色包含授予该角色的所有权限以及授予该角色的其他角色的所有权限。新角色最初是空的。您可以使用GRANT语句向角色添加权限。

如果您创建一个角色NOT IDENTIFIED或IDENTIFIED EXTERNALLY或BY password ，那么Oracle数据库授予您与该角色ADMIN OPTION 。但是，如果您创建一个IDENTIFIED GLOBALLY角色，则数据库不会授予您该角色。无法直接向用户或角色授予全局角色。只能通过企业角色授予全局角色。

先决条件

您必须具有CREATE ROLE系统特权。

语义

角色

指定要创建的角色的名称。该名称必须满足“ 数据库对象命名规则 ”中列出的要求。Oracle建议该角色至少包含一个单字节字符，无论数据库字符集是否还包含多字节字符。角色名称的最大长度为30个字节。可以为单个用户一次启用的最大用户定义角色数为148。

在非CDB中，角色名称不能以C##或c##开头。

在CDB中，角色名称的要求如下：

  • 公共角色的名称必须以与COMMON_USER_PREFIX初始化参数指定的前缀不区分大小写的字符开头。默认情况下，前缀为C## 。

  • 本地角色的名称不得以与COMMON_USER_PREFIX初始化参数指定的前缀不区分大小写的字符开头。无论COMMON_USER_PREFIX的值COMMON_USER_PREFIX ，本地角色的名称都不能以C##或c##开头。

注意：

如果COMMON_USER_PREFIX的值是空字符串，则对常见或本地角色名称没有要求，但有一个例外：本地角色的名称永远不能以C##或c##开头。Oracle建议不要使用空字符串值，因为当PDB插入不同的CDB时，或者打开在创建普通用户时关闭的PDB时，它可能会导致本地角色和公共角色名称之间发生冲突。

某些角色由分发介质上提供的SQL脚本定义。

未经过身份确认的条款

指定NOT IDENTIFIED以指示此角色已由数据库授权，并且无需密码即可启用该角色。

身份证明条款

使用IDENTIFIED子句指示在使用SET ROLE语句启用角色之前，必须由指定方法授权用户。

通过密码

您可以使用带有BY password子句的密码创建本地角色 。这意味着您必须在启用角色时指定数据库的密码。

密码可以包含数据库字符集中除NULL字符(CHR(0))和双引号之外的任何字符。密码在语法上是一个标识符，可能需要用“ 数据库对象命名规则 ”的要求括在双引号中。您必须确保将数据库和需要启用该角色的客户端配置为支持包含密码的所有字符。

您可以在代理会话中启用受密码保护的角色。安全应用程序角色和受密码保护的角色都提供了在会话中启用角色的安全方法。Oracle建议在必须通过不安全的通道维护和传输密码的情况下，或者如果有多个人需要知道密码，则使用安全密码角色而不是密码保护角色。代理会话中受密码保护的角色适用于使用自动化设置角色的情况。

使用包

USING package子句允许您创建安全的应用程序角色 ，该角色只能由使用授权包的应用程序启用。如果未指定schema ，则数据库假定程序包位于您自己的模式中。

EXTERNALLY

指定EXTERNALLY以创建外部角色。 在启用角色之前，外部用户必须由外部服务（例如操作系统或第三方服务）授权。

根据操作系统，用户可能必须在启用角色之前为操作系统指定密码。

GLOBALLY

指定GLOBALLY以创建全局角色 。在登录时启用角色之前，必须授权全局用户使用企业目录服务的角色。

使用集合管理用户时，使用AS指定GLOBALLY以将目录组映射到全局角色。目录组由其域名标识。

示例：将目录用户映射到全局用户

CREATE USER scott_global全球识别为'cn = scott taylor，ou = sales，dc = abccorp，dc = com';

这有效地将名为'scott taylor'的目录用户在abccorp.com域的'sales'组织单元中映射到数据库全局用户'scott_global'。

集装箱条款

当您连接到CDB时， CONTAINER子句适用。但是，没有必要指定CONTAINER子句，因为它的默认值是唯一允许的值。

  • 要创建公共角色，必须连接到根。您可以选择指定CONTAINER = ALL ，这是连接到根目录时的默认值。

  • 要创建本地角色，您必须连接到PDB。您可以选择指定CONTAINER = CURRENT ，这是连接到PDB时的默认值。

手动升级非CDB架构Oracle数据库

此过程提供了升级非CDB体系结构Oracle数据库的步骤。

注意：

从Oracle Database 12c第1版（12.1）开始，不推荐使用非CDB体系结构。它可以在将来的版本中得到支持。

在使用此过程之前，请完成以下步骤：

  • 安装Oracle数据库软件

  • 准备新的Oracle主页

  • 运行升级前信息工具

脚步：

  1. 如果尚未执行此操作，请运行升级前信息工具。在继续之前，请查看升级前信息工具输出并更正输出中记录的所有问题。

    例如，在Linux或Unix系统上：

    $ ORACLE_HOME / jdk / bin / java -jar /opt/oracle/product/19.0.0/rdbms/admin/preupgrade.jar文件文本

  2. 确保您有适当的备份策略。

  3. 如果您还没有这样做，请准备新的Oracle主目录。

  4. （视情况而定）仅对于Oracle RAC环境，输入以下命令将CLUSTER_DATABASE的初始化参数值设置为FALSE：

    ALTER SYSTEM SET CLUSTER_DATABASE = FALSE SCOPE = SPFILE;
  5. 关闭数据库。例如：
    SQL> SHUTDOWN IMMEDIATE

  6. 如果您的操作系统是Windows，请完成以下步骤：

    1. 停止要升级的数据库的OracleService SID Oracle服务，其中SID是实例名称。例如，如果您的SID是ORCL ，则在命令提示符处输入以下内容：

      C：\> NET STOP OracleServiceORCL

    2. 使用ORADIM在命令提示符处删除Oracle服务。有关ORADIM语法和命令的完整列表，请参阅平台指南。

      例如，如果您的SID是ORCL ，则输入以下命令。 

      C：\> ORADIM -DELETE -SID ORCL

    3. 使用新Oracle Database发行版的ORADIM命令在命令提示符处为新版本Oracle Database创建服务。

      使用以下语法，其中SID是您的数据库SID， PASSWORD是您的系统密码， USERS是您要为最大用户数设置的值， ORACLE_HOME是您的Oracle主目录：

      C：\> ORADIM -NEW -SID SID -SYSPWD PASSWORD -MAXUSERS USERS -STARTMODE AUTO -PILE ORACLE_HOME \ DATABASE \ INIT SID .ORA

      大多数Oracle数据库服务使用Oracle软件安装所有者的权限登录系统。该服务以该用户的权限运行。ORADIM命令会提示您为此用户帐户提供密码。您可以使用ORADIM指定其他选项。

      在以下示例中，如果您的SID是ORCL ，则您的password （ SYSPWD ）是TWxy5791 ，最大用户数（ MAXUSERS ）是10，Oracle主路径是C:\ORACLE\PRODUCT\19.0.0\DB ，然后输入以下命令：

      C：\> ORADIM -NEW -SID ORCL -SYSPWD TWxy5791 -MAXUSERS 10 -STARTMODE AUTO -PFILE C：\ ORACLE \ PRODUCT \ 19.0.0 \ DB \ DATABASE \ INITORCL.ORA

      ORADIM将日志文件写入ORACLE_HOME\database目录。

      注意：

      如果您使用Oracle Home User帐户拥有Oracle主目录，则ORADIM命令会提示您输入该用户名和密码。

  7. 如果您的操作系统是Linux或UNIX，请执行以下检查：

    1. 您的ORACLE_SID设置正确

    2. oratab文件指向新的Oracle主目录

    3. 以下环境变量指向新的Oracle数据库目录：

      • ORACLE_HOME

      • 路径

    4. 客户端用于设置$ORACLE_HOME环境变量的任何脚本都必须指向新的Oracle主目录。

    注意：

    如果要升级Oracle Real Application Clusters数据库，请在Oracle Real Application Clusters数据库配置了实例的所有Oracle Grid Infrastructure节点上执行这些检查。

  8. 以新Oracle Database版本的Oracle安装所有者身份登录系统。

  9. 从新Oracle主目录中的admin目录启动新Oracle主目录中的SQL * Plus。

    例如：

    $ cd $ ORACLE_HOME / rdbms / admin $ pwd /u01/app/oracle/product/19.0.0/dbhome_1/rdbms/admin $ sqlplus

  10. 将SPFILE.ORA或INIT.ORA文件从旧的Oracle主目录复制到新的Oracle主目录。

  11. 使用具有SYSDBA权限的帐户连接到要升级的数据库：
    SQL> connect / as sysdba

  12. 以升级模式启动非CDB Oracle数据库：

    SQL>启动升级

    如果出现列出不支持的初始化参数的错误，请记下不支持的初始化参数并继续升级。下次关闭数据库时删除不支持的初始化参数。

    注意：

    以UPGRADE模式启动数据库使您可以基于早期的Oracle数据库版本打开数据库。它还将登录限制为AS SYSDBA会话，禁用系统触发器，并执行其他操作以准备升级环境。

  13. 退出SQL * Plus。

    例如：

    SQL> EXIT

  14. 使用升级所需的升级选项运行并行升级实用程序（ catctl.pl ）脚本。

    您可以使用dbupgrade shell命令将并行升级实用程序作为命令行shell命令运行，该命令位于Oracle_home /bin 。如果将PATH环境变量设置为包含Oracle_home/bin ，则可以直接从命令行运行该命令。例如：

    $ dbupgrade -d /u01/app/oracle/12.2.0/dbhome_1

    注意：

    • 运行Parallel Upgrade Utility命令时，请使用-d选项指定目标Oracle主目录的文件路径。使用-l选项指定要用于假脱机日志文件的目录。

  15. 成功升级后，数据库将关闭。重新启动实例，以便重新初始化系统参数以进行正常操作。例如：

    SQL> STARTUP

    在数据库关闭后重新启动，刷新所有缓存，清除缓冲区并执行其他内务处理活动。这些措施是确保升级的Oracle数据库软件的完整性和一致性的重要最后一步。

    注意：

    如果在启动数据库时遇到列出不支持的初始化参数的消息，则在重新启动之前从参数文件中删除不支持的初始化参数。如有必要，将SPFILE转换为PFILE ，以便您可以编辑该文件以删除参数。

  16. 运行catcon.pl以启动utlrp.sql ，并重新编译任何剩余的无效对象。

    例如：

    $ ORACLE_HOME / perl / bin / perl catcon.pl -n 1 -e -b utlrp -d'''。'''utlrp.sql

    因为使用-b utlrp运行命令，所以在脚本运行时会生成日志文件utlrp0.log 。日志文件提供重新编译的结果。

  17. 运行postupgrade_fixups.sql 。例如：

    SQL> @ postupgrade_fixups.sql

    注意：

    如果您未指定将脚本放在其他位置，则它位于默认路径Oracle_base /cfgtoollogs/ SID /preupgrade ，其中Oracle_base是您的Oracle基本主路径， SID是您唯一的数据库名称。

  18. 运行utlusts.sql 。该脚本验证所有问题是否已修复。

    例如：

    SQL> @ $ ORACLE_HOME / rdbms / admin / utllu122s.sql

    运行脚本时会生成日志文件utlu122s0.log ，该文件提供升级结果。您还可以在upg_summary.log查看升级报告。

    要查看有关数据库状态的信息，请在升级完成后随时多次运行utlusts.sql 。如果utlusts.sql脚本返回错误，或显示状态为VALID的组件，或者为该组件列出的版本不是最新版本，请参阅本指南中的故障排除部分。

  19. 使用DBMS_DST PL/SQL程序包升级时区文件，确保时区数据文件是最新的。您还可以在升级后调整时区数据文件。

  20. 退出SQL * Plus

    例如：

    SQL> EXIT

  21. （视情况而定）如果要升级Oracle Real Application Clusters数据库，请使用以下命令语法升级Oracle Clusterware中的数据库配置：

    srvctl升级数据库-db db-unique-name -oraclehome oraclehome

    在此语法示例中， db-unique-name是数据库名称（而不是实例名称）， oraclehome是升级数据库的Oracle主目录位置。除了早期版本中使用的简短命令行界面（CLI）选项之外， SRVCTL实用程序还支持长GNU样式选项。

  22. （视情况而定）仅对于Oracle RAC环境，在升级所有节点后，输入以下命令将CLUSTER_DATABASE的初始化参数值设置为TRUE，并启动数据库，其中db_unique_name是Oracle RAC数据库的名称：

    ALTER SYSTEM SET CLUSTER_DATABASE = TRUE SCOPE = SPFILE; srvctl start database -db db_unique_name

您的数据库现已升级。您已准备好完成升级后的过程。

警告：

如果您保留旧的Oracle软件，则永远不要使用旧软件启动升级的数据库。仅使用新Oracle Database主目录中的start命令启动Oracle数据库。

在删除旧的Oracle环境之前，请将该环境中的所有数据文件重定位到新的Oracle数据库环境。

Oracle Data Guard Broker

58.1使用DBMS_DG

Oracle数据库外部运行的应用程序可检测到某些条件，这些条件可能会保证Oracle Data Guard代理执行快速启动故障转移。由于可能的条件范围几乎是无限的，因此应由程序决定哪些条件需要快速启动故障转移。

发生此类情况时，应用程序将调用DBMS_DG 。 INITIATE_FS_FAILOVER过程，用于警告主INITIATE_FS_FAILOVER程序或快速启动故障转移目标备用数据库，应用程序希望立即进行快速启动故障转移。调用该过程的数据库然后通知观察者，只要备用数据库处于有效的快速启动故障转移状态（“观察”并且“同步”或“滞后”，立即启动快速启动故障转移） ）接受故障转移。如果配置未处于有效的快速启动故障转移状态，则INITIATE_FS_FAILOVER子程序将返回ORA错误消息（它不会发出异常信号），以通知调用应用程序无法执行快速启动故障转移。

注意：

如果您在多租户容器数据库（CDB）中工作，则DBMS_DG中的函数仅在根级别执行。确保您在根级别连接，而不是在单个可插拔数据库（PDB）级别连接。

58.2 DBMS_DG安全模型

DBMS_DG包以调用者的权限运行，并需要SYSDBA权限。

ORA-16817：不同步的快速启动故障转移配置

当配置未同步时，在最大可用快速启动故障转移配置中调用DBMS_DG.INITIATE_FS_FAILOVER 。

ORA-16819：快速启动故障转移观察器未启动

调用了DBMS_DG.INITIATE_FS_FAILOVER ，但尚未启动观察者。

ORA-16820：快速启动故障转移观察器不再观察此数据库

调用了DBMS_DG.INITIATE_FS_FAILOVER ，但配置检测到观察者可能没有运行。

1样本模式简介

多年来，Oracle使用简单的数据库模式SCOTT及其两个突出的表EMP和DEPT ，用于文档和培训中的各种示例。这些表不足以显示Oracle数据库和其他Oracle产品的基本功能。示例数据库模式可用于产品文档，课件，软件开发和应用程序演示。

1.1关于示例模式

示例数据库模式为每个Oracle数据库发行版中的示例提供了一个通用平台。示例模式是一组相互关联的数据库模式。该集提供了复杂性的方法：

  • 模式人力资源（ HR ）对于介绍基本主题非常有用。此架构的扩展支持Oracle Internet Directory演示。

  • 模式订单输入（ OE ）对于处理中间复杂性问题很有用。此架构中提供了许多数据类型，包括非标量数据类型。

  • 模式联机目录（ OC ）是在模式OE构建的对象关系数据库对象的集合。

  • 架构产品介质（ PM ）专用于打印介质数据类型。

  • 在主模式名称Information Exchange（ IX ）下收集的一组模式可用于演示Oracle Advanced Queuing功能。

  • 架构销售历史（ SH ）旨在允许具有大量数据的演示。此架构的扩展为高级分析处理提供支持。

1.2样本模式的设计原则

已经使用以下设计原则创建了示例数据库模式：

  • 简单易用 。Schemas HR和OE故意简单。它们提供了从简单到中级数据库使用的渐变路径。

  • 典型用户的相关性 。基本模式及其扩展将客户通常使用的功能带到了前台。只有最常用的数据库对象才会在模式中自动构建。整套模式提供了一个基础，人们可以扩展它以说明其他功能。

  • 可扩展性 。示例模式为添加对象以演示超出基本范围的功能提供了逻辑和物理基础。

  • 相关性 。示例模式旨在适用于电子商务和其他重要行业趋势（例如，XML）。当此目标与简单性目标冲突时，架构扩展将用于展示焦点趋势。

1.3样本模式的客户利益

示例模式提供的好处包括：

  • 背景的连续性 。当在任何地方遇到相同的表集时，用户，学生和开发人员可以花更少的时间熟悉模式，并花更多的时间来理解或解释技术概念。

  • 可用性 。客户可以在种子数据库中使用这些模式来运行Oracle数据库文档和培训材料中显示的示例。这种对示例的第一手访问有助于概念理解和应用程序开发。

  • 品质 。通过集中维护和测试构建示例模式的创建脚本以及针对模式运行的示例，Oracle Database文档和培训材料的质量得到了增强。

1.4样本模式概述

Oracle数据库示例模式基于虚构的示例公司，该公司通过各种渠道销售商品。该公司在全球范围内运营，以填补产品订单。它有几个部门，每个部门都由一个示例数据库模式表示。

  • Schema HR - Division Human Resources跟踪有关公司员工和设施的信息。

  • Schema OE - Division Order Entry通过各种渠道跟踪产品库存和公司产品的销售情况。

  • Schema PM - Division Product Media维护有关公司销售的每种产品的描述和详细信息。

  • Schema IX - Division Information Exchange通过B2B应用程序管理运输。

  • Schema SH - Division Sales跟踪业务统计数据以促进业务决策。

1.4.1 HR示例模式

在人力资源（ HR ）记录中，每个员工都有一个识别号，电子邮件地址，工作识别码，工资和经理。有些员工除了工资外还可以获得佣金。

该公司还跟踪组织内的工作信息。每份工作都有一份识别码，职称以及该工作的最低和最高工资范围。一些员工已经在公司工作了很长时间，并在公司内担任过不同的职位。员工辞职时，员工工作的持续时间，工作识别号和部门都会被记录下来。

样本公司区域多样化，因此它跟踪仓库和部门的位置。每个员工都被分配到一个部门，每个部门都由唯一的部门编号或短名称标识。每个部门都与一个位置相关联，每个位置都有一个完整的地址，其中包括街道名称，邮政编码，城市，州或省以及国家/地区代码。

在部门和仓库所在的地方，公司会记录详细信息，例如国家/地区名称，货币符号，货币名称以及国家/地区所在的地区。

B.1 Oracle RAC快速应用程序通知概述

Oracle RAC快速应用程序通知（FAN）功能提供了一个简化的API，用于通过回调机制访问FAN事件。此机制使第三方驱动程序，连接池和容器能够订阅，接收和处理FAN事件。这些API在本附录中称为Oracle RAC FAN API。

Oracle RAC FAN API提供FAN事件通知，用于开发能够充分利用Oracle Database HA功能的响应更快的应用程序。如果您不想使用通用连接池，但希望使用实现自己的连接池的FAN事件，则应使用Oracle RAC快速应用程序通知。

注意：

  • 如果您不想实现自己的连接池，那么您应该使用Oracle通用连接池来获得Oracle RAC快速应用程序通知的所有优势，以及许多其他好处。

  • 从Oracle Database 12 c第1版（12.1）开始，不支持隐式连接缓存（ICC）。Oracle建议改为使用通用连接池。

您的应用程序可以通过以下方式响应FAN事件：

  • 侦听Oracle RAC服务关闭和节点关闭事件。

  • 侦听表示任何Oracle RAC或全局数据服务（GDS）启动或重新启动的Oracle RAC服务启动事件。对于这些UP事件，FAN参数状态为UP ， event_type是以下之一： database ， instance ， service或servicemember 。

  • 支持Oracle Database Release 12 c和早期版本的FAN ONS事件语法和字段，例如，添加到每个事件的event_type和timezone字段， db_domain字段添加到除节点或公共网络事件之外的每个事件， percentf字段添加到运行时负载平衡（RLB）事件，等等。

  • 侦听负载平衡咨询事件并对其进行响应。

此功能公开FAN事件，这些事件是运行Oracle RAC的集群发送的通知，用于通知订户有关在服务级别或节点级别发生的事件。支持的FAN事件如下：

  • 服务起来

    service up事件通知连接池新实例可供使用，允许在新实例上创建会话。当前版本的Oracle RAC FAN API支持ServiceUpEvent客户端API，即simplefan.jar文件。

  • 服务下来

    服务停机事件通知受管资源已关闭且当前无法访问。有两种类型的服务停机事件：

    • 指示服务的特定实例已关闭且实例不再能够接受工作的事件。

    • 表示服务的所有情况都已关闭且服务不再能够接受工作的事件。

  • 节点向下

    节点关闭事件通知主机标识符标识的Oracle RAC节点已关闭且无法访问。当节点不再能够接受工作时，群集会发送节点关闭事件。

  • 计划下来

    计划的关闭事件包括除节点关闭事件之外的所有关闭事件。这些事件设置了以下两个字段： status=down和reason=user 。

  • 负载均衡咨询

    负载平衡咨询事件提供负载平衡算法的度量标准。定期发送负载平衡建议，以通知订户可用节点之间的推荐工作分配。

注意：

如果要实现自己的连接池，则只应使用Oracle RAC快速应用程序通知。否则，您应该使用Oracle通用连接池来获得Oracle RAC快速应用程序通知的所有优势，以及许多其他好处。

目的

PREDICTION_DETAILS返回选择中每行的预测详细信息。返回值是一个XML字符串，用于描述预测的属性。

对于回归，返回的详细信息指的是预测的目标值。对于分类和异常检测，返回的详细信息指的是最高概率类或指定的class_value 。

TOPN

如果为topN指定值，则该函数返回对预测（分数）影响最大的N属性。如果未指定topN ，则该函数将返回5个最有影响力的属性。

DESC，ASC或ABS

返回的属性按重量排序。属性的权重表示其对预测的正面或负面影响。对于回归，正权重表示更高的值预测;负权重表示较低值预测。对于分类和异常检测，正权重表示概率预测较高;负权重表示概率预测较低。

默认情况下， PREDICTION_DETAILS返回具有最高正权重（ DESC ）的属性。如果指定ASC ，则返回负权重最高的属性。如果指定ABS ，则返回权重最大的属性，无论是负数还是正数。结果按绝对值从最高到最低排序。输出中不包含权重为零的属性。

13.1关于期望最大化

混合模型的期望最大化（EM）估计是一种流行的概率密度估计技术，其用于各种应用中。Oracle Data Mining使用EM实现基于分布的聚类算法（EM-clustering）。

13.1.1期望步骤和最大化步骤

期望最大化是一种迭代方法。它以初始参数猜测开始。参数值用于计算当前模型的可能性。这是期望步骤。然后重新计算参数值以最大化可能性。这是最大化步骤。新参数估计值用于计算新的期望值，然后再次进行优化以最大化可能性。这个迭代过程一直持续到模型收敛。

13.1.2概率密度估计

在密度估计中，目标是构建一个密度函数，捕获给定种群的分布方式。在概率密度估计中，密度估计基于表示群体样本的观察数据。模型中的高数据密度区域对应于基础分布的峰值。

基于密度的聚类在概念上不同于基于距离的聚类（例如k -Means），其中重点在于最小化聚类间和最大化聚类内距离。由于其概率性质，基于密度的聚类可以计算群集分配中的可靠概率。它还可以自动处理缺失值。

13.2算法增强

尽管期望最大化（EM）已经很好地建立为基于分布的聚类算法，但它在其标准形式中提出了一些挑战。Oracle Data Mining实施包括重要的增强功能，例如可扩展的大量数据处理和自动参数初始化。本节将进一步介绍Oracle Data Mining用于解决EM群集固有限制的策略。

标准期望最大化的局限性：

  • 可伸缩性：EM具有线性可伸缩性以及记录和属性的数量。随着数据大小的增加（行和列），收敛的迭代次数趋于增加。对于复杂问题，EM收敛可能很慢并且可能对计算资源造成很大负担。

  • 高维度：EM对高维（宽）数据建模的能力有限。许多属性的存在减慢了模型收敛，并且该算法变得不能区分有意义的属性和噪声。因此，该算法在其找到相关性的能力方面受到损害。

  • 组件数量：EM通常要求用户指定组件数量。在大多数情况下，这不是用户可以提前知道的信息。

  • 参数初始化：选择合适的初始参数值会对模型的质量产生重大影响。已经用于EM的初始化策略通常在计算上是昂贵的。

  • 从组件到集群：EM模型组件通常被视为集群。这种方法可能会产生误导，因为内聚集群通常由多个组件建模。具有复杂形状的群集需要由多个组件建模。

13.2.2高维度

期望最大化（EM）的Oracle数据挖掘实现可以有效地处理具有数千个属性的高维数据。这是通过双重过程实现的：

  • 分析单列（非嵌套）属性的数据空间的成对相关性。EM混合模型中仅包含与其他属性显着相关的属性。该算法还可以配置为将维度限制为M个最相关的属性。

  • 测量相似类型事件的高维（嵌套）数值数据被投影到由EM建模的一组低维特征中。高维数值数据的一些例子是：文本，推荐，基因表达和市场篮子数据。

13.2.4参数初始化

选择适当的初始参数值会对解决方案的质量产生重大影响。期望最大化（EM）不能保证收敛到似然函数的全局最大值，而是可以收敛到局部最大值。因此，不同的初始参数值可导致不同的模型参数和不同的模型质量。

在模型搜索过程中，EM模型独立生长。随着新组件的添加，其参数将初始化为分布不合适的区域。

13.2.5从组件到集群

期望最大化（EM）模型组件通常被视为集群。但是，这种方法可能会产生误导。内聚集群通常由多个组件建模。EM中使用的概率密度函数的形状有效地预先确定了所识别的聚类的形状。例如，高斯密度函数可以识别单峰对称聚类。更复杂形状的集群需要由多个组件建模。

理想情况下，任意形状的高密度区域必须被解释为单个簇。为实现此目的，EM的Oracle数据挖掘实现构建了一个基于各个组件分布重叠的组件层次结构。Oracle Data Mining EM使用凝聚层次聚类。使用Bhattacharyya距离函数测量组分分布重叠。在层次结构中选择适当的截止级别会自动确定高级别集群的数量。

EM的Oracle Data Mining实现生成了模型组件到高级集群的分配。平均值，方差，模式，直方图和规则等统计数据还描述了高级别集群。该算法可以配置为在组件级别或集群级别生成集群分配。

13.3配置算法

配置期望最大化（EM）。

在Oracle数据挖掘中，期望最大化（EM）可以有效地建模非常大的数据集（行和列），而无需用户提供初始化参数或指定模型组件的数量。虽然该算法提供了合理的默认值，但它也提供了灵活性。

以下列表描述了EM的一些可配置方面：

  • 模型中是否包含独立的非嵌套列属性。该选项默认由系统确定。

  • 是否将伯努利或高斯分布用于数值属性。默认情况下，算法选择最合适的分布，并且各个属性可以使用不同的分布。当分布是用户指定的时，它用于所有数字属性。

  • 收敛标准是基于保留数据集还是基于贝叶斯信息准则（BIC）。收敛标准默认由系统确定。

  • 将新组件添加到模型所需的对数似然函数值的百分比改进。默认百分比为0.001。

  • 是否将群集定义为单个组件或组件组。默认情况下，群集与组件组相关联。

  • 模型中的最大组件数。如果启用模型搜索，则算法基于似然函数的改进或基于正则化（BIC）确定组件的数量，直到指定的最大值。

  • 凝聚聚类步骤的连锁功能是否使用分支内的最近距离（单连锁），分支内的平均距离（平均连锁），或分支内的最大距离（完全连锁）。默认情况下，算法使用单链接。

选择适当的初始参数值会对解决方案的质量产生重大影响。期望最大化（EM）不能保证收敛到似然函数的全局最大值，而是可以收敛到局部最大值。因此，不同的初始参数值可导致不同的模型参数和不同的模型质量。

在模型搜索过程中，EM模型独立生长。随着新组件的添加，其参数将初始化为分布不合适的区域。

14.2.5从组件到集群

期望最大化（EM）模型组件通常被视为集群。但是，这种方法可能会产生误导。内聚集群通常由多个组件建模。EM中使用的概率密度函数的形状有效地预先确定了所识别的聚类的形状。例如，高斯密度函数可以识别单峰对称聚类。更复杂形状的集群需要由多个组件建模。

理想情况下，任意形状的高密度区域必须被解释为单个簇。为实现此目的，EM的Oracle数据挖掘实现构建了一个基于各个组件分布重叠的组件层次结构。Oracle Data Mining EM使用凝聚层次聚类。使用Bhattacharyya距离函数测量组分分布重叠。在层次结构中选择适当的截止级别会自动确定高级别集群的数量。

EM的Oracle Data Mining实现生成了模型组件到高级集群的分配。平均值，方差，模式，直方图和规则等统计数据还描述了高级别集群。该算法可以配置为在组件级别或集群级别生成集群分配。

14.3配置算法

配置期望最大化（EM）。

在Oracle数据挖掘中，期望最大化（EM）可以有效地建模非常大的数据集（行和列），而无需用户提供初始化参数或指定模型组件的数量。虽然该算法提供了合理的默认值，但它也提供了灵活性。

以下列表描述了EM的一些可配置方面：

  • 模型中是否包含独立的非嵌套列属性。该选项默认由系统确定。

  • 是否将伯努利或高斯分布用于数值属性。默认情况下，算法选择最合适的分布，并且各个属性可以使用不同的分布。当分布是用户指定的时，它用于所有数字属性。

  • 收敛标准是基于保留数据集还是基于贝叶斯信息准则（BIC）。收敛标准默认由系统确定。

  • 将新组件添加到模型所需的对数似然函数值的百分比改进。默认百分比为0.001。

  • 是否将群集定义为单个组件或组件组。默认情况下，群集与组件组相关联。

  • 模型中的最大组件数。如果启用模型搜索，则算法基于似然函数的改进或基于正则化（BIC）确定组件的数量，直到指定的最大值。

  • 凝聚聚类步骤的连锁功能是否使用分支内的最近距离（单连锁），分支内的平均距离（平均连锁），或分支内的最大距离（完全连锁）。默认情况下，算法使用单链接。

目的

PREDICTION_DETAILS返回选择中每行的预测详细信息。返回值是一个XML字符串，用于描述预测的属性。

对于回归，返回的详细信息指的是预测的目标值。对于分类和异常检测，返回的详细信息指的是最高概率类或指定的class_value 。

TOPN

如果为topN指定值，则该函数返回对预测（分数）影响最大的N属性。如果未指定topN ，则该函数将返回5个最有影响力的属性。

DESC，ASC或ABS

返回的属性按重量排序。属性的权重表示其对预测的正面或负面影响。对于回归，正权重表示更高的值预测;负权重表示较低值预测。对于分类和异常检测，正权重表示概率预测较高;负权重表示概率预测较低。

默认情况下， PREDICTION_DETAILS返回具有最高正权重（ DESC ）的属性。如果指定ASC ，则返回负权重最高的属性。如果指定ABS ，则返回权重最大的属性，无论是负数还是正数。结果按绝对值从最高到最低排序。输出中不包含权重为零的属性。

语法选择

PREDICTION_DETAILS可以通过以下两种方式之一对数据进行评分：它可以将挖掘模型对象应用于数据，也可以通过执行构建和应用一个或多个瞬态挖掘模型的分析子句来动态挖掘数据。选择语法或分析语法 ：

  • 语法 - 使用第一种语法使用预定义的模型对数据进行评分。提供执行分类，回归或异常检测的模型的名称。

  • 分析语法 - 使用分析语法在没有预定义模型的情况下对数据进行评分。分析语法使用mining_analytic_clause ，它指定是否应为多个模型构建分区数据。mining_analytic_clause支持query_partition_clause和order_by_clause 。（参见“ analytic_clause :: = ” 。）

    • 对于分类，请指定FOR expr ，其中expr是一个表达式，用于标识具有字符数据类型的目标列。

    • 对于回归，请指定FOR expr ，其中expr是一个表达式，用于标识具有数字数据类型的目标列。

    • 对于异常检测，请指定关键字OF ANOMALY 。

在对分区模型进行评分时， PREDICTION_DETAILS函数的语法可以使用可选的GROUPING提示。请参阅GROUPING提示 。

在Oracle Database 11g第2版（11.2）之前，配置文件仅限于驻留在与最初创建文件时具有相同扇区大小（物理块大小）的磁盘上。这不是问题，因为在给定的代理配置中通常使用单个扇区大小。由于预期在给定代理配置中某处具有混合扇区大小，因此代理配置文件现在对基础扇区大小完全不敏感，只要扇区大小为4KB或更小。

在下面的步骤5中的升级处理期间，将配置文件的转换发生为对基础扇区大小不敏感。

以下步骤使用Oracle Database 12 c第2版（12.2），但您可以替换Oracle Database 12 c第1版（12.1）。

    DGMGRL>禁用FAST_START FAILOVER;

  2. 关闭10.n或11.n Oracle Data Guard代理，无论您运行的是哪个版本。

    1. 发出以下DGMGRL命令以禁用代理在Oracle Data Guard配置中对数据库的主动管理：

      DGMGRL>禁用配置;

    2. 发出以下SQL * Plus语句以停止代理：

      SQL> ALTER SYSTEM SET DG_BROKER_START = FALSE;

  3. 制作当前代理配置文件的副本，如以下初始化参数所示： DG_BROKER_CONFIG_FILE1和DG_BROKER_CONFIG_FILE2 。

  4. 将Oracle数据库软件升级到Oracle Database 12 c第2版（12.2）。有关逐步升级说明，请参阅适用于您的操作系统的Oracle数据库安装文档。

    还必须升级DGMGRL命令行界面，以便管理和监视在Oracle Database 12 c第2版（12.2）上运行的代理配置。在Oracle Database 12 c Release 2（12.2）上运行的DGMGRL不能用于管理在Oracle Database 10g或Oracle Database 11g上运行的Oracle Data Guard。

    注意：

    Oracle Database 12 c第2版（12.2）中提供的DGMGRL命令行界面支持Oracle Database 10g和Oracle Database 11g的现有DGMGRL命令行脚本。

    Oracle数据库11克 DGMGRL命令行脚本不能保证由Oracle数据库10g支撑。

  5. 升级到Oracle Database 12 c第2版（12.2）后，启动代理。例如：

    1. 发出以下SQL * Plus语句以启动代理：

      SQL> ALTER SYSTEM SET DG_BROKER_START = TRUE;

    2. 发出以下DGMGRL命令以启用代理对Oracle Data Guard配置中的数据库的主动管理：

      DGMGRL> ENABLE CONFIGURATION;

    代理首次在Oracle Database 12 c Release 2（12.2）上启动时，它会检测到Oracle Database 10g和Oracle Database 11 g代理配置文件的存在。它会自动升级它们以包含Oracle Database 12 c第2版（12.2）中引入的所有新属性。此自动转换是透明的，永久性的，并且只发生一次。

    注意：

    从步骤5中发生的这种自动升级开始，配置文件此后可以驻留在具有任何支持的磁盘扇区大小（物理块大小）直到并包括4KB扇区的磁盘上。

注意：

264.1 UTL_RECOMP概述

在主要版本升级（通常使所有PL / SQL和Java对象无效）之后，此脚本特别有用。尽管在使用时会自动重新编译无效对象，但在操作之前运行此脚本非常有用，因为这将消除或最小化由于运行时按需自动重新编译而导致的后续延迟。

并行重新编译可以利用多个CPU来减少重新编译无效对象所需的时间。并行度由RECOMP_PARALLEL过程的第一个参数指定。

通常，每个可用CPU的一个线程的并行设置提供了良好的初始设置。但请注意，重新编译无效对象的过程会将大量数据写入系统表，并且相当I / O密集。慢速磁盘系统可能是一个重要的瓶颈，并且可以从更高程度的并行性限制加速。

264.2 UTL_RECOMP操作说明

UTL_RECOMP有几个操作说明。

  • 此程序包使用作业队列进行并行重新编译。

  • 必须使用SQL*PLUS运行此程序包。

  • 您必须连接AS SYSDBA才能运行此脚本。

  • 此程序包需要使用VALID状态创建以下程序包：

    • STANDARD （ standard sql ）

    • DBMS_STANDARD （ dbmsstdx 。 sql ）

    • DBMS_JOB （ dbmsjob 。 sql ）

    • DBMS_RANDOM（dbmsrand.sql）

  • 在此程序包中运行条目时，数据库上不应有其他DDL。不遵循此建议可能会导致死锁。

264.3 UTL_RECOMP示例

这些示例显示了UTL_RECOMP可以重新编译对象的各种方法。

  • 按顺序重新编译所有对象：

    执行UTL_RECOMP.RECOMP_SERIAL（）;

  • 按顺序重新编译模式SCOTT对象：

    执行UTL_RECOMP.RECOMP_SERIAL（'SCOTT'）;

  • 使用4个并行线程重新编译所有对象：

    EXECUTE UTL_RECOMP.RECOMP_PARALLEL（4）;

  • 使用参数JOB_QUEUE_PROCESSES指定的线程数重新编译模式JOE中的对象：

    EXECUTE UTL_RECOMP.RECOMP_PARALLEL（NULL，'JOE'）;
