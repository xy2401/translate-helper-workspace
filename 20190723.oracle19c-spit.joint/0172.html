23.1关于SQL调整集

SQL调优集（STS）是一个数据库对象，可用作调优工具的输入。

数据库将SQL调优集存储在数据库提供的模式中。STS包括：

  • 一组SQL语句

  • 关联的执行上下文，例如用户架构，应用程序模块名称和操作，绑定值列表以及用于SQL编译游标的环境

  • 关联的基本执行统计信息，例如已用时间，CPU时间，缓冲区获取，磁盘读取，已处理的行，游标提取，执行次数，完成执行次数，优化程序成本和命令类型

  • 每个SQL语句的关联执行计划和行源统计信息（可选）

注意：

将STS与可插拔数据库一起使用时，数据可见性和权限要求可能会有所不同。有关可管理性功能如何在容器数据库（CDB）中工作的表，请参见“ Oracle数据库管理员指南” 。

23.1.1 SQL调优集的用途

通过STS，您可以在单个数据库对象中对SQL语句和相关元数据进行分组，您可以使用它来满足调整目标。

具体来说，SQL调优集实现了以下目标：

  • 为性能调优顾问提供输入

    您可以使用STS作为多个数据库顾问程序的输入，包括SQL Tuning Advisor，SQL Access Advisor和SQL Performance Analyzer。

  • 在数据库之间传输SQL

    您可以将SQL调优集从一个数据库导出到另一个数据库，从而在数据库之间传输SQL工作负载，以进行远程性能诊断和调优。当在生产数据库上发生次优执行SQL语句时，开发人员可能不希望直接在生产数据库上进行调查和调优。DBA可以将有问题的SQL语句传输到测试数据库，开发人员可以在其中安全地分析和调整它们。

23.3加载SQL调优集

要使用SQL语句加载STS，请使用DBMS_SQLTUNE或DBMS_SQLSET包中的LOAD_SQLSET过程。

填充STS的标准源是AWR，另一个STS或共享SQL区域。对于工作负载存储库和SQL调优集，预定义的表函数可以从源中选择列以填充新的STS。

下表描述了一些DBMS_SQLTUNE.LOAD_SQLSET过程参数。

表23-3 DBMS_SQLTUNE.LOAD_SQLSET参数

参数 描述

populate_cursor

指定从中填充STS的游标引用。

load_option

指定语句如何加载到STS中。可能的值为INSERT （默认值）， UPDATE和MERGE 。

DBMS_SQLTUNE.SELECT_CURSOR_CACHE函数根据指定的过滤器从共享SQL区域收集SQL语句。此函数返回每个数据源中找到的每个SQL ID或PLAN_HASH_VALUE对的一个SQLSET_ROW 。

使用DBMS_SQLTUNE.CAPTURE_CURSOR_CACHE_SQLSET函数（或等效的DBMS_SQLSET.CAPTURE_CURSOR_CACHE ）在指定的时间间隔内重复轮询共享的SQL区域。此函数比重复调用SELECT_CURSOR_CACHE和LOAD_SQLSET过程更有效。此功能有效地捕获整个工作负载，而AWR仅捕获高负载SQL语句的工作负载，或LOAD_SQLSET过程， LOAD_SQLSET访问数据源一次。

先决条件

本教程具有以下先决条件：

  • 提供给SELECT_CURSOR_CACHE函数的过滤器将作为当前用户运行的SQL语句的一部分进行评估。因此，它们使用该用户的安全权限执行，并且可以包含用户可以访问的任何构造和子查询，但不能包含。

  • 当前用户必须具有共享SQL区域视图的权限。

假设

本教程假设以下内容：

  • 您希望使用共享SQL区域中的语句加载名为SQLT_WKLD_STS的SQL调优集。

  • 您希望使用DBMS_SQLTUNE而不是DBMS_SQLSET来加载STS。

要加载STS：

  1. 在SQL * Plus或SQL Developer中，以具有必要权限的用户身份登录数据库。

  2. 运行DBMS_SQLTUNE.LOAD_SQLSET过程。

    例如，执行以下PL / SQL程序以使用属于sh模式的所有游标缓存语句填充SQL调优集：

    DECLARE c_sqlarea_cursor DBMS_SQLTUNE.SQLSET_CURSOR; BEGIN OPEN c_sqlarea_cursor FOR SELECT VALUE（p）FROM TABLE（DBMS_SQLTUNE.SELECT_CURSOR_CACHE（'module =''SQLT_WKLD''and parsing_schema_name =''SH'''））p; - 加载调优集DBMS_SQLTUNE.LOAD_SQLSET（sqlset_name =>'SQLT_WKLD_STS'，populate_cursor => c_sqlarea_cursor）;结束; /

23.5修改SQL调整集

使用DBMS_SQLTUNE或DBMS_SQLSET的DELETE_SQLSET过程从STS中删除SQL语句。

您可以使用UPDATE_SQLSET过程更新由STS名称和SQL ID标识的现有STS中的SQL语句（例如PRIORITY或OTHER ）的属性。

假设

本教程假定您要修改SQLT_WKLD_STS ，如下所示：

  • 您希望删除fetch count超过100的所有SQL语句。

  • 您希望将ID为fudq5z56g642p的SQL语句的优先级更改为1 。运行SQL Tuning Advisor时，可以使用优先级作为排名标准。

  • 您使用DBMS_SQLTUNE而不是DBMS_SQLSET 。

修改STS的内容：

  1. 在SQL * Plus或SQL Developer中，以具有必要权限的用户身份登录数据库。

  2. （可选）使用TABLE函数查询STS内容。

    例如，执行以下查询：

    从表中选择SQL_ID，ELAPSED_TIME，FETCHES，EXECUTIONS（DBMS_SQLTUNE.SELECT_SQLSET（'SQLT_WKLD_STS'））;

    示例输出显示如下：

    SQL_ID ELAPSED_TIME FETCHES EXECUTIONS ------------- ------------ ---------- ---------- 2cqsw036j5u7r 3407459 2 1 79f8shn041a1f 9453965 61258 1 bzmnj0nbvmz8t 401869 1 1 fudq5z56g642p 5300264 1 1

  3. 根据用户指定的条件删除SQL语句。

    使用basic_filter谓词从SQLSET_ROW属性上定义的STS中过滤SQL。以下示例删除STS中的所有语句，提取计数超过100：

    BEGIN DBMS_SQLTUNE.DELETE_SQLSET（sqlset_name =>'SQLT_WKLD_STS'，basic_filter =>'fetches> 100'）;结束; /

  4. 设置SQL语句的属性值。

    以下示例将语句2cqsw036j5u7r的优先级设置为1 ：

    BEGIN DBMS_SQLTUNE.UPDATE_SQLSET（sqlset_name =>'SQLT_WKLD_STS'，sql_id =>'2cqsw036j5u7r'，attribute_name =>'PRIORITY'，attribute_value => 1）;结束; /

  5. （可选）查询STS以确认已进行了预期的修改。

    例如，执行以下查询：

    SELECT SQL_ID，ELAPSED_TIME，FETCHES，EXECUTIONS，PRIORITY FROM TABLE（DBMS_SQLTUNE.SELECT_SQLSET（'SQLT_WKLD_STS'））;

    示例输出显示如下：

    SQL_ID ELAPSED_TIME FETCHES EXECUTIONS PRIORITY ------------- ------------ ---------- ---------- ---------- 2cqsw036j5u7r 3407459 2 1 1 bzmnj0nbvmz8t 401869 1 1 fudq5z56g642p 5300264 1 1

23.6.1.2当CON_DBID值不同时传输SQL调整集的基本步骤

传输STS时，必须在源数据库和目标数据库的con_dbid不同时重新映射STS中每个SQL语句的con_dbid 。

导致con_dbid值不同的情况包括：

  • 已重新启动其实例的单实例数据库

  • Oracle RAC数据库的不同实例

  • 不同的PDB

  • 非CDB和CDB

重新映射的基本步骤如下：

  1. 使用DBMS_SQLTUNE.PACK_STGTAB_SQLSET或DBMS_SQLSET.PACK_STGTAB将STS打包到临时表中。

  2. 使用DBMS_SQLTUNE.REMAP_STGTAB_SQLSET或DBMS_SQLSET.REMAP_STGTAB con_dbid映射登台表中的每个con_dbid 。

  3. 导出STS。

  4. 在目的地CDB中解压缩STS。

示例23-1将STS从非CDB传输到CDB时重新映射CON_DBID

在此示例中，您打算将名为STS_for_transport的STS从非CDB传输到CDB。在源非CDB上，您已使用DBMS_SQLTUNE.PACK_STGTAB_SQLSET过程将STS打包到源临时表src_stg_tbl 。目标CDB的容器ID是12345 。

在源非CDB中，执行以下命令：

VARIABLE con_dbid_src NUMBER; EXEC SELECT UNIQUE con_dbid INTO：con_dbid_src FROM src_stg_tbl; BEGIN DBMS_SQLTUNE.REMAP_STGTAB_SQLSET（staging_table_name =>'src_stg_tbl'，staging_schema_owner =>'dba1'，old_sqlset_name =>'STS_for_transport'，old_con_dbid =>：con_dbid_src，new_con_dbid => 12345）;结束;

您现在可以导出登台表的内容，然后继续使用正常的传输过程。

B.2 oclumon dumpnodeview

使用oclumon dumpnodeview命令以节点视图的形式查看系统监视器服务的日志信息。

使用说明

节点视图是Cluster Health Monitor在某个时间点为节点收集的所有度量标准的集合。Cluster Health Monitor尝试在每个节点上每五秒收集一次度量标准。某些指标是静态的，而其他指标是动态的。

显示详细输出时，节点视图由八个视图组成：

  • SYSTEM ：列出系统指标，例如CPU COUNT，CPU USAGE和MEM USAGE

  • 热门消费者 ：按以下格式列出最常用的流程：

    metric_name ：' process_name （ process_identifier ） utilization '

  • CPUS ：列出每个CPU的统计信息

  • 进程 ：列出进程指标，例如PID，名称，线程数，内存使用情况和文件描述符数

  • 设备 ：列出设备指标，例如磁盘读取和写入速率，队列长度以及每个I / O的等待时间

  • NICS ：列出网络接口卡指标，例如网络接收和发送速率，有效带宽和错误率

  • FILESYSTEMS ：列出文件系统指标，例如总空间，已用空间和可用空间

  • 协议错误 ：列出任何协议错误

生成仅包含SYSTEM和TOP CONSUMERS视图的摘要报告。

句法

oclumon dumpnodeview [-allnodes | -n node1 ...] [-last duration | -s timestamp -e timestamp ] [-i interval ] [-v | [-system] [ -  process] [ -  procag] [ -  device] [ -  filesystem] [ -  nic] [ -  protoerr] [ -  cpu] [ -  topconsumer]] [-format format type ] [-dir directory [-append] ]]

13.11将Java DOM API和JDBC与二进制XML一起使用

您可以使用Java DOM API进行XML和Java数据库连接（JDBC）来读取或写入从Oracle XML DB编码为二进制XML的XML数据。这样做涉及通常的读写程序。

二进制XML是XML Schema感知的，可以使用各种编码方案，具体取决于您的需求和数据。因此，为了操作二进制XML数据，您必须同时拥有关于相关XML模式和编码的数据和元数据。

对于存储在数据库中的XMLType数据，此元数据也存储在数据库中。但是，根据数据库和数据的设置方式，元数据可能与应用的数据不在同一服务器上。如果是这种情况，那么，在您从数据库读取或写入二进制XML数据之前，必须执行以下步骤：

  1. 为元数据创建上下文实例。

  2. 将此上下文与用于访问数据库中的二进制XML数据的数据连接相关联。数据连接可以是专用连接或连接池。您在类java.sql.使用getDedicatedConn()和getConnPool()方法java.sql.Connection分别获取这两种连接的句柄。

  1. 在类java.sql.创建XML数据连接对象java.sql.Connection 。

  2. 创建一个或多个元数据的上下文中，根据需要，使用方法BinXMLMetadataProviderFactory.createDBMetadataProvider()中包oracle.xml.binxml 。元数据上下文有时被称为元数据存储库。您可以从专用连接或连接池创建元数据上下文。

  3. 将元数据上下文与二进制XML数据连接相关联。使用方法DBBinXMLMetadataProvider.associateDataConnection()封装oracle.xml.binxml做到这一点。

  4. （可选）如果XML数据源自数据库之外，请使用方法oracle.xdb.XMLType.setFormatPref()指定要以二进制XML格式编码要发送到数据库的XML数据。这适用于DOM文档（类oracle.xdb.XMLType ）。如果未指定二进制XML，则数据将作为文本发送到数据库。

  5. 使用常用的Java方法从数据库读取和写入XML数据。只要需要对二进制XML文档进行编码或解码，就会使用元数据上下文自动获取必要的元数据。

  • 如果Oracle为pragma restrict_references返回错误“ORA：452,0，'子程序'％s'违反其关联的pragma'，则可能意味着违规是由于：

    • 当前程序或功能的问题

    • 在没有编译指示的情况下调用过程或函数，或者调用具有较少限制性编译指示的过程或函数

表142-13 ROWID_TO_EXTENDED函数参数

返回值

ROWID_TO_EXTENDED以扩展字符格式返回ROWID 。如果输入ROWID为NULL ，则该函数返回NULL 。如果提供了零值ROWID （00000000.0000.0000），则返回零值受限ROWID 。

例子

假定有一个称为表RIDS在架构SCOTT ，并且该表包含列ROWID_COL保持ROWIDs （限制），及列TABLE_COL指向在其他表SCOTT架构。您可以使用以下语句将ROWIDs转换为扩展格式：

UPDATE SCOTT.RIDS SET rowid_col = dbms_rowid.rowid_to_extended（rowid_col，'SCOTT“，TABLE_COL，0）;

使用说明

  • 如果模式和对象名称作为IN参数提供，则此函数将验证名为的表上的SELECT权限，并使用表的数据对象编号ROWID提供的受限ROWID转换为扩展的ROWID 。但是， ROWID_TO_EXTENDED返回一个值并不能保证转换后的ROWID实际引用表中的有效行，无论是在调用函数时，还是在实际使用扩展的ROWID时。

  • 如果未提供架构和对象名称（传递为NULL ），则此函数尝试获取由提供的受限ROWID指定的页面。它将存储在此ROWID中的文件编号视为绝对文件编号。如果文件已被删除，这可能会导致问题，并且在迁移之前已重复使用其编号。如果获取的页面属于有效表，则此表的数据对象编号用于转换为扩展的ROWID值。这是非常低效的，并且Oracle建议仅在目标表未知时作为最后的手段执行此操作。用户在使用转换后的值时仍必须知道正确的表名。

  • 如果提供了扩展的ROWID值，则根据从表名参数计算的数据对象编号验证输入扩展ROWID中的数据对象编号。如果两个数字不匹配，则引发INVALID_ROWID异常。如果它们匹配，则返回输入ROWID 。

  • ROWID_TO_EXTENDED不能与分区表一起使用。

DBMS_SYNC_REFRESH包提供了一个接口，用于执行物化视图的同步刷新。

167.1 DBMS_SYNC_REFRESH概述

同步刷新是Oracle Database Release 12 c中引入的一种刷新方法，它使您能够保持一组表和在其上定义的物化视图始终保持同步。

也可以看看：

有关使用同步刷新的详细信息，请参阅“Oracle数据库数据仓库指南”

167.2 DBMS_SYNC_REFRESH安全模型

此包的执行权限授予PUBLIC ，因此所有用户都可以执行此包中的过程以对其拥有的对象执行同步刷新。数据库管理员可以对数据库中的所有表和物化视图执行同步刷新操作。

通常，如果没有DBA权限的用户想要在另一个用户的表上使用同步刷新，他必须完成读取和写入该表的权限，用户必须具有SELECT或READ ， INSERT ， UPDATE和DELETE该表或物化视图的权限。有几个例外：

  • PURGE_REFRESH_STATS和ALTER_REFRESH_STATS_RETENTION函数

    这两个函数实现清除策略，可用于更改默认保留期。这些功能只能由数据库管理员执行。

  • CAN_SYNCREF_TABLE函数

    这是一个咨询功能，用于检查与指定表关联的所有物化视图的同步刷新资格。因此，此函数需要对与指定表关联的所有READ化视图具有SELECT或READ权限。

167.3.1 ABORT_REFRESH程序

此过程撤消PREPARE_REFRESH或EXECUTE_REFRESH对指定的同步刷新组所做的所有更改。它可以帮助您恢复到表和物化视图可用的状态，并在遇到意外错误时保持一致。

此过程已超载。

句法

DBMS_SYNC_REFRESH.ABORT_REFRESH（group_id IN NUMBER）; DBMS_SYNC_REFRESH.ABORT_REFRESH（group_id_list IN DBMS_UTILITY.NUMBER_ARRAY）;

参数

表167-2 ABORT_REFRESH过程参数

参数 描述

GROUP_ID

同步刷新组的组ID。

group_id_list

要中止同步刷新的同步刷新组的组ID数组。

使用说明

如果在PREPARE_REFRESH之后PREPARE_REFRESH ，则此过程将删除PREPARE_REFRESH创建的外部表，并解锁同步刷新组中的表和PREPARE_REFRESH化视图。

如果在EXECUTE_REFRESH失败后调用，则此过程通过撤消已成功完成的任何分区交换来将表的状态恢复到EXECUTE_REFRESH之前。

此过程释放放置在同步刷新组中的表上的锁，这些锁由PREPARE_REFRESH过程放置在它们上。有关这些锁的说明，请参见“ PREPARE_REFRESH过程 ” 。

仅当PREPARE_REFRESH或EXECUTE_REFRESH语句失败时， ABORT_REFRESH才会起作用。成功运行这些命令后无法使用它，并在这种情况下抛出错误。

167.3.2 ALTER_REFRESH_STATS_RETENTION过程

此过程会更改以天为单位指定的刷新历史记录保留值。它旨在与PURGE_REFRESH_HISTORY结合使用。除了执行它的权限之外，它还需要SYSDBA权限。

句法

DBMS_SYNC_REFRESH.ALTER_REFRESH_STATS_RETENTION（保留在数字中）;

参数

表167-3 ALTER_REFRESH_STATS_RETENTION过程参数

参数 描述

保留

保留时间（天）。刷新历史记录将至少保留这么多天。有效范围是1到365,000。

您可以将以下值用于特殊目的：

  • -1 - PREPARE_REFRESH永远不会清除刷新历史记录。

  • 0 - 永远不会保存旧刷新历史记录。PREPARE_REFRESH将删除所有刷新历史记录。

  • NULL - 将刷新历史记录保留更改为默认值。

167.3.3 CAN_SYNCREF_TABLE过程

此过程建议表及其相关的物化视图是否符合同步刷新的条件。它提供了对其分析的解释。如果不符合条件，您可以检查原因并尽可能采取适当的措施。

此过程列出了表的所有相关物化视图以及它们是否有资格进行同步刷新。请注意，实体化视图可能有资格进行同步刷新，即使基表可能不符合。

Oracle数据库数据仓库指南中详细讨论了用于同步刷新的物化视图的资格规则。

您可以通过两种方式调用CAN_SYNCREF_TABLE 。第一种是使用表，而第二种是创建VARRAY 。

句法

DBMS_SYNC_REFRESH.CAN_SYNCREF_TABLE（schema_name IN VARCHAR2，table_name IN VARCHAR2，statement_id IN VARCHAR2）; DBMS_SYNC_REFRESH.CAN_SYNCREF_TABLE（schema_name IN VARCHAR2，table_name IN VARCHAR2，output_array IN OUT Sys。CanSyncRefTypeArray）;

请注意，只有一个statement_id或output_array需要被提供给CAN_SYNCREF_TABLE 。

参数

表167-4 CAN_SYNCREF_TABLE过程参数

参数 描述

SCHEMA_NAME

基表的架构名称。

TABLE_NAME

基表的名称。

statement_id

一个字符串（ VARCHAR2(30) ），用于在输出定向到用户模式中名为SYNCREF_TABLE的表时，标识与调用CAN_SYNCREF_TABLE相关的行。

output_array

CAN_SYNCREF_TABLE在其中记录有关基表及其相关物化视图的同步刷新资格的信息的输出数组。

使用SYNCREF_TABLE

CAN_SYNCREF_TABLE的输出可以定向到名为SYNCREF_TABLE的表。用户负责创建SYNCREF_TABLE ;它可以在不再需要时丢弃。其结构如下：

CREATE TABLE SYNCREF_TABLE（statement_id VARCHAR2（30），schema_name VARCHAR2（30），table_name VARCHAR2（30），mv_schema_name VARCHAR2（30），mv_name VARCHAR2（30），符合条件的VARCHAR2（1），seq_num NUMBER，msg_number NUMBER，消息VARCHAR2（4000） ））;

使用VARRAY

您可以在PL / SQL VARRAY保存CAN_SYNCREF_TABLE的输出。此数组的元素是CanSyncRefMessage类型，它是在SYS模式中预定义的，如下所示：

TYPE CanSyncRefMessage IS OBJECT（schema_name VARCHAR2（30），table_name VARCHAR2（30），mv_schema_name VARCHAR2（30），mv_name VARCHAR2（30），符合条件的VARCHAR2（1），seq_num NUMBER，msg_number NUMBER，消息VARCHAR2（4000））;

数组类型CanSyncRefArrayType是CanSyncRefMessage对象的CanSyncRefMessage ，在SYS模式中预定义，如下所示：

TYPE CanSyncRefArrayType AS VARRAY（256）OF CanSyncRefMessage;

每个CanSyncRefMessage记录都提供有关基表资格的消息或同步刷新的从属物化视图。字段的语义与SYNCREF_TABLE相应字段的语义相同。但是， SYNCREF_TABLE具有一个statement_id字段，该字段在CanSyncRefMessage不存在，因为当使用VARRAY参数调用CAN_SYNCREF_TABLE时，不提供statement_id （因为它不是必需的）。

使用说明

此过程计划同步刷新操作的三个阶段，并执行与准备阶段本身相关的步骤。这些步骤包括识别事实表的分区和已更改的物化视图，以及根据更改计算其新值。分区的新值存储在称为外部表的表中，这些表在EXECUTE_REFRESH时交换到其对应的分区中。

在运行此过程之前，用户必须对组中的所有表运行PREPARE_STAGING_LOG 。即使对没有更改的日志记录，这也是必需的。用户还必须使用REGISTER_PARTITION_OPERATION在组中的表上注册任何分区操作。

此过程的一个副作用是正在准备的表被锁定在这个意义上：表的登台日志将被锁定以防止发生任何DML并且将禁用分区操作的注册。在您发出EXECUTE_REFRESH语句之前，这些锁定将一直有效。或者，您可以发出ABORT_REFRESH操作来释放这些锁。此过程的另一个副作用是从早期同步刷新操作的目录记录中清除;如果它们比保留期更早，则将其清除。

准备刷新作业的并行度继承自可以使用ALTER SESSION语句控制的会话参数。

可以使用GET_GROUP_ID(table_name)找到表的组ID。可以使用GET_GROUP_ID_LIST(table_name_list)找到表列表的组ID。可以使用GET_ALL_GROUP_IDS检索所有表列表的组ID。

默认情况下，同步刷新不会维护属于同步刷新组中的表和物化视图的全局索引。如果您希望这样做，可以在执行PREPARE_REFRESH之前设置事件31904，级别64。这将导致PREPARE_REFRESH生成的分区交换DDL语句附加UPDATE INDEXES子句，当它们由EXECUTE_REFRESH执行时，将保留全局索引。

167.3.9 PREPARE_STAGING_LOG过程

此过程收集基表的分段日志中的数据统计信息，并验证日志中的数据。

它可以在几种不同的模式下运行，从执行严格检查数据的强制模式到不进行检查的可信模式。您应该在加载登台日志之后和运行PREPARE_REFRESH之前运行此过程。

在强制模式（默认情况下）下，此过程将填充要删除或更新的行的列的缺失值。如果发现任何违反登台日志规则的行为，则会引发错误。您可以查询视图USER_SR_STLOG_EXCEPTIONS以获取有关异常的详细信息。

“ Oracle数据库数据仓库指南”中介绍了登台日志密钥的概念。

在强制模式下，此过程处理登台日志中的每个删除/更新行，如下所示：

  • 它使用密钥验证基表中是否存在行。

  • 对于要删除的行（ DMLTYPE$$为'D' ），它会验证基表中是否存在具有此键的行;如果在暂存日志中提供了非空的非键值，则验证这些值是否与基表中的相应列匹配;否则在例外表中记录一个异常。如果缺少任何非键列的值，则会从基表中的行填充这些值。

  • 对于要更新的行（ DMLTYPE$$是'UO'或'UN' ），它会验证基表中是否存在具有此键的行。在旧值行（ DMLTYPE$$是'UO' ）中，它进行相同的检查并执行与删除行相同的处理。在新值行（ DMLTYPE$$是'UN' ）中，它检查至少一列的值与其旧值不同;否则记录异常。

  • 在新值行（ DMLTYPE$$是'UN' ）中，列中的空值被解释为具有与列的旧值相同的值，除非旧值为非null且新值为null在这种情况下，列的新值被解释为null。这要求用户必须提供正在更新为NULL的旧值。

在默认强制模式下，此过程将验证是否为删除或更新操作最多指定了一次键。这意味着用户在进行更改合并时，必须将同一行的delete-insert合并到行'UO'和'UN ;'的更新操作中。必须将多个更新合并为单个更新;并且空行更改（例如同一行的insert-update-delete）不得出现在登台日志中。

在强制模式下完成的检查可能非常耗时。如果您对数据的完整性有信心，则可以选择较低级别的检查。您可以选择：

  • 通过选择psl_mode的DBMS_SYNC_REFRESH.INSERT_TRUSTED信任所有插入行（ DMLTYPE$$是'I')

  • 通过选择psl_mode的DBMS_SYNC_REFRESH.DELETE_TRUSTED信任所有删除行（ DMLTYPE$$为'D' ）

  • 通过选择psl_mode的DBMS_SYNC_REFRESH.UPDATE_TRUSTED信任所有更新行（ DMLTYPE$$是'UO '或'UN' ）

  • 通过选择psl_mode的DBMS_SYNC_REFRESH.TRUSTED信任所有三种类型的DML。

此外，您可以将psl_mode指定为上述标志的位掩码。例如， DBMS_SYNC_REFRESH.INSERT_TRUSTED + DBMS_SYNC_REFRESH_DELETE_TRUSTED会将插入和删除视为可信但不会更新。

12.1使用自动共享内存管理

自动共享内存管理通过在SGA中为以下内存池自动分配内存来简化SGA的配置：

  • 数据库缓冲区缓存（默认池）

  • 共享池

  • 大型游泳池

  • Java池

  • 溪流池

自动共享内存管理由SGA_TARGET参数控制。SGA_TARGET参数值的更改会自动调整这些内存池的大小。如果这些内存池设置为非零值，则自动共享内存管理将这些值用作最低级别。Oracle建议您根据应用程序组件正常运行所需的最小内存量来设置最小值。

以下内存缓存是手动调整大小的组件，不受自动共享内存管理控制：

  • 重做日志缓冲区

    使用LOG_BUFFER初始化参数调整重做日志缓冲区的大小，如“ 配置重做日志缓冲区 ”中所述 。

  • 其他缓冲区缓存（例如KEEP ， RECYCLE和其他非默认块大小）

    使用DB_KEEP_CACHE_SIZE初始化参数调整KEEP池的大小，如“ 配置KEEP池 ”中所述 。

    使用DB_RECYCLE_CACHE_SIZE初始化参数调整RECYCLE池的大小，如“ 配置RECYCLE池 ”中所述 。

  • 修复了SGA和其他内部分配

    固定SGA和其他内部分配使用DB_nK_CACHE_SIZE初始化参数调整大小。

当自动共享内存管理计算自动调整的内存池的值时，分配给这些内存缓存的内存将从SGA_TARGET参数的值中扣除。

12.1.2.1启用自动共享内存管理

要启用自动共享内存管理，请设置以下初始化参数：

  • STATISTICS_LEVEL为TYPICAL或ALL

  • SGA_TARGET为非零值

    SGA_TARGET参数可以设置为小于或等于SGA_MAX_SIZE初始化参数值的值。将SGA_TARGET参数的值设置为您要专门用于SGA的内存量。

12.1.2.2禁用自动共享内存管理

要禁用自动共享内存管理，请在实例启动时将SGA_TARGET参数的值动态设置为0。

这将禁用自动共享内存管理，并且将为每个内存池使用当前自动调整的大小。如有必要，您可以手动调整每个内存池的大小，如“ 手动调整SGA组件大小 ”中所述 。

如果系统未使用自动内存管理或自动共享内存管理，则必须手动配置以下SGA组件的大小：

  • 数据库缓冲缓存

    使用DB_CACHE_SIZE初始化参数调整数据库缓冲区高速缓存的大小，如“ 配置数据库缓冲区高速缓存 ”中所述 。

  • 共享池

  • 大型游泳池

  • Java池

    使用JAVA_POOL_SIZE初始化参数调整Java池的大小。

  • 溪流池

    使用STREAMS_POOL_SIZE初始化参数调整Streams池的大小。

    使用INMEMORY_SIZE初始化参数调整IM列存储的大小。

12.2.1 SGA尺寸单位

缓冲区缓存，共享池，大型池和Java池的内存以粒度为单位进行分配。如果SGA大小小于1 GB，则颗粒大小为4MB。如果SGA大小大于1 GB，则颗粒大小将更改为16 MB。在数据库实例启动时计算并修复粒度大小。在实例的生命周期中，大小不会更改。

要查看当前用于SGA的粒度大小，请使用V$SGA_DYNAMIC_COMPONENTS视图。相同的颗粒大小用于SGA中的所有动态组件。

12.2.2 SGA的最大大小

数据库实例可用的最大内存量在实例启动时由SGA_MAX_SIZE初始化参数的值确定。您可以将总SGA大小扩展为等于SGA_MAX_SIZE参数的值。SGA_MAX_SIZE参数的值默认为所有SGA组件的聚合设置。

如果未设置SGA_MAX_SIZE参数的值，则减小一个高速缓存的大小，并在必要时将该内存重新分配给另一个高速缓存。或者，您可以将SGA_MAX_SIZE参数的值设置为大于所有SGA组件的总和，例如缓冲区高速缓存和共享池。这样做可以动态增加缓存大小，而不必减小另一个缓存的大小。

注意：

无法动态调整SGA_MAX_SIZE参数的值。

12.2.3应用注意事项

配置内存时，请根据应用程序的需要适当调整内存缓存的大小。相反，调整应用程序对内存缓存的使用可以大大减少资源需求。有效使用内存缓存还可以减少相关资源（如锁存器，CPU和I / O系统）的负载。

为获得最佳性能，请考虑以下事项：

  • 设计缓存以最有效的方式使用操作系统和数据库资源。

  • 将内存分配给Oracle数据库内存结构，以最好地反映应用程序的需求。

  • 如果对现有应用程序进行了更改或添加，请调整Oracle数据库内存结构的大小以满足已修改应用程序的需要。

  • 如果应用程序使用Java，请调查是否需要修改Java池的默认配置。

12.2.4.1减少寻呼

当操作系统将内存驻留页面仅传输到磁盘以将新页面加载到内存中时，就会发生分页 。许多操作系统页面可以容纳大量不适合实际内存的信息。在大多数操作系统上，分页会降低性能。

要确定主机系统上是否发生了重要分页，请使用操作系统实用程序检查操作系统。如果发生重要的分页，则总系统内存可能不足以容纳为其分配内存的内存高速缓存。考虑增加系统上的总内存，或减少分配的内存量。

12.2.4.2将SGA安装到主存储器中

因为SGA的目的是将数据存储在存储器中以便快速访问，所以SGA应该驻留在主存储器中。如果SGA的页面交换到磁盘，则不再可以快速访问数据。在大多数操作系统上，分页的缺点明显超过了大SGA的优势。

12.4使用内存中的列存储提高查询性能

内存中列存储（IM列存储）是系统全局区域（SGA）的可选部分，它以列式格式存储表，分区和其他数据库对象的副本，并且此列式数据针对快速扫描进行了优化。由于IM列存储将数据库对象存储在内存中，因此与对存储在磁盘上的数据执行这些操作相比，Oracle数据库可以更快地对该数据执行扫描，查询，连接和聚合。

注意：

  • IM列存储和数据库缓冲区缓存存储相同的数据，但格式不同。IM列存储不会替换数据库缓冲区高速缓存中的基于行的存储，而是对其进行补充以实现更好的查询性能。

  • 从Oracle Database 12 c第1版（12.1.0.2）开始，可以使用IM列存储。

12.5.1关于Memoptimized Rowstore

Memoptimized Rowstore为诸如物联网（IoT）应用程序之类的应用程序提供高性能数据流，这些应用程序通常同时从大量客户端单行插入少量数据，并且还为客户端查询数据非常高频率。

Memoptimized Rowstore提供以下功能：

  • 快速摄取

    快速摄取优化了高频，单行数据插入数据库的处理。快速摄取使用大型池在将插件写入磁盘之前缓冲插件，从而提高数据插入性能。

  • 快速查找

    快速查找可以快速从数据库中检索数据，以进行高频查询。快速查找在SGA中使用称为memoptimize池的单独内存区域来缓冲从表中查询的数据，从而提高查询性能。

    注意：

    要使用快速查找，必须使用MEMOPTIMIZE_POOL_SIZE初始化参数为memoptimize池分配适当的内存大小。

12.5.2使用快速摄取

快速摄取优化了从物联网（IoT）应用程序等应用程序插入数据库的高频单行数据的处理。

快速摄取使用MEMOPTIMIZE_WRITE提示将数据插入指定为MEMOPTIMIZE FOR WRITE表中。数据库临时将这些插入缓冲在大型池中，并在将这些缓冲的插入写入磁盘时自动提交更改。无法回滚更改。

使用快速摄取的插入也称为延迟插入 ，因为它们最初在大型池中缓冲，然后由后台进程异步写入磁盘。

使用快速摄取将数据插入表中的步骤

以下是使用快速摄取将数据插入表中的步骤：

  1. 启用快速摄取表

    通过在CREATE TABLE或ALTER TABLE语句中指定MEMOPTIMIZE FOR WRITE子句来实现此任务。

    有关详细信息，请参阅“ 为快速摄取启用表 ” 。

  2. 为插入操作指定使用快速摄取的提示

    通过在INSERT语句中指定MEMOPTIMIZE_WRITE提示来实现此任务。

    有关详细信息，请参阅“ 为数据插入指定使用快速摄取的提示 ” 。

具有以下特征的表不能使用快速摄取：

  • 表格：

    • 磁盘压缩
    • 内存压缩

以下是快速摄取的一些其他注意事项：

  • 由于快速摄取缓冲大型池中的数据，因此在系统发生故障时可能会丢失数据。为避免数据丢失，客户端必须在执行插入后保留数据的本地副本，以便在数据写入磁盘之前，它可以在系统出现故障时重放插入。客户端可以使用DBMS_MEMOPTIMIZE程序来跟踪插入的持久性。将插入写入磁盘后，客户端可以销毁其插入数据的本地副本。

  • 查询不从大型池读取数据，因此在将其写入磁盘之前无法查询使用快速摄取插入的数据。

  • 必须同步父子事务以避免错误。例如，如果父数据尚未写入磁盘，则插入大型池的行的外键插入和更新可能会返回错误。

  • 快速摄取支持索引操作，类似于常规插入。但是，对于快速摄取，数据库在将数据写入磁盘时执行索引操作，而不是在将数据写入大型池时执行索引操作。

改变物质化的视图

目的

物化视图是包含查询结果的数据库对象。查询的FROM子句可以命名表，视图和其他物化视图。这些源对象统称为主表 （复制术语）或详细信息表 （数据仓库术语）。此引用使用术语主表来保持一致性。包含主表的数据库称为主数据库 。

使用ALTER MATERIALIZED VIEW语句可以通过以下一种或多种方式修改现有的物化视图：

  • 改变其存储特性

  • 更改其刷新方法，模式或时间

  • 改变它的结构，使它成为一种不同类型的物化视图

  • 启用或禁用查询重写

注意：

自动刷新：示例

以下语句将sales_by_month_by_state化视图（在“ 创建实体化聚合视图：示例 ”中创建）的默认刷新方法更改为FAST ：

ALTER MATERIALIZED VIEW sales_by_month_by_state REFRESH FAST;

物化视图的下一次自动刷新将是快速刷新，前提是它是一个简单的物化视图，其主表具有在物化视图创建或上次刷新之前创建的物化视图日志。

由于REFRESH子句未指定START WITH或NEXT值，因此当创建或最后更改sales_by_month_by_state化视图时，Oracle数据库将使用REFRESH子句建立的刷新间隔。

以下语句为sales_by_month_by_state化视图的自动刷新之间建立了新的间隔：

ALTER MATERIALIZED VIEW sales_by_month_by_state REFRESH NEXT SYSDATE + 7;

由于REFRESH子句未指定START WITH值，因此在创建或最后更改sales_by_month_by_state化视图时指定的START WITH和NEXT值建立的时间发生下一次自动刷新。

在下一次自动刷新时，Oracle数据库刷新实例化视图，评估NEXT表达式SYSDATE +7以确定下一个自动刷新时间，并继续每周自动刷新一次实例化视图。由于REFRESH子句未明确指定刷新方法，因此Oracle数据库继续使用CREATE MATERIALIZED VIEW或最新ALTER MATERIALIZED VIEW语句的REFRESH子句指定的refresh方法。

考虑新鲜：例子

以下语句指示Oracle数据库应将物化视图sales_by_month_by_state视为新的。即使您对sales_by_month_by_state的主表执行了分区维护操作，此语句也允许sales_by_month_by_state符合TRUSTED模式下的查询重写sales_by_month_by_state ：

ALTER MATERIALIZED VIEW sales_by_month_by_state CONSIDER FRESH;

作为上述语句的结果，自上次刷新实例化视图以来对基表执行的任何分区维护操作都不会应用于实例化视图。例如，如果在下次刷新实例化视图之前使用CONSIDER FRESH ，则基表中分区中数据的添加，删除或更改将不会反映在实例化视图中。有关更多信息，请参阅CONSIDER FRESH 。
