  • GET TRANSACTIONS - 从出站服务器的应用协调器接收事务。

  • SEND TRANSACTION - 将事务发送到XStream客户端应用程序。

  • WAIT FOR NEXT CHUNK -等待一个大事务下一组的LCR的。

  • TRANSACTION CLEANUP - 清理已应用的事务，包括从出站服务器的队列中删除LCR。

  • WAIT FOR CLIENT -等待XStream的客户端应用程序请求更多的LCR。

  • IDLE - 由于没有LCR发送到XStream客户端应用程序，因此无法正常工作。

  • BROWSING LCR - 浏览出站服务器的队列以获取下一个LCR。

  • EVALUATING RULES - 根据规则集评估LCR。

  • DEQUEUING LCR - 从出站服务器的队列中取出LCR。

  • SENDING LCR - 将LCR发送到XStream客户端应用程序。

  • WAITING FOR CAPTURE TO TERMINATE -等待拍摄过程中被禁用。

  • SUSPENDED DUE TO A DROPPED SUBSCRIBER -由于连接的用户下降了暂停。例如，可以在拆分或合并操作期间删除订户。

  • SUSPENDED FOR AUTO SPLIT / MERGE正在执行暂停，因为自动分割或合并操作- 。

  • WAITING ON EMPTY QUEUE -等待从捕获过程更LCR的。

  • WAITING FOR CLIENT -等待XStream的客户端应用程序请求更多的LCR。

  • WAITING FOR CAPTURE TO INITIALIZE -等待捕获过程完成数据字典建立。

  • 使用DBMS_HPROF.CREATE_TABLES子程序创建分层探查器数据库表以及持久存储分析原始探查器数据结果所需的其他数据结构。

  • 如果表已存在， DBMS_HPROF.CREATE_TABLES使用默认值（ FALSE ）调用DBMS_HPROF.CREATE_TABLES将引发错误。

  • 使用DBMS_HPROF.CREATE_TABLES(TRUE)删除以前创建的任何分层探查器表。

  • 使用DBMS_HPROF.CREATE_TABLES删除以前创建的任何分层探查器表。默认情况下， force_it为FALSE ;因此，要删除任何先前创建的分层探查器表，必须将force_it的值设置为TRUE 。

  • 如果trace_id条目为NULL ，则引发错误。

  • 如果原始探查器数据表中的trace_id条目不存在，则会引发错误。

  • 如果原始探查器数据表中的trace_id条目的原始数据为NULL或大小zero ，则会引发错误。

在Oracle数据库中运行JavaScript的好处

  • 重用现有技能和代码。

  • 避免驻留在Oracle数据库中的运输数据。在数据库中执行JavaScript可以在数据库中就地生成JSON文档并加快处理速度。这样可以避免将数据传输到外部基础架构，并在避免网络开销时提高性能。

在Oracle数据库中维护JavaScript源的首选方法是将它们作为Java资源加载到您的模式中，然后在数据库中运行脚本。

  • 在Oracle JVM中限制直接调用Nashorn类。

  • Oracle JVM中禁用所有脚本模式扩展。

  • 您可以使用标准脚本API从字符串或文件流运行脚本，但Oracle不建议使用它。以这种方式调用的脚本只能以沙盒模式运行，具有默认权限。

V $ PQ_TQSTAT

您可以使用V$PQ_TQSTAT动态性能视图监视并行执行性能。

举一个简单的例子，考虑两个表之间的散列连接，在一个只有两个不同值的列上进行连接。最好的情况是，这个哈希函数有一个哈希值给并行执行服务器A而另一个给并行执行服务器B.一个DOP为2，但是，如果它是4，那么至少有两个并行执行服务器没有工作。要发现此类偏差，请使用类似于以下示例的查询：

SELECT dfo_number，tq_id，server_type，process，num_rows FROM V $ PQ_TQSTAT ORDER BY dfo_number DESC，tq_id，server_type，process;

解决此问题的最佳方法可能是选择其他连接方法;嵌套循环连接可能是最佳选择。或者，如果一个连接表相对于另一个连接表较小，则可以使用PQ_DISTRIBUTE提示来提示BROADCAST分配方法。优化程序考虑BROADCAST分配方法，但要求OPTIMIZER_FEATURES_ENABLE设置为9.0.2或更高。

现在，假设您有一个具有高基数的连接键，但是一个值包含大部分数据，例如，按年计算的熔岩灯销售额。唯一一年销量很大的一年是1968年，而1968年的并行执行服务器则不堪重负。您应该使用与前一段中描述的相同的纠正措施。

V$PQ_TQSTAT视图提供表队列级别的消息流量的详细报告。V$PQ_TQSTAT数据仅在从执行并行SQL语句的会话中查询时有效。表队列是查询服务器组之间，并行执行协调器和查询服务器组之间，或查询服务器组和协调器之间的管道。表队列由PX SEND <partitioning type> （例如， PX SEND HASH ）和PX RECEIVE在操作列中明确表示。

对于每个查询服务器进程， V$PQ_TQSTAT都有一行，它在每个表队列中读取或写入。将10个使用者进程连接到10个生产者进程的表队列在视图中有20行。通过TQ_ID和表队列标识符对字节列和组进行TQ_ID ，以获得通过每个表队列发送的总字节数。将此与优化器估计值进行比较;大的变化可能表明需要使用更大的样本来分析数据。

计算按TQ_ID分组的字节方差。大的差异表明工作量不平衡。您应该调查大差异，以确定生产者是否开始使用不均等的数据分布，或者分布本身是否有偏差。如果数据本身存在偏差，则可能表示基数较低或不同值较少。

语义

或替换

指定OR REPLACE以使用不同的定义替换分析视图的现有定义。

力量和NOFORCE

指定FORCE以强制创建分析视图，即使它未成功编译也是如此。如果指定NOFORCE ，则分析视图必须成功编译，否则会发生错误。默认为NOFORCE 。

模式

指定用于创建分析视图的架构。如果未指定架构，则Oracle数据库会在您自己的架构中创建分析视图。

analytic_view

指定分析视图的名称。

sharing_clause

指定是否将分析视图创建为应用程序公共对象。指定METADATA共享分析视图的元数据，但其数据对每个容器都是唯一的。指定DATA共享分析视图对象;它的数据对于应用程序容器中的所有容器都是相同的，并且数据仅存储在应用程序根目录中。指定NONE排除分析视图的共享。

classification_clause

使用classification子句指定CAPTION或DESCRIPTION分类的值，并指定用户定义的分类。分类提供描述性元数据，应用程序可用于提供有关分析视图及其组件的信息。

您可以为同一对象指定任意数量的分类。分类的最大长度为4000字节。

对于CAPTION和DESCRIPTION分类，你可以使用快捷键DDL CAPTION ' caption '和DESCRIPTION ' description '或完全分类的语法。

您可以按语言更改分类值。要为CAPTION或DESCRIPTION分类指定语言，必须使用完整语法。如果未指定语言，则分类的语言值为NULL 。语言值必须为NULL或有效的NLS_LANGUAGE值。

using_clause

指定事实表或视图。允许外部和远程表。您可以在另一个模式中指定表或视图。您可以为表或视图指定别名。

dim_by_clause

指定分析视图的属性尺寸。

dim_key

指定属性维，事实表的列，属性维的列以及分析视图中相关的层次结构。

使用KEY关键字，在事实表中指定一个或多个列。

使用REFERENCES关键字，指定分析视图的尺寸标注属性尺寸的属性。每个属性必须是级别键。DISTINCT关键字支持使用非规范化事实表，其中属性维和事实数据位于同一个表中。使用事实表定义属性维时，请使用REFERENCES DISTINCT 。

使用HIERARCHIES关键字，在分析视图中指定使用属性维的层次结构。

dim_ref

指定属性维度。您可以为属性维指定别名，如果您多次使用同一维，或者您使用不同架构中具有相同名称的多个维，则必须使用该别名。

hier_ref

指定层次结构。您可以为层次结构指定别名。您可以将列表中的一个层次结构指定为默认值。如果未指定默认值，则列表中的第一个层次结构为默认层次结构。

measures_clause

指定分析视图的度量。

av_measure

在此分析视图中使用单个事实列或表达式来定义度量。度量可以是基本度量或计算度量。

base_measure_clause

通过可选地指定事实列或meas_aggregate_clause或两者来定义基本度量。如果未指定事实列，则分析视图将使用与度量具有相同名称的事实表的列。如果不存在同名的列，则会引发错误。

calc_measure_clause

通过指定分析视图表达式来定义计算的度量。表达式可以引用分析视图中的其他度量，但可能不引用事实列。计算的度量没有聚合子句，因为它们是通过聚合的基本度量来计算的。

有关分析视图表达式的语法和说明，请参阅分析视图表达式 。

default_measure_clause

指定要用作分析视图的默认度量的度量。如果未指定度量，则定义的第一个度量是默认值。

meas_aggregate_clausè

为基本度量指定默认聚合运算符。如果未指定聚合运算符，则使用default_aggregate_clause指定的运算符。

default_aggregate_clause

为分析视图中的所有基本度量指定默认聚合。如果未指定默认聚合，则默认值为SUM 。

cache_clause

指定缓存子句可在适当的实例化视图可用时改进查询响应时间。您可以指定一个或多个缓存规范。

cache_specification

为缓存子句指定一个或多个度量值组。要包括所有度量值组，请指定ALL 。每个度量值组可以包含一个或多个度量以及一个或多个级别分组。级别分组可以包含一个或多个级别规范。

level_specification

为缓存规范指定度量值组的级别分组的一个或多个级别。每个层次结构仅指定一个级别。必须存在包含层次结构级别的聚合值的实例化视图。

使用Passthrough SQL的含义

执行在非Oracle系统中隐式提交或回滚事务的直通SQL语句时，事务会受到影响。

例如，某些系统隐式提交包含数据定义语言（DDL）语句的事务。由于Oracle数据库被绕过，Oracle数据库不知道事务是在非Oracle系统中提交的。因此，可以提交非Oracle系统中的数据，而Oracle数据库中的事务则不会。

如果回滚Oracle数据库中的事务，则可能会发生Oracle数据库与非Oracle系统之间的数据不一致。这种情况导致全局数据不一致。

请注意，如果应用程序执行典型的COMMIT语句，Oracle数据库可以协调分布式事务与非Oracle系统。使用passthrough工具执行的语句是分布式事务的一部分。

12.2.1代码生成

您必须能够指定预编译器生成的代码类型，C兼容代码或C ++兼容代码。Pro * C / C ++默认生成C代码。C ++不是C的完美超集。生成的代码需要进行一些更改，以便可以通过C ++编译器进行编译。

例如，除了发出应用程序代码之外，预编译器还会调用其运行时库SQLLIB。 SQLLIB中的函数是C函数。SQLLIB没有特殊的C ++版本。因此，如果要使用C ++编译器编译生成的代码，Pro * C / C ++必须将SQLLIB中调用的函数声明为C函数。

对于C输出，预编译器将生成诸如的原型

void sqlora（unsigned long *，void *）;

但对于C ++兼容代码，预编译器必须生成

extern“C”{void sqlora（unsigned long *，void *）; };

您可以使用预编译器选项CODE控制Pro * C / C ++生成的代码类型。此选项有三个值：CPP，KR_C和ANSI_C。可以通过考虑SQLLIB函数sqlora的声明如何在CODE选项的三个值之间不同来说明这些选项之间的差异：

void sqlora（/ * _ unsigned long *，void * _ * /）; / * K＆R C * / void sqlora（unsigned long *，void *）; / * ANSI C * / extern“C”{/ * CPP * / void sqlora（unsigned long *，void *）; };

  • 生成C ++可编译代码。

  • 为输出文件提供特定于平台的文件扩展名（后缀），例如“。C“或”.cc“，而不是标准的”.c“扩展名。（您可以使用CPP_SUFFIX选项覆盖它。）

  • 导致PARSE选项的值默认为PARTIAL。您也可以指定PARSE = NONE。如果指定PARSE = FULL，则在预编译时发出错误。

  • 允许在代码中使用C ++样式//注释。当CODE = CPP时，SQL语句和PL / SQL块中也允许使用这种注释方式。

  • Pro * C / C ++识别以// +开头的SQL优化器提示。

  • 要求OTT（对象类型转换器）生成的头文件必须包含在声明部分中。

您可以通过将资源配置文件与资源配置文件相关联来为任何目的配置Oracle XML DB资源库资源，该资源配置文件定义资源的可配置参数。资源配置文件本身是Oracle XML DB Repository中的资源。

资源配置文件是符合XML模式XDBResConfig.xsd的XML文件，可在Oracle XML DB Repository的路径/sys/schemas/PUBLIC/xmlns.oracle.com/xdb/XDBResConfig.xsd 。此XML架构由Oracle XML DB定义，您无法对其进行更改。

您使用PL / SQL过程DBMS_RESCONFIG.addResConfig将资源映射到配置它的文件。单个资源配置文件也可以应用于存储库中的所有资源。在这种情况下，您使用PL / SQL过程DBMS_RESCONFIG.addRepositoryResConfig将其映射到整个存储库。

如果适用，可以使用相同的资源配置文件来配置多个资源。Oracle建议，只要相同的配置有意义，您就可以通过这种方式共享配置文件。这可以提高运行时性能。它还允许您在单个位置更新配置并使更改影响多个资源，从而简化了存储库管理。

避免创建多个等效的资源配置文件，因为这会对性能产生负面影响。如果Oracle XML DB检测到重复的资源配置文件，则会引发错误。

通常，您配置资源以用于特定应用程序。为了使资源由多个应用程序共享，不同的应用程序必须能够以不同的方式对其进行配置。您可以通过创建多个资源配置文件并将它们映射到同一资源来完成此操作。因此，每个资源与配置列表 ， 资源配置列表相关联。配置列表中的配置按列表顺序处理。

存储库本身具有资源配置文件列表，用于存储库范围的配置，这实际上意味着配置存储库中的所有资源。不得将同一配置文件用于存储库本身和特定资源。否则，会引发错误。如果在任何给定资源配置列表中出现多次相同的资源配置文件，也会引发错误。

注意：

如果您尝试为存储库范围的配置创建超过125个资源配置文件，则会引发错误。

新资源的资源配置列表基于新资源的父文件夹的所有资源配置文件的configuration元素中的信息。如果没有此类信息（文件中没有配置文件或defaultChildConfig元素），则使用存储库资源配置列表的configuration元素。如果还缺少该信息，则新资源具有空资源配置列表。

您可以通过从视图RESOURCE_VIEW列RES中提取元素/Resource/RCList ，或使用PL / SQL过程DBMS_RESCONFIG.getResConfigPath来查看特定资源的配置列表。您可以使用PL / SQL过程DBMS_RESCONFIG.getRepositoryResConfigPath查看整个存储库的配置列表。要修改存储库范围的配置列表，必须为您授予角色XDBADMIN 。

  1. 创建定义配置的资源配置文件。此XML文件必须符合XML架构XDBResConfig.xsd 。
  2. 将资源配置文件作为资源本身添加到存储库：配置资源。您可以使用PL / SQL函数DBMS_XDB_REPOS.createResource来执行此操作。

  4. 承诺。

注意：

在执行使用资源配置文件的任何操作之前，必须执行COMMIT操作。在您这样做之前，无论何时使用配置文件，都会引发ORA-22881“悬空REF”错误。

PL / SQL包DBMS_RESCONFIG提供了从配置列表中删除配置，获取配置列表中配置的路径列表等的其他过程。

注意：

如果删除其他资源引用的资源配置文件，则只要尝试访问已配置的资源，就会引发悬空REF错误。

22.3.2配置元素defaultChildConfig和配置

配置元素defaultChildConfig仅适用于文件夹。它包含要应用于文件夹中所有子资源的配置信息。元素defaultChildConfig具有一个或多个configuration子元素，每个元素都定义文件夹中资源的可能配置。

configuration元素具有以下子元素：

  • pre-condition （可选） - 此元素指定在path元素标识的资源配置（参见下一个）可用作默认配置之前必须满足的条件。如果缺少元素pre-condition ，则path针对的资源配置文件将应用于该文件夹中的所有资源。也就是说，前提条件被视为真实。

    pre-condition元素具有可选的existsNode子元素。existsNode元素具有必需的XPath子元素和可选的namespace子元素，两个字符串。它们分别定义了XPath 1.0表达式和命名空间，用于检查资源是否存在。如果该资源存在，则满足前提条件，因此由path标识的资源配置文件将用作文件夹中所有子资源的默认资源配置文件。XPath元素的第一个组件必须是Resource 。

    注意：

    元素XPath复杂XPath表达式会对性能产生负面影响。

    如果多个configuration元素具有真正的前提条件，则由其关联的path元素标识的每个资源配置文件都应用于该文件夹中的所有资源。

  • path （必需） - 此元素指定资源配置文件的绝对存储库路径，只要满足元素pre-condition条件指定的前提条件，该文件将用作新资源的缺省配置。

通常， path元素的值是当前资源配置文件的path ，即包含path元素的文件。示例22-1说明了这一点，假设资源配置文件位于存储库中的路径/cm/app_rc.xml 。在此示例中，前提条件是存在其内容为xml类型的Resource节点。满足该前提条件时， 示例22-1中的资源配置文件适用于与配置文件（ /cm/app_rc.xml ）相同的文件夹中的所有资源。

1.1.2实例调整

诊断和调优数据库性能讨论了调整和优化Oracle数据库实例所涉及的因素。

在考虑实例调优时，请注意数据库的初始设计，以避免可能导致性能问题的瓶颈。此外，您必须考虑：

  • 将内存分配给数据库结构

  • 确定数据库不同部分的I / O要求

  • 调整操作系统以获得数据库的最佳性能

安装并配置数据库实例后，必须在数据库运行时监视数据库，以检查与性能相关的问题。

1.1.2.1性能原则

性能调整需要与系统初始配置不同但相关的方法。配置系统涉及以有序方式分配资源，以便初始系统配置正常运行。

通过确定最重要的瓶颈并进行适当的更改来减少或消除该瓶颈的影响，从而推动调优。通常，无论是在系统处于预生产阶段还是处于生产状态之后，都会以反应方式执行调整。

1.1.2.2基线

最有效的调整方法是建立一个既定的性能基准，如果出现性能问题，可以用它来进行比较。大多数数据库管理员（DBA）都很了解他们的系统，并且可以轻松识别高峰使用期。例如，高峰期可以在上午10点到下午12点之间以及下午1点30分到下午3点之间。这可能包括午夜12点到早上6点的批处理窗口。

在站点识别这些高峰期并安装监视工具以收集高负载时间的性能数据非常重要。最理想的情况是，应在QA周期内应用程序处于初始试用阶段时配置数据收集。否则，应在系统首次投入生产时进行配置。

理想情况下，收集的基线数据应包括以下内容：

  • 应用程序统计信息（事务量，响应时间）

  • 数据统计

  • 操作系统统计

  • 磁盘I / O统计信息

  • 网络统计

在自动工作负载存储库中，基准由一系列快照标识，这些快照将保留以供将来进行比较。请参阅“ 自动工作负载存储库 ” 。

1.1.2.3症状和问题

性能调优中常见的缺陷是将问题的症状误认为是实际问题本身。重要的是要认识到许多性能统计数据表明症状，并且识别症状不足以实施补救措施。例如：

  • 物理I / O缓慢

    通常，这是由配置不当的磁盘引起的。但是，它也可能是由调整不佳的SQL发出的大量不必要的物理I / O引起的。

  • 锁定争用

    很少通过重新配置实例来锁定争用可调。相反，闩锁争用通常通过应用程序更改来解决。

  • CPU使用率过高

    CPU使用率过高通常意味着系统上的空闲CPU很少。这可能是由于系统规模不合适，未经过调整的SQL语句或低效的应用程序造成的。

1.1.2.4何时调整

有两种不同类型的调整：

  • 主动监控

  • 瓶颈消除

1.1.2.4.1主动监测

主动监控通常在定期调度的时间间隔内进行，其中检查多个性能统计信息以确定系统行为和资源使用情况是否已更改。主动监控也可视为主动调整。

通常，监视不会导致系统的配置更改，除非监视暴露出正在发展的严重问题。在某些情况下，经验丰富的性能工程师可以通过统计数据识别潜在问题，尽管通常会伴随性能下降。

当没有明显的性能下降作为主动行动时，试验或调整系统可能是一项危险的活动，导致不必要的性能下降。调整系统应被视为反应调整，并应遵循反应调整的步骤。

监视通常是更大容量规划练习的一部分，其中检查资源消耗以查看应用程序使用方式的变化，以及应用程序使用数据库和主机资源的方式。

1.1.2.4.2消除瓶颈

调整通常意味着修复性能问题。但是，调整应该是应用程序生命周期的一部分 - 通过分析，设计，编码，生产和维护阶段。通常，调优阶段一直持续到数据库处于生产阶段。此时，调整成为一个反应过程，识别和修复最重要的瓶颈。

通常，调整的目的是减少资源消耗或减少操作完成所用的时间。无论哪种方式，目标都是改善特定资源的有效使用。通常，性能问题是由特定资源的过度使用引起的。过度使用的资源是系统的瓶颈。在确定瓶颈和潜在修复方面有几个不同的阶段。这些将在以下章节中讨论。

请记住，不同形式的争用是可以通过在以下位置进行更改来解决的症状：

  • 应用程序的更改或应用程序的使用方式

  • Oracle的变化

  • 主机硬件配置的更改

通常，解决瓶颈的最有效方法是更改应用程序。

1.1.3 SQL调优

许多应用程序程序员认为SQL是一种消息传递语言，因为会发出查询并返回数据。但是，客户端工具通常会生成效率低下的SQL语句。因此，编写最佳SQL需要很好地理解数据库SQL处理引擎。对于高事务处理系统尤其如此。

通常，联机事务处理（OLTP）应用程序发出的SQL语句一次只能在相对较少的行上运行。如果索引可以指向所需的确切行，那么Oracle数据库可以构建一个准确的计划，以通过尽可能短的路径有效地访问这些行。在决策支持系统（DSS）环境中，选择性不那么重要，因为它们通常访问表的大多数行。在这种情况下，全表扫描很常见，甚至不使用索引。本书主要关注OLTP应用程序。

DRDA网关包的手动绑定

网关使用多个DB2包，它通常在网关连接到DB2实例的第一次上传和绑定。在某些客户环境中，连接用户标识可能没有执行绑定所需的权限，或者某些客户可能更喜欢手动绑定包而不是允许网关进行绑定。

在这种情况下，Oracle为手动绑定提供了一组预定义的包。这些软件包带有一些限制，必须通过设置特定的网关初始化参数来设置值，否则网关将尝试自动重新绑定软件包。

本节包含以下子部分：

  • 手动绑定DB2 UDB for z / OS的包

  • 手动绑定DB2 UDB for Linux_Unix_和Windows的包

手动绑定DB2 UDB for z / OS的包

执行以下步骤以手动绑定DB2 UDB for z / OS的包：

  1. 使用参数DSORG=PS ， RECFM=FB ， RECFM=FB LRECL=80和BLKSIZE=3120在z / OS上分配顺序数据集。例如，
    用户身份。DBRMFILE.XMIT
  2. 使用参数DSORG=PO ， RECFM=FB ， RECFM=FB LRECL=80和BLKSIZE=6160分配分区LRECL=80 BLKSIZE=6160 。例如，
    用户身份。TG4DRDA.CNTL
  3. 将以下文件FTP到BINARY模式中先前分配的顺序数据集：
    $ ORACLE_HOME / dg4db2 /管理/ dg4db2_zos_dbrm.xmit

    使用PUT命令替换顺序数据集内容。

  4. 将以下文件以ASCII模式FTP到先前分配的PDS：
    $ ORACLE_HOME / dg4db2 /管理/ dg4db2_zos_bind.jcl

    使用PUT命令将文件作为成员名称BIND放入PDS。

  5. 使用ISPF的TSO命令选项（选项6）发出RECEIVE命令：
    接收INDS（'用户ID。DBRMFILE.XMIT“）

    指定DA（ userid.DDODBC.DBRMLIB ）作为RECEIVE命令的参数。这将解压缩xmit文件并创建指定的PDS名称。

  6. 编辑BIND JCL （ userid.TG4DRDA.CNTL(BIND) ）并按照说明更新JCL 。更新后，提交JCL以执行包的实际绑定并授予包的执行权限。

要将这些软件包与网关一起使用，请在网关初始化文件中设置以下init参数：

  • HS_OPEN_CURSORS = 200

  • HS_FDS_PACKAGE_COLLID = NULLID

Oracle建议不要使用Oracle Database Gateway主目录中的侦听器。而是在Grid home中配置监听器。默认情况下，在群集配置期间创建本地侦听器，该侦听器在网格基础结构主目录之外运行，并侦听节点虚拟IP（VIP）的指定端口（默认为1521）。

支持单客户端访问名称（SCAN）

Oracle Database 11g第2版及更高版本的客户端使用支持单客户端访问名称（SCAN）连接到数据库。它为连接到Oracle RAC的客户端提供单一名称，即使您在群集中添加或删除节点，该名称也不会在群集的整个生命周期中发生更改。与SCAN连接的客户端可以使用简单的连接字符串，例如瘦JDBC URL或EZConnect，并实现负载平衡和客户端连接故障转移。

除了三个SCAN侦听器（每个虚拟IP地址一个）之外，在托管数据库实例的每个节点上都有一个节点侦听器。使用两层侦听器（SCAN侦听器和节点侦听器）的目的是分离Oracle RAC中侦听器的两个功能，首先是负载平衡连接，其次是产生和遗留会话。SCAN侦听器将接收来自客户端的连接请求，由GNS（网格命名服务）随机分发。然后，SCAN侦听器将使用负载平衡度量标准将请求智能地重定向到最能够提供所请求服务的节点上的节点侦听器。数据库实例向SCAN侦听器注册为远程侦听器，并将节点侦听器注册为本地侦听器。

无法使用SCAN配置Oracle数据库网关，SCAN是数据库连接到网关的单个名称。有两个原因。网关不适用于远程侦听器。与数据库不同，您可以在其中指定REMOTE_LISTENER以设置为SCAN侦听器，但在网关中不支持它。这对于SCAN侦听器将连接路由到节点侦听器至关重要。其次，网关不向群集注册，以便将其作为群集资源进行管理。

对于网关，当Oracle数据库和Oracle数据库网关在同一群集上运行时，SCAN不是很有用。可以在安装数据库的每个节点上安装和配置Oracle数据库网关，并且可以配置数据库，使每个实例连接到在同一节点上运行的网关。

本地监听器

应使用本地侦听器配置Oracle Database Gateway服务。它是本地监听器产生网关进程。这意味着监听器应该知道要生成哪个网关进程。使用Grid infrastructure home中的listener.ora添加Gateway SID。如果数据库的SCAN侦听器已在该节点上运行，则可以使用相同的listener.ora文件来配置本地侦听器。

负载平衡和透明应用程序故障转移（TAF）

Oracle Database Gateway本身不支持客户端（使用数据库主目录中的tnsnames）或服务器端（使用SCAN）负载平衡。网关级别的负载平衡不适用，因为网关进程当前专用于单个会话。但是，通过为每个数据库实例关联不同的Oracle Database Gateway实例，可以实现节点级负载平衡，即通过数据库上的负载平衡选择负载最小的节点。

Oracle Database Gateway支持连接故障转移功能。如果为Gateway配置客户端连接负载平衡，则其工作方式与故障转移类似。

每当发生数据库故障转移时，即会话从一个数据库实例迁移到另一个数据库实例（在新节点上），迁移的会话将使用该新节点中的网关实例。

默认情况下，Oracle Call Interface（OCI）应用程序可以使用三种类型的Oracle Net故障转移功能：

  • session ：设置为故障转移会话。如果用户连接丢失，则会自动为备份上的用户创建新会话。此类故障转移不会尝试恢复选择操作。

  • select ：设置为允许具有打开游标的用户在失败后继续获取它们。但是，此模式涉及正常选择操作中客户端的开销。

  • none ：这是默认值。没有使用故障转移功能。这也可以明确指定，以防止发生故障转移。

要使故障转移起作用，需要使用多个侦听器地址配置Database home中的tnsnames.ora 。

如果实例在连接后失败，则TAF应用程序将故障转移到另一个节点的侦听器，并保留正在进行的任何SELECT语句。

在以下用于负载平衡的tnsnames.ora示例中，仅用作故障转移，仅当gateway1-server上的网关不可用时，数据库才连接到主机gateway2-server的网关：

17 Oracle Text Alternative Spelling

本章介绍Oracle Text处理单词替代拼写的各种方法。它还记录了Oracle Text在德语，丹麦语和瑞典语中使用的替代拼写约定。

涵盖以下主题：

  • 备选拼写功能概述

  • 覆盖备用拼写功能

  • 替代拼写约定

17.1备用拼写功能概述

某些语言对某些单词有其他拼写形式。例如，德语单词Schoen也可以拼写为Schön 。

该单词的形式是原始的或标准化的 。该单词的原始形式是它在源文档中的显示方式。标准化形式是它如何被转换，如果它被转换的话。根据被索引的单词以及哪些系统首选项有效（这些将在本章中讨论），单词的规范化形式可能与原始形式相同。而且，规范化形式可以包括多于一个拼写。例如， Schoen的标准化形式是Schoen和Schön 。

Oracle Text通过以下方式处理备用单词表单的索引：

  • 替代拼写 - 替代表单的索引已启用

  • 基于字母转换的重音字母转换为非重音表示

  • 接受新德语拼写改革后的德语拼写

通过为BASIC_LEXER指定适当的属性来启用这些功能。例如，通过为ALTERNATE_SPELLING属性指定GERMAN ， DANISH或SWEDISH来启用备用拼写。例如，以下是如何在德语中启用备用拼写：

开始ctx_ddl.create_preference（'GERMAN_LEX'，'BASIC_LEXER'）; ctx_ddl.set_attribute（'GERMAN_LEX'，'ALTERNATE_SPELLING'，'GERMAN'）;结束;

要禁用备用拼写，请使用CTX_DDL.UNSET_ATTRIBUTE过程，如下所示：

开始ctx_ddl.unset_attribute（'GERMAN_LEX'，'ALTERNATE_SPELLING'）;结束;

Oracle Text在查找之前将查询术语转换为其规范化形式。因此，用户可以使用拼写查询单词。如果舒恩已被索引既是舍恩和舍恩 ，与舍恩查询返回包含任何形式的文件。

本节包含以下主题。

  • 替代拼写

  • 基本字母转换

  • 新德语拼写

17.1.1替代拼写

当瑞典语，德语或丹麦语有多种拼写方式时，Oracle Text通常会以原始形式对单词进行索引;也就是说，它出现在源文档中。

启用“备用拼写”时，Oracle Text会以其规范化形式对单词进行索引。因此，例如， Schoen被索引为Schoen和Schön ，对Schoen的查询将返回包含拼写的文档。（对Schön的查询也是如此。）

要启用备用拼写，请将BASIC_LEXER属性ALTERNATE_SPELLING设置为GERMAN ， DANISH或SWEDISH 。有关更多信息，请参见“ BASIC_LEXER ” 。

17.1.2基本字母转换

除了备选拼写之外，Oracle Text还处理基本字母转换。启用基地字母转换，用变音符号的字母，重音符，变音符号等被转换成自己的索引基本形式，所以未婚夫既是未婚夫和未婚妻被索引，和未婚夫的查询返回包含任何形式的文件。

要启用基本字母转换，请将BASIC_LEXER属性BASE_LETTER设置为YES 。有关更多信息，请参见“ BASIC_LEXER ” 。

如果还启用了备用拼写，则可能需要覆盖基本字母转换以防止出现意外结果。有关详细信息，请参阅“ 覆盖备用拼写功能 ” 。

通用与特定语言的基本字母转换

BASE_LETTER_TYPE属性会影响基本字母转换的发生方式。它有两个可能的值： GENERIC或SPECIFIC 。

GENERIC值是默认值，指定基本字母转换使用一个适用于所有语言的转换表。

SPECIFIC值表示将使用专门为您的语言定义的基本字母转换。这使您可以对您自己语言的单词使用重音敏感搜索，同时忽略来自其他语言的重音。

例如， GENERIC和西班牙语SPECIFIC表都会将é转换为e 。然而，他们对待字母n明显。所述GENERIC表对待N作为带有口音（实际上，一个波浪线）的n，所以将N个至n。西班牙语SPECIFIC表将ñ视为单独的字母表字母，因此不会对其进行转换。

17.1.3新德语拼写

1996年，德语的新拼写规则得到了所有德语国家代表的批准。例如，在拼写改革中， 潜能成为Potenzial ， Schiffahrt成为Schifffahrt ，而schneuzen成为schnäuzen 。

当BASIC_LEXER属性NEW_GERMAN_SPELLING设置为YES ，对包含新表单和传统表单的德语单词的CONTAINS查询将返回与两个表单匹配的文档。例如，对Potential的查询返回包含Potential和Potenzial的文档。默认设置为NO 。

注意：

在改革后的德语拼写中，传统上拼写为单词的许多单词，如soviel ，现在拼写为两个（ 所以viel ）。目前，Oracle Text不进行这些转换，也不进行从两个单词到一个单词的转换（例如， weh tun to wehtun ）。

变换后的单词的大小由源文档中单词的前两个字符确定;也就是说， schiffahrt成为schifffahrt ， Schiffahrt成为Schifffahrt ，而SCHIFFAHRT成为SCHIFFFAHRT 。

由于许多新的德语拼写都包含连字符，因此建议选择NEW_GERMAN_SPELLING用户将连字符定义为printjoin 。

有关设置此属性的详细信息，请参阅“ BASIC_LEXER ” 。

17.2覆盖备用拼写功能

即使词法分析器首选项指定了替代拼写功能，也可以覆盖它们。

使用备用拼写时，您可以覆盖基本字母转换，以防止具有备用拼写形式（如ü ， ö和ä）的字符也被转换为基本字母表单。

使用备用拼写覆盖基本字母转换

通过启用alternate_spelling导致的转换在base_letter转换之前执行，这有时会在启用两者时导致意外结果。

启用“备用拼写”时，Oracle Text会将双字母形式转换为单字母形式（例如， ue到ü ），以便可以在基本形式和替代形式中搜索单词。因此，启用备用拼写后，搜索Schoen将返回Schoen和Schön的文档。

但是，当启用基本字母转换时， Schlüssel中的ü转换为u ，产生不存在的单词（无论如何都是德语） Schlussel ，并且该单词以所有三种形式编入索引。

要阻止此二次转换，请将OVERRIDE_BASE_LETTER属性设置为TRUE。

OVERRIDE_BASE_LETTER只影响带变音符号的字母;例如，重音字母仍会转换为基本形式。

有关BASE_LETTER更多BASE_LETTER ，请参阅“ 基本字母转换 ” 。
