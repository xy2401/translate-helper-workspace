6.1.1关于ADO策略和IM列存储

ADO通过ADO策略管理IM列存储。您只能在段级别创建带有INMEMORY子句的ADO策略。

数据库将ADO策略视为对象的属性。ADO策略位于数据库级别，而不是实例级别。Oracle数据库支持以下类型的Database In-Memory ADO策略：

  • INMEMORY政策

    此策略使用INMEMORY属性标记对象，使其可用于IM列存储中的填充。在表级别设置时， INMEMORY属性仅适用于内部分区;因此，混合分区表的外部分区不受策略管理。

  • 重新压缩政策

    此策略更改INMEMORY对象的压缩级别。

  • NO INMEMORY政策

    此策略从IM列存储中删除对象并删除其INMEMORY属性。

Oracle数据库支持以下标准以确定何时应用策略：

  • 自修改对象以来的指定天数

    从DBA_HEAT_MAP_SEGMENT视图中的SEGMENT_WRITE_TIME列中获取此值。

  • 访问对象后指定的天数

    此值是DBA_HEAT_MAP_SEGMENT视图中SEGMENT_WRITE_TIME ， FULL_SCAN和LOOKUP_SCAN列中的较大值。

  • 自创建对象以来的指定天数

    从DBA_OBJECTS的CREATED列获取此值。

  • 用户定义的函数返回一个布尔值

6.1.2 ADO和IM列存储的目的

ADO将IM列存储作为新数据层进行管理。

您可以创建策略，以便在不经常访问IM列存储时从其中逐出对象，并在更频繁地访问对象时填充对象，从而提高查询性能。ADO使用热图统计信息管理IM列存储。

INMEMORY政策的目的

在许多数据库中，片段在创建后经过大量修改。为了最大限度地提高性能，当写入活动平息时，ADO可以在IM列存储中填充这些段。例如，如果您每天向表中添加一个分区，那么您可以创建一个策略，在创建后一天填充sales_2016_d100分区：

ALTER TABLE销售修改分区sales_2016_d100 ILM添加政策设置INMEMORY MEMCOMPRESS在创作1天后获得高优先级

类似地，您可能知道创建后两个月表上的写入活动会消退，并且希望在满足此时间条件时填充此对象：

更新表2016_ski_sales ILM添加政策设定内容存储为创作60天后的关键优先级

上述策略会导致2016_ski_sales表的所有现有分区和新分区继承该策略。当段符合策略条件时，数据库会使用指定的INMEMORY子句独立标记每个分区。如果该段已具有INMEMORY策略，则数据库将忽略新策略。

重新压缩政策的目的

您可能希望根据访问模式压缩IM列存储中的数据。例如，您可能希望在段上的DML活动停止后2天将段从DML压缩更改为查询压缩：

ALTER TABLE lineorders ILM ADD POLICY MODIFY INMEMORY MEMCOMPRESS在2天没有修改的情况下获得高价

如果未在IM列存储中填充对象，则此策略仅更改压缩属性。如果在IM列存储中填充对象，则ADO使用新的压缩级别重新填充对象。如果段尚未具有INMEMORY属性，则数据库将忽略该策略。

没有INMEMORY政策的目的

要优化IM列存储中的空间，您可能希望使用NO INMEMORY策略驱逐非活动段。此策略还可用于通过不频繁的查询来防止非活动段的填充。例如，如果特定销售分区的报告在一年中频繁运行，但通常不是每周运行，那么您可能希望在一周无法访问后逐出该分区：

ALTER TABLE销售修改分区sales_2015_q1 ILM添加政策7天后无法访问;

如果很少查询1998年的销售表，那么您可能希望在1天无法访问后逐出：

ALTER TABLE sales_1998 ILM添加政策在没有访问的1天之后没有任何内容;

永远不会阻止被驱逐段的查询。数据库总是可以通过传统的缓冲区缓存机制访问数据。

6.1.3.1热图的工作原理

启用后，热图会自动发现数据访问模式。ADO使用热图数据在数据库级别实现用户定义的策略。

热图自动跟踪行和段级别的使用信息。在行级别，热图跟踪数据修改时间，然后将这些时间聚合到块级别。在段级别，热图跟踪修改，全表扫描和索引查找的时间。

启用IM列存储时，热图会跟踪列数据的访问模式。例如， sales表可能是“热门”，而locations表可能是“冷”。ADO算法对于柱状数据和基于行的数据的工作方式相同。

数据库定期将热图数据写入数据字典。数据库在数据字典视图中公开热图数据。例如，要获取In-Memory对象的读取和写入时间，请查询ALL_HEAT_MAP_SEGMENT视图。

6.1.5为IM列存储创建ADO策略

您可以使用ADO策略根据热图统计信息为对象设置，修改或删除INMEMORY子句。

要创建ADO IM列存储策略，请在ALTER TABLE语句中指定ILM ADD POLICY子句，然后指定以下子句之一：

  • SET INMEMORY ......分割

    当您只想在DML活动平息时使用INMEMORY属性标记段时，此选项很有用。

  • 修改INMEMORY ......MEMCOMPRESS ......分割

    在频繁修改时，存储未压缩的数据或MEMCOMPRESS FOR DML级别的数据是合适的。替代压缩级别更适合查询。如果段上的活动从大多数写入转换为大多数读取，则可以使用MODIFY子句应用不同的压缩方法。

  • 没有内容......分割

    当对段的访问随时间减少（变为“冷”）时，此选项很有用，并且可防止由于随机访问而导致该段的填充。

先决条件

在使用ADO IM列存储策略之前，必须满足以下先决条件：

  • 通过将INMEMORY_SIZE初始化参数设置为非零值并重新启动数据库，为数据库启用IM列存储。

  • HEAT_MAP初始化参数必须设置为ON 。

    热图提供了段级别的数据访问跟踪和段级别和行级别的数据修改跟踪。

  • COMPATIBLE初始化参数必须设置为12.2.0或更高。

要创建ADO策略：

  1. 在SQL * Plus或SQL Developer中，以具有必要权限的用户身份登录数据库。

  2. 使用带有ILM ADD POLICY ...的ALTER TABLE语句ILM ADD POLICY ...INMEMORY条款。

6.2.1 自动内存中的目的

为确保始终填充工作数据集， Automatic In-Memory会自动驱逐冷（不经常访问）的段。

IM列存储仅删除填充的段（如果它被删除或移动）， INMEMORY选项被删除，或者IM ADO策略对其进行操作。当INMEMORY数据集的大小超过IM列存储的可用内存时，会发生内存压力，并且某些已填充的段将变为非活动状态。为获得最佳性能，IM列存储应包含最常查询的段，称为工作数据集 。

通常，工作数据集随许多应用程序的时间而变化。因此，优化需要DBA的定期监视和手动干预，以逐出IM存储元素或创建ADO IM策略。这两项任务都需要充分了解工作量。

通过自动驱逐冷段， Automatic In-Memory具有以下优点：

  • 改善了性能

    通过逐出冷段来减轻内存压力， 自动内存可以提高工作负载的性能，因为工作数据集驻留在IM列存储中。

  • 易于管理

    通过驱逐冷段来管理IM列存储器以减轻存储器压力涉及显着的用户干预。自动内存通过最少的用户干预解决了这些问题。

6.2.2 自动内存工作原理

数据驱逐单位是INMEMORY段。

INMEMORY段仅在其优先级为NONE时才有资格被驱逐。基本流程如下：

  1. 填充作业失败，这意味着IM列存储空间已用尽。

  2. 数据库使用符合条件的填充段的内部统计信息来定义要驱逐的对象集。统计数据类似于热图所使用的统计数据，但不需要启用热图。

  3. 对于集合中的每个段，数据库会检查是否为该段启用了ADO策略：

    • 如果启用的策略要求保留该段，则ADO策略将覆盖自动内存中 。数据库什么都不做。

    • 如果没有策略阻止驱逐，则自动内存中提交任务以驱逐段。

  4. W nnn进程会驱逐通过上述检查的任何段，从而释放IM列存储中的空间。

    对于被驱逐的段，保留INMEMORY属性。

例如，每晚批处理作业加载sales分区（优先级为NONE ），然后查询分区以触发填充。由于IM列存储几乎处于其最大容量，因此仅填充分区的一半行。无法完全填充新分区会触发自动内存 ，从而驱逐冷段。新分区的后续按需填充作业将完全填充新的sales分区。

6.2.3 自动内存中的用户界面

使用初始化参数INMEMORY_AUTOMATIC_LEVEL启用和禁用自动内存 。

初始化参数

系统级初始化参数INMEMORY_AUTOMATIC_LEVEL具有以下可能值：

  • OFF （默认）

    此选项禁用自动内存中 ，将IM列存储返回到其Oracle Database 12c第2版（12.2.0.1）行为。

  • 低

    在内存压力下，数据库会从IM列存储中驱逐冷段。

  • 介质

    此级别包括一个额外的优化，可确保首先填充因内存压力而未填充的任何热段。

注意：

自动内存不需要启用HEAT_MAP初始化参数。

Oracle建议您为工作数据集配置足够的内存以适应IM列存储。作为调整额外的自动内存中共享池要求的经验法则，将5 KB乘以SGA内存的INMEMORY段数。例如，如果10,000个段具有INMEMORY属性，则为自动内存中保留50 MB的共享池。

当网上fsck是一个安装文件系统上运行，在线fsck首先创建文件系统元数据的点对点即时快照。然后，联机fsck对此元数据快照执行文件系统检查。由于文件系统已挂载并处于活动状态，因此可以在在线fsck下创建或更新文件系统元数据。但是，联机fsck不会检查任何新的或更新的元数据，因为联机fsck仅检查初始元数据快照。

在线fsck使用称为写时复制（COW）文件的特殊文件。当在线fsck在文件系统上运行时，每当必须更新任何文件系统元数据时，首先将元数据的原始版本复制到COW文件中以保留它以进行文件系统检查。COW文件有助于保留在线fsck检查的元数据快照。

必须提前选择COW文件的大小。因此，COW文件具有有限的空间用于保留元数据的副本。如果COW文件空间不足而保留元数据副本，则可能会强制联机fsck中止。必须仔细选择COW文件的大小，以避免在执行在线fsck期间空间不足。COW文件的适当大小取决于在线fsck执行期间可能更改的文件系统元数据的数量，这取决于工作负载，可能需要凭经验确定。Oracle建议COW文件的大小从正在进行联机fsck的文件系统的5％开始。

COW文件必须放在单独的Oracle ACFS文件系统上。必须将单独的Oracle ACFS文件系统安装在与正在进行联机fsck的Oracle ACFS文件系统相同的节点上。

mount将文件系统附加到作为目录名称的安装点的Oracle ACFS层次结构。挂载发生在发出mount命令的节点上。如果文件系统未在此节点上处于卸载状态，则mount命令将返回错误。

并不总是可以将mount错误的原因返回到mount命令。发生这种情况时，Oracle ACFS会将失败原因写入系统控制台和关联的系统日志文件。

mount成功运行后， V$ASM_VOLUME视图中的MOUNTPATH字段将显示现在挂载文件系统的目录名称。

Oracle ACFS文件系统只应安装在一个安装点上。应在所有集群成员上使用相同的安装点名称。

如果在没有参数的情况下运行， mount命令将列出所有已安装的文件系统。

root权限是运行mount所必需的。

11.1关于Apriori

了解Apriori。

一个关联挖掘问题可以分解为以下子问题：

  • 查找以指定的最小频率发生的一组事务中的所有项目组合。这些组合被称为 频繁项目集 。

  • 计算表示频繁项目集中可能的项目共现的规则。

Apriori计算项目出现在频繁项目集中的概率，假设存在另一个或多个项目。

不建议使用关联规则挖掘来查找涉及具有大量项目的问题域中的罕见事件的关联。Apriori发现频率高于最小支持阈值的模式。因此，要查找涉及罕见事件的关联，算法必须以非常低的最小支持值运行。但是，这样做可能会爆炸枚举项目集的数量，尤其是在具有大量项目的情况下。这显着增加了执行时间。当数据具有大量属性时，分类或异常检测更适合发现罕见事件。

Apriori的构建过程支持并行执行。

11.2关联规则和频繁项集

Apriori算法计算表示频繁项集中的项之间的概率关系的规则。例如，从包含A，B和C的频繁项集导出的规则可能表明如果A和B包含在事务中，则可能还包括C。

关联规则指出项目或项目组意味着存在具有某种可能性的另一项目。与预测目标的决策树规则不同，关联规则仅表示相关性。

11.2.1先行和后续

关联规则的IF组件称为前提 。THEN组件被称为结果 。前因和后果是不相交的;他们没有共同的项目。

Oracle Data Mining支持关联规则，其中包含先行条件中的一个或多个项目以及后续条目中的单个项目。

11.2.2信心

规则有关联置信度，即在先行词发生的情况下发生后果的条件概率。您可以指定规则的最小置信度。

11.3 Apriori的数据准备

关联模型旨在使用交易数据。在事务数据中，案例标识符与每个案例的值之间存在一对多关系。每个案例ID /值对在单独的记录（行）中指定。

11.3.1本机事务数据和星型模式

了解交易数据的存储格式。

事务数据可以以本机事务格式存储，具有非唯一的案例ID列和值列，或者它可以存储在某些其他配置中，例如星型模式。如果数据未以本机事务格式存储，则必须将其转换为嵌套列，用于Apriori算法处理。

11.3.2项目和集合

在交易数据中，项目集合与每个案例相关联。该集合理论上包括该集合的所有可能成员。例如，理论上可以在单个市场篮子交易中购买所有产品。但是，实际上，在给定的交易中只存在所有可能项目的一小部分;市场购物篮中的商品仅代表商店中可供销售商品的一小部分。

11.3.3稀疏数据

通过稀疏性了解缺失的项目。

集合中缺少的项目表示稀疏性 。丢失的项目可能存在空值，或者可能只是丢失。

假定事务数据中的空值表示已知但在事务中不存在的值。例如，可以在单个交易中购买数百个可能项目中的三个项目。未购买的商品已知但在交易中不存在。

Oracle Data Mining假设事务数据稀疏。Apriori算法针对处理稀疏数据进行了优化。

注意：

Apriori不受自动数据准备的影响。

  • 置信水平γ

d -index被定义为最大整数d ，使得数据集至少包含d个最小长度为d的事务。它是Vapnik-Chervonenkis（VC）维度的上限。AR算法通过扫描数据集中所有事务的长度来计算数据集的d- index。

用户指定绝对误差ε和置信水平γ参数。大的d-指数，小的AR支撑，小的ε或大的γ可能导致大的样本量。样本大小理论上保证近似AR（来自采样）的支持和置信度的绝对误差小于ε ，而精确AR与概率（或置信水平）至少为γ 。在本文档中，此样本大小称为AR特定样本大小。

11.3.4.1抽样实施

当用户接通采样（样本大小仅计算ODMS_SAMPLING设定为ODMS_SAMPLING_ENABLE ）和不指定样本大小（ ODMS_SAMPLE_SIZE是未指定）。

使用说明

  1. 如果未指定ODMS_SAMPLING或将其设置为ODMS_SAMPLING_DISABLE ，则不对AR执行采样并获得准确的AR。

  2. 如果将ODMS_SAMPLING设置为ODMS_SAMPLING_ENABLE并且如果将ODMS_SAMPLE_SIZE指定为正整数，则使用用户指定的样本大小（ ODMS_SAMPLE_SIZE ）。在AR算法之前的一般数据准备阶段中执行采样。不计算AR特定的样本大小。获得近似AR。

    如果计算出的AR特定样本大小大于或等于数据集中的总事务大小，则不执行采样并获得准确的AR。

如果用户对AR的样本大小选择不太了解，建议不要指定ODMS_SAMPLE_SIZE ，仅指定采样参数的正确值，并让AR算法计算合适的AR特定样本大小。

11.5评估关联规则

最小支持和置信度用于影响关联模型的构建。支持和信心也是评估模型生成的规则质量的主要指标。此外，Oracle Data Mining还支持提升关联规则。这些统计测量可用于对规则进行排序，从而对预测的有用性进行排序。

11.5.1支持

一个人的支持 rule表示规则中的项目一起出现的频率。例如，谷物和牛奶可能在40％的交易中一起出现。如果是这样，以下规则各自支持40％：

谷物意味着牛奶意味着谷物

支持是包含前提中的所有项目以及总交易数量的交易比率。

支持可用概率表示法表示如下：

支持（A暗示B）= P（A，B）

11.5.2最低支持计数

最小支持Count定义每个规则必须满足的事务中的最小阈值。

当事务数未知时，支持百分比阈值参数可能很难设置。因此，支持也可以表示为事务计数，两个阈值中的较大者用于过滤不频繁的项集。默认值为1表示未应用此标准。

谷物意味着牛奶有80％的信心

置信度（A暗示B）= P（B / A），等于P（A，B）/ P（A）

11.5.5升降机

都必须使用支持和信心来确定规则是否有效。但是，有时候这两种措施都可能很高，但仍然会产生一种无用的规则。例如：

购买橙汁的便利店顾客也会以75％的信心购买牛奶。牛奶和橙汁的组合支持率为30％。

这听起来像是一个很好的规则，在大多数情况下，它会是。它具有很高的信心和高度的支持。但是，如果便利店顾客一般在90％的时间购买牛奶怎么办？在这种情况下，橙汁顾客实际上比一般顾客更不可能购买牛奶。

需要采取第三项措施来评估规则的质量。提升表示规则的强度，相对于先前和随后的共同发生，给予他们个人的支持。它提供了有关改进的信息，以及由此产生的后果概率的增加。提升定义如下。

（规则支持）/（支持（先行）*支持（后续））

这也可以定义为项目组合的置信度除以结果的支持。因此，在我们的牛奶示例中，假设40％的客户购买橙汁，其改进将是：

30％/（40％* 90％）

这是0.83 - 小于1的改进。

任何改善小于1的规则并不表示真正的交叉销售机会，无论其支持和信心有多高，因为它实际上提供的预测购买能力低于随机机会。

10.1关于Apriori

了解Apriori。

一个关联挖掘问题可以分解为以下子问题：

  • 查找以指定的最小频率发生的一组事务中的所有项目组合。这些组合被称为 频繁项目集 。

  • 计算表示频繁项目集中可能的项目共现的规则。

Apriori计算项目出现在频繁项目集中的概率，假设存在另一个或多个项目。

不建议使用关联规则挖掘来查找涉及具有大量项目的问题域中的罕见事件的关联。Apriori发现频率高于最小支持阈值的模式。因此，要查找涉及罕见事件的关联，算法必须以非常低的最小支持值运行。但是，这样做可能会爆炸枚举项目集的数量，尤其是在具有大量项目的情况下。这显着增加了执行时间。当数据具有大量属性时，分类或异常检测更适合发现罕见事件。

Apriori的构建过程支持并行执行。

10.2关联规则和频繁项集

Apriori算法计算表示频繁项集中的项之间的概率关系的规则。例如，从包含A，B和C的频繁项集导出的规则可能表明如果A和B包含在事务中，则可能还包括C。

关联规则指出项目或项目组意味着存在具有某种可能性的另一项目。与预测目标的决策树规则不同，关联规则仅表示相关性。

10.2.1先行和后续

关联规则的IF组件称为前提 。THEN组件被称为结果 。前因和后果是不相交的;他们没有共同的项目。

Oracle Data Mining支持关联规则，其中包含先行条件中的一个或多个项目以及后续条目中的单个项目。

10.2.2信心

规则有关联置信度，即在先行词发生的情况下发生后果的条件概率。您可以指定规则的最小置信度。

10.3 Apriori的数据准备

关联模型旨在使用交易数据。在事务数据中，案例标识符与每个案例的值之间存在一对多关系。每个案例ID /值对在单独的记录（行）中指定。

10.3.1本机事务数据和星型模式

了解交易数据的存储格式。

事务数据可以以本机事务格式存储，具有非唯一的案例ID列和值列，或者它可以存储在某些其他配置中，例如星型模式。如果数据未以本机事务格式存储，则必须将其转换为嵌套列，用于Apriori算法处理。

10.3.2项目和集合

在交易数据中，项目集合与每个案例相关联。该集合理论上包括该集合的所有可能成员。例如，理论上可以在单个市场篮子交易中购买所有产品。但是，实际上，在给定的交易中只存在所有可能项目的一小部分;市场购物篮中的商品仅代表商店中可供销售商品的一小部分。

10.3.3稀疏数据

通过稀疏性了解缺失的项目。

集合中缺少的项目表示稀疏性 。丢失的项目可能存在空值，或者可能只是丢失。

假定事务数据中的空值表示已知但在事务中不存在的值。例如，可以在单个交易中购买数百个可能项目中的三个项目。未购买的商品已知但在交易中不存在。

Oracle Data Mining假设事务数据稀疏。Apriori算法针对处理稀疏数据进行了优化。

10.3.4改进抽样

根据Riondato和Upfal的工作，关联规则（AR）可以使用具有性能保证的良好样本量。

AR算法通过以下输入计算样本大小：

  • d数据集的索引

  • 绝对误差ε

  • 置信水平γ

d -index被定义为最大整数d ，使得数据集至少包含d个最小长度为d的事务。它是Vapnik-Chervonenkis（VC）维度的上限。AR算法通过扫描数据集中所有事务的长度来计算数据集的d- index。

用户指定绝对误差ε和置信水平γ参数。大的d-指数，小的AR支撑，小的ε或大的γ可能导致大的样本量。样本大小理论上保证近似AR（来自采样）的支持和置信度的绝对误差小于ε ，而精确AR与概率（或置信水平）至少为γ 。在本文档中，此样本大小称为AR特定样本大小。

10.3.4.1抽样实施

当用户接通采样（样本大小仅计算ODMS_SAMPLING设定为ODMS_SAMPLING_ENABLE ）和不指定样本大小（ ODMS_SAMPLE_SIZE是未指定）。

使用说明

  1. 如果未指定ODMS_SAMPLING或将其设置为ODMS_SAMPLING_DISABLE ，则不对AR执行采样并获得准确的AR。

  2. 如果将ODMS_SAMPLING设置为ODMS_SAMPLING_ENABLE并且如果将ODMS_SAMPLE_SIZE指定为正整数，则使用用户指定的样本大小（ ODMS_SAMPLE_SIZE ）。在AR算法之前的一般数据准备阶段中执行采样。不计算AR特定的样本大小。获得近似AR。

  3. 如果ODMS_SAMPLING设置为ODMS_SAMPLING_ENABLE ODMS_SAMPLE_SIZE指定ODMS_SAMPLE_SIZE ，则计算AR指定的样本大小，然后在AR算法中执行采样。获得近似AR。

    注意：

    如果计算出的AR特定样本大小大于或等于数据集中的总事务大小，则不执行采样并获得准确的AR。

如果用户对AR的样本大小选择不太了解，建议不要指定ODMS_SAMPLE_SIZE ，仅指定采样参数的正确值，并让AR算法计算合适的AR特定样本大小。

10.5评估关联规则

最小支持和置信度用于影响关联模型的构建。支持和信心也是评估模型生成的规则质量的主要指标。此外，Oracle Data Mining还支持提升关联规则。这些统计测量可用于对规则进行排序，从而对预测的有用性进行排序。

10.5.1支持

一个人的支持 rule表示规则中的项目一起出现的频率。例如，谷物和牛奶可能在40％的交易中一起出现。如果是这样，以下规则各自支持40％：

谷物意味着牛奶意味着谷物

支持是包含前提中的所有项目以及总交易数量的交易比率。

支持可用概率表示法表示如下：

支持（A暗示B）= P（A，B）

10.5.2最低支持计数

最小支持Count定义每个规则必须满足的事务中的最小阈值。

当事务数未知时，支持百分比阈值参数可能很难设置。因此，支持也可以表示为事务计数，两个阈值中的较大者用于过滤不频繁的项集。默认值为1表示未应用此标准。

10.5.3信心

规则的置信度表示先行和后续出现在同一交易中的概率。

信心是给定前因后的条件概率。例如，谷物出现在50个交易中; 50个中的40个也可能包括牛奶。规则信心是：

谷物意味着牛奶有80％的信心

置信度是规则支持与包含前提的交易数量的比率。

置信度可用概率表示法表示如下。

置信度（A暗示B）= P（B / A），等于P（A，B）/ P（A）

10.5.5升降机

都必须使用支持和信心来确定规则是否有效。但是，有时候这两种措施都可能很高，但仍然会产生一种无用的规则。例如：

购买橙汁的便利店顾客也会以75％的信心购买牛奶。牛奶和橙汁的组合支持率为30％。

这听起来像是一个很好的规则，在大多数情况下，它会是。它具有很高的信心和高度的支持。但是，如果便利店顾客一般在90％的时间购买牛奶怎么办？在这种情况下，橙汁顾客实际上比一般顾客更不可能购买牛奶。

需要采取第三项措施来评估规则的质量。提升表示规则的强度，相对于先前和随后的共同发生，给予他们个人的支持。它提供了有关改进的信息，以及由此产生的后果概率的增加。提升定义如下。

（规则支持）/（支持（先行）*支持（后续））

这也可以定义为项目组合的置信度除以结果的支持。因此，在我们的牛奶示例中，假设40％的客户购买橙汁，其改进将是：

30％/（40％* 90％）

这是0.83 - 小于1的改进。

任何改善小于1的规则并不表示真正的交叉销售机会，无论其支持和信心有多高，因为它实际上提供的预测购买能力低于随机机会。

使用此子句可以创建编辑视图 。编辑视图是单表视图，它从基表中选择所有行并显示基表列的子集。您可以使用版本控制视图在升级等管理操作期间将应用程序与DDL更改隔离到基表。您可以通过查询USER_ ， ALL_和DBA_EDITIONING_VIEW数据字典视图来获取有关现有编辑视图与其基表的关系的信息。

编辑视图的注释

编辑视图在几个重要方面与传统视图不同：

  • 编辑视图仅用于为表中的列子集选择和提供别名。因此，用于创建编辑视图的语法比用于创建传统视图的语法更受限制。即使您指定FORCE ，任何违反后续限制的行为都会导致视图创建失败。

  • 您可以在编辑视图上创建DML触发器。在这种情况下，数据库将编辑视图视为触发器的基础对象。当DML操作以编辑视图本身为目标时，会触发此类触发器。如果DML操作以基表为目标，则不会触发它们。

  • 您无法在编辑视图上创建INSTEAD OF触发器。

版本化视图的限制

编辑视图受以下限制：

  • 在任何版本中，您只能为任何单个表创建一个版本视图。

  • 您不能指定object_view_clause ， XMLType_view_clause或BEQUEATH子句。

  • 您无法在版本控制视图上定义WITH CHECK OPTION约束。

仅当在应用程序根目录中创建视图时，此子句才适用。这种类型的视图称为应用程序公共对象，其数据可以与属于应用程序根目录的应用程序PDB共享。要确定如何共享视图数据，请指定以下共享属性之一：

有关可更新视图的说明

以下说明适用于可更新视图：

可更新视图是可用于插入，更新或删除基表行的视图。您可以创建一个本质上可更新的视图，或者您可以在任何视图上创建一个INSTEAD OF触发器以使其可更新。

要了解是否可以修改固有可更新视图的列以及以何种方式修改，请查询USER_UPDATABLE_COLUMNS数据字典视图。此视图显示的信息仅对固有可更新视图有意义。要使视图本身可更新，必须满足以下条件：

  • 视图中的每个列都必须映射到单个表的列。例如，如果视图列映射到TABLE子句（未引入的集合）的输出，则视图本身不可更新。

  • 视图不得包含以下任何构造：

    • 一组运算符
    • DISTINCT运算符
    • 聚合或分析函数
    • GROUP BY ， ORDER BY ， MODEL ， CONNECT BY或START WITH子句
    • SELECT列表中的集合表达式
    • SELECT列表中的子查询
    • 指定WITH READ ONLY子查询
    • 如Oracle数据库管理员指南中所述 ，加入了一些例外情况

  • 此外，如果固有可更新视图包含伪列或表达式，则无法使用引用任何这些伪列或表达式的UPDATE语句更新基表行。

  • 如果希望连接视图可更新，则必须满足以下所有条件：

    • DML语句必须仅影响连接下的一个表。

    • 对于INSERT语句，不能WITH CHECK OPTION创建视图，并且插入值的所有列必须来自密钥保留表 。密钥保留表是基表中的每个主键或唯一键值在连接视图中也是唯一的。

    • 对于UPDATE语句，不能WITH CHECK OPTION创建视图，并且必须从保留密钥的表中提取更新的所有列。

    • 对于DELETE语句，如果连接导致多个密钥保留表，则Oracle数据库将从FROM子句中指定的第一个表中删除，无论该视图是否是WITH CHECK OPTION创建的。

多租户环境中跟踪事件的管理

您应该了解在多租户环境中如何启用跟踪事件。

  • 当前会话的跟踪事件：在多租户环境中，从根或可插拔数据库（PDB）运行ALTER SESSION SET EVENTS SQL语句可以跟踪当前用户会话。如果从一个PDB切换到另一个PDB（通过使用ALTER SESSION SET CONTAINER语句），则仍然为新PDB启用跟踪。您无法在多租户容器数据库（CDB）中为单个PDB启用跟踪;它适用于所有PDB和根。请记住，必须具有ALTER SESSION SET CONTAINER系统特权才能从一个PDB移动到另一个PDB。

  • 跟踪所有数据库会话的事件：在多租户环境中，从根或特定PDB运行ALTER SYSTEM SET EVENTS语句可以跟踪容器数据库中的所有PDB。

在多租户环境中禁用跟踪事件

您应该了解在多租户环境中如何启用跟踪事件。

  • 当前会话的跟踪事件：在多租户环境中，从根或PDB运行ALTER SESSION SET EVENTS SQL语句会禁用当前用户会话的跟踪。如果从一个PDB切换到另一个PDB（通过使用ALTER SESSION SET CONTAINER语句），则仍然会禁用新PDB的跟踪。您无法禁用CDB中单个PDB的跟踪;它适用于所有PDB和根。请记住，必须具有ALTER SESSION SET CONTAINER系统特权才能从一个PDB移动到另一个PDB。

  • 跟踪所有数据库会话的事件：在多租户环境中，从根或特定PDB运行ALTER SYSTEM SET EVENTS语句会禁用对CDB中所有PDB的跟踪。

如果您没有使用DBCA创建单实例数据库的预配置映像（如上一节中所述），请执行以下步骤以完成转换：

  2. 通过使用REUSE关键字运行CREATE CONTROLFILE SQL语句重新创建控制文件，并根据Oracle RAC配置的需要指定MAXINSTANCES和MAXLOGFILES等。MAXINSTANCES建议默认值为32。

  3. 关闭数据库实例。

  4. 如果您的单实例数据库使用SPFILE，则使用以下SQL语句从SPFILE创建临时参数文件（PFILE）：

    CREATE PFILE = ' pfile_name从SPFILE = '' spfile_name '

  5. 将CLUSTER_DATABASE参数设置为TRUE ，并使用sid . parameter = value将INSTANCE_NUMBER参数设置为每个实例的唯一值sid . parameter = value语法。

    如果优化了单实例数据库的内存使用量，则调整系统全局区域（SGA）的大小，以避免在转换为Oracle RAC时进行交换和分页。您应该进行此调整，因为Oracle RAC需要大约350个字节用于每个缓冲区以容纳全局缓存服务（GCS）。例如，如果您有10,000个缓冲区，那么Oracle RAC需要大约350乘以10,000个字节的更多内存。因此，通过相应地更改DB_CACHE_SIZE和DB_nK_CACHE_SIZE参数来调整SGA的大小。

  6. 使用步骤4中创建的PFILE启动数据库实例。

  7. 如果您的单实例数据库使用自动撤消管理，则使用CREATE UNDO TABLESPACE SQL语句为每个其他实例创建一个撤消表空间。

  8. 创建每个附加实例至少有两个重做日志的重做线程。使用ALTER DATABASE SQL语句启用新的重做线程。然后，关闭数据库实例。

  9. 将Oracle密码文件从初始节点或从您工作的节点复制到群集数据库将具有实例的其他节点上的相应位置。为每个其他实例适当地替换每个密码文件中的ORACLE_SID名称。

  10. 将REMOTE_LISTENER参数设置为单客户端访问名称（SCAN）和端口。

  11. 配置数据库和实例，并为地址表项的网络服务条目LOCAL_LISTENER每个实例，为REMOTE_LISTENER中tnsnames.ora文件，并且复制tnsnames.ora文件中的所有节点。

  12. 从PFILE创建SPFILE。

  13. 创建包含以下条目的$ORACLE_HOME/dbs/init sid .ora文件，其中spfile_path_name是SPFILE的完整路径名：

    spfile = 'spfile_path_name '

    SQL>开始？/rdbms/admin/catclust.sql

    srvctl启动数据库-d db_name

使用rconfig实用程序将单实例数据库转换为Oracle RAC时，会发生以下更改：

  • 在转换期间， rconfig将目标Oracle RAC数据库置于归档日志模式，并为数据库启用归档。如果您不打算使用存档日志空间，则可以在转换完成后禁用存档日志记录。

1.4.1 SQL调优任务

调优会话的细节取决于许多因素，包括您是主动调整还是被动调整。

在主动SQL调优中 ，您经常使用SQL Tuning Advisor来确定是否可以使SQL语句更好地执行。在反应式SQL调优中 ，您可以更正用户遇到的与SQL相关的问题。

无论您是主动还是被动地进行调优，典型的SQL调优会话都涉及以下所有或大部分任务：

  1. 识别高负载SQL语句

    查看过去的执行历史记录，以查找负责大量应用程序工作负载和系统资源的语句。

  2. 收集与性能相关的数据

    优化程序统计信息对SQL调优至关重要。如果这些统计信息不存在或不再准确，则优化程序无法生成最佳计划。与SQL性能相关的其他数据包括语句访问的表和视图的结构，以及语句可用的任何索引的定义。

  3. 确定问题的原因

    通常，SQL性能问题的原因包括：

    • 设计效率低下的SQL语句

      如果编写SQL语句以便执行不必要的工作，那么优化器无法提高其性能。低效设计的例子包括

      • 忽略添加连接条件 ，这会导致笛卡尔连接

      • 使用提示将大表指定为连接中的驱动表

      • 指定UNION而不是UNION ALL

      • 为外部查询中的每一行执行子查询

    • 次优的执行计划

      查询优化器 （也称为优化器 ）是内部软件，用于确定哪个执行计划最有效。有时，优化器会选择具有次优访问路径的计划，这是数据库从数据库中检索数据的方法。例如，具有低选择性的查询谓词的计划可以在大表而不是索引上使用全表扫描 。

      您可以将执行最佳SQL语句的执行计划与语句的计划进行次优比较。这种比较以及诸如数据量变化之类的信息可以帮助确定性能下降的原因。

    • 缺少SQL访问结构

      缺少SQL访问结构（例如索引和物化视图）是SQL性能欠佳的典型原因。最佳访问结构集可以将SQL性能提高几个数量级。

    • 过时的优化程序统计信息

      当统计维护操作（自动或手动）无法跟上DML引起的对表数据的更改时， DBMS_STATS收集的统计信息可能会变得陈旧。由于表上的陈旧统计信息无法准确反映表数据，因此优化程序可以根据错误信息做出决策并生成次优执行计划。

    • 硬件问题

      次优性能可能与内存，I / O和CPU问题有关。

  4. 定义问题的范围

    解决方案的范围必须与问题的范围相匹配。考虑数据库级别的问题和语句级别的问题。例如，共享池太小，这会导致游标快速老化，从而导致许多硬解析。使用初始化参数来增加共享池大小可以修复数据库级别的问题并提高所有会话的性能。但是，如果单个SQL语句未使用有用的索引，则更改整个数据库的优化程序初始化参数可能会损害整体性能。如果单个SQL语句出现问题，那么适当范围的解决方案只能通过此语句解决此问题。

  5. 为次优执行SQL语句实施更正操作

    这些行为因环境而异。例如，您可以重写SQL语句以提高效率，通过重写语句以使用绑定变量来避免不必要的硬解析。您还可以使用equijoins，从WHERE子句中删除函数，并将复杂的SQL语句分解为多个简单语句。

    在某些情况下，您不是通过重写语句而是通过重构模式对象来提高SQL性能。例如，您可以索引新的访问路径 ，或者对连锁索引中的列重新排序。您还可以对表进行分区，引入派生值，甚至更改数据库设计。

  6. 防止SQL性能回归

    要确保最佳SQL性能，请验证执行计划是否继续提供最佳性能，并选择更好的计划（如果可用）。您可以使用优化程序统计信息，SQL配置文件和SQL计划基准来实现这些目标。

SQL Tuning Advisor是内部诊断软件，可识别有问题的SQL语句，并建议如何提高语句性能。

  • 构建SQL配置文件

  • 标识适合于次优计划的SQL语句

1.4.2.1.4自动索引

Oracle数据库可以持续监控应用程序工作负载，自动创建和管理索引。

注意：

有关不同版本和服务支持哪些功能的详细信息，请参阅“ Oracle数据库许可信息用户手册 ” 。

手动创建索引需要深入了解数据模型，应用程序和数据分布。DBA通常会选择创建哪些索引，然后从不修改他们的选择。结果，失去了改进的机会，并且不必要的索引可能成为性能责任。自动索引管理通过执行以下任务解决了此问题：

  • 不断监控工作量

  • 创建新索引

  • 重建索引，然后将其标记为不可用或不可见

  • 删除索引

  • 检查自动索引管理对性能的影响

索引功能实现为自动任务，以固定间隔在后台运行。在每次迭代时，该任务执行以下基本步骤：

  1. 根据列和列组使用情况标识候选索引。

  2. 在不可用和不可见模式下创建一组候选索引。这些索引不占用存储空间。

  3. 查询先前执行的语句的优先级列表，以确定候选索引是否值得重建。

  4. 编译语句以确定优化程序是否会选择新索引，然后重建优化程序选择的索引。

  5. 执行使用自动索引的语句

  6. 执行以下任一操作：

    • 当语句显着改善其性能时，将索引标记为可见。只有在验证并将索引标记为可见之后，数据库才会更改工作负载中语句的计划。在此之前，数据库不会使游标无效并继续使用旧计划。

    • 标记索引在提供不足的性能优势时无法使用。当使用其他报表的索引的概率较低或存在空间压力时，此操作以延迟方式发生。

  7. 使用SQL计划管理避免回归。索引可能会使一个语句受益，但会导致第二个语句中的性能回归。在这种情况下，数据库通过将索引标记为可见来优化第一个语句。为了防止第二个语句的回归，数据库使用SQL计划管理来保护它。

  8. 删除长时间未使用的索引。

您可以通过在DBMS_AUTO_INDEX包中执行以下过程来启用自动索引：

EXEC DBMS_AUTO_INDEX.CONFIGURE（'AUTO_INDEX_MODE'，'IMPLEMENT'）

您还可以使用DBMS_AUTO_INDEX包报告自动任务并配置设置，例如保留未使用的索引的时间。

25.3.3.2跳过损坏块时的含义

在跳过损坏的块时，查询可以在某些情况下返回不同的结果。

如果索引和表未同步，则在一个查询仅探测索引并且后续查询同时探测索引和表的情况下， SET TRANSACTION READ ONLY事务可能不一致。如果表块被标记为损坏，则两个查询返回不同的结果，从而破坏了只读事务的规则。解决此问题的一种方法是不要跳过SET TRANSACTION READ ONLY事务中的损坏。

选择链接的行时会出现类似的问题。对同一行的查询可能会也可能不会访问损坏，从而产生不同的结果。

25.4.2示例：检测腐败

一个示例说明了使用CHECK_OBJECT过程检测损坏。

CHECK_OBJECT过程检查指定的对象，并使用有关损坏和修复指令的信息填充修复表。如果要检查对象的一部分，可以选择指定范围，分区名称或子分区名称。

验证包括检查对象中之前未被标记为已损坏的所有块。对于每个块，检查事务和数据层部分的自身一致性。在CHECK_OBJECT期间，如果遇到具有损坏的缓冲区高速缓存标头的块，则跳过该块。

以下是为scott.dept表执行CHECK_OBJECT过程的scott.dept 。 

SET SERVEROUTPUT ON DECLARE num_corrupt INT; BEGIN num_corrupt：= 0; DBMS_REPAIR.CHECK_OBJECT（SCHEMA_NAME =>'SCOTT'，OBJECT_NAME =>'DEPT'，REPAIR_TABLE_NAME =>'REPAIR_TABLE'，CORRUPT_COUNT => num_corrupt）; DBMS_OUTPUT.PUT_LINE（'number corrupt：'|| TO_CHAR（num_corrupt））;结束; /

数字腐败：1

SELECT OBJECT_NAME，BLOCK_ID，CORRUPT_TYPE，MARKED_CORRUPT，CORRUPT_DESCRIPTION，REPAIR_DESCRIPTION FROM REPAIR_TABLE; OBJECT_NAME BLOCK_ID CORRUPT_TYPE MARKED_COR ------------------------------ ---------- ------ ------ ---------- CORRUPT_DESCRIPTION --------------------------------- --------------------------------------------- REPAIR_DESCRIPTION ---- -------------------------------------------------- ------------------------ DEPT 3 1 FALSE kdbchk：由不存在的事务表锁定的行= 0 slot = 0 lockid = 32 ktbbhitc = 1 mark阻止软件腐败

已损坏的块尚未标记为已损坏，因此这是提取任何有意义数据的时间。块被标记为损坏后，必须跳过整个块。

25.4.3示例：修复损坏的块

一个示例说明使用FIX_CORRUPT_BLOCKS过程修复损坏的块。

使用FIX_CORRUPT_BLOCKS过程根据CHECK_OBJECT过程生成的修复表中的信息修复指定对象中的损坏块。在更改块之前，将检查块以确保块仍然损坏。通过标记块软件损坏来修复损坏的块。执行修复时，修复表中的关联行将使用时间戳进行更新。

示例说明了使用DUMP_ORPHAN_KEYS过程查找指向损坏数据块的索引条目。

DUMP_ORPHAN_KEYS过程报告指向损坏数据块中的行的索引条目。对于每个索引条目，将在指定的孤立键表中插入一行。必须先前已创建孤立密钥表。
