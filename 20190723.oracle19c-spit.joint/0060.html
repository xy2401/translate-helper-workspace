5.2.4关于分区模型

引入分区模型来组织和表示多个模型。

Oracle Data Mining支持构建持久的Oracle Data Mining分区模型。分区模型将单个模型组织并表示为单个模型实体中的分区，使用户能够轻松构建和管理针对独立数据片段定制的模型。持久意味着分区模型具有磁盘表示。该产品管理分区模型的组织，并简化了对分区模型进行评分的过程。在评分时，您必须将分区列包含在USING子句中。

分区名称，键值和分区模型的结构在ALL_MINING_MODEL_PARTITIONS视图中可见。

5.2.4.1分区模型构建过程

要构建分区模型，Oracle Data Mining需要分区键。分区键通过设置表中的构建设置进行设置。

分区键是来自输入数据集的一个或多个列（最多16个）的逗号分隔列表。分区键基于分区键的离散值水平地对输入数据进行切片。也就是说，分区是作为列表值执行的，而不是针对连续值的范围分区。分区键仅支持数据类型为NUMBER和VARCHAR2 。

在构建过程中，输入数据集根据指定键的不同值进行分区。每个数据切片（唯一键值）都会产生自己的模型分区。此结果模型分区不是独立的，并且作为独立模型不可见。分区模型的最大分区数的默认值为1000分区。您还可以设置不同的最大分区值。如果输入数据集中的分区数超过定义的最大值，Oracle Data Mining将引发异常。

分区模型组织所有分区和分区特定功能共有的功能。常见功能包括以下元数据：

  • 型号名称

  • 挖掘功能

  • 挖掘算法

  • 所有分区引用的所有挖掘模型属性的超集（签名）

  • 一组通用的用户定义列转换

  • 任何用户指定的或默认的构建设置，都被解释为全局。例如，自动数据准备（ADP）设置

5.2.4.2分区模型中的DDL

通过以下DDL操作维护分区模型：

  • 删除模型或删除分区

  • 添加分区

5.2.4.2.1丢弃模型或丢弃分区

Oracle Data Mining支持为给定的分区名称删除单个模型分区。

如果只剩下一个分区，则无法显式删除该分区。相反，您必须在删除分区之前添加其他分区，或者您可以选择删除模型本身。删除分区模型时，将在单个原子操作中删除所有分区。从性能角度来看，由于DROP_PARTITION选项的有效行为，Oracle建议使用DROP_PARTITION后跟ADD_PARTITION而不是利用REPLACE选项。

5.2.4.2.2添加分区

Oracle Data Mining支持将单个分区或多个分区添加到现有分区模型。

添加基于输入数据集和现有分区模型的名称进行。该操作将输入数据集和现有分区模型作为参数。从输入数据集中提取分区键，并根据输入数据集构建模型分区。这些分区将添加到分区模型中。如果新分区的分区键与模型中的现有分区冲突，则可以从以下三种方法中进行选择以解决冲突：

  • ERROR ：终止ADD操作而不添加任何分区。

  • REPLACE ：替换找到冲突键的现有分区。

  • IGNORE ：消除具有冲突密钥的行。

如果输入数据集包含多个键，则操作将创建多个分区。如果模型中的分区总数增加到超过创建模型时指定的用户定义的最大值，则会出现错误。分区数的默认阈值为1000 。

5.2.4.3分区模型评分

了解分区模型的评分。

分区模型的评分与非分区模型的评分相同。数据挖掘功能的语法保持不变，但扩展为您提供可选提示。可选提示可能会影响查询的性能，该查询涉及对分区模型进行评分。

对于对分区模型进行评分，在构建分区键期间使用的签名列必须存在于评分数据集中。这些列组合在一起形成一个唯一的分区键。然后将唯一键映射到特定的底层模型分区，并使用标识的模型分区对该行进行评分。

评分所需的分区对象在查询执行期间按需加载，并根据系统全局区域（SGA）内存进行老化。

  • -无声

3.1.2分类和回归的数据集

了解数据集如何用于培训和测试模型。

3.1.3评分要求

大多数数据挖掘模型可以应用于称为评分的过程中的单独数据。Oracle Data Mining支持评分操作分类， 回归， 异常检测， 聚类，和特征提取。

评分过程将评分数据中的列名称与用于构建模型的列的名称相匹配。评分过程不要求所有列都存在于评分数据中。如果数据类型不匹配，Oracle Data Mining会尝试执行类型强制。例如，如果名为PRODUCT_RATING的列在训练数据中为VARCHAR2但在评分数据中为NUMBER ，则Oracle数据挖掘会有效地应用TO_CHAR()函数进行转换。

测试或评分数据中的列必须经历相同的操作转换为构建数据中的相应列。例如，如果构建数据中的AGE列已从数字转换为值CHILD ， ADULT和SENIOR ，则评分数据中的AGE列必须经历相同的转换，以便模型可以正确评估它。

注意：

Oracle Data Mining可以嵌入用户指定的转换模型中的指令，并在应用模型时重新应用它们。当转换指令嵌入到模型中时，您无需为测试或评分数据集指定它们。

Oracle Data Mining也支持自动数据准备（ADP）。启用ADP后，算法所需的转换将自动执行，并与任何用户指定的转换一起嵌入到模型中。

为直接NFS客户端创建oranfstab文件

Direct NFS使用配置文件oranfstab来确定可用的挂载点。

使用Direct NFS Client为要访问的每个NFS服务器创建一个具有以下属性的oranfstab文件：

  • 服务器

    NFS服务器名称。

  • 本地

    数据库主机上最多有四个路径，由IP地址或名称指定，使用在数据库主机上运行的ifconfig命令显示。

  • 路径

    最多四个到NFS服务器的网络路径，由IP地址或名称指定，如NFS服务器上的ifconfig命令所示。

  • 出口

    NFS服务器的导出路径。

  • 安装

    导出卷的相应本地安装点。

  • mnt_timeout

    指定（以秒为单位）Direct NFS Client在超时之前应等待成功安装的时间。此参数是可选的。默认超时为10分钟（600）。

  • nfs_version

    指定Direct NFS Client使用的NFS协议版本。可能的值包括NFSv3，NFSv4，NFSv4.1和pNFS。默认版本为NFSv3。如果选择NFSv4.x，则必须在oranfstab为nfs_version配置值。如果要将Direct NFS与并行NFS一起使用，请将nfs_version指定为pNFS 。

  • security_default

    指定适用于服务器条目的所有导出的NFS服务器路径的默认安全模式。此参数是可选的。 sys是默认值。请参阅security参数支持的安全级别的security_default参数说明。

  • 安全

    指定安全级别，以使用带有Direct NFS Client的Kerberos身份验证协议启用安全性。可以根据导出安装对指定此可选参数。security_default和security参数支持的安全级别为：

    • sys ：UNIX级别安全性AUTH_UNIX基于用户标识符（UID）和组标识符（GID）值的身份验证。这是安全性参数的默认值。
    • krb5 ：Direct NFS使用普通Kerberos身份验证运行。服务器被认证为它声称的真实服务器。
    • krb5i ：Direct NFS运行Kerberos身份验证和NFS完整性。对服务器进行身份验证，并检查每个邮件传输的完整性。
    • krb5p ：Direct NFS运行Kerberos身份验证和NFS隐私。服务器已通过身份验证，并且所有数据都已完全加密。

    security参数（如果已指定）优先于security_default参数。如果未指定这些参数，则sys是默认身份验证。

    对于NFS服务器Kerberos安全设置，请查看相关的NFS服务器文档。对于Kerberos客户端设置，请查看相关的操作系统文档。

  • dontroute

    指定传出消息不应由操作系统路由，而是使用它们绑定的IP地址发送。

    注意：

    dontroute选项是POSIX选项，有时在同一子网中具有多个路径的Linux系统上不起作用。

  • 管理

    使Direct NFS Client能够使用管理接口进行SNMP查询。如果SNMP在NFS服务器上的单独管理接口上运行，则可以使用此参数。默认值是服务器参数值。

  • 社区

    指定用于SNMP查询的社区字符串。默认值为public 。

选择'真'来自员工WHERE department_id NOT IN（10,20）;

SELECT'Frue'FROM employees WHERE department_id NOT IN（10,20，NULL）;

department_id！= 10 AND department_id！= 20 AND department_id！= null

更改资源成本

目的

使用ALTER RESOURCE COST语句指定或更改Oracle数据库计算会话中使用的总资源成本的公式。

虽然Oracle数据库监视其他资源的使用，但只有语法中显示的四种资源可能会导致会话的总资源成本。

此语句允许您对四个资源应用权重。然后，Oracle数据库将权重应用于为配置文件指定的这些资源的值，以建立计算总资源成本的公式。您可以使用CREATE PROFILE语句的COMPOSITE_LIMIT参数限制会话的此成本。如果会话的资源成本超出限制，则Oracle数据库将中止会话并返回错误。如果使用ALTER RESOURCE COST语句更改分配给每个资源的权重，则Oracle数据库会使用这些新权重来计算所有当前和后续会话的总资源成本。

语义

Oracle数据库通过首先将会话中使用的每个资源的数量乘以资源的权重，然后将所有四个资源的产品相加，来计算总资源成本。对于任何会话，此成本受用户配置文件中COMPOSITE_LIMIT参数的值限制。产品和总成本均以称为服务单位的单位表示 。

CPU_PER_SESSION

使用此关键字将权重应用于CPU_PER_SESSION资源。

CONNECT_TIME

使用此关键字将权重应用于CONNECT_TIME资源。

使用此子句将权重应用于LOGICAL_READS_PER_SESSION资源。逻辑读取包括从内存和磁盘读取的块。

使用此子句将权重应用于PRIVATE_SGA资源。仅当您使用共享服务器体系结构并在SGA中为会话分配私有空间时，此限制才适用。

整数

指定每个资源的权重。您为每个资源分配的权重决定了该资源的使用对总资源成本的贡献程度。如果您没有为资源分配权重，则权重默认为0，并且随后使用该资源不会影响成本。您指定的权重适用于数据库中的所有后续会话。

例子

改变资源成本：示例

以下语句为资源CPU_PER_SESSION和CONNECT_TIME分配权重：

ALTER RESOURCE COST CPU_PER_SESSION 100 CONNECT_TIME 1;

权重为会话建立此成本公式：

cost =（100 * CPU_PER_SESSION）+（1 * CONNECT_TIME）

在此示例中， CPU_PER_SESSION和CONNECT_TIME的值是DEFAULT配置文件中的值或会话用户的配置文件中的值。

由于前面的语句没有为资源LOGICAL_READS_PER_SESSION和PRIVATE_SGA赋予权重，因此这些资源不会出现在公式中。

如果为用户分配了COMPOSITE_LIMIT值为500的配置文件，则只要cost超过500，会话就会超过此限制。例如，使用0.04秒CPU时间和101分钟经过时间的会话超出限制。使用0.0301秒CPU时间和200分钟经过时间的会话也超出了限制。

随后可以使用另一个ALTER RESOURCE语句更改权重：

更改资源成本LOGICAL_READS_PER_SESSION 2 CONNECT_TIME 0;

这些新权重建立了一个新的成本公式：

cost =（100 * CPU_PER_SESSION）+（2 * LOGICAL_READ_PER_SECOND）

其中， CPU_PER_SESSION和LOGICAL_READS_PER_SECOND的值是DEFAULT配置文件中的值或此会话用户的配置文件中的值。

此ALTER RESOURCE COST语句以下列方式更改公式：

  • 该语句省略了CPU_PER_SESSION资源的权重。该资源已经分配了权重，因此资源保留在公式中并具有原始权重。

  • 该语句为LOGICAL_READS_PER_SESSION资源分配权重，因此该资源现在显示在公式中。

  • 该语句为CONNECT_TIME资源分配权重0，因此该资源不再出现在公式中。

  • 该语句省略了PRIVATE_SGA资源的权重。该资源尚未分配权重，因此资源仍未显示在公式中。

5.2开发Oracle Sharding应用程序

Sharding为最苛刻的OLTP应用程序提供线性可扩展性和完整的故障隔离，而不会影响Oracle数据库的企业质量：严格的一致性，SQL的全部功能，开发人员对JSON的敏捷性，安全性，高可用性，备份和恢复，生命周期管理等等。

Sharding是一种数据层架构，其中数据在独立数据库中水平分区。这种配置中的每个数据库都称为分片。所有分片一起组成一个逻辑数据库，称为分片数据库（SDB）。

分片适用于适用于分片数据库体系结构的OLTP应用程序。特别：

  • 应用程序必须具有明确定义的数据模型和数据分发策略，并且必须主要使用分片密钥访问数据。分片键的示例包括客户ID，帐号，country_id等。

  • 数据模型应该是具有单个根表的分层树结构。Oracle Sharding支持层次结构中的任意数量的级别。

  • 对于系统管理的分片方法，分片键必须基于具有高基数的列;此列中唯一值的数量必须远大于分片数量。例如，客户ID是分片密钥的良好候选者，而美国州名称则不是。

  • 分片键应该非常稳定;它的价值几乎永远不会改变。

  • 分片键必须存在于所有分片表中。这允许基于分片键创建一系列等分区表。分片键必须是根表主键的前导列。

  • 应使用分片键执行表系列中表之间的连接。

  • 复合分片可实现两级分片 - 一个按列表或范围，另一个按一致哈希。这是通过提供两个键的应用程序实现的：超级分片键和分片键。

  • 所有需要高性能和故障隔离的数据库请求只能访问与分片键的单个值相关联的数据。应用程序必须在建立数据库连接时提供分片键。如果是这种情况，请求将直接路由到相应的分片。

    只要它们都与同一个分片密钥相关，就可以在同一个会话中执行多个请求。此类事务通常访问10行或100行。单一分片交易的示例包括订单输入，查询和更新客户的账单记录，以及查询和更新订户的文档。

  • 必须从查询协调器执行必须访问与分片密钥的多个值相关联的数据或者分片密钥的值未知的数据库请求，该查询协调器在多个分片中协调查询的并行执行。

  • 应用程序使用Oracle集成连接池（UCP，OCI，ODP.NET，JDBC）连接到分片数据库。

  • 必须使用单独的连接池进行直接路由和代理路由。对于直接路由，必须为读写和只读工作负载创建单独的全局服务。仅在使用Data Guard复制时才会出现这种情况。对于代理路由，请在分片目录数据库上使用GDS $ CATALOG服务。

H使用ALTERNATE属性配置远程备用目标

从Oracle Database 12c第2版（12.2）开始，为活动目标失败而为远程备用数据库和远程同步实例创建备用日志归档目标的首选方法是使用GROUP和PRIORITY属性。

对于本地归档位置（ LOCATION= ...），如果原始归档目录由于阻止访问归档位置的磁盘或网络问题而变得不可用，则ALTERNATE属性仍用于提供高可用性。但是，为了向后兼容性，仅保留对远程日志归档目标（ SERVICE= ...）的ALTERNATE属性的使用。以下各节中提供的使用此方法的示例是创建远程同步实例的后续操作，但它们也适用于级联重做目标。

执行创建和配置远程同步实例中的步骤后，远程同步实例为通过WAN的远程站点上的终端备用数据库配置提供零数据丢失功能。要使配置保持受保护，但保护级别降低，如果与远程同步实例的通信丢失，您可以选择将终端备用数据库配置为自动成为备用目标。通过允许Oracle Data Guard直接从主服务器异步发送重做到终端备用数据库，暂时绕过远程同步实例，可以减少数据丢失的数量。

此配置使Oracle Data Guard能够继续以异步方式向终端备用boston发送重做，因为它无法再将重做直接发送到远程同步实例chicagoFS 。当远程同步实例再次可用时，Oracle Data Guard会自动重新同步远程同步实例chicagoFS并返回原始配置，其中主服务器将重做发送到远程同步实例，远程同步实例将该重做转发到终端备用数据库。同步完成后，备用目标（前一示例中的LOG_ARCHIVE_DEST_3 ）将再次变为休眠状态。

在上述情况下，使用异步重做传输直接在两个数据库之间设置ALTERNATE远程目标，因此在远程同步实例发生故障时，配置的保护级别会降低到最大性能，数据丢失在故障转移时。为了更好地防止系统或网络故障，可以配置一个额外的远程同步实例，为活动远程同步实例提供高可用性。

在以下配置中，一个远程同步实例是首选的活动远程同步实例，另一个是备用远程同步实例。如果首选远程同步实例由于某种原因失败，则配置备用远程同步实例可为配置提供持续保护，从而使配置保持最大可用性。如果主服务器在首选远程同步实例中检测到故障，则主服务器会自动开始发送到备用远程同步实例。如果首选远程同步实例然后重新建立自身，则主服务器切换回首选远程同步实例，并将备用远程同步实例重新置于备用状态。

在这些类型的配置中，主要仅使用两个远程同步实例中的一个来在任何给定时间重新分配重做。

将使用与创建和配置远程同步实例中给出的步骤相同的步骤创建第二个高可用性远程同步实例 ，然后成为现有远程同步实例的替代实例，而不是终端备用实例。完成后， chicago将配置如下参数（假设名称chicagoFS1为新的远程同步实例名称）。

13.57 ％ROWTYPE属性

%ROWTYPE属性允许您声明一条记录，该记录表示数据库表或视图的完整行或部分行。

对于完整或部分行的每个可见列，记录都有一个具有相同名称和数据类型的字段。如果行的结构发生变化，则记录的结构会相应地发生变化。使可见的可见列可见会更改使用%ROWTYPE属性声明的某些记录的结构。

记录字段不继承相应列的约束或初始值。

如果引用的字符列具有USING_NLS_COMP以外的排序规则，则不能使用%ROWTYPE属性。

目的

FEATURE_VALUE返回选择中每行的特征值。该值指的是最高值要素或指定的feature_id 。功能值以BINARY_DOUBLE形式返回。

语法选择

FEATURE_VALUE可以通过以下两种方式之一对数据进行评分：它可以将挖掘模型对象应用于数据，也可以通过执行构建和应用一个或多个瞬态挖掘模型的分析子句来动态挖掘数据。选择语法或分析语法 ：

  • 语法 - 使用第一种语法使用预定义的模型对数据进行评分。提供特征提取模型的名称。

  • 分析语法 - 使用分析语法在没有预定义模型的情况下对数据进行评分。包括INTO n ，其中n是要提取的mining_analytic_clause数，以及mining_analytic_clause ，它指定是否应为多个模型构建分区数据。mining_analytic_clause支持query_partition_clause和order_by_clause 。（参见“ analytic_clause :: = ” 。）

在对分区模型进行评分时， FEATURE_VALUE函数的语法可以使用可选的GROUPING提示。请参阅GROUPING提示 。

mining_attribute_clause

mining_attribute_clause标识要用作评分预测变量的列属性。使用分析语法调用函数时，此数据还用于构建瞬态模型。mining_attribute_clause行为与PREDICTION函数的描述相同。（参见“ mining_attribute_clause ” 。）

    ＃/ sbin / sysctl --system

    ＃/ sbin / sysctl -a
  4. 重新启动计算机，或运行sysctl --system以使/etc/sysctl.d/97-oracle-database-sysctl.conf文件中的更改在活动内核内存中可用。

  • 如果使用了Oracle预安装RPM，则内核参数设置位于/etc/sysctl.d/99-oracle-database-server-19c-preinstall-sysctl.conf文件中。

  • 仅包含要更改的内核参数值的行。对于信号量参数（ kernel.sem ），您必须指定所有四个值。如果任何当前值大于最小值，则指定较大的值。
