30 FAN事件的Oracle JDBC支持

从Oracle Database 12 c第2版（12.2.0.1）开始，Oracle JDBC驱动程序支持Oracle RAC快速应用程序通知（FAN）事件，用于计划内和计划外中断。这有助于第三方连接池利用Oracle RAC功能实现高可用性。不使用Oracle通用连接池（UCP）或WebLogic Server的Java应用程序现在可以利用此支持。例如，Oracle RAC服务器端的滚动升级等方案不会在应用程序中导致JDBC错误。

注意：

虽然Oracle JDBC驱动程序现在支持FAN事件，但Oracle UCP为所有FAN事件提供了更全面的支持。

  • 计划维护

30.2计划维护的安全排水API

对于计划的Oracle RAC维护，JDBC驱动程序支持安全耗尽API列表，这些API是与第三方Java连接池进行额外握手或集成工作所必需的。这些API用作排放点 ，驱动程序可以安全地关闭受计划维护影响的任何连接，而不会导致应用程序可见错误。以下是驱动程序FAN支持的安全排放API列表：

  • java.sql中。Connection.isValid（int timeout）

  • oracle.jdbc。OracleConnection.pingDatabase（）

  • oracle.jdbc。OracleConnection.pingDatabase（int timeout）

  • oracle.jdbc。OracleConnection.endRequest（）

  • 所有标准JDBC和Oracle JDBC扩展EXECUTE***调用Statement ， PreparedStatement和CallableStatement接口

对于标准JDBC和Oracle JDBC扩展EXECUTE***调用，执行的SQL命令字符串必须包含以下SQL提示作为SQL字符串中的第一个非注释标记：

/ * + CLIENT_CONNECTION_VALIDATION * /

合格的SQL被视为连接验证SQL。例如：

/ * + CLIENT_CONNECTION_VALIDATION * / SELECT 1 FROM DUAL

通常，第三方连接池会调用这些API。预计在检测到与此类调用的任何错误连接时，第三方连接池将关闭并从池中删除相关连接，以便应用程序不会看到任何错误。当应用程序本身调用这些API时，预计应用程序正在主动验证底层连接，并将关闭并删除检测到的任何错误连接。

30.3安装和配置用于FAN事件支持的Oracle JDBC驱动程序

Oracle JDBC驱动程序通过检查连接到的数据库服务器，以及除了Oracle JDBC驱动程序之外是否在应用程序环境中可用以下必需的JAR文件，自动确定是否为FAN事件启用Oracle JDBC支持：

  • simplefan.jar和ons.jar文件

    您必须从以下链接安装12.3.0.1版本的simplefan.jar和ons.jar文件，并将它们包含在CLASSPATH 。

    http://www.oracle.com/technetwork/database/application-development/jdbc/jdbc-ucp-122-3110062.html

    如果其中任何一个丢失，或者驱动程序无法加载，则禁用此功能。与第三方连接池一起使用时，这些JAR文件必须放在同一位置，连接池将在该位置检索并加载驱动程序JAR文件。

  • Oracle JDBC数据源

    您可以使用相同的典型Oracle JDBC数据源，例如oracle.jdbc.pool.OracleDataSource或oracle.jdbc.OracleDriver用于获取JDBC连接。与第三方连接池一起使用时，应用程序必须将这些类指定为连接池的连接工厂。

想要显式禁用此功能的应用程序可以将oracle.jdbc.fanEnabled属性设置为FALSE 。此属性既可用作系统属性，也可用作连接属性。对于使用通用连接池（UCP）或WebLogic Server Active GridLink（AGL）的应用程序，默认情况下此属性设置为FALSE 。否则，默认值为TRUE 。

注意：

  • 当JDBC驱动程序自动启用对FAN事件的支持时，如果在CLASSPATH上同时存在simplefan.jar和ons.jar文件，则调用getConnection方法可能会抛出异常，例如java.lang.IllegalArgumentException 。为避免这种情况，您可以执行以下任一操作：

    • 从CLASSPATH删除simplefan.jar或ons.jar 。

    • 将oracle.jdbc.fanEnabled属性设置为FALSE以显式禁用此功能。

  • 将oracle.jdbc.fanEnabled属性设置为TRUE可能无法启用Oracle JDBC支持FAN事件功能，因为该功能也取决于其他因素。

JDBC驱动程序需要对第三方连接池进行最少的配置更改或代码更改，以支持Oracle FAN事件。对于不需要任何配置更改或代码更改的连接池，假定它满足以下条件之一：

7.2.1创建不完整类型

直接或通过中间类型相互依赖的类型称为相互依赖。例如，您可能希望定义对象类型employee和department以这样的方式的一个属性employee是员工属于和一个属性的部门department是谁管理部门的员工。

如果使用箭头显示图表，显示一组相互依赖的类型之间的关系，则连接形成一个循环。要定义这样的循环依赖关系，必须对圆的至少一个段使用REF 。

例如，您可以定义例7-7中显示的类型。

示例7-7创建从属对象类型

- 需要Ex。 7-1和密码CONNECT user1  - 输入密码ALTER SESSION SET PLSQL_WARNINGS ='enable：all'; CREATE TYPE部门; //占位符 / CREATE TYPE员工AS OBJECT（名称VARCHAR2（30），部门REF部门，supv REF员工）; / CREATE TYPE emp_list AS员工表; / CREATE TYPE部门AS OBJECT （名称VARCHAR2（30），mgr REF员工，员工emp_list）; /

这是一组合法的相互依赖类型和SQL DDL语句的合法序列。Oracle数据库编译它没有错误。

请注意， 示例7-7中的代码会两次创建类型department 。第一个语句是一个可选的，不完整的department声明，作为employee的REF属性指向的占位符。声明不完整，因为它省略了AS OBJECT短语并且没有列出任何属性或方法。这些在稍后的完整声明中指定，以完成该类型。与此同时， department被创建为不完整的对象类型。这使得employee的编译能够顺利进行。

如果不创建不完整类型作为占位符，则引用缺少类型的类型仍会编译，但编译会继续出错。例如，如果department根本不存在，Oracle数据库会将其创建为不完整类型并编译有错误的employee 。然后， employee将在下次某些操作访问时重新编译。这一次，如果已经创建了它所依赖的所有类型并且满足了它的依赖性，那么它将编译而不会出错。

不完整类型还允许您创建包含尚未创建的子类型的REF属性的类型。要创建这样的超类型，首先要创建一个不完整类型的要引用的子类型。创建超类型后创建完整的子类型。

7.2.2完成不完整类型

当您创建了由不完整类型引用的所有类型时，请完成不完整类型的声明，因为不再需要它保持不完整。

完成该类型会重新编译它并使系统能够释放各种锁。使用CREATE TYPE语句完成类型。

  • 执行CREATE TYPE语句，该语句指定TYPE的属性和方法，如例7-7末尾所示。

此外，您必须完成数据库为您创建的任何不完整类型。如果，如上一节所述，您没有明确地将department创建为不完整类型，那么数据库就是这样做的。在这种情况下，您仍需要完成它。

您必须将不完整的对象类型作为对象类型完成：您无法将对象类型作为集合类型（嵌套表类型或数组类型）完成。唯一的选择是放弃类型。

7.2.3手动重新编译类型

如果创建了具有编译错误的类型，并且您尝试对其进行操作（例如创建表或插入行），则可能会收到错误。您需要在尝试操作之前重新编译该类型。您使用ALTER TYPE语句重新编译。

  • 执行ALTER TYPE typename COMPILE语句。成功编译类型后，再次尝试操作。

7.2.4使用带有类型和表依赖关系的CREATE或REPLACE TYPE

如果要替换的类型具有表或类型依赖项，则CREATE OR REPLACE TYPE语句将引发错误。这适用于对象，varrays和嵌套表类型。这也适用于涉及继承或类型组合的类型依赖项（将一种类型嵌入另一种类型）。后者可能是一种类型是另一种类型的属性的情况。

使用带有CREATE OR REPLACE TYPE语句的FORCE选项可以替换类型依赖项但不依赖于表依赖项的类型。表依赖性仍会导致错误。

  • 如果FORCE选项具有类型依赖性，请将FORCE选项与CREATE OR REPLACE TYPE语句一起使用以替换类型。

例7-8显示了由于类型依赖性而失败的CREATE OR REPLACE语句（第二个语句）。

7.2.6可替换表和列的类型依赖性

特定类型的可替换表或列不仅取决于该类型，还取决于该类型的所有子类型。

这是因为对于在类型的子类型中添加的每个属性，将隐藏列添加到表中。即使可替换表或列不包含该子类型的数据，也会添加隐藏列。

在示例7-10中 ， person_typ类型的persons表不仅依赖于person_typ ，还依赖于person_typ子类型student_typ和part_time_student_typ 。

如果尝试删除具有依赖类型，表或列的子类型，则DROP TYPE语句将返回错误并中止。因此，尝试删除part_time_student_typ会因为依赖persons表而引发错误。

如果存在依赖表或列但不包含要删除的类型的数据，则可以使用VALIDATE关键字删除该类型。VALIDATE关键字使Oracle数据库检查指定类型的实际存储实例，如果未找到，则删除该类型。这还会删除与该类型唯一的属性关联的隐藏列。

在示例7-10中 ，第一个DROP TYPE语句失败，因为part_time_student_typ具有从属表（ persons ）。但是如果persons包含part_time_student_typ实例（也没有任何其他依赖表或列），则VALIDATE关键字会导致第二个DROP TYPE语句成功。

7.1.5对象，类型和表的访问权限

对象类型仅使用EXECUTE特权。

但是，对象表使用与关系表相同的所有权限：

  • READ或SELECT允许您从表中访问对象及其属性。

  • UPDATE允许您修改表中对象的属性。

  • INSERT允许您向表中添加新对象。

  • DELETE允许您从表中删除对象。

类似的表和列权限规定了对象类型的表列的使用。

选择对象表的列不需要对象表类型的特权。但是，选择整个行对象可以。

14.2国家开发银行舰队的目的

CDB机队为许多CDB的可扩展性和集中管理提供数据库基础设施。

CDB车队在以下情况下非常有用：

  • 您必须为应用程序提供超过最大PDB数量（4096）的数量，这需要您创建多个CDB。

  • 单个配置中的不同PDB需要不同类型的服务器才能以最佳方式运行。

    例如，某些PDB可能会处理大量事务负载，而其他PDB主要用于监视，并且您希望为这些PDB提供适当的服务器资源，例如CPU，内存，I / O速率和存储系统。

  • 使用相同应用程序的不同PDB必须位于不同的位置。

跨CDB监测和诊断收集

主要CDB可以运行监控和报告在车队中的CDB上执行的应用程序。您可以在一个容器中安装监视应用程序，然后使用CDB视图和GV$视图来监视和处理整个CDB机群的诊断数据。在主要CDB中发布的跨容器查询可以在CDB机群中的所有PDB中自动执行。

软件即服务（SaaS）应用程序

在CDB机群中的不同应用程序容器中使用通用模式和通用应用程序对象，可以使用CONTAINERS子句或容器映射在CDB机群中的所有PDB上运行查询。为了确保跨CDB的通用应用程序架构，可以将应用程序安装在应用程序根目录中。

典型的用例涉及在主CDB中安装主应用程序根。应用程序根克隆驻留在机群中的每个其他CDB中。应用程序根克隆的代理PDB驻留在主应用程序根目录中。

数据库即服务（DBaaS）应用程序

主要CDB可以作为一个中心位置，您可以在其中收集和查看CDB车队中配置的所有或部分PDB的使用指标和状态。

微服务

微服务是面向服务的体系结构的专门化，用于构建灵活，可独立部署的软件系统。借助微服务，每个团队都可以部署和管理具有自定义扩展和可用性SLA的CDB机群。CDB可以使用不同的存储系统和配置设置，以满足不同类型的工作负载。主要CDB可以帮助中央DBA管理与每个单独的微服务相关联的CDB的集合。

14.4指定CDB车队成员

通过将LEAD_CDB_URI数据库属性设置为指向前导CDB的数据库链接来指定机群成员。

先决条件

您必须使用具有固定用户语义的数据库链接，这意味着用户名和密码位于链接定义中。链接不能使用连接的用户语义，在这种情况下，用户名和密码不在链接定义中。

指定CDB车队成员：

  1. 在SQL * Plus中，确保当前容器是要指定为fleet成员的CDB的根。

  2. （可选）通过运行以下查询来检查当前的LEAD_CDB_URI数据库属性：

    SELECT PROPERTY_VALUE FROM DATABASE_PROPERTIES PROPERERE_NAME ='LEAD_CDB_URI';

  3. 如果数据库链接不存在，则创建指向队列中主要CDB根目录的链接。

    数据库链接必须是固定的公共用户数据库链接。

  4. 将LEAD_CDB_URI数据库属性设置为LEAD_CDB_URI CDB的数据库链接的名称。

示例14-2指定CDB车队成员

此示例假定前导CDB为cdb1并且指向前导CDB的数据库链接不存在。它还假定网络已配置为使当前CDB可以使用lead_pod服务名称连接到cdb1 。

  1. 访问要指定为车队成员的CDB的根目录：

    ALTER SESSION SET CONTAINER = CDB $ ROOT;

  3. 将LEAD_CDB_URI属性设置为数据库链接的名称：

    ALTER DATABASE SET LEAD_CDB_URI ='dblink：LEAD_LINK';

17.1数据仓库运输概述

运输是将数据从一个系统移动到另一个系统的操作。在数据仓库环境中，最常见的传输要求是从以下位置移动数据：

  • 源系统到登台数据库或数据仓库数据库

  • 数据仓库的临时数据库

  • 数据集市的数据仓库

运输通常是ETL过程中较简单的部分之一，并且可以与过程的其他部分集成。例如，如数据仓库中的提取 ，分布式查询技术提供了一种提取和传输数据的机制。

17.2数据仓库中的运输机制简介

在仓库中传输数据有三种基本选择：

  • 使用平面文件的运输

  • 分布式运营的运输

  • 使用可传输表空间的传输

17.2.1使用平面文件进行运输

传输数据最常见的方法是使用FTP或其他远程文件系统访问协议等机制传输平面文件。使用数据仓库中的提取中讨论的技术将数据从源系统卸载或导出到平面文件中 ，然后使用FTP或类似机制将数据传输到目标平台。

由于源系统和数据仓库通常使用不同的操作系统和数据库系统，因此使用平面文件通常是在异构系统之间以最少的转换交换数据的最简单方法。但是，即使在同构系统之间传输数据时，平面文件通常也是数据传输最有效和最易于管理的机制。

17.2.2通过分布式操作进行的运输

具有或不具有网关的分布式查询可以是用于提取数据的有效机制。这些机制还将数据直接传输到目标系统，从而在一个步骤中提供提取和转换。根据对时间和系统资源的可容忍影响，这些机制非常适合于提取和转换。

与平面文件传输相反，传输的成功或失败立即被分布式查询或事务的结果识别。

17.2.3使用可传输表空间进行传输

Oracle可传输表空间是在两个Oracle数据库之间移动大量数据的最快方法。在引入可传输表空间之前，最具可伸缩性的数据传输机制依赖于移动包含原始数据的平面文件。这些机制要求将数据从源数据库卸载或导出到文件中。然后，在传输之后，将这些文件加载或导入到目标数据库中。可传输的表空间完全绕过卸载和重新加载步骤。

使用可传输的表空间，Oracle数据文件（包含表数据，索引和几乎所有其他Oracle数据库对象）可以直接从一个数据库传输到另一个数据库。此外，与导入和导出一样，可传输表空间除了传输数据外，还提供传输元数据的机制。

可传输表空间有一些限制：源系统和目标系统必须运行Oracle8 i （或更高版本），必须使用兼容的字符集，并且在Oracle Database 10g之前必须在同一操作系统上运行。有关如何在操作系统之间传输表空间的详细信息，请参见“ Oracle数据库管理员指南” 。

可传输表空间在数据仓库中最常见的应用是将数据从登台数据库移动到数据仓库，或将数据从数据仓库移动到数据集市。

将要传输的数据放入其自己的表空间中

必须将当前月份的数据放入单独的表空间中才能进行传输。在此示例中，您有一个表空间ts_temp_sales ，它包含当前月份数据的副本。使用CREATE TABLE ...AS SELECT语句，可以将当前月份的数据有效地复制到此表空间：

CREATE TABLE temp_jan_sales NOLOGGING TABLESPACE ts_temp_sales AS SELECT * FROM sales WHERE time_id BETWEEN '31 -DEC-1999'AND'01-FEB-2000';

执行此操作后，表空间ts_temp_sales将设置为只读：

ALTER TABLESPACE ts_temp_sales READ ONLY;

除非没有修改表空间的活动事务，否则无法传输表空间。将表空间设置为只读会强制执行此操作。

表空间ts_temp_sales可以是一个表空间，专门用于临时存储数据以供可传输表空间功能使用。在“ 将数据文件和导出文件复制到目标系统 ”之后 ，可以将此表空间设置为读/写，如果需要，可以删除表temp_jan_sales ，或者可以将表空间重新用于其他传输或其他传输目的。

在给定的可传输表空间操作中，传输给定表空间中的所有对象。虽然在此示例中仅传输一个表，但表空间ts_temp_sales可能包含多个表。例如，数据集市可能不仅刷新了新月的销售交易，而且还刷新了客户表的新副本。这两个表都可以在同一个表空间中传输。此外，此表空间还可以包含其他数据库对象，例如索引，这些对象也将被传输。

此外，在给定的可传输表空间操作中，可以同时传输多个表空间。这样可以更轻松地在数据库之间移动大量数据。但请注意，可传输表空间功能只能传输一组表空间，这些表空间包含一组完整的数据库对象，而不依赖于其他表空间。例如，索引不能在没有表的情况下传输，也不能在没有表的其余部分的情况下传输分区。您可以使用DBMS_TTS包来检查表空间是否可传输。

使用XML C ++类生成器

例6-1中的C代码片段演示了如何连接MySQL以及如何将行插入表中。更新连接凭据后，当使用liboramysql库而不是libmysqlclient库链接可执行文件时，此代码可以针对Oracle数据库保持不变。

虽然数据库模式和数据必须单独迁移到Oracle，尽管liboramysql库不会转换SQL语句，但迁移到Oracle数据库时会节省大量的工作量，因为不必对应用程序代码进行任何更改。

自定义C应用程序可以使用liboramysql库轻松迁移到Oracle数据库。

此外，您可以使用抽象使用libmysqlclient库并提供MySQL扩展或适配器的编程语言来迁移应用程序。这些语言包括PHP，Perl，Python和Ruby。虽然已经存在用于在C中实现的许多编程语言的本机Oracle适配器，但是将应用程序迁移到本机Oracle适配器通常需要更改应用程序代码。

安装和首次使用MySQL客户端库驱动程序

用于Oracle的MySQL客户端库驱动程序作为Linux中的liboramysql.so共享库中的文件提供，并作为Windows的oramysql.dll动态链接库（DLL）提供。该驱动程序还打包为Oracle Instant Client Basic和Basic Lite软件包的一部分，可从OTN下载。请参阅http://www.oracle.com/technetwork/topics/linuxsoft-082809.html和http://www.oracle.com/technetwork/topics/winsoft-085727.html 。

驱动程序必须安装在与Oracle客户端共享库相同的目录中，即Linux的libclntsh.so和Windows的oci.dll 。通常，必须将操作系统环境变量（Linux上的LD_LIBRARY_PATH或Windows上的PATH ）设置为包含此安装目录。

对于ORACLE_HOME安装，驱动程序库安装在Linux的$ORACLE_HOME/lib目录和Windows的%ORACLE_HOME%\bin目录中。对于Instant Client ZIP文件，该库位于instantclient_12_1目录中。对于Instant Client RPM安装，该库分别位于32位和64位Linux平台上的/usr/lib/oracle/12.1/client/lib或/usr/lib/oracle/12.1/client64/lib目录中。

使用MySQL客户端库驱动程序进行Oracle迁移概述

将基于C的MySQL应用程序迁移到Oracle数据库涉及以下步骤：

  1. 确认应用程序针对MySQL数据库运行。

    这可确保迁移过程从已知的功能基准开始。

  2. 将 libmysqlclient 库 替换为 liboramysql 库 。

    必须重新链接应用程序才能使用liboramysql库而不是libmysqlclient库。

  3. 将应用程序架构迁移到Oracle数据库。

    必须迁移模式才能使用Oracle DDL和类型。Oracle SQL Developer在此过程中提供帮助。

    有关更多详细信息，请参见“ Oracle SQL Developer用户指南” 。

  4. 查看应用程序使用的所有SQL语句。

    如有必要，请更改应用程序的SQL语句以使用Oracle语法，或实现SQL转换程序以在应用程序运行时自动执行转换。重写依赖于Oracle数据库不支持的MySQL功能的任何逻辑。

    请参阅JDBC和ODBC应用程序的SQL转换 。

  5. 更新应用程序的连接字符串以连接到Oracle数据库。

    在连接调用的host参数中使用Oracle Easy Connect语法或tnsnames.ora连接标识符。

  6. 使用Oracle数据库测试应用程序。

    根据Oracle数据库验证应用程序。

使用MySQL客户端库驱动程序for Oracle

liboramysql API与MySQL Commercial Connector / C 6.0兼容。用于Oracle数据库的MySQL驱动程序liboramysql将MySQL API调用转换为Oracle调用接口（OCI）调用，以及Oracle和MySQL数据类型之间的调用。

可以重新链接现有的基于MySQL的应用程序以使用liboramysql驱动程序，从而使Oracle数据库成为新的数据源。请注意， liboramysql驱动程序仅支持与Oracle数据库的连接。无法在同一应用程序中同时连接MySQL数据库和Oracle数据库。

有关数据类型映射和API兼容性的详细信息，请参阅Oracle MySQL客户端库驱动程序的API参考 。其他信息也可以在Oracle SQL Developer用户指南中找到 。

liboramysql驱动程序不转换SQL语句。您必须重写对Oracle数据库无效的语句。您可以直接在应用程序中执行此操作，也可以使用SQL转换程序执行此操作。还必须单独迁移应用程序架构和数据。Oracle SQL Developer自动执行此过程。

每当旧版本的Oracle数据库存在跨版本OCI连接时，您可以使用liboramysql驱动程序连接到这些旧版本。

使用liboramysql驱动程序重新链接应用程序

使用liboramysql库的基本步骤是重新链接应用程序以使用新库。liboramysql库与MySQL Commercial Connector / C 6.0.2软件包中的libmysqlclient.so库兼容，因此在迁移到Oracle数据库之前，必须使用MySQL Commercial Connector / C 6.0.2构建和验证版本敏感的应用程序。

从源代码编译的公共软件的安装脚本通常期望MySQL组件遵循预定义的系统目录结构。您可以使用Instant Client SDK的demo目录中的setuporamysql.sh脚本来实现此目的。

根据应用程序的不同，您可以使用以下一种或多种方法将应用程序与liboramysql库重新链接：

  • 直接使用 liboramysql 库 构建 。

    您可以更新构建脚本以使用liboramysql库并直接使用此Oracle库构建自定义应用程序。

  • 使用 liboramysql 库模拟MySQL Commercial Connector / C目录

    Instant Client SDK中的setuporamysql.sh库显示了如何创建模拟MySQL Commercial Connector / C安装的目录结构。您可以使用此模拟目录构建应用程序。

  • 使用 LD_PRELOAD 环境变量 。

    预配置的程序可能能够使用LD_PRELOAD环境变量与liboramysql库链接。但是，如果程序使用dlopen()方法，则更改此环境变量的值可能不起作用。

  • 复制 liboramysql 库。

    执行以下步骤将liboramysql库重命名为应用程序使用的MySQL客户端库名称：

    1. 使用ldd命令标识与应用程序链接的MySQL库：

      $ ldd yourprogram ...libmysqlclient.so.16 => /usr/lib/libmysqlclient.so.16(0x00007f9004e7f000)..

    2. 以Oracle软件所有者用户身份创建以下符号链接：

      $ ln -s $ ORACLE_HOME / lib / liboramysql12.so $ ORACLE_HOME / lib / libmysqlclient.so.16

    3. 对于之前使用libmysqlclient库的任何应用程序，将$ORACLE_HOME/lib添加到LD_LIBRARY_PATH环境变量中：

      $ export LD_LIBRARY_PATH = $ ORACLE_HOME / lib

  • 替换系统MySQL客户端库。

    重命名目标系统MySQL客户端库并将新库链接到其位置。因为此选项会影响使用MySQL的系统上的每个应用程序，并且只有在绝对必要时才应该这样做。

    #mv /usr/lib64/libmysqlclient.so.16 /usr/lib64/libmysqlclient.so.16.backup#ln -s $ ORACLE_HOME / lib / liboramysql12.so /usr/lib64/libmysqlclient.so.16

如果没有从源代码重建MySQL应用程序，则必须首先将应用程序与MySQL Commercial Connector / C 6.0.2软件包中的libmysqlclient.so库链接起来。这确保了与liboramysql库中的数据结构的二进制兼容性。

连接到Oracle数据库

mysql_real_connect（c，“localhost / pdborcl”，“myun”，“mypw”，NULL，0，NULL，0）;

全球化

应用程序期望的日期格式可以使用Oracle数据库的NLS_DATE_FORMAT环境变量进行设置，也可以在连接后使用等效的ALTER SESSION命令进行ALTER SESSION 。仅当在环境中设置了NLS_LANG才使用NLS_DATE_FORMAT环境变量。

由于MySQL数据库和Oracle数据库之间存在潜在差异，因此liboramysql库中的某些API必然会返回不同的结果。使用这些API的现有应用程序可能需要更改逻辑。有关这些差异的详细信息，请参阅Oracle MySQL客户端库驱动程序的API参考 。

一个新的特点

本附录着眼于改进和 Oracle预编译器1.8版提供的新功能。旨在满足专业软件开发人员的实际需求，这些功能将帮助您构建有效，可靠的应用程序。

A.1关于在不使用指示符变量的情况下获取NULL

对于Oracle预编译器ORA-01405和1.7版本，如果将NULL返回到主机变量，则将数据FETCH数据导入主变量而没有关联的指示符变量的源文件会在运行时返回ORA-01405消息。对于版本1.8，当您指定MODE=ORACLE和DBMS=V7 ，您还可以通过指定UNSAFE_NULL=YES来禁用ORA-01405消息。

在为Oracle数据库开发应用程序时，首选做法是为任何可能返回NULL主机变量包含指示符变量。但是，将应用程序从Oracle V6迁移到Oracle数据库版本7时， UNSAFE_NULL选项可以显着简化该过程。

有关更多信息，请参阅“ UNSAFE_NULL ”和“ 关于使用指示符变量 ” 。

A.1.1关于使用DBMS = V7和MODE = ORACLE

使用MODE=ORACLE和DBMS=V7预编译的应用程序在运行时返回ORA-01405错误，如果在没有关联的指示符变量时将NULL返回到主机变量。在使用指定的这些选项升级到Oracle数据库版本7时，您需要以两种方式之一迁移应用程序：

  • 修改源代码以包含必要的指示符变量

A.6动态SQL语句缓存

A.7可滚动游标

可滚动游标是Oracle执行SQL语句并存储在执行期间处理的信息的工作区。执行游标时，查询结果将放入一组称为结果集的行中。结果集可以按顺序或非顺序获取。非顺序结果集称为可滚动游标。可滚动游标使用户能够以前向，后向和随机方式访问数据库结果集的行。此可滚动游标使程序可以获取结果集中的任何行。有关可滚动游标的更多信息，请参阅Scrollable Cursors 。

A.8平台字节序支持

Oracle存储的unicode数据（UTF16）始终采用big-endian形式。目前，客户端应用程序在不同平台上运行Linux和Windows具有little-endian表示，Solaris具有big-endian表示。插入或选择UTF16数据时，Pro * Cobol不会在服务器和客户端之间转换字节序形式。这会导致PIC N变量中的UTF16（UCS2）字符串损坏。

可以使用命令行选项picn_endian维护PIC N变量中的平台字节序（Linux和Windows的Little-endian形式，Solaris的Big-endian形式）。

新命令行选项

picn_endian = {BIG | OS}

如果picn_endian = big，则PIC N变量与字符集ID AL16UTF16绑定。

如果picn_endian = os，则PIC N变量与字符集ID UCS2绑定。

此选项的默认值为“big”以保留当前行为。如果NLS_NCHAR不是AL16UTF16，则忽略此选项。

可以使用现有的Pro * Cobol命令行选项设置PIC N变量的字符集形式：

charset_picn = {nchar_charset |值为db_charset}

关于AutoUpgrade分析处理模式

AutoUpgrade分析（ analyze ）处理模式检查您的数据库以查看它是否已准备好进行升级。

在分析模式下运行AutoUpgrade时，AutoUpgrade仅从数据库中读取数据，并且不对数据库执行任何更新。您可以在正常工作时间使用“分析”模式运行“自动升级”。在设置目标发行版Oracle数据库主目录之前，可以在源Oracle数据库主目录上以分析模式运行AutoUpgrade。

您可以使用以下语法在Analyze模式下启动AutoUpgrade，其中Java-8-home是Java 8发行版的位置，或者是Java 8 home的环境变量集：

Java-8-home / bin / java -jar autoupgrade.jar -config yourconfig.txt -mode analyze

如果Java 8位于源Oracle主目录中，则使用以下语法在Analyze模式下启动AutoUpgrade，其中Oracle_home是Oracle主目录，或者是为Oracle主目录设置的环境变量， yourconfig.txt是您的配置文件：

Oracle_home / jdk8 / bin / java -jar autoupgrade.jar -config yourconfig.txt -mode analyze

说明autoupgrade-analyze-flow.eps的说明

AutoUpgrade Analyze模式生成两个输出文件，这些文件的名称为您检查的数据库的系统标识符（ SID ）：

  • SID .html ：使用Web浏览器查看此文件。

  • SID _preupgrade.log ：使用文本编辑器查看此文件。

每个报告都会识别出通过运行自动修复脚本或手动更正而无法更正的升级错误。如果发生错误，则会在用户日志文件中以及status.json文件中报告这些错误。

Analyze模式还在路径cfgtoollogs/upgrade/auto/status生成状态目录。此目录包含指示分析是成功还是失败的文件。该目录有两个JSON文件， status.json和progress.json ：

  • status.json ：包含升级最终状态的高级状态JSON文件。
  • progress.json ：一个JSON文件，包含代表配置文件执行的所有升级的当前进度。如果发生错误，则会在运行AutoUpgrade的用户的日志文件中以及status.json文件中报告这些错误。

如果服务器上没有目标数据库Oracle主目录，则必须在配置文件中将源主目录和目标Oracle主目录参数设置为相同的路径，以便可以运行自动升级分析处理模式。例如：

关于AutoUpgrade修复处理模式

AutoUpgrade Fixup（ fixups ）处理模式分析您的数据库，并执行必须更正的项目的修正，然后才能执行升级。

在Fixup模式下运行AutoUpgrade时，AutoUpgrade会执行它在Analyze模式下执行的检查。完成这些检查后，AutoUpgrade将在您开始升级之前执行修复早期版本数据库（源数据库）所需的所有自动修复。当您计划将数据库移动到其他平台时，使用Fixup模式准备数据库以进行升级。如果目标Oracle主目录不可用，则在修改配置文件以在源数据库系统上运行修订时，可以将target_home Oracle主目录值设置为与source_home Oracle主目录相同的路径。

警告：

Oracle建议您在Fixup模式下运行AutoUpgrade之前单独在Analyze模式下运行AutoUpgrade。Fixup模式可以更改源数据库。

作为升级准备的一部分，如果源数据库需要更正可能在升级期间导致错误的条件，则在Fixup模式下运行AutoUpgrade会自动修复源数据库。因为在Fixup模式下运行AutoUpgrade是您在移动到另一个系统时执行的一个步骤，所以它不会创建有保证的还原点。Oracle建议您在正常工作时间之外运行此模式。

使用以下语法在Fixup模式下启动AutoUpgrade，其中Java-8-home是Java 8发行版的位置，或者是Java 8 home的环境变量集：

Java-8-home / bin / java -jar autoupgrade.jar -config yourconfig.txt -mode fixup

如果Java 8位于源Oracle主目录中，则使用以下语法在Fixup模式下启动AutoUpgrade，其中Oracle_home是Oracle主目录，或者是为Oracle主目录设置的环境变量， yourconfig.txt是您的配置文件：

Oracle_home / jdk8 / bin / java -jar autoupgrade.jar -config yourconfig.txt -mode fixup

说明auttifgrade-fixup-flow.eps的说明

当AutoUpgrade在Fixup模式下运行时，它会通过运行在Analyze模式下运行的相同预检开始。然后，它运行源数据库的自动修复以准备升级，并生成一个高级状态文件，指示修复操作的成功或失败。如果发生错误，则会在运行AutoUpgrade的用户的日志文件中报告这些错误。

警告：

Fixup模式下的AutoUpgrade不会创建保证还原点。在Fixup模式下启动AutoUpgrade之前，请确保已备份数据库。

AutoUpgrade Deploy（ deploy ）处理模式执行数据库的实际升级。如有必要，它还可以对数据库执行修正。

在运行Deploy之前，您必须已安装目标版本，并且除了作为AutoUpgrade脚本的一部分运行的备份计划之外，您还必须具有备份计划。

使用以下语法在Deploy模式下启动AutoUpgrade，其中Oracle_home是Oracle主目录，或者为Oracle主目录设置的环境变量， yourconfig.txt是您的配置文件：

Oracle_home / jdk8 / bin / java -jar autoupgrade.jar -config yourconfig.txt -mode deploy

说明autoupgrade-deploy-flow.eps的说明

在部署模式下运行AutoUpgrade时，AutoUpgrade将对数据库执行所有升级操作，从预升级源数据库分析到升级后检查。每个操作都准备下一个操作。如果发生错误，则停止操作。所有错误都记录在运行AutoUpgrade脚本的用户的日志文件中。为每个操作生成高级状态文件，其显示操作的成功或失败。

关于AutoUpgrade升级处理模式

通过AutoUpgrade升级（ upgrade ）处理模式，您可以在无权访问源Oracle主目录的情况下升级目标Oracle主目录。

您可以使用升级模式将升级分为两部分：首先，如果您选择，您可以在其Oracle主目录上运行的源数据库上运行前缀模式。其次，使用升级模式在新主目录上执行数据库升级。要使用升级模式，在升级模式下运行AutoUpgrade之前，必须在目标Oracle主目录中启动并运行数据库。

使用以下语法在升级模式下启动AutoUpgrade，其中Oracle_home是Oracle主目录，或者为Oracle主目录设置的环境变量， yourconfig.txt是您的配置文件：
Oracle_home / jdk8 / bin / java -jar autoupgrade.jar -config yourconfig.txt -mode upgrade

说明autoupgrade-upgrade-flow.eps的说明

此命令在数据库上运行升级和重新编译操作。将Oracle数据库移动到与原始源系统不同的系统时，请使用此选项，以便不能使用“自动升级部署”模式。

由于AutoUpgrade在升级模式下运行，因此会将错误记录到运行AutoUpgrade脚本的用户的日志文件中。为每个操作生成高级状态文件，其显示操作的成功或失败。在升级模式下运行AutoUpgrade时，不会执行升级后操作。

创建SYNONYM

目的

使用CREATE SYNONYM语句创建同义词 ，它是表，视图，序列，运算符，过程，存储函数，包，物化视图，Java类架构对象，用户定义对象类型或其他同义词的替代名称。同义词会对其目标对象设置依赖关系，并在更改或删除目标对象时变为无效。

同义词提供数据独立性和位置透明性。无论哪个用户拥有表或视图，无论哪个数据库拥有表或视图，同义词都允许应用程序无需修改即可运行。但是，同义词不能替代数据库对象的权限。在用户可以使用同义词之前，必须向用户授予适当的权限。

您可以在以下DML语句中引用同义词： SELECT ， INSERT ， UPDATE ， DELETE ， FLASHBACK TABLE ， EXPLAIN PLAN和LOCK TABLE 。

您可以在以下DDL语句中引用同义词： AUDIT ， NOAUDIT ， GRANT ， REVOKE和COMMENT 。
