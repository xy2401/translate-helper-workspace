1.2属性图先决条件

虽然不需要特殊的配置操作，但使用属性图功能的要求如下。

  • 必须安装Oracle Spatial and Graph。

  • 必须enabled max_string_size ，

  • 必须将AL16UTF16 （而不是UTF8 ）指定为NLS_NCHAR_CHARACTERSET。

    AL32UTF8 (UTF8)应该是默认字符集，但AL16UTF16必须是NLS_NCHAR_CHARACTERSET。

  • 必须安装Java 8或更高版本。

    但是，不需要单独安装，因为JDK 8安装在$ ORACLE_HOME下。只需确保使用正确版本的Java来运行基于属性图的应用程序。

  • 如果使用属性图架构，如Oracle数据库的属性图架构对象中所述，则需要Oracle分区许可证。请注意，如果按照使用双表模式处理属性图中所述使用双表模式 ，则Oracle分区是可选的。

  • 如果使用SolrCloud进行文本索引，则必须单独下载SolrCloud并手动配置。

    有关使用SolrCloud和其他搜索引擎管理的文本索引的信息 ，请参阅管理属性图数据的文本索引 。

    有关下载和使用Apache Solr的信息， 请访问https://lucene.apache.org/solr/guide/ 。

1.3属性图特征

图形管理链接数据的网络，作为顶点和边的顶点，边和属性。

图形通常用于建模，存储和分析社交网络，网络安全，公用事业和电信，生命科学和临床数据以及知识网络中的关系。

典型的图形分析包括图形遍历，建议，查找社区和影响因素以及模式匹配。包括电信，生命科学和医疗保健，安全，媒体和出版在内的行业可以从图表中受益。

Oracle Special和Graph的属性图功能支持具有以下功能的用例：

  • 可伸缩的图形数据库

  • 基于开发人员的API，基于Tinkerpop蓝图和Java图形API

  • 通过与Apache Lucene，SolrCloud和Oracle Text集成进行文本搜索和查询

  • 脚本语言支持Groovy和Python

  • 并行的内存中图形分析引擎

  • 一套快速，可扩展的社交网络分析功能，包括排名，中心，推荐，社区检测，路径查找

  • 以Oracle定义的平面文件格式并行批量加载和导出属性图数据

  • 通过基于Groovy的控制台实现Java和Tinkerpop Gremlin API的可管理性

  • 属性图调整建议

使用DBUA在Linux，Unix和Windows系统上升级数据库

要使用DBUA图形用户界面升级数据库，请在新的Oracle主目录中执行这些步骤。

在Microsoft Windows系统（Windows）上，将DBUA作为Oracle数据库管理用户（具有操作系统分配的ORA_DBA角色的用户）或Oracle安装所有者帐户运行。

  1. 从安装了新数据库软件的Oracle主目录启动Oracle数据库升级助手（DBUA）。dbua可执行文件位于目录路径ORACLE_HOME /bin 。

    • 在Linux或Unix平台上，以具有SYSDBA权限的用户身份登录，并在Oracle数据库19c的新主目录中的系统提示符处输入以下命令：

      ./dbua

    • 在Windows操作系统上，依次选择“ 开始” ，“ 程序” ，“ Oracle HOME_NAME ，“ 配置和迁移工具” ，然后选择“ 数据库升级助手” 。

  2. 将显示“ 选择数据库”窗口如果您之前已发布Oracle数据库安装，则会列出这些安装可用于升级。

    说明dbua_1.png的说明

    如果您在任何DBUA窗口上需要帮助，或者如果您想查阅有关DBUA的更多文档，请单击“ 帮助”以打开联机帮助。

    如果需要，请输入所选数据库的SYSDBA用户名和密码。

    如果从没有SYSDBA权限的用户帐户运行DBUA，或者源数据库没有操作系统身份验证，则必须输入用户名和密码凭据才能为所选数据库启用SYSDBA权限。如果您使用的用户帐户具有SYSDBA权限，或者您使用了操作系统身份验证，则无需输入用户名和密码。

    选择后单击“ 下一步” 。

    注意：

    • 您一次只能选择一个数据库。

    • 对于单实例升级，如果数据库未出现在列表中，则检查/etc/oratab存在具有数据库名称的条目。如果未在此列出数据库，则指示DBUA升级特定数据库：

      • 如果您的单实例数据库未在/etc/oratab列出，并且DBUA可以连接到数据库，那么您可以通过使用命令行参数-sid Oracle_SID ， -oracleHome Oracle_home , and sysDBAPassword password启动DBUA来指示DBUA显式升级该数据库-oracleHome Oracle_home , and sysDBAPassword password作为命令行参数。例如：

        dbua -sid Oracle_SID -oracleHome /u01/app/oracle/18.1.0/dbhome1 -sysDBAUserName SYS -sysDBAPassword password
    • 如果您的帐户没有SYSDBA权限，或者您没有设置操作系统身份验证，则可以使用以下语法进行连接，其中mydb是您的Oracle数据库SID， username是具有SYSDBA权限的用户名， password是用户名的密码：
      dbua -sid mydb -oracleHome /u01/app/oracle/18.1.0/dbhome1 -sysDBAUserName  - username -sysDBAPassword  - password

    • Oracle Real Application Clusters（Oracle RAC）升级：如果数据库未出现在列表中，请输入以下crsctl命令以检查Oracle RAC实例：

      crsctl status resource -t

      您还可以输入以下命令来检查特定的Oracle RAC数据库，其中db_name是Oracle RAC数据库名称：

      crsctl status resource ora。 db_name .db

    • 在Microsoft Windows上，以下安全更改会影响身份验证和用户帐户：

      • 出于安全原因，不再支持使用NTLM协议的Windows NTS身份验证。Kerberos身份验证是唯一受支持的身份验证。在此版本中，NTS在Windows NT域或Windows NT控制器的域中不起作用。

      • Oracle使用标准Microsoft Windows用户帐户而不是Windows LocalSystem帐户来运行Oracle数据库服务。减少Oracle安装所有者的帐户访问权限可在Microsoft Windows上提供更好的安全性。

  3. 如果所选数据库是多租户容器数据库（CDB），则DBUA将显示“ 可插拔数据库”窗口。“ 可插拔数据库”窗口列出了CDB中包含的可插拔数据库。列出的PDB将作为所选CDB升级的一部分进行升级。


    说明dbua_2.png的描述

    您可以选择PDB的升级优先级。单击每个PDB的优先级列，然后输入升级优先级的数值，其中1先升级，2升级第二，依此类推。

    默认情况下， CDB$ROOT ， PDB$SEED以及插入CDB的所有PDB都会升级。如果您现在不想升级某些PDB，请拔下这些PDB。

    完成选择PDB和升级优先级后，单击“ 下一步” 。

  4. 仅限Windows平台：如果升级目标主目录是与Oracle主目录用户关联的安全主目录，则会打开“ 指定Oracle主目录用户密码”窗口。对于其他平台，请继续执行下一步。


    说明dbua3a-win.png的描述

    提供Oracle主用户名，并提供此用户帐户的密码，然后单击“ 下一步” 。

  5. 将打开“ 先决条件检查”窗口。DBUA分析数据库，执行预升级检查并根据需要显示警告。以下是DBUA检查的示例列表，以及DBUA对数据库执行的操作的列表：

    • 空数据库回收站。

    • 识别无效对象。

    • 识别已弃用和不支持的初始化参数。

    • 确定时区数据文件版本。

    分析需要几分钟才能完成。

    当DBUA完成其分析时，将再次显示“ 先决条件检查”窗口，显示检查结果。


    说明dbua_3.png的描述

    “ 先决条件检查”窗口显示DBUA已完成的检查以及发现的任何错误的严重性。当DBUA发现错误时，它指示哪些错误是可修复的，以及您可以采取哪些操作来纠正错误。

    如果出现DBUA可以修复的任何错误， 请选择“ 修复并再次检查” 。

    如果DBUA检测到无法纠正的错误， 请手动修复错误原因，然后选择“ 再次检查” 。

    如果DBUA没有发现错误或警告，则DBUA会自动绕过此窗口并进入下一个窗口。

    修复检测到的错误后，单击“ 下一步” 。

  6. 将显示“ 选择升级选项”窗口。


    说明dbua_4.png的描述

    该窗口提供以下选项：

    启用并行升级

    如果要在升级过程中启用并行性，请选择此选项。并行运行升级过程可根据可用于同时处理脚本和进程运行的CPU数量，减少执行升级所需的时间。

    在升级后重新编译无效对象

    升级完成后，此选项将重新编译所有无效的PL / SQL模块。如果在升级后阶段没有DBUA重新编译无效对象，则必须在升级数据库后手动重新编译无效对象。

    升级时区数据

    此选项更新此版本的时区数据文件。如果未选择此选项，则必须在升级后手动更新时区配置文件。

    指定要执行的自定义SQL脚本 。

    如果要在升级过程中运行自定义SQL脚本，请选择此选项。根据需要，单击“ 浏览”以获取“ 升级前”或“ 升级后”输入字段。导航到自定义SQL脚本所在的位置。

    完成选择后，单击“ 下一步” 。

  7. 将出现“ 选择恢复选项”窗口。要在升级期间发生故障时恢复数据库，请从以下选项之一中进行选择：

    • 使用闪回和保证还原点 。

      您可以创建新的保证还原点，也可以使用现有的还原点。如果使用现有还原点，则单击选择域以选择要使用的还原点。

      注意：

      如果要升级的数据库具有Oracle Data Guard物理备用数据库，则必须先在每个备用数据库上创建保证还原点，然后才能在主数据库上创建一个备用数据库。如果不首先在每个备用数据库上创建还原点，则必须在使用保证还原点降级主数据库后再次重新创建所有备用数据库。升级成功后，您必须手动删除备用数据库上所有保证还原点。

    • 使用RMAN备份

      您可以创建新的脱机RMAN备份，也可以使用现有备份。单击“ 浏览”以指定备份的路径。

    • 使用最新可用的RMAN备份

      您可以使用现有备份。单击“ 查看/编辑还原脚本”以选择要使用的备份。

    • 我有自己的备份和恢复策略 。

      仅当您具有适用于现有数据库的第三方备份解决方案时，才选择此选项。

    完成选择后，单击“ 下一步” 。

  8. 对于单实例数据库安装，将打开“ 配置网络”窗口。从源Oracle主目录中选择一个或多个要迁移到新升级的Oracle主目录的侦听器，或在安装期间创建新的侦听器。


    说明dbua_6.png的描述

    “网络配置”窗口的“侦听器选择”区域显示包含以下列的表：

    • 选择列。选择要更新的侦听器。

    • 名称此列显示侦听器名称。

    • 端口此列显示配置侦听器的端口。

    • Oracle Home此列显示配置了侦听器的Oracle主目录。

    • 状态此列显示侦听器状态（向上或向下）。

    • 迁移选择此列，然后选择“ 是”以进行迁移，或者选择“ 否”（如果您不想迁移）。

    您还可以选择创建新的侦听器。如果创建新侦听器，则提供侦听器名称，要放置的Oracle主目录以及要配置侦听器要监视的端口。

    做出选择后，DBUA会为您迁移的所有侦听器完成以下步骤：

    1. DBUA将选定的侦听器添加到目标Oracle主目录的listener.ora文件中，然后启动它。

    2. DBUA从旧（源） listener.ora文件中删除已升级数据库的条目。

    3. DBUA在源和目标Oracle数据库环境中重新加载listener.ora文件。

    注意：

    如果在同一侦听器中注册了其他数据库，则在侦听器迁移期间可能会影响其新的客户端连接请求。

    完成选择后，单击“ 下一步” 。

  9. 出现“ 配置管理”窗口。在“ 配置管理”窗口中，选择管理选项：


    说明dbua_7.png的描述

    • 配置企业管理器（EM）数据库快速

      Oracle Enterprise Manager Database Express是基于Web的数据库管理应用程序，内置于Oracle数据库中。EM Express替换了早期版本中可用的DB Control组件。如果选择配置Enterprise Manager Database Express，请输入EM Database Express端口号。例如：5500。您也可以选中该复选框以将快速端口配置为全局端口。

    • 注册企业管理器（EM）云控制

      注册Oracle Enterprise Manager Cloud Control会将数据库及其相关实体（如监听器，Oracle ASM磁盘组和Oracle Clusterware）添加为可以使用EM Cloud Control管理的目标。

      如果选择此选项，则必须在以下字段中提供信息：

      • OMS主机

      • OMS端口

      • EM管理员用户名

      • EM管理员密码

      • DBSNMP用户密码

      • ASMSNMP用户密码

    完成输入信息后，单击“ 下一步” 。

  10. “ 摘要”窗口打开。“摘要”窗口显示您为升级提供的信息。向下滚动列表以查看信息。摘要包括以下信息：

    • 源数据库

    • 目标数据库

    • 可插拔数据库

    • 升级前检查

    • 初始化参数更改

    • 时区升级

    检查您的选择。然后，选择要更改的项目的链接，或单击“上一步”转到上一页 ，或选择“ 完成” ：

    • 如果您在“ 摘要”窗口中看到要更正的信息，请单击要更新的项目上的链接，或单击“上一步”以向后浏览DBUA配置面试。

    • 如果在“ 摘要”窗口中看到的信息正确，请单击“ 完成” 。选择“ 完成”后，将开始升级。

    当DBUA开始升级时， 进度窗口会显示进度条。“ 进度”窗口显示一个表，其中显示了DBUA在升级期间完成的步骤。此表显示升级过程中的持续时间和升级步骤状态。如果您此时必须取消升级，DBUA会提供“ 停止”按钮。


    说明dbua_9.png的描述

    升级完成CDB根目录和每个PDB种子的升级后，“ 进度”窗口将状态标记为“已完成” 。

  11. 升级完成后，将打开“ 结果”窗口。“ 结果”窗口显示有关原始数据库以及已升级数据库的信息。“升级结果”报告还显示DBUA对初始化参数所做的更改。如果要升级多租户体系结构数据库，则“ 结果”窗口还会显示可插拔数据库以及升级后存储日志文件的目录。向下滚动以查看有关预升级检查的更多详细信息。如果升级成功，则“升级结果”字段将报告结果，并且您看不到警告消息。如果升级失败，如此示例图像所示，则“ 还原数据库”按钮将显示在显示字段下方的右下角。您可以单击此按钮以启动数据库还原。


    说明dbua_10.png的描述

  12. 可选：检查日志文件以获取有关升级过程的更多详细信息。如果设置了Oracle基本环境变量，则DBUA日志文件位于路径/ ORACLE_BASE /cfgtoollogs/dbua/upgradesession_ timestamp / SID 。如果未设置Oracle base，则DBUA日志文件位于路径/ ORACLE_HOME /cfgtoollogs/dbua/upgradesession_ timestamp / SID

    注意：

    升级结果窗口的HTML版本也保存在日志文件目录中。您可以单击此HTML窗口中的链接以在浏览器中查看日志窗口。

    如果您对升级结果感到满意，请单击“ 关闭”以退出DBUA。

  13. 升级完成后，执行本书中描述的升级后过程。完成升级后过程后，即可使用升级后的数据库。

警告：

为防止未经授权使用数据库，Oracle建议您在升级数据库后立即更改所有用户密码。

如果Oracle Database 12c及更高版本的默认安全设置已就绪，则密码必须至少为八个字符。不允许使用welcome和oracle等密码。

使用DBUA在Linux，Unix和Windows系统上升级数据库

要使用DBUA图形用户界面升级数据库，请在新的Oracle主目录中执行这些步骤。

在Microsoft Windows系统（Windows）上，将DBUA作为Oracle数据库管理用户（具有操作系统分配的ORA_DBA角色的用户）或Oracle安装所有者帐户运行。

  1. 从安装了新数据库软件的Oracle主目录启动Oracle数据库升级助手（DBUA）。dbua可执行文件位于目录路径ORACLE_HOME /bin 。

    • 在Linux或Unix平台上，以具有SYSDBA权限的用户身份登录，并在Oracle数据库19c的新主目录中的系统提示符处输入以下命令：

      ./dbua

    • 在Windows操作系统上，依次选择“ 开始” ，“ 程序” ，“ Oracle HOME_NAME ，“ 配置和迁移工具” ，然后选择“ 数据库升级助手” 。

  2. 将显示“ 选择数据库”窗口如果您之前已发布Oracle数据库安装，则会列出这些安装可用于升级。

    说明dbua_1.png的说明

    如果您在任何DBUA窗口上需要帮助，或者如果您想查阅有关DBUA的更多文档，请单击“ 帮助”以打开联机帮助。

    如果需要，请输入所选数据库的SYSDBA用户名和密码。

    如果从没有SYSDBA权限的用户帐户运行DBUA，或者源数据库没有操作系统身份验证，则必须输入用户名和密码凭据才能为所选数据库启用SYSDBA权限。如果您使用的用户帐户具有SYSDBA权限，或者您使用了操作系统身份验证，则无需输入用户名和密码。

    选择后单击“ 下一步” 。

    注意：

    • 您一次只能选择一个数据库。

    • 对于单实例升级，如果数据库未出现在列表中，则检查/etc/oratab存在具有数据库名称的条目。如果未在此列出数据库，则指示DBUA升级特定数据库：

      • 如果您的单实例数据库未在/etc/oratab列出，并且DBUA可以连接到数据库，那么您可以通过使用命令行参数-sid Oracle_SID ， -oracleHome Oracle_home , and sysDBAPassword password启动DBUA来指示DBUA显式升级该数据库-oracleHome Oracle_home , and sysDBAPassword password作为命令行参数。例如：

        dbua -sid Oracle_SID -oracleHome /u01/app/oracle/18.1.0/dbhome1 -sysDBAUserName SYS -sysDBAPassword password
    • 如果您的帐户没有SYSDBA权限，或者您没有设置操作系统身份验证，则可以使用以下语法进行连接，其中mydb是您的Oracle数据库SID， username是具有SYSDBA权限的用户名， password是用户名的密码：
      dbua -sid mydb -oracleHome /u01/app/oracle/18.1.0/dbhome1 -sysDBAUserName  - username -sysDBAPassword  - password

    • Oracle Real Application Clusters（Oracle RAC）升级：如果数据库未出现在列表中，请输入以下crsctl命令以检查Oracle RAC实例：

      crsctl status resource -t

      您还可以输入以下命令来检查特定的Oracle RAC数据库，其中db_name是Oracle RAC数据库名称：

      crsctl status resource ora。 db_name .db

    • 在Microsoft Windows上，以下安全更改会影响身份验证和用户帐户：

      • 出于安全原因，不再支持使用NTLM协议的Windows NTS身份验证。Kerberos身份验证是唯一受支持的身份验证。在此版本中，NTS在Windows NT域或Windows NT控制器的域中不起作用。

      • Oracle使用标准Microsoft Windows用户帐户而不是Windows LocalSystem帐户来运行Oracle数据库服务。减少Oracle安装所有者的帐户访问权限可在Microsoft Windows上提供更好的安全性。

  3. 如果所选数据库是多租户容器数据库（CDB），则DBUA将显示“ 可插拔数据库”窗口。“ 可插拔数据库”窗口列出了CDB中包含的可插拔数据库。列出的PDB将作为所选CDB升级的一部分进行升级。


    说明dbua_2.png的描述

    您可以选择PDB的升级优先级。单击每个PDB的优先级列，然后输入升级优先级的数值，其中1先升级，2升级第二，依此类推。

    默认情况下， CDB$ROOT ， PDB$SEED以及插入CDB的所有PDB都会升级。如果您现在不想升级某些PDB，请拔下这些PDB。

    完成选择PDB和升级优先级后，单击“ 下一步” 。

  4. 仅限Windows平台：如果升级目标主目录是与Oracle主目录用户关联的安全主目录，则会打开“ 指定Oracle主目录用户密码”窗口。对于其他平台，请继续执行下一步。


    说明dbua3a-win.png的描述

    提供Oracle主用户名，并提供此用户帐户的密码，然后单击“ 下一步” 。

  5. 将打开“ 先决条件检查”窗口。DBUA分析数据库，执行预升级检查并根据需要显示警告。以下是DBUA检查的示例列表，以及DBUA对数据库执行的操作的列表：

    • 空数据库回收站。

    • 识别无效对象。

    • 识别已弃用和不支持的初始化参数。

    • 确定时区数据文件版本。

    分析需要几分钟才能完成。

    当DBUA完成其分析时，将再次显示“ 先决条件检查”窗口，显示检查结果。


    说明dbua_3.png的描述

    “ 先决条件检查”窗口显示DBUA已完成的检查以及发现的任何错误的严重性。当DBUA发现错误时，它指示哪些错误是可修复的，以及您可以采取哪些操作来纠正错误。

    如果出现DBUA可以修复的任何错误， 请选择“ 修复并再次检查” 。

    如果DBUA检测到无法纠正的错误， 请手动修复错误原因，然后选择“ 再次检查” 。

    如果DBUA没有发现错误或警告，则DBUA会自动绕过此窗口并进入下一个窗口。

    修复检测到的错误后，单击“ 下一步” 。

  6. 将显示“ 选择升级选项”窗口。


    说明dbua_4.png的描述

    该窗口提供以下选项：

    启用并行升级

    如果要在升级过程中启用并行性，请选择此选项。并行运行升级过程可根据可用于同时处理脚本和进程运行的CPU数量，减少执行升级所需的时间。

    在升级后重新编译无效对象

    升级完成后，此选项将重新编译所有无效的PL / SQL模块。如果在升级后阶段没有DBUA重新编译无效对象，则必须在升级数据库后手动重新编译无效对象。

    升级时区数据

    此选项更新此版本的时区数据文件。如果未选择此选项，则必须在升级后手动更新时区配置文件。

    指定要执行的自定义SQL脚本 。

    如果要在升级过程中运行自定义SQL脚本，请选择此选项。根据需要，单击“ 浏览”以获取“ 升级前”或“ 升级后”输入字段。导航到自定义SQL脚本所在的位置。

    完成选择后，单击“ 下一步” 。

  7. 将出现“ 选择恢复选项”窗口。要在升级期间发生故障时恢复数据库，请从以下选项之一中进行选择：

    • 使用闪回和保证还原点 。

      您可以创建新的保证还原点，也可以使用现有的还原点。如果使用现有还原点，则单击选择域以选择要使用的还原点。

      注意：

      如果要升级的数据库具有Oracle Data Guard物理备用数据库，则必须先在每个备用数据库上创建保证还原点，然后才能在主数据库上创建一个备用数据库。如果不首先在每个备用数据库上创建还原点，则必须在使用保证还原点降级主数据库后再次重新创建所有备用数据库。升级成功后，您必须手动删除备用数据库上所有保证还原点。

    • 使用RMAN备份

      您可以创建新的脱机RMAN备份，也可以使用现有备份。单击“ 浏览”以指定备份的路径。

    • 使用最新可用的RMAN备份

      您可以使用现有备份。单击“ 查看/编辑还原脚本”以选择要使用的备份。

    • 我有自己的备份和恢复策略 。

      仅当您具有适用于现有数据库的第三方备份解决方案时，才选择此选项。

    完成选择后，单击“ 下一步” 。

1 Oracle Database的WebSphere MQ简介

以下主题概述了消息队列，WebSphere MQ以及访问WebSphere MQ队列时网关的角色。

消息队列简介

消息队列使分布式应用程序能够通过在应用程序之间发送消息来异步通信。

来自发送应用程序的消息存储在队列中，并由接收应用程序检索。应用程序通过向消息排队系统发送请求来通过队列发送或接收消息。发送和接收应用程序可以使用相同或不同的消息排队系统，允许消息排队系统处理从发送方队列到接收方队列的消息转发。

排队的消息可以存储在中间节点，直到系统准备好将它们转发到下一个节点。在目标节点处，消息存储在队列中，直到接收应用程序从队列中检索它们。即使网络或应用程序出现故障，也可以保证邮件传递。这为应用程序之间提供了可靠的通信通道。

底层模型（在不同环境之间存储和转发消息）的复杂性和细节由消息排队系统处理。通过保持这种抽象级别，可以开发分布式应用程序，而无需担心信息传输的细节。

因为发送和接收应用程序彼此独立地操作，所以发送应用程序较少依赖于远程应用程序的可用性，它们之间的网络以及运行接收应用程序的系统。这可以为参与的应用程序提供更高级别的可用性。

应用程序可以配置消息和消息队列操作以在特定模式下操作。例如，发送应用程序可以指定排队的消息应该在系统崩溃后继续存在。作为另一示例，接收应用可以指定来自队列的接收操作的最大等待时段（在接收队列上还没有消息可用的情况下）。

WebSphere MQ简介

WebSphere MQ是基于消息队列客户端和消息队列服务器模型的消息队列系统。

应用程序在队列管理器和队列所在的服务器节点上运行，或者在远程客户机节点上运行。应用程序只能从与其连接的队列管理器所拥有的队列发送或检索消息。

网关简介

Oracle Database Gateway for WebSphere MQ使Oracle应用程序能够与其他WebSphere MQ应用程序集成。

Oracle应用程序可以将消息发送到其他WebSphere MQ应用程序或从中接收消息。通过网关，Oracle应用程序可以访问WebSphere MQ消息队列远程过程调用（RPC）处理。

网关扩展了Oracle数据库可用的RPC工具，并允许任何客户端应用程序使用PL / SQL访问WebSphere MQ队列中的消息。网关提供由网关转换为MQI调用的PL / SQL过程。这些过程类似于MQI的调用和类型，但它们适用于充分利用与Oracle数据库的事务集成。

通过WebSphere MQ，网关可与各种平台上的任何其他WebSphere MQ系统进行通信，包括大型机，基于UNIX的系统，Microsoft Windows和其他桌面环境。网关不需要远程系统上的任何Oracle软件。该网关与现有WebSphere MQ应用程序集成，无需对这些应用程序进行任何更改，并使用户能够利用他们在这些应用程序中的投资，同时使他们能够最大限度地利用面向消息的系统的优势。

该网关还提供了一种将这些现有WebSphere MQ应用程序与新技术领域（如网络计算）集成的方法。任何Oracle应用程序可以调用PL / SQL程序，包括使用Oracle应用Server 11的3G应用。

B.1修补或升级Oracle数据库软件之前

在开始修补或升级Oracle数据库软件之前，需要考虑几点。

  • 如果您使用Oracle Data Guard代理来管理配置，请按照Oracle Data Guard Broker手册中的说明获取有关删除或禁用代理配置的信息。

  • 这些主题中描述的过程将与“ Oracle数据库升级指南”中包含的过程一起使用。

  • 检查nologging操作。如果已执行nologging操作，则必须更新备用数据库。有关详细信息，请参阅指定NOLOGGING子句后的恢复 。

  • 记下由于OFFLINE IMMEDIATE而需要恢复的任何表空间或数据文件。在升级之前，应该在线或离线恢复表空间或数据文件。

  • 在Oracle Data Guard配置中，所有物理和快照备用数据库都必须使用主数据库中密码文件的副本。如Oracle数据库12的c解除2（12.2.0.1），在主数据库上进行密码文件的变化将自动传播到备用数据库。（密码文件更改包括授予或撤销管理权限（ SYSDG ， SYSOPER ， SYSDBA等）的时间，以及更改具有管理权限的任何用户的密码时。）

    远程同步实例是自动更新功能的一个例外。更新的密码文件仍必须手动复制到远程同步实例，因为远程同步实例会接收重做，但不会应用它。在远程同步实例上手动更新密码文件时，包含来自主数据库的相同密码更改的重做会自动传播到设置为从该远程同步实例接收重做的任何备用数据库。应用重做时，备用数据库上的密码文件会更新。

如果您的配置中存在级联备用数据库，那么这些级联备用数据库必须遵循与任何其他备用数据库相同的规则，但应该最后关闭并首先在新家中重新启动。

B.2使用备用第一补丁修补Oracle数据库

Oracle Data Guard Standby-First Patch Apply为主数据库及其物理备用数据库之间的不同数据库主页软件提供支持。

提供此支持的目的是以滚动方式应用和验证Oracle修补程序和修补程序包，同时将主数据库的风险降至最低。例如，使用Data Guard Standby-First Patch Apply，您首先将数据库主补丁应用于物理备用数据库。备用数据库用于运行只读工作负载，如果是快照备用数据库，则用于读写工作负载，用于测试和评估补丁。在通过评估之后，然后将补丁安装在主系统上，更好地保证数据库主补丁的有效性和稳定性。

Oracle Data Guard Standby-First Patch Apply仅支持Oracle Databaseered Systems上的Oracle Database 11.2.0.1及更高版本的经过认证的临时修补程序和修补程序包（例如，修补程序集更新或Exadata数据库修补程序）（例如，Exadata， SuperCluster）和非工程系统。Data Guard Standby-First认证的补丁和补丁包在补丁自述文件中说明以下内容：

Data Guard Standby-First Installable

以下类型的补丁是Data Guard Standby-First认证的候选者：

  • 数据库主页临时补丁

  • Exadata捆绑包修补程序（例如，Exadata的每月和每季度数据库修补程序）

  • 数据库补丁集更新

更新可能会破坏运行不同数据库家庭软件的主要和物理备用系统之间的互操作性的模块的修补程序和修补软件包未经过“Data Guard Standby-First Installable”认证，并且未在修补程序自述文件中说明。

Oracle补丁集和主要版本升级不符合Data Guard Standby-First Patch Apply的条件。例如，从11.2.0.2到11.2.0.3或11.2到12.1的升级不符合条件。对数据库修补程序集和主要版本使用Data Guard临时逻辑备用滚动升级过程。

此外，从Oracle Database 11g第2版（11.2.0.1）开始，可以使用物理备用数据库以滚动方式安装符合条件的一次性修补程序，修补程序集更新（PSU）和关键修补程序更新（CPU）。有关此功能的更多信息，请参阅http://support.oracle.com的My Oracle Support说明1265700.1。

  4. 关闭物理备用数据库。

语义

发出CREATE CONTROLFILE语句时，Oracle数据库会根据您在语句中指定的信息创建新的控制文件。控制文件位于CONTROL_FILES初始化参数中指定的位置。如果该参数没有值，则数据库在默认控制文件目标中创建Oracle管理的控制文件，该文件是以下之一（按优先顺序排列）：

  1. DB_CREATE_ONLINE_LOG_DEST_ n初始化参数中指定的一个或多个控制文件。第一个目录中的文件是主控制文件。指定DB_CREATE_ONLINE_LOG_DEST_ n ，数据库不会在DB_CREATE_FILE_DEST或DB_RECOVERY_FILE_DEST （快速恢复区域）中创建控制文件。

  2. 如果没有为DB_CREATE_ONLINE_LOG_DEST_ n指定值，但为DB_CREATE_FILE_DEST和DB_RECOVERY_FILE_DEST设置了值，则数据库会在每个位置创建一个控制文件。DB_CREATE_FILE_DEST指定的位置是主控制文件。

  3. 如果仅为DB_CREATE_FILE_DEST指定了值，则数据库将在该位置创建一个控制文件。

  4. 如果仅为DB_RECOVERY_FILE_DEST指定了值，则数据库将在该位置创建一个控制文件。

如果没有为这些参数设置任何值，则数据库会在运行数据库的操作系统的默认位置创建控制文件。此控制文件不是Oracle托管文件。

如果省略任何子句，则Oracle数据库使用默认值而不是上一个控制文件的值。成功创建控制文件后，Oracle Database以初始化参数CLUSTER_DATABASE指定的模式安装数据库。如果未设置该参数，则默认值为FALSE ，并且数据库以EXCLUSIVE模式挂载。Oracle建议您关闭实例并对数据库中的所有文件进行完整备份。

不要在DATAFILE子句中包含只读表空间中的任何数据文件。您可以稍后将这些类型的文件添加到数据库中。此外，请勿在此子句中包含任何临时数据文件（临时文件）。

使用file_specification的datafile_tempfile_spec形式（请参阅file_specification ）列出操作系统文件系统中的常规数据文件和临时文件，或列出Oracle ASM磁盘组文件。使用ASM_filename形式时，必须使用ASM_filename的引用形式之一。有关不同语法形式的信息，请参阅ASM_filename 。

对DATAFILE的限制

您无法在此DATAFILE子句中指定file_specification的autoextend_clause 。

MAXLOGFILES条款

指定可以为数据库创建的最大联机重做日志文件组数。Oracle数据库使用此值来确定在控制文件中为重做日志文件的名称分配多少空间。默认值和最大值取决于您的操作系统。您指定的值不应小于任何重做日志文件组的最大GROUP值。

MAXLOGMEMBERS条款

指定重做日志文件组的最大成员数或相同副本。Oracle数据库使用此值来确定在控制文件中为重做日志文件的名称分配多少空间。最小值为1。最大值和默认值取决于您的操作系统。

MAXLOGHISTORY条款

仅当您在ARCHIVELOG模式下使用Oracle数据库时，此参数才有用。指定数据库自动介质恢复所需的最大归档重做日志文件组的当前估计值。数据库使用此值来确定在控制文件中为归档重做日志文件的名称分配多少空间。

最小值为0。默认值是MAXINSTANCES值的倍数，取决于您的操作系统。最大值仅受控制文件的最大大小限制。数据库将根据需要继续向控制文件的相应部分添加额外空间，这样，如果原始配置不再适用，则无需重新创建控制文件。因此，此参数的实际值最终可能会超出您指定的值。

MAXDATAFILES条款

在CREATE DATABASE或CREATE CONTROLFILE时指定控制文件的数据文件部分的初始大小。尝试添加数量大于MAXDATAFILES但小于或等于DB_FILES的文件会导致控制文件自动扩展，以便数据文件部分可以容纳更多文件。

实例可访问的数据文件数也受初始化参数DB_FILES 。

最大条款

指定可以同时装入和打开数据库的最大实例数。该值优先于初始化参数INSTANCES的值。最小值为1。最大值和默认值取决于您的操作系统。

ARCHIVELOG | NOARCHIVELOG

指定ARCHIVELOG以在重用之前归档重做日志文件的内容。本节准备介质恢复以及实例或系统故障恢复的可能性。

如果省略ARCHIVELOG子句和NOARCHIVELOG子句，则Oracle数据库默认选择NOARCHIVELOG模式。创建控制文件后，可以使用ALTER DATABASE语句在ARCHIVELOG模式和NOARCHIVELOG模式之间进行ALTER 。

强制记录

在创建控制文件后，使用此子句将数据库置于FORCE LOGGING模式。当数据库处于此模式时，Oracle数据库会记录数据库中的所有更改，但临时表空间和临时段的更改除外。此设置优先于您为各个表空间指定的任何NOLOGGING或FORCE LOGGING设置，并且与您为各个数据库对象指定的任何NOLOGGING设置无关。如果省略此子句，则在创建控制文件后，数据库将不会处于FORCE LOGGING模式。

注意：

FORCE LOGGING模式可以产生性能影响。有关何时使用此设置的信息，请参阅Oracle数据库管理员指南 。

SET STANDBY NOLOGGING

SET STANDBY NOLOGGING禁用备用数据库的日志记录。您可以在两种模式中指定它：

  • SET STANDBY NOLOGGING FOR DATA AVAILABILITY保证完整数据复制到备用数据库。主数据库和备用数据库在加载期间同步。在网络拥塞的情况下，主数据库将限制其负载。

XSLT XVM处理器

Oracle XVM包包括XSLT编译器和XVM。该软件包实现了1999年11月16日万维网联盟（W3C）建议书中规定的XSLT语言。

实现XSLT编译器和XVM可以将XSLT（版本1.0）编译为字节码格式，该格式由虚拟机执行。XVM是用于运行已编译的XSLT代码的CPU的软件实现。虚拟机假定编译器将XSLT样式表编译为一系列字节码或XSLT CPU的机器指令。字节码程序是与平台无关的2字节单元序列。它可以在不同的XVM上存储，缓存和运行。 XVM使用字节码程序转换XML实例文档。这种方法明确地将编译时与运行时计算分开，并指定了在指令之间交换数据的统一方式。

这种方法的好处是：

  • XSLT样式表可以编译，保存在文件中，并经常重复使用，即使在不同的平台上也是如此。

  • 与其他XSLT处理器相比，XVM速度更快，占用的内存更少。

  • 字节码与语言无关。从C或C ++ XSLT编译器生成的代码之间没有区别。

XVM用法示例

描述了使用包API的典型方案。

  1. 创建和使用XML元上下文对象。

    xctx = XmlCreate（＆err，...）;

  2. 创建和使用XSLT编译器对象。

    comp = XmlXvmCreateComp（xctx）;

  3. 编译XSLT样式表或XPath表达式并存储或缓存生成的字节码。

    code = XmlXvmCompileFile（comp，xslFile，baseuri，flags，＆err）;

    要么

    code = XmlXvmCompileDom（comp，xslDomdoc，flags，＆err）;

    要么

    code = XmlXvmCompileXPath（comp，xpathexp，namespaces＆err）;

  4. 创建和使用XVM对象。当XVM以Stack Overflow消息终止或需要较小的内存占用时，需要显式堆栈大小设置。请参阅XmlXvmCreate() 。 

    vm = XmlXvmCreate（xctx，“StringStack”，32，“NodeStack”，24，NULL）;

  5. 设置输出（可选）。默认是一个流。

    err = XmlXvmSetOutputDom（vm，NULL）;

    要么

    err = XmlXvmSetOutputStream（vm，＆xvm_stream）;

    要么

    err = XmlXvmSetOutputSax（vm，＆xvm_callback，NULL）;

  6. 将样式表字节码设置为XVM对象。可以与其他字节码重复。

    len = XmlXvmGetBytecodeLength（code，＆err）; err = XmlXvmSetBytecodeBuffer（vm，code，len）;

    要么

    err = XmlXvmSetBytecodeFile（vm，xslBytecodeFile）;

    err = XmlXvmTransformFile（vm，xmlFile，baseuri）;

    要么

    err = XmlXvmTransformDom（vm，xmlDomdoc）;

    要么

    obj =（xvmobj *）XmlXvmEvaluateXPath（vm，code，1,1，node）;

    node = XmlXvmGetOutputDom（vm）;

  9. 删除对象。

    XmlXvmDestroy（VM）; XmlXvmDestroyComp（COMP）; XmlDestroy（xctx）;

9.85 V $ STREAMS_APPLY_READER

V$STREAMS_APPLY_READER显示有关每个应用阅读器的信息。 应用程序阅读器是一个从队列中读取（出列）消息，计算消息依赖性和构建事务的过程。它以事务顺序将事务传递给协调器，以便分配给应用服务器。应用程序阅读器是应用程序进程，出站服务器或入站服务器的子组件。

11.1查询重写概述

当基表包含大量数据时，计算所需聚合或计算这些表之间的连接是昂贵且耗时的。在这种情况下，查询可能需要几分钟甚至几小时。由于物化视图包含已经预先计算的聚合和连接，Oracle数据库采用了一个非常强大的过程，称为查询重写，以使用物化视图快速回答查询。

创建和维护物化视图的主要好处之一是能够利用查询重写，该查询重写将表或视图表示的SQL语句转换为访问详细信息表中定义的一个或多个物化视图的语句。转换对最终用户或应用程序是透明的，不需要干预，也不需要引用SQL语句中的物化视图。由于查询重写是透明的，因此可以像索引一样添加或删除实例化视图，而不会使应用程序代码中的SQL无效。 “ Oracle什么时候重写查询？“描述了要重写查询必须满足的条件。

11.1.1关于查询重写和优化器

查询经过多次检查以确定它是否是查询重写的候选者。如果查询未通过任何检查，则查询将应用于详细信息表而不是物化视图。在响应时间和处理能力方面，无法重写可能代价高昂。

优化程序使用两种不同的方法来确定何时根据实例化视图重写查询。第一种方法将查询的SQL文本与实例化视图定义的SQL文本相匹配。如果第一个方法失败，则优化程序使用更通用的方法来比较查询和物化视图之间的连接，选择，数据列，分组列和聚合函数。

查询重写对以下类型的SQL语句中的查询和子查询进行操作：

  • 选择

  • CREATE TABLE ... AS SELECT

  • 插入...选择

它还对集合运算符UNION ， UNION ALL ， INTERSECT和MINUS子查询以及DML语句（如INSERT ， DELETE和UPDATE子查询进行操作。

维度，约束和重写完整性级别会影响是否重写查询以使用实例化视图。此外，可以通过REWRITE和NOREWRITE提示以及QUERY_REWRITE_ENABLED会话参数启用或禁用查询重写。

DBMS_MVIEW.EXPLAIN_REWRITE过程建议是否可以对查询进行查询重写，如果可以，则建议使用哪些实例化视图。它还解释了为什么无法重写查询。

11.1.2 Oracle何时重写查询？

只有满足一定数量的条件时才会重写查询：

  • 必须为会话启用查询重写。

  • 必须启用物化视图才能进行查询重写。

  • 重写完整性级别应允许使用物化视图。例如，如果物化视图不新鲜且查询重写完整性设置为ENFORCED ，则不使用物化视图。

  • 查询请求的全部或部分结果必须可以从存储在物化视图中的预先计算结果中获得。

为了测试这些条件，优化器可以依赖于用户使用约束和维度（其中包括层次结构，参照完整性和关键数据的唯一性等）声明的一些数据关系，等等。

11.2确保查询重写生效

您必须遵循几个条件才能启用查询重写：

  1. 单个物化视图必须具有ENABLE QUERY REWRITE子句。

    如果未完成此步骤，如启用实体化视图的查询重写中所述，则实例化视图永远不能进行查询重写。

  2. 会话参数QUERY_REWRITE_ENABLED必须设置为TRUE （默认值）或FORCE 。

    请参阅查询重写的初始化参数 。

  3. 必须通过将初始化参数OPTIMIZER_MODE设置为ALL_ROWS ， FIRST_ROWS或FIRST_ROWS_ n来使用基于成本的优化。

    请参阅查询重写的初始化参数 。

您可以使用DBMS_ADVISOR.TUNE_MVIEW过程来优化CREATE MATERIALIZED VIEW语句以启用常规QUERY REWRITE 。

11.2.1为物化视图启用查询重写

您可以使用ALTER MATERIALIZED VIEW语句或创建MATERIALIZED VIEW时指定ENABLE QUERY REWRITE ，如下所示：

CREATE MATERIALIZED VIEW join_sales_time_product_mv ENABLE QUERY REWRITE AS SELECT p.prod_id，p.prod_name，t.time_id，t.week_ending_day，s.channel_id，s.promo_id，s.cust_id，s.amount_sold FROM sales s，products p，times t WHERE s.time_id = t.time_id AND s.prod_id = p.prod_id;

NOREWRITE提示在SQL语句中禁用查询重写，覆盖QUERY_REWRITE_ENABLED参数， REWRITE提示（与mv_name使用时）将符合条件的物化视图限制为提示中指定的视图。

11.2.3控制查询重写

如果已指定ENABLE QUERY REWRITE子句，则物化视图仅适用于查询重写，最初在首次创建REWRITE化视图时或随后使用ALTER MATERIALIZED VIEW语句时。

您可以使用ALTER SYSTEM SET语句或初始化文件为所有会话设置先前描述的会话参数。对于给定用户的会话， ALTER SESSION可用于仅为该会话禁用或启用查询重写。一个例子如下：

ALTER SESSION SET QUERY_REWRITE_ENABLED = TRUE;

您可以为会话设置查询重写级别，从而允许不同的用户在不同的完整性级别上工作。可能的陈述是：

ALTER SESSION SET QUERY_REWRITE_INTEGRITY = STALE_TOLERATED; ALTER SESSION SET QUERY_REWRITE_INTEGRITY = TRUSTED; ALTER SESSION SET QUERY_REWRITE_INTEGRITY = ENFORCED;

您可以为QUERY_REWRITE_INTEGRITY参数设置的值如下：

  • ENFORCED

    这是默认模式。优化程序仅使用物化视图中的新数据，并仅使用基于ENABLED VALIDATED主键，唯一键或外键约束的那些关系。

  • 可信

    在TRUSTED模式下，优化器信任在维度和RELY约束中声明的关系是正确的。在此模式下，优化程序还使用基于视图的预构建物化视图或物化视图，并使用未强制执行的关系以及强制执行的关系。它还信任使用维度指定的ENABLED VALIDATED声明但不是ENABLED VALIDATED主要或唯一键约束和数据关系。此模式提供更强的查询重写功能，但如果您声明的任何可信关系不正确，也会产生不正确结果的风险。

  • STALE_TOLERATED

    在STALE_TOLERATED模式下，优化程序使用有效但具有过时数据的STALE_TOLERATED化视图以及包含新数据的视图。此模式提供最大重写功能，但会产生产生不准确结果的风险。

如果将重写完整性设置为最安全的级别ENFORCED ，则优化程序仅使用强制主键约束和参照完整性约束来确保查询结果与直接访问详细信息表时的结果相同。

如果重写完整性设置为ENFORCED以外的ENFORCED ，则有几种情况下重写的输出可能与没有重写的输出不同：

  • 物化视图可能与数据的主副本不同步。这通常是因为物化视图刷新过程在对物化视图的一个或多个详细信息表进行批量加载或DML操作之后处于未决状态。在某些数据仓库站点，这种情况是可取的，因为某些物化视图在某些时间间隔刷新并不罕见。

  • 维度对象隐含的关系无效。例如，层次结构中某个级别的值不会累积到恰好一个父值。

  • 存储在预构建的物化视图表中的值可能不正确。

  • 由于未强制的表或视图约束定义的错误数据关系，可能会出现错误的答案。

您可以在初始化参数文件中或使用ALTER SYSTEM或ALTER SESSION语句设置QUERY_REWRITE_INTEGRITY 。

11.2.5关于启用查询重写的权限

物化视图的使用不是基于用户在物化视图上具有的权限，而是基于用户对查询中的详细信息表或视图的权限。

系统特权GRANT QUERY REWRITE允许您在自己的模式中启用REWRITE化视图，以便仅在REWRITE化视图直接引用的所有表都在该模式中时才进行查询重写。GRANT GLOBAL QUERY REWRITE权限使您可以为查询重写启用REWRITE化视图，即使REWRITE化视图引用其他模式中的对象也是如此。或者，您可以对架构外的表和视图使用QUERY REWRITE对象特权。

使用实体化视图进行查询重写的权限与定义者权限过程的权限类似。

11.2.7如何验证是否发生查询重写

由于查询重写是透明的，因此必须采取特殊步骤来验证查询是否已被重写。当然，如果查询运行得更快，这应该表明已经发生了重写，但这不是证明。因此，要确认确实发生了查询重写，请使用EXPLAIN PLAN语句或DBMS_MVIEW.EXPLAIN_REWRITE过程。有关详细信息，请参阅“ 验证是否已发生查询重写 ” 。

11.3查询重写示例

此示例说明了使用实体化视图重写查询的强大功能。

请考虑以下物化视图cal_month_sales_mv ，它提供每月销售的美元金额的汇总：

CREATE MATERIALIZED VIEW cal_month_sales_mv ENABLE QUERY REWRITE as SELECT t.calendar_month_desc，SUM（s.amount_sold）AS $ FROM sales s，times t WHERE s.time_id = t.time_id GROUP BY t.calendar_month_desc;

让我们假设，在一个典型的月份，商店的销售数量约为一百万。因此，此物化聚合视图具有针对每个月销售的美元金额的预先计算的聚合。

请考虑以下查询，该查询要求每个日历月在商店销售的金额总和：

SELECT t.calendar_month_desc，SUM（s.amount_sold）FROM sales s，times t WHERE s.time_id = t.time_id GROUP BY t.calendar_month_desc;

如果没有先前的物化视图和查询重写功能，Oracle数据库必须直接访问sales表并计算sales金额的总和以返回结果。这涉及从sales表中读取数百万行，由于磁盘访问，这将总是增加查询响应时间。查询中的连接还将进一步减慢查询响应，因为需要在数百万行上计算连接。

在物化视图cal_month_sales_mv存在的情况下，查询重写将透明地将以前的查询重写为以下查询：

SELECT calendar_month，dollar FROM cal_month_sales_mv;

由于物化视图中只有几十行cal_month_sales_mv且没有连接，因此Oracle数据库会立即返回结果。
