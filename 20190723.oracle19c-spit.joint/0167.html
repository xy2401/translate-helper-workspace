关于初始化和调整并行执行的参数

您可以使用参数来初始化和调整并行执行。

Oracle数据库根据CPU_COUNT和PARALLEL_THREADS_PER_CPU数据库启动时的值计算并行执行参数的缺省值。还可以手动调整参数，增加或减少其值以适应特定的系统配置或性能目标。例如，在从不使用并行执行的系统上， PARALLEL_MAX_SERVERS可以设置为零。

您还可以手动调整并行执行参数。默认情况下启用并行执行。

PARALLEL_THREADS_PER_CPU

2

描述CPU在并行执行期间可以处理的并行执行进程或线程的数量。

强制会话的并行执行

Youu可以强制会话的并行性。

如果您确定要并行执行并希望避免为表设置DOP或修改所涉及的查询，则可以使用以下语句强制并行：

ALTER SESSION FORCE PARALLEL QUERY;

所有后续查询都是并行执行的，前提是没有违反任何限制。您还可以强制DML和DDL语句。此子句将覆盖会话中后续语句中指定的任何并行子句，但会被并行提示覆盖。

例如，在典型的OLTP环境中，表不是并行设置的，但是夜间批处理脚本可能希望并行地从这些表中收集数据。通过在会话中设置DOP，用户可以避免并行更改每个表，然后在完成时将其更改回串行。

PARALLEL_FORCE_LOCAL

PARALLEL_FORCE_LOCAL参数指定PARALLEL_FORCE_LOCAL执行的SQL语句是否仅限于Oracle RAC环境中的单个实例。

通过将此参数设置为TRUE ，可以将处理的并行服务器的范围限制为运行查询协调器的单个Oracle RAC实例。

PARALLEL_FORCE_LOCAL参数的建议值为FALSE 。

PARALLEL_MAX_SERVERS

PARALLEL_MAX_SERVERS参数指定实例的最大并行执行进程数和并行恢复进程数。

随着需求的增加，Oracle数据库会将实例启动时创建的数量增加到此值。

例如，将值设置为64使您可以同时运行四个并行查询，如果每个查询使用两个从属集，每组的DOP为8 。

当用户有太多进程时

当并发用户具有太多查询服务器进程时，可能发生内存争用（分页），I / O争用或过多的上下文切换。

这种争用可以将系统吞吐量降低到低于不使用并行执行的水平。仅当系统具有足够的内存和I / O带宽时才增加PARALLEL_MAX_SERVERS值。

您可以使用操作系统的性能监视工具来确定可以释放多少内存，交换空间和I / O带宽。查看CPU和磁盘的运行队列长度，以及系统上I / O操作的服务时间。验证系统是否有足够的交换空间来添加更多进程。限制查询服务器进程的总数可能会限制可以执行并行操作的并发用户数，但系统吞吐量趋于保持稳定。

何时限制使用使用者组的用户的资源数量

必要时，您可以通过为用户建立资源使用者组来限制给定用户可用的并行度。

这样做是为了限制任何一个用户或一组用户可以拥有的会话数，并发登录数和并行进程数。

处理并行执行语句的每个查询服务器进程都使用会话ID登录。每个进程都会计入用户的并发会话限制。例如，要将用户限制为10个并行执行进程，请将用户限制设置为11。一个进程用于并行执行协调器，另一个进程由两组查询服务器组成。这将允许并行执行协调器的一个会话和并行执行过程的10个会话。

PARALLEL_MIN_PERCENT

PARALLEL_MIN_PERCENT参数使用户可以等待可接受的DOP，具体取决于所使用的应用程序。

PARALLEL_MIN_PERCENT参数的建议值为0（零）。将此参数设置为0（零）以外的值会导致Oracle数据库在给定时间内系统无法满足请求的DOP时返回错误。例如，如果将PARALLEL_MIN_PERCENT设置为50（转换为50％），并且由于自适应算法或资源限制导致DOP降低50％或更多，则Oracle数据库将返回ORA-12827 。例如：

SELECT / * + FULL（e）PARALLEL（e，8）* / d.department_id，SUM（SALARY）FROM employees e，departments d WHERE e.department_id = d.department_id GROUP BY d.department_id ORDER BY d.department_id;

Oracle数据库使用以下消息进行响应：

ORA-12827：可用的并行查询从站不足

SHARED_POOL_SIZE

SHARED_POOL_SIZE参数指定共享池的内存大小。

除串行SQL执行所需的内存资源外，并行执行还需要内存资源。附加内存用于在查询服务器进程和查询协调器之间进行通信和传递数据。

Oracle数据库从共享池为查询服务器进程分配内存。调整共享池如下：

  • 允许共享池的其他客户端，例如共享游标和存储过程。

  • 请记住，较大的值可提高多用户系统的性能，但较小的值会占用较少的内存。

  • 然后，您可以监视并行执行使用的缓冲区数，并将shared pool PX msg pool与视图V$PX_PROCESS_SYSSTAT输出中报告的当前高水位标记进行V$PX_PROCESS_SYSSTAT 。

    注意：

    如果没有足够的可用内存，则会出现错误消息12853（PX缓冲区的内存不足：当前字符串 K，最大需要字符串 K）。这是因为PX缓冲区的SGA内存不足。必须重新配置SGA以至少具有（ MAX - CURRENT ）字节的附加内存。

默认情况下，Oracle数据库从共享池中分配并行执行缓冲区。

如果Oracle数据库在启动时显示以下错误，则应将SHARED_POOL_SIZE的值减小到足以使数据库启动：

ORA-27102：内存不足SVR4错误：12：空间不足

减少SHARED_POOL_SIZE的值后，您可能会看到错误：

ORA-04031：无法分配16084字节的共享内存（“SHARED池”，“未知对象”，“共享池堆”，“PX消息池”）

如果是，请执行以下查询以确定Oracle数据库无法分配16,084字节的原因：

SELECT VAME，SUM（BYTES）FROM V $ SGASTAT WHERE UPPER（POOL）='SHARED POOL'GROUP BY ROLLUP（NAME）;

您的输出应类似于以下内容：

NAME SUM（BYTES）-------------------------- ---------- PX msg pool 1474572 free memory 562132 2036704

如果指定SHARED_POOL_SIZE并且指定要保留的内存量大于池，则Oracle数据库不会分配它可以获取的所有内存。相反，它留下了一些空间。查询运行时，Oracle数据库会尝试获取所需内容。Oracle数据库使用560 KB，失败时需要另外16 KB。该错误不会报告所需的累计金额。确定需要多少内存的最佳方法是使用消息缓冲区的附加内存要求中的公式。

要解决当前示例中的问题，请增加SHARED_POOL_SIZE的值。如示例输出中所示， SHARED_POOL_SIZE大约为2 MB。根据可用的内存量，您可以将SHARED_POOL_SIZE的值增加到4 MB并尝试启动数据库。如果Oracle数据库继续显示ORA-4031消息，请逐渐增加SHARED_POOL_SIZE的值，直到启动成功为止。

消息缓冲区的附加内存要求

本主题讨论了使用并行执行计划时消息缓冲区和游标的附加内存要求。

确定共享池的初始设置后，必须计算消息缓冲区的其他内存要求，并确定游标需要多少额外空间。

消息缓冲区所需的内存

您必须增加SHARED_POOL_SIZE参数的值以适应消息缓冲区。消息缓冲区允许查询服务器进程相互通信。

Oracle数据库为生产者查询服务器和使用者查询服务器之间的每个虚拟连接使用固定数量的缓冲区。连接随着DOP的平方增加而增加。因此，并行执行使用的最大内存量受系统允许的最高DOP限制。您可以使用PARALLEL_MAX_SERVERS参数或使用策略和配置文件来控制此值。

要计算所需的内存量，请使用以下公式之一：

  • 对于SMP系统：

    mem in bytes =（3 x size x users x groups x connections）

  • 对于Oracle Real Application Clusters和MPP系统：

    mem in bytes =（（3 x local）+（2 x remote））x（size x users x groups）/ instances

每个实例使用公式计算的内存。

条款是：

  • SIZE = PARALLEL_EXECUTION_MESSAGE_SIZE

  • USERS =您希望以最佳DOP运行的并发并行执行用户数

  • GROUPS =每个查询使用的查询服务器进程组的数量

    一个简单的SQL语句只需要一个组。但是，如果您的查询涉及并行处理的子查询，则Oracle数据库会使用另一组查询服务器进程。

  • CONNECTIONS =（DOP 2 + 2 x DOP）

    如果您的系统是集群或MPP，那么您应考虑实例数，因为这会增加DOP。换句话说，在双实例集群上使用DOP为4会导致DOP为8。PARALLEL_MAX_SERVERS值乘以实例数除以4是保守估计值，用作起点。

  • LOCAL = CONNECTIONS / INSTANCES

  • REMOTE = CONNECTIONS - LOCAL

将此金额添加到共享池的原始设置。但是，在为这些内存结构中的任何一个设置值之前，还必须考虑游标的额外内存，如以下部分所述。

游标的额外内存

影响资源消耗的参数

本主题讨论了影响资源消耗的参数。

注意：

在考虑下面的部分，你应该阅读的描述MEMORY_TARGET和MEMORY_MAX_TARGET对细节的初始化参数。无需将PGA_AGGREGATE_TARGET初始化参数设置为MEMORY_TARGET自动调整SGA和PGA组件。

本节中讨论的第一组参数影响所有并行操作的内存和资源消耗，特别是并行执行。这些参数是：

  • PGA_AGGREGATE_TARGET

  • PARALLEL_EXECUTION_MESSAGE_SIZE

在影响并行DML和并行DDL的资源消耗的参数中讨论了第二个参数子集。

要控制资源消耗，您应该在两个级别配置内存：

  • 在数据库级别，系统使用来自操作系统的适当内存量。

  • 在操作系统级别以保持一致性。

    在某些平台上，您可能需要设置控制可用虚拟内存总量的操作系统参数，并在所有进程中进行总计。

数据仓库操作中使用的大部分内存（与OLTP相比）更具动态性。此内存来自Process Global Area（PGA），进程内存的大小和进程数可以有很大差异。在这种情况下，使用PGA_AGGREGATE_TARGET初始化参数来控制进程内存和进程数。显式设置PGA_AGGREGATE_TARGET和MEMORY_TARGET可确保仍然发生自动调整，但PGA_AGGREGATE_TARGET未调整到低于指定值。

您可以通过初始化参数的设置启用自动PGA内存管理，例如PGA_AGGREGATE_TARGET 。

交易

TRANSACTIONS参数影响并行DML和DDL下的事务数。

对于并行DML和DDL，每个查询服务器进程都会启动一个事务。并行执行协调器使用两阶段提交协议来提交事务;因此，正在处理的交易数量因DOP而增加。因此，您可能需要增加TRANSACTIONS初始化参数的值。

TRANSACTIONS参数指定最大并发事务数。TRANSACTIONS的默认值假定没有并行性。例如，如果您的DOP为20，则还有20个新服务器事务（如果有两个服务器集，则为40个）和1个协调器事务。在这种情况下，如果事务在同一实例中运行，则应将TRANSACTIONS增加21（或41）。如果未设置此参数，Oracle数据库会将其设置为等于1.1 x SESSIONS的值。如果您使用服务器管理的撤消，则此讨论不适用。

FAST_START_PARALLEL_ROLLBACK

如果系统在存在未提交的并行DML或DDL事务时失败，则可以使用FAST_START_PARALLEL_ROLLBACK参数在启动期间加快事务恢复。

FAST_START_PARALLEL_ROLLBACK参数控制恢复已终止事务时使用的DOP。终止事务是在系统发生故障之前处于活动状态的事务。默认情况下，DOP选择最多为CPU_COUNT参数值的CPU_COUNT 。

如果默认DOP不足，请将参数设置为HIGH 。这使得最大DOP最多为CPU_COUNT参数值的CPU_COUNT 。默认情况下，此功能可用。

DML_LOCKS

DML_LOCKS参数设置为考虑并行DML操作所持有的锁数。

DML_LOCKS参数指定DML锁的最大数量。其值应等于所有用户引用的所有表上的锁的总数。并行DML操作的锁定要求与串行DML非常不同。并行DML包含更多锁，因此您应该将DML_LOCKS参数的值增加相等的量。

关于OCRCONFIG

使用ocrconfig命令管理OCR。使用此实用程序，您可以导入，导出，添加，删除，还原，覆盖，备份，修复，替换，移动，升级或降级OCR。

  • OCRCONFIG可执行文件位于Grid_home/bin目录中

  • ocrconfig命令语法如下：

    ocrconfig -option

使用Utility帮助

要显示OCRCONFIG实用程序的帮助输出：

ocrconfig -help

特权和安全

要使用OCRCONFIG实用程序，您必须以具有管理权限的用户身份登录操作系统。

日志文件

OCRCONFIG实用程序在ORACLE_BASE /diag/crs/ host_name /crs创建一个日志文件。

要更改日志记录量，请编辑ORACLE_BASE /crsdata/ host_name /crsdiag/ocrconfig.ini文件中的路径。在这种情况下， ORACLE_BASE用于Oracle Grid Infrastructure软件主页。

ocrconfig -add

将OCR位置添加到Oracle自动存储管理（Oracle ASM）磁盘组。

ocrconfig -add + diskgroup_name

使用说明

  • 您添加的OCR位置必须存在，具有足够的权限，并且必须先装入才能添加它们。

  • 您必须以root身份运行此命令。

  • diskgroup_name变量是Oracle ASM磁盘组的名称。例如：

    • +newdg ：确保磁盘组存在且已装入

      磁盘组的名称前面必须加号（ + ）。

    也可以看看：

    • 有关创建OCR位置和设置正确权限的信息，请参阅“Oracle Grid Infrastructure安装和升级指南”

    • 有关Oracle ASM磁盘组管理的详细信息，请参见“Oracle自动存储管理管理员指南”

例

ocrconfig -backuploc

指定OCR备份目录位置。

句法

ocrconfig -backuploc + disk_group_name ocrconfig -local -backuploc dir_name

使用说明

  • 您必须以root身份运行此命令。

  • 使用-local选项指定OLR备份目录位置。

  • 对于OCR备份位置， disk_group_name变量必须是Oracle ASM磁盘组。例如：

    +bkupdg ：确保Oracle ASM磁盘组存在并挂载在所有节点上

    Oracle ASM磁盘组名称前面必须加号（+）。

  • 对于OLR备份位置， dir_name变量可以是完整目录路径名。例如：

    • Grid_home /cdata/cluster3/ ：确保该位置存在

    • d:\cdata\cluster3 ：确保该位置存在

    在Linux或UNIX系统上生成OLR备份的默认位置是Grid_home /cdata/ host_name ，其中host_name是要备份的OLR所在节点的名称。用于生成OLR备份的Windows默认位置使用相同的路径结构。

#ocrconfig -backuploc + bkupdg

要指定OLR备份位置：

full_directory_path -local -backuploc full_directory_path

9.1动态SQL

大多数数据库应用程序执行特定的工例如，一个简单的程序可能会提示用户输入员工编号，然后更新EMP和DEPT表中的行。在这种情况下，您知道在预编译时UPDATE语句的组成。也就是说，您知道可能更改哪些表，为每个表和列定义的约束，可能更新的列以及每列的数据类型。

但是，某些应用程序必须在运行时接受（或构建）和处理各种SQL语句。例如，通用报表编写者必须为其生成的各种报表构建不同的SELECT语句。在这种情况下，语句的构成在运行时才会被识别。这些陈述可以并且可能会从执行变为执行。它们恰当地称为动态 SQL语句。

与静态SQL语句不同，动态SQL语句未嵌入源程序中。相反，它们存储在运行时输入或由程序构建的字符串中。它们可以交互式输入或从文件中读取。

9.2动态SQL的优缺点

接受和处理动态定义的SQL语句的主机程序比普通的嵌入式SQL程序更通用。可以使用对SQL知之甚少或根本不了解的用户的输入以交互方式构建动态SQL语句。

例如，您的程序可能只是提示用户在SELECT，UPDATE或DELETE语句的WHERE子句中使用搜索条件。更复杂的程序可能允许用户从列出SQL操作，表和视图名称，列名称等的菜单中进行选择。因此，动态SQL允许您编写高度灵活的应用程序。

但是，一些动态查询需要复杂的编码，特殊数据结构的使用以及更多的运行时处理。虽然您可能没有注意到增加的处理时间，但除非您完全理解动态SQL概念和方法，否则您可能会发现编码很困难。

9.3何时使用动态SQL

实际上，静态SQL几乎可以满足您的所有编程需求。仅在需要开放式灵活性时才使用动态SQL。当预编译时未知以下一项或多项时，建议使用它：

  • SQL语句的文本（命令，子句等）

  • 主机变量的数量

  • 主机变量的数据类型

  • 对数据库对象的引用，例如列，索引，序列，表，用户名和视图

9.4动态SQL语句的要求

要表示动态SQL语句，字符串必须包含有效DML或DDL SQL语句的文本，但不包含EXEC SQL子句，主机语言分隔符或语句终止符。

在大多数情况下，字符串可以包含虚拟主机变量。它们在SQL语句中保存实际主机变量的位置。因为虚拟主机变量只是占位符，所以不要声明它们并且可以将它们命名为任何你喜欢的（不允许使用连字符）。例如，Oracle不区分以下两个字符串

'从EMP删除MGR =：MGRNUMBER和JOB =：JOBTITLE''从EMP删除MGR =：M和JOB =：J'

9.5如何处理动态SQL语句

通常，应用程序会提示用户输入SQL语句的文本以及语句中使用的主机变量的值。然后Oracle 解析 SQL语句。也就是说，Oracle检查SQL语句以确保它遵循语法规则并引用有效的数据库对象。解析还涉及检查数据库访问权限，保留所需资源以及查找最佳访问路径。

接下来，Oracle 将主机变量绑定到SQL语句。也就是说，Oracle获取主机变量的地址，以便它可以读取或写入它们的值。

如果该语句是查询，则定义SELECT变量，然后Oracle FETCH它们，直到检索到所有行。然后关闭光标。

然后Oracle 执行 SQL语句。也就是说，Oracle执行SQL语句请求的内容，例如从表中删除行。

可以使用主机变量的新值重复执行SQL语句。

9.6.5指南

使用所有四种方法，必须将动态SQL语句存储在字符串中，该字符串必须是主变量或带引号的文字。将SQL语句存储在字符串中时，请省略关键字EXEC SQL和语句终止符。

使用方法2和3，必须在预编译时知道输入主变量的占位符数和输入主变量的数据类型。

每个后续方法对应用程序施加的约束较少，但编码起来更加困难。通常，使用最简单的方法。但是，如果方法1将重复执行动态SQL语句，请使用方法2来避免为每次执行重新解析。

方法4提供了最大的灵活性，但需要复杂的编码和对动态SQL概念的充分理解。通常，仅当您不能使用方法1,2或3时才使用方法4。

9.6.5.1避免常见错误

如果使用字符数组存储动态SQL语句，请在存储SQL语句之前对数组进行空白填充。这样，你就清除了无关紧要的角色。当您将数组重用于不同的SQL语句时，这一点尤为重要。通常，在存储SQL语句之前始终初始化（或重新初始化）主机字符串。

不要空终止主机字符串。Oracle不会将null终止符识别为字符串结束标记。相反，Oracle将其视为SQL语句的一部分。

如果使用VARCHAR变量存储动态SQL语句，请确保在执行PREPARE或EXECUTE IMMEDIATE语句之前正确设置（或重置）VARCHAR的长度。

EXECUTE重置SQLCA中的SQLWARN警告标志。因此，要捕获诸如无条件更新（由省略WHERE子句引起）之类的错误，请在执行PREPARE语句之后但在执行EXECUTE语句之前检查SQLWARN标志。

9.11使用方法3

9.13.1需要SQLDA

要处理这种动态查询，您的程序必须发出DESCRIBE SELECT LIST命令并声明一个名为SQL Descriptor Area（SQLDA）的数据结构。因为它包含查询选择列表中列的描述，所以此结构也称为选择描述符 。

同样，如果动态SQL语句包含输入主机变量的未知数量的占位符，则USING子句无法在预编译时建立主机变量列表。

9.13.2 DESCRIBE声明

DESCRIBE初始化描述符以保存选择列表项或输入主变量的描述。

如果提供选择描述符，则DESCRIBE SELECT LIST语句将检查准备好的动态查询中的每个选择列表项，以确定其名称，数据类型，约束，长度，比例和精度。然后它将此信息存储在选择描述符中。

9.16使用PL / SQL

Pro * COBOL将PL / SQL块视为单个SQL语句。因此，像SQL语句一样，PL / SQL块可以存储在字符串主变量或文字中。将PL / SQL块存储在字符串中时，省略关键字EXEC SQL EXECUTE，关键字END-EXEC和语句终止符。

但是，Pro * COBOL处理SQL和PL / SQL的方式有两点不同：

  • 所有PL / SQL主机变量都应以与输入主机变量相同的方式处理，无论它们是输入还是输出主变量（或两者）。

  • 您无法从PL / SQL块进行FETCH，因为它可能包含任意数量的SQL语句。但是，您可以使用游标变量实现类似的功能。

9.16.1使用方法1

如果PL / SQL块不包含主机变量，则可以使用方法1以常规方式执行PL / SQL字符串。

9.16.2使用方法2

如果PL / SQL块包含已知数量的输入和输出主机变量，则可以使用方法2以常规方式PREPARE和EXECUTE PL / SQL字符串。

您必须将所有主变量放在USING子句中。完成PL / SQL字符串EXECUTE后，USING子句中的主机变量会在PREPARE之后替换字符串中的相应占位符。虽然Pro * COBOL将所有PL / SQL主机变量视为输入主机变量，但值是正确分配的。输入（程序）值分配给输入主变量，输出（列）值分配给输出主变量。

PREPARE之后的PL / SQL字符串中的每个占位符必须与USING子句中的主机变量相对应。因此，如果相同的占位符在PREPAREd字符串中出现两次或多次，则每个外观必须对应于USING子句中的主变量。

9.16.3使用方法3

方法2和3是相同的，除了方法3允许完成FETCH。由于您无法从PL / SQL块进行FETCH，因此请改用方法2。

9.16.4使用方法4

如果PL / SQL块包含未知数量的输入或输出主机变量，则必须使用方法4。

要使用方法4，请为所有输入和输出主机变量设置一个绑定描述符。执行DESCRIBE BIND VARIABLES将有关输入和输出主机变量的信息存储在绑定描述符中。因为您使用与输入主机变量关联的方法引用所有PL / SQL主机变量，所以执行DESCRIBE SELECT LIST无效。

在主机语言补充中详细介绍了使用方法4绑定描述符。

请注意，在动态SQL方法4中，主机数组不能绑定到具有“table”类型参数的PL / SQL过程。

9.16.5注意

不要在将动态处理的PL / SQL块中使用ANSI样式的注释（ - - ...），因为忽略了行尾字符。因此，ANSI样式的注释扩展到块的末尾，而不仅仅是行的末尾。相反，使用C风格的评论（/ * ...* /）。

2 Sharded Database Schema Design

为了获得分片的好处，分片数据库的模式应该以最大化在单个分片上执行的数据库请求数的方式设计。

以下主题描述了用于此目的的对象：

  • Sharded Tables
    分片表是一个表，在多个数据库之间划分为更小且更易于管理的部分，称为分片。
  • Sharded Table Family
    分片表族是一组以相同方式分片的表。
  • 重复的表格
    除了分片表，SDB还可以包含在所有分片上重复的表。
  • 在所有碎片上创建的非表对象
    除了重复的表之外，其他模式对象（例如用户，角色，视图，索引，同义词，函数，过程和包）以及非模式数据库对象（例如表空间，表空间集，目录和上下文）可以是在所有分片上创建。
  • Sharded数据库中的DDL执行
    要在SDB中创建模式，必须在分片目录数据库上发出DDL命令，该数据库在分片上执行之前验证DDL并在本地执行它们。
  • Sharded数据库中的PL / SQL过程执行
    与在配置中的所有分片上执行DDL语句的方式相同，Oracle提供的某些PL / SQL过程也是如此。这些特定过程调用的行为就像它们是分片DDL语句一样，它们被传播到所有分片，由目录跟踪，并在每次将新分片添加到配置时运行。
  • 在Shards上生成唯一序列号
    Oracle Sharding允许您为非主键列生成跨分片的全局唯一序列号，并由分片数据库处理。

2.1分类表

分片表是一个表，在多个数据库之间划分为更小且更易于管理的部分，称为分片。

分片的主要好处包括：

  • 线性可扩展性 。分片消除了性能瓶颈，并且可以通过添加分片来线性扩展性能和容量。

  • 故障遏制 。Sharding是一种无共享硬件基础架构，可消除单点故障，例如共享磁盘，SAN和集群件，并提供强大的故障隔离 - 一个分片的故障或速度降低不会影响其他分片的性能或可用性。

  • 数据的地理分布 。当数据必须位于特定的管辖区域时，分片可以将特定数据存储在靠近其消费者的位置并满足法规要求。

  • 滚动升级 。一次在一个分片上应用配置更改不会影响其他分片，并允许管理员首先测试一小部分数据的更改。

  • 云部署的简单性 。分片非常适合在云中部署。可以根据需要调整碎片的大小以适应可用的任何云基础设施并且仍然实现所需的服务级别。Oracle Sharding支持内部部署，云和混合部署模型。

应用程序将数据库池视为单个逻辑数据库。通过向池中添加数据库（分片），应用程序可以在任何平台上弹性地将数据，事务和用户扩展到任何级别。Oracle数据库支持最多可扩展1000个分片。

分片和分区

Oracle Sharding本质上是分布式分区，因为它通过支持跨分片的表分区分发来扩展分区。

基于分片键，分区在表空间级别的分片中分布。密钥的示例包括客户ID，帐号和国家ID。分片键支持以下数据类型：

  • 数

  • 整数

  • SMALLINT

  • 生的

  • （N）VARCHAR

  • （N）VARCHAR2

  • （N）CHAR

  • 日期

  • TIMESTAMP

分片表的每个分区都驻留在一个单独的表空间中，每个表空间都与一个特定的分片相关联。根据分片方法，可以自动建立关联或由管理员定义关联。

即使分片表的分区驻留在多个分片中，对于应用程序，该表的外观和行为与单个数据库中的分区表完全相同。应用程序发出的SQL语句永远不必引用分片或依赖于分片数量及其配置。

CREATE SHARDED TABLE客户（cust_id NUMBER NOT NULL，名称VARCHAR2（50），地址VARCHAR2（250），区域VARCHAR2（20），类VARCHAR2（3），注册DATE CONSTRAINT cust_pk PRIMARY KEY（cust_id））PARTITION BY CONSISTENT HASH（cust_id ）PARTITIONS AUTO TABLESPACE SET ts1;

上表由一致哈希分区，哈希是一种在可伸缩分布式系统中常用的特殊类型的哈希分区。此技术自动跨分片传播表空间，以提供均匀的数据和工作负载分布。请注意，不支持分片表上的全局索引，但支持本地索引。

表空间集

Oracle Sharding创建和管理表空间作为一个称为表空间集的单元。PARTITIONS AUTO子句指定应自动确定分区数。这种类型的散列在分片之间迁移数据方面提供了更大的灵活性和效率，这对于弹性可伸缩性非常重要。

表空间是SDB中数据分布的逻辑单元。跨分片的分区分布是通过在驻留在不同分片上的表空间中自动创建分区来实现的。为了最小化多分片连接的数量，相关表的相应分区始终存储在同一分片中。分片表的每个分区都存储在单独的表空间中。

注意：

表空间集仅支持Oracle托管文件。

不能独立于整个表空间集删除或更改单个表空间。

TABLESPACE SET不能与用户定义的分片方法一起使用。

2.2 Sharded Table Family

分片表族是一组以相同方式分片的表。

通常，数据库表之间存在父子关系，子表（外键）中的引用约束引用父表的主键。通过这种关系链接的多个表通常形成树状结构，其中每个孩子具有单个父母。一组这样的表被称为表族。表系列中没有父表的表称为根表。表系列中只能有一个根表。

使用CREATE TABLE创建分片表族

创建分片表系列的推荐方法是使用引用分区指定表之间的父子关系。

Customers-Orders-LineItems模式的相应CREATE TABLE语句如下所示。第一个语句创建表系列的根表 - Customers。 

CREATE SHARDED TABLE客户（CustNo NUMBER NOT NULL，Name VARCHAR2（50），Address VARCHAR2（250），CONSTRAINT RootPK PRIMARY KEY（CustNo））PARTITION BY CONSISTENT HASH（CustNo）PARTITIONS AUTO TABLESPACE SET ts1;

以下两个语句创建Orders和LineItems表，它们是Customers表的子项和孙项。

CREATE SHARDED TABLE Orders（OrderNo NUMBER NOT NULL，CustNo NUMBER NOT NULL，OrderDate DATE，CONSTRAINT OrderPK PRIMARY KEY（CustNo，OrderNo），CONSTRAINT CustFK FOREIGN KEY（CustNo）REFERENCES Customers（CustNo））PARTITION BY REFERENCE（CustFK）;
CREATE SHARDED TABLE LineItems（CustNo NUMBER NOT NULL，LineNo NUMBER（2）NOT NULL，OrderNo NUMBER（5）NOT NULL，StockNo NUMBER（4），Quantity NUMBER（2），CONSTRAINT LinePK PRIMARY KEY（CustNo，OrderNo，LineNo）， CONSTRAINT LineFK FOREIGN KEY（CustNo，OrderNo）REFERENCES Orders（CustNo，OrderNo））PARFERTION BY REFERENCE（LineFK）;

在上面的示例语句中，系列中所有表的相应分区存储在同一个表空间集-TS1中。但是，可以为每个表指定单独的表空间集。

通过引用进行分区简化了语法，因为仅为根表指定了分区方案。此外，在根表上执行的分区管理操作会自动传播到其后代。例如，在向根表添加分区时，会在其所有后代上创建新分区。

请注意，在上面的示例语句中，用作分片键的分区列CustNo存在于所有三个表中。尽管参数分区通常允许子表与父表等分，而不必复制子表中的键列。这样做的原因是引用分区需要父表中的主键，因为必须在用于将子项链接到其父项的子表的外键约束中指定主键。但是，分片表上的主键必须与分片键相同，或者包含分片键作为前导列。这使得可以在不与其他分片协调的情况下实施主键的全局唯一性 - 这是线性可伸缩性的关键要求。

总而言之，在分片数据库中使用引用分区表需要遵守以下规则：

  • 分片表上的主键必须与分片键相同，或者与分片键前缀的其他列相同。这是在不与其他分片协调的情况下强制实施主键的全局唯一性所必需的。

  • 引用分区需要父表中的主键，因为必须在子表的外键约束中指定主键以将子项链接到其父表。例如，要将LineItems（子）表链接到Orders（父）表，您需要Orders表中的主键。第二条规则意味着Orders表中的主键以CustNo值为前缀。（这是一个不特定于Oracle Sharding的现有分区规则。）

在某些情况下，创建引用分区所需的主键和外键约束是不可能或不可取的。对于这种情况，在表系列中指定父子关系要求所有表都是显式CREATE SHARDED TABLE ，并且每个子表都是使用CREATE SHARDED TABLE中的PARENT子句创建的，该子句包含其父级的名称。语法示例如下所示。 

CREATE SHARDED TABLE客户（CustNo NUMBER NOT NULL，Name VARCHAR2（50），Address VARCHAR2（250），region VARCHAR2（20），class VARCHAR2（3），signup DATE）PARTITION BY CONSISTENT HASH（CustNo）PARTITIONS AUTO TABLESPACE SET ts1;创建SHARDED TABLE订单（OrderNo NUMBER，CustNo NUMBER NOT NULL，OrderDate DATE） PARENT客户按一致哈希分区（CustNo）PARTITIONS AUTO TABLESPACE SET ts1; CREATE SHARDED TABLE LineItems（LineNo NUMBER，OrderNo NUMBER，CustNo NUMBER NOT NULL，StockNo NUMBER，Quantity NUMBER） PARENT Customers PARTITION BY CONSISTENT HASH（CustNo）PARTITIONS AUTO TABLESPACE SET ts1;

由于在所有CREATE SHARDED TABLE语句中都完全指定了分区方案，因此任何表都可以独立地进行子分区。引用分区不允许这样做，其中只能为根表指定子分区，并且子表划分方案对于表系列中的所有表都是相同的。

请注意，此方法仅支持两级表系列，即所有子级必须具有相同的父级和子级才能存在。只要父表中的分区列存在于所有子表中，这不是限制。

大块

分片之间的数据迁移单位是一个块。块是一组表空间，用于存储表系列中所有表的相应分区。块包含来自一组相关表的每个表的单个分区。这保证了来自不同分片表的相关数据可以一起移动。创建SDB时，将指定每个分片中的块数。

Sharded数据库中的多个表系列

注意：

在Oracle Database 19c中，Oracle Sharding包括对多个表系列的支持。此功能仅适用于系统管理的分片数据库。复合和用户定义的分片数据库仅支持一个表系列。

分片数据库可以有多个表系列，其中来自不同表系列的所有数据都位于相同的块中，这些块包含来自共享相同散列键范围的不同表系列的分区。跨表族查询应该是最小的，并且只在分片协调器上执行。每个表族都与不同的全局服务相关联。来自不同表系列的应用程序各自拥有自己的连接池和服务，并使用自己的分片键路由到正确的分片。

每个表族都由其根表标识。不同表族中的表不应相互关联。每个表族都应该有自己的分片键定义，而在子表中具有相同分片键列的相同限制在每个表族中仍然适用。这意味着来自不同表系列的所有表都以相同的方式进行分片，并将相同的哈希值分成相同数量的块，每个块包含来自所有表系列的数据。

以下示例说明如何使用PARENT子句创建多个表系列。

如果现有表系列已使用表空间集并且您尝试将其用于另一个表系列，则抛出ORA-3850。

GDSCTL> MODIFY SERVICE -GDSPOOL shdpool -TABLE_FAMILY sales.customer -SERVICE sales

跨表系列的连接可能效率不高，如果您有许多此类连接，或者它们对性能至关重要，则应使用重复的表而不是多个表系列。

2.3重复的表格

除了分片表，SDB还可以包含在所有分片上重复的表。

对于许多应用程序，通过在所有分片中复制只读或主要读取表，可以最大化单个分片处理的数据库请求数。对于通常与分片表一起访问的相对较小的表，此策略是一个不错的选择。每个分片中具有相同内容的表称为重复表。

SDB包括跨分片水平分区的分片表，以及复制到所有分片的重复表。重复的表包含参考信息，例如，每个分片共有的Stock Items表。分片和重复表的组合使得与分片键关联的所有事务都可以由单个分片处理。该技术实现了线性可扩展性和故障隔离。

作为需要复制表的示例，请考虑Sharded Table Family中描述的表族 。数据库模式还可能包含Products表，该表包含为此表系列创建的分片中的所有客户共享的数据，并且无法按客户编号进行分片。要在订单处理期间防止多分片查询，必须在所有分片上复制整个表。

2.5 Sharded数据库中的DDL执行

要在SDB中创建模式，必须在分片目录数据库上发出DDL命令，该数据库在分片上执行之前验证DDL并在本地执行它们。 因此，分片目录数据库包含分片数据库中存在的所有对象的本地副本，并充当SDB架构的主副本。如果目录验证和DDL的执行成功，则DDL会自动传播到所有分片，并按照分片目录上的顺序应用。

如果在DDL传播期间分片已关闭或无法访问，则目录会跟踪无法应用于分片的DDL，然后在分片备份时应用它们。当新的分片添加到SDB时，在SDB中执行的所有DDL在客户端可访问之前以相同的顺序应用于分片。

您可以通过两种方式在SDB中发布DDL：

  • 使用GDSCTL sql命令。

    以这种方式发出DDL时，GDSCTL会等待所有分片完成执行DDL并返回执行状态。例如

    GDSCTL> sql“create tablespace set tbsset”

  • 使用GDS$CATALOG. sdbname使用SQL * Plus连接到分片目录数据库GDS$CATALOG. sdbname服务。例如

    SQL> create tablespace set tbsset;

    在分片目录数据库上发出DDL命令时，它会在本地完成执行时返回状态，但DDL到所有分片的传播会在后台异步发生。

验证DDL传播

您可以使用GDSCTL show ddl和config shard命令检查DDL传播到分片的状态。在分片目录上使用SQL * Plus执行DDL时，必须执行此检查，因为SQL * Plus不会在所有分片上返回执行状态。当分片上的DDL失败时，该分片上的所有其他DDL都将被阻止，直到故障得到解决并执行GDSCTL recover shard命令。请注意，用户必须具有GSM_ADMIN权限才能执行这些GDSCTL命令。

在本地和全球创建对象

使用GDSCTL sql命令发出创建对象的DDL时，将在所有分片上创建对象。还在分片目录数据库中创建该对象的主副本。存在于所有分片上的对象和目录数据库称为SDB对象 。

使用SQL * Plus连接到分片目录时，可以创建两种类型的对象：SDB对象和本地对象。本地对象是仅存在于分片目录中的传统对象。例如，本地对象可用于管理目的，或者它们可以由源自目录数据库的多分片查询使用，以生成和存储报告。

在SQL * Plus会话中创建的对象类型（SDB或本地）取决于是否在会话中启用了SHARD DDL模式。默认情况下，在SDB用户的分片目录数据库上启用此模式 - 存在于所有分片和分片目录数据库中的用户。在会话中启用SHARD DDL创建的所有对象都是SDB对象。要创建本地对象，SDB用户必须首先运行alter session disable shard ddl 。禁用SHARD DDL创建的所有对象都是本地对象。要在会话中启用SHARD DDL ，SDB用户必须运行alter session enable shard ddl 。

有关SHARD DDL会话参数的详细信息，请参阅ALTER SESSION 。

创建SDB用户以创建架构对象

仅存在于分片目录数据库中的本地用户没有在SDB中创建模式对象的权限。因此，创建SDB模式的第一步是通过以SYSDBA连接到分片目录数据库，启用SHARD DDL并执行CREATE USER命令来创建SDB用户。当SDB用户连接到目录数据库时，默认情况下启用SHARD DDL模式。

注意：

如果本地用户启用SHARD DDL模式，则可以创建非模式SDB对象，例如表空间，目录和上下文;但是，它们无法创建模式SDB对象，例如表，视图，索引，函数，过程等。

分片对象不能对本地对象有任何依赖性。例如，您无法在本地表上创建所有分片视图。

2.5.1 Sharded数据库的DDL语法扩展

Oracle Sharding引入了对SQL DDL语句的更改。具有此语法的DDL语句只能对分片数据库执行。

请注意，不需要更改查询和DML语句来支持Oracle Sharding，并且对DDL语句的更改非常有限。大多数现有的DDL语句在分片数据库上的工作方式相同，语法和语义与常规Oracle数据库相同。

创建TABLESPACE SET

这是为Oracle Sharding引入的新声明。它的语法类似于CREATE TABLESPACE 。

CREATE TABLESPACE SET tablespace_set [IN SHARDSPACE shardspace] [USING TEMPLATE（{MINIMUM EXTENT size_clause | BLOCKSIZE integer [K] | logging_clause | FORCE LOGGING | ENCRYPTION tablespace_encryption_spec | DEFAULT [table_compression] storage_clause | {ONLINE | OFFLINE} | extent_management_clause | segment_management_clause | flashback_mode_clause} ...）];

该语句创建一个表空间集，可用作一个或多个分片表和索引的逻辑存储单元。表空间集由多个Oracle表空间组成，这些表空间分布在分片空间中的分片中。

请注意，在系统管理的分片中，分片数据库中只有一个默认分片空间。表空间集中的表空间数自动确定，并且等于相应分片空间中的块数。

表空间集中的所有表空间都是bigfile并具有相同的属性。属性在USING TEMPLATE子句中指定。此子句与典型表空间的permanent_tablespace_clause相同，但不能在datafile_tempfile_spec子句中指定数据文件名。表空间集中每个表空间的数据文件名称将自动生成。

请注意，表空间集只能包含永久表空间，没有系统，撤消或临时表空间集。

例

使用模板创建TAPLESPACE SET TSP_SET_1在SHARDSPACE sgr1中（数据文件大小100米扩展管理本地区域空间管理自动）;

ALTER TABLESPACE SET

无法修改表空间集的shardspace属性。表空间集的所有其他属性都可以像常规永久表空间一样进行更改。由于表空间集中的表空间是bigfile，因此不支持ADD DATAFILE和DROP DATAFILE子句。

DROP TABLESPACE SET和PURGE TABLESPACE SET

这些语句的语法和语义类似于DROP和PURGE TABLESPACE语句。

创建表

此语句已扩展为允许创建分片和重复表以及表系列的规范。

句法

CREATE [{GLOBAL TEMPORARY | SHARDED | DUPLICATED}] TABLE [架构。 ] table {relational_table | object_table | XMLType_table} [PARENT [schema。 ]表];

CREATE TABLE语句的以下部分旨在支持Oracle Sharding：

  • SHARDED和DUPLICATED关键字表示表内容分别跨分片分区或在所有分片上重复。DUPLICATED关键字是创建重复表的唯一语法更改。下面描述的所有其他更改仅适用于分片表。

  • PARENT子句将分片表链接到其表族的根表。

  • 要创建分片表，请使用TABLESPACE SET代替TABLESPACE 。包含TABLESPACE所有子句都扩展为包含TABLESPACE SET 。

  • table_partitioning_clauses三个子句： consistent_hash_partitions ， consistent_hash_with_subpartitions和partition_set_clause 。 

    table_partitioning_clauses :: = {range_partitions | hash_partitions | list_partitions | composite_range_partitions | composite_hash_partitions | composite_list_partitions | reference_partitioning | system_partitioning | consistent_hash_partitions | consistent_hash_with_subpartitions | partition_set_clause }

当前版本中分片表的限制：

  • 没有为分片表设置默认表空间。

  • 临时表无法分片或重复。

  • 不支持以索引为单位的分片表。

  • 分片表不能包含嵌套表列或标识列。

  • 在分片表上定义的主键约束必须包含分片列。不支持引用重复表列的分片表的列上的外键约束。

  • 分片表不支持系统分区和间隔范围分区。通过一致性散列进行分区不支持单个散列分区的规范。

  • PARTITION BY或PARTITIONSET BY子句中使用的分片表中的列不能是虚拟列。

以下不支持当前版本中的重复表：

  • 系统和引用分区表

  • LONG，abstract（支持MDSYS数据类型），REF数据类型

  • 没有主键的最大列数为999

  • nologging ， parallel ， inmemory选项

  • 重复表中的XMLType列不能在非ASSM表空间中使用

例

CREATE SHARDED TABLE客户（cust_id NUMBER NOT NULL，名称VARCHAR2（50），地址VARCHAR2（250），location_id VARCHAR2（20），类VARCHAR2（3），signup_date DATE，CONSTRAINT cust_pk PRIMARY KEY（cust_id，class））PARTITIONSET BY LIST （类）由一致哈希分区（cust_id）PARTITIONS AUTO（PARTITIONSET金币值（'gld'）TABLESPACE SET ts2，PARTITIONSET银价值（'slv'）TABLESPACE SET ts1）;

更改表

系统管理或复合分片数据库中的分片表不支持以下选项：

  • 改名

  • 添加外键约束

  • 单个分区和子分区上的所有操作

  • 分片上与分区相关的所有操作，TRUNCATE分区，UNUSABLE LOCAL INDEXES和REBUILD UNUSABLE LOCAL INDEXES除外

重复表不支持以下内容：

  • 数据类型：long，abstract（支持MDSYS数据类型），REF

  • 列选项：矢量编码，不可见列，嵌套表

  • 对象类型

  • 集群表

  • 外部表

  • ILM政策

  • PARENT条款

  • 闪回表操作

  • 系统和参考分区

  • 启用nologging选项

  • 截断表

  • 删除重复的表实体化视图日志

  • 删除分片上的重复表实体化视图

  • 更改分片上的物化视图（重复表）

改变会话

会话级SHARD DDL参数设置针对分片目录数据库发出的DDL的范围。 

ALTER SESSION {ENABLE | DISABLE} SHARD DDL ;

启用SHARD DDL ，会话中发出的所有DDL都在分片目录和所有分片上执行。禁用SHARD DDL ，仅对分片目录数据库执行DDL。默认情况下，SDB用户（存在于所有分片和目录中的用户）启用SHARD DDL 。要创建SDB用户，必须在运行CREATE USER之前启用SHARD DDL参数。

  2. 使用alter session enable shard ddl启用分片DDL。
  3. 使用名为SYS.EXEC_SHARD_PLSQL的特定于分片的PL / SQL过程运行目标过程。

请注意目标过程调用规范中需要双引号，因为调用规范本身是exec_shard_plsql的字符串参数。

如果目标过程在分片目录数据库上正确执行，则它将排队等待处理所有当前部署的分片。目录上的目标过程执行中的任何错误都将返回到SQL * Plus会话。可以使用与DDL相同的方式跟踪在分片上执行期间的错误。

2.7在碎片上生成唯一序列号

Oracle Sharding允许您为非主键列生成跨分片的全局唯一序列号，并由分片数据库处理。

当customer_id是分片键时，客户通常需要为非主键列生成唯一ID，例如order_id。对于这种情况，此功能允许您跨分片生成唯一的序列号，而不需要您管理应用程序中给定非主键列的全局唯一性。

新对象SHARDED SEQUENCE支持此功能。在分片目录上创建分片序列，但在每个分片上都有一个实例。每个实例生成单调递增的数字，这些数字属于与其他分片上使用的范围不重叠的范围。因此，每个生成的数字都是全球唯一的。

例如，可以使用分片序列来为按客户ID分片的表生成唯一的订单号。使用客户ID作为密钥建立与分片的连接的应用程序可以使用分片序列的本地实例来生成全局唯一的订单号。

请注意，分片序列生成的数字不能立即用作插入此分片的新行的分片键，因为键值可能属于另一个分片，插入将导致错误。要插入新行，应用程序应首先生成分片键的值，然后使用它连接到相应的分片。生成分片键新值的典型方法是在分片目录上使用常规（非分片）序列。

如果单个分片密钥生成器成为瓶颈，则可以使用分片序列来实现此目的。在这种情况下，应用程序应连接到随机分片（使用全局服务而不指定分片键），从分片序列中获取唯一键值，然后使用键值连接到相应的分片。

为了支持此功能，新的SEQUENCE对象的条款， SHARD和NOSHARD ，包括在SEQUENCE对象的DDL语法，如下面的CREATE语句的语法。 

创建| ALTER SEQUENCE [架构。 ]序列[{INCREMENT BY | START WITH}整数| {MAXVALUE整数| NOMAXVALUE} | {MINVALUE整数| NOMINVALUE} | {CYCLE | NOCYCLE} | {CACHE整数| NOCACHE} | {订购| NOORDER} | {SCALE {EXTEND | NOEXTEND} | NOSCALE} | {SHARD {EXTEND | NOEXTEND} | NOSHARD} ]

NOSHARD是序列的默认值。如果指定了SHARD子句，则此属性将在序列对象的字典表中注册，并使用DBA_SEQUENCES ， USER_SEQUENCES和ALL_SEQUENCES视图显示。

当SHARD被指定， EXTEND和NOEXTEND子句定义分片序列的行为。指定EXTEND ，生成的序列值全部为长度（x + y） ，其中x是大小为4的SHARD偏移量的长度（对应于最大碎片数量的宽度，即1000）序列值的开头， y是序列MAXVALUE / MINVALUE的最大位数。

为默认设置SHARD条文是NOEXTEND 。使用NOEXTEND设置，生成的序列值最多与MAXVALUE / MINVALUE序列中的最大位数一样宽。此设置对于与使用序列填充固定宽度列的现有应用程序集成非常有用。在指定了SHARD NOEXTEND的序列上调用NEXTVAL时，如果生成的值需要比序列的MAXVALUE / MINVALUE更多的表示位数，则抛出用户错误。

如果还使用SHARD子句指定了SCALE子句，则序列会在分片内为多个实例和会话生成可伸缩值，这些实例和会话是全局唯一的。当使用SHARD和SCALE关键字指定EXTEND ，生成的序列值都是长度（x + y + z） ，其中x是大小为4的前置SHARD偏移的长度， y是可缩放偏移的长度（默认值为 6）， z是MAXVALUE / MINVALUE序列中的最大位数。

注意：

使用SHARD子句时，请勿在序列上指定ORDER 。使用SHARD生成全局无序值。如果需要ORDER则在每个节点上本地创建序列。

SHARD关键字将与CACHE和NOCACHE操作模式结合使用。

207.4.6 PROCESSXSL函数和过程

此函数转换输入XMLDocument 。

在调用此过程之前，应对默认处理器行为进行任何更改。如果处理失败，则会引发应用程序错误。

句法

使用给定的DOMDocument和样式表转换输入XMLDocument ，并返回结果文档片段：

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，xmldoc IN DOMDOCUMENT），RETURN DOMDOCUMENTFRAGMENT;

使用给定文档将输入XMLDocument转换为URL和Stylesheet ，并返回结果文档片段：

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，url IN VARCHAR2，RETURN DOMDOCUMENTFRAGMENT;

使用给定文档将输入XMLDocument转换为CLOB和Stylesheet ，并返回结果文档片段：

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，clb IN CLOB）返回DOMDOCUMENTFRAGMENT;

使用给定的DOMDOCUMENT和样式表转换输入XMLDocument ，并将输出写入指定的文件：

DBMS_XSLPROCESSOR.DBMS_XSLPROCESSOR。（p IN处理器，ss IN样式表，xmldoc IN DOMDOCUMENT，dir IN VARCHAR2，fileName IN VARCHAR2）;

使用给定的URL和样式表转换输入XMLDocument ，并将输出写入指定目录中的指定文件：

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，url IN VARCHAR2，dir IN VARCHAR2，fileName IN VARCHAR2）;

使用给定的DOMDOCUMENT和样式表转换输入XMLDocument ，并将输出写入CLOB ：

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，xmldoc IN DOMDOCUMENT，cl IN OUT CLOB）;

使用给定的DOMDOCUMENTFRAGMENT和样式表转换输入XMLDocument ，并返回结果文档片段：

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，xmldf IN DOMDOCUMENTFRAGMENT）返回DOMDOCUMENTFRAGMENT;

使用给定的DOMDocumentFragment和样式表转换输入XMLDocumentFragment ，并将输出写入指定目录中的指定文件：

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，xmldf IN DOMDOCUMENTFRAGMENT，dir IN VARCHAR2，文件名IN VARCHAR2）;

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，xmldf IN DOMDOCUMENTFRAGMENT，buf IN OUT VARCHAR2）;

DBMS_XSLPROCESSOR.PROCESSXSL（p IN处理器，ss IN样式表，xmldf IN DOMDOCUMENTFRAGMENT，cl IN OUT CLOB）;
