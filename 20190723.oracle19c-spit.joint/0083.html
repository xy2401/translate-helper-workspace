13.28 EXECUTE IMMEDIATE语句

EXECUTE IMMEDIATE语句在单个操作中构建并运行动态SQL语句。

本机动态SQL使用EXECUTE IMMEDIATE语句来处理大多数动态SQL语句。

警告：

使用动态SQL时，要注意SQL注入，存在安全风险。有关SQL注入的详细信息，请参阅“ SQL注入 ” 。

语义

execute_immediate_statement

dynamic_sql_stmt

字符串文字，字符串变量或表示SQL语句的字符串表达式。其类型必须是CHAR ， VARCHAR2或CLOB 。

注意：

如果dynamic_sql_statement是SELECT语句，并且省略了into_clause和bulk_collect_into_clause ，则execute_immediate_statement永远不会执行。

例如，此语句从不递增序列：

into_clause

指定用于存储语句返回的列值的变量或记录。有关此子句的更多信息，请参阅“返回到子句 ” 。

对into_clause的限制

当且仅当dynamic_sql_stmt返回单行时才使用。

bulk_collect_into_clause

指定一个或多个集合，用于存储语句返回的行。有关此子句的更多信息，请参阅“返回到子句 ” 。

对bulk_collect_into_clause的限制

当且仅当dynamic_sql_stmt可以返回多行时使用。

dynamic_returning_clause

在单个变量或记录中返回受动态SQL语句影响的行的列值。有关此子句的更多信息，请参阅“返回到子句 ” 。

对dynamic_returning_clause的限制

当且仅当dynamic_sql_stmt具有RETURNING INTO子句时才使用。

using_clause

使用位置表示法指定绑定变量。

注意：

如果在dynamic_sql_statement重复占位符名称，请注意占位符与绑定变量关联的方式取决于动态SQL语句的类型。有关详细信息，请参阅“ 动态SQL语句中的重复占位符名称” 。

using_clause的限制

  • 当且仅当dynamic_sql_stmt包含绑定变量的占位符时才使用。

  • 如果dynamic_sql_stmt有一个RETURNING INTO子句（ static_returning_clause ），然后using_clause只能包含IN绑定变量。根据定义， RETURNING INTO子句中的绑定变量是OUT绑定变量。

IN，OUT，IN OUT

对于带有RETURNING子句的DML语句，可以将OUT绑定变量放在RETURNING INTO子句中，而不指定参数模式，该模式始终为OUT 。

bind_argument

一个表达式，其值在运行时替换dynamic_sql_stmt中的相应占位符。

在每一个占位符dynamic_sql_stmt必须与相关联bind_argument中USING子句或RETURNING INTO子句（或两者）或与在限定可变INTO子句。

您可以使用绑定变量的不同值重复运行dynamic_sql_stmt 。您会产生一些开销，因为EXECUTE IMMEDIATE在每次执行之前准备动态字符串。

注意：

可以按任何顺序评估绑定变量。如果程序确定评估顺序，那么在程序执行此操作时，其行为是不确定的。

  • bind_argument不能是由string索引的关联数组。

  • bind_argument不能是保留字NULL 。

目的

使用CROSSCHECK命令将备份和副本的物理实际与RMAN存储库中的逻辑记录同步。

也可以看看：

先决条件

RMAN必须连接到必须启动的目标数据库实例。

磁盘交叉检查不需要维护通道。如果您使用媒体管理器但尚未为其配置自动通道，则必须在CROSSCHECK之前使用运行ALLOCATE CHANNEL FOR MAINTENANCE 。例如，如果您在SBT通道上创建了备份，但尚未为此设备配置自动SBT通道，则必须先手动分配SBT通道，然后CROSSCHECK才能检查备份。此外，如果使用不同的介质管理器选项（池，服务器，库等）执行备份，则必须为每个组合分配维护通道。

CROSSCHECK验证所有指定的备份和副本，即使它们是在以前的数据库版本中创建的。

对于预备份备份，CDB必须以读写模式打开，并且必须以SYSDBA或SYSBACKUP权限的身份以普通用户身份进行连接。

使用说明

RMAN始终在其执行操作的每个目标数据库的控制文件中维护有关备份的元数据。如果将RMAN与恢复目录一起使用，则RMAN还会维护恢复目录中每个已注册数据库的元数据。

如果备份在磁盘上，则CROSSCHECK确定文件的标头是否有效。如果备份在磁带上，则RMAN会在RMAN存储库中查询要检查的备份片的名称和位置。RMAN将此元数据发送到目标数据库服务器，该服务器向媒体管理软件查询备份。然后，媒体管理软件检查其媒体目录，并使用备份状态向服务器报告。

跟踪路线实用程序（ 在Linux和UNIX环境中，TRCROUTE使管理员能够发现从客户端到服务器的连接路径或路由。如果TRCROUTE遇到问题，则它会向客户端返回错误堆栈而不是单个错误。这些附加错误消息使故障排除更容易

比较条件将一种表达与另一种表达进这种比较的结果可以是TRUE ， FALSE或UNKNOWN 。

比较条件不支持大对象（LOB）。但是，您可以使用PL / SQL程序来比较CLOB数据。

比较数值表达式时，Oracle使用数字优先级来确定条件是否比较NUMBER ， BINARY_FLOAT或BINARY_DOUBLE值。有关数字优先级的信息，请参阅数字优先级。

在比较字符表达式时，Oracle使用数据类型比较规则中描述的规则 。这些规则定义了在比较之前如何对齐表达式的字符集，使用二进制或语言比较（整理），空白填充比较语义的使用，以及对归类键施加的限制所产生的限制，包括报告错误ORA-12742: unable to create the collation key 。

11使用JSON Merge修补程序更新JSON文档

您可以使用Oracle SQL函数json_mergepatch更新JSON文档的特定部分。您传递一个JSON Merge Patch文档，该文档指定对指定的JSON文档所做的更改。JSON Merge Patch是IETF标准。

您可以在UPDATE语句中使用json_mergepatch来更新JSON列中的文档。例11-3说明了这一点。

您可以在SELECT列表中使用json_mergepatch来修改所选文档。修改后的文档可以进一步返回或处理。例11-4说明了这一点。

JSON Merge Patch适用于更新主要使用对象作为结构的JSON文档，而不使用显式null值。您不能使用它来添加，删除或更改数组元素（除非显式替换整个数组）。并且您不能使用它将字段的值设置为null 。

JSON Merge Patch有点像UNIX patch实用程序：你给它（1）补丁的源文档和（2）指定要进行的更改的补丁文档，它返回更新的源文档的副本（修补） 。补丁文档指定源文档和结果文档之间的差异。对于UNIX patch ，差异是UNIX diff实用程序输出的形式。对于JSON Merge Patch，源和补丁都是JSON文档。

您可以将JSON Merge Patch视为合并源和补丁的内容。合并两个对象时，一个来自源，一个来自补丁，一个成员在一个对象中而不在另一个对象中的成员保留在结果中。一个例外是当源对象没有这样的字段时，将忽略字段值为null的补丁成员。

合并具有相同字段的对象成员时：

  • 如果补丁字段值为null ，则从源中删除该字段 - 它不包含在结果中。

  • 否则，该字段保留在结果中，但其值是将源字段值与补丁字段值合并的结果 。也就是说，在这种情况下，合并操作是递归的 - 它向下潜入其值本身就是对象的字段。

更确切地说，JSON Merge Patch的行为如下：

  • 如果补丁 不是 JSON对象，则通过补丁替换源。

  • 否则（补丁是对象），请执行以下操作：

    1. 如果源 不是对象，则表现为它是空对象（ {} ）。

    2. 迭代补丁对象的（ p-field : p–value ）成员。

      • 如果修补程序成员的p-value null则从源中删除相应的成员。

      • 否则， 递归 ： 将相应源字段的值替换为使用p-value （作为下一个补丁） 合并修补该值（作为下一个源）的结果。

如果补丁字段值null没有特殊含义（使用该字段删除相应的源成员），则可以将其用作字段值，以将相应的源字段值设置为null 。特殊删除行为意味着您无法将源字段值设置为null 。

例子：

  • 修补程序成员"PONumber":99999使用字段PONumber覆盖源成员， 将其值替换为修补程序指定的值99999 。

    json_mergepatch('{"User":"ABULL", "PONumber":1600}', '{"PONumber":99999}')导致{"User":"ABULL", "PONumber":99999} 。

  • 修补程序成员"tracking":123456使用字段tracking覆盖缺少的源成员，将该修补程序成员添加到结果中。并且源成员"PONumber":1600使用字段PONumber覆盖缺少的补丁成员 - 它保留在结果中。

    json_mergepatch('{"PONumber":1600}', '{"tracking":123456}')导致{"PONumber":1600, "tracking":123456}" 。

  • 补片部件"Reference":null覆盖与场源构件Reference ，从结果中移除 。

    json_mergepatch('{"PONumber":1600, "Reference":"ABULL-20140421"}', '{"Reference":null}')导致{"PONumber":1600} 。

  • 补丁值[1,2,3]会覆盖相应的源值[4,5,6] ， 替换它。

    json_mergepatch('{"PONumber":1600, "LineItems":[1, 2, 3]}', '{"LineItems":[4,5,6]}')导致{"PONumber":1600, "LineItems":[4, 5, 6]} 。

注意：

合并补丁程序 - 特别是对于非对象补丁没有递归行为这一事实 - 意味着您无法单独添加，删除或替换数组的值。要进行此类更改，您必须替换整个数组 。例如，如果源文档具有成员Phone:["999-555-1212", "415-555-1234"]然后要删除第二个电话号码，您可以使用其内容具有"Phone":["999-555-1212"]成员的补丁"Phone":["999-555-1212"] 。

确定Oracle自动存储管理的存储要求

要确定使用Oracle ASM的存储要求，必须确定所需的设备数量和可用磁盘空间量。

若要完成此任务，请按照下列步骤操作：
  1. 根据要部署的群集配置，规划Oracle ASM磁盘组要求。Oracle Domain Services Clusters将Oracle Clusterware文件和Grid Infrastructure Management Repository（GIMR）存储在单独的Oracle ASM磁盘组上，因此需要配置两个单独的Oracle ASM磁盘组，一个用于OCR和投票文件，另一个用于GIMR。
  2. 确定是否要将Oracle ASM用于Oracle数据库文件，恢复文件和Oracle数据库二进制文件。Oracle数据库文件包括数据文件，控制文件，重做日志文件，服务器参数文件和密码文件。

    注意：

    • 您不必对Oracle数据库文件和恢复文件使用相同的存储机制。您可以将共享文件系统用于一种文件类型，将Oracle ASM用于另一种文件类型。

    • Oracle Clusterware文件有两种类型：OCR文件和投票文件。您可以使用Oracle ASM或共享文件系统在Oracle Standalone Cluster部署中存储OCR和投票文件，但必须使用Oracle ASM在Oracle Domain Services集群部署上存储OCR和投票文件。

    • 如果数据库文件存储在共享文件系统上，则可以继续将其用于数据库文件，而不是将其移动到Oracle ASM存储。

  3. 选择要用于Oracle ASM磁盘组的Oracle ASM冗余级别。

    除使用外部冗余外，Oracle ASM将所有Oracle Clusterware文件镜像到磁盘组中的不同故障组中。仲裁失败组是一种特殊类型的失败组，当投票文件存储在普通或高冗余磁盘组中时，它包含投票文件的镜像副本。包含Oracle Clusterware文件（OCR和投票文件）的磁盘组比其他磁盘组具有更高的最小故障组数，因为投票文件存储在Oracle ASM磁盘组的仲裁故障组中。

    仲裁失败组是一种特殊类型的故障组，用于存储Oracle Clusterware投票文件。仲裁失败组用于确保指定故障组的法定数量可用。当Oracle ASM装入包含Oracle Clusterware文件的磁盘组时，仲裁失败组用于确定是否可以在丢失一个或多个故障组时装入磁盘组。仲裁失败组中的磁盘不包含用户数据，因此在确定存储用户数据的冗余要求时不考虑仲裁失败组。

    冗余级别如下：

    • 高冗余

      在高冗余磁盘组中，Oracle ASM使用三向镜像来提高性能并提供最高级别的可靠性。高冗余磁盘组至少需要三个磁盘设备（或三个故障组）。高冗余磁盘组中的有效磁盘空间是其所有设备中磁盘空间总和的三分之一。

      对于Oracle Clusterware文件，高冗余磁盘组至少需要五个磁盘设备，并提供五个投票文件和一个OCR（一个主副本和两个副本）。例如，您的部署可能包含三个常规故障组和两个仲裁故障组。请注意，并非所有故障组都可以是仲裁故障组，即使投票文件需要所有五个磁盘。通过高冗余，群集可以在丢失两个故障组后继续存在。

      虽然高冗余磁盘组确实提供了高级别的数据保护，但在决定选择高冗余磁盘组之前，应考虑额外存储设备的更高成本。

    • 正常冗余

      在常规冗余磁盘组中，为了提高性能和可靠性，Oracle ASM默认使用双向镜像。普通冗余磁盘组至少需要两个磁盘设备（或两个故障组）。普通冗余磁盘组中的有效磁盘空间是其所有设备中磁盘空间总和的一半。

      对于Oracle Clusterware文件，普通冗余磁盘组至少需要三个磁盘设备，并提供三个投票文件和一个OCR（一个主副本和一个副本）。例如，您的部署可能包含两个常规故障组和一个仲裁故障组。通过正常冗余，群集可以在丢失一个故障组后继续存在。

      如果您没有使用存储阵列来提供针对存储的数据丢失的独立保护，那么Oracle建议您选择正常冗余。

    • 外部冗余

      外部冗余磁盘组至少需要一个磁盘设备。外部冗余磁盘组中的有效磁盘空间是其所有设备中磁盘空间的总和。

      由于Oracle ASM不镜像外部冗余磁盘组中的数据，因此Oracle建议您对存储设备（如RAID）或其他提供自己的数据保护机制的类似设备使用外部冗余。

    • Flex冗余

      Flex冗余磁盘组是一种冗余磁盘组，具有灵活的文件冗余，镜像拆分和冗余更改等功能。Flex磁盘组可以将具有不同冗余要求的文件合并到一个磁盘组中。它还为数据库提供了更改其文件冗余的功能。磁盘组是文件组的集合，每个文件组与一个数据库相关联。配额组定义磁盘组中一组数据库的最大存储空间或配额限制。

      在Flex冗余磁盘组中，Oracle ASM使用Oracle ASM元数据的三向镜像来提高性能并提供可靠性。对于数据库数据，您可以选择不镜像（不受保护），双向镜像（镜像）或三向镜像（高）。Flex冗余磁盘组至少需要三个磁盘设备（或三个故障组）。

    • 扩展冗余

      扩展冗余磁盘组具有与Flex冗余磁盘组类似的功能。配置Oracle扩展群集时，可以使用扩展冗余。扩展冗余扩展了Oracle ASM数据保护，通过在每个站点的不同故障组中放置足够的数据副本来覆盖站点故障。站点是故障组的集合。对于具有三个站点（例如，两个数据站点和一个仲裁故障组）的扩展冗余，最小磁盘数为七（两个数据站点各三个磁盘，两个数据站点之外一个仲裁故障组）。扩展冗余支持的最大站点数为3。在扩展冗余磁盘组中，每个站点都维护文件组属性指定的用户数据冗余。每个站点都可以托管给定磁盘组的数据故障组和仲裁故障组。例如，如果文件组冗余指定为2或3，则每个站点分别有2个或3个镜像，前提是有足够的故障组来容纳镜像。有关为扩展群集选择冗余级别的详细信息，请参阅关于Oracle Extended Clusters 。

    也可以看看：

    有关Flex磁盘组的文件组和配额组的详细信息，请参见“Oracle自动存储管理管理员指南”

    注意：

    创建磁盘组后，可以更改磁盘组的冗余级别。例如，您可以将普通或高冗余磁盘组转换为Flex冗余磁盘组。在Flex冗余磁盘组中，文件冗余可以在三个可能的值之间更改：不受保护，镜像或高。

  4. 确定Oracle Clusterware文件以及数据库文件和恢复文件所需的磁盘空间总量。

    如果系统上正在运行Oracle ASM实例，则可以使用现有磁盘组来满足这些存储要求。如有必要，可以在数据库安装期间将磁盘添加到现有磁盘组。

    请参阅Oracle Clusterware存储空间要求以确定安装Oracle Clusterware文件的最小磁盘数和最小磁盘空间要求，以及安装启动器数据库，其中您将投票文件放在单独的磁盘组中。

  5. 确定分配单元大小。

    每个Oracle ASM磁盘都分为分配单元（AU）。分配单元是磁盘组内的基本分配单位。您可以从1,2,4,8,16,32或64 MB中选择AU Size值，具体取决于特定的磁盘组兼容级别。对于Flex磁盘组，AU大小的默认值设置为4 MB。对于外部，正常和高冗余，默认AU大小为1 MB。

  6. 对于Oracle Clusterware安装，还必须为Oracle ASM元数据添加额外的磁盘空间。您可以使用以下公式计算OCR和投票文件以及Oracle ASM元数据的磁盘空间要求（以MB为单位）：
    total = [2 * ausize * disks] + [redundancy *（ausize *（all_client_instances + nodes + disks + 32）+（64 * nodes）+ clients + 543）]

    redundancy =镜像数：external = 1，normal = 2，high = 3，flex = 3。

    ausize =元数据AU大小，以兆字节为单位

    all_client_instance =所有数据库客户端和ACFS代理实例的总和

    nodes =集群中的节点数。

    clients - 每个节点的数据库实例数。

    disks - 磁盘组中的磁盘数。

    例如，对于四节点Oracle RAC安装，在正常冗余磁盘组中使用三个磁盘，则需要额外的5293 MB空间： [2 * 4 * 3] + [2 * (4 * (4 * (4 + 1)+ 30)+ (64 * 4)+ 533)] = 5293 MB

  7. （可选）识别Oracle ASM磁盘组设备的故障组。

    如果您打算使用普通或高冗余磁盘组，则可以通过关联自定义故障组中的一组磁盘设备来进一步保护数据库免受硬件故障的影响。默认情况下，每个设备都包含在其故障组中。但是，如果正常冗余磁盘组中的两个磁盘设备连接到同一主机总线适配器（HBA），则如果适配器发生故障，则磁盘组将变为不可用。此示例中的HBA是单点故障。

    例如，为了避免此类故障，您可以使用两个HBA结构路径，每个路径有两个磁盘，并为连接到每个适配器的磁盘定义故障组。此配置将使磁盘组能够容忍一个HBA结构路径的故障。

    注意：

    您可以在安装Oracle Grid Infrastructure期间定义自定义故障组。您还可以使用GUI工具ASMCA，命令行工具asmcmd或SQL命令在安装后定义故障组。如果定义自定义故障组，则必须为常规冗余磁盘组指定至少两个故障组，为高冗余磁盘组指定三个故障组。

  8. 如果您确定系统上不存在合适的磁盘组，请安装或识别要添加到新磁盘组的相应磁盘设备。在识别适当的磁盘设备时，请遵循以下准则：
    • 磁盘设备必须由执行Oracle Grid Infrastructure安装的用户拥有。
    • Oracle ASM磁盘组中的所有设备必须具有相同的大小并具有相同的性能特征。
    • 不要将单个物理磁盘上的多个分区指定为磁盘组设备。Oracle ASM希望每个磁盘组设备都位于单独的物理磁盘上。
    • 尽管您可以将逻辑卷指定为Oracle ASM磁盘组中的设备，但Oracle不建议使用它们，因为它会增加Oracle ASM不必要的复杂性。 Oracle建议，如果选择使用逻辑卷管理器，则使用逻辑卷管理器表示单个逻辑单元号（LUN），而不进行条带化或镜像，以便最大限度地减少对附加存储层的存储性能的影响。
  9. 如果使用在网络文件系统（NFS）上创建的Oracle ASM磁盘组进行存储，请确保遵循在NFS上配置Oracle ASM磁盘组的准则中所述的建议。

实现Java的安全性

实现Java程序安全性的过程。

使用Oracle XML Developer's Kit保护XSLT处理

在使用Oracle XML Developer's Kit进行XSLT处理之前，您需要通过设置一些配置选项来保护它。

您必须为XSLProcessor API和JAXP转换器API启用安全处理，如下所示。

  • 使用此Java代码为XSLProcessor启用安全处理模式：

    processor.setAttribute（XMLConstants。FEATURE_SECURE_PROCESSING，布尔值。真正）;

  • 使用此Java代码为JAXP转换器API的SAXTransformerFactory启用安全处理模式：

    factory.setFeature（XMLConstants。FEATURE_SECURE_PROCESSING，true）;

这些设置一起通过以下方式提高了XML解析的安全性：

  • 默认情况下，设置会阻止对外部资源的访问。

    然后开发人员可以使用org.xml.sax.EntityResolver或javax.xml.transform.URIResolver解决XSL样式表中的include和import元素。他们可以使用Java接口URIResolver提供对受信任外部资源的XPath或XSLT功能访问。

  • 这些设置会阻止使用Java反射来执行Java代码。

    开发人员可以将受信任的Java类转换为接口javax.xml.xpath.XPathFunction并使用接口javax.xml.xpath.XPathFunctionResolver来解决它们。

  • 设置限制递归实体扩展。

    这有助于避免因过多的资源消耗而导致的常见拒绝服务攻击。

    注意：

    虽然这可以限制递归实体扩展 ，但XDK不能限制XML转换期间的资源消耗。XSL样式表可以包含无限循环或以其他方式消耗资源。因此，不得从不受信任的来源或外部实体接受XSL样式表。测试所有使用的样式表 ，以确保它们不会消耗过多的资源。

此外，Oracle建议对从用户或外部不可信实体获取的数据或文件执行XSLT处理的所有代码检查是否为XSLProcessor API和JAXP转换器API启用了安全处理。不这样做会打开一个安全漏洞。在XSL处理器受到保护之后，不允许访问外部资源和运行Java扩展功能。

任意安全豁免

如果要任意覆盖安全限制，可以使用以下选项：

  • 执行以下任何修改以允许访问某些外部源：

    • 使用setURIResolver()方法将URIResolver设置为SAXTransformerFactory或XSLProcessor 。

    • 使用setEntityResolver()方法将EntityResolver设置为XSLProcessor 。

  • 使用Java接口oracle.xml.xslt.XSLSecurityManager用于注册Java类和方法的白名单，您可以使用它们使用Java反射扩展函数。

  • 使用XSLProcessor方法setAttribute()注册XSLSecurityManager以运行您知道安全的扩展函数：

    processor.setAttribute（XSLProcessor。SECURITY_MANAGER，securityManager）;

  • 在XSLSecurityManager上使用简单的API来实现Java类和方法的白名单：

    boolean checkExtensionFunction（String className，String fnName）;

安全地使用Oracle XML Parser

与许多XML解析器一样，默认情况下，XDK解析器会尝试解析外部引用。攻击者可以利用此行为来执行XML外部实体（XXE）和XML实体扩展（XEE）攻击。要避免此漏洞，请禁用对外部资源的任意引用，并禁用不受约束的实体扩展。

要实现全局安全设置（阻止在系统中创建的所有解析器使用外部实体），请将Java系统属性oracle.xdkjava.security.resolveEntityDefault为false 。

要阻止特定解析器的实体扩展，或限制实体扩展的级别数，请设置属性XMLParser.RESOLVE_ENTITY_DEFAULT或XMLParser.ENTITY_EXPANSION_DEPTH到解析器。

  1. 在前面示例中显示的位置将no_ri_open设置为FALSE 。

  2. 开发定制的打开，读取和关闭回调函数，并使用这些函数初始化OraSteam 。

  3. 覆盖访问xctx与OraSteam

xmlctx.hpp头文件使用CXmlCtx(bool no_ri_open) API初始化xctx上下文并设置参数no_ri_open 。当no_ri_open参数设置为true ，使用xctx解析器API将无法访问外部资源。当no_ri_open参数设置为false ，使用xctx解析器API将能够访问外部资源。

为何选择数据库文件系统？

从概念上讲，数据库文件系统是放置在存储在数据库表中的文件和目录之上的文件系统接口。

应用程序通常使用标准SQL数据类型BLOB和CLOB来存储和检索Oracle数据库中的文件，医疗图像，发票图像，文档，视频和其他文件等文件。与传统文件系统相比，Oracle数据库提供了更好的安全性，可用性，健壮性，事务性能和可伸缩性。存储在数据库中的文件以及关系数据将自动备份，使用Data Guard同步到灾难恢复站点，并一起恢复。

数据库文件系统（DBFS）是Oracle数据库的一项功能，使用户可以更轻松地访问和管理存储在数据库中的文件。使用此接口，对数据库中文件的访问不再局限于专门编写为使用BLOB和CLOB编程接口的程序。现在可以使用任何作用于文件的操作系统（OS）程序透明地访问数据库中的文件。例如，ETL（提取，转换和加载）工具可以透明地将暂存文件存储在数据库中，而基于文件的应用程序可以从数据库功能（如最大可用性体系结构（MAA））中受益，而无需对应用程序进行任何更改。

关于DBFS

DBFS与NFS类似，它提供的共享网络文件系统看起来像本地文件系统，同时具有服务器组件和客户端组件。

DBFS的核心是DBFS Content API，它是Oracle数据库中的PL / SQL接口。它连接到DBFS Content SPI，这是一个程序化接口，允许支持不同类型的存储。

在编程级别，客户端调用DBFS Content API来执行特定功能，例如删除文件。然后，DBFS Content API 删除文件函数调用DBFS内容SPI来执行该功能。

在用户定义的DBFS中，用户必须根据DBFS内容SPI中的规范以及规范中的其他功能实现删除功能。

DBFS服务器

在DBFS中，文件服务器是Oracle数据库。

文件作为SecureFiles LOB存储在数据库表中。数据库中文件系统的实现称为DBFS内容存储库，例如DBFS SecureFiles存储库。DBFS内容存储库允许每个数据库用户创建一个或多个可由客户端装入的文件系统。每个文件系统都有自己的专用表来保存文件系统内容。

DBFS内容SPI支持不同类型的存储，如下所示：

  • DBFS SecureFiles Store：一个DBFS内容存储库，它使用带有SecureFiles LOB列的表来存储文件系统数据。它实现了类似POSIX的文件系统功能。

  • DBFS分层存储：DBFS内容存储，允许将文件写入Oracle Recovery Manager（RMAN）支持的任何磁带存储单元或云存储系统。

  • 用户定义的存储：用户定义的内容存储。这允许用户在Oracle数据库内编写自己的文件系统，而无需编写任何操作系统代码。

C PL / SQL程序限制

本附录描述了PL / SQL语言强加的程序限制。PL / SQL基于编程语言Ada。因此，PL / SQL使用描述性中间属性表达式Ada（DIANA）的变体，这是一种树形结构的中间语言。它使用称为接口定义语言（IDL）的元交换来定义。DIANA由编译器和其他工具在内部使用。

在编译时，PL / SQL源文本被翻译成系统代码。DIANA和子程序或包的系统代码都存储在数据库中。在运行时，它们将加载到共享内存池中。DIANA用于编译依赖的子程序;系统代码只是运行。

在共享内存池中，包规范，ADT规范，独立子程序或匿名块仅限于67108864（2 ** 26）DIANA节点，这些节点对应于标识符，关键字，运算符等标记。这允许大约6,000,000行代码，除非您超出PL / SQL编译器施加的限制，其中一些在表C-1中给出。

操作：请参阅文档以获取有效模式集，然后重试。

UDI-00012：表模式导入仅允许来自一个模式的对象

原因：选择了多个模式进行导入。

操作：重试导入仅指定一个模式。

UDI-00013：内部impdp错误。

原因：致命的内部数据泵客户端错误。

操作：联系Oracle客户支持。

UDI-00014：参数值无效，' string '

原因：无法解析指定的参数或参数值。

操作：请参阅文档以获取有效的参数值。

UDI-00015：参数' string '的上下文或作业状态无效。

原因：指定参数的作业状态或上下文无效。

操作：请参阅文档以了解正确的参数用法。

UDI-00016：' string '参数需要一个值。

原因：指定的参数没有值。

操作：请参阅文档以了解正确的参数用法。

UDI-00017：数据泵作业已在执行。

原因：请求的操作的数据泵作业状态不正确。

操作：请参阅文档以了解正确用法。

UDI-00018：数据泵客户端与数据库版本string不兼容

原因：导入实用程序的版本高于数据库版本，因此不兼容。

操作：使用与数据库相同版本的Import实用程序。

UDI-00019：' string '参数列表太长。

原因：请求的参数列表最多超过4000个字符。

操作：减少指定数量的参数值并重新运行。

UDI-00020：Data Pump客户端与Data Pump API不兼容。

原因：导出实用程序的版本高于数据泵服务器，因此不兼容。

操作：使用与服务器相同版本的Export实用程序。

UDI-00022：连接字符串\ n中指定的不受支持的管理权限

原因：连接字符串指定了数据泵不支持的管理权限。

操作：指定数据泵支持的管理权限。这些是SYSDBA和SYSBACKUP。

UDI-00050：无法设置参数的处理。

原因：参数管理器初始化期间发生意外错误。

操作：记录后面的消息并将其作为导入内部错误报告给Oracle客户支持。

UDI-00051：无法NLS_NUMERIC_CHARACTERS设置所需的默认。

原因：导入无法将NLS_NUMERIC_CHARACTERS设置为“。”，

操作：记录随附的消息并将其作为导入内部错误报告给Oracle客户支持。

UDI-00052：操作错误的数据泵客户端： string

原因：导入客户端尝试附加到导出作业。

操作：附加到现有作业时使用适当的客户端

UDI-00053：无法识别的客户端命令' string '

原因：无效的交互式客户端命令。

操作：请参阅文档以获取有效的客户端命令。

UDI-00054：已弃用的旧版模式参数' string '在或上指定： string

原因：指定的传统模式参数没有相应的数据泵映射。

操作：查看错误消息中引用的位置（（“命令行”，“参数文件”或“环境变量”）并删除参数。
