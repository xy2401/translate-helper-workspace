6比较SQL试验

在构建更改后的SQL性能数据之后，您可以通过使用SQL Performance Analyzer运行比较分析，将在更改前的SQL试验中收集的性能数据与更改后的SQL试验进行比较。 比较分析完成后，您可以生成一个报告，以识别由于系统更改而改进，保持不变或退化的SQL语句。SQL Performance Analyzer报告计算了每个SQL语句性能变化的两个主要影响度量：

  • 对工作量的影响

    这表示在考虑执行频率之后，此更改对SQL语句的影响百分比对工作负载的累积执行时间的影响。例如，导致SQL语句的累积执行时间从101秒提高到1秒的更改（其中其余工作负载在更改之前的总执行时间为99秒）将具有50％（2x）的值这个测量。

  • 对SQL的影响

    这表示此更改对SQL语句的响应对SQL语句的响应时间的影响百分比。例如，导致SQL语句的响应时间从10秒提高到1秒的更改将具有此测量的90％（10x）值。

本章介绍如何比较和分析来自更改前和更改后SQL测试的性能数据，并包含以下主题：

  • 使用Oracle Enterprise Manager比较SQL试验

  • 使用API比较SQL试验

用于比较SQL试验的主要接口是Oracle Enterprise Manager。如果由于某种原因Oracle Enterprise Manager不可用，您可以使用DBMS_SQLPA PL / SQL包比较SQL试验。

6.1.2.4关于SQL Performance Analyzer活动报告
SQL Performance Analyzer活动报告是HTML文件，使用Web托管的交互式用户界面显示所有报告数据。 与Oracle Enterprise Manager中提供的SQL Performance Analyzer报告类似，活动报告包含有关每个类别的顶级SQL语句的信息（例如改进，回归和更改的计划），其中包含更改前和更改后的统计信息，解释计划和任务摘要。

SQL Performance Analyzer活动报告比传统的HTML或文本报告更有用，因为它们提供与Oracle Enterprise Manager类似的用户界面，但即使在数据库不可用时，甚至在删除数据库之后，也可以查看它们。因此，活动报告提供了传统报告和动态Oracle Enterprise Manager分析的优势，但消除了两者的缺点。此外，活动报告包含有关比较分析的更多信息，并提供更多用户交互选项。强烈建议您使用活动报告而不是HTML或文本报告。

活动报告用户界面组件与Oracle Enterprise Manager中显示的组件非常相似。有关用户界面组件的说明，请参阅“使用Oracle Enterprise Manager查看SQL Performance Analyzer报告 ”中描述的相关章节。

6.1.3使用Oracle Enterprise Manager调整回归的SQL语句

在查看SQL Performance Analyzer报告之后，您应该在比较SQL性能之后调整所识别的任何回归SQL语句。 如果有大量SQL语句似乎已退化，则应尝试确定根本原因并进行系统级更改以纠正问题。如果只有少数SQL语句退化，请考虑使用以下调整方法之一为它们实现点解决方案：

  • 创建SQL计划基准

  • 运行SQL Tuning Advisor

在调优回归的SQL语句之后，您应该使用SQL Performance Analyzer测试这些更改。在测试系统上运行新的SQL试用版，然后进行第二次比较（在此新SQL试用版和第一个SQL试用版之间）以验证结果。一旦SQL Performance Analyzer显示性能稳定，测试就完成了。从此步骤实施修复程序到生产系统。

从Oracle Database 11g第1版开始，SQL Tuning Advisor在调优SQL语句时执行备用计划分析。SQL Tuning Advisor在当前系统中搜索以前的执行计划，包括第一次SQL试验的计划。如果第一个SQL试用版的执行计划与第二个SQL试用版的执行计划不同，则SQL Tuning Advisor将推荐第一个SQL试用版的计划。如果这些执行计划产生更好的性能，您可以使用第一个SQL试验中的计划创建计划基准。

注意：

SQL Performance Analyzer在完成远程SQL试用后，不提供创建SQL计划基准或直接运行SQL Tuning Advisor的选项。在这种情况下，您需要使用API手动传输SQL调优集并在远程数据库上完成相应的过程。

6.2.1使用API分析SQL性能

在构建更改后的SQL性能数据之后，您可以将性能数据的更改前版本与更改后的版本进行比较。 使用DBMS_SQLPA.EXECUTE_ANALYSIS_TASK过程或函数运行比较分析。

要比较更改前和更改后的SQL性能数据：

  1. 打电话给 EXECUTE_ANALYSIS_TASK过程或函数使用以下参数：

    • 将task_name参数设置为SQL Performance Analyzer任务的名称。

    • 将execution_type参数设置为COMPARE PERFORMANCE 。此设置将分析和比较两个版本的SQL性能数据。

    • 使用execution_name参数指定名称以标识execution_name 。如果未指定，它将由SQL Performance Analyzer生成并由函数返回。

    • 使用execution_params参数指定两个版本的SQL性能数据。execution_params参数被指定为指定执行的（ 名称 ， 值 ）对。设置与比较和分析SQL性能数据相关的执行参数，如下所示：

      • 将execution_name1参数设置为第一次执行的名称（在进行系统更改之前）。此值应对应于“ 使用API创建预更改SQL试验 ”中指定的execution_name参数的值。

      • 将execution_name2参数设置为第二次执行的名称（在进行系统更改后）。此值应对应于在系统更改后执行SQL工作负载时“ 使用API创建更改后SQL试验 ”中指定的execution_name参数的值。如果调用者未指定执行，则默认情况下SQL Performance Analyzer将始终比较最后两个任务执行。

      • 设置comparison_metric参数以指定要在性能影响分析中使用的执行统计信息的表达式。可能的值包括以下度量标准或它们的任意组合： elapsed_time （默认值）， cpu_time ， buffer_gets ， disk_reads ， direct_writes ， optimizer_cost和io_interconnect_bytes 。

      有关可以设置用于比较的其他可能参数，请参阅Oracle Database PL / SQL包和类型参考中的DBMS_SQLPA包的说明。

    以下示例说明了函数调用：

    EXEC DBMS_SQLPA.EXECUTE_ANALYSIS_TASK（task_name =>'my_spa_task'， -  execution_type =>'COMPARE PERFORMANCE'， -  execution_name =>'my_exec_compare'， -  execution_params => dbms_advisor.arglist（ - 'comparison_metric'，'buffer_gets'））;

  2. 打电话给 REPORT_ANALYSIS_TASK函数使用以下参数：

    • 将task_name参数设置为SQL Performance Analyzer任务的名称。

    • 将execution_name参数设置为要使用的执行名称。此值应与要为其生成报告的执行的execution_name参数匹配。

6.2.3使用API比较SQL调优集

你可以比较两个 SQL调优使用DBMS_SQLPA包进行设置。例如，在使用数据库重放时，您可能在工作负载捕获期间捕获了生产系统上的SQL调优集，并在工作负载重放期间在测试系统上捕获了另一个SQL调整集。然后，您可以使用SQL Performance Analyzer来比较这些SQL调优集，而无需重新执行SQL语句。在您进行系统更改之前和之后已经有另一个实用程序来运行工作负载（例如自定义脚本）的情况下，这非常有用。

比较SQL调优集时，SQL Performance Analyzer使用SQL调优集中捕获的运行时统计信息来执行其比较分析，并报告在一个SQL调优集中找到但在另一个SQL调优集中找不到的任何新的或缺少的SQL语句。还会报告两个SQL调优集之间的执行计划的任何更改。对于两个SQL调优集中的每个SQL语句，报告每个SQL语句的改进和回归结果 - 基于每次执行的平均统计值计算 - 以及基于累积统计值计算的整个工作负载。

要使用API比较SQL调优集：

  1. 创建SQL Performance Analyzer任务：

    VAR aname varchar2（30）; EXEC：aname：='compare_s2s'; EXEC：aname：= DBMS_SQLPA.CREATE_ANALYSIS_TASK（task_name =>：aname）;

    在创建期间不必将SQL调整集关联到任务。

  2. 创建第一个SQL试用版并转换第一个SQL调优集：

    EXEC DBMS_SQLPA.EXECUTE_ANALYSIS_TASK（task_name =>：aname， -  execution_type =>'convert sqlset'， -  execution_name =>'first trial'， -  execution_params => DBMS_ADVISOR.ARGLIST（'sqlset_name'，'my_first_sts'， - 'sqlset_owner'， '应用'））;

    使用SQLSET_NAME和SQLSET_OWNER任务参数指定SQL调优集的名称和所有者。SQL性能分析器任务不会复制SQL调优集的内容。相反，对SQL调优集的引用与新SQL试验相关联，在此示例中为“第一次试用”。

  3. 创建第二个SQL试用版并将其与要比较的第二个SQL调优关联：

    EXEC DBMS_SQLPA.EXECUTE_ANALYSIS_TASK（task_name =>：aname， -  execution_type =>'convert sqlset'， -  execution_name =>'second trial'， -  execution_params => DBMS_ADVISOR.ARGLIST（'sqlset_name'，'my_second_sts'， - 'sqlset_owner'， '应用'））;

  4. 通过运行比较分析来比较两个SQL试验（或SQL调优集）的性能数据：

    EXEC DBMS_SQLPA.EXECUTE_ANALYSIS_TASK（task_name =>：aname， -  execution_type =>'compare'， -  execution_name =>'comparison'， -  execution_params => DBMS_ADVISOR.ARGLIST（'workload_impact_threshold'，0， - 'sql_impact_threshold'，0））;

    在此示例中，工作负载和每SQL影响阈值设置为0％以进行比较（默认值为1％）。

  5. 比较分析完成后，使用DBMS_SQLPA.REPORT_ANALYSIS_TASK函数生成SQL Performance Analyzer报告。

    有关使用API生成SQL Performance Analyzer报告的信息，请参阅“ 使用API分析SQL性能 ” 。

6.2.4使用API调整回归的SQL语句

在查看SQL Performance Analyzer报告之后，您应该调整任何一个在比较SQL性能后识别的SQL语句。如果有大量SQL语句似乎已退化，则应尝试确定根本原因并进行系统级更改以纠正问题。如果只有少数SQL语句退化，请考虑使用SQL Tuning Advisor为它们实现点解决方案，或创建SQL计划基准以指示优化程序在将来选择原始执行计划。

使用API调整回归的SQL语句：

  • 使用以下命令为SQL Performance Analyzer执行创建SQL调优任务 DBMS_SQLTUNE包中的CREATE_TUNING_TASK函数：

    BEGIN DBMS_SQLTUNE.CREATE_TUNING_TASK（spa_task_name =>'my_spa_task'，spa_task_owner =>'immchan'，spa_compare_exec =>'my_exec_compare'）; DBMS_SQLTUNE.EXECUTE_TUNING_TASK（spa_task_name =>'my_spa_task'）;结束; /

    此示例创建并执行SQL调优任务，以调整在名为my_exec_compare的SQL Performance Analyzer任务名为my_spa_task的比较性能执行中退化的SQL语句。在这种情况下，使用此版本的CREATE_TUNING_TASK函数调用很重要。否则，可以从捕获它们的生产系统在环境中调整SQL语句，这不会反映系统更改。

    注意：

    如果选择在单独的数据库上远程执行SQL工作负载，则不应使用此版本的CREATE_TUNING_TASK函数调用来调优回归的SQL语句。相反，您应该调整远程数据库上SQL试验所标识的任何回归，因为应用程序模式不在运行SQL Performance Analyzer的数据库上。因此，您需要在架构所在的数据库和进行更改的位置运行SQL Tuning Advisor。

6.2.5使用API调整远程SQL试验中的回归SQL语句

如果选择在单独的数据库上远程执行SQL工作负载，则应调整远程数据库上SQL试验标识的任何回归，而不是SQL性能分析器任务所在的系统。

使用API调整远程SQL试用版中的回归SQL语句：

  1. 在运行SQL Performance Analyzer的系统上，将已回归的SQL语句的子集创建为SQL调优集：

    DECLARE sqlset_cur DBMS_SQLTUNE.SQLSET_CURSOR; BEGIN DBMS_SQLTUNE.CREATE_SQLSET（'SUB_STS1'，'测试目的'）; OPEN sqlset_cur FOR SELECT value（p）FROM table（DBMS_SQLTUNE.SELECT_SQLPA_TASK（task_name =>'SPA_TASK1'，execution_name =>'COMP'，level_filter =>'REGRESSED'））p; DBMS_SQLTUNE.LOAD_SQLSET（'SUB_STS1'，sqlset_cur）;关闭sqlset_cur;结束; /

    除了'REGRESSED' ，您可以使用其他过滤器为SQL调整集选择SQL语句，例如'CHANGED' ， 'ERRORS'或'CHANGED_PLANS' 。有关更多信息，请参阅Oracle Database PL / SQL包和类型参考 。

  2. 创建一个临时表到将导出SQL调优集的位置：

    BEGIN DBMS_SQLTUNE.CREATE_STGTAB_SQLSET（table_name =>'STG_TAB1'，schema_name =>'JOHNDOE'，tablespace_name =>'TBS_1'，db_version => DBMS_SQLTUNE.STS_STGTAB_11_1_VERSION）;结束; /

    使用db_version参数指定将导出和调整SQL调优集的相应数据库版本。在此示例中，将使用格式创建临时表，以便可以将其导出到运行Oracle Database 11g第1版的系统，稍后将使用SQL Tuning Advisor进行调整。对于其他数据库版本，请参阅该版本的Oracle Database PL / SQL包和类型参考 。

  3. 将SQL调优集导出到临时表：

    BEGIN DBMS_SQLTUNE.PACK_STGTAB_SQLSET（sqlset_name =>'SUB_STS1'，sqlset_owner =>'JOHNDOE'，staging_table_name =>'STG_TAB1'，staging_schema_owner =>'JOHNDOE'，db_version => DBMS_SQLTUNE.STS_STGTAB_11_1_VERSION）;结束; /

  4. 使用选择的机制（例如Oracle Data Pump或数据库链接）将登台表移动到远程数据库（执行SQL工作负载的位置）。

  5. 在远程数据库上，从登台表导入SQL调优集：

    BEGIN DBMS_SQLTUNE.UNPACK_STGTAB_SQLSET（sqlset_name =>'SUB_STS1'，staging_table_name =>'STG_TAB1'，replace => TRUE）;结束; /

  6. 通过运行SQL Tuning Advisor调整SQL调优集中的回归SQL语句：

    BEGIN sts_name：='SUB_STS1'; sts_owner：='JOHNDOE'; tune_task_name：='TUNE_TASK1'; tname：= DBMS_SQLTUNE.CREATE_TUNING_TASK（sqlset_name => sts_name，sqlset_owner => sts_owner，task_name => tune_task_name）; EXEC DBMS_SQLTUNE.SET_TUNING_TASK_PARAMETER（：tname，'APPLY_CAPTURED_COMPILENV'，'FALSE'）; exec_name：= DBMS_SQLTUNE.EXECUTE_TUNING_TASK（tname）;结束; /

    注意：

    本APPLY_CAPTURED_COMPILENV使用的APPLY_CAPTURED_COMPILENV参数仅受Oracle Database 11g第1版及更新版本的支持。如果要从早期版本的Oracle数据库测试数据库升级，则SQL Tuning Advisor将使用存储在SQL调优集中的环境变量。

11.2.1标准类型与Oracle类型

oracle.sql的Oracle数据类型以与数据库使用的相同位格式存储数据。在Oracle Database 10g之前的Oracle JDBC驱动程序版本中，Oracle数据类型通常更有效。从Oracle Database 10g开始 ，JDBC驱动程序已大幅更新。因此，在大多数情况下，标准Java类型优先于oracle.sql.的数据类型oracle.sql.* 。特别是java.lang.String比oracle.sql.更有效oracle.sql.CHAR 。

通常，Oracle建议您使用Java标准类型。例外情况是：

  • 使用oracle.jdbc.OracleData而不是java.sql.SqlData如果OracleData功能更适合您的需求。

  • 使用oracle.sql.NUMBER而不是java.lang.如果需要保留浮点数的确切值，请Double 。Oracle NUMBER是十进制表示，Java Double和Float是二进制表示。从一种格式转换为另一种格式可能导致所表示的实际值略有不同。另外，可以使用两种格式表示的值的范围是不同的。

    使用oracle.sql.NUMBER而不是java.math.BigDecimal在性能至关重要且您没有操纵值时，只需读取和写入它们。

  • 使用oracle.sql.DATE或oracle.sql.如果您使用的是早于JDK 6的JDK版本，则为TIMESTAMP 。使用java.sql.Date或java.sql.如果您使用的是JDK 6或更高版本，请使用Timestamp 。

    注意：

    由于JDK 6之前的所有Java版本中都存在错误，因此构建了java.lang.Date和java.lang.Timestamp对象很慢，尤其是在多线程环境中。JDK 6中修复了此错误。

  • 使用oracle.sql.仅当您拥有来自某个外部源的数据时才会显示CHAR ，该外部源已在Oracle字符集编码中表示。在所有其他情况下，您应该使用java.lang.String 。

  • STRUCT ， ARRAY ， BLOB ， CLOB ， REF和ROWID都是相应JDBC标准接口类型的所有实现类。因此，使用Oracle扩展类型没有任何好处，因为它们与JDBC标准类型相同。

  • BFILE ， TIMESTAMPTZ和TIMESTAMPLTZ在JDBC标准中没有任何表示。您必须使用这些Oracle扩展。

  • 在所有其他情况下，您应该使用标准JDBC类型而不是Oracle扩展。

注意：

如果将oracle.sql数据类型转换为Java标准数据类型，则使用oracle.sql数据类型的好处将丢失。

11.2.2关于转换SQL NULL数据

Java通过Java值null表示SQL NULL数据。Java数据类型分为两类：基本类型（如byte ， int和float ）和对象类型（如类实例）。基元类型不能表示null 。相反，它们将null存储null值，如JDBC规范所定义。当您尝试解释结果时，这可能会导致歧义。

相反，Java对象类型可以表示null 。Java语言定义了对应于可以表示null每个基本类型的对象容器类型。必须将对象容器类型用作SQL数据的目标，以便检测SQL NULL而不会产生歧义。

11.2.3关于测试NULL

您不能使用关系运算符来相互比较NULL值或使用其他价值观。例如，即使COMMISSION_PCT列包含一个或多个NULL值，以下SELECT语句也不返回任何行。 

PreparedStatement pstmt = conn.prepareStatement（“SELECT * FROM EMPLOYEES WHERE COMMISSION_PCT =？“）; pstmt.setNull（1，java.sql。类型。VARCHAR）;

该下一个示例显示了在某些返回值可能为NULL时如何比较值的相等性。下面的代码返回所有的FIRST_NAME从EMPLOYEES表是NULL ，如果不存在用于205价值COMM 。

PreparedStatement pstmt = conn.prepareStatement（“SELECT FIRST_NAME FROM EMPLOYEES WHERE COMMISSION_PCT =？或（（COMM是空）和（？IS NULL））“）; pstmt.setBigDecimal（1，new BigDecimal（205））; pstmt.setNull（2，java.sql。类型。VARCHAR）;

11.3结果集和语句扩展

11.4.7.1输入数据绑定

绑定数据有三种输入方式：

  • 直接绑定，数据本身放在绑定缓冲区中

  • 流绑定数据流的绑定

  • LOB绑定创建临时lob，使用LOB API放置在LOB中的数据，以及LOB定位器的字节放在绑定缓冲区中

这三种绑定在性能上有一些差异，对批处理有影响。直接绑定很快，批处理很好。流绑定较慢，可能需要多次往返，并关闭批处理。LOB绑定非常慢，需要多次往返。批处理工作，但可能是一个坏主意。它们还具有不同的大小限制，具体取决于SQL语句的类型。

对于SQL参数，标准参数类型（如RAW和VARCHAR2的长度由目标列的大小确定。对于PL / SQL参数，大小限制为固定的字节数，即32766。

在Oracle数据库10g第2版，某些变化到作出setString ， setCharacterStream ， setAsciiStream ， setBytes ，和setBinaryStream的方法PreparedStatement 。这些API的原始行为是：

  • setString ：直接绑定字符

  • setCharacterStream ：流绑定字符

  • setAsciiStream ：字节的流绑定

  • setBytes ：字节的直接绑定

  • setBinaryStream ：字节的流绑定

从Oracle数据库10g第2版开始，提供了基于数据大小和SQL语句类型的绑定模式之间的自动切换。

setBytes和setBinaryStream

对于SQL，直接绑定用于最大2000的大小，并且流绑定用于更大。

对于PL / SQL，直接绑定用于最大为32766的大小，而LOB绑定用于更大。

setString，setCharacterStream和setAsciiStream

对于SQL，直接绑定最多使用32766个Java字符，而流绑定用于更大的。这与字符集无关。

对于PL / SQL，您必须注意数据库字符集中的字符数据的字节大小或国家字符集，具体取决于使用参数形式的设置。直接绑定用于字节长度小于32766且LOB绑定用于更大的数据。

对于固定长度字符集，将Java字符数据的长度乘以固定字符大小（以字节为单位），并将其与限制值进行比较。对于可变长度字符集，有三种基于Java字符长度的情况，如下所示：

  • 如果字符长度小于32766除以最大字符大小，则使用直接绑定。

  • 如果字符长度大于32766除以最小字符大小，则使用LOB绑定。

  • 如果字符长度介于两者之间，并且如果转换后的字节的实际长度小于32766，则使用直接绑定，否则使用LOB绑定。

注意：

当PL / SQL过程嵌入在SQL语句中时，绑定操作是不同的。

服务器端内部驱动程序具有以下附加限制：

  • setString ， setCharacterStream ，并setASCIIStream的API不支持SQL CLOB列时，以字符数据大小超过32767个字节

  • 当数据大小超过32767字节时，SQL BLOB列不支持setBytes和setBinaryStream API

注意：

不要将这些API与服务器端内部驱动程序一起使用，而不必仔细检查客户端代码中的数据大小。

29.4.1进行脱机表空间和数据文件的用户管理备份

备份脱机表空间时请注意以下准则：

  • 您不能使SYSTEM表空间或具有活动还原段的表空间脱机。以下技术不能用于此类表空间。

  • 假设表在表空间Primary ，其索引在表空间Index 。将表空间Index脱机而离开表空间Primary联机可能会在针对位于Primary节点中的索引表发出数据操作语言（DML）时导致错误。仅当优化程序选择的访问方法必须访问Index表空间中的Index时，才会出现此问题。

要备份脱机表空间：

  1. 在开始备份表空间之前，通过查询DBA_DATA_FILES视图来标识表空间的数据文件。例如，假设您要备份users表空间。在SQL * Plus中输入以下语句：
    SELECT TABLESPACE_NAME，FILE_NAME FROM SYS.DBA_DATA_FILES WHERE TABLESPACE_NAME ='USERS'; TABLESPACE_NAME FILE_NAME ------------------------------- ----------------- --------------- USERS /oracle/oradata/trgt/users01.dbf

    在此示例中，/ /oracle/oradata/trgt/users01.dbf是与users表空间中的数据文件对应的完全指定的文件名。

  2. 如果可能，请使用普通优先级使表空间脱机，因为它可以保证您可以随后将表空间联机而无需恢复。例如：
    SQL> ALTER TABLESPACE用户OFFLINE NORMAL;

    ％cp /oracle/oradata/trgt/users01.dbf / d2 / users01_'date“+％m_％d_％y”'。dbf

    ALTER TABLESPACE用户在线;

    ALTER SYSTEM ARCHIVE LOG CURRENT;

    SELECT TABLESPACE_NAME，FILE_NAME FROM SYS.DBA_DATA_FILES WHERE TABLESPACE_NAME ='USERS'; TABLESPACE_NAME FILE_NAME ------------------------------- ----------------- --- USERS /oracle/oradata/trgt/users01.dbf USERS /oracle/oradata/trgt/users02.dbf
  2. 标记在线表空间备份的开头。例如，以下语句标记表空间users的联机备份的开始：
    SQL> ALTER TABLESPACE用户BEGIN BACKUP;

    警告：

    如果不使用BEGIN BACKUP标记联机表空间备份的开头并等待此语句在启动联机表空间副本之前完成，则生成的数据文件副本不可用于后续恢复操作。尝试恢复此类备份存在风险，并且可能返回导致数据不一致的错误。例如，尝试的恢复操作可能会发出模糊文件警告，并可能导致无法打开的数据库不一致。

  3. 使用操作系统命令备份联机表空间的联机数据文件。例如，Linux和UNIX用户可能会输入：
    ％cp /oracle/oradata/trgt/users01.dbf / d2 / users01_'date“+％m_％d_％y”'。dbf％cp /oracle/oradata/trgt/users02.dbf / d2 / users02_'date“+ ％M_％D_％Y“”。DBF
  4. 备份联机表空间的数据文件后，使用END BACKUP选项运行SQL语句ALTER TABLESPACE 。例如，以下语句结束表空间users的在线备份：
    SQL> ALTER TABLESPACE用户END BACKUP;
  5. 归档未归档的重做日志，以便归档恢复表空间备份所需的重做。例如，输入：
    SQL> ALTER SYSTEM ARCHIVE LOG CURRENT;

    警告：

    如果您未能将表空间从备份模式中删除，则Oracle数据库会继续将此表空间中的数据块副本写入联机重做日志，从而导致性能问题。此外，如果您尝试在表空间仍处于备份模式的情况下关闭数据库，则会收到ORA-01149错误。

29.4.2.2进行多个用户管理的在线读/写表空间备份

备份多个在线表空间时，可以串行或并行备份它们。根据您的需要，使用以下任一过程：

  • 并行备份在线表空间

  • 备份在线表空间

29.4.2.2.1并行备份在线表空间

您可以同时创建需要在备份模式下进行备份的多个表空间的数据文件副本。但请注意，通过将所有表空间放在联机模式中，如果受影响的表空间上存在大量更新活动，则可以生成大型重做日志，因为重做必须包含每个已更改数据文件中每个已更改数据块的副本。在使用此处概述的过程之前，请务必考虑可能重做的大小。

要并行备份联机表空间：

  1. 通过一起发出所有必需的ALTER TABLESPACE语句来准备联机表空间以进行备份。例如，将表空间users ， tools和indx置于备份模式，如下所示：
    SQL> ALTER TABLESPACE用户BEGIN BACKUP; SQL> ALTER TABLESPACE工具BEGIN BACKUP; SQL> ALTER TABLESPACE indx BEGIN BACKUP;

    如果要备份所有表空间，可以使用以下命令：

    SQL> ALTER DATABASE BEGIN BACKUP;
  2. 备份联机表空间的所有文件。例如，Linux或UNIX用户可能使用*.dbf后缀备份数据文件，如下所示：
    ％cp $ ORACLE_HOME / oradata / trgt / *。dbf / disk2 / backup /
  3. 将表空间从备份模式中取出，如以下示例所示：
    SQL> ALTER TABLESPACE用户END BACKUP; SQL> ALTER TABLESPACE工具END BACKUP; SQL> ALTER TABLESPACE indx END BACKUP;

    同样，如果要同时处理所有数据文件，则可以使用ALTER DATABASE命令而不是ALTER TABLESPACE ：

    SQL> ALTER DATABASE END BACKUP;
  4. 归档联机重做日志，以便恢复表空间备份所需的重做可用于以后的介质恢复。例如，输入：
    SQL> ALTER SYSTEM ARCHIVE LOG CURRENT;
29.4.2.2.2备份在线表空间

您可以将所有需要联机备份的表空间放在备份模式中，一次一个。 Oracle建议使用串行备份选项，因为它可以最小化ALTER TABLESPACE之间的时间... BEGIN/END BACKUP语句。在线备份期间，会为表空间生成更多重做信息，因为整个数据块都会复制到重做日志中。

要连续备份在线表空间：

  1. 准备用于在线备份的表空间。例如，要将表空间users置于备份模式，请输入以下内容：
    SQL> ALTER TABLESPACE用户BEGIN BACKUP;

    在这种情况下，您可能不希望使用ALTER DATABASE BEGIN BACKUP将所有表空间同时置于备份模式，因为在线模式下为表空间生成了不必要的重做日志信息量。

  2. 备份表空间中的数据文件。例如，输入：
    ％cp /oracle/oradata/trgt/users01.dbf / d2 / users01_'date“+％m_％d_％y”'。dbf
  3. 使表空间退出备份模式。例如，输入：
    SQL> ALTER TABLESPACE用户END BACKUP;
  4. 对每个剩余的表空间重复此过程。
  5. 归档未归档的重做日志，以便归档恢复表空间备份所需的重做。例如，输入：
    SQL> ALTER SYSTEM ARCHIVE LOG CURRENT;
29.4.2.3在实例失败或关机中止后结束备份

以下情况可能导致表空间备份失败并且不完整：

  • 备份完成，但你没有运行ALTER TABLESPACE ... END BACKUP声明。

  • 实例故障或SHUTDOWN ABORT中断备份。

每当需要从故障中恢复时，如果数据文件在尝试打开时处于备份模式，则数据库不会打开数据文件，直到发出恢复命令或数据文件被取出为止。备份模式。

在高可用性情况下，以及在没有数据库管理员（DBA）监视数据库的情况下，用户干预的要求是不可容忍的。因此，您可以编写执行以下操作的故障恢复脚本：

  2. 运行ALTER DATABASE END BACKUP语句

或者，您可以在系统因备份模式下的表空间失败后采取以下手动措施：

  • 恢复数据库并避免完全发出END BACKUP语句。

29.8在SUSPEND模式下进行用户管理的备份

  • 在挂起的数据库中进行备份

29.8.1关于暂停/恢复功能

某些第三方工具允许您镜像一组磁盘或逻辑设备，即在另一个位置维护主数据的精确副本，然后拆分镜像 。拆分镜像涉及分离副本，以便您可以单独使用它们。

使用SUSPEND / RESUME功能，您可以暂停数据库的I / O，然后拆分镜像并备份分割镜像。通过使用此功能补充备份模式功能，您可以挂起数据库I / O，以便不会执行新的I / O.然后，您可以访问已挂起的数据库以进行备份，而不会受到I / O干扰。

通常，您不需要使用SUSPEND / RESUME来进行分割镜像备份，但是如果您的系统要求数据库缓存在分割卷之前没有脏缓冲区 ，则必须使用它。在拆分操作发生时，某些RAID设备可以从暂停写入中受益;您的RAID供应商可以建议您的系统是否会受益于此功能。

ALTER SYSTEM SUSPEND语句通过暂停数据文件头，数据文件和控制文件的I / O来挂起数据库。当数据库挂起时，所有预先存在的I / O操作都可以完成;但是，任何新的数据库I / O访问尝试都会排队。

ALTER SYSTEM SUSPEND和ALTER SYSTEM RESUME语句在数据库上运行，而不仅仅是实例。如果在Oracle RAC配置中的一个系统上输入ALTER SYSTEM SUSPEND语句，则内部锁定机制会跨实例传播暂停请求，从而暂停给定群集中所有活动实例的I / O操作。

29.8.2在挂起的数据库中进行备份

成功暂停数据库后，您可以将数据库备份到磁盘或中断镜像。由于挂起数据库并不能保证立即终止I / O，因此Oracle建议您在ALTER SYSTEM SUSPEND语句之前加上BEGIN BACKUP语句，以便将表空间置于备份模式。

您必须使用传统的用户管理备份方法来备份分割镜像。RMAN无法进行数据库备份或复制，因为这些操作需要读取数据文件头。数据库备份完成或镜像重新启动后，您可以使用ALTER SYSTEM RESUME语句恢复正常的数据库操作。

在不拆分镜像的情况下备份挂起的数据库可能会导致数据库中断，因为在此期间数据库无法访问。但是，如果通过拆分镜像进行备份，则中断是名义上的。中断时间取决于要刷新的缓存大小，数据文件的数量以及中断镜像所需的时间。

请注意SUSPEND/RESUME功能的以下限制：

  • 在Oracle RAC配置中，请勿在原始节点挂起时启动新实例。

  • ALTER SYSTEM SUSPEND或ALTER SYSTEM RESUME语句不启动检查点。

  • 在数据库挂起时，您不能使用IMMEDIATE ， NORMAL或TRANSACTIONAL选项发出SHUTDOWN 。

  • 在挂起的数据库上发出SHUTDOWN ABORT会重新激活数据库。这可以防止介质恢复或故障恢复进入无响应状态。

要在SUSPEND模式下进行分割镜像备份：

  1. 将数据库表空间置于备份模式。例如，要将表空间users置于备份模式，请输入：
    ALTER TABLESPACE用户BEGIN BACKUP;

    如果要备份数据库的所有表空间，则可以使用：

    ALTER DATABASE BEGIN BACKUP;

    警告：

    不要使用ALTER SYSTEM SUSPEND语句替代将表空间置于备份模式。

  2. 如果镜像系统在发生磁盘写入时分割镜像时出现问题，则挂起数据库。例如，发出以下语句：
    ALTER SYSTEM SUSPEND;
  3. 通过查询V$INSTANCE视图验证数据库是否已挂起。例如：
    从V $ INSTANCE中选择DATABASE_STATUS; DATABASE_STATUS -----------------暂停

  5. 结束数据库暂停。例如，发出以下语句：

29.10使用第三方快照技术进行备份

您可以使用存储快照优化来获取数据库的第三方快照，而无需将数据库置于备份模式。快照必须符合本节中描述的要求。

存储快照优化具有以下优势：

  • 消除了将数据库置于备份模式所带来的复杂性和开销。

  • 使用RECOVER ...在一个步骤中执行恢复RECOVER ...SNAPSHOT TIME命令。您可以恢复到当前时间或拍摄快照后的某个时间点。

要使用快照存储优化，第三方快照技术必须符合以下要求：

  • 快照期间数据库崩溃一致。

  • 快照保留每个文件的写入顺序。

  • 快照技术存储快照完成的时间。

如果供应商无法保证符合这些要求，则必须将数据文件放入使用带有BEGIN BACKUP子句的ALTER DATABASE或ALTER TABLESPACE语句的备份模式 。在创建快照之前，将数据文件置于备份模式。当表空间处于备份模式时，数据库会在修改块之前将整个块的前映像写入重做流。数据库还记录联机重做日志中块的更改。备份模式也冻结数据文件检查点，直到文件从备份模式中删除。Oracle数据库执行此安全措施，因为它无法保证第三方备份工具在复制数据块之前复制文件头。创建快照后，立即使用带有END BACKUP子句的ALTER DATABASE或ALTER TABLESPACE命令将数据文件从备份模式中取出。您无需等到快照实际复制到备份媒体以结束备份模式。

注意：

当备份的数据库驻留在Oracle自动存储管理（Oracle ASM）上时，RMAN不支持基于存储的复制技术。

PRCA-01000：ASM磁盘组{0}不存在

原因：尝试无法检索未配置的ASM磁盘组。

操作：如果指定的磁盘组不存在，则使用sqlplus创建磁盘组。在ASM中挂载磁盘组，使用CRS注册磁盘组。

PRCA-01001：ASM磁盘组资源不存在

原因：没有向CRS注册磁盘组资源。

操作：使用sqlplus创建磁盘组并在ASM中装入磁盘组，该磁盘组使用CRS注册磁盘组。

PRCA-01002：无法删除ASM磁盘组{1}的CRS资源{0}

原因：尝试删除指定ASM磁盘组的CRS资源失败。

操作：检查随附的错误消息以获取详细信息。

PRCA-01003：无法创建ASM {0}资源{1}

原因：尝试为ASM添加CRS资源失败。

操作：确保CRS堆栈正在运行，运行srvctl add asm的用户也是CRS主目录的所有者，并检查随附的错误消息以获取详细信息。

PRCA-01005：无法检索ASM {0}的服务器参数文件

原因：无法检索ASM服务器参数文件。

操作：确保CRS堆栈正在运行，已配置ASM资源，并且用户对ASM资源具有读取权限后，请重试该请求。

PRCA-01006：无法更新ASM {1}的服务器参数文件{0}

原因：无法更新ASM服务器参数文件。

操作：确保CRS堆栈正在运行，已配置ASM资源，并且用户对ASM资源具有更新权限后，请重试该请求。

PRCA-01007：ASM {0}未使用初始化参数：{1}

原因：指定的ASM实例不使用显示的初始化参数。

操作：检查随附的错误消息以获取详细信息。

PRCA-01016：无法删除ASM实例{1}的ASM资源{0}

原因：尝试删除指定的ASM配置失败。

操作：检查随附的错误消息以获取详细信息。

PRCA-01017：ASM {0}不存在

原因：尚未配置ASM。

操作：配置ASM，然后重试。

PRCA-01018：无法检索ASM {0}的ASM实例列表

原因：无法检索ASM实例列表。

操作：确保crs堆栈已启动且已配置ASM资源。

PRCA-01019：ASM {0}未运行

原因： ASM资源没有任何正在运行的实例。

操作：使用'srvctl start asm'命令启动asm资源。

PRCA-01020：无法为卷设备{1}创建ACFS文件系统资源{0}

原因：无法创建指定卷设备路径的ACFS文件系统资源。可能的原因：1）CRS堆栈不可用。2）ASM实例未运行。3）用户没有使用具有管理访问权限的帐户运行。 （root，Administrator等）4）无法识别用户请求资源使用的挂载点。

关于领域

领域是必须为给定应用程序保护的数据库模式，数据库对象和数据库角色的分组。

将领域视为数据库对象的保护区域。模式是数据库对象（如表，视图和包）的逻辑集合，而角色是特权的集合。通过将模式和角色安排到功能组中，您可以控制用户对这些组使用系统特权的能力，并防止数据库管理员或具有系统特权的其他强大用户访问未经授权的数据。Oracle Database Vault不会替换现有Oracle数据库中的自主访问控制模型。对于领域和命令规则，它在此模型的顶部充当层。

Oracle Database Vault提供两种类型的域：常规域和强制域。两种域类型都可以选择性地保护整个模式或模式中的关键对象，例如表和索引。对于常规领域，对象所有者或已被授予对象权限的用户可以在没有领域授权的情况下执行查询或DML操作，但必须具有领域授权才能执行DDL操作。强制领域为领域内的对象提供更强大的保护。强制域阻止基于对象权限和基于系统权限的访问，并且不允许具有对象权限的用户在没有域授权的情况下执行查询，DML或DDL操作。换句话说，如果对象受到强制领域的保护，即使对象所有者也无法在没有适当领域授权的情况下访问他或她自己的对象。

对于使用Oracle闪回技术的数据库，常规域和强制域都会对闪回表强制执行相同的行为。如果用户被授权到领域，则用户可以在受领域保护的表上执行FLASHBACK TABLE SQL语句。

对于使用信息生命周期管理（ILM）的数据库，Database Vault管理员可以使用DBMS_MACADM.AUTHORIZE_MAINTENANCE_USER和DBMS_MACADM.UNAUTHORIZE_MAINTENANCE_USER过程来控制谁可以对受域保护的对象执行ILM操作。

您可以将模式，模式中特定类型的所有对象或模式中的单个对象注册到领域中。创建领域后，您可以注册一组架构对象或角色（安全对象）以进行领域保护，并授权一组用户或角色访问安全对象。受常规域保护的对象允许DML访问具有直接对象授权的用户。

例如，您可以创建领域以保护会计部门中使用的所有现有数据库模式。该领域禁止未经授权的任何用户使用系统权限来访问安全的记帐数据。当整个模式受到保护时，模式中的所有对象都受到保护，包括表，索引，过程和其他对象。

您可以在Oracle Database Vault中创建的域上运行报告。您可以在开发，测试甚至生产阶段使用模拟模式来仅记录域违规而不是阻止访问。这使您可以使用Database Vault领域快速测试应用程序。

您可以使用Oracle Enterprise Manager Cloud Control中的Oracle Database Vault Administrator页面配置域。或者，您可以使用Oracle Database Vault提供的PL / SQL接口和包来配置域。

强制领域限制用户访问领域内的对象

默认情况下，拥有或具有对象权限的用户可以在没有显式域授权的情况下访问受域保护的对象。

您可以选择配置域以通过将其配置为强制域来阻止这些用户的访问。强制领域阻止基于系统特权的访问以及基于对象特权的访问。这意味着即使对象所有者无权访问该领域，也无法访问该领域。仅当用户或角色有权这样做时，用户才能访问强制领域中的安全对象。

强制领域具有以下附加特征：

  • 如果同一对象上有多个强制域，则必须先授权所有必需域上的用户或角色，然后才能访问受保护对象。

  • 如果角色受强制域保护，则除了域所有者之外，不能从受保护角色授予或撤消任何特权。

  • 您可以将在早期版本中创建的常规域更新为必需域。这样，您可以阻止所有者访问和对象特权用户访问受领域保护的对象。

  • SYS拥有的对象已受到数据字典保护的保护，并且不受Oracle Database Vault单独保护。

强制领域具有以下好处：

  • 强制域 可以阻止对象所有者和对象特权用户。 在以前的版本中，阻止这些用户只能通过定义复杂的命令规则来完成。

  • 强制领域为访问控制提供了更灵活的配置。 例如，假设您希望允许用户访问具有特定条件的对象，例如在白天的特定时间范围内。您无法向该用户授予对象权限，因为领域不会阻止对象权限。您只能向用户授予系统权限，然后使用规则将此用户授权到域，或直接对命令创建命令规则。这些解决方案在计算成本方面要么非常昂贵，要么是不合需要的，因为它们需要向用户过度授予诸如系统特权之类的特权。使用强制域时，您只需要为用户授予对象权限，并具有特定条件的规则，然后授权该用户成为域所有者或参与者。因此，对于强制领域，Oracle Database Vault策略具有更大的灵活性，而无需授予用户过多的权限。

  • 强制领域在修补程序升级期间添加了一层保护。 在修补程序升级期间，数据库管理员可能需要直接访问受领域保护的对象才能对该对象执行修补程序。如果存在包含敏感数据的表（例如社会保险号），则可以在修补程序升级期间使用强制域保护这些表免受管理员访问。当修补完成且数据库管理员不再需要访问对象时，您可以禁用强制域保护，然后重新启用正常的应用程序域保护，以便应用程序保护可以恢复到正常状态。

  • 您可以使用强制域在运行时保护表。 在运行时，应用程序数据可以存储在许多表中。最好让单个用户（如运行时模式）访问这些表，以便您可以维护数据的完整性和正确性。如果应用程序数据分散在许多不同的模式中，那么具有对象权限的模式所有者和用户可以在他们直接登录数据库时更改数据。为了确保用户无法在不经过运行时模式的过程的情况下更新这些表，您可以使用强制域来保护表，以便只有授权用户的过程才能访问它们。由于常规域不会阻止对象所有者和对象特权用户，因此可以使用强制域来阻止它们。这样，只有授权用户才能在运行时访问这些表。

  • 您可以通过阻止更改已配置的角色来冻结安全设置。

关于Realm-Secured Objects

领域安全对象定义领域 - 领域保护的一组模式和数据库对象和角色。

您可以创建以下类型的保护：

  • 来自多个数据库帐户或模式的对象可以位于同一领域。

  • 一个对象可以属于多个领域。

    如果对象属于多个领域，则Oracle Database Vault会检查领域是否有适当的授权。对于SELECT ，DDL和DML语句，只要用户是其中一个领域的参与者，并且如果命令规则允许，则允许用户输入的命令。对于多个领域中数据库角色的GRANT和REVOKE操作，执行GRANT或REVOKE操作的人员必须是领域所有者。架构所有者可以对受多个常规域保护的对象执行DML操作。

    如果其中一个领域是强制领域，那么想要访问该对象的用户必须是领域所有者或强制领域的参与者。在授权检查过程中，将忽略非强制域。如果有多个强制域保护对象，则必须在所有强制域中授权想要访问该对象的用户。

  • SYS拥有的对象已受到数据字典保护的保护，并且不受Oracle Database Vault单独保护。

关于领域授权

领域授权建立了一组数据库帐户和角色，用于管理或访问领域中受保护的对象。

您可以向帐户或角色授予域授权，以允许在以下情况下使用其系统特权：

  • 当用户必须创建或访问领域安全对象时

  • 当用户必须授予或撤消领域安全角色时

已被授予领域授权作为领域所有者或领域参与者的用户可以使用其系统特权来访问领域中的安全对象。

请注意以下事项：

  • 领域所有者不能将其他用户作为所有者或参与者添加到他们的领域。只允许具有DV_OWNER或DV_ADMIN角色的用户将用户作为所有者或参与者添加到领域。

  • 已被授予DV_OWNER角色的用户可以将自己添加到领域授权中。

  • 领域所有者（但不是领域参与者）可以授予或撤销领域安全角色，或者授予或撤销领域安全对象的对象权限给任何人。

  • 用户可以被授予领域所有者或领域参与者，但不能同时授予两者。但是，您可以更新现有领域授权的授权类型。

使用“编辑域”页面来管理域授权。您可以创建，编辑和删除领域授权。

多租户环境中的领域授权

在多租户环境中，常见域授权的规则和行为类似于其他常见对象的授权。

共同领域的本地授权

对公共领域的本地授权是指用户对该用户正在访问的PDB的授权。

对共同领域的本地授权规则如下：

  • 通常被授予DV_OWNER或DV_ADMIN角色的用户可以向普通用户，公共角色，本地用户和本地角色授予本地授权。常见的DV_OWNER或DV_ADMIN用户还可以从PDB中的公共DV_OWNER删除本地授权。

  • 本地Database Vault管理员可以在PDB中本地授权（即，向本地用户和普通用户授予本地授权）。常见的Database Vault管理员还可以在每个PDB中授予授权。通用域授权只能由应用程序根目录中的公共Database Vault管理员授予。

  • 公共Database Vault管理员可以在PDB中向公共领域添加或删除本地授权。

  • 如果普通用户仅具有对公共领域的本地授权，则该用户无法访问除本地授权之外的任何其他PDB中的公共领域。

  • 普通用户或共同角色可以同时具有对公共领域的本地授权和公共授权。从公共领域删除普通用户的本地授权不会影响普通用户的公共授权。从公共领域删除普通用户的公共授权不会影响普通用户的本地授权。

共同领域的共同授权

公共领域的公共授权是指普通用户或公共角色在应用程序根目录中的授权，而授权在启用Database Vault的每个容器中生效。

对共同领域的本地授权规则如下：

  • 通常被授予DV_OWNER或DV_ADMIN角色的用户可以向应用程序根目录中的公共用户或角色授予公共领域授权。此通用Database Vault管理员可以在应用程序根目录中执行常见授权的删除。

  • 此通用授权适用于已在CDB中启用Database Vault的容器。

  • 如果普通用户被授权到应用程序根目录中的公共领域，则该用户可以访问由应用程序根目录中的公共领域和任何应用程序PDB保护的对象。

  • 与公共领域关联的任何规则集都必须是通用规则集。添加到与公共授权关联的公共规则集的规则不能涉及任何本地对象。

领域如何运作

当适当特权的数据库帐户发出影响领域内对象的SQL语句时，会发生一组特殊的活动。

这些权限包括DDL，DML， EXECUTE ， GRANT ， REVOKE,或SELECT权限。

  1. SQL语句是否影响领域保护的对象？

    如果是，则转到步骤2 。如果不是，那么领域不会影响SQL语句。转到第7步。如果受命令影响的对象在任何领域都不受保护，则领域不会影响正在尝试的SQL语句。

  2. 领域是强制领域还是常规领域？

    如果是，则转到步骤4 。如果是常规域，则转到步骤3 。

  3. 数据库帐户是否使用系统特权来执行SQL语句？

    如果是，则转到步骤4 。如果否，则转到步骤6 。如果会话仅对SELECT ， EXECUTE和DML语句对相关对象具有对象特权，则不会强制实施领域保护。领域可防止对受领域保护的对象或角色使用任何系统特权。即使具有受常规域保护的对象的对象权限的用户也无法执行DDL操作。

    请记住，如果O7_DICTIONARY_ACCESSIBILITY初始化参数已设置为TRUE ，则非SYS用户可以访问SYS模式对象。为了更好的安全性，请确保将O7_DICTIONARY_ACCESSIBILITY设置为FALSE 。

  4. 数据库帐户是领域所有者还是领域参与者？

    如果是，则转到步骤5 。否则，将发生域违规，并且不允许该语句成功。如果命令是一个GRANT或REVOKE由境界保护的作用，或者GRANT或REVOKE由境界保护的对象上的对象特权，那么会话必须直接或间接授权的境界所有者通过角色。

  5. 数据库帐户的领域授权是否有条件地基于规则集？

    如果是，则转到步骤6 。如果不是，则转到步骤7 。

  6. 规则集评估为TRUE吗？

    如果是，则转到步骤7 。如果不是，则存在域违规，因此不允许SQL语句成功。

  7. 命令规则是否阻止命令执行？

    如果是，则存在命令规则违规并且SQL语句失败。如果不是，则没有域或命令规则违规，因此命令成功。

    例如， HR帐户可以具有DROP ANY TABLE权限，并且可以是HR领域的所有者，但是命令规则可以防止HR丢弃HR模式中的任何表，除非它在其每月维护窗口期间。命令规则适用于ANY系统特权和对象特权的使用，并在领域检查后进行评估。

此外，由于会话在领域中被授权，因此并不意味着帐户可以完全控制由领域保护的对象。境界授权不会隐授予额外的特权的帐户。该帐户仍必须具有系统权限或对象权限才能访问对象。例如，帐户或角色可以具有SELECT ANY表特权，并且是HR领域的参与者。这意味着授予该角色的帐户或帐户可以查询HR.EMPLOYEES表。作为领域的参与者并不意味着帐户或角色可以DROP HR.EMPLOYEES表。Oracle Database Vault不会替换现有Oracle数据库中的自主访问控制模型。对于领域和命令规则，它在此模型的顶部充当层。

请注意以下事项：

  • 默认情况下，保护域中的表不会保护视图。无论视图是在表添加到域之前还是之后创建的，都应将任何必须保护的视图添加到域中。

  • 对于调用者访问领域受保护对象的正确过程，必须授权该过程的调用者。

  • 请注意，如果已向PUBLIC授予对表的访问权限，则域保护不会保护表。例如，如果将SELECT ON table_name授予PUBLIC ，则每个用户都可以访问table_name （除非该表受强制域保护），即使此表受域保护也是如此。作为最佳实践，从PUBLIC撤消不必要的权限。

关于领域中的授权

领域通过系统权限保护数据不被访问。

领域不会为数据所有者或参与者提供额外的权限。

领域授权提供了一种运行时机制，用于逻辑检查是否应允许或拒绝用户的命令来访问命令中指定的对象并继续执行。

系统特权是彻底的数据库特权，例如CREATE ANY TABLE和DELETE ANY TABLE 。这些权限通常适用于模式，并且不需要对象权限。数据字典视图（如DBA_SYS_PRIVS ， USER_SYS_PRIVS和ROLE_SYS_PRIVS列出了数据库帐户和角色的系统特权。数据库授权通常适用于不受领域保护的对象。但是，必须授权用户作为领域所有者或参与者才能成功地使用他或她对系统所保护的对象的系统特权。领域违规会阻止使用系统特权，并且可以进行审计。

强制域阻止基于对象特权的访问和基于系统特权的访问。这意味着即使对象所有者无权访问该领域，也无法访问该领域。仅当用户或角色有权这样做时，用户才能访问强制领域中的安全对象。

访问受领域保护的对象

您可以通过领域保护对象，但仍然可以访问属于此领域受保护对象的对象。

例如，假设您围绕特定表创建领域。但是，您希望用户能够在此表上创建索引。您可以按照以下方案完成此操作，具体取决于以下方案。

  • 用户没有CREATE ANY INDEX权限。 作为表的域所有者，将CREATE INDEX ON table权限授予必须创建索引的用户。

  • 用户具有CREATE ANY INDEX权限。 在这种情况下，创建另一个领域并将所有索引类型作为安全对象，并将该用户参与者授权授予该领域。（请记住，单独使用CREATE ANY INDEX权限不足以让非领域参与者在受领域保护的表中创建索引。）

  • 您希望所有数据库管理员都能够创建索引，并且他们具有CREATE ANY INDEX权限。 在数据保护领域中，指定除索引类型之外要保护的所有对象类型。这允许所有管理员为受保护的表创建索引。

设计领域指南

Oracle为设计领域提供了一套指南。

  • 根据构成数据库应用程序的模式和角色创建领域。

    使用维护应用程序对象所需的最小和特定角色和系统特权定义数据库角色，并将角色授予指定帐户。然后，您可以将角色添加为领域的授权成员。对于受领域保护且应用程序所需的对象的对象级特权，请创建角色并将这些最小和特定对象级特权授予该角色，然后向该命名帐户授予此角色。在大多数情况下，除非已使用ANY风格的系统特权，否则不需要在领域中授权这些类型的角色。使用最小权限原则的模型适用于任何数据库应用程序。

  • 数据库对象可以属于多个领域，并且可以在多个领域中授权帐户或角色。

    要提供对数据库模式子集的有限访问（例如，仅HR模式中的EMPLOYEES表）或受领域保护的角色，请创建仅具有最少所需对象和授权的新领域。

  • 如果要将角色作为被授予者添加到领域，请创建一个领域来保护角色。这样做可以防止已被授予GRANT ANY ROLE系统特权的用户（例如SYSTEM用户帐户）将角色授予自己。

  • 如果要将SYS用户帐户添加到领域授权，则必须显式添加用户SYS而不是通过角色（例如DBA角色）添加。

  • 请注意当前允许您计划添加为领域授权的角色的权限。

    如果SYS或SYSTEM等帐户首次创建角色并且Oracle Database Vault管理员将此角色添加为领域授权，则可能会意外授予角色的领域授权，并且不明显。这是因为创建角色的帐户在创建时隐式授予该角色。

  • 有时您必须暂时放松对行政任务的领域保护。安装管理器（ DV_ADMIN或DV_OWNER ）登录，而不是禁用DV_OWNER ，将指定的帐户添加到域的授权帐户，并将授权规则设置为“已启用”。然后在启用的规则集中，启用规则集的所有审核。您可以在管理任务完成时删除领域授权。

  • 如果要向新用户授予ANY特权，Oracle建议您将数据库管理用户添加到Oracle系统特权和角色管理领域，以便此用户可以授予其他用户ANY特权（如果需要）。例如，使用命名帐户执行ANY操作的GRANT ，可以审核这些操作，从而为问责制创建审计跟踪。

  • 如果删除已受域保护的表，索引或角色，然后使用相同名称重新创建它，则不会还原域保护。您必须为新表，索引或角色重新创建领域保护。但是，您可以自动对指定模式中的所有未来表，索引和角色实施保护。例如，要对所有将来的表实施保护：

    BEGIN DBMS_MACADM.ADD_OBJECT_TO_REALM（' realm_name '，' schema_name '，'％'，'TABLE'）;结束; /

领域如何影响绩效

领域可以在各种情况下影响数据库性能，例如使用DDL和DML操作。

  • 受领域保护的对象上的DDL和DML操作对Oracle数据库没有可测量的影响。 Oracle建议您围绕整个架构创建领域，然后授权特定用户仅执行与其分配的任务相关的特定操作。对于更细粒度的控制，您可以定义各个表周围的域并授权用户对它们执行某些操作，并且还具有围绕整个架构的领域以保护整个应用程序。但请注意，此类配置可能会降低性能，但它确实允许您向架构中的某些对象授予域授权。

  • 审计会影响绩效。 为了获得最佳性能，Oracle建议您使用细粒度审计而不是审计所有操作。

  • 定期检查系统性能。 您可以通过运行Oracle Enterprise Manager（包括Oracle Enterprise Manager云控制，默认情况下随Oracle Database安装），自动工作负载存储库（AWR）和TKPROF 。
