8.189 V $ RSRC_PDB

V$RSRC_PDB通过可插拔数据库（PDB）显示与当前活动资源使用者组相关的数据。

当STATISTICS_LEVEL设置为TYPICAL或ALL ，此视图包含有关CPU利用率和等待时间的信息，即使未设置资源管理器计划或资源管理器计划不监视CPU或会话资源也是如此。

当多租户容器数据库（CDB）更改其资源计划时，将重置V$RSRC_PDB中的统计信息。它们不受PDB资源计划更改的影响。

V$RSRC_PDB涵盖所有PDB的相同时间段。此视图专门用于比较不同PDB之间的统计信息。

由于V$RSRC_PDB不包含消费者组级别的信息，因此对于比较PDB中的使用者组没有用。

6.14对象视图层次结构

对象视图层次结构是一组对象视图，每个对象视图基于类型层次结构中的不同类型。视图层次结构中的子视图是在超级视图下创建的，类似于在超类型下创建类型层次结构中的子类型的方式。

视图层次结构中的每个对象视图都填充有单个类型的对象，但是给定视图上的查询也隐式地查询其子视图。因此，对象视图层次结构为您提供了一种简单的方法来构建查询，这些查询可以返回给定特定级别或更高级别的多态对象集。

例如，假设您具有以下类型层次结构，其中person_typ为根：

图6-1对象类型层次结构


“图6-1对象类型层次结构”的说明

如果已基于此类型层次结构创建了对象视图层次结构，并且在每种类型上构建了对象视图，则可以查询与您感兴趣的特化级别相对应的对象视图。例如，您可以查询student_typ的视图以获取仅包含学生的结果集，包括兼职学生。

您可以将对象视图层次结构的根视图基于类型层次结构中的任何类型：您不需要以根类型启动对象视图层次结构。您也不需要将对象视图层次结构扩展到类型层次结构的每个叶子或覆盖每个分支。但是，您不能跳过下降线中的干预子类型。任何子视图必须基于其直接超级视图类型的直接子类型。

就像一个类型可以有多个兄弟子类型一样，对象视图可以有多个兄弟子视图。但是，基于给定类型的子视图只能参与一个对象视图层次结构：两个不同的对象视图层次结构不能各自具有基于相同子类型的子视图。

子视图从其超级视图继承对象标识符（OID）。无法在任何子视图中明确指定OID。

根视图可以使用WITH OBJECT ID子句显式指定对象标识符。如果OID是系统生成的，或者根视图中未指定子句，则只有在根视图基于同样使用系统生成的OID的表或视图时，才能创建子视图。

视图下的查询确定视图是否可更新。要使视图可更新，其查询必须不包含连接，集合运算符，聚合函数， GROUP BY子句， DISTINCT子句，伪列或表达式。这同样适用于子视图。

如果视图不可更新，则可以定义INSTEAD OF触发器以执行适当的DML操作。请注意，子视图不会继承INSTEAD OF触发器。

视图层次结构中的所有视图必须位于同一模式中。

注意：

您可以创建不可实例化的类型的视图。不可实例化的类型不能具有实例，因此通常在创建这种类型的对象视图时没有意义。但是，不可实例化的类型可以具有可实例化的子类型。通过创建不可实例化类型的对象视图，您可以将对象视图层次结构基于包含不可实例化类型的类型层次结构。

6.14.1创建对象视图层次结构

您可以通过在根视图下创建子视图来构建对象视图层次结构。您可以通过在CREATE VIEW语句中使用UNDER关键字来执行此操作，如示例6-17中所示 。

相同的对象视图层次结构可以基于不同的底层存储模型。换句话说，底层表的各种布局或设计可以产生相同的对象视图层次结构。底层存储模型的设计会影响对象视图层次结构的性能和可更新性。

描述了三种可能的存储模型。在第一个是平面模型，对象视图层次结构中的所有视图都基于同一个表。在第二个水平模型中，每个视图与不同的表一一对应。在第三个是垂直模型，视图是使用连接构建的。

要执行任何这些存储模型，请首先创建示例6-16中所示的类型。

示例6-16为存储模型示例创建类型

CREATE TYPE person_typ AS OBJECT（ssn NUMBER，名称VARCHAR2（30），地址VARCHAR2（100））NOT FINAL; / CREATE TYPE student_typ UNDER person_typ（deptid NUMBER，major VARCHAR2（30））NOT FINAL; / CREATE TYPE employee_typ UNDER person_typ（ empid NUMBER，mgr VARCHAR2（30））; /

PL / SQL基本块覆盖概述

DBMS_PLSQL_CODE_COVERAGE包使您可以在基本块级别收集数据。PL / SQL开发人员和测试工程师使用代码覆盖率测试结果作为其标准质量保证指标的一部分。

代码覆盖率是自动化测试所涵盖的代码百分比的度量。具有高代码覆盖率的程序比具有低代码覆盖率的程序具有更少的包含错误的机会。最重要的是测试套件执行的基本块的百分比。基本块是没有分支的线性代码段。基本块具有单个入口点（基本块中没有代码是跳转指令的目标）和单个出口点（只有最后一条指令或异常，可以将执行点移动到不同的基本块） 。通过目视检查代码无法预测基本块边界。编译器生成在运行时执行的块。

通过收集基本块级别的覆盖范围，可以准确地导出单位级别的覆盖率信息。可以生成实用程序来报告和可视化测试覆盖率结果，并帮助识别测试涵盖，部分覆盖或未涵盖的代码。

编写测试用例来运用一些基本块并不总是可行的。通过使用COVERAGE编译指示标记它们，可以从coverage计算中排除这些块。可以将源代码标记为不可覆盖单个基本块或一系列基本块。

A.1在云控制中使用IM列存储的会议先决条件

在启用数据库以使用IM列存储之前，请确保COMPATIBLE设置为12.1.0.0或更高版本。

要设置兼容级别，请按照下列步骤操作：

  1. 在Enterprise Manager的Database Home页面中，从Administration菜单中选择Initialization Parameters以导航到Initialization Parameters页面。

    您可以使用此页面设置或更改兼容级别。

  2. 搜索COMPATIBLE初始化参数。

    参数的类别是Miscellaneous。

  3. 将值更改为12.1.0.0 ，然后单击“ 应用” 。

    Cloud Control会提示您重新启动数据库。重新启动数据库后，新值将生效。

要设置或更改IM列存储的大小，请按照下列步骤操作：

  1. 在Enterprise Manager的Database Home页面中，从Administration菜单中选择Initialization Parameters以导航到Initialization Parameters页面。

  2. 搜索参数INMEMORY_SIZE 。参数的类别是In-Memory。

  3. 更改值，然后单击“ 应用” 。

    您可以将值设置为高于最小大小100 MB的任何值。

    然后，系统将提示您重新启动数据库。

A.2使用内存列存储中央主页监视数据库对象的内存支持

使用内存中列存储中心主页来监视对数据库对象（如表，索引，分区和表空间）的内存内支持。您可以查看对象的内存中功能并监视其内存中使用情况统计信息。

您可以在内存列存储中心主页上完成以下操作：

  • 内存中对象访问热图显示IM列存储中的前100个对象及其相对大小，并显示对象的访问频率，由不同颜色表示。要激活热图，必须在初始化参数文件中打开热图的选项。通常，在激活地图之前有一天的等待期。您可以使用日期选择器选择热图中显示的对象的日期范围。您还可以使用滑块控制颜色的粒度。

  • 使用“配置”部分可以查看状态设置，例如内存中查询，内存中强制和默认内存中子句。单击“编辑”以导航到“初始化参数”页面，您可以在其中更改此部分中显示的值和设置。使用“性能”部分查看“活动会话”的度量标准。

  • 使用“对象摘要”部分可以查看压缩因子和有关填充对象使用的内存的数据。通过从页面上的View In-Memory Enabled Object Statistics链接向下钻取，弹出窗口中提供了In-Memory Enabled Object Statistics。

  • 使用“内存中对象分发”部分可以按百分比查看内存中使用的各种对象的分布。该部分包括一个图表，显示分区，子分区，非分区表和非分区物化视图的分布。每个的数值显示在图表上方。

  • 使用“内存中对象搜索”部分搜索为内存使用指定的对象。输入要搜索的参数后，单击“搜索”。结果表显示了找到的每个对象的名称及其大小，内存大小，磁盘大小，内存中百分比及其内存中参数。您还可以搜索内存中或非内存中的访问对象。如果启用了热图，则“内存对象搜索”框的“视图”字段的下拉列表中将显示“访问的对象”选项。选择“访问的对象”时，可以根据前100个对象进行过滤，这些对象的访问数据可以是内存中的，也可以不是内存中的。您可以选择时间范围并搜索该范围内的对象。如果选择“所有内存中的对象”选项，则可以根据内存大小查看内存中前100个对象的列表。

如果您在Oracle RAC环境中工作，则可以通过在热图上方和右侧的实例选择框中选择实例来快速在实例之间移动。

  4. （可选）您可以单击“ 分区”选项卡。

    若要为分区指定IM列存储详细信息，请从“分区”选项卡中的表中选择它，然后单击“ 高级选项” 。

  6. 在表级别，列级别和分区级别输入所有必需的IM列存储详细信息后，单击“ 显示SQL”以查看生成的SQL。单击“ 确定”以创建表。

SQL * Plus概述

SQL * Plus是一个交互式批处理查询工具，随每个Oracle数据库安装一起安装。它有一个命令行用户界面。

还有SQL * Plus Instant Client，它是一个独立的命令行界面，可在支持OCI Instant Client的平台上使用。SQL * Plus Instant Client连接到任何可用的Oracle数据库，但不需要自己的Oracle数据库安装。有关OCI Instant Client的更多信息，请参阅Oracle Call Interface Programmer's Guide 。

SQL * Plus有自己的命令和环境，它提供对Oracle数据库的访问。它使您能够输入和执行SQL，PL / SQL，SQL * Plus和操作系统命令来执行以下操作：

  • 格式化，对查询结果执行计算，存储和打印

  • 检查表和对象定义

  • 开发和运行批处理脚本

  • 执行数据库管理

您可以使用SQL * Plus以交互方式生成报告，以批处理方式生成报告，并将结果输出到文本文件，屏幕或HTML文件以便在Internet上浏览。您可以使用SQL * Plus的HTML输出工具动态生成报告。

为网关创建SNAP-IX配置文件

Oracle Database Gateway for APPC需要一组存储的定义，称为侧信息配置文件，用于支持网关和网关服务器之间的连接。每个配置文件由配置文件名称和配置文件类型组成，配置文件类型是描述配置文件的一组字段。给定配置文件类型中的字段通常是操作参数值和与配置文件相关的其他SNA配置文件的名称的混合。APPC的每个功能部分，例如模式，远程事务程序名称和逻辑单元（LU），由不同的配置文件类型描述。

独立对抗的LU

网关配置可以适应独立或依赖逻辑单元。如果您选择使用从属LU，或仅限于使用从属LU，则网关可以正常运行。如果正确定义了从属LU，则无需更改网关的配置，也不需要对网关服务器进行任何更改。但是，Oracle建议您为网关使用独立的LU，因为它们支持多个并行会话或会话。这意味着多个Oracle客户端应用程序可以通过独立LU与同一网关服务器同时处于活动状态。

与独立LU相比，从属LU仅支持单个活动会话。CP（节点的控制点）将来自已经活动的对话后面的网关的每个附加会话请求排队。换句话说，对话LU是单线程的。

依赖LU的操作影响是第一个客户端应用程序可以通过网关与网关服务器发起对话，但是该会话处于活动状态（可能是几秒，几分钟或几小时，具体取决于客户端应用程序和事务的方式设计），启动与同一网关服务器的会话的任何其他客户端应用程序似乎挂起，因为它等待前一个会话。

如果生产应用程序在任何时候确实只使用一个对话，那么应该没有问题。但是，在某些时候，您可能需要额外的并发对话来进行测试或进行其他应用程序开发。进行多个会话需要在远程主机上定义其他从属LU。必须将其他配置条目添加到SNAP-IX。应定义附加的侧信息配置文件以使用新的从属LU。应创建网关实例并将其配置为使用这些新的辅助信息配置文件。

为网关创建SNA定义

SNAP-IX定义存储在以下文件中，这些文件位于/etc/opt/sna目录中：

  • SNA节点定义：sna_node.cfg

  • SNA域定义： sna_domn.cfg

这些文件是使用。创建和维护的 xsnaadmin工具。SNA定义的维护通常由具有管理权限的用户完成。以下信息适用于为网关创建SNA定义的用户。在阅读本节之前，您必须具备一些SNA知识。

样品SNAP-IX定义

$ORACLE_HOME/dg4appc/sna子目录包含一组用于网关的示例SNAP-IX定义文件，这些文件是使用 xsnaadmin 。这些示例文件是sna_domn.cfg和sna_node.cfg 。SNA定义非常特定于主机和SNA网络。因此，如果没有为本地主机和SNA网络定制，所提供的样本定义将无法工作。

配置SNAP-IX

本节介绍使用xsnaadmin为SNAP-IX创建SNA定义的过程。所有配置都使用xsnaadmin的各种下拉菜单和面板xsnaadmin 。以下配置说明遵循提供的示例。请为您的本地主机和SNA网络定制各种SNA值。

启动xsnaadmin

使用以下命令调用xsnaadmin 。必须正确设置DISPLAY环境变量。如果从本地控制台运行xsnaadmin ，则应已设置DISPLAY 。如果从远程X显示器运行xsnaadmin ，则将DISPLAY设置为该显示器的主机名或IP地址。 

$ DISPLAY = < your_display >：0 $ export DISPLAY $ xsnaadmin＆

在启动xsnaadmin ，将打开主屏幕并显示本地SNA节点的当前配置。

配置SNA节点

要配置SNA节点，您需要执行以下操作：

  1. 从“服务”菜单中，选择“ 配置节点参数” 。
  2. 在“节点参数”对话框中，根据需要输入APPN支持类型，控制点名称，控制点别名和节点ID。控制点名称由SNA网络名称和本地主机的CP名称组成。
  3. 单击确定 。
添加端口

要添加新端口，请从“服务”菜单中选择“ 连接”和“新端口” 。

  1. 在“添加到节点名”对话框中，选择“端口类型”，然后单击“ 确定” 。
  2. 在SAP对话框中，输入端口名称和网卡号。端口名称用于逻辑命名您正在使用的物理网卡，用于将服务访问端口绑定到SNA协议的卡。通常，您可以接受对话框中提供的值。但是，如果需要其他网卡，请输入dmesg命令报告的卡号。
  3. 单击确定 。
创建一个链接站

定义端口后，您必须创建链接站。Link Station表示网关服务器的远程主机的SNA节点。但在创建Link Station之前，必须按照以下过程中的说明创建远程节点定义：

  1. 从“服务”菜单中，选择“ APPC”和“添加远程节点” 。

  2. 在对话框中，输入远程节点的SNA CPNAME，然后单击“ 确定” 。

现在您可以按如下方式创建Link Station：

  1. 从“服务”菜单中，选择“ 连接”和“新建链接站” 。在对话框中，选择先前定义的端口，然后单击“ 确定” 。
  2. 在“链接站”对话框中，输入“链接站”的名称，选择“SNA端口名称”，“链接激活类型”和“LU流量类型”。要获得最大的灵活性，请选择任意选项。
  3. 对于独立LU流量，请指定远程节点名称。单击“ 远程节点”并选择先前创建的节点，然后单击“ 确定” 。选择“远程”节点的类型，通常是“网络”节点。
  4. 对于从属LU流量，请指定远程节点的角色，通常为“主机”，本地节点ID和可选的远程节点ID。
  5. 指定联系信息。联系信息包含远程主机的MAC地址以及SAP编号。
  6. 单击“ 高级”以获取Link Station的其他参数。令牌环参数对话框显示链接站的其他参数。这些参数会影响初始XID联系和重新传输时间和限制。您通常不需要更改默认值。
  7. 单击确定 。
创建本地LU

完成远程节点定义后，为本地主机创建本地LU名称，如下所示：

  1. 从“服务”菜单中，选择“ APPC”和“新建本地LU” 。在本地LU对话框中，输入本地LU的名称和别名。此名称必须与UNIX主机的远程网关服务器主机上的VTAM定义相对应。
  2. 单击确定 。
创建合作伙伴LU

现在定义一个Partner LU，它代表网关服务器用于通信的LU。

  1. 从“服务”菜单中，选择“远程节点上的 APPC和新伙伴LU和伙伴LU” 。
  2. 在“合作伙伴LU”对话框中，输入合作伙伴LU名称和特征。合作伙伴LU名称包含SNA网络名称以及远程LU的LU名称。启用并行会话支持。该位置是远程节点名称的名称。您可以单击“ 位置 ”以获取列表。
  3. 单击确定 。
创建模式和CPI-C配置文件

在创建了本地和远程LU定义后，创建必要的模式和CPI-C定义。

  1. 从“服务”菜单中，选择“ APPC和模式” 。在“模式”对话框中，单击“ 新建”以添加新模式。
  2. 在“模式”对话框中，输入“模式名称”和其他会话参数。网关模式的推荐名称是CICSPGA 。请与Remote Host系统管理员联系以获取适当的模式参数。
  3. 单击确定 。
  4. 现在已经定义了模式，创建CPI-C侧信息配置文件，网关将其用作连接名称。从菜单中选择APPC和CPI-C 。
  5. 在“CPI-C目标名称”对话框中，单击“ 新建”以添加新的“配置文件”。
  6. 在CPI-C目标对话框中，输入配置文件名称，本地LU名称，伙伴TP，伙伴LU和模式以及安全选项。伙伴TP名称是主机事务程序的名称或要在TIP中覆盖的虚拟值。
  7. 对于本地LU，您可以指定特定的LU或选择默认的LU。对于合作伙伴LU，请输入完整的LU名称或先前创建的别名。
  8. 输入ORAPLU62作为模式名称。选择要使用的这些会话的安全类型。这会影响会话授权的完成方式。
  9. 单击确定 。

2.7目录

目的

使用CATALOG命令执行以下操作：

  • 在RMAN存储库的磁盘上添加备份片和映像副本

  • 将数据文件副本记录为RMAN存储库中的0级增量备份，这使您可以将其用作增量备份策略的一部分

也可以看看：

Oracle数据库备份和恢复用户指南 ，了解如何管理存储在目录中的目标数据库记录

先决条件

您必须连接到目标数据库，该数据库必须已安装或打开。如果RMAN连接到恢复目录，则必须打开目录数据库。

您正在编目的文件必须满足以下条件：

  • 它必须不存在于SBT设备上。

  • 如果它是用户管理的副本，则它必须是数据文件副本，控制文件副本，归档重做日志或备份片。

使用说明

RMAN将所有用户管理的备份视为映像副本。编目时，RMAN不检查文件是否被操作系统实用程序正确复制：它只检查标题。

在Data Guard环境中使用RMAN时，需要恢复目录。恢复目录支持具有相同DBID但DB_UNIQUE_NAME值不同的所有主数据库和备用数据库的统一文件命名空间。因此，恢复目录会跟踪所有主数据库和备用数据库的数据库文件名，以及创建联机重做日志，备用重做日志，临时文件，归档重做日志文件，备份集和映像副本的位置。

“ Data Guard环境中的RMAN备份 ”解释了RMAN如何处理在不同主数据库和备用数据库上进行的备份。通常，在一个数据库上进行的磁带备份可由环境中的任何数据库访问，而磁盘备份只能由创建它们的数据库访问。

如果连接的目标数据库可以访问备份，则RESTORE和RECOVER等RMAN命令在不同的数据库中透明运行。您可以手动将磁盘备份从环境中的一个主机传输到另一个主机，然后对备份进行编目。如果备份位于共享磁盘上，则可以使用CHANGE RESET DB_UNIQUE_NAME将备份与新数据库关联。

BACKUPPIECE ' filename '

指定要添加到RMAN存储库的备份片段的名称（请参阅示例2-40 ）。

备份必须在磁盘上。RMAN在对其进行编目之前验证备份件头。RMAN可以编译先前数据库化身的备份片段。

您可以选择在以下情况下编制备份片段：

  • 您使用操作系统实用程序复制或移动备份片段，并希望RMAN可以使用它。

  • 已删除备份片的RMAN元数据，但备份片仍然存在。如果您在暂时不可用的备份片上运行DELETE命令，则会发生这种情况。

  • 您在Data Guard环境中的一个数据库主机上进行NOCATALOG备份，并将备份片移动到另一个数据库主机上的相同位置。在这种情况下，恢复目录没有原始备份的记录。

  • 您不使用恢复目录，必须重新创建控制文件，从而丢失所有RMAN存储库数据。对备份进行编目使它们再次可用。

  • 禁用控制文件自动备份时，备份控制文件，然后备份存档的重做日志文件。您可以还原和装入控制文件，但必须对包含在控制文件后备份的归档重做日志文件的备份片段进行编目。

如果指定备份片段列表，则RMAN会尝试对给定列表中的所有片段进行编目，即使其中一些片段失败也是如此。编目备份片段会在V$BACKUP_PIECE创建一个新行。备份集仅在所有备份片段编目时可用，否则它仅处于部分可用状态。

注意：如果RMAN在数据库的COMPATIBLE参数设置为11.0.0或更高版本时创建服务器参数文件备份，则备份将与此数据库关联。在这种情况下，即使您将RMAN连接到另一个数据库并显式编目备份片，也可以与此备份关联的DB_UNIQUE_NAME不会更改。例如，如果当COMPATIBLE为11.0.0时，如果RMAN使用DB_UNIQUE_NAME 'NEWYORK'备份数据库的服务器参数文件，则RMAN无法使用在数据库NEWYORK创建的服务器参数文件备份来还原数据库BOSTON上的服务器参数文件。

指定要添加到RMAN存储库的控制文件副本的文件名。控制文件副本可以是由以下命令之一创建的普通或备用控制文件副本：

  • SQL： ALTER DATABASE BACKUP CONTROLFILE

注意： RMAN可以在还原操作期间自动将主数据库控制文件备份转换为备用控制文件。

您可以使用此数据文件副本作为基本级别0备份来执行增量备份。

指定数据文件副本的标记。

START WITH ' string_pattern '

安装Oracle Standalone Cluster

完成此过程以安装Oracle Standalone Cluster的Oracle Grid Infrastructure软件。

  1. 作为grid用户，下载Oracle Grid Infrastructure图像文件并将文件解压缩到Grid主目录。例如：
    $ mkdir -p /u01/app/19.0.0/grid $ chown grid：oinstall /u01/app/19.0.0/grid $ cd /u01/app/19.0.0/grid $ unzip -q download_location/grid.zip

    grid.zip是Oracle Grid Infrastructure映像zip文件的名称。

    注意：

    • 您必须将zip映像软件解压缩到您希望Grid主页所在的目录中。

    • 仅将Oracle Grid Infrastructure映像文件下载并复制到本地节点。在安装过程中，软件将复制并安装在群集中的所有其他节点上。

  2. 配置共享磁盘以与Oracle ASM筛选器驱动程序一起使用：
    1. 以root用户身份登录，并将环境变量ORACLE_HOME设置为Grid home的位置。

      对于C shell：

      $ su root #setenv ORACLE_HOME /u01/app/19.0.0/grid

      对于bash shell：

      $ su root #export ORACLE_HOME = / u01 / app / 19.0.0 / grid
    2. 使用Oracle ASM命令行工具（ASMCMD）配置磁盘设备以与Oracle ASM筛选器驱动程序一起使用。
      #cd /u01/app/19.0.0/grid/bin#./asmcmd afd_label DATA1 / dev / sdb --init＃。/ asmcmd afd_label DATA2 / dev / sdc --init＃。/ asmcmd afd_label DATA3 / dev / sdd - 在里面
    3. 验证设备是否已标记为与Oracle ASMFD一起使用。
      ＃。/ asmcmd afd_lslbl / dev / sdb＃。/ asmcmd afd_lslbl / dev / sdc＃。/ asmcmd afd_lslbl / dev / sdd
  3. 以grid用户身份登录，并通过运行以下命令启动Oracle Grid Infrastructure安装程序：
    $ /u01/app/19.0.0/grid/gridSetup.sh
    安装程序启动，出现“选择配置选项”窗口。
  4. 选择“ 为新群集配置网格基础结构 ”选项，然后单击“ 下一步” 。
    将出现“选择群集配置”窗口。
  5. 选择“ 配置Oracle独立群集 ”选项，然后单击“ 下一步” 。
    选择“ 配置为扩展群集”选项可将Oracle RAC群集扩展到两个或多个单独的站点，每个站点都配有自己的存储。
    将出现“网格即插即用信息”窗口。
  6. 在“ 群集名称”和“ 扫描名称”字段中，输入在整个企业网络中唯一的群集和群集扫描的名称。

    如果已将域名服务器（DNS）配置为发送到子域GNS服务的GNS虚拟IP地址名称解析请求，则可以选择配置GNS ，如本指南中所述。

    对于集群成员节点公用和VIP网络地址，根据要配置的集群类型提供所需的信息：

    • 如果您计划使用通过GNS配置和解析的DHCP地址的自动群集配置，则您只需提供DNS上配置的GNS VIP名称。

    • 如果计划使用手动群集配置，并在DNS上配置并解析固定IP地址，则提供群集的SCAN名称，以及每个群集成员节点的公用名称和VIP名称。例如，您可以选择基于节点名称的公共前缀的名称。群集名称可以是mycluster ，群集SCAN名称可以是mycluster-scan 。

    单击下一步 。
    将出现“群集节点信息”窗口。
  7. 在群集节点表的“公共主机名”列中，您应该看到本地节点，例如node1.example.com 。

    以下是有关节点IP地址的其他信息的列表：

    • 仅对于本地节点，OUI会自动填写公共和VIP字段。如果您的系统使用供应商集群件，则OUI可能会填充其他字段。

    • 主机名和虚拟主机名不是域限定的。如果在安装期间在地址字段中提供域，则OUI将从该地址中删除该域。

    • 标识为专用于私有IP地址的接口不应作为公共接口访问。使用Cache Fusion的公共接口可能会导致性能问题。

    • 输入公共节点名称时，请使用每个节点的主要主机名。换句话说，使用/bin/hostname命令显示的名称。

    1. 单击“ 添加”以将另一个节点添加到群集。
    2. 输入第二个节点的公共名称（ node2 ）和虚拟IP名称（ node2-vip ），然后单击“ 确定” 。为所有群集节点提供虚拟IP（VIP）主机名，或者不提供。
      您将返回“群集节点信息”窗口。您现在应该看到群集节点表中列出的所有节点。
    3. 确保选中所有节点，然后单击窗口底部的SSH连接按钮。
      窗口的底部面板显示SSH连接信息。
    4. 输入Oracle软件所有者（ grid ）的操作系统用户名和密码。如果已在节点之间配置SSH连接，则选择“ 重用用户主目录中存在的私钥和公钥”选项。单击设置 。
      将出现一个消息窗口，指示可能需要几分钟才能在节点之间配置SSH连接。在短暂的一段时间后，将出现另一个消息窗口，指示已在群集节点之间建立无密码SSH连接。单击“ 确定”继续。
    5. 返回“群集节点信息”窗口时，单击“ 下一步”继续。
    将出现“指定网络接口使用”窗口。
  8. 选择显示的每个网络接口的使用类型。
    验证每个接口是否具有与之关联的正确接口类型。如果您具有Oracle Clusterware不应使用的网络接口，请将网络接口类型设置为“不使用” 。例如，如果您只有两个网络接口，则将公共接口设置为使用值为Public ，并将专用网络接口设置为使用ASM和私有值。
    单击下一步 。将出现“存储选项信息”窗口。
  9. 选择Oracle Cluster Registry（OCR）和投票文件的存储选项：
    1. 选择“ 使用Oracle Flex ASM存储”以在Oracle ASM磁盘组上存储OCR和表决文件。
    2. 选择“ 使用共享文件系统”将OCR和投票文件存储在共享文件系统上，然后单击“ 下一步” 。将出现“创建网格基础结构管理存储库选项”窗口
  10. 选择是否要为Oracle Standalone Cluster安装创建Grid Infrastructure Management存储库，然后单击“ 下一步” 。
    如果在此窗口中选择“ 是 ”，则会显示“网格基础结构管理存储库选项”窗口。否则，将显示“创建ASM磁盘组”窗口。
  11. 选择是否要将Grid Infrastructure Management存储库存储在单独的Oracle ASM磁盘组中，然后单击“ 下一步” 。
    将出现“创建ASM磁盘组”窗口。
  12. 提供Oracle ASM磁盘组的名称和规范。
    1. 在“ 磁盘组名称”字段中，输入磁盘组的名称 ，例如DATA。
    2. 选择此磁盘组的冗余级别。建议选择正常。
    3. 在“添加磁盘”部分中，选择要添加到此磁盘组的磁盘。

      在“添加磁盘”部分中，您应该看到在步骤2中标记的磁盘。如果没有看到磁盘，请单击“ 更改发现路径”按钮并提供磁盘的路径和模式匹配，例如/dev/sd*

      在安装期间，使用默认发现字符串时，标记为Oracle ASMFD磁盘或Oracle ASMLIB磁盘的磁盘将列为候选磁盘。但是，如果磁盘的标头状态为MEMBER，则它不是候选磁盘。

    4. 如果要使用Oracle ASM筛选器驱动程序（Oracle ASMFD）来管理Oracle ASM磁盘设备，请选择“ 配置Oracle ASM筛选器驱动程序 ”选项。
      如果要在Linux系统上进行安装，并且希望使用Oracle ASM筛选器驱动程序（Oracle ASMFD）来管理Oracle ASM磁盘设备，则必须在启动Oracle Grid Infrastructure安装之前卸载Oracle ASM库驱动程序（Oracle ASMLIB）。
    完成为磁盘组提供信息后，单击“ 下一步” 。
  13. 如果选择为GIMR使用其他磁盘组，则会显示“网格基础结构管理存储库选项”窗口。提供GIMR磁盘组的名称和规格。
    1. 在“ 磁盘组名称”字段中，输入磁盘组的名称 ，例如DATA。
    2. 选择此磁盘组的冗余级别。建议选择正常。
    3. 在“添加磁盘”部分中，选择要添加到此磁盘组的磁盘。
    完成为磁盘组提供信息后，单击“ 下一步” 。
    将出现“指定ASM密码”窗口。
  14. 为Oracle ASM SYS和ASMSNMP帐户选择相同的密码，或为每个帐户指定不同的密码，然后单击“ 下一步” 。
    出现“故障隔离支持”窗口。
  15. 选择“不使用智能平台管理界面（IPMI）”选项 ，然后单击“ 下一步” 。
    将出现“指定管理选项”窗口。
  16. 如果您的企业中安装了Enterprise Manager Cloud Control，请选择“ 注册企业管理器（EM）云控制 ”选项并提供EM配置信息。如果您的企业中未安装Enterprise Manager Cloud Control，请单击“ 下一步”继续。
    将显示“特权操作系统组”窗口。
  17. 接受Oracle ASM管理的默认操作系统组名称，然后单击“ 下一步” 。
    将出现“指定安装位置”窗口。
  18. 指定用于Oracle Grid Infrastructure安装的Oracle基础的目录，然后单击“ 下一步” 。Oracle基本目录必须与Oracle主目录不同。
    如果您按照步骤1中的说明将Oracle Grid Infrastructure安装文件复制到Oracle Grid主目录中，则Oracle基本目录的默认位置应显示为/u01/app/grid 。
    如果以前未在此计算机上安装Oracle软件，则会显示“创建库存”窗口。
  19. 如果需要，更改库存目录的路径。然后单击“ 下一步” 。
    如果您使用与本书中的示例相同的目录名称，则它应显示/u01/app/oraInventory 。oraInventory目录的组名应显示oinstall 。
    将出现“根脚本执行配置”窗口。
  20. 选择“ 自动运行配置脚本 ”选项。 输入root用户或sudo帐户的凭据，然后单击“ 下一步” 。
    或者，您可以在安装程序提示时在安装过程结束时以root用户身份手动运行脚本。
    将出现“执行先决条件检查”窗口。
  21. 如果任何检查的状态为“失败”且不可修复，则必须手动更正这些问题。解决问题后，可以单击“ 再次检查”按钮让安装程序重新检查要求并更新状态。根据需要重复，直到所有检查的状态为Succeeded。单击下一步 。

    将出现“摘要”窗口。

      [grid @ node1 grid] $ cd / u01 / app / oraInventory [grid @ node1 oraInventory] $ su

      [root @ node1 oraInventory]＃。/ orainstRoot.sh

      [grid @ node1 grid] $ ssh node2 [grid @ node2 grid] $ cd / u01 / app / oraInventory [grid @ node2 oraInventory] $ su

      [root @ node2 oraInventory]＃。/ orainstRoot.sh

      [root @ node1 oraInventory] #cd /u01/app/19.0.0/grid [root @ node1 grid]＃。/ root.sh

      在提示符处按Enter键以接受默认值。

      注意：

      您必须在第一个节点上运行root.sh脚本并等待它完成。您可以在除运行脚本的最后一个节点之外的所有其他节点上同时运行root.sh脚本。与第一个节点一样，最后一个节点上的root.sh脚本必须单独运行。

    6. root.sh脚本在node1完成后，转到在此步骤的c部分中打开的终端窗口。作为node2上的root用户，输入以下命令：

      [root @ node2 oraInventory] #cd /u01/app/19.0.0/grid [root @ node2 grid]＃。/ root.sh

支持Oracle Clusterware升级的干运行验证

从Oracle Grid Infrastructure 19c开始，Oracle Grid Infrastructure安装向导（ gridSetup.sh ）使您能够执行干运行模式升级以检查系统的升级准备情况。

多个ASMB

鉴于+ ASM1已安装dg1但未安装dg2，并且+ ASM2已安装dg2但未安装dg1，则多个ASMB项目允许数据库通过同时连接到两个ASM实例来同时使用dg1和dg2。我们现在可以拥有ASMBn，而不仅仅是ASMB。

此功能允许DB使用多个磁盘组，即使给定的ASM实例没有安装所有磁盘组，也可以提高Real Application Clusters（RAC）堆栈的可用性。

奇偶校验保护文件

REDUNDANCY文件类型属性指定文件组的冗余。PARITY值指定冗余的单奇偶校验。奇偶校验设置适用于一次写入文件，例如存档日志和备份集。

当两个或三个ASM镜像用于与数据库备份操作相关联的文件时，会消耗大量空间。备份文件是一次写入文件，此功能允许保护奇偶校验而不是传统镜像。结果可节省大量空间。

安全集群通信

当与单一网络支持一起使用时，安全群集通信可保护群集互连免受常见安全威胁。安全群集通信包括消息摘要机制，防止模糊测试，并使用传输层安全性（TLS）在群集成员之间提供隐私和数据完整性。

零停机Oracle Grid Infrastructure补丁

零停机Oracle Grid Infrastructure Patching可以在不中断数据库操作的情况下修补Oracle Grid Infrastructure。修补程序以非滚动方式应用，一次修补一个节点，同时节点上的数据库实例保持运行。零停机Oracle Grid Infrastructure Patching支持具有两个或更多节点的集群上的Oracle Real Application Clusters（Oracle RAC）数据库。

零停机时间网格基础结构修补通过允许客户执行Oracle Grid Infrastructure的滚动补丁而不中断正在修补的节点上的数据库操作而不影响这些数据库实例的容量或性能，显着提高了数据库可用性。

1.2关于在Oracle数据库中使用Java

您可以在数据库中编写和加载Java应用程序，因为它是一种具有许多安全功能的安全语言。Java的开发是为了防止任何人篡改Java代码所在的操作系统。某些语言（如C）可能会在数据库中引入安全问题。但是，由于Java的设计，Java是一种可以在数据库中使用的健壮语言。

虽然Java语言为开发人员带来了许多好处，但提供以可伸缩方式支持Java服务器应用程序的JVM实现是一项挑战。本节讨论以下挑战：

1.2.1 Java和RDBMS：一种强大的组合

Oracle数据库为Java应用程序提供了动态数据处理引擎，该引擎支持复杂查询和相同数据的不同视图。所有客户端请求都被组合为数据查询以便立即处理，并且动态生成查询结果。

Java和Oracle数据库的结合可帮助您创建基于组件的，以网络为中心的应用程序，这些应用程序可随业务需求的变化而轻松更新。此外，您还可以将应用程序和数据存储从桌面移到智能网络和以网络为中心的服务器上。更重要的是，您可以从任何客户端设备访问这些应用程序和数据存储。

1.2.2关于多线程

多线程是Java语言的关键可伸缩性功能之一。Java语言和类库使得用Java编写多线程应用程序比使用许多其他语言更简单，但是编写可靠，可伸缩的多线程代码在任何语言中仍然是一项艰巨的任务。

Oracle数据库服务器可以有效地为数千名用户安排工作。Oracle JVM利用Oracle数据库的会话体系结构，为数百到数千个用户同时运行Java应用程序。虽然Oracle数据库支持JLS和JCK所需的Java语言级线程，但是使用数据库范围内的线程不会增加可伸缩性。通过使用数据库的嵌入式可伸缩性，消除了编写多线程Java服务器的需要。

您应该使用Oracle数据库的工具通过编写单线程Java应用程序来安排用户。数据库可以在每个应用程序之间调度进程，因此，您无需管理线程即可实现可伸缩性。您仍然可以编写多线程Java应用程序，但多个Java线程不会提高服务器的性能。

多线程创建的一个复杂因素是线程与自动存储管理或垃圾收集的交互。在通用JVM中运行的垃圾收集器不知道正在运行哪些Java语言线程或底层操作系统如何调度它们。非Oracle数据库模型和Oracle JVM模型之间的区别如下：

  • 非Oracle数据库模型

    单个用户映射到单个Java线程，单个垃圾收集器管理来自所有用户的所有垃圾。不同的技术通常涉及不同寿命和大小的对象的分配和收集。大量多线程应用程序的结果充其量取决于对本机线程的操作系统支持，这可能是不可靠的并且在可伸缩性方面受到限制。这种实现的高度可扩展性尚未得到令人信服的证明。

  • Oracle JVM模型

    即使成千上万的用户连接到服务器并运行相同的Java代码，每个用户也会体验它，就像他或她在自己的JVM上运行自己的Java代码一样。 Oracle JVM的职责是利用操作系统进程和线程以及Oracle数据库的可扩展方法。作为这种方法的结果，Oracle JVM的垃圾收集器更加可靠和高效，因为它永远不会从多个用户收集垃圾。

1.2.3记忆空间管理

垃圾收集是Java自动存储管理功能的主要功能，无需Java开发人员明确分配和释放内存。因此，这消除了C和C ++程序中常见的大量内存泄漏。但是，垃圾收集会导致程序执行速度和占用空间的开销。

垃圾收集给寻求提供高度可扩展和快速Java平台的JVM开发人员带来了挑战。Oracle JVM通过以下方式应对这些挑战：

  • Oracle JVM使用Oracle数据库调度工具，可以有效地管理多个用户。

  • 对多个用户一致地执行垃圾收集，因为垃圾收集集中在单个会话中的单个用户。Oracle JVM具有优势，因为内存管理器的工作负担和复杂性不会随着用户数量的增加而增加。内存管理器在单个会话中执行对象的分配和收集，这通常转换为单个用户的活动。

  • Oracle JVM根据使用的内存类型使用不同的垃圾收集技术。这些技术提供高效率和低开销。

两种类型的存储空间是调用空间和会话空间。

记忆空间 描述

呼叫空间

它是一种快速且廉价的存储器。它主要存在于呼叫的长度。呼叫存储空间分为新旧段。所有新对象都在新内存中创建。在几次清除中幸存下来的物体被移入旧记忆中。

会话空间

这是一种昂贵的，性能方面的记忆。它主要存在于会话的长度。此处存在所有static字段以及在调用生命周期之外存在的任何对象。

Oracle JVM中的垃圾收集算法遵循以下规则：

  1. 在新的调用空间中创建新对象。

  2. 清除以设定的间隔发生。一些程序员经常在短时间内创建对象。在新的调用空间内快速创建和垃圾收集这些类型的对象。这被称为清除 。

  3. 在几次扫气迭代中幸存的任何对象都被认为是可以存在一段时间的对象。这些对象从新的调用空间移出到旧的调用空间。搬家期间，他们也被压缩了。旧的呼叫空间被清除或垃圾收集较少，因此提供了更好的性能。

  4. 在调用结束时，除调用之外存在的任何对象都将移动到会话空间中。

图1-6说明了前面文本中列出的步骤。这种方法适用于针对对象的类型和生命周期调整的复杂分配和收集方案。例如，新对象在快速且廉价的呼叫存储器中分配，用于快速分配和访问。Java static字段中保存的对象将迁移到更珍贵和昂贵的会话空间。

1.2.4足迹

正在运行的Java程序的占用空间受许多因素的影响：

  • 程序的大小

    程序的大小取决于类和方法的数量以及它们包含的代码量。

  • 程序的复杂性

    程序的复杂性取决于程序运行时Oracle JVM使用的核心类库的数量，而不是程序本身。

  • JVM使用的空间量

    JVM使用的空间量取决于JVM分配的对象数，这些对象的大小以及调用期间必须保留的对象数。

  • 垃圾收集器和内存管理器能够处理程序运行的需求

    这不能经常确定。分配对象的速度以及它们被其他对象保持的方式会影响该因素的重要性。

从可伸缩性的角度来看，同时支持多个客户端的关键是每个用户的最小会话占用空间。通过将用户的所有只读数据（例如Java字节码）放在共享内存中，Oracle JVM将每用户会话占用空间降至最低。针对呼叫和会话存储器应用适当的垃圾收集算法以维持用户会话的小占用空间。Oracle JVM使用以下类型的垃圾收集算法来维护用户的会话内存：

  • 对短命物体的代际清除

  • 标记和延迟扫描集合，用于在单个调用的生命周期中存在的对象

  • 为长期对象复制收集器，即在会话中跨越调用的对象

1.2.5 Oracle JVM的性能

该通过在标准JVM上嵌入类似于HotSpot的创新型Just-In-Time编译器，可以增强Oracle JVM的性能。独立于平台的Java字节码在JVM之上运行，JVM与特定硬件平台交互。无论何时在软件中添加级别，性能都会降低。由于Java需要通过中介来解释字节码，因此与使用平台相关语言（如C）开发的应用程序相比，Java应用程序存在一定程度的低效率。为了解决此问题，一些JVM供应商创建了本机编译器。本机编译器将Java字节码转换为依赖于平台的本机代码，从而消除了解释器步骤并提高了性能。

下表描述了两种本机编译方法：

编译器 描述

即时（JIT）编译

JIT编译器在运行时快速将Java字节码编译为特定于平台或本机的机器代码。这些编译器不会生成要在平台上运行的可执行文件。相反，它们提供来自Java字节码的平台相关代码，这些代码在翻译后直接运行。JIT编译器应该用于经常运行且速度更接近其他语言（如C语言）开发的代码的Java代码。

Oracle数据库使用实时（JIT）编译以本机编译的形式提供其核心Java类库，例如JDBC代码。JIT编译器在没有任何插件支持的情况下启用，并且适用于Oracle支持的所有平台。

Java的另一个强大功能是动态类加载。类加载器从磁盘加载类并将它们放在解释所需的特定于JVM的内存结构中。类加载器在CLASSPATH定位类，并仅在程序运行时使用它们时才加载它们。这种方法适用于applet，在服务器环境中存在以下问题：

问题 描述 解

可预测性

当程序第一次运行时，类加载操作会造成严重的惩罚。一个简单的程序可以使Oracle JVM加载许多核心类以支持其需求。程序员不能轻易预测或确定加载的类数。

Oracle JVM动态加载类，就像使用任何其他JVM一样。遇到相同的一次性加载速度命中。但是，因为类被加载到共享内存中，所以这些类的其他用户不会再次加载类，并且它们将使用相同的预加载类。

可靠性

动态类加载的一个好处是它支持程序更新。例如，您将更新服务器上的类，下载程序并动态加载它的客户端会在下次使用该程序时查看更新。服务器程序倾向于强调可靠性。作为开发人员，您必须知道每个客户端都运行特定的程序配置。您不希望客户端无意中加载您不打算加载的某些类。

Oracle数据库在运行时将上载和解析操作与类加载操作分开。您使用loadjava工具将开发的Java代码上载到服务器。您可以在安装时指定解析程序，而不是使用CLASSPATH 。解析器类似于CLASSPATH ，但允许您指定类所在的模式。这种分辨率与类加载的分离可确保您始终了解用户运行的程序。

发布 - 订阅模型简介

由于数据库是企业中最重要的信息资源，因此Oracle为企业信息传递和消息传递创建了一个发布 - 订阅解决方案，以补充此角色。

网络技术和产品可在大量计算机，应用程序和用户之间实现高度连接。在这些环境中，为分布式系统提供异步通信非常重要，这些分布式系统以松散耦合和自治的方式运行，并且需要操作免受网络故障的影响。此要求由各种中间件产品填充，这些产品的特征在于消息传递，面向消息的中间件（MOM），消息队列或发布 - 订阅。

通过发布和订阅范例进行通信的应用程序要求发送应用程序（发布者）在不明确指定收件人或了解预期收件人的情况下发布邮件。类似地，接收应用程序（订户）必须仅接收订户已注册感兴趣的那些消息。

发送者和接收者之间的这种分离通常由发布者和订阅者之间的中间实体来完成，该实体用作间接级别。该中间实体是表示主题或频道的队列。图23-1说明了发布和订阅功能。

发布 - 订阅概念

队列

队列是支持感兴趣的命名主题概念的实体。队列可以表征为持久性或非持久性（轻量级）。

持久队列充当消息的持久容器。消息以延迟和可靠的模式传递。

非持久或轻量级队列的底层基础结构以轻量级，最佳一次方式将发布的消息推送到连接的客户端。

代理人

发布者和订阅者在内部表示为代理。

代理是持久性逻辑订阅实体，通过订阅表达对队列的兴趣。代理具有属性，例如关联的订阅，地址和消息的传递模式。在此上下文中，代理是发布者或订阅者的电子代理。

客户

客户端是瞬态物理实体。客户端的属性包括客户端程序运行的物理进程，节点名称和客户端应用程序逻辑。多个客户可以代表单个代理进行操作。如果获得授权，同一客户可以代表多个代理商行事。

统治队列

使用消息格式属性或消息头属性上的预定义运算符集将队列规则指定为条件表达式。每个队列都有一个关联的消息内容格式，用于描述该队列所表示的消息的结构。消息格式可能是非结构化的（ RAW ），也可能是定义良好的结构（ADT）。这允许基于主题或内容的订阅。

订户

订户（代理）可以使用规则在队列上指定订阅。订户是持久的并存储在目录中。

数据库事件发布框架

数据库是发布信息的重要来源。提出了一个事件框架，以允许数据库事件发布的声明性定义。当这些预定义事件发生时，框架会检测并发布此类事件。这允许以事件驱动的方式将信息主动传递给最终用户，作为发布 - 订阅功能的一部分。

注册

注册是指代客户代表代理人的相关交付信息的过程。与代理/客户端分离相关的订阅和注册之间存在重要区别。

订阅表示代理对特定队列的兴趣。它没有说明必须在何处以及如何进行交付。传递信息是与客户端相关联的物理属性，它是逻辑代理（订户）的瞬时表现形式。代表代理的特定客户端进程通过关联主机和端口来指示传递信息，指示传递将在何处进行，以及回调，指示如何进行传递。

发布消息

发布者使用适当的排队接口将消息发布到队列。接口可能取决于实现队列的模型。例如，enqueue调用表示发布消息。

规则引擎

当消息发布或发布到给定队列时，规则引擎从该队列上定义的与发布消息匹配的所有规则中提取候选规则集。

订阅服务

对应于给定队列上的候选规则列表，可以评估与候选规则匹配的订户组。反过来，可以确定并通知与该订阅列表相对应的代理集。

发帖

该队列通知所有已注册的客户端相应的已发布消息。这个概念叫做发布 。当队列必须通知所有感兴趣的客户端时，它会将消息发布到所有已注册的客户端。

收到一条消息

订阅者可以通过以下任何机制接收消息：

  • 代表订户执行的客户端进程使用注册机制指定回调。当消息与订阅者的订阅匹配时，发布机制然后异步调用回调。消息内容可以传递给回调函数（仅限非持久队列）。

  • 代表订户执行的客户端进程使用注册机制指定回调。然后，发布机制异步调用回调函数，但没有完整的消息内容。这用作客户端的通知，客户端随后以拉取方式检索消息内容（仅限持久队列）。

  • 代表订户行动的客户端进程仅以定期或其他适当的方式从队列中检索消息。虽然消息被延迟，但是没有异步传递到最终客户端。
