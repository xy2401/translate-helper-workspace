提交确认概述

注意：

如果您计划实施commit-confirm，那么您应该已经配置了组件。根据您的平台，请参阅适用于POWER系统（64位）上的IBM AIX的Oracle数据库网关APPC安装和配置指南的第12章，Linux x86-64，SPARC上的Oracle Solaris（64位）和HP- UX Itanium或适用于Microsoft Windows的APPC安装和配置指南的Oracle数据库网关的第9章，了解其配置。

Commit-confirm是两阶段提交的特殊实现，它允许不支持完全两阶段提交的数据库或网关与支持完全两阶段提交的其他数据库或网关一起参与分布式更新事务。在此实现中，在准备好所有其他站点之后，commit-confirm站点始终是第一个提交的站点。这允许所有站点保持同步，因为如果提交确认站点无法成功提交，则可以回滚所有其他站点。

在Oracle分布式事务中，与该事务关联的所有工作都分配了一个公共标识符，称为 Oracle全球交易ID。该标识符保证是唯一的，因此它可以用于专门标识特定的分布式事务。提交确认支持的关键要求是提交确认站点（在本例中为APPC的Oracle数据库网关）能够将Oracle全局事务ID作为其工作单元的一部分进行记录，以便如果发生故障，网关的恢复处理可以通过该事务的日志条目的存在与否来确定特定Oracle全局事务ID的状态。每次提交或回滚操作后都会生成一个新的Oracle全局事务ID。

APPC的Oracle数据库网关使用LU6.2 SYNCLEVEL 1实现commit-confirm。这类似于单站点更新的实现，具有Oracle站点和上的资源的附加优势网关访问的OLTP可以更新并保持同步。主要区别在于提交确认实现需要在OLTP事务中进行一些额外的编程，以执行恢复支持所需的事务日志记录。

支持的OLTP

由于commit-confirm使用LU6.2 SYNCLEVEL 1 ，因此任何支持APPC的OLTP都可以支持它，包括用于z / OS和IMS / TM的CICS Transaction Server。适用于APPC的Oracle数据库网关为z / OS和IMS / TM的CICS Transaction Server提供了示例提交确认应用程序。

使用CICS Transaction Server for z / OS，标准命令级EXEC CICS接口可用于所有APPC通信。此外，如果首选，可以使用CPI-C接口。使用EXEC CICS接口在COBOL中编写的示例DB2更新事务随网关一起提供。CICS Transaction Server for z / OS支持的任何语言都可用于编写提交确认事务。

使用IMS / TM时，必须使用CPI-C接口，使IMS事务成为“显式APPC事务”，如IBM IMSCICS Transaction Server for z / OS手册中所述。这是必要的，因为它是IMS事务可以访问LU6.2 SYNCLEVEL 1控制流的唯一方法。当使用“隐含的APPC”，其中来自IOPCB的“GU”和IOPCB的“ISRT”用于接收和发送数据时，IMS事务无法访问LU6.2 SYNCLEVEL 1控制流，使其无法实现使用此方法进行commit-confirm。使用CPI-C APPC接口在COBOL中编写的示例DLI数据库更新事务随网关一起提供。IMS和CPI-C支持的任何语言都可用于编写提交确认事务。

支持Commit-Confirm所需的组件

需要以下组件来支持commit-confirm：

  • 适用于APPC服务器的Oracle数据库网关

    当在网关初始化文件中指定PGA_CAPABILITY=COMMIT_CONFIRM时，网关服务器支持commit-confirm。当网关服务器在启用了commit-confirm的情况下运行时，它将连接到本地Oracle数据库，在该数据库中维护提交确认事务日志，类似于存储在DBA_2PC_PENDING表中的Oracle两阶段提交日志。网关的事务日志存储在PGA_CC_PENDING表中。对于每个正在进行的事务，该表存储在该表中，并且在该事务完成之前保持该行。PGA_CC_PENDING中行的生命周期通常很短，仅从网关收到提交的时间开始，直到Oracle数据库完成所有提交处理并告知网关忘记事务为止。

    应保留提交确认网关SID，仅用于调用实现commit-confirm的更新事务。当PGA_CAPABILITY设置为COMMIT_CONFIRM时，记录设置中涉及一些额外的开销。应该通过单独的网关SID调用只读事务， PGA_CAPABILITY设置为READ_ONLY这样它们就不会产生额外的开销。

  • 记录服务器

    必须有一个Oracle数据库可供网关服务器用于存储PGA_CC_PENDING表。为了获得最佳性能和可靠性，Oracle建议此Oracle数据库与网关服务器位于同一系统上。

  • OLTP提交 - 确认事务日志

    必须为正在访问的OLTP系统定义提交确认事务日志数据库。此数据库必须是可恢复的，并且必须可由OLTP作为与OLTP应用程序的数据库相同的工作单元的一部分访问，以便事务日志数据库的更新将与应用程序的数据库的更新保持同步。工作。

    commit-confirm事务日志数据库只需要包含 Oracle全局事务ID和日期/时间戳。Oracle全局事务ID长度为169个字节，必须是关键字段。日期/时间戳用于清除在某些故障情况后可留在日志中的旧条目。

    为简单起见，特定OLTP下的所有提交确认应用程序应共享相同的提交确认事务日志。

  • OLTP事务日志代码

    必须将代码添加到由commit-confirm网关调用的每个OLTP事务中，以执行网关的commit-confirm实现所需的事务日志记录。此代码必须从网关接收Oracle全局事务ID，并将该信息写入OLTP提交确认事务日志数据库。为了获得最大的灵活性和易用性，可以将此代码编写为可从OLTP系统上的任何提交确认事务调用的子例程。

    此代码必须在第一个APPC接收之前的每个commit-confirm事务开始时执行，然后在事务中的每个COMMIT或ROLLBACK之后立即执行。这可确保在每个工作单元开始时完成日志记录。

  • OLTP忘记/恢复事务

    必须在OLTP系统上创建单独的APPC事务，该事务可以由网关启动，以便在成功提交事务后忘记事务并在恢复处理期间查询事务的状态。此事务在忘记处理期间从OLTP提交确认事务日志数据库中删除特定Oracle全局事务ID的条目，并在恢复处理期间从OLTP提交确认事务日志数据库查询特定Oracle全局事务ID的条目。

    注意：

    确保正确配置网关初始化参数和OLTP参数，如适用于POWER系统上的IBM AIX的APPC安装和配置指南的Oracle数据库网关（64位），Linux x86-64，Oracle Solaris的第11章中所述。 在SPARC（64位）和HP-UX Itanium或适用于Microsoft Windows的APPC安装和配置指南的Oracle数据库网关的第8章中，具体取决于您的平台。

应用设计要求

在设计用于APPC的Oracle数据库网关的提交确认应用程序时，必须满足一些要求，以便网关在发生故障时确定事务的状态。如果不满足这些要求，则尝试将应用程序与commit-confirm网关一起使用将产生不可预测的结果。

由提交确认网关调用的OLTP事务必须完成的第一件事是从网关接收O racle全局事务ID并将其记录到OLTP commit-confirm事务日志数据库中。这必须在OLTP事务和Oracle应用程序之间的正常数据流开始之前完成。网关始终将Oracle全局事务ID作为第一个数据项发送。

如果OLTP事务是一次性交易，这是唯一需要的改变。如果交易是一个持久性事务执行多个工作单元（发出多个提交或回滚），然后必须在每次COMMIT或ROLLBACK之后接收并记录新的Oracle全局事务ID。

Oracle全局事务ID由网关在可变长度记录中发送，最大长度为202字节。前32个字节包含一个特殊的二进制字符串，用于验证数据是来自网关而不是来自其他应用程序。下一个1字节是保留字段。接下来是Oracle全局事务ID，最大长度为169个字节。您必须记录保留字段和Oracle全局事务ID，以及日期/时间戳和您要记录的任何其他信息。请注意，Oracle全局事务ID必须是日志数据库的关键字段，以便忘记/恢复事务可以使用Oracle全局事务ID直接访问日志条目。

注意：

如果您的OLTP是IMS / TM，则必须将提交确认事务日志数据库的PCB添加到PSB，以用于将与提交确认网关一起使用的每个事务。该PCB必须是PSB中的第一块PCB。

提交确认架构

Oracle Database Gateway for APPC中的commit-confirm实现的体系结构包含三个主要组件：

  • Oracle数据库

  • 适用于APPC服务器的Oracle数据库网关（网关服务器）

  • 记录服务器（包含表PGA_CC_PENDING和PGA_CC_LOG的Oracle数据库）

本节描述每个组件在commit-confirm操作中的作用以及这些组件如何交互。

组件

该 Oracle数据库是commit-confirm体系结构中的控制组件。它告诉网关服务器何时提交事务以及何时回滚事务。它对参与分布式事务的所有其他服务器也是如此。发生故障时，Oracle数据库充当集成服务器，驱动每个参与服务器（包括网关服务器）中的恢复过程。

该网关服务器执行将指令从Oracle数据库转换为LU6.2操作的任务，然后将事务记录到日志记录服务器中。网关服务器将日志信息存储在日志记录服务器上的名为PGA_CC_PENDING的表中。如果在事务处理期间发生故障，则网关服务器确定应将哪个错误返回到Oracle数据库。

该日志服务器是可用的Oracle数据库网关服务器，用于存储和访问其提交确认日志信息。日志服务器不必是作为集成服务器的Oracle数据库。由于日志记录服务器是网关提交确认操作的组成部分，因此它驻留的最佳位置与网关服务器位于同一系统上。这允许网关服务器和日志记录服务器之间的通信使用进程间通信，从而在组件之间提供高速，低开销的本地连接。

互动

组件之间存在一组特定的交互。他们是：

  • Oracle数据库< - >网关服务器

    Oracle数据库驱动网关服务器的所有操作。根据Oracle应用程序的请求，集成服务器可以指示网关服务器开始新的Oracle事务，启动提交序列，启动回滚序列或启动遗忘序列。它还可以调用网关远程过程调用（RPC）函数（ PGAINIT ， PGAXFER ， PGATERM ）代表Oracle应用程序。

  • 网关服务器< - >日志服务器

    网关服务器调用日志服务器以从其PGA_CC_PENDING表中插入和删除行。这实际上是通过调用a来完成的日志服务器中的PL / SQL存储过程PGA_CC_LOG ，用于减少网关服务器执行其日志记录所需的打开游标数。网关服务器只需要一个光标进行日志记录。

提交确认流程

Oracle应用程序和Oracle应用程序之间成功提交的控制流程 OLTP事务在下一节中描述， 如图5-1所示 。该图假设Oracle和OLTP资源都已更新。 Commit-Confirm Logic Flow中的以下步骤逐步概述了提交确认逻辑流程。

提交 - 确认逻辑流程，一步一步

  1. 应用程序向Oracle数据库发出COMMIT 。
  2. Oracle数据库将PREPARE发送给除网关之外的分布式事务中的每个参与者。
  3. 每个参与者准备其数据库更新并响应PREPARE OK到Oracle数据库。
  4. Oracle数据库将COMMIT发送到网关。网关从Oracle数据库接收COMMIT ，并将新的挂起事务行插入PGA_CC_PENDING表。
  5. 网关将APPC CONFIRM发送到OLTP应用程序。OLTP应用程序以最后一个APPC RECEIVE的状态的形式接收CONFIRM请求。
  6. OLTP应用程序使用适当的OLTP函数发出COMMIT 。OLTP提交自上次COMMIT以来应用程序所做的所有数据库更新，包括commit-confirm事务日志更新。
  7. 提交数据库更新后，OLTP将控制权返回给应用程序，并返回指示COMMIT状态的返回码。
  8. OLTP应用程序将APPC CONFIRMED发送到网关。
  9. 网关接收CONFIRMED并将COMMIT OK返回给Oracle数据库。
  10. Oracle数据库将COMMIT发送给除网关之外的分布式事务中的每个参与者。
  11. 每个参与者提交其数据库更新并响应COMMIT OK到Oracle数据库。
  12. Oracle数据库将FORGET发送到网关。
  13. 网关接收FORGET并在OLTP上与FORGET/RECOVERY事务启动新的APPC对话，向其发送FORGET请求和APPC CONFIRM 。 FORGET/RECOVERY事务接收FORGET请求，并从当前Oracle事务的commit-confirm事务日志中删除该条目，并提交删除。
  14. FORGET/RECOVERY事务向网关发送APPC CONFIRMED以指示FORGET已处理，然后终止。网关接收CONFIRMED或删除从挂起的事务行PGA_CC_PENDING表。
  15. 网关将FORGET OK返回给Oracle数据库。
  16. Oracle数据库将控制权返回给Oracle应用程序。

语义

或替换

指定OR REPLACE以使用不同的定义替换属性维度的现有定义。

力量和NOFORCE

指定FORCE以强制创建属性维，即使它未成功编译也是如此。如果指定NOFORCE ，则属性维必须成功编译，否则会发生错误。默认为NOFORCE 。

模式

指定要在其中创建属性维度的架构。如果未指定架构，则Oracle数据库会在您自己的架构中创建属性维度。

attr_dimension

指定属性维的名称。

sharing_clause

指定是否将属性维创建为应用程序公共对象。指定METADATA共享属性维度的元数据，但其数据对每个容器都是唯一的。指定DATA共享属性维对象;它的数据对于应用程序容器中的所有容器都是相同的，并且数据仅存储在应用程序根目录中。指定NONE从共享中排除属性维度。

classification_clause

使用classification子句指定CAPTION或DESCRIPTION分类的值，并指定用户定义的分类。分类提供描述性元数据，应用程序可用于提供有关分析视图及其组件的信息。

您可以为同一对象指定任意数量的分类。分类的最大长度为4000字节。

对于CAPTION和DESCRIPTION分类，你可以使用快捷键DDL CAPTION ' caption '和DESCRIPTION ' description '或完全分类的语法。

您可以按语言更改分类值。要为CAPTION或DESCRIPTION分类指定语言，必须使用完整语法。如果未指定语言，则分类的语言值为NULL 。语言值必须为NULL或有效的NLS_LANGUAGE值。

尺寸类型

属性维度可以是STANDARD或TIME类型。STANDARD类型属性维度具有STANDARD类型级别。TIME类型属性维度的每个级别都是TIME类型之一。默认的DIMENSION TYPE是STANDARD 。

attr_dim_using_clause

使用USING关键字，指定表或视图。您可以使用AS关键字为表或视图指定别名。（可选）您可以指定连接路径。当属性维使用在雪花模式中组织的表时，使用连接路径指定连接。

join_path_clause

join path子句指定不同表中列之间的连接条件。对于USING子句中包含的每个连接路径， join_path_name参数指定的连接路径的名称必须是唯一的。

join_condition

连接条件由一个或多个连接条件元素组成; AND操作包含每个附加连接条件元素。

join_condition_element

在连接条件元素中，左侧的列引用必须来自与右侧的列引用不同的表。

attributes_clause

指定一个或多个attr_dim_attribute_clause子句。

attr_dim_attribute_clause

从attr_dim_using_clause源指定一列。除非使用AS关键字指定别名，否则该属性具有列的名称。您可以为每个属性指定分类。

attr_dim_level_clause

在属性维度中指定级别。级别指定提供级别成员的键和可选备用键属性。

如果key属性没有NULL值，那么您可以指定NOT NULL ，这是默认值。如果它确实有一个或多个NULL值，则指定SKIP WHEN NULL 。

等级类型

STANDARD类型属性维度具有STANDARD类型级别。您不需要为STANDARD类型属性维指定LEVEL TYPE 。

在TIME类型属性维度中，您必须指定级别类型。级别的类型可以是时间类型之一。即使级别成员的值不是该类型，您也必须指定时间类型。例如，您可能具有SEASON级别，其值为季节名称。在定义级别时，必须指定任何一种时间级别类型，例如QUARTERS 。应用程序可以将级别类型名称用于其选择的任何目的。

确定

使用DETERMINES关键字，您可以指定此级别确定的属性维度的其他属性。如果属性对于另一个属性的每个值只有一个值，则第一个属性的值确定另一个属性的值。例如，QUARTER_ID属性对MONTH_ID属性的每个值只有一个值，因此您可以在MONTHS级别的DETERMINES短语中包含QUARTER_ID属性。

key_clause

指定一个或多个属性作为级别的键。

alternate_key_clause

指定一个或多个属性作为级别的备用键。

dim_order_clause

指定级别成员的顺序。

all_clause

（可选）为隐式ALL级别指定MEMBER NAME ， MEMBER CAPTION和MEMBER DESCRIPTION值。默认情况下， MEMBER NAME值为ALL。

一个设计一个维度模型

本指南使用Global模式作为示例。本附录探讨了虚构的全球计算公司的业务需求，并讨论了如何根据这些要求设计数据模型。

本附录包含以下主题：

  • 案例研究场景

  • 确定所需的业务事实

  • 为全球计算设计维度模型

A.1案例研究情景

虚构的全球计算公司成立于1990年。Global Computing在全球范围内向客户分发计算机硬件和软件组件。销售和营销部门尚未达到预算编号。因此，该部门一直面临着制定成功的销售和营销战略的挑战。

Global Computing在竞争激烈的市场中运营。竞争对手众多，客户对价格特别敏感，利润率往往较窄。为了实现盈利增长，Global Computing必须增加其盈利能力最强的产品的销售额。

Global Computing当前业务的各种因素表明销售额和利润下降：

  • 传统上，Global Computing的第三季度销售额较低（7月至9月）。然而，近期其他季度的销售额也低于预期。该公司经历了多次增长，但由于没有明显的原因，过去两年的第一季度销售额与前几年相比有所下降。

  • Global凭借其最新的销售渠道 - 互联网 - 取得了成功。尽管该渠道的销售额在增长，但总体利润却在下降。

  • 也许最重要的因素是个人电脑的利润 - 以前是全球计算大部分利润的来源 - 正在迅速下降。

全球计算必须了解这些因素如何影响其业务。

当前报告由IT部门完成，IT部门每月生成一些标准报告。任何临时报告都是根据需要进行处理的，并受到有限IT人员的时间限制。销售和营销部门对于报告请求的延迟响应已经普遍存在投诉。IT部门对于经常改变主意或要求提供进一步信息的分析师的投诉也很多。

销售和营销部门一直在努力解决缺乏及时销售产品，购买者以及购买方式的信息。在与CIO的会议上，销售和营销副总裁表示，“当我获得信息时，它已不再有用。我只能在每个月末获得信息，而且它没有我完成工作所需的细节。“

A.1.1报告要求

当被要求更具体地了解她的需求时，销售和营销副总裁确定了以下要求：

  • 特定客户，地区和细分市场的趋势销售数据。

  • 能够为现场销售人员提供信息和一些分析功能。由于销售人员遍布全球，因此Web界面将是首选。

  • 有关邮购，电话和电子邮件销售的详细信息，按月和按季度进行，并与过去的时间段进行比较。信息必须确定每个渠道的销售时间，方式和内容。

  • 关于产品的保证金信息，以了解每笔销售的美元贡献。

  • 了解销售额，单位和保证金的前一年和一年前的百分比变化。

  • 通过临时分组执行数据分析的能力。

CIO已经与他的团队讨论了这些要求，并得出结论，针对生产订单输入系统的标准报告解决方案不够灵活，无法提供所需的分析功能。业务分析的报告要求非常多样化，以至于预计的开发成本以及预期的请求周转时间将使该解决方案变得不可接受。

CIO团队建议使用分析工作区来支持分析。该团队建议销售和营销部门的IT部门与企业IT部门合作，构建一个满足其信息分析需求的分析工作区。

A.1.2业务目标

开发团队确定项目必须满足的以下高级业务目标：

  • Global Computing的战略目标是通过增加高利润产品的销售额和增加整体销售量来增加公司利润。

  • 销售和营销部门的目标是：

    • 分析行业趋势并针对特定细分市场。

    • 分析销售渠道，增加利润。

    • 确定产品趋势并制定开发适当渠道的策略。

A.1.3信息要求

一旦确定了业务目标，就可以确定有助于实现这些目标的信息类型。要了解最终用户如何检查分析工作区中的数据，进行广泛的访谈非常重要。通过与关键最终用户的访谈，您可以确定他们对业务的看法以及他们想要回答的业务分析问题类型。

A.1.3.1业务分析问题

Global Computing的销售和营销副总裁，销售人员和市场分析师的访谈揭示了以下业务分析问题：

  • 哪些产品有利可图？

  • 谁是我们的客户，他们购买的是什么以及如何购买？

  • 什么账户最有利可图？每个分销渠道的表现如何？

  • 该业务是否仍存在季节性差异？

我们可以详细检查每个业务分析问题。

哪些产品有利可图？

此业务分析问题包含以下问题：

  • 在任何月份，季度或年度以及任何分销渠道中，任何项目，产品系列或产品类别的总销售额百分比是多少？这个百分比的销售额与一年前有何不同？

  • 任何特定月份的任何项目的单位价格，单位成本和单位成本是多少？任何月份的任何商品的价格，成本和保证金趋势是多少？

  • 在任何分销渠道，任何地理区域或细分市场中，哪些项目在任何月份，季度或年度中最有利可图？盈利能力如何从前一时期发生变化？盈利能力与上一期间的变化百分比是多少？

  • 哪些项目的盈利能力与前一时期相比变化最大？

  • 在任何分销渠道，任何地理区域或细分市场中，哪些项目对任何月份，季度或年度的总盈利能力贡献最大？

  • 什么项目在任何特定月份的单位保证金最高？

  • 总之，趋势是什么？

谁是我们的客户，他们购买的是什么以及如何购买？

此业务分析问题包含以下问题：

  • 任何月份，季度或年度的任何项目，产品系列或产品类别的销售额是多少？

  • 在任何分销渠道，地理区域或细分市场中，任何商品，产品系列或产品类别的销售额是多少？

  • 销售额如何与上一期间相比有所变化？上一期的销售变化百分比是多少？

  • 销售额与一年前相比有何变化？一年前的销售变化百分比是多少？

  • 总之，趋势是什么？

哪些帐户最赚钱？

此业务分析问题包含以下问题：

  • 哪个帐户在任何分销渠道，任何项目，产品系列或产品类别中的任何月份，季度或年度最有利可图？

  • 对于任何分销渠道和任何产品，任何月份，季度或年度的销售额和扩展保证金（毛利润）是多少？

  • 账户盈利能力与前一时期相比如何？

  • 与前一时期相比，哪些帐户的销售额增幅最大？

  • 上一期的销售变化百分比是多少？盈利率的百分比变化是否与销售额变化百分比相同？

  • 总之，趋势是什么？

每个分销渠道的表现如何？

此业务分析问题包含以下问题：

  • 对于任何项目，产品系列或产品类别，或任何地理区域或细分市场，每个分销渠道的总销售额占销售额的百分比是多少？

  • 每个分销渠道的盈利能力是什么：直销，目录销售和互联网？

  • 是最新的分销渠道，互联网，“蚕食”目录销售？客户只是简单地转换订购方式，还是互联网分销渠道覆盖其他客户？

  • 总之，趋势是什么？

该业务是否仍存在季节性差异？

此业务分析问题包含以下问题：

  • 特定商品或产品系列是否有可识别的季节性销售模式？

  • 季节性销售模式如何因地理位置而异？

  • 季节性销售模式如何因细分市场而异？

  • 与去年相比，季节性销售模式是否存在差异？

A.1.3.2信息要求摘要

通过检查用户想要执行的分析类型，我们可以确定以下分析的关键要求：

  • 全球计算对盈利能力分析有很强的需求。公司必须了解产品，帐户，细分市场和分销渠道的盈利能力。它还必须了解盈利趋势。

  • 全球计算必须了解销售额如何随时间变化。公司必须按产品，地理区域，细分市场和分销渠道了解这些季节性趋势。

  • 全球计算需要临时销售分析。分析必须确定哪些产品销售给谁，何时销售这些产品以及客户如何购买这些产品。

  • 执行趋势分析的能力对全球计算非常重要。

A.3为全球计算设计维度模型

“ 业务目标 ”确定了支持Global Computing分析要求的业务事实。接下来，我们确定数据模型中的维度，级别和属性。我们还确定每个维度内的关系。生成的数据模型用于设计全局模式，维模型和分析工作区。

A.3.1识别尺寸

四个维度用于组织数据库中的事实：

  • 产品显示数据如何因产品而异。

  • 客户显示数据如何因客户或地理区域而异。

  • 频道显示数据如何根据每个分发渠道而变化。

  • 时间显示数据随时间的变化情况。

A.3.2识别级别

现在我们已经确定了维度，我们可以确定每个维度内的摘要级别。Global Computing的分析要求表明：

  • 有三种分销渠道：销售，目录和互联网。这三个值是数据仓库中最低级别的详细信息，并按通道级别进行分组。从最高级别的汇总顺序到最低级别的详细信息，级别为Total和Channel。

  • Global通过向客户发货的方式和市场细分来执行客户和地理分析。出货和市场是客户维度中的两个层次结构。在每种情况下，数据模型中最低级别的详细信息是“收货地点”位置。

    • 在分析客户出货量时，汇总级别（从最高到最低）：总计，地区，仓库和出货地。

    • 通过市场细分进行分析时，汇总级别（从最高到最低）：总计，市场细分，帐户和发货到。

  • “产品”维度有四个级别（从最高到最低）：总计，类别，族和项目。

  • 时间维度有四个级别（从最高到最低）：总计，年份，季度和月份。维度有两个层次结构：日历和财务。

所有维度都具有总计级别作为最高级别的摘要。添加此最高级别可在应用程序用户分析数据时提供额外的灵

A.3.3识别层次结构

我们可以识别组织每个维度中的级别的层次结构。为了识别层次结构，我们以正确的汇总顺序对级别进行分组，并以支持已识别的分析类型的方式对级别进行分组。

对于渠道和产品维度，全球计算每个维度只需要一个层次结构。对于Customer维度，Global Computing需要两个层次结构。客户维度内的分析往往是按地理区域或市场细分。因此，我们将级别组织为两个层次结构，即发货和分段。随着时间的推移分析还需要两个层次结构，即日历层次结构和财务层次结构。

A.3.4识别存储的措施

“ 识别所需的业务事实 ”列出了支持全球计算分析要求所需的21个业务事实。在这个数字中，只需从事务数据库中获取四个事实：

  • 单位

  • 销售

  • 单价

  • 单位成本

所有其他事实都可以从这些基本事实中得出。可以根据需要在分析工作空间中计算派生的事实。如果经验表明这些衍生事实中的一些被大量使用并且计算对系统造成了明显的负担，则可以计算这些事实中的一些并将其作为数据维护过程存储在分析工作空间中。

9.3通过XML创建关系视图：将XML节点映射到列

要将XMLType表的多个级别的数据作为关系视图中的单个行公开， XMLType SQL / XML函数XMLTable应用于每个级别。只要XMLType表中的文档与视图中的行之间存在一对多 （1：N）关系，就应使用此技术。

也就是说，您使用与分解单个级别相同的一般方法（请参阅创建基于XML的关系视图：每个XML文档一行 ）：定义构成视图的列，并将XML节点映射到这些列。但在这种情况下，您将XMLTable应用于要XMLTable并存储在关系列中的每个文档级别。

例如，每个PurchaseOrder元素都包含一个LineItems元素，该元素又包含一个或多个LineItem元素。每个LineItem元素都有子元素，例如Description和ItemNumber属性。要使这些较低级别的数据可作为关系值访问，请使用XMLTable来同时投影PurchaseOrder元素和LineItem集合。

当元素PurchaseOrder被分解时，其后代LineItem元素被映射到XMLType类型的列，其中包含XML片段。然后将该列传XMLTable对XMLTable的第二次调用，将其作为多列关系值分解为其各个部分。

例9-2说明了这一点。它使用XMLTable实现XMLType表po_binaryxml的文档与关系视图purchaseorder_detail_view的行之间的一对多 （1：N）关系。该视图提供对集合的各个成员的访问，并将集合成员公开为一组行。

在示例9-2中 ，对于存储在XMLType表po_binaryxml中的XML文档中的每个LineItem元素，视图purchaseorder_detail_view有一行。

7在联机事务处理环境中使用分区

分区功能对OLTP系统非常有用。

由于在线事务处理（OLTP）系统及其用户群的爆炸式增长，除了数据仓库环境之外，分区对OLTP系统特别有用

分区通常用于OLTP系统，以减少争用，同时支持非常大的用户群。它还有助于解决OLTP系统面临的法规要求，包括以经济有效的方式存储大量数据。

本章包含以下部分：

  • 什么是在线交易处理系统？

  • 在线事务处理环境中的性能

  • 在线事务处理环境中的可管理性

什么是在线交易处理系统？

在线事务处理（OLTP）系统是当今企业中常见的数据处理系统。OLTP系统的典型示例是订单输入，零售销售和金融交易系统。

OLTP系统主要通过与数据仓库环境不同的特定数据使用来表征，但是一些特征（例如，具有大量数据和与生命周期相关的数据使用和重要性）是相同的。

OLTP环境的主要特征是：

  • 响应时间短

    OLTP环境的性质主要是任何类型的交互式临时使用，例如电话推销员输入电话调查结果。OLTP系统需要较短的响应时间，以便用户保持高效。

  • 小额交易

    OLTP系统通常读取和操作高选择性的少量数据;数据处理大多是简单的，复杂的连接比较少见。始终存在混合的查询和DML工作负载。例如，许多呼叫中心员工中的一个检索每个呼叫的客户详细信息并在查看过去与客户的通信时输入客户投诉。

  • 数据维护操作

    拥有必须定期或临时运行的报告程序和数据更新程序并不罕见。这些程序在用户继续处理其他任务时在后台运行，可能需要大量的数据密集型计算。例如，大学可以开始批量工作，将学生分配到课程，而学生仍然可以在线注册课程。

  • 用户人数众多

    OLTP系统可能具有非常大的用户群，其中许多用户试图同时访问相同的数据。例如，在线拍卖网站可以有数十万（如果不是数百万）用户同时访问其网站上的数据。

  • 高并发性

    由于用户数量大，响应时间短，交易量小，OLTP环境中的并发性非常高。对数千个并发用户的要求并不罕见。

  • 大数据量

    OLTP系统可能会变得非常大，具体取决于应用程序类型，用户群和数据保留时间。例如，银行的每个客户都可以访问在线银行系统，该系统显示过去12个月的所有交易。

  • 高可用性

    OLTP系统的可用性要求通常非常高。不可用的OLTP系统可能会影响非常大的用户群，如果OLTP系统不可用，组织可能会遭受重大损失。例如，证券交易系统在交易时间内具有极高的可用性要求。

  • 与生命周期相关的数据使用

    与数据仓库环境类似，OLTP系统通常会经历不同的数据访问模式。例如，在月末，计算每个活动帐户的月利息。

以下是OLTP环境分区的好处：

  • 支持更大的数据库

    作为高可用性策略的一部分，备份和恢复可以在较低的粒度级别上执行，以有效地管理数据库的大小。OLTP系统通常在备份期间保持联机状态，用户可以在备份运行时继续访问系统。备份过程不应对在线用户造成重大性能下降。

    分区有助于减少OLTP系统的空间需求，因为数据库对象的一部分可以压缩存储，而其他部分可以保持未压缩。针对未压缩行更新事务比压缩数据更新更有效。

    分区可以将数据透明地存储在不同的存储层上，以降低保留大量数据的成本。

  • 用于数据维护的分区维护操作（而不是DML）

    对于数据维护操作（清除是最常见的操作），您可以利用Oracle数据库在线索引维护功能来利用分区维护操作。分区管理操作生成的重做次数少于等效的DML操作。

  • 通过消除热点可能获得更高的并发性

    OLTP环境的一个常见场景是单调增加索引值，用于强制执行主键约束，从而创建高并发和潜在争用的区域：每个新插入尝试更新同一组索引块。分区索引，特别是散列分区索引，可以帮助缓解这种情况。

在线事务处理环境中的性能

OLTP环境中的性能很大程度上依赖于有效的索引访问，因此选择最合适的索引策略变得至关重要。

以下部分讨论了决定是否在OLTP环境中对索引进行分区的最佳实践。

  • 决定是否分区索引

  • 如何在索引组织表上使用分区

决定是否分区索引

由于查询的选择性和OLTP应用程序的高并发性，正确的索引策略的选择无疑是在OLTP环境中使用分区的重要决策。通过较少的争用，应用程序可以支持更大的用户群。

以下基本规则解释了各种可能的索引结构的主要优点和权衡：

  • 如果选择了索引访问路径，则非分区索引虽然大于单个分区索引段，但总是会导致单个索引探测（或扫描）;表只有一个段。对于分区表和非分区表，数据访问时间和正在访问的块数是相同的。

    非分区索引不提供分区自治，并且需要对影响rowid的每个分区维护操作执行索引维护操作（例如，删除，截断，移动，合并，合并或拆分操作）。

  • 使用分区索引时，始终存在多个段。只要Oracle数据库无法修剪到单个索引段，数据库就必须访问多个段。这可能会导致更高的I / O要求（与非分区索引的一个探测相比， n个索引段探测）并且可能对运行时性能产生影响（可测量或不可测量）。对于所有分区索引都是如此。

    分区索引可以是本地分区索引或全局分区索引。本地分区索引始终从表继承分区键，并与表分区完全对齐。因此，任何类型的分区维护操作都几乎不需要索引维护工作。例如，删除或截断分区不会产生任何可衡量的索引维护开销;本地索引分区被删除或截断。

    未与表对齐的分区索引称为全局分区索引。与本地索引不同，表和索引分区之间没有关系。全局分区索引可以灵活地选择最适合有效分区索引访问的分区键。分区维护操作通常会影响全局分区索引的更多（如果不是全部）分区，具体取决于索引的操作和分区键。

  • 在某些情况下，为索引创建多个段可能有利于提高性能。在OLTP环境中使用序列来创建人工密钥是很常见的。因此，您创建单调递增的键值，这会导致许多插入过程竞争相同的索引块。引入全局分区索引（例如，在键列上使用全局散列分区）可以缓解这种情况。例如，如果您有这样一个索引的四个哈希分区，那么您现在有四个要插入数据的索引段，从而将插入过程中这些段的并发性降低了四倍。

强制唯一性是OLTP环境的重要数据库功能。可以使用非分区索引和分区索引强制实施唯一性。但是，由于分区索引提供了分区自治，因此必须满足以下要求才能实现唯一索引：

  • 非分区索引可以强制任何给定列或列组合的唯一性。与非分区表相比，分区表的非分区索引的行为没有区别。

  • 分区索引的每个分区都被视为自治段。要强制执行这些段的自治，您始终必须将分区键列包含为唯一键定义的子集。

    • 全局分区索引必须始终至少以索引列的第一个前导列（分区全局索引的分区列）为前缀。

    • 唯一本地索引必须将表的分区键作为唯一键定义的子集。

例7-1显示了在orders_oltp表的order_id列上创建唯一索引。使用序列号填充OLTP应用程序中的order_id 。唯一索引使用散列分区来减少对单调递增的order_id值的争用。然后使用唯一键来创建主键约束。

示例7-1创建唯一索引和主键约束

创建独特的索引orders_pk ON orders_oltp（order_id）全局分区哈希（order_id）（分区p1表格tbs1，分区p2表格tbs2，分区p3表格tbs3，分区p4表格tbs4）NOLOGGING; ALTER TABLE orders_oltp ADD CONSTRAINT orders_pk PRIMARY KEY（order_id）USING INDEX;

如何在索引组织表上使用分区

当您的工作负载适合使用索引组织表时，您必须考虑如何在索引组织表和任何二级索引上使用分区。

您必须根据与常规堆表上的索引相同的注意事项来决定是否对索引组织表上的二级索引进行分区。您可以对索引组织表进行分区，但分区键必须是主键的子集。对索引组织表进行分区的一个常见原因是减少争用;这通常使用散列分区来实现。

对索引组织表进行分区的另一个原因是能够基于主键列物理地分离数据集。例如，应用程序托管公司可以通过对公司标识符进行列表分区来物理地为不同客户分离应用程序实例。在这种情况下查询通常可以利用索引分区修剪，缩短索引扫描的时间。具有索引组织表和分区的ILM方案不太常见，因为它们需要日期列作为主键的一部分。

在线事务处理环境中的可管理性

除了性能优势之外，分区还可以为OLTP环境中的大型对象提供最佳数据管理。

Oracle数据库中的每个分区维护操作都可以扩展为原子地包括全局和本地索引维护，从而可以执行任何分区维护操作，而不会影响OLTP环境的24x7可用性。

由于ILM方案，OLTP系统中的分区维护操作经常发生。在这些场景中，[range | interval]分区表，或[range | interval] - *复合分区表，很常见。

分区维护操作的一些业务案例包括围绕应用程序数据分离的场景。例如，零售公司在单个模式中为多个分支运行相同的应用程序。根据分支收入，应用程序（作为单独的分区）存储在更高效的存储上。列表分区或list- * composite partitioning是此类业务案例的常见分区策略。

您可以在OLTP系统中使用散列分区或表的哈希子分区，以获得与数据仓库环境中实现的性能优势类似的性能优势。每日OLTP工作负载的大部分由相对较小的操作组成，并且是串行执行的。但是，定期批处理操作可以并行执行，并受益于散列分区和子分区可以为分区连接提供的分发改进。例如，可以并行执行月末利息计算以在夜间批处理窗口内完成。

分区维护操作对具有本地索引的分区表的影响

发生分区维护操作时，Oracle数据库会锁定受影响的表分区以进行任何DML操作，但ONLINE MOVE除外。

除DROP或TRUNCATE操作外，受影响分区中的数据仍可完全访问任何SELECT操作。由于本地索引在逻辑上与表（数据）分区耦合，因此只有受影响的表分区的本地索引分区必须作为分区维护操作的一部分进行维护，从而实现索引维护的最佳处理。

例如，使用ALTER TABLE MOVE ONLINE功能将旧分区从高端存储层移动到低成本存储层时，数据和索引始终可用于SELECT和DML操作;必要的索引维护是更新现有索引分区以反映数据的新物理位置。如果在归档后删除旧分区，则其本地索引分区也会被删除，对于全局索引，已删除已删除分区的孤立条目，从而启用仅影响数据字典的分秒分区维护操作。

分区维护操作对全局索引的影响

每当在分区或非分区表上定义全局索引时，不同的表分区和索引之间就没有相关性。因此，任何分区维护操作都会影响所有全局索引或索引分区。

包含本地索引的表的分区将被锁定，以防止针对影响表分区的DML操作，但ONLINE MOVE操作除外。但是，与本地索引的索引维护不同，任何全局索引仍可完全用于DML操作，并且不会影响OLTP系统的联机可用性。

从概念上和技术上讲，分区维护操作的全局索引的索引维护与语义相同的DML操作所需的索引维护相当，但DROP和TRUNCATE除外，其中全局索引维护延迟到稍后的时间点。有关管理全局索引的更多信息，请参阅全局分区索引管理 。

例如，删除旧分区在语义上等同于使用SQL DELETE语句DELETE旧分区的所有记录。在DML情况下，必须从任何全局索引中删除已删除数据集的所有索引条目作为标准索引维护操作，这不会影响SELECT和DML操作的索引的可用性。

DROP PARTITION也不会影响索引可用性，但可以使必要的索引维护与初始数据删除分离，而不会影响全局索引的可用性。在这种情况下，丢弃操作表示最佳方法：数据被移除而没有传统DELETE操作的开销，并且全局索引以非侵入方式维护。

OLTP环境中的通用分区维护操作

两种最常见的分区维护操作是删除数据以及将数据重定位到成本较低的存储层设备上。

  • 删除（清除）旧数据

  • 将较旧的分区移动或合并到低成本存储层设备

删除（清除）旧数据

使用DROP或TRUNCATE操作会根据分区键标准删除旧数据。

删除操作会删除数据和分区元数据，而TRUNCATE操作仅删除数据但保留元数据。将分别删除所有本地索引分区，并将其截断。异步全局索引维护是针对分区或非分区全局索引完成的，并且完全可用于select和DML操作。

以下示例从orders_oltp表中删除早于2006年1月的所有数据。作为drop语句的一部分，将执行UPDATE GLOBAL INDEXES语句，以便全局索引在整个维护操作中保持可用。作为此操作的一部分，将删除任何本地索引分区。 

ALTER TABLE orders_oltp DROP PARTITION p_before_jan_2006 UPDATE GLOBAL INDEXES;
将较旧的分区移动或合并到低成本存储层设备

使用MOVE或MERGE操作作为信息生命周期管理（ILM）策略的一部分，您可以将旧分区重新定位到最具成本效益的存储层。

使用ALTER TABLE ONLINE MOVE功能可以使数据同时用于查询和DML操作。维护本地索引并可能作为合并或移动操作的一部分重新定位。标准索引维护是针对分区或非分区全局索引完成的，并且完全可用于select和DML操作。

以下示例显示如何在orders_oltp表中合并2006年1月和2006年2月的分区，并将它们存储在不同的表空间中。作为此操作的一部分，任何本地索引分区也会移动到ts_low_cost表空间。UPDATE INDEXES子句确保所有索引在操作的整个过程中和操作后都可用，无需额外的重建。 

ALTER TABLE orders_oltp MERGE PARTITIONS p_2006_jan，p_2006_feb INTO PARTITION p_before_mar_2006 COMPRESS TABLESPACE ts_low_cost UPDATE INDEXES;

3.11 RMAN

目的

使用RMAN命令从操作系统命令行启动RMAN。

先决条件

您必须在操作系统命令行而不是在RMAN提示符下发出RMAN命令和任何选项。

指定和验证与数据库的RMAN连接的方式与SQL * Plus与数据库的连接相同。唯一的区别是与目标或辅助数据库的RMAN连接需要SYSBACKUP权限。

也可以看看：

  • Oracle数据库管理员指南 ，了解使用SQL * Plus时的数据库连接选项

  • “Oracle数据库备份和恢复用户指南”了解如何使用SYSBACKUP管理权限

警告：

良好的安全性要求不要在命令行上以纯文本形式输入密码。仅在RMAN提示请求时在RMAN中输入密码。请参阅Oracle数据库安全指南以了解密码保护。

使用说明

您在操作系统提示符下输入的命令名称取决于操作系统。例如，在Linux和UNIX系统上以小写形式输入rman 。

如果在操作系统命令行上未指定CATALOG或NOCATALOG而启动RMAN，则除非执行CONNECT CATALOG命令，否则RMAN会话实际上处于NOCATALOG模式（请参阅示例3-48 ）。如果维护恢复目录，则最佳做法是在执行RMAN操作之前将RMAN连接到恢复目录。

18使用碎片的OCI接口

本章介绍使用Oracle Sharding的OCI界面。

碎片是一组数据库中的位置，每个数据库存储数据的某些部分。存储在每个数据库中的数据部分由一组块表示 ，其中每个块与一定范围的数据相关联。

要发出读取或写入块的请求，必须将应用程序路由到在连接启动步骤中存储该块的相应数据库（分片）。通过使用数据密钥完成此路由。数据密钥通过指定其分片密钥或通过指定其超级分片密钥来路由到特定的块，从而能够路由到特定的块。为了获得包含您希望操作的块的正确分片的连接，必须在连接到分片Oracle数据库之前为应用程序指定密钥，以获得独立连接或从OCI会话池获取的连接。对于OCI会话池，您必须在从池中检出连接之前指定数据密钥。

对于OCI会话池和独立连接，形成分片密钥和分片组密钥以及与底层连接建立会话的步骤包括：

  1. 通过调用分配的分片密钥描述符OCIDescriptorAlloc()并指定描述符类型参数作为OCI_DTYPE_SHARDING_KEY以形成分片键。

    1. 通过调用OCIShardingKeyColumnAdd()多次添加分片键的所有列，以形成完整的分片键。

    2. 调用OCIAttrSet()并指定OCI_ATTR_SHARDING_KEY属性以在身份验证句柄上设置分片键。

  2. 分配通过调用碎片组密钥描述符OCIDescriptorAlloc()并指定描述符类型参数作为OCI_DTYPE_SHARDING_KEY以形成分片组密钥。

    1. 通过调用OCIShardingKeyColumnAdd()多次添加分片键的所有组列，以形成完整的分片组键。

    2. 调用OCIAttrSet()并指定OCI_ATTR_SUPER_SHARDING_KEY属性以在身份验证句柄上设置分片组键。

  3. 使用上一步中包含分片密钥和分片组密钥信息的初始化身份验证句柄调用OCISessionGet()以获取数据库与分片键和分片组键指定的分块组指定的分片和块的连接。

对于自定义池，形成分片键和分片组键以及从池中检出连接的步骤包括：

  1. 如果自定义池中没有现有连接，请直接转到步骤7。否则，请执行以下步骤。

  2. 通过调用分配的分片密钥描述符OCIDescriptorAlloc()并指定描述符类型参数作为OCI_DTYPE_SHARDING_KEY以形成分片键。通过调用OCIShardingKeyColumnAdd()多次添加分片键的所有列，以形成完整的分片键。

  3. 分配通过调用碎片组密钥描述符OCIDescriptorAlloc()并指定描述符类型参数作为OCI_DTYPE_SHARDING_KEY以形成分片组密钥。通过调用OCIShardingKeyColumnAdd()多次添加分片键的所有组列，以形成完整的分片组键。

  4. 使用分片键，超级分片键描述符和连接字符串调用OCIShardInstancesGet() ，以返回包含指定分片键描述符和超级分片键描述符的所需块的实例名称。

  5. 检查自定义池中的每个连接，以查看它是否指向OCIShardInstancesGet()返回其名称的实例之一。要检查自定义池中的连接指向的实例，可以使用服务上下文句柄（ OCISvcCtx * ）上的OCI_ATTR_INSTNAME获取实例名称。

剑OCIShardInstancesGet（void ** shTopoCtx，OCIError * errhp，const OraText * connstr，ub4 constrl，OCIShardingKey * shardingKey，OCIShardingKey * superShardingKey，OCIShardinst *** shardinsts，ub4 numShardInsts，ub4 mode）;

如果分片拓扑缓存尚未具有映射，则返回的值可能为NULL ，因为没有与OCI_SESSGET_CUSTOM_POOL建立连接，或者到目前为止使用OCI_SESSGET_CUSTOM_POOL建立的连接未连接到包含分片键的块的分片，请求超级分片密钥。在这两种情况下，自定义池应建立明确使用一个新的连接OCISessionGet()与OCI_ATTR_SHARDING_KEY和OCI_ATTR_SUPER_SHARDING_KEY属性设置为适用与OCI_SESSGET_CUSTOM_POOL启用模式。这样做会将有关新分片的信息添加到OCI分片拓扑缓存中。后续的OCIShardInstancesGet()调用将查找属于这些分OCIShardInstancesGet()块的键范围，并返回这些分片实例名称。

E.2.1使用RMAN创建备用数据库的目的

您可以使用手动技术或RMAN DUPLICATE命令从主数据库的备份创建备用数据库。

您可以使用手动技术或RMAN DUPLICATE命令从主数据库的备份创建备用数据库。使用RMAN创建备用数据库与手动技术相比具有以下优势：

  • RMAN可以通过复制主数据库当前使用的文件来创建备用数据库。无需备份。

  • RMAN可以通过将主数据库的备份还原到备用站点来创建备用数据库。因此，在创建备用数据库期间，主数据库不受影响。

  • RMAN自动重命名文件，包括Oracle托管文件（OMF）和目录结构。

  • RMAN从备份还原存档的重做日志文件并执行介质恢复，以便同步备用数据库和主数据库。

E.2.2使用RMAN创建备用数据库的基本概念

使用RMAN创建备用数据库的过程与创建重复数据库的过程几乎相同。

您需要修改“ Oracle数据库备份和恢复用户指南”中描述的复制过程，以解决备用数据库特有的问题。

要使用DUPLICATE命令创建备用数据库，必须将目标连接到主数据库并指定FOR STANDBY选项。您无法连接到备用数据库并创建其他备用数据库。RMAN通过还原和装入控制文件来创建备用数据库。RMAN可以使用主数据库控制文件的现有备份，因此您无需为备用数据库创建控制文件备份。

与没有 FOR STANDBY OPTION的DUPLICATE创建的重复数据库不同，备用数据库不会获得新的DBID。因此，请勿在备份数据库中注册恢复目录。

E.2.2.1活动数据库和基于备份的复制

使用RMAN创建备用数据库时，必须在主动和基于备份的复制之间进行选择。

如果指定FROM ACTIVE DATABASE ，则RMAN将数据文件直接从主数据库复制到备用数据库。必须安装或打开主数据库。

如果未指定FROM ACTIVE DATABASE ，则RMAN将执行基于备份的复制。RMAN将主数据文件的备份还原到备用数据库。创建和恢复备用数据库所需的所有备份和归档重做日志文件必须可由备用主机上的服务器会话访问。除非执行SET UNTIL命令，否则RMAN将还原最新的数据文件。

E.2.2.2 RMAN环境中的DB_UNIQUE_NAME值

与没有FOR STANDBY选项的DUPLICATE创建的重复数据库不同，备用数据库不会获得新的DBID。

在Oracle Data Guard环境中使用RMAN时，请将其连接到恢复目录。恢复目录可以存储环境中所有主数据库和备用数据库的元数据。不要在恢复目录中显式注册备用数据库。

E.2.2.3恢复备用数据库

默认情况下，RMAN在创建后不会恢复备用数据库。

RMAN将挂起备用数据库，但不会将备用数据库置于手动或受管恢复模式。RMAN断开连接并且不执行备用数据库的介质恢复。

要使RMAN在创建备用数据库后恢复备用数据库，备用控制文件必须可用于恢复。必须满足以下条件：

  • 备用数据库的结束恢复时间必须大于或等于备用控制文件的检查点SCN。

  • 必须在备用站点上提供包含备用控制文件的检查点SCN的归档重做日志文件以进行恢复。

确保满足这些条件的一种方法是在备份主数据库上的控制文件后发出ALTER SYSTEM ARCHIVE LOG CURRENT语句。此语句归档主数据库的联机重做日志文件。然后，使用RMAN备份最新的归档重做日志文件，或将归档的重做日志文件移动到备用站点。

使用DUPLICATE命令的DORECOVER选项指定您希望RMAN恢复备用数据库。创建备用数据库文件后，RMAN执行以下步骤：

  1. RMAN开始媒体恢复。如果恢复需要存档的重做日志文件，并且日志文件尚未存在于磁盘上，则RMAN会尝试还原备份。

  2. RMAN将备用数据库恢复到指定的时间，系统更改号（SCN）或日志文件序列号，或者恢复到最新的归档重做日志文件（如果未指定前面的任何一个）。

  3. RMAN离开后安装介质恢复备用数据库完成，但没有放置在手动或管理恢复模式的备用数据库。

E.2.2.3.1备用数据库重做日志文件

RMAN自动在备用数据库上创建备用重做日志文件。

创建日志文件后，备用数据库会根据日志文件的常规规则维护和归档它们。

如果使用基于备份的复制，则在备用数据库上命名备用重做日志文件时，唯一的选项是日志文件的文件名，如备用控制文件中所指定。如果备用数据库上的日志文件名必须与主文件名不同，则一个选项是通过设置LOG_FILE_NAME_CONVERT为备用重做日志指定文件名在备用初始化参数文件中。

在备用数据库上指定备用重做日志文件的文件名时，请注意以下限制：

  • 如果主数据库和备用数据库对日志文件使用不同的命名约定，则必须使用LOG_FILE_NAME_CONVERT参数来命名备用重做日志文件。

  • 你不能使用 SET NEWNAME或 CONFIGURE AUXNAME命令重命名备用重做日志文件。

  • 您不能使用DUPLICATE命令的LOGFILE子句为备用重做日志文件指定文件名。

  • 要使备用数据库上的备用重做日志文件名与主要重做日志文件名相同，必须指定DUPLICATE命令的NOFILENAMECHECK子句。否则，即使备用数据库是在其他主机上创建的，RMAN也会发出错误信号。

E.2.2.4备用数据库的密码文件

如果使用活动数据库复制，则RMAN始终将密码文件复制到备用主机，因为备用数据库上的密码文件必须是目标数据库上密码文件的精确副本。

如果使用活动数据库复制，则RMAN始终将密码文件复制到备用主机，因为备用数据库上的密码文件必须是目标数据库上密码文件的精确副本。在这种情况下， PASSWORD FILE子句不是必需的。RMAN会覆盖辅助实例的任何现有密码文件。对于基于备份的复制，您必须将主数据库上使用的密码文件复制到备用数据库，以便Oracle Data Guard发送日志。

111.2 DBMS_NETWORK_ACL_UTILITY示例

此程序包中的CONTAINS_HOST函数指示域或子网是否包含给定的主机或IP地址。

它可以与DBMS_NETWORK_ACL_ADMIN包中的CHECK_PRIVILEGE_ACLID函数一起使用，以确定影响用户访问网络主机的权限的权限分配。CONTAINS_HOST函数的返回值也可用于按优先顺序排列ACL分配。

例1

例如，SCOTT允许连接到www.hr.example.com ：

SELECT host，lower_port，upper_port，acl，DECODE（DBMS_NETWORK_ACL_ADMIN.CHECK_PRIVILEGE_ACLID（aclid，'SCOTT'，'connect'），1，'GRANTED'，0，'DENIED'，NULL）特权FROM（SELECT host，acl，aclid， lower_port，upper_port，DBMS_NETWORK_ACL_UTILITY.CONTAINS_HOST（'www.hr.example.com'，host）优先级FROM dba_network_acls）WHERE优先级> 0 ORDER BY优先级DESC，lower_port nulls LAST; HOST LOWER_PORT UPPER_PORT ACL PRIVILEGE -------------------- ---------- ---------- ----- --------------- --------- www.hr.example.com 80 80 /sys/acls/www.xml GRANTED www.hr.example.com 3000 3999 /sys/acls/www.xml GRANTED www.hr.example.com /sys/acls/www.xml GRANTED * .hr.example.com /sys/acls/all.xml * .example.com / sys / acls /all.xml

例2

例如， SCOTT允许为www.hr.example.com进行域名解析：

SELECT host，acl，DECODE（DBMS_NETWORK_ACL_ADMIN.CHECK_PRIVILEGE_ACLID（aclid，'SCOTT'，'resolve'），1，'GRANTED'，0，'DENIED'，null）特权FROM（SELECT host，acl，aclid，DBMS_NETWORK_ACL_UTILITY.CONTAINS_HOST（ 'www.hr.example.com'，host）优先级从dba_network_acls WHERE lower_port IS NULL AND upper_port IS NULL）WHERE优先级> 0 ORDER BY优先级DESC; HOST ACL PRIVILEGE ---------------------- ------------------------- --- --------- www.hr.example.com /sys/acls/hr-www.xml GRANTED * .hr.example.com /sys/acls/hr-domain.xml * .example .com /sys/acls/corp-domain.xml

请注意，“resolve”权限仅在分配时没有任何端口范围的ACL中生效（当lower_port和upper_port为NULL ）。因此，该示例在查询中不包含lower_port和upper_port列。

20 DBVERIFY：脱机数据库验证实用程序

DBVERIFY是一个外部命令行实用程序，用于执行物理数据结构完整性检查。

DBVERIFY可用于脱机或联机数据库，也可用于备份文件。主要在需要确保备份数据库（或数据文件）在还原之前有效时，或者在遇到数据损坏问题时作为诊断帮助时，使用DBVERIFY。因为可以对脱机数据库运行DBVERIFY，所以完整性检查要快得多。

DBVERIFY检查仅限于缓存管理的块（即数据块）。因为DBVERIFY仅用于数据文件，所以它不适用于控制文件或重做日志。

DBVERIFY有两个命令行界面。使用第一个接口，可以指定单个数据文件的磁盘块以进行检查。使用第二个界面，您可以指定要检查的段。两个接口都使用dbv命令启动。以下部分提供了这些接口的说明：

2.1.1空对象和属性

值为NULL的对象称为atomically null。

原子空对象不同于对其所有属性具有空值的对象。

在具有空值的对象中，如果表列，对象属性，集合或集合元素已初始化为NULL或尚未初始化，则它可能为NULL 。通常，稍后将NULL值替换为实际值。当所有属性都为null时，您仍然可以更改这些属性并调用对象的子程序或方法。使用原子空对象，您可以不执行这些操作。

2.1.7 REF列和属性规则

可以通过使用约束来强制执行REF列和属性的规则。

在Oracle数据库中，可以使用SCOPE子句或引用约束子句来约束或约束REF列或属性。当REF列不受约束时，它可以将对象引用存储到相应对象类型的任何对象表中包含的行对象。

Oracle数据库无法确保存储在此类列中的对象引用指向有效和现有的行对象。因此， REF列可能包含不指向任何现有行对象的对象引用。这种REF值被称为悬空参考。

SCOPE约束可以应用于特定的对象表。存储在具有SCOPE约束的列中的所有REF值都指向SCOPE子句中指定的表的行对象。然而， REF值可能悬空。

可以使用类似于外键规范的REFERENTIAL约束来约束REF列。引用约束的规则适用于此类列。也就是说，存储在这些列中的对象引用必须指向指定对象表中的有效和现有行对象。

无法为REF列指定PRIMARY KEY约束。但是，您可以为此类列指定NOT NULL约束。

2.1.8名称解析

有几种方法可以解析Oracle数据库中的名称。

Oracle SQL允许您在某些关系操作中省略限定表名。

例2-7使用点表示法进行名称解析

在某些情况下，对象关系功能要求您指定表别名。

2.1.8.1何时需要表别名

可能需要表别名以避免解决引用的问题。

Oracle数据库要求您使用表别名来限定对子程序或对象属性的任何点符号引用，以避免内部捕获和解决引用的类似问题。

内部捕获是由使用非限定名称引起的情况。例如，如果向depts添加assignment列并忘记更改查询，Oracle Database会自动重新编译查询，以便内部SELECT使用depts表中的assignment列。

2.1.9使用远程数据库使用用户定义类型的限制

对象或用户定义的类型（具体地说，使用SQL CREATE TYPE语句声明的TYPE ，而不是在PL / SQL包中声明的类型）目前仅在单个数据库中有用。

Oracle数据库限制使用数据库链接，如下所示：

  • 您无法连接到远程数据库以在远程表上选择，插入或更新用户定义的类型或对象REF 。

    您可以将CREATE TYPE语句与可选关键字OID一起使用，以创建允许在多个数据库中使用对象类型的用户指定的对象标识符（OID）。请参阅Oracle Database Data Cartridge Developer's Guide中有关为对象类型分配OID的讨论。

  • 您不能在PL / SQL代码中使用数据库链接来声明远程用户定义类型的本地变量。

  • 您无法在PL / SQL远程过程调用中传达用户定义的类型参数或返回值。
