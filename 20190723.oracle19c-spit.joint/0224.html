24.1.1关于视图

视图是表或表组合的逻辑表示。实质上，视图是存储的查询。

视图从其所基于的表中获取其数据。这些表称为基表 。基表可能反过来是实际表，也可能是视图本身。在视图上执行的所有操作实际上都会影响视图的基表。您可以以与表格几乎相同的方式使用视图。您可以像查看标准表一样查询，更新，插入和删除视图。

视图可以提供驻留在其他表和视图中的数据的不同表示（例如子集或超集）。视图非常强大，因为它们允许您为不同类型的用户定制数据表示。

注意：

一种特殊类型的视图是版本视图，它仅用于支持使用基于版本的重新定义在线升级应用程序。管理视图的本节其余部分描述了除编辑视图之外的所有视图。有关版本视图和基于版本的重新定义的讨论，请参阅“ Oracle数据库开发指南” 。

24.1.2.1创建视图

您可以使用CREATE VIEW语句CREATE VIEW 。

要创建视图，您必须满足以下要求：

  • 要在架构中创建视图，您必须具有CREATE VIEW权限。要在另一个用户的模式中创建视图，您必须具有CREATE ANY VIEW系统特权。您可以显式或通过角色获取这些权限。

  • 必须已显式授予视图所有者（无论是您还是其他用户）访问视图定义中引用的所有对象的权限。所有者无法通过角色获得这些权限。此外，视图的功能取决于视图所有者的权限。例如，如果视图的所有者仅具有Scott的emp表的INSERT权限，则该视图只能用于将新行插入emp表，而不能用于SELECT ， UPDATE或DELETE行。

  • 如果视图的所有者打算将视图的访问权限授予其他用户，则所有者必须已使用GRANT OPTION或具有ADMIN OPTION的系统特权接收对基础对象的对象特权。

您可以使用CREATE VIEW语句CREATE VIEW 。每个视图都由引用表，物化视图或其他视图的查询定义。与所有子查询一样，定义视图的查询不能包含FOR UPDATE子句。

以下语句在hr.departments表中创建数据子集的视图：

CREATE VIEW departments_hq AS SELECT department_id，department_name，location_id FROM hr.departments WHERE location_id = 1700 WITH CHECK OPTION CONSTRAINT departments_hq_cnst;

定义departments_hq视图的查询仅引用位置1700中的行。此外， CHECK OPTION使用约束（名为departments_hq_cnst ）创建视图，以便针对视图发出的INSERT和UPDATE语句不会导致视图无法选择的行。例如，以下INSERT语句使用departments_hq视图成功地在departments表中插入一行，该视图包含位置为1700的所有行：

INSERT INTO departments_hq VALUES（300，'NETWORKING'，1700）;

但是，以下INSERT语句返回错误，因为它尝试为位置2700插入行，但无法使用departments_hq视图进行选择：

INSERT INTO departments_hq VALUES（301，'TRANSPORTATION'，2700）;

可以构造视图，指定WITH READ ONLY子句，该子句阻止通过视图对基表执行任何更新，插入或删除。如果未指定WITH子句，则视图具有一些限制，本质上是可更新的。

您还可以使用不可见列创建视图。例如，以下语句创建departments_hq_man视图并使manager_id列不可见：

CREATE VIEW departments_hq_man（department_id，department_name，manager_id INVISIBLE，location_id）AS SELECT department_id，department_name，manager_id，location_id FROM hr.departments WHERE location_id = 1700 WITH CHECK OPTION CONSTRAINT departments_hq_man_cnst;

24.1.2.3在视图创建时扩展定义查询

创建视图时，Oracle数据库会将顶级视图查询中的任何通配符（*）扩展为列列表。生成的查询存储在数据字典中;任何子查询都保持不变。

扩展列列表中的列名称用引号括起来，以说明最初使用引号输入基础对象的列的可能性，并要求它们使查询在语法上正确。

例如，假设dept视图创建如下：

CREATE VIEW dept AS SELECT * FROM scott.dept;

数据库将dept视图的定义查询存储为：

选择“DEPTNO”，“DNAME”，“LOC”来自scott.dept;

使用错误创建的视图不会扩展通配符。但是，如果最终编译视图时没有错误，则会扩展定义查询中的通配符。

24.1.2.4创建带错误的视图

如果CREATE VIEW语句中没有语法错误，则即使无法执行视图的定义查询，数据库也可以创建视图。在这种情况下，视图被视为“创建有错误”。

例如，当创建引用不存在的表或现有表的无效列的视图时，或者当视图所有者没有所需的权限时，仍可以创建视图并将其输入到数据字典中。但是，该视图尚不可用。

要创建包含错误的视图，必须包含CREATE VIEW语句的FORCE子句。 

创建力量视图...;

默认情况下，具有错误的视图将创建为INVALID 。当您尝试创建此类视图时，数据库将返回一条消息，指示视图是在创建时出错的。如果条件稍后更改以便可以执行无效视图的查询，则可以重新编译视图并使其有效（可用）。有关更改条件及其对视图的影响的信息，请参阅“ 管理对象依赖关系 ” 。

24.1.3替换视图

您可以通过删除并重新创建CREATE VIEW或发出包含OR REPLACE子句的CREATE VIEW语句来替换视图。

要替换视图，您必须具有删除和创建视图所需的所有权限。如果视图的定义必须更改，则必须替换视图;您不能使用ALTER VIEW语句来更改视图的定义。您可以通过以下方式替换视图：

  • 您可以删除并重新创建视图。

    注意：

    删除视图时，将从角色和用户撤消所有相应对象权限的授予。重新创建视图后，必须重新授予权限。

  • 您可以使用包含OR REPLACE子句的CREATE VIEW语句重新定义视图。OR REPLACE子句替换视图的当前定义并保留当前的安全授权。例如，假设您创建了如前所示的sales_staff视图，此外，您还为角色和其他用户授予了多个对象权限。但是，现在必须重新定义sales_staff视图以更改WHERE子句中指定的部门编号。您可以使用以下语句替换当前版本的sales_staff视图：

    创建或替换视图sales_staff AS SELECT empno，ename，deptno FROM emp WHERE deptno = 30 WITH CHECK OPTION CONSTRAINT sales_staff_cnst;

在替换视图之前，请考虑以下效果：

  • 替换视图将替换数据字典中的视图定义。视图引用的所有基础对象都不受影响。

  • 如果先前已定义CHECK OPTION的约束但未包含在新视图定义中，则会删除约束。

  • 依赖于替换视图的所有视图都变为无效（不可用）。此外，依赖的PL / SQL程序单元可能会变为无效，具体取决于新版本视图中的更改。例如，如果只有视图的WHERE子句发生更改，则依赖的PL / SQL程序单元仍然有效。但是，如果对视图列的数量或视图列名称或数据类型进行了任何更改，则相关的PL / SQL程序单元将失效。有关数据库如何管理此类依赖关系的更多信息，请参阅“ 管理对象依赖关系 ” 。

24.1.4在查询中使用视图

您可以查询视图。您还可以对视图执行数据操作语言（DML）操作，但有一些限制。

要对视图发出查询或INSERT, UPDATE或DELETE语句，您必须分别对视图具有SELECT, READ ， INSERT, UPDATE,或DELETE对象权限，无论是显式还是通过角色。

可以使用与表相同的方式查询视图。例如，要查询Division1_staff视图，请输入引用该视图的有效SELECT语句：

SELECT * FROM Division1_staff; ENAME EMPNO JOB DNAME ---------------------------------------------- -------- CLARK 7782经理会计师事务所7839总裁会计师事务所7934 CLERK会计师事务所7499销售人员销售人员7521销售员销售人员James 7900 CLERK SALES TURNER 7844销售人员销售马丁7654销售员销售额7698经理销售额

通过一些限制，可以使用视图将行插入，更新或从基表中删除。以下语句使用sales_staff视图在emp表中插入新行：

INSERT INTO sales_staff VALUES（7954，'OSTER'，30）;

对视图的DML操作的限制按列出的顺序使用以下条件：

  1. 如果视图由包含SET或DISTINCT运算符， GROUP BY子句或组函数的查询定义，则无法使用视图将行插入，更新或从基表中删除。
  2. 如果使用WITH CHECK OPTION定义视图，则如果视图无法从基表中选择行，则无法在基表（使用视图）中插入或更新行。
  3. 如果视图中省略了没有DEFAULT子句的NOT NULL列，则无法使用视图将行插入基表。
  4. 如果视图是使用表达式创建的，例如DECODE(deptno, 10, "SALES", ...) ，则无法使用视图将行插入或更新到基表中。

由sales_staff视图的WITH CHECK OPTION创建的约束仅允许将部门号为30的行插入到emp表中或在其中更新。或者，假设sales_staff视图由以下语句定义（即，排除deptno列）：

CREATE VIEW sales_staff AS SELECT empno，ename FROM emp WHERE deptno = 10 WITH CHECK OPTION CONSTRAINT sales_staff_cnst;

考虑此视图定义，您可以更新现有记录的empno或ename字段，但不能通过sales_staff视图将行插入emp表，因为该视图不允许您更改deptno字段。如果您在deptno字段上定义了DEFAULT值10，那么您可以执行插入操作。

当用户尝试引用无效视图时，数据库会向用户返回错误消息：

ORA-04063：视图'view_name'有错误

当视图存在但由于其查询中的错误而无法使用时会返回此错误消息（最初创建时是否有错误或者是否已成功创建但后来由于基础对象被更改或删除而变得无法使用）。

24.1.5.1更新加入视图

可更新的连接视图（也称为可修改的连接视图 ）是一个视图，它在SELECT语句的顶级FROM子句中包含多个表，并且不受WITH READ ONLY子句的限制。

可更新连接视图的规则如下表所示。据说满足这些标准的视图本质上是可更新的。

规则 描述

一般规则

连接视图上的任何INSERT ， UPDATE或DELETE操作一次只能修改一个基础基表。

UPDATE规则

连接视图的所有可更新列都必须映射到密钥保留表的列 。有关密钥保留表的讨论，请参阅“ 密钥保留表 ” 。如果WITH CHECK OPTION子句定义视图，则所有连接列和重复表的所有列都不可更新。

DELETE规则

只要连接中只有一个密钥保留表，就可以删除连接视图中的行。密钥保留表可以在FROM子句中重复。如果WITH CHECK OPTION子句定义视图并重复密钥保留表，则无法从视图中删除行。

INSERT规则

INSERT语句不能显式或隐式引用非密钥保留表的列 。如果WITH CHECK OPTION子句定义连接视图，则不允许INSERT语句。

有数据字典视图，指示连接视图中的列是否具有固有的可更新性。有关这些视图的说明，请参阅“ 使用UPDATABLE_ COLUMNS视图 ” 。

注意：

还有一些其他限制和条件会影响连接视图是否具有固有的可更新性。特定于Oracle数据库SQL语言参考中的CREATE VIEW语句的描述中列出。

如果视图本身不可更新，则可以通过在其上创建INSTEAD OF触发器来使其可更新。有关触发器的信息，请参见Oracle Database PL / SQL语言参考 。

此外，如果视图是其他嵌套视图上的连接，则其他嵌套视图必须可合并到顶级视图中。有关可合并和不可合并视图的讨论，以及更一般地说，优化程序如何优化引用视图的语句，请参阅“ Oracle数据库SQL调优指南”。

24.1.5.2密钥保留表

密钥保留表的概念是理解修改连接视图的限制的基础。如果表的每个键也可以是连接结果的键，则表是密钥保留的。因此，密钥保留表通过连接保留其密钥。

注意：

不必选择表的一个或多个键来进行密钥保存。如果选择了一个或多个键就足够了，那么它们也是连接结果的键。

表的密钥保留属性不依赖于表中的实际数据。相反，它是其架构的属性。例如，如果在emp表中每个部门中最多只有一个员工，那么deptno在emp和dept的连接结果中将是唯一的，但dept仍然不是一个保存密钥的表。

如果从emp_dept选择所有行，则结果为：

EMPNO ENAME DEPTNO DNAME LOC ---------- ---------- ------- -------------- ---- ------- 7782 CLARK 10会计纽约7839 KING 10会计纽约7934 MILLER 10会计纽约7369 SMITH 20研究DALLAS 7876 ADAMS 20研究DALLAS 7902 FORD 20研究DALLAS 7788 SCOTT 20研究DALLAS 7566 JONES 20研究DALLAS选择了8行。

在此视图中， emp是一个保留密钥的表，因为empno是emp表的一个键，也是连接结果的一个键。 dept 不是保存密钥的表，因为虽然deptno是dept表的一个键，但它不是连接的关键。

当序列被读入序列高速缓存时，生成序列值并将其存储在高速缓存条目中。然后可以快速访问这些值。

此CREATE SEQUENCE语句创建seq2序列，以便序列的50个值存储在SEQUENCE缓存中：

创建序列seq2 CACHE 50 ;

然后可以从高速缓存中读取seq2的前50个值。访问第51个值时，将从磁盘读取接下来的50个值。

为CACHE选择一个较高的值可以访问更多连续的序列号，从磁盘到序列缓存的读取次数更少。但是，如果存在实例故障，则缓存中的所有序列值都将丢失。如果事务在导出运行时继续访问序列号，则导出和导入后也可以跳过缓存的序列号。

如果在CREATE SEQUENCE语句中使用NOCACHE选项，则SEQUENCE的值不会存储在序列高速缓存中。在这种情况下，每次访问序列都需要磁盘读取。此类磁盘读取对序列的访问速度较慢。此CREATE SEQUENCE语句创建SEQ3序列，以便其值永远不会存储在缓存中：

创建序列seq3 NOCACHE;

24.2.4.3使序列可扩展

通过在CREATE SEQUENCE或ALTER SEQUENCE语句中指定SCALE子句，可以使序列可伸缩 。

27.1.2.1数据库角色和ACL将权限映射到用户

数据库角色将权限映射到用户。在细粒度访问控制的上下文中，ACL将权限映射到用户，而角色只是一组用户。

授予数据库角色权限，就像可以授予数据库用户权限一样。数据库角色充当将数据库特权映射到数据库用户（和应用程序）的中介：角色被授予特权，然后该角色被授予用户（赋予他们特权）。

在数据库角色的概念中，一组用户与授予这些用户的一组权限之间的界限有点模糊：该角色可用于对映射到用户的权限进行分组，并将用户分组到权限已映射。映射是通过定义角色并将其授予用户来完成的，而传统的数据库术语认为角色与授予它的权限集是一样的。

在细粒度访问控制的上下文中，使用不同的机制（ 访问控制列表 （ACL））作为将权限映射到用户的中介。角色只是一组用户。在此上下文中，将权限与用户和角色相关联的行为不是数据库授权。它是一个声明性ACL条目，以及ACL的运行时评估和ACL冲突的解决方案。

27.1.3特权：许可

特权是可以授予或拒绝委托人的特定权利或许可。特权是聚合或原子。聚合特权包括其他特权;原子特权没有。

  • 聚合权限 - 包含其他权限的权限。

  • 原子权限 - 不包含其他权限的权限。它无法细分。

当权限数量变大时，聚合权限简化了可用性，并且它们促进了ACL客户端之间的互操作性。

聚合特权保留其标识：它们不会分解为相应的原子（叶）特权。在WebDAV术语中，Oracle数据库聚合特权不是抽象的。这意味着聚合特权充当一组指向其组件特权的指针，而不是这些组件的副本。因此，即使组件的定义发生更改，聚合权限也始终是最新的。

授予主体的权限集控制该主体是否可以对其保护的数据执行给定操作。例如，如果主体（数据库用户） HR想要对给定资源执行read操作，则必须在读取操作之前将主权HR授予读取权限。

27.4 ACL和ACE

ACL用于保护资源，在Oracle数据库中，资源是Oracle XML DB Repository中的资源（文件和文件夹）。ACL由ACE组成。

访问控制列表（ACL）是一种标准安全机制，在某些语言（如Java）和某些操作系统（如Microsoft Windows）中使用。ACL也是WebDAV标准的一部分。

可以使用WebDAV访问存储库资源，其保护ACL充当WebDAV ACL。每个存储库资源都受某些ACL的保护。无论如何通过WebDAV，SQL或任何其他方式访问资源，都会强制执行保护资源的ACL。

在Oracle XML DB Repository中创建新资源时，默认情况下，其父文件夹上的ACL用于保护资源。创建资源后，可以在其上设置新的ACL。

Oracle数据库中的ACL是根据Oracle数据库ACL XML模式验证的XML文档，该模式位于/sys/schemas/PUBLIC/xmlns.oracle.com/xdb/acl.xsd中的Oracle XML DB存储库中。ACL本身作为资源库中的资源进行存储和管理。

在主体对受ACL保护的数据执行操作之前，将检查受保护数据的用户权限。检查的权限集取决于要执行的操作。

聚合权限由其他权限组成。存储ACL时，它所引用的聚合权限充当其组件权限的指针集。

所有ACL都存储在表XDB$ACL ，该表由数据库模式（用户帐户） XDB 。这是一个基于XML模式的XMLType表。此表中的每一行（以及每个ACL）都有一个系统生成的对象标识符（ OID ），可以作为名为OBJECT_ID的列进行访问。

每个Oracle XML DB资源库资源都有一个名为ACLOID的属性。 ACLOID存储保护资源的ACL的OID。ACL本身就是一种资源，ACL的XMLRef属性（例如/sys/acls/all_all_acl.xml ）是表XDB$ACL中包含XDB$ACL内容的行的REF 。这两个属性构成了存储Oracle XML DB资源的表XDB$RESOURCE与表XDB$ACL之间的链接。

27.4.2 ACL和ACE评估

在允许主体访问受ACL保护的存储库资源之前，将检查权限。通过按顺序评估该主体的保护ACL来完成此检查。对于每个此类ACL，将按顺序检查其中适用于主体的ACE。

如果一个ACE向当前用户授予某个特权，而另一个ACE拒绝该用户的特权，则会产生冲突。有两种可能的方法来管理同一主体的ACE之间的冲突。

  • 默认行为（称为ace-order ）是仅使用针对给定主体发生的第一个 ACE。该委托人的其他ACE无效。在这种情况下， ACE订单是相关的 。

  • 但是，您可以将数据库配置为使用替代行为deny-trumps-grant 。在这种情况下，任何对给定委托人具有子deny ACE都会deny该委托人的许可，无论该委托人是否有其他具有grant子女的ACE。在这种情况下， deny始终优先于grant ，而ACE订单则无关紧要 。

您可以通过在配置文件xdbconfig.xml配置参数acl-evaluation-method设置为ace-order或deny-trumps-grant来配置ACL评估行为。默认配置文件指定ace-method ，但是在没有给出acl-evaluation-method时使用的元素acl-evaluation-method的默认值是deny-trumps-grant 。

注意：

在Oracle Database 11g第1版之前的版本中，只有一种ACL评估行为可用： deny-trumps-grant （尽管未在配置文件中指定）。

使用ace-order作为默认行为的更改对数据库版本之间的升级和降级具有重要影响。请参阅升级或降级现有Oracle XML DB安装 。

27.4.3 ACL验证

创建ACL时，将根据ACL的XML架构验证ACL，并运行一些正确性测试，例如确保ACE的开始和结束日期按时间顺序排列。在ACL之间的关系的ACL创建时间没有完全检查。

这种对ACL正确性的完整检查称为ACL有效性检查，但不能与其XML 模式有效性混淆。要使ACL有效（作为ACL），它还必须是XML模式有效的，但反之则不成立。

只要评估ACL以检查主体是否具有某些操作的适当权限，就会在运行时进行完整的ACL有效性检查。如果此检查发现ACL无效，则ACL将授予的所有权限将被拒绝给指定的主体。

27.6 ACL缓存

由于ACL会检查每次访问它们保护的数据，因此ACL检查操作的性能对此类数据（包括Oracle XML DB存储库资源）的性能至关重要。在Oracle XML DB中，通过使用多个缓存来实现此存储库操作所需的性能。

ACL保存在数据库实例中所有会话共享的缓存中。更新ACL时，其在缓存中的条目以及依赖于它的所有对象都将失效。下次使用ACL时，会将其新副本带入缓存。Oracle建议您尽可能在资源之间共享ACL。

给定ACL授予给定用户特权的会话特定缓存。此高速缓存中的条目具有由Oracle XML DB配置文件（ xdbconfig.xml ）中的元素<acl-max-age>指定的超时（以秒为单位）。为获得最佳性能，请将此超时设置为尽可能大。但这里有一个权衡：超时越大，当前会话获取更新的ACL所需的时间越长。

在使用具有LDAP主体（LDAP组或用户）的ACL时，Oracle XML DB还维护缓存以提高性能。这些缓存的目标是最小化与LDAP服务器的网络通信。一个是共享缓存，它将LDAP GUID映射到相应的LDAP昵称和专有名称（DN）。在显示ACL文档（或从XMLType实例转换为CLOB或VARCHAR 2值）时使用此方法。要清除此缓存，请使用过程DBMS_XDBZ.purgeLDAPCache 。另一个缓存是特定于会话的，并将LDAP组映射到其成员（嵌套成员资格）。每当Oracle XML DB第一次遇到LDAP组时（在会话中），它就会从LDAP服务器获取该组的嵌套成员资格。因此，最好使用尽可能少的成员和嵌套级别的组。

27.7存储库资源和数据库表安全性

通过在用于存储它们的表上启用层次结构，提供了用于访问基于REF的存储库资源的统一安全机制。当特定资源不需要基于ACL的安全性时，可以通过使用PL / SQL过程DBMS_XDBZ.disable_hierarchy来关闭ACL检查来优化其访问。

Oracle XML DB Repository中的资源有两种类型：

  • 基于LOB（内容存储在作为资源一部分的LOB中）。访问权限仅由保护资源的ACL确定。

  • 基于REF （内容是XML数据并存储在数据库表中）。除了资源的ACL权限之外，用户还必须具有存储XML内容的基础表或视图的相应权限。

由于基于REF的资源的内容可以存储在表中，因此可以使用表上的SQL查询直接访问此数据。统一访问控制机制是访问所需的权限独立于访问方法（例如，FTP，HTTP或SQL）的机制。要使用ACL提供统一的安全机制，在引用表中的行的资源插入Oracle XML DB之前，必须先使基础表启用层次结构 。

XML架构注册生成的默认表是启用层次结构的。使用Oracle XML DB注册XML架构时，启用层次结构是默认行为。您还可以使用过程DBMS_XDBZ.enable_hierarchy在注册后启用层次结构。

在资源表上启用层次结构执行以下操作：

  • 添加两个隐藏列以存储ACLOID和引用表中行的资源的OWNER 。

  • 向表中添加行级安全性（RLS）策略，每当对表执行SELECT ， UPDATE或DELETE操作时，该策略都会检查ACL。

  • 创建一个名为path-index trigger的数据库触发器，该触发器确保每当在存储内容的XMLType表中更新相应的行时，都会更新资源的最后修改信息。

27.8 Oracle XML DB与LDAP的集成

您可以允许轻量级目录访问协议（LDAP）用户使用Oracle XML DB的功能，包括ACL。

典型方案是单个共享数据库模式（用户），多个LDAP用户映射到该模式。此映射在Oracle Internet Directory中维护。最终用户可以使用LDAP用户名和密码登录数据库。然后，它们会自动映射到相应的共享数据库架构。（用户可以使用SQL或任何受支持的Oracle XML DB协议登录。）隐式ACL解析基于当前LDAP用户和相应的LDAP组成员身份信息。

在将Oracle用户和组（也称为LDAP角色）用作Oracle XML DB ACL中的主体之前，必须满足以下先决条件：

  • 必须设置Oracle Internet Directory，并且必须向其注册数据库。

  • 必须在数据库和Oracle Internet Directory之间设置SSL身份验证。

6.1.1关于控制文件

控制文件跟踪数据库的物理组件。它是数据库用于查找数据库使用的所有其他文件的根文件。由于控制文件的重要性，Oracle建议对控制文件进行多路复用 ，或者具有多个相同的副本。 对于使用Oracle Database Configuration Assistant（DBCA）创建的数据库，将自动创建控制文件的两个副本并使其保持同步。

如果任何控制文件失败，则您的数据库将变为不可用。但是，如果您有控制文件副本，则可以关闭数据库并从副本重新创建失败的控制文件，然后重新启动数据库。另一个选项是从CONTROL_FILES初始化参数中删除失败的控制文件，并使用其余的控制文件重新启动数据库。

6.1.2关于联机重做日志文件

每个Oracle数据库都有一组两个或多个联机重做日志文件。联机重做日志文件集合统称为数据库的重做日志 。 重做日志由重做条目组成，也称为重做记录 。

联机重做日志存储对数据所做更改的副本。如果故障需要从备份恢复数据文件，则可以从联机重做日志文件中获取已还原数据文件中缺少的最新数据更改，因此工作永远不会丢失。联机重做日志文件用于在硬件，软件或介质发生故障后恢复数据库。为了防止涉及联机重做日志文件本身的故障，Oracle数据库可以复用联机重做日志文件，以便可以在不同磁盘上维护两个或多个相同的联机重做日志文件副本。

数据库的联机重做日志包含一组联机重做日志文件。组由在线重做日志文件及其多路复用副本组成。每个相同的副本都被视为该组的成员。每个组由一个数字定义，例如组1。

图6-2显示了具有三个联机重做日志组和每个组中有两个成员的数据库的配置。对于每个组，成员存储在单独的磁盘上以获得最大可用性。例如，组1的成员是重做日志文件A_LOG1和B_LOG1 。

6.1.4关于回滚段

回滚段是用于在早期版本的Oracle数据库中跟踪数据库的撤消信息的数据库结构。现在，管理撤消信息的首选方法是使用撤消表空间。 有关更多信息，请参阅“ 管理撤消数据 ” 。

注意：

Oracle数据库使用SYSTEM回滚段来执行系统事务。它是在创建数据库时自动创建的，并且在实例启动时始终处于联机状态。它位于SYSTEM表空间中。您不需要执行任何操作来管理SYSTEM回滚段。

6.1.5关于数据文件

数据文件是存储数据库中数据的操作系统文件。数据以Oracle专有格式写入这些文件，其他程序无法读取。 Tempfiles是一类特殊的数据文件，仅与临时表空间相关联。

数据文件可以分解为以下组件：

  • 分割

    段包含特定类型的数据库对象。例如，表存储在表段中，索引存储在索引段中。数据文件可以包含许多段。

  • 程度

    范围是段内的连续数据块集。Oracle数据库以一个范围为单位为段分配空间。当段的现有扩展区已满时，数据库将为该段分配另一个扩展区。

  • 数据块

    数据块 （也称为数据库块 ）是数据库存储的最小I / O单位。范围由几个连续的数据块组成。数据库在创建数据库时使用默认块大小。

    创建数据库后，如果不重新创建数据库，则无法更改默认块大小。但是，可以创建块大小不同于默认块大小的表空间。

段，范围和数据块都是逻辑结构。只有Oracle数据库可以确定文件中有多少数据块。操作系统仅识别文件和操作系统块，而不识别Oracle数据库文件中的数据块数。每个数据块映射到一个或多个操作系统块。

6.1.6关于表空间

数据库分为称为表空间的逻辑存储单元，它将相关的逻辑结构（例如表，视图和其他数据库对象）分组。例如，可以将所有应用程序对象分组到单个表空间中，以简化维护操作。

表空间由一个或多个物理数据文件组成。分配给表空间的数据库对象存储在该表空间的物理数据文件中。

创建Oracle数据库时，已存在一些表空间，例如SYSTEM和SYSAUX 。

表空间提供了一种在存储上物理定位数据的方法。定义包含表空间的数据文件时，请指定这些文件的存储位置。例如，您可以将某个表空间的数据文件位置指定为指定的主机目录（表示某个磁盘卷）或指定的Oracle自动存储管理磁盘组。然后，分配给该表空间的任何模式对象都将位于指定的存储位置。表空间还提供备份和恢复单元。Oracle数据库的备份和恢复功能使您可以在表空间级别进行备份或恢复。

6.1.6.1本地管理的表空间
表空间内的空间管理涉及跟踪可用（空闲）和已用空间，以便在数据插入和删除期间有效分配空间。本地管理的表空间将空间分配信息保留在表空间内，而不是数据字典中，从而提供更好的性能。 默认情况下，Oracle数据库使用自动段管理将所有新创建的表空间设置为本地管理，这一功能可进一步提高性能。
6.1.6.2表空间类型
表空间有三种类型。 例如：

  • 常驻

    您使用永久表空间来存储您的用户和应用程序数据。Oracle数据库使用永久表空间来存储永久数据，例如系统数据。为每个用户分配一个默认的永久表空间。

  • 解开

    以自动撤消管理模式运行的数据库透明地创建和管理撤消表空间中的撤消数据。Oracle数据库使用撤消数据回滚事务，提供读取一致性，帮助恢复数据库，以及启用Oracle闪回查询等功能。数据库实例只能有一个活动的撤消表空间。

  • 临时

    临时表空间用于存储临时数据，这将在SQL语句执行排序操作时创建。创建数据库时，Oracle数据库会获取临时表空间。如果要创建临时表空间组，则可以创建另一个临时表空间。在典型情况下，您不必创建其他临时表空间。如果您有一个非常大的数据库，那么您可以配置其他临时表空间。

    组成临时表空间的物理文件称为tempfiles，而不是数据文件。

    TEMP表空间通常用作未明确分配临时表空间的用户的默认临时表空间。

    用户可以在创建表空间后对其进行读写。这是默认值。

  • 只读

    如果创建表空间为只读，则在将状态更改为“读写”之前，无法写入表空间。您不太可能创建只读表空间，但在向其写入不需要修改的数据后，可能会将其更改为该状态。

  • 离线

    如果表空间的状态为“脱机”，则没有用户可以访问它。在对与该表空间关联的数据文件执行维护或恢复之前，您可以将表空间的状态更改为“脱机”。

6.1.6.4自动扩展表空间
您可以将表空间设置为在达到其大小限制时自动将其自身扩展指定的数量。如果未启用autoextend，则会在表空间达到其临界或警告阈值大小时收到警报。 关键和警告阈值参数具有您可以随时更改的默认值。这些参数还会导致为自动扩展表空间生成警报，这些表空间接近其指定的大小限制。您可以通过手动增加表空间大小来响应大小警报。您可以通过增加一个或多个表空间数据文件的大小或通过向表空间添加另一个数据文件来实现此目的。

注意：

虽然将表空间称为自动扩展是很常见的，但自动扩展是数据文件属性，而不是表空间属性。也就是说，当您创建包含表空间的数据文件时，您可以指示这些数据文件是否自动扩展。具有自动扩展数据文件的表空间被视为自动扩展表空间。您可以为自动扩展数据文件指定最大大小。

6.1.6.5加密表空间
加密表空间主要通过数据库以外的方式保护您的数据免受未经授权的访问。 例如，当加密的表空间写入备份介质以便从一个Oracle数据库传输到另一个Oracle数据库或者前往场外设施进行存储时，它们仍然是加密的。此外，加密表空间可保护试图绕过数据库安全功能并直接通过操作系统文件系统访问数据库文件的用户的数据。

您可以加密任何永久表空间以保护敏感数据。加密表空间时，所有表空间块都会加密。支持加密所有段类型，包括表，集群，索引，LOB，表和索引分区等。表空间加密对您的应用程序完全透明，因此无需修改应用程序。

6.5.1关于撤消数据

当事务修改数据时，Oracle数据库会在修改原始数据之前复制它。修改后的数据的原始副本称为撤消数据 。 由于以下原因，保存此信息是必要的：

  • 如果需要回滚，则撤消对数据库所做的任何未提交的更改。可能需要回滚，因为用户想要撤消误导或无意的事务的更改，或者它可以是恢复操作的一部分。

  • 提供读取一致性 ，这意味着每个用户都可以获得一致的数据视图，即使可能正在对数据进行其他更改。使用读取一致性，用户会话不会看到在其他用户会话中进行的未提交的更改（有时称为脏读 ）。例如，如果用户在上午10:00发出查询并且查询持续15分钟，则查询结果将在上午10:00反映数据的整个状态，而不管其他用户在执行更新或插入操作之后查询已开始。

  • 启用某些Oracle闪回功能，例如Oracle闪回查询和Oracle闪回表，使您可以查看或恢复以前某个时间点的数据。

撤消表空间

通过自动撤消管理，撤消数据存储在撤消表空间。撤消表空间具有超出永久表空间的其他属性。可以有多个撤消表空间，但只有一个可以为Oracle实例激活。

使用Database Configuration Assistant（DBCA）创建数据库时，它会创建一个名为UNDOTBS1的自动扩展撤消表空间，最大扩展名为32,767 MB。

撤消保留

Oracle数据库自动确保在提交事务之前，永远不会覆盖活动事务正在使用的撤消数据。提交事务后，可以重用或覆盖该撤消数据占用的空间。在这种情况下，如果撤消表空间中的空间变得稀缺，则可以覆盖撤消数据。

即使在提交了事务之后，保留（不覆盖）其撤消数据，确保Oracle闪回功能的成功以及长时间运行的查询的读取一致性也很有用。为此，数据库维护并自动调整撤消保留期 。保留年龄小于撤消保留期的已承诺撤消数据，供查询或Oracle闪回操作使用。

格式模型

格式模型是一个字符文字，用于描述存储在字符串中的日期时间或数字数据的格式。格式模型不会更改数据库中值的内部表示形式。将字符串转换为日期或数字时，格式模型将确定Oracle数据库如何解释字符串。在SQL语句中，您可以使用格式模型作为TO_CHAR和TO_DATE函数的参数来指定：

  • Oracle用于从数据库返回值的格式

  • 您为Oracle指定要存储在数据库中的值的格式

  • 字符串HH24:MI:SS 17:45:29 '的日期时间格式模型是' HH24:MI:SS '。

RR日期时间格式元素

RR datetime格式元素类似于YY日期时间格式元素，但它为在其他世纪存储日期值提供了额外的灵活性。RR日期时间格式元素允许您通过仅指定年份的最后两位数来存储21世纪的20世纪日期。

如果将TO_DATE函数与YY日期时间格式元素一起使用，则返回的年份始终与当前年份的前2位数字相同。如果使用RR datetime格式元素，则返回值的世纪根据指定的两位数年份和当前年份的最后两位数而变化。

那是：

  • 如果指定的两位数年份是00到49，那么

    • 如果当前年份的最后两位数字是00到49，那么返回的年份与当前年份的前两位数字相同。

    • 如果当前年份的最后两位数字是50到99，那么返回年份的前2位数字比当前年份的前2位数字大1。

  • 如果指定的两位数年份是50到99，那么

    • 如果当前年份的最后两位数字是00到49，那么返回年份的前2位数字比当前年份的前2位数字少1。

    • 如果当前年份的最后两位数字是50到99，那么返回的年份与当前年份的前两位数字相同。

140.1 DBMS_RLS概述

支持细粒度访问控制的功能基于动态谓词，其中安全规则未嵌入到视图中，但是在DML语句中引用基表或视图时，在语句分析时获取。

表/视图或同义词的动态谓词由PL / SQL函数生成，该函数通过PL / SQL接口与安全策略相关联。例如：

DBMS_RLS.ADD_POLICY（'hr'，'employees'，'emp_policy'，'hr'，'emp_sec'，'select'，'user_ctx'，'time'）;

每当在HR模式下的EMPLOYEES表在查询或子查询（ SELECT ）中引用时，服务器就会调用EMP_SEC函数（在HR模式下）。此函数返回特定于EMP_POLICY策略的当前模式的EMP_POLICY 。策略函数可以基于函数调用期间可用的会话环境变量生成谓词。这些变量通常以应用程序上下文的形式出现。策略可以指定安全相关列和这些语句类型的任意组合： INDEX ， SELECT ， INSERT ， UPDATE或DELETE 。

然后，服务器生成一个包含文本的瞬态视图：

SELECT * FROM hr.employees WHERE P1

这里， P1 （例如， SAL > 10000，甚至是子查询）是从EMP_SEC函数返回的谓词。服务器将EMPLOYEES表视为视图，并像普通视图一样进行视图扩展，但视图文本是从瞬态视图而不是数据字典中获取的。

如果谓词包含子查询，则策略函数的所有者（定义者）用于解析子查询中的对象并检查这些对象的安全性。换句话说，对受策略保护的对象具有访问权限的用户不需要了解有关策略的任何信息。不需要为任何基础安全策略授予对象权限。此外，用户不需要对策略函数EXECUTE特权，因为服务器使用函数definer的权限进行调用。

注意：

瞬态视图可以保留父对象的可更新性，因为它是从仅具有谓词的单个表或视图派生的;也就是说，没有JOIN ， ORDER BY ， GROUP BY等等。

DBMS_RLS还提供了删除或启用安全策略的接口。例如，您可以使用以下PL / SQL语句删除或启用EMP_POLICY ：

DBMS_RLS.DROP_POLICY（'hr'，'employees'，'emp_policy'）; DBMS_RLS.ENABLE_POLICY（'hr'，'employees'，'emp_policy'，TRUE）;

140.2 DBMS_RLS安全模型

使用子查询创建瞬态视图时，将执行安全检查。拥有策略函数的模式（生成动态谓词）是瞬态视图定义的安全检查和对象查找。
