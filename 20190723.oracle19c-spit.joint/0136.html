14.1 LOB数据类型

在Oracle Database 10g之前，LOB的最大大小为2 ^ 32字节。自Oracle Database 10g以来，此限制已被删除，并且最大大小限制为可用物理存储的大小。

Oracle数据库支持以下四种LOB数据类型：

  • 二进制大对象（BLOB）

    此数据类型用于非结构化二进制数据。

  • 字符大对象（CLOB）

    此数据类型用于字符数据。

  • 国字大物（NCLOB）

    此数据类型用于国家字符数据。

  • BFILE

    此数据类型用于存储在数据库表空间之外的操作系统文件中的大型二进制数据对象。

BLOB，CLOB和NCLOB持久存储在数据库表空间中，对这些数据类型执行的所有操作都在事务控制之下。

BFILE是Oracle专有数据类型，它提供对位于三级存储设备（如硬盘，网络安装文件系统，CD-ROM，PhotoCD和DVD）上的数据库表空间外部的数据的只读访问。BFILE数据不受事务控制，不由数据库备份存储。

PL / SQL语言支持LOB数据类型，JDBC接口允许将IN参数传递给PL / SQL过程或函数，以及检索OUT参数或返回。PL / SQL对包括LOB在内的所有数据类型使用值语义，但仅对BFILE使用引用语义。

14.2 Oracle SecureFiles

Oracle Database 11g第1版（11.1）引入了Oracle SecureFiles，这是一个全新的LOB存储。

以下功能可通过现有API透明地将JDBC SecureFiles用于JDBC程序：

  • SecureFile压缩使用户能够压缩数据以节省磁盘空间。

  • SecureFile加密引入了一种新的加密工具，允许随机读取和写入加密数据。

  • 重复数据删除使Oracle数据库能够自动检测重复的LOB数据，并通过仅存储一个数据副本来节省空间。

  • LOB数据路径优化包括存储层上方的逻辑高速缓存和新的高速缓存模式。

  • 高性能空间管理。

setLobOptions和getLobOptions API在PL / SQL包和类型参考中描述，可以通过可调用语句从JDBC访问。

以下Oracle SecureFiles功能通过对现有API的更新在数据库中实现：

14.3.1简化机制

14.3.4 CallableSatement和IN OUT参数

作为PL / SQL要求，用作IN OUT参数的输入和输出的Java类型必须相同。由本章中描述的扩展完成的类型的自动切换可能会导致此问题。

请考虑您具有存储过程的IN OUT CLOB参数，并且您希望使用setString方法来设置此参数的值。对于任何IN和OUT参数，绑定必须是相同类型。除非您确定数据大小，否则自动切换输入模式将导致问题。例如，如果已知输入和输出数据都不会大于32766字节，则可以使用setString方法作为输入参数，并将OUT参数注册为Types.VARCHAR并使用getString方法作为输出参数。

更好的解决方案是将存储过程更改为具有单独的IN和OUT参数。也就是说，如果你有：

CREATE PROCEDURE clob_proc（c IN OUT CLOB）;

然后，将其更改为：

CREATE PROCEDURE clob_proc（c_in IN CLOB，c_out OUT CLOB）;

另一种解决方法是使用容器块来进行调用。clob_proc过程可以用Java String包装以用于prepareCall语句，如下所示：

“DECLARE c_temp; BEGIN c_temp：=？; clob_proc（c_temp）; ？ ：= c_temp;结束;”

在任何一种情况下，您都可以对第一个参数使用setString方法，对Types.使用registerOutParameter方法Types.CLOB就秒了。

14.5关于使用临时LOB

您可以使用临时LOB来存储瞬态数据。数据存储在临时表空间而不是常规表空间中。在不再需要它们之后，您应该释放临时LOB。如果不这样做，那么LOB在临时表空间中占用的空间将不会被回收。

您可以将临时LOB插入表中。执行此操作时，将创建并存储LOB的永久副本。

注意：

在某些情况下，插入临时LOB可能更为可取。例如，当LOB数据相对较小并且复制数据的开销小于数据库往返检索空定位符的成本时。请记住，数据最初存储在服务器上的临时表空间中，然后移动到永久存储中。

使用static方法createTemporary创建临时LOB，在oracle.sql.定义oracle.sql.BLOB和oracle.sql.CLOB类。您使用freeTemporary方法释放临时LOB。

您还可以使用JDBC 4.0中提供的连接工厂方法创建临时LOB / CLOB或NCLOB。

您可以通过调用isTemporary方法来测试LOB是否是临时的。如果通过调用createTemporary方法创建LOB，则isTemporary方法返回true ，否则返回false 。

您可以通过调用freeTemporary方法释放临时LOB。在结束会话或致电之前释放任何临时LOB。

注意：

  • 如果您没有释放临时LOB，那么它将使该数据库中的LOB使用的存储不可用。经常无法释放临时LOB将导致填充具有不可用LOB存储的临时表空间。

  • 当从获取数据ReultSet与是临时的LOB的列，使用getClob或getBlob方法代替getString或getBytes 。

  • JDBC 4.0方法free ，存在于java.sql.Blob ， java.sql.Clob和java.sql.NClob接口取代了freeTemporary方法。

14.6关于使用Open和Close方法打开持久LOB

本节讨论如何打开和关闭LOB。使用oracle.sql.的以下方法提供此功能的JDBC实现oracle.sql.BLOB和oracle.sql.CLOB接口：

  • void open（int模式）

  • void close（）

  • boolean isOpen（）

注意：

  • 从Oracle Database 12 c第1版（12.1）开始， oracle.sql.BLOB和oracle.sql.不推荐使用CLOB类，并将其替换为oracle.jdbc.OracleBlob和oracle.jdbc.OracleClob接口。Oracle建议您尽可能使用java.sql包中提供的方法，以java.sql Oracle特定扩展的oracle.jdbc包中提供的标准兼容性和方法。有关这些接口的更多信息，请参阅MoS Note 1364193.1。

  • 您无需打开和关闭LOB。出于性能原因，您可以选择打开和关闭它们。

如果不在Open/Close调用操作中包装LOB操作，则对LOB的每次修改都将隐式打开和关闭LOB，从而触发域索引上的任何触发器。请注意，在这种情况下，只要进行LOB修改，LOB上的任何域索引都将更新。因此，域LOB索引始终有效，并且可以在同一事务中的任何时间使用。

如果将LOB操作包装在Open/Close调用操作中，则不会为每个LOB修改触发触发器。相反，域索引上的触发器将在Close调用时触发。例如，您可以设计应用程序，以便在调用close方法之前不会更新域索引。但是，这意味着LOB上的任何域索引在Open/Close调用之间都无效。

您通过调用open或open(int)方法打开LOB。然后，您可以在没有与该LOB触发相关联的任何触发器的情况下读取和写入LOB。完成访问LOB后，通过调用close方法关闭LOB。当您关闭LOB时，将触发与LOB关联的任何触发器。

您可以通过调用isOpen方法来检查LOB是打开还是关闭。如果通过调用open(int)方法打开LOB，则参数的值必须是MODE_READONLY或MODE_READWRITE ，如oracle.sql.所定义oracle.sql.BLOB和oracle.sql.CLOB类。如果使用MODE_READONLY打开LOB，则任何写入LOB的尝试都将导致SQL异常。

注意：

  • 如果在关闭事务打开的所有LOB之前提交事务，则会发生错误。打开LOB的开放性被丢弃，但事务已成功提交。因此，提交了对事务中的LOB和非LOB数据所做的所有更改，但域索引的触发器不是固定的。

  • open和close方法仅适用于持久性LOB。close方法与用于临时LOB的free或freeTemporary方法不同。free和freeTemporary方法释放存储并使LOB无法使用。另一方面， close方法向数据库指示对LOB的修改已完成，并且应触发触发器并更新索引。在调用close方法后，LOB仍然可用。

1.1 OLAP Java API概述

Oracle OLAP Java API是一个应用程序编程接口，可通过OLAP选项访问Oracle数据库中的联机分析处理（OLAP）技术。本主题列出了OLAP Java API客户端应用程序可以执行的操作，描述了OLAP Java API中的类，描述了维度数据模型中的对象，并讨论了用于联机分析处理的组织数据。

1.1.1 OLAP Java API可以做什么

使用OLAP Java API，您可以开发执行以下操作的客户端应用程序。

  • 在与Oracle数据库实例的JDBC连接中建立一个或多个用户会话。多个用户会话可以共享相同的连接和相同的元数据对象缓存。

  • 使用数据库管理OLAP事务。

  • 使用OLAP元数据对象实现维度数据模型。

  • 创建和维护分析工作区。

  • 创建逻辑元数据对象并将它们映射到关系源。

  • 将元数据对象部署为分析工作区或关系表和视图，并将对象提交到数据库。

  • 浏览元数据以发现可供查看或分析的数据。

  • 构造多维数据的分析查询。使最终用户能够根据用户的需要创建指定和操作数据的查询（例如，选择，聚合和计算数据）。

  • 当应用程序用户优化其分析时，修改查询，而不是完全重新定义它们。

  • 检索结构化以便以多维格式显示的查询结果。

1.1.3描述维度数据模型

数据仓库和OLAP应用程序基于数据的多维视图。该视图在包括以下维对象的维数据模型中实现。

多维数据集

多维数据集是具有相同维度集的度量的容器。多维数据集通常对应于单个关系事实表或视图。立方体的度量包含事实，维度使事实数据成形。通常，尺寸形成立方体的边缘，测量数据是立方体的主体。例如，您可以组织销售到多维数据集中的产品单位的数据，该多维数据集的边包含来自时间，产品，客户和渠道维度的成员的值，其主体包含来自销售单位数量的度量值和销售额度量的值。

OLAP Java API中的元数据对象不表示多维数据集边缘的OLAP概念，但边缘通常包含在使用OLAP Java API的应用程序的设计中。每条边包含一个或多个维度的成员值。尽管对立方体上的边数没有限制，但是数据通常被组织用于沿三条边的显示目的，这三条边被称为行边，列边和页边。

措施

度量包含多维数据集中的事实数据。度量值按维度进行组织和标识。措施通常是多方面的。每个度量值由一组唯一的维度成员标识。这组维度成员称为元组 。

尺寸

维度包含唯一值列表，用于标识度量中的数据并对其进行分类。常用尺寸是客户，产品和时间。通常，维度具有一个或多个层次结构，这些层次结构将维度成员组织为父子关系。

通过指定要对数据执行的维度成员，度量和计算，最终用户可以制定业务问题并获取其查询的答案。例如，使用按月对数据进行分类的时间维度，按单位项目对数据进行分类的产品维度，以及包含按月销售的产品单位数量的数据的度量，您可以制定一个查询，询问是否销售产品单位在1月或6月份较高。

层次结构

层次结构是维度的组件，用于将维度成员组织为父子关系。通常，在客户端应用程序的用户界面中，最终用户可以通过在父母和孩子之间向下钻取或向上钻取来扩展或折叠层次结构。父维度成员的度量值是子值的聚合。

维度可以具有多个层次结构。例如，时间维度可以具有日历年度层次结构和会计年度层次结构。层次结构可以是基于级别的，也可以是基于值的。

在基于级别的层次结构中，父级必须处于比该父级子级更高级别的级别。在多维数据集中，父项的度量值通常根据子项的值进行汇总。例如，时间维度可能包含年，季度和月份的级别。月份级别包含基础数据，这是最详细的数据。四分之一的度量值是该季度子项的月份值的汇总，一年的度量值是一年中子项的季度的汇总。通常，每个级别都映射到关系维度表中的不同列。

在基于值的层次结构中，父维度成员和子维度成员通常来自关系表中的同一列。另一列标识成员的父级。例如，值层次结构可以包含公司的所有员工，并为每个拥有一个员工的员工标识经理。包括经理在内的所有员工都来自同一专栏。另一栏将包含员工的经理。

级别

级别是基于级别的层次结构的组件。级别可以与多个层次结构相关联。维度成员只能属于一个级别。

级别通常对应于维度表或视图中的列。基本级别是主键。

属性

属性包含与维度成员相关的信息。最终用户可以使用属性来选择数据。例如，最终用户可以通过使用具有每个产品的描述性名称的属性来选择一组产品。属性包含在维度中。

查询

查询是特定数据集的规范。OLAP Java API中的术语查询是指Source对象，该对象指定一组数据，并且可以包括聚合，计算或使用该数据执行的其他操作。数据及其上的操作定义查询的结果集。在本文档中，通用术语查询是指Source对象。

API在oracle.olapi.syntax包中有一个Query类。Query表示多行，多列结果集，类似于关系表，SQL SELECT语句或OLAP函数。在将维度或度量映射到关系表或视图时，可以使用Query对象。

1.1.5组织OLAP数据

OLAP Java API使Java应用程序（包括applet）可以访问驻留在Oracle数据仓库中的数据。数据仓库是一种关系数据库，专为查询和分析而设计，而不是用于事务处理。仓库数据通常符合星型模式，星型模式是关系数据库的维度数据模型。星型模式由一个或多个事实表和一个或多个维度表组成。事实表具有包含维度表的外键的列。通常，通过提取转换传输（ETT）工具从事务处理数据库创建数据仓库。

要使数据仓库中的数据可供OLAP Java API应用程序访问，数据库管理员必须确保根据Oracle OLAP支持的组织配置数据仓库。星型模式就是这样一个组织，但不是唯一的组织。

在仓库中组织数据后，您可以使用OLAP Java API应用程序设计多维数据集，度量，维度等的OLAP维数据模型，并创建实现该模型的逻辑OLAP元数据对象。您将元数据对象映射到仓库中的数据并构建分析工作区。构建分析工作区使用OLAP元数据对象表示的数据填充OLAP视图和其他存储结构。您还可以使用Analytic Workspace Manager执行相同的任务。

OLAP Java API应用程序可以获取由Analytic Workspace Manager或OLAP Java API创建的OLAP元数据对象。它可以使用元数据对象来创建对仓库中的数据进行操作的查询。

分析工作区中的仓库数据集合是OLAP Java API提供访问权限的数据存储。当然，用户可以访问的数据范围受数据库管理员授予用户的权限的限制。

除了确保已正确准备数据和元数据之外，还必须确保应用程序用户可以与数据存储建立JDBC连接，并且用户具有允许他们访问数据的数据库权限。有关建立连接的信息，请参阅发现元数据 。

1.2通过OLAP Java API访问数据

Oracle OLAP元数据对象组织和描述客户端应用程序可用的数据。元数据对象也包含其他信息，例如数据的数据类型。但是，您无法直接从元数据对象检索数据。要指定所需的数据，必须创建查询。在指定数据时，通常必须指定一个或多个维度成员值。要检索指定的数据，请创建一个Cursor 。本主题简要介绍了这些操作。

查询OLAP元数据对象中包含的数据的另一种方法是通过SQL查询Oracle OLAP为元数据对象创建的视图。有关查询这些视图的信息，请参阅了解OLAP Java API元数据中的 “ 使用OLAP视图 ” 。

1.2.1创建查询

查询由oracle.olapi.data.source.表示oracle.olapi.data.source.Source对象。您从元数据对象获取Source ，并使用该Source对象指定要获取的数据。Source类具有用于选择和执行数据操作的方法。您可以使用这些方法以用户需要的任何方式操作数据。有关Source对象的信息，请参阅了解源对象和使用源方法进行查询 。

1.2.2指定维度成员

Oracle OLAP维度的成员通常组织为一个或多个层次结构。某些层次结构具有基于级别的父子关系，而某些层次结构具有基于值的关系。每个维度成员的值必须是唯一的。

OLAP Java API使用三部分格式来唯一标识维度成员。格式包含维度成员的层次结构，级别和值，从而标识维度中的唯一值。唯一值的第一部分是层次结构对象的名称，第二部分是级别对象的名称，第三部分是级别中成员的值。唯一值的部分由值分隔字符串分隔，默认情况下为双冒号（ :: :）。以下是名为TIME_AWJ的维度中名为CALENDAR_YEAR的层次结构中名为YEAR的级别的唯一成员值的TIME_AWJ 。

CALENDAR_YEAR :: YEAR :: CY2001

唯一值的第三部分是本地值。前面示例中的本地值标识日历年2001。

1.2.3创建游标

要检索Source指定的数据，请创建oracle.olapi.data.cursor.该Source Cursor 。然后，您可以使用此Cursor从数据存储中请求和检索数据。您可以指定Cursor在每次提取操作中检索的数据量（例如，足以填充用户界面中的40个单元格表）。然后，Oracle OLAP可以有效地管理为应用程序检索的数据块的计时，大小调整和缓存，因此您无需这样做。有关Cursor对象的信息，请参阅了解游标类和概念以及检索查询结果 。

OLAP元数据对象由CreateAndBuildAW.java和SpecifyAWValues程序创建和构建。这些元数据对象包括以下内容：

  • GLOBAL_AWJ，它是包含其他对象的分析工作区。

  • PRODUCT_AWJ，这是产品的维度。它有一个名为PRODUCT_PRIMARY的层次结构。层次结构的最低级别具有产品项标识符，而较高级别具有产品系列，类和总产品标识符。

  • CUSTOMER_AWJ，这是客户的一个维度。它有两个层次结构，名为SHIPMENTS和MARKETS。每个层次结构的最低级别具有客户标识符，更高级别具有仓库，区域和总客户，以及帐户，市场细分和总市场标识符。

  • TIME_AWJ，是时间值的维度。它有一个名为CALENDAR_YEAR的层次结构。最低级别具有月份标识符，其他级别具有季度和年份标识符。

  • CHANNEL_AWJ，这是销售渠道的一个维度。它有一个名为CHANNEL_PRIMARY的层次结构。最低级别具有销售渠道标识符，而较高级别具有总渠道标识符。

  • UNITS_CUBE_AWJ，它是一个包含度量COST，SALES和UNITS的多维数据集。 COST具有产品单位成本的价值。SALES具有产品单位销售的美元金额。UNITS具有销售的产品数量的价值。立方体的大小由所有四个维度决定。多维数据集的聚合方法是SUM ，其中每个父项的每个值都是父项子项的值的总和。

  • PRICE_CUBE_AWJ，它是一个包含度量UNIT_COST和UNIT_PRICE的多维数据集。 UNIT_COST有单位的费用。UNIT_PRICE有单位的价格。多维数据集的大小由PRODUCT_AWJ和TIME_AWJ维度决定。多维数据集的聚合方法是AVG ，其中每个父级的值是父级子级的值的平均值。

有关发现分析工作区的OLAP元数据的程序示例，请参阅发现元数据 。

1.4 OLAP Java API应用程序执行的任务

使用OLAP Java API的客户端应用程序通常执行以下任务：

  1. 连接到数据存储并创建DataProvider和UserSession 。

  2. 创建或发现元数据对象。

  3. 根据需要部署，映射和构建元数据对象。

  4. 指定选择和操作数据的查询。

  5. 检索查询结果。

本主题的其余部分简要介绍了这些任务，本指南的其余部分提供了有关如何完成这些任务的详细信息。

任务1：连接到数据存储并创建DataProvider和UserSession

通过标识有关目标Oracle数据库实例的一些信息并在JDBC连接方法中指定此信息，可以连接到数据存储。建立连接后，您将创建一个DataProvider并使用它和连接来创建UserSession 。有关连接和创建DataProvider和UserSession详细信息，请参阅“ 发现元数据 ”中的“ 连接到Oracle OLAP ” 。

任务2：创建或发现元数据对象

您使用DataProvider获取MdmMetadataProvider 。MdmMetadataProvider允许访问数据存储中的所有元数据对象。接下来，您获得MdmRootSchema通过调用对象getRootSchema的方法MdmMetdataProvider 。MdmRootSchema对象包含数据库中的所有OLAP元数据对象。从MdmRootSchema ，您可以获取当前用户有权访问的模式的MdmDatabaseSchema对象。MdmDatabaseSchema表示由SQL语句SELECT username FROM all_users返回的命名Oracle数据库用户。

从MdmDatabaseSchema ，您可以发现架构拥有的现有元数据对象，也可以创建新的元数据对象。getMeasures和getDimensions等方法获取MdmDatabaseSchema拥有的所有度量或维度。findOrCreateAW和findOrCreateCube等方法获取分析工作区或多维数据集（如果存在），或者创建一个（如果尚不存在）。

从MdmDatabaseSchema包含的顶级元数据对象（例如分析工作区，多维数据集或维度）中，您可以获取它包含的对象。例如，从MdmPrimaryDimension ，您可以获取与其关联的层次结构，级别和属性。确定了用户可用的元数据对象后，您可以向用户显示相关的对象列表，以进行数据选择和操作。

有关元数据对象的说明，请参阅了解OLAP Java API元数据 。有关如何发现可用元数据的信息，请参阅发现元数据 。

任务3：部署，映射和构建对象

如果创建新的MdmCube或MdmPrimaryDimension ，则必须将其部署为分析工作区对象或关系OLAP（Rolap）对象。要部署多维数据集，可以调用MdmCube方法，例如findOrCreateAWCubeOrganization 。要部署dimension ，可以调用MdmPrimaryDimension方法，例如findOrCreateAWPrimaryDimensionOrganization 。

如果创建表示数据的新元数据对象，则必须指定将元数据对象映射到关系源表或视图的Expression ，或Oracle OLAP用于生成数据的Expression 。对于分析工作区包含的对象，可以在映射后构建元数据对象。有关创建元数据，部署，映射和构建元数据对象的信息，请参阅创建元数据和分析工作区 。

任务4：通过查询选择和计算数据

OLAP Java API应用程序可以构建针对数据存储的查询。典型的应用程序用户界面为用户提供了选择数据和指定使用数据执行的操作的方法。然后，数据操作代码将这些指令转换为针对数据存储的查询。查询可以像选择维度成员一样简单，也可以是复杂的，包括几个聚合和计算，涉及由维度成员选择指定的度量值。

表示查询的OLAP Java API对象是Source 。表示数据的元数据对象是MdmSource类的扩展。从MdmSource （例如MdmMeasure或MdmPrimaryDimension ，您可以获取Source对象。用的方法Source对象，就可以产生其他Source指定一个选择的元素的对象Source ，或指定的计算或其它操作上的值执行Source 。

如果要实现简单的用户界面，则可以仅使用Source对象的方法来选择和操作用户在界面中指定的数据。但是，如果要为用户提供多步骤选择过程以及修改查询或撤消其选择中的各个步骤的能力，则应设计并实现Template类。在每个Template的代码中，您使用Source类的方法，但Template类本身允许您动态修改和优化最复杂的查询。此外，您可以编写通用Template类，并在应用程序的各个部分重用它们。

有关使用Source对象的信息，请参阅了解源对象 。有关使用Template对象的信息，请参阅创建动态查询 。

任务5：检索查询结果

当OLAP Java API应用程序的用户正在选择，计算，组合和通常操作数据时，他们还希望查看其工作结果。这意味着应用程序必须从数据存储中检索查询的结果集，并以多维形式显示数据。要通过OLAP Java API检索查询的结果集，请为Source指定查询创建Cursor 。

您还可以获取Oracle OLAP为查询生成的SQL。为此，您SQLCursorManager为Source创建SQLCursorManager而不是创建Cursor 。SQLCursorManager的generateSQL方法返回Source指定的SQL。然后，您可以通过OLAP Java API之外的方法检索数据。

由于OLAP Java API旨在处理数据的多维视图，因此Source可以具有多维结果集。例如， Source可以代表MdmMeasure其由四个尺寸MdmPrimaryDimension对象。每个MdmPrimaryDimension都有一个关联的Source 。您可以通过将维度的Source对象连接到度量的Source来创建查询。生成的查询将度量的Source作为基础，并将维度的Source对象作为输出。

一个Cursor的查询Source具有相同的结构Source 。例如，刚刚提到的Source的Cursor具有作为度量数据的基值。Cursor还有四个输出。输出的值是维度的Source对象的值。

要通过Cursor检索所有数据项，可以循环遍历多维Cursor结构。该设计非常适合于绘制计算机屏幕的标准用户界面对象的要求。它特别适用于以多维格式显示数据。

有关使用Source对象指定查询的详细信息，请参阅了解源对象 。有关使用Cursor对象检索数据的更多信息，请参阅了解游标类和概念 。有关SQLCursorManager类的更多信息，请参阅Oracle OLAP Java API Reference 。

语义

本节介绍file_specification的语义。有关其他信息，请参阅指定数据文件，临时文件，重做日志文件或Oracle ASM磁盘组或磁盘组文件的SQL语句。

datafile_tempfile_spec

如果数据库存储位于文件系统或Oracle ASM磁盘组中，请使用此子句指定数据文件和临时文件的属性。

redo_log_file_spec

如果数据库存储位于文件系统或Oracle ASM磁盘组中，请使用此子句指定重做日志文件的属性。

文件名

对存储在文件系统中的文件使用filename 。filename可以指定新文件或现有文件。对于新文件：

  • 如果您未使用Oracle Managed Files，则必须同时指定filename和SIZE子句，否则语句将失败。指定不带大小的文件名时，Oracle会尝试重用现有文件，并在文件不存在时返回错误。

  • 如果您正在使用Oracle管理文件，那么filename是可选的，因为是规范的其余条款。在这种情况下，Oracle数据库会为该文件创建一个唯一的名称，并将其保存在由以下初始化参数之一指定的目录中：

    • DB_RECOVERY_FILE_DEST （用于日志文件和控制文件）

    • DB_CREATE_FILE_DEST初始化参数（适用于任何类型的文件）

    • DB_CREATE_ONLINE_LOG_DEST_ n初始化参数，对于日志文件优先于DB_CREATE_FILE_DEST和DB_RECOVERY_FILE_DEST 。

对于现有文件，请指定数据文件，临时文件或重做日志文件成员的名称。filename只能包含7位ASCII或EBCDIC字符集中的单字节字符。多字节字符无效。

文件名可以包含路径前缀。如果未指定此类路径前缀，则数据库会添加默认存储位置的路径前缀，该位置与平台相关。

重做日志文件组可以包含一个或多个成员（副本）。必须根据操作系统的约定完全指定每个filename 。

数据库解释filename的方式还取决于您是否使用SIZE和REUSE子句指定它。

  • 如果仅指定filename ，或者使用REUSE子句但没有SIZE子句，则该文件必须已存在。

  • 如果使用SIZE指定filename但没有REUSE ，则该文件必须是新文件。

  • 如果使用SIZE和REUSE指定filename ，则该文件可以是new或existing。如果文件存在，则以新大小重复使用。如果它不存在，则数据库将忽略REUSE关键字并创建指定大小的新文件。

以字节为单位指定文件的大小。使用K ， M ， G或T指定大小，以千字节，兆字节，千兆字节或太字节为单位。

  • 对于撤消表空间，必须为每个数据文件指定SIZE子句。对于其他表空间，如果文件已存在，或者您正在创建Oracle托管文件，则可以省略此参数。

  • 表空间的大小必须比其中包含的对象的大小总和大一个块。

BLOCKSIZE条款

指定BLOCKSIZE以覆盖依赖于操作系统的扇区大小。如果省略此子句，则数据库将使用依赖于操作系统的扇区大小作为块大小。

将重做日志文件添加到512字节扇区磁盘或具有512字节仿真的4KB扇区磁盘时，新文件的块大小必须是原始平台基本块大小或4KB。

  • 如果将重做日志文件添加到512字节扇区磁盘，则必须指定512或1024（或1K）作为块大小，具体取决于您的平台。

  • 如果将重做日志文件添加到4KB扇区磁盘（本机），则必须指定4096或4K作为块大小。

  • 如果将重做日志文件添加到具有512字节仿真的4KB扇区磁盘，则可以指定512,1024（或1K）或4096（或4K）作为块大小，具体取决于您的平台。

日志组中的所有日志必须具有相同的块大小。在不同磁盘上创建的两个日志组可以具有不同的块大小。但是，混合配置会在每个日志切换时引入开销。Oracle建议您使用相同的块大小创建所有日志文件。

当使用4K扇区大小时，此子句很有用，但您希望优化磁盘空间使用而不是性能。在这种情况下，您可以通过指定BLOCKSIZE 512或对于HP-UX BLOCKSIZE 1024来覆盖操作系统扇区大小。

REUSE

指定REUSE以允许Oracle重用现有文件。

  • 如果该文件已存在，则Oracle将重用文件名并应用新大小（如果指定SIZE ）或保留原始大小。

  • 如果该文件不存在，则Oracle忽略此子句并创建该文件。

对REUSE条款的限制

除非指定了filename否则无法指定REUSE 。

每当Oracle使用现有文件时，该文件的先前内容都将丢失。

autoextend_clause

autoextend_clause对数据文件和临时文件有效，但对重做日志文件无效。使用此子句可启用或禁用新数据文件或现有数据文件或临时文件的自动扩展。如果省略此子句，则：

  • 对于Oracle托管文件：

    • 如果指定SIZE ，则Oracle数据库会创建一个指定大小的文件，并禁用AUTOEXTEND 。

    • 如果未指定SIZE ，则数据库将创建启用了AUTOEXTEND的100M文件。当需要自动扩展时，数据库将文件扩展为原始大小或100MB，以较小者为准。您可以通过指定NEXT子句来覆盖此默认行为。

  • 对于用户管理的文件，无论是否指定了SIZE ，Oracle都会创建一个禁用AUTOEXTEND的文件。

上

指定ON以启用autoextend。

关闭

指定OFF以关闭自动扩展（如果已打开）。关闭autoextend时， NEXT和MAXSIZE的值设置为零。如果在后续语句中重新打开autoextend，则必须重置这些值。

下一个

使用NEXT子句指定在需要更多扩展区时自动分配的下一个磁盘空间增量的大小（以字节为单位）。默认值是一个数据块的大小。

MAXSIZE

使用MAXSIZE子句指定允许自动扩展数据文件的最大磁盘空间。

无限

如果您不想限制Oracle可以分配给数据文件或临时文件的磁盘空间，请使用UNLIMITED子句。

对 autoextend_clause的 限制

您不能在CREATE CONTROLFILE语句或ALTER DATABASE CREATE DATAFILE子句中将此子句指定为datafile_tempfile_spec一部分。

例子

指定日志文件：示例

以下语句创建一个名为payable的数据库，该数据库具有两个重做日志文件组，每个组包含两个成员和一个数据文件：

CREATE DATABASE应付LOGFILE GROUP 1（'diska：log1.log'，'diskb：log1.log'）SIZE 50K，GROUP 2（'diska：log2.log'，'diskb：log2.log'）SIZE 50K DATAFILE'diskc ：dbone.dbf'SIZE 30M;

LOGFILE子句中的第一个文件规范指定GROUP值为1的重做日志文件组。该组的成员名为' diska:log1.log '和' diskb:log1.log '，每个大小为50千字节。

LOGFILE子句中的第二个文件规范指定GROUP值为2的重做日志文件组。该组的成员名为“ diska:log2.log ”和“ diskb:log2.log ”，大小也为50千字节。

DATAFILE子句中的文件规范指定名为“ diskc:dbone.dbf ”的数据文件，大小为30兆字节。

每个文件规范都指定了SIZE参数的值并省略了REUSE子句，因此这些文件都不存在。Oracle必须创建它们。

添加日志文件：示例

以下语句将另一个带有两个成员的重做日志文件组添加到payable数据库：

ALTER DATABASE应付ADD LOGFILE GROUP 3（'diska：log3.log'，'diskb：log3.log'）SIZE 50K REUSE;

ADD LOGFILE子句中的文件规范指定GROUP值为3的新重做日志文件组。这个新组的成员名为' diska:log3.log '和' diskb:log3.log '，每个大小为50千字节。由于文件规范指定了REUSE子句，因此每个成员都可以（但不必）已经存在。

以下语句在迁移目标磁盘4k_disk_a和4k_disk_b上添加了日志文件组5和成员日志文件。执行此语句后，您可以使用switch_logfile_clause将具有512字节块大小的磁盘上的现有日志文件切换为具有4K块大小的日志。 

ALTER DATABASE添加LOGFILE GROUP 5（'4k_disk_a：log5.log'，'4k_disk_b：log5.log'）SIZE 100M BLOCKSIZE 4096 REUSE;

指定数据文件：示例

“ Data Guard环境中的RMAN备份 ”解释了备份的关联和可访问性之间的区别。在Data Guard环境中，创建备份或副本的数据库与该文件相关联。如果可以访问备份，则可以在连接到Data Guard环境中的任何数据库时使用CHANGE ， DELETE和CROSSCHECK等维护命令进行备份。通常，RMAN将在任何数据库上创建的磁带备份视为环境中所有数据库都可访问，而磁盘备份只能由创建它们的数据库访问。

4.5.6上下文索引示例：使用FILTER BY和ORDER BY进行查询处理

若要为混合查询启用更高效的查询处理和更好的响应时间，请使用FILTER BY和ORDER BY子句，如以下示例所示：

此外，当查询具有匹配的ORDER BY条件时，通过指定ORDER BY pub_date desc, Oracle Text将确定是否将SORT推送到Oracle Text索引行源以获得更好的响应时间。

4.6创建CTXCAT索引

CTXCAT索引类型非常适合索引小文本片段和相关信息。此索引类型提供比CONTEXT索引更好的结构化查询性能。

  • CTXCAT索引和DML

  • 关于CTXCAT子索引及其成本

  • 创建CTXCAT子索引

  • 创建CTXCAT索引

4.6.1 CTXCAT索引和DML操作

CTXCAT索引是事务性的。在基表上执行插入，更新和删除时，Oracle Text会自动同步索引。与CONTEXT索引不同，不需要CTX_DDL.SYNC_INDEX 。

注意：

插入而不调用触发器的应用程序（如SQL * Loader）不会导致自动索引同步，如本节所述。

4.6.2关于CTXCAT子索引及其成本

CTXCAT索引包含您定义为索引集的一部分的子索引。您可以在一列或多列上创建子索引，以提高混合查询性能。但是，Oracle Text创建CTXCAT索引所需的时间取决于其总大小， CTXCAT索引的总大小与以下因素直接相关：

  • 要编入索引的文本总数

  • 索引集中的子索引数

  • 构成子索引的基表中的列数

索引集中的许多组件索引也会降低插入，更新和删除操作的性能，因为必须更新更多索引。

由于创建CTXCAT索引所需的索引时间和磁盘空间成本增加，因此在将其添加到索引集之前，请仔细考虑每个组件索引为应用程序提供的查询性能优势。

注意：

您可以使用I_ROWID_INDEX_CLAUSE的BASIC_STORAGE来加速CTXCAT索引的创建。Oracle Text Reference中描述了该子句。

Oracle Data Redaction和DML和DDL操作

数据编校策略保护的列上的Oracle数据编辑和查询

对受Oracle Data Redaction策略保护的列包含DISTINCT或ORDER BY子句的查询可能返回0行。

发生这种情况是因为编校保留了查询的语义。例如，查询如SELECT * table_name WHERE sensitive_column LIKE ' value ';可能会导致返回0行，因为编辑的值与WHERE子句中输入的' value '值不匹配。要解决此问题，请重写查询以包含内联视图，以便语义层可以找到列。例如，而不是以下查询：

SELECT DISTINCT sensitive_column FROM table_name ORDER BY sensitive_column ;

编写查询如下：

SELECT sensitive_column FROM（SELECT DISTINCT sensitive_column FROM table_name ORDER BY sensitive_column ）;

Oracle数据编辑和Oracle虚拟专用数据库

Oracle Data Redaction不会影响Oracle虚拟专用数据库策略，因为包含VPD谓词的VPD内联视图会对实际值起作用。

Oracle Data Redaction与Oracle Virtual Private Database的区别在于以下方面：

  • Oracle Data Redaction提供了比Oracle虚拟专用数据库更多的编辑功能，后者仅支持NULL编辑。许多应用程序不能使用NULL编辑，因此Data Redaction是这些应用程序的一个很好的解决方案。

  • Oracle虚拟专用数据库策略可以是静态，动态和上下文敏感的，而数据编辑策略仅允许静态和上下文敏感的策略表达式。

  • Data Redaction仅允许在表或视图上定义一个策略，而您可以在对象上定义多个虚拟专用数据库策略。

  • 数据编辑是指应用程序用户尝试使用同义词访问受数据编辑策略保护的对象，但（与Oracle虚拟专用数据库不同）数据编辑不支持直接在同义词本身上创建策略。

36.3 DBMS_PREDICTIVE_ANALYTICS

数据挖掘可以发现埋藏在大量数据中的有用信息。但是，通常情况下，获取这些结果所需的编程接口和数据挖掘专业知识过于复杂，无法被广泛的受众使用，这些受众可以从使用Oracle数据挖掘中获益。

DBMS_PREDICTIVE_ANALYTICS包通过自动化整个数据挖掘过程（从数据预处理到模型构建到评分新数据）来解决这两个复杂问题。该软件包提供了一个重要的工具，使广泛的用户，特别是业务分析师可以进行数据挖掘。

36.3.1.1 DBMS_PREDICTIVE_ANALYTICS概述

DBMS_PREDICTIVE_ANALYTICS自动化部分数据挖掘过程。

根据常用的流程模型，数据挖掘需要以下步骤：

  1. 了解业务问题。

  2. 理解数据。

  3. 准备挖掘数据。

  4. 使用准备好的数据创建模型。

  5. 评估模型。

  6. 部署并使用该模型对新数据进行评分。

DBMS_PREDICTIVE_ANALYTICS自动执行此过程的步骤3 - 5的部分内容。

预测分析程序分析和准备输入数据，使用输入数据创建和测试挖掘模型，然后使用输入数据进行评分。评分结果将返回给用户。操作完成后，不会保留模型和支持对象。

36.3.1.2 DBMS_PREDICTIVE_ANALYTICS安全模型

DBMS_PREDICTIVE_ANALYTICS包由用户SYS拥有，并作为数据库安装的一部分安装。包的执行权限授予公开。程序包中的例程使用调用者的权限运行（使用当前用户的权限运行）。

DBMS_PREDICTIVE_ANALYTICS包公开了Oracle数据挖掘选项所利用的API。希望调用此程序包中的过程的用户需要CREATE MINING MODEL系统特权（以及CREATE TABLE和CREATE VIEW系统特权）。

远程插入Rowsource

远程插入行源功能允许远程插入，要求本地Oracle数据通过Oracle数据库和Oracle数据库网关工作。此功能要求Oracle数据库和Oracle Database Gateway为12.2或更高版本。

通过Oracle数据库设计，必须在数据库链接站点执行某些分布式语句。但在某些情况下，执行这些查询所需的数据必须从原始Oracle数据库中获取。在同构连接下，远程Oracle数据库将回调源Oracle数据库以获取此类数据。但是在异构连接中，这是不可行的，因为这意味着外部数据存储必须查询只能由发出查询的Oracle实例提供的回调函数或数据。通常，这些类型的语句不能通过Oracle数据库网关支持。

以下类别的SQL语句会导致回调：

  • 任何带有子选择的DML，它指的是Oracle数据库中的一个表。

  • 任何DELETE ， INSERT ， UPDATE或“ SELECT...FOR UPDATE..."包含需要在原始Oracle数据库中执行的SQL函数或语句的SQL语句。

    这些SQL函数包括USER ， USERENV和SYSDATE ;并涉及从原始Oracle数据库中选择数据。

  • 任何涉及Oracle数据库中的表以及远程表中的LONG或LOB列的SQL语句。

可以通过远程插入行源功能工作的远程INSERT语句的示例如下：

INSERT INTO gateway_table @gateway_link select * from local_table ;

使用过程功能，网关可以执行Teradata数据库中定义的存储过程。没有必要重新链接网关或将过程定义到网关，但过程的访问权限必须允许网关访问。

标准PL / SQL语句用于执行存储过程。

网关支持三种互斥模式的存储过程：

  • 普通模式：只能访问IN / OUT参数

CHAR语义学

此功能允许网关可选地在CHAR语义模式下运行。而不是总是将Teradata CHAR列描述为CHAR(n BYTE) ，此功能将它们描述为CHAR(n CHAR)和VARCHAR(n CHAR) 。该概念类似于Oracle数据库CHAR Semantics。您需要指定HS_NLS_LENGTH_SEMANTICS=CHAR才能激活此选项。有关更多详细信息，请参阅初始化参数 。

多字节字符集比率抑制

此功能可选地抑制从Teradata数据库到涉及多字节字符集的Oracle数据库的比率扩展（例如，从US7ASCII到AL32UTF8 ，或从KO16MSWIN949到KO16KSC5601 ）。默认情况下，Oracle网关采用最差的比率来防止数据被截断或分配大小不足的缓冲区。但是，如果您具有Teradata数据库的特定知识并且不希望进行扩展，则可以指定HS_KEEP_REMOTE_COLUMN_SIZE参数来禁止扩展。有关更多详细信息，请参阅初始化参数 。

IPv6支持

除了Oracle数据库和网关之间的完全IPv6支持外，此网关和Teradata之间还支持IPv6。有关更多详细信息，请参阅初始化参数中的 HS_FDS_CONNECT_INFO参数。

网关会话IDLE超时

您可以选择使用网关参数HS_IDLE_TIMEOUT自动终止长空闲网关会话。具体来说，当网关会话空闲超过指定的时间限制时，网关会话将终止，并回滚任何挂起的更改。

架构注意事项

Teradata中不存在Oracle的架构概念。Teradata将查询中包含的模式名称识别为数据库名称。在Oracle数据库中，对象的模式与该对象的所有者相同。但是，当在数据字典视图（例如ALL_TABLES检索或引用OWNER字段时， OWNER字段实际上引用了Teradata数据库名称。

查询数据字典表时，返回以下结果：

  • ALL_*数据字典表，返回每个Teradata数据库的数据。Teradata数据库名称在OWNER ， INDEX_OWNER或TABLE_OWNER列中返回，具体取决于所引用的数据字典表。

  • ALL_USERS数据字典表，每个Teradata数据库名称都在USERNAME列中返回。

  • USER_*数据字典表，根据所引用的数据字典表，为OWNER或TABLE_OWNER列返回默认Teradata数据库的数据。如果未定义默认Teradata数据库，则使用DBC Teradata系统数据库。

命名规则

命名对象的规则

Oracle和Teradata使用不同的数据库对象命名规则。例如，每个对象名称允许的最大字符数可以不同。此外，使用单引号和双引号，区分大小写以及使用字母数字字符都可以不同。

区分大小写

除非您使用双引号字符包围标识符，否则Oracle数据库默认为大写。例如，要引用名为emp的Teradata表，请输入带双引号字符的名称，如下所示：

SQL> SELECT * FROM“emp”@TERA;

但是，要从Oracle应用程序中引用名为Scott的Teradata数据库中名为emp的Teradata表，请输入以下内容：

SQL> SELECT * FROM“Scott”。“emp”@TERA;

如果名为SCOTT的Teradata数据库中名为emp的Teradata表（一个完全由大写字母组成的名称），则可以输入不带双引号字符的所有者名称，如下所示：

SQL> SELECT * FROM SCOTT。“emp”@TERA;

要么

SQL> SELECT * FROM scott。“emp”@TERA;

Oracle建议您使用双引号字符包围所有Teradata对象名称，并使用Teradata数据字典中显示的对象名称的确切字母大小写。参照中列出的支持的Oracle数据字典的表或视图时，则不需要此约定数据字典 。

如果无法根据这些约定更改现有应用程序，请在Oracle中创建视图以将Teradata名称与正确的字母大小写相关联。例如，要仅使用大写名称从现有Oracle应用程序引用Teradata表emp ，请定义以下视图：

SQL> CREATE VIEW EMP（EMPNO，ENAME，SAL，HIREDATE）AS SELECT“empno”，“ename”，“sal”，“hiredate”FROM“emp”@TERA;

使用此视图，应用程序可以发出如下语句：

SQL> SELECT EMPNO，ENAME FROM EMP;

使用视图是一种解决方案，可复制源自Teradata数据字典的数据字典信息。只要在Teradata数据库中更改了相应表的数据定义，就必须准备好更新Oracle视图定义。

二进制文字表示法

Oracle SQL使用由单引号括起的十六进制数字来表示要比较或插入到定义为数据类型RAW列中的文字值。

此表示法不会转换为与Teradata VARBINARY和BINARY数据类型兼容的语法（ ff由单引号括起，后跟十六进制数字）。

行选择

Teradata在返回任何行之前评估所有选定行的查询条件。如果一个或多个行的评估过程中存在错误，则即使其余行满足条件，也不会返回任何行。

Oracle逐行评估查询条件，并在评估成功时返回一行。返回行直到一行未通过评估。

空绑定变量

对于VARCHAR绑定变量，网关将空绑定变量作为NULL值传递给Teradata数据库。

锁定

  • 读访问可能会阻止写访问。

  • 写访问可能会阻止读访问。

  • 语句级读取一致性无法保证。

要防止此错误，请在游标循环外移动COMMIT或ROLLBACK语句。

Teradata LONG VARCHAR数据类型

使用LONG VARCHAR数据类型时，以下限制适用：

  • 不支持的SQL函数不能在访问定义为Teradata数据类型LONG VARCHAR的列的SQL语句中使用。

  • 当数据长度超过80个字符时，不能使用SQL * Plus从定义为Teradata数据类型LONG VARCHAR的列中选择数据。Oracle建议使用Pro * C或Oracle Call Interface访问Teradata数据库中的此类数据。

  • LONG VARCHAR数据类型必须为NULLABLE才能使INSERT或UPDATE正常工作。

  • 包含LONG VARCHAR列的表必须在表上定义唯一索引，否则表必须具有用作主键的单独列。

  • 无法通过传递查询读取LONG VARCHAR数据。

网关不支持DBMS_SQL包的PL / SQL函数COLUMN_VALUE_LONG 。
