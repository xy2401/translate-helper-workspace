193 DBMS_XDBT

DBMS_XDBT包为管理员提供了一种方便的机制，可以在Oracle XML DB层次结构上设置CONTEXT索引。

该软件包包含创建默认首选项，创建索引和设置CONTEXT索引自动同步的过程

DBMS_XDBT包还包含一组包变量，用于描述索引的配置设置。这些旨在涵盖安装可能需要的基本自定义，但绝不是完整的集。

193.1 DBMS_XDBT概述

DBMS_XDBT包中包含一组包变量，用于描述索引的配置设置。

DBMS_XDBT包可以通过以下方式使用：

  • 自定义程序包以设置适当的配置。

  • 使用DROPPREFERENCES过程删除任何现有的索引首选项

  • 使用CREATEPREFERENCES过程过程创建新的索引首选项

  • 使用CREATEINDEX Procedure过程创建CONTEXT索引

  • 使用CONFIGUREAUTOSYNC过程设置索引的自动同步

193.2 DBMS_XDBT安全模型

由XDB拥有， DBMS_XDBT包必须由SYS或XDB创建。 EXECUTE特权被授予SYS或XDB 。此包中的子程序使用当前用户的权限执行。

193.3 DBMS_XDBT操作说明

配置设置或包变量可用于自定义DBMS_XDBT包。

可以使用PL / SQL过程或匿名块来自定义DBMS_XDBT包，以设置相关的包变量，配置设置，然后执行这些过程。更通用的方法是通过适当地修改此包或作为副本来引入适当的自定义。必须将系统配置为使用作业队列，并且可以通过USER_JOBS目录视图查看作业。

193.4.2 CREATEDATASTOREPREF程序

此过程为XML DB层次结构上的CONTEXT索引创建用户数据存储区首选项。

句法

DBMS_XDBT.CREATEDATASTOREPREF;

使用说明

  • 可以修改d atastore首选项的名称;请参阅DatastorePref配置设置。

  • 默认的USER数据存储过程还会过滤传入的文档。DBMS_XDBT包提供了一组控制过滤过程的配置设置。

  • SkipFilter_Types数组包含正则表达式列表。具有与这些表达式之一匹配的mime类型的文档不会编制索引。文档元数据的某些属性（如作者）仍未编入索引。

    • NullFilter_Types数组包含正则表达式列表。具有与这些表达式之一匹配的mime类型的文档不会被过滤;但是，他们仍然被编入索引。这旨在用于基于文本的文档，例如HTML，XML和纯文本。

    • 所有其他文档都通过IFILTER API使用INSO过滤器。

XS_ACL包的安全模型

XS_ACL包是在SYS架构下创建的。

DBA角色被授予ADMIN_ANY_SEC_POLICY权限，该权限允许它跨所有模式管理ACL，安全类和安全策略等模式对象。

如果已为模式授予RESOURCE角色，则用户可以在自己的模式中管理模式对象。RESOURCE角色和XS_RESOURCE应用程序角色包括管理模式中的模式对象所需的ADMIN_SEC_POLICY权限，以及在授予的模式中管理策略工件以在应用程序中实现策略管理。

如果已授予APPLY_SEC_POLICY权限，则用户可以管理架构上的策略实施。使用此权限，用户可以在授予的模式中管理策略实施，以在应用程序内实现策略管理。

3.1自动数据库诊断监视器概述

ADDM是Oracle数据库中内置的诊断软件。ADDM检查并分析在自动工作负载存储库（AWR）中捕获的数据，以确定可能的数据库性能问题。ADDM然后执行以下操作：

  • 找出性能问题的根本原因

  • 提供纠正它们的建议

  • 量化预期收益

  • 标识不需要采取任何措施的区域。

本节包含以下主题：

  • ADDM分析

  • ADDM建议

  • 适用于Oracle Real Application Clusters的ADDM

  • 适用于多租户环境的ADDM

3.1.1 ADDM分析

ADDM分析在每个AWR快照之后执行（默认情况下每小时执行一次），结果将保存在数据库中。然后，您可以使用Cloud Control查看结果。

ADDM分析从上到下执行，首先识别症状，然后改进分析以找出性能问题的根本原因。ADDM使用DB时间统计来识别性能问题。数据库时间（DB）时间是数据库在处理用户请求时所花费的累积时间，包括非空闲的所有用户会话的等待时间和CPU时间。

数据库性能调优的目标是减少给定工作负载的系统DB时间。通过减少DB时间，数据库可以通过使用相同或更少的资源来支持更多用户请求。ADDM报告使用大部分数据库时间作为问题区域的系统资源，并按相关数据库时间花费的降序对其进行排序。

3.1.2 ADDM建议

除诊断性能问题外，ADDM还建议使用可能的解决方案。适当时，ADDM建议您选择多种解决方案。ADDM 建议包括以下内容：

  • 硬件变化

    添加CPU或更改I / O子系统配置

  • 数据库配置

    更改初始化参数设置

  • 架构更改

    散列分区表或索引，或使用自动分段空间管理（ASSM）

  • 申请变更

    使用序列的缓存选项或使用绑定变量

  • 使用其他顾问

    在高负载SQL语句上运行SQL Tuning Advisor或在热对象上运行Segment Advisor

ADDM的优势适用于生产系统。即使在开发和测试系统上，ADDM也可以提供潜在性能问题的早期警告。

性能调优是一个迭代过程。解决一个问题可能会导致瓶颈转移到系统的另一部分。即使利用ADDM分析，也可能需要多个调整周期才能达到理想的性能水平。

3.1.3 Oracle Real Application Clusters的ADDM

在Oracle Real Application Clusters（Oracle RAC）环境中，您可以使用ADDM分析数据库集群的吞吐量性能。ADDM for Oracle RAC将DB时间视为所有数据库实例的数据库时间总和，并报告在群集级别具有重要意义的结果。例如，当单独考虑时，每个群集节点的DB时间可能是无关紧要的，但是聚合DB时间对于整个群集可能是一个重要问题。

3.2.2设置DBIO_EXPECTED参数

I / O性能的ADDM分析部分取决于单个参数DBIO_EXPECTED ，它描述了I / O子系统的预期性能。DBIO_EXPECTED的值是读取单个数据库块所需的平均时间（以微秒为单位）。Oracle数据库使用默认值10毫秒，这是大多数硬盘驱动器的适当值。您可以根据硬件的特性选择不同的值。

要确定DBIO_EXPECTED初始化参数的正确设置：

  1. 测量硬件的单个数据库块的平均读取时间。

    必须对随机I / O进行此测量，如果使用标准硬盘驱动器，则包括寻道时间。硬盘驱动器的典型值介于5000和20000微秒之间。

    也可以看看：

    Oracle数据库性能调优指南 ，了解如何评估存储子系统的I / O功能

  2. 为所有后续ADDM执行设置一次值。

    EXECUTE DBMS_ADVISOR.SET_DEFAULT_TASK_PARAMETER（'ADDM'，'DBIO_EXPECTED'，8000）;

3.2.3管理AWR快照

3.4自动数据库诊断监视器结果的解释

ADDM分析结果表示为一组发现。每个ADDM发现都属于以下三种类型之一：

  • 问题

    描述数据库性能问题根本原因的调查结果

  • 症状

    包含通常会导致一个或多个问题发现的信息的调查结果

  • 信息

    用于报告系统中没有性能影响的区域的结果

通过估计DB时间的部分量化每个问题发现这是由性能问题引起的。

当特定问题有多种原因时，ADDM可能会报告多个结果。在这种情况下，这些多个发现的影响可能包含相同的DB时间部分。由于性能问题可能会重叠，因此总结报告结果的影响可能会产生高于DB时间100％的数字。例如，如果系统执行许多读取I / O操作，ADDM可能会报告由于I / O活动作为一个发现而导致50％DB时间的SQL语句，以及负责75％DB时间的小型缓冲区高速缓存。另一个发现。

问题发现可以与推荐列表相关联减少性能问题的影响。每个建议都有一个好处，即在实施建议时可以保存的DB时间部分的估计值。当多个建议与ADDM发现相关联时，建议可能包含解决相同问题的替代方案。在这种情况下，收益的总和可能高于发现的影响。您无需应用所有建议来解决同一问题。

建议由行动组成和理由 。您必须应用建议的所有操作才能获得预期收益。基本原理解释了为什么建议采取这些行动，并提供实施这些行动的其他信息。ADDM动作可以呈现多种解决方案。如果是这种情况，那么选择最简单的解决方案来实现。

250.1 UTL_CALL_STACK概述

UTL_CALL_STACK包为PL / SQL程序员提供了一个接口，以获取有关当前正在执行的程序的信息，包括来自动态和词汇堆栈的子程序名称以及这些堆栈的深度。

各个函数返回给定动态深度的子程序名称，单元名称，所有者名称，版本名称和行号。更多函数返回错误堆栈信息。此类信息可用于创建更具启发性的错误日志和应用程序执行跟踪。

动态深度

PL / SQL子程序的执行实例的动态深度是递归定义的。

  • 当前正在执行的子程序实例的动态深度为1。

  • 否则，子程序实例的动态深度比它调用的子程序的动态深度多一个。

  • 如果存在由执行子程序调用或调用的SQL，Java或其他非PL / SQL上下文，则它占用调用堆栈上的级别，就像它是子程序一样。

在一个调用堆栈的情况下， A调用B ，调用C ，调用D ，调用E ，调用F ，调用E ，这个堆栈可以写成一个下面有动态深度的行：

ABCDEFE 7 6 5 4 3 2 1

词汇深度

PL / SQL子程序的词法深度是递归定义的。

  • 单位的词法深度，匿名块，触发器或ADT是一（1）。

  • 在另一个对象中定义的子程序的词法深度是一个加上该对象的词法深度。

块不会影响词汇深度。

错误深度

错误深度是错误堆栈上的错误数。

例如，请考虑以下匿名块。

开始开始......（1）提高zero_divide;其他人然后提出no_data_found时的异常;结束;当别人那么例外......（2）结束;

（1）处的误差深度为零，（2）处的误差深度为2。

回溯

回溯是从抛出异常到检查回溯的位置的跟踪。

考虑一个调用堆栈，其中A调用B调用C和C引发异常。如果在C中检查了回溯，则回溯将具有一个单位C，并且回溯深度将为1。如果在A中检查，它将有三个单位，A，B和C，回溯深度为3。

在没有异常的情况下，回溯的深度为零。

250.2 UTL_CALL_STACK安全模型

UTL_CALL_STACK上的EXECUTE被授予PUBLIC 。

UTL_CALL_STACK包不显示包装的程序单元。例如，考虑一个调用堆栈，其中程序单元A调用B ，调用C ，然后调用UTL_CALL_STACK来确定子程序列表。如果程序单元B被包装，则子程序列表仅显示程序单元C

250.3 UTL_CALL_STACK操作说明

某些操作说明适用于UTL_CALL_STACK 。

  • 编译器优化可以改变词法，动态和回溯深度。

  • 超过RPC边界不支持UTL_CALL_STACK 。例如，如果A调用远程过程B，则B将无法使用UTL_CALL_STACK获取有关A的信息。

  • 词法单元信息可通过PL / SQL条件编译功能获得，因此不会通过UTL_CALL_STACK 。

该网关系统标识符 （SID）是标识网关实例的字母数字字符串。对于要访问的每个Sybase数据库，您需要一个网关实例，因此需要一个网关SID。SID用作初始化参数文件的文件名的一部分。默认SID是 dg4sybs 。

您可以定义网关SID，但使用默认的dg4sybs更容易，因为您不需要更改初始化参数文件名。但是，如果要访问两个Sybase数据库，则需要两个网关SID，每个网关用于网关的每个实例。如果您只有一个Sybase数据库并且希望有时使用一组网关参数设置来访问它，而其他时候使用不同的网关参数设置，那么您将需要多个网关SID用于单个Sybase数据库。

37.1.4.1关于物化视图日志

主表上需要物化视图日志，以根据主表对物化视图执行快速刷新。

为主表创建实例化视图日志时，Oracle数据库会创建基础表作为实例化视图日志。物化视图日志可以保存主表中已更新的行的主键，rowid或对象标识符。物化视图日志还可以包含其他列，以支持使用子查询快速刷新实例化视图。

物化视图日志表的名称是MLOG$_ master_table_name 。物化视图日志在与目标主表相同的模式中创建。一个物化视图日志可以在其主表上支持多个物化视图。如上一节所述，只要在目标主表上发生DML事务，内部触发器就会将更改信息添加到物化视图日志中。

以下是物化视图日志的类型：

  • 主键 ：物化视图根据受影响行的主键记录对主表的更改。

  • 行ID ：物化视图根据受影响行的rowid记录对主表的更改。

  • 对象ID ：实体化视图根据受影响的行对象的对象标识符记录对主对象表的更改。

  • 组合 ：物化视图根据三个选项的任意组合记录对主表的更改。可以根据主键， ROWID和受影响行的对象标识符记录更改。这样的物化视图日志支持主键， ROWID和对象物化视图，这对于具有基于主表的所有三种类型的物化视图的环境很有帮助。

组合物化视图日志的工作方式与仅跟踪一种类型值的物化视图日志相同，但记录的值不止一种。例如，组合物化视图日志可以跟踪受影响行的主键和rowid。

虽然基于主键和rowid的物化视图日志之间的差异很小（一个使用主键记录受影响的行，而另一个记录使用物理rowid影响行），但实际影响很大。使用rowid物化视图和物化视图日志会使重组和截断主表变得困难，因为它会阻止您的ROWID物化视图快速刷新。如果重组或截断你的主表，那么你的ROWID的物化视图必须是COMPLETE ，因为主表的的rowid已经改变刷新。

注意：

  • 您可以使用BEGIN_TABLE_REORGANIZATION和END_TABLE_REORGANIZATION程序在DBMS_MVIEW包重组主表。有关更多信息，请参见Oracle Database PL / SQL包和类型参考 。

  • 在线重新定义表是另一种重组主表的可能方法，但在具有物化视图日志和物化视图的主表上不允许在线重新定义。在没有物化视图日志的主表上允许在线重新定义。请参阅“ 在线重新定义表 ” 。

37.1.4.2在物化视图日志中记录的列

创建实例化视图日志时，可以在必要时向日志添加列。

要在实例化视图上执行快速刷新，必须将以下类型的列添加到实例化视图日志中：

  • 子查询的WHERE子句中引用的列，它不是equi-join的一部分，不是主键列。这些列称为过滤列。

  • 如果子查询是多对多或一对多，则equi-join中的列不是主键列。如果子查询是多对一，则不需要将连接列添加到实例化视图日志中。

无法将集合列添加到实例化视图日志中。此外，使用完全刷新的物化视图不需要物化视图日志。

例如，请考虑以下DDL：

1）创建物化视图oe.customers快速刷新2）SELECT * FROM oe.customers@orc1.example.com c 3）在哪里存在4）（SELECT * FROM oe.orders@orc1.example.com o 5）WHERE c .customer_id = o.customer_id AND o.order_total> 20000）;

请注意前面DDL的第5行， WHERE子句中引用了三列。列orders.customer_id和customers.customer_id作为equi-join子句的一部分引用。由于customers.customer_id是主键列，因此默认情况下会记录该列，但orders.customer_id不是主键列，因此必须添加到orders.customer_id化视图日志中。此外，列orders.order_total是一个额外的过滤器列，因此必须记录。

因此，加orders.customer_id和orders.order_total的物化视图日志oe.orders表。

建议您分析计划的物化视图的定义查询，并确定必须将哪些列添加到物化视图日志中。如果尝试创建或刷新需要添加列而不将列添加到实例化视图日志的实例化视图，则实体化视图创建或刷新可能会失败。

注意：

要在实例化视图上执行快速刷新，如果连接列不是主键且子查询是多对多或一对多，则必须将子查询中的连接列添加到实例化视图日志中。如果子查询是多对一，则不需要将连接列添加到实例化视图日志中。

37.3.1刷新组

为了保留多个物化视图之间的参照完整性和事务（读取）一致性，Oracle数据库可以刷新单个物化视图作为刷新组的一部分。

刷新刷新组中的所有物化视图后，组中所有物化视图的数据对应于相同的事务一致时间点。

父主题： 组织机制

37.3.2刷新组大小

在决定刷新组的大小时，需要考虑一些权衡因素。

Oracle数据库针对大型刷新组进行了优化。因此，如果组中的物化视图相似，则大型刷新组的刷新速度比小型刷新组中的相同数量的物化视图更快。例如，刷新具有100个物化视图的刷新组比刷新五个刷新组（每个刷新组具有20个物化视图）更快。此外，大型刷新组使您只需调用一个PL / SQL子程序即可刷新更多的物化视图。

执行刷新时必须保持网络连接。如果在刷新期间连接丢失或中断，则会回滚所有更改，以使数据库保持一致。因此，在难以维护网络连接的情况下，请考虑使用较小的刷新组。

还有一个优化空刷新。也就是说，如果自上次刷新特定物化视图后主表没有更改，那么在刷新期间物化视图几乎不需要额外的时间。

37.4刷新过程

要确保实例化视图与其主表一致，必须定期刷新实例化视图。

  • 关于刷新过程
    物化视图刷新是一种有效的批处理操作，它使物化视图反映其主表的更新当前状态。
  • 刷新类型
    Oracle数据库可以使用快速，完整或强制刷新来刷新实例化视图。
  • 启动刷新
    创建刷新组时，可以配置该组，以便Oracle数据库按计划的时间间隔自动刷新组的实例化视图。相反，您可以省略计划信息，以便必须手动或“按需”刷新刷新组。在不总是具有网络连接的系统上执行刷新时，手动刷新是理想的解决方案。
  • 约束和刷新
    要在刷新实例化视图期间避免任何完整性约束违规，请在每个实例化视图上进行非主键完整性约束可延迟。

父主题： 只读物化视图体系结构

37.4.1关于刷新过程

物化视图刷新是一种有效的批处理操作，它使物化视图反映其主表的更新当前状态。

物化视图的数据不一定与其主表的当前数据匹配。物化视图是其主表的事务（读）一致反映，因为数据存在于特定时间点（即，在创建时或发生刷新时）。要使物化视图的数据与其主表的数据保持相对最新，必须定期刷新物化视图。

主表中的行可以在物化视图的刷新之间多次更新，但刷新仅使用当前数据更新实例化视图中的行。例如，自上次刷新实例化视图以来，主表中的行可能会更新10次，但结果仍然只是在下次刷新期间物化视图中相应行的一次更新。

确定如何以及何时刷新每个物化视图以使其更新。例如，基于应用程序更新的主表的物化视图可能需要频繁刷新。相反，基于相对静态主表的物化视图通常需要不经常刷新。总之，分析应用程序特征和要求以确定适当的物化视图刷新间隔。

要刷新实例化视图，Oracle数据库支持多种刷新类型和启动刷新的方法。

父主题： 刷新过程

37.4.2刷新类型

Oracle数据库可以使用快速，完整或强制刷新来刷新实例化视图。

  • 完全刷新
    要执行物化视图的完整刷新 ，管理物化视图的服务器将执行物化视图的定义查询，该查询基本上会重新创建物化视图。
  • 快速刷新
    要执行快速刷新 ，管理实例化视图的主表首先标识自最近一次刷新实例化视图以来主表中发生的更改，然后将这些更改应用于实例化视图。
  • 强制刷新
    要对物化视图执行强制刷新 ，管理物化视图的服务器会尝试执行快速刷新。如果无法快速刷新，则Oracle数据库会执行完全刷新。

父主题： 刷新过程

37.4.2.1完全刷新

要执行物化视图的完整刷新 ，管理物化视图的服务器将执行物化视图的定义查询，该查询基本上会重新创建物化视图。

要刷新实例化视图，查询的结果集将替换现有的实例化视图数据。Oracle数据库可以对任何物化视图执行完整刷新。根据满足定义查询的数据量，完全刷新可能比快速刷新花费更长的时间。

注意：

如果对物化视图使用完全刷新，则将其PCTFREE设置为0和 PCTUSED到99以获得最大效率。

父主题： 刷新类型

37.4.2.2快速刷新

要执行快速刷新 ，管理实例化视图的主表首先标识自最近一次刷新实例化视图以来主表中发生的更改，然后将这些更改应用于实例化视图。

当主表的更改很少时，快速刷新比完全刷新更有效，因为参与的服务器和网络复制的数据量较少。仅当主表具有实例化视图日志时，才能快速刷新实例化视图。此外，为了使快速刷新比完全刷新更快， CREATE MATERIALIZED VIEW语句中的每个连接列都必须有一个索引。

使用SQL * Loader在主表上直接路径加载后，快速刷新不会应用直接路径加载期间发生的更改。此外，快速刷新不会应用主表上其他类型的批量加载操作所导致的更改。这些操作的示例包括带有APPEND提示的INSERT语句和INSERT ... SELECT * FROM语句。

如果您具有基于分区主表的物化视图，则可以使用分区更改跟踪（PCT）来标识哪些物化视图行对应于特定分区。PCT还用于支持在物化视图的主表上进行分区维护操作后的快速刷新。只有满足几个条件，才能实现基于PCT的物化视图刷新。

37.4.3.1计划刷新

设置组的刷新间隔时，请考虑以下特征：

  • 指定刷新间隔的日期或日期表达式必须计算到将来的时间点。

  • 刷新间隔必须大于执行刷新所需的时间长度。

  • 相对日期表达式计算到相对于最近刷新日期的时间点。如果网络或系统故障干扰计划的组刷新，则相对日期表达式的评估可能会相应更改。

  • 无论最近的刷新日期如何，显式日期表达式都会评估特定时间点。

  • 考虑您的环境对陈旧数据的容忍度：如果容差很小，则经常刷新;而如果存在高容忍度，则刷新次数较少。

  • 七天的间隔指定为：

18.2使用ORACLE_LOADER访问驱动程序创建分区混合表

本主题介绍如何使用ORACLE_LOADER访问驱动程序创建分区混合表。

混合分区表是一种通过允许某些分区驻留在数据库段中以及外部文件或源中的某些分区来扩展Oracle分区的功能。这显着增强了大数据SQL的分区功能，其中表的大部分可以驻留在外部分区中。

例18-2例子

以下是创建分区混合l表的语句示例：

CREATE TABLE hybrid_pt（time_id日期，客户编号）TABLESPACE TS1外部分区属性（类型ORACLE_LOADER默认目录data_dir0访问参数（字段终止于'，'）拒绝限制无限制分区范围（time_id）（PARTITION century_18 VALUES少于（'01） -01-1800'）EXTERNAL，< - 空外部分区PARTITION century_19值少于（'01 -01-1900'）外部默认目录data_dir1位置（'century19_data.txt'），PARTITION century_20 VALUES少于（'01  - 01-2000'）外部位置（'century20_data.txt'），PARTITION year_2000值小于（'01 -01-2001'）TABLESPACE TS2，PARTITION pmax VALUE小于（MAXVALUE））;

在此示例中，该表包含内部和外部分区。表中内部分区的默认表空间是TS1 。添加了一个EXTERNAL PARTITION ATTRIBUTES子句，用于指定在表级别应用于表中外部分区的参数。该子句对于混合分区表是必需的。在这种情况下，通过ORACLE_LOADER访问驱动程序访问外部分区，并在子句中指定访问驱动程序所需的参数。在分区级别，在每个外部分区中指定EXTERNAL子句，以及应用于该分区的任何外部参数。

4.1.1.2表示层

如果客户端和数据库服务器在不同的操作系统上运行，则会出现字符集差异。表示层解决了任何差异。它针对每个连接进行了优化，以便在需要时执行转换。

客户端/服务器应用程序使用的表示层是 双任务共同（TTC） 。TTC在客户端和数据库服务器上提供不同字符集或格式之间的字符集和数据类型转换。在初始连接时，TTC负责评估内部数据和字符集表示的差异，并确定两台计算机之间是否需要进行转换。

4.1.1.3 Oracle Net Foundation Layer

该 Oracle Net基础层负责建立和维护客户端应用程序与数据库服务器之间的连接，以及在它们之间交换消息。由于透明网络基板（TNS）技术，Oracle Net基础层可以执行这些任务 。TNS为所有行业标准OSI传输和网络层协议提供单一，通用的接口。TNS支持对等应用程序连接，其中两台或多台计算机可以直接相互通信，而无需任何中间设备。

在客户端，Oracle Net基础层接收客户端应用程序请求并解决所有通用计算机级连接问题，例如：

  • 数据库服务器或目标的位置

  • 连接涉及多少个协议

  • 如何根据每个服务器的功能处理客户端和数据库服务器之间的中断

在服务器端，Oracle Net基础层执行与客户端相同的任务。它还与侦听器一起使用以接收传入的连接请求。

除了建立和维护连接之外，Oracle Net基础层还通过命名方法进行通信，以解析名称并使用安全服务来确保安全连接。

4.4.1.1.1 CIDR表示法

无类别域间路由（CIDR）是一种将IP地址分组为独立于地址值的子网的方法。无类路由旨在克服IP类系统中地址空间的耗尽以及路由表大小的无法管理的增长。

CIDR表示网络中第一个地址的网络，以十进制表示网络前缀的比特大小，用斜杠（ / ）分隔。例如， 2001:0db8::/32表示地址的前32位标识网络，而其余位标识网络中的主机。

CIDR对IPv4地址使用类似表示法。例如，在符号192.0.2.0/24中，地址的前24位表示网络前缀。DBMS_NETWORK_ACL_ADMIN程序包提供用于管理访问控制列表的API，支持IPv6和IPv4地址和子网的CIDR表示法。

4.4.2关于使用SSL协议的TCP / IP

具有安全套接字层（SSL）协议的TCP / IP使客户端上的Oracle应用程序能够通过TCP / IP和SSL与远程数据库进行通信。 SSL在Oracle Wallet中存储身份验证数据，例如证书和私钥。当客户端启动与数据库服务器的连接时，SSL使用证书在两者之间执行握手。在握手期间，会发生以下过程：

  • 客户端和数据库服务器协商由一组身份验证，加密和数据完整性类型组成的密码套件，以应用于它们交换的消息。

  • 根据其配置，数据库服务器在使用客户端公钥加密的消息中将其证书发送到客户端。数据库服务器还可以在同一消息中发送对客户端证书的请求。客户端使用自己的私钥解密此消息，然后验证数据库服务器的证书是否具有证书颁发机构的签名。

  • 如果需要，客户端可以将用户的证书发送到数据库服务器。证书确保用户的信息正确并且公钥实际上属于该用户。

数据库检查用户证书以验证它是否具有证书颁发机构的签名。

2.1.1监督数据挖掘

监督学习也被称为定向学习。学习过程由先前已知的从属属性或目标。定向数据挖掘试图将目标的行为解释为一组独立属性或预测变量的函数。

监督学习通常会导致预测模型。这与无监督学习形成对比，其中目标是模式检测。

监督模型的构建涉及培训 ，该过程软件分析许多已知目标值的情况。在训练过程中，模型“学习”进行预测的逻辑。例如，寻求识别可能响应促销的客户的模型必须通过分析已知在过去已响应或未响应促销的许多客户的特征来进行培训。

2.1.1.1监督学习：测试

建筑（培训）需要单独的数据集并测试一些预测模型。构建数据（训练数据）和测试数据必须具有相同的列结构。通常，一个大型表或视图分为两个数据集：一个用于构建模型，另一个用于测试模型。

将模型应用于测试数据的过程有助于确定基于一个选定样本构建的模型是否可推广到其他数据。特别是，它有助于避免这种现象过度拟合，当模型的逻辑太适合构建数据并且因此具有很小的预测能力时可能发生。

2.1.1.2监督学习：评分

同时应用数据称为评分数据，是实际的应用模型的人口。例如，您可以构建一个模型来标识经常购买某个产品的客户的特征。要获取在某个商店购物并且可能购买相关产品的客户列表，您可以将该模型应用于该商店的客户数据。在这种情况下，商店客户数据是评分数据。

大多数监督学习可以应用于感兴趣的人群。主要的监督采矿技术， 分类和回归 ，都可以用于评分。

Oracle Data Mining不支持另一个受监督函数Attribute Importance的评分操作。这种类型的模型建立在感兴趣的人群上，以获得有关该人口的信息;它们不能应用于单独的数据。属性重要性模型返回并排列在预测目标值时最重要的属性。

2.3数据准备

模型的质量在很大程度上取决于用于构建（训练）它的数据的质量。在任何给定的数据挖掘项目中花费的大部分时间都用于数据准备。必须仔细检查，清理和转换数据，并且必须应用与算法相关的数据准备方法。

由于应用模型的任何数据，无论是用于测试还是用于评分，数据准备过程都变得更加复杂，必须经历与用于训练模型的数据相同的变换。

2.3.1 Oracle数据挖掘简化了数据准备

Oracle Data Mining提供了几个可以显着简化数据准备过程的功能：

  • 嵌入式数据准备： 用于训练模型的转换嵌入在模型中，并在模型应用于新数据时自动执行。如果为模型指定转换，则只需指定一次。

  • 自动数据准备（ADP）：Oracle Data Mining支持自动数据准备模式。当ADP处于活动状态时，Oracle Data Mining会自动执行算法所需的数据转换。转换指令与任何用户指定的转换指令一起嵌入到模型中。

  • 自动管理缺失值和稀疏数据： Oracle Data Mining跨采矿算法使用一致的方法来处理稀疏性和缺失值。

  • 透明度：Oracle Data Mining提供模型详细信息，这些信息是模型内部属性的视图。由于反向转换可以将对转换后的属性值映射到可由用户解释的形式，因此可以深入了解模型的内部细节。在可能的情况下，属性值将反转为原始列值。逆变换也应用于监督模型的目标，因此得分结果与原始目标的单位具有相同的单位。

  • 用于自定义数据准备的工具：Oracle Data Mining在DBMS_DATA_MINING_TRANSFORM PL / SQL包中提供了许多常见的转换例程。您可以使用这些例程，或在SQL中或两者中开发自己的例程。SQL语言非常适合在数据库中实现转换。您可以使用自定义转换说明以及ADP或ADP。

2.4数据库内评分

评分是将数据挖掘算法应用于新数据。在传统的数据挖掘中，模型使用远程系统上的专用软件构建，并部署到另一个系统进行评分。这是一个麻烦的，容易出错的过程，容易受到安全违规和数据同步的困扰。

使用Oracle Data Mining，评分简单而安全。评分引擎和数据都驻留在数据库中。评分是SQL语言的扩展，因此挖掘结果可以轻松地合并到应用程序和报告系统中。

2.4.1并行执行和易于管理

所有Oracle Data Mining评分例程都支持并行执行以评估大型数据集。

在数据库评分提供了性能优势。所有Oracle Data Mining评分例程都支持并行执行，可显着减少执行复杂查询和评分大型数据集所需的时间。

数据库内挖掘可最大限度地减少支持数据挖掘计划所需的IT工作量。使用标准数据库技术，可以轻松地在更新的数据上刷新（重新创建）模型并重新部署。部署是立即的，因为评分查询保持不变;只在数据库中替换基础模型。

关于Oracle ASM Flex磁盘组

Oracle ASM Flex磁盘组是支持Oracle ASM文件组和配额组的磁盘组类型。

通常，除磁盘组级别外，灵活磁盘组还允许用户以数据库的粒度管理存储。

以下是Flex磁盘组的特征：

  • Flex磁盘组中的文件冗余非常灵活，可以在数据库级别进行存储管理。

  • Flex磁盘组的文件组用于描述数据库文件。

    每个数据库都有自己的文件组，除磁盘组级别外，还可以在文件组级别完成存储管理。例如，您可以为不同的文件组指定不同的冗余和重新平衡设置。文件组与配额组关联，从而实现轻松的配额管理。

  • Flex磁盘组的冗余设置设置为FLEX REDUNDANCY并且Flex磁盘组的每个文件组都有自己的冗余属性设置。

  • Flex磁盘组通常可以容忍两个故障，与HIGH冗余磁盘组相同。

    但是，如果磁盘组的故障组少于五个，则它只能容忍一个故障。此限制是磁盘组的仲裁要求的结果。

  • 数据库容忍的故障取决于Flex磁盘组中其关联文件组的冗余设置。

  • Flex磁盘组至少需要三个故障组。

  • 如果具有FLEX REDUNDANCY的磁盘组只有两个常规故障组和一个仲裁故障组，则即使为Flex磁盘组中的数据库文件选择了HIGH REDUNDANCY ，也只会存储两个数据副本。此行为是仅有两个常规故障组的结果。但是，如果稍后添加其他常规故障组，则会自动创建第三个数据副本。添加常规故障组后，如果一个故障组丢失并且重新平衡成功完成，则在第二个故障组脱机后，磁盘组可以保持挂载状态。

  • 可以使用ALTER DISKGROUP SQL语句的CONVERT REDUNDANCY TO FLEX子句对NORMAL或HIGH冗余磁盘组执行迁移（转换）到Flex磁盘组。您无法将EXTERNAL冗余磁盘组迁移到FLEX冗余磁盘组。

    注意：

    • 迁移到Flex磁盘组时，磁盘组必须至少具有三个故障组。

    • 必须启用虚拟分配元数据（VAM）迁移磁盘组属性，并且在迁移之前完成迁移，而不使用限制性安装。

  • Flex磁盘组可以创建时间点数据库克隆。

    时间点数据库克隆是特定数据库的完整副本，而不是底层存储系统。仅复制数据文件。创建其他文件，或根据需要引用其他文件以创建克隆。有关Flex磁盘组和时间点数据库克隆的信息，请参阅关于使用Oracle ASM Flex磁盘组的时间点数据库克隆 。

  • 分配单元（AU）的默认大小为4 M.

  • 虚拟磁盘组启用并需要虚拟分配的元数据（VAM）。

  • Flex磁盘组要求COMPATIBLE.ASM和COMPATIBLE.RDBMS磁盘组属性设置为12.2或更高。

  • 要在Flex磁盘组上创建Oracle ADVM卷， COMPATIBLE.ADVM磁盘组属性必须设置为12.2或更高。

V$ASM_DISKGROUP视图中的REQUIRED_MIRROR_FREE_MB和USABLE_FILE_MB列报告没有Flex磁盘组的值。

关于Oracle ASM文件组

Oracle ASM文件组是一组共享同一组属性和特征的文件。

文件组的一个主要好处是能够为共享同一磁盘组的每个数据库使用不同的可用性规范。

Oracle ASM文件组的属性集包括冗余，重新平衡重建优先级，重新平衡功率限制，客户端兼容性，条带化，配额组和访问控制列表。

以下列表提供了有关文件组的重要说明。

  • 磁盘组至少包含一个文件组，即默认文件组。

  • 磁盘组可以包含多个文件组。

  • 磁盘组必须具有FLEX或EXTENDED冗余才能包含文件组。

  • 磁盘组可以存储属于多个数据库的文件，每个数据库都有一个单独的文件组。

  • 数据库在磁盘组中只能有一个文件组。

  • 数据库可以跨多个磁盘组，其中多个文件组位于不同的磁盘组中。属于多个磁盘组中的数据库的文件组应该具有相同的名称，以便一致性和易于识别。

  • 文件组只能属于一个磁盘组。

  • 文件组只能描述一个数据库，PDB，CDB，卷或集群。

  • Flex磁盘组中的Oracle ADVM卷的文件组始终具有关联卷的名称。文件组中只能有一个卷。

  • 文件组只能属于一个配额组。

  • 自动创建的文件组与通用配额组关联。

  • 创建数据库，PDB或CDB时，如果现有文件组的客户端ID或名称与数据库，PDB或CDB的ID或名称匹配，则该文件组用于描述其文件。否则，将为数据库，PDB或CDB创建新的文件组。

  • 由于create database操作而自动创建的文件组被标识为系统创建的。删除数据库时，将删除此系统创建的文件组。但是，如果手动创建文件组，则在删除相关数据库时不会自动删除该文件组。您必须显式删除手动创建的文件组。

关于Oracle ASM文件组属性

以下列表总结了文件组的属性。

  • COMPATIBLE.CLIENT

    此文件组属性的当前值是文件打开操作期间的最后一个已知客户端兼容性。

    必须将COMPATIBLE.RDBMS磁盘组属性设置为小于或等于：

    • 每个文件组的COMPATIBLE.CLIENT （设置时）

    • COMPATIBLE数据库初始化参数（对于连接到Oracle ASM的每个数据库）

    • 磁盘组的COMPATIBLE.ASM

    您必须是Oracle ASM管理员才能修改此属性。

  • DBCLONE_STATUS

    此文件组属性指定数据库拆分操作的状态。无法手动设置此属性。

    在与此文件组关联的数据库的拆分操作期间，文件组属性包含以下值之一。

    • 制备

      准备数据库以进行拆分的过程已经开始，但数据库尚未准备好进行拆分。

    • 备

      准备过程已完成，数据库已准备好拆分。

    • 分裂

      已经开始拆分数据库文件的过程。

    • 分裂完成

      数据库已被拆分。

    • DROPPING

      删除准备好的数据库克隆的过程已经开始。

    • 失败

      克隆操作失败。重新平衡操作应该删除不完整的克隆。

  • 所有者

    此文件组属性指定OWNER设置。要设置OWNER属性，必须将ACCESS_CONTROL.ENABLED磁盘组属性设置为TRUE以启用Oracle ASM文件访问控制。指定为所有者的用户必须是磁盘组的成员。

    只有Oracle ASM管理员的OWNER ，该OWNER的的USER_GROUP与文件组关联，以及成员USER_GROUP与文件组关联可以修改标记有文件组的属性A database administrator can modify this property 。仅当ACCESS_CONTROL.ENABLED为true时才会出现ACCESS_CONTROL.ENABLED情况。

    您可以使用V$ASM_FILEGROUP视图的USER_NUMBER和USER_INCARNATION列显示所有者。USER_NUMBER和USER_INCARNATION的值可以与V$ASM_USER视图的USER_NUMBER和INCARNATION列连接，以显示用户名。

    默认值为NULL 。

    您必须是Oracle ASM管理员才能修改此属性。

  • PARENT_FILEGROUP_NUMBER

    此文件组属性指定父文件组编号。无法手动设置此属性。

    准备拆分数据库时，将为影子文件创建子文件组。子文件组自动设置PARENT_FILEGROUP_NUMBER属性。

  • POWER_LIMIT

    此文件组属性指定重新平衡功率，该功能确定此文件组的磁盘重新平衡的并行重定位数。有效设置为1到1024 。重新平衡文件组时，将按如下方式应用功率限制值：

    1. 如果已在ALTER DISKGROUP REBALANCE SQL语句或ASMCMD rebal命令中明确指定了power选项，则ALTER DISKGROUP语句或ASMCMD rebal命令中指定的功率值将用于重新平衡所有文件组。

    2. 如果未在SQL重新平衡语句或ASMCMD重新平衡命令中指定电源选项，则文件组POWER_LIMIT属性值（如果已设置）适用于文件组重新平衡。

    3. 如果未在SQL重新平衡语句或ASMCMD重新平衡命令中指定电源选项，并且尚未为文件组设置POWER_LIMIT属性，则电源值默认为文件组重新平衡的ASM_POWER_LIMIT初始化参数的值。

    您必须是数据库管理员才能修改此属性。

  • 优先

    此文件组属性指定重新平衡优先级，并确保在其他文件之前重新平衡某些类型的文件。优先级最高的文件组首先完全重新平衡，然后是具有次高优先级的文件组，依此类推到最低优先级。

    您可以将重新平衡优先级设置为HIGHEST ， HIGH ， MEDIUM ， LOW或LOWEST 。默认值为MEDIUM 。

    您必须是数据库管理员才能修改此属性。

  • QUOTA_GROUP

    此文件组属性指定此文件组所属的配额组的名称。有效设置是现有配额组名称。默认值为GENERIC 。磁盘组中的每个文件组都必须是配额组的成员。

    您必须是Oracle ASM管理员才能修改此属性。

  • 冗余

    此文件类型属性指定文件组的冗余。每种文件类型的默认冗余设置都是从系统模板派生的。有效设置为HIGH ， MIRROR ， PARITY和UNPROTECTED 。

    为减少与Oracle ASM相关的数据保护开销，为Flex磁盘组提供单奇偶校验保护。奇偶校验冗余仅限于不需要持续更新的数据，以减少对奇偶校验保护的执行影响。PARITY设置适用于一次写入文件，例如存档日志和备份集，并且不支持数据文件和读/写文件。PARITY在Flex磁盘组中至少需要三个常规（非仲裁）故障组。如果在创建奇偶校验文件时有三个或四个故障组，则每个奇偶校验范围集都有两个数据范围。对于双向镜像文件，这种情况会产生50％的冗余开销，而不是100％的冗余开销。如果在创建奇偶校验文件时存在五个或更多故障组，则每个奇偶校验范围集具有四个数据范围。这种情况导致25％的冗余开销。

    创建文件后，可以更改文件的冗余。如果冗余已更改，则V$ASM_FILE的REMIRROR列包含Y以指示文件需要新镜像，启动重新平衡以使新冗余生效。重新平衡完成后， REMIRROR列中的值包含N

    将文件组冗余属性从HIGH ， MIRROR或UNPROTECTED设置修改为PARITY设置时，文件组中现有文件的冗余不会更改。此行为也适用于从PARITY更改为HIGH ， MIRROR或UNPROTECTED设置。但是，将来创建的任何文件都采用新的冗余设置。

    如果文件组中的冗余增加，则重新平衡完成后， V$ASM_FILE的REDUNDANCY列的值将增加。但是，如果文件组中的冗余减少，则在重新平衡运行之前， V$ASM_FILE的REDUNDANCY列的值会立即降低。

    更改文件类型粒度或整个文件组的冗余会影响所有文件类型。

    您必须是数据库管理员才能修改此属性。

  • STRIPING

    此文件类型属性指定每种文件类型的STRIPING设置。通常，每种文件类型的默认值都是足够的，不会更改。

    大多数文件都有COARSE条带，但默认情况下具有FINE条带的控制文件除外。有效设置为COARSE和FINE 。默认值为COARSE或从文件类型模板派生。

    STRIPING仅影响新创建的文件，并且在创建文件后无法更改。

    您必须是数据库管理员才能设置此属性。

  • USER_GROUP

    此文件组属性指定USER_GROUP设置。要设置USER_GROUP属性，必须将ACCESS_CONTROL.ENABLED磁盘组属性设置为TRUE以启用Oracle ASM文件访问控制。指定的用户组必须是现有用户组。

    只有Oracle ASM管理员的OWNER ，该OWNER的的USER_GROUP与文件组关联，以及成员USER_GROUP与文件组关联可以修改标记有文件组的属性A database administrator can modify this property 。仅当ACCESS_CONTROL.ENABLED为true时才会出现ACCESS_CONTROL.ENABLED情况。

    您可以使用V$ASM_FILEGROUP视图的USERGROUP_NUMBER和USERGROUP_INCARNATION列显示用户组信息。USERGROUP_NUMBER和USERGROUP_INCARNATION的值可以与V$ASM_USERGROUP视图的USERGROUP_NUMBER和INCARNATION列结合使用，以显示用户组名称。

    在文件组中创建的文件将继承USER_GROUP属性设置。

    克隆继承USER_GROUP属性设置。

    默认值为NULL 。

    您必须是Oracle ASM管理员才能修改此属性。

关于Oracle ASM文件组的配额组

配额组定义分配给一组Oracle ASM文件组的配额。文件组只属于一个配额组。有关文件组的信息，请参阅关于Oracle ASM文件组 。

配额可以描述不同数据库使用的空间总和。

以下列表提供了有关文件组的配额组的重要说明。

  • 文件组只能属于一个配额组。

  • 配额组不能跨越多个磁盘组。

  • 配额组描述一个文件组或同一磁盘组中的多个文件组使用的空间聚合。

    PDB的文件组和该PDB的克隆的文件组可以是文件组将共享公共配额组的方案。

    默认情况下，文件组克隆会继承父级的配额组。但是，克隆可能属于不同的配额组。

  • 创建文件并调整其大小时，将强制执行配额。

  • 配额是物理空间。

    因此，如果配额限制已设置为10 M，则双向镜像6 M文件超出配额。

  • 每个配额组都有两个值：限制和当前使用的空间。

    可以将限制值设置为低于当前使用的空间。此操作可防止为与此配额组关联的文件组描述的文件分配任何其他空间。

  • 无论目标配额组是否有足够的空间用于文件组，都可以将文件组从一个配额组移动到另一个配额组。

    此功能使Oracle ASM管理员可以停止特定文件组描述的任何文件分配额外空间。

您可以使用SQL和ASMCMD命令管理配额组。以下示例说明如何使用SQL语句将配额组添加到磁盘组，以及如何将文件组从其当前配额组移动到新配额组。

ALTER DISKGROUP DiskGroup_2 ADD QUOTAGROUP QuotaGroup_QGRP3 SET'quota'= 10m;

示例4-20将文件组移动到新的配额组

ALTER DISKGROUP DiskGroup_2 MODIFY FILEGROUP FileGroup_PDB1 SET'quota_group'='QuotaGroup_QGRP3';

关于Oracle ASM扩展磁盘组

Oracle ASM扩展磁盘组具有Flex磁盘组的所有功能，其特定设计在扩展（拉伸）群集环境中具有高可用性，该环境包含跨多个物理上分离的站点的节点。

以下是扩展磁盘组的特征：

  • 扩展磁盘组的冗余设置设置为EXTENDED REDUNDANCY ，扩展磁盘组的每个文件组都有自己的冗余属性设置。

  • 为每个站点指定扩展磁盘组中的文件和文件组的冗余，而不是为每个磁盘组指定。

    这意味着由REDUNDANCY设置确定的份数乘以数据站点的数量。如果REDUNDANCY设置为MIRROR ，那么指定两个副本，导致两个数据站点总共四个副本。

  • 扩展磁盘组可以容忍整个站点的丢失，以及另一个站点中最多两个故障组的丢失。

    这意味着除了故障组级别之外，扩展磁盘组还可以容忍站点级别的故障。

  • 创建磁盘组期间，所有数据站点必须具有相同数量的故障组。

  • 为每个磁盘组指定配额组，而不是为每个站点指定配额组。

  • 配额组限制是所有站点中所有副本所需的物理空间。

    例如，对于两个数据站点，冗余设置为MIRROR的6 M文件使用24 M的配额限制。

  • 必须总共有三个站点，两个数据站点和一个仲裁站点才能创建扩展磁盘组。每个数据站点应具有三个故障组，仲裁站点应具有一个故障组。

  • 对于数据库文件，冗余设置由扩展磁盘组的文件组的冗余设置决定。

  • COMPATIBLE.ASM和COMPATIBLE.RDBMS磁盘组属性必须设置为12.2或更高。

  • 最小分配单位（AU）大小为4 M.

  • 不支持从其他磁盘组类型转换为扩展磁盘组。

  • 如果已配置Oracle Exadata，则可以从单元推断出故障组的SITE规范。否则，可以在创建磁盘组时使用SQL语句指定站点规范。

关于使用Oracle ASM Flex磁盘组进行时间点数据库克隆

在创建时间点数据库克隆之前，请查看以下要求。

注意：

  • 仅在Oracle ASM flex和扩展磁盘组中支持时间点数据库克隆。

  • 只有Oracle Database 18c（版本18.1或更高版本）支持时间点数据库克隆。

  • 时间点数据库克隆需要磁盘组兼容性属性COMPATIBLE.ASM和COMPATIBLE.RDBMS设置为18.0或更高版本。

  • 源数据库（父）必须是可插拔数据库。创建的数据库克隆（子）是同一容器数据库下的可插拔数据库。

Oracle ASM支持创建时间点数据库克隆，该克隆是可插拔数据库的镜像副本。要克隆的数据库的所有数据文件都应存在于单个磁盘组中。克隆操作可用于快速配置相同的副本。您可以在数据库克隆中指定数据库数据文件的冗余设置，或者该设置可以默认为正在克隆的源数据库中的数据文件的冗余。

在克隆过程中创建的数据文件副本最初与源相同。但是，在克隆过程完成后，对源的更新不会导致对已克隆的子项进行任何更改。V$ASM_DBCLONE_INFO视图显示源数据库，克隆数据库及其文件组之间的关系。应在Oracle ASM实例上运行V$ASM_DBCLONE_INFO 。

在本主题中显示的示例中，源数据库存在于DATA flex磁盘组中。

数据库克隆包含以下步骤：

  1. 准备镜像副本

    在此步骤中，Oracle ASM为其他数据副本分配空间。此过程涉及创建克隆文件并将其与源文件链接。但是，此时不会复制数据;在重镜像期间完成复制。在重新平衡的准备阶段进行重镜像，这是作为此步骤的一部分启动的。任何时候都只能存在一个镜像副本。给定数据库或可插拔数据库的所有数据文件都应驻留在单个磁盘组中。必须先使用镜像副本创建数据库克隆，然后才能创建另一个镜像副本。

    将创建一个文件组，并且所有克隆的文件都将成为此文件组的一部分。文件组中的数据库数据文件的冗余设置为您指定的值，或者默认值为源文件组的冗余设置。与克隆关联的配额组与源数据库的配额组相同。

    与克隆关联的Oracle ASM文件访问控制属性与与源数据库关联的属性相同。例如，数据库克隆的文件访问控制用户组属性与源数据库的属性相同。

    在此步骤结束时以及开始下一步之前，您可以删除准备好的镜像副本，如删除准备好的镜像副本中所述。

  2. 拆分镜像副本并创建数据库克隆

    准备和复制步骤必须在开始此步骤之前完成。您可以在Oracle ASM实例上显示V$ASM_DBCLONE_INFO视图，以查询正在进行的准备操作的状态。当文件组准备好拆分时， V$ASM_DBCLONE_INFO视图的DBCLONE_STATUS列指示PREPARED 。

      例如，以下LDIF语法将所有权从OracleNetAdmins组更改为另一个名为ExampleSecurityAdmins组。该组可以在Oracle Context内部或外部。 

      dn：cn = OracleNetAdmins，cn = OracleContext，...changetype：modify add：owner owner：cn = ExampleSecurityAdmins

    2. （可选）指定要作为所有者删除的组。

      dn：cn = OracleNetAdmins，cn = OracleContext，...changetype：modify delete：owner owner： DN of group to delete

  2. 在命令行中输入以下命令以刷新文件：

    $ ldapmodify -h directory_host -p port -D binddn -q -f ldif_file

    在上面的命令中， directory_host是目录服务器主机， port是目录服务器的侦听TCP / IP端口， binddn是目录管理员或用户DN， ldif_file是输入文件名。如果未指定端口，则使用默认端口389。-q选项提示输入单个绑定密码。

7.3使用监听器控制实用程序管理监听器

Oracle Net Services提供的工具可帮助您启动，停止，配置和控制每个网络组件。Listener Control实用程序使您可以管理侦听器。该实用程序由拥有Oracle安装的用户或指定组的成员在运行侦听器的同一台计算机上启动。该实用程序的基本语法如下：

lsnrctl command [ listener_name ]

例如，以下命令启动名为lsnr的侦听lsnr ：

lsnrctl START lsnr

您还可以在LSNRCTL>程序提示符下发出监听器控制实用程序命令。要获取提示，请在操作系统命令行中输入不带参数的lsnrctl 。运行lsnrctl ，将启动该实用程序，您可以从程序提示符输入必要的命令。

例如：

lsnrctl LSNRCTL> START lsnr
