新的ServerJDBCTheme（名字）

构造一个新的动态sql主题（JDBC主题）。例如：


    var jdbcTCounties = new OM.server。ServerJDBCTheme（ 'theme_jdbc_counties'）;

在实例化ServerJDBCTheme之后，可以添加其他属性。例如：


    jdbcTCounties.setDataSourceName（ 'mvdemo'）;
    jdbcTCounties.setSRID（ '8307'）;
    jdbcTCounties.setGeometryColumnName（ '的geom'）;
    var sql ='select totpop，poppsqmi，county，state_abrv，geom from countyies';
    jdbcTCounties.setQuery（SQL）;
    jdbcTCounties.addInfoColumn（{column：'state_abrv'，name：'State'}）;
    jdbcTCounties.addInfoColumn（{column：'county'，name：'County'}）;

可以将这样的主题添加到服务器映射请求实例中。
例如：


  • var req = new OM.server。ServerMapRequest（基本URL）;

  • req.addTheme（jdbcTCounties）;


警告：由于ServerJDBCTheme将向数据库发送大量不可重用的SQL查询，

O2F-00301：尝试从INTYPE文件中读取时出错

原因：尝试从INTYPE文件读取时发生操作系统错误。

操作：尝试使用编辑器打开INTYPE文件以确保您能够访问该文件。

O2F-00302：无法分配内存

原因： OTT无法分配内存。

行动：联系ORACLE客户支持。

O2F-00303：未指定INTYPE文件

原因：未指定所需的选项INTYPE。

操作：将INTYPE选项设置为包含要转换的OTT的类型名称的文件的名称。可以在命令行或CONFIG文件中指定INTYPE选项。

O2F-00304：INTYPE文件的文件名无效

原因：为INTYPE文件指定的文件名在语法上不正确。

操作：确保正确拼写INTYPE文件的文件名。

O2F-00305：打开INTYPE文件进行读取时出错

原因：尝试打开INTYPE文件时发生操作系统错误。

操作：确保正确拼写INTYPE文件的文件名。尝试使用编辑器打开INTYPE文件，以确保您可以访问该文件。

O2F-00306：无法关闭INTYPE文件

原因：尝试关闭INTYPE文件时发生操作系统错误。

操作：尝试使用编辑器打开INTYPE文件以确保您能够访问该文件。

O2F-00307：无法读取INTYPE文件

原因：未读取INTYPE文件的第一个标记。尝试从INTYPE文件读取时发生操作系统错误，或者INTYPE文件的所有行都是空行或注释行。

操作：尝试使用编辑器打开INTYPE文件以确保您能够访问该文件。确保INTYPE文件包含一个或多个类型规范。

O2F-00308：INTYPE文件中未显示任何类型规范

原因： INTYPE文件中没有类型规范，或者由于先前报告的错误而无法读取第一个类型规范。

操作：修复以前报告的任何错误。确保INTYPE文件中有一个或多个类型规范。

O2F-00309：用户定义类型的名称是非法或缺失的

原因：在类型规范开头的关键字TYPE之后，应该是用户定义类型的名称。没有看到合法的类型名称。

操作：确保类型规范在语法上是正确的，并且类型名称拼写正确。

O2F-00310：标识符名称非法或缺失

原因： AS之后预计会有合法的C或C ++标识符，但是没有看到。

操作：使用合法的C或C ++标识符，包括不以数字开头的字母，数字和下划线。

O2F-00311：文件名非法或缺失

原因：预计会有合法的文件名，但是没有看到。

操作：确保您的TYPE文件语法正确。使用正确的文件名。

O2F-00312：非法或缺少类型版本字符串

原因：预计会出现类型版本名称，但未见到。

操作：确保您的TYPE文件语法正确。使用正确的版本名称。

O2F-00313：属性名称非法或缺失

原因： INTYPE文件中需要一个属性名称，但没有看到。

操作：确保您的TYPE文件语法正确。使用正确的属性名称。

O2F-00314：AS预期但未在INTYPE文件中看到

原因： INTYPE文件中的语法不正确。

操作：确保您的TYPE文件语法正确。

O2F-00315：类型规范中列出了两次类型的属性

原因：在INTYPE文件的类型规范中列出了两次用户定义类型的属性。第二次出现被忽略。

操作：删除属性的两次出现之一。

O2F-00316：在类型规范的末尾看到意外的令牌

原因： INTYPE文件中存在语法错误。

操作：检查文档以确保INTYPE文件中的类型规范在语法上是正确的。检查您的文档以确保类型规范

O2F-00317：打开ERRTYPE文件进行写入时出错

原因：尝试打开ERRTYPE文件进行写入时发生操作系统错误。

操作：确保正确拼写ERRTYPE文件名。尝试创建或修改ERRTYPE文件以确保您具有该文件的写入权限。

O2F-00318：写入OUTTYPE文件时出错

原因：尝试写入OUTTYPE文件时发生操作系统错误。

操作：确保您具有写入文件的操作系统权限。另外，请确保磁盘空间可用。

O2F-00319：未指定OUTTYPE文件

原因：未指定所需的选项OUTTYPE。

操作：将OUTTYPE选项设置为应写入OTT生成的TYPE文件的文件的名称。可以在命令行或CONFIG文件中指定OUTTYPE选项。

O2F-00320：OUTTYPE文件的文件名无效

原因：为OUTTYPE文件指定的文件名在语法上不正确。

操作：确保正确拼写OUTTYPE文件的文件名。

O2F-00321：打开OUTTYPE文件进行写入时出错

原因：尝试打开OUTTYPE文件时发生操作系统错误。

操作：确保正确拼写OUTTYPE文件的文件名。确保您具有操作系统权限才能打开该文件。

O2F-00322：无法关闭OUTTYPE文件

原因：尝试关闭OUTTYPE文件时发生操作系统错误。

操作：尝试使用编辑器打开OUTTYPE文件以确保您能够访问该文件。

O2F-00323：OTT设施O2F中的内部错误

原因： OTT的O2F组件发生内部OTT错误。

行动：联系ORACLE客户支持。

O2F-00324：无法从数据库字符集转换为编译器字符集

原因：当OTT被调用到编译器字符集（通常为ASCII或EBCDIC）时，无法从有效的客户端字符集转换类型，属性或方法的名称。

操作：首先将名称从数据库字符集转换为客户端字符集，然后从客户端字符集转换为编译器字符集。客户端字符集以与平台相关的方式指定（在UNIX平台上，通过设置NLS_LANG环境变量来指定客户端字符集）。使用可以转换为编译器字符集的客户端字符集。编译器字符集本身就是一个这样的字符集。在INTYPE文件中为无法成功转换为客户端字符集的任何数据库对象名称指定同义词。

O2F-00325：名称中的所有字符均不是合法标识符字符

原因：数据库实体名称中的任何字符都不能转换为可能以C或C ++标识符开头的字符，例如字母或下划线。

操作：在INTYPE文件中为名称指定显式转换。

O2F-00326：此名称中的某些字符不是合法标识符字符

原因：数据库实体名称中的一个或多个字符无法转换为可能出现在C或C ++标识符中的字符，例如字母，下划线或数字。无法翻译的字符被替换为下划线。

操作：确保名称拼写正确，并且由空格，行尾或等号（“=”）终止。您可以使用已添加下划线的已翻译名称，也可以在INTYPE文件中为名称指定显式转换。

O2F-00327：CASE选项具有非法值

原因： INTYPE文件中CASE选项的值不合法。

操作：将为CASE选项指定的值更改为以下合法值之一：SAME，LOWER，UPPER或OPPOSITE。

O2F-00328：非法的INITFUNC名称

原因： INTYPE文件中为INITFUNC选项指定的函数名称不是合法的C或C ++标识符。

操作：将INITFUNC函数的名称指定为合法的C或C ++标识符。此名称可以在INTYPE文件中给出，在命令行中给出，或者从INITFILE名称派生。

O2F-00329：预期的类型或选项规范

原因： INTYPE文件中需要一个类型或选项规范，但没有看到。

操作：检查INTYPE文件是否存在语法错误。提供类型或选项规范。类型规范以单词TYPE开头。选项规范以CASE，INITFILE或INITFUNC开头。

O2F-00330：名称长度超过265个字节

原因： INTYPE文件中看到的名称长度超过265个字节。

操作：选择较短的名称。

O2F-00331：INTYPE文件中出现语法上非法的名称

原因： INTYPE文件中使用的名称出现语法错误。

操作：如果引用了名称，请确保引号位于名称的开头和末尾。如果出现句点（将模式名称与类型名称分开），请确保模式名称和类型名称都存在。

O2F-00332：数据库链接可能不会出现在用户定义类型的名称中

原因：在用户定义类型的名称中看到了数据库链接。此功能尚不支持。

操作：在OTT连接的数据库中声明用户定义的类型。

O2F-00333：在INTYPE文件中出现关键字的位置出现意外令牌

原因： INTYPE文件中预计会出现TYPE等关键字，但会看到错误或拼写错误的关键字。

操作：验证您的TYPE规范是否遵循文档中描述的语法。验证您的关键字拼写是否正确。

O2F-00334：在INTYPE文件中多次列出用户定义的类型

原因：用户定义类型的相同名称出现在多个类型规范中。类型名称的重复规范将被忽略。

操作：将有关类型的所有信息（包括所有类型名称和属性名称转换）放在单个TYPE规范中，并消除重复的TYPE规范。

O2F-00335：OTT的O2F组件内部错误

原因：请求数据库模式中的所有类型都是数据库。

操作：列出要在INTYPE文件中处理的类型。联系ORACLE客户支持。

O2F-00336：子系统报告错误：\ n

原因：发出请求以生成数据库模式中所有类型的声明。访问数据库时子系统中发生错误。

操作：检查sybsystem报告的错误消息以确定其原因。列出要在INTYPE文件中处理的类型。

O2F-00337：未找到架构，或架构中找不到类型

原因：发出请求以生成数据库模式中所有类型的声明。找不到架构，在架构中找不到用户声明的类型，或者无法连接到Oracle。

操作：确保为OTT指定了登录/密码字符串，并且可以使用它访问Oracle数据库。确保USER_TYPES表至少包含一行。

2 SQL Performance Analyzer简介

您可以在生产系统或与生产系统非常相似的测试系统上运行SQL Performance Analyzer。 在生产系统上测试系统更改将影响系统的吞吐量，因为SQL Performance Analyzer必须执行您正在测试的SQL语句。在系统上进行的任何全局更改以测试性能影响也可能会影响系统的其他用户。如果系统更改不会影响许多会话或SQL语句，则可以接受在生产系统上运行SQL Performance Analyzer。但是，对于系统范围的更改 - 例如数据库升级 - 不建议使用生产系统。相反，请考虑在单独的测试系统上运行SQL Performance Analyzer，以便您可以在不影响生产系统的情况下测试系统更改的效果。使用测试系统还可确保生产系统上运行的其他工作负载不会影响SQL Performance Analyzer执行的分析。在测试系统上运行SQL Performance Analyzer是推荐的方法和此处描述的方法。如果选择在生产系统上运行SQL Performance Analyzer，则将生产系统替换为适用的测试系统。

2.1捕获SQL工作负载

在运行SQL Performance Analyzer之前，在生产系统上捕获一组SQL语句，这些语句表示您要分析的SQL工作负载。

捕获的SQL语句应包含以下信息：

  • SQL文本

  • 执行环境

    • SQL绑定，它是执行SQL语句和生成准确执行统计信息所需的绑定值

    • 解析可以编译SQL语句的模式

    • 编译环境，包括执行SQL语句的初始化参数

  • 执行SQL语句的次数

捕获SQL工作负载对生产系统的性能影响可忽略不计，不应影响吞吐量。包含更多SQL语句的SQL工作负载将更好地表示应用程序或数据库的状态。这将使SQL Performance Analyzer能够更准确地预测系统更改对SQL工作负载的潜在影响。因此，您应该捕获尽可能多的SQL语句。理想情况下，您应捕获应用程序调用或正在数据库上运行的所有SQL语句。

您可以将捕获的SQL语句存储在SQL调优集中，并将其用作SQL Performance Analyzer的输入源。SQL调优集是一个数据库对象，包括一个或多个SQL语句，以及它们的执行统计信息和执行上下文。SQL语句可以从不同的源加载到SQL调优集中，包括游标缓存，自动工作负载存储库（AWR），SQL跟踪文件和现有的SQL调优集。使用SQL调优集捕获SQL工作负载使您能够：

  • 将SQL文本和任何必要的辅助信息存储在单个持久数据库对象中

  • 在SQL调优集中填充，更新，删除和选择捕获的SQL语句

  • 加载和合并来自各种数据源的内容，例如自动工作负载存储库（AWR）或游标缓存

  • 从捕获SQL工作负载的系统导出SQL调优集，并将其导入另一个系统

  • 将SQL工作负载重用为其他顾问程序的输入源，例如SQL Tuning Advisor和SQL Access Advisor

2.2设置测试系统

您应该配置测试数据库环境以尽可能地匹配生产系统的数据库环境。通过这种方式，SQL Performance Analyzer可以更准确地预测系统更改对SQL性能的影响。

2.4测量更改前的SQL性能

在进行系统更改之前创建一个更改前的SQL试用版。 您可以使用以下方法生成SQL Performance Analyzer的SQL试用所需的性能数据：

  • 测试执行

    此方法测试通过SQL Performance Analyzer执行SQL语句。这可以在运行SPA Performance Analyzer的数据库上或远程数据库上完成。

  • 解释计划

    此方法仅通过SQL Performance Analyzer为SQL语句生成执行计划。这可以在运行SPA Performance Analyzer的数据库上或远程数据库上完成。与EXPLAIN PLAN语句不同，使用explain plan方法的SQL试验EXPLAIN PLAN绑定值考虑在内并生成实际的执行计划。

  • 转换SQL调优集

    此方法转换存储在SQL调优集中的执行统计信息和计划。这仅适用于API。

测试执行方法运行工作负载中包含的每个SQL语句以完成。在执行期间，SQL Performance Analyzer生成执行计划并计算工作负载中每个SQL语句的执行统计信息。SQL调优集中的每个SQL语句都与其他SQL语句分开执行，而不保留其初始执行顺序或并发性。每个SQL语句至少执行两次，直到执行超时（最多10次）为止。第一次执行用于加热缓冲区缓存。然后，所有后续执行都将用于根据SQL语句的平均值计算SQL语句的运行时执行统计信息。执行SQL语句的实际次数取决于执行SQL语句所需的时间。长时间运行的SQL语句将仅执行第二次，并将使用此执行的执行统计信息。其他（运行速度更快）的SQL语句多次执行，并且它们的执行统计信息在这些执行中取平均值（计算中不使用第一次执行的统计信息）。通过对多次执行的统计信息进行平均，SQL Performance Analyzer可以为每个SQL语句计算更准确的执行统计信息。为避免对数据库造成潜在影响，不支持DDL。默认情况下，仅执行DML的查询部分。使用API，您可以使用EXECUTE_FULLDML任务参数执行完整的DML。不支持并行DML，除非删除并行提示，否则不会执行查询部分。

根据其大小，执行SQL工作负载可能会耗费大量时间和资源。使用说明计划方法，您可以选择仅生成执行计划，而无需收集执行统计信息。此技术缩短了运行试验的时间并减少了对系统资源的影响，但无法进行全面的性能分析，因为在分析过程中只有执行计划可用。但是，与使用EXPLAIN PLAN命令生成计划不同，SQL Performance Analyzer在生成执行计划时为优化程序提供绑定值，这可以更准确地预测执行SQL语句时的计划。

在这两种情况下，您都可以使用数据库链接在单独的数据库上远程执行SQL工作负载。SQL Performance Analyzer将使用数据库链接建立与远程数据库的连接，在该数据库上执行SQL语句，收集每个SQL语句的执行计划和运行时统计信息，并将结果存储在本地数据库的SQL试验中可以用于以后的分析。在您希望以下情况下，此方法很有用：

  • 测试数据库升级

  • 在运行其他版本的Oracle数据库的系统上执行SQL工作负载

  • 将SQL Performance Analyzer分析的结果存储在单独的测试系统中

  • 使用不同的硬件配置在多个系统上执行测试

  • 即使您在生产系统上使用旧版本的Oracle数据库，也请使用SQL Performance Analyzer中的最新功能

执行SQL工作负载后，生成的执行计划和运行时统计信息将存储在a中 SQL试用版。

您还可以使用存储在SQL调优集中的执行统计信息和计划来构建SQL试用版。虽然此方法仅支持API，但在您有另一种方法来运行工作负载（例如数据库重放或其他应用程序测试工具）的情况下，它可能很有用，并且您不需要SQL Performance Analyzer来驱动工作负载。测试系统。在这种情况下，如果在测试运行期间捕获SQL调优集，则可以使用SQL Performance Analyzer从这些SQL调优集构建SQL试验，以查看更全面的分析报告。与标准SQL性能分析器报告（每个试验中只有一个执行计划）和一组执行带有一组绑定的SQL语句生成的执行统计信息不同，您可以生成一个报告，比较从SQL调优集构建的SQL试验显示来自两个试验的所有执行计划，在多个执行中可能存在许多不同的绑定集。

2.7比较性能测量

SQL Performance Analyzer会在更改之前和之后比较SQL语句的性能，并生成一个报告，标识执行计划中的任何更改或SQL语句的性能。

SQL Performance Analyzer测量系统更改对SQL工作负载的总体执行时间以及工作负载中每个单独SQL语句的响应时间的影响。默认情况下，SQL Performance Analyzer使用已用时间作为比较指标。或者，您可以从各种可用的SQL运行时统计信息中选择用于比较的度量标准，包括：

  • CPU时间

  • 用户I / O时间

  • 缓冲区得到

  • 物理I / O.

  • 优化成本

  • I / O互连字节

  • 这些指标的任何组合都以表达式的形式出现

如果您选择仅在SQL试验中生成解释计划，则SQL Performance Analyzer将使用存储在SQL执行计划中的优化程序成本。

比较完成后，生成的数据将生成到SQL Performance Analyzer中报告，比较更改前和更改后的SQL性能。可以将SQL Performance Analyzer报告视为HTML，文本或活动报告。活动报告使用交互式用户界面提供深入报告，即使在与数据库或Oracle Enterprise Manager断开连接时也可以执行详细分析。

  • 读写

    在此模式下，用户可以查看分析工作区对象和数据，以及创建或更改对象。用户可以导出或导入对象。只有一个用户可以在读写模式下打开分析工作区，但任何数量的其他用户都可以在只读模式下打开它。这是默认模式。

2.5从Oracle OLAP 10 g升级元数据

通过将对象另存为XML模板并将XML导入到其他模式，可以将Oracle OLAP 10 g分析工作区升级到OLAP 11 g或12 c 。升级过程可以保持原始分析工作区的可访问性和未更改状态。

先决条件：

  • OLAP 10 g分析工作区可以使用OLAP标准表单元数据。

  • 原始关系源数据必须可用于加载到新的分析工作区。如果数据位于不同的模式中或表名不同，则必须在升级后将维对象重新映射到新的关系源。

  • 您可以在与OLAP 10 g分析工作区相同的架构中创建OLAP 12 c分析工作区。但是，如果选择在其他架构中创建OLAP 12 c分析工作区，则必须为新用户授予适当的权限，如“ 向DBA和应用程序开发人员授予权限 ”中所述 。

要升级OLAP 11 g分析工作区，请执行以下操作：

请注意，在需要对登台表进行额外索引的数据加载情况下，SQL * Loader可能是更好的选择。有关SQL * Loader与外部表之间的加载行为如何不同的详细信息，请参阅SQL * Loader和外部表之间的行为差异。

从Oracle Database 12 c第2版（12.2.0.1）开始，您可以对外部表中包含的数据进行分区，这样您就可以利用分区存储在数据库中的表时提供的相同性能改进（例如，分区修剪） 。

注意：

外部表可用作SQL语句中的内联外部表，因此无需在数据字典中创建外部表作为持久数据库对象。有关其他信息，请参见Oracle数据库SQL语言参考 。

创建外部表时，请指定以下属性：

  • TYPE - 指定外部表的类型。每种类型的外部表都由其自己的访问驱动程序支持。

    • ORACLE_LOADER - 这是默认的访问驱动程序。它将数据从外部表加载到内部表。数据必须来自文本数据文件。（ ORACLE_LOADER访问驱动程序无法执行卸载;也就是说，它无法将数据从内部表移动到外部表。）

    • ORACLE_DATAPUMP - 此访问驱动程序可以执行加载和卸载。数据必须来自二进制转储文件。从外部表加载内部表是通过从二进制转储文件中获取来完成的。通过填充外部表的二进制转储文件来完成从内部表到外部表的卸载。ORACLE_DATAPUMP访问驱动程序只能在使用SQL CREATE TABLE AS SELECT语句创建外部表的过程中写入转储文件。创建转储文件后，可以多次读取它，但不能修改它（也就是说，不能执行DML操作）。

    • ORACLE_HDFS - 提取存储在Hadoop分布式文件系统（HDFS）中的数据。

    • ORACLE_HIVE - 提取存储在Apache HIVE中的数据。

  • DEFAULT DIRECTORY - 指定用于未明确命名目录对象的所有输入和输出文件的默认目录。使用目录对象而不是目录路径指定位置。您必须在创建外部表之前创建目录对象;否则，会产生错误。有关更多信息，请参阅数据文件和输出文件的位置 。

  • ACCESS PARAMETERS - 描述外部数据源并实现指定的外部表的类型。每种类型的外部表都有自己的访问驱动程序，该驱动程序提供对该类外部表唯一的访问参数。访问参数是可选的。请参阅访问参数 。

  • LOCATION - 指定外部表的数据文件。

    • 对于ORACLE_LOADER和ORACLE_DATAPUMP ，文件在表单directory:file中命名。directory部分是可选的。如果缺少，则默认目录用作文件的目录。如果您使用的是ORACLE_LOADER访问驱动程序，则可以在文件名中使用通配符：星号（*）表示多个字符，一个问号（？）表示单个字符。

    • 对于ORACLE_HDFS ， LOCATION子句是目录或文件的统一资源标识符（URI）列表。没有与URI关联的目录对象。

    • 对于ORACLE_HIVE ，不使用LOCATION子句。而是读取Hadoop HCatalog表以获取有关数据源位置（可以是文件或其他数据库）的信息。

14.2数据文件和输出文件的位置

数据文件和输出文件必须位于服务器上。您必须具有一个目录对象，该对象指定从中读取和写入文件的位置。

注意：

本节中有关目录对象的信息不适用于ORACLE_HDFS访问驱动程序或ORACLE_HIVE访问驱动程序的数据文件。使用ORACLE_HDFS驱动程序，数据的位置使用目录或文件的URI列表指定，并且没有与URI关联的目录对象。 ORACLE_HIVE驱动程序未指定数据源位置;它读取Hive Metastore表以获取该信息，因此不需要目录对象。

访问驱动程序在数据库服务器内运行。此行为与SQL * Loader不同，后者是一个客户端程序，用于将要加载的数据发送到服务器。这种差异具有以下含义：

  • 服务器需要访问访问驱动程序可以加载的文件。

  • 服务器必须创建和写入访问驱动程序创建的输出文件：日志文件，错误文件，丢弃文件以及ORACLE_DATAPUMP访问驱动程序创建的任何转储文件。

要指定读取和写入文件的位置，访问驱动程序要求您使用目录对象。目录对象将名称映射到文件系统上的目录名称。例如，以下语句创建一个名为ext_tab_dir的目录对象，该对象映射到位于/usr/apps/datafiles.

CREATE DIRECTORY ext_tab_dir AS'/ usr / apps / datafiles';

DBA或任何用户都可以使用CREATE ANY DIRECTORY权限创建目录对象。

注意：

要在Oracle Real Applications Cluster（Oracle RAC）配置中使用外部表，必须确保目录对象路径位于群集范围的文件系统上。

创建目录后，创建目录对象的用户必须将该目录的READ和WRITE权限授予其他用户。必须显式授予这些特权，而不是使用角色分配。例如，要允许服务器代表ext_tab_dir指定的目录中的用户scott读取文件，创建目录对象的用户必须执行以下命令：

GRANT READ ON DIRECTORY ext_tab_dir TO scott;

SYS用户是唯一可以拥有目录对象的用户，但SYS用户可以授予其他用户创建目录对象的权限。对目录对象的READ或WRITE权限仅表示Oracle数据库代表您读取或写入该文件。除非您具有相应的操作系统权限，否则不能直接访问Oracle数据库之外的那些文件。同样，Oracle数据库需要操作系统的许可才能读取和写入目录中的文件。

14.4外部表使用期间的数据类型转换

读取外部表时以及写入外部表时可能会发生转换错误。

读取外部表时的转换错误

从外部表中选择行时，访问驱动程序将执行任何必要的转换，以使数据源中的数据与外部表中相应列的数据类型相匹配。根据所需的数据和转换类型，转换可能会遇到错误。

为了说明从外部表读取时可能发生的数据转换问题的类型，假设您创建以下外部表KV_TAB_XT其中包含两列：数据类型为VARCHAR2(4) KEY和数据类型为NUMBER VAL 。

SQL> CREATE TABLE KV_TAB_XT（KEY，VARCHAR2（4），VAL NUMBER）2 ORGANIZATION EXTERNAL 3（DEFAULT DIRECTORY DEF_DIR1 LOCATION（'key_val.csv'））;

外部表KV_TAB_XT使用访问参数的默认值，这意味着：

  • 记录由新行分隔。

  • 数据文件和数据库具有相同的字符集。

  • 数据文件中的字段具有相同的名称，并且与外部表中的列的顺序相同。

  • 字段的数据类型是CHAR(255) 。

  • 每个字段的数据以逗号结束。

KV_TAB_XT外部表的数据文件中的记录应为：

  • 一个字符串，最长4个字节。如果字符串为空，则该字段的值为NULL 。

  • 终止逗号。

  • 一串数字字符。如果字符串为空，则此字段的值为NULL 。

  • 可选的终止逗号。

当访问驱动程序从数据文件中读取记录时，它会验证数据文件中KEY字段的值的长度是否小于或等于4，并且它尝试转换数据中VAL字段的值文件到Oracle号码。

如果KEY字段的值的长度大于4，或者VAL的值中存在非数字字符，则ORACLE_LOADER访问驱动程序拒绝该行，这会导致写入该行的行的副本文件和写入日志文件的错误消息。

所有访问驱动程序都必须处理来自外部表的源的字段的数据类型的转换以及外部表的列的数据类型。以下是访问驱动程序执行的转换和检查类型的一些示例：

  • 将字符数据从源数据使用的字符集转换为数据库使用的字符集。

  • 从字符数据转换为数字数据。

  • 从数字数据转换为字符数据。

  • 从字符数据转换为日期或时间戳。

  • 从日期或时间戳转换为字符数据。

  • 从字符数据转换为间隔数据类型。

  • 从间隔数据类型转换为字符数据。

  • 验证字符列的数据值长度是否超过该列的长度限制。

当访问驱动程序遇到执行所需转换或验证的错误时，它可以决定如何处理错误。当ORACLE_LOADER和ORACLE_DATAPUMP访问驱动程序遇到错误时，它们会拒绝该记录并将错误消息写入日志文件。就好像该记录不在数据源中。当ORACLE_HDFS和ORACLE_HIVE访问驱动程序遇到错误时，遇到错误的字段的值将设置为NULL。这与Hive如何处理Hadoop中的错误的行为一致。

即使访问驱动程序已从数据源转换数据以匹配外部表列的数据类型，访问外部表的SQL语句也可能需要其他数据类型转换。如果任何这些额外的转换遇到错误，则整个语句将失败。（例外情况是，如果您使用SQL语句中的DML错误日志记录功能来处理这些错误。）这些转换与执行SQL语句时通常需要的转换相同。例如，假设您更改的定义KV_TAB_XT外部表只具有字符数据类型的列，然后执行INSERT语句从外部表将数据加载到具有另一个表NUMBER列数据类型VAL ：

SQL> CREATE TABLE KV_TAB_XT（KEY VARCHAR2（20），VAL VARCHAR2（20））2 ORGANIZATION EXTERNAL 3（DEFAULT DIRECTORY DEF_DIR1 LOCATION（'key_val.csv'））; 4 CREATE TABLE KV_TAB（KEY VARCHAR2（4），VAL NUMBER）; 5插入KV_TAB SELECT * FROM KV_TAB_XT;

在此示例中，如果VAL的数据包含非数字字符，则访问驱动程序不会拒绝记录，因为外部表中VAL的数据类型现在为VARCHAR2 （而不是NUMBER ）。但是，SQL执行现在需要处理从KV_TAB_XT字符数据类型到KV_TAB_XT中的数字数据类型的KV_TAB 。如果外部表中VAL的值中存在非数字字符，则SQL会引发转换错误并回滚所有已插入的行。为避免SQL执行中的转换错误，请尝试使外部表中列的数据类型与将使用这些列的值的其他表或函数所期望的数据类型相匹配。

写入外部表时的转换错误

ORACLE_DATAPUMP访问驱动程序允许您使用CREATE TABLE AS SELECT语句将数据卸载到外部表中。如果SELECT表达式中列的数据类型与外部表中列的数据类型不匹配，则会发生数据转换。如果SQL遇到转换数据类型的错误，则SQL将中止该语句，并且数据文件将不可读。

为避免导致操作失败的转换错误问题，外部表中列的数据类型应与用于写入外部表的源表或表达式中的列的数据类型相匹配。这并不总是可行的，因为外部表不支持所有数据类型。在这些情况下，必须将源表中不受支持的数据类型转换为外部表可以支持的数据类型。以下CREATE TABLE语句显示了以下示例：

CREATE TABLE LONG_TAB_XT（LONG_COL CLOB）组织外部...从LONG_TAB中选择TO_LOB（LONG_COL）;

名为LONG_TAB的源表具有LONG列，因此正在创建的外部表中的相应列LONG_TAB_XT必须是CLOB ，并且用于填充外部表的SELECT子查询必须使用TO_LOB运算符来加载列。

使用授权，角色和最小权限启用用户访问

本主题说明如何向用户授予权限和角色以限制对数据的访问。它还解释了最小特权概念的重要性，将安全应用程序角色作为一种自动过滤掉尝试登录应用程序的用户的方法。

用户权限是执行操作的权限 ，例如更新或删除数据。您可以授予用户执行这些操作的权限。角色被命名为组合在一起的权限集合，通常用于使用户能够执行与其作业相关的一组任务。例如，名为clerk的角色可以使职员执行创建，更新和删除文件等操作。clerk_mgr角色可以包括clerk角色，以及一些额外的权限，例如批准职员的费用报告或管理他们的绩效评估。

向用户授予权限时，应用最小权限原则： 仅授予用户所需的权限 。如果可能，请勿直接向用户授予权限。而是创建一个角色，定义用户需要的权限集，然后向用户授予此角色。例如，授予用户fred CREATE SESSION权限，以便他可以登录到数据库会话。但是对于他的工作所需的权限，例如UPDATE TABLE权限，授予他一个具有这些权限的角色。

您可以将应用程序设计为自动向尝试登录的用户授予角色，前提是用户满足您指定的条件。为此，您需要创建一个安全的应用程序角色 ，该角色与PL / SQL过程（或包含多个过程的PL / SQL包）相关联。该过程验证用户：如果用户验证失败，则用户无法登录。如果用户通过了验证，则该过程授予用户一个角色，以便他或她可以使用该应用程序。用户只有在登录应用程序时才具有此角色。用户注销时，该角色将被撤消。

有几种方法可以控制用户对应用程序数据的访问级别。

  • Oracle虚拟专用数据库（VPD）： VPD使您可以创建可以限制行和列级别的数据库访问的策略。实质上，VPD将动态WHERE子句添加到针对应用了VPD安全策略的表，视图或同义词发出的SQL语句。

    例如，假设用户输入SELECT * FROM OE.ORDERS语句。VPD策略可以将此语句转换为以下语句，以便只有拥有该订单的销售代表才能查看此数据：

    SELECT * FROM OE.ORDERS WHERE SALES_REP_ID = SYS_CONTEXT（'USERENV'，'SESSION_USER'）;

  • Oracle数据编辑： Oracle Data Redaction在用户尝试访问数据时（即查询执行时）在运行时屏蔽数据。该解决方案在动态生产系统中运行良好，其中数据不断变化。在数据被编辑期间，正常执行所有数据处理，并保留后端参照完整性约束。您通常会编辑敏感数据，例如信用卡或社会保险号。

    您可以通过以下方式屏蔽数据：

    • 完全编辑，其中整个数据被屏蔽。 例如，数字37828224可以显示为零。

    • 部分编辑，其中只有一部分数据被编辑。 使用此类型，数字37828224可以显示为***** 224。

    • 随机编辑，其中数据显示为随机数据。 数据37828224可以显示为93204857。

    • 正则表达式，您可以在其中根据搜索模式编辑数据。 您可以在完整和部分编校中使用正则表达式。例如，您可以编辑电子邮件地址的用户名，以便只显示域：电子邮件地址jsmith@example.com中的jsmith可以替换为[redacted]以便电子邮件地址显示为[redacted]@example.com 。

    • 无编辑，使您可以测试编校策略的内部操作，而不会影响对其上定义策略的表的查询结果。 您可以使用此选项在将编校策略定义应用于生产环境之前对其进行测试。

  • Oracle Label Security： Oracle Label Security在行级别保护数据库表，并根据站点的需要为这些行分配不同级别的安全性。可以为包含高度敏感数据的行分配一个标题为HIGHLY SENSITIVE的标签;敏感度较低的行可以标记为SENSITIVE ，依此类推。所有用户都可以访问的行可以标记为PUBLIC 。您可以根据需要创建任意数量的标签，以满足您网站的安全要求。

    例如，当用户fred （谁是低级员工）登录时，他将只看到PUBLIC标签定义的所有用户可用的数据。然而，当他的导演hortensia登录时，她可以看到所有被分配了高HIGHLY SENSITIVE标签的敏感数据。

  • Oracle Database Vault： Oracle Database Vault使您可以限制对数据的管理访问。默认情况下，管理员（例如具有SYSDBA权限的用户SYS ）可以访问数据库中的所有数据。管理员通常必须执行性能调整，备份和恢复等任务。但是，他们不需要访问您的工资记录。通过Database Vault，您可以创建限制管理员操作的策略，但不会阻止他或她执行正常的管理活动。

    例如，典型的Database Vault策略可能会阻止管理员访问和修改HR.EMPLOYEES表。您可以创建经过微调的策略，这些策略会施加限制，例如限制管理员可以登录的小时数，可以使用的计算机，是否可以使用动态工具登录数据库等等。此外，如果管理员尝试违反策略，您可以创建生成警报的策略。

什么是祈求者的权利和定义者的权利？

创建过程或函数（即程序单元）时，可以对其进行设计，使其以所有者（您）的权限或调用它的人员的权限运行。Definer的权利使用所有者的权限运行程序单元，并且调用者的权利使用运行它的人的权限运行程序单元。例如，假设用户harold创建一个更新表orders 。用户harold然后将此过程的EXECUTE特权授予用户hazel 。如果harold使用definer权限创建了该过程，那么该过程将期望orders表位于harold的模式中。如果他使用调用者的权限创建它，那么该过程将在hazel的模式中查找orders表。

要将程序单元指定为定义者权限或调用者权限，请在创建语句中使用AUTHID子句。如果省略AUTHID子句，则创建具有定义者权限的程序单元。

审核用户活动

您可以创建审核策略以审核数据库中的特定操作。然后，Oracle数据库将这些操作记录在审计跟踪中。数据库强制审计某些操作并将这些操作写入审计跟踪。您创建的审核策略可以很简单，例如审核特定用户的所有操作，或者复杂，例如测试特定条件并在违反这些条件时发送电子邮件警报。

安装Oracle数据库时，可以选择审核数据库的方式。

  • 统一审计：在统一审计中，所有审计跟踪都写入单个审计跟踪，可通过V$UNIFIED_AUDIT_TRAIL和GV$UNIFIED_AUDIT_TRAIL动态视图查看。此审计跟踪不仅包括Oracle数据库特定的操作，还包括在Oracle Real Application Security，Oracle Recovery Manager，Oracle Database Vault和Oracle Label Security环境中执行的操作。来自这些来源的审计记录以统一，统一的格式呈现。您可以创建命名审核策略，并根据需要启用和禁用它们。如果要使用统一审核，则必须将数据库迁移到该审核。

  • 统一审计和预发布12 c 审计的混合：在大多数情况下，此选项使您可以使用预发布12 c审计，其中审计记录使用自己的格式写入不同位置。但是，可以使用Release 12 c功能，例如在多租户环境中使用审计。此类审核是新数据库和升级数据库的默认设置。

在这两种情况下，将数据库升级到Oracle Database 12 c第1版（12.1.0.1）时，都会保留先前版本的审核记录。如果您决定迁移以完全使用统一审计，则可以归档这些早期记录，然后清除审计跟踪。完成迁移后，新的审核记录将写入统一审核跟踪。

Oracle Flex ASM概述

Oracle Flex ASM使Oracle ASM实例能够在与数据库服务器不同的物理服务器上运行。通过此部署，较大的Oracle ASM实例集群可以支持更多数据库客户端，同时减少整个系统的Oracle ASM占用空间。

使用Oracle Flex ASM时，Oracle ASM客户端配置为可直接访问存储。

使用Oracle Flex ASM，您可以将所有存储要求合并到一组磁盘组中。所有这些磁盘组都由在单个集群中运行的一小组Oracle ASM实例安装和管理。您可以使用基数设置指定Oracle ASM实例的数量。默认值为三个实例。

集群是一组提供组成员资格服务的节点。每个群集都具有全局唯一的名称。每个群集都有一个或多个Hub节点。Hub节点可以访问Oracle ASM磁盘。每个群集至少有一个专用网络和一个公共网络。如果群集将使用Oracle ASM进行存储，则它至少具有一个Oracle ASM网络。单个网络既可以用作专用网络，也可以用作Oracle ASM网络。出于安全原因，Oracle ASM网络永远不应公开。群集中只能运行一个Oracle Flex ASM配置。

Oracle ASM实例可以在Oracle Flex ASM中的多个配置中运行：

  • 可直接访问Oracle ASM磁盘的本地Oracle ASM客户端（标准Oracle ASM集群）

  • 可直接访问Oracle ASM磁盘的Oracle Flex ASM客户端

  • Oracle ACFS通过Oracle ASM代理实例进行访问

  • Oracle IOServer（IOS）与Oracle ASM磁盘组的基于网络的连接

9.138 V $ XSTREAM_APPLY_SERVER
