27.7.1 OCITypeArrayByName（）

给定一组名称时获取一个TDO数组。

目的

给定一组名称时获取一个TDO数组。

注意：

OCITypeArrayByName()不支持包级别类型。

句法

sword OCITypeArrayByName（OCIEnv * envhp，OCIError * errhp，const OCISvcCtx * svc，ub4 array_len，const OraText * schema_name []，ub4 s_length []，const OraText * type_name []，ub4 t_length []，const OraText * version_name []， ub4 v_length []，OCIDuration pin_duration，OCITypeGetOpt get_option，OCIType * tdo []）;

参数

envhp（IN / OUT）

OCI环境句柄在对象模式下初始化。有关详细信息，请参阅OCIEnvNlsCreate() ， OCIEnvCreate()和OCIInitialize() （不建议使用）的说明。

errhp（IN / OUT）

OCI错误句柄。如果有错误， OCI_ERROR err记录，此函数返回OCI_ERROR 。通过调用OCIErrorGet()获取诊断信息。

svc（IN）

OCI服务处理。

array_len（IN）

要检索的schema_name或type_name或version_name条目的数量。

schema_name（IN，可选）

与要检索的类型关联的模式名称数组。如果指定，数组必须具有array_len元素。如果提供0，则假定为默认模式;否则， schema_name必须具有array_len数量的元素。可以为一个或多个条目提供零（0），以指示这些条目需要默认模式。

s_length（IN）

schema_name长度数组，每个条目对应schema_name数组中对应的schema_name条目的长度（以字节为单位）。该数组必须具有array_len数量的元素，或者如果未指定schema_name则它必须为0。

type_name（IN）

要检索的类型的名称数组。这必须有array_len个元素。

t_length（IN）

type_name数组中类型名称长度的数组，以字节为单位。

version_name（IN）

将忽略版本名称，并返回所请求类型的最新版本。由于类型演变在9.0版本中可用，因此试图访问更改类型的9.0之前的应用程序会生成错误。必须使用最新的类型定义修改，重新编译和重新链接这些应用程序。

要检索的类型的版本名称数组对应。这可以是0表示检索最新版本，或者它必须具有array_len数量的元素。

如果提供0，则假定最新版本，否则它必须具有array_len数量的元素。可以为一个或多个条目提供零（0），以指示那些条目需要当前版本。

v_length（IN）

version_name数组中版本名称长度的数组（以字节为单位）。

pin_duration（IN）

检索到的类型的引脚持续时间（例如，直到当前事务结束）。有关每个选项的说明，请参阅oro.h

get_option（IN）

加载类型的选项。它可以是两个值之一：

  • OCI_TYPEGET_HEADER （仅加载标题）

  • OCI_TYPEGET_ALL （已加载TDO和所有ADO和MDO）

tdo（OUT）

指向对象缓存中每个固定类型的指针的输出数组。它必须有array_len指针的空间。使用OCIObjectGetObjectRef()获取每个固定类型描述符的CREF。

评论

获取指向与架构或类型名称数组关联的现有类型的指针。

您可以使用get_option参数来控制每次往返加载的TDO部分。

如果任何必需参数为NULL或者与模式或类型名称条目关联的任何对象类型不存在，则此函数返回错误。

要检索单个类型而不是数组，请使用OCITypeByName() 。

注意：

通过此函数获得的TDO（TDO数组或表定义）属于与传入的服务句柄（连接）对应的高速缓存的逻辑分区。如果跨逻辑分区使用TDO或表，则行为未知且可能在发布之间发生更改。

关于Oracle ASM过滤器驱动程序

Oracle ASM筛选器驱动程序（Oracle ASMFD）是一个内核模块，驻留在Oracle ASM磁盘的I / O路径中。Oracle ASM使用筛选器驱动程序验证对Oracle ASM磁盘的写入I / O请求。

Oracle ASMFD通过在每次重新启动系统时无需重新绑定与Oracle ASM一起使用的磁盘设备，从而简化了磁盘设备的配置和管理。

Oracle ASM筛选器驱动程序拒绝任何无效的I / O请求。此操作可消除意外覆盖Oracle ASM磁盘，从而导致磁盘组中的磁盘和文件损坏。例如，Oracle ASM筛选器驱动程序筛选出可能导致意外覆盖的所有非Oracle I / O.

关于Oracle ASMLIB和Oracle ASM过滤器驱动程序

从Oracle Grid Infrastructure 12 c第2版（12.2）开始，如果系统上已安装Oracle ASMLIB，则无法安装和配置Oracle ASM筛选器驱动程序（Oracle ASMFD）。如果要安装和配置Oracle ASMFD，必须先卸载Oracle ASMLIB。

关于分区表和Oracle ASM筛选器驱动程序

Oracle自动存储管理12.2第1版（12.2.0.1）中的Oracle ASM过滤器驱动程序（ASMFD）不支持扩展分区表。

  1. 在任何节点上以Oracle Grid Infrastructure所有者身份登录，以在此过程中运行命令。
  2. 列出现有磁盘组：
    $ $ ORACLE_HOME / bin / asmcmd lsdg
  3. 列出关联的磁盘：
    $ $ ORACLE_HOME / bin / asmcmd lsdsk -G diskgroup
  4. 检查Oracle ASM是否处于活动状态：
    $ $ ORACLE_HOME / bin / srvctl状态asm
  5. 停止数据库并卸载所有节点上的磁盘组：
    $ $ ORACLE_HOME / bin / srvctl stop diskgroup -diskgroup diskgroup -f
  6. 通过对Hub节点上的每个磁盘运行以下命令，标记磁盘组中的所有现有磁盘：
    $ $ ORACLE_HOME / bin / asmcmd afd_label 标签 disk_path --migrate
  7. 扫描所有Hub节点上的磁盘：
    $ $ ORACLE_HOME / bin / asmcmd afd_scan
  8. 启动数据库并在所有节点上装入磁盘组：
    $ $ ORACLE_HOME / bin / srvctl启动diskgroup -diskgroup diskgroup

8.1关于数据源

8.1.1 JNDI的Oracle数据源支持概述

JNDI标准为应用程序提供了一种查找和访问远程服务和资源的方法。这些服务可以是任何企业服务。但是，对于JDBC应用程序，这些服务将包括数据库连接和服务。

JNDI使应用程序能够在访问这些服务时使用逻辑名称，从应用程序代码中删除特定于供应商的语法。JNDI具有将逻辑名称与所需服务的特定源相关联的功能。

所有Oracle JDBC数据源都是JNDI可引用的。开发人员不需要使用此功能，但通过JNDI逻辑名访问数据库使代码更具可移植性。

注意：

使用JNDI功能需要jndi.jar文件位于CLASSPATH环境变量中。此文件包含在安装CD上的Java产品中。您必须单独将其添加到CLASSPATH环境变量。

8.1.2数据源的特性和属性

通过将数据源功能与JNDI一起使用，您无需注册特定于供应商的JDBC驱动程序类名，并且可以对URL和其他属性使用逻辑名称。这可确保打开数据库连接的代码可移植到其他环境。

DataSource接口和Oracle实现

JDBC数据源是实现标准javax.sql.的类的实例javax.sql.DataSource接口：

public interface DataSource {Connection getConnection（）抛出SQLException; Connection getConnection（String username，String password）抛出SQLException; ...}

Oracle使用oracle.jdbc.pool包中的OracleDataSource类实现此接口。重载的getConnection方法返回与数据库的连接。

要使用其他值，可以使用适当的setter方法设置属性。对于备用用户名和密码，您还可以使用将这些参数作为输入的getConnection方法。这将优先于属性设置。

注意：

OracleDataSource类和所有子类实现java.io.Serializable和javax.naming.Referenceable接口。

8.2数据库URL和数据库说明符

数据库URL是字符串。完整的URL语法是：

jdbc：oracle： driver_type ：[ 用户名 / 密码 ] @ database_specifier

注意：

  • 内部服务器端驱动程序kprb使用隐式连接。服务器端驱动程序的数据库URL在driver_type之后结束。

8.2.1支持Internet协议版本6

此版本的Oracle JDBC驱动程序支持JDBC URL中的Internet协议版本6（IPv6）地址和解析为IPv6地址的计算机名称。IPv6是由Internet工程任务组（IETF）设计的新的网络层协议，用于替换当前版本的Internet协议，Internet协议版本4（IPv4）。IPv6的主要优点是大地址空间，源自128位地址。IPv6还在路由，网络自动配置，安全性，服务质量等方面改进了IPv4。

注意：

  • IPv6客户端只能支持具有双协议支持的IPv6服务器或服务器，即支持IPv6和IPv4协议。相反，IPv6服务器只能支持IPv6客户端或双协议客户端。

  • IPv6仅支持单实例数据库服务器，而不支持Oracle RAC。

如果要在URL中使用文字IPv6地址，则应将左括号（ [ ）和右括号（ ] ）中包含的文字地址括起来。例如： [2001:0db8:7654:3210:FEDC:BA98:7654:3210] 。因此，使用IPv6地址的JDBC URL将如下所示：

jdbc：oracle：thin：@（DESCRIPTION =（ADDRESS =（PROTOCOL = tcp）（HOST = [2001：0db8：7654：3210：FEDC：BA98：7654：3210]）（PORT = 5521））（CONNECT_DATA =（SERVICE_NAME） = sales.example.com））

注意：

在数据库初始化期间启用Java时，将加载IPv6支持所需的所有新System类。因此，如果您的应用程序没有任何IPv6寻址，那么您无需更改代码即可使用IPv6功能。但是，如果您的应用程序只有IPv6或IPv6和IPv4寻址，那么您应该在命令行中设置java.net.preferIPv6Addresses系统属性。这使Oracle JVM能够加载适当的库。这些库加载一次，如果不重新启动Java进程就无法重新加载。

199 DBMS_XMLPARSER

使用DBMS_XMLPARSER ，您可以访问XML文档的内容和结构。XML描述了一类数据XML文档对象。它部分描述了处理它们的计算机程序的行为。通过构造，XML文档符合SGML文档。

XML文档由称为实体的存储单元组成，其中包含已解析或未解析的数据。解析数据由字符组成，其中一些字符形成字符数据，其中一些形成标记。标记编码文档的存储布局和逻辑结构的描述。XML提供了一种对存储布局和逻辑结构施加约束的机制。

称为XML处理器的软件模块用于读取XML文档并提供对其内容和结构的访问。假设XML处理器代表另一个模块（称为应用程序）执行其工作。XML处理器（或解析器）的PL / SQL实现遵循W3C XML规范REC-xml-19980210，并包含XML处理器在必须读取XML数据方面所需的行为以及它必须提供给应用程序的信息。

此PL / SQL XML解析器的默认行为是构建可由DOM API访问的解析树，如果找到DTD则验证它（否则，它是非验证的），并在指定错误日志时记录错误。如果解析失败，则会引发应用程序错误。

29.1.1存储大纲迁移的目的

如果您依赖存储的大纲来维护计划稳定性并防止性能下降，那么您可以安全地从存储的大纲迁移到SQL计划基准。迁移后，您可以保持相同的计划稳定性，同时受益于SQL计划管理框架提供的功能。

存储的轮廓具有以下缺点：

  • 存储的轮廓不能随着时间的推移自动发展。因此，存储的轮廓在创建时可能是最佳的，但在数据库更改后会成为次优计划，从而导致性能下降。

  • 存储轮廓中的提示可能会变得无效，就像删除索引上的索引提示一样。在这种情况下，数据库仍然使用轮廓但排除了无效提示，产生的计划通常比原始计划或优化程序生成的当前最佳成本计划更差。

  • 对于SQL语句，优化程序只能选择当前指定类别中存储大纲中定义的计划。优化程序无法从不同类别的其他存储轮廓或当前基于成本的计划中进行选择，即使它们可以提高性能。

  • 存储的大纲是一种反应性调整技术，这意味着您只需使用存储的大纲来解决性能问题。例如，您可以实现存储的大纲来更正变为高负载的SQL语句的计划。在这种情况下，您使用存储的大纲而不是在语句变为高负载之前主动调整语句。

存储的大纲迁移PL / SQL API通过以下方式帮助解决上述问题：

  • SQL计划基准使优化程序能够使用相同的最佳计划，并允许此计划随着时间的推移而发展。

    对于指定的SQL语句，您可以在验证它们之后将新计划添加为SQL计划基准，以免导致性能下降。

  • SQL计划基准可防止计划因无效提示而无法生成。

    如果存储在计划基准中的提示变为无效，则优化程序可能无法重现该计划。在这种情况下，优化程序会选择备用的可重现计划基准或优化程序生成的当前最佳成本计划。

  • 对于特定的SQL语句，数据库可以维护多个计划基准。

    优化程序可以从特定SQL语句的一组最佳计划中进行选择，而不是按存储的大纲要求限制为每个类别的单个计划。

迁移过程包括以下阶段：

  1. 用户调用一个函数，指定要迁移的轮廓。

  2. 数据库按如下方式处理轮廓：

    1. 数据库复制计划基准所需的大纲中的信息。

      数据库直接复制或根据大纲中的信息进行计算。例如，SQL语句的文本存在于两个模式中，因此数据库可以将文本从大纲复制到基线。

    2. 数据库重新提示提示以获取不在大纲中的信息。

    3. 数据库创建基线。

  3. 数据库在第一次选择SQL计划基准时执行SQL语句时获取缺少的信息。

    编译环境和执行统计信息仅在解析和编译计划基准时在执行期间可用。

29.2准备存储大纲迁移

目标是准备确定哪些存储的轮廓符合迁移条件。

准备存储的大纲迁移：

  1. 使用SYSDBA权限或DBMS_SPM程序包的EXECUTE权限将SQL * Plus连接到数据库。

  2. 查询数据库中存储的轮廓。

    以下示例查询尚未迁移到SQL计划基准的所有存储大纲：

    SELECT NAME，CATEGORY，SQL_TEXT FROM DBA_OUTLINES MIGRATED ='NOT-MIGRATED';

  3. 确定哪些存储的大纲符合迁移资格的以下先决条件：

    • 该语句不能是运行时INSERT AS SELECT语句。

    • 该语句不得引用远程对象。

    • 此声明不得为私有存储大纲。

  4. 决定是否迁移属于指定大纲类别的所有轮廓，指定的存储轮廓或轮廓。

    如果您不决定迁移所有大纲，请确定要迁移的大纲或类别。

  5. 确定迁移到SQL计划基准的存储轮廓是使用固定计划还是非固定计划 ：

    • 固定计划

      固定计划被冻结。如果使用存储在计划基准中的提示可以重现固定计划，则优化程序始终选择最低成本的固定计划基准而不是未修复的计划基准。实质上，固定计划基线充当具有有效提示的存储轮廓。

      当数据库可以根据存储在计划基准中的提示解析语句并创建具有与计划基准中的计划哈希值相同的计划哈希值的计划时，可以重现固定计划。如果其中一个提示变为无效，则数据库可能无法创建具有相同计划哈希值的计划。在这种情况下，该计划是不可再现的。

      如果在使用其提示进行解析时无法再现固定计划，则优化程序会选择不同的计划，该计划可以是以下任一种：

      • SQL计划基线的另一个计划

      • 当前基于成本的计划由优化程序创建

      在某些情况下，由于计划不同而需要进行SQL调优，因此会发生性能回归。

    • 非固定计划

      如果计划基准不包含固定计划，则SQL计划管理在为SQL语句选择计划时会平等地考虑计划。

  6. 在开始实际迁移之前，请确保Oracle数据库满足以下先决条件：

    • 数据库必须是Enterprise Edition。

    • 数据库必须处于打开状态且不处于挂起状态。

    • 数据库不得处于受限访问（DBA），只读或迁移模式。

    • Oracle调用接口（OCI）必须可用。

29.3迁移大纲以利用SQL计划管理功能

您可以将存储的大纲迁移到SQL计划基准。

这项任务的目标如下：

  • 允许SQL计划管理从SQL语句的计划基准中的所有计划中进行选择，而不是在迁移后应用相同的固定计划

  • 通过向基线添加新计划，允许SQL计划基线在数据库更改面前发展

假设

本教程假设以下内容：

  • 您迁移所有轮廓。

    要迁移特定轮廓，请使用DBMS_SPM.MIGRATE_STORED_OUTLINE函数。

  • 您希望基线的模块名称与迁移的轮廓的类别名称相同。

  • 您不希望修复SQL计划。

    默认情况下，生成的计划不是固定的，SQL计划管理在为SQL语句选择计划时会平等地考虑所有计划。这种情况允许计划发展的高级功能捕获SQL语句的新计划，验证其性能，并将这些新计划接受到计划基准中。

要将存储的轮廓迁移到SQL计划基准：

  1. 使用适当的权限将SQL * Plus连接到数据库。

  2. 调用PL / SQL函数MIGRATE_STORED_OUTLINE 。

    以下示例PL / SQL块将所有存储的轮廓迁移到固定基线：

    DECLARE my_report CLOB; BEGIN my_outlines：= DBMS_SPM.MIGRATE_STORED_OUTLINE（attribute_name =>'all'）;结束; /

29.4迁移轮廓以保留存储的轮廓行为

您可以将存储的大纲迁移到SQL计划基准，并通过创建固定计划基准来保留存储的大纲的原始行为。

对于同一SQL语句，固定计划优先于其他计划。如果计划得到修复，那么计划基线就无法发展。数据库不会将新计划添加到包含固定计划的计划基准。

假设

本教程假设以下内容：

  • 您只想迁移名为firstrow的类别中存储的轮廓。

  • 您希望基线的模块名称与迁移的轮廓的类别名称相同。

要将存储的轮廓迁移到计划基线：

  1. 使用适当的权限将SQL * Plus连接到数据库。

  2. 调用PL / SQL函数MIGRATE_STORED_OUTLINE 。

    以下示例PL / SQL块将firstrow类别中存储的轮廓迁移到固定基线：

    DECLARE my_report CLOB; BEGIN my_outlines：= DBMS_SPM.MIGRATE_STORED_OUTLINE（attribute_name =>'category'，attribute_value =>'firstrow'，fixed =>'YES'）;结束; /

    迁移后，SQL计划基准位于模块firstrow和类别DEFAULT 。

13.1关于数据库缓冲区高速缓存

对于许多类型的操作，Oracle数据库使用缓冲区高速缓存来存储从磁盘读取的数据块。Oracle数据库绕过特定操作的缓冲区高速缓存，例如排序和并行读取。

要有效地使用数据库缓冲区缓存，请调整应用程序的SQL语句，以避免不必要的资源消耗。要实现此目标，请验证经常执行的SQL语句和执行许多缓冲区获取的SQL语句是否经过良好调整。

使用并行查询时，请考虑将数据库配置为使用数据库缓冲区高速缓存，而不是直接读入程序全局区域（PGA）。当系统具有大量内存时，此配置可能是适当的。

13.2.3解释缓冲区高速缓存命中率

在决定是否增加或减少缓冲区高速缓存大小之前，应首先检查缓冲区高速缓存命中率。

低缓存命中率并不一定意味着增加缓冲区缓存的大小将有利于性能。此外，高缓存命中率可能错误地指示缓冲区缓存的大小适合工作负载。

要解释缓冲区缓存命中率，请考虑以下因素：

  • 通过一次执行处理或优化SQL语句，避免重复扫描频繁访问的数据。

    重复扫描相同的大型表或索引可以人为地夸大低缓存命中率。检查经常执行的具有大量缓冲区获取的SQL语句，以确保这些SQL语句的执行计划是最佳的。

  • 避免通过在客户端程序或中间层缓存频繁访问的数据来重新查询相同的数据。

  • 在运行OLTP应用程序的大型数据库中，许多行只能访问一次（或从不）。因此，在使用它之后，没有任何目的将块保持在存储器中。

  • 不要不断增加缓冲区缓存大小。

    如果数据库执行全表扫描或不使用缓冲区高速缓存的操作，则缓冲区高速缓存大小的连续增加不起作用。

  • 在发生大型全表扫描时，请考虑较差的命中率。

    在长全表扫描期间访问的数据库块位于最近最少使用（LRU）列表的尾端，而不是列表的头部。因此，在执行索引查找或小型表扫描时，块会比读取的块更快地老化。

    注意：

    短表扫描是在特定大小阈值下对表执行的扫描。小表的定义最多为缓冲区高速缓存的2％或20，以较大者为准。

13.2.4增加分配给数据库缓冲区高速缓存的内存

如果缓存命中率较低并且调整了应用程序以避免执行全表扫描，请考虑增加缓冲区缓存的大小。如果可能，请动态调整缓冲池的大小，而不是关闭实例以执行此更改。

要增加数据库缓冲区缓存的大小：

  1. 将DB_CACHE_ADVICE初始化参数的值设置为ON 。

  2. 允许缓冲区缓存统计信息稳定下来。

  3. 检查V$DB_CACHE_ADVICE视图中的顾问数据，以确定显着减少执行的物理I / O量所需的下一个增量，如“ 使用V $ DB_CACHE_ADVICE视图 ”中所述 。

  4. 如果可以在不导致系统分页的情况下分配缓冲区高速缓存所需的额外内存，则分配此内存。

  5. 要增加分配给缓冲区高速缓存的内存量，请增加DB_CACHE_SIZE初始化参数的值。

    DB_CACHE_SIZE参数指定数据库标准块大小的默认高速缓存的大小。要创建和使用块大小不是数据库标准块大小的表空间（例如可传输表空间），请为每个使用的块大小配置单独的高速缓存。使用DB_ n K_CACHE_SIZE参数配置所需的非标准块大小（其中n为DB_ n K_CACHE_SIZE或32而不是标准块大小）。

注意：

  • 无论缓存是默认标准块大小缓存， KEEP还是RECYCLE缓存，还是非标准块大小缓存，选择缓存大小的过程都是相同的。

  • 当缓存显着调整大小（大于20％）时，将丢弃旧缓存顾问值，并将缓存顾问设置为新大小。否则，通过现有值的插值将旧缓存顾问值调整为新大小。

13.2.5减少分配给数据库缓冲区高速缓存的内存

如果高速缓存命中率很高，则缓冲区高速缓存可能足够大以存储最频繁访问的数据。如果是这种情况并且另一个内存结构需要内存，请考虑减小缓冲区高速缓存的大小。

要减小数据库缓冲区缓存的大小：

  1. 检查V$DB_CACHE_ADVICE视图中的顾问数据，以确定减小缓冲区高速缓存的大小是否会显着增加物理I / O的数量，如“ 使用V $ DB_CACHE_ADVICE视图 ”中所述 。

  2. 要减少分配给缓冲区高速缓存的内存量，请减小DB_CACHE_SIZE初始化参数的值。

13.3配置多个缓冲池

对于大多数系统，单个默认缓冲池通常就足够了。但是，具有应用程序缓冲池详细知识的数据库管理员可以从配置多个缓冲池中受益。

对于具有非典型访问模式的段，请考虑将这些段中的块存储在两个单独的缓冲池中： KEEP池和RECYCLE池。如果段的访问模式经常被访问（有时被称为热门）或不经常访问（例如每天只能由批处理作业访问一次的大段），则该段的访问模式可能是非典型的。

运用多个缓冲池使您可以解决这些不规则问题。您可以使用KEEP池来维护缓冲区缓存中经常访问的段，以及RECYCLE池以防止对象占用缓冲区缓存中不必要的空间。当对象与缓冲区高速缓存关联时，该对象的所有块都放在该高速缓存中。Oracle数据库为未分配给特定缓冲池的对象维护DEFAULT缓冲池。默认缓冲池大小由DB_CACHE_SIZE初始化参数确定。每个缓冲池使用相同的LRU替换策略。例如，如果KEEP池的大小不足以存储分配给它的所有段，则最旧的块会从缓存中老化。

通过将对象分配给适当的缓冲池，您可以：

  • 减少或消除I / O.

  • 将对象隔离或限制为单独的缓存

13.3.1.1随机访问大段

当使用大型或无限索引范围扫描访问非常大的段（与缓冲区高速缓存的大小相比）时，LRU老化方法可能会出现问题。可以认为占非实质读物的大部分（超过10％）的任何单个片段都非常大。对大段的随机读取可能导致包含其他段的数据的缓冲区从缓存中老化。大段最终消耗了大部分缓冲区缓存，但它不会从缓存中受益。

非常频繁访问的段不受大段读取的影响，因为它们的缓冲区经常被加热以使它们不会老化缓冲区高速缓存。但是，该问题会影响温度段，这些段不会频繁访问，无法承受由大段读取引起的缓冲区老化。解决此问题有三种选择：

  • 如果访问的对象是索引，请确定索引是否是选择性的。如果没有，请调整SQL语句以使用更具选择性的索引。

  • 如果调整了SQL语句，请将大段移动到单独的RECYCLE缓存中，这样它就不会影响其他段。RECYCLE缓存应小于DEFAULT缓冲池，并且应该更快地重用缓冲区。

  • 或者，考虑将小的热段移动到不用于大段的单独KEEP高速缓存中。调整KEEP缓存的大小以最大限度地减少缓存中的未命中。通过将查询访问的段存储在KEEP缓存中，可以使特定查询的响应时间更加可预测，以确保它们不会老化。

在Oracle Real Application Cluster（Oracle RAC）环境中，请考虑为每个数据库实例创建多个缓冲池。没有必要为数据库的每个实例定义同一组缓冲池。在实例中，缓冲池可以是不同的大小或未定义的。根据该实例的应用程序要求调整每个实例。

13.3.5.2检查特定段的缓冲区高速缓存使用模式

确定池中具有多个缓冲区的另一种方法是计算在给定时间单个对象使用的缓冲区高速缓存的百分比。

要计算单个对象使用的缓冲区高速缓存的百分比：

  1. 通过查询，找到该段的Oracle数据库内部对象编号 DBA_OBJECTS视图：

    SELECT data_object_id，object_type FROM DBA_OBJECTS WHERE object_name = UPPER（' segment_name '）;

    因为两个对象可以具有相同的名称（如果它们是不同类型的对象），请使用OBJECT_TYPE列来标识感兴趣的对象。

  2. 在SEGMENT_NAME的缓冲区缓存中查找缓冲区数量：

    SELECT COUNT（*）缓冲区FROM V $ BH WHERE objd = data_object_id_value ;

    对于data_object_id_value ，请使用上一步中的DATA_OBJECT_ID值。

  3. 查找数据库实例中的缓冲区数量：

    SELECT name，block_size，SUM（buffers）FROM V $ BUFFER_POOL GROUP BY name，block_size HAVING SUM（buffers）> 0;

  4. 计算缓冲区与总缓冲区的比率，以获取SEGMENT_NAME当前使用的缓存百分比：

    segment_name = [buffers（Step2）/ total buffers（Step3）]使用的％缓存

注意：

此方法仅适用于单个段。对于分区对象，请为每个分区运行查询。

13.3.6配置KEEP池

KEEP缓冲池的目的是将对象保留在内存中，从而避免I / O操作。保留在内存中的每个对象都会导致权衡。将频繁访问的块保留在缓存中更有益。避免在缓存中保留不常使用的块，因为这样可以减少其他更活跃块的空间

如果应用程序中存在经常引用的某些段，则考虑将这些段中的块存储在KEEP缓冲池中。保留在KEEP池中的典型段是经常使用的小型参考表。要确定哪些表是候选表，请通过查询V$BH视图来检查候选表中的块数，如“ 检查缓冲区高速缓存使用模式 ”中所述 。

要配置KEEP池：

  1. 计算KEEP缓冲池的大致大小。

    KEEP缓冲池的大小取决于要保留在缓冲区高速缓存中的对象。要估计其大小，请添加分配给此池的所有对象使用的块。

    如果您收集了有关段的统计信息，请查询DBA_TABLES.BLOCKS和DBA_TABLES 。 EMPTY_BLOCKS确定使用的块数。

  2. 在不同时间拍摄系统性能的两个快照。

    使用V$DB_CACHE_ADVICE视图从KEEP池查询每个快照的数据，如“ 使用各个缓冲池的V $ DB_CACHE_ADVICE视图 ”中所述 。

  3. 从较旧的值中减去physical reads ， block gets和consistent gets的更新值，并使用结果计算命中率。

    缓冲池命中率为100％可能不是最佳的。通常，您可以减小KEEP缓冲池的大小，并仍然保持足够高的命中率。将从KEEP缓冲池中删除的块分配给其他缓冲池。

  4. 通过将DB_KEEP_CACHE_SIZE参数的值设置为所需大小，将内存分配给KEEP缓冲池。

    KEEP池的内存不是默认池的子集。

注意：

如果对象的大小增加，则它可能不再适合KEEP缓冲池。您将开始从缓存中丢失块。

13.3.7配置RECYCLE池

您可以为属于那些您不想保留在内存中的段的块配置RECYCLE缓冲池。RECYCLE池的目的是保留很少扫描或不经常引用的段。如果应用程序随机访问非常大的对象的块，则存储在缓冲池中的块不可能在其老化之前被重用。无论缓冲池的大小如何（考虑到可用物理内存量的约束），都是如此。因此，不需要缓存对象的块;缓存缓冲区可以分配给其他对象。

不要太快丢弃内存中的块。如果缓冲池太小，则在事务或SQL语句完成执行之前，块可能会从缓存中老化。例如，应用程序可能会从表中选择一个值，使用该值处理某些数据，然后更新记录。如果在SELECT语句之后从高速缓存中删除该块，则必须再次从磁盘读取该块以执行更新。应在用户事务期间保留该块。

要配置RECYCLE POOL：

  • 通过将DB_RECYCLE_CACHE_SIZE参数的值设置为所需大小，将内存分配给RECYCLE缓冲池。

    RECYCLE池的内存不是默认池的子集。

13.4配置重做日志缓冲区

对缓冲区高速缓存中的数据块进行更改的服务器进程会将重做数据生成到日志缓冲区中。如果满足以下任一条件，则日志写入器进程（LGWR）开始写入将条目从重做日志缓冲区复制到联机重做日志：

  • 重做日志缓冲区至少变为满三分之一

13.5配置数据库缓存模式

从Oracle Database 12 c第1版（12.1.0.2）开始，有两个数据库缓存模式：以前版本的Oracle数据库中使用的默认数据库缓存模式，以及此版本中新增的强制完全数据库缓存模式。在默认缓存模式下，当用户查询大型表时，Oracle数据库并不总是缓存基础数据。在强制完全数据库缓存模式下，Oracle数据库假定缓冲区缓存足够大，可以缓存整个数据库并尝试缓存查询访问的所有块。

本节包含以下主题：

  • 默认数据库缓存模式

  • 强制完全数据库缓存模式

  • 确定何时使用强制完全数据库缓存模式

  • 验证数据库缓存模式

注意：

从Oracle Database 12 c第1版（12.1.0.2）开始，可以使用强制完全数据库缓存模式。

13.5.1默认数据库缓存模式

默认情况下，Oracle数据库使用执行全表扫描时的默认数据库缓存模式。在默认缓存模式下，当用户查询大型表时，Oracle数据库并不总是缓存底层数据，因为这样做可能会从缓冲区缓存中删除更多有用的数据。

如果Oracle数据库实例确定有足够的空间来缓存缓冲区高速缓存中的完整数据库，并且这样做是有益的，那么实例会自动将完整数据库缓存在缓冲区高速缓存中。

如果Oracle数据库实例确定没有足够的空间来缓存缓冲区缓存中的完整数据库，则：

  • 只有当表大小小于缓冲区高速缓存大小的2％时，才会将较小的表加载到内存中。

  • 对于中型表，Oracle数据库分析最后一次表扫描与缓冲区高速缓存的老化时间戳之间的间隔。如果在上一次表扫描中重用的表的大小大于剩余的缓冲区高速缓存大小，则表将被缓存。

  • 除非您为KEEP缓冲池显式声明表，否则通常不会将大表加载到内存中。

注意：

在默认缓存模式下，Oracle数据库实例不会在缓冲区缓存中缓存NOCACHE LOB。

13.5.2强制完全数据库缓存模式

随着更多内存添加到数据库，缓冲区缓存大小可能会不断增长。在某些情况下，缓冲区高速缓存的大小可能会变得如此之大，以至于整个数据库都可以放入内存中。在执行全表扫描或访问LOB时，将整个数据库缓存在内存中的能力可以极大地提高数据库性能。

在强制完全数据库缓存模式，当数据库的大小小于数据库缓冲区高速缓存大小时，Oracle数据库将整个数据库缓存在内存中。所有数据文件（包括使用SecureFiles的NOCACHE LOB和LOBS）在访问时都会加载到缓冲区缓存中。

13.5.3确定何时使用强制完全数据库缓存模式

要提高表扫描和LOB数据访问的数据库性能，尤其是对受I / O吞吐量或响应时间限制的工作负载，请考虑使用force full 数据库缓存模式，只要数据库缓冲区缓存的大小大于数据库的大小。

在以下情况下，请考虑使用强制完全数据库缓存模式：

  • 逻辑数据库大小（或实际使用的空间）小于Oracle RAC环境中每个数据库实例的单个缓冲区高速缓存。这也适用于非Oracle RAC数据库。

  • 对于Oracle RAC环境中良好分区的工作负载（通过实例访问），逻辑数据库大小小于所有数据库实例的组合缓冲区高速缓存大小的80％。

  • 数据库使用SGA_TARGET或MEMORY_TARGET。

  • 需要缓存NOCACHE LOB。除非使用强制数据库缓存，否则永远不会缓存NOCACHE LOB。

对于前三种情况，您应定期监控系统性能，以验证性能数据是否符合您的预期。

当一个Oracle RAC数据库实例使用强制完全数据库缓存模式时，Oracle RAC环境中的所有其他数据库实例也将使用强制完全数据库缓存模式。

在多租户环境中，强制完整数据库缓存模式适用于整个容器数据库（CDB），包括其所有可插拔数据库（PDB）。

磁盘组兼容性概述

通过推进磁盘组兼容性设置，您可以使用以后版本中提供的新Oracle ASM功能。例如，磁盘组兼容性属性设置为19.0的磁盘组可以利用Oracle ASM 19c功能。有关为兼容性属性设置组合启用的功能的信息，请参阅表4-4 。

确定兼容性的磁盘组属性是COMPATIBLE.ASM ， COMPATIBLE.RDBMS 。和COMPATIBLE.ADVM 。 COMPATIBLE.ASM和COMPATIBLE.RDBMS属性设置确定系统可以分别用于Oracle ASM和数据库实例类型的最低Oracle数据库软件版本号。例如，如果Oracle ASM兼容性设置为19.0 ，并且RDBMS兼容性设置为12.2.0.1 ，则Oracle ASM软件版本必须至少为19.0 ，Oracle数据库客户端软件版本必须至少为12.2.0.1 。COMPATIBLE.ADVM属性确定Oracle ASM动态卷管理器功能是否可以在磁盘组中创建卷。

创建磁盘组时，可以在CREATE DISKGROUP SQL语句中指定磁盘组兼容性属性设置。ALTER DISKGROUP SQL语句可以更新现有磁盘组的兼容属性设置。有关兼容属性设置的有效组合的示例，请参阅表4-3 。

您还可以使用ASMCMD setattr命令设置和更新磁盘组属性。有关ASMCMD setattr命令的信息，请参阅setattr 。

注意：

  • 磁盘组兼容性设置确定您的环境是否可以使用最新的Oracle ASM功能。

  • 磁盘组兼容性设置只能高级;您无法恢复到较低的兼容性设置。有关更多信息，请参阅恢复磁盘组兼容性 。

  • 必须先提前COMPATIBLE.ASM属性，然后才能推进其他磁盘组兼容性属性，并且其值必须大于或等于其他磁盘组兼容性属性的值。

将CREATE DISKGROUP与兼容性属性一起使用

创建磁盘组时，可以使用CREATE DISKGROUP语句指定磁盘组的兼容性设置。

以下示例创建一个正常冗余磁盘组data1 ，其Oracle ASM兼容性设置为19.0 ，并且RDBMS兼容性设置为缺省值（ COMPATIBLE.RDBMS缺省值小于或等于19.0 ）：

CREATE DISKGROUP data1 DISK'/ dev / sd *'ATTRIBUTE'compliances.asm'= '19 .0';

以下示例创建一个正常冗余磁盘组data2 ，其ASM，RDBMS和ADVM兼容性设置为19.0 ：

CREATE DISKGROUP data2 DISK'/ dev / sd *'ATTRIBUTE'compxliance.asm'= '19 .0'，'compatible.rdbms'= '19 .0'，'compatible.advm'= '19 .0';

使用带有兼容性属性的ALTER DISKGROUP

创建磁盘组后，可以使用ALTER DISK GROUP SQL语句更改兼容性属性。ALTER DISK GROUP SQL语句确保Oracle在提交更改之前可以提高指定磁盘组的兼容性。

运行ALTER DISKGROUP时，所有受影响的数据库和文件系统都应处于联机状态，以确保提高兼容性不会减少数据库和文件系统访问。提高磁盘组兼容性时，必须提高COMPATIBLE 。 COMPATIBLE之前的ASM属性。 RDBMS或COMPATIBLE.ADVM属性可确保兼容属性设置的有效组合，如表4-3所示。您只能在单个ALTER DISKGROUP语句中前进一个兼容性属性。

以下示例将磁盘组data3的Oracle ASM兼容性提升到19.0 。Oracle ASM实例必须位于Oracle ASM 19c或更高版本才能访问data3磁盘组。 

ALTER DISKGROUP data3 SET ATTRIBUTE'compliances.asm'= '19 .0';

以下示例将磁盘组data3的COMPATIBLE.RDBMS和COMPATIBLE.ADVM兼容性属性提升为10.0 。此示例假定COMPATIBLE.ASM的值设置为19.0 。 

ALTER DISKGROUP data3 SET ATTRIBUTE'compxliance.rdbms'= '19 .0'，ALTER DISKGROUP data3 SET ATTRIBUTE'compatired.advm'= '19 .0';

还原磁盘组兼容性

您无法将磁盘组兼容性还原为以前的值。相反，您必须创建一个新磁盘组。

推进磁盘组兼容性属性的值是不可逆转的操作。如果您提前了磁盘组兼容性设置，则无法将值更改回以前的设置。要还原为先前的值，必须使用旧兼容性属性设置创建新磁盘组，然后将磁盘组中的数据库文件还原到新磁盘组。

当您恢复到具有旧兼容性属性设置的新磁盘组时，可能无法使用最新的Oracle ASM功能。例如，如果将磁盘组兼容性恢复为11.2之前的值，则Oracle ACFS功能不可用。

例如，您可以执行以下过程将磁盘组还原为以前的兼容性设置：

  1. 如果Oracle ASM SPFILE位于磁盘组中，请将此SPFILE移出磁盘组：

    1. 与SQL * Plus连接到Oracle ASM实例。

    2. 在文件系统中创建PFILE。

      例如：

      SQL> CREATE PFILE'$ ORACLE_HOME / dbs / asmspfile.ora'FROM SPFILE;

  2. 如果OCR和投票文件位于磁盘组中，请将它们移出此磁盘组。

  3. 备份所有必须保存的文件。

    1. 备份数据库文件。

    2. 如果Oracle ACFS文件系统安装在磁盘组上的Oracle ADVM卷上，则必须备份文件系统中的操作系统文件或将其复制到文件系统装入点之外的位置。

  4. 使用SQL CREATE DISKGROUP创建新磁盘组，指定磁盘组属性设置的先前值。

  5. 使用ALTER DATABASE MOVE DATAFILE SQL语句或Recovery Manager（RMAN）将数据库文件移动或还原到新创建的磁盘组。

  6. 使用SQL DROP DISKGROUP INCLUDING CONTENTS删除具有高级磁盘组兼容性设置的磁盘组，以删除磁盘组及其内容。此SQL语句还会删除任何Oracle ACFS文件系统及其内容。
