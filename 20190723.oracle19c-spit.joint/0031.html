2.2 CDB_ *视图

对于每个DBA_ *视图，都定义了CDB_ *视图。 在多租户容器数据库（CDB）的根目录中，CDB_ *视图可用于获取有关根和可插拔数据库（PDB）中包含的表，表空间，用户，权限，参数等的信息。

只有具有SYSDBA系统特权或SELECT ANY DICTIONARY特权或SELECT_CATALOG_ROLE角色的用户或具有授予它们的直接特权的用户才能查询CDB_ *视图。

CDB_ *视图是容器数据对象。当连接到根的用户查询CDB_ *视图时，查询结果将取决于视图用户的CONTAINER_DATA属性。SQL ALTER USER语句的CONTAINER_DATA子句用于设置和修改用户的CONTAINER_DATA属性。

无论谁拥有底层DBA_ *视图，CDB_ *视图都归SYS所有。

默认情况下，连接到根的用户只能看到与根有关的数据。

CDB_ *视图包含以下隐藏列：

  • CON$NAME ：此列包含给定CDB_ *行表示其数据的容器的名称

  • CDB$NAME ：此列显示CDB的名称，该CDB的数据是给定的CDB_ *行所代表的

在PDB中，CDB_ *视图仅显示通过相应DBA_ *视图可见的对象。

除了在给定DBA_ *视图中找到的所有列之外，相应的CDB_ *视图还包含CON_ID列，该列标识其给定CDB_ *行表示的数据的容器。在非CDB中， CON_ID列的值将为0 。

从根容器查询时，CDB视图可以从CDB中的不同容器返回数据。这些对象将隐式地将数据转换为根容器（AL32UTF8）的字符集，然后将结果返回给用户。转换为AL32UTF8时，某些字符集可能具有字符扩展（表示字符所需的字节数更多），因此如果视图列宽度无法容纳来自给定PDB的数据，则可能会出现数据截断。

1.2物体的优点

通常，对象类型模型类似于C ++和Java中的类机制。

与类一样，对象的可重用性使得可以更快，更高效地开发数据库应用程序。通过本机支持数据库中的对象类型，Oracle数据库使应用程序开发人员能够直接访问其应用程序使用的数据结构。

与纯粹的关系方法相比，对象提供了其他优势，例如：

  • 对象可以与数据一起封装操作

  • 对象是高效的

  • 对象可以代表部分整体关系

对象可以与数据一起封装操作

数据库表仅包含数据。对象可以包括执行可能对该数据执行的操作的能力。

因此，诸如采购订单之类的对象可能包括计算所购买的所有商品的成本的方法。或者客户对象可能有方法返回客户的购买历史和付款方式。应用程序可以简单地调用方法来检索信息。

对象是高效的

使用对象类型可以提高效率：

  • 对象类型及其方法与数据一起存储在数据库中，因此可供任何应用程序使用。开发人员不需要在每个应用程序中重新创建类似的结构和方法。这也确保了开发人员使用一致的标准。

  • 您可以将一组相关对象作为单个单元进行提取和操作。从服务器获取对象的单个请求可以检索与其连接的其他对象。引用SQL对象类型的列时，将检索整个对象。

对象可以代表部分整体关系

对象类型允许您表示部分 - 整体关系。

例如，在库存物品的关系表中，活塞和发动机可以具有相同的状态。使用对象可以减少将活塞表示为引擎的一部分的需要，其具有具有主键 - 外键关系的多个表的复杂模式。对象可以将其他对象作为属性，属性对象也可以具有自己的对象属性。可以通过互锁对象类型以这种方式构建整个零件清单层次结构。

从Oracle Database 12 c第2版（12.2）及更高版本开始，您必须手动升级依赖于Oracle维护类型的用户表。

如果数据库具有依赖于Oracle维护类型的用户表（例如， AQ队列表），则在升级后运行utluptabdata.sql命令，以对受Oracle维护更改影响的任何用户表执行ALTER TABLE UPGRADE类型。此行为更改使用户表在升级期间保持READ ONLY状态。禁止用户使用SYSDBA权限（ AS SYSDBA ）登录应用程序，并更改依赖于Oracle维护类型的应用程序表。

要确定数据库升级完成后需要升级的表，请连接到数据库AS SYSDBA ，然后运行以下查询：

COLUMN所有者FORMAT A30 COLUMN table_name FORMAT A30 SELECT DISTINCT所有者，table_name FROM dba_tab_cols WHERE data_upgraded ='NO'ORDER BY 1,2;

此查询列出未列为UPGRADED所有表。但是， utluptabdata.sql脚本仅升级依赖于Oracle维护类型的表。如果查询列出了任何表，则运行utluptabdata.sql脚本以在依赖用户表上执行ALTER TABLE UPGRADE命令，以便将这些Oracle维护类型升级到该类型的最新版本。

您必须使用具有ALTER权限的用户帐户运行utluptabdata.sql脚本，该用户帐户依赖于Oracle维护类型的所有表，或者授予SYSDBA系统权限的用户，并且记录在AS SYSDBA 。

当参数SERVEROUTPUT设置为ON ， utluptabdata.sql脚本将显示所有已升级表的名称，并列出表升级期间遇到的任何错误。要将服务器输出设置为ON ，请运行以下命令：

SET SERVEROUTPUT ON @ utluptabdata.sql

关于快速恢复区和快速恢复区磁盘组

快速恢复区域是与恢复相关的所有Oracle数据库文件的统一存储位置。为最近的数据启用快速备份可以减少向系统管理员请求检索备份磁带以进行恢复操作的请求。

数据库管理员可以将DB_RECOVERY_FILE_DEST参数定义为快速恢复区域的路径，以启用磁盘备份和快速恢复数据。在init.ora文件中启用快速恢复时，Oracle数据库会将所有RMAN备份，存档日志，控制文件自动备份和数据库副本写入快速恢复区域。RMAN通过删除不再需要的过时备份和归档文件来自动管理快速恢复区中的文件。

Oracle建议您创建快速恢复区磁盘组。Oracle Clusterware文件和Oracle数据库文件可以放在同一磁盘组中，您也可以将快速恢复文件放在同一磁盘组中。但是，Oracle建议您创建单独的快速恢复磁盘组以减少存储设备争用。

通过设置DB_RECOVERY_FILE_DEST参数来启用快速恢复区域。快速恢复区域的大小使用DB_RECOVERY_FILE_DEST_SIZE设置。作为一般规则，快速恢复区域越大，它变得越有用。为便于使用，Oracle建议您在存储设备上创建一个可包含至少三天恢复信息的快速恢复区磁盘组。理想情况下，快速恢复区域足以容纳所有数据文件和控制文件的副本，联机重做日志以及使用保留策略下保留的数据文件备份恢复数据库所需的归档重做日志文件。

多个数据库可以使用相同的快速恢复区域。例如，假设您已在具有150 GB存储空间的磁盘上创建了快速恢复区磁盘组，由3个不同的数据库共享。您可以根据每个数据库的重要性为每个数据库设置快速恢复的大小。例如，如果database1是您最不重要的数据库，则database2更重要，而database3最重要，那么您可以为每个数据库设置不同的DB_RECOVERY_FILE_DEST_SIZE设置以满足每个数据库的保留目标：30 GB用于database1,50 GB对于database2，对于database3为70 GB。

验证Linux上的磁盘I / O调度程序

为了获得Oracle ASM的最佳性能，Oracle建议您使用Deadline I / O Scheduler。

磁盘I / O调度程序对磁盘I / O的请求进行重新排序，延迟或合并，以实现更高的吞吐量和更低的延迟。Linux有多个磁盘I / O调度程序可用，包括Deadline，Noop，Anticipatory和Completely Fair Queuing（CFQ）。
在每个群集节点上，输入以下命令以验证是否已配置Deadline磁盘I / O调度程序以供使用：
#cat / sys / block / $ {ASM_DISK} / queue / scheduler noop [deadline] cfq

在此示例中，默认磁盘I / O调度程序为Deadline， ASM_DISK是Oracle自动存储管理（Oracle ASM）磁盘设备。

在某些虚拟环境（VM）和特殊设备（如快速存储设备）上，上述命令的输出可能为none 。操作系统或VM绕过内核I / O调度并将所有I / O请求直接提交给设备。请勿在此类环境中更改I / O计划程序设置。

如果默认磁盘I / O调度程序不是Deadline，则使用rules文件设置它：

  1. 使用文本编辑器为Oracle ASM设备创建UDEV规则文件：

    #vi /etc/udev/rules.d/60-oracle-schedulers.rules

  2. 将以下行添加到规则文件并保存：

    ACTION ==“add | change”，KERNEL ==“sd [az]”，ATTR {queue / rotating} ==“0”，ATTR {queue / scheduler} =“deadline”

  3. 在群集系统上，将规则文件复制到群集上的所有其他节点。例如：

    $ scp 60-oracle-schedulers.rules root @ node2：/etc/udev/rules.d/

  4. 加载规则文件并重新启动UDEV服务。例如：

    1. Oracle Linux和Red Hat Enterprise Linux

      #udevadm control --reload-rules

    2. SUSE Linux Enterprise Server

      #etet/init.d boot.udev restart

  5. 验证磁盘I / O调度程序是否设置为截止日期。

关于快速恢复区和快速恢复区磁盘组

快速恢复区域是与恢复相关的所有Oracle数据库文件的统一存储位置。为最近的数据启用快速备份可以减少向系统管理员请求检索备份磁带以进行恢复操作的请求。

数据库管理员可以将DB_RECOVERY_FILE_DEST参数定义为快速恢复区域的路径，以启用磁盘备份和快速恢复数据。在init.ora文件中启用快速恢复时，Oracle数据库会将所有RMAN备份，存档日志，控制文件自动备份和数据库副本写入快速恢复区域。RMAN通过删除不再需要的过时备份和归档文件来自动管理快速恢复区中的文件。

Oracle建议您创建快速恢复区磁盘组。Oracle Clusterware文件和Oracle数据库文件可以放在同一磁盘组中，您也可以将快速恢复文件放在同一磁盘组中。但是，Oracle建议您创建单独的快速恢复磁盘组以减少存储设备争用。

通过设置DB_RECOVERY_FILE_DEST参数来启用快速恢复区域。快速恢复区域的大小使用DB_RECOVERY_FILE_DEST_SIZE设置。作为一般规则，快速恢复区域越大，它变得越有用。为便于使用，Oracle建议您在存储设备上创建一个可包含至少三天恢复信息的快速恢复区磁盘组。理想情况下，快速恢复区域足以容纳所有数据文件和控制文件的副本，联机重做日志以及使用保留策略下保留的数据文件备份恢复数据库所需的归档重做日志文件。

多个数据库可以使用相同的快速恢复区域。例如，假设您已在具有150 GB存储空间的磁盘上创建了快速恢复区磁盘组，由3个不同的数据库共享。您可以根据每个数据库的重要性为每个数据库设置快速恢复的大小。例如，如果database1是您最不重要的数据库，则database2更重要，而database3最重要，那么您可以为每个数据库设置不同的DB_RECOVERY_FILE_DEST_SIZE设置以满足每个数据库的保留目标：30 GB用于database1,50 GB对于database2，对于database3为70 GB。

Oracle Fleet Patching and Provisioning（Oracle FPP）是一种用于配置和修补Oracle主目录的软件生命周期管理方法。

通过Oracle Fleet Patching和Provisioning，您可以安装集群，配置，修补和升级Oracle Grid Infrastructure，Oracle数据库和Oracle Restart家园。支持的版本是11.2,12.1,12.2,18c和19c。您还可以使用Oracle FPP配置应用程序和中间件。单个群集（称为Oracle Fleet Patching and Provisioning Server）存储和管理标准化图像（称为黄金映像），可以将其配置到任意数量的节点。您可以安装Oracle Grid Infrastructure集群配置，例如Oracle Standalone Clusters，Oracle Member Clusters和Oracle Member Cluster for Applications。部署后，您可以展开和收缩群集和Oracle RAC数据库。

您可以在云计算环境中的一组远程节点上从单个群集中配置Oracle Grid Infrastructure，您可以将Oracle家庭模板存储为Oracle软件的图像（称为黄金映像），例如数据库，中间件和应用程序。

Oracle Fleet补丁和配置

使用Oracle FPP部署Oracle软件具有以下优势：

  • 通过黄金映像和已部署软件的管理系列确保标准化并实现高度自动化。

  • 支持变更管理。通过标准化的Oracle家庭，管理员可以更好地控制托管的Oracle软件，并可以通过单一位置轻松管理软件的大规模部署和维护，以进行变更管理。

  • 最大限度地减少修补和升级期间的停机时间，简化回滚，并使大型系统的配置更容易，更高效。

  • 减少修补软件映像的累积时间，因为单个Oracle主目录可用于许多数据库实例。

6.1为什么使用对象视图

正如视图是虚拟表一样，对象视图是虚拟对象表。视图中的每一行都是一个对象：您可以调用其方法，使用点表示法访问其属性，并创建指向它的REF 。

您可以运行面向对象的应用程序，而无需将现有表转换为不同的物理结构。为此，您可以使用对象视图来原型化或转换到面向对象的应用程序，因为视图中的数据可以从关系表中获取并访问，就像将表定义为对象表一样。

可以像关系视图一样使用对象视图来仅显示您希望用户看到的数据。例如，您可以创建一个对象视图，该视图显示employee表中的选定数据，但省略有关工资的敏感数据。

使用对象视图可以带来更好的性能。构成对象视图行的关系数据作为一个单元遍历网络，可能节省许多往返。

您可以将关系数据提取到客户端对象缓存中，并将其映射到C结构或C ++或Java类中， 所以3GL应用程序可以像本机类一样操纵它。您还可以使用面向对象的功能，例如复杂对象检索和关系数据。

  • 您可以通过合成关系数据中的对象以新方式查询数据。您可以使用对象解除引用而不是使用多个表编写复杂连接来查看来自多个表的数据。

  • 您可以从对象视图中固定对象数据，并使用客户端对象缓存中的数据。通过专门的对象检索机制在对象缓存中检索这些合成对象时，可以减少网络流量。

  • 在视图中创建对象模型时，您可以获得极大的灵活性，从而可以继续开发模型。如果需要更改对象类型，则只需使用新定义替换无效视图即可。

  • 通过在视图中使用对象，您不会对底层存储机制的特性施加任何限制。出于同样的原因，您不受当前技术限制。例如，您可以合并来自关系表的对象，这些对象是并行化和分区的。

  • 您可以从相同的基础数据创建不同的复杂数据模型。

目的

NULLIF比较expr1和expr2 。如果它们相等，则该函数返回null。如果它们不相等，则函数返回expr1 。您不能为expr1指定文字NULL 。

如果两个参数都是数值数据类型，则Oracle数据库确定具有较高数字优先级的参数，隐式将另一个参数转换为该数据类型，并返回该数据类型。如果参数不是数字，则它们必须是相同的数据类型，否则Oracle将返回错误。

NULLIF函数在逻辑上等效于以下CASE表达式：

CASE WHEN expr1 = expr2 THEN NULL ELSE expr1 END

在Java集成的Oracle数据库中，Java应用程序存在于数据库会话的上下文中。Oracle JVM会话完全类似于传统的Oracle会话。每个Oracle JVM会话都维护客户端在会话中的调用之间访问的Java应用程序的状态。

图2-1展示了每个Java客户端如何启动数据库会话，作为在数据库中运行Java应用程序的环境。每个Java数据库会话都有一个单独的垃圾收集器，会话内存和调用内存。

在会话的上下文中，客户端执行以下操作：

  1. 连接到数据库并打开会话。

  2. 在数据库中运行Java。这被称为呼叫。

  3. 继续在会话中工作，根据需要执行尽可能多的呼叫。

  4. 结束会议。

在会话中，客户端具有自己的Java环境。客户端似乎为每个会话启动了一个单独的，单独的JVM，尽管实现效率比这似乎意味着更高效。在会话中，Oracle JVM管理应用程序的可伸缩性。来自单个客户端的每个呼叫都在其自己的会话中进行管理，并且来自每个客户端的呼叫将单独处理。Oracle JVM最大限度地在客户端之间共享只读数据，并强调每个会话增量的最小量足迹，以最大化多个客户的性能。

关于Oracle Cluster Domain和Oracle Domain Services Cluster

Oracle Cluster Domain是Oracle Clusterware 12c第2版中引入的新集群部署体系结构的一种选择。

Oracle Cluster Domain使您能够为私有数据库云标准化，集中和优化Oracle Real Application Clusters（Oracle RAC）部署。多个群集配置在Oracle Cluster Domain下分组以进行管理，并使用该Oracle Cluster Domain中可用的共享服务。Oracle Cluster Domain中的群集配置包括Oracle Domain Services Cluster和Oracle Member Clusters。

Oracle域服务集群为Oracle集群域中的其他集群提供集中式服务。这些服务包括：

  • 集中式Grid Infrastructure Management Repository（为Oracle Cluster Domain中的每个集群提供MGMTDB）

  • 跟踪文件分析器（TFA）服务，用于Oracle Clusterware和Oracle数据库的目标诊断数据收集

  • 整合Oracle ASM存储管理服务，包括使用Oracle ACFS。

  • 可选的Oracle Fleet补丁和配置服务，用于安装集群，以及配置，修补和升级Oracle Grid Infrastructure和Oracle数据库主目录。配置Oracle域服务群集时，还可以选择配置Oracle Fleet修补和配置服务器。

Oracle域服务集群为Oracle成员集群提供这些集中式服务。Oracle成员集群使用这些服务进行集中管理并减少其本地资源使用。

COLUMN_VALUE伪列

当您引用不带COLUMNS子句的XMLTable构造时，或者当您使用TABLE集合表达式引用标量嵌套表类型时，数据库将返回具有单个列的虚拟表。此伪COLUMN_VALUE名称为COLUMN_VALUE 。

在XMLTable的上下文中，返回的值是数据类型XMLType 。例如，以下两个语句是等效的，并且两者的输出都显示COLUMN_VALUE作为要返回的列的名称：

SELECT * FROM XMLTABLE（'<a> 123 </a>'）; COLUMN_VALUE --------------------------------------- <a> 123 </a> SELECT COLUMN_VALUE FROM（XMLTable（'<a> 123 </a>））; COLUMN_VALUE ---------------------------------------- <a> 123 </a>

在TABLE集合表达式的上下文中，返回的值是集合元素的数据类型。以下语句创建了创建表：多级集合示例中所示的两级嵌套表，以显示COLUMN_VALUE在此上下文中的COLUMN_VALUE ：

创建类型电话作为数字表; / CREATE TYPE phone_list AS TABLE OF phone; /

下一个语句使用COLUMN_VALUE从phone类型中进行选择：

选择t。COLUMN_VALUE FROM TABLE（电话（1,2,3））t; COLUMN_VALUE ------------ 1 2 3

在嵌套类型中，您可以在选择列表和TABLE集合表达式中使用COLUMN_VALUE伪列：

选择t。COLUMN_VALUE FROM TABLE（phone_list（phone（1,2,3）））p，TABLE（p。COLUMN_VALUE）t; COLUMN_VALUE ------------ 1 2 3

关键字COLUMN_VALUE也是Oracle数据库为没有列或属性名称的内部嵌套表的标量值生成的名称，如下面的示例所示。在此上下文中， COLUMN_VALUE不是伪列，而是实际列名。

关于安装修复脚本

Oracle Universal Installer检测何时不满足安装的最低要求，并创建称为修复脚本的shell脚本，以完成不完整的系统配置步骤。

如果Oracle Universal Installer检测到不完整的任务，则会生成runfixup.sh脚本（ runfixup.sh ）。您可以运行修复脚本，然后单击“ 修复并再次检查” 。fixup脚本修改了内存中的持久性参数设置和参数，因此您不必重新启动系统。

Fixup脚本执行以下任务：

  • 如有必要，将内核参数设置为成功安装所需的值，包括：

    • 共享内存参数。

    • 打开文件描述符和UDP发送/接收参数。

  • 在Oracle Inventory（中央库存）目录上创建和设置权限。

  • 如有必要，为Oracle Inventory目录和操作系统特权组创建或重新配置安装所有者的主要和辅助组成员身份。

  • 如有必要，将shell限制设置为所需的值。

注意：

使用修复脚本无法确保满足安装Oracle数据库的所有先决条件。您仍必须验证是否满足所有预安装要求以确保安装成功。

Oracle Universal Installer与Cluster Verification Utility（CVU）完全集成，可自动执行Oracle Grid Infrastructure或Oracle Real Application Clusters（Oracle RAC）安装的许多先决条件检查。您还可以通过运行cluvfy命令手动执行各种CVU验证。

在Oracle Grid Infrastructure安装期间，您可以选择安装和配置Oracle自动存储管理筛选器驱动程序（Oracle ASMFD）。Oracle ASMFD有助于防止Oracle ASM磁盘和磁盘组中的文件损坏。

Oracle ASM筛选器驱动程序（Oracle ASMFD）拒绝Oracle软件未发出的写入I / O请求。此写入筛选器有助于防止具有管理权限的用户无意中覆盖Oracle ASM磁盘，从而防止Oracle ASM磁盘和磁盘组中的文件损坏。对于磁盘分区，受保护的区域是Oracle ASMFD管理的磁盘上的区域，假设分区表未被用户保持不变。

配置Oracle ASM（包括Oracle ASMFD）时，请勿修改或删除Oracle ASM磁盘的内容，也不要修改任何文件，包括配置文件。

关于多个Oracle Homes支持

Oracle数据库支持多个Oracle主目录。您可以在同一系统上，在不同的Oracle主目录中多次安装此版本或更早版本的软件。

仔细选择安装点名称可以使Oracle软件更易于管理。按照Optimal Flexible Architecture（OFA）规则配置多个Oracle主目录具有以下优势：

  • 您可以在同一系统上，在不同的Oracle主目录中多次安装此发行版或该软件的早期版本。但是，您无法将一个Oracle数据库版本的产品安装到另一个版本的Oracle主目录中。例如，您无法将Oracle Database 19c软件安装到现有的Oracle 18c Oracle主目录中。

  • 不同用户拥有的不同版本的多个数据库可以同时共存。

  • 您必须在与早期版本的Oracle数据库分开的新Oracle主目录中安装新的Oracle数据库发行版。

    您无法在一个Oracle主目录中安装多个版本。Oracle建议您根据Optimal Flexible Architecture（OFA）指南为每个版本创建单独的Oracle Database Oracle主目录。

  • 在生产中，Oracle数据库服务器软件版本必须与前几个数字（主要版本，维护版本和修补程序版本号）的Oracle数据库字典版本相同。

  • 后来的Oracle Database版本可以访问早期的Oracle Database版本。但是，此访问权限仅用于升级。例如，如果18c数据库以升级模式启动，Oracle Database 19c可以访问Oracle数据库18c。

  • 如果两个产品处于同一版本级别，则Oracle Database Client可以安装在同一Oracle数据库主目录中。例如，您可以将Oracle Database Client 12.2.0.1安装到现有的Oracle Database 12.2.0.1主目录中，但不能将Oracle Database Client 12.2.0.1安装到现有的Oracle Database 12.1.0.2主目录中。如果在安装客户端之前应用修补程序集，则必须再次应用修补程序集。

  • 目录和文件的结构化组织以及数据库文件的一致命名简化了数据库管理。

  • 当数据库管理员添加，移动或删除Oracle主目录时，登录主目录不会有风险。

  • 您可以在与生产数据库所在的Oracle主目录不同的目录中测试Oracle主目录中的软件升级。

每次使用replacement_string search_string REPLACE返回char 。如果省略replacement_string或null，则删除所有出现的search_string 。如果search_string为null，则返回char 。

REPLACE提供与TRANSLATE函数提供的功能相关的功能。TRANSLATE提供单字符，一对一替换。REPLACE允许您将一个字符串替换为另一个字符串以及删除字符串。

根据Oracle维护类型升级表

从Oracle Database 12 c第2版（12.2）及更高版本开始，您必须手动升级依赖于Oracle维护类型的用户表。

如果数据库具有依赖于Oracle维护类型的用户表（例如， AQ队列表），则在升级后运行utluptabdata.sql命令，以对受Oracle维护更改影响的任何用户表执行ALTER TABLE UPGRADE类型。此行为更改使用户表在升级期间保持READ ONLY状态。禁止用户使用SYSDBA权限（ AS SYSDBA ）登录应用程序，并更改依赖于Oracle维护类型的应用程序表。

要确定数据库升级完成后需要升级的表，请连接到数据库AS SYSDBA ，然后运行以下查询：

COLUMN所有者FORMAT A30 COLUMN table_name FORMAT A30 SELECT DISTINCT所有者，table_name FROM dba_tab_cols WHERE data_upgraded ='NO'ORDER BY 1,2;

此查询列出未列为UPGRADED所有表。但是， utluptabdata.sql脚本仅升级依赖于Oracle维护类型的表。如果查询列出了任何表，则运行utluptabdata.sql脚本以在依赖用户表上执行ALTER TABLE UPGRADE命令，以便将这些Oracle维护类型升级到该类型的最新版本。

您必须使用具有ALTER权限的用户帐户运行utluptabdata.sql脚本，该用户帐户依赖于Oracle维护类型的所有表，或者授予SYSDBA系统权限的用户，并且记录在AS SYSDBA 。

当参数SERVEROUTPUT设置为ON ， utluptabdata.sql脚本将显示所有已升级表的名称，并列出表升级期间遇到的任何错误。要将服务器输出设置为ON ，请运行以下命令：

SET SERVEROUTPUT ON @ utluptabdata.sql
