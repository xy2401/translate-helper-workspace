C.1关于Java的内容存储库API（JCR）

JCR 1.0为应用程序定义了一个标准Java API，以便与内容存储库进行交互。内容存储库是节点树，每个节点可以具有一个或多个子节点。每个节点只有一个父节点，但根节点除外，它没有父节点。

除子节点外，节点还可以具有一个或多个属性。属性是一个简单的名称/值对。例如，表示内容存储库中特定文件的节点具有名为jcr:created的属性，其值为文件的创建日期。

每个属性都有一个属性类型。例如， jcr:created属性具有DATE属性类型，要求其值为有效日期/时间。

类似地，每个节点都有一个节点类型。例如，表示文件的节点具有节点类型nt:file 。节点类型控制节点可能具有或必须具有的子节点和属性。例如，类型为nt:file所有节点都必须具有jcr:created属性。

由于节点和属性已命名，因此可以通过路径进行寻址。JCR支持绝对路径和相对路径。例如，绝对路径

/我的文件/ pictures / puppy.jpg / jcr：已创建

解析为文件puppy.jpg的jcr:created属性。此属性也可以通过以下相对路径相对于“ My Documents文件夹进行寻址：

图片/小狗.jpg / JCR：创建

节点和属性名称可以是名称空间限定的。与XML一样，JCR使用冒号分隔的名称空间前缀来表示名称空间限定名称，例如jcr:created 。与XML不同，JCR在存储库范围的命名空间注册表中记录命名空间前缀到URI的映射，例如，将jcr前缀映射到URI http://www.jcp.org/jcr/1.0 。

5.1 JSON对象中的唯一对应重复字段

默认情况下，字段名称对于给定的JSON对象不必是唯一的。但是，只有在没有任何对象具有重复的字段名称时，才能指定特定的JSON数据被认为是格式良好的。

JSON标准未指定字段名称对于给定的JSON对象是否必须是唯一的。这意味着，先验的，格式良好的JSON对象可以具有多个具有相同字段名称的成员。这是在Oracle数据库中处理JSON数据的默认行为，因为检查重复的名称需要额外的时间。

只有当它包含的所有对象都具有唯一的字段名称时，您才可以指定特定的JSON数据被认为是格式正确的，即，没有对象具有重复的字段名称。通过使用关键字WITH UNIQUE KEYS与SQL / JSON条件is json 。脚1

如果未指定UNIQUE KEYS ，或者使用关键字WITHOUT UNIQUE KEYS ，则对象可以具有重复的字段名称，并且仍然可以认为格式正确。

Oracle数据库采用的评估始终只使用具有给定字段名称的一个对象成员;具有相同字段名称的任何其他成员将被忽略。未指定使用多个这样的成员中的哪一个。

在格式良好的JSON数据中是否允许重复字段名称与Oracle是否使用严格或松散语法来确定格式良好是正交的。

5.2关于Strict和Lax JSON语法

JSON的Oracle默认语法是松散的。特别是：它反映了对象字段的JavaScript语法; Boolean和null值不区分大小写;并且它对于数字，空格和Unicode字符的转义更加宽容。

标准ECMA-404（ JSON数据交换格式 ）和ECMA-262（ ECMAScript语言规范 ）定义了JSON语法。

根据这些规范，每个JSON字段和每个字符串值必须用双引号（ " ）括起来。Oracle支持这种严格的JSON语法 ，但它不是默认语法。

在JavaScript表示法中，对象文字中使用的字段可以（但不一定）用双引号括起来。它也可以用单引号（ ' ）括起来。Oracle还支持这种宽松的JSON语法 ，它是默认语法。

此外，在实践中，一些JavaScript实现（但不是JavaScript标准）允许以下一个或多个：

  • 关键字true ， false和null大小写变体（例如， TRUE ， True ， TrUe ， fALSe ， NulL ）。

  • 在数组的最后一个元素或对象的最后一个成员之后的额外逗号（ , ）（例如， [a, b, c , ] ， {a:b, c:d , } ）。

  • 带有一个或多个前导零的数字（例如， 0042.3 ）。

  • 在小数点前缺少0的小数（例如， .14而不是0.14 ）。

  • 小数点后没有小数部分的数字（例如， 342.或1.e27 ）。

  • 数字前面的加号（ + ），表示该数字是非负数（例如，+ +1.3 ）。

此语法也允许作为Oracle默认（松散）JSON语法的一部分。（有关严格的数字语法，请参阅JSON标准。）

除了ASCII空格字符（U + 0020）之外，在引用字段或字符串值之外使用时，JSON标准将以下字符定义为无效（忽略）空格：

  • 标签，水平标签（ HT ， ^I ，小数9，U + 0009， \t ）

  • 换行，换行符（ LF ， ^J ，十进制10，U + 000A， \n ）

  • 回车（ CR ， ^M ，十进制13，U + 000D， \r ）

但是，松散的JSON语法将所有 ASCII控制字符（Control + 0到Control + 31）以及ASCII空格字符（十进制32，U + 0020）视为（无关紧要的）空格字符。以下是控制字符：

  • 空（ NUL ， ^@ ，十进制0，U + 0000， \0 ）

  • 贝尔（ NEL ， ^G ，小数7，U + 0007， \a ）

  • 垂直选项卡（ VT ， ^K ，十进制11，U + 000B）

  • 逃生（ ESC ， ^[ ，十进制27，U + 001B， \e ）

  • 删除（ DEL ， ^? ,十进制127，U + 007F）

ASCII空格字符（U + 0020）是唯一允许，未转义，引用字段或字符串值中的空白字符。对于松散和严格的JSON语法都是如此。

对于严格和宽松的JSON语法，引用的对象字段和字符串值可以包含任何Unicode字符，但其中一些必须进行转义，如下所示：

  • 除了由以下转义序列表示的字符外，不允许使用ASCII控制字符： \b （退格）， \f （换页）， \n （换行符，换行符）， \r （回车符）和\t （ \t标签，水平标签）。

  • 双引号（ " ），斜杠（ / ）和反斜杠（ \ ）字符也必须进行转义（前面有反斜杠）： \" ， \/和\\分别。

在松懈的JSON语法， 未引述的对象字段可以包含除空白任何Unicode字符和JSON结构特征-左和右括号（ [ ， ] ）和大括号（ { ， } ），结肠癌（ : ），和逗号（ , ），但不允许转义序列。

通过使用ASCII转义语法\u后跟表示Unicode代码点的四个ASCII十六进制数字，任何Unicode字符也可以包含在名称或字符串中。

请注意，其他不可打印或可能显示为空格的Unicode字符（例如不间断空格字符（U + 00A0）） 不会被视为严格或宽松JSON语法的空白。

10 VLDB的存储管理

VLDB环境中的数据库文件的存储管理包括高可用性，性能和可管理性方面。

数据仓库环境中的存储性能通常转换为I / O吞吐量（MB / s）。对于联机事务处理（OLTP）系统，每秒I / O请求数（IOPS）是性能的关键度量。

基于硬件的镜像

大多数外部存储设备都支持不同的RAID（独立磁盘冗余阵列）级别。

VLDB环境中最常用的高可用性硬件RAID级别是RAID 1和RAID 5。虽然在VLDB环境中不太常用，但也可以使用其他高可用性RAID级别。

RAID 1镜像

RAID 1是一种基本的镜像技术。

已写入存储的每个存储块都按照RAID设置的定义存储在不同的物理设备上两次。RAID 1提供容错功能，因为如果一个设备出现故障，则会有另一个镜像设备可以响应数据请求。RAID 1设置中的两个写操作是在存储级别生成的。RAID 1至少需要两个物理磁盘才能生效。

存储设备通常提供在请求进入时读取主要或镜像的功能，与为高可用性设计的其他RAID配置相比，这可以产生更好的性能。RAID 1是最简单的硬件高可用性实现，但需要存储数据所需的存储量的两倍。RAID 1通常与RAID 0 + 1配置中的RAID 0（条带化）结合使用。在最简单的RAID 0 + 1配置中，单个条带跨两个物理设备进行镜像。

RAID 5镜像

RAID 5至少需要3个存储设备，但RAID 5组通常使用4到6个设备。

使用RAID 5时，对于写入设备的每个数据块，计算奇偶校验并将其存储在不同的设备上。在读取操作时，检查奇偶校验。奇偶校验计算发生在存储层中。RAID 5为设备故障提供高可用性，因为可以根据存储在其他设备上的奇偶校验来重建设备的内容。

RAID 5提供良好的读取性能。通过存储层中的奇偶校验计算可以减慢写入性能。RAID 5不需要两倍的存储量，而是需要较小的百分比，具体取决于RAID 5组中的设备数量。RAID 5相对复杂，因此并非所有存储设备都支持RAID 5设置。

使用Oracle ASM进行镜像

Oracle自动存储管理（Oracle ASM）提供基于软件的镜像功能。

Oracle ASM支持正常冗余（镜像）和高冗余（三重镜像）。Oracle ASM还支持使用外部冗余，在这种情况下，Oracle ASM不会执行其他镜像。可以将Oracle ASM正常冗余与RAID 1硬件镜像进行比较。

使用Oracle ASM镜像，镜像由数据库服务器生成。因此，与使用基于硬件的镜像相比，使用Oracle ASM镜像时，写操作需要更多的I / O吞吐量。根据您的配置和硬件RAID控制器的速度，Oracle ASM镜像或硬件RAID可能会为数据加载带来瓶颈。

在Oracle ASM中， 故障 组的定义可实现冗余，因为Oracle ASM跨故障组的边界镜像数据。例如，在VLDB环境中，您可以为每个磁盘阵列定义一个故障组，在这种情况下，Oracle ASM可确保镜像数据存储在不同的磁盘阵列上。这样，您不仅可以在磁盘阵列中的单个磁盘发生故障时幸免于难，而且您甚至可以在整个磁盘阵列发生故障或所有通道故障到该磁盘阵列时幸免于难。硬件RAID配置通常不支持这种容错。

使用正常冗余的Oracle ASM需要将存储数据所需的磁盘空间增加一倍。高冗余需要三倍的磁盘空间。

性能

要从存储设备获得最佳吞吐量，多个磁盘必须并行工作。

使用称为条带化的技术可以实现最佳吞吐量，该技术将数据块存储在跨多个设备的均衡切片（条带）中。Striping支持存储配置以获得良好的性能和吞吐量。

最佳存储设备性能是寻道时间和访问磁盘上连续块之间的权衡。在VLDB环境中，1 MB条带大小为OLTP系统和数据仓库系统提供了最佳性能和吞吐量的良好平衡。在数据库环境中有三种条带化选项：

  • 使用Oracle ASM进行基于软件的条带化

  • 不使用Oracle ASM的基于软件的条带化

基于硬件的条带化

大多数外部存储设备都提供条带化功能。最常用的提高存储性能的条带化技术是RAID 0和RAID 5。

  • RAID 0条带化

RAID 0条带化

RAID 0至少需要两个设备才能实现。

写入设备的数据块被拆分，并使用条带大小交替存储在设备上。该技术使得能够将多个设备和多个信道用于设备。

RAID 0尽管有RAID名称，但并不冗余。RAID 0配置中的设备丢失会导致数据丢失，并且应始终与关键环境中的某些冗余结合使用。在RAID 0 + 1配置中，使用RAID 0的数据库实现通常与RAID 1（基本镜像）结合使用。

RAID 5条带化

RAID 5配置使用特定于硬件的条带大小在RAID组中的可用设备上传播数据。

因此，使用多个设备和通道来读取和写入数据。由于其更复杂的奇偶校验计算，并非所有存储设备都支持RAID 5配置。

使用Oracle ASM进行条带化

Oracle自动存储管理（Oracle ASM）始终以作为磁盘组呈现给它的所有设备进行条带化。

磁盘组是您在其中创建数据文件的逻辑存储池。默认的Oracle ASM条带大小通常是VLDB的良好条带大小。

在磁盘组中使用具有相同性能特征的磁盘。磁盘组中的所有磁盘也应具有相同的大小，以实现最佳数据分配，从而获得最佳性能和吞吐量。磁盘组应跨越尽可能多的物理轴以获得最佳性能。VLDB的磁盘组配置不必与非VLDB的磁盘组配置不同。

Oracle ASM可以在先前条带化的存储设备之上使用。如果使用此类配置，请确保不要通过定义跨越逻辑设备的磁盘组来引入热点，这些逻辑设备实际上可能使用相同的资源（磁盘，控制器或通道到磁盘）而不是其他可用资源。始终确保Oracle ASM条带在所有物理设备上均匀分布。

信息生命周期管理

在信息生命周期管理（ILM）环境中，您无法在所有设备上使用条带化，因为所有数据都将分布在所有存储池中。

在ILM环境中，不同的存储池通常具有不同的性能特征。表空间不应跨越存储池，同一表空间的数据文件不应存储在多个存储池中。

应将ILM环境中的存储配置为在存储池中的所有设备上使用条带化。如果使用Oracle ASM，则应创建不同存储池的单独磁盘组。使用此方法，表空间不会将数据文件存储在不同的磁盘组中。可以使用分区表的分区移动操作在表空间之间在线移动数据，或者在未对表进行分区时使用DBMS_REDEFINITION包。

分区放置

如果您跨所有可用设备进行条带化并在所有可用资源之间分配负载，则不需要考虑分区放置。

如果无法跨所有可用设备对数据文件进行条带化，请考虑使用分区放置来优化所有可用硬件资源（物理磁盘轴，磁盘控制器和磁盘通道）的使用。

I / O密集型查询或DML操作应充分利用所有可用资源。将数据库对象分区存储在特定的表空间中，每个表空间使用一组不同的硬件资源，使您可以将所有资源用于针对单个分区数据库对象的操作。通过使用适当的分区技术，确保I / O密集型操作可以使用所有资源。

散列分区和散列子分区在唯一或几乎唯一的列或列集上，散列分区数等于2的幂是唯一一种在使用分区放置来优化I / O资源时可能导致均匀工作负载分布的技术利用。其他分区和子分区技术可能会产生类似的好处，具体取决于您的应用程序。

Bigfile表空间

Oracle数据库支持创建bigfile表空间。

bigfile表空间由单个数据或临时文件组成，最高可达128 TB。使用bigfile表空间可以显着减少数据库的数据文件数量。Oracle数据库支持在单个数据文件上进行并行RMAN备份和还原。

因此，使用bigfile表空间没有任何缺点，您可以选择使用bigfile表空间来显着减少数据和临时文件的数量。

文件分配是一个串行进程。如果对表和自动可扩展数据文件使用自动分配，则无论是否使用bigfile表空间，大型数据加载都会受到扩展文件所需时间的影响。但是，如果预分配数据文件并使用多个数据文件，则会生成多个进程以同时添加数据文件。

Oracle数据库文件系统（DBFS）

Oracle数据库文件系统（DBFS）利用数据库的优势来存储文件，以及数据库在有效管理关系数据方面的优势，以便为存储在数据库中的文件实现标准文件系统接口。

使用此接口，在数据库中存储文件不再局限于专门编写为使用BLOB和CLOB编程接口的程序。现在可以使用任何作用于文件的操作系统（OS）程序透明地访问数据库中的文件。

Oracle数据库文件系统（DBFS）在存储在数据库表中的文件和目录之上创建标准文件系统接口。使用DBFS，服务器是数据库。文件作为Oracle SecureFiles LOB存储在数据库表中。一组PL / SQL过程实现文件系统访问原语，例如create，open，read，write和list目录。数据库中文件系统的实现称为DBFS内容存储库。DBFS内容存储库允许每个数据库用户创建一个或多个可由客户端装入的文件系统。每个文件系统都有自己的专用表来保存文件系统内容。

可扩展性和可管理性

存储可扩展性和管理是VLDB环境中的重要因素。

VLDB的一个非常重要的特征是它的大尺寸。大尺寸引入了以下挑战：

  • 简单的统计数据表明存储组件更有可能失败，因为VLDB使用更多组件。

  • VLDB中的小的相对增长可能相当于显着的绝对增长，导致可能添加许多设备。

  • 尽管其尺寸，性能和（通常）可用性要求与较小的系统没有区别。

您选择的存储配置应该能够应对这些挑战。无论是故意还是意外添加或删除存储，您的系统都应从性能和高可用性角度保持最佳状态。

条纹和镜像一切（相同）

Oracle多年来一直推荐使用条带和镜像一切（SAME）方法，这是一种优化高可用性，性能和可管理性的方法。

为了进一步简化配置，SAME方法中建议使用1 MB的固定条带大小作为OLTP和数据仓库系统的良好起点。Oracle ASM实现了SAME方法并在其上添加了自动化。

相同和可管理性

为了实现最高性能，SAME方法建议对尽可能多的物理设备进行条带化。

这可以在没有Oracle ASM的情况下实现，但是如果存储配置发生更改（例如，通过添加或删除设备），则设备上的数据库文件的布局应该更改。Oracle ASM在后台自动执行此任务。在大多数非Oracle ASM环境中，重新条带化是一项主要任务，通常涉及手动干预。

在ILM环境中，您将SAME方法应用于每个存储池。

特定于VLDB的Oracle ASM设置

VLDB的Oracle自动存储管理配置与非VLDB的Oracle ASM配置类似。

某些参数值（例如Oracle ASM实例的内存分配）可能需要更高的值。

Oracle数据库支持Oracle ASM变量分配单元。大型变量分配单元对于使用大型顺序I / O操作的环境非常有用。一般而言，VLDB，特别是大型数据仓库，是利用大型分配单元的良好候选环境。分配单元可以设置在1 MB到64 MB之间，功率为2（即1,2,4,8,16,32和64）。如果您的工作负载包含大量扫描大型表的查询，那么您应该使用大型Oracle ASM分配单元。对于非常大的数据仓库系统，请使用64 MB。大型分配单元还可降低Oracle ASM的内存要求并改善Oracle ASM启动时间。

BasicFiles LOB的一般性能指南

您可以使用BasicFiles LOB实现最高性能。

使用这些指南可以最大限度地提高BasicFiles LOB的性能：

  • 可能时，读/写一次大数据块：

    由于LOB很大，您可以通过一次读取和写入大块LOB值来获得最佳性能。这在几个方面有所帮助：

    1. 如果从客户端访问LOB并且客户端与服务器位于不同的节点，那么大的读/写会减少网络开销。

    2. 如果使用NOCACHE选项，则每次小的读/写都会产生I / O.读/写大量数据可减少I / O.

    3. 写入LOB会创建新版本的LOB块。因此，一次写入少量数据会导致每个小写入的新版本的成本。如果启用了日志记录，则块也会存储在重做日志中。

  • 将OCILobRead2()和OCILobWrite2()与回调一起使用：

    这样数据就可以流入和流出LOB。确保在输入的amount参数中设置整个写入的长度。只要有可能，读取和写入LOB 块大小的倍数 。

  • 使用Checkout / Check-in Model进行LOB：

    LOB针对以下操作进行了优化：

    • SQL UPDATE替换整个LOB值

    • 将整个LOB数据复制到客户端，修改客户端上的LOB数据，将整个LOB数据复制回数据库。这可以使用OCILobRead2 （）和OCILobWrite2 （）与流完成。

  • 经常提交更改。

临时LOB

除了关于LOB性能的“ LOB性能指南 ”中描述的指南之外，以下是使用临时LOB的一些指导原则：

  • 使用PGA内存来存储临时LOB以提高性能。

  • 使用单独的临时表空间进行临时LOB存储，而不是使用默认系统表空间

    这可以避免在将数据从持久LOB复制到临时LOB时发生设备争用。

    如果在应用程序中使用新提供的增强型SQL语义功能，那么SQL和PL / SQL中会比以前静默创建更多临时LOB。确保用于存储这些临时LOB的临时表空间 足够大，以便您的应用程序使用。特别是，当您使用以下内容时，将以静默方式创建这些临时LOB：

    • LOB上的SQL函数

    • LOB上的PL / SQL内置字符函数

    • 从VARCHAR2 / RAW到CLOB s / BLOB的变量赋值。

    • 执行LONG to- LOB迁移

  • 如果在LOB上使用SQL运算符，则PGA内存和临时表空间必须足够大，以容纳SQL运算符生成的临时LOB。

  • 释放从SQL查询和PL / SQL程序返回的临时LOB

    在PL / SQL，C（OCI），Java和其他编程接口中，SQL查询结果或PL / SQL程序执行返回临时LOB以进行LOB上的操作/函数调用。例如：

    SELECT substr（CLOB_Column，4001,32000）FROM ...

    如果在PL / SQL中执行查询，则在PL / SQL程序块结束时自动释放返回的临时LOB。您还可以随时显式释放临时LOB。在OCI和Java中，必须显式释放返回的临时LOB。

    如果没有正确释放从SQL查询返回的临时LOB，则会填充临时表空间，您可能会发现性能下降。

  • 在PL / SQL中，使用 NOCOPY尽可能通过引用传递临时LOB参数。

    也可以看看：

    有关通过引用和参数别名传递参数的更多信息，请参见Oracle Database PL / SQL语言参考

  • 利用临时LOB上的缓冲区缓存。

    将CACHE参数设置为true创建的临时LOB在缓冲区缓存中移动。否则，临时LOB将直接从磁盘读取并直接写入磁盘。

  • 为获得最佳性能，临时LOB使用读取时的引用，写入语义上的复制。将临时LOB定位器分配给另一个定位器时，不会复制物理LOB数据。使用任一LOB定位符的后续READ操作指的是相同的物理LOB数据。在赋值后的第一个WRITE操作中，复制物理LOB数据以保留LOB值语义，即，确保每个定位符指向唯一的LOB值。此性能考虑主要适用于PL / SQL和OCI环境。

    在PL / SQL中，有关读取，写入语义复制的参考说明如下：

    LOCATOR1 BLOB; LOCATOR2 BLOB; DBMS_LOB.CREATETEMPORARY（LOCATOR1，TRUE，DBMS_LOB.SESSION）; - 在此赋值操作中不复制LOB数据：LOCATOR2：= LOCATOR; - 这些读取操作引用相同的物理LOB副本：DBMS_LOB.READ（LOCATOR1，...）; DBMS_LOB.GETLENGTH（LOCATOR2，...）; -  LOB数据的物理副本在WRITE上进行：DBMS_LOB.WRITE（LOCATOR2，...）;

    在OCI中，为了确保LOB定位符和数据的值语义， OCILobLocatorAssign()用于复制临时LOB定位符和LOB数据。OCILobLocatorAssign()不会往返服务器。当LOB更新在与LOB更新API相同的往返中发生时，进行物理临时LOB复制，如下所示：

    OCILobLocator * LOC1; OCILobLocator * LOC2; OCILobCreateTemporary（...LOC1，......TRUE，OCI_DURATION_SESSION）; / *以下呼叫不会产生往返。* / OCILobLocatorAssign（...LOC1，LOC2）; / *读取操作指的是相同的物理LOB副本。* / OCILobRead2（...LOC1 ...）/ *为了制作* LOB数据的新副本并写入新的LOB副本，会产生一次往返。* / OCILobWrite2（...LOC1 ...）/ * LOC2看不到与LOC1相同的LOB数据。* / OCILobRead2（...LOC2 ...）

    如果不打算使用LOB值语义，那么可以使用C指针来实现引用语义，如下所示：

    OCILobLocator * LOC1; OCILobLocator * LOC2; OCILobCreateTemporary（...LOC1，......TRUE，OCI_DURATION_SESSION）; / *指针被复制。LOC1和LOC2指的是相同的LOB数据。* / LOC2 = LOC1; / *写入LOC2。* / OCILobWrite2（...LOC2 ...）/ * LOC1看到对LOC2的更改。* / OCILobRead2（...LOC1 ...）

  • 对临时LOB使用OCI_OBJECT模式

    要提高LOB分配上临时LOB的性能， OCI_OBJECT模式用于OCILobLocatorAssign() 。在OCI_OBJECT模式下，数据库尝试最小化要执行的深层副本的数量。因此，在OCI_OBJECT模式下对源临时LOB完成OCILobLocatorAssign()之后，源和目标定位器指向相同的LOB，直到通过任一LOB定位器进行任何修改。

LOB访问统计

在Oracle Database 10g第2版之后，用户可以使用三个特定于LOB的会话级统计信息：LOB读取，LOB写入和LOB写入未对齐。

可以通过V$MYSTAT ， V$SESSTAT和V$SYSSTAT动态性能视图访问会话统计V$SYSSTAT 。要查询这些视图，必须为用户授予SELECT_CATALOG_ROLE ， SELECT ON SYS.V_$MYSTAT视图和SELECT ON SYS.V_$STATNAME视图的SELECT ON SYS.V_$STATNAME 。

LOB读取定义为在会话/系统中执行的LOB API读取操作的数量。单个LOB API读取可以对应于多个物理/逻辑磁盘块读取。

LOB写入定义为在会话/系统中执行的LOB API写入操作的数量。单个LOB API写入可以对应于多个物理/逻辑磁盘块写入。

LOB写入未对齐定义为LOB API写入操作的数量，其起始偏移量或缓冲区大小未与LOB的内部块大小对齐。与块边界对齐的写入是最有效的写入操作。LOB的内部块大小可通过LOB API获得（例如，使用PL / SQL，通过DBMS_LOB.GETCHUNKSIZE() ）。

168.1 DBMS_TDB概述

在许多情况下，Oracle支持在具有相同endian格式的平台之间传输数据库。但是，即使endian格式相同，数据库也必须经过转换过程才能从一个平台移动到另一个平台。传输数据库的过程也需要先决条件，例如将数据库以只读方式打开。

DBMS_TDB包有两个用途：

  • 确认Oracle支持将数据库从给定的源平台传输到给定的目标平台

  • 确定要传输的数据库是否已准备好进行传输，如果没有，则确定阻止数据库传输的条件

使用Recovery Manager CONVERT DATABASE命令执行实际转换。有关传输数据库的要求，跨平台转换数据库以进行传输的过程以及在转换过程中使用DBMS_TDB子程序的示例的完整讨论，请参阅“ Oracle数据库备份和恢复用户指南” 。

2.15删除

目的

使用DELETE命令执行以下操作：

  • 删除物理备份和副本。

  • 删除稀疏数据库的过时备份。

  • 更新目标控制文件中的存储库记录以显示文件已删除。例如， V$BACKUP_PIECE.STATUS中备份件的记录显示值D

  • 从恢复目录中删除已删除文件的存储库记录（如果使用目录）。例如， RC_BACKUP_PIECE不再包含已删除备份片段的行。

也可以看看：

BACKUP了解BACKUP ...DELETE INPUT命令

先决条件

RMAN必须连接到目标数据库，该数据库必须安装或打开。

RMAN使用所有已配置的通道执行删除。如果对未配置为自动通道的设备上的文件使用DELETE ，则必须使用ALLOCATE CHANNEL FOR MAINTENANCE 。例如，如果使用SBT通道进行备份，但只配置了磁盘通道，则必须手动为DELETE分配SBT通道。在仅磁盘文件上使用DELETE时，需要自动或手动分配的维护通道。

对于预备份备份，CDB必须以读写模式打开，并且必须以SYSDBA或SYSBACKUP权限的身份以普通用户身份进行连接。

使用说明

最佳做法是运行CROSSCHECK以更新存储库中的备份和副本的状态，然后运行DELETE以删除所需的文件。以交互方式运行RMAN时， DELETE会在删除列表中的任何文件之前显示文件列表并提示进行确认。如果您确认，则RMAN会在删除时显示每个项目。从命令文件中读取命令时，RMAN不会提示确认。

您可以通过LIST ， V$视图或恢复目录视图（如果使用目录）查看RMAN存储库中记录的备份和副本的状态。备份的存储库记录可能无法反映其物理状态。例如，用户使用Linux rm命令删除磁盘备份。rm无法更新备份记录，因此RMAN存储库将文件显示为可用，尽管它已不存在。

注意：如果使用KEEP UNTIL TIME子句进行备份，则在指定的KEEP时间过后，此备份将过时，并由DELETE OBSOLETE 。对于KEEP时间已过期的归档备份，RMAN不考虑备份保留策略。

注意： DELETE...将备份存储到零数据丢失恢复设备（通常称为恢复设备）时，无法使用OBSOLETE命令。

   obsOperandList

指定用于确定哪些备份和副本已过时的条件。

另请参见： obsOperandList
   DEVICE TYPE    deviceSpecifier

将删除限制为仅在指定设备类型上创建的过时备份和副本。

另请参见： deviceSpecifier

使用DBUA在Linux，Unix和Windows系统上升级数据库

要使用DBUA图形用户界面升级数据库，请在新的Oracle主目录中执行这些步骤。

在Microsoft Windows系统（Windows）上，将DBUA作为Oracle数据库管理用户（具有操作系统分配的ORA_DBA角色的用户）或Oracle安装所有者帐户运行。

  1. 从安装了新数据库软件的Oracle主目录启动Oracle数据库升级助手（DBUA）。dbua可执行文件位于目录路径ORACLE_HOME /bin 。

    • 在Linux或Unix平台上，以具有SYSDBA权限的用户身份登录，并在Oracle数据库19c的新主目录中的系统提示符处输入以下命令：

      ./dbua

    • 在Windows操作系统上，依次选择“ 开始” ，“ 程序” ，“ Oracle HOME_NAME ，“ 配置和迁移工具” ，然后选择“ 数据库升级助手” 。

  2. 将显示“ 选择数据库”窗口如果您之前已发布Oracle数据库安装，则会列出这些安装可用于升级。

    说明dbua_1.png的说明

    如果您在任何DBUA窗口上需要帮助，或者如果您想查阅有关DBUA的更多文档，请单击“ 帮助”以打开联机帮助。

    如果需要，请输入所选数据库的SYSDBA用户名和密码。

    如果从没有SYSDBA权限的用户帐户运行DBUA，或者源数据库没有操作系统身份验证，则必须输入用户名和密码凭据才能为所选数据库启用SYSDBA权限。如果您使用的用户帐户具有SYSDBA权限，或者您使用了操作系统身份验证，则无需输入用户名和密码。

    选择后单击“ 下一步” 。

    注意：

    • 您一次只能选择一个数据库。

    • 对于单实例升级，如果数据库未出现在列表中，则检查/etc/oratab存在具有数据库名称的条目。如果未在此列出数据库，则指示DBUA升级特定数据库：

      • 如果您的单实例数据库未在/etc/oratab列出，并且DBUA可以连接到数据库，那么您可以通过使用命令行参数-sid Oracle_SID ， -oracleHome Oracle_home , and sysDBAPassword password启动DBUA来指示DBUA显式升级该数据库-oracleHome Oracle_home , and sysDBAPassword password作为命令行参数。例如：

        dbua -sid Oracle_SID -oracleHome /u01/app/oracle/18.1.0/dbhome1 -sysDBAUserName SYS -sysDBAPassword password
    • 如果您的帐户没有SYSDBA权限，或者您没有设置操作系统身份验证，则可以使用以下语法进行连接，其中mydb是您的Oracle数据库SID， username是具有SYSDBA权限的用户名， password是用户名的密码：
      dbua -sid mydb -oracleHome /u01/app/oracle/18.1.0/dbhome1 -sysDBAUserName  - username -sysDBAPassword  - password

    • Oracle Real Application Clusters（Oracle RAC）升级：如果数据库未出现在列表中，请输入以下crsctl命令以检查Oracle RAC实例：

      crsctl status resource -t

      您还可以输入以下命令来检查特定的Oracle RAC数据库，其中db_name是Oracle RAC数据库名称：

      crsctl status resource ora。 db_name .db

    • 在Microsoft Windows上，以下安全更改会影响身份验证和用户帐户：

      • 出于安全原因，不再支持使用NTLM协议的Windows NTS身份验证。Kerberos身份验证是唯一受支持的身份验证。在此版本中，NTS在Windows NT域或Windows NT控制器的域中不起作用。

      • Oracle使用标准Microsoft Windows用户帐户而不是Windows LocalSystem帐户来运行Oracle数据库服务。减少Oracle安装所有者的帐户访问权限可在Microsoft Windows上提供更好的安全性。

  3. 如果所选数据库是多租户容器数据库（CDB），则DBUA将显示“ 可插拔数据库”窗口。“ 可插拔数据库”窗口列出了CDB中包含的可插拔数据库。列出的PDB将作为所选CDB升级的一部分进行升级。


    说明dbua_2.png的描述

    您可以选择PDB的升级优先级。单击每个PDB的优先级列，然后输入升级优先级的数值，其中1先升级，2升级第二，依此类推。

    默认情况下， CDB$ROOT ， PDB$SEED以及插入CDB的所有PDB都会升级。如果您现在不想升级某些PDB，请拔下这些PDB。

    完成选择PDB和升级优先级后，单击“ 下一步” 。

  4. 仅限Windows平台：如果升级目标主目录是与Oracle主目录用户关联的安全主目录，则会打开“ 指定Oracle主目录用户密码”窗口。对于其他平台，请继续执行下一步。


    说明dbua3a-win.png的描述

    提供Oracle主用户名，并提供此用户帐户的密码，然后单击“ 下一步” 。

  5. 将打开“ 先决条件检查”窗口。DBUA分析数据库，执行预升级检查并根据需要显示警告。以下是DBUA检查的示例列表，以及DBUA对数据库执行的操作的列表：

    • 空数据库回收站。

    • 识别无效对象。

    • 识别已弃用和不支持的初始化参数。

    • 确定时区数据文件版本。

    分析需要几分钟才能完成。

    当DBUA完成其分析时，将再次显示“ 先决条件检查”窗口，显示检查结果。


    说明dbua_3.png的描述

    “ 先决条件检查”窗口显示DBUA已完成的检查以及发现的任何错误的严重性。当DBUA发现错误时，它指示哪些错误是可修复的，以及您可以采取哪些操作来纠正错误。

    如果出现DBUA可以修复的任何错误， 请选择“ 修复并再次检查” 。

    如果DBUA检测到无法纠正的错误， 请手动修复错误原因，然后选择“ 再次检查” 。

    如果DBUA没有发现错误或警告，则DBUA会自动绕过此窗口并进入下一个窗口。

    修复检测到的错误后，单击“ 下一步” 。

  6. 将显示“ 选择升级选项”窗口。


    说明dbua_4.png的描述

    启用并行升级

    在升级后重新编译无效对象

    升级完成后，此选项将重新编译所有无效的PL / SQL模块。如果在升级后阶段没有DBUA重新编译无效对象，则必须在升级数据库后手动重新编译无效对象。

    升级时区数据

    此选项更新此版本的时区数据文件。如果未选择此选项，则必须在升级后手动更新时区配置文件。

    指定要执行的自定义SQL脚本 。

    如果要在升级过程中运行自定义SQL脚本，请选择此选项。根据需要，单击“ 浏览”以获取“ 升级前”或“ 升级后”输入字段。导航到自定义SQL脚本所在的位置。

    完成选择后，单击“ 下一步” 。

    • 使用闪回和保证还原点 。

      您可以创建新的保证还原点，也可以使用现有的还原点。如果使用现有还原点，则单击选择域以选择要使用的还原点。

      注意：

      如果要升级的数据库具有Oracle Data Guard物理备用数据库，则必须先在每个备用数据库上创建保证还原点，然后才能在主数据库上创建一个备用数据库。如果不首先在每个备用数据库上创建还原点，则必须在使用保证还原点降级主数据库后再次重新创建所有备用数据库。升级成功后，您必须手动删除备用数据库上所有保证还原点。

    • 使用RMAN备份

      您可以创建新的脱机RMAN备份，也可以使用现有备份。单击“ 浏览”以指定备份的路径。

    • 使用最新可用的RMAN备份

      您可以使用现有备份。单击“ 查看/编辑还原脚本”以选择要使用的备份。

    • 我有自己的备份和恢复策略 。

      仅当您具有适用于现有数据库的第三方备份解决方案时，才选择此选项。

    完成选择后，单击“ 下一步” 。

您将创建一个Oracle Label Security策略，该策略将使用以下级别的敏感性来管理对当前和以前员工的访问：

  • SENSITIVE仅允许访问当前员工。用户Ida Neau将被分配这个级别。
  • HIGHLY_SENSITIVE可以访问前雇员。用户Susan Mavris将被分配到这个级别。此级别比SENSITIVE更高级别，这意味着它还将提供对SENSITIVE保护的行的访问。换句话说，Susan Mavris可以访问前雇员和当前员工记录。

第1步：创建角色和用户帐户

您创建的角色将允许授予它的任何用户在HR.EMPLOYEES表上具有SELECT权限。用户帐户适用于两名人力资源员工Susan Mavris和Ida Neau。

  1. 以具有创建角色，授予权限和创建用户帐户权限的用户身份登录SQL * Plus。
    例如：
    sqlplus sec_admin输入密码： 密码
  2. 创建角色如下：
    创建角色HR_ROLE;
  3. 将HR.EMPLOYEES的SELECT权限授予HR_ROLE 。
    在HR.EMPLOYEES上选择HR_ROLE;
  4. 为Susan Mavris和Ida Neau创建用户帐户，并授予他们HR_ROLE角色。
    GRANT CONNECT，HR_ROLE到SMAVRIS通过密码识别; GRANT CONNECT，HR_ROLE to INEAU通过密码识别;

第2步：创建Oracle Label Security策略容器

作为Oracle Label Security管理员，您必须创建并启用策略容器。

  1. 以可以创建和管理Oracle Label Security策略的用户身份进行连接。
    例如：
    sqlplus psmith_ols输入密码： 密码
  2. 创建策略。
    BEGIN SA_SYSDBA.CREATE_POLICY（policy_name =>'HR_OLS_POL'，column_name =>'OLS_COL'）;结束; /

    在本规范中，省略了default_options参数，因为您可以稍后在另一个过程中添加它。

  3. 启用该策略。
    EXEC SA_SYSDBA.ENABLE_POLICY（'HR_OLS_POL'）;

步骤3：为Oracle Label Security策略创建两级组件

创建Oracle Label Security策略容器后，您就可以创建两个级别来表示两种不同的灵敏度级别。

  1. 如有必要，请以可以创建和管理Oracle Label Security策略的用户身份进行连接。
    例如：
    sqlplus psmith_ols输入密码： 密码
  2. 创建级别如下：
    BEGIN SA_COMPONENTS.CREATE_LEVEL（policy_name =>'HR_OLS_POL'，level_num => 3000，short_name =>'HS'，long_name =>'HIGHLY_SENSITIVE'）; SA_COMPONENTS.CREATE_LEVEL（policy_name =>'HR_OLS_POL'，level_num => 2000，short_name =>'S'，long_name =>'SENSITIVE'）;结束; /

    在本规范中：

    • policy_name将级别与您刚刚创建的策略容器相关联。
    • level_num确定用户可以拥有多少访问权限。级别号3000使用户能够访问此级别及其下的任何级别编号，在本例中为级别编号2000 。换句话说，被授权具有HIGHLY_SENSITIVE级别的用户也可以访问分配给SENSITIVE级别的数据。
    • short_name对于短手的名字long_name水平，并会在其它程序被用来指long_name水平的版本。

第4步：为级别创建数据标签

数据标签标记数据记录以与Oracle Label Security策略一起使用。

在此过程中，数据标签将指定用户Susan Mavris和Ida Neau将在HR.EMPLOYEES表中看到的行。标记为HS行将对应于要分配给Susan Mavris的HS （ HIGHLY_SENSITIVE ）级别，标记为S行将对应于要分配给Ida Neau的S （ SENSITIVE ）级别。
  1. 如有必要，请以可以创建和管理Oracle Label Security策略的用户身份进行连接。
    例如：
    sqlplus psmith_ols输入密码： 密码
  2. 创建数据标签，如下所示：
    BEGIN SA_LABEL_ADMIN.CREATE_LABEL（policy_name =>'HR_OLS_POL'，label_tag => 3100，label_value =>'HS'，data_label => TRUE）; SA_LABEL_ADMIN.CREATE_LABEL（policy_name =>'HR_OLS_POL'，label_tag => 2100，label_value =>'S'，data_label => TRUE）;结束; /

    在本规范中：

    • label_tag由Oracle Label Security在内部用于标识级别。与级别不同，它不管理带有标签的任何类型的层次结构。
    • data_label设置为TRUE以便标签可以应用于行数据。

步骤5：为Oracle Label Security Policy设置用户授权

设置用户授权需要将用户与策略以及与Oracle Label Security策略关联的最低和最高级别相关联。

  1. 如有必要，请以可以创建和管理Oracle Label Security策略的用户身份进行连接。
    例如：
    sqlplus psmith_ols输入密码： 密码
  2. 授权用户如下：
    BEGIN SA_USER_ADMIN.SET_LEVELS（policy_name =>'HR_OLS_POL'，user_name =>'SMAVRIS'，max_level =>'HS'，min_level =>'S'）; SA_USER_ADMIN.SET_LEVELS（policy_name =>'HR_OLS_POL'，user_name =>'INEAU'，max_level =>'S'，min_level =>'S'）;结束; /

    在本规范中，省略了def_level （默认级别）和row_level参数，以便它们的值可以默认为max_level参数设置。

步骤6：将Oracle Label Security Policy应用于HR Schema

将策略应用于HR模式后，必须启用与HR的策略关联。

  1. 如有必要，请以可以创建和管理Oracle Label Security策略的用户身份进行连接。
    例如：
    sqlplus psmith_ols输入密码： 密码
  2. 将策略应用于HR模式。
    BEGIN SA_POLICY_ADMIN.APPLY_TABLE_POLICY（policy_name =>'HR_OLS_POL'，schema_name =>'HR'，table_name =>'EMPLOYEES'，table_options =>'READ_CONTROL'）;结束; /

    之前，当您使用SA_SYSDBA.CREATE_POLICY过程创建策略时，未设置default_options参数，该参数定义策略强制选项。因此，必须使用table_options参数SA_POLICY_ADMIN.APPLY_TABLE_POLICY在此处设置策略实施。 READ_CONTROL在SELECT语句处理期间强制执行OLS策略，用户稍后将执行该策略。（它也适用于UPDATE和DELETE语句处理。）

  3. 启用策略与HR模式的关联。
    BEGIN SA_POLICY_ADMIN.ENABLE_TABLE_POLICY（policy_name =>'HR_OLS_POL'，schema_name =>'HR'，table_name =>'EMPLOYEES'）;结束; /

步骤7：将策略标签添加到HR.EMPLOYEES表数据

Oracle Label Security管理员和HR用户都将策略标签添加到EMPLOYEE_ID列中的HR.EMPLOYEES表数据。

  1. 如有必要，请以可以创建和管理Oracle Label Security策略的用户身份进行连接。
    例如：
    sqlplus psmith_ols输入密码： 密码
  2. 将READ权限授予HR用户的HR_OLS_POL策略。
    BEGIN SA_USER_ADMIN.SET_USER_PRIVS（policy_name =>'HR_OLS_POL'，user_name =>'HR'，privileges =>'READ'）;结束; /
  3. 以HR用户身份连接。
    connect hr输入密码： 密码
  4. 执行以下UPDATE语句以将HIGHLY_SENSITIVE级别应用于已离开公司的用户的员工ID。
    此UPDATE语句控制Susan Mavris对HR.EMPLOYEES表的访问权限，因为她已获得HIGHLY_SENSITIVE级别的授权。 
    UPDATE员工SET ols_col = CHAR_TO_LABEL（'HR_OLS_POL'，'HS'）WHERE UPPER（employee_id）IN（200,101,102,176,201,122,114）;
  5. 执行以下UPDATE语句以将SENSITIVE级别应用于公司当前员工的员工ID。
    此UPDATE语句控制Ida Neau对HR.EMPLOYEES表的访问权限，因为她已获得SENSITIVE级别的授权。 
    UPDATE员工SET ols_col = CHAR_TO_LABEL（'HR_OLS_POL'，'S'）WHERE UPPER（employee_id）NOT IN（200,101,102,176,201,122,114）;

    此UPDATE语句转换为“将SENSITIVE标签应用于任何不是前雇员的员工”。

    SELECT FIRST_NAME，LAST_NAME，EMPLOYEE_ID，LABEL_TO_CHAR（OLS_COL）OLS_LABEL来自HR.EMPLOYEES ORDER by OLS_COL;
