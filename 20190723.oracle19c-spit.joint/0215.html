D初始化参数

init.ora文件中的Oracle数据库初始化参数与网关初始化参数不同。使用特定于代理的机制在初始化参数文件中设置网关参数，或使用DBMS_HS包在Oracle数据字典中设置它们。网关启动时，网关初始化参数文件必须可用。

以下主题包含可以为每个网关及其描述设置的网关初始化参数的列表。主题还描述了初始化参数文件语法。

  • 用于SQL Server初始化参数的Oracle Database Gateway

初始化参数文件语法

初始化参数文件的语法如下：

  • 该文件是一系列命令。

  • 每个命令应该从单独的一行开始。

  • 行尾被视为命令终止符（除非使用反斜杠进行转义）。

  • 如果初始化参数文件中存在语法错误，则所有设置都不会生效。

  • 设置参数值如下：

    [SET] [PRIVATE] 参数 = 值

    哪里：

    parameter是初始化参数名称。它是一个以字母开头的字符串，由字母，数字和下划线组成。初始化参数名称区分大小写。

    value是初始化参数值。它区分大小写。初始化参数值是：

    • 一串不包含任何反斜杠，空格或双引号的字符（“）

    • 带引号的字符串，以双引号开头，以双引号结尾。以下内容可以在带引号的字符串中使用：

      • 反斜杠（\）是转义字符

      • \ n插入一个新行

      • \ t插入一个标签

      • \“插入双引号

      • \\插入一个反斜杠

      该行末尾的反斜杠继续下一行的字符串。如果反斜杠在任何其他字符之前，则忽略反斜杠。

    例如，要为代理启用跟踪，请按如下所示设置HS_FDS_TRACE_LEVEL初始化参数：

     HS_FDS_TRACE_LEVEL = ON

    SET和PRIVATE是可选关键字。您不能将它们用作初始化参数名称。大多数参数仅作为初始化参数需要，因此您通常不需要使用SET或PRIVATE关键字。如果未指定SET或PRIVATE ，则该参数仅用作代理的初始化参数。

    SET指定除了用作初始化参数之外，还将参数值设置为代理进程的环境变量。将SET用于驱动程序或非Oracle系统需要的参数值作为环境变量。

    PRIVATE指定初始化参数应该是代理的私有参数，不应上载到Oracle数据库。大多数初始化参数不应该是私有的。但是，如果您在初始化参数文件中存储密码等敏感信息，则可能不希望将其上载到服务器，因为上载时初始化参数和值未加密。将初始化参数设置为私有可防止上载发生，并且它们不会出现在动态性能视图中。仅当参数值包含敏感信息（如用户名或密码）时，才使用PRIVATE作为初始化参数。

    SET PRIVATE指定将参数值设置为代理进程的环境变量，并且也是私有的（不会传输到Oracle数据库，不会出现在动态性能视图或图形用户界面中）。

HS_CALL_NAME

属性 描述

默认值

没有

价值范围

不适用

指定可在SQL语句中引用的远程函数。该值是远程函数及其所有者的列表，以分号分隔，格式如下：

owner_name.function_name

例如：

owner1。A1; owner2。A2; owner3。A3

如果未为远程功能指定所有者名称，则默认所有者名称将成为用于连接到远程数据库的用户名（在创建异构服务数据库链接时指定，如果未在数据库链接中指定，则从用户会话中获取） 。

所有者名称和函数名称的条目区分大小写。

HS_DB_DOMAIN

属性 描述

默认值

世界

价值范围

1到199字符

为非Oracle系统指定唯一的网络子地址。HS_DB_DOMAIN初始化参数类似于Oracle Database Reference描述的DB_DOMAIN初始化参数。如果使用Oracle Names服务器，则需要HS_DB_DOMAIN初始化参数。HS_DB_NAME和HS_DB_DOMAIN初始化参数定义非Oracle系统的全局名称。

注意：

HS_DB_NAME和HS_DB_DOMAIN初始化参数必须组合以在协作服务器环境中形成唯一地址。

HS_DB_INTERNAL_NAME

属性 描述

默认值

01010101

价值范围

1到16十六进制字符

指定标识异构服务代理连接到的实例的唯一十六进制数。激活全局名称服务时，此参数的值将用作事务ID的一部分。当事务需要两阶段提交恢复操作时，指定非唯一编号可能会导致问题。

HS_DB_NAME

属性 描述

默认值

HO

价值范围

1至8字符

为给予非Oracle系统的数据存储指定唯一的字母数字名称。这个名字标识了协作服务器环境中的非Oracle系统。HS_DB_NAME和HS_DB_DOMAIN初始化参数定义非Oracle系统的全局名称。

HS_DESCRIBE_CACHE_HWM

属性 描述

默认值

100

价值范围

1到4000

指定异构服务使用的描述高速缓存中的最大条目数。这个限制被称为描述缓存高水位线。缓存包含异构服务重用的映射表的描述，因此它不必重新访问非Oracle数据存储。

如果要访问许多映射表，请增加高水位标记以提高性能。增加高水位标记会以内存使用为代价提高性能。

HS_LANGUAGE

属性 描述

默认值

系统专用

价值范围

任何有效的语言名称（最多255字符）

为异构服务提供非Oracle数据源的字符集，语言和区域信息。该值必须使用以下格式：

语言 [_ 领土 。 character_set ]

注意：

全球化支持初始化参数会影响错误消息，SQL Service的数据以及分布式外部过程中的参数。

字符集

理想情况下，Oracle数据库和非Oracle数据源的字符集是相同的。在几乎所有情况下， HS_LANGUAGE应设置为与Oracle数据库字符集完全相同，以实现最佳字符集映射和性能。如果它们不相同，则异构服务会尝试将非Oracle数据源的字符集转换为Oracle数据库字符集，然后再返回。翻译会降低性能。在某些情况下，异构服务无法将字符从一个字符集转换为另一个字符集。

注意：

指定的字符集必须是安装代理程序的平台上的操作系统字符集的超集。

由于更多Oracle数据库和非Oracle数据库使用Unicode作为数据库字符集，因此最好还以Unicode字符集运行网关。为此，您必须设置HS_LANGUAGE=AL32UTF8 。但是，当网关在Windows上运行时，Microsoft ODBC驱动程序管理器界面只能以双字节字符集UCS2交换数据。这导致所述缓冲液和柱尺寸的额外比率扩展。有关如何调整到正确尺寸的说明，请参阅HS_FDS_REMOTE_DB_CHARSET 。

语言

HS_LANGUAGE初始化参数的语言组件确定：

  • 日期的日期和月份名称

  • 日期和时间的AD，BC，PM和AM符号

  • 默认排序机制

请注意，Oracle不确定通用异构服务消息（ ORA-25000到ORA-28000 ）的错误消息的语言。它们由Oracle数据库中的会话设置控制。

领土

territory子句指定日期和星期编号的约定，默认日期格式，十进制字符和组分隔符以及ISO和本地货币符号。请注意，Oracle数据库与非Oracle数据源之间的全球化支持级别取决于网关的实现方式。

HS_LONG_PIECE_TRANSFER_SIZE

属性 描述

默认值

64 KB

价值范围

任何值最多2 GB

设置要传输的LONG数据的大小。较小的块大小意味着更少的内存需求，但更多的往返来获取所有数据。较大的单件尺寸意味着更少的往返次数，但更多的内存需要在内部存储中间件。因此，初始化参数可用于调整系统以获得最佳性能，在往返和内存要求之间以及网络延迟或响应时间之间进行最佳权衡。

HS_OPEN_CURSORS

属性 描述

默认值

50

价值范围

1到Oracle的OPEN_CURSORS初始化参数的值

定义在与非Oracle系统实例的一个连接上可以打开的最大游标数。

该值永远不会超过Oracle数据库中打开游标的数量。因此，建议在Oracle数据库中设置与OPEN_CURSORS初始化参数相同的值。

HS_RPC_FETCH_REBLOCKING

属性 描述

默认值

上

价值范围

OFF或ON

控制异构服务是否尝试优化Oracle数据库与连接到非Oracle数据存储的异构服务代理之间的数据传输性能。

可能的值如下：

  • OFF禁用对已获取数据的重新锁定，以便立即将数据从代理发送到服务器。

  • ON启用重新锁定，这意味着从非Oracle系统获取的数据将缓冲在代理中，并且在获取的数据量等于或高于HS_RPC_FETCH_SIZE初始化参数的值之前不会发送到Oracle数据库。但是，当提取指示不存在更多数据或非Oracle系统报告错误时，将立即返回任何缓冲数据。

HS_RPC_FETCH_SIZE

属性 描述

默认值

50000

价值范围

1到10000000

调整内部数据缓冲以优化服务器和代理进程之间的数据传输速率。

增加该值可以减少传输给定数量的数据所需的网络往返次数，但是也倾向于增加数据带宽并减少在发出查询和完成查询的所有提取之间所测量的等待时间。然而，增加提取大小会增加查询的初始提取结果的延迟，因为在其他数据可用之前不会传输第一次提取结果。

HS_TIME_ZONE

属性 描述

'[+ | - ] hh：mm'的默认值

派生自NLS_TERRITORY初始化参数

'[+ | - ] hh：mm'的值范围

任何有效的日期时间格式掩码

指定当前SQL会话的默认本地时区位移。格式掩码[+ | - ] hh：mm，用于指示UTC之前或之后的小时和分钟（协调世界时 - 以前的格林威治标准时间）。例如：

HS_TIME_ZONE = [+ | - ] hh：mm

HS_FDS_ENCRYPT_SESSION

属性 描述

默认值

没有

价值范围

{NONE | SSL | NOTRUST_SSL}

句法

HS_FDS_ENCRYPT_SESSION = {无| SSL | NOTRUST_SSL}

指定加密SQL Server会话的方式。有效值为：

  • NONE ：网关和SQL Server之间传输的数据未加密。（默认）。

  • SSL ：使用SSL加密网关和SQL Server之间传输的数据。

  • NOTRUST_SSL ：此选项等同于SSL设置，初始化参数为HS_FDS_VALIDATE_SERVER_CERT = DISABLED

HS_FDS_TRANSACTION_ISOLATION

属性 描述

默认值

READ_COMMITTED

价值范围

{ READ_UNCOMMITTED | READ_COMMITTED | REPEATABLE_READ | SERIALIZABLE }

句法

HS_FDS_ISOLATION_LEVEL = {{READ_UNCOMMITTED | READ_COMMITTED | REPEATABLE_READ | SERIALIZABLE}

HS_FDS_TRANSACTION_ISOLATION指定用于网关在非Oracle数据库上打开的事务的隔离级别。

READ_UNCOMMITTED ， READ_COMMITTED ， REPEATABLE_READ和SERIALIZABLE的隔离级别是SQL标准中定义的四个隔离级别，由ANSI和ISO / IEC采用。有关它们的其他信息，请参阅Oracle数据库概念 。

在指定低于所使用的Oracle事务隔离级别的隔离级别时要小心，因为网关事务将具有与Oracle数据库事务中发生的不同的可预防读取现象。

HS_FDS_VALIDATE_SERVER_CERT

属性 描述

默认值

启用

价值范围

{启用|禁用}

句法

HS_FDS_VALIDATE_SERVER_CERT = {ENABLED | DISABLED}

指定当通过HS_FDS_ENCRYPT_SESSION启用SSL加密时，驱动程序是否验证数据库服务器发送的证书。使用SSL服务器身份验证时，服务器发送的任何证书都必须由受信任的证书颁发机构颁发。有效值为：

  • ENABLED ：网关验证数据库服务器发送的证书。来自服务器的任何证书必须由信任库文件中的受信任证书颁发机构颁发。使用HS_FDS_TRUSTSTORE_FILE和HS_FDS_TRUSTSTORE_PASSWORD初始化参数指定信任库信息。

  • DISABLED ：网关不验证数据库服务器发送的证书。

HS_TRANSACTION_MODEL

属性 描述

默认值

COMMIT_CONFIRM

价值范围

COMMIT_CONFIRM ， READ_ONLY ， SINGLE_SITE ， READ_ONLY_AUTOCOMMIT ， SINGLE_SITE_AUTOCOMMIT

指定事务更新非Oracle数据库时使用的事务模型的类型。

可能的值如下：

  • COMMIT_CONFIRM提供对非Oracle数据库的读写访问，并允许网关成为分布式更新的一部分。要使用commit-confirm模型，必须在非Oracle数据库中创建以下项：

    • 事务日志表。默认表名称为HS_TRANSACTION_LOG 。可以使用HS_FDS_TRANSACTION_LOG参数设置不同的名称。必须将事务日志表授予设置为public的SELECT ， DELETE和INSERT权限。

    • 恢复帐户。帐户名称分配有HS_FDS_RECOVERY_ACCOUNT参数。

    • 恢复帐户密码。密码分配有HS_FDS_RECOVERY_PWD参数。

  • READ_ONLY提供对非Oracle数据库的读访问权限。

  • SINGLE_SITE提供对非Oracle数据库的读写访问。但是，网关无法参与分布式更新。

  • READ_ONLY_AUTOCOMMIT提供对不使用日志记录的非Oracle数据库的只读访问权限。

  • SINGLE_SITE_AUTOCOMMIT提供对非Oracle数据库的读写访问，无需记录。网关无法参与分布式更新。此外，立即提交对非Oracle数据库的任何更新。

HS_FDS_CONNECT_INFO

属性 描述

默认值

没有

价值范围

不适用

HS_FDS_CONNECT_INFO ，描述与非Oracle系统的连接。

默认初始化参数文件已有此参数的条目。网关的HS_FDS_CONNECT_INFO语法如下：

对于UNIX：

HS_FDS_CONNECT_INFO = host_name [[： port_number ] | / [ instance_name ]] [/ database_name ]

其中， host_name是托管SQL Server数据库的计算机的主机名或IP地址， port_number是SQL Server的端口号， instance_name是计算机上运行的SQL Server实例， database_name是SQL Server数据库名称。

可以使用变量port_number或instance_name中的任何一个，但不能同时使用它们。可选地，它们都可以省略。变量database_name始终是可选的。省略特定值时需要斜杠（ / ）。例如，以下所有条目均有效：

HS_FDS_CONNECT_INFO = host_name / instance_name / database_name HS_FDS_CONNECT_INFO = host_name // database_name HS_FDS_CONNECT_INFO = host_name ： port_name // database_name HS_FDS_CONNECT_INFO = host_name / instance_name HS_FDS_CONNECT_INFO = host_name

对于Windows：

HS_FDS_CONNECT_INFO = host_name / [ instance_name ] [/ database_name ]

其中， host_name是托管SQL Server数据库的计算机的主机名或IP地址， instance_name是计算机上运行的SQL Server实例， database_name是SQL Server数据库名称。

instance_name和database_name都是可选的。如果省略instance_name并提供了database_name ，则需要斜杠（ / ）。这可以显示如下：

HS_FDS_CONNECT_INFO = host_name // database_name

此版本支持IPv6格式，因此您可以输入IPv6格式来代替hostname ，但是您需要在IPv6规范中包含方括号。

例如：

HS_FDS_CONNECT_INFO = [2001：将0db8：20C：f1ff：fec6：38af]：端口号/ ...

17.1关于I / O.

每个Oracle数据库都在磁盘上读取或写入数据，从而生成磁盘I / O.许多软件应用程序的性能本质上受到磁盘I / O的限制。占用大部分CPU时间等待I / O活动完成的应用程序被认为是I / O绑定的 。

Oracle数据库的设计使得如果应用程序编写得很好，其性能不应受I / O的限制。如果I / O系统以容量或接近容量运行并且无法在可接受的时间内为I / O请求提供服务，则调整I / O可以增强应用程序的性能。但是，如果应用程序不受I / O限制（例如，当CPU是限制因素时），则调整I / O无法提高性能。

在设计I / O系统时，请考虑以下数据库要求：

  • 存储，例如最小磁盘容量

  • 可用性，例如连续（24 x 7）或营业时间

  • 性能，例如I / O吞吐量和应用程序响应时间

许多I / O设计计划存储和可用性要求，并假设性能不会成为问题。这并非总是如此。最佳地，要配置的磁盘和控制器的数量应由I / O吞吐量和冗余要求确定。然后可以根据存储要求确定磁盘的大小。

在开发I / O设计计划时，请考虑使用Oracle自动存储管理（Oracle ASM） 。Oracle ASM是一个集成的高性能数据库文件系统和磁盘管理器，它基于数据库应该管理存储而不需要管理员来执行存储的原则。

Oracle建议您使用Oracle ASM进行数据库文件存储，而不是使用原始设备或操作系统文件系统。Oracle ASM提供以下主要优势：

  • 条带化

  • 镜像

  • 在线存储重新配置和动态重新平衡

  • 托管文件的创建和删除

17.2.1使用操作系统或硬件条带化布置文件

如果您的操作系统具有LVM软件或基于硬件的条带化，则可以使用这些工具分发I / O.使用LVM或硬件条带化时要做出的决定包括条带深度和条带宽度 。

  • 条带深度是条带的大小，有时称为条带单元。

  • 条带宽度是条带深度和条带集中的驱动器数量的乘积。

明智地选择这些值，以便系统能够维持所需的吞吐量。对于Oracle数据库，合理的条带深度范围为256 KB到1 MB。不同类型的应用程序受益于不同的条带深度。最佳条纹深度和条带宽度取决于以下内容：

  • 请求的I / O大小

  • I / O请求的并发性

  • 物理条纹边界与块大小边界的对齐

  • 拟议系统的可管理性

除了I / O大小之外，并发度还有助于确定理想的条带深度。选择条带宽度和条纹深度时请考虑以下事项：

  • 在低并发（顺序）系统上，确保没有单个I / O访问同一磁盘两次。例如，假设条带宽度为四个磁盘，条带深度为32K。如果Oracle服务器进程发出单个1MB I / O请求（例如，对于全表扫描），则条带中的每个磁盘必须执行八个I / O以返回所请求的数据。为避免这种情况，平均I / O的大小应小于条带宽度乘以条带深度。如果不是这种情况，则Oracle数据库向操作系统发出的单个I / O请求会导致对同一磁盘的多个物理I / O请求。

  • 在高并发（随机）系统上，确保没有单个I / O请求被分解为多个物理I / O调用。如果不这样做，则会增加系统中执行的物理I / O请求的数量，从而严重降低I / O响应时间。

17.2.1.2 I / O请求的并发

在具有高度并发小I / O请求的系统中，例如在传统的OLTP环境中，保持条带深度较大是有益的。使用大于I / O大小的条纹深度称为粗粒度条纹 。在高并发系统中，条带深度可以如下，其中n > 1：

n * DB_BLOCK_SIZE

粗粒度条带化允许阵列中的磁盘为多个I / O请求提供服务。通过这种方式，大量并发I / O请求可由一组条带化磁盘提供服务，并且I / O设置成本最低。粗粒度条带化努力最大化整体I / O吞吐量。如同全表扫描一样，多块读取将在条带深度很大并且可以从一个驱动器进行服务时受益。数据仓库环境中的并行查询也是粗粒度条带化的候选者，因为许多单独的进程都会发出单独的I / O.如果在没有高并发请求的系统中使用粗粒度条带化，则可能导致热点。

在具有少量大I / O请求的系统中，例如在传统DSS环境或低并发OLTP系统中，保持条带深度较小是有益的。这称为细晶粒条纹 。在此类系统中，条带深度如下所示，其中n小于多块读取参数，例如DB_FILE_MULTIBLOCK_READ_COUNT ：

n * DB_BLOCK_ SIZE

细粒度条带化允许单个I / O请求由多个磁盘提供服务。精细纹理条纹力求最大化单个I / O请求或响应时间的性能。

17.2.1.4拟议系统的可管理性

对于LVM，最简单的管理配置是在所有可用磁盘上具有单个条带卷的配置。在这种情况下，条带宽度包含所有可用磁盘。所有数据库文件都驻留在该卷中，从而有效地均匀分配负载。在大多数情况下，这种单卷布局可提供足够的性能。

单卷配置仅在与允许轻松恢复的RAID技术结合使用时才可行，例如RAID 1。否则，丢失单个磁盘意味着同时丢失所有文件，从而执行完整的数据库还原和恢复。

除了性能之外，还存在可管理性问题：系统的设计必须允许简单地添加磁盘，以允许数据库增长。挑战是在保持负载均衡的同时这样做。

例如，初始配置可能涉及在64个磁盘上创建单个条带卷，每个磁盘为16 GB。这是主数据的总磁盘空间为1 TB。系统运行后的某个时间，必须添加额外的80 GB（即五个磁盘）以应对未来的数据库增长。

使此空间可用于数据库的选项包括创建包含五个新磁盘的第二个卷。但是，如果这些新磁盘无法维持放置在其上的文件所需的I / O吞吐量，则可能会产生I / O瓶颈。

另一种选择是增加原始卷的大小。LVM正变得足够复杂，允许动态重新配置条带宽度，从而允许在系统联机时添加磁盘。这开始使得在生产环境中单个条带卷上的所有文件的放置都是可行的。

如果您的LVM不支持动态地将磁盘添加到条带，那么您可能需要选择更小，更易管理的条带宽度。然后，当添加新磁盘时，系统可以按条带宽度增长。

在前面的示例中，八个磁盘可能是更易于管理的条带宽度。这仅在八个磁盘能够每秒维持所需数量的I / O时才可行。因此，当需要额外的磁盘空间时，可以添加另外八个磁盘条带，从而保持I / O在卷上保持平衡。

注意：

条带宽度越小，您需要花时间在卷上分发文件的可能性越大，程序就越接近手动分发I / O.

17.2.2手动分发I / O.

如果您的系统没有LVM或硬件条带化，则必须通过根据每个文件的I / O要求分发文件，在可用磁盘上手动平衡I / O.为了确定文件放置，您应该熟悉数据库文件的I / O要求和I / O系统的功能。如果您不熟悉此数据并且没有要分析的代表性工作负载，则可以先进行猜测，然后在知道使用情况时调整布局。

要手动条带化磁盘，您需要将文件的存储要求与其I / O要求相关联。

  1. 通过检查文件和磁盘的大小来评估数据库磁盘存储要求。

  2. 确定每个文件的预期I / O吞吐量。确定哪些文件具有最高的I / O速率以及哪些文件没有很多I / O.在所有可用磁盘上布置文件，以便均衡I / O速率。

一种流行的手动I / O分发方法建议将常用表与其索引分开。这是不正确的。在事务过程中，首先读取索引，然后读取表。由于这些I / O按顺序发生，因此表和索引可以存储在同一磁盘上而不会发生争用。仅仅因为数据文件包含索引或表数据而仅分离数据文件是不够的。仅当该文件的I / O速率影响数据库性能时，才应决定隔离文件。

17.2.3何时分离文件

无论您使用的是操作系统条带化还是手动I / O分配，如果I / O系统或I / O布局无法支持所需的I / O速率，那么您需要分离具有高I / O速率的文件从剩余的文件。您可以在计划阶段或系统生效后识别此类文件。

隔离文件的决定应仅由I / O速率，可恢复性问题或可管理性问题驱动。（例如，如果您的LVM不支持条带宽度的动态重新配置，那么您可能需要创建较小的条带宽度，以便能够一次添加n个磁盘以创建具有相同配置的新条带。）

在隔离文件之前，请验证瓶颈是否真的是I / O问题。 调查瓶颈产生的数据确定哪些文件具有最高的I / O速率。

以下部分描述了如何隔离以下文件类型：

  • 表，索引和TEMP表空间

  • 重做日志文件

  • 存档的重做日志

17.2.3.1表，索引和TEMP表空间

如果具有高I / O的文件是属于包含表和索引的表空间的数据文件，则通过调整SQL或应用程序代码来确定是否可以减少这些文件的I / O.

如果具有高I / O的文件是属于TEMP表空间的数据文件，则调查是否调整执行磁盘排序的SQL语句以避免此活动，或调整排序。

在调整应用程序以避免不必要的I / O之后，如果I / O布局仍然无法维持所需的吞吐量，那么请考虑隔离高I / O文件。

17.2.3.2重做日志文件

如果高I / O文件是重做日志文件，则考虑从其他文件中拆分重做日志文件。可能的配置可包括以下内容：

  • 将所有重做日志放在一个磁盘上，而不包含任何其他文件。还考虑可用性;同一组的成员应位于不同的物理磁盘和控制器上，以实现可恢复性。

  • 将每个重做日志组放在不存储任何其他文件的单独磁盘上。

  • 使用操作系统条带化工具在多个磁盘上划分重做日志文件。（在这种情况下无法进行手动条带化。）

  • 避免将RAID 5用于重做日志。

重做日志文件由Log Writer（LGWR）进程按顺序写入 。如果同一磁盘上没有并发活动，则可以加快此操作。将单独的磁盘专用于重做日志文件通常可确保LGWR顺利运行，无需进一步调整。如果您的系统支持异步I / O但当前未配置此功能，请测试以查看使用此功能是否有益。与LGWR相关的性能瓶颈很少见。

17.2.4.2将存档日志移动到不同的磁盘

如果归档的重做日志文件与其他文件在同一组磁盘上条带化，则当数据库归档重做日志时，这些磁盘上的任何I / O请求都会受到影响。将归档的重做日志文件移动到单独的磁盘可提供以下好处：

  • 存档可以以非常高的速率执行（使用顺序I / O）。

  • 归档目标磁盘上的响应时间降低不会影响其他任何内容。

存档日志的磁盘数由存档日志生成速率和所需的存档存储量决定。

17.2.4.3将重做日志移动到单独的磁盘

在高度更新的OLTP系统中，重做日志是写入密集型的。将重做日志文件移动到与其他磁盘和归档重做日志文件分开的磁盘具有以下优点：

  • 写入重做日志以尽可能高的速率执行。因此，事务处理性能处于最佳状态。

  • 任何其他I / O都不会影响重做日志的写入。

重做日志的磁盘数量主要取决于重做日志大小，与当前技术磁盘大小相比，这通常较小。通常，具有两个磁盘的配置（可能镜像到四个磁盘以实现容错）就足够了。特别是，通过在两个磁盘上交替使用重做日志文件，将重做日志信息写入一个文件不会影响读取已完成的重做日志以进行存档。

17.2.6.1读取

17.3.2运行I / O校准

使用DBMS_RESOURCE_MANAGER访问Oracle数据库的I / O校准功能。 CALIBRATE_IO程序。此过程向数据库文件发出I / O密集型只读工作负载（由1兆字节的随机I / O组成），以确定最大IOPS（每秒I / O请求数）和MBPS（兆字节I / O）每秒可以由存储子系统维持。

I / O校准分两步进行：

  • 在使用DBMS_RESOURCE_MANAGER进行I / O校准的第一步中。 CALIBRATE_IO过程，该过程向所有数据库实例中的所有数据文件发出随机数据库块大小的读取，默认情况下为8 KB。此步骤在输出参数max_iops中提供数据库可以维持的最大IOPS。值max_iops是OLTP数据库的重要度量标准。输出参数actual_latency提供此工作负载的平均延迟。当您需要特定的目标延迟时，可以使用输入参数max_latency指定目标延迟（指定数据库块大小的IO请求的最大可容忍延迟（以毫秒为单位））。

  • 使用DBMS_RESOURCE_MANAGER进行校准的第二步。 CALIBRATE_IO过程随机发出，1 MB读取来自所有数据库实例的所有数据文件。第二步产生输出参数max_mbps ，它指定数据库可以承受的最大I / O MBPS。此步骤为数据仓库提供了重要指标。

如果用户提供num_physical_disks输入参数，则可以更有效地运行校准，该参数指定数据库存储系统中的近似物理磁盘数。

由于运行I / O工作负载的开销，I / O校准应该仅在数据库空闲时或非高峰时间执行，以最小化I / O工作负载对正常数据库工作负载的影响。

要运行I / O校准并评估Oracle数据库使用的存储子系统的I / O功能，请使用DBMS_RESOURCE_MANAGER 。 CALIBRATE_IO程序：

SET SERVEROUTPUT ON DECLARE lat INTEGER; iops INTEGER; mbps INTEGER; BEGIN  -  DBMS_RESOURCE_MANAGER.CALIBRATE_IO（<DISKS>，<MAX_LATENCY>，iops，mbps，lat）; DBMS_RESOURCE_MANAGER.CALIBRATE_IO（2,10，iops，mbps，lat）; DBMS_OUTPUT.PUT_LINE（'max_iops ='|| iops）; DBMS_OUTPUT.PUT_LINE（'latency ='|| lat）; dbms_output.put_line（'max_mbps ='|| mbps）;结束; /

运行DBMS_RESOURCE_MANAGER 。 CALIBRATE_IO程序，请考虑以下事项：

  • 仅在使用相同存储子系统的数据库上一次运行一次校准。如果同时在使用相同存储子系统的单独数据库中运行校准，则校准将失败。

  • 停顿数据库以最小化实例上的I / O.

  • 对于Oracle Real Application Clusters（Oracle RAC）配置，请确保打开所有实例以跨节点校准存储子系统。

  • 对于Oracle Real Application Clusters（Oracle RAC）数据库，将同时从所有实例生成工作负载。

  • num_physical_disks输入参数是可选的。通过将num_physical_disks参数设置为数据库存储系统中的近似物理磁盘数，可以更快，更准确地进行校准。

  • 在某些情况下，允许异步I / O用于数据文件，但用于提交异步I / O的I / O子系统可能会最大化，并且I / O校准无法继续。在这种情况下，请参阅特定于端口的文档，以获取有关检查系统上异步I / O的最大限制的信息。

  2. 根据测试名称创建Orion输入文件。例如，创建名为mytest.lun的文件。在输入文件列表中要测试的原始卷或文件。每行添加一个卷名。不要在.lun文件中添加注释或其他任何内容。

    例如，Orion输入文件可能包含以下内容：

    / dev / raw / raw1 / dev / raw / raw2 / dev / raw / raw3 / dev / raw / raw4 / dev / raw / raw5 / dev / raw / raw6 / dev / raw / raw7 / dev / raw / raw8

    有关更多信息，请参阅“ 猎户座输入文件 ” 。

  3. 验证是否可以使用命令dd或其他等效文件查看实用程序访问输入文件中指定的所有卷，例如mytest.lun。例如，对于典型的完整性检查，请在Linux系统上尝试以下操作：
    $ dd if = / dev / raw / raw1 of = / dev / null bs = 32k count = 1024

    根据您的平台，您使用的文件查看实用程序及其界面可能不同。

  4. 验证您的平台是否已安装必要的库以执行异步I / O.Orion测试完全依赖于异步I / O.在Linux和Solaris上，库libaio必须位于标准的lib目录中，或者可以通过shell环境的库路径变量访问（通常是LD_LIBRARY_PATH或LIBPATH ，具体取决于您的shell）。Windows具有内置的异步I / O库，因此该问题不适用。
  5. 作为Orion的第一个测试，使用–run和oltp或dss选项。如果数据库主要是OLTP，则使用–run oltp 。如果数据库主要用于数据仓库或分析，则使用–run dss 。

    例如，使用以下命令使用默认输入文件名orion.lun运行类似OLTP的工作负载：

    $ ./orion-run oltp

    Orion生成的I / O负载级别考虑了正在测试（或使用–num_disks参数指定）的磁盘轴数。请记住，主轴的数量可能与输入文件中指定的卷数相关也可能不相关，具体取决于这些卷的映射方式。

  6. “ 猎户座输出文件 ”部分提供了显示猎户座输出文件的示例结果。使用示例文件mytest_summary.txt是验证输入参数和分析输出的良好起点。示例文件mytest_*.csv包含多个I / O性能度量的逗号分隔值。

17.4.6猎户座故障排除

  1. 如果在< testname > .lun文件中指定的一个或多个卷上收到I / O错误：

    • 验证是否可以使用文件复制程序（如dd以与测试，读取或写入相同的模式访问卷。

    • 验证您的主机操作系统版本是否可以执行异步I / O.

    • 在Linux和Solaris上，库libaio必须位于标准的lib目录中，或者可以通过shell环境的库路径变量访问（通常是LD_LIBRARY_PATH或LIBPATH，具体取决于您的shell）。

  2. 如果您在NAS存储上运行：

    • 必须正确安装文件系统才能运行Orion。有关说明，请参阅Oracle安装指南（例如，“适用于Linux x86的数据库安装指南”中的“使用NAS设备”附录B部分）。

    • mytest.lun文件应包含一个或多个现有文件路径。Orion不适用于目录或挂载点。该文件必须足够大才能进行有意义的测试。此文件的大小应表示数据文件的最终预期大小（例如，使用几年后）。

    • 您可能会看到在Linux上通过NFS进行异步I / O（包括2.6内核）的性能很差。

    • 如果您正在进行读取测试并且读取未触及未初始化或先前写入的文件的未触摸块，则某些智能NAS系统可能会通过返回归零块来“伪造”读取。发生这种情况时，您会看到意外的良好性能。

      解决方法是在执行读取测试之前使用dd等工具编写所有块。

  3. 如果在Windows上运行Orion：在原始分区上进行测试需要临时将分区映射到驱动器号并在test.lun文件中指定这些驱动器号。

  4. 如果在x86_64系统上运行Orion 32位Linux / x86二进制文件：请从运行相同Linux版本的32位计算机上复制32位libaio.so文件。

  5. 如果您要测试大量磁盘（ num_disks大于30）：

    • 您应该使用-duration选项（有关详细信息，请参阅可选参数部分）为每个数据点指定一个较长的持续时间（如120秒或更长）。由于Orion试图保持所有主轴在特定负载水平下运行，因此每个数据点都需要一个加速时间，这意味着测试的持续时间更长。

    • 您可能会收到以下错误消息，指示您增加持续时间值：

      指定更长的持续时间值。

      持续时间为锭子数量的2倍似乎是一个很好的经验法则。根据您的磁盘技术，您的平台可能需要更多或更少的时间。

  6. 如果您收到有关Orion使用的库的错误：

    • Linux / Solaris：请参阅I / O错误故障排除。

    • 仅限NT：不要移动/删除分发中包含的Oracle库。这些必须与orion.exe位于同一目录中。

  7. 如果你看到的表现数字“令人难以置信的好”：

    • 您可能在Orion程序和磁盘轴之间的某处有一个大的读取或写入缓存，或读取和写入缓存。通常，存储阵列控制器具有最大的影响。找出此缓存的大小，并使用-cache_size advanced选项将其指定给Orion（有关详细信息，请参阅可选参数部分）。

    • 与沿途的一个或多个缓存相比，卷的总大小可能非常小。尝试关闭缓存。如果共享存储的其他卷在生产环境中显示重要的I / O活动（并最终使用共享缓存的大部分），则需要这样做。

  8. 如果Orion报告了长时间的估计运行时间：

    • 当-num_disks为高时，运行时间会增加。Orion内部使用线性公式来确定使给定数量的磁盘饱和所需的时间。

    • -cache_size参数会影响运行时，即使未指定它也是如此。Orion默认情况下每个数据点缓存加热两分钟。如果已关闭缓存，请指定-cache_size 0 。

    • 如前所述，当指定long -duration值时，运行时间会增加。

188.1 DBMS_XDB_CONFIG概述

PL / SQL包DBMS_XDB_CONFIG是用于DBA的PL / SQL的Oracle XML DB资源应用程序接口（API），用于配置其系统。此API提供了使用PL / SQL访问和管理Oracle XML DB存储库资源的功能和过程。它包括管理资源安全性和Oracle XML DB配置的方法。

Oracle XML DB Repository以XML为模型，为任何数据提供数据库文件系统。存储库将路径名（或URL）映射到XMLType的数据库对象，并为这些对象提供管理功能。

PL / SQL包DBMS_XDB_CONFIG是一个可用于管理以下所有内容的API：

  • Oracle XML DB资源

  • 基于访问控制列表（ACL）的Oracle XML DB安全性。ACL是访问控制条目（ACE）的列表，用于确定哪些主体（用户和角色）可以访问哪些资源。

  • Oracle XML DB配置

188.2 DBMS_XDB_CONFIG安全模型

由XDB拥有， DBMS_XDB_CONFIG包必须由SYS或XDB创建。 EXECUTE特权被授予PUBLIC 。此包中的子程序使用当前用户的权限执行。

132.1 DBMS_REDEFINITION概述

为了实现在线重新定义，使用增量可维护的本地物化视图。这些日志跟踪主表的更改，并在刷新同步期间由实例化视图使用。

132.2 DBMS_REDEFINITION安全模型

DBMS_REDEFINITION包中的子程序使用调用者的权限（具有当前用户的权限）运行。

有两种模式：

  • 在USER模式下，具有CREATE TABLE和CREATE MVIEW权限的用户可以重新定义驻留在其自己的模式中的表。

  • 在FULL模式下，具有ANY特权的用户可以在任何模式中重新定义表。

132.4 DBMS_REDEFINITION操作说明

以下操作说明适用于DBMS_REDEFINITION。

  • CONS_USE_PK和CONS_USE_ROWID是常量，用作START_REDEF_TABLE过程和CAN_REDEF_TABLE过程中 “options_flag”参数的输入。CONS_USE_ROWID用于指示重新定义应使用rowid完成，而CONS_USE_PK意味着重新定义应使用主键或伪主键（这是具有NOT NULL约束的所有组件列的唯一键）完成。

  • CONS_INDEX ， CONS_MVLOG ， CONS_TRIGGER和CONS_CONSTRAINT用于指定在REGISTER_DEPENDENT_OBJECT过程和UNREGISTER_DEPENDENT_OBJECT过程 （参数“ dep_type ”）中注册 （un）的从属对象的类型。

    CONS_TRIGGER ==>依赖对象的类型为TRIGGER

    CONS_CONSTRAINT ==>依赖对象类型的类型为CONSTRAINT

    CONS_MVLOG ==>依赖对象的类型为MATERIALIZED VIEW LOG

  • CONS_ORIG_PARAMS用作COPY_TABLE_DEPENDENTS过程中 “ copy_indexes ”参数的输入 。使用此参数意味着使用与原始索引相同的存储参数将原始表上的索引复制到临时表上。

132.6 DBMS_REDEFINITION示例

以下示例演示了DBMS_REDEFINITION功能。

我们分别创建两个表EMP和EMP_INT作为原始表和临时表：

CREATE TABLE“EMP”（“EMPNO”NUMBER（4,0）PRIMARY KEY，“ENAME”VARCHAR2（10），“JOB”VARCHAR2（9），“MGR”NUMBER（4,0），“HIREDATE”DATE，“ SAL“NUMBER（7,2），”COMM“NUMBER（7,2），”DEPTNO“（2,0））TABLESPACE myts; CREATE TABLE“EMP_INT”（“EMPNO”NUMBER（4,0）PRIMARY KEY，“ENAME”VARCHAR2（10），“JOB”VARCHAR2（9），“MGR”NUMBER（4,0），“HIREDATE”DATE，“ SAL“NUMBER（7,2），”COMM“NUMBER（7,2），”DEPTNO“NUMBER（2,0））TABLESPACE compressed_ts;

定期多步重新定义

DBMS_REDEFINITION.START_REDEF_TABLE（'SCOTT'，'EMP'，'EMP_INT'，ENABLE_ROLLBACK => TRUE）; DBMS_REDEFINITION.FINISH_REDEF_TABLE（'SCOTT'，'EMP'，'EMP_INT'）;

假设在将表EMP从表空间myts到compressed_ts之后，DBA想要评估应用程序的性能2天。可以运行sync_interim_table SYNC_INTERIM_TABLE过程以使两个表保持同步（例如，每小时）。 

DBMS_REDEFINITION.SYNC_INTERIM_TABLE（'SCOTT'，'EMP'，'EMP_INT'）;

案例1 - DBA对性能不满意，因此决定回滚。

DBMS_REDEFINITION.ROLLBACK（'SCOTT'，'EMP'，'EMP_INT'）;

案例2 - DBA对性能感到满意，因此决定不回滚。

DBMS_REDEFINITION.ABORT_ROLLBACK（'SCOTT'，'EMP'，'EMP_INT'）;

这终止了回滚的可能性。

单步重新定义

DBMS_REDEFINITION.REDEF_TABLE（'SCOTT'，'EMP'，'ROW STORE COMPRESS ADVANCED'，enable_rollback => TRUE）;

注意：

使用REDEF_TABLE过程重新定义表时，不支持联机表重新定义回滚。

64.1 DBMS_EPG概述

DBMS_EPG包是PL / SQL用户开发和部署PL / SQL Web应用程序的平台。嵌入式PL / SQL网关是在Oracle数据库中的XML数据库HTTP服务器中运行的网关的嵌入式版本。它在数据库中提供了mod_plsql的核心功能，但不需要Apache支持的Oracle HTTP服务器。

为了通过HTTP从浏览器访问PL / SQL应用程序，必须创建数据库访问描述符（DAD）并将其映射到虚拟路径。DAD是用于数据库访问的一组配置值，虚拟路径映射使应用程序可以在XML DB HTTP Server的虚拟路径下访问。DAD在XML DB HTTP Server中表示为servlet。

64.2 DBMS_EPG安全模型

需要XDBADMIN角色来调用配置界面。它可以由数据库用户“ XDB ”调用。

任何用户都可以调用授权接口。
