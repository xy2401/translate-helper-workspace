使用DBMS_FILE_TRANSFER包在平台之间转换数据

您可以使用GET_FILE或PUT_FILE的议事DBMS_FILE_TRANSFER包中的数据文件传输过程中的平台之间的数据转换。

当您使用这些过程之一在源平台和目标平台之间移动数据文件时，每个数据文件中的每个块都将转换为目标平台的字节顺序。

本节使用一个示例来描述如何使用DBMS_FILE_TRANSFER包将数据文件转换为其他平台。该示例做出以下假设：

  • GET_FILE过程将传输数据文件。

  • mytable.342.123456789数据文件正在转移到另一个平台。

  • 源平台的字节顺序与目标平台的字节顺序不同。

  • 源数据库的全局名称是dbsa.example.com 。

  • 源数据库和目标数据库都使用Oracle自动存储管理（Oracle ASM）。

注意：

您还可以使用DBMS_FILE_TRANSFER包在具有相同字节顺序的平台之间传输数据文件。

完成以下步骤以通过使用GET_FILE过程传输数据文件来转换数据文件：

  1. 使用SQL * Plus以可以创建目录对象的管理用户身份连接到源数据库。
  2. 创建目录对象以存储要传输到目标数据库的数据文件。

    例如，要为+data/dbsa/datafile目录创建名为sales_dir_source的目录对象，请执行以下SQL语句：

    创建或替换目录sales_dir_source AS'+ data / dbsa / datafile';

    创建目录对象时，指定的文件系统目录必须存在。

  3. 使用SQL * Plus作为管理用户连接到目标数据库，管理用户可以创建数据库链接，创建目录对象，并在DBMS_FILE_TRANSFER包中运行这些过程。
  4. 创建从目标数据库到源数据库的数据库链接。

    源数据库中的已连接用户必须对您在步骤2中创建的目录对象具有读取权限。

  5. 创建目录对象以存储要从源数据库传输的数据文件。

    将在DBMS_FILE_TRANSFER程序包中运行该过程的本地数据库用户必须对目录对象具有写权限。

    例如，要为+data/dbsb/datafile目录创建名为sales_dir_target的目录对象，请运行以下SQL语句：

    创建或替换目录sales_dir_target AS'+ data / dbsb / datafile';
  6. 运行DBMS_FILE_TRANSFER包中的GET_FILE过程以传输数据文件。

    例如，运行以下过程以使用您在步骤4中创建的数据库链接将mytable.342.123456789数据文件从源数据库传输到目标数据库：

    BEGIN DBMS_FILE_TRANSFER.GET_FILE（source_directory_object =>'sales_dir_source'，source_file_name =>'mytable.342.123456789'，source_database =>'dbsa.example.com'，destination_directory_object =>'sales_dir_target'，destination_file_name =>'mytable'）;结束; /

注意：

在此示例中，目标数据文件名是mytable 。Oracle ASM不允许在GET_FILE过程的destination_file_name参数中使用完全限定的文件名表单。

32.6成本敏感决策

成本是用户指定的数字，偏向分类。该算法使用正数来惩罚比较便宜的结果更昂贵的结果。数字越大表示成本越高。

该算法使用负数来支持比不太有利的结果更有益的结果。较低的负数表示较高的收益。

所有分类算法都可以使用评分成本。您可以在成本矩阵表中指定成本，也可以在评分时指定内联成本。如果您指定内联成本且模型也具有关联的成本矩阵，则仅使用内联成本。PREDICTION ， PREDICTION_SET和PREDICTION_COST函数支持成本。

只有决策树算法可以使用成本来偏向模型构建。如果要创建包含成本的决策树模型，请创建成本矩阵表，并在模型的CLAS_COST_TABLE_NAME设置中提供其名称。如果在构建模型时指定成本，则在评分时使用用于创建模型的成本矩阵。如果要使用不同的成本矩阵表进行评分，请先删除现有的成本矩阵表，然后添加新的成本矩阵表。

DROP TYPE

         

         

            

               

                  
目的

                  
对象类型使用PL / SQL定义。有关创建，更改和删除对象类型的完整信息，请参阅Oracle Database PL / SQL语言参考 。
                  

                  
使用DROP TYPE语句删除对象类型，varray或嵌套表类型的规范和正文。
                  

               

               
               

                  
先决条件

                  
对象类型，varray或嵌套表类型必须位于您自己的架构中，或者您必须具有DROP ANY TYPE系统特权。
                  

               

               
               

                  
句法

               

               
               

                  
drop_type :: =

                  

插图说明drop_type.eps

                  
               

               
               

                  
语义

               

               
               

                  
模式

                  
指定包含类型的架构。如果省略schema ，则Oracle数据库假定该类型位于您自己的模式中。
                  

               

               
               

                  
TYPE_NAME

                  
指定要删除的对象，varray或嵌套表类型的名称。您只能删除没有类型或表依赖项的类型。

                  
如果type_name是超类型，则除非您还指定FORCE否则此语句将失败。如果指定FORCE ，则数据库将根据此超类型使所有子类型无效。
                  

                  
如果type_name是统计类型，则除非您还指定FORCE否则此语句将失败。如果指定FORCE ，则数据库首先解除与type_name关联的所有对象的关联，然后删除type_name 。
                  

                  

                     
也可以看看：

                     
ASSOCIATE统计和撇清统计的统计类型的详细信息

                  

                  
如果type_name是已与统计信息类型关联的对象类型，则数据库首先尝试取消将type_name与统计信息类型取消关联，然后删除type_name 。但是，如果使用统计信息类型收集了统计信息，则数据库将无法将type_name与统计信息类型取消关联，并且此语句将失败。
                  

                  
如果type_name是indextype的实现类型，则indextype将标记为INVALID 。

                  
如果type_name上定义了公共同义词，则数据库也将删除同义词。
                  

                  
除非指定FORCE ，否则只能删除作为独立模式对象且没有依赖关系的对象类型，嵌套表或varray类型。这是默认行为。
                  

                  

                     
也可以看看：

                     
CREATE INDEXTYPE

                  

               

               
               

                  
力

                  
指定FORCE以删除类型，即使它具有依赖的数据库对象。Oracle数据库标记UNUSED所有依赖于要删除的类型的列，并且这些列变得不可访问。
                  

                  

                     
注意：

                     
Oracle建议您不要指定FORCE来删除具有依赖项的对象类型。此操作无法恢复，可能导致从属表或列中的数据无法访问。
                     

                  

               

               
               

                  
VALIDATE

                  
如果在删除类型时指定VALIDATE ，则Oracle数据库会在其任何超类型的可替换列中检查此类型的存储实例。如果未找到此类实例，则数据库将完成放置操作。
                  

                  
该子句仅对子类型有意义。Oracle建议使用此选项安全地删除没有任何显式类型或表依赖性的子类型。

4.5.1 UCP中收获连接的概述

               

                  
此功能有助于确保池中始终可以使用一定数量的连接，并有助于最大限度地提高性能。如果应用程序缓存连接句柄，则此功能特别有用。通常出于性能原因执行缓存，因为它最小化了连接参与事务所需的状态的重新初始化。

                  
例如，从池中借用连接，使用必要的会话状态进行初始化，然后保存在上下文对象中。以这种方式保持连接可能会导致连接池耗尽可用连接。如果适用，连接收集功能将回收借用的连接，并允许重用连接。

                  
使用HarvestableConnection接口控制连接收集，并使用两个池属性配置或启用连接收集：连接收集触发计数和连接收获最大计数。实现连接收集功能时，接口和属性一起使用。
                  

               

            
4.5.2设置与可收集的连接

               

                  

                     
 setConnectionHarvestable(boolean)方法oracle.ucp.jdbc.HarvestableConnection接口控制是否将收获连接。当启用连接收集时，此方法用作锁定机制。例如，当在事务中使用连接时，该方法在连接上设置为false ，并且不得收获。事务完成后，该方法在连接上设置为true ，并且可以在需要时获取连接。
                     

                     

                        
注意：

                        
默认情况下，启用连接收获功能时，所有连接都是可收集的。如果启用此功能，则应始终使用setConnectionHarvestable方法显式控制连接是否可收集。
                        

                     

                     
以下示例演示如何使用setConnectionHarvestable方法指示在连接收集功能尝试收集连接时连接不可收集：
Connection conn = pds.getConnection（）; （（HarvestableConnection）conn）。setConnectionHarvestable（false）;

                  
               

            

            
4.5.3设置收获触发计数

               

                  

                     
连接收集触发计数指定触发连接收集的可用连接阈值。例如，如果连接收获触发计数设置为10，则当池中的可用连接数降至10时，将触发连接收集。
                     

                     
值为Integer.MAX_VALUE （默认为2147483647）表示禁用了连接收集。默认值为Integer.MAX_VALUE 。

                     
以下示例演示了通过配置连接收集触发计数来启用连接收集。
pds.setConnectionHarvestTriggerCount（2）;

                  
               

            
4.5.4设置收获最大计数

               

                  

                     
连接收获最大计数属性指定在达到收获触发计数后应将多少借用连接返回到池。实际收获的连接数可以是从0到连接收获最大计数值的任何值。最近最少使用的连接首先被收获，这使得非常活跃的用户会话能够保持最大的连接。
                     

                     
收获最大计数值的范围可以从0到最大连接属性值。默认值为1 。如果指定了超出范围的值，则抛出SQLException。
                     

                     
以下示例演示如何配置连接收获最大计数。
pds.setConnectionHarvestMaxCount（5）;

                        
注意：

                        
    
                           
  • 
                              
    如果同时启用了连接收集和放弃的连接超时功能，则超时处理不会回收指定为不可维护的连接。
    
                           
    • 
                           
  • 
                              
    如果同时启用了连接收集和生存时间连接超时功能，则超时处理将回收指定为不可维护的连接。

1.111 ENABLE_DDL_LOGGING

         

         

            

               
ENABLE_DDL_LOGGING启用或禁用将数据定义语言（DDL）语句子集写入DDL日志。
               

               

               
               

                  

                     
                        

                           属性
                           描述

DDL日志是一个与警报日志具有相同格式和基本行为的文件，但它只包含数据库发出的DDL语句。仅为RDBMS组件创建DDL日志，并且仅当ENABLE_DDL_LOGGING初始化参数设置为true时才创建。当此参数设置为false ，DDL语句不包含在任何日志中。

    当REMOTE_LOGIN_PASSWORDFILE设置为shared ：
    
                        
      
                           
    • 
                              
      SYS密码无法更改。如果您尝试，密码更改操作将失败，并显示“ORA-28046：不允许更改SYS的密码”。
      
                           
      • 
                           
    • 
                              
      具有SYS*管理员权限（ SYSDBA ， SYSOPER ， SYSASM ， SYSBACKUP ， SYSDG ， SYSKM ）的任何用户的密码都无法更改。如果您尝试，密码更改操作将失败，并显示“ORA-01999：密码文件无法在共享模式下更新”。
      
                           
      • 
                           
    • 
                              
      不允许向单个用户授予SYS*管理员权限（ SYSDBA ， SYSOPER ， SYSASM ， SYSBACKUP ， SYSDG ， SYSKM ）。例如， grant sysdba to scott失败，并显示“ORA-01999：密码文件无法在SHARED模式下更新”。同样，撤消SYS*管理员权限也会失败。
                              
      
                           
      • 
                           
    • 
                              
      如果密码文件不存在，则行为与将REMOTE_LOGIN_PASSWORDFILE设置为none相同。
                              
      
                           
      • 
                        
    
                     
    • 
                     
  • 
                        
    独家
    
                        
    密码文件只能由一个数据库使用。密码文件可以包含SYS和非SYS用户。
                        
    
                        
    当REMOTE_LOGIN_PASSWORDFILE设置为exclusive ，如果密码文件不存在，则行为与将REMOTE_LOGIN_PASSWORDFILE设置为none相同。
                        
    
                     
    • 
                     
  • 
                        
    没有
    
                        
    Oracle忽略任何密码文件。因此，必须由操作系统对特权用户进行身份验证。
    
                     
    • 
                  

                  

                     
注意：

                     
如果更改REMOTE_LOGIN_PASSWORDFILE来exclusive或shared从none ，则确保密码文件与字典密码同步。

processes = 400 open_cursors = 250 locallistener =

父主题： 升级Oracle数据库

                  

               

            

            
使用并行升级实用程序并行升级数据字典

               

                  
现在，在Oracle数据库中升级数据字典与并行升级实用程序并行完成，从而加快了升级过程。

                  
而不是一个SQL进程加载数据字典，您现在可以有多个进程，具体取决于您的服务器的CPU容量。并行升级工具（ catctl.pl ，您可以使用命令行运行dbupgrade在Linux和Unix和dbupgrade.com在Windows上）将数据加载到字典中尽可能快，并按照可在做任何工作normal模式而不是upgrade模式。使用并行升级实用程序升级数据库可减少整体停机时间。
                  

               

               

                  

                     

                        
父主题： 数据字典升级和升级状态显示的改进

                     

                  

               

               
            
更改为升级状态设置

               

                  
从Oracle Database 12 c第 1版及更高版本开始，UPGRADED状态的含义已更改。
                  

                  
升级完成后，如果升级过程中没有错误，则Oracle Server，Oracle Multimedia（ORDIM），Spatial（SDO）和XDB的状态均设置为UPGRADED 。此行为与12.1之前的版本不同。在早期版本中，升级后，Oracle Server，Oracle Multimedia（ORDIM），Spatial（SDO）和XDB的状态设置为VALID 。在12.1及更高版本中， UPGRADED状态表示数据字典已加载且没有任何错误。如果将任何错误记录到registry$error ，则升级的状态将设置为INVALID 。

                  
升级完成后如果没有错误，则数据库的状态将设置为UPGRADED 。此行为与早期版本不同，早期版本在升级后将数据库的状态设置为VALID 。UPGRADED状态表示已加载数据字典而没有任何错误。如果将任何错误记录到registry$error ，则升级的状态将设置为INVALID 。

               

               

                  

                     

                        
父主题： 数据字典升级和升级状态显示的改进

                     

                  

               

               
            
运行重新编译utlrp.sql脚本后更改为状态

               

                  
为确保数据库用户在升级后获得预期的性能，Oracle建议您运行utlrp.sql脚本。
                  

                  
在升级数据库之后的早期版本的Oracle数据库中， VALID意味着数据字典中的所有对象都已重新编译并可以使用。在Oracle Database 12c及更高版本中，获取VALID状态具有不同的含义。在正常模式下运行utlrp.sql脚本会重新编译数据字典对象，并将数据字典从UPGRADED状态移动到VALID状态。
                  

                  
除非在升级之后和首次启动升级的数据库之前运行了utlrp.sql否则Oracle无法保证数据库升级有效 。执行升级后未运行utlrp.sql强制数据字典对象在首次访问时进行编译。访问数据库的第一个用户在初始访问期间访问的每个无效对象的性能成本都会受到影响。重新编译这些无效对象后，将返回正常处理。运行utlrp.sql可确保数据库在升级后可以使用。发现的任何错误都会立即报告。您可以在升级的数据库的用户遇到问题之前修复它们。

1.249 PARALLEL_MAX_SERVERS

         

         

            

               
PARALLEL_MAX_SERVERS指定实例的最大并行执行进程数和并行恢复进程数。 随着需求的增加，Oracle数据库会将实例启动时创建的数量增加到此值。

可修改

注意：

                     
此参数适用于独占模式以及Real Application Clusters环境中的并行执行。

                  

                  
在实例上以默认并行度运行的并发并行用户数取决于实例的内存初始化参数设置。例如，如果设置了MEMORY_TARGET或SGA_TARGET初始化参数，那么concurrent_parallel_users的数量= 4 。如果MEMORY_TARGET设置MEMORY_TARGET或SGA_TARGET ，则检查PGA_AGGREGATE_TARGET 。如果为PGA_AGGREGATE_TARGET设置了值，则concurrent_parallel_users = 2 。如果没有为PGA_AGGREGATE_TARGET设置值，则concurrent_parallel_users = 1 。
                  

                  
如果将此参数设置得太低，则在查询处理期间，某些查询可能没有可用的并行执行过程。如果将其设置得太高，则在高峰期可能会出现内存资源短缺，从而降低性能。

                  

                     
注意：

DROP RESTORE POINT

         

         

            

               

                  
目的

                  
使用DROP RESTORE POINT语句从数据库中删除正常还原点或保证还原点。
                  

                  
    
                     
  • 
                        
    您无需删除正常的还原点。必要时，数据库会自动删除最旧的还原点，如restore_point的语义中所述。但是，如果要重用该名称，则可以删除正常的还原点。
                        
    
                     
    • 
                     
  • 
                        
    保证还原点不会自动删除。因此，如果要从数据库中删除保证还原点，则必须使用此语句显式执行此操作。
    
                     
    • 
                  

                  

                     
也可以看看：

                     
有关创建和使用还原点的信息，请创建RESTORE POINT ， FLASHBACK DATABASE和FLASHBACK TABLE

                  

               

               
               

                  
先决条件

                  
要删除正常还原点，必须具有SELECT ANY DICTIONARY ， FLASHBACK ANY TABLE ， SYSBACKUP或SYSDG系统特权。
                  

                  
要删除保证还原点，你必须符合下列条件之一 ：

                  
    
                     
  • 
                        
    您必须连接AS SYSDBA ，或AS SYSBACKUP或AS SYSDG 。
    
                     
    • 
                     
  • 
                        
    您必须已被授予SYSDBA权限，并且正在使用多租户数据库。
                        
    
                     
    • 
                     
  • 
                        
    您必须以SYS用户身份运行，并且正在使用多租户数据库。
                        
    
                     
    • 
                  

                  
连接到多租户容器数据库（CDB）时，可以删除还原点，如下所示：

                  
    
                     
  • 
                        
    要删除正常的CDB还原点，当前容器必须是根，并且您必须具有SELECT ANY DICTIONARY或FLASHBACK ANY TABLE系统特权，可以在根目录中SYSBACKUP或在本地授予，也可以具有SYSDBA ， SYSBACKUP或SYSDG系统特权一般授予。
                        
    
                     
    • 
                     
  • 
                        
    要删除保证的CDB还原点，当前容器必须是根，并且必须具有通常授予的SYSDBA ， SYSBACKUP或SYSDG系统特权。
                        
    
                     
    • 
                     
  • 
                        
    要删除正常的PDB还原点，当前容器必须是根，并且您必须具有通常授予的SELECT ANY DICTIONARY ， FLASHBACK ANY TABLE ， SYSDBA ， SYSBACKUP或SYSDG系统特权，或者当前容器必须是其中的PDB如果要创建还原点，则必须具有SELECT ANY DICTIONARY ， FLASHBACK ANY TABLE ， SYSDBA ， SYSBACKUP或SYSDG系统特权， SYSDG特权通常授予或在该PDB中本地授予。
    
                     
    • 
                     
  • 
                        
    要删除保证的PDB还原点，当前容器必须是根，并且您必须具有通常授予的SYSDBA ， SYSBACKUP或SYSDG系统特权，或者当前容器必须是要在其中创建还原点的PDB和您必须具有SYSDBA ， SYSBACKUP或SYSDG系统特权， SYSDG特权通常授予或在该PDB中本地授予。
    
                     
    •

语义

               

               
               

                  
restore_point

                  
指定要删除的还原点的名称。

               

               
               

                  
对于可插入的数据库

                  
此子句使您可以在连接到根时删除PDB还原点。对于pdb_name ，请指定包含要删除的还原点的PDB的名称。
                  

                  
如果您已连接到要从中删除还原点的PDB，则无需指定此子句。但是，如果指定此子句，则必须指定要连接的PDB的名称。

8.1.3数据挖掘的数据库调优注意事项

数据库内存管理可以对数据挖掘产生重大影响。正确的尺寸程序全局区域（PGA）内存对于模型构建，复杂查询和批处理评分非常重要。从数据挖掘的角度来看， 系统全局区域（SGA）通常不太重要。但是，必须调整SGA的大小以适应实时评分，这会将模型加载到SGA中的共享游标中。在大多数情况下，您可以配置数据库以自动管理内存。为此，请在调整参数MEMORY_TARGET指定总的最大内存大小。通过自动内存管理，Oracle Database可根据需要在SGA和实例PGA之间动态交换内存，以满足处理需求。
                  

                  
大多数数据挖掘算法都可以利用在数据库中启用并行执行。INIT.ORA参数控制并行执行的行为。
