10.1.1收集统计数据

默认情况下，Oracle Text使用基于开销的优化程序（CBO）来确定查询的最佳执行计划。

要使优化程序能够更好地估算成本，请计算要查询的表的统计信息：

ANALYZE TABLE <table_name> COMPUTE STATISTICS;

或者，估算表格样本的统计数据：

ANALYZE TABLE <table_name> ESTIMATE STATISTICS 1000 ROWS;

要么

ANALYZE TABLE <table_name> ESTIMATE STATISTICS 50 PERCENT;

您还可以与DBMS_STATS.GATHER_TABLE_STATS过程并行收集统计信息：

开始DBMS_STATS.GATHER_TABLE_STATS（'owner'，'table_name'，estimate_percent => 50，block_sample => TRUE，degree => 4）;结束 ;

这些语句收集与table_name,关联的所有对象的统计信息table_name,包括表列和与表关联的任何索引（b-tree，bitmap或Text域）。

要重新收集表的统计信息，请根据需要多次输入ANALYZE语句或使用DBMS_STATS包。

通过收集有关Text域索引的统计信息，Oracle数据库中的CBO可以执行以下任务：

  • 估计CONTAINS谓词的选择性

  • 估计使用Oracle Text索引的I / O和CPU成本（即使用域索引处理CONTAINS谓词的成本）

  • 估计每次CONTAINS调用的I / O和CPU成本

了解CONTAINS谓词的选择性对于包含多个谓词的查询很有用，例如在结构化查询中。通过这种方式，CBO可以更好地决定是使用域索引来评估CONTAINS还是将CONTAINS谓词应用为后置过滤器。

假设如下：

  • 作者列的类型为VARCHAR2 ，年份列的类型为NUMBER.

  • author列上的b树索引。

  • 结构化author谓词对于CONTAINS谓词和year谓词具有高度选择性。也就是说，结构化谓词（author ='King'）相对于year和CONTAINS谓词单独返回的行数要少得多，比如分别返回5行和1000行和1500行。

注意：

10.1.3重新收集统计数据

同步索引后，您可以重新收集单个索引的统计信息以更新成本估算。

如果在同步之前重新分析了基表，则在同步后分析索引就足够了，而无需重新分析整个表。

要重新收集统计信息，请输入以下语句之一：

ANALYZE INDEX <index_name> COMPUTE STATISTICS;
ANALYZE INDEX <index_name>估算统计样本50％;

10.1.4删除统计信息

删除与表关联的统计信息：

ANALYZE TABLE <table_name> DELETE STATISTICS;

删除一个索引的统计信息：

ANALYZE INDEX <index_name> DELETE STATISTICS;

10.2.2使用FIRDER_ROWS（n）提示修改ORDER BY查询的响应时间

当您需要ORDER BY查询的第一行时，Oracle建议您使用基于成本的FIRST_ROWS(n)提示。

注意：

由于FIRST_ROWS(n)提示是基于成本的，因此Oracle建议您在使用此提示之前收集有关表的统计信息。

如果您希望在最短的时间内获得前n行，则使用FIRST_ROWS(n)提示。例如，考虑以下PL / SQL块，它使用游标检索查询的前10个命中，并使用FIRST_ROWS(n)提示来优化响应时间：

声明光标c是从articles_tab中选择/ * + FIRST_ROWS（10）* / article_id，其中包含（article，'Omophagia'）> 0 order by pub_date desc;开始
for c in c loop insert into t_s values（i.pk，i.col）;当c％rowcount> 11时退出;结束循环;
结束; /

c cursor是一个SELECT语句， 它以排序顺序返回包含单词omophagia的rowid。代码循环通过游标以提取前10行。这些行存储在临时t_s表中。

使用FIRST_ROWS(n)提示，当返回前N个命中的成本较低时，优化器指示Oracle Text索引以分数排序顺序返回rowid。

在没有提示的情况下，Oracle数据库会在Oracle Text索引返回满足CONTAINS谓词的未排序顺序的所有行之后对rowid进行排序。检索整个结果集需要时间。

因为此查询中只需要前10个匹配，所以使用提示可以获得更好的性能。

注意：

当您只需要查询的前几个匹配时，请使用FIRST_ROWS(n)提示。当您需要整个结果集时，请不要使用此提示，因为它可能会导致性能不佳。

  • 分区键列上的范围搜索：此查询将搜索限制为也是分区键的列上的特定值范围。例如，考虑对日期范围的查询：

    SELECT storyid FROM storytab WHERE CONTAINS（story，'oliver'）> 0和pub_date BETWEEN'1-OCT-93'AND'1-NOV-93';

  • ORDER BY分区键列：此查询仅需要前n命中， ORDER BY子句命名分区键。考虑price列上的ORDER BY查询以获取前20个点击：

    SELECT * FROM（
    SELECT itemid FROM item_tab WHERE CONTAINS（item_desc，'cd player'）> 0 ORDER BY price）WHERE ROWNUM <20;

    在此示例中，对于按价格分区的表，查询可能只需要从第一个分区获取命中以满足查询。

10.2.7使用CONTEXT索引的BIG_IO选项改进响应时间

Oracle Text提供了 BIG_IO选项，用于提高广泛使用IO操作的CONTEXT索引的查询性能。查询性能改进主要用于存储在旋转磁盘上的数据，而不是存储在固态磁盘上的数据。

启用BIG_IO选项时， CONTEXT索引会为每个唯一标记文本创建一个具有一个大对象（LOB）数据类型的标记类型对。具有相同文本但不同标记类型的标记对应于$I表中的不同行。

启用了BIG_IO选项的索引应该将令牌LOB创建为SecureFile LOB，以便将数据按顺序存储在多个块中。此方法可以改善查询的响应时间，因为查询现在可以执行更长的顺序读取而不是许多短读取。

注意：

如果使用SecureFiles，则必须将COMPATIBLE设置为11.0或更高。此外，您必须在自动段空间管理（ASSM）表空间上创建LOB。将现有Oracle Text索引迁移到SecureFiles时，请使用ASSM表空间。要帮助将现有索引迁移到SecureFiles，可以扩展ALTER INDEX REBUILD以提供仅影响$ I表的存储首选项。

要使用BIG_IO索引选项创建CONTEXT索引，首先通过将其BIG_IO存储属性的值设置为YES,来创建基本存储首选项YES,然后在创建CONTEXT索引时指定此存储首选项。

以下示例创建基本mystore存储首选项，并将其BIG_IO存储属性的值设置为YES:

exec ctx_ddl.create_preference（'mystore'，'BASIC_STORAGE'）; exec ctx_ddl.set_attribute（'mystore'，'BIG_IO'，'YES'）;

要禁用BIG_IO选项，请通过将其BIG_IO存储属性的值设置为NO,来更新现有存储首选项（ mystore ） NO,然后重建索引。 

exec ctx_ddl.set_attribute（'mystore'，'BIG_IO'，'NO'）; alter index idx rebuild（'replace storage mystore'）;

警告：

请勿使用替换元数据操作来禁用BIG_IO索引选项。它可以使索引处于不一致状态。

要在不重建索引的情况下为分区索引启用BIG_IO选项，请通过将其BIG_IO存储属性的值设置为YES,来修改基本存储首选项，使用ctx_ddl.replace_index_metadata,替换全局索引元数据ctx_ddl.replace_index_metadata,然后在REBUILD模式下为每个ctx_ddl.replace_index_metadata,调用optimize_index分区索引表的分区。

以下示例为idx分区索引启用BIG_IO选项：

exec ctx_ddl.set_attribute（'mystore'，'BIG_IO'，'YES'）; exec ctx_ddl.replace_index_metadata（'idx'，'replace storage mystore'）; exec ctx_ddl.optimize_index（'idx'，'rebuild'，part_name =>'part1'）;

注意：

如果过程仅在启用BIG_IO选项的情况下修改现有索引表，则不会导致重新索引数据。

注意：

由于BIG_IO索引选项执行较长的顺序读取，因此使用BIG_IO索引选项的查询需要大型程序全局区域（PGA）内存。

10.2.8使用CONTEXT索引的SEPARATE_OFFSETS选项改进响应时间

Oracle Text提供了 SEPARATE_OFFSETS选项用于提高使用IO操作的CONTEXT索引的查询性能，其查询主要是单字或布尔查询。

SEPARATE_OFFSETS选项为TEXT.类型的标记创建不同的过帐列表结构TEXT. SEPARATE_OFFSETS选项不是在发布列表中散布docids，频率，信息长度（偏移信息的长度）和偏移，而是在发布列表的开头存储所有docids和频率，以及所有信息长度和偏移量。发布列表的结尾。发布开头的标题包含有关docids和偏移之间的边界点的信息。由于docids和offsets的分离减少了查询读取数据的时间，因此可以缩短查询响应时间。

当您将其与BIG_IO选项一起使用以及具有非常长的发布的令牌时，最好实现SEPARATE_OFFSETS选项的性能。

要使用SEPARATE_OFFSETS选项创建CONTEXT索引，请首先通过将其SEPARATE_OFFSETS存储属性的值设置为T.来创建基本存储首选项T.接下来，在创建CONTEXT索引时指定此存储首选项。

以下示例创建基本mystore存储首选项，并将其SEPARATE_OFFSETS存储属性的值设置为T:

exec ctx_ddl.create_preference（'mystore'，'BASIC_STORAGE'）; exec ctx_ddl.set_attribute（'mystore'，'SEPARATE_OFFSETS'，'T'）;

要禁用SEPARATE_OFFSETS选项，请通过将其SEPARATE_OFFSETS存储属性的值设置为F,来更新现有存储首选项（ mystore ） F,然后重建索引。 

exec ctx_ddl.set_attribute（'mystore'，'SEPARATE_OFFSETS'，'F'）; alter index idx rebuild（'replace storage mystore'）;

警告：

不要使用替换元数据操作来禁用SEPARATE_OFFSETS索引选项，因为它可能使索引处于不一致状态。

要为分区索引启用SEPARATE_OFFSETS选项而不重建索引，请通过将其SEPARATE_OFFSETS存储属性的值设置为T,来修改基本存储首选项，使用ctx_ddl.replace_index_metadata,替换全局索引元数据ctx_ddl.replace_index_metadata,然后在REBUILD模式下调用optimize_index分区索引表中的每个分区。

以下示例为分区的idx索引启用SEPARATE_OFFSETS选项：

exec ctx_ddl.set_attribute（'mystore'，'SEPARATE_OFFSETS'，'T'）; exec ctx_ddl.replace_index_metadata（'idx'，'replace storage mystore'）; exec ctx_ddl.optimize_index（'idx'，'rebuild'，part_name =>'part1'）;

注意：

如果过程仅在启用了SEPARATE_OFFSETS选项的情况下修改现有索引表，则不会重新索引数据。

10.2.9使用STEX_ITAB，STAGE_ITAB_MAX_ROWS和STAGE_ITAB_PARALLEL语境索引选项改进响应时间

Oracle Text提供了 STAGE_ITAB选项，用于提高CONTEXT索引的查询性能，该索引广泛使用插入，更新和删除操作进行近实时索引。

如果不使用STAGE_ITAB索引选项，则在将新文档添加到CONTEXT索引时，将调用SYNC_INDEX以使文档可搜索。这个调用在创建新行$I表，并增加了碎裂$I表。结果是查询性能下降。

启用STAGE_ITAB索引选项时，会发生以下情况：

  • 有关新文档的信息存储在$G staging表中，而不是$I表中。此存储可确保$I表不会碎片化，并且不会降低查询性能。

  • $H b-tree索引在$G表上创建。$G表和$H b-tree索引等价于$I表和$X b-tree索引。

使用MERGE优化模式优化$G表中的行并将它们移动到$I表。

注意：

$G表存储在KEEP池中。要提高查询性能，应使用新的stage_itab_max_rows选项分配足够的KEEP池内存并维护足够大的$G表大小。

要使用STAGE_ITAB索引选项创建CONTEXT索引，请首先通过将其STAGE_ITAB存储属性的值设置为YES.来创建基本存储首选项YES.接下来，在创建CONTEXT索引时指定此存储首选项。

以下示例创建基本mystore存储首选项，并将其STAGE_ITAB存储属性的值设置为YES:

exec ctx_ddl.create_preference（'mystore'，'BASIC_STORAGE'）; exec ctx_ddl.set_attribute（'mystore'，'STAGE_ITAB'，'YES'）;

您还可以使用ALTER INDEX语句的rebuild选项为现有的非分区CONTEXT索引启用STAGE_ITAB索引选项。 

alter index IDX rebuild parameters（'replace storage mystore'）;

要为未分区的CONTEXT索引禁用STAGE_ITAB选项，请通过将其STAGE_ITAB存储属性的值设置为NO,来更新现有存储首选项（ mystore ） NO,然后重建索引。 

exec ctx_ddl.set_attribute（'mystore'，'STAGE_ITAB'，'NO'）; alter index idx rebuild（'replace storage mystore'）;

此操作使用MERGE优化模式运行优化过程，然后删除$G表。

ALTER INDEX语句的rebuild选项不适用于分区的CONTEXT索引，用于启用和禁用STAGE_ITAB选项。

以下示例为分区的CONTEXT idx索引启用STAGE_ITAB选项：

alter index idx参数（'add stage_itab'）;

以下示例禁用分区的CONTEXT idx索引的STAGE_ITAB选项：

alter index idx parameters（'remove stage_itab'）;

从Oracle Database 12 c第2版（12.2）开始，当$G超过100万行时， $G的内容在索引同步期间自动移动到$I此值由STORAGE首选项的STAGE_ITAB_MAX_ROWS属性控制。

注意：

如果偶尔的索引同步需要很长时间，您可以通过使用STORAGE首选项的STAGE_ITAB_PARALLEL属性来减小STAGE_ITAB_MAX_ROWS参数的值或增加并行度。如果将STAGE_ITAB_MAX_ROWS的值设置为非常小的值，则$G的内容将频繁移至$I因此，不要将其设置为非常低的值。

注意：

要对CONTEXT索引使用STAGE_ITAB索引选项，必须指定g_index_clause和g_table_clause BASIC_STORAGE首选项。

默认情况下，如果在升级到Oracle Database Release 18c之前已在索引中启用了STAGE_ITAB ，则不会启用STAGE_ITAB_AUTO_OPT 。如果STAGE_ITAB和AUTO_OPTIMIZE在现有的索引被启用，则必须禁用AUTO_OPTIMIZE启用前STAGE_ITAB_AUTO_OPT.

以下示例创建基本mystore存储首选项，并将其STAGE_ITAB_AUTO_OPT存储属性的值设置为TRUE:

exec ctx_ddl.create_preference（'mystore'，'basic_storage'）; exec ctx_ddl.set_attribute（'mystore'，'stage_itab'，'TRUE'）; exec ctx_ddl.set_attribute（'mystore'，'stage_itab_auto_opt'，'TRUE'）; exec ctx_ddl.set_attribute（'mystore'，'stage_itab_parallel'，16）;

10.4 Oracle Text中的复合域索引

Oracle数据库中可扩展性框架的复合域索引（CDI）功能使结构化列可以由Oracle Text编制索引。因此，单个Oracle Text索引行源可以满足文本和一个或多个结构化标准。以下类型的查询的性能得到改进：

  • SQL语句在SQL WHERE子句中使用结构化条件查询

  • 具有结构化ORDER BY条件的Oracle Text查询

  • 前两种查询类型的组合

与连续的b树索引或位图索引一样，随着FILTER BY和ORDER BY列数量的增加，应用程序的数据操作语言（DML）性能会下降。在针对响应时间优化SCORE -sort下推的情况下， SCORE和结构化排序下推的结构化排序或组合也针对响应时间进行了优化，但不针对吞吐量进行了优化。但是，使用DOMAIN_INDEX_SORT或FIRST_ROWS (n)提示强制将排序推送到CDI，同时获取整个命中列表可能会导致查询响应时间不佳。

10.5使用CDI进行性能调整

因为您可以将FILTER BY列映射到MDATA,所以可以通过限制RANGE和LIKE.支持的功能来优化相等搜索的查询性能LIKE.但是，如果FILTER BY列包含顺序值或具有非常高的基数，Oracle建议不要将FILTER BY列映射到MDATA 。这样做可能会导致非常长而窄的$I表并降低$X性能。这种顺序列的一个示例可以是使用DATE标记的列。对于此类顺序列，建议映射到SDATA 。

使用以下提示将SORT和FILTER BY谓词推入或不推入CDI：

  • DOMAIN_INDEX_SORT:查询优化器尝试将适用的排序条件推送到指定的CDI。

  • DOMAIN_INDEX_NO_SORT:查询优化器尝试不将排序条件推送到指定的CDI。

  • DOMAIN_INDEX_FILTER （ 表名 索引名称 ）：查询优化器尝试将适用的FILTER BY谓词推送到指定的CDI。

  • DOMAIN_INDEX_NO_FILTER （ 表名 索引名称 ）：查询优化器不会尝试将适用的FILTER BY谓词推送到指定的CDI中。

注意：

domain_index_filter提示不会强制查询优化器使用CDI。相反，如果CBO选择使用CDI，那么它也应该将过滤谓词推送到索引中。要强制查询优化器选择CDI索引，您还需要使用INDEX提示。

10.6使用跟踪解决索引和查询瓶颈

Oracle Text包含一个跟踪功能，使您能够识别索引和查询中的瓶颈。

Oracle Text提供了一组预定义的跟踪。 每条迹线都由唯一编号标识。CTX_OUTPUT包含此编号的符号。

每个跟踪都会测量特定的数字量，例如在文本查询期间选择的$I行数。

跟踪是累积计数器，因此用法如下：

  1. 用户启用跟踪。

  2. 用户执行一个或多个操作。Oracle Text测量活动并在跟踪中累积结果。

  3. 用户检索跟踪值，该值是在步骤2中完成的所有操作的总值。

  4. 用户将跟踪重置为0。

  5. 用户从步骤2开始。

因此，例如，如果在步骤2中用户运行两个查询，并且查询1从$I,选择15行$I,并且查询2从$I,选择17行$I,则在步骤3中，跟踪的值为32（15 + 17） ）。

跟踪与会话相关联 - 它们可以测量在单个会话中发生的操作，相反，无法跨会话进行测量。

在并行同步或优化期间，当且仅当当前启用了跟踪时，才会将跟踪配置文件复制到从属会话。每个从器件累积自己的跟踪，并在终止之前将所有跟踪值隐式写入从属日志文件。

10.7使用并行查询

通常，并行查询对于决策支持系统（DSS）和在线分析处理（OLAP）是最佳的。它们也适用于具有大量数据集合的分析系统，具有较少并发用户数的多个CPU，或者 Oracle Real Application Clusters（Oracle RAC）节点。

Oracle Text支持以下并行查询：

  • 本地上下文索引的并行查询

  • 跨Oracle RAC节点并行化查询

10.7.1本地上下文索引的并行查询

并行查询是指本地CONTEXT索引的并行处理。根据索引的并行度和各种系统属性，Oracle确定要生成以处理索引的并行查询从属的数量。每个并行查询从属处理一个或多个索引分区。此默认查询行为适用于并行创建的本地索引。

但是，对于具有大量并发用户的高负载系统，并行查询的查询吞吐量通常更差;如果查询是串行运行的，那么前N个命中通常可以通过前几个分区来满足。例如，使用ORDER BY分区键列进行典型的前N个文本查询：

select * from（从stories_tab中选择story_id，其中包含（...）> 0 order by publication_date desc）其中rownum <= 10;

这些文本查询通常在并行查询时表现更差 。

您可以在使用ALTER INDEX语句进行并行索引操作后禁用并行查询：

更改索引<文本索引名称> NOPARALLEL;更改索引<文本索引名称> PARALLEL 1;

您还可以启用或增加并行度：

更改索引<文本索引名称>并行<并行度>;

10.7.2跨Oracle RAC节点并行化查询

Oracle Real Application Clusters（Oracle RAC）为提高查询吞吐量提供了出色的解决方案。如果您可以通过轻量级查询加载从Oracle Text获得良好性能，那么随着查询负载的增加，您可以期望从Oracle RAC获得出色的可伸缩性。

您可以通过物理分区文本数据和Oracle Text索引（使用本地分区索引）并确保分区由单独的Oracle RAC节点处理来实现Oracle Text性能的进一步改进。这样，您可以避免跨多个节点重复缓存内容，从而最大限度地发挥Oracle RAC缓存融合的优势。

在Oracle 10g第1版中，必须在创建索引时将每个Oracle Text索引分区强制为单独的数据库文件。这使得Oracle RAC中的“重新制作”功能可以强制实现数据库文件关联，其中每个节点都集中在特定的数据库文件上，因此也是特定的Oracle Text索引分区。

在Oracle 10g第2版及更高版本中，Oracle支持数据库对象级关联，这使得将索引对象（ $I和$R表）分配给特定节点变得更加容易。

尽管Oracle RAC提供了用于提高查询吞吐量和性能的解决方案，但它并不一定能够在扩展数据卷时继续获得相同的性能改进。您更有可能通过增加系统全局区域（SGA）缓存可用的内存量或通过对数据进行分区来查看改进，以便查询不必命中所有表分区以提供所需的查询结果集。

10.8使用阻止操作调整查询

如果发出具有多个谓词的查询，则可以在执行计划中导致阻止操作。例如，请考虑以下混合查询：

从mytab中选择docid，其中包含（text，'oracle'，1）> 0 AND colA> 5 AND colB> 1 AND colC> 3;

假设所有谓词都是非选择性的，colA，colB和colC都有位图索引。Oracle数据库中的CBO选择以下执行计划：

表格通过ROWIDS BITMAP转换为ROWIDS BITMAP和BITMAP索引COLA_BMX BITMAP索引COLB_BMX BITMAP索引COLC_BMX来自ROWIDS的BITMAP转换按区域索引MYINDEX排序

由于BITMAP AND是阻塞操作，因此Oracle Text必须在运行BITMAP AND操作之前临时保存从Oracle Text域索引返回的rowid和score对。

Oracle Text尝试将这些rowid和score对保存在内存中。但是，当结果集的大小超过SORT_AREA_SIZE初始化参数时，Oracle Text会将这些结果溢出到磁盘上的临时段。

由于将结果保存到磁盘会导致额外开销，因此可以通过增加SORT_AREA_SIZE参数来提高性能。 

alter session set SORT_AREA_SIZE = <新的内存大小，以字节为单位>;

例如，将缓冲区设置为大约8兆字节。

alter session set SORT_AREA_SIZE = 8300000;

10.9.1什么是查询性能？

答：查询性能有两种衡量标准：

  • 响应时间：获得单个查询答案的时间

  • 吞吐量：在任何给定时间段内可以运行的查询数量;例如，每秒查询一次

这两项措施是相关的，但它们并不相同。在负载很重的系统中，您需要最大吞吐量，而在负载相对较轻的系统中，您可能需要最短的响应时间。此外，某些应用程序需要查询以向用户提供所有匹配，而其他应用程序仅需要来自有序集的前20个匹配。区分这两种情况很重要。

10.9.2什么是最快的Oracle Text查询类型？

答：最快的查询类型符合以下条件：

  • 单个CONTAINS子句

  • WHERE子句中没有其他条件

  • 没有ORDER BY子句

  • 仅返回结果的第一页（例如，前10或20次点击）

10.9.3我应该收集有关表格的统计数据吗？

答：是的。通过收集表的统计信息，Oracle Text可以进行基于成本的分析。这有助于Oracle Text为您的查询选择最有效的执行计划。

如果您的查询始终是纯文本查询（没有结构化谓词且没有连接），则应删除Oracle Text索引的统计信息。

10.9.4我的数据大小如何影响查询？

答案： Oracle Text索引可以提供rowid的速度不受实际数据大小的影响。Oracle Text查询速度与必须从索引表中提取的行数，请求的命中数，查询生成的命中数以及是否存在排序有关。

10.9.5我的数据格式如何影响查询？

答：文档格式（纯ASCII文本，HTML或Microsoft Word）对查询速度没有影响。文档在索引时过滤为纯文本，而不是查询时间。

数据的清洁度有所不同。用于发布的拼写检查和子编辑文本往往具有比非正式文本（包括拼写错误和缩写）小得多的总词汇量（因此索引表的大小）。对于给定的索引内存设置，额外文本占用内存，创建更多碎片行，并对查询响应时间产生负面影响。

10.9.6索引查找和功能查找之间有什么区别

答案：内核可以使用索引查找和功能查找来查询Oracle Text索引。在索引查找中，第一个也是最常见的情况是，内核向Oracle Text索引询问满足特定文本搜索的所有rowid。这些rowid分批返回。

在函数查找中，内核将各个rowid传递给Oracle Text索引，并询问该特定rowid是否满足某个文本标准。函数查找最常用于非常有选择性的结构化子句，因此必须根据Oracle Text索引检查几个rowid。以下是功能查找有用的搜索示例：

SELECT ID，SCORE（1），来自MYTABLE的文字
在START_DATE ='1992年10月21日'< - 高度选择性和包含（TEXT，'commonword'）> 0 < - 非选择性

功能调用还用于按结构化列排序的Oracle Text查询（例如日期，价格）以及Oracle Text查询是否包含非选择性单词。

10.9.7查询涉及哪些表？

答案：所有查询都会查看索引令牌表。表的名称具有DR$indexname$I的形式，并包含令牌列表（ TOKEN_TEXT列）以及有关令牌发生的行和字位置的信息（ TOKEN_INFO列）。

行信息存储为内部docid值，必须转换为外部rowid值。您使用的表取决于查找的类型：

  • 对于功能查找，请使用$K表， DR$indexname$K.这个简单的索引组织表（IOT）包含每个docid / rowid对的行。

  • 对于索引查找，请使用$R表， DR$indexname$R.此表包含BLOB列中的rowid的完整列表。

从Oracle Database 12 c第2版（12.2） SMALL_R_ROW,引入了一个新的存储属性SMALL_R_ROW,来减小$R行的大小。它按需填充$R行而不是创建22个静态行，从而减少了数据操作语言争用。当并行插入，更新和删除操作尝试锁定相同的$R行时，会发生争用。

通过检查SQL跟踪并查找$K或$R表，可以轻松找出是否正在使用功能查询或索引查找。

注意：

这些内部索引表可能会随发行版的不同而变化。Oracle建议您不要直接访问应用程序中的这些表。

10.9.8如何减少$ R表争用？

基表删除和更新操作期间的$R争用已成为过去几年中反复出现的主题。目前，每个$R索引表有22个静态行，每行最多可包含2亿个rowid。当并行插入，更新和删除操作尝试锁定插入或删除操作的相同$R行时，会发生争用。此版本中的以下增强功能减少了争用：

  • 每个$R行可以包含的最大rowid数是70,000，这意味着每行存储1 MB的数据。要使用此功能，必须设置SMALL_R_ROW存储属性。

  • $R行是按需创建的，而不是仅填充预定数量的行。

10.9.9排序结果是否会减慢纯文本查询的速度？

答：是的，确实如此。

如果Oracle Text没有排序，那么它可以在找到它们时返回结果。当应用程序需要一次只显示一页结果时，此方法更快。

10.9.10如何更快地进行ORDER BY分数查询？

答案：如果使用FIRST_ROWS(n)提示，按相关性排序（ SCORE(n) ）可能会很快。在这种情况下，Oracle Text在从Oracle Text索引表中获取时执行高速内部排序。

以下是此查询的示例：

SELECT / * + FIRST_ROWS（10）* / ID，SCORE（1），TEXT FROM mytable WHERE CONTAINS（TEXT，'searchterm'，1）> 0 ORDER BY SCORE（1）DESC;

值得注意的是，除了单个CONTAINS.之外， WHERE子句中不能有其他标准CONTAINS.

10.9.12宽基表列的外线LOB存储是否提高了性能？

答：是的。通常， SELECT语句从基表中选择多个列。由于Oracle Text将列提取到内存，因此将大型基表列（如大对象（LOB））存储在线外会更有效，尤其是当这些列很少更新但经常被选中时。

当LOB存储在行外时，在查询期间只需要将LOB定位符提取到内存。外部存储减少了基表的有效大小。它使Oracle Text更容易将整个表缓存到内存中，因此降低了从基表中选择列的成本，并加快了文本查询的速度。

此外，缓存在内存中的较小基表允许在查询期间缓存更多索引表数据，从而提高性能。

10.9.13如何在多个列上加速CONTAINS查询？

答案：最快的查询类型是只有一个CONTAINS子句而WHERE子句中没有其他条件的WHERE 。

考虑以下多个CONTAINS查询：

SELECT title，isbn FROM booklist WHERE CONTAINS（title，'horse'）> 0 AND CONTAINS（abstract，'racing'）> 0

您可以通过节搜索和WITHIN运算符获得相同的结果：

SELECT title，isbn FROM booklist WHERE CONTAINS（alltext，'horse WITHIN title AND racing WITHIN abstract'）> 0

此查询比单个CONTAINS子句更快地完成。要使用这样的查询，您必须将所有数据复制到单个文本列中以进行索引，并在每个列的数据周围添加节标记。您可以在建立索引之前使用PL / SQL过程执行此操作，也可以在索引USER_DATASTORE期间使用USER_DATASTORE数据存储区将带有text列的结构化列合成为一个文档。

10.9.14我可以在查询中进行多次扩展吗？

答案：查询中使用的每个不同的单词都需要从索引表中提取至少一行。因此，最好尽可能减少扩展次数。

你不应该使用扩展，如通配符，词库，词干和模糊匹配，除非它们是任务所必需的。通常，一些扩展（例如，10到20）不会造成困难，但在查询中避免大量扩展（80或100）。使用查询反馈机制确定任何特定查询表达式的扩展数。

对于通配符和词干查询，您可以通过创建前缀，子字符串或词干索引来避免从查询时间到索引时间的术语扩展。查询性能的增加是以更长的索引时间和增加的磁盘空间为代价的。

前缀和子字符串索引可以提高通配符性能。您可以使用BASIC_WORDLIST首选项启用前缀和子字符串索引。以下示例为前缀和子字符串索引设置wordlist首选项。对于前缀索引，它指定Oracle Text创建3到4个字符长的令牌前缀：

结束

结束;

10.9.18何时应使用CTXCAT指数？

答案： CTXCAT索引在文本处于小块（仅几行）时效果最佳，并且您希望搜索根据某些结构化条件（如数字或日期）限制或排序结果集。

例如，考虑在线拍卖网站。每个待售商品都有简短描述，当前竞标价格以及拍卖的开始和结束日期。用户可能希望在说明中查看带有古董柜的所有记录，当前出价低于500美元。因为他对新发布的项目特别感兴趣，所以他希望按拍卖开始时间排序结果。

对于CONTEXT索引上的CONTAINS结构化查询，此搜索并不总是有效。响应时间可根据结构和显著变化CONTAINS子句，由于结构化的交点和CONTAINS条款或在Oracle文本查询排序在查询时间计算的。

通过在CTXCAT索引中包含结构化信息，可以确保查询响应时间始终处于最佳范围，而不管搜索条件如何。由于文本和结构化查询之间的交互是在索引期间预先计算的，因此查询响应时间是最佳的。

10.9.19何时CTXCAT索引不合适？

答：创建索引所需的时间和空间存在差异。CTXCAT索引需要更长时间才能创建，并且它们使用的磁盘空间比CONTEXT索引多得多。如果您的磁盘空间CTXCAT ，请仔细考虑CTXCAT索引是否适合您。

使用查询运算符，您可以使用查询模板在CATSEARCH查询中使用更丰富的CONTEXT语法。单个CATSEARCH查询语法的旧限制不再成立。

10.10.1索引需要多长时间？

答：索引文本是一个资源密集型过程。索引的速度取决于硬件的功率。索引速度取决于CPU和I / O容量。有足够的I / O容量来读取原始数据并写出索引条目，CPU是限制因素。

采用Intel x86（Core 2架构，2.5GHz）CPU的测试表明，Oracle Text每天可为每个CPU核心索引大约100 GB的文本。随着CPU时钟速度的增加和CPU架构变得更加高效，预计这种速度将会提高。

其他因素（例如文档格式，数据位置以及对用户定义的数据存储，过滤器和词法分析器的调用）可能会影响索引速度。

10.10.2我应该使用哪种索引内存设置？

答案：您可以使用DEFAULT_INDEX_MEMORY和MAX_INDEX_MEMORY系统参数设置索引内存。您还可以使用参数字符串中的CREATE INDEX memory参数在运行时设置索引内存。

您应该将DEFAULT_INDEX_MEMORY值设置为尽可能高，而不会导致分页。

您还可以通过增加SORT_AREA_SIZE系统参数来提高索引性能。

Oracle建议您使用大型索引内存设置。大型设置，甚至高达数百兆字节，可以提高索引速度并减少最终索引的碎片。但是，如果将索引内存设置设置得太高，则内存分页会降低索引速度。

使用并行索引，每个流都需要自己的索引内存。处理非常大的表时，可以不同地调整数据库系统全局区域（SGA）以进行索引和检索。对于查询，您希望尽可能多地在SGA块缓冲区缓存中缓存信息。因此，您应该为块缓冲区缓存分配大量内存。因为这种方法对索引没有任何影响，所以最好减小SGA的大小，以便在索引期间为大型索引内存设置腾出更多空间。

您可以在Oracle数据库初始化文件中设置SGA的大小。

10.10.3索引需要多少磁盘开销？

答案：开销，索引表所需的空间量，在原始文本卷的大约50％到200％之间变化。通常，较大量的文本会导致较小的开销，但许多小记录比较少的大记录使用更多的开销。此外，干净的数据（例如发布的文本）比诸如电子邮件或讨论笔记之类的脏数据需要更少的开销，因为脏数据可能包括许多拼写错误和缩写的单词。

纯文本索引小于组合文本和主题索引。前缀和子字符串索引使索引显着增大。

10.10.4我的数据格式如何影响索引？

答：对于格式化文档（如Microsoft Word文件），您可以期望更低的存储开销，因为文档与其中保存的实际文本相比往往非常大。因此，1 GB的Word文档可能只需要50 MB的索引空间，而1 GB的纯文本可能需要500 MB，因为后者的纯文本数量是后者的十倍。

索引时间不那么明确。尽管要编制索引的文本数量的减少具有明显的效果，但您必须将其与使用AUTO_FILTER过滤器或其他用户定义的过滤器过滤文档的成本进行平衡。

10.10.5并行索引可以提高性能吗？

答：当您拥有大量数据和多个CPU时，并行索引可以提高索引性能。

使用PARALLEL关键字创建最多包含三个独立索引进程的索引，具体取决于您的资源。 

CREATE INDEX index_name ON table_name（column_name）INDEXTYPE IS ctxsys.context PARAMETERS（'...'）PARALLEL 3;

您还可以使用并行索引在分区表上创建本地分区索引。但是，索引性能仅在多个CPU的情况下才会提高。

注意：

使用PARALLEL创建本地分区索引可启用并行查询。（并行创建非分区索引不会启用并行查询处理。）

并行查询会降低查询吞吐量，尤其是在负载较重的系统上。因此，Oracle建议您在并行索引后禁用并行查询。为此，请使用ALTER INDEX NOPARALLEL.

10.10.6创建本地分区索引时，如何提高索引性能？

11.2异常处理概述

异常（PL / SQL运行时错误）可能由设计错误，编码错误，硬件故障和许多其他来源引起。您无法预测所有可能的异常，但您可以编写异常处理程序，让您的程序继续在其存在的情况下运行。

任何PL / SQL块都可以有一个异常处理部分，它可以有一个或多个异常处理程序。例如，异常处理部分可以具有以下语法：

ex_name_1 THEN statements_1之外的异常 - 异常处理程序WHEN ex_name_2 OR ex_name_3 THEN statements_2 - 异常处理程序WHEN OTHERS THEN statements_3 - 异常处理程序END;

在前面的语法示例中， ex_name_ n是异常的名称，而statements_ n是一个或多个语句。（有关完整的语法和语义，请参阅“ 异常处理程序 ” 。）

当在块的可执行部分中引发异常时，可执行部分停止并控制对异常处理部分的转移。如果ex_name_1 ，那么statements_1运行。如果ex_name_2或ex_name_3 ，则statements_2运行。如果引发任何其他异常，则statements_3运行。

在异常处理程序运行后，控制转移到封闭块的下一个语句。如果没有封闭块，那么：

  • 如果异常处理程序在子程序中，则控制将在调用之后的语句处返回到调用者。

  • 如果异常处理程序位于匿名块中，则控件将传输到主机环境（例如，SQL * Plus）

如果在没有异常处理程序的块中引发异常，则异常传播。也就是说，异常在连续的封闭块中再现它，直到一个块有一个处理程序或者没有封闭块（有关更多信息，请参阅“ 异常传播 ” ）。如果没有异常处理程序，则PL / SQL会向调用者或主机环境返回未处理的异常错误，这将确定结果（有关更多信息，请参阅“ 未处理的异常 ” ）。

11.2.2异常处理程序的优点

使用异常处理程序进行错误处理使程序更易于编写和理解，并降低了未处理异常的可能性。

如果没有异常处理程序，您必须检查可能发生的每个可能的错误，然后处理它。很容易忽略可能出现的错误或可能发生错误的地方，特别是如果错误无法立即检测到（例如，在计算中使用错误数据之前可能无法检测到错误数据）。错误处理代码分散在整个程序中。

使用异常处理程序，您无需知道每个可能的错误或任何可能发生的错误。您只需在每个可能发生错误的块中包含异常处理部分。在异常处理部分中，您可以包含特定和未知错误的异常处理程序。如果块中的任何地方（包括子块内）发生错误，则异常处理程序会处理它。错误处理代码在块的异常处理部分中被隔离。

您可以确定错误处理代码的精度。对于所有除零错误，错误的数组索引等，您可以拥有一个异常处理程序。您还可以通过将该语句放在具有自己的异常处理程序的块中来检查单个语句中的错误。

11.2.3避免和处理例外的指南

为了使您的程序尽可能可靠和安全：

  • 使用错误检查代码和异常处理程序。

    在错误的输入数据可能导致错误的地方使用错误检查代码。错误输入数据的示例是不正确的或空的实际参数以及不返回行或超出预期的行的查询。使用不同输入数据的不同组合测试代码，以查看可能出现的错误。

    有时您可以使用错误检查代码来避免引发异常，如例11-7所示 。

  • 在可能发生错误的地方添加异常处理程序。

    在算术计算，字符串操作和数据库操作期间尤其可能出现错误。错误也可能源于与您的代码无关的问题 - 例如，磁盘存储或内存硬件故障 - 但您的代码仍然必须采取纠正措施。

  • 设计您的程序，以便在数据库未处于您期望的状态时工作。

    例如，您查询的表可能添加或删除了列，或者它们的类型可能已更改。通过使用%TYPE限定符声明标量变量并使用%ROWTYPE限定符记录变量来保存查询结果，可以避免出现问题。

  • 尽可能为命名异常编写异常处理程序，而不是使用OTHERS异常处理程序。

    了解预定义异常的名称和原因。如果您知道数据库操作可能会引发没有名称的特定内部定义的异常，那么请为它们指定名称，以便您可以专门为它们编写异常处理程序。

  • 让异常处理程序输出调试信息。

    如果将调试信息存储在单独的表中，请使用自治例程，以便即使您回滚主子程序所执行的工作也可以提交调试信息。有关自治例程的信息，请参阅“ AUTONOMOUS_TRANSACTION Pragma ” 。

  • 对于每个异常处理程序，请仔细决定是将其提交事务，将其回滚还是让它继续。

    无论错误的严重程度如何，您都希望使数据库保持一致状态并避免存储错误数据。

  • 通过在每个PL / SQL程序的顶层包含OTHERS异常处理程序来避免未处理的异常。

    使OTHERS异常处理程序中的最后一个语句成为RAISE或调用RAISE_APPLICATION_ERROR过程。（如果您不遵循此惯例，并且启用了PL / SQL警告，那么您将获得PLW-06009。）有关RAISE或调用RAISE_APPLICATION_ERROR ，请参阅“ 明确提高异常 ” 。

11.9未处理的例外情况

如果没有引发异常的处理程序，PL / SQL会向调用者或宿主环境返回未处理的异常错误，从而确定结果。

如果存储的子程序以未处理的异常退出，则PL / SQL不会回滚子程序所做的数据库更改。

FORALL语句多次运行一个DML语句，在VALUES和WHERE子句中使用不同的值。如果一组值引发未处理的异常，则PL / SQL将回滚先前在FORALL语句中进行的所有数据库更改。有关更多信息，请参阅“ 立即处理FORALL异常 ”和“ FORALL语句完成后处理FORALL异常 ” 。

小费：

通过在每个PL / SQL程序的顶层包含OTHERS异常处理程序来避免未处理的异常。

11.12处理异常后重试事务

要在处理它引发的异常后重试事务，请使用以下技术：

  1. 将事务包含在具有异常处理部分的子块中。
  2. 在子块中，在事务开始之前，标记保存点。
  3. 在子块的异常处理部分中，放置一个异常处理程序，该处理程序回滚到保存点，然后尝试更正问题。
  4. 将子块放在LOOP语句中。
  5. 在子块中，在结束事务的COMMIT语句之后，放入一个EXIT语句。

    如果事务成功，则执行COMMIT和EXIT语句。

    如果事务失败，则控制转移到子块的异常处理部分，并且在异常处理程序运行之后，循环重复。

您可以使用触发器或存储的子程序来创建分布式查询。该分布式查询由本地Oracle数据库实例分解为相应数量的远程查询，这些远程查询被发送到远程节点以供执行。远程节点运行查询并将结果发送回本地节点。然后，本地节点执行任何必要的后处理，并将结果返回给用户或应用程序。

创建自动列表分区表

自动列表分区方法允许按需创建列表分区。

自动列表分区表类似于常规列表分区表，但该分区表更易于管理。您可以仅使用已知的分区键值创建自动列表分区表。当数据加载到表中时，如果加载的分区键值与任何现有分区不对应，则数据库会自动创建新分区。由于分区是按需自动创建的，因此自动列表分区方法在概念上与现有的间隔分区方法类似。

除非您可以调整数据，否则对值非常频繁更改的数据类型的自动列表分区不太适合此方法。例如，当格式未被剥离时，具有日期值的SALES_DATE字段将每秒增加。每个SALES_DATE值，例如05-22-2016 08:00:00 05-22-2016 08:00:01等，都会生成自己的分区。为避免创建大量分区，您必须了解要输入的数据并进行相应调整。例如，您可以将SALES_DATE日期值截断为一天或某个其他时间段，具体取决于所需的分区数。

CREATE和ALTER TABLE SQL语句使用附加子句进行更新，以指定AUTOMATIC或MANUAL列表分区。自动列表分区表在创建时必须至少有一个分区。由于为新的和未知的分区键值自动创建了新分区，因此自动列表分区不能具有DEFAULT分区。

您可以检查*_PART_TABLES视图的AUTOLIST列以确定表是否是自动列表分区的。

创建参考分区表

要创建引用分区表，请在CREATE TABLE语句中指定PARTITION BY REFERENCE子句。

PARTITION BY REFERENCE子句指定引用约束的名称，此约束成为分区引用约束，该约束用作表中引用分区的基础。必须启用并强制执行引用约束。

与其他分区表一样，您可以指定对象级缺省属性，也可以选择指定基于每个分区覆盖对象级缺省值的分区描述符。

如果提供了分区描述符，则所描述的分区数必须与引用表中的分区或子分区数完全相等。如果父表是复合分区表，则该表对其父表的每个子分区都有一个分区;否则该表为其父分区的每个分区都有一个分区。

无法为引用分区表的分区指定分区边界。

可以命名引用分区表的分区。如果未明确命名分区，则它将从父表中的相应分区继承其名称，除非此继承名称与现有显式名称冲突。在这种情况下，分区具有系统生成的名称。

创建间隔引用分区表

您可以将间隔分区表用作父表以进行引用分区。将记录插入引用分区表时，将创建与父表中的间隔分区对应的引用分区表中的分区。

在子表中创建间隔分区时，分区名称将从关联的父表片段继承。如果子表具有表级缺省表空间，则它将用作新间隔分区的表空间;否则，表空间继承自父表片段。

引用分区表不允许使用SQL ALTER TABLE SET INTERVAL语句，但可以在具有引用分区子项的表上运行。特别是， ALTER TABLE SET INTERVAL从目标表中删除interval属性，并将任何interval-reference子项转换为普通的reference-partitioned表。此外，SQL ALTER TABLE SET STORE IN语句不允许用于引用分区表，但可以在具有引用分区子项的表上运行。

使用只读分区或子分区创建表

您可以将表，分区和子分区设置为只读状态，以保护数据免受任何用户或触发器的无意DML操作。

任何更新设置为只读的分区或子分区中的数据的尝试都会导致错误，而更新分区或设置为读写的子分区中的数据会成功。

指定键列上的分区

对于范围分区和散列分区表，最多可以指定16个分区键列。

当分区键由多个列组成时，使用多列分区，后续列定义比前面的列更高的粒度。最常见的情况是分解的DATE或TIMESTAMP键，由分隔的列组成，包括年，月和日。

在评估多列分区键时，仅当第一个值不能唯一标识单个目标分区时，数据库才使用第二个值，并且仅当第一个值和第二个值未确定正确的分区时才使用第三个值，依此类推。仅当分区绑定与该值完全匹配且为下一个分区定义了相同的绑定时，值才能确定正确的分区。只有当所有先前的第（n-1）的多列键的值完全匹配分区的第（n-1）边界的第 n列进行了研究。例如，仅当第一列与分区边界值完全匹配时，才会评估第二列。如果所有列值都与分区的所有绑定值完全匹配，则数据库会确定该行不适合此分区，并将下一个分区视为匹配项。

对于非确定性边界定义（具有至少一列的相同值的连续分区），分区边界值变为包含值，表示“小于或等于”边界。这与确定性边界形成对比，其中值始终被视为“小于”边界。

创建多列范围分区表以实施等分区

本主题中的示例显示如何创建多列范围分区表以强制执行大小相等的分区。

以下示例说明了对表supplier_parts使用多列分区方法，存储有关哪些供应商提供哪些部件的信息。要在相同大小的分区中分发数据，仅基于supplier_id对表进行分区是不够的，因为某些供应商可能提供数十万个部件，而其他供应商仅提供少量特殊部件。相反，您在（ supplier_id ， partnum ）上对表进行分区以手动强制执行相等大小的分区。

无论partnum值如何， supplier_id <10的每一行都存储在分区p1 。仅当supplier_id = 10时才评估列partnum ，并且当partnum > = 200时，相应的行将插入到分区p1 ， p2或甚至p3 。要为supplier_parts范围实现大小相等的分区，您可以选择复合范围 - 散列分区表，由supplier_id分区的范围，由partnum分区的partnum 。

为多列分区表定义分区边界必须遵守某些规则。例如，考虑一个在三个列a ， b和c上进行范围分区的表。各个分区的范围值表示如下：

P0（a0，b0，c0）P1（a1，b1，c1）P2（a2，b2，c2）......Pn（an，bn，cn）

您为每个分区提供的范围值必须遵循以下规则：

  • a0必须小于或等于a1 ， a1必须小于或等于a2 ，依此类推。

  • 如果a0 = a1 ，则b0必须小于或等于b1 。如果a0 < a1 ，那么b0和b1可以有任何值。如果a0 = a1且b0 = b1 ，则c0必须小于或等于c1 。如果b0 < b1 ，那么c0和c1可以有任何值，依此类推。

  • 如果a1 = a2 ，则b1必须小于或等于b2 。如果a1 < a2 ，则b1和b2可以具有任何值。如果a1 = a2且b1 = b2 ，则c1必须小于或等于c2 。如果b1 < b2 ，那么c1和c2可以有任何值，依此类推。

表格中插入以下三条记录：

前两个记录插入到分区p1 ，由supplier_id唯一标识。但是，第三条记录被插入到分区p2 ;它严格匹配分区p1所有范围边界值，因此数据库会考虑以下分区进行匹配。partnum的值满足<200的标准，因此将其插入到分区p2 。

多个块大小的分区限制

在具有不同块大小的表空间的数据库中创建分区对象时要小心。

在这样的表空间中存储分区对象受到一些限制。具体来说，以下实体的所有分区必须驻留在相同块大小的表空间中：

  • 常规表格

  • 索引

  • 索引组织表的主键索引段

  • 索引组织表的溢出段

  • LOB列存储在行外

因此：

  • 对于每个传统表，该表的所有分区必须存储在具有相同块大小的表空间中。

  • 对于每个索引组织表，所有主键索引分区必须驻留在相同块大小的表空间中，并且该表的所有溢出分区必须驻留在具有相同块大小的表空间中。但是，索引分区和溢出分区可以驻留在不同块大小的表空间中。

  • 对于每个索引（全局或本地），该索引的每个分区必须驻留在相同块大小的表空间中。但是，在同一对象上定义的不同索引的分区可以驻留在不同块大小的表空间中。

  • 对于每个LOB列，该列的每个分区必须存储在具有相等块大小的表空间中。但是，不同的LOB列可以存储在不同块大小的表空间中。

创建或更改分区表或索引时，为每个实体的分区和子分区显式指定的所有表空间必须具有相同的块大小。如果没有为实体显式指定表空间存储，则默认情况下数据库使用的表空间必须具有相同的块大小。因此，您必须了解分区对象的每个级别的默认表空间。

使用XMLType或对象表和列时的分区遵循分区的基本规则。

出于本讨论的目的，术语集合 表用于以下两类：（1） XMLType表或列内的有序集合表，以及（2）对象表或列内的嵌套表。

Oracle数据库提供了一个LOCAL关键字，用于为Collection Table分配一个分区基表。这是此版本中的默认行为。早期版本中的默认设置不是将Collection Table与分区基表进行均分。现在，您必须指定GLOBAL关键字以存储带有分区基表的未分区Collection表。

支持外线（OOL）表分区。但是，您无法创建具有外联表的同一XML架构的两个表。此限制意味着无法对具有OOL表的模式执行交换分区，因为不可能有两个具有相同模式的表。

创建可插入的数据库

目的

使用CREATE PLUGGABLE DATABASE语句创建可插拔数据库（PDB）。

此语句使您可以执行以下任务：

  • 使用种子作为模板创建PDB

    使用create_pdb_from_seed子句通过将多租户容器数据库（CDB）中的种子用作模板来创建PDB。与种子关联的文件将复制到新位置，然后复制的文件将与新PDB关联。

  • 通过克隆现有PDB或非CDB来创建PDB

    使用create_pdb_clone子句通过复制现有PDB或非CDB然后将副本插入CDB来创建PDB。与现有PDB或非CDB关联的文件将复制到新位置，复制的文件将与新PDB关联。

  • 通过将未插入的PDB或非CDB插入CDB来创建PDB

    使用create_pdb_from_xml子句使用XML元数据文件将未插入的PDB或非CDB插入CDB。

  • 通过引用另一个PDB来创建代理PDB。代理PDB提供对引用的PDB的完全功能访问。

    使用create_pdb_clone子句并指定AS PROXY FROM以创建代理PDB。

  • 创建应用程序容器，应用程序种子或应用程序PDB

    使用create_pdb_from_seed ， create_pdb_clone或create_pdb_from_xml clause 。要创建应用程序容器，必须指定AS APPLICATION CONTAINER子句。要创建应用程序种子，必须指定AS SEED子句。

注意：

在CDB中建立新的PDB时，新PDB提供的服务名称可能会与现有服务名称冲突。可以发生冲突的命名空间是允许访问CDB的侦听器的命名空间。在该命名空间内，非CDB的默认服务，CDB的默认服务，PDB的默认服务和用户定义的服务的名称之间可能存在冲突。例如，如果同一计算机系统上的两个或多个CDB使用相同的侦听器，并且新建立的PDB具有与这些CDB中的另一个PDB相同的服务名称，则发生冲突。

创建PDB时，可以为任何潜在的冲突服务名称指定新名称。请参阅service_name_convert子句。如果在创建PDB后发现服务名称冲突，则不得尝试操作导致与现有服务名称冲突的PDB。如果冲突名称是PDB默认服务的名称，则必须重命名PDB。如果冲突名称是PDB中用户创建的服务的名称，则必须删除该服务并使用具有相同目的和属性的非冲突名称创建一个名称。

也可以看看：

  • 有关多租户架构和概念的更多信息，请参阅“Oracle多租户管理员指南” 。

  • ALTER PLUGGABLE DATABASE和DROP PLUGGABLE DATABASE有关修改和删除PDB的信息

先决条件

您必须连接到CDB。 CDB必须处于打开状态并处于READ WRITE模式。

要创建PDB或应用程序容器，当前容器必须是根，并且您必须具有通常授予的CREATE PLUGGABLE DATABASE系统特权。

要创建应用程序种子或应用程序PDB，当前容器必须是应用程序根目录，应用程序容器必须处于打开状态并处于READ WRITE模式，并且您必须具有CREATE PLUGGABLE DATABASE系统特权，可以是通常授予的，也可以是本地授予的应用容器。

要指定create_pdb_clone子句：

  • 如果src_pdb_name引用同一CDB中的PDB，则必须在将在其中创建新PDB的CDB的根目录中以及在克隆的PDB中具有CREATE PLUGGABLE DATABASE系统特权。

  • 如果src_pdb_name引用远程数据库或非CDB中的PDB，则必须在CDB的根目录中具有CREATE PLUGGABLE DATABASE系统特权，其中将创建新的PDB，并且远程用户必须具有CREATE PLUGGABLE DATABASE src_pdb_name引用的PDB或非CDB中的系统特权。

有关PDB创建的先决条件的详细信息，请参阅“ Oracle Multitenant管理员指南” 。

PDB名称在CDB中必须是唯一的，并且在通过特定侦听器到达其实例的所有CDB的范围内必须是唯一的。

ADMIN USER

使用此子句可以创建一个管理用户，该用户可以被授予在PDB上执行管理任务所需的权限。对于admin_user_name ，请指定要创建的用户的名称。使用IDENTIFIED BY子句指定admin_user_name的密码。Oracle数据库在PDB中创建本地用户，并向该用户授予PDB_DBA本地角色。

pdb_dba_roles

此子句允许您向PDB_DBA角色授予一个或多个角色。使用此子句授予具有PDB管理用户所需权限的角色。对于role ，请指定预定义角色。有关预定义角色的列表，请参阅“ Oracle数据库安全指南” 。

您还可以使用GRANT语句在创建PDB PDB_DBA角色授予PDB_DBA角色。在您为PDB_DBA角色授予适当的权限PDB_DBA ， SYS和SYSTEM用户可以在PDB上执行管理任务。

parallel_pdb_creation_clause

该子句指示CDB使用并行执行服务器将新PDB的数据文件复制到新位置。这可能导致更快地创建PDB。

平行

如果指定PARALLEL ，则CDB会自动选择要使用的并行执行服务器的数量。如果COMPATIBLE初始化参数设置为12则这是默认值。 2或更高。

PARALLEL 整数

使用integer指定要使用的并行执行服务器的数量。CDB可以忽略此设置，具体取决于当前数据库负载和可用并行执行服务器的数量。如果指定值0或1，则CDB不会并行创建PDB。这可能导致PDB创建时间更长。

default_tablespace

如果指定此子句，则Oracle数据库会创建一个smallfile表空间并将其设置为PDB的默认永久表空间。 Oracle数据库将默认表空间分配给未指定其他永久表空间的任何非SYSTEM用户。default_tablespace子句与CREATE DATABASE语句具有相同的语义。有关完整信息，请参阅CREATE DATABASE文档中的default_tablespace 。

pdb_storage_clause

使用此子句指定PDB的存储限制。

  • 使用MAXSIZE将PDB中所有表空间可以使用的存储量限制为使用size_clause指定的值。此限制包括属于PDB的表空间的数据文件和临时文件的大小。指定MAXSIZE UNLIMITED以强制执行无限制。

  • 使用MAX_AUDIT_SIZE限制可以通过统一的审计OS溢出（可以使用的存储量.bin格式）文件，在PDB与指定的值size_clause 。指定MAX_AUDIT_SIZE UNLIMITED以强制执行无限制。

  • 使用MAX_DIAG_SIZE限制自动诊断信息库（ADR）中的诊断（跟踪文件和事件转储）的存储量，PDB可以将其用于使用size_clause指定的值。指定MAX_DIAG_SIZE UNLIMITED以强制执行无限制。

如果省略此子句，或指定STORAGE UNLIMITED ，则PDB没有存储限制。这相当于指定STORAGE (MAXSIZE UNLIMITED MAX_AUDIT_SIZE UNLIMITED MAX_DIAG_SIZE UNLIMITED ）。

file_name_convert

使用此子句可确定数据库如何为PDB生成文件名（例如数据文件和钱包文件）。

  • 对于filename_pattern ，指定在与种子关联的文件名称中找到的字符串（使用种子创建PDB时），与源PDB关联（克隆PDB时），或在XML文件中列出（将PDB插入时）一个CDB）。

  • 对于replacement_filename_pattern ，请指定替换字符串。

在生成与新PDB关联的文件的名称时，Oracle数据库将使用replacement_filename_pattern替换filename_pattern 。

文件名模式与Oracle Managed Files管理的文件或目录不匹配。

您可以指定FILE_NAME_CONVERT = NONE ，这与省略此子句相同。如果省略此子句，则数据库首先尝试使用Oracle Managed Files生成文件名。如果您未使用Oracle Managed Files，则数据库使用PDB_FILE_NAME_CONVERT初始化参数来生成文件名。如果未设置此参数，则会发生错误。

service_name_convert

使用此子句根据源PDB的服务名称重命名新PDB的用户定义服务。当新PDB的服务名称与CDB中的现有服务名称冲突时，可能会导致插件违规。此子句使您可以避免这些违规。

  • 对于service_name ，指定在PDB种子中找到的服务的名称（在使用应用程序种子在应用程序容器中创建PDB时）或在源PDB中（在克隆PDB或将PDB插入CDB时）。

  • 对于replacement_service_name ，请指定服务的替换名称。

Oracle数据库将使用正在创建的PDB中的服务的替换服务名称。

您可以指定SERVICE_NAME_CONVERT = NONE ，这与省略此子句相同。

service_name_convert的 限制

service_name_convert子句受以下限制：

  • 您无法更改PDB的默认服务名称。默认服务与PDB具有相同的名称。

  • 使用create_pdb_from_seed子句从CDB种子创建PDB时，不能指定此子句，因为CDB种子没有用户定义的服务。但是，在使用create_pdb_from_seed子句从应用程序种子创建应用程序PDB时，可以指定此子句。

path_prefix_clause

使用此子句可确保与PDB关联的目录对象的文件路径仅限于指定目录或其子目录。此子句还确保与PDB关联的以下文件仅限于指定的目录：PDB的Oracle XML存储库，使用CREATE PFILE语句创建的文件以及Oracle钱包的导出目录。创建PDB后，无法修改此子句的设置。此子句不影响Oracle Managed Files创建的文件。

  • 对于path_name ，请指定操作系统目录的绝对路径名。单引号是必需的，结果是路径名称区分大小写。Oracle数据库使用path_name作为与PDB关联的所有文件路径的前缀。

    请务必指定path_name以便在将相对路径附加到路径名时正确形成结果路径名。例如，在UNIX系统上，请确保使用正斜杠（ / ）结束path_name ，例如：

    PATH_PREFIX ='/ disk1 / oracle / dba / salespdb /'

  • 对于directory_object_name ，请指定CDB根目录中存在的目录对象的名称（ CDB$ROOT ）。目录对象指向用于PATH_PREFIX的绝对路径。

  • 如果指定PATH_PREFIX = NONE ，则与PDB关联的目录对象的相对路径将被视为绝对路径，并且不限于特定目录。

省略path_prefix_clause等同于指定PATH_PREFIX = NONE 。

在为PDB指定path_prefix_clause之后，现有目录对象可能无法按预期工作，因为PATH_PREFIX字符串始终作为前缀添加到PDB中的所有本地目录对象。 path_prefix_clause仅适用于用户创建的目录对象。它不适用于Oracle提供的目录对象。

tempfile_reuse_clause

创建PDB时，Oracle数据库会将临时文件与新PDB关联。根据您创建PDB的方式，临时文件可能已经存在并且可能以前曾使用过。

指定TEMPFILE REUSE以指示数据库格式化TEMPFILE用与新PDB关联的临时文件（如果已存在）。如果指定此子句且临时文件不存在，则数据库将创建临时文件。

如果未指定TEMPFILE REUSE且与新PDB关联的临时文件已存在，则数据库将返回错误并且不会创建PDB。

user_tablespaces_clause

此子句允许您指定要在新PDB中使用的表空间。 SYSTEM ， SYSAUX和TEMP表空间在所有PDB中都可用，并且不能在本节中指定。

您可以使用此子句将多个模式的数据分成不同的PDB。例如，当您将非CDB移动到PDB，并且非CDB具有多个模式，每个模式支持不同的应用程序时，您可以使用此子句将属于每个模式的数据分隔为单独的PDB，假设每个模式在非CDB中使用单独的表空间。

  • 指定tablespace以使表空间在新PDB中可用。您可以在逗号分隔列表中指定多个表空间。

  • 指定ALL以使新PDB中的所有表空间可用。这是默认值。

  • 指定ALL EXCEPT以使新PDB中的所有表空间可用，但指定的表空间除外。

  • 指定NONE以仅在新PDB中提供SYSTEM ， SYSAUX和TEMP表空间。

当CDB的兼容级别为12.2或更高时，此子句排除的表空间在新PDB中脱机创建，并且它们没有与之关联的数据文件。当CDB的兼容级别低于12.2 ，此子句排除的表空间在新PDB中处于脱机状态，属于这些表空间的所有数据文件都是未命名和脱机的。

{SNAPSHOT COPY |没有数据 }

这些子句仅在使用create_pdb_clone子句克隆PDB时适用。克隆非CDB时不适用。默认情况下，数据库根据为克隆PDB指定的设置创建新PDB中可用的每个表空间。这些子句允许您覆盖这些设置，如下所示：

  • SNAPSHOT COPY - 使用存储快照克隆表空间。

  • NO DATA - 克隆表空间的数据模型定义，但不克隆表空间的数据。

{COPY |移动| NOCOPY}

当您使用create_pdb_from_xml子句插入PDB时，这些子句适用。默认情况下，数据库根据为插入PDB指定的设置创建在新PDB中可用的每个表空间。这些子句允许您覆盖这些设置，如下所示：

  • COPY - 将表空间文件复制到新位置。

  • MOVE - 将表空间文件移动到新位置。

  • NOCOPY - 不要将表空间文件复制或移动到新位置。

standbys_clause

使用此子句指定新PDB是否包含在一个或多个备用CDB中。如果在备用CDB中包含PDB，则在备用恢复期间，备用CDB将搜索PDB的数据文件。如果找不到数据文件，则备用恢复将停止，您必须将数据文件复制到正确的位置，然后才能重新启动恢复。

  • 指定cdb_name以在指定的备用CDB中包含新PDB。您可以在逗号分隔列表中指定多个备用CDB名称。

  • 指定ALL以在所有备用CDB中包含新PDB。这是默认值。

  • 指定ALL EXCEPT以在所有备用CDB中包括新PDB，但指定的备用CDB除外。

  • 指定NONE以从所有备用CDB中排除新PDB。当从所有备用CDB中排除PDB时，PDB的数据文件未命名，并在所有备用CDB上标记为脱机。如果在备用数据库上找不到PDB的数据文件，则备用恢复不会停止。如果在创建PDB后实例化新的备用CDB，则必须在新备用CDB上明确禁用PDB以进行恢复。

通过将数据文件复制到正确的位置，使PDB联机并将其标记为已启用以进行恢复，可以在备用CDB上排除备用CDB上的PDB后启用PDB。

logging_clause

使用此子句指定在PDB中创建的表空间的缺省日志记录属性。 logging属性控制是否在重做日志文件（ LOGGING ）中记录某些DML操作（ NOLOGGING ）。默认为LOGGING 。

创建表空间时，可以通过指定CREATE TABLESPACE语句的logging_clause来覆盖缺省日志记录属性。

有关此子句的完整说明，请参阅logging_clause 。

create_file_dest_clause

默认情况下，新创建的PDB从根目录继承其Oracle Managed Files设置。如果root使用Oracle Managed Files，则PDB也使用Oracle Managed Files。PDB与root共享Oracle托管文件的相同基本文件系统目录，并且具有以PDB的GUID命名的自己的子目录。如果根不使用Oracle Managed Files，则PDB也不使用Oracle Managed Files。

此子句允许您覆盖默认行为。您可以为PDB启用或禁用Oracle Managed Files，并为PDB的文件指定不同的基本文件系统目录或Oracle ASM磁盘组。

  • 指定NONE以禁用PDB的Oracle Managed Files。

  • 指定directory_path_name或diskgroup_name以为PDB启用Oracle Managed Files。

    指定directory_path_name以指定PDB文件的基本文件系统目录。指定操作系统目录的完整路径名。该目录必须存在且Oracle进程必须对该目录具有适当的权限。单引号是必需的，结果是路径名称区分大小写。

    指定diskgroup_name以指定PDB文件的默认Oracle ASM磁盘组。

如果指定的值不是NONE ，则数据库会在PDB中使用SCOPE=SPFILE隐式设置DB_CREATE_FILE_DEST初始化参数。

create_pdb_clone

此子句使您可以通过将源克隆到目标PDB来创建新的PDB。源可以是本地CDB中的PDB，远程CDB中的PDB或非CDB。目标PDB是源的克隆。

如果源是本地CDB中的PDB，则可以插入或拔出源PDB。如果源是远程CDB中的PDB，则必须插入源PDB。

如果源是远程CDB中的非CDB或PDB，则包含目标PDB的源和CDB必须满足以下要求：

  • 它们必须具有相同的endian格式。

  • 它们必须具有兼容的字符集和国家字符集，这意味着：

    • 源字符集中的每个字符都可在本地CDB字符集中使用。

    • 源字符集中的每个字符在本地CDB字符集中具有相同的代码点值。

  • 它们必须安装相同的数据库选项集。

PDB中使用源非CDB或PDB的默认临时表空间的用户使用新PDB的默认临时表空间。在非CDB或PDB中使用非默认临时表空间的用户将继续在新PDB中使用相同的本地临时表空间。

您可以使用相同的命令克隆联合PDB或隔离的PDB。唯一的区别是您必须提供的密钥库密码用于不同的密钥库。

快照复制

只有在克隆PDB时才能指定SNAPSHOT COPY 。克隆非CDB时不支持此子句。源PDB可以位于本地CDB或远程CDB中。 SNAPSHOT COPY子句指示数据库使用存储快照克隆源PDB。这减少了创建克隆所需的时间，因为数据库不需要制作源数据文件的完整副本。

当您使用SNAPSHOT COPY子句创建源PDB的克隆并且CLONEDB初始化参数设置为FALSE ，源PDB文件的基础文件系统必须支持存储快照。此类文件系统包括Oracle自动存储管理群集文件系统（Oracle ACFS）和直接NFS客户端存储。

当您使用SNAPSHOT COPY子句创建源PDB的克隆并且CLONEDB初始化参数设置为TRUE ，源PDB的文件的基础文件系统可以是任何本地文件系统，网络文件系统（NFS）或群集已启用Direct NFS的文件系统。但是，只要存在任何克隆，源PDB就必须保持打开的只读模式。

Direct NFS Client使Oracle数据库可以直接访问网络连接存储（NAS）设备，而不是使用操作系统内核NFS客户端。如果PDB文件存储在Direct NFS客户端存储上，则必须满足以下附加要求：

  • 源PDB文件必须位于NFS卷上。

  • 存储凭据必须存储在透明数据加密密钥库中。

  • 存储用户必须具有在承载源PDB文件的卷上创建和销毁快照所需的权限。

  • 必须使用ADMINISTER KEY MANAGEMENT ADD SECRET SQL语句将凭据存储在密钥库中。

当您使用SNAPSHOT COPY子句创建源PDB的克隆时，只要存在任何克隆，以下限制将适用于源PDB：

  • 它不能被拔掉。

  • 它不能丢弃。

使用SNAPSHOT COPY子句创建的PDB克隆无法拔出。他们只能被丢弃。尝试拔出使用SNAPSHOT COPY子句创建的克隆会导致错误。

对于在Oracle Real Application Clusters（Oracle RAC）环境中使用SNAPSHOT COPY子句创建的PDB，必须装入必须访问PDB文件的每个节点。对于在Linux或UNIX平台上运行的Oracle RAC数据库，必须装入基础NFS卷。如果Oracle RAC数据库在Windows平台上运行并使用Direct NFS进行共享存储，则必须使用创建的卷export和mount条目更新所有节点上的oranfstab文件。

使用新的PDB GUID命名和标记存储克隆。您可以查询DBA_PDB_HISTORY视图的CLONETAG列以查看存储克隆的克隆标记。

pdb_refresh_mode_clause

REFRESH MODE子句仅在克隆PDB时适用。源PDB必须位于远程CDB中，即必须使用FROM src_pdb_name @ dblink子句指定源PDB。

此子句允许您指定PDB的刷新模式。您可以使用此子句创建可刷新的PDB 。源PDB中的更改可以手动或自动传播到可刷新的PDB。此操作称为刷新。您可以指定以下刷新模式：

  • MANUAL - 此模式允许您通过发出ALTER PLUGGABLE DATABASE REFRESH语句随时手动刷新可刷新的PDB。

  • EVERY refresh_interval MINUTES或HOURS - 此模式指示数据库在选定时间单位，分钟或小时的每个refresh_interval刷新可刷新的PDB。如果选择MINUTES ，则refresh_interval必须小于3000。如果选择HOURS ，则refresh_interval必须小于2000。此模式还允许您通过发出ALTER PLUGGABLE DATABASE REFRESH语句随时手动刷新PDB。

  • NONE - 如果指定此模式，则克隆PDB不是可刷新的PDB。数据库无法自动刷新PDB，您无法手动刷新PDB。如果指定此模式，则以后无法将PDB更改为可刷新的PDB。这是默认值。

只能在READ ONLY模式下打开可刷新的PDB。必须关闭可刷新的PDB才能进行刷新。如果在尝试执行手动刷新时未关闭，则会发生错误。如果在数据库尝试自动刷新时未关闭，则刷新将推迟到下一次计划刷新。

RELOCATE

使用此子句将PDB从一个CDB重定位到另一个CDB。数据库首先将源PDB克隆到目标PDB，然后删除源PDB。数据库还将与PDB关联的文件移动到新位置。此操作是以最短的停机时间重新定位PDB的最快方法。PDB的停机时间大约是将PDB文件从旧位置复制到新位置所需的时间。源PDB可以在READ WRITE模式下打开，并在重定位操作期间完全正常工作。

您可以使用AVAILABILITY关键字指定可用性级别。默认可用性为NORMAL 。如果指定AVAILABILITY MAX ，则执行其他操作以确保工作负载在源和目标之间的持久连接中平稳迁移。
