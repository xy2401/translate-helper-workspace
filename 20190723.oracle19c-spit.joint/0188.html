24.1.3 RESOURCE_VIEW和PATH_VIEW之间的区别

PATH_VIEW包括特定资源的所有路径名。RESOURCE_VIEW包含RESOURCE_VIEW的可能路径名之一 。PATH_VIEW还包括链接属性。为了获得更好的性能，请尽可能使用RESOURCE_VIEW ，而不是PATH_VIEW 。

图24-3说明了这种差异。

由于许多Internet应用程序只需要一个URL来访问资源，因此RESOURCE_VIEW广泛适用。

PATH_VIEW包含链接属性和资源属性，而RESOURCE_VIEW仅包含资源属性。

因为它处理多个路径的信息， PATH_VIEW访问可能会更慢。如果使用RESOURCE_VIEW那么数据库可以利用只需要一个路径的事实;索引可以做更少的工作来确定所有可能的路径。

注意：

使用RESOURCE_VIEW ，如果使用函数under_path或equals_path指定路径，则无论指定的路径是否是使用RESOURCE_VIEW通常与该资源一起显示的任意路径，该函数都会查找RESOURCE_VIEW 。

24.2使用RESOURCE_VIEW和PATH_VIEW的Oracle SQL函数

与公共视图RESOURCE_VIEW和PATH_VIEW一起使用的Oracle SQL函数包括under_path ， equals_path ， path和depth 。

  • UNDER_PATH SQL函数
    Oracle SQL函数under_path使用Oracle XML DB Repository的分层索引来返回特定路径下所有硬链接的路径。此索引旨在加速遍历路径时的访问（最常见的用法）。
  • EQUALS_PATH SQL函数
    使用Oracle SQL函数equals_path查找具有给定路径名的资源。它在功能上等同于深度限制为零的under_path 。
  • PATH SQL函数
    Oracle SQL函数path将给定pathname参数下的资源的相对路径名返回到函数under_path或equal_path 。
  • 深度SQL函数
    Oracle SQL函数depth返回指定起始路径下资源的文件夹深度。

24.7 Oracle XML DB存储库操作的性能指南

提供了用于提高资源创建和查询等存储库操作性能的指南。

包含大量资源的文件夹会对并发产生负面影响，尤其是在创建或删除许多资源时。根据经验，不要包含超过10,000个资源的文件夹。此经验限制基于数据库块大小和平均文件名长度。

如果批量创建资源，请至少每1,000个资源执行一次COMMIT操作。如果您经常提交或者您提交的次数少于每1,000次资源创建，性能就会受到负面影响。

创建文件资源时，该文件资源是已知XML模式的基于XML模式的文档，请将XML模式URL指定为PL / SQL函数DBMS_XDB_REPOS.createResource的参数。这节省了预先准备文档以确定XML模式。

Oracle XML DB使用配置文件xdbconfig.xml来配置系统和协议环境。此文件包含一个元素参数resource-view-cache-size ，它定义RESOURCE_VIEW缓存的动态内存中的大小。默认值为1048576 。

可以通过调整resource-view-cache-size来改进对RESOURCE_VIEW和PATH_VIEW的一些查询的性能。通常，缓存大小越大，查询越快。默认的resource-view-cache-size适用于大多数情况，但您可能希望在查询大量的RESOURCE_VIEW时扩大resource-view-cache-size元素。

以下元素的默认限制是软限制。超出这些限制时，系统会自动适应。

  • xdbcore-loadableunit-size - 此元素指示可装入单元（分区）可以以千字节为单位增长的最大大小。将分区读入内存或在使用新文档时构建分区时，将构建分区，直到达到最大大小。默认值为16 KB。

  • xdbcore-xobmem-bound - 此元素指示允许文档占用的最大内存（以KB为单位）。默认值为1024 KB。一旦文档超过此数量，就会换出一些可加载的单元（分区）。

96.1 DBMS_LOGMNR概述

Oracle LogMiner是Oracle数据库的一部分，使您可以通过SQL接口查询联机和归档的重做日志文件。DBMS_LOGMNR包提供了启动和停止LogMiner并指定感兴趣的重做日志文件所需的大多数工具。

对用户数据或数据库字典所做的所有更改都记录在Oracle重做日志文件中，以便可以执行数据库恢复操作。您可以利用重做日志文件中记录的数据来完成其他任务，例如：

  • 精确定位数据库的逻辑损坏（例如在应用程序级别发生的错误）可能已经开始

  • 确定在事务级别执行细粒度恢复时必须采取的操作。

  • 通过趋势分析进行性能调整和容量规划。

  • 跟踪在数据库上执行的任何数据操作语言（DML）和数据定义语言（DDL）语句，它们的执行顺序以及执行它们的人员。

指示V$LOGMNR_CONTENTS视图上的选择操作以跳过正在分析的重做日志文件中的任何损坏并继续处理。仅当重做日志文件中的块（而不是重做日志文件的标头）损坏时，此选项才有效。您应该检查V$LOGMNR_CONTENTS视图中的INFO列以确定LogMiner跳过的损坏块。当跳过重做日志文件中的损坏时， OPERATION列包含值CORRUPTED_BLOCKS ， STATUS列包含值1343.

DDL_DICT_TRACKING

如果正在使用的LogMiner字典是平面文件或重做日志文件，则LogMiner会在发生DDL事件时更新其内部字典。这可确保为构建LogMiner内部字典后修改的对象维护正确的SQL_REDO和SQL_UNDO信息。必须打开LogMiner连接的数据库。

此选项不能与DICT_FROM_ONLINE_CATALOG选项一起使用，并且在使用的LogMiner字典是在Oracle9 i之前提取到平面文件的字典时，不能使用此选项。

DICT_FROM_ONLINE_CATALOG

指示LogMiner使用当前在线数据库字典而不是平面文件中包含的LogMiner字典或正在分析的重做日志文件。

此选项不能与DDL_DICT_TRACKING选项一起使用。LogMiner连接的数据库必须与生成重做日志文件的数据库相同。

如果数据库中的表定义与重做日志文件中的表定义不匹配，则期望在V$LOGMNR_CONTENTS视图的STATUS列中看到值2 。

DICT_FROM_REDO_LOGS

如果设置，LogMiner希望在指定的重做日志文件中找到LogMiner字典。重做日志文件与指定DBMS_LOGMNR.ADD_LOGFILE过程或与DBMS_LOGMNR.START_LOGMNR程序。

NO_SQL_DELIMITER

如果设置，则SQL分隔符（分号）不会放在重构SQL语句的末尾。这对于打开游标然后执行重构语句的应用程序很有用。

NO_ROWID_IN_STMT

如果设置，则ROWID子句不包含在重构的SQL语句中。如果启用了补充日志记录，则重做日志文件可能已包含已修改行的逻辑唯一标识符。

使用此选项时，必须确保在源数据库中在适当级别启用了补充日志记录，并且感兴趣的表中不存在重复行。 LogMiner不保证逻辑行标识符的唯一性。

PRINT_PRETTY_SQL

如果设置，LogMiner将重构的SQL语句格式化以便于阅读。这些重构的SQL语句不可执行。

STRING_LITERALS_IN_STMT

如果设置，SQL_REDO和SQL_UNDO将使用文字表示数字，日期时间和间隔列类型。

要指定多个选项，请在它们之间使用加号（+）。例如：

EXECUTE DBMS_LOGMNR.START_LOGMNR（OPTIONS =>  -  DBMS_LOGMNR.DDL_DICT_TRACKING + DBMS_LOGMNR.DICT_FROM_REDO_LOGS）;

  • 关于开始和结束时间或SCN范围，请记住以下几点：

    • 如果既未指定startTime也未指定startScn参数，则LogMiner将设置startScn参数以使用包含最旧更改的重做日志文件中的最低SCN值。

    • 如果同时指定时间和SCN值，LogMiner将使用SCN值或忽略时间值。

    • 如果指定了起始和结束时间或SCN值，并且它们位于LogMiner重做日志文件列表中，则LogMiner将挖掘由这些值指示的日志。

    • 如果指定了不在LogMiner重做日志文件列表中的开始和结束时间或SCN值，并指定了DBMS_LOGMNR.START_LOGMNR ，则指定：

      • 对于startTime或startScn值为0 ，则LogMiner重做日志文件列表中的最低SCN将用作startScn

      • startTime或startScn值的非零数字，然后返回错误

      • 对于endTime或endScn值为0或非零数字，则LogMiner重做日志文件列表中的最高SCN将用作endScn

    • 如果指定了开始和结束时间或SCN值，并且在LogMiner重做日志文件列表中找不到它们，并指定了DBMS_LOGMNR.START_LOGMNR ，则指定：

      • 0表示startTime或startScn值，然后返回错误。

      • startTime或startScn值大于数据库的归档重做日志文件中的任何值，然后LogMiner开始在联机重做日志文件中进行挖掘。LogMiner将继续处理联机重做日志文件，直到它在从V$LOGMNR_CONTENTS视图返回行之前发现或超出请求的起始点。

      • endTime或endScn参数值，指示将来的时间或SCN，然后LogMiner在其挖掘时包含联机重做日志文件。查询V$LOGMNR_CONTENTS视图时，将在对数据库进行更改时从此视图返回行，并且在LogMiner看到超出请求的结束点的更改之前不会停止。

      • 0的endTime或endScn参数值，然后LogMiner的包括联机重做日志文件时，它的地雷。查询V$LOGMNR_CONTENTS视图时，将在对数据库进行更改时从此视图返回行，并且在您输入CTL + C或终止PL / SQL游标之前不会停止。

根据W3C的定义，“SOAP是一种用于在分散的分布式环境中交换信息的轻量级协议。它是一个基于XML的协议，由三部分组成：一个信封，用于定义用于描述消息中的内容以及如何处理消息的框架，一组用于表示应用程序定义的数据类型实例的编码规则，以及一个用于表示的约定远程过程调用和响应。“

SOAP消息的结构如下：

[SOAP消息（XML文档）[SOAP信封[SOAP标题？element *] [SOAP body（element * | Fault）？]]]

SOAP接口的Oracle C ++实现包括SOAP数据类型， SoapException方法， ConnectRef方法和MsgFactory方法。

31.1响应当前控制文件子集的丢失

如果永久性介质故障已损坏数据库的一个或多个控制文件且至少有一个当前控制文件未被介质故障损坏，请使用以下过程恢复数据库。

本节包含以下主题：

  • 将多路复用控制文件复制到默认位置

  • 将多路复用控制文件复制到非默认位置

31.1.1将多路复用控制文件复制到默认位置

如果包含丢失的控制文件的磁盘和文件系统完好无损，则只需将完整的控制文件复制到丢失的控制文件的位置即可。在这种情况下，您不必编辑CONTROL_FILES初始化参数。

要通过复制多路复用控制文件来替换损坏的控制文件：

  1. 如果实例仍在运行，请将其关闭：
    SQL> SHUTDOWN ABORT
  2. 纠正导致介质故障的硬件问题。如果无法快速修复硬件问题，请通过将损坏的控制文件还原到备用存储设备来继续进行数据库恢复，如“ 将多路复用控制文件复制到非默认位置 ”中所述 。
  3. 使用数据库当前控制文件的完整多路复用副本来复制损坏的控制文件。例如，要将bad_cf.f替换为good_cf.f ，您可以输入：
    ％cp /oracle/good_cf.f /oracle/dbs/bad_cf.f
  4. 启动一个新实例并装入并打开数据库。例如，输入：
    SQL> STARTUP

31.1.2将多路复用控制文件复制到非缺省位置

如果包含丢失的控制文件的磁盘和文件系统不完整，则无法将良好的控制文件复制到丢失的控制文件的位置。在这种情况下，您必须更改CONTROL_FILES初始化参数以指示缺少的控制文件的新位置。

要将控制文件还原到非默认位置：

  1. 如果实例仍在运行，请将其关闭：
    SQL> SHUTDOWN ABORT
  2. 如果无法更正导致介质故障的硬件问题，请将完整的控制文件复制到备用位置。

    例如，要将良好版本的control01.dbf复制到新磁盘位置，可以发出以下命令：

    ％cp /disk1/oradata/trgt/control01.dbf /new_disk/control01.dbf
  3. 编辑数据库的参数文件，以便 CONTROL_FILES参数反映所有控制文件的当前位置，并排除未还原的所有控制文件。

    假设初始化参数文件包含以下设置：

    CONTROL_FILES = '/磁盘1 / ORADATA / TRGT / control01.dbf'， '/ bad_disk / control02.dbf'

    您可以编辑CONTROL_FILES初始化参数，如下所示：

    CONTROL_FILES = '/磁盘1 / ORADATA / TRGT / control01.dbf'， '/ new_disk / control02.dbf'
  4. 启动一个新实例并装入并打开数据库。例如：
    SQL> STARTUP

查找跟踪文件和警报日志的最简单方法是运行以下SQL查询： SELECT NAME, VALUE FROM V$DIAG_INFO 。

31.2.1使用默认位置中的备份控制文件进行恢复

如果可能，将控制文件还原到其原始位置。这样，您就可以避免在初始化参数文件中指定新的控制文件位置。

要将备份控制文件还原到其默认位置：

  1. 如果实例仍在运行，请将其关闭：
    SQL> SHUTDOWN ABORT
  2. 纠正导致介质故障的硬件问题。
  3. 将备份控制文件还原到服务器参数文件或初始化参数文件中CONTROL_FILES参数中指定的所有位置。例如，如果/disk1/oradata/trgt/control01.dbf和/disk2/oradata/trgt/control02.dbf是服务器参数文件中列出的控制文件位置，则使用操作系统实用程序将备份控制文件还原到这些地点：
    ％cp /backup/control01.dbf /disk1/oradata/trgt/control01.dbf%cp/backup/control02.dbf /disk2/oradata/trgt/control02.dbf
  4. 启动新实例并装入数据库。例如，输入：
    SQL> STARTUP MOUNT
  5. 通过USING BACKUP CONTROLFILE子句执行RECOVER命令开始恢复。如果执行不完全恢复，请指定UNTIL CANCEL 。例如，输入：
    SQL> RECOVER DATABASE使用备份控制文件直到取消
  6. 应用提示的存档日志。如果您随后收到另一条消息，指出缺少所需的归档日志，则可能意味着必要的重做记录位于联机重做日志中。当实例失败时，在联机日志中找到未归档的更改时，可能会发生这种情况。

    例如，假设您看到以下内容：

    ORA-00279：更改55636生成于11/08/2002 16:59:47线程1 ORA-00289：建议：/oracle/work/arc_dest/arcr_1_111.arc ORA-00280：更改55636为线程1按顺序＃111指定日志：{<RET> =建议|文件名|自动|取消}

    您可以指定在线重做日志的名称，然后按Enter键（您可能需要尝试几次，直到找到正确的日志）：

    ORACLE_HOME /oradata/redo01.dbf已应用日志。媒体恢复完成。

    如果无法访问在线日志，则可以取消恢复而不应用它们。如果所有数据文件都是最新的，并且如果需要在联机日志中重做进行恢复，则无法在不应用联机日志的情况下打开数据库。如果无法访问联机日志，则必须使用“ 重新创建控制文件 ”中描述的过程重新创建控制文件 。

  7. 完成恢复后，使用RESETLOGS选项打开数据库：
    SQL> ALTER DATABASE OPEN RESETLOGS;

31.2.3使用备份控制文件通过添加的数据文件进行恢复

如果使用备份控制文件的数据库恢复通过CREATE TABLESPACE或ALTER TABLESPACE ADD DATAFILE操作前滚，则数据库会在为添加的文件应用重做记录时停止恢复，并允许您确认文件名。

例如，假设发生以下事件序列：

  1. 您备份数据库。

  2. 您创建一个包含以下数据文件的表空间： /disk1/oradata/trgt/test01.dbf和/disk1/oradata/trgt/test02.dbf 。

  3. 您可以通过CREATE TABLESPACE操作还原备份控制文件并执行介质恢复。

应用CREATE TABLESPACE重做数据时，您可能会看到以下错误：

ORA-00283：由于错误而取消恢复会话ORA-01244：通过介质恢复添加到控制文件的未命名数据文件ORA-01110：数据文件11：'/ disk1 / odata / strgt / test02.dbf'ORA-01110：数据文件10：'/ disk1 / oddata / strgt / test01.dbf'

要通过ADD DATAFILE操作恢复：

  1. 查看通过查询V$DATAFILE 。添加的文件，如下例所示：
    SELECT FILE＃，NAME FROM V $ DATAFILE; FILE＃NAME --------------- ---------------------- 1 / disk1 / oradata / trgt / system01。 dbf。。。10 / disk1 / oradata / trgt / UNNAMED00001 11 / disk1 / oradata / trgt / UNNAMED00002

    • 打开alert_ SID .log ，其中包含有关每个未命名文件的原始文件位置的消息。

    • 从错误消息和V$DATAFILE导出每个未命名文件的原始文件位置：每个未命名文件对应于错误消息中具有相同文件号的文件。

  3. 发出ALTER DATABASE RENAME FILE语句以重命名数据文件。例如，输入：
    ALTER DATABASE RENAME FILE'/ db / UNNAMED00001'to'/disk1/oradata/trgt/test01.dbf'; ALTER DATABASE RENAME FILE'/ db / UNNAMED00002'to'/disk1/oradata/trgt/test02.dbf';
  4. 通过发出恢复声明继续恢复。例如：
    使用备份控制文件直到取消恢复自动数据库

31.2.4使用备份控制文件恢复只读表空间

如果您在只读或慢速介质上有只读表空间，那么在使用“只读”或“慢速介质”进行恢复时可能会遇到错误或性能不佳 USING BACKUP CONTROLFILE选项。当备份控制文件指示在备份控制文件时读/写表空间时，会出现这种情况。在这种情况下，介质恢复可能会尝试写入文件。对于只读介质，数据库会发出错误消息，指出它无法写入文件。对于慢速介质，例如由磁带备份的分层存储系统，性能可能会受到影响。

要避免这些问题，请使用当前控制文件而不是备份来恢复数据库。如果必须使用备份控制文件，那么如果只读表空间没有出现介质故障，也可以避免此问题。使用备份控制文件时，您可以使用以下备选方法来恢复只读和慢速介质：

  • 在使用备份控制文件进行恢复之前，将只读表空间中的数据文件脱机，然后在介质恢复后将文件联机。

  • 使用正确版本的控制文件进行恢复。如果恢复完成时表空间是只读的，则控制文件备份必须来自表空间为只读时。同样，如果恢复后表空间是读/写，则控制文件必须是从表空间读/写时开始。

31.4备份不可用时重新创建数据文件

  • 创建原始数据文件后写入的所有归档日志文件都可用

  • 控制文件包含损坏文件的名称（即，控制文件是最新的，或者是在将损坏的数据文件添加到数据库后进行的备份）

    注意：

    您不能使用ALTER DATABASE语句的CREATE DATAFILE子句为SYSTEM表空间重新创建任何数据文件，因为必需的重做不可用。

要重新创建用于恢复的数据文件：

  1. 创建一个空数据文件以替换没有相应备份的损坏数据文件。例如，假设数据文件/disk1/oradata/trgt/users01.dbf已损坏，并且没有可用的备份。以下语句在disk2上重新创建原始数据文件（大小相同）：
    ALTER DATABASE CREATE DATAFILE'/disk1/oradata/trgt/users01.dbf'AS'/disk2/users01.dbf';

    此语句创建一个与丢失文件大小相同的空文件。数据库查看控制文件和数据字典中的信息以获取大小信息。旧数据文件重命名为新数据文件。

  2. 在空数据文件上执行介质恢复。例如，输入：
    RECOVER DATAFILE'/disk2/users01.dbf'
  3. 创建原始数据文件后写入的所有归档日志必须在恢复期间应用于丢失数据文件的新空版本。

31.5恢复NOLOGGING表和索引

您可以使用。创建表和索引 CREATE TABLE AS SELECT语句。您还可以指定数据库使用NOLOGGING选项创建它们。将表或索引创建为NOLOGGING ，数据库不会为操作生成重做日志记录。因此，即使在ARCHIVELOG模式下运行，也无法恢复使用NOLOGGING创建的对象。

注意：

如果您无法承受丢失使用NOLOGGING创建的表或索引，请在创建不可恢复的表或索引后进行备份。

请注意，执行介质恢复时，某些表或索引是正常创建的，而其他表是使用NOLOGGING选项创建的，否则NOLOGGING对象会被RECOVER操作标记为逻辑损坏。任何访问不可恢复对象的尝试都会返回 ORA-01578错误消息。删除NOLOGGING对象并根据需要重新创建它们。

因为可以使用NOLOGGING选项创建表，然后在该表上使用LOGGING选项创建索引，所以在执行介质恢复后，索引不会被标记为逻辑损坏。该表是不可恢复的（因此在恢复后标记为损坏），因此索引指向损坏的块。必须删除索引，必要时必须重新创建表和索引。

31.6恢复可传输表空间

Oracle数据库的可传输表空间功能使用户能够将一组表空间从一个数据库传输到另一个数据库。将表空间传输到数据库就像创建一个包含已加载数据的表空间。由于以下原因，使用此功能通常是一个优势：

  • 它比使用Data Pump Export或SQL * Loader实用程序更快，因为它只涉及复制数据文件和集成元数据

  • 您可以使用它来移动索引数据，从而避免重建索引的必要性

要恢复可传输的表空间，请使用以下过程：

  1. 如果数据库已打开，则使传输的表空间脱机。例如，如果要恢复users表空间，请发出以下语句：

    ％cp /backup/users.dbf $ ORACLE_HOME / oradata / trgt / users01.dbf
  3. 像往常一样恢复表空间。例如，输入：
    RECOVER TABLESPACE用户

  2. 重新初始化损坏的日志组。例如，要清除重做日志组2 ，请发出以下语句：

31.7.2.1.2清除不活动的，未归档的重做

清除尚未归档的重做日志允许在不归档的情况下重用它。如果在日志中的最后一次更改之前启动备份，则此操作会使备份无法使用，除非在日志中第一次更改之前文件已脱机。因此，如果您需要已清除的日志文件来恢复备份，则无法恢复该备份。清除尚未归档的重做日志会阻止由于缺少日志而从备份中完全恢复。

要清除尚未归档的非活动联机重做日志组，请执行以下操作：

  1. 如果数据库已关闭，则启动新实例并装入数据库：
    SQL> STARTUP MOUNT
  2. 使用UNARCHIVED关键字清除日志。

    例如，要清除日志组2 ，请发出以下SQL语句：

    SQL> ALTER DATABASE CLEAR UNARCHIVED LOGFILE GROUP 2;

    如果存在需要清除日志使其联机的脱机数据文件，则需要关键字UNRECOVERABLE DATAFILE 。必须删除数据文件，因为正在清除使数据文件联机所需的重做日志，并且没有它的副本。例如，输入：

    SQL> ALTER DATABASE CLEAR UNARCHIVED LOGFILE GROUP 2 UNRECOVERABLE DATAFILE;
  3. 使用操作系统实用程序立即备份数据库中的所有数据文件，以便您可以使用备份进行完全恢复，而无需依赖已清除的日志组。例如，输入：
    ％cp /disk1/oracle/dbs/*.dbf / disk2 / backup

  • 通过在当前配置的重做日志文件名下重新创建重做日志文件，将其重新定位到备用介质上

  • 重新使用当前配置的日志文件名重新创建重做日志文件，因为名称本身无效或不可用（例如，由于介质故障）

在这些情况下， ALTER DATABASE CLEAR LOGFILE语句（在收到I / O错误之前）成功通知控制文件正在清除日志并且不需要存档。在CLEAR LOGFILE语句尝试创建新重做日志文件并向其写入零的步骤中发生I / O错误。这个事实反映在V$LOG.CLEARING_CURRENT 。

C.2数据类型的实现

本节讨论DATE ， TIMESTAMP和浮点数据类型。

DATE和TIMESTAMP

Oracle DATE和TIMESTAMP数据类型的语义与具有相同名称的ODBC数据类型不完全对应。Oracle DATE数据类型包含日期和时间信息。SQL_DATE数据类型仅包含日期信息。Oracle TIMESTAMP数据类型还包含日期和时间信息，但它在小数秒内具有更高的精度。Oracle ODBC驱动程序将Oracle DATE和TIMESTAMP列的数据类型报告为SQL_TIMESTAMP以防止信息丢失。同样，Oracle ODBC驱动程序将SQL_TIMESTAMP参数绑定为Oracle TIMESTAMP值。

C.10启用EXEC语法

如果SQL Server EXEC语句的语法可以很容易地转换为等效的Oracle过程调用而不需要对其进行任何更改，那么如果启用此选项，Oracle ODBC Driver可以将其翻译。

SQL Server过程的完整名称最多包含四个标识符：

  • 服务器名称

  • 数据库名称

  • 所有者名称

  • 程序名称

名称的格式为：

[[[服务器] [数据库]。] [OWNER_NAME]。] PROCEDURE_NAME

在将Microsoft SQL Server数据库迁移到Oracle数据库期间，每个SQL Server过程或函数的定义都将转换为其等效的Oracle数据库语法，并在Oracle数据库的模式中定义。迁移过程通常通过以下方式之一重新组织（并在模式中创建）：

  • 所有过程都迁移到一个模式（默认选项）。

  • 在一个SQL Server数据库中定义的所有过程都将迁移到使用该数据库名称命名的模式。

  • 一个用户拥有的所有过程都将迁移到使用该用户名命名的模式。

要支持这三种组织迁移过程的方法，可以指定其中一个模式名称选项来转换过程名称。已转换的Oracle过程调用中的对象名称不区分大小写。

C.12.2 ODBC API中的Unicode支持

ODBC API使用W和A后缀约定支持Unicode和ANSI入口点。ODBC应用程序开发人员不显式调用带后缀的入口点。使用UNICODE和_UNICODE预处理程序定义编译的ODBC应用程序会生成相应的调用。例如，对SQLPrepare的调用将编译为SQLPrepareW 。

C数据类型SQL_C_WCHAR已添加到ODBC接口，以允许应用程序指定输入参数编码为Unicode或请求以Unicode返回的列数据。宏SQL_C_TCHAR对于必须同时构建为Unicode和ANSI的应用程序非常有用。 SQL_C_TCHAR宏编译为Unicode应用程序的SQL_C_WCHAR和ANSI应用程序的SQL_C_CHAR 。

C.12.3 SQLGetData性能

SQLGetData函数允许ODBC应用程序指定在获取数据后接收列的数据类型。OCI要求Oracle ODBC Driver在获取之前指定数据类型。在这种情况下，Oracle ODBC驱动程序使用有关列的数据类型的信息（如数据库中所定义）来确定如何最好地默认通过OCI获取列。

获取数据时使用Oracle Database Client的默认编码。但是，ODBC应用程序可能会覆盖此缺省值，并通过将列或参数绑定为WCHAR数据类型将数据作为Unicode获取。

C.13.1通用ODBC编程指南

应用以下编程指南以提高ODBC应用程序的性能：

  • 如果应用程序经常连接和断开数据源，则启用连接池。与重新建立连接相比，重用池化连接非常有效。

  • 最小化必须准备语句的次数。在可能的情况下，使用绑定参数使语句可以重用于不同的参数值。准备一次语句并多次运行它比为每个SQLExecute准备语句要高效得多。

  • 不要在SELECT语句中包含您知道应用程序无法检索的列;特别是LONG专栏。由于数据库服务器协议的性质，Oracle ODBC驱动程序必须获取LONG列的全部内容（如果它包含在SELECT语句中），无论应用程序是绑定列还是执行SQLGetData操作。

  • 如果您正在执行不更新数据源的事务，请将ODBC SQLSetConnectAttr函数的SQL_ATTR_ACCESS_MODE属性设置为SQL_MODE_READ_ONLY 。

  • 如果未使用ODBC转义子句，则将ODBC SQLSetConnectAttr函数或ODBC SQLSetStmtAttr函数的SQL_ATTR_NOSCAN属性设置为true。

  • 使用ODBC SQLFetchScroll函数而不是ODBC SQLFetch函数从具有大量行的表中检索数据。

C.13.2数据源配置选项

本节讨论以下ODBC数据源配置选项的性能影响：

  • 启用结果集

    此选项支持从过程调用返回结果集（例如， RefCursor ）。默认设置是启用结果集的返回。

    Oracle ODBC驱动程序必须查询数据库服务器以确定过程及其数据类型的参数集，以确定是否存在任何RefCursor参数。在第一次准备和执行任何过程时，该查询引起额外的网络往返。

  • 启用LOB

    此选项支持插入和更新LOB。默认设置已启用。

    Oracle ODBC驱动程序必须查询数据库服务器以确定INSERT或UPDATE语句中每个参数的数据类型，以确定是否存在任何LOB参数。第一次准备并运行INSERT或UPDATE时，此查询会产生额外的网络往返。

    也可以看看：

    有关LOB的更多信息，请参见Oracle Database SecureFiles和Large Objects Developer's Guide

    注意：

    LOB数据压缩使您能够压缩SecureFiles以获得磁盘，输入输出和重做日志记录。这可以降低成本，因为压缩可以最有效地利用空间并提高SecureFiles的性能，因为压缩会降低输入输出和重做日志记录。

    LOB数据加密提供增强的数据库安全性。虽然加密数据可用于随机读写，但数据更安全。

    数据压缩和加密会消耗一些额外的内存。

  • 将TIMESTAMP绑定为DATE

    将SQL_TIMESTAMP参数绑定为适当的Oracle数据库数据类型。如果此选项设置为TRUE ，则SQL_TIMESTAMP将绑定为Oracle DATE数据类型。如果此选项设置为FALSE ，则SQL_TIMESTAMP将绑定为Oracle TIMESTAMP数据类型，这是默认值。

  • 启用关闭游标

    ODBC函数的SQL_CLOSE选项SQLFreeStmt应该使用语句关闭关联的游标并丢弃所有挂起的结果。应用程序可以通过再次运行语句重新打开游标，而无需再次执行SQLPrepare。这种情况的典型情况是应用程序需要空闲一段时间但重新使用相同的SQL语句。当应用程序空闲时，它可能想要释放任何相关的服务器资源。

    分层Oracle ODBC驱动程序的OCI不支持关闭游标的功能。因此，默认情况下， SQL_CLOSE选项在Oracle ODBC驱动程序中无效。游标和相关资源在数据库上保持打开状态。

    启用此选项会导致关联的游标在数据库服务器上关闭。但是，这会导致SQL语句的解析上下文丢失。ODBC应用程序可以在不调用SQLPrepare情况下再次运行该语句。但是，在内部，Oracle ODBC Driver必须准备并运行该语句。启用此选项会对准备语句一次并重复运行的应用程序产生严重的性能影响。

    只有在需要释放服务器上的资源时才应启用此选项。

  • 获取缓冲区大小

    将odbc.ini文件中的Fetch Buffer Size（ FetchBufferSize ）设置为以字节为单位指定的值。此值是确定Oracle ODBC驱动程序从Oracle数据库一次预取多少行数据到客户端缓存所需的内存量，而不管应用程序在单个查询中请求的行数，从而提高性能。

    通常一次获取少于20行数据的应用程序的响应时间有所改善，特别是在慢速网络连接或高负载服务器上。将此设置得太高会对响应时间产生负面影响或消耗大量内存。默认值为64,000个字节。您应该为应用程序选择最佳值。

    当存在LONG和LOB数据类型时，Oracle ODBC驱动程序预取的行数不是由获取缓冲区大小决定的。包含LONG和LOB数据类型可以最大限度地降低性能，并可能导致过多的内存使用。Oracle ODBC驱动程序忽略了获取缓冲区大小，并且仅在存在LONG和LOB数据类型时预取了一定数量的行。

C.13.3 DATE和TIMESTAMP数据类型

如果在WHERE子句中使用数据库中的DATE列并且该列具有索引，则可能会对性能产生影响。例如：

SELECT * FROM EMP WHERE HIREDATE =？

在此示例中， HIREDATE列上的索引可用于使查询快速运行。但是，由于HIREDATE是DATE值而Oracle ODBC驱动程序将参数值作为TIMESTAMP ，因此Oracle数据库的查询优化器必须应用转换函数。为了防止不正确的结果（如果参数值具有非零小数秒可能会发生），优化程序将转换应用于HIREDATE列，从而产生以下语句：

SELECT * FROM EMP WHERE TO_TIMESTAMP（HIREDATE）=？

但是，这会禁止在HIREDATE列上使用索引。相反，服务器执行表的顺序扫描。如果表有很多行，那么这可能需要很长时间。作为这种情况的解决方法，Oracle ODBC驱动程序具有将TIMESTAMP绑定为DATE的连接选项。启用此选项后，Oracle ODBC驱动程序将SQL_TIMESTAMP参数绑定为Oracle DATE数据类型而不是Oracle TIMESTAMP数据类型。这使查询优化器能够使用DATE列上的任何索引。

注意：

此选项仅适用于Microsoft Access或将DATE列绑定为TIMESTAMP列的其他类似程序。当存在实际的TIMESTAMP列或可能发生数据丢失时，不应使用它。Microsoft Access使用选择的任何列作为主键来运行此类查询。

C.14错误消息

发生错误时，Oracle ODBC驱动程序返回本机错误号， SQLSTATE （ODBC错误代码）和错误消息。驱动程序从驱动程序检测到的错误和Oracle数据库返回的错误中获取此信息。

本机错误

对于数据源中发生的错误，Oracle ODBC驱动程序返回Oracle数据库返回给它的本机错误。当Oracle ODBC驱动程序或驱动程序管理器检测到错误时，Oracle ODBC驱动程序返回零的本机错误。

SQLSTATE

对于数据源中发生的错误，Oracle ODBC Driver将返回的本机错误映射到适当的SQLSTATE 。当Oracle ODBC驱动程序或驱动程序管理器检测到错误时，它会生成相应的SQLSTATE 。

错误信息

对于数据源中发生的错误，Oracle ODBC Driver会根据Oracle数据库返回的消息返回错误消息。对于Oracle ODBC驱动程序或驱动程序管理器中发生的错误，Oracle ODBC驱动程序会根据与SQLSTATE关联的文本返回错误消息。
