9插入，更新和加载JSON数据概述

您可以使用标准数据库API在Oracle数据库中插入或更新JSON数据。您可以通过创建将其公开给数据库的外部表，直接处理文件系统文件中包含的JSON数据。您可以使用JSON Merge Patch更新JSON文档。

使用标准数据库API插入或更新JSON数据

由于使用标准SQL数据类型存储JSON数据，因此用于插入或更新VARCHAR2和大对象（LOB）列的所有标准数据库API都可用于包含JSON文档的列。对于这些API，存储的JSON文档只不过是一串字符。

您通过使用SQL条件指定JSON列必须仅包含格式良好的JSON数据is json作为检查约束。数据库处理此检查约束与任何其他检查约束相同 - 它强制执行有关列内容的规则。因此，使用包含JSON文档的VARCHAR2 ， BLOB或CLOB类型的列与使用该类型的任何其他列没有什么不同。

对JSON列中的文档进行更新操作需要替换整个文档。您可以对JSON文档进行细粒度的修改，但是当您需要将更改保存到磁盘时，将编写整个更新的文档。

如果列的数据类型为VARCHAR2或CLOB ，则将JSON文档插入JSON列很简单 - 请参阅示例4-2 。更新这样的列也是如此。

但是，如果使用SQL * Plus等命令行工具将数据插入BLOB类型的JSON列中，或者更新此类数据，则必须将JSON数据正确转换为二进制格式 。例9-1是其部分例子。它假设表my_table有一个JSON列， json_doc ，它使用BLOB存储。

使用外部表来处理文件系统文件中的JSON数据

通过外部表，可以轻松访问作为文件系统中的单独文件存储的JSON文档。每个文件都可以作为外部表中的一行公开给Oracle数据库。外部表还可以提供对NoSQL数据库生成的转储文件内容的访问。您可以使用外部JSON文档表实际上直接查询文件系统文件中的数据 。如果您只需要在一次性操作中处理来自所有文件的数据，这将非常有用。

但是，如果您需要对文档进行多次查询，特别是如果不同的查询从外部表的不同行（不同的文档）中选择数据，那么为了获得更好的性能，请考虑将数据从外部表复制到普通的数据库表中，使用INSERT作为SELECT语句 - 请参见例10-4 。将JSON数据加载到普通表的JSON列后，您可以索引内容，然后您可以以重复，有选择的方式有效地查询数据。

使用JSON合并修补程序更新JSON文档

您可以使用Oracle SQL函数json_mergepatch或PL / SQL对象类型方法json_mergepatch()来更新JSON文档的特定部分。在这两种情况下，您都提供了一个JSON Merge Patch文档，该文档以声明方式指定对指定的JSON文档所做的更改。JSON Merge Patch是IETF标准。

概观

olsnodes命令提供参与集群的所有节点的节点列表和其他信息。

您可以使用此命令快速检查群集是否正常运行，并且所有节点都已注册为群集成员。此命令还提供了一种获取节点编号的简便方法。

操作说明

使用信息

当使用-n all选项时，Cluster Verification Utility（CLUVFY）使用此命令获取节点名称列表。

此命令实用程序位于$ORA_CRS_HOME/bin目录中。如果启动了CRS守护程序，则只能使用此命令。

特权和安全

您可以以root用户，安装Oracle Clusterware的用户或安装Oracle数据库的用户身份运行此命令。

OLSNODES命令摘要

olsnodes命令不使用关键字，但接受一个或多个选项。表C-1中描述了可用选项。

句法

olsnodes [[-n] [-i] [-s] [-t] [node_name | -l [-p]] | [-c]] [-a] [-g] [-v]

如果发出不带任何命令参数的olsnodes命令，该命令将返回集群中节点的列表：

[root @ node1] #olsnodes node1 node2 node3 node4

    在数据库安装期间，您可以选择文件系统或Oracle ASM作为Oracle数据库文件的存储机制。同样，您还可以选择文件系统或Oracle ASM作为恢复文件的存储机制。

  2. 选择要用于您创建的每个Oracle ASM磁盘组的Oracle ASM冗余级别。

    您为Oracle ASM磁盘组选择的冗余级别确定Oracle ASM如何镜像磁盘组中的文件，并确定所需的磁盘数和磁盘空间量，如下所示：

    • 外部冗余

      此选项不允许Oracle ASM镜像磁盘组的内容。Oracle建议您在磁盘组包含提供其自身数据保护的设备（如RAID设备）或数据库不需要不间断访问数据时选择此冗余级别。

    • 正常冗余

      为了优化正常冗余磁盘组的性能和可靠性，默认情况下，Oracle ASM对数据文件使用双向镜像，对控制文件使用三向镜像。此外，您还可以为磁盘组中的各个文件选择镜像特征。

      如果使用双向镜像，则正常冗余磁盘组至少需要两个故障组（或两个磁盘设备）。普通冗余磁盘组中的有效磁盘空间是其所有设备的磁盘空间总和的一半。

      对于大多数安装，Oracle建议您使用常规冗余磁盘组。在Oracle Exadata上，Oracle建议您使用高冗余磁盘组来增强防止故障的保护。

    • 高冗余

      默认情况下，磁盘组的内容是三向镜像的。要创建具有高冗余的磁盘组，必须至少指定三个故障组（至少三个设备）。

      虽然高冗余磁盘组提供高级别的数据保护，但在决定使用此冗余级别之前，必须考虑更高的额外存储设备成本。

    • Flex冗余

      Flex冗余磁盘组是一种新的磁盘组类型，具有灵活的文件冗余，镜像拆分和冗余更改等功能。Flex磁盘组可以将具有不同冗余要求的文件合并到一个磁盘组中。它还为数据库提供了更改其文件冗余的功能。

      对于数据库数据，您可以选择不镜像（不受保护），双向镜像（镜像）或三向镜像（高）。Flex冗余磁盘组至少需要三个磁盘设备（或三个故障组）。

  3. 确定数据库文件和恢复文件所需的磁盘空间总量。

    如果系统上正在运行Oracle ASM实例，则可以使用现有磁盘组来满足这些存储要求。如有必要，可以在数据库安装期间将磁盘添加到现有磁盘组。

    有关Oracle ASM磁盘空间要求，请参阅“Oracle 数据库安装指南 ”中的“Linux上的Oracle ASM磁盘空间要求”。

    • 磁盘设备必须由执行网格安装的用户拥有。

      请咨询系统管理员，以确定Oracle ASM使用的磁盘是否在存储级别进行镜像。如果是，请选择外部以获得冗余。如果磁盘未在存储级别进行镜像，则为冗余选择“正常”。

    • 每个Oracle ASM磁盘都分为分配单元（AU）。分配单元是磁盘组内的基本分配单位。您可以从1,2,4,8,16,32或64 MB中选择AU Size值，具体取决于特定的磁盘组兼容级别。对于Flex磁盘组，默认值为4 MB，对于所有其他磁盘组类型，默认值为1 MB。在工程系统上，默认值为4 MB。

    • 从版本12.2开始，您可以在Oracle Grid Infrastructure安装期间在“ 创建ASM磁盘组”屏幕中指定自定义故障组。

    • 您还可以使用GUI工具ASMCA，命令行工具asmcmd或SQL命令在安装后定义自定义故障组。

    • 如果定义自定义故障组，则对于仅包含数据库文件的故障组，必须为正常冗余磁盘组指定至少两个故障组，为高冗余磁盘组指定三个故障组。

  5. 如果您确定系统上不存在合适的磁盘组，请安装或识别要添加到新磁盘组的相应磁盘设备。

    在识别适当的磁盘设备时，请遵循以下准则：

    • 磁盘设备必须由执行Oracle Grid Infrastructure安装的用户拥有。

    • Oracle ASM磁盘组中的所有设备必须具有相同的大小并具有相同的性能特征。

    • 不要将单个物理磁盘上的多个分区指定为磁盘组设备。Oracle ASM希望每个磁盘组设备都位于单独的物理磁盘上。

    • 尽管您可以将逻辑卷指定为Oracle ASM磁盘组中的设备，但Oracle不建议使用它们，因为它会增加Oracle ASM不必要的复杂性。 Oracle建议，如果选择使用逻辑卷管理器，则使用逻辑卷管理器表示单个逻辑单元号（LUN），而不进行条带化或镜像，以便最大限度地减少对附加存储层的存储性能的影响。

注意：

JIT编译器取代了早期版本的Oracle数据库中使用的编译器。

10.1.1 Oracle JVM JIT概述

从Oracle 11g第 1版（11.1）开始，有一个用于Oracle JVM环境的JIT编译器。用于Oracle JVM的JIT编译器可以更快地执行，因为它可以在没有外部机制的情况下管理代码的失效，重新编译和存储。基于动态收集的分析数据，此编译器透明地选择Java方法来编译本机机器代码，并动态地使它们可用于运行Java会话。此外，编译器可以利用Oracle JVM的类解析模型，可选择在数据库调用，会话或实例之间保留已编译的Java方法。当已知在语义上Java代码没有改变时，这种持久性避免了跨会话或实例的不必要的重新编译的开销。

JIT编译器由一个名为的新布尔值初始化参数控制 java_jit_enabled 。当使用java_jit_enabled参数值为true大量使用的Java方法时，JIT编译器会自动将Java方法编译为本机代码，并使其可供实例中的所有会话使用。将java_jit_enabled参数设置为true还会导致进一步停止JIT编译，并还原任何已编译的方法以进行解释。VM会在必要时自动重新编译Java方法的本机代码，例如在重新解析包含的Java类之后。

注意：

在Linux上，Oracle JVM JIT使用需要访问/dev/shm目录的POSIX共享内存。/dev/shm目录的类型应为tmpfs ，您必须按如下方式挂载此目录：

  • 使用rw和execute权限设置它

  • 没有设置noexec或nosuid

如果未使用正确的安装选项，则在安装数据库期间可能会发生以下故障：

ORA-29516：Aurora断言失败：joez.c断言失败：方法java / lang / Object。<init>的批量加载失败; shm对象空间不足

JIT编译器在实例的单个后台进程中作为MMON从属运行。因此，当JIT编译器正在运行并主动编译方法时，您可能会看到此后台进程消耗的CPU和内存资源等同于活动用户Java会话。

10.1.2 JIT编译的优点

以下是使用JIT编译优于早期版本的Oracle数据库中使用的编译技术的优点：

  • JIT编译透明地工作

  • JIT编译可加快Java类的性能

  • 当已知语义上的Java代码没有改变时，JIT存储的编译代码避免了跨会话或实例重新编译Java程序。

  • JIT编译不需要C编译器

  • JIT编译消除了一些数组边界检查

  • JIT编译消除了块内的常见子表达式

  • JIT编译消除了空方法

  • JIT编译定义了局部变量的寄存器分配区域

  • JIT编译消除了流量分析的需要

  • JIT编译限制内联代码

10.1.3 Oracle Database 11g中引入的方法

从11g版本1（11.1）开始， DBMS_JAVA包已经使用以下新的公共方法进行了增强，以提供Java入口点来控制同步方法编译并恢复到解释的方法执行：

set_native_compiler_option

此过程将本机编译器选项设置为当前架构的指定值。如果不允许optionName给出的选项具有重复值，则忽略该值。 

PROCEDURE set_native_compiler_option（optionName VARCHAR2，value VARCHAR2）;

unset_native_compiler_option

此过程取消设置当前架构的本机编译器选项/值对。如果不允许optionName给出的选项具有重复值，则忽略该值。 

PROCEDURE unset_native_compiler_option（optionName VARCHAR2，value VARCHAR2）;

compile_class

此函数编译由当前模式中的classname标识的类定义的所有方法。它返回成功编译的方法数。如果该类不存在，则会发生ORA-29532 (Uncaught Java exception) 。 

FUNCTION compile_class（classname VARCHAR2）返回NUMBER;

uncompile_class

此函数取消编译由当前模式中的classname标识的类定义的所有方法。它返回成功未编译的方法数。如果参数permanentp的值非零，则将这些方法标记为永久动态不可编译。否则，它们有资格进行将来的动态重新编译。如果该类不存在，则会发生ORA-29532 (Uncaught Java exception) 。

注意：

Oracle强烈建议您以自动撤消管理模式运行数据库，而不是使用回滚段。除非必须这样做才能与早期版本的Oracle数据库兼容，否则请勿使用回滚段。有关自动撤消管理的信息，请参阅Oracle数据库管理员指南 。

目的

本节假定您的数据库以回滚撤消模式运行（ UNDO_MANAGEMENT初始化参数设置为MANUAL或根本不设置）。如果数据库以自动撤消模式运行（ UNDO_MANAGEMENT初始化参数设置为AUTO ，这是默认值），则用户创建的回滚段无关紧要。

语义

ROLLBACK_SEGMENT

指定现有回滚段的名称。

线上

指定ONLINE以使回滚段联机。创建回滚段时，它最初处于脱机状态，不可用于事务。此子句使回滚段在线，使其可用于您的实例的事务。使用初始化参数ROLLBACK_SEGMENTS启动实例时，也可以在线引入回滚段。

也可以看看：

离线

指定OFFLINE以使回滚段脱机。

  • 如果回滚段不包含回滚活动事务所需的任何信息，则Oracle数据库会立即使其脱机。

  • 如果回滚段确实包含活动事务的信息，则数据库使回滚段不可用于将来的事务，并在提交或回滚所有活动事务后使其脱机。

当回滚段处于脱机状态时，任何实例都可以将其联机。

要查看回滚段是联机还是脱机，请查询数据字典视图DBA_ROLLBACK_SEGS STATUS列。联机回滚段的值为IN_USE 。脱机回滚段的值为AVAILABLE 。

您无法使SYSTEM回滚段脱机。

3.6处理缺失值

Oracle Data Mining区分 稀疏数据和包含的数据 随机缺失值 。后者意味着某些属性值未知。疏另一方面，数据包含假设已知的值，尽管它们未在数据中表示。

稀疏数据的典型示例是市场篮子数据。在数百或数千种可用物品中，只有少数存在于个别情况（篮子或交易）中。所有项目值都是已知的，但它们并非都包含在购物篮中。当前值具有数量，而未表示的项目是稀疏的（已知数量为零）。

Oracle Data Mining按如下方式解释缺少的数据：

  • 随机丢失：假设在具有简单数据类型（未嵌套）的列中缺少值，则随机丢失。

  • 稀疏：嵌套列中缺少值表示稀疏性。

3.6.1示例：缺少值还是稀疏数据？

本节中的示例说明了Oracle Data Mining如何将数据标识为稀疏或随机丢失。

3.6.1.1销售表中的稀疏性

销售表包含一组产品的销售点数据，这些产品在一段时间内在多个商店中销售给不同的客户。特定客户只购买少数产品。客户未购买的产品不会在销售表中显示为行。

如果您要计算出客户为每种产品花费的金额，那么未购买的产品的推断量为零。该值不是随机的或未知的;它为零，即使表中没有行。

请注意，销售数据的尺寸标注（按产品，商店，客户和时间），通常表示为挖掘的嵌套数据。

由于嵌套列中的缺失值始终表示稀疏性，因此必须确保此解释适用于要挖掘的数据。例如，当尝试从大型电影数据库的用户挖掘包含电影评级的多记录案例数据集时，缺失的评级是未知的（随机丢失），但Oracle数据挖掘将数据视为稀疏并推断评级为零缺失值。

3.6.1.2客户数据表中缺少值

客户数据表包含有关客户的人口统计数据。案例ID列是客户ID。属性包括年龄，教育，职业，性别，家庭大小等。并非所有数据都适用于每个客户。任何缺失值都被认为是随机丢失的。例如，如果数据中不存在客户1的年龄和客户2的职业，那么该信息就是未知的。它并不表示稀疏性。

请注意，客户数据未标注尺寸。案例与其每个属性之间存在一对一的映射。没有任何属性是嵌套的。

  • 第一个标记的可能值是JAR，RESOURCE或SHARED_DATA，其中RESOURCE表示Java资源对象，SHARED_DATA表示Java共享数据对象，JAR表示数据库驻留JAR对象。此标记是可选的，并且所有值都不区分大小写。如果存在其中一个，则URL称为JAR说明符。否则，它被称为SCHEMA说明符。

  • 第三个令牌的值是SCHEMA。这是必需的令牌，不区分大小写。

  • 第四个标记是必需的标记，它被解释为模式名称。

  • 第五个标记是JAR说明符所必需的，并且禁止用于SCHEMA说明符。它被解释为由第四个标记标识的模式中的对象的名称（如果存在）。

  • 搜索名称以.class结尾

5.2.2在UCP中创建标签回调

要创建标签回调，应用程序将实现oracle.ucp.ConnectionLabelingCallback接口。每个连接池创建一个回调。该接口提供以下两种方法：

  • 成本方法

  • 配置方法

成本方法

此方法考虑标签匹配差异来预测配置连接的成本。在连接请求时，连接池使用此方法选择配置成本最低的连接。

public int cost（Properties requestedLabels，Properties currentLabels）;

配置方法

在将选定连接返回给应用程序之前，此方法由所选连接上的连接池调用。该方法用于设置连接的状态，并在连接中应用或删除任何标签。

public boolean configure（Properties requestedLabels，Connection conn）;

连接池迭代池中可用的每个连接。对于每个连接，它调用cost方法。cost方法的结果是一个integer ，表示将连接重新配置为所需状态所需的成本估算。值越大，重新配置连接的成本越高。连接池始终返回具有最低成本值的连接。算法如下：

  • 如果cost方法为连接返回0 ，则连接是匹配的。连接池不会在找到的连接上调用configure方法，并按原样返回连接。

  • 如果cost方法返回大于0的值，则连接池将迭代，直到找到成本值为0的连接或用尽可用连接。

  • 如果池已遍历所有可用连接，则连接的最低成本为Integer.MAX_VALUE （默认情况下为2147483647），则池中的任何连接都无法满足连接请求。池创建并返回新连接。如果池已达到最大池大小（它无法创建新连接），则池将抛出SQL异常或等待指定连接等待超时属性。

  • 如果池已遍历所有可用连接，并且连接的最低成本低于Integer.MAX_VALUE ，然后在连接上调用configure方法并返回连接。如果多个连接小于Integer.MAX_VALUE ，返回最低成本的连接。

注意：

成本为0并不意味着requestedLabels等于currentLabels 。

重新启用触发器后，只要UPDATE语句更改了员工的作业，数据库就会触发该触发器。如果在禁用触发器时更新了员工的工作，则数据库不会自动触发此员工的触发器，直到另一个事务再次更改job_id 。

15.1关于显式语义分析

在Oracle数据库12c第2版中，引入了显式语义分析（ESA）作为Oracle数据挖掘用于特征提取的无监督算法。从Oracle Database 18c开始，ESA作为分类的监督算法得到了增强。

作为特征提取算法，ESA不会发现潜在特征，而是使用现有知识库中表示的显式特征。作为特征提取算法，ESA主要用于计算文本文档的语义相似度和显式主题建模。作为分类算法，ESA主要用于对文本文档进行分类。ESA的特征提取和分类版本也可以应用于数字和分类输入数据。

ESA的输入是一组属性向量。每个属性向量都与一个概念相关联。该概念是特征提取的特征，或者是分类的目标类。对于特征提取 ，只有一个属性向量可以与任何特征相关联。对于分类 ，训练集可以包含与任何给定目标类相关联的多个属性向量。与一个目标类相关的这些行通过ESA算法聚合为一个。

ESA的输出是稀疏属性概念矩阵，其包含最重要的属性 - 概念关联。通过每个属性 - 概念对的权重值来捕获关联的强度。属性概念矩阵存储为反向索引，列出每个属性的最重要概念。

注意：

对于特征提取 ，ESA算法不会投影原始要素空间，也不会降低其维数。ESA算法使用有限或无信息的属性集过滤掉特征。

的范围 ESA处理的分类任务与分类算法（如朴素贝叶斯和支持向量机）不同 。ESA可以执行大规模分类 ，其中不同类的数量高达数十万。大规模分类需要巨大的训练数据集，其中一些类具有大量训练样本，而其他类在训练数据集中稀疏地表示。

15.1.1用ESA评分

学习使用显式语义分析（ESA）进行评分。

ESA的典型特征提取应用程序是识别给定输入的最相关特征并对其相关性进行评分。评分ESA模型会在概念特征空间中生成数据投影。如果ESA模型是从任意文档集合构建的，则每个文档都被视为一个特征。然后很容易识别集合中最相关的文档。特征提取功能包括： FEATURE_DETAILS ， FEATURE_ID ， FEATURE_SET ， FEATURE_VALUE和FEATURE_COMPARE 。

ESA的典型分类应用是预测给定文档的类别并估计预测的概率。作为分类算法，ESA实现以下评分函数： PREDICTION ， PREDICTION_PROBABILITY ， PREDICTION_SET ， PREDICTION_DETAILS ， PREDICTION_COST 。

15.1.2对大型ESA模型进行评分

在大量文本文档上构建显式语义分析（ESA）模型可以产生具有许多特征或标题的模型。用于评分的模型信息作为共享（共享池大小）库高速缓存对象加载到系统全局区域（SGA）中。不同的SQL预测查询可以引用此对象。当模型大小很大时，必须将数据库中的SGA参数设置为适合大对象的足够大小。

如果SGA太小，则每次引用时都需要重新加载模型，这可能会导致性能下降。

15.2文本挖掘的ESA

了解显式语义分析（ESA）如何用于文本挖掘。

显性知识通常以文本形式存在。多个知识库可用作文本文档的集合。这些知识库可以是通用的，例如，维基百科，或特定于域。数据准备将文本转换为捕获属性 - 概念关联的向量。即使ESA没有任何共同的词，ESA也能够量化文档的语义相关性。函数FEATURE_COMPARE可用于计算语义相关性。

15.3 ESA的数据准备

自动数据准备将输入向量标准化为显式语义分析（ESA）的单位长度。

当具有简单数据类型（未嵌套）的列中存在缺失值时，ESA将使用模式替换缺少的分类值，并使用均值替换缺少的数值。当嵌套列中缺少值时，ESA会将它们解释为稀疏。该算法用零和用零向量的稀疏分类数据替换稀疏数值数据。Oracle Data Mining数据准备将输入文本转换为实数向量。这些数字代表了文本中各个单词的重要性。

15.4显式语义分析中的术语

讨论与显式语义分析（ESA）相关的术语。

多目标分类

这些大规模分类中的培训项目属于几个类别。在这种情况下分类的目标是检测一个项目的可能的多个目标类别。这种分类称为多目标分类。扩展了基于ESA的分类的目标列。允许集合作为目标列值。基于ESA的分类中目标的集合类型为ORA_MINING_VARCHAR2_NT 。

大规模分类

大规模分类适用于包含大量类别的本体，通常范围为数十或数十万。这种大规模分类还需要通常不平衡的巨大训练数据集，即，一些类可能具有大量训练样本，而其他类可能在训练数据集中稀疏地表示。大规模分类通常会导致给定测试用例的多个目标类分配。

主题建模

主题建模是指文档最重要主题的派生。主题建模可以是明确的或潜在的。显式主题建模导致从给定文档的预定义集合中选择最相关的主题。明确的主题有名称，可以用语言表达。潜在主题建模识别一组文档集合的潜在主题。这些潜在主题的子集与正在检查的每个文档相关联。潜在主题没有口头描述或有意义的解释。
