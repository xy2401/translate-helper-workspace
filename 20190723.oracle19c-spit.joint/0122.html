LOB定位器的基于签名的安全性

从此版本开始，您可以为大对象（LOB）定位器配置基于签名的安全性。

此功能增强了Oracle数据库LOB的安全性，尤其是在分布式环境中使用LOB数据类型（CLOB和BLOB）的实例时。

LOB签名密钥可以位于多租户PDB中，也可以位于独立的非多租户数据库中。您可以通过执行ALTER DATABASE DICTIONARY ENCRYPT CREDENTIALS SQL语句来启用LOB签名密钥凭证的ALTER DATABASE DICTIONARY ENCRYPT CREDENTIALS ;否则，凭证以模糊格式存储。如果选择以加密格式存储LOB签名密钥，则数据库或PDB必须具有打开的TDE密钥库。

默认用户帐户现在只有架构

使用Oracle数据库版本18c中的仅模式帐户功能，大多数Oracle数据库提供的模式（用户）现在都会删除其密码，以防止用户对这些帐户进行身份验证。

此增强功能不会影响示例模式。示例模式仍然使用其默认密码进行安装。

对于仅作为模式的默认模式，管理员仍然可以使用密码更改这些帐户（如果需要对模式进行身份验证），但Oracle建议之后将模式更改回仅模式帐户。

此功能的好处是管理员不再需要定期轮换这些Oracle数据库提供的模式的密码。此功能还可以降低使用默认密码侵入这些帐户的攻击者的安全风险。

自动支持SASL和非SASL Active Directory连接

从此版本开始，Microsoft Active Directory连接支持简单身份验证和安全层（SASL）以及传输层安全性（TLS）绑定。

对于集中管理的用户，Oracle数据库最初将尝试使用SASL bind连接到Active Directory。如果Active Directory服务器拒绝SASL绑定连接，则Oracle数据库将在没有SASL绑定的情况下再次自动尝试连接，但仍使用TLS进行保护。

Active Directory管理员负责配置Active Directory服务器的连接参数，但不需要配置数据库以匹配此新的Active Directory连接增强功能。数据库将自动从使用SASL调整为不使用SASL绑定。

只能审核顶级SQL语句的能力

统一审计顶级报表功能使您可以审计数据库中的顶级用户（或直接用户）活动，但无需收集间接用户活动审计数据。

4.4端口号限制

了解端口号的限制。如果需要配置侦听器以使用系统端口号（在1到1024范围内），请使用此处描述的过程配置侦听器。

Oracle允许端口号从1到65535。但是，通常会保留低于1024的端口号（ 众所周知的端口或系统端口 ）。通常，只有特权进程才能侦听1024以下端口上的TCP连接。

如果需要将侦听器配置为侦听小于1024的端口号，请完成以下过程：

注意：

这个程序是一个指导原则。您的操作系统可能需要不同的过程。

  1. 使用Oracle Net Configuration Assistant或Oracle Net Manager为侦听器配置协议地址和其他配置参数。

  2. 以具有侦听器的计算机上的root用户身份登录。

  3. 为侦听器可执行文件（ tnslsnr ）和从属共享库设置文件所有权和访问权限，以便只能由root用户修改这些文件。

  4. 从root目录开始，确保在这些文件的路径名中找到的各个目录的权限共享相同的所有权和访问权限。

  5. 以root用户身份启动侦听器。

  6. 在系统提示符下输入以下命令：

    tnslsnr listener_name -user user -group group

    在上面的命令中，使用以下选项：

    表4-3 tnslsnr实用程序选项

    选项 描述

    LISTENER_NAME

    指定要配置的侦听器的名称。如果省略，则使用默认名称LISTENER 。

    用户

    指定在不需要超级用户（ root ）权限时希望侦听器使用其权限的用户。执行特权操作后，侦听器会不可逆转地放弃root权限。

    组

    指定在不需要超级用户（ root ）组权限时希望侦听器使用其权限的组。执行特权操作后，侦听器会不可逆转地放弃root组权限。

    在此步骤中，侦听器从root更改为您指定的用户和组权限。除了监听配置的端点所需的系统调用之外，所有操作都使用指定的用户和组权限完成。侦听器将恢复为root用户以侦听保留的地址，例如小于1024的TCP端口。

    在侦听器开始侦听listener.ora文件中配置的所有端点之后，它会不可逆地切换到指定的用户和组。在那时候。监听器放弃它最初拥有的root权限。-user和-group命令行参数仅接受以数字形式指定的用户和组标识符。

    例如，要运行具有root权限的名为mylsnr的侦听器，并使用具有37555的用户标识符（UID）的Oracle用户的权限，以及具有组标识符（GID）16的OSDBA组dba成员资格，在操作系统提示符处输入以下命令：

    tnslsnr mylsnr -user 37555 -group 16

  7. 侦听器启动后，您可以使用侦听器控件实用程序对其进行管理。

    警告：

    • Oracle建议运行侦听器进程的用户是oracle用户，或具有侦听器进程通常在操作系统上运行的特权的类似特权用户。

    • 不要让侦听器进程以root用户身份运行。以超级用户身份运行进程是一个安全漏洞。

storage_clause

目的

storage_clause允许您指定Oracle数据库应如何存储永久数据库对象。临时段的存储参数始终使用关联表空间的默认存储参数。存储参数会影响访问存储在数据库中的数据所需的时间以及数据库中空间的使用效率。

也可以看看：

“Oracle自动存储管理管理员指南”，用于讨论存储参数的影响

创建集群，索引，物化视图，物化视图日志，回滚段，表，LOB，varray，嵌套表或分区时，可以为分配给这些对象的段指定存储参数的值。如果省略任何存储参数，则Oracle将使用为该对象所在的表空间指定的该参数的值。如果没有为表空间指定值，则数据库使用默认值。

注意：

为了向后兼容，支持本地管理的表空间中对象的存储参数规范。如果使用本地管理的表空间，则可以在这些表空间中创建对象时省略这些存储参数。

更改集群，索引，物化视图，物化视图日志，回滚段，表，varray，嵌套表或分区时，可以更改存储参数的值。新值仅影响未来的范围分配。

storage_clause是physical_attributes_clause一部分，因此您可以在任何可以指定物理属性子句的语句中指定此子句（请参阅physical_attributes_clause ）。此外，您可以在以下语句中指定storage_clause ：

  • CREATE CLUSTER和ALTER CLUSTER ：设置或更改集群的存储特性以及集群中的所有表 （请参阅CREATE CLUSTER和ALTER CLUSTER ）。

  • CREATE INDEX和ALTER INDEX ：设置或更改为表索引或索引分区创建的索引段的存储特征，或为用于强制执行主键或唯一约束的索引创建的索引段 （请参阅CREATE INDEX和ALTER INDEX ） 。

  • ENABLE ......USING CREATE TABLE或ALTER TABLE INDEX子句：设置或更改系统创建的索引的存储特性， 以强制实施主键或唯一约束 。

  • CREATE MATERIALIZED VIEW和ALTER MATERIALIZED VIEW ：设置或更改物化视图，其分区中的一个，或Oracle生成以维持物化视图的索引的存储特性（见CREATE MATERIALIZED VIEW和ALTER MATERIALIZED VIEW ）。

  • CREATE MATERIALIZED VIEW LOG和ALTER MATERIALIZED VIEW LOG ：设置或更改的物化视图日志的存储特性（见CREATE物化视图日志和ALTER物化视图日志 ）。

  • CREATE ROLLBACK SEGMENT和ALTER ROLLBACK SEGMENT ：设置或更改回滚段的存储特性（请参阅CREATE ROLLBACK SEGMENT和ALTER ROLLBACK SEGMENT ）。

  • CREATE TABLE和ALTER TABLE ：设置非聚簇表或其某个分区或子分区的LOB或varray数据段的存储特性，或嵌套表的存储表 （请参阅CREATE TABLE和ALTER TABLE ）。

  • CREATE TABLESPACE和ALTER TABLESPACE ：设置或更改在表空间中创建的对象的默认存储特性（请参阅CREATE TABLESPACE和ALTER TABLESPACE ）。对表空间存储参数的更改仅影响在表空间中创建的新对象或为段分配的新扩展区。

  • constraint ：指定用于强制执行约束的索引（及其分区，如果它是分区索引）的存储 （请参阅约束 ）。

语义

本节介绍storage_clause的参数。有关其他信息，请参阅为其设置或重置特定数据库对象的这些存储参数的SQL语句。

注意：

对于本地管理的表空间， storage_clause的解释方式不同。对于本地管理的表空间，Oracle数据库使用INITIAL ， NEXT ， PCTINCREASE和MINEXTENTS来计算首次创建对象时分配的扩展区数。创建对象后，将忽略这些参数。有关更多信息，请参见CREATE TABLESPACE 。

初始

指定对象的第一个范围的大小。在创建架构对象时，Oracle会为此范围分配空间。有关该子句的信息，请参阅size_clause 。

在本地管理的表，Oracle使用的值INITIAL ，与当地管理-的类型一起AUTOALLOCATE或UNIFORM -and的值MINEXTENTS ， NEXT和PCTINCREASE ，以确定所述段的初始大小。

  • 通过AUTOALLOCATE管理，Oracle使用INITIAL设置来优化分配的扩展区数。可以分配64K，1M，8M和64M的范围。在段创建期间，系统选择等于或小于INITIAL的这四种大小中的最大大小，并分配与达到INITIAL设置所需的大小一样多的范围。例如，如果将INITIAL设置为4M，则数据库将创建四个1M范围。

  • 对于UNIFORM扩展区管理，扩展区的数量由初始段大小和在表空间创建时指定的统一扩展区大小确定。例如，在具有1M范围的统一本地管理表空间中，如果指定INITIAL值为5M，则Oracle将创建五个1M范围。

    考虑这种比较：使用AUTOALLOCATE ，如果将INITAL设置为72K，则初始段大小将为128K（大于INITIAL ）。数据库无法分配小于64K的扩展区，因此必须分配两个64K扩展区。如果使用UNIFORM扩展区大小为24K将INITIAL设置为72K，则数据库将分配三个24K扩展区以等于72K。

在字典管理的表空间中，默认的初始范围大小为5个块，所有后续范围都舍入为5个块。如果在表空间创建时指定了MINIMUM EXTENT ，则将范围大小四舍五入为MINIMUM EXTENT的值。

对INITIAL的限制

您不能在ALTER语句中指定INITIAL 。

下一个

以字节为单位指定要分配给对象的下一个范围的大小。有关该子句的信息，请参阅size_clause 。

在本地管理的表，对于任何用户提供的值NEXT被忽略的尺寸和NEXT由Oracle如果表被设定为AUTOALLOCATE程度管理决定。在UNIFORM表空间中， NEXT的大小是在表空间创建时指定的统一范围大小。

在字典管理的表空间中，默认值是5个数据块的大小。最小值是1个数据块的大小。最大值取决于您的操作系统。对于少于5个数据块的值，Oracle将值舍入到数据块大小的下一个倍数。对于大于5个数据块的值，Oracle会将最接近的值舍入到最小化碎片的值。

PCTINCREASE

在本地管理的表空间中，Oracle数据库在创建段期间使用PCTINCREASE的值来确定初始段大小，并在后续空间分配期间忽略此参数。

在字典管理的表空间中，指定第三个和后续扩展区在前一个扩展区上的增长百分比。默认值为50，表示每个后续范围比前一个范围大50％。最小值为0，表示第一个之后的所有范围都是相同的大小。最大值取决于您的操作系统。Oracle将每个新范围的计算大小舍入为数据块大小的最接近倍数。如果通过在ALTER语句中指定PCTINCREASE参数来更改其值，则Oracle将使用此新值和最近分配的范围的大小计算下一个范围的大小。

对PCTINCREASE的限制

您不能为回滚段指定PCTINCREASE 。回滚段的PCTINCREASE值始终为0。

MINEXTENTS

在本地管理的表空间中，Oracle数据库使用MINEXTENTS的值与PCTINCREASE ， INITIAL和NEXT结合使用来确定初始段大小。

在字典管理的表空间中，指定创建对象时要分配的扩展区总数。默认值和最小值为1，这意味着Oracle仅分配初始范围，但回滚段除外，其默认值和最小值均为2。最大值取决于您的操作系统。

  • 在本地管理的表空间中， MINEXTENTS用于计算分配的初始空间量，等于INITIAL * MINEXTENTS 。此后，此值设置为1，这将反映在DBA_SEGMENTS视图中。

  • 在字典管理的表空间中， MINEXTENTS只是必须分配给段的最MINEXTENTS数。

如果MINEXTENTS值大于1，则Oracle将根据INITIAL ， NEXT和PCTINCREASE存储参数的值计算后续扩展区的大小。

通过在ALTER语句中指定MINEXTENTS来更改MINEXTENTS的值时，可以从当前值减小该值，但不能增加它。将MINEXTENTS重置为较小的值可能很有用，例如，在TRUNCATE之前......DROP STORAGE语句，如果要确保在TRUNCATE操作之后段将保持最小数量的范围。

对MINEXTENTS的限制

MINEXTENTS存储参数受以下限制：

  • MINEXTENTS不适用于表空间级别。

  • 您不能更改ALTER语句中的MINEXTENTS值或驻留在本地管理的表空间中的对象的值。

MAXEXTENTS

此存储参数仅对字典管理的表空间中的对象有效。指定Oracle可以为该对象分配的扩展区总数，包括第一个扩展区。除了回滚段之外，最小值为1，回滚段始终至少为2。默认值取决于您的数据块大小。

对MAXEXTENTS的限制

对于驻留在本地管理的表空间中的对象，将忽略MAXEXTENTS ，除非ALLOCATION_TYPE的值对于DBA_TABLESPACES数据字典视图中的表空间是USER 。

无限

如果要根据需要自动分配扩展区，请指定UNLIMITED 。Oracle建议将此设置作为最小化碎片的方法。

不要将此子句用于回滚段。这样做可能会导致长时间运行的恶意DML事务继续创建新的扩展区，直到磁盘已满。

注意：

您在未指定storage_clause情况下创建的回滚段具有与创建回滚段的表空间相同的存储参数。因此，如果您创建一个MAXEXTENTS UNLIMITED的表空间，则回滚段将具有相同的默认值。

MAXSIZE

MAXSIZE子句允许您指定存储元素的最大大小。对于LOB存储， MAXSIZE具有以下效果

  • 如果在LOB_parameters指定RETENTION MAX ，则在从撤消空间回收任何空间之前，LOB段将增加到指定的大小。

  • 如果在LOB_parameters指定RETENTION AUTO ， MIN或NONE ，则指定的大小是LOB段大小的硬限制，与撤消保留无关。

无限

如果您不想限制存储元素的磁盘空间，请使用UNLIMITED子句。该子句与LOB_parameters的RETENTION MAX规范不兼容。如果同时指定两者，则数据库使用RETENTION AUTO和MAXSIZE UNLIMITED 。

FREELISTS

在具有手动段空间管理的表空间中，Oracle数据库使用FREELISTS存储参数通过增加段中插入点的数量来提高OLTP系统中空间管理的性能。在具有自动段空间管理的表空间中，忽略此参数，因为数据库适应不同的工作负载。

在具有手动段空间管理的表空间中，对于表空间和回滚段以外的对象，请为表，分区，集群或索引指定每个空闲列表组的空闲列表数。此参数的默认值和最小值为1，表示每个空闲列表组包含一个空闲列表。此参数的最大值取决于数据块大小。如果指定的FREELISTS值太大，则Oracle会返回指示最大值的错误。

如果指定了该条款是无效的或有用SECUREFILE的参数LOB_parameters 。如果同时指定SECUREFILE参数和FREELISTS ，则数据库将以静默方式忽略FREELISTS规范。

对FREELISTS的限制

除了创建或更改表空间或回滚段之外，您可以在任何语句的storage_clause中指定FREELISTS 。

自由组

在具有手动段空间管理的表空间中，Oracle数据库使用此存储参数的值对Oracle Real Application Clusters环境中的段可用空间进行静态分区。此分区通过避免段元数据的实例间传输来提高空间分配和释放的性能。在具有自动段空间管理的表空间中，此参数将被忽略，因为Oracle会动态适应实例间工作负载。

在具有手动段空间管理的表空间中，指定要创建的数据库对象的空闲列表组的数量。此参数的默认值和最小值为1。Oracle使用Oracle Real Application Clusters（Oracle RAC）实例的实例编号将每个实例映射到一个空闲列表组。

每个空闲列表组使用一个数据库块。因此：

  • 如果没有为INITIAL指定足够大的值来覆盖每个空闲列表组的最小值加一个数据块，那么Oracle INITIAL的值增加到必要的数量。

  • 如果要在统一的本地管理的表空间中创建对象，并且扩展区大小不足以容纳空闲列表组的数量，则创建操作将失败。

如果指定了该条款是无效的或有用SECUREFILE的参数LOB_parameters 。如果同时指定SECUREFILE参数和FREELIST GROUPS ，则数据库将以静默方式忽略FREELIST GROUPS规范。

对FREELIST组的限制

您只能在CREATE TABLE ， CREATE CLUSTER ， CREATE MATERIALIZED VIEW ， CREATE MATERIALIZED VIEW LOG和CREATE INDEX语句中指定FREELIST GROUPS参数。

最佳

OPTIMAL关键字仅与回滚段相关。它为回滚段指定了最佳字节大小。有关该子句的信息，请参阅size_clause 。

Oracle尝试通过在活动事务不再需要其数据时动态释放扩展区来为回滚段维护此大小。Oracle在不降低OPTIMAL值以下的回滚段总大小的情况下，尽可能多地释放扩展区。

OPTIMAL的值不能小于MINEXTENTS ， INITIAL ， NEXT和PCTINCREASE参数最初分配的空间。最大值取决于您的操作系统。Oracle将值舍入到数据块大小的下一个倍数。

空值

为回滚段指定没有最佳大小的NULL ，这意味着Oracle永远不会释放回滚段的扩展区。这是默认行为。

BUFFER_POOL

BUFFER_POOL子句允许您为架构对象指定默认缓冲池或缓存。对象的所有块都存储在指定的缓存中。

  • 如果为分区表或索引定义缓冲池，则分区将从表或索引定义继承缓冲池，除非被分区级定义覆盖。

  • 对于索引组织表，可以分别为索引段和溢出段指定缓冲池。

BUFFER_POOL参数的限制

BUFFER_POOL受以下限制：

  • 您不能为群集表指定此子句。但是，您可以为群集指定它。

  • 您不能为表空间或回滚段指定此子句。

保持

指定KEEP将段中的块放入KEEP缓冲池。维护适当大小的KEEP缓冲池可让Oracle将架构对象保留在内存中以避免I / O操作。KEEP优先于您为表，集群，物化视图或物化视图日志指定的任何NOCACHE子句。

回收

指定RECYCLE将段中的块放入RECYCLE池。适当大小的RECYCLE池减少了默认池为RECYCLE池的对象数量占用不必要的缓存空间。

默认

指定DEFAULT以指示默认缓冲池。这是未分配给KEEP或RECYCLE对象的默认值。

FLASH_CACHE

FLASH_CACHE子句允许您覆盖自动缓冲区高速缓存策略，并指定特定模式对象在高速缓存中的缓存方式。要使用此子句，必须在系统上配置数据库智能闪存缓存（闪存缓存）。闪存缓存是存储在闪存盘（一种使用闪存的存储设备）上的数据库缓冲区缓存的扩展。由于闪存比磁盘快，因此数据库可以通过缓存闪存缓存中的缓冲区而不是从磁盘读取来提高性能。

保持

如果希望模式对象缓冲区保持缓存在闪存缓存中，只要闪存缓存足够大，请指定KEEP 。

没有

指定NONE以确保模式对象缓冲区永远不会缓存在闪存缓存中。这允许您为更频繁访问的对象保留闪存缓存空间。

默认

如果希望将架构对象缓冲区从主内存中老化后写入闪存缓存，然后使用标准缓冲区缓存替换算法将其老化到闪存缓存中，请指定DEFAULT 。如果配置了闪存缓存且未指定KEEP或NONE则这是默认设置。

8.1.4.1修改数据库初始化参数

如果数据库由Oracle Database QoS Management管理，则必须将服务器池中运行的每个数据库实例的CPU_COUNT参数设置为相同的值。

在每台服务器上，该服务器上运行的所有数据库实例的CPU_COUNT值的总和必须小于或等于物理CPU计数。例如，如果您的服务器有8个CPU，并且在此服务器上运行了两个数据库实例，则对于要由Oracle数据库QoS管理管理的数据库，必须设置每个数据库实例的CPU_COUNT参数，以便服务器上所有实例的CPU_COUNT参数值最多为8或更少。例如，一个实例上的CPU_COUNT=3 ，另一个实例上的CPU_COUNT=4 ，或者一个实例上的CPU_COUNT=6 ，另一个实例上的CPU_COUNT=2 。

注意：

默认情况下，在服务器上启动的每个数据库的CPU计数都设置为为该服务器安装的物理CPU数。

如果在服务器池中运行多个数据库，则使用CPU_COUNT的缺省设置将导致Oracle数据库QoS管理报告违规。要避免此错误，请使用Oracle Enterprise Manager或SQL * Plus手动配置SPFILE中的CPU_COUNT值。

  • 使用SQL * Plus修改Oracle RAC数据库的所有实例的CPU_COUNT数据库初始化参数：
    ALTER SYSTEM SET cpu_count = n SCOPE = BOTH SID ='*';

    在上一个命令中， n是数据库实例应使用的CPU数。

    CPU_COUNT是默认情况下未设置的动态参数。应将其设置为数据库实例随时应使用的最大CPU数。服务器上所有实例的CPU_COUNT值的总和不应超过该服务器的物理CPU数。作为最佳实践，Oracle建议对CPU_COUNT使用最小值2。

8.1.7关于Solaris上的多CPU绑定和服务质量管理

使用多CPU绑定（MCB）和Oracle数据库服务质量（QoS）管理需要系统管理员和数据库管理员（DBA）之间的密切通信。

多CPU绑定（MCB）是Oracle Solaris项目资源管理功能，用于将项目绑定到特定的一组CPU，但不能独占绑定CPU。MCB允许其他进程也使用这些CPU并允许分区重叠。Oracle Solaris 11.3支持MCB。Linux系统上的控制组（CGroup）是另一种系统管理员方法，通过将CPU和服务器资源分配给特定应用程序来管理服务器资源。

在测量和监视模式下使用时，MCB对Oracle数据库服务质量（QoS）管理的使用没有影响。在管理模式下为一组服务器使用Oracle数据库服务质量（QoS）管理时，Oracle QoS Management当前支持四种资源控制：

  1. 消费者组映射 ：非CDB或PDB中的竞争工作负载之间的CPU份额。

  2. 容器数据库（CDB）资源计划 ：CDB内竞争PDB之间的CPU份额

  3. 实例锁定 ：共同托管的数据库实例之间的CPU /线程

  4. 服务器池基数 ：提供数据库的服务器池中的服务器数量

MCB成为实例锁定的问题，因为Oracle数据库服务质量（QoS）管理可能会建议操作系统不会遵守的CPU_COUNT更改。如果在这种情况下实施建议的操作，则目标工作负载可能仍会有一些改进，因为捐赠者数据库会丢失CPU。这将导致资源管理器不安排尽可能多的并行会话，这将在不使用硬分区时提供帮助。但是，预计的绩效改善将被夸大。

1数据磁带简介

Oracle数据库提供数据磁带以支持对象模式下的自定义应用程序开发。

除了在关系模型下有序和安全地管理数据之外，Oracle还为在对象模型下组织的数据提供支持。对象类型和其他功能（如大对象（ LOB ），外部过程，可扩展索引和查询优化）可用于构建功能强大，可重用，基于服务器的组件，这些组件称为 数据墨盒 。

1.1数据磁带概述

数据盒式磁带利用Oracle可扩展性架构框架扩展了Oracle服务器的功能。通过此框架，您可以捕获与用户定义的数据类型中的专用或特定于域的数据相关联的业务逻辑和流程。提供新行为而不需要其他属性的数据盒式磁带可以选择使用包而不是用户定义的类型。无论哪种方式，您都可以确定服务器如何解释，存储，检索和索引应用程序数据。数据盒式磁带打包此功能，创建插入服务器的软件组件，并将其功能扩展到新域，使数据库本身可扩展 。

您可以自定义可扩展数据库管理系统的索引和查询优化机制，并为用户定义的业务对象和丰富类型提供专门的服务或更有效的处理。通过可扩展性接口向服务器注册实现时，可以指示服务器实现自定义处理指令，而不是自己的默认进程。

可扩展性接口由服务器调用以执行自定义索引或优化为数据盒式磁带实现的行为的函数组成。接口由Oracle定义;作为盒式磁带开发人员，您必须实现具有应用程序所需特殊行为的功能或接口。通常，您将函数实现为对象类型的静态方法。实现可扩展索引接口的对象类型称为 indextype ;实现可扩展优化接口的对象类型称为a 统计类型 。

数据磁带具有以下主要特征：

  • 数据盒式磁带是基于服务器的 。它们的组成部分驻留在服务器上或从服务器访问。服务器运行所有数据盒式磁带进程，或将这些进程作为外部过程分派。

  • 数据盒带扩展服务器 。它们定义新的类型和行为，使服务器能够以组件形式执行其他方式不可用的进程。数据盒式磁带可以在其应用程序中使用这些新类型和行为。

  • 数据盒带与服务器集成在一起 。Oracle可扩展性框架定义了一组接口，这些接口将数据盒式磁带与服务器引擎的组件集成在一起，允许特定于域的索引，针对CPU资源的特定于域的优化访问，以及针对盒式磁带的I / O访问的特定于域的优化数据。

  • 数据盒式磁带已打包。数据盒式磁带作为一个整体安装。安装后，数据盒式磁带会处理每个用户的所有访问问题，包括验证架构和权限。

1.2数据磁带的使用

大多数行业都在发展复杂的模型来处理构成其业务本质的复杂数据对象 。这些数据对象都是涉及不同信息单元和对它们执行的操作的结构。

给数据对象的简单名称通常隐藏了相当大的复杂性。例如，银行业有许多不同类型的银行账户。每个银行账户都有客户人口统计信息，余额信息，交易信息以及体现其行为的规则（存款，取款，应计利息等）。使用数据盒带及其对象关系扩展时，应用程序编程人员和独立软件供应商可以将业务逻辑封装在与Oracle服务器集成的软件组件中，并对其进行增强，以支持对业务对象建模的数据类型，流程和逻辑。

商业模式越来越发达复杂的数据对象，信息技术使得有必要处理新的和复杂的数据，如卫星图像，X射线，动物声音，地震振动和化学模型。现在经常存储和检索，查询和分析复杂和多媒体数据类型。

1.3.1.1.4大对象

大对象类型，或LOB S，处理图像，视频剪辑，文档和其他形式的非结构化数据的存储需求。LOB的存储优化了空间要求和高效访问。

LOB由定位器和相关的二进制或字符数据组成。定位器与其他表列一起存储。内部LOB （ BLOB ， CLOBs和NCLOB ）可以将数据存储在单独的数据库存储区域中。外部LOB （ BFILE ）将数据存储在数据库表空间的操作系统文件中。表可以包含多个LOB列，而不是每个表的单个LONG RAW列的限制。每个LOB列可以存储在单独的表空间中，甚至可以存储在不同的辅助存储设备上。

您可以使用Oracle SQL数据定义语言（DDL）扩展创建，修改和删除包含LOB的表和对象类型。使用Oracle SQL数据操作语言（DML）语句，您可以插入和删除完整的LOBs 。还有一组广泛的语句，用于在Java，PL / SQL和Oracle调用接口中分段读取，编写和操作LOB 。

对于内部LOB类型，定位器和相关数据都完全参与Oracle服务器的事务模型。BFILEs的数据不参与交易;但是，Oracle服务器事务完全支持BFILE定位器。

与标量数量不同， LOB值无法通过内置索引方案进行索引。但是，您可以使用各种LOB API来构建模块，包括用户定义类型的方法，以访问和操作LOB内容。您可以定义驻留在LOB的数据的语义，并使用可扩展的索引框架来操作此数据。

1.3.1.3可扩展索引

基本数据库管理系统支持几种类型的访问方法，例如 B +树和哈希索引，在一组有限的数据类型上，例如数字和字符串。对于像整数和小字符串这样的简单数据类型，索引的所有方面都可以由数据库系统轻松处理。随着数据通过添加文本，空间，图像，视频和音频信息变得更加复杂，它需要复杂的数据类型和专门的索引技术。

复杂数据类型具有特定于应用程序的格式，索引要求和选择谓词。例如，有许多不同的文档编码方式（ODA，XML，纯文本）和信息检索技术（关键字，全文布尔，相似性和概率）。同样的， R树是索引空间数据的有效方法。为了使您能够定义业务需求所需的索引类型，Oracle提供了一个可扩展的索引框架。

此类用户定义的索引称为域索引，因为它们在特定于应用程序的域中索引数据。盒式磁带负责定义索引结构，在加载和更新操作期间维护索引内容，以及在查询处理期间搜索索引。物理索引可以作为表存储在Oracle数据库中，也可以作为文件存储在外部。

域索引是架构对象。它由作为用户定义类型的方法实现的例程创建，管理和访问 indextype 。构建域索引中描述了indextype必须实现的例程以及例程必须执行的操作。例程的实现特定于应用程序，因此必须由盒式开发人员完成。

使用可扩展索引，应用程序必须具有以下过程：

  • 定义域索引的结构。

  • 将索引数据存储在Oracle数据库的内部或外部。

  • 管理，检索和使用索引数据来评估用户查询。

当数据库系统处理域索引的物理存储时，数据盒式磁带必须具有以下过程：

  • 定义索引的格式和内容。盒式磁带定义了一个可以容纳复杂数据对象的索引结构。

  • 构建，删除和更新域索引。墨盒构建并维护索引结构。由于索引被建模为元组集合，因此它们直接支持就地更新。

  • 访问和解释索引的内容。通过处理数据库查询的内容相关子句，盒式磁带成为查询处理的不可或缺的组成部分。

典型的关系数据库和对象关系数据库管理系统不支持可扩展索引。因此，许多应用程序为关系数据库表中的复杂数据维护基于文件的索引。完成以下任务需要大量代码和工作量：

  • 保持外部索引与相关关系数据之间的一致性。

  • 支持涉及表格值和外部索引的复合查询。

  • 使用多种形式的持久存储（如文件和数据库）管理系统，执行备份，恢复，存储分配等。

通过支持可扩展索引，Oracle 服务器显着降低了开发涉及对复杂数据类型的高性能访问的解决方案所需的工作量。

1.3.1.4可扩展优化器

该 可扩展优化器允许用户定义的函数和索引收集统计信息，例如选择性和成本函数，并为SQL语句生成执行计划。优化程序在选择查询计划时使用此信息，从而扩展优化程序以使用用户提供的信息。基于规则的优化器保持不变。

一个优化器生成的执行计划包括FROM子句中每个表的访问方法，以及一个名为 连接 FROM子句中表的顺序 。系统定义的访问方法包括索引，散列簇和表扫描。对于连接顺序中的每个表，优化程序通过生成一组连接顺序或排列，计算每个顺序的成本以及选择成本最低的表来选择计划。连接顺序的成本是访问方法和连接方法成本的总和。

成本模型是一组用于计算给定操作成本的算法。它可以包括有关查询运行的物理环境的不同级别的详细信息。当前成本模型包括磁盘访问次数和网络成本估算，并进行微调。

优化程序还使用有关查询中引用的对象的统计信息来计算成本和 选择性 ，或查询选择的表中行的分数（介于0和100之间，百分比）。该 DBMS_STATS包中包含用于生成这些统计信息的方法。

可扩展性允许用户定义新的运算符，索引类型和域索引，并允许控制优化程序使用的三个主要组件来选择执行计划：统计，选择性和成本。

1.3.2可扩展性接口

19 DBMS_APPLICATION_INFO

应用程序开发人员可以将DBMS_APPLICATION_INFO程序包与Oracle Trace和SQL跟踪工具一起使用，以记录数据库中正在执行的模块或事务的名称，以便以后在跟踪各种模块的性能和调试时使用。

本章包含以下主题：

  • 概观

19.1 DBMS_APPLICATION_INFO概述

注册应用程序允许系统管理员和性能调整专家按模块跟踪性能。系统管理员还可以使用此信息来跟踪模块的资源使用情况。当应用程序向数据库注册时，其名称和操作将记录在V$SESSION和V$SQLAREA视图中。

19.2 DBMS_APPLICATION_INFO安全模型

无需其他特权。DBMSAPIN.SQL脚本已作为标准数据库创建的一部分运行。

注意：

DBMS_APPLICATION_INFO的公共同义词在创建之前不会被删除，因此您可以将公共同义词重定向到指向您自己的包。

19.3 DBMS_APPLICATION_INFO操作说明

每次用户进入该模块时，您的应用程序都应自动设置模块名称和操作名称。模块名称可以是Oracle Forms应用程序中表单的名称，也可以是Oracle预编译器应用程序中代码段的名称。操作名称通常应该是模块中当前事务的名称或描述。

如果要基于模块收集自己的统计信息，可以通过在另一个模式中编写此软件包的版本来实现此软件包的包装，该模式首先收集统计信息，然后调用软件包的SYS版本。然后可以将DBMS_APPLICATION_INFO的公共同义词更改为指向DBA的软件包版本。

G.2.1使用旧语法执行到物理备用数据库的切换

在主数据库上启动切换，并在目标备用数据库上完成切换。

本主题介绍如何切换到物理备用数据库。

  1. 验证主数据库是否可以切换到备用角色。

    查询主数据库上V$DATABASE视图的SWITCHOVER_STATUS列。例如：

    SQL> SELECT SWITCHOVER_STATUS FROM V $ DATABASE; SWITCHOVER_STATUS ----------------- TO STANDBY选择1行

    值TO STANDBY或SESSIONS ACTIVE表示主数据库可以切换到备用角色。如果这两个值均未返回，则无法进行切换，因为重做传输配置错误或无法正常运行。有关配置和监视重做传输的信息，请参阅重做传输服务 。

  2. 在主数据库上发出以下SQL语句以将其切换到备用角色：在主数据库上发出以下SQL语句以将其切换到备用角色：
    SQL> ALTER DATABASE承诺切换到物理待机;

    此语句将主数据库转换为物理备用数据库。在切换之前，当前控制文件将备份到当前SQL会话跟踪文件。如果需要，这使得可以重建当前控制文件。

  3. 挂载以前的主数据库。例如：例如：
    SQL> STARTUP MOUNT;

    此时，在切换过程中，原始主数据库是物理备用数据库。

  4. 查询备用数据库上V$DATABASE视图的SWITCHOVER_STATUS列，以验证切换目标是否已准备好切换到主角色。例如：
    SQL> SELECT SWITCHOVER_STATUS FROM V $ DATABASE; SWITCHOVER_STATUS -----------------要选择PRIMARY 1行

    TO PRIMARY或SESSIONS ACTIVE值表示备用数据库已准备好切换到主要角色。如果未返回这些值，请验证重做应用是否处于活动状态，并且已配置重做传输并且正常工作。继续查询此列，直到返回的值为TO PRIMARY或SESSIONS ACTIVE 。

  5. 在目标物理备用数据库上发出以下SQL语句，以将其切换为主要角色：
    SQL> ALTER DATABASE使用SESSION SHUTDOWN进行切换到主切换;

    注意：

    如果在步骤4中执行的查询返回TO PRIMARY值，则可以从switchover语句中省略WITH SESSION SHUTDOWN子句。

  6. 打开新的主数据库：
    SQL> ALTER DATABASE OPEN;
  7. 在新的物理备用数据库上启动重做应用：
    SQL> ALTER DATABASE RECOVER MANAGED STANDBY DATABASE  - >断开会话;
  8. 如果已在Oracle Data Guard配置中的任何其他物理备用数据库中停止，请重新启动重做应用：
    SQL> ALTER DATABASE RECOVER MANAGED STANDBY DATABASE  - >断开会话;

G.2.2使用旧语法执行到物理备用数据库的故障转移

这些步骤描述了如何执行到物理备用数据库的故障转移。

  1. 如果可能，请装入主数据库并将任何未发送的已归档和当前重做从主数据库刷新到备用数据库。如果此操作成功，即使主数据库不处于零数据丢失数据保护模式，也可以进行零数据丢失故障转移。

    确保Redo Apply在目标备用数据库中处于活动状态。

    挂载，但不要打开主数据库。如果无法装入主数据库，请转到步骤2。

    在主数据库上发出以下SQL语句：

    SQL> ALTER SYSTEM FLUSH REDO TO target_db_name ;

    对于target_db_name ，请指定备用数据库的DB_UNIQUE_NAME ，该数据库将接收从主数据库刷新的重做。

    此语句刷新从主数据库到备用数据库的任何未发送的重做，并等待将该重做应用于备用数据库。

    如果此语句完成且没有任何错误，请转到步骤5。如果语句以任何错误完成，或者由于您不能再等待语句完成而必须停止语句，请继续执行步骤2。

  2. 验证备用数据库是否具有每个主数据库重做线程的最近归档的重做日志文件。

    为此，请查询目标备用数据库上的V$ARCHIVED_LOG视图，以获取每个重做线程的最高日志序列号。

    例如：

    SQL> SELECT UNIQUE THREAD #AS THREAD，MAX（SEQUENCE＃） - > OVER（PARTITION BY thread＃）AS VAST ARCHIVED_LOG;线程最后---------- ---------- 1 100

    如果可能，将每个主数据库重做线程的最近归档的重做日志文件复制到备用数据库（如果该数据库不存在），并进行注册。必须为每个重做线程执行此操作。

    例如：

    SQL> ALTER DATABASE注册PHYSICAL LOGFILE'vilespec1';
  3. 查询目标备用数据库上的V$ARCHIVE_GAP视图，以确定目标备用数据库上是否存在任何重做间隙。

    例如：

    SQL> SELECT THREAD＃，LOW_SEQUENCE＃，HIGH_SEQUENCE＃FROM V $ ARCHIVE_GAP; THREAD＃LOW_SEQUENCE＃HIGH_SEQUENCE＃---------- ------------- -------------- 1 90 92

    在此示例中，间隙包括具有序列号90,91和92的归档重做日志文件，用于线程1。

    如果可能，将任何缺少的归档重做日志文件从主数据库复制到目标备用数据库，并将它们注册到目标备用数据库。必须为每个重做线程执行此操作。

    例如：

    SQL> ALTER DATABASE注册PHYSICAL LOGFILE'vilespec1';
  4. 重复步骤3，直到解决所有间隙。（在步骤3中执行的查询仅显示最高间隙的信息，因此在解决间隙后，必须重复查询，直到不再返回任何行。）

    如果在执行步骤2到步骤4之后，您无法解决归档重做日志文件中的所有空白（例如，因为您无权访问托管故障主数据库的系统），则会丢失一些数据在故障转移期间发生。

  5. 在目标备用数据库上发出以下SQL语句以停止重做应用：
    SQL> ALTER DATABASE RECOVER MANAGED STANDBY DATABASE CANCEL;
  6. 完成应用所有收到的重做数据：
    SQL> ALTER DATABASE RECOVER MANAGED STANDBY DATABASE FINISH;

    如果此语句完成且没有任何错误，则继续执行步骤7。

    如果发生错误，则不会应用某些接收的重做数据。尝试解决错误原因并重新发出语句，然后再继续执行下一步。

    如果在步骤3和步骤4中存在未解决的重做间隙，则会收到错误消息，指出存在重做间隙。

    如果无法解决错误情况，则仍可通过在目标备用数据库上发出以下SQL语句来执行故障转移（丢失一些数据）：

    SQL> ALTER DATABASE激活物理备用数据库;

    当ACTIVATE语句完成时，继续执行步骤9。

  7. 验证目标备用数据库是否已准备好成为主数据库。

    为此，请查询目标备用数据库上V$DATABASE视图的SWITCHOVER_STATUS列。例如：

    SQL> SELECT SWITCHOVER_STATUS FROM V $ DATABASE; SWITCHOVER_STATUS -----------------要选择PRIMARY 1行

    TO PRIMARY或SESSIONS ACTIVE值表示备用数据库已准备好切换到主要角色。如果未返回这些值，请验证Redo Apply是否处于活动状态并继续查询此视图，直到返回TO PRIMARY或SESSIONS ACTIVE 。

  8. 在目标备用数据库上发出以下SQL语句，以将物理备用数据库切换为主要角色：
    SQL> ALTER DATABASE使用SESSION SHUTDOWN进行切换到主切换;

    注意：

    如果在上一步骤中执行的SWITCHOVER_STATUS列的查询返回TO PRIMARY值，则可以从switchover语句中省略WITH SESSION SHUTDOWN子句。

  9. 打开新的主数据库：
    SQL> ALTER DATABASE OPEN;
  10. Oracle建议您现在备份新的主数据库。
  11. 如果已在Oracle Data Guard配置中的任何其他物理备用数据库中停止，请重新启动重做应用：
    SQL> ALTER DATABASE RECOVER MANAGED STANDBY DATABASE  - >断开会话;
  12. （可选）还原失败的主数据库。

    故障转移后，可以使用使用闪回数据库 将故障主 数据库 转换为备用数据库或使用RMAN备份将故障主数据库转换为备用数据库中描述的方法将原始主数据库转换为新主数据库的物理备用数据库，或者可以使用创建物理备用数据库的分步说明中描述的方法从新主数据库的备份中将其重新创建为物理备用数据库 。

    原始主数据库以备用角色运行后，可以执行切换以将其还原为主角色。

G.3.1切换失败，因为没有传输重做数据

如果切换未成功完成，您可以在V$ARCHIVED_LOG视图中查询SEQUENCE#列，以查看从原始主数据库传输的最后一个重做数据是否已应用于备用数据库。

如果最后一个重做数据未传输到备用数据库，则可以手动将包含重做数据的存档重做日志文件从原始主数据库复制到旧备用数据库，并使用SQL ALTER DATABASE REGISTER LOGFILE file_specification语句进行ALTER DATABASE REGISTER LOGFILE 。如果您随后开始应用服务，则会自动应用存档的重做日志文件。查询V$DATABASE视图中的SWITCHOVER_STATUS列。如果SWITCHOVER_STATUS列返回TO PRIMARY或SESSIONS ACTIVE则现在可以切换到主要角色：

SQL> SELECT SWITCHOVER_STATUS FROM V $ DATABASE; SWITCHOVER_STATUS -----------------要选择PRIMARY 1行

有关SWITCHO其他有效值的信息，请参阅与Oracle Data Guard相关的SWITCHO V$DATABASE视图的VER_STATUS列。

要继续切换，请按照使用旧语法执行切换到物理备用数据库中的说明进行操作，然后再次尝试将目标备用数据库切换为主要角色。

G.3.2切换因ORA-01102错误而失败

如果您收到ORA-01102错误，这些是可能的原因和解决方案。

假设备用数据库和主数据库位于同一站点上。在ALTER DATABASE COMMIT TO SWITCHOVER TO PHYSICAL STANDBY和ALTER DATABASE COMMIT TO SWITCHOVER TO PRIMARY语句之后，成功执行，关闭并重新启动物理备用数据库和主数据库。

注意：

如果物理备用数据库在实例启动后尚未以只读方式打开，则无需关闭并重新启动物理备用数据库。

但是，第二个数据库的启动失败， ORA-01102 cannot mount database in EXCLUSIVE mode错误。

如果未在备用数据库（原始主数据库）使用的初始化参数文件中设置DB_UNIQUE_NAME参数，则可能会在切换期间发生这种情况。如果未设置备用数据库的DB_UNIQUE_NAME参数，则备用数据库和主数据库都使用相同的安装锁定，并在启动第二个数据库期间导致ORA-01102错误。

操作：将DB_UNIQUE_NAME= unique_database_name添加到备用数据库使用的初始化参数文件中，然后关闭并重新启动备用数据库和主数据库。

切换后不应用G.3.3重做数据

如果切换后归档的重做日志文件未应用于新的备用数据库，则可能是因为切换后未正确设置某些环境或初始化参数。

行动：

  • 检查新主站点上的tnsnames.ora文件和新备用站点上的listener.ora文件。备用站点上的侦听器应该有条目，主站点上应该有相应的服务名称。

  • 如果尚未启动，则在备用站点启动侦听器。

  • 检查LOG_ARCHIVE_DEST_ n初始化参数是否已设置为正确地将重做数据从主站点传输到备用站点。例如，在主站点查询V$ARCHIVE_DEST固定视图，如下所示：

    SQL> SELECT DEST_ID，STATUS，DESTINATION from V $ ARCHIVE_DEST;

    如果没有看到与备用站点对应的条目，则需要设置LOG_ARCHIVE_DEST_ n和LOG_ARCHIVE_DEST_STATE_ n初始化参数。

  • 验证在备用站点上是否正确设置了LOG_ARCHIVE_FORMAT初始化参数。

  • 在备用站点，设置 DB_FILE_NAME_CONVERT和 LOG_FILE_NAME_CONVERT初始化参数。将STANDBY_FILE_MANAGEMENT初始化参数设置为AUTO以使备用站点能够自动添加在主站点上创建的新数据文件。

G.3.4切换失败并重新启动后回滚

对于发生错误且无法继续切换的情况下的物理备用数据库，仍可能将新的物理备用数据库还原为主要角色。

采取以下步骤：

  1. 关闭并装入新的备用数据库（旧的主数据库）。
  2. 在新的备用数据库上启动重做应用。
  3. 验证新备用数据库是否已准备好切换回主角色。查询新备用数据库上V$DATABASE视图的SWITCHOVER_STATUS列。例如：
    SQL> SELECT SWITCHOVER_STATUS FROM V $ DATABASE; SWITCHOVER_STATUS ----------------- TO_PRIMARY选择了1行

    TO PRIMARY或SESSIONS ACTIVE值表示新备用数据库已准备好切换到主要角色。继续查询此列，直到返回的值为TO PRIMARY或SESSIONS ACTIVE 。

  4. 发出以下语句以将新备用数据库转换回主要角色：
    SQL> ALTER DATABASE使用SESSION SHUTDOWN进行切换到主切换;

    如果此语句成功，则数据库将以主数据库角色运行，您无需执行任何其他步骤。

    如果此声明不成功，请继续步骤5。

  5. 当启动切换以将角色从主服务器更改为物理服务器时，会在日志目录中写入跟踪文件。此跟踪文件包含重新创建原始主控制文件所需的SQL语句。找到跟踪文件并将SQL语句提取到临时文件中。从SQL * Plus执行临时文件。这会将新备用数据库还原为主角色。
  6. 关闭原始物理备用数据库。
  7. 创建一个新的备用控制文件。这是重新同步主数据库和物理备用数据库所必需的。将物理备用控制文件复制到原始物理备用系统。为备用数据库创建控制文件描述了如何创建物理备用控制文件。
  8. 重新启动原始物理备用实例。

    如果此过程成功并且启用了存档间隙管理，则FAL进程会启动并将任何丢失的存档重做日志文件重新存档到物理备用数据库。强制主数据库上的日志切换并检查主数据库和物理备用数据库上的警报日志，以确保归档的重做日志文件序列号正确无误。

    有关存档间隙管理和设置存档跟踪的信息，请参阅手动间隙分辨率 ，以获取有关查找跟踪文件的信息。

  9. 再次尝试切换。

    此时，Oracle Data Guard配置已回滚到其初始状态，您可以再次尝试切换操作（在纠正可能导致初始切换失败的任何问题之后）。

  2. 删除或重命名任何旧网关目录。

升级和迁移注意事项

升级注意事项如下：

  • PGAU 来自Gateway版本8或9的控制文件是向上兼容的，您无需更改它们。

  • 升级后，PG数据字典（PG DD ）包含所有早期条目而不进行修改。添加了新的PGAU控制信息以及一些列以支持新功能，但升级不会更改任何客户条目。

  • 所有必须重新编译Oracle Database Gateway for APPC版本4.0.1或更早版本的TIP，因为以下内容发生了更改：

    • PL / SQL兼容性

    • 网关服务器RPC接口

    • UTL_PG接口

  • 如果您以前使用SNA通信协议在网关上生成的现有TIP并且您希望使用新的TCP / IP功能，则必须由PGAU使用强制NLS_LANGUAGE和Side Profile Settings重新生成TIP。在DEFINE TRANSACTION命令中指定适当的ASCII字符集。

    这是因为网关假定正在使用IMS Connect中的用户出口，这将在ASCII和EBCDIC字符集之间进行转换。

    警告：

    升级了早期版本的PGAU无法访问PG数据字典（PG DD）。

关于卸载工具

安装前，安装介质中提供了卸载工具（ deinstall ），安装后可在Oracle主目录中使用。它位于路径$ORACLE_HOME/deinstall 。

deinstall命令停止Oracle软件，并删除操作系统上的Oracle软件和配置文件。

该命令使用以下语法，其中变量内容以斜体表示：

deinstall -home home的完整路径 [-silent] [-checkonly] [-local] [-paramfile 输入参数属性文件的完整路径 ] [-params name1 = value name2 = value 。 。 。] [-o 保存文件的目录的完整路径 ] [-help | -H]

选项是：

  • -无声

    使用此标志以静默或响应文件模式运行命令。如果使用-silent标志，则必须使用-paramfile标志，并提供一个参数文件，其中包含要卸载或取消配置的Oracle主目录的配置值。

    您可以通过使用-checkonly标志运行deinstall来生成要使用或修改的参数文件。然后， deinstall命令会发现要从中卸载和取消配置的Oracle主目录中的信息。它会生成属性文件，然后您可以将其与-silent选项一起使用。

    您还可以修改位于响应文件夹中的模板文件deinstall.rsp.tmpl 。

  • -checkonly

    使用此标志可以检查Oracle软件主目录配置的状态。使用-checkonly标志运行该命令不会删除Oracle配置。-checkonly标志生成一个可以与deinstall命令一起使用的参数文件。

  • -本地

    在多节点环境中使用此标志可在群集中取消配置Oracle软件。

    使用此标志运行deconfig时，它会在本地节点（运行deconfig的节点）上取消配置并卸载Oracle软件。在远程节点上，它取消配置Oracle软件，但不卸载Oracle软件。

  • -paramfile complete path of input parameter property file

    使用此标志可以使用默认位置以外的参数文件运行deconfig。使用此标志时，请提供参数文件所在的完整路径。

    参数文件的默认位置取决于deconfig的位置：

    • 从安装介质或阶段位置： $ORACLE_HOME/inventory/response

    • 来自Oracle Technology Network的解压缩归档文件： /ziplocation/response

    • 从已安装的Oracle主目录安装后： $ORACLE_HOME/deinstall/response

  • -params [name1= value name 2= value name3= value . . .]

    将此标志与参数文件一起使用，以覆盖要在已创建的参数文件中更改的一个或多个值。

  • -o complete path of directory for saving response files

    使用此标志提供除保存属性文件（ deinstall.rsp.tmpl ）的默认位置以外的路径。

    参数文件的默认位置取决于deconfig的位置：

    • 安装前从安装介质或阶段位置： $ORACLE_HOME/

    • 来自Oracle Technology Network的解压缩归档文件： /ziplocation/response/

    • 从已安装的Oracle主目录安装后： $ORACLE_HOME/deinstall/response

  • -help | -H

    使用help选项（ -help或-h ）获取有关命令选项标志的其他信息。

使用ILM管理Oracle数据库中的数据

借助信息生命周期管理（ILM），您可以使用适用于该数据的规则和规则来管理Oracle数据库中的数据。

今天的信息有各种类型，例如电子邮件消息，照片或在线事务处理（OLTP）系统中的订单。在您了解数据的类型及其使用方式后，您就可以了解其演变和最终处置可能是什么。

每个组织面临的一个挑战是了解其数据如何发展和增长，监控其使用情况如何随时间变化，并决定它应该存活多久，同时遵守现在适用于该数据的所有规则和法规。信息生命周期管理（ILM）旨在通过流程，策略，软件和硬件的组合解决这些问题，以便可以在数据生命周期的每个阶段使用适当的技术。

关于Oracle数据库for ILM

Oracle数据库为实现ILM解决方案提供了理想的平台。

Oracle数据库平台提供以下功能：

  • 应用透明度

    应用程序透明度在ILM中非常重要，因为它意味着无需自定义应用程序，并且还可以对数据进行各种更改，而不会对使用该数据的应用程序产生任何影响。数据可以在其生命周期的不同阶段轻松移动，并且可以使用数据库优化对数据的访问。另一个重要的好处是，应用程序透明性提供了快速适应任何新法规要求所需的灵活性，同样对现有应用程序没有任何影响。

  • 细粒度数据

    Oracle可以在非常细粒度的级别上查看数据并对相关数据进行分组，而存储设备只能查看字节和块。

  • 低成本存储

    由于需要保留如此多的数据，因此使用低成本存储是实施ILM的关键因素。由于Oracle可以利用多种类型的存储设备，因此可以以最低的成本保留最大量的数据。

  • 可执行的合规性政策

    如果出于合规性原因保留信息，则必须向监管机构表明，数据是按照规定保留和管理的。在Oracle数据库中，可以定义安全性和审核策略，这些策略强制执行并记录对数据的所有访问。

Oracle数据库管理所有类型的数据

信息生命周期管理关注组织中的所有数据。

此数据不仅包括结构化数据，例如OLTP系统中的订单或数据仓库中的销售历史，还包括非结构化数据，如电子邮件，文档和图像。Oracle数据库支持使用BLOB和Oracle SecureFiles存储非结构化数据，Oracle Text中提供了一个复杂的文档管理系统。

如果组织中的所有信息都包含在Oracle数据库中，那么您可以利用数据库提供的特性和功能来管理和移动数据在其生命周期中的演变，而无需管理多种类型的数据存储。

法规要求

许多组织必须保留特定时间段的特定数据。不遵守这些规定可能导致组织不得不支付巨额罚款。

世界各地的各种监管要求，如美国的Sarbanes-Oxley，HIPAA，DOD5015.2-STD和欧盟的欧洲数据隐私指令，正在改变组织管理数据的方式。这些法规规定了必须保留哪些数据，是否可以更改数据，以及必须保留多长时间，这可能是30年或更长时间。

这些法规经常要求保护电子数据免受未经授权的访问和更改，并且对数据的所有更改以及由谁进行审计跟踪。Oracle数据库可以保留大量数据，而不会影响应用程序性能。它还包含限制访问和防止未经授权的数据更改所需的功能，并可通过Oracle Audit Vault和Database Firewall进一步增强。Oracle数据库还提供加密功能，可以证明高权限用户没有故意修改数据。使用闪回数据技术，您可以将行的所有版本存储在防篡改历史存档中。

在线档案的好处

在线存档有多种好处。

在数据的生命周期中，当数据不再被定期访问并且被认为有资格进行存档时，通常会出现这一点。传统上，数据将从数据库中删除并存储在磁带上，您可以以非常低的成本存储大量信息。今天，不再需要将数据存档到磁带，而是可以保留在数据库中，或者传输到中央在线存档数据库。所有这些信息都可以使用低成本存储设备进行存储，每千兆字节的成本非常接近磁带的成本。

将所有数据保存在Oracle数据库中用于存档目的有很多好处。最重要的好处是数据始终可以立即获得。因此，不会浪费时间来定位存档数据的磁带，并确定磁带是否可读并且仍然是可以加载到数据库中的格式。

如果数据已存档多年，则可能还需要开发时间来编写程序以从磁带存档将数据重新加载到数据库中。这可能证明是昂贵且耗时的，特别是如果数据非常陈旧的话。如果数据保留在数据库中，那么这不是问题，因为它是在线的，并且是最新的数据库格式。

保存数据库中的历史数据不再影响备份数据库所需的时间和备份大小。使用RMAN备份数据库时，它仅在备份中包含已更改的数据。由于历史数据更改的可能性较小，因此在备份数据后，不会再次备份。

另一个需要考虑的重要因素是如何从数据库中物理删除数据，特别是如果要将数据从生产系统传输到中央数据库存档。Oracle提供了使用可传输表空间或分区在数据库之间快速移动此数据的功能，这些表空间或分区将数据作为一个完整的单元移动。

当需要从数据库中删除数据时，最快的方法是删除一组数据。这是通过将数据保存在自己的分区中来实现的。可以删除分区，这是一个非常快速的操作。但是，如果由于必须维护数据关系而无法使用此方法，则必须发出传统的SQL删除语句。您不应低估发出删除声明所需的时间。

如果需要从数据库中删除数据，并且将来可能需要将数据返回到数据库，则考虑以数据库格式（如可传输表空间）删除数据，或使用XML Oracle数据库以开放格式提取信息的能力。

考虑将数据在线存档到Oracle数据库中，原因如下：

  • 磁盘的成本接近磁带的成本，因此您可以节省查找包含数据的磁带的时间以及恢复该数据的成本

  • 数据在需要时保持在线状态，为您提供更快的访问以满足业务需求

  • 在线数据意味着立即访问，因此监管机构对未能生成数据的罚款不太可能

  • 当前应用程序可用于访问数据，因此您无需浪费资源来构建新应用程序

第1步：定义数据类

要有效使用信息生命周期管理，请在实施信息生命周期管理解决方案之前首先查看组织中的所有数据。

查看数据后，确定以下内容：

  • 哪些数据很重要，哪里存储，哪些必须保留

  • 这些数据如何在组织内流动

  • 随着时间的推移，这些数据会发生什么变化，是否仍然需要

  • 所需的数据可用性和保护程度

  • 法律和业务要求的数据保留

在了解了数据的使用方式之后，可以在此基础上对数据进行分类。最常见的分类类型是年龄或日期，但其他类型也是可能的，例如产品或隐私。也可以使用混合分类，例如隐私和年龄。

要以不同方式处理数据类，必须将数据物理分离。首次创建信息时，通常会经常访问信息，但随着时间的推移，可能很少引用信息。例如，当客户下订单时，他们会定期查看订单以查看其状态以及订单是否已发货。订单到货后，他们可能永远不会再次参考该订单。此订单也将包含在常规报告中，以便查看正在订购的商品，但是，随着时间的推移，这些订单将无法在任何报告中显示，并且如果有人进行涉及此的详细分析，则可能仅在将来引用数据。例如，订单可以按财务季度Q1，Q2，Q3和Q4以及历史订单进行分类。

使用这种方法的优点是，当数据按行类别在行级别进行分组时（在此示例中为订单日期），Q1的所有订单都可以作为自包含单元进行管理，其中作为订单对于Q2将驻留在不同的类别。这可以通过使用分区来实现。由于分区对应用程序是透明的，因此数据在物理上是分开的，但应用程序仍然可以找到所有订单。

第2步：为数据类创建存储层

由于Oracle数据库可以利用许多不同的存储选项，因此实施信息生命周期管理解决方案的第二步是建立所需的存储层。

虽然您可以根据需要创建任意数量的存储层，但建议的起点是以下层：

  • 高性能

    高性能存储层是存储所有重要且经常访问的数据（例如保存Q1订单的分区）的地方。此层在高性能存储设备上使用更小，更快的磁盘。

  • 低成本

    低成本存储层是存储频率较低的数据的位置，例如保存Q2，Q3和Q4订单的分区。此层使用大容量磁盘构建，例如模块化存储阵列或低成本ATA磁盘中的磁盘，这些磁盘可提供最大量的廉价存储。

  • 在线档案

    在线存档存储层是存储很少访问或修改的所有数据的地方。此存储层可能非常大并且存储最大数量的数据。您可以使用各种技术来压缩数据。存储在低成本存储设备（如ATA驱动器）上，数据仍然可以联机并可用，其成本仅略高于将此信息存储在磁带上，而没有将数据存档到磁带所带来的缺点。如果将联机归档存储层标识为只读，则无法更改数据，并且在初始数据库备份后不需要后续备份。

  • 离线存档（可选）

    脱机归档存储层是可选层，因为它仅在需要从数据库中删除数据并以其他某种格式（如磁带上的XML）存储时使用。

第3步：创建数据访问和迁移策略

实施信息生命周期管理解决方案的第三步是确保只有授权用户才能访问数据并指定如何在数据生命周期内移动数据。

随着数据的老化，有多种技术可以在存储层之间迁移数据。

控制对数据的访问

数据的安全性是信息生命周期管理的另一个非常重要的部分，因为数据的访问权限可能会在其生命周期内发生变化。

此外，可能存在对如何访问数据提出严格要求的监管要求。

可以使用数据库功能保护Oracle数据库中的数据，例如：

  • 数据库安全

  • 查看

  • 虚拟专用数据库

虚拟专用数据库（VPD）定义了对数据库的非常细粒度的访问级别。安全策略确定可以查看哪些行以及可见的列。可以定义多个策略，以便不同的用户和应用程序看到相同数据的不同视图。例如，大多数用户可以看到Q1，Q2，Q3和Q4的信息，而只有授权用户才能查看历史数据。

安全策略在数据库级别定义，并透明地应用于所有数据库用户。这种方法的好处是它提供了一个安全和受控的环境来访问数据，这些环境不能被覆盖，并且可以在不需要任何应用程序更改的情况下实现。此外，可以定义只读表空间，以确保数据不会更改。

使用分区移动数据

在其生命周期中，必须移动数据并且分区是可以使用的技术。

移动数据可能由于以下原因而发生：

  • 为了提高性能，高性能磁盘上只保留有限数量的订单

  • 不再频繁访问数据并使用有价值的高性能存储，必须将其移至低成本存储设备

  • 法律要求要求信息在给定的时间间隔内始终可用，并且必须以尽可能低的成本安全地保存

有多种方法可以在Oracle数据库中物理移动数据，以利用不同的存储层。例如，如果数据已分区，则包含Q2订单的分区可以从高性能存储层在线移动到低成本存储层。由于数据在数据库中移动，因此可以在物理上移动数据，而不会影响需要它的应用程序或导致普通用户中断。

有时必须移动单个数据项而不是一组数据。例如，假设数据是根据隐私级别进行分类的，并且现在要向公众提供秘密的报告。如果分类从机密更改为公共，并且数据按其隐私分类进行分区，则该行将自动移动到包含公共数据的分区。

每当数据从其原始来源移动时，确保所选择的过程符合任何法规要求是非常重要的，例如，数据不能被更改，不受未经授权的访问，易于阅读和存储在批准的位置。

第4步：定义并实施合规性政策

信息生命周期管理解决方案的第四步是创建合规性策略。

当数据分散和分散时，必须在每个数据位置定义和实施合规性策略，这很容易导致忽略合规性策略。但是，使用Oracle数据库提供存储数据的中心位置意味着可以非常轻松地实施合规性策略，因为它们都是从一个中心位置进行管理和实施的。

在定义合规性策略时，请考虑以下方面：

  • 数据保留

  • 不变性

  • 隐私

  • 审计

  • 呼气

数据保留

保留策略描述了如何保留数据，保留数据的时间以及数据生命周期后发生的情况。

保留策略的一个示例是记录必须以其原始形式存储，不允许进行任何修改，必须保留七年，然后才能删除。使用Oracle数据库安全性，可以确保数据保持不变，并且只有授权的进程才能在适当的时间删除数据。还可以通过ILM Assistant中的生命周期定义来定义保留策略。

不变性

不变性涉及向外部方证明数据是完整的并且未被修改。

密码或数字签名可由Oracle数据库生成，并保留在数据库内部或外部，以显示数据未被更改。

隐私

Oracle数据库提供了几种确保数据隐私的方法。

使用虚拟专用数据库（VPD）定义的安全策略可以严格控制对数据的访问。此外，可以对单个列进行加密，以便查看原始数据的任何人都无法查看其内容。

审计

Oracle数据库可以跟踪对数据的所有访问和更改。

可以在表级别或通过细粒度审计来定义这些审计功能，该审计指定生成审计记录的条件。使用Oracle Audit Vault和Database Firewall可以进一步增强审计。

操作：指定ABORT或CONTINUE。

操作：确保查询的结束日期晚于开始日期。

PRGR-00172：无法获取客户端“{0}”的工作副本漂移元数据

PRGR-00173：凭据“{0}”不存在。
