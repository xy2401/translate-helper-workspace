6.1什么是Unicode标准？

Unicode标准是一种字符编码系统，它定义了世界上大多数口语中的每个字符。

为了克服现有字符编码的限制，一些组织在20世纪80年代后期开始致力于创建全局字符集。随着20世纪90年代中期万维网的发展，对此的需求变得更大。互联网改变了公司的业务方式，重点放在全球市场上，这使得普遍性的特征成为主要要求。

全局字符集需要满足以下条件：

  • 包含所有主要的生活脚本

  • 支持遗留数据和实施

  • 足够简单，应用程序的单个实现足以在全球范围内使用

全局字符集还应具有以下功能：

  • 支持多语言用户和组织

  • 符合国际标准

  • 实现全球数据交换

现在广泛使用的Unicode标准满足全局字符集的所有要求和功能。它为每个角色提供唯一的代码值，无论平台，程序或语言如何。它还定义了许多字符属性和处理规则，有助于正确一致地实现复杂的多语言文本处理。双向行为，断字和换行是这种复杂处理的例子。

Unicode标准已被许多软件和硬件供应商采用。许多操作系统和浏览器现在都支持该标准。其他标准（如XML，Java，JavaScript，LDAP和WML）都需要Unicode标准。它还与ISO / IEC 10646标准同步。

Oracle数据库在Oracle Database 7中引入了Unicode标准字符编码作为现在过时的数据库字符集AL24UTFFSS。从那时起，每个版本都进行了逐步改进，以使支持与新发布的标准版本同步。

6.2.1代码点和补充字符

Unicode标准的第一个版本是一个16位的固定宽度编码，它使用两个字节来编码每个字符。这样可以表示65,536个字符。但是，需要支持更多角色，尤其是对中国，日本和韩国市场非常重要的其他CJK表意文字。

Unicode标准的当前定义为标准中定义的每个字符分配一个数字。这些数字称为代码点，范围为0到10FFFF十六进制。用于表示字符代码点的Unicode表示法是前缀“U +”，后跟十六进制代码点值。代码点值用非重要零填充到最小长度为4。代码点U + 0000到U + FFFF的字符称为基本多语言平面字符。代码点U + 10000到U + 10FFFF的字符称为增补字符。

添加增补字符增加了Unicode 16位固定宽度编码格式的复杂性;然而，这仍远不如管理Unicode之前使用的数百种遗留编码那么复杂。

6.2.2 Unicode编码表单

Unicode标准定义了一些编码形式，它们是从Unicode代码点到代码单元的映射。代码单元是应用程序处理的整数值。代码单元可以具有8位，16位或32位。标准编码形式为：UTF-8，UTF-16和UTF-32。标准及其相关技术报告中还提到了两种兼容性编码：UCS-2和CESU-8。不同Unicode编码之间的转换是标准中定义的简单逐位操作。

6.2.2.1 UTF-8编码格式

UTF-8就是 Unicode的8位编码形式。它是一个可变宽度编码和ASCII的严格超集 。这意味着ASCII字符集中的每个字符都以UTF-8提供，具有相同的字节表示。一个Unicode字符可以用UTF-8编码形式的1字节，2字节，3字节或4字节表示。来自欧洲和中东脚本的字符以1或2个字节表示。大多数亚洲脚本中的字符以3个字节表示。补充字符以4个字节表示。

UTF-8是用于HTML和大多数Internet浏览器的Unicode编码。

UTF-8的好处如下：

  • 欧洲脚本的紧凑存储要求，因为它是ASCII的严格超集

  • 易于在基于ASCII的字符集和UTF-8之间进行迁移

6.2.2.2 UTF-16编码格式

UTF-16 是Unicode的16位编码形式。一个字符可以用UTF-16中的一个16位整数值（两个字节）或两个16位整数值（四个字节）表示。Basic Multilingual Plane中的所有字符（日常文本中使用的大多数字符）都以两个字节表示。补充字符以四个字节表示。编码单个补充字符的两个代码单元（整数值）称为代理对。

UTF-16是Java J2SE 5.0以及Microsoft Windows 2000以来用于Java内部处理的主要Unicode编码。

UTF-16优于UTF-8的好处如下：

  • 亚洲脚本的更紧凑的存储，因为大多数常用的亚洲字符用两个字节表示。

  • 与Java和Microsoft客户端的兼容性更好

6.2.2.3 UCS-2编码表

UCS-2不是官方的Unicode编码形式。在引入补充字符之前，该名称最初来自ISO / IEC 10646标准的旧版本。因此，它目前用于指代从支持补充字符和代理对而剥离的UTF-16编码形式。也就是说，代理对在UCS-2中被处理为两个单独的字符。支持UCS-2但不支持UTF-16的应用程序不应处理包含增补字符的文本，因为在将文本分成片段时，它们可能会错误地分割代理对。他们通常也无法显示此类文本。

UCS-2是用于Java J2SE 5.0之前和Microsoft Windows NT之前的内部处理的Unicode编码。

6.2.2.4 UTF-32编码格式

UTF-32是Unicode的32位编码形式。每个Unicode代码点由单个32位固定宽度整数值表示。如果是最简单的编码形式，但空间效率很低。对于英文文本，与UTF-8相比，它的存储要求增加了四倍，与UTF16相比，增加了一倍。因此，UTF-32有时在内部文本处理中用作中间形式，但通常不用于信息交换。

在Java中，从J2SE 5.0版开始，所选API已经过增强，可以对32位格式的字符进行操作，并以int值存储。

6.2.2.5 CESU-8编码表

CESU-8不是核心Unicode标准的一部分。它在The Unicode Consortium发布的Unicode技术报告＃26中有所描述。CESU-8是与UTF-8相同的兼容性编码形式，除了它表示补充字符。在CESU-8中，补充字符表示为代理对，如UTF-16。要获得补充字符的CESU-8编码，首先以UTF-16编码字符，然后将每个代理代码单元视为具有相同值的代码点。然后，将UTF-8编码规则（位转换）应用于每个代码点。这将产生两个三字节表示，总共六个字节。

CESU-8只有两个好处：

  • 它具有与UTF-16相同的二进制排序顺序。

  • 它每个字符使用相同数量的代码（一个或两个）。这对字符串处理中的字符长度语义很重要。

通常，应尽可能避免使用CESU-8编码形式。

6.3在数据库中实现Unicode解决方案

Unicode字符可以通过两种方式存储在Oracle数据库中：

  • 您可以创建一个数据库，使您可以将UTF-8编码的字符存储为SQL CHAR数据类型（ CHAR ， VARCHAR2 ， CLOB和LONG) 。

  • 您可以在SQL NCHAR数据类型（ NCHAR ， NVARCHAR2和NCLOB ）中以UTF-16或CESU-8编码格式存储Unicode数据。SQL NCHAR数据类型称为Unicode数据类型，因为它们仅用于存储Unicode数据。

注意：

如果在单个数据库中运行的不同应用程序需要，您可以组合使用两种Unicode解决方案。

以下部分介绍了如何使用这两种Unicode解决方案以及如何在它们之间进行选择：

  • 为数据库启用多语言支持

  • 使用Unicode数据类型启用多语言支持

  • 如何在Unicode解决方案之间进行选择

6.3.1为数据库启用多语言支持

数据库字符集指定要在SQL CHAR数据类型中使用的编码以及表名，列名和SQL语句等元数据。启用Unicode标准的数据库是具有符合Unicode标准的字符集作为数据库字符集的数据库。有两个实现Unicode标准的数据库Oracle字符集。

  • AL32UTF8

    AL32UTF8 字符集实现UTF-8编码格式，并支持最新版本的Unicode标准。它以一个，两个，三个或四个字节对字符进行编码。补充字符需要四个字节。它适用于基于ASCII的平台。

    AL32UTF8是Oracle数据库任何新部署的推荐数据库字符集，因为它为多语言应用程序提供了最佳支持，例如Internet网站和跨国公司的应用程序。

  • UTF8

    该 UTF8字符集实现CESU-8编码格式，并以一个，两个或三个字节对字符进行编码。它适用于基于ASCII的平台。

    插入UTF8数据库的补充字符以CESU-8编码形式存储。每个字符由两个三字节代码表示，因此总共占用六个字节的存储器。

    UTF8字符集中的字符属性不保证更新超出Unicode标准版本3.0。

    Oracle建议您切换到AL32UTF8以完全支持增补字符和最新版本的Unicode标准。

注意：

  • 创建数据库时指定数据库字符集。Oracle建议使用AL32UTF8作为数据库字符集。AL32UTF8是Unicode编码UTF-8的正确实现。从Oracle Database 12c第2版开始，AL32UTF8在使用Oracle Universal Installer（OUI）和Oracle Database Configuration Assistant（DBCA）创建数据库时用作默认数据库字符集。

  • 不要使用UTF8作为数据库字符集，因为它不是Unicode编码UTF-8的正确实现。如果在预期使用UTF-8处理的情况下使用UTF8字符集，则可能会发生数据丢失和安全问题。对于Web相关数据（例如XML和URL地址）尤其如此。

  • AL32UTF8和UTF8字符集彼此不兼容，因为它们具有不同的最大字符宽度。AL32UTF8的最大字符宽度为4个字节，而UTF8的最大字符宽度为3个字节。

  • 如果未明确在CREATE DATABASE语句中指定CHARACTER SET子句，则数据库字符集默认为US7ASCII（EBCDIC平台除外）。

6.3.2使用Unicode数据类型启用多语言支持

将Unicode数据存储在数据库中的替代方法是使用SQL NCHAR数据类型（ NCHAR ， NVARCHAR2 ， NCLOB ）。无论数据库字符集的定义方式如何，都可以将Unicode字符存储在这些数据类型的列中。NCHAR数据类型NCHAR Unicode数据类型，这意味着它存储以Unicode编码形式编码的数据。

Oracle建议在AL32UTF8数据库中使用SQL CHAR ， VARCHAR2和CLOB数据类型来存储Unicode字符数据。某些数据库功能不支持SQL NCHAR ， NVARCHAR2和NCLOB数据类型。最值得注意的是，Oracle Text和XML DB不支持这些数据类型。

您可以使用NVARCHAR2和NCHAR数据类型创建表。为NCHAR和NVARCHAR2列指定的列长度始终等于字符数而不是字节数：

CREATE TABLE product_information（product_id NUMBER（6），product_name NVARCHAR2（100），product_description VARCHAR2（1000））;

    为SQL NCHAR数据类型指定UTF8时，存储在SQL数据类型中的数据采用CESU-8编码格式。不推荐使用UTF8字符集。

6.3.3如何在Unicode解决方案之间进行选择

Oracle建议您在数据库字符集AL32UTF8中部署所有新的Oracle数据库，并使用SQL VARCHAR2 ， CHAR和CLOB数据类型来存储字符数据。仅在以下情况下才应考虑SQL NVARCHAR2 ， NCHAR和NCLOB数据类型：

  • 您有一个具有非Unicode数据库字符集和遗留应用程序的现有数据库，迁移到Unicode的业务成本将是不可接受的，您需要在应用程序的一小部分或在应用程序中添加对多语言数据的支持。一个小的新模块，单独的数据库没有多大意义，或者

  • 您需要创建一个必须支持多语言数据的应用程序，该应用程序必须可以安装在客户部署的任何Oracle数据库中。

对于启用Unicode标准的数据库中的数据库字符集，请始终选择AL32UTF8。对于国家字符集，请选择AL16UTF16。如果考虑选择不推荐使用的UTF8，因为英文字符数据的存储要求较低，请首先考虑其他选项，例如数据压缩或增加磁盘存储。如果数据库中积累了大量数据，以后迁移到AL16UTF16可能会很昂贵。

注意：

  • Oracle建议使用AL32UTF8作为数据库字符集。AL32UTF8是Unicode编码UTF-8的正确实现。从Oracle Database 12c第2版开始，AL32UTF8在使用Oracle Universal Installer（OUI）和Oracle Database Configuration Assistant（DBCA）创建数据库时用作默认数据库字符集。

  • 不要使用UTF8作为数据库字符集，因为它不是Unicode编码UTF-8的正确实现。如果在预期使用UTF-8处理的情况下使用UTF8字符集，则可能会发生数据丢失和安全问题。对于Web相关数据（例如XML和URL地址）尤其如此。

  • AL32UTF8和UTF8字符集彼此不兼容，因为它们具有不同的最大字符宽度。AL32UTF8的最大字符宽度为4个字节，而UTF8的最大字符宽度为3个字节。

场景1：具有Unicode标准启用数据库的Unicode解决方案

一个美国人运行Java应用程序的公司希望在下一版本的应用程序中添加德语和法语支持。他们想在以后添加日语支持。该公司目前有以下系统配置：

  • 现有数据库具有US7ASCII的数据库字符集。

  • 现有数据库中的所有字符数据都由ASCII字符组成。

  • PL / SQL存储过程在数据库中使用。

  • 数据库大约为300 GB， CLOB列中存储的数据非常少。

  • 每晚的停机时间为4小时。

在这种情况下，典型的解决方案是为数据库字符集选择AL32UTF8，原因如下：

  • 数据库非常大，计划的停机时间很短。将数据库快速迁移到Unicode字符集至关重要。由于数据库位于US7ASCII中，因此使数据库支持Unicode标准的最简单，最快捷的方法是使用Database Migration Assistant for Unicode（DMU）将数据库字符集切换为AL32UTF8。CLOB以外的列不需要数据转换，因为US7ASCII是AL32UTF8的子集。

  • 由于大多数代码都是用Java和PL / SQL编写的，因此将数据库字符集更改为AL32UTF8不太可能破坏现有代码。应用程序中会自动启用Unicode支持。

场景2：Unicode数据类型的Unicode解决方案

欧洲人主要在Windows平台上运行其遗留应用程序的公司希望添加一个用Visual C / C ++编写的新的小型Windows应用程序。新应用程序将使用现有数据库来支持日语和中文客户名称。该公司目前有以下系统配置：

  • 现有数据库的数据库字符集为WE8MSWIN1252。

  • 现有数据库中的所有字符数据均由西欧字符组成。

  • 数据库大约500 GB，有很多CLOB列。

  • 新应用程序不需要支持全文搜索和XML存储

典型的解决方案是采取以下措施：

  • 使用NCHAR和NVARCHAR2数据类型来存储Unicode字符

  • 将WE8MSWIN1252保留为数据库字符集

  • 使用AL16UTF16作为国家字符集

解决方案的原因是：

  • 将现有数据库迁移到Unicode数据库需要数据转换，因为数据库字符集是WE8MSWIN1252（Windows Latin-1字符集），它不是AL32UTF8的子集。此外，许多数据存储在CLOB列中。当从单字节数据库字符集（例如US7ASCII或WE8MSWIN1252）迁移到AL32UTF8时，必须转换数据库中的所有CLOB值，即使它们仅包含ASCII字符。因此，将数据转换为AL32UTF8会产生大量开销。

  • 仅在新应用程序中支持其他语言。它不依赖于现有的应用程序或模式。在新模式中使用Unicode数据类型并保持现有模式不变更为简单。

  • 只有客户名称列需要Unicode字符集支持。使用单个NCHAR列可满足客户的要求，而无需迁移整个数据库。

  • 新应用程序不需要不支持SQL NCHAR数据类型的数据库功能。

  • SQL NCHAR数据类型的长度定义为字符数。这与在Windows C / C ++程序中使用wchar_t字符串时的处理方式相同。这降低了编程复杂性。

  • 使用现有模式的现有应用程序不受影响。

6.5.2以多种语言存储数据

Unicode字符集包括世界上大多数书面语言的字符，但它不包含有关给定字符所属语言的信息。换句话说， ä等字符不包含有关瑞典语或德语字符的信息。为了以用户期望的语言提供信息，存储在Unicode数据库中的数据应该用数据所属的语言信息来标记。

数据库模式有许多方法可以将数据与语言相关联。以下部分讨论了实现此目标的示例步骤。

使用数据存储语言信息

WHERE language_id = SYS_CONTEXT（'userenv'，'LANG'）

6.5.3在LOB数据类型中以多种语言存储文档

您可以将文档以多种语言存储在CLOB ， NCLOB或BLOB数据类型中，并设置Oracle Text以启用文档的内容搜索。

当数据库字符集是多字节时， CLOB列中的数据存储在AL16UTF16字符集中，例如UTF8或AL32UTF8。这意味着转换数据时，英文文档所需的存储空间会翻倍。CLOB列中的亚洲语言文档的存储所需的存储空间比使用AL32UTF8的LONG列中的相同文档少，通常减少约30％，具体取决于文档的内容。

无论数据库字符集还是国家字符集， NCLOB格式的文档也存储在AL16UTF16字符集中。存储空间要求与CLOB数据相同。将文档内容插入NCLOB列时，文档内容将转换为UTF-16。如果要将多语言文档存储在非Unicode数据库中，请选择NCLOB 。但是，不支持使用Oracle Text对NCLOB进行内容搜索。

BLOB格式的文档按原样存储。插入和检索期间不会发生数据转换。但是，SQL字符串操作函数（如LENGTH或SUBSTR ）和排序规则函数（如NLS_SORT和ORDER BY ）不能应用于BLOB数据类型。

6.5.4.1创建多路复用器

创建多路复用器的第一步是为所支持的每种语言创建特定于语言的词法分析器首选项。以下示例使用PL / SQL过程创建英语，德语和日语词法分析器：

ctx_ddl.create_preference（'english_lexer'，'basic_lexer'）; ctx_ddl.set_attribute（ 'english_lexer'， 'index_themes'， '是'）; ctx_ddl.create_preference（'german_lexer'，'basic_lexer'）; ctx_ddl.set_attribute（ 'german_lexer'， '复合'， '德国'）; ctx_ddl.set_attribute（ 'german_lexer'， 'alternate_spelling'， '德语'）; ctx_ddl.set_attribute（ 'german_lexer'， 'mixed_case'， '是'）; ctx_ddl.create_preference（'japanese_lexer'，'JAPANESE_VGRAM_LEXER'）;

在创建特定于语言的词法分析器首选项之后，需要在单个多路复用器首选项下将它们聚集在一起。首先，使用MULTI_LEXER对象创建多路复用器首选项：

ctx_ddl.create_preference（ 'global_lexer'， 'multi_lexer'）;

现在使用add_sub_lexer调用将特定于语言的词法分析器添加到多路镜首选项：

ctx_ddl.add_sub_lexer（'global_lexer'，'german'，'german_lexer'）; ctx_ddl.add_sub_lexer（'global_lexer'，'japanese'，'japanese_lexer'）; ctx_ddl.add_sub_lexer（'global_lexer'，'default'，'english_lexer'）;

这提名了german_lexer首选项来处理德语文档， japanese_lexer首选项处理日语文档，以及english_lexer首选项来处理其他所有内容，使用DEFAULT作为语言。

6.5.4.2为CLOB数据类型中存储的文档创建索引

多路复用器根据表中的语言列决定每行使用哪个词法分析器。这是一个字符列，用于在文本列中存储文档的语言。使用Oracle语言名称标识此列中文档的语言。例如，如果使用CLOB数据类型来存储文档，则将语言列添加到存储文档的表中：

CREATE TABLE globaldoc（doc_id NUMBER PRIMARY KEY，语言VARCHAR2（30），文本CLOB）;

要为此表创建索引，请使用多路复用器首选项并指定语言列的名称：

CREATE INDEX globalx ON globaldoc（text）indextype IS ctxsys.context parameters（'lexer global_lexer language column language'）;
6.5.4.3为BLOB数据类型中存储的文档创建索引

除语言列外，还必须在存储文档的表中添加字符集和格式列。字符集列使用Oracle字符集名称存储文档的字符集。format列指定文档是文本文档还是二进制文档。例如， CREATE TABLE语句可以指定名为characterset和format列：

CREATE TABLE globaldoc（doc_id NUMBER PRIMARY KEY，语言VARCHAR2（30），characterset VARCHAR2（30），format VARCHAR2（10），text BLOB）;

您可以将文字处理或电子表格文档放入表格中，并在format列中指定binary 。对于HTML，XML和文本格式的文档，您可以将它们放入表中并在format列中指定text 。

由于存在用于指定字符集的列，因此可以将文本文档存储在不同的字符集中。

创建索引时，请指定格式和字符集列的名称：

CREATE INDEX globalx ON globaldoc（text）indextype是ctxsys.context参数（'filter inso_filter lexer global_lexer语言列语言格式列格式charset column characterset'）;

如果所有文档都是文本格式，则可以使用charset_filter 。charset_filter将数据从charset列中指定的charset转换为数据库字符集。

2错误处理和诊断

本章讨论适用于Pro * FORTRAN的错误报告和恢复。它补充了Oracle预编译程序员指南的第8章。

您将学习如何声明和使用SQLSTA状态变量和SQLCOD状态变量，以及如何包含SQL通信区域（SQLCA）。您还将学习如何声明和启用Oracle通信区域（ORACA）。

本章包含以下部分：

  • 错误处理替代方案

  • 关于MODE = {ANSI | ANSI14}时使用状态变量

2.1错误处理替代方案

Pro * FORTRAN预编译器支持四个状态变量，用作错误处理机制：

  • SQLCOD

  • SQLSTA

  • SQLCA（使用WHENEVER声明）

  • ORACA

预编译器MODE选项控制ANSI / ISO合规性。SQLCOD，SQLSTA和SQLCA变量的可用性取决于MODE设置。无论MODE设置如何，您都可以声明和使用ORACA变量。有关更多信息，请参阅“ 关于使用Oracle通信区 ” 。

当MODE = {ORACLE | ANSI13}时，您必须声明SQLCA状态变量。接受SQLCOD和SQLSTA声明（不推荐），但不会将其识别为状态变量。有关更多信息，请参阅“ 关于使用SQL通信区 ” 。

当MODE = {ANSI | ANSI14}时，您可以使用SQLCOD，SQLSTA和SQLCA变量中的任何一个，两个或全部三个。要确定哪种变量（或变量组合）最适合您的应用，请参阅“ 关于在MODE = {ANSI | ANSI14}时使用状态变量 ” 。

2.1.1 SQLCOD和SQLSTA

SQLCOD存储错误代码和“未找到”条件。它仅保留与SQL-89的兼容性，并已在SQL：1999及该标准的后续版本中删除。

与SQLCOD不同，SQLSTA存储错误和警告代码并使用标准化编码方案。执行SQL语句后，Oracle服务器将状态代码返回到当前范围内的SQLSTA变量。状态代码指示SQL语句是成功执行还是引发异常（错误或警告条件）。为了提高互操作性 （系统轻松交换信息的能力），SQL标准预定义了所有常见的SQL异常。

2.1.2 SQLCA

SQLCA是一种类似记录的主机语言数据结构。Oracle在每个可执行的 SQL语句之后更新SQLCA。（在声明性陈述之后，SQLCA值未定义。）通过检查存储在SQLCA中的Oracle返回码，您的程序可以确定SQL语句的结果。这可以通过两种方式完成：

  • 使用WHENEVER语句进行隐式检查

  • 显式检查SQLCA变量

您可以使用WHENEVER语句，对SQLCA变量进行显式检查，或者同时执行这两种操作。通常，使用WHENEVER语句是首选，因为它更容易，更便携，并且符合ANSI标准。

2.1.3 ORACA

当需要有关运行时错误的信息比SQLCA提供的更多信息时，可以使用ORACA，其中包含游标统计信息，SQL语句数据，选项设置和系统统计信息。

ORACA是可选的，无论MODE设置如何，都可以声明。有关ORACA状态变量的更多信息，请参阅“ 关于使用Oracle通信区 ” 。

2.2关于MODE = {ANSI | ANSI14}时使用状态变量

当MODE = {ANSI | ANSI14}时，您必须至少声明一个 - 您可以声明两个或全部三个 - 以下状态变量：

  • SQLCOD

  • SQLSTA

  • SQLCA

在检查可执行的SQL和PL / SQL语句之后，通过使用您自己的代码显式检查SQLCOD或SQLSTA，您的程序可以检索最新的可执行SQL语句的结果。您的程序也可以隐式检查SQLCA（使用WHENEVER SQLERROR和WHENEVER SQLWARNING语句），也可以显式检查SQLCA变量。

注意：

当MODE = {ORACLE | ANSI13}时，您必须声明SQLCA状态变量。有关更多信息，请参阅使用SQL通信区域。

2.2.1一些历史信息

Oracle Pro * FORTRAN预编译器对状态变量和变量组合的处理从1.5版开始发展。

2.2.2版本1.5

Pro * FORTRAN版本1.5假定存在状态变量SQLCOD，无论它是否在声明部分中声明;事实上，预编译器从未注意到SQLCOD是否被声明 - 它只是假定它是。仅当存在SQLCA的INCLUDE时，SQLCA才会用作状态变量。

2.2.3版本1.6

从Pro * FORTRAN版本1.6开始，预编译器不再假定存在SQLCOD状态变量且不需要。预编译器要求声明SQLCA，SQLCOD或SQLSTA 中的至少一个。

当且仅当满足以下条件中的至少一个时，SQLCOD才被识别为状态变量：

  • 它在Declare Section中声明，具有完全正确的数据类型。

  • 预编译器找不到其他状态变量。

如果预编译器在声明段找到SQLSTA声明（ 完全正确类型的课程），或者找到一个包括SQLCA的，它不会擅自SQLCOD声明。

2.2.4版本1.7

由于Pro * FORTRAN版本1.5允许在声明部分之外声明SQLCOD变量，同时声明SQLCA，因此Pro * FORTRAN版本1.6及更高版本会出现兼容性问题。添加了一个新选项ASSUME_SQLCODE = {YES | NO}（默认为NO）以在版本1.6.7中修复此问题，并将其记录为版本1.7中的新功能。

当ASSUME_SQLCODE = YES，并且当SQLSTA和/或SQLCA被声明为状态变量时，预编译器假定声明SQLCOD是否在声明部分或正确类型中声明。这导致版本1.6.7及更高版本在这方面的行为类似于版本1.5。有关预编译器选项ASSUME_SQLCODE的信息，请参见ASSUME_SQLCODE 。

2.2.5关于声明状态变量

本节介绍如何声明SQLCOD和SQLSTA。有关声明SQLCA状态变量的信息，请参阅“ 关于声明SQLCA ” 。

2.2.6声明SQLCOD

SQLCOD必须在Declare Section 内部或外部声明为4字节整数变量。在下面的示例中，SQLCOD在Declare Section之外声明：

*声明主机和指标变量。EXEC SQL BEGIN DECLARE SECTION ...EXEC SQL END DECLARE SECTION *声明状态变量。INTEGER * 4 SQLCOD

如果在Declare Section之外声明，仅当ASSUME_SQLCODE = YES时，SQLCOD才被识别为状态变量。当MODE = {ORACLE | ANSI13 | ANSI14}时，将忽略SQLCOD变量的声明。

访问本地SQLCOD受其程序范围的限制。在每次SQL操作之后，Oracle都会将状态代码返回到当前范围内的SQLCOD。因此，您的程序可以通过显式检查SQLCOD或使用WHENEVER语句隐式检查来了解最新SQL操作的结果。

在特定编译单元中声明SQLCOD而不是SQLCA时，预编译器会为该单元分配内部SQLCA。您的主程序无法访问内部SQLCA。

2.2.7声明SQLSTA

SQLSTA 必须被声明为定义段内的5个字符数的字母数字字符串，如显示在下面的例子：

EXEC SQL BEGIN DECLARE SECTION ...CHARACTER * 5 SQLSTA ......EXEC SQL END DECLARE SECTION

当MODE = {ORACLE | ANSI13}时，将忽略SQLSTA声明。声明SQLCA是可选的。

2.3.4状态代码

每个可执行的SQL语句都返回SQLCA变量SQLCDE中的状态代码，您可以使用WHENEVER SQLERROR隐式检查或使用您自己的FORTRAN代码显式检查。

最近执行的SQL语句处理的行数记录在SQLCA变量SQLERD（3）中。对于OPEN游标上的重复FETCH，SQLERD（3）保持获取的行数的运行总和。

2.3.7解析错误偏移量

在执行SQL语句之前，Oracle必须解析它;检查它以确保它遵循语法规则并引用有效的数据库对象。如果Oracle发现错误，则会在SQLCA变量SQLERD（5）中存储偏移量，您可以显式检查该变量。偏移量指定SQL语句中解析错误开始时的字符位置。第一个角色占据零位。例如，如果偏移量为9，则解析错误从第十个字符开始。

如果您的SQL语句不会导致解析错误，Oracle会将SQLERD（5）设置为零。如果解析错误从第一个字符开始，Oracle也将SQLERD（5）设置为零，该字符占据零位。因此，仅当SQLCDE为负时才检查SQLERD（5），这意味着发生了错误。

2.3.8错误消息文本

SQLCA变量SQLEMC中提供了错误代码和Oracle错误消息。例如，您可以将以下语句放在错误处理例程中：

2.3.10关于使用WHENEVER声明

默认情况下，Pro * FORTRAN预编译器会忽略Oracle错误和警告条件并继续处理（如果可能）。要执行自动条件检查和错误处理，您需要WHENEVER语句。

使用WHENEVER语句，您可以指定Oracle检测到错误，警告条件或“未找到”条件时要采取的操作。这些操作包括继续下一个语句，调用子例程，分支到带标签的语句或停止。

使用以下语法对WHENEVER语句进行编码：

EXEC SQL WHENEVER <condition> <action>

您可以让Oracle自动检查SQLCA是否存在以下任何一种情况，这些情况在错误处理和诊断中进行了描述：

  • SQLWARNING

  • SQLERROR

  • 未找到

当Oracle检测到上述条件之一时，您可以让程序执行以下任何操作：

  • 继续

  • DO subroutine_call

  • GOTO statement_label

  • 停

2.3.11范围

因为WHENEVER是一个声明性陈述，它的范围是位置的，而不是逻辑的。它测试源文件中跟随它的所有可执行SQL语句，而不是程序逻辑流程。因此，在要测试的第一个可执行SQL语句之前编写WHENEVER语句。

WHENEVER语句一直有效，直到被另一个检查相同条件的WHENEVER语句取代。

小费：

您可能希望将WHENEVER语句放在包含SQL语句的每个程序单元的开头。这样，一个程序单元中的SQL语句将不会在另一个程序单元中引用WHENEVER操作，从而在编译或运行时导致错误。

  • NULL - 捕获文件未加密

  • AES128 - 使用AES128加密捕获文件

  • AES256 - 使用AES256加密捕获文件

  • 成功调用START_CAPTURE后发送到数据库的所有用户请求都会在给定的dir目录中记录给定的持续时间，前提是指定了一个。如果未指定持续时间，则捕获将无限期持续，直到执行FINISH_CAPTURE过程 。

  • 工作负载捕获一旦开始，将继续记录跨数据库实例关闭和启动的用户请求达指定的持续时间，或直到执行FINISH_CAPTURE ，以先发生者为准。

  • 可以使用工作负载过滤器（如关于ADD_FILTER过程所描述的）来仅捕获发送到数据库的用户请求的子集。默认情况下，如果未定义工作负载筛选器，则会捕获所有用户请求。

  • 无论如何定义工作负载过滤器，都不会捕获从Oracle数据库后台进程（如SMON ， PMON ， MMON ）和Oracle数据库调度程序作业（如DBMS_SCHEDULER程序包中详述）启动的工作负载。这些活动应在适当配置的重播系统上自动进行。

  • 为工作负载定义明确的起点非常重要，这样在重新启动捕获的工作负载之前，可以将重放系统恢复到该点。要为工作负载捕获定义明确的起点，最好在执行START_CAPTURE时不要有任何活动的用户会话。如果正在进行的会话具有正在进行的事务，则在后续数据库重放中不会正确地重放这些事务，因为仅重放在START_CAPTURE之后执行调用的事务的那部分。

  • 对于加密捕获， START_CAPTURE过程依赖于软件密钥库。标识符为oracle.rat.database_replay.encryption （区分大小写）。

  • 您必须在auto-login模式下配置软件密钥库。否则，如果数据库在捕获期间被退回，则捕获将自动终止。

8.1.1在生产系统上启用SQL跟踪

Oracle9 i使用SQL Trace工具来收集各个SQL语句的性能数据。 SQL Trace生成的信息存储在SQL跟踪文件中。SQL Performance Analyzer从这些文件中使用以下信息：

  • 发生解析的SQL文本和用户名

  • 绑定每次执行的值

  • CPU和经过的时间

  • 物理读取和逻辑读取

  • 处理的行数

  • 每个SQL语句的执行计划（仅在SQL语句的游标关闭时捕获）

虽然有可能启用对于实例的SQL跟踪，建议您为会话子集启用SQL跟踪。为实例启用SQL跟踪工具后，实例中执行的所有SQL语句的性能统计信息将存储到SQL跟踪文件中。以这种方式使用SQL Trace会对性能产生严重影响，并可能导致系统开销增加，CPU使用率过高以及磁盘空间不足。这是必需的跟踪级别设置为4以捕获绑定值以及执行计划。

8.1.3构建SQL调优集

要构建SQL调优集：

  1. 将SQL跟踪文件复制到SQL Performance Analyzer系统上的目录。

  2. 为此目录创建目录对象。

  3. 使用DBMS_SQLTUNE 。 SELECT_SQL_TRACE函数从SQL跟踪文件中读取SQL语句。

    对于每个SQL语句，仅收集单个执行的信息。不捕获每个SQL语句的执行频率。因此，在对运行Oracle数据库10g第1版及更早版本的生产系统执行比较分析时，应忽略SQL Performance Analyzer报告中的工作负载级统计信息，并仅评估执行级别的性能更改。

8.1.1.1关于使用跟踪文件和警报日志监控错误

跟踪文件和警报日志包含有关错误的信息。

每个服务器和后台进程都可以写入关联的跟踪文件 。当进程检测到内部错误时，它会将有关错误的信息转储到其跟踪文件中。写入跟踪文件的某些信息适用于数据库管理员，其他信息适用于Oracle支持服务。跟踪文件信息还用于调整应用程序和实例。

注意：

严重错误还会在自动诊断信息库中创建事件和事件转储。有关详细信息，请参阅诊断和解决问题 。

警报日志是消息和错误的时间顺序日志，包括以下各项：

  • 发生的所有内部错误（ ORA-00600 ），块损坏错误（ ORA-01578 ）和死锁错误（ ORA-00060 ）

  • 管理操作，例如一些CREATE ， ALTER和DROP语句以及STARTUP ， SHUTDOWN和ARCHIVELOG语句

  • 与共享服务器和调度程序进程的功能相关的消息和错误

  • 在物化视图的自动刷新期间发生的错误

  • 在数据库和实例启动时具有非默认值的所有初始化参数的值

Oracle数据库使用警报日志记录这些操作，作为在操作员控制台上显示信息的替代方法（尽管某些系统还在控制台上显示信息）。如果操作成功，则会在警报日志中写入“已完成”消息以及时间戳。

警报日志作为XML格式的文件和文本格式的文件进行维护。您可以使用任何文本编辑器查看警报日志的任一格式，也可以使用ADRCI实用程序查看剥离了XML标记的XML格式的文件版本。

定期检查警报日志和跟踪实例的文件，以了解后台进程是否遇到错误。例如，当日志写入器进程（LGWR）无法写入日志组的成员时，会将指示问题性质的错误消息写入LGWR跟踪文件和警报日志。此类错误消息表示已发生媒体或I / O问题，应立即更正。

除了其他重要的统计信息之外，Oracle数据库还会将初始化参数的值写入警报日志。

后台和服务器进程的警报日志和所有跟踪文件将写入自动诊断存储库，其位置由DIAGNOSTIC_DEST初始化参数指定。跟踪文件的名称是特定于操作系统的，但每个文件通常包括写入文件的进程的名称（例如LGWR和RECO）。

8.1.1.3.1跟踪文件分段和MAX_DUMP_FILE_SIZE

Oracle数据库可以根据您使用MAX_DUMP_FILE_SIZE初始化参数指定的限制自动分段跟踪文件。达到限制时，数据库使用序号重命名当前跟踪文件，并使用原始名称创建空文件。

下表描述了如何根据MAX_DUMP_FILE_SIZE设置对跟踪文件进行分段。

表8-1 MAX_DUMP_FILE_SIZE参数和跟踪文件分段

MAX_DUMP_FILE_SIZE设置 跟踪文件分段

无限

跟踪文件未分段。

大于15M

跟踪文件在MAX_DUMP_FILE_SIZE设置的1/5的边界上进行分段。尺寸小于此边界的跟踪文件不会被分段。例如，如果MAX_DUMP_FILE_SIZE设置为100M ，则边界为20 MB（100 MB的1/5）。

15M或更少

跟踪文件未分段。

最多可以有五个段，但段的总组合大小不能超过MAX_DUMP_FILE_SIZE限制。当跟踪文件的所有段的组合大小超过指定的限制时，将删除第一个段之后的最旧段，并创建一个新的空段。因此，跟踪文件始终包含最新的跟踪信息。不会删除第一个段，因为它可能包含有关进程初始状态的相关信息。

分段可改善跟踪文件的空间管理。具体而言，分段使您可以通过以下方式管理跟踪文件：

  • 您可以在不再需要时清除旧跟踪文件。

  • 您可以使用较小的跟踪文件诊断问题，并隔离必须为事件打包服务（IPS）打包的跟踪文件。

注意：

任何涵盖事件的时间范围的段都不会被删除。它除了五个默认段之外还保留。

  • 实例的名称设置为inst1 。

  • 常量DBMS_ALERT.OBJECT_TYPE_SERVICE指定设置阈值的对象类型。在此示例中，服务名称为main.regress.rdbms.dev.us.example.com 。

8.2.1监控锁

锁是防止访问相同资源的事务之间的破坏性交互的机制。资源可以是用户对象（如表和行），也可以是用户不可见的系统对象，例如内存和数据字典行中的共享数据结构。

8.3.1关于对象隔离

对象隔离隔离了引发错误的对象，并监视对象是否对系统产生影响。

某些Oracle数据库错误（例如ORA-00600和ORA-07445 ）通常会导致进程中止，从而导致数据库中止。遇到此类错误时，对象隔离会尝试隔离导致错误的资源，以便数据库可以继续运行。资源在内存中是隔离的，因此它不会影响数据库的其余部分。V$QUARANTINE视图存储有关当前隔离的对象的信息。

大多数数据库资源都可能引发可能导致数据库中止的错误。例如，库高速缓存存储器对象可能引发此类错误。

在多租户环境中，多租户容器数据库（CDB）在某些情况下可以使用对象隔离来隔离和中止可插拔数据库（PDB），该数据库引发了严重错误而不是中止CDB。

在重新启动数据库之前，隔离的资源通常会保持隔离状态。如果资源被隔离为CDB中的PDB，则资源将被隔离，直到PDB关闭并重新打开。

11直方图

直方图是一种特殊类型的列统计信息，它提供有关表列中数据分布的更多详细信息。直方图将值分类为“桶”，因为您可以将硬币分类到桶中。

基于NDV和数据的分布，数据库选择要创建的直方图的类型。（在某些情况下，创建直方图时，数据库会对内部预定的行数进行采样。）直方图的类型如下：

  • 频率直方图和最高频率直方图

  • 高度平衡直方图（传统）

  • 混合直方图

本章包含以下主题：

  • 直方图的目的
    默认情况下，优化程序假定列中不同值的行的统一分布。
  • Oracle数据库创建直方图时
    如果DBMS_STATS收集表的统计信息，并且查询引用了此表中的列，则Oracle数据库会根据先前的查询工作负载根据需要自动创建直方图。
  • Oracle数据库如何选择直方图类型
    Oracle数据库使用多个条件来确定要创建的直方图：频率，顶部频率，高度平衡或混合。
  • 使用直方图时的基数算法
    对于直方图，基数算法取决于诸如端点数和值之类的因素，以及列值是流行还是非流行。
  • 频率直方图
    在频率直方图中 ，每个不同的列值对应于直方图的单个桶。因为每个值都有自己的专用存储桶，所以某些存储桶可能有很多值，而其他存储桶很少。
  • 顶部频率直方图
    顶部频率直方图是频率直方图的变化，其忽略统计上无关紧要的非流行值。
  • 高度平衡直方图（遗留）
    在传统的高度平衡直方图中，列值被分成桶，以便每个桶包含大致相同的行数。
  • 混合直方图
    混合直方图结合了基于高度的直方图和频率直方图的特征。这种“两全其美”的方法使优化器能够在某些情况下获得更好的选择性估计。

父主题： 优化程序统计信息

11.1直方图的目的

默认情况下，优化程序假定列中不同值的行的统一分布。

对于包含数据偏斜的列（列中数据的非均匀分布），直方图使优化器能够为涉及这些列的过滤器和连接谓词生成准确的基数估计。

例如，一家位于加利福尼亚的书店将95％的书籍运往加利福尼亚州，4％运往俄勒冈州，1％运往内华达州。图书订单表有300,000行。表列存储订单的发货状态。用户查询运往俄勒冈州的图书数量。如果没有直方图，优化器会假设均匀分布为300000/3（NDV为3），估计基数为100,000行。通过此估计，优化程序将选择全表扫描 。使用直方图，优化程序会计算出4％的图书被运送到俄勒冈州，并选择索引扫描。

父主题： 直方图

11.2 Oracle数据库创建直方图时

如果DBMS_STATS收集表的统计信息，并且查询引用了此表中的列，则Oracle数据库会根据先前的查询工作负载根据需要自动创建直方图。

基本流程如下：

  1. 您为一个表运行DBMS_STATS ，其METHOD_OPT参数设置为默认的SIZE AUTO 。

  2. 用户查询该表。

  3. 数据库记录前面查询中的谓词并更新数据字典表SYS.COL_USAGE$ 。

  4. 您再次运行DBMS_STATS ，导致DBMS_STATS查询SYS.COL_USAGE$以确定哪些列需要基于先前查询工作负载的直方图。

AUTO功能的后果包括以下内容：

  • 随着查询随时间的变化， DBMS_STATS可能会更改它收集的统计信息。例如，即使表中的数据没有更改，查询和DBMS_STATS操作也可能导致引用这些表的查询计划发生更改。

  • 如果收集表的统计信息而不查询表，则数据库不会为此表中的列创建直方图。要使数据库自动创建直方图，您必须运行一个或多个查询以填充SYS.COL_USAGE$的列使用信息。

SQL> CREATE TABLE sales2 AS SELECT * FROM sh_ext; SQL> CREATE INDEX sh_12c_idx1 ON sales2（prod_id）; SQL> CREATE INDEX sh_12c_idx2 ON sales2（cust_id，time_id）;

11.4使用直方图时的基数算法

对于直方图，基数算法取决于诸如端点数和值之类的因素，以及列值是流行还是非流行。

本节包含以下主题：

  • 端点号和值

  • 流行和非流行的价值观
    直方图中值的流行度影响基数估计算法。
  • 铲斗压缩
    在某些情况下，为了减少存储桶的总数，优化程序会将多个存储桶压缩到一个存储桶中。

父主题： 直方图

11.4.1端点号和值

端点号是唯一标识存储桶的编号。在频率和混合直方图中，端点号是当前和先前桶中包含的所有值的累积频率。

例如，端点号为100的存储桶表示当前和所有先前存储桶中的值的总频率为100。在高度平衡的直方图中，优化器按顺序编号，从0或1 。在所有情况下，端点号都是桶号。

端点值是存储桶中值范围中的最高值。例如，如果存储桶仅包含值52794和52795 ，则端点值为52795 。

父主题： 使用直方图时的基数算法

11.4.2流行和非流行价值观

直方图中值的流行度影响基数估计算法。

具体而言，基数估计受到如下影响：

  • 热门价值观

    流行值作为多个桶的端点值出现。优化器通过首先检查它是否是存储桶的端点值来确定值是否受欢迎。如果是，则对于频率直方图，优化器从当前桶的端点号中减去前一个桶的端点号。混合直方图已经为每个端点单独存储此信息。如果此值大于1，则该值很受欢迎。

    优化程序使用以下公式计算其热门值的基数估计值：

    流行值的基数=（表中的行数）*（此值跨越的端点数/端点总数）

  • 非流行的价值观

    任何不受欢迎的值都是非流行值 。优化程序使用以下公式计算非流行值的基数估计值：

  • 顶频直方图的标准不适用。

    这是说明前n个频繁值占据的行的百分比小于阈值p的另一种方式，其中p是(1-(1/ n ))*100 。

  • DBMS_STATS统计信息收集过程中的estimate_percent参数设置为AUTO_SAMPLE_SIZE 。

    如果用户指定自己的百分比，则数据库会创建频率或高度平衡的直方图。

应用程序设计和调优原理

Oracle开发人员必须设计，创建和调整数据库应用程序，以便实现安全性和性能目标。

以下应用程序设计和调优原则是有用的指导原则：

  • 了解Oracle数据库的工作原理

    作为开发人员，您希望在最少的时间内针对Oracle数据库开发应用程序，这需要利用数据库体系结构和功能。例如，不了解Oracle数据库数据并发控制和多版本读取一致性可能会使应用程序破坏数据的完整性，运行缓慢并降低可伸缩性。了解Transaction Guard和Application Continuity的工作原理可以避免编写不必要的异常处理代码。

  • 除非您有充分的理由不使用它们，否则请使用绑定变量

    当查询使用绑定变量时，数据库可以编译一次并将查询计划存储在共享池中 。如果再次执行相同的语句，则数据库可以执行软解析并重用该计划。相比之下， 硬解析需要更长时间并使用更多资源。使用绑定变量来允许软解析非常有效，并且是数据库打算让开发人员工作的方式。

  • 在数据库服务器而不是客户端中实现完整性约束

    使用主键和外键可以在多个应用程序中重用数据。在客户端中编写规则意味着其他客户端在针对数据库运行时无权访问这些规则。

  • 使用代表性数据和会话活动构建测试环境

    模拟您的实时生产环境的测试环境提供了多种好处。例如，您可以对应用程序进行基准测试，以确保它可以扩展并运行良好。此外，您可以使用测试环境来衡量更改对数据库的性能影响，并确保升级和修补程序正常工作。

  • 以良好的性能为目标设计数据模型

    通常，尝试使用通用数据模型会导致性能不佳。精心设计的数据模型尽可能高效地回答最常见的查询。例如，数据模型应使用提供最佳性能的索引类型。部署之后的调整是不合需要的，因为逻辑和物理结构的改变可能是困难的或不可能的。

  • 定义明确的绩效目标并保留指标的历史记录

    开发的一个重要方面是确切地确定应用程序的执行和扩展程度。例如，使用包含预期用户负载，每秒事务数，可接受响应时间等的度量标准。良好实践要求您保留性能指标的历史记录。通过这种方式，您可以主动和被动地监控性能。

  • 检测应用程序代码

    良好的开发实践涉及向应用程序添加调试代码。生成跟踪文件的功能对于调试和诊断性能问题很有用。

ODBC和JDBC

ODBC是一种标准API，它使应用程序能够连接到数据库，然后准备和运行SQL语句。

ODBC独立于编程语言，数据库和操作系统。ODBC的目标是使任何应用程序都能访问任何数据库中包含的数据。

数据库驱动程序是位于应用程序和数据库之间的软件。驱动程序将应用程序发出的API调用转换为数据库可以处理的命令。通过使用ODBC驱动程序，应用程序可以访问任何数据源，包括存储在电子表格中的数据。ODBC驱动程序执行ODBC标准和数据库之间的所有映射。

JDBC是一种低级Java接口，它使Java应用程序能够与Oracle数据库进行交互。与ODBC一样，JDBC是独立于供应商的API。 JDBC标准由Sun Microsystems定义，并通过java.sql接口实现。

以下区别在应用程序开发中很重要：

LOB概述

大对象（LOB）数据类型使您能够以二进制或字符格式存储和操作大块非结构化数据。

大对象（LOB）数据类型提供对数据的有效，随机，分段访问。

内部LOB

内部LOB将数据存储在数据库本身而不是外部文件中。

内部LOBS包括以下内容：

  • CLOB （字符LOB），用于在数据库字符集中存储大量文本，如文本或XML文件

  • NCLOB （国家字符集LOB），用于存储Unicode数据

  • BLOB （二进制LOB），它将大量二进制信息存储为比特流，不受字符集转换的影响

数据库以不同于其他数据类型的方式存储LOB。创建LOB列会隐式创建LOB段和LOB索引。包含LOB段和LOB索引的表空间始终存储在一起，可能与包含该表的表空间不同。

注意：

有时，数据库可以在表本身中存储少量LOB数据，而不是在单独的LOB段中。

LOB段将数据存储在称为块的块中 。块是逻辑上连续的数据块集，是LOB的最小分配单位。表中的一行存储一个名为LOB定位符的指针，该指针指向LOB索引。查询表时，数据库使用LOB索引快速定位LOB块。

数据库管理LOB段的读取一致性与其他数据不同。数据库不是使用撤消数据来记录更改，而是将前映像存储在段本身中。当事务更改LOB时，数据库会分配新块并保留旧数据。如果事务回滚，则数据库将更改回滚到索引，该索引指向旧块。

外部LOB

BFILE （二进制文件大对象，或LOB）是外部大对象。

BFILE是外部LOB，因为数据库存储指向操作系统中文件的指针。外部数据是只读的。

BFILE使用目录对象来定位数据。消耗的空间量取决于目录对象名称的长度和文件名的长度。

BFILE不使用与内部LOBS相同的读取一致性机制，因为二进制文件在数据库外部。如果文件中的数据发生更改，则从同一二进制文件重复读取可能会产生不同的结果。

SecureFiles的

SecureFiles LOB存储是两种存储类型之一;另一种类型是BasicFiles LOB存储。

SECUREFILE LOB参数启用高级功能，包括压缩和重复数据删除（高级压缩选项的一部分）和加密（高级安全选项的一部分）。从Oracle Database 12c开始 ，SecureFiles是LOB的默认存储机制。
