18.1.1优化器统计顾问的目的

Optimizer Statistics Advisor检查如何收集优化程序统计信息。

顾问程序会自动诊断现有收集统计信息的实践中的问题。该顾问不收取新的或者另一套优化的统计数据。顾问的输出是结果和建议的报告，它可以帮助您遵循收集统计数据的最佳实践。

优化程序统计信息在确定查询的执行计划中起着重要作用。因此，优化程序收集和维护准确和最新的统计信息至关重要。为此，优化器提供了DBMS_STATS包，该包从发行版发布到发行版。通常，用户根据特定工作负载开发自己的策略来收集统计信息，然后使用自行开发的脚本来实现这些策略。

  • Optimizer Statistics Advisor的优点
    与传统方法相比，基于顾问的方法提供了更好的可扩展性和可维护性。

18.1.1.1传统的基于脚本的方法的问题

脚本方法的优点是通常会测试和审查脚本。但是，次优遗留脚本的所有者可能不会更改它们，以免导致计划更改。

传统方法存在以下问题：

  • 旧版脚本可能无法跟上新的最佳实践，这些实践可能会随发行版的不同而变化。

    通常，后续版本会增加直方图，采样，工作负载监控，并发性以及其他与优化器相关的功能。例如，从Oracle Database 12c开始 ，Oracle建议设置AUTO_SAMPLE_SIZE而不是百分比。但是，旧脚本通常会指定采样百分比，这可能会导致执行计划不理想。

  • 资源浪费在不必要的统计数据收集上。

    脚本可以在同一个表上每天多次收集统计信息。

  • 自动统计信息收集作业无法保证准确和最新的统计信息。

    例如，有时自动统计信息收集作业未运行，因为初始化参数组合会禁用它，或者作业终止。此外，由于资源限制，或者因为太多对象需要统计信息收集，有时自动作业维护窗口不足。在收集所有统计信息之前停止运行的作业会导致某些对象无统计信息或过时统计信息，从而导致计划不理想。

  • 统计信息有时可能会丢失，陈旧或不正确。

    例如，表与其索引之间或具有主键 - 外键关系的表之间的统计信息可能不一致。或者，统计信息收集作业可能已被意外禁用，或者您可能不知道脚本已失败。

  • 缺乏对问题的了解可能是耗时且资源密集的。

    例如，服务请求可能会寻求解决问题，而不会意识到问题是由次优统计引起的。诊断可能需要花费大量时间通过电子邮件发送有问题的查询脚本，启用跟踪和调查跟踪。

  • 推荐的修复可能不可行。

    性能工程师可能会建议更改维护统计信息的应用程序代在一些组织中，这一要求可能难以或不可能满足。

18.1.1.2优化器统计顾问的优点

与传统方法相比，基于顾问的方法提供了更好的可扩展性和可维护性。

如果最新实践在新版本中发生更改，则Optimizer Statistics Advisor会在其规则中对这些实践进行编码。通过这种方式，顾问始终提供最新的建议。

顾问程序分析您当前如何收集统计信息（使用手动脚本，显式设置参数等），现有统计信息收集作业的有效性以及收集的统计信息的质量。优化程序统计顾问不收集新的或另一套优化的统计，所以不影响工作量。相反，Optimizer Statistics Advisor会分析存储在数据字典中的信息，然后将结果和建议存储在数据库中。

Optimizer Statistics Advisor与传统方法相比具有以下优势：

  • 提供易于理解的报告

    顾问应用规则来生成结果，建议和操作。

  • 提供脚本以实现必要的修复， 而无需更改应用程序代码

    实施建议的操作时，每次执行改进的语句都会产生收益。例如，如果您设置全局首选项以使样本大小为AUTO_SAMPLE_SIZE而不是次优百分比，那么基于改进统计信息的每个计划都可以从此更改中受益。

  • 在维护窗口中每天AUTO_STATS_ADVISOR_TASK一次名为AUTO_STATS_ADVISOR_TASK的预定义任务

    要运行自动作业， STATISTICS_LEVEL初始化参数必须设置为TYPICAL或ALL 。

  • 在DBMS_STATS包中提供API，使您可以手动创建和运行任务，将结果和建议存储在数据字典视图中，生成任务报告，并在必要时实施更正

  • 与现有工具集成

    顾问程序与SQL Tuning Advisor和AWR集成，后者汇总了Optimizer Statistics Advisor结果。

18.1.2.1.1优化器统计顾问的规则

Optimizer Statistics Advisor规则是Oracle提供的标准，Optimizer Statistics Advisor通过该标准执行检查。

这些规则体现了基于当前功能集的Oracle最佳实践。如果最佳实践在发行版之间发生变化，那么Optimizer Statistics Advisor规则也会发生变化。

顾问将规则组织到以下类中：

  • 系统

    此类检查统计信息收集的首选项，自动统计信息收集作业的状态，SQL计划指令的使用等。此类中的规则在V$STATS_ADVISOR_RULES.RULE_TYPE具有值SYSTEM 。

  • 手术

    此类检查统计信息收集是否使用默认值，使用SET_ * _STATS过程创建测试统计信息，等等。此类中的规则在V$STATS_ADVISOR_RULES.RULE_TYPE的值为OPERATION 。

  • 宾语

    此类检查统计信息的质量，统计信息的陈旧性，不必要的统计信息收集等。此类中的规则在V$STATS_ADVISOR_RULES.RULE_TYPE具有值OBJECT 。

规则检查以下问题：

  • 如何收集统计数据

    例如，一个规则可能指定初始化参数的建议设置。另一条规则可能指定应在架构级别收集统计信息。

  • 何时收集统计数据

    例如，顾问程序可能会建议应启用自动统计信息收集作业的维护窗口，或者应该扩展窗口。

  • 如何提高统计数据收集的效率

    例如，规则可能指定应在DBMS_STATS使用默认参数，或者不应手动设置统计信息。

在V$STATS_ADVISOR_RULES ，每个规则都有一个唯一的字符串ID，可在DBMS_STATS过程和报告中使用。您可以使用规则过滤器指定Optimizer Statistics Advisor应检查的规则。但是，您无法编写新规则。

18.1.2.1.2优化器统计顾问的结果

优化程序统计顾问检查存储在数据库中的证据并得出未遵循规则的结论时的结果。

要生成结果，Optimizer Statistics Advisor会执行一项任务，该任务会自动或手动调用。此任务分析存储在数据字典中的统计信息历史记录，统计信息操作日志以及SYSAUX中存在的当前统计信息足迹。例如，顾问程序查询DBA_TAB_STATISTICS和DBA_IND_STATISTICS以确定统计信息是否过时，或者行数之间是否存在差异。

通常，优化程序统计顾问会在未遵循或违反特定规则时生成查找结果，但某些结果（如对象陈旧性）仅提供信息。例如，一个发现可能显示DBMS_STATS.GATHER_TABLE_STATS使用了ESTIMATE_PERCENT=>0.01 ，这违反了ESTIMATE_PERCENT=>AUTO_SAMPLE_SIZE规则。

一个发现恰好对应一条规则。但是，规则可以产生许多发现。

18.1.2.1.3优化器统计顾问的建议

根据每个发现，Optimizer Statistics Advisor会就如何获得更好的统计数据提出建议。

例如，顾问程序可能会在收集统计信息时发现违反不使用抽样的规则，并建议改为指定AUTO_SAMPLE_SIZE 。顾问将建议存储在DBA_ADVISOR_RECOMMENDATIONS 。

单个发现可能存在多个建议。在这种情况下，您必须进行调查以确定要遵循的建议。每个建议都包含一个或多个基本原理，用于解释Optimizer Statistics Advisor为何提出建议。在某些情况下，调查结果可能不会产生建议。

18.1.2.1.4优化器统计顾问的操作

Optimizer Statistics Advisor操作是实现建议的SQL或PL / SQL脚本。如果可行，建议有相应的行动。顾问程序将操作存储在DBA_ADVISOR_ACTIONS 。

例如，Optimizer Statistics Advisor执行执行以下步骤的任务：

  1. 检查规则

    顾问程序检查是否符合应避免过时统计的规则。

  2. 产生发现

    顾问发现许多对象没有统计信息。

  3. 生成推荐

    顾问建议收集没有统计信息的对象的统计信息。

  4. 产生行动

    顾问程序生成执行DBMS_STATS.GATHER_DATABASE_STATS的PL / SQL脚本，提供需要收集统计信息的对象列表。

18.1.2.2优化器统计顾问的操作模式

Optimizer Statistics Advisor支持自动和手动模式。

  • 自动化

    预定义任务AUTO_STATS_ADVISOR_TASK每天在维护窗口中自动运行一次。该任务作为自动优化器统计信息收集客户端的一部分运行。自动化任务会生成结果和建议，但不会自动实施操作。

    对于任何其他任务，您可以配置自动化任务，并生成报告。如果报告建议操作，则可以手动实施操作。

  • 手册

    您可以使用DBMS_STATS.CREATE_ADVISOR_TASK函数创建自己的任务，然后使用EXECUTE_ADVISOR_TASK过程随时运行它。

    与自动化任务不同，手动任务可以自动执行操作。或者，您可以将任务配置为生成PL / SQL脚本，然后可以手动运行该脚本。

18.2.3.1关于Optimizer Statistics Advisor的过滤器

过滤器是使用DBMS_STATS将Optimizer Statistics Advisor任务限制为用户指定的规则，模式或操作集。

过滤器可用于包含或排除特定结果集。例如，您可以将顾问程序任务配置为仅包含对sh架构的建议。此外，您可以排除所有违反规则的过时统计信息。过滤器的主要优点是能够忽略您不感兴趣的建议，并减少顾问任务的开销。

创建过滤器的最简单方法是单独或组合使用以下DBMS_STATS过程：

  • CONFIGURE_ADVISOR_OBJ_FILTER

    使用此过程包括或排除指定的数据库模式或对象。对象过滤器接受所有者名称和对象名称，并支持通配符（ % ）。

  • CONFIGURE_ADVISOR_RULE_FILTER

    使用此过程包括或排除指定的规则。通过查询V$STATS_ADVISOR_RULES获取规则名称。

  • CONFIGURE_ADVISOR_OPR_FILTER

    使用此过程包括或排除指定的DBMS_STATS操作。通过查询DBA_OPTSTAT_OPERATIONS获取操作的ID和名称。

对于前面的函数，您可以指定过滤器应用的操作类型： EXECUTE ， REPORT ， SCRIPT和IMPLEMENT 。您还可以组合类型，如EXECUTE + REPORT 。 Null表示过滤器适用于所有类型的顾问程序操作。

关于Oracle数据库 - Microsoft Active Directory集成

集中管理的用户提供与Microsoft Active Directory的更简单集成，以允许用户的集中身份验证和授权。

Active Directory服务器操作系统的最低版本要求是Microsoft Windows Server 2008 R2。

通过此集成，组织可以使用Active Directory通过单个目录以及其他信息技术服务集中管理多个Oracle数据库中的用户和角色。Active Directory用户可以使用存储在Active Directory中的凭据对Oracle数据库进行身份验证。Active Directory用户还可以使用Active Directory组与数据库用户（模式）和角色相关联。Microsoft Active Directory用户可以映射到独占或共享Oracle数据库用户（模式），并通过目录中的组成员身份与数据库角色相关联。当用户登录时，Oracle数据库会遵守Active Directory帐户策略（例如密码到期时间和在指定次数的失败登录尝试后锁定）。

在Oracle Database 18c第1版（18.1）之前，可以通过配置Oracle Enterprise User Security以及安装和配置Oracle Internet Directory（或Oracle Universal Directory），将数据库用户身份验证和授权与Active Directory集成。此体系结构仍然可用，并且将继续由必须在受信任数据库，复杂企业角色之间使用Oracle企业域和当前用户数据库链接以及具有单个位置来审核数据库访问权限和角色的用户使用。

大多数组织没有这些复杂的要求。相反，他们可以使用Active Directory集中管理的用户（CMU）。此集成适用于希望使用Active Directory作为其集中身份管理解决方案的组织。Oracle Net Naming Services继续像以前一样使用目录服务。

组织可以对CMU和Active Directory使用Kerberos，PKI或密码身份验证。将CMU与Active Directory一起使用可向后兼容当前支持的Oracle数据库客户端。这意味着LDAP绑定操作不用于密码身份验证，您需要将一个Oracle筛选器添加到Active Directory以及Active Directory架构的扩展以存储密码验证程序。使用Kerberos或PKI的组织无需添加过滤器或扩展Active Directory架构。

Oracle Database-Active Directory集成对以下类型的用户特别有用：

  • 当前使用强身份验证的用户，如Kerberos或公钥基础结构（PKI）。这些用户已经使用集中式身份管理系统

  • 目前使用Oracle企业用户安全性，Oracle Internet Directory，Oracle Unified Directory，Oracle Virtual Directory以及需要与Active Directory集成的用户。

如何使用Microsoft Active Directory集中管理用户

通过将Microsoft Active Directory用户和组直接映射到Oracle数据库用户和角色，可以实现集成。

使用Microsoft Active Directory的集中管理用户支持的用户

具有Active Directory的CMU支持专门映射的用户，映射到共享模式的用户和管理用户。

这些用户如下：

  • 使用共享模式访问Oracle数据库的目录用户。

    这种类型的目录用户可以通过成为映射到共享模式（数据库用户）的目录组的一部分来连接到数据库中的共享模式。使用共享模式允许对数据库用户进行集中式Active Directory管理，并且是使用独占模式的推荐最佳实践（下面将介绍）。即使只有一个用户与架构关联（例如，负责数据库备份的管理员），通过仅在Active Directory中进行更改而不是对所有管理员进行更改，更容易管理添加其他备份管理员或删除现有管理员相关数据库也是如此。

    将使用映射到Active Directory中的组的全局角色为用户提供适合其任务的其他权限。使用此设计，用户可以在组织内更改其任务，并通过Active Directory中的新组获得新的数据库权限。

    Active Directory用户可能会意外（或故意）成为Active Directory中多个组的成员，这些组映射到同一数据库上的不同共享模式。用户还可以具有到数据库模式的独占映射。如果用户在登录时有多个可能的架构映射，则适用以下优先规则：

    • 如果存在用户的独占映射，则该映射优先于任何其他共享映射。

    • 如果用户存在多个共享模式映射，则具有最低模式ID（ USER_ID ）的共享用户映射优先。

    Oracle建议每个用户只有一个可能的映射，因此不会发生意外的模式映射。

  • 专有映射的全局用户，他们是两层和三层应用程序中的常规Oracle数据库用户，或者是在数据库中具有直接权限授予的用户。

    Oracle建议您通过全局角色向这些用户授予权限。这种类型的权限授予通过集中管理用户的权限和角色来促进授权管理，而不必登录到每个数据库以更新用户的权限和角色。

  • 具有以下管理权限的管理全局用户： SYSDBA ， SYSOPER ， SYSBACKUP ， SYSDG ， SYSKM和SYSRAC 。

    您无法通过全局角色授予这些管理权限。要授予具有这些管理权限的Active Directory用户，必须将目录用户映射到已授予数据库用户帐户的系统管理权限的数据库用户（独占或使用共享模式）。

关于使用dsi.ora文件

您使用dsi.ora文件为集中管理的用户指定Active Directory服务器。

您必须手动创建dsi.ora文件以标识Active Directory服务器。dsi.ora文件为所有可插拔数据库提供Active Directory连接信息（如果它位于可放置ldap.ora文件的相同位置）。一个dsi.ora在PDB特定钱夹位置文件的优先级高于主dsi.ora仅针对该PDB文件。

注意：

创建dsi.ora文件时，Oracle数据库按以下顺序搜索它：

  1. 如果WALLET_LOCATION设置包含在sqlnet.ora文件中，则对于非多租户数据库或多租户数据库的根容器，Oracle会在sqlnet.ora指定的位置搜索它。对于多租户数据库的PDB，Oracle在WALLET_LOCATION_specified_in_sqlnet.ora / pdb_guid目录中的每个PDB钱包位置中搜索它。
  2. 如果sqlnet.ora文件中未包含WALLET_LOCATION设置，则Oracle数据库会在默认钱包位置中搜索它。
  3. 如果Oracle数据库无法在钱包位置找到dsi.ora ，则Oracle数据库dsi.ora以下顺序搜索它。这些是Oracle数据库搜索ldap.ora文件的相同位置。
    1. $LDAP_ADMIN环境变量设置
    2. $ORACLE_HOME/ldap/admin目录
    3. $TNS_ADMIN环境变量设置
    4. $ORACLE_HOME/network/admin目录

Oracle建议您在$ORACLE_BASE下使用可写文件的目录，而不是在$ORACLE_HOME 。从Oracle Database 18c开始，您可以选择将$ORACLE_HOME目录设置为只读。因此，您应该将dsi.ora文件放在$ORACLE_HOME之外的目录中，以适应将来版本的dsi.ora配置。

Oracle还建议您仅使用dsi.ora为集中管理的用户标识Active Directory服务器。如果dsi.ora和ldap.ora都在同一数据库中为Active Directory的集中管理用户配置，并且都位于同一目录中，则dsi.ora优先于ldap.ora文件。如果它们位于不同的目录中，则Oracle使用它在上面的位置优先级列表中找到的第一个来查找Active Directory服务器。如果第一个找到的dsi.ora或ldap.ora的目录服务器类型不是Active Directory，则不会启用集中管理的用户。

您可以为多租户数据库中的各个PDB指定dsi.ora文件。特定dsi.ora PDB的dsi.ora将覆盖该PDB的共享dsi.ora或ldap.ora的公共设置。不同的PDB可以连接到CMU的不同Active Directory服务器。单个PDB的dsi.ora文件与该PDB的钱包位于同一目录中。

如果设置了sqlnet.ora文件中的WALLET_LOCATION参数，则单个PDB的dsi.ora文件将位于WALLET_LOCATION_specified_in_sqlnet.ora / pdb_guid/目录中的每个PDB钱包中。

如果未设置sqlnet.ora文件中的WALLET_LOCATION参数，则多租户数据库中单个容器的默认钱包位置为$ORACLE_BASE/admin/ db_unique_name / pdb_guid /wallet/目录。要使每个PDB使用默认钱包位置，您不能在sqlnet.ora设置WALLET_LOCATION 。

要查找db_unique_name ，请连接到CDB根目录并执行以下查询：

SELECT DB_UNIQUE_NAME FROM V $ DATABASE;

要从CDB根目录中查找pdb_guid ，请执行以下查询：

SELECT PDB_NAME，GUID FROM DBA_PDBS;

在sqlnet.ora设置或不设置WALLET_LOCATION参数具有以下效果：

  • 如果未在sqlnet.ora设置WALLET_LOCATION ，则还可以将dsi.ora放在CDB根容器的默认wallet目录中，该容器位于$ORACLE_BASE/admin/ db_unique_name /wallet目录中。但是，这只会将CDB根容器连接到Active Directory，而不是整个CDB数据库。
  • 如果在sqlnet.ora中设置了WALLET_LOCATION ，那么您可以将dsi.ora放在该钱包位置，这也只会将CDB根容器连接到Active Directory，而不是整个CDB数据库。

为了保持向后兼容性（如果您的站点需要它），Oracle过滤器可以生成密码验证程序，以便与版本11g，12c和18c的Oracle数据库客户端一起使用。Oracle密码过滤器使用名为ORA_VFR_MD5 （对于WebDAV ）， ORA_VFR_11G （对于版本11g）和ORA_VFR_12C （对于版本12c和18c）的Active Directory组来确定要生成哪些Oracle数据库密码验证器。必须在Active Directory中创建这些组，才能为组成员用户生成Oracle密码验证程序。这些是单独的组，用于指定应为Active Directory用户生成哪些特定验证程序。例如，如果十个目录用户需要登录到仅与Oracle数据库版本18c和12c客户端通信的新创建的Oracle数据库版本18c数据库，则Active Directory组ORA_VFR_12C将有十个Active Directory用户作为成员。Oracle过滤器只有在使用Active Directory更改密码时才会为这十个Active Directory用户生成12C验证程序（18c验证程序与12c验证程序相同）。如果Active Directory用户不再需要登录Oracle数据库，为了清除为Active Directory用户生成的Oracle密码验证程序，请从任何ORA_VFR组中删除该用户，并ORA_VFR重置密码（或要求更改密码）用户。您也可以手动清除此用户的orclCommonAttribute属性。用户从ORA_VFR组中删除后，将不再生成Oracle密码验证程序。

关于为集中管理用户配置授权

您可以在Active Directory中管理数据库的用户授权。

大多数Oracle数据库用户将映射到共享数据库架构（用户）。这可以最大限度地减少在雇用目录用户，在公司内部更换作业或离开公司时必须在每个Oracle数据库中完成的工作。目录用户将分配给映射到Oracle数据库全局用户（架构）的Active Directory组。当用户登录数据库时，数据库将查询Active Directory以查找用户所属的组。如果部署使用共享模式，则其中一个组将映射到共享数据库模式，并且将为该用户分配该数据库模式。用户将拥有授予数据库模式的角色和权限。由于将为同一共享数据库模式分配多个用户，因此只应将最小的角色和特权集授予共享模式。在某些情况下，不应向共享模式授予任何特权和角色。将通过数据库全局角色为用户分配适当的角色和模式集。全局角色映射到Active Directory组。这样，即使将不同的用户映射到同一数据库共享模式，它们也可以具有不同的角色和特权。新雇用的用户将被分配到映射到共享模式的Active Directory组，然后分配给映射到全局角色的一个或多个其他组，以获得完成其任务所需的其他角色和特权。共享模式和全局角色的组合允许集中授权管理，并且对操作上的数据库进行最小的更改。必须首先为数据库配置映射到相应Active Directory组的共享模式和全局角色集，然后在Active Directory中进行用户授权管理。

Active Directory用户也可以专门映射到数据库全局用户。这需要数据库中的新用户直接映射到Active Directory用户。新用户和离开的用户将需要更新他们所属的每个数据库。

需要管理权限（如SYSOPER和SYSBACKUP Active Directory用户无法通过全局角色授予这些权限。管理权限只能授予架构，而不能授予角色。但即使在具有管理权限的这些情况下，共享模式也可用于提供用户授权管理的简便性。使用共享模式与SYSOPER权限将允许新用户可以很容易地加入到映射到架构中的Active Directory组SYSOPER ，而不必在数据库中创建一个新用户的模式。即使只有一个用户被分配到共享模式，它仍然可以集中管理。

使用全局角色向用户授予权限和角色时，请记住会话中启用的最大角色数为150。

授权支持以下类型的全局用户映射：

  • 映射共享全局用户，通过将目录组映射到共享模式，将用户分配给共享数据库模式（用户）的目录。作为组成员的目录用户可以通过此共享模式连接到数据库。使用共享模式允许在Active Directory中集中管理用户授权。

  • 独占全局用户映射，其中专用数据库用户专门映射到目录用户。不像共享数据库模式那样常见，通过使用SQL * Plus或模式用户为两层或三层应用程序创建此用户以进行直接数据库访问。Oracle建议您通过全局角色向这些用户授予数据库权限，这有助于授权管理。但是，这些用户也可以在Oracle数据库中拥有直接权限授予，但不建议这样做。这是因为两层和三层应用程序可以使用全局用户作为数据库架构，因此全局用户对架构对象具有作为所有者的完全数据库权限。

目录用户通常是多个组的成员。但是，只应将其中一个组映射到共享架构。

将目录组映射到共享数据库全局用户

数据库的大多数用户将通过目录组中的成员身份映射到共享全局数据库用户（架构）。

必须先创建Active Directory组，然后才能将数据库全局用户映射到该组。您可以在用户需要登录数据库之前随时将Active Directory用户添加到该组。在数据库端，您必须具有CREATE USER和ALTER USER权限才能执行这些映射。此配置可用于具有密码身份验证，Kerberos身份验证和公钥基础结构（PKI）身份验证方法的用户。
您可以将共享应用程序的相同数据库架构的用户分配到Active Directory组。共享Oracle数据库全局用户（即共享架构）映射到Active Directory组。这样，该组的任何Active Directory用户都可以通过该共享全局用户帐户登录数据库。虽然数据库全局用户帐户由组成员共享，但在数据库内跟踪和审核Active Directory用户的身份验证身份（Windows域及其samAccountName ）和企业身份（DN）。
  1. 以已被授予CREATE USER或ALTER USER系统特权的用户身份登录数据库实例。
  2. 使用指定Active Directory组的DN的IDENTIFIED GLOBALLY AS子句执行CREATE USER或ALTER USER语句。
    例如，要将production.examplecorp.com域的sales组织单位中名为widget_sales_group的目录组映射到名为WIDGET_SALES的共享数据库全局用户：
    CREATE USER widget_sales IDENTIFIED GLOBALLY AS'CN = widget_sales_group，OU = sales，DC = production，DC = examplecorp，DC = com';
    在所有成员widget_sales_group将被分配给widget_sales当他们登录到数据库共享模式。

Oracle数据库与Microsoft Active Directory帐户策略的集成

作为Oracle数据库 - Microsoft Active Directory集成的一部分，当Active Directory用户登录Oracle数据库时，Oracle数据库会强制执行Active Directory帐户策略。

Active Directory帐户策略设置包括密码策略，帐户锁定策略和Kerberos策略。Oracle数据库为Active Directory中的集中管理用户强制执行所有帐户策略。例如，Oracle会阻止具有帐户状态的Active Directory用户，例如password expired ， password must change ， account locked out或account disabled用以登录数据库。如果使用Kerberos身份验证，则Oracle会阻止具有过期Kerberos票证的Active Directory用户登录数据库。如果您使用密码身份验证，则在尝试使用不正确的密码登录Oracle数据库时，用户连续发出指定次数的失败尝试后，Active Directory用户帐户将在Active Directory上锁定指定的时间段。通过强制执行帐户锁定策略，Oracle可以有效防止针对Active Directory用户帐户的密码猜测攻击。

  • 使用UPDATE更改BFILE的引用目标。

目录名称规范

您必须具有CREATE ANY DIRECTORY系统特权才能创建目录。

路径名称不能包含两个点（“..”）。DIRECTORY对象的命名约定与表和索引的命名约定相同。也就是说，普通标识符以大写形式解释，但是分隔标识符按原样解释。例如，以下声明：

CREATE DIRECTORY“Mary_Dir”AS'/ usr / home / mary';

然后目录对象名称是' Mary_Dir '。调用BFILENAME时使用' SCOTT_DIR '和' Mary_Dir '。例如：

BFILENAME（'SCOTT_DIR'，'afile'）BFILENAME（'Mary_Dir'，'afile'）
在Windows平台上

在Windows平台上，目录名称不区分大小写。因此以下两个语句引用相同的目录：

CREATE DIRECTORY“big_cap_dir”AS“g：\ data \ source”; CREATE DIRECTORY“small_cap_dir”AS“G：\ DATA \ SOURCE”;

BFILE安全

BEFILE安全性涉及BFILE安全模型和相关的SQL语句。与BFILE安全性相关的主要SQL语句是：

  • SQL DDL： CREATE和REPLACE或ALTER DIRECTORY对象

  • SQL DML： GRANT和REVOKE DIRECTORY对象的READ系统和对象权限

所有权和特权

DIRECTORY对象是系统拥有的对象。

有关系统拥有的对象的更多信息，请参见Oracle数据库SQL语言参考 。Oracle数据库支持两个新的系统特权，这些特权仅授予DBA：

  • CREATE ANY DIRECTORY ：用于创建或更改DIRECTORY对象创建

  • DROP ANY DIRECTORY ：删除DIRECTORY对象

读取DIRECTORY对象的权限

DIRECTORY对象的READ权限使您可以读取位于该目录下的文件。DIRECTORY对象的创建者自动获得READ权限。

如果您已使用GRANT选项授予READ权限，则可以将此权限授予其他用户/角色，并将其添加到您的权限域。

注意：

READ权限仅在DIRECTORY 对象上定义，而不是在单个文件上定义。因此，无法为同一目录中的文件分配不同的权限。

它表示的物理目录可能具有或不具有Oracle Server进程的相应操作系统特权（在本例中为读取 ）。

DBA有责任确保以下内容：

  • 物理目录存在

  • 已对文件，目录和指向它的路径启用了对Oracle Server进程的读取权限

  • 在数据库用户访问文件的整个过程中，该目录仍然可用，并且读取权限保持启用状态

该权限只是意味着就Oracle服务器而言，您可以从目录中的文件中读取。在实际文件操作时，PL / SQL DBMS_LOB包和OCI API检查并强制执行这些权限。

警告：

因为CREATE ANY DIRECTORY和DROP ANY DIRECTORY权限可能会将服务器文件系统暴露给所有数据库用户，所以DBA应该谨慎地将这些权限授予普通数据库用户以防止安全漏洞。

目录使用指南

DIRECTORY功能的主要目标是为DBA提供一种简单，灵活，非侵入性但安全的机制，以管理对服务器文件系统中大文件的访问。但要实现此目标，DBA在使用DIRECTORY对象时遵循这些准则非常重要：

  • 不要将DIRECTORY对象映射到数据文件目录。不应将DIRECTORY对象映射到包含Oracle数据文件，控制文件，日志文件和其他系统文件的物理目录。篡改这些文件（意外或其他）可能会破坏数据库或服务器操作系统。

  • 只有DBA才应具有系统权限。应谨慎使用诸如CREATE ANY DIRECTORY （最初授予DBA）之类的系统特权，而不是不加区分地授予其他用户。在大多数情况下，只有数据库管理员才具有这些权限。

  • 授予DIRECTORY权限时请小心。DIRECTORY对象的权限应该小心地授予不同的用户。在向用户授予权限时，同样适用于使用WITH GRANT OPTION子句。

  • 数据库运行时，请勿删除或替换DIRECTORY对象。在数据库运行时，不应随意删除或替换DIRECTORY对象。如果发生这种情况，那么与此DIRECTORY对象关联的所有文件的所有会话的操作都将失败。此外，如果在成功关闭这些文件之前执行DROP或REPLACE命令，则在程序中丢失对这些文件的引用，并且在关闭会话之前不会释放与这些文件关联的系统资源。

    例如，PL / SQL用户唯一的办法就是运行一个调用DBMS_LOB的程序块。 FILECLOSEALL并重新启动文件操作，或完全退出其会话。因此，必须谨慎使用这些命令，最好是在维护停机期间。

  • 撤消用户对DIRECTORY对象的权限时要小心。使用REVOKE语句撤消用户对DIRECTORY对象的权限会导致来自用户会话的相关文件的所有后续操作失败。您必须重新获取关闭文件的权限，或者在会话中运行FILECLOSEALL并重新启动文件操作。

通常，使用DIRECTORY对象来管理文件访问是操作系统级别的系统管理工作的扩展。通过一些规划，可以将文件逻辑组织到具有Oracle进程的READ权限的合适目录中。

可以使用映射到这些物理目录的READ权限创建DIRECTORY对象，并授予对这些目录的访问权限的特定数据库用户。

BFILE定位器变量

A BFILE定位器变量的运行方式与任何其他自动变量相同。关于文件操作，它像文件描述符一样操作，作为大多数传统编程语言的标准输入/输出库的一部分。

这意味着一旦定义并初始化BFILE定位器，并打开此定位器指向的文件，所有后续操作直到关闭此文件必须使用此定位器或此定位器的本地副本在同一程序块中完成。

BFILE指南

使用BFILE时请注意以下准则：

  • 在相同的嵌套级别打开和关闭同一程序块中的文件。BFILE定位器变量可以像任何标量一样用作其他过程，成员方法或外部函数标注的参数。但是，建议您在同一嵌套级别打开和关闭同一程序块中的文件。

  • 在将对象刷新到数据库之前设置BFILE值。如果对象包含BFILE ，则必须在将对象刷新到数据库之前设置BFILE值，从而插入新行。换句话说，您必须在OCILobFileSetName()之后和OCIObjectNew()之前OCIObjectFlush() 。

  • 在插入或更新BFILE之前指示DIRECTORY对象名称和文件名。插入或更新BFILE而不指示DIRECTORY对象名称和文件名是错误的。

    此规则也适用于在插入或更新语句中对BFILE使用OCI绑定变量的用户。在发出insert或update语句之前，必须使用DIRECTORY对象名和文件名初始化OCI绑定变量。

  • 在插入或更新之前初始化BFILE

    注意：

    OCISetAttr()不允许用户将BFILE定位符设置为NULL 。

  • 在使用SQL插入或更新带有BFILE的行之前，必须将BFILE初始化为以下之一：

    • NULL （如果使用OCI绑定变量则不可能）

    • DIRECTORY对象名称和文件名

  • 路径名称在其规范中的任何位置都不能包含两个点（“..”）。文件名不能以两个点开头。

17.1.1什么是Oracle托管文件？

使用Oracle Managed Files简化了Oracle数据库的管理。Oracle Managed Files使您（DBA）无需直接管理组成Oracle数据库的操作系统文件。

使用Oracle Managed Files，您可以指定数据库在数据库对象级别自动创建，命名和管理文件的文件系统目录。例如，您只需要指定要创建表空间;您不需要使用DATAFILE子句指定表空间的数据文件的名称和路径。此功能适用于逻辑卷管理器（LVM）。

数据库内部使用标准文件系统接口根据以下数据库结构的需要创建和删除文件：

  • 表空间

  • 重做日志文件

  • 控制文件

  • 存档日志

  • 阻止更改跟踪文件

  • 闪回日志

  • RMAN备份

通过初始化参数，指定要用于特定类型文件的文件系统目录。然后，数据库确保在不再需要时创建和删除唯一文件（Oracle托管文件）。

此功能不会影响管理文件的创建或命名，例如跟踪文件，审计文件，警报日志和核心文件。

17.1.3什么是逻辑卷管理器？

逻辑卷管理器（LVM）是大多数操作系统都可用的软件包。有时它被称为逻辑磁盘管理器（LDM）。它允许将多个物理磁盘的各个部分组合成一个连续的地址空间，该地址空间作为一个磁盘显示在更高层的软件上。

LVM可以使逻辑卷具有比任何底层物理磁盘更好的容量，性能，可靠性和可用性特征。它使用镜像，条带化，串联和RAID 5等技术来实现这些特性。

某些LVM允许在创建逻辑卷后更改其特性，即使在使用它时也是如此。卷可以调整大小或镜像，也可以重新定位到不同的物理磁盘。

17.1.4什么是文件系统？

文件系统是在连续磁盘地址空间内构建的数据结构。文件管理器（FM）是一个操作文件系统的软件包，但它有时也称为文件系统。

所有操作系统都有文件管理器。文件管理器的主要任务是将磁盘空间分配和释放到文件系统中的文件中。

文件系统允许将磁盘空间分配给大量文件。每个文件都显示为Oracle数据库等应用程序的连续地址空间。这些文件在文件系统的磁盘空间内实际上可能不是连续的。可以创建，读取，写入，调整大小和删除文件。每个文件都有一个与之关联的名称，用于引用该文件。

文件系统通常构建在由LVM构建的逻辑卷之上。因此，特定文件系统中的所有文件都具有从底层逻辑卷继承的相同性能，可靠性和可用性特征。文件系统是由文件系统中的所有文件共享的单个存储池。如果文件系统空间不足，则该文件系统中的所有文件都不会增长。一个文件系统中的可用空间不会影响另一个文件系统中的空间。但是，某些LVM / FM组合允许在文件系统中添加或删除空间。

操作系统可以支持多个文件系统。构建多个文件系统以为不同文件提供不同的存储特性，以及将可用磁盘空间划分为不会相互影响的池。

17.1.5使用Oracle托管文件的好处

Oracle Managed Files提供了多种好处。

请考虑使用Oracle托管文件的以下好处：

  • 它们使数据库的管理更容易。

    无需创建文件名并定义特定的存储要求。一组一致的规则用于命名所有相关文件。文件系统定义存储的特征以及分配它的池。

  • 它们可以减少管理员指定错误文件所导致的损坏。

    每个Oracle托管文件和文件名都是唯一的。在两个不同的数据库中使用相同的文件是一个常见错误，可能导致非常大的停机时间和丢失已提交的事务。使用引用同一文件的两个不同名称是另一个导致严重损坏的错误。

  • 它们减少了过时文件所浪费的磁盘空间。

    Oracle Database在不再需要时会自动删除旧的Oracle Managed Files。大型系统中浪费了大量磁盘空间，因为没有人确定是否仍然需要特定文件。这还简化了删除磁盘上不再需要的文件的管理任务，并防止了删除错误文件的错误。

  • 它们简化了测试和开发数据库的创建。

    您可以最大限度地减少在文件结构和命名方面做出决策所花费的时间，并且您可以减少文件管理任务。您可以更好地专注于满足测试或开发数据库的实际要求。

  • Oracle Managed Files使便携式第三方工具的开发变得更加容易。

    Oracle Managed Files消除了在SQL脚本中放置特定于操作系统的文件名的需要。

17.1.6 Oracle托管文件和现有功能

使用Oracle Managed Files不会消除任何现有功能。

现有数据库能够像往常一样运行。新文件可以创建为托管文件，而旧文件则以旧方式管理。因此，数据库可以混合使用Oracle托管和非托管文件。

17.2.1启用Oracle托管文件的初始化参数

下表列出了允许使用Oracle Managed Files的初始化参数。

初始化参数 描述

DB_CREATE_FILE_DEST

定义默认文件系统目录或Oracle ASM磁盘组的位置，当创建操作中未指定文件规范时，数据库将创建数据文件或临时文件。如果未指定DB_CREATE_ONLINE_LOG_DEST_ n ， DB_CREATE_ONLINE_LOG_DEST_用作重做日志和控制文件的缺省位置。

DB_CREATE_ONLINE_LOG_DEST_ n

在创建操作中未指定文件规范时，为重做日志文件和控制文件创建定义默认文件系统目录或Oracle ASM磁盘组的位置。通过更改n ，您可以多次使用此初始化参数，其中n指定重做日志或控制文件的多路复用副本。您最多可以指定五个多路复用副本。

DB_RECOVERY_FILE_DEST

定义的位置快速恢复区，它是默认文件系统目录或Oracle ASM磁盘组，其中数据库在未使用格式选项时创建RMAN备份，未配置其他本地目标时的归档日志以及闪回日志。如果未指定DB_CREATE_ONLINE_LOG_DEST_ n ，还用作重做日志和控制文件的默认位置或重做日志和控制文件的多路复用副本。指定此参数时，还必须指定DB_RECOVERY_FILE_DEST_SIZE初始化参数。

这些参数指定的文件系统目录必须已存在;数据库不会创建它们。该目录还必须具有允许数据库在其中创建文件的权限。

只要没有为创建文件的操作明确指定位置，就会使用默认位置。数据库创建文件名，因此创建的文件是Oracle托管文件。

这两个初始化参数都是动态的，可以使用ALTER SYSTEM或ALTER SESSION语句进行设置。

17.2.2设置DB_CREATE_FILE_DEST初始化参数

DB_CREATE_FILE_DEST初始化参数指定重要数据库文件的位置。

在初始化参数文件中包含DB_CREATE_FILE_DEST初始化参数，以标识要创建的数据库服务器的缺省位置：

  • 数据文件

  • 临时文件

  • 重做日志文件

  • 控制文件

  • 阻止更改跟踪文件

您可以指定文件系统目录的名称，该目录将成为为这些实体创建操作系统文件的默认位置。以下示例将/u01/app/oracle/oradata为创建Oracle托管文件时使用的默认目录：

DB_CREATE_FILE_DEST ='/ u01 / app / oracle / oradata'

17.3.2 Oracle托管文件的命名方式

Oracle Managed Files的文件名符合文件命名的Optimal Flexible Architecture（OFA）标准。

注意：

本节中描述的命名方案仅适用于在操作系统文件系统中创建的文件。Oracle自动存储管理管理员指南中介绍了在Oracle自动存储管理（Oracle ASM）磁盘组中创建的文件的命名方案。

指定的名称旨在满足以下要求：

  • 数据库文件很容易与所有其他文件区分开来。

  • 一种数据库类型的文件很容易与其他数据库类型区分开来。

  • 文件明确与特定于文件类型的重要属性相关联。例如，数据文件名可以包括表空间名称以允许容易地将数据文件关联到表空间，或者存档的日志名称可以包括线程，序列和创建日期。

没有两个Oracle托管文件具有相同的名称。用于创建Oracle托管文件的名称由三个来源构成：

  • 默认创建位置

  • 根据文件类型选择的文件名模板。该模板还取决于操作系统平台以及是否使用Oracle自动存储管理。

  • 由Oracle数据库或操作系统创建的唯一字符串。这可确保文件创建不会损坏现有文件，并且文件不会被误认为是其他文件。

作为一个具体示例，Oracle Managed Files的文件名在Solaris文件系统上具有以下格式：

destination_prefix /o1_mf_%t_%u_.dbf

哪里：

  • destination_prefix是destination_location / db_unique_name / datafile

    哪里：

    • destination_location是DB_CREATE_FILE_DEST指定的位置

    • db_unique_name是目标数据库的全局唯一名称（ DB_UNIQUE_NAME初始化参数）。如果没有DB_UNIQUE_NAME参数，则使用DB_NAME初始化参数值。

  • ％t是表空间名称。

  • ％u是一个八字符的字符串，保证唯一性

例如，假设以下参数设置：

DB_CREATE_FILE_DEST = / u01 / app / oracle / oradata DB_UNIQUE_NAME = PAYROLL

那么示例数据文件名将是：

/u01/app/oracle/oradata/PAYROLL/datafile/o1_mf_tbs1_2ixh90q_.dbf

其他文件类型的名称类似。其他平台上的名称也类似，受限于平台的命名规则。

  • 如果仅为DB_RECOVERY_FILE_DEST指定了值，则数据库将在该位置创建一个控制文件。

17.4 Oracle托管文件的操作

只要使用文件名标识现有文件，就会在SQL语句中接受Oracle Managed Files的文件名。

与其他文件名一样，这些文件名存储在控制文件中，如果使用Recovery Manager（RMAN）进行备份和恢复，则存储在RMAN目录中。它们在可用于监视数据文件和临时文件的所有常用固定和动态性能视图中都可见（例如， V$DATAFILE或DBA_DATA_FILES ）。

以下是使用数据库生成的文件名的语句的一些示例：

SQL> ALTER DATABASE 2> RENAME FILE'/u01/oradata/mydb/datafile/o1_mf_tbs01_ziw3bopb_.dbf'3> TO'/u01/oradata/mydb/tbs0101.dbf'; SQL> ALTER DATABASE 2> DROP LOGFILE'/u01/oradata/mydb/onlinelog/o1_mf_1_wo94n2xi_.log'; SQL> ALTER TABLE emp 2> ALLOCATE EXTENT 3>（DATAFILE'/u01/oradata/mydb/datafile/o1_mf_tbs1_2ixfh90q_.dbf'）;

您可以像对应的非Oracle托管文件一样备份和还原Oracle托管数据文件，临时文件和控制文件。使用数据库生成的文件名不会影响逻辑备份文件（如导出文件）的使用。这对于表空间时间点恢复（TSPITR）和可传输表空间导出文件尤为重要。

在某些情况下，Oracle Managed Files的行为方式不同，包括删除文件或重命名文件的操作，以及涉及备用数据库的操作。

  • 删除数据文件和临时文件
    与不由数据库管理的文件不同，当删除Oracle托管数据文件或临时文件时，将从控制文件中删除文件名，并自动从文件系统中删除该文件。
  • 删除重做日志文件
    删除Oracle托管重做日志文件时，将删除其Oracle托管文件。您指定要删除的组或成员。
  • 重命名文件
    对于Oracle Managed Files，重命名文件的SQL语句实际上并不重命名操作系统上的文件，而是更改控制文件中的名称。
  • 管理备用数据库
    备用数据库中的数据文件，控制文件和重做日志文件可以由数据库管理。这与Oracle Managed Files是否在主数据库上使用无关。

    由于DATAFILE子句不存在且设置了DB_CREATE_FILE_DEST初始化参数，因此将在缺省文件系统（此方案中为/u01/oradata中创建SYSTEM表空间数据文件。文件名由数据库唯一生成。该文件是可自动扩展的，初始大小为100 MB，最大大小不受限制。该文件是Oracle托管文件。为SYSAUX表空间创建了类似的数据文件。

    由于不存在LOGFILE子句，因此会创建两个重做日志组。每个日志组都有两个成员，其中一个成员位于DB_CREATE_ONLINE_LOG_DEST_1位置，另一个成员位于DB_CREATE_ONLINE_LOG_DEST_2位置。文件名由数据库唯一生成。创建的日志文件大小为100 MB。日志文件成员是Oracle托管文件。

    同样，由于CONTROL_FILES初始化参数不存在，并且指定了两个DB_CREATE_ONLINE_LOG_DEST_ n初始化参数，因此会创建两个控制文件。位于DB_CREATE_ONLINE_LOG_DEST_1位置的控制文件是主控制文件;位于DB_CREATE_ONLINE_LOG_DEST_2位置的控制文件是多路复用的副本。文件名由数据库唯一生成。它们是Oracle托管文件。假设存在服务器参数文件，则生成CONTROL_FILES初始化参数。

    自动撤销管理模式中指定，但因为没有指定一个撤销表空间和DB_CREATE_FILE_DEST初始化参数设置，默认撤消的表空间UNDOTBS中所指定的目录中创建DB_CREATE_FILE_DEST 。数据文件是一个可自动扩展的20 MB数据文件。它是Oracle托管文件。

    最后，指定了名为dflttmp的默认临时表空间。因为DB_CREATE_FILE_DEST包含在参数文件中，所以dflttmp的临时文件是在该参数指定的目录中创建的。临时文件为100 MB，可自动扩展，最大大小不受限制。它是Oracle托管文件。

  3. 管理控制文件

    生成数据库时创建了控制文件，并将CONTROL_FILES初始化参数添加到参数文件中。如果需要，DBA可以使用CREATE CONTROLFILE语句重新创建控制文件或为数据库构建新文件。

    必须在DATAFILE和LOGFILE子句中使用正确的Oracle托管文件名。ALTER DATABASE BACKUP CONTROLFILE TO TRACE语句生成具有正确文件名的脚本。或者，可以通过从V$DATAFILE ， V$TEMPFILE和V$LOGFILE视图中进行选择来找到文件名。以下示例为示例数据库重新创建控制文件：

    CREATE CONTROLFILE REUSE DATABASE示例LOGFILE GROUP 1（'/ u02 / oradata / SAMPLE / onlinelog / o1_mf_1_0orrm31z_.log'，'/ u03 / oradata / Sample_onlinelog / o1_mf_1_ixfvm8w9_.log'），GROUP 2（'/ u02 / oradata / SAMPLE /onlinelog/o1_mf_2_2xyz16am_.log'，'/ u03 / oradata / Sample/onlinelog / o1_mf_2_q89tmp28_.log'）NORESETLOGS DATAFILE'/u01/oradata/SAMPLE/datafile/o1_mf_system_cmr7t30p_.dbf'，'/ u01 / oradata / SAMPLE / datafile / o1_mf_sysaux_cmr7t88p_.dbf'，'/ u01 / odata / Sample / dataata / o1_mf_sys_undo_2ixfh90q_.dbf'，'/ u01 / oradata / SAMPLE / dataata / o1_mf_dflttmp_157se6ff_.tmp'MAXLOGFILES 5 MAXLOGHISTORY 100 MAXDATAFILES 10 MAXINSTANCES 2 ARCHIVELOG;

    此语句创建的控制文件的位置由创建数据库时生成的CONTROL_FILES初始化参数指定。REUSE子句导致覆盖任何现有文件。

  4. 管理重做日志

    要创建一组新的重做日志文件，DBA可以使用ALTER DATABASE ADD LOGFILE语句。以下语句添加一个日志文件，其中包含DB_CREATE_ONLINE_LOG_DEST_1位置中的成员以及DB_CREATE_ONLINE_LOG_DEST_2位置中的成员。这些文件是Oracle托管文件。 

    SQL> ALTER DATABASE ADD LOGFILE;

    通过指定完整的文件名继续添加和删除日志文件成员。

    GROUP子句可用于删除日志组。在以下示例中，将自动删除与每个Oracle托管日志文件成员关联的操作系统文件。 

    SQL> ALTER DATABASE DROP LOGFILE GROUP 3;
  5. 管理表空间

    sample数据库中未来表空间创建的所有数据文件的缺省存储是DB_CREATE_FILE_DEST初始化参数（在此方案中为/u01/oradata ）指定的位置。在初始化参数DB_CREATE_FILE_DEST指定的文件系统中创建未指定文件名的任何数据文件。例如：

    SQL> CREATE TABLESPACE tbs_1;

    前面的语句创建一个表空间，其存储位于/u01/oradata 。创建一个初始值为100 MB的数据文件，它是可自动扩展的，具有无限的最大大小。数据文件是Oracle托管文件。

    删除表空间后，将自动删除表空间的Oracle Managed Files。以下语句删除了用于其存储的表空间和所有Oracle托管文件：

    SQL> DROP TABLESPACE tbs_1;

    第一个数据文件填满后，数据库不会自动创建新的数据文件。通过添加另一个Oracle托管数据文件，可以向表空间添加更多空间。以下语句在DB_CREATE_FILE_DEST指定的位置添加另一个数据文件：

    SQL> ALTER TABLESPACE tbs_1添加数据文件;

    可以通过更改初始化参数来更改默认文件系统。这不会更改任何现有数据文件。它只会影响未来的创作。这可以使用以下语句动态完成：

    SQL> ALTER SYSTEM SET DB_CREATE_FILE_DEST ='/ u04 / oradata';
  6. 存档重做信息

    对于Oracle托管文件，重做日志文件的归档与非托管文件的归档没有区别。可以使用LOG_ARCHIVE_DEST_ n初始化参数指定归档重做日志文件的文件系统位置。文件名基于LOG_ARCHIVE_FORMAT参数或其默认值形成。存档日志不是Oracle管理文件。

  7. 备份，还原和恢复

    由于Oracle托管文件与标准操作系统文件兼容，因此您可以使用操作系统实用程序来备份或还原Oracle托管文件。所有用于备份，还原和恢复数据库的现有方法都适用于Oracle托管文件。

  5. 管理表空间

17.5.3方案3：将Oracle托管文件添加到现有数据库

示例说明了如何将Oracle Managed Files添加到现有数据库。
