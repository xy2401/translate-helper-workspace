12.9 BREAK

句法

BRE [AK] [ON report_element [ action [ action ]]] ......

其中report_element的语法为{ column | expr | ROW | REPORT}

和动作的语法[SKI [P] n | [SKI [P]] PAGE] [ NODUP [LICATES ] | DUP [LICATES]]

指定报表中发生更改的位置以及要执行的格式化操作，例如：

  • 禁止显示给定列的重复值

  • 每次给定列值更改时跳过一行

  • 印花每次给定列值更改时或在报告结束时计算的数字。

    请参阅COMPUTE命令。

输入BREAK，不带任何条款列出当前的BREAK定义。

条款

在列 [ action [ action ]]

当您包含操作时，指定SQL * Plus在指定列（称为中断列）中发生中断时执行的操作 ）。 （ 列不能附加表或视图。要实现此目的，您可以在SQL语句中为该列添加别名。）中断是三个事件之一，列或表达式的值更改，行的输出或报表的结尾

省略操作时，BREAK ON列会禁止在列中打印重复值，并在报表中标记一个位置，其中SQL * Plus将执行您在相应的COMPUTE命令中指定的计算。

您可以指定ON 列一次或多次。如果指定多个 ON条款，如

在JOB_ID上的DEPARTMENT_ID SKIP页面上休息 - 在SALIAL SKIP 1上跳过1

第一个ON子句表示最外面的中断（在本例中为ON DEPARTMENT_ID），最后一个ON子句表示最内部中断（在本例中为ON SALARY）。SQL * Plus在输出的每一行中搜索指定的中断，从最外面的中断开始，然后按照输入子句的顺序继续到最里面。在该示例中，SQL * Plus搜索DEPARTMENT_ID值的更改，然后搜索JOB_ID，然后搜索SALARY。

接下来，SQL * Plus执行从为最里面的中断指定的操作开始的操作，并以相反的顺序向最外面的中断进行操作（在这种情况下，从ON SALARY的SKIP 1到ON DEPARTMENT_ID的SKIP PAGE）。SQL * Plus执行每个操作，包括为初始搜索中遇到的第一个中断指定的操作。

例如，如果在给定行中JOB_ID的值发生更改 - 但DEPARTMENT_ID和SALARY的值保持不变 - 在打印行之前，SQL * Plus会跳过两行（一行是SKIP 1 ON SALARY，另一行是SKIP 1 ON JOB_ID的结果）。

每当使用ON 列时 ，您还应该使用ORDER BY SQL SELECT中的子句命令。通常，BREAK命令中使用的列应该在ORDER BY子句中以相同的顺序出现（尽管ORDER BY子句中指定的所有列都不需要出现在BREAK命令中）。这可以防止在报告中无意义的点发生中断。

如果使用本节前面指定的BREAK命令，则以下SELECT命令将生成有意义的结果：

SELECT DEPARTMENT_ID，JOB_ID，SALARY，LAST_NAME FROM EMP_DETAILS_VIEW WHERE SALARY> 12000 ORDER BY DEPARTMENT_ID，JOB_ID，SALARY，LAST_NAME;

具有相同DEPARTMENT_ID的所有行在一个页面上一起打印，并且在该页面内，具有相同JOB_ID的所有行在组中打印。在每组作业中，具有相同SALARY的那些作业按组打印。LAST_NAME中的中断不会导致任何操作，因为LAST_NAME未显示在BREAK命令中。

在BREAK命令中，空值被认为彼此相等，但不等于其他任何内容。这与WHERE子句中的空值处理不同。

ON expr [ action [ action ]]

包含操作时，指定SQL * Plus在表达式值更改时要执行的操作。

如果省略行动，打破expr的抑制，其中SQL * Plus将执行你在相应的COMPUTE命令中指定的计算expr的重复值和标志的印刷。

您可以使用涉及一个或多个表列的表达式或分配给SQL SELECT中的报表列的别名或SQL * Plus COLUMN 命令。如果在BREAK命令中使用表达式，则必须完全按照SELECT命令中的显示输入expr 。例如，如果SELECT命令中的表达式为+ b，则不能在BREAK命令中使用b + a或（a + b）来引用SELECT命令中的表达式。

为ON 列提供的信息也适用于ON expr 。 

在行[ 行动 [ 行动 ]]

包含操作时，指定SQL * Plus在SQL SELECT时执行的操作命令返回一行。无论您在BREAK命令中指定的位置如何，ROW中断都将成为最里面的中断。当你在一行上BREAK时，你应该总是指定一个动作。 

ON REPORT [ 行动 ]

在报告中标记一个位置，其中SQL * Plus将执行您在相应的COMPUTE命令中指定的计算。使用BREAK ON REPORT与COMPUTE一起打印总计或其他“大”计算值。

无论您在BREAK命令中指定的位置如何，REPORT中断都将成为最外层的中断。

请注意，SQL * Plus不会跳过报告末尾的页面，因此您无法使用BREAK ON REPORT SKIP PAGE。

SKI [P] n

在打印发生中断的行之前跳过n行。除非PREFORMAT设置为ON，否则BREAK SKIP n在SET MARKUP HTML ON模式下不起作用。

[SKI [P]] PAGE

在打印中断发生的行之前，跳过定义为页面的行数。可以使用SET命令的PAGESIZE子句设置每页的行数。请注意，PAGESIZE仅更改SQL * Plus认为是页面的行数。因此，SKIP PAGE可能并不总是导致物理分页，除非您还指定了NEWPAGE 0。另请注意，如果要在报表中打印的最后一行数据之后有中断，SQL * Plus将不会跳过该页面。

当值与前一行中列的值重复时，打印空白而不是break列的值。

聚合函数

聚合函数根据行组而不是单行返回单个结果行。聚合函数可以出现在选择列表和ORDER BY和HAVING子句中。它们通常与SELECT语句中的GROUP BY子句一起使用，其中Oracle Database将查询的表或视图的行划分为组。在包含GROUP BY子句的查询中，选择列表的元素可以是聚合函数， GROUP BY表达式，常量或涉及其中一个的表达式。Oracle将聚合函数应用于每组行，并为每个组返回单个结果行。

如果省略GROUP BY子句，则Oracle会将选择列表中的聚合函数应用于查询的表或视图中的所有行。您可以在HAVING子句中使用聚合函数，以根据聚合函数的结果从输出中消除组，而不是根据查询的表或视图的各个行的值。

创建类型

目的

对象类型使用PL / SQL定义。因此，本节提供了一些常规信息，但有关语法和语义的详细信息，请参阅Oracle Database PL / SQL语言参考 。

使用CREATE TYPE语句创建对象类型 ， SQLJ对象类型 ，命名变量数组（ varray ）， 嵌套表类型或不完整对象类型的 规范 。您可以使用CREATE TYPE和CREATE TYPE BODY语句创建对象类型。CREATE TYPE语句指定对象类型的名称，其属性，方法和其他属性。CREATE TYPE BODY语句包含实现该类型的方法的代码。

注意：

  • 如果创建的对象类型的类型规范仅声明属性但没有方法，则无需指定类型体。

  • 如果创建SQLJ对象类型，则无法指定类型主体。该类型的实现被指定为Java类。

不完整类型是由转发类型定义创建的类型。它被称为“不完整”，因为它有一个名称，但没有属性或方法。它可以被其他类型引用，因此可以用于定义相互引用的类型。但是，必须先完全指定类型，然后才能使用它来创建表或对象列或嵌套表类型的列。

先决条件

要在自己的模式中创建类型，您必须具有CREATE TYPE系统特权。要在另一个用户的模式中创建类型，您必须具有CREATE ANY TYPE系统特权。您可以显式获取这些权限，也可以通过角色授予它们。

要创建子类型，您必须具有UNDER ANY TYPE系统特权或超类型的UNDER对象特权。

必须为该类型的所有者显式授予EXECUTE对象特权才能访问该类型定义中引用的所有其他类型，或者必须为类型所有者授予EXECUTE ANY TYPE系统特权。所有者无法通过角色获得这些权限。

如果类型所有者打算授予其他用户访问该类型的权限，则必须使用GRANT OPTION或具有ADMIN OPTION的EXECUTE ANY TYPE系统特权为所有者授予所有者EXECUTE对象特权。否则，类型所有者没有足够的权限将该类型的访问权限授予其他用户。

用户定义的数据类型声明为不可持久的数据类型

在创建数据类型时，可以将用户定义的数据类型指定为不可持久化。非持久性类型的实例不能在磁盘上持久存在。Perisistable数据类型包括以下内容：

  • ANSI支持的数据类型，例如NUMERIC ， DECIMAL ， REAL 。

  • Oracle内置数据类型，例如NUMBER ， VARCHAR2 ， TIMESTAMP 。

  • Oracle提供的数据类型，例如ANYDATA ， XML Type ， ORDImage 。

SQL用户定义数据类型的规则

  • 持久化类型不能具有非持久类型的属性或元素。

  • 非持久类型可以具有可持久类型和非持久类型的属性或元素。

  • 子类型必须从其超类型继承持久性属性。

  • REF类型是可持久的，并且只能保存对可持久类型对象的引用。

  • 您无法在磁盘上保留不可持久类型的实例。如果创建的表的类型已声明为不可持久化，则CREATE TABLE语句将失败。以下操作同样会失败：

    • 使用非持久类型的列创建或更改关系表。

    • 使用非可持久类型的列创建对象表。

    • 将不可持久类型的实例存储在磁盘上持久存储的ANYDATA实例中。

这些步骤描述了如何切换到物理备用数据库。

注意：

如果存在连接主数据库和备用数据库的远程同步实例（或首选远程同步实例和备用远程同步实例的组合），则切换到备用数据库的过程与本主题中描述的过程相同。远程同步实例是可用还是不可用不会影响切换。在切换期间，主服务器和备用服务器必须能够直接相互通信，并执行无法远程同步实例的切换角色转换步骤。有关如何正确设置此类配置的示例，请参阅Oracle Data Guard概念和管理中的 “使用远程同步实例”，以便远程同步实例可以在切换后为两个数据库的新角色提供服务。

  1. 验证目标备用数据库是否已准备好进行切换。

    新的switchover语句具有VERIFY选项，可以检查切换所需的许多条件。检查的一些项目是：Redo Apply是否在切换目标上运行;切换目标的发布版本是否为12.1或更高版本;切换目标是否同步;以及是否有MRP运行。

    假设主数据库具有BOSTON的DB_UNIQUE_NAME ，并且切换目标备用数据库具有CHICAGO的DB_UNIQUE_NAME 。在主数据库BOSTON ，发出以下SQL语句以验证切换目标CHICAGO是否已准备好进行切换：

    SQL> ALTER DATABASE SWITCHOVER到CHICAGO VERIFY;第1行的错误：ORA-16470：重做应用未在切换目标上运行

    如果此操作成功，则会返回Database Altered消息，但在此示例中返回了ORA-16470错误。此错误意味着切换目标CHICAGO尚未准备好进行切换。必须在切换操作之前启动重做应用。

    启动Redo Apply后，再次发出以下语句：

    SQL> ALTER DATABASE SWITCHOVER到CHICAGO VERIFY;第1行的错误：ORA-16475：成功发出警告，查看警报日志了解更多详情

    切换目标CHICAGO已准备好进行切换。但是， ORA-16475错误指示的警告可能会影响切换性能。警报日志包含类似于以下内容的消息：

    SWITCHOVER VERIFY警告：切换目标具有需要清除的脏的联机重做日志文件。清除在线重做日志文件需要时间。这可能会减慢切换过程。

    您可以解决问题，或者如果切换性能不重要，则可以忽略这些警告。在确定了必要的修复后，再次发出以下SQL语句：

    SQL> ALTER DATABASE SWITCHOVER到CHICAGO VERIFY;数据库改变了

    切换目标CHICAGO现在可以进行切换。

    如果发生错误，请装入旧的主数据库（ BOSTON ）和旧的备用数据库（ CHICAGO ）。在这两个数据库，查询DATABASE_ROLE从V$DATABASE 。 BOSTON和CHICAGO有三种可能的数据库角色组合。下表描述了这些组合，并为每种情况提供了可能的原因和高级补救措施。有关特定错误情况的详细信息，请参阅Oracle Data Guard概念和管理中的 “ Oracle Data Guard故障排除”。

    操作：有关阻止BOSTON切换到备用角色的错误的详细信息，请参阅警报日志，采取必要的操作来修复错误，必要时重新打开BOSTON其中一个节点，然后重复步骤1中的切换过程。

9.1寻找最佳属性

有时过多的信息会降低数据挖掘的效率。为构建和测试模型而组装的一些数据属性列不会为模型提供有意义的信息。有些确实会降低模型的质量和准确性。

例如，您希望收集有关特定人群的大量数据，因为您希望预测该组中某种疾病的可能性。其中一些信息（可能是其中的大部分）对疾病的易感性几乎没有影响。诸如每个家庭的汽车数量之类的属性可能无论如何都没有效果。

不相关的属性会给数据增加噪音并影响模型的准确性。噪声会增加模型的大小以及模型构建和评分所需的时间和系统资源。

具有许多属性的数据集可以包含相关的属性组。这些属性实际上测量相同的底层特征。它们在构建数据中的存在可能会扭曲算法的逻辑并影响模型的准确性。

宽数据（许多属性）通常呈现数据挖掘算法的处理挑战。模型属性是算法使用的处理空间的维度。处理空间的维数越高，算法处理所涉及的计算成本就越高。

为了最小化噪声，相关性和高维度的影响，某种形式的降维有时是数据挖掘的理想预处理步骤。特征选择和提取是降维的两种方法。

  • 特征选择 ：选择最相关的属性

  • 特征提取 ：将属性组合成一组新的特征

9.2关于特征选择和属性重要性

找出最重要的预测因素是一些数据挖掘项目的目标。例如，模型可能试图找出构成高信用风险的客户的主要特征。

Oracle Data Mining支持属性重要性挖掘功能，该功能根据属性在预测目标时的重要性对属性进行排名。属性重要性实际上不执行特征选择，因为所有预测变量都保留在模型中。在真实特征选择中，从模型中移除排名低于给定重要性阈值的属性。

特征选择可用作预处理步骤，以提高预测建模的计算效率。当启用自动数据准备（ADP）时，Oracle Data Mining在决策树算法和朴素贝叶斯内实现功能选择以进行优化。可以将广义线性模型（GLM）配置为执行特征选择作为预处理步骤。

9.2.1属性重要性和评分

Oracle Data Mining不支持属性重要性的评分操作。属性重要性的结果是根据其预测影响排序的构建数据的属性。排名和重要性度量可用于选择分类模型的训练数据。

9.3关于特征提取

特征提取是属性约简过程。不像功能选择，选择并保留最重要的属性，功能提取实际上转换属性。变换后的属性或特征是原始属性的线性组合。

特征提取过程会产生更小，更丰富的属性集。功能的最大数量可由用户指定或由算法确定。默认情况下，算法会确定它。

基于提取的特征构建的模型可以具有更高的质量，因为更少和更有意义的属性描述数据。

特征提取将具有较高维度的数据集投影到较小数量的维度上。因此，它对于数据可视化是有用的，因为当复杂数据集减少到二维或三维时，可以有效地可视化。

特征提取的一些应用是潜在语义分析，数据压缩，数据分解和投影以及模式识别。特征提取还可用于提高监督学习的速度和有效性。

特征提取可用于提取文档集合的主题，其中文档由一组关键字及其频率表示。每个主题（特征）由关键字组合表示。然后可以根据发现的主题来表示集合中的文档。

9.3.1特征提取和评分

Oracle Data Mining支持特征提取的评分操作。作为无监督挖掘函数，特征提取不涉及目标。应用时，特征提取模型将输入转换为一组要素。

9.4属性重要性和特征提取算法

了解用于属性重要性和特征提取的算法。

Oracle Data Mining支持以下属性重要性算法：

  • 最小描述长度

  • CUR矩阵分解

Oracle Data Mining支持以下特征提取算法：

  • 显式语义分析 （ESA） 。

  • 非负矩阵分解 （NMF） 。

  • 奇异值分解 （SVD）和预测成分分析 （PCA） 。

8.1寻找最佳属性

有时过多的信息会降低数据挖掘的效率。为构建和测试模型而组装的一些数据属性列不会为模型提供有意义的信息。有些确实会降低模型的质量和准确性。

例如，您希望收集有关特定人群的大量数据，因为您希望预测该组中某种疾病的可能性。其中一些信息（可能是其中的大部分）对疾病的易感性几乎没有影响。诸如每个家庭的汽车数量之类的属性可能无论如何都没有效果。

不相关的属性会给数据增加噪音并影响模型的准确性。噪声会增加模型的大小以及模型构建和评分所需的时间和系统资源。

具有许多属性的数据集可以包含相关的属性组。这些属性实际上测量相同的底层特征。它们在构建数据中的存在可能会扭曲算法的逻辑并影响模型的准确性。

宽数据（许多属性）通常呈现数据挖掘算法的处理挑战。模型属性是算法使用的处理空间的维度。处理空间的维数越高，算法处理所涉及的计算成本就越高。

为了最小化噪声，相关性和高维度的影响，某种形式的降维有时是数据挖掘的理想预处理步骤。特征选择和提取是降维的两种方法。

  • 特征选择 ：选择最相关的属性

  • 特征提取 ：将属性组合成一组新的特征

8.2关于特征选择和属性重要性

找出最重要的预测因素是一些数据挖掘项目的目标。例如，模型可能试图找出构成高信用风险的客户的主要特征。

Oracle Data Mining支持属性重要性挖掘功能，该功能根据属性在预测目标时的重要性对属性进行排名。属性重要性实际上不执行特征选择，因为所有预测变量都保留在模型中。在真实特征选择中，从模型中移除排名低于给定重要性阈值的属性。

特征选择可用作预处理步骤，以提高预测建模的计算效率。当启用自动数据准备（ADP）时，Oracle Data Mining在决策树算法和朴素贝叶斯内实现功能选择以进行优化。可以将广义线性模型（GLM）配置为执行特征选择作为预处理步骤。

8.2.1属性重要性和评分

Oracle Data Mining不支持属性重要性的评分操作。属性重要性的结果是根据其预测影响排序的构建数据的属性。排名和重要性度量可用于选择分类模型的训练数据。

8.3关于特征提取

特征提取是属性约简过程。不像功能选择，选择并保留最重要的属性，功能提取实际上转换属性。变换后的属性或特征是原始属性的线性组合。

特征提取过程会产生更小，更丰富的属性集。功能的最大数量可由用户指定或由算法确定。默认情况下，算法会确定它。

基于提取的特征构建的模型可以具有更高的质量，因为更少和更有意义的属性描述数据。

特征提取将具有较高维度的数据集投影到较小数量的维度上。因此，它对于数据可视化是有用的，因为当复杂数据集减少到二维或三维时，可以有效地可视化。

特征提取的一些应用是潜在语义分析，数据压缩，数据分解和投影以及模式识别。特征提取还可用于提高监督学习的速度和有效性。

特征提取可用于提取文档集合的主题，其中文档由一组关键字及其频率表示。每个主题（特征）由关键字组合表示。然后可以根据发现的主题来表示集合中的文档。

8.3.1特征提取和评分

Oracle Data Mining支持特征提取的评分操作。作为无监督挖掘函数，特征提取不涉及目标。应用时，特征提取模型将输入转换为一组要素。

8.4属性重要性和特征提取算法

了解用于属性重要性和特征提取的算法。

Oracle Data Mining支持以下属性重要性算法：

  • 最小描述长度

  • CUR矩阵分解

Oracle Data Mining支持以下特征提取算法：

  • 显式语义分析 （ESA） 。

  • 非负矩阵分解 （NMF） 。

  • 奇异值分解 （SVD）和预测成分分析 （PCA） 。

注意：

从失败阶段查找和重新启动多租户升级的过程

要从失败阶段重新启动多租户升级，请首先确定哪个PDB创建了错误，然后在其相应的日志文件中搜索错误。

要识别导致多租户升级失败的PDB，请查看升级摘要报告，或查看catupgrd0.log;此日志包含文件末尾的升级摘要报告。
使用此过程检查每个日志文件以查找错误。
  1. 找到有错误的日志文件。

    例如：

    Linux和Unix

    行'catupgrd * .log时出现$ grep -i'错误

    视窗

    C：\> find / I“行”错误“catupgrd * .log
    grep或find命令显示发现错误的日志文件的文件名。
  2. 检查每个有错误的日志文件，并确定第一个错误发生的位置。使用您选择的文本编辑器查看每个日志文件。在第 ' 行 '中搜索第一次出现的短语错误 。当您找到该短语时，然后从错误（反向搜索）向后搜索PHASE_TIME___START 。

    例如：

    PHASE_TIME___START 15 15-01-16 08:49:41

    PHASE_TIME___START之后的PHASE_TIME___START是发生错误的阶段号。在此示例中，阶段编号为15。

    每个日志文件都可能出错。重复检查短语PHASE_TIME___START ，并为包含错误的每个日志文件识别包含错误的阶段号，并标识包含最低阶段号的日志文件。

    包含最低阶段编号的日志文件是重新启动阶段编号，即重新启动升级的阶段编号。

    例如：

    阶段15中发生catupgrd0.log错误：PHASE_TIME___START 15 15-01-16 08:49:41第19阶段发生catupgrd1.log错误：PHASE_TIME___START 19 15-01-16 08:50:01

    在此示例中，重新启动阶段编号为15。确保您确定在所有日志文件中看到的第一个错误，以便您可以从该阶段重新启动升级。

  3. 通过将目录切换到正在运行的并行升级实用程序重新启动从失败的阶段升级（ catctl.pl ，您可以使用命令行运行dbupgrade在Linux和Unix和dbupgrade.cmd在Windows上）。使用-p标志指示您要从阶段重新启动升级，并提供重新启动阶段号。在多租户数据库中，也可以使用-c ' PDBname '语法来使用-c标志，其中PDBname是发生故障的PDB的名称。

    例如：

    Linux或UNIX系统上的非CDB Oracle数据库：

    cd $ ORACLE_HOME / bin dbupgrade -p 15

    Windows系统上多租户Oracle数据库（CDB）中的PDB：

    cd $ ORACLE_HOME / bin dbupgrade -p 15 -c'PDB1'
    在这两个示例中，升级都从阶段15重新启动，使用-p标志进行标识。在多租户示例中，带有错误的PDB使用-c标志进行标识。
    在这些示例中，升级从阶段15开始并一直运行到升级结束。
  4. （可选）您还可以使用-P标志指定停止阶段来运行包含错误的阶段。使用停止阶段允许升级仅重新运行发生错误的阶段。您可以通过在出错的阶段运行错误来快速确定错误是否已修复，而无需运行整个升级。

    例如，使用Perl脚本Parallel Upgrade Utility命令选项：

    cd $ ORACLE_HOME / rdbms / admin $ ORACLE_HOME / perl / bin / perl catctl.pl -p 15 -P 15 -c'PDB1'
    在确认错误的阶段中确定错误后，您可以在该阶段之后恢复升级。

    例如，如果您已确认PDB1多租户数据库升级的第15阶段中的错误已得到修复，那么您可以在Linux和Unix系统上使用以下命令在阶段16继续升级：

    cd $ ORACLE_HOME / bin dbupgrade -p 16 -c'PDB1'

13.67 WHILE LOOP声明

WHILE LOOP语句在条件为TRUE时运行一个或多个语句。 WHILE LOOP语句在条件变为FALSE或NULL时结束，或者当循环内的语句将控制转移到循环外或引发异常时结束。

  • “ 退出声明 ”

  • “ GOTO声明 ”

为模式 - 私有语义网络添加了支持

现在可以在常规用户的模式中创建语义网络。此类网络称为模式专用语义网络，因为关联的数据库对象是在网络所有者的架构中创建的，并且网络所有者具有对这些对象的专有权限。（DBA用户也具有此类权限，网络所有者或DBA可以授予和撤消其他用户的权限。）

在以前的版本中，唯一的方案是MDSYS用户拥有的单个语义网络，并且可以访问整个数据库。该方案仍然受支持，但我们鼓励您使用模式专用语义网络。

有关更多信息，请参阅语义网络 。

以前称为RDF语义图的功能名称现在称为RDF知识图 。

语义网络和语义数据这两个术语在本指南中仍广泛使用。

为Oracle Database In-Memory添加了支持

可以轻松地将RDF数据加载到内存中，以利用Oracle Database In-Memory功能。现在可以使用SEM_APIS.ENABLE_INMEMORY过程将语义网络加载到内存中。此外，可以在虚拟模型级别使用内存中虚拟列，以将RDF术语的词值添加到MDSYS.RDF_LINK $表的内存中表示，从而减少评估SPARQL查询所需的联接数。

有关更多信息，请参阅Oracle数据库内存中的RDF支持 。

父主题： Oracle数据库发行版18.1中的更改

通过复合分区为语义网络添加支持

现在可以使用列表哈希复合分区来创建语义网络。通过该方案，语义网络最初由模型id进行列表分区，然后使用RDF谓词ID的散列对每个分区进行子分区。复合分区通过增加并行化和更好的查询优化器统计信息来提高SPARQL查询性能。

有关更多信息，请参阅语义网络 。

父主题： Oracle数据库发行版18.1中的更改

增强的CLOB支持批量加载操作

现在可以使用SEM_APIS.BULK_LOAD_FROM_STAGING_TABLE有效地加载包含具有长文字的RDF四边形（大小超过4000字节的RDF对象值）的临时表。

SEM_APIS.LOAD_INTO_STAGING_TABLE中还添加了新选项，以便在从外部表加载时更好地处理长文字。VC_ONLY选项仅将对象值不大于4000字节的RDF四元组加载到临时表中，而CLOB_ONLY选项仅加载对象值大于4000字节的RDF四元组。这些选项允许非常有效的两阶段批量加载，其中在一个批量加载操作中加载仅VARCHAR数据，而在第二个批量加载操作中加载仅CLOB数据。

父主题： Oracle数据库发行版18.1中的更改

对Turtle和Trig RDF格式的本机支持

现在可以将Turtle和Trig RDF格式直接加载到Oracle数据库中，而无需第三方工具。通过SEM_APIS.UPDATE_MODEL执行的SPARQL LOAD操作现在可以解析并插入以Turtle和Trig格式序列化的RDF数据，以及先前支持的N-Triple和N-Quad格式。

现在可以直接从Turtle或N-Triple格式指定的R2RML映射创建RDF视图。新的R2RML_STRING和R2RML_STRING_FMT参数已添加到SEM_APIS.CREATE_RDFVIEW_MODEL，以便R2RML映射字符串可用于创建RDF视图模型。
