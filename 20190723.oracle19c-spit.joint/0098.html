B OCI示范计划

列出示例程序，其中包含示例代码，用于演示OCI句柄的分配和使用。

Oracle提供了说明OCI调用使用的代码示例。这些程序仅用于演示目的，并不保证在所有操作系统上运行。

您必须按照“ Oracle数据库示例安装指南”中的说明安装演示程序。程序的位置，名称和可用性可能因不同的操作系统而异。在Linux或UNIX工作站上，程序安装在$ ORACLE_HOME/rdbms/demo目录中。对于Windows系统，请参阅OCI for Windows入门 。

在Linux或UNIX平台上进行OCI客户端应用程序开发所需的OCI头文件位于$ORACLE_HOME/rdbms/public目录中。demo_rdbms.mk文件位于$ORACLE_HOME/rdbms/demo目录中，并作为makefile的示例。在Windows系统上， make.bat是samples目录中的类似文件。makefile中有说明。

除非您显着修改demo_rdbms.mk文件，否则只要demo_rdbms.mk文件包含$ORACLE_HOME/rdbms/public目录，您就不会受到更改的影响。确保高度自定义的makefile在INCLUDE路径中具有$ORACLE_HOME/rdbms/public目录。

开发用于构建OCI应用程序或外部过程的新makefile应该包括通过将自己的宏添加到链接行来自定义makefile 。但是，Oracle要求您保留demo makefile中提供的宏，因为这样可以更轻松地维护您自己的makefile 。

当应用程序需要特定的头文件或SQL文件时，这些文件也包含在演示程序文件中。查看演示程序开头的注释中的信息，以获取有关运行程序的设置和提示。

34.1使用数据库管理方法

从Oracle Database 11g第1版开始， oracle.jdbc.添加了两个JDBC方法，即startup和shutdown oracle.jdbc.OracleConnection接口，使您可以启动和关闭Oracle数据库实例。这类似于从SQL * Plus启动或关闭数据库实例的方式。

要使用startup和shutdown方法，您必须：

  • 与服务器建立专用连接。您无法通过调度程序连接到共享服务器。

  • 以SYSDBA或SYSOPER连接。要以SYSDBA或SYSOPER与Oracle JDBC驱动程序连接，需要相应地设置INTERNAL_LOGON连接属性。

要使用JDBC Thin驱动程序以SYSDBA身份登录，必须将服务器配置为使用密码文件。例如，要将system/manager配置为使用JDBC Thin驱动程序作为SYSDBA进行连接，请执行以下操作：

  1. 从命令行键入：
    orapwd file = $ ORACLE_HOME / dbs / orapw entries = 5输入密码： 密码
  2. 以SYSDBA身份连接到数据库，并从SQL * Plus运行以下命令：
    GRANT SYSDBA TO系统; PASSWORD系统更改系统密码新密码： 密码重新输入新密码： 密码
  3. 编辑init.ora并添加以下行：
    REMOTE_LOGIN_PASSWORDFILE = EXCLUSIVE
  4. 重新启动数据库实例。

与JDBC Thin驱动程序相反，JDBC OCI驱动程序可以在本地连接为SYSDBA或SYSOPER ，而无需在服务器上指定密码文件。

34.2使用启动方法

要使用startup方法启动数据库实例，应用程序必须首先以PRELIM_AUTH模式中的SYSDBA或SYSOPER连接到数据库，这是数据库关闭时允许的唯一连接模式。您可以通过将连接属性PRELIM_AUTH设置为true来执行此操作。在PRELIM_AUTH模式下，您只能启动已关闭的数据库实例。您无法在此模式下运行任何SQL语句。

例

startup方法将使用服务器参数文件启动数据库。Oracle JDBC驱动程序不支持使用客户机参数文件启动数据库。

24数据库驻留连接池

数据库驻留连接池（DRCP）是服务器中的连接池，在许多客户端之间共享。您应该在连接池中使用DRCP，其中活动连接的数量远小于打开的连接数。由于可以共享来自DRCP池的连接的连接池的实例数量增加，因此使用DRCP带来的好处会增加。DRCP提高了数据库服务器的可扩展性，并解决了与中间层连接池相关的资源浪费问题。

24.1数据库驻留连接池概述

在中间层连接池中，每个连接缓存都维护与服务器的最小连接数。每个连接表示服务器上的已用资源。在任何给定时间都不使用所有这些打开的连接，这意味着存在不必要地占用服务器资源的未使用资源。在多中间层方案中，这些连接不与任何其他中间层共享，并且即使其中一些空闲也保留在缓存中。但是，大量此类中间层连接池会显着增加与数据库服务器的非活动连接数，并浪费大量数据库资源，因为所有连接都不会同时保持活动状态。

例如，在中间层连接池中，如果最小池大小为200，则连接池具有到服务器的200个连接，并且数据库服务器具有与这些连接关联的200个服务器进程。如果有30个中间层具有最小大小为200的连接池，则服务器运行6000（200 * 30）个相应的服务器进程。通常，平均只有5％的连接，反过来，服务器进程在任何给定时间都在使用。因此，在6,000个服务器进程中，在任何给定时间只有300个服务器进程处于活动状态。这导致服务器上超过5,700个未使用的服务器进程。这些未使用的进程是服务器上浪费的资源。

数据库驻留连接池实现在服务器端创建一个池，该池在多个客户端池之间共享。这显着降低了服务器上的内存消耗，因为服务器上的服务器进程数量减少，并且增加了数据库服务器的可伸缩性。

24.4 DRCP标记

DRCP使您可以请求服务器连接池将服务器进程与特定标记名称相关联。您可以将标记应用于给定连接，并在以后检索该标记连接。连接标记增强了会话池，因为您可以轻松地检索特定会话。

从Oracle Database 12 c第2版（12.2.0.1）开始，DRCP为多个标记提供支持。默认情况下，由于与现有DRCP应用程序的兼容性因素，此功能被禁用。设置oracle.jdbc.UseDRCPMultipletag连接属性为TRUE用于在DRCP应用程序中启用此功能。

使用DRCP标记时，请记住以下几点：

  • 标记的键和值不能为NULL或Empty。

  • 指定多个标记时，最左边的标记具有最高优先级，最右边的标记具有最低优先级。

  • 在检索标记连接时，如果未找到完全匹配（所有标记都不匹配），则会搜索部分匹配。

14.7创建功能声明

CREATE FUNCTION语句创建或替换独立函数或调用规范。

独立函数是存储在数据库中的函数（返回单个值的子程序）。

注意：

使用CREATE FUNCTION语句创建的独立函数与在PL / SQL块或包中声明和定义的函数不同。有关更多信息，请参见“ 函数声明和定义 ”和CREATE PACKAGE语句 。

调用规范声明了Java方法或C函数，以便可以从PL / SQL调用它。您还可以使用SQL CALL语句来调用此类方法或子程序。调用规范告诉数据库在调用时调用哪个Java方法或哪个共享库中的命名函数。它还告诉数据库为参数和返回值进行的类型转换。

注意：

要从SQL语句中调用，存储的函数必须遵守控制副作用的某些规则。请参阅“ 子程序副作用 ” 。

8.3.1关于Oracle Data Pump

Oracle Data Pump由两个命令行客户端和两个PL / SQL包组成。命令行客户端expdp和impdp为Data Pump导出和导入实用程序提供了易于使用的界面。您可以使用expdp和impdp导出和导入整个模式或数据库。

Data Pump导出实用程序将架构对象（包括构成挖掘模型的表和元数据）写入转储文件集。Data Pump导入实用程序从转储文件集中检索架构对象（包括模型表和元数据），并将其还原到目标数据库中。

expdp和impdp不能用于导出/导入单个挖掘模型。

10.1.3 Oracle数据库资源管理器未启用和资源计划错误

Oracle数据库服务质量（QoS）管理要求数据库资源管理器使用特定的资源计划。

10.1.5服务器池标记为无法管理

如果Oracle数据库QoS管理无法正确测量或预测该服务器池中部署的性能类的性能，则会将服务器池标记为无法管理。

在以下情况下，服务器池可能被视为无法管理：

  1. 服务器池中的服务器具有不同的物理CPU计数。

  2. 无法检索每个数据库实例的CPU计数。

  3. 服务器上所有数据库实例的已配置CPU计数总和大于其物理CPU计数。

  4. Singleton服务部署在最大大小为1的服务器池中。

  5. Oracle数据库QoS管理无法收集所有性能类的度量标准，或者收集的度量标准不包含有效数据。

  6. 服务器池具有未启用Oracle数据库QoS管理的数据库。

如果将服务器添加到服务器池，则Oracle会在新服务器上为服务器池中的策略管理数据库启动实例和服务。在新服务器上启动实例时，Oracle会检查现有实例的SPFILE以获取CPU_COUNT设置，并将此值用于新实例。默认情况下，如果SPFILE中没有CPU_COUNT设置， CPU_COUNT启动新服务器上的实例，并将CPU_COUNT设置为服务器的物理CPU数。这不是受支持的配置，将导致服务器池标记为无法管理。此外，如果修改CPU_COUNT参数但未将更改存储在SPFILE中，则CPU_COUNT参数可能会在新启动的实例上设置为错误的值，从而导致配置违规。

确保要由Oracle Database QoS Management管理的数据库和数据库实例符合“支持的数据库配置”中记录的要求。

10.1.6性能等级缺少度量标准

Oracle Database QoS Management通过查询集群中的托管数据库实例来捕获度量标准。

在以下情况下不显示度量标准：

  1. 数据库请求（正在完成的工作）与为特定性能类定义的分类器不匹配，并且显示“无需”。

  2. 首先评估性能类的更通用的分类器。因此，具有更具体分类器的Performance Class显示为具有“No demand”。例如，如果sales_pc Performance Class使用分类器service=Sales且sales_search Performance Class使用分类器（ s ervice=Sales and action=Search) ，那么，如果sales_pc Performance Class在Edit Policy Set Performance Class中首先列出订单屏幕，然后任何使用sales服务的工作请求都将放在sales_pc Performance Class中，包括执行Search操作的那些。sales_search性能类不生成任何度量标准，并且该性能等级显示“无需求”。

  3. 数据库实例被过度使用，度量标准查询达到超时限制，并显示“不完整数据”。

  4. 数据库不会为Performance Class生成相互一致的数据，并显示“Nonconforming data”。例如，对超过一秒的数据库请求进行分类的性能类将导致此响应。

  5. 通过查询收集的度量标准不会通过Oracle数据库QoS管理执行的完整性验证检查，并显示“不合格数据”。

Oracle数据库本身支持JavaScript对象表示法（JSON）数据，具有关系数据库功能，包括事务，索引，声明性查询和视图。

本文档介绍如何使用数据库语言和功能来处理存储在Oracle数据库中的JSON数据。特别是，它介绍了如何将SQL和PL / SQL与JSON数据一起使用。

注意：

Oracle还提供了一系列简单Oracle文档访问 （ SODA ）API，用于访问存储在数据库中的JSON数据。SODA专为无模式应用程序开发而设计，无需了解关系数据库功能或SQL和PL / SQL等语言。它允许您在Oracle数据库中创建和存储文档集合，检索它们并查询它们，而无需知道文档如何存储在数据库中。

SODA有几种实现方式：

  • SODA for REST - 使用能够进行HTTP调用的任何语言，表示性状态转移（REST）请求执行收集和文档操作。

  • SODA for Java - Java类和接口代表数据库，集合和文档。

  • 用于PL / SQL的SODA - PL / SQL对象类型表示集合和文档。

  • SODA for C - Oracle调用接口（OCI）句柄代表集合和文档。

有关SODA 的信息，请将Oracle视为文档存储 。

1.1 Oracle数据库中的JSON概述

JSON数据和XML数据可以类似的方式在Oracle数据库中使用。与关系数据不同，两者都可以存储，索引和查询， 而无需任何定义数据的模式 。Oracle数据库本身支持JSON和关系数据库功能，包括事务，索引，声明性查询和视图。

JSON数据通常存储在NoSQL数据库中，例如Oracle NoSQL Database和Oracle Berkeley DB。这些允许存储和检索不基于任何模式的数据，但它们不提供关系数据库的严格一致性模型。

为了弥补这个缺点，关系数据库有时与NoSQL数据库并行使用。使用存储在NoSQL数据库中的JSON数据的应用程序必须自己确保数据完整性。

Oracle数据库对JSON的本机支持避免了这种解决方法。它提供了与JSON一起使用的关系数据库功能的所有好处，包括事务，索引，声明性查询和视图。

结构化查询语言（SQL）查询是声明性的。使用Oracle数据库，您可以使用SQL将JSON数据与关系数据连接起来。您可以相关地投影JSON数据，使其可用于关系流程和工具。您还可以在数据库中查询存储在外部表中的Oracle数据库外部的JSON数据。

您可以像访问其他数据库数据一样访问存储在数据库中的JSON数据，包括使用Oracle调用接口（OCI），Microsoft .NET Framework和Java数据库连接（JDBC）。

在Oracle数据库中，使用常见的SQL数据类型VARCHAR2 ， CLOB和BLOB存储JSON数据（与XML数据不同，后者使用抽象SQL数据类型XMLType存储）。Oracle建议您始终使用is_json检查约束来确保列值是有效的JSON实例（请参阅示例4-1 ）。

根据定义，文本JSON数据使用Unicode编码（UTF-8或UTF-16）进行编码。您可以使用存储在非Unicode字符集中的文本数据，就像它是JSON数据一样，但在这种情况下，Oracle数据库会在处理数据时自动将字符集转换为UTF-8。

数据库表中的JSON列

Oracle数据库对可用于存储JSON文档的表没有任何限制。包含JSON文档的列可以与任何其他类型的数据库数据共存。表还可以包含多个包含JSON文档的列。

将Oracle数据库用作JSON文档存储时，包含JSON列的表通常也会包含一些非JSON管理列。这些通常跟踪有关JSON文档的元数据。

如果您使用JSON为主要关系应用程序添加灵活性，那么您的某些表可能还有一个JSON文档列，您可以使用该列来管理不直接映射到关系模型的应用程序数据。

  • SQL / JSON 生成函数json_object ， json_array ， json_objectagg和json_arrayagg 。他们收集SQL数据以生成JSON数据（作为SQL值）。

  • 点表示法 ，其作用类似于查询函数json_value和json_query的组合，类似于SQL对象访问表达式，即抽象数据类型（ADT）的属性点表示法。这是查询数据库中JSON数据的最简单方法。

关于Oracle ASM文件访问控制

Oracle ASM文件访问控制限制文件对作为SYSDBA连接的特定Oracle ASM客户端的访问。

Oracle ASM客户端通常是一个数据库，它被标识为拥有数据库实例主目录的用户。Oracle ASM文件访问控制使用此用户名来标识数据库。Oracle ASM文件访问控制根据操作系统数据库所有者的有效用户标识号限制访问。例如，在表3-2中 ，数据库被标识为oracle1和oracle2 。

Oracle ASM使用文件访问控制来确定为已在Oracle ASM实例上进行AS SYSDBA身份验证的数据库提供的其他权限。这些附加权限包括修改和删除某些文件，别名和用户组的功能。

您可以设置用户组以指定与Oracle ASM文件共享相同访问权限的数据库列表。用户组是数据库列表，任何验证AS SYSDBA的数据库都可以创建用户组。但是，只有组的创建者可以删除它或修改其成员列表。

每个Oracle ASM文件都有三类权限：所有者，组和其他。每个类别都没有权限，只读权限或读写权限。

文件所有者通常是文件的创建者，可以在任何所有者，组或其他类别中为文件分配权限。所有者还可以更改与文件关联的组。

创建文件时，组不会自动与该文件关联。必须使用asmcmd chgrp命令手动设置组，因为Oracle ASM文件访问控制不具有用户主组的概念。

管理Oracle ASM文件访问控制时，Oracle建议您以SYSDBA身份连接到磁盘组中文件的所有者或计划所有者的数据库实例。

要为磁盘组中的文件设置Oracle ASM文件访问控制，请执行以下步骤：

  1. 更改新的或现有的磁盘组以设置Oracle ASM文件访问控制磁盘组属性。

    对于新创建的磁盘组，应在创建磁盘组中的任何文件之前设置磁盘组属性。

  2. 对于在设置Oracle ASM文件访问控制磁盘组属性之前存在于磁盘组中的文件，必须显式设置这些现有文件的权限和所有权。

    在为文件设置所有权或权限之前，请确保用户存在。在设置所有权或权限之前，必须先关闭该文件。

  3. （可选）您可以创建用户组，这些用户组是对Oracle ASM文件具有相同访问权限的数据库用户组。

为Oracle ASM文件访问控制设置磁盘组属性

要管理磁盘组的Oracle ASM文件访问控制，必须设置ACCESS_CONTROL.ENABLED和ACCESS_CONTROL.UMASK磁盘组属性。您可以通过使用ALTER DISKGROUP SQL语句更改磁盘组来设置属性，也可以使用ASMCMD setattr命令设置属性。

在现有磁盘组上设置文件访问控制时，除非运行ALTER DISKGROUP SET PERMISSION SQL语句或ASMCMD chmod命令来限制权限，否则所有人都可以访问以前创建的文件。

必须将COMPATIBLE.ASM和COMPATIBLE.RDBMS磁盘组属性设置为11.2或更高才能启用Oracle ASM文件访问控制。

控制Oracle ASM文件访问控制的磁盘组属性如下：

  • ACCESS_CONTROL.ENABLED

    此属性确定是否为磁盘组启用Oracle ASM文件访问控制。

    值可以是true或false 。默认值为false 。

    如果该属性设置为true ，则访问Oracle ASM文件受访问控制。如果为false ，则任何用户都可以访问磁盘组中的每个文件。所有其他操作的行为与此属性无关。

  • ACCESS_CONTROL.UMASK

    此属性确定在为拥有该文件的用户，同一用户组中的用户以及不在该用户组中的其他用户创建Oracle ASM文件时屏蔽了哪些权限。此属性适用于磁盘组上的所有文件。

    值可以是三位数{ 0 |的组合2 | 6 } { 0 | 2 | 6 } { 0 | 2 | 6 }。默认值为066 。

    设置为0屏蔽掉任何内容。设置为2屏蔽写入权限。设置为6屏蔽读取和写入权限。

    在设置ACCESS_CONTROL.UMASK磁盘组属性之前，必须将ACCESS_CONTROL.ENABLED属性设置为true以启用Oracle ASM文件访问控制。

例4-11显示了如何使用SQL * Plus为磁盘组启用Oracle ASM文件访问控制。在此示例中， umask权限设置为026 ，它允许所有者进行读写访问（不使用0屏蔽任何内容），对组中的用户进行读取访问（使用2屏蔽写入权限），以及不访问其他人（掩码） 6 ）不在组中的所有访问权限。

示例4-11设置Oracle ASM文件访问控制

ALTER DISKGROUP data1 SET ATTRIBUTE'access_control.enabled'='true'; ALTER DISKGROUP data1 SET ATTRIBUTE'access_control.umask'='026';

使用SQL语句管理Oracle ASM文件访问控制

您可以使用ALTER DISKGROUP SQL语句来管理Oracle ASM磁盘组的文件访问控制。这些SQL语句可用于数据库和Oracle ASM实例。

使用SQL * Plus管理用户时，用户必须是现有操作系统用户，并且其用户名必须具有相应的操作系统用户ID。但是，只能验证与Oracle ASM实例位于同一群集中的用户。

注意：

修改文件的所有权时，即使文件的所有权发生更改，也无法在文件仍处于打开状态时删除文件的先前所有者。

支持磁盘组访问控制的SQL语句是：

  • ALTER DISKGROUP ADD USERGROUP WITH MEMBER

    将Oracle ASM用户组添加到磁盘组。用户组名称限制为最多30个字符。MEMBER子句中标识的数据库的操作系统用户必须位于磁盘组中，如V$ASM_USER所示，否则该命令将返回错误。任何经过身份验证为SYSASM或SYSDBA用户都可以创建新的用户组。例如：

    SQL> SELECT group_number，os_name FROM V $ ASM_USER; GROUP_NUMBER OS_NAME ------------ ------------------------------------ ---------------------------- 1 oracle1 1 oracle2 ...SQL> ALTER DISKGROUP数据添加USERGROUP'test_grp1'WITH MEMBER'oracle1'，'oracle2';

  • ALTER DISKGROUP DROP USERGROUP

    从磁盘组中删除Oracle ASM用户组。删除组可能会留下一些没有有效组的文件。要使这些文件具有有效组，您必须手动将与这些文件关联的组更新为有效组。

    SQL> ALTER DISKGROUP数据DROP USERGROUP'test_grp1';

  • ALTER DISKGROUP MODIFY USERGROUP ADD MEMBER

    将用户添加到指定的用户组。用户必须位于磁盘组中，如V$ASM_USER所示，否则命令将返回错误。只有组的创建者或Oracle ASM管理员才能修改组成员身份。 

    SQL> ALTER DISKGROUP数据MODIFY USERGROUP'test_grp2'ADD MEMBER'oracle2';

  • ALTER DISKGROUP MODIFY USERGROUP DROP MEMBER

    从指定的用户组中删除用户。如果成员不在用户组中，则返回错误。只有组的创建者或Oracle ASM管理员才能修改组成员身份。

    SQL> ALTER DISKGROUP数据MODIFY USERGROUP'test_grp2'DROP MEMBER'oracle2';

  • ALTER DISKGROUP ADD USER

    将操作系统（OS）用户添加到Oracle ASM磁盘组，以便这些用户可以拥有磁盘组的访问权限。如果磁盘组中存在用户，如V$ASM_USER所示，则该命令会记录错误并继续添加其他用户（如果有）。

    当数据库实例访问该磁盘组并创建文件时，正在运行的数据库实例的操作系统用户将自动添加到磁盘组。但是，要使数据库实例读取磁盘组中的文件而不创建任何文件，则必须使用ADD USER子句将与该数据库关联的操作系统用户添加到磁盘组。此外，在设置Oracle ASM文件访问控制磁盘组属性之后和创建新文件之前，您可以使用此子句立即将与数据库关联的操作系统用户添加到现有磁盘组。 

    SQL> ALTER DISKGROUP数据ADD USER'oracle1';

  • ALTER DISKGROUP DROP USER

    从Oracle ASM磁盘组中删除操作系统用户。如果用户不在磁盘组中，则此命令会记录错误并继续删除其他用户（如果有）。

    如果用户拥有同一Oracle ASM磁盘组上的任何文件，则此命令将失败并显示错误，除非指定了CASCADE关键字。如果是后一种情况，则删除用户以及用户拥有的所有文件。

    如果用户拥有的任何文件当前处于打开状态，则DROP USER命令将失败，并且不会删除任何文件。 

    SQL> ALTER DISKGROUP数据DROP USER'oracle1';

  • ALTER DISKGROUP REPLACE USER

    在Oracle ASM磁盘组中将一个操作系统用户替换为另一个操作系统用户。替换用户不能是当前位于磁盘组用户列表中的用户。

    如果命令成功，则当前用户以前拥有的所有文件现在都由替换用户拥有。将自动从磁盘组的用户列表中删除当前用户。可以将当前用户添加回磁盘组，但此用户不拥有任何文件。

    更改打开文件的所有者时，在重新启动集群中的所有实例之前，不能使用所有者的文件删除新所有者。

    只有Oracle ASM管理员才能运行此命令。

    SQL> ALTER DISKGROUP data REPLACE USER'oracle1'WITH'oracle2';

  • ALTER DISKGROUP SET PERMISSION

    修改Oracle ASM文件的权限。设置read only允许已经一个文件read write权限撤销的write权限。只有文件所有者或Oracle ASM管理员才能更改文件的权限。 

    SQL> ALTER DISKGROUP数据SET PERMISSION OWNER =读写，GROUP =只读，OTHER =无FOR FILE'+ data / controlfile.f';

  • ALTER DISKGROUP SET OWNERSHIP

    将文件的所有者或组分别更改为指定的用户或用户组名称。如果指定的用户或用户组名称不存在，则此命令将失败并显示错误。只有文件所有者或Oracle ASM管理员才能运行此命令，只有Oracle ASM管理员才能更改所有者。此外，用户组名称必须存在，并且文件的所有者必须是该组的成员。

    SQL> ALTER DISKGROUP数据SET OWNERSHIP OWNER ='oracle1'，GROUP ='test_grp1'FOR FILE'+ data / controlfile.f';

注册是Oracle将代理信息存储在数据字典中的操作。

代理商无需注册。如果未注册代理，Oracle会将有关代理的信息存储在内存中而不是数据字典中。当涉及代理的会话终止时，此信息将不再可用。

自 注册是一种操作，数据库管理员在其中设置初始化参数，使代理自动将信息上载到数据字典中。当HS_AUTOREGISTER初始化参数设置为TRUE （默认值）时，会发生自注册。

使用代理自注册来避免配置不匹配

异构服务（HS）配置数据存储在Oracle数据库服务器的数据字典中。

由于代理可以是远程的，因此可以单独管理，因此有几种情况可能导致服务器和代理之间的配置不匹配。例如：

  • 可以在单独的计算机上新安装代理，以便Oracle数据库服务器不具有代表代理的异构服务配置数据的异构服务数据字典内容。

  • 可以新安装Oracle数据库服务器，并且缺少现有代理和非Oracle数据存储所需的异构服务配置数据。

  • 可以将非Oracle实例从旧版本升级到较新版本，从而需要修改异构服务配置数据。

  • 远程站点上的异构服务代理可以升级到新版本或修补，需要修改异构服务配置数据。

  • 非Oracle站点上的数据库管理员（DBA）可以以影响异构服务配置数据的方式更改代理设置（可能用于调整或测试目的）。

代理自注册允许在所有这些场景中成功运行异构服务。具体而言，代理自注册可增强任何Oracle数据库与任何异构服务代理（如果每个版本均为8.0.3或更高版本）之间的互操作性。此功能的基本机制是能够将代理中的异构服务配置数据上载到服务器。

自注册可自动更新驻留在Oracle数据库数据字典中的异构服务配置数据。此更新确保在初次使用以前未注册的代理时，代理自注册上载只需执行一次。实例信息上载在每个连接上，而不是存储在服务器数据字典中。

FDS_CLASS和FDS_CLASS_VERSION

Oracle数据库或第三方供应商为每个单独的异构服务代理和版本定义FDS_CLASS和FDS_CLASS_VERSION 。

异构服务将这些名称连接起来形成FDS_CLASS_NAME ， FDS_CLASS_NAME用作访问服务器数据字典中的类信息的主键。

FDS_CLASS应指定要访问的非Oracle数据存储的类型， FDS_CLASS_VERSION应指定非Oracle数据存储和连接到它的代理的版本号。当代理的任何组件发生更改时， FDS_CLASS_VERSION也必须更改为唯一标识新版本。初始化每个连接时会上载此信息。

先决条件

Oracle数据库管理员指南中描述的创建可传输表空间集的限制适用于从备份传输表空间，但要求使表空间为只读。

该 SYSAUX表空间不能是恢复集的一部分，恢复集是要传输的表空间集。RMAN强制在辅助集中包含SYSAUX表空间，辅助集包含表空间传输所需的数据文件和其他文件。

TRANSPORT TABLESPACE不会转换 endian格式。如果目标平台具有不同的endian格式，则在运行TRANSPORT TABLESPACE使用CONVERT命令转换可传输集数据文件的endian格式。

如果删除表空间，则以后不能使用TRANSPORT TABLESPACE将此表空间包含在可传输表空间集中，即使TRANSPORT TABLESPACE的SCN早于删除表的SCN也是如此。如果重命名表空间，则无法使用TRANSPORT TABLESPACE创建可重定位表空间之前的时间点的可传输表空间集。

备份和备份元数据

您必须备份所有需要的表空间（包括辅助集中的表空间）和恢复到目标时间点所需的归档重做日志文件。

如果不使用恢复目录，并且数据库已重新使用包含有关所需备份的元数据的控制文件记录，则该命令将失败，因为RMAN无法找到备份。您可以使用CATALOG将备份添加到RMAN存储库，但如果数据库正在覆盖控制文件记录，则可能会丢失其他备份的记录。

数据泵导出和导入

因为RMAN使用了数据泵导出和导入实用程序，如果要传输的表空间使用，则不能使用TRANSPORT TABLESPACE XMLType 。在这种情况下，您必须使用Oracle数据库管理员指南中的过程 。

如果表空间目标中存在导出转储文件名称下的文件，则TRANSPORT TABLESPACE在调用Data Pump Export时失败。如果要重复以前的TRANSPORT TABLESPACE作业，请确保删除以前的输出文件，包括导出转储文件。

表空间和列加密

以下数据库加密功能都使用 Oracle软件密钥库：透明数据加密（TDE）列加密，其功能在列级别，以及TDE表空间加密。请注意对加密或包含加密列的表空间的以下限制：

  • 如果要传输加密的表空间，则必须手动将密钥库复制到目标数据库。

  • 如果目标数据库具有现有密钥库，则无法将密钥库从源数据库复制到目标数据库。因此，您无法将加密数据传输到已具有密钥库的数据库。如果使用TDE列加密对列进行加密，则可以将它们导出到受密码保护的导出文件中，并将数据导入目标数据库。

8.1共享池支持概述

在UCP中，池实例与数据源具有一对一映射。每个数据源都创建自己的连接池实例，并且该实例不可由其他数据源访问或共享，即使它们在内部创建和缓存到同一数据库和服务的连接也是如此。在此体系结构中，会创建大量隔离的连接池，这会导致可伸缩性问题，因为数据库只能扩展到一定数量的连接。

从Oracle Database 12 c第2版（12.2.0.1）开始，UCP支持连接到同一数据库的多个数据源，以共享同一个连接池。此公共连接池称为共享池。共享池优化系统资源，以便在Oracle Database Multitenant环境中实现多租户Java应用程序的可扩展部署。当每个数据源上的负载不均匀时，此功能可提供更大的灵活性。当创建每个数据源的单个池时，则无法将空闲资源从空闲连接池移动到加载的连接池。但是，当使用共享池时，可以通过共享和重新利用数据源之间的连接以有效的方式利用连接。因此，此功能可减少数据库连接总数，并提高数据库服务器的资源使用率，可诊断性，可管理性和扩展性。

以下是可以实现此功能的两种方案：

  • 使用共享池的单个多租户数据源

  • 每个租户使用共享池的一个数据源

8.2支持共享池的先决条件

以下是多租户数据源使用共享池的先决条件：

  • 您必须通过XML配置文件提供共享池的初始配置。您可以通过系统属性oracle.ucp.jdbc.xmlConfigFile为UCP指定初始XML配置文件。应将初始XML配置文件的位置指定为URI。例如， file:/user_directory/ucp.xml 。

    configuration.xsd模式文件包含在ucp.jar文件中以供参考。创建UCP XML配置文件时请参阅此文件。

  • 在重新配置共享池期间，应通过重新配置API提供更新的池属性。

  • 始终将应用程序服务用于共享池所用的服务，以及各个租户数据源特定服务。使用管理服务或默认PDB服务时，不会重新调整或重用连接。

  • 通过共享池访问的各种服务必须是同质的，也就是说，它们应具有与应用程序连续性（AC），DRCP（数据库驻留连接池）等类似的属性。

  • 必须使用单个用户配置共享池，并且此用户应该是CDB上配置的普通用户。普通用户应具有以下权限 - CREATE SESSION ， ALTER SESSION和SET CONTAINER 。普通用户还应具有DBMS_SERVICE_PRVT包的执行权限。

    注意：

    • 如果普通用户需要每个租户具有特定角色或启用密码的角色，则应在相应的租户数据源属性中指定这些角色。

    • SET CONTAINER语句的优点是，如果存在与不同PDB的现有连接，则池不必创建到PDB的新连接。池可以使用现有连接，并可以通过SET CONTAINER语句连接到所需的PDB。

  • 仅当池中的连接总数达到连接重新调整阈值（如果在池上配置）和最小池大小时，才会在共享池中的各个租户连接之间重新进行连接重新调整。

  • 为XML配置文件中的共享池指定的URL必须具有LONG格式，并明确指定服务名称。不支持短格式或简易连接URL。

新功能

  • 数据库内存等待填充

    DBMS_INMEMORY_ADMIN.POPULATE_WAIT函数启动优先级大于或等于指定优先级的所有INMEMORY对象的填充，并返回填充的状态值。用户指定的时间间隔指定函数在将值返回给调用者之前等待的最长时间。

    • 支持ORACLE_HIVE和ORACLE_BIGDATA驱动程序。

    • 支持并行查询。

    请参阅“ 内存表 ” 。

  • Oracle Database Resource Manager自动为Database In-Memory启用

    当INMEMORY_SIZE大于0 ，将自动启用资源管理器。

    请参阅“ IM动态扫描的界面 ” 。

  • Oracle Data Guard多实例重做应用支持IM列存储

    将初始化参数ENABLE_IMC_WITH_MIRA设置为TRUE可在Active Data Guard备用数据库上同时启用IM列存储和Data Guard多实例重做应用。默认情况下， ENABLE_IMC_WITH_MIRA为FALSE 。

    请参阅Oracle数据库参考以了解有关ENABLE_IMC_WITH_MIRA更多信息。

新功能

以下主要功能是此版本中的新功能。

  • 自动内存

    此功能使用段和列使用情况统计信息自动管理IM列存储的内容。如果填充作业因IM列存储已满而失败，则自动内存中驱逐非活动段以为活动段腾出空间。

    请参阅“ 配置自动内存中 ” 。

  • 内存中动态扫描

    IM动态扫描通过使用轻量级进程线程自动并透明地并行化表扫描。当Oracle资源管理器感知到CPU资源空闲并且可以利用它们来加速查询时，它们会分配这些线程。

    请参阅“ 内存中动态扫描 ” 。

  • IM表达式窗口捕获

    您可以定义任意长度的表达式捕获窗口 ，以确保仅考虑在此窗口中出现的表达式进行内存中实现。当您知道代表整个工作负载的小间隔时，此机制特别有用。例如，在交易窗口期间，经纪公司可以收集表达式集，并在IM列存储中实现它们，以加速整个工作负载的未来查询处理。

    请参阅“ 表达式捕获间隔 ” 。

  • 外部表的内存支持

    您可以将外部表填充到IM列存储中。此功能对于组合内部和外部数据的分析查询非常有用。

    请参阅“ 内存外部表 ”和“ 使用DBMS_INMEMORY.POPULATE填充内存外部表：示例 ” 。

  • 内存中优化算法

    对于使用QUERY LOW压缩的表格， NUMBER列使用优化格式进行编码，以便在硬件中进行本机计算。 使用此格式的聚合和算术运算的SIMD矢量处理可以实现显着的性能提升。当INMEMORY_OPTIMIZED_ARITHMETIC设置为ENABLE时，该功能ENABLE 。

    请参阅“ 优化内存中算术 ” 。

  • 大型对象（LOB）的增强性能

    在以前的版本中，尽管在IM列存储中填充了LOB和LOB指针，但数据库通过使用缓冲区缓存来满足查询。在此版本中，将范围谓词应用于标量列或SQL运算符应用于LOB列的内存中查询受益于SIMD向量处理。

    IM列存储为IMCU内的LOB（小于4 KB的LOB）提供连续存储。对于外部LOB，IM列存储仅存储LOB定位符，即40个字节。前面的规则有一个例外。IMEU可以为定义为LOB数据类型的JSON列分配最多32 KB的连续存储。IMEU以OSON（二进制JSON）格式存储这些列。

    请参阅“ CPU架构：SIMD矢量处理 ” 。

  • 一列上的内存中连接组

    您可以使用以下语法为单个列上的自CREATE INMEMORY JOIN GROUP jg_name ( table_name ( column_name ))创建连接组： CREATE INMEMORY JOIN GROUP jg_name ( table_name ( column_name )) 。

    请参阅“ 使用加入组优化联接 ” 。

  • 加入团体

    连接组是用户创建的对象，它列出了可以有意义地连接的两列。在某些查询中，连接组使数据库能够消除解压缩和散列列值的性能开销。加入组需要IM列存储。

    请参阅“ 使用加入组优化联接 ” 。

安全要求概述

之前实施安全方案，您必须了解环境中现有的安全要求和期望。由于您要启用对不同系统上不同数据库的应用程序访问，因此必须合并多个安全文化。在开发安全方案时，最严格的安全要求占上风。当您将不同的系统连接到一个操作整体时，具有最严格安全要求的系统通常会规定其他系统能够做什么和不能做什么。

网关安全包括两个主要问题：

  • 允许访问特定网关实例和OLTP的用户和应用程序

  • 用户和应用程序能够执行的OLTP事务

您可以在网关体系结构中的多个点控制访问。主要选项将在以下部分中讨论。每个OLTP都提供对远程事务程序（RTP）访问的控制，具有基于用户ID的本机授权机制。 OLTP的产品文档中描述了这些功能。以下主题中的信息包括网关设施如何确定对特定OLTP连接有效的用户标识。

当网关涉及RPC请求时，安全机制对网关遇到的每个系统组件都有效。遇到的第一个系统组件是应用程序工具或3GL程序。遇到的最后一个系统组件是OLTP。

以下每个部分都标识了该组件中可用的组件和安全性处理类型。每个部分都提供了主要功能和参数的摘要。有关Oracle和非Oracle产品的非网关组件的详细信息，请参阅产品特定的文档。

验证应用程序登录

应用程序必须连接到使用该网关之前的Oracle数据库。您使用的登录身份验证类型决定了生成的Oracle用户ID，并且可能会影响网关操作。

有两种基本类型的身份验证：

  • Oracle身份验证。

    使用Oracle身份验证，每个Oracle用户标识都有一个Oracle已知的关联密码。当应用程序连接到服务器时，它会提供用户标识和密码。Oracle确认存在用户标识，并且密码与存储在数据库中的密码相匹配。

  • 操作系统认证

    通过操作系统身份验证，服务器的底层操作系统负责身份验证。使用操作系统身份验证访问使用IDENTIFIED EXTERNALLY属性（而不是密码）创建的Oracle用户标识。要登录此类用户ID，应用程序会为用户ID提供斜杠（/），但不提供密码。

    要执行操作系统身份验证，服务器会确定请求者的操作系统用户ID，可选地为其添加固定前缀，并将结果用作Oracle用户ID。服务器确认用户ID存在并在外部标识，但未进行密码检查。基本假设是用户在登录操作系统时进行了身份验证。

    操作系统身份验证并非在所有平台上都可用，并且在某些平台上不可用 Oracle Net（客户端 - 服务器）和多线程服务器配置。请参阅Windows Oracle数据库文档和“ Oracle数据库网络服务管理员指南”，以确定配置中此功能的可用性。

有关验证应用程序登录的详细信息，请参阅“ Oracle数据库管理员指南” 。

定义和控制数据库链接

这些主题讨论了使用TCP / IP或SNA通信协议的网关用户的数据库链接。

  • 链接可访问性

  • 链接和CONNECT子句

链接可访问性

数据库链接的第一个控制点是给定用户是否可以访问它。一个公共数据库链接可以由任何用户ID使用。私有数据库链接只能由创建它的用户使用。数据库链接可用性取决于其打开网关会话的能力。该 Oracle数据库不区分使用类型（例如只读和更新或写入）或可以访问哪些远程对象。这些区别是访问的OLTP的责任。

链接和CONNECT子句

该 CONNECT子句是另一个与安全相关的属性数据库链接。您可以使用CONNECT子句指定显式用户标识和密码，该标识可以与用户的Oracle用户标识和密码不同。首次打开数据库链接连接时，此CONNECT用户标识和密码组合将发送到网关。根据特定于网关的选项，网关可能会将该用户标识和密码发送到要验证的OLTP。

如果使用Oracle身份验证创建了没有CONNECT子句的数据库链接，则在打开连接时CONNECT用户的Oracle用户ID和密码发送到网关。如果用户登录到 Oracle数据库具有操作系统身份验证，然后网关不会从Oracle数据库接收用户ID或密码。经过系统身份验证的Oracle用户不可能使用没有CONNECT子句定义的网关数据库链接。但是，如果您的OLTP提供基于网关的u ser ID映射工具用户连接的LU名称，如果同一网关实例上的所有用户都可以使用相同的OLTP用户ID，则可以进行此类连接。

有关数据库链接的更多信息，请参阅“ Oracle数据库管理员指南” 。

使用SNA安全验证

本节中的信息仅适用于采用SNA通信协议的网关用户的安全需求。当网关收到启动远程事务程序的RPC请求时，网关尝试启动与OLTP的APPC对话。在对话开始之前，会话必须在Windows的逻辑单元之间启动（ LU）和OLTP LU。

APPC SNA Server（Microsoft Host Integration Server或IBM Communication Server v6.1.1或更高版本）提供对Windows平台的支持。

SNA及其各种访问方法实现，包括VTAM和 SNA Server，在会话启动时提供安全验证，允许每个LU验证其合作伙伴。在网关和OLTP应用程序开始对话并处理对话级安全数据之前，此验证完全由网络软件执行。如果使用会话级安全性，则必须在其中建立正确的密码信息 Microsoft Windows的SNA配置文件以及要访问的OLTP中的类似参数结构。有关此主题的详细信息，请参阅相应的通信软件产品文档。

指定SNA会话安全性

该网关初始化文件的PGA_SECURITY_TYPE参数允许您指定确定安全行为的三个选项之一 LU6.2与OLTP分配的会话。这些选项是SNA LU6.2体系结构的一部分，但它们的精确行为可能因特定的OLTP系统而异。

SNA安全选项安全=无

如果指定PGA_SECURITY_TYPE=NONE ，则网关不会处理客户端用户标识和密码。会话分配SNA选项SECURITY=NONE 。

SNA安全选项SECURITY = PROGRAM

如果你指定 PGA_SECURITY_TYPE=PROGRAM ，然后网关使用SNA选项SECURITY=PROGRAM分配对话，并将以下信息发送给 OLTP：

  • 如果使用提示用户ID和密码覆盖，然后发送指定的用户ID和密码，而不管数据库链接规范如何。

  • 如果数据库链接有明确的 CONNECT信息，然后发送指定的用户ID和密码。

  • 如果数据库链接没有CONNECT子句，并且应用程序使用显式用户标识和密码登录到Oracle，则会发送Oracle用户标识和密码。

  • 如果应用程序使用操作系统身份验证登录到Oracle，并且数据库链接缺少显式CONNECT信息，则不会发送任何用户ID和密码。如果未发送用户标识和密码，并且未将OLTP配置为分配默认用户标识，则连接将失败。

通常，SNA选项SECURITY=PROGRAM告诉OLTP使用可用的任何身份验证机制来验证用户ID /密码组合。例如，如果用于z / OS的CICS Transaction Server是OLTP，然后可以使用RACF。但是，情况并非总是如此，因为每个OLTP都可以配置为以其他方式处理入站用户ID。

1.1关于Sharding

Sharding是一种数据层架构，其中数据在独立数据库中水平分区。

每个数据库都托管在具有自己的本地资源（CPU，内存，闪存或磁盘）的专用服务器上。 在这种配置中的每个数据库称为碎片 。 所有分片一起组成一个逻辑数据库，称为分片数据库（SDB） 。

水平分区涉及跨分片拆分数据库表，以便每个分片包含具有相同列但行的不同子集的表。以这种方式分割的表也称为分片表 。

分片基于无共享硬件基础架构，它消除了单点故障，因为分片不共享物理资源，如CPU，内存或存储设备。碎片在软件方面也松散耦合;他们不运行集群件。

碎片通常托管在专用服务器上。这些服务器可以是商用硬件或工程系统。分片可以在单实例或Oracle RAC数据库上运行。它们可以放置在本地，云端，或混合本地和云配置中。

从数据库管理员的角度来看，SDB由多个可以集体或单独管理的数据库组成。但是，从应用程序的角度来看，SDB看起来像一个单独的数据库：这些分片中的分片数量和数据分布对数据库应用程序是完全透明的。

分片适用于适用于分片数据库体系结构的自定义OLTP应用程序。使用分片的应用程序必须具有明确定义的数据模型和数据分布策略（一致的散列，范围，列表或复合），主要使用分片键访问数据。分片键的示例包括customer_id ， account_no或country_id 。

1.2分片的好处

Sharding为最苛刻的OLTP应用程序提供线性可扩展性和完整的故障隔离。

分片的主要好处包括：

  • 线性可扩展性 。分片消除了性能瓶颈，并且可以通过添加分片来线性扩展性能和容量。

  • 故障遏制 。Sharding是一种无共享硬件基础架构，可消除单点故障，例如共享磁盘，SAN和集群件，并提供强大的故障隔离 - 一个分片的故障或速度降低不会影响其他分片的性能或可用性。

  • 数据的地理分布 。当数据必须位于特定的管辖区域时，分片可以将特定数据存储在靠近其消费者的位置并满足法规要求。

  • 滚动升级 。一次在一个分片上应用配置更改不会影响其他分片，并允许管理员首先测试一小部分数据的更改。

  • 云部署的简单性 。分片非常适合在云中部署。可以根据需要调整碎片的大小以适应可用的任何云基础设施并且仍然实现所需的服务级别。Oracle Sharding支持内部部署，云和混合部署模型。

与实现分片的NoSQL数据存储不同，Oracle Sharding在不牺牲企业RDBMS功能的情况下提供分片的好处。例如，Oracle Sharding支持：

  • 关系模式

  • 数据库分区

  • ACID属性和读取一致性

  • SQL和其他编程接口

  • 复杂的数据类型

  • 在线架构更改

  • 多核可扩展性

  • 高级安全性

  • 压缩

  • 高可用性功能

  • 企业级备份和恢复

1.3 Oracle Sharding架构的组件

Oracle Sharding是适用于OLTP应用程序的可扩展性和可用性功能。它支持在不共享硬件或软件的Oracle数据库池中分发和复制数据。

应用程序将数据库池视为单个逻辑数据库。通过向池中添加数据库（分片），应用程序可以在任何平台上弹性地将数据，事务和用户扩展到任何级别。Oracle数据库支持最多可扩展1000个分片。

下图说明了Oracle Sharding的主要体系结构组件：

  • 分片数据库（SDB） - 单个逻辑Oracle数据库，在不共享硬件或软件的物理Oracle数据库（分片）池中水平分区

  • 碎片 - 托管分片数据库子集的独立物理Oracle数据库

  • 全局服务 - 提供对SDB中数据的访问的数据库服务

  • 分片目录 - 支持自动分片部署，分片数据库集中管理和多分片查询的Oracle数据库

  • Shard director - 网络侦听器，支持基于分片键的高性能连接路由

  • 连接池 - 在运行时，通过跨池连接路由数据库请求来充当分片导向器

  • 管理界面 - GDSCTL（命令行实用程序）和Oracle企业管理器（GUI）

分片数据库和碎片

碎片是独立的Oracle数据库，它们托管在具有自己的本地资源的数据库服务器上：CPU，内存和磁盘。分片之间不需要共享存储。

分片数据库是分片的集合。碎片可以全部放置在一个区域中，也可以放置在不同的区域。Oracle Sharding上下文中的区域表示数据中心或多个数据中心，这些数据中心位于网络附近。

使用Dataguard等Oracle复制技术复制碎片以实现高可用性（HA）和灾难恢复（DR）。对于HA，备用分片可以放置在放置主分片的同一区域中。对于DR，备用分片位于另一个区域。

全球服务

全局服务是对传统数据库服务概念的扩展。全局服务支持传统数据库服务的所有属性。对于分片数据库，为全局服务设置了其他属性 - 例如，数据库角色，复制滞后容差，客户端和分片之间的区域关联性等。对于读写事务工作负载，将创建单个全局服务以从SDB中的任何主分片访问数据。对于使用Active Data Guard的高可用分片，可以创建单独的只读全局服务。

碎片目录

分片目录是一个专用的Oracle数据库，它是SDB配置数据的持久存储，在分片数据库的集中管理中起着关键作用。所有配置更改（例如添加和删除分片和全局服务）都在分片目录上启动。SDB中的所有DDL都通过连接到分片目录来执行。

分片目录还包含SDB中所有重复表的主副本。分片目录使用物化视图自动将更改复制到所有分片中的重复表。分片目录数据库还充当查询协调器，用于处理多分片查询和未指定分片键的查询。

使用Oracle Data Guard进行分片目录高可用性是建议的最佳实践。分片目录的可用性对SDB的可用性没有影响。分片目录的中断仅影响在完成自动故障转移到备用分片目录所需的短暂时间内执行维护操作或多分片查询的能力。OLTP事务继续由SDB路由和执行，不受目录中断的影响。

碎片总监

Oracle Database 12 c引入了全局服务管理器，以根据数据库角色，负载，复制滞后和位置来路由连接。为了支持Oracle Sharding，全局服务管理器支持基于数据位置的连接路由。在Oracle Sharding的背景下，全球服务经理被称为分片导演。

分片导向器是全局服务管理器的特定实现，它充当连接到SDB的客户端的区域侦听器。导演维护SDB的当前拓扑图。根据连接请求期间传递的分片密钥，导向器将连接路由到适当的分片。

对于典型的分片数据库，每个区域的专用低端商用服务器上都安装了一组分片导向器。要实现高可用性，请部署多个分片导向器。您可以在给定区域中部署最多5个分片导向器。

以下是分片导演的主要功能：

  • 维护有关SDB配置和分片可用性的运行时数据

  • 测量其自身和其他区域之间的网络延迟

  • 充当客户连接到SDB的区域监听器

  • 管理全球服务

  • 执行连接负载平衡

连接池

Oracle数据库支持数据访问驱动程序（如OCI，JDBC和ODP.NET）中的连接池。这些驱动程序可以识别作为连接请求的一部分指定的分片键。同样，JDBC客户端的Oracle通用连接池（UCP）可以识别连接URL中指定的分片密钥。 Oracle UCP还使非Oracle应用程序客户端（如Apache Tomcat和WebSphere）能够与Oracle Sharding一起使用。

Oracle客户端使用UCP缓存路由信息根据应用程序提供的分片键将数据库请求直接路由到相应的分片。这种数据相关的数据库请求路由消除了额外的网络跳跃，减少了大容量OLTP应用程序的事务延迟。

在与分片的初始连接期间缓存路由信息，该分片是使用分片导向器建立的。在缓存范围内对分片密钥的后续数据库请求将直接路由到分片，绕过分片导向器。

与UCP一样，分片导向器可以处理连接字符串中指定的分片密钥并缓存路由信息。但是，UCP使用已建立的连接路由数据库请求，而分片导向器将连接请求路由到分片。当分片变得不可用或分片拓扑发生更改时，路由缓存会自动刷新。对于高性能，数据相关的路由，Oracle建议在访问SDB中的数据时使用连接池。

SDB的管理接口

您可以使用两个界面部署，管理和监视Oracle Sharded数据库：Oracle Enterprise Manager云控制和GDSCTL。

Cloud Control通过图形用户界面实现分片数据库的生命周期管理。您可以管理和监控SDB的可用性和性能，还可以执行添加和部署分片，服务，分片导向器和其他分片组件等任务。

GDSCTL是一个命令行界面，它提供了一种简单的声明方式，用于指定SDB的配置并自动部署。创建SDB只需要几个GDSCTL命令，例如：

  • 创建SHARDCATALOG

  • ADD GSM和START GSM （创建和启动分片导向器）

  • CREATE SHARD （每个碎片）

  • 部署

GDSCTL DEPLOY命令自动创建分片及其各自的侦听器。此外，此命令会自动部署用于管理员指定的分片级高可用性的复制配置。

1.4应用适用于分片

Oracle Sharding适用于适用于分片数据库的OLTP应用程序。

从未打算分片的现有应用程序需要进行一定程度的重新设计才能实现分片架构的优势。在某些情况下，它可能与提供分片密钥一样简单，在其他情况下，可能无法按分片数据库的要求对数据和工作负载进行水平分区。

许多面向客户的Web应用程序（如电子商务，移动和社交媒体）非常适合分片。此类应用程序具有明确定义的数据模型和数据分布策略（散列，范围，列表或复合），并且主要使用分片密钥访问数据。分片键的示例包括客户ID，帐号和country_id。应用程序通常还需要对数据进行部分去标准化，以便在分片时表现良好。

访问与分片密钥的单个值相关联的数据的OLTP事务是分片数据库的主要用例，例如查找和更新客户的记录，订户文档，金融交易，电子商务交易等。由于分片模式中具有相同分片值的所有行都保证在同一分片上，因此这些事务始终是单分片并以最高性能执行并提供最高级别的一致性。支持多分片操作，但性能和一致性降低。此类事务包括简单聚合，报告等，并且相对于由单个分片OLTP事务支配的工作负载，在分片应用程序中起次要作用。
