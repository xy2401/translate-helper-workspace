新功能

本节列出了Oracle Database 19c版19.1 PL / SQL包和类型参考中的主要新功能。

以下功能是此版本中的新增功能：

  • DBMS_ADDM

    可插拔数据库（PDB）的自动数据库诊断监视器（ADDM）支持。您现在可以使用ADDM分析PDB中的AWR数据，以识别和解决与性能相关的问题。将以下子程序中的两个新参数CDB_TYPE_OVERRIDE和READ_ONLY_TYPE_OVERRIDE添加到DBMS_ADDM程序包：
    • ANALYZE_DB程序
    • ANALYZE_INST过程
    • ANALYZE_PARTIAL过程
    • COMPARE_CAPTURE_REPL AY_REPORT功能
    • COMPARE_DATABASES功能
    • COMPARE_INSTANCES功能
    • COMPARE_REPLAY_REPLA Y_REPORT功能
    • REAL_TIME_ADDM_REPORT功能

  • DBMS_AUTO_INDEX

    新包DBMS_AUTO_INDEX提供了一个界面，用于配置自动索引并生成包含数据库中已执行的自动索引操作的详细信息的报告。

    请参阅“ DBMS_AUTO_INDEX ”

  • DBMS_CREDENTIAL

    您还可以使用DBMS_CREDENTIAL.CREATE_CREDENTIAL过程来管理云对象存储的凭据。impdb的credential参数指定凭证对象的名称，该凭据对象包含访问对象存储桶所需的用户名和密码。您还可以使用数据库属性DEFAULT_CREDENTIAL指定默认凭据。

  • DBMS_HADOOP

    Oracle数据库现在支持混合分区表。混合分区表是一个分区表，其中一些分区驻留在数据库中，一些分区驻留在数据库外部的外部文件中，例如操作系统文件或Hadoop分布式文件系统（HDFS）文件。

    新的过程CREATE_HYBRID_PARTNED_TABLE被添加到DBMS_HADOOP包中。

  • DBMS_INMEMORY_ADMIN

    新功能POPULATE_WAIT被添加到DBMS_INMEMORY_ADMIN包中。此函数确保在允许应用程序访问之前已填充指定的In-Memory对象。

  • DBMS_MEMOPTIMIZE

    以下新子程序作为Memoptimized Rowstore快速摄取功能的一部分添加到DBMS_MEMOPTIMIZE包中：

    • GET_APPLY_HWM_SEQID

    • GET_WRITE_HWM_SEQID

    • WRITE_END

    请参阅“ DBMS_MEMOPTIMIZE ”

  • DBMS_MEMOPTIMIZE_ADMIN

    新包DBMS_MEMOPTIMIZE_ADMIN提供了一个接口，用于将大型池中缓冲的Memoptimized Rowstore数据刷新到磁盘。

    请参见“ DBMS_MEMOPTIMIZE_ADMIN ”

  • DBMS_METADATA

    以下由datapump的元数据层使用的新转换参数将添加到DBMS_METADATA.SET_TRANSFORM_PARAM过程：
    • OMIT_ENCRYPTION_CLAUSE
    • DWCS_CVT_IOTS
    • DWCS_CVT_CONSTRAINTS
    • CONSTRAINT_USE_DEFAULT_INDEXES
    • CONSTRAINT_NAME_FROM_INDEXES

  • DBMS_MVIEW

    Oracle数据库现在支持混合分区表。混合分区表是一个分区表，其中一些分区驻留在数据库中，一些分区驻留在数据库外部的外部文件中，例如操作系统文件或Hadoop分布式文件系统（HDFS）文件。

    新参数SKIP_EXT_DATA将添加到DBMS_MVIEW.REFRESH过程中。

  • DBMS_SQLQ

    新包DBMS_SQLQ提供了一个用于配置SQL语句及其执行计划的隔离阈值的接口。跨越隔离阈值的SQL语句将被隔离，并且不允许在数据库中再次运行。

    请参阅“ DBMS_SQLQ ”

  • DBMS_SQLDIAG新SQL修复顾问函数DBMS_SQLDIAG.SQL_DIAGNOSE_AND_REPAIR

    新的SQL Repair Advisor函数DBMS_SQLDIAG.SQL_DIAGNOSE_AND_REPAIR创建诊断任务，执行它，并接受SQL补丁建议，该SQL补丁建议生成严重错误。此功能可帮助用户诊断和修复问题。

  • DBMS_SODA

    DBMS_SODA是一个实现简单Oracle文档访问（SODA）的PL / SQL包，它允许您将Oracle数据库用作NoSQL文档存储。此版本中添加了其他功能和特性。

  • DBMS_STATS

    新的布尔参数REALTIME_STATS被添加到GET_TABLE_STATS和GET_COLUMN_STATS过程。

  • DBMS_XPLAN

    新的COMPARE_PLANS函数已添加到DBMS_XPLAN包中。此功能将列表中的每个计划与参考计划进行比较并返回报告。

  • 基于密钥的消息传递

    以下新参数添加到GET_QUEUE_PARAMETER ， SET_QUEUE_PARAMETER和UNSET_QUEUE_PARAMETER的程序DBMS_AQADM作为基于密钥的消息功能的一部分包：

    • 分片队列参数
      • SHARD_NUM
      • KEY_BASED_ENQUEUE
      • STICKY_DEQUEUE
    • 基于密钥的参数
      • AQ $ KEY_TO_SHARD_MAP
      • AQ $ GET_KEY_SHARD_INST

    请参阅“ DBMS_AQADM ”

不推荐的功能

Oracle Database 19c版19.1 PL / SQL包和类型参考中不推荐使用的功能列表。

不推荐使用DBMS_SESSION.IS_ROLE_ENABLED函数。

请改用DBMS_SESSION.CURRENT_IS_ROLE_ENABLED或DBMS_SESSION.SESSION_IS_ROLE_ENABLED函数。

不受支持的功能

本节列出了Oracle Database 19c版19.1 PL / SQL包和类型参考中不支持的功能。

此版本不支持以下功能：

  • 支持CONTINUOUS_MINE选项

    Oracle Database 19c中不支持DBMS_LOGMNR.START_LOGMNR DBMS_LOGMNR.START_LOGMNR包的continuous_mine选项，并且该选项不再可用。LogMiner包的CONTINUOUS_MINE功能已过时。它在Oracle Database 12c第2版（12.2）中已弃用。没有替换功能。

    不支持Oracle Streams

    从Oracle Database 19c开始，不支持Oracle Streams功能。使用Oracle GoldenGate替换Oracle Streams的所有复制功能。

  • Oracle多媒体

    从Oracle Database 19c开始，不支持Oracle多媒体功能。将Oracle SecureFiles与开源或第三方产品一起用于元数据提取以及图像处理和转换，而不是Oracle多媒体。

    也可以看看：

    有关更多信息，请参见Oracle数据库升级指南

  • 不支持LOG_ARCHIVE_DEST_n的MAX_CONNECTIONS属性

    Oracle Data Guard重做传输的LOG_ARCHIVE_DEST_n参数的MAX_CONNECTIONS属性已过时。Oracle Database 19c中不支持它。

    MAX_CONNECTIONS属性可能会干扰Oracle Database 11g中引入的新重做传输流机制，并增加了解决间隙所需的时间。为防止出现这些类型的错误，Oracle已取消支持并删除了此属性。

  • 支持扩展数据类型支持（EDS）

    Oracle Database 19c中不支持扩展数据类型支持（EDS）功能。现在，Logical Standby和Oracle GoldenGate都支持EDS功能支持的所有数据类型。

    扩展数据类型支持（EDS）功能为逻辑备用数据库提供了一种机制，以支持缺少基于本机重做的支持的某些Oracle数据类型。例如，EDS用于复制具有SDO_GEOMETRY列的表。但是，从Oracle Database 12c第2版（12.2）开始，没有任何EDS支持的Oracle数据类型本身不受逻辑备用数据库或Oracle GoldenGate支持。Oracle Database 19c（19.1）不支持此功能。

2.3建议失败

目的

使用ADVISE FAILURE命令显示指定故障的修复选项。此命令打印Data Recovery Advisor识别的故障摘要，并隐式关闭所有已修复的打开故障。

建议的工作流程是在RMAN会话中运行以下命令： LIST FAILURE显示故障， ADVISE FAILURE显示修复选项， REPAIR FAILURE修复故障。

先决条件

RMAN必须连接到目标数据库。请参阅CONNECT和RMAN命令以了解如何以TARGET身份连接到数据库。

必须启动目标数据库实例。目标数据库必须是单实例数据库，并且不能是物理备用数据库，尽管它可以是逻辑备用数据库。

在当前版本中，Data Recovery Advisor仅支持单实例数据库。不支持Oracle Real Application Clusters（Oracle RAC）数据库。

使用说明

在提出修复策略之前，Data Recovery Advisor会验证修复可行性。例如，Data Recovery Advisor会检查介质恢复所需的所有备份和归档重做日志文件是否可用。ADVISE FAILURE输出指示Data Recovery Advisor认为对于给定的一组故障最佳的修复策略。ADVISE FAILURE命令可以生成手动和自动修复选项。

手动修复选项

手动修复选项是强制性的或可选的。可选操作可以比自动修复更快速或更容易地修复故障。例如，Data Recovery Advisor可能会建议将故障转移到备用数据库，以替代主数据库修复。

在其他情况下，唯一的选择是手动，因为自动修复是不可行的。例如，I / O故障通常无法自动修复。此外，有时无法诊断故障，因为操作系统或磁盘子系统返回的数据不足。

自动修复选项

每个自动修复选项都是单个修复或一组修复步骤（有关命令输出的说明，请参阅表2-1 ）。当修复选项具有包含多个修复步骤的脚本时， ADVISE FAILURE生成脚本，以便修复步骤的顺序正确。单个维修始终将关键故障一起修复。您必须修复关键故障，但您也可以同时修复非关键故障。您可以按随机顺序，逐个或成组修复非关键故障。

Oracle RAC和Data Recovery Advisor

如果数据故障导致Oracle RAC数据库的所有实例关闭，则可以在单实例模式下装入数据库，并使用Data Recovery Advisor检测和修复控制文件， SYSTEM数据文件和字典故障。您还可以启动运行状况检查以测试其他数据库组件的数据故障。此方法不会检测其他群集实例本地的数据故障，例如，无法访问的数据文件。

13.1条款可以进入

ACCESSIBLE BY子句限制对单元和子程序的访问。

ACCESSIBLE BY子句限制其他单元对单元或子程序的访问。访问者列表 （也称为白名单）明确列出了可能具有访问权限的那些单元。可以在包中的各个子程序上定义访问者列表。除了在包本身上定义的访问者列表（如果有的话）之外，还检查该列表。此列表可能仅限制对子程序的访问 - 它无法扩展访问权限。此代码管理功能可用于防止无意中使用内部子程序。例如，将包重组为两个包可能不方便或不可行：一个用于需要限制访问的少量过程，另一个用于需要公共访问的其余单元。

ACCESSIBLE BY子句可能出现在对象类型，对象类型主体，包和子程序的声明中。

ACCESSIBLE BY子句可以出现在以下SQL语句中：

  • ALTER TYPE语句

  • 创建功能声明

  • CREATE PROCEDURE声明

  • CREATE PACKAGE声明

  • CREATE TYPE语句

  • CREATE TYPE BODY声明

指定可以调用实体的已存储PL / SQL单元。

每个accessor指定可以访问包含ACCESSIBLE BY子句的实体的另一个PL / SQL实体。

当出现ACCESSIBLE BY子句时，只有该子句中指定的实体可以访问该子句出现的实体。

accessor可能在ACCESSIBLE BY子句中出现多次。

ACCESSIBLE BY子句只能在单元声明中出现一次。

不需要存在在accessor命名的实体。

当调用具有ACCESSIBLE BY子句的实体时，它会在执行所有其他检查后强制执行其他访问检查。这些检查是：

  • 被调用的单元必须包括一个accessor以相同的UNIT_NAME和unit_kind作为调用单元。

  • 如果accessor包含模式 ，则调用单元必须位于该模式中 。

  • 如果accessor不包含模式 ，则调用者必须来自与调用实体相同的模式。

unit_kind

指定单位是FUNCTION ， PACKAGE ， PROCEDURE ， TRIGGER还是TYPE 。

使用说明

unit_kind是可选的，但建议在单位具有相同名称时指定它以避免歧义。例如，可以定义与函数同名的触发器。

只有当呼叫是直接的时， ACCESSIBLE BY子句才允许访问。如果通过静态SQL， DBMS_SQL或动态SQL进行访问，则检查将失败。

将根据包规范的访问者列表检查对包规范或包体的初始化过程的任何调用。

一个单位总是可以访问自己。单元中的项目可以引用同一单元中的另一个项目。

对受保护子程序的RPC调用将始终失败，因为在编译时或运行时没有可用于检查调用有效性的上下文。

从条件编译指令调用受保护的子程序将失败。

关于sqlnet.ora文件中的密钥库位置

Oracle建议您使用WALLET_ROOT和TDE_CONFIGURATION参数来配置密钥库位置，但另一种方法是配置sqlnet.ora文件。

但是，如果您确实要配置sqlnet.ora文件，请注意建议使用WALLET_ROOT和TDE_CONFIGURATION参数的基于实例初始化文件的方法。它避免了使用sqlnet.ora时可能出现的配置不一致的可能性，因为每次连接到数据库都会重新读取sqlnet.ora参数文件的内容。使用基于实例初始化文件的方法，因为WALLET_ROOT参数在数据库实例启动时只读取一次，所以不存在不一致的风险。在多租户环境中，使用WALLET_ROOT初始化参数初始化参数比使用sqlnet.ora更WALLET_ROOT ，因为它使每个租户PDB能够执行独立的管理操作。

如果尚未设置WALLET_ROOT和TDE_CONFIGURATION初始化参数，则Oracle数据库会检查sqlnet.ora文件以获取密钥库的目录位置。这适用于密钥库是软件密钥库，硬件模块安全性（HSM）密钥库还是Oracle密钥库密钥库。您必须编辑sqlnet.ora文件以定义计划创建的密钥库的目录位置。确保预先存在此目录。优选地，该目录应该是空的。

必须在Oracle Real Application Clusters（Oracle RAC）或多租户环境中编辑sqlnet.ora文件时，请注意以下行为：

  • 在Oracle RAC环境中：如果要使用srvctl实用程序，并且要在sqlnet.ora配置文件中包含环境变量，则必须在操作系统和srvctl环境中设置这些环境变量。Oracle建议您将密钥库放在共享文件系统上，例如Oracle自动存储管理（ASM）或NFS。

  • 在多租户环境中：如果在多租户环境中将sqlnet.ora用于TDE配置，则仅支持联合模式。仅当WALLET_ROOT和TDE_CONFIGURATION参数用于TDE配置时，才支持隔离模式。

在sqlnet.ora文件中，必须设置ENCRYPTION_WALLET_LOCATION参数以指定密钥库位置。但是，请注意，不推荐使用ENCRYPTION_WALLET_LOCATION ，从Oracle Database版本19c开始，支持使用WALLET_ROOT和TDE_CONFIGURATION初始化参数。

默认情况下， sqlnet.ora文件位于$ORACLE_BASE/network/admin目录（如果设置了ORACLE_BASE环境变量）或$ORACLE_HOME/network/admin位置，或者位于TNS_ADMIN环境变量设置的位置。确保已正确设置TNS_ADMIN环境变量以指向正确的sqlnet.ora文件。如果未在sqlnet.ora文件中设置密钥库位置，则V$ENCRYPTION_WALLET视图将显示默认位置。您可以在V$ENCRYPTION_WALLET视图中检查密钥库的位置和状态。

6.1 Oracle Data Guard保护模式

Oracle Data Guard提供三种保护模式：最大可用性，最高性能和最大保护。

在以下保护模式的描述中，同步备用数据库应该是满足配置的数据保护模式的最低要求且没有重做间隙的数据库。Redo间隙在Redo Gap Detection and Resolution中进行了讨论。

最高可用性

此保护模式可提供最高级别的数据保护，而不会影响主数据库的可用性。在正常操作下，事务不会提交，直到恢复这些事务所需的所有重做数据都已写入联机重做日志并且基于用户配置，以下情况之一为真：

  • 已在备用数据库中收到重做，已启动备用重做日志的I / O，并将确认发送回主数据库

  • redo已被接收并写入待机状态的备用重做日志，并将确认发送回主数据库

如果主服务器没有从至少一个同步备用数据库接收到确认，那么它就像它处于最高性能模式一样运行，以保持主数据库可用性，直到它再次能够将其重做流写入同步备用数据库。

如果主数据库发生故障，则此模式可确保在Oracle Data Guard配置中至少有一个同步备用数据库时不会发生数据丢失。有关支持最高可用性和相关权衡所需的重做传输设置的信息，请参阅“ 最高可用性模式下的性能与保护 ” 。

一旦Oracle Data Guard将重做数据写入备用重做日志文件中的持久存储，主要事务就被视为受保护。完成后，确认将快速返回主数据库，以便它可以继续进行下一个事务。这最大限度地减少了同步传输对主数据库吞吐量和响应时间的影响。要在备用数据库中完全受益于完整的Oracle Data Guard验证，请确保以实时应用模式运行，以便重新更改以尽可能快的速度应用于备用数据库。Oracle Data Guard会发出任何检测到的损坏信号，以便立即采取纠正措施。

最高可用性模式下的性能与保护

使用最高可用性模式时，了解使用LOG_ARCHIVE_DEST_ n属性SYNC / AFFIRM与SYNC / NOAFFIRM （FastSync）的可能结果非常重要。 ）这样您就可以做出最适合您需求的选择。

当使用SYNC/AFFIRM执行传输时，主服务器执行写操作并等待确认重做已同步传输到物理备用数据库并写入磁盘。SYNC/AFFIRM传输提供额外的保护优势，代价是由完成备用重做日志的I / O所需的时间所导致的性能影响。

使用SYNC/NOAFFIRM执行SYNC/NOAFFIRM ，主服务器执行写操作，仅等待确认已在备用数据库上接收到数据， 而不是等待已写入磁盘。SYNC/NOAFFIRM传输可以在多个同时发生故障的特殊情况下以潜在的数据丢失风险为代价提供性能优势。

考虑到这些定义，假设您在主站点遇到灾难性故障，同时在备用站点断电。数据是否丢失取决于所使用的传输模式。在SYNC/AFFIRM的情况下，其中有一个检查以确认数据被写入备用数据库上的磁盘，因此没有数据丢失，因为当系统恢复时，数据将在备用数据库上可用。在SYNC/NOAFFIRM的情况下，其中没有检查数据是否已在备用数据库上写入磁盘，可能会有一些数据丢失。

这种保护 mode可提供最高级别的数据保护，而不会影响主数据库的性能。这是通过允许事务在这些事务生成的所有重做数据写入在线日志后立即提交来实现的。重做数据也写入一个或多个备用数据库，但这是在事务承诺方面异步完成的，因此主数据库性能不受传输重做数据和从备用数据库接收确认所需的时间的影响。

与最大可用性模式相比，此保护模式提供的数据保护略少，对主数据库性能的影响最小。

这是默认保护模式。

最大保护

最大保护与最大可用性类似，但在发生多个故障事件时提供额外级别的数据保护。与最大可用性不同，最大可用性允许主服务器在无法从备用数据库接收确认时继续处理，最大保护会关闭主数据库，而不是允许它继续处理不受保护的事务。

由于此数据保护模式优先考虑主数据库可用性的数据保护，因此Oracle建议至少使用两个备用数据库来保护以最大保护模式运行的主数据库，以防止单个数据库故障导致主数据库关闭。

目的

PREDICTION返回选择中每行的预测。返回预测的数据类型取决于函数是执行回归，分类还是异常检测。

  • 回归 ：返回每行的预期目标值。返回值的数据类型是目标的数据类型。

  • 分类 ：返回每行最可能的目标类（或指定成本的最低成本目标类）。返回值的数据类型是目标的数据类型。

  • 异常检测 ：每行返回1或0 。典型行分类为1。与其他数据显着不同的行归类为0。

cost_matrix_clause

成本是最小化最有害的错误分类的偏向因素。您可以为分类或异常检测指定cost_matrix_clause 。成本与回归无关。cost_matrix_clause行为与“ 预测成本 ”的描述相同。

语法选择

PREDICTION可以通过以下两种方式之一对数据进行评分：它可以将挖掘模型对象应用于数据，或者可以通过执行构建和应用一个或多个瞬态挖掘模型的分析子句来动态评分数据。选择语法或分析语法 ：

  • 语法 ：使用此语法使用预定义的模型对数据进行评分。提供执行分类，回归或异常检测的模型的名称。

  • 分析语法 ：使用分析语法在没有预定义模型的情况下对数据进行评分。分析语法使用mining_analytic_clause ，它指定是否应为多个模型构建分区数据。mining_analytic_clause支持query_partition_clause和order_by_clause 。（参见“ analytic_clause :: = ” 。）

    • 对于回归，请指定FOR expr ，其中expr是一个表达式，用于标识具有数字数据类型的目标列。

    • 对于Classification，请指定FOR expr ，其中expr是一个表达式，用于标识具有字符数据类型的目标列。

    • 对于异常检测，请指定关键字OF ANOMALY 。

在对分区模型进行评分时， PREDICTION函数的语法可以使用可选的GROUPING提示。请参阅GROUPING提示 。

mining_attribute_clause

mining_attribute_clause标识要用作评分预测变量的列属性。

  • 如果指定USING * ，则使用输入行中存在的所有相关属性。

16.1关于指数平滑

指数平滑方法广泛用于预测。

指数平滑方法已被广泛用于半个多世纪的预测中。它具有战略，战术和运营层面的应用。例如，在战略层面，预测用于预测投资回报，增长和创新的影响。在战术层面，预测用于预测成本，库存需求和客户满意度。在操作层面，预测用于设定目标和预测质量以及与标准的一致性。

在其最简单的形式中， 指数平滑是一种移动平均方法，其具有单个参数，其模拟过去水平对未来值的指数减小的影响。通过各种扩展， 指数平滑涵盖了比竞争对手更广泛的模型，例如Box-Jenkins自回归集成移动平均（ARIMA）方法。Oracle Data Mining使用最先进的状态空间方法实现指数平滑 ，该方法包含单一错误源（SSOE）假设，可提供理论和性能优势。

指数平滑扩展到以下：

  • 混合和匹配错误类型（加法或乘法），趋势（加法，乘法或无）和季节性（加法，乘法或无）的模型矩阵

  • 阻尼趋势的模型。

  • 直接处理具有缺失值的不规则时间序列和时间序列的模型。

注意：

有关更多信息，请参阅Ord，JK等， 时间序列预测：单一误差状态空间方法的案例，工作文件 ，计量经济学和商业统计系，莫纳什大学，VIC 3800，澳大利亚，2005年4月2日。

16.1.1指数平滑模型

指数平滑模型是一类直观，灵活且可扩展的广泛预测模型。

该类成员包括简单的单参数模型，可将未来预测为先前水平和当前冲击的线性组合。扩展可以包括线性或非线性趋势的参数，趋势阻尼，简单或复杂的季节性，相关序列，预测方程中的各种形式的非线性，以及不规则时间序列的处理。

指数平滑假设一系列无限延伸到过去，但过去对未来的影响，平稳地和指数地快速衰减。平滑的衰减速率由一个或多个平滑常数表示。平滑常数是模型估计的参数。该假设通过使用等效递归公式来实际建模现实世界数据，该递归公式仅基于基于先前历史的当前水平的估计和仅依赖于当前条件的该估计的冲击来表达。该程序需要对第一次观察之前的时间段进行估计，该时间段包含所有先前的历史记录。该初始观察是另外的模型参数，其值通过建模过程估计。

ESM的组件（例如趋势和季节性扩展）可以具有加法或乘法形式。更简单的加法模型假设冲击，趋势和季节性是递归公式中的线性效应。

16.1.2简单指数平滑

简单指数平滑假设数据围绕静止均值波动，没有趋势或季节性模式。

在简单指数平滑模型中，每个预测（平滑值）被计算为先前观察的加权平均值，其中权重根据平滑常数α的值指数地减小。平滑常数α近1的值几乎都是最近观察的重量。α接近零的值允许远距离过去的观察具有很大的影响。

16.1.3趋势但没有季节性的模型

加性（线性）趋势的优选形式有时被称为Holt方法或双指数平滑。

具有趋势的模型添加平滑参数γ并且可选地添加阻尼参数φ。阻尼参数平滑地抑制过去线性趋势对未来水平估计的影响，通常提高准确性。

16.1.4具有季节性但没有趋势的模型

当时间序列平均值不随时间变化（静止），但受季节性波动影响时，适当的模型具有季节性参数但没有趋势。

假设季节性波动在长度为m的周期内平衡，其中m是季节数。例如，当输入数据按季度汇总时，可以使用m = 4。对于具有附加误差的模型，季节性参数必须总和为零。对于具有乘法误差的模型，季节性参数的乘积必须为1。

16.1.5具有趋势和季节性的模型

Holt和Winters在指数平滑模型 （ESM）中引入了趋势和季节性。原始模型，也称为Holt-Winters或三指数平滑，被认为是一种加性趋势和乘法季节性。扩展包括具有加性和乘性趋势，季节性和误差的各种组合的模型，具有和不具有趋势阻尼。

16.1.6预测间隔

为了计算预测区间， 指数平滑模型 （ESM）分为三类。

最简单的类是线性模型，其中包括简单的ESM，Holt的方法和附加的Holt-Winters。2类模型（乘法误差，附加分量）对违反常态假设进行近似校正。3类模式使用简单的模拟方法来计算预测间隔。

16.2.1输入数据

时间序列分析，需要有序的输入数据。因此，每个数据行必须由[index，value]对组成，其中索引指定排序。

当CREATE_MODEL过程用于启动指数平滑 （ESM）模型构建时， CASE_ID_COLUMN_NAME指定用于计算输入索引的列， TARGET_COLUMN_NAME指定用于计算观察到的时间序列值的列。时间列包含Oracle编号，Oracle日期，时间戳，带时区的时间戳或带有本地时区的时间戳。输入时间序列根据CASE_ID （时间标签）的值进行排序。case id列不能包含缺失值。值列可以包含表示为NULL缺失值。 ESM还支持分区模型，在这种情况下，输入表包含指定分区的额外列。具有相同分区ID的所有[index，value]对形成一个完整的时间序列。尽管所有模型都使用相同的模型设置，但指数平滑为每个分区独立构建模型。

数据属性可能会导致警告消息或忽略设置。如果用户指定具有乘法趋势，乘法季节性或两者的模型并且数据包含值Y t <= 0，则忽略设置，则将模型类型设置为默认值。如果系列包含的值少于用户指定季节的数量，则会忽略季节性规范并发出警告。

16.2.2累积

对于指数平滑算法，当datetime日期类型（ date ， datetime timestamp ， timestamp with timezone timestamp with local timezone或timestamp with local timezone ） timestamp with timezone ，将应用累积过程。

case id可以是NUMBER列，其排序索引表示值在时间序列序列中的位置。case id列也可以是日期类型。根据用户指定的累积窗口累积日期类型。无论类型如何，case id用于将列转换为等间隔的时间序列。对于NUMBER类型的案例ID，不应用累积。例如，考虑有关促销活动的时间序列。时间列包含每个事件的日期，日期可以不等间隔。用户必须指定间隔间隔，间隔间隔是累积或变换的等间隔时间序列的间隔。在该示例中，如果用户将间隔指定为月，则从原始时间序列生成每个日历月具有利润的等间隔时间序列。设置EXSM_INTERVAL用于指定间隔间隔。用户还必须为EXSM_ACCUMULATE指定一个值，例如EXSM_ACCU_MAX ，在这种情况下，等间隔的月度系列将包含该月份所有事件的最大利润作为观察到的时间序列值。

16.2.3缺失值

输入时间序列可以包含缺失值。目标列中的NULL条目表示缺少值。当time列的类型为datetime时，累积过程也会引入缺失值。设置EXSM_SETMISSING可用于指定如何处理缺失值。特殊值EXSM_MISS_AUTO表示，如果系列包含缺失值，则将其视为不规则的时间序列。

注意：

缺少值处理设置必须与模型设置兼容，否则将引发错误。

16.2.4预测

可以应用指数平滑模型（ESM）通过指定预测窗口来进行预测。

设置EXSM_PREDICTION_STEP可用于指定预测窗口。当时间列属于日期时间类型时，预测窗口以间隔数（设置EXSM_INTERVAL ）表示。如果时间列是数字，则预测窗口是要预测的步骤数。无论时间序列是规则的还是不规则的， EXSM_PREDICTION_STEP指定预测窗口。

16.2.5分区的并行性

Oracle Advanced Analytics通过分区支持并行。

例如，用户可以选择PRODUCT_ID作为一个分区列，并可以为模型构建中的不同产品生成预测。虽然为每个分区构建了一个独特的平滑模型，但所有分区都共享相同的模型设置。例如，如果将EXSM_MODEL设置为EXSM_SIMPLE ，则所有分区模型都将是简单的指数平滑模型。来自不同分区的时间序列可以分布到不同的进程并且并行处理。每个时间序列的模型是串行构建的。

时间序列分析，需要有序的输入数据。因此，每个数据行必须由[index，value]对组成，其中索引指定排序。

当CREATE_MODEL过程用于启动指数平滑 （ESM）模型构建时， CASE_ID_COLUMN_NAME指定用于计算输入索引的列， TARGET_COLUMN_NAME指定用于计算观察到的时间序列值的列。时间列包含Oracle编号，Oracle日期，时间戳，带时区的时间戳或带有本地时区的时间戳。输入时间序列根据CASE_ID （时间标签）的值进行排序。case id列不能包含缺失值。值列可以包含表示为NULL缺失值。 ESM还支持分区模型，在这种情况下，输入表包含指定分区的额外列。具有相同分区ID的所有[index，value]对形成一个完整的时间序列。尽管所有模型都使用相同的模型设置，但指数平滑为每个分区独立构建模型。

数据属性可能会导致警告消息或忽略设置。如果用户指定具有乘法趋势，乘法季节性或两者的模型并且数据包含值Y t <= 0，则忽略设置，则将模型类型设置为默认值。如果系列包含的值少于用户指定季节的数量，则会忽略季节性规范并发出警告。

15.2.2累积

对于指数平滑算法，当datetime日期类型（ date ， datetime timestamp ， timestamp with timezone timestamp with local timezone或timestamp with local timezone ） timestamp with timezone ，将应用累积过程。

case id可以是NUMBER列，其排序索引表示值在时间序列序列中的位置。case id列也可以是日期类型。根据用户指定的累积窗口累积日期类型。无论类型如何，case id用于将列转换为等间隔的时间序列。对于NUMBER类型的案例ID，不应用累积。例如，考虑有关促销活动的时间序列。时间列包含每个事件的日期，日期可以不等间隔。用户必须指定间隔间隔，间隔间隔是累积或变换的等间隔时间序列的间隔。在该示例中，如果用户将间隔指定为月，则从原始时间序列生成每个日历月具有利润的等间隔时间序列。设置EXSM_INTERVAL用于指定间隔间隔。用户还必须为EXSM_ACCUMULATE指定一个值，例如EXSM_ACCU_MAX ，在这种情况下，等间隔的月度系列将包含该月份所有事件的最大利润作为观察到的时间序列值。

15.2.3缺失值

输入时间序列可以包含缺失值。目标列中的NULL条目表示缺少值。当time列的类型为datetime时，累积过程也会引入缺失值。设置EXSM_SETMISSING可用于指定如何处理缺失值。特殊值EXSM_MISS_AUTO表示，如果系列包含缺失值，则将其视为不规则的时间序列。

注意：

缺少值处理设置必须与模型设置兼容，否则将引发错误。

15.2.4预测

可以应用指数平滑模型（ESM）通过指定预测窗口来进行预测。

设置EXSM_PREDICTION_STEP可用于指定预测窗口。当时间列属于日期时间类型时，预测窗口以间隔数（设置EXSM_INTERVAL ）表示。如果时间列是数字，则预测窗口是要预测的步骤数。无论时间序列是规则的还是不规则的， EXSM_PREDICTION_STEP指定预测窗口。

15.2.5分区的并行性

Oracle Advanced Analytics通过分区支持并行。

例如，用户可以选择PRODUCT_ID作为一个分区列，并可以为模型构建中的不同产品生成预测。虽然为每个分区构建了一个独特的平滑模型，但所有分区都共享相同的模型设置。例如，如果将EXSM_MODEL设置为EXSM_SIMPLE ，则所有分区模型都将是简单的指数平滑模型。来自不同分区的时间序列可以分布到不同的进程并且并行处理。每个时间序列的模型是串行构建的。

19.2 NCHAR，NVARCHAR2，NCLOB和defaultNChar属性

默认情况下，是oracle.jdbc.OraclePreparedStatement接口以与在数据库字符集中编码的方式相同的方式处理所有列的数据类型。但是，从Oracle Database 10g开始 ，如果将oracle.jdbc.defaultNChar系统属性的值设置为true ，则JDBC会将所有字符列视为国家/地区语言。

defaultNChar的默认值为false。如果defaultNChar值为false，则必须调用setFormOfUse(<column_Index>, OraclePreparedStatement.FORM_NCHAR)方法用于那些特别需要国家语言字符的列。例如：

PreparedStatement pstmt = conn.prepareStatement（“插入TEST值（？，？，？）“）; pstmt.setFormOfUse（1，OraclePreparedStatement。FORM_NCHAR）; pstmt.setString（1，myUnicodeString1）; // NCHAR列pstmt.setFormOfUse（2，OraclePreparedeStatement。FORM_NCHAR）; pstmt.setString（2，myUnicodeString2）; // NVARCHAR2列

如果要将defaultNChar的值设置为true ，请在命令行中指定以下内容：

java -Doracle.jdbc.defaultNChar = true myApplication

如果您愿意，则还可以将defaultNChar指定为连接属性并访问NCHAR ， NVARCHAR2或NCLOB数据。 

属性props = new Properties（）; props.put（的OracleConnection。CONNECTION_PROPERTY_DEFAULTNCHAR，“true”）; //设置URL，用户名，密码等。...Connection conn = DriverManager.getConnection（props）;

如果defaultNChar值为true ，那么您应该为不需要国家语言字符的列调用setFormOfUse(<column_Index>, FORM_CHAR) 。例如：

PreparedStatement pstmt = conn.prepareStatement（“插入TEST值（？，？，？）“）; pstmt.setFormOfUse（3，OraclePreparedStatement。FORM_CHAR）; pstmt.setString（3，myString）; // CHAR列

注意：

如果将defaultNChar的值设置为true ，然后访问CHAR列，则数据库将隐式将所有CHAR数据转换为NCHAR 。此转换对性能产生重大影响。

注意：

  • 始终使用java.lang.字符数据的String而不是oracle.sql.CHAR 。提供CHAR仅用于向后兼容。

  • 您还可以使用setObject方法访问国家字符集类型，但如果使用setObject方法，则必须将目标数据类型指定为Types.NCHAR ， Types.NCLOB ， Types.NVARCHAR或Types.LONGNVARCHAR 。

注意：

在Oracle数据库中，SQL字符串将转换为数据库字符集。因此，您需要牢记以下内容：

  • 在Oracle Database 10g第 1版（10.1）及更早版本中，JDBC驱动程序不支持包含在数据库字符集中无法表示的Unicode字符的任何NCHAR文字（n'...'）。在数据库字符集中无法表示的所有Unicode字符都会损坏。

  • 如果Oracle数据库10g第 2版（10.2）JDBC驱动程序连接到Oracle数据库10g第 2版（10.2）数据库服务器，则所有NCHAR文字（n'...'）都将转换为Unicode文字（u'。 ..'）和所有非ASCII字符都转换为相应的Unicode转义序列。这是自动完成的，以防止数据损坏。

  • 如果Oracle数据库10g发行版2（10.2）JDBC驱动程序连接到Oracle数据库10g发行版1（10.1）或更早版本的数据库服务器，则不会转换NCHAR文字（n'...'）并且不会转换任何字符在数据库中无法表示的字符集会被破坏。

9.4.1连接亲和性概述

UCP JDBC连接池利用Oracle RAC数据库提供的关联功能。连接关联需要使用Oracle JDBC驱动程序和Oracle RAC数据库版本11.1.0.6或更高版本。

连接关联性是一种性能功能，它使连接池能够选择针对特定Oracle RAC实例的连接。池使用运行时连接负载平衡（如果已配置）来选择Oracle RAC实例以创建第一个连接，然后创建后续连接，并且对同一实例具有亲缘关系。

9.4.1.1基于交易的亲和力

基于事务的关联是与Oracle RAC实例的关联，可以由客户端应用程序或失败事件释放。当需要与Oracle RAC实例的长期关联性或者重定向到新的Oracle RAC实例的成本（在性能方面）很高时，应用程序通常使用此类型的关联。分布式事务是基于事务的关联的一个很好的例子。在分布式事务中登记的XA连接在事务持续期间与Oracle RAC实例保持密切关系。在这种情况下，如果在分布式事务期间将连接重定向到其他Oracle RAC实例，则应用程序将产生显着的性能成本。

9.4.1.2 Web会话亲和性

Web会话关联是与Oracle RAC实例的关联，可以由实例，客户端应用程序或失败事件释放。Oracle RAC实例使用提示与连接池进行通信，无论是否已在实例上启用或禁用了关联。Oracle RAC实例可能会根据许多因素（如性能或负载）禁用关联。如果Oracle RAC实例不再支持关联，则刷新池中的连接以使用新实例，并再次建立关联。

当预期与Oracle RAC实例的短期关联性或者重定向到新的Oracle RAC实例的成本（在性能方面）最小时，应用程序通常使用此类型的关联。例如，邮件客户端会话可能会使用与Oracle RAC实例的Web会话关联来提高性能，并且如果将连接重定向到其他实例，则相对不受影响。

9.4.1.3 Oracle RAC数据亲和性

从Oracle Database Release 18c开始，UCP支持Oracle RAC Data Affinity。在Oracle RAC数据库上启用Data Affinity时，对关联表的数据进行分区，使表的特定分区或行子集与特定的Oracle RAC数据库实例相关联。由于改进了缓存局部性并减少了RAC实例之间的节点间同步和块ping，因此亲和性可以为应用程序带来更高的性能和可伸缩性。

https://docs.oracle.com/middleware/1212/coherence/COHDG/api_dataaffinity.htm#COHDG5577

要使用Oracle RAC Data Affinity功能，通过UCP访问数据库的客户端必须在其连接请求中提供数据关联密钥。在为启用关联的RAC数据库池连接时，UCP具有以下功能：

  1. UCP在池启动时了解包含Oracle RAC实例中数据分区的数据关联的拓扑。

  2. 需要利用Oracle RAC Data Affinity功能的UCP连接请求使用分片密钥构建器提供数据关联密钥，并使用连接构建器，如下所示：

    您仍然可以在不提供数据关联密钥的情况下启用对Oracle RAC Data Affinity启用的连接请求。但是，在这种情况下，您将看不到Oracle RAC Data Affinity功能的好处。

  3. UCP确定请求中提供的分片键的关联实例，并检查池中是否存在该实例的连接。如果存在连接，则它用于提供请求。如果池中不存在匹配的连接，则回退到运行时负载平衡会为请求选择连接并为其提供服务。如果需要创建新连接以提供请求，则将请求路由到与提供的分片（数据亲和性）密钥对应的关联实例。

  4. 当存在HA事件或Oracle RAC上的数据分区的关联性发生更改时，UCP会使其数据分区的拓扑与服务器端保持同步。

12.2.3防止过度配合

原则上，决策树算法可以使树的每个分支生长得足够深，以便对训练样例进行完美分类。虽然这有时是一种合理的策略，但事实上，当数据中存在噪声时，或者当训练样本的数量太小而无法产生真正目标函数的代表性样本时，它会导致困难。在这两种情况中的任何一种情况下，这种简单的算法都可以产生过度拟合训练样例的树。过度拟合是一种情况，其中模型能够准确地预测用于创建模型的数据，但在呈现给它的新数据上表现不佳。

为防止过度拟合，Oracle Data Mining支持控制树增长的自动修剪和可配置限制条件 。限制条件可以在条件满足后防止进一步分裂。修剪会删除具有无关紧要的预测能力的分支。

3.9 RESYNC CATALOG

目的

使用RESYNC CATALOG命令在恢复目录模式中使用目标数据库控制文件中的元数据执行元数据的完全重新同步。您还可以使用FROM CONTROLFILECOPY子句将当前控制文件与控制文件副本中的RMAN元数据重新同步。

通常，在以下情况下运行RESYNC CATALOG ：

  • 执行自动执行重新同步的RMAN命令时，恢复目录不可用。

  • 目标数据库以ARCHIVELOG模式运行，因为在发生联机重做日志切换或归档重做日志时，恢复目录不会自动更新。

  • 您对目标数据库的物理结构进行了更改，例如添加或删除表空间。与日志归档一样，当物理架构更改时，恢复目录不会自动更新。

  • RMAN作为TARGET连接到备用数据库。您希望使用有关在此数据库上执行的RMAN操作的元数据更新恢复目录。

  • RMAN作为TARGET连接到备用数据库。您希望使用有关主数据库上的物理更改的元数据更新恢复目录（请参阅示例3-45 ）。

先决条件

RMAN必须作为TARGET连接到已安装或打开的数据库，并作为CATALOG连接到恢复目录数据库。运行具有多个数据库的RESYNC CATALOG时，每个目标数据库都需要网络连接。

使用说明

重新同步已完全或部分。如果已满，并且目标数据库已挂载当前控制文件（但不是新创建的控制文件或比先前使用的控制文件更新的控制文件），则RMAN将更新物理模式的所有已更改记录：数据文件，表空间，重做线程和联机重做日志。如果数据库已打开，则RMAN还会获取有关回滚段的数据。如果重新同步是部分的，则RMAN不会重新同步有关物理架构或回滚段的元数据。

如果在命令执行时装入了目标控制文件并且目录数据库可用，则在使用RMAN命令时，RMAN会根据需要自动重新同步恢复目录。在对数据库进行结构更改（添加或删除数据库文件，创建新的化身等）或更改RMAN持久性配置之后，RMAN会执行完全重新同步。

从Oracle Database 11g开始 ，单个恢复目录模式可以跟踪Data Guard环境中所有数据库的数据库文件名。此目录模式还会跟踪为所有数据库创建联机重做日志，备用重做日志，临时文件，归档重做日志文件，备份集和映像副本的位置。如果RMAN作为TARGET连接到备用数据库，则如果备用控制文件包含有关主数据库上的物理架构更改的信息，则RMAN将隐式执行完全重新同步。

10配置OLTP

以下部分介绍如何配置联机事务处理OLTP。

  • 如果您的通信协议是TCP / IP ：继续为TCP / IP环境配置OLTP 。

    注意：

    在使用TCP / IP支持IMS Connect的网关上，如果需要输入区分大小写，则必须在IMS TRANSACT宏中指定EDIT=ULC 。使用SNA支持时，无需在IMS TRANSACT宏中指定EDIT=ULC 。

配置SNA环境的OLTP

配置OLTP以与APPC的Oracle Database Gateway通信的步骤取决于您使用的OLTP以及运行OLTP的平台。适用于z / OS，IMS / TM，APPC / MVS和的CICS Transaction Server z / OS是目前支持的 OLTPs。从以下部分中选择与OLTP相对应的说明：

  • 为z / OS配置CICS Transaction Server

  • 配置IMS / TM

  • 配置APPC / MVS

    注意：

    如果这是您第一次配置OLTP，则只需执行OLTP的配置步骤。

为z / OS配置CICS Transaction Server

如果您的OLTP是用于z / OS的CICS Transaction Server，则执行以下步骤以配置与网关的通信：

  1. 配置MVS 用于SNA通信包的VTAM，它将使APPC连接到您的系统。至少一个独立的 LU必须可用于网关。
  2. 检查 CICS Transaction Server for z / OS使用的VTAM日志模式表。 （表名在CICS的VTAM APPL定义中的MODETAB参数中指定。）确保具有并行会话和同步级别支持的APPC会话存在条目。

    $ORACLE_HOME/dg4appc/sna目录中的oraplu62.asm文件包含一个示例模式条目，包括指示模式条目中所需值的注释。

  3. 使用文件传输工具，将以下文件从$ORACLE_HOME/dg4appc/demo/CICS目录传输到运行CICS Transaction Server for z / OS的z / OS系统：

    • dfhcsdup.jcl ：JCL运行CICS DFHCSDUP实用程序

    • pgaflip.asm ：CICS FLIP事务的汇编源

    • pgaflip.jcl ：用于汇编和链接CICS FLIP事务的JCL

  4. 使用中的注释 dfhcsdup.jcl文件，定制JCL和输入语句以匹配您的系统设置，并提交它以进行批量执行。执行此步骤将更新Transaction Server for z / OS系统定义。
  5. 使用pgaflip.jcl文件注释中的说明，定制JCL以匹配您的系统设置，并将其提交以进行批量执行。执行此步骤将pgaflip.asm文件组装并链接到您的Transaction Server for z / OS可访问的加载模块库中。 CICS启动过程中的DFHRPL DD语句。
  6. 登录到您的CICS Transaction Server for z / OS并输入以下事务：
    CEDA INSTALL GROUP（ORAPGA）

    此事务将安装与UNIX上的网关进行APPC通信的CICS连接和会话定义。它还安装了随网关提供的示例CICS程序和事务的定义。

您的CICS Transaction Server for z / OS配置现已完成。

配置IMS / TM

如果您的OLTP是IMS / TM，则执行以下步骤以配置IMS / TM和z / OS以与网关通信：

  1. 为APPC配置IMS系统。
  2. 为SNA APPC连接到UNIX配置MVS VTAM。除非您使用适用于LU6.2应用程序的IMS LU6.1适配器，否则网关必须至少使用一个独立的LU。在这种情况下，您必须为每个并发会话定义一个从属LU。例如，如果要支持10个并发会话，则必须定义10个相关LU。
  3. 检查IMS / TM使用的VTAM logmode表。表名由VTAM APPL定义中的MODETAB参数指定。对于APPC / IMS，请确保具有同步级支持和并行会话支持的APPC会话存在条目。$ORACLE_HOME/dg4appc/sna目录中的oralu62.asm和oraplu62.asm文件分别包含单会话和并行会话支持的样本模式条目。样本包含指示模式条目中所需值的注释。
  4. 使用文件传输工具，将以下文件从$ORACLE_HOME/dg4appc/demo/IMS目录传输到运行IMS / TM的z / OS系统：

    • pgaflip.asm是IMS FLIP事务的汇编源

    • pgaflip.jcl是用于汇编和链接IMS FLIP事务的JCL

    • imsgen.asm是IMS FLIP事务的IMS阶段1 gen定义

  5. 将imsgen.asm文件中的语句添加到IMS stage 1 gen并运行IMS stage 1和stage 2 gens。使用联机更改实用程序启用新的事务定义。
  6. 使用pgaflip.jcl文件中的注释，定制JCL以匹配您的系统设置并提交它以进行批量执行。这会将pgaflip.asm文件组装并链接到您的IMS / TM系统可访问的加载模块库，并为FLIP事务创建PSB和ACB。
  7. 执行系统上所需的任务，以使新事务可用于IMS / TM。根据您的系统设置，您可能必须重新启动IMS。

IMS / TM配置现已完成。

配置APPC / MVS

如果您的OLTP是APPC / MVS，请执行以下步骤以配置APPC / MVS以与网关通信：

  1. 配置MVS 用于SNA APPC连接到UNIX的VTAM。至少有一个独立的LU必须可供网关使用。
  2. 检查 APPC / MVS使用的VTAM日志模式表。 （表名由APPC / MVS的VTAM APPL定义中的MODETAB参数指定。）确保具有SYNCLEVEL和并行会话支持的APPC会话存在条目。oraplu62.asm文件中 $ORACLE_HOME/dg4appc/sna目录包含一个示例模式条目，包括指示模式条目中所需值的注释。
  3. 分配分区数据集（ 放置样本文件的z / OS系统上的PDS）。PDS应分配RECFM=FB ， RECFM=FB LRECL=80 ，以及适合BLKSIZE在设备类型的BLKSIZE 。一个目录块需要大约两个3390磁盘空间的磁道。Oracle建议命名此分区数据集（PDS） ORAPGA.APPCMVS.SAMPLIB 。
  4. 使用文件传输工具，从中传输以下文件 $ORACLE_HOME/dg4appc/demo/MVS目录到您在上一步中分配的z / OS PDS，使用以下指定的成员名称：

    • pgaflip.jcl ：JCL，用于添加APPC / MVS TP配置文件并定义事务的执行环境。将此文件作为成员PGAFLIPJ存储在z / OS PDS中。

    • pgaflip.rex ：APPC / MVS PGAFLIP事务的REXX源。将此文件作为成员PGAFLIP存储在z / OS PDS中。

  5. 使用pgaflip.jcl文件中的注释，定制JCL以匹配您的系统设置并提交它以进行批量执行。执行此步骤定义PGAFLIP事务的APPC / MVS TP配置文件，并将其存储在APPC / MVS配置文件数据集中。确保更改JCL中的数据集名称以匹配在步骤3中分配的z / OS PDS的名称。

APPC / MVS配置现已完成。
