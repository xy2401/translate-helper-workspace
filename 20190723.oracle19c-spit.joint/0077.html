目的

CLUSTER_DETAILS返回选择中每行的集群详细信息。返回值是一个XML字符串，用于描述最高概率集群或指定cluster_id的属性。

TOPN

如果为topN指定值，则该函数返回最影响群集分配的N属性（分数）。如果未指定topN ，则该函数将返回5个最有影响力的属性。

DESC，ASC或ABS

返回的属性按重量排序。属性的权重表示其对群集分配的正面或负面影响。正权重表示分配的可能性增加。负权重表示分配的可能性降低。

默认情况下， CLUSTER_DETAILS返回具有最高正权重（ DESC ）的属性。如果指定ASC ，则返回负权重最高的属性。如果指定ABS ，则返回权重最大的属性，无论是负数还是正数。结果按绝对值从最高到最低排序。输出中不包含权重为零的属性。

语法选择

CLUSTER_DETAILS可以通过以下两种方式之一对数据进行评分：它可以将挖掘模型对象应用于数据，也可以通过执行构建和应用一个或多个瞬态挖掘模型的分析子句来动态挖掘数据。选择语法或分析语法 ：

  • 语法 - 使用第一种语法使用预定义的模型对数据进行评分。提供聚类模型的名称。

  • 分析语法 - 使用分析语法在没有预定义模型的情况下对数据进行评分。包括INTO n ，其中n是要计算的集群数，以及mining_analytic_clause ，它指定是否应为多个模型构建分区数据。mining_analytic_clause支持query_partition_clause和order_by_clause 。（参见analytic_clause :: = 。）

在对分区模型进行评分时， CLUSTER_DETAILS函数的语法可以使用可选的GROUPING提示。请参阅GROUPING提示 。

mining_attribute_clause

mining_attribute_clause标识要用作评分预测变量的列属性。使用分析语法调用函数时，这些预测变量也用于构建瞬态模型。mining_attribute_clause行为与PREDICTION函数的描述相同。（参见mining_attribute_clause :: = 。）

C句法和语义检查

通过检查嵌入式SQL语句和PL / SQL块的语法和语义，Pro * C / C ++预编译器可帮助您快速查找和修复编码错误。本附录向您介绍如何使用SQLCHECK选项来控制检查的类型和范围。本附录包含以下主题：

C.1什么是句法和语义检查？

语法规则指定语言元素如何排序以形成有效语句。因此， 语法检查会验证关键字，对象名称，运算符，分隔符等是否正确放置在SQL语句中。例如，以下嵌入式SQL语句包含语法错误： 

执行SQL删除EMP，DEPTNO = 20; - 拼写错误的关键字WHERE EXEC SQL INSERT INTO EMP COMM，SAL VALUES（NULL，1500）; - 列名COMM和SAL周围缺少括号

语义规则指定如何进行有效的外部引用。因此， 语义检查验证对数据库对象和主机变量的引用是否有效以及主机变量数据类型是否正确。例如，以下嵌入式SQL语句包含语义错误： 

EXEC SQL DELETE FROM empp WHERE deptno = 20; - 不存在的表，EMPP EXEC SQL SELECT * FROM emp WHERE ename =：emp_name; - 未声明的主机变量emp_name

SQL语法和语义的规则在Oracle数据库SQL语言参考中定义。

C.2关于控制检查的类型和范围

您可以通过在命令行上指定SQLCHECK选项来控制检查的类型和范围。使用SQLCHECK，检查的类型可以是语法，语义或两者。检查的范围可包括以下内容：

  • 数据定义语句（例如CREATE和GRANT）

  • 数据操作语句（例如SELECT和INSERT）

  • PL / SQL块

但是，SQLCHECK无法检查动态SQL语句，因为它们在运行时才完全定义。

您可以为SQLCHECK指定以下值：

  • 语义学或全部

  • 句法

默认值为SYNTAX。

使用SQLCHECK不会影响对数据控件，游标控制和动态SQL语句执行的正常语法检查。

C.3关于指定SQLCHECK = SEMANTICS

当SQLCHECK = SEMANTICS时，预编译器检查语法和语义

  • 数据操作语句（INSERT，UPDATE等）

  • PL / SQL块

  • 主机变量数据类型

以及。的语法

  • 数据定义语句（CREATE，ALTER等）

但是，只对使用AT db_name子句的数据操作语句进行语法检查。

当SQLCHECK = SEMANTICS时，预编译器通过使用嵌入式DECLARE TABLE语句或通过连接到Oracle服务器并访问数据字典在命令行上指定USERID选项来获取语义检查所需的信息。如果在DECLARE TABLE语句中定义了数据操作语句或PL / SQL块中引用的每个表，则无需连接到Oracle服务器。

如果连接到Oracle服务器，但在数据字典中找不到某些所需信息，则必须使用DECLARE TABLE语句提供缺少的信息。如果数据字典定义发生冲突，则DECLARE TABLE定义将覆盖它。

如果在主机程序中嵌入PL / SQL块，则必须指定SQLCHECK = SEMANTICS。

在检查数据操作语句时，预编译器使用Oracle数据库SQL语言参考中的语法规则，但使用更严格的语义规则集。特别是，完成了更严格的数据类型检查。因此，当SQLCHECK = SEMANTICS时，为早期版本的Oracle编写的现有应用程序可能无法成功预编译。

在预编译新程序或要求更严格的数据类型检查时指定SQLCHECK = SEMANTICS。

C.3.1关于启用语义检查

当SQLCHECK = SEMANTICS时，预编译器可以通过以下任一方式获取语义检查所需的信息：

  • 连接到Oracle服务器并访问数据字典。

  • 使用嵌入式DECLARE TABLE和DECLARE TYPE语句。

C.3.1.1关于连接Oracle服务器

要进行语义检查，预编译器可以连接到Oracle数据库，该数据库维护主机程序中引用的表，类型和视图的定义。

连接到Oracle服务器后，预编译器访问数据字典以获取所需信息。数据字典存储表和列名称，表和列约束，列长度，列数据类型等。

如果在数据字典中找不到某些所需信息（例如，因为您的程序引用了尚未创建的表），则必须使用DECLARE TABLE语句提供缺少的信息（本附录稍后将对此进行讨论）。

要连接到Oracle服务器，请使用以下语法在命令行上指定USERID选项： 

USERID =用户名/密码

其中用户名和密码包含有效的Oracle用户标识。如果省略密码，系统会提示您输入密码。

如果您指定的是用户名和密码，而不是用户名和密码

USERID = /

预编译器尝试自动连接到Oracle服务器。仅当现有Oracle用户名与前缀为“CLUSTER $”的操作系统ID匹配时，或者在INIT.ORA文件中将参数OS_AUTHENT_PREFIX设置为任何值时，尝试才会成功。例如，如果您的操作系统ID是MBLAKE，则只有在CLUSTER $ MBLAKE是有效的Oracle用户名时，自动连接才会成功。

如果省略USERID选项，则预编译器必须从嵌入的DECLARE TABLE语句中获取所需的信息。

如果您尝试连接到Oracle服务器但不能（例如，因为数据库不可用），则会发出错误消息，并且您的程序未预编译。

C.3.1.2关于使用DECLARE TABLE

预编译器可以在不连接Oracle服务器的情况下进行语义检查。要进行检查，预编译器必须从嵌入的DECLARE TABLE语句中获取有关表和视图的信息。因此，必须在DECLARE TABLE语句中定义数据操作语句或PL / SQL块中引用的每个表。

DECLARE TABLE语句的语法是：

EXEC SQL DECLARE table_name TABLE（col_name col_datatype [DEFAULT expr] [NULL | NOT NULL]，...）;

其中expr是一个整数，可以在CREATE TABLE语句中用作默认列值。

对于用户定义的对象数据类型，大小是可选的，因为它未被使用。

如果使用DECLARE TABLE定义已存在的数据库表，则预编译器将使用您的定义，忽略数据字典中的定义。

C.3.1.3关于使用DECLARE TYPE

类似地，对于TYPE，有一个DECLARE TYPE语句，其语法是：

EXEC SQL DECLARE类型TYPE [AS OBJECT（col_name col_datatype，...）]| [AS VARRAY（size）OF element_type] | [AS TABLE OF object_type];

这允许在预编译时SQLCHECK = SEMANTICS时对用户定义的类型进行更好的类型检查。当SQLCHECK = SYNTAX时，DECLARE TYPE语句仅用作文档，并被注释掉并被忽略。

C.4关于指定SQLCHECK = SYNTAX

当SQLCHECK = SYNTAX时，预编译器检查Oracle数据库SQL语言参考中记录的SQL语句的语法：

  • 数据操作语句

  • 主机变量表达式

未进行语义检查，并且适用以下限制：

  • 未尝试与Oracle服务器建立连接，USERID变为无效选项。如果指定USERID，则会发出警告消息。

  • DECLARE TABLE和DECLARE TYPE语句被忽略;它们仅用作文档。

  • 不允许使用PL / SQL块。如果预编译器找到PL / SQL块，则会发出错误消息。

检查数据操作语句时，预编译器使用Oracle语法规则。这些规则向下兼容，因此在迁移预编译程序时请指定SQLCHECK = SYNTAX。

C.5关于输入SQLCHECK选项

您可以在线或在命令行上输入SQLCHECK选项。但是，您在内联中指定的检查级别不能高于您在命令行上指定的级别（或默认接受）。例如，如果在命令行上指定SQLCHECK = SYNTAX，则不能指定SQLCHECK = SEMANTICS inline。

9.2选择时间序列模型

了解如何选择时间序列模型。

时间序列数据可能包含可能影响预测准确性的模式。例如，在经济增长期间，销售可能会出现上升趋势。特定季节的销售额可能会增加（夏季的泳衣）。为了适应这样的系列，选择包含趋势，季节效果或两者的模型会很有用。

目的

目的

TOPN

如果为topN指定值，则该函数返回最影响群集分配的N属性（分数）。如果未指定topN ，则该函数将返回5个最有影响力的属性。

DESC，ASC或ABS

返回的属性按重量排序。属性的权重表示其对群集分配的正面或负面影响。正权重表示分配的可能性增加。负权重表示分配的可能性降低。

默认情况下， CLUSTER_DETAILS返回具有最高正权重（ DESC ）的属性。如果指定ASC ，则返回负权重最高的属性。如果指定ABS ，则返回权重最大的属性，无论是负数还是正数。结果按绝对值从最高到最低排序。输出中不包含权重为零的属性。

使用此子句可以修改区域映射的默认刷新方法和模式。该子句与ALTER MATERIALIZED ZONEMAP和CREATE MATERIALIZED ZONEMAP具有相同的语义。有关此条款的完整信息，请参阅zonemap_refresh_clause在文档中CREATE MATERIALIZED ZONEMAP 。

使用此子句可启用或禁用区域映射以进行修剪。该子句与ALTER MATERIALIZED ZONEMAP和CREATE MATERIALIZED ZONEMAP具有相同的语义。有关此子句的完整信息，请参阅ENABLE |关于CREATE MATERIALIZED ZONEMAP的文档中的DISABLE ZONEMAP

COMPILE

此子句允许您显式编译区域映射。在DDL操作更改其一个或多个基表的结构后，此操作将验证区域映射。通常没有必要发出此子句，因为Oracle数据库会在使用之前自动编译需要编译的区域映射。但是，如果要显式编译区域映射，则可以使用此子句执行此操作。

编译区域映射的结果取决于是否以影响区域映射的方式更改基表。例如，如果将列添加到基表，则编译后区域映射将有效，因为更改不会影响区域映射。但是，如果从基表中删除区域映射的定义子查询中包含的列，则编译后区域映射将无效。

您可以通过查询ALL_ ， DBA_和USER_ZONEMAPS数据字典视图的COMPILE_STATE列来确定区域映射是否需要编译。如果列的值为NEEDS_COMPILE ，则区域映射需要编译。

重建

此子句允许您显式重建区域映射。此操作将刷新区域映射中的数据。该子句在以下情况下很有用：

  • 您可以使用此子句刷新按需刷新区域映射的数据。有关更多信息，请参阅CREATE MATERIALIZED ZONEMAP文档中的ON DEMAND子句。

  • 无论区域映射的默认刷新模式如何，您都必须在区域映射的一个基表上的EXCHANGE PARTITION操作之后发出此子句。

  • 如果区域映射标记为不可用，则必须发出此子句以将其标记为可用。您可以通过查询ALL_ ， DBA_和USER_ZONEMAPS数据字典视图的UNUSABLE列来确定区域映射是否标记为不可用。

UNUSABLE

指定此子句以使区域映射不可用。后续查询将不使用区域映射，数据库将不再维护区域映射。您可以通过发布ALTER MATERIALIZED ZONEMAP使区域地图再次可用...REBUILD声明。

您可以使用数据库重放来捕获生产系统上的工作负载，并使用原始工作负载的准确时间，并发性和事务特征在测试系统上重放它。这使您可以在不影响生产系统的情况下测试系统更改的效果。

9.1工作负载捕获

使用数据库重放的第一步是捕获生产工作负载。捕获工作负载涉及将外部客户端发出的所有请求记录到Oracle数据库。

启用工作负载捕获后，将跟踪所有定向到Oracle数据库的外部客户端请求并将其存储在二进制文件中捕获文件 - 在文件系统上。您可以指定捕获文件的存储位置。工作负载捕获开始后，所有外部数据库调用都将写入捕获文件。捕获文件包含有关客户端请求的所有相关信息，例如SQL文本，绑定值和事务信息。不捕获后台活动和数据库调度程序作业。这些捕获文件与平台无关，可以传输到另一个系统。

9.3工作负载重播

捕获的工作负载经过预处理后，可以在测试系统上重放。 在工作负载重放阶段，Oracle数据库通过使用生产系统的相同时序，并发和事务依赖性重新创建所有捕获的外部客户端请求，在测试系统上执行工作负载捕获阶段记录的操作。

Database Replay使用一个名为的客户端程序重播客户端以重新创建工作负载捕获期间记录的所有外部客户端请求。根据捕获的工作负载，您可能需要一个或多个重播客户端才能正确重放工作负载。提供校准工具以帮助确定特定工作负载所需的重放客户端的数量。由于重放了整个工作负载（包括DML和SQL查询），因此重放系统中的数据应尽可能与捕获系统中的数据逻辑相似。这将最大限度地减少重放差异，并实现更可靠的重播分析。

9.4分析和报告

重播工作负载后，将为您提供深入报告，以便对工作负载捕获和重放进行详细分析。

工作负载捕获报告和工作负载重放报告提供有关工作负载捕获和重放的基本信息，例如重放期间遇到的错误以及DML或SQL查询返回的行中的数据偏差。还提供了工作负载捕获和工作负载重放之间的若干统计信息（例如数据库时间，平均活动会话和用户调用）的比较。

重放比较期间报告可用于执行一次工作负荷重放与其捕获或同一捕获的另一次重放的高级别比较。还提供了一个分歧摘要，分析是否发生任何数据分歧以及是否存在任何重大的性能变化。此外，自动数据库诊断监视器（ADDM）结果也包含在这些报告中。

对于高级分析，可以使用自动工作负载存储库（AWR）报告来详细比较工作负载捕获和工作负载重放之间的性能统计信息。这些报告中提供的信息非常详细，可以预期工作负载捕获和重放之间存在一些差异。此外，Workload Intelligence还会对工作负载捕获期间记录的数据进行操作，以创建描述工作负载的模型。此模型可用于标识作为工作负载一部分执行的模板中的重要模式。对于每个模式，您可以查看重要的统计信息，例如给定模式的执行次数以及模式在执行期间消耗的数据库时间。

SQL Performance Analyzer报告可用于将工作负载捕获中的SQL调优集与工作负载重放中的另一个SQL调优集或两个工作负载重放中的两个SQL调优集进行比较。将SQL调优集与数据库重放进行比较提供了比SQL性能分析器测试执行更多的信息，因为它考虑并显示了每个SQL语句的所有执行计划，而SQL性能分析器测试执行仅为每个SQL语句为每个SQL语句生成一个执行计划。此外，SQL语句在更真实的环境中执行，因为Database Replay捕获所有绑定值并更准确地再现PL / SQL包状态等动态会话状态。建议您首先运行SQL Performance Analyzer测试执行作为完整性测试，以确保在使用数据库重放执行负载和货币测试之前，SQL语句没有退化并且测试系统已正确设置。

除了使用重放差异信息来分析给定系统更改的重放特征外，还应使用应用程序级验证过程来评估系统更改。考虑开发一个脚本来评估重播的整体成功。例如，如果在工作负载捕获期间处理10,000个订单，则应验证在重放期间还处理了类似数量的订单。

重播分析完成后，您可以在工作负载捕获时将数据库还原到其原始状态，并重复工作负载重放以测试对系统的其他更改。

30.3.1分箱

分档也称为离散化，是一种降低连续和离散数据基数的技术。分箱将相关值组合在一起以减少不同值的数量。

分档可以显着提高资源利用率和模型构建响应时间，而不会显着降低模型质量。分箱可以通过加强属性之间的关系来提高模型质量。

监督分箱是智能分箱的一种形式，其中数据的重要特征用于确定箱边界。在有监督的分箱中，箱边界由单预测器决策树识别，该决策树考虑与目标的联合分布。有监督的分箱可用于数字和分类属性。

30.3.2标准化

归一化是减少数值数据范围的最常用技术。大多数规范化方法将单个变量的范围映射到另一个范围（通常为0,1）。

30.3.3异常值处理

如果值与列中的大多数其他值明显不同，则该值被视为异常值。异常值的存在可能对数据产生偏移影响，并且可能干扰转换的有效性，例如归一化或分级。

异常处理方法，如修剪或可以实现裁剪以最小化异常值的影响。

异常值表示有问题的数据，例如，由于仪器的异常情况导致的读数错误。但是，在某些情况下，特别是在商业领域，异常值是完全有效的。例如，在人口普查数据中，一些最富有的人的收入可能与一般人群有很大差异。不要将此信息视为异常值，因为它是数据的重要组成部分。您需要领域知识来确定异常值处理。

目的

PREDICTION_SET返回一组预测，其中包含选择中每行的概率或成本。返回值是与字段名对象的VARRAY PREDICTION_ID和PROBABILITY或COST 。预测标识符是Oracle NUMBER ;概率和成本字段是BINARY_DOUBLE 。

PREDICTION_SET可以执行分类或异常检测。对于分类，返回值是指预测的目标类。对于异常检测，返回值是指1 （对于典型行）或0 （对于异常行）的分类。

bestN和cutoff

您可以指定bestN和cutoff来限制函数返回的预测数。默认情况下， bestN和cutoff都为null，并返回所有预测。

  • bestN是最可能或最便宜的N预测。如果多个预测共享第N个概率或成本，则该函数选择其中一个。

  • cutoff是一个值阈值。仅返回概率大于或等于cutoff或成本小于或等于cutoff预测。要仅按cutoff过滤，请为bestN指定NULL 。如果函数使用带COST MODEL AUTO的cost_matrix_clause ，则忽略cutoff 。

您可以使用cutoff指定bestN ，以返回大于或等于cutoff的N最可能的预测。如果使用成本，请指定带有cutoff bestN ，以返回小于或等于截止值的N最低成本预测。

cost_matrix_clause

您可以指定cost_matrix_clause作为偏差因子，以最大限度地减少最有害的错误分类。 cost_matrix_clause行为与“ PREDICTION_COST ”中描述的相同。

PREDICTION_SET可以通过以下两种方式之一对数据进行评分：它可以将挖掘模型对象应用于数据，也可以通过执行构建和应用一个或多个瞬态挖掘模型的分析子句来动态挖掘数据。选择语法或分析语法 ：

    • 对于分类，请指定FOR expr ，其中expr是一个表达式，用于标识具有字符数据类型的目标列。

“新增功能”主题提供有关此版本中新增功能的信息。

以下功能是此版本中的新增功能：

  • Oracle LogMiner

    LogMiner包的CONTINUOUS_MINE功能已过时。它在Oracle Database 12 c第 2版（12.2）中已弃用。没有替换功能。

  • REST API

    在诸如Exalogic弹性云软件（EECS）和Oracle数据仓库云服务（DWCS）等托管云服务中，用户无权访问操作系统。他们只能访问数据库服务。加载数据并将客户从内部部署迁移到此类服务很困难。客户可以将其数据泵转储文件上载到对象库，并使用新的对象库REST API将它们加载到数据库中。 Impdp客户端CLI现在接受名为credential的新命令行参数。

    credential参数是凭证对象的名称，其中包含访问对象库所需的用户名和密码。当用户连接到数据库以运行impdp ，Datapump层将验证凭据是否存在以及用户是否有权读取凭据。如果有任何错误，则会将错误返回给impdp客户端。

  • Oracle Data Pump导出和导入

    • 只要文件设置为read-only ，DBA现在可以导入安装在两个不同数据库上的表空间文件。KEEP_READ_ONLY和NO_BITMAP_REBUILD被添加为TRANSPORTABLE参数值。

    • DBA现在可以更轻松地确定导出将花费多长时间，并发现闭包检查未报告的无法预料的问题。Data Pump Export命令行模式TTS_CLOSURE_CHECK参数用于指示作为Data Pump可传输表空间操作的一部分执行的闭包检查的程度。

    • 引入了一个新的转换参数OMIT_ENCRYPTION_CLAUSE ，该参数使Oracle Data Pump抑制与使用加密列的对象关联的任何加密子句。

    • 当有多个用户在给定数据库环境中执行数据泵作业时，更改的参数MAX_DATAPUMP_JOBS_PER_PDB可以为DBA提供对资源利用率的更多控制。默认值： 100 ，范围： 0到250或Auto - 50％的SESSIONS。

      引入了一个新参数MAX_DATAPUMP_PARALLEL_PER_JOB ，当有多个用户在给定数据库环境中执行数据泵作业时，它为数据库管理员提供了对资源利用率的更多控制。

  • Oracle外部表

    您现在可以使用ORACLE_LOADER访问驱动程序创建分区混合表。

48.1使用DBMS_DBCOMP

DBMS_DBCOMP.DBCOMP过程假定有一个主数据库和一个或多个物理备用数据库。在块比较之前，应至少安装数据库。

注意：

逻辑备用数据库，远程同步实例和级联备用数据库不能是DBMS_DBCOMP.DBCOMP过程的目标数据库。

在此过程运行时，您可以通过查询V$SESSION_LONGOPS视图来监视比较操作的进度。例如：

SQL> SELECT target_desc，sofar，totalwork FROM V $ SESSION_LONGOPS WHERE opname ='BlockCompare';

输出可能如下所示：

TARGET_DESC SO FAR TOTALWORK ----------------------------- --------- -------- - 比较块23914 340639丢失写入0 0

结果可以解释如下：

  • “ Compared Blocks值是扫描的块数（包括那些未使用比较回调函数的块）。该数字是所有从站的总和（即，每个从站负责主站点和备用站点之间的连接）。例如，如果有两个从站并且每个从站比较了1000个块，则SOFAR的值应为2000。

  • “ Lost Writes值是在主要和备用数据库中找到的丢失写入数。而且，它是所有奴隶的总和。对于Lost Writes ， SOFAR的值应始终等于TOTALWORK的值。

  • SOFAR的值应始终小于或等于TOTALWORK的值。

  • 当块比较从属进程完成时， SOFAR的值应该等于TOTALWORK的值。

48.3.1 DBCOMP程序

使用DBCOMP过程比较主数据库和一个或多个物理备用数据库上的块。

句法

DBMS_DBCOMP.DBCOMP（数据文件IN varchar2，输出文件IN varchar2，block_dump IN布尔值）;

参数

参数 描述
数据文件 这可以是数据文件名或数据文件号。指定'ALL'以比较所有数据文件。
输出文件 此值用作输出文件名称中的前缀。默认情况下，输出文件放在dbs /目录中，但输出文件参数可以包含相对目录路径或绝对目录路径。有关输出文件中包含的信息的说明，请参见使用说明。

使用说明

  • 输出文件包含以下列：

    • ID ：块类型。例如，类型02是撤消块类型，06是KTB管理的数据块类型。

    • TOTAL ：特定块类型中的块总数。

    • CORR ：损坏块的数量。

    • SKIPPED ：未经比较跳过（忽略）的块对数，例如直接加载块，RMAN优化块和闪回优化块

    • DIFFV ：主数据库和备用数据库之间具有不同版本号的块数。

    • SAMEV ：主数据库和备用数据库之间具有相同版本号的块对数。

    • SAMEV＆C ：具有相同版本和相同校验和的块对数。

    • ENCERR ：与加密相关的错误的块对数（通常是因为钱包未打开）。

    • LWLOC ：本地数据库上丢失的写入次数。

    • LWRMT ：远程数据库上丢失的写入次数。

    • DIFFPAIR ：具有相同版本但校验和不同的块对的数量，以及块比较回调函数得出的块对的数量不同。

9.1.1关于将对象存储为列或行

您可以将关系表中的对象存储为列对象，也可以将对象存储为对象表。那些在它们所在的关系数据库之外有意义的对象应该可以作为对象表中的行对象引用。否则，它们应作为列对象存储在关系表中。

有关表存储的介绍，请参见“ 如何在表中存储对象 ” 。

  • 关系表中的列对象存储

9.1.1.1关系表中的列对象存储

列对象存储类似于共同构成对象的等效标量列集的存储。

不同之处在于维护任何非集合列对象及其嵌入对象属性的原子空值的额外开销。这些值称为空指示符（有时是空图像），为每个列对象指定列对象是否为空以及每个嵌入对象属性是否为空。

9.1.1.2对象表中的行对象存储

行对象存储在对象表中。对象表是一种特殊的表，它保存对象并提供这些对象的属性的关系视图。对象表在逻辑上和物理上类似于关系表，其关键表的列类型对应于存储在对象表中的对象类型的顶级属性。关键区别在于对象表可以选择包含附加对象标识符（OID）列和索引。

9.1.2对象标识符（OID）的存储注意事项

对象表中的行对象有两种类型的对象标识符，可以存储和引用。

对象标识符（OID）允许相应的行对象从其他对象或从关系表引用。称为REF内置数据类型表示此类引用。REF使用对象标识符（OID）指向行对象。

您可以使用系统生成的OID或基于主键的OID。

“ 引用行对象 ”

9.1.2.1系统生成的对象标识符（OID）

系统生成的OID是对象表中行对象的默认值。

Oracle为每个行对象分配一个唯一的系统生成的OID，长度为16个字节，自动编入索引以进行有效的基于OID的查找。OID列相当于具有额外的16字节主键列。在分布式环境中，系统生成的唯一标识符使Oracle可以明确地识别对象。

对象标识符列是Oracle用于构造对行对象的引用的隐藏列。Oracle不提供对象标识符内部结构的访问权限。这种结构可以随时改变。应用程序仅涉及使用对象引用来获取和导航对象。

9.1.2.2基于主键的对象标识符（OID）

如果存在主键列，Oracle允许选择将行对象的主键值指定为其对象标识符。

您不必使用系统生成的OID，而是使用带有此子句的CREATE TABLE语句， OBJECT IDENTIFIER IS PRIMARY KEY 。这指定系统使用主键列作为表中对象的OID。这样，您可以使用现有列作为对象的OID，或使用应用程序生成的OID，这些OID小于Oracle生成的16字节全局唯一OID。

您可以以与关系表中的外键类似的方式对存储对这些行对象的引用的列强制执行引用完整性。

注意：

9.1.2.3系统生成与基于主键的OID

基于主键的标识符使得将数据加载到对象表中变得更快更容易。相反，系统生成的对象标识符需要使用一些用户指定的密钥重新映射，特别是当还存储对它们的引用时。

如果对对象表使用系统生成的OID，Oracle会在存储这些OID的列上维护索引。系统生成的OID需要为此索引提供额外的存储空间，并为每个行对象提供额外的16字节存储空间。

但是，如果每个主键值需要超过16个字节的存储空间并且您有大量REF ，则使用主键可能需要比系统生成的OID更多的空间，因为每个REF都是主键的大小。

目的

使用FLASHBACK DATABASE语句将数据库返回到过去时间或系统更改号（SCN）。此语句提供了执行不完整数据库恢复的快速替代方法。

在FLASHBACK DATABASE操作之后，为了具有对闪回数据库的写访问权，必须使用ALTER DATABASE OPEN RESETLOGS语句重新打开它。

先决条件

您必须具有SYSDBA ， SYSBACKUP或SYSDG系统特权。

如果您连接到多租户容器数据库（CDB）：

  • 要闪回CDB，必须连接到root，并且必须具有通常授予的SYSDBA ， SYSBACKUP或SYSDG系统特权。

  • 要闪回PDB，您必须连接到根目录，并且必须具有通常授予的SYSDBA ， SYSBACKUP或SYSDG系统特权，或者您必须连接到要闪回的PDB，并且必须具有SYSDBA ， SYSBACKUP ，或SYSDG系统特权，通常授予或在该PDB中本地授予。

必须为数据库准备一个快速恢复区。除非您将数据库闪回到保证还原点，否则必须使用ALTER DATABASE FLASHBACK ON语句将数据库置于FLASHBACK模式。必须装入数据库但不能打开。

此外：

  • 数据库必须以ARCHIVELOG模式运行。

  • 必须使用当前控制文件装入数据库，但不能打开。控制文件不能是备份或重新创建。从备份还原或重新创建数据库控制文件时，将丢弃所有现有的闪回日志信息。

  • 数据库必须不包含使用SQL语句ALTER TABLESPACE禁用闪回功能的联机表空间...FLASHBACK OFF 。

发出FLASHBACK DATABASE语句时，Oracle数据库首先验证是否所有必需的归档和联机重做日志都可用。如果它们可用，则它会将数据库中的所有当前在线数据文件还原到SCN或此语句中指定的时间。

  • 保留在数据库中的闪回数据量由DB_FLASHBACK_RETENTION_TARGET初始化参数和快速恢复区域的大小控制。您可以通过查询V$FLASHBACK_DATABASE_LOG视图来确定可以回退数据库的时间。

  • 如果数据库中仍有足够的数据来执行闪回，则可以使用标准恢复过程将数据库恢复到过去的某个时间点。

  • 如果一组数据文件的数据不足，则数据库将返回错误。在这种情况下，您可以使这些数据文件脱机并重新发出该语句以还原数据库的其余部分。然后，您可以尝试使用标准恢复过程恢复脱机数据文件。

闪回PDB的限制

  • 您无法闪回代理PDB。

  • 如果CDB处于共享撤消模式，则只能将PDB闪回到干净的PDB还原点。有关更多信息，请参阅CREATE RESTORE POINT的CLEAN子句。

数据库

如果要闪回非CDB或CDB，则可以选择指定要闪回的数据库的名称。如果省略database ，则Oracle数据库会闪回由初始化参数DB_NAME的值标识的数据库。

如果要闪回PDB并且当前容器是根，则使用database指定要闪回的PDB的名称。如果要闪回PDB并且当前容器是该PDB，则可以选择使用database来指定PDB名称。

到SCN条款

指定系统更改编号（SCN）：

  • TO SCN将数据库恢复到指定SCN的状态。

  • TO BEFORE SCN将数据库恢复到其指定SCN之前的系统更改号的状态。

您可以通过查询V$DATABASE视图的CURRENT_SCN列来确定当前SCN。这反过来允许您将SCN保存到假脱机文件，例如，在运行高风险批处理作业之前。

到TIMESTAMP条款

指定有效的日期时间表达式。

  • TO TIMESTAMP将数据库恢复到指定时间戳的状态。

  • TO BEFORE TIMESTAMP将数据库恢复到指定时间戳之前一秒的状态。

您可以将时间戳表示为与确定值（例如SYSDATE的偏移量，或者表示为绝对系统时间戳。

恢复点条款

指定此子句可将数据库闪回到指定的还原点。如果尚未启用闪回数据库，则这是您可以在此FLASHBACK DATABASE语句中指定的唯一子句。如果数据库不处于FLASHBACK模式，如上面“先决条件”部分所述，那么这是您可以为此语句指定的唯一子句。

指定TO BEFORE RESETLOGS闪数据库恢复到刚刚过去的RESETLOGS操作之前（ ALTER DATABASE OPEN RESETLOGS ）。

关于使用安全的外部密码存储区

如果要使用安全的外部密码存储，请按照以下信息中的说明配置Oracle钱包;否则，可以交互式提供密码，您可以跳过此部分。

在运行extusrupgrade脚本之前， extusrupgrade客户端Oracle钱包配置为安全外部密码存储，以便应用程序可以使用存储在钱包中的密码凭据连接到数据库。在客户端Oracle钱包中存储数据库密码凭证，无需在应用程序代码，批处理作业或脚本中嵌入密码。这样可以降低在脚本和应用程序代码中以明文形式显示密码的风险，并允许您更轻松地管理用户帐户的密码策略，而无需在密码更改时更改应用程序代码或脚本。

有关使用mkstore命令行实用程序将客户端配置为使用外部密码存储库的步骤，请参阅配置客户端以使用外部密码存储库。

使用SSL外部用户转换脚本

SSL外部用户转换脚本具有以下语法：

$ ORACLE_HOME / rdbms / bin / extusrupgrade --dbconnectstring 数据库连接字符串 --dbuser 数据库用户 [-dbalias 数据库用户密码别名 ] [-wallet_location 钱包位置 ] [-a] [-l username1，username2，... ][-f filename ] [-o] [-h]注意：-a升级所有合格用户-l由comma -f升级的用户升级列表由文件指定的用户升级列表。每行一个用户名-o将所有合格用户输出到标准输出。不与其他选项结合使用-h显示此帮助。

database connect string的格式应为hostname ： port_no ： sid ，其中hostname是运行数据库的主机的名称， port_no是侦听器端口号， sid是数据库实例的系统标识符。

如果使用mkstore命令行实用程序创建了安全外部密码存储库，则使用语法<alias, username, password>使用mkstore CreateCredential命令在钱包中创建dbuser凭据。例如， dbmanager1, system, password 。

接下来，启用Oracle钱包的自动登录。这允许运行extusrupgrade脚本的管理员用户访问extusrupgrade服务，而无需提供必要的凭据。

现在，您可以在命令行上使用数据库别名参数-dbalias <database user password alias>和钱包位置参数-wallet_location <wallet location>来运行extusrupgrade转换脚本。

以下示例假定钱包具有使用语法<alias, username, password>定义的dbuser凭证<alias, username, password>如dbmanager1, system, password 。例如，钱包位置显示为/oracle/product/19.1.0/db_1/wallets 。

使用-a选项转换所有经过SSL身份验证的外部用户。这是一个例子：

extusrupgrade --dbconnectstring mymachine：1521：ORA001 --dbuser system -dbialis dbmanager1 -wallet_location /oracle/product/19.1.0/db_1/wallets -a

使用-l选项指定要转换的逗号分隔的用户列表。例如：

extusrupgrade --dbconnectstring mymachine：1521：ORA001 --dbuser system -dbialis dbmanager1 -wallet_location /oracle/product/19.1.0/db_1/wallets -l user1，user2，user3

使用-f选项指定包含要转换的用户列表的文件。例如：

extusrupgrade --dbconnectstring mymachine：1521：ORA001 --dbuser system -dbialis dbmanager1 -wallet_location /oracle/product/19.1.0/db_1/wallets -f usernames.txt

指定文件中的每一行应该有一个用户名。这是一个示例usernames.txt文件：

用户＃1用户> 2个用户，3个用户4用户5

必须使用-f选项转换用户名中包含特殊字符（例如＃）的用户。

注意：

您可以在同一命令中组合-l和-f选项。该脚本结合了-l和-f选项的用户列表。如果使用-a选项以及-l option和-f选项，则忽略-a选项。

您可以使用-o选项将SSL身份验证的外部用户列表打印到标准输出设备。输出列出了可以使用extusrupgrade脚本转换的用户。-o选项不能与任何其他选项组合使用。 

extusrupgrade --dbconnectstring mymachine：1521：ORA001 --dbuser system -dbialis dbmanager1 -wallet_location /oracle/product/19.1.0/db_1/wallets -o

此示例输出可能是：

user1 user2 user3

小费：

您可以将命令输出重定向到文件以获取可以转换的用户列表。然后，您可以编辑该文件并将其与-f选项一起使用。

将全球用户转换为外部用户

Oracle数据库10g及更高版本允许经过SSL身份验证的外部用户和经过SSL身份验证的全局用户在数据库中共存。以前的版本限制所有SSL用户必须是全局用户或外部用户，具体取决于是否使用Oracle Internet Directory来验证用户。

如果您希望用户即使在Oracle Internet Directory不可用时也能够连接到数据库，则应将用户配置为外部用户。您可以使用SSL外部用户转换脚本将经过SSL身份验证的全局用户转换为经过SSL身份验证的外部用户。

如果使用mkstore命令行实用程序创建了安全外部密码存储区，并使用语法<alias, username, password>使用mkstore CreateCredential命令在钱包中创建了dbuser凭据。例如， dbmanager1, system, password 。例如，钱包位置显示为/oracle/product/19.1.0/db_1/wallets 。现在，您可以在运行extusrupgrade转换脚本时在命令行上使用数据库别名参数-dbalias <database user password alias>和钱包位置参数-wallet_location <wallet location> 。请注意，如果您已启用Oracle钱包的自动登录，则运行extusrupgrade脚本的管理员用户可以访问extusrupgrade服务，而无需提供必要的凭据。

例如：

extusrupgrade --dbconnectstring mymachine：1521：ORA001 --dbuser system -dbialis dbmanager1 -wallet_location /oracle/product/19.1.0/db_1/wallets -l user1，user2

上面的示例将两个全局用户转换为外部用户。

23.1 CDB中的DBMS_SCHEDULER调用

除了两个调度程序全局属性之外，大多数调度程序调用的工作方式与在非CDB中的调用方式相同。

JOB_QUEUE_PROCESSES初始化参数指定可以为执行DBMS_JOB作业和Oracle Scheduler（ DBMS_SCHEDULER ）作业创建的每个实例的最大作业从站数。值的范围是0到4000 （默认值）。

要限制CDB环境中的作业从站，可以在以下位置设置JOB_QUEUE_PROCESSES ：

  • CDB根

    将JOB_QUEUE_PROCESSES设置为Scheduler可以在整个数据库实例中同时使用的最大从属进程数。

    如果CDB根目录中的JOB_QUEUE_PROCESSES为0 ，则无论PDB级别的JOB_QUEUE_PROCESSES设置如何， DBMS_JOB和Oracle Scheduler作业都无法在根或任何PDB中运行。

  • PDB

    将JOB_QUEUE_PROCESSES设置为此PDB的最大并发作业数。实际数量取决于资源管理器分配的资源以及其他容器中的需求。当多个PDB请求作业时，Oracle Scheduler会尝试为所有PDB提供公平的进程份额。

    如果PDB中的JOB_QUEUE_PROCESSES为0 ，则无论CDB根目录中的JOB_QUEUE_PROCESSES设置如何， DBMS_JOB和Oracle Scheduler作业都无法在此PDB中运行。

您必须在PDB级别设置所有全局Oracle Scheduler属性。例如，如果使用DBMS_SCHEDULER.SET_ATTRIBUTE在根数据库中设置EMAIL_SENDER属性，则它将应用于在根中运行的作业，而不是在特定PDB中运行的作业。如果为PDB选择新的EMAIL_SENDER ，则必须在此PDB中设置全局属性。

23.2 CDB中的工作协调员和从属流程

与CDB相关的主要变化是工作协调员流程。

在非CDB中，协调器查看准备运行的所有作业，选择要运行的子集，并将它们分配给作业从站。它还会打开和关闭窗口，这会更改对数据库有效的资源计划。

这基本上是CDB内部发生的事情，除了以下情况：

  • 从所有PDB中选择作业

    协调器查看根数据库和所有子PDB，并根据作业优先级，作业计划开始时间以及运行作业的资源可用性选择作业。后一标准取决于工作的消费者群体和当前有效的资源计划。协调员不会试图对每个PDB公平。确保PDB中的作业不会缺乏的唯一方法是为其分配足够的资源。

  • Windows在PDB和根数据库级别中打开

    在非CDB中，在任何给定时间只能打开一个窗口。在CDB中，有两个级别的窗口。在PDB级别，窗口可用于设置资源计划，该资源计划在属于该PDB的消费者组之间分配资源。在根数据库级别，Windows可以将资源分配给不同的PDB。因此，在任何时候，根数据库中都可以打开一个窗口，每个PDB中都有一个窗口。

  • 作业从站切换到它所属的特定PDB

    作业从属基本上与非CDB中的相同，除了当从属执行作业时，它切换到作业所属的PDB然后执行它。其余的代码基本上没有变化。

23.3 DBMS_JOB和DBMS_SCHEDULER

仅当您向提交DBMS_JOB作业的模式授予CREATE JOB特权时，才能在PDB中使用DBMS_JOB 。

DBMS_JOB接口使用DBMS_SCHEDULER实现。因此，Oracle建议您从DBMS_JOB切换到DBMS_SCHEDULER 。

对于调度程序，协调器选择要从每个PDB运行的作业。此外，对于调度程序，从属进程在执行作业之前切换到PDB;否则，代码基本不变。

23.4关闭PDB的过程

如果使用immediate选项关闭PDB，则协调器将终止在PDB中运行的作业，并且必须先恢复作业，然后才能再次运行。

在Oracle RAC数据库中，协调器在大多数情况下可以在该PDB处于打开状态的另一个实例上恢复作业。因此，如果第一个实例上的协调器可以找到PDB仍处于打开状态的另一个实例，则会将作业移动到那里。在某些情况下，可能无法将作业移动到另一个实例。例如，如果相关的PDB未在其他任何位置打开，则无法移动作业。此外，当作业设置了INSTANCE_ID属性时，无法将作业移动到另一个实例。在这种情况下，作业无法运行，直到该实例上的PDB再次打开。

在非Oracle RAC案例中，不会出现移动作业的问题。只有在再次打开PDB后才能恢复已终止的作业。

23.5新的和更改的CDB视图

一些CDB视图特定于CDB，而其他视图具有CDB特定列。

  • V$和GV$视图有一个CON_ID列，用于标识其数据由CDB_*行表示的容器。在非CDB中， CON_ID NULL 。

  • CDB_*视图对应于所有Scheduler DBA_*视图。

    在PDB中，这些视图仅显示通过相应DBA_*视图可见的对象，但所有对象在根中都可见。CDB_*视图包含在给定DBA_*视图和列（ CON_ID ）中找到的所有列。在非CDB中，此列为NULL 。
