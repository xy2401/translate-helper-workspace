Oracle Database 12 c第2版（12.2.0.1）中的更改

以下是Oracle Database 2 Day DBA for Oracle Database 12 c Release 2（12.2.0.1）中的更改。

新功能

Oracle Enterprise Manager Database Express（EM Express）中有几个新功能。

EM Express具有以下新功能：

  • 资源管理器支持多租户容器数据库（CDB），可插拔数据库（PDB）和非CDB。您可以创建，编辑，删除，激活和停用资源计划，以帮助您在这些数据库中有效地管理资源。

    有关EM Express中的资源管理器功能的信息，请参阅“ 管理数据库资源 ” 。

  • SQL性能分析器（SPA）和SQL性能分析器快速检查（SPA快速检查）支持，使您可以评估数据库更改对SQL性能的影响

    有关EM Express中SPA和SPA快速检查功能的信息，请参阅“ 分析数据库更改对SQL性能的影响 ” 。

  • 在CDB的根目录中，您可以配置单个端口（称为全局端口），这使您可以使用EM Express使用CDB的HTTPS端口连接到CDB中的所有PDB

    有关配置全局端口的信息，请参阅“ 为PDB启动EM Express ” 。

  • CDB和PDB的其他Performance Hub支持

    请参阅“ 使用Performance Hub监控性能 ”中的表，以获取可用于不同数据库（CDB，PDB和非CDB）的选项卡列表。

  • 支持管理新类型的PDB

    您可以使用EM Express管理可刷新的PDB，代理PDB，应用程序根和应用程序PDB。请参阅“ 关于可以在EM Express中管理的PDB ”和“ 访问应用程序根目录的容器页面 ” 。

    EM Express支持代理PDB的管理。代理PDB提供上下文以执行SQL语句并在代理PDB中执行远程PDB中的操作。

    在代理PDB中执行的所有语句（ ALTER DATABASE ， CREATE PLUGGABLE DATABASE ， ALTER PLUGGABLE DATABASE和DROP PLUGGABLE DATABASE语句除外）都将在远程PDB中执行。

  • 将远程CDB中的PDB热克隆到另一个CDB

    有关将PDB从远程CDB克隆到另一个CDB的信息，请参阅“ 通过使用EM Express从远程CDB克隆PDB来创建PDB ” 。

  • 将PDB从远程CDB重定位到另一个CDB，停机时间最短

    有关将PDB从远程CDB重定位到另一个CDB的信息，请参阅“重定位远程PDB ” 。

  • 创建可刷新的PDB

    有关在通过从远程CDB克隆现有PDB创建新PDB时使用“ 刷新选项”字段创建可刷新PDB的信息，请参阅“ 通过使用EM Express从远程CDB克隆PDB来创建PDB ” 。

  • 根据需要刷新可刷新的PDB

    有关刷新可刷新PDB的信息，请参阅“ 按需刷新PDB”。

  • 用于管理Oracle Data Guard备用数据库的新EM Express功能

    您可以使用EM Express：

    • 查看Performance Hub中备用数据库的实时性能统计信息

      有关查看备用数据库性能统计信息的信息，请参见“ 关于在备用数据库上查看性能统计信息” 。

    • 使用SQL Tuning Advisor调整备用数据库的SQL查询

      有关调整备用数据库的SQL语句的信息，请参见“ 关于调整备用数据库上的SQL语句 ”和“ 调整备用数据库上的 SQL语句”。

新功能

除管理非CDB之外，您还可以使用Oracle Enterprise Manager Database Express（EM Express）来管理多租户容器数据库（CDB）和可插拔数据库（PDB）。

  • 有关启动EM Express以管理CDB的信息，请参阅“ 为CDB启动EM Express ” 。

EM Express为Oracle Database 12 c第1版（12.1.0.2）中引入的内存数据库功能提供支持：

  • “内存管理”页面上的“分配细分”和“分配历史记录”图表显示了“内存区域”信息。

  • CPU消耗量分为常规CPU使用率（CPU使用率与内存中操作无关）和内存中CPU使用率（内存中操作的CPU使用率）（如果适用）。这个CPU故障显示在几个地方：

    • 在“监视的SQL执行详细信息”页面上，显示CPU活动的任何位置。

    • 在“SQL详细信息”页面上的“活动％”列中。

    • 在Performance Hub的“活动”选项卡上，选择“按等级类消耗资源”时。

14.1自动收集诊断

本节介绍自动诊断收集概念。

如果Oracle Trace File Analyzer检测到任何问题，则会执行以下操作：

  • 运行必要的诊断并在出现问题时收集所有相关的日志数据

  • 在问题发生时修剪日志文件，以便Oracle跟踪文件分析器仅收集诊断所需的内容

  • 从集群中的所有节点收集并打包所有已修剪的诊断，整合单个节点上的所有内容

  • 在Oracle Trace File Analyzer存储库中存储诊断集合

  • 向您发送问题的电子邮件通知以及准备上载到Oracle Support的诊断收集的详细信息

Oracle Trace File Analyzer使用泛洪控制机制。重复错误不会使系统充满自动收集。

识别事件会触发集合的起点，五分钟后，Oracle Trace File Analyzer会开始收集诊断数据。五分钟后开始是一起捕捉任何其他相关事件。如果事件在五分钟后仍然发生，则诊断收集继续等待。Oracle跟踪文件分析器等待30秒，没有事件发生，最多再过5分钟。

如果事件在10分钟后仍然发生，则会发生诊断收集。一个新的收集点开始。

收集完成后，Oracle跟踪文件分析器会将包含集合位置的电子邮件通知发送给相关收件人。

如果您的环境可以与oracle.com建立连接，则可以使用Oracle Trace File Analyzer将集合上载到服务请求。

设定并行度

平行度 （DOP） 的 程度是与单个操作相关联的并行处理服务器的数量。

并行执行旨在有效地使用多个CPU。Oracle数据库并行执行框架使您能够明确选择特定的并行度，或依靠Oracle数据库自动控制它。

本节包含以下主题：

  • 手动指定并行度

  • 默认的并行度

  • 自动并行度

  • 确定自动DOP中的并行度

  • 控制自动并行度

  • 自适应并行

手动指定并行度

可以从Oracle数据库为表和索引请求特定的并行度（DOP）。

例如，您可以使用以下内容在表级别设置固定DOP：

ALTER TABLE销售PARALLEL 8; ALTER TABLE客户PARALLEL 4;

在此示例中，仅访问sales表的查询请求DOP为8，并且访问customers表的查询请求DOP为4。访问sales表和customers表的查询使用DOP 8进行处理，并可能分配16个并行执行服务器（因为生产者/消费者模型）。无论何时指定不同的DOP，Oracle数据库都使用更高的DOP。

您还可以使用语句级别或对象级别并行提示来请求特定DOP。

仅当PARALLEL_DEGREE_POLICY设置为MANUAL或LIMITED时，表或索引的PARALLEL子句中指定的DOP才会生效。

Oracle Database Resource Manager可以限制语句的实际运行时DOP。

默认的并行度

如果指定了PARALLEL子句但未列出任何并行度（DOP），则该对象将获得默认DOP。

例如，您可以使用以下SQL语句将表设置为默认DOP。

ALTER TABLE销售PARALLEL;

默认并行性使用公式根据系统配置确定DOP，如下所示：

  • 对于单个实例，DOP = PARALLEL_THREADS_PER_CPU x CPU_COUNT

  • 对于Oracle RAC配置，DOP = PARALLEL_THREADS_PER_CPU x sum(CPU_COUNT)

默认情况下， sum(CPU_COUNT)是群集中的CPU总数。但是，如果您使用Oracle RAC服务来限制并行操作可以执行的节点数，则sum(CPU_COUNT)是属于该服务的节点上的CPU总数。例如，在4节点Oracle RAC集群上，每个节点具有8个CPU核心且没有Oracle RAC服务，默认DOP将为2 x（8 + 8 + 8 + 8）= 64。

您还可以使用语句级别或对象级别并行提示来请求默认DOP。

仅当PARALLEL_DEGREE_POLICY设置为MANUAL时，表或索引的PARALLEL子句中指定的默认DOP才会生效。

默认DOP算法旨在使用最大资源，并假设如果可以使用更多资源，则操作会更快完成。默认DOP以单用户工作负载为目标，不建议在多用户环境中使用。

Oracle数据库资源管理器可以限制SQL语句的实际运行时DOP。

自动并行度

自动并行度（Auto DOP）使Oracle数据库能够自动决定语句是否应该并行执行以及它应该使用什么DOP。

以下是启用Auto DOP时并行语句处理的摘要。

  1. 发出SQL语句。

  2. 解析该语句，优化器确定执行计划。

  3. 将检查PARALLEL_MIN_TIME_THRESHOLD初始化参数指定的阈值限制。

    1. 如果预期执行时间小于阈值限制，则SQL语句将以串行方式运行。

    2. 如果预期执行时间大于阈值限制，则语句将根据优化程序计算的DOP并行运行，包括任何已定义资源限制的因子分解。

确定自动DOP中的并行度

通过自动并行度（DOP），优化器可根据该语句的资源要求自动确定语句的DOP。

优化程序使用所有扫描操作的成本（例如全表扫描或索引快速完全扫描）以及执行计划中所有CPU操作的成本来确定必要的DOP。

但是，优化器会限制实际的最大DOP，以确保并行执行服务器不会压倒系统。此限制由参数PARALLEL_DEGREE_LIMIT设置。此参数的默认值为CPU ，这意味着DOP受系统上CPU数量（ PARALLEL_THREADS_PER_CPU * sum(CPU_COUNT) ）的限制，也称为默认DOP。此默认DOP可确保单个用户操作不会使系统崩溃。通过调整此参数设置，您可以控制优化程序可以为SQL语句选择的最大DOP。如果使用Oracle数据库资源管理器限制DOP，优化程序可以进一步限制可以选择的最大DOP。

注意：

PARALLEL_DEGREE_LIMIT的值AUTO与CPU值具有相同的功能。

要计算SQL语句的操作成本，Auto DOP使用有关系统硬件特征的信息。硬件特性包括I / O校准统计数据，因此应收集这些统计数据。

如果未运行I / O校准以收集所需的统计数据，则使用默认校准值来计算操作成本和DOP。

可以使用PL / SQL DBMS_RESOURCE_MANAGER.CALIBRATE_IO过程收集I / O校准统计信息。如果物理硬件没有改变，I / O校准是一次性操作。

控制自动并行度

有几个初始化参数可以控制自动并行度（自动DOP）。这些初始化参数包括PARALLEL_DEGREE_POLICY ， PARALLEL_DEGREE_LIMIT ， PARALLEL_MIN_TIME_THRESHOLD和PARALLEL_MIN_DEGREE 。

初始化参数PARALLEL_DEGREE_POLICY控制是否启用Auto DOP，并行语句排队和内存中并行执行。此参数具有以下可能值：

  • 手册

    此设置禁用自动DOP，并行语句排队和内存中并行执行。它将并行执行的行为恢复为Oracle Database 11g第2版（11.2）之前的行为，这是默认设置。

    对于PARALLEL_DEGREE_POLICY的默认设置MANUAL ，系统仅在对象上显式设置DOP或在SQL语句中指定并行提示时才使用并行执行。使用的DOP正是指定的。不会发生并行语句排队和内存中并行执行。

  • 有限

    此设置为某些语句启用Auto DOP，但禁用并行语句排队和内存中并行执行。自动DOP仅应用于访问使用PARALLEL子句显式声明的表或索引的语句，而不指定显式DOP。指定了DOP的表和索引使用显式DOP设置。

    如果希望Oracle数据库仅针对触及特定对象子集的SQL语句子集自动确定DOP，则将PARALLEL_DEGREE_POLICY设置为LIMITED并设置并行属性，而不在该对象子集上指定显式DOP。

  • 汽车

    此设置为所有语句启用Auto DOP，还启用并行语句排队和内存中并行执行。

    如果希望Oracle数据库自动决定所有SQL语句的DOP，请将PARALLEL_DEGREE_POLICY设置为AUTO 。

  • 自适应

    此设置启用Auto DOP，并行语句排队和内存中并行执行，类似于AUTO值。此外，还启用了性能反馈。

PARALLEL_DEGREE_LIMIT初始化参数指定Auto DOP可在系统范围内使用的最大DOP。要对最大DOP进行更细粒度的控制，可以使用Oracle数据库资源管理器。

PARALLEL_MIN_TIME_THRESHOLD初始化参数指定要为Auto DOP考虑的语句的最小估计执行时间。首先，优化器计算SQL语句的串行执行计划。如果估计的执行时间大于PARALLEL_MIN_TIME_THRESHOLD的值，则该语句将成为Auto DOP的候选者。

PARALLEL_MIN_ DEGREE初始化参数控制由自动并行度计算的最小并行度。但是，如果PARALLEL_MIN_DEGREE的值大于CPU_COUNT的值，或者对象是Oracle拥有的，则PARALLEL_MIN_DEGREE没有影响，例如在字典表上创建的字典表或视图。

自适应并行

随着系统负载的增加，自适应多用户算法降低了并行度。

使用Oracle数据库自适应并行功能时，数据库在SQL执行时使用算法来确定并行操作是应该接收请求的DOP还是降低其DOP以确保系统不会过载。

在通过使用高DOP积极使用并行执行的系统中，当仅少数操作并行运行时，自适应算法调低DOP。虽然算法仍然确保最佳资源利用率，但用户可能会遇到不一致的响应时间。不建议在需要确定性响应时间的环境中仅使用自适应并行功能。通过数据库初始化参数PARALLEL_ADAPTIVE_MULTI_USER控制自适应PARALLEL_ADAPTIVE_MULTI_USER 。

9.1空间信息和数据挖掘应用

Oracle Data Mining允许从数据库中自动发现知识。其技术包括发现不同数据属性之间的隐藏关联，基于某些样本的数据分类，以及聚类以识别内在模式。可以实现空间数据以包含在数据挖掘应用程序中。

因此，Oracle数据挖掘可以使您发现具有位于特定区域（社区，城市或地区）的地址的销售潜在客户更有可能观看特定的电视节目或对特定的广告征集做出积极响应。（地址被地理编码为经度/纬度点并存储在Oracle Spatial and Graph几何对象中。）

在许多应用中，特定位置的数据受到邻域中数据的影响。例如，房屋的价值很大程度上取决于附近其他房屋的价值。这种现象称为空间相关 （或邻域影响），并在物化空间相关中进一步讨论。Oracle Spatial and Graph中的空间分析和挖掘功能使您可以通过以下几种方式使用数据项的位置属性来利用空间关联：将数据合并（离散化）到区域（例如将数据分类到北部，南部，东部和西部）区域），用于实现邻域的影响（例如每个商店半径2英里范围内的顾客数量），以及用于识别共同定位的数据项（例如视频租赁店和披萨店）。

要执行空间数据挖掘，可以使用专题图层为一组空间数据实现空间谓词和关系。每个层包含有关特定种类的空间数据（即具有特定“主题”）的数据，例如公园和娱乐区域，或人口统计收入数据。空间实现可以在应用数据挖掘技术之前作为预处理步骤执行，也可以作为空间挖掘的中间步骤执行， 如图9-1所示。

9.2区域模式检测的空间分组

空间分级 （空间离散化）将位置值离散化为与地理区域相关联的少数组。

  • 对经度/纬度坐标进行反向地理编码，以获取指定（对于美国位置）邮政编码，城市，州和国家/地区的地址

9.4主机托管挖掘

共置是在相同位置存在两个或更多个空间对象，或者彼此距离非常近。共置模式可以指示空间数据对象之间关于其非空间属性的有趣关联。

例如，数据挖掘应用程序可以发现，与视频商店共处的餐馆的特定比萨饼连锁店的特许经营店的销售额高于未与视频商店共存的餐馆。

支持两种类型的托管挖掘：

  • 数据挖掘表中的项目的共置。给定数据层，此方法可识别多个要素的共置。例如，捕食者和猎物物种可以在动物栖息地中共存，而高销售的比萨饼餐馆可以与高销售视频商店共存。您可以使用参考特征方法（使用一个特征作为参考，其他特征作为主题属性，并实现参考特征的所有邻居）或基于缓冲区的方法（实现指定的所有窗口内的所有项目）尺寸）。

  • 与主题层的主机托管。给定了几个数据层，此方法可识别跨层的共置。例如，考虑到湖泊层和植被层，湖泊可以与高植被区域共处。您实现数据，向数据挖掘表添加分类和数字空间关系，并应用Oracle数据挖掘关联规则机制。

Oracle数据库网关概述

网关技术由两部分组成：具有连接到非Oracle系统的通用技术的组件，所有非Oracle系统都通用，称为异构服务，以及特定于非Oracle的组件网关连接的系统。异构服务与Oracle数据库网关结合使用，可以从Oracle环境透明地访问非Oracle系统。

异构数据访问是一个影响很多公司的问题。许多公司运行几个不同的数据库系统这些系统中的每一个都存储数据，并具有一组针对它运行的应用程序。在一个数据库系统中整合这些数据通常很难 - 很大程度上是因为许多针对一个数据库运行的应用程序可能没有与另一个数据库运行的等效项。在迁移到一个统一数据库系统之前，各种异构数据库系统必须能够互操作。

Oracle数据库网关提供了从Oracle环境透明地访问驻留在非Oracle系统中的数据的功能。这种透明性消除了应用程序开发人员定制其应用程序以访问来自不同非Oracle系统的数据的需要，从而减少了开发工作并提高了应用程序的移动性。可以使用Oracle和非Oracle系统的一致Oracle接口开发应用程序。

关于异构服务技术

异构服务提供了连接到非Oracle系统的通用技术。作为数据库的集成组件，异构服务可以利用数据库的功能，例如强大的SQL解析和分布式优化功能。

异构服务扩展了Oracle SQL引擎，以识别远程非Oracle系统的SQL和过程功能以及获取必要数据字典信息所需的映射。异构服务提供两种类型的转换：将Oracle SQL转换为非Oracle系统的正确方言的能力，以及以本地格式显示非Oracle系统的元数据的数据字典转换。对于没有可用转换的情况，可以使用异构服务的传递功能向非Oracle系统发出本机SQL。

异构服务还维护Oracle与远程非Oracle系统之间的事务协调。

关于Oracle Database Gateway for ODBC

Oracle Database Gateway for ODBC适用于需要动态查询功能从Oracle数据库连接到非Oracle系统的低端数据集成解决方案。

可以使用Oracle Database Gateway for ODBC访问与本章中描述的ODBC标准兼容的任何数据源。

远程非Oracle系统的功能，SQL映射，数据类型转换和接口包含在网关中。网关与异构服务交互，以提供Oracle和非Oracle系统之间的透明连接。

用于ODBC体系结构的Oracle Database Gateway

网关使用ODBC驱动程序使用Oracle Database Gateway for ODBC访问非Oracle数据存储。您使用的驱动程序必须与网关位于同一台计算机上。非Oracle系统可以与Oracle数据库位于同一台计算机上，也可以位于不同的计算机上。

网关可以安装在运行非Oracle系统的计算机上，运行Oracle数据库的计算机上，也可以作为独立计算机安装在第三台计算机上。每种配置都有其优点和缺点。确定网关安装位置时的注意事项是网络流量，操作系统平台可用性，硬件资源和存储。

注意：

Oracle虚拟专用数据库和Oracle Label Security之间有什么区别？

Oracle数据库企业版免费提供Oracle虚拟专用数据库（VPD）。

Oracle Label Security是Oracle数据库企业版的附加安全选项。

Oracle VPD是一个术语，用于几个强大的安全功能，如细粒度访问控制（FGAC），应用程序上下文和全局应用程序上下文。VPD策略使用PL/SQL编写，可以分配给单个表或视图。访问受VPD保护的表或视图的信息请求将根据分配给表或视图的策略进行修改。

VPD策略可以像在工作时间强制访问一样简单。VPD策略可以通过将单个行中的属性值与应用程序上下文值进行比较来限制访问。全局应用程序上下文允许跨多个数据库会话访问应用程序上下文，从而减少或消除为每个用户会话创建单独的应用程序上下文的需要。

Oracle Label Security是行级安全性的开箱即用解决方案。无需编码或软件开发，允许管理员完全专注于策略。Oracle Label Security提供了一个界面，用于创建策略，指定实施选项，定义数据敏感性标签，建立用户标签授权以及保护单个表或方案。

数据敏感性标签提供了一种强大而灵活的限制数据访问的方法。例如，属于不同组织或公司的数据可以使用数据敏感性标签分开，并通过更改数据敏感性标签在公司之间选择性地共享。

根据安全策略的复杂性，Oracle虚拟专用数据库可能是实施安全策略的首选方法。Oracle Label Security最适合需要基于信息敏感性的访问控制决策的情况。

语义

模式

指定将在其中创建维度的架构。如果未指定schema ，则Oracle数据库会在您自己的架构中创建维度。

尺寸

指定维度的名称。该名称必须满足“ 数据库对象命名规则 ”中列出的要求。

level_clause

level_clause定义维度中的级别。级别定义维层次结构和属性。

水平

指定级别的名称。

level_table。 level_column

指定级别中的列。您最多可以指定32列。您在此子句中指定的表必须已存在。

空跳时跳过

指定此子句以指示如果指定的级别为NULL ，则跳过级别。此子句允许您通过跳过指定级别的备用路径来保留父子关系的层次结构链。请参阅hierarchy_clause 。

尺寸级别列的限制

维度级别列受以下限制：

  • 级别中的所有列必须来自同一个表。

  • 如果不同级别的列来自不同的表，则必须指定dimension_join_clause 。

  • 您指定的列集必须是此级别的唯一列。

  • 您指定的列不能在任何其他维度中指定。

  • 除非使用SKIP WHEN NULL指定级别，否则每个level_column必须为非null。非空列不需要具有NOT NULL约束。您为其指定SKIP WHEN NULL的列不能具有NOT NULL约束。

hierarchy_clause

hierarchy_clause定义维度中的级别的线性层次结构。每个层次结构在维度中的级别之间形成一系列父子关系。维度中的层次结构彼此独立。他们可能（但不一定）有共同的列。

维度中的每个级别最多应在此子句中指定一次，并且每个级别必须已在level_clause.命名level_clause.

等级制度

指定层次结构的名称。此名称在维度中必须是唯一的。

child_level

指定与父级别具有n ：1关系的级别的名称。level_columns的child_level不能为null，并且每个child_level值唯一地确定下一个命名的parent_level的值。

如果子level_table与父level_table level_table不同，则必须在dimension_join_clause指定它们之间的连接关系。

parent_level

指定级别的名称。

dimension_join_clause

dimension_join_clause允许您为其列包含在多个表中的维度指定内部等值连接关系。仅当层次结构中指定的列不在同一个表中时，才需要此子句。

child_key_column

指定一个或多个与父级别中的列连接兼容的列。

如果未指定每个child_column的模式和表，则会从hierarchy_clause的CHILD OF关系推断出模式和表。如果确实指定了child_key_column的模式和列，则模式和表必须与hierarchy_clause中parent_level子parent_level中的模式和列表相匹配。

parent_level

指定级别的名称。

加入维度的限制

加入尺寸受以下限制：

  • 您只能为同一层次结构中的给定一对级别指定一个dimension_join_clause 。

  • child_key_columns必须为非null，并且父键必须是唯一且非null。您无需定义约束来强制执行这些条件，但如果这些条件不正确，查询可能会返回不正确的结果。

  • 每个子键必须与parent_level表中的键连接。

  • 不允许自我加入。child_key_columns不能与parent_level在同一个表中。

  • 所有子键列必须来自同一个表。

  • 子键列的数量必须与parent_level的列数相匹配，并且列必须是可连接的。

  • 除非父级别包含多个列，否则无法指定多个子键列。

attribute_clause

attribute_clause允许您指定由层次结构级别唯一确定的列。level的列必须都与dependent_columns属于同一个表。该dependent_columns不必已在指定level_clause 。

例如，如果层次结构级别是city ， state和country ，那么city可能决定mayor ， state可能决定governor ， country可能决定president 。

extended_attribute_clause

此子句允许您为一个或多个级别到列关系指定属性名称。使用此子句创建的attribute_clause类型与使用attribute_clause创建的attribute_clause类型没有区别。唯一的区别是此子句允许您为与级别名称不同的属性指定名称。

关于受信任的存储程序单元

Oracle数据库存储过程，函数和包是以编译形式存储在数据库中的PL / SQL语句集。

函数和过程之间的唯一区别是函数返回值而过程不返回。可信存储的程序单元与Oracle数据库中的任何其他存储程序单元一样：基础逻辑是相同的。

包是一组过程和函数，以及它们使用的游标和变量，作为一个单元存储。包有两个部分，包规范和包体。包规范声明包所包含的公共过程，函数和变量的外部定义。包体包含过程和函数的实际文本，以及任何私有过程和变量。

受信任的存储程序单元是已被授予一个或多个Oracle Label Security权限的存储过程，函数或程序包。可信存储的程序单元通常用于让用户以受控方式执行特权操作，或者更新多个标签上的数据。这是允许用户访问超出授权范围的数据的最佳方法。

受信任的存储程序单元提供对特权使用的细粒度控制。虽然您可以为许多用户授予权限，但是应该非常谨慎地授予权限，因为它可能违反为您的应用程序建立的安全策略。您可以识别需要特权的任何应用程序操作，并将其实现为受信任的程序单元，而不是为用户分配权限。为这些存储的程序单元授予权限时，可以有效地限制用户所需的Oracle Label Security权限。该方法采用最小特权原则。

例如，如果具有标签CONFIDENTIAL的用户需要将数据插入SENSITIVE行，则可以将WRITEUP权限授予用户有权访问的受信任存储程序。以这种方式，用户可以通过可信存储的程序执行任务，同时保持在CONFIDENTIAL级别。

可信程序单元代表用户执行所有动作。因此，您可以有效地将安全策略封装到可以进行验证以确保其有效的模块中。

受信任的存储程序单元如何运行

受信任的存储程序单元使用自己的权限和调用者的标签运行。

以这种方式，可信存储程序单元可以对由用户标签约束的行集执行特许操作。

Oracle数据库系统和对象特权旨在捆绑到角色中。然后根据需要授予用户角色。相比之下，Oracle Label Security权限只能分配给用户或存储的程序单元。这些受信任的存储程序单元提供了比角色更易于管理的机制，以控制Oracle Label Security权限的使用。

执行受信任的存储程序单元

在Oracle Label Security下，程序调用上的所有标准Oracle数据库控件（关于对表和模式的访问）仍然有效。

Oracle Label Security通过控制对行的访问来补充这些安全机制。

执行受信任的存储程序单元时，有效的策略权限是调用用户权限和程序单元权限的并集。无论受信任的存储程序单元是否调用另一个受信任的程序单元或不受信任的程序单元，被调用的程序单元都以与原始程序单元相同的权限运行。

如果执行一系列不可信和可信存储的程序单元，则第一可信程序单元将从该点确定整个调用序列的特权。请考虑以下顺序：

  • 程序A（不信任）
  • 程序B使用WRITEUP
  • 程序C使用WRITEDOWN
  • 程序D（不信任）

这里，过程B，C和D都以WRITEUP权限运行，因为B是序列中的第一个可信过程。当序列结束时，与过程B相关的特权对后续过程不再有效。

注意：

Oracle Label Security捕获受信任程序单元中引发的未处理异常。这意味着可能无法向用户显示错误消息。因此，在授予权限之前，应始终彻底测试和调试任何程序单元。
