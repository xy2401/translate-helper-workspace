16.1什么是LOB？

使用LOB（大对象）列来存储大量数据（最大大小为4千兆字节），例如ASCII文本，国家字符文本，各种图形格式的文件和声波形式。

16.1.1内部LOB

内部LOB（BLOB，CLOB，NCLOB）存储在数据库表空间中，并具有数据库服务器的事务支持（Commit，Rollback等等。与它们一起工作）。

BLOB（二进制LOB）存储非结构化二进制（也称为“原始”）数据，例如视频剪辑。

CLOB（字符LOB）存储来自数据库字符集的大块字符数据。

NCLOB（国家字符LOB）存储来自国家字符集的大块字符数据。

16.1.2外部LOB

外部LOB是数据库表空间之外的操作系统文件，它们没有来自数据库服务器的事务支持。

BFILEs （二进制文件）将数据存储在外部二进制文件中。BFILE可以是GIF，JPEG，MPEG，MPEG2，文本或其他格式。

16.1.3 BFILE的安全性

DIRECTORY对象用于访问和使用BFILE。DIRECTORY是包含该文件的服务器文件系统中的实际物理目录的逻辑别名（存储在服务器中）。仅当授予DIRECTORY对象的访问权限时，才允许用户访问该文件。

  • DDL（数据定义语言）SQL语句CREATE，REPLACE，ALTER和DROP与DIRECTORY数据库对象一起使用。

  • DML（数据管理语言）SQL语句用于GRANT和REVOKE DIRECTORY对象的READ系统和对象权限。

CREATE DIRECTORY指令的示例是：

EXEC SQL创建或替换目录“Mydir”AS'/ usr / home / mydir';

仅当您使用DML（数据操作语言）语句（例如GRANT）授予其权限时，其他用户或角色才能读取该目录。例如，允许用户scott读取目录/usr/home/mydir BFILES：

EXEC SQL GRANT阅读目录“Mydir”TO scott;

一次会话中最多可同时打开10个BFILES。可以通过设置SESSION_MAX_OPEN_FILES参数来更改此默认值。

16.1.4 LOB与长期和长期的比较

LOB在许多方面与旧的LONG和LONG RAW数据类型不同。

  • LOB和LONG RAW的最大LOB大小为4千兆字节，而2千兆字节。

  • 您可以在LOB上使用随机和顺序访问方法;您只能在LONG和LONG RAW上使用顺序访问方法。

  • LOB（NCLOB除外）可以是您定义的对象类型的属性。

  • 表可以有多个LOB列，但只能有一个LONG或LONG RAW列。

Oracle建议将现有的LONG和LONG Raw属性迁移到LOB。甲骨文计划在未来的版本中终止对LONG和LONG RAW的支持。有关迁移的详细信息，请参见“ Oracle®数据库迁移指南”

16.1.5 LOB定位器

LOB定位符指向实际的LOB内容。检索LOB时返回定位器，而不是LOB的内容。LOB定位器无法保存在一个事务或会话中，并在以后的事务或会话中再次使用。

16.1.6临时LOB

您可以创建临时 LOB，就像本地变量一样，以帮助您使用数据库LOB。临时LOB不与任何表关联，只能由其创建者访问，具有定位器（访问方式），并在会话结束时删除。

临时BFILES不支持。临时LOB仅允许在INSERT，UPDATE或DELETE语句的WHERE子句中作为输入变量（IN值）。它们也被允许作为INSERT语句插入的值或UPDATE语句的SET子句中的值。临时LOB没有来自数据库服务器的事务支持，这意味着您不能对它们执行COMMITS或ROLLBACK。

临时LOB定位器可以跨越事务。当服务器异常终止时，以及从数据库SQL操作返回错误时，它们也会被删除。

16.1.7 LOB缓冲子系统

LBS（LOB缓冲子系统）是用户存储器的一个区域，用作客户端地址空间中一个或多个LOB的缓冲区。

缓冲具有这些优点，特别是对于客户端上的应用程序，它对LOB的特定区域执行许多小的读取和写入：

  • LBS减少了到服务器的往返次数，因为您通过对LOB进行多次读/写来填充缓冲区，然后在执行FLUSH指令时写入服务器。

  • 缓冲还减少了服务器上LOB更新的总数。这可以创建更好的LOB性能并节省磁盘空间。

Oracle提供了一个简单的缓冲子系统不是缓存。Oracle不保证缓冲区的内容始终与服务器LOB值同步。使用FLUSH语句实际在服务器LOB中写入更新。

通过其定位器执行LOB的缓冲读/写。启用缓冲的定位器提供LOB的一致读取版本，直到您通过该定位器执行写入。

在用于缓冲的WRITE之后，定位器成为更新的定位器，并提供对通过缓冲子系统看到的最新LOB版本的访问。所有进一步缓冲的写入LOB只能通过此更新的定位器完成。涉及缓冲LOB操作的事务无法跨用户会话进行迁移。

LBS由用户管理，用户负责通过使用FLUSH语句更新服务器LOB值。它是单用户和单线程。使用ROLLBACK和SAVEPOINT操作来保证服务器LOB中的正确性。Oracle不保证对缓冲LOB操作的事务支持。要确保缓冲LOB更新的事务语义，必须维护逻辑保存点以在发生错误时执行回滚。

16.2.3初始化LOB

有不同的技术用于初始化不同类型的LOB。每个都在本节中描述。

16.2.3.1内部LOB

要将BLOB初始化为空，请使用EMPTY_BLOB()函数，或使用ALLOCATE SQL语句。对于CLOB和NCLOB，请使用EMPTY_CLOB()函数。 有关EMPTY_BLOB()和EMPTY_CLOB()更多信息，请参见Oracle数据库SQL语言参考 。

这些函数仅在INSERT语句的VALUES子句中或在UPDATE语句中作为SET子句的源允许。

例如：

EXEC SQL INSERT INTO lob_table（a_blob，a_clob）VALUES（EMPTY_BLOB（），EMPTY_CLOB（））;

ALLOCATE语句分配LOB定位器并将其初始化为空。因此，以下代码等同于前面的示例：

#include <oci.h> ...OCIBlobLocator * blob; OCIClobLocator * clob; EXEC SQL ALLOCATE：blob; EXEC SQL ALLOCATE：clob; EXEC SQL INSERT INTO lob_table（a_blob，a_clob）VALUES（：blob，：clob）;
16.2.3.2外部LOB

使用LOB FILE SET语句以这种方式初始化BFILE和FILENAME的DIRECTORY别名：

#include <oci.h> ...char * alias =“lob_dir”; char * filename =“image.gif”; OCIBFileLocator * bfile; EXEC SQL ALLOCATE：bfile; EXEC SQL LOB FILE SET：bfile DIRECTORY =：alias，FILENAME =：filename; EXEC SQL INSERT INTO file_table（a_bfile）VALUES（：bfile）;

有关DIRECTORY对象命名约定和DIRECTORY对象特权的完整说明，请参阅Oracle数据库高级应用程序开发人员指南 。

或者，您可以在INSERT或UPDATE语句中使用BFILENAME（'directory'，'filename'）函数来初始化特定行的BFILE列或属性，并提供实际物理目录和文件名的名称：

EXEC SQL INSERT INTO file_table（a_bfile）VALUES（BFILENAME（'lob_dir'，'image.gif'））RETURNING a_bfile INTO：bfile;

注意：

BFILENAME（）不检查目录或文件名的权限，或物理目录是否确实存在。使用BFILE定位器的后续文件访问将执行这些检查，并在文件不可访问时返回错误。

16.2.3.3临时LOB

使用嵌入式SQL LOB CREATE TEMPORARY语句首次创建临时LOB时，它将初始化为空。EMPTY_BLOB（）和EMPTY_CLOB（）函数不能与临时LOB一起使用。

16.2.3.4释放LOB

FREE语句用于释放ALLOCATE语句保留的内存：

EXEC SQL FREE：a_blob;

16.3 LOB声明规则

以下是使用LOB语句的规则：

16.3.1对于所有LOB语句

当使用SQL LOB语句操作LOB时，这些一般限制和限制适用：

  • EXEC SQL LOB语句中不允许使用FOR子句，因为在这些语句中只能使用一个LOB定位符。

  • 不支持分布式LOB。虽然您可以在任何新的嵌入式SQL LOB语句中使用AT数据库子句，但是不能在同一SQL LOB语句中混合使用不同数据库连接创建的LOB定位符或ALLOCATEd。

  • 对于LOB READ和WRITE操作，OCI提供了一种回调机制，客户端可以指定一个回调函数，该函数将在每次读取或写入一段LOB值时执行。嵌入式SQL LOB方法不支持此功能。

  • OCI提供了一种机制，允许用户创建和指定自己的持续时间，可以在创建临时LOB时使用。还有一种机制可以指定缓冲区高速缓存用于临时LOB上的READ和WRITE操作。此接口不支持这些功能。

16.3.2对于LOB缓冲子系统

对于LBS，必须遵循以下规则：

  • 读取或写入访问中的错误在下次访问服务器时报告。因此，错误恢复必须由您（用户）编码。

  • 使用缓冲写入更新LOB时，请勿使用绕过LOB缓冲子系统的方法更新相同的LOB。

  • 启用缓冲的更新LOB定位器可以作为IN参数传递给PL / SQL过程，但不能作为IN OUT或OUT参数传递。返回错误，当尝试返回更新的定位符时，也会返回错误。

  • 不允许使用已更新的定位器的ASSIGN来缓冲到另一个定位器。

  • 您可以使用缓冲写入附加到LOB值，但起始偏移量必须是LOB结束后的一个字符。 LBS不允许APPEND语句在数据库服务器的LOB中产生零字节填充或空格。

  • 主机定位器绑定变量的字符集，数据库服务器CLOB必须相同。

  • 只有ASSIGN，READ和WRITE语句才能与启用缓冲的定位器一起使用。

  • 与启用缓冲的定位器一起使用时，以下语句会导致错误：APPEND，COPY，ERASE，DESCRIBE（仅限LENGTH）和TRIM。如果将这些语句与未启用缓冲的定位器一起使用，如果定位器指向的LOB正由另一个定位器以缓冲模式访问，则也会返回错误。

    注意：

    必须在之前由LOB缓冲子系统启用的LOB上使用FLUSH语句。

  • 提交交易。

  • 从当前事务迁移到另一个事务。

  • 禁用LOB上的缓冲区操作。

  • 从外部过程执行返回到PL / SQL例程。

    注意：

    如果从带有locator参数的PL / SQL块调用外部callout，则所有缓冲（包括ENABLE语句）都应在外部过程中完成。

请遵循以下食谱：

  • 呼叫外部标注。

  • 启用定位器进行缓冲。

  • 使用定位器读取或写入。

  • FLUSH LOB（LOB永远不会被隐式刷新）。

  • 禁用定位器进行缓冲。

  • 返回PL / SQL中的函数/ procedure /方法。

你必须明确地FLUSH一个LOB。

16.3.3对于主机变量

对LOB语句使用以下规则和注释：

  • src和dst可以引用内部或外部LOB定位器，但file仅指外部定位器。

  • 数字主机值（ amt, src_offset, dst_offset等）声明as 4字节无符号整数变量。值限制在0到4千兆字节之间。

  • NULL的概念是LOB定位器的一部分。LOB语句中不需要指示符变量。NULL不能与数值变量（如amt, src_offset ，从而导致错误。

  • 偏移值src_offset和dst_offset默认值为1。

16.4 LOB声明

这些陈述按字母顺序排列。在出现的所有语句中， 数据库指的是数据库连接

16.4.1附录

目的

此语句在另一个LOB的末尾附加一个LOB值。

句法

EXEC SQL [AT [：] database ] LOB APPEND： src TO： dst ;

主机变量

src（IN）

内部LOB定位器，唯一引用源LOB。

dst（IN OUT）

唯一引用目标LOB的内部LOB定位器。

使用说明

数据从源LOB复制到目标LOB的末尾，将目标LOB扩展到最大4千兆字节。如果LOB超出4千兆字节，则会发生错误。

源和目标LOB必须已存在，并且必须初始化目标LOB。

源LOB和目标LOB都必须具有相同的内部LOB类型。为任一类型的定位器启用LOB缓冲是错误的。

16.4.2分配

目的

将LOB或BFILE定位器分配给另一个。

句法

EXEC SQL [AT [：] database ] LOB ASSIGN： src to： dst ;

主机变量

src（IN）

从中复制LOB或BFILE定位器源。

dst（IN OUT）

LOB或BFILE定位器复制到。

使用说明

赋值后，两个定位器都引用相同的LOB值。目标LOB定位器必须是有效的初始化（ALLOCATEd）定位器。

对于内部LOB，仅当目标定位器存储在表中时，源定位器的LOB值才会复制到目标定位器的LOB值。对于Pro * C / C ++，发出包含目标定位符的对象的FLUSH将复制LOB值。

将BFILE定位器分配给内部LOB定位器时会返回错误，反之亦然。如果src和dst LOB的类型不同，也是一个错误。

如果源定位器用于已启用缓冲的内部LOB，并且源定位器已用于通过LOB缓冲子系统修改LOB值，并且缓冲区自WRITE以来未被FLUSHed，则源定位器不能是分配给目标定位器。这是因为每个LOB只有一个定位器可以通过LOB缓冲子系统修改LOB值。

16.4.3关闭（对于LOB）

目的

关闭一个打开的LOB或BFILE。

句法

EXEC SQL [AT [：] database ] LOB CLOSE： src;

主机变量

src（IN OUT）

要关闭的LOB或BFILE的定位器。

使用说明

使用不同的定位器或使用相同的定位器关闭相同的LOB两次是错误的。对于外部LOB，如果BFILE存在但尚未打开，则不会产生错误。

在关闭所有先前打开的LOB之前执行COMMIT是一个错误。在事务ROLLBACK时，所有仍处于打开状态的LOB将在不先关闭的情况下被丢弃。

16.4.4复制

目的

将全部或部分LOB值复制到第二个LOB中。

句法

EXEC SQL [AT [：] database ] LOB COPY： amt FROM： src [AT： src_offset ] TO： dst [AT： dst_offset ];

主机变量

amt（IN）

要复制的BLOB或CLOB和NCLOB的字符的最大字节数。

src（IN）

源LOB的定位器。

src_offset（IN）

这是CLOB或NCLOB的字符数，以及BLOB的字节数，从LOB开头的1开始。

dst（IN）

目标LOB的定位器。

dst_offset（IN）

目的地偏移量。与src_offset相同的规则。

使用说明

如果数据已经存在于目标的偏移量以及更远的位置，则会被源数据覆盖。如果目标的偏移量超出当前数据的末尾，则从当前数据的末尾到源的新写入数据的开头，将零字节填充符（BLOB）或空格（CLOB）写入目标LOB。

如果目标LOB超出目标LOB的当前长度，则扩展目标LOB以容纳新写入的数据。将此LOB扩展到4 GB以上是一个运行时错误。

尝试从未初始化的LOB进行复制也是错误的。

源LOB和目标LOB都必须属于同一类型。不能为任何一个定位器启用LOB缓冲。

amt变量表示要复制的最大数量。如果在复制指定的数量之前到达源LOB的末尾，则操作将以ORA-22993错误终止。

要使临时LOB永久化，必须使用COPY语句将临时LOB明确复制为永久LOB。

16.4.5创建临时

目的

创建一个临时LOB。

句法

EXEC SQL [AT [：] database ] LOB CREATE TEMPORARY： src;

主机变量

src（IN OUT）

在执行之前，当IN， src是一个LOB定位器之前是ALLOCATEd。

执行后，当OUT， src是一个LOB定位器，它将指向一个新的空临时LOB。

使用说明

成功执行后，定位器指向新创建的临时LOB，该临时LOB独立于表驻留在数据库服务器上。临时LOB为空，长度为零。

在会话结束时，释放所有临时LOB。对临时LOB的READ和WRITE永远不会通过缓冲区缓存。

16.4.6禁用缓冲

目的

禁用LOB定位器的LOB缓冲。

句法

EXEC SQL [AT [：] database ] LOB DISABLE BUFFERING： src;

主变量

src（IN OUT）

内部LOB定位器。

使用说明

该语句不支持BFILE。后续的读取或写入不会通过LBS完成。

注意：

使用FLUSH BUFFER命令永久更改，因为此语句不会隐式刷新LOB缓冲子系统中所做的更改。

16.4.7启用缓冲

目的

为LOB定位器启用LOB缓冲。

句法

EXEC SQL [AT [：] database ] LOB ENABLE BUFFERING： src;

主变量

src（IN OUT）

内部LOB定位器。

使用说明

该语句不支持BFILE。随后的读取和写入通过LBS完成。

16.4.8删除

目的

从给定的偏移量开始擦除给定数量的LOB数据。

句法

EXEC SQL [AT [：] database ] LOB ERASE： amt FROM： src [AT： src_offset ];

主机变量

amt（IN OUT）

输入是要擦除的字节数或字符数。返回的输出是擦除的实际数字。

src（IN OUT）

内部LOB定位器。

src_offset（IN）

从LOB开始的偏移量，从1开始。

使用说明

该语句不支持BFILE。

执行后， amt返回已擦除的实际字符数/字节数。如果在擦除所请求的字符/字节数之前达到LOB值的末尾，则实际数量和请求的数量将不同。如果LOB是空的， amt将表明，0个字符/字节都被擦除。

对于BLOB，擦除意味着零字节填充程序覆盖现有的LOB值。对于CLOB，擦除意味着空格会覆盖现有的LOB值。

16.4.10文件集

目的

在BFILE定位器中设置DIRECTORY别名和FILENAME。

句法

EXEC SQL [AT [：] database ] LOB FILE SET： file DIRECTORY =： alias ，FILENAME =： filename;

主机变量

文件（IN OUT）

BFILE定位器，其中设置了DIRECTORY别名和FILENAME。

别名（IN）

要设置的DIRECTORY别名。

文件名（IN）

要设置的FILENAME。

使用说明

给定的BFILE定位器在此语句中使用之前必须首先进行ALLOCATEd。

要加载的最大字节数。

文件（IN OUT）

源BFILE定位器。

从文件开头偏移的字节数，从1开始。

输出是写入的实际字符数或字节数。

用于写入LOB数据的缓冲区。

LOB定位器。

dst_offset（IN）

110.1 DBMS_NETWORK_ACL_ADMIN概述

DBMS_NETWORK_ACL_ADMIN包提供管理网络访问控制列表（ACL）的接口。

ACL用于控制用户通过PL / SQL网络实用程序包（包括UTL_TCP ， UTL_HTTP ， UTL_SMTP和UTL_INADDR）从数据库访问外部网络服务和资源。

Oracle建议您不要在新应用程序中使用已弃用的子程序。对不推荐使用的功能的支持仅用于向后兼容

注意：

Oracle Database 12 c中不推荐使用此过程。虽然出于向后兼容性的原因，程序仍在程序包中可用，但Oracle建议使用APPEND_HOST_ACE过程和APPEND_WALLET_ACE过程 。

句法

DBMS_NETWORK_ACL_ADMIN.ADD_PRIVILEGE（acl IN VARCHAR2，主要IN VARCHAR2，is_grant IN BOOLEAN，权限IN VARCHAR2，位置IN PLS_INTEGER DEFAULT NULL，start_date IN TIMESTAMP WITH TIMESTAMP DEFAULT NULL，end_date IN TIMESTAMP WITH TIMESTAMP DEFAULT NULL）;

参数

表110-4 ADD_PRIVILEGE函数参数

参数 描述

ACL的名称。相对路径将相对于“/ sys / acls”

主要

授予或拒绝特权的委托人（数据库用户或角色）。区分大小写。

is_grant

授予或拒绝特权。

特权

授予或拒绝网络权限

开始日期

访问控制条目（ACE）的开始日期。指定后，ACE仅在指定日期及之后有效。如果将权限添加到现有ACE，则将忽略start_date 。

结束日期

访问控制条目（ACE）的结束日期。指定后，ACE将在指定日期后过期。end_date必须大于或等于start_date 。如果将权限添加到现有ACE，则将忽略end_date 。

要删除权限，请使用DELETE_PRIVILEGE过程 。

BEGIN DBMS_NETWORK_ACL_ADMIN.ADD_PRIVILEGE（acl =>'us-example-com-permissions.xml'，principal =>'ST_USERS'，is_grant => TRUE，privilege =>'connect'）END;

110.7.6 ASSIGN_ACL程序

此过程将访问控制列表（ACL）分配给主机，域或IP子网，如果指定，则分配TCP端口范围。

注意：

Oracle Database 12 c中不推荐使用此过程。虽然出于向后兼容性的原因，程序仍在程序包中可用，但Oracle建议使用APPEND_HOST_ACE过程和APPEND_WALLET_ACE过程 。

句法

DBMS_NETWORK_ACL_ADMIN.ASSIGN_ACL（acl IN VARCHAR2，host IN VARCHAR2，lower_port IN PLS_INTEGER DEFAULT NULL，upper_port IN PLS_INTEGER DEFAULT NULL）;

参数

表110-9 ASSIGN_ACL功能参数

参数 描述

ACL

ACL的名称。相对路径将相对于“ /sys/acls ”。

主办

要为其分配ACL的主机。主机可以是主机的名称或IP地址。通配符可用于指定域或IP子网。主机或域名不区分大小写。

lower_port

如果不是NULL则TCP端口范围的下限

upper_port

TCP端口范围的上限。如果为NULL ，则假定为lower_port 。

使用说明

  • 只能为任何主机，域或IP子网分配一个ACL，如果指定，则为TCP端口范围。将新访问控制列表分配给网络目标时，Oracle数据库将取消分配分配给同一目标的先前访问控制列表。但是，Oracle数据库不会删除访问控制列表。您可以使用DROP_ACL过程删除访问控制列表。要删除访问控制列表分配，请使用UNASSIGN_ACL过程 。

  • 分配给域的ACL的优先级低于分配给其他ACL的子域，后者的优先级低于分配给各个主机的ACL。因此，对于给定的主机，例如“www.us.example.com”，以下域以递减优先级列出：

    - www.us.example.com

    - * .us.example.com

    - * .example.com

    - * .com

    - *

    同样，分配给子网的ACL的优先级低于分配较小子网的其他ACL，其优先级低于分配给各个IP地址的ACL。因此，对于给定的IP地址，例如“192.168.0.100”，以下优先级列出以下子网：

    - 192.168.0.100

    - 192.168.0。*

    - 192.168*

    - 192。*

    - *

  • 端口范围仅适用于ACL中的“连接”权限分配。 ACL中的“resolve”权限分配仅在将ACL分配给没有端口范围的主机时才有效。

    对于“连接”权限分配，分配给没有端口范围的主机的ACL优先级低于分配给具有端口范围的同一主机的其他ACL。

  • 指定TCP端口范围时， lower_port和upper_port不能为NULL ， upper_port必须大于或等于lower_port 。端口范围不得与已分配的同一主机的任何其他端口范围重叠。

  • 要删除分配，请使用UNASSIGN_ACL过程 。

例子

BEGIN DBMS_NETWORK_ACL_ADMIN.ASSIGN_ACL（acl =>'us-example-com-permissions.xml'，host =>'* .us.example.com'，lower_port => 80）;结束;

15.2.2.1.3管理服务的工具

Oracle建议使用SRVCTL实用程序来创建和修改服务。或者，您可以使用DBMS_SERVICE包。

SRVCTL

如果您的单实例数据库由Oracle Restart管理，或者您的Oracle RAC数据库由Oracle Clusterware管理，则使用服务器控制（SRVCTL）实用程序来创建，修改或删除该服务。

要使用SRVCTL实用程序为PDB创建服务，请使用add service命令并在-pdb参数中指定PDB。如果未指定-pdb ，则该服务与根关联。

要使用SRVCTL实用程序modify service的PDB属性，请使用modify service命令并在-pdb参数中指定PDB。要使用SRVCTL实用程序remove PDB的服务，请使用remove service命令。

您可以使用其他SRVCTL命令来管理服务，例如start service ， stop service和重relocate service命令，即使它们不包含-pdb参数也是-pdb 。

使用SRVCTL实用程序创建或修改服务时，不验证PDB名称。但是，尝试使用无效的PDB名称启动服务会导致错误。

DBMS_SERVICE

如果您的数据库不是由Oracle Restart或Oracle Clusterware管理的，则使用DBMS_SERVICE包来创建或删除数据库服务。

DBMS_SESSION存在于根级别和每个PDB中。它由SYS在每个级别拥有和执行。PDB管理员无法停止，重定位或测试另一个PDB拥有的服务的连接。

使用DBMS_SERVICE包创建服务时，服务的PDB属性将设置为当前容器。因此，要使用DBMS_SERVICE包创建PDB属性设置为特定PDB的DBMS_SERVICE ，请在PDB是当前容器时运行CREATE_SERVICE过程。如果在当前容器是根时使用CREATE_SERVICE过程创建服务，则该服务与根关联。

您无法使用DBMS_SERVICE包修改服务的PDB属性。但是，您可以在一个PDB中删除服务，并在不同的PDB中创建类似的服务。在这种情况下，新服务具有创建它的PDB的PDB属性。

您还可以使用其他DBMS_SERVICE子程序来管理服务，例如START_SERVICE和STOP_SERVICE过程。您可以使用DBMS_SERVICE. * _CONNECTION_TEST用于在计划维护期间检查数据库连接的运行状况的过程。使用DELETE_SERVICE过程删除服务。

  1. 在SQL * Plus中，确保当前容器是引用的PDB。
    请参阅“ 连接到PDB ” 。
  2. 运行ALTER PLUGGABLE DATABASE CONTAINERS HOST语句并指定新主机名，或包含RESET关键字以将主机名返回到其默认设置，即默认设置，即引用的PDB的主机名。
  3. 删除并重新创建引用引用的PDB的代理PDB，以重新建立每个代理PDB及其引用的PDB的通信。

logging_clause

注意：

从Oracle Database 12c第1版（12.1.0.2）开始提供此子句。

  • ENABLE FORCE NOLOGGING将PDB置于强制nologging模式，这样不会记录PDB中的任何更改。无法在架构对象级别覆盖强制nologging模式。

本节包含以下主题：

  • 刷新PDB
    您可以刷新作为可刷新克隆创建的PDB。
  • 切换可刷新的克隆PDB
    您可以切换源PDB及其可刷新克隆PDB的角色。

父主题： 在数据库级别修改PDB

15.4.4.1刷新PDB

您可以刷新作为可刷新克隆创建的PDB。

手动刷新PDB时，自上次刷新以来对源PDB所做的更改将传播到正在刷新的PDB。您可以手动刷新配置为自动刷新的PDB。

先决条件

要刷新PDB，必须将PDB创建为包含REFRESH MODE MANUAL或REFRESH MODE EVERY minutes子句的克隆。

  1. 在SQL * Plus中，确保当前容器是要刷新的PDB。
  2. 如果PDB未关闭，则关闭PDB。例如，发出以下SQL语句：
    ALTER PLOGGABLE DATABASE CLOSE IMMEDIATE;
  3. 发出以下SQL语句：
    ALTER PLOGGABLE DATABASE REFRESH;

15.4.4.2切换可刷新克隆PDB

您可以切换源PDB及其可刷新克隆PDB的角色。

以下语句执行切换：

ALTER PLUGGABLE DATABASE refresh_mode FROM clonepdb @ dblink SWITCHOVER ;

您不能为refresh_mode指定REFRESH MODE NONE 。FROM子句中指定的数据库链接必须指向克隆PDB所在的CDB的根目录。

切换完成后，源PDB成为可刷新的克隆PDB，只能在READ ONLY模式下打开。

先决条件

您必须满足以下先决条件：

  • 发出ALTER PLUGGABLE DATABASE ...时必须连接到源PDB ALTER PLUGGABLE DATABASE ...SWITCHOVER 。

  • 如果源PDB和克隆PDB位于单独的CDB中，则数据库链接中指定的用户必须在源PDB中具有相同的名称和密码并克隆PDB。

要切换源和克隆PDB的角色：

  1. 在SQL * Plus或SQL Developer中，登录到源PDB。

  2. 执行ALTER PLUGGABLE DATABASE refresh_mode FROM clonepdb@dblink SWITCHOVER语句。

    语句完成后，当前连接的PDB现在是可刷新的克隆PDB。

  3. （可选）刷新克隆PDB：

    ALTER PLOGGABLE DATABASE REFRESH;

15.5启动和关闭PDB

当前容器是PDB时，可以使用SQL * Plus STARTUP命令打开PDB，使用SQL * Plus SHUTDOWN命令关闭PDB。

  • 使用STARTUP命令启动PDB
    当前容器是PDB时，SQL * Plus STARTUP命令打开PDB。
  • 使用SHUTDOWN命令关闭PDB
    当前容器是PDB时，SQL * Plus SHUTDOWN命令关闭PDB。

父主题： 管理PDB

15.5.1使用STARTUP命令启动PDB

当前容器是PDB时，SQL * Plus STARTUP命令打开PDB。

使用STARTUP命令的以下选项打开PDB：

  • 力

    在以读/写模式重新打开PDB之前关闭它。指定此选项时，不允许其他选项。

  • 限制

    仅允许PDB中具有RESTRICTED SESSION系统特权的用户访问PDB。

    如果既未指定OPEN READ WRITE也未指定OPEN READ ONLY且指定了RESTRICT ，则当PDB所属的CDB是物理备用数据库时，PDB以只读模式打开。否则，PDB以读/写模式打开。

  • 打开open_pdb_options

    以读/写模式或只读模式打开PDB。指定OPEN READ WRITE或OPEN READ ONLY 。如果未指定RESTRICT ，则READ WRITE始终为默认值。

要在当前容器是PDB时发出STARTUP命令，必须满足以下先决条件：

  • 当前用户必须具有SYSDBA ， SYSOPER ， SYSBACKUP或SYSDG管理权限，并且必须在PDB中共同授予或本地授予该权限。用户必须在连接时分别使用AS SYSDBA ， AS SYSOPER ， AS SYSBACKUP或AS SYSDG来执行权限。

  • 排除使用FORCE选项，PDB必须处于挂载模式才能打开它。

  • 要将PDB置于挂载模式，PDB必须处于打开的只读或开放读/写模式。

要使用STARTUP命令修改PDB：

  1. 在SQL * Plus中，确保当前容器是PDB。

  2. 运行STARTUP命令。

15.5.2使用SHUTDOWN命令关闭PDB

当前容器是PDB时，SQL * Plus SHUTDOWN命令关闭PDB。

在PDB上成功发出SHUTDOWN命令后，它处于挂载模式。

可以使用以下SHUTDOWN模式：

  • 仅指定SHUTDOWN ，PDB将以正常模式关闭。

  • 指定SHUTDOWN IMMEDIATE ，PDB将以立即模式关闭。

  • 指定SHUTDOWN ABORT ，将强制关闭PDB。

    对于单实例CDB，指定SHUTDOWN ABORT时需要PDB介质恢复。对于Oracle Real Application Clusters（Oracle RAC）CDB，如果SHUTDOWN ABORT命令关闭最后一个打开的实例，则需要PDB介质恢复。

先决条件

要在当前容器是PDB时发出SHUTDOWN命令，必须满足以下先决条件：

  • 当前用户必须具有SYSDBA ， SYSOPER ， SYSBACKUP或SYSDG管理权限，并且必须在PDB中共同授予或本地授予该权限。用户必须在连接时分别使用AS SYSDBA ， AS SYSOPER ， AS SYSBACKUP或AS SYSDG来执行权限。

  • 要关闭PDB，PDB必须是开放的。

要使用SHUTDOWN命令修改PDB：

  1. 在SQL * Plus中，确保当前容器是PDB。

  2. 运行SHUTDOWN命令。

注意：

  • 当前容器是PDB时， SHUTDOWN命令仅关闭PDB，而不关闭CDB实例。

  • PDB没有SHUTDOWN命令，相当于非CDB的SHUTDOWN TRANSACTIONAL 。
