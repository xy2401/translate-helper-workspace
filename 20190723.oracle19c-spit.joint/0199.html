12.2.1.1优化器统计信息首选项的用途

首选项使您可以在某些对象需要的设置与默认设置不同时自动维护优化程序统计信息。

首选项使您可以更精细地控制Oracle数据库收集统计信息的方式。您可以在以下级别设置优化程序统计信息首选项：

  • 表

  • 架构

  • 数据库（所有表格）

  • 全局（没有首选项的表和将来创建的任何表）

用于设置首选项的DBMS_STATS过程具有SET_*_PREFS形式的名称。

父主题： 关于优化程序统计信息首选项

12.2.1.2统计首选项示例

使用SET_ * _PREFS过程的pname参数设置首选项。

您可以设置的首选项包括但不限于以下内容：

  • ESTIMATE_PERCENT

    此首选项确定要估计的行的百分比。

  • 同时

    此首选项确定数据库是否在多个对象上同时收集统计信息，或者一次收集一个对象。

  • STALE_PERCENT

    此首选项确定表中的行百分比，在数据库认为统计信息过时且需要重新收集之前必须更改这些行。

  • AUTO_STAT_EXTENSIONS

    当设置为非默认值ON ，此首选项使SQL计划指令能够根据工作负载中谓词中列的使用情况触发列组统计信息的创建。

  • 增加的

    此首选项确定数据库是否维护分区表的全局统计信息，而不执行全表扫描。可能的值为TRUE和FALSE 。

    例如， INCREMENTAL的默认设置为FALSE 。在更新最后几个分区时，可以为范围分区表将INCREMENTAL设置为TRUE 。此外，在非分区表上执行分区交换操作时，Oracle建议您将INCREMENTAL设置为TRUE ，将INCREMENTAL_LEVEL为TABLE 。通过这些设置， DBMS_STATS在此表上收集表级别概要。

  • INCREMENTAL_LEVEL

    当INCREMENTAL首选项设置为TRUE时，此首选项控制要收集的概要。它需要两个值： TABLE或PARTITION 。

  • APPROXIMATE_NDV_ALGORITHM

    此首选项控制在使用增量统计信息计算分区表的不同值的数量时使用的算法。

  • ROOT_TRIGGER_PDB

    此首选项控制是接受还是拒绝从CDB中的应用程序根触发的统计信息收集。

    默认情况下，聚集在应用程序根目录元数据联表统计信息时，如果统计应用PDB是陈旧的，数据库不会触发统计数据收集应用PDB。设置为TRUE ， ROOT_TRIGGER_PDB将触发应用程序PDB上的统计信息收集，然后在应用程序根目录中派生全局统计信息。

12.2.1.3 DBMS_STATS设置统计首选项的过程

DBMS_STATS.SET_*_PREFS过程更改DBMS_STATS.GATHER_*_STATS过程使用的参数的缺省值。要查询当前首选项，请使用DBMS_STATS.GET_PREFS函数。

设置统计信息首选项时，优先顺序为：

  1. 表首选项（为特定表设置，模式中的所有表或数据库中的所有表）

  2. 全球偏好

  3. 默认偏好

下表总结了相关的DBMS_STATS过程。

表12-1设置优化程序统计信息首选项的DBMS_STATS过程

程序 范围

SET_TABLE_PREFS

仅限指定表。

SET_SCHEMA_PREFS

指定架构中的所有现有表。

此过程为指定模式中的每个表调用SET_TABLE_PREFS 。调用SET_SCHEMA_PREFS不会影响运行后创建的任何新表。新表对所有参数使用GLOBAL_PREF值。

SET_DATABASE_PREFS

数据库中的所有用户定义的模式。通过将ADD_SYS参数设置为true可以包括系统拥有的模式，例如SYS和SYSTEM 。

此过程为指定模式中的每个表调用SET_TABLE_PREFS 。调用SET_DATABASE_PREFS不会影响运行后创建的任何新对象。新对象对所有参数使用GLOBAL_PREF值。

SET_GLOBAL_PREFS

任何没有现有表首选项的表。

除非设置了表首选项或在DBMS_STATS.GATHER_*_STATS语句中显式设置了参数，否则所有参数都默认为全局设置。SET_GLOBAL_PREFS所做的SET_GLOBAL_PREFS会影响运行后创建的任何新对象。新对象对所有参数使用SET_GLOBAL_PREFS值。

随着SET_GLOBAL_PREFS ，你可以设置参数的默认值AUTOSTATS_TARGET 。此附加参数控制在夜间维护窗口中运行的自动统计信息收集作业对哪些对象产生影响。可能值AUTOSTATS_TARGET是ALL ， ORACLE ，和AUTO （默认值）。

您只能在全局级别设置CONCURRENT首选项。您无法使用SET_GLOBAL_PREFS设置首选项INCREMENTAL_LEVEL 。

12.2.1.4统计首选项覆盖

preference_overrides_parameter统计信息首选项确定在收集优化程序统计信息时是否使用统计信息首选项覆盖参数的输入值。通过这种方式，您可以控制数据库何时遵循传递给统计信息收集过程的参数值。

当preference_overrides_parameter设置为FALSE （默认值）时，将遵循统计信息收集过程的输入值。设置为TRUE ，将忽略输入值。

使用DBMS_STATS的SET_TABLE_PREFS ， SET_SCHEMA_PREFS或SET_GLOBAL_PREFS过程设置preference_overrides_parameter选项。无论是否设置了preference_overrides_parameter ，数据库都使用相同的优先顺序来设置统计信息：

  1. 表首选项（为特定表设置，模式中的所有表或数据库中的所有表）

  2. 全球偏好

  3. 默认偏好

示例12-1在表级别覆盖统计信息首选项

在此示例中，遗留脚本显式设置estimate_percent而不是使用推荐的AUTO_SAMPLE_SIZE 。您的目标是阻止用户使用这些脚本在sh.costs表上设置首选项。

表12-2在表级别覆盖统计信息首选项

行动 描述 
SQL> SELECT DBMS_STATS.GET_PREFS（'estimate_percent'，'sh'，'costs'）AS“STAT_PREFS”FROM DUAL; STAT_PREFS ---------- DBMS_STATS.AUTO_SAMPLE_SIZE

对于无要求estimate_percent设置为sh.costs或在全球范围内，所以优先默认为AUTO_SAMPLE_SIZE 。

 
SQL> EXEC DBMS_STATS.SET_TABLE_PREFS（'sh'，'cost'，'preference_overrides_parameter'，'true'）; PL / SQL过程成功完成。

默认情况下，Oracle数据库接受传递给GATHER_*_STATS过程的首选项。要覆盖这些参数，可以使用SET_TABLE_PREFS仅为costs表将preference_overrides_parameter首选项设置为true 。 

SQL> EXEC DBMS_STATS.GATHER_TABLE_STATS（'sh'，'cost'，estimate_percent => 100）; PL / SQL过程成功完成。

在收集sh.costs统计信息时，您尝试将estimate_percent设置为100 。但是，由于此表的preference_overrides_parameter为true ，因此Oracle数据库不支持estimate_percent=>100设置。相反，数据库使用AUTO_SAMPLE_SIZE收集统计信息，这是默认值。

示例12-2覆盖全局级别的统计信息首选项

在这个例子中，您将estimate_percent至5在全球范围内，这意味着这种偏好适用于所有的表中没有一个表的首选项集数据库。然后，在sh.sales表上设置覆盖，该表没有表级首选项集，以防止用户覆盖其脚本中的全局设置。

表12-3覆盖全局级别的统计信息首选项

行动 描述 
SQL> SELECT DBMS_STATS.GET_PREFS（'estimate_percent'，'sh'，'sales'）AS“STAT_PREFS”FROM DUAL; STAT_PREFS ---------- DBMS_STATS.AUTO_SAMPLE_SIZE

对于无要求estimate_percent设置为sh.sales或在全球范围内，所以优先默认为AUTO_SAMPLE_SIZE 。

 
SQL> EXEC DBMS_STATS.SET_GLOBAL_PREFS（'estimate_percent'，'5'）; PL / SQL过程成功完成。

对于没有设置表首选项的数据库中的每个表，使用SET_GLOBAL_PREFS过程将estimate_percent首选项设置为5 。 

SQL> SELECT DBMS_STATS.GET_PREFS（'estimate_percent'，'sh'，'sales'）AS“STAT_PREFS”FROM DUAL; STAT_PREFS ---------- 5

由于sh.sales没有首选项集，因此全局设置适用于此表。现在，对sh.sales首选项的查询显示estimate_percent设置为5 ，这是全局设置。 

SQL> EXEC DBMS_STATS.SET_TABLE_PREFS（'sh'，'sales'，'preference_overrides_parameter'，'true'）; PL / SQL过程成功完成。

您可以使用SET_TABLE_PREFS仅为sh.sales表将preference_overrides_parameter选项设置为true 。 

SQL> EXEC DBMS_STATS.GATHER_TABLE_STATS（'sh'，'sales'，estimate_percent => 10）; PL / SQL过程成功完成。

在收集sh.sales统计信息时，您尝试将estimate_percent设置为10 。但是，由于preference_overrides_parameter对于sales表为true ，并且由于定义了全局首选项，因此Oracle Database实际上使用全局设置5收集统计信息。

12.3.2手动设置动态统计级别

确定对所有SQL语句有益的数据库级设置可能很困难。

假设

本教程假设以下内容：

  • 您希望对以下查询进行正确的选择性估计，该查询在两个相关列上具有WHERE子句谓词：

    SELECT * FROM sh.customers WHERE cust_city ='洛杉矶'和cust_state_province ='CA';

  • 前面的查询使用串行处理。

  • sh.customers表包含符合查询条件的932行。

  • 您已经收集了sh.customers表的统计信息。

  • 您在cust_city和cust_state_province列上创建了一个索引。

  • OPTIMIZER_DYNAMIC_SAMPLING初始化参数设置为默认级别2 。

    来自sh.customers的解析计划WHERE cust_city ='洛杉矶'和cust_state_province ='CA';

12.4管理SQL计划指令

SQL计划指令是优化程序可用于生成更优计划的附加信息和指令。

本指南介绍如何使用DBMS_NETWORK_ACL_ADMIN PL / SQL包为数据库用户和角色配置访问控制。

此功能增强了网络连接的安全性，因为它限制了数据库用户可以使用PL / SQL网络实用程序包UTL_TCP ， UTL_SMTP ， UTL_MAIL ， UTL_HTTP和UTL_INADDR连接的外部网络主机。 DBMS_LDAP和DBMS_DEBUG_JDWP PL / SQL包;和HttpUriType类型。否则，获得数据库访问权限的入侵者可能会恶意攻击网络，因为默认情况下，PL / SQL实用程序包是使用授予PUBLIC用户的EXECUTE特权创建的。这些PL / SQL网络实用程序包以及DBMS_NETWORK_ACL_ADMIN和DBMS_NETWORK_ACL_UTILITY程序包支持IP版本4（IPv4）和IP版本6（IPv6）地址。本指南介绍了如何管理两个版本的访问控制。有关IPv4和IPv6表示法如何与Oracle数据库配合使用的详细信息，请参阅“ Oracle数据库网络服务管理员指南” 。

配置外部网络服务访问控制的语法

您可以使用DBMS_NETWORK_ACL_ADMIN.APPEND_HOST_ACE过程向用户授予访问控制权限。

此过程将具有指定权限的访问控制条目（ACE）附加到给定主机的ACL，如果ACL尚不存在，则创建ACL。结果配置驻留在SYS架构中，而不是创建它的用户的架构。

语法如下：

BEGIN DBMS_NETWORK_ACL_ADMIN.APPEND_HOST_ACE（host =>' host_name '，lower_port => null | port_number ，upper_port => null | port_number ，ace => ace_definition ）;结束;

在本规范中：

  • host ：输入host的名称。它可以是主机的主机名或IP地址。您可以使用通配符指定域或IP子网。（有关在域名中使用通配符的优先顺序，请参阅多个访问控制列表分配中的主机优先顺序。）主机或域名不区分大小写。示例如下：

    host =>'www.example.com'，host =>'* example.com'，

  • lower_port :(可选）对于TCP连接，请输入端口范围的下边界。仅将此设置用于connect权限。省略它以获得resolve权限。默认值为null ，表示没有端口限制（即ACL适用于所有端口）。端口号范围介于1和65535之间。

    例如：

    lower_port => 80，

  • upper_port :(可选）对于TCP连接，请输入端口范围的上边界。仅将此设置用于connect权限。省略它以获得resolve权限。默认值为null ，表示没有端口限制（即ACL适用于所有端口）。端口号范围介于1和65535之间

    例如：

    upper_port => 3999）;

    如果输入lower_port的值并将upper_port为null （或只是省略它），则Oracle数据库假定upper_port设置与lower_port相同。例如，如果将lower_port设置为80并省略upper_port ，则upper_port设置假定为80 。

    在访问控制列表分配中指定端口范围时，访问控制列表中的resolve特权无效。

  • ace ：使用XS$ACE_TYPE常量定义ACE，格式如下：

    ace => xs $ ace_type（privilege_list => xs $ name_list（' privilege '），principal_name =>' user_or_role '，principal_type => xs $ ace_type_user ））;

    在本规范中：

    • privilege_list ：输入以下一个或多个特权，这些特权不区分大小写。用单引号括起每个权限，并用逗号分隔每个权限（例如， 'http', 'http_proxy' ）。

      对于更严格的访问控制，如果用户仅使用UTL_HTTP ， HttpUriType ， UTL_SMTP或UTL_MAIL仅授予http ， http_proxy或smtp权限，而不是connect权限。

      - http ：通过UTL_HTTP包和HttpUriType类型向主机发出HTTP请求

      - http_proxy ：通过UTL_HTTP包和HttpUriType类型通过代理发出HTTP请求。如果用户通过代理发出HTTP请求，则必须将http_proxy与http特权结合使用。

      - smtp ：通过UTL_SMTP和UTL_MAIL包向主机发送SMTP

      - resolve ：通过UTL_INADDR包解析网络主机名或IP地址

      - connect ：通过UTL_TCP ， UTL_SMTP ， UTL_MAIL ， UTL_HTTP和DBMS_LDAP包或HttpUriType类型授予用户连接主机网络服务的权限

      - jdwp ：用于Java或PL / SQL存储过程的Java Debug Wire Protocol调试操作。有关更多信息，请参阅为Java调试线协议操作配置网络访问 。

    • principal_name ：输入数据库用户名或角色。除非您使用双引号输入（例如， '"ACCT_MGR'" ），否则此值是不区分大小写的。

    • principal_type ：为数据库用户或角色输入XS_ACL.PTYPE_DB 。您必须指定PTYPE_DB因为principal_type值默认为PTYPE_XS ，用于指定Oracle Database Real Application Security应用程序用户。

DECLARE req_context UTL_HTTP.REQUEST_CONTEXT_KEY; req UTL_HTTP.REQ; BEGIN req_context：= UTL_HTTP.CREATE_REQUEST_CONTEXT（wallet_path =>'file： path_to_directory_containing_wallet '，wallet_password =>' wallet_password '| NULL）; req：= UTL_HTTP.BEGIN_REQUEST（url =>' URL_to_application '，request_context =>' request_context '| NULL）; ...结束;

    wallet_path =>'file：/ oracle / wallets / hr_wallet'，

    url =>'www.hr_access.example.com'，

仅使用客户端证书进行身份验证

只要用户在ACL钱包中被授予了相应的权限，只有客户端证书才能对用户进行身份验证。

身份验证应在远程Web服务器上成功，用户可以使用GET_RESPONSE函数继续检索HTTP响应。

使用密码进行身份验证

DECLARE req_context UTL_HTTP.REQUEST_CONTEXT_KEY; req UTL_HTTP.REQ;开始 ...UTL_HTTP.SET_AUTHENTICATION_FROM_WALLET（r => HTTP_REQUEST ，alias =>' alias_to_retrieve_credentials_stored_in_wallet '，scheme =>'AWS | Basic'，for_proxy => TRUE | FALSE）;结束;

在本规范中：

  • r ：输入您在上一节中创建的UTL_HTTP.BEGIN_REQUEST过程中定义的HTTP请求。例如：

    r => req，

  • alias ：输入用于标识和检索存储在Oracle钱包中的用户名和密码凭据的别名。例如，假设用于标识此用户名和密码凭据的别名是hr_access 。 

    alias =>'hr_access'，

    • AWS ：指定Amazon Simple Storage Service（S3）方案。仅在配置对Amazon.com网站的访问时才使用此方案。（有关此设置的详细信息，请与Amazon联系。）

    例如：

    scheme =>'基本'，

  • for_proxy ：指定HTTP身份验证信息是否用于访问HTTP代理服务器而不是Web服务器。默认值为FALSE 。

    例如：

    for_proxy => TRUE）;

指定一组网络主机

您可以使用通配符指定一组网络主机。

  • 要将访问控制列表分配给一组网络主机，请使用星号（*）通配符。

无类别域间路由（ CIDR ）表示法定义了如何对IPv4和IPv6地址进行分类，以便在Internet上路由IP数据包。DBMS_NETWORK_ACL_ADMIN程序包支持IPv4和IPv6地址的CIDR表示法。此程序包考虑IPv4映射的IPv6地址或子网等效于其表示的IPv4本机地址或子网。例如， ::ffff:192.0.2.1相当于192.0.2.1 ， ::ffff:192.0.2.1/120相当于192.0.2.* 。

多个访问控制列表分配中主机的优先顺序

分配给域的访问控制列表的优先级低于分配给子域的访问控制列表。

对于分配给主机及其域的多个访问控制列表，分配给主机的访问控制列表优先于分配给域的访问控制列表。

分配给域的访问控制列表的优先级低于分配给子域的访问控制列表。例如，Oracle数据库首先选择分配给主机server.us.example.com的访问控制列表，然后再分配给分配给其域的其他访问控制列表。如果为子域分配了其他访问控制列表，则其优先顺序如下：

  1. server.us.example.com

  2. * .us.example.com

  3. * .example.com的

  4. * .COM

  5. *

8.1 XStream简介

XStream In使远程客户端应用程序能够从另一个系统（例如非Oracle数据库或文件系统）将信息发送到Oracle数据库。

XStream In提供了一个高效的，基于事务的接口，用于从客户端应用程序向Oracle数据库发送信息。XStream In可以通过多种方式使用进入Oracle数据库的信息，包括数据复制，审计和更改数据捕获。XStream In支持OCI和Java接口。

与直接对Oracle数据库进行DML更改的OCI客户端应用程序相比，XStream In对于Oracle数据库的近实时，基于事务的异构DML更改更有效。

XStream In使用以下Oracle Replication功能：

  • DML的高性能处理改变，可选地具有并行性

  • 使用应用处理程序应用流程功能，例如SQL生成，冲突检测和解决，错误处理以及自定义处理

  • 以最少的网络往返流式传输信息的网络

  • 规则，规则集和基于规则的转换

    在入站服务器使用的规则上指定基于规则的自定义转换时，调用转换函数的用户是入站服务器的应用用户。

XStream In支持Oracle Replication支持的所有数据类型，包括LOB， LONG ， LONG RAW和XMLType 。客户端应用程序以块的形式将LOB和XMLType数据发送到入站服务器。几个块包含单列值LOB， LONG ， LONG RAW或XMLType数据类型。

8.2.3入站服务器的LCR处理选项

入站服务器可以直接应用LCR，也可以将LCR发送到应用处理程序进行处理。您的LCR处理选项取决于入站服务器接收的LCR是行LCR还是DDL LCR。

默认情况下，入站服务器直接应用LCR。入站服务器在LCR中标识的数据库对象上执行LCR的更改。入站服务器成功应用LCR中的更改，或者，如果遇到冲突或应用错误，则尝试使用冲突处理程序或称为错误处理程序的用户指定过程来解决错误。

如果冲突处理程序可以解决冲突，那么它应用LCR或丢弃LCR中的更改。如果错误处理程序可以解决错误，那么它应该应用LCR（如果适用）。错误处理程序可以通过在应用之前修改LCR来解决错误。如果冲突处理程序或错误处理程序无法解决错误，则入站服务器会将事务以及与事务关联的所有LCR放入错误队列中。

您可以使用应用处理程序以自定义方式处理LCR，而不是直接应用LCR。使用apply处理程序时，入站服务器会将LCR传递给SQL语句的集合或用户定义的PL / SQL过程进行处理。应用处理程序可以自定义方式处理LCR。

  • 声明DML处理程序
    语句DML处理程序使用一组SQL语句来处理行LCR。
  • 过程DML处理程序

8.2.3.1.1语句DML处理程序

语句DML处理程序使用一组SQL语句来处理行LCR。

语句DML处理程序具有以下特征：

  • 机制：SQL语句的集合

  • LCR的类型：行LCR

  • 范围：一张桌子上的一次操作

  • 每个入站服务器允许的数量：可以在同一个表上为同一操作指定许多和多个

语句DML处理程序中包含的每个SQL语句都具有唯一的执行序列号。当调用语句DML处理程序时，它按顺序从具有最低执行序列号的语句到具有最高执行序列号的语句执行其语句。执行序列号可以是正数，负数或十进制数。

对于与入站服务器关联的每个表，您可以设置单独的语句DML处理程序来处理行LCR中的以下每种操作类型：

  • 插入

  • UPDATE

  • 删除

当入站服务器收到对指定表执行指定操作的行LCR时，将调用语句DML处理程序。例如， hr.employees表可以有一个语句DML处理程序来处理INSERT操作，另一个语句DML处理程序来处理UPDATE操作。或者， hr.employees表可以为每种类型的操作使用相同的语句DML处理程序。

您可以在同一个表上为同一操作指定多个语句DML处理程序。在这种情况下，这些语句DML处理程序可以按任何顺序执行，并且每个语句DML处理程序接收入站服务器接收的原始行LCR的副本。

8.2.3.1.2过程DML处理程序

过程DML处理程序使用PL / SQL过程来处理行LCR。

过程DML处理程序具有以下特征：

  • 机制：用户定义的PL / SQL过程

  • LCR的类型：行LCR

  • 范围：一张桌子上的一次操作

  • 每个入站服务器允许的数量：很多，但只能在同一个表上为同一操作指定一个

对于与入站服务器关联的每个表，您可以设置单独的过程DML处理程序来处理行LCR中的以下每种类型的操作：

  • 插入

  • UPDATE

  • 删除

  • LOB_UPDATE

当入站服务器收到对指定表执行指定操作的行LCR时，将调用过程DML处理程序。例如， hr.employees表可以有一个用于处理INSERT操作的过程DML处理程序和一个用于处理UPDATE操作的不同过程DML处理程序。或者， hr.employees表可以为每种类型的操作使用相同的过程DML处理程序。

PL / SQL过程可以执行行LCR的任何定制处理。例如，如果您希望每个插入到源数据库的特定表中以导致插入目标数据库中的多个表，那么您可以创建一个用户定义的PL / SQL过程来处理表上的INSERT操作以实现此目的。与语句DML处理程序不同，过程DML处理程序可以修改行LCR中的列值。

8.2.3.4预先处理程序

8.2.4入站服务器和受限制的会话

启用和禁用受限会话会影响入站服务器。

在系统启动期间通过发出STARTUP RESTRICT语句启用受限会话时，即使数据库关闭时它们正在运行，也不会启动入站服务器。禁用受限会话时，将启动未停止的每个入站服务器。

当SQL语句ALTER SYSTEM ENABLE RESTRICTED SESSION在正在运行的数据库中启用受限会话时，它不会影响任何正在运行的入站服务器。这些入站服务器继续运行并将LCR发送到XStream客户端应用程序。如果在受限会话中启动了已停止的入站服务器，则在禁用受限会话之前，入站服务器实际上不会启动。

8.2.5入站服务器组件

入站服务器由以下子组件组成：读取器服务器，协调器进程和一个或多个应用服务器。

入站服务器由以下子组件组成：

  • 从XStream客户端应用程序接收LCR的读取器服务器 。读取器服务器是计算逻辑变更记录（LCR）之间的依赖关系并将LCR组装成事务的过程。然后，阅读器服务器将组装的事务返回给协调器进程。

    您可以通过查询V$XSTREAM_APPLY_READER动态性能视图来查看入站服务器的阅读器服务器的状态。

  • 协调器进程 ，它从读取器服务器获取事务并将它们传递给应用服务器。协调器进程名称是AP nn ，其中nn可以包括字母和数字。协调程序进程是Oracle后台进程。

    您可以通过查询V$XSTREAM_APPLY_COORDINATOR动态性能视图来查看协调程序进程的状态。

  • 一个或多个应用服务器 ，将LCR作为DML或DDL语句应用于数据库对象，或者将LCR传递给相应的应用处理程序。应用服务器还可以将LCR排入DBMS_APPLY_ADM.SET_ENQUEUE_DESTINATION过程指定的队列的持久队列部分。每个应用服务器都是一个进程。如果应用服务器遇到错误，则它会尝试使用用户指定的冲突处理程序或错误处理程序来解决错误。如果应用服务器无法解决错误，则它将回滚事务并将整个事务（包括其所有LCR）放入错误队列中。

    当应用服务器提交已完成的事务时，已应用此事务。当应用服务器将事务放入错误队列并提交时，也会应用此事务。

    您可以通过查询V$XSTREAM_APPLY_SERVER动态性能视图来查看入站服务器的每个应用服务器的状态。

8.2.7入站服务器的错误队列

错误队列包含数据库的所有当前应用错误。如果数据库中有多个入站服务器，则错误队列包含每个入站服务器的应用错误。

受信任的用户可以通过查询DBA_APPLY_ERROR数据字典视图或使用Oracle Enterprise Manager Cloud Control来查看应用错误。DBA_APPLY_ERROR数据字典视图使受信任用户可以查看有关其他用户的应用错误的信息。不受信任的用户可以通过查询ALL_APPLY_ERROR数据字典视图来查看应用错误。此视图仅显示不受信任用户的应用错误。

此外，受信任的用户可以通过查询DBA_APPLY_ERROR_MESSAGES数据字典视图来查看有关应用错误的更多详细信息。不受信任的用户可以通过查询ALL_APPLY_ERROR_MESSAGES数据字典视图来查看有关应用错误的更多详细信息。这些视图包含有关在错误事务中导致错误的行的信息。

错误队列存储有关在数据库中运行的入站服务器无法成功应用的事务的信息。交易可以包括许多LCR。在应用期间发生未处理的错误时，入站服务器会自动将事务中满足入站服务器规则集的所有LCR移动到错误队列。

您可以更正导致错误的条件，然后重新执行导致错误的事务。例如，您可以修改表中的行以更正导致错误的条件。

如果更正了导致错误的条件，则可以使用EXECUTE_ERROR或EXECUTE_ALL_ERRORS过程在错误队列中重新执行事务，也可以使用DELETE_ERROR或DELETE_ALL_ERRORS过程从错误队列中删除事务。这些过程位于DBMS_APPLY_ADM包中。

在错误队列中重新执行事务时，可以指定事务由最初将错误放入错误队列的用户执行，或者由重新执行事务的用户执行。此外，当您在错误队列中重新执行事务时，将使用入站服务器的当前标记。

重新执行的事务使用任何相关的应用处理程序和冲突解决处理程序。如果要解决错误，必须在执行错误队列之前修改错误队列中的行LCR，然后可以配置过程DML处理程序来处理导致错误队列中的错误的行LCR。在这种情况下，DML处理程序可以修改行LCR以避免重复相同的错误。当您重新执行包含行LCR的错误时，行LCR将传递给DML处理程序。例如，语句DML处理程序可能插入与插入行LCR中存在的值不同的值，而过程DML处理程序可能会修改行LCR中的一个或多个列以避免重复相同的错误。

8.3 LCR和XStream的位置

客户端应用程序将LCR流式传输到入站服务器中的XStream。

本节介绍与入站服务器的LCR位置相关的概念。

每个位置必须以支持字节比较的格式（例如base-16编码）进行编码。该位置对于客户端应用程序使用XStream In接口发送的事务流的总顺序至关重要。

以下位置对于入站服务器很重要：

  • 应用的低位表示已应用小于或等于该值的LCR。

    当LCR已执行，发送到应用处理程序或移动到错误队列时，入站服务器将应用LCR。

  • 溢出位置表示位置小于或等于此值的LCR已应用或从内存溢出到硬盘。

  • 应用的高位表示已应用的LCR的最高位置。

    当commit_serialization apply参数设置为入站服务器的DEPENDENT_TRANSACTIONS ，可能会在具有较低提交位置的LCR之前应用具有较高提交位置的LCR。发生这种情况时，应用的高位置与应用的低位置不同。

  • 处理后的低位置是应用的低位置或溢出位置的较高值。

    处理后的低位是入站服务器不再需要任何LCR的位置。此位置对应于应用捕获过程捕获的更改的Oracle Apply流程的最旧SCN。

    处理后的低位表示入站服务器已处理位置小于或等于此位置的LCR。如果客户端重新连接到入站服务器，则它必须仅发送位置大于处理的低位置的LCR，因为入站服务器丢弃位置小于或等于处理的低位置的任何LCR。

如果客户端应用程序异常停止，则客户端应用程序与入站服务器之间的连接将自动中断。重新启动后，客户端应用程序从入站服务器检索已处理的低位置，并指示其捕获代理程序从此处理的低位置开始检索更改。

要使用XStream In界面限制客户端应用程序的恢复时间，客户端应用程序可以定期向入站服务器发送活动（如空事务）。行LCR可以包括提交事务控制指令。当没有LCR要发送到服务器时，客户端应用程序可以发送带有commit指令的行LCR，以提前入站服务器处理的低位置。此活动充当确认，以便提前入站服务器处理的低位置。

8.4.1在大型事务的性能中优化XStream

对于小事务，XStream In不会开始应用逻辑更改记录（LCR），直到入站服务器从源接收事务的提交LCR。作为性能优化，入站服务器可以使用eager apply在接收提交LCR之前开始应用大型事务。

eager_size apply参数控制在eager apply开始之前入站服务器接收的最小LCR数。当事务中的LCR数超过eager_size apply参数的值时，入站服务器开始应用LCR。此参数的默认值为9500。您可以修改参数值以优化XStream在您的环境中的性能。

大型事务可能需要额外的应用服务器来应用LCR。在为事务启动eager之后，入站服务器可以自动创建其他应用服务器以应用LCR。max_parallelism apply参数控制入站服务器的最大应用服务器数。

如果入站服务器自动创建其他应用服务器，并且其中一些服务器在一段时间内处于空闲状态，则XStream In会确定不再需要它们并自动将其删除。但是，应用服务器的数量永远不会低于parallelism apply参数指定的值。这些应用服务器的任何统计信息都聚合为应用服务器0（零）。

对于使用急于申请大额交易的入站服务器，该值eager_size应用参数必须小于该值txn_lcr_spill_threshold适用参数。当txn_lcr_spill_threshold的值低于eager_size ，事务在eager apply开始之前溢出到磁盘，并且入站服务器无法使用eager申请已溢出到磁盘的事务。

8.4.2避免事务跟踪中的潜在瓶颈

XStream In跟踪它应用于数据库的更改，以避免在重新启动入站服务器时重新应用事务。

当optimize_progress_table apply参数设置为TRUE ，默认的XStream In会在重做日志中跟踪其进度。使用重做日志可以避免进度表中DML更改导致的潜在瓶颈和争用。

当optimize_progress_table参数设置为FALSE ，XStream In使用表进行跟踪。在高容量环境中，此表可能是潜在的瓶颈。

应用数据库必须处于存档日志模式，然后才能在重做日志中执行应用跟踪。如果optimize_progress_table参数设置为TRUE但apply数据库未处于存档日志模式，则忽略optimize_progress_table的设置，并且XStream In使用表进行跟踪。

8.4.3优化事务应用调度

当目标表上的约束与源表上的约束匹配时，可以通过将入站服务器的compute_lcr_dep_on_arrival apply参数设置为Y来优化依赖性计算。

如果约束不匹配，则将此apply参数设置为N （默认值）。

如果此apply参数设置为Y ，则在接收到事务的LCR时计算依赖关系。如果此apply参数设置为N ，则仅在接收到已提交事务的所有LCR之后才计算依赖关系。

无论compute_lcr_dep_on_arrival应用参数设置如何，密钥列的前映像必须在入站服务器接收的LCR中可用。键列包括主键列，外键列和唯一约束列。在XStream配置中，入站服务器在XStream Out配置中应用捕获进程捕获的更改，补充日志记录可确保所需信息位于LCR中。

8.5.3应用用户对入站服务器所需的权限

入站服务器在其应用用户的安全域中应用LCR。

apply用户必须具有应用更改所需的权限，包括以下权限：

  • 应用数据操作语言（DML）所需的权限更改为其他模式中的表（当入站服务器接收对其他模式中的表的DML更改时）

  • 将数据定义语言（DDL）更改所需的权限更改为数据库（当入站服务器接收DDL更改时）

  • 对入站服务器使用的规则集的EXECUTE特权

  • 对为正规则集中的规则指定的所有基于规则的自定义转换函数的EXECUTE特权

  • 任何应用处理程序的EXECUTE特权

入站服务器只能与一个用户关联，但一个用户可以与许多入站服务器关联。

通过为GRANT_ADMIN_PRIVILEGE过程中的privilege_type参数指定APPLY为具有DBMS_XSTREAM_AUTH包的apply用户授予权限。

8.6.4 XStream In和多租户环境

多租户环境使Oracle数据库能够包含一组可移植的模式，对象和相关结构，这些模式在逻辑上作为单独的数据库出现在应用程序中。

这个自包含的集合称为可插拔数据库（PDB）。多租户容器数据库（CDB）包含PDB。它还可以包含应用程序容器。应用程序容器是CDB的可选组件，它由应用程序根目录和与之关联的所有应用程序PDB组成。应用程序容器存储一个或多个应用程序的数据。应用程序容器共享应用程序元数据和公共数据在CDB中，以下各项都是容器：CDB根，每个PDB，每个应用程序根和每个应用程序PDB。

在CDB中，入站服务器仅限于从一个源数据库接收LCR，并且仅在当前容器中执行更改（一个PDB，一个应用程序根，一个应用程序PDB或CDB根）。单个入站服务器无法将更改应用于CDB中的多个容器。

当入站服务器位于CDB根目录中时，应用用户必须是普通用户。当入站服务器位于应用程序根目录中时，应用用户必须是普通用户或应用程序公共用户。当入站服务器位于PDB或应用程序PDB中时，应用用户可以是普通用户或本地用户。

注意：

XStream不会同步应用程序根容器中所做的更改。不要使用XStream In复制来复制在应用程序根容器中完成的操作。您可以在目标中的应用程序根容器中手动应用这些更改。请注意，仍可以复制在PDB中完成的操作。

    关于命令规则

    命令规则保护影响一个或多个数据库对象的Oracle数据库SQL语句。

    这些语句可以包括SELECT ， ALTER SYSTEM ，数据库定义语言（DDL）和数据操作语言（DML）语句。

    要自定义和强制执行命令规则，请将其与规则集关联，规则集是一个或多个规则的集合。命令规则在运行时强制执行。命令规则会影响尝试使用它保护的SQL语句的任何人，无论对象存在的领域如何。

    除了基本的Oracle Database DDL和DML语句之外，您还可以使用命令规则来保护各种SQL语句。例如，您可以保护与Oracle闪回技术一起使用的语句。

    除了将命令规则与命令关联之外，命令规则还具有以下属性：

    • 命令规则保护的SQL语句

    • 命令规则影响的对象的所有者

    • 命令规则影响的数据库对象

    • 是否启用了命令规则

    • 关联的规则集

    命令规则可以分类如下：

    • 具有系统范围的命令规则。 对于此类型，在大多数情况下，您只能为每个数据库实例创建一个命令规则。

    • 特定于模式的命令规则。 特定于模式的命令规则的示例是DROP TABLE语句的命令规则。您只能为每个架构创建一个CONNECT命令规则。

    • 特定于对象的命令规则。 例如，使用命令规则定义中包含的特定表为DROP TABLE语句创建命令规则。

    当用户执行受命令规则影响的语句时，Oracle Database Vault会首先检查领域授权。如果未发现域违规并且启用了关联的命令规则，则Database Vault将评估关联的规则集。如果所有规则集都评估为TRUE ，则该语句被授权进一步处理。如果任何规则集的计算结果为FALSE ，则不允许执行该语句并引发命令规则违规。

    您可以定义一个命令规则，该命令规则使用CONNECT事件的因子来在通常的步骤 - 用户身份验证过程，因子初始化和Oracle Label Security集成完成后允许或拒绝会话。例如，您可以配置一个命令规则，允许在工作时间之后授权BIZAPP模式中的DDL语句（如CREATE TABLE ， DROP TABLE和ALTER TABLE ，但不能在工作时间授权。

    您可以运行在Oracle Database Vault中创建的命令规则的报告。

    您无法创建阻止SYS执行SYS拥有的过程的命令规则。

    ALTER SESSION和ALTER SYSTEM命令规则

    您可以创建不同类型的ALTER SESSION和ALTER SYSTEM命令规则，为这些SQL语句提供细粒度控制。

    创建这些类型的命令规则的过程如下：

    • DBMS_MACADM.CREATE_COMMAND_RULE创建ALTER SESSION和ALTER SYSTEM命令规则，这些规则使用来自相应SQL语句的子句，例如ADVISE ， CLOSE DATABASE LINK ， COMMIT IN PROCEDURE ， SET for ALTER SESSION ，或ARCHIVE_LOG ， CHECK DATAFILES ， CHECKPOINT和SET for ALTER SYSTEM 。

    • DBMS_MACADM.CREATE_SESSION_EVENT创建特定于ALTER SESSION SET EVENTS SQL语句的命令规则

    • DBMS_MACADM_CREATE_SYSTEM_EVENT创建特定于ALTER SYSTEM SET EVENTS SQL语句的命令规则。

    要创建这些命令规则，请使用相应的Database Vault过程在创建语句中指定子句的参数（如果适用）。如果ALTER SESSION或ALTER SYSTEM命令规则使用SET EVENTS设置，则可以使用特殊参数指定事件，组件和操作。

    例如，对于ALTER SYSTEM命令规则，您可以从ALTER SYSTEM SQL语句指定SECURITY子句及其RESTRICTED SESSION参数。要指定RESTRICTED SESSION是TRUE还是FALSE ，您必须创建Database Vault规则和规则集，该规则集可以测试此序列号的有效性。

    要了解此概念的工作原理，请首先创建以下规则和规则集，这些规则和规则集旨在检查RESTRICTED SESSION参数是否设置为TRUE ：

    EXEC DBMS_MACADM.CREATE_RULE（'RESTRICTED SESSION TRUE'，'UPPER（PARAMETER_VALUE）=''TRUE'''）; BEGIN DBMS_MACADM.CREATE_RULE_SET（rule_set_name =>'检查RESTRICTED SESSION为TRUE'，description =>'检查受限会话是否为真'，启用=> DBMS_MACUTL.G_YES，eval_options => DBMS_MACUTL.G_RULESET_EVAL_ALL，audit_options => DBMS_MACUTL.G_RULESET_AUDIT_FAIL + DBMS_MACUTL .G_RULESET_AUDIT_SUCCESS，fail_options => DBMS_MACUTL.G_RULESET_FAIL_SILENT，fail_message =>'RESTRICTED SESSION不为TRUE'，fail_code => 20461，handler_options => DBMS_MACUTL.G_RULESET_HANDLER_FAIL，handler =>''，is_static => false）;结束; / EXEC DBMS_MACADM.ADD_RULE_TO_RULE_SET（检查RESTRICTED SESSION为TRUE'，'RESTRICTED SESSION TRUE'）;

    使用规则和规则集，您就可以创建一个ALTER SYSTEM命令规则来检查RESTRICTED SESSION参数：

    BEGIN DBMS_MACADM.CREATE_COMMAND_RULE（command =>'ALTER SYSTEM'，rule_set_name =>'Check RESTRICTED SESSION为TRUE'，object_owner =>'％'，object_name =>'％'，enabled => DBMS_MACUTL.G_YES，clause_name =>'SECURITY '，parameter_name =>'RESTRICTED SESSION'，scope => DBMS_MACUTL.G_SCOPE_LOCAL）;结束; /

    在这个例子中：

    • rule_set_name检查RESTRICTED SESSION是设置为TRUE还是FALSE 。在多租户环境中，必须在与命令规则相同的位置创建规则集和规则：在应用程序根目录中或在PDB中本地。

    • 对于这种ALTER SESSION或ALTER SYSTEM命令规则， object_owner和object_name必须始终设置为% 。

    • enabled允许您启用或禁用命令规则，或使用模拟模式将违反命令规则的日志记录到指定的日志表。日志数据描述错误，例如使用的用户名或SQL语句。

    • clause_name指定ALTER SYSTEM SQL语句的SECURITY子句

    • parameter_name指定SECURITY子句中的RESTRICTED SESSION参数

    • scope将命令规则设置为当前PDB的本地规则。关联的规则集和规则也必须是当前PDB的本地规则。如果要在应用程序根目录中创建命令规则，则作为普通用户，可以将scope设置为DBMS_MACUTL.G_SCOPE_COMMON并从应用程序根目录运行该过程（及其随附的规则集和规则创建过程）。

      • 适用对象名称：输入命令规则影响的数据库对象的名称，或指定％以选择所有数据库对象，包括表，过程，视图，统一审计策略等。如果从“对象所有者”列表中选择了对象所有者，则此属性是必需的。

      • 规则集：从列表中，选择要与命令规则关联的规则集。此属性是必需的。

        如果规则集的计算结果为true，则SQL语句将成功。如果计算结果为false，则语句将失败，然后Oracle Database Vault会引发命令规则违规。（您可以使用Oracle Database Vault Reports中讨论的命令规则配置问题报告来跟踪规则违规。）与规则集关联的任何审核和自定义事件处理都作为命令规则处理的一部分发生。

        有关规则集的详细信息，请参阅配置规则集。

    命令规则如何工作

    命令规则遵循一组步骤来检查其关联的组件。

    Realms Work如何描述当数据库帐户发出影响领域内对象的SELECT ，DDL或DML语句时会发生什么。

    发出SELECT ，DDL或DML语句时会发生以下操作：

    1. Oracle Database Vault会查询需要应用的所有命令规则。

      对于SELECT ，DDL和DML语句，可能会应用多个命令规则，因为对象所有者和对象名称支持通配符表示法。

      您可以将规则集与命令规则和领域授权相关联。Oracle Database Vault首先评估领域授权规则集，然后评估适用于正在评估的命令类型的规则集。

    2. 对于适用的每个命令规则，Oracle Database Vault会评估其关联的规则集。

    3. 如果任何适用的命令规则的关联规则集返回false或错误，则Oracle Database Vault会阻止该命令执行。否则，该命令被授权进一步处理。针对审计和事件处理程序的规则集的配置指示发生的审计或自定义处理。

      命令规则覆盖对象权限。也就是说，如果对象受命令规则保护，即使对象的所有者也无法访问该对象。您可以禁用命令规则或命令的规则集。如果禁用命令规则，则命令规则不会执行它旨在处理的检查。如果禁用规则集，则规则集始终计算为TRUE 。但是，如果要禁用特定命令的命令规则，则应禁用命令规则，因为规则集可能与其他命令规则或域授权相关联。

    设计命令规则的指南

    Oracle提供了设计命令规则的指南。

    • 创建更精细的命令规则，因为它们更容易维护。

      例如，如果要阻止在特定模式对象上发生SELECT语句，则设计多个命令规则以停止那些特定模式对象上的SELECT语句，而不是创建一般命令规则来阻止模式级别中的SELECT语句。

    • 在为CONNECT事件设计规则时，请注意包含不会无意中锁定任何所需用户连接的逻辑。如果任何帐户被意外锁定，请询问已被授予DV_ADMIN或DV_OWNER角色的用户登录并更正导致锁定问题的规则。CONNECT命令规则不适用于具有DV_OWNER和DV_ADMIN角色的用户。这可以防止不正确配置的CONNECT命令规则导致完全锁定。

      如果帐户已被锁定，您可以禁用Oracle Database Vault，更正导致锁定问题的规则，然后重新启用Oracle Database Vault。即使禁用了Oracle Database Vault，您仍然可以使用Database Vault Administrator和Database Vault PL / SQL包。有关禁用和重新启用Database Vault的说明，请参阅禁用和启用Oracle Database Vault。

    • 有时，您必须暂时放松管理任务的已启用命令规则。而不要禁用命令规则，有安全管理器（与该帐户DV_ADMIN或DV_OWNER登录，设置规则设置为启用 ，启用审核成功或失败，名为默认规则集角色）启用，并设置任务完成后，命令规则返回其原始规则集。（请注意，在统一审核环境中，此设置不起作用。相反，您必须创建统一的审核策略。“Oracle数据库安全指南”介绍了如何为Database Vault创建统一的审核策略。）

    • 在设计命令规则时，请小心考虑自动化过程，例如备份，这些过程可能会被无意中禁用。您可以通过创建允许在一系列Oracle Database Vault因素为真时（例如，正在使用的程序），正在使用的帐户或客户端所在的计算机或网络上的命令来执行这些任务的帐户程序正在运行。

    • 您可以使用模拟模式测试命令规则的开发阶段，该模式启用命令规则，但会将有关它的详细信息写入日志文件。

    命令规则的性能取决于与命令规则关联的规则集中的规则的复杂性。
