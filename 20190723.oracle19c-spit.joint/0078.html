关于与Microsoft Active Directory的直接集成

Oracle数据库18c版，18.1版及更高版本支持使用新的集中管理用户功能与Microsoft Active Directory（MSAD）直接集成。

从Oracle数据库18c版本18.1开始，Oracle数据库引入了集中管理的用户，可以直接使用Microsoft Active Directory对用户进行身份验证和授权。通过集中管理的用户，可以集中管理访问数据库的用户，以改善组织的安全状况。企业用户（Microsoft Active Directory中的用户）可以独占映射到数据库帐户，或者许多企业用户（在Microsoft Active Directory组中）可以映射到数据库中的共享帐户。Microsoft Active Directory组还可以映射到数据库全局角色，该角色为用户提供超出其登录帐户（独占或共享）授予的其他权限和角色。通过集中管理的用户，可以使用密码，Kerberos和PKI证书对用户进行身份验证。

对于开始不需要某些更复杂的企业用户安全功能（如可信数据库链接）的新目录服务项目的组织，使用Microsoft Active Directory的集中管理用户允许您使用其当前的Microsoft Active Directory服务作为其集中用户管理和集中式数据库访问授权。对于新的目录服务项目，这可能是实现目录服务的首选选项，因为这样可以降低维护和开发方面的复杂性和操作成本。

企业用户还可以使用Oracle Internet Directory，它是Oracle身份管理基础结构的一部分。如果您的组织使用Microsoft Active Directory等第三方目录来存储和管理用户条目，则可以将其与Oracle Internet Directory集成以管理企业用户安全性。如果您的组织需要某些更复杂的企业用户安全功能（如可信数据库链接），这可能是您的首选选项。

10.1概述

启动或关闭数据库等特殊操作由数据库管理员（DBA）执行。该 DBA具有未分配给普通用户的某些权限。本章概述的命令通常由DBA使用。

有关SQL * Plus中的安全性和角色的详细信息，请参阅SQL * Plus安全性 。

10.2数据库启动和关闭简介

并非所有用户都可以使用Oracle数据库。要打开或关闭数据库，或启动或关闭实例，您必须具有DBA权限或以SYSOPER或SYSDBA身份连接。其他用户无法更改Oracle数据库的当前状态。

10.2.1数据库启动

要启动数据库：

  1. 启动一个实例

    实例控制后台进程和内存区域分配以访问Oracle数据库。

  2. 装入数据库

    挂载数据库会将其与先前启动的实例相关联。

  3. 打开数据库

    打开数据库使其可用于正常的数据库操作。

要在不安装数据库的情况下启动Oracle数据库实例，请输入

启动NOMOUNT

示例10-2安装数据库

要启动实例，请装入数据库，但保持数据库关闭，然后输入

启动安装

其中SALES是INITSALES.ORA参数文件中DB_NAME参数中指定的数据库名称。

有关数据库启动的更多信息，请参阅启动和关闭 。有关启动数据库的更多信息，请参阅STARTUP命令。

10.2.2 PDB启动

可插拔数据库（PDB）是存在于统一数据库中的模式和模式对象的自包含集合。

要启动可插拔数据库：

  1. 使用/NOLOG参数启动SQL * Plus：
    sqlplus / nolog
  2. 使用简易连接或网络服务名称发出CONNECT命令以连接到PDB。
  3. 发出STARTUP命令。

有关数据库启动的更多信息，请参阅启动和关闭 。有关启动数据库的更多信息，请参阅STARTUP命令。

10.2.3数据库关闭

关闭数据库涉及三个步骤：

  1. 关闭数据库

    关闭数据库时，SGA中的所有数据库和恢复数据都将写入数据文件和重做日志文件，并关闭所有在线数据文件。

  2. 卸载数据库

    卸载数据库会取消数据库与实例的关联，并关闭数据库的控制文件。

  3. 关闭实例

    关闭实例会从内存中回收SGA，并终止构成Oracle数据库实例的后台Oracle数据库进程。

  2. 运行SHUTDOWN命令。

10.3重做日志文件

每个Oracle数据库都有一组两个或多个重做日志文件。数据库的重做日志文件集合统称为数据库的重做日志。

重做日志用于记录对数据所做的更改。例如，如果存在数据库故障，则使用重做日志来恢复数据库。为防止涉及重做日志本身的故障，Oracle数据库具有镜像重做日志，因此可以在不同磁盘上维护两个或多个重做日志副本。

10.3.1 ARCHIVELOG模式

在ARCHIVELOG模式下操作数据库可以归档在线重做日志。

SQL ALTER SYSTEM命令可以从磁盘故障和实例故障中完全恢复，因为对数据库所做的所有更改都将永久保存在归档重做日志中。

有关重做日志文件和数据库归档模式的更多信息，请参阅ARCHIVE LOG命令。

介绍

通常，Oracle Universal Installer以交互模式运行，这意味着它会提示您在图形用户界面（GUI）屏幕中提供信息。或者，您可以在非交互模式下运行Oracle Universal Installer。非交互模式也称为静默模式或静默安装。

在以下情况下，您可能希望使用非交互模式在Microsoft Windows（64位）上安装Oracle Database Gateway：

  • 您需要以无人值守的方式将Oracle组件部署到多个节点。您可以从操作系统调度程序或站点通常使用的其他作业子系统安排非交互式安装模式。

示例6-1 Oracle Grid Infrastructure组和用户示例

该命令将创建以下Oracle Grid Infrastructure组和用户：

  • Oracle中央库存组或oraInventory组（ oinstall ），其成员将此组作为其主要组。该组的成员被授予OINSTALL系统特权，该特权授予写入oraInventory目录的权限以及其他相关的安装二进制特权。

  • OSASM组（ asmadmin ），在安装期间与Oracle Grid Infrastructure关联，其成员被授予SYSASM管理Oracle ASM的权限。

  • 适用于ASM组的OSDBA（ asmdba ），与安装期间的Oracle Grid Infrastructure存储相关联。其成员包括网格和任何数据库安装所有者，例如oracle1和oracle2 ，他们被授予对Oracle ASM的访问权限。使用Oracle ASM进行存储的任何其他安装所有者也必须成为该组的成员。

  • 适用于Oracle ASM（ asmoper ）的ASM组的asmoper ，在安装期间与Oracle Grid Infrastructure相关联。asmoper组的成员被授予有限的Oracle ASM管理员权限，包括启动和停止Oracle ASM实例的权限。

  • 一个Oracle网格基础架构安装所有者（ grid ），与oraInventory的组（ oinstall ）作为其主组，并且与OSASM（ asmadmin ）组和ASM（该OSDBA asmdba ）基团作为辅助组。

  • /u01/app/oraInventory 。群集上Oracle安装的中央清单。此路径grid:oinstall拥有，以使其他Oracle软件所有者能够写入中央库存。

  • grid:oinstall拥有OFA兼容的挂载点/u01 grid:oinstall安装前的grid:oinstall ，以便Oracle Universal Installer可以写入该路径。

  • 网格安装所有者/u01/app/grid的Oracle基础grid:oinstall具有775权限的grid:oinstall ，并在安装过程中更改为755权限。

  • 网格所有者/u01/app/19.0.0/grid由grid:oinstall拥有775（drwxdrwxr-x）权限。这些权限是安装所必需的，并且在安装过程中更改为root:oinstall具有755权限（ drwxr-xr-x ）。

2.2.2健康检查报告包含哪些内容？

    • 名称，例如，群集名称

    • 收集的日期和时间

    • 运行检查的用户的名称，例如root

  • 以前的报告结果作为附件

116.3.1 CLEANUP_GIDX过程

作为使用异步全局索引维护的先前分区维护操作的结果，全局索引可以包含指向不再存在的数据段的条目。这些陈旧的索引行在表或索引的任何操作期间都不会导致任何正确性问题或损坏，无论这些是查询，DML，DDL还是分析。此过程将识别并清除这些全局索引，以确保存储和性能方面的效率。

句法

DBMS_PART.CLEANUP_GIDX（schema_name_in IN VARCHAR2 DEFAULT NULL，table_name_in IN VARCHAR2 DEFAULT NULL）;

116.3.2 CLEANUP_ONLINE_OP过程

自动升级日志文件结构

AutoUpgrade实用程序生成包含作业状态和配置文件的日志文件结构

自动升级日志文件基本路径

使用全局参数autoupg_log_dir设置AutoUpgrade日志文件路径。默认情况下，global参数具有以下定义：

global.autoupgr_log_dir = /数据库/ jobmgr
AutoUpgrade配置和状态文件路径与您使用global.autoupgr_log_dir建立的目录路径相关。

/ cfgtoollogs /升级/汽车

自动配置工具日志目录（ /cfgtoollogs/upgrade/auto ）包含有关AutoUpgrade作业管理器处理的每个作业的特定信息。您可以使用这些目录中的信息来确定作业开始或停止时的问题。

AutoUpgrade实用程序维护所有AutoUpgrade作业的AutoUpgrade日志文件：

autoupgrade.log

autoupgrade.log文件包含两个日志文件，提供有关AutoUpgrade作业启动和停止的更多详细信息，这些文件可以帮助您进行调试：

  • user_autoupgrade.log ：作业信息，其格式设置为增强可读性。

  • autoupgrade_err.log ：作业运行时发生的任何意外异常的报告。

/ CONFIG_FILES

config_files目录包含AutoUpgrade内部运行时配置和全局临时文件。

/状态

/status目录包含JSON作业状态文件。它包含两个目录：

  • status.json ：此目录包含以JSON文件格式完成的所有作业的最终作业状态。

  • progress.json ：此目录包含当前以JSON文件格式运行的所有作业的进度。

目录中的每个模块都包含其执行的操作的状态文件。该模块采用以下格式，其中前缀dbname是数据库名称， operation是执行的升级操作，后缀status是该操作的完成状态：

dbname _ operation-name 。 status

该操作的成功或失败由后缀指示，后缀是.success ，表示该操作的成功执行，或.failure ，表示该操作失败。例如，以下模块名称表示在数据库sales成功运行预检操作：

sales_prechecks.success

操作模块名称可以是以下之一：

  • preupgrade ：preupgrade阶段，可以在其中运行自定义脚本。

  • prechecks ：升级检查在开始升级之前完成。

  • grp ：保证还原点（使用Oracle Flashback技术）

  • prefixups ：preupgrade fixups在开始升级之前运行

  • drain ：在开始升级之前完成或迁移现有作业的阶段。

  • dbupgrade ：升级发生的阶段。

  • postchecks ：升级完成后运行升级后检查的阶段。

  • postfixups ：运行升级后修正的阶段。

  • postupgrade ：可以运行自定义升级后脚本的阶段。

单个作业和数据库日志文件目录

由AutoUpgrade调度程序启动的每个作业都将获得一个具有该作业标识符前缀的目录。在该作业目录中，作业中的每个数据库都在路径/database/logs/ sid被赋予一个日志目录，其中sid是数据库的系统标识符。例如，在配置文件中标识的作业为sales1且数据库系统标识符为sales ，数据库销售的日志文件位于以下路径中：

sales.log_dir = /数据库/日志/销售额1

日志目录包含为该数据库执行的所有任务的所有相关日志文件。默认情况下，在/database/logs目录下创建由SID标识的目录。如果您选择以这种方式设置配置文件，则每个数据库作业都可以具有单独的日志目录。

/#### （工作号码）

单个作业运行放在由运行编号标识的子目录中，格式为/ #### ，其中####表示运行编号。例如： 0004 。运行号目录包含以下日志文件：

  • autoupgrade_err.log ：报告作业运行时发生的任何意外异常。

  • autoupgrade.log ：自动升级跟踪日志文件。提供作业的详细日志，以识别作业运行期间发生的任何问题。

  • user_autoupgrade.log ：自动升级作业状态文件，其格式设置为增强可读性。

/预升级

preupgrade目录（ /preupgrade ）包含以下文件和日志文件：

  • preupgrade_source.log ：跟踪日志文件。此文件提供详细的日志，可帮助识别在升级前作业阶段发生的问题。

  • SOURCE.html ：图形用户界面，显示有关数据库状态的报告。

  • SOURCE_preupgrade.log ：显示数据库状态报告的文本用户界面。

/ dbupgrade

数据库升级目录（ /dbupgrade ）包含与数据库升级关联的所有日志文件：

  • autoupgrade YYYYMMDDHHMNSC dbname .log ：源数据库的日志文件，由运行升级的日期标识，以及数据库名称，表示并行性。格式：

    • YYYY ：年份

    • MM ：月

    • DD ：日

    • HH ：小时

    • MN ：分钟

    • SC ：第二

    • dbname ：数据库名称，其中dbname是数据库名称。

  • catupgrade YYYYMMDDHHMNSC dbnameN .log ：源数据库的日志文件，由运行升级的日期标识。

    格式：

    • YYYY ：年份

    • MM ：月

    • DD ：日

    • HH ：小时

    • MN ：分钟

    • SC ：第二

    • dbnameN ：数据库名称，其中dbname是数据库名称， N表示并行度：CDB ROOT为0 ... 3，非CDB数据库为0 ... 1，PDB为0 ... 1。

/ TEMP

临时AutoUpgrade文件（ /temp ）。此目录可以包含升级期间使用的PFILE等文件。

创建使用Oracle ASM的数据库

建议创建数据库的方法是使用Database Configuration Assistant（DBCA）。但是，如果使用CREATE DATABASE语句手动CREATE DATABASE ，则Oracle ASM使您能够以最少的输入创建数据库及其所有基础文件。

以下示例显示了CREATE DATABASE语句的使用，其中Oracle ASM自动创建和管理数据库文件。 此示例使用以下Oracle ASM文件创建数据库：

  • 磁盘组data SYSTEM表空间数据文件。

  • 磁盘组data SYSAUX表空间数据文件。表空间在本地管理，具有自动段空间管理功能。

  • 具有两个在线日志组的多路复用联机重做日志，每个组在data和fra （快速恢复区域）中各有一个。

  • 如果启用了自动撤消管理模式，则在目录data删除表空间数据文件。

  • 如果没有为CONTROL_FILES初始化参数指定值，则为两个控制文件，一个在data ，另一个在fra （快速恢复区域）中。data中的控制文件是主控制文件。

初始化参数文件中包含以下初始化参数设置：

DB_CREATE_FILE_DEST = + data DB_RECOVERY_FILE_DEST = + fra DB_RECOVERY_FILE_DEST_SIZE = 10G

可以在SQL提示符下运行以下语句：

SQL> CREATE DATABASE orcl;

在Oracle ASM中创建表空间

当Oracle ASM为永久表空间（或临时表空间的临时文件）创建数据文件时，数据文件将设置为自动可扩展，具有无限的最大大小和100 MB的默认大小。您可以使用AUTOEXTEND子句覆盖此默认扩展性，使用SIZE子句覆盖默认大小。

Oracle ASM将属性应用于数据文件，如数据文件的系统默认模板中所指定，如“ 管理磁盘组模板 ”中的表所示。您还可以创建和指定自己的模板。

由于表空间历史记录，表空间中的文件可能位于Oracle ASM文件和非Oracle ASM文件中。RMAN命令可以将非Oracle ASM文件重定位到Oracle ASM磁盘组，并使Oracle ASM文件可以作为非Oracle ASM文件重新定位。

例如，如果存在名为data的磁盘组，则可以使用以下SQL语句在该磁盘组中创建表空间tblspace ：

SQL> CREATE TABLESPACE tblspace DATAFILE'+ data';

以下示例说明了默认情况下Oracle ASM的用法。此示例使Oracle ASM能够为您创建和管理表空间数据文件，使用适用于大多数情况的Oracle提供的默认值。

假设以下初始化参数设置：

DB_CREATE_FILE_DEST = +数据

以下语句创建表空间及其数据文件：

SQL> CREATE TABLESPACE tblspace;

在Oracle ASM中创建重做日志

可以在多个磁盘组中创建联机重做日志，可以隐式在初始化参数文件中创建，也可以在ALTER DATABASE...显式创建ALTER DATABASE...ADD LOGFILE语句。每个联机日志在多个磁盘组中应该有一个日志成员。将自动生成日志文件成员的文件名。由于系统错误而创建的所有部分创建的重做日志文件都将自动删除。

例如，如果存在名为data1和data2磁盘组，则可以使用以下SQL语句创建每个磁盘组中包含成员的日志文件：

SQL> ALTER DATABASE ADD LOGFILE（'+ data1'，'+ data2'）;

以下示例创建一个日志文件，其中包含每个磁盘组data1和data2 。这些参数设置包含在初始化参数文件中：

DB_CREATE_ONLINE_LOG_DEST_1 = + data1 DB_CREATE_ONLINE_LOG_DEST_2 = + data2

可以在SQL提示符下运行以下语句：

SQL> ALTER DATABASE ADD LOGFILE;

在Oracle ASM中创建控制文件

可以在多个磁盘组中显式创建控制文件。将自动生成控制文件的文件名。

如果必须按名称指定控制文件，则可以使用别名文件名来引用Oracle ASM文件并使用易于理解的名称。在创建控件文件的规范中使用别名使您可以稍后使用用户指定的名称引用控件文件。也可以在CONTROL_FILES初始化参数中将别名指定为控制文件名。

创建控制文件时，应使用其Oracle ASM文件名的文件引用形式将存储在Oracle ASM磁盘组中的数据文件和日志文件提供给CREATE CONTROLFILE命令。但是，使用RESETLOGS选项需要使用文件创建表单来指定日志文件。

22.1关于O-Cluster

O-Cluster是一种快速，可扩展的基于网格的聚类算法，非常适合挖掘大型高维数据集。该算法可以生成高质量的集群，而无需依赖用户定义的参数。

O-Cluster的目标是识别数据中的高密度区域，并将密集区域分成簇。它使用轴平行的单维（正交）数据投影来识别密度区域。该算法寻找分裂点，这些分裂点导致不重叠并且大小平衡的不同群集。

O-Cluster通过创建二叉树层次结构来递归操作。叶簇的数量是自动确定的。可以将算法配置为限制最大簇数。

22.1.1分区策略

分区策略是指在属性直方图中发现密度区域的过程。该过程因数字和分类数据而异。当两者都存在于数据中时，算法单独执行搜索，然后比较结果。

在选择分区时，算法平衡了两个目标：找到分离良好的集群，并创建大小平衡的集群。以下段落详细说明了如何识别数字和分类属性的分区。

22.1.1.1分区数值属性

为了找到最有效的切割平面，O-Cluster在属性直方图中搜索高密度（峰值）箱之间的低密度（谷值）箱。O-Cluster试图找到一对在它们之间具有谷的峰，其中峰和谷直方图计数之间的差异在统计上是显着的。

灵敏度级别参数指定可被视为峰值的最低密度。灵敏度是数字数据的可选参数。它可用于过滤分裂点候选者。

22.1.1.2对分类属性进行分区

分类值没有与之关联的内在顺序。因此，不可能应用用于分割数值的直方图峰值和谷值的概念。

相反，单个值的计数形成直方图。具有大量计数的箱被解释为具有高密度的区域。聚类目标是分离这些高密度区域并有效地降低数据的熵（随机性）。

O-Cluster识别沿各个投影具有最高熵的直方图。熵是以高于灵敏度水平的箱数来衡量的。O-Cluster将两个最大的bin放入单独的分区，从而创建一个拆分谓词。其余的箱子被随机分配给两个结果分区。

22.1.2主动采样

O-Cluster算法在有限大小的数据缓冲区上运行。它使用了主动采样机制来处理不适合内存的数据集。

在处理初始随机样本之后，O-Cluster识别不再感兴趣的案例。这种情况属于冻结分区，进一步分裂的可能性很小。这些情况被来自模糊区域的示例替换，其中需要进一步的信息（附加情况）来找到良好的分裂平面并继续分区。如果只能在较低置信度下找到有效分割，则分区被认为是不明确的。

与冻结分区相关联的案例被标记为从缓冲区中删除。它们被属于模糊分区的案例所取代。更新模糊分区的直方图并重新评估分裂点。

22.1.3流程

O-Cluster算法评估分区中所有投影的可能分裂点，选择最佳分割点，并将数据分成两个新分区。该算法通过在新创建的分区内搜索好的切割平面来进行。因此，O-Cluster创建二叉树结构，将输入空间划分为没有重叠或间隙的矩形区域。

主要处理阶段是：

  1. 加载缓冲区。将所有案例从初始缓冲区分配到单个活动根分区。

  2. 沿每个活动分区的正交单维投影计算直方图。

  3. 找到活动分区的最佳分裂点。

  4. 标记模糊和冻结分区。

  5. 如果存在有效分隔符，请将活动分区拆分为两个新的活动分区，然后从步骤2开始。

  6. 在当前缓冲区上的所有递归分区完成后重新加载缓冲区。继续加载缓冲区，直到再次填充缓冲区，或者达到数据集的末尾，或者直到案例数等于数据缓冲区大小。

    注意：

    O-Cluster最多需要一次通过数据

22.1.4评分

发现的集群 O-Cluster用于生成可用于对新数据进行评分的贝叶斯概率模型。生成的概率模型是混合模型，其中混合分量由数值属性的独立正态分布和分类属性的多项分布的乘积表示。

23.1关于O-Cluster

O-Cluster是一种快速，可扩展的基于网格的聚类算法，非常适合挖掘大型高维数据集。该算法可以生成高质量的集群，而无需依赖用户定义的参数。

O-Cluster的目标是识别数据中的高密度区域，并将密集区域分成簇。它使用轴平行的单维（正交）数据投影来识别密度区域。该算法寻找分裂点，这些分裂点导致不重叠并且大小平衡的不同群集。

O-Cluster通过创建二叉树层次结构来递归操作。叶簇的数量是自动确定的。可以将算法配置为限制最大簇数。

23.1.1分区策略

分区策略是指在属性直方图中发现密度区域的过程。该过程因数字和分类数据而异。当两者都存在于数据中时，算法单独执行搜索，然后比较结果。

在选择分区时，算法平衡了两个目标：找到分离良好的集群，并创建大小平衡的集群。以下段落详细说明了如何识别数字和分类属性的分区。

23.1.1.1分区数值属性

为了找到最有效的切割平面，O-Cluster在属性直方图中搜索高密度（峰值）箱之间的低密度（谷值）箱。O-Cluster试图找到一对在它们之间具有谷的峰，其中峰和谷直方图计数之间的差异在统计上是显着的。

灵敏度级别参数指定可被视为峰值的最低密度。灵敏度是数字数据的可选参数。它可用于过滤分裂点候选者。

23.1.1.2对分类属性进行分区

分类值没有与之关联的内在顺序。因此，不可能应用用于分割数值的直方图峰值和谷值的概念。

相反，单个值的计数形成直方图。具有大量计数的箱被解释为具有高密度的区域。聚类目标是分离这些高密度区域并有效地降低数据的熵（随机性）。

O-Cluster识别沿各个投影具有最高熵的直方图。熵是以高于灵敏度水平的箱数来衡量的。O-Cluster将两个最大的bin放入单独的分区，从而创建一个拆分谓词。其余的箱子被随机分配给两个结果分区。

23.1.2主动采样

O-Cluster算法在有限大小的数据缓冲区上运行。它使用了主动采样机制来处理不适合内存的数据集。

在处理初始随机样本之后，O-Cluster识别不再感兴趣的案例。这种情况属于冻结分区，进一步分裂的可能性很小。这些情况被来自模糊区域的示例替换，其中需要进一步的信息（附加情况）来找到良好的分裂平面并继续分区。如果只能在较低置信度下找到有效分割，则分区被认为是不明确的。

与冻结分区相关联的案例被标记为从缓冲区中删除。它们被属于模糊分区的案例所取代。更新模糊分区的直方图并重新评估分裂点。

23.1.3流程

O-Cluster算法评估分区中所有投影的可能分裂点，选择最佳分割点，并将数据分成两个新分区。该算法通过在新创建的分区内搜索好的切割平面来进行。因此，O-Cluster创建二叉树结构，将输入空间划分为没有重叠或间隙的矩形区域。

主要处理阶段是：

  1. 加载缓冲区。将所有案例从初始缓冲区分配到单个活动根分区。

  2. 沿每个活动分区的正交单维投影计算直方图。

  3. 找到活动分区的最佳分裂点。

  4. 标记模糊和冻结分区。

  5. 如果存在有效分隔符，请将活动分区拆分为两个新的活动分区，然后从步骤2开始。

  6. 在当前缓冲区上的所有递归分区完成后重新加载缓冲区。继续加载缓冲区，直到再次填充缓冲区，或者达到数据集的末尾，或者直到案例数等于数据缓冲区大小。

    注意：

    O-Cluster最多需要一次通过数据

23.1.4评分

发现的集群 O-Cluster用于生成可用于对新数据进行评分的贝叶斯概率模型。生成的概率模型是混合模型，其中混合分量由数值属性的独立正态分布和分类属性的多项分布的乘积表示。

3.2关于属性

属性是数据挖掘中使用的数据项。在预测模型中， 属性是影响给定结果的预测因子。在描述性模型中，属性是为自然分组或关联分析的信息项。例如，员工数据表包含职位，雇用日期，工资，年龄，性别等属性。

3.2.1数据属性和模型属性

数据属性是用于构建，测试或评分模型的数据集中的列。模型属性是模型内部使用的数据表示。

数据属性和模型属性可以相同。例如，名为SIZE的列，其值为S ， M和L ，是算法用于构建模型的属性。在内部，模型属性SIZE很可能与从中派生它的数据属性相同。

另一方面， SALES_PROD产品的销售数字的嵌套列SALES_PROD与模型属性不对应。data属性可以是SALES_PROD ，但每个产品及其相应的销售数字（嵌套列中的每一行）都是一个模型属性。

转换还会导致数据属性和模型属性之间的差异。例如，转换可以将计算应用于两个数据属性，并将结果存储在新属性中。new属性是没有相应数据属性的模型属性。其他转换（例如分箱，归一化和异常值处理）会导致模型的属性表示与案例表中的数据属性不同。

3.2.3数字，分类和非结构化文本

解释数字，分类和非结构化文本属性。

模型属性是数值， 分类，或非结构化的（ 文本）。数据属性（案例表中的列）具有Oracle数据类型，如“列数据类型”中所述 。

数字的理论上，属性可以具有无限数量的值。值具有隐式顺序，它们之间的差异也是有序的。Oracle Data Mining将NUMBER ， FLOAT ， BINARY_DOUBLE ， BINARY_FLOAT ， DM_NESTED_NUMERICALS ， DM_NESTED_BINARY_DOUBLES和DM_NESTED_BINARY_FLOATS为数字。

明确的属性具有标识有限数量的离散类别或类的值。没有与值相关联的隐式顺序。一些分类是二元的：它们只有两个可能的值，例如是或否，或者是男性或女性。其他分类是多类的：它们具有两个以上的值，例如小，中和大。

默认情况下，Oracle Data Mining将CHAR和VARCHAR2解释为分类，但是这些列也可以标识为非结构化数据列（ 文本）。Oracle Data Mining将DM_NESTED_CATEGORICALS列解释为分类。CLOB ， BLOB和BFILE始终包含非结构化数据。

该分类模型的目标是分类的。（如果分类模型的目标是数字，则将其解释为分类。）回归模型的目标是数字。属性重要性模型的目标是分类或数字。

3.2.4模型签名

该 model signature是用于构建模型的数据属性集。签名中的部分或全部属性必须存在才能进行评分。该模型以尽力而为的方式考虑任何缺失的列。如果存在具有相同名称但数据类型不同的列，则模型会尝试转换数据类型。如果存在额外的未使用的列，则忽略它们。

模型签名不一定包括构建数据中的所有列。特定于算法的标准可能导致模型忽略某些列。其他列可以通过转换来消除。签名中仅包含实际用于构建模型的数据属性。

该目标和案例ID列不包含在签名中。

3.2.6模型细节

模型细节揭示了有关模型属性及其算法处理的信息。Oracle建议用户利用相应算法的模型详细信息视图。

转型与反向变换表达式与模型属性相关联。在创建模型的算法处理之前，将转换应用于数据属性。在构建模型之后，将反向变换应用于模型属性，以便模型详细信息以原始数据属性的形式表示，或尽可能接近它。

反向转换支持模型透明度。它们提供了算法在内部使用的数据视图，但其格式对用户有意义。
