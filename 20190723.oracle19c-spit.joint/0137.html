B UTL_RAW包

使用Oracle Visual Workbench开发通过网关访问WebSphere MQ的应用程序。Oracle Visual Workbench定义了一个用于访问WebSphere MQ的接口，并自动为Oracle应用程序生成PL / SQL代码（MIP）以与网关进行交互。有关Oracle Visual Workbench的更多信息，请参阅适用于WebSphere MQ安装的Oracle Procedural Gateway Visual Workbench和适用于Microsoft Windows的用户指南（32位） 。

消息数据类型

发送到WebSphere MQ队列或从WebSphere MQ队列检索的消息由MIP过程作为无类型数据传输。

在数据配置文件中定义时 MIP，MIP将消息数据从Oracle数据类型转换为接收应用程序理解的目标数据类型。消息数据在发送到WebSphere MQ队列之前打包到RAW数据类型的缓冲区中。接收消息时应用相同的转换过程。MIP从缓冲区解包消息并将其转换为指定的Oracle数据类型。

该 MIP使用UTL_RAW包的功能来执行消息数据转换。UTL_RAW包是一个PL / SQL包，其中包含使用RAW数据类型和PL / SQL数据类型转换和打包通过WebSphere MQ队列来回发送的消息数据的过程。

必要时，可以使用UTL_RAW函数增强生成的MIP中的消息数据转换。如果MIP中没有定义数据配置文件，您可以使用UTL_RAW函数创建自己的数据转换过程，在发送消息之前和收到消息后立即调用这些函数。

UTL_RAW包不包含在网关中。它随每个Oracle数据库一起提供。有关安装UTL_RAW软件包的信息，请参阅Oracle DBA。

UTL_RAW.OVERLAY

UTL_RAW 。 OVERLAY用新的原始数据字符串替换原始值的一部分。如果新数据短于覆盖区域的长度，则填充新数据以使其足够长。如果新数据长于覆盖区域，则忽略额外字节。如果指定的覆盖区域超出输入值的长度，则根据指定的长度扩展输入值。如果指定覆盖区域的起始位置超过输入值的长度，则将输入值填充到指定的位置，然后使用新数据进一步扩展输入值。

A VALUE_ERROR引发异常任何下列条件：

  • 用于覆盖输入值的新数据为null或长度为0

  • 未定义要覆盖的输入值部分

  • 要覆盖的部分的长度超过32 767字节

  • 要覆盖的字节数定义为小于0

  • 用于开始叠加操作的输入值内的位置定义为小于1

句法

结果 ：= UTL_RAW.OVERLAY（ new_bytes ， 输入 ， 位置 ， 长度 ， 填充 ）;

哪里：

  • result是保存函数输出值的变量。它是RAW数据类型。

  • new_bytes是新值， RAW数据类型的字节字符串，用于覆盖输入。从最左边的字节开始，从new_bytes中选择字节。

  • input是要覆盖的数据类型RAW的输入值。

  • position是输入中的位置，从1开始编号，开始重叠。该值必须大于0。默认值为1。

  • length是要覆盖的字节数。这必须大于或等于0。默认值是new_bytes的长度。

  • pad是一个单字节值，用于在长度超过覆盖长度或位置超过input长度时进行填充。默认值为X'00'。

UTL_RAW.SUBSTR

UTL_RAW 。 SUBSTR从原始值中删除字节。如果指定正数作为要删除的字节的起始点，则SUBSTR从输入值的开头开始计数以查找第一个字节。如果指定负数， UTL_RAW 。 SUBSTR从输入值的末尾向后计数以找到第一个字节。

A VALUE_ERROR引发异常任何下列条件：

  • 开始删除的位置指定为0

  • 要删除的字节数指定为小于0

句法

result ：= UTL_RAW.SUBSTR（ input ， position [， length ]）;

哪里：

  • result是保存函数输出值的变量。它是RAW数据类型。该值是在指定的字节或字节从input ，或该值是空值，如果input是空的。

  • input是RAW数据类型的输入值，从中提取其部分字节。

  • position是从中开始提取的字节位置。该值不能为0。如果position值为负，则SUBSTR从输入结束向后计数。

  • length是position后从input提取的字节数。该值必须大于0。未指定时，将返回输入结尾的所有字节。

UTL_RAW.TRANSLATE

UTL_RAW 。 TRANSLATE根据您指定的方案更改原始值中某些字节的值。将输入值中的字节与匹配的字符串进行比较，当发现匹配时，替换字符串中相同位置的字节将复制到结果中。如果偏移量超过替换字符串的长度，则从结果中省略。输入值中未出现在匹配字符串中的字节将复制到结果值。仅使用匹配字符串中最左边的一个字节，并忽略后续的重复出现。

如果匹配字符串包含的字节数多于替换字符串，则匹配字符串末尾的额外字节在替换字符串中没有对应的字节。输入值中与这些字节匹配的任何字节都将从结果值中省略。

A VALUE_ERROR引发异常任何下列条件：

  • 输入值为null或长度为0

  • 匹配字符串为null或长度为0

  • 替换字符串为null或长度为0

句法

result ：= UTL_RAW.TRANSLATE（ input ， match ， replace_bytes ）;

哪里：

  • result是保存函数输出值的变量。它是数据类型RAW 。

  • input是要更改的数据类型RAW的输入值。

  • match指定要在input搜索的字节0codes并更改为replace_bytes 。它是数据类型RAW 。

  • replace_bytes指定替换match指定的代码的字节代码。它是数据类型RAW 。

UTL_RAW.TRANSLITERATE

UTL_RAW 。 TRANSLITERATE使用替换字符串的相应字节替换匹配字符串中所有出现的任何字节。将输入值中的字节与匹配字符串进行比较，如果未找到它们，则将它们不加改变地复制到结果值。如果找到它们，则它们将在结果值中替换替换字符串中相同偏移量处的字节，或者当偏移量超出替换字符串长度时指定的填充字节。仅使用匹配字符串中最左边的一个字节。后续重复出现将被忽略。UTL_RAW的结果值。 TRANSLITERATE始终与输入值的长度相同。

如果替换字符串比匹配字符串短，则当所选匹配字符串字节在替换字符串中没有对应字节时，填充字节将被放置在结果值中。当输入值为null或长度为0时，将引发VALUE_ERROR异常。

UTL_RAW 。 TRANSLITERATE不同于UTL_RAW 。通过以下方式进行TRANSLATE ：

  • 替换字符串中未定义的输入值中的字节将使用您指定的值进行填充

  • 结果值始终与输入值的长度相同

句法

result ：= UTL_RAW.TRANSLITERATE（ input ， replace_bytes ， match ， pad ）;

哪里：

  • Result是函数的输出值。它是数据类型RAW 。

  • Input是要更改的数据类型RAW的输入值。

  • Replace_bytes指定相应的匹配字节更改的字节代码。该值可以是对RAW数据类型有效的任何长度。默认值为空值，并根据需要使用pad有效地扩展到匹配的长度。

  • Match指定要在input Match的byte代码。该值可以是对RAW数据类型有效的任何长度。默认值为X'00'到X'FF'。

  • Pad是一个单字节值，用于在replace_bytes短于匹配时扩展replace_bytes的长度。默认值为X'00'。

    UTL_RAW 。 TRANSLATE与UTL_RAW不同。 TRANSLITERATE函数有以下几种方式：

    • 用于匹配和替换字符串的原始值没有默认值

    • 在结果值中省略了替换字符串中未定义的输入值中的字节

    • 结果值可以短于输入值

UTL_RAW.XRANGE

UTL_RAW 。 XRANGE 返回包含指定范围内所有有效单字节代码的原始值。如果起始字节值大于结束字节值，则结果字节的连续开始于起始字节，从X'FF'换行到X'00'，并在结束字节处结束。

指定时，起始字节和结束字节的值必须是单字节原始值。

句法

结果 ：= UTL_RAW.XRANGE（ 开始 ， end ）;

哪里：

  • start是一个单字节代码开始。默认值为X'00'。

  • end是以字节end的单字节代码。默认为X'FF'。

1.2为什么使用Oracle Pro * C / C ++预编译器

Oracle Pro * C / C ++预编译器使您可以在应用程序中使用SQL的强大功能和灵活性。方便易用的界面使您的应用程序可以直接访问Oracle。

与许多应用程序开发工具不同，Pro * C / C ++允许您创建高度自定义的应用程序。例如，您可以创建包含最新窗口和鼠标技术的用户界面。您还可以创建在后台运行的应用程序，而无需用户交互。

此外，Pro * C / C ++可帮助您微调应用程序。它允许密切监视资源使用，SQL语句执行和各种运行时指示器。使用此信息，您可以更改程序参数以获得最佳性能。

虽然预编译为应用程序开发过程增加了一个步骤，但它可以节省时间。预编译器（而不是您）将每个嵌入式SQL语句转换为对Oracle运行时库（SQLLIB）的调用。Pro * C / C ++预编译器还分析主机变量，定义结构到列的映射，并且使用SQLCHECK = FULL，对嵌入式SQL语句执行语义分析。

1.3为什么使用SQL

如果要访问和操作Oracle数据，则需要SQL。无论是通过SQL * Plus交互式使用SQL还是嵌入应用程序，都取决于手头的工作。如果作业需要C或C ++的程序处理能力，或者必须定期完成，请使用嵌入式SQL。

SQL已成为首选的数据库语言，因为它灵活，功能强大且易于学习。它是非过程性的，它允许您指定您想要的内容，而无需指定如何操作。一些类似英语的语句可以轻松地一次操作一行或多行Oracle数据。

您可以执行任何SQL（而不是SQL * Plus） 来自应用程序的声明。例如，你可以

  • 动态创建，更改和DROP数据库表

  • SELECT，INSERT，UPDATE和DELETE数据行

  • COMMIT或ROLLBACK事务

在将SQL语句嵌入应用程序之前，可以使用SQL * Plus以交互方式测试它们。通常，从交互式SQL切换到嵌入式SQL只需要进行微小的更改。

1.4为什么使用PL / SQL

作为SQL的扩展，PL / SQL是一种事务处理语言，支持过程构造，变量声明和强大的错误处理。在同一个PL / SQL块中，您可以使用SQL和所有PL / SQL扩展。

嵌入式PL / SQL的主要优点是性能更好。与SQL不同，PL / SQL提供了逻辑分组SQL语句并将其以块的形式发送到Oracle而不是逐个发送到Oracle的能力。这减少了网络流量和处理开销。

1.8.8字符指针有什么特别之处吗？

是。当Pro * C / C ++绑定输入或输出主变量时，它必须知道长度。当您使用VARCHAR [n]或声明char [n]类型的主机变量时，Pro * C / C ++知道声明的长度。但是当你使用字符指针作为主变量，并使用malloc()来定义程序中的缓冲区时，Pro * C / C ++无法知道长度。

在输出时，您不仅必须分配缓冲区，而且必须使用一些非空字符填充它，然后将其终止。在输入或输出时，Pro * C / C ++调用strlen()来获取缓冲区的长度。

在RMAN恢复备份之前，必须满足RESTORE命令的所有规范。除非受DEVICE TYPE子句的限制，否则RESTORE命令将在所有已配置通道的设备类型上搜索备份。如果存储库中没有可用备份满足所有指定条件，则RMAN将返回错误，指示无法还原该文件。

如果仅使用手动分配的通道，则如果在您为其分配通道的介质上没有可用的备份，则备份作业可能会失败。配置自动通道使RMAN更有可能找到并恢复满足指定条件的备份。

14.2.2关于RMAN还原故障转移

RMAN自动使用还原故障转移来跳过损坏或无法访问的备份并查找可用的备份。如果未找到备份或包含损坏的数据，RMAN会自动查找从中还原所需文件的另一个备份。

RMAN生成指示其正在执行的故障转移类型的消息。例如，当RMAN故障转移到同一文件的另一个备份时，它会生成类似于以下内容的消息：

14.2.4关于RMAN还原操作和ASM

使用自动存储管理（ASM）磁盘组时，仅当数据文件的全名（包括化身）与现有数据文件的名称不匹配时，RMAN还原操作才会创建数据文件的新副本。

完全限定的ASM文件名的格式为+diskgroup/dbname/filetype/filetypetag.file.incarnation 。首次还原控制文件然后还原其他数据库文件时，控制文件中的数据文件名可能与现有数据文件的名称不匹配，因此会重新创建数据文件。

使用以下方法之一确保在还原或重复操作期间不重新创建现有数据文件：

  • 在控制文件中，为每个数据文件使用别名。别名不得包含ASM化身编号。

  • 还原控制文件之后和还原其他数据库文件之前，请使用CATALOG命令确保在还原的控制文件中编目现有数据文件。接下来，使用SWITCH命令使还原的控制文件指向现有数据文件。

  • 在还原数据文件之前和还原控制文件之后，使用SET NEWNAME重命名数据文件。

14.2.5关于RMAN还原优化

RMAN使用还原优化来避免在可能的情况下从备份还原数据文件。如果数据文件存在于正确的位置且其标头包含预期信息，则RMAN不会从备份还原数据文件。

注意：

还原优化仅检查数据文件头。它不会扫描数据文件正文中的已损坏块。

您可以使用RESTORE命令的FORCE选项覆盖此行为并无条件地还原所请求的文件。

当恢复多个数据文件的操作被中断时，恢复优化特别有用。例如，假设除了一个数据文件已恢复之外，完整数据库还原在遇到电源故障之后。如果再次运行相同的RESTORE命令，则RMAN仅还原在上一次尝试期间未还原的单个数据文件。

复制数据库时也使用还原优化。如果副本中的数据文件位于具有正确标头内容的正确位置，则不会复制数据文件。与RESTORE不同， DUPLICATE不支持FORCE选项。要强制RMAN复制由于还原优化而跳过的数据文件，请在运行DUPLICATE命令之前从副本中删除数据文件。

14.3.2关于数据库化身

只要使用RESETLOGS选项打开数据库，就会创建数据库化身。

完全恢复后，您可以在没有OPEN RESETLOGS情况下恢复正常操作。但是，在使用备份控制文件进行DBPITR或恢复之后，必须使用RESETLOGS选项打开数据库，从而创建数据库的新版本。当两个不同的重做流具有相同的SCN但是在不同的时间发生时，数据库需要新的化身以避免混淆。如果您将错误的重做应用于您的数据库，那么您就会破坏它。

单个数据库的多个化身的存在决定了RMAN如何处理不在当前化身路径中的备份。通常，当前的数据库化身是正确使用的。然而，在某些情况下，将数据库重置为先前的化身是最好的方法。例如，您可能不满意已执行的时间点恢复的结果，并希望将数据库返回到RESETLOGS之前的时间。了解数据库化身有助于为这种情况做好准备。

14.3.2.1关于RMAN OPEN RESETLOGS操作

使用RESETLOGS选项打开数据库时，RMAN会执行某些操作。

执行的操作如下：

  • 归档当前的联机重做日志（如果可访问），然后擦除联机重做日志的内容并将日志序列号重置为1。

    例如，如果当前打开RESETLOGS ，当前联机重做日志为序列1000和1001，则数据库归档日志1000和1001，然后将联机重做日志重置为序列1和2。

  • 如果当前不存在，则创建联机重做日志文件。

  • 将控制文件中的重做线程记录和联机重做日志记录初始化为新数据库化身的开头。

    更具体地说，数据库将重做线程状态设置为关闭，将重做线程记录中的当前线程序列设置为1，将每个重做线程的线程检查点设置为日志序列1的开头，从每个线程中选择一个重做日志，将其序列初始化为1，依此类推。

  • 使用新的RESETLOGS SCN和时间戳更新所有当前数据文件和联机重做日志以及所有后续归档重做日志。

由于数据库不会将归档重做日志应用于数据文件，除非RESETLOGS SCN和时间戳匹配，因此RESETLOGS要求可防止您使用不是来自当前化身的直接父化身的归档日志来破坏数据文件。在下一节中将更全面地解释化身之间的关系。

在以前的版本中，建议您在OPEN RESETLOGS之后立即备份数据库。因为您现在可以像任何其他备份一样轻松恢复预RESETLOGS备份，因此可以选择新的数据库备份。要通过RESETLOGS执行恢复，必须在最近一次备份后生成所有归档日志，并且必须至少有一个控制文件（当前，备份或创建）。

14.3.2.2数据库化身之间的关系

数据库化身可以相互建立以下关系：

  • 当前的化身是数据库当前正在运行的化身 。

  • 在OPEN RESETLOGS操作之后当前化身分支的化身是当前化身的父母化身。

  • 父母化身的父母是祖先的化身 。任何祖先化身的父母也是当前化身的祖先。

  • 当前化身的直接祖先路径始于最早的化身，并且仅包括当前化身的祖先，父母的化身或当前的化身。

化身编号用于唯一标记和标识重做流。图14-1说明了一个经历了几个化身的数据库，每个化身都有不同的化身编号。

14.3.2.3关于PDB的化身

可插拔数据库（PDB）版本是多租户容器数据库（CDB）的子实现，表示为( database_incarnation, pdb_incarnation ) 。

14.3.2.4关于孤立备份

当数据库经历多个化身时，某些备份可能成为孤立备份。孤立备份是在数据库的化身期间创建的备份，这些备份不在直接祖先路径中。

假设场景如图14-1所示。如果化身3是当前的化身，则以下备份是孤立的：

  • SCN 1000之后的所有备份都来自化身1

  • SCN 2000之后的所有备份来自化身2

相反，以下备份不是孤立的，因为它们位于直接祖先路径中：

  • 所有来自SCN 1000之前的化身1的备份

  • 来自SCN 2000之前的化身2的所有备份

  • 所有来自化身的备份3

当您打算将数据库还原到不在直接祖先路径中的SCN时，可以使用孤立备份。如果从最早的备份到要恢复的点存在连续的归档日志路径，则RMAN可以从父级和祖先的版本恢复备份并恢复到当前时间，即使是在OPEN RESETLOGS操作中也是如此。如果从尚未放弃备份中表示的更改的化身还原控制文件，则RMAN还可以还原和恢复孤立备份。

14.3.2.5关于孤立的PDB备份

当您在可插拔数据库（PDB）上执行时间点恢复，然后使用RESETLOGS选项打开PDB时，可能会导致孤立PDB备份。

将PDB恢复到指定的时间点后，当您使用RESETLOGS选项打开PDB时，将创建PDB的新版本。孤立PDB备份是在SCN或时间值介于SCN或恢复PDB的时间与SCN或在RESETLOGS模式下打开PDB的时间之间创建的备份。V$PDB_INCARNATION视图的END_RESETLOGS_SCN列包含在RESETLOGS模式下打开PDB的SCN。

21.8重启后OSWatcher参数不同或意外不同

当Oracle跟踪文件分析器管理OSWatcher时，在安装或重新启动后，OSWatcher将作为非特权用户启动，例如：

  • Oracle RAC系统上的grid

  • oracle对非Oracle RAC系统

Oracle建议不要以root身份停止并重新启动OSWatcher。

例如：

tfactl oswbb stop tfactl start oswbb 20 72（间隔20秒，保留72小时）

然后OSWatcher以root身份运行，直到它停止并以oracle或grid重新启动，或者重新启动。在任何一种情况下，参数都保存在属性文件中。除非为间隔和保留期指定了其他参数，否则将使用OSWatcher默认值（30,48）。从Oracle Trace File Analyzer版本12.1.2.5.2开始，为每个用户维护一个OSWatcher属性文件。每次启动OSWatcher时，间隔或保留时间的参数都会对该用户保持不变。在早期版本中，如果OSWatcher启动参数与预期不同，那么这是因为OSWatcher已停止并以root用不同参数启动。这些设置将在重新启动后保持不变，因为只有一个属性文件。

在12.1.2.5.2及更高版本中，如果有重新启动，则必须始终使用oracle或grid属性中的参数启动OSWatcher。如果OSWatcher停止并在重新启动之前以root用不同参数重新启动，则OSWatcher启动参数会有所不同。从root属性获取的参数在重新引导后不得生效。参数必须恢复为oracle属性的参数。

参数不同，持久性设置也会更改，因为Oracle支持人员会推荐不同的设置来调查问题。在这种情况下，停止并以正常参数重新启动OSWatcher作为非特权用户。

注意：

如果OSWatcher已安装并正在运行，并且未由Oracle跟踪文件分析器管理，则Oracle跟踪文件分析器将遵循该安装和参数。列出oswbb工具状态时，状态必须为NOT RUNNING ，即不由Oracle Trace File Analyzer管理。

5使用远程同步实例

Oracle Data Guard远程同步实例是远程Oracle Data Guard目标，它接受来自主数据库的重做，然后将该重做发送到Oracle Data Guard配置的其他成员。

远程同步实例管理控制文件，将重做接收到备用重做日志（SRL），并将这些SRL归档到本地归档重做日志，但这是与备用数据库的相似性结束的地方。远程同步实例没有用户数据文件，无法打开访问权限，无法运行重做应用，并且永远不能在主要角色中运行或转换为任何类型的备用数据库。

远程同步实例是Oracle Active Data Guard远程同步功能的一部分，该功能需要Oracle Active Data Guard许可证。

远程同步实例占用的磁盘和处理资源非常少，但却能够在没有数据丢失的情况下故障转移到终端目标，并且可以卸载其他类型开销的主数据库（例如，重做传输）。

在为远程同步实例提供服务时，也可以使用主服务器在为典型备用目标服务时可用的所有重做传输选项。在维护终端目标时，所有重做传输选项都可用于远程同步实例（例如，执行重做传输压缩，如果您拥有Oracle Advanced Compression选项的许可证）。

许多配置都使用异步传输将主数据库重新发送到备用数据库，但有可能在故障转移时丢失一些数据。使用同步重做传输实现零数据丢失可能不是一个可行的选择，因为由于两个数据库之间的网络延迟而对主要的提交响应时间产生影响。

创建靠近主服务器的远程同步实例的好处是可以将提交响应时间的影响降至可接受的阈值（由于主同步和远程同步实例之间的网络延迟较小），同时允许更高的数据保护保证 - 如果主要是失败，并假设远程同步实例在失败时同步，远程同步实例和终端备用数据库将协调从远程同步实例到备用数据库的最终重做装运，以便发送任何尚未提供给备用数据库的重做然后执行零数据丢失故障转移。

5.1创建远程同步实例

创建远程同步实例类似于创建物理备用数据库，但远程同步实例中不存在数据文件。

因此，在远程同步实例上，无需复制数据文件或从备份还原数据文件。创建远程同步实例后，将修改配置以在最高可用性模式下从主数据库同步向重同步实例发送重做，然后远程同步实例将实时异步转发重做。最后，原始异步备用（称为终端备用）配置为在远程同步实例的通信中断的情况下充当远同步实例的备用。

注意：

在包含远程同步实例的配置中，主数据库和远程备用数据库之间仍必须存在直接网络连接。主数据库和远程备用数据库之间的直接连接用于执行运行状况检查和切换处理任务。它不用于重做传输，除非备用数据库已配置为备用目标，以防远程同步实例失败并且没有配置备用远程同步来维护保护级别。

5.2替代目的地

执行创建和配置远程同步实例中的步骤后，远程同步实例通过将重做转发到WAN上的远程站点的终端备用数据库，为配置提供零数据丢失功能。要使配置在远程同步实例无法访问的情况下保持受保护，您必须配置备用数据库的备用重做传输路径。这是使用LOG_ARCHIVE_DEST_ n参数的GROUP和PRIORITY属性完成的。（从Oracle Database 12 c第2版（12.2.0.1）开始， GROUP和PRIORITY属性已替换远程重做目标的ALTERNATE属性。）

使用日志归档目标组的概念增加了可能的备用远程目标的数量。日志归档目标组指定多个归档目标，可用于从远程同步实例或通过级联将重做分发到多个目标。然后可以对组中的目标进行优先级排序，以便主数据库上一次只有一个目标处于活动状态。如果活动目标不可用，则其他目标可用于激活。要扩展数据库的可能存档目标数，可以指定多个组。

5.2.1将日志归档目标分配给组

使用LOG_ARCHIVE_DEST_ n初始化参数的GROUP属性将日志归档目标分配给组。

如果使用日志归档目标组，则只要组中至少有一个目标仍然可用，则至少有一个目标保持启用且处于活动状态。未分配给组的日志归档目标的行为与Oracle Database 12 c第2版（12.2.0.1）之前的日志归档目标的行为相同。

一个组中最多可以有30个日志归档目标。日志归档目标组由其组编号引用，该组编号在创建组时分配。组的编号从1到8。日志归档目标组包含一组远程（ SERVICE= ...）目标。（日志归档目标组中不支持本地归档（ LOCATION= ...）目标，并且必须使用ALTERNATE属性作为备用本地归档位置。见ALTERNATE 。

组中的一个日志归档目标始终处于活动状态，而其他日志归档目标可在活动日志归档目标发生故障时使用。当失败的目标再次变为可用时，如果当前活动目标失败，则它将变为合格，但它不会立即生效，除非所有其他组成员也不可用。例如，以下声明可用于将三个远程同步实例指定为同一组的成员并具有相同的优先级（组中的优先级将在下一节中介绍）。这些是示例参数定义，不包含所有必需的属性。不要逐字使用它们。在此示例中，只有第一个目标处于活动状态，第二个目标可用于在目标1变为不可用时接管。

LOG_ARCHIVE_DEST_2 ='SERVICE = chicagoFS SYNC VALID_FOR =（ONLINE_LOGFILES，PRIMARY_ROLE）GROUP = 1'LOG_ARCHIVE_DEST_STATE_2 = ENABLE LOG_ARCHIVE_DEST_3 ='SERVICE = chicagoFS1 SYNC VALID_FOR =（ONLINE_LOGFILES，PRIMARY_ROLE）GROUP = 1'LOG_ARCHIVE_DEST_STATE_3 = ALTERNATE

注意：

由于日志归档目标组将替换LOG_ARCHIVE_DEST_ n ALTERNATE属性，因此不允许将ALTERNATE属性与不在默认组（其中GROUP指定为1到8）中的日志归档目标一起使用。

5.2.2 为组中的日志归档目标分配优先级

使用LOG_ARCHIVE_DEST_ n初始化参数的PRIORITY属性在日志归档目标组中分配目标首选项，可以控制故障恢复机制，尤其是对于组内的多个成员。

在上一节中，两个远程同步实例目标没有优先级，这意味着当在第一个目标发生故障后激活备用目标时，它将保持为活动目标，直到它失败。优先级用于确定在启动数据库或远程同步实例或目标失败时组中哪个日志归档目标处于活动状态。在以下情况下，日志归档目标将变为活动状态：主数据库以读/写模式打开，安装远程同步实例，或者以只读模式挂载或打开备用数据库。可以将相同的优先级值分配给组中的多个日志归档目标。优先级值是1到8范围内的整数。数字越低表示优先级越高。默认优先级为1（最高优先级）。

当先前失败的目的地再次可用时，优先级将发挥作用。默认情况下，分配给同一组的一组日志归档目标具有相同的优先级。因此，如果一个目标失败，则会对该集的另一个成员进行故障转移。当失败的目标再次可用时，它不会成为活动目标，因为两个目标具有相同的优先级。如果第二个目标在第一个目标再次可用后失败，则数据库将以相同的优先级故障转移到第一个目标或组中的另一个目标。如果在活动目标失败之前另一个目标始终可用，则此循环可无限重复。

继续前面的示例，可以将优先级添加到日志归档目标，以控制目标何时变为活动状态。在以下示例中，添加了第三个远程同步实例，但优先级较低：

LOG_ARCHIVE_DEST_2 ='SERVICE = chicagoFS SYNC VALID_FOR =（ONLINE_LOGFILES，PRIMARY_ROLE）GROUP = 1 PRIORITY = 1'LOG_ARCHIVE_DEST_STATE_2 = ENABLE LOG_ARCHIVE_DEST_3 ='SERVICE = chicagoFS1 SYNC VALID_FOR =（ONLINE_LOGFILES，PRIMARY_ROLE）GROUP = 1 PRIORITY = 1'LOG_ARCHIVE_DEST_STATE_3 = ALTERNATE LOG_ARCHIVE_DEST_4 = 'SERVICE = chicagoFS2 ASYNC VALID_FOR =（ONLINE_LOGFILES，PRIMARY_ROLE）GROUP = 1 PRIORITY = 2'LOG_ARCHIVE_DEST_STATE_4 = ALTERNATE

此声明会导致以下行为：

  • 主要船舶重做到两个首选远程同步实例中的第一个，即chicagoFS

  • 如果chicagoFS变得不可用，那么主要运送到chicagoFS1 。

  • 如果chicagoFS再次可用，则不会发生故障恢复。它成为chicagoFS1的替代品，因为优先级相同。

  • 如果chicagoFS和chicagoFS1变得不可用，则主要发送到chicagoFS2 （在这种情况下通过ASYNC重做传输模式）。

  • 如果在将主数据库发送到chicagoFS2时chicagoFS或chicagoFS1可用，则主数据库将故障回复到该可用的首选日志存档目标。

5.2.3 运送到组中的多个活动目的地

您还可以使用PRIORITY属性配置组，以便在首选目标失败时将其发送到多个目标。

支持单个组内多个活动目标的机制是，定义最低优先级（ PRIORITY=8 ）以激活该组中该优先级的目标，通常用于将重做直接发送到目标备用数据库。以下日志归档目标声明显示了如何配置它。在此示例中，有一个远程同步实例将重做转发到两个终端备用数据库：

LOG_ARCHIVE_DEST_2 ='SERVICE = chicagoFS SYNC VALID_FOR =（ONLINE_LOGFILES，PRIMARY_ROLE）GROUP = 1 PRIORITY = 1'LOG_ARCHIVE_DEST_STATE_2 = ENABLE LOG_ARCHIVE_DEST_3 ='SERVICE = boston ASYNC VALID_FOR =（ONLINE_LOGFILES，PRIMARY_ROLE）GROUP = 1 PRIORITY = 8'LOG_ARCHIVE_DEST_STATE_3 = ALTERNATE LOG_ARCHIVE_DEST_4 = 'SERVICE = newyork ASYNC VALID_FOR =（ONLINE_LOGFILES，PRIMARY_ROLE）GROUP = 1 PRIORITY = 8'LOG_ARCHIVE_DEST_STATE_4 = ALTERNATE

此声明会导致以下行为：

  • 主要船舶重做到首选的远程同步实例chicagoFS 。

  • 如果chicagoFS不可用，则主要服务器以ASYNC模式直接发送到终端备用服务器boston和newyork 。

  • 虽然运到boston和newyork ，如果chicagoFS变为可用，则主站直接运到boston和newyork ，并开始，而不是运到chicagoFS 。

5.2.4 使用多个日志归档目标组

多个日志归档目标组可用于特定于站点的高可用性注意事项，也可用于通过大型级联（读取器场）配置分发服务。

下面的声明建立与多个日志归档的目的地组chicagoFS和chicagoFS1在组1和chicagoFS3和chicagoFS4在组2：

如果首选远程同步实例由于某种原因失败，则配置备用远程同步实例会将配置的保护级别保持在配置的最高可用性保护级别。

在上述两个示例中，配置的保护级别将超出最大可用性，因为重做不再以SYNC模式运送。为了更好地防止系统或网络故障，可以配置一个额外的远程同步实例，为活动远程同步实例提供高可用性。在此配置中，一个是首选的活动远程同步实例，另一个是备用远程同步实例。

如果主服务器在首选远程同步实例中检测到故障，则主服务器会自动开始发送到备用远程同步实例。在这些类型的配置中，主数据库仅使用一个远程同步实例在任何给定时间重新分配重做。

要保持最高可用性保护级别，请在主数据库附近配置两个远程同步实例，并将它们设置为相互保护。然后，如果活动远程同步实例变得不可用，则主数据库可以自动开始以同步模式将重做发送到备用远程同步实例，从而保持最高可用性的提升保护级别。但是在这种情况下，两个远程同步实例具有相同的优先级，当一个远程同步实例接管另一个实例时，它仍然是活动的远程同步实例，直到它失败。为了确保在两个远程同步实例都失败的情况下继续将重做发送到终端备用数据库，终端备用数据库的配置与PRIORITY=2 。（如果有多个终端备用数据库，则为它们使用PRIORITY=8 ）。

5.4远程同步实例支持的保护模式

在最高性能或最高可用性模式下支持远程同步实例。

5.4.1最高可用性模式配置中的远程同步实例

在最大可用性模式下，远程同步实例相对靠近主数据库以最小化网络延迟，而主要服务使用SYNC传输进行远程同步实例。

注意：

在最大可用性模式下，可以将主同步实例和远同步实例分开的距离没有架构限制。实际距离限制取决于给定应用程序对同步配置中网络延迟影响的容忍度。此外，通过使用新的Oracle Data Guard FastSync功能（ SYNC/NOAFFIRM ），可以降低任何给定距离的性能影响。请参阅“ 最高可用性模式下的性能与保护 ” 。

在远程同步实例的主服务器上建立的LOG_ARCHIVE_DEST_ n上支持SYNC/AFFIRM和SYNC/NOAFFIRM语义。有关使用每个模式的权衡信息，请参阅Oracle Data Guard保护模式 。

当主服务器使用SYNC传输服务远程同步实例时，所有提交的重做都驻留在远程同步实例的磁盘上。这允许远程同步实例使用其中一个终端备用目标，以便在主数据库丢失时进行无数据丢失故障转移。

远程同步实例使用ASYNC传输将传入的重做传送到可能更远的终端备用数据库。这样就不会将数据丢失保护扩展到远离主数据库的目的地，以便直接使用SYNC传输进行服务，因为会导致事务吞吐量降低。即使配置中只有一个备用目标，远程同步实例也是有益的。

5.4.2最高性能模式配置中的远程同步实例

在最高性能模式下，主数据库使用ASYNC重做传输为远程同步实例目标服务。

无论主数据实例与远程同步实例之间的物理距离如何都是如此，因为当使用ASYNC传输服务目标时，高网络延迟不会影响事务吞吐量。

在最高性能模式下，远程同步实例可以使管理多个远程目标的Oracle Data Guard配置受益。虽然每个ASYNC目标对主数据库性能的影响几乎为零，但如果有许多远程目标（例如，多个Oracle Active Data Guard备用数据库构成一个读取器服务器场），则效果可以变得可测量。使用远程同步实例时，添加到配置的每个远程目标的增量效果为零。此外，重做传输压缩也可以卸载到远程同步实例。当使用远程同步实例时，主要仅需要为远程同步实例提供服务，然后远程同步实例为其余配置提供服务;目的地越多，性能效益越大。

系统生成的文件名或完全限定的文件名

在Oracle ASM中创建的每个文件都会获得系统生成的文件名，也称为完全限定的文件名。这类似于本地文件系统中的完整路径名。

Oracle ASM根据以下方案生成文件名：

+磁盘组/的databaseName /的fileType / fileTypeTag.fileNumber.incarnation

完全限定文件名的示例如下：

+数据/ ORCL / CONTROLFILE / Current.256.541956473

在以前的完全限定文件名中， data是磁盘组名称， orcl是数据库名称， CONTROLFILE是文件类型，依此类推。

ASMCMD仅支持斜杠（/）。文件名不区分大小写，但具有案例保持性。如果键入路径名称为小写，ASMCMD将保留小写。

有关Oracle ASM文件名的更多信息，请参阅完全限定的文件名表 。

目录

与其他文件系统一样，Oracle ASM目录是文件的容器，Oracle ASM目录可以是其他目录的树结构的一部分。完全限定的文件名表示目录的层次结构，其中加号（+）表示根目录。在每个磁盘组中，Oracle ASM自动创建一个目录层次结构，该层次结构对应于磁盘组中完全限定文件名的结构。此层次结构中的目录称为系统生成的目录。

ASMCMD使您可以使用cd （更改目录）命令在此目录层次结构中上下移动。ASMCMD ls （列表目录）命令列出当前目录的内容，而pwd命令则打印当前目录的名称。

启动ASMCMD时，当前目录设置为root（+）。对于具有两个磁盘组的Oracle ASM实例（例如， data和fra ，输入以根目录作为当前目录的ls命令将生成以下输出：

ASMCMD> ls data / fra /

以下示例演示如何导航Oracle ASM目录树（请参阅前面显示的完全限定文件名）：

ASMCMD> cd + data / orcl / CONTROLFILE ASMCMD> ls Current.256.541956473 Current.257.541956475

您可以使用ASMCMD mkdir命令将自己的目录创建为系统生成的目录的子目录。您创建的目录可以包含子目录，您可以使用cd命令导航系统生成的目录和用户创建的目录的层次结构。

以下示例在磁盘组data中的orcl下创建目录mydir ：

ASMCMD> mkdir + data / orcl / mydir

注意：

目录orcl是系统生成的目录。data内容表示磁盘组data的内容。

如果使用-p标志启动ASMCMD，则ASMCMD会将当前目录显示为其提示的一部分。请参阅在ASMCMD提示中包括当前目录 。 

ASMCMD [+]> cd data / orcl ASMCMD [+ data / orcl]>

ASMCMD保留您输入的目录的大小写。

别号

别名是文件名，它们是系统生成的文件名的引用或指针。但是，别名是用户友好的名称。别名类似于UNIX或Linux计算机中的符号链接。您可以创建别名以简化Oracle ASM文件名管理。您可以使用mkalias ASMCMD命令或SQL ALTER DISKGROUP命令创建别名。

别名至少具有磁盘组名称作为其完整路径的一部分。您可以在磁盘组级别或任何系统生成的或用户创建的子目录中创建别名。以下是别名的示例：

+ data / ctl1.f + data / orcl / ctl1.f + data / mydir / ctl1.f

如果使用-l标志运行ASMCMD ls （列表目录），则会使用别名引用的系统生成的文件列出每个别名。

ctl1.f => + data / orcl / CONTROLFILE / Current.256.541956473

有关别名的更多信息，请参阅Alias Oracle ASM文件名表单 。

绝对路径和相对路径

运行接受文件名或目录名作为参数的ASMCMD命令时，可以将名称用作绝对路径或相对路径。

绝对路径是指文件或目录的完整路径。绝对路径以加号（+）开头，后跟磁盘组名称，后跟目录树中的后续目录。绝对路径包括直到到达感兴趣的文件或目录的目录。完整的系统生成的文件名（也称为完全限定的文件名）是文件的绝对路径的示例。

使用绝对路径使命令可以访问文件或目录，而不管当前目录的设置位置。以下rm命令使用文件名的绝对路径：

ASMCMD [+] > rm +data/orcl/datafile/users.259.555341963

以下cd命令使用目录的绝对路径。

ASMCMD [+ data / mydir]> cd + data / orcl / CONTROLFILE

相对路径仅包括不属于当前目录的文件名或目录名的一部分。也就是说，文件或目录的路径是相对于当前目录的。

在以下示例中， rm命令对文件undotbs1.272.557429239 ，在这种情况下，该文件是相对路径。ASMCMD将当前目录附加到命令参数以获取文件的绝对路径。在此示例中，这是+data/orcl/DATAFILE/undotbs1.272.557429239 。 

ASMCMD [+]> cd + data ASMCMD [+ data]> cd orcl / DATAFILE ASMCMD [+ data / orcl / DATAFILE]> ls EXAMPLE.269.555342243 SYSAUX.257.555341961 SYSTEM.256.555341961 UNDOTBS1.258.555341963 UNDOTBS1.272.557429239 USERS.259.555341963 ASMCMD [+ data / orcl / DATAFILE]> rm undotbs1.272.557429239

目录的路径也可以是相对的。您可以通过为cd命令提供目录参数来上升或下移当前目录树分支的层次结构，该命令的路径是相对于当前目录的。

此外，您可以使用伪目录“ . ”和“ .. ”而不是目录名。“ . ”伪目录是当前目录。“ .. ”伪目录是当前目录的父目录。

以下示例演示如何使用相对目录路径和伪目录：

ASMCMD [+ data / orcl]> cd DATAFILE ASMCMD [+ data / orcl / DATAFILE]> cd ..ASMCMD [+ data / orcl]>

通配符

通配符*和%在绝对或相对路径中的任何位置匹配零个或多个字符，这样可以节省完整目录或文件名的输入。两个通配符的行为完全相同。有各种ASMCMD命令可以接受通配符，例如cd ， du ， find ， ls ， lsattr ， lsdg ， lsdsk ， lsgrp ， lsusr和rm 。

如果在使用带有cd通配符时通配符模式仅匹配一个目录，则cd会将目录更改为该目标。如果通配符模式匹配多个目录，则ASMCMD不会更改目录，而是返回错误。

如果以非交互模式运行ASMCMD命令，则对于大多数操作系统，必须将通配符括在引号中。有关更多信息，请参阅以非交互模式运行ASMCMD 。

  4. 输入exit或quit命令退出ASMCMD。

您可以指定--privilege选项以选择连接类型， SYSASM或SYSDBA 。默认值为SYSASM ，在管理Oracle ASM实例时使用。连接到数据库实例时以SYSDBA身份连接。

例如：

$ asmcmd --privilege sysasm

您可以使用asmcmd命令指定-v选项，以显示一些命令的附加信息，以帮助用户诊断问题，如以下示例所示：

$ asmcmd -v debug

如果在asmcmd操作期间必须记录消息，则会将消息写入alert.log文件或trace.trc文件。

使用-v errors ， -v warnings或-v normal display level启动asmcmd时，将创建或更新alert.log文件。使用-v info或-v debug显示级别启动asmcmd时，将创建或更新trace.trc文件。

默认情况下， alert.log是在$ORACLE_BASE/diag/asmcmd/user_ username / hostname /alert/目录中创建的文件。默认情况下， trace.trc文件在$ORACLE_BASE/diag/asmcmd/user_ username / hostname /trace/目录中创建。

在某些情况下，可以设置$ORACLE_BASE和$ORACLE_HOME来覆盖alert.log和trace.trc文件的默认位置。

$ asmcmd -p ASMCMD [+]> cd data ASMCMD [+ data]>

    13.6 Bfile类

    Bfile类定义BFILE类型的对象的公共属性。 BFILE是存储在Oracle数据库之外的操作系统文件中的大型二进制文件。Bfile对象包含指向BFILE的逻辑指针，而不是BFILE本身。

    Bfile类的方法使您能够执行与Bfile对象相关的特定任务。

    ResultSet和Statement类的方法（如getBfile()和setBfile()使您可以访问SQL BFILE值。

    NULL Bfile对象上唯一有效的方法是setName（） ， isNull（）和operator =（） 。

    Bfile对象可以通过以下方式初始化：

    • setName（）方法。该BFILE然后可以通过将本修改BFILE到表中，然后使用它检索SELECT...FOR UPDATE 。 write()方法修改BFILE ;但是，仅在提交事务时才将刷新的数据刷新到表中。请注意，不需要INSERT操作。

    • 为其分配初始化的Bfile对象。

    9.1检索查询结果

    查询是OLAP Java API Source ，它指定要从数据存储中检索的数据以及希望Oracle OLAP对数据执行的任何计算。Cursor是检索或获取 Source指定的结果集的对象。为Source创建Cursor涉及以下步骤：

    1. 获取的主要Source来自MdmObject或创建派生Source ，通过对业务DataProvider或Source 。有关获取或创建Source对象的信息，请参阅了解源对象 。
    2. 如果Source是派Source ，则提交您在其中创建Source的Transaction 。要提交Transaction ，调用commitCurrentTransaction你的方法TransactionProvider 。有关提交Transaction更多信息，请参阅使用TransactionProvider 。如果Source是主要Source ，那么您不需要提交Transaction 。
    3. 通过调用DataProvider的createCursorManager方法并将该方法传递给Source创建CursorManager 。
    4. 通过调用CursorManager的createCursor方法创建一个Cursor 。

    5.1分片队列

    分片队列会增加enqueue-dequeue吞吐量，尤其是在Oracle RAC实例中，因为允许来自不同队列会话的消息并行出列。队列的每个分片都是根据会话中的排队时间排序的，并且跨分片排序是最好的努力。分片队列自动管理表分区，以便入队者和出队者不会相互竞争。此外，分片队列使用内存中的消息缓存来优化性能并减少排队和出队的磁盘和CPU开销。

    分片队列的优点和权衡包括：

    • 分片队列提供Oracle Real Application Clusters（Oracle RAC）上单个队列的可伸缩性，尤其是在每个用户在每个实例上都有多个dequeuers的情况下。

    • 分片队列将增加内存使用量以获得性能。

    5.1.3分片队列和本机JMS支持

    Sharded队列原生支持：

    • 非耐用订户

    • JMS有效负载

    • 优先级

    分片队列支持持久消息和非持久消息。非持久性消息存储在消息高速缓存内的内存中，不存储在磁盘上。因此，在实例崩溃或关闭时会丢失非持久消息。

    Sharded队列本身支持两种订阅者以满足JMS要求：

    • 非持久订户：只有在订户处于活动状态时发布消息时，这些订户才会收到有关其所选主题的消息。此订阅在不同会话中不可共享。

    • 持久订阅者：这些订阅者接收在主题上发布的所有消息，包括在订阅者处于非活动状态时发布的消息。多个数据库会话可以共享相同的订阅。

    分片队列不使用ADT来存储JMS有效负载。JMS消息存储在数据库的标量列中。JMS消息类型（如TEXT ， BYTES ， MAP ， STREAM和OBJECT根据有效负载大小和类型将JMS有效负载存储在队列表中的标量TEXT / RAW或CLOB/BLOB列中。JMS消息属性存储在队列表的CLOB （SecureFile）列中，并为用户定义的属性定义了属性访问函数。有效负载和用户属性被选入RAW ， VARCHAR2或安全文件列，而不是存储为ADT。 JMS标头属性和JMS提供程序信息存储在它们自己的标量列中。

    分片队列支持0（最低优先级）和9（最高优先级）之间的整数优先级值，默认值为优先级4，由JMS标准定义。

    5.1.4分片队列和分区

    分片队列自动管理用于队列表的基础分区表。这种分区管理可以在前台或后台进行。每个分片都提供排队消息的会话级排序。每个入队会话都分配了一个分片。每个碎片由一系列子碎片组成。每个子硬盘都映射到一个分区。在入队时，消息会自动分配给表分区。

    根据需要自动创建新分区，因为当dequeuers无法跟上入队者时队列表需要增长。当分区中的所有消息都已出列且不再需要时，分区将被截断并重用。消息缓存根据dequeuers的要求自动将消息从分区加载到内存中。不应在分片队列下面的分区表上创建全局索引。通常不建议在分区表上使用本地索引。如果需要这样的索引并导致性能下降，则应考虑非分片队列。

    5.1.5分片队列和Oracle Real Application Clusters（Oracle RAC）

    分片队列自动提供队列会话排序，同时尽可能避免跨实例通信。有时需要跨实例通信。例如，如果分片队列在一个Oracle RAC实例上具有单个入队会话而在另一个实例上具有单个出队会话，则分片队列将在Oracle RAC实例之间转发消息。消息转发对于排队事务是非原子的，以提高性能。如果Dequeuers连接到其分片中没有消息的实例，则可以获得ORA-25228。

    在大多数情况下，请考虑为每个Oracle RAC实例上的每个订户或单个使用者队列设置多个dequeue，以提高吞吐量并减少跨实例开销。本指南的一个例外是当您使用指定单个消息的出列选择器时。如果要通过Oracle RAC数据库中的消息标识符使分片队列中的消息出列，则必须连接到为包含该消息的分片分配出列所有权的实例。否则，该消息将无法用于出列到出队会话。如果所有出列都在单个实例上执行，则消息将自动转发到此实例。因此，对于通过消息ID广泛出列的分片单用户队列，请考虑将分片队列的所有出队会话连接到单个实例。类似地，对于通过消息ID广泛出列的分片多消息队列，考虑让每个用户的所有出队会话连接到单个实例。服务可用于简化将出列会话连接到特定实例。

    5.1.6 Sharded队列限制

    分片队列目前不支持以下Oracle数据库功能：

    • 留言保留

    • 交易分组

    • 不支持匿名发布订阅者通知和OCI回调通知。支持PL / SQL回调通知。

    • 消息网关

    • JMS的Oracle扩展，例如JMS传播和远程订阅者

    • 每个队列表有多个队列。通过CREATE_SHARDED_QUEUE接口创建分片队列。

    • 排序除了消息优先级后跟入队时间（在JMS标准中指定）

    • JDBC thick（OCI）驱动程序。

    • 分片和非分片队列之间的传播

    • 消息转换

    5.1.7 Sharded队列调整

    5.2.1.4在单个非分片队列表运行入队和出队过程兼

    某些环境必须以恒定流程处理消息，要求将进程并行和出列并行。如果消息传递系统只有一个队列表和一个队列，则所有进程必须同时在同一段区域上工作。这在提供大量消息时排除了合理的性能水平。

    并发进程的最佳数量取决于可用的系统资源。例如，在四CPU系统上，从两个并发排队和两个并发排队过程开始是合理的。如果系统无法提供所需数量的消息 ，则使用多个订户进行负载平衡，而不是增加进程数。

    调整队列中的入队和出队率，以便在通常情况下队列大小保持较小且有界。增长和收缩的队列将使索引和IOT失去平衡，这将影响性能。

    对于多用户队列，使用多个订户进行负载平衡而不是增加进程数将减少争用。可以使用多个队列表来获得水平可伸缩性。

    有关调整分片队列的信息，请参阅Sharded Queues Tuning 。

    5.2.1.5在单个非分片队列表中系列地运行入队和出队进程

    当enqueue和dequeue进程串行运行时，同一数据段上的争用低于并发进程的争用。但是，系统传递消息所花费的总时间比它们同时运行的时间长。

    增加进程数有助于排队和出队。当进程数增加时，对于入队者而言，消息吞吐率可能高于出队者，尤其是对于单个消费者队列。多个消费者队列上的出列过程可以更好地扩展。

    5.2.1.7非分片队列的其他技巧

    这些是一些其他持久消息传递基本调优技巧。

    • 确保收集统计信息，以便选择用于检索邮件的最佳查询计划。默认情况下，队列表无法自动收集统计信息。建议的用法是使用代表队列消息加载来收集统计信息并锁定它们。

    • 队列表索引和IOT由AQ后台进程自动合并。但是，如果需要，它们必须继续受到监控和合并。通过自动空间段管理（ASSM），可以将在线收缩操作用于相同目的。均衡的索引可以减少队列监视CPU的消耗，并确保最佳的入队出队性能。

    • 确保运行足够的队列监视器进程以执行后台任务。队列监视器还必须运行其他关键的后台活动。多个qmn进程共享负载;确保有足够的。这些是自动调整的，但如果需要，可以强制为最小数量。

    • 建议等待时间的队列仅用于专用服务器进程。在共享服务器环境中，共享服务器进程专用于呼叫持续时间的出队操作，包括等待时间。许多此类进程的存在可能导致严重的性能和可伸缩性问题，并可能导致共享服务器进程死锁。

    • 长时间运行的出列事务会使队列中的队列争用恶化，必须避免。

    • 将多个使用者队列上的多个出列操作批处理为单个事务可提供最佳吞吐量。

    • 如果不使用消息优先级，则使用NEXT作为导航模式。这提供了相同的语义，但提高了性能。

    • 如果在BROWSE模式下出现，然后是REMOVE则使用REMOVE_NODATA出列模式。

    5.2.2传播调整技巧

    传播可以被认为是一种特殊的出列操作，在远程（或本地）队列表中有一个额外的INSERT 。单个计划的传播不会跨多个作业队列进程并行化。相反，它们是负载平衡的。传播可以被认为是一种特殊的出列操作，在远程（或本地）队列表中有一个额外的INSERT 。单个计划的传播不会跨多个作业队列进程并行化。相反，它们是负载平衡的。

    为了获得更好的可伸缩性，请根据可用的系统资源（CPU）配置传播计划的数量。

    事务性和非事务性 （默认）队列表的传播速率在一定程度上有所不同，因为Oracle数据库高级队列确定非事务性队列的批处理大小，而对于事务性队列，批量大小主要由用户应用程序决定。

    优化传播分批进行。如果远程队列位于不同的数据库中，则Oracle数据库高级队列使用排序算法来避免需要两阶段提交。当必须将消息发送到同一目的地中的多个队列时，会多次发送该消息。如果必须将消息发送到目标位置的同一队列中的多个使用者，则只发送一次。

    在每个伙伴关系和状态表（PST）刷新之后或从磁盘读取PST到内存时计算站点状态。可能的状态包括：

    • UNKNOWN ：这是执行任何检查之前的状态。

    • UNAVAILABLE ：这是站点中所有磁盘都处于脱机状态的状态。

    • COMPROMISED ：当有足够的磁盘或PST副本脱机时的状态，这样该站点就无法再提供数据可用性。

    • AVAILABLE ：当站点内有足够的冗余以提供数据可用性以防其他站点受损或不可用时的状态。

    操作：修改每个指示的服务器类别，使其属性设置为“ACTIVE_CSS_ROLE = hub”。否则，在对使用任何指示的服务器类别的每个服务器池执行以下两个更正操作之一后，删除指示的服务器类别：（1）如果指示了多个类别，则从其SERVER_CATEGORY属性中删除该服务器类别。否则，请删除服务器池。（2）修改服务器池的SERVER_CATEGORY，将指定的服务器类别替换为未配置叶子角色的服务器类别。

    PRVH-00501：ASM过滤器驱动程序发现diskstring“{0}”无效。

    原因： Cluster Verification Utility检查发现指示的ASM筛选器驱动程序发现diskstring包含无效OS路径名的路径名。

    操作：使用命令'asmcmd afd_dsget'获取当前发现字符串。确保使用命令'asmcmd afd_dsset <afd_diskstring>'将发现字符串正确设置为操作系统接受的路径名，然后重试该操作。

    PRVH-00510：ASM实例使用的IPC协议在群集节点上不一致。ASM实例在节点“{3}”上的节点“{1}”和“{2}”协议上使用“{0}”协议。

    原因：群集验证实用程序（CVU）发现ASM实例使用的IPC协议在群集节点中不一致。

    操作：确保ASM实例在所有群集节点上使用相同的协议进行IPC通信。运行命令'make -f $ ORACLE_HOME / rdbms / lib / ins_rdbms.mk ipc_rds ioracle'将协议更改为'rds'或命令'make -f $ ORACLE_HOME / rdbms / lib / ins_rdbms.mk ipc_g ioracle'来更改适用节点上的'udp'协议，使IPC协议在整个群集节点上保持一致。有关更多详细信息，请参阅MOS注释2345406.1。

    PRVH-00511：ASM实例不在节点“{0}”上使用IPC协议“RDS”进行IPC通信。

    原因：群集验证实用程序（CVU）发现ASM实例使用的IPC协议不是指定节点上的RDS。

    操作：通过运行命令'make -f $ ORACLE_HOME / rdbms / lib / ins_rdbms.mk ipc_rds ioracle'，确保ASM实例在指定节点上使用RDS协议进行IPC通信。有关更多详细信息，请参阅MOS注释2345406.1。

    PRVH-00512：ASM实例在节点“{0}”上使用IPC协议“UDP”进行IPC。

    原因：群集验证实用程序（CVU）发现ASM实例使用的进程间通信（IPC）协议不是指定节点上的可靠数据报套接字（RDS）。在Oracle Exadata环境中，用户数据报协议（UDP）链接的Oracle二进制升级失败。

    操作：通过运行命令'make -f $ ORACLE_HOME / rdbms / lib / ins_rdbms.mk ipc_rds ioracle'，确保ASM实例在指定节点上使用IPC的RDS协议。有关更多详细信息，请参阅MOS注释2345406.1。

    PRVH-00513：ASM实例在节点“{0}”上使用IPC协议“RDS”用于IPC。

    原因：群集验证实用程序（CVU）发现ASM实例使用的进程间通信（IPC）协议不是指定节点上的用户数据报协议（UDP）。在非Oracle Exadata环境中，可靠数据报套接字（RDS）链接Oracle二进制升级失败。

    操作：通过运行命令'make -f $ ORACLE_HOME / rdbms / lib / ins_rdbms.mk ipc_g'，确保ASM实例在指定节点上使用UDP协议进行IPC。

    PRVH-00516：尝试设置远程执行框架目录“{0}”时，无法从节点“{1}”检索CVU远程执行框架版本

    原因：无法完成需要远程执行的操作，因为尝试检索CVU远程执行框架版本失败。随附的消息提供详细的故障信息。

    操作：确保执行检查的用户具有足够的权限在指定的目录下执行可执行文件。有关执行失败信息，请参阅随附的错误消息，解决问题并重试操作。
