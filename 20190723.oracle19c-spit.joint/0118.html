260.1 UTL_MAIL安全模型

默认情况下不安装UTL_MAIL ，因为SMTP_OUT_SERVER配置要求及其涉及的安全风险。在安装UTL_MAIL ，您应该采取措施防止SMTP_OUT_SERVER定义的端口被数据传输淹没.

此包现在是一个调用者权限包，并且调用用户将需要在分配给他想要连接的远程网络主机的访问控制列表中授予的连接权限。

注意：

有关管理细粒度访问的详细信息，请参阅“ Oracle数据库安全指南” 。

260.2 UTL_MAIL操作说明

您必须安装UTL_MAIL并定义SMTP_OUT_SERVER.

  • 要安装UTL_MAIL：
    sqlplus sys / <pwd> SQL> @ $ ORACLE_HOME / rdbms / admin / utlmail.sql SQL> @ $ ORACLE_HOME / rdbms / admin / prvtmail.plb

从NLS_LANG返回字符集ID或从NLS_NCHAR返回国家字符集ID。

目的

从NLS_LANG返回字符集ID或从NLS_NCHAR返回国家字符集ID。

参数

val（IN / OUT）

尺寸（IN）

指定给定输出值的大小，该值仅适用于字符串数据。每条信息的最大长度是 OCI_NLS_MAXBUFSZ个字节。对于数字数据，将忽略此参数。

项目（IN）
指定从全球化支持环境变量获取的其中一个值：

OCI_SUCCESS ;或OCI_ERROR 。

什么是流式验证器？

流式验证器使用XML事件，它是XML文档的表示，类似于Simple API for XML（SAX）事件。XML事件具有开始标记，结束标记和注释。生产者驱动SAX事件，消费者驱动XML事件。

流式验证器与较旧的模式验证器共享软件，并从中获取大部分功能。内存开销小于旧验证器中使用的DOM表示。只有一次通过该文件。流式验证器是在Oracle Database 11g第1版（11.1）中引入的。

流验证有两种模式：

  • 透明模式事件将返回给应用程序。

  • 不透明模式事件不会返回到应用程序，但会返回指示文档验证过程成功或失败的错误。

在文档验证之前，必须创建常规验证上下文，并且必须使用此上下文加载相关模式。然后必须创建pull解析器（或另一个事件生成器）的XML事件上下文。然后将该事件上下文提供给流式验证器，以便它可以从生产者请求事件。

还支持将模式DOM传递给XmlSchemaLoad API。

使用透明模式

描述了透明模式的基本用法。

应用程序首先调用XmlEvCreateSVCtx( ）。此调用创建并返回xmlctx类型的事件上下文，必须在后续调用流式验证器时传递该事件上下文。必须通过调用XmlEvDestroyCtx()来终止创建的事件上下文。

在创建事件上下文之后，应用程序通过调用XmlEvNext()重复将验证提升到下一个事件，该XmlEvNext()返回下一个事件的类型。其他API接口允许应用程序检索与上一个事件相关的信息。

透明模式下的错误处理

没有有效事件的概念。有效性是文档的属性，而不是文档的单个项目和事件。

错误是：

  • XML_EVENT_FATAL_ERROR - 当XML事件的生产者报告此错误时，流验证器将此事件返回给应用程序并停止验证过程。

  • XML_EVENT_ERROR - 当验证错误发生时，流验证程序将此事件返回给应用程序。然后，应用程序可以调用XmlEvGetError()以获取有关错误的更多信息。

如果应用程序未收到任何XML_EVENT_ERROR或XML_EVENT_FATAL_ERROR事件，则该文档有效。因此，应用程序必须处理这些事件而不是忽略它们。

不缓存这些错误，并且相关信息不可用于以后检索。

C.1 JDBC和多线程

Oracle JDBC驱动程序为使用Java多线程的应用程序提供全面支持，并对其进行高度优化。对连接的受控串行访问（例如连接缓存提供的连接）是必要和鼓励的。但是，Oracle强烈反对在多个线程之间共享数据库连接。避免允许多个线程同时访问连接。如果多个线程必须共享连接，请使用规范的开始使用/结束使用协议。

在处理多线程应用程序时，请牢记以下几点：

  • 将Connection对象用作局部变量。

  • 在退出方法之前关闭finally块中的连接。例如：

    连接conn = null;尝试{...终于{if（conn！）= null）conn.close（）; }

  • 不要在线程之间共享Connection对象。

  • 永远不要在JDBC对象上同步，因为它是由驱动程序在内部完成的。

  • 使用Statement.setQueryTimeout方法设置执行查询的时间，而不是从其他线程取消长时间运行的查询。

  • 将Statement.cancel方法用于SQL操作，如SELECT ， UPDATE或DELETE 。

  • 将Connection.cancel方法用于SQL操作，如COMMIT ， ROLLBACK等。

  • 不要使用Thread.interrupt方法。

C.2.1禁用自动提交模式

自动提交模式指示数据库是否在每次SQL操作后发出自动COMMIT操作。例如，如果您使用不同的绑定变量重复相同的语句，则处于自动提交模式在时间和处理工作方面可能是昂贵的。

默认情况下，新连接对象处于自动提交模式。但是，您可以使用连接对象的setAutoCommit方法（ java.sql.禁用自动提交模式java.sql.Conection或oracle.jdbc.OracleConnection 。

在自动提交模式下， COMMIT操作在语句完成或下一次执行时发生，以先到者为准。在声明返回的情况下 ResultSet对象，语句在检索到结果集的最后一行或结果集已关闭时完成。在更复杂的情况下，单个语句可以返回多个结果以及输出参数值。这里，在检索到所有结果和输出参数值时发生COMMIT 。

如果使用setAutoCommit(false)调用禁用自动提交模式，则必须使用连接对象的commit或rollback方法手动提交或回滚操作组。

C.2.2标准提取大小和Oracle行预取

Oracle JDBC连接和语句对象允许您在查询期间填充结果集时指定每次访问数据库时预取到客户端的行数。您可以在连接对象中设置一个值，该值影响通过该连接生成的每个语句，并且您可以在任何特定语句对象中覆盖该值。连接对象中的默认值为10。将数据预取到客户端可减少到服务器的往返次数。

类似地，并且具有更大的灵活性，JDBC 2.0使您能够指定每次行程要获取的行数，包括语句对象（影响后续查询）和结果集对象（影响行重新获取）。默认情况下，结果集使用生成它的语句对象的值。如果未设置JDBC 2.0提取大小，则默认使用Oracle连接行预取值。

5.1关于分类

分类是一种数据挖掘功能，可将集合中的项目分配给目标类别或类。分类的目标是准确地预测数据中每个案例的目标类。例如，分类模型可用于将贷款申请人识别为低，中或高信用风险。

分类任务以已知类分配的数据集开始。例如，可以基于一段时间内许多贷款申请人的观察数据来开发预测信用风险的分类模型。除历史信用评级外，数据还可能跟踪就业历史，房屋所有权或租赁，居住年限，投资数量和类型等。信用评级是目标，其他属性是预测变量，每个客户的数据构成一个案例。

分类是离散的，并不意味着秩序。连续的浮点值表示a 数字而非分类目标。具有数字目标的预测模型使用回归算法，而不是分类算法。

最简单的类型分类问题是二元分类。在二进制分类中，目标属性只有两个可能的值：例如，高信用评级或低信用评级。多类目标具有两个以上的值：例如，低，中，高或未知的信用评级。

在模型构建（训练）过程中，分类算法找到预测变量的值与目标值之间的关系。不同的分类算法使用不同的技术来寻找关系。这些关系在模型中进行了总结，然后可以将其应用于类别分配未知的不同数据集。

通过将预测值与一组测试数据中的已知目标值进行比较来测试分类模型。分类项目的历史数据通常分为两个数据集：一个用于构建模型;另一个用于测试模型。

应用分类模型会导致每种情况的类别分配和概率。例如，将客户分类为低，中或高值的模型还可以预测每个客户的每个分类的概率。

分类在客户细分，业务建模，营销，信用分析以及生物医学和药物响应建模方面有许多应用。

5.2测试分类模型

通过将分类模型应用于具有已知目标值的测试数据并将预测值与已知值进行比较来测试分类模型。

测试数据必须与用于构建模型的数据兼容，并且必须以与构建数据相同的方式准备。通常，构建数据和测试数据来自相同的历史数据集。一部分记录用于构建模型;其余记录用于测试模型。

测试指标用于评估模型预测已知值的准确程度。如果模型运行良好并满足业务要求，则可以将其应用于新数据以预测未来。

5.2.2升降机

提升测量分类模型的预测比随机生成的预测更好的程度。

Lift仅适用于二进制分类，需要指定正类。如果模型本身没有二进制目标，则可以通过将一个类指定为正数并将所有其他类组合为一个负类来计算提升。

可以计算出许多统计数据来支持升力的概念。基本上，升力可以理解为两个百分比的比率：模型对正确分类的百分比与测试数据中实际正分类的百分比。例如，如果市场调查中40％的客户对过去的促销活动做出了积极响应（积极分类），并且该模型准确预测了其中75％，则通过将.75除以.40获得升力。由此产生的升力为1.875。

根据分位数计算提升，每个分位数包含相同数量的个案。数据在评分后被分成分位数。它按阳性等级的概率从最高到最低排序，因此阳性预测的最高浓度在最高分位数中。典型的分位数是10。

Lift通常用于衡量营销应用程序中响应模型的性能。响应模型的目的是识别具有潜在高浓度的营销活动积极响应者的人群。电梯揭示了必须征求多少人口以获得最高百分比的潜在响应者。

5.2.3接收机工作特性（ROC）

ROC是用于比较分类模型中的预测目标值和实际目标值的度量。

与Lift一样，ROC适用于二进制分类，需要指定一个正类。

您可以使用ROC深入了解模型的决策能力。模型准确预测负面或正面课程的可能性有多大？

ROC衡量概率阈值变化的影响。概率阈值是模型用于分类的决策点。二进制分类的默认概率阈值为0.5。当预测概率为50％或更高时，模型预测该类。当概率小于50％时，预测另一类。（在多类分类中，预测类是以最高概率预测的类。）

5.2.3.3 ROC和模型偏差

模型的ROC曲线表示其混淆矩阵中所有可能的值组合。

概率阈值的变化会影响模型的预测。例如，如果用于预测正类的阈值从0.5变为0.6，则进行较少的正预测。这会影响混淆矩阵中值的分布：真假阳性的数量和真假阴性的数量不同。

您可以使用ROC查找产生最高总体准确度或最高每类准确度的概率阈值。例如，如果准确预测正类很重要，但是你不关心负类的预测误差，那么你可以降低正类的阈值。这可能使模型偏向于积极的阶级。

  • 真阳性：测试数据中的阳性病例，预测概率大于或等于概率阈值（正确预测）。

  • 假阴性：测试数据中的阳性病例，预测概率严格小于概率阈值（错误预测）。

  • 误报：测试数据中的负面情况，预测概率大于或等于概率阈值（错误预测）。

  • 真正的正分数 ：命中率。 （真阳性/（真阳性+假阴性））

  • 误报率：误报率。 （误报/ （假阳性+真阴性））

5.3偏向分类模型

成本，先验概率和类权重是偏差的方法分类模型。

5.3.1费用

一个成本矩阵是影响模型决策的机制。一个成本矩阵可以使模型最小化代价高昂的错误分类。它还可以使模型最大化有益的准确分类。

例如，如果模型将信用不良的客户归类为低风险，则此错误代价很高。成本矩阵可以偏向模型以避免此类错误。成本矩阵还可用于偏向模型，以支持具有最差信用历史的客户的正确分类。

ROC是评估模型在不同概率阈值下的行为方式的有用指标。在给定不同的使用场景的情况下，您可以使用ROC帮助您找到给定分类器的最佳成本。您可以使用此信息创建成本矩阵以影响模型的部署。

5.3.1.1成本与准确性

比较成本矩阵和混淆矩阵的成本和准确性，以评估模型质量。

像一个混淆矩阵，成本矩阵是n- by- n矩阵，其中n是类的数量。混淆矩阵和成本矩阵都包括基于给定测试数据集的实际和预测结果的每种可能组合。

混淆矩阵用于测量准确性，正确预测与预测总数的比率。成本矩阵用于指定不同预测的准确性的相对重要性。在大多数商业应用中，在评估模型质量时，除了准确性之外还要考虑成本。

5.4分类算法

了解Oracle Data Mining中使用的不同分类算法。

神谕数据挖掘提供以下分类算法：

  • 决策树

    决策树自动生成规则，这些规则是显示用于构建树的逻辑的条件语句。

  • 显式语义分析

    显式语义分析（ESA）旨在对文本数据进行预测。该算法可以处理具有数十万个类的用例。

  • 朴素贝叶斯

    朴素贝叶斯使用贝叶斯定理，这是一种通过计算历史数据中值的频率和值组合来计算概率的公式。

  • 广义线性模型（GLM）

    GLM是一种流行的线性建模统计技术。Oracle Data Mining实现了GLM，用于二进制分类和回归。GLM提供广泛的系数统计和模型统计以及行诊断。GLM也支持置信区间。

  • 随机森林

    Random Forest是一种功能强大且流行的机器学习算法，可带来显着的性能和可扩展性优势。

  • 支持向量机（SVM）

    SVM是一种基于线性和非线性回归的功能强大的先进算法。Oracle Data Mining实现了用于二进制和多类分类的SVM。

注意：

Oracle Data Mining使用Naive Bayes作为默认值分类算法。

4.1关于分类

分类是一种数据挖掘功能，可将集合中的项目分配给目标类别或类。分类的目标是准确地预测数据中每个案例的目标类。例如，分类模型可用于将贷款申请人识别为低，中或高信用风险。

分类任务以已知类分配的数据集开始。例如，可以基于一段时间内许多贷款申请人的观察数据来开发预测信用风险的分类模型。除历史信用评级外，数据还可能跟踪就业历史，房屋所有权或租赁，居住年限，投资数量和类型等。信用评级是目标，其他属性是预测变量，每个客户的数据构成一个案例。

分类是离散的，并不意味着秩序。连续的浮点值表示a 数字而非分类目标。具有数字目标的预测模型使用回归算法，而不是分类算法。

最简单的类型分类问题是二元分类。在二进制分类中，目标属性只有两个可能的值：例如，高信用评级或低信用评级。多类目标具有两个以上的值：例如，低，中，高或未知的信用评级。

在模型构建（训练）过程中，分类算法找到预测变量的值与目标值之间的关系。不同的分类算法使用不同的技术来寻找关系。这些关系在模型中进行了总结，然后可以将其应用于类别分配未知的不同数据集。

通过将预测值与一组测试数据中的已知目标值进行比较来测试分类模型。分类项目的历史数据通常分为两个数据集：一个用于构建模型;另一个用于测试模型。

应用分类模型会导致每种情况的类别分配和概率。例如，将客户分类为低，中或高值的模型还可以预测每个客户的每个分类的概率。

分类在客户细分，业务建模，营销，信用分析以及生物医学和药物响应建模方面有许多应用。

4.2测试分类模型

通过将分类模型应用于具有已知目标值的测试数据并将预测值与已知值进行比较来测试分类模型。

测试数据必须与用于构建模型的数据兼容，并且必须以与构建数据相同的方式准备。通常，构建数据和测试数据来自相同的历史数据集。一部分记录用于构建模型;其余记录用于测试模型。

测试指标用于评估模型预测已知值的准确程度。如果模型运行良好并满足业务要求，则可以将其应用于新数据以预测未来。

4.2.2升降机

提升测量分类模型的预测比随机生成的预测更好的程度。

Lift仅适用于二进制分类，需要指定正类。如果模型本身没有二进制目标，则可以通过将一个类指定为正数并将所有其他类组合为一个负类来计算提升。

可以计算出许多统计数据来支持升力的概念。基本上，升力可以理解为两个百分比的比率：模型对正确分类的百分比与测试数据中实际正分类的百分比。例如，如果市场调查中40％的客户对过去的促销活动做出了积极响应（积极分类），并且该模型准确预测了其中75％，则通过将.75除以.40获得升力。由此产生的升力为1.875。

根据分位数计算提升，每个分位数包含相同数量的个案。数据在评分后被分成分位数。它按阳性等级的概率从最高到最低排序，因此阳性预测的最高浓度在最高分位数中。典型的分位数是10。

Lift通常用于衡量营销应用程序中响应模型的性能。响应模型的目的是识别具有潜在高浓度的营销活动积极响应者的人群。电梯揭示了必须征求多少人口以获得最高百分比的潜在响应者。

4.2.3接收机工作特性（ROC）

ROC是用于比较分类模型中的预测目标值和实际目标值的度量。

与Lift一样，ROC适用于二进制分类，需要指定一个正类。

您可以使用ROC深入了解模型的决策能力。模型准确预测负面或正面课程的可能性有多大？

ROC衡量概率阈值变化的影响。概率阈值是模型用于分类的决策点。二进制分类的默认概率阈值为0.5。当预测概率为50％或更高时，模型预测该类。当概率小于50％时，预测另一类。（在多类分类中，预测类是以最高概率预测的类。）

4.2.3.2曲线下面积

ROC曲线下面积（AUC）测量二元分类模型的区分能力。AUC越大，分配实际阳性病例的可能性越高，并且阳性的概率高于实际阴性病例。AUC度量对于具有不平衡目标分布的数据集尤其有用（一个目标类占据另一个目标类）。

4.2.3.3 ROC和模型偏差

模型的ROC曲线表示其混淆矩阵中所有可能的值组合。

概率阈值的变化会影响模型的预测。例如，如果用于预测正类的阈值从0.5变为0.6，则进行较少的正预测。这会影响混淆矩阵中值的分布：真假阳性的数量和真假阴性的数量不同。

您可以使用ROC查找产生最高总体准确度或最高每类准确度的概率阈值。例如，如果准确预测正类很重要，但是你不关心负类的预测误差，那么你可以降低正类的阈值。这可能使模型偏向于积极的阶级。

成本矩阵是用于改变模型评分的概率阈值的便利机制。

4.3偏向分类模型

成本，先验概率和类权重是偏差的方法分类模型。

4.3.1费用

一个成本矩阵是影响模型决策的机制。一个成本矩阵可以使模型最小化代价高昂的错误分类。它还可以使模型最大化有益的准确分类。

例如，如果模型将信用不良的客户归类为低风险，则此错误代价很高。成本矩阵可以偏向模型以避免此类错误。成本矩阵还可用于偏向模型，以支持具有最差信用历史的客户的正确分类。

ROC是评估模型在不同概率阈值下的行为方式的有用指标。在给定不同的使用场景的情况下，您可以使用ROC帮助您找到给定分类器的最佳成本。您可以使用此信息创建成本矩阵以影响模型的部署。

4.3.1.1成本与准确性

比较成本矩阵和混淆矩阵的成本和准确性，以评估模型质量。

像一个混淆矩阵，成本矩阵是n- by- n矩阵，其中n是类的数量。混淆矩阵和成本矩阵都包括基于给定测试数据集的实际和预测结果的每种可能组合。

混淆矩阵用于测量准确性，正确预测与预测总数的比率。成本矩阵用于指定不同预测的准确性的相对重要性。在大多数商业应用中，在评估模型质量时，除了准确性之外还要考虑成本。

4.4分类算法

了解Oracle Data Mining中使用的不同分类算法。

神谕数据挖掘提供以下分类算法：

  • 决策树

    决策树自动生成规则，这些规则是显示用于构建树的逻辑的条件语句。

  • 显式语义分析

    显式语义分析（ESA）旨在对文本数据进行预测。该算法可以处理具有数十万个类的用例。

  • 朴素贝叶斯

    朴素贝叶斯使用贝叶斯定理，这是一种通过计算历史数据中值的频率和值组合来计算概率的公式。

  • 广义线性模型（GLM）

    GLM是一种流行的线性建模统计技术。Oracle Data Mining实现了GLM，用于二进制分类和回归。GLM提供广泛的系数统计和模型统计以及行诊断。GLM也支持置信区间。

  • 随机森林

    Random Forest是一种功能强大且流行的机器学习算法，可带来显着的性能和可扩展性优势。

  • 支持向量机（SVM）

    SVM是一种基于线性和非线性回归的功能强大的先进算法。Oracle Data Mining实现了用于二进制和多类分类的SVM。

注意：

Oracle Data Mining使用Naive Bayes作为默认值分类算法。

2数据库应用程序的连接策略

数据库连接是客户端进程和数据库实例之间的物理通信路径。数据库会话是数据库实例内存中的逻辑实体，表示当前用户登录数据库的状态。会话从用户通过数据库进行身份验证到用户断开连接或退出数据库应用程序的时间为止。单个连接可以在其上建立0,1或更多会话。

Oracle Real-World Performance小组调查的大多数OLTP性能问题都与应用程序连接策略有关。因此，设计良好的连接策略对于应用程序开发至关重要，尤其是在必须扩展以满足不断增长的需求的企业环境中。

连接池的设计指南

连接池是可由应用程序使用的Oracle数据库连接的缓存。

在运行时，应用程序从池请求连接。如果池包含可以满足请求的连接，则它返回与应用程序的连接。应用程序使用连接对数据库执行工作，然后将连接返回到池。然后，释放的连接可用于下一个连接请求。

在静态连接池中 ，池包含固定数量的连接，无法创建更多以满足需求。因此，如果池找不到空闲连接以满足新的应用程序请求，则请求排队或返回错误。但是，在动态连接池中 ，池会创建一个新连接，然后将其返回给应用程序。理论上，动态连接池可以增加和减少池中的连接数，从而节省在维护不必要的连接时会丢失的系统资源。但是，实际上，动态连接池策略允许潜在的连接风暴和过度订阅问题。

连接风暴

连接风暴是竞争条件，其中应用程序服务器启动越来越多的连接请求，但数据库服务器CPU无法立即计划它们，这会导致应用程序服务器创建更多连接。

在连接风暴期间，数据库连接的数量可以在不到一分钟的时间内从数百到数千。

动态连接池特别容易发生连接风暴。随着连接请求数量的增加，数据库服务器相对于CPU核心数量变得过度订阅。在任何给定时间，只有一个进程可以在CPU核心上运行。因此，如果服务器上存在32个核心，那么一次只能处理32个进程。如果应用程序服务器创建了数百或数千个连接，那么CPU就会忙于尝试跟上系统上争用时间的进程数。

在数据库内部，等待活动随着活动会话数量的增加而增加。您可以通过查看ASH和AWR报告中的等待事件来观察此活动。典型的等待事件包括队列上的锁存器，行缓存对象，无锁存器，enq：TX - 索引争用和缓冲区忙等待。随着等待事件的增加，事务吞吐量会降低，因为会话无法执行工作。由于服务器计算机是超额订阅的，因此监视工具进程必须在CPU上争取时间。在最极端的情况下，使用键盘变得不可能，使调试变得困难。

防止连接风暴的准则：使用静态池

Oracle Real-World Performance组建议您使用静态连接池而不是动态连接池。

经过多年诊断连接风暴，Oracle Real-World Performance小组发现动态连接池通常会为必要的工作负载使用过多的进程。一个流行的神话是动态连接池根据需要创建连接，并在不需要时减少它们。实际上，当连接池耗尽时，应用程序服务器会使数据库连接池的大小快速增加。会话数量随着系统负载的增加而增加，导致所有会话变为活动时出现性能问题。

由于动态连接池可能会使系统快速稳定，因此Oracle Real-World Performance组建议您使用静态连接池而不是动态连接池。

减少连接数可以减轻CPU的压力，从而缩短响应时间并提高吞吐量。这个结果可能看似矛盾。性能提升的原因如下：

  • 更少的会话意味着数据库中与争用相关的等待事件更少。在减少连接之后，以前在锁存器上消耗循环并仲裁争用的CPU可能会花费更多时间处理数据库事务。

  • 随着连接数量的减少，连接可以在CPU上保持更长的时间。因此，与这些进程关联的所有内存页都驻留在CPU缓存中。它们在调度方面变得越来越有效，并且在内存中停滞不前

根据经验，Oracle Real-World Performance组建议将％user与％system系统CPU利用率的比率提高到90/10，并且数据库服务器上每个CPU核心平均不超过10个进程。连接数应基于CPU核心数而不是CPU核心线程数。例如，假设一台服务器有2个CPU，每个CPU有18个核心。每个CPU核心有2个线程。根据Oracle Real-Wold Performance组指南，应用程序可以与数据库实例建立36到360个连接。

登录策略的设计指南

所有数据库开发人员面临的一个问题是应用程序如何以及何时登录数据库以启动事务。

在次优设计中，数据库应用程序为每个SQL请求执行以下步骤：

  1. 登录数据库。

  2. 发出SQL请求，例如INSERT或UPDATE语句。

  3. 退出数据库。

使用登录/注销策略的应用程序可能满足功能要求。此外，当每秒的事务数量较少时，它们可能表现良好。但是，登录和退出数据库是一项资源密集型操作。Oracle Real-World Performance小组发现，使用此类算法的应用程序无法很好地扩展，并且可能导致严重的性能问题，尤其是在与动态连接池一起使用时。登录/注销策略通常不使用连接池。

如果应用程序使用登录/注销设计，并且DBA和开发人员没有意识到问题的根源，那么最初的症状可能是数据库吞吐量低且响应时间过长而且不稳定。对数据库的诊断调查可能表明相对较少的会话处于活动状态，而资源争用较少。

次优性能的线索是每秒的登录次数接近每秒的事务数。当使用每个事务状态的login / loutout时，数据库实例和操作系统在后台执行大量工作以创建新进程，数据库连接和相关的存储区域。其中许多步骤都是序列化的，导致低CPU利用率和低事务吞吐量。

由于上述原因，Oracle Real-World Performance组强烈建议针对任何必须扩展以支持大量事务的应用程序进行登录/注销设计。

防止程序化会话泄漏的设计指南

程序丢失连接时发生会话泄漏 ，但其会话在数据库实例中保持活动状态。泄漏的会话以编程方式丢失给应用程序。

优化设计的应用程序可防止会话泄漏。通常，由于应用程序捕获的异常，会发生会话泄漏。如果应用程序没有正确处理异常，那么它可以在不执行提交或回滚的情况下终止连接，从而泄漏会话。

会话泄漏可能会导致严重的数据库性能和数据完整性问题。通常，问题采取以下形式：

  • 排水连接池

  • 锁定泄漏

  • 逻辑腐败

耗尽的连接池

设计缺陷会导致连接池耗尽。

例如，假设应用程序设计缺陷导致它一致地泄漏会话。即使泄漏率很低，动态连接池也会导致会话数量不断增加，从而无法使用。

其结果是减少了可用的连接池，并防止其余连接跟上工作负载。无法使用的会话数量会增加，直到池中没有可用的连接。

检查会话泄漏

会话泄漏是由于应用程序或应用程序服务器中的问题而发生的，无法单独从数据库中修复。需要在应用程序或应用程序服务器中解决该问题。检查会话泄漏的简单方法是修改连接池以使用与数据库的一个连接并测试应用程序。使用一个连接进行测试可以更容易地找到应用程序中问题的根本原因。

锁漏

锁泄漏通常是会话泄漏的副作用。

例如，处于批量更新过程中的泄漏会话可能会在表中的多行上保留TX锁定。如果泄漏的会话持有锁，则想要获取锁的会话形成泄漏会话后面的队列。

持有锁的程序正在等待来自客户端的交互以释放该锁，但由于连接以编程方式丢失，因此将不会发送消息。因此，数据库无法提交或回滚会话中活动的任何事务。

逻辑腐败

泄漏的会话可以包含对数据库的未提交的更改。例如，当意外地释放数据库连接时，事务处于其工作的中途。

这种情况可能会导致以下问题：

  • 应用程序向UI报告错误。在这种情况下，客户可能会抱怨他们已经丢失了工作，例如，商业订单或航班时刻表

  • 即使没有发生提交或回滚，UI也会收到提交消息。这是最糟糕的情况，因为后续事务可能会从泄漏的会话中提交自己的工作和一半的事务。在这种情况下，数据库在逻辑上已损坏。

关于运行时连接负载平衡

Oracle Real Application Clusters（Oracle RAC）是一个数据库选项，其中单个数据库由多个节点上的多个实例托管。Oracle RAC共享磁盘集群数据库方法提高了可伸缩性。可以轻松添加或释放节点以满足当前需求并提高可用性，因为如果一个节点发生故障，另一个节点可以承担其工作负载。Oracle RAC为数据库添加了高可用性和故障转移容量，因为所有实例都可以访问整个数据库。

工作请求在连接时（ 连接时负载平衡 ，由Oracle Net Services提供）和运行时（ 运行时连接负载平衡 ）之间得到平衡 。对于Oracle RAC环境，会话池使用通过快速应用程序通知（FAN）事件从Oracle RAC负载平衡通报接收的服务指标来平衡应用程序会话请求。进入会话池的工作请求可以使用当前服务性能分布在提供服务的Oracle RAC实例中。

当应用程序创建会话时，会发生连接时间负载平衡。创建会话时，池化会话必须在Oracle RAC实例中良好分布，以确保每个实例上的会话可以执行工作。

当应用程序从现有会话池中选择会话时（因此是非常频繁的活动），会发生运行时连接负载平衡。运行时连接负载平衡将工作请求路由到最能为工作提供服务的会话池中的会话。对于仅在一个实例上支持服务的会话池，池中的第一个可用会话就足够了。当池支持跨多个实例的服务时，必须跨实例分配工作，以便提供更好服务或具有更大容量的实例获得更多工作请求。

OCI，OCCI，JDBC和ODP.NET客户端应用程序都支持运行时连接负载平衡。

为一个或多个表指定PRIORITY CRITICAL时，IM列存储中这些表的填充是在没有指定优先级， PRIORITY LOW ， PRIORITY MEDIUM和PRIORITY HIGH表之前完成的。

9.2.1 ORA-16596：数据库不是Oracle Data Guard代理配置的一部分

向代理发出了请求，但您已连接的数据库实例不再是代理配置的一部分。

解

通过您知道属于代理配置的另一个数据库重新连接到配置。确认代理配置中存在一个数据库，该数据库的名称与返回ORA-16596错误的数据库的db_unique_name值相匹配。

如果您尝试启用配置，但其中一个数据库的代理配置文件被意外删除或过时，也会发生此问题。在这种情况下，从代理配置中删除数据库，手动删除该备用数据库的配置文件（而不是主数据库），然后再次尝试启用配置。启用配置后，您可以使用Cloud Control Add Standby Database向导并选择Add existing standby database选项，也可以使用DGMGRL命令行界面并发出ADD DATABASE命令。

9.2.2主服务器上的重做累积不会发送到某些备用数据库

通过查看Cloud Control中的“日志文件详细信息”页面，您已确定日志文件正在主数据库上累积，并且未归档到代理配置中的某些备用数据库。

解

要缩小问题范围，请执行以下操作：

  • 验证主数据库的状态是否处于TRANSPORT-ON状态（不是TRANSPORT-OFF ）。

  • 验证有问题的备用数据库的LogShipping数据库属性的值是否为ON 。

  • 使用LogXptStatus属性检查主数据库上的重做传输服务的状态。如果重做传输服务有错误，则使用错误消息确定进一步的检查和解决操作。例如：

    • 如果错误指示备用数据库不可用，则需要重新启动备用数据库。

    • 如果错误指示没有侦听器，则需要重新启动侦听器。

    • 如果错误指示备用数据库没有本地目标，则需要设置备用位置以存储主数据库中的归档重做日志文件。

9.2.3许多日志文件在备用数据库上接收但未应用

解

归档重做日志文件可能未应用于备用数据库的原因有很多。调查日志文件正在构建的原因并排除有效原因。

如果备用数据库的当前状态不正常：

  • 如果这是逻辑备用数据库，请检查是否发生了失败的事务。

  • 如果要在调查问题时禁止显示错误，可以暂时禁用数据库的代理管理。

9.4故障转移操作期间的故障排除问题

虽然故障转移操作可能会失败，但不太可能。如果确实发生错误，则可以遵循一些指导原则来解决问题，然后重试代理故障转移。

9.4.1故障转移到物理备用数据库失败

这些步骤描述了如何从失败的代理故障转移恢复到物理备用数据库。

9.4.1.1失败的代理完成物理故障转移

检查目标备用数据库上的警报日志文件和代理日志文件（drc * .log），以确定故障原因并更正问题。

如果可以更正报告的问题，请重试故障转移操作。如果无法更正报告的问题，或者在报告的问题得到纠正后故障转移操作再次失败，请执行以下步骤：

  1. 连接到目标备用数据库并使用FORCE选项禁用快速启动故障转移（如果已启用）。

  2. 然后你可以：

    • 连接到另一个物理备用数据库并尝试代理完成故障转移。

    • 执行代理立即故障转移到目标物理备用数据库。

  3. 恢复原始主数据库和禁用状态为需要恢复的任何旁观者物理备用数据库（ ORA-16661 ）。

  4. 如果在步骤1中禁用了快速启动故障转移，则重新启用它。

9.4.1.2失败的经纪人即时物理故障转移

检查目标备用数据库上的警报日志文件和代理日志文件（drc * .log），以确定故障原因并更正问题。

如果问题可以解决，请重试代理立即进行故障转移。否则，连接到另一个物理备用数据库并尝试完成代理或立即进行故障转移。

9.4.2故障转移到逻辑备用数据库失败

如果故障转移到逻辑备用数据库失败，则需要遵循以下步骤。

  1. 检查目标备用数据库上的警报日志文件和代理日志文件（drc * .log），以确定故障原因并更正问题。

  2. 连接到目标备用数据库并使用FORCE选项禁用快速启动故障转移（如果已启用）。

  3. 重试代理故障转移。

  4. 恢复旧的主数据库。将从新主数据库的副本重新创建所有旁观者备用数据库。

  5. 如果在步骤1中禁用了快速启动故障转移，则重新启用它。

如果代理故障转移继续失败，则应在Oracle Data Guard配置中的所有数据库上停止代理（将DG_BROKER_START初始化参数设置为FALSE ）。从所有数据库中删除Oracle Data Guard代理配置文件。使用Oracle Data Guard概念和管理中的角色转换准则尝试手动故障转移。

注意：

您可以使用DGMGRL ENABLE CONFIGURATION和DISABLE CONFIGURATION命令启用或禁用代理配置。您无法使用Cloud Control禁用配置。如果以前使用DGMGRL禁用了Cloud Control，则只能使用Cloud Control启用配置。

二进制XML使得在XML文本和压缩二进制XML之间进行编码和解码成为可能。应用程序编程接口（API）在二进制XML之上提供，供XML应用程序直接使用。XML文档片段的压缩和解压缩有助于增量处理。

二进制XML存储格式 - Java

二进制XML是一种紧凑的XML-Schema-aware XML数据编码，但它可以与不基于XML模式的XML数据一起使用。您还可以将二进制XML用于数据库外部的XML数据（例如，在客户端应用程序中）。二进制XML允许对XML文档进行编码和解码，从文本到二进制，二进制到文本。二进制XML是具有本机数据库数据类型的解析后持久XML。

可以使用二进制XML存储选项创建XMLType表和列。可以通过所有现有的结构化查询语言（SQL）运算符和函数以及对XMLType进行操作的过程语言/结构化查询语言（PL / SQL）API来访问和操作二进制格式的XML数据。

与非结构化存储相比，二进制XML提供了更高效的数据库存储，更新，索引，查询性能和片段提取。它可以将数据和元数据一起存储或单独存储。

二进制XML处理器

二进制XML处理器是用于处理二进制XML到文本和XML文本到二进制XML的组件的抽象术语。它还可以提供用于存储模式的缓存。二进制XML处理器可以发起或接收网络协议请求。

二进制XML处理器的基类是BinXMLProcessor 。

使用二进制XML的模型

在应用程序中使用二进制XML有几种模型。这些小节描述了使用二进制XML的术语和模型。

二进制XML的使用术语

描述了与二进制XML使用相关的术语。

  • doc-id ：每个编码的XML文档都由唯一的doc-id标识。它是一个16字节的全局用户标识符（GUID）或像URL一样的不透明字节序列。

  • 令牌表 ：当文本XML文档没有与之关联的架构时，则使用令牌（或符号）表来最小化重复项的空间。

  • 词汇表id ：可以是令牌表的schema-id或名称空间统一资源标识符（URI）。

  • schema-id ：作用于二进制XML处理器的模式的唯一不透明二进制标识符。schema-id对于二进制XML处理器是唯一的，并且仅在该二进制XML处理器的范围内可识别。即使架构发展，schema-id仍保持不变。schema-id表示整个模式文档集，包括导入和包含的模式。

  • schema version ：每个带注释的模式都有一个与之关联的版本号。版本号被指定为系统级注释的一部分。当演进模式时，它由二进制XML处理器递增（即，使用二进制XML处理器注册相同模式的新版本）。

  • 部分有效性 ：使用模式的二进制XML流编码意味着至少部分有效的模式。部分有效性意味着不对唯一键，keyref，标识符（ID）或DTD属性（如IDREF）进行验证。

独立模型

这是二进制XML的最简单的使用场景。有一个二进制XML处理器。

唯一可用的存储库是本地内存中的词汇表缓存，它不是持久的，只能在二进制XML处理器的生命周期中使用。所有模式必须在编码之前使用二进制XML Processor预先注册，或者在XML Processor看到xsi:SchemaLocation标记时自动注册它们。对于解码，模式已在词汇表缓存中可用。

客户/服务器模型

在客户端 - 服务器方案中，二进制XML处理器使用Java数据库连接（JDBC）连接到数据库。假设在编码之前向数据库注册了XML模式。

以下是如何实现这一目标的示例：

BEGIN DBMS_XMLSCHEMA.registerSchema（SCHEMAURL =>'http://xmlns.oracle.com/xdb/documentation/purchaseOrder.xsd',SCHEMADOC => bfilename（'XMLDIR'，'purchaseOrder.xsd'），CSID => nls_charset_id（' AL32UTF8'），GENTYPES => FALSE，OPTIONS => REGISTER_BINARYXML）;结束; /

除非为数据指定单独的连接（使用associateDataConnection() ），否则假定使用单个连接存储和检索所有数据和元数据以进行编码和解码。

带存储库的Web服务模型

在这种情况下，有多个客户端，每个客户端运行二进制XML处理器。一个编码，另一个解码。有一个公共存储库（不一定是数据库）连接到所有客户端以进行元数据存储。它可以是文件系统或其他一些存储库。

第一个二进制XML处理器确保在执行编码之前向存储库注册了模式，或者可以在编码时使用xsi:schemaLocation标记自动注册模式。第二个二进制XML处理器用于解码，不知道模式的位置，并从存储库中获取模式。

如果第一个二进制XML处理器注册了一个模式，而第二个二进制XML处理器在存储库中注册了相同的模式，则二进制XML处理器不会编译模式，而只是返回本地词汇表缓存中现有编译模式的vocabulary-id 。

BinXMLProcessor不是线程安全的，因此访问存储库的多个线程或客户端必须实现自己的线程安全方案。

没有存储库的Web服务模型

在这种情况下，有多个客户端，每个客户端运行二进制XML处理器。编码和解码可以在不同的客户端上进行。没有共同的元数据存储库。

编码器必须确保传递给下一个客户端的二进制数据独立于模式：即具有内联标记定义。这可以通过在编码期间使用setProperty()方法设置schemaAware = false和inlineTokenDefs = true来实现。解码时，不需要架构。

二进制XML for Java的组件

描述了用于Java的二进制XML的组件。

这些是组件：

  • 二进制XML编码 - 二进制XML编码器将XML 1.0信息集转换为二进制XML。

  • 二进制XML解码 - 二进制XML解码器将二进制XML转换为XML信息集。

  • 二进制XML词汇表管理，包括模式管理和令牌管理。

二进制XML编码

编码器是从BinXMLStream创建的。它将XML文本作为输入，并将编码的二进制XML输出到BinXMLStream 。它使用流式SAX读取XML文本。 XML文本的编码基于解析XML文本的结果。

在编码器上设置schemaAware标志，指定编码是模式感知还是无模式。

对于模式感知编码，编码器确定是否已向词汇表管理器注册具有指定模式URL的模式。对于基于存储库或基于数据库的处理器，编码器基于架构URL向存储库或数据库查询已编译的架构。如果模式在数据库中可用，则以二进制XML格式从存储库或数据库中获取该模式，并向本地词汇表管理器注册。词汇表是架构。

还设置一个标志，指示编码生成独立于模式的二进制XML流。在这种情况下，生成的二进制XML流包含内联的所有标记定义，并且不依赖于模式或外部标记集。

如果编码是模式感知的，则编码器使用来自模式对象的数据类型信息来更有效地编码SAX流。存在与每个模式内置数据类型相关联的默认编码数据类型。使用模式的二进制XML流编码意味着至少部分有效使用模式（对于部分有效性，不对唯一密钥，keyref或ID或DTD属性（如IDREF）进行验证）。如果已知数据对于模式完全有效，则编码的二进制XML流存储该信息。

二进制XML解码

二进制XML解码器将二进制XML转换为XML信息集。解码器是从BinXMLStream创建的;它从此流中读取二进制XML并输出SAX事件或提供拉式InfosetReader API以读取已解码的XML。

如果没有架构与BinXMLStream相关联，则令牌定义可以在BinXMLStream内联或存储在令牌集中。如果相应命名空间的标记未存储在本地词汇表缓存中，则从存储库中提取标记集。

架构管理

对于元数据持久性，Oracle建议您使用DB二进制XML处理器。在这种情况下，模式和令牌集已在数据库中注册。词汇表管理器从数据库中提取模式或令牌集，并将其缓存在本地词汇表缓存中以进行编码和解码。

如果必须使用不是数据库的持久性元数据存储库，则可以插入自己的元数据存储库。您必须实现与此存储库BinXMLMetadataProvider进行通信的接口。

二进制XML词汇管理的模式注册

使用本地二进制XML处理器在本地注册XML模式。它包含一个词汇表管理器，用于维护用户在其存在期间提交的所有XML模式。与本地二进制XML处理器关联的词汇表管理器不提供XML架构持久性。

如果再次注册相同的XML模式（相同的模式位置和相同的目标名称空间），则不会对其进行解析，并返回现有的词汇表标识符。

如果注册了具有相同目标命名空间和不同模式位置的新XML模式，则使用新模式定义扩充现有XML模式定义。如果发生冲突，则会引发错误。

架构识别

每个模式由词汇表id标识。词汇表id在处理器的范围内，并且在处理器内是唯一的。任何使用模式验证的文档都需要使用最新版本的模式进行验证。

架构注释

二进制XML模式注释只能出现在XML模式中的元素<xsd:appInfo>中。词汇表管理器在XML模式注册期间解释用户级和系统级注释。所有其他模式注释（例如与数据库相关的注释）都将被忽略。

用户级注释

用户级注释由用户在注册前指定。

encodingType - 此注释可以在xsd:element ， xsd:attribute或xsd:simpleType元素中使用。它指示用于编码元素或属性的节点值的数据类型。对于字符串，此版本仅支持Unicode（UTF-8）编码的8位编码。

系统级注释

词汇管理器在注册时添加系统级注释。你无法覆盖它们。

令牌管理

令牌集可以从数据库或元数据存储库中获取，缓存在本地词汇表管理器中，并用于解码。在编码时，可以将令牌集推送到存储库以实现持久性。

通过在编码器上设置标志，也可以将令牌定义作为二进制XML流的一部分包括在内。

架构注册概述

您使用二进制XML处理器注册XML模式。架构是一个文本文件，可以包含用户级注释。作为注册过程的一部分，处理器添加系统级注释。然后，Schema Builder将处理生成的带注释的模式，以构建XML模式对象。

二进制XML

二进制XML处理器可以与数据库通信，以进行各种类型的二进制XML操作，包括存储和检索二进制XML模式，令牌集和二进制XML流。

DBBinXMLMetadataProvider对象可以使用专用JDBC连接进行实例化，也可以使用连接池来访问模式和令牌集等词汇表信息。处理器还与一个或多个数据连接相关联以访问XML数据。

数据库通信涉及以下方面：

  1. 使用词汇表ID或架构URL提取已编译的二进制XML架构

    要检索已编译的二进制XML模式以进行编码，将根据模式URL查询数据库。为了解码二进制XML模式，根据词汇表ID从数据库中获取它。

  2. 使用架构URL存储未编译的二进制XML架构并检索词汇表ID。

    在编码期间遇到xsi:schemaLocation标记时，会在数据库中注册模式以在数据库中进行持久存储。从数据库中检索与模式相关联的词汇表id以及编译模式的二进制版本;使用从数据库返回的词汇表id构建编译的架构对象并将其存储在本地缓存中。

  3. 使用命名空间URL检索二进制令牌集。

    如果要解码的二进制流与用于解码的令牌表相关联，则使用元数据连接从数据库中取出这些令牌表。

  4. 使用命名空间URL存储二进制令牌集

    如果XML文本已经编码而没有模式，那么它会生成一组令牌定义的令牌。这些令牌表可以持久存储在数据库中。元数据连接用于将令牌集传输到数据库。

  5. 具有远程存储选项的二进制XML流

    您有责任创建一个包含带有二进制XML的XMLType列的表，用于存储编码结果并检索二进制XML以进行解码。可以使用Oracle Net Services和JDBC实现与数据库的通信。从JDBC查询的输出结果集中获取XMLType对象。可以从XMLType对象创建用于读取二进制数据或写出二进制数据的BinXMLStream 。必须扩展XMLType类以支持读取和写入二进制XML数据。

持久存储元数据

您可以为元数据提供持久的后端存储。

本地词汇表管理器和缓存将元数据信息存储在内存中，用于BinXMLProcessor的生命BinXMLProcessor 。您可以通过实现接口BinXMLMetadataProvider并将其插入BinXMLProcessor来插入您自己的后端存储以获取元数据。目前，每个处理器仅支持一个元数据提供程序。

您必须编写一个FileBinXMLMetadataProvider实现了接口BinXMLMetadataProvider 。编码器和解码器使用这些API从持久的后端存储中访问元数据。设置持久存储的配置信息：例如， FileBinXMLMetadataProvider类中的文件系统的根目录。实例化FileBinXMLMetadataProvider并将其插入BinXMLProcessor 。

  • 设置PATH

    要运行调用SQLJ转换sqlj脚本，而不必完全指定其路径，请验证PATH环境变量是否已更新为包含以下内容：

    ORACLE_HOME / bin

    对Microsoft Windows使用反斜杠（\）。将ORACLE_HOME替换为您的实际Oracle主目录。

  • 设置CLASSPATH

    更新CLASSPATH环境变量以包括当前目录以及以下内容：

    ORACLE_HOME /sqlj/lib/translator.jar

    对Microsoft Windows使用反斜杠（\）。将ORACLE_HOME替换为您的实际Oracle主目录。

    在CLASSPATH包含以下运行时库：

    ORACLE_HOME /sqlj/lib/runtime12.jar

    此外，您必须在CLASSPATH包含以下JDBC JAR文件：

    ORACLE_HOME / jdbc / lib / ojdbc6.jar ORACLE_HOME / jdbc / lib / ojdbc7.jar

    注意：

    • 要在JDK 6环境中翻译或运行SQLJ程序，你应该在类路径中有ojdbc6.jar ，并且要在JDK 7环境中翻译或运行SQLJ程序，你应该在类路径中有ojdbc7.jar 。确保在运行时选择正确的JDBC JAR以连接到Oracle数据库。

    • 如果不添加运行时库，则无法运行SQLJ转换程序。您必须在CLASSPATH指定运行时库和转换程序库。

      要查看SQLJ是否已正确安装，并查看SQLJ，JDBC和Java的版本信息，请运行以下命令：

      ％sqlj -version-long

验证sqljutl包的安装

在Oracle数据库实例中联机检查存储过程和函数需要sqljutl包。在为支持Java的数据库安装服务器端Java虚拟机（JVM）期间，程序包将自动安装在SYS模式下。如果您的数据库未启用Java，则必须手动安装此程序包。

如果要验证sqljutl的安装， sqljutl从SQL * Plus发出以下SQL命令：

描述sys.sqljutl

这应该导致包的简要描述。

如果您收到一条消息，指出无法找到该软件包，或者您要安装软件包的更新版本，则可以使用SQL * Plus安装它来运行sqljutl.sql脚本，该脚本位于：

ORACLE_HOME /sqlj/lib/sqljutl.sql

验证SQLJ转换程序与数据库的连接

如果SQLJ转换器能够连接到数据库，那么它可以在转换期间提供对SQL操作的在线语义检查。SQLJ转换器是用Java编写的，它使用JDBC从您指定的数据库连接中获取所需的信息。您可以使用sqlj脚本命令行或使用SQLJ属性文件（默认情况下为sqlj.properties提供联机语义检查的连接参数。

仍在demo目录中时，编辑sqlj.properties文件并根据需要更新，注释或取消注释sqlj.password ， sqlj.url和sqlj.driver行，以反映您的数据库连接信息。有关帮助，请参阅sqlj.properties文件中的sqlj.properties 。

以下是使用Oracle JDBC OCI驱动程序时适当的驱动程序，URL和密码设置的示例。

sqlj.url = jdbc：oracle：oci：@sqlj.driver = oracle.jdbc。OracleDriver sqlj.password = hr

只要为转换时连接指定用户名，就会启用联机语义检查。您可以通过取消注释sqlj.properties文件中的sqlj.user行或使用-user命令行选项来指定用户名。user ， password ， url和driver选项都可以在命令行或属性文件中设置。
