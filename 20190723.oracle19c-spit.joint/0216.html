4.1满足方法4的特殊要求

在研究方法4的要求之前，您应该对术语select-list项和占位符感到满意。选择列表项是查询中关键字SELECT后面的列或表达式。例如，以下动态查询包含三个选择列表项：

从EMP中选择ENAME，JOB，SAL + COMM，DEPTNO = 20

占位符是伪绑定（输入）变量，它们在SQL语句中保存实际绑定变量的位置。您没有声明占位符，可以将它们命名为您喜欢的任何名称。绑定变量的占位符最常用于SET，VALUES和WHERE子句中。例如，以下动态SQL语句每个都包含两个占位符：

插入EMP（EMPNO，DEPTNO）值（：E，：D）从部门删除DEPTNO =：DNUM和LOC =：DLOC

注意：

占位符不能引用表名或列名。

4.1.1什么使方法4特别？

与方法1,2和3不同，动态SQL方法4允许您的程序

  • 接受或构建包含未知数量的选择列表项或占位符的动态SQL语句

  • 明确控制Oracle和FORTRAN类型之间的数据类型转换

要为程序添加此灵活性，必须为Oracle运行时库提供其他信息。

4.1.2 Oracle需要哪些信息？

Pro * FORTRAN预编译器为所有可执行动态SQL语句生成对Oracle的调用。如果动态SQL语句不包含选择列表项或占位符，则Oracle不需要其他信息来执行该语句。以下DELETE语句属于此类别：

*动态SQL语句STMT ='从EMP删除DEPTNO = 30'

但是，大多数动态SQL语句包含绑定变量的选择列表项或占位符，如以下UPDATE语句所示：

*带占位符的动态SQL语句STMT ='UPDATE EMP SET COMM =：C WHERE EMPNO =：E'

要执行包含绑定变量的选择列表项和/或占位符的动态SQL语句，Oracle需要有关将保存输出或输入值的程序变量的信息。具体而言，Oracle需要以下信息：

  • 选择列表项的数量和绑定变量的数量

  • 每个选择列表项和绑定变量的长度

  • 每个选择列表项和绑定变量的数据类型

  • 每个输出变量的内存地址，用于存储选择列表项的值，以及每个绑定变量的地址

例如，要写入select-list项的值，Oracle需要相应输出变量的地址。

4.1.3信息存储在哪里？

Oracle需要的关于绑定变量的选择列表项或占位符的所有信息（除了它们的值）都存储在称为SQL描述符区域（SQLDA）的程序数据结构中。

选择列表项的描述存储在选择的SQLDA中，绑定变量的占位符描述存储在绑定的SQLDA中 。

select-list项的值存储在输出缓冲区中;绑定变量的值存储在输入缓冲区中。您可以使用库例程SQLADR将这些数据缓冲区的地址存储在select或bind SQLDA中，以便Oracle知道在何处写入输出值和读取输入值。

值如何存储在这些数据缓冲区中？输出值使用游标进行FETCH，输入值由程序填充，通常来自用户交互输入的信息。

4.1.4如何获取信息？

您可以使用DESCRIBE语句来帮助获取Oracle所需的信息。DESCRIBE SELECT LIST语句检查每个选择列表项以确定其名称，数据类型，约束，长度，比例和精度，然后将此信息存储在选择的SQLDA中供您使用。例如，您可以使用选择列表名称作为打印输出中的列标题。DESCRIBE还存储SQLDA中的选择列表项的总数。

DESCRIBE BIND VARIABLES语句检查每个占位符以确定其名称和长度，然后将此信息存储在输入缓冲区中并绑定SQLDA供您使用。例如，您可以使用占位符名称来提示用户输入绑定变量的值。

4.2了解SQL描述符区域（SQLDA）

本节详细介绍了SQLDA数据结构。您将学习如何声明它，它包含哪些变量，如何初始化它们以及如何在程序中使用它们。

4.2.1 SQLDA的目的

对于包含未知数量的选择列表项或绑定变量占位符的动态SQL语句，需要方法4。要处理这种动态SQL语句，程序必须显式声明SQLDAs，也称为描述符 （不要与某些FORTRAN编译器生成的CHARACTER变量描述符混淆）。每个描述符都是一个名为COMMON的块，您必须将其复制或硬编码到程序中。

选择描述符包含选择列表项的描述和输出缓冲区的地址，其中存储了选择列表项的名称和值。

注意：

选择列表项的名称可以是列名，列别名，也可以是SAL + COMM等表达式的文本。

绑定描述符包含绑定变量和指示符变量的描述以及存储绑定变量和指示符变量的名称和值的输入缓冲区的地址。

请记住，一些描述符变量包含地址，而不是值。因此，您必须声明数据缓冲区以保存值。您可以决定所需输入和输出缓冲区的大小。因为FORTRAN不支持指针，所以必须使用库子例程SQLADR来获取输入和输出缓冲区的地址。您将学习如何在“ 关于使用SQLADR ”一节中调用SQLADR。

4.2.2多个SQLDA

如果您的程序有多个活动的动态SQL语句，则每个语句必须具有自己的SQLDA。您可以声明任意数量的具有不同名称的SQLDA。例如，您可以声明三个名为SEL1，SEL2和SEL3的选择SQLDAs，以便您可以从三个同时打开的游标进行FETCH。但是，非并发游标可以重用SQLDA。

4.3关于使用SQLDA变量和数组

本节介绍每个SQLDA变量的用途和用法。在示例中，使用先前给出的任意SQLDA文件名，描述符名称和数据缓冲区名称。

4.3.1 N变量

此变量指定可以DESCRIBEd的选择列表项或占位符的最大数量。例如，SELN确定选择描述符数组中的元素数。

在发出DESCRIBE命令之前，必须将此变量设置为描述符数组的维度。在DESCRIBE之后，必须将其重置为变量DESCRIBEd的实际数量，该变量存储在F变量中。

4.3.2 F变量

这是DESCRIBE命令找到的选择列表项或占位符的实际数量。

F变量由DESCRIBE设置。如果F变量为负数，则DESCRIBE命令找到过多的选择列表项或占位符以获取描述符的大小。例如，如果将SELN设置为10但DESCRIBE找到11个选择列表项，则SELF设置为-11。如果发生这种情况，则无法在不重新分配描述符的情况下处理SQL语句。

在DESCRIBE命令之后，必须将N变量设置为等于F变量。

4.3.3 S阵列

此数组包含存储在动态SQL语句中出现的选择列表或占位符名称的数据缓冲区的地址。

在发出DESCRIBE命令之前，必须使用SQLADR设置S数组的元素。

DESCRIBE指示Oracle将第J个选择列表项或占位符的名称存储在SELS（J）或BNDS（J）寻址的缓冲区中。如果SELS和BNDS的元素寻址名为SELSB和BNDSB的数据缓冲区数组的元素，则Oracle将第J个选择列表或占位符名称存储在SELSB（J）或BNDSB（J）中。

4.3.4 M阵列

此数组包含存储选择列表或占位符名称的数据缓冲区的长度。缓冲区由S数组的元素寻址。

必须在发出DESCRIBE命令之前设置M数组的元素。每个选择列表或占位符名称缓冲区可以具有不同的长度。

4.3.5 C阵列

此数组包含select-list或占位符名称的实际长度。DESCRIBE将实际长度数组设置为每个选择列表或占位符名称中的字符数。

4.3.6 L阵列

此数组包含存储在数据缓冲区中的select-list或bind-variable值的长度。

4.3.7选择描述符

DESCRIBE将长度数组设置为每个选择列表项的预期最大值。但是，您可能希望在发出FETCH命令之前重置某些长度。FETCH最多返回n个字符，其中n是FETCH之前的SELL（J）值。

长度的格式因Oracle数据类型而异。对于字符选择列表项，DESCRIBE将SELL（J）设置为选择列表项的最大长度（以字节为单位）。对于NUMBER个选择列表项，分别在变量的低字节和下一个字节中返回缩放和精度。您可以使用库子例程SQLPRC从SELL中提取精度和比例值。请参阅“提取精度和比例”一节。

必须在FETCH之前将SELL（J）重置为数据缓冲区所需的长度。例如，在将NUMBER强制转换为FORTRAN CHARACTER字符串时，将SELL（J）设置为数字的精度加上符号和小数点的2。将NUMBER强制转换为FORTRAN REAL时，将SELL（J）设置为系统上REAL的长度。有关强制数据类型长度的更多信息，请参阅“转换数据”部分。

4.3.8绑定描述符

必须在发出OPEN命令之前设置长度数组。

由于Oracle使用SELV（J）或BNDV（J）中的地址间接访问数据缓冲区，因此它不知道该缓冲区中值的长度。如果要更改Oracle用于第J个select-list或bind-variable值的长度，请将SELL（J）或BNDL（J）重置为所需的长度。每个输入或输出缓冲区可以具有不同的长度。

4.3.9 T阵列

此数组包含select-list或bind-variable值的数据类型代码。这些代码确定在存储在SELV元素寻址的数据缓冲区中时如何转换Oracle数据。 “转换数据”一节中介绍了此主题。

4.3.10选择描述符

DESCRIBE将数据类型代码数组设置为选择列表中项目的内部数据类型（例如，VARCHAR2，CHAR，NUMBER或DATE）。

在FETCHing之前，您可能希望重置某些数据类型，因为Oracle数据类型的内部格式可能难以处理。出于显示目的，将选择列表值的数据类型强制转换为VARCHAR2通常是个好主意。对于计算，您可能希望将数字从Oracle强制转换为FORTRAN格式。请参阅“强制数据类型”部分。

SELT（J）的高位被设置为指示第J个选择列表列的空/非空状态。在发出OPEN或FETCH命令之前，必须始终清除该位。您使用库子例程SQLNUL来检索数据类型代码并清除null / not null位。请参阅“处理空/非空数据类型”一节。

对于选择描述符，DESCRIBE SELECT LIST可以返回任何Oracle内部数据类型。通常，与字符数据一样，内部数据类型与您要使用的外部数据类型完全对应。但是，一些内部数据类型映射到可能难以处理的外部数据类型。因此，您可能希望重置SELT描述符数组中的某些元素。

4.4.8例外情况

在某些情况下，DESCRIBE SELECT LIST返回的内部数据类型可能不适合您的目的。两个例子是DATE和NUMBER。当您描述DATE选择列表项时，Oracle将数据类型代码12返回给SELT数组。除非您在FETCH之前重置代码，否则日期值将以其7字节内部格式返回。要以默认字符格式获取日期，必须将数据类型代码从12更改为1（VARCHAR2），并将SELL值从7增加到9。

同样，当您描述一个NUMBER选择列表项时，Oracle会将数据类型代码2返回给SELT数组。除非您在FETCH之前重置代码，否则数值将以其内部格式返回，这可能不是您想要的。因此，将代码从2更改为1（VARCHAR2），3（INTEGER），4（FLOAT）或其他一些适当的数据类型。

11.1 RMAN报告概述

本节介绍RMAN报告的目的和基本概念。

11.1.1 RMAN报告的目的

作为备份和恢复策略的一部分，您应定期运行指示已备份内容的报告。您可以确定哪些数据文件需要备份或哪些文件最近未备份。此外，您还可以预览出现问题时RMAN必须还原的备份。

备份和恢复的另一个重要方面是监视空间使用情况。如果备份到磁盘，则磁盘可能会填满，这可能会导致性能问题甚至导致数据库停止。您可以使用RMAN确定备份是否为过时备份 ，因此可以删除。

您可能还需要获取有关RMAN作业的历史信息。例如，您可能想知道已发出了多少备份作业，每个备份作业的状态（例如，是否失败或已完成），作业何时开始和完成，以及执行了哪种类型的备份。

11.1.2 RMAN报告的基本概念

RMAN存储执行操作的每个数据库的元数据。

RMAN始终将其RMAN元数据存储库存储在目标数据库的控制文件中。例如，假设您使用RMAN备份prod1和prod2数据库。RMAN存储元数据的备份prod1在控制文件prod1 ，以及用于备份的元数据prod2在控制文件prod2 。

（可选）您可以将RMAN与恢复目录一起使用 。在这种情况下，RMAN在单独的恢复目录数据库中的一组表中维护另一个元数据存储库 。例如，您可以在prod3创建恢复目录。您可以在此恢复目录中注册多个目标数据库。例如，如果您注册prod1和prod2存储在恢复目录prod3 ，然后约其备份RMAN将元数据存储prod1和prod2在恢复目录架构 。

11.5.1查询过去和当前RMAN作业的详细信息

RMAN作业是在RMAN会话中执行的一组命令。因此，一个RMAN作业可以包含多个命令。

例如，您可以在单个会话中执行两个单独的BACKUP命令和一个RECOVER COPY命令。RMAN备份作业是在一个RMAN作业中执行的一组BACKUP命令。例如，在同一RMAN作业中执行的BACKUP DATABASE和BACKUP ARCHIVELOG ALL命令构成单个RMAN备份作业。

视图V$RMAN_BACKUP_JOB_DETAILS和V$RMAN_BACKUP_SUBJOB_DETAILS及其相应的恢复目录版本提供了RMAN备份作业的详细信息。例如，视图显示备份所花费的时间，已发布的备份作业数，每个备份作业的状态（例如，是否失败或已完成），作业何时开始和完成以及备份的类型执行。SESSION_KEY列是发生备份作业的RMAN会话的唯一键。

RMAN备份的写入次数通常少于读取次数。由于RMAN压缩， OUTPUT_BYTES_PER_SEC列不能用作备份速度的度量。测量备份速度的相应列是INPUT_BYTES_PER_SEC 。读取和写入数据之间的比率在COMPRESSION_RATIO列中描述。

要查询有关过去和当前RMAN作业的详细信息：

  1. 将SQL * Plus连接到要查询其备份历史记录的数据库。
  2. 查询V$RMAN_BACKUP_JOB_DETAILS视图，以获取有关备份类型，状态以及开始和结束时间的信息。

    以下查询显示按会话密钥排序的备份作业历史记录，该密钥是RMAN会话的主键：

    COL状态格式a9 COL小时格式化999.99 SELECT SESSION_KEY，INPUT_TYPE，STATUS，TO_CHAR（START_TIME，'mm / dd / yy hh24：mi'）start_time，TO_CHAR（END_TIME，'mm / dd / yy hh24：mi'）end_time，ELAPSED_SECONDS / 3600小时来自V $ RMAN_BACKUP_JOB_DETAILS订购SESSION_KEY;

    以下示例输出显示备份作业历史记录：

11.6查询恢复目录视图

LIST ， REPORT和SHOW命令提供了访问控制文件和恢复目录中数据的最简单方法。

但是，您有时也可以从恢复目录视图中获取有用信息，这些视图驻留在恢复目录模式中并使用RC_前缀。

11.6.1关于恢复目录视图

RMAN从目标数据库控制文件中获取备份和恢复元数据，并将其存储在恢复目录的表中。恢复目录视图是从这些表派生的。恢复目录视图未针对用户查询进行规范化或优化。

通常，恢复目录视图不像RMAN报告命令那样用户友好。例如，当您启动RMAN并连接到目标数据库时，只有在发出LIST ， REPORT和SHOW命令时才能获取此目标数据库的信息。如果在同一恢复目录中注册了10个不同的目标数据库，则目录视图的任何查询都会显示所有10个数据库的所有化身的元数据。您通常必须在视图之间执行复杂的选择和连接，以提取有关数据库化身的有用信息。

大多数目录视图在数据库中都有相应的V$视图。例如， RC_BACKUP_PIECE对应于V$BACKUP_PIECE 。恢复目录视图和相应的V$视图之间的主要区别是每个恢复目录视图都包含有关在恢复目录中注册的所有目标数据库的元数据。V$视图仅包含有关其自身的信息。

11.6.1.1关于已注册数据库的唯一标识符

大多数恢复目录视图包含DB_KEY和DBINC_KEY列。可以通过主键（ DB_KEY列值）或DBID （32位唯一数据库标识符）唯一标识在恢复目录中注册的每个数据库。数据库的每个版本都由DBINC_KEY列唯一标识。

您可以使用DB_KEY和DBINC_KEY来检索目标数据库的特定化身的记录。然后，您可以与大多数其他目录视图执行连接，以隔离属于此化身的记录。

目录和V$视图之间的一个重要区别是，不同的唯一标识符系统用于备份和恢复文件。例如，许多V$视图（如V$ARCHIVED_LOG使用RECID和STAMP列来形成连接的主键。相应的恢复目录视图使用派生值作为其主键，并将此值存储在单个列中。例如， RC_ARCHIVED_LOG的主键是AL_KEY列。AL_KEY列值是RMAN在LIST命令输出中显示的主键。

11.6.1.2关于Data Guard环境中的唯一标识符

在Data Guard环境中查询恢复目录时，需要考虑特殊注意事项。

在Data Guard环境中，多个数据库共享相同的DBID。多个视图包含DB_UNIQUE_NAME列，该列指示记录所属的数据库化身的DB_UNIQUE_NAME 。Data Guard环境中的所有数据库共享相同的DBID，但具有不同的DB_UNIQUE_NAME值。

DB_UNIQUE_NAME目录不知道数据库名称，则DB_UNIQUE_NAME值为null ，对于在恢复目录中注册的Oracle9 i数据库。此外，当数据库升级到Oracle Database 11g或更高版本时，列值为null ，但恢复目录架构尚未协调所有文件的数据库名称。

在恢复目录视图中，主数据库及其关联的备用数据库共享相同的DB_KEY 。但是，Data Guard环境中的每个数据库都具有唯一的RC_SITE.SITE_KEY值。例如，主数据库prod及其备用数据库standby1的DB_KEY值均为1，而prod的SITE_KEY为3， standby1的SITE_KEY为30。

某些恢复目录视图没有DB_UNIQUE_NAME列，但包含SITE_KEY列。您可以使用SITE_KEY列与RC_SITE.SITE_KEY连接，以确定与文件关联的数据库的DB_UNIQUE_NAME 。如“ 关于Data Guard环境中的RMAN文件管理”中所述，Data Guard环境中的每个文件都与创建它的主数据库或备用数据库相关联。

11.6.2查询目标DB_KEY或DBID值的目录视图

DB_KEY值是注册数据库的主键，仅在恢复目录中使用。最简单的方法是获取DB_KEY是使用目标数据库的DBID，每当您将RMAN作为TARGET连接到数据库时，都会显示该DBID。

DBID区分RMAN恢复目录中注册的数据库。

假设您要获取有关在恢复目录中注册的数据库的信息。

要查询目录以获取有关数据库当前版本的信息：

  1. 确定要查看其记录的数据库的DBID。

    您可以通过查看RMAN连接到数据库时显示的输出，查询V$RMAN_OUTPUT或查询V$DATABASE视图来获取DBID。以下示例将SQL * Plus连接到所需的数据库并查询DBID：

    SQL> CONNECT / AS SYSBACKUP SQL> SELECT DBID 2 FROM V $ DATABASE; DBID --------- 598368217
  2. 启动SQL * Plus并以恢复目录的所有者身份连接到恢复目录数据库。
  3. 获取在步骤1中获取其DBID的数据库的数据库密钥。

    要获取数据库的DB_KEY ，请运行以下查询，其中dbid_of_target是在步骤1中获取的DBID：

    SELECT DB_KEY FROM RC_DATABASE WHERE DBID = dbid_of_target ;
  4. 查询在步骤1中获取DBID的数据库当前版本的记录。

    要获取有关目标数据库当前化身的信息，请指定目标数据库DB_KEY值并使用RC_DATABASE_INCARNATION视图执行连接。使用WHERE条件指定CURRENT_INCARNATION列值设置为YES 。例如，要获取有关DB_KEY值为1的目标数据库的当前版本中的备份集的信息，请按以下方式查询：

    SELECT BS_KEY，BACKUP_TYPE，COMPLETION_TIME FROM RC_DATABASE_INCARNATION i，RC_BACKUP_SET b我在哪里。DB_KEY = 1和我。DB_KEY = b。DB_KEY和我。CURRENT_INCARNATION ='是';

管理统一审计跟踪

默认情况下启用审核，但您可以控制何时将审核记录写入磁盘。

  • 何时何地创建审计记录？
    始终启用审核。Oracle数据库在审计的SQL语句的执行阶段期间或之后生成审计记录。
  • 被强制审计的活动
    始终会审计某些安全性敏感的数据库活动，并且无法禁用此类审核配置。
  • 游标如何影响审计？
    对于游标中每次执行可审计操作，Oracle数据库会将一条审计记录插入审计跟踪中。
  • 将统一审计跟踪记录写入AUDSYS架构
    Oracle数据库自动将审计记录写入AUDSYS模式中的内部关系表。
  • 将Unified Audit Trail Records写入SYSLOG或Windows事件查看器
    您可以通过设置初始化参数将统一审计跟踪记录写入SYSLOG或Windows事件查看器。
  • 将审计记录写入操作系统时
    在数据库表无法接受统一审计记录的情况下，这些记录将写入操作系统溢出审计文件（ .bin格式）。
  • 将操作系统审计记录移动到统一审计跟踪中
    已写入溢出审计文件的审计记录可以移动到统一审计跟踪数据库表中。
  • 禁用统一审核
    您可以禁用统一审核。
  • 使用Oracle Data Pump导出和导入Unified Audit Trail
    您可以在Oracle Database Pump导出和导入转储文件中包含统一审计跟踪。

何时何地创建审计记录？

始终启用审核。Oracle数据库在审计的SQL语句的执行阶段期间或之后生成审计记录。

当程序单元运行时，Oracle数据库会根据需要单独审核PL / SQL程序单元中的SQL语句。

为了提高统一审计跟踪的读取性能，统一审计记录立即写入磁盘到AUDSYS模式中的内部关系表。在之前的版本中，统一审计记录已写入SecureFile LOB。如果您已迁移到Oracle Database 12 c第 1版（12.1）中的统一审核，则可以手动将统一审核记录从SecureFile LOBS传输到此内部表。如果您使用的数据库版本支持分区表，则此内部表是分区表。在这种情况下，您可以使用DBMS_AUDIT_MGMT.ALTER_PARTITION_INTERVAL过程修改表的分区间隔。此表的分区版本基于EVENT_TIMESTAMP时间戳作为分区键，默认分区间隔为一个月。如果数据库版本不支持分区，则内部表是常规的非分区表。

审计跟踪记录的生成和插入与提交的用户事务无关。也就是说，即使回滚用户事务，审计跟踪记录仍将提交。

在数据库用户连接到数据库时生效的统一审计策略中的语句和权限审计选项在会话期间保持有效。会话已处于活动状态时，设置或更改语句或权限统一审核选项不会在该会话中生效。修改后的语句或权限审计选项仅在当前会话结束并创建新会话时生效。

相反，对架构对象审计选项的更改对于当前会话立即生效。

默认情况下，审计跟踪记录将写入SYSAUX表空间中的AUDSYS模式。您可以使用DBMS_AUDIT_MGMT.SET_AUDIT_TRAIL_LOCATION过程指定不同的表空间，包括加密的表空间。

为统一审计跟踪启用syslog和Windows事件查看器捕获

要将统一审计跟踪记录的子集写入UNIX syslog或Windows事件查看器，必须设置UNIFIED_AUDIT_SYSTEMLOG初始化参数。

在Microsoft Windows系统上， UNIFIED_AUDIT_SYSTEMLOG的默认值为FALSE 。在UNIX系统上，没有默认值。在Oracle数据库Real Application Clusters（Oracle RAC）环境中，将此参数设置为每个Oracle RAC实例上的相同值。
  1. 找到init.ora初始化文件，该文件默认位于$ORACLE_HOME/db s目录中。
  2. 编辑init.ora文件以包含UNIFIED_AUDIT_SYSTEMLOG参数。
    • 在Windows上，将UNIFIED_AUDIT_SYSTEMLOG设置为TRUE或FALSE 。 TRUE将syslog值写入Windows事件查看器; FALSE禁用该参数。例如：
      UNIFIED_AUDIT_SYSTEMLOG = TRUE
    • 在UNIX系统上，使用以下语法：
      UNIFIED_AUDIT_SYSTEMLOG =' facility_clause 。 priority_clause '

      在本规范中：

      • facility_clause是指您将编写审计跟踪记录的工具。有效的选择是USER和LOCAL 。如果输入LOCAL ，则可以选择附加0 - 7以指定syslog记录的本地自定义工具。

      • priority_clause指的是对记录进行分类的警告类型。有效选择包括NOTICE ， INFO ， DEBUG ， WARNING ， ERR ， CRIT ， ALERT和EMERG 。

    例如：
    UNIFIED_AUDIT_SYSTEMLOG ='LOCAL7。EMERG”
  3. 退出SQL * Plus。
  4. 将审计文件目标添加到syslog配置文件/etc/syslog.conf 。
    例如，假设您已将UNIFIED_AUDIT_SYSTEMLOG设置为LOCAL7.EMERG ，输入以下内容：
    local7.emerg /var/log/audit.log

    此设置将所有紧急消息记录到/var/log/audit.log文件中。

  5. 重新启动syslog记录器。
    $ / etc / rc.d / init.d / syslog restart

    现在，所有审计记录将通过syslog守护程序在文件/var/log/audit.log捕获。

  6. 重新登录数据库实例。
  7. 重启数据库。
    例如：
    关机立即启动

将审计记录写入操作系统时

在数据库表无法接受统一审计记录的情况下，这些记录将写入操作系统溢出审计文件（ .bin格式）。

在以下情况下写入数据库表的能力可能会失败：审计表空间处于脱机状态，表空间是只读的，表空间已满，数据库是只读的，依此类推。统一审核记录将继续写入操作系统溢出文件，直到操作系统磁盘空间已满。此时，当操作系统中没有空间用于审计记录时， ORA-02002 error while writing to audit trail错误时，用户可审计事务将因ORA-02002 error while writing to audit trail失败。为防止出现此问题，Oracle建议您定期清除审计跟踪。

将操作系统审计记录移动到统一审计跟踪中

已写入溢出审计文件的审计记录可以移动到统一审计跟踪数据库表中。

当数据库不可写时（例如在数据库装入期间），如果数据库已关闭，或者它是只读的，则Oracle数据库会将审计记录写入这些外部文件。这些外部文件的默认位置是$ORACLE_BASE/audit/$ORACLE_SID目录。

您可以通过运行DBMS_AUDIT_MGMT.LOAD_UNIFIED_AUDIT_FILES过程将文件加载到数据库中。请注意，如果要在外部文件中移动大量操作系统审核记录，性能可能会受到影响。

在数据库可写时，将这些文件中的审计记录移动到AUDSYS模式审计表：

  1. 以已被授予AUDIT_ADMIN角色的用户身份登录数据库实例。

    例如：

    CONNECT aud_admin输入密码： 密码已连接。

    在多租户环境中，登录要在其中将审计跟踪记录移动到统一审计跟踪的PDB。

    例如：

    CONNECT aud_admin @hrpdb输入密码： 密码已连接。

    要查找可用的PDB，请运行show pdbs命令。要检查当前PDB，请运行show con_name命令。

  2. 确保数据库是开放和可写的。

    对于非CDB体系结构，要查找数据库是否可打开，请查询V$DATABASE视图。

    例如，在CDB环境中：

    从V $ DATABASE中选择名称，OPEN_MODE;名称OPEN_MODE --------------- ---------- HRPDB READ WRITE

    在多租户环境中，您可以运行show pdbs命令以查找有关与当前实例关联的PDB的信息。

  3. 运行DBMS_AUDIT_MGMT.LOAD_UNIFIED_AUDIT_FILES过程。
    EXEC DBMS_AUDIT_MGMT.LOAD_UNIFIED_AUDIT_FILES;

审计记录立即加载到AUDSYS模式审计表中，然后从$ORACLE_BASE/audit/$ORACLE_SID目录中删除。

    立即关闭

要设置清除作业的时间戳，请执行以下操作：

  1. 以已被授予AUDIT_ADMIN角色的用户身份登录数据库实例。

    在多租户环境中，登录要在其中安排清除作业的根或PDB。在大多数情况下，您可能希望在各个PDB上安排清除作业。

    例如，要登录名为hrpdb的PDB：

    CONNECT aud_admin @hrpdb输入密码： 密码已连接。
  2. 运行DBMS_AUDIT_MGMT.SET_LAST_ARCHIVE_TIMESTAMP PL / SQL过程以设置时间戳。

    例如：

    BEGIN DBMS_AUDIT_MGMT.SET_LAST_ARCHIVE_TIMESTAMP（AUDIT_TRAIL_TYPE => DBMS_AUDIT_MGMT.AUDIT_TRAIL_UNIFIED，LAST_ARCHIVE_TIME => '12 -OCT-2013 06：30：00.00'，RAC_INSTANCE_NUMBER => 1，CONTAINER => DBMS_AUDIT_MGMT.CONTAINER_CURRENT）;结束; /

    • AUDIT_TRAIL_TYPE指定审计跟踪类型。DBMS_AUDIT_MGMT.AUDIT_TRAIL_UNIFIED为统一审计跟踪设置它。

      对于仍具有先前版本的传统审计数据的升级数据库：

      • DBMS_AUDIT_MGMT.AUDIT_TRAIL_AUD_STD用于传统的标准审计跟踪表AUD$ 。（此设置不适用于只读数据库。）

      • DBMS_AUDIT_MGMT.AUDIT_TRAIL_FGA_STD用于传统的细粒度审计跟踪表FGA_LOG$ 。（此设置不适用于只读数据库。）

      • DBMS_AUDIT_MGMT.AUDIT_TRAIL_OS用于具有.aud扩展名的传统操作系统审计跟踪文件。（此设置不适用于Windows事件日志条目。）

      • DBMS_AUDIT_MGMT.AUDIT_TRAIL_XML用于XML传统操作系统审计跟踪文件。

    • LAST_ARCHIVE_TIME以AUDIT_TRAIL_UNIFIED ， AUDIT_TRAIL_AUD_STD和AUDIT_TRAIL_FGA_STD以及AUDIT_TRAIL_OS和AUDIT_TRAIL_XML的本地时区中的YYYY-MM-DD HH:MI:SS.FF UTC（协调世界时）格式指定时间戳。

    • RAC_INSTANCE_NUMBER指定Oracle RAC安装的实例编号。此设置与单实例数据库无关。如果指定了DBMS_AUDIT_MGMT.AUDIT_TRAIL_AUD_STD或DBMS_AUDIT_MGMT.AUDIT_TRAIL_FGA_STD审计跟踪类型，则可以省略RAC_INSTANCE_NUMBER参数。这是因为即使对于Oracle RAC安装，也只有一个AUD$或FGA_LOG$表。默认值为NULL 。您可以通过在SQL * Plus中发出SHOW PARAMETER INSTANCE_NUMBER命令来查找当前实例的实例编号。

    • CONTAINER将时间戳应用于多租户环境。DBMS_AUDIT_MGMT.CONTAINER_CURRENT指定当前的PDB; DBMS_AUDIT_MGMT.CONTAINER_ALL适用于多租户环境中的所有PDB。

    通常，在设置时间戳之后，可以使用DBMS_AUDIT_MGMT.CLEAN_AUDIT_TRAIL PL / SQL过程删除在时间戳日期之前创建的审核记录。

    • DBMS_AUDIT_MGMT.AUDIT_TRAIL_OS用于具有.aud扩展名的操作系统审计跟踪文件。（此设置不适用于Windows事件日志条目。）

    • DBMS_AUDIT_MGMT.AUDIT_TRAIL_XML用于XML操作系统审计跟踪文件。

    • DBMS_AUDIT_MGMT.AUDIT_TRAIL_FILES用于操作系统和XML审计跟踪文件。

    • DBMS_AUDIT_MGMT.AUDIT_TRAIL_ALL用于所有审计跟踪记录，即数据库审计跟踪和操作系统审计跟踪类型。（此设置不适用于只读数据库。）

启用或禁用审计跟踪清除作业

DBMS_AUDIT_MGMT.SET_PURGE_JOB_STATUS过程启用或禁用审计跟踪清除作业。

在多租户环境中，运行DBMS_AUDIT_MGMT.SET_PURGE_JOB_STATUS过程取决于清除作业的位置，该位置由DBMS_MGMT.CREATE_PURGE_JOB过程的CONTAINER参数确定。如果已将CONTAINER设置为CONTAINER_ALL （在根目录中创建清除作业），则必须从根运行DBMS_AUDIT_MGMT.SET_PURGE_JOB_STATUS过程。如果已将CONTAINER设置为CONTAINER_CURRENT ，则必须从创建它的PDB运行DBMS_AUDIT_MGMT.SET_PURGE_JOB_STATUS过程。

  • 要启用或禁用审计跟踪清除作业，请使用DBMS_AUDIT_MGMT.SET_PURGE_JOB_STATUS PL / SQL过程。

    例如，假设您已在hrpdb PDB中创建了清除作业：

    CONNECT aud_admin @hrpdb输入密码： 密码已连接。BEGIN DBMS_AUDIT_MGMT.SET_PURGE_JOB_STATUS（AUDIT_TRAIL_PURGE_NAME =>'Audit_Trail_PJ'，AUDIT_TRAIL_STATUS_VALUE => DBMS_AUDIT_MGMT.PURGE_JOB_ENABLE）;结束; /

在这个例子中：

  • AUDIT_TRAIL_PURGE_NAME指定名为Audit_Trail_PJ的清除作业。要查找现有清除作业，请查询DBA_AUDIT_MGMT_CLEANUP_JOBS数据字典视图的JOB_NAME和JOB_STATUS列。

  • AUDIT_TRAIL_STATUS_VALUE接受以下任一属性：

    • DBMS_AUDIT_MGMT.PURGE_JOB_ENABLE启用指定的清除作业。

    • DBMS_AUDIT_MGMT.PURGE_JOB_DISABLE禁用指定的清除作业。

清除存档时间戳设置

DBMS_AUDIT_MGMT.CLEAR_LAST_ARCHIVE_TIMESTAMP过程可以清除存档时间戳设置。

要查找审计跟踪日志清除的历史记录，可以使用以下条件查询UNIFIED_AUDIT_TRAIL数据字典视图： OBJECT_NAME是DBMS_AUDIT_MGMT ， OBJECT_SCHEMA是SYS ， SQL_TEXT设置为LIKE %DBMS_AUDIT_MGMT.CLEAN_AUDIT_TRAIL% 。

  • 要清除存档时间戳设置，请使用DBMS_AUDIT_MGMT.CLEAR_LAST_ARCHIVE_TIMESTAMP PL / SQL过程指定审计跟踪类型，并为多租户环境指定容器类型。

    例如，假设您已在hrpdb PDB中创建了清除作业：

    CONNECT aud_admin @hrpdb输入密码： 密码已连接。BEGIN DBMS_AUDIT_MGMT.CLEAR_LAST_ARCHIVE_TIMESTAMP（AUDIT_TRAIL_TYPE => DBMS_AUDIT_MGMT.AUDIT_TRAIL_UNIFIED，CONTAINER => DBMS_AUDIT_MGMT.CONTAINER_CURRENT）;结束; /

在这个例子中：

  • 为统一审计跟踪AUDIT_TRAIL_TYPE 。如果AUDIT_TRAIL_TYPE属性设置为DBMS_AUDIT_MGMT.AUDIT_TRAIL_OS或DBMS_AUDIT_MGMT.AUDIT_TRAIL_XML ，则无法将RAC_INSTANCE_NUMBER设置为0 。如果将AUDIT_TRAIL_TYPE为DBMS_AUDIT_MGMT.AUDIT_TRAIL_UNIFIED则可以省略RAC_INSTANCE_NUMBER设置。

  • CONTAINER将时间戳应用于多租户环境。DBMS_AUDIT_MGMT.CONTAINER_CURRENT指定本地PDB; DBMS_AUDIT_MGMT.CONTAINER_ALL适用于所有数据库。
