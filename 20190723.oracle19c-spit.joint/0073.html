Oracle RAC环境中的Oracle Label Security Policy功能

在一个实例上进行的策略更改可立即用于Oracle Real Application Clusters（Oracle RAC）环境中的其他实例。

没有必要重新启动其他实例来获取更改。

在一个数据库实例上进行的重要更改会自动传播到其他实例。一个例子是创建一个新政策。另一个是改变政策选择。

传播这些变化确保了两个有价值的保护：

  • 该表的所有用户都遵循相同的策略

  • 如果任何实例失败，其他实例继续其工作将使用在该失败之前立即生效的相同策略和参数。因此，如果已启用或禁用策略，则在所有情况下都会将其视为此类策略。

如果管理员使用表D-1中列出的策略函数在一个实例中更改策略信息，Oracle Label Security将存储有关该函数调用更改的相关信息。新信息可立即供Oracle RAC中的其他活动实例使用，从而实现受影响策略的用户之间的一致性。

Oracle Label Security中的透明应用程序故障转移

会话信息保留在透明应用程序故障转移上。

通过表D-2中列出的会话功能对会话信息的任何更改都将保留在透明应用程序故障转移上。

例如，假设用户Scott使用默认标签Top Secret登录。如果他调用sa_session.set_label（）将其会话标签更改为Secret ，并且发生故障转移到另一个实例，他将看不到任何更改，但他的会话标签仍为Secret 。

保留当前用户会话信息意味着对用户可以看到或影响的数据的访问权限和限制保持不变。尽管存在故障转移，但用户只能查看并影响故障转移之前可访问的表和行。如果不是保留，则故障转移到另一个实例可能导致或使用户能够看到不同的数据集。

只要使用表D-2中列出的某个会话功能，Oracle Label Security就会存储有关该函数调用所更改内容的相关信息。

使用Optimal Flexible Architecture（OFA）运行多个版本。

最佳灵活体系结构（OFA）是一组用于高效可靠的Oracle数据库和Oracle Grid Infrastructure部署的配置指南。Oracle建议您根据Oracle数据库安装的OFA体系结构标准部署所有Oracle软件安装。遵循OFA标准有助于确保您的安装更易于维护，并且更容易从Oracle支持获得快速帮助。

OFA提供以下好处：

  • 在磁盘上组织大量复杂的软件和数据，有助于避免设备瓶颈和性能不佳

  • 促进日常管理任务，例如软件和数据备份功能，这些功能通常容易受到数据损坏

  • 简化多个Oracle数据库的管理

  • 有助于消除数据字典中可用空间的碎片，隔离其他碎片，并有助于最大限度地减少资源争用

  • 协助数据库管理员部署有效的企业数据管理策略

如果您当前没有使用OFA标准，那么切换到OFA标准涉及修改目录结构和重定位数据库文件。

关于最优灵活架构标准

Oracle Optimal Flexible Architecture（OFA）规则可帮助您组织数据库软件并配置数据库，以允许不同用户拥有的不同版本的多个数据库共存。

在早期的Oracle数据库版本中，OFA规则通过隔离碎片和最小化争用来提供最佳系统性能。在当前版本中，OFA规则提供数据库管理和支持的一致性，并简化扩展或添加数据库或添加其他硬件。

默认情况下，Oracle Universal Installer将Oracle数据库组件放置在目录位置，并具有符合OFA规则的权限。Oracle建议您根据OFA准则在安装介质上配置所有Oracle组件。

Oracle建议您接受OFA默认值。如果数据库很大，或者您计划拥有多个数据库，那么遵循OFA规则尤其有价值。

注意：

OFA通过其自动诊断存储库（ADR）诊断数据帮助识别ORACLE_BASE，以正确收集事件。

关于多个Oracle Homes支持

Oracle数据库支持多个Oracle主目录。您可以在同一系统上，在不同的Oracle主目录中多次安装此版本或更早版本的软件。

仔细选择安装点名称可以使Oracle软件更易于管理。按照Optimal Flexible Architecture（OFA）规则配置多个Oracle主目录具有以下优势：

  • 您可以在同一系统上，在不同的Oracle主目录中多次安装此发行版或该软件的早期版本。但是，您无法将一个Oracle数据库版本的产品安装到另一个版本的Oracle主目录中。例如，您无法将Oracle Database 19c软件安装到现有的Oracle 18c Oracle主目录中。

  • 不同用户拥有的不同版本的多个数据库可以同时共存。

  • 您必须在与早期版本的Oracle数据库分开的新Oracle主目录中安装新的Oracle数据库发行版。

    您无法在一个Oracle主目录中安装多个版本。Oracle建议您根据Optimal Flexible Architecture（OFA）指南为每个版本创建单独的Oracle Database Oracle主目录。

  • 在生产中，Oracle数据库服务器软件版本必须与前几个数字（主要版本，维护版本和修补程序版本号）的Oracle数据库字典版本相同。

  • 后来的Oracle Database版本可以访问早期的Oracle Database版本。但是，此访问权限仅用于升级。例如，如果18c数据库以升级模式启动，Oracle Database 19c可以访问Oracle数据库18c。

  • 如果两个产品处于同一版本级别，则Oracle Database Client可以安装在同一Oracle数据库主目录中。例如，您可以将Oracle Database Client 12.2.0.1安装到现有的Oracle Database 12.2.0.1主目录中，但不能将Oracle Database Client 12.2.0.1安装到现有的Oracle Database 12.1.0.2主目录中。如果在安装客户端之前应用修补程序集，则必须再次应用修补程序集。

  • 目录和文件的结构化组织以及数据库文件的一致命名简化了数据库管理。

  • 当数据库管理员添加，移动或删除Oracle主目录时，登录主目录不会有风险。

  • 您可以在与生产数据库所在的Oracle主目录不同的目录中测试Oracle主目录中的软件升级。

对于要被视为JSON数据的表列，它必须具有is json检查约束。但是在视图的情况下，以下任何一个条件都足以将列视为JSON数据：

  • 底层数据有一个is json检查约束。

  • 该列来自使用SQL / JSON函数json_query 。

  • 该列来自使用JSON生成函数，例如json_object 。

  • 该列使用SQL函数treat关键字AS JSON 。

如果is json检查约束确定表列是JSON数据，并且如果稍后停用该约束，则该列仍然在视图中列出。如果删除了检查约束，则会从视图中删除该列。

注意：

如果检查约束将条件is json与使用逻辑条件OR另一个条件相结合 ，则该列不会在视图中列出。在这种情况下，不确定列中的数据是否为JSON数据。例如，约束jcol is json OR length(jcol) < 1000 不能确保列jcol中的数据是JSON数据。

使用不区分大小写的密码版本检查帐户

使用这些过程可以确定要升级的Oracle数据库是否具有使用不区分大小写的密码版本的帐户或配置参数。

默认情况下，在Oracle Database 12 c第 2版（12.2）及更高版本中，不会生成或允许10G密码版本。

如果未将SQLNET.ALLOWED_LOGON_VERSION_SERVER设置为允许不区分大小写的版本的许可身份验证协议，并且您不希望将使用不区分大小写的密码版本进行身份验证的用户帐户锁定在数据库之外，则必须识别受影响的帐户，并且确保他们使用区分大小写的密码版本。

示例2-1查找使用不区分大小写（10G）版本的用户帐户

以管理用户身份登录SQL * Plus，并输入以下SQL查询：

来自DBA_USERS的SELECT USERNAME，PASSWORD_VERSIONS;

以下结果显示了帐户的密码版本：

USERNAME PASSWORD_VERSIONS ------------------------------ ----------------- JONES 10G 11G 12C ADAMS 10G 11G CLARK 10G 11G PRESTON 11G BLAKE 10G

在此示例中，使用的每个用户帐户密码验证版本的背景不同：

  • JONES是在Oracle Database 10G中创建的，当SQLNET.ALLOWED_LOGON_VERSION_SERVER参数的设置设置为8时， JONES的密码在Oracle Database 12C重置。因此，此密码重置创建了所有三个版本。11G和12C使用区分大小写的密码。

  • ADAMS和CLARK最初是使用10G版本创建的，然后是从早期版本导入的11G版本。然后，这些帐户密码在11G重置，不推荐使用的参数SEC_CASE_SENSITIVE_LOGON设置为TRUE。

  • BLAKE的密码是使用10G版本创建的，密码尚未重置。因此，用户BLAKE继续使用10G密码版本，该版本使用不区分大小写的密码。

用户BLAKE在升级之前只有10G密码版本：

SQL> SELECT USERNAME，PASSWORD_VERSIONS FROM DBA_USERS; USERNAME PASSWORD_VERSIONS ------------------------------ ----------------- BLAKE 10G

如果在不采取任何进一步操作的情况下升级到新的Oracle数据库版本，则此帐户将无法访问。确保在升级之前未将系统配置为独占模式（通过将SQLNET.ORA参数SQLNET.ALLOWED_LOGON_VERSION_SERVER设置为更宽松的身份验证模式）。

示例2-2使用不区分大小写的密码修复帐户

完成以下过程：

  1. 使用以下SQL查询查找仅具有10G密码版本的帐户：

    从DBA_USERS中选择USERNAME，其中（PASSWORD_VERSIONS ='10G'或PASSWORD_VERSIONS ='10G HTTP'）和USERNAME <>'ANONYMOUS';

  2. 通过将SQLNET.ORA参数SQLNET.ALLOWED_LOGON_VERSION_SERVER的设置编辑为适合受影响帐户的级别，配置系统以使其不以独占模式运行。例如：

    SQLNET.ALLOWED_LOGON_VERSION_SERVER = 11

    进行此更改后，继续升级。

  3. 升级完成后，使用以下命令语法使您在步骤1中找到的帐户失效，其中username是在步骤1中从查询返回的用户的名称：

    ALTER USER用户username PASSWORD EXPIRE;

  4. 询问已过期密码的用户是否登录。

  5. 当这些用户登录时，系统会提示他们重置密码。除了10G密码版本之外，系统还会在内部为其帐户生成缺少的11G和12C密码版本。10G密码版本仍然存在，因为系统正在允许模式下运行。

  6. 确保用户连接的客户端软件具有O5L_NP功能标志。

    注意：

    所有Oracle Database 11.2.0.4及更高版本的客户端以及所有Oracle Database 12.1及更高版本的客户端都具有O5L_NP功能。其他客户端需要CPUOct2012补丁才能获取O5L_NP功能。

    O5L_NP功能标志记录在Oracle Database Net Services Reference中的参数SQLNET.ALLOWED_LOGON_VERSION_SERVER 。

  7. 在所有客户端都具有O5L_NP功能后，使用以下过程将服务器安全性提升回独占模式：

    1. 从实例初始化文件中删除SEC_CASE_SENSITIVE_LOGON设置，或将SEC_CASE_SENSITIVE_LOGON实例初始化参数设置为TRUE 。例如：

      SEC_CASE_SENSITIVE_LOGON = TRUE

    2. 卸下SQLNET.ALLOWED_LOGON_VERSION_SERVER从服务器参数SQLNET.ORA文件，或者通过改变的值设置回独占模式SQLNET.ALLOWED_LOGON_VERSION_SERVER在服务器SQLNET.ORA文件传回12 。例如：

      SQLNET.ALLOWED_LOGON_VERSION_SERVER = 12

  8. 使用以下SQL查询查找仍具有10G密码版本的帐户：

    从DBA_USERS中选择USERNAME，其中PASSWORD_VERSIONS类似于'％10G％'和USERNAME <>'ANONYMOUS';

  9. 使用步骤8中从查询返回的帐户列表使所有仍具有10G密码版本的帐户失效。使用以下语法使帐户过期，其中username是查询返回的列表上的名称：

    ALTER USER用户username PASSWORD EXPIRE;

  10. 请求您已过期帐户的用户登录其帐户。

    当用户登录时，系统会提示他们重置密码。系统内部仅为其帐户生成11G和12C密码版本。由于系统在独占模式下运行，因此不再生成10G密码版本。

  11. 通过重新运行步骤1中的查询，检查系统是否以安全模式运行。确保找不到用户。当查询找不到用户时，此结果表示系统中不存在10G密码版本。

SQL> SHOW PARAMETER SEC_CASE_SENSITIVE_LOGON NAME TYPE VALUE ------------------------------------ ------ ----- ------------------------------ sec_case_sensitive_logon boolean FALSE
您可以使用以下命令更改此参数：
SQL> ALTER SYSTEM SET SEC_CASE_SENSITIVE_LOGON = TRUE;系统改变了。

注意：

除非该参数的值SQLNET.ALLOWED_LOGON_VERSION_SERVER改为一个版本比更宽容12 ，如11 ，不要设置SEC_CASE_SENSITIVE_LOGON参数设置为FALSE。

Multiset运算符

Multiset运算符将两个嵌套表的结果组合到一个嵌套表中。

与multiset运算符相关的示例要求创建两个嵌套表并使用如下数据加载：

首先，制作名为customers_demo的oe.customers表的副本：

CREATE TABLE customers_demo AS SELECT * FROM customers;

接下来，创建一个名为cust_address_tab_typ的表类型。创建嵌套表列时将使用此类型。 

CREATE TYPE cust_address_tab_typ AS TABLE OF cust_address_typ; /

现在，在customers_demo表中创建两个嵌套表列：

ALTER TABLE customers_demo ADD（cust_address_ntab cust_address_tab_typ，cust_address2_ntab cust_address_tab_typ）NESTED TABLE cust_address_ntab STORE AS cust_address_ntab_store NESTED TABLE cust_address2_ntab STORE AS cust_address2_ntab_store;

最后，使用oe.customers表的cust_address列中的数据将数据加载到两个新的嵌套表列中：

更新customers_demo cd SET cust_address_ntab = CAST（MULTISET（SELECT cust_address FROM customers c WHERE c.customer_id = cd.customer_id）as cust_address_tab_typ）; UPDATE customers_demo cd SET cust_address2_ntab = CAST（MULTISET（SELECT cust_address FROM customers c WHERE c.customer_id = cd.customer_id）as cust_address_tab_typ）;

存储脚本中的替换变量

RMAN支持在存储的脚本中使用替换变量。 &1表示放置第一个值的位置， &2表示放置第二个值的位置，依此类推。必须引用特殊字符。

替换变量语法是& integer后跟可选的句点，例如&1.3 。可选句点是变量的一部分，并替换为值，从而使替换文本后面紧跟另一个整数。例如，如果将值mybackup传递给包含替换变量&1.3的命令文件，则替换的结果为mybackup3 。要创建结果mybackup.3 ，在&1..3使用两个句点。

使用替换变量创建存储脚本时，必须在创建时提供示例值。您可以在启动RMAN时USING子句提供这些值（请参阅RMAN ），或在提示时输入它们（请参阅示例2-74 ）。

使用不区分大小写的密码版本检查帐户

使用这些过程可以确定要升级的Oracle数据库是否具有使用不区分大小写的密码版本的帐户或配置参数。

默认情况下，在Oracle Database 12 c第 2版（12.2）及更高版本中，不会生成或允许10G密码版本。

如果未将SQLNET.ALLOWED_LOGON_VERSION_SERVER设置为允许不区分大小写的版本的许可身份验证协议，并且您不希望将使用不区分大小写的密码版本进行身份验证的用户帐户锁定在数据库之外，则必须识别受影响的帐户，并且确保他们使用区分大小写的密码版本。

示例8-1查找使用不区分大小写（10G）版本的用户帐户

以管理用户身份登录SQL * Plus，并输入以下SQL查询：

来自DBA_USERS的SELECT USERNAME，PASSWORD_VERSIONS;

以下结果显示了帐户的密码版本：

USERNAME PASSWORD_VERSIONS ------------------------------ ----------------- JONES 10G 11G 12C ADAMS 10G 11G CLARK 10G 11G PRESTON 11G BLAKE 10G

在此示例中，使用的每个用户帐户密码验证版本的背景不同：

  • JONES是在Oracle Database 10G中创建的，当SQLNET.ALLOWED_LOGON_VERSION_SERVER参数的设置设置为8时， JONES的密码在Oracle Database 12C重置。因此，此密码重置创建了所有三个版本。11G和12C使用区分大小写的密码。

  • ADAMS和CLARK最初是使用10G版本创建的，然后是从早期版本导入的11G版本。然后，这些帐户密码在11G重置，不推荐使用的参数SEC_CASE_SENSITIVE_LOGON设置为TRUE。

  • BLAKE的密码是使用10G版本创建的，密码尚未重置。因此，用户BLAKE继续使用10G密码版本，该版本使用不区分大小写的密码。

用户BLAKE在升级之前只有10G密码版本：

SQL> SELECT USERNAME，PASSWORD_VERSIONS FROM DBA_USERS; USERNAME PASSWORD_VERSIONS ------------------------------ ----------------- BLAKE 10G

如果在不采取任何进一步操作的情况下升级到新的Oracle数据库版本，则此帐户将无法访问。确保在升级之前未将系统配置为独占模式（通过将SQLNET.ORA参数SQLNET.ALLOWED_LOGON_VERSION_SERVER设置为更宽松的身份验证模式）。

  • 问题：服务器耗尽空闲物理内存导致的内存压力导致操作系统Swapper进程长时间运行将内存移动到磁盘。交换可防止时间关键的集群进程运行并最终导致节点被驱逐。

    解决方案： Memory Guard会提前检测内存压力并导致工作耗尽以释放内存。

  • 问题：专用互连上的网络拥塞可能导致时间关键的节点间或存储I / O具有过多的延迟或丢弃的数据包。这种类型的故障通常会累积，可以及早发现，纠正或缓解。

    解决方案：如果服务器配置中的更改导致此问题，则群集验证实用程序（CVU）会在问题持续一个多小时后检测到它。但是，Oracle Cluster Health Advisor会在几分钟内检测到该问题并提供纠正措施。

  • 问题：由拉线或网络接口卡（NIC）引起的专用互连网络故障可能会立即导致被逐出的节点。

    解决方案：虽然无法及早检测到这些类型的网络故障，但可以通过使用Cluster Health Monitor和Oracle Trace File Analyzer来查明故障时间和所涉及的网络接口，从而缩小原因。

数据库可用性问题示例

数据库可用性问题可能导致Oracle数据库或数据库的某个实例无响应，从而无法供用户使用。

此类问题的示例如下：

  • 问题：失控查询或挂起可以将关键数据库资源（例如锁，锁存器或CPU）拒绝到其他会话。拒绝关键数据库资源会导致数据库或数据库实例对应用程序无响应。

    解决方案： Hang Manager检测并自动解决这些类型的挂起。此外，Oracle Cluster Health Advisor还会检测，识别并通知数据库管理员此类挂起并提供适当的更正操作。

  • 问题：拒绝服务（DoS）攻击，漏洞或简单的软件错误可能导致数据库或数据库实例无响应。

    解决方案： Oracle ORAchk提供的已知问题及其解决方案的主动建议可以防止此类事件发生。如果不阻止这些问题，那么除了Cluster Health Monitor收集的数据之外，Oracle Trace File Analyzer自动收集日志可以加快纠正这些问题。

  • 问题：配置更改可能导致难以排除故障的数据库中断。例如，对oracle.bin文件的不正确权限可能会阻止创建会话进程。

    解决方案：使用Cluster Verification Utility和Oracle ORAchk加速识别和纠正这些类型的问题。您可以使用Oracle ORAchk生成差异报告，以查看两个报告的基线比较和差异列表。您还可以查看Cluster Verification Utility创建的配置报告，以验证您的系统是否满足Oracle安装的条件。

1.1.2性能问题

性能问题是威胁系统性能的运行时问题。

性能问题可能由软件问题（错误，配置问题，数据争用等）或客户端问题（需求，查询类型，连接管理等）引起。

服务器和数据库性能问题交织在一起，难以分离。根据它们的来源对它们进行分类更容易：数据库服务器或客户端。

数据库服务器性能问题的示例

  • 问题：偏离配置中的最佳实践可能会导致数据库服务器性能问题。

    解决方案： Oracle ORAchk会在Oracle ORAchk定期运行时检测配置问题，并通知数据库管理员相应的更正设置。

  • 问题：瓶颈资源或构造不良的SQL语句可能导致数据库服务器性能问题。

    解决方案： Oracle数据库服务质量（QoS）管理标记这些问题，并在问题使服务级别协议（SLA）面临风险时生成通知。Oracle Cluster Health Advisor会检测问题何时超出正常运行条件，并通知数据库管理员采取纠正措施。

  • 问题：会话可能导致其他会话减慢等待阻塞会话释放其资源或完成其工作。

    解决方案： Hang Manager检测到这些会话链并自动终止root持有者会话以缓解瓶颈。

  • 问题：未解决的已知问题或未修补的错误可能导致数据库服务器性能问题。

    解决方案：可以通过自动Oracle ORAchk报告检测这些问题，并使用相关修补程序或解决方法进行标记。Oracle ORAchk定期得到增强，可以在现有产品或新产品领域中包含新的关键问题。

数据库客户端引起的性能问题示例

  • 问题：当服务器托管的数据库实例多于其资源且客户端负载可以管理时，性能会因等待CPU，I / O或内存而受到影响。

    解决方案： Oracle ORAchk和Oracle数据库QoS管理检测这些问题何时是错误配置的结果，例如超额订购CPU，内存或后台进程。Oracle ORAchk和Oracle数据库QoS管理系统会通过纠正措施通知您。

  • 问题：配置错误的参数（如SGA和PGA分配，会话或进程数，CPU计数等）可能导致数据库性能下降。

    解决方案： Oracle ORAchk和Oracle Cluster Health Advisor分别检测设置和结果，并通过建议的纠正措施自动通知您。

  • 问题：客户端连接的激增可能会超出服务器或数据库容量，从而导致超时错误和其他性能问题。

    解决方案： Oracle数据库QoS管理和Oracle Cluster Health Advisor自动检测性能下降。此外，Oracle数据库QoS管理和Oracle Cluster Health Advisor会通过纠正措施通知您，以缓解瓶颈并恢复性能。

目的

FEATURE_DETAILS返回选择中每行的要素详细信息。返回值是一个XML字符串，用于描述最高值要素或指定feature_id的属性。

TOPN

DESC，ASC或ABS

返回的属性按重量排序。属性的权重表示其对要素值的正面或负面影响。正权重表示更高的特征值。负权重表示较低的特征值。

    • 
                     
  •

    • 
                  

                  
在对分区模型进行评分时， FEATURE_DETAILS函数的语法可以使用可选的GROUPING提示。请参阅GROUPING提示 。

语义

                  
您可以通过两种方式执行例程。您可以通过名称，使用routine_clause发出对例程本身的调用，也可以使用object_access_expression调用表达式类型内的例程。
                  

               

               
               

                  
routine_clause

                  
指定要调用的函数或过程的名称，或解析为函数或过程的同义词。

                  
当您调用类型的成员函数或过程时，如果第一个参数（ SELF ）是null IN OUT参数，则Oracle数据库将返回错误。如果SELF是null IN参数，则数据库返回null。在这两种情况下，都不会调用函数或过程。
                  

                  
对功能的限制

                  
如果例程是函数，则需要INTO子句。
                  

                  
模式

                  
指定独立例程或包含例程的包或类型所在的模式。如果未指定schema ，则Oracle数据库假定例程在您自己的模式中。
                  

                  
类型 或 包装

                  
指定定义例程的类型或包。

                  
@dblink

                  
在分布式数据库系统中，指定包含独立例程的数据库的名称，或包含该例程的包或函数。如果省略dblink ，则Oracle Database将在您的本地数据库中查找。

object_access_expression

                  
如果您有一个对象类型的表达式，例如类型构造函数或绑定变量，那么您可以使用这种表达式来调用该类型中定义的例程。在此上下文中， object_access_expression仅限于方法调用。

争论

                  
如果例程接受参数，则指定例程的一个或多个参数。您可以对argument使用位置，命名或混合表示法。例如，以下所有表示法都是正确的：
CALL my_procedure（arg1 => 3，arg2 => 4）
CALL my_procedure（3,4）CALL my_procedure（3，arg2 => 4）

  • 
                        
    CALL语句传递的参数的数据类型必须是SQL数据类型。它们不能是PL / SQL专用数据类型，例如BOOLEAN 。

    • 
                     
  • 
                        
    作为例程的IN OUT或OUT参数的任何argument必须对应于主机变量表达式。
                        
    
                     
    • 
                     
  • 
                        
    参数的数量（包括任何返回参数）限制为1000。
    
                     
    •

    •
