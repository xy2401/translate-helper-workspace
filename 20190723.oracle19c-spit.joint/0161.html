关于Real Application Security Diagnostics

Real Application Security使用跨越后端数据库，应用程序服务器和应用程序实例的集成基础架构。Real Application Security组件包括诊断功能，使您能够在Real Application Security系统中查找，诊断和解决问题。

Real Application Security诊断使用Oracle Database 12 c第1版（12.1）及更高版本中提供的数据库诊断框架（DFW）。功能诊断允许您跟踪，调查和解决功能故障。您可以使用异常状态转储，基于事件的跟踪或默认跟踪来研究和解决功能问题。通过性能诊断，您可以识别和解决性能问题。

如何检查哪些ACL与当前用户的行相关联

要查找哪些ACL与当前用户的特定行相关联，请使用ORA_GET_ACLIDS函数。ORA_GET_ACLIDS函数返回与当前应用程序用户的数据安全策略启用表的行实例关联的ACL IDS列表。如果已授予对当前行的访问权限，则此函数将捕获与匹配数据域约束关联的所有ACL标识符。有关参考信息，请参阅“ ORA_GET_ACLIDS函数 ” ，有关教程信息，请参阅“ 关于检查特权的ACL ” 。

如何在ACL中为用户授予特权

要查找是否在ACL中授予了权限，请使用ORA_CHECK_ACL函数。ORA_CHECKACL函数根据ACL列表检查应用程序用户是否具有查询的应用程序权限。如果已将指定的应用程序权限授予应用程序用户，则ORA_CHECKACL将返回1 。如果它们未被授予应用程序用户，则返回0 。有关参考信息，请参阅“ ORA_CHECK_ACL函数 ” ;有关教程信息，请参阅“ 关于检查特权的ACL ” 。

要列出与表的每一行关联的ACLID（例如， EMPLOYEE表），用户可以使用以下查询：

从EMPLOYEE emp中选择ORA_GET_ACLIDS（emp）;

要为EMPLOYEE表的每一行授予特权（例如SELECT ，列出结果，用户可以执行以下查询：

从EMPLOYEE emp中选择ORA_CHECK_ACL（ORA_GET_ACLIDS（emp），'SELECT'）;

关于异常状态转储

当一个异常发生时，Real Application Security组件的状态信息将转储到跟踪文件中。异常状态转储类似于飞机失事的碰撞现场证据。

失败（如内部错误或服务器崩溃）会导致为每个组件调用诊断数据提取（DDE）例程。这会将当前系统，会话和进程状态信息转储到跟踪文件中。您可以稍后使用转储到跟踪文件中的状态信息来分析故障原因。

关于基于事件的跟踪

事件- 基于跟踪可用于跟踪与特定Real Application Security组件相关的事件。基于事件的跟踪有助于跟踪导致失败的事件。例如，事件号46148用于跟踪应用程序会话事件，例如createSession和attachSession 。

关于内存中跟踪

在记忆中跟踪是一种主动跟踪机制，用于诊断间歇性和难以复制的错误。内存中跟踪机制在内存缓冲区中记录组件状态更改和事件。发生故障时，会将其转储到跟踪文件中。内存中跟踪类似于用于飞机失事调查的黑匣子数据。

关于统计

真实应用安全组件统计信息有助于识别Real Application Security系统中的性能问题。统计信息包括关键数据，如会话创建操作数，主要失效，角色启用操作等。

  1. 在属性文件中指定日志记录选项。例如，
    handlers = java.util.logging。ConsoleHandler .level = SEVERE java.util.logging。ConsoleHandler.level = FINEST java.util.logging。ConsoleHandler.formatter = java.util.logging。SimpleFormatter oracle.security.xs.level = FINEST

    java -Djava.util.logging.config.file = logging.properties

对于简单的数据库操作， V$SQL_MONITOR有一个用于并行执行协调器进程的条目和一个用于每个并行执行服务器进程的条目。每个条目在V$SQL_PLAN_MONITOR都有对应的条目。由于为并行执行SQL语句分配的进程正在协作执行相同的操作，因此这些条目共享相同的执行键（ SQL_ID ， SQL_EXEC_START和SQL_EXEC_ID的组合）。

当满足以下任一条件时，Oracle数据库会监视复合数据库操作：

  • 使用DBMS_SQL_MONITOR.BEGIN_OPERATION启动数据库操作，并且该操作消耗了至少5秒的CPU或I / O时间。

  • 通过在DBMS_SQL_MONITOR.BEGIN_OPERATION FORCE_TRACKING设置为Y来强制跟踪操作。

对于复合数据库操作，每一行都包含一个操作，该操作的统计信息是在SQL语句和PL / SQL子程序中累积的，这些子程序在操作的同一会话中运行。

V$SQL_MONITOR视图包含V$SQL可用的统计信息的子集。但是，与V$SQL不同，监视统计信息不会累积多次执行。相反， V$SQL_MONITOR一个条目专用于单个执行SQL语句。如果数据库监视同一SQL语句的两次执行，则每次执行在V$SQL_MONITOR都有一个单独的条目。

主键是SQL_ID ， SQL_EXEC_START和SQL_EXEC_ID列的组合。

V$SQL_MONITOR有一个用于并行执行协调器进程的条目，以及一个用于每个并行执行服务器进程的条目。每个条目在V$SQL_PLAN_MONITOR都有对应的条目。由于为并行执行SQL语句而分配的进程正在协作执行相同的操作，因此这些条目共享相同的执行键（复合SQL_ID ， SQL_EXEC_START和SQL_EXEC_ID ）。您可以聚合执行密钥以确定并行执行的总体统计信息。

当正在执行的SQL语句正在执行时， V$SQL_MONITOR中的统计信息通常几乎实时刷新，每秒一次。执行结束后，不会立即删除监控信息。相反，它保存在V$SQL_MONITOR中至少一分钟。在监视新语句时，最终将删除该条目以回收其空间。

18.1 Oracle Messaging Gateway先决条件

Messaging Gateway为每个Messaging Gateway代理使用一个Oracle Scheduler作业。如果JOB_QUEUE_PROCESSES数据库初始化参数的值为零，则不会运行任何Oracle Scheduler作业。如果该值不为零，则它实际上成为可以并发运行的Scheduler作业和作业队列作业的最大数量。

验证是否已设置非零值，并且该值足以容纳要启动的每个Messaging Gateway代理的Scheduler作业。

17.8.1 GetCapabilities操作（WCS）

GetCapabilities操作允许WCS客户端检索WCS服务器提供的服务和覆盖元数据。

所有WCS服务器必须为GetCapabilities请求实现KVP协议。用户通过使用KVP协议（HTTP GET请求）向URL发送GetCapabilities请求，开始与WCS服务器交互。例如：

http：// host：port / oraclespatial / wcs / <数据源名称> ？服务= WCS＆请求=获得性能

GetCapabilities操作返回描述服务的XML文档以及客户端可以请求的coverage的简要描述。客户端通常会运行GetCapabilities操作并缓存其结果以在整个会话中使用，或者将其重用于多个会话。

GetCapabilities响应包括以下内容：

  • WCS响应版。如果请求未指定所需的响应版本，则服务器返回支持的最新版本。

  • ServiceIdentification的配置文件列表标识OGC接口标准一致性类。

    <wcs：ServiceMetadata> <wcs：formatSupported> image / tiff </ wcs：formatSupported> <wcs：formatSupported> image / jp2 </ wcs：formatSupported> <wcs：formatSupported> application / x-ogc-nitf </ wcs：formatSupported > <wcs：formatSupported> application / x-ogc-aaigrid </ wcs：formatSupported> <wcs：formatSupported> image / png </ wcs：formatSupported> <wcs：formatSupported> image / jpeg </ wcs：formatSupported> </ wcs ：ServiceMetadata>

  • 扩展元素允许WCS扩展标准定义其各自的额外服务元数据。此元素包括16.8.2 12-039 OGC Web覆盖服务接口标准 - 扩展扩展元素，如：最近邻居，双线性，双二次，......它还包括16.8.1 11-053r1 OGC Web覆盖服务接口标准 - CRS扩展包括crsSupported元素。

17.8.2 DescribeCoverage Operation（WCS）

DescribeCoverage操作允许客户端请求WCS服务器提供的一个或多个coverage的详细元数据，并提供对域和范围集中预期的数据量的估计。DescribeCoverage请求提供覆盖标识符列表，并提示服务器为每个标识符返回相应coverage的描述。以下是XML / POST DescribeCoverage请求示例：

<wcs：DescribeCoverage xmlns：xsi =“http://www.w3.org/2001/XMLSchema-instance”xmlns：wcs =“http://www.opengis.net/wcs/2.0”xsi：schemaLocation =“http ：//schemas.opengis.net/wcs/2.0 ../wcsAll.xsd“service =”WCS“version =”2.0.1“> <wcs：CoverageId> C0001 </ wcs：CoverageId> <wcs：CoverageId> C0002 </ wcs：CoverageId> ...</ WCS：DescribeCoverage>

DescribeCoverage响应包含一个coverage元数据列表，一个用于请求中传递的每个coverage标识符。 Coverage元数据是一个XML文档，类型为gml:Grid for GridCoverages ，类型为gml:rectifiedGrid for RectifiedGridCoverages 。这些文档的xsd模式可以在http://schemas.opengis.net/wcs/2.0/中找到。

gml:Grid元素隐式定义网格，网格是由两组或更多组曲线组成的网络，其中每组的成员以算法的方式与其他组的成员相交。网格内的感兴趣区域以其gml:limits给出，其是矩形区域的对角相对角的网格坐标。 gml:axisLabels提供了网格轴的标签列表（ gml:axisName已被弃用）。 gml:dimension指定网格的维度。

gml:limits元素包含一个gml:GridEnvelope 。包络的gml:low和gml:high属性元素是整数列表，它们是坐标元组。坐标被测量为沿着每个轴的栅格原点与“矩形”感兴趣区域的对角相对角的偏移。

整流网格是网格，在网格坐标和外部坐标参考系统的坐标之间存在仿射变换。它通过指定网格“原点”的位置（在某些几何空间中）和指定帖子位置的矢量来定义。

17.8.3 GetCoverage操作（WCS）

GetCoverage操作通常在GetCapabilities和DescribeCoverage操作响应显示允许的请求和可用的数据之后运行。GetCoverage操作返回以众所周知的coverage格式编码的coverage（即，一组地理位置的值或属性）。

GetCoverage请求提示WCS服务处理从服务的产品中选择的特定覆盖范围并返回派生的覆盖范围。

WCS核心标准定义了域子集化操作，该操作从指定请求包络（“边界框”）内的覆盖范围传递相对于覆盖范围包络的所有数据 - 更准确地说，是请求包络与覆盖范围包络的交集。

域子集化细分为修剪和切片 。修剪操作标识尺寸以及下限和上限（两者都必须位于coverage的域内）并提供覆盖范围，其范围在指定的维度中减少到这些新的更窄的限制。生成的coverage的维度与输入coverage的维度相同。以下是DimensionTrim元素的示例：

<wcs：DimensionTrim> <wcs：Dimension> N </ wcs：Dimension> <wcs：TrimLow> 8.16270027015798 </ wcs：TrimLow> <wcs：TrimHigh> 8.34362402047258 </ wcs：TrimHigh> </ wcs：DimensionTrim>

域切片操作接收维度和位置（必须位于coverage的域内）并提供覆盖率，该覆盖率是在指定的切割位置获得的提供的覆盖率的切片。与原始覆盖范围相比，生成的覆盖范围减小了一个。

修剪和切片都可以在请求中组合，也可以根据需要组合在一起。但是，在任何请求中，每个维度最多可以应用一个操作。以下是DimensionSlice元素的示例：

<wcs：DimensionSlice> <wcs：Dimension> N </ wcs：Dimension> <wcs：SlicePoint> 8.16270027015798 </ wcs：SlicePoint> </ wcs：DimensionSlice>

将返回coverage的编码格式由format和mediaType elelemts的组合指定。支持的格式是服务器的Capabilities文档中列出的格式，如果配置了GDAL，则默认为application/gml+xml或image/jpeg 。例如：

<wcs：format> image / jpeg </ wcs：format> <wcs：mediaType> multipart / related </ wcs：mediaType>

D初始化参数

init.ora文件中的Oracle数据库初始化参数与网关初始化参数不同。使用特定于代理的机制在初始化参数文件中设置网关参数，或使用DBMS_HS包在Oracle数据字典中设置它们。

网关启动时，网关初始化参数文件必须可用。

以下主题包含可以为每个网关及其描述设置的网关初始化参数的列表。主题还描述了初始化参数文件语法。

初始化参数文件语法

本主题说明初始化参数文件的语法。

  • 该文件是一系列命令。

  • 每个命令应该从单独的一行开始。

  • 行尾被视为命令终止符（除非使用反斜杠进行转义）。

  • 如果初始化参数文件中存在语法错误，则所有设置都不会生效。

  • 设置参数值如下：

    [SET] [PRIVATE] 参数 = 值

    哪里：

    parameter是初始化参数名称。它是一个以字母开头的字符串，由字母，数字和下划线组成。初始化参数名称区分大小写。

    value是初始化参数值。它区分大小写。初始化参数值是：

    • 一串不包含任何反斜杠，空格或双引号（“）的字符。

    • 带引号的字符串，以双引号开头，以双引号结尾。以下内容可以在带引号的字符串中使用：

      • 反斜杠（\）是转义字符

      • \ n插入一个新行

      • \ t插入一个标签

      • \“插入双引号

      • \\插入一个反斜杠

      该行末尾的反斜杠继续下一行的字符串。如果反斜杠在任何其他字符之前，则忽略反斜杠。

    例如，要为代理启用跟踪，请按如下所示设置HS_FDS_TRACE_LEVEL初始化参数：

     HS_FDS_TRACE_LEVEL = ON

    SET和PRIVATE是可选关键字。您不能将它们用作初始化参数名称。大多数参数仅作为初始化参数需要，因此您通常不需要使用SET或PRIVATE关键字。如果未指定SET或PRIVATE ，则该参数仅用作代理的初始化参数。

    SET指定除了用作初始化参数之外，还将参数值设置为代理进程的环境变量。将SET用于驱动程序或非Oracle系统需要的参数值作为环境变量。

    PRIVATE指定初始化参数应该是代理的私有参数，不应上载到Oracle数据库。大多数初始化参数不应该是私有的。但是，如果您在初始化参数文件中存储密码等敏感信息，则可能不希望将其上载到服务器，因为上载时初始化参数和值未加密。将初始化参数设置为私有可防止上载发生，并且它们不会出现在动态性能视图中。仅当参数值包含敏感信息（如用户名或密码）时，才使用PRIVATE作为初始化参数。

    SET PRIVATE指定将参数值设置为代理进程的环境变量，并且也是私有的（不会传输到Oracle数据库，不会出现在动态性能视图或图形用户界面中）。

107.1.1 CLIENT_ID_STAT_DISABLE过程

此过程将禁用所有实例的统计信息累积，并从CLIENT_ID_STAT_ENABLE过程启用的V$CLIENT_STATS视图中删除累积结果。

句法

DBMS_MONITOR.CLIENT_ID_STAT_DISABLE（client_id IN VARCHAR2）;

参数

表107-2 CLIENT_ID_STAT_DISABLE过程参数

参数 描述

CLIENT_ID

禁用统计信息聚合的客户端标识符

例子

要禁用累积：

EXECUTE DBMS_MONITOR.CLIENT_ID_STAT_DISABLE（'janedoe'）;

107.1.2 CLIENT_ID_STAT_ENABLE过程

此过程启用给定客户端标识符的统计信息收集。

统计信息收集对于数据库是全局的，并且跨实例启动和重新启动是持久的也就是说，为同一数据库的所有实例启用统计信息，包括重新启动。可以通过V$CLIENT_STATS视图查看统计信息。

句法

DBMS_MONITOR.CLIENT_ID_STAT_ENABLE（client_id IN VARCHAR2）;

参数

表107-3 CLIENT_ID_STAT_ENABLE过程参数

参数 描述

CLIENT_ID

启用统计信息聚合的客户端标识符

例子

要为具有给定客户端ID的客户端启用统计信息累积：

EXECUTE DBMS_MONITOR.CLIENT_ID_STAT_ENABLE（'janedoe'）;

参数

表107-4 CLIENT_ID_TRACE_DISABLE过程参数

参数 描述

禁用SQL跟踪的客户端标识符

EXECUTE DBMS_MONITOR.CLIENT_ID_TRACE_DISABLE（'janedoe'）;

107.1.4 CLIENT_ID_TRACE_ENABLE过程

此过程将为数据库全局跟踪给定客户端标识符。

句法

DBMS_MONITOR.CLIENT_ID_TRACE_ENABLE（client_id IN VARCHAR2，等待IN BOOLEAN DEFAULT TRUE，绑定IN BOOLEAN DEFAULT FALSE，plan_stat IN VARCHAR2 DEFAULT NULL）;

参数

表107-5 CLIENT_ID_TRACE_ENABLE过程参数

参数 描述

CLIENT_ID

启用了SQL跟踪的数据库会话标识符

等待

如果为TRUE ，则跟踪中存在等待信息

结合

如果为TRUE ，则跟踪中存在绑定信息

plan_stat

我们转储行源统计信息的频率。值应该是' NEVER '，' FIRST_EXECUTION '（相当于NULL ）或' ALL_EXECUTIONS '。

使用说明

  • 跟踪将写入多个跟踪文件，因为多个Oracle影子进程可以代表给定的客户端标识符工作。

  • 对所有实例启用跟踪，并在重新启动时保持持久性。

例子

EXECUTE DBMS_MONITOR.CLIENT_ID_TRACE_ENABLE（'janedoe'，TRUE，FALSE）;
