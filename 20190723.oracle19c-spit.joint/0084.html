7.1.1支持的服务器池配置

在Oracle数据库服务质量（QoS）管理可以管理群集资源之前，必须将服务器池标记为性能策略中的托管。如果服务器池不由Oracle数据库QoS管理管理，则Oracle数据库QoS管理服务器不会看到服务器池。Oracle数据库QoS管理不管理未标记服务器池中的任何工作负载或资源。

您可以选择由Oracle Database QoS Management管理的服务器池，即使它们不包含任何数据库实例，或者当前大小为零。这使您可以在工作负载启动之前预配置服务器池以供Oracle Database QoS Management进行管理。此外，如果将服务器池配置为最小大小为零，则Oracle数据库QoS管理可以从该服务器池中删除服务器，以便为更高优先级的工作负载提供资源。在这种情况下，即使服务器池中没有要管理的Oracle数据库QoS管理的资源，仍会监视在该服务器池中运行的任何工作负载，并且如果可能，将资源分配给该服务器池以支持工作负载。

Oracle Database QoS Management的所有功能都可用于运行Oracle Database 12 c第 2版软件的Oracle RAC数据库。对于Oracle Database 11g第 2版和Oracle Database 12 c第 1版Oracle RAC数据库，您的Oracle RAC数据库必须是策略管理数据库，或者您只能在仅测量或监视（12.1.0.2）模式下使用Oracle数据库QoS管理。

对于策略管理的数据库，应将数据库服务创建为UNIFORM服务，这意味着该服务由在指定服务器池中运行的每个可用实例提供。如果您的应用程序需要SINGLETON服务，那么要在管理模式下使用Oracle数据库QoS管理，该服务必须在最大大小为1的服务器池中运行。如果在最大大小超过1的服务器池中使用SINGLETON服务，则Oracle数据库QoS管理会报告配置违规。如果为此数据库的管理模式选择了“仅测量”，则可以使用SINGLETON服务任何配置。

Oracle数据库QoS管理支持共享服务器池的多个数据库。如果您有多个使用相同服务器池的数据库，则使用服务器池的每个数据库必须启用Oracle数据库QoS管理（如果它们都已定义性能类）。Oracle数据库QoS管理还支持Oracle RAC One Node数据库（有时称为单例数据库 ），但如果管理其性能类而不是简单地测量或监视，则这些数据库必须使用最大大小为1的服务器池。

创建数据库时，数据库实例的CPU_COUNT初始化参数的缺省值设置为运行实例的每个节点上的物理CPU数量的值。如果在同一节点上有多个数据库实例，并且在管理模式下有性能类，则必须为每个实例调整CPU_COUNT的值，以便在节点上运行的每个实例的CPU_COUNT总和等于或小于该节点上的CPU的物理数量。此外，对于数据库的每个实例， CPU_COUNT的值必须相同。例如，对于sales数据库，你不能有CPU_COUNT一套四个的sales1实例和CPU_COUNT设置两个用于sales2例如，如果两个实例都在同一个服务器池。

从Oracle Database 12 c第 2版（12.2）开始，Oracle数据库QoS管理为多租户数据库提供全面支持。在多租户架构中，Oracle数据库充当多租户容器数据库（CDB）。CDB包括零个，一个或多个客户创建的可插拔数据库（PDB）。PDB是模式，模式对象和非模式对象的可移植集合，在Oracle Net客户端中显示为非CDB。 Oracle数据库之前，所有Oracle数据库12c的是非的CDB。

Oracle Multitenant选项使您可以合并数据和代码，而无需更改现有模式或应用程序。从连接Oracle Net的客户端看，PDB的行为与非CDB相同。作用于整个非CDB的操作在整个CDB上以相同的方式起作用，例如，在使用Oracle Data Guard和数据库备份和恢复时。因此，非CDB的用户，管理员和开发人员在合并数据库之后具有基本相同的体验。通过整合硬件和共享数据库内存和文件，可以降低硬件，存储，可用性和人工成本。例如，单个服务器上的100个PDB可以共享一个数据库实例和一组数据库文件，从而需要更少的硬件和更少的人员。

Oracle数据库QoS管理支持：

  • 可插拔数据库（PDB）中的模式合并，通过调整PDB中运行的性能类的CPU份额

  • 通过调整同一多租户容器数据库（CDB）中不同PDB之间的CPU份额进行数据库整合

  • 通过调整服务器池大小（如果使用策略管理的数据库）调整相同物理服务器上托管的所有CDB的CPU数量和CDB实例总数，进行多个CDB合并

此功能无缝集成到Oracle Enterprise Manager Cloud Control中的Oracle数据库QoS管理页面中。

7.1.3支持的服务配置

Oracle Database QoS管理需要Oracle Clusterware管理的数据库服务。Oracle Database QoS Management管理的所有工作负载都必须使用Oracle Clusterware管理的数据库服务连接到数据库。您不能使用默认数据库服务; Oracle Clusterware不管理默认数据库服务。

用于连接数据库的服务应该是UNIFORM 。如果您的应用程序需要SINGLETON服务，那么，要使用Oracle数据库QoS管理，该服务必须在最大大小为1的服务器池中运行。

Oracle RAC高可用性框架监视数据库及其服务，并使用快速应用程序通知（FAN）发送事件通知。Oracle Clusterware和Oracle Net Services根据服务配置中指定的规则为服务提供负载平衡。这些规则是：

  • 连接负载平衡目标 ：使用该实例的当前工作负载和连接类型（ LONG或SHORT ）将连接路由到实例，以确定哪个实例可以提供最佳性能。对于Oracle数据库QoS管理，连接负载平衡目标应设置为LONG ，这样，如果将新服务器分配给服务的服务器池，则新连接将更快地迁移到新服务器。通过将连接更快地迁移到新服务器，可以在所有可用实例之间更快地平衡工作负载，并且工作负载的响应时间可以更快地得到改善。

  • 运行时连接负载平衡目标 ：负载平衡顾问数据用于确定哪个实例最符合为服务指定的目标。这两个目标是SERVICE_TIME ，负载平衡咨询数据基于实例中完成的工作所用的时间，以及THROUGHPUT ，负载平衡咨询数据基于实例中完成工作的速率。对于Oracle数据库QoS管理，对于使用服务器池的所有数据库服务，运行时连接负载平衡目标应设置为SERVICE_TIME ，但可选的最大值为1的服务器池除外。

运行时连接负载平衡向连接池发送有关如何跨Oracle RAC数据库中的实例平衡连接请求的建议。负载平衡咨询还提供有关如何将传入工作定向到为该工作提供最佳服务质量的实例的建议。这最大限度地减少了以后重新安置工作的需要。

要配置服务的负载平衡目标，请使用服务器控制（SRVCTL）实用程序，如以下示例所示，或使用企业管理器：

srvctl修改服务-db db_name -service service_name -rlbgoal SERVICE_TIME -clbgoal LONG

7.1.4支持的工作负载和目标类型

查看Oracle数据库QoS管理支持的数据库工作负载和目标的类型。

在Oracle数据库QoS管理的初始版本中，仅支持联机事务处理（OLTP）工作负载。唯一受支持的性能目标是数据库请求的平均响应时间。Oracle数据库QoS管理旨在管理开放式工作负载或需求与响应时间无关的工作负载。

对应用程序工作负载的数据库请求的平均响应时间必须小于1秒，最好是平均响应时间小于0.5秒。性能类中的每个数据库请求在资源使用方面应该是同质的。如果工作负载中的数据库请求的子集使用的资源比其他请求多得多，那么您应该创建一个新的性能类来包含需要更多资源的数据库请求。

Oracle数据库QoS管理不支持涉及并行查询的工作负载。默认情况下，无论使用哪种服务连接到数据库，并行查询都在数据库的所有可用实例上运行;工作负载不包含或限制为仅在提供服务的那些实例上运行。出于类似的原因，Oracle数据库QoS管理不支持涉及大量涉及GV $视图查询的数据库请求的工作负载。

对于由Oracle Database QoS Management管理的工作负载，数据库连接必须使用由Oracle Clusterware管理的数据库服务。启动连接的客户端或应用程序必须是JDBC（胖或瘦）客户端或OCI客户端。使用与数据库的遗留连接的工作负载不由Oracle数据库QoS管理进行管理。

8.1.1叶级属性

对象类型类似于树结构，其中分支表示属性。作为对象的属性使用自己的属性生成子分支。

最终，每个分支都以一个内置类型的属性结束;例如NUMBER ， VARCHAR2或REF ，或集合类型，例如VARRAY或嵌套表。原始对象类型的每个叶级属性都存储在表列中。

不是集合类型的叶级属性称为对象类型的叶级标量属性 。

以下主题涉及“ 如何在表中存储对象 ”中对象表和关系表的讨论。

8.1.2行对象如何跨列拆分

在对象表中 ，Oracle数据库将每个叶级标量或REF属性的数据存储在单独的列中。

注意：

每个VARRAY也存储在一列中，除非它太大。Oracle数据库将嵌套表类型的叶级属性存储在与对象表关联的单独表中。您必须将这些表声明为对象表声明的一部分。

在对象表中检索或更改行对象的属性时，数据库会对表的列执行相应的操作。访问行对象本身的值会调用该类型的默认构造函数，使用对象表的列作为参数并生成对象的副本。

数据库将系统生成的对象标识符存储在隐藏列中。数据库使用对象标识符为对象构造REF 。

8.1.3具有列对象的表的隐藏列

当使用对象类型的列定义表（关系表）时，数据库会向表中添加对象类型的叶级属性的隐藏列。

每个对象类型列还有一个对应的隐藏列，用于存储列对象的NULL信息（即顶级对象和嵌套对象的原子空值）。

8.1.4可替换列和对象表的隐藏列

可替换的列或对象表不仅具有列的对象类型的每个属性的隐藏列，而且还具有对象类型的任何子类型中添加的每个属性的隐藏列。

隐藏列存储插入可替换列中的任何子类型实例的这些属性的值。

除了类型判别列和空图像列之外，以下内容与person_typ的可替换列相关联，由例8-1创建

  • person_typ每个属性的隐藏列： idno ， name和phone

  • person_typ的子类型的属性的隐藏列

创建子类型时，数据库会自动将子类型中新属性的隐藏列添加到包含新子类型的任何祖先类型的可替换列的表中。这些改进了表以存储新类型的数据。如果由于某种原因无法添加列，则回滚创建子类型。

使用带有VALIDATE选项的DROP TYPE删除子DROP TYPE时，数据库会自动删除对于不包含数据的子类型唯一的属性的隐藏列。如果这些列包含数据，则会引发错误。

8.1.6 REF的存储

当数据库将REF构造为行对象时，构造的REF由对象标识符（OID），对象表的一些元数据以及可选的ROWID 。

一个的尺寸REF中的列REF类型取决于与列相关联的存储要求，如下：

  • 如果该列声明为REF WITH ROWID ，则数据库将ROWID存储在REF列中。对于受约束的REF列中的对象引用，将忽略ROWID提示。

  • 如果将列声明为带有SCOPE子句的REF ，则由于省略了对象表元数据和ROWID ，该列会减少。范围内的REF长度为16个字节。

8.1.7嵌套表的内部布局

嵌套表的行存储在单独的存储表中。每个嵌套表列都有一个关联的存储表。存储表包含该列中所有嵌套表的所有元素。存储表具有隐藏的NESTED_TABLE_ID列，该列具有系统生成的值，允许Oracle数据库将嵌套的表元素映射回适当的行。

您可以通过使存储表按索引进行组织来加速检索整个集合的查询。在STORE AS子句中包含ORGANIZATION INDEX子句。

嵌套表类型可以包含对象或标量：

  • 如果元素是对象，则存储表就像一个对象表：对象类型的顶级属性成为存储表的列。但是，您无法将REF构造到嵌套表中的对象，因为嵌套表行没有对象标识符列。

  • 如果元素是标量，则存储表包含一个名为COLUMN_VALUE列，其中包含标量值。

21.1关于神经网络

Oracle Data Mining中的 神经网络专为分类和回归等挖掘功能而设计。

在机器学习中，人工神经网络是一种受生物神经网络启发的算法，用于估计或近似依赖于大量通常未知输入的函数。人工神经网络由大量互连的神经元组成，这些神经元在彼此之间交换消息以解决特定问题。他们通过实例学习并在学习过程中调整神经元之间连接的权重。神经网络能够解决各种任务，如计算机视觉，语音识别和各种复杂的业务问题。

21.1.1神经元和激活功能

神经元是神经网络的构建块。

神经元采用具有不同权重的一个或多个输入，并且具有取决于输入的输出。通过将每个神经元的输入与权重相加并将总和馈入激活函数来实现输出。

Sigmoid函数通常是激活函数的最常见选择，但也使用其他非线性函数，分段线性函数或阶梯函数。以下是激活功能的一些示例：

  • Logistic Sigmoid功能

  • 线性功能

  • Tanh功能

  • Arctan功能

  • 双极乙状结肠功能

21.1.2损失或成本函数

损失函数或成本函数是将一个或多个变量的一个或多个事件映射到直觉上表示与事件相关联的一些“成本”的实数的函数。

优化问题旨在最小化损失函数。根据问题的性质和数学需要选择损失函数的形式。

以下是不同场景的不同损失函数：

  • 二进制分类：交叉熵函数。

  • 多级分类：softmax功能。

  • 回归：平方误差函数。

21.1.3前向后向传播

了解前向传播。

前向传播通过对先前层神经元值进行加权求和并应用激活函数来计算损失函数值。向后传播计算相对于网络中所有权重的损失函数的梯度。利用在用户指定的区域内均匀分布的一组随机数（通过设置权重边界）或由相邻层中的节点数量（数据驱动）定义的区域来初始化权重。梯度被馈送到优化方法，该优化方法又使用它们来更新权重，以试图使损失函数最小化。

21.1.4优化求解器

理解优化求解器。

优化求解器用于搜索损失函数的最优解，以找到损失（成本）函数的极值（最大值或最小值）。

Oracle Data Mining实现了限制内存Broyden-Fletcher-Goldfarb-Shanno（L-BFGS）以及线搜索。L-BFGS是一种准牛顿法。该方法使用由梯度评估指定的秩1更新来近似Hessian矩阵。此方法仅需要有限的内存量。L-BFGS用于查找下降方向，线搜索用于查找适当的步长。保存在L-BFGS解算器中的历史副本数由LBFGS_HISTORY_DEPTH定义。当迭代次数小于历史深度时，由L-BFGS计算的Hessian是准确的。当迭代次数大于历史深度时，由L-BFGS计算的Hessian是近似值。因此，历史深度不能太小或太大，因为计算可能太慢。通常，该值介于3和10之间。

21.1.5正规化

理解正规化。

正规化是指引入额外信息以解决不适定问题或防止过度拟合的过程。当统计模型描述随机误差或噪声而不是基础关系时，可能会出现不适合或过度拟合。典型的正则化技术包括L1范数正则化，L2范数正则化和保持。

保留通常用于大型训练日期集，而L1范数正则化和L2范数正则化主要用于小型训练日期集。

21.1.6收敛检查

这将检查是否已达到最佳解决方案以及优化的迭代是否已结束。

在L-BFGS求解器中，收敛标准包括最大迭代次数，梯度的无穷范数和相对误差容限。对于保持正则化，收敛标准检查测试数据集的损失函数值，以及迄今为止学到的最佳模型。当模型在特定次数的迭代（由NNET_HELDASIDE_MAX_FAIL指定）变得更糟，或者损失函数接近于零，或者测试数据的相对误差小于容差时，终止训练。

21.1.7 LBFGS_SCALE_HESSIAN

定义LBFGS_SCALE_HESSIAN 。

它指定了如何在每次迭代开始时设置逆Hessian的初始近似值。如果该值设置为LBFGS_SCALE_HESSIAN_ENABLE ，那么我们用Oren-Luenberger缩放近似初始逆Hessian。如果它被设置为LBFGS_SCALE_HESSIAN_DISABLE ，那么我们在每次迭代开始时使用identity作为逆Hessian的近似值。

21.2神经网络的数据准备

了解为神经网络准备数据。

该算法自动将分类数据“爆炸”成一组二进制属性，每个类别值一个。Oracle Data Mining算法自动处理缺失值，因此不需要缺失值处理。

该算法自动用模式替换缺失的分类值，并用均值替换缺失的数值。神经网络需要数字输入的标准化。该算法使用z得分归一化。仅对二维数字列（非嵌套）进行标准化。规范化将数字属性的值放在相同的比例上，并防止具有较大原始比例的属性偏向解决方案。神经网络通过相应列中的最大绝对值来缩放嵌套列中的数值。

20.1关于神经网络

Oracle Data Mining中的 神经网络专为分类和回归等挖掘功能而设计。

在机器学习中，人工神经网络是一种受生物神经网络启发的算法，用于估计或近似依赖于大量通常未知输入的函数。人工神经网络由大量互连的神经元组成，这些神经元在彼此之间交换消息以解决特定问题。他们通过实例学习并在学习过程中调整神经元之间连接的权重。神经网络能够解决各种任务，如计算机视觉，语音识别和各种复杂的业务问题。

20.1.1神经元和激活功能

神经元是神经网络的构建块。

神经元采用具有不同权重的一个或多个输入，并且具有取决于输入的输出。通过将每个神经元的输入与权重相加并将总和馈入激活函数来实现输出。

Sigmoid函数通常是激活函数的最常见选择，但也使用其他非线性函数，分段线性函数或阶梯函数。以下是激活功能的一些示例：

  • Logistic Sigmoid功能

  • 线性功能

  • Tanh功能

  • Arctan功能

  • 双极乙状结肠功能

20.1.2损失或成本函数

损失函数或成本函数是将一个或多个变量的一个或多个事件映射到直觉上表示与事件相关联的一些“成本”的实数的函数。

优化问题旨在最小化损失函数。根据问题的性质和数学需要选择损失函数的形式。

以下是不同场景的不同损失函数：

  • 二进制分类：交叉熵函数。

  • 多级分类：softmax功能。

  • 回归：平方误差函数。

20.1.3前向后向传播

了解前向传播。

前向传播通过对先前层神经元值进行加权求和并应用激活函数来计算损失函数值。向后传播计算相对于网络中所有权重的损失函数的梯度。利用在用户指定的区域内均匀分布的一组随机数（通过设置权重边界）或由相邻层中的节点数量（数据驱动）定义的区域来初始化权重。梯度被馈送到优化方法，该优化方法又使用它们来更新权重，以试图使损失函数最小化。

20.1.4优化求解器

理解优化求解器。

优化求解器用于搜索损失函数的最优解，以找到损失（成本）函数的极值（最大值或最小值）。

Oracle Data Mining实现了限制内存Broyden-Fletcher-Goldfarb-Shanno（L-BFGS）以及线搜索。L-BFGS是一种准牛顿法。该方法使用由梯度评估指定的秩1更新来近似Hessian矩阵。此方法仅需要有限的内存量。L-BFGS用于查找下降方向，线搜索用于查找适当的步长。保存在L-BFGS解算器中的历史副本数由LBFGS_HISTORY_DEPTH定义。当迭代次数小于历史深度时，由L-BFGS计算的Hessian是准确的。当迭代次数大于历史深度时，由L-BFGS计算的Hessian是近似值。因此，历史深度不能太小或太大，因为计算可能太慢。通常，该值介于3和10之间。

20.1.5正规化

理解正规化。

正规化是指引入额外信息以解决不适定问题或防止过度拟合的过程。当统计模型描述随机误差或噪声而不是基础关系时，可能会出现不适合或过度拟合。典型的正则化技术包括L1范数正则化，L2范数正则化和保持。

保留通常用于大型训练日期集，而L1范数正则化和L2范数正则化主要用于小型训练日期集。

20.1.6收敛检查

这将检查是否已达到最佳解决方案以及优化的迭代是否已结束。

在L-BFGS求解器中，收敛标准包括最大迭代次数，梯度的无穷范数和相对误差容限。对于保持正则化，收敛标准检查测试数据集的损失函数值，以及迄今为止学到的最佳模型。当模型在特定次数的迭代（由NNET_HELDASIDE_MAX_FAIL指定）变得更糟，或者损失函数接近于零，或者测试数据的相对误差小于容差时，终止训练。

20.1.7 LBFGS_SCALE_HESSIAN

定义LBFGS_SCALE_HESSIAN 。

它指定了如何在每次迭代开始时设置逆Hessian的初始近似值。如果该值设置为LBFGS_SCALE_HESSIAN_ENABLE ，那么我们用Oren-Luenberger缩放近似初始逆Hessian。如果它被设置为LBFGS_SCALE_HESSIAN_DISABLE ，那么我们在每次迭代开始时使用identity作为逆Hessian的近似值。

20.2神经网络的数据准备

了解为神经网络准备数据。

该算法自动将分类数据“爆炸”成一组二进制属性，每个类别值一个。Oracle Data Mining算法自动处理缺失值，因此不需要缺失值处理。

该算法自动用模式替换缺失的分类值，并用均值替换缺失的数值。神经网络需要数字输入的标准化。该算法使用z得分归一化。仅对二维数字列（非嵌套）进行标准化。规范化将数字属性的值放在相同的比例上，并防止具有较大原始比例的属性偏向解决方案。神经网络通过相应列中的最大绝对值来缩放嵌套列中的数值。

承诺

目的

使用COMMIT语句结束当前事务并在事务中执行永久性所有更改。事务是Oracle数据库视为单个单元的一系列SQL语句。此语句还会擦除事务中的所有保存点并释放事务锁。

在您提交交易之前：

  • 您可以通过查询已修改的表来查看在事务期间所做的任何更改，但其他用户无法查看更改。提交事务后，更改对提交后执行的其他用户的语句可见。

  • 您可以使用ROLLBACK语句回滚（撤消）在事务期间所做的任何更改（请参阅ROLLBACK ）。

在以下情况下，Oracle数据库会发出隐式COMMIT ：

  • 在任何语法上有效的数据定义语言（DDL）语句之前，即使语句导致错误

  • 在完成没有错误的任何数据定义语言（DDL）语句之后

您还可以使用此语句：

  • 手动提交不确定的分布式事务

  • 终止由SET TRANSACTION语句开始的只读事务

Oracle建议您在断开与Oracle数据库的连接之前，使用COMMIT或ROLLBACK语句显式结束应用程序中的每个事务，包括最后一个事务。如果您没有显式提交事务并且程序异常终止，则会自动回滚最后一个未提交的事务。

大多数Oracle实用程序和工具的正常退出会导致提交当前事务。从Oracle预编译器程序正常退出不会提交事务，并依赖Oracle数据库回滚当前事务。

写条款

使用此子句指定将提交操作生成的重做信息写入重做日志的优先级。此子句可以通过减少延迟来提高性能，从而消除等待重做日志的I / O.在满足以下条件的情况下，使用此子句可以改善响应时间要求严格的环境中的响应时间：

  • 更新事务的数量很大，需要将重做日志频繁写入磁盘。

  • 应用程序可以容忍丢失异步提交的事务。

  • 等待重做日志写入发生的延迟对整体响应时间有很大影响。

您可以指定WAIT | NOWAIT和IMMEDIATE | BATCH条款的任何顺序。

注意：

等待| NOWAIT

使用这些子句指定控件何时返回给用户。

  • WAIT参数确保仅在相应的重做在联机重做日志中持久化后才会返回提交。无论是在BATCH模式还是IMMEDIATE模式下，当客户端从此COMMIT语句成功返回时，该事务已被提交到持久介质。成功写入日志后发生崩溃可能会阻止成功消息返回到客户端。在这种情况下，客户端无法判断事务是否已提交。

  • 无论对重做日志的写入是否已完成， NOWAIT参数都会使提交返回给客户端。此行为可以增加事务吞吐量。使用WAIT参数，如果收到提交消息，则可以确保没有数据丢失。

注意：

使用NOWAIT ，在收到提交消息之后但在写入重做日志记录之前发生崩溃可能会错误地向事务指示其更改是持久的。

如果省略此子句，则事务将以WAIT行为提交。

立即|批量

使用这些子句指定何时将重做写入日志。

  • IMMEDIATE参数使日志写入器进程（LGWR）将事务的重做信息写入日志。此操作选项强制执行磁盘I / O，因此可以降低事务吞吐量。

  • BATCH参数使重做被重缓冲到重做日志，以及其他同时执行的事务。收集到足够的重做信息后，将启动重做日志的磁盘写入。此行为称为“组提交”，因为多个事务的重做在单个I / O操作中写入日志。

如果省略此子句，则事务将以IMMEDIATE行为提交。

8.5.2构造函数和类型演化

属性值构造函数可以省去为类型定义自己的构造函数的麻烦。但是，您必须为类型中声明的每个属性提供值，否则构造函数调用将无法编译。

如果稍后进化类型，此要求可能会产生问题，特别是因为属性值构造函数是隐式的，并且在代码中不可见，这与用户定义的构造函数不同。更改类型的属性时，类型的属性值构造函数也会更改。如果添加属性，则更新的属性值构造函数需要新属性的值;否则，现有代码中的任何属性值构造函数调用都会失败。

8.5.3用户定义构造函数的优点

与属性值构造函数不同，用户定义的构造函数不需要为类型的每个属性显式设置值。

用户定义的构造函数可以包含任意类型的任意数量的参数，并且这些参数不需要直接映射到类型属性。定义构造函数时，可以将属性初始化为任何适当的值。对于您未提供值的任何属性，系统初始化为NULL 。

如果您进化类型 - 例如，通过向该用户定义的构造函数添加属性调用，则不需要更改。当类型发展时，用户定义的构造函数不会自动修改，因此它们的签名保持不变。但是，如果不希望将new属性初始化为NULL ，则可能需要更改构造函数的定义。

8.5.5重载和隐藏的构造函数

您可以像其他类型方法一样重载用户定义的构造函数。

用户定义的构造函数不是继承的，因此在超类型中定义的用户定义的构造函数不能隐藏在子类型中。但是，如果用户定义的构造函数的签名与属性值构造函数的签名完全匹配，则用户定义的构造函数会隐藏并因此取代其类型的属性值构造函数。

对于要匹配的签名，用户定义的构造函数的参数的名称和类型（在隐式SELF参数之后）必须与该类型的属性的名称和类型相同。用户定义的构造函数的参数模式（在隐式SELF参数之后）必须为IN 。

如果属性值构造函数未被具有相同名称和签名的用户定义构造函数隐藏，则仍可以调用属性值构造函数。

请注意，如果您进化类型 - 例如，通过添加属性 - 类型的属性值构造函数的签名会相应地更改。这可能导致以前隐藏的属性值构造函数再次变得可用。

7.1关于群集

聚类分析查找彼此相似的数据对象集群。群集的成员彼此更像是与其他群集的成员相似。不同的集群可以具有共同的成员。聚类分析的目标是找到高质量的聚类，使得聚类间相似性低并且聚类内相似性高。

聚类与分类一样，用于分割数据。与分类不同，聚类模型将数据分段为先前未定义的组。分类模型通过将数据分配给在目标中指定的先前定义的类来对数据进行分段。群集模型不使用目标。

聚类对于探索数据很有用。当有很多情况并且没有明显的分组时，您可以使用聚类算法查找自然分组。

聚类可以作为一个有用的数据预处理步骤，以识别可以构建监督模型的同类组。

您也可以使用群集异常检测。将数据分段到群集后，您会发现某些情况不适合任何群集。这些情况是异常或异常值。

7.1.1如何计算集群？

有几种不同的聚类计算方法。Oracle Data Mining支持以下方法：

  • 基于密度 ：这种类型的聚类查找数据的基础分布，并估计数据中高密度区域如何与分布中的峰值相对应。高密度区域被解释为集群。基于密度的聚类估计是概率性的。

  • 基于距离 ：这种类型的聚类使用距离度量来确定数据对象之间的相似性。距离度量标准测量群集中的实际案例与群集的原型案例之间的距离。原型案例称为质心 。

  • 基于网格 ：这种类型的聚类将输入空间划分为超矩形单元格，并识别相邻的高密度单元格以形成聚类。

7.1.2评分新数据

尽管群集是一种无监督的挖掘功能，但Oracle Data Mining支持群集的评分操作。以概率方式对新数据进行评分。

7.1.3分层聚类

Oracle Data Mining支持的聚类算法执行分层聚类。叶簇是算法生成的最终簇。层次结构中较高的集群是中间集群。

7.1.3.1规则

规则描述了每个数据簇。规则是一种条件语句，用于捕获用于将父集群拆分为子集群的逻辑。规则描述了为群集分配概率的条件。

7.1.3.2支持和信心

支持和 置信度是描述群集规则和案例之间关系的指标。支持是规则所适用的案例的百分比。置信度是指由此规则描述的案例实际分配给集群的概率。

7.2评估聚类模型

由于已知的类不用于聚类，因此聚类的解释会带来困难。您如何知道集群是否可以可靠地用于业务决策？

Oracle Data Mining集群模型支持高度模型透明度。您可以通过检查聚类算法生成的信息来评估模型：例如，基于距离的聚类的质心。而且，因为聚类过程是在分层结构中，您可以评估与每个集群在层次结构中的位置相关的规则和其他信息。

6.1关于群集

聚类分析查找彼此相似的数据对象集群。群集的成员彼此更像是与其他群集的成员相似。不同的集群可以具有共同的成员。聚类分析的目标是找到高质量的聚类，使得聚类间相似性低并且聚类内相似性高。

聚类与分类一样，用于分割数据。与分类不同，聚类模型将数据分段为先前未定义的组。分类模型通过将数据分配给在目标中指定的先前定义的类来对数据进行分段。群集模型不使用目标。

聚类对于探索数据很有用。当有很多情况并且没有明显的分组时，您可以使用聚类算法查找自然分组。

聚类可以作为一个有用的数据预处理步骤，以识别可以构建监督模型的同类组。

您也可以使用群集异常检测。将数据分段到群集后，您会发现某些情况不适合任何群集。这些情况是异常或异常值。

6.1.1如何计算集群？

有几种不同的聚类计算方法。Oracle Data Mining支持以下方法：

  • 基于密度 ：这种类型的聚类查找数据的基础分布，并估计数据中高密度区域如何与分布中的峰值相对应。高密度区域被解释为集群。基于密度的聚类估计是概率性的。

  • 基于距离 ：这种类型的聚类使用距离度量来确定数据对象之间的相似性。距离度量标准测量群集中的实际案例与群集的原型案例之间的距离。原型案例称为质心 。

  • 基于网格 ：这种类型的聚类将输入空间划分为超矩形单元格，并识别相邻的高密度单元格以形成聚类。

6.1.2评分新数据

尽管群集是一种无监督的挖掘功能，但Oracle Data Mining支持群集的评分操作。以概率方式对新数据进行评分。

6.1.3分层聚类

Oracle Data Mining支持的聚类算法执行分层聚类。叶簇是算法生成的最终簇。层次结构中较高的集群是中间集群。

6.1.3.1规则

规则描述了每个数据簇。规则是一种条件语句，用于捕获用于将父集群拆分为子集群的逻辑。规则描述了为群集分配概率的条件。

6.1.3.2支持和信心

支持和 置信度是描述群集规则和案例之间关系的指标。支持是规则所适用的案例的百分比。置信度是指由此规则描述的案例实际分配给集群的概率。

6.2评估聚类模型

由于已知的类不用于聚类，因此聚类的解释会带来困难。您如何知道集群是否可以可靠地用于业务决策？

Oracle Data Mining集群模型支持高度模型透明度。您可以通过检查聚类算法生成的信息来评估模型：例如，基于距离的聚类的质心。而且，因为聚类过程是在分层结构中，您可以评估与每个集群在层次结构中的位置相关的规则和其他信息。

该用于DRDA的Oracle数据库提供程序是一个网络前端，它使客户端程序可以使用分布式关系数据库体系结构（DRDA）协议。

用于DRDA的Oracle数据库提供程序实现了完整DRDA第4版规范的大部分，以及其他DRDA服务器产品的若干方面（例如 IBM的DB2）兼容性。它是一个独立于数据库的协议，它仅提供Oracle数据库提供的功能。用于DRDA产品的Oracle数据库提供程序使现有DB2应用程序客户能够在迁移时利用其当前在应用程序技术方面的投资 DB2服务器。

    请参阅“ 配置高频自动优化器统计信息收集 ” 。

    分区移动，合并和合并操作维护全局和分区级统计信息。

    请参阅“ 在线统计信息收集 ” 。

    没有SELECT_CATALOG_ROLE数据库用户可以为自己的SQL语句生成和查看SQL Monitor报告，包括执行计划和性能指标。

    请参阅“ 生成和访问SQL监视器报告 ” 。

    在某些情况下，数据库可以通过访问表统计信息而不是表本身来满足简单的聚合查询，从而显着减少查询响应时间。

    请参阅“ 基于统计的查询转换 ” 。

    请参阅“ 关于隔离的SQL计划 ” 。

    请参阅“ 自动索引 ” 。

32.1使用DBMS_AW_STATS

多维数据集和维度是支持多维分析的一流数据对象。它们存储在称为分析工作区的容器中。Oracle数据库的OLAP选项提供了多维对象和分析。

优化程序统计信息用于为连接两个多维数据集视图或将多维数据集视图连接到表或表视图的查询创建执行计划。它们还用于查询重写到多维数据集物化视图。您只需为这些类型的查询生成统计信息。

针对单个多维数据集的查询不使用优化程序统计信息。这些查询在分析工作区内自动优化。

32.2.1分析程序

此过程生成多维数据集或多维数据集维度的优化程序统计信息。

这些统计信息用于生成一些执行计划，如“ 使用DBMS_AW_STATS ”中所述 。

对于多维数据集，统计信息适用于与多维数据集关联的所有度量和计算度量。这些统计包括：

  • 数据值的平均长度

  • 最大数据值的长度

  • 最小值

  • 不同值的数量

  • 空值的数量

对于维度，统计信息适用于维度及其属性，级别和层次结构。这些统计包括：

  • 值的平均长度

  • 最大值的长度

  • 最小值

  • 最大值

DBMS_AW_STATS.ANALYZE（inname IN VARCHAR2）;
