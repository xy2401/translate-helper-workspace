A.2.1在模式级别启用GeoRaster

必须为将使用GeoRaster功能的每个数据库架构启用GeoRaster。

默认情况下，在最初安装Oracle Spatial and Graph后，将禁用GeoRaster功能。GeoRaster只能在模式范围内启用（即不适用于整个数据库），并且必须为将使用GeoRaster功能的每个模式启用GeoRaster。

要启用GeoRaster，请对将启用GeoRaster 的每个架构执行以下步骤：

  1. 以该架构的用户身份连接到数据库。例如：
    CONNECT SCOTT / <password-for-scott>
  2. 输入以下语句：
    执行sdo_geor_admin.enableGeoRaster;
  3. 通过检查以下语句是否返回TRUE验证现在是否已启用GeoRaster：
    SELECT sdo_geor_admin.isGeoRasterEnabled FROM DUAL;
  4. 确保此模式的用户具有CREATE TRIGGER权限（GeoRaster正常工作所需的权限）。如果用户没有CREATE TRIGGER权限（或者您不知道），请以DBA用户身份连接并向用户授予该权限。例如：
    CONNECT / AS SYSDBA GRANT创建触发器TO scott;

如果的GeoRaster表已创建并用数据填充，那么数据库升级后的GeoRaster自动为表的模式启用，您不需要为架构重新启用的GeoRaster。（只需确保为用户授予CREATE TABLE权限。）

A.4增加纵坐标阵列的大小以支持非常大的几何

如果需要支持超过1,048,576个纵坐标的几何，则必须按照本节中的说明进行操作。但是，这样做涉及大量额外工作（运行脚本，迁移现有空间数据），某些数据库停机时间以及一些注意事项和限制。因此，除非您需要，否则不应执行本节中的操作。

要增加纵坐标数组的大小以支持最多10M纵坐标的几何，请按照下列步骤操作：

  1. 确保没有用户使用任何空间表或Oracle Spatial and Graph或Locator功能。
  2. 以具有SYSDBA权限的SYS用户身份（SYS AS SYSDBA）连接到数据库，并在出现提示时输入SYS帐户密码。
  3. 如果包含SDO_GEOMETRY列的表上存在任何sdoupggeom.sql化视图，请在执行下一步（运行sdoupggeom.sql ） 之前删除所有此类sdoupggeom.sql 。

    升级操作完成后，您可以重新创建这些物化视图。

  4. 输入以下语句：

    • Linux： @$ORACLE_HOME/md/admin/sdoupggeom.sql

    • Windows： @%ORACLE_HOME%\md\admin\sdoupggeom.sql

    sdoupggeom.sql脚本的一个操作是自动迁移所有空间数据以适应新的SDO_ORDINATE_ARRAY定义。此脚本可能需要很长时间才能完成，并且时间量将取决于数据库中存在的空间数据量。

执行这些步骤后，以下注意事项和限制适用：

  • 使用旧SDO_ORDINATE_ARRAY定义创建的任何现有可传输表空间都不起作用。

  • 如果使用具有旧SDO_ORDINATE_ARRAY定义的数据库上的原始导出实用程序创建导出文件，并且如果需要将该文件导入到使用新SDO_ORDINATE_ARRAY定义的数据库中，则必须使用原始导入实用程序指定TOID_NOVALIDATE标志，如下例所示：

    imp scott / <password> file = states.dmp tables = states TOID_NOVALIDATE = MDSYS.SDO_GEOMETRY，MDSYS.SDO_ORDINATE_ARRAY，MDSYS.SDO_ELEM_INFO_ARRAY

  • 如果计划在源数据库上执行sdoupggeom.sql之后使用Oracle Data Pump导入数据，则还必须在导入操作后在目标（目标）数据库上运行sdoupggeom.sql 。

15 SQL / JSON条件JSON_EXISTS

SQL / JSON条件json_exists允许您将SQL / JSON路径表达式用作行过滤器，以根据JSON文档的内容选择行。您可以在CASE表达式或SELECT语句的WHERE子句中使用条件json_exists 。

条件json_exists检查JSON数据中是否存在特定值：如果值存在则返回true，如果不存在则返回false。更准确地说，如果json_exists目标数据与一个或多个JSON值匹配， json_exists返回true。如果没有匹配的JSON值，则返回false。

您还可以使用json_exists创建用于JSON数据的位图索引 - 请参见例26-1 。

错误处理程序ERROR ON ERROR ， FALSE ON ERROR和TRUE ON ERROR适用。默认值为FALSE ON ERROR 。处理程序在发生任何错误时生效，但是当给定的JSON数据格式不正确时（通过使用松散语法），通常会发生错误。与条件的情况不同is json而is not json ，条件json_exists 期望它检查的数据是格式良好的JSON数据。

json_exists的第二个参数是SQL / JSON路径表达式，后跟可选的PASSING子句和可选的错误子句。

与json_exists使用的SQL / JSON路径表达式的可选过滤器表达式可以引用SQL / JSON变量，其值通过将它们与PASSING子句绑定从SQL传递。此类变量支持以下SQL数据类型： VARCHAR2 ， NUMBER ， BINARY_DOUBLE ， DATE ， TIMESTAMP和TIMESTAMP WITH TIMEZONE 。

15.1使用带有JSON_EXISTS的过滤器

您可以将SQL / JSON条件json_exists与具有一个或多个过滤器表达式的路径表达式一起使用，以选择包含匹配数据的文档。过滤器允许您测试是否存在具有满足各种条件的特定字段的文档。

SQL / JSON条件json_exists对包含与SQL / JSON路径表达式匹配的数据的文档返回true。如果路径表达式包含过滤器，则与应用该过滤器的路径匹配的数据也必须满足过滤器，以便json_exists为包含数据的文档返回true。

过滤器适用于紧接在其之前的路径，并且测试是（a）给定文档是否具有与该路径匹配的一些数据，以及（b）匹配数据是否满足过滤器。如果这两个条件都成立，则json_exists为文档返回true。

紧接在过滤器之前的路径表达式定义了其中使用的模式的范围。过滤器中的符号（ @ ）表示该路径所针对的数据，该路径称为过滤器的当前项 。例如，在路径表达式$.了LineItem？（@。部分。UPCCode == 85391628927) ， @指的是数组LineItems的出现。

示例15-1 JSON_EXISTS：没有过滤器的路径表达式

此示例选择具有订单项描述包含UPC代码条目的订单项的采购订单文档。

SELECT po.po_document FROM j_purchaseorder po WHERE json_exists（po.po_document，'$。了LineItem。部分。UPCCode'）;

示例15-2 JSON_EXISTS：路径表达式过滤器中的当前项和范围

此示例显示了三种等效方法来选择具有行项目的文档，该行项目的部分包含值为85391628927的UPC代码。 

SELECT po.po_document FROM j_purchaseorder po WHERE json_exists（po.po_document，'$？（@。了LineItem。部分。UPCCode == 85391628927）'）; SELECT po.po_document FROM j_purchaseorder po WHERE json_exists（po.po_document，'$。LineItems ？（@。部分。UPCCode == 85391628927）'）; SELECT po.po_document FROM j_purchaseorder po WHERE json_exists（po.po_document，'$。了LineItem。部分 ？（@。UPCCode == 85391628927）'）;

  • 在第一个查询中，过滤器的范围是上下文项，即整个采购订单。 @指的是上下文项。

  • 在第二个查询中，过滤器作用域是一个LineItems数组（以及它的每个元素，隐式）。 @指的是该数组的一个元素。

  • 在第三个查询中，过滤器作用域是LineItems数组中元素的Part字段。 @指的是Part字段。

例15-3 JSON_EXISTS：过滤条件取决于当前项

该实施例中选择具有既与具有UPC代码的一部分的行项目购买订单文件85391628927 和与订单数量大于3的订单项。每个过滤器的范围，即当前项，在本例中是上下文项。每个过滤条件独立应用（到同一文档）;这两个条件并不一定适用于同一订单项。 

SELECT po.po_document FROM j_purchaseorder po WHERE json_exists（po.po_document，'$？（@。了LineItem。部分。UPCCode == 85391628927 && @。了LineItem。数量> 3）'）;

示例15-4 JSON_EXISTS：过滤递减

此示例与示例15-3类似，但它的行为完全不同。它选择具有包含UPC代码且订单数量大于3的零件的订单项的采购订单文档。过滤器中当前项目的范围较低;它不是上下文项，而是LineItems数组元素。也就是说， 同一行项必须满足两个条件，因为json_exists返回true。 

SELECT po.po_document FROM j_purchaseorder po WHERE json_exists（po.po_document，'$。LineItems ？（@。部分。UPCCode == 85391628927 && @。数量> 3）'）;

示例15-5 JSON_EXISTS：路径表达式使用路径表达式存在条件

此示例显示如何缩减过滤器的一部分，同时将另一部分限定在文档（上下文项）级别。它选择具有值为"ABULL"的User字段的采购订单文档和具有包含UPC代码且订单数量大于3的部分的行项目的文档。也就是说，它选择由例15-4选择的相同文档，以及作为用户具有"ABULL"所有文档。path-expression谓词exists的参数是一个路径表达式，它指定特定的行项目;如果找到匹配，则谓词返回true，即，如果存在任何此类行项目。

15.2 JSON_EXISTS为JSON_TABLE

SQL / JSON条件json_exists可以视为SQL / JSON函数json_table 。

例15-6说明了等价：两个SELECT语句具有相同的效果。

除了可能帮助您更好地理解json_exists ，这种等效性实际上很重要，因为它意味着您可以使用任何一种来获得相同的效果。

特别是，如果您json_exists使用json_exists ，或者将它与json_value或json_query （也可以使用json_table表示）结合使用，要访问相同的数据，那么单次调用json_table带来数据优势。只解析一次。

因此，优化器通常会自动将json_exists ， json_value和json_query （任意组合）的多次调用重写为较少的json_table调用。

示例15-6使用JSON_TABLE表示的JSON_EXISTS

SELECT select_list FROM table WHERE json_exists （ column ， json_path error_handler ON ERROR）; SELECT select_list FROM table ， json_table （ column ， '$' error_handler ON ERROR COLUMNS （“ COLUMN_ALIAS ” NUMBER EXISTS PATH json_path ）） AS “ JT ” WHERE jt.column_alias = 1 ;

100.1 DBMS_MEMOPTIMIZE概述

DBMS_MEMOPTIMIZE包提供用于管理在大池和memoptimize池中缓冲的Memoptimized Rowstore数据的接口。

Memoptimized Rowstore提供以下功能：

  • 快速摄取

    快速摄取优化了高频单行数据插入的处理。快速摄取使用SGA中的大型池来缓冲插入，然后再将它们写入磁盘。

  • 快速查找

    快速查找可以快速检索高频查询的数据。快速查找在SGA中使用称为memoptimize池的单独内存区域来缓冲从表中查询的数据。

DBMS_MEMOPTIMIZE包提供与Memoptimized Rowstore相关的以下操作：

  • 快速摄取操作：

    • 在所有会话中提供已从大型池成功写入磁盘的行序列号的低高水位标记（低HWM）。

    • 提供已写入当前会话的大池的行的高水位线（HWM）序列号。

    • 将当前会话中从大型池中的所有快速接收数据刷新到磁盘。

  • 快速查找操作：

    • 从memoptimize池中删除表的数据。

    • 在memoptimize池中填充表的数据。

术语“熄灯”源于计算中心位于一个房间并包含许多服务器，这些服务器保持锁定和密钥以及黑暗中。在正常操作下，人工管理员没有输入房间，房间内的所有操作都是自动化的。

逻辑备用数据库

逻辑备用数据库采用标准的Oracle归档重做日志文件，将它们转换回SQL事务，然后将它们应用于开放备用数据库。尽管可以与最终用户访问同时应用更改，但通过重新生成的SQL事务维护的表允许对逻辑备用数据库的用户进行只读访问。由于数据库已打开以允许应用重构的SQL事务，因此它与主数据库在物理上不同。数据库表可以与其主数据库对等体具有不同的索引和物理特性，但这些表必须从应用程序访问角度保持逻辑一致性，以履行其作为备用数据源的角色。

手动故障转移

首先确定故障转移的DBA启动故障转移，然后通过单击企业管理器中的FAILOVER或发出DGMGRL FAILOVER命令来调用故障转移。“ 手动 ”一词用于将此类故障转移与快速启动故障转移进行对比。

另请参见 快速启动故障转移 。

观察者

DGMGRL客户端，持续监视主数据库和目标备用数据库，评估是否需要进行故障转移，并在条件允许时启动快速启动故障转移。

物理备用数据库

物理备用数据库是主数据库的精确副本。当主数据库处于打开和活动状态时，物理备用数据库使用重做应用来应用从主数据库接收的重做数据。重做应用可以在已装入的物理备用数据库实例上运行。如果已购买Oracle Active Data Guard选件的许可证，则Redo Apply也可以在打开的物理备用数据库实例上运行。有关更多详细信息，请参阅Oracle Data Guard概念和管理 。

主数据库

生成数据库，从中创建和维护一个或多个备用数据库。每个备用数据库都与一个且仅一个主数据库相关联。但是，单个主数据库可以支持多个备用数据库。Oracle Data Guard代理监视器（DMON）使用主数据库维护二进制配置文件的主副本，确保每个备用数据库的文件副本在进行更改时保持最新。

代理使用DB_UNIQUE_NAME初始化参数中的值来引用此数据库，该参数被定义为全局唯一。

只读模式

可以打开允许查询但不允许修改的数据库的模式。

物理备用数据库可以只读方式打开，以便可以执行查询。如果已购买Oracle Active Data Guard选件的许可证，则可以在Redo Apply处于活动状态时打开物理备用数据库。此功能称为实时查询。有关更多详细信息，请参阅Oracle Data Guard概念和管理 。

重做申请

物理备用数据库通过Redo Apply与主数据库保持同步，该数据库恢复从主数据库接收的重做数据并将重做应用于物理备用数据库。

复职

恢复是将故障转移操作后已禁用的数据库（包括旧主数据库）转换为新主数据库的可用备用数据库的过程。必须在数据库上启用闪回数据库才能恢复它。

快照备用数据库

快照备用数据库是完全可更新的备用数据库，它是从物理备用数据库创建的。在快照备用数据库上，接收重做数据，但在将快照备用数据库转换回物理备用数据库之前不会应用重做数据。

SQL Apply

逻辑备用数据库通过SQL Apply与主数据库保持同步，SQL Apply将从主数据库接收的重做数据转换为SQL语句，然后在备用数据库上执行SQL语句。

备用数据库

使用主数据库备份创建的主数据库的副本。通过随时间从主数据库到每个备用数据库应用归档重做数据，备用数据库与主数据库保持同步。备用数据库可以接管主数据库的处理，提供几乎连续的数据库可用性。备用数据库具有自己的服务器参数文件，控制文件和数据文件。它还具有代理配置文件的副本，在主数据库实例中运行的DMON进程的指导下保持最新。

代理通过其全局唯一名称引用备用数据库，该名称存储在其DB_UNIQUE_NAME初始化参数中。

另请参见 逻辑备用数据库 ， 物理备用数据库和快照备用数据库 。

47.1 DBMS_DB_VERSION概述

DBMS_DB_VERSION包指定Oracle版本号和其他对基于Oracle版本的简单条件编译选择有用的信息。

Oracle Database 12c第2版的软件包如下所示。

包DBMS_DB_VERSION是VERSION CONSTANT PLS_INTEGER：= 12; -  RDBMS版本号RELEASE CONSTANT PLS_INTEGER：= 2; -  RDBMS版本号ver_le_9_1 CONSTANT BOOLEAN：= FALSE; ver_le_9_2 CONSTANT BOOLEAN：= FALSE; ver_le_9 CONSTANT BOOLEAN：= FALSE; ver_le_10_1 CONSTANT BOOLEAN：= FALSE; ver_le_10_2 CONSTANT BOOLEAN：= FALSE; ver_le_10 CONSTANT BOOLEAN：= FALSE; ver_le_11_1 CONSTANT BOOLEAN：= FALSE; ver_le_11_2 CONSTANT BOOLEAN：= FALSE; ver_le_11 CONSTANT BOOLEAN：= FALSE; ver_le_12_1 CONSTANT BOOLEAN：= FALSE; ver_le_12_2 CONSTANT BOOLEAN：= TRUE; ver_le_12 CONSTANT BOOLEAN：= TRUE; END DBMS_DB_VERSION;

布尔常量遵循命名约定。每个常量都给出一个布尔表达式的名称。例如：

  • VER_LE_9_1表示版本<= 9且版本<= 1

  • VER_LE_10_2表示版本<= 10并且发布<= 2

  • VER_LE_10表示版本<= 10

这些布尔常量的典型用法是：

$ IF DBMS_DB_VERSION.VER_LE_10 $ THEN版本10及更早版本代码$ ELSIF DBMS_DB_VERSION.VER_LE_11 $ THEN版本11代码$ ELSE版本12及更高版本代码$ END

此代码结构将保护对版本12的代码的任何引用。它还会阻止控制包常量DBMS_DB_VERSION 。在版本10下编译程序时引用VER_LE_11 。类似的观察也适用于版本11。即使静态常量VER_LE_11未在版本10数据库中定义，此方案仍然有效，因为如果DBMS_DB_VERSION条件编译会保护$ELSIF免于评估。 VER_LE_10为TRUE 。

12.17定义

句法

DEF [INE] [ 变量 ] | [ variable = text ]

指定一个用户或预定义变量并为其分配CHAR值，或列出单个变量或所有变量的值和变量类型。

条款

变量

表示要分配或列出其值的用户或预定义变量。

文本

表示要分配给变量的CHAR值。如果文本包含标点符号或空格，请将文本用单引号括起来。 

变量 = 文本

定义（命名）替换变量并为其赋值 CHAR值。

输入DEFINE后面变量列表中选择值和变量的类型。 输入DEFINE，不带子句列出所有替换变量的值和类型。

用法

定义的变量保留其值，直到您：

  • 输入引用变量的新DEFINE命令

  • 进入UNDEFINE 命令引用变量

  • 输入ACCEPT 命令引用变量

  • 引用COLUMN的NEW_VALUE或OLD_VALUE子句中的变量命令然后在SELECT中引用该列命令

  • 退出SQL * Plus

无论何时运行存储的查询或脚本，SQL * Plus都会替换每个变量的值替代变量引用变量 （形式和变量或&& 变量 ）。直到取消定义变量 SQL * Plus将不提示您输入变量的值，在这个环节。

如果已定义变量的值扩展到多行（使用SQL * Plus命令延续字符），则SQL * Plus将使用空格替换每个延续字符和回车符。例如，SQL * Plus解释

DEFINE TEXT ='ONE-TWO-THREE'

如

DEFINE TEXT ='一两三'

您应该避免使用可能与传递给它们的值相同的名称定义变量，因为可能会出现意外结果。如果为定义的变量提供的值与变量名匹配，则使用匹配变量的内容而不是提供的值。

SQL * Plus启动时预定义了一些变量。输入DEFINE以查看其定义。

例子

要将值MANAGER分配给变量POS，请键入：

DEFINE POS =经理

如果执行包含对＆POS的引用的命令，则SQL * Plus会将值MANAGER替换为＆POS，并且不会提示您输入POS值。

要将CHAR值20分配给变量DEPARTMENT_ID，请键入：

DEFINE DEPARTMENT_ID = 20

即使输入数字20，SQL * Plus也会为DEPARTMENT_ID分配一个CHAR值，该值包含两个字符，即2和0。

要列出DEPARTMENT_ID的定义，请输入

DEFINE DEPARTMENT_ID
DEFINE DEPARTMENT_ID =“20”（CHAR）

此结果显示DEPARTMENT_ID的值为20。

10.2.1配置内存初始化参数

您可以修改以下数据库初始化参数以调整内存使用情况，以更准确地反映您的应用程序需求：

  • SHARED_POOL_SIZE

    共享池内存由JVM中的类加载器使用。平均而言，类加载器为每个加载的类使用大约8 KB的内存。在将类加载和解析到数据库中时使用共享池内存。在数据库中编译源代码或在数据库中使用Java资源对象时也会使用它。

    使用loadjava工具时，将loadjava消耗SHARED_POOL_SIZE指定的内存。数据库初始化过程需要将SHARED_POOL_SIZE设置为96 MB，因为它会加载大约8,000个类的Java二进制文件并对其进行解析。在创建调用规范时以及系统在运行时跟踪动态加载的Java类时，也会使用SHARED_POOL_SIZE资源。

  • JAVA_POOL_SIZE

    Oracle JVM内存管理器主要使用JAVA_POOL_SIZE进行Java方法和类定义的内存表示，以及在共享服务器模式下以调用结束时迁移到会话空间的静态Java状态。在第一种情况下，您将与所有Java用户共享内存成本。在第二种情况下， JAVA_POOL_SIZE的值根据每个会话的静态变量中保存的实际状态量而变化。但是，Oracle建议最小值为50 MB。

  • JAVA_SOFT_SESSIONSPACE_LIMIT

    此参数允许您在会话中指定Java内存使用的软限制，如果必须增加Java内存限制，则会警告您。每次分配内存时，都会根据此限制检查分配的总内存。

    当用户的会话Java状态超过此大小时，Oracle JVM会生成一个写入跟踪文件的警告。虽然此警告是一条信息性消息，对您的应用程序没有影响，但您应该了解并管理已部署类的内存要求，尤其是与会话空间的使用相关的内存要求。

    注意：

    此参数仅适用于共享服务器环境。

  • JAVA_MAX_SESSIONSPACE_SIZE

    如果在服务器中运行的可由用户调用的Java程序可以以其内存使用不自限的方式使用，则此设置可能对会话量的硬限制有用空间可供它使用。默认值为4 GB。这个限制特意设置得非常高，通常是不可见的。

    当用户的会话Java状态尝试超过此大小时，应用程序可能会收到内存不足故障。

    注意：

    此参数仅适用于共享服务器环境。

10.2.2关于Java池内存

Java池内存是SGA的一个子集，它由Java专门用于必须以页面方式对齐的内存。这包括大多数，但不是所有用于Java类共享定义的内存。Java池内存的其他用途取决于Oracle数据库服务器运行的模式。

专用服务器中使用的Java池内存

以下是在专用服务器中使用的Java池内存的构成：

  • 每个Java类的大多数共享部分都在使用中。

    这包括只读存储器，例如代码向量和方法。总的来说，每个类平均大约4 KB到8 KB。

  • 每个会话的每会话Java状态都不是。

    对于专用服务器，它存储在程序全局区域（PGA）内的用户全局区域（UGA）中，而不是SGA内。

在专用服务器下，所需的Java池内存总量取决于运行的应用程序，通常介于10到50 MB之间。

共享服务器中使用的Java池内存

以下内容构成共享服务器中使用的Java池内存：

  • 每个Java类的大多数共享部分都在使用中

    这包括只读存储器，例如矢量和方法。总的来说，每个类的内存通常平均为4 KB到8 KB左右。

  • 一些UGA用于每个会话内存

    特别是，始终在Java池中分配跨数据库调用保持使用的对象的内存。

    由于Java池内存大小有限，因此必须估计应用程序的总需求并乘以应用程序要创建的并发会话数，以计算必需的Java池内存总量。每个UGA都会根据需要增长和缩小。但是，所有组合在一起的UGA必须能够适应整个固定Java池空间。

在共享服务器下，Java池可能很大。Java密集型多用户应用程序可能需要超过100 MB。

注意：

如果要在服务器上编译代码，而不是在客户端上编译并加载到服务器，那么您可能需要比默认20 MB更大的JAVA_POOL_SIZE 。

减少启用Java的会话数

数据库中Java的顶级调用由客户端应用程序或实用程序发出。如果每个客户端都有专用服务器，那么大规模部署会导致数据库服务器上的资源大量消耗，并且还会导致资源浪费。您可以使用客户端连接池或数据库驻留连接池（DRCP）来减少数据库进程和会话的数量。

10.2.4纠正内存不足错误

如果在加载类时内存不足，则它可能会无提示失败，从而在数据库中留下无效的类。稍后，如果您尝试调用或解决任何无效的类，则会在运行时抛出ClassNotFoundException或NoClassDefFoundException实例。如果要加载损坏的类文件，您将获得相同的异常。您应该执行以下操作：

  • 验证该类实际上是否包含在要加载到服务器的集合中。

  • 使用loadjava -force选项强制加载新类以替换已驻留在服务器中的类。

  • 使用loadjava -resolve选项在加载过程中尝试解析类。这使您可以在加载时而不是在运行时捕获缺少的类。

  • 通过连接到包含该类的模式中的数据库，仔细检查新加载的类的状态，然后运行以下命令：

    SELECT * FROM user_objects WHERE object_name = dbms_java.shortname（''）;

    STATUS字段应为VALID 。如果loadjava工具抱怨内存问题或失败，例如连接丢失，则增加SHARED_POOL_SIZE和JAVA_POOL_SIZE ，然后重试。

4.1关于回归

回归是一种数据挖掘功能，可以预测连续体中的数值。利用回归技术可以预测利润，销售，抵押贷款利率，房屋价值，面积，温度或距离。例如，回归模型可用于根据位置，房间数量，批量和其他因素预测房屋的价值。

回归任务以一个已知目标值的数据集开始。例如，可以基于一段时间内许多房屋的观测数据来开发预测房屋价值的回归模型。除了价值，数据还可以跟踪房屋的年龄，面积，房间数量，税收，学区，购物中心的距离等。房屋价值可以是目标，其他属性是预测变量，每个房屋的数据构成一个案例。

在模型构建（训练）过程中，回归算法估计目标的值作为构建数据中每个案例的预测变量的函数。预测变量和目标之间的这些关系在模型中进行了总结，然后可以将其应用于目标值未知的不同数据集。

通过计算测量预测值与预期值之间差异的各种统计数据来测试回归模型。回归项目的历史数据通常分为两个数据集：一个用于构建模型，另一个用于测试模型。

回归建模在趋势分析，业务规划，营销，财务预测，时间序列预测，生物医学和药物响应建模以及环境建模方面具有许多应用。

4.1.1回归如何工作？

您不需要了解回归分析中使用的数学来开发和使用质量回归模型进行数据挖掘。但是，理解一些基本概念是有帮助的。

回归分析旨在确定函数的参数值，使函数最适合您提供的一组数据观察。以下等式用符号表示这些关系。它表明回归是估计连续目标（ y ）的值作为一个或多个预测变量（ x 1 ， x 2 ，...， x n ）的函数（ F ）的过程，一组参数（θ） 1，θ2，...，θn），以及差错（e的度量）。 

y = F （ x ，θ）+ e

预测变量可以理解为自变量，目标可以理解为因变量。该误差（也称为残差 ）是因变量的预期值和预测值之间的差值。回归参数也称为回归系数 。

训练回归模型的过程涉及找到最小化误差测量的参数值，例如，误差平方和。

有不同的回归函数族和不同的误差测量方法。

4.1.1.6置信区间

回归模型预测评分数据中每个案例的数字目标值。除了预测，一些回归算法可以识别置信界限，即预测值可能存在的区间的上下边界。

3.1关于回归

回归是一种数据挖掘功能，可以预测连续体中的数值。利用回归技术可以预测利润，销售，抵押贷款利率，房屋价值，面积，温度或距离。例如，回归模型可用于根据位置，房间数量，批量和其他因素预测房屋的价值。

回归任务以一个已知目标值的数据集开始。例如，可以基于一段时间内许多房屋的观测数据来开发预测房屋价值的回归模型。除了价值，数据还可以跟踪房屋的年龄，面积，房间数量，税收，学区，购物中心的距离等。房屋价值可以是目标，其他属性是预测变量，每个房屋的数据构成一个案例。

在模型构建（训练）过程中，回归算法估计目标的值作为构建数据中每个案例的预测变量的函数。预测变量和目标之间的这些关系在模型中进行了总结，然后可以将其应用于目标值未知的不同数据集。

通过计算测量预测值与预期值之间差异的各种统计数据来测试回归模型。回归项目的历史数据通常分为两个数据集：一个用于构建模型，另一个用于测试模型。

回归建模在趋势分析，业务规划，营销，财务预测，时间序列预测，生物医学和药物响应建模以及环境建模方面具有许多应用。

自动

自动生成继续恢复操作所需的下一个归档重做日志文件的名称。Oracle数据库使用LOG_ARCHIVE_DEST（或LOG_ARCHIVE_DEST_ 1）和LOG_ARCHIVE_FORMAT参数（或其默认值）生成目标重做日志文件名。如果找到该文件，则应用该文件中包含的重做。如果找不到该文件，SQL * Plus会提示您输入文件名，并将生成的文件名显示为建议。

如果未指定AUTOMATIC或LOGFILE，则SQL * Plus会提示您输入文件名，并提示生成的文件名。您可以接受生成的文件名，也可以使用完全限定的文件名替换它。如果您知道归档文件名与Oracle数据库将生成的文件名不同，则可以使用LOGFILE子句指定文件名来节省时间。

指定从中读取归档重做日志文件组的位置。location的值必须是完全指定的文件位置。如果省略此参数，则SQL * Plus假定归档重做日志文件组位于初始化参数LOG_ARCHIVE_DEST或LOG_ARCHIVE_DEST_1指定的位置。如果已使用SET LOGSOURCE设置文件，请不要指定FROM。

partial_database_recovery

要在表空间上执行介质恢复，必须装入或打开数据库，并且表空间必须处于脱机状态。

要对数据文件执行介质恢复，数据库可以保持打开状态并在损坏的数据文件脱机的情况下装入（除非该文件是SYSTEM表空间的一部分）。

在使用RECOVER命令之前，您必须从先前的备份中恢复已损坏的数据文件的副本。确保您可以访问所有存档和联机重做日志文件，这些文件可以追溯到备份时。

关于Oracle Label Security审核

Oracle Label Security审计通过跟踪其自身的管理操作和策略权限来补充标准Oracle数据库审计。

您可以使用SA_AUDIT_ADMIN程序包或Oracle Enterprise Manager来设置和更改Oracle Label Security策略的审核选项。

创建新策略时，该策略的标签列将添加到数据库审计跟踪中。无论是启用还是禁用审计，都会创建标签列，而与是否使用数据库审计或操作系统审计无关。每当将记录写入审计表时，每个策略都为该记录提供标签以指示会话标签。管理员可以创建审核视图以显示这些标签。请注意，在审计表中，标签不控制对行的访问，而只记录行的敏感性。

您指定的审核选项仅适用于后续会话，而不适用于当前会话。即使禁用了审核，也可以指定审核选项。仅制作这些规范不会产生任何开销。启用Oracle Label Security审核后，这些选项将生效，并且会创建超出标准Oracle数据库审核所创建的开销。

请注意，Oracle Label Security不会为写入操作系统审计试用版的审计数据提供标签。即使启用了操作系统审核，所有Oracle Label Security审核记录也会直接写入数据库审核跟踪。如果禁用审核，则不会生成任何Oracle Label Security审核记录。

如果您使用的是传统审核（非统一审核），则传统审核跟踪仅列出操作编号。要查找相应的审计操作名称，可以查询LABACSYS.OLS$AUDIT_ACTIONS系统表。您必须具有AUDIT_VIEWER ， AUDIT_ADMIN或policy _DBA角色才能查询此表。

5 DRDA的Oracle数据库提供程序的管理和自定义

考虑各种管理和定制问题。

使用Oracle数据库提供程序为DRDA迁移步骤

虽然将现有DB2应用程序迁移到Oracle数据库是特定于数据和目标的，但一般方法有以下六个步骤：

  1. 为DRDA软件安装和配置Oracle数据库提供程序

  2. 在Oracle数据库中为DRDA对象安装Oracle数据库提供程序

  3. 管理DRDA包权限

  4. 迁移DB2数据

  5. 重新定位应用程序

  6. 调整SQL转换和数据类型

使用Oracle数据库提供程序进行DRDA的注意事项

在为DRDA软件安装Oracle数据库提供程序之前，组织必须考虑几个操作和资源问题。在确定最佳安装是作为独立的Oracle主目录，现有Oracle数据库中的Oracle主目录，还是完全独立于Oracle数据库的计算机上时，计算机和网络资源的灵活性和性能至关重要。此外，必须使用安装的所有可能DB2客户端的性质是一个决定性因素;在此上下文中，DB2被视为客户端。

为DRDA安装Oracle数据库提供程序的先决条件

在Oracle数据库中为DRDA对象安装Oracle数据库提供程序之前，必须将一个或多个用户指定为DRDA管理员，并具有管理员角色。

同样，指定将通过Oracle数据库提供程序访问Oracle数据库以获取DRDA或DB2应用程序的用户，并向他们授予DRDA用户的角色和权限。设置DRDA用户权限和配置的某些方面可能需要延迟，直到迁移过程进一步。这主要涉及应用程序使用的特定DRDA包，以及任何特定的SQL转换或数据类型调整。如果在迁移之前识别应用程序的包，则这些包可以作为包授权工作流的一部分应用。

迁移DB2数据

在DB2中，可以在任意模式下创建对象，而在Oracle数据库中模式名称不是任意的。因此，在将数据从DB2迁移到Oracle时，必须仔细考虑模式。在Oracle中，必须在实际用户的模式中分配所有模式对象，例如表，视图，同义词等。这显然会影响这些对象的命名，创建和访问方式。

请考虑以下示例： USER1创建表"USER1"."TABLE1"和"USER2"."TABLE2" 。在DB2中， TABLE1和TABLE2归USER1 ，因为USER1是它们的创建者。在Oracle中，表"USER2"."TABLE2"由用户USER2拥有。此外， USER1不能创造TABLE2除非USER1具有CREATE ANY TABLE privelege。相反， USER2必须创建TABLE2 ，然后授予USER1访问权限。

从DB2迁移到Oracle的数据也必须根据Oracle数据类型进行定义。虽然Oracle使用ANSI定义的数据类型名称，但它们不一定具有与DB2实现相同的范围限制或语义。要准确建模现有DB2应用程序数据类型，请查看适用于DRDA的Oracle数据库提供程序数据字典部分。

在使用适当的数据类型创建模式和对象后，可以将数据导入Oracle。

应用程序有两大类： 本机应用程序和远程应用程序 。

重新定位原生应用程序

典型的DB2应用程序称为本机，因为它们通过内部IPC机制直接与本地DB2系统交互。这些应用程序使用嵌入式SQL编程，并使用DB2 SQL PreProcessor。预处理源生成执行计划 ，该计划存储在数据库资源模块（DBRM）中。用户必须在程序运行之前上载或将执行计划绑定到本地DB2实例。

以下步骤应由IBM DB管理员执行。

    DB2有一个内部通信系统，用于连接远程DB2实例。要处理远程实例，请将记录插入SYSIBM.IPNAMES表， SYSIBM.LOCATIONS表，也可以插入SYSIBM.USERNAMES表。

    有关DB2 Communications Database工具的说明，请参阅IBM DB2文档。

重新定位远程应用程序

远程应用程序通常不直接绑定到本地DB2。此类应用程序通常被称为网络感知或面向网络，并具有远程服务器位置配置属性，用于指定连接的内容和位置。

此类应用程序通过网络协议将Oracle数据库提供程序用于DRDA。重新定位此类应用程序很容易配置，如以下步骤所示。

  1. 更改应用程序的配置选项以使用Oracle数据库提供程序中为DRDA配置的主机名（或IP地址），端口号和RDB名称。一个例子是通过ODBC，其中DSN条目包含网络参数。

    在此示例中， Network和PortNumber参数对应于插入先前使用的DB2 Communication Database示例中的Linkname和Location条目。Database参数对应于Location名称。所有这些DATA_PORT与RDB_MAP配置的DRDA数据库提供程序的DATA_PORT和RDB_MAP参数相对应。

    以下是odbc.ini文件的示例。 

    [DRDAAS] Network = remotehost.remotedomain.com PortNumber = 1446数据库= DRDAAS
  2. 执行应用程序的包资源绑定操作。

    通常，这由应用程序本身隐式处理，或者记录为将应用程序访问和资源设置为远程DB2实例的一次性步骤。有关绑定说明的具体应用，请参阅文档。

翻译SQL语句和键入数据类型

某些应用程序可能具有超出SQL转换的自动转换机制的特定于DB2的SQL，或者可能期望查询中的特定列具有非常特定的数据类型。在这种情况下，可能需要手动插入SQL替换语句，或添加特定于项的数据类型操作。

例如，假设应用程序具有特定的SQL语句，该语句具有以下特定于DB2的语法： SELECT LOG2(COL1) FROM TABLE1 。要在Oracle中正常工作，需要将SQL转换为此语句： SELECT LOG10(COL1,2) FROM TABLE1 。

通过SQL Translation的Register工具，可以为此SQL语句注册直接转换，如注册SQL替换语句一节中所示。请注意，这必须由执行SQL语句的用户完成;请记住，SQL Translation Profile必须作为该用户的资源创建。

注册SQL转换器后，当应用程序发出原始SQL时，它会隐式转换为新的SQL和进程。

在某些非常特殊的情况下，应用程序客户端要求以非常特定的格式返回查询中的选定项的数据类型。

假设翻译查询SELECT LOG10(COL1,2) FROM TABLE1返回DECFLOAT34数据类型，但应用程序无法处理它，可以隐式地将数据类型强制转换为另一种兼容类型。

如果应用程序支持DOUBLE PRECISION数据类型，则可以使用TYPEMAP工具添加注册按需数据类型转换中描述的此特定强制。

7.1关于使用命令行SQL * Plus创建报告

除了纯文本输出之外，SQL * Plus命令行界面还允许您生成完整的网页，可以嵌入网页的HTML输出或CSV格式的数据。您可以使用SQLPLUS -MARKUP“HTML ON”或SET MARKUP HTML ON SPOOL ON来生成使用<HTML>和<BODY>标记自动封装的完整HTML页面。您可以使用SQLPLUS -MARKUP“CSV ON”或SET MARKUP CSV ON来生成CSV格式的报告。

默认情况下，使用MARKUP HTML ON检索的数据以HTML格式输出，但您可以选择将输出定向到HTML <PRE> 标记，以便它在Web浏览器中显示与SQL * Plus中显示的完全相同。有关这些命令的更多信息，请参见SQLPLUS MARKUP选项和SET MARKUP命令。

在程序脚本中嵌入SQL * Plus时，SQLPLUS -MARKUP“HTML ON”非常有用。启动时，它会在执行任何命令之前输出HTML和BODY标记。所有后续输出都是HTML格式，直到SQL * Plus终止。

-SILENT和-RESTRICT命令行选项可与-MARKUP有效地一起使用，以抑制SQL * Plus提示和标题信息的显示，并限制某些命令的使用。

SET MARKUP HTML ON SPOOL ON为每个后续假脱机文件生成一个HTML页面。执行SPOOL OFF或SQL * Plus退出时，将关闭假脱机文件中的HTML标记。

您可以使用SET MARKUP HTML ON SPOOL OFF来生成适合嵌入现有网页的HTML输出。以这种方式生成的HTML输出没有<HTML>或<BODY>标记。

通过在SQL * Plus命令行中使用-M [ARKUP] CSV ON选项，可以在登录用户会话时启用CSV标记。有关更多信息，请参阅SQL * Plus程序语法 。登录到用户会话后，可以使用SET MARKUP CSV ON命令启用CSV标记。

您可以使用DELIMITER选项指定分隔符。您还可以使用QUOTE OFF输出不带引号的文本。

您可以使用SET FEEDBACK命令的ONLY选项来禁止显示查询返回的数据。显示查询选择和返回的行数。

7.1.1创建HTML报告

在SQL * Plus会话期间，以交互方式使用SET MARKUP命令将HTML写入假脱机文件。您可以在a中查看输出网页浏览器。

SET MARKUP HTML ON SPOOL ON仅指定SQL * Plus输出将是HTML编码的，它不会创建或开始写入输出文件。必须使用SQL * Plus SPOOL命令开始生成假脱机文件。然后，此文件包含HTML标记，包括<HTML>和</ HTML>。

创建HTML文件时，指定.html或.htm文件扩展名是非常重要和方便的，它们是HTML文件的标准文件扩展名。这使您可以轻松识别输出文件的类型，还可以使Web浏览器识别并正确显示HTML文件。如果未指定扩展名，则使用默认的SQL * Plus文件扩展名。

你用 SPOOL OFF或退出以将最终HTML标记附加到假脱机文件，然后将其关闭。如果输入另一个SPOOL文件名命令，则关闭当前假脱机文件，如SPOOL OFF或EXIT，并创建具有指定名称的新HTML假脱机文件。

您可以使用SET MARKUP命令根据需要启用或禁用HTML输出。
