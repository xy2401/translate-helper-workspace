27.1关于支持向量机

支持向量机（SVM）是一种功能强大，最先进的算法，具有基于Vapnik-Chervonenkis理论的强大理论基础。SVM具有强大的正则化特性。正则化是指将模型推广到新数据。

27.1.1 SVM的优点

Oracle Data Mining SVM实现包括两种类型的解算器，一种内点法（IPM）求解器和a 子梯度下降（SGD）求解器。IPM求解器提供非常稳定和准确的解决方案，但是，它可能无法处理非常高维度的数据。对于高维数据，例如文本，评级等，SGD求解器是更好的选择。两个求解器都具有高度可扩展的并行实现，可以处理大量数据。

27.1.2 Oracle数据挖掘中SVM的优点

Oracle Data Mining有自己的支持向量机（SVM）专有实现，它利用了算法的许多优点，同时补偿了SVM框架中固有的一些限制。Oracle Data Mining SVM提供了生产质量数据挖掘系统所需的可扩展性和可用性。

27.1.2.1可用性

解释Oracle Data Mining中支持向量机（SVM）的可用性。

可用性是一项重大改进，因为SVM经常被视为专家的工具。该算法通常需要数据准备，调整和优化。Oracle Data Mining最大限度地减少了这些要求。您无需成为专家即可在Oracle Data Mining中构建高质量的SVM模型。例如：

  • 大多数情况下不需要数据准备。

  • 默认调整参数通常是足够的。

27.1.2.2可扩展性

了解如何扩展支持向量机（SVM）的数据。

处理非常大的数据集时，通常需要采样。但是，Oracle Data Mining SVM不需要采样，因为算法本身使用分层采样来根据需要减小训练数据的大小。

Oracle Data Mining SVM经过高度优化。它通过优化小型工作集向全局解决方案逐步建立模型。训练模型直到在当前工作集上收敛，然后模型适应新数据。该过程迭代地继续，直到满足收敛条件。高斯内核使用缓存技术来管理工作集。

27.1.3基于内核的学习

了解基于内核的函数来转换支持向量机（SVM）的输入数据。

SVM是一种基于内核的算法。内核是一种将输入数据转换为解决问题的高维空间的函数。核函数可以是线性的或非线性的。

Oracle Data Mining支持线性和高斯（非线性）内核。

在Oracle数据挖掘中， 线性核函数简化为训练数据中原始属性的线性方程。当训练数据中有许多属性时，线性内核很有效。

高斯内核将训练数据中的每个情况转换为n维空间中的点，其中n是个案数。该算法试图将点分成具有同质目标值的子集。高斯核使用非线性分离器，但在核空间内构造线性方程。

27.3 SVM的数据准备

支持向量机（SVM）算法本身在数字属性上运行。SVM对数字属性使用z-score标准化。仅对二维数字列（非嵌套）进行标准化。该算法自动将分类数据“爆炸”成一组二进制属性，每个类别值一个。例如，具有married或single价值的婚姻状况的字符列将转换为两个数字属性： married和single 。新属性的值可以是1 （true）或0 （false）。

当具有简单数据类型（非嵌套）的列中缺少值时，SVM会将它们随机解释为缺失。该算法自动用模式替换缺失的分类值，并用均值替换缺失的数值。

当嵌套列中存在缺失值时，SVM会将它们解释为稀疏。该算法自动用零和矢量稀疏分类数据替换稀疏数值数据。

27.3.1标准化

支持向量机需要数字输入的标准化。规范化将数字属性的值放在相同的比例上，并防止具有较大原始比例的属性偏向解决方案。归一化还可以最大限度地降低溢出和下溢的可能性。

27.3.2 SVM和自动数据准备

了解手动或通过支持向量机（SVM）的自动数据准备（ADP）处理和转换数据。

SVM算法自动处理缺失值处理和分类数据的转换，但规范化和异常值检测必须由自动数据准备（ADP）处理或手动准备。ADP对SVM执行最小 - 最大标准化。

注意：

Oracle建议您使用SVM自动数据准备。 ADP执行的转换适用于大多数模型。

27.4 SVM分类

支持向量机（SVM） 分类基于定义决策边界的决策平面的概念。决策平面是在具有不同类别成员资格的一组对象之间进行分隔的平面。SVM找到定义分隔符的向量（“支持向量”），给出最广泛的类分离。

SVM分类支持二进制，多类和多目标分类 。多目标将多个类别标签与一行相关联。目标类型是ORA_MINING_VARCHAR2_NT类型的集合。

了解何时在支持向量机（SVM）中为数据实现类权重。

27.5一类SVM

神谕数据挖掘使用支持向量机（SVM）作为异常检测的一类分类器。当SVM用于异常检测时，它具有分类挖掘功能但没有目标。

应用单类SVM模型时，会为评分数据中的每个案例生成预测和概率。如果预测为1，则认为该情况是典型的。如果预测为0，则该情况被认为是异常的。此行为反映了使用正常数据训练模型的事实。

您可以使用SVMS_OUTLIER_RATE构建设置指定预期异常的数据百分比。如果您知道“可疑”案例的数量是您的人口的一定百分比，那么您可以将离群率设置为该百分比。该模型在应用于一般人群时大致识别出许多“罕见”案例。

27.6 SVM回归

了解如何使用epsilon-insensitivity loss函数来解决支持向量机（SVM）中的回归问题。

SVM使用epsilon不敏感的损失函数来解决回归问题。

SVM回归试图找到连续函数，使得最大数据点数位于epsilon范围的不敏感管内。落在真实目标值的epsilon距离内的预测不会被解释为错误。

epsilon因子是SVM回归的正则化设置。它平衡了误差幅度和模型稳健性，以实现对新数据的最佳推广。

26.1关于支持向量机

支持向量机（SVM）是一种功能强大，最先进的算法，具有基于Vapnik-Chervonenkis理论的强大理论基础。SVM具有强大的正则化特性。正则化是指将模型推广到新数据。

26.1.1 SVM的优点

Oracle Data Mining SVM实现包括两种类型的解算器，一种内点法（IPM）求解器和a 子梯度下降（SGD）求解器。IPM求解器提供非常稳定和准确的解决方案，但是，它可能无法处理非常高维度的数据。对于高维数据，例如文本，评级等，SGD求解器是更好的选择。两个求解器都具有高度可扩展的并行实现，可以处理大量数据。

26.1.2 Oracle数据挖掘中SVM的优点

Oracle Data Mining有自己的支持向量机（SVM）专有实现，它利用了算法的许多优点，同时补偿了SVM框架中固有的一些限制。Oracle Data Mining SVM提供了生产质量数据挖掘系统所需的可扩展性和可用性。

26.1.2.1可用性

26.1.3基于内核的学习

了解基于内核的函数来转换支持向量机（SVM）的输入数据。

SVM是一种基于内核的算法。内核是一种将输入数据转换为解决问题的高维空间的函数。核函数可以是线性的或非线性的。

Oracle Data Mining支持线性和高斯（非线性）内核。

26.3 SVM的数据准备

支持向量机（SVM）算法本身在数字属性上运行。SVM对数字属性使用z-score标准化。仅对二维数字列（非嵌套）进行标准化。该算法自动将分类数据“爆炸”成一组二进制属性，每个类别值一个。例如，具有married或single价值的婚姻状况的字符列将转换为两个数字属性： married和single 。新属性的值可以是1 （true）或0 （false）。

当具有简单数据类型（非嵌套）的列中缺少值时，SVM会将它们随机解释为缺失。该算法自动用模式替换缺失的分类值，并用均值替换缺失的数值。

当嵌套列中存在缺失值时，SVM会将它们解释为稀疏。该算法自动用零和矢量稀疏分类数据替换稀疏数值数据。

26.3.1标准化

支持向量机需要数字输入的标准化。规范化将数字属性的值放在相同的比例上，并防止具有较大原始比例的属性偏向解决方案。归一化还可以最大限度地降低溢出和下溢的可能性。

SVM分类支持二进制，多类和多目标分类 。多目标将多个类别标签与一行相关联。目标类型是ORA_MINING_VARCHAR2_NT类型的集合。

26.5一类SVM

34.2.1升级前步骤

之前升级时，必须删除在其中创建的任何数据挖掘模型 Java和在中创建的任何挖掘活动 Oracle Data Miner Classic（早期版本的 Oracle Data Miner）。

警告：

在Oracle Database 12c中 ，Oracle Data Mining不支持Java API，Oracle Data Miner Classic无法针对Oracle Database 12 c运行。

34.2.1.1删除用Java创建的模型

如果10 g或11 g数据库包含使用Java创建的模型，请在升级数据库之前使用DBMS_DATA_MINING.DROP_MODEL例程删除模型。

34.2.1.2删除Oracle Data Miner Classic中创建的挖掘活动

如果您的数据库包含挖掘活动在Oracle Data Miner Classic中，删除挖掘活动并在升级数据库之前删除存储库。跟着这些步骤：

  1. 使用Data Miner Classic用户界面删除挖掘活动。

  2. 在SQL * Plus或SQL Developer中，删除这些表：

    DM4J $ ACTIVITIES DM4J $结果DM4J $ TRANSFORMS

    以及这些观点：

    DM4J $ MODEL_RESULTS_V DM4J $ RESULTS_STATE_V

完成这些步骤后，数据库中的任何模式中都必须没有前缀为DM4J$表或视图。

34.2.2.1.1从版本10 g升级

在 Oracle Data Mining 10g，数据挖掘元数据和PL / SQL包存储在DMSYS模式中。在Oracle Data Mining 11 g和12 c中 ， DMSYS不再存在;数据挖掘元数据对象存储在SYS 。

当Oracle Database 10g升级到12 c时 ，所有数据挖掘元数据对象和PL / SQL包都将从DMSYS迁移到SYS 。成功迁移后，将删除DMSYS模式及其关联对象。删除DMSYS后， SYS.DBA_REGISTRY视图不再将Oracle Data Mining列为组件。

升级到Oracle Database 12 c后 ，您无法再切换到数据挖掘评分引擎（ DMSE ）。评分引擎不会在Oracle数据库11 g或12c的存在。

34.2.2.1.2从版本11 g升级

如果你将Oracle Database 11g升级到Oracle Database 12c ，并且数据库先前已从Oracle Database 10g升级，然后DMSYS架构可能仍然存在。如果升级过程检测到DMSYS ，则会在升级期间显示警告消息并丢弃DMSYS 。

司机 描述

JDBC瘦驱动程序

1.6.6 Internet协议版本6支持

从Oracle Database 11g第2版开始，Oracle JVM支持数据库中Java代码的URL和机器名中的Internet协议版本6（IPv6）地址，这些地址将解析为IPv6地址。IPv6是由Internet工程任务组（IETF）设计的新的网络层协议，用于替换当前版本的Internet协议，Internet协议版本4（IPv4）。IPv6的主要优点是大地址空间，源自128位地址。IPv6还在诸如路由，网络自动配置，安全性，服务质量等方面改进了IPv4。

以下系统属性使您可以配置IPv6首选项：

java.net.preferIPv4Stack

如果操作系统上有IPv6，则底层本机套接字将是IPv6套接字。这使Java应用程序能够连接并接受来自IPv4和IPv6主机的连接。如果您的应用程序优先使用IPv4套接字，则可以将此属性设置为true 。如果将该属性设置为true ，则表示该应用程序无法与IPv6主机通信。

java.net.preferIPv6Addresses

即使操作系统上有IPv6，但出于向后兼容性原因，地址也会映射到IPv4。例如，依赖于仅访问IPv4服务的应用程序将受益于此类映射。如果要更改首选项以使用IPv4地址上的IPv6地址，则可以将java.net.preferIPv6Addresses属性设置为true 。这允许在预期应用程序连接到IPv6服务的环境中测试和部署应用程序。

注意：

在数据库初始化期间启用Java时，将加载IPv6支持所需的所有新System类。因此，如果您的应用程序没有包含在软件代码中的任何地址，那么您无需更改代码即可使用IPv6功能。

SQL翻译配置文件

SQL Translation Profile是一个数据库模式对象，用于指导非Oracle方言中的SQL语句如何转换为Oracle SQL方言。它还指导Oracle错误代码和SQLSTATES如何转换为其他供应商的SQL方言。

如果要将为非Oracle SQL数据库编写的客户端应用程序迁移到Oracle，可以创建SQL转换配置文件并对其进行配置以转换应用程序的SQL语句和错误。在运行时，应用程序为Oracle数据库中的连接设置配置文件以转换其SQL语句和错误。使用oracle.jdbc.sqlTranslationProfile属性设置此配置文件。

必要时，您可以使用服务器上的SQL翻译配置文件注册SQL语句和错误的自定义翻译。翻译SQL语句或错误时，首先查找自定义翻译，然后仅在找不到匹配项时调用翻译器。

请参阅“ SQL转换框架的体系结构 ”和“ 设置SQL转换配置文件 ” 。

转换JDBC标准参数标记

在提交用于转换的SQL语句之前，JDBC驱动程序在内部转换JDBC标准参数标记（ ? ）到格式的Oracle样式参数标记:b<n> 。

这里，参数标记的命名格式为:b<n> ，其中n是一个增量数，用于指定（ ? ）的位置JDBC PreparedStatement标记。

考虑UPDATE employees SET salary = salary * ?WHERE employee_id =？ PreparedStatement语句，其中，第一个参数标记（ ? ）将成为:b1和第二个参数标记（ ? ）将成为:b2 。

转换后，驱动程序将以下查询发送到服务器进行转换：

UPDATE员工SET薪水=薪水*：b1 WHERE employee_id =：b2

请注意，任何包含“？“如果将processEscapes属性的值更改为FALSE ，则在连接转换阶段期间参数标记失败。要成功转换，必须保留processEscapes属性的默认值。

转换参数标记有助于驱动程序自动重新排序转换时发生的任何参数更改。在转换时，必须在Oracle样式参数标记版本中注册必须在服务器上注册的任何自定义转换;服务器接收语句。请注意，自定义转换在Oracle样式中必须具有与原始查询中相同数量的参数标记。

有关支持的JDBC API的更多信息，请参阅JDBC应用程序的SQL转换API参考 。

执行翻译的Oracle方言查询

将JDBC标准参数标记转换为Oracle样式参数标记后，驱动程序将往返服务器以将查询转换为Oracle方言。一旦服务器接收到翻译的查询，驱动程序透明地处理参数中的任何重新排序，并且查询作为普通查询执行。

如果由于转换不可用而无法翻译查询，则服务器可能会引发错误或返回NULL ，具体取决于 DBMS_SQL_TRANSLATOR.ATTR_RAISE_TRANSLATION_ERROR配置文件属性。如果服务器返回NULL ，则假定原始的未翻译查询是由驱动程序翻译并执行的查询。

驱动程序将翻译保留在本地缓存中以保存将来的往返。

请注意，JDBC驱动程序可以支持由DBMS_SQL_TRANSLATOR.ATTR_RAISE_TRANSLATION_ERROR属性的任一值设置的转换错误（当由于转换不可用而无法转换查询时）。但是，必须在建立连接之前在服务器上设置该值。由于在会话中间更改此属性的值可能会导致行为不一致，因此Oracle建议您不要在会话期间翻转此属性的值。有关TRANSLATE_SQL过程的详细信息，请参阅Oracle Database PL / SQL包和类型参考 。

错误翻译

如果在查询执行期间抛出任何SQLException ，则驱动程序将透明地访问服务器并将Oracle代码中的异常转换为原始特定于供应商的代码。因此，生成的SQLException具有特定于供应商的代码和SQLSTATE以及特定于Oracle的SQLException作为原因。

与查询翻译类似，自定义错误翻译也可以在服务器上注册，并优先于标准翻译。DBMS_SQL_TRANSLATOR.ATTR_RAISE_TRANSLATION_ERROR属性对自定义错误转换的影响与查询转换相同。

请注意，只有在建立与服务器的连接后才会转换错误。因此，对于在建立与服务器的连接之前发生的错误，使用错误消息转换 。

关于BY和EXCEPT条款的注释

以下注释适用于BY和EXCEPT子句：

  • 如果多次AUDIT ......为同一个统一审计策略指定了BY ...语句，然后为每个语句中指定的用户联合启用策略。

  • 如果多次AUDIT ......EXCEPT ...语句是为同一个统一审计策略指定的，然后只有最近指定的语句生效。也就是说，除了最近一次AUDIT指定的用户外，所有用户都启用了该策略...EXCEPT ...声明。

  • 如果使用BY子句启用策略，并且您希望使用EXCEPT子句启用它，则必须首先使用NOAUDIT ...BY ...语句为当前启用了策略的所有用户禁用策略，然后使用AUDIT启用策略...EXCEPT ...声明。

  • 如果使用EXCEPT子句启用策略，并且您希望使用BY子句启用它，则必须首先使用NOAUDIT语句禁用审核策略。请注意，您不能使用NOAUDIT语句指定EXCEPT子句。然后，您可以使用AUDIT启用该策略...BY ...声明。

对BY和EXCEPT条款的限制

你不能指定一个AUDIT ......BY ...声明和AUDIT ......EXCEPT ...同一统一审计政策的声明。如果您尝试这样做，则会发生错误。

by_users_with_roles

指定此子句仅为已直接授予指定角色的用户启用policy 。如果您随后将其中一个角色授予其他用户，则该策略会自动向该用户授予权限。如果您随后从用户撤消其中一个角色，则该策略将不再适用于该用户。

当您连接到CDB时，如果policy是一个通用的统一审核策略，那么role必须是一个共同的角色。如果policy是本地统一审核策略，则role必须是您所连接的容器中的公共角色或本地角色。

在角色上启用本地审核策略

可以对本地角色以及公共角色启用本地审核策略。在公共角色上启用本地审核策略时，如果在本地或通常在容器中向用户授予公共角色，则会生成审核记录。

在角色上启用通用审核策略

通用审核策略只能在常用角色上启用。当对公共角色启用公共审核策略时，它会在ROOT容器中共同或本地授予用户公共角色时生成审核记录。

在[NOT]成功的时候

指定WHENEVER SUCCESSFUL仅审核SUCCESSFUL SQL语句和操作。

指定WHENEVER NOT SUCCESSFUL仅审核失败或导致错误的SQL语句和操作。

如果省略此子句，则无论成功与否，Oracle数据库都会执行审核。

  • RMAN客户端的版本必须等于目标数据库的版本。

  • 辅助数据库的版本必须等于RMAN客户端的版本。

  • 恢复目录模式的版本必须大于或等于RMAN客户端的版本。

C.4恢复目录导出和导入的跨版本兼容性

数据泵恢复目录的导出通常用作备份其内容的方法。在计划使用Data Pump Export进行恢复目录的逻辑备份时，必须牢记与跨Oracle Oracle数据库的数据库导出使用相关的某些兼容性问题。

无法将更高版本数据库的导出导入到在早期版本下运行的数据库中。必须使用用于恢复目录的最早版本的Oracle数据库中的导出实用程序导出恢复目录数据。

例如，要从Oracle Database 11g第2版（11.2）数据库导出恢复目录数据并将其导入Oracle数据库10g第1版（10.1.0.5）数据库以进行灾难恢复，必须使用Oracle中的导出实用程序数据库10g第1版（10.1.0.5）执行导出操作。否则，导入操作失败。

分层查询

如果表包含分层数据，则可以使用分层查询子句以分层顺序选择行：

hierarchical_query_clause :: =


hierarchy_query_clause.eps插图的描述

condition可以是如所描述的任何条件的条件 。

START WITH指定层次结构的根行。

CONNECT BY指定层次结构的父行和子行之间的关系。

  • 即使数据中存在CONNECT BY循环， NOCYCLE参数也会指示Oracle数据库从查询中返回行。将此参数与CONNECT_BY_ISCYCLE伪列一起使用，以查看哪些行包含循环。有关更多信息，请参阅CONNECT_BY_ISCYCLE Pseudocolumn 。

  • 在分层查询中，必须使用PRIOR运算符限定condition一个表达式以引用父行。例如，

    ...PRIOR expr = expr或... expr = PRIOR expr

    如果CONNECT BY condition是复合的，那么只有一个条件需要PRIOR运算符，尽管您可以有多个PRIOR条件。例如：

    CONNECT BY last_name！='King'AND PRIOR employee_id = manager_id ......CONNECT BY PRIOR employee_id = manager_id和PRIOR account_mgr_id = customer_id ...

    PRIOR是一元运算符，与一元+和 - 算术运算符具有相同的优先级。它为分层查询中当前行的父行计算紧接着的表达式。

    在将列值与相等运算符进行比较时，最常用的是PRIOR 。（ PRIOR关键字可以位于运营商的任何一侧。）PRIOR使Oracle使用列中父行的值。在CONNECT BY子句中理论上可以使用等号（=）以外的运算符。但是，这些其他运算符创建的条件可能会导致可能组合的无限循环。在这种情况下，Oracle在运行时检测循环并返回错误。

CONNECT BY条件和PRIOR表达式都可以采用不相关子查询的形式。但是， CURRVAL和NEXTVAL不是有效的PRIOR表达式，因此PRIOR表达式不能引用序列。

您可以使用CONNECT_BY_ROOT运算符进一步细化分层查询，以限定选择列表中的列。此运算符通过不仅返回直接父行而是返回层次结构中的所有祖先行来扩展分层查询的CONNECT BY [ PRIOR ]条件的功能。

Oracle处理分层查询，如下所示：

  • 首先计算连接（如果存在），无论是在FROM子句中还是在WHERE子句谓词中指定了连接。

  • 评估CONNECT BY条件。

  • 将评估任何剩余的WHERE子句谓词。

然后，Oracle使用来自这些评估的信息，使用以下步骤形成层次结构：

  1. Oracle选择层次结构的根行 - 满足START WITH条件的那些行。

  2. Oracle选择每个根行的子行。每个子行必须满足CONNECT BY条件相对于其中一个根行的条件。

  3. Oracle选择连续几代子行。Oracle首先选择步骤2中返回的行的子项，然后选择这些子项的子项，依此类推。Oracle总是通过评估关于当前父行的CONNECT BY条件来选择子项。

  4. 如果查询包含没有连接的WHERE子句，则Oracle将从层次结构中删除不满足WHERE子句条件的所有行。Oracle分别为每一行评估此条件，而不是删除不满足条件的行的所有子项。

  5. Oracle按照图9-1中所示的顺序返回行。在图中，孩子出现在父母的下方。有关分层树的说明，请参见图3-1 。

为了查找父行的子节点，Oracle计算父行的CONNECT BY条件的PRIOR表达式以及表中每行的另一个表达式。条件为真的行是父项的子项。CONNECT BY条件可以包含其他条件以进一步过滤查询选择的行。

如果CONNECT BY条件导致层次结构中出现循环，则Oracle将返回错误。如果一行既是父行（或祖父母或直接祖先）又是另一行的子（或孙子或直接后代），则会发生循环。

注意：

在分层查询中，不要指定ORDER BY或GROUP BY ，因为它们将覆盖CONNECT BY结果的层次结构顺序。如果要订购同一父级的兄弟节点行，请使用ORDER SIBLINGS BY子句。请参阅order_by_clause 。

在升级之前使用Oracle Data Pump迁移数据

在安装新的Oracle数据库软件之前，请使用此Oracle Data Pump过程从源数据库导出数据。然后将数据导入目标升级数据库。

  1. 安装新的Oracle数据库软件，并确保将其修补到最新的Oracle软件包修补程序或修补程序集更新（BP或PSU）。Oracle Database的安装步骤将在特定于操作系统的Oracle文档中介绍。

  2. 使用当前数据库附带的导出实用程序从当前数据库导出数据。

    导出期间和之后，当前数据库不能用于更新。如果导出后用户可以使用当前数据库进行更新，那么在使当前数据库可用之前，必须先安装过程，以便在导入完成后将当前数据库中所做的更改复制到新数据库。

    要获得可以更新导出数据库的一致点，请使用以下选项之一：

    • 在参数文件中设置FLASHBACK_TIME=SYSTIMESTAMP ，以便您导出的表中的数据获取的图像表示您要导出的所有表的相同单个时间点的表数据的已提交状态。

    • 使用FLASHBACK_SCN选择Export可用于启用闪回查询实用程序的特定系统更改编号（SCN）。

    使用闪回选项可增加UNDO的使用和保留。

  3. 创建新数据库。如果新数据库位于同一服务器上，并且它与当前数据库具有相同的名称，则在创建新数据库之前关闭当前数据库。

    （选项）您可以从源数据库更改存储参数。您可以在新数据库中预先创建表空间，用户和表，以通过更改存储参数来改善空间使用。使用SQL * Plus预创建表时，可以在原始数据库兼容模式下运行数据库，也可以考虑导入期间发生的特定数据定义转换。

  4. 在新的Oracle数据库环境中启动SQL * Plus，然后启动Oracle数据库实例。

    例如：

    $ SQLPLUS / AS SYSDBA SQL> STARTUP

  5. 如果您有预先创建的项目，请为数据泵导入指定TABLE_EXISTS_ACTION=APPEND选项。

    注意：

    如果在与源数据库相同的服务器上创建新数据库，并且您不想覆盖源数据库数据文件，则必须预先创建表空间并为数据泵导入设置以下参数值：

    • REUSE_DATAFILES=N表示数据泵导入

      （选项）您可以使用REMAP_DATAFILE, REMAP_TABLESPACE和REMAP_TABLE选项重新映射引用，以将转储文件集中的数据库名称导出到导入数据库中的新的非冲突名称。

    • DESTROY=N表示原始导入。

  6. 使用新数据库导入实用程序导入从当前数据库导出的对象。要将导入会话中的信息性消息和错误消息保存到文件，请使用以下参数：

    • 数据泵导入的LOGFILE参数

    • 原始Import的LOG参数

  7. 导入后，检查导入日志文件，以获取有关已成功完成的特定对象的导入的信息。如果出现故障，请检查有关任何失败对象的信息。

  8. 进一步使用导入方案，或使用创建数据库对象的SQL脚本来清理不完整的导入（或者可能启动全新的导入）。

    注意：

    如果数据泵导出或导入作业遇到不可恢复的错误，则可以在纠正导致故障的条件后重新启动作业。作业从故障点自动继续。

  9. 如果在导出后对当前数据库进行了更改，请确保在将这些更改提供给用户之前将这些更改传播到新数据库。请参阅此过程中的步骤1。

  10. 按照第4章“Oracle数据库的升级后任务”中的说明完成升级后所需的升级后任务。

使用网络链接导入完整的Oracle数据库

此数据库导出/导入方法是Oracle Data Pump迁移的替代方法，在迁移到其他存储系统时非常有用。

设置数据库链接并使用数据泵导入实用程序（ impdp ）执行从源数据库到目标数据库的完整数据库导入。使用此方法迁移数据意味着不会写入转储文件，因此您不必复制转储文件。使用不同的存储系统时，此方法特别有用。但是，您必须在Oracle数据库导出期间的早期版本Oracle Data Pump软件限制的范围内工作。

注意：

要避免互操作性错误，请确保在开始升级之前已将适当的补丁集应用于要升级的数据库。

  1. 确保源数据库中的导出用户具有DATAPUMP_EXP_FULL_DATABASE角色。

    您必须在创建数据库链接时指定此导出用户。

  2. 确保目标数据库中的导入用户具有DATAPUMP_IMP_FULL_DATABASE角色。

  3. 创建并测试源数据库和目标数据库之间的数据库链接。

  4. 使用以下命令语法启动数据泵导出，其中import_user是导入用户的用户名， db_link是导出用户拥有的数据库链接的名称：

    impdp import_user NETWORK_LINK = db_link FULL = Y NOLOGFILE = Y;

    注意：

    在导入数据库上运行此命令会隐式触发导出Oracle数据库上的数据泵导出操作（ expdp ）。

  5. 导入操作的日志文件写入DATA_PUMP_DIR目录。您可以通过运行以下命令来发现此目录的位置：

    SQL> select * from dba_directories，其中DIRECTORY_NAME喜欢'DATA_PUMP_DIR';

    在执行此命令时，请注意XDB存储库不会在完整数据库导出和导入操作中移动。LONG列和NESTED列也不会移动。但是，会移动用户创建的XML架构。

注意：

导入操作将在新目标服务器上重新创建用户。dba_users表中用户的创建日期显示实际的导入日期。到期日期更新为creation_date的值加上password_life_time列。目标数据库中dba_users表中的创建日期与源数据库中dba_users表的值不同。

降级Oracle数据库时的数据泵要求

您可以使用Data Pump获取向下兼容的转储文件。

将Oracle Data Pump与降级过程一起使用时，降级的Oracle数据库版本必须不超过要降级的版本的一个版本。

使用以下Data Pump Export方法之一获取向下兼容的转储文件：

  • 使用当前版本Oracle Database home中包含的Data Pump Export实用程序，并将VERSION参数设置为要降级的早期目标的版本。

    除非您使用VERSION参数设置为目标数据库的版本号创建了这些转储文件集，否则Data Pump Import无法读取由Oracle Data Pump版本创建的转储文件集。因此，执行降级的最佳方法是使用Data Pump Export，并将VERSION参数设置为要降级的目标数据库的版本号。

  • 使用带有VERSION参数的NETWORK_LINK参数将数据泵导出实用程序与DOWNGRADE一起使用。

    如果要降级的数据库的兼容级别早于您创建的导出转储文件集的版本，那么如果您希望从的Oracle数据库的兼容级别仍然可以通过数据库链接传输数据出口在两个主要版本号内。例如，如果一个数据库是Oracle Database 18c，则另一个数据库必须是12 c release 1（12.1）或12 c release 2（12.2）。您可以使用Data Pump Export以便在上一次转储文件导出期间将VERSION设置为不兼容的值进行恢复。

    注意：

    如果在安装新版本的Oracle数据库后提高兼容级别，以便它使用新版本中的功能，则使用这些新功能的对象无法降级。例如，如果使用Oracle Database 18c提供的长标识符，则具有这些长标识符的对象不能降级。降级后，当您尝试导入这些对象，并且数据泵导入实用程序尝试重新创建使用长标识符的对象时，您会收到错误。
