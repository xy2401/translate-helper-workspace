16.1关于广义线性模型

介绍广义线性模型（GLM）。

GLM包括并扩展了线性模型的类别。

线性模型做出一系列限制性假设，最重要的是，目标（因变量y ）正常分布，条件是具有恒定方差的预测变量值，而不管预测的响应值如何。线性模型及其限制的优点包括计算简单性，可解释的模型形式，以及计算关于拟合质量的某些诊断信息的能力。

广义线性模型放宽了这些限制，这些限制在实践中经常被违反。例如，二进制（是/否或0/1）响应在类之间没有相同的方差。此外，线性模型中的项之和通常可以具有非常大的范围，包括非常负的和非常正的值。对于二进制响应示例，我们希望响应是范围[0,1]中的概率。

广义线性模型通过两种机制适应违反线性模型假设的响应：链接函数和方差函数。链接函数将目标范围转换为潜在的无穷大到+无穷大，以便可以保持简单形式的线性模型。方差函数将方差表示为预测响应的函数，从而适应具有非恒定方差的响应（例如二元响应）。

Oracle Data Mining包括GLM系列模型中最受欢迎的两个成员，其最受欢迎的链接和方差函数：

  • 具有同一性链接和方差函数的线性回归等于常数1（响应值范围内的常数方差）。

  • 使用logit链接和二项式方差函数进行逻辑回归 。

广义线性模型（GLM）是一种参数化建模技术。参数模型对数据的分布做出假设。当满足假设时，参数模型可以比非参数模型更有效。

开发此类模型的挑战涉及评估假设的满足程度。因此，质量诊断是开发质量参数模型的关键。

16.2.3置信区间

通过广义线性模型（GLM）预测置信区间。

GLM具有预测置信区间的能力。除了预测每行的最佳估计和概率（仅分类）之外，GLM还识别其中预测（回归）或概率（分类）所在的区间。间隔的宽度取决于模型的精度和用户指定的置信度。

置信水平衡量模型是否确定真实值位于由模型计算的置信区间内。置信水平的热门选择是95％。例如，模型可能预测员工的收入为12.5万美元，并且您可以95％确定其在$ 90K和$ 160K之间。默认情况下，Oracle Data Mining支持95％的置信度，但可以配置该值。

注意：

使用系数统计返回置信区间。您还可以使用PREDICTION_BOUNDS SQL函数来获取模型预测的置信区间。

16.2.4岭回归

理解岭回归在数据中对奇点（精确多重共线性）的使用。

16.3可扩展的特征选择

神谕 Data Mining支持高度可扩展和自动化的版本特征选择和广义线性模型的生成。此功能可以增强算法的性能并提高准确性和可解释性。线性回归和二元Logistic回归均可使用特征选择和生成。

16.3.1特征选择

特征选择是选择要包含在模型中的术语的过程。模型中的术语越少，人类就越容易理解其含义。此外，某些列可能与模型尝试预测的值无关。删除此类列可以提高模型的准确性。

16.3.1.1配置功能选择

特征选择是广义线性模型的构建设置。默认情况下不启用它。配置为功能选择时，算法会自动确定适当的默认行为，但可以使用以下配置选项：

  • 特征选择标准可以是AIC，SBIC，RIC或α投资。当特征选择标准是α投资时，特征接受可以是严格的也可以是放松的。

  • 可以指定最大功能数。

  • 功能可以在最终模型中修剪。修剪基于线性回归的t统计或逻辑回归的wald统计。

16.3.1.2特征选择和岭回归

特征选择和岭回归是互斥的。启用特征选择后，算法不能使用脊。

注意：

如果将模型配置为同时使用特征选择和岭回归，则会出现错误。

16.3.2特征生成

特征生成是将术语转换添加到模型中的过程。特征生成增强了模型的功能，以适应目标和预测变量之间更复杂的关系。

16.5 GLM求解器

了解广义班轮模型（GLM）的不同求解器。

GLM算法支持四种不同的求解器：Cholesky，QR，随机梯度下降（SGD），L-BFGS和交替方向乘法器（ADMM）。Cholesky和QR求解器采用经典分解方法。Cholesky求解器与QR求解器相比速度更快，但在数值上不太稳定。QR解算器在没有正则化的帮助下处理更好的秩缺陷问题。

SGD和L-BFGS ADMM求解器最适合高维数据。SGD求解器采用随机梯度下降优化算法，而L-BFGS ADMM在交替方向乘法器框架内使用Broyden-Fletcher-Goldfarb-Shanno优化算法。SGD速度快但对参数敏感，需要适当缩放的数据才能实现良好的收敛。L-BFGS算法解决了无约束优化问题，并且比SGD更稳定，更稳健。此外，L-BFGS与ADMM结合使用，从而实现高效的分布式优化方法，并且通信成本低。

16.6.1线性回归的数据准备

了解广义线性模型（GLM）的自动数据准备（ADP）。

启用自动数据准备（ADP）时，算法会根据输入数据属性和其他设置选择转换。对于数值数据，变换可以包括以下中的一个或多个：减去平均值，按标准偏差缩放，或执行相关变换（Neter等，1990）。如果将相关变换应用于数字数据，则它也应用于分类属性。

在标准化之前，分类属性被分解为N-1列，其中N是属性基数。爆炸转换期间省略了最常见的值（模式）。在频率最高关系的情况下，属性值按字母顺序按升序排序，并且在爆炸期间省略列表中的第一个值。无论是否启用ADP，都会发生此爆炸转换。

在高基数分类属性的情况下，所描述的变换（爆炸后标准化）可以增加构建数据大小，因为结果数据表示是密集的。要减少内存，磁盘空间和处理要求，请使用其他方法。在这种情况下，必须谨慎使用VIF统计数据。

16.6.2 Logistic回归的数据准备

分类属性分解为N -1列，其中N是属性基数。爆炸转换期间省略了最常见的值（模式）。在频率最高关系的情况下，属性值按字母顺序按升序排序，并且在爆炸期间省略列表中的第一个值。无论是否启用自动数据准备（ADP），都会发生此爆炸转换。

启用ADP时，数值属性按标准偏差进行缩放。这种可变性的度量被计算为每个属性相对于原点的标准偏差（不是均值）（Marquardt，1980）。

16.6.3缺少值

在构建或应用模型时，Oracle数据挖掘会自动使用模式将分类属性的均值和缺失值替换为数值属性的缺失值。

您可以配置“广义线性模型”以覆盖缺失值的默认处理。使用ODMS_MISSING_VALUE_TREATMENT设置，您可以使算法删除训练数据中具有缺失值的行，而不是使用均值或模式替换它们。但是，在应用模型时，Oracle Data Mining会执行通常的均值/模式缺失值替换。因此，评分生成的统计信息可能与构建模型时生成的统计信息不匹配。

如果要在模型评分中删除缺少值的行，则必须显式执行转换。要使构建和应用统计信息匹配，必须在执行应用操作之前从评分数据中删除带有NULL的行。您可以通过创建视图来完成此操作。

CREATE VIEW viewname AS SELECT * from tablename WHERE column_name1为NOT NULL且column_name2为NOT NULL且column_name3为NOT NULL .....

注意：

在Oracle Data Mining中，嵌套数据中的缺失值表示稀疏性，而不是随机丢失的值。

值ODMS_MISSING_VALUE_DELETE_ROW仅对没有嵌套列的表有效。如果此值与嵌套数据一起使用，则会引发异常。

8.2 PDB搬迁的目的

这种技术是在最短或没有停机时间内移动PDB的最快方法。否则，拔出源PDB需要PDB中断，直到PDB插入目标CDB。

8.3 PDB搬迁的工作原理

该操作将与PDB关联的文件移动到新位置，将PDB添加到目标CDB，然后打开PDB。

PDB重定位执行源PDB数据文件的在线块级副本，重做和撤消，而源PDB使用活动会话打开。当目标PDB由于ALTER PLUGGABLE DATABASE OPEN语句而联机时，Oracle数据库终止活动会话并关闭源PDB。

Oracle Database 18c引入了数据库通用会话耗尽，其中活动会话可以在计时器下正常退出。计时器到期后，Oracle数据库终止所有活动会话并将它们重新连接到重定位的PDB。对于在数据库服务和PDB级别调用的所有维护操作，默认情况下启用此功能：停止服务，重定位服务，重定位PDB和停止PDB。

PDB重定位服务在Oracle RAC和单实例环境中可用，其中PDB在同一集群数据库中的CDB实例之间或不同的CDB之间重定位。在Oracle Clusterware中，Rapid Home Provisioning功能可自动执行PDB重定位。在Oracle RAC环境中进行细粒度重定位的一个示例是PDB实例之间的服务重定位。Oracle RAC和Oracle Clusterware提供了丰富的高可用性环境，可进一步减少重定位期间对连接客户端的影响。例如，共享存储可以最小化或消除复制数据文件的必要性。

8.3.1重定位或停止PDB时的服务器会话耗尽

计划维护的关键要求是耗尽或失败PDB会话，以便不中断应用程序工作。

从Oracle Database 18c开始，数据库会在PDB重新定位或停止时自动标记要排空的会话。数据库使用一组可扩展的规则来确定何时耗尽数据库会话，该会话一直持续到规则满足为止。规则包括以下内容：

  • 标准应用程序服务器测试有效性

  • 自定义SQL测试的有效性

  • 请求边界正在使用中，并且没有活动请求

  • 请求边界正在使用中，当前请求已结束

  • 会话具有一个或多个可恢复的会话状态，并且可以在故障转移时重新创建

典型的用例是应用程序服务器和池化应用程序，它们在从连接池借用，返回池连接以及批量提交时测试连接。在耗尽会话时，数据库会自动拦截连接测试，关闭连接，然后返回测试的失败状态。接收到失败状态后，应用层可以请求不同的连接。这样，应用程序不会中断。

对于最小化对客户端影响的最佳配置，请考虑将Oracle RAC与FAN和应用程序连续性一起使用。

8.3.2.1公共监听器网络中的PDB重定位

当源和目标位置共享公共侦听器网络时，不需要转发客户端连接，因为SQL * Net层隐式转发客户端连接。

可用性正常

当侦听器网络很常见时，在CREATE PLUGGABLE DATABASE ...指定AVAILABILITY NORMAL子句CREATE PLUGGABLE DATABASE ...RELOCATE 。此选项是默认选项。以下情况是AVAILABILITY NORMAL典型用例：

  • 共享监听器

    如果在旧位置和新位置对PDB使用相同的侦听器，则在重定位完成时，新连接将自动路由到新位置。这种情况通常是同一主机中CDB之间的重定位。在这种情况下，PDB在其新位置与监听器重新注册。不需要额外的连接处理。

  • 交叉注册的听众

    如果PDB使用不同的侦听器，并且您通过配置local_listener和remote_listener参数来使用各自侦听器的交叉注册，则重定位是无缝的。PDB服务的可用性和位置会自动注册到两个听众。这种情况通常是数据中心内主机之间的重定位，可能是为了实现负载平衡。

在共享和交叉注册的侦听器环境中，来自所有数据库的服务将发布到公共侦听器网络。因此，公共侦听器网络立即知道重定位PDB的服务。为避免服务名称空间冲突，PDB服务定义在公共侦听器网络中必须是唯一的。

公共监听网络中的重定位阶段

  1. 用户发出CREATE PLUGGABLE DATABASE ...RELOCATE AVAILABILITY NORMAL 。

    此步骤执行源PDB从其原始位置到其目标位置的热克隆。源PDB将数据文件，撤消块和重做块从隐式开始SCN标记复制到目标PDB。

    完成此步骤后，将存在此PDB的两个事务一致性副本：一个位于源容器中，另一个位于目标容器中。在操作期间，源PDB上的处理继续不间断。连接到源PDB的应用程序或应用程序的用户不知道正在进行重定位。

    所有现有应用程序连接以及在此步骤中创建的新连接将继续连接到源PDB。

  2. 用户发出ALTER PLUGGABLE DATABASE OPEN 。

    以下操作在后台发生：

    1. 目标PDB隐式设置结束SCN标记，并应用完成介质恢复所需的任何重做或撤消以满足隐式结束SCN标记。

    2. 在目标PDB上进行介质恢复时，Oracle数据库会在源PDB上启动活动会话耗尽。

    3. PDB服务向监听器注册，并且可在目标CDB上使用。

    4. 源PDB已关闭。

    5. 目标PDB以读/写模式打开。

      此步骤完成PDB重定位到目标CDB。在操作结束时，连接指向新重定位的PDB。

      在读/写模式下打开PDB后，其状态为NORMAL 。如果尝试以只读模式打开PDB，数据库将返回错误。

8.3.2.2隔离监听器网络中的PDB重定位

当独立侦听器不使用交叉注册时，目标CDB和源CDB中的侦听器不了解彼此或其各自的已发布服务。

可用性最大

CREATE PLUGGABLE DATABASE ...的AVAILABILITY MAX子句CREATE PLUGGABLE DATABASE ...RELOCATE隐式指示SQL * Net层重新配置原始侦听器。在数据中心之间重新定位PDB时，这种情况可能很常见。在更新Oracle Internet Directory（OID）或LDAP服务器或修改客户端连接时，此配置是临时的。

如果本地侦听器重定向到Oracle RAC配置中的单个客户端访问名称（SCAN）侦听器，则此侦听器可能需要进一步将客户端连接请求重定向到另一个群集节点。默认情况下，Oracle Net侦听器不支持多个重定向。因为任何SCAN听众可路由连接请求到任何节点，设置ALLOW_MULTIPLE_REDIRECTS_ listener_name参数到listener_name每SCAN收听者，并且将其设置在每一个listener.ora集群中的文件。例如，如果SCAN侦听器名为listener_scan1 ， listener_scan2和listener_scan3 ，则每个目标主机上的listener.ora文件应具有以下设置：

ALLOW_MULTIPLE_REDIRECTS_LISTENER_SCAN1 = YES ALLOW_MULTIPLE_REDIRECTS_LISTENER_SCAN2 = YES ALLOW_MULTIPLE_REDIRECTS_LISTENER_SCAN3 = YES

警告：

不要为节点侦听器设置ALLOW_MULTIPLE_REDIRECTS_ listener_name参数，因为它可能允许在某些网络配置中进行无限重定向循环。

孤立的侦听器网络中的重定位阶段

  1. 用户发出CREATE PLUGGABLE DATABASE ...RELOCATE AVAILABILITY MAX 。

    此步骤执行源PDB从其原始位置到其目标位置的热克隆。源PDB将数据文件，撤消块和重做块从隐式开始SCN标记复制到目标PDB。

  2. 用户发出ALTER PLUGGABLE DATABASE OPEN 。

    以下操作在后台发生：

    1. 目标PDB隐式设置结束SCN标记，并应用完成介质恢复所需的任何重做或撤消以满足隐式结束SCN标记。

    2. 在目标PDB上进行介质恢复时，Oracle数据库会在源PDB上启动活动会话耗尽。

    3. LISTENER_NETWORKS初始化参数在源PDB中使用转发地址进行隐式更新，并且源CDB的侦听器PDB服务使用转发地址进行更新。

    4. 在介质恢复完成时，目标PDB以只读模式打开。

      在此阶段，仅允许对目标PDB进行查询。查询的行为与在源PDB上运行的完全相同。但是，尝试DML的连接无法完成。

    5. 只读连接立即转发到新的主机侦听器，新的读/写连接被转发到新的主机侦听器，在那里它们旋转直到目标PDB以一致的状态打开。

    6. 源PDB执行SHUTDOWN IMMEDIATE ，终止持久连接。

    7. 目标PDB以读/写模式打开。

      此步骤完成PDB重定位到目标CDB。在操作结束时，连接指向新重定位的PDB。

      在读/写模式下打开PDB后，其状态为NORMAL 。如果尝试以只读模式打开PDB，数据库将返回错误。

注意：

称为逻辑删除PDB的工件保留在源CDB中以保护PDB的命名空间并保留侦听器转发配置，直到更新完成。在源CDB的根目录中，逻辑删除PDB在V$CONTAINERS可见，状态为RELOCATED 。更改应用程序连接字符串以提供与目标PDB的直接连接时，可以从源CDB中删除逻辑删除PDB。

8.5使用CREATE PLUGGABLE DATABASE重定位PDB

CREATE PLUGGABLE DATABASE ...RELOCATE语句将PDB移动到其他容器。

目标CDB是PDB重定位到的CDB。目标PDB是重新定位的PDB。在CREATE PLUGGABLE DATABASE ...RELOCATE操作完成后，Oracle数据库从源CDB移动PDB，然后将其添加到目标CDB。

一般先决条件

在“ 表5-3 ”中解决适用于重新定位PDB的问题。该表描述了您必须根据不同因素指定的CREATE PLUGGABLE DATABASE子句。此外，完成“ PDB创建的一般先决条件”中描述的先决条件 。

具体先决条件

在重新定位PDB之前，您必须满足以下先决条件：

  • 源CDB必须处于本地撤消模式。

  • 在源CDB中，您必须在所有数据库实例中保存服务并打开PDB的状态。以管理员身份登录CDB root，并发出以下语句：

    ALTER PLOGGABLE DATABASE ALL SAVE STATE INSTANCES = ALL;

    此步骤确保PDB重定位操作自动启动目标CDB中的PDB服务。

  • 在目标CDB中，当前用户必须具有CDB根目录中的CREATE PLUGGABLE DATABASE系统特权。

  • 如果目标CDB未处于ARCHIVELOG模式，则PDB必须在操作期间处于打开的只读模式。如果目标CDB处于ARCHIVELOG模式，则此要求不适用。

  • 以下先决条件适用于数据库链接：

    • 数据库链接必须启用从目标CDB到源CDB的连接。数据库链接必须连接到源CDB的根目录。如果PDB是应用程序PDB，则数据库链接必须连接到其应用程序根目录。

    • 数据库链接在PDB的当前CDB中连接的用户必须具有CREATE PLUGGABLE DATABASE系统特权或SYSOPER管理特权。

    • 如果数据库链接连接到PDB当前CDB中的CDB根目录，则数据库链接所连接的用户必须是普通用户。

  • PDB当前CDB的平台和重新安置的CDB必须满足以下要求：

    • 它们必须具有相同的字节顺序。

    • 安装在源平台上的数据库选项必须与目标平台上安装的数据库选项相同或者是其子集。

  • 如果PDB要重定位到的CDB的字符集不是AL32UTF8，则PDB的当前CDB和要重定位的CDB必须具有兼容的字符集和国家字符集。

    如果PDB要重定位到的CDB的字符集是AL32UTF8，则此要求不适用。

  • 如果要创建应用程序PDB，则源PDB和目标应用程序容器必须具有相同的应用程序名称和版本。

重新安置PDB：

  1. 在SQL * Plus中，以具有CREATE PLUGGABLE DATABASE系统特权的用户身份登录到目标CDB。

  2. 确保当前容器是目标CDB或目标应用程序容器的根。

  3. 运行CREATE PLUGGABLE DATABASE ...RELOCATE语句与FROM子句。

17.1关于广义线性模型

介绍广义线性模型（GLM）。

GLM包括并扩展了线性模型的类别。

线性模型做出一系列限制性假设，最重要的是，目标（因变量y ）正常分布，条件是具有恒定方差的预测变量值，而不管预测的响应值如何。线性模型及其限制的优点包括计算简单性，可解释的模型形式，以及计算关于拟合质量的某些诊断信息的能力。

广义线性模型放宽了这些限制，这些限制在实践中经常被违反。例如，二进制（是/否或0/1）响应在类之间没有相同的方差。此外，线性模型中的项之和通常可以具有非常大的范围，包括非常负的和非常正的值。对于二进制响应示例，我们希望响应是范围[0,1]中的概率。

广义线性模型通过两种机制适应违反线性模型假设的响应：链接函数和方差函数。链接函数将目标范围转换为潜在的无穷大到+无穷大，以便可以保持简单形式的线性模型。方差函数将方差表示为预测响应的函数，从而适应具有非恒定方差的响应（例如二元响应）。

Oracle Data Mining包括GLM系列模型中最受欢迎的两个成员，其最受欢迎的链接和方差函数：

  • 具有同一性链接和方差函数的线性回归等于常数1（响应值范围内的常数方差）。

  • 使用logit链接和二项式方差函数进行逻辑回归 。

17.5 GLM求解器

了解广义班轮模型（GLM）的不同求解器。

GLM算法支持四种不同的求解器：Cholesky，QR，随机梯度下降（SGD），L-BFGS和交替方向乘法器（ADMM）。Cholesky和QR求解器采用经典分解方法。Cholesky求解器与QR求解器相比速度更快，但在数值上不太稳定。QR解算器在没有正则化的帮助下处理更好的秩缺陷问题。

SGD和L-BFGS ADMM求解器最适合高维数据。SGD求解器采用随机梯度下降优化算法，而L-BFGS ADMM在交替方向乘法器框架内使用Broyden-Fletcher-Goldfarb-Shanno优化算法。SGD速度快但对参数敏感，需要适当缩放的数据才能实现良好的收敛。L-BFGS算法解决了无约束优化问题，并且比SGD更稳定，更稳健。此外，L-BFGS与ADMM结合使用，从而实现高效的分布式优化方法，并且通信成本低。

17.6.1线性回归的数据准备

了解广义线性模型（GLM）的自动数据准备（ADP）。

启用自动数据准备（ADP）时，算法会根据输入数据属性和其他设置选择转换。对于数值数据，变换可以包括以下中的一个或多个：减去平均值，按标准偏差缩放，或执行相关变换（Neter等，1990）。如果将相关变换应用于数字数据，则它也应用于分类属性。

在标准化之前，分类属性被分解为N-1列，其中N是属性基数。爆炸转换期间省略了最常见的值（模式）。在频率最高关系的情况下，属性值按字母顺序按升序排序，并且在爆炸期间省略列表中的第一个值。无论是否启用ADP，都会发生此爆炸转换。

在高基数分类属性的情况下，所描述的变换（爆炸后标准化）可以增加构建数据大小，因为结果数据表示是密集的。要减少内存，磁盘空间和处理要求，请使用其他方法。在这种情况下，必须谨慎使用VIF统计数据。

17.6.2 Logistic回归的数据准备

分类属性分解为N -1列，其中N是属性基数。爆炸转换期间省略了最常见的值（模式）。在频率最高关系的情况下，属性值按字母顺序按升序排序，并且在爆炸期间省略列表中的第一个值。无论是否启用自动数据准备（ADP），都会发生此爆炸转换。

启用ADP时，数值属性按标准偏差进行缩放。这种可变性的度量被计算为每个属性相对于原点的标准偏差（不是均值）（Marquardt，1980）。

在构建或应用模型时，Oracle数据挖掘会自动使用模式将分类属性的均值和缺失值替换为数值属性的缺失值。

您可以配置“广义线性模型”以覆盖缺失值的默认处理。使用ODMS_MISSING_VALUE_TREATMENT设置，您可以使算法删除训练数据中具有缺失值的行，而不是使用均值或模式替换它们。但是，在应用模型时，Oracle Data Mining会执行通常的均值/模式缺失值替换。因此，评分生成的统计信息可能与构建模型时生成的统计信息不匹配。

如果要在模型评分中删除缺少值的行，则必须显式执行转换。要使构建和应用统计信息匹配，必须在执行应用操作之前从评分数据中删除带有NULL的行。您可以通过创建视图来完成此操作。

CREATE VIEW viewname AS SELECT * from tablename WHERE column_name1为NOT NULL且column_name2为NOT NULL且column_name3为NOT NULL .....

注意：

在Oracle Data Mining中，嵌套数据中的缺失值表示稀疏性，而不是随机丢失的值。

值ODMS_MISSING_VALUE_DELETE_ROW仅对没有嵌套列的表有效。如果此值与嵌套数据一起使用，则会引发异常。

10.1 Oracle数据库高级队列概述

Oracle Streams是一种新的信息共享功能，可提供复制，消息队列，数据仓库加载和事件通知。它也是Oracle数据库高级队列（AQ）的基础。

高级队列是集成的消息队列功能，它公开了Oracle Database AQ的消息队列功能，使应用程序能够：

  • 执行与Oracle数据库中的SQL操作类似的消息队列操作

  • 通过AQ队列中的消息异步通信

  • 与数据库集成，实现消息队列的前所未有的操作简单性，可靠性和安全性

  • 审核并跟踪消息

  • 支持同步和异步通信模式

在OCCI应用程序中使用AQ的优点包括：

  • 创建以一致，可靠，安全和自主的方式相互通信的应用程序

  • 将消息存储在数据库表中，将数据库的可靠性和可恢复性带到消息传递基础结构中

  • 自动保留数据库中的消息以进行审计和商业智能

  • 创建利用消息传递的应用程序，而无需处理不同的安全性，数据类型或操作模式

  • 利用数据库的事务特征

由于传统的消息传递解决方案具有单个订户队列，因此必须为彼此通信的每对应用程序创建一个队列。AQ的发布/订阅协议可以轻松地将其他应用程序（订阅者）添加到多个应用程序之间的对话中。

4.7.3优化传播

AQ传播作业由Oracle Scheduler运行。可以通过这些方式安排传播。

  • 传播永久运行或指定持续时间的专用计划。此模式提供最低的传播延迟。

  • 传播在指定时间间隔内定期运行的定期计划。当传播可以以批处理模式运行时，可以使用此方法。

  • 基于事件的系统，其中当存在要传播的消息时开始传播。此模式可以更有效地利用可用资源，同时仍能提供快速响应时间。

管理员可以选择最符合应用程序性能要求的计划。

Oracle Scheduler将为传播计划启动所需数量的作业队列进程。由于调度程序优化了吞吐量，因此如果系统负载很重，则可能无法运行某些传播作业。资源管理器可用于更好地控制调度决策。特别地，将传播作业与不同资源组相关联可以允许调度的公平性，这在重负载情况下可能是重要的。

In setting the number of JOB_QUEUE_PROCESSES, DBA应该知道此数字由必须传播消息的队列数以及必须传播消息的目标（而不是队列）数决定。

调度算法处理传播。该算法优化了可用的作业队列进程，并最大限度地减少了消息在排入源队列后显示在目的地所需的时间，从而提供了近OLTP操作。该算法可以处理无限数量的调度和各种类型的故障。传播尝试最佳地使用可用作业队列进程时，要启动的作业队列进程数也取决于与传播无关的作业的存在，例如复制作业。因此，使用以下指南从调度算法中获得最佳结果非常重要。

调度算法使用作业队列过程如下（对于此讨论，活动调度是具有有效当前窗口的调度）：

  • 如果活动计划的数量少于作业队列进程数量的一半，则获取的作业队列进程数对应于活动计划的数量。

  • 如果活动计划的数量超过作业队列进程数量的一半，则在获取作业队列进程数量的一半之后，将多个活动计划分配给所获取的作业队列进程。

  • 如果系统过载（所有计划都忙于传播），则根据可用性，可以获得比作业队列进程总数少一个的其他作业队列进程。

  • 如果进程处理的活动调度都没有传播消息，则释放该作业队列进程。

  • 该算法通过将时间表从高负载进程转移到轻负载进程来执行自动负载平衡，从而不会过度加载进程。

4.7.4处理传播失败

调度算法对处理故障具有强大的支持。这些是阻止消息传播的常见故障。

  • 数据库链接失败

  • 远程数据库不可用

  • 远程队列不存在

  • 远程队列未启动

  • 尝试将消息排入远程队列时发生安全性违规

在所有这些情况下， DBA_QUEUE_SCHEDULES视图中会报告相应的错误消息。

当调度中发生错误时，在作为失败次数的函数的重试周期之后再次尝试该调度中的消息传播。重试超过系统定义的最大值后，将禁用计划。

如果修复了导致错误的问题并启用了计划，则指示上一个错误日期，时间和消息的错误字段将继续显示错误信息。仅当在该计划中成功传播消息时，才会重置这些字段。

什么时候“ ?“占位符与prepareStatement()方法一起使用，驱动程序内部更改” ?“对于Oracle风格的参数，因为服务器端转换器只能使用Oracle风格的标记。这是区分绑定变量所必需的。如果不是，则绑定变量顺序的任何变化都是无法区分的。替换的oracle样式标记遵循以下格式:b< n >其中< n >是增量编号。例如， exec sample_proc(?,?) 变为exec sample_proc(:b1,:b2) 。

为了进一步举例说明，考虑一种供应商格式的场景，其中选择前三行的供应商查询是SELECT * FROM employees WHERE first_name=?AND employee_id =？TOP 3 。查询必须转换为oracle方言。在这种情况下，将在服务器上注册以下转换：

从：

SELECT * FROM employees WHERE first_name =：b1 AND employee_id =：b2 TOP 3

至： 

SELECT * FROM employees WHERE first_name =：b1 AND employee_id =：b2 AND ROWNUM <= 3

有关更多信息，请参见SqlTranslationVersion和“ JDBC应用程序的SQL转换 ” 。

27.2.1 SQL计划管理的好处

SQL计划管理可以改进或保留数据库升级以及系统和数据更改中的SQL性能。

具体而言，好处包括：

  • 安装新优化程序版本的数据库升级通常会导致一小部分SQL语句的计划更改。

    大多数计划更改都会导致改进或无性能变化。但是，某些计划更改可能会导致性能下降。SQL计划基准可显着降低升级产生的潜在回归。

    升级时，数据库仅使用计划基准中的计划。数据库将不在当前基线中的新计划放入保留区域，然后对其进行评估以确定它们是否使用的资源少于基准中的当前计划。如果计划表现更好，那么数据库会将它们提升到基线;否则，数据库不会提升它们。

  • 正在进行的系统和数据更改可能会影响某些SQL语句的计划，从而可能导致性能下降。

    SQL计划基准有助于最大限度地降低性能回归并稳定SQL性能。

  • 新应用程序模块的部署将新的SQL语句引入数据库。

    应用程序软件可以使用在标准测试配置中为新语句开发的适当SQL执行计划。如果系统配置与测试配置明显不同，则数据库可以随时间推移SQL计划基准以产生更好的性能。

27.2.2 SQL计划基准和SQL配置文件之间的差异

SQL配置文件和SQL计划基准都通过确保优化程序仅使用最佳计划来帮助提高SQL语句的性能。

配置文件和基线都是使用提示在内部实现的。但是，这些机制存在显着差异，包括：

  • 通常，SQL计划基准是主动的，而SQL配置文件是反应性的。

    通常，您会在发生重大性能问题之前创建SQL计划基准。SQL计划基准阻止优化程序在将来使用次优计划。

    调用SQL Tuning Advisor时，数据库会创建SQL配置文件，通常只有在 SQL语句显示高负载症状后才会执行此操作。SQL配置文件主要用于提供持续解决导致计划不理想的优化程序错误。因为SQL配置文件机制是被动的，所以当发生剧烈的数据库更改时，它无法保证稳定的性能。

27.3.1.1自动初始计划捕获的资格

要有资格进行自动计划捕获，执行的语句必须是可重复的，并且任何捕获筛选器都不能排除它。

默认情况下，数据库将所有可重复的SQL语句视为符合捕获条件，但以下情况除外：

  • 未指定AS SELECT子句时的CREATE TABLE

  • DROP TABLE

  • 插入 ...VALUES

第一次检查资格是重复执行。如果语句执行的次数少于两次，则数据库不认为它符合SQL计划基准的条件。如果语句至少执行两次，那么根据定义它是可重复的，因此数据库认为它有资格进行进一步检查。

注意：

SQL计划管理不保护使用EXPLAIN PLAN解释但尚未执行的语句。

  • 没有经常执行到重要

  • 不是资源密集型的

  • 不够复杂，无法从SQL计划管理中受益

对于指定的参数，过滤器包括（ allow=>TRUE ）或排除（ allow=>FALSE ）具有指定值的语句的计划。要获得捕获资格，任何过滤器都不得排除可重复的声明。DBMS_SPM.CONFIGURE过程支持SQL文本的过滤器，解析模式名称，模块和操作。

27.4计划选择

SQL计划选择是优化程序根据存储的计划历史记录检测计划更改的能力，以及使用SQL计划基准来选择计划以避免潜在的性能回归。

当数据库执行SQL语句的硬解析时，优化程序会生成最佳成本计划。默认情况下，优化程序然后尝试在语句的SQL计划基准中查找匹配计划。如果不存在计划基准，则数据库将运行具有最佳成本计划的语句。

如果存在计划基准，则优化程序行为取决于新生成的计划是否在计划基准中：

  • 如果新计划在基准中，则数据库使用找到的计划执行语句。

  • 如果新计划不在基准中，则优化程序会将新生成的计划标记为未接受，并将其添加到计划历史记录中。优化程序行为取决于计划基准的内容：

    • 如果计划基准中存在固定计划，则优化程序将使用具有最低成本的固定计划。

    • 如果计划基准中不存在固定计划，则优化程序将使用具有最低成本的基准计划。

    • 如果计划基准中不存在可重现的计划，如果基准中的每个计划都引用了已删除的索引，则可能会发生这种情况，然后优化程序使用新生成的基于成本的计划。

27.5.1计划演变的目的

通常，语句的SQL计划基准以一个已接受的计划开始。

但是，某些SQL语句在不同条件下使用不同计划执行时表现良好。例如，具有绑定变量的SQL语句可能具有多个最佳计划，这些绑定变量的值导致不同的选择性。创建物化视图或索引或重新分区表可能会使当前计划比其他计划更昂贵。

如果从未将新计划添加到SQL计划基准中，则某些SQL语句的性能可能会降低。因此，有时需要将新接受的计划发展为SQL计划基准。计划演变通过在将新计划包含在SQL计划基准之前验证新计划的性能来防止性能回归。

27.5.2计划进化如何运作

计划演变涉及验证和添加计划。

具体而言，计划发展包括以下不同步骤：

  1. 验证

    优化程序确保未接受的计划至少与SQL计划基准中的已接受计划执行（称为计划验证 ）。

  2. 添加

    在数据库证明未接受的计划与已接受的计划一致之后，数据库会将计划添加到基准。

在计划演变的标准情况下，优化程序按顺序执行上述步骤，以便SQL计划管理无法使用新计划，直到优化程序验证相对于SQL计划基准的计划性能。但是，您可以将SQL计划管理配置为执行一个步骤而不执行另一个步骤。下图显示了计划演变的可能路径。

ENABLE条款

指定ENABLE以并行执行会话中的后续语句。这是DDL和查询语句的缺省值。

  • DML ：如果指定了并行提示或并行子句，则DML语句以并行模式执行。

  • DDL ：如果指定了parallel子句，则DDL语句以并行模式执行。

  • QUERY ：如果指定了并行提示或并行子句，则以并行模式执行查询。

对ENABLE条款的限制

您不能使用ENABLE指定可选的PARALLEL integer 。

禁用条款

指定DISABLE以串行方式执行会话中的后续语句。这是DML语句的默认值。

  • DML ：DML语句是串行执行的。

  • DDL ：DDL语句是串行执行的。

  • QUERY ：查询是按顺序执行的。

对DISABLE子句的限制

您不能使用DISABLE指定可选的PARALLEL integer 。

FORCE条款

FORCE强制在会话中并行执行后续语句。如果未指定parallel子句或提示，则使用默认的并行度。此子句将覆盖会话中后续语句中指定的任何parallel_clause ，但会被并行提示覆盖。

  • DML ：如果没有违反并行DML限制，则会话中的后续DML语句将使用默认的并行度执行，除非在此子句中指定了度数。

  • DDL ：会话中的后续DDL语句以默认的并行度执行，除非在此子句中指定了度。产生的数据库对象将与它们相关联的主流并行度。

    指定FORCE DDL自动导致在此会话中创建的所有表都使用默认的并行级别创建。效果与在CREATE TABLE语句中指定parallel_clause （具有默认度）相同。

  • QUERY ：使用默认的并行度执行后续查询，除非在此子句中指定了度数。

PARALLEL 整数

指定一个整数以显式指定并行度：

  • 对于FORCE DDL ，该度将覆盖后续DDL语句中的任何并行子句。

  • 对于FORCE DML和QUERY ，该度数将覆盖当前为数据字典中的表存储的程度。

  • 通过提示在语句中指定的学位将覆盖被强制的学位。

无论此子句如何，以下类型的DML操作都不会并行化：

  • 群集表的操作

  • 嵌入式函数的操作，可以写入或读取数据库或包状态

  • 具有可触发的触发器的表的操作

  • 对包含对象类型或LONG或LOB数据类型的表或模式对象的操作

DEFAULT_COLLATION

使用此子句设置会话的默认排序规则。

  • 使用collation_name指定会话的默认排序规则。您可以指定任何有效命名的排序规则或伪排序规则的名称。此归类将成为有效的架构默认归类 。此排序规则分配给表，视图和物化视图，这些视图随后在会话期间在任何模式中创建。会话的默认排序规则不会传播到使用数据库链接连接到当前会话的任何远程会话。

  • 如果指定NONE ，则会话没有默认排序规则。在这种情况下，特定架构的默认排序规则将成为该架构的有效架构默认排序规则 。该默认排序规则分配给表，视图和物化视图，这些视图随后在会话期间在模式中创建。

在上述任一情况下，您可以覆盖有效的架构默认排序规则，并通过为表，物化视图或表格的CREATE或ALTER语句指定DEFAULT COLLATION子句，为特定表，物化视图或视图指定默认排序规则。视图。

有效的模式默认排序规则也会影响DDL语句CREATE FUNCTION ， CREATE PACKAGE ， CREATE PROCEDURE ， CREATE TRIGGER和CREATE TYPE 。有关这些语句的更多详细信息，请参阅Oracle Database PL / SQL语言参考 。

您可以使用以下语句查询会话的默认排序规则：

SELECT SYS_CONTEXT（'USERENV'，'SESSION_DEFAULT_COLLATION'）FROM DUAL;

仅当COMPATIBLE初始化参数设置为12.2或更高，并且MAX_STRING_SIZE初始化参数设置为EXTENDED ，才能指定SET DEFAULT_COLLATION子句。

会话参数和ALTER SESSION

以下参数仅为会话参数，不是初始化参数：

约束[S]

句法： 

CONSTRAINT [S] = {IMMEDIATE | DEFERRED | DEFAULT}

CONSTRAINT[S]参数确定何时强制执行由可延迟约束指定的条件。

  • IMMEDIATE表示在每个DML语句之后立即检查由可延迟约束指定的条件。此设置相当于在会话中的每个事务开始时发出SET CONSTRAINTS ALL IMMEDIATE语句。

  • DEFERRED表示在提交事务时检查由可延迟约束指定的条件。此设置相当于在会话中的每个事务开始时发出SET CONSTRAINTS ALL DEFERRED语句。

  • DEFAULT将每个事务开始时的所有约束恢复到其初始状态DEFERRED或IMMEDIATE 。

CURRENT_SCHEMA

句法： 

CURRENT_SCHEMA = 架构

CURRENT_SCHEMA参数将会话的当前架构更改为指定的架构。在会话期间对模式对象的后续非限定引用将解析为指定模式中的对象。该设置在会话期间或您发出另一个ALTER SESSION SET CURRENT_SCHEMA语句之前一直存在。

此设置提供了一种方便的方法，可以对当前用户以外的模式中的对象执行操作，而无需使用模式名称限定对象。此设置更改当前架构，但它不会更改会话用户或当前用户，也不会为会话用户提供会话的任何其他系统或对象权限。

ERROR_ON_OVERLAP_TIME

句法： 

ERROR_ON_OVERLAP_TIME = {TRUE |假}

ERROR_ON_OVERLAP_TIME参数确定Oracle数据库应如何处理模糊边界日期时间值 - 在此情况下，不清楚日期时间是标准时间还是夏令时。

  • 指定TRUE以返回模糊重叠时间戳的错误。

  • 指定FALSE以将模糊重叠时间戳默认为标准时间。这是默认值。

有关边界日期时间值的更多信息，请参阅“ 夏令时支持 ” 。

FLAGGER

句法： 

FLAGGER = {ENTRY |关闭}

FLAGGER参数指定FIPS标记（在联邦信息处理标准127-2中指定），当发出的SQL语句是SQL-92的入门级的扩展时，会导致生成错误消息（官方，ANSI X3.135） -1992，现在被SQL：2016取代的标准。FLAGGER仅是会话参数，不是初始化参数。

在会话中设置标记后，后续的ALTER SESSION SET FLAGGER语句将起作用，但会生成消息ORA-00097。这允许在不断开会话的情况下更改FIPS标记。OFF关闭标记。

INSTANCE

句法： 

INSTANCE = 整数

通过设置INSTANCE参数，您可以访问另一个实例，就像连接到自己的实例一样。INSTANCE只是一个会话参数，不是初始化参数。在Oracle Real Application Clusters（Oracle RAC）环境中，每个Oracle RAC实例都会保留磁盘空间的静态或动态所有权，以便根据此参数的设置获得最佳DML性能。

ISOLATION_LEVEL

句法： 

ISOLATION_LEVEL = {SERIALIZABLE | READ COMMITTED}

ISOLATION_LEVEL参数指定如何处理包含数据库修改的事务。ISOLATION_LEVEL只是一个会话参数，不是初始化参数。

  • SERIALIZABLE指示会话中的事务使用SQL标准中指定的可序列化事务隔离模式。如果可序列化事务尝试执行DML语句，该语句在可序列化事务开始时更新当前由另一个未提交事务更新的行，则DML语句将失败。可序列化的事务可以看到自己的更新。

  • READ COMMITTED表示会话中的事务将使用默认的Oracle数据库事务行为。如果事务包含需要由另一个事务持有的行锁的DML，则DML语句将等待直到释放行锁。

注意：

可序列化事务不适用于延迟段创建或间隔分区。尝试将数据插入到没有创建段的空表中，或者插入到没有段的间隔分区表的分区中会导致错误。

STANDBY_MAX_DATA_DELAY

句法： 

STANDBY_MAX_DATA_DELAY = { 整数 |没有 }

在Active Data Guard环境中，此会话参数可用于为非管理用户向处于实时查询模式的物理备用数据库发出的查询指定特定于会话的应用延迟容差（以秒为单位）。此功能允许将查询从主数据库安全地卸载到物理备用数据库，因为可以检测备用数据库是否已变得无法接受。

如果将STANDBY_MAX_DATA_DELAY设置为默认值NONE ，则无论该数据库上的应用延迟如何，都将执行发送到物理备用数据库的查询。

如果STANDBY_MAX_DATA_DELAY设置为非零值，则仅当应用延迟小于或等于STANDBY_MAX_DATA_DELAY才会执行发送到物理备用数据库的查询。否则，将返回ORA-3172错误以警告客户端应用延迟太大。

如果STANDBY_MAX_DATA_DELAY设置为0 ，则发送到物理备用数据库的查询将保证返回与在主数据库上发出查询完全相同的结果，除非备用数据库滞后于主数据库，在这种情况下是ORA-3172返回ORA-3172错误。

时区

句法： 

TIME_ZONE ='[+ | - ] hh：mi '| LOCAL | DBTIMEZONE | ' time_zone_region '

TIME_ZONE参数指定当前SQL会话的默认本地时区偏移或区域名称。TIME_ZONE仅是会话参数，不是初始化参数。要确定当前会话的时区，请查询内置函数SESSIONTIMEZONE （请参阅SESSIONTIMEZONE ）。

  • 指定格式掩码（ '[+|-] hh : mi ' ），表示UTC之前或之后的小时和分钟（协调世界时 - 以前的格林威治标准时间）。hh:mi的有效范围是-12：00到+14：00。

  • 指定LOCAL以将当前SQL会话的默认本地时区偏移量设置为在当前SQL会话启动时建立的原始默认本地时区偏移量。

  • 指定DBTIMEZONE以将当前会话时区设置为与为数据库时区设置的值匹配。如果指定此设置，则DBTIMEZONE函数将数据库时区作为UTC偏移或时区区域返回，具体取决于数据库时区的设置方式。

  • 指定有效的time_zone_region 。要查看有效时区区域名称列表，请查询V$TIMEZONE_NAMES动态性能视图的TZNAME列。如果指定此设置，则SESSIONTIMEZONE函数将返回区域名称。

注意：

夏令时功能需要时区区域名称。这些名称存储在两种类型的时区文件中：一个大文件和一个小文件。其中一个文件是默认文件，具体取决于您所使用的环境和Oracle数据库版本。有关时区文件和名称的更多信息，请参阅“ Oracle数据库全球化支持指南” 。

您还可以使用ORA_SDTZ环境变量设置默认客户端会话时区。有关此变量的更多信息，请参阅“ Oracle数据库全球化支持指南” 。

USE_PRIVATE_OUTLINES

句法： 

USE_PRIVATE_OUTLINES = {TRUE | FALSE |分类名称 }

USE_PRIVATE_OUTLINES参数允许您控制私有轮廓的使用。如果启用此参数并发出概述的SQL语句，优化程序将从会话专用区域检索大纲，而不是在启用USE_STORED_OUTLINES时使用的公共区域。如果会话专用区域中不存在大纲，则优化器将不使用大纲来编译语句。USE_PRIVATE_OUTLINES不是初始化参数。

  • TRUE导致优化器在编译请求时使用存储在DEFAULT类别中的私有轮廓。

  • FALSE指定优化程序不应使用存储的私有轮廓。这是默认值。如果启用了USE_STORED_OUTLINES ，则优化器将使用存储的公共轮廓。

  • category_name使优化器在编译请求时使用存储在category_name类别中的轮廓。

对USE_PRIVATE_OUTLINES的限制

如果启用了USE_STORED_OUTLINES则无法启用此参数。

USE_STORED_OUTLINES

注意：

存储的大纲已被弃用。它们仍然支持向后兼容性。但是，Oracle建议您使用SQL计划管理。有关SQL计划管理的详细信息，请参阅“ Oracle数据库SQL调优指南” 。

句法：

USE_STORED_OUTLINES = {TRUE | FALSE |分类名称 }

USE_STORED_OUTLINES参数确定优化程序是否将使用存储的公共轮廓来生成执行计划。USE_STORED_OUTLINES不是初始化参数。

  • TRUE使优化器在编译请求时使用存储在DEFAULT类别中的轮廓。

  • FALSE指定优化程序不应使用存储的轮廓。这是默认值。

  • category_name使优化器在编译请求时使用存储在category_name类别中的轮廓。

对USED_STORED_OUTLINES的限制

如果启用了USE_PRIVATE_OUTLINES则无法启用此参数。

例子

启用并行DML：示例

发出以下语句以为当前会话启用并行DML模式：

ALTER SESSION ENABLE PARALLEL DML;

强制分布式事务：示例

以下事务将员工记录插入由数据库链接remote标识的数据库上的employees表中，并从local标识的数据库上的employees表中删除员工记录：

ALTER SESSION ADVISE COMMIT; INSERT INTO员工@远程VALUES（8002，'Juan'，'Fernandez'，'juanf @ example.com'，NULL，TO_DATE（'04 -OCT-1992'，'DD-MON-YYYY'），'SA_CLERK'， 3000，NULL，121,20）; ALTER SESSION ADVISE ROLLBACK; DELETE FROM employees @ local WHERE employee_id = 8002;承诺;

此事务具有两个带有ADVISE子句的ALTER SESSION语句。如果事务变得有疑问，则通过第一个ALTER SESSION语句向remote发送建议' COMMIT '，并且通过第二个语句向local发送建议' ROLLBACK '。

关闭数据库链接：示例

此语句使用数据库链接更新local数据库上的jobs表，提交事务并显式关闭数据库链接：

UPDATE jobs @ local SET min_salary = 3000 WHERE job_id ='SH_CLERK';承诺; ALTER SESSION CLOSE DATABASE LINK local;

动态更改日期格式：示例

以下语句将会话的默认日期格式动态更改为'YYYY MM DD-HH24:MI:SS' ：

ALTER SESSION SET NLS_DATE_FORMAT ='YYYY MM DD HH24：MI：SS';

Oracle数据库使用新的默认日期格式：

SELECT TO_CHAR（SYSDATE）今天FROM DUAL;今天------------------- 2001 04 12 12:30:38

动态更改日期语言：示例

以下语句将日期格式元素的语言更改为法语：

ALTER SESSION SET NLS_DATE_LANGUAGE =法语; SELECT TO_CHAR（SYSDATE，'Day DD Month YYYY'）今天FROM DUAL;今天--------------------------- Jeudi 12 Avril 2001

更改ISO货币：示例

以下语句将ISO货币符号动态更改为美国领土的ISO货币符号：

ALTER SESSION SET NLS_ISO_CURRENCY = America; SELECT TO_CHAR（SUM（薪水），'C999G999D99'）员工总数;总计------------------ USD694,900.00

更改十进制字符和组分隔符：示例

以下语句将十进制字符动态更改为逗号（，），将组分隔符更改为句点（。）：

ALTER SESSION SET NLS_NUMERIC_CHARACTERS ='，。';

使用数字格式元素时，Oracle数据库会返回这些新字符：

ALTER SESSION SET NLS_CURRENCY ='FF'; SELECT TO_CHAR（SUM（薪水），'L999G999D99'）员工总数; TOTAL --------------------- FF694.900,00

更改NLS货币：示例

以下语句将本地货币符号动态更改为“ DM ”：

ALTER SESSION SET NLS_CURRENCY ='DM'; SELECT TO_CHAR（SUM（薪水），'L999G999D99'）员工总数; TOTAL --------------------- DM694.900,00

更改NLS语言：示例

以下语句将动态更改为法语，显示错误消息的语言：

ALTER SESSION SET NLS_LANGUAGE = FRENCH;会议修改。SELECT * FROM DMP; ORA-00942：表ou vindististante

更改语言排序顺序：示例

以下语句将语言排序顺序动态更改为西班牙语：

ALTER SESSION SET NLS_SORT = XSpanish;

Oracle数据库根据字符串在西班牙语语言排序顺序中的位置对字符值进行排序。

启用查询重写：示例

此语句在当前会话中为尚未显式禁用的所有物化视图启用查询重写：

ALTER SESSION SET QUERY_REWRITE_ENABLED = TRUE;
