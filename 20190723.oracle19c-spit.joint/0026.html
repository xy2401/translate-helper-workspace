1.2.5 Memory Guard简介

Memory Guard是一个Oracle Real Application Clusters（Oracle RAC）环境功能，用于监控集群节点，以防止由于内存不足而导致的节点压力。

默认启用，Memory Guard：

  • 每分钟一次分析过度提交的内存条件

  • 如果任何服务器存在风险，则会发出警报

  • 通过自动关闭服务器到新连接来保护应用程序

  • 在服务器上以事务方式停止所有CRS管理的服务

  • 一旦内存压力消退，重新打开服务器连接

由于过多的打开会话或失控的工作负载，企业数据库服务器可以使用所有可用内存。内存不足可能导致事务失败，或者在极端情况下，导致节点重新启动并导致应用程序资源可用性降低。

Memory Guard自动从Cluster Health Monitor收集每个节点内存的度量标准，以确定节点是否内存不足。如果内存不足，则Memory Guard会阻止创建新的数据库会话，从而允许现有工作负载完成并释放内存。当缓解内存压力时，会自动启动新会话。

分段LONG数据类型

现在，网关在代理和Oracle数据库之间分段管理数据，从而消除了大量内存和网络带宽需求。

早期版本的网关对LONG数据类型的支持有限。LONG是一种Oracle数据类型，可以存储最多2 GB的字符数据或原始数据（ LONG RAW ）。这些早期版本将LONG数据量限制为4 MB，因为它们将LONG数据视为单个数据。这会导致内存和网络带宽限制可以处理的数据大小。当前网关扩展了功能，以支持完整的2 GB异构LONG数据。

HS_LONG_PIECE_TRANSFER_SIZE异构服务初始化参数可用于设置已传输片段的大小。例如，考虑从异构源中检索2 GB的LONG数据。较小的一块需要较少的内存，但需要更多的往返才能检索所有数据。较大的一块需要较少的往返次数，但它需要更大的内存来在内部存储中间件。初始化参数可用于调整系统以获得最佳性能，即在往返和内存要求之间进行最佳权衡。如果未设置初始化参数，则系统使用64 KB作为默认块大小。

注意：

不要将此功能与客户端上LONG数据的零碎操作混淆。客户端上的零碎提取和插入操作与早期版本的网关一起工作，并且他们继续这样做。客户端的唯一区别是，早期版本的网关能够获取最多4 MB的LONG数据，现在他们可以检索2 GB的LONG数据。这是一项重大改进，因为4 MB仅占数据类型容量的0.2％。

11.4在Oracle数据库中安全使用Runtime.exec功能

本节适用于DBA和安全管理员，并提供在Oracle数据库中运行的Java应用程序中安全使用Java SE功能Runtime.exec指南。java.lang.自Java 9开始，Java SE库中的Java虚拟机（Java VM）支持的Runtime.exec方法跨越了新的操作系统（OS）进程，并在新进程中执行指定的命令和参数。如果存在SecurityManager （在数据库中运行的Java VM始终如此），则在新OS过程开始之前执行对相关路径名的文件执行权限的安全检查。如果您是DBA或安全管理员，则您有责任有选择地向有权运行服务器端OS命令的数据库用户授予相应的文件读取，写入和执行权限。此外，Oracle强烈建议使用dbms_java.set_runtime_exec_credentials过程来控制生成的命令的OS用户标识，如以下各节所述。

按设计， Runtime.exec和java.lang.的相关功能java.lang.ProcessBuilder和java.lang.Process类无法控制与新创建的进程关联的用户的标识。在大多数Java实现中，包括Java VM的默认行为，分叉进程使用父进程的标识运行，父进程是Oracle数据库中的Oracle OS用户。出于安全原因，建议运行Runtime.exec功能分叉的进程，并授予较少权限的操作系统标识。dbms_java.set_runtime_exec_credentials过程提供了将指定的数据库用户/模式绑定到特定OS帐户的机制。如果您是DBA，则应将发出Runtime.exec调用的数据库用户绑定到操作系统帐户，并尽可能降低功耗。以下调用将数据库用户/架构DBUSER与OS osuser帐户相关联：

dbms_java.set_runtime_exec_credentials（'DBUSER'，'osuser'，'ospass'）;

其结果是，操作系统进程催生运行Runtime.exec颁发命令DBUSER与身份运行osuser 。您必须是SYS用户才能使用set_runtime_exec_credentials过程。

手动数据锁定

Oracle数据库始终自动执行锁定，以确保数据并发性，数据完整性和语句级读取一致性。但是，您可以覆盖Oracle默认锁定机制。这在以下情况下非常有用：

  • 当您的应用程序在事务持续时间内需要一致的数据而不反映其他事务的更改时，您可以通过使用显式锁定，只读事务，可序列化事务或通过覆盖默认锁定来实现事务级读取一致性。

  • 当您的应用程序要求事务具有对资源的独占访问权限，以便事务不必等待其他事务完成时，您可以在事务持续期间显式锁定数据。

您可以在两个级别覆盖自动锁定：

  • 交易 。您可以使用以下SQL语句覆盖事务级锁定：

    • SET TRANSACTION ISOLATION LEVEL

    • LOCK TABLE

    • SELECT ......FOR UPDATE

    在事务提交或回滚后释放由这些语句获取的锁。

  • 会议 。会话可以使用ALTER SESSION SET ISOLATION LEVEL语句设置所需的事务隔离级别。

注意：

覆盖Oracle默认锁定时，数据库管理员或应用程序开发人员应确保数据完整性得到保证，数据并发性是可接受的，并且无法实现死锁，或者如果可能，应妥善处理死锁。有关这些条件的更多信息，请参阅Oracle数据库概念 。

关于Oracle数据库的默认SGA权限的更改

从Oracle Database 12c第2版（12.2.0.1）开始，默认情况下，读取和写入系统全局区域（SGA）的权限仅限于Oracle软件安装所有者。

在以前的版本中，Oracle安装所有者帐户和OSDBA组的成员都可以访问共享内存。Oracle Database 12c第2版（12.2）及更高版本中的更改将默认情况下限制为Oracle安装所有者帐户，这提供了比以前的配置更高的安全性。但是，此更改可能会阻止无权访问Oracle安装所有者帐户的DBA管理数据库。

Oracle数据库初始化参数ALLOW_GROUP_ACCESS_TO_SGA确定Oracle数据库安装所有者帐户（Oracle文档示例中的oracle ）是否是唯一可以读取和写入数据库系统全局区域（SGA）的用户，或者OSDBA组的成员是否可以读取SGA。在Oracle Database 12c第2版（12.2）和更高版本中，此参数的默认值为FALSE，因此只有Oracle数据库安装所有者具有对SGA的读写权限。默认情况下，将删除对SGA的组访问权限。此更改会影响所有Linux和UNIX平台。

如果OSDBA组的成员需要对SGA的读访问权限，则可以将初始化参数ALLOW_GROUP_ACCESS_TO_SGA设置从FALSE更改为TRUE。 Oracle强烈建议您接受将SGA访问权限限制为oracle用户帐户的默认权限。

4.5 Java会话状态的两层持续时间

在11g版本1（11.1）之前，Java会话状态是单层的，其中包括与运行Java相关的所有值，例如系统属性值和静态变量值，会话期间加载的类集等等。这种状态的持续时间用于从RDBMS会话中第一次调用Java方法开始，并且由于调用了java.lang. ，它过去一直持续到JVM退出java.lang.System.exit或类似的OracleRuntime方法，未捕获的异常，致命错误或RDBMS会话的结束。这需要修改Java代码，也无法在RDBMS调用结束之前完全完成会话终止。这使得无法在同一个调用中启动新的Java会话。

从11g版本1（11.1）开始，Java会话状态分为两层。一层具有更长的持续时间，并且它包含另一层的持续时间。较短层的持续时间与之前相同，也就是说，它在调用Java方法时启动，在JVM退出时结束。当第一次在RDBMS会话中调用Java方法时，较长层的持续时间开始。此会话将持续到RDBMS会话结束或通过调用函数dbms_java.endsession_and_related_state显式终止会话。通过在DBMS_JAVA包中添加以下两个PL / SQL函数来解决这个问题，该函数考虑了两种Java会话持续时间：

  • FUNCTION endsession RETURN VARCHAR2;

    此函数清除当前RDBMS会话中先前执行Java所剩余的任何Java会话状态。返回值是指示所采取的操作的消息。

  • 功能endsession_and_related_state RETURN VARCHAR2;

    此函数清除当前RDBMS会话中先前执行Java所剩余的任何Java会话状态以及与运行Java相关的所有支持数据，例如属性设置和输出规范。返回值是指示所采取的操作的消息。

与运行Java相关的大多数值都保留在较短的层中。对多个JVM调用有用的值已移至较长层。例如， dbms_java.set_property建立的系统属性值和输出重定向规范。

由于此新功能，更改为仅架构帐户的默认帐户将无法用于密码身份验证。此功能的好处是管理员不再需要定期轮换这些Oracle数据库提供的模式的密码。此功能还可以降低使用默认密码侵入这些帐户的攻击者的安全风险。

如果要在升级期间阻止将这些Oracle帐户设置为仅架构帐户，则必须在开始升级之前为该帐户设置有效的强密码，或者在升级后为这些帐户设置有效的强密码，或者在登录升级的Oracle数据库之前解锁帐户。

升级后，管理员还可以为仅架构帐户启用密码身份验证。但是，为了更好的安全性，Oracle建议您将这些帐户保留为仅模式帐户。

手动配置存储设备路径持久性

您可以通过创建规则文件来维护存储文件路径持久性。

默认情况下，设备文件命名方案udev在服务器启动时动态创建设备文件名，并将其所有权分配给root。如果udev应用默认设置，则它会更改磁盘的Oracle设备文件名和所有者，从而使服务器重新启动时无法访问磁盘。例如，用户grid拥有的名为/dev/sdd的设备上的表决磁盘可能位于重新启动服务器后由root拥有的名为/dev/sdf的设备上。

如果使用Oracle ASMFD，则无需在udev确保权限和设备路径持久性。

如果不使用Oracle ASMFD，则必须创建自定义规则文件。Linux供应商定制他们的udev配置并使用不同的顺序来读取规则文件。例如，在启动udev某些Linux发行版中，它会依次执行规则文件中定义的规则（配置指令）。这些文件位于路径/etc/udev/rules.d/ 。规则文件以词法顺序读取。例如，文件10-wacom.rules中的规则在规则文件90-ib.rules中的规则之前被解析和执行。

在udev规则文件中指定设备信息时，请确保在显示的顺序中的所有其他特征之前指定了OWNER，GROUP和MODE。例如，要在UDEV行上包含特征ACTION，请在OWNER，GROUP和MODE之后指定ACTION。

规则文件描述相同设备的位置，在受支持的Linux内核版本上，最后读取的文件是应用的文件。

升级后或其他数据库配置更改后，Oracle强烈建议您在Oracle数据库上运行代表性工作负载后重新收集固定对象统计信息。

固定对象是X$表及其索引。V$性能视图通过X$表定义。收集固定对象统计信息对于数据库性能很有价值，因为这些统计信息可帮助优化器生成良好的执行计划，从而提高数据库性能。未能获得具有代表性的统计数据可能会导致执行计划不理想，从而导致严重的性能问题。

SQL> execute dbms_stats.gather_fixed_objects_stats;

关于Oracle数据库的默认SGA权限的更改

从Oracle Database 12c第2版（12.2.0.1）开始，默认情况下，读取和写入系统全局区域（SGA）的权限仅限于Oracle软件安装所有者。

在以前的版本中，Oracle安装所有者帐户和OSDBA组的成员都可以访问共享内存。Oracle Database 12c第2版（12.2）及更高版本中的更改将默认情况下限制为Oracle安装所有者帐户，这提供了比以前的配置更高的安全性。但是，此更改可能会阻止无权访问Oracle安装所有者帐户的DBA管理数据库。

Oracle数据库初始化参数ALLOW_GROUP_ACCESS_TO_SGA确定Oracle数据库安装所有者帐户（Oracle文档示例中的oracle ）是否是唯一可以读取和写入数据库系统全局区域（SGA）的用户，或者OSDBA组的成员是否可以读取SGA。在Oracle Database 12c第2版（12.2）和更高版本中，此参数的默认值为FALSE，因此只有Oracle数据库安装所有者具有对SGA的读写权限。默认情况下，将删除对SGA的组访问权限。此更改会影响所有Linux和UNIX平台。

如果OSDBA组的成员需要对SGA的读访问权限，则可以将初始化参数ALLOW_GROUP_ACCESS_TO_SGA设置从FALSE更改为TRUE。 Oracle强烈建议您接受将SGA访问权限限制为oracle用户帐户的默认权限。

2.6服务器上的用户界面

Oracle数据库在服务器上提供所有核心Java类库，包括与用户界面的表示相关的库。但是，在服务器上运行的代码尝试在服务器上实现或显示用户界面是不合适的。不应期望也不允许在Oracle JVM环境中运行应用程序的用户与运行Oracle数据库的服务器的显示和输入硬件交互或依赖它们。

为了解决不支持显示器，键盘或鼠标的平台上的兼容性问题，Java 1.4概述了无头抽象窗口工具包（AWT）支持。Headless AWT API引入了一个新的public运行时异常类java.awt.HeadlessException 。如果平台不支持显示，则Applet类的构造函数，所有重量级组件以及依赖于本机显示设备的Toolkit和GraphicsEnvironment类中的许多方法都会更改为抛出HeadlessException 。在Oracle数据库中，仅在客户端应用程序上支持用户界面。因此，Oracle JVM是一个无头平台，如果调用这些方法，则抛出HeadlessException 。

在Headless AWT中，允许大多数不涉及访问底层本机显示或输入设备的AWT计算。实际上，Headless AWT非常强大，因为它为程序员提供了对字体，成像，打印，颜色和ICC操作的访问。例如，在Oracle JVM中运行的应用程序可以解析，操作和写出映像，只要它们不尝试在服务器上物理显示它。标准JVM实现可以在Headless模式下启动，方法是提供-Djava.awt.headless=true属性，并使用与Oracle JVM相同的Headless AWT限制运行。Oracle JVM完全符合关于Headless AWT的Java Compatibility Kit（JCK）。

也可以看看：

http://www.oracle.com/technetwork/articles/javase/headless-136834.html

Oracle JVM采用类似的方法来提供声音支持。为了进行声音播放或录制，不允许Oracle JVM中的应用程序访问底层声音系统。相反，系统声音资源似乎以与尝试访问资源的方法的声音API规范一致的方式不可用。例如， javax.sound.midi.MidiSystem试图访问底层系统资源的声音抛出MidiUnavailableException检查异常信号，系统不可用。但是，与Headless AWT支持类似，Oracle数据库支持允许声音文件操作的API，不受原生声音设备的影响。Oracle JVM在实现声音API时也完全符合JCK。

在源数据库和重复数据库中为数据库文件使用相同的名称

必须满足某些条件才能对源和重复数据库中的文件使用相同的名称。

最简单的复制策略是将重复数据库配置为使用与源数据库相同的目录结构和文件名。只有在复制到远程主机时，才能使用相同的目录结构和名称。

使用相同的目录结构和文件名意味着您的环境满足以下要求：

  • 如果源数据库使用ASM磁盘组，则重复数据库必须使用具有相同名称的ASM磁盘组。

  • 如果源数据库文件是Oracle Managed Files，则辅助实例必须将DB_CREATE_FILE_DEST参数设置为与源数据库相同的目录位置。虽然源和目标主机上的目录相同，但Oracle数据库会选择重复文件的相对名称。

  • 如果源数据库中的数据库文件的名称包含路径，则此路径名在重复数据库中必须相同。

  • 对于Oracle Real Application Clusters（RAC）环境，请对源数据库和目标数据库的ORACLE_SID参数使用相同的值。

按建议配置环境时，不需要其他配置来命名重复文件。

挖掘模型是执行数据挖掘的数据库模式对象。

关于升级Oracle OLAP数据安全策略

从Oracle Database 12 c开始，安全策略的存储方式不同。如果现有Oracle数据库是11 g版本11.2，则删除安全角色。

在Oracle数据库12c中，甲骨文OLAP使用Oracle真正应用安全（奥拉斯）来存储OLAP数据安全策略，而不是可扩展的数据安全（XDS），它在Oracle数据库11克版本中使用。

当您升级从11版克至新的Oracle数据库版本的Oracle数据库，任何XDS数据安全策略被自动转换为奥拉斯。

注意：

11 g版 Oracle数据库实例中定义的数据安全角色不会自动转换为ORAS。在升级Oracle数据库版本11 克至当前的Oracle数据库版本中，您必须删除11克数据库中定义的任何数据的安全性角色。升级后，您可以使用新版本的Oracle Database Analytic Workspace Manager再次定义数据安全角色。

如果在不删除11g数据安全角色的情况下升级11g数据库，则在以后的Oracle数据库版本中，包含数据安全角色的任何数据安全策略都将无效。

启用IPMI的要求

您必须配置以下硬件和软件才能使用IPMI管理群集节点：

  • 每个集群成员节点都需要一个底板管理控制器（BMC），其运行的固件与IPMI 1.5或更高版本兼容，它支持LAN上的IPMI，并配置为使用LAN进行远程控制。

  • 群集需要IPMI的管理网络。这可以是共享网络，但Oracle建议您配置专用网络。

  • BMC使用的每个集群成员节点的端口必须连接到IPMI管理网络。

  • 每个集群成员必须连接到管理网络。

  • 某些服务器平台在断电时将其网络接口置于省电模式。在这种情况下，它们可能仅在较低的链路速度下运行（例如，100 MB，而不是1 GB）。对于这些平台，BMC所连接的网络交换机端口必须能够自动协商到较低的速度，否则IPMI将无法正常工作。

注意：

IPMI通过基板管理控制器（BMC）的网络接口在物理硬件平台上运行。根据您的系统配置，IPMI启动的服务器重启可能会影响服务器上托管的所有虚拟环境。有关详细信息，请与硬件和OS供应商联系。

关于自动内存管理安装选项

确定是否要在安装期间配置自动内存管理。

在典型安装期间，使用Oracle Database Configuration Assistant（DBCA）创建数据库，并启用自动内存管理。如果选择“ 高级安装”，则可以手动指定内存分配，也可以启用自动内存管理。

如果数据库实例的总物理内存大于4 GB，则在数据库安装和创建期间无法选择Oracle自动内存管理选项。而是使用自动共享内存管理。自动共享内存管理根据需要自动在各种组件之间分配可用内存，从而允许系统最大限度地使用所有可用的SGA内存。

通过自动内存管理，Oracle数据库实例可以自动管理和调整内存。通过自动内存管理，您可以选择内存目标，实例会自动在系统全局区域（SGA）和实例程序全局区域（实例PGA）之间分配内存。随着内存需求的变化，实例动态地在SGA和实例PGA之间重新分配内存。

您可以在数据库安装期间或之后启用自动内存管理。安装后启用自动内存管理涉及关闭并重新启动数据库。

注意：

默认情况下，在具有4 GB以上RAM的节点上执行典型安装时，将禁用自动内存管理。
