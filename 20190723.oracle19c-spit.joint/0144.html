acfsutil加入

目的

加密Oracle ACFS文件系统，目录或单个文件。

语法和说明

acfsutil encr on -h acfsutil encr on -m mount_point [-a {AES} -k {128 | 192 | 256}] [[-r] path [ path ...]]

acfsutil encr on -h显示帮助文本并退出。

表16-84包含acfsutil encr on命令可用的选项。

表16-84 acfsutil encr on命令的选项

选项 描述

-m mount_point

指定安装文件系统的目录。

-a algorithm

指定目录或文件的加密算法类型。高级加密标准（AES）是此版本支持的唯一加密算法。

-k key_length

指定目录或文件的加密密钥长度。

-r

在path标识的现有目录文件夹下递归指定加密。

路径

指定目录的绝对路径或相对路径。允许多个路径值。

此命令无法在受域保护的文件上运行。

-a和-k的缺省值由运行acfsutil encr set时指定的卷参数确定。要在卷级别设置密钥长度，请使用acfsutil encr set命令。

只有管理员才能在Oracle ACFS文件系统上运行此命令（ -m选项没有指定path ）。如果指定了-m选项而没有path ，则挂载点下的所有文件都将被加密。

path选项可以指定读写快照中文件或目录的路径。如果在根目录中使用该命令指定了-r选项，则该命令不会横向显示下的快照.ACFS目录。如果在文件系统级别指定了加密操作，则该操作不会处理快照中的文件和目录.ACFS/snaps/目录。

使用-r选项运行acfsutil encr on ，该命令以递归方式加密指定的目录，但不在文件系统级别启用加密。

只有具有root或系统管理员权限的用户才能运行此命令以在文件系统上启用加密。文件所有者还可以运行此命令以对目录或文件启用加密。

例子

目的

-f [ -r ] path ......

为指定的路径生成新的文件加密密钥，然后使用新密钥加密数据。

如果指定了-r，则在path下递归执行重定密钥操作。

path指定目录的绝对路径或相对路径。允许多个路径值。

-v

为指定的安装点生成新的卷加密密钥（VEK），然后使用新密钥加密文件系统中的所有文件加密密钥。提示钱包密码，因为必须访问钱包才能存储新的VEK。

-k key_length

指定path指定的目录或文件的密钥长度。

此命令无法在受安全领域保护的文件上运行。

-a和-k的缺省值由运行acfsutil encr set时指定的卷参数确定。

path选项可以指定读写快照中文件或目录的路径。如果在根目录中使用该命令指定了-r选项，则该命令不会横向显示下的快照.ACFS目录。如果在文件系统级别指定了重定密钥操作，则该操作不会处理快照中的文件和目录.ACFS/snaps/目录。

如果Oracle Key Vault是文件系统的密钥库，则在使用-v选项生成新卷密钥时，必须设置Oracle Key Vault主环境变量（ OKV_HOME ）。如果客户端配置为使用Oracle Key Vault的密码，则必须在提示时输入相同的密码。

acfsutil encr set

目的

设置或更改Oracle ACFS文件系统的加密参数。

语法和说明

acfsutil encr set -h acfsutil encr set [[-a {AES} -k {128 | 192 | 256}] [-e] | -u] -m mount_point

acfsutil encr set -h显示帮助文本并退出。

表16-86包含acfsutil encr set命令可用的选项。

表16-86 acfsutil encr set命令的选项

选项 描述

-a algorithm

指定算法。高级加密标准（ AES ）是此版本支持的默认值和唯一加密。如果指定了-k则必须指定算法。

-k { 128 | 192 | 256 }

指定密钥长度。密钥长度设置为音量级别。默认值为192 。如果指定了-a则必须指定。

-e

指定使用Oracle Key Vault作为密钥库。

-u

退出加密。解密文件系统中的所有加密文件，并将文件系统恢复到在文件系统上运行acfsutil encr set之前的状态。如果正在使用安全性，则只能在安全性退出后运行此命令。要删除安全性，请参阅“ acfsutil sec prepare ” 。

-m mount_point

指定安装文件系统的目录。

在运行acfsutil encr set命令之前，必须先运行acfsutil encr init命令。

acfsutil encr set命令配置文件系统的加密参数，透明地生成卷加密密钥，并将生成的密钥存储在先前使用acfsutil encr init命令配置的密钥存储区中。

如果在群集上初始化审核，则此命令还会在文件系统上启用Oracle ACFS加密审核源。启用此审计源时执行的操作与直接运行acfsutil audit enable命令时执行的操作相同。有关更多信息，请参阅“ acfsutil audit enable ” 。

另外， acfsutil encr set创建mount_point /.包含收集审计和诊断数据的日志文件（ encr- hostname _fsid.log ）的Security /encryption/logs/目录。

存储密钥时的密码要求取决于加密密钥存储的配置方式。如果使用acfsutil encr init指定了-p ，则运行此命令需要密码。

在使用-e选项将Oracle Key Vault指定为密钥存储区之前，必须先配置Oracle Key Vault。如果要选择Oracle Key Vault作为文件系统的密钥存储区，则在使用-e选项运行命令时，必须设置Oracle Key Vault主环境变量（ OKV_HOME ）。如果客户端配置为使用Oracle Key Vault的密码，则必须在提示时输入相同的密码。

也可以看看：

有关配置Oracle Key Vault的信息，请参见“Oracle Key Vault管理员指南”

如果文件系统中存在任何快照，则不允许使用acfsutil encr set –u命令。

只有具有root或系统管理员权限的用户才能运行acfsutil encr set命令。

21.1 SQL / JSON生成函数概述

您可以使用SQL / JSON函数json_object ， json_array ， json_objectagg和json_arrayagg从数据库中的非JSON数据构造JSON数据。JSON数据作为SQL值返回。

这些生成函数可以直接从SQL查询中轻松构造JSON数据。它们允许将非JSON数据表示为JSON对象和JSON数组。您可以通过嵌套对这些函数的调用来生成复杂的分层JSON文档。嵌套子查询可以生成表示一对多关系的JSON集合。脚1

从非JSON数据构造JSON数据的最佳方法

使用SQL / JSON生成函数的替代方法通常容易出错或效率低下。

  • 使用字符串连接生成JSON文档很容易出错。特别是，关于何时以及如何转义特殊字符（例如双引号（ " ）），必须遵守许多复杂的规则。很容易忽视或误解这些规则，这可能导致生成不正确的JSON数据。

  • 从数据库读取非JSON结果集并使用客户端应用程序代码生成JSON数据通常效率很低，特别是由于网络开销。当将一对多关系表示为JSON数据时，通常需要多个SELECT操作来收集所需的所有非JSON数据。如果要生成的文档代表多个级别的一对多关系，那么这种技术可能非常昂贵。

SQL / JSON生成函数不会遇到这样的问题;它们是为从非JSON数据库数据构造JSON数据而设计的。

  • 它们总是构造格式良好的JSON文档。

  • 通过将SQL子查询与这些函数一起使用，您可以使用单个SQL语句生成一整套JSON文档，从而可以优化生成操作。

  • 因为只将生成的文档返回给客户端，所以网络开销最小化：每个文档最多生成一次往返。

SQL / JSON生成函数

  • 函数json_object和json_array构造一个JSON对象或数组。在最简单的情况下， json_object将SQL名称 - 值对作为参数， json_array将SQL值作为参数。

  • 函数json_objectagg和json_arrayagg是聚合 SQL函数。它们分别将分组的SQL查询的行中包含的信息转换为JSON对象和数组。参数的评估分别确定对象成员和数组元素的数量;也就是说，结果的大小反映了当前查询的数据。

    对于json_objectagg和json_arrayagg ，未指定对象成员和数组元素的顺序。对于json_arrayagg ，您可以在json_arrayagg调用中使用ORDER BY子句来控制数组元素顺序。

SQL / JSON生成函数返回的结果

默认情况下，生成的JSON数据作为SQL VARCHAR2值从生成函数返回。您可以使用可选的RETURNING子句指定VARCHAR2大小或指定CLOB或BLOB返回值。当BLOB是返回类型时，字符集是AL32UTF8。

处理SQL / JSON生成函数的输入值

SQL / JSON生成函数将SQL值作为输入，并将其作为返回的SQL值内的JSON值呈现。输入值如何呈现为JSON取决于它们的SQL数据类型。

SQL / JSON生成函数的可选行为

您可以选择指定SQL NULL -handling子句， RETURNING子句和关键字STRICT 。

  • NULL -handling子句 - 确定如何处理由输入求值产生的SQL NULL值。

    • NULL ON NULL - 输出SQL NULL值转换为JSON null以进行输出。这是json_object和json_objectagg的默认行为。

    • ABSENT ON NULL - 输入SQL NULL值不会导致相应的输出。这是json_array和json_arrayagg的默认行为。

  • RETURNING子句 - 用于函数返回值的SQL数据类型。默认值为VARCHAR2(4000) 。

  • STRICT关键字 - 如果存在，则检查返回的JSON数据，以确保其格式正确。如果存在STRICT且返回的数据格式不正确，则会引发错误。

21.2 SQL / JSON生成函数的输入值处理

SQL / JSON生成函数将SQL值作为输入，并将其作为返回的SQL值内的JSON值呈现。输入值如何呈现为JSON取决于它们的SQL数据类型。

默认情况下，SQL NUMBER值在输出中呈现为JSON编号，SQL VARCHAR2值呈现为JSON字符串，依此类推。例如，默认情况下， VARCHAR2值'{}'呈现为JSON字符串"{}" 。

在某些情况下，您知道或期望输入值实际上已经表示JSON数据，并且您希望将此信息传递给生成函数，以便保持值不变。例如，如果输入为'{}'那么您可能希望将其解释（呈现）为空JSON 对象 {} ，而不是JSON 字符串 "{}" 。

您可以通过在输入表达式之后添加关键字FORMAT JSON来声明由此产生的值被视为已经表示JSON数据。同样，您可以将带有关键字AS JSON SQL函数treat应用于生成函数输入值 - 效果与使用FORMAT JSON相同。

在许多情况下，Oracle可以自动确定输入值实际上是JSON数据，在这种情况下，输入被视为后面跟着显式的FORMAT JSON声明。例如，如果值表达式本身是对SQL / JSON生成函数或函数json_query的调用，则就是这种情况。

如果没有为给定的输入值指定FORMAT JSON ，并且Oracle 无法确定该值是JSON数据，则假定它是普通（非JSON）SQL数据。在这种情况下，它被序列化如下（任何其他SQL值引发错误）：

  • 用户定义的SQL对象类型的实例呈现为JSON对象，其字段名称取自对象属性名称，其字段值取自对象属性值（递归应用JSON生成）。

  • SQL集合类型的实例呈现为JSON数组，其元素值取自集合元素值（递归应用JSON生成）。

  • VARCHAR2 ， CLOB或NVARCHAR值用双引号（ " ）括起来，并在必要时转义字符以符合JSON标准。

  • 数值（ NUMBER ， BINARY_DOUBLE或BINARY_FLOAT ）呈现为JSON编号。（没有引用。）

    正数和负数无穷大的数值以及作为数字运算的未定义结果的值（“非数字”或NaN ）不能表示为JSON数字。它们分别呈现为JSON字符串： "Inf" ， "-Inf" "Inf"和"Nan" 。

  • RAW或BLOB值呈现为十六进制JSON字符串（带双引号， " ）。

  • 与时间相关的值（ DATE ， TIMESTAMP ， TIMESTAMP WITH TIME ZONE ， TIMESTAMP WITH LOCAL TIME ZONE ， INTERVAL YEAR TO MONTH或INTERVAL DAY TO SECOND ）以支持的ISO 8601格式呈现，结果用双引号括起来（ " ）。

  • BOOLEAN PL / SQL值呈现为JSON true或false 。（没有引用。）

  • 无论NULL数据类型如何， NULL值都呈现为JSON null 。（没有引用。）

注意：

对于数据类型CLOB和BLOB ，空实例与NULL区分开来，并呈现为空JSON字符串（ "" ）。但是，对于数据类型VARCHAR2 ， NVARCHAR2 ，和RAW ，的Oracle SQL把一个空（零长度）值作为NULL ，所以不要指望这样的值，以呈现为一个JSON字符串。

21.3 SQL / JSON函数JSON_OBJECT

SQL / JSON函数json_object从关系（SQL）数据构造JSON对象。

它接受任意数量的参数，每个参数可以是以下任何一个：

  • 显式字段名称 - 值对。示例： answer : 42 。

  • 表名或别名或视图名称，后跟一个圆点和一个星号通配符（ .* ）。示例： t1.* 。（名称或别名也可以使用数据库模式名称作为前缀，如myschema.t1.* 。）

  • 关系列名称，可能以表名或别名开头，或者后跟点（ . ）的视图名称。示例： t1.address 。

或者，它接受单个参数，即星号通配符（ * ）或用户定义的SQL对象类型的单个实例。

描述星号通配符使用的另一种方法是它遵循SQL SELECT列表中允许使用通配符的方式。

名称 - 值对参数指定生成的JSON对象的对象成员（除非值表达式求值为SQL NULL且应用了ABSENT ON NULL子句）。名称和值是SQL表达式。名称表达式必须求值为SQL 字符串 。值表达式必须求值为可以呈现为JSON值的SQL值。名称和值表达式通过关键字分离VALUE或冒号（ : ）。

注意：

某些客户端驱动程序可能会在将查询发送到数据库之前尝试扫描查询文本并识别绑定变量。在某些情况下， json_object冒号作为名称 - 值分隔符可能会被误解为引入绑定变量。您可以使用关键字VALUE作为分隔符来避免此问题（ 'Name' VALUE Diderot ），或者您可以简单地将对的值部分括在括号中： 'Name':(Diderot) 。

如果使用列名作为参数，则对于给定的数据行，column-name参数指定的对象成员将列名称作为其字段名称，将列值作为字段值。无论是否引用，您提供的列名都将区分大小写 。例如，如果使用Email作为列名参数，则EMAIL列中的数据用于生成字段名称为Email对象成员。

您可以使用星号（ * ） 通配符作为快捷方式，以显式指定给定表或视图的所有列以生成对象成员。生成的对象字段名称是大写的列名称。您可以将通配符与表，视图或表别名一起使用。

您可以传递用户定义的SQL对象类型的单个实例，而不是传递用于定义单个JSON对象成员的SQL表达式。这将生成一个JSON对象，其字段名称取自对象属性名称，其字段值取自对象属性值（递归应用JSON生成）。

配置网关初始化参数文件

  2. 自定义初始化参数文件

选择网关的系统标识符

该网关系统标识符 （SID）是标识网关实例的字母数字字符串。对于要访问的每个Teradata数据库，您需要一个网关实例，因此需要一个网关SID。SID用作初始化参数文件的文件名的一部分。默认SID是 dg4tera 。

您可以定义网关SID，但使用默认的dg4tera更容易，因为您不需要更改初始化参数文件名。但是，如果要访问两个Teradata数据库，则需要两个网关SID，一个用于网关的每个实例。如果您只有一个Teradata数据库并且希望有时使用一组网关参数设置来访问它，而其他时候使用不同的网关参数设置，那么您将需要多个网关SID用于单个Teradata数据库。

自定义初始化参数文件

启动网关时，初始化参数文件必须可用。在安装过程中，将创建以下默认初始化参数文件：

 ORACLE_HOME \ dg4tera \ admin \ initdg4tera.ora

其中ORACLE_HOME是安装网关的目录。

此初始化文件用于默认网关SID。如果未使用dg4tera作为网关SID，则必须使用在上一步选择网关的系统标识符中选择的SID重命名初始化参数文件。此默认初始化参数文件足以启动网关，验证安装是否成功以及运行演示脚本。

可以使用许多初始化参数来修改网关行为。有关可以设置的初始化参数的完整列表，请参阅初始化参数 。对初始化参数所做的更改仅在下一个网关会话中生效。最重要的参数是HS_FDS_CONNECT_INFO ，它描述了与非Oracle系统的连接。

默认初始化参数文件已有此参数的条目。HS_FDS_CONNECT_INFO的语法如下：

HS_FDS_CONNECT_INFO = ip_address ： port_number [/ database_name ]

是Teradata数据库服务器的端口号。

是Teradata数据库名称。database_name变量是可选的

  • Oracle Net Listener侦听的Oracle Net地址列表

创建数据库链接

连接到Oracle数据库的任何Oracle客户端都可以通过网关访问Teradata数据。Oracle客户端和Oracle数据库可以驻留在不同的计算机上。网关仅接受来自Oracle数据库的连接。

首次在Oracle会话中使用时，通过数据库链接建立与网关的连接。在此上下文中，连接是指Oracle数据库与网关之间的连接。在Oracle会话结束之前，连接将保持建立状态。另一个会话或用户可以访问相同的数据库链接，并获得与网关和Teradata数据库的不同连接。

数据库链接在网关会话期间处于活动状态。如果要在会话期间关闭数据库链接，可以使用ALTER SESSION语句执行此操作。

要访问Teradata服务器，您必须创建数据库链接。公共数据库链接是最常见的数据库链接。

注意：

    • HS_FDS_RECOVERY_ACCOUNT为您设置用于恢复的Teradata用户帐户的用户名。

注意：

通过使用HS_FDS_TRANSACTION_LOG初始化参数指定其他名称，可以使用除HS_TRANSACTION_LOG之外的日志表的其他名称。

1.1 Oracle Data Guard和Broker概述

Oracle Data Guard可确保企业数据的高可用性，数据保护和灾难恢复。Oracle Data Guard提供了一整套服务，可以创建，维护，管理和监视一个或多个备用数据库，从而使生产Oracle数据库能够在灾难和数据损坏中幸免于难。Oracle Data Guard将这些备用数据库维护为主数据库的事务一致性副本。如果主数据库由于计划内或计划外中断而变得不可用，则Oracle Data Guard允许您将任何备用数据库切换到生产角色，从而最大程度地减少与中断相关的停机时间。Oracle Data Guard可与传统的备份，恢复和集群技术一起使用，也可与闪回数据库功能一起使用，以提供高级别的数据保护和数据可用性。

Oracle Data Guard和Oracle RAC One Node

从Oracle Database 11g第2版（11.2.0.2）开始，Oracle Data Guard和Oracle Data Guard代理与Oracle Real Application Clusters One Node（Oracle RAC One Node）完全集成。

1.1.1 Oracle Data Guard配置和代理配置

Oracle Data Guard配置包含一个主数据库以及备用数据库，远程同步实例和直接从主数据库接收重做的零数据丢失恢复设备的组合。 Oracle Data Guard配置的成员由Oracle Net连接，可能在地理上分散。只要成员可以相互通信，对成员的位置没有限制。

Oracle Data Guard代理在逻辑上将这些成员分组为代理配置 ，允许代理作为集成单元一起管理和监视它们。您可以使用Oracle Enterprise Manager云控制（云控制）或Oracle Data Guard命令行界面管理代理配置。

1.1.2 Oracle Data Guard Broker

Oracle Data Guard代理是一个分布式管理框架，可自动化和集中化Oracle Data Guard配置的创建，维护和监视。

以下列表描述了代理自动化和简化的一些操作：

  • 创建包含一个主数据库和零个或多个备用数据库（物理，逻辑或快照），远程同步实例和零数据丢失恢复设备以及任何关联的重做传输服务和日志应用服务的Oracle Data Guard配置。数据库可以是单实例，Oracle Real Application Clusters（Oracle RAC）或Oracle RAC One Node数据库。

  • 将备用数据库（物理，逻辑或快照），远程同步实例和零数据丢失恢复设备添加到现有Data Guard配置。

  • 管理配置的保护模式。

  • 使用单个命令调用切换或故障转移，以启动和控制配置中所有数据库的复杂角色更改。

  • 配置故障转移是在丢失主数据库时自动发生的，无需人工干预即可提高可用性。

  • 监控整个配置的状态，捕获诊断信息，报告统计信息（如重做应用率和重做生成率），并使用集中监控，测试和性能工具快速检测问题。

  • 评估数据库之间是否正确配置了网络。（请参阅有效的网络配置 。）

1.2 Oracle Data Guard Broker的优点

代理的接口提高了可用性，并集中管理和监视Oracle Data Guard配置。

该代理作为Oracle数据库企业版和个人版的一项功能提供，还与Oracle数据库和云控制集成。这些代理属性带来以下好处：

防灾：

通过自动执行配置和监视Oracle Data Guard配置所需的许多手动任务，代理可增强Oracle Data Guard中固有的高可用性，数据保护和灾难保护功能。可以通过客户端访问Oracle Data Guard配置中的任何系统，从而消除任何单点故障。如果主数据库发生故障，代理会自动执行任何一个备用数据库的进程以替换主数据库并接管生产处理。Oracle Data Guard提供的数据库可用性使您可以更轻松地保护数据。

Oracle Real Application Clusters（Oracle RAC）数据库具有更高的可用性和可扩展性：

虽然Oracle Data Guard代理通过维护主数据库的事务一致性副本来增强灾难保护，但Oracle Data Guard配置了Oracle高可用性解决方案（如Oracle Real Application Clusters（Oracle RAC）数据库），进一步增强了任何给定副本的可用性和可伸缩性那个数据库。Oracle RAC数据库的站点内高可用性补充了Oracle Data Guard代理提供的站点间保护。

例如，如果Oracle RAC主数据库的一个实例意外失败，Oracle Clusterware会尝试恢复失败的实例并保持主数据库可用。从Oracle Data Guard角度来看，只要至少一个集群数据库实例继续将重做数据传输到备用数据库，主数据库就仍然可用。如果Oracle Clusterware无法恢复失败的实例，则Oracle RAC数据库将继续使用一个活动较少的实例自动运行。如果主数据库的最后一个实例失败，则Oracle RAC数据库提供了一种故障转移到指定备用数据库的方法。如果主数据库的最后一个实例失败，并且启用了快速启动故障转移，则代理可以通过自动故障转移到预定的备用数据库来继续提供高可用性。

该代理与Oracle Clusterware集成，因此可以平滑无缝地进行数据库角色更改。这在计划的角色切换的情况下尤其明显（例如，当前一个主数据库承担备用角色时，指示物理备用数据库接管主角色）。代理和Oracle Clusterware一起工作以暂时中止主数据库上的服务可用性，完成两个数据库的实际角色更改，在此期间Oracle Clusterware与代理一起工作，以便在旧主数据库上根据需要正确地重新启动实例，然后启动服务在新的主数据库上定义。代理管理基础Oracle Data Guard配置及其数据库角色，而Oracle Clusterware管理依赖于这些角色的服务可用性。依赖Oracle Clusterware管理服务可用性的应用程序只会在Oracle Data Guard配置中发生角色更改时才会看到临时的服务暂停。

请注意，虽然Oracle Clusterware有助于维护Oracle RAC数据库的各个实例的可用性，但代理会协调在多个地理位置分散的位置维护数据库的一个或多个物理或逻辑副本的操作，以提供灾难保护。代理和Oracle Clusterware共同为Oracle的高可用性架构提供了坚实的基础。

支持Oracle RAC One节点：

从Oracle Database 11g第2版（11.2.0.2）开始，Oracle Data Guard和Oracle Data Guard代理与Oracle Real Application Clusters One Node（Oracle RAC One Node）完全集成。此支持通过利用Oracle RAC配置中提到的许多Oracle Clusterware优势，在本地站点提供更高级别的可用性。

自动创建Oracle Data Guard配置：

代理可帮助您逻辑定义和创建Oracle Data Guard配置。代理使用Oracle Net Services自动在Oracle Data Guard配置的成员之间进行通信。成员可以是本地的或远程的，通过LAN连接或通过WAN在地理上分散。

Cloud Control提供了一个向导，可自动执行创建代理配置所涉及的复杂任务，包括：

  • 添加现有备用数据库或从通过Cloud Control获取的现有备份创建的新备用数据库

  • 配置备用控制文件，服务器参数文件和数据文件

  • 初始化与备用数据库的通信

  • 创建备用重做日志文件

  • 如果计划使用快速启动故障转移，请启用闪回数据库

虽然DGMGRL无法自动创建新备用数据库，但您可以使用DGMGRL命令配置和监视现有备用数据库，包括使用Cloud Control创建的备用数据库。

轻松配置其他备用数据库：

创建Oracle Data Guard配置后，可以为每个Oracle Data Guard配置添加新的或现有的备用数据库。Cloud Control提供“添加备用数据库”向导，指导您完成添加更多数据库的过程。

简化，集中和扩展管理：

您可以发出命令来管理代理配置的许多方面。这些包括：

  • 简化配置的所有组件的管理，包括主数据库，备用数据库，零数据丢失恢复设备，重做传输服务和日志应用服务。

  • 协调数据库状态转换并动态更新数据库属性，代理在代理配置文件中记录更改，该文件包含有关配置中所有数据库的信息。代理将更改传播到配置中的所有数据库及其服务器参数文件。

  • 简化配置保护模式的控制（最大化保护，最大化可用性或最大化性能）。

  • 调用Cloud Control验证操作以确保重做传输服务和日志应用服务已配置并正常运行。

简化的切换和故障转移操作：

代理通过允许您使用Cloud Control中的单个键单击或DGMGRL命令行界面上的单个命令来调用切换和故障转移（在本文档中称为 手动故障转移）。对于无人值守管理 ，您可以启用快速启动故障转移，以允许代理确定是否需要进行故障转移，并自动启动故障转移到预先指定的目标备用数据库，而无需DBA干预。可以将快速启动故障转移配置为在没有数据丢失或可配置的数据丢失量的情况下发生。

快速启动故障转移使您可以在不需要手动干预的情况下提高可用性，从而降低管理成本。通过手动故障转移，您可以准确控制故障转移的时间和目标备用数据库。无论您选择哪种方法，代理都会协调配置中所有数据库的角色转换。故障转移完成后，代理会发布快速应用程序通知（FAN）事件，以通知应用程序新主服务器可用。

请注意，您可以使用DBMS_DG PL / SQL程序包使应用程序在遇到特定条件时启动快速启动故障转移。有关更多信息，请参阅应用程序启动的快速启动故障转移 。

只需一个命令即可为配置中所有数据库的切换或故障转移操作启动复杂角色更改。代理自动切换和故障转移到代理配置中的指定备用数据库。通过Cloud Control，您可以从一组可行的备用数据库中选择新的主数据库（已启用并正在运行，具有正常状态）。DGMGRL SWITCHOVER和FAILOVER命令仅要求您在自动启动并完成配置中多个数据库的切换或故障转移操作中的许多步骤之前指定目标备用数据库。

介绍数据挖掘功能的概念。

3.1.1监督数据挖掘

监督学习也被称为定向学习。学习过程由先前已知的从属属性或目标。定向数据挖掘试图将目标的行为解释为一组独立属性或预测变量的函数。

监督学习通常会导致预测模型。这与无监督学习形成对比，其中目标是模式检测。

监督模型的构建涉及培训 ，该过程软件分析许多已知目标值的情况。在训练过程中，模型“学习”进行预测的逻辑。例如，寻求识别可能响应促销的客户的模型必须通过分析已知在过去已响应或未响应促销的许多客户的特征来进行培训。

3.1.1.1监督学习：测试

建筑（培训）需要单独的数据集并测试一些预测模型。构建数据（训练数据）和测试数据必须具有相同的列结构。通常，一个大型表或视图分为两个数据集：一个用于构建模型，另一个用于测试模型。

将模型应用于测试数据的过程有助于确定基于一个选定样本构建的模型是否可推广到其他数据。特别是，它有助于避免这种现象过度拟合，当模型的逻辑太适合构建数据并且因此具有很小的预测能力时可能发生。

3.1.1.2监督学习：评分

同时应用数据称为评分数据，是实际的应用模型的人口。例如，您可以构建一个模型来标识经常购买某个产品的客户的特征。要获取在某个商店购物并且可能购买相关产品的客户列表，您可以将该模型应用于该商店的客户数据。在这种情况下，商店客户数据是评分数据。

大多数监督学习可以应用于感兴趣的人群。主要的监督采矿技术， 分类和回归 ，都可以用于评分。

Oracle Data Mining不支持另一个受监督函数Attribute Importance的评分操作。这种类型的模型建立在感兴趣的人群上，以获得有关该人口的信息;它们不能应用于单独的数据。属性重要性模型返回并排列在预测目标值时最重要的属性。

Oracle Data Mining支持k -Means的增强版本。它通过定义超越了经典实现群集的分层父子关系。

非负矩阵分解

特征提取

非负矩阵分解（NMF）使用原始属性的线性组合生成新属性。该线性组合的系数是非负的。在模型应用期间，NMF模型将原始数据映射到模型发现的新属性（特征）集。

一类支持向量机

异常检测

一类 SVM构建一个类的概要。应用模型时，它会识别与该配置文件不同的案例。这允许检测不一定彼此相关的罕见情况。

正交分区聚类

聚类

正交分区聚类（o-cluster）创建基于网格的分层聚类模型。该算法创建用于定义属性空间中密集区域的聚类。灵敏度参数定义基线密度水平。

奇异值分解与主成分分析

特征提取

奇异值分解（SVD）和主成分分析（PCA）是正交线性变换，其在捕获数据的基础方差时是最佳的。此属性对于减少高维数据的维度和支持有意义的数据可视化非常有用。

除了降维，SVD和PCA还有许多其他重要的应用，例如数据去噪（平滑），数据压缩，矩阵求逆和求解线性方程组。

3.3数据准备

模型的质量在很大程度上取决于用于构建（训练）它的数据的质量。在任何给定的数据挖掘项目中花费的大部分时间都用于数据准备。必须仔细检查，清理和转换数据，并且必须应用与算法相关的数据准备方法。

由于应用模型的任何数据，无论是用于测试还是用于评分，数据准备过程都变得更加复杂，必须经历与用于训练模型的数据相同的变换。

3.3.1 Oracle数据挖掘简化了数据准备

Oracle Data Mining提供了几个可以显着简化数据准备过程的功能：

  • 嵌入式数据准备： 用于训练模型的转换嵌入在模型中，并在模型应用于新数据时自动执行。如果为模型指定转换，则只需指定一次。

  • 自动数据准备（ADP）：Oracle Data Mining支持自动数据准备模式。当ADP处于活动状态时，Oracle Data Mining会自动执行算法所需的数据转换。转换指令与任何用户指定的转换指令一起嵌入到模型中。

  • 自动管理缺失值和稀疏数据： Oracle Data Mining跨采矿算法使用一致的方法来处理稀疏性和缺失值。

  • 透明度：Oracle Data Mining提供模型详细信息，这些信息是模型内部属性的视图。由于反向转换可以将对转换后的属性值映射到可由用户解释的形式，因此可以深入了解模型的内部细节。在可能的情况下，属性值将反转为原始列值。逆变换也应用于监督模型的目标，因此得分结果与原始目标的单位具有相同的单位。

  • 用于自定义数据准备的工具：Oracle Data Mining在DBMS_DATA_MINING_TRANSFORM PL / SQL包中提供了许多常见的转换例程。您可以使用这些例程，或在SQL中或两者中开发自己的例程。SQL语言非常适合在数据库中实现转换。您可以使用自定义转换说明以及ADP或ADP。

3.3.2案例数据

大多数数据挖掘算法作用于单记录案例数据，其中每个案例的信息存储在单独的行中。案例的数据属性存储在列中。

在事务中组织数据时，一个案例（一个事务）的数据存储在许多行中。交易数据的一个例子是市场篮子数据。除了可以对本机事务数据进行操作的关联规则之外，Oracle数据挖掘算法需要单记录案例组织。

3.3.2.1嵌套数据

Oracle Data Mining支持嵌套列中的属性。事务表可以转换为嵌套列，并包含在单记录案例数据表中。同样，星型模式可以转换为嵌套列。通过嵌套数据转换，Oracle Data Mining可以有效地挖掘源自多个源和配置的数据。

3.3.3文本数据

准备和转换非结构化文本数据以进行数据挖掘。

Oracle Data Mining自动将CLOB列和长VARCHAR2列解释为非结构化文本。此外，您可以将short VARCHAR2 ， CHAR ， BLOB和BFILE列指定为非结构化文本。非结构化文本包括数据项，如网页，文档库，Power Point演示文稿，产品规格，电子邮件，报告中的注释字段以及呼叫中心备注。

Oracle Data Mining使用Oracle Text实用程序和术语加权策略来转换非结构化文本以进行挖掘。在文本转换中，提取文本术语并在文本索引中给出数值。文本转换过程可针对模型和单个属性进行配置。一旦转换，文本可以通过数据挖掘算法挖掘。

3.4数据库内评分

评分是将数据挖掘算法应用于新数据。在传统的数据挖掘中，模型使用远程系统上的专用软件构建，并部署到另一个系统进行评分。这是一个麻烦的，容易出错的过程，容易受到安全违规和数据同步的困扰。

使用Oracle Data Mining，评分简单而安全。评分引擎和数据都驻留在数据库中。评分是SQL语言的扩展，因此挖掘结果可以轻松地合并到应用程序和报告系统中。

3.4.1并行执行和易于管理

所有Oracle Data Mining评分例程都支持并行执行以评估大型数据集。

在数据库评分提供了性能优势。所有Oracle Data Mining评分例程都支持并行执行，可显着减少执行复杂查询和评分大型数据集所需的时间。

数据库内挖掘可最大限度地减少支持数据挖掘计划所需的IT工作量。使用标准数据库技术，可以轻松地在更新的数据上刷新（重新创建）模型并重新部署。部署是立即的，因为评分查询保持不变;只在数据库中替换基础模型。

配置RMAN for Oracle RAC概述

RMAN使您可以备份，还原和恢复数据文件，控制文件，服务器参数文件（SPFILE）和归档重做日志文件。RMAN包含在Oracle数据库中，不需要单独安装。您可以从命令行运行RMAN，也可以在Oracle Enterprise Manager的备份管理器中使用RMAN。

Oracle RAC中的归档模式

要归档重做日志文件，Oracle RAC数据库必须处于ARCHIVELOG模式。

您可以运行ALTER DATABASE SQL语句来更改Oracle RAC中的归档模式，因为数据库由本地实例装入，但在任何情况下都不会打开。您无需修改参数设置即可运行此语句。

配置RMAN快照控制文件位置

快照控制文件是RMAN在特定于操作系统的位置创建的数据库控制文件的副本。

RMAN创建快照控制文件，以便在重新同步恢复目录或备份控制文件时，它具有一致的控制文件版本。

您可以为快照控制文件的位置指定群集文件系统或原始设备目标。此文件在群集中的所有节点之间共享，并且必须可由群集中的所有节点访问。运行以下RMAN命令以确定快照控制文件的已配置位置：

显示SNAPSHOT CONTROLFILE NAME;

您可以更改快照控制文件的已配置位置。例如，在Linux和UNIX系统上，您可以通过在RMAN提示符处输入以下内容将快照控制文件位置指定为$ORACLE_HOME/dbs/scf/snap_prod.cf ：

配置SNAPSHOT CONTROLFILE NAME为'$ ORACLE_HOME / dbs / scf / snap_prod.cf';

此命令为每个群集数据库实例设置快照控制文件位置的配置。因此，请确保执行备份的所有节点共享$ORACLE_HOME/dbs/scf目录。

CONFIGURE命令在RMAN会话中创建持久设置。因此，除非要更改快照控制文件的位置，否则无需再次运行此命令。

要删除快照控制文件，必须先更改快照控制文件位置，然后删除旧位置的文件，如下所示：

配置SNAPSHOT CONTROLFILE NAME为' new_name ';删除控制文件的副本;

在Oracle RAC中使用RMAN管理存档的重做日志

当节点生成归档重做日志时，Oracle数据库始终将日志的文件名记录在目标数据库的控制文件中。如果您使用的是恢复目录，则RMAN还会在重新同步时在恢复目录中记录归档的重做日志文件名。

您使用的归档重做日志命名方案很重要，因为当节点写入其文件系统上具有特定文件名的日志时，该文件必须可由必须访问此归档重做日志的任何节点读取。例如，如果node1将日志归档到/oracle/arc_dest/log_1_100_23452345.arc ，则node2只有在其自己的文件系统上读取/oracle/arc_dest/log_1_100_23452345.arc时才能备份此归档重做日志。

您选择的备份和恢复策略取决于您为每个节点配置归档目标的方式。无论是一个节点还是所有节点都执行归档重做日志备份，都必须确保备份所有归档重做日志。如果在恢复期间使用RMAN并行性，则执行恢复的节点必须具有对群集中所有归档重做日志的读取权限。

多个节点可以并行恢复存档日志。但是，在恢复期间，只有一个节点应用存档日志。因此，执行恢复的节点必须能够访问恢复操作所需的所有存档日志。默认情况下，数据库确定在恢复操作期间使用的最佳并行线程数。您可以使用RECOVER命令中的PARALLEL子句来更改并行线程的数量。

归档重做日志的准则和注意事项

主要考虑因素是确保在恢复期间可以从每个节点读取所有归档重做日志，如果可能，还可以在备份期间读取。在恢复期间，如果从执行恢复的节点可以看到存档的日志目标，则Oracle数据库可以成功恢复存档的日志数据。

4.1.2撤消空间

数据库使用撤消空间来存储用于读取一致性，恢复和回滚语句的数据。此数据存在于一个或多个撤消表空间中。如果使用Database Configuration Assistant（DBCA）创建数据库，则会自动创建撤消表空间。要手动创建撤消表空间，请将UNDO TABLESPACE子句添加到CREATE DATABASE语句中。

为了自动管理撤消数据，Oracle数据库使用自动撤消管理 ，透明地创建和管理撤消段。启用自动撤消管理，将UNDO_MANAGEMENT初始化参数设置为AUTO （默认设置）。如果未指定，则UNDO_MANAGEMENT初始化参数使用AUTO设置。Oracle强烈建议使用自动撤消管理，因为它可以显着简化数据库管理，并且无需手动调整撤消（回滚）段。支持使用回滚段的手动撤消管理以实现向后兼容。

4.1.3重做日志文件

重做日志文件的大小会影响性能，因为数据库写程序和归档程序进程的行为取决于重做日志大小。通常，较大的重做日志文件可提供更好的性能。较小的日志文件会增加检查点活动并降低性能。

虽然重做日志文件的大小不会影响LGWR性能，但它可能会影响DBWR和检查点行为。 检查点频率受多种因素影响，包括日志文件大小和FAST_START_MTTR_TARGET初始化参数的设置。如果将FAST_START_MTTR_TARGET参数设置为限制实例恢复时间，则Oracle数据库会根据需要自动尝试检查点。在这种情况下，日志文件的大小应足够大，以避免由于日志文件大小不足而进行额外的检查点。可以通过从V$INSTANCE_RECOVERY视图查询OPTIMAL_LOGFILE_SIZE列来获取最佳大小。您还可以在Oracle Enterprise Manager云控制（云控制）的“ 重做 日志 组”页面上获取调整建议。

可能无法始终为重做日志文件提供特定大小的建议，但是在100 MB到几千兆字节范围内的重做日志文件被认为是合理的。根据系统生成的重做次数调整联机重做日志文件的大小。粗略的指南是每20分钟最多切换一次日志文件。

4.1.4表空间

如果使用Database Configuration Assistant（DBCA）创建数据库，则种子数据库会自动包含必要的表空间。如果选择不使用DBCA，则必须在创建数据库后创建额外的表空间。

除SYSTEM和SYSAUX表空间外，所有数据库都应具有多个表空间。这些额外的表空间包括：

  • 一个临时表空间，用于排序等操作

  • undo表空间，包含读取一致性，恢复和撤消语句的信息

  • 至少一个用于应用程序的表空间（在大多数情况下，应用程序需要多个表空间）

对于包含许多数据文件的超大型表空间，您可以运行多个ALTER TABLESPACE ...并行ADD DATAFILE语句。在创建表空间期间，组成表空间的数据文件将使用特殊的空块图像进行初始化。临时文件未初始化。

Oracle数据库执行此操作以确保它可以完整地写入所有数据文件，但如果以串行方式完成，这显然是一个漫长的过程。因此，同时运行多个CREATE TABLESPACE语句以加速创建表空间。对于永久表，在表空间创建的本地和全局扩展区管理之间进行选择会极大地影响性能。对于与读取相比具有中等到大的插入，修改或删除操作的永久表空间，请选择本地扩展区管理。

4.2.1表压缩

您可以以压缩格式存储堆组织表，该格式对任何类型的应用程序都是透明的。数据库块中的压缩数据是自包含的，这意味着在块中可以获得在块中重新创建未压缩数据所需的所有信息。块也在缓冲区缓存中压缩。表压缩不仅减少了磁盘存储，还减少了内存使用，特别是缓冲区缓存要求。通过减少访问表所需的I / O操作量并增加缓冲区高速缓存命中的可能性来实现性能改进。

Oracle数据库具有高级压缩选项，使您可以提高任何类型的应用程序工作负载（包括数据仓库和OLTP应用程序）的性能，同时减少数据库所需的磁盘存储。您可以对所有类型的数据使用高级压缩功能，包括结构化数据，非结构化数据，备份数据和网络数据。

估计压缩因子

表压缩的工作原理是消除各个块中的列值重复。块中所有行和列中的重复值在块的开头存储一次，即所谓的块的符号表。所有出现的此类值都将替换为对符号表的简短引用。具有更多重复值的块中的压缩更高。

在压缩大表之前，您应该估计预期的压缩因子。压缩因子被定义为以未压缩形式存储信息所需的块数除以压缩存储所需的块数。可以通过对要压缩的表的少量代表性数据块进行采样并且比较针对未压缩和压缩情况的每个块的平均记录数来估计压缩因子。经验表明，大约1000个数据块提供了非常准确的压缩因子估计。请注意，您采样的块越多，结果就越准确。

调整以实现更好的压缩比

在许多情况下，Oracle数据库实现了良好的压缩因子，无需进行特殊调整。作为DBA或应用程序开发人员，您可以尝试通过在压缩发生时重新组织记录来调整压缩因子。在某些情况下，调整可以略微改善压缩因子，而在其他情况下则可以。

要提高压缩系数，必须增加数据块中值重复的可能性。可实现的压缩因子取决于特定列或列对的基数（表示列值重复的可能性）以及这些列的平均行长度。表压缩不仅压缩单个列的重复值，而且尽可能尝试使用多列值对。如果不详细了解数据分布，则很难预测最佳顺序。

使用属性聚簇表

属性聚簇表是一个堆组织表，它根据用户指定的聚簇指令将数据存储在磁盘上。指令确定存储在表中的数据是基于指定的列排序，还是基于允许多列I / O减少的特殊算法。属性集群仅适用于批量插入操作 - 例如INSERT/*+APPEND*/或ALTER TABLE ...MOVE PARTITION命令 - 在传统DML中被忽略。

通过结合区域映射减少物理I / O，使用属性聚簇表可以显着降低表扫描的I / O成本。此外，它还可以改善数据压缩，因为当相同的值在磁盘上彼此更接近时，可以更容易地压缩数据。

4.3共享服务器的性能注意事项

使用共享服务器可减少数据库主机上的进程数和内存量。共享服务器对于有许多OLTP用户执行间歇性事务的数据库是有益的。

对于与数据库具有高连接速率的系统，使用共享服务器而不是专用服务器通常也更好。对于共享服务器，当收到连接请求时，调度程序可用于处理并发连接请求。但是，对于专用服务器，将为每个连接请求顺序初始化特定于连接的专用服务器。

当使用共享服务器体系结构时，某些数据库功能的性能可以得到改善，并且当使用共享服务器体系结构时，某些数据库功能的性能可能会略微降低。例如，可以防止会话在并行执行处于活动状态时迁移到另一个共享服务器。

即使在处理了客户端的请求之后，会话仍然可以保持不可迁移，因为并非所有用户信息都已存储在 UGA。如果服务器要处理来自客户端的请求，那么未存储在UGA中的用户状态部分将是不可访问的。为避免这种情况，单个共享服务器通常需要保持绑定到用户会话。

16.1选择XMLType存储模型和索引方法简介

XMLType是一种抽象的SQL数据类型，它提供了不同的存储和索引模型，以最好地适应您的XML数据及其使用。因为它是一种抽象数据类型，所以您的应用程序和数据库查询可以获得灵活性：所有XMLType操作都可以使用相同的接口。

不同的应用程序以不同的方式使用XML数据有时它是由关系数据源构建的，因此它是相对结构化的。有时它用于提取，转换和加载（ETL）操作，在这种情况下它也是非常结构化的。有时它用于自由格式文档（非结构化或半结构化），如书籍和文章。

对于不同类型的数据，检索方法也可以是不同的。以数据为中心的用例通常涉及一组固定的查询，而以文档为中心的用例通常涉及任意（ad-hoc）查询。

由于XML的使用范围很广，因此没有一种通用的存储模型可以为每个用例提供最佳性能和灵活性。Oracle XML DB为XMLType提供了两种存储模型，以及适用于这些不同存储模型的几种索引方法。您可以定制性能和功能，以最好地适应您拥有的XML数据类型以及使用它的方式。

因此，要做出的一个关键决定是使用哪种XMLType存储模型来处理哪些XML数据。本章帮助您为给定的用例选择最佳存储选项。

XMLType表和列可以通过以下方式存储：

  • 二进制XML存储 - 这也称为解析后持久性 。它是Oracle XML DB的默认存储模型。它是一种专为XML数据设计的解析后二进制格式。二进制XML是紧凑的，并且支持XML模式。二进制XML存储的最大优点是灵活性 ：您可以将其用于基于XML模式的文档或不基于XML模式的文档。您可以将其与XML模式一起使用，该模式允许高数据可变性或者显着或意外地发展。此存储模型还提供有效的部分更新和可流式查询评估。

  • 对象关系存储 - 这也称为 结构化存储和基于对象的持久性 。该存储模型表示XML数据的实体关系（ER）分解。它通过已知的或多或少固定的查询集为高度结构化数据提供最佳性能。查询性能与关系数据的性能相匹配，可以在适当的位置执行更新。

注意：

从Oracle Database 12c第1版（12.1.0.1）开始， 不推荐使用 XMLType的非结构化（ CLOB ）存储模型。请改用二进制XML存储。

如果您拥有存储为CLOB数据的XMLType数据，请考虑使用Oracle GoldenGate将其移至二进制XML存储格式。如果文档保真度对于特定XML文档很重要，则将其副本存储在关系CLOB列中。

Oracle XML DB支持XMLType数据的以下索引类型。

  • 对象关系存储的B树功能索引

  • 二进制XML存储上的XML搜索索引

  • XMLIndex包含二进制XML存储上的结构化和非结构化组件

  • 在二进制XML存储上为XMLIndex （结构化和非结构化组件）自动创建的辅助表上的B树索引

不同的用例需要XMLType存储模型和索引的不同组合。

16.4 XMLType存储模型注意事项

对于大多数用例，Oracle建议您使用XMLType二进制XML存储。对象关系存储在特殊情况下是适当的。

除非满足以下所有条件 ，否则对象关系存储是不合适的：

  • 您有一个XML模式，它严格指定您要存储在给定XMLType列或表中的所有XML文档的详细数据格式。您的应用程序是以数据为中心的。

  • 您的数据不是特别稀疏（不包括许多空或缺少的元素）。
