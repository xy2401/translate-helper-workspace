11适用于DRDA的Oracle数据库提供程序的安全和存储注意事项

在使用Oracle数据库提供程序进行DRDA将DB2 DRDA应用程序转换为Oracle之前，必须考虑几个安全性和存储问题。

适用于DRDA的Oracle数据库提供程序的安全性和存储概述

适用于DRDA的Oracle数据库提供程序使用Oracle数据库的安全和授权模型;这确保了对用户数据的正确SQL访问。

DRDA指定了两种主要授权模型： REQUESTER和OWNER 。它们被指定为包的属性PKGATHRUL 。

  • 在REQUESTER模型中，当前登录的用户是授权控件。

    REQUESTER模型也称为动态SQL规则，它用于执行在运行时构造的SQL。这是Oracle数据库为SQL访问实现的安全模型。

  • 在OWNER模型中，存储的USERID （包所有者， PKGOWNID ）在语句执行期间实例化不同的USERID 。

    OWNER模型也称为静态SQL规则，它用于包中的预绑定SQL。Oracle数据库未实现等效的安全模型。

由于Oracle数据库仅实现了一种安全模型，因此存储和动态创建的包中的所有SQL都在动态SQL规则模型下执行。这不会覆盖使用AUTHID DEFINER属性创建的任何PL / SQL对象的实例化模式。

有关AUTHID DEFINER详细信息，请参见Oracle®DatabasePL / SQL语言参考 。

用于DRDA的Oracle数据库提供程序中的身份验证和加密

DRDA要求在可以执行对数据库（ACCRDB）的任何访问之前执行用户认证（ACCSEC（SECMEC），SECCHK）。

认证服务

DRDA提供两种类型的身份验证服务：专用和多路复用。

  • 在专用的情况下，AS提供对单个专用数据库的访问，而无需连接到另一个数据库。稍后在ACCRDB中传递的rdbnam必须与目标数据库名称匹配;否则，会话访问失败并显示RDBAFLRM消息。

    在此模式下，ACCSEC不需要rdbnam实例变量来执行身份验证，因为无法在数据库之间进行选择。因此，AS在ACCSEC处理时可以自由地与单个数据库建立会话关联，并验证所请求的安全机制（SECMEC）是否可用。发送SECCHK命令时，将维护初始数据库会话并执行完整身份验证。

    此模式必须与不支持多路服务器的旧AR一起使用。如果AR在此模式下提供rdbnam，则必须针对连接的数据库对其进行验证。

  • 在多路复用模式下，AS可以同时访问许多数据库。在此模式下，需要ACCSEC中的rdbnam实例变量来验证请求的安全机制。AS连接到请求的数据库以进行安全验证。发送SECCHK命令时，将维护初始数据库会话并执行完整身份验证。

    较新的AR支持此模式，它将rdbnam作为ACCSEC的一部分提供。如果AR在此模式下不在ACCSEC中提供rdbnam，则会生成错误并将RDBAFLRM响应发送到ACCSEC。

加密服务

适用于DRDA的Oracle数据库提供程序支持以下安全机制：

  • EUSRIDPWD - 加密的用户ID和密码机制

  • EUSRIDNWPWD - 加密的用户ID，密码和新密码机制

  • USRIDPWD - 用户ID和密码机制

  • USRIDNWPWD - 用户ID，密码和新密码机制

  • USRENCPWD - 用户ID和密码加密机制

在将安全机制验证为可访问之后，可以继续实际的安全授权（SECCHK）。

BNF pdb_refresh_mode_clause的规范

pdb_refresh_mode_clause的BNF规范：

刷新模式（（手动）|（每次“refresh_interval”（HOURS | MINUTES））|（NONE））

如何阅读BNF规范

您从左到右阅读BNF表示。表示遵循以下符号规则：

  1. Oracle关键字和命令是大写的，并按原样指定。在pdb_refresh_mode_clause中，Oracle关键字是REFRESH，MODE，EVERY，HOURS，MINUTES，NONE和SWITCHOVER。
  2. 括号用于对项目进行分组。分组使语法更易于阅读。例如，MANUAL，EVERY和NONE选项分组在括号内。选项HOURS和MINUTES的分组清楚地表明它们属于每个选项。
  3. 方括号（[]）包含可选项。
  4. 由竖线分隔的分组中的项目意味着在给定时间仅需要其中一个项目。
  5. 用户定义的输入（如参数和子句）用双引号括起来。例如，pdb_refresh_mode_clause中的“refresh_interval”。
  6. 标点符号用双引号括起来。双引号表示标点符号被解释为字符串。
  7. 标点符号前面的正斜杠字符（“/”）表示标点符号不会被解释为字符串。例如，在句法元素之后的/“，”表示句法元素可以重复。

创建ROLLBACK SEGMENT

注意：

Oracle强烈建议您以自动撤消管理模式运行数据库，而不是使用回滚段。除非必须这样做才能与早期版本的Oracle数据库兼容，否则请勿使用回滚段。有关自动撤消管理的信息，请参阅Oracle数据库管理员指南 。

目的

使用CREATE ROLLBACK SEGMENT语句创建回滚段 ，该回滚段是Oracle数据库用于存储反转或撤消事务所做更改所必需的数据的对象。

本节中的信息假定您的数据库未在自动撤消模式下运行（ UNDO_MANAGEMENT初始化参数设置为MANUAL或根本未设置）。如果数据库以自动撤消模式运行（ UNDO_MANAGEMENT初始化参数设置为AUTO ，这是默认值），则不允许回滚段。但是，抑制了回滚段操作中生成的错误。

此外，如果您的数据库具有本地管理的SYSTEM表空间，则无法在任何字典管理的表空间中创建回滚段。相反，您必须使用自动撤消管理功能或创建本地管理的表空间来保存回滚段。

注意：

表空间可以有多个回滚段。通常，多个回滚段可提高性能。

  • 表空间必须在线才能为其添加回滚段。

  • 创建回滚段时，它最初处于脱机状态。要使其可用于Oracle数据库实例的事务，请使用ALTER ROLLBACK SEGMENT语句将其联机。要在启动数据库时自动使其联机，请将段名称添加到ROLLBACK_SEGMENT初始化参数的值。

要在SYSTEM表空间以外的表空间中使用对象：

  • 如果使用回滚段进行撤消，则至少有一个回滚段（ SYSTEM回滚段除外）必须在线。

  • 如果以自动撤消模式运行数据库，则至少有一个UNDO表空间必须联机。

Oracle数据库必须经常访问回滚段。因此，Oracle强烈建议您不要通过省略此子句显式或隐式地在SYSTEM表空间中创建回滚段。此外，为了避免包含回滚段的表空间的高争用，它不应包含其他对象（如表和索引），并且它应该需要最小的扩展区分配和释放。

要实现这些目标，请在本地管理的表空间中创建回滚段，并禁用自动分配 - 使用带有UNIFORM设置的EXTENT MANAGEMENT LOCAL子句创建的表空间。不支持AUTOALLOCATE设置。

Oracle ACFS概述

Oracle自动存储管理群集文件系统（Oracle ACFS）是一种多平台，可扩展的文件系统和存储管理技术，可扩展Oracle自动存储管理（Oracle ASM）功能以支持所有客户文件。

Oracle ACFS支持Oracle数据库文件和应用程序文件，包括可执行文件，数据库数据文件，数据库跟踪文件，数据库警报日志，应用程序报告，BFILE和配置文件。其他支持的文件是视频，音频，文本，图像，工程图和所有其他通用应用程序文件数据。Oracle ACFS符合适用于Linux和UNIX的POSIX标准，适用于Windows的Windows标准。

Oracle ACFS文件系统与Oracle ASM通信，并使用Oracle ASM存储进行配置，如下图所示。

图11-1 Oracle ACFS存储层


“图11-1 Oracle ACFS存储层”的说明

Oracle ACFS利用Oracle ASM功能，该功能支持：

  • Oracle ACFS动态文件系统调整大小

  • 通过直接访问Oracle ASM磁盘组存储，实现最佳性能

  • 在Oracle ASM磁盘组存储中平衡分配Oracle ACFS，以提高I / O并行性

  • 通过Oracle ASM镜像保护机制实现数据可靠性

Oracle ACFS建立并维护与Oracle ASM实例的通信，以参与Oracle ASM状态转换，包括Oracle ASM实例和磁盘组状态更新以及磁盘组重新平衡。Oracle自动存储管理与Oracle ACFS和Oracle ASM动态卷管理器（Oracle ADVM）提供对所有客户数据的支持，并在Oracle Restart（独立）上的多个供应商平台和操作系统环境中提供一组通用的Oracle存储管理工具和服务和群集配置。

Oracle ACFS与Oracle Clusterware技术紧密结合，直接参与Clusterware集群成员资格状态转换和Oracle Clusterware基于资源的高可用性（HA）管理。此外，Oracle安装，配置，验证和管理工具已更新，以支持Oracle ACFS。

可以使用本机操作系统文件系统工具和标准应用程序编程接口（API）访问和管理Oracle ACFS。Oracle ACFS也可以使用Oracle ASM Configuration Assistant进行管理。可以使用行业标准网络连接存储（NAS）文件访问协议访问Oracle ACFS：网络文件系统（NFS）和通用Internet文件系统（CIFS）。但是，Windows上的CIFS客户端在与Oracle ACFS Linux，Solaris或AIX服务器连接时不能使用ACL，但可以在Windows上将ACL与Oracle ACFS一起使用。

除了共享文件数据外，Oracle ACFS还提供其他存储管理服务，包括支持Oracle Grid Infrastructure集群范围的安装注册表，动态在线文件系统调整大小以及每个文件系统的多个空间高效快照。

Oracle ACFS通过提供以下功能，为整体Oracle存储管理做出贡献：

  • 与Oracle ASM和Oracle Clusterware技术集成的通用独立服务器和集群文件系统解决方案

  • 跨多个供应商平台和操作系统的一组通用文件系统功能，提供本机操作系统或第三方文件系统解决方案的替代方案

  • 独立和群集范围共享Oracle数据库主目录，所有Oracle数据库文件和应用程序数据

  • 统一，一致的共享文件访问和所有客户应用程序文件的群集范围命名

  • 与Oracle Clusterware高可用性资源集成

Oracle ACFS可容纳大容量存储容量和大量群集节点。它有效地管理大量文件系统，文件，并支持具有exabyte功能的文件和文件系统容量的小型和大型文件。Oracle ACFS为具有数百万个文件的大型目录提供优化的快速目录查找。

Oracle ACFS提供对稀疏文件的支持。Oracle ACFS稀疏文件极大地有利于NFS客户端写入操作，这些操作通常由NFS服务器和关联的Oracle ACFS文件系统无序接收。通常，当应用程序写入超出文件末尾时，将分配存储并将零插入超出旧文件末尾和新文件的开头。使用此功能，文件中将保留一个孔而不是插入的零。当读取孔时，Oracle ACFS会在内存中用零填充这些漏洞。稀疏文件功能有利于NFS性能以及有意执行此类写入的其他应用程序的性能和磁盘利用率。此外，对于本质上稀疏的文件，也减少了时间和存储优势，这意味着它们具有大量未使用的空间，例如用于虚拟机的一些图像文件。对于稀疏文件支持， COMPATIBLE.ADVM磁盘组属性必须设置为12.2或更高。

Oracle ACFS文件系统通常安装在所有群集节点上，以便为群集提供单个名称空间，以便每个节点保持与装入的文件系统相同的视图和访问功能。如果成员发生故障，另一个集群成员将代表发生故障的成员快速恢复任何未完成的元数据事务。恢复后，可以恢复其他活动集群成员和任何远程客户端系统的访问。

  • 包含引用对象的表或视图的OID，长度为16个字节。

  • Rowid提示，长度为10个字节。

9.3.2 REF列的完整性约束

在参照完整性约束REF列确保有针对行对象REF 。

REF的参照完整性约束创建与在关系数据上指定主键/外键关系相同的关系。通常，应尽可能使用参照完整性约束，因为它们是确保REF的行对象存在的唯一方法。但是，您无法在嵌套表中的REF上指定参照完整性约束。

9.3.3 Scoped REF的性能和存储注意事项

范围限制的REF被约束为仅包含对指定对象表的引用。将列类型，集合元素或对象类型属性声明为REF时，可以指定范围内的REF 。

通常，您应该使用范围化的REF而不是未范围的REF因为更有效地存储范围的REF 。而无范围的REF至少需要36个字节才能存储（如果使用rowid则更多），范围内的REF仅存储为其目标对象的OID，并且可能需要少于16个字节，具体取决于引用的OID是否是系统生成的或基于主键。系统生成的OID需要16个字节;基于主键（基于PK）的OID需要足够的空间来存储主键值，该主键值可能小于16个字节。但是，基于PK的OID的REF （必须在选择时动态构建）可能在内存中占用比系统生成的OID的REF更多的空间。

除了需要更少的存储空间之外，范围内的REF通常使优化器能够优化将范围内的REF取消引用到更有效的连接的查询。这种优化是不可能的未范围REF是因为优化器无法确定无范围含有桌REF在查询优化时间s。

与参照完整性约束不同，作用域REF不确保引用的行对象存在;它们只确保引用的对象表存在。因此，如果为行对象指定作用域REF然后删除行对象，则作用域REF将成为悬空REF因为引用的对象不再存在。

注意：

引用完整性约束是隐式的。

如果应用程序设计要求引用的对象分散在多个表中，则无范围的REF非常有用。因为作用域REF会忽略rowid提示，所以如果rowid提示的性能增益（如“ 使用WITH ROWID选项进行对象访问的性能改进 ”中所述 ）超过了存储保存的优势，则应使用未作用域的REF 。使用作用域REF的查询优化。

9.3.3.1 Scoped REF的索引

您可以使用CREATE INDEX命令在作用域的REF列上构建索引。这允许您使用索引有效地评估取消引用作用域REF的查询。此类查询将隐式转换为联接。对于某些类型的查询，Oracle可以使用范围内的REF列上的索引来有效地评估连接。

例如，假设对象类型address_objtyp用于创建名为address_objtab的对象表：

CREATE TABLE address_objtab OF address_objtyp;

可以创建一个people_reltab2表，其定义与例9-2中显示的people_reltab表相同，只是REF用于地址。接下来，可以在address_ref列上创建索引。

示例9-3在Scoped REF列上创建索引

CREATE TABLE people_reltab2（id NUMBER（4）CONSTRAINT pk_people_reltab2 PRIMARY KEY，name_obj name_objtyp，address_ref REF address_objtyp SCOPE IS address_objtab，phones_ntab phone_ntabtyp）NESTED TABLE phones_ntab STORE AS phone_store_ntab2; CREATE INDEX address_ref_idx ON people_reltab2（address_ref）;

以下查询取消引用address_ref ：

SELECT id FROM people_reltab2 p WHERE p.address_ref.state ='CA';

执行此查询时， address_ref_idx索引用于有效地评估它。这里， address_ref是一个作用域的REF列，它存储对address_objtab对象表中存储的地址的引用。Oracle隐式地将前面的查询转换为带有连接的查询：

SELECT p.id FROM people_reltab2 p，address_objtab a WHERE p.address_ref = REF（a）AND a.state ='CA';

Oracle查询优化器可能会创建一个计划来执行嵌套循环连接，并将address_objtab作为外部表，并使用address_ref范围的REF列上的索引查找匹配的地址。

9.3.4使用WITH ROWID选项进行对象访问的性能改进

如果为REF列指定了WITH ROWID选项，Oracle将维护REF引用的对象的rowid。然后，Oracle可以使用REF包含的rowid直接找到引用的对象，而无需从OID索引中获取rowid。因此，您使用WITH ROWID选项指定rowid提示。维护rowid需要更多的存储空间，因为rowid会将10个字节添加到REF的存储要求中。

绕过OID索引搜索可提高应用程序中REF遍历（导航访问）的性能。根据以下因素，实际性能增益可能因应用而异：

  • OID索引有多大。

  • OID索引是否缓存在缓冲区缓存中。

  • 应用程序执行了多少次REF遍历。

WITH ROWID选项只是一个提示，因为当您使用此选项时，Oracle会使用REF的OID检查行对象的OID。如果两个OID不匹配，Oracle将使用OID索引。对于具有参照完整性约束的REF ，范围REF不支持rowid提示。 。

了解朴素贝叶斯算法。

朴素贝叶斯算法基于条件概率。它使用贝叶斯定理， 通过计算历史数据中的值的频率和值的组合来计算概率的公式。

贝叶斯定理根据已发生的另一事件的概率，找出事件发生的概率。如果B代表从属事件而A代表先前事件，贝叶斯定理可以如下陈述。

注意：

问题（B给出A）= Prob（A和B）/ Prob（A）

为了计算给定A的B的概率，算法计算A和B一起出现的情况的数量，并将其除以A单独出现的情况的数量。

假设您想确定21岁以下的客户增加支出的可能性。在这种情况下，先验条件（ A ）是“在21以下”，并且从属条件（ B ）是“增加支出”。

如果培训数据中有100个客户，其中25个是21岁以下增加支出的客户，那么：

问题（A和B）= 25％

如果100个客户中有75个不满21，那么：

问题（A）= 75％

贝叶斯定理预测，21岁以下的客户中有33％可能会增加支出（25/75）。

两种情况一起发生的情况称为成对 。在实施例20-1中 ，所有病例的25％是成对的。

仅发生先前事件的情况称为单身 。在例20-1中 ，75％的案例是单身。

这些步骤描述了如何切换到物理备用数据库。

注意：

如果存在连接主数据库和备用数据库的远程同步实例（或首选远程同步实例和备用远程同步实例的组合），则切换到备用数据库的过程与本主题中描述的过程相同。远程同步实例是可用还是不可用不会影响切换。在切换期间，主服务器和备用服务器必须能够直接相互通信，并执行无法远程同步实例的切换角色转换步骤。有关如何正确设置此类配置的示例，请参阅Oracle Data Guard概念和管理中的 “使用远程同步实例”，以便远程同步实例可以在切换后为两个数据库的新角色提供服务。

  1. 验证目标备用数据库是否已准备好进行切换。

    新的switchover语句具有VERIFY选项，可以检查切换所需的许多条件。检查的一些项目是：Redo Apply是否在切换目标上运行;切换目标的发布版本是否为12.1或更高版本;切换目标是否同步;以及是否有MRP运行。

    假设主数据库具有BOSTON的DB_UNIQUE_NAME ，并且切换目标备用数据库具有CHICAGO的DB_UNIQUE_NAME 。在主数据库BOSTON ，发出以下SQL语句以验证切换目标CHICAGO是否已准备好进行切换：

    SQL> ALTER DATABASE SWITCHOVER到CHICAGO VERIFY;第1行的错误：ORA-16470：重做应用未在切换目标上运行

    启动Redo Apply后，再次发出以下语句：

    SWITCHOVER VERIFY警告：切换目标具有需要清除的脏的联机重做日志文件。清除在线重做日志文件需要时间。这可能会减慢切换过程。

    您可以解决问题，或者如果切换性能不重要，则可以忽略这些警告。在确定了必要的修复后，再次发出以下SQL语句：

    SQL> STARTUP;

创建目录

目的

使用CREATE DIRECTORY语句创建目录对象。目录对象指定服务器文件系统上的目录的别名，外部二进制文件LOB（ BFILE ）和外部表数据位于该目录中。在PL / SQL代码和OCI调用中引用BFILE时，可以使用目录名，而不是硬编码操作系统路径名，以提高管理灵活性。

所有目录都在单个命名空间中创建，并且不属于单个模式。通过将目录的对象权限授予特定用户，可以保护对存储在目录结构中的BFILE的访问。

先决条件

您必须具有CREATE ANY DIRECTORY系统特权才能创建目录。

创建目录时，会自动为该目录授予READ ， WRITE和EXECUTE对象权限，您可以将这些权限授予其他用户和角色。DBA还可以将这些权限授予其他用户和角色。

对目录的WRITE权限与外部表有关。它们让被授权者确定外部表代理是否可以将日志文件或错误文件写入目录。

对于文件存储，还必须创建相应的操作系统目录，Oracle自动存储管理（Oracle ASM）磁盘组或Oracle ASM磁盘组中的目录。您的系统或数据库管理员必须确保操作系统目录对Oracle数据库进程具有正确的读写权限。

为目录授予的权限是独立于为操作系统目录定义的权限而创建的，两者可能完全对应也可能不完全对应。例如，如果示例用户hr被授予目录对象的READ权限但相应的操作系统目录没有为Oracle数据库进程定义的READ权限，则会发生错误。

限制

打开BFILE对象时，目录对象路径或文件名中不允许使用符号链接。检查整个目录路径和文件名，如果找到任何符号链接，则返回以下错误：

ORA-22288：文件或LOB操作FILEOPEN路径中的软链接失败

解决方法

如果您尝试打开的数据库目录对象或文件名包含符号链接，请将其更改为提供实际路径和文件名。

ALL_SYNONYMS描述了当前用户可访问的同义词。

以下条件确定ALL_SYNONYMS显示的同义词列表：

  • 登录用户拥有的所有私有同义词，即使指向的基础对象不可访问。

  • 所有公共同义词，即使指向的基础对象不可访问。

  • 由不同用户拥有的所有私有同义词，其中由该同义词或任何嵌套同义词链指向的最终基础对象，由于对登录用户的授予或对角色的授予而被知道是可访问的。本次会议的效果。

  • 如果当前会话具有以下任何特权，则会显示直接指向本地对象的所有同义词，因为假定会话可以访问这些对象：

    • 锁定任何表格

    • 读一个表

    • 选择任何表格

    • 插入任何表

    • 更新任何表格

    • 删除任何表格

    排除指向远程对象的同义词，因为刚刚列出的系统权限不会自动传达对这些远程对象的访问权限。此外，如果同义词指向除表和视图之外的对象（例如序列，PL / SQL过程等），则此规则可能会显示最终解析为此会话无法访问的对象的同义词。

  • 不同用户拥有的所有私有同义词（同义词是通过数据库链接）都被排除在外。

13.16覆盖率

COVERAGE编译指示标记PL / SQL代码，这对于测试覆盖率是不可行的。这些标记提高了覆盖度量标准。

COVERAGE编译指示标记PL / SQL源代码，以指示代码可能无法对覆盖进行可行的测试。该编译指示标记了特定的代码部分。标记不可行代码可提高用于评估已完成测试的覆盖度指标的质量。

coverage_pragma

COVERAGE编译指示可能出现在任何声明或声明之前。

coverage_pragma_argument

COVERAGE pragma参数必须具有以下值之一：

  • '不可行'

  • 'NOT_FEASIBLE_START'

  • 'NOT_FEASIBLE_END'

当COVERAGE编译指示出现参数'NOT_FEASIBLE'时，它标记整个基本块，其中包括pragma后面的第一个声明或语句的开头。

参数为“NOT_FEASIBLE_START”的COVERAGE pragma可能出现在任何声明或任何声明之前。必须后跟COVERAGE编译指示，其参数为“NOT_FEASIBLE_END”。第二个pragma可能出现在任何声明或任何声明之前。它必须出现在与第一个pragma相同的PL / SQL块中，而不是出现在任何嵌套的子程序定义中。

一对关联的COVERAGE编译指示标记从基本块的开头不可行的基本块，包括第一个pragma之后的第一个语句或声明的开头到基本块的末尾，包括第二个之后的第一个语句或声明附注。

COVERAGE编译指示，其范围包括内部子程序的定义或声明，不会将该子程序的块标记为不可行。
