本章讨论在数据仓库环境中提取，传输，转换和加载数据的过程。它包括以下主题：

15.1数据仓库中ETL概述

你必须加载你的定期数据仓库，以便它可以促进业务分析。为此，必须提取来自一个或多个操作系统的数据并将其复制到数据仓库中。数据仓库环境中的挑战是在许多系统上集成，重新排列和整合大量数据，从而为商业智能提供新的统一信息库。

从源系统提取数据并将其带入数据仓库的过程通常称为ETL ，它代表提取，转换和加载。请注意，ETL指的是一个广泛的过程，而不是三个明确定义的步骤。首字母缩略词ETL可能过于简单化，因为它省略了运输阶段，并暗示该过程的每个其他阶段都是不同的。然而，整个过程称为ETL。

ETL的方法和任务多年来一直是众所周知的，并不一定是数据仓库环境所特有的：各种专有应用程序和数据库系统是任何企业的IT骨干。数据必须在应用程序或系统之间共享，尝试集成它们，为至少两个应用程序提供相同的世界图片。这种数据共享主要通过类似于现在称为ETL的机制来解决

15.1.1数据仓库中的ETL基础知识

在ETL过程中会发生什么？以下任务是该过程中的主要操作：

  • 数据仓库中的数据提取

  • 数据仓库中的数据传输

15.1.1.1数据仓库中数据的提取

在提取期间，从许多不同的源（包括数据库系统和应用程序）中识别和提取所需数据。通常，不可能识别感兴趣的特定子集，因此必须提取比必要更多的数据，因此相关数据的识别将在稍后的时间点完成。根据源系统的功能（例如，操作系统资源），在此提取过程中可能会发生一些转换。提取的数据大小从数百千字节到千兆字节不等，具体取决于源系统和业务情况。对于两个（逻辑上）相同的提取之间的时间差异也是如此：时间跨度可以在天/小时和分钟之间变化到接近实时。例如，Web服务器日志文件可以在很短的时间内轻松增长到数百兆字节。

15.1.1.2数据仓库中数据的传输

在提取数据之后，必须将其物理地传输到目标系统或中间系统以进行进一步处理。根据所选择的运输方式，在此过程中也可以进行一些转换。例如，通过网关直接访问远程目标的SQL语句可以将两列连接为SELECT语句的一部分。

本节中许多示例的重点是可伸缩性。许多Oracle数据库的长期用户都是使用PL / SQL编写复杂数据转换逻辑的专家。这些章节为许多此类数据操作操作提供了替代方案，特别强调利用Oracle新SQL功能的实现，特别是对于ETL和并行查询基础结构。

15.2数据仓库的ETL工具

设计和维护ETL过程通常被认为是数据仓库项目中最困难和资源密集的部分之一。许多数据仓库项目使用ETL工具来管理此过程。例如，Oracle Data Integrator（ODI）提供ETL功能并利用固有的数据库功能。其他数据仓库构建器在数据库内部或外部创建自己的ETL工具和流程。

除了提取，转换和加载的支持之外，还有一些其他任务对于成功的ETL实现很重要，作为数据仓库日常操作的一部分，并支持进一步的增强。除了支持设计数据仓库和数据流之外，这些任务通常由ODI等ETL工具解决。

Oracle不是ETL工具，也不为ETL提供完整的解决方案。但是，Oracle确实提供了一组丰富的功能，可供ETL工具和自定义ETL解决方案使用。Oracle提供了在Oracle数据库之间传输数据，转换大量数据以及将新数据快速加载到数据仓库的技术。

15.2.1数据仓库中的日常操作

必须按特定顺序安排和处理连续的加载和转换。根据操作的成功或失败或部分操作，必须跟踪结果，然后可以启动后续的替代过程。通常由ETL工具（如Oracle Data Integrator（ODI））解决对进度的控制以及操作的业务工作流的定义。

15.2.2数据仓库的演变

由于数据仓库是一个生动的IT系统，因此源和目标可能会发生变化。必须在系统的生命周期内维护和跟踪这些更改，而不会覆盖或删除旧的ETL流程信息。为了建立并保持对仓库中信息的信任程度，可以在理想情况下在将来的任何时间点重建仓库中每个单独记录的处理流程。

目的

DECODE逐个将expr与每个search值进行比较。如果expr等于search ，则Oracle数据库返回相应的result 。如果未找到匹配项，则Oracle返回default 。如果省略default ，则Oracle返回null。

参数可以是任何数字类型（ NUMBER ， BINARY_FLOAT或BINARY_DOUBLE ）或字符类型。

  • 如果expr和search是字符数据，那么Oracle使用非填充比较语义对它们进行比较。 expr ， search和result可以是CHAR ， VARCHAR2 ， NCHAR或NVARCHAR2任何数据类型。返回的字符串是VARCHAR2数据类型，与第一个result参数的字符集相同。

  • 如果第一个search-result对是数字，则Oracle会比较所有search-result表达式和第一个expr以确定具有最高数字优先级的参数，隐式将剩余参数转换为该数据类型，并返回该数据类型。

search ， result和default值可以从表达式派生。Oracle数据库使用短路评估 。数据库仅在将其与expr进行比较之前评估每个search值，而不是在将任何search值与expr进行比较之前评估所有search值。因此，如果先前的search等于expr ，Oracle从不评估search 。

甲骨文自动转换expr并且每个search值到第一的数据类型search相比较之前的值。Oracle会自动将返回值转换为与第一个result相同的数据类型。如果第一个result的数据类型为CHAR或者第一个result为null，则Oracle会将返回值转换为数据类型VARCHAR2 。

在DECODE函数中，Oracle认为两个空值是等效的。如果expr为null，则Oracle返回第一个也为null的search result 。

DECODE函数中的最大组件数（包括expr ， searches ， results和default ）为255。

OLAP Java API客户端软件是一组Java包，包含实现Oracle OLAP的Java编程接口的类。具有OLAP选项的Oracle数据库提供OLAP Java API和其他必需的类库作为Java归档（JAR）文件。作为应用程序开发人员，您必须将所需的JAR文件复制到开发Java应用程序的计算机上，或者使其可供开发环境访问。

当Java应用程序调用OLAP Java API对象的方法时，它使用OLAP Java API客户端软件与Oracle OLAP进行通信，Oracle OLAP位于Oracle数据库实例中。OLAP Java API客户端软件和Oracle OLAP之间的通信是通过Java数据库连接（JDBC）API提供的，该API是用于连接到关系数据库的标准Java接口。另一个必需的JAR文件为导入和导出OLAP Java API元数据对象XML提供支持。

要在开发应用程序时使用OLAP Java API类，请将它们导入Java代码中。将应用程序交付给用户时，请在应用程序中包含OLAP Java API类。您还必须确保用户可以访问JDBC。

要开发OLAP Java API应用程序，您必须具有Java Development Kit（JDK），例如Oracle JDeveloper中的一个。用户必须具有Java Runtime Environment（JRE），其版本号与您用于开发的JDK兼容。

A.2必需的类库

您的应用程序开发环境必须包含以下文件：

  • olap_api.jar文件，其中包含OLAP Java API类库。

  • ojdbc6.jar文件，它是一个Oracle JDBC（Java数据库连接）库，包含连接到Oracle数据库实例所需的类。Oracle安装包括JDBC文件。您必须使用该JDBC文件，而不是来自其他Oracle产品的文件或来自其他供应商的产品。

  • xmlparserv2.jar文件，其中包含提供XML解析支持的类。

  • Java Development Kit（JDK）1.6版。Oracle数据库安装不提供JDK。如果您使用Oracle JDeveloper作为开发环境，那么您的计算机上已经安装了JDK。但是，请确保在JDeveloper中使用正确版本的JDK。有关获取和使用其他JDK的信息，请访问Oracle Technology Network Java网站http://www.oracle.com/technetwork/java/index.html 。

目的

FIRST_VALUE是一个分析函数。它返回有序值集中的第一个值。如果集合的第一个值是零，则函数返回NULL ，除非您指定IGNORE NULLS 。此设置对数据密集化很有用。

注意：

此语法的两种形式具有相同的行为。顶部分支是ANSI格式，Oracle建议ANSI兼容。

{ RESPECT | IGNORE } NULLS确定expr空值是否包含在计算中或从计算中消除。默认值为RESPECT NULLS 。如果指定IGNORE NULLS ，则FIRST_VALUE返回集合中的第一个非空值，如果所有值都为null，则返回NULL 。有关数据密集化的示例，请参阅“ 使用分区外连接：示例 ” 。

您不能通过使用FIRST_VALUE或expr任何其他分析函数来嵌套分析函数。但是，您可以为expr使用其他内置函数表达式。有关有效形式的expr信息，请参阅“ 关于SQL表达式 ” 。

例子

以下示例为Department 90中的每个员工选择薪水最低的员工的姓名。

SELECT employee_id，last_name，salary，hire_date，FIRST_VALUE（last_name）OVER（ORDER BY salary ASC ROWS UNBOUNDED PRECEDING）AS fv FROM（SELECT * FROM employees WHERE department_id = 90 ORDER BY hire_date）; EMPLOYEE_ID LAST_NAME SALARY HIRE_DATE FV ----------- ------------------------- --------- -  --------- ------- 102 De Haan 17000 13-JAN-01 De Haan 101 Kochhar 17000 21-SEP-05 De Haan 100 King 24000 17-JUN-03 De Haan

该示例说明了FIRST_VALUE函数的非确定性。Kochhar和DeHaan的工资相同，相邻的行也是如此。Kochhar首先出现，因为子查询返回的行由hire_date排序。但是，如果子查询返回的行按hire_date按降序排序，如下一个示例所示，则函数返回不同的值：

SELECT employee_id，last_name，salary，hire_date，FIRST_VALUE（last_name）OVER（ORDER BY salary ASC ROWS UNBOUNDED PRECEDING）AS fv FROM（SELECT * FROM employees WHERE department_id = 90 ORDER by hire_date DESC ）; EMPLOYEE_ID LAST_NAME SALARY HIRE_DATE FV ----------- ------------------------- --------- -  --------- ------- 101 Kochhar 17000 21-SEP-05 Kochhar 102 De Haan 17000 13-JAN-01 Kochhar 100 King 24000 17-JUN-03 Kochhar

Oracle Database Release 19c引入了一个Java库，它为反应流提取提供支持，使客户能够有效地将数据流式传输到Oracle数据库。

22.1用于Reactive Streams摄取的Java库概述

新的Java库使Java应用程序能够不断地从大量客户端接收和接收数据。需要在Oracle数据库表中以行的形式保留大量记录的IoT代理或Telco应用程序可以使用此库。使用Oracle数据库的直接路径加载方法来插入数据，它使摄取过程无阻塞且速度极快。此外，通过使用通用连接池（UCP），该库还提供了数据库的多种高可用性和可伸缩性功能，如支持表分区，Oracle RAC连接关联和分片。

22.2用于反应流摄取的Java库体系结构

当大量客户端使用数据库以表行的形式保存信息并且不希望被阻塞等待来自数据库的同步响应时，新的Java库使客户能够有效地将数据流式传输到Oracle数据库。该库包含以下模块：

  • 反应流摄取（RSI）库

    这是公开API以处理逻辑的核心模块。它支持Oracle Database Release 19c的Memoptimized RowStore快速摄取功能，可以快速无缝地将用户信息保存在数据库中。它使用UCP进行连接池和管理，包括分片拓扑知识和Oracle Real Application Cluster（RAC）数据库的快速应用程序通知（FAN）感知。

    要使用此库，必须将以下JAR文件添加到CLASSPATH ：
    • fis.jar
    • ojdbc9.jar
    • ucp.jar
    • ons.jar
  • JDBC API

    此模块是增强的JDBC API的集合，它使用OracleConnection接口中提供的新prepareDirectPath方法添加对直接路径加载的支持，该方法创建从PreparedStatement接口派生的新接口。使用此接口对最终用户是透明的，您需要对应用程序代码进行很少的更改以执行直接路径加载操作。

    以下代码段显示了如何使用此功能：

    PreparedStatement preparedStatement = oracleConnection。prepareDirectPath（“SCOTT”，“EMP”，new String [] {“EMPNO”，“ENAME”}）;在循环中{preparedStatement.setInt（1，someIntValue）; preparedStatement.setString（2，someStringValue）：preparedStatement.addBatch（）; prepareStatement.executeBatch（）; oracleConnection.commit（）; //或oracleConnection.rollback（）;
  • UCP图书馆

    此模块是现有UCP API的扩展，它为分片数据库提取分片拓扑并侦听FAN事件。HSI库使用UCP能够正确地重用连接。

    UCP库识别用户指定的分片键，并允许它们连接到特定的分片和块。在创建新连接时，UCP将分片键范围缓存到分片的位置，并允许进一步的连接请求绕过全局服务管理器（分片导向器），从而实现更快的路径密钥访问。UCP还可以通过提供分片键来选择池中的可用连接。这允许通过使用缓存的路由拓扑重新使用到各种分片的连接。

    RSI库使用UCP公开的分片API来获得指定分片键的正确连接。RSI中的每条记录都可以映射到特定的唯一块ID，然后可以使用唯一的块ID将这些记录组合在一起。当RSI库有足够的记录发送到数据库时，它会从UCP借用一个特定于块的连接，并使用该连接将记录插入到分片数据库中。

22.3用于Reactive Streams摄取的Java库限制

在发生意外崩溃或错误的情况下，数据流的反应流摄取可能无法将所有数据摄取到数据库中，因为库在尝试将它们存储在数据库中时可能会丢失一些记录。数据丢失量取决于以下两个值：

  • MaxNoOfRecord

    这是要在内存中缓冲的最大记录数。此值是数据库崩溃时可能丢失的最大记录数。在绩效和潜在的记录损失之间存在权衡。

  • MaxBufferInterval

    这是在内存中保存记录的最长时间。如果此参数的值较低，则内存中可能丢失的记录数也会减少。在绩效和潜在的记录损失之间存在权衡。

除了由于数据库崩溃导致数据丢失的可能性之外，该库还具有以下限制：

  • 不支持触发器。
  • 不检查参照完整性。
  • 不支持群集表。
  • 不支持加载远程对象。
  • 必须将LONG数据类型指定为最后一列。
  • 不支持包含流的LONG数据类型
  • 不支持加载VARRAY列。
  • 所有分区列必须出现在任何LOB数据类型之前。

采用非CDB作为PDB

您可以使用DBMS_PDB.DESCRIBE过程将非CDB（移动）到PDB中。

要使用DBMS_PDB包将非CDB用作PDB：

  1. 如果CDB不存在，请创建CDB。

  2. 确保非CDB处于事务一致状态。

  3. 将非CDB置于只读模式。

  4. 连接到非CDB，并运行DBMS_PDB.DESCRIBE过程以构造描述非CDB的XML文件。

    当前用户必须具有SYSDBA管理权限。用户必须在连接时使用AS SYSDBA行使权限。

    例如，要在/disk1/oracle目录中生成名为ncdb.xml的XML文件，请运行以下过程：

    BEGIN DBMS_PDB.DESCRIBE（pdb_descr_file =>'/ disk1 / oracle / ncdb.xml'）;结束; /

    该过程成功完成后，您可以使用XML文件和非CDB数据库文件将非CDB插入CDB。

  5. 运行DBMS_PDB.CHECK_PLUG_COMPATIBILITY函数以确定非CDB是否与CDB兼容。

    运行该功能时，请设置以下参数：

    • pdb_descr_file - 将此参数设置为XML文件的完整路径。

    • pdb_name - 指定新PDB的名称。如果省略此参数，则使用XML文件中的PDB名称。

    例如，要确定/disk1/oracle/ncdb.xml文件描述的非CDB是否与当前CDB兼容，请运行以下PL / SQL块：

    SET SERVEROUTPUT ON DECLARE兼容的CONSTANT VARCHAR2（3）：= CASE DBMS_PDB.CHECK_PLUG_COMPATIBILITY（pdb_descr_file =>'/ disk1 / oracle / ncdb.xml'，pdb_name =>'NCDB'）当为“是”时，“否”结束; BEGIN DBMS_OUTPUT.PUT_LINE（兼容）;结束; /

    如果输出为YES ，则非CDB兼容，您可以继续下一步。如果输出为NO ，则非CDB不兼容，您可以检查PDB_PLUG_IN_VIOLATIONS视图以查看它不兼容的原因。在继续之前，必须纠正所有违规行为。例如，应通过运行升级或datapatch实用程序来解决任何版本或修补程序不匹配问题。纠正违规后，再次运行DBMS_PDB.CHECK_PLUG_COMPATIBILITY以确保非CDB与CDB兼容。

  6. 关闭非CDB。

  7. 插入非CDB。

    例如，以下SQL语句插入非CDB，将其文件复制到新位置，并且仅包含非CDB中的tbs3用户表空间：

    创建可插入的数据库ncdb使用'/disk1/oracle/ncdb.xml'COPY FILE_NAME_CONVERT =（'/ disk1 / oracle / dbs /'，'/ disk2 / oracle / ncdb /'）USER_TABLESPACES =（'tbs3'）;

    如果没有违规，则不要打开新的PDB。您将在以下步骤中打开它。

    1. 访问PDB。

      @ $ ORACLE_HOME / RDBMS /管理/ noncdb_to_pdb.sql

    该脚本在更改完成后打开PDB，执行更改并关闭PDB。

  9. 以读/写模式打开新PDB。

    您必须在Oracle数据库的读/写模式下打开新的PDB，才能完成新PDB与CDB的集成。如果尝试以只读模式打开PDB，则会返回错误。在读/写模式下打开PDB后，其状态为NORMAL 。

  10. 备份PDB。

    除非备份PDB，否则无法恢复PDB。

创建锁定概况

目的

使用CREATE LOCKDOWN PROFILE语句创建PDB锁定配置文件。您可以在多租户容器数据库（CDB）中使用PDB锁定配置文件来限制PDB中的用户操作。

创建PDB锁定配置文件后，可以使用ALTER LOCKDOWN PROFILE语句向配置文件添加限制。您可以限制与某些数据库功能，选项和SQL语句关联的用户操作。

将锁定配置文件分配给PDB时，该PDB中的用户无法执行对配置文件禁用的操作。要分配锁定配置文件，请为PDB_LOCKDOWN初始化参数的值设置其名称。您可以为单个PDB或CDB或应用程序容器中的所有PDB分配锁定配置文件，如下所示：

  • 如果在连接到CDB根目录时设置PDB_LOCKDOWN ，则锁定配置文件将应用于CDB中的所有PDB。它不适用于CDB根目录。

  • 如果在连接到应用程序根目录时设置PDB_LOCKDOWN ，则锁定配置文件将应用于应用程序根目录和应用程序容器中的所有PDB。

  • 如果在连接到特定PDB时设置PDB_LOCKDOWN ，则锁定配置文件将应用于该PDB并覆盖CDB或应用程序容器（如果存在）的锁定配置文件。

也可以看看：

  • ALTER LOCKDOWN PROFILE和DROP LOCKDOWN PROFILE

  • 有关PDB锁定配置文件的更多信息，请参见“Oracle数据库安全指南

DBMS_JOB和Oracle Scheduler共享相同的作业协调器和作业从属，它们都由JOB_QUEUE_PROCESSES参数控制。

如果在非CDB或CDB根目录中将JOB_QUEUE_PROCESSES的值设置为0，则DBMS_JOB作业和Oracle Scheduler作业将不会在非CDB或根目录中运行。

如果在非CDB或CDB根目录中将JOB_QUEUE_PROCESSES设置为1到4000的值，则将运行DBMS_JOB作业和Oracle Scheduler作业。为计划程序自动调整为Oracle Scheduler作业创建的实际作业从站数，具体取决于多种因素，包括可用资源，资源管理器设置和当前正在运行的作业。但是，在实例上运行DBMS_JOB作业和Oracle Scheduler作业的作业从属总数不能超过该实例的JOB_QUEUE_PROCESSES值。

物化视图使用Oracle Scheduler进行自动刷新。将JOB_QUEUE_PROCESS设置为0将禁用此功能以及使用Oracle Scheduler或DBMS_JOB任何其他功能。

6.1关于异常检测

异常检测的目标是识别在看似同质的数据中不寻常的情况。异常检测是检测欺诈，网络入侵和其他罕见事件的重要工具，这些事件具有重要意义但很难找到。

异常检测可用于解决以下问题：

  • 执法机构汇编有关非法活动的数据，但不包括合法活动的数据。如何标记可疑活动？

    执法数据全部属于一类。没有反例。

  • 一家保险机构处理数百万份保险索赔，因为他们知道极少数是欺诈行为。如何识别欺诈性索赔？

    索赔数据包含非常少的反例。它们是异常值。

6.1.1一级分类

了解异常检测作为训练数据中的一类分类。

异常检测是一种分类形式。异常检测实现为一类分类，因为在训练数据中只表示一个类。异常检测模型预测数据点是否对于给定分布是典型的。非典型数据点可以是异常值，也可以是先前看不见的类的示例。

通常，必须对包含每个类的示例和反例的数据训练分类模型，以便模型可以学习区分它们。例如，预测药物副作用的模型必须接受包括对药物的广泛响应的数据的训练。

一类分类器开发通常描述训练数据中的典型情况的简档。偏离轮廓被识别为异常。一类分类器有时被称为积极安全模型，因为它们试图识别“好”行为并假设所有其他行为都是坏的。

注意：

解决一类分类问题可能很困难。单类分类器的准确性通常不能与使用有意义的反例建立的标准分类器的准确性相匹配。

异常检测的目标是在以前无法获得信息的情况下提供一些有用的信息。但是，如果有足够的“罕见”案例那样分层抽样产生一个训练集，其中有足够的反例用于标准分类模型，那么这通常是一个更好的解决方案。

6.1.2单类数据的异常检测

在单类数据中，所有情况都具有相同的分类。反例，另一个类的实例很难指定或收集昂贵。例如，在文本文档分类中，很容易在给定主题下对文档进行分类。但是，本主题之外的文档范围可能非常大且多样。因此，指定其他类型的文档作为反例是不可行的。

异常检测可用于查找特定类型文档的异常实例。

6.1.3寻找异常值的异常检测

异常值是不常见的情况，因为它们超出了数据正常的分布范围。例如，人口普查数据显示，家庭收入中位数为70,000美元，家庭平均收入为80,000美元，但一个或两个家庭的收入为20万美元。这些案例可能被识别为异常值。

距正态分布中心的距离表示给定点相对于数据分布的典型程度。每个案例都可以根据它是典型的还是非典型的概率进行排名。

异常值的存在会对许多形式的数据挖掘产生有害影响。在挖掘数据之前，您可以使用异常检测来识别大纲。

5.1关于异常检测

异常检测的目标是识别在看似同质的数据中不寻常的情况。异常检测是检测欺诈，网络入侵和其他罕见事件的重要工具，这些事件具有重要意义但很难找到。

异常检测可用于解决以下问题：

  • 执法机构汇编有关非法活动的数据，但不包括合法活动的数据。如何标记可疑活动？

    执法数据全部属于一类。没有反例。

  • 一家保险机构处理数百万份保险索赔，因为他们知道极少数是欺诈行为。如何识别欺诈性索赔？

    索赔数据包含非常少的反例。它们是异常值。

5.1.1一级分类

了解异常检测作为训练数据中的一类分类。

异常检测是一种分类形式。异常检测实现为一类分类，因为在训练数据中只表示一个类。异常检测模型预测数据点是否对于给定分布是典型的。非典型数据点可以是异常值，也可以是先前看不见的类的示例。

通常，必须对包含每个类的示例和反例的数据训练分类模型，以便模型可以学习区分它们。例如，预测药物副作用的模型必须接受包括对药物的广泛响应的数据的训练。

一类分类器开发通常描述训练数据中的典型情况的简档。偏离轮廓被识别为异常。一类分类器有时被称为积极安全模型，因为它们试图识别“好”行为并假设所有其他行为都是坏的。

注意：

解决一类分类问题可能很困难。单类分类器的准确性通常不能与使用有意义的反例建立的标准分类器的准确性相匹配。

异常检测的目标是在以前无法获得信息的情况下提供一些有用的信息。但是，如果有足够的“罕见”案例那样分层抽样产生一个训练集，其中有足够的反例用于标准分类模型，那么这通常是一个更好的解决方案。

5.1.2单类数据的异常检测

在单类数据中，所有情况都具有相同的分类。反例，另一个类的实例很难指定或收集昂贵。例如，在文本文档分类中，很容易在给定主题下对文档进行分类。但是，本主题之外的文档范围可能非常大且多样。因此，指定其他类型的文档作为反例是不可行的。

异常检测可用于查找特定类型文档的异常实例。

5.1.3寻找异常值的异常检测

异常值是不常见的情况，因为它们超出了数据正常的分布范围。例如，人口普查数据显示，家庭收入中位数为70,000美元，家庭平均收入为80,000美元，但一个或两个家庭的收入为20万美元。这些案例可能被识别为异常值。

距正态分布中心的距离表示给定点相对于数据分布的典型程度。每个案例都可以根据它是典型的还是非典型的概率进行排名。

异常值的存在会对许多形式的数据挖掘产生有害影响。在挖掘数据之前，您可以使用异常检测来识别大纲。

DROP INDEX

目的

使用DROP INDEX语句从数据库中删除索引或域索引。

删除全局分区索引，范围分区索引或散列分区索引时，也会删除所有索引分区。如果删除复合分区索引，则也会删除所有索引分区和子分区。

此外，当您删除域索引时：

  • Oracle数据库调用适当的例程。

  • 如果任何统计信息与域索引相关联，则Oracle数据库会将统计信息类型与FORCE子句解除关联，并删除使用统计信息类型收集的用户定义统计信息。

    也可以看看：

    • CREATE INDEX和ALTER INDEX以获取有关创建和修改索引的信息

    • 有关域索引的更多信息，请参阅CREATE INDEX的domain_index_clause

    • 关于统计类型关联的更多信息，关联统计和DISASSOCIATE STATISTICS

先决条件

索引必须位于您自己的架构中，或者您必须具有DROP ANY INDEX系统特权。

说明drop_index.eps的描述

语义

模式

指定包含索引的架构。如果省略schema ，则Oracle数据库假定索引位于您自己的模式中。

指数

指定要删除的索引的名称。删除索引时，分配给索引的所有数据块都将返回到包含索引的表空间。

线上

指定ONLINE以指示在删除索引时将允许对表或分区执行DML操作。

力

FORCE仅适用于域索引。即使indextype例程调用返回错误或索引标记为IN PROGRESS此子句也会删除域索引。如果没有FORCE ，如果其索引类型例程调用返回错误或索引标记为IN PROGRESS ，则不能删除域索引。

{DEFERRED | IMMEDIATE}无效

此子句允许您控制数据库何时在删除索引时使相关游标无效。它具有与ALTER INDEX语句相同的语义，并添加了以下内容：当您删除带有DEFERRED INVALIDATION的索引时，Oracle数据库将立即使其计划中引用已删除索引的任何DML语句或查询无效。

见{DEFERRED | IMMEDIATE}有关此子句的完整语义的ALTER INDEX文档中的INVALIDATION 。

删除索引的限制

以下限制适用于删除索引：

  • 如果索引或其任何索引分区标记为IN_PROGRESS则无法删除域索引。

  • 删除域索引，集群索引或队列表上的索引时，无法指定ONLINE子句。
