16管理撤消

对于默认安装，Oracle数据库会自动管理撤消。通常不需要DBA干预。但是，如果您的安装使用Oracle闪回操作，则可能需要执行一些撤消管理任务以确保这些操作成功。

  • 什么是撤消？
    Oracle数据库创建和管理用于回滚或撤消对数据库的更改的信息。这些信息包括交易行为的记录，主要是在交易之前。这些记录统称为撤消 。
  • 自动撤消管理简介
    Oracle数据库可以自动管理撤消信息和空间。
  • 设置最小撤消保留期限
    您可以通过设置UNDO_RETENTION初始化参数来指定最小撤消保留期限（以秒为单位）。
  • 调整固定大小的撤消表空间
    使用固定大小的撤消表空间，自动调整撤消保留通常可以获得更好的结果。如果您决定使用固定大小的撤消表空间，则Undo Advisor可以帮助您估算所需的容量。
  • 管理撤消表空间
    您可以通过完成创建，更改和删除任务来管理撤消表空间。您还可以切换撤消表空间并为撤消空间建立用户配额。
  • 迁移到自动撤消管理
    如果您当前正在使用回滚段来管理撤消空间，Oracle强烈建议您将数据库迁移到自动撤消管理。
  • 管理临时撤消
    默认情况下，临时表的撤消记录存储在撤消表空间中，并以重做方式记录，这与对持久表管理撤消的方式相同。但是，您可以使用TEMP_UNDO_ENABLED初始化参数将临时表的撤消与永久表的撤消分开。当此参数设置为TRUE ，临时表的撤消称为临时撤消 。
  • 撤消空间数据字典视图
    您可以在自动撤消管理模式下查询一组视图，以获取有关撤消空间的信息。

16.2.1自动撤消管理概述

Oracle提供了一种全自动机制，称为自动撤消管理，用于管理撤消信息和空间。通过自动撤消管理，数据库管理撤消表空间中的还原段。

自动撤消管理是新安装的数据库的默认模式。使用Database Configuration Assistant（DBCA）创建数据库时，将自动创建名为UNDOTBS1的自动扩展撤消表空间。

您还可以显式创建撤消表空间。“ 创建撤消表空间 ”中介绍了创建撤消表空间的方法 。

数据库实例启动时，数据库会自动选择第一个可用的撤消表空间。如果没有可用的撤消表空间，则实例将在没有撤消表空间的情况下启动，并将撤消记录存储在SYSTEM表空间中。建议不要这样做，并且会向警报日志文件写入警报消息，以警告系统在没有撤消表空间的情况下运行。

如果数据库包含多个撤消表空间，则可以选择在启动时指定要使用特定的撤消表空间。这是通过设置UNDO_TABLESPACE初始化参数来完成的，如下例所示：

UNDO_TABLESPACE = undotbs_01

如果初始化参数中指定的表空间不存在，则STARTUP命令失败。UNDO_TABLESPACE参数可用于将特定的撤消表空间分配给Oracle Real Application Clusters环境中的实例。

数据库也可以在手动撤消管理模式下运行 。在此模式下，撤消空间通过回滚段进行管理，不使用撤消表空间。

注意：

回滚段的空间管理很复杂。Oracle强烈建议将数据库保留为自动撤消管理模式。

16.2.2.1关于撤消保留期限

启用自动撤消管理时，始终存在当前的撤消保留期 ，这是Oracle数据库在覆盖之前尝试保留旧撤消信息的最短时间。

提交事务后，不再需要撤消数据进行回滚或事务恢复。但是，出于一致的读取目的，长时间运行的查询可能需要此旧的撤消信息来生成数据块的旧图像。此外，几个Oracle闪回功能的成功还取决于旧的撤消信息的可用性。由于这些原因，希望尽可能长时间地保留旧的撤销信息。

旧的（已提交的）早于当前撤消保留期的撤消信息据说已过期 ，其空间可供新事务覆盖。具有小于当前撤消保留期的年龄的旧撤消信息被认为是未过期的并且被保留以用于一致的读取和Oracle闪回操作。

Oracle数据库根据撤消表空间大小和系统活动自动调整撤消保留期。您可以通过设置UNDO_RETENTION初始化参数来指定最小撤消保留期（以秒为单位）。此参数对撤消保留的确切影响如下：

  • 对于固定大小的撤消表空间，将忽略UNDO_RETENTION参数。数据库始终根据系统活动和撤消表空间大小调整撤消保留期以获得最佳保留期。有关详细信息，请参阅“ 撤消保留的自动调整 ” 。

  • 对于启用了AUTOEXTEND选项的撤消表空间，数据库会尝试遵守UNDO_RETENTION指定的最短保留期。当空间不足时，表空间会自动扩展，而不是覆盖未过期的撤消信息。如果为自动扩展的撤消表空间指定了MAXSIZE子句，则在达到最大大小时，数据库可能会开始覆盖未过期的撤消信息。由DBCA自动创建的UNDOTBS1表空间是自动扩展的。

16.2.2.2撤消保留的自动调整

Oracle数据库根据撤消表空间的配置方式自动调整撤消保留期。

  • 如果使用AUTOEXTEND选项配置撤消表空间，则数据库会动态调整撤消保留期，使其比系统上运行时间最长的活动查询稍长。但是，此保留期可能不足以适应Oracle闪回操作。导致snapshot too old错误的Oracle闪回操作表明您必须进行干预以确保保留足够的撤消数据以支持这些操作。为了更好地适应Oracle闪回功能，您可以将UNDO_RETENTION参数设置为等于预期最长的Oracle闪回操作的值，也可以将撤消表空间更改为固定大小。

  • 如果撤消表空间是固定大小，则数据库会动态调整撤消保留期，以便为该表空间大小和当前系统负载提供最佳保留。此最佳可能保留时间通常显着大于运行时间最长的活动查询的持续时间。

    如果您决定将撤消表空间更改为固定大小，则必须选择足够大的表空间大小。如果选择的撤消表空间大小太小，则可能发生以下两个错误：

    • DML可能会失败，因为没有足够的空间来容纳新事务的撤消。

    • 长时间运行的查询可能会失败，一个snapshot too old的错误，这意味着有读一致性还原数据不足。

    有关详细信息，请参阅“ 调整固定大小的撤消表空间 ” 。

注意：

LOB不支持自动调整撤消保留。这是因为LOB的撤消信息存储在段本身中而不是存储在撤消表空间中。对于LOB，数据库尝试遵守UNDO_RETENTION指定的最小撤消保留期。但是，如果空间变小，则可能会覆盖未到期的LOB撤消信息。

16.2.2.3保留保证

为了保证长时间运行的查询或Oracle闪回操作的成功，您可以启用保留保证。

如果启用了保留保证，则保证指定的最小撤消保留;数据库永远不会覆盖未过期的撤消数据，即使这意味着由于撤消表空间中的空间不足导致事务失败。如果未启用保留保证，则数据库可以在空间不足时覆盖未过期的撤消，从而降低系统的撤消保留。默认情况下禁用此选项。

警告：

启用保留保证可能导致多个DML操作失败。谨慎使用。

通过使用CREATE DATABASE或CREATE UNDO TABLESPACE语句创建撤消表空间的RETENTION GUARANTEE子句，可以启用保留保证。或者，您可以稍后在ALTER TABLESPACE语句中指定此子句。您可以使用RETENTION NOGUARANTEE子句禁用保留保证。

您可以使用DBA_TABLESPACES视图来确定撤消表空间的保留保证设置。名为RETENTION的列包含值GUARANTEE ， NOGUARANTEE或NOT APPLY ，其中NOT APPLY用于除undo表空间之外的表空间。

16.2.2.4撤消保留调整和警报阈值

对于固定大小的撤消表空间，数据库根据数据库统计信息和撤消表空间的大小计算最佳保留期。

为了实现最佳的撤消管理，数据库不是基于表空间大小的100％进行调整，而是根据表空间大小的85％调整撤消保留期，或者根据使用的空间的警告警报阈值百分比调整，以较低者为准。（警告警报阈值默认为85％，但可以更改。）因此，如果将撤消表空间的警告警报阈值设置为低于85％，则可能会减小撤消保留期的调整大小。有关表空间警报阈值的详细信息，请参阅“ 管理表空间警报 ” 。

16.4调整固定大小的撤消表空间大小

使用固定大小的撤消表空间，自动调整撤消保留通常可以获得更好的结果。如果您决定使用固定大小的撤消表空间，则Undo Advisor可以帮助您估算所需的容量。

要使用Undo Advisor，首先要估算这两个值：

  • 预期运行时间最长的查询的长度

  • Oracle闪回操作所需的最长时间间隔

    例如，如果您希望过去运行最多48小时的Oracle闪回查询，则您的Oracle闪回要求为48小时。

然后，您将获取这两个值的最大值，并将该值用作Undo Advisor的输入。

16.5.4切换撤消表空间

您可以从使用一个撤消表空间切换到另一个。由于UNDO_TABLESPACE初始化参数是动态参数，因此ALTER SYSTEM SET语句可用于分配新的撤消表空间。

以下语句切换到新的undo表空间：

ALTER SYSTEM SET UNDO_TABLESPACE = undotbs_02;

假设undotbs_01是当前的undo表空间，在此命令成功执行后，实例使用undotbs_02代替undotbs_01作为其undo表空间。

如果存在切换到的表空间存在以下任何条件，则会报告错误并且不会发生切换：

  • 表空间不存在

  • 表空间不是撤消表空间

  • 表空间已被另一个实例使用（仅在Oracle RAC环境中）

执行切换操作时数据库处于联机状态，并且在执行此命令时可以执行用户事务。当切换操作成功完成时，在切换操作开始后启动的所有事务都将分配给新的撤消表空间中的事务表。

切换操作不会等待旧的撤消表空间中的事务提交。如果旧的撤消表空间中存在任何挂起的事务，则旧的撤消表空间将进入PENDING OFFLINE模式（状态）。在此模式下，现有事务可以继续执行，但新用户事务的撤消记录不能存储在此撤消表空间中。

即使在切换操作成功完成后，撤销表空间也可以在此PENDING OFFLINE模式下存在。PENDING OFFLINE undo tablespace不能被其他实例使用，也不能被删除。最终，在提交所有活动事务后，撤消表空间自动从PENDING OFFLINE模式进入OFFLINE模式。从那时起，撤消表空间可用于其他实例（在Oracle Real Application Cluster环境中）。

如果UNDO TABLESPACE的参数值设置为''（两个单引号），则切换当前的撤消表空间，并切换下一个可用的撤消表空间。请谨慎使用此语句，因为可能没有可用的撤消表空间。

以下示例取消分配当前的undo表空间：

ALTER SYSTEM SET UNDO_TABLESPACE ='';

16.7.1关于管理临时撤消

临时撤消记录存储在数据库的临时表空间中，因此不会记录在重做日志中。启用临时撤消后，临时表空间使用的某些段会存储临时撤消，这些段称为临时撤消段 。

启用临时撤消后，可能需要增加临时表空间的大小以考虑撤消记录。

启用临时撤消可带来以下好处：

  • 临时撤消可以减少撤消表空间中存储的撤消量。

    撤消表空间中的较少撤消可以导致撤消记录的更实际的撤消保留期要求。

  • 临时撤消会减小重做日志的大小。

    性能得到改善，因为重做日志中写入的数据较少，而解析重做日志记录的组件（如LogMiner）性能更好，因为要解析的重做数据较少。

  • 临时撤消使用Oracle Active Data Guard选项对物理备用数据库中的临时表启用数据操作语言（DML）操作。但是，必须在主数据库上发出创建临时表的数据定义语言（DDL）操作。

您可以为特定会话或整个系统启用临时撤消。使用ALTER SESSION语句为会话启用临时撤消时，会话将创建临时撤消，而不会影响其他会话。使用ALTER SYSTEM语句为系统启用临时撤消时，所有现有会话和新会话都会创建临时撤消。

当会话第一次使用临时对象时， TEMP_UNDO_ENABLED会话的其余部分设置TEMP_UNDO_ENABLED初始化参数的当前值。因此，如果为会话启用了临时撤消，并且会话使用临时对象，则无法为会话禁用临时撤消。同样，如果为会话禁用临时撤消并且会话使用临时对象，则无法为会话启用临时撤消。

对于具有Oracle Active Data Guard选项的物理备用数据库，默认情况下启用临时撤消。由于默认设置， TEMP_UNDO_ENABLED初始化参数对具有Active Data Guard选项的物理备用数据库没有影响。

仅当数据库的兼容级别为12.0.0或更高时，才能启用临时撤消。

3.3关于RMAN频道

RMAN信道表示到设备的一个数据流，并且对应于一个数据库服务器会话。在备份或恢复操作期间，通道从输入设备读取数据，对其进行处理，并将其写入输出设备。

RMAN客户端指示数据库服务器会话执行所有备份和恢复任务。会话的构成取决于操作系统。例如，在Linux上，服务器会话对应于服务器进程，而在Windows上，它对应于数据库服务中的线程。RMAN客户端本身不执行备份，还原或恢复操作。

大多数RMAN命令由通道执行，通道必须配置为跨RMAN会话保持，或者在每个RMAN会话中手动分配。如图3-2所示 ，通道通过在实例上启动服务器会话，建立从RMAN客户端到目标或辅助数据库实例的连接。

3.3.2关于RMAN自动和手动通道

RMAN可以使用自动通道或手动通道进行备份和恢复操作。

您可以使用CONFIGURE CHANNEL命令配置跨RMAN会话使用磁盘或磁带的通道。这种技术称为自动信道分配 。RMAN预先配置了一个DISK通道，可用于备份到磁盘。

当您运行可以使用自动通道的命令时，RMAN会自动使用您在CONFIGURE命令中指定的选项分配通道。对于BACKUP命令，RMAN仅分配备份到指定媒体所需的通道类型。对于RESTORE命令和 RMAN维护命令 ，RMAN为执行命令所需的设备类型分配所有必需的通道。RMAN确定自动通道的名称。

您也可以手动分配频道。每个手动分配的通道使用与数据库的单独连接。手动分配通道时，为其指定用户定义的名称，例如dev1或ch2 。

运行命令时可用于设备的通道数确定RMAN在执行命令时是否并行读取或写入此设备。当工作并行完成时，文件的备份由多个通道完成。每个通道可以备份多个文件，但除非执行多节备份 ，否则不会有多个文件备份任何文件。

3.4关于RMAN存储库

RMAN存储库是有关RMAN用于备份，恢复和维护的目标数据库的元数据的集合。

RMAN始终将其元数据存储在控制文件中。控制文件中此元数据的版本是数据库的RMAN备份的权威记录。这是保护控制文件是备份策略的重要部分的一个原因。RMAN可以仅使用控制文件执行所有必要的备份和恢复操作，以存储RMAN存储库信息，并维护满足配置的保留策略所需的所有记录。

您还可以创建恢复目录 ，该目录是存储在Oracle数据库模式中的RMAN元数据的存储库。控制文件的备份活动记录空间有限，而恢复目录可以存储更长的历史记录。您可以通过创建包含所有数据库的RMAN元数据的单个恢复目录来简化备份和恢复管理。

恢复目录的所有者可以向其他数据库用户授予或撤消对目录的受限访问。每个受限用户都对其自己的元数据具有完全读/写访问权限，称为虚拟专用目录 。当数据库中存在一个或多个虚拟专用目录时，该数据库仅包含一组目录表。这些表归基本恢复目录所有者所有。基本恢复目录的所有者控制每个虚拟专用目录用户可以访问的数据库。

某些RMAN功能仅在您使用恢复目录时才起作用。例如，您可以在恢复目录中创建存储的脚本 ，并使用此脚本执行RMAN作业。其他RMAN命令与管理恢复目录特别相关，因此如果RMAN未连接到恢复目录，则不可用（并且不需要）。

恢复目录仅由RMAN维护。目标数据库实例永远不会直接访问目录。在更新存储库的任何操作之后，以及在某些操作之前，RMAN会将有关数据库结构，归档重做日志，备份集和数据文件副本的信息传播到目标数据库控制文件的恢复目录中。

3.5.1关于RMAN与媒体管理器的交互

在执行备份或还原到媒体管理器之前，必须分配一个或多个通道来处理与媒体管理器的通信。您还可以为媒体管理器配置默认通道。默认通道用于使用媒体管理器且尚未明确分配通道的所有备份和恢复任务。

RMAN不会发出特定命令来加载，标记或卸载磁带。备份时，RMAN为媒体管理器提供字节流，并将唯一名称与此流关联。当RMAN必须还原备份时，它会要求介质管理器检索字节流。存储流的方式和位置的所有细节完全由媒体管理器处理。例如，媒体管理器标记并跟踪每个磁带上的磁带和文件名，并自动加载和卸载磁带，或向操作员发出信号。

某些媒体管理器支持代理复制功能，它们处理数据文件和备份设备之间的整个数据移动。这些产品可能使用诸如存储和媒体子系统之间的高速连接之类的技术来减少主数据库服务器上的负载。RMAN提供需要备份或还原到媒体管理器的文件列表，这反过来会做出有关如何以及何时移动数据的所有决策。

3.6关于快速恢复区

快速恢复区域是可选的磁盘位置，可用于存储与恢复相关的文件。

创建不同备份和恢复相关文件的组件不了解彼此或存储数据的文件系统的大小。通过基于磁盘的自动备份和恢复，您可以创建快速恢复区域 （也称为恢复区域 ），从而自动管理与备份相关的文件。

快速恢复区域最大限度地减少了为备份相关文件手动管理磁盘空间的需要，并平衡了不同类型文件之间的空间使用。通过这种方式，快速恢复区域简化了数据库的持续管理。Oracle建议您启用恢复区域以简化备份管理。

创建恢复区域时，在磁盘上选择一个位置并设置存储空间的上限。您还可以设置备份保留策略 ，该策略控制恢复所需的备份文件的时长。数据库管理用于备份，存档重做日志以及此空间内数据库的其他恢复相关文件的存储。当RMAN必须回收新文件的空间时，不再需要的文件有资格删除。

3.7.2.1关于Data Guard环境中备份的可互换性

RMAN命令使用恢复目录元数据在Data Guard环境中的不同物理数据库之间透明地运行。例如，您可以备份物理备用数据库上的表空间，并在主数据库上还原和恢复它。同样，您可以备份主数据库上的表空间，并在物理备用数据库上还原和恢复它。

备用控制文件和非备用控制文件的备份是可互换的。例如，您可以还原主数据库上的备用控制文件和物理备用数据库上的主控制文件。这种可互换性意味着您可以将控制文件备份卸载到Data Guard环境中的一个数据库。RMAN在数据库的还原和恢复期间自动更新数据库文件的文件名。

3.7.2.2关于Data Guard环境中备份的关联

恢复目录通过将每个数据库文件或备份文件与DB_UNIQUE_NAME相关联来跟踪Data Guard环境中的文件。创建文件的数据库与该文件相关联。

例如，如果RMAN使用唯一名称standby1备份数据库，则standby1与此备份关联。除非您使用CHANGE ，否则备份仍与创建它的数据库相关联...RESET DB_UNIQUE_NAME命令将备份与其他数据库关联。

3.7.2.3关于Data Guard环境中备份的可访问性

备份的可访问性与其关联不同。在Data Guard环境中，恢复目录将磁盘备份视为只能与它们关联的数据库访问，而在一个数据库上创建的磁带备份可供所有数据库访问。

如果备份文件未与任何数据库关联，则在恢复目录视图中描述它的行对于SITE_KEY列显示为null 。默认情况下，RMAN将SITE_KEY为null的文件与它们所连接的数据库关联为TARGET 。

RMAN命令（如BACKUP ， RESTORE和CROSSCHECK可用于任何可访问的备份。例如，对于RECOVER COPY操作，RMAN仅将与数据库关联的映像副本视为可以恢复。RMAN认为磁盘和磁带上的增量备份有资格恢复映像副本。在数据库恢复中，RMAN仅考虑与数据库关联的磁盘备份以及磁带上的所有文件都有资格恢复。

为了说明备份可访问性的差异，假设数据库prod和standby1驻留在不同的主机上。RMAN将prod上的数据文件1备份到生产主机上的/prmhost/disk1/df1.dbf ，也备份到磁带。RMAN备份数据文件1上standby1到/sbyhost/disk2/df1.dbf备用主机上，并且还到磁带上。如果RMAN连接到数据库prod ，则无法使用RMAN命令对备用主机上的/sbyhost/disk2/df1.dbf备份执行操作。然而，RMAN会考虑上所做的磁带备份standby1有资格得到恢复。

注意：

您可以将备份从备用主机传输到主要主机，反之亦然，将TARGET连接到此主机上的数据库，然后使用CATALOG命令对备份进行编目。在目标数据库对文件进行编目之后，该文件将与目标数据库关联。

除了支持标准JDBC应用程序编程接口（API）之外，Oracle驱动程序还具有支持Oracle特定数据类型和增强性能的扩展。

Oracle提供以下JDBC驱动程序：

  • 瘦司机

    该 JDBC Thin驱动程序是纯Java，Type IV驱动程序，可以在应用程序中使用。它独立于平台，在客户端不需要任何其他Oracle软件。JDBC Thin驱动程序使用Oracle Net Services与服务器通信以访问Oracle数据库。

    JDBC Thin驱动程序通过提供Oracle Net Services的实现来实现与数据库的直接连接 Java套接字。该驱动程序支持TCP / IP协议，并且需要数据库服务器上的TCP / IP套接字上的TNS侦听器。

    注意：

    除非您具有仅由特定驱动程序支持的功能，否则Oracle建议您使用Thin驱动程序。

  • Oracle调用接口（OCI）驱动程序

    它在客户端用于Oracle客户端安装。它只能用于应用程序。

    该 JDBC OCI驱动程序是与Java应用程序一起使用的Type II驱动程序。它需要特定于平台的OCI库。它支持所有已安装的Oracle Net适配器，包括进程间通信（IPC），命名管道，TCP / IP和Internetwork Packet Exchange / Sequenced Packet Exchange（IPX / SPX）。

    JDBC OCI驱动程序使用Java和C的组合编写，使用本机方法调用C-entry点，将JDBC调用转换为对OCI的调用。这些调用使用Oracle Net Services与数据库通信。

    JDBC OCI驱动程序在安装它的客户端计算机上使用OCI库，C-entry点，Oracle Net，核心库和其他必需文件。

    OCI是一种API，使您能够创建使用第三代语言的本机过程或函数调用的应用程序来访问Oracle数据库并控制SQL语句处理的所有阶段。

  • 服务器端瘦驱动程序

    它在功能上类似于客户端瘦驱动程序。但是，它用于在数据库服务器上运行的代码，并且需要访问同一服务器或任何层上的远程服务器上的另一个会话。

    该 JDBC服务器端Thin驱动程序提供与在客户端运行的JDBC Thin驱动程序相同的功能。但是，JDBC服务器端Thin驱动程序在Oracle数据库中运行，并访问远程数据库或同一数据库上的不同会话，以便与数据库中的Java一起使用。

    此驱动程序在以下方案中很有用：

    • 从充当中间层的Oracle数据库实例访问远程数据库服务器

    • 从另一个内部访问Oracle数据库会话，例如从Java存储过程访问

    从客户端应用程序或从服务器内部使用JDBC Thin驱动程序不会影响代码。

  • 服务器端内部驱动程序

    它用于在数据库服务器上运行并访问同一会话的代码。也就是说，代码运行并访问单个Oracle会话中的数据。

    该 JDBC服务器端内部驱动程序支持在Oracle数据库中运行的任何Java代码，例如在Java存储过程中，并访问同一数据库。它允许Oracle Java虚拟机（Oracle JVM）直接与SQL引擎通信，以便与数据库中的Java一起使用。

    JDBC服务器端内部驱动程序，Oracle JVM，数据库和SQL引擎都在同一地址空间内运行，因此，网络往返问题无关紧要。程序通过使用函数调用来访问SQL引擎。

    注意：

    服务器端内部驱动程序不支持Statement类的cancel和setQueryTimeout方法。

    JDBC服务器端内部驱动程序与客户端驱动程序完全一致，并支持相同的功能和扩展。

1.2选择合适的驱动程序

为应用程序或applet选择JDBC驱动程序时，请考虑以下事项：

  • 通常，除非您需要特定于OCI的功能，例如支持非TCP / IP网络，否则请使用JDBC Thin驱动程序。

  • 如果您想要最大的可移植性和性能，请使用JDBC Thin驱动程序。您可以使用JDBC Thin驱动程序从应用程序连接到Oracle数据库。

  • 如果要在安全套接字层（SSL）/传输层安全性（TLS）上使用轻量级目录访问协议（LDAP），请使用JDBC Thin驱动程序。

  • 如果要为Oracle客户端环境编写客户端应用程序并且需要特定于OCI驱动程序的功能（例如支持非TCP / IP网络），请使用JDBC OCI驱动程序。

  • 对于在数据库服务器中运行并需要访问同一数据库实例中的远程数据库或另一个会话的代码，请使用JDBC服务器端Thin驱动程序。

  • 如果您的代码在数据库服务器内运行并且需要在会话中本地访问数据，那么使用JDBC服务器端内部驱动程序来访问该服务器。

使用群集资源活动日志来监视群集资源故障

群集资源活动日志提供有关资源故障的精确和特定信息，与诊断日志分开。

如果Oracle Clusterware管理的资源出现故障，则Oracle Clusterware会在Grid Infrastructure Management资源库中的群集资源活动日志中记录有关故障的消息。由于资源，托管节点或网络出现问题，可能会发生故障。群集资源活动日志提供资源故障原因的统一视图。

对集群资源活动日志的写入使用活动ID进行标记，并且任何相关数据都将获得相同的父活动ID，并嵌套在父数据下。例如，如果Oracle Clusterware正在运行并且您运行crsctl stop clusterware -all命令，则所有活动都会获取活动ID，相关活动将使用相同的父活动ID进行标记。在每个节点上，该命令在父ID下创建子ID，并使用相应的活动ID标记每个相应的活动。此外，各个节点上的每个资源基于父ID创建子ID，从而创建活动ID的层次结构。通过活动ID层次结构，您可以分析数据以查找特定活动。

例如，您可能拥有许多资源，这些资源之间具有复杂的依赖关系，并且具有数据库服务。星期五，您会看到所有资源都在一个节点上运行，但是当您在星期一返回时，每个资源都在不同的节点上，并且您想知道原因。使用crsctl query calog命令，可以查询群集资源活动日志，以查找涉及这些资源和数据库服务的所有活动。输出提供了完整的流，您可以查询父服务故障转移ID中的每个子ID，并具体查看发生的情况和原因。

您可以使用筛选器查询群集资源活动日志中的任意数量的字段。例如，您可以查询特定操作系统用户（如root所写的所有活动。crsctl query calog命令生成的输出可以以表格格式或XML格式显示。

群集资源活动日志是当前Oracle Clusterware日志记录和警报日志消息的附件。

注意：

Oracle Clusterware不会将包含安全相关信息（如登录凭据）的消息写入群集活动日志。

Oracle Clusterware诊断和警报日志数据

查看此内容以了解Oracle Clusterware如何使用ADR的特定于集群件的方面。

Oracle Clusterware使用Oracle数据库故障诊断基础结构来管理诊断数据及其警报日志。因此，大多数诊断数据驻留在自动诊断存储库（ADR）中，ADR是位于安装期间指定的基目录下的目录和文件的集合。

ADR目录结构

Oracle Clusterware ADR数据是在称为ADR基础的根目录下编写的。由于ADR以外的组件使用此目录，因此它也可以称为Oracle基础。您指定在Oracle Grid Infrastructure安装期间用作基础的文件系统路径，并且只有在重新安装Oracle Grid Infrastructure时才能更改。

ADR文件驻留在ADR主目录中。在给定主机上运行的Oracle Clusterware的ADR主目录始终具有以下结构：

ORACLE_BASE / diag / crs / host_name / crs

在前面的示例中， ORACLE_BASE是您在安装Oracle Grid Infrastructure时指定的Oracle基本路径， host_name是主机的名称。在Windows平台上，此路径使用反斜杠（ \ ）分隔目录名称。

在ADR主页下面是特定类型ADR数据的各种目录。最感兴趣的目录是incident 。trace目录包含Oracle Clusterware守护程序和命令行程序编写的所有正常（非事件）跟踪文件以及Oracle Clusterware警报日志的简单文本版本。该组织与Oracle Clusterware 12 c第 1版（12.1.0.2）之前的版本有很大不同，其中诊断日志文件是在每个守护程序的不同目录下编写的。

要更改日志级别，请编辑ORACLE_BASE /crsdata/ host_name /crsdiag/ocrcheck.ini文件。

跟踪目录中的文件

从Oracle Clusterware 12 c第 1版（12.1.0.2）开始，Oracle Clusterware程序编写的诊断数据文件称为跟踪文件，并具有.trc文件扩展名，并一起显示在 ADR主页的trace子目录。这些文件的命名约定通常使用可执行程序名称作为文件名，可能根据程序类型使用其他数据进行扩充。

Oracle Clusterware命令行程序编写的跟踪文件在跟踪文件名中包含操作系统进程ID（PID），以区分同一命令程序的多个调用中的数据。例如，CRSCTL写入的跟踪数据使用以下名称结构： crsctl_ PID .trc 。在此示例中， PID是显示为十进制数字的操作系统进程ID。

Oracle Clusterware守护程序程序编写的跟踪文件在文件名中不包含PID，并且它们还受到影响命名的文件轮换机制的影响。旋转意味着当前守护程序跟踪文件达到特定大小时，文件将关闭，重命名，并打开新的跟踪文件。这会发生固定次数，然后丢弃守护程序中最旧的跟踪文件，保持旋转设置为固定大小。

大多数Oracle Clusterware守护程序使用的文件大小限制为10 MB，旋转集大小为10个文件，因此总共保留了100 MB的跟踪数据。给定守护程序的当前跟踪文件只使用程序名称作为文件名;旋转中的旧文件会在文件名后附加一个数字。例如，Oracle高可用性服务守护程序（OHASD）当前正在写入的跟踪文件名为ohasd.trc ;最近ohasd_出的跟踪文件名为ohasd_ n .trc ，其中n是一个不断增加的十进制整数。具有最高n的文件实际上是最近存档的跟踪，具有最低n的文件是最早的。

Oracle Clusterware代理是守护程序，其跟踪文件受特殊命名约定的约束，这些约定指示代理的来源（无论是由OHASD还是Cluster Ready Services守护程序（CRSD）生成）和代理程序的操作系统用户名运行。因此，代理的名称结构是：

origin_executable_user_name

注意：

名称结构中的前两个下划线（_）是文字的，并包含在跟踪文件名中。user_name的下划线不是文件命名约定的一部分。

在前面的示例中， origin是ohasd或crsd ， executable是可执行程序名称， user_name是操作系统用户名。此外，由于它们是守护程序，因此代理程序跟踪文件受前面描述的旋转机制的约束，因此在发生旋转后会出现带有附加_n后缀的文件。

Oracle Clusterware警报日志

除了跟踪文件之外，Oracle Clusterware ADR主目录中的trace子目录包含简单文本Oracle Clusterware警报日志。它的名称始终为alert.log 。警报日志也作为XML文件写入ADR主目录的alert子目录中，但最容易读取文本警报日志。

当Oracle Clusterware出现问题或问题时，警报日志是第一个查看的日志。与Oracle数据库实例警报日志不同，Oracle Clusterware警报日志中的消息将被标识，记录和本地化（转换）。Oracle Clusterware警报消息是针对发生的大多数重要事件或错误编写的。

注意：

写入Oracle Clusterware跟踪文件的消息和数据通常不会记录和翻译，主要由My Oracle Support用于问题诊断。

事件跟踪文件

Oracle Clusterware程序中会出现某些错误，这些错误会引发ADR事件。在大多数情况下，应将这些错误报告给My Oracle Support进行诊断。事件的发生通常会在Oracle Clusterware警报日志中生成一条或多条描述性消息。

除了警报消息之外，事件还会导致受影响的程序产生特殊的，单独的包含与错误相关的诊断数据的跟踪文件。这些特定于事件的跟踪文件收集在 ADR主目录的incident子目录而不是trace子目录。报告错误时，会收集正常跟踪文件和事件跟踪文件并将其提交给Oracle。

其他诊断数据

除ADR数据外，Oracle Clusterware还会收集或使用与问题诊断相关的其他数据。从Oracle Clusterware 12 c第 1版（12.1.0.2）开始，此数据位于ADR使用的相同基本路径下，但在具有以下形式的单独目录结构中： ORACLE_BASE /crsdata/ host_name 。在此示例中， ORACLE_BASE是您在安装Grid Infrastructure时指定的Oracle基本路径， host_name是主机的名称。

在此目录中，在给定主机上有几个子目录。如果出现问题，则最感兴趣的两个子目录被命名 core和 output 。当用于核心文件的正常ADR位置不可用时（例如，在程序中初始化ADR服务之前）， core目录是Oracle Clusterware守护程序核心文件的编写位置。output目录是Oracle Clusterware守护程序重定向其C标准输出和标准错误文件的位置。这些文件通常使用由可执行文件名称组成的名称结构，其中字符OUT附加到.trc文件扩展名（如跟踪文件）。例如，Cluster Time Synchronization Service守护程序的重定向标准输出名为octssdOUT.trc 。通常，守护进程对这些文件写的很少，但在某些故障情况下，可能会在那里写入重要数据。

Oracle Extended Clusters中的存储拆分

当两个或多个不同站点之间的专用网络可用且处于联机状态但存储网络发生故障时，会发生存储拆分。

当Oracle自动存储管理（Oracle ASM）在具有三个站点（两个数据站点和仲裁站点）的典型扩展群集配置中检测到存储拆分时，其中一个数据站点将终止并隔离自身及其包含的其余节点集群。如果Oracle ASM尝试在隔离的站点上启动，则会发生错误CRS-2971。

解决问题，如下：

  1. 解决导致存储拆分的站点间连接问题。
  2. 确保所有Oracle ASM磁盘组都安装在未隔离的站点上，如下所示：
    SELECT group_number，name，state FROM v $ asm_diskgroup_stat;
  3. 通过对每个磁盘组运行以下命令，获取属于这些磁盘组的联机磁盘列表：
    SELECT路径FROM v $ asm_disk_stat WHERE group_number = group_number AND state ='NORMAL'AND mode_status ='ONLINE';
  4. 对于您在上一步中获得的每条路径，请确保可以从隔离的站点访问磁盘，如下所示：
    asmcmd lsdsk -I  - 成员' path '
  5. 如果上述验证成功，则重新启动隔离的站点，如下所示：
    crsctl修改集群站点site_name -s rejuvenate

    crsctl stop testdns -address address

    crsctl stop ip -A IP_name / netmask / interface_name

关于在SQL中使用LOB

您可以使用SQL VARCHAR2语义（如SQL字符串运算符和函数）访问CLOB和NCLOB数据类型。（ LENGTH函数可以与BLOB数据类型以及CLOB和NCLOB 。）这些技术在以下情况下是有益的：

  • 对LOB进行操作时，这些LOB的大小相对较小（最多约100K字节）。

  • 将数据库从LONG列迁移到LOB数据类型后，现有PL / SQL应用程序中包含的任何SQL字符串函数将在迁移后继续工作。

在以下情况下不建议使用SQL语义：

  • 使用随机访问和分段提取等高级功能时，必须使用LOB API。

  • 使用SQL语义对大小相对较大（大于1MB）的LOB执行操作可能会影响性能。在这种情况下，建议使用LOB API。

    注意：

    SQL语义与持久和临时LOB一起使用。（SQL语义不适用于BFILE列，因为BFILE是只读数据类型。）

表7-1中描述的INSTR ， SUBSTR ， LENGTH和LIKE函数的代码点语义根据传递给函数的参数的数据类型而不同。这些函数使用不同的代码点语义，具体取决于参数是VARCHAR2还是CLOB类型，如下所示：

  • 当参数是CLOB ，UCS2代码点语义用于所有字符集。

  • 当参数是字符类型（例如VARCHAR2 ，默认的代码点语义用于给定的字符集：

    • UCS2代码点语义用于AL16UTF16和UTF8字符集。

    • UCS4代码点语义用于所有其他字符集，例如AL32UTF8。

  • 如果要将字符数据存储在CLOB或NCLOB ，请注意，在UCS2代码点中指定了读取或写入CLOB或NCLOB数据的任何API的数量和偏移量参数。在某些字符集中，完整字符包含一个或多个称为代理项对的UCS2代码点。在这种情况下，您必须确保指定的金额或偏移量不会切割为完整字符。这可以避免读取或写入部分字符。

  • 在SQL函数的情况下以及在通过LOB API进行读/写的情况下，Oracle数据库有助于检测读取或写入边界上的半代理对。行为如下：

    • 如果起始偏移位于代理对的中间，则读取和写入操作都会引发错误。

    • 如果读取的数量仅读取部分字符，则将数量增加或减少1以读取完整字符。

      注意：

      输出量可能与输入量不同。

    • 如果写入量覆盖部分字符，则会引发错误以防止由于覆盖目标CLOB或NCLOB中的部分字符而导致现有数据损坏。

      注意：

      此检查仅适用于CLOB或NCLOB中的现有数据。您必须确保写操作的传入缓冲区以完整字符开头和结尾。

关于LOB的VARCHAR2和RAW语义

与VARCHAR2和RAW数据类型一起使用的这些语义也适用于LOB：

  • 在CLOB上定义CHAR缓冲区

    您可以为CLOB定义VARCHAR2 ，为BLOB列定义RAW 。您还可以为VARCHAR2和RAW列定义CLOB和BLOB类型。

  • 选择CLOB列到CHAR缓冲区或VARCHAR2

    如果将CLOB列选择到VARCHAR2变量中，则会检索存储在CLOB列中的数据并将其放入CHAR缓冲区。如果缓冲区不足以包含所有CLOB数据，则抛出截断错误，并且不会将数据写入缓冲区。成功完成SELECT操作后， VARCHAR2变量将保存为常规字符缓冲区。

    相反，当CLOB列被选择到本地CLOB变量中时，将获取CLOB定位符。

  • 选择BLOB列到RAW

    当BLOB列被选择到RAW变量中时， BLOB数据被复制到RAW缓冲区中。如果BLOB的大小超过缓冲区的大小，则抛出截断错误，并且没有数据写入缓冲区。

从SQL函数返回的LOB

从SQL函数返回LOB时，返回的结果是临时LOB。

您的应用程序应将临时LOB视为SELECT操作返回的数据的本地存储，如下所示：

  • 在PL / SQL中，临时LOB与其他本地PL / SQL程序变量具有相同的生存期（持续时间）。它可以作为PL / SQL局部变量传递给后续的SQL或PL / SQL VARCHAR2函数或查询。临时LOB在程序块结束时超出范围，此时释放LOB。这些语义与PL / SQL VARCHAR2变量的语义相同。但是，您可以随时使用DBMS_LOB.FREETEMPORARY()调用来释放本地临时LOB所占用的资源。

    注意：

    如果SQL语句返回LOB或LOB是PL / SQL函数或过程的OUT参数，则必须测试它是否是临时LOB，如果是，则在完成后将其释放。

  • 在OCI中，从SQL查询返回的临时LOB始终处于会话持续时间，除非存在用户定义的持续时间，在这种情况下，临时LOB处于用户定义的持续时间。

IS NULL和IS NOT NULL与VARCHAR2和CLOB一起使用

您可以将IS NULL和IS NOT NULL运算符与LOB列一起使用。

当与LOB一起使用时， IS NULL和IS NOT NULL运算符确定LOB定位符是否存储在行中。

注意：

在SQL 92标准中，a 长度为零的字符串与NULL字符串不同。与VARCHAR2相比，传递LOB时，IS NULL的返回值会有所不同：

  • 将长度为零的初始化LOB传递给IS NULL函数时，返回零（ FALSE ）。这些语义符合SQL标准。

  • 将长度为零的VARCHAR2传递给IS NULL函数时，将返回TRUE 。

与LOB一起使用的子句

除了用于比较LOB值的函数之外，在WHERE子句的谓词中允许使用LOB作为参数的SQL函数。

例如， LENGTH函数可以包含在WHERE子句的谓词中：

CREATE TABLE t（n NUMBER，c CLOB）;插入值（1，'abc'）; SELECT * FROM t WHERE c IS NOT NULL; SELECT * FROM t WHERE LENGTH（c）> 0; SELECT * FROM t WHERE c LIKE'％a％'; SELECT * FROM t WHERE SUBSTR（c，1,2）LIKE'％b％'; SELECT * FROM t WHERE INSTR（c，'b'）= 2;
