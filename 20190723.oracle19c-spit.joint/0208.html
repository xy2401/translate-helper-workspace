82 DBMS_HS_PASSTHROUGH

DBMS_HS_PASSTHROUGH PL / SQL包允许您直接将语句发送到非Oracle系统，而不会被Oracle服务器解释。如果非Oracle系统允许在Oracle中没有等效语句的语句中进行操作，这将非常有用。

82.1 DBMS_HS_PASSTHROUGH概述

您可以使用PL / SQL包DBMS_HS_PASSTHROUGH直接在非Oracle系统上执行直通SQL语句。使用此包执行的任何语句都在与标准SQL语句相同的事务中执行。

82.2 DBMS_HS_PASSTHROUGH操作说明

DBMS_HS_PASSTHROUGH包是一个虚拟包。它概念上位于非Oracle系统中。但实际上，对此程序包的调用被异构服务拦截并映射到一个或多个异构服务调用。反过来，驱动程序将这些异构服务调用映射到非Oracle系统的API。客户端应用程序应通过数据库链接调用包中的过程，其方式与调用非Oracle系统存储过程的方式完全相同。异构服务完成的特殊处理对用户是透明的。

82.3.10 FETCH_ROW功能

此函数从结果集中提取行。

结果集使用SQL SELECT语句定义。当没有更多要提取的行时，会引发异常NO_DATA_FOUND 。在可以获取行之前，必须打开游标，并且必须解析SQL语句。

句法

在快速模式下，不使用SQL * Loader控制文件。而是，SQL * Loader使用在ALL_TAB_COLUMNS视图中找到的表列定义来确定输入字段顺序和数据类型。对于大多数其他设置，它假定您可以使用命令行参数覆盖的默认值。

注意：

用于SQL * Loader express模式的唯一有效参数是本章中描述的参数。任何其他参数将被忽略或可能导致错误。

13.3.1坏

默认值：默认值取决于是否指定了任何数据文件（使用DATA参数）。请参见SQL * Loader Express模式使用的默认值 。

目的

BAD参数指定坏文件的位置和名称。

语法和说明

BAD = [ 目录 /] [ 文件名 ]

坏文件存储在插入期间导致错误或格式不正确的记录。如果指定BAD参数，则必须提供目录或文件名，或两者都提供。如果未指定BAD参数，并且存在拒绝记录，则使用默认文件名。

directory变量指定写入错误文件的目录。规范可以包括设备名称或网络节点。

filename变量指定在您的平台上被识别为有效的文件名。您必须仅指定名称（和扩展名，如果需要除.bad之外的.bad ）。文件名中的任何空格或标点符号必须用单引号括起来。

directory和filename的值确定如下：

  • 如果使用文件名但没有目录指定BAD参数，则该目录默认为当前目录。

  • 如果使用目录但没有文件名指定BAD参数，则使用指定的目录，并将默认值用于文件名和扩展名。

BAD参数适用于与指定的DATA参数匹配的所有文件（如果已指定）。如果未指定data参数，它将应用于一个数据文件（ table-name.dat ）。

限制

警告：

  • 如果文件名（默认值或您指定的文件名）已存在，则会覆盖该文件名或创建新版本，具体取决于您的操作系统。

  • 如果正在加载多个数据文件，则建议您不要指定BAD参数，也不要仅指定坏文件的目录。

例

13.3.2字符

默认值： NLS_LANG环境变量中指定的NLS客户端字符集

目的

CHARACTERSET参数指定用于加载的字符集（默认值除外）。

语法和说明

CHARACTERSET = character_set_name

CHARACTERSET参数指定SQL * Loader输入数据文件的字符集。如果未指定CHARACTERSET参数，则所有数据文件的缺省字符集是会话字符集，该字符集由NLS_LANG环境变量定义。只有字符数据（SQL * Loader数据类型CHAR, VARCHAR, VARCHARC,数字EXTERNAL以及日期时间和间隔数据类型的字段）受数据文件的字符集影响。

  • 没有

13.3.3 CSV

默认值：如果未在命令行中指定CSV参数，则SQL * Loader express假定正在加载的CSV文件包含没有嵌入字符且没有机箱的数据。

如果在命令行中指定了CSV=WITHOUT_EMBEDDED ，则SQL * Loader express假定正在加载的CSV文件包含没有嵌入字符且可选地用“”括起来的数据。

目的

CSV参数提供的选项允许您指定要加载的逗号分隔值（CSV）格式文件是否包含嵌入了记录终止符的字段。

语法和说明

CSV = [WITH_EMBEDDED | WITHOUT_EMBEDDED]

此参数的有效选项如下：

  • WITH_EMBEDDED - 此选项意味着记录中的字段中可能包含（嵌入）记录终止符。记录终止符是换行符。默认分隔符是TERMINTATED BY ","和OPTIONALLY_ENCLOSED_BY '"' 。必须附上嵌入式记录终止符。

  • WITHOUT_EMBEDDED - 此选项表示记录中的字段中不包含（嵌入）记录终止符。记录终止符是换行符。默认分隔符是TERMINTATED BY ","和OPTIONALLY_ENCLOSED_BY '“'。

限制

  • 通常，文件可以并行处理（一次拆分并由多个执行服务器处理）。但对于具有嵌入式记录终止符的CSV格式文件，该文件必须仅由一个执行服务器处理。因此，当CSV=WITH_EMBEDDED时，将禁用数据文件中的并行处理。

例

以下示例将数据文件作为具有嵌入式记录终止符的CSV格式文件进行处理。

> sqlldr hr TABLE = employees CSV = WITH_EMBEDDED

默认值： EXECUTE

目的

EXTERNAL_TABLE参数指示SQL * Loader是否使用外部表选项加载数据。

语法和说明

EXTERNAL_TABLE = [NOT_USED | GENERATE_ONLY |执行]

有三个可能的值：

  • NOT_USED - 表示使用传统或直接路径模式执行加载。

  • GENERATE_ONLY - 使用SQL * Loader日志文件中的外部表放置加载所需的所有SQL语句。可以编辑和自定义这些SQL语句。通过在SQL * Plus中执行这些语句，可以在不使用SQL * Loader的情况下稍后完成实际加载。

  • EXECUTE - SQL * Loader express模式中的默认值。尝试执行使用外部表进行加载所需的SQL语句。但是，如果任何SQL语句返回错误，则加载尝试将停止。语句在执行时放在日志文件中。这意味着如果SQL语句返回错误，则加载所需的其余SQL语句将不会放在日志文件中。

请注意，外部表选项使用数据库中的目录对象来指示存储所有数据文件的位置，并指示创建输出文件（如坏文件和丢弃文件）的位置。您必须具有对包含数据文件的目录对象的READ访问权限，并且您必须具有对创建输出文件的目录对象的WRITE访问权限。如果数据文件或输出文件的位置没有现有目录对象，则SQL * Loader将生成SQL语句以创建一个。因此，指定EXECUTE选项时，您必须具有CREATE ANY DIRECTORY权限。如果希望在加载结束时删除目录对象，则还必须具有DROP ANY DIRECTORY权限。

注意：

EXTERNAL_TABLE= EXECUTE限定符告诉SQL * Loader创建一个外部表，该表可用于加载数据，然后执行INSERT语句来加载数据。必须将外部表中的所有文件标识为位于目录对象中。SQL * Loader尝试使用已存在且您有权访问的目录对象。但是，如果SQL * Loader找不到匹配的目录对象，则它会尝试创建临时目录对象。如果您没有创建新目录对象的权限，则操作将失败。

要解决此问题，请使用EXTERNAL_TABLE= GENERATE_ONLY来创建SQL * Loader将尝试执行的SQL语句。提取这些SQL语句并将对目录对象的引用更改为您有权访问的目录对象。然后，执行这些SQL语句。

限制

  • 没有

例

sqlldr hr TABLE = employees EXTERNAL_TABLE = NOT_USED

13.3.12 FIELD_NAMES

默认值： NONE

目的

FIELD_NAMES参数用于覆盖数据库表中列的顺序的字段。（默认情况下，SQL * Loader Express使用ALL_TAB_COLUMNS视图中的表列定义来确定输入字段顺序和数据类型。）

此参数何时有用的示例是输入文件中的数据与表中的列的顺序不同。在这种情况下，您可以在数据文件中包含字段名称记录（类似于表的列标题行），并使用FIELD_NAMES参数通知SQL * Loader处理第一个记录中的字段名称以确定顺序的领域。

语法和说明

FIELD_NAMES = [ALL | ALL_IGNORE |第一个| FIRST_IGNORE |没有]

此参数的有效选项如下：

  • ALL - 为每个数据文件处理字段名称记录。

  • ALL_IGNORE - 忽略所有数据文件中的第一个（字段名称）记录并正常处理数据记录。

  • FIRST - 在第一个数据文件中，处理第一个（字段名称）记录。对于所有其他数据文件，没有字段名称记录，因此正常处理数据文件。

  • FIRST_IGNORE - 在第一个数据文件中，忽略第一个（字段名称）记录，并使用表列顺序作为字段顺序。

  • NONE - 任何数据文件中都没有字段名称记录，因此数据文件会正常处理。这是默认值。

限制

  • 如果任何字段名称具有大小写混合大小写或特殊字符（例如，空格），则必须使用OPTIONALLY_ENCLOSED_BY参数或ENCLOSED_BY参数来指示应保留该大小写，并且应将特殊字符作为字段名称的一部分包含在内。

例

如果要将包含列标题的CSV文件加载到表中，并且输入文件中每行中的字段与表中的列的顺序相同，则可以使用以下命令：

> sqlldr hr TABLE = employees CSV = WITHOUT_EMBEDDED FIELD_NAMES = FIRST_IGNORE

13.3.21 TRIM

默认值：传统和直接路径加载的默认值为LDRTRIM 。外部表加载的默认值是LRTRIM 。

目的

语法和说明

TRIM = [LRTRIM | NOTRIM | LTRIM | RTRIM | LDRTRIM]

  • 如果该字段是指定了OPTIONALLY _ ENCLOSED _ BY的分隔字段，并且特定实例缺少可选的外壳，则将从左侧修剪空格。

  • 如果加载是默认外部表加载并且发生错误导致SQL * Loader express模式使用直接路径加载，则LDRTRM用作修剪方法，即使您指定了不同的方法或已接受外部表默认LRTRIM 。将显示一条消息，提醒您进行此更改。

    要使用NOTRIM ，请使用带有PRESERVE BLANKS子句的控制文件。

例

以下示例读取字段，修剪右侧的所有空格（尾随空格）。

> sqlldr hr TABLE = employees TRIM = RTRIM

7.2存储管理

除非所有者另行指定，否则分析工作区将存储在所有者的默认表空间中。OLAP使用的所有表空间都应指定EXTENT MANAGEMENT LOCAL 。使用默认参数创建的表空间可能无效地使用资源。您应该创建适合存储分析工作区的撤消，永久和临时表空间。

7.2.1创建撤消表空间

使用EXTENT MANAGEMENT LOCAL子句创建一个undo表空间，如下例所示：

创建UNDO TABLESPACE olapundo DATAFILE'$ ORACLE_BASE / oradata / undo.dbf'SIZE 64M REUSE AUTOEXTEND on next 8M MAXSIZE UNLIMITED EXTENT MANAGEMENT LOCAL;

创建撤消表空间后，更改系统参数文件以包括以下设置，然后按照“ 设置数据库初始化参数 ”中的说明重新启动数据库。 

UNDO_TABLESPACE = 表空间 UNDO_MANAGEMENT = AUTO

7.2.2为OLAP使用创建永久表空间

每个维对象占据至少一个范围。固定范围大小可能会浪费大部分分配的空间。例如，如果对象为64K且范围设置为1M的统一大小（默认值），则仅使用范围的一小部分。

使用EXTENT MANAGEMENT LOCAL和SEGMENT SPACE MANAGEMENT AUTO子句创建永久表空间，如下例所示：

CREATE TABLESPACE glo DATAFILE'$ ORACLE_BASE / oradata / glo.dbf'SIZE 64M REUSE AUTOEXTEND on next 8M MAXSIZE UNLIMITED EXTENT MANAGEMENT LOCAL SATERMENT SPACE AUTO;

7.2.3为OLAP使用创建临时表空间

无论更改是数据加载还是数据分析的结果，Oracle OLAP都使用临时表空间来存储多维数据集中数据的所有更改。保存多维数据集会将更改移动到永久表空间并清除临时表空间。

此用法在表空间中创建了许多范围。适合Oracle OLAP使用的临时表空间应指定EXTENT MANAGEMENT LOCAL子句和小尺寸的UNIFORM SIZE子句，如下例所示：

CREATE TEMPORARY TABLESPACE glotmp TEMPFILE'$ ORACLE_BASE / oradata / glotmp.tmp'SIZE 50M REUSE AUTOEXTEND on next 5M MAXSIZE UNLIMITED EXTENT MANAGEMENT LOCAL UNIFORM SIZE 256K;

7.2.4跨存储资源传播数据

Oracle数据库提供了出色的存储管理工具来简化日常任务。自动存储管理（ASM）提供了一个简单的存储管理界面，可将数据库存储虚拟化为磁盘组。您可以管理一小组磁盘组，ASM可以自动在这些磁盘组中放置数据库文件。

ASM在所有可用存储资源上均匀分布数据，以优化性能和利用率。添加或删除磁盘后，ASM会自动重新平衡磁盘组中的文件。

由于OLAP是Oracle数据库的一部分，因此您可以使用ASM来管理关系数据和维度数据。

强烈建议ASM用于分析工作区。使用ASM管理的系统比文件系统更快，并且比原始设备更易于管理。ASM在使用Oracle RAC的系统和没有Oracle RAC的系统上优化分析工作区的性能。

但是，您不需要ASM来使用Oracle OLAP。您仍然可以跨多个磁盘传播数据，只需定义表空间，如下例所示：

CREATE TABLESPACE glo DATAFILE'disk1 / oradata / glo1.dbf'SIZE 64M REUSE AUTOEXTEND ON NEXT 8M MAXSIZE 1024M EXTENT MANAGEMENT LOCAL SATERMENT SPACE AUTO AUTO; ALTER TABLESPACE glo ADD DATAFILE'disk2 / oradata / glo2.dbf'SIZE 64M REUSE AUTOEXTEND ON NEXT 8M MAXSIZE 1024M，'disk3 / oradata / glo3.dbf'SIZE 64M REUSE AUTOEXTEND ON NEXT 8M MAXSIZE UNLIMITED;

7.3.4维护日志

第一次使用Analytic Workspace Manager将数据加载到多维数据集或维度时，它会创建多个日志。这些日志存储在与分析工作区相同的模式中的表中：

  • 多维数据集生成日志 ：包含有关生成期间发生的事件的信息。使用此日志来确定构建是否产生了您期望的结果，如果不是，为什么不产生。无论是通过Analytic Workspace Manager，数据库物化视图刷新子系统还是PL / SQL过程，每当刷新多维数据集或维度时，日志都会不断更新。您可以随时查询日志以评估构建进度并估计完成时间。默认表名称为CUBE_BUILD_LOG 。

  • 多维数据集维度编译日志 ：包含OLAP聚合多维数据集时在维度层次结构验证期间发生的错误。默认表名是CUBE_DIMENSION_COMPILE 。

  • 多维数据集操作日志 ：包含所有OLAP引擎事件的消息和调试信息。默认表名称为CUBE_OPERATIONS_LOG 。

  • 多维数据集拒绝记录日志 ：标识因未满足预期格式而被拒绝的任何记录。默认表名为CUBE_REJECTED_RECORDS 。

通过这些日志，您可以跟踪长时间运行的进程的进度，然后使用结果来分析性能特征。它们提供的信息可帮助您诊断和解决在开发和维护多维数据集期间可能发生的问题。它们还有助于诊断查询多维数据集时的性能问题。

您还可以运行$ ORACLE_HOME / olap / admin / utlolaplog.sql脚本来创建包含一些有用视图的构建日志。

7.4.2创建和删除分区

作为数据维护的一部分，OLAP引擎会自动创建和删除分区，因为会在分区维度中添加和删除成员。

例如，假设在示例“全局分析”工作空间中，“单位”多维数据集在“时间”维度上使用“日历”层次结构和“日历季度”级别进行分区。OLAP引擎为每个Calendar Quarter及其子项创建一个分区。默认顶部分区包含Calendar Years和Fiscal层次结构的所有成员。如果Global有三年的数据，那么Units多维数据集有13个分区：每个Calendar Year的四个底部分区，加上顶部分区。

数据刷新通常会创建新的时间段并删除旧的时间段。只要将“日历季度”值加载到“时间”维度中，就会将相应的分区添加到多维数据集中。每当从“时间”维度中删除“日历季度”值时，都会从多维数据集中删除相应的空分区。

7.4.3并行性

您可以通过启用并行处理来提高数据维护的性能。有两个级别的并行性：

  • 并行作业执行：使用多个进程加载和聚合数据。

  • 并行更新：使用多个进程将数据从临时表空间移动到永久表空间。

这些并行进程受以下因素控制：

  • 可以并行聚合的对象数。每个多维数据集和每个分区（包括顶部分区）都可以使用单独的进程。

    您可以在Analytic Workspace Manager的多维数据集属性表的“分区”选项卡上控制多维数据集中的分区数。

  • 用户被授权运行的并发数据库进程数。

    这个数字是由 JOB_QUEUE_PROCESSES参数。如果您具有SYS权限，则可以使用以下SQL命令获取当前参数设置：

    SHOW PARAMETER JOB_QUEUE_PROCESSES

  • 对于并行更新，您分配给作业的进程数。在指定任务处理选项时，或在多维数据集的“物化视图”选项卡上，可以在Analytic Workspace Manager的维护向导中指定进程数。

  • 分配给SQL以从关系源表中获取行的进程数。当PARALLEL_DEGREE_POLICY设置为AUTO或LIMITED ，数据库可以分配其他进程来执行SQL语句。

假设多维数据集在“季度”级别上进行分区，并且多维数据集包含三年的数据。多维数据集具有3 * 4 = 12个底部分区， JOB_QUEUE_PROCESSES设置为8，并且您将并行性选项设置为4用于构建。当PARALLEL_DEGREE_POLICY设置为其默认值MANUAL时，Oracle数据库以这种方式处理多维数据集：

  1. 使用单个过程连续加载和构建立方体的尺寸。

  2. 使用4个进程并行加载和构建12个底部分区。一旦一个过程完成，另一个过程开始直到所有12个过程完成。

    此多维数据集可以使用JOB_QUEUE_PROCESSES允许的8个进程，但构建设置限制为4。

  3. 加载并构建顶部分区。

当PARALLEL_DEGREE_POLICY设置为AUTO或LIMITED ，Oracle数据库可能会分配多于指定的进程。

7.5分析立方体和尺寸

7.7关于备份和恢复分析工作区

您可以使用与数据库其余部分相同的工具和过程来备份和恢复分析工作区。

Oracle Recovery Manager（RMAN）是一款功能强大的工具，可简化，自动化和改进备份和恢复操作的性能。RMAN支持基于用户指定的恢复窗口，可重新启动的备份和还原以及测试还原/恢复的一次性备份配置，备份自动管理和归档日志。

RMAN实现了一个恢复窗口来控制备份何时到期。这使您可以通过执行数据库或表空间时间点恢复来建立可以发现逻辑错误并修复受影响对象的时间段。RMAN还会自动将不再需要的备份过期，以将数据库还原到恢复窗口中的某个时间点。即使RMAN存储库不可用，控制文件自动备份也允许还原或恢复数据库。

  • 度量文件夹 ：在度量文件夹中保存度量列表。它不保存对象。

模板存储元数据，而不是数据。您可以将模板存储在小型文本文件或数据库表中。从模板重新创建对象时，您必须有权访问源数据。

7.10.4使用查询重写

查询重写更改查询以从实例化视图中选择数据，而不是从主表中计算结果集。转换对客户端完全透明，并且不需要在SQL语句中提及实体化视图。对于多维数据集物化视图，查询是针对星型或雪花式架构的表或视图编写的，并将其转换为针对多维数据集物化视图的查询。这种转换可以显着改善运行时性能。

查询重写需要有关多维数据集和维度的优化程序统计信息。您可以通过生成和检查其执行计划来发现是否重写了查询。

Oracle数据库使用两个初始化参数来控制查询重写：

  • QUERY_REWRITE_ENABLED ：为数据库启用或禁用全局查询重写。

  • QUERY_REWRITE_INTEGRITY ：确定查询重写监视具有源数据的QUERY_REWRITE_INTEGRITY化视图的一致性的程度。使用重写到多维数据集物化视图时，建议使用trusted或stale tolerated设置。

多维数据集物化视图的管理与任何其他物化视图相同，只是多维数据集物化视图必须与分析工作区位于同一模式中。用户需要 GLOBAL QUERY REWRITE特权，可以重写除了自己的模式之外的物化视图。但是，所有者可以从任何架构访问实例化视图，而无需其他权限。

34.1 DBMS_COMPARISON概述

DBMS_COMPARISON包是Oracle提供的包，可用于比较两个数据库中的数据库对象。此包还使您可以聚合数据库对象，以便它们在不同的数据库中保持一致。通常，此程序包用于在多个数据库中共享数据库对象的环境中。当多个数据库中存在相同数据库对象的副本时，数据库对象是共享数据库对象 。多个数据字典视图包含有关使用DBMS_COMPARISON包进行比较的信息。

共享数据库对象可能由数据复制维护。例如，物化视图或Oracle Streams组件可能会复制数据库对象并将其维护在多个数据库中。自定义应用程序也可能维护共享数据库对象。共享数据库对象时，它可以在共享它的数据库中发散。您可以使用此包来标识共享数据库对象中的差异。确定差异后，您可以选择使用此程序包同步共享数据库对象。

要比较在两个不同数据库中共享的数据库对象，请完成以下常规步骤：

  1. 运行此程序包中的CREATE_COMPARISON过程以创建比较 。比较标识要比较的数据库对象，并指定比较参数。

  2. 运行此程序包中的COMPARE函数以比较两个数据库中的数据库对象并识别差异。当没有找到差异时，此函数返回TRUE当找到差异时，此函数返回FALSE 。此函数还会使用比较结果填充数据字典视图。为每次执行COMPARE功能生成单独的比较结果。

  3. 如果要检查比较结果，请查询以下数据字典视图：

    • DBA_COMPARISON_SCAN

    • USER_COMPARISON_SCAN

    • DBA_COMPARISON_SCAN_VALUES

    • USER_COMPARISON_SCAN_VALUES

    • DBA_COMPARISON_ROW_DIF

    • USER_COMPARISON_ROW_DIF

  4. 如果存在差异，并且您希望在两个数据库中同步数据库对象，则在此程序包中运行CONVERGE过程。

在DBMS_COMPARISON包中创建与CREATE_COMPARISON过程的比较后，可以使用COMPARE函数随时运行比较。每次运行COMPARE函数时，它都会在相应的数据字典视图中记录比较结果。当调用此包中的子程序并指定比较结果中的扫描时，可能会修改比较结果。例如，运行RECHECK函数时可能会修改比较结果。

单次执行COMPARE功能的比较结果可以包括一次或多次扫描 。扫描会在单个时间点检查共享数据库对象中某些或所有行的差异。您可以多次比较数据库对象，并且唯一的扫描ID可识别比较结果中的每个扫描。

存储桶是正在比较的数据库对象中的一系列行。通过将数据库对象拆分为范围并独立比较范围，存储桶可以提高性能。每次比较将要比较的行划分为适当数量的桶，并通过扫描比较每个桶。

每次COMPARE功能将桶拆分成较小的桶时，它会对较小的桶执行新的扫描。分析较大存储桶的扫描是每次扫描的父扫描 ，该扫描分析较大存储桶被拆分的较小存储桶。比较结果中的根扫描是最高级别的父扫描。根扫描没有父级。

您可以使用RECHECK功能重新检查扫描，并且可以使用CONVERGE过程收敛扫描。如果要重新检查或收敛所有行比较结果，请在相应的子程序中指定比较结果的根扫描ID。如果要在比较结果中重新检查或收集部分行，请指定包含差异的扫描的扫描ID。

34.2 DBMS_COMPARISON安全模型

可以通过两种方式之一控制此程序包的安全性。

  • 将此程序包上的EXECUTE授予选定的用户或角色。

  • 向选定的用户或角色授予EXECUTE_CATALOG_ROLE 。

如果程序包中的子程序是从存储过程中运行的，则必须直接为程序包授予运行子程序的用户EXECUTE特权。它不能通过角色授予。

DBMS_COMPARISON包中的每个子程序都有一个comparison_name参数。当前用户必须是指定比较的所有者才能在DBMS_COMPARISON包中运行子程序。

要运行COMPARE函数， RECHECK函数或CONVERGE过程，以下用户必须对共享数据库对象的每个副本具有SELECT或READ权限：

  • 本地数据库中的比较所有者

  • 使用数据库链接时，比较所有者通过数据库链接连接的远程数据库中的用户

CONVERGE过程还需要为数据库中的其中一个用户提供额外的权限，以便对共享数据库对象进行更改。用户必须对此数据库上的共享数据库对象具有INSERT ， UPDATE和DELETE权限。

此外，在将local_converge_tag或remote_converge_tag参数设置为非NULL值的情况下运行CONVERGE过程时，必须满足以下附加要求：

  • 如果本地表“wins”，那么CONVERGE过程的调用者通过数据库链接连接的远程数据库的用户必须被授予DBMS_STREAMS_ADM包的EXECUTE_CATALOG_ROLE或EXECUTE特权。

  • 如果远程表“wins”，则必须在本地数据库上的CONVERGE过程的调用者授予DBMS_STREAMS_ADM包的EXECUTE_CATALOG_ROLE或EXECUTE特权。

数据库管理员（DBA）可以控制其他用户拥有的某些DBMS_COMPARISON功能和过程。此控件适用于DROP_COMPARISON和PURGE_COMPARISON 。当需要删除由另一个用户创建的比较时，此DBA覆盖在清理操作中特别有用

DBMS_COMPARISON包的索引列要求

本节讨论数字，时间戳和间隔列。其中包括以下数据类型：

  • 间隔列有以下几种数据类型： INTERVAL YEAR TO MONTH和INTERVAL DAY TO SECOND 。

对于DBMS_COMPARISON程序包支持的所有扫描模式，数据库对象必须具有以下类型的索引之一：

  • 数字，时间戳，间隔或DATE数据类型列上的单列索引

  • 仅包含数字，时间戳，间隔或DATE数据类型列的组合索引。复合索引中的每一列必须具有NOT NULL约束或者是主键的一部分。

对于支持扫描模式CMP_SCAN_MODE_FULL和CMP_SCAN_MODE_CUSTOM ，数据库对象必须具有以下类型的索引之一：

  • 数字，时间戳，间隔， DATE ， VARCHAR2或CHAR数据类型列上的单列索引

  • 仅包含数字，时间戳，间隔， DATE ， VARCHAR2或CHAR列的组合索引。复合索引中的每一列必须具有NOT NULL约束或者是主键的一部分。

如果数据库对象没有这些类型的索引之一，则DBMS_COMPARISON包不支持数据库对象。例如，如果数据库对象在NVARCHAR2列上只有一个索引，则DBMS_COMPARISON包不支持它们。或者，如果数据库对象只有一个索引，并且它是包含NUMBER列和NCHAR列的复合索引，则DBMS_COMPARISON包不支持它们。

使用CREATE_COMPARISON过程中的index_schema_name和index_name参数创建比较时，可以指定索引。如果指定索引，请确保索引中的列满足用于比较的扫描模式的要求。

比较中的索引列必须唯一标识比较中涉及的每一行。以下约束满足此要求：

  • 主键约束

  • 对一个或多个非NULL列的唯一约束

如果表上不存在这些约束，则使用CREATE_COMPARISON过程中的index_schema_name和index_name参数指定其列满足此要求的索引。

当单个索引值同时标识本地行和远程行时，这两行必须是复制表中同一行的副本。此外，同一行的每对副本必须始终具有相同的索引值。

如果有多个索引列，则COLUMN_POSITION的索引COLUMN_POSITION 1 ，是复合索引中的前导索引列。

分区概述

在Oracle数据库中， 分区使您能够将非常大的表和索引分解为更小且更易于管理的分区 。每个分区都是一个独立的对象，它有自己的名称和可选的自己的存储特性。

对于说明分区的类比，假设HR经理有一个包含员工文件夹的大框。每个文件夹列出员工的雇用日期。通常会针对特定月份雇用的员工进行查询。满足此类请求的一种方法是创建雇员雇用日期的索引，该索引指定分散在整个框中的文件夹的位置。相比之下，分区策略使用许多较小的盒子，每个盒子包含在给定月份雇用的员工的文件夹。

使用较小的盒子有几个优点。当被要求为6月份雇用的员工检索文件夹时，人力资源经理可以检索6月份的框。此外，如果任何小盒子暂时损坏，其他小盒子仍然可用。移动办公室也变得更容易，因为经理可以移动几个小盒子而不是移动一个重箱子。

从应用程序的角度来看，只存在一个模式对象。SQL语句不需要修改即可访问分区表。分区对许多不同类型的数据库应用程序很有用，特别是那些管理大量数据的应用程序。好处包括：

  • 提高可用性

    分区的不可用性不会导致对象不可用。查询优化器会自动从查询计划中删除未引用的分区，以便在分区不可用时查询不受影响。

  • 更轻松地管理模式对象

    分区对象具有可以集体或单独管理的片段。DDL语句可以操作分区而不是整个表或索引。因此，您可以分解资源密集型任务，例如重建索引或表。例如，您可以一次移动一个表分区。如果出现问题，则只需重做分区移动，而不是表移动。此外，删除分区可避免执行大量DELETE语句。

  • 减少OLTP系统中共享资源的争用

    在某些OLTP系统中，分区可以减少对共享资源的争用。例如，DML分布在许多段而不是一个段上。

  • 增强了数据仓库中的查询性能

    在数据仓库中 ，分区可以加快即席查询的处理速度。例如，包含一百万行的销售表可以按季度进行分区。

分区特征

表或索引的每个分区必须具有相同的逻辑属性，例如列名，数据类型和约束。

例如，表中的所有分区共享相同的列和约束定义。但是，每个分区可以具有单独的物理属性，例如它所属的表空间。

分区键

分区键是一组一列或多列，用于确定分区表中每行应该到达的分区。每行明确地分配给单个分区。

在sales表中，您可以将time_id列指定为范围分区的键。数据库根据此列中的日期是否落在指定范围内，将行分配给分区。Oracle数据库使用分区键自动将插入，更新和删除操作定向到相应的分区。

分区策略

Oracle Partitioning提供了几种分区策略，用于控制数据库如何将数据放入分区。基本策略是范围，列表和散列分区。

单级分区仅使用一种数据分发方法，例如，仅列出分区或仅进行范围分区。在复合分区中 ，通过一种数据分配方法对表进行分区，然后使用第二种数据分配方法将每个分区进一步划分为子分区。例如，您可以为channel_id使用列表分区，为time_id使用范围子分区。

区间分区是范围分区的扩展。

如果插入的数据超出现有范围分区，则Oracle数据库会自动创建指定时间间隔的分区。例如，您可以创建一个销售历史记录表，用于在单独的分区中存储每个月的数据。

间隔分区使您可以避免显式创建范围分区。您可以对范围分区的每个表使用间隔分区，并对新分区使用固定间隔。除非您创建具有不同间隔的范围分区，或者除非始终设置特定分区属性，否则请考虑使用间隔分区。

按间隔分区时，必须至少指定一个范围分区。范围分区键值确定范围分区的高值，称为转换点 。数据库会自动为值超出转换点的数据创建间隔分区。每个间隔分区的下边界是前一个范围或间隔分区的包含上边界。因此，在示例4-2中 ，值01-JAN-2011在分区p2 。

参考分区

在引用分区中 ，子表的分区策略仅通过与父表的外键关系来定义。对于父表中的每个分区，子表中只存在一个对应的分区。父表将父记录存储在特定分区中，子表将子记录存储在相应的分区中。

例如， orders表是line_items表的父line_items ，其中主键和外键在order_id定义。表格按引用分区。例如，如果数据库存储在分区订购233 Q3_2015的orders ，那么数据库中存储的所有行项目在分区顺序233 Q3_2015的line_items 。如果将分区Q4_2015添加到orders ，则数据库会自动将Q4_2015添加到line_items 。

引用分区的优点是：

  • 通过对父表和子表使用相同的分区策略，可以避免重复所有分区键列。该策略减少了非规范化的手动开销，并节省了空间。

  • 父表上的维护操作自动发生在子表上。例如，将分区添加到主表时，数据库会自动将此添加内容传播到其后代。

  • 数据库自动使用父表和子表中分区的分区连接，从而提高性能。

您可以将引用分区与所有基本分区策略一起使用，包括间隔分区。您还可以将引用分区表创建为复合分区表。

复合分区

在复合分区中 ，表由一种数据分配方法分区，然后使用第二种数据分配方法将每个分区进一步细分为子分区。因此，复合分区结合了基本数据分配方法。给定分区的所有子分区都代表数据的逻辑子集。

复合分区具有以下几个优点：

  • 根据SQL语句，一维或二维上的分区修剪可以提高性能。

  • 查询可能能够在任一维度上使用完全或部分分区连接。

  • 您可以执行单个表的并行备份和恢复。

  • 分区数大于单级分区，这可能有利于并行执行。

  • 如果许多语句可以从分区修剪或分区连接中受益，则可以实现滚动窗口以支持历史数据，并仍在另一个维度上进行分区。

  • 您可以根据分区键的标识以不同方式存储数据。例如，您可能决定以只读的压缩格式存储特定产品类型的数据，并保持其他产品类型数据不压缩。

本地分区索引

在本地分区索引中 ，索引在相同的列上进行分区，具有相同数量的分区和与其表相同的分区范围。

每个索引分区只与基础表的一个分区相关联，因此索引分区中的所有键仅指存储在单个表分区中的行。通过这种方式，数据库自动将索引分区与其关联的表分区同步，使每个表索引对独立。

本地分区索引在数据仓库环境中很常见。本地索引具有以下优点：

  • 可用性增加，因为在分区中使数据无效或不可用的操作仅影响此分区。

  • 分区维护得以简化。移动表分区时，或者当数据超出分区时，只能重建或维护关联的本地索引分区。在全局索引中，必须重建或维护所有索引分区。

  • 如果发生分区的时间点恢复，则可以将索引恢复到恢复时间（请参阅“ 数据文件恢复 ” ）。整个索引不需要重建。

CREATE INDEX hash_sales_idx ON hash_sales（time_id）LOCAL;

本地前缀和非前缀索引

本地分区索引可以是前缀或非前缀的。

索引子类型定义如下：

  • 本地前缀索引

    在这种情况下，分区键位于索引定义的前沿。在Range Partitioning中的time_range_sales示例中，表在time_id上按范围分区。此表上的本地前缀索引将time_id作为其列表中的第一列。

  • 本地非前缀索引

    在这种情况下，分区键不在索引列列表的前沿，并且根本不需要在列表中。在本地分区索引中的hash_sales_idx示例中，索引是本地非前缀的，因为分区键product_id不在前沿。

两种类型的索引都可以利用分区消除 （也称为分区修剪 ），当优化器通过排除考虑的分区来加速数据访问时，会发生分区消除 。查询是否可以消除分区取决于查询谓词 。使用本地前缀索引的查询始终允许索引分区消除，而使用本地非前缀索引的查询可能不允许。

全局分区索引

全局分区索引是B树索引，它独立于创建它的基础表进行分区。单个索引分区可以指向任何或所有表分区，而在本地分区索引中，索引分区和表分区之间存在一对一奇偶校验。

通常，全局索引对OLTP应用程序很有用，因为快速访问，数据完整性和可用性非常重要。在OLTP系统中，表可以由一个键分区，例如employees.department_id列，但是应用程序可能需要使用许多不同的键访问数据，例如， employee_id或job_id 。全局索引在此方案中很有用。

部分索引是与关联的分区表的索引属性相关联的索引。

通过关联，您可以指定索引哪些表分区。部分索引具有以下优点：

  • 未编制索引的表分区可避免消耗不必要的索引存储空间。

  • 负载和查询的性能可以提高。

    在Oracle Database 12c之前，交换分区操作需要对关联的全局索引进行物理更新，以使其保持可用状态。从Oracle Database 12c开始 ，如果分区维护操作中涉及的分区不是部分全局索引的一部分，则索引仍然可用，而无需任何全局索引维护。

  • 如果在创建索引时仅索引某些表分区，并且稍后索引其他分区，则可以减少索引创建所需的排序空间。

分片表概述

在Oracle数据库中，分片使您可以将大型表拆分为更易于管理的部分，称为分片，可以存储在多个数据库中。

每个数据库都托管在具有自己的本地资源（CPU，内存，闪存或磁盘）的专用服务器上。 在这种配置中的每个数据库称为碎片 。 所有分片一起组成一个逻辑数据库，称为分片数据库（SDB） 。

水平分区涉及跨分片拆分数据库表，以便每个分片包含具有相同列但行的不同子集的表。以这种方式分割的表也称为分片表 。

分片基于无共享硬件基础架构，它消除了单点故障，因为分片不共享物理资源，如CPU，内存或存储设备。碎片在软件方面也松散耦合;他们不运行集群件。

碎片通常托管在专用服务器上。这些服务器可以是商用硬件或工程系统。分片可以在单实例或Oracle RAC数据库上运行。它们可以放置在本地，云端，或混合本地和云配置中。

从数据库管理员的角度来看，SDB由多个可以集体或单独管理的数据库组成。但是，从应用程序的角度来看，SDB看起来像一个单独的数据库：这些分片中的分片数量和数据分布对数据库应用程序是完全透明的。

分片适用于适用于分片数据库体系结构的自定义OLTP应用程序。使用分片的应用程序必须具有明确定义的数据模型和数据分布策略（一致的散列，范围，列表或复合），主要使用分片键访问数据。分片键的示例包括customer_id ， account_no或country_id 。

Sharded Tables

分片表是一个表，在多个数据库之间划分为更小且更易于管理的部分，称为分片。

分片的主要好处包括：

  • 线性可扩展性 。分片消除了性能瓶颈，并且可以通过添加分片来线性扩展性能和容量。

  • 故障遏制 。Sharding是一种无共享硬件基础架构，可消除单点故障，例如共享磁盘，SAN和集群件，并提供强大的故障隔离 - 一个分片的故障或速度降低不会影响其他分片的性能或可用性。

  • 数据的地理分布 。当数据必须位于特定的管辖区域时，分片可以将特定数据存储在靠近其消费者的位置并满足法规要求。

  • 滚动升级 。一次在一个分片上应用配置更改不会影响其他分片，并允许管理员首先测试一小部分数据的更改。

  • 云部署的简单性 。分片非常适合在云中部署。可以根据需要调整碎片的大小以适应可用的任何云基础设施并且仍然实现所需的服务级别。Oracle Sharding支持内部部署，云和混合部署模型。

应用程序将数据库池视为单个逻辑数据库。通过向池中添加数据库（分片），应用程序可以在任何平台上弹性地将数据，事务和用户扩展到任何级别。Oracle数据库支持最多可扩展1000个分片。

分片和分区

Oracle Sharding本质上是分布式分区，因为它通过支持跨分片的表分区分发来扩展分区。

基于分片键，分区在表空间级别的分片中分布。密钥的示例包括客户ID，帐号和国家ID。分片键支持以下数据类型：

  • 数

  • 整数

  • SMALLINT

  • 生的

  • （N）VARCHAR

  • （N）VARCHAR2

  • （N）CHAR

  • 日期

  • TIMESTAMP

分片表的每个分区都驻留在一个单独的表空间中，每个表空间都与一个特定的分片相关联。根据分片方法，可以自动建立关联或由管理员定义关联。

即使分片表的分区驻留在多个分片中，对于应用程序，该表的外观和行为与单个数据库中的分区表完全相同。应用程序发出的SQL语句永远不必引用分片或依赖于分片数量及其配置。

例4-5 Sharded Table

熟悉的表分区SQL语法指定了如何跨分片对行进行分区。例如，以下SQL语句创建一个分片表，根据分片键cust_id在分片上水平分区表：

CREATE SHARDED TABLE客户（cust_id NUMBER NOT NULL，名称VARCHAR2（50），地址VARCHAR2（250），区域VARCHAR2（20），类VARCHAR2（3），注册DATE CONSTRAINT cust_pk PRIMARY KEY（cust_id））PARTITION BY CONSISTENT HASH（cust_id ）PARTITIONS AUTO TABLESPACE SET ts1;

上表由一致哈希分区，哈希是一种在可伸缩分布式系统中常用的特殊类型的哈希分区。此技术自动跨分片传播表空间，以提供均匀的数据和工作负载分布。请注意，不支持分片表上的全局索引，但支持本地索引。

表空间集

Oracle Sharding创建和管理表空间作为一个称为表空间集的单元。PARTITIONS AUTO子句指定应自动确定分区数。这种类型的散列在分片之间迁移数据方面提供了更大的灵活性和效率，这对于弹性可伸缩性非常重要。

表空间是SDB中数据分布的逻辑单元。跨分片的分区分布是通过在驻留在不同分片上的表空间中自动创建分区来实现的。为了最小化多分片连接的数量，相关表的相应分区始终存储在同一分片中。分片表的每个分区都存储在单独的表空间中。

注意：

表空间集仅支持Oracle托管文件。

不能独立于整个表空间集删除或更改单个表空间。

TABLESPACE SET不能与用户定义的分片方法一起使用。

视图概述

视图是一个或多个表的逻辑表示。实质上，视图是存储的查询。

视图从其所基于的表中派生其数据，称为基表 。基表可以是表或其他视图。对视图执行的所有操作实际上都会影响基表。您可以在大多数使用表的地方使用视图。

注意：

物化视图使用标准视图中的不同数据结构。

通过视图，您可以为不同类型的用户定制数据表示。视图通常用于：

  • 通过限制对表的一组预定行或列的访问来提供额外级别的表安全性

    例如， 图4-8显示了staff视图如何不显示基表employees的salary或commission_pct列。

  • 隐藏数据复杂性

    例如，可以使用连接定义单个视图， 连接是多个表中相关列或行的集合。但是，该视图隐藏了这一信息实际上源自几个表的事实。查询还可能使用表信息执行大量计算。因此，用户可以在不知道如何执行连接或计算的情况下查询视图。

  • 以与基表不同的视角呈现数据

    例如，可以重命名视图的列，而不会影响视图所基于的表。

  • 将应用程序与基表定义的更改隔离开来

观点的特征

与表不同，视图不是分配存储空间，视图也不包含数据。相反，视图由查询定义，该查询从视图引用的基表中提取或派生数据。因为视图基于其他对象，所以除了用于定义数据字典中的视图的查询的存储之外，它不需要存储。

视图依赖于其引用的对象，这些对象由数据库自动处理。例如，如果删除并重新创建视图的基表，则数据库将确定新基表是否为视图定义所接受。

视图中的数据操作

因为视图是从表中派生的，所以它们有许多相似之处。用户可以查询视图，并且通过一些限制，他们可以在视图上执行DML。对视图执行的操作会影响视图的某个基表中的数据，并受基本表的完整性约束和触发器的影响。

以下示例创建hr.employees表的视图：

CREATE VIEW staff_dept_10 AS SELECT employee_id，last_name，job_id，manager_id，department_id FROM employees WHERE department_id = 10 WITH CHECK OPTION CONSTRAINT staff_dept_10_cnst;

定义查询仅引用部门10的行。CHECK OPTION使用约束创建视图，以便针对视图发出的INSERT和UPDATE语句不会导致视图无法选择的行。因此，可以插入部门10中的雇员的行，但不插入部门30的行。

如何在视图中访问数据

Oracle数据库将视图定义存储在数据字典中，作为定义视图的查询的文本。

在SQL语句中引用视图时，Oracle数据库将执行以下任务：

  1. 将查询（如果可能）与具有定义视图和任何基础视图的查询的视图合并

    Oracle数据库优化合并查询，就像您在不引用视图的情况下发出查询一样。因此，Oracle数据库可以在任何引用的基表列上使用索引，无论这些列是在视图定义中引用还是在针对视图的用户查询中引用。

    有时，Oracle数据库无法将视图定义与用户查询合并。在这种情况下，Oracle数据库可能不会使用引用列上的所有索引。

  2. 解析共享SQL区域中的合并语句

    仅当没有现有共享SQL区域包含类似语句时，Oracle数据库才会解析引用新共享SQL区域中的视图的语句。因此，视图提供了与共享SQL相关的减少内存使用的好处。

  3. 执行SQL语句

以下示例说明了查询视图时的数据访问。假设您根据employees和departments表创建employees_view ：

CREATE VIEW employees_view AS SELECT employee_id，last_name，salary，location_id FROM employees JOIN departments USING（department_id）WHERE department_id = 10;

用户执行以下employees_view查询：

SELECT last_name FROM employees_view WHERE employee_id = 200;

Oracle数据库合并视图和用户查询以构造以下查询，然后执行该查询以检索数据：

SELECT last_name FROM employees，departments WHERE employees.department_id = departments.department_id AND departments.department_id = 10 AND employees.employee_id = 200;

为了具有固有的可更新性，视图必须满足几个标准。例如，一般规则是连接视图上的INSERT ， UPDATE或DELETE操作一次只能修改一个基表。以下USER_UPDATABLE_COLUMNS数据字典视图查询显示staff_dept_10_30视图是可更新的：

对象视图

正如视图是虚拟表一样， 对象视图是虚拟对象表。视图中的每一行都是一个对象，它是对象类型的一个实例。对象类型是用户定义的数据类型。

您可以检索，更新，插入和删除关系数据，就像它存储为对象类型一样。您还可以使用作为对象数据类型的列定义视图，例如对象， REF和集合（嵌套表和VARRAY ）。

与关系视图一样，对象视图只能呈现数据库管理员希望用户看到的数据。例如，对象视图可以显示有关IT程序员的数据，但可以省略有关工资的敏感数据。以下示例基于此对象创建employee_type对象，然后创建视图it_prog_view ：

CREATE TYPE employee_type AS OBJECT（employee_id NUMBER（6），last_name VARCHAR2（25），job_id VARCHAR2（10））; / CREATE VIEW it_prog_view employee_type WITH OBJECT IDENTIFIER（employee_id）AS SELECT e.employee_id，e.last_name，e.job_id FROM employees e WHERE job_id ='IT_PROG';

对象视图在原型设计或转换到面向对象的应用程序时非常有用，因为视图中的数据可以从关系表中获取并访问，就像将表定义为对象表一样。您可以运行面向对象的应用程序，而无需将现有表转换为不同的物理结构。

物化视图概述

物化视图是预先作为模式对象存储或“物化”的查询结果。查询的FROM子句可以命名表，视图或物化视图。

物化视图通常用作复制中的主表和数据仓库中的事实表 。物化视图汇总，计算，复制和分发数据。它们适用于各种计算环境，例如：

  • 在数据仓库中，物化视图可以计算和存储从总和函数（例如总和和平均值）生成的数据。

    摘要是一种聚合视图，它通过预先计算连接和聚合操作并将结果存储在表中来缩短查询时间。物化视图等同于摘要。您还可以使用物化视图来计算有或没有聚合的连接。

  • 在使用XStream和Oracle GoldenGate实现的物化视图复制中，视图包含单个时间点的表的完整或部分副本。物化视图在分布式站点复制数据并同步在多个站点执行的更新。当数据库不总是连接到网络时，这种复制形式适用于现场销售等环境。

  • 在移动计算环境中，物化视图可以将数据子集从中央服务器下载到移动客户端，定期刷新中央服务器并将客户端更新传播到中央服务器。

物化视图共享索引和非物化视图的一些特征。

  • 它们包含实际数据并占用存储空间。

  • 当主表中的数据发生更改时，可以刷新它们。

  • 当用于查询重写操作时，它们可以提高SQL执行的性能。

  • 它们的存在对SQL应用程序和用户是透明的。

物化视图的刷新方法

数据库通过在更改基表后刷新数据来维护物化视图中的数据。刷新方法可以是增量或完全刷新。

完全刷新

完整刷新将执行定义实例化视图的查询。最初创建实例化视图时会发生完全刷新，除非实体化视图引用预构建表，或者将表定义为BUILD DEFERRED 。

完全刷新可能很慢，尤其是在数据库必须读取和处理大量数据的情况下。您可以在创建物化视图后随时执行完整刷新。

增量刷新

增量刷新 （也称为快速刷新 ）仅处理对现有数据的更改。此方法无需从头开始重建实体化视图。仅处理更改可能会导致非常快的刷新时间。

您可以按需或定期刷新物化视图。或者，您可以在与基表相同的数据库中配置实例化视图，以便在事务提交对基表的更改时进行刷新。

快速刷新有以下两种形式：

  • 基于日志的刷新

    在这种类型的刷新中，物化视图日志或直接加载器日志会记录对基表的更改。物化视图日志是一种模式对象，它记录对基表的更改，以便可以递增地刷新在基表上定义的物化视图。每个物化视图日志都与单个基表相关联。

  • 分区更改跟踪（PCT）刷新

    PCT刷新仅在基表分区时有效。PCT刷新会删除受影响的物化视图分区或受影响的数据部分中的所有数据，然后重新计算它们。数据库使用修改后的基表分区来标识视图中受影响的分区或数据部分。在基表上发生分区维护操作时，PCT刷新是唯一可用的增量刷新方法。

就地和不合适的刷新

对于完整和增量方法，数据库可以就地刷新实例化视图，直接在视图上刷新语句，或者不合适。

异地刷新会创建一个或多个外部表，对它们执行刷新语句，然后使用外部表切换实例化视图或受影响的分区。此技术在刷新期间实现了高可用性，尤其是在刷新语句需要很长时间才能完成时。

Oracle Database 12c引入了同步刷新，这是一种异地刷新。同步刷新不会修改基表的内容，而是使用同步刷新包中的API，通过同时将这些更改应用于基表和物化视图来确保一致性。此方法使一组表和在其上定义的物化视图始终同步。在数据仓库中，同步刷新方法非常适合，原因如下：

  • 增量数据的加载受到严格控制，并定期发生。

  • 表及其物化视图通常以相同的方式分区，或者它们的分区通过功能依赖关联。

尺寸概述

典型的数据仓库有两个重要组件：维度和事实。

维度是用于指定业务问题的任何类别，例如，时间，地理位置，产品，部门和分销渠道。事实是与特定维度值集合相关联的事件或实体，例如，销售单位或利润。

多维请求的示例包括以下内容：

  • 显示所有产品的总销售额，在2013年和2014年的地理位置（从州到国家/地区）的聚合级别不断提高。

  • 对我们的运营进行跨表格分析，显示2013年和2014年南美洲地区的开支。包括所有可能的小计。

  • 根据2014年汽车产品销售收入列出亚洲十大销售代表，并对其佣金进行排名。

许多多维问题需要汇总数据和数据集比较，通常跨时间，地理或预算。

创建维度允许更广泛地使用查询重写功能。通过透明地重写查询以使用实体化视图，数据库可以提高查询性能。

维度的层次结构

维度表是一种逻辑结构，用于定义列对或列集之间的分层（父/子）关系。

例如，维度可以指示一行中的city列表示state列的值，而state列表示country列的值。

在客户层面，客户可以前往城市，州，国家，次区域和地区。数据分析通常从维度层次结构中的较高级别开始，如果情况需要进行此类分析，则逐渐向下钻取。

子级别的每个值都与父级别的一个且仅一个值相关联。层次关系是从层次结构的一个级别到层次结构中的下一个级别的功能依赖性。

维度没有分配给它的数据存储。维度信息存储在维度表中，而事实信息存储在事实表中。

同义词概述

同义词是架构对象的别名。例如，您可以为表或视图，序列，PL / SQL程序单元，用户定义的对象类型或其他同义词创建同义词。因为同义词只是一个别名，所以除了数据字典中的定义之外，它不需要存储。

同义词可以简化数据库用户的SQL语句。同义词对于隐藏底层模式对象的标识和位置也很有用。如果必须重命名或移动基础对象，则只能重新定义同义词。基于同义词的应用程序继续工作而无需修改。

您可以创建私人和公共同义词。私有同义词位于特定用户的架构中，该用户可以控制其他人的可用性。公共同义词由名为PUBLIC的用户组拥有，并且每个数据库用户都可以访问。
