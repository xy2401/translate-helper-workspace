服务器池和基于策略的管理概述

Oracle Clusterware管理的资源包含在称为服务器池的逻辑服务器组中。这是在Oracle Clusterware 11g第 2版（11.2）中引入的。

资源托管在共享基础架构上，并包含在服务器池中 。Oracle Clusterware管理的资源示例包括数据库实例，数据库服务，应用程序VIP和应用程序组件。

在Oracle Cluster中，您可以使用服务器池在集群成员节点上运行特定类型的工作负载，同时提供简化的管理选项。您可以使用群集配置策略集来跨群集提供群集策略的动态管理。

您可以使用Oracle Clusterware 11g第 2版（11.2）服务器池模型继续管理Oracle Clusterware标准群集中的资源，也可以使用传统的固定非服务器池方法手动管理资源。

服务器池和基于策略的管理

通过基于策略的管理，管理员可以指定运行服务器的服务器池（不包括通用池和空闲池）。

例如，数据库管理员使用SRVCTL为托管数据库或数据库服务的服务器创建服务器池。集群件管理员使用CRSCTL为非数据库使用创建服务器池，例如为托管应用程序的服务器创建服务器池。

基于政策的管理：

  • 根据定义的策略启用在线服务器重新分配，以满足工作负载容量要求

  • 保证为策略定义的关键工作分配所需资源

  • 确保在必要时进行隔离，以便您可以在群集中为应用程序和数据库提供专用服务器

  • 允许将策略配置为根据业务需求或应用程序需求更改池，以便池在适当的时间提供所需的容量

服务器池提供资源隔离，以防止在一个服务器池中运行的应用程序访问在另一个服Oracle Clusterware在服务器池之间提供细粒度的角色分离。此功能在具有由不同组管理的集群环境的组织中维护这些组之间所需的管理角色分离。

Oracle Clusterware可以有效地在群集中分配服务器。创建服务器池时定义的服务器池属性根据IMPORTANCE服务器池属性指定服务器的放置和优先级。

服务器池如何工作

服务器池将群集划分为托管单例和统一应用程序的逻辑服务器组。应用程序可以是数据库服务或非数据库应用程序。当应用程序工作负载分布在服务器池中的所有服务器上时，应用程序是统一的。当应用程序在服务器池中的单个服务器上运行时，它是单例 。Oracle Clusterware角色分离管理确定对服务器池的访问和使用。

您使用服务器控制（SRVCTL）实用程序管理包含Oracle RAC数据库的服务器池。使用Oracle Clusterware Control（CRSCTL）实用程序来管理所有其他服务器池。只有群集管理员才有权创建顶级服务器池。

数据库管理员使用服务器控制（SRVCTL）实用程序来创建和管理将包含Oracle RAC数据库的服务器池。群集管理员使用Oracle Clusterware Control（CRSCTL）实用程序来创建和管理所有其他服务器池，例如非数据库应用程序的服务器池。只有群集管理员才有权创建顶级服务器池。

顶级服务器池：

  • 逻辑划分群集

  • 始终是独占的，这意味着一个服务器在某个特定时间点只能驻留在一个特定的服务器池中

默认服务器池

安装Oracle Clusterware时，会自动创建两个内部服务器池： 通用和 免费最初，新安装中的所有服务器都分配给Free服务器池。服务器自动从Free迁移到新定义的服务器池。

免费服务器池

该空闲服务器池包含未分配给任何其他服务器池的服务器。Free服务器池的属性受到限制，如下所示：

  • 用户无法编辑SERVER_NAMES ， MIN_SIZE和MAX_SIZE

  • 用户可以编辑IMPORTANCE和ACL

通用服务器池

该通用服务器池存储不在顶级服务器池中且不受策略管理的任何服务器。托管非策略管理的应用程序的服务器（例如管理员管理的数据库）将静态分配给通用服务器池。

通用服务器池的属性受到限制，如下所示：

  • 没有人可以修改通用服务器池的配置属性（所有属性都是只读的）

  • 如果要在其上创建数据库的服务器是以下之一，则只能在通用池中创建管理员管理的数据库：

    • 在线并存在于通用服务器池中

    • 在线并存在于免费服务器池中，在这种情况下，Oracle Clusterware将服务器移动到通用服务器池中

    • 在线并存在于任何其他服务器池中，并且用户是集群管理员或允许使用服务器池的服务器，在这种情况下，服务器将移动到通用服务器池中

    • 脱机，用户是群集管理员

您可以使用SRVCTL或CRSCTL分别为数据库和其他应用程序创建服务器池。

如果使用SRVCTL创建服务器池，则只能使用本节中描述的服务器池属性的子集。如果使用CRSCTL创建服务器池，则可以使用整组服务器池属性。

Oracle Clusterware如何使用服务器池分配新服务器

Oracle Clusterware按以下顺序将新服务器分配给服务器池：

  1. 通用服务器池

  2. 用户创建的服务器池

  3. 免费服务器池

Oracle Clusterware继续将服务器分配给服务器池，直到满足以下条件：

  1. 直到所有服务器池按其最小值（ MIN_SIZE ）的重要性顺序填充。

  2. 直到所有服务器池按其最大值（ MAX_SIZE ）的重要性顺序填充。

  3. 默认情况下，未放置在服务器池中的任何服务器都将进入免费服务器池。

    您可以修改免费服务器池的IMPORTANCE属性。如果空闲服务器池的IMPORTANCE属性的值大于一个或多个其他服务器池，则一旦满足其MIN_SIZE属性的值，空闲服务器池将接收任何剩余的服务器。

  • 对于IMPORTANCE值低于缺陷服务器池的服务器池，Oracle Clusterware可以从这些服务器池中获取服务器，即使这意味着服务器数量低于MIN_SIZE属性的值。

使用默认属性管理服务器池

默认情况下，每个服务器池都配置有以下用于管理服务器池的属性选项：

  • MIN_SIZE ：服务器池应包含的最小服务器数。

    如果服务器池中的服务器数低于此属性的值，则Oracle Clusterware会自动将服务器从其他位置移动到服务器池中，直到服务器数达到属性值。

  • MAX_SIZE ：服务器池应包含的最大服务器数。

  • IMPORTANCE ：从0到1000排名集群中的所有其他服务器池之间的服务器池中的号码（0为最重要的）。

此外，您还可以分配其他属性以提供更精细的服务器池管理，作为其中的一部分群集配置策略。EXCLUSIVE_POOLS和SERVER_CATEGORY等属性可以帮助您为服务器池创建策略，从而提高性能并在服务器池中构建调优设计管理。

服务器分类概述

通过服务器分类，您可以使用处理器类型，内存和其他区分系统功能等属性将服务器组织到特定类别中。

您可以配置服务器池以将池的合格成员限制为共享特定属性集的服务器类别。最初，服务器池被限制为一组基本属性，表征服务器属于给定池，无法区分服务器类型;所有服务器都被认为与其处理器，物理内存和其他特性相同。服务器分类现在提供了区分这些服务器的方法。

注意：

如果创建服务管理（Oracle数据库QoS管理）的Oracle数据库质量方针，然后通过设置服务器池指令覆盖分类的服务器，并使用CRSCTL命令policy和policyset名词被禁用。此外，如果您从使用Oracle Clusterware策略切换到使用Oracle数据库QoS管理策略，则会替换Oracle Clusterware策略，因为这两种策略类型无法共存。Oracle建议您在切换策略之前使用crsctl status policyset -file file_name创建备份。

负载感知资源放置

您可以配置数据库，以便Oracle Clusterware了解给定数据库的CPU要求和限制。

Oracle Clusterware使用此信息仅将数据库资源放置在具有足够CPU数，内存量或两者的服务器上。

如果已在Oracle Clusterware中配置了具有CPU或内存要求的资源，则Oracle Clusterware将仅尝试在满足这些要求的服务器上启动这些资源。对于数据库资源，具体而言，您可以将CPU或内存值分别配置为CPU_COUNT和MEMORY_TARGET实例参数。

注意：

配置实例参数需要以下内容：

  • 对于CPU（实例囚笼），必须启用资源管理器

  • 对于内存，必须为数据库使用自动内存管理

添加或修改数据库实例时，可以配置负载感知资源放置，如下所示：

$ srvctl modify database -db db_unique_name -cpucount cpu_count -memorytarget memory_target

在前面的示例中， cpu_count指的是工作负载CPU的数量， memory_target指的是资源使用的目标内存（MB）。

数据库服务器的内存压力管理

由于过多的打开会话或失控的工作负载，企业数据库服务器可以使用所有可用内存。内存不足可能导致事务失败，或者在极端情况下，导致服务器重新启动以及丢失应用程序的宝贵资源。Oracle数据库QoS管理实时检测服务器上的内存压力，并将新会话重定向到其他服务器，以防止使用压力服务器上的所有可用内存。

启用Oracle数据库QoS管理并管理Oracle Clusterware服务器池时，Cluster Health Monitor会发送一个度量流，该群集流可将有关群集服务器的内存资源的实时信息提供给Oracle数据库QoS管理。此信息包括以下内容：

  • 可用内存量

  • 当前正在使用的内存量

如果Oracle数据库QoS管理确定节点处于内存压力下，则Oracle Clusterware管理的数据库服务将在该节点上停止，从而阻止创建新连接。释放内存压力后，该节点上的服务将自动重新启动，并且侦听器开始向该服务器发送新连接。内存压力可以通过多种方式缓解（例如，通过关闭现有会话或通过用户干预）。

将新会话重新路由到不同的服务器可以保护内存压力服务器上的现有工作负载，并使服务器保持可用状态。管理服务器的内存压力为管理Oracle RAC数据库上托管的应用程序的服务级别增加了新的资源保护功能。

值为0阻止重新平衡操作。然后可以在以后进行手动或自动重新平衡。您可能希望推迟重新平衡，直到对数据库的需求减少或完成所有磁盘组更改为止。

执行以下过程以使磁盘联机或脱机：

    • 使用ASM_POWER_LIMIT为Rebalance Power设置的值必须是整数。页面上的文本描述了Rebalance Power的值如何影响快速镜像重新同步操作。如果ASM_POWER_LIMIT设置为0 ，则Oracle ASM不会执行自动重新平衡。有关重新平衡电源设置和ASM_POWER_LIMIT初始化参数的信息，请参阅“ ASM_POWER_LIMIT ” 。

    • 默认情况下，Rebalance Power值使用ASM_POWER_LIMIT初始化参数的值。

    • Oracle Enterprise Manager显示“脱机确认”页面，此页面上的文本说明在Oracle Enterprise Manager使所选磁盘脱机后，Oracle ASM实例将等待您为DISK_REPAIR_TIME初始化参数设置的值指定的时间间隔在Oracle ASM删除磁盘之前。在磁盘修复期间，您可以修复磁盘，然后将它们重新联机。对于您成功联机的每个磁盘，将取消挂起的删除操作并开始重新同步操作。

删除磁盘组

删除磁盘组时，可以选择删除磁盘组及其所有文件。如果任何数据库文件处于打开状态，则无法删除磁盘组。仅当您指定包含 内容选项时，Oracle ASM才会删除所有磁盘组内容。删除磁盘组后，可以将其成员磁盘添加到其他磁盘组或将其用于其他目的。

删除磁盘组的一个原因是更改冗余级别（正常，高或外部）。由于无法更改磁盘组的冗余，因此必须删除磁盘组，然后使用适当的冗余重新创建磁盘组。在这种情况下，您必须在删除磁盘组之前备份或移动磁盘组数据。

要删除磁盘组：

配置磁盘组兼容性属性

为磁盘组指定磁盘 兼容性值，以使系统能够使用最新的Oracle ASM功能。

注意：

您无法撤消磁盘组兼容性值的设置。

  • ASM 兼容性

    指定值，该值是Oracle ASM实例装入此磁盘组所需的最低软件版本。

  • 数据库 兼容性

    指定必须小于或等于Oracle ASM兼容性的值。这是数据库实例使用磁盘组中的文件所需的最低软件版本。如果您使用的数据库兼容性值大于Oracle ASM兼容性，则Oracle Enterprise Manager会显示错误。

    注意：

    您无法修改处于滚动迁移状态的Oracle ASM群集的数据库兼容性。

  • ASM 卷

    指定11.2或更高的值以使用Oracle ASM Dynamic Volume Manager（Oracle ADVM）功能。必须将磁盘组的ASM兼容性设置为11.2或更高才能启用Oracle ADVM。此外，必须在受支持的环境中加载Oracle ADVM卷驱动程序。

有关磁盘组兼容性属性的信息，请参阅“ 磁盘组兼容性 ” 。

34.1.1 DML和DDL事务

分布式事务中支持某些DML和DDL操作。

以下是分布式事务中支持的DML和DDL操作：

  • CREATE TABLE AS SELECT

  • 删除

  • INSERT （默认和直接加载）

  • UPDATE

  • LOCK TABLE

  • 选择

  • SELECT FOR UPDATE

您可以并行执行DML和DDL语句，并按顺序INSERT直接加载语句，但请注意以下限制：

  • 所有远程操作都必须是SELECT语句。

  • 这些语句不能是另一个分布式事务中的子句。

  • 如果INSERT ， UPDATE或DELETE语句的table_expression_clause中引用的表是远程的，则执行是串行而不是并行。

  • 发出并行DML / DDL或直接加载INSERT后，您无法执行远程操作。

  • 如果事务开始使用XA或OCI，它将按顺序执行。

  • 不能对源自并行操作的事务执行环回操作。例如，您不能引用实际上是本地对象的同义词的远程对象。

  • 如果在事务中执行除SELECT之外的分布式操作，则不会并行化DML。

34.2.6.3承诺点强度

在准备阶段开始时确定的提交点站点仅从参与事务的节点中选择。发生以下事件序列：

  1. 在全局协调器直接引用的节点中，数据库选择具有最高提交点强度的节点作为提交点站点。

  2. 最初选择的节点确定它必须从中获取该事务的信息的任何节点是否具有更高的提交点强度。

  3. 在事务中直接引用的具有最高提交点强度的节点或具有较高提交点强度的其中一个服务器将成为提交点站点。

  4. 在确定最终提交点站点之后，全局协调器向参与该事务的所有节点发送准备响应。

34.3.1关于两阶段提交机制

与本地数据库上的事务不同，分布式事务涉及更改多个数据库上的数据。因此，分布式事务处理更复杂，因为数据库必须协调提交或回滚事务中的更改作为独立单元。换句话说，整个事务提交，或整个事务回滚。

数据库使用两阶段提交机制确保分布式事务中数据的完整性。在准备阶段 ，事务中的发起节点要求其他参与节点承诺提交或回滚事务。在提交阶段 ，发起节点要求所有参与节点提交事务。如果无法获得此结果，则会要求所有节点回滚。

分布式事务中的所有参与节点应执行相同的操作：它们应该全部提交或全部执行事务的回滚。数据库自动控制和监视分布式事务的提交或回滚，并使用两阶段提交机制维护全局数据库 （参与事务的数据库集合）的完整性。这种机制是完全透明的，不需要用户或应用程序开发人员进行编程。

34.3.2.1关于准备阶段

  • 记录重做日志中的信息，以便随后可以提交或回滚事务，而不管中间的故障

  • 在已修改的表上放置分布式锁，以防止读取

当节点响应它准备提交的全局协调器时，准备好的节点承诺稍后提交或回滚事务，但不会单独决定是提交还是回滚事务。承诺意味着如果此时发生实例故障，则节点可以使用联机日志中的重做记录将数据库恢复回准备阶段。

34.3.2.3准备阶段的步骤

两阶段提交过程中的准备阶段包括特定步骤。

  2. 节点检查事务是否更改自身或其后代的数据。如果数据没有变化，则节点跳过剩余的步骤并返回只读响应（参见“ 只读响应 ” ）。

这些操作保证节点可以随后在节点上提交或回滚事务。然后，准备好的节点等待，直到从全局协调器接收到COMMIT或ROLLBACK请求。

分布式事务可能会在以下方面变得不确定：

  • 运行Oracle数据库软件的服务器系统崩溃

  • 断开了分布式处理中涉及的两个或多个Oracle数据库之间的网络连接

  • 发生未处理的软件错误

当系统，网络或软件问题得到解决时，RECO进程会自动解决不确定的事务。在RECO可以解决事务之前，数据将被锁定以进行读取和写入。数据库阻止读取，因为它无法确定要为查询显示哪个版本的数据。

34.4.4疑难事务的系统变更号的相关性

系统更改号 （SCN）是已提交的数据库版本的内部时间戳。Oracle数据库服务器使用SCN时钟值来保证事务的一致性。

例如，当用户提交事务时，数据库会在重做日志中记录此提交的SCN。

数据库使用SCN来协调不同数据库之间的分布式事务。例如，数据库以下列方式使用SCN：

  1. 应用程序使用数据库链接建立连接。

  2. 分布式事务在所涉及的所有数据库中提交最高的全局SCN。

  3. 提交全局SCN将发送到事务中涉及的所有数据库。

SCN对于分布式事务很重要，因为它们用作事务的同步提交时间戳，即使事务失败也是如此。如果事务变得不确定，管理员可以使用此SCN来协调对全局数据库所做的更改。事务提交的全局SCN也可用于稍后识别事务，例如，在分布式恢复中。

注意：

您无法将保护模式从最高性能模式更改为最大保护模式。您必须先将保护模式更改为最大可用性，然后再更改为最大保护模式。

6.7场景6：启用快速启动故障转移并启动观察器

您可以在连接到代理配置中的任何数据库时，从任何站点（包括观察站点）启用快速启动故障转移。

启用快速启动故障转移不会触发故障转移。相反，它允许监视配置的观察者在条件允许故障转移时启动快速启动故障转移。本节介绍启用快速启动故障转移的步骤，并启动配置保护模式设置为最大可用性模式的观察器。

  1. 确保在主备用数据库和目标备用数据库上配置备用重做日志。在配置备用重做日志之前，必须停止日志应用服务。

    也可以看看：

    有关配置备用重做日志文件的说明，请参阅Oracle Data Guard概念和管理

  2. 如果主数据库和备用数据库直接接收重做而不是通过远程同步实例，则将主数据库和备用数据库配置为通过SYNC或FASTSYNC模式接收重做。如果任一数据库通过远程同步实例接收重做，则将远同步实例配置为通过SYNC或FASTSYNC模式接收重做，并将数据库配置为通过ASYNC模式接收重做。例如：
    DGMGRL>编辑数据库'North_Sales'设置属性'LogXptMode'='SYNC';属性“LogXptMode”已更新

    DGMGRL>编辑数据库'North_Sales'设置属性FastStartFailoverTarget ='South_Sales';属性“FastStartFailoverTarget”已更新

6.9.2禁用配置和数据库

当您禁用代理配置或其任何数据库时，您将禁用代理对它们的管理，并且实际上无法使用DGMGRL来管理和监视它们。

但是，禁用代理管理代理配置不会影响基础Oracle Data Guard配置或数据库的实际操作。例如，Oracle Data Guard配置中的重做传输服务和日志应用服务继续保持不变，但您无法再管理它们。

6.9.2.1禁用配置

必须使用DISABLE CONFIGURATION命令禁用对包括主数据库在内的整个代理配置的管理。

例如：

DGMGRL>禁用配置;

禁用主数据库的代理管理的唯一方法是使用DISABLE CONFIGURATION命令; DISABLE DATABASE命令仅禁用备用数据库的管理。同样， DISABLE FAR_SYNC命令仅禁用远程同步实例的管理。

注意：

如果在连接到备用数据库或远程同步实例时禁用配置管理，则必须连接到主数据库（即控制文件角色为主数据库的数据库）才能重新启用配置。

禁用代理管理配置成员不会从代理配置文件中删除该成员。您可以通过输入相应的ENABLE CONFIGURATION或ENABLE DATABASE命令来重新启用使用DGMGRL（或Cloud Control）来管理成员的能力。

6.9.2.2禁用备用数据库

当您暂时不希望代理管理和监视备用数据库时，可以使用DISABLE DATABASE命令。

您可以显式禁用备用数据库的代理管理，以防止在启用其余配置时启用备用数据库。以下示例说明如何禁用South_Sales备用数据库。 

DGMGRL> DISABLE DATABASE'South_Sales';禁用。

注意：

如果启用了快速启动故障切换并且要禁用的数据库是目标备用数据库，则无法从配置中禁用备用数据库。

注意：

如果在连接到备用数据库时禁用备用数据库的管理，则必须连接到主数据库或其他已启用的备用数据库，以重新启用备用数据库的代理管理。

警告：

如果在代理配置中禁用备用数据库的代理管理，则在丢失主数据库时，代理不能将该备用数据库用作故障转移目标。

在最大保护模式或最大可用性模式下运行时，代理会阻止您禁用支持保护模式的最后一个备用数据库。

警告：

6.14.1监控配置任务1：检查配置状态

代理配置的状态是代理配置中所有数据库和实例的聚合状态。

您可以先检查配置状态，以确定是否需要采取进一步的操作。如果配置状态为SUCCESS，则代理配置中的所有内容都将正常运行。但是，如果您看到WARNING或ERROR状态，则配置中出现问题。

例如，在以下显示中，您可以看到主数据库有多个警告：

DGMGRL> SHOW CONFIGURATION;配置 -  DRSolution保护模式：MaxPerformance成员：North_Sales  - 主数据库警告：ORA-16809：检测到数据库的多个警告South_Sales  - 物理备用数据库快速启动故障转移：禁用配置状态：警告

6.14.2监控配置任务2：检查数据库状态

要识别主数据库上的警告，请使用SHOW DATABASE命令显示其状态。

例如：

DGMGRL> SHOW DATABASE'North_Sales';数据库 -  North_Sales角色：PRIMARY预期状态：TRANSPORT-ON实例：north_sales1警告：ORA-16737：备用“red_Sort”的重做传输服务有错误警告：ORA-16714：属性LogArchiveTrace的值与之不一致数据库设置警告：ORA-16715：重做传输相关属性备用数据库“South_Sales”的ReopenSecs不一致数据库状态：警告

6.14.3监控配置任务3：检查LogXptStatus Monitorable属性

步骤2中的SHOW DATABASE输出显示错误警告ORA-16737。

要确定确切的传输错误，请使用LogXptStatus属性：

DGMGRL> SHOW DATABASE'North_Sales''LogXptStatus';日志传输状态PRIMARY_INSTANCE_NAME STANDBY_DATABASE_NAME状态north_sales1 South_Sales ORA-12541：TNS：没有侦听器

输出显示物理备用数据库的侦听器未运行。要修复此错误，请启动物理备用数据库South_Sales的侦听South_Sales 。

6.14.4监视配置任务4：检查InconsistentLogXptProps Monitorable属性

要识别重做传输数据库属性ReopenSecs的不一致值，可以使用InconsistentLogXptProps ReopenSecs属性。

例如，对于步骤2中SHOW DATABASE显示中显示的警告是ORA-16715，这很有用。 

DGMGRL> SHOW DATABASE'North_Sales''InconsistentLogXptProps';不一致的日志传输属性INSTANCE_NAME STANDBY_NAME PROPERTY_NAME MEMORY_VALUE BROKER_VALUE south_sales1 South_Sales ReopenSecs 600 300

当前数据库内存值（600）与Oracle Data Guard代理的属性值（300）不同。如果您认为代理的属性值是正确的，则可以通过使用相同的值重新编辑备用数据库的属性来修复不一致性，如以下示例所示：

DGMGRL>编辑数据库'South_Sales'Set PROPERTY'ReopenSecs'= 300;属性“ReopenSecs”已更新

您还可以重新启用备用数据库或将主数据库的状态重置为TRANSPORT-ON以修复此不一致。

以下主题讨论Oracle Messaging Gateway（MGW）如何将消息格式从一个消息传递系统转换为另一个消息传递系统。在Oracle Database Advanced Queuing和另一个系统之间移动消息时，通常需要进行转换，因为不同的消息传递系统具有不同的消息格式。Java消息服务 （JMS）消息是一种特殊情况。JMS消息只能传播到JMS目标，从而使转换成为一个简单的过程。

20.1转换Oracle Messaging Gateway非JMS消息

MGW在传播期间将源消息传递系统的本机消息格式转换为目标消息传递系统的本机消息格式。MGW使用规范类型和以Oracle数据库高级队列为中心的模型进行转换。

20.1.1非JMS消息转换过程概述

当MGW传播消息时，消息将从源队列的本机格式转换为目标队列的本机格式。

本机消息通常包含消息头和消息正文。标头包含该消息传递系统中所有消息所具有的固定标头字段，例如Oracle Database Advanced Queuing中的消息属性和WebSphere MQ中的固定标头。正文包含消息内容，例如Oracle数据库高级队列有效内容，WebSphere MQ消息正文或整个TIB / Rendezvous消息。MGW转换消息头和消息体组件。

图20-1 显示了如何在两个阶段中转换非JMS消息。消息首先从源队列的本机格式转换为MGW内部消息格式，然后将其从内部消息格式转换为目标队列的本机格式。

20.1.2 Oracle Messaging Gateway规范类型

MGW定义了规范类型，以支持Oracle数据库高级队列和非Oracle消息传递系统之间的消息转换。规范类型是Oracle数据库中PL / SQL Oracle类型形式的消息类型表示。规范类型是RAW ， SYS.MGW_BASIC_MSG_T和SYS.MGW_TIBRV_MSG_T 。

WebSphere MQ传播支持规范类型SYS.MGW_BASIC_MSG_T和RAW 。 TIB / Rendezvous传播支持规范类型SYS.MGW_TIBRV_MSG_T和RAW 。

20.2.2 WebSphere MQ Outbound Propagation

如果没有为出站传播提供消息转换，则Oracle数据库高级队列源队列有效内容类型必须为SYS.MGW_BASIC_MSG_T或RAW 。如果指定了消息转换，则转换的目标ADT必须为SYS.MGW_BASIC_MSG_T ，但源ADT可以是Oracle Database Advanced Queuing支持的任何ADT。

如果Oracle数据库高级队列队列有效负载是RAW，则生成的WebSphere MQ消息将消息体设置为RAW字节的值，并且默认情况下， format字段设置为值“ MGW_Byte ”。

如果Oracle数据库高级队列队列有效负载或转换目标ADT为SYS.MGW_BASIC_MSG_T ，则消息将映射到WebSphere MQ本机消息，如下所示：

  • WebSphere MQ固定标头字段基于规范消息的内部Oracle数据库高级排队消息属性和SYS.MGW_BASIC_MSG_T.header属性，如“ WebSphere MQ消息标头映射 ”中所述 。

  • 如果规范消息具有TEXT主体，则WebSphere MQ格式头字段设置为MQFMT_STRING除非MGW_MQ_format属性MGW_MQ_format覆盖。邮件正文被视为文本。

  • 如果规范消息具有RAW主体，则WebSphere MQ格式头字段设置为“ MGW_Byte ”，除非MGW_MQ_format属性MGW_MQ_format覆盖。消息正文被视为原始字节。

  • 如果规范消息同时具有TEXT和RAW正文，则消息转换失败。

  • 如果规范消息既没有TEXT也没有RAW主体，则没有设置消息体，并且WebSphere MQ格式头字段是MQFMT_NONE 。

  • 如果规范消息具有设置了小值和大值的TEXT主体（ SYS.MGW_BASIC_MSG_T.TEXT_BODY.small_value和SYS.MGW_BASIC_MSG_T.TEXT_BODY.large_value不为空），则消息转换失败。

  • 如果规范消息具有设置了小值和大值的RAW正文（ SYS.MGW_BASIC_MSG_T.RAW_BODY.small_value和SYS.MGW_BASIC_MSG_T.RAW_BODY.large_value不为空），则消息转换失败。

如果使用值TRUE指定作业选项PreserveMessageID ，那么WebSphere消息头的correlationId字段将设置为AQ源消息标识符。correlationId值将是"AQMSGID:"+ AQ_msgid形式的24字节值，其中AQ_msgid表示16字节的数据库AQ消息标识符。

  • 生成的消息包含名为MGW_RAW_MSG的字段，其值为TibrvMsg.OPAQUE 。字段ID设置为0。

20.3.3 TIB / Rendezvous入站传播

如果没有为入站传播提供传播转换，则Oracle数据库高级队列目标队列有效内容类型必须为RAW或SYS.MGW_TIBRV_MSG_T 。如果指定了传播转换，则转换的目标ADT可以是Oracle数据库高级队列支持的任何ADT，但转换的源ADT必须是SYS.MGW_TIBRV_MSG_T 。

如果Oracle数据库高级队列队列有效负载或转换源ADT为SYS.MGW_TIBRV_MSG_T ，则：

  • 源TIB / Rendezvous消息中的每个字段都将转换为SYS.MGW_TIBRV_MSG_T类型的结果消息的SYS.MGW_TIBRV_MSG_T 。

  • 该MGW剂提取从源TIB /楼台消息发送主题名称和设置send_subject在属性SYS.MGW_TIBRV_MSG_T 。发送主题名称通常与注册的传播源队列的主题名称相同，但在使用通配符时可能会有所不同。

  • 该MGW剂从源TIB /楼台消息中提取所述回复主题名称，如果它存在，并设置reply_subject在属性SYS.MGW_TIBRV_MSG_T 。

  • 如果RAW数据大小大于32KB，则会发生消息转换失败。这是由于对绑定变量允许的数据大小的限制。此外，实际限制是32512而不是32767。

安全散列算法（SHA）

一种算法，通过从给定数据生成160位加密消息摘要值来确保数据完整性。如果修改数据中的单个位，则安全散列算法校验和数据更改。以使得安全散列算法生成与原始数据的结果相同的结果的方式伪造给定数据集被认为是计算上不可行的。

一种算法，它获取长度小于264位的消息并生成160位消息摘要。具有较大消息摘要的算法使其更安全地抵御暴力冲突和反转攻击。
