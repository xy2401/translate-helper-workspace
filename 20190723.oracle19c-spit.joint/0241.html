3.2.1关于使用连接池

对于许多中间层应用程序，应为大量线程启用与数据库的连接。因为每个线程存在的时间相对较短，所以为每个线程打开与数据库的连接将是低效的连接使用，并且会导致性能低下。

通过使用连接池功能，您的应用程序可以创建一小组可用于大量线程的连接，从而使您可以非常有效地使用数据库资源。

3.2.2使用无状态连接池

无状态连接池专门设计用于需要短连接时间且不处理状态注意事项的应用程序。无状态连接池的主要优点是提高了性能，因为消除了耗时的连接和身份验证协议。

无状态连接池创建并维护一组可由多个线程使用的无状态，经过身份验证的数据库连接。一旦线程使用其连接完成，它应该将连接释放回池。如果没有可用的连接，则生成新的连接。因此，池中的连接数可以动态增加。

池中的某些连接可能会使用特定属性进行标记。用户可以请求默认连接，设置某些属性，例如全球化支持设置，然后标记它并将其返回到池中。当需要具有相同属性的连接时，可以对具有相同标记的连接进行请求，并且可以重用具有相同标记的池中的多个连接之一。可以更改或重置连接上的标记。

还可以通过Stateless Connection Pooling接口创建和维护代理连接。

无状态连接池通过多路复用连接来提高中间层应用程序的可伸缩性。但是，来自StatelessConnectionPool连接不应用于长事务，因为长时间保持连接会导致并发性降低。

注意：

  • OCCI不检查连接标签对的正确性。您有责任确保具有不同客户端属性的连接不具有相同的标记。

  • 在将连接释放回池之前，您的应用程序应提交或回滚任何打开的事务。如果不这样做，Oracle会在释放连接时自动提交任何打开的事务。

有两种类型的无状态连接池：

  • 同构池是使用在创建池时提供的用户名和密码对所有连接进行身份验证的池。因此，所有连接都具有相同的身份验证上下文此类池中不允许代理连接。

  • 异构池中的不同用户名可以对不同的连接进行身份验证。代理连接也可以存在于异构池中，前提是在服务器上授予创建它们的必要权限。此外，异构池支持外部身份验证。

3.2.3关于数据库驻留连接池

企业级应用程序通常必须处理大量并发用户会话，这些会话实现为与数据库的持久连接。创建和管理这些连接的内存开销对数据库的性能有重大影响。

数据库驻留连接池通过提供用于处理大量应用程序连接的专用服务器池来解决持久连接过多的问题，从而使数据库能够扩展到数万个并发连接。它显着减少了数据库层的内存占用，并提高了数据库和中间层的可伸缩性。数据库驻留连接池专为具有多进程应用程序服务器和多个中间层的体系结构而设计，这些中间层无法容纳中间层中的连接池。

数据库驻留连接池结构紧密遵循用于连接到Oracle数据库实例的默认专用模型;但是，它消除了为每个连接分配特定服务器的开销。在服务器层上，大多数连接在任何给定时间都处于非活动状态，并且每个连接都消耗内存。因此，支持高连接量的数据库系统面临快速耗尽所有可用内存的风险。数据库驻留连接池允许连接使用专用服务器，该服务器结合了Oracle服务器进程和用户会话。一旦连接变为非活动状态，它就会将其资源返回到池中，供类似的连接使用。

在能够进行全面连接池的多线程中间层中，未使用的连接问题有些不同。随着中间层数的增加，每个中间层私下拥有与数据库的多个连接;这些连接不能与其他中间层共享。在数据库上定位连接池可以实现跨类似客户端的连接共享。

数据库驻留连接池支持基于密码的身份验证，语句缓存，标记和快速应用程序通知 。您还可以将客户端无状态连接池与数据库驻留连接池一起使用。

请注意，从数据库驻留连接池保持连接的客户端持久连接到后台连接代理进程。Connection Broker实现池功能并将入站客户端连接多路复用到专用服务器进程池。不使用连接池的客户端使用专用服务器进程。

3.2.3.1管理数据库驻留连接池

要实现数据库驻留连接池，必须首先由具有SYSDBA权限的用户在系统上启用它。有关启动和维护数据库驻留连接池所需的步骤，请参阅示例3-12 。

3.4.4关于流式读取和写入

OCCI支持通过将数据分成一系列小块来插入和检索非常大的列的流接口。这种方法最大限度地减少了客户端内存需求。此流接口可以与参数化语句（如SELECT和各种DML命令）一起使用，也可以与PL / SQL块中的可调用语句一起使用。流支持的数据类型是BLOB ， CLOB ， LONG ， LONG RAW ， RAW和VARCHAR2 。

  • 可写流对应于SELECT / DML语句中的绑定变量或可调用语句中的IN参数。

  • 对于写入，它将数据传递给绑定变量或IN或IN/OUT参数

  • 对于读取，它从OUT或IN/OUT参数中获取数据

ResultSet类的getStream（）方法返回可用于读取数据的流对象。

这些类的status()方法确定流操作的状态。

3.5.3关于通过设置预取计数来优化性能

虽然ResultSet方法一次检索一行数据，但实际从服务器获取数据不需要为查询的每一行进行网络往返。为了最大限度地提高性能，您可以在每次往返服务器的过程中设置要预取的行数。

您可以通过设置要通过setPrefetchRowCount( )方法预取的行数，或通过设置用于通过setPrefetchMemorySize()方法预取的内存大小来实现此目的。

如果设置这两个属性，则会预取指定的行数，除非首先达到指定的内存限制。如果首先达到指定的内存限制，则预取将返回与调用setPrefetchMemorySize()方法定义的内存空间中的行数相同的行数。

默认情况下，打开预取并且数据库始终提取额外的行。要关闭预取，请将预取行数和内存大小都设置为0 。

如果LONG ，LOB或Opaque Type列（例如XMLType ）是查询的一部分，则预取不起作用。

此状态表示应用程序需要在执行完成之前在OUT或IN/OUT参数中读取一些流数据。读取流后，调用status方法以查明是否应读取更多流数据，或者执行是否已完成。

3.10关于处理异常

如果不成功，每个OCCI方法都可以生成异常。此异常的类型为SQLException 。OCCI使用C ++标准模板库（STL），因此OCCI方法也可以抛出STL可以抛出的任何异常。

STL异常派生自标准异常类。exception::what()方法返回指向错误文本的指针。在catch块期间，错误文本保证有效

SQLException类包含Oracle特定的错误号和消息。它派生自标准异常类，因此它也可以通过使用exception::what()方法获取错误文本。

此外， SQLException类有两个可用于获取错误信息的方法。getErrorCode()方法返回Oracle错误号。可以通过getMessage()方法获取exception::what()返回的相同错误文本。getMessage()方法返回一个STL字符串，以便可以像任何其他STL字符串一样进行复制。

根据您的错误处理策略，您可以选择以不同于标准异常的方式处理OCCI异常，或者您可以选择不区分这两者。

如果您认为区分OCCI异常和标准异常并不重要，则catch块可能类似于以下内容：

catch（exception＆excp）{cerr << excp.what（）<< endl; }

如果您决定以不同于标准异常的方式处理OCCI异常，则catch块可能如下所示：

catch（SQLException＆sqlExcp）{cerr << sqlExcp.getErrorCode <<“：”<< sqlExcp.getErrorMessage（）<< endl; } catch（exception＆excp）{cerr << excp.what（）<< endl; }

在前面的catch块中，第一个块捕获SQL异常，第二个块捕获非SQL异常。如果要反转这两个块的顺序，则永远不会捕获SQL异常。由于SQLException是从标准异常派生的，因此标准异常catch块也会处理SQL异常。

  • 管理存储在磁盘或磁带上的数据库备份

  • 管理RMAN存储库中这些备份的记录

RMAN维护的一个重要部分是删除不再需要的备份。如果配置快速恢复区域，则数据库会自动自动删除此区域中不需要的文件;即便如此，您可能希望从磁带中删除备份和副本。您甚至可能需要删除整个数据库。您可以使用RMAN命令执行这些任务。

快速恢复区可能需要偶尔进行维护。例如，快速恢复区域可能已满，在这种情况下，您可以为其添加空间。或者，您可能希望更改恢复区域的位置。

RMAN存储库可能无法反映磁盘或磁带上文件的真实状态。例如，用户可以使用操作系统实用程序从磁盘中删除备份。在这种情况下，RMAN存储库显示文件存在时不存在。在类似情况下，包含RMAN备份的磁带可能已损坏。您可以使用RMAN维护命令来使用准确的信息更新存储库。

12.2.1关于控制文件记录

控制文件包含两种类型的记录：循环重用记录和非圆形重用记录。

循环重用记录包含非关键信息，如果需要，可以覆盖这些信息。这些记录包含由数据库不断生成的信息。当所有可用记录槽已满时，数据库会扩展控制文件以为新记录腾出空间或覆盖最旧记录。CONTROL_FILE_RECORD_KEEP_TIME初始化参数指定记录在可以重用之前的最小年龄（以天为单位）。

非圆形重用记录包含不经常更改且无法覆盖的关键信息。非圆重用记录中的一些信息示例包括数据文件，联机重做日志文件和重做线程。

在备份目标数据库时，数据库会将这些备份记录在控制文件中。为防止控制文件因添加新记录而变得过大，如果记录超过您指定的阈值，则可以重复使用这些记录。CONTROL_FILE_RECORD_KEEP_TIME初始化参数确定记录在被覆盖之前的最小年龄（以天为单位）：

CONTROL_FILE_RECORD_KEEP_TIME = 整数

例如，如果参数值为14 ，那么任何14天或更早的记录都是重用的候选者。覆盖记录中的信息丢失。首先使用可用于重用的最旧记录。

当数据库必须将新的RMAN存储库记录添加到控制文件中，但没有记录超过阈值时，数据库会尝试扩展控制文件的大小。如果底层操作系统阻止了控制文件的扩展（例如，由于磁盘已满），则数据库将覆盖控制文件中最旧的记录。

数据库将覆盖记录在自动诊断存储库（ADR）中的警报日志中。对于它覆盖的每个记录，数据库在警报日志中记录一个类似于以下内容的条目：

kccwnc：写入以下控制文件记录：RECID＃72 Recno 72记录时间戳07/28/06 22:15:21线程= 1 Seq＃= 3460备份集密钥：stamp = 372031415，count = 17低scn：0x0000.3af33f36 07/27/06 21:00:08下一个scn：0x0000.3af3871b 07/27/06 23:23:54 Resetlogs scn和time scn：0x0000.00000001

12.2.2防止丢失控制文件记录

防止因覆盖控制文件记录而丢失RMAN元数据的最佳方法是使用恢复目录。

如果您无法使用恢复目录，则可以采取以下措施：

  • 将CONTROL_FILE_RECORD_KEEP_TIME值设置为略长于您必须保留的最旧文件。例如，如果您每周备份一次整个数据库，则必须将每个备份保留至少7天。将CONTROL_FILE_RECORD_KEEP_TIME设置为诸如10或14类的值。CONTROL_FILE_RECORD_KEEP_TIME的默认值为7天。

    警告：

    无论您是否使用恢复目录，在CONTROL_FILE_RECORD_KEEP_TIME设置为0时都不要使用RMAN。如果这样做，那么您可能会丢失备份记录。

  • 将控制文件存储在文件系统中而不是存储在原始设备上，以便它可以扩展。

  • 监视警报日志以确保Oracle数据库不会覆盖控制文件记录。警报日志位于自动诊断存储库（ADR）中 。

如果使用快速恢复区域，请遵循以下准则以避免控制文件无法保存满足备份保留策略所需的所有快速恢复区域文件的记录的情况：

  • 如果控制文件的块大小不是最大值，则使用更大的块大小，最好是32千字节。

    要实现此目标，必须将SYSTEM表空间块大小设置为大于或等于控制文件块大小，并在更改DB_BLOCK_SIZE后重新创建控制文件。快速恢复区域中的文件已重新编目，但磁带上文件的记录将丢失。

  • 通过将快速恢复区域中的文件备份到磁带等第三级存储，使其可以在快速恢复区域中进行删除。

    例如，您可以使用BACKUP RECOVERY AREA将快速恢复区中的文件专门备份到媒体管理器。

  • 如果备份保留策略使备份和归档日志长于业务要求，则可以通过将保留策略更改为更短的恢复窗口或更低的冗余度，使快速恢复区域中的更多文件符合删除条件。

12.2.3保护控制文件

如果您没有使用恢复目录来存储RMAN元数据，那么保护每个目标数据库控制文件是非常重要的。

要保护控制文件：

您可以使用以下策略来保护控制文件。

  1. 通过多路复用或操作系统镜像创建控制文件的冗余副本。

    通过这种方式，数据库可以在丢失控制文件子集的情况下继续运行，而无需从备份中恢复控制文件。Oracle建议您在不同的磁盘上至少使用两个多路复用或镜像控制文件。

  2. 配置控制文件自动备份功能。

    在这种情况下，RMAN会在您运行某些RMAN命令时自动备份控制文件。如果控制文件自动备份可用，RMAN可以还原服务器参数和备份控制文件，并装入数据库。装入控制文件后，您可以还原数据库的其余部分。

  3. 保留数据库DBID的记录。

    如果丢失了控制文件，则可以使用DBID恢复数据库。

12.3.3管理快速恢复区域中闪回日志的空间

除了设置闪回保留目标或使用保证还原点之外，您无法直接管理快速恢复区域中的闪回日志。

不过，您可以整体管理快速恢复区域空间，以最大限度地利用可用于保留闪回日志的空间。通过这种方式，您可以提高实现闪回目标的可能性。

要为闪回日志腾出空间，请使用BACKUP RECOVERY AREA ， BACKUP BACKUPSET等命令将快速恢复区的其他内容BACKUP到磁带。Oracle数据库会自动从快速恢复区域中删除过时的文件。如果将备份卸载到磁带仍然无法创建足够的空间来满足备份保留策略和闪回保留目标，则在快速恢复区域中分配更多空间。

从Oracle Database Release 19c开始，简化了快速恢复区域中的空间管理。Oracle数据库监视快速恢复区域中的闪回日志，并自动删除超出保留期的闪回日志。当保留目标减少时，将立即删除超出保留期的闪回日志。

必须将COMPATIBLE初始化参数设置为19.0.0或更高，才能自动删除闪回日志。

在突然的工作负载峰值导致创建大量闪回日志的情况下，在删除超出保留期的闪回日志之前，将监视工作负荷几天。如果在不久之后发生另一个峰值工作负载，则可以避免重新创建闪回日志的开销。

注意：

您无法备份闪回日志。因此，在将快速恢复区内容备份到磁带时， BACKUP RECOVERY AREA操作不包括闪回日志。

12.3.4响应完全快速恢复区域

如果RMAN保留策略要求保留一组大于快速恢复区磁盘配额的备份，或者保留策略设置为NONE ，则快速恢复区域可以完全填充，没有可回收空间。

当可回收空间小于15％时，数据库发出警告警报，当可回收空间小于3％时，数据库发出严重警报。要向DBA警告此情况，会在警报日志和DBA_OUTSTANDING_ALERTS表（由Enterprise Manager使用）中添加一个条目。然而，数据库继续占用快速恢复区域的空间，直到没有可回收的空间。

当恢复区域完全填满时，显示的错误如下所示，其中nnnnn是所需的字节数， mmmmm是磁盘配额：

ORA-19809：恢复文件超出限制ORA-19804：无法从mmmmm限制中回收nnnnn字节磁盘空间

当没有文件可以删除时，您有多种选择来解决完整的快速恢复区域：

  • 增加可用磁盘空间并增加DB_RECOVERY_FILE_DEST_SIZE以反映额外空间。

  • 将备份从快速恢复区域移动到第三级存储（如磁带）。

    将所有恢复区文件备份到磁带的一种便捷方法是BACKUP RECOVERY AREA命令。将备份从恢复区域传输到磁带后，可以从快速恢复区域中删除文件。闪回日志无法在恢复区域外备份，也无法由BACKUP RECOVERY AREA BACKUP 。

  • 对于使用操作系统实用程序DELETE的任何文件，请运行DELETE 。

    如果使用主机操作系统命令删除文件，则数据库不会知道生成的可用空间。您可以运行RMAN CROSSCHECK命令让RMAN重新检查快速恢复区域的内容并识别过期文件，然后使用DELETE EXPIRED命令从RMAN存储库中删除每个过期的备份 。

  • 确保您的保证还原点是必要的。如果没有，删除它们。

12.3.5将快速恢复区域更改为新位置

使用ALTER SYSTEM命令更改快速恢复区域的位置。

如果必须将数据库的快速恢复区域移动到新位置，请执行以下步骤：

  1. 在目标数据库上启动SQL * Plus并更改DB_RECOVERY_FILE_DEST初始化参数。例如，输入以下命令将目标设置为ASM磁盘组disk1 ：
    ALTER SYSTEM SET DB_RECOVERY_FILE_DEST ='+ disk1'SCOPE = BOTH SID ='*';

    更改此参数后，将在新位置创建所有新的快速恢复区域文件。

  2. 在旧的闪回恢复位置中保留或移动永久文件，闪回日志和瞬态文件。

    如果将现有文件保留在闪回恢复中，则数据库会删除旧快速恢复区域中的临时文件，因为它们有资格删除。

    如果必须将旧文件移动到新的快速恢复区域，请参阅“ Oracle自动存储管理管理员指南” 。使用RMAN将数据库文件移入和移出ASM磁盘组的过程在将文件移入和移出快速恢复区域时起作用。

12.3.6禁用快速恢复区域

ALTER SYSTEM命令可用于禁用快速恢复区域。

在禁用快速恢复区域之前，必须首先删除所有保证的还原点，然后关闭闪回数据库。

然后，您可以将DB_RECOVERY_FILE_DEST初始化参数设置为空字符串以禁用快速恢复区域。例如，使用以下SQL语句更改正在运行的数据库上的参数：

ALTER SYSTEM SET DB_RECOVERY_FILE_DEST =''SCOPE = BOTH SID ='*';

数据库不再为存储在旧DB_RECOVERY_FILE_DEST位置的文件提供快速恢复区域的空间管理功能。但是，RMAN存储库仍然知道这些文件，并且这些文件可用于备份和还原活动。

12.3.7响应文件创建过程中的实例崩溃

通常，快速恢复区域是自我维护的。但是，当在快速恢复区域中创建文件期间实例崩溃时，数据库可能会将文件保留在快速恢复区域中。

发生这种情况时，警报日志包含以下错误，其中location是快速恢复区域的位置：

ORA-19816：警告：文件可能存在于数据库未知的位置 。

在这种情况下，使用RMAN命令CATALOG RECOVERY AREA重新编目任何此类文件。如果相关文件的文件头已损坏，则使用操作系统实用程序手动删除该文件。

12.4.3将备份记录添加到RMAN存储库

您可以使用CATALOG命令使RMAN知道存储库中未记录的存档日志的存在，或者通过RMAN以外的方式创建的数据库文件的副本。

12.4.3.1关于编目操作

目标数据库控制文件保留目标数据库和所有RMAN备份生成的所有归档重做日志的记录。CATALOG命令的目的是在没有要管理的RMAN文件记录时将元数据添加到存储库。

在以下情况下运行RMAN CATALOG命令：

  • 您使用操作系统实用程序来复制数据文件，存档日志或备份片。在这种情况下，存储库没有它们的记录。

  • 使用备份控制文件执行恢复，并在恢复期间更改存档目标或格式。在这种情况下，存储库不包含有关恢复所需的存档日志的信息，您必须对这些日志进行编目。

  • 您希望将数据文件副本编目为0级备份，从而使您可以稍后使用数据文件副本作为增量备份策略的基础来执行增量备份。

  • 您希望编制在迁移到更高版本之前创建的Oracle7数据库文件的用户管理副本，或者在开始使用RMAN之前创建的Oracle8及更高版本数据库文件的目录。如果数据库在迁移后但在您有机会备份迁移的数据库之前失败，则可以使用这些数据文件副本来恢复数据库。

每当您创建用户管理的副本时（例如，使用UNIX cp命令复制数据文件），请务必对其进行编目。制作用户管理的副本时，您可以使用ALTER TABLESPACE...BEGIN/END BACKUP语句使数据文件副本脱离在线表空间。虽然RMAN不会创建此类数据文件副本，但您可以使用CATALOG命令将它们添加到恢复目录中，以便RMAN知道它们。

要对用户管理的副本进行编目，必须为：

  • 可在磁盘上访问

  • 单个文件的完整映像副本

  • 数据文件副本，控制文件副本，归档重做日志副本或备份片副本

例如，如果将数据文件存储在镜像磁盘驱动器上，则可以通过断开镜像来创建用户管理的副本。在此方案中，使用CATALOG命令在断开镜像后通知RMAN存在用户管理的副本。在改造镜子之前，运行CHANGE...UNCATALOG命令通知RMAN文件副本不再存在。

12.5删除RMAN备份和归档重做日志

您可以使用RMAN DELETE命令删除存档的重做日志和RMAN备份。

对于磁盘上的备份，删除备份会从磁盘中删除备份文件。对于SBT设备上的备份，RMAN DELETE命令指示介质管理器删除磁带上的备份片或代理副本。在任何一种情况下，RMAN都会更新RMAN存储库以反映删除。

注意：

在CDB中，只有在以具有SYSDBA或SYSBACKUP权限的用户身份连接到root时，才能删除存档日志。连接到PDB时，无法删除存档的重做日志。

12.5.1删除RMAN备份概述

每个RMAN备份在RMAN存储库中生成相应的记录。该记录存储在控制文件中。如果使用了恢复目录，则在从控制文件重新同步恢复目录后，也可以在恢复目录中找到该记录。

例如，如果生成完整数据库备份集，则可以在V$BACKUP_SET查看此备份集的记录。如果使用恢复目录，则还可以访问RC_BACKUP_SET目录视图中的记录。

V$控制文件视图和恢复目录视图在存储信息的方式上有所不同，这会影响RMAN处理存储库记录的方式。恢复目录RMAN存储库存储在实际数据库表中，而存储库的控制文件版本存储在控制文件的内部结构中。

使用RMAN命令删除备份或归档重做日志文件时，RMAN执行以下操作：

  • 从操作系统中删除物理文件（如果文件仍然存在）

  • 将控制文件中的文件记录更新为DELETED状态

  • 从恢复目录表中删除文件记录（如果RMAN连接到恢复目录）

由于存储控制文件数据的方式，RMAN无法从控制文件中删除记录，只能将其更新为DELETED状态。但是，由于恢复目录表是普通数据库表，因此RMAN会从中删除行，其方式与从任何表中删除行的方式相同。

24.1什么是对象类型转换器？

对象类型转换器（OTT）有助于开发在Oracle数据库中使用用户定义类型的C语言应用程序。

使用SQL CREATE TYPE语句，您可以创建对象类型。这些类型的定义存储在数据库中，可用于创建数据库表。填充这些表后，OCI程序员可以访问存储在表中的对象。

访问对象数据的应用程序必须能够以宿主语言格式表示数据。这是通过将对象类型表示为C结构来实现的。虽然程序员可以手动编写结构声明来表示数据库对象类型，但如果涉及许多类型，这可能非常耗时且容易出错。OTT通过自动生成适当的结构声明来消除对这种手动编码的需要。在OCI中，应用程序还必须调用OTT生成的初始化函数。

除了创建表示存储数据类型的结构之外，OTT还生成并行指示符结构，指示对象类型或其字段是否为NULL 。

对象类型转换器（OTT）将对象类型和命名集合类型的数据库定义转换为可包含在OCI应用程序中的C结构声明。

您必须显式调用OTT才能将数据库类型转换为C表示形式。

在大多数操作系统上，在命令行上调用OTT。它将intype文件作为输入，它生成一个outtype文件和一个或多个C 头文件以及一个可选的实现文件 。以下是调用OTT的命令示例：

ott userid = scott intype = demoin.typ outtype = demoout.typ code = c hfile = demo.h \ initfile = demov.c

此命令使OTT使用用户名scott连接到数据库。提示用户输入密码。

实现文件（ demov.c ）包含初始化类型版本表的函数，其中包含有关已翻译的用户定义类型的信息。

本章后面的部分将更详细地描述每个参数。

示例demoin.typ文件：

CASE = LOWER TYPE SCOTT.EMPTYPE AS emptype VERSION =“$ 8.0”HFILE = demo.h

OTT完成转换后，头文件包含intype文件中指定的每种类型的C结构表示，以及与每种类型对应的NULL指示符结构。例如，假设intype文件中列出的雇员类型定义如例24-1所示。

  • 类型名称与同一范围内的另一个标识符冲突。例如，如果程序使用来自不同模式的两个具有相同名称的类型，则会发生这种情况。

  • 程序员喜欢不同的名字。

24.4.3空指示符结构

24.6.3初始化函数的任务

C初始化函数提供有关OTT处理的类型的版本信息。它向类型版本表添加每个OTT处理的对象数据类型的名称和版本标识符。

类型规范的结构如下，其中[]表示内部的可选输入：

TYPE type_name [AS type_identifier ] [VERSION [=] version_string ] [HFILE [=] hfile_name ] [TRANSLATE { member_name [AS identifier ]} ...]

type_name的语法是：

schema_name是拥有给定对象数据类型的模式的名称， type_name是该类型的名称。默认架构是运行OTT的用户的架构。默认数据库是本地数据库。

  • type_name是Oracle数据库对象数据类型的名称。

  • type_identifier是用于表示类型的C标识符。如果省略type_identifier ，则使用默认名称映射算法。

  • version_string是上次调用OTT生成代码时使用的类型的版本字符串。版本字符串由OTT生成并写入outtype文件，当稍后执行OTT时，该文件可用作intype文件。版本字符串不会影响OTT的操作，但最终用于选择应在正在运行的程序中使用的对象数据类型的版本。

  • hfile_name是头文件的名称，其中显示相应结构或类的声明。如果省略hfile_name ，则在生成声明时使用命令行HFILE参数指定的文件。

  • member_name是要转换为identifier的属性（数据成员）的名称。

  • identifier是用于表示用户程序中的属性的C标识符。可以通过这种方式为任意数量的属性指定标识符。默认名称映射算法用于未提及的属性。

24.7.6 SCHEMA_NAMES用法

SCHEMA_NAMES参数会影响OTT所连接的默认架构中的类型名称是否使用outtype文件中的架构名称进行限定。

此参数会影响OTT所连接的默认架构中的类型名称是否使用outtype文件中的架构名称进行限定。

除默认架构之外的架构中的类型名称始终使用outtype文件中的架构名称进行限定。

模式名称或其缺席确定在程序执行期间在哪个模式中找到类型。

有三种设置：

  • schema_names=ALWAYS （默认）

    outtype文件中的所有类型名称都使用模式名称限定。

  • schema_names = IF_NEEDED

    outtype文件中属于默认架构的类型名称不使用架构名称限定。与往常一样，属于其他模式的类型名称使用模式名称限定。

  • schema_names = FROM_INTYPE

    intype文件中提到的类型在outtype文件中使用模式名称限定，当且仅当它在intype文件中使用模式名称限定时。默认模式中未在intype文件中提及但由于类型依赖性而必须生成的类型仅在OTT遇到的依赖于它的第一个类型使用模式名称编写时才使用模式名称编写。但是，不在OTT所连接的默认架构中的类型始终使用显式架构名称编写。

OTT生成的outtype文件是Pro * C / C ++的输入参数。从Pro * C / C ++的角度来看，它是Pro * C / C ++ intype文件。此文件将数据库类型名称与C结构名称匹配。在运行时使用此信息以确保在结构中选择了正确的数据库类型。如果在outtype文件（Pro * C / C ++ intype文件）中出现带有模式名称的类型，则在程序执行期间会在命名模式中找到该类型。如果类型显示没有架构名称，则在程序连接的默认架构中找到类型，这可能与OTT使用的默认架构不同。

24.7.7默认名称映射

当OTT为对象类型或属性创建C标识符名称时，它会将名称从数据库字符集转换为合法的C标识符。

首先，名称从数据库字符集转换为OTT使用的字符集。接下来，如果在intype文件中提供了结果名称的翻译，则使用该翻译。否则，OTT将字符逐字符转换为编译器字符集，并应用CASE选项。以下更详细地描述该过程。

当OTT读取数据库实体的名称时，该名称将自动从数据库字符集转换为OTT使用的字符集。为了使OTT成功读取数据库实体的名称，必须在OTT字符集中找到名称的所有字符，尽管字符在两个字符集中可以具有不同的编码。

保证OTT使用的字符集包含所有必需字符的最简单方法是使其与数据库字符集相同。但请注意，OTT字符集必须是编译器字符集的超集。也就是说，如果编译器字符集是7位ASCII，则OTT字符集必须包含7位ASCII作为子集，如果编译器字符集是7位EBCDIC，则OTT字符集必须包含7位EBCDIC作为一个子集。用户通过设置NLS_LANG环境变量或某些其他特定于操作系统的机制来指定OTT使用的字符集。

一旦OTT读取了数据库实体的名称，它就会将名称从OTT使用的字符集转换为编译器的字符集。如果名称的翻译出现在intype文件中，OTT将使用该翻译。

否则，OTT会尝试使用以下步骤翻译名称：

  1. 如果OTT字符集是多字节字符集，则名称中具有单字节等效字符的所有多字节字符将转换为那些单字节等效字符。

  2. 名称从OTT字符集转换为编译器字符集。编译器字符集是单字节字符集，例如US7ASCII。

  3. 字母的大小是根据有效的CASE选项设置的，并且任何在C标识符中不合法或在编译器字符集中没有翻译的字符都将被下划线替换。如果至少有一个字符被下划线替换，OTT会发出警告消息。如果名称中的所有字符都被下划线替换，OTT会给出错误消息。

逐个字符的名称转换不会改变编译器字符集中出现的下划线，数字或单字节字母，因此不会更改合法的C标识符。

例如，名称翻译可以将重音单字节字符（例如带有变音符号的“o”或带有重音符号的“a”翻译为“o”或“a”），并且可以将多字节字母翻译为其单字节等效字符。如果名称包含缺少单字节等效项的多字节字符，则名称转换通常会失败。在这种情况下，用户必须在intype文件中指定名称翻译。

OTT不检测由两个或多个数据库标识符映射到同一C名称引起的命名冲突，也不检测数据库标识符映射到C关键字的命名问题。

24.7.8 OTT对文件名比较的限制

目前，OTT通过比较用户在命令行或intype文件中提供的文件名来确定两个文件是否相同。

但是当OTT需要知道两个文件名是否引用同一文件时，可能会出现一个潜在的问题。例如，如果OTT生成的文件foo.h要求写入类型声明foo1.h ，并写入到另一个类型声明/private/elias/foo1.h ，OTT应该产生一个#include指令，如果这两个文件是相同，如果文件不同，则有两个#includes指令。但实际上，它会得出结论，这两个文件是不同的，并会生成两个#includes指令，如下所示：

#ifndef FOO1_ORACLE #include“foo1.h”#endif #ifndef FOO1_ORACLE #include“/private/elias/foo1.h”#endif

如果foo1.h和/private/elias/foo1.h是不同的文件，则只包含第一个文件。如果foo1.h和/private/elias/foo1.h是同一个文件，则写入冗余的#include指令。

因此，如果在命令行或intype文件中多次提到文件，则每次提及该文件时应使用完全相同的文件名。

24.7.9 Microsoft Windows上的OTT命令

当前版本中Microsoft Windows上的OTT可执行文件是ott.bat ，而不是早期版本中的ott.exe 。

这可能会破坏Windows批处理脚本，因为脚本在执行ott后立即退出。要解决此问题，应在Windows批处理脚本中按如下方式调用OTT：

14.11 DROP_PHRASE

指定同义词库名称。

狼BT动物
