138 DBMS_RESULT_CACHE

DBMS_RESULT_CACHE包提供了一个接口，允许DBA管理SQL结果缓存和PL / SQL函数结果缓存使用的共享池的那部分。

这两个缓存都使用相同的基础结构。因此，例如， DBMS_RESULT_CACHE 。 BYPASS确定是绕过两个缓存还是使用两个缓存，以及DBMS_RESULT_CACHE 。 FLUSH刷新SQL查询的所有缓存结果和PL / SQL函数的所有缓存结果。

138.3.5 BYPASS程序

此过程为结果缓存设置旁路模式。

它设置以下旁路模式之一：

  • 打开旁路模式时，意味着不再使用缓存结果，并且不会在缓存中保存新结果。

  • 当旁路模式关闭时，缓存恢复正常操作。

句法

DBMS_RESULT_CACHE.BYPASS（bypass_mode IN BOOLEAN，session IN BOOLEAN）;

参数

表138-6 BYPASS程序参数

参数 描述

bypass_mode

  • TRUE =>结果缓存使用率缓存

  • FALSE =>结果缓存使用率已打开

会议

  • TRUE =>适用于当前会话

  • FALSE （默认值）=>适用于所有会话

使用说明

此操作是特定于数据库实例的。

例子

当需要在正在运行的系统中热修补PL / SQL代码时，可以使用此操作。如果将代码补丁应用于结果缓存函数直接或传递依赖的PL / SQL模块，则不会自动刷新与结果缓存函数关联的缓存结果（如果实例未重新启动/退回）。这必须手动实现。

要在修补过程中确保正确性，请执行以下步骤：

  1. 将结果缓存置于旁路模式，并刷新现有结果。

    BEGIN DBMS_RESULT_CACHE.BYPASS（TRUE）; DBMS_RESULT_CACHE.FLUSH;结束; /

    如果在Oracle Real Application Clusters环境中，则必须在每个实例上执行此步骤。

  2. 应用PL / SQL代码补丁。

  3. 通过关闭缓存旁路模式继续使用结果缓存。

    BEGIN DBMS_RESULT_CACHE.BYPASS（FALSE）;结束; /

    如果在Oracle Real Application Clusters环境中，则必须在每个实例上执行此步骤。

以下列表总结了基于COMPATIBLE.ADVM磁盘组属性值的-oi选项和元数据块大小的使用。

  • 如果COMPATIBLE.ADVM设置为12.2或更高，则默认情况4096数据块大小为4096字节。

  • 如果COMPATIBLE.ADVM设置为小于12.2 ，则块大小设置为512字节。

Oracle ACFS文件系统只应安装在一个安装点上。应在所有集群成员上使用相同的安装点名称。

root权限是运行mount所必需的。

适用于AIX环境的Oracle ACFS命令行工具

本主题提供AIX操作系统的Oracle ACFS命令的摘要。

在AIX上使用Oracle Grid Infrastructure安装Oracle ACFS和Oracle ADVM期间，将为Oracle ACFS和Oracle ADVM操作创建多个系统安全类（授权）。这些授权可以执行以下操作：

  • 配置Oracle ACFS和Oracle ADVM设备

  • 删除Oracle ACFS和Oracle ADVM设备

  • Oracle ACFS和Oracle ADVM设备的定义

这些授权收集在一个角色（ oracle_devmgmt ）中，该角色默认分配给Oracle Grid Infrastructure用户和Oracle ASM管理员。您可以在安装后运行lsrole或rolelist命令以查看此新角色。可以随时运行这些命令，以确保用户具有管理Oracle ACFS和Oracle ADVM的适当权限。

例如，以Oracle ASM管理员身份运行以下命令：

$ rolelist -e oracle_devmgmt

在AIX操作系统上进行初始安装后，运行根脚本的shell不会继承新角色。如果用户不存在该角色，则必须在执行Oracle ACFS或Oracle ADVM操作之前运行swrole命令。

例如，以Oracle ASM管理员身份运行以下命令：

$ swrole oracle_devmgmt

或者，您可以打开新shell以执行Oracle ACFS或Oracle ADVM操作。

在删除Oracle Grid Infrastructure期间，将从系统中删除oracle_devmgmt角色及其关联的授权。

使用-c release_version选项指定的值必须大于或等于磁盘组的COMPATIBLE.ADVM值，并且必须小于或等于正在运行的Oracle Grid Infrastructure版本。如果未指定—c release_version ，则使用COMPATIBLE.ADVM的值。设置兼容性后，无法降级。—c release_version选项可用于无法或不希望更新磁盘组的COMPATIBLE.ADVM和COMPATIBLE.ASM的情况，但您希望使用需要增加兼容性的Oracle ACFS功能。更新Oracle ACFS兼容性后，将无法再使用较早的Oracle Grid基础结构版本安装文件系统。要更改现有文件系统的兼容性，请参阅acfsutil compat set 。有关Oracle ASM磁盘组兼容性属性的信息，请参阅磁盘组兼容性 。

对于512字节元数据块大小格式，最小文件系统大小为200 MB，对于4 KB元数据块大小格式，最小文件系统大小为512 MB。

必须加载Oracle ACFS驱动程序才能使mkfs正常工作。

4 KB扇区/元数据

以下列表总结了基于COMPATIBLE.ADVM磁盘组属性值的-oi选项和元数据块大小的使用。

  • 如果COMPATIBLE.ADVM设置为12.2或更高，则默认情况4096数据块大小为4096字节。

  • 如果COMPATIBLE.ADVM设置为小于12.2 ，则块大小设置为512字节。

  • 如果COMPATIBLE.ADVM设置为小于12.2且逻辑扇区大小不是512字节，则命令失败。

对于普通用户IO请求，用户数据IO继续支持小至512字节的传输。当文件系统的Oracle ADVM卷的逻辑磁盘扇区大小为4 K时，用户Direct IO请求应在4 K文件偏移上对齐，并且是4 KB长度的倍数，以获得最佳性能。请注意，只有4 K元数据块大小格式支持具有4 K逻辑磁盘扇区的ADVM卷。

必须将COMPATIBLE.ADVM值设置为12.2或更高，以格式化逻辑磁盘大小为4096的Oracle ADVM卷，或使用-oi=4096选项格式化逻辑磁盘扇区大小为512字节的Oracle ADVM卷。

业务目录服务提供给定区域中的业务列表，并匹配指定的名称或类别。

业务目录数据来自此类数据的第三方提供商。这些提供商可能具有不同的业务类别，甚至不同的层次结构。各种方法中的统一模式是企业按主题和位置分类。位置组件很好理解;例如，对于美国，可以使用邮政编码或城市和州的组合以及可选的特定地址来确定开始搜索的位置。

85.1 DBMS_INMEMORY概述

此程序包包含用于填充和重新填充IM列存储以及从指定表中删除IM表达式的过程。

IM人口和人口增长

当数据库从磁盘读取现有行格式数据，将其转换为列式格式，然后将其存储在IM列存储中时，会发生内存中的填充（填充）。只有具有INMEMORY属性的对象INMEMORY资格进行填充。

将磁盘上的现有数据转换为柱状格式的群体与将将新数据加载到IM列存储中的重新填充不同。在其柱状数据经历显着的DML活动之后自动发生重新填充。

当对象具有INMEMORY属性且NONE以外的优先级时，数据库将根据内部优先级队列逐渐填充IM列存储中的对象。优先级为NONE对象仅在进行完全扫描时才会填充。

DBMS_INMEMORY.POPULATE过程强制对象的直接填充。DBMS_INMEMORY.POPULATE_WAIT函数强制满足指定优先级条件的对象群，并指定填充必须完成的超时间隔。DBMS_INMEMORY.REPOPULATE过程强制立即重新填充对象。

也可以看看：

Oracle Database In-Memory指南 ，了解有关IM群体的更多信息

IM表达式

IM表达式在IM列存储中填充频繁评估的查询表达式，以供后续重用。IM表达式具体化为隐藏虚拟列，前缀为字符串SYS_IME ，并且以与非虚拟列相同的方式访问。

使用DBMS_INMEMORY_ADMIN.IME_CAPTURE_EXPRESSIONS ，数据库将20个最热门的表达式作为SYS_IME列添加到各自的表中，并应用默认的INMEMORY列压缩子句。如果在上一次调用期间添加的任何SYS_IME列不再位于最新表达式列表中，则数据库会将其属性更改为NO INMEMORY 。

无论属性是INMEMORY还是NO INMEMORY ，表的最大SYS_IME列数为50。达到表的50表达式限制后，数据库将不会添加新的SYS_IME列。要为新表达式腾出空间，必须使用DBMS_INMEMORY.IME_DROP_EXPRESSIONS或DBMS_INMEMORY_ADMIN.IME_DROP_ALL_EXPRESSIONS过程手动删除SYS_IME列。

2.1.1嵌入式SQL语句

术语嵌入式SQL是指放置在应用程序中的SQL语句。因为应用程序包含SQL语句，所以它被称为a 宿主程序 ，其编写语言称为 主持语言 。例如，使用Pro * COBOL预编译器，您可以在COBOL宿主程序中嵌入SQL语句。

例如，要操作和查询Oracle数据，请使用INSERT ， UPDATE ， DELETE和SELECT语句。INSERT向数据库表添加数据行， UPDATE修改行， DELETE删除不需要的行， SELECT检索符合搜索条件的行。

Oracle预编译器支持所有Oracle语句。例如，强大的SET ROLE语句允许您动态管理数据库权限。角色是授予用户或其他角色的相关系统和对象权限，相关系统或对象权限的命名组。角色定义存储在Oracle数据字典中。您的应用程序可以使用SET ROLE语句根据需要启用和禁用角色。

只有SQL语句 - 而不是SQL * Plus语句 - 在应用程序中才有效。（SQL * Plus具有用于设置环境参数，编辑和报告格式的其他语句。）

2.1.4静态与动态SQL语句

大多数应用程序旨在处理静态 SQL语句和固定事务。在这种情况下，您知道运行时之前每个SQL语句和事务的组成。也就是说，您知道将发出哪些SQL命令，哪些数据库表可能会更改，哪些列将更新，等等。

但是，某些应用程序需要在运行时接受和处理任何有效的SQL语句。因此，在此之前您可能不知道所涉及的所有SQL命令，数据库表和列。

动态 SQL是一种高级编程技术，它允许程序在运行时接受或构建SQL语句，并对数据类型转换进行显式控制。

2.1.5嵌入式PL / SQL块

Oracle预编译器将PL / SQL块视为单个嵌入式SQL语句。因此，您可以将PL / SQL块放在可以放置SQL语句的应用程序中的任何位置。要在主机程序中嵌入PL / SQL，只需声明要与PL / SQL共享的变量，并将PL / SQL块与关键字EXEC SQL EXECUTE和END-EXEC括起来。

从嵌入式PL / SQL块，您可以灵活，安全地操作Oracle数据，因为PL / SQL支持所有SQL数据操作和事务处理命令。有关PL / SQL的更多信息，请参阅使用嵌入式PL / SQL 。

2.1.6主机和指标变量

一个 host变量是以宿主语言声明并与Oracle共享的标量或数组变量，这意味着您的程序和Oracle都可以引用其值。主机变量是Oracle与程序之间通信的关键。

您的程序使用输入主机变量将数据传递给Oracle。Oracle使用输出主机变量将数据和状态信息传递给您的程序。该程序为输入主变量赋值; Oracle将值分配给输出主机变量。

可以在可以使用表达式的任何地方使用主机变量。但是，在SQL语句中，主机变量必须以冒号（:)为前缀，以使它们与Oracle对象区分开来。

您可以将任何主机变量与可选的指示符变量相关联。指示符变量是一个整数变量，它“指示”其主变量的值或条件。您可以使用指示符变量为输入主变量分配空值，并检测输出主变量中的空值或截断值。 null是缺失，未知或不适用的值。

在SQL语句中，指示符变量必须以冒号为前缀并附加到其关联的主变量（除非为了提高可读性，否则在指示符变量之前使用可选关键字INDICATOR ）。

2.1.7 Oracle数据类型

通常，主机程序将数据输入Oracle，Oracle将数据输出到程序。Oracle将输入数据存储在数据库表中，并将输出数据存储在程序主机变量中。要存储数据项，Oracle必须知道其数据类型 ，该数据类型指定存储格式和有效值范围。

Oracle识别两种数据类型： 内部和外部 。内部数据类型指定Oracle如何在数据库列中存储数据。Oracle还使用内部数据类型来表示数据库伪列，它返回特定数据项但不是表中的实际列。

外部数据类型指定数据在主机变量中的存储方式。当主机程序向Oracle输入数据时，如有必要，Oracle会在输入主机变量的外部数据类型和数据库列的内部数据类型之间进行转换。当Oracle将数据输出到主机程序时，如有必要，Oracle会在数据库列的内部数据类型和输出主机变量的外部数据类型之间进行转换。

2.1.8数组

Oracle预编译器允许您定义数组主机变量（称为主机数组 ）并使用单个SQL语句对它们进行操作。使用数组SELECT ， FETCH ， DELETE ， INSERT和UPDATE语句，您可以轻松查询和操作大量数据。

2.1.9数据类型等价

Oracle预编译器通过允许等效数据类型为应用程序增加了灵活性。这意味着您可以自定义Oracle解释输入数据和格式化输出数据的方式。

在逐个变量的基础上，您可以将等效支持的主机语言数据类型添加到Oracle外部数据类型。

2.1.10私有SQL区域，游标和活动集

为了处理SQL语句，Oracle打开了一个名为a的工作区 私有SQL区域 。私有SQL区域存储执行SQL语句所需的信息。称为游标的标识符允许您命名SQL语句，访问其私有SQL区域中的信息，并在某种程度上控制其处理。

对于静态SQL语句，有两种类型的游标： 隐式和显式 。Oracle隐式声明所有数据定义和数据操作语句的游标，包括只返回一行的SELECT语句（查询）。但是，对于返回多行的查询，要处理超出第一行，必须显式声明游标（或使用主机数组）。

检索到的行集称为活动集 ;它的大小取决于符合查询搜索条件的行数。您使用显式游标来标识当前正在处理的行，称为当前行 。

想象一下，这组行被返回到终端屏幕。屏幕光标可以指向要处理的第一行，然后指向下一行，依此类推。类似地，显式游标“指向”活动集中的当前行，允许程序一次处理一行。

2.1.11交易

事务是一系列逻辑上相关的SQL语句（例如，两个UPDATE ，它们将一个银行帐户存入贷方并借记另一个银行帐户），Oracle将其视为一个单元，因此语句带来的所有更改都是永久性的或在同一时间撤消时间。当前事务包括自上次执行数据定义， COMMIT或ROLLBACK语句以来执行的所有数据操作语句。

为了帮助确保数据库的一致性，Oracle预编译器允许您使用COMMIT ， ROLLBACK和SAVEPOINT语句定义事务。COMMIT使当前事务期间所做的任何更改成为永久性的。ROLLBACK结束当前事务并撤消自事务开始以来所做的任何更改。SAVEPOINT标记交易中的当前点;与ROLLBACK ，它撤消了部分交易。

2.1.12错误和警告

执行嵌入式SQL语句时，它会成功或失败，并可能导致错误或警告。您需要一种方法来处理这些结果。Oracle预编译器提供了四种错误处理机制：

  • SQLCODE状态变量

  • SQLSTATE状态变量

  • SQL通信区（SQLCA）和WHENEVER语句

  • Oracle通信区域（ORACA）

SQLCODE / SQLSTATE状态变量

执行SQL语句后，Oracle Server会将状态代码返回给名为SQLCODE或SQLSTATE的变量。状态代码指示SQL语句是成功执行还是导致错误或警告条件。

SQLCA和WHENEVER声明

SQLCA是一种数据结构，它定义了Oracle用于将运行时状态信息传递给程序的程序变量。使用SQLCA，您可以根据Oracle对刚尝试的工作的反馈采取不同的操作。例如，您可以验证DELETE语句是否成功，如果成功，则删除了多少行。

使用WHENEVER语句，您可以指定Oracle检测到错误或警告条件时要自动执行的操作。这些操作包括继续下一个语句，调用子例程，分支到带标签的语句或停止。

ORACA

如果需要有关运行时错误的更多信息，而不是SQLCA提供的信息，则可以使用ORACA。 ORACA是一种处理Oracle通信的数据结构。它包含游标统计信息，有关当前SQL语句的信息，选项设置和系统统计信息。

3.1.3.1使用JDBC

JDBC是一种行业标准API，允许您将SQL语句嵌入为Java方法参数。JDBC基于X / Open SQL调用级接口（CLI），符合SQL-92标准的入门级。每个供应商（例如Oracle）都通过实现来创建其JDBC实现标准java.sql包的接口。Oracle提供了以下实现这些标准接口的JDBC驱动程序：

  • JDBC Thin驱动程序，100％纯Java解决方案，可用于客户端应用程序或applet，无需安装Oracle客户端。

  • JDBC OCI驱动程序，用于客户端应用程序并需要安装Oracle客户端。

  • 嵌入在Oracle数据库中的服务器端JDBC驱动程序。

使用JDBC是执行以下任务的分步过程：

  1. 获取连接句柄
  2. 为所需的SQL操作创建某种类型的语句对象
  3. 分配要绑定到SQL操作的任何本地变量
  4. 执行操作
  5. （可选）检索结果集

这个过程对于许多应用程序来说已经足够了，但对于任何复杂的语句来说都很麻烦动态SQL操作（直到运行时才知道操作）需要JDBC。但是，在典型的应用程序中，这代表了少数SQL操作。

3.1.3.2使用SQLJ

SQLJ提供了一种行业标准的方法，可以通过一个简单的步骤将任何静态SQL操作直接嵌入到Java源代码中，而无需多个JDBC步骤。 Oracle SQLJ符合X3H2-98-320美国国家标准协会（ANSI）标准。

SQLJ由一个翻译器和一个运行时组件组成，该翻译器是一个支持标准SQLJ编程语法的预编译器。在.sqlj文件中创建SQLJ源代码后，使用.sqlj程序处理它。翻译器将SQLJ源代码转换为标准Java源代码，SQL操作转换为对SQLJ运行时的调用。在Oracle Database SQLJ实现中，转换程序调用Java编译器来编译Java源代码。运行SQLJ应用程序时，SQLJ运行时调用JDBC与数据库进行通信。

SQLJ还使您能够在运行时捕获SQL语句中的错误。JDBC代码是纯Java，直接编译。编译器无法检测SQL错误。另一方面，当您翻译SQLJ代码时，翻译器在语义和语法上分析嵌入式SQL语句，在开发期间捕获SQL错误，而不是允许最终用户在运行应用程序时捕获它们。

集成开发环境（IDE）（例如Oracle JDeveloper）可以在构建SQLJ程序时对其进行转换，编译和自定义。Oracle JDeveloper是一个基于Microsoft Windows的Java编程可视化开发环境。如果您没有使用IDE，请使用前端SQLJ实用程序sqlj 。您可以按如下方式运行它：

3.1.3.6与PL / SQL的交互

所有Oracle JDBC驱动程序都与Oracle SQL和PL / SQL无缝通信，请务必注意SQLJ与PL / SQL进行互操作。您可以开始使用SQLJ，而无需重写任何PL / SQL存储过程。Oracle SQLJ包含用于调用PL / SQL存储过程的语法，还允许您在SQLJ语句中嵌入匿名PL / SQL块。

Oracle JVM的命令行界面类似于使用JDK或JRE shell命令。您可以：

  • 使用标准的-classpath语法指示要在何处查找要加载的类

  • 使用标准-D语法设置系统属性

接口是一个PL / SQL函数，它接受一个字符串（ VARCHAR2 ）参数，将其解析为命令行输入，如果它正确形成，则在Oracle JVM中运行指示的Java方法。为此，PL / SQL包DBMS_JAVA提供以下功能：

  • runjava

  • runjava_in_current_session

runjava

此函数将Java命令行作为其唯一参数，并在Oracle JVM中运行它。成功完成后返回值为null，否则返回错误消息。命令行的格式与JDK shell命令的格式相同，即：

[选项开关] name_of_class_to_execute [arg1 arg2 ... argn]

您可以使用选项开关-classpath, -D, -Xbootclasspath,和-jar. 此函数与runjava_in_current_session函数的不同之处在于，它在运行当前命令之前清除会话中以前使用Java时剩余的任何Java状态。特别是，这对于保证在类初始化时从-classpath和-D参数派生的静态变量值反映当前命令行中这些开关的值是必要的。 

FUNCTION runjava（cmdline VARCHAR2）RETURN VARCHAR2;

runjava_in_current_session

此函数与runjava函数相同，只是在执行当前命令行之前，它不会清除会话中先前使用Java的剩余Java状态。 

功能runjava_in_current_session（cmdline VARCHAR2）RETURN VARCHAR2;

句法

命令行的语法具有以下形式：

[-options] classname [arguments ...][-options] -jar jarfile [arguments ...]

选项

-classpath -D -Xbootclasspath -Xbootclasspath / a -Xbootclasspath / p -cp

注意：

第一种形式的作用是使用参数运行classname标识的类的main方法。第二种形式的作用是运行由JAR标识的JAR文件的清单中的Main-Class属性标识的类的main方法。这类似于JDK / JRE如何解释此语法。

3.1.5使用客户端存根的概述

Oracle数据库10g引入了客户端存根（以前称为本机Java接口），用于调用服务器端Java代码。它是一种简化的应用程序集成。客户端和中间层Java应用程序可以直接在数据库中调用Java，而无需定义PL / SQL包装器。客户端存根使用服务器端Java类反射功能。

在以前的版本中，从数据库客户端调用Java存储过程和函数需要对相关的PL / SQL包装器进行Java数据库连接（JDBC）调用。必须使用SQL签名和Java实现手动发布每个包装器。这有以下缺点：

  • 签名仅允许具有直接SQL等效的Java类型

  • Java中发布的异常未正确返回

从Oracle Database 12 c第2版（12.2.0.1）开始，您可以使用Oracle JVM Web服务调用实用程序生成客户端存根。

2.8.1登录时有选择地捕获用户

配置Oracle Health Check Collections Manager以捕获用户详细信息并为用户分配Oracle Health Check Collections Manager角色。

在登录期间自动捕获用户可自动执行用户管理。您无需手动创建用户。

默认情况下，Oracle Health Check Collections Manager：

  • 捕获使用LDAP身份验证登录的用户的详细信息

  • 为其分配Oracle Health Check Collections Manager角色，例如DBA角色。

注意：

Oracle Health Check Collections Manager角色特定于Oracle Health Check Collections Manager，并不等同于系统特权。例如，DBA角色未授予SYSDBA系统特权。

16.1.1 RMAN验证的目的

16.1.2 RMAN验证的基本概念

数据库可防止导致无法使用的备份文件或损坏的已还原数据文件的操作。

数据库自动执行以下操作：

  • 在恢复或恢复数据文件时阻止访问它们

  • 每次只允许每个数据文件执行一次还原操作

  • 确保以正确的顺序应用增量备份

  • 在备份文件中存储信息以允许检测损坏

  • 每次读取或写入时检查一个块，以便在检测到损坏后立即报告

16.1.2.1关于校验和和损坏的块

损坏的块是一个已更改的块，因此它与Oracle数据库期望找到的块不同。

块损坏可能由几种不同的故障引起，包括但不限于以下内容：

  • 磁盘和磁盘控制器出现故障

  • 记忆力不好

  • Oracle数据库软件缺陷

DB_BLOCK_CHECKSUM是一个数据库初始化参数，用于控制数据库中块的校验和以及数据库中的联机重做日志文件（而不是备份）的写入。如果DB_BLOCK_CHECKSUM是typical ，则数据库在正常操作期间计算每个块的校验和，并在将其写入磁盘之前将其存储在块的头中。当数据库稍后从磁盘读取块时，它会重新计算校验和并将其与存储的值进行比较。如果值不匹配，则块已损坏。

默认情况下， BACKUP命令计算每个块的校验和并将其存储在备份中。BACKUP命令忽略DB_BLOCK_CHECKSUM的值，因为此初始化参数适用于数据库中的数据文件，而不是备份。

16.1.2.2关于物理和逻辑块损坏

在物理损坏（也称为媒体损坏）中，数据库根本无法识别块：校验和无效，块包含全零，或块的页眉和页脚不匹配。

注意：

默认情况下， BACKUP命令计算每个块的校验和并将其存储在备份中。如果指定NOCHECKSUM选项，则RMAN在创建备份时不会对块执行校验和。

在逻辑损坏中 ，块的内容在逻辑上是不一致的。逻辑损坏的示例包括行段或索引条目的损坏。如果RMAN检测到逻辑损坏，则会将该日志记录在警报日志和服务器会话跟踪文件中。

默认情况下，RMAN不检查逻辑损坏。但是，如果在BACKUP命令中指定CHECK LOGICAL ，则RMAN会测试数据和索引块是否存在逻辑损坏，例如行片段或索引条目的损坏，并将它们记录在自动诊断存储库（ADR）中的警报日志中。如果在备份或还原文件时使用具有以下配置的RMAN，则它会检测可能检测到的所有类型的块损坏：

  • 在数据库的初始化参数文件中，设置DB_BLOCK_CHECKSUM=typical以便数据库自动计算数据文件校验和（不是用于备份，而是用于数据库使用的数据文件）

  • 不要在BACKUP命令之前使用SET MAXCORRUPT以便RMAN不会容忍任何未标记的块损坏。

  • 在BACKUP命令， 不指定NOCHECKSUM写备份时，这样RMAN计算选项校验

  • 在BACKUP和RESTORE命令中，指定CHECK LOGICAL选项，以便RMAN检查逻辑和物理损坏

16.1.2.3关于RMAN备份中损坏块的限制

您可以使用SET MAXCORRUPT命令设置RMAN备份文件中允许的未标记损坏总数。默认值为零，这意味着RMAN不会容忍任何类型的未标记的损坏块。

如果在备份期间RMAN遇到未标记的损坏块时超出了MAXCORRUPT限制，则RMAN将终止备份。否则，RMAN将新检测到的损坏块写入备份，并使用特殊标头指示该块已标记为已损坏。您可以使用VALIDATE命令确定哪些块被标记为已损坏并找到任何未标记的损坏块。

因为RMAN允许在备份中标记损坏的块，并且因为可以指示RMAN允许未标记的损坏块在备份中被标记为损坏（当使用MAXCORRUPT时），所以可以恢复具有标记为多个块的数据文件腐败。如果备份此恢复的数据文件（假设没有发生新的损坏），即使没有MAXCORRUPT设置，备份MAXCORRUPT成功。这是因为先前标记的损坏不会阻止RMAN完成备份。

Oracle数据库支持用于检测，修复和监视块损坏的不同技术。

该技术取决于损坏是否是块间损坏或内部损坏 。在intrablock损坏中，损坏发生在块本身内。这种损坏可以是物理的也可以是逻辑的。在块间损坏中，损坏发生在块之间，并且只能是逻辑的。

Oracle Database Advanced Queuing（AQ）允许您在队列中共享数据和事件。它可以在数据库中或从一个数据库传播信息到另一个数据库，将信息路由到指定的目的地。它为捕获和管理事件以及与其他数据库和应用程序共享事件提供了功能和灵活性。

您可以使用AQ执行以下所有操作：

  • 捕获数据库中的更改。 您可以配置后台捕获过程以捕获对表，数据库模式或整个数据库所做的更改。捕获过程捕获重做日志中的更改，并将每个捕获的更改格式化为逻辑更改记录（LCR）。在重做日志中生成更改的数据库称为源数据库。

  • 将事件排入队列。 可以在队列中暂存两种类型的事件：LCR和用户消息。捕获过程将LCR事件排入您指定的队列。然后，队列可以在同一数据库或其他数据库中共享LCR事件。您还可以使用用户应用程序显式地将用户事件排入队列。这些显式排队的事件可以是LCR或用户消息。

  • 将事件从一个队列传播到另一个队列。 这些队列可能位于同一数据库中，也可能位于不同的数据库中。

  • 出列事件。 后台应用程序可以使事件出列。您还可以使用用户应用程序显式地使事件出列。

  • 在数据库中应用事件。 您可以配置应用进程以应用队列中的所有事件或仅应用您指定的事件。您还可以配置应用进程以调用自己的PL / SQL子程序来处理事件。

    应用LCR事件和处理其他类型事件的数据库称为目标数据库。在一些配置中，源数据库和目标数据库可以是相同的。

37.6.1将包含许多PDF的AQ XML消息存储为一条记录

您可以使用Oracle数据库高级队列（AQ）在企业之间交换XML文档，其中每个消息都包含XML标头，XML附件（XML数据流），DTD和PDF文件。数据可以存储在数据库表中，例如队列表。

您可以通过以下方式实现此目的：

  • 通过使用（ CLOB ， RAW ，...）属性定义对象类型，并将其存储为单个消息。

CREATE TYPE mypayload_type as OBJECT（xmlDataStream CLOB，dtd CLOB，pdf BLOB）;

37.6.2在邮件入队后添加新收件人

您可以使用队列表来支持邮件分配。
