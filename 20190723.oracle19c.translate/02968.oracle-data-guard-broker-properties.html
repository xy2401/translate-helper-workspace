8 Oracle Data Guard Broker属性

使用Oracle Data Guard代理属性来管理实例或整个配置。

代理配置和数据库属性可帮助您查看和控制整个代理配置，单个配置成员，重做传输服务和日志应用服务的行为。

  • 配置属性

  • 可监视（只读）属性

  • 可配置的属性

属性具有配置范围范围，数据库范围范围，远程同步实例范围或特定于实例的范围。配置范围的属性控制代理在配置的所有成员上的行为。这些属性的值均匀地应用于配置的所有成员。

数据库范围的属性允许您查看或控制特定数据库的行为。如果数据库（主数据库或备用数据库）是由多个实例组成的Oracle RAC数据库，则此类属性的值将统一应用于该数据库的所有实例。

远程同步实例范围的属性允许您查看或控制远程同步实例的行为。

特定于实例的属性允许您查看或控制单个数据库实例的行为。这样的属性存在于Oracle RAC数据库的所有实例中，但其值可能因特定实例而异。

注意：

您可以使用Oracle Data Guard代理命令行界面（DGMGRL）来查看或修改代理属性。

Oracle Enterprise Manager云控制（云控制）在“编辑属性”页面上显式显示了一些代理属性。来自其他属性的信息可以隐式合并到Cloud Control显示的其他Web页面中。有关Cloud Control如何呈现该属性的信息，请参阅每个属性的个别描述。

8.1配置属性

配置属性控制代理配置的行为。

您可以使用DGMGRL或Cloud Control查看和动态更新这些属性的值。但是，某些属性只能通过DGMGRL更新。

配置属性具有配置范围;意味着您为属性设置的值均匀地应用于配置中的每个成员。

使用EDIT CONFIGURATION SET PROPERTY命令设置EDIT CONFIGURATION SET PROPERTY ，如以下示例所示。

例1

8.1.1 BystandersFollowRoleChange

BystandersFollowRoleChange配置属性确定在故障转移期间 （值= ALL ）或故障转移（值= NONE ） 之后是否评估旁观者备用数据库。

ALL和NONE选项的描述如下：

  • ALL - 在故障转移过程中，代理确定旁观者备用数据库是在故障转移目标备用数据库之前还是之后（即，将成为新主数据库的备用数据库）。

    如果旁观者备用在前面，它们将被禁用状态为ORA-16661作为故障转移操作的一部分。故障转移完成后，必须恢复旁观者备用数据库。代理通过闪回将备用数据库恢复到目标备用数据库成为主数据库的SCN，并设置从新主数据库到备用数据库的重做传输配置。

    如果旁观者备用设备落后，那么代理只需将重做传输配置从即将成为新的主设备设置到这些备用设备并完成故障转移过程。

  • NONE - 在故障转移过程中，代理不会评估旁观者备用状态作为故障转移操作的一部分。它们被标记为已禁用，状态为ORA-16661以便稍后进行评估。代理只需完成故障转移即可尽快生成新的主数据库。

    故障转移完成后，您可以恢复旁观者备用数据库。在恢复旁观者的过程中，经纪人确定旁观者是在新的主要人员之前还是之后。如果旁观者位于新主节点之前，则代理会自动将备用节点闪回到目标备用数据库成为主数据库的SCN，并设置从新主节点到备用数据库的重做传输。（即使不需要闪回，代理也会设置从新主服务器到这些备用服务器的重做传输配置。）

    NONE选项减少了故障转移的处理时间，但禁用了配置中所有旁观者数据库的代理管理。如果启用了快速启动故障转移，则观察器会在故障转移完成后自动恢复备用数据库。否则，您必须在故障转移完成后手动恢复备用数据库。

8.1.13 PreferredObserverHosts

PreferredObserverHosts配置属性允许您在主数据库处于主要角色时列出您希望作为主观察者主机的主机。

8.1.14 PrimaryLostWriteAction

PrimaryLostWriteAction配置属性确定备用数据库检测到主数据库中发生丢失写入时采取的操作。

注意：

PrimaryLostWriteAction配置属性仅在Oracle Database 11g第2版（11.2.0.4）和Oracle Database 12c第1版（12.1.0.2）及更高版本中可用。

可能的行动如下：

  • CONTINUE - 即使备用数据库检测到主数据库中发生了丢失的写入，主数据库也会继续运行。这是默认操作。

  • SHUTDOWN - 如果备用数据库检测到主数据库中发生了丢失写入，则禁用快速启动故障切换，并且主数据库执行关闭中止。不会发生自动故障转移。

  • FAILOVER - 如果在最高性能模式下启用快速启动故障转移，并且配置在为FastStartFailoverLagLimit属性指定的可接受限制内，则观察器将启动故障转移。

  • FORCEFAILOVER - 如果启用了快速启动故障切换（在最大性能或最大可用性模式下），则观察器将启动故障切换。此选项会导致数据丢失故障转移。

对于FAILOVER和FORCEFAILOVER选项，如果禁用快速启动故障转移，则不会发生故障转移，但主服务器将关闭。

如果备用数据库检测到主数据库中发生了丢失写入，则在满足以下任一条件时禁用快速启动故障转移：

  • PrimaryLostWriteAction设置为CONTINUE
  • PrimaryLostWriteAction设置为SHUTDOWN
  • PrimaryLostWriteAction设置为FAILOVER ，保护模式设置为最大可用性或最大保护

诊断信息将写入主数据库和检测到丢失写入的备用数据库中的数据库警报和代理日志。

8.2.4 LsbyFailedTxnInfo（逻辑备用失败的事务信息）

LsbyFailedTxnInfo属性标识导致日志应用服务停止的失败事务。

此属性包含一个字符串，其中包含DBA_LOGSTDBY_EVENTS视图中的以下值：

  • XIDUSN ：交易ID撤消段号

  • XIDSLT ：交易ID槽号

  • XIDSQN ：交易ID序列号

  • STATUS_CODE ：属于STATUS消息的状态（或Oracle错误代码）

  • STATUS ：进程当前活动的描述或停止日志应用服务的原因

事务ID和状态信息由一串数字符号（###）分隔。

此属性适用于逻辑备用数据库。

MAX_SGA是最大系统全局区域， MAX_SERVERS是并行查询服务器的最大数量。这些值由LsbyParameters属性中的一串数字符号（###） LsbyParameters 。

此属性适用于逻辑备用数据库。

8.3.10 DbFileNameConvert

DbFileNameConvert配置属性将物理备用数据文件名与主数据文件文件名区分开来。

如果将数据文件添加到主数据库，则此属性会将主数据库上的数据文件名转换为物理备用数据库上的数据文件。

此属性用于以下情况：

  • 在物理备用安装时，如果物理备用系统上的数据文件文件路径与主数据库系统不同，则它用于将主数据文件文件名重命名为备用数据文件文件名。

  • 在主数据库上创建新数据文件时，如果StandbyFileManagement配置属性设置为'AUTO' ， StandbyFileManagement在物理备用数据库上创建相应的新数据文件。Oracle使用DbFileNameConvert属性中的数据文件文件路径映射信息来确定新备用数据文件的备用文件路径。如果StandbyFileManagement属性设置为'MANUAL' ，则必须将相应的文件添加到物理备用数据库。

注意：

将数据库添加到配置时，代理会将此属性的初始值设置为DB_FILE_NAME_CONVERT初始化参数的内存中值。此参数的内存中值和服务器参数文件（SPFILE）值可能不同。如果要使用参数的内存中值，则启用数据库，代理将确保将参数的SPFILE值设置为内存中值。如果要使用SPFILE值，请将属性值设置为存储在SPFILE中的参数值。然后启用数据库。

8.3.22 LogXptMode

LogXptMode配置属性使您可以设置重做传输服务。

您将每个配置成员上的重做传输服务设置为以下模式之一：

  • 同步

    使用LOG_ARCHIVE_DEST_ n初始化参数的SYNC和AFFIRM属性为此配置成员配置重做传输服务。备用重做日志文件是必需的。最大保护或最大可用性数据保护模式需要此模式。此重做传输服务可为主数据库提供最高级别的数据保护，但也会产生最高的性能影响。

    使用LOG_ARCHIVE_DEST_ n初始化参数的ASYNC和NOAFFIRM属性为此配置成员配置重做传输服务。备用重做日志文件是必需的。此模式可为主数据库提供中等级别的数据保护，并且比SYNC产生更低的性能影响。

  • FASTSYNC

    使用LOG_ARCHIVE_DEST_ n初始化参数的SYNC和NOAFFIRM属性为此配置成员配置重做传输服务。此模式仅在最大可用性保护模式下可用。由于FASTSYNC传输模式使用LOG_ARCHIVE_DEST_n参数的NOAFFIRM属性，因此可能会丢失数据。

8.3.36.1重做路由规则

RedoRoutes属性设置为包含一个或多个重做路由规则的字符串。

每个规则都包含在一组括号中，如下所示：

（redo_routing_rule_1）[（redo_routing_rule_ n ）]

重做路由规则包含重做源字段和由冒号分隔的重做目标字段：

（重做源：重做目的地）

重做源字段必须包含关键字LOCAL或ANY ，或以逗号分隔的DB_UNIQUE_NAME值列表，如下所示：

{LOCAL |任何| db_unique_name_1，[，db_unique_name_ n ]}

  • LOCAL关键字是本地数据库名称的别名。此关键字不能用于远程同步实例。

  • ANY关键字是配置中任何数据库的别名。

  • 无法通过使用LOCAL关键字显式或隐式地将数据库指定为在给定数据库中定义的多个重做路由规则中的重做源。

重做目标字段必须包含关键字ALL或以逗号分隔的重做目标组列表，每个目标组都包含具有可选优先级属性和可选重做传输模式属性的目标数据库：

{ALL [xpt_mode] | redo_dest_group_1 [，redo_dest_group_ n ]}

  • ALL关键字是配置中所有可能目标的别名。

  • redo_dest_group_ n是：

    {db_unique_name_1 [xpt_mode] | （db_unique_name_1 [xpt_mode] [PRIORITY = n ] [，db_unique_name_ n [xpt_mode] [PRIORITY = n ]]）}

    可选的xpt_mode指定用于将重做发送到关联目标的重做传输模式。它可以具有以下三个值之一： ASYNC ， SYNC或FASTSYNC 。如果未指定重做传输属性，则使用的传输模式将是重做目标的LogXptMode属性指定的传输模式。

    可选的[ PRIORITY= n ]可以有n = 1~8。远同步成员的默认值为1，非远同步成员的默认值为8。重要的是要了解不同的组和优先级设置如何影响各种条件下的重做传输。以下示例描述了一些示例情况。

示例1：组内的不同优先级

假设代理配置中有三个成员：
  • PRI （主数据库）
  • SB1 （备用数据库）
  • FS1 （远程同步实例）
主数据库PRI生成备用数据库SB1使用的重做日志。由于存在远程同步实例FS1 ，因此备用数据库有两种可能的重做传递路径：
  • （路径1） PRI - > FS1 - > SB1
  • （路径2） PRI - > SB1
假设（路径1）比（路径2）更合乎需要。这可以用RedoRoutes属性表示如下：
  • PRI - RedoRoutes = (local : ( FS1 PRIORITY=1, SB1 PRIORITY=2 ) )
  • FS1 - RedoRoutes = ( PRI : SB1 )

如PRI RedoRoutes属性中所指定，主（ PRI ）有两个目的地：一个用于FS1 ， PRIORITY=1 ，一个用于SB1 ， PRIORITY=2 。较小的优先级数字意味着较高的优先级，因此主PRI尝试将重做日志发送到FS1 。

如果FS1可用，则主要装载到FS1 ，其PRIORITY=1 。如果FS1不可用，则主要装载到SB1 ，其中PRIORITY=2 。当FS1再次变为活动状态时，主要设备将再次发送给它，因为优先级1高于优先级2。

示例2：组内的相同优先级

假设您将另一个远程同步实例FS2添加到示例1中设置的配置中，然后按如下方式更新RedoRoutes属性：
  • PRI - RedoRoutes = (local : ( FS1 PRIORITY=1, FS2 PRIORITY=1 ) )
  • FS1 - RedoRoutes = ( PRI : SB1 )
  • FS2 - RedoRoutes = ( PRI : SB1 )

主PRI现在有两个目的地， FS1和FS2 ，具有相同的优先级。主要必须选择FS1或FS2 。假设主要选择FS1 。

如果FS1可用，则主要装载到FS1 。如果FS1不可用，则主要装载到FS2 。即使在FS1再次变为活动状态之后，主数据库也继续传送到FS2因为FS1和FS2具有相同的优先级。如果FS2发生故障，则主要发送到FS1 。

示例3：多个组

假设您将一个备用数据库SB2添加到配置中，然后按如下所示更新RedoRoutes属性，以便主数据库有两个目标组：
  • PRI - RedoRoutes = (local : ( FS1 PRIORITY=1, SB1 PRIORITY=2 ), ( FS2 PRIORITY=1, SB2 PRIORITY=2 ) )
  • FS1 - RedoRoutes = ( PRI : SB1 )
  • FS2 - RedoRoutes = ( PRI : SB2 )

一般规则是每个组都有一个活动的重做路径。（有关多个活动重做路径的方案，请参见示例4。）主要为第一组建立一个重做传递路径（ FS1 PRIORITY=1 ， SB1 PRIORITY=2 ），并为第二组建立另一个重做传递路径（ FS2 PRIORITY=1 ， SB2 PRIORITY=2 ）。

  • 如果FS1和FS2都可用，则主要装载到FS1和FS2 。

  • 如果FS1不可用且FS2可用，则主要装载到SB1和FS2 。

  • 如果FS1可用且FS2不可用，则主要装载到FS1和SB2 。

  • 如果FS1和FS2都不可用，则主要装载到SB1和SB2 。

示例4：PRIORITY属性设置为8

设置PRIORITY=8具有特殊含义。如果主船重做到与目的地PRIORITY=8 ，那么它必须运送到每PRIORITY=8目的地。假设您按如下所示更新RedoRoutes属性，以便主服务器有一个包含三个目标的组：
  • PRI - RedoRoutes = (local : ( FS1 PRIORITY=1, SB1 PRIORITY=8, SB2 PRIORITY=8 ) )
  • FS1 - RedoRoutes = ( PRI : SB1, SB2 )

这些设置导致以下行为：

  • 如果FS1可用，则主要装载到FS1 。

  • 如果FS1不可用，则主要装载到SB1和SB2因为它们都具有PRIORITY=8

  • 如果FS1再次变为活动状态，则主要设备将发送到FS1 。

高级重做传输设置的使用说明

以下使用说明适用于高级重做传输设置：

  • RedoRoutes属性的默认值为NULL，在主数据库中将其视为(LOCAL : ALL) 。

  • 如果重做源字段指定当前主数据库，则重做路由规则处于活动状态。如果规则处于活动状态，则主数据库重做将由定义规则的数据库发送到该规则的重做目标字段中指定的每个目标。

  • 必须为级联目标明确指定ASYNC重做传输属性，以实现到该目标的实时级联。

  • 无法配置RedoRoutes属性，以便在将物理备用数据库转换为快照备用数据库时，快照备用数据库会将重做数据发送给另一个成员。

  • 仅当重做源字段设置为LOCAL才能为逻辑备用数据库设置RedoRoutes属性。
