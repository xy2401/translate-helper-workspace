自动升级配置文件示例

使用这些示例可以了解如何修改自己的AutoUpgrade配置文件。

以下示例是具有两个数据库的AutoUpgrade配置文件。

示例3-1具有两个数据库条目的自动升级配置文件

##全局日志记录目录适用于所有作业＃global.autoupg_log_dir = / database / jobmgr＃顶级日志记录目录（必需）## Database 1＃upg1.dbname = hr＃数据库名称（必填）upg1.source_home = / database / apps / 11204 / dbhome_1＃Source Home（必填）upg1.target_home = / database / apps / 18100 / dbhome_1＃Target home（必填）upg1.sid = hr1＃Oracle Sid（必填）upg1.start_time = now＃操作开始时间（必需）upg1.pdbs = *＃全部或以逗号分隔的pdb列表（可选）upg1.log_dir = / database / logs / hr＃本地日志目录（必需）upg1.upgrade_node = nodename1＃将运行操作的升级节点（必需）upg1.run_utlrp = yes #yes（默认）运行utlrp作为升级的一部分，no跳过它（可选）upg1.timezone_upg = yes＃yes（默认）升级时区（如果需要），不跳过它（可选） upg1.target_version = 18.1＃Oracle Home Target版本号（必填）## Database 2＃upg2.dbname = sales upg2.source_home = / database / apps / 12202 / dbhome_1 upg2.target_home = / d atabase / apps / 18100 / dbhome_1 upg2.sid = sales1 upg2.start_time = now upg2.log_dir = / database / logs / sales upg2.upgrade_node = nodename2 upg2.timezone_upg = no upg1.target_version = 12.2

参数upgrade_node引用实际系统名称（ hostname ），而不是指定给主机名的别名。您还可以使用关键字localhost来引用当前系统。

示例3-2使用AutoUpgrade配置文件条目标准化升级

在以下配置文件中，您可以了解如何使用AutoUpgrade配置文件条目来标准化其数据库配置。全局PFILE条目应用于配置文件中的所有数据库。本地PFILE条目仅应用于配置文件中的特定数据库。这些PFILE值的语法遵循PFILE配置的相同Oracle规则。 

##示例全局pfile配置条目#global.del_during_upgrade_pfile = / database / pfiles / global_during_delinit.ora global.add_during_upgrade_pfile = / database / pfiles / global_during_addinit.ora global.del_after_upgrade_pfile = / database / pfiles / global_after_delinit.ora global.add_after_upgrade_pfile = / database /pfiles/global_after_addinit.ora##示例本地pfile配置条目#upg1.del_during_upgrade_pfile = / database / pfiles / hr_during_delinit.ora upg1.add_during_upgrade_pfile = / database / pfiles / hr_during_addinit.ora upg1.del_after_upgrade_pfile = / database / pfiles / hr_after_delinit.ora upg1.add_after_upgrade_pfile = /数据库/ PFILES / hr_after_addinit.ora

在AutoUpgrade过程中，将创建文件during_upgrade_pfile_dbname.ora和after_upgrade_pfile_dbname.ora 。这些文件用于在升级期间和升级后启动数据库。如果要在升级期间或升级后更改系统参数，则可以修改这两个文件。

首先应用全局PFILE条目，然后应用由作业前缀upgl指定的本地upgl条目。在这两个配置文件中，首先应用参数del_upgrade_pfile中的条目，然后是参数add_upgrade_pfile条目。在这些参数PFILE配置条目或者直接施加到PFILE during_upgrade_pfile_dbname.ora或到PFILE after_upgrade_pfile_dbname.ora取决于哪个， PFILE靶向。

操作：

  • del_during_upgrade_pfile从during_upgrade_pfile_dbname.ora删除条目
  • add_during_upgrade_pfile向during_upgrade_pfile_dbname.ora添加条目。
  • del_after_upgrade_pfile从after_upgrade_pfile_dbname.ora删除条目
  • add_after_upgrade_pfile向after_upgrade_pfile_dbname.ora.添加条目after_upgrade_pfile_dbname.ora.

参数del_during_upgrade_pfile和del_after_upgrade_pfile引用的文件在每一行上都列出了一个数据库参数。您不能向参数添加任何前缀，因为整行是参数名称的一部分。请考虑以下示例：

## global.del_during_upgrade_pfile #process * .open_cursors

此配置设置的结果是从配置文件中列出的每个数据库的PFILE删除对processes参数的所有引用，但不删除对open_cursors参数的引用：仅删除具有前缀的open_cursors实例。但是，从PFILE删除的参数包括所有带前缀的参数。例如，使用此语法删除*.processes和instance_name.processes 。

参数add_during_upgrade_pfile和add_after_upgrade_pfile引用的文件在每行上都列出了一个参数，格式为parameter = value 。如果需要，可以将值字段留空。如果参数以*.为前缀*.或instancename. ，那些引用不会添加到修改后的PFILE.要更新现有参数的值，必须先将其删除。然后，您可以使用所需的值添加参数。请考虑以下示例：

## global.add_during_upgrade_pfile＃processes = 400 * .open_cursors = 250 instancename.locallistener =

此全局配置文件条目导致将以下条目添加到配置文件中列出的每个数据库的PFILE：

processes = 400 open_cursors = 250 locallistener =

参数after_upgrade_pfile_dbname用于在升级后过程中创建数据库SPFILE。
