目录对象

DIRECTORY对象有助于管理BFILE数据类型的访问和使用。

DIRECTORY对象为数据库服务器文件系统上的物理目录指定逻辑别名 ，在该目录下要访问的文件所在的位置。仅当在DIRECTORY对象上授予所需的访问权限时，才能访问服务器文件系统中的文件。您还可以使用Oracle Enterprise Manager Cloud Control来管理DIRECTORY对象。

也可以看看：

  • 在Oracle数据库SQL语言参考中 CREATE DIRECTORY

  • 有关Oracle Enterprise Manager云控制的说明，请参见“ Oracle数据库管理员指南”

初始化BFILE定位器

DIRECTORY对象提供了管理文件位置的灵活性，而不是强制您对应用程序中物理文件的绝对路径名进行硬编码。

目录对象名称与SQL和PL / SQL中的BFILENAME函数或OCI中的OCILobFileSetName()一起使用，用于初始化BFILE定位符。

警告：

数据库不验证您指定的目录和路径名是否确实存在。您应该注意在操作系统中指定有效的目录。如果操作系统使用区分大小写的路径名，请确保以正确的格式指定目录。不需要指定终止斜杠（例如， /tmp/不是必需的，只需使用/tmp ）。

目录规范在路径中的任何位置都不能包含“..”（例如， /abc/def/hij.. ）。

如何将操作系统文件与BFILE关联

要将操作系统文件关联到BFILE ，首先要创建一个DIRECTORY对象，该对象是操作系统文件的完整路径名的别名。

要将现有操作系统文件与特定表的相关数据库记录相关联，请使用Oracle SQL DML（数据操作语言）。例如：

  • 使用INSERT初始化BFILE列以指向服务器文件系统中的现有文件。

  • 使用UPDATE更改BFILE的引用目标。

  • 将BFILE初始化为NULL ，然后稍后更新它以使用BFILENAME函数引用操作系统文件。

  • OCI用户还可以使用OCILobFileSetName()来初始化BFILE定位符变量，然后在INSERT语句的VALUES子句中使用该变量。

目录示例

以下语句关联文件Image1.gif 和 image2.gif 记录的key_value分别为21和22。' IMG '是一个DIRECTORY对象，表示存储Image1.gif和image2.gif的物理目录。

您可能需要设置类似于以下内容的数据结构才能使用某些示例：

CREATE TABLE Lob_table ( Key_value NUMBER NOT NULL, F_lob BFILE)    插入Lob_table值（21，BFILENAME（'IMG'，'Image1.gif'））;插入Lob_table值（22，BFILENAME（'IMG'，'image2.gif'））;

以下UPDATE语句将目标文件更改为 image3.gif为key_value为22的行。 

   UPDATE Lob_table SET f_lob = BFILENAME（'IMG'，'image3.gif'）WHERE Key_value = 22;

警告：

数据库不会扩展DIRECTORY对象中指定的环境变量或BFILE定位器的文件名。例如，指定：

BFILENAME（'WORK_DIR'，'$ MY_FILE'）

其中MY_FILE是操作系统中定义的环境变量，无效。

BFILENAME和初始化

BFILENAME是一个内置函数，用于初始化BFILE列以指向外部文件。

一旦使用SQL DML将物理文件与记录相关联，就可以使用PL / SQL DBMS_LOB包和OCI执行对BFILE后续读取操作。但是，这些文件在通过BFILE访问时是只读的，因此无法通过BFILE更新或删除它们。

作为结果参考为主对于BFILEs语义，可以在同一记录中有多个BFILE列，或者在同一个文件中有不同的记录。例如，以下UPDATE语句将行的BFILE列设置为 key_value在= 21 lob_table 指向与key_value = 22的行相同的文件。 

UPDATE lob_table SET f_lob =（SELECT f_lob FROM lob_table WHERE key_value = 22）WHERE key_value = 21;

在初始化方面考虑BFILENAME - 它可以初始化以下值：

  • BFILE列

  • 在PL / SQL模块中声明的BFILE （自动）变量

BFILE数据类型的特征

使用BFILE数据类型具有以下优点：

  • 如果您对特定BFILE是临时的并且在您正在使用的模块中是有限的，那么您可以在变量上使用与BFILE相关的API，而无需将其与数据库中的列相关联。

  • 因为您不必在服务器端表中创建BFILE列，初始化此列值，然后使用SELECT检索此列值，您将保存到服务器的往返。

BFILENAME的OCI对应物是OCILobFileSetName() ，它可以以类似的方式使用。

目录名称规范

您必须具有CREATE ANY DIRECTORY系统特权才能创建目录。

路径名称不能包含两个点（“..”）。DIRECTORY对象的命名约定与表和索引的命名约定相同。也就是说，普通标识符以大写形式解释，但是分隔标识符按原样解释。例如，以下声明：

创建或更换目录scott_dir AS'/ usr / home / scott';

创建或重新定义名称为“ SCOTT_DIR ”（大写）的DIRECTORY对象。但是，如果DIRECTORY名称使用了分隔标识符，则如下面的语句所示

CREATE DIRECTORY“Mary_Dir”AS'/ usr / home / mary';

然后目录对象名称是' Mary_Dir '。调用BFILENAME时使用' SCOTT_DIR '和' Mary_Dir '。例如：

BFILENAME（'SCOTT_DIR'，'afile'）BFILENAME（'Mary_Dir'，'afile'）
在Windows平台上

在Windows平台上，目录名称不区分大小写。因此以下两个语句引用相同的目录：

CREATE DIRECTORY“big_cap_dir”AS“g：\ data \ source”; CREATE DIRECTORY“small_cap_dir”AS“G：\ DATA \ SOURCE”;

BFILE安全

BEFILE安全性涉及BFILE安全模型和相关的SQL语句。与BFILE安全性相关的主要SQL语句是：

  • SQL DDL： CREATE和REPLACE或ALTER DIRECTORY对象

  • SQL DML： GRANT和REVOKE DIRECTORY对象的READ系统和对象权限

所有权和特权

DIRECTORY对象是系统拥有的对象。

有关系统拥有的对象的更多信息，请参见Oracle数据库SQL语言参考 。Oracle数据库支持两个新的系统特权，这些特权仅授予DBA：

  • CREATE ANY DIRECTORY ：用于创建或更改DIRECTORY对象创建

  • DROP ANY DIRECTORY ：删除DIRECTORY对象

读取DIRECTORY对象的权限

DIRECTORY对象的READ权限使您可以读取位于该目录下的文件。DIRECTORY对象的创建者自动获得READ权限。

如果您已使用GRANT选项授予READ权限，则可以将此权限授予其他用户/角色，并将其添加到您的权限域。

注意：

READ权限仅在DIRECTORY 对象上定义，而不是在单个文件上定义。因此，无法为同一目录中的文件分配不同的权限。

它表示的物理目录可能具有或不具有Oracle Server进程的相应操作系统特权（在本例中为读取 ）。

DBA有责任确保以下内容：

  • 物理目录存在

  • 已对文件，目录和指向它的路径启用了对Oracle Server进程的读取权限

  • 在数据库用户访问文件的整个过程中，该目录仍然可用，并且读取权限保持启用状态

该权限只是意味着就Oracle服务器而言，您可以从目录中的文件中读取。在实际文件操作时，PL / SQL DBMS_LOB包和OCI API检查并强制执行这些权限。

警告：

因为CREATE ANY DIRECTORY和DROP ANY DIRECTORY权限可能会将服务器文件系统暴露给所有数据库用户，所以DBA应该谨慎地将这些权限授予普通数据库用户以防止安全漏洞。

目录使用指南

DIRECTORY功能的主要目标是为DBA提供一种简单，灵活，非侵入性但安全的机制，以管理对服务器文件系统中大文件的访问。但要实现此目标，DBA在使用DIRECTORY对象时遵循这些准则非常重要：

  • 不要将DIRECTORY对象映射到数据文件目录。不应将DIRECTORY对象映射到包含Oracle数据文件，控制文件，日志文件和其他系统文件的物理目录。篡改这些文件（意外或其他）可能会破坏数据库或服务器操作系统。

  • 只有DBA才应具有系统权限。应谨慎使用诸如CREATE ANY DIRECTORY （最初授予DBA）之类的系统特权，而不是不加区分地授予其他用户。在大多数情况下，只有数据库管理员才具有这些权限。

  • 授予DIRECTORY权限时请小心。DIRECTORY对象的权限应该小心地授予不同的用户。在向用户授予权限时，同样适用于使用WITH GRANT OPTION子句。

  • 数据库运行时，请勿删除或替换DIRECTORY对象。在数据库运行时，不应随意删除或替换DIRECTORY对象。如果发生这种情况，那么与此DIRECTORY对象关联的所有文件的所有会话的操作都将失败。此外，如果在成功关闭这些文件之前执行DROP或REPLACE命令，则在程序中丢失对这些文件的引用，并且在关闭会话之前不会释放与这些文件关联的系统资源。

    例如，PL / SQL用户唯一的办法就是运行一个调用DBMS_LOB的程序块。 FILECLOSEALL并重新启动文件操作，或完全退出其会话。因此，必须谨慎使用这些命令，最好是在维护停机期间。

  • 撤消用户对DIRECTORY对象的权限时要小心。使用REVOKE语句撤消用户对DIRECTORY对象的权限会导致来自用户会话的相关文件的所有后续操作失败。您必须重新获取关闭文件的权限，或者在会话中运行FILECLOSEALL并重新启动文件操作。

通常，使用DIRECTORY对象来管理文件访问是操作系统级别的系统管理工作的扩展。通过一些规划，可以将文件逻辑组织到具有Oracle进程的READ权限的合适目录中。

可以使用映射到这些物理目录的READ权限创建DIRECTORY对象，并授予对这些目录的访问权限的特定数据库用户。

共享服务器（多线程服务器）模式下的BFILE

数据库不支持共享服务器（多线程服务器）模式下的BFILE数据类型的会话迁移。这意味着在打开的BFILE实例上的操作可以在对共享服务器的调用结束之后持续存在。

在共享服务器会话中， BFILE操作绑定到一个共享服务器，它们无法从一个服务器迁移到另一个服务器。

外部LOB（BFILE）定位器

对于BFILE ，该值存储在服务器端操作系统文件中;换句话说，在数据库外部。引用该文件的BFILE定位器存储在该行中。

当BFILE表中的两行引用相同的文件时

如果在DBMS_LOB使用的BFILE定位器变量。 FILEOPEN （例如L1）被分配给另一个定位变量（例如L2），然后L1和L2都指向同一个文件。

这意味着带有BFILE列的表中的两行可以引用相同的文件或两个不同的文件 - 这是一个精明的开发人员可能会转向优势的事实，但这对于粗心的人来说可能是一个陷阱。

BFILE定位器变量

A BFILE定位器变量的运行方式与任何其他自动变量相同。关于文件操作，它像文件描述符一样操作，作为大多数传统编程语言的标准输入/输出库的一部分。

这意味着一旦定义并初始化BFILE定位器，并打开此定位器指向的文件，所有后续操作直到关闭此文件必须使用此定位器或此定位器的本地副本在同一程序块中完成。

BFILE指南

使用BFILE时请注意以下准则：

  • 在相同的嵌套级别打开和关闭同一程序块中的文件。BFILE定位器变量可以像任何标量一样用作其他过程，成员方法或外部函数标注的参数。但是，建议您在同一嵌套级别打开和关闭同一程序块中的文件。

  • 在将对象刷新到数据库之前设置BFILE值。如果对象包含BFILE ，则必须在将对象刷新到数据库之前设置BFILE值，从而插入新行。换句话说，您必须在OCILobFileSetName()之后和OCIObjectNew()之前OCIObjectFlush() 。

  • 在插入或更新BFILE之前指示DIRECTORY对象名称和文件名。插入或更新BFILE而不指示DIRECTORY对象名称和文件名是错误的。

    此规则也适用于在插入或更新语句中对BFILE使用OCI绑定变量的用户。在发出insert或update语句之前，必须使用DIRECTORY对象名和文件名初始化OCI绑定变量。

  • 在插入或更新之前初始化BFILE

    注意：

    OCISetAttr()不允许用户将BFILE定位符设置为NULL 。

  • 在使用SQL插入或更新带有BFILE的行之前，必须将BFILE初始化为以下之一：

    • NULL （如果使用OCI绑定变量则不可能）

    • DIRECTORY对象名称和文件名

  • 路径名称在其规范中的任何位置都不能包含两个点（“..”）。文件名不能以两个点开头。

关于从BFILE读取数据

您可以从BFILE读取数据。

也可以看看：

表11-1 ，列出了针对每个编程环境提供的BFILE和API的操作。

使用说明

使用此操作时请注意以下事项。

流媒体读取OCI

读取大量BFILE数据的最有效方法是通过OCILobRead2()启用流机制，并使用轮询或回调。为此，请使用offset参数指定读取的起始点，如下所示：

ub8 char_amt = 0; ub8 byte_amt = 0; ub4 offset = 1000; OCILobRead2（svchp，errhp，locp，＆byte_amt，＆char_amt，offset，bufp，bufl，OCI_ONE_PIECE，0,0,0,0）;

使用轮询模式时 ，请务必在每次OCILobRead2()调用后查看byte_amt参数的值，以查看缓冲区中读取了多少字节，因为缓冲区可能未完全填满。

使用回调时 ，输入到回调的lenp参数指示缓冲区中填充的字节数。请务必在回调处理期间检查lenp参数，因为整个缓冲区可能没有填充数据。

金额参数

  • 调用DBMS_LOB.READ ，amount参数可以大于数据的大小;但是，amount参数应小于或等于缓冲区的大小。在PL / SQL中，缓冲区大小限制为32K。

  • 调用OCILobRead2() ，可以为byte_amt参数传递值0（零）以读取到BFILE的末尾。

使用FILECLOSEALL关闭所有打开的BFILE

你可以关闭所有打开的BFILE 。

您负责在程序终止之前关闭所有BFILE实例。例如，您必须在PL / SQL块或OCI程序终止之前关闭任何打开的BFILE实例。

即使在发生应用程序异常或意外终止的情况下，您也必须关闭打开的BFILE实例。在这些情况下，如果未关闭BFILE实例，则数据库仍将其视为已打开。确保您的异常处理策略不允许BFILE实例在这些情况下保持打开状态。
