1 Oracle Database In-Memory简介

Oracle数据库内存（Database In-Memory）是一套功能，可以极大地提高实时分析和混合工作负载的性能。内存中列存储（IM列存储）是Database In-Memory的关键功能。

注意：

Database In-Memory功能需要Oracle Database In-Memory选项。有关不同版本和服务支持哪些功能的详细信息，请参阅“ Oracle数据库许可信息用户手册 ”。

本章包含以下主题：

  • 分析应用面临的挑战
    传统上，为分析查询获得良好性能意味着满足几个要求。
  • 单格式方法
    传统上，关系数据库以行或列格式存储数据。内存和磁盘以相同的格式存储数据。
  • Oracle数据库内存中解决方案
    Oracle数据库内存（Database In-Memory）功能集包括内存中列存储（IM列存储），高级查询优化和可用性解决方案。
  • 数据库内存的先决条件
    所有Database In-Memory功能都需要Oracle Database In-Memory选项。IM列存储不需要特殊硬件。
  • 数据库内存的主要任务
    对于受益于IM列存储的查询，唯一需要的任务是调整IM列存储大小，并指定用于填充的对象。查询优化和可用性功能需要其他配置。
  • IM列存储的工具
    管理IM列存储或其他Database In-Memory功能不需要特殊工具或实用程序。完全支持SQL * Plus，SQL Developer和Oracle Enterprise Manager（企业管理器）等管理工具。

传统上，为分析查询获得良好性能意味着满足几个要求。

在典型的数据仓库或混合使用数据库中，要求包括以下内容：

  • 您必须了解用户访问模式。

  • 您必须提供良好的性能，这通常需要创建索引，物化视图和OLAP多维数据集。

例如，如果为表（1个主键和2个外键索引）创建1到3个索引以为OLTP应用程序提供良好性能，则可能需要创建其他索引以为分析查询提供良好性能。

1.2单格式方法

传统上，关系数据库以行或列格式存储数据。内存和磁盘以相同的格式存储数据。

Oracle数据库在数据块中连续存储行。例如，在具有三行的表中，Oracle数据块存储第一行，然后存储第二行，然后存储第三行。每行包含该行的所有列值。以行格式存储的数据针对事务处理进行了优化。例如，更新少量行中的所有列可能仅修改少量块。

为了解决与分析查询相关的问题，一些数据库供应商引入了列式格式。列式数据库存储选定的列 - 而不是行 - 连续。例如，在大型销售表中，销售ID位于一列中，销售区域位于不同的列中。

分析工作负载在扫描时访问几列，但扫描整个数据集。因此，柱状格式对分析来说效率最高。由于列是单独存储的，因此分析查询只能访问所需的列，并避免读取不必要的数据。例如，按地区划分的销售总额报告可以快速处理多行，而只访问几列。

数据库供应商通常会强制客户在列式和基于行的格式之间进行选择 。例如，如果数据格式是列式的，则数据库以列式格式在内存和磁盘上存储数据。获得一种格式的优势意味着失去备用格式的优点。应用程序既可以实现快速分析，也可以实现快速交易，通过以单一格式存储数据无法解决混合使用数据库的性能问题。

1.3.1.1 IM列存储

IM列存储以压缩的列式格式维护表，分区和单个列的副本，该格式针对快速扫描进行了优化。

视频：

IM列存储位于内存区域中 ，该区域是系统全局区域（SGA）的可选部分。IM列存储不会替换基于行的存储或数据库缓冲区缓存，而是对其进行补充 。该数据库使数据能够以行为基础和列式格式存储在内存中，从而提供两全其美的优势。IM列存储提供独立于磁盘格式的表数据的附加事务一致性副本。

  • 列（非虚拟或虚拟）

  • 表（内部或外部），物化视图或分区

    注意：

  • 表空间

如果在表空间级别指定了INMEMORY属性，则默认情况下会为IM列存储启用表空间中的所有新表和实例化视图。在Database In-Memory的上下文中， 填充是将磁盘上基于行的数据自动转换为IM列存储中的列数据。您可以在IM列存储中配置数据库对象列的全部或子集以用于填充。同样，对于分区表或物化视图，您可以配置全部或部分分区以进行填充。

例如，您可以将sh模式中的三个表配置为IM列存储： customers ， products和sales 。IM列存储按列而不是按行存储每个表的数据，并将每个列分成单独的行子集。称为内存中压缩单元（IMCU）的特殊容器存储表段中行的子集的所有列。

1.3.1.2高级查询优化

Database In-Memory包括针对分析查询的多种性能优化。

优化包括：

  • 表达式是一个或多个值，运算符和SQL函数（仅限DETERMINISTIC ）的组合，可以解析为某个值。默认情况下， 内存中表达式 （IM表达式）优化使DBMS_INMEMORY_ADMIN.IME_CAPTURE_EXPRESSIONS过程能够识别并填充IM列存储中的“热”表达式。IM表达式具体化为隐藏虚拟列，但其访问方式与非虚拟列相同。

  • 连接组是用户定义的对象，它指定经常用于连接一组表的一组列。在某些查询中，连接组使数据库能够消除解压缩和散列列值的性能开销。

  • 对于将小维度表连接到大型事实表的聚合查询， 内存中聚合 （IM聚合）使用VECTOR GROUP BY操作来增强性能。此优化在事实表的扫描期间而不是之后聚合数据。

  • 在IM列存储中， 重新填充是在IMCU中的数据被显着修改之后自动更新IMCU。如果IMCU具有陈旧的条目但不满足陈旧性阈值，则后台进程可能会引发涓流重新填充 ，这是IM列存储的逐渐重新填充。

1.3.1.3高可用性支持

可用性是指按需访问应用程序，服务或功能的程度。

Database In-Memory支持以下可用性功能：

  • 内存中快速启动 （IM FastStart）减少了数据库实例重新启动时将数据填充到IM列存储中的时间。IM FastStart通过以压缩的列式格式定期保存磁盘上IM列存储中当前填充的数据副本来实现此目的。

  • Oracle Real Application Clusters（Oracle RAC）环境中的每个节点都有自己的IM列存储。可以在每个节点上填充完全不同的对象，或者在群集中的所有IM列存储中分布较大的对象。在工程系统中，也可以在每个节点上的IM列存储中显示相同的对象。

  • 从Oracle Database 12c第2版（12.2）开始，Active Data Guard环境中的备用数据库支持IM列存储。

1.3.2.1改进数据扫描的性能

柱状格式为扫描大量数据提供了快速吞吐量。

IM列存储使您能够实时分析数据，使您能够探索不同的可能性并执行迭代。具体而言，IM列存储可以显着提高执行以下操作的查询的性能：

  • 扫描许多行并应用使用< ， > ， =和IN等运算符的过滤器

  • 从表或具有许多列的物化视图中选择几列，例如访问100列中的5列的查询

  • 使用SQL运算符选择LOB列

视频：

对于大多数数字和短字符串数据类型，列式格式使用固定宽度列。此优化支持快速矢量处理，使数据库能够更快地回答查询。

由于以下原因，IM列存储的扫描速度比基于行的数据的扫描速度快：

  • 消除缓冲区缓存开销

    IM列存储以纯内存中的列式格式存储数据。数据不会保留在数据文件中（或生成重做），因此数据库可以避免将数据从磁盘读入缓冲区缓存的开销。

  • 数据修剪

    数据库仅扫描查询所需的列，而不是整行数据。此外，数据库使用存储索引和内部字典来仅读取特定查询所需的IMCU。例如，如果查询请求商店ID小于8的商店的所有销售，则数据库可以使用IMCU修剪来消除不包含此值的IMCU 。

  • 压缩

    传统上，压缩的目标是节省空间。在IM列存储中，压缩的目标是加速扫描。数据库使用允许对压缩格式应用WHERE子句谓词的算法自动压缩列数据。根据所应用的压缩类型，Oracle数据库可以以压缩格式扫描数据， 而无需先对其进行解压缩。因此，数据库必须在IM列存储中扫描的数据量小于数据库缓冲区高速缓存中的相应卷。

  • 矢量处理

    每个CPU核心都扫描本地内存中的列。要将数据作为数组（集合）处理，扫描使用SIMD向量指令。例如，查询可以读取单个CPU指令中的一组值，而不是逐个读取值。CPU核心的矢量扫描比行扫描快几个数量级。

例如，假设用户执行以下即席查询：

SELECT cust_id，time_id，channel_id FROM sales WHERE prod_id BETWEEN 14和29

使用缓冲区缓存时，数据库通常会扫描索引以查找产品ID，使用rowid将磁盘中的行提取到缓冲区缓存中，然后丢弃不需要的列值。在缓冲区高速缓存中以行格式扫描数据需要许多CPU指令，并且可能导致CPU效率不理想。

使用IM列存储时，数据库只能扫描请求的sales列，完全避免使用磁盘。以列式格式扫描数据只需要CPU的必要列，从而提高效率。每个CPU内核使用SIMD向量指令扫描本地内存列。

1.3.2.2提高连接性能

Bloom过滤器是一种低内存数据结构，用于测试集合中的成员资格。IM列存储利用Bloom过滤器来提高连接的性能。

Bloom过滤器通过将小维度表上的谓词转换为大型事实表上的过滤器来加速联接。当使用一个大事实表执行多个维度的连接时，此优化非常有用。事实表上的维度键具有许多重复值。扫描性能和重复值优化可将连接速度提高数个数量级。

1.3.2.3改进聚合性能

分析的一个重要方面是通过聚合数据来确定模式和趋势。当数据存储在IM列存储中时，聚合和复杂的SQL查询运行得更快。

在Oracle数据库中，聚合通常涉及GROUP BY子句。传统上，数据库使用SORT和HASH运算符。从Oracle Database 12c第1版（12.1）开始，数据库提供了VECTOR GROUP BY转换，以实现高效的内存中基于阵列的聚合。

在事实表扫描期间，数据库将聚合值累积到内存数组中，并使用有效的算法来执行聚合。基于主键和外键关系的连接针对星型模式和雪花模式进行了优化。

1.3.3改进混合工作负载的性能

虽然OLTP应用程序无法从访问IM列存储中的数据中受益，但双内存格式可间接提高OLTP性能。

当所有数据都存储在行中时，提高分析查询性能需要创建访问结构。标准方法是创建分析索引，物化视图和OLAP多维数据集。例如，一个表可能需要3个索引来提高OLTP应用程序的性能（1个主键和2个外键索引）和10-20个附加索引，以提高分析查询的性能。虽然此技术可以提高分析查询性能，但它会降低OLTP性能。在表中插入行需要修改表上的所有索引。随着索引数量的增加，插入速度降低。

将数据填充到IM列存储中时，可以删除分析访问结构。此技术减少了存储空间和处理开销，因为需要更少的索引，物化视图和OLAP多维数据集。例如，插入会导致修改1-3个索引而不是11-23个索引。

虽然IM列存储可以显着提高业务应用程序，临时分析查询和数据仓库工作负载中的分析查询的性能，但使用索引查找执行短事务的纯OLTP数据库受益更少。IM列存储不会提高以下类型查询的性能：

  • 具有复杂谓词的查询

  • 一个选择多列的查询

  • 返回许多行的查询

1.3.4 Exadata Flash Cache的内存支持

并非所有标记为INMEMORY对象INMEMORY可以同时适合DRAM内存。如果使用Oracle Exadata存储服务器软件，则Exadata智能闪存缓存可用作补充内存。

启用IM列存储后，Exadata智能闪存缓存会自动将数据重新格式化为内存中列式格式。在以前的Exadata版本中，只有混合列压缩数据符合IM列式格式的闪存存储。从Oracle Exadata系统软件18 c （18.1.0）开始，对压缩（包括OLTP压缩）和未压缩表进行重新格式化。

使用此格式，Smart Scan支持大多数Database In-Memory性能增强，包括连接和聚合。此外，将未压缩和OLTP压缩的数据块重新格式化为IM柱状格式可以显着减少所需的闪存量。

Exadata智能闪存缓存在以下阶段转换数据：

  1. Oracle Exadata以传统的列式格式缓存符合条件的扫描数据，以便数据立即可用。此格式是列式的，但它与IM列存储使用的格式不同。

  2. 在后台，Oracle Exadata将数据重新格式化为较低优先级的纯IM列存储格式。后台写入可防止干扰主要工作负载。

如果数据库未运行OLTP工作负载，则数据仓库工作负载可能会占用100％的闪存缓存。但是，OLTP工作负载将数据仓库工作负载限制为不超过闪存缓存的50％。此优化可确保分析扫描不会牺牲OLTP工作负载性能。

默认情况下，Exadata智能闪存缓存使用MEMCOMPRESS FOR CAPACITY LOW级别压缩数据。若要更改压缩级别或完全禁用列式格式，请使用ALTER TABLE ...NO CELLMEMORY声明。

1.3.5高可用性支持

IM列存储完全集成到Oracle数据库中。支持所有高可用性功能。

列式格式不会更改Oracle数据库磁盘存储格式。因此，缓冲区缓存修改和重做日志记录功能相同。完全支持RMAN，Oracle Data Guard和Oracle ASM等功能。

在Oracle Real Application Clusters（Oracle RAC）环境中，默认情况下每个节点都有自己的IM列存储。根据您的要求，您可以以不同方式填充对象：

  • 每个节点上都填充了不同的表。例如， sales事实表位于一个节点上，而products维度表位于不同的节点上。

  • 单个表分布在不同节点之间。例如，相同散列分区表的不同分区在不同节点上，或者单个非分区表的不同rowid范围在不同节点上。

  • 某些对象出现在每个节点上的IM列存储中。例如，您可以在每个节点中填充products维度表，但是跨不同节点分发sales事实表的分区。

1.3.6易于采用

Database In-Memory易于实现，无需更改应用程序。

数据库内存采用的关键方面包括：

  • 易于部署

    无需用户管理的数据迁移。数据库以行格式在磁盘上存储数据，并在填充IM列存储时自动将行数据转换为列格式。

  • 兼容现有应用程序

    无需更改应用程序。优化程序自动利用列式格式。如果您的应用程序连接到数据库并发出SQL，那么它可以从Database In-Memory功能中受益。

  • 完全SQL兼容性

    Database In-Memory对SQL没有限制。无论是使用Oracle分析函数还是自定义PL / SQL代码，分析查询都会受益。

  • 便于使用

    无需复杂的设置。INMEMORY_SIZE初始化参数指定保留供IM列存储使用的内存量。DDL语句中的INMEMORY子句指定要填充到IM列存储中的对象或列。通过配置IM列存储，您可以立即提高现有分析工作负载和即席查询的性能。

1.6.1 In-Memory Advisor

In-Memory Advisor是一个可下载的PL / SQL包，用于分析数据库中的分析处理工作负载。

In-Memory Advisor根据SQL计划基数，活动会话历史记录（ASH），并行查询使用情况和其他统计信息将分析处理与其他数据库活动区分开来。In-Memory Advisor根据统计信息和启发式压缩因子估计IM列存储中对象的大小。

顾问根据以下内容估算分析处理性能改进因素：

  • 消除等待事件，例如用户I / O等待，群集传输等待和缓冲区高速缓存锁存等待

  • 查询与特定压缩类型相关的处理优势

  • 针对特定压缩类型的解压缩成本启发式

  • SQL计划基数，结果集中的列数等

输出是一个报告，它建议IM列存储的大小以及将从内存中填充的对象列表。顾问程序还会生成一个SQL * Plus脚本，该脚本使用INMEMORY子句更改建议的对象。

内存中的Advisor 不包含在存储的PL / SQL包中。您必须从Oracle Support下载该软件包。

1.6.4 Oracle Data Pump和IM列存储

您可以使用impdp命令的TRANSFORM=INMEMORY:y选项导入为IM列存储启用的数据库对象。

使用此选项，Oracle Data Pump将为具有一个对象的所有对象保留IM列存储子句。如果指定了TRANSFORM=INMEMORY:n选项，Data Pump将从具有一个的所有对象中删除IM列存储子句。

您还可以使用TRANSFORM=INMEMORY_CLAUSE: string选项在导入期间覆盖转储文件中数据库对象的IM列存储子句。例如，您可以使用此选项更改导入的数据库对象的IM列存储压缩。
