1企业用户安全介绍

企业用户安全性是Oracle Database 12 c第1版（12.1）企业版的重要组成部分。它使您能够解决大量企业数据库用户的管理和安全挑战。企业用户是在目录中定义的那些用户。他们的身份在整个企业中保持不变。企业用户安全性依赖于Oracle身份管理基础架构，后者又使用符合LDAP的目录服务来集中存储和管理用户。

用户管理的挑战

管理员必须使整个企业的用户信息保持最新且安全。随着应用程序和用户数量的增加，此任务变得更加困难。通常，每个用户在不同的数据库上有多个帐户，这意味着每个用户必须记住多个密码。结果是用户记住的密码太多，管理员有太多帐户无法有效管理。

有数千名用户访问数据库帐户，用户管理需要大量资源。多个应用程序使用的公共信息（例如用户名，电话号码以及系统角色和权限）通常在整个企业中分散。这些数据越来越多，变得多余，不一致，难以管理。

除了用户和帐户管理问题之外，这些条件也会产生安全问题。例如，每当用户离开公司或更换工作时，该用户的权限应在同一天更改，以防止他们被滥用。但是，大型企业通常会在多个数据库中分布许多用户帐户，并且管理员可能无法进行此类及时更改。

同样，如果您的用户拥有太多密码，他们可能会将其写下来，以便其他人轻松复制。他们可能会选择易于记忆的密码，使其他人容易猜到密码，并为多个应用程序使用相同的密码，从而可能会因密码泄露而产生更广泛的后果。所有这些用户跟踪多个密码的努力都可能危及企业安全。

企业用户安全：大图

企业用户安全性依赖于Oracle Internet Directory（一种LDAP兼容的目录服务）提供的身份管理服务，解决了用户，管理和安全方面的挑战。身份管理是在组织中管理网络实体的完整安全生命周期的过程。它通常是指组织的应用程序用户的管理，其中安全生命周期中的步骤包括帐户创建，暂停，权限修改和帐户删除。

Oracle Internet Directory如何实现身份管理

Oracle Internet Directory使用身份管理领域的概念来组织目录信息树（DIT）中的信息，这是一种由目录对象条目组成的分层树状结构。在目录中，有关对象的每个信息集合称为条目。该对象可以是人，但也可以是关于联网设备的信息，例如配置信息。要在DIT中命名和标识目录对象的位置，每个条目都会分配一个唯一的可分辨名称（DN）。条目的DN由条目本身及其父条目组成，它们以升序连接，从条目本身到DIT中的根（顶部）条目。

关于身份管理领域

身份管理领域是目录条目的子树，所有这些都由相同的管理策略管理。例如，企业中有权访问Intranet的所有员工可能属于一个领域，而访问企业的公共应用程序的所有外部用户可能属于另一个领域。使用不同的领域使企业能够隔离用户群并在每个领域中实施不同的管理策略，例如密码策略或命名策略。用于标识登录标识的默认昵称属性是uid，它在每个身份管理领域中设置

关于身份管理领域特定的Oracle上下文

每个身份管理领域都有一个特定于领域的Oracle Context（领域Oracle Context），用于存储该领域的Oracle产品信息。领域Oracle Context存储应用程序数据，用户的命名和定位方式，用户必须如何进行身份验证，组位置和权限分配，这些都特定于领域Oracle Context所在的特定身份管理领域。

企业用户与数据库用户的比较

数据库用户通常使用CREATE USER语句在数据库中定义，如下所示：

CREATE USER用户名IDENTIFIED BY密码;

这将创建一个与用户模式关联的数据库用户，该用户可以访问数据库并使用CONNECT命令使用密码进行身份验证，如下所示：

CONNECT username @ database_service_name输入密码：

必须在他们需要访问的每个数据库中创建数据库用户，并且他们可以为每个数据库选择不同的密码。数据库用户权限由每个数据库中的本地角色控制。

相反，企业用户在LDAP兼容目录（例如Oracle Internet Directory）中集中供应和管理，以进行数据库访问。企业用户在名为专有名称（DN）的目录中具有唯一标识。当企业用户登录到数据库时，数据库使用其DN对这些用户进行身份验证。

企业用户在数据库中定义为全局用户。全局用户可以拥有自己的模式，也可以在他们访问的数据库中共享全局模式。您可以通过两种不同的方式在CREATE USER语句中使用GLOBALLY子句来创建企业用户。

您可以使用AS子句指定用户的目录DN，如以下语句所示：

CREATE USER username IDENTIFIED GLOBALLY AS'<目录用户条目的DN>';

在这种情况下，它们具有专门为它们分配的模式。

或者，您可以使用AS子句指定空字符串，如以下语句所示：

CREATE USER用户名IDENTIFIED GLOBALLY AS'';

使用AS子句指定空字符串时，该目录将经过身份验证的用户映射到相应的数据库架构。在这种情况下，可以根据设置并存储在Oracle Internet Directory中的映射信息将多个用户映射到共享模式。

注意：

您还可以使用以下语法创建共享架构：

CREATE USER用户名 IDENTIFIED GLOBALLY;

这与指定空字符串相同。

当企业用户通过SSL连接到数据库时，他们不使用密码。相反，他们使用以下CONNECT命令，该命令根据客户端的sqlnet.ora文件中的信息查找钱包位置：

CONNECT / @ database_service_name

经过密码验证的企业用户使用相同的CONNECT语句作为常规数据库用户连接到数据库。例如，经过密码验证的企业用户使用以下语法连接到数据库：

CONNECT username @ database_service_name输入密码：

当数据库从企业用户接收连接请求时，数据库引用用于用户身份验证和授权（角色）信息的目录。

关于企业用户架构

如果企业安全管理员将它们映射到共享模式，企业用户可以保留其各自的数据库模式（独占模式）或共享模式。

本节包含以下主题：

  • 私人或独家架构

  • 共享架构

私人或独家架构

如果用户希望将其各自的模式保留在他们访问的数据库中，则必须完成以下任务：

  • 在目录中创建企业用户

  • 为他们访问的每个数据库中的每个用户创建一个全局用户架构

在每个访问的数据库上为每个企业用户创建单独的帐户会产生很大的开销。相反，创建在每个数据库中访问单个通用共享模式的企业用户可以提高企业用户解决方案的效率。

共享架构

要获得企业用户解决方案的真正好处，您可以为企业用户使用共享架构。对于此策略，请完成以下任务：

  • 在目录中创建企业用户

  • 在每个数据库中创建一个共享模式

  • 在Oracle Internet Directory中创建单个共享模式映射

将企业用户映射到他们访问的每个数据库上的通用共享模式可以大大减少为每个企业用户创建单独模式的开销。

共享模式企业用户可以映射到他们访问的所有数据库上的通用共享模式，也可以在某些数据库上使用独占模式，在其他数据库上使用共享模式。共享模式映射存储在目录中。

企业用户如何使用数据库链接访问数据库资源

数据库链接是存储在本地数据库或网络定义中的网络对象，用于标识远程数据库，该数据库的通信路径以及可选的用户名和密码。定义后，数据库链接用于访问远程数据库。Oracle数据库支持连接的用户链接，固定用户链接和当前用户链接。

企业用户可以使用以下三种类型的数据库链接：

  • 连接的用户链接由在远程服务器上具有帐户的本地用户访问。

  • 固定用户链接包含用户名和密码作为链接定义的一部分。

  • 当前用户数据库链接允许企业用户访问远程数据库上的对象，而无需在链接执行期间传递认证信息，或者在链接定义中存储认证信息。

    它们需要SSL用于数据库网络连接，这意味着必须为数据库获取和维护公钥基础结构（PKI）凭据。当前用户数据库链接可用于仅作为企业用户连接到远程数据库。

关于企业用户安全目录条目

在目录中，有关对象的信息集合称为条目。对于企业用户安全性，用户，角色和数据库等元素是目录对象，有关这些对象的信息作为条目存储在目录中。

目录中的每个条目都由DN唯一标识。 DN会准确地告诉您条目在目录条目层次结构中的位置，通常称为目录信息树（DIT） 。

注意：

对于Oracle数据库10g及更高版本，数据库必须在Oracle Internet Directory的完整身份管理领域中注册。

企业用户

企业用户是在目录中定义和管理的用户 。每个企业用户在整个企业中都具有唯一标识。企业用户条目可以驻留在身份管理领域内的任何位置，但在领域Oracle Context中除外。

注意：

在9.0.4或更高版本的Oracle Internet Directory中创建企业用户时，请使用9.0.4或更高版本的Oracle Internet Directory附带的工具，例如Delegated Administration System（DAS）。即使您的数据库是9 i或9 i R2，也不要使用9 i或9 i R2 Enterprise Security Manager GUI工具在9.0.4或更高版本的Oracle Internet Directory中创建用户。

仅使用Oracle Application Server 10g附带的基于DAS的工具（如Oracle Internet Directory自助服务控制台）在身份管理领域中创建企业用户。

以下各节中描述的条目只能驻留在领域Oracle Context中 。

企业角色

企业角色是一个目录对象，其作用类似于容纳一个或多个数据库全局角色 的容器。每个全局角色都在特定数据库中定义，在该数据库中为其分配权限，然后使用企业角色在目录中对其进行管理。可以为企业用户分配企业角色，该角色确定他们对数据库的访问权限。图1-3显示了OracleDefaultDomain下名为Manager的企业角色示例。

目录中的企业域子树由三种类型的条目组成：企业角色条目，用户模式映射以及该域的企业域管理员组。企业域用于管理适用于多个数据库的信息。企业域中包含的所有用户架构映射条目都适用于域中的所有数据库。如果需要将不同的用户架构映射应用于各个数据库，请使用数据库服务器条目，这将在下一节中讨论。

企业角色适用于域中的特定数据库，如上一节中所述。企业角色，域级映射和域管理员组都使用Oracle Enterprise Manager进行管理。

密码策略

密码策略是一组适用于身份管理领域中所有用户密码的规则。密码策略包括密码复杂性，最小密码长度等的设置。它们还包括帐户锁定和密码到期设置。

Oracle Internet Directory中为每个身份管理领域定义了密码策略条目。Oracle Internet Directory中的密码策略是Oracle Database for Enterprise User Security可以使用的标准Oracle Internet Directory条目。

Oracle Internet Directory确保所有企业用户密码都符合域的密码策略条目中指定的规则。在验证企业用户时，数据库与Oracle Internet Directory通信。它要求Oracle Internet Directory报告任何密码策略违规。如果数据库从Oracle Internet Directory获得策略违规响应，则会向用户闪烁相应的警告或错误消息。

数据库报告以下事件：

  • 当用户密码即将过期时，它会闪烁警告，并显示用户更改密码的剩余天数。

  • 密码过期时会闪烁警告，并通知用户剩余的宽限登录次数。

  • 当用户密码过期且用户没有任何宽限登录时，它会显示错误。

  • 由于重复登录尝试失败而导致用户帐户被锁定时，会显示错误。

    注意：

    对于Enterprise SYSDBA用户，启用失败的登录计数，并在数据库启动或关闭时更新。

  • 如果管理员已禁用用户帐户，则会显示错误。

  • 如果用户帐户处于非活动状态，则会显示错误。

企业用户登录尝试访问数据库，更新Oracle Internet Directory中的用户帐户状态。例如，根据目录的密码策略，连续失败的数据库登录尝试导致帐户被锁定在目录中。

企业用户安全中使用的共享模式概述

用户不一定需要在每个数据库中设置个人帐户或模式。或者，他们可以连接到 共享模式，并被授予对与目标应用程序关联的对象的访问权限。例如，假设用户Tom，Dick和Harriet需要访问Finance数据库上的Payroll应用程序。它们不需要在数据库中创建唯一对象，因此不需要自己的模式，但它们确实需要访问Payroll模式中的对象。

Oracle数据库支持将存储在企业目录中的多个用户映射到单个数据库上的共享模式。用户与模式的分离通过减少数据库上的用户帐户数量来降低管理成本。这意味着除了在目录中创建用户之外，您不需要为每个用户（用户架构）创建帐户。相反，您可以在企业目录中创建用户，并将该用户映射到共享模式。其他企业用户也可以映射到该架构。

例如，如果Tom，Dick和Harriet都访问Sales和Finance数据库，则无需为每个数据库上的每个用户创建一个帐户。相反，您可以在每个数据库上创建一个共享模式，例如GUEST ，这三个用户都可以访问。然后，可以使用企业角色为这三个用户授予对Sales或Finance数据库中对象的单独访问权限。典型环境最多可以将5,000个企业用户映射到一个共享模式，并且可以为每个用户分配一组企业角色。

Oracle建议您创建一个单独的共享模式，该模式不包含用作入口点的对象。然后，通过企业角色授予对其他模式中的应用程序对象的访问权限。否则，可能会无意中或恶意删除或更改应用程序对象。

总之，共享模式具有以下优点：

  • 共享模式消除了为每个企业用户在每个数据库上建立专用数据库模式的需要。

  • 每个企业用户都可以映射到用户需要访问的每个数据库上的共享模式。用户连接到数据库时，用户连接到共享模式。

  • 共享模式降低了管理企业用户的成本。

如何为企业用户配置共享架构

要配置共享模式，本地数据库管理员（DBA）必须在数据库中至少创建一个数据库模式。企业用户可以映射到此架构。

在以下示例中，管理员创建共享架构并将用户映射到它：

  1. 管理员在HR数据库上创建名为EMPLOYEE的全局共享模式和全局角色HRMANAGER 。

  2. 管理员使用Oracle Internet Directory自助服务控制台和Oracle Enterprise Manager在目录中创建和管理企业用户和角色。例如，管理员创建企业用户Harriet和名为MANAGER的企业角色。然后，管理员将HRMANAGER的HR数据库全局角色HRMANAGER给企业角色MANAGER 。

  3. 管理员将企业角色分配给目录中的企业用户。例如，管理员将企业角色MANAGER分配给Harriet。

  4. 管理员使用Oracle Enterprise Manager将目录中的用户Harriet映射到HR数据库上的共享模式EMPLOYEE 。

当Harriet连接到HR数据库时，她会自动连接到EMPLOYEE模式，并被赋予HRMANAGER的全局角色。可以将多个企业用户映射到同一共享架构。例如，企业安全性管理员可以创建另一个企业用户Scott并将Scott映射到EMPLOYEE模式。从那时起，Harriet和Scott在连接到HR数据库时自动使用EMPLOYEE模式，但每个模式可以具有不同的角色并且可以单独审核。

企业用户如何映射到架构

全局模式（使用CREATE USER IDENTIFIED GLOBALLY AS ''创建的模式）可以由一个企业用户（独占模式）拥有，也可以由多个企业用户共享（共享模式）。单个企业用户与其独占架构之间的映射作为用户DN和架构名称之间的关联存储在数据库中。企业用户和共享模式之间的映射是通过一个或多个映射对象在目录中完成的。映射对象用于将用户的可分辨名称（DN）映射到用户将访问的数据库模式。您可以使用Oracle Enterprise Manager创建映射对象。此映射可以是以下之一：

  • 入门级（完整DN）映射

    此方法将单个目录用户的DN与数据库上的特定架构相关联。它为每个用户生成一个映射条目。

  • 子树级（部分DN）映射

    此方法允许多个企业用户共享其部分DN以访问相同的共享架构。如果已将多个企业用户分组到目录树中的某个公共根目录下，则此方法很有用。这些用户共享的子树可以映射到数据库上的共享模式。例如，您可以将工程部门的子树中的所有企业用户映射到错误数据库上的一个共享模式BUG_APP_USER。请注意，子树的根未映射到指定的架构。

    当企业用户连接到数据库时，数据库将从网络（在SSL的情况下）或从目录（在密码和经过Kerberos身份验证的企业用户的情况下）检索用户的DN。

    在确定将用户连接到哪个架构时，数据库使用用户DN和以下优先规则：

    1. 它在本地（在数据库中）查找独占模式。

    2. 如果它在本地找不到独占模式，则它会搜索目录。在目录中，它在数据库服务器条目下查找，首先查看入门级映射，然后查看子树级映射。

    3. 如果它在服务器条目下没有找到映射条目，则它在企业域条目下查找，首先查看入门级映射，然后查看子树级映射。

    4. 如果它在本地找不到独占模式或数据库中没有适用的映射条目，则数据库拒绝连接。否则，数据库将用户连接到适当的架构。

例如，假设Harriet正在尝试连接到HR数据库，但数据库找不到Harriet的独占模式（在数据库中）。在这种情况下，会发生以下事件：

  1. HR数据库在目录中查找与Harriet的DN的用户模式映射。该目录具有Harriet到共享模式EMPLOYEE的映射，并返回此模式。

  2. 数据库记录Harriet并将其连接到EMPLOYEE模式。

  3. 数据库从目录中检索此用户的此数据库的全局角色。

  4. 数据库还从其自己的表中检索与用户映射到的数据库模式关联的任何本地角色和特权。

  5. 数据库使用全局角色和本地角色来确定用户可以访问的信息。

继续此示例，假设企业角色MANAGER包含HR数据库上的全局角色ANALYST和Payroll数据库上的USER 。当具有企业角色MANAGER Harriet连接到HR数据库时，她在该数据库上使用模式EMPLOYEE 。

  • 她对HR数据库的权限由以下因素确定：

    • 全球角色ANALYST

    • 与HR数据库上的EMPLOYEE模式关联的任何本地角色和特权

  • 当Harriet连接到Payroll数据库时，她的权限由以下因素决定：

    • 全球角色USER

    • 与Payroll数据库上的EMPLOYEE模式关联的任何本地角色和特权

您可以通过向数据库模式授予角色和特权来向指定的用户组授予特权。除个人企业角色外，共享此类架构的每个用户都将获得这些本地角色和权限。但是，执行此操作时应谨慎，因为映射到此共享模式的每个用户都可以行使分配给它的权限。因此，Oracle不建议为共享模式授予角色和特权。

企业用户代理

有时，企业用户需要以另一个用户身份连接到数据库，暂时拥有目标用户的授权和权限。此功能对于中型工具或应用程序特别有用，这些工具或应用程序通常作为企业用户在各种数据库中运行，其身份在Oracle Internet Directory中作为条目建立。这样的应用程序可以在切换最终用户身份的同时维持单个数据库连接，从而依次提供每个授权用户名称的功能。

企业用户安全11g第1版（11.1）通过引入单会话模型来增强代理机制的效率。两会话代理模型需要为代理用户和目标用户维护单独的会话。在新模型中，目标用户的安全上下文中仅维护一个会话。这导致性能的改善。

企业用户安全11g第1版（11.1）及更高版本允许更精细地为企业用户分配代理权限。企业用户可以被单独授予作为本地数据库用户代理的权限。权限不再需要与数据库中用户的共享模式相关联。

能够为企业用户单独分配代理权限意味着权限可以更具体。另一方面，为共享模式分配权限会强制您为映射到模式的所有用户分配相同的权限。这可能导致无根据的权利和特权。

创建企业用户代理权限并将其存储在Oracle Internet Directory中。权限允许一个或多个企业用户或组作为目标数据库用户进行代理。权限可以应用于特定数据库或企业域中的所有数据库。

默认情况下，域管理员可以管理企业域的目录中的代理权限。使用Oracle Enterprise Manager配置和管理这些权限。

关于为企业用户安全性使用当前用户数据库链接

Oracle数据库支持 当前用户数据库通过SSL身份验证的网络连接进行链接。当前用户数据库链接允许您自己连接到第二个数据库，或者在从该用户拥有的存储过程中使用时连接到另一个用户。这种访问仅限于程序的范围。当前用户数据库链接的安全优势在于，其他用户的凭据未存储在数据库链接定义中，并且不通过数据库之间的网络连接发送。相反，这些链接的安全性基于相互信任，相互认证以及数据库本身之间的安全网络连接。

例如，当前用户数据库链接允许Harriet（Finance数据库的用户）通过作为企业用户Scott连接来程序性地访问Accounts Payable数据库。

为了让Harriet访问当前用户数据库链接以连接到模式Scott，Scott必须是两个数据库中的全局模式（在IDENTIFIED GLOBALLY中创建）。但是，Harriet可以通过以下三种方式之一识别用户：

  • 通过密码

  • GLOBALLY

  • EXTERNALLY

要在第一个数据库Finance中创建Scott作为全局用户，您必须输入

CREATE USER Scott全球识别为'CN = Scott，O = nmt'

这样Scott就拥有了独家架构。然后，Scott可以映射到第二个数据库Accounts Payable中的共享模式。为了使当前用户数据库链接起作用，在第一个数据库中为Scott创建的模式无法与其他用户共享。

当前用户数据库链接仅在单个企业域内的可信数据库之间运行。域中的数据库相互信任以对用户进行身份验证。您可以使用Oracle Enterprise Manager将企业域指定为受信任的域。使用Oracle Enterprise Manager为域启用当前用户数据库链接时，它们将适用于该域中的所有数据库。但是，域中的每个数据库都必须具有自己的PKI凭据，并使用SSL对其他数据库进行身份验证。要将数据库指定为受信任的企业域的一部分，请使用PL / SQL包DBMS_DISTRIBUTED_TRUST_ADMIN.要获取可信服务器列表，请使用TRUSTED_SERVERS视图。

注意：

Oracle Advanced Security，Oracle数据库企业版的一个选项， 不支持数据库链接上的RADIUS身份验证。

集中安全证书的安全方面

除了企业用户集中化及其相关凭证所带来的一般好处之外，还应审查一些与安全相关的好处和风险。以下部分描述了这些好处和风险：

  • 与集中安全凭证管理相关的安全利益

  • 与集中安全凭证管理相关的安全风险

与集中安全凭证管理相关的安全利益

集中管理可以更轻松，更快速地管理用户，凭据和角色，并快速撤消用户对整个企业中所有应用程序和数据库的权限。通过集中管理，管理员可以在一个位置删除用户以撤消所有全局权限，从而最大限度地降低保留非预期权限的风险。

集中管理可以集中组织的安全专业知识。专业的安全感知管理员可以管理企业用户安全的所有方面，包括目录安全性，用户角色和权限以及数据库访问。这是对传统模型的重大改进，其中DBA通常负责其管理的数据库上的所有内容，包括安全性。

与集中安全凭证管理相关的安全风险

虽然Oracle Internet Directory是一个安全的存储库，但在任何可公开访问的存储库中集中凭据存在安全挑战和固有风险。虽然集中凭证可以至少像分布式凭证一样安全地受到保护，但集中化的本质增加了无意识凭证暴露给未授权方的后果。因此，必须限制管理员的权限以在目录中设置限制性访问控制列表（ACL），并在临时位于目录之外时实施安全凭证保护中的良好安全实践。

密码验证企业用户数据库登录信息的安全性

在所有基于密码的安全身份验证方法中，服务器使用密码验证程序对客户端进行身份验证，通常是必须严格保护的密码的哈希版本。对Oracle数据库的基于密码的身份验证也不例外。有一个密码验证程序，它也必须受到保护。如果验证程序本地存储在数据库中或集中存储在目录中，则为true。请注意，密码验证程序不能用于派生原始密码。

企业用户的数据库密码可以存储在中央目录服务中，以供多个数据库访问。它可以被用户有权访问的所有可信数据库查看和共享。虽然存储在目录中的密码验证程序不是明文密码，但仍然需要保护它免受随意或未经授权的访问。因此非常 重要的是在目录中定义与密码相关的ACL，尽可能限制，同时仍然启用必要的访问和可用性。（请注意，Oracle数据库支持Oracle Internet Directory支持的所有验证程序类型。）

Oracle工具帮助在目录中设置ACL，以在身份管理领域创建期间保护这些密码验证程序。Oracle建议的方法旨在平衡安全性和可用性考虑因素。如果您需要最高安全性并且可以为所有用户设置钱包，则只需要从用户到数据库的SSL连接。这种仅限SSL的方法可以绕过整个目录密码保护问题。

可信数据库的意义何在

SSL提供强身份验证，因此可确保数据库以其自己的身份进行信任。使用密码验证的企业用户安全性，其中数据库密码验证器集中存储在目录中并在多个数据库之间共享，允许密码验证的企业用户登录的每个数据库必须是受信任的数据库。每个数据库都可以访问共享密码验证程序，因此必须信任每个数据库以遵守以下安全预防措施：

  • 必须信任每个数据库以防止自己篡改服务器代码，以便恶意用户不会滥用数据库身份来访问目录中的密码验证程序。

  • 必须信任每个数据库以保护其PKI和其他凭据免遭盗窃，以便恶意用户无法使用它们来访问存储在目录中的密码验证程序。

保护数据库密码验证程序

每个身份管理领域中的OraclePasswordAccessibleDomains组在创建领域时自动创建，并且可以使用Oracle Internet Directory工具（如Oracle Internet Directory自助服务控制台）进行管理。具有必须在目录中查看用户的数据库密码验证程序的成员数据库的企业域将放置在此组中。

对于选定的域，确定哪些数据库可以接受经过密码验证的连接。使用Oracle Internet Directory自助服务控制台将包含这些数据库的域放入OraclePasswordAccessibleDomains组。用户子树上的ACL允许访问包含数据库使用的密码验证程序的目录属性。

拒绝所有其他用户访问此属性。阻止对密码验证者属性进行匿名读取访问的ACL位于目录树的根目录下。

请注意，对于可用性，默认情况下，OracleDefaultDomain是OraclePasswordAccessibleDomains组的成员。如果需要，可以将其删除。

在企业域中定义数据库成员身份的注意事项

定义域的数据库成员身份时，请考虑以下条件：

  • 当前用户数据库链接 s运行只有一个内数据库之间的企业域 。使用这些链接需要这些数据库之间以及管理它们的DBA之间的相互信任。

  • 企业用户的接受身份验证类型是在域级别定义的。因此，应相应地定义域中的数据库成员资格。

    注意：

    如果一个或多个数据库仅支持基于SSL的证书身份验证，则不能将它们与密码验证的数据库组合在同一域中。

  • 企业角色在域级别定义。要跨多个数据库共享企业角色 ，数据库必须是同一域的成员。
