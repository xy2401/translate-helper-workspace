关于使用统一审核策略和审核的审核活动

您可以使用统一审核策略和AUDIT SQL语句审核多种类型的活动。

您可以审核的活动类型如下：

  • 用户帐户（包括使用SYSDBA管理权限登录的管理用户），角色和权限

  • 对象操作，例如删除表或运行过程

  • 应用上下文值

  • Oracle数据库真实应用程序安全性，Oracle恢复管理器，Oracle数据挖掘，Oracle数据泵，Oracle SQL * Loader直接路径事件，Oracle Database Vault和Oracle Label Security中的活动

要完成此操作，请根据您要审核的内容，使用以下内容：

  • 统一审计政策。 统一审核策略是一组命名的审核设置，使您可以审核数据库中用户行为的特定方面。要创建策略，请使用CREATE AUDIT POLICY语句。策略可以像审核单个用户的活动一样简单，也可以创建使用条件的复杂审核策略。您可以在数据库中一次有多个审核策略生效。审核策略可以包含系统范围和特定于对象的审核选项。您将为一般活动（包括标准审计）执行的大多数审计都需要使用审计策略。

  • AUDIT和NOAUDIT SQL语句。 AUDIT和NOAUDIT SQL语句使您可以分别启用和禁用审核策略。AUDIT语句还允许您包含或排除策略的特定用户。AUDIT和NOAUDIT语句还使您能够审计应用程序上下文值。

  • 对于Oracle Recovery Manager，您不会创建统一的审核策略。 UNIFIED_AUDIT_TRAIL视图自动捕获通常审核的Recovery Manager事件。

创建统一审核策略的最佳实践

您可以在数据库中一次启用多个策略，但理想情况下，可以限制已启用策略的数量。

统一审计策略语法的设计使您可以编写一个涵盖数据库所需的所有审计设置的策略。一个好的做法是将相关选项分组到单个策略中，而不是创建多个小策略。这使您可以更轻松地管理策略。例如， 使用预定义统一审核策略的审核活动中描述的默认审核策略均包含一个统一审核策略中的多个审核设置。

限制用户会话的已启用审核策略的数量具有以下好处：

  • 它减少了将审计策略的详细信息加载到会话的UGA内存中的登录开销。如果启用的策略计数较少，则加载策略信息所花费的时间较少。

  • 它减少了会话的UGA内存消耗，因为需要在UGA内存中缓存较少数量的策略。

  • 它使内部审计检查功能更有效，从而确定是否为其关联事件生成审计记录。

创建统一审计策略的语法

要创建统一的审核策略，必须使用CREATE AUDIT POLICY语句。

创建统一审核策略时，Oracle数据库会将其存储在SYS模式所拥有的第一个类对象中，而不是存储在创建策略的用户的模式中。

例25-1显示了CREATE AUDIT POLICY语句的语法。

示例25-1 CREATE AUDIT POLICY语句的语法

CREATE审核策略POLICY_NAME {{privilege_audit_clause [action_audit_clause] [role_audit_clause]} | {action_audit_clause [role_audit_clause]} | {role_audit_clause}} [WHEN audit_condition EVALUATE PER {STATEMENT | SESSION | INSTANCE}] [ONLY TOPLEVEL] [CONTAINER = {CURRENT |所有}];

在本规范中：

  • privilege_audit_clause描述与权限相关的审计选项。有关详细信息，请参阅审核系统权限 。配置权限审计选项的详细语法如下：

    privilege_audit_clause ：= PRIVILEGES privilege1 [， privilege2 ]

  • action_audit_clause和standard_actions描述与对象操作相关的审计选项。请参阅审核对象操作 。语法如下：

    action_audit_clause ：= { standard_actions | component_actions } [， component_actions ] standard_actions ：= ACTIONS action1 [ON { schema 。 obj_name | DIRECTORY directory_name | MINING MODEL 架构 。 obj_name }] [， action2 [ON { schema 。 obj_name | DIRECTORY directory_name | MINING MODEL 架构 。 obj_name }]

  • component_actions使您可以为Oracle Label Security，Oracle Database Real Application Security，Oracle Database Vault，Oracle Data Pump或Oracle SQL * Loader创建审核策略。有关详细信息，请参阅“ 使用统一审核策略进行审核活动”下的相应部分和“审核声明 ”。语法是：

    component_actions ：= ACTIONS COMPONENT = [OLS | XS] action1 [， action2 ] | ACTIONS COMPONENT = DV DV_action ON DV_object_name | ACTIONS COMPONENT = DATAPUMP [EXPORT |进口|所有] | ACTIONS COMPONENT = DIRECT_LOAD [LOAD |全部]

  • role_audit_clause使您可以审核角色。请参阅审核角色 。语法是：

    role_audit_clause ：= ROLES role1 [， role2 ]

  • WHEN audit_condition EVALUATE PER使您可以指定一个函数来为审计策略和评估频率创建条件。必须在WHEN条件中包含EVALUATE PER子句。请参阅为统一审核策略创建条件 。语法是：

    何时' audit_condition ：= 函数操作value_list '评估每个{语句|会话|实例}
  • ONLY TOPLEVEL允许用户ONLY TOPLEVEL审核为作为此审核策略的一部分配置的操作执行的顶级操作。请参阅仅审核顶级语句 。

  • CONTAINER允许用户仅审核为作为此审核策略的一部分配置的操作执行的顶级操作。请参阅多租户环境中的统一审核策略或审核设置 。

此语法旨在审核策略中列出的任何组件。例如，假设您创建以下策略：

CREATE AUDIT POLICY table_pol PRIVILEGES CREATE ANY TABLE，DROP ANY TABLE ROLES emp_admin，sales_admin;

审计线索能捕捉到需要的SQL语句CREATE ANY TABLE系统特权或DROP ANY TABLE系统特权或任何系统权限直接授予角色emp_admin或任何系统权限直接授予角色sales_admin 。（请注意，它会审核直接授予的权限，而不是通过角色递归授予的权限。）

创建策略后，必须使用AUDIT语句启用它。（可选）您可以将策略应用于一个或多个用户，从策略中排除一个或多个用户，并指定在审计操作成功，失败或成功或失败时是否写入审计记录。请参阅为用户和角色启用和应用统一审核策略 。

关于系统权限审核

系统特权审核审核使用系统特权的活动，例如READ ANY TABLE.

审计功能，过程，包和触发器

您可以审核函数，过程，PL / SQL包和触发器。

您可以审核的区域如下：

  • 您可以单独审核独立函数，独立过程和PL / SQL包。

  • 如果为所有执行启用审计，Oracle数据库将审计数据库中的所有触发器以及PL / SQL包中的所有函数和过程。

  • 在PL / SQL存储过程或存储函数上审计EXECUTE操作时，数据库仅考虑查找过程或函数的能力，并在为审计目的确定操作成功或失败时授权其执行。因此，如果指定WHENEVER NOT SUCCESSFUL子句，则仅审计无效对象错误，不存在的对象错误和授权失败;执行过程或函数期间遇到的错误不会被审计。如果指定WHENEVER SUCCESSFUL子句，则无论是否在执行期间遇到错误，都会审计未被无效对象错误，不存在的对象错误或授权失败阻止的所有执行。

统一审计跟踪自动捕获Oracle虚拟专用数据库（VPD）策略中使用的谓词。

您无需创建统一的审核策略来捕获VPD谓词审核信息。

Oracle虚拟专用数据库策略功能的审核策略

审计可能会影响动态VPD策略，静态VPD策略和上下文相关的VPD策略。

  • 动态策略： Oracle数据库评估策略函数两次，一次是在SQL语句解析期间，另一次是在执行期间。结果，为每个评估生成两个审计记录。

  • 静态策略： Oracle数据库评估策略函数一次，然后将其缓存在SGA中。因此，只生成一个审计记录。

  • 上下文相关策略： Oracle Database在语句解析期间执行一次策略函数。因此，只生成一个审计记录。

使用条件配置统一审核策略

CREATE AUDIT POLICY语句中的WHEN子句定义审计策略中的条件。

  • 使用以下语法创建使用条件的统一审核策略：

    CREATE AUDIT POLICY policy_name action_privilege_role_audit_option [WHEN function_ operation_ value_list_1 [[AND | OR] function_ operation_ value_list_n ]评估每个语句|会话| INSTANCE];

在本规范中：

  • action_privilege_role_audit_option是指系统操作，对象操作，权限和角色的审计选项。

  • WHEN定义条件。它有以下组件：

    • function使用以下类型的函数：

      数字函数，例如BITAND ， CEIL ， FLOOR和LN POWER

      返回字符值的字符函数，例如CONCAT ， LOWER和UPPER

      返回数值的字符函数，例如LENGTH或INSTR

      环境和标识符函数，例如SYS_CONTEXT和UID 。对于SYS_CONTEXT ，在大多数情况下，您可能希望使用USERENV命名空间，这在Oracle数据库SQL语言参考中有所描述。

    • operation可以是以下任何操作符： AND ， OR ， IN ， NOT IN ， = ， < ， > ， <>

    • value_list指的是您正在测试的条件。

    您可以为每个function_operation_value_list集包含其他条件，以AND或OR分隔。

    编写WHEN子句时，请遵循以下准则：

    • 将整个function operation value设置括在单引号中。在该子句中，将每个引用的组件包含在两对单引号内。不要使用双引号。

    • WHEN条件不要超过4000个字节。

  • EVALUATE PER指的是以下选项：

    • STATEMENT评估发生的每个相关可审计语句的条件。

    • SESSION仅在会话期间评估条件一次，然后在会话的剩余时间内缓存并重新使用结果。Oracle数据库在第一次使用策略时评估条件，然后将结果存储在UGA内存中。

    • INSTANCE仅在数据库实例生存期内评估条件一次。在Oracle数据库评估条件之后，它会缓存并重新使用结果，以用于实例生命周期的剩余部分。与SESSION评估一样，评估在第一次需要时进行，然后结果存储在UGA内存中。

请记住，在创建策略后，必须使用AUDIT语句来启用它。

您可以在统一审计跟踪中捕获应用程序上下文值。

此功能使您可以在执行审计语句时捕获数据库应用程序设置的任何应用程序上下文值。

如果您计划审核Oracle Label Security，则此功能会捕获数据库审计跟踪的会话标签活动。审计跟踪记录为指定的上下文属性值对检索的所有值。

应用程序上下文审核设置或审核策略具有会话静态语义。换句话说，如果为用户启用了新策略，则后续用户会话将看到此命令的效果。会话建立后，将加载策略和上下文设置，后续的AUDIT语句对该会话没有影响。

对于多租户环境，应用程序上下文审核策略仅适用于当前PDB。

为Oracle Database Vault配置统一审核策略

CREATE AUDIT POLICY语句中的ACTIONS和ACTIONS COMPONENT子句可以为Oracle Database Vault事件创建统一的审核策略。

  • 使用以下语法创建Oracle Database Vault统一审核策略：

    CREATE AUDIT POLICY policy_name ACTIONS action1 [， action2 ] ACTIONS COMPONENT = DV DV_action ON DV_object [， DV_action2 ON DV_object2 ]

在本规范中：

  • DV_action是以下之一：

    • Realm Violation ， Realm Success ， Realm Access

    • Rule Set Failure ， Rule Set Success ， Rule Set Eval

    • Factor Error, Factor Null ， Factor Validate Error ， Factor Validate False ， Factor Trust Level Null ， Factor Trust Level Neg ， Factor All

  • DV_objects是以下之一：

    • Realm_Name

    • Rule_Set_Name

    • Factor_Name

如果对象是以较低或大小写形式创建的，则必须将DV_objects括在双引号中。如果您使用所有大写字母创建了对象，则可以省略引号。

例如，要审核Database Vault帐户管理领域中的域违规：

CREATE AUDIT POLICY audit_dv ACTIONS CREATE TABLE，SELECT ACTIONS COMPONENT = DV Realm Violation ON“Database Vault Account Management”;

您可以构建更复杂的策略，例如包含条件的策略。请记住，在创建策略后，必须使用AUDIT语句来启用它。

CONTAINER子句特定于CREATE AUDIT POLICY语句的多租户环境使用。

要在CDB环境或应用程序容器环境中创建本地或公共（CDB或应用程序）统一审计策略，请在CREATE AUDIT POLICY语句中包含CONTAINER子句。

  • 使用以下语法创建本地或通用统一审核策略：

    CREATE AUDIT POLICY policy_name action1 [， action2 ] [CONTAINER = {CURRENT |所有}];

在本规范中：

  • CURRENT将审核策略设置为当前PDB的本地策略。

  • ALL使审计策略成为通用审计策略，即整个多租户环境可用。

创建审计策略dict_updates SYS.USER上的操作更新$，删除SYS.USER $，更新SYS.LINK $，删除SYS.LINK $ CONTAINER = ALL;

  • 您可以为CREATE AUDIT POLICY语句设置CONTAINER子句，但不能为ALTER AUDIT POLICY或DROP AUDIT POLICY 。如果要更改现有统一审核策略的范围以使用此设置，则必须删除并重新创建策略。

  • 对于AUDIT报表，您可以设置CONTAINER的审核设置条款只有当你有尚未迁移到12版和更高的审计功能的Oracle数据库。您不能在用于启用统一审核策略的AUDIT语句中使用CONTAINER子句。

  • 如果您在PDB中，那么您只能将CONTAINER子句设置为CURRENT ，而不是ALL 。如果在PDB中省略设置，则默认值为CONTAINER = CURRENT 。

  • 如果您位于根目录中，则可以将CONTAINER子句设置为CURRENT如果希望策略仅应用于根），或者设置为ALL如果希望策略应用于整个CDB）。如果省略CONTAINER子句，则默认为CONTAINER = CURRENT 。

  • 对象：

    • 通用审计策略只能有公共对象，本地审计策略可以同时具有本地对象和公共对象。

    • 如果涉及的对象是本地的，则无法将CONTAINER设置为ALL 。它们必须是共同的对象。

  • 对于特权：

    • 如果涉及的用户帐户是本地帐户和普通帐户的混合，则可以将CONTAINER设置为CURRENT （或省略CONTAINER子句）。这将创建仅适用于当前PDB的本地审核配置。

    • 如果涉及的用户是本地用户，则无法将CONTAINER设置为ALL 。他们必须是普通用户。

    • 如果将CONTAINER设置为ALL并且未指定用户列表（使用AUDIT语句中的BY子句），则配置将应用于每个PDB中的所有公共用户。

      ALTER PLUGGABLE DATABASE APPLICATION BEGIN INSTALL 列出SQL语句。用分号分隔每个语句。
 ALTER PLOGGABLE DATABASE应用程序结束安装

    创建并启用审核策略后，无论是在数据库中还是从脚本中定义审核策略，都会审核对应用程序公共对象的所有用户访问权限。

在被审计用户（或已被授予与策略关联的角色的用户）登录到数据库实例之后，策略才会生效。换句话说，如果在审核用户登录时创建并启用策略，则策略无法收集审核数据;用户必须先注销然后再次登录才能开始审核。一旦设置会话并对其进行审核，则设置将持续与用户会话一样长，然后在会话结束时结束。

您可以为单个用户或角色启用审核策略。通过为角色启用审核策略，您可以为直接授予该角色的一组用户启用该策略。将角色直接授予新用户后，该策略将自动应用于该用户。从用户撤消角色后，该策略将不再适用于该用户。

如果已为会话启用统一审核策略，则此现有会话将看不到删除策略的影响。在此之前，统一审计政策的设置仍然有效。但是，对于与对象相关的统一审计策略，效果是即时的。

您可以通过查询AUDIT_UNIFIED_POLICIES数据字典视图找到现有统一审核策略的列表。

在删除审核策略之前禁用审核策略时，请确保使用与启用它相同的设置禁用它。例如，假设您启用了logon_pol策略，如下所示：

审计政策logon_pol BY HR，OE;

在删除之前，您的NOAUDIT语句必须包括HR和OE用户，如下所示：

NOAUDIT POLICY logon_pol BY HR，OE;

在多租户环境中，您只能从根目录中删除通用审计策略，而只能从应用它的PDB中删除本地审计策略。

删除统一审核策略

要删除统一审核策略，必须先将其禁用，然后运行DROP AUDIT POLICY语句将其删除。

  • 使用以下语法删除统一审核策略：

    DROP AUDIT POLICY policy_name ;

在多租户环境中，统一审计策略删除适用于当前PDB。如果统一审核策略是作为通用统一审核策略创建的，则无法从本地PDB中删除它。

关于细粒度审计

通过细粒度审计，您可以创建策略，以定义审计发生时必须具备的特定条件。

您无法使用细粒度审计创建统一审计策略，但可以使用细粒度审计来创建非常自定义的审计设置，例如审计访问数据的时间。

这使您可以根据内容监控数据访问。它提供查询的粒度审计，以及INSERT ， UPDATE和DELETE操作。您可以使用细粒度审计来审核以下类型的操作：

  • 在晚上9点到早上6点之间或周六和周日访问桌子

  • 使用公司网络外部的IP地址

  • 选择或更新表列

  • 修改表列中的值

通常，细粒度审计策略基于表对象上的简单用户定义SQL谓词作为选择性审计的条件。在获取期间，只要满足行的策略条件，就会审核查询。

使用统一审计策略的审计活动和AUDIT语句中描述的审计策略可以执行细粒度审计策略可以执行的大多数操作，但以下操作除外：

  • 审核特定列。 您可以审核包含敏感信息的特定相关列，例如工资或社会安全号码。

  • 使用事件处理程序。 例如，您可以编写一个函数，在更新午夜不应更改的审计列时，向安全管理员发送电子邮件警报。

注意：

  • 只有基于成本的优化才支持细粒度审计。对于使用基于规则的优化的查询，在应用行过滤之前进行细粒度审计检查，这可能导致不必要的审计事件触发器。

创建细粒度审计策略的语法

DBMS_FGA.ADD_POLICY过程包括许多设置，例如使用处理程序进行复杂审计的能力。

DBMS_FGA.ADD_POLICY过程语法如下：

DBMS_FGA.ADD_POLICY（object_schema IN VARCHAR2 DEFAULT NULL object_name IN VARCHAR2，policy_name IN VARCHAR2，audit_condition IN VARCHAR2 DEFAULT NULL，audit_column IN VARCHAR2 DEFAULT NULL handler_schema IN VARCHAR2 DEFAULT NULL，handler_module IN VARCHAR2 DEFAULT NULL，enable IN BOOLEAN DEFAULT TRUE，statement_types IN VARCHAR2 DEFAULT SELECT，audit_trail IN BINARY_INTEGER DEFAULT NULL，audit_column_opts IN BINARY_INTEGER DEFAULT ANY_COLUMNS，policy_owner IN VARCHAR2 DEFAULT NULL）;

在本规范中：

  • object_schema指定要审核的对象的模式。（如果为NULL ，则假定当前的登录用户模式。）

  • object_name指定要审核的对象的名称。

  • policy_name指定要创建的策略的名称。确保此名称是唯一的。

  • audit_condition指定行中的布尔条件。允许NULL并充当TRUE 。有关详细信息，请参阅特定列和行的审核 。如果指定NULL或没有审计条件，则对具有该策略的表执行的任何操作都会创建审计记录，无论是否返回行。

    请遵循以下准则：

    • 不要在audit_condition设置中包含在同一基表上执行auditable语句的函数。例如，假设您创建了一个在HR.EMPLOYEES表上执行INSERT语句的函数。策略的audit_condition包含此函数，它适用于INSERT语句（由statement_types设置）。使用策略时，函数将递归执行，直到系统内存不足。这可能会引发错误ORA-1000: maximum open cursors exceeded或ORA-00036: maximum number of recursive SQL levels (50) exceeded ORA-1000: maximum open cursors exceeded ORA-00036: maximum number of recursive SQL levels (50) exceeded 。

    • 不要从策略条件中的函数发出DBMS_FGA.ENABLE_POLICY或DBMS_FGA.DISABLE_POLICY语句。

  • audit_column指定要审核的一个或多个列，包括隐藏列。如果设置为NULL或省略，则审计所有列。这些可以包括Oracle Label Security隐藏列或对象类型列。如果访问或影响任何列，则默认值NULL导致审计。

  • handler_schema ：如果在违反策略时使用警报触发响应，则指定包含事件处理程序的模式的名称。默认值NULL使用当前架构。另请参阅教程：向细粒度审核策略添加电子邮件警报 。

  • handler_module指定事件处理程序的名称。包括事件处理程序所在的包。只有在处理了与查询中的审计条件匹配的第一行后，才会调用此函数。

    请遵循以下准则：

    • 不要创建递归的细粒度审计处理程序。例如，假设您创建了一个在HR.EMPLOYEES表上执行INSERT语句的处理程序。与此处理程序关联的策略适用于INSERT语句（由statement_types参数设置）。使用策略时，处理程序以递归方式执行，直到系统内存不足。这可能会引发错误ORA-1000: maximum open cursors exceeded或ORA-00036: maximum number of recursive SQL levels (50) exceeded ORA-1000: maximum open cursors exceeded ORA-00036: maximum number of recursive SQL levels (50) exceeded 。

    • 不要从策略处理程序发出DBMS_FGA.ENABLE_POLICY或DBMS_FGA.DISABLE_POLICY语句。这样做会引发ORA-28144: Failed to execute fine-grained audit handler错误。
