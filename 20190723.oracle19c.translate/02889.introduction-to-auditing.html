什么是审计？

审计是从数据库用户和非数据库用户监视和记录已配置的数据库操作。

“Nondatabase用户”是指使用CLIENT_IDENTIFIER属性在数据库中识别的应用程序用户。要审核此类用户，您可以使用统一的审核策略条件，细粒度的审核策略或Oracle数据库Real Application Security。

您可以基于单个操作（例如，执行的SQL语句的类型）或可以包括用户名，应用程序，时间等的数据组合进行审计。

您可以为成功和失败的活动配置审核，并在审核中包括或排除特定用户。在多租户环境中，您可以审核可插拔数据库（PDB）的各个操作或整个多租户容器数据库（CDB）中的各个操作。除了审核数据库提供的标准活动之外，审计还可以包括来自Oracle数据库真实应用程序安全性，Oracle恢复管理器，Oracle数据泵，Oracle数据挖掘，Oracle Database Vault，Oracle Label Security和Oracle SQL * Loader直接路径事件的活动。 。

默认情况下启用审核。所有审计记录都以统一格式写入统一审计跟踪，并通过UNIFIED_AUDIT_TRAIL视图提供。这些记录驻留在AUDSYS模式中。默认情况下，审计记录存储在SYSAUX表空间中。Oracle建议您为统一审计跟踪配置不同的表空间。请注意，对于除Enterprise Edition之外的大多数Oracle数据库版本，您只能将表空间关联一次以进行统一审计。在为统一审计跟踪生成任何审计记录之前，应执行此关联。关联表空间后，无法对其进行修改，因为仅在Enterprise Edition上支持分区。

您可以使用以下任一方法配置审核：

  • 将审核设置分组到一个统一的审核策略中 您可以创建一个或多个统一审核策略，以定义数据库所需的所有审核设置。使用统一审核策略和AUDIT语句的审核活动描述了如何完成此操作。

  • 使用默认的统一审核策略之一。 Oracle数据库提供三种默认的统一审计策略，其中包含大多数监管机构所需的标准审计设置。请参阅使用预定义的统一审核策略审核活动 。

  • 创建细粒度的审核策略。 您可以创建细粒度的审核策略来捕获数据，例如操作发生的时间。请参阅使用细粒度审计审计特定活动 。

Oracle建议您审核数据库。审计是强制执行强有力的内部控制的有效方法，以便您的网站符合“萨班斯 - 奥克斯利法案”中规定的法规遵从要求。这使您可以监视业务操作，并查找可能偏离公司策略的任何活动。这样做可转换为对数据库和应用程序软件的严格控制访问，确保按计划应用修补程序并防止临时更改。通过创建有效的审计策略，您可以为审计和合规人员生成审计记录。选择审计并确保其满足您的业务合规性需求。

为什么要使用审计？

您通常使用审计来监视用户活动。

审计可用于完成以下任务：

  • 实现行动的责任。 这些包括在特定架构，表或行中执行的操作，或影响特定内容的操作。

  • 根据用户的责任，阻止用户（或其他人，如入侵者）采取不当行为。

  • 调查可疑活动。 例如，如果用户正在从表中删除数据，则安全性管理员可以审核与数据库的所有连接以及数据库中所有表的所有成功和不成功删除行。

  • 通知审核员未授权用户的操作。 例如，未经授权的用户可能正在更改或删除数据，或者用户拥有的权限超出预期，这可能导致重新评估用户授权。

  • 监视和收集有关特定数据库活动的数据。 例如，数据库管理员可以收集有关正在更新哪些表，执行了多少逻辑I / O或在高峰时间连接多少并发用户的统计信息。

  • 检测授权或访问控制实现的问题。 例如，您可以创建预期永远不会生成审计记录的审计策略，因为数据受到其他方式的保护。但是，如果这些策略生成审计记录，那么您将知道其他安全控制未正确实现。

  • 解决合规性的审计要求。 以下法规具有共同的审计相关要求：

    • 萨班斯 - 奥克斯利法案

    • 健康保险流通与责任法案（HIPAA）

    • 资本计量与资本标准的国际趋同：经修订的框架（巴塞尔协议II）

    • 日本隐私法

    • 欧盟隐私和电子通信指令

审计的最佳实践

您应遵循审核的最佳实践指南。

  • 作为一般规则，设计审计策略以收集满足合规性要求所需的信息量，但要关注导致最大安全性问题的活动。 例如，审计数据库中的每个表都不实用，但审计包含包含敏感数据（如工资）的列的表是。通过统一和细粒度审计，您可以使用一些机制来设计专注于审计特定活动的审计策略。

  • 定期归档和清除审计跟踪数据。 有关更多信息，请参阅清除审计跟踪记录 。

统一审计跟踪的好处

统一审计跟踪的好处很多。

例如：

  • 启用统一审核后，它不依赖于先前版本中使用的初始化参数。有关这些初始化参数的列表，请参见表G-1 。

  • 对于Oracle数据库安装的所有已审计组件，审计记录（包括SYS审计跟踪的记录）放在一个位置并采用一种格式，而不是必须查看不同的位置以查找不同格式的审计跟踪。该统一视图使审计人员能够共同关联来自不同组件的审计信息。例如，如果在INSERT语句期间发生错误，则标准审计可以指示错误号和已执行的SQL。特定于Oracle Database Vault的信息可以指示是否由于命令规则违规或域违规而发生此错误。请注意，将有两个具有不同AUDIT_TYPE审核记录。通过这种统一， SYS审计记录显示为AUDIT_TYPE设置为Standard Audit 。

  • 审计跟踪的管理和安全性也通过单一审计跟踪得到改善。

  • 整体审计绩效大大提高。默认情况下，审计记录会自动写入AUDSYS架构中的内部关系表。

  • 您可以创建命名审核策略，以便审核本节开头列出的受支持组件以及SYS管理用户。此外，您可以在策略中构建条件和排除项。

  • 如果您使用的是Oracle Audit Vault和Database Firewall环境，则统一审计跟踪可以极大地方便收集审计数据，因为所有这些数据都来自一个位置。

关于混合模式审计

混合模式审计既可以实现传统（即，比版本12c更早的版本的审计功能），也可以启用新的审计功能（统一审计）。

创建新数据库时，默认情况下数据库使用混合模式审核。

您可以使用以下两种模式之一启用数据库：混合模式审核或纯统一审核模式。尽管在这两种模式下都启用了统一审核的功能，但它们之间存在差异。在混合模式下，您可以将新的统一审计工具与传统审计工具一起使用。在纯统一审计中，您只使用统一审计工具。

混合模式审计的能力

混合模式审核提供了多种功能。

这些功能如下：

  • 它允许使用所有现有的审计初始化参数： AUDIT_TRAIL ， AUDIT_FILE_DEST ， AUDIT_SYS_OPERATIONS和AUDIT_SYSLOG_LEVEL 。

  • 它仅将强制审计记录写入传统审计跟踪。

  • 它基于标准审计配置的标准审计记录，并将这些记录写入AUDIT_TRAIL初始化参数指定的审计跟踪。

    但是，请注意，标准审计跟踪记录也是基于统一审计策略生成的，并且只有这些审计记录才会写入统一审计跟踪。由统一审计策略生成的标准审计记录遵循统一审计策略启用的语义。

  • 管理用户会话生成SYS审核记录。如果AUDIT_SYS_OPERATIONS初始化参数设置为TRUE则写入这些记录。此过程仅将记录写入传统审计跟踪。但是，为管理用户启用统一审核策略时，这些统一审核记录也会写入统一审核跟踪。

  • 写入传统审计跟踪的审计记录格式与Oracle Database 11g第2版中的格式相同。

  • 默认情况下，Oracle数据库会立即将统一审计记录写入AUDSYS架构中的内部关系表。有关详细信息，请参阅将统一审计跟踪记录写入AUDSYS架构 。

  • 编写审计记录的性能成本等于将审计记录生成和写入传统审计跟踪和统一审计跟踪所需的时间总和。

  • 混合模式审核提供了统一审核模式功能的一目了然。Oracle建议您在熟悉新样式的审核策略和审核跟踪后迁移到统一审核模式。要迁移到统一审核，请参阅“ Oracle数据库升级指南” 。

谁可以进行审计？

Oracle为执行审计的用户提供了两个角色： AUDIT_ADMIN和AUDIT_VIEWER 。

要执行任何类型的审核，您必须被授予AUDIT_ADMIN角色。审计员可以在授予AUDIT_VIEWER角色后查看审计数据。

这些角色提供的权限如下：

  • AUDIT_ADMIN角色。 通过此角色，您可以创建统一且细粒度的审计策略，使用AUDIT和NOAUDIT SQL语句，查看审计数据以及管理审计跟踪管理。仅将此角色授予受信任的用户。

  • AUDIT_VIEWER角色。 此角色使用户可以查看和分析审计数据。它为DBMS_AUDIT_UTIL PL / SQL包提供EXECUTE特权。需要此角色的用户通常是外部审计员。

注意：

在以前的版本中，允许用户在没有任何其他权限的情况下向自己的模式中的对象添加和删除审计配置。不再允许这种能力。

关于多租户环境中的审计

您可以在多租户环境中使用统一审核。

您可以将审核设置应用于单个PDB或CDB，具体取决于策略的类型。在多租户环境中，每个PDB（包括根）都具有自己的统一审计跟踪。

有关更多信息，请参阅以下部分：

  • 使用CREATE AUDIT POLICY和AUDIT语句创建的统一审计策略：您可以为根PDB和单个PDB创建策略。

  • 细粒度审计策略：您只能为单个PDB创建策略，而不能为根创建策略。

  • 清除审计跟踪：您可以对根PDB和单个PDB执行清除操作。
