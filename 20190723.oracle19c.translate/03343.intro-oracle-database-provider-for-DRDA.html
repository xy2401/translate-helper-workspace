1 DRDA的Oracle数据库提供程序简介

考虑用于DRDA技术的Oracle数据库提供程序的一些基本概念。

什么是DRDA的Oracle数据库提供程序？

该用于DRDA的Oracle数据库提供程序是一个网络前端，它使客户端程序可以使用分布式关系数据库体系结构（DRDA）协议。

用于DRDA的Oracle数据库提供程序实现了完整DRDA第4版规范的大部分，以及其他DRDA服务器产品的若干方面（例如 IBM的DB2）兼容性。它是一个独立于数据库的协议，它仅提供Oracle数据库提供的功能。用于DRDA产品的Oracle数据库提供程序使现有DB2应用程序客户能够在迁移时利用其当前在应用程序技术方面的投资 DB2服务器。

调用使用DRDA协议的客户端程序或系统申请请求者（ ARS）。提供提供DRDA协议服务的服务器程序或系统，例如用于DRDA的Oracle数据库提供程序应用服务器（ 如）。

编写为使用DRDA协议的应用程序，可以是直接AR，也可以是中间接口（如嵌入式SQL），通常不需要通过Oracle数据库提供程序为DRDA更改其现有代码以连接到Oracle数据库。只有最小的应用程序配置更改，例如重定向，是成功更改客户端应用程序环境以使用Oracle数据库所必需的。

发布信息

用于DRDA的Oracle数据库提供程序是一个网络前端，它使客户端程序能够使用分布式关系数据库体系结构（DRDA）协议连接到Oracle数据库。DRDA是一种独立于数据库的协议。它仅提供Oracle数据库提供的功能。用于DRDA的Oracle数据库提供程序使现有DB2应用程序客户能够在从DB2服务器迁移时利用其当前的应用程序技术投资。

DB2客户端应用程序

DB2应用程序有两个通用类：

  • 远程DB2应用程序

  • 原生DB2应用程序

远程DB2应用程序

远程应用程序使用 DRDA数据协议与目标服务器数据库通信。协议的体系结构是包含以下内容的客户端/服务器模型的示例， 如图1-1所示 ：

  • 客户组件， DRDA申请请求人（AR）

  • 网络基板，如 TCP / IP网络或 SNA / APPC网络

  • 服务器组件，Application Server（AS）

图1-1 DRDA连接模型

应用程序使用AR与AS进行通信，AS又与数据库进行通信。在此配置中，应用程序间接了解网络，因为AR连接到网络。该应用程序不需要直接了解网络连接。

通常，DRDA AR实现提供可直接调用的API，其可由应用程序编写器（例如ODBC）编码。此API也可以作为语言的一部分调用用于翻译源代码的预处理器嵌入式SQL语句等效嵌入式API调用。这与Oracle的概念类似 OCI API和Oracle Pro * C预处理器产品。在这两种情况下，应用程序都与实际的数据库连接无关;只有特定的API调用才能将应用程序附加到数据库。

在网络中， 客户/服务器架构有跨平台互操作性：客户端和服务器可以在任何支持的计算机平台上运行。例如，IBM制造可用的DB2数据库服务器产品 AS / 400， z / OS中， VM， VSE， Linux，几个 Unix平台，以及微软Windows。IBM还生产客户，例如IBM DB2 Connect产品，可在多个平台上使用。这种安排使客户端能够与多个服务器通信，并且可以容易地重定向到不同的服务器，该服务器可以位于相同或不同的远程主机上。

远程应用程序的示例包括：

  • 基于ODBC的应用程序

  • 的Java / 基于JDBC的应用程序

  • DB2数据库，用于远程数据库到数据库的连接

  • DB2 Connect，用于重定向本机应用程序

  • 使用多种可用AR实现之一的自定义应用程序

ODBC-和支持JDBC的应用程序可能是重定向使用Oracle数据库提供程序用于DRDA，只需对应用程序本身进行很少或不进行任何更改。

IBM的DB2产品有对DRDA的本机支持，其中DB2可以是请求者，服务器或两者。本书仅讨论DB2所属的场景 Oracle服务器的请求者。

原生DB2应用程序

Oracle Database Provider for DRDA支持本机应用程序，但它们需要现有的DB2数据库服务器来重定向网络。这是因为本机DB2应用程序与DB2服务器更紧密地交织在一起。此类应用程序使用本地和专有API直接与特定DB2服务器通信。虽然此类应用程序无法直接连接到其他数据库，但它们可以使用远程节点连接机制间接连接到远程数据库。如图1-2所示 。

图1-2本机应用程序远程连接模型

这不是一种理想的方法，因为仅使用完整的DB2服务器来提供对本机应用程序的远程访问是非常昂贵的。这是因为许可模式的“每处理器”结构，以及磁盘和内存占用。

完整的多处理器DB2服务器的一个更有吸引力的替代方法是使用本地应用程序，例如 DB2 Connect，提供远程连接。在这种情况下，应用程序的访问权限可以通过DRDA转换为网络连接， 如图1-3所示 。

图1-3 DB2 Connect连接模型的DB2 Connect替换

在某些情况下，无法使用备用本机应用程序启用程序替换DB2数据库服务器。此类申请包括以下内容：

  • CICS DB2连接的应用程序 Z / OS

  • DB2 / 400本机应用程序

  • DB2 for z / OS本机应用程序

  • DB2 for Linux中， Unix，和视窗原生应用

在这些情况下，应用程序可以通过连接代理，通过本地DB2数据库服务器。虽然这不是一种理想的方法，但它减少了对DB2服务器产品的投资。如果应用程序不使用DB2数据库产品，那么DB2服务器的数量可能会减少到作为应用程序代理所必需的DB2实例。

适用于DRDA的Oracle数据库提供程序的使用方案

部分远程DB2应用程序和本机DB2应用程序描述了针对DRDA的Oracle数据库提供程序的一些可能的使用方案。显然，两者兼而有之远程和本机应用程序可能是重新定位以通过Oracle数据库提供程序为DRDA使用Oracle数据库。

因为用于DRDA的Oracle数据库提供程序是一种网络解决方案网络基础设施应具有足够的过剩容量，以在重新定位本机应用程序时适应增加的负载。在远程应用程序的情况下，客户端和服务器之间的数据流不会发生显着变化。

虽然大多数场景都围绕标准AR（通常由IBM提供），但有一种情况是Application Server取代 Oracle Access Manager for AS / 400已停产。Access Manager是一种客户端产品，它使本机DB2 / 400应用程序能够连接到Oracle数据库，就像它是远程DB2数据库一样。Access Manager在。上运行 AS / 400作为 DB2 / 400 API插件并使用OCI方法连接到Oracle数据库， 如图1-4所示 。
