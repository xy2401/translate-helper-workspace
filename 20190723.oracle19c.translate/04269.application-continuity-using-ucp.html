10.1使用UCP确保应用程序连续性的概述

Oracle Database 12 c Release 1（12.1）介绍了应用程序连续性功能，提供通用的，独立于应用程序的基础结构。应用程序连续性可以在发生计划内或计划外中断后，从应用程序角度恢复工作，该中断可能与修复，配置更改或修补程序应用程序之后的系统，通信或硬件有关。

要使用Application Continuity，必须先配置数据源。之后，使用以下两个功能之一在您的应用程序中使用通用连接池（UCP）实现应用程序连续性：

  • 使用连接标签实现应用程序连续性

  • 使用连接初始化回调实现应用程序连续性

10.2配置应用程序连续性的数据源

要在启用池的数据源上使用“应用程序连续性”功能，应用程序必须在oracle.ucp.jdbc.上进行以下调用oracle.ucp.jdbc.PoolDataSource接口：

// pds是一个PoolDataSource pds.setConnectionFactoryClassName（“oracle.jdbc.replay。OracleDataSourceImpl“）;

始终连接到服务，而不是使用SID。使用SID语法进行连接时，不支持应用程序连续性。

在针对Oracle Real Application Clusters（Oracle RAC）或Data Guard运行时，应用程序还应启用快速连接故障转移（FCF），如以下代码段所示：

pds.setFastConnectionFailoverEnabled（真）;

10.3对应用程序连续性使用连接标签

连接标签使应用程序可以将任意名称/值对附加到连接。应用程序可以从连接池请求与所需标签的连接。

连接标签设置每个连接请求的初始状态。如果应用程序使用连接标记或标记连接的好处，则应为Application Continuity注册标记回调，以在故障转移时初始化干净连接。

每次Application Continuity从底层数据源获取新连接时，都会执行标记回调。回调在正常连接签出期间以及重放期间执行。因此，在重放期间完全重新创建在运行时创建的状态。初始化必须是幂等的。

只要事务在回调调用结束时完成（提交或回滚），回调就可以执行事务。应用程序连续性重复在回调实现中编码的任何操作，包括此类事务。如果在执行UCP标记回调期间发生中断，则应用程序连续性可以作为重放尝试的一部分多次执行回调。同样，回调操作必须是幂等的。

10.4使用连接初始化回调应用程序连续性

如果应用程序因无法更改而无法使用连接标记，则会为此类应用程序提供连接初始化回调。

注册时，每次从池中借用连接时以及在可恢复错误后每次成功重新连接时都会执行初始化回调。
