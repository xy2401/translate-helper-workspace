Oracle ACFS和文件访问和管理安全性

Oracle ACFS支持Linux环境的传统Unix风格文件访问控制类（用户，组，其他）和Windows安全模型，包括Windows平台的文件访问控制列表（ACL）。

大多数Oracle ACFS管理操作由具有Linux环境的root或Oracle ASM管理权限的用户以及Windows平台上具有Windows管理权限的用户执行。任何系统用户都可以访问文件系统的常规Oracle ACFS信息。

为了支持Oracle ACFS管理，Oracle建议将Oracle ASM管理员角色授予root特权用户，因为许多常见的Oracle ACFS文件系统管理任务（包括mount，umount，fsck，driver load和driver unload）都是root特权操作。Oracle ASM管理员可以执行不需要root权限的其他特权Oracle ACFS文件系统操作。如果未向root特权用户授予Oracle ASM管理员角色，则可以使用norootsuid和nodev mount选项限制对Oracle ACFS文件系统的访问。

为具有安全基础结构功能的Oracle ACFS文件系统提供了额外的细粒度访问控制。

Oracle ACFS和Grid Infrastructure安装

Oracle Grid Infrastructure包括Oracle Clusterware，Oracle ASM，Oracle ACFS，Oracle ADVM和驱动程序资源软件组件，它们使用Oracle Universal Installation（OUI）工具安装到Grid Infrastructure主目录中。

Oracle ACFS配置

在Grid Infrastructure安装和可操作的Oracle Clusterware之后，您可以使用Oracle ASM Configuration Assistant（ASMCA）启动Oracle ASM实例并创建Oracle ASM磁盘组，Oracle ADVM卷和Oracle ACFS文件系统。或者，可以使用SQL * Plus和ASMCMD命令行工具创建Oracle ASM磁盘组和Oracle ADVM卷。可以使用操作系统命令行工具创建文件系统。

Oracle ACFS文件系统使用基于Oracle ADVM的操作系统存储设备进行配置，这些设备是在创建Oracle ADVM动态卷文件后自动创建的。创建卷文件及其关联的卷设备文件后，可以创建文件系统并将其绑定到该操作系统存储设备。创建之后，可以安装Oracle ACFS文件系统，之后授权用户和执行文件和文件系统操作的应用程序可以访问它。

Oracle ACFS基于资源的管理摘要

以下列表提供了Oracle ACFS基于资源的管理的摘要。

  • 启动Oracle ASM实例时，将动态加载Oracle ACFS，Oracle内核服务（OKS）和Oracle ADVM驱动程序。

    • Oracle ACFS

      此驱动程序处理所有Oracle ACFS文件和目录操作。

    • Oracle ADVM

      此驱动程序为Oracle ADVM卷文件提供块设备服务，文件系统使用这些文件来创建文件系统。

    • Oracle内核服务驱动程序（OKS）

      此驱动程序为Oracle ACFS和Oracle ADVM提供内存分配，同步原语和分布式锁定服务的可移植驱动程序服务。

    驱动程序作为单个资源集进行管理。有关其他信息，请参阅“ Oracle ACFS驱动程序资源管理 ”和“ Oracle ACFS驱动程序命令 ” 。

  • 创建卷时，Oracle ADVM会创建名为ora.的资源ora.DISKGROUP . VOLUME .advm 。此资源通常通过Oracle ASM的透明高可用性调用进行管理，无需用户交互。但是，用户可以选择使用SRVCTL命令界面来启动和停止卷，以及在Oracle ASM重新启动后控制卷的默认状态。这在大型集群或Oracle Flex ASM集群中尤其有用，因为可以对其他节点上的卷进行操作。

    此外，Oracle Clusterware堆栈中的其他资源可以使用这些Oracle ADVM资源来维护依赖关系链。依赖链确保程序运行所需的资源可用。例如，如果资源正在监视备份到Oracle ADVM卷的备份应用程序，则备份应用程序需要确保它在其START和STOP依赖关系列表中指定了Oracle ADVM卷资源。由于Oracle ADVM卷资源将启用卷，因此可确保在备份开始之前卷可用。

  • Oracle ACFS文件系统可以使用Oracle ACFS或Oracle Clusterware命令行工具手动挂载或卸载，也可以根据Oracle Clusterware资源操作自动挂载或卸载。

    例如，托管Oracle数据库主目录的文件系统在关联的Oracle数据库资源的依赖关系列表中命名，这样在数据库资源上发出启动会导致挂载依赖的Oracle ACFS托管数据库主文件系统。

    Oracle ACFS文件系统资源提供以下操作：

    • 安装

      在START操作期间，资源将文件系统安装在资源中配置的路径上。Oracle ACFS文件系统资源要求Oracle ASM堆栈的所有组件都处于活动状态（卷设备，ASM），并确保它们在尝试装入之前处于活动状态。

    • UNMOUNT

      在STOP操作期间，资源尝试卸载文件系统。

  • Oracle为Oracle Highly Available NFS提供了两种资源类型。有关更多信息，请参阅Oracle ACFS NAS最高可用性扩展 。

与所有Oracle Clusterware资源一样，这些资源通过监视底层设备，文件系统或驱动程序来提供高可用性，以确保对象保持可用。如果底层对象变得不可用，则每个资源都会尝试再次使基础对象可用。

高可用性操作

以下是高可用性资源的操作：

  • Oracle ACFS资源

    此资源尝试卸载文件系统。卸载成功后，资源将重新装入文件系统，使文件系统再次可用。如果在卸载期间进程在文件系统上处于活动状态，则资源将标识并终止这些进程。

  • Oracle ADVM资源

    此资源尝试禁用任何卷设备，然后重新启用卷设备。此时，任何已配置的Oracle ACFS资源都可以重新装入文件系统。如果在此期间卷上的进程处于活动状态，则资源将标识并终止进程。

创建Oracle ACFS资源

可以使用以下方法创建Oracle ACFS资源：

  • Oracle ASM Configuration Assistant（ASMCA）提供了一个公开最常用功能的GUI。在所有情况下，创建文件系统资源都不会格式化基础文件系统。尝试启动资源需要用户手动或使用ASMCA格式化文件系统。

  • SRVCTL提供了一个高度灵活的命令行实用程序，用于通过filesystem对象创建Oracle ACFS文件系统资源。通过此机制创建的Oracle ACFS资源可以访问完整的功能集，包括服务器池。

  • acfsutil命令提供了使用registry对象创建Oracle ACFS文件系统资源的替代方法。通过此方法创建的Oracle ACFS资源可以访问一组有限的选项。

SRVCTL和acfsutil命令之间的区别是：

  • 通过SRVCTL创建并指定服务器池或节点列表的Oracle ACFS资源仅安装在其中一个节点上。 （节点本地）

  • 通过SRVCTL创建的Oracle ACFS资源可以利用Oracle Server Pools。

  • 通过acfsutil命令创建的Oracle ACFS资源和指定节点列表将安装在所有列出的节点上。 （节点本地）

  • 通过acfsutil命令创建的Oracle ACFS资源是在AUTOSTART设置为ALWAYS情况下创建的。

  • 通过SRVCTL创建的Oracle ACFS资源允许高级应用程序ID功能。使用此功能可以由管理员设置资源类型。设置类型后，其他资源可以依赖于此类型，允许使用不同的节点本地文件系统来实现每个节点的依赖性。在一个简化的示例中，这将允许管理员在集群的每个节点上的/log目录中安装不同的设备，并且能够运行Apache资源。Apache资源将在其资源依赖关系结构中指定新类型，而不是指定单个资源。

  • 通过SRVCTL创建的Oracle ACFS资源可以指定其他AUTOSTART参数。这些参数可用于防止资源在堆栈启动时启动，始终强制启动资源，或仅在以前运行时启动资源。

  • 通过SRVCTL创建的Oracle ACFS资源可以访问加速器卷等功能。

SRVCTL和acfsutil命令的常见元素是：

  • 用户

    这是可以对资源执行操作的其他用户。默认情况下，您必须是root用户才能启动和停止Oracle ACFS资源。

  • 选项

    这些是在资源启动时应该用于挂载文件系统的挂载选项。

节点本地或群集文件系统

创建Oracle ACFS文件系统资源时，可以创建节点本地文件系统或创建群集范围的文件系统。

  • 节点本地

    此文件系统类型仅限于可以装入的节点数。根据它是使用SRVCTL还是acfsutil命令创建的，它可能只安装在一个节点，节点子集或所有已配置的节点上。在某些情况下，这可能与完整群集配置相同，但如果将新节点添加到群集，则不会自动挂载文件系统而不修改允许节点列表。

  • 群集范围

    这种类型的文件系统安装在集群的所有节点上，没有例外。将新成员添加到群集后，文件系统将自动在其上可用。某些配置（例如Oracle数据库或Oracle HANFS）需要此类资源。

监视Oracle ACFS资源

与所有Oracle Clusterware资源类似，Oracle ACFS资源使您可以监视系统的状态。您可以使用以下命令执行此监视：

  • 使用SRVCTL命令

    运行命令srvctl status filesystem或srvctl status volume ，该命令的输出将报告文件系统是否已装入或是否已启用卷，以及哪些节点为true。

  • 使用CRSCTL命令

    运行crsctl status resource命令时，将报告每个可用资源的ONLINE状态，无论是通过已安装的文件系统还是已启用的卷。无论是通过卸载的文件系统还是禁用的卷，都会报告每个不可用资源的OFFLINE状态。其他状态可能会显示在此输出的STATUS字段中。

停止Oracle ACFS资源

您可以使用以下方法停止Oracle ACFS文件系统资源：

  • 您可以停止整个Oracle Clusterware堆栈。停止Oracle Clusterware堆栈时，将自动停止所有Oracle ACFS资源。

  • 要停止单个资源，可以将SRVCTL管理命令与Oracle ACFS文件系统或卷对象一起使用。如果有其他资源依赖于您尝试停止的资源，则该命令可能需要-force选项。

  • 您可以使用手动操作，例如在文件系统上运行unmount或使用ASMCMD或SQL * Plus命令手动停止卷。在这种情况下，Oracle ACFS资源会自动转换为OFFLINE状态。

对于某些情况，非Oracle Grid Infrastructure使用挂载点可以防止在内核中卸载和禁用卷。例如：

  • 网络文件系统（NFS）

  • Samba /通用Internet文件系统（CIFS）

如果前面的任一示例都反映了您的情况，请确保在启动堆栈关闭，文件系统卸载或卷禁用之前停止使用该功能。

此外，某些用户空间进程和系统进程可能会以阻止Oracle Grid Infrastructure堆栈在修补程序或升级期间关闭的方式使用文件系统或卷设备。如果出现此问题，请使用lsof和fuser命令（Linux和UNIX）或handle和wmic命令（Windows）来标识在Oracle ACFS文件系统和Oracle ADVM卷上处于活动状态的进程。要确保这些进程不再处于活动状态，请卸载所有Oracle ACFS文件系统或Oracle ADVM卷并发出Oracle Clusterware关闭。否则，Oracle Clusterware关闭期间可能会出现与Oracle ACFS文件系统或Oracle ADVM卷上的活动相关的错误，从而阻止Oracle Clusterware成功关闭。

Oracle ACFS资源限制

Oracle ACFS具有以下资源限制：

  • 所有Oracle ACFS资源都需要root权限才能创建。

  • 所有Oracle ACFS资源都需要删除root权限。

  • 所有Oracle ACFS文件系统资源都需要root权限才能执行操作，例如启动和停止资源，但可以配置为允许其他用户（如数据库用户）执行此操作。在这种情况下，必须使用root用户配置资源。

  • 所有Oracle ADVM卷资源都允许ASMADMIN用户对其执行操作。

  • 所有Oracle ACFS资源仅在Oracle RAC模式下可用。Oracle Restart配置不支持Oracle ACFS资源。有关Oracle ACFS和Oracle Restart的更多信息，请参阅Oracle ACFS和Oracle Restart 。

Oracle ACFS和卸载或关闭操作

在关闭Oracle ASM实例或卸载磁盘组之前，卸载使用Oracle ADVM卷设备文件配置的任何活动文件系统非常重要。卸载文件系统后，将删除对Oracle ASM文件的所有打开引用，并且可以卸除关联的磁盘组或关闭实例。

如果在关联的Oracle ACFS处于活动状态时强制关闭或失败Oracle ASM实例或磁盘组，则文件系统将处于脱机错误状态。如果当前在Oracle ADVM卷文件上安装了任何文件系统，则在不先卸载这些文件系统的情况下，不应使用SHUTDOWN ABORT命令终止Oracle ASM实例。否则，应用程序会遇到IO错误，并且在Oracle ASM存储被隔离之前，可能无法将终止时写入的Oracle ACFS用户数据和元数据刷新到存储。如果无法卸载文件系统，则应在发出SHUTDOWN ABORT操作之前运行两个sync （1）命令将缓存的文件系统数据和元数据刷新到持久存储。

任何后续尝试访问脱机文件系统都会返回错误。从该状态恢复文件系统需要卸载并重新安装Oracle ACFS文件系统。卸载活动文件系统，即使是脱机活动文件系统，也需要使用文件系统停止所有应用程序，包括任何shell引用。例如，先前的更改目录（ cd ）到文件系统目录中。Linux fuser或lsof命令或Windows handle有关进程和打开文件的命令列表信息。

Oracle ACFS安全性

Oracle ACFS安全性为Oracle ACFS文件系统提供基于域的安全性，使您能够创建域以指定用户和组的安全策略，以确定对文件系统对象的访问。

此安全功能在操作系统提供的访问控制之上提供了更细粒度的访问控制。Oracle ACFS安全性可以使用加密功能来保护存储在Oracle ACFS文件系统中的域安全文件的内容。

Oracle ACFS安全性使用领域，规则，规则集和命令规则来实施安全策略。

  • Oracle ACFS安全领域是一组文件或目录，其安全供用户或一组用户访问。领域使用规则集定义，规则集包含应用细粒度访问控制的规则组。Oracle ACFS安全领域也可用作容器以启用加密。

  • Oracle ACFS安全规则是布尔表达式，根据规则所基于的系统参数求值为true或false。

  • Oracle ACFS规则集是规则集合。规则集根据规则集包含的规则的评估值计算为TRUE或FALSE 。

  • Oracle ACFS命令规则是文件系统操作与规则集的关联。例如，文件系统创建，删除或重命名操作与规则集的关联。命令规则与Oracle ACFS领域相关联。

必须将现有操作系统用户指定为第一个Oracle ACFS安全管理员，并且必须将现有操作系统组指定为安全管理员admin组。安全管理员必须是指定安全组的成员。可以将其他用户指定为安全管理员。Oracle ACFS安全管理员可以基于每个领域管理Oracle ACFS文件系统的加密。Oracle ACFS安全管理员使用安全领域密码进行安全操作身份验证，而不是用户的操作系统密码。

第一个安全管理员是在Oracle ACFS安全性初始化期间使用由root用户运行的acfsutil sec init命令创建的。创建第一个安全管理员后，将为管理员分配一个可由管理员更改的密码。每次安全管理员运行acfsutil sec命令时，系统都会提示管理员输入安全密码。管理员的安全领域密码存储在安全性初始化过程中创建的钱包中。此钱包位于Oracle Cluster Registry（OCR）中。

将记录Oracle ACFS安全性的审核和诊断数据。日志文件包括已运行的acfsutil命令，安全性或系统管理员权限的使用以及域检查授权失败等运行时故障等信息。

只有在文件系统上未启用审核时，才会将审核事件（例如，域创建或已启用加密）写入这些日志文件。如果启用了审核，则会将这些事件写入审核跟踪。无论是否启用审计，始终会将与安全性和加密相关的诊断消息写入sec- hostname_fsid .log文件。

日志将写入以下文件：

  • mount_point /.Security /realm/logs/sec- hostname_fsid .log

    该目录使用acfsutil sec prepare命令创建，并受Oracle ACFS安全性保护。

  • GRID_HOME /log/ hostname /acfs/security/acfssec.log

    记录到此文件的消息适用于与特定文件系统无关的命令，例如acfsutil sec init 。该目录在安装期间创建，并由root用户拥有。

当活动日志文件增长到预定义的最大大小（10 MB）时，文件将自动移动到log_file_name .bak ，通知管理员，并继续记录常规日志文件名。通知管理员后，管理员必须存档并删除log_file_name .bak文件。如果活动日志文件增长到最大大小并且log_file_name .bak文件存在，则日志记录将停止，直到删除备份文件。删除备份日志文件后，日志记录将自动重新启动。

Oracle ACFS安全性可保护以下对象免受未经授权的访问：

  • 领域安全目录和用户文件

    目录和文件驻留在由Oracle ACFS安全性保护的文件系统上。

  • Oracle ACFS安全目录（ mount_point /.Security ）及其内容

    安全目录包含纯文本格式的日志文件和XML格式的安全元数据备份文件。Oracle ACFS安全性生成的日志文件只能由有效的Oracle ACFS安全管理员访问。

  • Oracle ACFS安全对象

    这些对象是用于管理Oracle ACFS安全性的安全领域，规则和规则集。

访问Oracle ACFS文件系统安全领域中的文件必须由安全领域和基础操作系统权限授权，例如Linux上的（所有者，组，其他）权限和Windows上的访问控制列表（ACL）。首先检查对域安全文件的每次访问以获得安全领域授权。如果访问权限由安全领域授权，则底层操作系统访问控制检查将检查对文件的访问。如果两个检查都通过，则允许访问领域安全文件。

使用Oracle ACFS安全性时请注意以下事项：

  • Oracle ACFS安全性不对网络上发送的数据提供任何保护。

  • 受领域保护的文件的副本不受域保护，除非该副本是在受安全领域保护的目录中进行的。

    某些应用程序（如vi编辑器）会在修改文件时重新创建文件。修改后的文件将另存为临时文件，原始文件将被删除，临时文件将以原始文件名作为目标名称进行复制。此过程会创建一个新文件。如果新文件是在受域保护的目录中创建的，则域的安全策略也适用于新文件。如果未在受域保护的目录中创建新文件，则新文件不受域保护。如果您计划复制受领域保护的文件，则应确保父目录也受安全领域保护。

    安全策略也适用于在受域保护的目录中创建的任何临时文件。

要在Linux上使用Oracle ACFS安全功能，必须将ASM和ADVM的磁盘组兼容性属性设置为11.2.0.2或更高版本。要在Windows上使用Oracle ACFS安全功能，必须将ASM和ADVM的磁盘组兼容性属性设置为11.2.0.3或更高版本。 。

Oracle ACFS加密

Oracle ACFS加密使您可以加密存储在磁盘上的数据（静态数据）。

加密功能以加密格式保护Oracle ACFS文件系统中的数据，以防止在数据丢失或被盗时未经授权使用数据。加密和非加密文件都可以存在于同一Oracle ACFS文件系统中。

某些加密功能需要系统管理员权限。此功能包含用于启动，设置和重新配置加密的命令。

系统管理员和Oracle ACFS安全管理员可以启动加密操作。此外，非特权用户可以为他们拥有的文件启动加密。

Oracle ACFS加密提供两种类型的加密密钥：

  • 文件加密密钥

    这是文件的密钥，用于加密文件中的数据。

  • 卷加密密钥

    这是文件系统的密钥，用于加密文件加密密钥。

您必须首先创建加密密钥库，然后指定文件系统级加密参数并标识目录。如果用户具有访问文件数据的适当权限，则用户无需额外的步骤即可读取加密文件。

Oracle ACFS加密支持Oracle Cluster Registry（OCR）和Oracle Key Vault作为密钥库。OCR和Oracle Key Vault都可以在同一个集群中使用。但是，单个文件系统使用OCR或Oracle Key Vault作为密钥存储，但不能同时使用两者。Oracle Key Vault目前仅适用于Linux上的文件系统。

如果您使用OCR作为密钥存储区，则应在创建或更新加密密钥后备份OCR，以确保存在包含文件系统的所有卷加密密钥（VEK）的OCR备份。

Oracle ACFS加密可保护存储在辅助存储上的数据免受被盗或直接访问存储介质的威胁。数据永远不会以明文形式写入二级存储。即使物理存储被盗，如果没有加密密钥也无法访问存储的数据。加密密钥永远不会以明文形式存储。密钥要么被混淆，要么使用用户提供的密码加密。

Oracle ACFS安全管理员可以基于每个领域管理加密参数。将文件置于域安全性之后，不允许对该文件执行文件级加密操作。即使领域安全性允许文件所有者或root用户打开文件，也会阻止文件级加密操作。受域保护的文件的加密完全由Oracle ACFS安全管理员管理，该管理员可以在安全领域级别启用和禁用文件加密。

将目录添加到安全领域后，目录中创建的所有文件都将继承领域级加密参数，而不是目录或文件系统级参数。从最后一个安全领域中删除文件时，将对文件进行加密或解密以匹配文件系统级加密状态。如果已使用安全领域参数加密，则不会对文件进行重新加密以匹配文件系统级参数。

系统管理员无法在文件系统或文件级别重新加密域安全文件。要确保使用最新的卷加密密钥（VEK）加密所有域安全文件，必须先从所有域中删除加密，然后重新启用加密。此操作会使用最新的VEK重新加密所有文件。

记录Oracle ACFS加密的审计和诊断数据。日志文件包括已运行的acfsutil命令，安全性或系统管理员权限的使用以及运行时故障等信息。日志将写入以下文件：

  • mount_point /.Security /encryption/logs/encr- hostname_fsid .log

    如果启用了安全性，则使用acfsutil encr set命令创建目录，并受Oracle ACFS安全性保护。

Oracle ACFS压缩

在指定的Oracle ACFS文件系统上启用Oracle ACFS压缩以获取通用文件。Oracle数据库文件不支持Oracle ACFS压缩。

在应用程序写入文件后，异步执行缓存IO压缩。在文件系统上启用压缩时，不压缩现有文件，仅压缩新创建的文件。禁用压缩时，压缩文件不会解压缩。压缩文件与压缩单元相关联，压缩算法在此单元上运行。默认单位大小目前为32 K. lzo是默认的压缩算法，是目前唯一支持的压缩算法。

acfsutil compress命令使用acfsutil compress on和acfsutil compress off设置和重置文件系统的compress off 。要显示压缩操作的压缩状态和有效性，请使用acfsutil compress info命令。acfsutil info fs和acfsutil info file命令已得到增强，可报告Oracle ACFS压缩状态。

压缩文件比非压缩文件消耗更少的磁盘空间。但是，对于使用该文件的应用程序，报告的大小等于未压缩的文件大小，而不是较小的压缩大小。某些实用程序（如ls -l ）会报告文件的未压缩大小。诸如du ， acfsutil compress info和acfsutil info file实用程序报告压缩文件的实际磁盘分配。

请注意以下有关Oracle ACFS压缩的信息。

  • 用于保存数据库文件的Oracle ACFS文件系统不支持Oracle ACFS压缩。相反，请使用Oracle Advanced Compression。

  • 压缩文件不支持环回安装。这包括供Oracle ACFS远程服务使用的文件。如果环回设备与压缩文件相关联，则对环回设备的读写操作将失败。

  • 环回设备可以与已启用压缩的Oracle ACFS文件系统上的未压缩文件相关联。

  • 对于Oracle Grid Infrastructure 12 c发行版2（12.2.0.1），Linux和AIX上支持Oracle ACFS压缩。

  • Oracle ACFS压缩仅支持从Oracle Grid Infrastructure 12c第2版（12.2.0.1）开始提供的基于Oracle ACFS快照的复制。

  • ADVM磁盘组兼容性必须设置为12.2或更高。

关于Oracle ACFS审核

Oracle ACFS安全性和加密都是审计源，Oracle ACFS审计管理器可以启用和禁用这些源。这些源由于执行Oracle ACFS安全性或加密命令而生成事件。

Oracle ACFS安全管理员可以在领域级别启用审计，以便还可以审计安全违规和授权，以及启用安全审计以审计安全管理员执行的所有事件。必须先启用Oracle ACFS安全源，然后才能使用Oracle ACFS域安全审核。

设置领域审核策略以审核所有命令规则的所有授权和违规可以使审计跟踪快速增加到其最大大小。管理员应仔细调整审计级别以满足其要求，并注意审计策略生成更详细的审计输出需要额外的主动监视和管理，例如归档和清除审计跟踪和审计跟踪备份文件。

除了生成文件系统审计源之外，Oracle ACFS审计还允许在每个领域单独设置细粒度审计策略。Oracle ACFS审核功能为审核库收集器提供了基础架构，可将数据导入Oracle Audit Vault和Database Firewall。收集器与Oracle ACFS分开，用作将Oracle ACFS审核数据导入Audit Vault Server的方法。

审计源的配置和管理职责分为Oracle ACFS审计经理和Oracle ACFS审计员角色。系统管理员有权在Oracle ACFS审计管理器和Oracle ACFS审计员操作系统（OS）组之间添加和删除用户。

Oracle ACFS审计经理可以访问审计源的内容，并可以读取审计数据;但是，审计经理无法修改审计来源。整个集群中的Oracle ACFS审计管理器集是相同的。

Oracle ACFS审计员负责查看和分析审计源的内容，例如向Oracle ACFS审计经理指示哪些记录已经过分析和存档，并且可以安全清除。Oracle ACFS审核员应该是系统中唯一可以访问审核源内容的用户。Oracle ACFS审核员没有删除或清除审核记录所需的权限。整个集群中的Oracle ACFS审核员集是相同的。

审计归档过程将审计跟踪日志文件（.log）重命名为审计跟踪备份文件（.log.bak）并生成XML文件，该文件可由Audit Vault Server导入。Audit Vault Server仅具有对审计跟踪目录的读访问权限，并在此情况下充当审计员。在审计库服务器中导入XML文件中的数据后，审计员功能会将审计跟踪备份文件标记为已read ，然后审计管理器可以执行清除以删除审计跟踪备份文件和XML文件。

要为Oracle ACFS文件系统配置审核，请运行acfsutil audit init命令以初始化Oracle ACFS的审核，然后运行acfsutil audit enable以在指定的文件系统上启用Oracle ACFS加密或安全性的审核。

审计跟踪文件

审计跟踪文件由一组审计记录组成。每个审计记录代表一个事件。审计跟踪文件位于mount_point /.Security/audit目录。

Oracle ACFS审计生成的审计跟踪文件可用于以下内容：

  • 使用文本查看工具由Oracle ACFS审核员进行手动审核

  • 导入Oracle Audit Vault和Database Firewall

  • 可以解析和导入审计源的第三方产品

审计跟踪文件由审计记录组成。有几种不同类型的审计记录，每种记录代表一种独特的事件类型，并包含与诊断事件相关的不同信息。事件类型是：

  • 文件访问事件

  • 特权使用事件

  • 评估结果事件

在审计跟踪文件中输入的审计记录字段的组合取决于事件类型。

每条记录都作为一组字段名称和值写入审计跟踪文件。根据记录的类型，字段的数量和类型可能会有所不同。字段由名称和值对组成，格式为字段 名称 ： value ，后跟行尾字符。

审计跟踪文件中可以出现的审计记录字段在以下列表中描述。括号中的字符串是出现在审计跟踪日志文件中的字段名称。

  • 时间戳（ Timestamp ）：事件发生的时间，始终以UTC指定。时间戳的格式为： MM/DD/YYYY HH:MM:SS UTC

  • 事件代码（ Event ）：标识事件类型的代码。有关评估结果代码的列表，请参阅“ 文件访问事件 ”和“ 权限使用事件 ” 。

  • 来源（ Source ）： Oracle ACFS

  • 用户标识（ User ）：触发事件的用户。在Linux平台上，这是用户ID，在Windows上，这是用户SID。

  • 组标识（ Group ）：触发事件的用户的主要组。在Linux平台上，这是用户主要组的ID，在Windows上，这是用户主要组的SID。

  • 进程标识（ Process ）：当前进程ID。

  • 主机名（ Host ）：记录事件的主机。

  • 应用程序名称（ Application ）：当前进程的应用程序名称。

  • 领域名称（ Realm ）：被侵犯的领域的名称，或者被授权并且正在保护文件的领域。

  • 文件名（ File ）：用户正在访问的文件名。

  • 评估结果（ Evaluation Result ）：该字段包含有关执行的命令结果的信息。有关评估结果代码列表，请参阅“ 评估结果事件 ” 。

  • 文件系统标识（ FileSystem-ID ）：

  • 消息（ Message ）：消息字段包含有关所执行命令及其结果的信息。

Oracle ACFS复制

Oracle ACFS基于快照的复制可以将Oracle ACFS文件系统通过网络复制到远程站点，从而为文件系统提供灾难恢复功能。

Oracle ACFS复制可以将装入的文件系统或装入的文件系统的快照指定为复制存储 位置 。Oracle ACFS复制关系的源Oracle ACFS位置称为主要位置。Oracle ACFS复制关系的目标Oracle ACFS位置称为备用位置。

注意：

  • Oracle ACFS复制功能仅支持每个主要位置的一个备用位置。

  • 只要复制处于活动状态，备用位置就是只读位置。可以创建备用数据库的读写快照。

  • 运行Linux，Solaris或AIX的主站点可以复制到运行支持Oracle ACFS的任何操作系统的备用站点。运行Windows的主站点只能复制到运行Windows的备用站点。

    请注意，在Oracle ACFS复制的任何跨平台使用中都不会修改应用程序数据。Oracle ACFS复制仅在不同操作系统之间传输时确保其元数据的有效性。

  • 主站点和备用站点应运行相同版本的Oracle Grid Infrastructure软件。升级站点时，请先更新备用站点。

  • Oracle Restart不支持Oracle ACFS复制。

  • 包含复制位置的备用文件系统不支持Oracle Key Vault密钥库。

  • 无法在包含复制位置的主文件系统上撤消Oracle ACFS加密。

    您无法撤消具有活动快照的文件系统上的加密。如果要在包含活动复制位置的主文件系统上撤消加密，请首先终止复制。复制停止后，再撤消加密并再次启动复制。

站点可以托管主要和备用位置。例如，如果存在群集站点A和B ，则可以将站点A托管的主文件系统复制到站点B上的备用快照。同时，站点B托管的主快照可以复制到站点A的备用文件系统。但是，Oracle ACFS文件系统或快照不能同时用作主要和备用位置。

Oracle ACFS基于快照的复制通过记录主要位置的快照来运行。将初始快照传输到备用位置后，通过在主要连续快照到备用位置之间传输更改来继续复制。这些复制操作可以以恒定 模式发生（一旦前一个操作完成就启动新操作），或者可以按固定间隔进行调度。此复制解决方案本质上是异步的。

Oracle ACFS复制最初使用主站点上的快照功能来外部化初始快照的内容，以及稍后两个指定快照之间的差异。结果称为快照复制流。然后，复制过程使用备用站点上的快照功能将此流应用于备用位置，从而创建主要位置的副本。

在主服务器上，因为复制通过比较连续快照来工作，所以在托管主服务器的站点上有足够的可用磁盘空间来包含每个快照中记录的主服务器版本以及当前主要内容，这一点至关重要。此外，必须始终有足够的空间来创建所需的快照。不再需要时，将删除每个复制快照。

在备用数据库上，将在每个复制操作结束时创建备份快照。此快照记录备用数据库的最新一致内容，如果在当前复制操作期间发生永久性中断，则可用于恢复这些内容。完成以下复制操作后，将删除每个备份快照，因此必须始终可以创建备份快照。此外，快照中捕获的备用版本和当前备用内容必须存在足够的空间。

您应确保主文件系统和备用文件系统的磁盘空间不足。如果任一文件系统的可用存储空间不足，则应通过从文件系统或任何存在的读写快照中删除文件来扩展文件系统或释放空间。您还可以配置自动调整大小以避免空间不足。

如果主文件系统空间不足并且您决定通过删除文件来释放空间，那么您应该只删除未复制的文件。已复制的文件已存储在快照挂起传输到备用文件系统中，并且不会被删除。您可以删除不是由复制创建的任何Oracle ACFS快照。

Oracle ACFS复制使用ssh实用程序作为主群集和备用群集之间的传输。要启用ssh的自动使用，复制需要配置两种密钥 。必须在启用复制的每个节点上提供这些密钥。

  • 在每个群集中的每个节点上，系统管理员用户（非Windows系统上的用户root或Windows上的本地SYSTEM ）必须为另一个群集中的每个节点存储主机 密钥 。

  • 在每个群集中的每个节点上，指定的非特权用户（ 应用 用户 ）必须具有为root或本地SYSTEM存储的公钥，该公钥被授权以该节点上的应用 用户身份登录。通常，此公钥被定义为另一个集群的节点上的根（或本地SYSTEM ）的公钥/私钥对的一部分。

这些密钥是必需的，因为在主要主机上运行的复制必须能够使用主机密钥对要向其发送数据的备用主机进行身份验证。此外，在主要主机上运行的复制必须能够作为备用主机上的应用 用户使用用户密钥登录以更新备用位置。此外，复制还能够反转主要和备用角色所扮演的角色。要使此角色转换操作成功，主要和备用主机需要配置相同类型的ssh密钥。有关更多信息，请参阅配置ssh以与Oracle ACFS复制一起使用 。

请注意，Windows上本身不提供ssh 。有关所需密钥及其配置方式的更多信息，以及有关在Windows上安装和配置ssh更多信息，请参阅配置Oracle ACFS基于快照的复制 。

  • 有足够的网络带宽来支持主要和备用位置之间的复制。

  • 托管主要和备用位置的站点的配置使备用文件系统能够跟上主要位置的更改速率。

  • 已正确配置ssh主机密钥和用户密钥。

Oracle ACFS标记

Oracle ACFS标记为一组文件分配一个公共命名属性。

Oracle ACFS Replication可以使用此标记选择具有唯一标记名称的文件，以便复制到其他远程集群站点。标记选项可避免复制整个Oracle ACFS文件系统。

Oracle ACFS使用扩展属性实现标记。默认情况下，某些编辑工具和备份实用程序不保留原始文件的扩展属性;你必须设置一个特定的开关。以下列表描述了一些常用实用程序的必要要求和开关设置，以确保在原始文件上保留Oracle ACFS标记名称。

  • cp命令需要标志来保留标记名称。

    在Linux上安装coreutils库（版本coreutils-5.97-23.el5_4.1.src.rpm或coreutils-5.97-23.el5_4.2.x86_64.rpm或更高版本）以安装支持扩展属性保存的cp命令版本使用--preserve=xattr开关和支持扩展属性保留的mv命令，无需任何开关。

    cp不保留分配给符号链接文件的标记名称。

    保存文件和目录上的标记名称所需的cp开关是：

    • Linux： - --preserve=xattr

    • Solaris： -@

    • AIX： -U

    • Windows：无需切换

    • Windows：不可用

  • emacs要求将copy backup-by-copying选项设置为非零值，以保留原始文件名上的标记名称而不是备份副本。必须将此选项添加到.emacs文件中。

  • pax文件传输实用程序需要标志来保留标记名称。

    • Linux： pax不保留标记名称

    • Solaris： -@需要保留或恢复文件和目录的标记名称，但不保留符号链接文件的标记名称

    • AIX：不可用

    • Windows：不可用

具有审计，加密和安全性的Oracle ACFS复制

可以在已配置复制的Oracle ACFS文件系统上启用审计，加密和基于域的安全功能。使用与主文件系统相同的审核，安全性或加密策略来保护复制的备用文件系统。对于此复制环境，主文件系统和备用文件系统必须都是12.1或更高版本的安装。有关Oracle ACFS复制的更多信息，请参阅“ Oracle ACFS复制 ” 。

为确保成功复制，备用文件系统必须是通用文件系统，而不包含审核，加密或安全元数据。Oracle ACFS不支持使用曾经具有安全性或加密功能，然后删除安全性或加密的备用文件系统。本节列出了Oracle ACFS审核，加密和安全性必须满足的其他条件。

请注意以下有关Oracle ACFS审核文件系统的信息：

  • 在复制启用审计的文件系统或审核复制的文件系统之前，必须在备用文件系统上初始化审计。

  • 主文件系统上存在的审核策略将复制到备用文件系统，并且在主文件系统上执行的所有策略操作都将在备用文件系统上生成。

  • 备用文件系统上存在两组审计跟踪。来自主文件系统的路径将作为普通文件复制到备用文件系统。文件系统活动可以在备用文件系统上生成事件，这些事件记录在备用文件系统的审计跟踪中。审计跟踪名称有助于区分两组路径，因为它们包含主机名和FSID。

请注意以下有关Oracle ACFS加密文件系统的信息：

  • 主文件系统上的加密文件在备用文件系统上保持加密，具有相同的密钥和加密参数（算法和密钥长度）。

  • 在主文件系统上完成的加密操作将在备用文件系统上重放 - 开，关和重新密钥。

  • 可以在复制文件系统之前或之后启用加密。在任何一种情况下，如果由于尚未在备用文件系统上运行acfsutil encr init则在备用文件系统上透明地创建加密钱包。

  • 备用文件系统不支持受密码保护的钱包。如果要用作备用文件系统的站点上已存在PKCS钱包，则管理员必须使用acfsutil keystore migrate命令将所有密钥传输到SSO钱包。

请注意以下有关Oracle ACFS安全文件系统的信息：

  • 在复制启用安全性的文件系统之前，应初始化备用文件系统以确保安全性。

  • 规则，规则集和领域被复制到备用文件系统，并且备用文件系统上存在相同的策略。在策略和文件保护方面，备用文件系统完全相同。

  • 可以在启用安全性的文件系统上启用复制，也可以在复制的文件系统上启用安全性。作为安全准备的一部分，还在备用文件系统上启用了安全性。

  • 在文件系统上一起进行安全性和复制不需要任何额外的用户干预或其他步骤。

  • 可以在备用文件系统上设置一组不同的安全管理员或安全管理员组。

Oracle ACFS加速器卷

使用加速器卷可以通过减少访问和更新Oracle ACFS元数据的时间来提高性能。您应该在磁盘组上创建加速器卷，其存储空间明显快于主卷存储。例如，可以使用固态磁盘（SSD）存储。使用ASMCMD volcreate命令创建Oracle ADVM卷。有关volcreate命令的信息，请参阅volcreate 。

建议的加速器容量大小取决于工作负载。对于具有多个扩展区的文件尤其有用，尤其是在经常更新扩展区元数据的情况下。您可以使用acfsutil info file命令查看文件范围的报告。数据库文件通常具有许多范围，并且在使用Oracle ACFS快照时，范围元数据会经常更新。受益于加速器的工作负载是压缩文件系统。

如果Oracle ACFS无法在加速器上为关键元数据分配空间，则该元数据将存储在主卷上。根据元数据更新的频率，它可能会对性能产生不成比例的影响。如果慢速元数据与快速元数据在同一事务中写入，则慢速元数据会降低整个操作的性能。

建议的起始加速器大小最小为文件系统大小的0.6％。如果正在使用许多快照代表数据库工作负载的几个时间点，则建议每个快照额外增加0.4％。例如，具有5个快照的文件系统可能需要加速器，其大小为主卷大小的2.6％。可以将acfsutil size配置为根据需要随主卷自动增长加速器。加速器以64兆字节为单位增加。加速器体积的最小尺寸为256 M. mkfs要求初始加速器尺寸至少为主体积的0.4％

加速器卷链接到使用mkfs命令指定的主卷。安装文件系统时，仅指定主卷。如果在装入卷的文件系统之后加速器卷因任何原因而无法访问，则文件系统将脱机。只有一个存储加速器卷可以与Oracle ACFS文件系统关联。将加速器卷与文件系统关联后，无法将该卷与文件系统解除关联。

可以使用mkfs命令的-a选项在Linux环境中创建加速器卷。要在Linux上创建加速器卷， COMPATIBLE.ADVM的值必须至少为12.2 。有关用于管理加速器卷的命令的信息，请参阅：

Oracle ACFS高可用性网络文件系统

用于Oracle Grid Infrastructure的高可用性网络文件系统（HANFS）通过在高可用虚拟IP（HAVIP）上公开NFS导出并使用Oracle Clusterware代理确保VIP和NFS导出，提供NFS v2，v3或v4导出路径的不间断服务总是在线。虽然基本NFS支持文件锁定，但HANFS不支持NFS文件锁定。

注意：

  • 此功能依赖于主机上可用的有效NFS服务器配置。在尝试使用Oracle ACFS NFS导出功能之前，必须配置NFS服务器。

  • Windows上不提供此功能。

  • Oracle Restart配置不支持此功能。

  • 在为其创建至少一个文件系统导出资源之前，无法启动HAVIP。

要为Oracle Grid Infrastructure设置高可用性NFS，请执行以下步骤：

  1. 添加并注册新的HAVIP资源。

    例如：

    #srvctl add havip -id hrexports -address my_havip_name

    在该示例中， my_havip_name在域名服务器（DNS）中映射到VIP地址，并在安装文件系统时由客户端系统使用。

    srvctl add havip的初始处理确保：

    • 使用的地址是静态的，而不是动态的

    • 任何DNS名称仅解析为一个主机，而不是循环多个DNS解析

    • 网络资源和提供的IP地址和已解析的名称位于同一子网中

    • 该名称未使用

    SRVCTL使用id创建适当的HAVIP名称，确保它是唯一的。作为最后的验证步骤，SRVCTL确保ora.net#.network的network资源（如果提供）存在。这一步后，SRVCTL增加了一个新havip型ora.havip.type用的名称ora. id .havip 。在此示例中，名称为ora. hrexports .havip 。

    下一个SRVCTL修改HAVIP启动依赖性，例如active dispersion ;设置停止依赖项;并确保适当设置description属性（如果提供）。

  2. 创建共享Oracle ACFS文件系统。

    适用于Oracle Grid Infrastructure的高可用性NFS仅适用于为群集范围的可访问性配置的Oracle ACFS文件系统，并且不支持为在群集节点的特定子集上进行访问而配置的Oracle ACFS文件系统。非Oracle ACFS文件系统不支持高可用性NFS。

  3. 注册Oracle ACFS文件系统。

    例如：

    $ srvctl add filesystem -device / dev / asm / d1volume1-295 -volume VOLUME1 \ -diskgroup HR_DATA -mountpath / oracle / cluster1 / acfs1

  4. 创建Oracle ACFS文件系统导出资源。

    例如：

    #srvctl add exportfs -id hrexports -path / oracle / cluster1 / acfs1 -name hrexport1

    创建文件系统导出资源后，您可以启动在步骤1中创建的HAVIP，以使用srvctl start havip命令导出文件系统。

    NFS安装选项FSID将添加到任何导出选项，利用基础Oracle ACFS文件系统的FSID以及唯一标识符。此FSID选项提供节点之间的可靠故障转移，并允许使用快照装入。

    已配置导出的默认安装和导出选项是NFS服务器的默认值。

    完全限定的相对路径将转换为绝对路径。不完全限定的相对路径不被接受为导出路径。

    VIP会尝试根据可用的文件系统和其他正在运行的VIP找到要运行的最佳服务器，但这种分散只发生在CSS成员更改事件期间，例如加入或离开群集的节点。

    注意：

    不建议单独启动和停止出口;应通过HAVIP的启动和停止操作提供此功能。

    当HAVIP未运行时，导出可以存在于不同的节点上。启动关联的HAVIP后，导出将聚集在单个节点上。

    正在使用而HAVIP运行提高NFS错误停止出口客户estale ，必须卸除并重新装入文件系统。

    在客户端上安装HANFS导出的文件系统时，建议使用以下CLIENT挂载选项：

    硬，INTR，重传= 10000

Oracle ACFS HANFS现在支持带有NFS锁的HANFS NFS v4。此功能仅适用于特定的操作系统（OS）平台。要激活此功能，必须在安装Oracle Grid Infrastructure软件后执行其他步骤。请注意，完成这些步骤后，群集的OS NFS服务器功能由Oracle Clusterware堆栈管理。此外，某些OS NFS配置文件的位置将从其默认位置移动到指定的Oracle ACFS文件系统。

  • 检查是否支持此平台：支持acfshanfs supported

＃grid_home / bin / acfshanfs addnode -nfsv4lock -volume volume_device

卷使用Oracle ACFS文件系统格式化，并安装在指定的Oracle ACFS集群件挂载点上例如在Linux上：

/ dev / asm / nfs-81 / var / lib / nfs类型acfs（rw）

群集域中的Oracle ACFS远程服务

Oracle ACFS为Oracle数据库成员集群上的本机Oracle ACFS功能提供Oracle ACFS远程服务，而无需连接本地存储（InDirect存储成员集群）。通过在Oracle域服务集群（DSC）上使用Oracle ACFS部署，Oracle ACFS远程服务可用于Oracle应用程序集群和Oracle数据库成员集群，从而实现灵活的基于文件系统的应用程序和数据库部署。

支持的Oracle ACFS远程服务系统

使用Oracle ACFS远程服务的Oracle数据库成员集群的要求包括：

  • Oracle ACFS远程服务需要Open-iSCSI iSCSI启动器才能进行传输配置。建议使用最新版本的Open-iSCSI。

  • Oracle ACFS远程服务仅支持Linux。支持的版本是RedHat Enterprise Linux 6和7以及Oracle Linux 6和7。

使用Oracle ACFS远程服务的DSC的要求是：

  • Oracle ACFS远程服务仅支持Oracle Linux 7和RedHat Linux 7。

  • Oracle ACFS远程服务需要UEK3，UEK4或更高版本的内核。

  • Oracle ACFS远程服务至少需要以下软件包的最低版本：

    • 蟒蛇，2.7.5-34.0.1.el7.x86_64

    • 蟒蛇-rtslib-2.1.fb57-3.el7.noarch

    • 蟒蛇-configshell-1.1.fb18-1.el7.noarch

    • targetcli-2.1.fb41-3.el7.noarch

    • 蟒蛇 - 六1.9.0-2.el7.noarch

您可以运行acfsremote supported命令来确定Oracle ACFS远程服务是否支持您当前的系统。

Oracle ACFS远程服务的设置和最佳实践

Oracle ACFS远程服务的设置包括以下内容：

  1. 创建运输

    在DSC上创建传输，以提供使Oracle数据库成员集群能够访问导出的方法。Oracle ACFS远程服务支持iSCSI传输，创建导出并将iSCSI目标分配给特定成员群集节点启动器（IQN），并仅为该指定的成员群集节点提供访问。

    确保每个Oracle数据库成员集群都有自己的独立传输。您可以通过使用VLAN或多个网络来实现此目的。此设置提供通过传输网络隔离网络活动，类似于iSCSI网络。对于完全冗余，应配置每个成员集群的多个传输，从而减轻传输链路故障。此外，每个Oracle数据库成员集群的多个传输都支持多路径。可以将Oracle数据库成员集群配置为共享单个传输。请注意， advmutil transport命令不会强制隔离。

  2. 创建存储库

    在DSC上为Oracle ACFS远程服务使用的存储创建存储库。此存储库作为单个Oracle ACFS文件系统启动。要获得最佳设置，请创建单独的Oracle ASM Flex冗余磁盘组以供Oracle ACFS远程服务使用。该组应包含一个高冗余Oracle ADVM卷和一个用于每个Oracle数据库成员集群的常规冗余卷。此配置代表隔离，管理和空间效率之间的权衡。通过此设置，存储库与卷级别的其他成员集群隔离，并且只有一个磁盘组可供管理。每个成员集群使用一个磁盘组可以实现进一步的隔离。通过使用普通和高冗余卷实现空间效率，使关键文件能够存储在高冗余量中。

    创建卷后，使用Oracle ACFS文件系统对其进行格式化。然后运行srvctl add filesystem以在堆栈启动期间为automount注册文件系统。automount确保在启动堆栈后立即可以使用存储库，以便Oracle Database成员集群可以利用其存储。可以配置存储库以进行自动调整大小，如果文件系统空间不足，则可以为Oracle ACFS文件系统调整大小。如果启用了自动调整大小，请确保单个成员集群不使用所有可用磁盘组空间。

    您可以使用ASMCA GUI工具执行管理Oracle ADVM卷的步骤。

  3. 创建导出

    在域服务集群（DSC）上运行advmutil export命令，以在DSC上的存储库中创建导出。如果成员集群正在运行，则成员集群将获取新导出。如果Oracle Clusterware堆栈未在成员集群上运行，或者未引导成员集群，则成员集群将在下次启动Oracle Clusterware堆栈时查找并使用新导出。这通常发生在成员集群的引导时。由于导出目标的方式，导出可能需要几分钟才能在成员群集上显示为卷。导出在成员集群上反映相同的冗余保护级别（高，正常，外部），作为存储在域服务集群上的基础Oracle ACFS远程服务存储库。

  4. 使用成员群集上的导出

    添加导出后，成员集群会在延迟后在/dev/asm/目录中显示新的卷设备。可以在客户端 ADVM 设备字段下的DSC上的advmutil export list输出中找到与此卷设备关联的名称。可以使用Oracle ACFS命令格式化此卷设备，使用标准Oracle ACFS命令挂载，并使用srvctl命令将其添加到Oracle Clusterware堆栈。导出是可以托管Oracle ACFS文件系统的完整Oracle ADVM卷，可以使用任何标准Oracle ACFS命令进行管理。有关创建Oracle ACFS文件系统的简单示例，请参阅创建Oracle ACFS文件系统 。

Oracle ACFS远程服务的性能调优

Oracle ACFS远程服务的性能调整包括以下内容：

  • Oracle ACFS远程服务使用iSCSI作为标准传输，因此可以在您的环境中使用iSCSI所需的任何适用的特定于站点的调整。通常，这包括标准专用传输网络并限制该网络上的主机数量。作为配置Oracle ACFS远程服务导出的组件，Oracle ACFS远程服务为每个导出启用更长的超时和更高的队列深度。

  • 合并Oracle ACFS远程服务的存储时，必须考虑系统调整注意事项。

    • 您正在有效地将多个客户端成员集群的IO聚合到DSC中，DSC充当这些成员集群的存储阵列。确保DSC上的存储设置可以处理所有连接成员集群的聚合负载。

    • DSC需要适当的网络带宽来服务所有连接的客户端。与iSCSI调整一起，可能需要额外的网络抽头来提供额外的网络带宽来处理所有成员集群请求。

    • iSCSI的IO服务利用CPU，因此确保足够的可用CPU是免费的是高优先级。通常，为Oracle ACFS远程服务提供服务的DSC应该为其他活动运行最小的处理负载，并且应该被视为实用程序集群，以满足其他人的需求。

Oracle ACFS远程服务的Oracle Clusterware更改

Oracle Clusterware for Oracle ACFS远程服务的更改包括：

在域服务群集上，Oracle ACFS远程服务启用一种新的资源类型。此类型是HAVIP（ transport_vip类型）的变体。该资源使iSCSI Transport VIP可以在DSC中移动。多个Oracle数据库成员集群传输可以共享一个VIP，但这不是推荐的做法。当使用多个传输VIP时，它们会尝试在整个群集中分散，以减少单个节点上网络的过载。创建传输VIP时，它依赖于网络以及托管存储库的任何Oracle ACFS资源。后续存储库创建会修改依赖项以包含新存储库。此修改可确保存储库和传输VIP在群集服务重定位和恢复期间协同工作以实现可用性。使用advmutil transport自动创建和删除传输VIP，但可以使用srvctl * havip命令查看和管理传输VIP。

在InDirect存储成员群集中，Oracle ACFS远程服务提供了四种新资源： ora.acfsrm ， ora.ccmb ， ora.acfsremote和ora.acfsrd 。这些资源协同工作以管理成员集群上的Oracle ACFS远程服务。可以使用srvctl * acfsrapps命令查看这些资源的状态。在成员集群上，没有卷资源。卷高可用性由Oracle ACFS远程服务功能集管理，因此Oracle ACFS资源不依赖于ADVM卷资源。相反，Oracle ACFS资源依赖于新资源。使用srvctl * filesystem命令创建Oracle ACFS资源。

支持Oracle ACFS远程服务的命令

除了Cluster Domain中的Oracle ACFS Oracle ACFS命令行工具中描述的Oracle ACFS命令外 ，以下命令还支持Oracle ACFS远程服务功能。

  • acfsutil compat

    Oracle ACFS主要使用磁盘组的兼容性设置来确定要启用的功能。在InDirect存储成员集群中，没有关联的磁盘组来控制此兼容性设置。每个文件系统使用acfsutil compat命令控制其功能。有关acfsutil compat信息，请参阅acfsutil compat get和acfsutil compat set 。

  • acfsutil info file

    对针对配置为由Oracle ACFS远程服务使用的文件运行acfsutil info file ，该文件将显示其他文件属性Remote 。这个属性将文件标识为Oracle ACFS远程服务正在使用的文件，并且在文件解除配置之前，执行一些操作;例如截断，修改属性和使用普通文件系统命令删除;是不允许的。此操作可防止使用此文件中断对成员集群的服务。有关acfsutil info file info ，请参阅acfsutil信息文件 。

  • acfsutil info storage

    在Oracle数据库成员集群上运行时， acfsutil info storage不会显示有关域服务集群（DSC）磁盘组和文件系统的信息。仅显示有关成员集群本身的信息。由于DSC为多个成员群集提供服务，因此此操作可防止过度共享信息。有关acfsutil info storage info ，请参阅acfsutil信息存储 。

  • crsctl create member_cluster_configuration

    acfs选项可用于crsctl create member_cluster_configuration命令。使用Oracle ACFS远程服务创建成员集群需要acfs选项。
