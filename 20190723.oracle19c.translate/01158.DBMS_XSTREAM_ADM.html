208.1 DBMS_XSTREAM_ADM概述

该软件包提供了用于配置出站服务器的接口，这些服务器将数据库更改从Oracle数据库传输到其他系该软件包还提供了用于配置入站服务器的接口，这些服务器将数据库更改从其他系统流式传输到Oracle数

在这两种情况下，数据库更改都封装在逻辑更改记录（LCR）中。此外，其他系统可以是Oracle系统或非Oracle系统，例如非Oracle数据库或文件系统。

XStream出站服务器可以使用C或Java以编程方式从Oracle数据库中流出LCR。收到LCR后，另一个系统可以任何自定义方式处理它们。例如，另一个系统可以将LCR的内容保存到文件中，通过XStream入站服务器将LCR发送到Oracle数据库，或者生成SQL语句并在任何Oracle或非Oracle数据库上执行它们。

XStream入站服务器接受来自另一个系统的LCR，并将它们应用于Oracle数据库，或使用应用处理程序以自定义方式处理它们。

XStream可以在多租户容器数据库（CDB）中使用。CDB是Oracle数据库，包括零个，一个或多个用户创建的可插拔数据库（PDB）。

208.2 DBMS_XSTREAM_ADM安全模型

要确保运行此程序包中子程序的用户具有必要的权限，请配置XStream管理员并在使用此程序包时以XStream管理员身份进行连接。

管理员执行以下任何操作时，必须授予DBA角色：

  • 连接时作为与出站服务器的已配置连接用户不同的用户运行ADD_OUTBOUND过程

  • 运行ALTER_OUTBOUND过程以更改捕获进程的捕获用户或出站服务器的连接用户

  • 运行CREATE_OUTBOUND过程，因为此过程会创建捕获过程

  • 运行ALTER_INBOUND过程以更改入站服务器的应用用户

  • 连接时以连接的用户身份运行ADD_INBOUND过程，该用户与配置的入站服务器的应用用户不同

当管理员不需要执行上述任务时，不需要DBA角色。

208.4.1 ADD_COLUMN过程

此过程添加或删除基于声明规则的转换，该转换将列添加到满足指定规则的行逻辑更改记录（行LCR）。

对于在指定规则的计算结果为TRUE时执行的转换，规则必须位于XStream客户端的正规则集中。XStream客户端包括捕获进程，传播和应用进程。

此过程已超载。column_value和column_function参数是互斥的。

注意：

  • ADD_COLUMN转换不能添加以下数据类型的列： BLOB ， CLOB ， NCLOB ， BFILE ， LONG ， LONG RAW ， ROWID ，用户定义类型（包括对象类型， REF ，varrays，嵌套表）和Oracle提供的类型（包括任何类型，XML类型，空间类型和媒体类型）。

  • 声明性转换只能转换行LCR。因此，运行此过程时必须指定DML规则。如果指定了DDL规则，则该过程会引发错误。

208.4.2 ADD_GLOBAL_PROPAGATION_RULES过程

此过程要么将全局规则添加到传播的正规则集，要么将全局规则添加到传播的否定规则集，如果不存在则创建指定的传播。

此过程已超载。此过程的一个版本包含两个OUT参数，另一个版本不包含。

句法

DBMS_XSTREAM_ADM.ADD_GLOBAL_PROPAGATION_RULES（streams_name IN VARCHAR2 DEFAULT NULL，source_queue_name IN VARCHAR2，destination_queue_name IN VARCHAR2，include_dml IN BOOLEAN DEFAULT TRUE，include_ddl IN BOOLEAN DEFAULT FALSE，include_tagged_lcr IN BOOLEAN DEFAULT FALSE，source_database IN VARCHAR2 DEFAULT NULL，dml_rule_name OUT VARCHAR2，ddl_rule_name OUT VARCHAR2， inclusion_rule IN BOOLEAN DEFAULT TRUE，and_condition IN VARCHAR2 DEFAULT NULL，queue_to_queue IN BOOLEAN DEFAULT NULL）; DBMS_XSTREAM_ADM.ADD_GLOBAL_PROPAGATION_RULES（streams_name IN VARCHAR2 DEFAULT NULL，source_queue_name IN VARCHAR2，destination_queue_name IN VARCHAR2，include_dml IN BOOLEAN DEFAULT TRUE，include_ddl IN BOOLEAN DEFAULT FALSE，include_tagged_lcr IN BOOLEAN DEFAULT FALSE，source_database IN VARCHAR2 DEFAULT NULL，inclusion_rule IN BOOLEAN DEFAULT TRUE，and_condition IN VARCHAR2 DEFAULT NULL，queue_to_queue IN BOOLEAN DEFAULT NULL）;

参数

表208-3 ADD_GLOBAL_PROPAGATION_RULES过程参数

参数 描述

streams_name

传播的名称。不要指定所有者。

如果指定的传播不存在，则该过程自动创建它。

如果对于同一源队列和目标队列（包括数据库链接）存在NULL和传播，则该过程使用此传播。

如果对于相同的源队列和目标队列（包括数据库链接）存在NULL且不存在传播，则该过程将使用系统生成的名称自动创建传播。

source_queue_name

源队列的名称，指定为[ schema_name .] queue_name 。当前数据库必须包含源队列，并且队列必须是ANYDATA类型。

例如，要在strmadmin架构中指定名为streams_queue的源队列，请为此参数输入strmadmin.streams_queue 。

如果未指定架构，则当前用户是默认用户。

destination_queue_name

如果目标队列位于远程数据库中，则指定为[ schema_name .] queue_name [ @dblink_name ]的目标队列的名称（包括数据库链接）。队列必须是ANYDATA类型。

例如，要在strmadmin架构中指定名为streams_queue的目标队列并使用名为dbs2.net的数据库链接， strmadmin.streams_queue@dbs2.net为此参数输入strmadmin.streams_queue@dbs2.net 。

如果未指定架构，则当前用户是默认用户。

如果省略数据库链接，则该过程使用当前数据库的全局名称，并且源队列和目标队列必须位于同一数据库中。

注意：不允许使用连接限定符。

include_dml

如果为TRUE ，则该过程为DML更改创建规则。如果为FALSE ，则该过程不会创建DML规则。不允许NULL 。

include_ddl

如果为TRUE ，则该过程为DDL更改创建规则。如果为FALSE ，则该过程不会创建DDL规则。不允许NULL 。

include_tagged_lcr

如果为TRUE ，则该过程不会向生成的规则添加有关Oracle Replication标记的条件。因此，无论逻辑更改记录（LCR）是否具有非NULL标记，这些规则都可以评估为TRUE 。如果将规则添加到传播的正规则集中，则始终考虑LCR用于传播，而不管它是否具有非NULL标记。如果将规则添加到正规则集，则将此参数设置为TRUE适用于数据库的完整（例如，备用）副本。如果将规则添加到传播的否定规则集中，则是否丢弃LCR不依赖于LCR的标记。

如果为FALSE ，则该过程会向每个生成的规则添加一个条件，该条件仅在LCR具有NULL Oracle Replication标记时才会将规则评估为TRUE 。如果将规则添加到传播的正规则集中，则仅当LCR包含NULL标记时才考虑将LCR用于传播。如果将规则添加到正规则集，则将此参数设置为FALSE可能适用于update-anywhere配置，以避免将更改发送回其源数据库。如果将规则添加到传播的否定规则集中，则只有在具有NULL标记的情况下才能丢弃LCR。

通常，如果inclusion_rule参数设置为FALSE ，则为此参数指定TRUE 。

source_database

源数据库的全局名称。源数据库是发生更改的位置。如果为NULL ，则该过程不会将有关源数据库的条件添加到生成的规则中。

如果您不包含域名，则该过程会自动将其附加到数据库名称。例如，如果指定DBS1且域为.NET ，则该过程DBS1.NET自动指定DBS1.NET 。

Oracle建议您为传播规则指定源数据库。

dml_rule_name

如果include_dml为TRUE ，则此参数包含DML规则名称。

如果include_dml为FALSE ，则此参数包含NULL 。

ddl_rule_name

如果include_ddl为TRUE ，则此参数包含DDL规则名称。

如果include_ddl为FALSE ，则此参数包含NULL 。

inclusion_rule

如果inclusion_rule为TRUE ，则该过程将规则添加到传播的正规则集。

如果inclusion_rule为FALSE ，则该过程将规则添加到传播的否定规则集。

在任何一种情况下，如果规则集不存在，系统都会创建规则集。

and_condition

如果为非NULL ，则使用AND子句以下列方式将指定条件附加到系统生成的规则条件：

（ system_condition ）AND（ and_condition ）

指定条件中的变量必须为:lcr 。例如，要指定仅当Oracle Replication标记是十六进制等效于'02' ，过程生成的全局规则才会求值为TRUE ，请指定以下条件：

：lcr.get_tag（）= HEXTORAW（''''''）

:lcr指定条件中的:lcr将转换为:dml或:ddl ，具体取决于生成的规则。如果要指定依赖于LCR类型（行或DDL）的LCR成员子程序，请确保该过程仅生成相应的规则。

特别是，如果你指定一个LCR成员子程序，这只是为行的LCR有效，然后指定TRUE为include_dml参数和FALSE为include_ddl参数。如果指定的LCR成员子程序只能用于DDL LCR的是有效的，然后指定FALSE为include_dml参数和TRUE为include_ddl参数。

另请参阅： 逻辑更改记录类型

queue_to_queue

如果为TRUE或NULL ，则此过程创建的新传播是队列传播的队列。队列到队列传播始终具有自己的传播作业，并且当目标队列是Oracle Real Application Clusters（Oracle RAC）数据库中的缓冲队列时，使用服务进行自动故障转移。

如果为FALSE ，则此过程创建的新传播是队列到dblink的传播。队列到数据链路的传播可以与使用相同数据库链接的其他传播共享传播作业，并且不支持Oracle RAC环境中的自动故障转移。

该过程无法将队列更改为现有传播的队列属性。如果存在指定的传播，则该过程对于每个设置按以下方式运行：

  • 如果为TRUE且指定的传播不是队列传播的队列，则该过程会引发错误。

  • 如果FALSE和指定的传播是队列传播的队列，则该过程会引发错误。

  • 如果为NULL ，则该过程不会将队列更改为传播的队列属性。

使用说明

此过程使用当前用户配置传播。在特定源队列和目标队列之间只允许一次传播。

此过程将规则添加到XStream客户端规则集。

208.4.4 ADD_OUTBOUND过程

此过程创建一个XStream出站服务器，使LCR从指定的队列中出列。出站服务器将LCR流出到XStream客户端应用程序。

此过程既不创建捕获过程也不创建队列。要创建出站服务器，捕获进程和具有一个过程调用的队列，请使用CREATE_OUTBOUND过程 。

要单独创建捕获过程，请使用以下某个包：

  • DBMS_XSTREAM_ADM

  • DBMS_CAPTURE_ADM

要单独创建队列，请使用DBMS_XSTREAM_ADM包中的SET_UP_QUEUE过程。

此过程已超载。一个table_names参数是类型VARCHAR2 ，另一个table_names参数是类型DBMS_UTILITY.UNCL_ARRAY 。此外，一个schema_names参数是类型VARCHAR2 ，另一个schema_names参数是类型DBMS_UTILITY.UNCL_ARRAY 。这些参数使您能够以不同的方式输入表和模式的列表，并且是互斥的。

注意：

  • 客户端应用程序可以创建多个会话。每个会话只能连接到一个出站服务器，每个出站服务器一次只能为一个会话服务。但是，不同的客户端应用程序会话可以连接到不同的出站服务器有关附加到出站服务器的信息，请参阅“ Oracle调用接口程序员指南”和“ Oracle数据库XStream Java API参考 ”。

  • 此过程启用它创建的出站服务器。

  • 从Oracle Database 11g第2版（11.2.0.2）开始， capture_name ， start_scn和start_time参数包含在此过程中。

此过程始终更改指定的出站服务器。当满足以下任一条件时，此过程还可以更改出站服务器的捕获过程：

  • 捕获过程由此包中的CREATE_OUTBOUND过程创建。

  • 捕获进程使用的队列由CREATE_OUTBOUND过程创建。

要检查此过程是否可以更改出站服务器的捕获过程，请在ALL_XSTREAM_OUTBOUND视图中查询CAPTURE_NAME列。当捕获过程的名称出现在此视图的CAPTURE_NAME列中时， ALTER_OUTBOUND过程可以管理捕获过程的规则或更改捕获过程的捕获用户。当CAPTURE_NAME这种观点的列是NULL ，则ALTER_OUTBOUND程序不能管理采集过程。

此过程已超载。一个table_names参数是类型VARCHAR2 ，另一个table_names参数是类型DBMS_UTILITY.UNCL_ARRAY 。此外，一个schema_names参数是类型VARCHAR2 ，另一个schema_names参数是类型DBMS_UTILITY.UNCL_ARRAY 。这些参数使您能够以不同方式输入表和模式列表，并且是互斥的。

注意：

从Oracle Database 11g第2版（11.2.0.2）开始，此过程中包含start_scn和start_time参数。

句法

DBMS_XSTREAM_ADM.ALTER_OUTBOUND（server_name IN VARCHAR2，table_names IN DBMS_UTILITY.UNCL_ARRAY，schema_names IN DBMS_UTILITY.UNCL_ARRAY，add IN BOOLEAN DEFAULT TRUE，capture_user IN VARCHAR2 DEFAULT NULL，connect_user IN VARCHR2 DEFAULT NULL，comment IN VARCHAR2 DEFAULT NULL，inclusion_rule IN BOOLEAN DEFAULT TRUE， start_scn IN NUMBER DEFAULT NULL，start_time IN TIMESTAMP DEFAULT NULL，include_dml IN BOOLEAN DEFAULT TRUE，include_ddl IN BOOLEAN DEFAULT FALSE，source_database IN VARCHAR2 DEFAULT NULL，source_container_name IN VARCHAR2 DEFAULT NULL）; DBMS_XSTREAM_ADM.ALTER_OUTBOUND（server_name IN VARCHAR2，table_names IN VARCHAR2 DEFAULT NULL，schema_names IN VARCHAR2 DEFAULT NULL，添加IN BOOLEAN DEFAULT TRUE，capture_user IN VARCHAR2 DEFAULT NULL，connect_user IN VARCHAR2 DEFAULT NULL，comment IN VARCHAR2 DEFAULT NULL，inclusion_rule IN BOOLEAN DEFAULT TRUE， start_scn IN NUMBER DEFAULT NULL，start_time IN TIMESTAMP DEFAULT NULL，include_dml IN BOOLEAN DEFAULT TRUE，include_ddl IN BOOLEAN DEFAULT FALSE，source_database IN VARCHAR2 DEFAULT NULL，source_container_name IN VARCHAR2 DEFAULT NULL）;

208.4.24 MERGE_STREAMS程序

此过程将从一个捕获进程流出的流与从另一个捕获进程流出的流合并。

通常，此过程用于合并使用此程序包中的SPLIT_STREAMS过程拆分的两个流。SPLIT_STREAMS过程在拆分流时克隆原始流的组件。因此，本节中的信息使用以下术语：

  • 拆分之前的流具有原始队列，原始捕获过程和原始传播。

  • 由SPLIT_STREAMS过程拆分的流具有克隆队列，克隆捕获过程和克隆传播。

MERGE_STREAMS_JOB过程调用此过程。MERGE_STREAMS_JOB过程确定流是否在用户指定的合并阈值内，以便可以安全地合并流。如果流不在合并阈值内，则MERGE_STREAMS_JOB过程不执行任何操作。通常，最好运行MERGE_STREAMS_JOB过程，而不是直接运行MERGE_STREAMS过程。

但是，您可以选择在满足以下条件时直接运行MERGE_STREAMS过程：

  • 拆分流的目标处的问题已得到纠正，目标队列可以接受更改。

  • 分割流使用的克隆捕获过程已启动并正在捕获更改。

  • 目标数据库上的应用进程正在应用克隆捕获进程捕获的更改。

  • 该CAPTURE_MESSAGE_CREATE_TIME在GV$XSTREAM_CAPTURE克隆捕获过程的观点已经赶上或几乎赶上了，该CAPTURE_MESSAGE_CREATE_TIME原捕获过程。克隆的捕获过程可能永远无法完全赶上原始捕获过程。因此，当克隆的捕获进程几乎赶上原始捕获进程时，您可以合并拆分流。

MERGE_STREAMS过程执行以下操作：

  1. 停止克隆的捕获过程。

  2. 停止原始捕获过程。

  3. 将克隆的传播复制回原始传播。传播与复制后的原始传播名称相同。

  4. 从这两个SCN值的较低SCN值开始原始捕获过程：

    • 已确认克隆传播的SCN。

    • 传播由原始捕获过程捕获的更改的其他传播的最低确认SCN。

    当原始捕获进程启动时，它可能会重新捕获已捕获的更改，或者它可能捕获已克隆捕获进程已捕获的更改。在任何一种情况下，相关的申请流程都会丢弃他们收到的任何重复变更。

  5. 丢弃克隆的传播。

  6. 删除克隆的捕获过程。

  7. 删除克隆的队列。

此过程确定原始捕获进程和克隆的捕获进程是否在指定的合并阈值内。如果它们在合并阈值内，则此过程运行MERGE_STREAMS过程以合并两个流。

通常，此过程用于合并使用此程序包中的SPLIT_STREAMS过程拆分的两个流。SPLIT_STREAMS过程在拆分流时克隆原始流的组件。因此，本节中的信息使用以下术语：

  • 拆分之前的流具有原始队列，原始捕获过程和原始传播。

  • 由SPLIT_STREAMS过程拆分的流具有克隆队列，克隆捕获过程和克隆传播。

如果在拆分流的SPLIT_STREAMS过程中将auto_merge_threshold参数设置为正数，则合并作业将根据其计划自动运行MERGE_STREAMS_JOB过程。为schedule_name参数指定调度名称，并在自动运行MERGE_STREAMS_JOB过程时为merge_job_name参数指定合并作业名称。合并作业及其计划由SPLIT_STREAMS过程创建。

如果在拆分流的SPLIT_STREAMS过程中将auto_merge_threshold参数设置为NULL或0 （零），则可以手动运行MERGE_STREAMS_JOB过程。在这种情况下，它不会自动运行。

208.4.27 RECOVER_OPERATION程序

此过程为分离和合并操作提供了选项，这些操作因遇到错误而停止。

此过程要么前滚操作，回滚操作，要么清除有关操作的所有元数据。拆分和合并操作可能在XStream Out环境中运行，其中多个出站服务器使用相同的捕获过程。

此过程仅可以使用split_threshold和merge_threshold捕获进程参数设置为非NULL值来执行拆分和合并操作的这些操作，以启用自动拆分和合并。

当操作正在进行时，有关操作的信息存储在以下数据字典视图中：

  • DBA_RECOVERABLE_SCRIPT

  • DBA_RECOVERABLE_SCRIPT_PARAMS

  • DBA_RECOVERABLE_SCRIPT_BLOCKS

  • DBA_RECOVERABLE_SCRIPT_ERRORS

数据字典视图填充在包含捕获过程的数据库中。

操作成功完成后，有关操作的元数据将从DBA_RECOVERABLE_SCRIPT视图移至DBA_RECOVERABLE_SCRIPT_HIST视图。其他视图DBA_RECOVERABLE_SCRIPT_PARAMS ， DBA_RECOVERABLE_SCRIPT_BLOCKS和DBA_RECOVERABLE_SCRIPT_ERRORS保留有关操作的信息，直到30天后自动清除该操作。

当其中一个操作遇到错误并停止时，有关操作的元数据仍保留在这些视图中。在这种情况下，您可以使用RECOVER_OPERATION过程前滚，回滚或清除有关操作的元数据。如果选择前滚操作，则在DBA_RECOVERABLE_SCRIPT_ERRORS之前更正导致DBA_RECOVERABLE_SCRIPT_ERRORS报告的错误的条件。

在包含捕获过程的数据库上运行RECOVER_OPERATION过程。

注意：

要运行RECOVER_OPERATION过程，两个数据库都必须是Oracle Database 10 g第2版或更高版本的数据库。

208.4.39 SPLIT_STREAMS过程

该过程将从捕获过程流出的一个流从捕获过程流出的所有其他流中分离出来。

此过程适用于Oracle Replication环境，其中捕获进程捕获传播到两个或多个目标数据库的更改。当传播的一个目标停止接受捕获的更改时，更改将保留在捕获进程的队列中。队列可能会增长并开始将LCR泄漏到硬盘，从而降低Oracle Replication环境的性能。由于多种原因，目标可能会停止接受更改。例如，目标数据库可能已关闭。

具体而言，此过程执行以下操作：

  1. 在运行捕获进程的数据库中创建新队列。新队列称为克隆队列，因为它是原始流使用的队列的克隆。新队列将由新的克隆捕获进程使用，它将成为新的克隆传播的源队列。

  2. 创建一个新传播，将LCR从步骤1中创建的源队列传播到现有目标队列。新传播称为克隆传播，因为它是原始流使用的传播的克隆。克隆的传播使用与原始传播相同的规则集。

  3. 停止捕获过程。

  4. 查询确认SCN的原始传播。已确认的SCN是应用过程确认的最后一个SCN，它应用传播发送的更改。

  5. 创建一个新的捕获过程。新的捕获过程称为克隆捕获过程，因为它是原始流使用的捕获过程的克隆。该过程将克隆捕获进程的启动SCN设置为查询的已确认SCN的值。克隆的捕获过程使用与原始捕获过程相同的规则集。

  6. 丢弃原始传播。

  7. 启动原始捕获过程，并将启动SCN设置为步骤4中查询的已确认SCN。

  8. 如果auto_merge_threshold参数设置为正数，则创建Oracle Scheduler作业，以根据其计划以设定的时间间隔运行MERGE_STREAMS_JOB过程。当两个流在指定的合并阈值内时， MERGE_STREAMS_JOB过程运行MERGE_STREAMS过程以自动合并流。

SPLIT_STREAMS过程完成运行后，将禁用克隆的捕获过程。解决目标数据库中的问题并且目标队列可以接受更改时，应使用DBMS_CAPTURE_ADM程序包中的START_CAPTURE过程启动克隆捕获过程。

如果原始捕获进程是下游捕获进程，则必须先配置克隆捕获进程，以便在启动克隆捕获进程之前从源数据库中读取重做日志。
