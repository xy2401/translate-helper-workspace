  • UNPROTECTED - 数据库未受保护（这通常发生在主数据库已挂载且未打开时）

SWITCHOVER_STATUS

VARCHAR2（20）

指示是否允许切换：

  • NOT ALLOWED - 在主数据库上，此状态表示没有有效且已启用的备用数据库。在备用数据库上，此状态表示尚未从主数据库收到切换请求。

  • SESSIONS ACTIVE - 数据库具有活动会话。在物理备用数据库上，必须指定WITH SESSION SHUTDOWN SQL子句以在此状态下执行角色转换。在逻辑备用数据库上，可以在此状态下执行角色转换，但在所有当前事务都已提交之前，角色转换将不会完成。

  • SWITCHOVER PENDING - 在物理备用数据库上，此状态表示已从主数据库接收并正在处理切换请求。在此瞬态状态下，物理备用数据库无法切换到主要角色。

  • SWITCHOVER LATENT - 在物理备用数据库上，此状态表示切换请求处于挂起状态，但原始主数据库已切换回主角色。

  • TO PRIMARY - 数据库已准备好切换到主要角色。

  • TO STANDBY - 数据库已准备好切换到物理或逻辑备用角色。

  • TO LOGICAL STANDBY - 数据库已从逻辑备用数据库接收到数据字典，并准备切换到逻辑备用角色。

  • RECOVERY NEEDED - 在物理备用数据库上，此状态表示在数据库切换到主角色之前必须应用其他重做。

  • PREPARING SWITCHOVER - 在主数据库上，此状态表示正在从逻辑备用数据库接收数据字典，以准备切换到逻辑备用角色。在逻辑备用数据库上，此状态表示数据字典已发送到主数据库和其他备用数据库。

  • PREPARING DICTIONARY - 在逻辑备用数据库上，此状态表示正在将数据字典发送到主数据库和其他备用数据库，以准备切换到主要角色。

  • FAILED DESTINATION - 在主数据库上，此状态表示一个或多个备用目标处于错误状态。

  • RESOLVABLE GAP - 在主数据库上，此状态表示一个或多个备用数据库具有可通过从主数据库或其他备用数据库中提取缺少的重做来自动解决的重做间隙。

  • UNRESOLVABLE GAP - 在主数据库上，此状态表示一个或多个备用数据库具有无法通过从主数据库或其他备用数据库获取缺少的重做来自动解决的重做间隙。

  • LOG SWITCH GAP - 在主数据库上，此状态表示由于最近的日志切换，一个或多个备用数据库缺少重做。

DATAGUARD_BROKER

VARCHAR2（8）

Data Guard经纪人信息：

  • ENABLED - 数据库是代理配置的一部分，并且启用了数据库的代理管理

  • DISABLED - 数据库是代理配置的一部分，数据库的代理管理已禁用。如果用户禁用了数据库或配置的代理管理，或者由于角色更改而禁用了代理管理（例如，在故障转移操作后禁用了旧主服务器），则会显示此值。

GUARD_STATUS

VARCHAR2（7）

保护数据不被更改：

  • ALL - 表示阻止SYS以外的所有用户更改数据库中的任何数据。

  • STANDBY - 表示阻止SYS以外的所有用户更改逻辑备用数据库维护的任何数据库对象。

  • NONE - 表示数据库中所有数据的正常安全性。

SUPPLEMENTAL_LOG_DATA_MIN

VARCHAR2（8）

确保LogMiner（以及构建在LogMiner技术上的任何产品）具有足够的信息来支持链式行和各种存储安排，例如集群表：

  • NO - 未启用任何数据库范围的补充日志记录指令。

    在CDB中，值NO表示在CDB中的所有PDB中未启用最小补充日志记录。

  • IMPLICIT - 启用最小补充日志记录，因为启用了主键，唯一键和外键补充日志记录的全部或组合

  • YES - 通过ALTER DATABASE ADD SUPPLEMENTAL LOG DATA语句启用最小补充日志记录。

    在CDB中，值为YES表示在CDB中的所有PDB中启用最小补充日志记录。

另请参阅： Oracle数据库SQL语言参考，以获取有关ALTER DATABASE ADD SUPPLEMENTAL LOG DATA语句的其他信息

SUPPLEMENTAL_LOG_DATA_PK

VARCHAR2（3）

对于具有主键的所有表，指示每当执行更新（ YES ）时是否将主键的所有列放入重做日志中（ NO ）。

当CDB中出现值YES ，表示在CDB中的所有PDB中启用了主键补充日志记录。

当CDB中出现NO值时，查询CDB中每个PDB的DBA_SUPPLEMENTAL_LOGGING视图中的PRIMARY_KEY列，以查看是否在PDB中启用了主键补充日志记录。

另见： Oracle数据库SQL语言参考有关详细信息， ALTER DATABASE ADD SUPPLEMENTAL LOG supplemental_id_key_clause声明

SUPPLEMENTAL_LOG_DATA_UI

VARCHAR2（3）

对于具有唯一键的所有表，指示如果修改了任何唯一键列（ YES ）或不是（ NO ），是否将属于唯一键的所有其他列放入重做日志中。

当CDB中出现值YES ，表示在CDB中的所有PDB中启用了该值。

当CDB中出现NO值时，请在DBA_SUPPLEMENTAL_LOGGING视图中查询CDB中每个PDB的UNIQUE_INDEX列，以查看是否在PDB中启用了唯一列补充日志记录。

另见： Oracle数据库SQL语言参考有关详细信息， ALTER DATABASE ADD SUPPLEMENTAL LOG supplemental_id_key_clause声明

当CDB中出现NO值时，查询ALL_COLUMN中每个PDB的DBA_SUPPLEMENTAL_LOGGING视图中的ALL_COLUMN列，以查看是否在PDB中启用了所有列补充日志记录。

VARCHAR2（19）

PRIMARY_DB_UNIQUE_NAME

VARCHAR2（30）

对于任何备用数据库（物理，逻辑或快照），此列将包含此备用数据库上次从当前重做的主数据库的DB_UNIQUE_NAME 。

如果此备用数据库自上次启动后未收到任何当前重做，则此列将为空。

对于先前已经处于待机状态的主数据库，此列将包含此数据库在充当备用数据库时从其接收当前重做的最后一个主数据库的DB_UNIQUE_NAME 。

对于从未进入待机状态的主数据库，此列将为null。

另请参阅： Oracle Data Guard概念和管理 ，以获取Oracle程序包复制到逻辑备用数据库的Oracle程序包列表
