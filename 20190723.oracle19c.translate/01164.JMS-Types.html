277.1 JMS类型概述

Java消息服务（JMS）是用于访问消息传递系统的众所周知的公共标准接口。Oracle JMS（OJMS）基于Oracle高级队列（AQ）和关系数据库系统（RDBMS）实现JMS。消息作为OJMS特定ADT存储在队列中。Java客户端使用OJMS包对这些消息进行入队，出列和操作。

另一方面，PL / SQL用户使用DBMS_AQ包将JMS消息入队和出列，并在本章中使用成员函数来填充和解释它们。Oracle Database Advanced Queuing为以下JMS ADT提供此类成员函数：

  • 水性$ _jms_map_message

277.3 Java与PL / SQL数据类型

数据类型不会在PL / SQL和Java之间进行一对一映射。

PL / SQL中不存在某些Java类型，例如BYTE和SHORT 。选择PL / SQL类型INT来表示这些类型。如果用于保存Java BYTE或SHORT值的PL / SQL INT值超出Java强制执行的相应范围，则会引发超出范围的错误。

其他Java类型在PL / SQL中具有多个具有不同功能的对应类型。Java String可以由VARCHAR2和CLOB ，但VARCHAR2的最大限制为4000字节。从map，stream和bytes消息类型中检索TEXT数据时，始终返回CLOB 。在更新地图，流和字节消息类型时，用户可以提交VARCHAR2或CLOB 。

类似地，Java BYTE ARRAY可以由RAW和BLOB ， RAW的最大大小为32767。从地图，流和字节消息类型中检索BYTE ARRAY数据时，始终返回BLOB 。更新地图，流和字节消息类型时，用户可以提交RAW或BLOB 。

使用Java存储过程对Oracle数据库高级排队消息进行编码和解码

映射，流，字节和其他消息之间的主要区别在于消息有效负载由JAVA编码为字节流。因此，在PL / SQL中检索和更新这些有效负载需要Oracle JAVA存储过程。

消息有效负载在处理期间存储在两个位置。在PL / SQL端，它作为JMS消息ADT的数据成员存储，在Jserv端，它作为静态变量存储。（Jserv是Oracle数据库中的JVM。）处理有效负载时，首先将有效负载数据转换为Jserv端的静态变量。初始化静态变量后，将对此静态变量执行消息有效内容的所有后续更新。在处理结束时，有效负载数据被刷回PL / SQL端。

Oracle提供了维护Jserv静态变量状态的成员过程，并在调用这些成员过程时强制执行规则。这些程序在以下ADT中：

  • 水性$ _jms_bytes_message

  • 水性$ _jms_map_message

  • 水性$ _jms_stream_message

初始化Jserv静态变量

在进行任何其他调用以处理有效内容数据之前，必须正确初始化Jserv静态变量。这是通过调用prepare或clear_body过程完成的。prepare过程使用PL / SQL ADT中的有效负载数据初始化静态变量，而clear_body将静态变量初始化为空有效负载（空散列表或流）。

注意：

在任何其他调用正确初始化Jserv静态变量之前调用prepare或clear_body过程非常重要。通常这两种方法在开始时调用一次。但是对于一条消息，可以多次调用它们。在没有先调用flush过程的情况下调用这两种方法会消除对消息所做的所有更新。

获取有效负载数据返回PL / SQL

调用flush过程会将对Jserv静态变量所做的更改同步回PL / SQL ADT。如果希望所做的更改反映在ADT有效负载中，则需要flush调用。将更改同步到ADT非常重要，因为重要的是ADT有效负载。

垃圾收集静态变量

clean过程强制静态变量的垃圾收集。它是清理和免费的JVM内存。完成处理消息后立即执行此操作可以避免内存泄漏。

使用消息存储：静态变量集合

Oracle使用一组静态变量来处理Jserv端的消息有效负载，而不是单个静态变量。此集合称为消息存储库。每个映射，字节或流消息类型在一个会话中都有自己的消息存储。

Oracle使用操作ID参数来定位要在消息存储库中处理的正确静态变量。诸如prepare和clear_body类的初始化调用为用户提供操作ID，该ID在以后的消息访问中使用。

用户完成消息处理后，必须使用操作ID调用clean过程来清理消息存储。这可以避免可能的内存泄漏。消息ADT的clean_all静态过程aq$_jms_bytes_message ， aq$_jms_map_message和aq$_jms_stream_message清除其相应消息存储的所有静态变量。

为流和字节消息强制执行的只读和只写模式

根据JMS规范，当收到消息时，其正文是只读的。用户可以调用clear_body方法使主体可写。此方法将擦除当前消息正文并将消息正文设置为空。

OJMS JAVA API遵循JMS规范设置的规则。但是，在更新PL / SQL中的JMS消息ADT时，Oracle会有选择地强制执行该规则：

  • 映射消息

    放宽限制，因为在现有地图有效负载之上添加更多条目是用户更新有效负载的便捷方式。因此，地图消息没有只读或只写模式。

  • 流和字节消息

    限制并未放宽，因为这些有效负载在读取和写入数据时使用流。在流的中间很难更新有效载荷。Oracle在处理流和字节消息有效负载时强制执行只读和只写模式。调用prepare过程以只读模式初始化消息有效负载。调用clear_body过程以只写模式初始化消息有效负载。

    调用reset过程会将指针重置为流的开头，并将模式从只写切换为只读。reset过程将对消息有效内容的更新保留在Jserv静态变量中。

    另一方面， prepare过程使用PL / SQL ADT中的有效负载覆盖Jserv静态变量中的消息有效负载。

    Oracle为用户提供成员函数get_mode来查询模式。

字节和流消息之间的差异

字节消息的成员函数与流消息的成员函数不完全相同。使用Java ObjectOutputStream消息进行编码，使用Java DataOutputStream字节消息进行编码。在流消息中，每个基本类型都被写入并作为Java对象读取，但在字节消息中，它们根据DataOutputStream的编码机制被写入并作为原始字节读取。

对于流消息， read_bytes方法对字节流进行操作，直到由相应的write_bytes方法写入的字节数组字段的write_bytes 。read_bytes字节消息方法在字节流上工作到整个字节流的末尾。这就是为什么aq$_bytes_message的read_bytes成员过程还需要一个length参数来告诉它要读多长时间。

您不会看到由bytes消息引发的类型转换错误，因为字节消息不支持类型转换。

方法get_unsigned_byte和get_unsigned_short可用于字节消息，但不适用于流消息。这是因为流消息读取Java对象，并且没有Java对象作为无符号字节或无符号短消息。

方法read_string和write_string方法不适用于字节消息。字节消息ADT必须强制执行某些字符编码。它有方法read_utf和write_utf ，它们支持utf-8编码。

注意：

字节消息写入的所有数据都使用DataOutputStream作为基础。有关如何将数据编码为字节的详细信息，请参阅JDK API文档JavaSoft.com。

获取和设置字节，映射和流消息作为RAW字节

字节，映射和流消息类型的有效负载在数据库中存储为RAW或BLOB 。在此版本中，Oracle数据库高级队列提供以下成员函数来设置这些有效负载并将其作为原始字节获取，而无需解释它们：

set_bytes（payload IN BLOB）set_bytes（payload IN RAW）get_bytes（payload OUT BLOB）get_bytes（payload OUT RAW）

这些函数是为Oracle9 i Release 2（9.2）中的字节消息提供的。

277.5一般信息和特定信息之间的上传和下传

OJMS ADT aq$_jms_message用于表示常规消息，因此不同类型的消息可以驻留在同一Oracle Database Advanced Queuing队列中。Oracle数据库高级队列支持通过支持此ADT和特定消息类型的ADT之间的向上转换和向下转换来检索和填充aq$_jms_message 。

要读取aq$_jms_message ，必须先根据其message_type字段将其向下转换为特定的消息类型

要填充aq$_jms_message ，您必须首先填充特定消息并将其上传到aq$_jms_message 。这样可以避免将其他特定消息ADT的所有成员函数复制到此ADT。它还保证此ADT的操作与其他特定消息ADT一致。
