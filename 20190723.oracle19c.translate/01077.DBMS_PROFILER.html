127 DBMS_PROFILER

该软件包提供了一个界面来分析现有的PL / SQL应用程序并识别性能瓶颈。然后，您可以收集并持久存储PL / SQL分析器数据。

127.1 DBMS_PROFILER概述

该软件包可以收集分析器（性能）数据，以提高性能或确定PL / SQL应用程序的代码覆盖率。应用程序开发人员可以使用代码覆盖率数据来集中他们的增量测

使用此界面，您可以为会话中执行的所有命名库单元生成分析信息。分析器在PL / SQL虚拟机级别收集信息。此信息包括每行执行的总次数，执行该行所花费的总时间，以及在该行的特定执行上花费的最小和最大时间。

注意：

可以推断已收集数据的PL / SQL单元的代码覆盖率数字。

分析信息存储在数据库表中。这样可以查询数据：您可以构建可自定义的报告（摘要报告，最热线，代码覆盖率数据等）。您可以分析数据。

127.2 DBMS_PROFILER安全模型

探查器仅收集用户具有CREATE权限的单元的数据;您无法使用该程序包来分析已授予EXECUTE ONLY访问权限的单元。通常，如果用户可以调试单元，则同一用户可以对其进行分析。但是，可以分析单元是否已编译DEBUG. Oracle建议正在分析的模块应该编译为DEBUG,因为这提供了有关数据库中单元的其他信息。

注意：

DBMS_PROFILER将处理以NATIVE模式编译的任何程序单元视为您没有CREATE权限，也就是说，您将不会获得任何输出。

127.3 DBMS_PROFILER操作说明

这些注释描述了典型的运行，如何解释输出，以及两种异常生成方法。

典型的运行

提高应用程序性能是一个迭代过程。每次迭代都涉及以下步骤：

  1. 使用启用了探查器数据收集的一个或多个基准测试运行应用程序。

  2. 分析分析器数据并识别性能问题。

  3. 解决问题。

PL / SQL探查器使用“运行”的概念支持此过程。运行涉及通过启用了探查器数据收集的基准测试来运行应用程序。您可以通过调用START_PROFILER和STOP_PROFILER函数来控制运行的开始和结束。

用户必须首先在探查器用户的架构中创建数据库表以收集数据。PROFTAB 。 SQL脚本创建持久存储探查器数据所需的表和其他数据结构。

请注意，运行PROFTAB.SQL会删除当前表。PROFTAB.SQL脚本位于RDBMS/ADMIN目录中。某些PL / SQL操作（例如PL / SQL单元的第一次执行）可能涉及对目录表进行I / O以加载正在执行的PL / SQL单元的字节代码。此外，在第一次调用包过程或函数时，可能需要一些时间来执行包初始化代码。

为避免计算此开销，请在收集配置文件数据之前“ 预热 ”数据库。为此，请运行应用程序一次，而不收集探查器数据。

您可以允许对系统的所有用户进行分析，例如，分析程序包的所有用户，而不管使用者是谁。在这种情况下， SYSADMIN应该使用修改过的PROFTAB.SQL脚本：

  • 创建探查器表和序列

  • 对这些表授予SELECT / INSERT / UPDATE并对所有用户进行排序

  • 定义表和序列的公共同义词

    注意：

    不要改变表的实际字段。

典型的运行包括：

  • 在运行中启动探查器数据收集。

  • 执行需要分析器和代码覆盖率数据的PL / SQL代码。

  • 停止探查器数据收集，将收集的运行数据写入数据库表

    注意：

    当用户断开连接时，不会自动存储收集的探查器数据。您必须发出对FLUSH_DATA或STOP_PROFILER函数的显式调用，以在会话结束时存储数据。停止数据收集存储收集的数据。

在应用程序执行时，分析器数据被收集在持续运行期间的内存数据结构中。您可以在运行期间的中间点调用FLUSH_DATA函数来获取增量数据并释放已分配的分析器数据结构的内存。刷新收集的数据涉及将收集的数据存储在先前创建的数据库表中。

解释输出

表plsql_profiler_data为生成代码的源单元的每一行包含一行。行＃值指定哪个源行。如果该行存在，并且该行中的total_occur值> 0，则执行与该行关联的一些代码。如果该行存在且total_occur值为0，则不会执行与该行关联的代码。如果表中不存在该行，则不会为该行生成任何代码，因此不应在报告中提及该行

如果单个语句的源是在一行上，则为该语句生成的任何代码都将归于该行号。（在某些情况下，例如简单的声明，或者由于优化，不需要代码）。要获取覆盖率信息，应使用PLSQL_OPTIMIZE_LEVEL=1编译单位。

如果语句跨越多行，则为该语句生成的任何代码都将归因于该范围中的某些行，但不保证该范围中的每一行都将具有归属于它的代码。在这种情况下，行＃值集合中将存在间隙。特别是，多行SQL相关语句可能看起来在一行（通常是第一行）。这是因为PL / SQL将处理后的文本文本传递给SQL引擎;因此，就PL / SQL而言，整个SQL语句是一个不可分割的操作。

当多个语句位于同一行时，探查器将组合每个语句的出现次数。如果一条线嵌入了控制流，这可能会造成混淆。例如，如果'then ...'和'else ...'在同一行，则无法确定是否采用了'then'或'else' 。

通常，如果每个语句都在自己的行上，则最容易解释分析器和覆盖率报告。

两种异常生成方法

此程序包中的每个例程都有两个版本，可用于确定报告错误的方式。

  • 一种函数，它将成功/失败作为状态值返回，并且永远不会引发异常

  • 如果成功则返回正常的过程，如果失败则引发异常

在每种情况下，功能和过程的参数是相同的。只报告错误的方法不同。如果存在错误，则函数返回的错误代码与程序引发的异常之间存在对应关系。

127.5.6 START_PROFILER函数和过程

此函数在用户会话中启动探查器数据收集。

START_PROFILER函数有两种重载形式;一个返回已启动运行的运行编号，以及调用的结果。另一个不返回运行编号。第一种形式旨在与控制分析器的基于GUI的工具一起使用。

句法

DBMS_PROFILER.START_PROFILER（run_comment IN VARCHAR2：= sysdate，run_comment1 IN VARCHAR2：=''，run_number OUT BINARY_INTEGER）RETURN BINARY_INTEGER; DBMS_PROFILER.START_PROFILER（run_comment IN VARCHAR2：= sysdate，run_comment1 IN VARCHAR2：=''）RETURN BINARY_INTEGER; DBMS_PROFILER.START_PROFILER（run_comment IN VARCHAR2：= sysdate，run_comment1 IN VARCHAR2：=''，run_number OUT BINARY_INTEGER）; DBMS_PROFILER.START_PROFILER（run_comment IN VARCHAR2：= sysdate，run_comment1 IN VARCHAR2：=''）;
