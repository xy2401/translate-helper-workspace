20.1 DBMS_APPLY_ADM概述

DBMS_APPLY_ADM程序包提供用于启动，停止和配置Oracle Apply进程，XStream出站服务器和XStream入站服务器的接口。

此程序包包括用于配置应用程序处理程序，设置消息的入队目标以及指定消息的执行指令的子程序。此程序包还提供管理子程序，用于为目标数据库中的对象设置实例化SCN。该软件包还包括用于管理应用错误的子程序。

XStream入站服务器和出站服务器可用于多租户容器数据库（CDB）中的XStream配置。CDB是Oracle数据库，包括零个，一个或多个用户创建的可插拔数据库（PDB）。

注意：

  • 为简单起见，本章将应用进程，XStream出站服务器和XStream入站服务器称为应用组件 。本章在必要时标识特定类型的应用组件。

  • 使用XStream需要购买Oracle GoldenGate产品的许可证。

20.2 DBMS_APPLY_ADM安全模型

可以通过将此程序包上的EXECUTE授予选定的用户或角色，或者通过向所选用户或角色授予EXECUTE_CATALOG_ROLE来控制此程序包的安全性。

如果程序包中的子程序是从存储过程中运行的，则必须直接为程序包授予运行子程序的用户EXECUTE特权。它不能通过角色授予。

使用DBMS_APPLY_ADM程序包管理Oracle Replication配置时，需要为用户授予Oracle Replication管理员的权限。

当DBMS_APPLY_ADM包用于管理XStream配置时，它要求为用户授予XStream管理员的权限。

20.4.1 ALTER_APPLY过程

此过程会更改应用组件。

句法

DBMS_APPLY_ADM.ALTER_APPLY（apply_name IN VARCHAR2，rule_set_name IN VARCHAR2 DEFAULT NULL，remove_rule_set IN BOOLEAN DEFAULT FALSE，message_handler IN VARCHAR2 DEFAULT NULL remove_message_handler IN BOOLEAN DEFAULT FALSE，ddl_handler IN VARCHAR2 DEFAULT NULL，remove_ddl_handler IN BOOLEAN DEFAULT FALSE，apply_user IN VARCHAR2 DEFAULT NULL，apply_tag IN RAW DEFAULT NULL，remove_apply_tag IN BOOLEAN DEFAULT FALSE，precommit_handler IN VARCHAR2 DEFAULT NULL，remove_precommit_handler IN BOOLEAN DEFAULT FALSE，negative_rule_set_name IN VARCHAR2 DEFAULT NULL，remove_negative_rule_set IN BOOLEAN DEFAULT FALSE）;

参数

表20-2 ALTER_APPLY过程参数

参数 描述

apply_name

要更改的应用组件的名称。您必须指定现有应用组件的名称。不要指定所有者。

rule_set_name

应用组件的正规则集的名称。正规则集包含指示应用组件应用消息的规则。

如果要为apply组件使用正规则集，则必须以[ schema_name .] rule_set_name形式指定现有规则集。例如，要在名为job_apply_rules的hr模式中指定正规则集，请输入hr.job_apply_rules 。如果未指定架构，则当前用户是默认用户。

如果指定的规则集不存在，则返回错误。

如果指定NULL并且remove_rule_set参数设置为FALSE ，则此过程将保留指定的apply组件的任何现有正规则集。如果指定NULL并且remove_rule_set参数设置为TRUE ，则此过程将从指定的apply组件中删除任何现有的正规则集。

remove_rule_set

如果为TRUE ，则该过程将删除指定的apply组件的正规则集。如果删除应用组件的正规则集，并且apply组件没有负规则集，则apply组件将使其队列中的所有消息出列。

如果删除应用组件的正规则集，并且apply组件存在否定规则集，则apply组件会使其队列中未被否定规则集丢弃的所有消息出列。

如果为FALSE ，则该过程将保留指定的应用组件的正规则集。

如果rule_set_name参数为非NULL ，则应将此参数设置为FALSE 。

MESSAGE_HANDLER

用户定义的过程，用于处理应用组件的队列中的非LCR消息。

有关消息处理程序过程的详细信息，请参阅CREATE_APPLY过程中的“ 使用说明 ” 。

remove_message_handler

如果为TRUE ，则该过程将删除指定的apply组件的消息处理程序。

如果为FALSE ，则该过程将保留指定的apply组件的任何消息处理程序。

如果message_handler参数为非NULL ，则此参数应设置为FALSE 。

ddl_handler

用户定义的过程，用于处理应用组件队列中的DDL逻辑更改记录（DDL LCR）。

所有应用的DDL LCR都会自动提交。因此，如果DDL处理程序调用DDL LCR的EXECUTE成员过程，则会自动执行提交。

有关DDL处理程序过程的详细信息，请参阅CREATE_APPLY过程中的“ 使用说明 ” 。

remove_ddl_handler

如果为TRUE ，则该过程将删除指定的apply组件的DDL处理程序。

如果为FALSE ，则该过程将保留指定的apply组件的任何DDL处理程序。

如果ddl_handler参数为非NULL ，则应将此参数设置为FALSE 。

apply_user

安全域中的应用组件的用户将满足其规则集的消息出列，将消息直接应用于数据库对象，运行基于规则的自定义转换，并运行应用处理程序。如果为NULL ，则不更改apply用户。

如果指定了非NULL值来更改apply用户，则必须为调用ALTER_APPLY过程的用户授予DBA角色。只有SYS用户可以将apply_user设置为SYS 。

如果更改apply用户，则此过程将对apply组件使用的队列授予新的apply user dequeue特权。它还将用户配置为队列的安全队列用户。

除了此过程授予的权限之外，您还应该向apply用户授予以下权限：

  • 在apply对象上执行DML和DDL更改所需的权限

  • 应用组件使用的规则集的EXECUTE特权

  • 对规则集中使用的所有基于规则的转换函数的EXECUTE特权

  • 所有应用处理程序过程的EXECUTE特权

这些权限可以直接授予apply用户，也可以通过角色授予。

此外，必须为apply用户授予对所有包（包括Oracle提供的包）的EXECUTE特权，这些包在apply组件运行的子程序中调用。必须直接向apply用户授予这些权限。他们不能通过角色授予。

默认情况下，通过运行此程序包中的CREATE_APPLY过程，将此参数设置为创建应用程序组件的用户。

注意：如果使用DROP USER. . .删除了应用组件的apply USER. . . CASCADE ，然后自动删除应用组件。

apply_tag

添加到由指定的应用组件生成的重做条目的二进制标记。标签是二进制值，可用于跟踪LCR。

仅当运行apply组件的数据库中的捕获进程捕获apply组件所做的更改时，该标记才是相关的。如果是，则捕获的更改包括此参数指定的标记。

如果为NULL （默认值），则不会更改apply组件的apply标记。

以下是十六进制值为17的标记示例：

HEXTORAW（'17' ）

remove_apply_tag

如果为TRUE ，则过程将指定的apply组件的apply标记设置为NULL ，并且apply组件生成带有NULL标记的重做条目。

如果为FALSE ，则该过程将保留指定的apply组件的任何apply标记。

如果apply_tag参数为非NULL ，则应将此参数设置为FALSE 。

precommit_handler

用户定义的过程，可以在apply组件处理apply组件之前接收队列中的内部提交指令。通常，预提交处理程序用于审核由应用组件处理的事务的提交信息。

内部提交指令以下列方式排队：

  • 当捕获进程捕获行LCR时，捕获进程会将包含行LCR的事务的commit指令排入队列。

  • 当用户或应用程序将消息排入队列然后发出COMMIT语句时，commit指令会自动排队。

对于捕获的行LCR，commit指令包含来自源数据库的事务的提交SCN。对于用户消息，提交SCN由应用组件生成。

预提交处理程序必须符合以下限制：

  • 任何提交的工作必须是自主交易。

  • 任何回滚都必须是在过程中创建的命名保存点。

如果预提交处理程序引发异常，则回滚整个应用事务，并将事务中的所有消息移动到错误队列。

有关预提交处理程序过程的详细信息，请参阅CREATE_APPLY过程中的“ 使用说明 ” 。

remove_precommit_handler

如果为TRUE ，则该过程将删除指定的apply组件的预提交处理程序。

如果为FALSE ，则该过程将保留指定的apply组件的任何预提交处理程序。

如果precommit_handler参数为非NULL ，则此参数应设置为FALSE 。

negative_rule_set_name

应用组件的负规则集的名称。否定规则集包含指示应用组件放弃消息的规则。

如果要对apply组件使用否定规则集，则必须以[ schema_name .] rule_set_name形式指定现有规则集。例如，要在名为neg_apply_rules的hr模式中指定否定规则集，请输入hr.neg_apply_rules 。如果未指定架构，则当前用户是默认用户。

如果指定的规则集不存在，则返回错误。

如果指定NULL并且remove_negative_rule_set参数设置为FALSE ，则该过程将保留任何现有的否定规则集。如果指定NULL并且remove_negative_rule_set参数设置为TRUE ，则该过程将删除任何现有的否定规则集。

如果为应用组件指定正规则集和负规则集，则始终首先评估否定规则集。

remove_negative_rule_set

如果为TRUE ，则该过程将删除指定的apply组件的否定规则集。如果删除应用组件的否定规则集，并且apply组件没有正规则集，则apply组件将使其队列中的所有消息出列。

如果删除了应用组件的否定规则集，并且应用组件存在正规则集，则apply组件会将其队列中未被正规则集丢弃的所有消息出列。

如果为FALSE ，则该过程将保留指定的应用组件的否定规则集。

如果negative_rule_set_name参数为非NULL ，则此参数应设置为FALSE 。

当您更改以下一个或多个ALTER_APPLY过程参数的值时，将自动停止并重新启动应用组件：

ALTER_APPLY过程和XStream出站服务器

以下使用说明适用于此过程和XStream出站服务器：

  • 您不能为出站服务器指定应用处理程序。出站服务器忽略以下参数的设置： message_handler ， ddl_handler和precommit_handler 。

20.4.3 COMPARE_OLD_VALUES程序

此过程指定是否将行逻辑更改记录（行LCR）中的一个或多个列的旧值与应用期间目标站点上相应列的当前值进行比较。

此过程仅适用于UPDATE和DELETE操作，因为只有这些操作才会导致行LCR中的旧列值。默认设置是比较所有列的旧值。

此过程已超载。column_list和column_table参数是互斥的。

句法

DBMS_APPLY_ADM.COMPARE_OLD_VALUES（object_name IN VARCHAR2，column_list IN VARCHAR2，操作IN VARCHAR2 DEFAULT'UPDATE'，比较IN BOOLEAN DEFAULT TRUE，apply_database_link IN VARCHAR2 DEFAULT NULL）; DBMS_APPLY_ADM.COMPARE_OLD_VALUES（object_name IN VARCHAR2，column_table IN DBMS_UTILITY.LNAME_ARRAY，operation IN VARCHAR2 DEFAULT'UPDATE'，比较IN BOOLEAN DEFAULT TRUE，apply_database_link IN VARCHAR2 DEFAULT NULL）;

参数

OBJECT_NAME

指定为[ schema_name .] object_name的源表的名称。例如， hr.employees 。如果未指定架构，则当前用户是默认用户。

column_list中

表中以逗号分隔的列名列表。条目之间必须没有空格。

冲突检测

默认情况下，apply组件使用行LCR中的旧列值来检测冲突。您可以选择不比较旧列值以避免特定表的冲突检测。例如，如果您不想在应用期间比较一组列的旧值，则使用COMPARE_OLD_VALUES过程在column_list或column_table参数中指定列集， compare参数设置为FALSE 。

此外，当compare_key_only apply组件参数设置为Y ，将禁用自动冲突检测，并且apply组件仅使用主键和唯一键列来标识行LCR的表行。当compare_key_only apply组件参数设置为N ，启用自动冲突检测，并且apply组件使用行LCR中的所有旧值来标识行LCR的表行。

注意：

  • 当应用组件存在于行LCR中并且应用组件参数compare_key_only设置为N时，应用组件将非键列的旧值进行比较。

  • 如果在column_list或column_table指定了键列并且compare参数设置为FALSE则此过程会引发错误。

apply用户是安全域中的应用组件使满足其规则集的消息出列，将消息直接应用于数据库对象，运行为应用组件规则配置的基于规则的自定义转换，以及运行为应用组件配置的应用处理程序的用户。此用户必须具有应用更改所需的权限。此过程为apply组件使用的队列授予apply user dequeue特权，并将用户配置为队列的安全队列用户。

  • 对规则集中使用的所有基于规则的转换函数的EXECUTE特权

  • 所有应用处理程序过程的EXECUTE特权

仅当运行apply组件的数据库中的捕获进程捕获apply组件所做的更改时，该标记才是相关的。如果是，则捕获的更改包括此参数指定的标记。

TRUE或FALSE 。

注意：创建应用组件后，无法更改apply_captured设置。

precommit_handler

用户定义的过程，可以在apply组件处理apply组件之前接收队列中的内部提交指令。通常，预提交处理程序用于审核由应用组件处理的事务的提交信息。

内部提交指令以下列方式排队：

  • 当捕获进程捕获行LCR时，捕获进程会将包含行LCR的事务的commit指令排入队列。

  • 当同步捕获捕获行LCR时，由同步捕获排队的持久性LCR被组织到消息组中。同步捕获在事务中记录每个持久LCR中的事务标识符。

  • 当用户或应用程序将消息排入队列然后发出COMMIT语句时，commit指令会自动排队。

对于由捕获进程或同步捕获捕获的行LCR，commit指令包含来自源数据库的事务的提交SCN。对于由用户或应用程序排队的消息，提交SCN由应用组件生成。

预提交处理程序必须符合以下限制：

  • 任何提交的工作必须是自主交易。

  • 任何回滚都必须是在过程中创建的命名保存点。

如果预提交处理程序引发异常，则回滚整个应用事务，并将事务中的所有消息移动到错误队列。

有关预提交处理程序过程的详细信息，请参阅“ 使用说明 ” 。

用户过程具有以下参数：

  • in_anydata ：应用组件传递给过程的消息的ANYDATA封装。单个事务可以包含多个消息。消息可以是行逻辑更改记录（行LCR），DDL逻辑更改记录（DDL LCR）或用户消息。

  • error_record ： DBA_APPLY_ERROR数据字典视图中标识事务的行

  • error_message_number ： in_anydata参数中ANYDATA对象的消息号，从1开始

  • messaging_default_processing ：如果为TRUE ，则apply组件继续处理in_anydata参数中的消息，该参数可以包括执行DML或DDL语句以及调用apply处理程序。

    如果FALSE ，则该应用部件跳过处理所述消息in_anydata参数，并移动到在下一消息in_anydata参数。

  • out_anydata ： ANYDATA对象由用户过程处理，如果messaging_default_processing为TRUE则由apply组件使用。

如果在用户过程中使用EXECUTE LCR成员过程EXECUTE LCR，则直接执行LCR，并且messaging_default_processing参数应设置为FALSE 。在这种情况下，LCR不会传递给任何应用处理程序。

使用用户过程处理错误事务会导致以下结果之一：

  • 用户过程修改事务，以便可以成功执行。

  • 用户过程无法进行必要的修改，并且在尝试执行事务时会引发错误。在这种情况下，事务将回滚并保留在错误队列中。

以下限制适用于用户过程：

  • 不要执行COMMIT或ROLLBACK语句。这样做可能会危及事务的一致性。

  • 不要修改LCR中的LONG ， LONG RAW或LOB列数据。

  • 如果in_anydata参数中的ANYDATA对象是行LCR，则如果messaging_default_processing参数设置为TRUE ，则out_anydata参数必须为行LCR。

  • 如果in_anydata参数中的ANYDATA对象是DDL LCR，则如果messaging_default_processing参数设置为TRUE ，则out_anydata参数必须为DDL LCR。

  • 运行用户过程的用户必须具有DBA_APPLY_ERROR数据字典视图的SELECT或READ权限。

注意：

包含事务指令的LCR（例如COMMIT和ROLLBACK ）不会传递给用户过程。

出站服务器不会将错误事务排入错误队列。此过程不能与XStream出站服务器一起使用。

EXECUTE_ERROR过程和XStream入站服务器

此过程对应用进程和入站服务器的功能相同。

  • [ schema_name .] package_name.procedure_name

  • [ schema_name .] procedure_name

如果过程在包中，则必须指定package_name 。调用SET_DML_HANDLER过程的用户必须具有指定过程的EXECUTE特权。此外，如果未指定schema_name ，则调用SET_DML_HANDLER过程的用户是默认值。

  • hr是schema_name 。

  • employees_default是procedure_name 。

以下限制适用于用户过程：

  • 不要执行COMMIT或ROLLBACK语句。这样做可能会危及包含LCR的事务的一致性。

  • 如果使用行LCR的EXECUTE成员过程操作行，则不要尝试在行操作中操作多行。您必须手动构造和执行任何操作多行的DML语句。

  • 如果命令类型为UPDATE或DELETE ，则使用LCR的EXECUTE成员过程重新提交的行操作必须包含旧值列表中的整个键。密钥是主键或具有至少一个NOT NULL列的最小唯一索引，除非SET_KEY_COLUMNS过程指定了替换键。如果没有指定的键，则该键包含所有非LOB ，非LONG和非LONG RAW列。

  • 如果命令类型为INSERT ，则使用LCR的EXECUTE成员过程重新提交的行操作应包括新值列表中的整个键。否则，可能存在重复的行。密钥是主键或具有至少一个NOT NULL列的最小唯一索引，除非SET_KEY_COLUMNS过程指定了替换键。如果没有指定的键，则该键包含所有表列，但以下数据类型的列除外：LOB， LONG ， LONG RAW ，用户定义的类型（包括对象类型， REF ，varrays，嵌套表） ）和Oracle提供的类型（包括Any类型，XML类型，空间类型和媒体类型）。

PROCEDURE user_procedure （ parameter_name IN ANYDATA）;

这里， user_procedure代表过程的名称， parameter_name代表传递给过程的参数的名称。传递给过程的参数是行LCR的ANYDATA封装。

如果表中的两行或更多行的所有列值相同，则表具有重复行，不包括LOB， LONG和LONG RAW列。您可以使用SET_KEY_COLUMNS过程为目标数据库中的表指定替换主键列。如果为目标数据库中具有重复行的表指定了替换主键列，并且allow_duplicate_rows apply组件参数设置为Y ，则满足以下要求以使表数据在源数据库和目标数据库中保持同步：

  • 确保在源数据库中为在目标数据库中指定为替换键列的列指定了补充日志记录。替换键列必须位于源数据库的无条件日志组中。

  • 确保替换键列唯一标识目标数据库中表中的每一行。

本节的其余部分提供了有关这些要求的更多详细信息。

如果表没有键且allow_duplicate_rows apply组件参数设置为Y ，则仅将具有UPDATE或DELETE命令类型的单行LCR应用于其中一个重复行。在这种情况下，如果源数据库中的表和目标数据库中的表具有相应的重复行，那么更改源数据库中所有重复行的更改也会更改目标数据库中的所有重复行应用由更改产生的LCR。

例如，假设源数据库中的表有两个重复的行。对重复行执行更新，从而产生两行LCR。在目标数据库中，一行LCR应用于其中一个重复行。此时，目标数据库中的行不再重复，因为其中一行已更改。当第二行LCR应用于目标数据库时，行再次重复。同样，如果在源数据库上对这些重复行执行了删除，则在应用源更改产生的行LCR时，将在目标数据库中删除这两行。

为表指定替换主键列时，使用替换主键列在应用期间使用表中的行标识行LCR。如果为目标数据库中具有重复行的表指定了替换主键列，并且allow_duplicate_rows apply组件参数设置为Y ，则对源数据库中的重复行执行的更新可能会导致行LCR为应用于目标数据库。具体来说，如果更新未更改指定为替换主键列的列之一，则可以在目标数据库中多次更新相同的重复行，而其他重复行可能不会更新。

此外，如果替换键列没有唯一地标识目标数据库中表中的每一行，则标识有多行的行LCR可以更新任何一行。在这种情况下，行LCR中的更新可能不会应用于目标数据库中表中的正确行。

当应用组件确定表中的行是否重复时，它会忽略替换主键列。仅当行中的所有列值相同（不包括LOB， LONG和LONG RAW列）时，apply组件才会确定行是重复的。因此，如果单个更新或删除更改表中的两个或更多非重复行，则应用组件始终会引发错误。

例如，考虑具有列c1 ， c2和c3的表，在该表上使用SET_KEY_COLUMNS过程将列c1指定为替换主键。如果两行具有c1列的相同键值，但c2或c3列的值不同，则apply组件不会将这些行视为重复。如果更新或删除修改了多行，因为行中的c1值相同，则无论allow_duplicate_rows apply组件参数的设置如何，apply组件都会引发错误。

STOP_APPLY强制参数和commit_serialization应用参数

下表描述了STOP_APPLY过程和commit_serialization apply组件参数中force参数的每个设置的应用组件行为。在所有情况下，apply组件在停止时回滚所有未完成的事务。

力 commit_serialization 应用组件行为

真正

充分

apply组件立即停止，不应用任何未完成的事务。

真正

DEPENDENT_TRANSACTIONS

当apply组件停止时，本地应用的某些事务可能在某个时间点在源数据库中提交，而不是在本地未应用的某些事务。

假

充分

如果任何此类事务正在进行中，则应用组件在提交顺序中应用下一个未提交的事务后停止。

假

DEPENDENT_TRANSACTIONS

在停止之前，apply组件应用所有事务，这些事务的提交时间早于应用的事务，并且具有最近的提交时间。

例如，假设commit_serialization apply组件参数设置为DEPENDENT_TRANSACTIONS并且有三个事务：事务1具有最早的提交时间，事务2在事务1之后提交，事务3具有最新的提交时间。还假设apply组件已应用事务1和事务3，并且在运行STOP_APPLY过程时正在应用事务2。在这种情况下，如果force参数设置为TRUE ，则不应用事务2，并且apply组件停止（事务2被回滚）。但是，如果force参数设置为FALSE ，则在应用组件停止之前应用事务2。

如果commit_serialization apply组件参数设置为FULL则会产生不同的方案。例如，假设commit_serialization apply组件参数设置为FULL并且有三个事务：事务A具有最早的提交时间，事务B在事务A之后提交，而事务C具有最新的提交时间。在这种情况下，apply组件已应用事务A，并且在运行STOP_APPLY过程时正在应用事务B和C.在这种情况下，如果force参数设置为TRUE ，则不应用事务B和C，并且apply组件停止（事务B和C被回滚）。但是，如果force参数设置为FALSE ，则在应用组件停止之前应用事务B，并回滚事务C.
