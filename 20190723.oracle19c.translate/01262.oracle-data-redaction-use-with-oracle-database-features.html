Oracle Data Redaction一般使用指南

了解使用Oracle Data Redaction的使用指南非常重要。

  • 不要在SQL语句中的GROUP BY子句中使用的SQL表达式中包含任何编辑的列。Oracle不支持这一点，并引发ORA-00979: not a GROUP BY expression错误。发生这种情况是因为内部SELECT列表中的表达式必须由Data Redaction修改，但这会导致在处理GROUP BY子句（当前未由Data Redaction更新）时不再找到它，导致这种情况无意错误信息。

  • 不要在SQL语句中包含任何编辑的列，这些列在SQL语句的DISTINCT子句和ORDER BY子句中都使用。Oracle不支持此操作，并引发错误： ORA-01791: not a SELECTed expression 。发生这种情况是因为内部SELECT列表中的表达式必须由Data Redaction修改，但这会导致在处理GROUP BY子句时不再找到它，从而导致出现此意外错误消息。

  • 一个ORA-28094: SQL construct not supported by data redaction如果查询涉及引发错误UNION节录列和各分支UNION不具有相同的节录政策。要避免ORA-28094错误，请确保查询具有以下属性：

    • 当UNION的列具有编校策略时， UNION每个分支中的相应列必须使用对其所有属性具有相同值的编校策略：

      • 功能类型

      • 功能参数或REGEXP参数

      • 政策表达

      • 启用标志

      它可以是不同的编校策略，但所有这些属性必须相同。

    • 在内联视图中， UNION不能具有子查询或任何涉及编辑列的SQL表达式。

Oracle Data Redaction和DML和DDL操作

Oracle Data Redaction会影响DML和DDL操作，尤其是对于针对具有编辑列的表发布临时SQL的用户。

请注意以下事项：

  • 即使未显示结果，Oracle Data Redaction也会将DML语句的RETURNING INTO子句视为查询。发送到缓冲区的结果是将RETURNING INTO子句作为普通SQL查询运行而不是作为DML语句的一部分运行时显示的结果。如果应用程序对包含RETURNING INTO值的缓冲区执行进一步处理，请考虑更改应用程序，因为它可能不希望在缓冲区中找到编辑的值。

  • 如果编辑的列在DML或DDL操作中显示为源，则Oracle Data Redaction会将此视为试图绕过策略并使用ORA-28081: Insufficient privileges - the command references a redacted object阻止它ORA-28081: Insufficient privileges - the command references a redacted object错误，除非您有EXEMPT REDACTION POLICY系统特权。在内部，Oracle Data Pump会发出这些类型的操作，因此如果需要执行具有编辑列的表的模式级导出，您可能还需要向用户授予EXEMPT REDACTION POLICY系统特权。

Oracle数据编辑和嵌套函数，内联视图和WHERE子句

您可以将Oracle Data Redaction与嵌套函数，内联视图和SELECT语句中的WHERE子句一起使用。

Oracle Data Redaction策略的工作方式如下：

  • 嵌套函数在最里面编辑。 例如，在SELECT SUM(AVG(TO_NUMBER(((X))) FROM HR.EMPLOYEES WHERE ... ，首先编辑TO_NUMBER函数，然后是AVG ，然后是SUM函数。

  • 内联视图最外面的编辑。 例如，在SELECT XYZ … AS SELECT A… AS SELECT B… AS SELECT C… ，首先编辑SELECT XYZ ，然后是AS SELECT A ，然后是AS SELECT B ，依此类推。AS SELECT C最后编辑。

  • WHERE子句永远不会被编辑。 数据编辑仅编辑列SELECT列表中的数据。

数据编校策略保护的列上的Oracle数据编辑和查询

对受Oracle Data Redaction策略保护的列包含DISTINCT或ORDER BY子句的查询可能返回0行。

发生这种情况是因为编校保留了查询的语义。例如，查询如SELECT * table_name WHERE sensitive_column LIKE ' value ';可能会导致返回0行，因为编辑的值与WHERE子句中输入的' value '值不匹配。要解决此问题，请重写查询以包含内联视图，以便语义层可以找到列。例如，而不是以下查询：

SELECT DISTINCT sensitive_column FROM table_name ORDER BY sensitive_column ;

编写查询如下：

SELECT sensitive_column FROM（SELECT DISTINCT sensitive_column FROM table_name ORDER BY sensitive_column ）;

Oracle数据编辑和聚合函数

聚合函数可能会影响Oracle Data Redaction策略的性能开销。

对于调用聚合函数的SQL查询，可能会发现由于编校而导致的性能开销。

多租户环境中的Oracle数据编辑

在多租户环境中，Oracle Data Redaction策略仅适用于当前可插拔数据库（PDB）中的对象。

您无法为多租户容器数据库（CDB）创建数据编辑策略。这是因为您为其创建数据编辑策略的对象通常驻留在PDB中。如果您具有SYSDBA权限，则可以通过运行SHOW PDBS命令列出CDB中的所有PDB。

与CDB根一样，您无法在应用程序根目录中创建数据编辑策略。

Oracle数据编辑和Oracle虚拟专用数据库

Oracle Data Redaction不会影响Oracle虚拟专用数据库策略，因为包含VPD谓词的VPD内联视图会对实际值起作用。

Oracle Data Redaction与Oracle Virtual Private Database的区别在于以下方面：

  • Oracle Data Redaction提供了比Oracle虚拟专用数据库更多的编辑功能，后者仅支持NULL编辑。许多应用程序不能使用NULL编辑，因此Data Redaction是这些应用程序的一个很好的解决方案。

  • Oracle虚拟专用数据库策略可以是静态，动态和上下文敏感的，而数据编辑策略仅允许静态和上下文敏感的策略表达式。

  • Data Redaction仅允许在表或视图上定义一个策略，而您可以在对象上定义多个虚拟专用数据库策略。

  • 数据编辑是指应用程序用户尝试使用同义词访问受数据编辑策略保护的对象，但（与Oracle虚拟专用数据库不同）数据编辑不支持直接在同义词本身上创建策略。

Oracle Data Redaction的Oracle数据泵安全模型

DATAPUMP_EXP_FULL_DATABASE角色包括强大的EXEMPT REDACTION POLICY系统特权。

请记住，默认情况下， DBA角色被授予DATAPUMP_EXP_FULL_DATABASE角色以及DATAPUMP_IMP_FULL_DATABASE 。

这使得被授予这些角色的用户可以免于数据编辑策略。这意味着，当您导出具有在其上定义的数据编辑策略的对象时，受保护表中的实际数据将被复制到数据泵目标系统，而不会被编辑。具有这些角色的用户（包括被授予DBA角色的用户）能够查看目标系统中的实际数据。

但是，默认情况下，与数据泵源系统中的任何表和视图关联的所有数据编辑策略也包含在导出和导入操作中（与对象本身一起）并应用于目标系统中的对象，因此当用户查询目标系统中的对象时，数据仍会被编辑。

使用EXPDP实用程序access_method参数导出数据

Oracle Data Pump可以从包含具有数据编辑策略的对象的模式导出数据。

如果您使用Oracle Data Pump使用Data Pump默认设置（ direct_path ）执行完整数据库导出操作，并且如果收到您不理解的错误消息，请使用此部分以更好的方式重复操作理解错误。

如果您尝试使用Oracle数据泵导出（ EXPDP ）实用程序并将access_method参数设置为direct_path以从包含已在其上定义数据编辑策略的对象的模式导出数据，则可能会出现以下错误消息，并且导出操作失败：

ORA-31696：无法使用客户端指定的DIRECT_PATH方法导出/导入TABLE_DATA：“ schema.table ”

仅当您以未授予EXP_FULL_DATABASE角色的用户身份执行架构级别导出时，才会出现此问题。在完整数据库导出期间不会发生这种情况，这需要EXP_FULL_DATABASE角色。EXP_FULL_DATABASE角色包括EXEMPT REDACTION POLICY系统特权，该特权绕过数据编辑策略。

要查找基础问题，请再次尝试EXPDP调用，但不要将access_method参数设置为direct_path 。相反，使用automatic或external_table 。根本问题可能是权限问题，例如：

ORA-28081：权限不足 - 该命令引用编校对象。

将数据导入受Oracle Data Redaction保护的对象

在导入操作期间，请注意不要无意中删除保护导入数据的数据编校策略。

考虑使用Oracle Data Pump Export（ EXPDP ）实用程序导出的源表没有Oracle Data Redaction策略的情况。但是，使用Oracle Data Pump Import（ IMPDP ）将数据导入到的目标表具有Oracle Data Redaction策略。

在数据泵导入操作期间，正在导入的对象上的数据编辑策略的状态取决于IMPDP命令的CONTENT选项。

  • 如果在IMPDP命令中使用CONTENT=ALL或CONTENT=METADATA_ONLY选项，则会删除目标表上的数据编辑策略。您必须重新创建数据编辑策略。

  • 如果在IMPDP命令中使用CONTENT=DATA_ONLY ，则不会删除目标表上的数据编辑策略。
