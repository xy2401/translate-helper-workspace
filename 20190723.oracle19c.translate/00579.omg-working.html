19.1配置Oracle Messaging Gateway代理

消息通过Messaging Gateway代理在Oracle Database Advanced Queuing和非Oracle消息传递系统之间传播。Messaging Gateway代理作为Oracle数据库服务器的外部进程运行。

Messaging Gateway支持给定数据库的多个代理。将自动创建名为DEFAULT_AGENT默认代理。可以创建其他命名代理，以在Oracle RAC环境中提供传播作业隔离和分组以及扩展。对于单实例非Oracle RAC环境，默认代理通常就足够了。

您可以使用DBMS_MGWADM 。 CREATE_AGENT以创建其他Messaging Gateway代理。Messaging Gateway默认代理DEFAULT_AGENT在安装Messaging Gateway时自动创建，并且始终存在。

可以在创建代理时使用代理用户，连接信息，数据库服务和资源限制配置代理，也可以在以后使用DBMS_MGWADM 。 ALTER_AGENT 。必须使用已授予角色MGW_AGENT_ROLE的数据库用户配置Messaging Gateway代理，然后才能启动代理。

您可以使用DBMS_MGWADM 。 ALTER_AGENT为Messaging Gateway代理设置资源限制。例如，您可以设置Messaging Gateway代理进程的堆大小以及代理进程使用的传播线程数。默认值为64 MB内存堆和一个传播线程。对于命名代理程序，还可以在DBMS_MGWADM创建代理程序时指定这些值。 CREATE_AGENT 。

19.2.3 Oracle Messaging Gateway代理调度程序作业

Messaging Gateway使用Scheduler作业来启动Messaging Gateway代理。调用过程DBMS_MGWADM.STARTUP时会创建此作业。运行作业时，它会调用外部过程，该过程在外部进程中创建Messaging Gateway代理。该作业在以下后删除：

  • 代理程序因为调用了DBMS_MGWADM.SHUTDOWN而关闭

  • 代理终止，因为发生了不可重新启动的错误

Messaging Gateway使用DBMS_SCHEDULER创建可重复的Scheduler作业，重复间隔为一分钟。这份工作归SYS 。可重复的作业使Messaging Gateway代理能够在给定的作业实例因数据库关闭，数据库故障或可重新启动的错误而结束时自动重新启动。在给定时间只运行一个Messaging Gateway代理作业实例。

每个代理使用Scheduler作业类来指定代理的Scheduler作业的服务亲缘关系。将使用DBMS_MGWADM指定的数据库服务配置作业类。 CREATE_AGENT或DBMS_MGWADM 。 ALTER_AGENT 。数据库管理员负责设置数据库服务。如果未指定数据库服务，则使用映射到每个实例的默认数据库服务。

Messaging Gateway默认代理程序使用的Scheduler作业类的名称是SYS 。 MGW_JOBCLS_DEFAULT_AGENT 。默认代理程序使用的Scheduler作业是SYS 。 MGW_JOB_DEFAULT_AGENT 。

Messaging Gateway命名代理程序使用的Scheduler作业类的名称是SYS 。 MGW_JOBCLS_ < agent_name >。命名代理程序使用的Scheduler作业是SYS 。 MGW_JOB_ < agent_name >。

如果代理程序作业遇到错误，则将错误分类为可重新启动的错误或不可重新启动的错误。可重新启动的错误表示如果要重新启动代理作业，可能会消失的问题。不可重新启动的错误表示可能会持续存在的问题，并且如果代理作业重新启动，则会再次遇到此问题。ORA-01089（正在进行立即关闭）和ORA-28576（与外部程序的RPC连接丢失）是可重启错误的示例。ORA-06520（错误加载外部库）是不可重新启动的错误的示例。

Messaging Gateway使用数据库关闭触发器。如果Messaging Gateway代理正在关闭的实例上运行，则触发器会通知代理关闭，并且在收到通知后，代理将终止当前运行。作业计划程序将自动安排作业在将来再次运行。

如果由于数据库故障或代理作业检测到可重新启动的错误而导致Messaging Gateway代理作业实例结束，则不会删除作业，并且作业计划程序将自动安排作业在将来再次运行。

19.2.4在Oracle RAC上运行Oracle Messaging Gateway代理

虽然Messaging Gateway作业启动和关闭原则对于Oracle Real Application Clusters（Oracle RAC）和非Oracle RAC环境是相同的，但Oracle RAC环境还是要记住一些事项。

即使在Oracle RAC环境中，也可以运行每个已配置的Messaging Gateway代理的单个进程。例如，如果已使用代理用户配置了默认代理和两个命名代理，则可以同时运行所有三个代理中的一个实例。与每个代理关联的数据库服务确定代理的Scheduler作业的服务亲和性，从而确定代理进程可以运行的数据库实例。

在Oracle RAC环境中关闭数据库实例时，仅当Messaging Gateway代理正在关闭的实例上运行时，Messaging Gateway关闭触发器才会通知代理关闭。作业计划程序将自动安排作业在将来再次运行，或者在另一个实例上运行，或者如果作业只能在正在关闭的实例上运行，则重新启动该实例。

Oracle建议将Messaging Gateway代理程序建立的所有数据库连接都发送到运行Messaging Gateway代理程序进程的实例。这可确保Oracle RAC环境中的正确故障转移行为。

如果Messaging Gateway代理已与数据库服务关联，则除非在正在运行的实例上当前启用该服务，否则代理的Scheduler作业将不会运行。关闭数据库时，Oracle会停止对该数据库的所有服务，您可能需要在启动数据库时手动重新启动服务。

19.3配置消息系统链接

作为非Oracle消息传递系统的客户端运行，Messaging Gateway代理通过消息传递系统链接与非Oracle消息传递系统进行通信。消息传递系统链接是Messaging Gateway代理与非Oracle消息传递系统之间的一组连接。

要配置非Oracle消息传递系统的消息传递系统链接，用户必须提供代理的信息以建立与非Oracle消息传递系统的连接。用户可以指定最大消息传递连接数。

代理名称将与每个邮件系统链接相关联。这是在创建链接时完成的，无法更改。然后，与链接相关联的代理负责处理使用与该链接相关联的已注册队列的所有传播作业。如果在创建邮件系统链接时未指定代理名称，则将使用Messaging Gateway默认代理。

为支持事务和持久性消息的非Oracle消息传递系统配置消息传递系统链接时，必须指定入站和出站传播的日志队列的本机名称，以保证一次性消息传递。日志队列只能由Messaging Gateway代理使用。没有其他程序可以将日志队列的消息入队或出列队列。入站日志队列和出站日志队列可以引用相同的物理队列，但如果它们引用不同的物理队列，则可以实现更好的性能。

唯一一个Messaging Gateway代理应该访问传播日志队列。这可以确保给定的日志队列仅包含该代理处理的那些传播作业的日志记录，并且代理可以自由地丢弃它可能遇到的任何其他日志记录。

配置消息传递系统链接时，用户还可以指定options参数。options参数是一组SYS.MGW_PROPERTY类型的{name，value}对。

19.3.3创建WebSphere MQ链接以使用SSL

Messaging Gateway允许对IBM WebSphere MQ和WebSphere MQ JMS连接进行SSL支持。本节介绍如何配置Messaging Gateway以将SSL用于WebSphere MQ Base Java链接，并且相同的信息适用于WebSphere MQ JMS链接。Messaging Gateway配置没有区别。

要将SSL用于WebSphere MQ连接，需要以下内容：

  • 配置为使用SSL的WebSphere MQ通道。

  • 信任库和可选的密钥库文件，位于Messaging Gateway代理进程可访问的位置。在Oracle RAC环境中，使用相同路径规范的Messaging Gateway代理进程可能运行的所有实例都必须可以访问这些文件。

  • 使用DBMS_MGWADM 。 CREATE_MSGSYSTEM_LINK使用所需的SSL相关链接选项创建WebSphere MQ链接。至少应设置MQ_SSLCIPHERSUITE属性以指定通道使用的SSL密码套件。

  • 使用DBMS_MGWADM 。 SET_OPTION为分配给链接的Messaging Gateway代理设置某些JSSE Java属性。

JSEE相关属性：

  • j ava.net.ssl.keyStore

    此属性用于指定密钥库的位置。密钥库是用于各种目的的密钥材料的数据库，包括身份验证和数据完整性。

  • java.net.ssl.keyStorePassword

    此属性用于指定密钥库的密码。此密码用于在访问密钥库之前检查密钥库中数据的完整性。

  • j ava.net.ssl.trustStore

    此属性用于指定信任库的位置。信任库是在决定哪些客户端和服务器受信任时使用的密钥库。

  • java.net.ssl.trustStorePassword

    此属性用于指定信任库的密码。此密码用于在访问信任库之前检查信任库中数据的完整性。

19.5.1传播工作概述

传播作业指定传播的消息以及消息的传播方式。

消息传递网关允许双向消息传播。出站传播将消息从Oracle数据库高级队列移动到非Oracle消息传递系统。入站传播将消息从非Oracle消息传递系统移动到Oracle数据库高级队列。

如果传播源是队列（点对点），则Messaging Gateway代理会将所有消息从源队列移动到目标队列。如果传播源是主题（发布/订阅），则Messaging Gateway代理会在消息传递系统中创建传播源队列的订户。在创建订户后，代理仅移动发布到源队列的消息。

传播邮件时，Messaging Gateway代理会将邮件从源邮件系统中的格式转换为目标邮件系统中的格式。用户可以通过提供消息定制消息转换变换 。如果消息转换失败，则消息将被移动到异常队列（如果已提供），以便代理可以继续为订阅者传播消息。

Messaging Gateway异常队列与Oracle Database Advanced Queuing异常队列不同。当邮件转换失败时，Messaging Gateway会将邮件移动到Messaging Gateway异常队列。在MAX_RETRIES对消息进行队列尝试后，Oracle数据库高级队列将消息移动到Oracle数据库高级队列异常队列。

移动到Oracle数据库高级队列异常队列的消息可能会导致关联的Messaging Gateway传播作业出现无法恢复的故障。为避免此问题，任何Oracle数据库高级队列队列的MAX_RETRIES参数（用作Messaging Gateway传播作业的传播源）都应设置为远大于16的值。

如果传播源队列的消息传递系统支持消息选择，则可以为传播订户指定消息选择规则。仅传播满足消息选择器的消息。

用户还可以指定传播作业选项以控制消息的传播方式，例如JMS消息传递模式和TIB / Rendezvous队列策略的选项。

19.5.6删除传播作业

您可以通过调用DBMS_MGWADM来删除Messaging Gateway传播作业。 REMOVE_JOB 。

  • 从消息传递系统中删除可能由Messaging Gateway创建的关联订户

  • 从正在删除的作业的日志队列中删除传播日志记录

Messaging Gateway可能无法执行清理，因为：

  • Messaging Gateway代理未运行

  • 非Oracle邮件系统未运行

  • Messaging Gateway代理无法与源或目标邮件系统进行交互

如果Messaging Gateway清理因任何原因失败，则正在删除的传播作业将处于DELETE_PENDING状态。在以下情况下，Messaging Gateway代理会尝试以DELETE_PENDING状态清除传播作业：

  • DBMS_MGWADM 。 REMOVE_JOB并且Messaging Gateway代理正在运行。

  • Messaging Gateway代理正在启动并找到DELETE_PENDING状态的传播作业。

DBMS_MGWADM 。 REMOVE_JOB有一个force参数，允许您强制从Messaging Gateway配置中删除传播作业，而不将其置于DELETE_PENDING状态。这在清除失败或您希望在Messaging Gateway代理未运行时删除传播作业时非常有用。

强制删除传播作业可能导致在日志队列中留下过时的日志记录，并且消息传递系统中的订阅可能导致不必要的消息累积。Oracle建议不要将force选项用于DBMS_MGWADM 。如果可能的话， REMOVE_JOB 。

DBMS_MGWADM 。 REMOVE_JOB过程可用于删除新样式和旧样式作业。支持强制和非强制删除。如果在完成非强制删除时Messaging Gateway代理未运行，则该作业将被标记为删除暂挂，并且此时不会删除基础订户和计划。重新启动代理程序并执行清理工作或强制DBMS_MGWADM将删除作业（订户/计划对）。执行REMOVE_JOB 。为了确保同时删除订户/计划对，如果首次调用DBMS_MGWADM ，则会发生错误。 REMOVE_JOB然后尝试调用DBMS_MGWADM 。 REMOVE_SUBSCRIBER或DBMS_MGWADM 。 UNSCHEDULE_PROPAGATION用于旧式作业。

MQ_BrokerControlQueue

WebSphere MQ JMS使用此属性。它指定代理控制队列的名称，并对应于WebSphere MQ JMS管理工具属性BROKERCONQ 。 WebSphere MQ缺省值为SYSTEM.BROKER.CONTROL.QUEUE 。

MQ_BrokerPubQueue

WebSphere MQ JMS使用此属性。它指定代理发布队列的名称，并对应于WebSphere MQ JMS管理工具属性BROKERPUBQ 。 WebSphere MQ缺省值为SYSTEM.BROKER.DEFAULT.STREAM 。

MQ_BrokerQueueManager

WebSphere MQ JMS使用此属性。它指定代理队列管理器的名称，并对应于WebSphere MQ管理工具属性BROKERQMGR 。如果未设置，则不使用默认值。

MQ_BrokerVersion

WebSphere MQ JMS使用此属性。它指定代理版本号，并对应于WebSphere MQ JMS管理工具属性BROKERVER 。 WebSphere MQ缺省值为0 。

MQ_ccdtUrl

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指定要使用的客户机通道定义表（CCDT）的URL字符串。如果未设置，则不使用CCDT。如果使用CCDT，则使用SYS 。 MGW_MQSERIES_PROPERTIES 。 channel link属性必须为NULL 。

MQ_ccsid

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指定用于转换WebSphere MQ消息头中的信息的字符集标识符。这应该是字符集的整数值（例如， 819 ），而不是描述性字符串。如果未设置，则使用WebSphere MQ缺省字符集819 。

MQ_JmsDurSubQueue

WebSphere MQ JMS使用此属性。它仅适用于WebSphere MQ JMS主题链接。SYS.MGW_MQSERIES_PROPERITES属性inbound_log_queue和outbound_log_queue指定用于传播日志记录的WebSphere MQ JMS主题的名称。此属性指定日志主题订阅者从中检索持久订阅消息的WebSphere MQ队列的名称。WebSphere MQ缺省队列是SYSTEM.JMS.D.SUBSCRIBER.QUEUE 。

MQ_PubAckInterval

WebSphere MQ JMS使用此属性。它指定需要来自代理的确认并且对应于WebSphere MQ JMS管理工具属性PUBACKINT发布请求之间的间隔（消息数）。 WebSphere MQ缺省值为25 。

MQ_ReceiveExit

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指定实现MQReceiveExit接口的类的完全限定Java类名。此类必须位于Messaging Gateway代理的CLASSPATH中。没有默认值。

MQ_ReceiveExitInit

WebSphere MQ JMS使用此初始化字符串。它由WebSphere MQ JMS传递给MQ_ReceiveExit指定的类的构造函数，并对应于WebSphere MQ JMS管理工具属性RECEXITINIT 。没有默认值。

MQ_SecurityExit

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指定实现MQSecurityExit接口的类的完全限定Java类名。此类必须位于Messaging Gateway代理的CLASSPATH中。没有默认值。

MQ_SecurityExitInit

WebSphere MQ JMS使用此初始化字符串。它由WebSphere MQ JMS传递给MQ_SecurityExit指定的类的构造函数，并对应于WebSphere MQ JMS管理工具属性SECEXITINIT 。没有默认值。

MQ_SendExit

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指定实现MQSendExit接口的类的完全限定Java类名。此类必须位于Messaging Gateway代理的CLASSPATH中。没有默认值。

MQ_SendExitInit

WebSphere MQ JMS使用此初始化字符串。它由WebSphere MQ JMS传递给MQ_SendExit指定的类的构造MQ_SendExit 。它对应于WebSphere MQ JMS管理工具属性SENDEXITINIT 。没有默认值。

MQ_SSLCipherSuite

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指定要使用的CipherSuite;例如， SSL_RSA_WITH_RC4_128_MD5 。这对应于WebSphere MQ SSLCIPHERSUITE管理属性。

MQ_SSLCrl

WebSphere MQ JMS使用此属性。它指定了一个以空格分隔的LDAP服务器列表，可用于证书吊销列表（CRL）检查。如果未设置，则不进行CRL检查。这对应于WebSphere MQ SSLCRL管理属性。WebSphere MQ Base Java不支持此选项，而是在需要CRL检查时必须使用客户机通道定义表（CCDT）。

MQ_SSLFipsRequired

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指示IBM Java JSSE FIPS提供程序（IBMSJSSEFIPS）是否支持正在使用的CipherSuite。值应为TRUE或FALSE 。默认值为FALSE 。这对应于WebSphere MQ SSLFIPSREQUIRED管理属性。

MQ_SSLPeerName

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指定队列管理器证书必须匹配的可分辨名称（DN）模式，以便建立连接。如果未设置，则不执行DN检查。这对应于WebSphere MQ SSLPEERNAME管理属性。

MQ_SSLResetCount

WebSphere MQ Base Java和WebSphere MQ JMS使用此属性。它指定在重新协商密钥之前发送和接收的总字节数。如果未设置，则不会重新协商密钥。这对应于WebSphere MQ SSLRESETCOUNT管理属性。

19.7.4可选的外部队列配置属性

MQ_CharacterSet

WebSphere MQ JMS使用此属性。它仅用于向JMS队列或主题的出站传播。它指定用于编码发送到目标的文本字符串的字符集。它应该是字符集的整数值（例如， 1208 ）而不是描述性字符串。Messaging Gateway使用的默认值为1208 （UTF8）。

MQ_JmsDurSubQueue

WebSphere MQ JMS使用此属性。它是一个字符串，表示订阅者在此主题上从中检索持久订阅消息的WebSphere MQ队列的名称。它仅适用于WebSphere MQ JMS主题。WebSphere MQ缺省队列是SYSTEM.JMS.D.SUBSCRIBER.QUEUE 。

MQ_JmsTargetClient

WebSphere MQ JMS使用此属性。它仅用于向JMS队列或主题的出站传播。支持的值为TRUE和FALSE 。 TRUE表示WebSphere MQ应将消息存储为JMS消息。FALSE表示WebSphere MQ应以非JMS格式存储消息，以便非JMS应用程序可以访问它。默认值为TRUE 。

MQ_openOptions

WebSphere MQ Base Java使用此属性。它指定用于WebSphere MQ Base Java MQQueueManager.accessQueue方法的openOptions参数的MQQueueManager.accessQueue 。没有价值。但是如果给出了一个，则Messaging Gateway代理会将MQOO_OUTPUT添加到enqueue（ put ）操作的指定值。为出列（ get ）操作添加了MQOO_INPUT_SHARED 。对于enqueue / put操作，默认值为MQOO_OUTPUT ; MQOO_INPUT_SHARED用于出队/获取操作。

19.7.5可选的作业配置属性

本节介绍可以使用DBMS_MGWADM的options参数指定的可选传播作业属性。 CREATE_JOB和DBMS_MGWADM 。 ALTER_JOB 。

AQ_MsgProperties

TIB / Rendezvous使用此属性。它指定了在消息传播期间如何使用Oracle Database AQ消息属性。支持的值为TRUE和FALSE 。默认值为FALSE 。

对于出站传播作业，如果值为TRUE （不区分大小写），则Messaging Gateway代理会将大多数Oracle Database AQ消息属性的字段添加到传播到TIB / Rendezvous主题的消息。

对于入站传播作业，如果值为TRUE （不区分大小写），则Messaging Gateway代理将在源消息中搜索具有保留名称的字段，如果存在，则使用其值设置相应的Oracle Database AQ消息属性。如果该字段不存在或者没有预期的数据类型，则将使用默认值。

JMS_DeliveryMode

WebSphere MQ JMS和Oracle JMS使用此属性。传播目标是JMS消息传递系统时，可以使用此属性。它设置JMS MessageProducer排队到传播目标队列的消息的传递模式。默认值为PRESERVE_MSG 。支持的值及其关联的交付模式为：

  • PERSISTENT （ DeliveryMode.PERSISTENT ）

  • NON_PERSISTENT （ DeliveryMode.NON_PERSISTENT ）

  • PRESERVE_MSG （使用源JMS消息的传递模式）

JMS_NoLocal

WebSphere MQ JMS和Oracle JMS使用此属性。当传播源是JMS消息传递系统时，您可以使用它。它设置JMS TopicSubscriber的noLocal参数。TRUE表示不会传播通过相同的Messaging Gateway链接发布到此主题的消息。默认值FALSE表示将从主题传播此类消息。

MsgBatchSize

任何受支持的邮件系统都可以使用此属性。它指定在一个事务中传播的最大消息数（如果有）。默认值为30。

PreserveMessageID

WebSphere MQ Base Java，WebSphere MQ JMS，TIB / Rendezvous和Oracle JMS使用此属性。它指定当邮件传播到目标邮件系统时，Messaging Gateway是否应保留原始邮件标识符。具体细节取决于所涉及的消息传递系统的功能。支持的值为TRUE和FALSE 。默认值为FALSE 。

RV_discardAmount

TIB / Rendezvous使用此属性。它指定队列的丢弃量。它仅对入站传播作业有意义。默认值为0。

RV_limitPolicy

TIB / Rendezvous使用此属性。它指定了解决队列限制溢出的限制策略。它仅对入站传播作业有意义。默认值为DISCARD_NONE 。支持的值及其关联的限制策略包括： DISCARD_NONE ， DISCARD_FIRST ， DISCARD_LAST和DISCARD_NEW 。

  • DISCARD_NONE （ TibrvQueue.DISCARD_NONE ）

  • DISCARD_FIRST （ TibrvQueue.DISCARD_FIRST ）

  • DISCARD_LAST （ TibrvQueue.DISCARD_LAST ）

  • DISCARD_NEW （ TibrvQueue.DISCARD_NEW ）

RV_maxEvents

TIB / Rendezvous使用此属性。它指定队列的最大事件限制。它仅对入站传播作业有意义。默认值为0。
