13.1关于Oracle GoldenGate

Oracle GoldenGate是一个异构复制系统，集成了对Oracle和其他数据库之间复制的支持。

Oracle GoldenGate与Oracle数据库集成，可以从重做日志中捕获DML和DDL更改，并在目标上应用更改。Oracle GoldenGate实时跨Oracle数据库或异构数据库提供事务数据的捕获，路由，转换和交付。

当Oracle GoldenGate捕获源自Oracle数据库的更改时，每个更改都封装在行逻辑更改记录（LCR）中。行LCR是DML行更改的结构化表示。每行LCR包括操作类型，旧列值和新列值。多行LCR可以是单个数据库事务的一部分。

13.2关于自动冲突检测和解决方案

当Oracle GoldenGate在Oracle数据库之间复制更改时，您可以在Oracle数据库中配置和管理Oracle GoldenGate自动冲突检测和解决方案。

注意：

本书中的文档适用于Oracle数据库12 c第2版（12.2）及更高版本中特定的自动冲突检测和解决功能，该功能在Oracle数据库中配置。它还需要Oracle GoldenGate 12.3及更高版本。还有一个用于冲突检测和解决的一般Oracle GoldenGate功能，称为Oracle GoldenGate冲突检测和解决（CDR）。Oracle GoldenGate CDR在Replicat参数文件中配置，并在Oracle GoldenGate文档中进行了说明。

您只能为单个表配置以下类型的自动冲突检测和解决方案之一：

  • Oracle Database 12 c第2版（12.2）特有的自动冲突检测和解决功能

  • Oracle GoldenGate CDR

  • 自动冲突检测和解决方案
    您可以在Oracle GoldenGate配置中配置自动冲突检测和解决方案，该配置可在Oracle数据库之间复制表。要配置冲突检测和解决的一个表，调用ADD_AUTO_CDR在程序DBMS_GOLDENGATE_ADM包。
  • 最新时间戳冲突检测和解决方案
    使用时间戳冲突检测时，如果行逻辑更改记录（行LCR）的旧列列表中的时间戳与表中相应行的时间戳不同，则会发生冲突。
  • Delta冲突检测和解决方案
    使用delta冲突检测时，如果行逻辑更改记录（行LCR）的旧列列表中的值与表中相应行的值不同，则会发生冲突。
  • 列组
    列组是复制表中一个或多个列的逻辑分组。添加列组时，将对列组中的列执行冲突检测和解决，与表中的其他列分开。

13.2.1自动冲突检测和解决方案

您可以在Oracle GoldenGate配置中配置自动冲突检测和解决方案，该配置可在Oracle数据库之间复制表。要配置冲突检测和解决的一个表，调用ADD_AUTO_CDR在程序DBMS_GOLDENGATE_ADM包。

自动冲突检测和解决不需要更改应用程序，原因如下：

  • Oracle数据库自动创建和维护不可见的时间戳列。

  • 插入，更新和删除使用delete tombstone日志表来确定是否删除了行。

  • 可以检测到LOB列冲突。

  • Oracle数据库自动配置所需列的补充日志记录。

需要补充日志记录以确保每行LCR具有检测和解决冲突所需的信息。当对表执行DML操作时，补充日志记录会在重做日志中为表的列放置其他信息。为Oracle GoldenGate冲突检测和解决配置表时，将自动为表中的所有列配置补充日志记录。

13.2.2最新时间戳冲突检测和解决方案

使用时间戳冲突检测时，如果行逻辑更改记录（行LCR）的旧列列表中的时间戳与表中相应行的时间戳不同，则会发生冲突。

当您运行ADD_AUTO_CDR在程序DBMS_GOLDENGATE_ADM包配置表进行自动的Oracle GoldenGate冲突检测和解决，一个隐藏的时间戳列添加到表。此隐藏时间戳列记录行更改的时间，此信息用于检测和解决冲突。

应用行LCR时， INSERT ， UPDATE或DELETE操作可能会发生冲突。下表描述了每种类型的冲突以及如何使用最新的时间戳冲突检测和解决方案解决冲突。

表13-1最新时间戳冲突检测和解决方案

手术 冲突检测 解决冲突

插入

当表具有与列LCR中的新值相同的键列值时，将检测到冲突。

如果行LCR的时间戳晚于表行中的时间戳，则行LCR中的值将替换表中的值。

如果行LCR的时间戳早于表行中的时间戳，则丢弃行LCR，并保留表值。

UPDATE

在以下每种情况下都会检测到冲突：

  • 行LCR中的时间戳值与表中相应行的时间戳值不匹配。

  • 行LCR中的列组中的旧值与相应表行中的列值不匹配。列组是复制表中一个或多个列的逻辑分组。

  • 表行不存在。如果该行位于逻辑删除表中，则将其称为更新 - 删除冲突。

如果值不匹配且行LCR的时间戳晚于表行中的时间戳，则行LCR中的值将替换表中的值。

如果存在值不匹配且行LCR的时间戳早于表行中的时间戳，则丢弃行LCR，并保留表值。

如果表行不存在且行LCR的时间戳晚于逻辑删除表行中的时间戳，则行LCR从UPDATE操作转换为INSERT操作并插入表中。

如果表行不存在且行LCR的时间戳早于逻辑删除表行中的时间戳，则丢弃行LCR。

如果表行不存在且逻辑删除表中没有对应的行，则行LCR将从UPDATE操作转换为INSERT操作并插入表中。

删除

在以下每种情况下都会检测到冲突：

  • 行LCR中的时间戳值与表中相应行的时间戳值不匹配。

  • 表行不存在。

如果行LCR的时间戳晚于表中的时间戳，则从表中删除该行。

如果行LCR的时间戳早于表中的时间戳，则丢弃行LCR，并保留表值。

如果删除成功，则通过将行LCR插入逻辑删除表来记录行LCR。

如果表行不存在，则通过将行LCR插入逻辑删除表来记录行LCR。

13.2.3 Delta冲突检测和解决方案

使用delta冲突检测时，如果行逻辑更改记录（行LCR）的旧列列表中的值与表中相应行的值不同，则会发生冲突。

要配置三角洲冲突检测和解决的一个表，运行ADD_AUTO_CDR_DELTA_RES在程序DBMS_GOLDENGATE_ADM包。增量分辨率方法不依赖于时间戳或额外的分辨率列。使用delta冲突解决方案，可以通过将行LCR中的旧值和旧值之间的差值添加到表中的值来解决冲突。此解决方法通常用于财务数据，例如帐户余额。例如，如果同时在两个站点更新银行余额，则融合值会计入所有借方和贷方。

“图13-1 Delta冲突检测和分辨率”的说明

此示例显示正在数据库A和数据库B中复制的行。“ Balance列指定为执行增量冲突解决的列， TS1列是不可见的时间戳列，用于跟踪每次更改“ Balance列的时间。几乎同时对两个数据库中的行中的Balance值进行更改（数据库A中的@T20和数据库B中的@T22 ）。这些更改会导致冲突，并且delta冲突解决方案用于以下列方式解决冲突：

  • 在数据库A中， Balance的值从100更改为110 。因此，该值增加了10。

  • 在数据库B中， Balance的值从100更改为120 。因此，该值增加了20。

  • 要解决数据库A中的冲突，请将行LCR中的新旧值与表中的值之间的差值。LCR中新旧值之间的差异为20（120-100 = 20）。因此，表（110）中的当前值增加20，使得冲突解决后的值为130。

  • 要解决数据库B中的冲突，请将行LCR中的新旧值与表中的值之间的差值。LCR中新旧值之间的差异为10（110-100 = 10）。因此，表（120）中的当前值增加10，使得冲突解决后的值为130。

在delta冲突解决后， Balance列的值对于数据库A和数据库B中的行是相同的。

13.2.4列组

列组是复制表中一个或多个列的逻辑分组。添加列组时，将对列组中的列执行冲突检测和解决，与表中的其他列分开。

使用ADD_AUTO_CDR过程为Oracle GoldenGate冲突检测和解析配置表时，表中的所有标量列都将添加到默认列组。要为表定义其他列组，请运行ADD_AUTO_CDR_COLUMN_GROUP过程。表中不属于用户定义列组的任何列都保留在表的默认列组中。

列组使不同的数据库几乎同时更新同一行中的不同列，而不会导致冲突。为表配置列组时，即使不同的数据库更新表中的同一行，也可以避免冲突。如果更新更改了不同列组中列的值，则不会检测到冲突。下图提供了一个说明列组的示例。

图13-2列组


“图13-2列组”的说明

此示例显示正在数据库A和数据库B中复制的行。为每个数据库上的复制表配置以下两个列组：

  • 一个列组包括Office列。此列组的不可见时间戳TS1 。

  • 另一个列组包括Title和Salary列。此列组的不可见时间戳列是TS2 。

这些列组使数据库A和数据库B几乎同时更新同一行而不会引起冲突。具体而言，进行了以下更改：

  • 在数据库A中， Office的值已从1080更改为1030 。

  • 在数据库B中， Title的值从MTS1变为MTS2 。

由于Office列和Title列位于不同的列组中，因此将复制更改而不会检测到冲突。结果是，在复制每个更改后，两个数据库中行中的值相同。

分段LOB更新

由LOB WRITE ， LOB ERASE和LOB TRIM组成的一组lob操作是分段LOB更新。如果为包含LOB列的表配置了冲突检测和解决方案，则每个LOB列都放在其自己的列组中，并且列组具有自己的隐藏时间戳列。时间戳列在第一个分段LOB操作上更新。

对于LOB列，将通过以下方式检测并解决冲突：

  • 如果LOB列组的时间戳晚于行中相应的LOB列组，则应用分段LOB更新。

  • 如果LOB列组的时间戳早于行中相应的LOB列组，则保留表行中的LOB。

  • 如果表中不存在该行，则会引发错误。

13.3.1配置最新时间戳冲突检测和解决方案

该ADD_AUTO_CDR在程序DBMS_GOLDENGATE_ADM包配置最新的时间戳冲突检测和解决。ADD_AUTO_CDR_COLUMN_GROUP过程添加了可选列组。

使用最新的时间戳冲突检测和解决方案，当行LCR的时间戳列与相应表行的时间戳不匹配时，将检测到冲突。如果时间戳稍后，则应用行LCR。否则，丢弃行LCR，并且不改变表行。运行ADD_AUTO_CDR过程时，它会为指定表中的每一行添加一个不可见的时间戳列，并配置时间戳冲突检测和解决方案。使用ADD_AUTO_CDR_COLUMN_GROUP过程添加一个或多个列组时，它会为列组添加时间戳，并为列组配置时间戳冲突检测和解决方案。
您可以使用配置的Oracle GoldenGate管理员GRANT_ADMIN_PRIVILEGE在程序DBMS_GOLDENGATE_ADM包。
  1. 以Oracle GoldenGate管理员身份连接到每个数据库。
  2. 运行ADD_AUTO_CDR过程并指定要配置的表以获取最新的时间戳冲突检测和解决方案。
  3. 可选： 运行ADD_AUTO_CDR_COLUMN_GROUP过程并在表中指定一个或多个列组。
  4. 在复制表的每个Oracle数据库中重复前面的步骤。

此示例配置hr.employees表的最新时间戳冲突检测和解决方案。 

BEGIN DBMS_GOLDENGATE_ADM.ADD_AUTO_CDR（schema_name =>'hr'，table_name =>'employees'）;结束; /

13.4管理自动冲突检测和解决方案

您可以使用DBMS_GOLDENGATE_ADM程序包管理Oracle数据库中的Oracle GoldenGate自动冲突检测和解决方案。

  • 改变表的冲突检测和解决方案
    DBMS_GOLDENGATE_ADM包中的ALTER_AUTO_CDR过程会改变表的冲突检测和解决方案。
  • 更改列组
    ALTER_AUTO_CDR_COLUMN_GROUP过程会更改列组。
  • 清除墓碑行
    PURGE_TOMBSTONES过程删除在指定日期和时间之前记录的逻辑删除行。此过程将删除为数据库中的冲突解决配置的所有表的逻辑删除行。
  • 从表中删除冲突检测和解决方案
    DBMS_GOLDENGATE_ADM包中的REMOVE_AUTO_CDR过程从表中删除自动冲突检测和解决方案。此过程还会删除为表配置的所有列组和delta冲突检测和解决方案。
  • 删除列组
    REMOVE_AUTO_CDR_COLUMN_GROUP过程删除列组。
  • 删除Delta冲突检测和解决方案
    该REMOVE_AUTO_CDR_DELTA_RES在程序DBMS_GOLDENGATE_ADM包删除增量冲突检测和解决为一列。

13.4.3清除墓碑行

例13-6清除墓碑行

EXEC DBMS_GOLDENGATE_ADM.PURGE_TOMBSTONES（'2015-12-01 15：00：00.000000 EST'）;
