59 NNL-00001至NNL-01078

NNL-00001：请求参数“ string ”无效

原因：用户输入了无效的请求命令参数。

行动：提供有效的论据。

NNL-00002：此操作不允许请求参数“ string ”

原因：用户输入了请求命令参数，该参数对于特定请求命令无效。

行动：提供有效的论据。

NNL-00003：内存耗尽

原因：程序内存不足

操作：检查计算机上的可用内存量，以确保有足够的内存来运行此可执行文件。如有必要，通过运行较少的程序释放内存，然后再试一次。

NNL-00004：无效的命令参数“ string ”

原因：用户输入了无效的命令参数。

行动：提供有效的论据。

NNL-00005：未设置任何服务器。首先使用“SET SERVER”命令

原因：用户尝试发出一个命令，该命令要求首先设置默认的托管Oracle名称服务器。

操作：使用SET SERVER命令设置默认服务器，然后重新发出该命令。或者，如果命令允许，则使用特定服务器名称重新发出它，以便控制程序不需要使用默认值。

NNL-00006：超时，服务器没有响应

原因：控制程序向Oracle Names服务器发出请求，没有及时响应。控制程序和服务器之间的网络连接可能有故障，服务器可能负载很重，或者控制程序和服务器之间可能存在大量网络延迟。后者对于WAN链接尤其如此。

行动：重新发布命令;如果它仍然失败，请检查正在管理的服务器是否正常运行，以及服务器和控制程序之间的网络连接是否正常工作。

NNL-00007：服务器响应错误

原因：控制程序向Oracle Names服务器发出请求，服务器响应错误。

操作：有关更多详细信息，请启用跟踪并重新执行失败操作。如果错误仍然存在，请联系全球客户支持。

NNL-00008：服务器无法理解请求

原因：控制程序向Oracle Names服务器发出请求，服务器不理解该请求。

操作：如果服务器和控制程序具有不同的版本，则它们可能无法通信。检查文档中的版本兼容性约束。如果服务器和控制程序兼容，请打开跟踪以获取更多详细信息，然后重新执行失败的操作。如果错误仍然存在，请联系全球客户支持。

NNL-00009：服务器密码加密失败

原因：控制程序尝试并无法加密通过SET PASSWORD命令或namesctl.server_password配置参数提供的服务器密码。

操作：用户通常看不到的内部错误。联系全球客户支持。

NNL-00010：获取服务器“ string ”的地址时出错

原因：控制程序尝试将服务器名称转换为其对应的地址，但失败。失败的原因很多。控制程序的SQLNET.ORA文件必须存在于正确的系统特定位置，并且必须由控制程序读取。该文件必须包含具有有效名称服务器地址的names.preferred_servers条目。如果用户使用SET SERVER命令更改了受管服务器，则新服务器必须正在运行，并且必须能够解析该地址或将请求转发到可以解析该地址的另一台服务器。或者，具有有效名称服务器名称和地址的TNSNAMES.ORA文件必须存在于正确的系统特定位置。

操作：有关更多详细信息，请启用跟踪并重新执行失败操作。如果错误仍然存在，请联系全球客户支持。

NNL-00011：不存在名称为“ string ”的服务器

原因：用户发出了SET SERVER请求，并且当前托管的Oracle Names服务器无法将该名称转换为地址。或者，用户向特定服务器发出控制程序请求，并且该服务器的名称不能被转换为地址。这两个问题几乎总是由错误输入服务器名称引起的。

操作：使用有效的服务器名称重新发出请求。

NNL-00012：无效的“ string ”值“ string ”

原因：第一个参数是数据类型，第二个参数是值。用户提供了指定类型的无效数据值。

操作：使用正确的数据重新发出失败的操作。

NNL-00013：未执行，已提供密码的权限被拒绝

原因：用户向Oracle Names服务器发出了请求，并且随请求提供的密码不正确。

操作：检查控制程序发送到服务器的密码是否适用于该服务器。您可以使用SET PASSWORD控制程序命令设置新密码。

NNL-00014：未知数据类型“ string ”

原因：用户发出了包含未知数据类型的名称服务器请求。

操作：服务器可能没有关于类型的信息，但会自动提取它以响应失败的请求。重新发出失败的请求;如果服务器在此期间获取了相应的数据类型信息，则错误将消失。如果错误仍然存在，则数据类型可能无效。

NNL-00015：类型“ string ”具有“ string ”语法，没有文字表示

原因：并非所有数据类型语法都可以表示为文本。此消息表示用户为具有非文本表示的类型发出了名称服务器请求。

操作：使用具有文本表示的语法的数据类型重新发出操作。

NNL-00016：无法启动服务器，错误获取程序目录的名称

原因：用户发出了STARTUP请求，但控制程序无法启动服务器，因为它无法找到服务器可执行文件所在目录的名称。

操作：用户通常看不到的内部错误。联系全球客户支持。

NNL-00017：无法启动服务器，程序“ string ”不存在

原因：用户发出STARTUP请求，但控制程序无法启动服务器，因为它无法找到服务器可执行文件。

操作：确保系统的oracle可执行文件目录中存在服务器可执行文件的副本。

NNL-00018：警告：无法联系默认名称服务器

原因：控制程序在启动期间无法联系其默认名称服务器。导致此错误的原因有很多种。控制程序的SQLNET.ORA文件必须存在于正确的系统特定位置，并且必须由控制程序读取。该文件必须包含具有有效名称服务器地址的names.preferred_servers条目。首选服务器列表中的第一个服务器必须正在运行，并且控制程序和该服务器之间必须存在网络连接。

行动：检查以上所有情况是否属实。如果控制程序仍无法联系其默认服务器，请打开跟踪并重新启动控制程序。如果错误仍然存在，请联系全球客户支持。

NNL-00019：警告：服务器已更改或忽略您的请求

原因：用户向Oracle Names服务器发出请求，并且该请求以某种方式被更改。通常，用户指定了某种类型的无效参数，服务器要么忽略该请求，要么使用合理的默认值替换invalid参数。

操作：使用控制程序检查您尝试更改的值的当前状态。服务器的日志文件还可能包含有关问题的确切原因的信息。使用正确的参数重新发出操作。

NNL-00020：警告：没有TNSNAMES.ORA文件，SQLNET.ORA中没有首选名称服务器

原因：控制程序无法找到TNSNAMES.ORA文件，无法在SQLNET.ORA中找到默认名称服务器。控制程序将在此状态下运行，但在设置默认服务器之前将无法解析任何名称，并且必须使用其原始TNS地址设置该服务器。

操作：控制程序的SQLNET.ORA文件必须存在于正确的系统特定位置，并且必须可由控制程序读取。该文件必须包含具有有效名称服务器地址的names.preferred_servers条目。首选服务器列表中的第一个服务器必须正在运行，并且控制程序和该服务器之间必须存在网络连接。或者，具有有效名称服务器信息的TNSNAMES.ORA文件必须存在于正确的系统特定位置。

NNL-00022：错误将跟踪级别从string更改为string ，忽略级别更改

原因：用户尝试更改控制程序的本地跟踪级别，并且程序在更改级别时遇到错误。通常问题是第一次启用跟踪（通过将跟踪级别从OFF更改为其他内容）并且跟踪文件不能由控制程序写入。

操作：检查跟踪文件是否可写。您的文档将告诉您跟踪文件的名称。
