5 DRDA的Oracle数据库提供程序的管理和自定义

考虑各种管理和定制问题。

使用Oracle数据库提供程序为DRDA迁移步骤

虽然将现有DB2应用程序迁移到Oracle数据库是特定于数据和目标的，但一般方法有以下六个步骤：

  1. 为DRDA软件安装和配置Oracle数据库提供程序

  2. 在Oracle数据库中为DRDA对象安装Oracle数据库提供程序

  3. 管理DRDA包权限

  4. 迁移DB2数据

  5. 重新定位应用程序

  6. 调整SQL转换和数据类型

使用Oracle数据库提供程序进行DRDA的注意事项

在为DRDA软件安装Oracle数据库提供程序之前，组织必须考虑几个操作和资源问题。在确定最佳安装是作为独立的Oracle主目录，现有Oracle数据库中的Oracle主目录，还是完全独立于Oracle数据库的计算机上时，计算机和网络资源的灵活性和性能至关重要。此外，必须使用安装的所有可能DB2客户端的性质是一个决定性因素;在此上下文中，DB2被视为客户端。

为DRDA安装Oracle数据库提供程序的先决条件

在Oracle数据库中为DRDA对象安装Oracle数据库提供程序之前，必须将一个或多个用户指定为DRDA管理员，并具有管理员角色。

同样，指定将通过Oracle数据库提供程序访问Oracle数据库以获取DRDA或DB2应用程序的用户，并向他们授予DRDA用户的角色和权限。设置DRDA用户权限和配置的某些方面可能需要延迟，直到迁移过程进一步。这主要涉及应用程序使用的特定DRDA包，以及任何特定的SQL转换或数据类型调整。如果在迁移之前识别应用程序的包，则这些包可以作为包授权工作流的一部分应用。

管理DRDA包权限

为了通过适用于DRDA的Oracle数据库提供程序从DRDA或DB2应用程序成功访问Oracle数据库，必须具有程序包授权。至少，必须收集有关应用程序及其用户的以下信息：

  • 包集合ID，例如NULLID

  • 包名称，例如DSNPBD3

  • 包版本名称（如果适用），例如01或NULL

  • 必须访问数据库的Oracle用户的名称，例如DRDAUSR

还必须为包表示的应用程序指定SQL转换配置文件名称。

迁移DB2数据

在DB2中，可以在任意模式下创建对象，而在Oracle数据库中模式名称不是任意的。因此，在将数据从DB2迁移到Oracle时，必须仔细考虑模式。在Oracle中，必须在实际用户的模式中分配所有模式对象，例如表，视图，同义词等。这显然会影响这些对象的命名，创建和访问方式。

请考虑以下示例： USER1创建表"USER1"."TABLE1"和"USER2"."TABLE2" 。在DB2中， TABLE1和TABLE2归USER1 ，因为USER1是它们的创建者。在Oracle中，表"USER2"."TABLE2"由用户USER2拥有。此外， USER1不能创造TABLE2除非USER1具有CREATE ANY TABLE privelege。相反， USER2必须创建TABLE2 ，然后授予USER1访问权限。

从DB2迁移到Oracle的数据也必须根据Oracle数据类型进行定义。虽然Oracle使用ANSI定义的数据类型名称，但它们不一定具有与DB2实现相同的范围限制或语义。要准确建模现有DB2应用程序数据类型，请查看适用于DRDA的Oracle数据库提供程序数据字典部分。

在使用适当的数据类型创建模式和对象后，可以将数据导入Oracle。

重新定位应用程序以使用Oracle数据库

以下示例显示如何迁移DB2 z / OS应用程序。迁移DB2 / LUW或DB2 / 400应用程序时，您需要遵循类似的步骤。有关每个产品的等效步骤的详细信息，请参阅IBM文档。

注意：

此示例仅适用于DB2 for z / OS。

应用程序有两大类： 本机应用程序和远程应用程序 。

重新定位原生应用程序

典型的DB2应用程序称为本机，因为它们通过内部IPC机制直接与本地DB2系统交互。这些应用程序使用嵌入式SQL编程，并使用DB2 SQL PreProcessor。预处理源生成执行计划 ，该计划存储在数据库资源模块（DBRM）中。用户必须在程序运行之前上载或将执行计划绑定到本地DB2实例。

执行计划包含应用程序源中嵌入的所有静态SQL，以及其他属性，如位置 ，也称为当前服务器。默认情况下，当前服务器为空;这表明服务器位于本地DB2实例上。但是，可以通过为Current Server属性设置新值来重新定位执行计划以在另一台服务器上运行所有操作。

以下步骤应由IBM DB管理员执行。

  1. 在DB2 Communications Database中创建位置条目。

    DB2有一个内部通信系统，用于连接远程DB2实例。要处理远程实例，请将记录插入SYSIBM.IPNAMES表， SYSIBM.LOCATIONS表，也可以插入SYSIBM.USERNAMES表。

    有关DB2 Communications Database工具的说明，请参阅IBM DB2文档。

    以下命令在DB2 Communications Database中插入链接名REMHOST ，位置条目DRDAAS和可选的用户名映射条目。linkname指定运行Oracle数据库提供程序以获取DRDA的计算机的主机名或IP地址。该位置指定一个RDB名称，该名称使用Oracle数据库提供程序为DRDA监听的链接名称和端口号。这些对应于用于DRDA配置参数DATA_PORT和RDB_MAP Oracle数据库提供程序。请注意，位置名称必须与RDB_MAP参数中指定的RDB()值完全匹配。 

    INSERT INTO SYSIBM.IPNAMES（LINKNAME，SECURITY_OUT，USERNAMES，IPADDR）VALUES（'REMHOST'，'P'，'O'，'remotehost.remotedomain.com'）;插入SYSIBM.LOCATIONS（LOCATION，LINKNAME，PORT）价值观（'DRDAAS'，'REHMOST'，'1446'）; INSERT INTO SYSIBM.USERNAMES（TYPE，AUTHID，LINKNAME，NEWAUTHID，PASSWORD）VALUES（'O'，''，'REMHOST'，'DRDAUSER'，'userpwd'）;
  2. 将应用程序计划远程绑定到适用于DRDA的Oracle数据库提供程序。

    插入位置条目后，您必须远程绑定应用程序执行计划。下面的代码结合计划DSNPBD3通过DSN命令处理器IKJEFT01 。请注意，位置DRDAAS作为集合ID的前缀。 

    BIND PACKAGE（DRDAAS.NULLID）成员（DSNPBD3） -  ACT（REP）ISO（CS）CURRENTDATA（YES）编码（EBCDIC）
  3. 将程序包本地绑定到当前服务器。

    远程绑定计划后，使用当前服务器选项重新绑定本地计划以重新执行目标。下面的代码结合计划DSNPBD3通过DSN命令处理器IKJEFT01 。

    请注意，必须通过程序包列表PKLIST在远程计划中引用该计划，并且必须同时指定程序包引用中的位置DRDAAS.NULLID.DSNPBD3 ，并指定包含该位置的CURRENTSERVER选项。 

    绑定计划（DSNPBD3） -  PKLIST（DRDAAS.NULLID.DSNPBD3） -  ACT（REP）ISO（CS）CURRENTDATA（是）编码（EBCDIC） -  CURRENTSERVER（DRDAAS）
  4. 远程绑定计划并在本地重新绑定后，应用程序使用计划DSNPBD3运行，隐式通过本地DB2与Oracle数据库提供程序进行DRDA远程连接，并远程执行计划的所有操作。本地DB2仍然是此配置中的传递协调器。

重新定位远程应用程序

远程应用程序通常不直接绑定到本地DB2。此类应用程序通常被称为网络感知或面向网络，并具有远程服务器位置配置属性，用于指定连接的内容和位置。

此类应用程序通过网络协议将Oracle数据库提供程序用于DRDA。重新定位此类应用程序很容易配置，如以下步骤所示。

  1. 更改应用程序的配置选项以使用Oracle数据库提供程序中为DRDA配置的主机名（或IP地址），端口号和RDB名称。一个例子是通过ODBC，其中DSN条目包含网络参数。

    在此示例中， Network和PortNumber参数对应于插入先前使用的DB2 Communication Database示例中的Linkname和Location条目。Database参数对应于Location名称。所有这些DATA_PORT与RDB_MAP配置的DRDA数据库提供程序的DATA_PORT和RDB_MAP参数相对应。

    以下是odbc.ini文件的示例。 

    [DRDAAS] Network = remotehost.remotedomain.com PortNumber = 1446数据库= DRDAAS
  2. 执行应用程序的包资源绑定操作。

    通常，这由应用程序本身隐式处理，或者记录为将应用程序访问和资源设置为远程DB2实例的一次性步骤。有关绑定说明的具体应用，请参阅文档。

翻译SQL语句和键入数据类型

某些应用程序可能具有超出SQL转换的自动转换机制的特定于DB2的SQL，或者可能期望查询中的特定列具有非常特定的数据类型。在这种情况下，可能需要手动插入SQL替换语句，或添加特定于项的数据类型操作。

例如，假设应用程序具有特定的SQL语句，该语句具有以下特定于DB2的语法： SELECT LOG2(COL1) FROM TABLE1 。要在Oracle中正常工作，需要将SQL转换为此语句： SELECT LOG10(COL1,2) FROM TABLE1 。

通过SQL Translation的Register工具，可以为此SQL语句注册直接转换，如注册SQL替换语句一节中所示。请注意，这必须由执行SQL语句的用户完成;请记住，SQL Translation Profile必须作为该用户的资源创建。

注册SQL转换器后，当应用程序发出原始SQL时，它会隐式转换为新的SQL和进程。

在某些非常特殊的情况下，应用程序客户端要求以非常特定的格式返回查询中的选定项的数据类型。

假设翻译查询SELECT LOG10(COL1,2) FROM TABLE1返回DECFLOAT34数据类型，但应用程序无法处理它，可以隐式地将数据类型强制转换为另一种兼容类型。

如果应用程序支持DOUBLE PRECISION数据类型，则可以使用TYPEMAP工具添加注册按需数据类型转换中描述的此特定强制。

注册SQL替换语句

示例5-1注册SQL替换语句

已为应用程序包分配了配置文件名称DB2ZOS 。

connect DRDAUSER / userpwd执行dbms_sql_translator.register_sql_translation（'DB2ZOS'，'SELECT LOG2（COL1）FROM TABLE1'，'SELECT LOG10（COL1,2）FROM TABLE1'）

注册按需数据类型转换

示例5-2注册按需数据类型转换

连接DRDAADM / adminpwd执行DBMS_DRDAAS_ADMIN.SET_TYPEMAP（'NULLID'，'DSNPBD3'，NULL，'TABLE1：LOG10（COL1,2）'，'NUMBER = DOUBLE'）执行DBMS_DRDAAS_ADMIN.SET_TYPEMAP（'NULLID'，'DSNPBD3'，NULL ，'TABLE1：LOG10（COL1,2）'，'NUMBER（0，-127）= DOUBLE'）
