12.1.1关于非CDB的资源管理器解决方案

在非CDB中，当数据库资源分配决策留给操作系统时，您可能会遇到工作负载管理问题。

这些问题包括：

  • 开销过高：

    当服务器进程数量很多时，Oracle数据库服务器进程之间的操作系统上下文切换会导致过多的开销。

  • 调度效率低下

    操作系统在保存锁存器时对数据库服务器进行计划，这是低效的。

  • 资源分配不当

    操作系统在所有活动进程中平均分配资源，并且不能将一个任务优先于另一个任务。

  • 无法管理特定于数据库的资源，例如并行执行服务器和活动会话

    使用资源管理器有助于通过允许数据库更好地控制硬件资源的分配方式来克服这些问题。在具有多个并发用户会话的环境中，这些会话运行具有不同优先级的作业，因此不应平等对待所有会话。资源管理器使您能够根据会话属性将会话分组，然后以优化应用程序环境的硬件利用率的方式将资源分配给这些组。

借助EM Express中的资源管理器功能，您可以：

  • 创建资源使用者组（使用者组），根据其处理需求将用户会话一起收集到资源使用者组（使用者组）中。

  • 通过将CPU的份额分配给不同的使用者组来设置分配可用CPU的CPU指令。例如，在数据仓库中，可以向ROLAP（关系在线分析处理）应用程序提供比批处理作业更多的共享。

  • 设置并行服务器指令，以限制使用者组成员执行的任何操作的并行度。

  • 设置并行服务器指令，用于管理并行语句队列中并行语句的顺序。来自关键应用程序的并行语句可以在来自低优先级用户组的并行语句之前排队。

  • 设置并行服务器指令，以限制一组用户可以使用的并行执行服务器的数量。这可确保所有可用的并行执行服务器不会仅分配给一组用户。

  • 设置失控查询指令，用于检测会话或调用何时消耗超过指定数量的CPU，物理I / O，逻辑I / O或已用时间，然后自动终止会话或调用，或切换到使用者组具有较低的资源分配或对组可以使用的CPU百分比的限制。逻辑I / O（也称为缓冲区I / O）是指缓冲区高速缓存中缓冲区的读写。当在内存中找不到请求的缓冲区时，数据库执行物理I / O将缓冲区从磁盘或闪存缓存复制到内存中，然后执行逻辑I / O以读取缓存的缓冲区。

12.1.2关于CDB的资源管理器解决方案

资源管理器可以为CDB提供更有效的资源使用。

当CDB的资源分配决策留给操作系统时，您可能会遇到以下工作负载管理问题：

  • PDB之间资源分配不当

    操作系统在所有活动进程中平均分配资源，并且不能将一个任务优先于另一个任务。因此，一个或多个PDB可能使用过多的系统资源，而其他PDB则缺乏资源。

  • 在单个PDB中不恰当地分配资源

    连接到单个PDB的一个或多个会话可能使用过多的系统资源，而连接到同一PDB的其他会话则缺乏资源。

  • PDB的性能不一致

    当其他PDB在不同时间竞争更多系统资源或更少系统资源时，单个PDB可能执行不一致。

  • 缺乏PDB的资源使用数据

    资源使用数据对于监控和调整PDB至关重要。如果非CDB是系统上运行的唯一数据库，则可以使用操作系统监视工具来收集非CDB的资源使用情况数据。但是，在CDB中，操作系统监视工具不再有用，因为系统上运行了多个PDB。

资源管理器通过允许CDB更多地控制如何在PDB之间和PDB内分配硬件资源来帮助克服这些问题。

在具有多个PDB的CDB中，一些PDB通常比其他PDB更重要。资源管理器使您可以确定特定PDB的资源使用的优先级并限制它们。

使用资源管理器，您可以：

  • 指定不同的PDB应接收不同的系统资源份额，以便为更重要的PDB分配更多资源

  • 限制特定PDB的CPU使用率

  • 限制特定PDB可以使用的并行执行服务器的数量

  • 限制特定PDB的内存使用量

  • 指定特定PDB所需的最小内存量

  • 限制连接到单个PDB的不同会话的资源使用情况

  • 限制特定PDB生成的I / O.

  • 监控PDB的资源使用情况

12.1.3关于消费者群体

资源使用者组（使用者组）是根据其处理需求组合在一起的用户会话的集合。创建会话时，会根据您设置的映射规则将其自动映射到使用者组。

由于资源管理器将资源（例如CPU）分配给使用者组，因此当会话成为使用者组的成员时，其资源分配由使用者组的分配确定。

数据字典中始终存在特殊的使用者组。它们无法修改或删除。他们是：

  • SYS_GROUP

    这是用户帐户SYS或SYSTEM创建的所有会话的初始使用者组。会话到使用者组映射规则可以覆盖此初始使用者组。

  • OTHER_GROUPS

    此使用者组包含尚未分配给使用者组的所有会话。每个资源计划（计划）都必须包含指向OTHER_GROUPS的指令。

任何活跃的非CDB计划中不得超过28个消费者群体，并且任何有效的PDB计划中不得超过8个消费者群体。

12.1.4关于计划

资源计划（计划）是指定如何分配资源的计划指令（指令）的容器。

除了为每个Oracle数据库预定义的计划外，您还可以创建任意数量的计划。

您可以为非CDB，CDB和PDB创建计划。但是，在特定的非CDB，CDB或PDB中，一次只有一个计划处于活动状态。当计划处于活动状态时，其每个指令都会控制资源分配。

在非CDB计划或PDB计划中，指令指定如何将资源分配给非CDB或PDB中的使用者组。

在CDB计划中，指令指定如何将资源分配给CDB中的PDB。

12.1.5关于指令

资源计划指令（指令）指定如何分配资源。

非CDB计划或PDB计划中的指令将消费者组与计划相关联，并指定如何将资源分配给该消费者组。根据属于计划的指令集将资源分配给使用者组。计划与其指令之间存在父子关系。每个指令引用一个使用者组，并且该计划没有两个指令可以引用同一个使用者组。

非CDB计划或PDB计划中的指令有几种方法可以限制消费者组的资源分配。例如，它可以控制使用者组获得的CPU占总CPU的百分比。

根据非CDB计划或PDB计划中的指令集将资源分配给消费者组。计划与其指令之间存在父子关系。每个指令引用一个使用者组，同一计划的两个指令不能引用同一个使用者组。

CDB计划中的指令指定如何将资源分配给CDB中的PDB。在CDB计划中，您可以为CDB中的无，部分或全部PDB明确定义指令。每个CDB计划还有一个PDB的默认指令。当CDB计划是活动计划时，默认指令用于没有为其明确定义的指令的任何PDB。

12.1.6关于非CDB计划

12.1.7关于管理CDB和PDB工作负载

在CDB中，您可以使用EM Express中的资源管理器功能来管理多个PDB中的多个工作负载，从而竞争系统和CDB资源。

在CDB中，您可以在两个基本级别上管理资源：

  • CDB级别：您可以管理争用系统和CDB资源的多个PDB的工作负载。您可以指定如何为PDB分配资源，还可以限制特定PDB的资源利用率。

  • PDB级别：您可以管理每个PDB中的工作负载。

资源管理器分两步分配资源：

  1. 它将系统资源的一部分分配给每个PDB。

  2. 在特定PDB中，它将步骤1中分配的一部分系统资源分配给连接到PDB的每个会话。

注意：

根管理器中的所有活动都由资源管理器自动管理。

12.1.7.1关于CDB计划

在CDB中，PDB可能具有不同的优先级。您可以创建CDB计划，以根据这些优先级将资源分配到CDB中的不同PDB。

CDB计划中的指令集指定如何将资源分配给CDB中的PDB。

CDB计划与其指令之间存在父子关系。

每个指令引用一个PDB。

指令控制以下资源分配给PDB：

  • 中央处理器

  • 并行执行服务器

  • 记忆

指令可以根据您为每个PDB指定的共享值来控制对PDB的资源分配。较高的股票价值会带来更多资源。例如，您可以通过将第一个PDB的共享值设置为第二个PDB的共享值的两倍来指定为一个PDB分配两倍于分配给第二个PDB的资源。

您还可以为PDB指定利用率限制利用率限制将资源分配限制为PDB。例如，它可以控制PDB获得的CPU数量占CDB可用总CPU的百分比。

您可以同时使用共享和利用率限制来精确控制分配给CDB中每个PDB的资源。

注意：

Oracle建议您不要在CDB资源计划中使用parallel_server_limit指令。

12.1.7.2关于PDB计划

CDB计划确定分配给每个PDB的资源量。 PDB计划确定如何将分配给特定PDB的资源分配给该PDB内的消费者组。

PDB计划类似于非CDB的计划。与非CDB的计划在非CDB中的消费者组之间分配资源的方式相同，PDB计划在PDB中的消费者组之间分配资源。

PDB计划必须包含一个指令，该指令将资源分配给名为OTHER_GROUPS的使用者组。 OTHER_GROUPS适用于没有映射到计划中任何其他使用者组的所有会话。

要使用“快速设置”创建新的资源计划：
  1. 转到资源管理：所有计划页面。

    有关详细信息，请参阅“ 查看所有计划 ” 。

  2. 单击快速设置 。

    出现“快速设置”向导，“资源计划”选项卡处于活动状

  3. 在“资源计划”选项卡上，选择选项卡上字段的值：
    领域 描述
    计划名称 输入要用于新计划的名称。
    评论 （可选）输入新计划的注释。
    属性 选择要用于为计划创建使用者组的属性。选择是：

    • 服务

    • 用户

    • 程序

    快速设置最多可以创建7个使用者组，默认情况下，它会对消费者组映射执行1对1服务/用户/程序。您可以选择将多个服务/用户/程序分组到一个使用者组中。

    如果有超过7个服务，用户或程序，则禁用该属性的“快速设置”。例如，如果服务少于7个，则可以选择“ 服务”属性，但如果有超过7个用户和程序，则“ 用户”和“ 程序”属性不可供选择。

    中央处理器 如果要为计划指定CPU指令，请选择此选项。
    并行服务器 如果要为计划指定并行服务器指令，请选择此选项。
    失控查询 如果要为计划指定失控查询指令，请选择此选项。
  4. 单击向右箭头按钮。

    将出现“快速设置”向导的“使用者组映射”选项卡。

  5. 将服务选择到相应的使用者组中。
    默认情况下，“快速安装”在“使用者组映射”选项卡上分配的使用者组名称使用命名方案CG_ service ，CG_ user或CG_ program 。您可以为使用者组指定其他名称。此外，您可以选择将多个服务/用户/程序分组到一个使用者组中。
  6. 单击向右箭头按钮。

    将出现“快速设置”向导的“资源分配”选项卡。在此选项卡上，您可以为计划设置资源指令。

    此选项卡的内容取决于您在快速设置向导的第一个选项卡（资源计划选项卡）上选择的指令（ CPU ， 并行服务器和失控查询 ）。

    如果您选择了CPU指令，则会显示“ 共享”列。在每个使用者组的“ 共享”列中，输入该使用者组的相应份额数。

    如果选择了并行服务器指令，则会出现并行服务器限制（％）列。在每个使用者组的并行服务器限制（％）列中，指定一个值，该值限制使用者组可以使用的并行执行服务器的数量。

    如果选择了失控查询指令，则会显示失控查询度量标准列表和“ 操作”列。默认情况下，在失控查询指标列表中选择经过时间限制指标，但您可以选择列表中的任何指标。对于每个使用者组，请为所选度量标准设置最大值，并在“ 操作”列中，指定在任何单个调用超过为度量标准设置的限制时要采取的操作（“ 取消SQL”或“ 终止会话” ）。

  7. 单击确定 。

    确认消息告诉您计划已成功创建。在确认消息中单击“ 确定”后，新计划将显示在“资源管理：所有计划”页面上的计划列表中。

12.3.1关于实例囚笼

当多个Oracle数据库实例在单个多CPU服务器上运行时，您可以使用实例锁定更有效地使用硬件资源。

设置实例锁定的典型原因是服务器整合 - 更有效地使用可用的硬件资源。在单个服务器上运行多个实例时，实例会争用CPU。一个资源密集型数据库实例可能会显着降低其他实例的性能。例如，在具有四个数据库实例的16-CPU系统上，操作系统可能在该实例的高负载期间在大多数CPU上运行一个数据库实例。这可能会降低其他三个实例的性能。

限制每个数据库实例的CPU消耗的一种简单方法是使用实例锁定。实例囚笼是一种使用初始化参数来限制实例可以同时使用的CPU数量的方法。

在前面的示例中，如果使用实例锁定将四个实例中的每个实例的CPU数限制为四个，则一个实例可能会干扰其他实例的可能性更小。当约束到四个CPU时，实例可能会受CPU限制。这是资源管理器根据您为实例设置的资源计划开始在各种数据库会话中分配CPU的工作。因此，实例锁定和资源管理器一起提供了一种简单有效的方法来管理单个服务器上的多个实例。
