66 DBMS_FGA

DBMS_FGA包提供细粒度的安全功能。

本章包含以下主题：

  • 安全模型

  • 操作说明

  • DBMS_FGA子程序摘要

66.1 DBMS_FGA安全模型

您必须具有DBMS_FGA程序包的AUDIT_ADMIN角色或EXECUTE特权才能创建审核策略。DBMS_FGA是一个调用者权限包。

要分析和审核数据，您必须具有AUDIT_VIEWER角色。由于审计功能可能会捕获所有用户环境和应用程序上下文值，因此策略管理应仅由特权用户执行。策略事件处理程序模块以模块所有者的权限执行。

66.2 DBMS_FGA操作说明

此软件包仅适用于基于成本的优化。基于规则的优化器可能会生成不必要的审计记录，因为审计监视可以在行筛选之前进行。

对于基于规则的优化程序和基于成本的优化程序，您可以查询UNIFIED_AUDIT_TRAIL视图的SQL_TEXT和SQL_BINDS列，以分析SQL文本和发出的相应绑定变量。

66.3.1 ADD_POLICY过程

此过程使用提供的谓词作为审计条件创建审计策略。

句法

DBMS_FGA.ADD_POLICY（     object_schema IN VARCHAR2 DEFAULT NULL，object_name IN VARCHAR2，   policy_name IN VARCHAR2，audit_condition IN VARCHAR2 DEFAULT NULL，audit_column IN VARCHAR2 DEFAULT NULL，handler_schema IN VARCHAR2 DEFAULT NULL，     handler_module IN VARCHAR2 DEFAULT NULL，启用IN BOOLEAN DEFAULT TRUE，statement_types IN VARCHAR2 DEFAULT SELECT，audit_trail IN BINARY_INTEGER DEFAULT NULL，audit_column_opts IN BINARY_INTEGER DEFAULT ANY_COLUMNS ，policy_owner IN VARCHAR2 DEFAULT NULL）;

参数

表66-2 ADD_POLICY过程参数

参数 描述

object_schema

要审计的对象的模式。如果为NULL ，则假定当前架构。

OBJECT_NAME

要审核的对象的名称

POLICY_NAME

策略的唯一名称。请勿输入空格或逗号等特殊字符。如果要对策略名称使用特殊字符，请将名称括在引号中。

audit_condition

连续的条件，指示监视条件。允许NULL并充当TRUE 。

audit_column

要检查访问的列。这些可以包括OLS隐藏列或对象类型列。如果访问或影响任何列，则默认值NULL导致审计。

handler_schema

包含事件处理程序的模式。默认值NULL会导致使用当前架构。

handler_module

事件处理程序的函数名称;如有必要，包括包名称。只有在处理了与查询中的审计条件匹配的第一行后，才会调用此函数。如果过程因异常而失败，则用户SQL语句也将失败。

启用

如果为TRUE ，则启用策略，这是默认值

statement_types

此策略适用的SQL语句类型：仅INSERT ， UPDATE ， DELETE或SELECT

AUDIT_TRAIL

在尚未迁移到统一审计的环境中，细粒度审计记录的目标（DB或XML）。还指定是否填充FGA_LOG$系统表中的LSQLTEXT和LSQLBIND列。

audit_column_opts

在查询引用audit_column参数中指定的任何列时或仅在引用所有此类列时，确定是否审计语句

policy_owner时

拥有细粒度审计政策的用户。但是，此设置不是用户提供的参数。Oracle Data Pump客户端在内部使用此设置来适当地重新创建细粒度的审计策略。

使用说明

  • 表或视图最多可以应用256个细粒度审计策略。

  • 如果未指定object_schema ，则假定为当前架构。

  • 不应将FGA策略应用于诸如LOB列之类的外部列。

  • 每个审核策略都单独应用于查询。但是，无论返回多少行满足该策略的audit_condition ，最多只能为每个策略生成一个审计记录。换句话说，每当返回任意数量的行满足表中定义的审计条件时，将为每个此类策略生成一个审计记录。

  • 如果在其上定义了具有FGA策略的表接收到快速路径插入或向量更新，则在执行任何此类操作之前会自动禁用该提示。禁用提示允许根据策略的条款进行审核。（快速路径插入的一个示例是语句INSERT-WITH-APPEND-hint 。）

  • audit_condition必须是布尔表达式，可以使用插入，更新或删除的行中的值来计算。表达式还可以使用函数，例如USER或SYS_CONTEXT函数。

    表达式不能使用AND和OR等运算符组合条件。 audit_condition可以为NULL （或省略），它被解释为TRUE ，但不能包含以下元素：

    • 子查询或序列

    • 使用SYS_CONTEXT函数访问时， USERENV命名空间的以下属性：

      • CURRENT_SQL

      • CURRENT_SQL_LENGTH

      • CURRENT_BIND

    • 任何使用伪列LEVEL, PRIOR, or ROWNUM 。

    为实现此目的，不再需要指定审计条件“1 = 1”以强制审计影响指定列（“ audit_column ”）的所有指定语句（“ statement_types ”）。即使没有处理任何行， audit_condition的NULL值也会导致审计发生，因此将审计具有此策略的表上的所有操作。

  • 使用创建策略的用户的权限评估audit_condition 。

  • 对于audit_condition设置，不包括功能，执行auditable语句在同一个基地台，在audit_condition设置。例如，假设您创建了一个在HR.EMPLOYEES表上执行INSERT语句的函数。策略audit_condition包含此函数，它适用于INSERT语句（由statement_types参数设置）。使用策略时，函数将递归执行，直到系统内存不足。这可能会引发错误ORA-1000: maximum open cursors exceeded或ORA-00036: maximum number of recursive SQL levels (50) exceeded ORA-1000: maximum open cursors exceeded ORA-00036: maximum number of recursive SQL levels (50) exceeded 。

  • 不要在条件中从策略函数发出DBMS_FGA.ENABLE_POLICY或DBMS_FGA.DISABLE_POLICY语句。

  • 审计功能（ handler_module ）是管理员的警报机制。这种功能所需的接口如下：

    PROCEDURE fname （ object_schema VARCHAR2， object_name VARCHAR2， policy_name VARCHAR2）AS ...

    其中fname是过程的名称， object_schema是被审计表的模式的名称， object_name是要审计的表的名称， policy_name是要强制执行的策略的名称。审计功能将以功能所有者的权限执行。

  • 如果已迁移到统一审计，则省略audit_trail参数，因为审计记录将自动写入统一审计跟踪。

  • 请注意，敏感数据（如信用卡信息）可以以明文形式记录。

  • audit_trail参数（如果使用）指定将编写细粒度审计跟踪的位置以及是否包含查询的SQL Text和SQL Bind变量信息（通常在名为LSQLTEXT和LSQLBIND列中）：

    • 如果audit_trail包含XML，则将细粒度的审计记录写入存储在SQL中的AUDIT_FILE_DEST语句指定的目录中的XML格式的操作系统文件。 （默认的AUDIT_FILE_DEST是基于Unix的系统上的$ORACLE_BASE/admin/$DB_UNIQUE_NAME/adump ，以及Windows系统上的$ ORACLE_BASE\admin\$DB_UNIQUE_NAME\adump 。）

    • 如果audit_trail包含DB ，则审计记录将写入数据库中的SYS.FGA_LOG$表。但是，对于只读数据库，无论audit_trail设置如何，Oracle数据库都会将细粒度的审计记录写入XML文件。

    • 如果audit_trail包含EXTENDED ，则查询的SQL Text和SQL Bind变量信息将包含在审计跟踪中。

    • 例如：

      • 将audit_trail设置为DBMS_FGA.DB会将审计跟踪发送到数据库中的SYS.FGA_LOG$表，并省略SQL Text和SQL Bind。

      • 将audit_trail设置为DBMS_FGA.DB + DBMS_FGA.EXTENDED会将审计跟踪发送到数据库中的SYS.FGA_LOG$表，并包括SQL Text和SQL Bind。

      • 将audit_trail设置为DBMS_FGA.XML会将审计跟踪写入发送到操作系统的XML文件中，并省略SQL Text和SQL Bind。

      • 将audit_trail设置为DBMS_FGA.XML + DBMS_FGA.EXTENDED将审计跟踪写入发送到操作系统的XML文件中，并包括SQL Text和SQL Bind。

    audit_trail参数显示在ALL_AUDIT_POLICIES视图中。

  • 您可以使用以下命令更改操作系统目标：

    ALTER SYSTEM SET AUDIT_FILE_DEST = 新目录 DEFERRED

  • 在许多平台上，XML审计文件名为process_name_processId .xml ，例如ora_2111.xml 。或者，在Windows上，XML审计文件名为process_name_ThreadId .xml （如果进程未作为线程运行，则为process_name_ProcessId .xml ）。

  • audit_column_opts参数确定是否审计语句

    • 当查询引用audit_column参数中指定的任何列时（ audit_column_opts = DBMS_FGA.ANY_COLUMNS ），或者

    • 仅在引用所有此类列时（ audit_column_opts = DBMS_FGA.ALL_COLUMNS ）。

    默认值为DBMS_FGA.ANY_COLUMNS 。

    ALL_AUDIT_POLICIES视图还显示audit_column_opts 。

  • 当audit_column_opts设置为DBMS_FGA.ALL_COLUMNS ，只有在语句中显式引用audit_column提到的所有列时，才会审计SQL语句。并且必须在同一SQL语句或子选择中引用这些列。

    所有这些列必须引用单个表/视图或别名。

    如果SQL语句选择不同表别名中的列，则不会审计该语句。

  • 每个XML审计记录都包含元素AUDIT_TYPE和EXTENDED_TIMESTAMP ，后者以UTC区域打印（没有时区信息）。使用V$XML_AUDIT_TRAIL视图检索的值将转换为会话时区并打印。

  • 对于SQL_TEXT和SQL_BIND元素值（ CLOB类型列），动态视图仅显示前4000个字符。对于此类SQL_TEXT和SQL_BIND值，基础XML文件可能包含超过4000个字符。

  • 对于大量XML审计文件，使用SQL * Loader或类似工具将V$XML_AUDIT_TRAIL加载到数据库表时，查询速度会更快。当AUDIT_TRAIL = OS时，XML审计文件大于写入OS文件的等效文件。

  • 错误处理与AUDIT_TRAIL = OS时相同。如果在将审计记录写入磁盘时发生任何错误，包括AUDIT_FILE_DEST标识的目录已满，则审计操作将失败。记录警报消息。

  • 策略事件处理程序模块将以模块所有者的权限执行。

  • 不要创建递归的细粒度审计处理程序。例如，假设您创建了一个在HR.EMPLOYEES表上执行INSERT语句的处理程序。与此处理程序关联的策略适用于INSERT语句（由statement_types参数设置）。使用策略时，处理程序以递归方式执行，直到系统内存不足。这可能会引发错误ORA-1000: maximum open cursors exceeded或ORA-00036: maximum number of recursive SQL levels (50) exceeded ORA-1000: maximum open cursors exceeded ORA-00036: maximum number of recursive SQL levels (50) exceeded 。另请参阅“ Oracle数据库安全指南” ，了解有关创建细粒度审核策略的信息。

  • 细粒度的审计处理程序模块不应该包含显式的COMMIT ， ROLLBACK和DDL语句。

  • audit_trail参数（ XML和XML+EXTENDED ）的值使细粒度审计记录以XML格式写入操作系统文件。动态视图V$ XML_AUDIT_TRAIL通过SQL查询从DBA可用的XML文件中获取此类审计记录，从而提高了可用性。查询此视图会导致AUDIT_FILE_DEST目录中的所有XML文件（扩展名为.xml所有文件）都以关系表格式进行解析和显示。

    存储在操作系统文件中的审计记录可能比数据库存储的审计记录更安全，因为访问可能需要DBA没有的文件权限。审计记录的操作系统存储还提供更高的可用性，因为即使数据库暂时无法访问，此类记录仍然可用。

    DBA_COMMON_AUDIT_TRAIL视图包含标准和细粒度审计记录的V$ XML_AUDIT_TRAIL动态视图的内容。

    请注意，仅当未启用统一审核时，才会填充V$XML_AUDIT_TRAIL视图。如果已启用统一审核，则可以在UNIFIED_AUDIT_TRAIL数据字典视图中查询审核跟踪记录。

也可以看看：

“Oracle数据库安全指南” ，以获取为细粒度审计策略创建电子邮件警报处理程序的示例

例子

DBMS_FGA.ADD_POLICY（object_schema =>'scott'，object_name =>'emp'，policy_name =>'mypolicy1'，audit_condition =>'sal <100'，audit_column =>'comm，sal'，handler_schema => NULL，handler_module = > NULL，enable => TRUE，statement_types =>'INSERT，UPDATE'，audit_column_opts => DBMS_FGA.ANY_COLUMNS，policy_owner =>'sec_admin）;

66.3.2 DISABLE_POLICY过程

句法

DBMS_FGA.DISABLE_POLICY（     object_schema IN VARCHAR2，object_name IN VARCHAR2，   policy_name IN VARCHAR2）;

参数

表66-3 DISABLE_POLICY过程参数

参数 描述

object_schema

要审计的对象的模式。如果为NULL ，则假定当前架构。

OBJECT_NAME

要审核的对象的名称

POLICY_NAME

策略的唯一名称

DBMS_FGA.DISABLE_POLICY（object_schema =>'scott'，object_name =>'emp'，policy_name =>'mypolicy1'）;

66.3.3 DROP_POLICY过程

此过程将删除审核策略。

句法

DBMS_FGA.DROP_POLICY（object_schema IN VARCHAR2，object_name IN VARCHAR2，   policy_name IN VARCHAR2）;

参数

表66-4 DROP_POLICY过程参数

参数 描述

object_schema

要审计的对象的模式。如果为NULL ，则假定当前架构。

OBJECT_NAME

要审核的对象的名称

POLICY_NAME

策略的唯一名称

使用说明

DBMS_FGA过程会导致当前DML事务（如果有）在操作之前提交，除非它们位于DDL事件触发器内。对于DDL事务， DBMS_FGA过程是DDL事务的一部分。object_schema的默认值为NULL 。如果为NULL ，则假定当前架构。

注意：

如果删除DBMS_FGA.ADD_POLICY过程的object_name参数中指定的对象，或者删除创建审核策略的用户，Oracle数据库将自动删除审核策略。

例子

DBMS_FGA.DROP_POLICY（object_schema =>'scott'，object_name =>'emp'，policy_name =>'mypolicy1'）;
