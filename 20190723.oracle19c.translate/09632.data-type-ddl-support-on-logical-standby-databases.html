C.1.1逻辑备用数据库中支持的数据类型

这些是逻辑备用数据库支持的数据类型。

逻辑备用数据库支持以下数据类型：

  • 抽象数据类型（ADT）和ADT表

    • ADT不能包含任何不支持作为顶级列类型的数据类型（例如，嵌套表，PKREF，BFILE，不支持的不透明类型）。

    • 对于要支持ADT列的表，必须有一个主键（或至少一个唯一约束或唯一索引），它只包含标量顶级列（标量ADT属性不能是这种候选键的一部分）。

  • BINARY_DOUBLE

  • BINARY_FLOAT

  • BLOB ， CLOB和NCLOB存储为BasicFile和SecureFiles。SecureFiles可以进行压缩，加密或重复数据删除。SecureFiles支持要求主数据库以11.2或更高的兼容性运行。请参阅对SecureFiles LOB的支持

  • CHAR

  • 日期

  • 间隔年至月

  • 间隔第二天

  • 长

  • LONG RAW

  • NCHAR

  • 数

  • NVARCHAR2

  • 存储为VARRAY的对象（集合除外）

  • Oracle Text

  • 生的

  • 多媒体（请参阅逻辑备用数据库中不支持的数据类型中列出的例外情况。）：

    • ORDAudio

    • ORDDataSource （内部）

    • 的ORDDicom

    • ORDDoc

    • 的ORDImage

    • ORDSource （内部）

    • ORDVideo

  • 空间（请参阅逻辑备用数据库中不支持的数据类型中列出的例外。）

  • TIMESTAMP

  • TIMESTAMP与TIMEZONE

  • TIMESTAMP与本地时区

  • VARCHAR和VARCHAR2

  • 假定以下主数据库兼容性要求，所有存储模型的XMLType数据：

    • 以CLOB格式存储的XMLType要求主数据库以11.0或更高的兼容性设置运行（从Oracle Database 12 c第1版（12.1）开始，不推荐使用存储为CLOB XMLType 。）

    • 以对象关系格式或二进制XML格式存储的XMLType要求主数据库运行Oracle Database 11g第2版（11.2.0.3）或更高版本，重做兼容性设置为11.2.0.3或更高版本

注意：

SQL Apply不支持具有在ADT， LOB或XMLType列上执行DML的函数调用的语句。

注意：

从Oracle Database 12c第1版（12.1）开始，当COMPATIBLE初始化参数设置为12.0或更高版本且MAX_STRING_SIZE初始化参数设置为EXTENDED时， VARCHAR2 ， NVARCHAR2和RAW数据类型的最大大小已增加到32 KB 。在大多数情况下，逻辑备用数据库支持增加的大小。有关已知限制，请参阅“ 确保可以唯一标识主数据库中的表行 ” 。

C.1.1.1兼容性要求

SQL Apply对这些功能的支持对主数据库有兼容性要求：

  • 多字节CLOB支持要求主数据库以10.1或更高的兼容性运行。

  • 没有LOB和溢出的IOT支持要求主数据库以10.1或更高的兼容性运行。

  • LOB和Overflow的IOT支持要求主数据库以10.2或更高的兼容性运行。

  • TDE支持要求主数据库以11.1或更高的兼容性运行。

  • 基本压缩和高级行压缩要求主数据库以11.1或更高的兼容性运行。

  • 混合列压缩支持依赖于底层存储系统。

C.1.1.2不透明类型限制

C.3支持透明数据加密（TDE）

Oracle Data Guard SQL Apply可用于为启用了透明数据加密（TDE）的主数据库提供数据保护。

使用逻辑备用数据库为具有高级安全要求的应用程序提供数据保护时，请考虑以下事项：

  • 当主数据库和备用数据库都以11.1或更高的兼容级别运行时，使用服务器持有密钥的透明数据加密表将复制到逻辑备用数据库上。

  • Oracle Database 11g第2版（11.2）及更高版本中的逻辑备用数据库支持硬件安全模块上下文中的透明数据加密。

在逻辑备用数据库的上下文中，您希望复制具有加密列的表时，必须考虑以下限制：

  1. 要转换加密的重做记录，SQL Apply必须能够访问包含透明数据加密密钥的打开的钱包。因此，在创建密钥后，必须将包含密钥的钱包从主数据库复制到备用数据库。

  2. 每次更改主密钥时，必须将钱包从主数据库复制到逻辑备用数据库。

  3. Oracle建议您不要在逻辑备用数据库上重新生成主密钥，而逻辑备用数据库正在从主数据库复制加密表。这样做可能会导致SQL Apply在遇到加密的重做记录时停止。

  4. 您可以在逻辑备用数据库上重新生成复制表的加密密钥。这要求在发出重定密钥命令之前将保护设置降低到NONE 。

  5. 复制的加密表可以使用与主数据库中使用的列不同的加密方案。例如，如果HR.EMPLOYEES表的SALARY列使用AES192加密算法在主数据库中加密，则可以使用AES256加密算法在逻辑备用数据库中对其进行加密。或者， SALARY列可以在逻辑备用数据库中保持未加密状态。

C.4支持表空间加密

Oracle Data Guard SQL Apply可用于为启用了表空间加密的主数据库提供数据保护。

在这种情况下， 支持透明数据加密（TDE）中列出的限制1,2和3适用。

主节点上的表空间的加密，重新键控或解密不会触发逻辑备用数据库上相同操作的需要。但是，逻辑备用数据库也必须具有重新键入的能力。

注意：

在某些情况下，当SQL Apply挖掘并应用重做记录以对加密表空间中的表所做的更改时，未加密形式的用户数据记录可能会保留很长一段时间。如果这不可接受，则发出以下命令将与SQL Apply的挖掘组件相关的所有元数据表移动到加密的表空间：

SQL> DBMS_LOGMNR_D.SET_TABLESPACE（NEW_TABLESPACE =>'ENCRYPTED_LOGMNR_TS'）;

C.5支持行级安全性和细粒度审计

从Oracle Database 11g开始 ，逻辑备用数据库可以自动复制通过DBMS_RLS和DBMS_FGA PL / SQL包提供的安全环境。

由于安全环境是透明维护的，因此当服务器故障转移到备用数据库时，此支持简化了安全性考虑因素的管理。它还确保应用于主数据的访问控制策略可以自动转发到备用数据库，并且备用数据可以透明地提供相同级别的保护。如果使用11g新创建备用服务器，则默认启用此复制;否则必须在适当的时候由DBA启用。

支持复制这些PL / SQL程序包要求主数据库和备用数据库都以11.1或更高的兼容性设置运行。

它还要求引用的表是逻辑备用维护对象。例如，具有rowid列的表没有由Logical Standby维护其数据，因此不会维护引用该表的DBMS_RLS和DBMS_FGA调用。

C.5.1行级安全性

行级安全性（也称为虚拟专用数据库（VPD））是一种在访问表，视图或同义词时以精细粒度级别强制实施安全性的功能。

当用户直接或间接访问受VPD策略保护的表，视图或同义词时，服务器会动态修改用户的SQL语句。修改创建由实现安全策略的函数返回的WHERE条件（称为谓词）。使用可以在函数中表达或返回的任何条件，动态地，对用户透明地修改语句。VPD策略可以应用于SELECT ， INSERT ， UPDATE ， INDEX和DELETE语句。通过使用DBMS_RLS包来应用安全策略来实现VPD。

当在主服务器上执行DBMS_RLS过程时，在重做中捕获附加信息，该重做允许在备用数据库上逻辑重建和执行过程调用。Logical Standby支持复制VPD的辅助对象，例如上下文，数据库登录触发器及其支持包。您必须确保将这些对象放在维护的架构中，并且没有配置任何会停止复制的DDL跳过。

C.5.2细粒度审计

细粒度审计提供了一种审计选择语句的方法。

DBMS_FGA程序包允许捕获访问表的所有select语句以及访问的数据。FGA策略可以应用于特定列，甚至只应用于返回指定谓词返回TRUE行的那些select语句。

当在主DBMS_FGA上执行DBMS_FGA过程时， DBMS_FGA附加信息捕获到重做，以允许在备用数据库上逻辑重建和执行过程调用。

C.5.3跳过和启用PL / SQL复制

PL / SQL可以使用skip和skip_error规则配置为与DDL语句完全相同，只是不支持在包和过程上使用通配符。

例如，要跳过VPD的所有方面，请执行以下操作：

DBMS_LOGSTDBY.Skip（stmt =>'PL / SQL'，schema_name =>'SYS'，object_name =>'DBMS_RLS'，use_like => FALSE）;

指定的模式是定义包的模式。要跳过包中的单个过程，语法如下：

DBMS_LOGSTDBY.Skip（stmt =>'PL / SQL'，schema_name =>'SYS'，object_name =>'DBMS_RLS.ADD_POLICY'，use_like => FALSE）;

要在某些模式或表上跳过VPD，必须使用跳过过程。skip过程传递要执行的完全限定的PL / SQL语句，例如：

DBMS_RLS.DROP_POLICY（object_schema =>'SCOTT，object_name =>'EMP'，policy_name =>'MYPOLICY'）;

然后，该过程可以解析语句以决定是跳过它，应用它还是停止应用并让DBA采取补偿措施。

与DDL不同，PL / SQL上的跳过过程不支持返回替换语句。

C.6 Oracle Label Security

从Oracle Database 12 c第2版（12.2）开始，您可以使用Oracle Data Guard数据库滚动升级将临时逻辑备用数据库和PL / SQL升级到使用Oracle Label Security（OLS）的数据库到新的Oracle数据库版本和补丁集包， DBMS_ROLLING 。

C.7 Oracle Database Vault

Oracle Data Guard滚动升级支持使用Oracle Database Vault的数据库。

C.11.1支持的PL / SQL提供的包

Oracle PL / SQL提供的软件包不会修改系统元数据或用户数据，不会在归档的重做日志文件中留下足迹，因此可以安全地在主数据库上使用。

此类包的示例是DBMS_OUTPUT ， DBMS_RANDOM ， DBMS_PIPE ， DBMS_DESCRIBE ， DBMS_TRACE ， DBMS_METADATA ， DBMS_CRYPTO 。

只要修改后的数据属于逻辑备用数据库中支持的数据类型中列出的受支持数据类型，SQL Apply就支持不修改系统元数据但可能修改用户数据的Oracle PL / SQL提供的包。此类包的示例是DBMS_LOB ， DBMS_SQL和DBMS_TRANSACTION 。

Oracle数据卫士逻辑备用支持通过以下软件包执行的操作复制： DBMS_DDL ， DBMS_FGA ， SDO_META ， DBMS_REDACT ， DBMS_REDEFINITION ， DBMS_RLS ， DBMS_SQL_TRANSLATOR ， DBMS_XDS ， DBMS_XMLINDEX和DBMS_XMLSCHEMA 。

要确定逻辑备用数据库中支持哪些包，您可以查询DBA_LOGSTDBY_PLSQL_SUPPORT视图。例如，您可以运行以下查询以找出通用逻辑备用数据库中支持的包：

SQL> SELECT OWNER，PKG_NAME FROM DBA_LOGSTDBY_PLSQL_SUPPORT  - >其中support_level ='ALWAYS';

要确定在使用DBMS_ROLLING包完成的滚动升级的上下文中支持DBMS_ROLLING包，您可以查询DBA_LOGSTDBY_PLSQL_SUPPORT视图，如下所示：

SQL> SELECT OWNER，PKG_NAME FROM DBA_LOGSTDBY_PLSQL_SUPPORT  - >其中support_level ='DBMS_ROLLING';

C.11.2不支持的PL / SQL提供的包

SQL Apply不支持修改系统元数据的Oracle PL / SQL提供的软件包，因此它们的影响在逻辑备用数据库上不可见。

此类包的示例是DBMS_JAVA ， DBMS_REGISTRY ， DBMS_ALERT ， DBMS_SPACE_ADMIN ， DBMS_REFRESH ， and DBMS_AQ 。

此外，物理备用滚动升级不支持DBMS_RESOURCE_MANAGER程序包。

C.11.2.1支持DBMS_JOB

已提供对DBMS_JOB特定支持。在主数据库上创建的作业将在备用数据库上复制，但只要备用数据库保持其备用角色，就不会运行。

在切换或故障转移的情况下，原始主数据库上计划的作业将自动开始在新的主数据库上运行。

您还可以在逻辑备用数据库中创建作业。只要逻辑备用数据库维持备用角色，这些作业就会运行。

C.11.2.2支持DBMS_SCHEDULER

已提供对DBMS_SCHEDULER特定支持，以允许作业在备用数据库上运行。

调度工作的一个新的属性在Oracle数据库11克称为创建database_role其内容符合database_role的属性V$DATABASE 。创建调度程序作业时，它默认为本地角色，因此在备用database_role上创建的作业默认为LOGICAL STANDBY的database_role 。作业计划程序仅执行特定于当前角色的作业。在切换或故障转移时，调度程序会自动切换到特定于新角色的正在运行的作业。

除了使用DBMS_ROLLING PL / SQL包完成滚动升级的上下文之外，调度程序作业不会复制到备用数据库。但是，可以使用DBMS_SCHEDULER.Set_Attribute过程在新角色下激活现有作业。或者，可以克隆应该在两个角色中运行的作业，并使副本特定于另一个角色。DBA_SCHEDULER_JOB_ROLES视图显示哪些作业特定于哪个角色。

调度程序作业在逻辑备用数据库上运行时遵循数据库防护。因此，要运行需要修改未维护表的作业，请将数据库防护设置为STANDBY 。 （无法在PL / SQL块中使用ALTER SESSION DISABLE GUARD语句并使其生效。）

C.11.3在逻辑备用中处理XML和XDB PL / SQL包

Logical Standby支持所有存储模型的XMLType数据，并具有一些兼容性要求。

要求如下：

  • 以CLOB格式存储的XMLType要求主数据库以11.0或更高的兼容性设置运行（从Oracle Database 12 c第1版（12.1）开始，不推荐使用存储为CLOB XMLType 。）

  • 以对象关系格式或二进制XML格式存储的XMLType要求主数据库运行Oracle Database 11g第2版（11.2.0.3）或更高版本，重做兼容性设置为11.2.0.3或更高版本

有几个与XML结合使用的PL / SQL包不完全受支持。

Logical Standby支持的PL / SQL包和过程仅修改内存结构;它们不会修改存储在数据库中的数据。这些程序包不会生成重做，因此不会复制到逻辑备用数据库。

某些与逻辑备用数据库不支持但需要在逻辑备用数据库上进行相应调用以便继续复制活动的XML和XDB相关的PL / SQL包和过程会进行检测，以便在主数据库上调用这些过程会产生额外的表示过程调用的重做记录。当SQL Apply遇到此类重做记录时，它会停止并在DBA_LOGSTDBY_EVENTS表中写入错误消息，指示过程名称。这允许DBA在适当的时间在逻辑备用数据库上调用相应的过程，以便在主数据库中生成的后续重做记录可以在逻辑备用数据库中成功应用。有关处理这些不受支持的过程的详细信息，请参阅DBMS_XMLSCHEMA架构，通过补偿订购敏感的不受支持的PL / SQL 。

以下软件包包含不受支持的过程：

  • DBMS_XMLSCHEMA （如果兼容性设置为12.0.0或更高，则支持。）

  • DBMS_XMLINDEX

除了这些包之外，Logical Standby不支持对XDB架构的任何修改。XDB模式中的对象被视为系统元数据，并且不会复制对它们的直接修改。

Logical Standby不支持由Oracle XML DB存储库管理的表，也称为支持层次结构的表。这些表用于存储XML数据，可以使用FTP和HTTP协议以及普通的SQL访问来访问。有关这些表的更多信息，请参阅Oracle XML DB Developer's Guide 。

在创建逻辑备用数据库之前，必须在主数据库上标识不受支持的数据库对象。

注意：

逻辑备用数据库不支持排序的哈希群集表。

从逻辑备用支持的角度来看，数据库上有三种类型的对象：

  • 由SQL Apply显式维护的对象

  • 由SQL Apply隐式维护的对象

  • 不由SQL Apply维护的对象

Oracle数据库附带的某些模式（例如， SYSTEM ）包含由SQL Apply隐式维护的对象。但是，如果在SYSTEM放置用户定义的表，则即使它具有受支持的数据类型列，也不会进行维护。要发现SQL Apply不维护哪些对象，必须运行两个查询。第一个查询如下：

C.14.2逻辑备用数据库中AUD $和FGA_LOG $的复制

逻辑备用数据库支持审计和细粒度审计。

对主数据库的AUD$和FGA_AUD$表所做的更改将在逻辑备用数据库中进行复制。

AUD$表和FGA_AUD$表都有一个DBID列。如果DBID值是主数据库的值，则根据主数据库上的活动将该行复制到逻辑备用数据库。如果DBID值是逻辑备用数据库的值，则由于逻辑备用数据库中的本地活动而插入该行。

之后，逻辑备用数据库承担的角色转换的结果主要角色（无论是切换或故障），则AUD$和FGA_AUD$在新的主 （最初的逻辑备用），并在新的逻辑备用表（原primary）不一定是同步的。因此，新主逻辑数据库中的AUD$或FGA_AUD$表中的所有行都不可能出现在新的逻辑备用数据库中。但是，在数据库处于主要角色时插入的所有AUD$和FGA_LOG$将被复制并存在于逻辑备用数据库中。

C.15分布式事务和XA支持

您可以使用几种不同的方法执行分布式事务。

  • 使用数据库链接以协调的方式修改多个数据库中的表。

  • 使用XA接口，由提供的PL / SQL包中的DBMS_XA包或通过OCI或JDBC库公开。XA接口实现X / Open分布式事务处理（DTP）架构。

使用这两种方法之一在分布式事务期间对主数据库所做的更改将复制到逻辑备用数据库。

但是，不会复制分布式事务状态。逻辑备用数据库不会继承此类事务的不确定或准备状态，并且不会使用在主数据库中为XA事务使用的相同全局事务标识符来复制更改。因此，如果在提交分布式事务之前故障转移到逻辑备用数据库，则更改将在逻辑备用数据库中回滚。即使主数据库上的分布式事务处于准备状态并且已成功完成两阶段提交协议的第一阶段，也会发生此回滚。切换操作等待所有活动的分布式事务完成，并且不受此限制的影响。

XA事务可以通过两种方式执行：

  • 紧密耦合，不同的XA分支共享锁

  • 松散耦合，不同的XA分支不共享锁

无论COMPATIBLE参数值如何，都支持复制由松散耦合的XA分支所做的更改。只有COMPATIBLE=11.2或更高版本才支持复制由Oracle RAC主服务器（在11g第1版中引入）中的紧密耦合分支所做的更改。

C.16支持SecureFiles LOB

当数据库兼容级别设置为11.2或更高时，支持SecureFiles LOB。

可以在主数据库的SecureFiles LOB列上启用透明数据加密和数据压缩。

完全支持SecureFiles LOB列和SecureFiles数据库文件系统（DBFS）操作的重复数据删除。

如果SQL Apply遇到由不支持的操作生成的重做，则它会ORA-16211: Unsupported record found in the archived redo log错误中ORA-16211: Unsupported record found in the archived redo log时停止。要继续，请使用DBMS_LOGSTDBY.SKIP为受影响的表添加跳过规则，然后重新启动SQL Apply。

C.17支持数据库文件系统（DBFS）

数据库文件系统（DBFS）在存储在数据库表中的文件和目录之上创建标准文件系统接口，这使您可以更轻松地访问和管理存储在数据库中的文件。

逻辑备用数据库支持数据库文件系统（DBFS）。有关DBFS的更多信息，请参见Oracle Database SecureFiles和Large Objects Developer's Guide 。

C.18字符集考虑因素

有关字符集的注意事项。

  • 不支持使用Data Guard配置，其中主数据库和逻辑备用数据库具有不同的字符集。

  • 只有在使用DBMS_ROLLING进行滚动升级时，才支持多租户容器数据库（CDB）具有混合字符集的DBMS_ROLLING 。混合字符集意味着CDB$ROOT和一个或多个CDB的可插拔数据库（PDB）具有不同的字符集。
