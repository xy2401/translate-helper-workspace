GRANT

目的

使用GRANT语句授予：

  • 用户和角色的系统权限。表18-1列出了系统特权（由操作的数据库对象组织）。

  • 对用户，角色和程序单元的角色。授予的角色可以是用户定义的（本地的或外部的）或预定义的。有关预定义角色的列表，请参阅“ Oracle数据库安全指南” 。

  • 特定对象对用户和角色的对象特权。表18-2列出了对象权限（由操作的数据库对象组织）。

注意：

全局角色（使用IDENTIFIED GLOBALLY创建）通过企业角色授予，无法使用GRANT语句授予。

授权数据库用户的注意事项

您可以通过数据库和GRANT语句以外的方式对数据库用户进行授权。

  • 许多Oracle数据库特权都是通过提供的PL / SQL和Java包授予的。有关这些权限的信息，请参阅相应程序包的文档。

  • 某些操作系统具有允许您使用初始化参数OS_ROLES向Oracle数据库用户授予角色的OS_ROLES 。如果您选择通过操作系统工具向用户授予角色，则您也不能使用GRANT语句向用户授予角色，但您可以使用GRANT语句将系统权限授予用户，将系统特权和角色授予其他角色。

有关Oracle自动存储管理的说明

经过身份验证的AS SYSASM用户可以使用此语句将系统特权SYSASM ， SYSOPER和SYSDBA授予当前节点的Oracle ASM密码文件中的用户。

关于可编辑对象的注释

授予可编辑对象的对象权限的GRANT操作可实现当前版本中的对象。有关版本和可编辑对象的更多信息，请参见“ Oracle数据库开发指南” 。

先决条件

要授予系统特权 ，必须满足以下条件之一：

  • 您必须已被授予GRANT ANY PRIVILEGE系统特权。在这种情况下，如果您将系统特权授予角色，则授予该角色的用户不具有该特权，除非在用户的会话中启用该角色。

  • 您必须已使用ADMIN OPTION授予系统特权。在这种情况下，如果将系统特权授予角色，则授予该角色的用户具有该特权，无论该角色是否在用户的会话中启用。

要将角色授予用户或其他角色 ，您必须直接使用ADMIN OPTION授予角色，或者您必须已被授予GRANT ANY ROLE系统特权，或者您必须已创建角色。

要将角色授予您自己架构中的程序单元 ，您必须直接使用ADMIN OPTION或DELEGATE OPTION授予该角色，或者您必须已被授予GRANT ANY ROLE系统特权，或者您必须已创建角色。

要将角色授予其他用户架构中的程序单元 ，您必须是用户SYS并且该角色必须由架构所有者创建或直接授予架构所有者。

要为用户授予对象特权 ，通过指定on_object_clause的ON USER子句，您必须是授予该特权的用户，或者您必须已WITH GRANT OPTION授予该用户的对象特权，或者您必须已被授予GRANT ANY OBJECT PRIVILEGE系统特权。如果只能因为具有GRANT ANY OBJECT PRIVILEGE而授予用户对象权限，则*_TAB_PRIVS视图的GRANTOR列将显示授予该权限的用户，而不是发出GRANT语句的用户。

要为所有其他类型的对象授予对象特权 ，您必须拥有该对象，或者该对象的所有者必须已WITH GRANT OPTION授予您对象特权，或者您必须已被授予GRANT ANY OBJECT PRIVILEGE系统特权。如果您具有GRANT ANY OBJECT PRIVILEGE ，则只有在对象所有者可以授予相同的对象权限时才能授予对象权限。在这种情况下， *_TAB_PRIVS视图的GRANTOR列显示对象所有者，而不是发出GRANT语句的用户。

要指定CONTAINER子句，必须连接到多租户容器数据库（CDB）。要指定CONTAINER = ALL ，当前容器必须是根。

语义

grant_system_privileges

使用这些子句授予系统特权。

system_privilege

指定要授予的系统特权。表18-1列出了由操作的数据库对象组织的系统特权。

  • 如果您向用户授予权限，则数据库会将该权限添加到用户的权限域。用户可以立即行使特权。Oracle建议您仅向受信任的用户授予ANY权限。

  • 如果为角色授予特权，则数据库会将该特权添加到角色的特权域。已被授予并已启用该角色的用户可立即行使该权限。已被授予该角色的其他用户可以启用该角色并行使该权限。

    也可以看看：

    向用户 授予系统特权 ：示例和“ 向角色授予系统特权：示例 ”

  • 如果您向PUBLIC授予权限，则数据库会将权限添加到每个用户的权限域。所有用户都可以立即执行该权限授权的操作。Oracle建议不要向PUBLIC授予系统权限。

角色

指定要授予的角色。您可以授予Oracle数据库预定义角色或用户定义角色。

  • 如果您向用户授予角色，则数据库会将该角色提供给用户。用户可以立即启用角色并在角色的权限域中行使权限。

    对于安全应用程序角色，您无需直接向用户授予此类角色。假设用户通过了适当的安全策略，您可以让关联的PL / SQL包执行此操作。有关更多信息，请参阅USING包和Oracle数据库安全指南的CREATE ROLE语义

  • 如果将角色授予其他角色 ，则数据库会将授予角色的权限域添加到被授予者角色的权限域。已被授予被授予者角色的用户可以启用它并在授予的角色的权限域中行使权限。

  • 如果您向PUBLIC授予角色，则数据库会使该角色对所有用户可用。所有用户都可以立即启用角色并在角色的权限域中行使权限。

所有特权

指定ALL PRIVILEGES以授予表18-1中列出的所有系统特权，但SELECT ANY DICTIONARY ， ALTER DATABASE LINK和ALTER PUBLIC DATABASE LINK特权除外。

grantee_identified_by

grantee_identified_by子句允许您在授予用户系统权限和角色时为其分配密码。您必须指定相同数量的用户和密码。第一个密码分配给第一个用户，第二个密码分配给第二个用户，依此类推。如果存在指定的用户，则数据库将重置用户的密码。如果指定的用户不存在，则数据库将使用密码创建用户。

WITH ADMIN OPTION

指定WITH ADMIN OPTION以使被授权者能够：

  • 将权限或角色授予其他用户或角色，除非该角色是GLOBAL角色

  • 撤消其他用户或角色的权限或角色

  • 更改权限或角色以更改访问它所需的授权

  • 删除权限或角色

  • 将角色授予被授权者模式中的程序单元。

  • 从受让人的架构中的程序单元撤消角色。

如果在未指定WITH ADMIN OPTION情况下向用户授予系统特权或角色，然后随后将该特权或角色授予用户WITH ADMIN OPTION ，则该用户对特权或角色具有ADMIN OPTION 。

要从用户撤消对系统特权或角色的ADMIN OPTION ，您必须完全撤消用户的特权或角色，然后在没有ADMIN OPTION情况下将权限或角色授予用户。

WITH DELEGATE选项

只有在向用户授予角色时才能指定此子句。

指定WITH DELEGATE OPTION以使被授权者能够：

  • 将角色授予被授权者模式中的程序单元

  • 从受让人的架构中的程序单元撤消角色

如果授予一个角色给用户，而无需指定WITH DELEGATE OPTION ，并随后角色授予用户WITH DELEGATE OPTION ，然后用户具有DELEGATE OPTION的角色。

要撤消用户对角色的DELEGATE OPTION ，您必须完全撤消用户的角色，然后在没有DELEGATE OPTION情况下将角色授予用户。

授予系统权限和角色的限制

权限和角色受以下限制：

  • 特权或角色在授予的特权和角色列表中不能出现多次。

  • 您无法为自己授予角色。

  • 您无法将IDENTIFIED GLOBALLY角色授予任何内容。

  • 您无法将IDENTIFIED EXTERNALLY角色授予全局用户或全局角色。

  • 您不能循环授予角色。例如，如果您将角色banker授予角色teller ，那么您无法随后将teller授予banker 。

  • 您不能将IDENTIFIED BY角色， IDENTIFIED USING角色或IDENTIFIED EXTERNALLY角色授予其他角色。

grant_object_privileges

使用这些子句授予对象权限。

object_privilege

指定要授予的对象权限。表18-2列出了对象权限，这些权限由可以授予它们的对象类型组织。将可编辑对象的对象特权授予用户或角色时，该对象将在授予的版本中实现。有关可编辑对象类型和版本的信息，请参阅CREATE EDITION 。

注意：

要将视图上的SELECT授予另一个用户，您必须拥有该视图下的所有对象，或者必须已在所有这些底层对象上授予SELECT对象特权WITH GRANT OPTION 。即使被授权者已经对这些底层对象具有SELECT权限，也是如此。

要将视图上的READ授予另一个用户，您必须拥有该视图下的所有对象，或者必须已在所有这些底层对象上授予READ或SELECT对象特权WITH GRANT OPTION 。即使被授权者已对这些基础对象具有READ或SELECT权限，也是如此。

对象权限的限制

在要授予的权限列表中，权限不能出现多次。

所有[特权]

指定ALL以授予您使用GRANT OPTION授予的对象的所有权限。拥有包含对象的模式的用户自动拥有GRANT OPTION对象的所有权限。提供关键字PRIVILEGES是为了语义清晰，并且是可选的。

柱

指定要授予权限的表或视图列。只有在授予INSERT ， REFERENCES或UPDATE权限时才能指定列。如果未列出列，则被授予者对表或视图中的所有列具有指定的权限。

有关现有列对象授予的信息，请查询USER_ ， ALL_或DBA_COL_PRIVS数据字典视图。

grant_roles_to_programs

使用此子句将角色授予程序单元。这些角色称为基于代码的访问控制（CBAC）角色。

角色

指定要授予的角色。您可以授予Oracle数据库预定义角色或用户定义角色。该角色必须由程序单元的模式所有者创建或直接授予该模式所有者。

program_unit

指定要授予角色的程序单元。您可以指定PL / SQL函数，过程或包。如果未指定schema ，则Oracle数据库假定函数，过程或包位于您自己的模式中。

集装箱条款

如果当前容器是可插拔数据库（PDB）：

  • 指定CONTAINER = CURRENT以在本地向用户或角色授予系统特权，对象特权或角色。仅在当前PDB中将权限或角色授予用户或角色。

CREATE TABLE dependent（dependno NUMBER，dependname VARCHAR2（10），employee NUMBER CONSTRAINT in_emp REFERENCES hr.employees（employee_id））;

约束in_emp确保dependent表中的所有依赖项都对应于模式hr employees表中的employees 。
