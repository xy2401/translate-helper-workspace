32管理分布式数据库

管理分布式数据库包括管理全局名称，管理数据库链接以及创建位置和语句透明度等任务。

  • 管理分布式系统中的全局名称
    在分布式数据库系统中，每个数据库都应具有唯一的全局数据库名称 。全局数据库名称唯一标识系统中的数据库。分布式系统中的主要管理任务是管理全局数据库名称的创建和更改。
  • 创建数据库链接
    要支持对分布式数据库系统中的数据和模式对象的应用程序访问，必须创建所有必需的数据库链接。
  • 使用共享数据库链接
    使用标准数据库链接引用远程服务器的每个应用程序都在本地数据库和远程数据库之间建立连接。许多同时运行应用程序的用户可能会导致本地和远程数据库之间的大量连接。共享数据库链接使您可以限制本地服务器和远程服务器之间所需的网络连接数。
  • 管理数据库链接
    管理数据库链接包括诸如关闭它们，删除它们以及限制它们的活动连接数等任务。
  • 查看有关数据库链接的信息
    每个数据库的数据字典存储数据库中所有数据库链接的定义。您可以使用数据字典表和视图来获取有关链接的信息。
  • 创建位置透明度
    配置必要的数据库链接后，可以使用各种工具从用户隐藏数据库系统的分布式特性。换句话说，用户可以像访问本地对象一样访问远程对象。
  • 管理声明透明度
    在分布式数据库中，某些SQL语句可以引用远程表。

    示例说明管理数据库链接。

32.1.2确定是否强制执行全局命名

您为本地数据库上的链接指定的名称取决于本地数据库是否强制执行全局命名。

如果本地数据库强制执行全局命名，则必须使用远程数据库全局数据库名称作为链接的名称。例如，如果您连接到本地hq服务器并且想要创建指向远程mfg数据库的链接，并且本地数据库强制执行全局命名，则必须使用mfg全局数据库名称作为链接名称。

您还可以将服务名称用作数据库链接名称的一部分。例如，如果使用服务名称sn1和sn2连接到数据库hq.example.com ，并强制执行全局命名，则可hq.example.com hq创建以下链接名称：

  • HQ.EXAMPLE.COM@SN2

32.1.4更改全局数据库名称中的域

使用ALTER DATABASE语句更改数据库全局名称中的域。

创建数据库后，更改初始化参数DB_DOMAIN不会影响全局数据库名称或数据库链接名称的解析。

以下示例显示重命名语句的语法，其中database是数据库名称， domain是网络域：

ALTER DATABASE RENAME GLOBAL_NAME TO database.domain ;

使用以下过程更改全局数据库名称中的域：

  1. 确定当前的全局数据库名称。例如，问题：
    SELECT * FROM GLOBAL_NAME; GLOBAL_NAME ------------------------------------------------- --------------------------- SALES.EXAMPLE.COM
  2. 使用ALTER DATABASE语句重命名全局数据库名称。例如，输入：
    ALTER DATABASE RENAME GLOBAL_NAME转发给sales.us.example.com;
  3. 查询GLOBAL_NAME表以检查新名称。例如，输入：
    SELECT * FROM GLOBAL_NAME; GLOBAL_NAME ------------------------------------------------- --------------------------- SALES.US.EXAMPLE.COM

32.1.5更改全局数据库名称：方案

方案说明了更改全局数据库名称。

在此方案中，您将更改本地数据库的全局数据库名称的域部分。您还可以使用部分指定的全局名称创建数据库链接，以测试Oracle数据库如何解析名称。您发现数据库使用本地数据库的当前全局数据库名称的域部分解析部分名称，而不是初始化参数DB_DOMAIN的值。

  1. 您连接到SALES.US.EXAMPLE.COM并查询GLOBAL_NAME数据字典视图以确定当前数据库全局名称：
    CONNECT SYSTEM@sales.us.example.com SELECT * FROM GLOBAL_NAME; GLOBAL_NAME ------------------------------------------------- --------------------------- SALES.US.EXAMPLE.COM
  2. 您查询V$PARAMETER视图以确定DB_DOMAIN初始化参数的当前设置：
    SELECT NAME，VALUE FROM V $ PARAMETER WHERE NAME ='db_domain';名称值--------- ----------- db_domain US.EXAMPLE.COM
  3. 然后，使用仅部分指定的全局名称创建名为hq的数据库的数据库链接：
    创建数据库链接使用'销售';

    数据库通过将本地数据库的全局数据库名称的域部分附加到链接中指定的数据库的名称来扩展此链接的全局数据库名称。

  4. 您查询USER_DB_LINKS以确定数据库用于解析部分指定的全局数据库名称的域名：
    从USER_DB_LINKS中选择DB_LINK; DB_LINK ------------------ HQ.US.EXAMPLE.COM

    此结果表明本地数据库的全局数据库名称的域部分是us.example.com 。创建数据库链接时，数据库使用此域解析部分数据库链接名称。

  5. 因为您收到sales数据库将移至日本的消息，所以您将sales数据库重命名为sales.jp.example.com ：
    ALTER DATABASE RENAME GLOBAL_NAME TO sales.jp.example.com; SELECT * FROM GLOBAL_NAME; GLOBAL_NAME ------------------------------------------------- --------------------------- SALES.JP.EXAMPLE.COM
  6. 您再次查询V$PARAMETER并发现DB_DOMAIN的值未更改，但您重命名了全局数据库名称的域部分：
    SELECT NAME，VALUE FROM V $ PARAMETER WHERE NAME ='db_domain';名称值--------- ----------- db_domain US.EXAMPLE.COM

    此结果表明DB_DOMAIN初始化参数的值独立于ALTER DATABASE RENAME GLOBAL_NAME语句。ALTER DATABASE语句确定全局数据库名称的域，而不是DB_DOMAIN初始化参数（尽管最好将DB_DOMAIN更改为反映新域名）。

  7. 您创建另一个数据库链接到数据库supply ，然后查询USER_DB_LINKS看到数据库是如何解决的全局数据库名的域部分supply ：
    创建数据库链接供应使用'供应';从USER_DB_LINKS中选择DB_LINK; DB_LINK ------------------ HQ.US.EXAMPLE.COM SUPPLY.JP.EXAMPLE.COM

    此结果表明数据库使用域jp.example.com.解析部分指定的链接名称jp.example.com. 创建链接时使用此域，因为它是本地数据库的全局数据库名称的域部分。该数据库不使用DB_DOMAIN解决部分链接名称时初始化参数设置。

  8. 然后，您会收到以前的信息有问题的消息： sales将在ASIA.JP.EXAMPLE.COM域中，而不是JP.EXAMPLE.COM域。因此，您重命名全局数据库名称，如下所示：
    ALTER DATABASE RENAME GLOBAL_NAME TO sales.asia.jp.example.com; SELECT * FROM GLOBAL_NAME; GLOBAL_NAME ------------------------------------------------- --------------------------- SALES.ASIA.JP.EXAMPLE.COM
  9. 您查询V$PARAMETER以再次检查参数DB_DOMAIN的设置：
    SELECT NAME，VALUE FROM V $ PARAMETER WHERE NAME ='db_domain';名称值---------- ----------- db_domain US.EXAMPLE.COM

    结果表明参数文件中的域设置与发出任一ALTER DATABASE RENAME语句之前的域设置相同。

  10. 最后，您创建一个指向warehouse数据库的链接，然后再次查询USER_DB_LINKS以确定数据库如何解析部分指定的全局名称：

    同样，您会看到数据库使用本地数据库的全局数据库名称的域部分来在链接创建期间扩展部分链接名称。

32.2创建数据库链接

要支持对分布式数据库系统中的数据和模式对象的应用程序访问，必须创建所有必需的数据库链接。

  • 获取创建数据库链接所需的权限
    数据库链接是本地数据库中的一个指针，用于访问远程数据库上的对象。要创建专用数据库链接，您必须已被授予适当的权限。
  • 指定链接类型
    创建数据库链接时，必须确定谁有权访问它。
  • 指定链接用户
    数据库链接定义从一个数据库到另一个数据库的通信路径。当应用程序使用数据库链接访问远程数据库时，Oracle数据库代表本地应用程序请求在远程数据库中建立数据库会话。创建专用或公用数据库链接时，可以通过创建固定用户，当前用户和连接的用户数据库链接来确定链接将建立连接的远程数据库上的哪个模式。
  • 使用连接限定符在链接名称中指定服务名称
    在某些情况下，您可能希望具有指向同一远程数据库的多个相同类型的数据库链接（例如，public），但是使用不同的通信路径建立与远程数据库的连接。

32.2.3指定链接用户

数据库链接定义从一个数据库到另一个数据库的通信路径。当应用程序使用数据库链接访问远程数据库时，Oracle数据库代表本地应用程序请求在远程数据库中建立数据库会话。创建专用或公用数据库链接时，可以通过创建固定用户，当前用户和连接的用户数据库链接来确定链接将建立连接的远程数据库上的哪个模式。

  • 创建固定用户数据库链接
    当应用程序使用固定用户数据库链接时，本地服务器始终建立与远程数据库中固定远程模式的连接。当应用程序使用该链接访问远程数据库时，本地服务器还会通过网络发送固定用户的凭据。
  • 创建连接的用户和当前用户数据库链接
    连接的用户和当前用户数据库链接不包含链接定义中的凭据。用于连接到远程数据库的凭据可能会更改，具体取决于引用数据库链接的用户和应用程序执行的操作。

32.2.4使用连接限定符在链接名称中指定服务名称

在某些情况下，您可能希望具有指向同一远程数据库的多个相同类型的数据库链接（例如，public），但是使用不同的通信路径建立与远程数据库的连接。

这种策略有用的一些情况是：

  • 远程数据库是Oracle Real Application Clusters配置的一部分，因此您可以在本地节点上定义多个公共数据库链接，以便可以建立与远程数据库的特定实例的连接。

  • 有些客户端使用TCP / IP连接到Oracle数据库服务器，而其他客户端使用DECNET。

为了实现此类功能，数据库允许您在数据库链接名称中创建具有可选服务名称的数据库链接。创建数据库链接时，服务名称被指定为数据库链接名称的尾部，由@符号分隔，如@sales 。此字符串称为连接限定符 。

例如，假设远程数据库hq.example.com在Oracle Real Application Clusters环境中管理。hq数据库有两个名为hq_1和hq_2实例。本地数据库可以包含以下公共数据库链接，以定义到hq数据库的远程实例的路径：

创建公共数据库链接hq.example.com@hq_1使用'string_to_hq_1';创建公共数据库链接hq.example.com@hq_2使用'string_to_hq_2';使用'string_to_hq'创建公共数据库链接hq.example.com;

请注意，在前两个示例中，服务名称只是数据库链接名称的一部分。服务名称的文本不一定表示如何建立连接;此信息在USING子句的服务名称中指定。另请注意，在第三个示例中，未将服务名称指定为链接名称的一部分。在这种情况下，就像将服务名称指定为链接名称的一部分一样，实例由USING字符串确定。

要使用服务名称指定特定实例，请在全局对象名称的末尾包含服务名称：

SELECT * FROM scott.emp @ hq.example.com @ hq_1

请注意，在此示例中，有两个@符号。

专用/共享服务器

32.4.1关闭数据库链接

在通过链接访问的每个远程数据库上进程处于活动状态的意义上，链接是打开的。这种情况会产生以下后果：

  • 如果20个用户打开会话并访问本地数据库中的同一公共链接，则会打开20个数据库链接连接。

  • 如果20个用户打开会话并且每个用户访问专用链接，则打开20个数据库链接连接。

  • 如果一个用户启动会话并访问20个不同的链接，则打开20个数据库链接连接。

关闭会话后，会话中活动的链接将自动关闭。您可能有机会手动关闭链接。例如，关闭链接时：

  • 由链接建立的网络连接很少在应用程序中使用。

  • 必须终止用户会话。

要关闭链接，请发出以下语句，其中linkname指的是链接的名称：

ALTER SESSION CLOSE DATABASE LINK linkname ;

请注意，此语句仅关闭当前会话中处于活动状态的链接。

32.6.2.1创建同义词

所有同义词都是模式对象，存储在创建它们的数据库的数据字典中。为了简化通过数据库链接的远程表访问，同义词允许单字访问远程数据，隐藏特定对象名称和同义词用户的位置。

您可以为以下内容创建同义词：

  • 表

  • 类型

  • 查看

  • 物化观点

  • 序列

  • 程序

  • 功能

  • 包

创建同义词的语法是：

CREATE [PUBLIC] SYNONYM synonym_name FOR [ schema 。] object_name [@ database_link_name ];

哪里：

  • PUBLIC是一个关键字，指定此同义词可供所有用户使用。省略此参数会使同义词变为私有，并且只能由创建者使用。公共同义词只能由具有CREATE PUBLIC SYNONYM系统特权的用户CREATE PUBLIC SYNONYM 。

  • synonym_name指定用户和应用程序引用的备用对象名称。

  • schema指定object_name指定的对象的模式。省略此参数使用创建者的模式作为对象的模式。

  • object_name指定表，视图，序列，物化视图，类型，过程，函数或包。

  • database_link_name指定标识远程数据库和架构的数据库链接， object_name指定的对象位于该架构中。

同义词必须是其架构的唯一命名对象。如果架构包含架构对象并且存在具有相同名称的公共同义词，则当拥有架构的用户引用该名称时，数据库始终会找到架构对象。

示例：创建公共同义词

假设在分布式数据库系统中的每个数据库中，为存储在hq数据库中的scott.emp表定义了公共同义词：

创建公共SYNONYM emp for scott.emp@hq.example.com;

您可以设计员工管理应用程序，而不必考虑应用程序的使用位置，因为表格scott.emp@hq.example.com的位置被公共同义词隐藏。应用程序中的SQL语句通过引用公共同义词emp访问该表。

此外，如果将emp表从hq数据库移动到hr数据库，则只需更改系统节点上的公共同义词。员工管理应用程序继续在所有节点上正常运行。

32.6.2.2管理权限和同义词

同义词是对实际对象的引用。有权访问特定模式对象的同义词的用户还必须具有底层模式对象本身的权限。

例如，如果用户尝试访问同义词但对其标识的表没有权限，则会发生错误，指示表或视图不存在。

假设scott创建本地同义词emp作为远程对象scott.emp@sales.example.com. scott的别名scott.emp@sales.example.com. scott 无法将同义词的对象权限授予另一个本地用户。 scott无法为同义词授予本地权限，因为此操作相当于授予sales数据库上远程emp表的权限，这是不允许的。此行为与作为本地表或视图的别名的同义词的权限管理不同。

因此，当同义词用于位置透明度时，您无法管理本地权限。基础对象的安全性完全由远程节点控制。例如，用户admin无法为emp_syn同义词授予对象权限。

与视图或过程定义中引用的数据库链接不同，同义词中引用的数据库链接通过首先查找在解析对同义词的引用时生效的模式所拥有的专用链接来解析。因此，为了确保所需的对象分辨率，在同义词的定义中指定底层对象的模式尤为重要。

32.6.3.1使用本地过程引用远程数据

过程或函数（独立或在包中）可以包含引用远程数据的SQL语句。

当用户或应用程序调用fire_emp过程时，很明显远程表正在被修改。

CREATE PROCEDURE fire_emp（enum NUMBER）AS BEGIN DELETE FROM emp WHERE empno = enum;结束;

32.7管理语句透明度

在分布式数据库中，某些SQL语句可以引用远程表。

该数据库允许以下标准DML语句引用远程表：

  • SELECT （查询）

  • 插入

  • UPDATE

  • 删除

  • 选择...FOR UPDATE （异构系统中并不总是支持）

  • 锁表

查询包括联接，聚合，子查询和SELECT...FOR UPDATE可以引用任意数量的本地和远程表和视图。例如，以下查询连接来自两个远程表的信息：

SELECT e.empno，e.ename，d.dname FROM scott.emp@sales.division3.example.com e，jward.dept@hq.example.com d WHERE e.deptno = d.deptno;

在同构环境中， UPDATE ， INSERT ， DELETE和LOCK TABLE语句可以引用本地和远程表。无需编程即可更新远程数据。例如，下面的语句插入新行插入到远程表emp在scott.sales从选择行模式emp表中的jward在本地数据库架构：

INSERT INTO scott.emp@sales.division3.example.com SELECT * FROM jward.emp;

对报表透明度的限制：

声明透明度有几个限制。

  • 更新远程非Oracle数据库系统上的对象的数据操作语言语句不能引用本地Oracle数据库上的任何对象。例如，如下所示的语句将导致引发错误：

    INSERT INTO remote_table @ link as SELECT * FROM local_table;

  • 在单个SQL语句中，所有引用的LONG和LONG RAW列，序列，更新表和锁定表必须位于同一节点。

  • 除了通过远程执行DBMS_SQL包的过程之外，数据库不允许在同类系统中使用远程DDL语句（例如， CREATE ， ALTER和DROP ），如下例所示：

    DBMS_SQL.PARSE@link_name（crs，'drop table emp'，v7）;

    请注意，在异构系统中，传递工具允许您执行DDL。

  • ANALYZE语句的LIST CHAINED ROWS子句不能引用远程表。

  • 在分布式数据库系统中，无论语句（或语句的一部分）执行的位置如何，数据库始终会针对本地服务器评估与环境相关的SQL函数，例如SYSDATE ， USER ， UID和USERENV 。

    注意：

    Oracle数据库仅支持用于查询的USERENV函数。

  • 一些性能限制与远程对象的访问有关：

    • 远程视图没有统计数据。

    • 可能无法优化分区表上的查询。

    • 远程表不会考虑超过20个索引。

    • 复合索引使用的列不超过20列。

  • Oracle数据库实现中对分布式读取一致性的限制可导致一个节点相对于另一个节点过去。根据读取一致性，查询可能最终检索一致但过时的数据。请参阅“ 管理读取一致性 ”以了解如何管理此问题。

SELECT * FROM emp@sales.division3.example.com;

32.8.5示例5：创建公共当前用户数据库链接

示例说明了如何创建公共当前用户数据库链接。

以下示例以连接用户身份连接到本地数据库，并创建指向sales数据库的公共链接（使用其网络服务名称sldb ）。以下语句创建公共当前用户数据库链接：

CONNECT bart @ local CREATE PUBLIC DATABASE LINK sales.division3.example.com使用'sldb'连接到CURRENT_USER;

注意：

要使用此链接，当前用户必须是全局用户。

此数据库链接的后果如下：

假设scott创建本地过程fire_emp ，它从远程emp表中删除一行，并将fire_emp执行权限授予ford 。 

CONNECT scott @ local_db CREATE PROCEDURE fire_emp（enum NUMBER）AS BEGIN DELETE FROM emp@sales.division3.example.com WHERE empno = enum;结束;对fire_emp执行GRANT EXECUTE TOd;

现在，假设ford连接到本地数据库并运行scott的过程：

CONNECT ford @ local_db EXECUTE PROCEDURE scott.fire_emp（enum 10345）;

当ford执行过程scott.fire_emp ，该过程在scott的权限下运行。因为使用了当前的用户数据库链接，所以连接建立到scott的远程模式，而不是ford的远程模式。请注意， scott必须是全局用户，而ford不必是全局用户。

注意：

如果使用连接的用户数据库链接，则连接将是ford的远程模式。有关调用者权限和特权的更多信息，请参阅Oracle Database PL / SQL语言参考 。

您可以使用到scott的远程模式的固定用户数据库链接来完成相同的结果。
