关于受信任的存储程序单元

Oracle数据库存储过程，函数和包是以编译形式存储在数据库中的PL / SQL语句集。

函数和过程之间的唯一区别是函数返回值而过程不返回。可信存储的程序单元与Oracle数据库中的任何其他存储程序单元一样：基础逻辑是相同的。

包是一组过程和函数，以及它们使用的游标和变量，作为一个单元存储。包有两个部分，包规范和包体。包规范声明包所包含的公共过程，函数和变量的外部定义。包体包含过程和函数的实际文本，以及任何私有过程和变量。

受信任的存储程序单元是已被授予一个或多个Oracle Label Security权限的存储过程，函数或程序包。可信存储的程序单元通常用于让用户以受控方式执行特权操作，或者更新多个标签上的数据。这是允许用户访问超出授权范围的数据的最佳方法。

受信任的存储程序单元提供对特权使用的细粒度控制。虽然您可以为许多用户授予权限，但是应该非常谨慎地授予权限，因为它可能违反为您的应用程序建立的安全策略。您可以识别需要特权的任何应用程序操作，并将其实现为受信任的程序单元，而不是为用户分配权限。为这些存储的程序单元授予权限时，可以有效地限制用户所需的Oracle Label Security权限。该方法采用最小特权原则。

例如，如果具有标签CONFIDENTIAL的用户需要将数据插入SENSITIVE行，则可以将WRITEUP权限授予用户有权访问的受信任存储程序。以这种方式，用户可以通过可信存储的程序执行任务，同时保持在CONFIDENTIAL级别。

可信程序单元代表用户执行所有动作。因此，您可以有效地将安全策略封装到可以进行验证以确保其有效的模块中。

受信任的存储程序单元如何运行

受信任的存储程序单元使用自己的权限和调用者的标签运行。

以这种方式，可信存储程序单元可以对由用户标签约束的行集执行特许操作。

Oracle数据库系统和对象特权旨在捆绑到角色中。然后根据需要授予用户角色。相比之下，Oracle Label Security权限只能分配给用户或存储的程序单元。这些受信任的存储程序单元提供了比角色更易于管理的机制，以控制Oracle Label Security权限的使用。

示例：受信任的存储程序单元

具有READ权限的受信任存储程序单元可以读取所有未受保护的数据以及受此策略保护的所有数据。

例如，考虑负责创建采购预测报告的用户。用户必须对所有购买量执行求和操作。无论用户自己的标签是否授权访问各个采购订单。在此示例中创建求和过程的语法如下：

创建功能sum_purchases RETURN NUMBER是psum NUMBER; BEGIN SELECT SUM（金额）INTO psum FROM purchase_orders;返回psum;结束sum_purchases;

以这种方式，程序单元可以收集最终用户不能收集的信息，并且可以通过求和使其可用。

请注意，要运行SUM_PURCHASES ，需要在此过程中为用户授予标准Oracle Database EXECUTE对象特权。

创建和编译受信任的存储程序单元

您可以创建和编译可信存储的程序单元，以便在Oracle Label Security中使用。

  • 创建受信任的存储程序单元
    您可以使用与创建标准过程，函数或包相同的方式创建受信任的存储程序单元。
  • 可信存储程序单元的权限
    Oracle Label Security管理员可以在授予特权之前验证存储的程序单元代码的正确性。
  • 重新编译受信任的存储程序单元
    自动或手动（使用ALTER PROCEDURE ）重新编译受信任的存储程序单元不会影响其Oracle Label Security权限。
  • 重新创建受信任的存储程序单元
    如果在受信任的存储程序单元上执行CREATE或REPLACE操作，则会撤消Oracle Label Security特权。
  • 执行受信任的存储程序单元
    在Oracle Label Security下，程序调用上的所有标准Oracle数据库控件（关于对表和模式的访问）仍然有效。

可信存储程序单元的权限

Oracle Label Security管理员可以在授予特权之前验证存储的程序单元代码的正确性。

通常，另一个用户（例如开发者）创建存储的程序单元。每当重新创建或替换受信任的存储程序单元时，Oracle Label安全性都会删除其权限。然后，Oracle Label Security管理员必须再次验证代码并再次授予权限。

Oracle Label Security管理员应仔细检查程序单元代码，并评估要授予它的权限。例如，受信任包中的过程不应执行特权数据库操作，然后将结果或状态信息写入包的公共变量中。通过这种方式，您可以确保不会发生违反站点的Oracle Label Security策略的情况。

重新创建受信任的存储程序单元

如果在受信任的存储程序单元上执行CREATE或REPLACE操作，则会撤消Oracle Label Security特权。

这限制了滥用程序的Oracle Label Security权限的可能性。

请注意，即使已删除Oracle Label Security权限，过程，函数或程序包仍可以运行。

如果您重新创建过程，函数或包，则应仔细查看其文本。如果您确定重新创建的程序单元未违反站点的Oracle Label Security策略，则可以再次授予其所需的权限。

在必须经常更换受信任的存储程序单元的开发环境中（例如，在实时系统的前几个月），建议根据需要创建可以授予适当的Oracle Label Security权限的脚本。

执行受信任的存储程序单元

在Oracle Label Security下，程序调用上的所有标准Oracle数据库控件（关于对表和模式的访问）仍然有效。

Oracle Label Security通过控制对行的访问来补充这些安全机制。

执行受信任的存储程序单元时，有效的策略权限是调用用户权限和程序单元权限的并集。无论受信任的存储程序单元是否调用另一个受信任的程序单元或不受信任的程序单元，被调用的程序单元都以与原始程序单元相同的权限运行。

如果执行一系列不可信和可信存储的程序单元，则第一可信程序单元将从该点确定整个调用序列的特权。请考虑以下顺序：

  • 程序A（不信任）
  • 程序B使用WRITEUP
  • 程序C使用WRITEDOWN
  • 程序D（不信任）

这里，过程B，C和D都以WRITEUP权限运行，因为B是序列中的第一个可信过程。当序列结束时，与过程B相关的特权对后续过程不再有效。

注意：

Oracle Label Security捕获受信任程序单元中引发的未处理异常。这意味着可能无法向用户显示错误消息。因此，在授予权限之前，应始终彻底测试和调试任何程序单元。

设置和返回标签信息的工作原理

SA_UTL包具有使用数字标签值返回有关会话安全属性的当前值的信息的功能。

虽然这些功能可以在不受信任的程序单元中使用，但它们主要用于受信任的存储程序单元。

请注意，这些是公共功能;您不需要policy_ DBA角色来使用它们。此外，每个函数都有一个并行的SA_SESSION函数，该函数以字符串格式返回相同的标签。
