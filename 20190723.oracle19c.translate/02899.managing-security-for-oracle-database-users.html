关于普通用户

Oracle提供两种类型的普通用户：CDB普通用户和应用程序普通用户。

CDB普通用户是一个数据库用户，其单一标识和密码在CDB根目录以及每个现有和未来的可插拔数据库（PDB）中都是已知的，包括任何应用程序根。所有Oracle提供的管理用户帐户（例如SYS和SYSTEM ）都是CDB公共用户，可以在系统容器中导航。CDB普通用户可以在不同的PDB中拥有不同的权限。例如，用户SYSTEM可以在PDB之间切换，并使用当前PDB中授予SYSTEM的权限。但是，如果其中一个PDB启用了Oracle Database Vault，则当此用户连接到该PDB时，数据库保管库限制（例如SYSTEM不允许创建用户帐户）将应用于SYSTEM 。 Oracle建议您不要更改Oracle提供的CDB普通用户的权限。

CDB通用用户可以执行应用程序通用用户可以执行的所有任务，前提是已为该用户授予了适当的权限。

应用程序通用用户是在应用程序根目录中创建的用户帐户，并且仅在此应用程序容器中是通用的。换句话说，应用程序公共用户无法访问整个CDB环境，如CDB普通用户。应用程序通用用户负责创建（包括插入），打开，关闭，拔出和删除应用程序PDB等活动。此用户可以在应用程序根目录中创建应用程序公共对象只有在连接到应用程序根目录时，才能创建应用程序通用用户。用户访问应用程序公共对象的能力与本地和CDB公共对象具有相同的权限。例如，PDB中与应用程序根关联的本地用户只能访问该用户具有权限的PDB中的对象。在应用程序根目录中，您通常可以为将在整个应用程序容器中应用的CDB公共对象授予特权。

这两种类型的普通用户都负责管理各自根目录中的公共对象。如果CDB公共用户或应用程序公共用户具有适当的权限，则该用户也可以在PDB中执行操作，例如向本地用户授予权限。这些用户还可以在本地向每个容器授予普通用户不同的权限。

CDB和应用程序普通用户都可以执行以下活动：

  • 授予普通用户或公共角色的权限。也就是说，CDB普通用户可以向公共用户或角色授予特权，并且该特权适用的范围由发布该语句的容器（CDB根，应用程序根或PDB）以及是否通常授予权限（在CDB根目录或应用程序根目录中）。连接到应用程序根目录的CDB公共用户通常可以授予CDB公共对象的权限，并且该权限将应用于整个应用程序容器。

如何插入PDB会影响CDB普通用户

将非CDB作为PDB插入CDB会影响Oracle提供的管理和用户创建的帐户和权限。

这会影响这些CDB通用用户帐户的密码，以及新插入的数据库中所有帐户的权限。

执行以下操作：

  • Oracle提供的管理帐户与现有的公共用户帐户合并。

  • 用户创建的帐户与现有的用户创建的公共用户帐户合并。

  • 现有CDB通用用户帐户的密码优先于非CDB帐户的密码。

  • 如果您已在其原始非CDB中修改了用户帐户的权限，则会保存这些权限，但它们仅适用于将PDB插入CDB时创建的PDB，作为本地授予的权限。例如，假设您已在非CDB db1为用户SYSTEM授予了一个名为hr_mgr的角色。将db1数据库添加到CDB后， SYSTEM只能在db1 PDB中使用hr_mgr角色，而不能在任何其他PDB中使用。

将PDB（例如， pdb_1 ）从一个CDB（ cdb_1 ）插入另一个CDB（ cdb_2 ）时，可能出现以下两种情况：

  • cdb_1有普通用户c##cdb1_user 。 cdb_2没有这个用户。

    c##cdb1_user仍在PDB_1但此帐户已被锁定。要恢复此帐户，您必须关闭pdb_1 ，在c##cdb1_user的根目录中重新创建常用用户c##cdb1_user cdb_2 ，然后重新打开pdb_1 。

  • cdb_1和cdb_2都有共同的用户c##common_user 。

    两个c##common_user帐户都已合并。 c##common_user在cdb_2保留其密码。在cdb_2分配给它但在cdb_1中没有分配给它的任何权限都保留在pdb_1 。

关于本地用户

在多租户环境中，本地用户是仅存在于单个PDB中的数据库用户。

本地用户可以具有管理权限，但这些权限仅适用于创建本地用户帐户的PDB。本地用户帐户具有以下特征，可将其与常见用户帐户区分开来：

  • 本地用户帐户无法创建公共用户帐户或通常授予他们权限。具有适当权限的普通用户可以创建和修改公共或本地用户帐户，并通常或本地授予和撤消权限。本地用户可以创建和修改本地用户帐户，也可以向给定PDB中的公共或本地用户授予本地权限。

  • 您可以授予本地用户帐户常用角色。但是，与公共角色关联的权限仅适用于本地用户的PDB。

  • 本地用户帐户必须仅在其PDB中是唯一的。

  • 使用适当的权限，本地用户可以访问公共用户架构中的对象。例如，如果公共用户已授予本地用户访问权限的权限，则本地用户可以访问公共用户的模式中的表。

  • 您可以通过版本启用本地用户帐户，但不能启用普通用户帐户。

如果要创建自己有权创建用户的用户，请在GRANT语句中包含WITH ADMIN OPTION子句。例如：

GRANT创建会话以使用ADMIN OPTION进行lbrown;

与授予您权限的所有用户帐户一样，仅将这些权限授予可信用户。

在多租户环境中，您必须具有通常授予的CREATE USER系统特权才能创建公共用户帐户。要创建本地用户帐户，您必须在将在其中创建本地用户帐户的PDB中具有通常授予的CREATE USER权限或本地授予的CREATE USER权限。

关于为用户分配默认表空间

每个用户都应该有一个默认的表空间。

在用户架构中创建架构对象且DDL语句未指定包含该对象的表空间时，Oracle数据库会将该对象存储在用户的默认表空间中。

通过表空间，您可以将用户数据与系统数据分开，例如存储在SYSTEM表空间中的数据。您可以使用CREATE USER或ALTER USER语句为ALTER USER分配默认表空间。所有用户的默认表空间的默认设置是SYSTEM表空间。如果用户没有创建对象，并且没有权限这样做，那么此默认设置就可以了。但是，如果用户可能创建任何类型的对象，则应特别为用户分配默认表空间，例如USERS表空间。使用SYSTEM以外的表空间可以减少数据字典对象与相同数据文件的用户对象之间的争用。通常，不要将用户数据存储在SYSTEM表空间中。

您可以使用CREATE TABLESPACE SQL语句在创建数据库时创建SYSTEM之外的永久缺省表空间，以用作永久对象的数据库缺省值。通过将用户数据与系统数据分离，可以降低SYSTEM表空间出现问题的可能性，这在某些情况下可能导致整个数据库无法正常运行。系统用户不使用此默认永久表空间，即SYS ， SYSTEM和OUTLN ，其默认永久表空间为SYSTEM 。无法删除指定为默认永久表空间的表空间。要实现此目标，必须首先将另一个表空间指定为默认永久表空间。您可以使用ALTER TABLESPACE SQL语句将默认永久表空间更改为另一个表空间。请注意，这将影响执行ALTER DDL语句后创建的所有用户或对象。

您还可以在用户创建期间设置用户默认表空间，稍后使用ALTER USER语句进行ALTER USER 。更改用户默认表空间仅影响更改设置后创建的对象。

为用户指定默认表空间时，还要在该表空间上指定配额。

关于为用户分配表空间配额

除临时表空间外，您可以为每个用户分配任何表空间的表空间配额。

分配配额可完成以下任务：

  • 具有创建特定类型对象权限的用户可以在指定的表空间中创建这些对象。

  • Oracle数据库将可分配用于存储指定表空间中用户对象的空间量限制为配额量。

默认情况下，用户对数据库中的任何表空间都没有配额。如果用户具有创建架构对象的权限，则必须分配配额以允许用户创建对象。至少，为用户分配默认表空间的配额，以及为其创建对象的其他表空间的其他配额。您可以为表空间分配的最大空间量为2 TB。如果需要更多空间，请为QUOTA子句指定UNLIMITED 。

您可以为用户分配每个表空间中特定磁盘空间量的单个配额，或者为所有表空间分配无限量的磁盘空间。特定配额可防止用户的对象在数据库中使用太多空间。

您可以在创建用户时为用户表空间分配配额，也可以稍后添加或更改配额。（您可以通过查询USER_TS_QUOTAS视图找到现有用户配额。）如果新配额小于旧配额，则以下条件仍然有效：

  • 如果用户已超过新的表空间配额，则在这些对象的组合空间小于新配额之前，不能为表空间中的用户分配更多空间。

  • 如果用户未超过新的表空间配额，或者表空间中用户的对象使用的空间属于新的表空间配额，则可以为用户的对象分配空间直至新配额。

关于为用户分配临时表空间

您应该为每个用户分配一个临时表空间。

当用户执行需要临时段的SQL语句时，Oracle数据库会将该段存储在用户的临时表空间中。执行排序或连接操作时，系统会创建这些临时段。临时段由SYS拥有，它在所有表空间中都具有资源特权。

要创建临时表空间，可以使用CREATE TEMPORARY TABLESPACE SQL语句。

如果没有为用户显式分配临时表空间，则Oracle Database会为用户分配在创建数据库时指定的默认临时表空间，或者稍后通过ALTER DATABASE语句。如果没有显式分配默认临时表空间，则默认为SYSTEM表空间或系统管理员建立的其他永久默认值。分配要专门用作临时表空间的表空间可消除临时段和其他类型段之间的文件争用。

注意：

如果您的SYSTEM表空间是本地管理的，则必须为用户分配特定的默认（本地管理）临时表空间。可能不允许它们默认使用SYSTEM表空间，因为临时对象不能放在本地管理的永久表空间中。

您可以在创建用户时为用户设置临时表空间，稍后使用ALTER USER语句进行ALTER USER 。您还可以建立表空间组，而不是分配单个临时表空间。

用户的配置文件

配置文件是由属性定义的一组限制，用于数据库资源和对数据库的密码访问。

配置文件可以应用于多个用户，使他们能够共享这些属性。

您可以在创建用户时指定配置文件。CREATE USER语句的PROFILE子句为CREATE USER分配配置文件。如果未指定配置文件，则Oracle数据库会为用户分配默认配置文件。

例如：

CREATE USER jward IDENTIFIED BY password DEFAULT TABLESPACE data_ts QUOTA 100M ON test_ts QUOTA 500K ON data_ts TEMPORARY TABLESPACE temp_ts PROFILE职员 CONTAINER = CURRENT;

在多租户环境中，可以将不同的配置文件分配给根和PDB中的公共用户。当普通用户登录到PDB时，其设置适用于会话的配置文件取决于设置是密码相关还是资源相关。

  • 密码相关的配置文件设置是从分配给根目录中的公共用户的配置文件中提取的。例如，假设您将公共配置文件c##prof （其中FAILED_LOGIN_ATTEMPTS设置为1 ）分配给根目录中的公共用户c##admin 。在PDB中，为用户分配了本地配置文件local_prof （其中FAILED_LOGIN_ATTEMPTS设置为6 ）普通用户c##admin只有在尝试登录分配了loc_prof的PDB loc_prof允许一次失败的登录尝试。

  • 在PDB中分配给用户的配置文件中指定的与资源相关的配置文件设置可以在不咨询分配给根中的公共用户的配置文件中的资源相关设置的情况下使用。例如，如果在local_prof中分配给用户c##admin的配置文件local_prof将SESSIONS_PER_USER设置为2 ，那么当c##admin登录到PDB loc_prof时，只允许2个并发会话分配给他，无论在根目录中分配给他的配置文件中此设置的值如何。

用户会话级别的限制

当用户连接到数据库时，会创建会话。会话使用CPU时间和内存，您可以在其上设置限制。

您可以在会话级别设置多个资源限制。如果用户超出会话级资源限制，则Oracle数据库终止（回滚）当前语句并返回指示已达到会话限制的消息。此时，当前事务中的所有先前语句都是完整的，并且用户可以执行的唯一操作是COMMIT ， ROLLBACK或disconnect（在这种情况下，当前事务已提交）。所有其他操作都会产生错误。即使在提交或回滚事务之后，用户也无法在当前会话期间完成任何更多工作。

数据库调用级别的限制

每次用户运行SQL语句时，Oracle数据库都会执行几个步骤来处理该语句。

在SQL语句处理期间，作为不同执行阶段的一部分，对数据库进行了多次调用。为防止任何一次调用过度使用系统，Oracle数据库允许您在调用级别设置多个资源限制。

如果用户超出了调用级别资源限制，则Oracle数据库会暂停对该语句的处理，回滚该语句并返回错误。但是，当前事务的所有先前语句保持不变，并且用户会话保持连接。

CPU时间限制

当对Oracle数据库进行SQL语句和其他调用时，处理调用需要CPU时间。

平均呼叫需要少量的CPU时间。但是，涉及大量数据或失控查询的SQL语句可能会占用大量CPU时间，从而减少了可用于其他处理的CPU时间。

为防止不受控制地使用CPU时间，可以为每次调用的CPU时间和会话期间用于Oracle数据库调用的CPU时间总量设置固定或动态限制。在呼叫或会话使用的CPU百分之一秒（0.01秒）内设置和测量限制。

逻辑读取的限制

输入/输出（I / O）是数据库系统中最昂贵的操作之一。

I / O密集型的SQL语句可以独占内存和磁盘使用，并导致其他数据库操作竞争这些资源。

其他资源的限制

您可以控制用户并发会话和空闲时间的限制。

其他资源的限制如下：

  • 您可以限制每个用户的并发会话数。 每个用户只能创建最多预定义数量的并发会话。

  • 您可以限制会话的空闲时间。 如果会话中的调用之间的时间达到空闲时间限制，则回滚当前事务，终止会话，并将会话的资源返回给系统。下一个调用将收到一个错误，指示用户不再连接到该实例。此限制设置为经过的分钟数。

    注意：

    会话终止后不久，因为它已超过空闲时间限制，进程监视器（PMON）后台进程将在终止会话后清除。在PMON完成此过程之前，终止的会话仍计入任何会话或用户资源限制。

  • 您可以限制每个会话的已用连接时间。 如果会话的持续时间超过了已用时间限制，则回滚当前事务，删除会话，并将会话资源返回给系统。此限制设置为经过的分钟数。

    注意：

    Oracle数据库不会持续监视已用空闲时间或连接时间。这样做会降低系统性能。相反，它每隔几分钟检查一次。因此，在Oracle数据库强制执行限制并终止会话之前，会话可以略微超出此限制（例如，5分钟）。

  • 您可以限制会话的专用系统全局区域（SGA）空间（用于私有SQL区域）的数量。 此限制仅在使用共享服务器配置的系统中很重要。否则，私有SQL区域位于程序全局区域（PGA）中。此限制设置为实例的SGA中的内存字节数。使用字符K或M指定千字节或兆字节。

删除用户帐户时，Oracle数据库会从数据字典中删除用户帐户和关联的架构。它还会立即删除用户模式中包含的所有模式对象（如果有）。

注意：

  • 如果必须保留用户架构和关联对象但必须拒绝用户访问数据库，则撤消用户的CREATE SESSION特权。

  • 不要尝试删除SYS或SYSTEM用户。这样做会破坏您的数据库。
