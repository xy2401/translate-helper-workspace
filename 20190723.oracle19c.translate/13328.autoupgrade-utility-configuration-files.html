AutoUpgrade实用程序配置文件

AutoUpgrade配置文件包含执行Oracle数据库升级所需的所有信息。

AutoUpgrade配置文件包含全局和本地配置参数。默认情况下，全局参数适用于配置文件寻址的所有数据库。为特定数据库指定时，本地配置参数会覆盖配置文件指定的全局参数。

  • AutoUpgrade用户配置文件的全局参数
    要为配置文件中解决的所有Oracle数据库升级指定参数的默认行为，可以使用可选的AutoUpgrade全局参数。
  • AutoUpgrade配置文件的本地参数
    要为AutoUpgrade实用程序升级配置特定Oracle数据库的信息，请在AutoUpgrade本地参数中提供信息。
  • AutoUpgrade配置文件的可本地修改的全局参数
    可本地修改的全局参数是您在全局设置的参数，并且根据需要在本地设置，以便您可以更好地控制AutoUpgrade作业处理。

AutoUpgrade用户配置文件的全局参数

要为配置文件中解决的所有Oracle数据库升级指定参数的默认行为，可以使用可选的AutoUpgrade全局参数。

使用说明

所有全局参数都是可选的。所有全局参数都采用global前缀。

表3-1 Oracle Database AutoUpgrade Utility的全局配置参数

参数 描述

add_after_upgrade_pfile

指定要在升级PFILE后添加其参数的PFILE的路径和文件名。此规范适用于用户配置文件中的所有数据库。

例：

global.add_after_upgrade_pfile = / path / to / my / add_after.ora
add_during_upgrade_pfile

指定要在PFILE升级期间添加其参数的PFILE的路径和文件名。此规范适用于用户配置文件中的所有数据库。 

global.add_during_upgrade_pfile = / path / to / my / add_during.ora

after_action

指定要在所有升级作业成功完成后运行的自定义用户脚本的路径和文件名。您使用的脚本必须采用name . ext的形式name . ext （例如， myscript.sh ，以便AutoUpgrade可以识别您要运行的脚本类型。允许的扩展选项：

  • Unix shell（ .sh ）

  • Microsoft Windows批处理（ .bat ， .cmd ）

  • Microsoft Windows PowerShell（ .ps1 ）

默认情况下，如果脚本失败，则AutoUpgrade继续运行。如果操作系统检测到脚本失败，请使用Y标志指定AutoUpgrade停止。如果脚本的状态不是0 ，则认为完成失败。

例子：

如果脚本失败，则停止AutoUpgrade：

global.after_action = /path/to/my/script .sh Y.

如果脚本失败，则继续AutoUpgrade：

global.after_action = /path/to/my/script .sh

autoupg_log_dir

设置AutoUpgrade使用的日志文件和属于全局模块的临时文件的位置。如果不使用此参数设置路径，则日志文件将放置在运行AutoUpgrade的当前位置。

例：

global.autoupg_log_dir = /path/to/my/global/log/dir

before_action

指定在启动升级作业之前要为所有升级运行的自定义用户脚本。您使用的脚本必须采用name . ext的形式name . ext （例如， myscript.sh ），以便AutoUpgrade可以识别您要运行的脚本类型。如果要在特定升级作业之前运行脚本，请使用local参数（ local.before_action ）指定该脚本

允许的扩展选项：

  • Unix shell（ .sh ）

  • Microsoft Windows批处理（ .bat ， .cmd ）

  • Microsoft Windows PowerShell（ .ps1 ）

默认情况下，如果脚本失败，则AutoUpgrade继续运行。如果操作系统检测到脚本失败，请使用Y标志指定AutoUpgrade停止。如果脚本的状态不是0 ，则认为完成失败。

例子：

如果脚本失败，则停止AutoUpgrade：

global.before_action = /path/to/my/script .sh Y.

如果脚本失败，则继续AutoUpgrade：

global.before_action = /path/to/my/script .sh
del_after_upgrade_pfile

指定要在PFILE升级期间删除其参数的PFILE的路径和文件名。此规范适用于用户配置文件中的所有数据库。

例：

global.del_after_upgrade_pfile = / path / to / my / del_after.ora
del_during_upgrade_pfile

指定要在PFILE升级期间删除其参数的PFILE的路径和文件名。此规范适用于用户配置文件中的所有数据库。

例：

global.del_during_upgrade_pfile = / path / to / my / del_during.ora

remove_underscore_parameters

在升级期间和升级之后，从配置文件中的所有Oracle数据库中删除PFILE文件中的下划线（隐藏）参数。Underscore参数只能由Oracle Support的建议使用。

例：

global.remove_underscore_parameters = YES

恢复

（可选）生成用于数据库还原的保证还原点（GRP）。如果选择此选项，则必须由DBA手动执行数据库备份和数据库还原。

选项：

[是的|没有]

默认值为yes 。

例：

global.restoration =无

target_home

为配置文件中指定的所有数据库设置全局目标主目录。使用此选项可避免多次指定相同的target_home 。可以在本地覆盖此参数。

例：

global.target_home = /目标/ ORACLE /家

upgradexml

生成upgrade.xml文件。选项： [yes | no]

upgrade.xml等同于早期版本中指定XML参数时preupgrade包生成的文件。此文件在分析模式（ mode -analyze ）期间创建。它是在为AutoUpgrade日志文件定义的preupgrade目录中生成的。

例：

global.upgradexml = YES

AutoUpgrade配置文件的本地参数

要为AutoUpgrade实用程序升级配置特定Oracle数据库的信息，请在AutoUpgrade本地参数中提供信息。

使用说明

本地参数优先于AutoUpgrade配置文件中设置的任何全局参数。除非用（可选）表示，否则所有本地参数都是必需的。所有本地参数都带有前缀（在示例中，由您定义的值标识以标识特定数据库或升级。前缀标识参数在配置文件中应用的特定升级作业。

示例：配置文件中第一次升级的参数集使用前缀sales ，配置文件中下一次升级的参数集使用前缀employees ：

sales.dbname = hr sales.source_home = / u01 / app / oracle / 11.2.0.4 / dbhome1。。。employees.dbname = sales.cdb employees.source_home- / 03 / app / oracle / 18.0.0.0 / dbhome1

表3-2 Oracle Database AutoUpgrade实用程序的本地配置参数。

参数 描述

add_after_upgrade_pfile

（可选）指定要在升级后添加其参数的PFILE的路径和文件名。

例：

sales3.add_after_upgrade_pfile = /路径/要/我的/ pfile_add.ora

add_during_upgrade_pfile

（可选）指定要在升级期间添加其参数的PFILE的路径和文件名。

例：

sales3.add_during_upgrade_pfile = /路径/要/我的/ newpfile.ora

after_action

（可选）指定在完成由前缀地址标识的数据库的升级作业后要执行的自定义操作。

您使用的脚本必须采用name . ext的形式name . ext （例如， myscript.sh ，以便AutoUpgrade可以识别您要运行的脚本类型。允许的扩展选项：

  • Unix shell（ .sh ）

  • Microsoft Windows批处理（ .bat ， .cmd ）

  • Microsoft Windows PowerShell（ .ps1 ）

  • Oracle SQL文件（ .sql ），只有前缀指定的本地操作。

默认情况下，如果脚本失败，则AutoUpgrade继续运行。如果操作系统检测到脚本失败，请使用Y标志指定AutoUpgrade停止。如果脚本的状态不是0 ，则认为完成失败。

与全局after_action参数相反，本地after_action参数可以指定SQL脚本，然后使用目标Oracle数据库二进制文件在数据库上运行该脚本。该脚本在CDB$ROOT上的内部部署数据库（非CDB）或多租户体系结构数据库（CDB）上运行。如果要运行其他特定于容器的操作，则必须在代码中设置它们。对于更复杂的方案，您可以在shell中运行特定于容器的操作。

例子：

在AutoUpgrade开始处理之前运行指定的脚本，如果脚本失败，则将Y标志设置为停止AutoUpgrade：

sales2.after_action = / user / path / script.sh Y.

在AutoUpgrade开始处理之前运行指定的脚本，AutoUpgrade设置为在脚本失败时继续运行：

sales3.after_action = /用户/路径/ script.sh

before_action

（可选）指定在为前缀寻址的特定数据库作业启动升级作业之前要执行的自定义操作。如果要在所有升级作业之前运行脚本，请使用local参数（ global.before_action ）指定该脚本

您使用的脚本必须采用name . ext的形式name . ext （例如， myscript.sh ），以便AutoUpgrade可以识别您要运行的脚本类型。允许的扩展选项：

  • Unix shell（ .sh ）

  • Microsoft Windows批处理（ .bat ， .cmd ）

  • Microsoft Windows PowerShell（ .ps1 ）

  • Oracle SQL文件（ .sql) ，只有前缀指定的本地操作。

默认情况下，如果脚本失败，则AutoUpgrade继续运行。如果操作系统检测到脚本失败，请使用Y标志指定AutoUpgrade停止。如果脚本的状态不是0 ，则认为完成失败。

与全局before_action参数相反，本地before_action参数可以使用早期版本的Oracle Database二进制文件指定SQL脚本，该脚本可以在源数据库Oracle主目录中的数据库上运行。该脚本在CDB$ROOT上的本地数据库（非CDB）或多租户架构数据库（CDB）上运行。如果要运行其他特定于容器的操作，则必须在代码中设置它们。对于更复杂的方案，您可以在shell中运行特定于容器的操作。

例子：

在AutoUpgrade开始处理之前运行指定的脚本，如果脚本失败，则将Y标志设置为停止AutoUpgrade：

sales.before_action = / user / path / script.sh Y.

在AutoUpgrade开始处理之前运行指定的脚本，AutoUpgrade设置为在脚本失败时继续运行：

sales4.before_action = /用户/路径/ script.sh

清单

（可选）指定清单的路径，您可以使用该清单覆盖AutoUpgrade执行的默认修订列表，例如由于策略或安全性问题而不希望自动实施的修正。

要在其他AutoUpgrade模式下使用此参数，必须在analyze模式下运行AutoUpgrade。AutoUpgrade完成分析后，您可以在预检查目录（ dbname _checklist.cfg ）下找到由数据库名称标识的清单文件。手动更新文件以排除您希望AutoUpgrade绕过的修正，并保存文件。再次运行AutoUpgrade时，可以指定指向您创建的清单文件的参数，并修改为各个数据库执行的修正。如果未指定清单文件路径，则在升级期间运行的修订集是在您指定的部署模式期间创建的清单文件的最新版本。

例：

sales.checklist = / U01 /应用/ ORACLE /升级的作业/ salesdb_checklist.cfg

在前面的示例中， salesdb_checklist.cfg是数据库salesdb的清单配置文件。

DBNAME

标识要升级的数据库的名称。

例：

sales.dbname = salesdb

del_after_upgrade_pfile

（可选）指定要在升级后删除其参数的PFILE的路径和文件名。

例：

sales3.del_after_upgrade_pfile = /路径/要/我的/ pfile_del.ora

del_during_upgrade_pfile

（可选）指定要在升级期间删除其参数的PFILE的路径和文件名。

例：

sales3.del_during_upgrade_pfile = /路径/要/我的/ oldpfile.ora

ENV

指定自定义操作系统环境变量，不包括ORACLE_SID，ORACLE_HOME，ORACLE_BASE和TNS_ADMIN。

使用案例：

使用此参数提供数据库sqlnet.ora文件中指示的环境设置，例如用于Oracle Wallet的安全套接字层密码套件。

例：

sales2.env = VARIABLE1：/ VALUE1 /，变量2：/ VALUE2 /

LOG_DIR

设置为数据库升级生成的日志文件的位置，这些数据库升级位于由参数前缀标识的升级作业中包含的数据库集中。AutoUpgrade基于该日志文件路径创建分层目录。例如，作业标识符前缀为sales ，其中log_dir标识为upgrade-jobs ， stage1 ， stage2和stagen表示升级的各个阶段：

/ u01 / app / oracle / upgrade-jobs / temp / / sales / / sales / stage1 / sales / stage2 / sales / stagen

您不能将通配符用于路径，例如波浪号（ ~ ）。您必须使用完整的路径。

例：

salesdb.log_dir = / U01 /应用/ ORACLE /升级的作业

PDBS

（可选）设置要在其上运行升级的PDB列表。此参数仅适用于多租户架构（CDB）数据库。如果运行AutoUpgrade的数据库是本地Oracle数据库，则忽略该参数。

PDB列表以逗号分隔。该列表可以包含PDB名称或星号（ * ），表示您要升级在运行AutoUpgrade时在CDB上打开的所有PDB。如果安装了PDB，则AutoUpgrade会忽略该PDB以进行分析部署。但是，如果PDB处于装载状态，并且部署模式是修正，部署或升级，则PDB以读写模式或升级模式或两者打开，以便可以运行这些阶段。

例：

sales1.pdbs = pdb1，pdb2，pdb n upgr1 .pdbs = *

恢复

（可选）生成用于数据库还原的保证还原点（GRP）。如果选择此选项，则必须由DBA手动执行数据库备份和数据库还原。

选项：

[是的|没有]

默认值为yes 。

例：

sales1.restoration =无

run_utlrp

（可选）启用或禁用在升级过程中运行utlrp 。

utlrp实用程序重新编译在数据库升级期间变为无效的所有数据字典对象。Oracle建议您在每次Oracle数据库升级后运行此实用程序。选项： yes ， no 。默认值已启用（ yes ）。

例：

prefix.run_utlrp = YES

SID

标识要升级的数据库的Oracle系统标识符（SID）。

例：

sales1.sid = salesdb

source_home

要升级的数据库的当前Oracle主目录（ORACLE_HOME）。此参数特别适用于模式analyze ， fixups和deploy 。对于模式upgrade ，源home和目标home值可以是相同的路径。

例：

sales2.source_home = /path/to/my/source/oracle/home

source_tns_admin_dir

（可选）指定源数据库主目录中TNS_ADMIN目录的路径。

例：

sales1.source_tns_admin_dir = / U01 /应用/ ORACLE / 12.2 / dbhome01 /网络/管理

开始时间

（可选）设置要运行的升级作业的未来开始时间。使用此参数可以调度升级作业以平衡服务器上的负载，并防止多个作业立即启动。

值必须now采用表格（立即开始），或采用00/00/0000 00:00:00表格。如果未设置值，则now为默认值。

例：

sales1.start_time = now sales2.start_time = 01/11/2011 01:30:15

允许值：

现在30/12/2018 15:30:00 01/11/2019 01:30:15 2/5/2019 3:30:50

如果启动了多个作业并将start_time值设置为now ，则AutoUpgrade将根据系统中可用的资源计划启动时间，这可能会导致相隔几分钟的作业的启动时间。

对于日期或时间元素使用错误的分隔符或使用错误的日期或小时格式的值无效。

例：

30-12-2018 15:30:00 01/11/2019 3:30:15 pm 2019/06/01 01:30:15

target_tns_admin_dir

（可选）指定目标数据库主目录中TNS_ADMIN目录的路径。

例：

sales1.target_tns_admin_dir = / U01 /应用程序/ ORACLE / 19.0.0.0 / dbhome01 /网络/管理

timezone_upg

（可选）启用或禁用作为AutoUpgrade过程的一部分运行时区升级。为了保持数据完整性，Oracle建议您在数据库升级时升级时区设置。特别是，当您拥有依赖于时区的数据时，例如timestamp with time zone表列的timestamp with time zone ，请升级时timestamp with time zone 。请注意，可以通过覆盖核对表文件上的fixup来禁用此设置。选项： yes ， no 。默认值已启用（ yes ）。

例：

sales1.timezone_upg = YES

upgrade_node

指定当前用户配置有效的节点。

此参数的目的是防止AutoUpgrade处理与AutoUpgrade配合使用的配置文件中列出的数据库，其中upgrade_node参数的值与当前主机名不对应。它无法远程运行AutoUpgrade。您可以使用关键字localhost作为通配符来指示应该处理本地主机上的数据库。

使用案例：

配置文件config.cfg包含10个数据库。其中五个数据库的upgrade_node值设置为denver01 。其余五个将upgrade_node的值设置为denver02 。如果使用配置文件config.cfg在服务器denver01上运行denver01 ，则AutoUpgrade仅处理upgrade_node设置为denver01的数据库。它忽略了upgrade_node设置为denver02的数据库。

例：

sales1.upgrade_node = denver01

AutoUpgrade配置文件的可本地修改的全局参数

可本地修改的全局参数是您在全局设置的参数，并且根据需要在本地设置，以便您可以更好地控制AutoUpgrade作业处理。

使用说明

可局部修改的全局参数是必需参数。您必须在AutoUpgrade配置文件中全局或本地定义这些参数。使用可本地修改的全局参数，您可以使用前缀global将值设置为AutoUpgrade配置文件中所有作业的全局参数，但使用同一配置文件中特定作业的本地作业前缀重置相同的参数。您还可以选择仅将本地可修改的全局参数设置为每个AutoUpgrade作业的本地参数。

当使用全局前缀和本地作业前缀设置本地可修改的全局参数时，本地修改的参数值将覆盖由参数使用的前缀标识的作业的全局参数值。您使用的语法格式为global.target_home = Global target Oracle home ， database .target_home= local target Oracle home .

例：

在AutoUpgrade配置文件中，所需参数target_home全局设置为一个Oracle主路径。但是在配置文件中，相同的参数在本地设置为不同的Oracle主路径。当AutoUpgrade处理配置文件中的作业时，它使用本地定义的target_home路径作为前缀upgrade3定义的作业，覆盖全局参数设置：

global.target_home = / u01 / app / oracle / 19.0.0.0 / dbhome01 upgrade3.target_home = / u03 / app / oracle3 / 12.2.0.1 / dbhome3

表3-3 AutoUpgrade配置文件的可本地修改的全局参数

参数 描述

target_version

（可选）指定您希望AutoUpgrade执行升级的目标发行版本。AutoUpgrade使用您在此参数中提供的发行版本信息来确保将正确的检查和修正用于要升级的目标Oracle数据库发行版。此参数的格式是有效Oracle版本的句点分隔值。

有效值

  • 12.2
  • 18
  • 19
  • 20

仅当目标主页不在系统上或目标主目录是12.2版本时，才需要此选项。否则，AutoUpgrade可以派生目标版本值。

例：

global.target_version = 18 employees.target_version = 12.2
target_home

指定目标Oracle主目录（ORACLE_HOME）路径。

例：

global.target_home = / u01 / app / oracle / 19.0.0 / dbhome01 sales4.target_home = / u04 / app / oracle4 / 19.0.0 / dbhome04
target_base

指定目标Oracle主目录的目标ORACLE_BASE路径。

例：

global.target_base = / u01 / app / oracle sales4.target_base = / u04 / app / oracle4
