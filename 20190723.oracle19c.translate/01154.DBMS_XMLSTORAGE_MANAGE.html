204.1 DBMS_XMLSTORAGE_MANAGE概述

DBMS_XMLSTORAGE_MANAGE包含在架构注册完成后管理和修改XML存储的过程。

使用此程序包中的子程序可以提高批量加载操作的性能。您可以在执行批量加载过程之前禁用索引和约束，然后再启用它们。

204.2 DBMS_XMLSTORAGE_MANAGE安全模型

由XDB拥有，DBMS_XMLSTORAGE_MANAGE包必须由SYS或XDB创建。 EXECUTE特权被授予PUBLIC 。

此包中的子程序使用当前用户的权限执行。

204.3.1 DISABLEINDEXESANDCONSTRAINTS程序

此过程禁用XMLType表和XMLType列的索引和约束。

句法

DBMS_XMLSTORAGE_MANAGE.DISABLEINDEXESANDCONSTRAINTS（owner_name IN VARCHAR2 DEFAULT USER，table_name IN VARCHAR2，column_name IN VARCHAR2 DEFAULT NULL，clear IN BOOLEAN DEFAULT FALSE）;

参数

表204-2 DISABLEINDEXESANDCONSTRAINTS过程参数

参数 描述

OWNER_NAME

所有者的名字

TABLE_NAME

正在执行过程的XMLType表的名称

列名

XMLType列名

明确

布尔值，当设置为TRUE时，在执行过程之前清除表的所有存储的索引和约束数据。默认值为FALSE ，不会清除它们。

使用说明

传递XMLTYPE表

对于XMLType表，必须传递要在其上执行批量加载操作的XMLType表名。对于XMLType列，您必须传递关系表名称和相应的XMLType列名称。

使用clear启用和禁用索引和约束

注意：

如果使用clear设置为TRUE调用DISABLEINDEXESANDCONTRAINTS过程，它将删除有关在过程的早期执行期间存储的XMLTYPE表或列的任何索引或约束信息。

因此，在调用DISABLEINDEXESANDCONTRAINTS过程并将clear设置为TRUE之前，必须确保在表或列上重新启用所有禁用的索引和约束。

理想情况下，建议您在第一次执行时将clear set设置为TRUE 。对于任何后续执行（由于禁用或启用索引时的错误）， clear应设置为FALSE ，即默认值。在批量加载操作之后成功重新启用所有索引和约束后，可以再次调用此过程，并将clear设置为TRUE以进行下一次批量加载操作。

例

以下示例说明了在DISABLEINDEXESANDCONSTRAINTS过程和ENABLEINDEXESANDCONSTRAINTS过程中使用clear 。

首先，在PURCHASEORDER_TAB表的comment元素上添加一个非NULL约束：

ALTER TABLE PURCHASEORDER_TAB ADD CONSTRAINT c1 check（“XMLDATA”。“comment”IS NOT NULL）;

然后，通过调用DISABLEINDEXESANDCONSTRAINTS过程将clear传递为TRUE禁用所有索引和约束：

BEGIN XDB.DBMS_XMLSTORAGE_MANAGE.DISABLEINDEXESANDCONSTRAINTS（USER，'PURCHASEORDER_TAB'，NULL，TRUE）;结束; /

接下来，执行违反ALTER表语句中的约束c1的批量加载操作（例如数据泵导入）。这不会引发错误，因为禁用了约束：

host impdp orexample / orexample directory = dir dumpfile = dmp.txt tables = OREXAMPLE.PURCHASEORDER_TAB content = DATA_ONLY;

注意：要查看禁用的约束和索引，请使用：

SELECT constraint_name，table_name，status FROM all_constraints WHERE owner = user;

最后，尝试使用ENABLEINDEXESANDCONSTRAINTS过程启用约束。它引发了一个错误，因为批量加载操作违反了c1 （非空约束）：

BEGIN XDB.DBMS_XMLSTORAGE_MANAGE.ENABLEINDEXESANDCONSTRAINTS（USER，'PURCHASEORDER_TAB'）;结束; /

要禁用所有索引和约束，请再次使用DISABLEINDEXESANDCONSTRAINTS ，但设置clear= FALSE （因为ENABLEINDEXESANDCONSTRAINTS未能成功完成）。注意：默认情况下， clear = FALSE ，因此我们不需要在下次调用中明确传递它。 

BEGIN xdb。DBMS_XMLSTORAGE_MANAGE.DISABLEINDEXESANDCONSTRAINTS（USER，'PURCHASEORDER_TAB'）;结束; /

然后，删除输入表中的错误行

DELETE FROM purchaseorder_tab p WHERE p.xmldata。“comment”IS NULL;

使用ENABLEINDEXESANDCONSTRAINTS重新启用索引和约束，并成功完成。 

BEGIN xdb。DBMS_XMLSTORAGE_MANAGE.ENABLEINDEXESANDCONSTRAINTS（USER，'PURCHASEORDER_TAB'）;结束; /

204.3.5 INDEXXMLREFERENCES过程

此过程在给定XML类型表的REF列或给定表的XML类型列上创建唯一索引。

如果该过程创建多个REF列，则会将_1 ， _2附加到其名称上。

句法

DBMS_XMLSTORAGE_MANAGE.INDEXXMLREFERENCES（owner_name IN VARCHAR2 DEFAULT USER，table_name IN VARCHAR2，column_name IN VARCHAR2 DEFAULT NULL）;

参数

表204-6 INDEXXMLREFERENCES过程参数

参数 描述

OWNER_NAME

所有者的名字

TABLE_NAME

正在编制索引的表

列名

列名称。XML类型表不需要。

INDEX_NAME

新创建的索引的名称

使用说明

仅当REF的作用域时才使用此过程。请参见SCOPEXMLREFERENCES过程 。

当在查询计划中出现基表和子表之间的连接时，索引的REF会带来更好的性能。

  • 如果基表具有比子表更高的选择性，则无需索引REF 。

  • 如果子表的选择性高于基表的选择性，并且如果不存在索引，则子表中的一行与基表的连接将导致基表的完整表扫描。

INDEXXMLREFERENCES不会在调用它的表的子表中递归索引REF 。为此，Oracle建议在XML_OUT_OF_LINE_TABLES或XML_NESTED_TABLES视图的循环内调用该过程。这将从视图中列的当前值创建索引名称。

注意：

此过程仅限于结构化存储模型。

204.3.9 SCOPEXMLREFERENCES过程

此过程用于范围所有XML引用。与无范围的REF类型相比，Scoped REF类型需要更少的存储空间并允许更高效的访问。

句法

DBMS_XMLSTORAGE_MANAGE.SCOPEXMLREFERENCES;

使用说明

  • 如果您在DBMS_XMLSTORAGE_MANAGE包中使用了SETOUTOFLINE Procedure以避免在XML模式注册期间引发'1000 column limit'错误，则还应使用SCOPEXMLREFERENCES过程 。

  • 在XML模式注册之后和加载XML实例数据之前使用SCOPEXMLREFERENCES ，使这些引用仅作为外部表的作用域。

注意：

此过程仅限于结构化存储模型。

204.3.10 XPATH2TABCOLMAPPING函数

此函数将路径表达式（以XPath表示法或DOT表示法）映射到相应的表名和列名。在用户想要在此表上创建索引或添加约束或重命名表以使查询执行计划更具可读性的情况下，这是必需的。

句法
