考虑建议的数据盒式磁带开发过程，包括过程步骤之间的关系和依赖关系。

2.1数据盒开发过程

要了解Data Cartridge开发过程，请将项目视为一个整体。

理解目的

开发数据盒式磁带的第一步是通过清楚地定义盒式磁带的新功能来建立您打算提供的特定于域的值。指定盒式磁带向用户公开的对象。

了解用户

如果盒式磁带的预期用户是软件开发人员，则盒式磁带的可扩展性至关重要。如果他们是最终用户，则磁带必须高度适应其预期的域。墨盒的设计应反映出对所有用户有清晰了解的商业模式。无论盒式磁带的大小如何，开发团队都必须彻底了解对象关系数据库管理系统并将其应用于盒式磁带域的问题。

规划项目

使用定义明确的软件开发流程，明确识别期望和可交付成果，并为Data Cartridge开发设定合理的里程碑。为项目安排适当的时间并获得可用资源技能的真实情况，使项目更有可能取得成功。

测试和安装

最后的步骤是测试应用程序并创建必要的安装脚本。

2.1.1实施项目

  • 在选择和设计对象时，请确保开发人员和最终用户熟悉并清楚地理解其名称和语义。

  • 在定义对象集合时，请考虑对象方法的SQL端与应用程序开发中使用的编程语言之间的接口。通过限制调用库例程的方法数量，避免对低级库入口点的大量调用，以及编写使用预取数据的大块代码，保持此接口尽可能简单。

  • 定义接口后，沿并行路径前进， 如图2-1所示 。您可以按适合可用资源的任何顺序继续路径。

    这些并行路径中最左边的路径打包现有的3GL代码，这些代码在运行时库（如DLL）中执行相关操作，可能在旧代码之上有新的入口点。库例程由对象方法代码的SQL组件调用。在可能的情况下，应使用3GL测试程序以独立方式测试此代码。

    中间路径定义并写入对象的类型规范和对象方法代码的PL / SQL组件。有些方法可以完全用PL / SQL编写，而有些方法可以调用外部库。如果您的应用程序需要外部库，请提供库定义和库入口例程的详细绑定。

    选择点的方向取决于操作数据时必须部署的访问方法的复杂性。如果您需要的查询方法相对简单，则可以构建常规索引。如果数据很复杂，则必须定义复杂的索引类型以利用Oracle的可扩展索引技术。如果您的项目使用多域查询，则应该使用Oracle的可扩展优化器技术。

    如果您的情况不涉及在多个域上执行查询，并且I / O是影响性能的唯一重要因素，则标准优化技术可能就足够了。但是，如果还有其他因素（如CPU成本影响性能），您仍可以使用可扩展优化器。

2.2安装和使用数据磁带

安装数据盒式磁带是组装其组件的过程，以便服务器可以找到它们并理解用户定义的类型定义。要正确放置这些组件，您必须：

  1. 在服务器中定义表和用户定义的类型。这通常通过运行SQL脚本来完成。
  2. 将动态链接库放在链接规范所期望的位置。
  3. 将联机文档，帮助文件和错误消息文件复制到托管位置。
  4. 通过运行SQL脚本向服务器注册用户定义的类型，这些脚本加载为盒式磁带定义的每个新类型。必须从特权帐户执行此步骤。
  5. 为磁带盒的用户授予必要的访问权限。

2.3数据盒组件的要求和指南

以下要求和准则适用于与数据盒式磁带关联的某些数据库对象。

2.3.1墨盒模式

组成每个盒式磁带的数据库组件必须安装在与盒式磁带同名的架构中。如果盒式磁带使用多个方案，则每个架构名称的前10字符必须与盒式磁带名称相同。注意，模式名称的Oracle中的长度被限制为30个字节，或30中的单字节语言字符。

必须放在盒式模式中的数据盒式磁带的数据库组件包括类型，表，视图，目录，库和包的名称。由于架构名称和用户名在Oracle中始终相同，因此架构名称的选择决定了用户名。

2.3.2弹药筒全球

某些数据库级磁带组件在范围内，因此对所有用户可见，而不是在单个用户或模式的范围内。这种全局变量的示例是角色，同义词和序列。所有全局名称都应以墨盒名称开头，并且格式如下：

C $ CARTRIDGEGLOBAL

2.3.3盒式磁带错误消息名称或错误代码

目前，错误代码ORA-20000保留用于使用Oracle产品的应用程序生成的所有错误。错误消息文本是可自定义的。您应该在以下格式中编写特定于盒式磁带的错误消息：

ORA-20000：C $ CARTRIDGE - NNNN ： ％s

哪里

  • C$ CARTRIDGE是发生错误的盒式磁带的名称

  • NNNN是该盒式磁带唯一的错误消息编号

  • %s是特定于墨盒的错误的描述

注意：

有关编写和管理错误消息的信息 ，请参阅Oracle数据库错误消息

2.3.4盒式磁带安装目录

Oracle建议您创建特定于供应商或客户组织的盒式磁带安装目录。此安装目录应包括盒式磁带的操作系统级组件，例如共享库，配置文件，目录和类似组件。此目录名称应与组织选择的前缀相同，并在平台的根目录下创建。

2.3.5盒式磁带文件

Oracle建议您将与每个盒式磁带关联的错误消息文件放入特定于盒式磁带库的子目录中。也是将配置文件保存在特定于磁带库的子目录中很方便。

2.3.6外部程序的共享库名称

共享库（ .so或.dll文件）可以放入盒式磁带安装目录（所有库名称必须唯一），或放入单独的目录中。如果您使用的是单独的目录，则文件名应以盒式磁带名称开头，不包括初始C$ 。如果有许多这样的库，每个名称应该从墨盒名称的前七个字母开始，再次排除C$ 。

2.4数据盒式磁带部署

在部署级别，您将面临几个常见问题。解决这些问题的最佳方法取决于您的应用程序的需求。以下列表包括应构成核对表基础的任务，以及一些建议的解决方案。

  • 您需要一种方法来安装和卸载盒式磁带组件。这包括库，数据库对象，平面文件，程序，配置工具，管理工具和其他对象。

  • 您应该允许安装多个版本的盒式磁带，以提供向后兼容性和可用性。将Oracle的迁移工具纳入您的战略。

  • 您必须跟踪安装的数据磁带，以支持依赖它们的其他磁带。

  • 您必须跟踪已安装组件的不同版本。

  • 您必须提供升级路径才能迁移到较新版本的磁带。同样，Oracle的迁移工具也很有帮助。

  • 要限制对特定用户和角色访问盒式磁带组件，请将Oracle的安全机制与根据需要在调用者和定义者权限下运行的过程相结合。

  • 您必须跟踪哪些用户有权访问盒式磁带以进行管理。考虑使用具有适当触发器的表。

  • 了解墨盒的安装位置通常是安全和管理问题。目前还没有简单的方法可以知道特定数据库中安装了哪些墨盒，或者用户可以访问墨盒或其任何组件。如果此信息对您的情况很重要，请通过任何方便的方法跟踪它。

2.4.1数据盒式磁带命名约定

考虑如何命名数据盒式磁带的组件;本讨论适用于独立软件供应商（ISV）和其他正在创建其他人使用的盒式磁带的供应商。

请注意，大多数说明数据磁带的示例都不遵循命名约定，主要是因为它们旨在尽可能简单和通用。随着您对该技术的熟悉程度的提高以及您考虑构建供其他人使用的数据盒式磁带，您应该了解并遵循这些命名约定。

命名约定假定为单字节字符集。

2.4.1.1数据磁带中命名约定的需要

在生产环境中，Oracle数据库可能安装了多个数据盒带。这些数据盒可以来自不同的开发组或供应商，因此可以单独开发。每个数据盒式磁带都包含数据库中的各种模式对象，以及操作系统级别可见的其他组件，例如共享库中的外部过程。如果多个数据盒式磁带试图对架构对象或操作系统级实体使用相同的名称，则结果将是错误且不一致的行为。

此外，由于数据盒式磁带运行时操作期间的异常情况可能导致Oracle服务器返回错误，因此防止不同数据盒式磁带的错误或消息代码之间发生冲突非常重要。例如，如果两个磁带盒针对不同的错误条件使用相同的错误代码，则可能会出现这些冲突。具有唯一错误和消息代码可确保轻松识别异常条件的来源。

2.4.1.2唯一名称格式

为防止多个数据盒式磁带组件具有相同的名称，Oracle建议采用以下约定来确保数据盒式磁带的唯一命名。此约定取决于每个组织开发选择唯一名称的数据盒式磁带。Oracle建议盒式磁带开发人员遵循以C$开头的唯一名称格式。

数据盒式磁带及其组件应具有以下格式的名称：

C $ pppptttm.ccccc

表2-1描述了此命名约定格式的各个部分。

Oracle建议名称中的所有字符除美元符号$ ，第二个字符为字母数字：字母，数字，下划线和连字符。

例如，Acme Cartridge Company选择并注册ACME的前缀。它提供音频数据盒式磁带和视频数据盒式磁带，并分别选择AUD和VID作为类型代码。它没有其他信息包含在墨盒名称中，因此它为杂项信息指示器选择任意数字1 。因此，两个墨盒名称是：

  • C $ ACMEAUD1

  • C $ ACMEVID1

对于每个盒式磁带，必须创建单独的架构，Acme使用盒式磁带名称作为架构名称。因此，必须在模式C$ACMEAUD1下创建音频盒式磁带的所有数据库组件，并且必须在架构C$ACMEVID1下创建视频盒式磁带的所有数据库组件。某些组件的示例可能包括：

  • C $ ACMEVID1.mf_rewind

  • C $ ACMEVID1.vid_ops_package

  • C $ ACMEVID1.vid_stream_lib

每个组织负责对象名称的C$pppp部分之后的特定命名要求。例如，Acme Cartridge公司必须确保其所有墨盒都具有唯一的名称，并且墨盒中的所有组件都具有唯一的名称。

2.4.3盒式目录结构和标准

您需要一些目录标准来指定放置二进制文件，支持文件，消息文件，管理文件和库的位置。

您还必须定义安装磁带的数据库用户。一种可能的解决方案是使用EXDSYS ，用于外部数据盒系统用户。

注意：

EXDSYS用户必须具有运行盒式磁带所需的特殊权限。此用户可以作为盒式磁带安装的一部分进行安装，但更好的解决方案是通过将此过程包含在标准数据库创建脚本中，使用户成为数据库安装的一部分。

2.4.4墨盒升级

管理员需要一种安全的方法将盒式磁带及其相关元数据升级到更新版本的盒式磁带。您还需要一个升级数据和删除过时数据的过程。这可能需要安装支持和数据库支持才能转移到更新的数据库盒式磁带

当磁带更换时，管理员还需要使用磁带类型更新表的方法。

2.4.5弹药筒对象的导入和导出

要导入和导出对象，您必须了解Oracle的导入和导出工具如何处理Oracle对象。特别是，您必须知道如何处理类型以及是否导入和导出类型方法，以及是否支持用户定义的方法。

2.4.6盒式磁带版本控制

必须解决两种类型的盒式磁带版本问题：内部和外部。

外部版本控制

外部版本控制更容易发生两个版本问题。您必须能够跟踪盒式磁带版本号，并根据版本信息在安装或配置时采取相应措施。

内部版本控制

内部版本控制是一个更难的问题。理想情况下，您希望有一种机制来支持数据库中多个版本的盒式磁带。这将提供向后兼容性并且还提供高可用性。

2.4.7弹药筒国际化

您可能希望将磁带国际化，以便它们可以支持多种语言并访问全球化支持工具以进行消息和解析。

Oracle建议数据盒组件名称使用ASCII字符集。

如果必须使用ASCII以外的字符集命名数据盒式磁带组件，Oracle会为您分配一个唯一的四字符前缀。但是，这会增加保留前缀所需的字节数。所有Oracle模式对象的名称必须适合30个字节。在ASCII中，这等于30个字符。例如，如果您有一个六字节字符集并请求一个四字符前缀字符串，Oracle可能会将您的请求截断为较少数量的字符。

2.4.8墨盒管理

规划和开发数据盒式磁带时，应考虑管理其使用所涉及的问题。

管理数据盒访问

  • 管理员如何知道谁可以访问墨盒？

    管理员必须管理对特定用户和角色的内部和外部组件（如程序和数据文件）的访问权限。

  • 管理员如何将对某些表，类型，视图和其他盒式磁带组件的访问限制为单个用户和角色？

    出于安全原因，必须允许管理员基于个人限制对类型的访问。

    每个墨盒设计都应该有自己的安全考虑因素。在构建复杂的数据盒式磁带时，您需要一种方法来识别盒式磁带的各种组件和盒式磁带的实例，以便管理员可以授予和撤消可识别组件的安全角色。

祈求者的权利

Invoker的权限是一项特殊权限，允许系统访问通常无法访问的数据库对象。特殊用户SYS具有此类权利。除非您向公众授予权限，否则您为安装和运行盒式磁带而创建的用户需要此权限。

数据盒式磁带配置

数据盒式磁带需要前端来处理部署问题，例如安装和配置工具。虽然每个数据盒式磁带可能具有不同的安全性需求，但是需要允许用户安装，配置和管理数据盒式磁带组件的基本前端。

这个前端可能只是某种形式的知识库或在线文档。无论如何，它应该是在线的，易于导航，并包含展示标准和起点的模板。

2.4.9数据盒式开发方法

通用数据墨盒开发方法。

在开发数据盒式磁带时，采取系统的方法。从简单易用的小任务开始，逐步构建全面的解决方案。

2.4.9.1创建数据盒式计划

创建数据磁带的计划概述。

  1. 试验以下示例： Power Demand Cartridge示例 ， PSBTREE：可扩展索引示例和流水线表函数：接口方法示例 。
  2. 创建自己的数据盒式磁带的原型，从单个用户定义的类型和一些数据元素和方法开始。在扩展盒式磁带的功能时，可以添加用户定义的类型，数据元素和方法，特定的索引类型和用户定义的运算符。
  3. 首先在SQL中完全实现您的方法，然后在需要时将标注添加到3GL代码中。
  4. 测试和调试墨盒。
2.4.9.2开发数据磁带

开发数据盒式磁带的一般开发过程

当您拥有可用的原型时，您可能希望遵循包含以下步骤的开发过程。
  1. 确定您的领域专业知识领域。
  2. 确定与持久性数据相关的专业领域。
  3. 考虑将这些区域中的一个或多个打包为新数据盒或作为现有盒的扩展的可行性。
  4. 使用面向对象的方法来帮助确定要包含在数据盒式磁带中的对象类型。
  5. 一次一个地构建和测试盒式磁带。
