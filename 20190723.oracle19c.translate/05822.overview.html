1概述

本章介绍使用OLAP选项在Oracle数据库中提供的强大分析资源。它包含以下主题：

  • Oracle数据库中的OLAP技术

  • 使用SQL工具和应用程序构建器开发报表和仪表板

  • 维度数据模型概述

1.1 Oracle数据库中的OLAP技术

Oracle数据库提供业界第一个也是唯一一个嵌入式OLAP服务器。在分析多个维度的数据时，Oracle OLAP提供本机多维存储和思考速度响应时间。该数据库为分析提供了丰富的支持，例如时间序列计算，预测，使用加法和非加法运算符的高级聚合以及分配运算符。这些功能使Oracle数据库成为一个完整的分析平台，能够支持整个商业智能和高级分析应用程序。

1.1.1多维技术的完全整合

通过将多维对象和分析集成到数据库中，Oracle提供了两全其美的优势：多维分析的强大功能以及Oracle数据库的可靠性，可用性，安全性和可伸缩性。

Oracle OLAP完全集成到Oracle数据库中。在技术层面，这意味着：

  • 多维数据集和其他维对象是Oracle数据字典中表示的第一类数据对象。

  • CREATE，ALTER，DROP和SELECT语句中的标准SQL语法支持多维数据集和其他维对象。

  • OLAP引擎在Oracle数据库的内核中运行。

  • 维度对象以其本机多维格式存储在Oracle数据库中。

  • 通过向Oracle数据库用户和角色授予和撤消权限，以标准方式管理数据安全性。

您组织的好处是巨大的。Oracle OLAP提供简单的功能：一个数据库，标准管理和安全性，标准接口和开发工具。

1.1.2易于应用开发

Oracle OLAP使您可以轻松地使用有趣的分析内容丰富数据库和应用程序。与提供专有接口的系统相比，对Oracle多维对象和计算的本机SQL访问极大地简化了开发任何类型的仪表板，报告，商业智能（BI）和分析应用程序的任务。此外，SQL访问意味着任何数据库应用程序都可以使用Oracle OLAP分析的强大功能，而不仅仅是传统的，有限的OLAP应用程序集合。

1.1.3易于管理

由于Oracle OLAP完全嵌入在Oracle数据库中，因此没有通常与独立OLAP服务器相关联的管理学习曲线。您可以利用现有的DBA员工，而不是投资专业的管理技能。

Oracle嵌入式OLAP技术的主要管理优势是自动化多维数据集维护。对于独立的OLAP服务器，刷新多维数据集的负担完全由管理员负责。这可能是一项复杂且可能容易出错的工作。您必须创建过程以从关系源中提取已更改的数据，将数据从源系统移动到运行独立OLAP服务器的系统，然后加载并重建多维数据集。在此过程中，您还必须对增量（更改的值）的安全性负责。

相比之下，使用Oracle OLAP，多维数据集刷新完全由Oracle数据库处理。数据库跟踪维对象的陈旧性，自动跟踪源表中的增量，并在刷新过程中自动仅应用更改的值。您只需按适当的时间间隔安排刷新，Oracle数据库将负责其他所有操作。

1.1.4安全

使用Oracle OLAP，标准Oracle数据库安全功能用于保护您的多维数据。

相反，对于独立的OLAP服务器，管理员必须管理两次安全性：一次在关系源系统上，一次在OLAP服务器系统上。此外，他们还必须管理从关系系统到独立OLAP系统的传输数据的安全性。

1.1.5无与伦比的性能和可扩展性

商业智能和分析应用程序由诸如上下层次结构和比较总体价值（如期间，父母份额，对未来时间段的预测以及无数类似计算）等行为主导。通常，这些操作在潜在的分层聚合的整个空间中基本上是随机的。由于Oracle OLAP预先计算或有效地计算定义的多维空间中的所有聚合，因此它为典型的商业智能应用程序提供了无与伦比的性能。

Oracle OLAP查询利用Oracle共享游标，大大降低了内存需求并提高了性能。

当Oracle数据库与Real Application Clusters（Oracle RAC）一起安装时，OLAP应用程序在性能，可伸缩性，故障转移和负载平衡方面可以获得与任何其他应用程序相同的优势。

1.1.6降低成本

所有这些功能都可以降低成本。由于可以利用现有的人员技能，因此降低了管理成本。此外，Oracle数据库可以管理维度对象的刷新，这是一项留给其他系统管理员的复杂任务。标准安全性也降低了管理成本。由于具有SQL知识的大量应用程序开发人员的可用性以及大量基于SQL的开发工具集合，因此可以更快地开发和部署应用程序，从而降低了应用程序开发成本。任何基于SQL的开发工具都可以利用Oracle OLAP。 Oracle OLAP有效管理聚合，使用共享游标和Oracle RAC，从而降低了硬件成本，从而可以从低成本的商品组件构建高度可扩展的系统。

1.3维度数据模型概述

维度对象是OLAP的组成部分。由于OLAP处于联机状态，因此必须快速提供答案;分析师在交互式会话期间提出迭代查询，而不是在隔夜运行的批处理作业中。而且因为OLAP也是分析的，所以查询很复杂。维度对象和OLAP引擎旨在实时解决复杂查询。

维对象包括多维数据集，度量，维度，属性，级别和层次结构。模型的简单性是固有的，因为它定义了代表真实世界业务实体的对象。分析师知道他们有兴趣检查哪些业务度量，哪些维度和属性使数据有意义，以及业务维度如何组织成级别和层次结构。

维度数据模型是高度结构化的。结构意味着管理数据之间关系的规则，并控制如何查询数据。多维数据集是维度模型的物理实现，因此针对维度查询进行了高度优化。OLAP引擎利用这种先天维度来执行行间计算的高效跨多维数据集连接，时间序列分析的外部联接和索引。尺寸预先连接到措施。作为立方体基础的技术基于索引的多维数组模型，该模型提供直接的小区访问。

OLAP引擎以与SQL引擎操作关系对象相同的方式操纵维对象。但是，由于OLAP引擎已经过优化以计算分析函数，并且维度对象已针对分析进行了优化，因此可以在OLAP中比在SQL中更快地计算分析和行函数。

维度模型使Oracle OLAP能够支持高端商业智能工具和应用程序 OracleBI Discoverer Plus OLAP， OracleBI电子表格插件， OracleBI Suite企业版， BusinessObjects Enterprise，和 Cognos ReportNet。

1.3.1立方体

多维数据集提供了一种组织具有相同形状的度量的方法，即它们具有完全相同的尺寸。可以轻松地分析和显示同一个多维数据集中的度量。

多维数据集通常对应于单个事实表或视图。

1.3.2措施

度量使用收集的有关业务操作的事实填充多维数据集的单元格。度量按维度组织，通常包括时间维度。

分析数据库包含历史数据的快照，这些快照源自事务数据库，遗留系统，联合源或其他数据源中的数据。通常认为三年的历史数据适用于分析应用。

在分析师使用这些措施为其决策提供信息时，措施是静态和一致的。它们定期在批处理窗口中更新：每周，每天或每天定期更新。某些管理员通过将时段添加到度量的时间维度来刷新其数据，并且还可以滚动相同数量的最旧时段。每次更新都会为该时间间隔提供特定业务活动的固定历史记录。其他管理员完全重建其数据，而不是执行增量更新。

定义度量的关键决策是最低级别的细节。用户可能永远不会查看此详细数据，但它确定了可以执行的分析类型。例如，市场分析师（不同于订单输入人员）不需要知道密歇根州安娜堡的Beth Miller在2006年7月6日下午2:34订购了10号蓝色波尔卡圆点连衣裙。但是他们可能想知道2006年夏天在美国中西部地区哪种颜色的服装最受欢迎。

基准水平决定了分析师是否可以得到这个问题的答案。对于这个特定问题，Time可以累积到几个月，Customer维度可以汇总到区域，Product维度可以汇总到具有颜色属性的项目（例如连衣裙）。但是，这一级别的汇总数据无法回答这个问题：女性一天中什么时候最有可能下订单？一个重要的决定是数据在加载到数据仓库之前的聚合程度。

计算度量返回在运行时根据存储在一个或多个度量中的数据计算的值。与关系视图一样，计算的度量将查询存储在存储在其他对象中的数据中。由于计算的度量不存储数据，因此可以在不增加数据库大小的情况下创建数十个。您可以将它们用作定义其他计算度量的基础，从而为您可以创建的计算类型增加深度。

1.3.3尺寸

维度包含一组用于标识和分类数据的唯一值。它们形成立方体的边缘，因此形成立方体内的度量。由于度量通常是多维的，因此度量中的单个值必须由每个维度的成员限定才有意义。例如，Sales度量有四个维度：时间，客户，产品和渠道。特定销售价值（43,413.50）仅在特定时间段（2006年2月），客户（Warren Systems），产品（便携式PC）和渠道（目录）合格时才有意义。

基本维度值对应于事实表的唯一键。

度量维度是将度量作为维度成员的维度。使用度量维度，您可以同时为多维数据集中的所有度量生成计算度量。此外，您不必为添加到多维数据集的每个度量创建一组新的计算度量。现有的计算度量适用于度量维度中的新度量。如果您经常创建新度量，这尤其有用。

1.3.4层次结构和级别

层次结构是一种在不同聚合级别组织数据的方法。在查看数据时，分析师使用维度层次结构来识别一个级别的趋势，深入到较低级别以确定这些趋势的原因，并汇总到更高级别以查看这些趋势对业务的更大部分有何影响。

1.3.4.1基于级别的层次结构

每个级别代表层次结构中的位置。基础（或最详细）级别之上的每个级别都包含其下级别的聚合值。不同级别的成员具有一对多的亲子关系 。例如， Q1-05和Q2-05是孩子2005 ，因此2005是父Q1-05和Q2-05 。

假设数据仓库包含每天三次（即每8小时）采集数据的快照。分析师通常可能更愿意查看已汇总为数天，数周，季度或年的数据。因此，时间维度需要具有至少五个级别的层次结构。

同样，具有特定目标的销售经理可能希望在其领土内的销售代表中分配该目标金额;分配需要维度层次结构，其中各个销售代表是特定区域的子值。

层次结构和级别具有多对多关系。层次结构通常包含多个级别，单个级别可以包含在多个层次结构中。

每个级别通常对应于维度表或视图中的列。基本级别是主键。

1.3.4.2基于值的层次结构

虽然层次结构通常由命名级别组成，但它们并非必须如此。维度成员之间的父子关系可能无法定义有意义的级别。例如，在员工维度中，每个经理都有一个或多个报表，这些报表形成父子关系。基于这些关系创建级别（例如个人贡献者，一级管理者，二级管理者等）可能对分析没有意义。同样，财务数据的订单项维度没有级别。这种类型的层次结构称为基于值的层次结构。

1.3.5属性

属性提供有关数据的其他信息。某些属性用于显示。例如，您可能有一个产品维度使用库存保留单位（SKU）作为维度成员。SKU是识别数千种产品的绝佳方式，但如果用于在报告或图表中标记数据，则对大多数人来说毫无意义。您可以定义描述性标签的属性。

您可能还有颜色，味道或大小等属性。这种类型的属性可用于数据选择和回答问题，例如：2005年夏季哪些颜色最受女性服装欢迎？这与前一个夏天相比如何？

时间属性可以提供有关时间维度的信息，这些信息在某些类型的分析中可能很有用，例如识别每个时间段的最后一天或天数。

每个属性通常对应于维度表或视图中的列。
