8.1关于模式对象

模式是数据库对象的集合。模式由数据库用户拥有，并与用户共享同一名称。模式对象是用户创建的逻辑结构。 某些对象（如表或索引）保存数据。其他对象（例如视图或同义词）仅由定义组成。

注意：

表空间和模式之间没有关系。同一模式中的对象可以使用不同表空间中的存储，表空间可以包含来自不同模式的数据。

命名模式对象

数据库中的每个对象都属于一个模式，并且在该模式中具有唯一的名称。如果多个数据库对象位于不同的模式中，则它们可以共享相同的名称。您可以使用模式名称明确引用对象。例如， hr.employees引用hr模式中名为employees的表。（ employees表由hr拥有。）术语数据库对象和模式对象可互换使用。

创建数据库对象时，必须确保在预期的架构中创建它。一种方法是以拥有该模式的用户身份登录数据库，然后创建该对象。通常，将属于单个应用程序的所有对象放在同一模式中。

架构对象名称必须遵守某些规则。除了在模式中是唯一的之外，模式对象名称不能超过30个字节，并且必须以字母开头。如果您尝试创建名称违反任何这些规则的对象，则数据库会引发错误。

DDL标签

您可以使用SQL或SQL Developer创建和操作模式对象。

使用SQL Developer创建模式对象时，可以单击“ DDL”选项卡以显示与使用图形用户界面指定的模式对象属性等效的SQL语句。SQL Developer提交此SQL语句以创建架构对象。即使该语句不完整，此选项也会显示该语句，因此您必须输入架构对象的所有规范才能查看SQL Developer提交的完整SQL语句。

8.2关于模式对象管理权限

作为数据库管理员（DBA），您可以在自己的架构和任何其他架构中创建，修改和删除架构对象。出于本讨论的目的，数据库管理员被定义为被授予DBA角色的任何用户。 这包括默认情况下的SYS和SYSTEM用户。Oracle建议仅将DBA角色授予需要管理类型访问权限的用户。

您可以启用其他用户来管理架构对象，而无需授予其DBA权限。例如，常见的场景是使应用程序开发人员能够在他或她自己的架构中创建，修改和删除架构对象。为此，您授予应用程序开发人员的RESOURCE角色。

8.4.1关于表格

该表是Oracle数据库中数据存储的基本单位。它包含所有用户可访问的数据。 每个表由列和行组成。例如，在employees表中，有名为last_name和employee_id列。表中的每一行代表一个不同的员工，并包含last_name和employee_id的值。

创建表时，指定表类型，并定义其列和约束。约束是有助于保持数据完整性的规则。

8.4.1.1关于表类型
Oracle数据库中最常见的表类型是关系表，它由类似于employees表的简单列构成。支持另外两种表类型： 对象表和XMLType表。 三种表类型中的任何一种都可以定义为永久或临时 。临时表包含仅在事务或会话期间存在的会话私有数据。它们在结果集必须暂时保存在内存中的应用程序中很有用，可能是因为结果集是通过运行多个操作构建的。

您可以在堆或索引组织结构中构建关系表。在堆结构中，行不以任何特定顺序存储。在索引组织表中，行顺序由一个或多个所选列中的值确定。对于某些应用程序，索引组织表提供增强的性能和更有效的磁盘空间使用。

本节介绍永久的堆组织表。有关其他表类型以及何时使用它们的信息，请参见Oracle数据库管理员指南和Oracle数据库概念 。有关使用SQL创建表所需的语法，请参见Oracle数据库SQL语言参考 。

8.4.1.2.1数据类型

数据类型属性定义要存储在列中的数据类型。创建表时，必须为其每个列指定数据类型。

数据类型定义每列可包含的值的域。例如， DATE列不能接受February 29的值（闰年除外）或值2或SHOE 。随后插入列中的每个值都采用列数据类型。例如，如果将17-JAN-2004插入日期列，则Oracle数据库会在验证它转换为有效日期后将该字符串视为日期值。

8.4.1.2.2 NOT NULL列约束
约束确定列的有效值。在SQL Developer中，您可以在“创建表”对话框页面的列级别定义的唯一约束是NOT NULL约束，该约束要求在插入或更新行时在列中包含值。 与“ 关于表级约束 ”中描述的其他约束（可以定义为列定义的一部分或表定义的一部分）不同， NOT NULL约束必须定义为列定义的一部分。

当必须为列提供数据以确保数据库的完整性时，请使用NOT NULL约束。例如，如果所有员工必须属于特定部门，则必须使用NOT NULL约束定义包含部门标识符的列。但是，如果数据可能未知，或者在添加或更改行时可能不存在，则不要将列定义为NOT NULL 。不能使用NOT NULL约束的列的示例是邮件地址中的第二个可选行。

数据库自动将NOT NULL约束添加到表的主键中包含的一列或多列。

8.4.1.2.3默认值
只要插入新行而没有为列提供值，该值就会自动存储在列中。您可以将默认值指定为文字或表达式。 但是，构造表达式的方式存在限制。有关详细信息，请参见Oracle Database SQL语言参考 。

8.4.1.3关于表级约束
在Oracle数据库中，您可以应用规则来保持数据的完整性。例如，在包含员工数据的表中，员工姓名列不能接受NULL作为值。 同样，在此表中，您不能拥有两个具有相同ID的员工。

Oracle数据库使您能够在表级别和列级别应用称为约束的数据完整性规则。尝试插入或更新违反约束的行的任何SQL语句都会导致错误，并且会回滚该语句。同样，如果任何现有行违反新约束，则任何尝试将新约束应用于填充表也会导致错误。

您可以在表级应用的约束类型如下：

  • 主键 - 要求列（或列的组合）是行的唯一标识符。主键列不允许NULL值。

  • 唯一键 - 要求任何两行都不能在指定列或列组合中具有重复值。列集被认为是唯一键。

  • 检查 - 要求列（或列的组合）满足表中每一行的条件。检查约束必须是布尔表达式。每次插入或更新行时都会对其进行评估。检查约束的示例是： SALARY > 0 。

  • 外键 - 要求对于特定列（或列的组合），子表中的所有列值都存在于父表中。包含外键的表称为依赖表或子表。外键引用的表称为父表。外键约束的一个示例是employees表的department列必须包含父部门表中存在的部门ID。

可以创建约束并通常使用不同的状态进行修改。选项包括启用或禁用，用于确定在添加或修改行时是否检查约束，以及延迟还是立即检查约束，这会导致约束验证分别在事务结束时或在语句结束时发生。

8.4.1.4.1用户定义的类型和大对象（LOB）
您的新表可以包含使用用户定义类型定义的一个或多个列。用户定义的类型允许单行中的单个列包含多个值。 多个值可以表示为数组，嵌套表或对象，其中对象类型表示真实世界的实体，例如采购订单。（检索采购订单类型列值可以返回一个包含订单号，客户号，金额等记录 ）。用户定义的类型是使用CREATE TYPE语句创建的，在Oracle数据库SQL语言参考中有详细描述。

大物件 （ LOB）列用于包含非结构化数据（例如文本或流视频），并且可以保存数TB的信息。

要使用SQL Developer创建LOB列，请在创建表时单击“ 高级”复选框。然后单击LOB Parameters以查看创建LOB列时可用的选项。有关创建LOB列的详细信息，请参见Oracle数据库SQL语言参考 。

8.4.1.4.2分区表和索引
您可以对表和索引进行分区 。分区有助于支持非常大的表和索引，使您能够将表和索引划分为更小且更易于管理的分区 。 不必修改SQL查询和DML语句来访问分区表和索引。分区对应用程序是透明的。

定义分区后，某些操作会变得更有效。例如，对于某些查询，数据库可以通过仅访问分区的子集而不是整个表来生成查询结果。这种技术（称为分区修剪 ）可以在提高性能方面提供数量级的增益。此外，数据管理操作可以在分区级别进行，而不是在整个表格上进行。这样可以减少数据加载等操作的时间;索引创建和重建;和备份和恢复。

每个分区都可以存储在自己的表空间中，与其他分区无关。由于不同的表空间可以位于不同的磁盘上，因此可以提供可以更好地调整可用性和性能的表结构。将分区存储在不同磁盘上的不同表空间中也可以优化可用的存储使用，因为经常访问的数据可以放在高性能磁盘上，而不经常检索的数据可以放在较便宜的存储上。

对于管理大量数据的许多类型的应用程序，分区很有用。在线事务处理（OLTP）系统通常受益于可管理性和可用性的改进，而数据仓库系统受益于提高的性能和可管理性。

要使用SQL Developer指定分区选项，请在创建表时单击“ 高级”复选框。然后单击“ 分区”以查看可用的分区选项。有关分区的详细信息，请参见Oracle数据库SQL语言参考 。

8.4.1.4.4压缩表

8.5.1关于索引

索引是与表关联的可选模式对象。您可以在表上创建索引以提高查询性能。正如指南中的索引可帮助您快速查找特定信息一样，Oracle数据库索引可以快速访问表数据。

您可以根据需要在表上创建任意数量的索引。您可以在表的一列或多列上创建每个索引。例如，在采购订单表中，如果在供应商编号列上创建索引，则可以按供应商编号顺序依次访问表的行，而无需对行进行实际排序。此外，您可以直接访问发给特定供应商的所有采购订单，而无需扫描整个表。

创建索引后，它将自动维护并由数据库使用。对表的数据或结构的更改（例如添加新行，更新行或删除行）将自动合并到所有相关索引中。这对用户来说是透明的。

某些索引是通过放置在表上的约束隐式创建的。例如，数据库会自动在主键约束或唯一键约束的列上创建索引。

8.5.1.1索引和性能
索引通常可以提高在单个现有行或少量现有行上运行的查询和DML语句的性能。 但是，过多的索引会增加添加，修改或删除行的语句的处理开销。

在添加其他索引之前，请检查数据库的性能以查询和DML。然后，您可以在添加新索引后比较性能。

8.5.1.2索引属性
可以使用索引属性的各种组合以多种方式创建索引。 主索引属性如下：

标准（B树）和位图

标准B树索引包含索引键中每个值的条目以及存储该值的行的磁盘地址。B树索引是Oracle数据库中的默认和最常见的索引类型。

位图索引使用位串来封装值和潜在的行地址。它比B树索引更紧凑，可以更有效地执行某些类型的检索。但是，对于一般用途，位图索引在表上的行操作期间需要更多开销，并且应主要用于数据仓库环境，如Oracle数据库数据仓库指南中所述 。

升序和降序

通过索引的默认搜索是从最低到最高值，其中字符数据按ASCII值排序，数字数据从最小到最大数字排序，日期从最早到最新值。此默认搜索方法在作为升序索引创建的索引中执行。您可以通过使用降序选项创建相关索引来使索引搜索反转搜索顺序。

专栏和功能

通常，索引条目基于表中一列或多列中的值。这是一个列索引。或者，您可以创建基于函数的索引，其中索引值是从表数据派生的。例如，要查找可以使用大写和小写字母的各种组合的字符数据，可以使用基于函数的索引（基于UPPER()函数）来查找值，就好像它们都是大写字母一样。

单列和连接

您可以只在一列上创建索引（称为单列索引） ，也可以在多列上创建索引 ，称为连锁索引 。当所有索引列都可能包含在频繁执行的SQL语句的WHERE子句中时，连接索引很有用。

未分区和分区

与表一样，您可以对索引进行分区。在大多数情况下，在对关联表进行分区时对索引进行分区以及使用与表相同的分区方案对索引进行分区是很有用的。（例如，如果按销售日期对表进行范围分区，则在销售日期创建索引，并使用与表分区相同的范围对索引进行分区。）这称为本地分区索引。但是，您不必使用与其表相同的分区方案对索引进行分区。您也可以创建一个非分区的，或 全球的 ， 分区表上的索引。

8.6.1关于视图

视图是一个或多个表或其他视图中的数据的自定义表示。您可以将它们视为存储的查询。 视图实际上不包含数据，而是从它们所基于的表中派生数据。这些表称为视图的基表 。

与表类似，可以查看，更新，插入和删除视图，但有一些限制。对视图执行的所有操作实际上都会影响视图的基表。视图可以通过限制对表的一组预定行和列的访问来提供额外的安全级别。它们还可以隐藏数据复杂性并存储复杂查询。

许多重要的视图都在SYS模式中。有两种类型： 静态数据字典视图和动态性能视图 。有关SYS模式中视图的完整描述，请参见Oracle数据库参考 。

静态数据字典视图

数据字典视图称为静态视图，因为它们不经常更改，仅在对数据字典进行更改时才会更改。数据字典更改的示例包括创建新表或向用户授予权限。

许多数据字典表有三个相应的视图：

  • DBA_视图显示整个数据库中的所有相关信息。DBA_视图仅供管理员使用。

    DBA_视图的一个示例是DBA_TABLESPACES ，它包含数据库中每个表空间的一行。

  • ALL_视图显示当前用户可访问的所有信息，包括来自当前用户的模式的信息，以及来自其他模式中的对象的信息（如果当前用户可通过特权或角色访问这些对象）。

    ALL_视图的一个示例是ALL_TABLES ，其中包含用户具有对象权限的每个表的一行。

  • USER_视图显示当前用户架构中的所有信息。查询这些视图不需要特殊权限。

    USER_视图的一个示例是USER_TABLES ，其中包含用户拥有的每个表的一行。

DBA_ ， ALL_和USER_视图中的列通常几乎相同。USER_视图通常没有OWNER列。

动态性能视图

动态性能视图监视正在进行的数它们仅供管理员使用。动态性能视图的名称以字符V$开头。因此，这些视图通常称为V$视图。

V$视图的一个示例是V$SGA ，它返回各种系统全局区域（SGA）内存组件的当前大小。

8.7.1关于存储在数据库中的程序代码

Oracle数据库提供了将程序代码存储在数据库中的功能。开发人员用PL / SQL或Java编写程序代码，并将代码存储在模式对象中。 作为DBA，您可以使用SQL Developer来管理程序代码对象，例如：

  • PL / SQL包，过程，函数和触发器

  • Java源代码（Java源代码）和已编译的Java类

您可以执行的操作包括创建，编译，创建同义词，授予权限以及显示这些代码对象的依赖关系。您还可以使用SQL Developer编辑和调试PL / SQL代码对象。您可以通过单击Schema子页面的Programs部分中的链接来访问这些对象的管理页面。

请注意，创建和管理程序代码对象主要是应用程序开发人员的责任。但是，作为DBA，您可能需要协助管理这些对象。程序代码对象最常见的任务可能是重新验证（编译）它们，因为如果它们所依赖的模式对象发生更改或被删除，它们可能会失效。

注意：

除程序代码对象之外的其他类型的模式对象可能变得无效。例如，如果删除表，则引用该表的任何视图都将变为无效。

8.7.2验证（编译）无效的模式对象

作为数据库管理员（DBA），可能会要求您重新验证已变为无效的架构对象。当对它们所依赖的对象进行更改时，架构对象（例如触发器，过程或视图）可能会失效。 例如，如果PL / SQL过程包含对表的查询，并且您修改了查询中引用的表列，则PL / SQL过程将变为无效。您可以通过编译来重新验证模式对象。

注意：

通过编译来重新验证存储程序代码的模式对象并不总是可能的。您可能必须先采取补救措施。例如，如果视图因其引用的表被删除而变为无效，则编译视图会生成一条错误消息，指示该表不存在。在重新创建表之前，无法验证视图。

8.8使用其他架构对象

除了使用SQL Developer管理表，索引，视图和程序代码之外，还可以使用SQL Developer来管理其他模式对象。 例如：

  • 序列

    序列是生成唯一整数的数据库对象。每次查询序列时，它会将其当前值递增指定的量并返回结果整数。可以由多个用户同时查询序列，并且每个用户接收唯一值。因此，使用序列为表中的主键提供值是一种简单的方法，可以保证键值是唯一的，无论将数据插入表中的用户数是多少。

  • 同义词

    同义词是任何架构对象的别名，例如表或视图。同义词提供了一种从应用程序或用户隐藏底层数据库结构的简便方法。同义词可以是私人的或公共的。公共同义词不必使用模式名称限定，而私有同义词则必须使用引用私有同义词的用户不是同义词所有者。例如，请考虑以下查询，该查询由已在HR.EMPLOYEES表上授予READ对象权限的用户发出：

    SELECT employee_id，salary FROM hr.employees ORDER BY salary

    现在假设你创建一个名为公共同义词PERSONNEL作为一个别名HR.EMPLOYEES表，您授予READ上的特权HR.EMPLOYEES表PUBLIC （所有数据库用户）。使用公共同义词，任何用户都可以发出以下更简单的查询：

    SELECT employee_id，工资人员ORDER BY工资

    创建此查询的用户无需知道包含人员数据的架构的名称。

    注意：

    如果用户拥有名为personnel的表，则在查询中使用该表。如果不存在此类表，则数据库将解析公共同义词并使用HR.EMPLOYEES表。

    同义词的另一个好处是，即使模式名称不同，您也可以在开发数据库中使用与生产数据库中相同的同义词。此技术使应用程序代码能够在两种环境中不加修改地运行。例如，前面的查询将不在于有一个开发数据库运行错误EMPLOYEES表中DEV1模式，如果PERSONNEL同义词在开发数据库中定义指向DEV1架构。

    因为同义词只是一个别名，所以除了数据字典中的定义之外，它不需要存储。要在查询中引用同义词，您必须拥有它指向的对象的权限。同义词本身无法保证。如果您将同义词的对象权限授予用户，则您将授予同义词指向的对象的权限。

  • 数据库链接

    数据库链接是指向另一个Oracle数据库的模式对象。您使用数据库链接来查询或更新远程数据库中的对象。数据库链接用于分布式数据库环境， Oracle数据库管理员指南中对此进行了描述。
