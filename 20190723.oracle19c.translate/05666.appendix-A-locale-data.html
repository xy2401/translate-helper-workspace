A.4字符集

Oracle数据库支持的字符集根据三大类列在以下部分中。

  • 推荐的数据库字符集

  • 其他字符集

  • 仅客户端字符集

另外，列出了公共字符集子集/超集组合。某些字符集只能用于某些数据类型。例如，AL16UTF16字符集只能用作NCHAR字符集，而不能用作数据库字符集。

注释部分还记录了字符集的其他独特功能，这些功能对用户或数据库管理员可能很重要。例如，信息包括字符集是否支持欧元货币符号，是否支持用户定义的字符，以及字符集是否是ASCII的严格超集。 （只有当所有架构数据都是新字符集的严格子集时，才可以使用Database Migration Assistant for Unicode将现有数据库迁移到新字符集。）

字符集表的注释列的关键是：

  • SB：单字节编码
  • MB：多字节编码
  • 修复：固定宽度多字节编码
  • ASCII：ASCII的严格超集
  • 欧元：支持欧元符号
  • UDC：支持用户定义的字符

Oracle数据库不记录单个代码页布局。有关特定字符集，其字符库和代码点值的特定详细信息，您可以使用Oracle Locale Builder。否则，您应参考实际的国家，国际或供应商特定标准。

A.6语言整理

Oracle数据库提供三种语言排序规则，单语，多语言和UCA。

通常创建单语校对以便以单一语言对字符数据进行排序，并以相应的语言命名。某些语言有多个排序规则，为每种语言实现多种排序标准。一些单语校对具有处理特殊语言案例的扩展版本。扩展版本的名称以字母X为前缀。这些特殊情况通常意味着一个字符的排序方式类似于两个字符的序列，或者两个字符的序列排序为一个字符。例如， ch和ll在XSPANISH中被视为单个字符。扩展的单语校对还可以定义特殊的语言特定的大写和小写规则，以覆盖字符集的标准规则。

通过将_CI或_AI分别附加到语言校对名称，可以将所有语言校对另外指定为不区分大小写或不区分重音。

A.9.5不支持 AL24UTFFSS字符集

Unicode数据库版本7中引入了Unicode字符集AL24UTFFSS，以支持UTF-8编码方案，并基于Unicode标准1.1。Oracle9 i中不支持AL24UTFFSS。Oracle数据库开始在Oracle8中提供Unicode数据库字符集UTF8，在Oracle9 i中提供 AL32UTF8。AL32UTF8字符集已更新，以符合Oracle Database 12 c第2版（12.2）中的Unicode 7.0和Oracle Database Release 18c及更高版本中的Unicode 9.0。

现有AL24UTFFSS数据库的迁移路径是在升级到Oracle Database 9 i或更高版本之前升级到UTF8。在尝试将现有数据库字符集迁移到UTF8之前，可以使用字符集扫描程序在Oracle8中进行数据分析。

A.9.6对Oracle数据库语言和区域定义文件的更新

自Oracle Database 10g以来，对某些语言和区域定义文件中的内容进行了更改。这些更新对于更正不再符合Oracle数据库支持的某些语言和区域中的本地约定的旧定义是必需的。这些更改包括对货币符号，月份名称和组分隔符的修改。一个例子是巴西的当地货币符号。这在Oracle数据库10g中从Cr$更新为R$ 。

请参阅$ORACLE_HOME/nls/data/old/data_changes.html文件中记录的“Oracle数据库语言和区域定义更改”表，以获取更改的详细列表。

您应该检查现有的应用程序代码，以确保使用此Oracle数据库发行版中包含的最新语言环境定义文件。如果您无法对特定于语言环境的代码进行更改以支持您的应用程序，则可以使用此Oracle数据库发行版中包含的Oracle9 i语言环境定义文件。

要恢复到Oracle9 i语言和区域行为：

  1. 关闭数据库。

  2. 从$ORACLE_HOME/nls/data/old目录运行脚本cr9idata.pl 。

  3. 将ORA_NLS10环境变量设置为新创建的$ORACLE_HOME/nls/data/9idata目录。

  4. 重启数据库。

需要为需要恢复到Oracle9 i定义文件的所有Oracle数据库客户端重复步骤2和3。

注意：

Oracle强烈建议您使用此Oracle数据库发行版中包含的最新语言环境定义文件。Oracle9 i语言环境定义文件将在以后的版本中得到支持。
