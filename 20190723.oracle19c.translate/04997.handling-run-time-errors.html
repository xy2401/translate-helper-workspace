9处理运行时错误

应用程序必须预测运行时错误并尝试从中恢复。本章提供了有关错误报告和恢复的深入讨论。您将学习如何使用SQLSTATE状态变量以及SQL通信区域（SQLCA）和WHENEVER指令来处理错误和状态更改。您还将学习如何使用Oracle通信区域（ORACA）诊断问题。本章包含以下主题：

9.1错误处理的必要性

每个应用程序的重要部分都必须用于错误处理。错误处理的主要原因是它允许程序在出现错误时继续运行。错误源于设计错误，编码错误，硬件故障，无效的用户输入和许多其他来源。

您无法预料到所有可能的错误，但您可以计划处理对您的程序有意义的某些类型的错误。对于Pro * C / C ++预编译器，错误处理意味着检测并从SQL语句执行错误中恢复。您还可以准备处理警告，例如“值截断”和状态更改，例如“数据结束”。在每个SQL数据操作语句之后检查错误和警告条件尤为重要，因为在处理表中的所有符合条件的行之前，INSERT，UPDATE或DELETE语句可能会失败。

9.2错误处理替代方案

您可以使用几种方法来检测应用程序中的错误和状态更改。本章介绍了这些替代方案，但是，没有针对您应该使用的方法提出具体建议。毕竟，该方法取决于您正在构建的应用程序或工具的设计。

9.2.1状态变量

您可以声明一个单独的状态变量SQLSTATE或SQLCODE，在每个可执行SQL语句之后检查其值，并采取适当的操作。该操作可能是调用错误报告函数，然后在错误无法恢复时退出程序。或者，您可以调整数据或控制变量并重试操作。

9.2.2 SQL通信区域

您可以使用的另一种方法是在程序中包含SQL通信区域结构（ sqlca ）。此结构包含在Oracle处理SQL语句后在运行时填充的组件。

注意：

在本指南中， sqlca结构通常使用SQL通信区域 （SQLCA）的首字母缩写来引用。当本指南引用C 结构中的特定组件时，将使用结构名称（ sqlca ）。

SQLCA在头文件sqlca.h定义，您使用以下任一语句将其包含在程序中：

  • EXEC SQL包含SQLCA;

  • #include <sqlca.h>

Oracle在每个可执行的 SQL语句之后更新SQLCA。（声明性陈述后SQLCA值不变。）通过检查存储在SQLCA中的Oracle返回码，您的程序可以确定SQL语句的结果。这可以通过以下两种方式完成：

  • 使用WHENEVER指令进行隐式检查

  • 显式检查SQLCA组件

您可以使用WHENEVER指令，对SQLCA组件进行代码显式检查，或同时执行这两项操作。

SQLCA中最常用的组件是状态变量（ sqlca.sqlcode ），以及与错误代码（ sqlca.sqlerrm.sqlerrmc ）关联的文本。其他组件包含警告标志和有关SQL语句处理的其他信息。

注意：

SQLCODE（大写）总是指一个单独的状态变量，而不是SQLCA的一个组件。 SQLCODE声明为整数 。当引用名为sqlcode的SQLCA的组件时，始终使用完全限定名称sqlca.sqlcode 。

如果需要有关运行时错误的更多信息，而不是SQLCA提供的信息，则可以使用ORACA。 ORACA是一个处理Oracle通信的C 结构 。它包含游标统计信息，有关当前SQL语句的信息，选项设置和系统统计信息。

9.3 SQLSTATE状态变量

预编译器命令行选项MODE控制ANSI / ISO合规性。当MODE = ANSI时，声明SQLCA数据结构是可选的。但是，您必须声明一个名为SQLCODE的单独状态变量。 SQL标准指定了一个名为SQLSTATE的类似状态变量，您可以使用或不使用SQLCODE。

执行SQL语句后，Oracle Server会将状态代码返回到当前范围内的SQLSTATE变量。状态代码指示SQL语句是成功执行还是引发异常（错误或警告条件）。为了提高互操作性 （系统轻松交换信息的能力），SQL标准预定义了所有常见的SQL异常。

与仅存储错误代码的SQLCODE不同，SQLSTATE存储错误和警告代码。此外，SQLSTATE报告机制使用标准化编码方案。因此，SQLSTATE是首选的状态变量。SQLCODE是SQL-92的弃用功能，仅为了与SQL-89的兼容性而保留。从SQL-92之后的SQL标准的所有版本中删除了SQLCODE。

9.3.1关于声明SQLSTATE

当MODE = ANSI时，您必须声明SQLSTATE或SQLCODE。声明SQLCA是可选的。当MODE = ORACLE时，如果声明SQLSTATE，则不使用它。

与存储有符号整数并且可以在Declare Section之外声明的SQLCODE不同，SQLSTATE存储5个字符的以null结尾的字符串，并且必须在Declare Section中声明。您将SQLSTATE声明为

char SQLSTATE [6]; / *大写是必需的。* /

注意：

必须使用正好 6个字符的维度声明SQLSTATE。

9.3.2 SQLSTATE值

SQLSTATE状态代码由一个2字符的类代码组成，后面紧跟一个3字符的子类代码 。除了类代码00（“成功完成”）之外，类代码表示一类异常。并且，除了子类代码000（“不适用”）之外，子类代码表示该类别中的特定异常。例如，SQLSTATE值'22012'由类代码22（“数据异常”）和子类代码012（“除以零”）组成。

SQLSTATE值中的五个字符中的每一个都是一个数字（0..9）或一个大写的拉丁字母（A ..Z）。以0..4范围内的数字或A范围内的字母开头的类代码。H保留用于预定义条件（SQL标准中定义的条件）。所有其他类代码都保留用于实现定义的条件。在预定义的类中，子类代码以0..4范围内的数字或范围A中的字母开头。H保留用于预定义的子条件。所有其他子类代码都保留用于实现定义的子条件。图9-1显示了编码方案。

9.3.3关于使用SQLSTATE

当您使用选项设置MODE = ANSI进行预编译时，以下规则适用于将SQLSTATE与SQLCODE或SQLCA一起使用。必须在Declare Section中声明SQLSTATE;否则，它会被忽略。

9.3.3.1如果您声明SQLSTATE

  • 声明SQLCODE是可选的。如果在Declare Section中声明SQLCODE，则Oracle Server会在每次SQL操作后将状态代码返回到SQLSTATE和SQLCODE。但是，如果在Declare Section之外声明SQLCODE，则Oracle仅将状态代码返回给SQLSTATE。

  • 声明SQLCA是可选的。如果声明SQLCA，Oracle会将状态代码返回给SQLSTATE和SQLCA。在这种情况下，为避免编译错误，请不要声明SQLCODE。

9.3.3.2如果您未声明SQLSTATE

  • 您必须在Declare Section内部或外部声明SQLCODE。每次SQL操作后，Oracle Server都会将状态代码返回给SQLCODE。

  • 声明SQLCA是可选的。如果声明SQLCA，Oracle会将状态代码返回给SQLCODE和SQLCA。

您可以通过使用自己的代码显式检查SQLSTATE或使用WHENEVER SQLERROR指令隐式检查SQLSTATE来了解最新的可执行SQL语句的结果。仅在可执行SQL语句和PL / SQL语句之后检查SQLSTATE。

9.4关于声明SQLCODE

当MODE = ANSI，并且您尚未声明SQLSTATE状态变量时，必须在Declare Section内部或外部声明一个名为SQLCODE的长整数变量。一个例子如下：

/ *声明主变量* / EXEC SQL BEGIN DECLARE SECTION; int emp_number，dept_number; char emp_name [20]; EXEC SQL END DECLARE SECTION; / *声明状态变量 - 必须大写* / long SQLCODE;

当MODE = ORACLE时，如果声明SQLCODE，则不使用它。

您可以声明多个SQLCODE。访问本地SQLCODE受其程序范围的限制。

在每次SQL操作之后，Oracle都会将状态代码返回到当前范围内的SQLCODE。因此，您的程序可以通过显式检查SQLCODE或使用WHENEVER指令隐式检查来了解最新SQL操作的结果。

在特定编译单元中声明SQLCODE而不是SQLCA时，预编译器会为该单元分配内部SQLCA。您的主程序无法访问内部SQLCA。如果声明SQLCA 和 SQLCODE，Oracle会在每次SQL操作后向两者返回相同的状态代码。

9.5使用SQLCA进行错误报告的关键组件

错误报告取决于SQLCA中的变量。本节重点介绍错误报告的关键组件。下一节将详细介绍SQLCA。

9.5.1状态代码

每个可执行的SQL语句都将状态代码返回给SQLCA变量sqlcode ，您可以使用WHENEVER指令隐式检查或使用您自己的代码显式检查。

零状态代码表示Oracle在未检测到错误或异常的情况下执行了该语句。正状态代码表示Oracle执行了该语句但检测到异常。负状态代码表示Oracle由于错误而未执行SQL语句。

9.5.2警告标志

警告标志在SQLCA变量sqlwarn[0]通过sqlwarn[7] ，您可以隐式或显式地检查。这些警告标志对于Oracle未被视为错误的运行时条件很有用。如果没有可用的指示符变量，Oracle会发出错误消息。

9.5.3行处理计数

最近执行的SQL语句处理的行数在SQLCA变量sqlca.sqlerrd[2] ，您可以显式检查。

严格来说，此变量不适用于错误报告，但它可以帮助您避免错误。例如，假设您希望从表中删除大约十行。删除后，检查sqlca.sqlerrd[2]并找到已处理的75行。为安全起见，您可能希望回滚删除并检查WHERE子句搜索条件。

9.5.4解析误差偏移

在执行SQL语句之前，Oracle必须解析它以确保它遵循语法规则并引用有效的数据库对象。如果Oracle发现错误，则会在SQLCA变量sqlca.sqlerrd[4]存储偏移量，您可以显式检查该变量。偏移量指定SQL语句中解析错误开始时的字符位置。与普通C字符串一样，第一个字符占据零位。例如，如果偏移量为9，则解析错误从第10 个字符开始。

解析错误偏移用于执行单独的准备/解析的情况。这是动态SQL语句的典型特征。

解析错误可能由丢失，错放或错误拼写的关键字，无效选项等引起。例如，动态SQL语句：

“UPDATE emp SET jib =：job_title WHERE empno =：emp_number”

导致解析错误

ORA-00904：列名无效

因为列名JOB拼写错误。sqlca.sqlerrd[4]值为15，因为错误的列名JIB从第16 个字符开始。

如果您的SQL语句不会导致解析错误，Oracle sqlca.sqlerrd[4]设置为零。如果解析错误从第一个字符（占据零位置）开始，Oracle也会将sqlca.sqlerrd[4]设置为零。因此，仅当sqlca.sqlcode为负时才检查sqlca.sqlerrd[4] ，这意味着发生了错误。

9.5.5错误消息文本

SQLCA变量SQLERRMC中提供了错误代码和Oracle错误消息。最多存储前70个字符的文本。要获取超过70个字符的消息的全文，请使用sqlglm()函数。

9.6使用SQL通信区域（SQLCA）

SQLCA是一种数据结构。其组件包含Oracle在执行SQL语句时更新的错误，警告和状态信息。因此，SQLCA始终反映最近SQL操作的结果。要确定结果，可以在SQLCA中检查变量。

您的程序可以有多个SQLCA。例如，它可能有一个全局SQLCA和几个本地SQLCA。访问本地SQLCA受其在程序范围内的限制。Oracle仅将信息返回到范围内的SQLCA。

注意：

当您的应用程序使用Oracle Net同时访问本地和远程数据库的组合时，所有数据库都会写入一个SQLCA。每个数据库没有不同的SQLCA。

9.6.1关于声明SQLCA

要么

9.7关于获取错误消息的全文

SQLCA可以容纳长达70个字符的错误消息。要获取更长（或嵌套）错误消息的完整文本，您需要使用sqlglm()函数。语法是 

void sqlglm（unsigned char * message_buffer，size_t * buffer_size，size_t * message_length）;

哪里：

句法 描述

message_buffer

是您希望Oracle在其中存储错误消息的文本缓冲区（Oracle空白填充到此缓冲区的末尾）。

缓冲区大小

是一个标量变量，指定缓冲区的最大大小（以字节为单位）。

message_length中

是标量变量，其中Oracle存储错误消息的实际长度（如果未截断）。

注意：

sqlglm()函数的最后两个参数的类型一般显示为size_t指针。但是在您的平台上，它们可能具有不同的类型。例如，在许多UNIX工作站端口上，它们是unsigned int * 。

您应该检查文件sqlcpr.h ，它位于系统的标准包含目录中，以确定这些参数的数据类型。

Oracle错误消息的最大长度为512个字符，包括错误代码，嵌套消息和表和列名称等消息插入。sqlglm()返回的错误消息的最大长度取决于您为buffer_size指定的值。

以下示例调用sqlglm()以获取最多200个字符的错误消息：

在使用多个运行时上下文的情况下，使用sqlglmt（）的版本，该版本采用上下文来获取正确的错误消息。

9.8关于使用WHENEVER指令

默认情况下，预编译程序会忽略Oracle错误和警告条件，并在可能的情况下继续处理。要执行自动条件检查和错误处理，您需要WHENEVER指令。

使用WHENEVER指令，您可以指定Oracle检测到错误，警告条件或“未找到”条件时要采取的操作。这些操作包括继续下一个语句，调用例程，分支到带标签的语句或停止。

您使用以下语法编写WHENEVER指令：

EXEC SQL WHENEVER <condition> <action>;

9.8.1 WHENEVER条件

您可以让Oracle自动检查SQLCA是否满足以下任何条件。

9.8.1.1 SQLWARNING

sqlwarn[0]已设置，因为Oracle返回警告（警告标志之一， sqlwarn[1]到sqlwarn[7] ，也设置）或SQLCODE具有+1403以外的正值。例如，当Oracle将截断的列值分配给输出主机变量时，会设置sqlwarn[0] 。

当MODE = ANSI时，声明SQLCA是可选的。但是，要使用WHENEVER SQLWARNING， 必须声明SQLCA。

9.8.1.2 SQLERROR

SQLCODE具有负值，因为Oracle返回了错误。

9.8.1.3未找到

SQLCODE的值为+1403（MODE = ANSI时为+100），因为Oracle找不到符合WHERE子句搜索条件的行，或者SELECT INTO或FETCH没有返回任何行。

当MODE = ANSI时，在没有行的INSERT之后，+ 100返回到SQLCODE。

9.8.2 WHENEVER行动

当Oracle检测到上述条件之一时 ，您可以让程序执行以下任何操作。

9.8.2.1继续

如果可能，您的程序将继续运行下一个语句。这是默认操作，相当于不使用WHENEVER指令。您可以使用它来关闭条件检查。

9.8.2.2 DO

您的程序将控制转移到程序中的错误处理功能。到达例程结束时，控制转移到失败的SQL语句后面的语句。

适用于进入和退出功能的通常规则。您可以将参数传递给由EXEC SQL WHENEVER调用的错误处理程序...DO ...指令，该函数可以返回一个值。

9.8.2.3 DO BREAK

实际的“break”语句放在你的程序中。在循环中使用此操作。当满足WHENEVER条件时，程序将退出其内部的循环。

9.8.2.4 DO继续

实际的“继续”语句放在您的程序中。在循环中使用此操作。当满足WHENEVER条件时，程序将继续进行其内部循环的下一次迭代。

9.8.2.5 GOTO label_name

您的程序分支到带标签的语句。标签名称可以是任意长度，但只有前31个字符是重要的。您的C编译器可能需要不同的最大长度。检查C编译器用户指南。

9.8.2.6停止

您的程序停止运行，并且回滚未提交的工作。

STOP生效只是在条件发生时生成一个exit()调用。小心。在断开与Oracle的连接之前，STOP操作不显示任何消息。

9.8.3 WHENEVER示例

如果你想要你的程序

  • 如果发生“未找到数据”条件，请转到close_cursor

  • 如果发生警告，请继续下一个语句

  • 如果发生错误，请转到error_handler

您必须在第一个可执行SQL语句之前编写以下WHENEVER指令： 

EXEC SQL WHENEVER NOT FOUND GOTO close_cursor; EXEC SQL WHENEVER SQLWARNING CONTINUE;执行SQL WHENEVER SQLERROR GOTO error_handler;

请注意程序如何检查SQLCA中的变量以确定操作过程。

9.8.4使用DO BREAK和DO CONTINUE

此示例说明了如何仅显示接收佣金的员工的员工姓名，薪水和佣金：

#include <sqlca.h> #include <stdio.h> main（）{char * uid =“scott / tiger”; struct {char ename [12];漂浮浮动通讯; }; / *捕获可能发生的任何连接错误。* / EXEC SQL WHENEVER SQLERROR GOTO哎呀; EXEC SQL CONNECT：uid; EXEC SQL DECLARE c CURSOR FOR SELECT ename，sal，comm FROM EMP ORDER BY ENAME ASC; EXEC SQL OPEN c; / *设置'BREAK'条件退出循环。* / EXEC SQL WHENEVER NOT FOUND DO BREAK; / * DO CONTINUE在发生错误时使循环在下一次迭代时开始。* / EXEC SQL WHENEVER SQLERROR DO CONTINUE; while（1）{EXEC SQL FETCH c INTO：emp; / * ORA-1405会导致'继续'发生。因此，只显示具有* / / *非NULL佣金的员工。* / printf（“％s％7.2f％9.2f \ n”，emp.ename，emp.sal，emp.comm）; } / *这个'CONTINUE'关闭'DO CONTINUE'，允许程序继续进行，如果发生任何进一步的错误，特别是CLOSE * / EXEC SQL WHENEVER SQLERROR CONTINUE; EXEC SQL CLOSE c;出口（EXIT_SUCCESS）; whoops：printf（“％。* s \ n”，sqlca.sqlerrm.sqlerrml，sqlca.sqlerrm.sqlerrmc）;出口（EXIT_FAILURE）; }

9.8.5 WHEEVER的范围

因为WHENEVER是一个声明性陈述，它的范围是位置的，而不是逻辑的。也就是说，它测试源文件中物理上跟随它的所有可执行SQL语句，而不是程序逻辑流。因此，在要测试的第一个可执行SQL语句之前编写WHENEVER指令。

WHENEVER指令保持有效，直到被另一个检查相同条件的WHENEVER指令取代。

在以下示例中，第一个WHENEVER SQLERROR指令被第二个取代，因此仅适用于CONNECT语句。WHENEVER SQLERROR指令同时适用于UPDATE和DROP语句，尽管控制的第一步 ，从第三步到流中的第二位。 

step1：EXEC SQL WHENEVER SQLERROR STOP; EXEC SQL CONNECT：用户名IDENTIFIED BY：密码; ...转到步骤3; step2：EXEC SQL WHENEVER SQLERROR CONTINUE; EXEC SQL UPDATE emp SET sal = sal * 1.10; ...step3：EXEC SQL DROP INDEX emp_index; ...

9.8.6 WHENEVER指南

以下指南将帮助您避免一些常见的陷阱。

9.8.6.1发表声明

通常，在程序中的第一个可执行SQL语句之前编写WHENEVER指令。这可确保捕获所有后续错误，因为WHENEVER指令对文件末尾保持有效。

9.8.6.2处理数据结束条件

在使用游标获取行时，您的程序应准备好处理数据结束条件。如果FETCH没有返回数据，程序应该退出fetch循环，如下所示： 

EXEC SQL WHENEVER NOT FOUND DO break; for（;;）{EXEC SQL FETCH ...EXEC SQL CLOSE my_cursor; ...

如果没有插入行，INSERT可以返回NOT FOUND。如果您不想捕获该条件，请在INSERT之前使用EXEC SQL WHENEVER NOT FOUND CONTINUE语句：

EXEC SQL WHENEVER NOT FOUND DO break; for（;;）{EXEC SQL FETCH ...EXEC SQL WHENEVER NOT FOUND CONTINUE; EXEC SQL INSERT INTO ...EXEC SQL CLOSE my_cursor; ...
9.8.6.3关于避免无限循环

如果WHENEVER SQLERROR GOTO指令分支到包含可执行SQL语句的错误处理例程，则如果SQL语句因错误而失败，则程序可能会进入无限循环。您可以通过在SQL语句之前编写WHENEVER SQLERROR CONTINUE来避免这种情况，如以下示例所示： 

执行SQL WHENEVER SQLERROR GOTO sql_error; ...sql_error：EXEC SQL WHENEVER SQLERROR CONTINUE; EXEC SQL ROLLBACK工作版; ...

如果没有WHENEVER SQLERROR CONTINUE语句，ROLLBACK错误将再次调用例程，启动无限循环。

不小心使用WHENEVER会导致问题。例如，如果DELETE语句设置NOT FOUND，则以下代码进入无限循环，因为没有行符合搜索条件：

/ *不当使用WHENEVER * / ...EXEC SQL WHENEVER NOT FOUND GOTO no_more; for（;;）{EXEC SQL FETCH emp_cursor INTO：emp_name，：salary; ...} no_more：EXEC SQL DELETE FROM emp WHERE empno =：emp_number; ...

下一个示例通过重置GOTO目标来正确处理NOT FOUND条件：

/ *正确使用WHENEVER * / ......EXEC SQL WHENEVER NOT FOUND GOTO no_more; for（;;）{EXEC SQL FETCH emp_cursor INTO：emp_name，：salary; ...} no_more：EXEC SQL WHENEVER NOT FOUND GOTO no_match; EXEC SQL DELETE FROM emp WHERE empno =：emp_number; ...no_match：......
9.8.6.4关于维护可寻址性

确保由WHENEVER GOTO指令控制的所有SQL语句都可以分支到GOTO标签。以下代码导致编译时错误，因为func1中的 labelA不在func2中的INSERT语句的范围内： 

func1（）{EXEC SQL WHENEVER SQLERROR GOTO labelA; EXEC SQL DELETE FROM emp WHERE deptno =：dept_number; ...labelA：......} func2（）{EXEC SQL INSERT INTO emp（job）VALUES（：job_title）; ...}

WHENEVER GOTO指令分支的标签必须与语句位于同一预编译文件中。

9.8.6.5关于错误后返回

如果程序在处理错误后必须返回，请使用DO routine_call操作。或者，您可以测试sqlcode的值，如以下示例所示：

...EXEC SQL UPDATE emp SET sal = sal * 1.10; if（sqlca.sqlcode <0）{/ *句柄错误* / EXEC SQL DROP INDEX emp_index;

只需确保没有WHENEVER GOTO或WHENEVER STOP指令处于活动状态。

9.9关于获取SQL语句的文本

在许多预编译器应用程序中，可以方便地知道正在处理的语句的文本，长度以及它包含的SQL命令（例如INSERT或SELECT）。对于使用动态SQL的应用程序尤其如此。

SQLStmtGetText()函数（旧名称： sqlgls()函数） - SQLLIB运行时库的部分 - 返回以下信息：

  • 最近解析的SQL语句的文本

  • 声明的有效长度

  • 语句中使用的SQL命令的功能代码

SQLStmtGetText()是线程安全的。发出静态SQL语句后，可以调用SQLStmtGetText() 。对于动态SQL方法1，在执行SQL语句后调用SQLStmtGetText() 。对于动态SQL方法2,3和4，只要语句已经PREPAREd，就可以调用SQLStmtGetText() 。

有关所有SQLLIB函数的新名称，另请参见“ SQLLIB公共函数的新名称 ” 。

SQLStmtGetText()的原型是

void SQLStmtGetText（dvoid * context，char * sqlstm，size_t * stmlen，size_t * sqlfc）;

context参数是运行时上下文。有关上下文的定义和使用，请参阅“ 上下文变量 ” 。

sqlstm参数是一个字符缓冲区，用于保存SQL语句的返回文本。您的程序必须静态声明缓冲区或为缓冲区动态分配内存。

stmlen参数是size_t变量。在调用SQLStmtGetText()之前，将此参数设置为sqlstm缓冲区的实际大小（以字节为单位） 。当SQLStmtGetText()返回时， sqlstm缓冲区包含SQL语句文本，空白填充到缓冲区的长度。stmlen参数返回返回的语句文本中的实际字节数，不计算空白填充。stmlen的最大值是特定于端口的，通常是最大整数大小。

9.10关于使用Oracle通信区域（ORACA）

SQLCA处理标准SQL通信ORACA处理Oracle通信。如果您需要有关运行时错误和状态更改的更多信息，请使用ORACA。它包含一组扩展的诊断工具。但是，使用ORACA是可选的，因为它会增加运行时开销。

除了帮助您诊断问题外，ORACA还允许您监视程序对Oracle资源（如SQL语句执行程序和游标缓存）的使用情况。

您的程序可以有多个ORACA。例如，它可能有一个全局ORACA和几个本地ORACA。访问本地ORACA受到程序范围的限制。Oracle仅将信息返回到范围内的ORACA。

9.10.1关于声明ORACA

要声明ORACA，请使用INCLUDE语句或#include预处理程序指令将其复制到程序中，如下所示：

EXEC SQL INCLUDE ORACA;

要么

#include <oraca.h>

如果您的ORACA必须是extern存储类，请在程序中定义ORACA_STORAGE_CLASS，如下所示：

#define ORACA_STORAGE_CLASS extern

如果程序使用Declare Section，则必须在其外部定义ORACA。

9.10.2关于启用ORACA

要启用ORACA，必须在命令行上指定ORACA选项

ORACA = YES

或者内联

EXEC ORACLE选项（ORACA = YES）;

然后，您必须通过在ORACA中设置标志来选择适当的运行时选项。

9.10.3 ORACA内容

ORACA包含选项设置，系统统计信息和扩展诊断，例如

  • SQL语句文本（可以指定保存文本的时间）

  • 发生错误的文件的名称（在使用子例程时很有用）

  • 错误在文件中的位置

  • 游标缓存错误和统计信息

ORACA包括几个选项标志。通过为这些标志分配非零值来设置这些标志，可以提供这些标志

9.10.5 ORACA的结构

本节介绍ORACA的结构，其组件以及它们可以存储的值。

9.10.5.1 oracaid

此字符串组件初始化为“ORACA”以标识Oracle通信区域。

9.10.5.2 oracabc

此整数组件保存ORACA数据结构的长度（以字节为单位）。

9.10.5.3 oracchf

如果设置了主DEBUG标志（ oradbgf） ，则此标志将启用游标缓存统计信息的收集，并允许您在每个游标操作之前检查游标缓存的一致性。

Oracle运行时库执行一致性检查，并可能发出错误消息，这些消息列在手册Oracle数据库错误消息中 。它们像Oracle错误消息一样返回到SQLCA。

此标志具有以下设置：

  • 禁用缓存一致性检查（默认值）。

  • 启用缓存一致性检查。

9.10.5.4 oradbgf

此主标志允许您选择所有DEBUG选项。它具有以下设置：

禁用所有DEBUG操作（默认值）。

启用所有DEBUG操作。

9.10.5.5 orahchf

如果设置了主DEBUG标志（ oradbgf ），则此标志告诉Oracle运行时库每次预编译器动态分配或释放内存时都要检查堆的一致性。这对于检测扰乱内存的程序错误很有用。

必须在发出CONNECT命令之前设置此标志，并且一旦设置，则无法清除;后续更改请求将被忽略。它具有以下设置：

  • 禁用堆一致性检查（默认值）。

  • 启用堆一致性检查。

9.10.5.6 orastxtf

此标志允许您指定何时保存当前SQL语句的文本。它具有以下设置：

  • 永远不要保存SQL语句文本（默认值）。

  • 仅在SQLERROR上保存SQL语句文本。

  • 将SQL语句文本保存在SQLERROR或SQLWARNING上。

  • 始终保存SQL语句文本。

SQL语句文本保存在名为orastxt的ORACA嵌入式结构中。
