8.1 Oracle全球化开发工具包概述

即使对于经验最丰富的开发人员来说，设计和开发全球化应用程序也是一项艰巨的任务。这通常是由于缺乏知识以及全球化概念和API的复杂性造成的。使用Oracle数据库编写应用程序的应用程序开发人员需要了解数据库的全球化支持体系结构，包括不同字符集，区域，语言和语言排序定义的属性。他们还需要了解其中间层编程环境的全球化功能，并了解它如何与数据库的语言环境模型进行交互和同步。最后，为了开发全球化的Internet应用程序，他们需要设计和编写能够同时支持在不同操作系统上运行的多个客户端的代码，这些客户端具有不同的字符集和区域设置要求。

神谕全球化开发工具包（GDK）简化了开发过程，降低了开发用于支持全球环境的Internet应用程序的成本。GDK包括用于Java和PL / SQL的全面编程API，代码示例和文档，可解决创建全局应用程序时遇到的许多设计，开发和部署问题。

GDK主要由两部分组成：GDK for Java和GDK for PL / SQL。 GDK for Java为Java应用程序提供全球化支持。GDK for PL / SQL为PL / SQL编程环境提供了全球化支持。GDK for Java和GDK for PL / SQL提供的功能并不相同。

8.2设计全球互联网应用

根据您的全球化和业务需求，有两种用于部署全局Web站点或全局Internet应用程序的体系结构模型。要部署哪种模型会影响Internet应用程序的开发方式以及应用程序服务器在中间层中的配置方式。这两个模型是：

  • 多个实例单语互联网应用

    仅支持单个二进制文件中的一个区域设置的Internet应用程序被归类为单语言应用程序。语言环境是指国家语言和语言所在的区域。例如，美国和英国的主要语言是英语。但是，这两个地区的日期格式有不同的货币和不同的约定。因此，美国和英国被认为是两个不同的地方。

    此级别的全球化支持适用于希望为每个应用程序实例支持一个区域设置的客户。用户需要具有不同的入口点才能访问不同语言环境的应用程序。仅当支持的语言环境数量较少时，此模型才可管理。

  • 单个实例多语言应用

    在单个二进制文件中同时支持多个语言环境的Internet应用程序被归类为多语言应用程序。此级别的全球化支持适用于希望同时支持Internet应用程序中的多个区域设置的客户。具有不同区域设置首选项的用户使用相同的入口点来访问应用程序。

    使用单语模型开发应用程序与使用多语言模型开发应用程序有很大不同。全球化开发工具包由库组成，可以使用任一架构模型协助开发全局应用程序。

每个应用程序服务器都配置为它所服务的语言环境。此部署模型假定Internet应用程序的一个实例在与中间层中的应用程序相同的语言环境中运行。

Internet应用程序以用于语言环境的本机编码访问后端数据库。以下是部署单语互联网应用程序的优点：

  • 各个区域设置的支持分为不同的服务器，以便可以在不同位置独立支持多个区域设置，并且可以相应地分配工作负载。例如，客户可能希望首先支持西欧语言环境，然后再支持日语（日本）等亚洲语言环境。

以下是部署单语互联网应用程序的缺点：

  • 需要额外的努力来维护和管理不同语言环境的多个服务器。不同的应用程序服务器需要不同的配置。

  • 所需的最小应用程序服务器数取决于应用程序支持的语言环境数，而不管站点流量是否达到应用程序服务器提供的容量。

  • 应用程序服务器的负载平衡仅限于同一语言环境的应用程序服务器组。

  • 应用程序服务器的多种配置需要更多的人力和机器QA资源。在不同语言环境中运行的Internet应用程序必须在相应的应用程序服

  • 它不是为支持多语言内容而设计的。例如，在此模型中无法轻松支持包含日语和阿拉伯数据的网页。

随着越来越多的语言环境得到支持，缺点很快就会超过优势。由于单语部署模型的限制和维护开销，此部署体系结构适用于仅支持一个或两个语言环境的应用程序。

要在单个应用程序实例中支持多个语言环境，应用程序可能需要执行以下操作：

  • 通过在语言环境的语言和文化约定中构造HTML页面，动态检测用户的语言环境并适应语言环境

  • 以Unicode处理字符数据，以便支持任何语言的数据。用户可以输入字符数据，也可以从后端数据库中检索字符数据。

  • 动态确定要用于HTML页面的HTML页面编码（或字符集），并将内容从Unicode转换为页面编码，反之亦然。

以下是部署多语言Internet应用程序的主要优点：

  • 对所有应用程序服务器使用单个应用程序服务器配置可简化部署配置，从而降低维护成本。

  • 性能调整和容量规划不依赖于Web站点支持的语言环境数。

  • 引入其他语言环境相对容易。新语言环境不需要额外的机器。

  • 跨不同语言环境测试应用程序可以在单个测试环境中完成。

  • 此模型可以在应用程序的同一实例中支持多语言内容。例如，在此模型中可以轻松支持包含日语，中文，英语和阿拉伯语数据的网页。

部署多语言Internet应用程序的缺点是它在应用程序开发期间需要额外的编码来处理动态区域设置检测和Unicode，当仅需要支持一种或两种语言时这是昂贵的。

当网站支持多种语言环境时，部署多语言Internet应用程序比部署单语应用程序更合适。

8.3开发全球互联网应用

构建支持不同语言环境的Internet应用程序需要良好的开发实践。

对于多语言Internet应用程序，应用程序本身必须了解用户的语言环境，并能够向用户显示适合语言环境的内容。无论客户端的语言环境如何，客户端都必须能够与应用程序服务器通信。然后，应用程序服务器与数据库服务器通信，交换数据，同时保持不同语言环境和字符集设置的首选项。开发多语言Internet应用程序时的一个主要考虑因素是能够根据用户的首选语言环境动态检测，缓存和提供适当的内容。

对于单语互联网应用程序，用户的语言环境始终是固定的，通常遵循运行时环境的默认语言环境。因此，语言环境配置更加简单。

8.3.1区域确定

要对区域设置感知或区域设置敏感，Internet应用程序必须能够确定用户的首选区域设置。

单语应用程序始终为具有相同语言环境的用户提供服务，并且该语言环境应该等效于相应编程环境的默认运行时语言环境。

多语言应用程序可以通过三种方式动态确定用户区域设置。每种方法都有优点和缺点，但它们可以在应用中一起使用以相互补充。可以通过以下方式确定用户区域设置：

  • 基于来自LDAP目录服务器的用户配置文件信息，例如Oracle Internet Directory或存储在数据库中的其他用户配置文件表

    用户配置文件的架构应包括首选的区域设置属性，以指示用户的区域设置。如果用户之前尚未登录，则这种确定区域设置用户的方法不起作用。

  • 基于浏览器的默认语言环境

    从浏览器获取默认的ISO区域设置。浏览器的默认ISO语言环境通过每个HTTP请求中的Accept-Language HTTP标头发送。如果Accept-Language标头为NULL ，则所需的语言环境应默认为英语。这种方法的缺点是Accept-Language标头可能不是用户区域设置的可靠信息源。

  • 根据用户选择

    允许用户从列表框或菜单中选择区域设置，并将应用程序区域设置切换为所选的区域设置。

8.3.2区域设置意识

要对区域设置感知或区域设置敏感，Internet应用程序需要确定用户的区域设置。确定用户的区域设置后，应用程序应：

  • 使用语言环境的语言构造HTML内容

  • 使用语言环境隐含的文化习俗

区域设置敏感函数（如日期，时间和货币格式）内置于各种编程环境中，例如Java和PL / SQL。应用程序可以根据用户区域设置的文化约定使用它们来格式化HTML页面。在不同的编程环境中，区域设置的表示方式不同。例如，法语（加拿大）语言环境在不同的环境中表示如下：

  • 在ISO标准中，它由fr-CA表示，其中fr是ISO 639标准中定义的语言代码， CA是ISO 3166标准中定义的国家代码。

  • 在Java中，它表示为使用fr构建的Java语言环境对象，法语的ISO语言代码，作为语言和CA ，加拿大的ISO国家/地区代码，作为国家/地区。Java语言环境名称是fr_CA 。

  • 在PL / SQL和SQL中，它主要由NLS_LANGUAGE和NLS_TERRITORY会话参数表示，其中NLS_LANGUAGE参数的值等于CANADIAN FRENCH并且NLS_TERRITORY参数的值等于CANADA 。

如果为多个编程环境编写应用程序，则必须在环境之间同步语言环境。例如，调用PL / SQL过程的Java应用程序应将Java语言环境映射到相应的NLS_LANGUAGE和NLS_TERRITORY值，并在调用PL / SQL过程之前更改参数值以匹配用户的语言环境。

全球化开发工具包for Java提供了一组Java类，以确保与Oracle数据库的区域设置敏感行为的一致性。

8.3.3本地化内容

对于支持多语言环境的应用程序，它必须能够以首选语言和用户的语言环境约定呈现内容。硬编码的用户界面文本必须首先从应用程序与任何图像文件一起外部化，以便将它们翻译成应用程序支持的不同语言。然后必须在单独的目录中暂存转换文件，并且应用程序必须能够根据用户区域设置找到相关内容。还可能需要特殊的应用程序处理来支持回退机制，因此如果用户首选的语言环境不可用，则会显示下一个最合适的内容。例如，如果加拿大法语内容不可用，则应用程序可能适合切换到法语文件。

GDK for Java补充了J2EE中现有的全球化功能。尽管J2EE平台已经为构建全球化应用程序提供了坚实的基础，但其全球化功能和行为可能与Oracle的功能完全不同。GDK for Java提供了中间层Java应用程序和数据库服务器之间的区域设置敏感行为的同步。

8.6.5在GDK中定义支持的应用程序区域设置

应用程序需要支持的语言环境数和语言环境名称基于应用程序的业务需求。应用程序支持的语言环境的名称将在应用程序配置文件中注册。以下示例显示应用程序配置文件的应用程序区域设置部分。它表示该应用程序支持德语（ de ），日语（ ja ）和英语（ en-US ），英语被定义为默认的回退应用程序语言环境。请注意，区域设置名称基于IANA约定。 

<application-locales> <locale> de </ locale> <locale> ja </ locale> <locale default =“yes”> en-US </ locale> </ application-locales>

当GDK框架检测到用户区域设置时，它会验证返回的区域设置是否是应用程序配置文件中支持的区域设置之一。验证算法如下：

  1. 从应用程序配置文件中检索支持的应用程序区域设置列表。
  2. 检查检测到的区域设置是否包含在列表中。如果它包含在列表中，则使用此语言环境作为当前客户端的语言环境。
  3. 如果检测到的语言环境中存在变体，则删除变体并检查生成的语言环境是否在列表中。例如，Java语言环境de_DE_EURO具有EURO变体。删除变体，以便生成的语言环境为de_DE 。
  4. 如果区域设置包含国家/地区代码，请删除国家/地区代码并检查生成的区域设置是否在列表中。例如，Java语言环境de_DE的国家代码为DE 。删除国家/地区代码，以便生成的区域设置为de 。
  5. 如果检测到的语言环境与列表中的任何语言环境都不匹配，则使用应用程序配置文件中定义的默认语言环境作为客户端语言环境。

通过执行步骤3和4，应用程序可以支持具有相同语言要求但具有与应用程序配置文件中定义的语言环境设置不同的语言环境设置的用户。例如，GDK可以支持de-AT （德语的奥地利变体）， de-CH （德语的瑞士变体）和de-LU （德语的卢森堡变体）语言环境。

GDK框架中的语言环境回退检测类似于Java Resource Bundle，除了它不受Java VM的默认语言环境的影响。发生此异常是因为可以在GDK区域设置回退操作期间使用应用程序默认区域设置。

如果应用程序配置文件中省略了application-locales部分，则GDK假定应用程序支持可从OraLocaleInfo.getCommonLocales方法返回的公共语言环境。

8.6.6处理GDK框架中的非ASCII输入和输出

该 HTML页面的字符集（或字符编码）是浏览器和Internet应用程序的一个非常重要的信息。浏览器需要解释此信息，以便它可以使用正确的字体和字符集映射表来显示页面。Internet应用程序需要知道，以便他们可以根据指定的编码安全地处理HTML表单中的输入数据。

页面编码可以翻译为用于Internet应用程序所服务的语言环境的字符集。

为了在不使用GDK框架的情况下正确指定HTML页面的页面编码，Internet应用程序必须：

  1. 确定给定语言环境的所需页面输入数据字符集编码。

  2. 为每个HTTP请求和HTTP响应指定相应的编码名称。

使用GDK框架的应用程序可以忽略这些步骤。不需要更改应用程序代码。字符集信息在GDK应用程序配置文件中指定。在运行时，GDK自动设置请求和响应对象的字符集。GDK框架不支持传入字符集与传出字符集的字符集不同的情况。

GDK应用程序框架支持以下方案来设置HTML页面的字符集：

  • 单个本地字符集专用于整个应用程序。这适用于单语互联网应用程序。根据字符集的属性，它可能支持多种语言。例如，大多数西欧语言可以由ISO-8859-1提供。

  • Unicode UTF-8用于所有内容，无论语言如何。这适用于使用Unicode进行部署的多语言应用程序。

  • 使用每种语言的本机字符集。例如，英语内容用ISO-8859-1表示，日语内容用Shift_JIS表示。这适用于使用每个语言环境的默认字符集映射的多语言Internet应用程序。这对于需要根据用户区域设置支持不同字符集的应用程序非常有用。例如，对于缺少Unicode字体或无法完全支持Unicode的Internet浏览器的移动应用程序，必须为每个请求确定字符集。

字符集信息在GDK应用程序配置文件中指定。以下是将UTF-8设置为所有应用程序页面的字符集的示例。

<页字符集> UTF-8 </页字符集>

<％Locale user_locale = request.getLocale（）; ResourceBundle rb = ResourceBundle.getBundle（“resource”，user_locale）; ％> <％= rb.getString（“欢迎”）％>

GDK框架简化了从资源包中检索文本字符串的过程。Localizer.getMessage()是资源包的包装器。 

<％Localizer.getMessage（“Welcome”）％>

<message-bundles> <resource-bundle name =“default”> resource </ resource-bundle> </ message-bundles>

此配置文件片段声明了一个默认资源包，其翻译内容位于“资源”Java包类中。可以在配置文件中指定多个资源包。要访问非默认包，请在getMessage方法中指定name参数。消息包机制使用OraResourceBundle GDK类来实现它。此类在Java行为之上提供特殊的语言环境回退行为。规则如下：

  • 如果给定的语言环境与可用资源包中的语言环境完全匹配，则将使用它。

  • 如果找不到新加坡华人资源包（ zh_SG ），它将回退到中国的中文资源包（ zh_CN ），用于简体中文翻译。

  • 如果找不到香港中文（ zh_HK ）的资源包，它将回退到台湾中文（ zh_TW ）的繁体中文翻译资源包。

  • 如果找不到澳门中文（ zh_MO ）的资源包，它将回退到台湾中文（ zh_TW ）的繁体中文翻译资源包。

8.7 GDK Java API

Java中的全球化特性和行为与Oracle数据库中提供的不同。例如，J2SE支持一组与Oracle数据库中的区域设置和字符集不同的区域设置和字符集。当用户的应用程序包含基于两种不同约定格式的数据时，这种不一致可能会让用户感到困惑。例如，使用Oracle约定格式化从数据库检索的日期，例如数字和日期格式以及语言排序顺序。但是，静态应用程序数据通常使用Java语言环境约定进行格式化。Java中的全球化功能也可能不同，具体取决于运行应用程序的JDK的版本。

在Oracle Database 10g之前，当需要应用程序合并Oracle全球化功能时，它必须与数据库建立连接并发出SQL语句。此类操作使应用程序变得复杂并生成与数据库的更多网络连接。

在Oracle数据库10g及更高版本中，GDK Java API将全球化功能扩展到中间层。通过使应用程序能够执行全局化逻辑，例如Oracle日期和数字格式以及中间层的语言排序，GDK Java API使开发人员能够消除数据库中昂贵的编程逻辑。GDK Java API还为XQuery提供标准合规性。这通过减少数据库处理负载以及减少应用程序层和后端之间不必要的网络流量来提高整体应用程序性能。

GDK Java API还提供高级全球化功能，例如语言和字符集检测，以及区域或语言的常见区域设置数据的枚举（例如，加拿大支持的所有时区）。大多数编程平台都不提供这些功能。如果没有GDK Java API，开发人员必须编写业务逻辑来处理应用程序内的这些进程。

GDK Java API的主要功能如下：

  • GDK中的Oracle语言环境信息

  • GDK中的Oracle区域设置映射

  • GDK中的Oracle字符集转换

  • GDK中的Oracle日期，数量和货币格式

  • GDK中的Oracle二进制和语言排序

  • GDK中的Oracle语言和字符集检测

  • 在GDK中Oracle转换的区域设置和时区名称

  • 将GDK与电子邮件程序一起使用

如果请求的语言环境的本地化内容不可用，则GDK框架可以通过将其映射到最接近的转换语言环境来触发语言环境回退机制。默认情况下，后备选项已关闭。这可以通过指定fallback="yes"来打开。

例如，假设应用程序仅支持以下转换： en ， de和ja ，并且en是应用程序的缺省语言环境。如果当前的应用程序区域设置是de-US ，那么它将回退到de 。如果用户选择zh-TW作为其应用程序区域设置，则它将回退到en 。

如果支持的应用程序区域设置的数量大于转换区域设置的数量，则通常需要回退机制。如果多个区域设置共享一个转换，则通常会发生一个例子是西班牙语。该应用程序可能需要支持多个西班牙语国家而不仅仅是西班牙，并提供一组翻译文件。

可以通过将name属性分配给非默认URL重写规则来指定多个URL重写规则。要使用非默认URL重写规则，必须将该名称作为重写URL方法的参数传递。例如：

<img src =“<％= ServletHelper.rewriteURL（”images / welcome.gif“，request）％>”> <img src =“<％= ServletHelper.rewriteURL（”US.gif“，”flag“，request） ％>“>
