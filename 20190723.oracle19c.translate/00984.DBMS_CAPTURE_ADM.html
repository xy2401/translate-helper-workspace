33 DBMS_CAPTURE_ADM

DBMS_CAPTURE_ADM程序包是一组Oracle Replication程序包之一，它提供了用于启动，停止和配置捕获进程的子程序。捕获的更改的来源是重做日志，捕获的更改的存储库是队列。

33.1 DBMS_CAPTURE_ADM概述

该软件包提供了启动，停止和配置捕获过程或同步捕获的接口。该软件包包括用于为实例化准备数据库对象的子程序。

捕获进程可以在多租户容器数据库（CDB）中的XStream配置中使用。CDB是Oracle数据库，包括零个，一个或多个用户创建的可插拔数据库（PDB）。

33.2 DBMS_CAPTURE_ADM安全模型

可以通过两种方式之一控制DBMS_CAPTURE_ADM安全性。

  • 将此程序包上的EXECUTE授予选定的用户或角色。

  • 向选定的用户或角色授予EXECUTE_CATALOG_ROLE 。

如果程序包中的子程序是从存储过程中运行的，则必须直接为程序包授予运行子程序的用户EXECUTE特权。它不能通过角色授予。

使用DBMS_CAPTURE_ADM程序包管理Oracle Replication配置时，需要为用户授予Oracle Replication管理员的权限。

当DBMS_CAPTURE_ADM包用于管理XStream配置时，它要求为用户授予XStream管理员的权限。

注意：

  • 必须为用户授予使用此程序包中的子程序执行某些管理任务的其他权限，例如设置捕获用户。如果子程序需要其他权限，则权限将在描述子程序的部分中记录。

  • 使用XStream需要购买Oracle GoldenGate产品的许可证。请参见Oracle Database XStream Guide 。

33.3.1 ABORT_GLOBAL_INSTANTIATION过程

此过程可逆转运行PREPARE_GLOBAL_INSTANTIATION ， PREPARE_SCHEMA_INSTANTIATION和PREPARE_TABLE_INSTANTIATION过程的影响。

具体而言，此过程执行以下操作：

  • 删除与数据库，模式和表实例化相关的数据字典信息

  • 删除PREPARE_GLOBAL_INSTANTIATION ， PREPARE_SCHEMA_INSTANTIATION和PREPARE_TABLE_INSTANTIATION过程启用的任何补充日志记录

句法

DBMS_CAPTURE_ADM.ABORT_GLOBAL_INSTANTIATION（容器IN VARCHAR2 DEFAULT' CURRENT'）;

参数

表33-2 ABORT_GLOBAL_INSTANTIATION过程参数

参数 描述

容器

CURRENT ， ALL或pdb_name 。

如果指定了CURRENT ，则此过程将删除当前容器的补充日志记录。

如果指定了ALL ，则此过程将删除当前CDB中所有容器的补充日志记录。

如果指定了pdb_name ，则此过程将删除指定PDB的补充日志记录。

ALL和pdb_name仅在从根调用过程时才有效。

33.3.5 ALTER_CAPTURE程序

此过程会改变捕获过程。

句法

DBMS_CAPTURE_ADM.ALTER_CAPTURE（capture_name IN VARCHAR2，rule_set_name IN VARCHAR2 DEFAULT NULL，remove_rule_set IN BOOLEAN DEFAULT FALSE，start_scn IN NUMBER DEFAULT NULL，use_database_link IN BOOLEAN DEFAULT NULL，first_scn IN NUMBER DEFAULT NULL，negative_rule_set_name IN VARCHAR2 DEFAULT NULL，remove_negative_rule_set IN BOOLEAN DEFAULT FALSE， capture_user IN VARCHAR2 DEFAULT NULL，checkpoint_retention_time IN NUMBER DEFAULT NULL，start_time IN TIMESTAMP DEFAULT NULL，oldest_scn IN NUMBER DEFAULT NULL）;

参数

表33-6 ALTER_CAPTURE过程参数

参数 描述

capture_name

要更改的捕获过程的名称。您必须指定现有的捕获进程名称。不要指定所有者。

rule_set_name

捕获过程的正规则集的名称。正规则集包含指示捕获过程捕获更改的规则。

要更改捕获过程的正规则集，请在[ schema_name .] rule_set_name形式中指定现有规则集。例如，要在名为job_capture_rules的hr模式中指定正规则集，请输入hr.job_capture_rules 。如果未指定架构，则当前用户是默认用户。

如果指定的规则集不存在，则返回错误。您可以使用DBMS_RULE_ADM包创建规则集并向其添加规则。

如果指定NULL并且remove_rule_set参数设置为FALSE ，则该过程将保留任何现有的正规则集。如果指定NULL并且remove_rule_set参数设置为TRUE ，则该过程将删除任何现有的正规则集。

remove_rule_set

如果为TRUE ，则该过程将删除指定捕获进程的正规则集。如果删除捕获进程的正规则集，并且捕获进程没有负规则集，则捕获进程将捕获对数据库中所有对象的所有受支持的更改，不包括SYS和SYSTEM模式中的数据库对象。

如果删除捕获进程的正规则集，并且捕获进程具有否定规则集，则捕获进程将捕获负规则集未放弃的所有受支持的更改。

如果为FALSE ，则该过程将保留指定捕获进程的正规则集。

如果rule_set_name参数为非NULL ，则确保将此参数设置为FALSE 。

start_scn

捕获进程开始捕获更改的数据库的有效SCN。SCN值必须大于或等于捕获过程的第一个SCN。此外，必须在重置其启动SCN之前停止捕获过程。

如果指定了无效的SCN或启用了捕获过程，则会返回错误。

use_database_link

如果为TRUE ，则下游数据库中的捕获进程使用指向源数据库的数据库链接，以用于与捕获进程相关的管理目的。如果您希望当前未使用数据库链接的捕获进程开始使用数据库链接，请指定TRUE 。在这种情况下，下游数据库中必须存在与源数据库的全局名称同名的数据库链接。

如果为FALSE ，则捕获进程正在源数据库上运行，或者下游数据库中的捕获进程未使用到源数据库的数据库链接。如果您希望当前使用数据库链接的捕获进程停止使用数据库链接，请指定FALSE 。在这种情况下，在添加或更改与这些对象相关的捕获流程规则时，必须手动准备源数据库对象以进行实例化。

如果为NULL ，则不更改捕获进程的此参数的当前值。

first_scn

重做日志中捕获进程可捕获的最低SCN更改。如果为捕获过程指定新的第一个SCN，则指定的第一个SCN必须满足以下要求：

  • 它必须大于捕获过程的当前第一个SCN。

  • 它必须小于或等于捕获过程的当前应用SCN。但是，如果捕获过程的当前应用SCN为零，则此要求不适用。

  • 它必须小于或等于捕获过程所需的检查点SCN。

如果指定的SCN不满足前三个要求，则返回错误。有关确定满足所有这些条件的SCN值的信息，请参阅“ 使用说明 ” 。

当第一个SCN被修改时，捕获过程从其LogMiner数据字典中清除信息，该信息是在较早的SCN上重新启动它所需的。有关LogMiner数据字典的详细信息，请参阅BUILD过程 。

如果指定的第一SCN高于捕获过程的当前开始SCN，则开始SCN自动设置为第一SCN的新值。

negative_rule_set_name

为捕获过程设置的否定规则的名称。否定规则集包含指示捕获过程放弃更改的规则。

要更改捕获过程的否定规则集，请在[ schema_name .] rule_set_name形式中指定现有规则集。例如，要在名为neg_capture_rules的hr模式中指定否定规则集，请输入hr.neg_capture_rules 。如果未指定架构，则当前用户是默认用户。

如果指定的规则集不存在，则返回错误。您可以使用DBMS_RULE_ADM包创建规则集并向其添加规则。

如果指定NULL并且remove_negative_rule_set参数设置为FALSE ，则该过程将保留任何现有的否定规则集。如果指定NULL并且remove_negative_rule_set参数设置为TRUE ，则该过程将删除任何现有的否定规则集。

如果为捕获过程指定了正规则集和负规则集，则始终首先评估否定规则集。

remove_negative_rule_set

如果为TRUE ，则该过程将删除指定捕获进程的否定规则集。如果删除捕获进程的否定规则集，并且捕获进程没有正规则集，则捕获进程将捕获对数据库中所有对象的所有受支持的更改，不包括SYS和SYSTEM模式中的数据库对象。

如果删除捕获进程的否定规则集，并且捕获进程存在正规则集，则捕获进程将捕获正规则集未放弃的所有更改。

如果为FALSE ，则该过程将保留指定捕获进程的否定规则集。

如果negative_rule_set_name参数为非NULL ，则确保将此参数设置为FALSE 。

capture_user

捕获进程在其安全域中捕获满足其规则集的更改的用户，并运行为捕获进程规则配置的基于规则的自定义转换。如果为NULL ，则不更改捕获用户。

要更改捕获用户，必须为调用ALTER_CAPTURE过程的用户授予DBA角色。只有SYS用户可以将capture_user设置为SYS 。

如果更改捕获用户，则此过程将授予捕获进程使用的队列的新捕获用户入队权限，并将用户配置为队列的安全队列用户。此外，请确保捕获用户具有以下权限：

  • 对捕获过程使用的规则集的EXECUTE特权

  • 对规则集中使用的所有基于规则的转换函数的EXECUTE特权

这些权限可以直接授予捕获用户，也可以通过角色授予。

此外，必须为捕获用户授予对所有包（包括Oracle提供的包）的EXECUTE特权，这些包在捕获过程运行的基于规则的转换中调用。必须直接向捕获用户授予这些权限。他们不能通过角色授予。

更改此参数的值时，将自动停止并重新启动捕获过程。

注意：如果使用DROP USER . . .删除捕获进程的捕获USER . . .CASCADE ，然后捕获过程也会自动删除。

checkpoint_retention_time

捕获进程在自动清除检查点之前保留检查点的天数，或者如果不应自动清除检查点， DBMS_CAPTURE_ADM.INFINITE 。如果为NULL ，则不更改检查点保留时间。

如果指定了数字，则捕获过程会在检查点生成后的指定天数内清除检查点。可以使用十进制值指定部分日期。例如， .25指定6小时。

清除检查点时，将清除与检查点对应的归档重做日志文件的LogMiner数据字典信息，并将捕获进程的first_scn重置为与下一个归档重做日志文件中的第一个更改对应的SCN值。

开始时间

捕获进程开始捕获更改的有效时间。必须在重置其开始时间之前停止捕获过程。

如果指定了无效时间或启用了捕获过程，则会返回错误。

start_scn和start_time参数是互斥的。

oldest_scn

目前正在处理的最早的SCN交易。

使用说明

如果要更改捕获过程的第一个SCN，则指定的值必须满足first_scn参数的描述中的first_scn 。

例子

以下查询确定数据库中每个捕获进程的当前第一个SCN，应用的SCN和所需的检查点SCN：

SELECT CAPTURE_NAME，FIRST_SCN，APPLIED_SCN，REQUIRED_CHECKPOINT_SCN来自DBA_CAPTURE;

捕获用户需求

capture_user参数指定捕获满足捕获过程规则集的更改的用户。此用户必须具有捕获更改所需的权限。此过程为捕获进程使用的队列授予捕获用户入队权限，并将用户配置为队列的安全队列用户。

此外，请确保捕获用户具有以下权限：

  • 对捕获过程使用的规则集的EXECUTE特权

  • 对正规则集中使用的所有基于规则的转换函数的EXECUTE特权

这些权限可以直接授予捕获用户，也可以通过角色授予。

注意：

  • 捕获用户不需要数据库对象的特权来捕获对数据库对象的更改。捕获过程可以将这些更改传递给基于规则的转换函数。因此，请确保在配置捕获过程时考虑安全隐患。

  • 在数据库中创建第一个捕获过程可能需要一些时间，因为在创建过程中数据字典会重复。
