28.1 Oracle Scheduler概述

Oracle数据库包括Oracle Scheduler，这是一个企业作业调度程序，可帮助您简化数百甚至数千个任务的调度。Oracle Scheduler（调度程序）由DBMS_SCHEDULER PL / SQL包中的过程和函数实现。

通过Scheduler，您可以控制在企业环境中执行各种计算任务的时间和位置。Scheduler可帮助您有效地管理和规划这些任务。通过确保无需人工干预即可完成许多常规计算任务，您可以降低运营成本，实施更可靠的例程，最大限度地减少人为错误，并缩短所需的时间窗口。

Scheduler提供复杂，灵活的企业调度功能，您可以使用它来：

  • 运行数据库程序单元

    您可以在本地数据库或一个或多个远程Oracle数据库上运行程序单元，即PL / SQL匿名块，PL / SQL存储过程和Java存储过程。

  • 运行外部可执行文件（数据库外部的可执行文件）

    您可以在本地系统或一个或多个远程系统上运行外部可执行文件 ，例如应用程序，shell脚本和批处理文件。远程系统不需要安装Oracle数据库;他们只需要一个Scheduler代理。调度程序代理可用于Oracle数据库和其他一些平台支持的所有平台。

  • 使用以下方法计划作业执行：

    • 基于时间的调度

      您可以安排作业在特定日期和时间运行，一次或重复。您可以定义复杂的重复间隔，例如“每个星期一和星期四凌晨3:00，公众假期除外”或“每个业务季度的最后一个星期三”。有关详细信息，请参阅“ 创建，运行和管理作业 ” 。

    • 基于事件的调度

      您可以启动作业以响应系统或业务事件。您的应用程序可以检测事件，然后发送调度程序信号。根据发送的信号类型，Scheduler启动特定作业。基于事件的调度的示例包括当文件到达系统时，当库存低于预定水平时，或者当事务失败时启动作业。从Oracle Database 11g第2版（11.2）开始，称为文件监视器的Scheduler对象简化了配置作业的任务，该作业在文件到达本地或远程系统时启动。有关详细信息，请参阅“ 使用事件启动作业 ” 。

    • 依赖性调度

      您可以将计划程序设置为根据一个或多个先前任务的结果运行任务。您可以定义包含分支和嵌套链的复杂依赖关系链。有关详细信息，请参阅“ 创建和管理作业链 ” 。

  • 根据业务需求确定工作的优先级。

    调度程序提供对竞争作业之间资源分配的控制，从而使作业处理与您的业务需求保持一致。这可以通过以下方式完成：

    • 按作业类控制资源

      您可以将具有共同特征和行为的作业分组为称为作业类的较大实体。您可以通过控制分配给每个类的资源来确定类之间的优先级。因此，您可以确保关键作业具有优先级和足够的资源来完成。例如，对于加载数据仓库的关键项目，您可以将所有数据仓库作业组合到一个类中，并通过为其分配大量可用资源来使其优先于其他作业。您还可以为作业类中的作业分配相对优先级。

    • 根据计划控制作业优先级

      您可以根据计划更改工作优先级。由于您对关键作业的定义可能会随着时间的推移而发生变化，因此调度程序还允许您在该时间范围内更改作业之间的优先级。例如，用于加载数据仓库的提取，传输和加载（ETL）作业在非高峰时段可能很关键，但在高峰时段则不然。此外，必须在业务季度结束时运行的作业可能需要优先于ETL作业。在这些情况下，您可以通过更改分配给每个类的资源来更改作业类之间的优先级。有关详细信息，请参阅“ 创建作业类 ”和“ 创建Windows ” 。

  • 管理和监控工作

    您可以管理和监控作业从创建到完成的多个状态。Scheduler会记录活动，以便您可以使用Oracle Enterprise Manager Cloud Control或SQL查询视图，轻松跟踪作业状态和作业的上次运行时间等信息。这些视图提供有关作业及其执行的有价值信息，可帮助您安排和更好地管理您的工作。例如，DBA可以轻松跟踪特定用户失败的所有作业。请参阅“ 调度程序数据字典视图 ” 。

    创建多目标作业（在一个数据库中定义但在多个远程主机上运行的作业）时，可以单独监视每个目标的作业状态，也可以监视父作业的整体状态。

    对于高级作业监视，您的应用程序可以订阅调度程序在事件队列中提供的作业状态更改通知。调度程序还可以在作业更改状态时发送电子邮件通知。

    请参阅“ 监视和管理计划程序 ” 。

  • 在集群环境中执行和管理作业

    集群是一组数据库实例，它们协作执行相同的任务。Oracle Real Application Clusters（Oracle RAC）提供可扩展性和可靠性，而无需对应用程序进行任何更改。Scheduler完全支持在这样的集群环境中执行作业。要平衡系统上的负载并获得更好的性能，还可以指定要在其中运行作业的数据库服务。有关详细信息，请参阅“ 调度程序和实际应用程序集群 ” 。

28.2作业和支持调度程序对象

您可以使用作业和其他调度程序对象进行任务调度。

  • 关于作业和支持计划程序对象
    要使用调度程序，请创建调度程序对象 。模式对象定义作业调度的内容，时间和位置。调度程序对象支持模块化方法来管理任务。模块化方法的一个优点是在创建与现有任务类似的新任务时可以重用对象。
  • 程式
    程序对象（程序）描述了调度程序要运行的内容。
  • 时间表
    计划对象（计划）指定作业运行的时间和次数。
  • 工作
    作业描述用户定义的任务。
  • 目的地
    您可以指定用于运行作业的外部和数据库目标。
  • 文件监视器
    文件观察程序对象（文件观察程序）定义文件的位置，名称和其他属性，这些文件到达系统会导致计划程序启动作业。
  • 证书
    凭据是存储在专用数据库对象中的用户名和密码对。
  • 链
    链是实现依赖关系调度的手段，其中作业启动取决于一个或多个先前作业的结果。
  • 工作类
    通过作业类，您可以为成员作业分配相同的属性，为成员作业设置资源分配，以及为优先级分组作业。
  • 视窗
    窗口是运行作业的时间间隔。
  • 组
    组指定Scheduler对象列表。
  • 不兼容性
    不兼容性定义（或不兼容性）指定不兼容的作业或程序，其中一次只能运行一个组。

28.2.1关于作业和支持调度程序对象

要使用调度程序，请创建调度程序对象 。模式对象定义作业调度的内容，时间和位置。调度程序对象支持模块化方法来管理任务。模块化方法的一个优点是在创建与现有任务类似的新任务时可以重用对象。

主要的Scheduler对象是作业。作业定义要执行的操作，操作的计划以及执行操作的位置。创建大多数其他调度程序对象以支持作业。

28.2.2程序

程序对象（程序）描述了调度程序要运行的内容。

一个程序包括：

  • 操作：例如，存储过程的名称，在操作系统文件系统中找到的可执行文件的名称（“外部可执行文件”）或PL / SQL匿名块的文本。

  • 类型： STORED_PROCEDURE ， PLSQL_BLOCK ， SQL_SCRIPT ， EXTERNAL_SCRIPT ， BACKUP_SCRIPT或EXECUTABLE ，其中EXECUTABLE表示外部可执行文件。

  • 参数数量：存储过程或外部可执行文件接受的参数数量。

程序是与作业分开的实体。作业在某个时间运行或由于某个事件发生，并调用某个程序。您可以创建指向现有程序对象的作业，这意味着不同的作业可以使用相同的程序并在不同的时间和不同的设置下运行程序。使用正确的权限，不同的用户可以使用相同的程序，而无需重新定义它。因此，您可以创建程序库，用户可以从现有程序列表中进行选择。

如果程序引用的存储过程或外部可执行文件接受参数，则在创建程序后，在单独的步骤中定义这些参数。您可以选择为每个参数定义默认值。

28.2.3时间表

计划对象（计划）指定作业运行的时间和次数。

计划可以由多个作业共享。例如，业务季度的结束可能是许多工作的共同时间框架。每次定义新作业时，工作创建者都可以指向命名的计划，而不是定义季度末计划。

有两种类型的时间表：

  • 时间表

    使用时间表，您可以安排作业立即运行或稍后运行。时间表包括开始日期和时间，可选的结束日期和时间以及可选的重复间隔。

  • 活动安排

    使用事件计划，您可以指定在发生特定事件时执行作业，例如库存低于阈值或文件到达系统。有关事件的更多信息，请参阅“ 使用事件启动作业 ” 。

28.2.4.1关于工作

作业对象（作业）是描述用户定义任务的元数据集合。它定义了必须执行的内容（操作），何时（一次性或定期调度或触发事件），何处（目标）以及凭据。作业具有所有者，该所有者是创建它的模式。

运行数据库程序单元的作业称为数据库作业 。运行外部可执行文件的作业称为外部作业 。

在一个或多个远程位置运行数据库程序单元的作业称为远程数据库作业 。在一个或多个远程位置运行外部可执行文件的作业称为远程外部作业 。

您可以通过指定一个或多个目标来定义作业的运行位置。目标也是Scheduler对象，将在本节后面介绍。如果未指定目标，则假定作业在本地数据库上运行。

28.2.4.2指定作业操作

您可以通过指定要运行的数据库程序单元或外部可执行文件或现有程序对象（程序）的名称来指定作业操作。

您可以通过以下方式之一指定作业操作：

  • 通过将数据库程序单元或外部可执行文件指定为作业属性来运行。这称为内联指定作业操作。

  • 通过将现有程序的名称指定为作业属性，该程序指定要运行的数据库程序单元或外部可执行文件。作业所有者必须具有该程序的EXECUTE特权或EXECUTE ANY PROGRAM系统特权。

28.2.4.3指定作业计划

您可以通过设置作业对象的属性或现有计划对象的名称（计划）来指定作业计划。

您可以通过以下方式之一指定作业计划：

  • 通过设置作业对象的属性来定义开始和结束日期以及重复间隔，或定义启动作业的事件。这称为内联计划。

  • 通过将现有计划的名称指定为作业属性，该计划定义开始和结束日期以及重复间隔，或定义事件。

28.2.4.4指定作业目的地

您可以通过多种不同方式指定作业目标。

您可以通过以下方式之一指定作业目标：

  • 通过将单个命名的目标对象指定为作业属性。在这种情况下，作业在一个远程位置运行。

  • 通过将命名目标组指定为作业属性，该目标组等同于远程位置列表。在这种情况下，作业将在所有远程位置上运行。

  • 通过不指定目标属性，在这种情况下作业在本地运行。该作业运行以下任一项：

    • 本地数据库（创建作业的数据库）上的数据库程序单元

    • 本地主机上的外部可执行文件，具体取决于作业操作类型

28.2.4.5指定作业凭据

您可以通过指定命名凭据对象或允许作业的凭证属性保持为NULL来指定作业凭证。

您可以通过以下方式之一指定作业凭据：

  • 通过将命名凭据对象指定为作业属性，该对象包含数据库用户名和密码（用于数据库作业）。

    作业以凭证中指定的用户身份运行。

  • 通过允许作业的凭证属性保持为NULL ，在这种情况下，本地数据库作业将作为作业所有者运行。（ 见表28-1 。）作业所有者是创建作业的架构。

注意：

本地数据库作业始终以用户为作业所有者的方式运行，并将忽略任何已命名的凭据。

28.2.5.1关于目的地

目标对象（目标）定义运行作业的位置。

有两种类型的目的地：

  • 外部目标：指定用于运行远程外部作业的远程主机名和IP地址。

  • 数据库目标：指定用于运行远程数据库作业的远程数据库实例。

运行外部可执行文件（外部作业）的作业必须指定外部目标，运行数据库程序单元（数据库作业）的作业必须指定数据库目标。

如果在创建作业时指定目标，则作业将在该目标上运行。如果未指定目标，则作业将在创建它的系统上本地运行。

您还可以创建目标组，该目标组由目标列表组成，并在创建作业时引用此目标组。在这种情况下，作业将在组中的所有目标上运行。

注意：

目标组还可以包含关键字LOCAL作为组成员，指示作业也在本地主机或本地数据库上运行。

28.2.5.2关于目标和调度程序代理

目标对象中指定的远程位置必须运行Scheduler代理，并且必须向创建作业的数据库注册代理。

Scheduler代理使本地调度程序能够与远程主机通信，在那里启动和停止作业，并将远程作业状态返回到本地数据库。有关完整的详细信息，请参阅“ 指定目标 ” 。

  • 外部目的地
    您无法显式创建外部目标。当您向该数据库注册Scheduler代理时，它们将在您的本地数据库中创建。
  • 数据库目的地
    使用DBMS_SCHEDULER.CREATE_DATABASE_DESTINATION过程创建数据库目标。

父主题： 目标

28.2.5.2.1外部目的地

您无法显式创建外部目标。当您向该数据库注册Scheduler代理时，它们将在您的本地数据库中创建。

分配给外部目标的名称是代理的名称。您可以在安装后配置代理名称，也可以接受默认代理名称，即主机名的第一部分（在第一个点分隔符之前）。例如，如果在主机dbhost1.us.example.com上安装代理，则代理名称默认为DBHOST1 。

28.2.5.2.2数据库目的地

使用DBMS_SCHEDULER.CREATE_DATABASE_DESTINATION过程创建数据库目标。

注意：

28.2.6文件监视器

文件观察程序对象（文件观察程序）定义文件的位置，名称和其他属性，这些文件到达系统会导致计划程序启动作业。

28.2.8链

链是实现依赖关系调度的手段，其中作业启动取决于一个或多个先前作业的结果。

链由使用依赖性规则组合的多个步骤组成。依赖关系规则定义了可用于启动或停止步骤或链本身的条件。条件可以包括先前步骤的成功，失败或完成或退出代码。可以在条件中使用逻辑表达式，例如AND / OR。从某种意义上说，链类似于决策树，有许多可能的路径来选择运行和何时运行。

在最简单的形式中，链由两个或多个调度程序对象（程序）组成，这些对象链接在一起用于单个组合目标。链的一个例子可能是“运行程序A后跟程序B，然后仅在程序A和B成功完成时运行程序C，否则等待一小时然后运行程序D.”

例如，您可能希望创建一个链，以组合成功金融交易所需的不同程序，例如验证和批准贷款申请，然后为贷款提供资金。

Scheduler作业可以指向链而不是指向单个程序对象。然后，这项工作可以启动这个链条。这项工作被称为连锁工作 。多个链作业可以指向同一个链，并且这些作业中的多个可以同时运行，从而创建同一链的多个实例，每个实例处于链中的不同进度点。

链中的每个位置称为步骤 。通常，在开始一组初始链步骤之后，连续步骤的执行取决于一个或多个先前步骤的完成。每个步骤都可以指向以下之一：

  • 程序对象（程序）

    该程序可以运行数据库程序单元（例如存储过程或PL / SQL匿名块）或外部可执行文件。

  • 另一条链（嵌套链）

    链可以嵌套到任何级别。

  • 事件计划，内联事件或文件监视器

    在启动指向事件计划的步骤或具有内联事件规范的步骤之后，该步骤将一直等到指定的事件被引发。同样，引用文件观察者内联或指向引用文件观察者的事件调度的步骤将等待，直到引发文件到达事件。对于文件到达事件或任何其他类型的事件，当事件发生时，该步骤完成，并且可以运行依赖于事件步骤的步骤。链中事件的常见示例是用户干预，例如批准或拒绝。

链中的多个步骤可以调用相同的程序或嵌套链。

对于每个步骤，您可以指定应在其上运行步骤的数据库目标或外部目标。如果未指定目标，则该步骤将在原始（本地）数据库或本地主机上运行。链中的每个步骤都可以在不同的目的地上运行。

28.2.9工作类

通过作业类，您可以为成员作业分配相同的属性，为成员作业设置资源分配，以及为优先级分组作业。

通常，只有在您担任Scheduler管理员角色时才创建作业类。

工作类提供了一种方法：

  • 将同一组属性值分配给成员作业

    每个作业类都指定一组属性，例如日志记录级别。将作业分配给作业类时，作业将继承这些属性。例如，您可以为清除所有工资核算作业的日志条目指定相同的策略。

  • 为成员作业设置服务亲和力

    您可以将作业类的service属性设置为所需的数据库服务名称。这将确定运行成员作业的Real Application Clusters环境中的实例，以及可选的分配给成员作业的系统资源。有关详细信息，请参阅“ 使用计划程序时的服务亲和性 ” 。

  • 为成员作业设置资源分配

    作业类提供数据库资源管理器和调度程序之间的链接，因为每个作业类都可以将资源使用者组指定为属性。然后，成员作业属于指定的使用者组，并根据当前资源计划中的设置分配资源。

    或者，您可以将resource_consumer_group属性保留为NULL ，并将作业类的service属性设置为所需的数据库服务名称。该服务又可以映射到资源使用者组。如果这两个resource_consumer_group和service属性设置，指定服务映射到资源消费群，在指定的资源消费群resource_consumer_group属性优先。

    有关将服务映射到使用者组的更多信息，请参阅使用Oracle Database Resource Manager管理资源 。

  • 将工作分组以确定优先级

    在同一作业类中，您可以将1-5的优先级值分配给单个作业，这样，如果类中的两个作业计划同时启动，则优先级较高的作业优先。这可以确保您没有那么重要的工作，无法及时完成更重要的工作。

    如果两个作业具有相同的分配优先级值，则具有较早开始日期的作业优先。如果没有为作业分配优先级，则其优先级默认为3。

    注意：

    工作优先级仅用于优先考虑同一班级中的工作。

    无法保证A类中的高优先级作业将在B类中的低优先级作业之前启动，即使它们共享相同的计划。在不同类别的作业之间进行优先级取决于当前资源计划以及每个作业类的指定资源使用者组或服务名称。

定义作业类时，尝试按功能对作业进行分类。考虑将作业划分为访问类似数据的组，例如营销，生产，销售，财务和人力资源。

要记住的一些限制是：

  • 工作必须是一个班级的一部分。创建作业时，可以指定作业所属的类。如果未指定类，则作业将自动成为类DEFAULT_JOB_CLASS一部分。

  • 在该类中仍有作业时删除一个类会导致错误。即使仍有作为该类成员的作业，也可以强制删除类，但随后会自动禁用引用该类的所有作业并将其分配给类DEFAULT_JOB_CLASS 。属于已经运行的已删除类的作业继续在作业开始时确定的类设置下运行。

28.2.10.2重叠Windows

虽然Oracle不推荐它，但Windows可以重叠。

由于一次只能激活一个窗口 ，因此以下规则用于确定窗口重叠时哪个窗口处于活动状态：

  • 如果具有相同优先级的窗口重叠，则处于活动状态的窗口将保持打开状态。但是，如果重叠是具有较高优先级的窗口，则较低优先级窗口将关闭，具有较高优先级的窗口将打开。当前运行的作业具有命名低优先级窗口的计划，可能会停止，具体取决于您在创建作业时分配的行为。

  • 一个自动关闭的打开窗口关闭。此时，先前的规则适用。

28.2.12不相容

不兼容性定义（或不兼容性）指定不兼容的作业或程序，其中一次只能运行一个组。

例如，如果作业A和B被定义为不兼容，则调度程序确保在任何给定时间只能运行其中一个， 即使它们各自的作业计划会导致它们同时运行。

可以在作业级别（默认）或程序级别定义不兼容性。例如，假设以下内容：

  • 作业J1和J2基于程序P1。

  • 作业J3，J4和J5，基于程序P2。

  • 乔布斯J6和J7基于程序P3。

在这种情况下：

  • 如果作业级不兼容定义指定J3，J4和J5，并且作业J3正在运行，则J4和J5在J3完成之前无法运行。

28.3.1.3多目标作业

多目标作业是一个作业，其实例在多个目标数据库或主机上运行，但可以从一个中央数据库进行控制和监视。

对于必须管理多个数据库或多个主机的DBA或系统管理员，多目标作业可以使管理变得更加容易。使用多目标作业，您可以：

  • 指定必须运行作业的多个数据库或主机。

  • 使用单个操作修改在多个目标上计划的作业。

  • 停止在一个或多个远程目标上运行的作业。

  • 确定每个远程目标上的作业实例的状态（运行，已完成，失败等）。

  • 确定作业实例集合的总体状态。

多目标作业可以被视为用于特定目的的单个实体，也可以被视为用于其他目的的独立运行作业的集合。创建或更改作业元数据时，多目标作业看起来像一个实体。但是，当作业实例正在运行时，它们最好被视为几乎完全相同的作业集合。在源数据库中创建的作业称为父作业 ，而在各个目标上运行的作业实例称为子作业 。

您可以通过将目标组分配给作业的destination_name属性来创建多目标作业。作业在预定时间或在检测到指定事件时在组中的所有目标上运行。可以将本地主机作为作业运行的目标之一包括在内。

对于其操作是数据库程序单元的作业，您必须在destination_name属性中指定数据库目标组。数据库目标组的成员包括数据库目标和关键字LOCAL ，后者表示原始（本地）数据库。对于其操作是外部可执行文件的作业，您必须在destination_name属性中指定外部目标组。外部目标组的成员包括外部目标和关键字LOCAL ，后者表示本地主机。

注意：

数据库目标不一定必须引用远程数据库;他们可以引用与创建作业的数据库在同一主机上运行的其他数据库实例。

多目标作业和时区

某些作业目标可能位于与创建父作业的数据库 （ 原始数据库 ）不同的时区中。在这种情况下，作业的开始时间始终基于原始数据库的时区。因此，如果您在英国伦敦创建父作业，请指定开始时间为晚上8:00，并在东京，洛杉矶和纽约指定目的地，则所有子作业将从伦敦时间晚上8:00开始。由于系统负载变化，需要重试的问题等，所有目的地的启动时间可能不准确。

基于事件的多目标作业

对于基于事件的多目标作业，当父作业在其主机上检测到该事件时，它将启动所有目标上的所有子作业。子作业本身不会检测各自主机上的事件。

28.3.1.5分离的工作

您使用分离的作业来启动在单独的进程中运行的脚本或应用程序，与调度程序独立且异步地运行。

分离的作业通常会启动另一个进程然后退出。退出时（作业操作完成时），分离的作业仍处于运行状态。运行状态表示作业启动的异步进程仍处于活动状态。当异步进程完成其工作时，它必须连接到数据库并调用DBMS_SCHEDULER 。 END_DETACHED_JOB_RUN ，结束作业。

当use_current_session参数设置为TRUE时，无法使用run_job执行分离的作业以手动触发执行。

如果作业指向其detached属性设置为TRUE （ 分离程序 ）的程序对象（程序），则detached 。

您在以下两种情况下使用分离的作业：

  • 等待启动的异步进程完成是不切实际的，因为这会不必要地保留资源。

    一个示例是向异步Web服务发送请求。Web服务可能需要数小时或数天才能响应，并且您不希望在等待响应时保留Scheduler作业slave。（有关作业从站的信息，请参阅“ 计划程序体系结构 ” 。）

  • 当无法等待启动的异步进程完成时，因为该进程会关闭数据库。

    一个示例是使用Scheduler作业来启动RMAN脚本，该脚本关闭数据库，进行冷备份，然后重新启动数据库。请参阅“ 创建分离的作业” 。

分离的工作如下：

  1. 当作业开始时，作业协调器为作业分配作业从站，作业从站运行分离程序中定义的程序操作。程序操作可以是PL / SQL块，存储过程或外部可执行文件。

  2. 程序操作执行另一个脚本或可执行文件的立即返回调用，此处称为进程A，然后退出。由于程序操作的工作已完成，因此作业从属服务器退出，但使作业处于运行状态。

  3. 进程A执行其处理。如果它对数据库运行任何DML，它必须提交其工作。处理完成后，进程A登录到数据库并调用END_DETACHED_JOB_RUN 。

  4. 分离的作业记录为已完成。

您还可以调用STOP_JOB来结束正在运行的分离作业。

28.3.1.6轻量级工作

当您有许多经常运行的短期作业时，请使用轻量级作业。在某些情况下，使用轻量级作业可以带来很小的性能提升。

轻量级作业具有以下特征：

  • 与常规作业不同，它们不是架构对象。

  • 与常规作业相比，它们具有明显更好的创建和删除时间，因为它们没有创建模式对象的开销。

  • 与普通工作相比，他们的平均会话创建时间更短。

  • 它们在磁盘上占用的空间很小，可用于作业元数据和运行时数据。

您可以通过设定指定一个轻量级的工作job_style作业属性为“ LIGHTWEIGHT ”。（默认作业样式为“ REGULAR ”。）

与程序和计划类似，常规作业是架构对象。常规作业提供了最大的灵活性，但在创建或删除时会产生一些开销。用户可以对作业上的权限进行细粒度控制，作业可以将其他用户拥有的程序或存储过程作为其操作。

如果需要创建相对较少数量的不经常运行的作业，则常规作业优先于轻量级作业。

轻量级作业必须引用程序对象（程序）来指定作业操作。创建轻量级作业时，必须已启用该程序，并且程序类型必须为“ PLSQL_BLOCK ”或“ STORED_PROCEDURE ”。由于轻量级作业不是架构对象，因此您无法为它们授予权限。轻量级作业从其指定的程序继承权限。因此，对程序具有特定权限集的任何用户都对轻量级作业具有相应的权限。

28.3.1.7内存中作业

当应在短时间内创建并运行许多作业时，请使用内存中作业。内存中的作业具有稍大的内存占用，但使用内存缓存来减少磁盘访问以及创建和执行作业所需的时间。性能提升可能很大。

可以使用以下类型的内存中作业：运行时（ IN_MEMORY_RUNTIME ）和完整（ IN_MEMORY_FULL ）。

  • 内存中运行时作业基于轻量级作业 ，因此它们是持久的。它们可以具有重复间隔并且可以多次运行。

    默认情况下，内存中作业与job_class DEFAULT_IN_MEMORY_JOB_CLASS相关联，其日志记录级别为NONE 。这意味着默认情况下，内存中作业不会在与Scheduler作业相关的视图中生成日志记录，从而提高了性能。

  • 内存中的完整作业仅存在于内存中，因此它们不是持久的。它们必须具有关联的程序，并且它们只能运行一次并被丢弃，因此它们不能具有重复间隔。因为它们没有磁盘上的备份，所以它们在创建或运行时不会生成重做，从而大大加快了它们的运行速度。

    内存中的完整作业仅存在于创建它们的实例中或存在于其中一个RAC实例中。它们不会传播到逻辑或物理备用实例，因此如果将主实例切换为备用实例，则它们将无法再运行（并将被丢弃）。

与Oracle数据库12c的开始，您可以使用几个新的脚本作业来运行SQL * Plus中，RMAN解释，或命令shell如自定义用户脚本cmd.exe为Windows和sh壳或其他解释为基于UNIX的系统。

这些可执行文件都需要OS凭据。这些脚本作业是：

  • SQL脚本作业：需要数据库目标。

    SQL脚本作业使用SQL * Plus解释器来运行Scheduler作业。因此，您现在可以使用所有SQL * Plus功能，包括查询输出格式。

    为了在产生后连接到数据库，SQL脚本作业需要一个身份验证步骤。用户可以在作业操作中使用内联，使用Scheduler提供的connect_credential功能进行身份验证。要使用connect_credential功能，用户需要设置作业的connect_credential_name属性。然后，作业尝试使用该connect_credential.的用户名，密码和角色连接到数据库connect_credential.

  • 外部脚本作业：需要正常的目标

    外部脚本作业生成一个新的shell解释器，允许一种简单的方法来运行命令行脚本。

  • 备份脚本作业：需要数据库目标。

    备份脚本作业提供了一种更直接的方法来指定创建和执行备份任务的RMAN脚本。

    为了在产生后连接到数据库，备份脚本作业需要一个身份验证步骤。用户可以在作业操作中使用内联，使用Scheduler提供的connect_credential功能进行身份验证。要使用connect_credential功能，用户需要设置作业的connect_credential_name属性。然后，作业尝试使用该connect_credential.的用户名，密码和角色连接到数据库connect_credential.

请注意，作业或程序操作必须指向每个解释器的相应脚本或具有适当的内联脚本。有关详细信息，请参阅job_action的参数CREATE_JOB子程序或program_action参数为CREATE_PROGRAM子程序。

28.3.2工作实例

作业实例表示特定的作业运行。计划仅运行一次的作业只有一个实例。具有重复计划或每次发生事件时运行的作业具有多个实例，每次运行该作业代表一个实例。

例如，计划仅在2009年10月8日星期二运行的作业有一个实例，每天中午运行一周的作业有七个实例，并且当文件到达远程系统时运行的作业具有每个文件到达事件的一个实例。

多目标作业为每个目标都有一个实例。如果多目标作业具有重复计划，则每个目标的每次作业运行都有一个实例。

创建作业时，只有一个条目添加到Scheduler的作业表中以表示作业。根据日志记录级别设置，每次作业运行时，都会在作业日志中添加一个条目。因此，如果创建具有重复计划的作业，则作业视图（ *_SCHEDULER_JOBS ）中有一个条目，作业日志中有多个条目。每个作业实例日志条目都提供有关特定运行的信息，例如作业完成状态以及开始和结束时间。每次运行的作业都会分配一个唯一的日志ID，该日志ID出现在作业日志和作业运行详细信息视图中（ *_SCHEDULER_JOB_LOG和*_SCHEDULER_JOB_RUN_DETAILS ）。

28.4.3.1关于工作协调员

作业协调器在数据库的控制下，控制和启动作业从属，利用作业表中的信息。

作业协调员后台进程（ cjqNNN ）自动启动并根据需要停止。在数据库启动时，作业协调器未启动，但数据库确实监视是否有任何要执行的作业，或者是否在不久的将来打开窗口。如果是，则启动协调器。

只要有作业或窗口运行，协调器就会继续运行。在一段时间的Scheduler不活动并且在不久的将来没有安排任何作业或窗口之后，协调器将自动停止。

当数据库确定是否启动作业协调器时，它会考虑作业的服务亲和性。例如，如果在不久的将来只安排了一个作业，并且此作业属于仅对四个Oracle RAC实例中的两个具有服务关联的作业类，则仅启动这两个实例的作业协调器。有关详细信息，请参阅“ 使用计划程序时的服务亲和性 ” 。

28.4.3.2工作协调员行动

作业协调员执行多个操作。

工作协调员：

  • 控制和生成作业从属

  • 查询作业表

  • 定期从作业表中获取作业并将它们放在内存缓存中。这可以减少磁盘的跳闸，从而提高性能

  • 从内存缓存中获取作业并将其传递给作业从属程序以执行

  • 当不再需要从属设备时，清理作业从属池

  • 没有安排任何工作时进入睡眠状态

  • 在即将执行新作业或使用CREATE_JOB过程创建作业时CREATE_JOB

  • 在数据库上，在数据库异常关闭后启动，恢复正在运行的所有作业。

您无需设置作业协调器检查作业表的时间;系统自动选择时间范围。

28.4.4工作如何执行

作业从属执行您提交的作业。

当工作时间被执行时，工作协调员会唤醒他们。他们收集元数据以从作业表中运行作业。

选择作业进行处理时，作业从站执行以下操作：

  1. 收集运行作业所需的所有元数据，例如，程序参数和权限信息。

  2. 以作业所有者的身份启动数据库会话，启动事务，然后开始执行作业。

  3. 作业完成后，从站提交并结束交易。

  4. 关闭会话。

28.4.5工作完成后

作业完成后，从站执行多个操作。

作业完成后，从站执行以下操作：

  • 如果需要，重新安排工作。

  • 更新作业表中的状态以反映作业是已完成还是计划再次运行。

  • 将条目插入作业日志表。

  • 更新运行计数，如有必要，更新失败并重试计数。

  • 清理。

  • 寻找新的工作（如果没有，他们去睡觉）。

调度程序根据需要动态调整从属池的大小。

28.5 Oracle Data Guard的调度程序支持

从Oracle Database 11g第1版（11.1）开始，Scheduler可以根据数据库是Oracle Data Guard环境中的主数据库还是逻辑备用数据库来运行作业。

对于物理备用数据库，对Scheduler对象所做的任何更改或主数据库上的Scheduler作业所做的任何数据库更改都会像任何其他数据库更改一样应用于物理备用数据库。

对于主数据库和逻辑备用数据库，还有一些附加功能使您可以指定只有在数据库处于主数据库或逻辑备用数据库的角色时才能运行作业。您可以使用DBMS_SCHEDULER.SET_ATTRIBUTE过程将database_role作业属性设置为以下两个值之一： 'PRIMARY'或'LOGICAL STANDBY' 。（要在两个角色中运行作业，您可以复制作业并将database_role设置为一个作业的'PRIMARY'和另一个作业的'LOGICAL STANDBY' ）。在切换或故障转移时，计划程序会自动切换到特定于新角色的正在运行的作业。DML被复制到作业事件日志，以便在故障转移时，有一个可用的记录，指出在主数据库上成功运行的内容，直到它失败。

从主数据库到逻辑备用数据库的调度程序作业的复制仅限于使用DBMS_ROLLING程序包完成的滚动升级中的升级目标。
