7.1.1关于语言支持

Oracle数据库使您能够以本机语言存储，处理和检索数据。可以存储在数据库中的语言都是用脚本编写的语言，这些脚本由Oracle支持的字符集编码。通过使用Unicode数据库和数据类型，Oracle数据库支持大多数当代语言。

可以为一部分语言提供附加支持。例如，数据库可以使用翻译的月份名称显示日期，并且可以根据文化惯例对文本数据进行排序。

在本文档中，术语“ 语言支持”是指附加的语言相关功能，而不是指存储特定语言文本的能力。例如，语言支持包括根据特定的语言环境和文化习俗显示日期或排序文本。此外，对于某些受支持的语言，Oracle数据库提供已翻译的服务器消息和数据库实用程序的已翻译用户界面。

7.1.4关于日历格式

不同的国家/地区使用不同的日历。

Oracle数据库存储每个区域的日历信息：

  • 一周的第一天

    周日在某些文化中，周一在其他文化中。由NLS_TERRITORY参数设置。

  • 日历年的第一周

    有些国家/地区使用周数进行调度，计划和簿记。在ISO标准中，本周数可能与日历年的周数不同。例如，2005年1月1日是ISO 2004年第53期。ISO周从星期一开始，到星期日结束。为了支持ISO标准，Oracle数据库提供IW日期格式元素，该元素返回ISO周号。一年中的第一个日历周由NLS_TERRITORY参数设置。

  • 一年中的天数和月数

    除了公历之外，Oracle数据库还支持六个日历系统，这是默认的。这些额外的日历系统是：

    • 日本帝国

      与公历有相同的月份和天数，但这一年从每个帝国时代的开始开始。

    • 中华民国官员

      与公历有相同的月份和天数，但这一年始于中华民国的成立。

    • 波斯语

      前六个月各有31天，接下来的五个月各有30天，上个月有29天或（闰年）30天。

    • 泰国佛使用佛教日历。

    • 阿拉伯语Hijrah有12个月和354或355天。

    • 英语Hijrah有12个月，354或355天。

    日历系统由NLS_CALENDAR参数指定。

  • 第一年的时代

    伊斯兰历法始于Hegira年。日本帝国的历法始于皇帝的统治时期（例如，1998年是平成时代的第十年）。

7.1.7关于长度语义

要使用字节长度计算字符串中的字符数，您必须知道字符集中每个字符的字节数。

在单字节字符集中，字符串中的字节数和字符数是相同的。在多字节字符集中，字符或代码点由一个或多个字节组成。在可变宽度字符集中，基于字节长度计算字符数可能是困难的。以字节为单位计算列长度称为字节语义 ，而以字符为单位测量列长度称为字符语义 。

字符语义对于指定不同宽度的多字节字符串的存储要求很有用。例如，在Unicode数据库（AL32UTF8）中，假设您必须具有VARCHAR2列，该列可以存储最多五个包含五个英文字符的中文字符。使用字节语义，此列需要15个字节用于中文字符，长度为3个字节，英文字符为5个字节，长度为1个字节，总共20个字节。使用字符语义，该列需要10个字符。

7.1.8关于Unicode和SQL国家字符数据类型

Unicode是一种字符编码系统，它定义了世界上大多数口语中的每个字符。在Unicode中，无论平台，程序或语言如何，每个字符都有唯一的代码。

您可以通过两种方式在Oracle数据库中存储Unicode字符：

  • 您可以创建一个Unicode数据库，使您可以将UTF-8编码的字符存储为SQL字符数据类型（ CHAR ， VARCHAR2 ， CLOB和LONG ）。

  • 您可以声明具有SQL国家字符数据类型的列和变量。

SQL国家字符数据类型是NCHAR ， NVARCHAR2和NCLOB 。它们也称为Unicode数据类型 ，因为它们仅用于存储Unicode数据。

在创建数据库时指定用于所有SQL国家字符数据类型的国家字符集。国家字符集可以是UTF8或AL16UTF16（默认）。

声明NCHAR或NVARCHAR2类型的列或变量时，指定的长度是字符数，而不是字节数。

7.4更改NLS参数值

您可以通过以下任何方式更改一个或多个NLS参数的值。

  • 更改所有SQL Developer连接的值，当前和未来。

  • 在客户端上，更改相应NLS环境变量的设置。

    仅在客户端上，NLS环境变量的新值将覆盖相应NLS参数的值。

    您可以使用环境变量为客户端指定依赖于区域设置的行为。例如，在Linux系统上，此语句将NLS_SORT环境变量的值设置为FRENCH ，从而覆盖NLS_SORT参数的值：

    ％setenv NLS_SORT FRENCH

    注意：

    环境变量可能与平台有关。

  • 仅使用具有以下语法的ALTER SESSION语句更改当前会话的值：

    ALTER SESSION SET parameter_name = parameter_value [ parameter_name = parameter_value ] ...;

    仅在当前会话中，新值将覆盖在上述所有方式中设置的值。

    您可以使用ALTER SESSION使用不同语言环境的设置测试您的应用程序。

  • 仅更改当前SQL函数调用的值。

    仅对于当前的SQL函数调用，新值将覆盖以上所有方式中设置的值。

  • 更改所有SQL Developer连接的NLS参数值
    您可以更改所有SQL Developer连接（当前和未来）的NLS参数值。
  • 更改当前SQL函数调用的NLS参数值
    其行为取决于NLS参数值的SQL函数称为区域设置相关 。一些与语言环境相关的SQL函数具有可选的NLS参数。

7.4.2更改当前SQL函数调用的NLS参数值

其行为取决于NLS参数值的SQL函数称为区域设置相关 。一些与语言环境相关的SQL函数具有可选的NLS参数。

具有可选NLS参数的本地相关函数是：

  • TO_CHAR

  • 至今

  • TO_NUMBER

  • NLS_UPPER

  • NLS_LOWER

  • NLS_INITCAP

  • NLSSORT

在上述所有函数中，您可以指定以下NLS参数：

  • NLS_DATE_LANGUAGE

  • NLS_DATE_LANGUAGE

  • NLS_NUMERIC_CHARACTERS

  • NLS_CURRENCY

  • NLS_ISO_CURRENCY

  • NLS_DUAL_CURRENCY

  • NLS_CALENDAR

  • NLS_SORT

在NLSSORT函数中，您还可以指定以下NLS参数：

  • NLS_LANGUAGE

  • NLS_TERRITORY

  • NLS_DATE_FORMAT

要在函数中指定NLS参数，请使用以下语法：

' parameter = value '[' parameter = value '] ...

假设您在评估此查询时希望NLS_DATE_LANGUAGE为AMERICAN ：

SELECT last_name FROM employees WHERE hire_date> '01 -JAN-1999';

在运行查询之前，您可以将NLS_DATE_LANGUAGE设置为AMERICAN ：

ALTER SESSION SET NLS_DATE_LANGUAGE = American ; SELECT last_name FROM employees WHERE hire_date> '01 -JAN-1999';

或者，您可以使用依赖于语言环境的SQL函数TO_DATE及其可选的NLS_DATE_LANGUAGE参数，在查询中将NLS_DATE_LANGUAGE设置为AMERICAN ：

SELECT last_name FROM employees WHERE hire_date> TO_DATE（'01 -JAN-1999'，'DD-MON-YYYY'， 'NLS_DATE_LANGUAGE = AMERICAN'） ;

小费：

在SQL函数中使用NLS参数的会话默认值通常会带来更好的性能。因此，仅在必须不使用默认NLS参数值的SQL语句中的语言环境相关SQL函数中指定可选NLS参数。

7.5关于单个NLS参数

可以使用许多单独的NLS参数。

  • 关于Locale和NLS_LANG参数
    区域设置是系统或应用程序运行的语言和文化环境。为Oracle数据库软件指定区域设置的最简单方法是设置NLS_LANG参数。
  • 关于NLS_LANGUAGE参数
    此参数指定数据库的默认语言。
  • 关于NLS_TERRITORY参数
    此参数指定日期格式，时间戳格式，小数和组分隔符，本地货币符号，ISO货币符号和双币种符号的默认约定。
  • 关于NLS_DATE_FORMAT参数
    此参数指定与TO_CHAR和TO_DATE函数一起使用的默认日期格式。
  • 关于NLS_DATE_LANGUAGE参数
    此参数指定由以下各项生成的日期和月份的名称和缩写语言：SQL函数TO_CHAR和TO_DATE ，默认日期格式（由NLS_DATE_FORMAT设置） ，以及默认语言等效项AM，PM，AD和符号的符号公元前。
  • 关于NLS_TIMESTAMP_FORMAT和NLS_TIMESTAMP_TZ_FORMAT参数

7.6.1在SQL和PL / SQL中表示Unicode字符串文字

在SQL或PL / SQL中有三种表示Unicode字符串文字的方法。

在SQL或PL / SQL中表示Unicode字符串文字的三种方法是：

  • N' 字符串 '

    示例： N'résumé' 。

    限制：请参阅“ 在字符集转换期间避免数据丢失 ” 。

  • NCHR（ 号码 ）

    SQL函数NCHR返回其二进制等效项是国家字符集中的数字的字符。返回的字符具有数据类型NVARCHAR2 。

    示例： NCHR(36)表示默认国家字符集AL16UTF16中的$。

    限制： NCHR（ 数字 ）值的可移植性仅限于使用相同国家字符集的应用程序。

  • UNISTR（' 字符串 '）

    SQL函数UNISTR将字符串转换为国家字符集。

    对于可移植性和数据保留，Oracle建议字符串仅包含ASCII字符和Unicode编码值。Unicode编码值的格式为\ xxxx ，其中xxxx是UCS-2编码格式的字符代码值的十六进制值。

    示例： UNISTR('G\0061ry')代表'Gary'

    ASCII字符转换为数据库字符集，然后转换为国家字符集。Unicode编码值直接转换为国家字符集。

7.6.2在字符集转换过程中避免数据丢失

作为SQL或PL / SQL语句的一部分，文字（带或不带前缀N ）编码在与语句其余部分相同的字符集中。在客户端，语句在客户端字符集中编码，该字符集由NLS_LANG参数确定。在服务器端，语句以数据库字符集编码。

当SQL或PL / SQL语句从客户端传输到数据库时，其字符集会相应地转换。如果数据库字符集不包含客户端在文本文字中使用的所有字符，则此转换中的数据将丢失。NCHAR字符串文字比CHAR文本文字更容易受到攻击，因为它们被设计为独立于数据库字符集。

为避免在转换为不兼容的数据库字符集时丢失数据，可以激活NCHAR文字替换功能。有关更多信息，请参阅“ Oracle数据库全球化支持指南” 。
