Oracle ACFS磁盘空间使用情况

Oracle ACFS在32位系统上支持64个已安装的文件系统，在64位系统上支持256个安装。但是，如果有足够的内存，可以安装更多的文件系统。

Oracle ACFS在文件系统中支持2 ^ 40（1万亿）个文件。已经测试了超过40亿个文件。文件系统中的目录数没有绝对限制;限制基于硬件资源。

Oracle ACFS预分配大型用户文件以提高写入数据时的性能。文件关闭时不会返回此存储，但在删除文件时会返回该存储。Oracle ACFS还会在节点首次挂载文件系统时分配本地元数据文件。这可能导致由于空间不足而导致安装失败，并且此存储的大部分必须是连续的。此存储大约为每节点64-128兆字节。

Oracle ACFS还可以保留本地位图，以便在搜索可用空间时减少对全局存储位图的争用。此磁盘空间被报告为in use的工具，如Linux的df即使一些空间，实际上可能没有尚未分配的命令。此本地存储池可以大到每个节点128兆字节，并且可以允许空间分配成功，即使命令（例如df ）报告的可用空间少于分配的空间。

Oracle ACFS错误处理

Oracle ACFS或其他文件系统使用Oracle ADVM卷时，Oracle ASM实例失败或强制关闭会导致I / O故障。必须关闭并重新打开卷才能再次访问卷。这需要卸载本地Oracle ASM实例失败时挂载的任何文件系统。重新启动实例后，必须在启用卷的情况下装入相应的磁盘组，然后重新装入文件系统。请参阅“ 取消注册，卸载和禁用卷和Oracle ACFS文件系统 ” 。

如果当前在Oracle ADVM卷文件上安装了任何文件系统，则在不先卸载这些文件系统的情况下，不应使用SHUTDOWN ABORT命令终止Oracle ASM实例。否则，应用程序会遇到I / O错误，并且在Oracle ASM存储被隔离之前，可能无法将终止时写入的Oracle ACFS用户数据和元数据刷新到存储。如果没有时间允许文件系统卸载，那么在发出SHUTDOWN ABORT操作之前，应该运行两个sync （1）命令将缓存的文件系统数据和元数据刷新到持久存储。

无论是由于Oracle ASM实例故障还是存储故障，Oracle ACFS都不会在元数据写入失败时中断操作系统环境。相反，Oracle ACFS将错误隔离到特定文件系统，使其处于脱机错误状态。从该点开始，该文件系统在该节点上成功的唯一操作是卸载操作。另一个节点恢复任何未完成的元数据事务，假设它可以将元数据写入存储。在解决I / O条件后，可以在脱线节点上重新安装文件系统。

如果存在引用文件系统的进程（例如，文件系统的目录是进程的当前工作目录），则管理员可能无法在文件系统处于脱机错误状态时卸载该文件系统。在这种情况下卸载文件系统时，有必要通过引用文件系统上的文件和目录来识别该节点上的所有进程，并使它们退出。Linux fuser或lsof命令或Window handle有关进程和打开文件的命令列表信息。

如果Oracle ACFS检测到从读取操作返回的文件元数据不一致（基于校验和或预期的类型比较），Oracle ACFS将采取适当的操作来隔离受影响的文件系统组件，并生成应尽快运行fsck或acfschkdsk的通知。每次挂载文件系统时，都会生成一个带有系统事件记录器消息的通知，直到运行fsck或acfschkdsk 。

Oracle ACFS和NFS

在Linux上通过NFS导出文件系统时，请使用-fsid=num exports选项。此选项强制用于与NFS客户端通信的文件句柄的文件系统标识部分是指定的编号，而不是从安装文件系统的块设备的主要编号和次编号派生的编号。您可以将任何32位数用于num ，但它在所有导出的文件系统中必须是唯一的。此外， num在集群成员中必须是唯一的，并且对于给定的文件系统，在集群的每个成员上必须是num 。这是必需的，因为不能保证Oracle ASM DVM块设备主要数字在同一节点的重新启动或群集中的不同节点之间保持相同。

使用High Availability NFS for Grid Home Clusters（HANFS）时，HANFS会自动处理上一段中描述的情况。有关HANFS的信息，请参阅“ Oracle Grid Infrastructure的高可用性网络文件存储 ” 。

Oracle ADVM的限制

本主题将讨论Oracle ADVM的限制。

Oracle ADVM卷的默认配置是8列和1 MB条带宽度。默认卷范围大小（64 MB）。

将Oracle ADVM动态卷上的列数设置为1有效地关闭Oracle ADVM卷的条带化。建议将列设置为8（默认值），以便使用数据库数据文件和其他文件获得最佳性能。

在Linux平台上，无论Oracle ASM磁盘组中底层存储的配置如何，Oracle ASM Dynamic Volume Manager（Oracle ADVM）卷设备都将创建为块设备。不要使用raw (8)将Oracle ADVM卷块设备映射到原始卷设备。

有关管理Oracle ADVM卷的ASMCMD命令的信息，请参阅使用ASMCMD管理Oracle ADVM 。

Oracle ACFS环回支持

Oracle ACFS支持Linux操作系统上的环回功能，可以将Oracle ACFS文件作为设备进行访问。

Oracle ACFS环回设备是一种操作系统伪设备，可以将Oracle ACFS文件作为块设备进行访问。此功能可与Oracle虚拟机（OVM）一起使用，以支持在Oracle ACFS文件系统中创建并通过Oracle ACFS环回设备呈现的OVM映像，模板和虚拟磁盘（虚拟磁盘）。

Oracle ACFS环回功能可提供超过NFS的性能提升。文件可以是稀疏的或非稀疏的。

除了一般的环回支持，Oracle ACFS还支持稀疏映像上的环回直接I / O（DIO）。

Oracle ACFS注册表资源管理

Oracle ACFS注册表资源仅支持Oracle Grid Infrastructure集群配置; Oracle Restart配置不支持它。请参阅“ Oracle ACFS和Oracle Restart ” 。

对于Oracle ASM 12 c第1版（12.1），Oracle ACFS注册表使用通过SRVCTL文件系统接口提供的标准单文件系统资源。有关更多信息，请参阅“ Oracle ACFS文件系统资源管理 ” 。使用SRVCTL使应用程序依赖于已注册的文件系统，如使用已注册的文件系统的管理srvctl filesystem 。默认情况下， acfsutil registry仅显示设置为始终挂载的文件系统，并将AUTO_START属性设置为always 。

Oracle ACFS注册表需要root权限才能注册和删除文件系统，但是，其他用户可以使用user选项来启动和停止（装载和卸载）文件系统。

Oracle ACFS文件系统资源管理

Oracle ACFS文件系统资源仅支持Oracle Grid Infrastructure集群配置; Oracle Restart配置不支持它。请参阅“ Oracle ACFS和Oracle Restart ” 。

Oracle ASM的配置助手（ASMCA）有利于甲骨文ACFS文件系统资源（创建ora. diskgroup . volume .acfs ）。在使用Database Configuration Assistant（DBCA）创建数据库期间，Oracle ACFS文件系统资源包含在其关联磁盘组的依赖关系列表中，因此停止磁盘组也会尝试停止任何相关的Oracle ACFS文件系统。

通常会创建Oracle ACFS文件系统资源以与应用程序资源依赖关系列表一起使用。例如，如果Oracle ACFS文件系统配置为用作Oracle数据库主目录，则为该文件系统创建的资源可以包含在Oracle数据库应用程序的资源依赖关系列表中。由于数据库应用程序的启动操作，此依赖性会导致文件系统和堆栈自动挂载。

Oracle ACFS文件系统资源的启动操作是装入文件系统。此Oracle ACFS文件系统资源操作包括确认关联的文件系统存储堆栈处于活动状态并装入磁盘组，启用卷文件，并在必要时创建装入点以完成装入操作。如果文件系统成功挂载，则资源状态设置为online ;否则，它被设置为offline 。

Oracle ACFS文件系统资源的检查操作将验证是否已装入文件系统。如果已挂载，则将资源状态设置为online状态，否则状态将设置为offline 。

Oracle ACFS文件系统资源的停止操作尝试卸载文件系统。如果由于打开引用而无法卸载文件系统，则stop操作将显示并记录包含引用的任何进程的进程标识符。

使用srvctl start和stop操作来管理Oracle ACFS文件系统资源可保持其正确的资源状态。

Oracle ACFS和Oracle Restart

Oracle Restart不支持此版本的基于root的Oracle ACFS资源。因此，不会自动执行以下操作：

  • 加载Oracle ACFS驱动程序

    在Linux上，驱动程序在系统引导时和系统关闭时自动加载和卸载。如果在系统运行或系统在其他操作系统（OS）版本上运行时需要执行操作，则可以使用acfsload命令手动加载或卸载驱动程序。有关更多信息，请参阅acfsload 。

  • 装载Oracle ACFS装入注册表中列出的Oracle ACFS文件系统

    Oracle Restart不支持Oracle ACFS安装注册表。但是，带有有效Oracle ASM设备的/etc/fstab文件中的Linux条目确实启用了关联的卷，并在系统启动时自动挂载，并在系统关闭时卸载。请注意，安装文件系统后不会应用高可用性（HA）恢复;该功能是一次性操作。

    有效的fstab条目具有以下格式：

    device mount_point acfs noauto 0 0

    例如：

    / dev / asm / dev1-123 / mntpoint acfs noauto 0 0

    上一个示例中的最后三个字段阻止Linux尝试自动挂载设备并尝试在设备上运行其他系统工具。当Oracle ASM实例在系统启动期间不可用时，此操作可防止出现错误。可以为文件系统安装添加其他标准fstab语法选项。

    如果在其他OS版本上需要安装或卸载操作，或者在系统启动后，您可以使用mount命令手动安装Oracle ACFS文件系统。有关信息，请参阅使用命令行工具管理Oracle ACFS 。

  • 挂载基于资源的Oracle ACFS数据库主文件系统

    不会为Oracle Restart配置创建与这些操作关联的Oracle ACFS资源。虽然Oracle Grid Infrastructure配置完全支持Oracle ACFS资源管理，但必须使用Oracle Restart配置中的备选（有时是手动）操作替换Oracle ACFS基于资源的管理操作。在尝试使用在Oracle Restart配置中注册基于root的资源的命令（例如srvctl ，会显示相应的错误。

Oracle ACFS预定义度量标准类型

Oracle ACFS提供ACFSMETRIC1_T和ACFSMETRIC2_T预定义的度量标准类型。

ACFSMETRIC1_T度量集是为存储虚拟化模型定义的。度量标准将作为选定的一组标记文件或文件系统中的所有文件的摘要记录进行维护。Oracle ACFS文件指标包括：读取次数，写入次数，平均读取大小，平均写入大小，最小和最大读取大小，最小和最大写入大小以及读取缓存（VM页面缓存）命中和未命中。

例：

typedef struct _ACFS_METRIC1 {ub2 acfs_version; ub2 acfs_type; ub4 acfs_seqno; ub8 acfs_nreads; ub8 acfs_nwrites; ub8 acfs_rcachehits; ub4 acfs_avgrsize; ub4 acfs_avgwsize; ub4 acfs_minrsize; ub4 acfs_maxrsize; ub4 acfs_minwsize; ub4 acfs_maxwsize; ub4 acfs_rbytes_per_sec; ub4 acfs_wbytes_per_sec; ub8 acfs_timestamp; ub8 acfs_elapsed_secs; } ACFS_METRIC1;

ACFSMETRIC2_T是Oracle ACFS写入描述记录的列表，其中包含每个写入的fileID ，起始偏移量，大小和序列号。序列号保留由插件驱动程序保留的Oracle ACFS写入记录顺序。序列号为应用程序提供了一种订购API返回的多个消息缓冲区的方法。它还提供了对丢弃的写入记录的检测，因为应用程序没有通过API足够快地耗尽消息缓冲区。

写记录包含在多个内存数组中。每个记录数组都可以使用API获取，缓冲区大小当前设置为1M。在获取的ioctl缓冲区的开头是一个描述数组的struct ，包括它包含的记录数。如果填充缓冲区，则内核缓冲区会删除最旧的写入记录，因为缓冲区的读取速度不够快。

例：

typedef struct _ACFS_METRIC2 {ub2 acfs_version; ub2 acfs_type; ub4 acfs_num_recs; ub8 acfs_timestamp; ACFS_METRIC2_REC acfs_recs [1]; } ACFS_METRIC2; typedef struct _ACFS_FILE_ID {ub8 acfs_fenum; ub4 acfs_genum; ub4 acfs_reserved1; } typedef struct _ACFS_METRIC2_REC {ACFS_FILE_ID acfs_file_id; ub8 acfs_start_offset; ub8 acfs_size; ub8 acfs_seq_num; } ACFS_METRIC2_rec;

Oracle ACFS插件API

目的

Oracle ACFS插件应用程序编程接口（API）从应用程序插件模块向本地插件启用的Oracle ACFS驱动程序发送和接收消息。

句法

sb8 acfsplugin_metrics（ub4 metric_type，ub1 * metrics，ub4 metric_buf_len，oratext * mountp）;
sb8 acfsfileid_lookup（ACFS_FILEID file_id，oratext * full_path，oratext * mountp）;

描述

Oracle ACFS应用程序插件模块使用acfsplugin_metrics API从Oracle ACFS驱动程序检索指标。首先必须使用acfsutil plugin enable命令启用Oracle ACFS驱动程序以进行插件通信。选定的应用程序插件度量标准类型模型必须与使用Oracle ACFS插件启用命令定义的插件配置相匹配。有关acfsutil plugin enable命令的信息，请参阅“ acfsutil plugin enable ” 。应用程序必须提供足够大的缓冲区来存储“ Oracle ACFS预定义度量标准类型 ”中描述的度量标准结构。

如果提供的缓冲区为NULL且metric_buf_len = 0 ，则返回值是保存所有当前收集的度量标准所需的大小。应用程序可以首先查询Oracle ACFS以查看需要多大的缓冲区，然后分配一个必要大小的缓冲区以传递回Oracle ACFS。

必须将提供路径提供给API，以标识正在引用的已启用插件的Oracle ACFS文件系统。

成功返回非负值： 0表示成功，没有更多要收集的指标， 1表示有更多指标可用，或2表示在间隔期间没有收集新指标。如果出现错误，则返回负值，并在Linux环境中设置errno或在Windows上调用SetLastError 。

使用度量标准类型＃2时，返回的度量标准包括ACFS_FILE_ID ，其中包含fenum和genum对。为了从fenum和genum对转换为文件路径，应用程序可以使用acfsfileid_lookup 。应用程序必须提供长度为ACFS_FILEID_MAX_PATH_LEN的缓冲区来保存路径。如果有多个指向文件的硬链接，则返回的路径是第一个。使用acfsutil info id时，这是相同的行为。

需要系统管理员或Oracle ASM管理员权限才能从启用插件的Oracle ACFS文件系统驱动程序发送和接收消息。

Oracle ACFS标记错误值

以下是发生故障时Linux，Solaris或AIX errno的值：

  • EINVAL - 标记名称语法无效或太长。

  • ENODATA - 此文件或目录不存在标记名称。

  • EACCES - 路径的路径前缀中的目录的搜索权限被拒绝;或者用户没有文件权限来读取标签名称。

acfsutil info ftrace

目的

显示与挂载点指定的Oracle ACFS文件系统关联的打开文件的跟踪条目。

表17-13 acfsutil info ftrace命令的选项

选项 描述

-s

仅显示打开文件的文件ID。

装入点

指定安装文件系统的目录。

acfsutil info ftrace命令显示已装载的Oracle ACFS文件系统上的打开文件列表。

Oracle ACFS内核驱动程序会跟踪内存中加载的文件。这些文件可能没有活动打开，可能只是缓存。打开文件跟踪能够确定哪些缓存文件控制块（FCB）具有活动打开或引用。此命令的目的是使您能够确定是否仍在引用文件以及哪些文件可能会阻止文件系统卸载。

最初运行acfsutil info ftrace ，该命令会尝试清除不再引用的任何缓存文件。此操作可能需要一些时间才能完成，因为必须将每个文件的已修改元数据和用户数据刷新到磁盘。

以下描述了acfsutil info ftrace命令的输出。请注意，文件可以引用常规文件或目录。

输出的基本格式是：

Fileid：％ID％，路径名：％PATH％[％OP％] Pid：％PID％Ppid：％PPID％经过时间：％TIME％Cmd：％CMD％...

以下列表中描述了这些字段。

  • %ID% ：数字文件标识符。这与acfsutil info id使用的编号相同。此值也称为Linux上的inode编号。

  • %PATH% ：基于%ID%生成的文件路径名。如果不可用，则可以显示N/A

  • %OP%:访问文件的操作类型。值可能如下：

    • LOOKUP ：指定的进程通过路径名查找此文件。

    • CREATE ：指定的进程创建了该文件。

    • NFS ：NFS进程代表客户端访问了该文件。

    • OPEN ：指定的进程打开文件。

    • MAP ：指定的进程将文件映射到内存中。

  • %PID% ：进程ID %PPID% ：父进程ID。此输出项可能不可用。

  • %TIME% ：从操作发生的经过时间。格式为：d（天），h（小时），m（分钟），s（秒）

  • %CMD% ：执行操作的进程的名称。

列出的每个文件可能会列出多个操作，具体取决于系统工作负载。显示的操作量仅限于节省内存。Oracle ACFS驱动程序为每个操作保留一个旋转日志，操作条目可以换行。结果，最旧的操作可能不是显示的第一个操作。

目的

选项 描述

-b

开始（启用）收集锁定统计信息。

-e

acfsutil日志

目的

检索内存诊断日志文件并管理调试设置。

语法和说明

acfsutil [-h] log acfsutil log [-f filename ] [-s] [-r n {K | M | G | T | P}] [-p {avd | ofs | oks}] [-l debuglevel ] [ -n consolelevel] [-o WAIT_TIME] [-q] [-c debugcontext] [-T FILE_TYPE] [-m 装入点 ] [-a] [-C] [-t]

有关运行Oracle ACFS acfsutil命令的信息，请参阅关于使用Oracle ACFS命令行工具 。

下表包含acfsutil log命令可用的选项。

表17-15 acfsutil log命令的选项

选项 描述

-f 文件名

将内存中的日志写入指定的文件。默认文件是当前目录中的oks.log 。

-s

显示内存中日志文件的大小。

-r n {K | M | G | T | P}

设置内存中日志文件的大小。

-p {avd | ofs | oks}

指定用于设置级别或查询设置的产品。默认所有产品：Oracle ADVM（ avd ），Oracle ACFS（ ofs ）和Oracle内核服务（ oks ）

-l debuglevel

设置内存中的调试级别。默认调试级别为2 。有效值为0 - 6 。

-n consolelevel

设置持久日志记录的调试级别。其他持久性日志配置设置由acfsutil plogconfig命令管理。

-o wait_time

设置所有节点上的日志大小，调试级别和产品值;等待wait_time指定的秒数;转储到所有节点的内存日志中;然后重置调试级别和日志大小。

-q

查询指定产品的调试设置。

例如： acfsutil log —p avd —q

-c debugcontext

设置调试上下文，仅限内部。

-T file_type

设置调试文件类型，仅限内部。

-m mount_point

指定仅调试指定安装点的文件系统。

-一个

将调试日志记录重置为记录所有文件系统。

-C

转储所有群集节点上的内存日志，也可以添加到-t选项。

-t

将所有Hang Manager线程信息转储到内存中和持久日志中。

使用acfsutil log命令可以管理内存诊断日志文件。如果未指定任何选项，则acfsutil log命令会默认检索和写入./oks.log内存日志。

-o选项执行以下操作：

  1. 将日志大小设置为500M ，将日志级别设置为5 ，并将所有节点上的内存中日志的产品设置为ofs （acfs）

  2. 显示信息性消息，例如阻止180秒，立即重现问题

  3. 等待指定的秒数后，显示所有节点上的转储日志

  4. 启动集群范围的日志转储

  5. 将日志级别重置为2并将日志大小重置为默认值

如果应更改默认产品，调试级别或日志大小设置，则-o选项可与p ， -l,和-r选项组合使用。

您必须是root用户或Oracle ASM管理员用户才能运行此命令。

将元数据从Oracle ACFS文件系统复制到单独的输出文件中。

语法和说明

acfsutil meta -h acfsutil meta [-v] [-g] [-g -O -C -S] [-O] [-C COW_filepath ] [-S COW_size ] [-q nn [K | M | G | T ]] [-l log_file_path ] [-o acfs_extent_offsets ] {-f record_oriented_metadata_output_file } [-a accel_device ] volume_device acfsutil meta {-e record_oriented_metadata_input_file [-i]} {-f output_filesystem_meta_file_prefix_name }

acfsutil meta -h显示帮助文本和退出。

有关运行Oracle ACFS acfsutil命令的信息，请参阅关于使用Oracle ACFS命令行工具 。

acfsutil meta命令用作元数据收集器，以将Oracle ACFS文件系统部分复制到单独的指定面向记录的输出文件中。元数据收集器读取由Oracle ACFS文件系统的卷设备名称指定的文件系统的内容。在此输入文件系统中搜索Oracle ACFS元数据，然后将找到的所有元数据写入指定的面向记录的输出文件。生成的面向记录的输出文件可以轻松地传输到另一个系统，在那里它可以扩展用于诊断和分析，而不会影响客户站点的原始文件系统。

-g选项仅收集已知的良好元数据。-g选项不应与损坏的文件系统一起使用，因为-g选项不会找到丢失的元数据。任何丢失的元数据在诊断文件系统损坏方面可能都很重要。如果文件系统处于良好的修复状态，则-g选项可以更快地收集元数据，因为它不需要扫描整个物理卷以查找丢失的元数据块。

在Linux上运行acfsutil meta -g命令时，Oracle ACFS在线检查程序（ fsck ）会自动运行。代表acfsutil meta -g运行的Oracle ACFS联机检查程序使用磁盘上的元数据指针acfsutil Oracle ACFS文件系统元数据，并将已读取的meta数据写入acfsutil meta -g元数据集合文件。有关Linux上的联机fsck命令的信息，请参阅适用于Linux环境的Oracle ACFS命令行工具 。

要使用acfsutil meta获取文件系统的最佳副本，请在运行acfsutil meta之前卸载文件系统。如果无法卸载文件系统，请避免在运行acfsutil meta修改内容或执行卷大小调整操作。

如果原始文件系统非常大，那么输出文件也可能非常大。尽可能压缩输出文件以减少存储空间和传输时间。

如果文件系统具有与之关联的加速器设备，则acfsutil meta还将加速器设备数据复制到面向记录的输出文件中。此操作自动发生。

在大多数情况下， acfsutil meta自动将加速器设备复制到面向记录的输出文件中。但是，如果您认为元收集器无法单独找到加速器设备，则可以使用-a选项在命令行上指定名称。例如，如果文件系统已损坏，则可能发生这种情况。请注意，使用-a选项会覆盖元收集器如何自动运行，因此应小心使用-a 。

输出文件不应放在指定为输入设备的Oracle ACFS设备上，因为metadata命令也可能处理输出文件。输出文件应放在可支持输出文件的文件系统上，该文件的大小与Oracle ACFS输入卷设备相同。除非文件系统已满并且包含所有元数据且几乎没有用户数据，否则输出文件不应该需要所有存储，这不太可能，但并非不可能。

应谨慎使用-q标志。指定-q ，元收集器不扫描并复制整个输入文件系统。相反，它仅扫描和复制预定数量的字节和被认为重要的某些数据结构。-q标志的主要用途是在没有足够时间运行完整版本的元数据收集器的情况下。除非调查问题的支持人员建议，否则不应使用-q标志。

应在要执行诊断和分析的系统上执行扩展面向记录的输出文件。例如，以下命令在具有足够存储空间的另一个文件系统上展开面向记录的元数据文件。

acfsutil meta -e record_oriented_metadata_input_file -f output_filesystem_meta_file_prefix_name

该命令的输出提供了适用于fsck或acfschkdsk稀疏文件。如果面向记录的元数据输入文件包括加速器卷，则使用附加了.acc后缀的相同输出文件名前缀创建第二稀疏输出文件。用于扩展文件的文件系统应支持稀疏文件。否则，生成的扩展文件可能非常大，包含无用的零，其中稀疏的孔可以节省空间。

在大多数情况下， fsck命令可以读取acfsutil meta扩展输出文件。但是，某些OS平台上的Oracle ACFS特定fsck命令可能无法正确访问输出文件，或者可能无法使用指定的标志。在这些情况下，您可以使用稍微修改过的fsck命令表单。例如：

  • 在Linux上，如果使用—x标志，请以此格式运行命令：

    /sbin/fsck.acfs -x filesystem_meta_file.acc filesystem_meta_file

  • 在Solaris上，如果使用—o x标志，请以此格式运行命令：

    / usr / lib / fs / acfs / fsck -ox = filesystem_meta_file.acc filesystem_meta_file

  • 在AIX上，以以下格式运行命令：

    / sbin / helpers / acfs / fsck filesystem_meta_file

  • 在Windows上，以以下格式运行命令：

    acfschkdsk filesystem_meta_file

了解Oracle ACFS I / O故障控制台消息

Oracle ACFS在特定于操作的系统事件日志中记录I / O故障的信息。

控制台消息具有以下格式：

[Oracle ACFS]：在operation_name op_type期间，设备device_name的 I / O失败（ error_code ）。
file_entry_num起始偏移量： 偏移量 。数据传输长度： io_length个字节。影响： acfs_type对象： object_type Oper。上下文： operation_context快照？： yes_or_no AcfsObjectID： acfs_object_id 。内部ACFS位置： code_location 。

控制台消息语法中的斜体变量对应于以下内容：

  • I / O失败

    Oracle ACFS针对失败的I / O看到的操作系统特定的错误代码（十六进制）。这可能表示存在硬件问题，或者可能表示由于某些其他原因未能启动I / O.

  • 设备

    涉及的设备，通常是ADVM设备文件，但在某些情况下，它可能是一个表示设备次要编号的字符串

  • 操作名称

    涉及的操作类型：

    user data ， metadata或paging

  • 操作类型

    涉及的操作类型：

    synch read ， synch write ， asynch read或asynch write

  • 文件条目号

    涉及的文件系统对象的Oracle ACFS文件条目号，以十进制数表示。acfsutil info fileid工具查找相应的文件名。

  • 抵消

    I / O的磁盘偏移量，以十进制数表示。

  • I / O长度

    I / O的长度（以字节为单位），以十进制数表示。

  • 受影响的文件系统对象

    指示所涉及的文件系统对象是节点本地的，或者是在群集范围内访问的资源。例如：

    Node或Cluster

  • 受影响的对象类型

    尽可能指示所涉及的文件系统对象类型的字符串。例如：

    Unknown ， User Dir. ， User Symlink ， User File ， Sys.Dir ， Sys.File或MetaData

    • Sys系统。迪尔。

      可见命名空间内的Oracle ACFS管理目录

    • SYS。文件

      可见命名空间内的Oracle ACFS管理文件

    • 元数据

      Oracle ACFS管理的资源位于可见命名空间之外

  • 运作环境

    发布I / O的代码上下文的更高级别视图。这供Oracle支持服务使用。例如：

    Unknown ， Read ， Write ， Grow ， Shrink ， Commit或Recovery

  • 快照

    如果可能确定所涉及的数据的指示来自快照。例如：

    Yes ， No ，或?

  • 文件系统的对象类型

    文件系统对象类型的内部标识符。供Oracle支持服务使用。

  • 代码的位置

    发出此消息的代码位置的内部标识符。供Oracle支持服务使用。
