Oracle Database 12 c第2版（12.2.0.1）中的更改

以下是Oracle数据库实际应用程序安全管理员和 Oracle数据库12 c版本2（12.2.0.1） 开发人员指南中的更改：

  • 新功能

  • 不推荐的功能

新功能

以下功能是此版本中的新增功能：

  • Real Application Security包括对特权范围的支持

    Oracle Database 12 c第2版（12.2）通过允许每个主体会话权限授予，通过作为本机Real Application Security应用程序用户的主体上的ACL设置来扩展Real Application Security模型，以授予会话管理权限。此外，Oracle Database 12c第2版（12.2）通过允许每个主要会话权限授予（尽管在主体上设置ACL作为动态角色）来扩展Real Application Security模型，仅授予SET_DYNAMIC_ROLES权限。特定于主体的ACL授权优先于系统级会话特权授予。它允许在主体特定ACL上设置否定授权。使用ACL允许在一组本机应用程序用户和动态角色上强制执行一组通用授权。

    此功能提供以下新API：

    • SET_ACL过程 - 为指定的应用程序用户或动态角色设置ACL。

    此功能通过添加acl参数增强了以下API：

    • CREATE_USER过程

    • CREATE_DYNAMIC_ROLE过程

    此功能通过显示在用户和/或动态角色上设置的ACL或两者来增强以下视图：

    • DBA_XS_USERS

    • DBA_XS_DYNAMIC_ROLES

    此功能添加了SET_DYNAMIC_ROLES权限，该权限在SESSION_SC安全性类中定义，以保护动态角色的启用和SET_DYNAMIC_ROLES ，作为附加会话的一部分并分配用户操作。

    见SET_ACL程序 ， CREATE_USER程序 ， CREATE_DYNAMIC_ROLE程序 ， DBA_XS_USERS ， DBA_XS_DYNAMIC_ROLES和SESSION_SC安全类安全等级获取更多信息。

    有关详细信息，请参阅关于实际应用程序安全性会话权限通过ACL确定范围 。

  • Real Application Security支持对DML语句的列级访问控制。这允许用户根据其授予的列级权限插入，更新和删除特定列值。

    从Oracle Database 12 c第2版（12.2）开始，具有所需权限的用户可以使用Data Security列安全性执行DML。这意味着：

    • 要更新行值，授权用户需要UPDATE行级UPDATE权限以及受保护列的列权限。

    • 要插入行，授权用户需要行级INSERT权限以及每个受保护列的列权限。如果INSERT语句未插入受保护列的值，则不需要列权限，并插入默认值（如果没有默认值，则为NULL ）。

    • 要删除行，授权用户只需要行级DELETE权限。列权限不是必需的。

    • 没有针对具有Data Security行级和列级安全性的DML公开数据。使用RETURNING INTO或使用参数sql92_security启用的DML语句需要行级SELECT权限以及列权限（如果列出现在RETURNING INTO子句中）。

  • Real Application Security包括对模式级安全策略管理的支持

    此功能增强了以下API：

    • GRANT_SYSTEM_PRIVILEGE添加schema参数的过程

    • REVOKE_SYSTEM_PRIVILEGE添加schema参数的过程

    此功能将ADMIN_SEC_POLICY权限扩展为策略管理的模式。

    有关更多信息，请参阅XS_ACL包 ， XS_DATA_SECURITY包和XS_SECURITY_CLASS包 。

    此功能在授予的模式中为策略实施添加APPLY_SEC_POLICY权限，以在应用程序中实现策略实施。

    在执行策略之前，将在以下API中检查APPLY_SEC_POLICY权限： APPLY_OBJECT_POLICY过程 ， REMOVE_OBJECT_POLICY过程 ， ENABLE_OBJECT_POLICY过程和DISABLE_OBJECT_POLICY过程 。

    此功能添加了两个审核操作：

    • AUDIT_GRANT_PRIVILEGE

      - 审核GRANT_SYSTEM_PRIVILEGE API

    • AUDIT_REVOKE_PRIVILEGE

      - 审核REVOKE_SYSTEM_PRIVILEGE API

    此功能增加了以下观点： ALL_XS_SECURITY_CLASSES ， ALL_XS_SECURITY_CLASS_DEP ， ALL_XS_PRIVILEGES ， ALL_XS_IMPLIED_PRIVILEGES ， ALL_XS_ACLS ， ALL_XS_ACES ， ALL_XS_POLICIES ， ALL_XS_REALM_CONSTRAINTS ， ALL_XS_INHERITED_REALMS ， ALL_XS_ACL_PARAMETERS ， ALL_XS_COLUMN_CONSTRAINTS ， ALL_XS_APPLIED_POLICIES和DBA_XS_PRIVILEGE_GRANTS 。

    有关详细信息，请参阅关于架构级别Real Application Security Policy Administration 。

  • Oracle Label Security支持Oracle Database Real Application Security

    对于user_name在参数SA_USER_ADMIN.SET_USER_LABELS过程，并在SA_USER_ADMIN.SET_USER_PRIVS程序Oracle数据库，用户名可以是一个Oracle数据库真正应用安全性的用户名。

    见SA_USER_ADMIN.SET_USER_LABELS手续Oracle标签安全管理员指南和SA_USER_ADMIN.SET_USER_PRIVS手续Oracle标签安全管理员指南了解更多信息。

    分配给Real Application Security用户的标签或Oracle Label Security权限在Real Application Security用户会话中强制实施。Oracle Label Security上下文是基于以下Real Application Security会话操作（ ATTACH_SESSION ， SWITCH_USER ， ASSIGN_USER ）和Real Application Security直接登录会话建立的。根据当前Real Application Security会话具有的标签或特权或两者，强制执行Oracle Label Security策略。

    请参阅将应用程序会话附加到传统数据库会话 ， 将应用程序用户分配给匿名应用程序会话 ， 将当前应用程序用户切换到当前应用 程序会话 中的另一个应用程序用户 ，以及在直接登录会话中建立Oracle标签安全上下文以获取更多信息。

  • 预定义的应用程序角色XSCONNECT

    允许授予此角色的用户连接到数据库。换句话说，未授予此预定义角色的用户无法连接到数据库。

    有关详细信息，请参阅常规应用程序角色 ， GRANT_ROLES过程和关于创建直接登录应用程序用户帐户 。

不推荐的功能

以下功能已弃用，将来的版本将不再支持：

  • 对于CREATE_USER过程

    不推荐使用参数STATUS的PASSWORDEXPIRED和LOCKED值。

    有关更多信息，请参见“ CREATE_USER过程 ” 。

  • 对于SET_USER_STATUS过程

    不推荐使用PASSWORDEXPIRED状态值。

    有关更多信息，请参见“ SET_USER_STATUS过程 ” 。

  • 对于SET_PASSWORD过程

    不推荐使用密码类型XS_MD4和XS_O3LOGON 。

    有关更多信息，请参见“ SET_PASSWORD过程 ” 。

  • 对于SET_VERIFIER过程

    不推荐使用验证程序类型XS_SALTED_MD5 ， XS_SHA1 ， XS_SASL_MD5 ， XS_MD5 ， XS_MD4和XS_O3LOGON 。

    有关更多信息，请参见“ SET_VERIFIER过程 ” 。
