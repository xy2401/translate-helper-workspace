4.1 Oracle数据库数据类型

Oracle数据库识别两种数据类型： 内部和外部 。内部数据类型指定Oracle数据库如何在数据库列中存储数据。

Oracle数据库还使用内部数据类型来表示数据库伪列。外部数据类型指定数据在主机变量中的存储方式。

4.1.2.1 CHAR

CHAR行为取决于选项PICX的设置。请参见“ PICX ” 。

4.1.2.2 CHARF

默认情况下，CHARF数据类型表示所有不变的字符主机变量。您使用CHARF数据类型来存储固定长度的字符串。在大多数平台上，CHARF值的最大长度为65535（64K）字节。请参见“ PICX ” 。

在输入上 。Oracle读取为输入主机变量指定的字节数， 不剥离尾随空白，然后存储在目标数据库列中的输入值。

如果输入值超过数据库列的已定义宽度，则Oracle会生成错误。如果输入值是全空的，则存储一串空格。

关于输出 。Oracle返回为输出主机变量指定的字节数，如有必要，返回空白填充，然后将输出值分配给目标主机变量。如果返回NULL，则不会覆盖变量的原始值。

如果输出值长于主机变量的声明长度，Oracle会在将值分配给主变量之前截断该值。如果指示符变量可用，Oracle会将其设置为输出值的原始长度。

4.1.2.3 CHARZ

CHARZ数据类型表示固定长度，以null结尾的字符串。在大多数平台上，CHARZ值的最大长度为65535字节。您通常不需要Pro * COBOL中的此外部类型。

4.1.2.5 DECIMAL

DECIMAL数据类型表示用于计算的压缩十进制数。在COBOL中，主变量必须是带有隐含小数点的带符号COMP-3字段。如果在数据转换期间丢失有效数字，则该值将被截断为声明的长度。

4.1.2.6显示

DISPLAY数据类型表示数字字符数据。DISPLAY数据类型引用COBOL“DISPLAY SIGN LEADING SEPARATE”编号，对于PIC S9（ n ）需要n + 1个字节的存储空间，对于PIC S9（ n ）V9（ d ）需要n + d + 1个字节的存储空间。

4.1.2.7 FLOAT

FLOAT数据类型表示具有小数部分或超过INTEGER数据类型容量的数字。FLOAT涉及COBOL数据类型COMP-1（4字节浮点）和COMP-2（8字节浮点）。

Oracle可以表示比浮点实现更精确的数字，因为Oracle数字的内部格式是十进制的。

注意：在SQL语句中，比较FLOAT值时，请使用SQL函数ROUND，因为FLOAT存储二进制（非十进制）数字;因此，分数不会完全转换。

4.1.2.8 INTEGER

INTEGER数据类型表示没有小数部分的数字。整数是带符号的2字节，4字节或8字节的二进制数。单词中字节的顺序取决于平台。您必须为输入和输出主变量指定长度。在输出时，如果列具有小数部分，则小数点后的数字将被截断。

4.1.2.9长

LONG数据类型表示固定长度的字符串。LONG数据类型与VARCHAR2数据类型类似，只是LONG值的最大长度为2147483647字节（2千兆字节）。

4.1.2.10 LONG RAW

LONG RAW数据类型表示固定长度的二进制数据或字节字符串。LONG RAW值的最大长度为2147483647字节（2千兆字节）。

LONG RAW数据类似于LONG数据，不同之处在于Oracle不会假设LONG RAW数据的含义，并且在将LONG RAW数据从一个系统传输到另一个系统时不进行字符集转换。

4.1.2.11 LONG VARCHAR

LONG VARCHAR数据类型表示可变长度字符串。LONG VARCHAR变量具有4字节长度字段，后跟字符串字段。字符串字段的最大长度为2147483643字节。在EXEC SQL VAR声明， 不包括4个字节的长度字段。

4.1.2.12 LONG VARRAW

LONG VARRAW数据类型表示二进制数据或字节字符串。LONG VARRAW变量有一个4字节长的字段，后跟一个数据字段。数据字段的最大长度为2147483643个字节。在EXEC SQL VAR声明， 不包括4个字节的长度字段。

4.1.2.13号码

NUMBER数据类型表示内部Oracle NUMBER格式，不能用COBOL数据类型表示。

4.1.2.14超越

OVER-PUNCH是COBOL语言的默认签名数字。数字以ASCII或EBCDIC格式保存在基数10中，每个字节的计算机存储器有一位数字。符号保存在其中一个字节的高位半字节中。它被称为OVER-PUNCH，因为该符号在第一个或最后一个字节中被“打孔”。默认符号位置将位于尾随字节上方。PIC S9（n）V9（m）TRAILING或PIC S9（n）V9（m）LEADING用于指定过冲。

4.1.2.15 RAW

RAW数据类型表示固定长度的二进制数据或字节字符串。在大多数平台上，RAW值的最大长度为65535字节。

RAW数据与CHAR数据类似，不同之处在于Oracle不会假定RAW数据的含义，并且在将RAW数据从一个系统传输到另一个系统时不进行字符集转换。

4.1.2.16 ROWID

ROWID数据类型是COBOL中的数据库行标识符。为了支持逻辑和物理ROWID（以及非Oracle表的ROWID），定义了Universal ROWID（UROWID）。对此数据类型使用SQL-ROWID假型（请参阅“ Universal ROWIDs ” ）。

您可以使用VARCHAR2主机变量以可读格式存储ROWID。当您选择ROWID或将ROWID提取到VARCHAR2主机变量时，Oracle会将二进制值转换为18字节的字符串，并以以下格式返回：

BBBBBBBB.RRRR.FFFF

其中BBBBBBBB是数据库文件中的块，RRRR是块中的行（第一行是0），FFFF是数据库文件。这些数字是十六进制的。例如，ROWID：
0000000E.000A.0007
指向第7个数据库文件中第15个块的第11行。

                           
通常，您将ROWID提取到VARCHAR2主变量中，并将主变量与UPDATE或DELETE语句的WHERE子句中的ROWID伪列进行比较。这样，您可以识别游标提取的最新行。有关示例，请参阅“ 模仿当前条款 ” 。
                           

                           
注意：如果您需要完全可移植性或应用程序通过透明网关与非Oracle数据库通信，请在声明VARCHAR2主机变量时指定最大长度为256（而不是18）字节。如果您的应用程序通过Oracle Open Gateway与非Oracle数据源通信，请指定最大长度为256个字节。虽然您可以对其内容不做任何假设，但主机变量在SQL语句中将正常运行。

4.1.2.19 VARCHAR

                     

                        

                           
VARCHAR数据类型表示可变长度字符串。VARCHAR变量具有2字节长度字段，后跟65533字节字符串字段。但是，对于VARCHAR数组元素，字符串字段的最大长度为65530字节。指定VARCHAR变量的长度时，请确保为length字段包含2个字节。对于较长的字符串，请使用LONG VARCHAR数据类型。在EXEC SQL VAR声明， 不包括2个字节的长度字段。

                        

                        
                     

                  
4.1.2.20 VARCHAR2

                     

                        

                           
VARCHAR2数据类型表示可变长度字符串。在大多数平台上，VARCHAR2值的最大长度为65535字节。

                           
指定VARCHAR2（ n ）值的最大长度（以字节为单位），而不是字符。因此，如果VARCHAR2（ n ）变量存储多字节字符，则其最大长度小于n个字符。
                           

                           
在输入上。Oracle读取为输入主机变量指定的字节数，删除任何尾随空白，然后将输入值存储在目标数据库列中。
                           

                           
如果输入值超过数据库列的已定义宽度，则Oracle会生成错误。如果输入值是所有SPACES，Oracle会将其视为NULL。

                           
如果字符值表示有效数字，Oracle可以将字符值转换为NUMBER列值。否则，Oracle会生成错误。

                           
关于输出。Oracle返回为输出主机变量指定的字节数，必要时返回空白填充，然后将输出值分配给目标主机变量。如果返回NULL，则Oracle使用空白填充主机变量。
                           

                           
如果输出值长于主机变量的声明长度，Oracle会在将值分配给主变量之前截断该值。如果指示符变量可用，Oracle会将其设置为输出值的原始长度。

                           
Oracle可以将NUMBER列值转换为字符值。字符主变量的长度决定了精度。如果主变量太短，则使用科学记数法。例如，如果将列值123456789选择为长度为6的主机变量，则Oracle会将值1.2E08返回到主机变量。

CURRVAL以指定的顺序返回当前数字。在引用CURRVAL之前，必须使用NEXTVAL生成序列号。

                           
LEVEL返回树结构中节点的级别编号。根是1级，根的子级是2级，孙子级是3级，依此类推。

                           
LEVEEL用于SELECT CONNECT BY语句，以将表的一些或所有行合并到树结构中。在ORDER BY或GROUP BY子句中，LEVEL隔离树中每个级别的数据。

NEXTVAL以指定的顺序返回下一个数字。创建序列后，您可以使用它为事务处理生成唯一的序列号。在以下示例中，名为partno的序列分配了部件号：
EXEC SQL INSERT INTO PARTS VALUES（PARTNO.NEXTVAL，：DESCRIPTION，：QUANTITY，：PRICE END EXEC。
如果事务生成序列号，则在提交或回滚事务时序列会递增。对NEXTVAL的引用将当前序列号存储在CURRVAL中。

                           
ROWNUM返回一个数字，表示从表中选择行的顺序。选择的第一行的ROWNUM为1，第二行的ROWNUM为2，依此类推。如果SELECT语句包含ORDER BY子句，则在排序完成之前将ROWNUM分配给选定的行。

                           
您可以使用ROWNUM来限制SELECT语句返回的行数。此外，您可以在UPDATE语句中使用ROWNUM为表中的每一行分配唯一值。在WHERE子句中使用ROWNUM不会停止处理SELECT语句;它只是限制了检索的行数。在WHERE子句中唯一有意义的使用ROWNUM是：
...WHERE ROWNUM <常数END-EXEC。
因为只有在检索到行时，ROWNUM的值才会增加。永远不会满足以下搜索条件，因为未检索到前四行：

ANSI日期

                  

                  
                  

                     
ANSI DATE基于DATE ，但不包含时间部分。（因此，它也没有时区。）ANSI DATE遵循DATE数据类型的ANSI规范。将ANSI DATE分配给DATE或timestamp数据类型时，Oracle DATE的时间部分和时间戳设置为零。将DATE或时间戳分配给ANSI DATE ，将忽略时间部分。
                     

                     
建议您使用包含日期和时间的TIMESTAMP数据类型。
                     

                  

                  
                  

                     
TIMESTAMP

                  

                  
                  

                     
TIMESTAMP数据类型是DATE数据类型的扩展。它存储DATE数据类型的年，月和日，加上小时，分钟和秒值。它没有时区。TIMESTAMP数据类型具有以下形式：
TIMESTAMP（小数秒）
其中fractional_seconds_precision （可选）指定SECOND日期时间字段的小数部分中的位数，并且可以是0到9范围内的数字。默认值为6。
                     

                  

                  
                  

                     
TIMESTAMP与时区

                  

                  
                  

                     
TIMESTAMP WITH TIME ZONE （ TSTZ ）是TIMESTAMP的变体，其值包含显式时区位移。时区位移是当地时间与UTC（协调世界时 - 以前的格林威治标准时间）之间的差异（以小时和分钟为单位）。TIMESTAMP WITH TIME ZONE数据类型具有以下形式：
具有时区的TIMESTAMP（fractional_seconds_precision）
其中fractional_seconds_precision可选地指定SECOND日期时间字段的小数部分中的位数，并且可以是0到9范围内的数字。默认值为6。
                     

                     
如果两个TIMESTAMP WITH TIME ZONE值代表UTC中的相同时刻，则它们被视为相同，无论数据中存储的TIME ZONE偏移如何。
                     

                  

                  
                  

                     
TIMESTAMP与当地时区

                  

                  
                  

                     
TIMESTAMP WITH LOCAL TIME ZONE （ TSLTZ ）是另一种变型TIMESTAMP其包括在它的值的时间段的位移。存储的格式与TIMESTAMP格式相同。此类型与TIMESTAMP WITH TIME ZONE不同之处在于，存储在数据库中的数据已标准化为数据库时区，并且时区位移不会存储为列数据的一部分。当用户检索数据时，Oracle会在用户的本地会话时区中返回该数据。
                     

                     
时区位移是当地时间与UTC（协调世界时 - 以前的格林威治标准时间）之间的差异（以小时和分钟为单位）。TIMESTAMP WITH LOCAL TIME ZONE数据类型具有以下形式：
TIMESTAMP（fractional_seconds_precision）与LOCAL TIME ZONE
其中fractional_seconds_precision可选地指定SECOND日期时间字段的小数部分中的位数，并且可以是0到9范围内的数字。默认值为6。
                     

                  

                  
                  

                     
间隔年至月

                  

                  
                  

                     
INTERVAL YEAR TO MONTH使用YEAR和MONTH日期时间字段存储一段时间。INTERVAL YEAR TO MONTH数据类型具有以下形式：
间隔年（年度精度）到月
其中可选的year_precision是YEAR日期时间字段中的位数。year_precision的默认值为2。
                     

                  

                  
                  

                     
间隔第二天

                  

                  
                  

                     
INTERVAL DAY TO SECOND以天，小时，分钟和秒为单位存储一段时间。INTERVAL DAY TO SECOND数据类型具有以下形式：

    • 
                        
  •

工作储存部分。 ...EXEC SQL BEGIN DECLARE SECTION END-EXEC。 01 EMP-NUMBER PIC S9（4）COMP VALUE ZERO。 01 EMP-NAME PIC X（10）VALUE SPACE。 01 SALARY PIC S9（5）V99 COMP-3。EXEC SQL END DECLARE SECTION END-EXEC。 ...程序部门。 ...显示“员工编号？“没有进展。接受EMP-NUMBER。执行SQL SELECT ENAME，SAL INTO：EMP-NAME，：来自EMP的雇佣人员EMPNO =：EMP-NUMBER END-EXEC。计算奖金= SALARY / 10。...

工作储存部分。 ...EXEC SQL BEGIN DECLARE SECTION END-EXEC。 01 EMPNO PIC S9（4）COMP VALUE ZERO。 01 ENAME PIC X（10）超值空间。 01 COMM PIC S9（5）V99 COMP-3。EXEC SQL END DECLARE SECTION END-EXEC。 ...程序部门。 ...EXEC SQL SELECT ENAME，COMM INTO：ENAME，：COMM from EMP WHERE EMPNO =：EMPNO END-EXEC。

4.3.2.2限制

4.5 VARCHAR变量

               

                  
COBOL字符串数据类型是固定长度。但是，Pro * COBOL允许您声明一个名为VARCHAR的可变长度字符串假型。 VARCHAR变量是一种假型，它允许您指定存储在数据库中的数据的确切长度，并指定要传递给数据库的数据的确切长度。

               
4.5.1声明VARCHAR变量

                  

                     
您可以通过将关键字VARYING添加到其声明来定义VARCHAR主机变量，如以下示例所示：
01 ENAME PIC X（15）变化。

                        
注意：

                        
在使用VARYING的定义中不允许PIC N和PIC G.要了解如何在VARCHAR变量中正确使用PIC N和PIC G，请参阅“ 隐式VARCHAR组项 ” 。
                        

                     

                     

                        
注意：

                        
Pro * COBOL为使用VARYING声明的变量名添加4个字符。因此，如果您的COBOL编译器支持变量名长度为n ，则Pro * COBOL支持的变量名称少于四个字符（ n -4）。
                        

                     

                     
在PERFORM和SEARCH语句中使用COBOL VARYING短语来增加下标和索引。不要将它与前面示例中的Pro * COBOL VARYING子句混淆。  
                     

                     
VARCHAR是扩展的Pro * COBOL数据类型或预先声明的组项。例如，Pro * COBOL扩展了VARCHAR声明
01 ENAME PIC X（15）变化。
到具有长度和字符串字段的组项中，如下所示：
01 ENAME。 05 ENAME-LEN PIC S9（4）COMP。 05 ENAME-ARR PIC X（15）。

长度字段（以-LEN为后缀）保存存储在字符串字段中的值的当前长度（以-ARR为后缀）。VARCHAR主机变量声明中的最大长度必须在1到9,999字节的范围内。

                     
使用VARCHAR变量的优点是您可以显式设置和引用长度字段。使用输入主变量，Pro * COBOL读取长度字段的值并使用字符串字段的许多字符。对于输出主变量，Pro * COBOL将长度值设置为存储在字符串字段中的字符串的长度。

                  

               
4.5.2隐式VARCHAR组项

                  

                     
当在命令行上指定预编译器选项VARCHAR = YES时，Pro * COBOL会将某些组项隐式识别为VARCHAR主变量。对于可变长度单字节字符类型，请使用以下结构（以单字节字符表示的长度 ）：
nn data-name-1。49 data-name-2 PIC S9（4）COMP。 49 data-name-3 PIC X（长度）。

nn必须是01到48。

                     
对于可变长度多字节NCHAR字符，请使用这些格式（长度以双字节字符表示）：
nn DATA-NAME-1。49 DATA-NAME-2 PIC S9（4）COMP。 49 DATA-NAME-3 PIC N（ 长度 ）。nn DATA-NAME-1。49 DATA-NAME-2 PIC S9（4）COMP。 49 DATA-NAME-3 PIC G（ 长度 ）。必须将这些组项结构中的基本项声明为级别49，以便Pro * COBOL将它们识别为VARCHAR主变量。

必须为Pro * COBOL指定VARCHAR = YES命令行选项才能识别VARCHAR组项的扩展形式。如果VARCHAR = NO，则任何类似于前面格式的声明将被解释为常规组项。如果VARCHAR = YES且组项声明格式与扩展VARCHAR格式看起来相似（但不相同），则该项将被解释为常规组项而不是VARCHAR组项。例如，如果指定了VARCHAR = YES，则编写以下内容：
01 LASTNAME。 48 LASTNAME-LEN PIC S9（4）COMP。 48 LASTNAME-TEXT PIC X（15）。

然后，由于48级而不是49级用于组项元素，因此该项被解释为常规组项而不是VARCHAR组项。

4.6.1 PIC X的默认值

                  

                     
PIC X变量的默认数据类型是CHARF（在8.0版之前是VARCHAR2。）预编译器命令行选项PICX用于向后兼容。只能在命令行或配置文件中输入PICX。有关详细信息，请参见“ PICX ” 。
                     

                  

               
4.6.2 PICX选项的影响

                  

                     
PICX选项确定Pro * COBOL如何处理字符串中的数据。PICX选项使您的程序可以使用ANSI定长字符串或保持与以前版本的数据库服务器和Pro * COBOL的兼容性。

                     
您必须使用PICX = VARCHAR2（不是默认值）才能获得与8.0之前的Pro * COBOL版本相同的结果。  或者，使用
EXEC SQL varname IS VARCHAR2 END-EXEC。
对于每个变量。

                  

               
4.6.3固定长度字符变量

                  

                     
使用PIC X（ n ）和PIC G（n）以及PIC N（n）数据类型声明固定长度字符变量。这些类型的变量根据其作为输入或输出变量的角色处理字符数据。
                     

                  
4.6.3.1输入

                     

                        
当PICX = VARCHAR2时，程序接口会在将值发送到数据库之前删除尾随空白。如果插入固定长度的CHAR列，Pro * COBOL会将尾随空白重新附加到数据库列的长度。但是，如果插入可变长度的VARCHAR2列，Pro * COBOL永远不会添加空格。

                        
当PICX = CHARF时，永远不会剥离尾随空白。

                        
多字节全局化支持数据的主机输入变量不会被删除尾随的双字节空格。假设length组件是以字符为单位的数据长度，而不是字节。
                        

                        
确保输入值不会被无关字符追踪。通常，这不是问题，因为当一个值被接受或MOVEd到PIC X（ n ）变量时，COBOL会将空白追加到变量的长度。
                        

                        
以下示例说明了这一点：
工作储存部分。 ...EXEC SQL BEGIN DECLARE SECTION END-EXEC。 01员工。 05 EMP-NAME PIC X（10）。05 DEPT-NUMBER PIC S9（4）VALUE 20 COMP。 05 EMP-NUMBER PIC S9（9）VALUE 9999 COMP。 05 JOB-NAME PIC X（8）。...EXEC SQL END DECLARE SECTION END-EXEC。 ...程序部门。 ...显示“员工姓名？“没有进展。接受EMP-NAME。*假设输入名称MILLER * EMP-NAME包含”MILLER“（4个尾随空白）将”SALES“移动到JOB-NAME。* JOB-NAME现在包含”SALES“（ 3个尾随空白）EXEC SQL INSERT INTO EMP（EMPNO，ENAME，DEPTNO，JOB）值（：EMP-NUMBER，：EMP-NAME，：DEPT-NUMBER，：JOB-NAME END-EXEC ....

如果使用PICX = VARCHAR2预编译最后一个示例且目标数据库列为VARCHAR2，程序接口将删除输入上的尾随空白，并将6个字符的字符串“MILLER”和5个字符的字符串“SALES”插入到数据库中。但是，如果目标数据库列为CHAR，则字符串将填充为列的宽度。

                        
如果使用PICX = CHARF预编译最后一个示例并且JOB列定义为CHAR（10），则插入该列的值为“SALES #####”（五个尾随空白）。但是，如果JOB列定义为VARCHAR2（10），则插入的值为“SALES ###”（三个尾随空白），因为主机变量声明为PIC X（8）。这可能不是你想要的，所以要小心。

                     

                  
4.6.3.2关于输出

                     

                        
PICX选项对固定长度字符变量的输出没有影响。当您使用PIC X（ n ）变量作为输出主变量时，Pro * COBOL会将其填空。在我们的示例中，当您的程序从数据库中获取字符串“MILLER”时，EMP-NAME包含值“MILLER ####”（带有四个尾随空格）。可以不加更改地使用此字符串作为另一个SQL语句的输入。

4.6.4.1输入

4.7通用ROWID

               

                  
数据库服务器中使用了两种表组织： 堆表和索引组织表 。
                  

                  
堆表是默认值。这是Oracle之前的所有表中使用的组织。物理行地址（ROWID）是一个永久属性，用于标识堆表中的行。物理ROWID的外部字符格式是base-64编码的18字节字符串。
                  

                  
索引组织表没有物理行地址作为永久标识符。为这些表定义了逻辑 ROWID。当您使用索引组织表中的SELECT ROWID ...语句时，ROWID是一个不透明的结构，包含表的主键，控制信息和可选的物理“猜测”。您可以在包含诸如“WHERE ROWID = ...”的子句的SQL语句中使用此ROWID来从表中检索值。
                  

                  
 通用 ROWID是在Oracle 8.1数据库版本中引入的。Universal ROWID可用于物理ROWID和逻辑ROWID。您可以使用通用ROWID访问堆表或索引组织表中的数据，因为表组织可以更改而不会影响应用程序。用于ROWID的列数据类型是UROWID（ 长度 ），其中length是可选的。
                  

                  
在所有新应用程序中使用通用ROWID。

                  
声明一个通用的ROWID，它使用假型SQL-ROWID，这样：
01 MY-ROWID SQL-ROWID。
通用ROWID的内存使用ALLOCATE语句分配：
EXEC SQL ALLOCATE：MY-ROWID END-EXEC。
在SQL DML语句中使用MY-ROWID，如下所示：
EXEC SQL SELECT ROWID INTO：MY-ROWID from MYTABLE WHERE ...END-EXEC。 ...EXEC SQL UPDATE MYTABLE SET ...WHERE ROWID =：MY-ROWID END-EXEC。 ...
使用FREE指令不再需要内存时释放内存：
EXEC SQL FREE：MY-ROWID END-EXEC。
您还可以选择使用宽度在18到4000之间的字符主机变量作为通用ROWID的主机绑定变量。堆表支持基于字符的通用ROWID，仅用于向后兼容。由于通用ROWID可以是可变长度，因此在选择时可以进行截断。

                  
像这样使用字符变量：
01 MY-ROWID-CHAR PIC X（4000）变化。 ...EXEC SQL ALLOCATE：MY-ROWID-CHAR END-EXEC。 EXEC SQL SELECT ROWID INTO：来自MYTABLE的MY-ROWID-CHAR ......END-EXEC。 ...EXEC SQL UPDATE MYTABLE SET ...WHERE ROWID =：MY-ROWID-CHAR END-EXEC。 ...EXEC SQL FREE：MY-ROWID-CHAR END-EXEC。

4.8全球化支持

               

                  
尽管广泛使用的7位或8位ASCII和EBCDIC字符集足以表示罗马字母，但某些亚洲语言（如日语）包含数千个字符。这些语言需要16位或更多，以表示每个字符。Oracle如何处理这些不同的语言？

4.9.1 NLS_LOCAL = YES限制

表格不允许。使用PIC N或PIC G数据类型声明的主机变量不能是表。

没有主机变量等价。使用EXEC SQL VAR语句无法使多字节NCHAR字符变量等效。
                     

                     
没有动态SQL。 Pro * COBOL中的NCHAR多字节字符串主变量不支持动态SQL。

4.10数据类型转换

               

                  
在预编译时，将为每个主机变量分配外部数据类型。例如，Pro * COBOL将INTEGER外部数据类型分配给PIC S9（ n ）COMP类型的主机变量。在运行时，SQL语句中使用的每个主机变量的数据类型代码都将传递给Oracle。Oracle使用代码在内部和外部数据类型之间进行转换。
                  

                  
在将SELECTed列值分配给输出主机变量之前，Oracle必须将源列的内部数据类型转换为主机变量的数据类型。同样，在将输入主机变量的值分配或比较到列之前，Oracle必须将主机变量的外部数据类型转换为目标列的内部数据类型。

                  
内部和外部数据类型之间的转换遵循通常的数据转换规则。例如，您可以将CHAR值1234转换为PIC S9（4）COMP值。但是，不能将CHAR值123465543（数字太大）或10F（数字不是十进制）转换为PIC S9（4）COMP值。同样，您无法将包含字母字符的PIC X（ n ）值转换为NUMBER值。
                  

                  
主机变量的数据类型必须与数据库列的数据类型兼容。您有责任确保价值可兑换。例如，如果您尝试将字符串值YESTERDAY转换为DATE列值，则会出现错误。内部和外部数据类型之间的转换遵循通常的数据转换规则。例如，您可以将CHAR值1234转换为2字节整数。但是，您不能将CHAR值65543 （数字太大）或10F （数字不是十进制）转换为2字节整数。同样，您不能将包含字母字符的字符串值转换为NUMBER值。
                  

                  
数字转换遵循Oracle初始化文件中Globalization Support参数指定的约定。例如，您的系统可能配置为识别逗号（，）而不是句点（。）作为小数字符。

4.12数据类型等价

               

                  
数据类型等效允许您控制Oracle解释输入数据的方式以及Oracle格式化输出数据的方式。您可以在逐个变量的基础上将等效支持的COBOL数据类型支持到外部数据类型。  
                  

               
4.12.1等同的有用性

                  

                     
数据类型等效在几个方面很有用。例如，假设您要在COBOL程序中使用可变长度字符串。您可以声明PIC X主机变量，然后将其与外部数据类型VARCHAR2等效。

                     
此外，您可以使用数据类型等效来覆盖默认数据类型转换。除非初始化文件中的全局化支持参数另行指定，否则如果在字符主机变量中选择DATE列值，Oracle将返回一个9字节的字符串，格式如下：
DD-MON-YY
但是，如果将字符主机变量等同于DATE外部数据类型，Oracle将以内部格式返回7字节值。

                  

               
4.12.2主机变量等价

                  

                     
默认情况下，Pro * COBOL为每个主机变量分配特定的外部数据类型。您可以通过将主机变量等效为外部数据类型来覆盖默认分配。这称为主变量等效 。

其中数据类型是：
SQL数据类型[（{length | precision，scale }）]

4.12.2.1 VAR语句中的CONVBUFSZ条款

                     

                        
EXEC SQL VAR语句可以有一个可选的CONVBUFSZ子句。您可以指定用于在字符集之间执行指定主机变量转换的运行时库中缓冲区的大小（以字节为单位）。
                        

                        
如果未使用CONVBUFSZ子句，运行时将根据主机变量字符大小（由NLS_LANG确定）与数据库字符集的字符大小之比自动确定缓冲区大小。这有时会导致创建LONG大小的缓冲区。允许数据库只有一个LONG列。如果有多个LONG值，则会引发错误。

                        
为避免此类错误，请使用长度小于LONG的长度。如果字符集转换的结果值超过CONVBUFSZ指定的长度，则Pro * COBOL将返回错误。

4.12.5 RAW和LONG RAW值

                  

                     
在字符主机变量中选择RAW或LONG RAW列值时，Oracle必须将内部二进制值转换为外部字符值。在这种情况下，Oracle将RAW或LONG RAW数据的每个二进制字节作为一对字符返回。每个字符代表半字节的十六进制等效值（半个字节）。例如，Oracle将二进制字节11111111作为字符对“FF”返回。SQL函数RAWTOHEX执行相同的转换。

                     
将字符主机值插入RAW或LONG RAW列时，还需要进行转换。host变量中的每对字符必须表示二进制字节的十六进制等效值。如果字符不表示半字节的十六进制值，则Oracle会发出错误消息。

4.13平台字节序支持

               

                  
Oracle存储的unicode数据（UTF16）总是处于大端。目前，客户端应用程序在不同平台上运行Linux和Windows几乎没有端点表示，Solaris有大端表示。插入或选择UTF16数据时，Pro * Cobol不会在服务器和客户端之间转换字节序形式。这会导致PIC N变量中的UTF16（UCS2）字符串损坏。

                  
可以使用命令行选项picn_endian维护PIC N变量中的平台字节序（Linux和Windows的Little endian，Solaris的Big endian）。
                  

                  

                     
新命令行选项

                  

                  
                  
picn_endian = {BIG | OS}
如果picn_endian = big，则PIC N变量与字符集ID AL16UTF16绑定。

                     
如果picn_endian = os，则PIC N变量与字符集ID UCS2绑定。

                     
此选项的默认值为“big”以保留当前行为。如果NLS_NCHAR不是AL16UTF16，则忽略此选项。

                     
可以使用现有的Pro * Cobol命令行选项设置PIC N变量的字符集形式：
charset_picn = {nchar_charset |值为db_charset}
