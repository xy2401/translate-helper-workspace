在运行DBMS_LOB例程之前提供LOB定位器

DBMS_LOB例程基于LOB 定位器工作 。为了成功完成DBMS_LOB例程， 在调用例程之前 ，必须提供表示存在于数据库表空间或外部文件系统中的LOB的输入定位器。

  • 持久性LOB：首先使用SQL定义包含LOB列的表，然后可以使用SQL初始化或填充这些LOB列中的定位器。

  • 外部 LOB ：定义一个DIRECTORY对象，该对象映射到包含您要访问的外部LOB的有效物理目录。这些文件必须存在，并具有Oracle Server要处理的READ权限。如果您的操作系统使用区分大小写的路径名，请在正确的大小写中指定目录。

一旦定义并创建了LOB，您就可以将LOB定位器SELECT到本地PL / SQL LOB变量中，并将此变量用作DBMS_LOB的输入参数，以访问LOB值。

DBMS_LOB操作中的偏移量和数量参数指南

以下准则适用于DBMS_LOB PL / SQL包中的过程中使用的偏移量和数量参数：

  • 对于所有格式的字符数据，固定宽度和变宽度 - amount和offset参数以字符为单位。这适用于CLOB和NCLOB数据类型的操作。

  • 对于二进制数据， offset和amount参数以字节为单位。这适用于BLOB数据类型的操作。

  • 使用以下程序时：

    • DBMS_LOB.LOADFROMFILE

    • DBMS_LOB.LOADBLOBFROMFILE

    • DBMS_LOB.LOADCLOBFROMFILE

    您不能使用大于您要加载的BFILE大小的值指定amount参数。要使用这些过程加载整个BFILE ，必须指定BFILE的确切大小或最大允许存储限制。

  • 使用DBMS_LOB.READ ， amount参数可以大于数据的大小。该量应小于或等于缓冲区的大小。缓冲区大小限制为32K。

OCI的固定宽度和变宽字符集规则

在OCI中，对于固定宽度的客户端字符集，以下规则适用：

  • CLOB和NCLOB ：offset和amount参数总是以字符为单位

  • BLOB和BFILE s：offset和amount参数总是以字节为单位

以下规则仅适用于宽度不同的客户端字符集：

  • 偏移参数 ：

    无论客户端字符集是否为宽度变化，offset参数始终如下：

    • CLOB和NCLOB ：字符

    • BLOB和BFILE ：以字节为单位

  • 金额参数 ：

    amount参数始终如下：

    • 引用服务器端LOB时：字符

    • 引用客户端缓冲区时：以字节为单位

  • OCILobFileGetLength（） ：

    无论客户端字符集是否为宽度变化，输出长度如下：

    • CLOB和NCLOB ：字符

    • BLOB和BFILE ：以字节为单位

  • OCILobRead2（） ：

    使用不同宽度的客户端字符集， CLOB和NCLOB ：

    • 输入金额以字符为单位。输入量是指从服务器端CLOB或NCLOB读取的字符数。

    • 输出量以字节为单位。输出量表示读入缓冲区bufp字节bufp 。

  • OCILobWrite2（） ：使用不同宽度的客户端字符集， CLOB和NCLOB ：

    • 输入量以字节为单位。输入量是指输入缓冲器bufp的数据字节数。

    • 输出量以字符为单位。输出量是指写入服务器端CLOB或NCLOB的字符数。

其他行动

对于所有其他LOB操作，无论客户端字符集如何， amount参数都是CLOB和NCLOB的字符。这些包括OCILobCopy2() ， OCILobErase2() ， OCILobLoadFromFile2()和OCILobTrim2() 。所有这些操作都是指服务器上的LOB数据量。

OCILobLocator指针分配

在OCI程序中分配OCILobLocator指针时必须特别小心 - 使用“=”赋值运算符。指针赋值创建LOB的浅表副本。指针分配后，源和目标LOB指向相同的数据副本。

这些语义与使用LOB API（例如OCILobAssign()或OCILobLocatorAssign()执行分配不同。使用这些API时，定位器在分配后在逻辑上指向数据的独立副本。

对于临时LOB，在执行指针赋值之前，必须确保通过调用OCIFreeTemporary()释放目标LOB定位器中的任何临时LOB。相反，当使用OCILobLocatorAssign()时，目标LOB定位器变量中的原始临时LOB（如果有）在分配发生之前自动释放。

OCI中的LOB定位器定义和绑定变量

在SQL语句中的define或out-bind变量中重用LOB定位器之前，必须使用OCIFreeTemporary()现有LOB定位器缓冲区中的任何临时LOB。

使用C ++（OCCI）处理LOB

Oracle C ++调用接口（OCCI）是一种用于操作Oracle数据库中的数据的C ++ API。OCCI被组织为一组易于使用的C ++类，它使C ++程序能够连接到数据库，运行SQL语句，在数据库表中插入/更新值，检索查询结果，在数据库中运行存储过程，并访问数据库模式对象的元数据。OCCI还提供了一个无缝接口，可以将用户定义类型的对象作为C ++类实例进行操作。

Oracle C ++调用接口（OCCI）旨在让您可以一起使用OCI和OCCI来构建应用程序。

与JDBC和ODBC相比，OCCI API具有以下优势：

  • OCCI包含比JDBC更多的Oracle功能。 OCCI提供JDBC不提供的OCI的所有功能。

  • OCCI提供编译性能。使用编译的程序，源代码尽可能靠近计算机编写。因为JDBC是解释的 API，所以它无法提供已编译API的性能。使用解释程序时，性能会下降，因为必须将每行代码单独解释为靠近计算机的代码。

  • OCCI通过智能指针提供内存管理。您不必担心管理OCCI对象的内存。这样可以生成性能更高的应用程序代码。

  • OCCI的导航访问使您能够直观地访问对象和调用方法。对象的更改将持续存在而无需编写相应的SQL语句。如果使用客户端缓存，则导航界面的性能优于对象界面。

  • 关于ODBC，OCCI API更易于使用。因为ODBC是基于C语言构建的，所以OCCI具有C ++提供的所有优点。此外，ODBC具有难以学习的声誉。相比之下，OCCI的设计便于使用。

JDBC驱动程序创建空LOB实例，而不进行数据库往返。在以下情况下可以使用空LOB：

  • 设置PreparedStatement API

  • 更新可更新结果集的API

  • STRUCT的属性值

  • ARRAY的元素值

    注意：

    空LOB是特殊标记LOB，但不是真正的LOB值。

JDBC应用程序无法读取或写入从前面的API创建的空LOB。一个如果您的应用程序尝试读取/写入空LOB，则会导致ORA-17098“无效的空lob操作”。

Oracle Data Provider for .NET概述（ODP.NET）

Oracle Data Provider for .NET（ODP.NET）是Oracle数据库的数据提供程序的实现。
