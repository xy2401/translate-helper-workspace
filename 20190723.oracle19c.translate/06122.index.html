兼容性和迁移问题

本节介绍从早期版本的Pro * C / C ++迁移时的兼容性问题。

32位和64位实现之间的兼容性

在支持32位和64位实现的平台上，在链接64位二进制文件之前，必须通过EXEC SQL INCLUDE语句重新预编译包含sqlca的应用程序。对于通过#include预处理程序语句包含sqlca.h的应用程序，必须重新编译以包含64位sqlca.h，然后再重新使用64位二进制文件。

将来，为了支持跨实现生成的代码兼容性，可以在支持32位和64位二进制文件的端口上仅提供sqlca.h的一个版本（64位版本）。

Pro * C / C ++配置文件

需要使用sys_include选项的相应路径更新Pro * C / C ++配置文件precomp / admin / pcscfg.cfg。环境变量可用于指定路径，例如，Unix系统的$ORACLE_HOME和Windows的%ORACLE_HOME% 。您的包含选项也应该适当更新。 

sys_include =（$ ORACLE_HOME / precomp / public，/ usr / include）<pre> sys_include = / usr / lib / gcc / i386-redhat-linux / 4.1.1 / include <pre> include =（$ ORACLE_HOME / precomp / public ）<pre> include = $ ORACLE_HOME / precomp / hdrs

LTYPE = SHORT

设置LTYPE = SHORT选项会导致.lis文件使用详细形式而不是列出整个程序的说明性表单生成。

Pro * C / C ++版本18c，版本18.1中的新功能

以下功能是此版本中的新增功能：

  • 在流量控制器模式下支持Oracle Connection Manager

    Traffic Director Mode中的Oracle Connection Manager是一个代理，位于支持的数据库客户端和数据库实例之间，以提高高可用性，连接多路复用和负载平衡。

早期版本中的新功能

本节列出了以前版本中Pro * C / C ++引入的新功能。

Pro * C / C ++ 12.2生产中的功能

  • 支持长标识符（对象长度为128字节）。在以前的版本中，对象长度限制为30个字节。

  • 支持Oracle Instant Client - Basic Light版本。

  • 新命令行选项trim_password ，用于防止由包含尾随空格的密码字符串导致的身份验证问题。

  • Pro * C / C ++与vc12的兼容性（Visual Studio 2013编译器）。

Pro * C / C ++ 12.1生产中的功能

  • 支持自动增量列

  • 支持32k列

  • 支持按内存预取

  • 支持SQL计划管理（SPM）

  • Pro * C / C ++在回滚期间不再无法设置SQLSTATE，MODE = ANSI且未声明SQLCODE（5891984）

  • 使用PIC N变量的“SELECT INTO”语句时，Pro * C / C ++不再抛出ORA-538976288错误（17189633）

  • 在使用88级初始化变量（20194289）预编译程序时，Pro * C / C ++不再抛出PCB-S-00400错误

  • Pro * C / C ++不再无法解析特定于Solaris平台的关键字__thread（20901503）

  • Oracle Instant Client的Pro * C / C ++演示makefile现在允许用户指定GCC路径（14591784）

  • Pro * C / C ++不再截断字符串，同时将数据从长列提取到长变量中（13589112）

  • Pro * C / C ++ Nvarchar2对象现在已正确映射到nvarchar2数据类型（11653552）

  • Pro * C / C ++现在支持EXEC SQL OBJECT GET构造中的OCIRaw处理（11653512）

  • Pro * C / C ++现在附带通用演示文件而不是OSD文件，旧的未使用文件不再发货（9909411）

  • Pro * C / C ++现在处理冗长数值的解析（9814506）

  • Pro * C / C ++现在接受IS OF子句中的TYPE标记（9535800）

  • Pro * C / C ++现在正确解释SQL语句中单引号之间的换行符（9531904）

  • 处理包含N字符串的SQL语句时，Pro * C / C ++不再存在内存损坏问题（9531638）

  • Pro * C / C ++现在可以在启用解析器统一时识别CURSOR WITH HOLD选项（8436316）

  • Pro * C / C ++现在将正确的“mode”参数值传递给OCIStmtFetch2（）（18509872）

  • Pro * C / C ++现在可以释放分配给列属性的内存（20051833）

  • 在删除表时遇到ORA-942错误后，Pro * C / C ++不再出现ORA-1008错误（20029428）

  • 在统一解析器模式下使用保持语法解析光标时，Pro * C / C ++不再抛出PCC-S-2201错误（20534275）

  • 使用common_parser = No（20808657）编译时，Pro * C / C ++不再抛出PLS-S-201错误

  • 在批量获取行期间，Varchar结构的Len属性不再被破坏（14013076）

  • 预编译长文件名时，Pro * C / C ++不再抛出分段错误（12565588）

在Pro * C / C ++版本12.1中修复了错误

  • 当outline和common_parser都设置为“yes”时，Pro * C / C ++解析器不再失败（12819524）

  • 当预编译器选项'code'设置为'kr_c'时，Pro * C / C ++不再为公共API生成ANSI原型（10250555）

  • 升级到11.2后，XA应用程序不再崩溃（10086495）

  • 来自CHAR（5）的VARRARY（5）的EXEC SQL COLLECTION GET现在表现正常（9531014）

  • 在64位平台上，SQLSQLDAAlloc（）不再返回无效的描述符句柄（9491931）

  • 升级到11g后，Pro * C / C ++不再在HP-UX上崩溃（7365514）
