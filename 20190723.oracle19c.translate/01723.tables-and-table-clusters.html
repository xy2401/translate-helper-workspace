模式对象简介

数据库模式是数据结构的逻辑容器，称为模式对象。模式对象的示例是表和索引。您可以使用SQL创建和操作模式对象。

架构对象依赖关系

某些模式对象引用其他对象，从而创建模式对象依赖项 。

例如，视图包含引用表或视图的查询 ，而PL / SQL子程序则调用其他子程序。如果对象A的定义引用对象B，则A是B上的依赖对象 ，B是A的引用对象 。

Oracle数据库提供了一种自动机制，可确保从属对象始终与其引用的对象保持同步。创建依赖对象时，数据库会跟踪依赖对象与其引用对象之间的依赖关系。当引用的对象以可能影响依赖对象的方式更改时，数据库会将依赖对象标记为无效。例如，如果用户删除表，则不会使用基于删除表的视图。

在依赖对象可用之前，必须针对引用对象的新定义重新编译无效的依赖对象。引用无效依赖对象时自动重新编译。

以下对过程test_proc状态的查询显示它是有效的：

SYS和SYSTEM架构

表格概述

表是Oracle数据库中数据组织的基本单元。

表格描述了一个实体 ，它对于必须记录哪些信息具有重要意义。例如，员工可以是实体。

Oracle数据库表分为以下基本类别：

  • 关系表

    关系表具有简单的列，是最常见的表类型。示例2-1显示了关系表的CREATE TABLE语句。

  • 对象表

    列对应于对象类型的顶级属性。请参见“ 对象表概述 ” 。

您可以创建具有以下组织特征的关系表：

  • 堆组织表不以任何特定顺序存储行。默认情况下， CREATE TABLE语句创建堆组织表。

  • 索引组织表根据主键值对行进行排序。对于某些应用程序，索引组织表可以提高性能并更有效地使用磁盘空间。请参阅“ 索引组织表概述 ” 。

  • 外部表是只读表，其元数据存储在数据库中，但其数据存储在数据库外部。请参阅“ 外部表概述 ” 。

表格是永久的或临时的。永久表定义和数据在会话中保持不变。临时表定义与永久表定义的持续方式相同，但数据仅在事务或会话期间存在 。临时表在必须临时保存结果集的应用程序中很有用，可能是因为结果是通过运行多个操作构建的。

列

表定义包括表名和列集。

列标识表所描述的实体的属性。例如，列employee_id在employees表是指雇员实体的雇员ID属性。

通常，在创建表时为每列提供列名， 数据类型和宽度。例如， employee_id的数据类型为NUMBER(6) ，表示此列只能包含宽度最多6位的数字数据。宽度可以由数据类型预先确定，与DATE 。

虚拟列

表可以包含虚拟列 ，与非虚拟列不同，它不占用磁盘空间。

数据库通过计算一组用户指定的表达式或函数，按需派生虚拟列中的值。例如，虚拟列income可以是salary和commission_pct列的函数。

隐形列

不可见列是用户指定的列，其值仅在按名称显式指定列时可见。您可以向表中添加不可见列而不影响现有应用程序，并在必要时使列可见。

通常，不可见列有助于迁移和发展在线应用程序。用例可能是使用SELECT *语句查询三列表的应用程序。向表中添加第四列会破坏应用程序，该应用程序需要三列数据。添加第四个不可见列使应用程序正常运行。然后，开发人员可以更改应用程序以处理第四列，并在应用程序上线时使列可见。

以下示例创建一个具有不可见列count的表products ，然后使不可见列可见：

CREATE TABLE产品（prod_id INT，count INT INVISIBLE）; ALTER TABLE产品MODIFY（计数可见）;

Oracle数据类型

每列都有一个数据类型 ，它与特定的存储格式，约束和有效的值范围相关联。值的数据类型将固定的一组属性与值相关联。

这些属性使Oracle数据库将一种数据类型的值与另一种数据类型的值区别对待。例如，您可以乘以NUMBER数据类型的值，但不能乘以RAW数据类型的值。

创建表时，必须为其每个列指定数据类型。随后插入列中的每个值都采用列数据类型。

Oracle数据库提供了几种内置数据类型。最常用的数据类型分为以下几类：

  • 字符数据类型

  • 数字数据类型

  • 日期时间数据类型

  • Rowid数据类型

  • 格式模型和数据类型

其他重要类型的内置类型包括原始对象（LOB）和集合。PL / SQL具有常量和变量的数据类型，包括BOOLEAN ，引用类型，复合类型（记录）和用户定义的类型。

字符数据类型

字符数据类型将字母数字数据存储在字符串中。最常见的字符数据类型是VARCHAR2 ，它是存储字符数据的最有效选项。

字节值对应于字符编码方案，通常称为字符集 。数据库字符集在数据库创建时建立。字符集的示例是7位ASCII，EBCDIC和Unicode UTF-8。

字符数据类型的长度语义可以以字节或字符来衡量。将字符串视为字节序列称为字节语义 。这是字符数据类型的默认值。将字符串视为一系列字符称为字符语义 。字符是数据库字符集的代码点。

VARCHAR2和CHAR数据类型
VARCHAR2数据类型存储可变长度字符文字。文字是固定数据值。

例如， 'LILA' ， 'St. George Island'和'101'都是字符文字; 5001是数字文字。字符文字用单引号括起来，以便数据库可以将它们与模式对象名称区分开来。

注意：

本手册可互换地使用术语文本文字 ， 字符文字和字符串 。

使用VARCHAR2列创建表时，指定最大字符串长度。在示例2-1中 ， last_name列的数据类型为VARCHAR2(25) ，这意味着存储在列中的任何名称最多为25个字节。

对于每一行，Oracle数据库将列中的每个值存储为可变长度字段，除非值超过最大长度，在这种情况下，数据库将返回错误。例如，在单字节字符集中，如果为行中的last_name列值输入10个字符，则行表中的列仅存储10个字符（10个字节），而不是25个字符。使用VARCHAR2可以减少空间消耗。

与VARCHAR2 ， CHAR存储固定长度的字符串。使用CHAR列创建表时，该列需要字符串长度。默认值为1个字节。数据库使用空格将值填充到指定的长度。

Oracle数据库使用非填充比较语义比较VARCHAR2值，并使用空白填充比较语义比较CHAR值。

NCHAR和NVARCHAR2数据类型
NCHAR和NVARCHAR2数据类型存储Unicode字符数据。

Unicode是一种通用编码字符集，可以使用单个字符集以任何语言存储信息。NCHAR存储与国家字符集对应的固定长度字符串，而NVARCHAR2存储可变长度字符串。

您在创建数据库时指定国家字符集。NCHAR和NVARCHAR2数据类型的字符集必须是AL16UTF16或UTF8 。两个字符集都使用Unicode编码。

使用NCHAR或NVARCHAR2列创建表时，最大大小始终为字符长度语义。字符长度语义是NCHAR或NVARCHAR2的默认长度语义。

数字数据类型

Oracle数据库数字数据类型存储固定和浮点数，零和无穷大。某些数字类型还存储值，这些值是操作的未定义结果，称为“非数字”或NaN 。

Oracle数据库以可变长度格式存储数字数据。每个值都以科学记数法存储，1个字节用于存储指数。数据库最多使用20个字节来存储尾数 ， 尾数是包含有效数字的浮点数的一部分。Oracle数据库不存储前导零和尾随零。

NUMBER数据类型

NUMBER数据类型存储固定和浮点数。数据库可以存储几乎任何数量的数字。保证此数据可在运行Oracle数据库的不同操作系统之间移植。对于必须存储数字数据的大多数情况，建议使用NUMBER数据类型。

您可以使用NUMBER ( p , s )格式指定定点数，其中p和s指的是以下特征：

  • 精确

    精度指定总位数。如果未指定精度，则列将完全按应用程序提供的值存储值，而不进行任何舍入。

  • 规模

    比例指定从小数点到最低有效数字的位数。正标度计数小数点右侧的数字，包括最低有效数字。负刻度计数小数点左边的数字，但不包括最低有效数字。如果指定不带刻度的精度，如NUMBER(6) ，则刻度为0。

浮点数字

Oracle数据库专门为浮点数提供两种数值数据类型： BINARY_FLOAT和BINARY_DOUBLE 。

这些类型支持NUMBER数据类型提供的所有基本功能。但是，虽然NUMBER使用小数精度，但BINARY_FLOAT和BINARY_DOUBLE使用二进制精度，这可以实现更快的算术计算，并且通常可以降低存储要求。

BINARY_FLOAT和BINARY_DOUBLE是近似数值数据类型。它们存储十进制值的近似表示，而不是精确表示。例如，值0.1不能由BINARY_DOUBLE或BINARY_FLOAT精确表示。它们经常用于科学计算。它们的行为类似于Java和XMLSchema中的数据类型FLOAT和DOUBLE 。

日期数据类型

DATE数据类型存储日期和时间。虽然日期时间可以用字符或数字数据类型表示，但DATE具有特殊的关联属性。

数据库在内部将日期存储为数字。日期存储在每个7字节的固定长度字段中，对应于世纪，年，月，日，小时，分钟和秒。

注意：

日期完全支持算术运算，因此您可以像使用数字一样添加和减去日期。

数据库根据指定的格式模型显示日期。格式模型是一种字符文字，用于描述字符串中日期时间的格式。标准日期格式为DD-MON-RR ，以01-JAN-11的形式显示日期。

RR类似于YY （年份的最后两位数字），但返回值的世纪根据指定的两位数年份和当前年份的最后两位数字而变化。假设在1999年数据库显示01-JAN-11 。如果日期格式使用RR ，则11指定2011 ，而如果格式使用YY ，则11指定1911 。您可以在数据库实例和会话级别更改默认日期格式。

Oracle数据库以24小时格式存储时间 - HH:MI:SS 。如果未输入时间部分，则默认情况下，日期字段中的时间为00:00:00 AM 。在仅限时间的条目中，日期部分默认为当前月份的第一天。

Rowid数据类型

存储在数据库中的每一行都有一个地址。Oracle数据库使用ROWID数据类型来存储数据库中每一行的地址（rowid）。

Rowid分为以下几类：

  • 物理rowid存储堆组织表，表簇以及表和索引分区中的行的地址。

  • 逻辑rowid在索引组织表中存储行的地址。

  • 外部rowid是外部表中的标识符，例如通过网关访问的DB2表。它们不是标准的Oracle数据库rowid。

称为通用rowid或urowid的数据类型支持所有类型的rowid。

使用Rowids
Oracle数据库在内部使用rowid来构建索引。

B树索引是最常见的类型，包含按键划分为范围的有序列表。每个键与一个rowid相关联，该rowid指向相关行的地址以便快速访问。

最终用户和应用程序开发人员也可以将rowid用于几个重要功能：

  • Rowid是访问特定行的最快方法。

  • Rowid提供了查看表的组织方式的功能。

  • Rowid是给定表中行的唯一标识符。

您还可以使用ROWID数据类型创建包含列的表。例如，您可以使用数据类型为ROWID的列定义异常表，以存储违反完整性约束的行的rowid。使用ROWID数据类型定义的列的行为与其他表列类似：可以更新值，依此类推。

ROWID伪列

Oracle数据库中的每个表都有一个名为ROWID的伪列。

伪列的行为类似于表列，但实际上并不存储在表中。您可以从伪列中进行选择，但不能插入，更新或删除它们的值。伪列也类似于没有参数的SQL 函数 。不带参数的函数通常为结果集中的每一行返回相同的值，而伪列通常为每一行返回不同的值。

ROWID伪列的值是表示每行地址的字符串。这些字符串的数据类型为ROWID 。通过执行SELECT或DESCRIBE列出表结构时，此伪列不明显，伪列也不占用空间。但是，可以使用保留字ROWID作为列名称使用SQL查询检索每行的rowid。

完整性约束是一种命名规则，用于限制表中一列或多列的值。

表组织

默认情况下，表被组织为堆，这意味着数据库将行放在最适合的位置而不是用户指定的顺序。因此，堆组织表是无序的行集合。

表压缩

数据库可以使用表压缩来减少表所需的存储量。

压缩可节省磁盘空间，减少数据库缓冲区缓存中的内存使用，并在某些情况下加快查询执行速度。表压缩对数据库应用程序是透明的。

基本表压缩和高级行压缩

基于字典的表压缩为堆组织表提供了良好的压缩比。

Oracle数据库支持以下类型的基于字典的表压缩：

  • 基本表压缩

    此类压缩适用于批量加载操作。数据库不压缩使用传统DML修改的数据。您必须使用直接路径INSERT操作， ALTER TABLE . . .MOVE操作或在线表重新定义以实现基本表压缩。

  • 高级行压缩

    此类压缩适用于OLTP应用程序，并压缩由任何SQL操作操纵的数据。数据库实现了竞争压缩比，同时使应用程序能够在与未压缩表上的DML大致相同的时间内执行DML。

对于前面的压缩类型，数据库以行主格式存储压缩行。一行的所有列都存储在一起，然后是下一行的所有列，依此类推。数据库使用对存储在块开头的符号表的简短引用来替换重复值。因此，数据库需要重新创建未压缩数据的信息存储在数据块本身中。

压缩数据块看起来很像普通数据块。大多数适用于常规数据块的数据库特性和功能也适用于压缩块。

您可以在表空间，表，分区或子分区级别声明压缩。如果在表空间级别指定，则默认情况下会压缩表空间中创建的所有表。

DML和混合列压缩

混合列压缩对不同类型的DML操作中的行锁定有影响。

直接路径载荷和常规插入

将数据加载到使用混合列压缩的表中时，可以使用传统插入或直接路径加载。直接路径加载会锁定整个表，从而降低并发性。

Oracle Database 12c第2版（12.2）增加了对混合列压缩格式的传统阵列插入的支持。传统阵列插件的优点是：

  • 插入的行使用行级锁，这会增加并发性。

  • 自动数据优化（ADO）和热图支持行级策略的混合列压缩。因此，即使在段的其他部分发生DML活动，数据库也可以对符合条件的块使用混合列压缩。

当应用程序使用传统的数组插入时，Oracle Database会在满足以下条件时以压缩单位存储行：

  • 该表存储在ASSM表空间中。

  • 兼容级别为12.2.0.1或更高版本。

  • 表定义满足现有的Hybrid Columnar Compression表约束，包括没有LONG类型的列，也没有行依赖性。

常规插入生成重做和撤消。因此，由传统DML语句创建的压缩单元与DML一起回滚或提交。数据库自动执行索引维护，就像存储在传统数据块中的行一样。

更新和删除

默认情况下，如果对单元中的任何行应用更新或删除，则数据库会锁定压缩单元中的所有行。要避免此问题，可以选择为表启用行级锁定。在这种情况下，数据库仅锁定受更新或删除操作影响的行。

表簇概述

表簇是一组表，它们共享公共列并在相同的块中存储相关数据。

当表被群集时，单个数据块可以包含来自多个表的行。例如，块可以存储来自employees和departments表的行，而不是仅存储来自单个表的行。

群集键是群集表共有的一列或多列。例如， employees和departments表共享department_id列。您在创建表群集时以及在创建添加到表群集的每个表时指定群集密钥。

群集键值是特定行集的群集键列的值。包含相同群集键值的所有数据（例如department_id=20 ）将物理存储在一起。无论多少行的不同表包含该值，每个群集键值仅在群集和群集索引中存储一次。

举个类比，假设一个人力资源经理有两个书柜：一个带有一箱员工文件夹，另一个带有一些部门文件夹。用户经常要求特定部门中所有员工的文件夹。为了使检索更容易，经理重新排列了一个书柜中的所有方框。她按部门ID划分方框。因此，部门20中的雇员的所有文件夹和部门20本身的文件夹都在一个框中;部门100中的员工文件夹和部门100的文件夹位于另一个框中，依此类推。

在主要查询（但未修改）表时，请考虑对表进行聚类，并且经常一起查询或连接表中的记录。由于表簇在相同的数据块中存储不同表的相关行，因此正确使用的表簇与非聚簇表相比具有以下优点：

  • 对于群集表的连接，磁盘I / O会减少。

  • 对于群集表的连接，访问时间得到改善。

  • 存储相关表和索引数据需要较少的存储空间，因为不会为每一行重复存储群集键值。

通常，在以下情况下，群集表不适用：

  • 这些表经常更新。

  • 这些表经常需要全表扫描 。

  • 这些表需要截断。

索引集群是一个表集群，它使用索引来定位数据。群集索引是群集密钥上的B树索引。必须先创建集群索引，然后才能将任何行插入到集群表中。

哈希集群概述

散列集群就像一个索引集群，除了索引键被散列函数替换。不存在单独的集群索引。在散列簇中，数据是索引。

使用索引表或索引集群，Oracle数据库使用存储在单独索引中的键值来查找表行。要在索引表或表集群中查找或存储行，数据库必须至少执行两个I / O：

  • 一个或多个I / O用于在索引中查找或存储键值

  • 另一个用于读取或写入表或表集群中的行的I / O.

要在哈希群集中查找或存储行，Oracle数据库会将哈希函数应用于该行的群集键值。生成的哈希值对应于集群中的数据块，数据库代表已发出的语句读取或写入。

散列是一种存储表数据的可选方法，以提高数据检索的性能。当满足以下条件时，散列簇可能是有益的：

  • 查询表比修改表更频繁。

  • 使用相等条件频繁查询散列键列，例如， WHERE department_id=20 。对于此类查询，将对群集键值进行哈希处理。哈希键值直接指向存储行的磁盘区域。

  • 您可以合理地猜测散列键的数量以及每个键值存储的数据的大小。

哈希群集创建

要创建散列集群，请使用与索引集群相同的CREATE CLUSTER语句，并添加散列键。群集的哈希值的数量取决于哈希键。

群集密钥（与索引群集的密钥一样）是由群集中的表共享的单个列或组合密钥。散列键值是插入到群集键列中的实际值或可能值。例如，如果群集键是department_id ，则哈希键值可以是10,20,30等。

Oracle数据库使用散列函数，该函数接受无限数量的散列键值作为输入，并将它们分类为有限数量的桶。每个存储桶都有一个唯一的数字ID，称为哈希值 。每个散列值映射到存储与散列键值对应的行的块的数据库块地址（部门10,20,30等）。

在以下示例中，可能存在的部门数为100，因此HASHKEYS设置为100 ：

CREATE CLUSTER employees_departments_cluster（department_id NUMBER（4））SIZE 8192 HASHKEYS 100;

创建employees_departments_cluster ，您可以在集群中创建employees和departments表。然后，您可以像在索引集群中一样将数据加载到散列集群中。

哈希群集存储

Oracle数据库为散列集群分配空间的方式与索引集群不同。

在Hash Cluster Creation的示例中， HASHKEYS指定可能存在的部门数，而SIZE指定与每个部门关联的数据的大小。数据库根据以下公式计算存储空间值：

HASHKEYS * SIZE / database_block_size

因此，如果在Hash Cluster Creation中显示的示例中块大小为4096字节，则数据库将至少200个块分配给散列簇。

Oracle数据库不限制可以插入群集的哈希键值的数量。例如，即使HASHKEYS为100 ，也没有什么可以阻止您在departments表中插入200个唯一的departments 。但是，当散列值的数量超过散列键的数量时，散列簇检索的效率会降低。

为了说明检索问题，假设图2-6中的块100完全填满了部门20的行。用户将具有department_id 43的新部门插入departments表中。部门数超过HASHKEYS值，因此数据库将department_id 43散列为散列值77，这与department_id 20使用的散列值相同。将多个输入值散列到相同的输出值称为散列冲突 。

当用户将行插入到部门43的集群中时，数据库不能在块100中存储这些行，该块已满。数据库将块100链接到新的溢出块，比如块200，并将插入的行存储在新块中。块100和200现在都有资格存储任一部门的数据。如图2-7所示，对部门20或43的查询现在需要两个 I / O来检索数据：块100及其相关块200。您可以通过使用不同的HASHKEYS值重新创建群集来解决此问题。

属性聚簇表概述

属性聚簇表是一个堆组织表，它根据用户指定的聚簇指令将数据存储在磁盘上。指令指定单个或多个表中的列。

指令如下：

  • CLUSTERING ...BY LINEAR ORDER指令根据指定的列对表中的数据进行排序。

    当查询限定clustering子句中指定的列的前缀时，请考虑使用BY LINEAR ORDER集群，这是默认设置。例如，如果sh.sales查询通常指定客户ID或客户ID和产品ID，则可以使用线性列顺序cust_id ， prod_id在表中对数据进行集群。

  • CLUSTERING ...BY INTERLEAVED ORDER指令使用特殊算法在一个或多个表中对数据进行排序，类似于Z顺序功能，允许多列I / O减少。

    当查询指定各种列组合时，请考虑使用BY INTERLEAVED ORDER聚类。例如，如果sh.sales查询以不同的顺序指定不同的维度，则可以根据这些维度中的列对sales表中的数据进行集群。

属性群集仅适用于直接路径INSERT操作。对于传统的DML，它被忽略了。

属性聚簇表的优点

属性集群表的主要优点是I / O减少，可以显着降低表扫描的I / O成本和CPU成本。I / O减少发生在区域中，或者通过在磁盘上更紧密的物理接近度来减少物理I / O以获得聚簇值。

属性聚簇表具有以下优点：

  • 您可以根据星型模式中的维度列对事实表进行集群。

    在星型模式中，大多数查询限定维度表而不是事实表，因此按事实表列进行集群无效。Oracle数据库支持对维度表中的列进行群集。

  • I / O减少可能在几种不同的情况下发生：

    • 与Oracle Exadata存储索引，Oracle In-Memory最小/最大修剪或区域映射一起使用时

    • 在OLTP应用程序中，查询限定前缀并使用具有线性顺序的属性集群

    • 在BY INTERLEAVED ORDER聚类的聚类列的子集上

  • 属性聚类可以改善数据压缩，并以这种方式间接提高表扫描成本。

    当相同的值在磁盘上彼此接近时，数据库可以更轻松地压缩它们。

  • Oracle数据库不会产生索引的存储和维护成本。

使用区域减少I / O.

区域是一组连续的数据块，用于存储相关列的最小值和最大值。当SQL语句包含存储在区域中的列的谓词时，数据库会将谓词值与区域中存储的最小值和最大值进行比较，以确定在SQL执行期间要读取的区域。

I / O减少是能够跳过不包含数据库满足查询所需数据的表或索引块。这种减少可以显着降低表扫描的I / O和CPU成本。

区域地图

区域映射是一种独立的访问结构，它将数据块划分为区域。Oracle数据库将每个区域映射实现为一种物化视图 。

只要在表上指定了CLUSTERING ，数据库就会自动在指定的集群列上创建区域映射。区域映射将列的最小值和最大值与属性聚簇表中的连续数据块相关联。属性聚簇表使用区域映射来执行I / O减少。

您可以创建不使用区域映射的属性聚簇表。您还可以创建没有属性群集表的区域映射。例如，您可以在表上创建区域映射，该表的行在一组列上自然排序，例如交易按时间排序的股票交易表。您可以执行DDL语句来创建，删除和维护区域映射。

具有线性排序的属性集群表

表的线性排序方案根据特定顺序中的用户指定属性将行划分为范围。Oracle数据库支持通过主 - 外键关系连接的单个或多个表的线性排序。

例如， sales表将cust_id和prod_id列划分为范围，然后将这些范围聚集在磁盘上。为表指定BY LINEAR ORDER指令时，如果谓词指定了指令中的前缀列或所有列，则可能会发生显着的I / O减少。

假设sales查询通常指定客户ID或客户ID和产品ID的组合。您可以创建属性聚簇表，以便此类查询从I / O减少中受益：

CREATE TABLE销售（prod_id NOT NULL NUMBER，cust_id NOT NULL NUMBER，amount_sold NUMBER（10,2）...）按线性顺序进行聚类（cust_id，prod_id）是在装载数据时使用材料化的ZONEMAP;

对cust_id和prod_id两个列进行限定的prod_id ，或者对cust_id前缀进行I / O减少的查询。仅限prod_id查询不会经历显着的I / O减少，因为prod_id是BY LINEAR ORDER子句的后缀。以下示例显示了数据库如何在表扫描期间减少I / O.

交错排序使用类似于Z顺序的技术 。

交叉排序使数据库能够基于群集列中的任何谓词子集来修剪I / O.交错排序对于数据仓库中的维层次结构非常有用。

与具有线性排序的属性集群表一样，Oracle数据库支持通过主 - 外键关系连接的单个或多个表上的交叉排序。除attribute-clustered表之外的表中的列必须通过外键链接并连接到attribute-clustered表。

大型数据仓库经常以星型模式组织数据。维度表使用父子层次结构，并通过外键连接到事实表 。通过交错顺序对事实表进行聚类使数据库能够使用特殊函数在表扫描期间跳过维列中的值。

例2-10交错排序示例

假设您的数据仓库包含sales事实表及其两个维度表： customers和products 。大多数查询都有customers表层次结构(cust_state_province, cust_city)和产品层次结构(prod_category, prod_subcategory) 。您可以对sales表使用交叉排序，如以下示例中的partial语句所示：

CREATE TABLE销售（prod_id NUMBER NOT NULL，cust_id NUMBER NOT NULL，amount_sold NUMBER（10,2）...）CLUSTERING销售JOIN产品ON（sales.prod_id = products.prod_id）JOIN customers ON（sales.cust_id = customers.cust_id）BY INTERLEAVED ORDER（（products.prod_category，products.prod_subcategory），（customers.cust_state_province，customers.cust_city） ）具有物质化的ZONEMAP;

注意：

BY INTERLEAVED ORDER子句中指定的列不必位于实际维度表中，但必须通过主 - 外键关系连接。

假设应用程序查询联接中的sales ， products和customers表。该查询指定谓词中的customers.prod_category和customers_cust_state_province列，如下所示：

SELECT cust_city，prod_sub_category，SUM（amount_sold）FROM sales，products，customers WHERE sales.prod_id = products.prod_id AND sales.cust_id = customers.cust_id AND customers.prod_category ='Boys'AND customers.cust_state_province ='England  -  Norfolk'GROUP BY cust_city，prod_sub_category;

在前面的查询中， prod_category和cust_state_province列是CREATE TABLE示例中显示的集群定义的一部分。在扫描sales表时，数据库可以查询区域映射并仅访问此区域中的rowid。

临时表的目的

临时表在必须缓冲结果集的应用程序中很有用。

例如，调度应用程序使大学生能够创建可选的学期课程安排。全局临时表中的行表示每个计划。在会话期间，计划数据是私有的。当学生选择计划时，应用程序将所选计划的行移动到永久表。在会话结束时，数据库会自动删除全局临时表中的计划数据。

私有临时表对动态报告应用程序很有用。例如，客户资源管理（CRM）应用程序可能无限期地作为同一用户连接，同时多个会话处于活动状态。每个会话为每个新事务创建一个名为ORA$PTT_crm的专用临时表。应用程序可以为每个会话使用相同的表名，但更改定义。数据和定义仅对会话可见。表定义将持续到事务结束或手动删除表为止。

临时表中的段分配

与永久表一样，全局临时表是在数据字典中静态定义的持久对象。对于专用临时表，元数据仅存在于内存中，但可以驻留在磁盘上的临时表空间中。

对于全局和专用临时表，数据库在会话首次插入数据时分配临时段。在会话中加载数据之前，该表显示为空。对于特定于事务的临时表，数据库在事务结束时释放临时段。对于特定于会话的临时表，数据库会在会话结束时释放临时段。

外部表概述

外部表访问外部源中的数据，就好像此数据位于数据库中的表中一样。

数据可以是提供访问驱动程序的任何格式。您可以使用SQL（串行或并行），PL / SQL和Java来查询外部表。

外部表的目的

当Oracle数据库应用程序必须访问非关系数据时，外部表非常有用。

例如，基于SQL的应用程序可能需要访问其记录采用以下格式的文本文件：

100，Steven，King，SKING，515.123.4567,17-JUN-03，AD_PRES，31944,150,90 101，Neena，Kochhar，NKOCHHAR，515.123.4568,21-SEP-05，AD_VP，17000,100,90 102，Lex，De Haan，LDEHAAN，515.123.4569,13-JAN-01，AD_VP，17000,100,90

您可以创建外部表，将文本文件复制到外部表定义中指定的位置，然后使用SQL查询文本文件中的记录。同样，您可以使用外部表来提供对JSON文档或LOB的只读访问权限。

在数据仓库环境中，外部表对于执行提取，转换和加载（ETL）任务很有价值。例如，外部表使您能够在转换阶段管理数据加载阶段。该技术消除了在数据库内分阶段数据的需要，以便为数据库内的进一步处理做准备。

您可以在虚拟或非虚拟列上对外部表进行分区。此外，您可以创建混合分区表 ，您可以将某些分区指定为内部分区，将某些分区指定为外部分区。外部分区（如内部分区）受益于性能增强，例如分区修剪和分区连接。例如，您可以使用分区外部表来分析存储在Hadoop分布式文件系统（HDFS）或NoSQL数据库中的大量非关系数据。

Oracle为外部表提供以下访问驱动程序：

  • ORACLE_LOADER （默认）

    使用SQL * Loader启用对外部文件的只读访问。您无法使用ORACLE_LOADER驱动程序创建，更新或附加到外部文件。

  • ORACLE_DATAPUMP

    使您可以卸载或加载外部数据。卸载操作从数据库读取数据并将数据插入到外部表中，由一个或多个外部文件表示。创建外部文件后，数据库无法更新或向其附加数据。加载操作读取外部表并将其数据加载到数据库中。

  • ORACLE_HDFS

    允许提取存储在Hadoop分布式文件系统（HDFS）中的数据。

  • ORACLE_HIVE

    允许访问存储在Apache Hive数据库中的数据。源数据可以存储在HDFS，HBase，Cassandra或其他系统中。与其他访问驱动程序不同，您无法指定位置，因为ORACLE_HIVE从外部元数据存储获取位置信息。

外部表创建

在内部，创建外部表意味着在数据字典中创建元数据。与普通表不同，外部表不描述存储在数据库中的数据，也不描述数据如何存储在外部。相反，外部表元数据描述了外部表层必须如何向数据库提供数据。

CREATE TABLE ...ORGANIZATION EXTERNAL语句有两个部分。外部表定义描述了列类型。此定义类似于一个视图，使SQL能够查询外部数据而无需将其加载到数据库中。语句的第二部分将外部数据映射到列。

外部表是只读的，除非使用ORACLE_DATAPUMP访问驱动程序使用CREATE TABLE AS SELECT创建。外部表的限制包括不支持索引列和列对象。

对象表概述

Oracle 对象类型是用户定义的类型，具有名称，属性和方法。对象表是一种特殊的表，其中每一行代表一个对象。对象类型使得可以将诸如客户和采购订单之类的现实世界实体建模为数据库中的对象。

对象类型定义逻辑结构，但不创建存储。以下示例创建名为department_typ的对象类型：

CREATE TYPE department_typ AS OBJECT（d_name VARCHAR2（100），d_address VARCHAR2（200））; /

以下示例创建对象类型department_typ名为departments_obj_t的对象表，然后在表中插入一行。departments_obj_t表的属性（列）是从对象类型的定义派生的。 

CREATE TABLE departments_obj_t OF department_typ; INSERT INTO departments_obj_t VALUES（'hr'，'10 Main St，Sometown，CA'）;

与关系列一样，对象表只能包含一种事物的行，即与表相同的声明类型的对象实例。默认情况下，对象表中的每个行对象都有一个关联的逻辑对象标识符（OID），该标识符在对象表中唯一标识它。对象表的OID列是隐藏列。
