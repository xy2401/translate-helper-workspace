17网络覆盖服务（WCS）支持

本章描述了Web覆盖服务接口标准（WCS）的Open GIS Consortium（OGC）标准的Oracle Spatial and Graph实现，该标准支持检索“覆盖范围”（根据OGC，“地理空间数据的电子编码，是，代表空间和时变现象的数字地理空间信息“）。

Oracle Spatial and Graph实现将称为Web Coverage Service或WCS。

网络覆盖服务（WCS）使地理空间数据的电子检索成为“覆盖范围”。WCS提供数据和描述，查询数据的语法，以及返回可以执行各种操作的数据的能力（可视化，解释，推断等）。WCS支持存储GridCoverage（没有坐标参考系统的GeoRaster对象）和RectifiedGridCoverage（带坐标参考系统的GeoRaster对象和带有仿射变换的地理参考）栅格类型

Oracle Spatial and Graph实现了以下OGC标准。

  • 09-110r4 WCS Core 2.0接口标准 - 核心。

  • 09-146r2 OGC GML应用程序模式 - 覆盖范围。

  • 09-147r3 OGC网络覆盖服务2.0接口标准 - KVP协议绑定扩展 - 勘误表。

  • 09-148r1 OGC Web Coverage Service 2.0接口标准 - XML / POST协议绑定扩展。

  • 09-149r1 OGC Web Coverage Service 2.0接口标准 - XML / SOAP协议绑定扩展。

  • 11-053r1 OGC网络覆盖服务接口标准 - CRS扩展

  • 12-039 OGC网络覆盖服务接口标准 - 扩展扩展。

  • 12-040 OGC网络覆盖服务接口标准 - 范围子集扩展。

  • 12-049 OGC网络覆盖服务接口标准 - 插值扩展。

  • 12-052 OGC WCS 2.0.1勘误表发布说明。

  • 12-100r1 OGC GML应用程序架构 - 覆盖范围 - GeoTIFF覆盖编码配置文件。

Oracle WCS还扩展了OGC标准，以支持所有GDAL支持的图像或栅格功能作为GetCoverage请求中的输出格式。

17.1 Web覆盖服务架构

在Oracle Spatial and Graph中，WCS实现为Java Web应用程序，可以部署在WebLogic 12.1.3或更高版本中。

所需的Java版本是JDK 1.8或更高版本。WCS的此实现打包在$ORACLE_HOME/md/jlib目录中的Spatial Web Services sdows.ear文件中。有关部署此文件的信息，请参阅部署和配置Spatial Web服务 。

WCS实现了三种协议绑定扩展，KVP（HTTP GET），XML / POST（HTTP / POST）和XML / SOAP。它还使用GDAL Java API生成GDAL支持的图像格式。必须在与部署Spatial Web Services的应用程序容器相同的系统上配置GDAL实例。

Oracle数据库中的SDO_WCS包包含初始化WCS模式，将GeoRaster对象发布为WCS coverage以及处理WCS请求的过程。

17.2 WCS的数据库模式

对于Web Coverage Service，本文档引用可以是用户模式和/或WCS模式的Oracle数据库模式。

  • 用户模式是用于存储GeoRaster对象的任何模式。

    在本文档中使用的一些示例中，名为SCOTT的数据库用户的模式是用户模式。

  • WCS模式是具有SDO_WCS_COVERAGE表的任何数据库模式，必须使用WCS管理控制台或SDO_WCS.Init过程创建该表 。覆盖信息存储在WCS模式中。

    在本文档的某些示例中，名为WCS_USER的专门创建的数据库用户的模式是WCS模式。

给定的Oracle数据库模式可以是用户模式，WCS模式，也可以是用户模式和WCS模式。

每个WCS实例都需要在应用程序容器中将WCS架构配置为JDBC数据源。通常，只为特定数据库配置一个WCS架构，但可以配置许多WCS实例以用于同一数据库。

17.3用于WCS的数据库对象

几个表和其他数据库对象用于实现WCS操作。

在WCS架构中：

  • 表SDO_WCS_COVERAGE包含所有已发布coverage的元数据。元数据包括coverage ID，栅格ID和栅格数据表。每行对应一个覆盖范围。

  • 序列SDO_WCS_COVERAGE_ID_SEQ用于生成唯一的覆盖ID值。

在用户架构中：

  • 当处理GetCoverage操作（WCS）请求时涉及重新投影或转换时，表WCS_TEMP_TABLE用作GeoRaster对象的临时存储。 GeoRaster对象保留在此表中，直到发送响应为止，之后将从表中删除对象。

  • 表WCS_TEMP_RDT是WCS_TEMP_TABLE表中GeoRaster对象的栅格数据表。

17.4使用WCS的PL / SQL子程序

SDO_WCS过程和函数使您可以执行包含以下操作的操作。

  • 初始化WCS架构。

  • 在需要重新投影或转换时创建临时表以存储GeoRaster。（CRS扩展或扩展扩展。）。

  • 授予和撤消WCS架构的权限。

  • 发布报道

  • 删除（取消发布）覆盖范围

SDO_WCS.Init创建SDO_WCS_COVERAGE表，使该数据库模式成为WCS模式。

SDO_WCS.PublishCoverage有两种格式。一种格式将GeoRaster对象发布为coverage，将元数据存储在SDO_WCS_COVERAGE表中，并为其分配唯一的coverage ID。另一种格式在指定列中发布所有未发布的GeoRaster对象。

应为每个用户模式执行一次SDO_WCS.CreateTempTable 。此过程创建一个GeoRaster表和一个RDT表，用于在处理GetCoverage操作（WCS）请求时涉及重投影或转换时临时存储GeoRaster对象。

17.5.1配置数据库模式

注意：

如果您计划使用相同的Oracle数据库模式来存储GeoRaster对象并通过WCS访问它们，请跳过本节并转至设置WCS数据源 。

要配置Oracle数据库模式，请按照下列步骤操作：

  1. 创建元数据表。为此，请使用SQL * Plus作为要成为WCS用户的用户（在此示例中，名为WCS_USER的用户）连接到Oracle数据库，并输入以下内容：

    CALL SDO_WCS.init（）;

  2. 以数据库用户身份连接（在此示例中，SCOTT，它拥有一个名为IMAGE的GeoRaster表），用于存储GeoRaster对象。

    connect scott / <password-for-scott> CALL SDO_WCS.createTempTable（）; CALL SDO_WCS.grantPrivilegesToWCS（'IMAGE'，'WCS_USER'）;

  3. 以WCS用户身份连接（在此示例中为WCS_USER），并从包含所需GeoRaster对象的用户表中发布GeoRaster图像（例如，从SCOTT.IMAGES表中的RASTER列）。

    连接wcs_user / <password-for-wcs_user> CALL SDO_WCS.publishCoverage（'SCOTT'，'IMAGE'，'RASTER'）;

17.5.2设置WCS数据源

每个数据库可以有多个WCS模式，每个模式对应于WebLogic中的一个数据源。您还可以在WebLogic Server中为WCS配置多个数据源。可以通过不同的URL访问每个数据源，其中URL的最后部分对应于WebLogic Server中配置的数据源名称。

以下是名为wcsdata1的WLS数据源的示例链接：

http：// localhost：80 / oraclespatial / wcs / wcsdata1？service = WCS＆version = 2.0.1＆request = GetCapabilities

请按照以下步骤配置WCS数据源。

  1. 登录WebLogic Server（WLS）控制台。

  2. 在左窗格中的“ 域结构”下 ，选择“ 数据源”

  3. 单击“ 新建” ，然后单击“ 通用数据源”

  4. 在“创建新的JDBC数据源”页面上，指定：

    1. 名称 ：任何没有空格的名称

    2. JNDI名称 ：从jdbc/开始的任何名称jdbc/

    3. 数据库类型 ： Oracle

  5. 对于数据库驱动程序，选择Oracle的驱动程序（精简）进行实例连接;版本：任何。

  6. 您可以跳过支持全局事务。

  7. 在“连接属性”页面上，指定：

    1. 数据库名称

    2. 主机名

    3. 港口

    4. 数据库用户名 （例如， WCS_USER ）

    5. 密码

  8. 在下一页上，测试此连接。如果连接成功，请为此部署选择适当的服务器名称： AdminServer或受管服务器名称。

17.6 WCS管理控制台

Oracle Spatial Web Services管理控制台是一个单页面应用程序，您可以通过以下格式转到URL来访问该应用程序。

HTTP：// <系统名称>：<端口> / oraclespatial /

在使用任何管理控制台页面之前，请从所有可用数据源名称列表中选择WCS数据源。（当前选择的数据源显示在右上角，您可以随时在其中进行更改。）

WCS管理控制台提供用于配置，测试和诊断问题的屏幕。它有以下标签：

  • 配置文件

  • 发布报道

  • 测试

  • 日志

配置文件选项卡

此选项卡允许管理员修改WCSConfig.xml文件，该文件允许您配置WCS日志记录，GDAL参数和GetCapabilities响应（ ServiceIdentification和ServiceProvider ）。

  • 可以配置日志记录属性：日志级别，日志文件大小限制和日志文件数。

  • 根据文件大小支持日志轮换。通过限制文件数量，可以限制日志文件占用的磁盘空间。日志文件在sdows java属性引用的目录内生成，它们的名称格式为wcs_ n<data source name> .log ，其中wcs_0.log包含最新的日志消息;当它达到文件大小限制时，将删除最旧的文件， wcs_0.log命名所有日志文件，以使wcs_0.log用于下一组日志消息。

  • 日志级别可以是SEVERE ， WARNING ， INFO （默认）， CONFIG ， FINE ， FINER ， FINEST或ALL 。

  • 日志大小限制是文件大小限制，以兆字节为单位（默认值为10）。

  • 日志文件计数是最大日志文件数。

使用gdalParameters元素配置GDAL数据库连接，如下所示：

  • <gdalParameters user =“<user>”password =“！<password>“connectionString =”<db_host：port：sid>“temporaryDirectory =”<directory>“/>

  • 指定的用户必须具有读取作为coverage发布的所有GeoRaster对象的权限。建议此用户与WCS用户相同。

  • 密码必须由服务器加密。您必须添加感叹号（！）密码属性开头的字符，使服务器加密密码。

  • 临时目录应指向GDAL用于生成输出文件的可写目录。示例： /tmp

要验证GDAL是否已正确配置，请转至Oracle Spatial Web Services管理控制台（ http://<host>:<port>/oraclespatial ）中的“关于”选项卡，如果配置正确，则显示GDAL版本。

可以通过取消注释相应的元素并指定所需的信息来配置ServiceIdentification和ServiceProvider ，遵守http://schemas.opengis.net/ows/2.0/owsGetCapabilities.xsd中的XML模式。

单击“ 保存更改”时 ，服务器将应用更改，并且不需要重新启动。

“发布覆盖率”选项卡

此选项卡允许您发布新的coverage。它需要管理员凭据。

您可以通过提供返回SDO_GEORASTER类型的单个列的选择查询来将GeoRaster对象发布为coverage。将发布select查询返回的所有GeoRaster对象。使用以下命令指定选择查询。（为简单起见，您可以从“选择栅格表（架构，表，列）”中选择任何条目，然后单击“ 填充查询参数”以填充这些字段。）

  • 列名 ：From From子句中列出的表中的SDO_GEORASTER列名

  • From子句 ：一个有效的FROM子句，包括模式名称，表名和别名（如果WHERE子句需要）。

  • Where子句 ：可以包含空间操作的有效WHERE子句。如果WHERE子句为空，则SELECT语句将返回指定表的所有行。

此页面显示包含所有GeoRaster对象的HTML表。您可以通过右键单击所需的行来选择发布或取消发布单个GeoRaster对象。

无法在WCS实例中多次发布GeoRaster。

测试标签

此选项卡允许您发送帖子请求。“测试”选项卡最初为空，在这种情况下，您需要发送GetCapabilities请求以填充以下元素：

  • 操作 ：一个HTML select元素，包含在最后一次GetCapabilities响应中发现的所有操作。

  • 覆盖范围 ：HTML选择元素，其中填充了最后一次GetCapabilities响应的内容。它包含收到的GetCapabilities响应的所有Coverage ID。

创建请求 ：使用对指定操作，coverage ID和操作URL的请求填充请求测试区域。

请求 ：文本区域，其内容将在发布请求中发送到操作URL。可以通过单击“ 创建请求”来填充此元素，并且可以根据需要编辑该请求。

操作URL ：发送请求的URL。可以通过单击“ 创建请求”来填充此元素，并且可以根据需要编辑该请求。

发送请求 ：使用请求的内容向操作URL发送HTTP发布请求。 HTTP post请求的响应将显示在Response中。

响应 ：填充了“发送请求”操作的响应的文本区域。

记录选项卡

此选项卡允许您可视化和下载WCS日志文件。日志文件在sdows java属性引用的目录中生成。使用“日志”选项卡需要管理员凭据

所有Oracle WCS日志文件都具有wcs_ n .log形式的文件名，其中n是连续编号，对于最新的日志文件， n为0（零）。

此选项卡显示wcs_<data_source_name>_0.log文件的内容，该文件包含WCS服务器生成的最新日志消息。但是，您可以选择其他日志文件以查看其内容。您还可以刷新显示以包括自页面加载或上次刷新以来生成的新日志消息。

下载允许您以zip格式下载所选的日志文件。

17.7 WCS的Oracle实现扩展

Oracle WCS扩展在WCS请求中的wcs：Extension元素内定义可选元素，以便您控制请求处理的以下方面。

  • GetCoverage请求中的GDAL CreationOption

  • GetCoverage请求中的CompressResponseFile

GetCoverage请求支持GDAL安装中包含的所有格式。您可以在Capabilities XML文档中查看完整的格式列表及其名称。

GetCoverage请求中的GDAL CreationOption

此元素在GetCoverage请求上向GDAL发送-co参数。每个GDAL驱动程序都定义自己的创建选项参数。

例子：

  • XML请求：

    <wcs：GetCoverage xmlns：wcs =“http://www.opengis.net/wcs/2.0”xmlns：gml =“http://www.opengis.net/gml/3.2”xmlns：xsi =“http：/ /www.w3.org/2001/XMLSchema-instance”的xsi：的schemaLocation = “http://www.opengis.net/wcs/2.0 http://schemas.opengis.net/wcs/2.0/wcsAll.xsd” 的xmlns ：ora =“http://www.oracle.com/spatial/wcs” service =“WCS”version =“2.0.1”> <wcs：Extension> <ora：CreationOption> WORLDFILE = YES </ ora：CreationOption> </ wcs：Extension> <wcs：CoverageId> C0005 </ wcs：CoverageId> <wcs：format> image / jpeg </ wcs：format> </ wcs：GetCoverage>

  • KVP请求：

    http：//.../oraclespatial/wcs？service = WCS＆version = 2.0.1＆request = GetCoverage＆format = image / jpeg＆coverageId = C0005＆ CO = WORLDFILE = YES

GetCoverage请求中的CompressResponseFile

此元素将生成的图像压缩为zip文件。

例子：

  • XML请求：

    <wcs：GetCoverage xmlns：wcs =“http://www.opengis.net/wcs/2.0”xmlns：gml =“http://www.opengis.net/gml/3.2”xmlns：xsi =“http：/ /www.w3.org/2001/XMLSchema-instance”的xsi：的schemaLocation = “http://www.opengis.net/wcs/2.0 http://schemas.opengis.net/wcs/2.0/wcsAll.xsd” 的xmlns ：ora =“http://www.oracle.com/spatial/wcs” service =“WCS”version =“2.0.1”> <wcs：Extension> <ora：CompressResponseFile> true </ ora：CompressResponseFile> </ wcs：Extension> <wcs：CoverageId> C0005 </ wcs：CoverageId> <wcs：format> image / jpeg </ wcs：format> </ wcs：GetCoverage>

  • KVP请求：

    http：//.../oraclespatial/wcs？service = WCS＆version = 2.0.1＆request = GetCoverage＆format = image / jpeg＆coverageId = C0005＆ COMPRESS = YES

17.8 WCS操作：使用XML示例的请求和响应

WCS提供三个主要操作，每个操作都有一个请求和响应格式。

当客户端执行任何WCS请求序列时，它应首先向服务器发出GetCapabilities请求以获取可用数据的最新列表。然后，它可以发出DescribeCoverage请求以查找有关所提供的特定覆盖范围的更多详细信息。要检索coverage或部分coverage，客户端会发出GetCoverage请求。

  • GetCapabilities操作（WCS）
  • DescribeCoverage Operation（WCS）
  • GetCoverage操作（WCS）

父主题： Web Coverage Service（WCS）支持

17.8.1 GetCapabilities操作（WCS）

GetCapabilities操作允许WCS客户端检索WCS服务器提供的服务和覆盖元数据。

所有WCS服务器必须为GetCapabilities请求实现KVP协议。用户通过使用KVP协议（HTTP GET请求）向URL发送GetCapabilities请求，开始与WCS服务器交互。例如：

http：// host：port / oraclespatial / wcs / <数据源名称> ？服务= WCS＆请求=获得性能

GetCapabilities操作返回描述服务的XML文档以及客户端可以请求的coverage的简要描述。客户端通常会运行GetCapabilities操作并缓存其结果以在整个会话中使用，或者将其重用于多个会话。

GetCapabilities响应包括以下内容：

  • WCS响应版。如果请求未指定所需的响应版本，则服务器返回支持的最新版本。

  • ServiceIdentification的配置文件列表标识OGC接口标准一致性类。

  • 在操作元件OperationsMetadata包含每个协议的每个WCS操作的URL。每个WCS协议的每个WCS操作可能具有不同的URL。

  • formatSupported在元件ServiceMetadata列表中的所有可用的输出格式由GetCoverage请求。此列表包括配置时支持GDAL的格式。例如：

    <wcs：ServiceMetadata> <wcs：formatSupported> image / tiff </ wcs：formatSupported> <wcs：formatSupported> image / jp2 </ wcs：formatSupported> <wcs：formatSupported> application / x-ogc-nitf </ wcs：formatSupported > <wcs：formatSupported> application / x-ogc-aaigrid </ wcs：formatSupported> <wcs：formatSupported> image / png </ wcs：formatSupported> <wcs：formatSupported> image / jpeg </ wcs：formatSupported> </ wcs ：ServiceMetadata>

  • 扩展元素允许WCS扩展标准定义其各自的额外服务元数据。此元素包括16.8.2 12-039 OGC Web覆盖服务接口标准 - 扩展扩展元素，如：最近邻居，双线性，双二次，......它还包括16.8.1 11-053r1 OGC Web覆盖服务接口标准 - CRS扩展包括crsSupported元素。

  • 此服务器提供的coverage列表包括一个coverage ID，它是DescribeCoverage和GetCoverage操作请求中使用的唯一标识符以及coverage子类型。

17.8.2 DescribeCoverage Operation（WCS）

DescribeCoverage操作允许客户端请求WCS服务器提供的一个或多个coverage的详细元数据，并提供对域和范围集中预期的数据量的估计。DescribeCoverage请求提供覆盖标识符列表，并提示服务器为每个标识符返回相应coverage的描述。以下是XML / POST DescribeCoverage请求示例：

<wcs：DescribeCoverage xmlns：xsi =“http://www.w3.org/2001/XMLSchema-instance”xmlns：wcs =“http://www.opengis.net/wcs/2.0”xsi：schemaLocation =“http ：//schemas.opengis.net/wcs/2.0 ../wcsAll.xsd“service =”WCS“version =”2.0.1“> <wcs：CoverageId> C0001 </ wcs：CoverageId> <wcs：CoverageId> C0002 </ wcs：CoverageId> ...</ WCS：DescribeCoverage>

DescribeCoverage响应包含一个coverage元数据列表，一个用于请求中传递的每个coverage标识符。 Coverage元数据是一个XML文档，类型为gml:Grid for GridCoverages ，类型为gml:rectifiedGrid for RectifiedGridCoverages 。这些文档的xsd模式可以在http://schemas.opengis.net/wcs/2.0/中找到。

gml:Grid元素隐式定义网格，网格是由两组或更多组曲线组成的网络，其中每组的成员以算法的方式与其他组的成员相交。网格内的感兴趣区域以其gml:limits给出，其是矩形区域的对角相对角的网格坐标。 gml:axisLabels提供了网格轴的标签列表（ gml:axisName已被弃用）。 gml:dimension指定网格的维度。

gml:limits元素包含一个gml:GridEnvelope 。包络的gml:low和gml:high属性元素是整数列表，它们是坐标元组。坐标被测量为沿着每个轴的栅格原点与“矩形”感兴趣区域的对角相对角的偏移。

整流网格是网格，在网格坐标和外部坐标参考系统的坐标之间存在仿射变换。它通过指定网格“原点”的位置（在某些几何空间中）和指定帖子位置的矢量来定义。

请注意，网格限制（后索引）和轴名称属性是从gml:GridType继承的，并且gml:RectifiedGrid添加了gml:origin属性（包含或引用gml:Point ）和一组gml:offsetVector属性。

17.8.3 GetCoverage操作（WCS）

GetCoverage操作通常在GetCapabilities和DescribeCoverage操作响应显示允许的请求和可用的数据之后运行。GetCoverage操作返回以众所周知的coverage格式编码的coverage（即，一组地理位置的值或属性）。

GetCoverage请求提示WCS服务处理从服务的产品中选择的特定覆盖范围并返回派生的覆盖范围。

WCS核心标准定义了域子集化操作，该操作从指定请求包络（“边界框”）内的覆盖范围传递相对于覆盖范围包络的所有数据 - 更准确地说，是请求包络与覆盖范围包络的交集。

域子集化细分为修剪和切片 。修剪操作标识尺寸以及下限和上限（两者都必须位于coverage的域内）并提供覆盖范围，其范围在指定的维度中减少到这些新的更窄的限制。生成的coverage的维度与输入coverage的维度相同。以下是DimensionTrim元素的示例：

<wcs：DimensionTrim> <wcs：Dimension> N </ wcs：Dimension> <wcs：TrimLow> 8.16270027015798 </ wcs：TrimLow> <wcs：TrimHigh> 8.34362402047258 </ wcs：TrimHigh> </ wcs：DimensionTrim>

域切片操作接收维度和位置（必须位于coverage的域内）并提供覆盖率，该覆盖率是在指定的切割位置获得的提供的覆盖率的切片。与原始覆盖范围相比，生成的覆盖范围减小了一个。

修剪和切片都可以在请求中组合，也可以根据需要组合在一起。但是，在任何请求中，每个维度最多可以应用一个操作。以下是DimensionSlice元素的示例：

<wcs：DimensionSlice> <wcs：Dimension> N </ wcs：Dimension> <wcs：SlicePoint> 8.16270027015798 </ wcs：SlicePoint> </ wcs：DimensionSlice>

将返回coverage的编码格式由format和mediaType elelemts的组合指定。支持的格式是服务器的Capabilities文档中列出的格式，如果配置了GDAL，则默认为application/gml+xml或image/jpeg 。例如：

<wcs：format> image / jpeg </ wcs：format> <wcs：mediaType> multipart / related </ wcs：mediaType>

本主题介绍了一些错误消息以及如何处理它们。

“未找到DataSource jdbc / wcs_admin_ds”GetCapabilities响应错误消息

OWS错误响应可能如下所示（为了便于阅读，此处重新格式化）：

<？xml version ='1.0'coding ='UTF-8'？> <ows：ExceptionReport xmlns：ows =“http://www.opengis.net/ows/2.0”xmlns：xsi =“http://www.w3.org/2001/XMLSchema-instance”xsi：schemaLocation =“ http://www.opengis.net/ows/2.0 http://schemas.opengis.net/ows/2.0/owsExceptionReport.xsd“version =”2.0.1“> <ows：异常exceptionCode =”NoApplicableCode“locator = “未找到DataSource jdbc / wcs_admin_ds”/> </ ows：ExceptionReport>

这意味着未配置WCS数据源或WebLogic Server无法连接到数据库。设置WCS数据源以获取有关配置WCS数据源的信息。

每次WebLogic Server启动时，WCS日志文件中都会显示“INFO：未找到GDAL”消息

如果不需要GDAL支持的格式，则可以忽略此消息。此消息表示未在WebLogic Server库中找到gdal.jar或未正确配置LD_LIBRARY_PATH。

  • 如果是java.lang.NoClassDefFoundError: org/gdal/gdal/gdal出现在WCS日志中，然后gdal.jar文件。

  • 如果是java.lang.UnsatisfiedLinkError: org.gdal.gdal.gdalJNI.GetDriverCount()I出现在WCS日志中，然后在LD_LIBRARY_PATH中找不到libgdal.so文件。

有关更多信息，请参阅为WCS服务器配置GDAL

GDALParameter错误响应

如果未正确配置GDAL，则会生成如下错误消息：

<？xml version ='1.0'coding ='UTF-8'？> <ows：ExceptionReport xmlns：ows =“http://www.opengis.net/ows/2.0”xmlns：xsi =“http://www.w3.org/2001/XMLSchema-instance”xsi：schemaLocation =“ http://www.opengis.net/ows/2.0 http://schemas.opengis.net/ows/2.0/owsExceptionReport.xsd“version =”2.0.1“> <ows：异常exceptionCode =”NoApplicableCode“locator = “GDALParameters”> <ows：ExceptionText> ... </ ows：ExceptionText> </ ows：Exception> </ ows：ExceptionReport>

严重：发现覆盖率java.sql时出错。SQLRecoverableException：没有更多数据要从套接字读取

WCS管理控制台“日志”选项卡上的“读取日志文件时出错”消息

如果经过身份验证的用户在给定的时间段内未与WCS服务器管理控制台交互，则浏览器中的用户会话可能已超时。在这种情况下，请刷新浏览器显示，然后重新进行身份验证以获取新会话。
