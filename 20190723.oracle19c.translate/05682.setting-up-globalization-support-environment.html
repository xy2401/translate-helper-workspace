3.1设置NLS参数

NLS（本地语言支持）参数确定客户端和服务器上的特定于语言环境的行为。可以通过以下方式指定NLS参数：

  • 作为服务器上的初始化参数

    您可以在初始化参数文件中包含参数以指定默认会话NLS环境。这些设置对客户端没有影响;他们只控制服务器的行为。例如：

    NLS_TERRITORY =“捷克共和国”

  • 作为客户端上的环境变量

    您可以使用可能与平台相关的NLS环境变量来指定客户端的依赖于区域设置的行为，还可以覆盖初始化参数文件中为会话设置的缺省值。例如，在UNIX系统上：

    ％setenv NLS_SORT FRENCH

  • 使用ALTER SESSION语句

    您可以使用在ALTER SESSION语句中设置的NLS参数来覆盖在初始化参数文件中为会话设置的默认值，或者由客户端使用环境变量设置的默认值。 

    SQL> ALTER SESSION SET NLS_SORT = FRENCH;

    也可以看看：

    有关ALTER SESSION语句的更多信息，请参见Oracle数据库SQL语言参考

  • 在SQL函数中

    您可以显式使用NLS参数来硬编码SQL函数中的NLS行为。此实践将覆盖在初始化参数文件中为会话设置的默认值，为具有环境变量的客户端设置的默认值，或者通过ALTER SESSION语句为会话设置的默认值。例如：

    TO_CHAR（hiredate，'DD / MON / YYYY'，'nls_date_language = FRENCH'）

    也可以看看：

    有关SQL函数的更多信息，包括TO_CHAR函数，请参见Oracle数据库SQL语言参考

3.2使用NLS_LANG环境变量选择区域设置

区域设置是运行系统或程序的文化和文化环境。设置NLS_LANG环境参数是指定Oracle数据库软件的区域设置行为的最简单方法。它设置客户端应用程序和数据库服务器使用的语言和区域。它还设置客户端的字符集，该字符集是客户端程序输入或显示的数据的字符集。

NLS_LANG在UNIX平台上设置为环境变量。NLS_LANG在Windows平台上的注册表中设置。

NLS_LANG参数有三个组件：语言，区域和字符集。使用以下格式指定它，包括标点符号：

NLS_LANG = language_territory.charset

例如，如果Oracle Universal Installer未填充NLS_LANG ，则其默认值为AMERICAN_AMERICA.US7ASCII 。语言为AMERICAN ，领域为AMERICA ，字符集为US7ASCII 。 NLS_LANG和其他NLS参数中的值不区分大小写。

NLS_LANG参数的每个组件控制全球化支持功能子集的操作：

  • 语言

    指定约定，例如用于Oracle数据库消息的语言，排序，日期名称和月份名称。每种受支持的语言都有唯一的名称;例如， AMERICAN ， FRENCH或GERMAN 。 language参数指定区域和字符集参数的默认值。如果未指定语言，则该值默认为AMERICAN 。

  • 领土

    指定约定，例如默认日期，货币和数字格式。每个支持的区域都有一个唯一的名称例如， AMERICA ， FRANCE或CANADA 。如果未指定区域，则该值将从语言值派生。

  • 字符集

    指定客户端应用程序使用的字符集（通常是与用户的终端字符集或OS字符集对应的Oracle数据库字符集）。每个支持的字符集都有一个独特的首字母缩写词，例如US7ASCII ， WE8ISO8859P1 ， WE8DEC ， WE8MSWIN1252或JA16EUC 。每种语言都有一个与之关联的默认字符集。

    注意：

    NLS_LANG定义的所有组件都是可选的;任何未指定的项目都使用其默认值。如果指定了区域或字符集，则必须包含前面的分隔符（区域的下划线（_），字符集的句点（。））。否则，该值将被解析为语言名称。

    例如，要仅设置NLS_LANG的区域部分，请使用以下格式： NLS_LANG=_JAPAN

NLS_LANG的三个组件可以在许多组合中指定，如以下示例所示：

NLS_LANG = AMERICAN_AMERICA.WE8MSWIN1252 NLS_LANG = FRENCH_CANADA.WE8ISO8859P1 NLS_LANG = JAPANESE_JAPAN.JA16EUC

请注意，可以设置不合逻辑的组合，但不能正常工作。例如，以下规范尝试使用西欧字符集来支持日语：

NLS_LANG = JAPANESE_JAPAN.WE8ISO8859P1

由于WE8ISO8859P1字符集不支持任何日语字符，因此如果对NLS_LANG使用此定义，则无法存储或显示日语数据。

3.2.1指定NLS_LANG的值

在UNIX操作系统C-shell会话中，您可以通过输入类似于以下示例的语句来指定NLS_LANG的值：

％setenv NLS_LANG FRENCH_FRANCE.WE8ISO8859P1

由于NLS_LANG是环境变量，因此客户端应用程序在启动时会读取它。当客户端连接到数据库服务器时，客户端将NLS_LANG定义的信息传递给服务器。

以下示例显示了NLS_LANG参数如何影响日期和数字格式。

示例3-1将NLS_LANG设置为American_America。WE8ISO8859P1

设置NLS_LANG以使语言为AMERICAN ，区域为AMERICA ，Oracle数据库字符集为WE8ISO8859P1 ：

％setenv NLS_LANG American_America。WE8ISO8859P1

输入SELECT语句：

SQL> SELECT last_name，hire_date，ROUND（薪水/ 8,2）薪水来自员工;

您应该看到类似于以下输出的结果：

LAST_NAME HIRE_DATE SALARY ------------------------- --------- ---------- ...Sciarra 30-SEP-05 962.5 Urman 07-MAR-06 975 Popp 07-DEC-07 862.5 ......

示例3-2将NLS_LANG设置为French_France。WE8ISO8859P1

设置NLS_LANG以使语言为FRENCH ，区域为FRANCE ，Oracle数据库字符集为WE8ISO8859P1 ：

％setenv NLS_LANG French_France。WE8ISO8859P1

然后， 例3-1中显示的查询返回以下输出：

LAST_NAME HIRE_DATE SALARY -------------- --------- --------- ...Sciarra 30/09/05 962,5 Urman 07/03/06 975 Popp 07/12/07 862,5 ...

请注意，日期格式和数字格式已更改。数字没有改变，因为基础数据是相同的。

3.2.2覆盖语言和地域规范

NLS_LANG参数设置服务器会话（例如，SQL命令执行）和客户端应用程序（例如，Oracle数据库工具中的显示格式）使用的语言和区域环境。使用此参数可确保数据库和客户端应用程序的语言环境自动相同。

NLS_LANG参数的语言和区域组件确定其他详细NLS参数的默认值，例如日期格式，数字字符和语言排序。如果已设置NLS_LANG参数，则可以在客户端环境中设置每个详细参数以覆盖默认值。

如果未设置NLS_LANG参数，则服务器会话环境将保持初始化，其值为NLS_LANGUAGE ， NLS_TERRITORY和初始化参数文件中的其他NLS实例参数。您可以修改这些参数并重新启动实例以更改默认值。

您可能希望在会话期间动态修改NLS环境。为此，您可以使用ALTER SESSION语句更改NLS_LANGUAGE ， NLS_TERRITORY和其他NLS参数。

注意：

您无法使用ALTER SESSION语句修改客户端字符集的设置。

ALTER SESSION语句仅修改会话环境。除非客户端显式检索新设置并修改其本地环境，否则不会修改本地客户端NLS环境。

3.2.4 NLS_LANG设置是否应与数据库字符集匹配？

该 NLS_LANG字符集应反映客户端操作系统字符集的设置。例如，如果数据库字符集是AL32UTF8且客户端在Windows操作系统上运行，则不应将AL32UTF8设置为NLS_LANG参数中的客户端字符集，因为没有UTF-8 WIN32客户端。相反， NLS_LANG设置应该反映客户端的代码页。例如，在英文Windows客户端上，代码页为1252。NLS_LANG的适当设置是AMERICAN_AMERICA.WE8MSWIN1252 。

正确设置NLS_LANG可以正确地从客户端操作系统字符集转换为数据库字符集。当这些设置相同时，Oracle数据库假定发送或接收的数据编码在与数据库字符集相同的字符集中，因此可能无法执行字符集验证或转换。如果客户端代码页和数据库字符集不同并且需要转换，则可能导致数据损坏。

3.3字符集参数

Oracle提供了一个环境变量NLS_OS_CHARSET ，用于处理客户端操作系统字符集与Oracle NLS客户端字符集不同的情况。

3.3.1 NLS_OS_CHARSET环境变量

如果客户端操作系统字符集与NLS_LANG环境变量指定的Oracle NLS客户端字符集不同，则应在Oracle客户端安装上设置NLS_OS_CHARSET环境变量。客户端OS字符集是用于表示OS字段中的字符的字符集，如机器名，程序可执行文件名和登录的用户名。在UNIX平台上，这通常是LANG环境变量或LC_ALL环境变量中指定的字符集。设置NLS_OS_CHARSET的示例是，如果在Linux客户端中LANG或LC_ALL中指定的区域设置字符集可以是zh_CN （简体中文），并且在NLS_LANG指定的Oracle客户端应用程序字符集可以是UTF8 。在这种情况下，必须将NLS_OS_CHARSET变量设置为等效的Oracle charset ZHT16GBK 。

必须将NLS_OS_CHARSET环境变量设置为与客户端OS字符集对应的Oracle字符集名称。

如果NLS_LANG对应于OS字符集， NLS_OS_CHARSET不需要设置NLS_OS_CHARSET 。NLS_OS_CHARSET不需要设置，在Windows平台上会被忽略。

NLS_LANGUAGE指定以下会话特征的默认约定：

  • 服务器消息的语言

  • 日期和月份名称及其缩写的语言（在SQL函数TO_CHAR和TO_DATE指定）

NLS_LANGUAGE =法语

请考虑以下服务器消息：

ORA-00942：表或视图不存在

当语言为法语时，服务器消息显示如下：

ORA-00942：表ou vin inexistante

服务器使用的消息存储在二进制格式文件中，这些文件位于$ORACLE_HOME/ product_name /mesg目录中，或者等效于操作系统。使用以下文件名约定，可以存在这些文件的多个版本，每个受支持的语言一个版本：

< product_id > < language_abbrev >。MSB

例如，包含法语服务器消息的文件称为oraf.msb ，因为ORA是产品ID（ <product_id> ）， F是法语的语言缩写（ <language_abbrev> ）。product_name是rdbms ，因此它位于$ORACLE_HOME/rdbms/mesg目录中。

如果NLS_LANG在客户环境中指定，则值NLS_LANGUAGE初始化参数文件在连接时覆盖。

消息存储在这些文件中的一个特定字符集中，具体取决于语言和操作系统。如果此字符集与数据库字符集不同，则消息文本将自动转换为数据库字符集。如果必要，如果客户端字符集与数据库字符集不同，则将其转换为客户端字符集。因此，消息在用户终端上正确显示，受字符集转换的限制。

实际安装的特定于语言的二进制消息文件取决于用户在产品安装期间指定的语言。仅安装英语二进制消息文件和用户指定的特定于语言的二进制消息文件。

NLS_LANGUAGE的默认值可能特定于操作系统。您可以通过更改初始化参数文件中的值然后重新启动实例来更改NLS_LANGUAGE参数。

此参数指定小数字符和组分隔符。例如，组分隔符是将整数组分隔为显示数千和数百万的字符。组分隔符是G编号返回的字符格式掩码。十进制字符分隔数字的整数和小数部分。设置NLS_NUMERIC_CHARACTERS会覆盖从NLS_TERRITORY设置派生的值。

示例3-18设置NLS_NUMERIC_CHARACTERS

3.9.5欧元的Oracle数据库支持

欧洲货币联盟（EMU）的12名成员采用欧元作为其货币。将NLS_TERRITORY设置为与EMU中的国家/地区（奥地利，比利时，芬兰，法国，德国，希腊，爱尔兰，意大利，卢森堡，荷兰，葡萄牙和西班牙）相对应，将NLS_CURRENCY和NLS_DUAL_CURRENCY的默认值设置为EUR 。

在过渡期（1999年至2001年）期间，Oracle数据库8 i及更高版本中提供了Oracle数据库对欧元的支持，如下所示：

  • NLS_CURRENCY被定义为该国家的主要货币

  • NLS_ISO_CURRENCY被定义为给定区域的ISO货币代码

  • NLS_DUAL_CURRENCY被定义为给定地区的次要货币符号（通常为欧元）

从Oracle Database 9 i Release 2开始， NLS_ISO_CURRENCY的值导致使用欧元的EMU成员国的ISO货币符号设置为EUR 。例如，假设NLS_ISO_CURRENCY设置为FRANCE 。输入以下SELECT语句：

SQL> SELECT TO_CHAR（order_total，'C099G999D99'）“TOTAL”FROM订单WHERE customer_id = 116;

您应该看到类似于以下输出的结果：

总计------- EUR006,394.80 EUR011,097.40 EUR014,685.80 EUR000,129.00

必须保留其过时的本地货币符号的客户可以通过在服务器上的初始化文件NLS_CURRENCY它们定义为参数以及作为客户端上的环境变量来覆盖NLS_DUAL_CURRENCY或NLS_CURRENCY的默认值。

注意：

还必须在客户端上设置NLS_LANG以使NLS_CURRENCY或NLS_DUAL_CURRENCY生效。

无法覆盖由NLS_ISO_CURRENCY值产生的ISO货币符号。

3.12.1 NLS_LENGTH_SEMANTICS

属性 描述

参数类型

串

参数范围

环境变量，初始化参数和ALTER SESSION

默认值

字节

价值范围

BYTE或CHAR

默认情况下，字符数据类型CHAR和VARCHAR2以字节为单位，而不是字符。因此，表定义中的规范CHAR(20)允许20个字节用于存储字符数据。

如果数据库这很好用字符集使用单字节字符编码方案，因为字符数与字节数相同。如果数据库字符集使用多字节字符编码方案，则字节数不再等于字符数，因为字符可以包含一个或多个字节。因此，必须谨慎选择列宽，以允许给定数量的字符具有最大可能的字节数。您可以通过在定义列大小时切换到字符语义来克服此问题。

NLS_LENGTH_SEMANTICS允许您使用字节或字符长度语义创建CHAR ， VARCHAR2和LONG列。NCHAR ， NVARCHAR2 ， CLOB和NCLOB列始终基于字符。现有列不受影响。

您可能需要使用字节语义以保持与现有应用程序的兼容性。

NLS_LENGTH_SEMANTICS不适用于在SYS模式中创建的表。数据字典总是使用字节语义。如果在表创建DDL中未指定长度限定符BYTE或CHAR则SYS拥有的表始终使用字节语义。

请注意，如果未在客户端上设置NLS_LENGTH_SEMANTICS环境变量，则客户端会话默认为数据库服务器上的NLS_LENGTH_SEMANTICS值。这使网络上的所有客户端会话具有相同的NLS_LENGTH_SEMANTICS行为。在单个客户端上设置环境变量可以为该客户端覆盖服务器初始化参数。

请注意，如果未在客户端上设置NLS_LENGTH_SEMANTICS环境变量，或者客户端通过Oracle JDBC Thin驱动程序连接，则客户端会话默认为客户端连接的实例的NLS_LENGTH_SEMANTICS初始化参数的值。出于兼容性原因，Oracle建议将此参数保留为未定义或设置为BYTE 。

注意：

Oracle强烈建议您不要在实例或服务器参数文件中将NLS_LENGTH_SEMANTICS参数设置为CHAR 。这可能导致许多现有安装脚本意外创建具有字符长度语义的列，从而导致运行时错误，包括缓冲区溢出。
