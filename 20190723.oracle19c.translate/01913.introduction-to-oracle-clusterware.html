1 Oracle Clusterware简介

Oracle Clusterware概念和组件。

Oracle Clusterware使服务器能够相互通信，从而使它们看起来像一个集合单元。这种服务器组合通常称为集群。虽然服务器是独立服务器，但每个服务器都有其他服务器与其他服务器通信。通过这种方式，单独的服务器看起来就像是应用程序和最终用户的一个系统。

Oracle Clusterware概述

Oracle Clusterware是便携式集群软件，可为整合环境提供全面的多层高可用性和资源管理。它支持独立服务器的群集，以便它们作为单个系统协作。

Oracle Clusterware是Oracle Real Application Clusters（Oracle RAC）的集成基础，是任何主要平台上所有应用程序的高可用性和资源管理框架。Oracle Clusterware首次与Oracle数据库10g第 1版（10.1）一起发布，作为Oracle多实例数据库Oracle RAC所需的集群技术。目的是利用云中的Oracle Clusterware在需要时提供企业级弹性，并在需要时随时随地动态地在线分配计算资源。

Oracle Flex集群

在Oracle Clusterware 12 c第 2版（12.2）中，所有集群都配置为Oracle Flex Clusters，这意味着集群配置了一个或多个集线器节点 ，可以支持大量节点。当前在旧版Oracle Clusterware中配置的群集将作为升级过程的一部分进行转换，包括激活Oracle Flex ASM（这是Oracle Flex群集的要求）。

集线器节点紧密连接，可直接访问共享存储。传统上，它们将部署为Oracle RAC或Oracle RAC One数据库实例的主机。群集中的其他节点与集线器节点的不同之处在于，它们不需要直接访问共享存储，而是通过集线器节点访问数据。

Oracle Flex Cluster中的所有节点都属于单个Oracle Grid Infrastructure集群。此体系结构集中了基于应用程序需求部署资源的策略决策，以考虑各种服务级别，负载，故障响应和恢复。

Oracle Flex Clusters由支持集群中其他节点的集线器节点组成。Oracle Flex Cluster中的Hub节点数必须至少为1，最多可以为64个，而其他受支持节点的数量可以更多。集线器节点可以托管不同类型的应用程序。

Oracle Flex Clusters可以与一个或多个Hub节点一起运行，但其他节点是可选的，并且只能作为包含至少一个Hub节点的群集的成员存在。

使用群集的好处包括：

  • 应用程序的可伸缩性（包括Oracle RAC和Oracle RAC One数据库）

  • 通过提供具有低成本商用硬件的可扩展系统，降低基础架构的总拥有成本

  • 能够进行故障转移

  • 通过向群集添加服务器来增加群集资源，从而提高对群集感知应用程序的吞吐量

  • 通过使应用程序能够在群集中的所有节点上运行，提高群集感知应用程序的吞吐量

  • 能够按计划的顺序对应用程序的启动进行编程，以确保以正确的顺序启动相关进程

  • 能够监控进程并在停止时重新启动它们

  • 消除因硬件或软件故障导致的意外停机

  • 减少或消除计划的软件维护停机时间

您可以配置Oracle Clusterware以管理用户应用程序和Oracle数据库的可用性。在Oracle RAC环境中，Oracle Clusterware会自动管理所有资源。Oracle Clusterware管理的所有应用程序和进程都是群集资源或本地资源 。

使用Oracle RAC需要Oracle Clusterware;它是Oracle RAC运行的平台所需的唯一集群件。虽然Oracle RAC继续支持特定平台上的许多第三方集群件产品，但您还必须安装和使用Oracle Clusterware。请注意，要安装和运行Oracle Clusterware的服务器必须使用相同的操作系统。

使用Oracle Clusterware消除了对专有供应商集群件的需求，并提供仅使用Oracle软件的好处。Oracle提供完整的软件解决方案，包括从磁盘管理到Oracle自动存储管理（Oracle ASM）到使用Oracle数据库和Oracle RAC进行数据管理的所有方面。此外，Oracle数据库功能（如Oracle服务）在与基础Oracle Clusterware高可用性框架一起使用时可提供高级功能。

除了二进制文件之外，Oracle Clusterware还有两个存储的组件：记录节点成员资格信息的投票文件和记录集群配置信息的Oracle集群注册表（OCR）。投票文件和OCR必须驻留在所有集群成员节点可用的共享存储上。

集群件架构

Oracle Clusterware在安装过程中为新集群提供了两种不同的部署体系结构选择。您可以选择域服务群集或成员群集 ，该群集用于承载应用程序和数据库。

域服务集群是Oracle Flex集群，具有一个或多个集线器节点（用于数据库实例）和零个或多个其他节点。共享存储在本地安装在每个集线器节点上，并且所有集线器节点都可以使用Oracle ASM实例。此外，在集群内本地存储和访问管理数据库。此部署还用于升级的预先存在的群集。

成员集群将多个集群配置分组以用于管理目的，并使用该集群域中可用的共享服务。该群集域中的群集配置为：

  • 域服务集群 ：为集群域中的其他集群提供集中服务的集群。服务可以包括集中式网格基础结构管理存储库（集群域中每个集群的管理数据库所在的存储库），跟踪文件分析器服务，可选的Fleet Patching和Provisioning服务，以及很可能是统一的Oracle ASM存储管理服务。

  • 数据库成员集群 ：旨在支持Oracle RAC或Oracle RAC的集群实例，其管理数据库已卸载到域服务集群，并且可以使用本地Oracle ASM存储管理进行配置或使用域服务集群提供的整合Oracle ASM存储管理服务。

  • 应用程序成员集群 ：配置为支持应用程序的集群，没有支持Oracle RAC或Oracle RAC One数据库实例所需的资源。此群集类型没有配置的本地共享存储，但它旨在为运行应用程序进程提供高度可用，可扩展的平台。

Oracle Clusterware硬件概念和要求

了解硬件概念和要求有助于确保成功部署Oracle Clusterware。

群集由一个或多个服务器组成。对于独立服务器，群集中的服务器（也称为群集成员或节点 ）对外部网络的访问是相同的。

注意：

许多硬件提供商已经验证了为群集提供单个部件号的群集配置。如果您不熟悉群集，请在购买硬件以创建群集时使用本节中的信息简化硬件采购工作。

作为群集一部分的节点需要第二个网络。该第二网络称为互连。因此，集群成员节点至少需要两个网络接口卡：一个用于公共网络，另一个用于专用网络。互连网络是使用交换机（或多个交换机）的专用网络，只有集群中的节点才能访问。脚1

注意：

Oracle不支持将交叉电缆用作Oracle Clusterware互连。

群集大小取决于群集上运行的工作负载的要求以及您在群集中配置的节点数。如果要实现群集以实现高可用性，请按如下方式为基础架构的所有组件配置冗余：

  • 公共网络至少有两个网络接口，绑定提供一个地址

  • 至少两个专用互连网络的网络接口

群集需要与群集中的每个服务器的存储共享连接。Oracle Clusterware支持网络文件系统（NFS），iSCSI，直接连接存储（DAS），存储区域网络（SAN）存储和网络连接存储（NAS）。

为了提供存储冗余，通常提供从每个服务器到群集感知存储的至少两个连接。根据您的I / O要求，可能会有更多连接。选择存储子系统时，请务必考虑整个群集的I / O要求。

大多数服务器至少有一个本地磁盘，它位于服务器内部。通常，此磁盘用于操作系统二进制文件;您也可以将此磁盘用于Oracle软件二进制文件。具有自己的Oracle二进制文件副本的每个服务器的好处是它增加了高可用性，因此一个二进制文件的损坏不会同时影响集群中的所有节点。它还允许滚动升级，从而减少停机时间。

Oracle Clusterware操作系统概念和要求

必须先安装并验证操作系统，然后才能安装Oracle Clusterware。

每台服务器必须具有经过您正在安装的Oracle Clusterware版本认证的操作系统。有关详细信息，请参阅适用于您的平台的Oracle Grid Infrastructure安装和升级指南或My Oracle Support（以前称为Oracle MetaLink ）中提供的认证矩阵。

安装并运行操作系统后，您可以安装Oracle Clusterware来创建群集。Oracle Clusterware独立于Oracle数据库安装。安装Oracle Clusterware后，可以在群集中的任何节点上安装Oracle数据库或Oracle RAC。

Oracle Clusterware软件概念和要求

Oracle Clusterware使用投票文件来提供防护和集群节点成员资格确定。Oracle Cluster Registry（OCR）提供群集配置信息。投票文件和OCR统称为Oracle Clusteware文件。

Oracle Clusterware文件必须存储在Oracle ASM上。如果Oracle ASM磁盘的基础存储不受硬件保护 （例如RAID），则Oracle建议您为OCR和投票文件配置多个位置。投票文件和OCR描述如下：

  • 投票文件

    Oracle Clusterware使用投票文件来确定哪些节点是群集的成员。您可以在Oracle ASM上配置投票文件，也可以在共享存储上配置投票文件。

    如果在Oracle ASM上配置投票文件，则无需手动配置投票文件。根据磁盘组的冗余，将创建适当数量的投票文件。

    如果您未在Oracle ASM上配置表决文件，那么为了实现高可用性，Oracle建议您在物理上独立的存储上至少有三个表决文件。这避免了单点故障。如果配置单个表决文件，则必须使用外部镜像来提供冗余。

    Oracle建议您不要使用五个以上的投票文件，即使Oracle支持最多15个投票文件。

  • Oracle Cluster Registry

    Oracle Clusterware使用Oracle Cluster Registry（OCR）来存储和管理有关Oracle Clusterware控制的组件的信息，例如Oracle RAC数据库，侦听器，虚拟IP地址（VIP）以及服务和任何应用程序。OCR将配置信息存储在树结构中的一系列键值对中。为确保群集高可用性，Oracle建议您定义多个OCR位置。此外：

    • 您最多可以有五个OCR位置

    • 每个OCR位置必须驻留在群集中所有节点均可访问的共享存储上

    • 如果它不是唯一的OCR位置，则可以在线替换发生故障的OCR位置

    • 您必须通过支持的实用程序（如Oracle Enterprise Manager，Oracle Clusterware Control Utility（CRSCTL），服务器控制实用程序（SRVCTL），OCR配置实用程序（OCRCONFIG）或数据库配置助手（DBCA））更新OCR

Oracle Clusterware网络配置概念

Oracle Clusterware通过自我管理群集的网络要求，实现动态Oracle Grid Infrastructure。

Oracle Clusterware支持对VIP地址和单客户端访问名称（SCAN）地址使用动态主机配置协议（DHCP）或无状态地址自动配置，但不支持使用公共地址。DHCP提供IPv4 VIP地址的动态分配，而无状态地址自动配置提供IPv6 VIP地址的动态分配。

节点VIP的使用在群集部署中是可选的。默认情况下，部署群集环境时会包含节点VIP。

使用Oracle RAC时，所有客户端都必须能够访问数据库，这意味着客户端必须分别将VIP和SCAN名称解析为所有VIP和SCAN地址。通过向群集添加网格命名服务（GNS）来解决此问题。GNS链接到公司域名服务（DNS），以便客户端可以将主机名解析为这些动态地址，并透明地连接到群集和数据库。Oracle支持使用没有DHCP或区域委派GNS在Oracle集群12℃（与Oracle的Flex ASM服务器集群，其可以不用区域委派或动态网络配置）。

注意：

Oracle 不支持在Windows上使用没有DHCP或区域委派的GNS。

Oracle集群12个C，A GNS实例被增强，以使服务于多个簇而不是仅一个，从而只有单个域必须在DNS委派给GNS。 GNS仍提供与先前版本的Oracle Clusterware相同的服务。

运行GNS服务器的集群称为服务器集群 。客户端群集将其名称通告给服务器群集。只有一个GNS守护程序进程可以在服务器群集上运行。Oracle Clusterware将GNS守护程序进程放在集群中的一个节点上以保持可用性。

在以前的单集群版本的GNS中，单个集群可以轻松地在其自身内定位GNS服务提供商。但是，在多集群环境中，客户端群集必须知道服务器群集的GNS地址。给定该地址，客户端群集可以找到在服务器群集上运行的GNS服务器。

要使GNS在服务器群集上运行，您必须具备以下条件：

  • DNS管理员必须委派区域供GNS使用

  • GNS实例必须在网络上的某个位置运行，并且不得被防火墙阻止

  • GNS提供的一组集群中的所有节点名称必须是唯一的

单客户端访问名称（SCAN）

Oracle Clusterware可以使用单客户端访问名称（SCAN）进行动态VIP地址配置，无需执行手动服务器配置。

SCAN是在DNS或GNS中注册到至少一个和最多三个IP地址的域名。使用GNS和DHCP时，Oracle Clusterware会为群集配置期间提供的SCAN名称配置VIP地址。

使用GNS时，从DHCP服务器获取节点VIP和三个SCAN VIP。如果新服务器加入群集，则Oracle Clusterware会从DHCP服务器动态获取所需的VIP地址，更新群集资源，并通过GNS访问服务器。

手动地址配置

您可以选择手动配置地址，而不是使用GNS和DHCP进行动态配置。

在手动地址配置中，您配置以下内容：

  • 每个节点的一个公共地址和主机名。

  • 每个节点的一个VIP地址。

    您必须为群集中的每个节点分配VIP地址。每个VIP地址必须与节点的公共IP地址位于同一子网上，并且应该是在DNS中分配了名称的地址。在安装Oracle Clusterware之前，每个VIP地址也必须在网络中未使用且无法访问。

  • 整个群集最多有三个SCAN地址。

    注意：

    SCAN必须解析为公共网络上的至少一个地址。为了实现高可用性和可伸缩性，Oracle建议您将SCAN配置为解析为公共网络上的三个地址。

Oracle Clusterware特定于平台的软件组件概述

在可操作的Oracle Clusterware中，在每个群集节点上运行各种特定于平台的进程或服务。

本节介绍这些过程和服务。

Oracle Clusterware技术堆栈

Oracle Clusterware由两个独立的技术堆栈组成：由Cluster Ready Services（CRS）守护程序（CRSD）锚定的高级技术堆栈和由Oracle高可用性服务守护程序（OHASD）锚定的低技术堆栈。

这两个技术堆栈有几个促进集群操作的过程。以下部分更详细地描述了这些技术堆栈。

集群就绪服务技术堆栈

集群就绪服务（CRS）技术堆栈利用多个流程来管理各种服务。

以下列表描述了这些过程：

  • 群集就绪服务（CRS） ：用于管理群集中的高可用性操作的主要程序。

    CRSD根据存储在OCR中的每个资源的配置信息来管理集群资源。这包括启动，停止，监视和故障转移操作。CRSD进程在资源状态更改时生成事件。安装Oracle RAC后，CRSD进程将监视Oracle数据库实例，侦听器等，并在发生故障时自动重新启动这些组件。

  • 群集同步服务（CSS） ：通过控制哪些节点是群集成员以及在节点加入或离开群集时通知成员来管理群集配置。如果您使用的是经过认证的第三方集群件，则CSS会处理与您的集群件的接口以管理节点成员身份信息。

    cssdagent进程监视集群并提供I / O防护。此服务以前由Oracle Process Monitor Daemon（ oprocd ）提供，在Windows上也称为OraFenceService 。cssdagent故障可能导致Oracle Clusterware重新启动节点。

  • Oracle ASM ：为Oracle Clusterware和Oracle Database提供磁盘管理。

  • 群集时间同步服务（CTSS） ：为Oracle Clusterware在群集中提供时间管理。

  • 事件管理（EVM） ：发布Oracle Clusterware创建的事件的后台进程。

  • 网格命名服务（GNS） ：处理外部DNS服务器发送的请求，对群集定义的名称执行名称解析。

  • Oracle代理（ oraagent ） ：扩展集群件以支持特定于Oracle的要求和复杂资源。当FAN事件发生时，此过程运行服务器标注脚本。此过程在Oracle Clusterware 11g第 1版（11.1）中称为RACG。

  • Oracle Notification Service（ONS） ：用于通信快速应用程序通知（FAN）事件的发布和订阅服务。

  • Oracle Root Agent（ orarootagent ） ：一种专门的oraagent进程，可帮助CRSD管理root拥有的资源，例如网络和Grid虚拟IP地址。

群集同步服务（CSS），事件管理（EVM）和Oracle Notification Services（ONS）组件与同一群集数据库环境中其他节点上的其他群集组件层进行通信。这些组件也是Oracle数据库，应用程序和Oracle Clusterware高可用性组件之间的主要通信链接。此外，这些后台进程监视和管理数据库操作。

Oracle高可用性服务技术堆栈

Oracle高可用性服务技术堆栈使用多个进程来提供Oracle Clusterware高可用性。

以下列表描述了Oracle高可用性服务技术堆栈中的过程：

  • appagent ：保护以前版本的Oracle Clusterware中使用的application资源类型的所有资源。

  • 集群记录器服务（ ologgerd ） ：从集群中的所有节点接收信息，并保留在基于Oracle Grid Infrastructure Management资源库的数据库中。此服务仅在群集中的两个节点上运行。

  • 网格进程间通信（GIPC） ：支持可执行冗余互连使用的支持守护程序。

  • 网格即插即用（GPNPD） ：提供对网格即插即用配置文件的访问，并协调群集节点之间的配置文件更新，以确保所有节点都具有最新的配置文件。

  • 多播域名服务（mDNS） ：由网格即插即用用于定位群集中的配置文件，并由GNS执行名称解析。mDNS进程是Linux和UNIX以及Windows上的后台进程。

  • Oracle代理（ oraagent ） ：扩展集群件以支持特定于Oracle的要求和复杂资源。此过程管理作为Oracle Clusterware所有者运行的守护程序，如GIPC，GPNPD和GIPC守护程序。

    注意：

    此过程与在Cluster Ready Services技术堆栈中运行的同名流程明显不同。

  • Oracle Root Agent（ orarootagent ） ：一种专门的oraagent进程，可帮助CRSD管理root拥有的资源，例如Cluster Health Monitor（CHM）。

    注意：

    此过程与在Cluster Ready Services技术堆栈中运行的同名流程明显不同。

  • scriptagent ：使用shell或批处理脚本保护应用application时，保护应用程序以外的资源类型的资源。

  • 系统监视器服务（ osysmond ） ：监视和操作系统度量标准收集服务，用于将数据发送到集群记录器服务。此服务在群集中的每个节点上运行。

传输层安全密码套件管理

Oracle Clusterware 19c提供CRSCTL命令以禁用给定的密码套件，并在Oracle Local Registry和Oracle Cluster Registry中存储禁用的密码套件详细信息，确保禁用列表中包含的密码套件不用于协商传输层安全性。

Oracle Clusterware技术堆栈使用GIPC库进行节点间和节点内通信。为了保护节点间通信信道，GIPC库使用传输层安全性。对于任何Oracle Clusterware版本，GIPC库都支持一组预编译的密码套件。随着时间的推移，密码套件可能会受到损害。在Oracle Clusterware 19c之前，无法禁用集合中包含的给定密码套件，以防止将来在任何新连接中使用它。

查询密码列表

要获取可用密码套件列表：
crsctl get cluster tlsciphersuite

将密码套件添加到禁用列表

要将密码套件添加到禁用列表：
crsctl set cluster disabledtlsciphersuite add cipher_suite_name

从禁用列表中删除密码套件

要从禁用列表中删除密码套件，请执行以下操作：
crsctl set cluster disabledtlsciphersuite delete cipher_suite_name

Oracle Clusterware版本兼容性

您可以在群集上安装不同版本的Oracle Clusterware，Oracle ASM和Oracle数据库。但是，您应该了解兼容性考虑因素。

在群集上安装不同版本的软件时，请遵循以下准则：

  • 您只能在群集中运行一个Oracle Clusterware安装，并且必须将其安装到自己的软件主目录（ Grid_home ）中。您使用的Oracle Clusterware的发行版必须等于或高于在集群中运行的Oracle ASM和Oracle RAC版本。您无法安装在群集上运行的Oracle Clusterware版本之后发布的Oracle RAC版本。换一种说法：

    • Oracle Clusterware 18c仅支持Oracle ASM 18c，因为Oracle ASM位于Oracle Grid Infrastructure主目录中，该主目录还包括Oracle Clusterware

    • Oracle Clusterware 18c支持Oracle Database 18c和Oracle Database 11g第 2版（11.2.0.3及更高版本）

    • Oracle ASM 18c需要Oracle Clusterware 18c，并支持Oracle Database 18c和Oracle Database 11g第 2版（11.2.0.3及更高版本）

    • Oracle Database 18c需要Oracle Clusterware 18c

      例如：

      • 如果您安装了Oracle Clusterware 18c作为您的集群件，那么您可以在一个节点上运行Oracle Database 11g第 2版（11.2.0.4）单实例数据库，并单独使用Oracle Real Application Clusters 12 c第 1版（12.1.0.2） ）和Oracle Real Application Clusters 11g版本2（11.2.0.4）数据库也在集群上运行。

      • 使用不同的Oracle ASM和Oracle Database版本时，每个版本的功能取决于早期软件版本的功能。因此，如果您安装Oracle Clusterware 18c并且稍后配置了Oracle ASM，并且使用Oracle Clusterware来支持现有的Oracle Database 11g第 2版（11.2.0.4）安装，则Oracle ASM功能仅等同于11 g版本2（11.2.0.4）发布版本。将磁盘组的兼容属性设置为正在使用的软件版本。

  • 集群中可以有多个Oracle数据库软件（包括单实例和Oracle RAC）的Oracle主目录。Oracle RAC数据库的所有节点的Oracle主目录必须相同。

  • 如果Oracle Clusterware和Oracle数据库主目录属于同一主要组，则可以使用不同的用户。

  • 与Oracle集群12c的开始，只能有在一个集群的Oracle ASM运行中的一个安装。Oracle ASM始终与Oracle Clusterware的版本相同，后者必须与Oracle数据库的版本相同（或更高）。

  • 要安装Oracle RAC 10 g ，还必须安装Oracle Clusterware。

  • Oracle建议您不要在同一台服务器上运行不同的群集软件，除非它们经过认证可以协同工作。但是，如果要将Oracle RAC添加到属于群集的服务器，则迁移到Oracle Clusterware或确保：

    • 您运行的集群件支持与Oracle RAC 18c一起运行。

    • 您已为Oracle Clusterware和其他供应商集群件安装了正确的选项，以便协同工作。

升级和修补Oracle Clusterware概述

就地修补将Oracle Clusterware软件替换为同一Grid主机中的较新版本。异地升级在节点上同时存在同一软件的两个版本，在不同的网格家庭中，但只有一个版本处于活动状态。

对于Oracle Clusterware 12 c ，Oracle支持就地或非现场修补。Oracle仅支持异地升级，因为Oracle Clusterware 12 c必须拥有自己的新Grid主页。

Oracle支持用于就地修补的修补程序包和一次性修补程序，但仅支持修补程序集和主要点发布作为非现场升级。

滚动升级可避免停机，并在软件升级到新版本时确保Oracle Clusterware的持续可用性。升级到Oracle Clusterware 12 c时 ，Oracle Clusterware和Oracle ASM二进制文件将作为单个二进制文件安装，称为Oracle Grid Infrastructure。您可以从Oracle Clusterware 11g第 2版（11.2）以滚动方式升级Oracle Clusterware和Oracle ASM。

在群集的某些节点关闭的情况下，Oracle支持强制升级。

Grid Infrastructure Management Repository概述

Grid Infrastructure Management Repository存储有关群集的信息，例如各种群集客户端收集和要求的实时性能数据和元数据。

在以前版本的Oracle Grid Infrastructure中，Grid Infrastructure Management Repository是Grid Infrastructure主目录中的独立数据库。创建的第一个磁盘组中需要磁盘空间，与Oracle Cluster Registry（OCR）和表决磁盘共享该空间。对于独立群集，您可以在Oracle Grid Infrastructure安装期间将Grid Infrastructure Management资源库与OCR备份文件一起安装到其自己的MGMT磁盘组中。

Oracle成员集群使用位于其域服务集群中的Grid Infrastructure Management Repository服务，其中Grid Infrastructure Management Repository是域服务集群的Grid Infrastructure Management Repository容器数据库（CDB）中的可插拔数据库（PDB）。这消除了运行本地Grid Infrastructure Management Repository并在成员集群的磁盘组中定位其数据文件的要求。

此外，您可以在安装Oracle Grid Infrastructure期间将本地Grid Infrastructure Management资源库和域服务群集Grid Grid Management Management存储库安装到MGMT磁盘组中。域服务群集必须位于与OCR和Voting磁盘分开的磁盘组中。

网格基础架构管理存储库：

  • 是一个Oracle数据库，用于存储Cluster Health Monitor收集的实时操作系统指标。您可以在群集上安装或升级到Oracle Clusterware 12 c期间配置Grid Infrastructure Management存储库。

    注意：

    如果要将Oracle Clusterware升级到Oracle Clusterware 12 c和OCR并且投票文件存储在原始或块设备上，则必须在升级软件之前将它们移动到Oracle ASM。

  • 必须在集线器节点上运行，并且必须支持在节点或存储发生故障时故障转移到另一个节点。

  • 通过专用网络与任何群集客户端（如Cluster Health Advisor，群集资源活动日志，Fleet Patching和Provisioning Server，OLOGGERD和OCLUMON）进行通信。Grid Infrastructure Management Repository仅与公共网络上的外部客户端，域服务群集和成员群集通信。

  • Oracle建议您将数据文件存储在自己的Oracle ASM磁盘组MGMT ，您可以在Oracle Grid Infrastructure安装期间创建该磁盘组。如果在安装期间未创建此磁盘组，则数据文件将与OCR和投票文件位于同一位置。

    Oracle增加了Oracle Clusterware共享存储要求，以适应Grid Infrastructure Management Repository，它可以是网络文件系统（NFS），群集文件系统或Oracle ASM磁盘组。

  • 使用mgmtdb和mgmtlsnr名词使用SRVCTL命令进行管理。

  • 由其每个客户端命令管理一个保留实用程序。

注意：

从Oracle Grid Infrastructure 19c开始，Grid Infrastructure Management Repository（GIMR）对于Oracle Standalone Cluster的新安装是可选的。Oracle Domain Services Clusters仍然需要安装GIMR作为服务组件。

域服务群集概述

域服务群集是面向服务的基础结构的基础，用于在单个管理域中部署群集系统。域服务群集是一个Oracle群集，仅用于为该管理域中的成员群集提供服务。

成员群集根据需要使用配置的服务。这些服务包括集中式域管理存储库，存储管理服务以及Fleet Patching和Provisioning。

配置域服务群集有许多好处，包括：

  • 易于部署和配置

    • 配置一次共享服务，重复使用多次

    • 无需为每个成员集群配置和配置共享存储

    • 部署数据库成员群集，可以选择使用共享存储管理服务或本地配置的存储

    • 部署应用程序成员集群，不支持数据库，消耗更少的资源并使用Oracle ACFS Remote进行共享存储

  • 集中共享服务

    • 将存储管理整合到域服务群集的好处：

      • 规模效率（节省存储和精力）

      • 同时管理所有存储

    • 数据库管理从本地群集到域管理存储库

    • 成员集群可以使用Oracle ACFS Remote访问域服务集群上的Oracle ACFS文件系统

群集域由域服务群集和使用所提供服务的成员群集组成。所有群集都使用相同的标准Oracle Clusterware，并使用Oracle Universal Installer进行部署。必须首先部署域服务群集，然后在成员群集部署和配置它们所需的服务时用于为其提供服务。

应用成员集群配置如下：

  • 不支持Oracle数据库和数据库资源，从而消耗更少的内存资源

  • 使用Oracle ACFS Remote访问从域服务群集配置的共享存储

  • 管理存储库以及由集中存储提供的Oracle Cluster Registry和投票文件的存储

数据库成员集群可以通过三种不同方式进行配置，具体取决于访问存储的方式。在所有情况下，管理存储库都存储在域管理存储库中。数据库成员群集存储访问的三种配置是：

  • 本地配置的共享存储，具有群集上的本地Oracle ASM实例

  • 直接访问通过域服务群集上的远程Oracle ASM管理的存储

  • 通过Oracle ACFS Remote和Oracle ASM IO服务间接访问域服务群集上的Oracle ASM存储

管理Oracle Clusterware环境概述

Oracle Clusterware为您提供了几种用于管理环境的不同实用程序。

以下列表描述了用于管理Oracle Clusterware环境的实用程序：

  • 群集运行状况监视器（CHM） ： 群集运行状况监视器检测并分析操作系统和群集资源相关的降级和故障，以便为许多Oracle Clusterware和Oracle RAC问题（如节点逐出）向用户提供更多详细信息。该工具持续跟踪节点，进程和设备级别的操作系统资源消耗。它收集并分析集群范围的数据。在实时模式下，当达到阈值时，该工具会向用户显示警报。对于根本原因分析，可以重放历史数据以了解发生故障时发生的情况。

  • 群集验证实用程序（CVU） ：CVU是一个命令行实用程序，可用于验证一系列群集和Oracle RAC特定组件。使用CVU验证共享存储设备，网络配置，系统要求和Oracle Clusterware以及操作系统组和用户。

    安装并使用CVU进行群集环境的预安装和安装后检查。在预安装和安装Oracle Clusterware和Oracle RAC组件期间，CVU特别有用，可确保您的配置满足最低安装要求。完成管理任务（例如节点添加和节点删除）后，还可以使用CVU验证配置。

  • Oracle Cluster Registry Configuration Tool（OCRCONFIG） ：OCRCONFIG是用于OCR管理的命令行工具。您还可以使用OCRCHECK和OCRDUMP实用程序来解决影响OCR的配置问题。

  • Oracle Clusterware Control（CRSCTL） ：CRSCTL是一个命令行工具，可用于管理Oracle Clusterware。使用CRSCTL进行常规集群件管理，单个资源管理，配置策略以及非数据库应用程序的服务器池。

    Oracle集群12c中引入了群集感知命令，使用它可以从任何节点集群中的另一节点上的群集在执行操作，或在集群中的所有节点上，根据操作。

    您可以使用crsctl命令监视群集资源（ crsctl status resource ）以及监视和管理除名称前缀为ora.*服务器池之外的服务器和服务器池，例如crsctl status server ， crsctl status serverpool ， crsctl modify serverpool ，和crsctl relocate server 。您还可以使用可选的节点特定参数-n或-all在整个集群上管理Oracle高可用性服务（ crsctl start | stop | enable | disable | config crs ）。您还可以使用CRSCTL在各个节点上管理Oracle Clusterware（ crsctl start | stop | enable | disable | config crs ）。

  • Oracle企业管理器 ：Oracle企业管理器具有云控制和网格控制GUI界面，用于管理单实例和Oracle RAC数据库环境。它还具有GUI界面，用于管理Oracle Clusterware以及Oracle Grid Infrastructure安装中配置的所有组件。Oracle建议您使用Oracle Enterprise Manager执行管理任务。

  • Oracle接口配置工具（OIFCFG） ：OIFCFG是用于单实例Oracle数据库和Oracle RAC环境的命令行工具。使用OIFCFG分配和释放组件的网络接口。您还可以使用OIFCFG来指导组件使用特定的网络接口并检索组件配置信息。

  • 服务器控制（SRVCTL） ：SRVCTL是一个命令行界面，可用于管理Oracle资源，例如群集中的数据库，服务或侦听器。

    注意：

    您只能使用SRVCTL来管理名称前缀为ora.服务器池ora.* 。

指挥评估概述

您可以使用Oracle Clusterware Control（CRSCTL）实用程序来评估当您使用CRSCTL命令管理Oracle Clusterware环境中的服务器，服务器池和策略时，将发生什么（命令评估）以及原因（合理的命令评估），而不实际使用变化。

除了向您展示计划内或计划外事件的后果之外，通过验证您可能拥有的有关Oracle Clusterware策略决策的任何假设，命令评估在策略管理环境中非常有用。合理的命令评估通过向您展示Oracle Clusterware通过各种CRSCTL命令的详细输出执行特定操作的原因进行扩展。

Oracle Clusterware高可用性框架和API概述

Oracle Clusterware提供了许多称为CLSCRS API的高可用性应用程序编程接口，您可以使用这些接口使Oracle Clusterware能够管理在群集中运行的应用程序或进程。CLSCRS API使您能够为所有应用程序提供高可用性。

您可以为应用程序定义VIP地址，以使用户能够独立于运行应用程序的群集中的节点访问应用程序。这被称为应用程序VIP。您可以定义多个应用程序VIP，通常为每个运行的应用程序定义一个应用程序VIP。应用程序VIP与应用程序相关，使其依赖于Oracle Clusterware定义的应用程序资源。

为了保持高可用性，Oracle Clusterware组件可以根据定义的高可用性规则响应状态更改以重新启动应用程序和进程。您可以通过向Oracle Clusterware注册应用程序并配置集群件来启动，停止或重定位应用程序进程，从而使用Oracle Clusterware高可用性框架。也就是说，您可以使用Oracle Clusterware创建监视，重定位和重新启动应用程序的配置文件，从而使自定义应用程序具有高可用性。

群集时间管理概述

群集时间同步服务（CTSS）可以检测群集中节点之间的时间同步问题。

CTSS作为Oracle Clusterware的一部分安装。如果它在系统上检测到时间同步服务（例如NTP或Chrony）或时间同步服务配置（有效或已损坏），则以观察者模式运行。例如，如果群集中的任何节点上都存在etc/ntp.conf文件，则即使没有运行时间同步软件，CTSS也会以观察者模式运行。

如果CTSS检测到群集中的任何节点上没有时间同步服务或时间同步服务配置，则CTSS进入活动模式并接管群集的时间管理。

如果CTSS在活动模式下运行而另一个非NTP时间同步软件正在运行，那么您可以通过创建名为etc/ntp.conf的文件来更改CTSS以在观察者模式下运行。CTSS在警报日志中输入有关观察者模式更改的条目。

当节点加入群集时，如果CTSS处于活动模式，则它会将这些节点上的时间与位于群集中一个节点上的参考时钟进行比较。如果两次之间存在差异且差异在某个步进限制内，则CTSS执行步进时间同步 ，即步进加入群集的节点的时间，前进或后退，以使它们与参考同步。

由于各种原因，群集中节点上的时钟与参考时钟（CTSS用作基础并且在群集中启动的第一个节点上）的周期性变得不同步。发生这种情况时，CTSS执行转换时间同步 ，即加速或减慢节点上的系统时间，直到它们与参考系统时间同步。在这种时间同步方法中，CTSS不会向后调整时间，这保证了系统时间的单调增加。

当Oracle Clusterware启动时，如果CTSS以主动模式运行且时间差异超出了步进限制（限制为24小时），则CTSS会在警报日志中生成警报，退出，并且Oracle Clusterware启动失败。您必须手动调整加入群集的节点的时间以与群集同步，之后Oracle Clusterware可以启动，CTSS可以管理节点的时间。

执行转换时间同步时，CTSS从不向后运行时间以与参考时钟同步。CTSS会定期向警报日志写入警报，其中包含有关调整节点时间以使其与参考时钟保持同步的频率的信息。

CTSS在以下情况下将条目写入Oracle Clusterware警报日志和syslog ：

  • 检测时间变化

  • 检测与参考节点的显着时间差

  • 模式从观察者切换到激活，反之亦然

运行CTSS以同步群集中的时间有助于解决Oracle Clusterware问题，因为您不必考虑不同节点上的一系列事件的时间偏移。

激活和停用群集时间管理

您可以激活CTSS以为您的群集提供时间管理服务。如果要使用其他群集时间同步服务，也可以将其停用。

要在群集中激活CTSS，必须在群集中的所有节点上停止和取消配置供应商时间同步服务。CTSS检测何时发生这种情况并假设集群的时间管理。

例如，要取消配置NTP，必须删除或重命名etc/ntp.conf文件。

同样，要停用群集中的CTSS：

  1. 在群集中的所有节点上配置供应商时间同步服务。CTSS检测到此更改并恢复为观察者模式。
  2. 使用crsctl check ctss命令确保CTSS以观察者模式运行。
  3. 在群集中的所有节点上启动供应商时间同步服务。
  4. 使用cluvfy comp clocksync -n all命令验证供应商时间同步服务是否正在运行。
