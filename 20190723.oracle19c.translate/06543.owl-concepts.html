2.1本体

本体论是特定领域中知识的共享概念化。

它由一组类，属性和可选实例组成。类通常与类层次结构（子类/超类关系）相关。类似地，属性可以通过属性层次结构（子属性/超级属性关系）来关联。属性可以是对称的或可传递的，或两者兼而有之。属性还可以具有为其指定的域，范围和基数约束。

基于RDFS的本体仅允许指定类层次结构，属性层次结构， instanceOf关系以及域和属性范围。

OWL本体建立在基于RDFS的本体上，另外允许指定属性特征。OWL本体可以进一步分类为OWL-Lite，OWL-DL和OWL Full。OWL-Lite将基数最小值和最大值限制为0或1。OWL-DL通过允许最小值和最大值来放宽此限制。OWL Full允许实例也被定义为类，OWL-DL和OWL-Lite本体不允许这样做。

支持的OWL子集描述了语义数据支持和不支持的OWL功能。

2.1.2支持的OWL子集

本节介绍支持的OWL词汇表子集。

Oracle数据库支持RDFS ++，OWLSIF和OWLPrime词汇表，它们具有增强的表达能力，以及OWL 2 RL。每个支持的词汇表都有相应的规则库;但是，这些规则库不需要填充，因为这三个词汇表的基本蕴涵规则是内部实现的。支持的词汇表如下：

  • RDFS ++：RDFS的最小扩展;这是RDFS加owl:sameAs和owl:InverseFunctionalProperty 。

  • OWLSIF：具有IF语义的OWL，具有完整性中的 pD *语义的词汇和语义，RDF模式的蕴涵的复杂性和复杂性以及涉及OWL词汇的语义扩展 ，作者：HJ Horst，Web Semantics 3,2（2005） ），79-115。

  • OWLPrime：以下OWL功能：

    • 基础知识：类，子类，属性，子属性，域，范围，类型

    • 性质特征：传递性，对称性，功能性，反函数性，逆性

    • 类比较：等价，不相交

    • 财产比较：等价

    • 个人比较：相同，不同

    • 类表达式：补码

    • 属性限制： hasValue ， someValuesFrom ， allValuesFrom

      与pD *一样，这些值限制支持的语义只是内涵（IF语义）。

  • OWL 2 RL：在W3C OWL 2 Web Ontology Language Profiles推荐（ http://www.w3.org/TR/owl2-profiles/#OWL_2_RL ）的“OWL 2 RL”部分中描述为：“OWL 2 RL profile适用于需要可扩展推理而不会牺牲太多表现力的应用程序。它旨在容纳OWL 2应用程序，这些应用程序可以交换语言的完整表现力以提高效率，以及RDF（S）应用程序需要OWL 2的一些额外表现力。“

    系统定义的规则库OWL2RL支持为OWL 2 RL定义的所有标准生产规则。与OWLPRIME ，用户不会在此OWL2RL规则库中看到任何规则。如果规则库OWL2RL尚不存在，则会自动创建。

    以下代码摘录使用OWL2RL规则库：

    CREATE TABLE m1_tpl（triple SDO_RDF_TRIPLE_S）COMPRESS;执行sem_apis.create_sem_model（'m1'，'m1_tpl'，'triple'，network_owner =>'RDFUSER'，network_name =>'NET1'）; - 将数据插入M1型。细节省略......- 现在使用OWL2RL规则库EXECUTE sem_apis.create_entailment（'m1_inf'，sem_models（'m1'）， sem_rulebases（'owl2rl'） ，network_owner =>'RDFUSER'，network_name =>'NET1'）运行推理;

    请注意，当使用OWL2RL规则库时，推理相关的优化（例如并行推理和RAW8）都适用。

  • OWL 2 EL：在W3C OWL 2 Web本体语言配置文件的推荐（ http://www.w3.org/TR/owl2-profiles/#OWL_2_EL ）的“OWL 2 EL”部分中描述为：“OWL 2 EL profile被设计为OWL 2的子集

    • 特别适用于使用定义了大量类和/或属性的本体的应用程序，

    • 捕获许多此类本体所使用的表达能力，以及

    • 可以在多项式时间内决定本体一致性，类表达式包含和实例检查。“

    OWL 2 EL本体的一个主要例子是生物医学本体SNOMED临床术语（SNOMED CT）。有关SNOMED CT的信息，请参阅： http://www.ihtsdo.org/snomed-ct/ ： http://www.ihtsdo.org/snomed-ct/

    系统定义的规则库OWL2EL支持EL语法。

    与OWLPRIME和OWL2RL ，用户不会在此OWL2EL规则库中看到任何规则，并且如果OWL2EL规则库尚不存在，则会自动创建。

    以下代码摘录使用OWL2EL规则库来对抗众所周知的SNOMED本体：

    CREATE TABLE snomed_tpl（triple SDO_RDF_TRIPLE_S）COMPRESS;执行sem_apis.create_sem_model（'snomed'，'snomed_tpl'，'triple'，network_owner =>'RDFUSER'，network_name =>'NET1'）compress; - 将数据插入模型SNOMED。细节省略......- 现在使用OWL2EL规则库EXECUTE运行推理sem_apis.create_entailment（'snomed_inf'，sem_models（'snomed'）， sem_rulebases（'owl2el'） ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

    请注意， OWL2EL规则库支持不包括自反对象属性（ ReflexiveObjectProperty ），因为反身对象属性会将每个人与自身链接，这可能会导致推理图的不必要且昂贵的扩展。

2.2.8优化owl：sameAs推理

您可以通过在执行OWLPrime蕴涵时在options参数中指定'OPT_SAMEAS=T'来优化大型owl:sameAs cliques的推理性能。（ clique是一个图表，其中每个节点都是双向连接到同一图表中的每个其他节点。）

根据OWL语义， owl:sameAs构造被视为等价关系，因此它是自反的，对称的和传递的。因此，在推理期间， owl:sameAs相关蕴涵的完全实现可以显着增加推断图的大小。请考虑以下示例三元组：

：John owl：sameAs：John1。：John owl：sameAs：John2。：John2：hasAge“32”。

将OWLPrime推断（指定了SAM组件）应用于此集合将生成以下新三元组：

：John1 owl：sameAs：John。：John2 owl：sameAs：John。：John1 owl：sameAs：John2。：John2 owl：sameAs：John1。：John owl：sameAs：John。：John1 owl：sameAs：John1。：John2 owl：sameAs：John2。：John：hasAge“32”。：John1：hasAge“32”。

在前面的示例中， :John ， :John1和:John2使用owl:sameAs关系相互连接;也就是说，他们是owl:sameAs成员owl:sameAs clique 。要为大型owl:sameAs cliques提供优化推理，您可以通过在执行OWLPrime蕴涵时在options参数中指定'OPT_SAMEAS=T'来合并owl:sameAs三元组而不会牺牲正确性。例如：

EXECUTE sem_apis.create_entailment（'M_IDX'，sem_models（'M'），sem_rulebases（'OWLPRIME'），null，null， 'OPT_SAMEAS = T' ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

当您指定此选项时，对于每个owl:sameAs clique，将选择来自clique的一个资源作为规范代表，并且该clique的所有推断都围绕该资源进行合并。使用前面的示例，如果:John1是clique代表，在合并之后，推断的图形将仅包含以下三元组：

：John1 owl：sameAs：John1。：John1：hasAge“32”。

owl:sameAs合并会产生一些开销。在推理期间，所有断言的模型都被复制到推理分区中，在那里它们与推断的三元组合并在一起。此外，对于非常大的断言图，合并和删除重复三元组会导致较大的运行时开销，因此建议仅对具有大量owl:sameAs关系和大团体大小的本体使用OPT_SAMEAS=T选项。

在OPT_SAMEAS = T选项被用于蕴涵之后， SEM_APIS.CREATE_ENTAILMENT对该蕴涵的所有后续使用也必须使用OPT_SAMEAS=T ，否则将报告错误。要禁用优化的sameAs处理，必须首先删除sameAs 。

2.2.9执行增量推理

增量推断可用于在三次添加之后有效地更新蕴涵（规则索引）。有两种方法可以为蕴涵启用增量推理：

  • 创建蕴涵时指定options参数值INC=T例如：

    EXECUTE sem_apis.create_entailment（'M_IDX'，sem_models（'M'），sem_rulebases（'OWLPRIME'），null，null， 'INC = T' ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

  • 使用SEM_APIS.ENABLE_INC_INFERENCE过程。

    如果使用此过程，则必须具有VALID状态。在调用该过程之前，如果您不拥有该蕴涵中涉及的模型，则必须确保相应的模型所有者已使用SEM_APIS.ENABLE_CHANGE_TRACKING过程为这些模型启用更改跟踪。

当为包含启用增量推断时，必须在调用该蕴涵的SEM_APIS.CREATE_ENTAILMENT过程时指定参数INC=T

蕴涵的增量推断取决于创建蕴涵所涉及的模型的应用表的触发器。这意味着增量推断仅在使用常规路径加载在所需的应用程序表中插入三元组时起作用，除非您在调用SEM_APIS.CREATE_ENTAILMENT过程中使用delta_in参数指定三元组，如下例所示将模型M_NEW的三元组添加到模型M ，并使用新的推论更新蕴涵M_IDX ：

EXECUTE sem_apis.create_entailment（'M_IDX'，sem_models（'M'），sem_rulebases（'OWLPRIME''），SEM_APIS.REACH_CLOSURE，null，null， sem_models（'M_NEW'） ，network_owner =>'RDFUSER'，network_name =>' NET1' ）;

如果增量推理调用中涉及多个模型， DEST_MODEL=指定要添加delta_in模型的目标模型，请在options参数中指定DEST_MODEL= <model_name> DEST_MODEL= <model_name> 。例如，以下原因导致模型M_NEW的语义数据被添加到模型M2 ：

执行sem_apis.create_entailment（'M_IDX'，sem_models（'M1'，'M2'，'M3'），sem_rulebases（'OWLPRIME''），SEM_APIS.REACH_CLOSURE，null，'DEST_MODEL = M2'， sem_models（'M_NEW'） ），network_owner =>'RDFUSER'，network_name =>'NET1'）;

使用增量推理时绕过传统路径加载要求的另一种方法是设置UNDO_RETENTION参数以覆盖执行批量加载时的蕴涵间隔。例如，如果最后一次蕴涵是在6小时前创建的，则UNDO_RETENTION值应设置为大于6小时;如果它小于那个，那么（考虑到繁重的工作量和有限的撤销空间），不能保证保留所有相关的撤销信息以进行增量推理。在这种情况下， SEM_APIS.CREATE_ENTAILMENT过程将回退到常规（非增量）推理。

要检查是否在模型上启用了更改跟踪，请使用SEM_APIS.GET_CHANGE_TRACKING_INFO过程。要获取有关蕴涵的增量推断的其他信息，请使用SEM_APIS.GET_INC_INF_INFO过程。

以下限制适用于增量推理：

  • 它不适用于优化的owl:sameAs处理（ OPT_SAMEAS ），用户定义的规则，启用VPD的模型或启用版本的模型。

  • 它仅支持添加三元组。通过更新或删除，将完全重建蕴涵。

  • 它取决于应用程序表上的触发器。

  • 增量推断中使用的列类型（RAW8或NUMBER）必须一致。例如，如果最初使用RAW8=T来构建蕴涵，那么对于每个后续的SEM_APIS.CREATE_ENTAILMENT调用，必须使用相同的选项。要将列类型更改为NUMBER，您必须删除并重建所需的内容。

2.2.10使用并行推理

并行推理可以通过利用多核或多CPU架构的功能来提高推理性能。要使用并行推理，请在使用SEM_APIS.CREATE_ENTAILMENT过程时指定DOP （并行度）关键字和适当的值。例如：

执行sem_apis.create_entailment（'M_IDX'，sem_models（'M'），sem_rulebases（'OWLPRIME'），sem_apis。REACH_CLOSURE，null， 'DOP = 4' ，network_owner =>'RDFUSER'，network_name =>'NET1'）;

指定DOP关键字会导致为Oracle选择的推理组件集启用并行执行

并行推断的成功在很大程度上取决于运行数据库的系统的良好硬件配置。关键是要有一个“平衡”系统，实现数据库性能调优和Oracle SQL并行执行的最佳实践。例如，不要将单个1 TB磁盘用于800 GB数据库，因为在单个物理磁盘上并行执行SQL语句甚至比在串行模式下执行SQL语句要慢。并行推理需要充足的内存;对于每个CPU核心，您应该至少有4 GB的内存。

并行推理最适合大型本体;但是，对于小型本体，推理性能也可以提高。

使用并行推断会产生一些临时存储开销。并行推理构建一个源表，其中包含基于所有源RDF / OWL模型和现有推断图的所有三元组。与存储数据和源模型索引所需的空间相比，此表可能使用额外10％到30％的存储空间。

2.2.11使用基于命名图的推理（全局和本地）

Oracle数据库中的默认推理从所提供的所有源模型或模型中获取所有断言的三元组，并在所有断言的三元组之上应用语义规则，直到达到推理闭包。即使给定的源模型包含一个或多个多个命名图形，它也没有区别，因为所有断言（无论是否是命名图形的一部分）都被视为来自单个图形的相同。（有关RDF语义图中命名图支持的介绍，请参阅命名图 。）

此默认推理可以被视为完全“全局”，因为它根本不考虑命名图。

但是，如果使用命名图，则可以覆盖默认推理，并使用以下任一功能考虑命名图：

  • 基于命名图的全局推理（NGGI），它将所有指定的命名图视为统一图。NGGI允许您缩小三元组的范围，同时实现极大的灵活性;它在基于命名图的全局推理（NGGI）中进行了解释 。

  • 基于命名图的局部推理（NGLI），它将每个指定的命名图视为一个单独的实体。NGLI在基于命名图的局部推断（NGLI）中进行了解释 。

对于使用NGGI和NGLI在一起，看到一个推荐的使用流量使用NGGI和NGLI一起 。

在创建蕴涵（规则索引）时，通过SEM_APIS.CREATE_ENTAILMENT过程的某些参数和选项指定NGGI或NGLI。

  • 基于命名图的全局推理（NGGI）
  • 基于命名图的局部推理（NGLI）
  • 一起使用NGGI和NGLI

2.2.11.1基于命名图的全局推理（NGGI）

基于命名图的全局推理（NGGI）使您能够缩小在命名图级别（与模型级别相对）进行推理的三元组范围。它还可以灵活地选择范围;例如，您可以包含零个或多个命名图形和/或默认图形中的三元组，并且可以包含指定模型中具有空图形名称的所有三元组。

例如，在医院应用程序中，您可能只想对描述特定医院患者的一组命名图中包含的所有信息应用推理规则。如果与患者相关的命名图仅包含与实例相关的断言（ABox），则可以指定一个或多个其他模式相关模型（TBox），如例2-6所示 。

  • include_default_g参数使指定模型中具有NULL图名的所有三元组都参与NGGI。在此示例中，模型hospital_ontology具有NULL图名称的所有三元组将包含在NGGI中。

  • include_named_g参数使得来自指定命名图（所有源模型）的所有三元组都参与NGGI。在此示例中，来自命名图形<urn:hospital1_patient1>和<urn:hospital1_patient2>三元组将包含在NGGI中。

2.2.11.2基于命名图的局部推理（NGLI）

基于命名图的局部推理（NGLI）将每个命名图视为单独的实体，而不是将图形视为单个统一图。推理逻辑在每个实体的边界内执行。您可以在默认图中指定与模式相关的断言（TBox），该默认图将参与每个命名图的推断。例如，基于具有名称G1的图形的推断三元组将在推断数据分区中被分配相同的图形名称G1 。

来自任何两个单独的命名图的断言将永远不会共同产生任何新的断言。

例如，假设以下内容：

  • 图G1包括以下断言：

    ：John：hasBirthMother：Mary。

  • 图G2包括以下断言：

    ：John：hasBirthMother：Bella。

  • 默认图表包含以下断言:hasBirthMother是一个owl:FunctionalProperty 。（此断言具有空图形名称。）

2.3使用语义运算符查询关系数据

模式专用语义网络不支持SEM_RELATED和SEM_DISTANCE。

  • 使用SEM_DISTANCE辅助操作员
  • 创建MDSYS.SEM_INDEXTYPE类型的语义索引
  • 当索引列不是第一个参数时，使用SEM_RELATED和SEM_DISTANCE

2.3.1使用SEM_RELATED运算符

参考Example：Disease Ontology中的本体示例，考虑以下需要语义匹配的查询： 查找诊断类型为“Immune_System_Disorder”的所有患者。 涉及语法匹配的PATIENTS表（在示例：疾病本体中描述）的典型数据库查询将不返回任何行，因为没有行具有包含精确值Immune_System_Disorder的DIAGNOSIS列。例如，以下查询不会返回任何行：

SELECT诊断FROM患者WHERE诊断='Immune_System_Disorder';

实施例2-9 SEM_RELATED操作员

但是，患者数据表中的许多行都是相关的，因为它们的诊断属于此类。示例2-9使用SEM_RELATED运算符（而不是词法相等）从患者数据表中检索所有相关行。（在此示例中，术语Immune_System_Disorder以名称空间为前缀，默认假设是表列中的值也具有名称空间前缀。但是，情况可能并非总是如此，如使用URIPREFIX时，如果值未存储为URI，则说明 。） 

选择诊断来自患者WHERE SEM_RELATED（诊断，'<http://www.w3.org/2000/01/rdf-schema#subClassOf>'，'<http://www.example.org/medical_terms/Immune_System_Disorder>' ，sem_models（'medical_ontology'），sem_rulebases（'owlprime'））= 1;

SEM_RELATED运算符具有以下属性：

SEM_RELATED（sub VARCHAR2，predExpr VARCHAR2，obj VARCHAR2，ontologyName SEM_MODELS，ruleBases SEM_RULEBASES，index_status VARCHAR2，lower_bound INTEGER，upper_bound INTEGER）RETURN INTEGER;

sub属性是要搜索的表列的名称。表格列中的术语通常是<subject，predicate，object>三元模式中的主题。

predExpr属性表示谓词，该谓词可以显示为从主题节点到对象节点的路径上的边的标签。

obj属性表示本体中的术语，必须在表中找到相关术语（由predExpr属性相关）（在sub属性指定的列中）。该术语通常是<subject，predicate，object>三重模式中的对象。（在使用相等运算符的查询中，这将是查询项。）

ontologyName属性是包含术语之间关系的本体的名称。

rulebases属性标识一个或多个规则库，其规则已应用于本体以推断新关系。当指定此属性时，将根据本体关系和推断的新关系来回答查询。

即使相关的蕴涵（在指定的规则库应用于本体时创建）没有有效状态， index_status可选属性也允许您查询数据。如果此属性为null，则如果蕴涵没有有效状态，则查询将返回错误。如果此属性不为null，则必须为字符串VALID ， INCOMPLETE或INVALID ，以指定查询成功所需的最小状态。由于OWL不保证单调性，因此在指定OWL规则库时不应使用值INCOMPLETE 。

lower_bound和upper_bound可选属性允许您指定相关术语之间关系的距离度量的界限。有关距离测量的说明，请参阅使用SEM_DISTANCE辅助操作员 。

如果两个输入项与本predExpr指定的predExpr关系相关，则SEM_RELATED运算符返回1，如果两个输入项不相关，则返回0。如果指定了下限和上限，则如果两个输入项与大于或等于lower_bound且小于或等于upper_bound的距离度量相关，则返回1。

2.3.2.1距离信息的计算

在推理（蕴涵）期间为以下属性生成距离：定义为传递属性的OWL属性，以及RDFS subClassOf和RDFS subPropertyOf属性。通过这些属性链接的两个术语之间的距离被计算为它们在分层类结构中的最短距离。通过其他属性链接的两个术语的距离是未定义的，因此设置为null。

原始模型中的每个传递属性链接（被视为分层类结构）具有1的距离，并且根据两个术语之间的链接的数量生成推断的三元组的距离。请考虑以下假设的示例场景：

  • 如果原始图包含C1 rdfs:subClassOf C2和C2 rdfs:subClassOf C3 ，则将导出C1 rdfs:subClassof of C3 。在这种情况下：

    • C1 rdfs:subClassOf C2 ：distance = 1，因为它存在于模型中。

    • C2 rdfs:subClassOf C3 ：distance = 1，因为它存在于模型中。

    • C1 rdfs:subClassOf C3 ：distance = 2，因为它是在推理过程中生成的。

  • 如果原始图包含P1 rdfs:subPropertyOf P2和P2 rdfs:subPropertyOf P3 ，则将导出P1 rdfs：subPropertyOf P3。在这种情况下：

    • P1 rdfs:subPropertyOf P2 ：distance = 1，因为它存在于模型中。

    • P2 rdfs:subPropertyOf P3 ：distance = 1，因为它存在于模型中。

    • P1 rdfs:subPropertyOf P3 ：distance = 2，因为它是在推理过程中生成的。

  • 如果原始图包含C1 owl:equivalentClass C2和C2 owl:equivalentClass C3 ，那么将导出C1 owl:equivalentClass C3 。在这种情况下：

    • C1 owl:equivalentClass C2 ：distance = 1，因为它存在于模型中。

    • C2 owl:equivalentClass C3 ：distance = 1，因为它存在于模型中。

    • C1 owl:equivalentClass C3 ：distance = 2，因为它是在推理期间生成的。

SEM_RELATED运算符使用用户定义的规则库。但是，尚不支持将SEM_DISTANCE运算符与用户定义的规则库一起使用，并将引发错误。

2.3.3创建MDSYS.SEM_INDEXTYPE类型的语义索引

使用SEM_RELATED运算符时，可以在包含本体术语的列上创建MDSYS.SEM_INDEXTYPE类型的语义索引。创建这样的索引将导致更有效地执行查询。CREATE INDEX语句必须包含INDEXTYPE IS MDSYS.SEM_INDEXTYPE子句，以指定要创建的索引的类型。

例2-12创建语义索引

例2-12使用Example：Disease Ontology中的本体，在PATIENTS表的DIAGNOSIS列上创建一个名为DIAGNOSIS_SEM_IDX的语义索引。 

CREATE INDEX diagnosis_sem_idx ON患者（诊断）INDEXTYPE IS MDSYS.SEM_INDEXTYPE;

构建索引的列（ 示例2-12中的 DIAGNOSIS）必须是SEM_RELATED运算符的第一个参数，以便使用索引。如果它不是第一个参数，则在执行查询期间不使用索引。

示例2-13创建指定模型和规则库的语义索引
