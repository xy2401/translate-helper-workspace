8错误处理和诊断

应用程序必须预测运行时错误并尝试从中恢复。本章提供了有关错误报告和恢复的深入讨论。您将学习如何使用状态变量SQLCODE，SQLSTATE，SQLCA（SQL通信区域）和WHENEVER语句来处理警告和错误。您还将学习如何使用状态变量ORACA（Oracle通信区域）诊断问题。讨论了以下主题：

  • 错误处理的必要性

  • 错误处理替代方案

  • 关于MODE = {ANSI | ANSI14}时使用状态变量

  • 关于使用SQL通信区域

  • 关于使用Oracle通信区域

8.1错误处理的必要性

每个应用程序的重要部分都必须用于错误处理。错误处理的主要好处是它允许程序在出现错误时继续运行。错误源于设计错误，编码错误，硬件故障，无效的用户输入和许多其他来源

您无法预料到所有可能的错误，但您可以计划处理对您的程序有意义的某些类型的错误。对于Oracle预编译器，错误处理意味着检测并从SQL语句执行错误中恢复。

您还可以准备处理警告，例如“值截断”和状态更改，例如“数据结束”。在每个数据操作语句之后检查错误和警告条件尤为重要，因为在处理表中的所有符合条件的行之前， INSERT ， UPDATE或DELETE语句可能会失败。

8.2错误处理替代方案

Oracle预编译器提供了四个状态变量，用作错误处理机制：

  • SQLCODE（Pro * FORTRAN中的SQLCOD）

  • SQLSTATE（Pro * FORTRAN中的SQLSTA）

  • SQLCA（使用WHENEVER声明）

  • ORACA

MODE选项（描述）控制ANSI / ISO合规性。SQLCODE，SQLSTATE和SQLCA变量的可用性取决于MODE设置。无论MODE设置如何，您都可以声明和使用ORACA变量。有关更多信息，请参阅关于使用Oracle通信区域 。

当MODE={ORACLE|ANSI13} ，您必须声明SQLCA状态变量。接受SQLCODE和SQLSTATE声明（不推荐），但不会将其识别为状态变量。有关更多信息，请参阅关于使用SQL通信区域 。

当MODE={ANSI|ANSI14} ，您可以使用SQLCODE，SQLSTATE和SQLCA变量中的任何一个，两个或全部三个。要确定哪种变量（或变量组合）最适合您的应用，请参阅关于在MODE = {ANSI | ANSI14}时使用状态变量 “。

8.2.1 SQLCODE和SQLSTATE

在Oracle预编译器的1.5版中，引入了SQLCODE状态变量作为SQL89标准ANSI / ISO错误报告机制。SQL92标准将SQLCODE列为不推荐使用的功能并定义了一个新的状态变量SQLSTATE（与Oracle预编译器的1.6版一起引入），作为首选的ANSI / ISO错误报告机制。

SQLCODE存储错误代码和“未找到”条件。它仅保留与SQL89的兼容性，可能会从该标准的未来版本中删除。

与SQLCODE不同，SQLSTATE存储错误和警告代码并使用标准化编码方案。执行SQL语句后，Oracle服务器将状态代码返回到当前范围内的SQLSTATE变量。状态代码指示SQL语句是成功执行还是引发异常（错误或警告条件）。为了提高互操作性 （系统轻松交换信息的能力），SQL92预定义了所有常见的SQL异常。

8.2.2 SQLCA

SQLCA是一种类似记录的主机语言数据结构。Oracle在每个可执行的 SQL语句之后更新SQLCA。（在声明性陈述之后，SQLCA值未定义。）通过检查存储在SQLCA中的Oracle返回码，您的程序可以确定SQL语句的结果。这可以通过两种方式完成：

  • 使用WHENEVER语句进行隐式检查

  • 显式检查SQLCA变量

您可以使用WHENEVER语句，对SQLCA变量进行显式检查，或者同时执行这两种操作。通常，使用WHENEVER语句是首选，因为它更容易，更便携，并且符合ANSI标准。

8.2.3 ORACA

如果需要有关运行时错误的更多信息，而不是SQLCA提供的信息，则可以使用ORACA，其中包含游标统计信息，SQL语句数据，选项设置和系统统计信息。

ORACA是可选的，无论MODE设置如何，都可以声明。有关ORACA状态变量的更多信息，请参阅“ 关于使用Oracle通信区域” 。

8.3关于MODE = {ANSI | ANSI14}时使用状态变量

当MODE={ANSI|ANSI14},您必须至少声明一个 - 您可以声明两个或全部三个 - 以下状态变量：

  • SQLCODE

  • SQLSTATE

  • SQLCA

在Pro * COBOL中，如果声明了SQLCA，则无法声明SQLCODE。同样，如果声明了SQLCODE，则无法声明SQLCA。存储Pro * COBOL错误代码的SQLCA数据结构中的字段也称为SQLCODE，因此如果声明了两个状态变量，则会发生错误。

通过在可执行SQL和PL / SQL语句之后使用您自己的代码显式检查SQLCODE和SQLSTATE，SQLCODE或SQLSTATE，您的程序可以获得最新可执行SQL语句的结果。您的程序也可以隐式检查SQLCA（使用WHENEVER SQLERROR和WHENEVER SQLWARNING语句），也可以显式检查SQLCA变量。

注意：

当MODE={ORACLE|ANSI13},您必须声明SQLCA状态变量。有关更多信息，请参阅关于使用SQL通信区 “。

8.3.1一些历史信息

Oracle预编译器对状态变量和变量组合的处理从1.5版开始发展。

8.3.2版本1.5

Oracle预编译器1.5版假定存在一个状态变量SQLCODE，无论它是否在Declare Section中声明;事实上，预编译器从不打扰注意是否有SQLCODE的声明 - 它只是假定声明存在。仅当存在SQLCA的INCLUDE ，SQLCA才会用作状态变量。

8.3.3版本1.6

从Oracle预编译器1.6版开始，预编译器不再假定存在SQLCODE状态变量，并且不需要它。预编译器要求声明SQLCA，SQLCODE或SQLSTATE 中的至少一个。

当且仅当满足以下条件中的至少一个时，SQLCODE才被识别为状态变量：

  • 它在Declare Section中声明，具有完全正确的数据类型。

  • 预编译器找不到其他状态变量。

如果预编译器发现一个SQLSTATE声明中声明段（ 完全正确类型的课程），或者找到一个INCLUDE SQLCA的，它不会擅自SQLCODE声明。

8.3.4版本1.7

由于Oracle预编译器的1.5版允许在同时声明SQLCA的同时在声明部分之外声明SQLCODE变量，因此预编译器版本1.6及更高版本会出现兼容性问题。添加了一个新选项ASSUME_SQLCODE={YES|NO} （默认为NO）以在版本1.6.7中修复此问题，并将其记录为版本1.7中的新功能。

当A SSUME_SQLCODE=YES,并且当SQLSTATE或SQLCA（仅限Pro * FORTRAN）被声明为状态变量时，预编译器假定SQLCODE被声明为无论是在声明部分还是在正确类型中声明它都是无效的。这导致版本1.6.7及更高版本在这方面的行为类似于版本1.5。有关预编译器选项ASSUME_SQLCODE的信息，请参阅“ASSUME_SQLCODE”。

8.3.9状态变量值

8.3.10 SQLCODE值

在每次SQL操作之后，Oracle都会将状态代码返回到当前范围内的SQLCODE变量。状态代码（表示SQL操作的结果）可以是以下任何数字：

0

Oracle在不检测错误或异常的情况下执行了SQL语句。

> 0

Oracle执行了该语句，但检测到异常。当Oracle无法找到满足WHERE子句中的条件的行或SELECT INTO或FETCH返回任何行时，就会发生这种情况。

当MODE={ANSI|ANSI14|ANSI13}, +100在没有行的INSERT之后MODE={ANSI|ANSI14|ANSI13}, +100返回到SQLCODE。当子查询不返回要处理的行时，可能会发生这种情况。

<0

由于数据库，系统，网络或应用程序错误，Oracle未执行该语句。这些错误是无法恢复的。当它们发生时，在大多数情况下，当前事务应该回滚。负返回码对应于Oracle数据库错误消息中列出的错误代码。

您可以通过使用您自己的代码显式检查SQLCODE或使用WHENEVER语句隐式检查来了解最新SQL操作的结果。

在特定预编译单元中声明SQLCODE而不是SQLCA时，预编译器会为该单元分配内部SQLCA。您的主程序无法访问内部SQLCA。如果声明SQLCA 和 SQLCODE（仅限Pro * FORTRAN），Oracle会在每次SQL操作后向两者返回相同的状态代码。

注意：

当MODE={ORACLE|ANSI13},将忽略SQLCODE的声明。

8.3.11 SQLSTATE值

8.4关于使用SQL通信区域

SQL通讯区域（ SQLCA）是一种类似记录的数据结构。其字段包含Oracle在SQL时更新的错误，警告和状态信息声明被执行。因此，SQLCA始终反映最近SQL操作的结果。要确定结果，可以在SQLCA中检查变量。

在允许本地和全局声明的主机语言中，您的程序可以有多个SQLCA。例如，它可能有一个全局SQLCA和几个本地SQLCA。访问本地SQLCA受其在程序范围内的限制。Oracle仅将信息返回给“活动”SQLCA。

另请注意，当您的应用程序使用时 SQL * Net同时访问本地和远程数据库的组合，所有数据库都写入一个SQLCA。每个数据库没有不同的SQLCA。有关更多信息，请参阅“ 并发登录 ”。

当MODE={ORACLE|ANSI13} ，需要SQLCA;如果未声明SQLCA，则会发生编译时错误。当MODE={ANSI|ANSI14} ，SQLCA是可选的，但是如果不声明SQLCA，则不能使用WHENEVER SQLWARNING语句。因此，如果要使用WHENEVER SQLWARNING语句，则必须声明SQLCA。

如果在特定编译单元中声明SQLCODE而不是SQLCA，则预编译器会为该单元分配内部SQLCA。您的主程序无法访问内部SQLCA。如果声明SQLCA 和 SQLCODE（仅限Pro * FORTRAN），Oracle会在每次SQL操作后向两者返回相同的状态代码。

当MODE={ANSI|ANSI14} ，必须声明SQLSTATE（请参阅SQLCODE和SQLSTATE SQLSTATE状态变量支持SQL92标准指定的SQLSTATE状态变量。您可以在有或没有SQLCODE的情况下使用SQLSTATE状态变量。有关更多信息，请参见表8-3和表8-4 。

8.4.6状态代码

每个可执行SQL语句都返回一个 SQLCA变量SQLCODE的状态代码，您可以使用WHENEVER语句隐式检查或使用您自己的代码显式检查。

警告标志在SQLCA变量SQLWARN（0）到SQLWARN（7）中返回，您可以隐式或显式检查。这些警告标志对于Oracle未被视为错误的运行时条件很有用。

8.4.8行处理计数

最近执行的SQL语句处理的行数在SQLCA变量SQLERRD（3）中返回，您可以显式检查它。

严格来说，此变量不适用于错误报告，但它可以帮助您避免错误。例如，假设您希望从表中删除大约十行。删除后，检查SQLERRD（3）并找到已处理的75行。为安全起见，您可能希望回滚删除并检查WHERE子句中的搜索条件。

8.4.9解析错误偏移量

在执行SQL语句之前，Oracle必须对其进行解析 ，即检查它以确保它遵循语法规则并引用有效的数据库对象。如果Oracle发现错误，则会在SQLCA变量SQLERRD（5）中存储偏移量，您可以显式检查该变量。偏移量指定SQL语句中解析错误开始时的字符位置。第一个角色占据零位。例如，如果偏移量为9，则解析错误从第十个字符开始。

默认情况下，在预编译时检查静态SQL语句的语法错误。因此，SQLERRD（5）对于调试程序在运行时接受或构建的动态SQL语句非常有用。

解析错误是由丢失，错放或错误拼写的关键字，无效选项，不存在的表等引起的。例如，动态SQL语句

UPDATE EMP SET JIB =：job_title WHERE EMPNO =：emp_number

导致解析错误

ORA-00904：列名无效

因为列名JOB拼写错误。SQLERRD（5）的值为15，因为错误的列名JIB从第16个字符开始。

如果您的SQL语句不会导致解析错误，Oracle会将SQLERRD（5）设置为零。如果解析错误从第一个字符（占据零位置）开始，Oracle也会将SQLERRD（5）设置为零。因此，仅当SQLCODE为负时才检查SQLERRD（5），这意味着发生了错误。

- 为函数调用声明变量msg_buffer CHARACTER（100）; buf_size INTEGER; msg_length INTEGER; set buf_size = 100;执行SQL WHENEVER SQLERROR DO sql_error; - 其他语句ROUTINE sql_error BEGIN  - 获取错误消息SQLGLM的全文（msg_buffer，buf_size，msg_length）;显示msg_buffer的内容;退出程序，出现错误END sql_error;

请注意，只有在发生SQL错误时才会调用SQLGLM。在调用SQLGLM 之前，始终确保SQLCODE为负数。如果在SQLCODE为零时调用SQLGLM，则会获得与先前SQL语句关联的消息文本。

8.4.22使用WHENEVER声明

默认情况下，预编译程序会忽略Oracle错误和警告条件，并在可能的情况下继续处理。执行自动条件检查和错误处理，使用WHENEVER语句。

使用WHENEVER语句，您可以指定Oracle检测到错误，警告条件或“未找到”条件时要采取的操作。这些操作包括继续下一个语句，调用例程，分支到带标签的语句或停止。

您使用以下语法编写WHENEVER语句：

EXEC SQL WHENEVER <condition> <action>;

8.4.31范围

因为WHENEVER是一个声明性陈述，它的范围是位置的，而不是逻辑的。它测试源文件中跟随它的所有可执行SQL语句，而不是程序逻辑流程。因此，您应该在要测试的第一个可执行SQL语句之前编写WHENEVER语句。

8.4.32指南

以下指南将帮助您避免一些常见的陷阱。

发表声明。 通常，在程序中的第一个可执行SQL语句之前编写WHENEVER语句。这可以确保所有后续错误都被捕获，因为WHENEVER语句对文件末尾保持有效。

处理数据结束条件。 在使用游标获取行时，您的程序应准备好处理数据结束条件。如果FETCH没有返回数据，程序应该分支到发出CLOSE命令的标记代码段，如下所示：

SQL WHENEVER NOT FOUND GOTO no_more; ...不再： ...EXEC SQL CLOSE my_cursor; ...

避免无限循环。 如果WHENEVER SQLERROR GOTO语句分支到包含可执行SQL语句的错误处理例程，则如果SQL语句因错误而失败，则程序可能会进入无限循环。您可以通过在SQL语句之前编写WHENEVER SQLERROR CONTINUE来避免这种情况，如以下示例所示：

执行SQL WHENEVER SQLERROR GOTO sql_error; ...sql_error：EXEC SQL WHENEVER SQLERROR CONTINUE; EXEC SQL ROLLBACK工作版; ...

如果没有WHENEVER SQLERROR CONTINUE语句， ROLLBACK错误将再次调用例程，启动无限循环。

不小心使用WHENEVER会导致问题。例如，如果DELETE语句设置NOT FOUND ，则以下代码进入无限循环，因为没有行符合搜索条件：

- 不当使用WHENEVER ......EXEC SQL WHENEVER NOT FOUND GOTO no_more; LOOP EXEC SQL FETCH emp_cursor INTO：emp_name，：salary; ...ENDLOOP; no_more：EXEC SQL DELETE FROM EMP WHERE EMPNO =：emp_number; ...

在下一个示例中，您可以通过重置GOTO目标来正确处理NOT FOUND条件：

- 正确使用WHENEVER ......EXEC SQL WHENEVER NOT FOUND GOTO no_more; LOOP EXEC SQL FETCH emp_cursor INTO：emp_name，：salary; ...ENDLOOP; no_more：EXEC SQL WHENEVER NOT FOUND GOTO no_match; EXEC SQL DELETE FROM EMP WHERE EMPNO =：emp_number; ...no_match：......

保持可寻址性。 使用允许本地和全局标识符的宿主语言，确保由WHENEVER GOTO语句控制的所有SQL语句都可以分支到GOTO标签。下面的代码将导致编译时错误，因为在labelA FUNC1不FUNC2 INSERT语句的范围之内：

FUNC1 BEGIN EXEC SQL WHENEVER SQLERROR GOTO labelA; EXEC SQL DELETE FROM EMP WHERE DEPTNO =：dept_number; ...labelA：......结束; FUNC2 BEGIN EXEC SQL插入EMP（JOB）值（：job_title）; ...结束;

WHENEVER GOTO语句分支的标签必须与语句位于同一预编译文件中。

错误后返回。 如果程序在处理错误后必须返回，请使用DO routine_call操作。或者，您可以测试SQLCODE的值，如以下示例所示：

EXEC SQL UPDATE EMP SET SAL = SAL * 1.10; IF sqlca.sqlcode <0 THEN  - 处理错误EXEC SQL DROP INDEX EMP_INDEX; ...

只需确保没有WHENEVER GOTO或WHENEVER STOP语句处于活动状态。

8.5关于使用Oracle通信区域

同样，SQLCA处理标准SQL通信; Oracle通讯区（ ORACA）处理Oracle通信。如果您需要有关运行时错误和状态更改的更多信息，请使用ORACA。它包含一组扩展的诊断工具。但是，使用ORACA是可选的，因为它增加了运行时开销。

除了帮助您诊断问题外，ORACA还允许您监视程序对Oracle资源（如SQL语句执行程序和游标缓存）的使用情况。

在允许本地和全局声明的主机语言中，您的程序可以有多个ORACA。例如，它可能有一个全局ORACA和几个本地ORACA。访问本地ORACA受到程序范围的限制。Oracle仅将信息返回给“活动”ORACA。只有在提交或回滚后才能使用该信息。

8.5.1声明ORACA

要声明ORACA，只需在主机语言源文件中包含它（使用EXEC SQL INCLUDE语句），如下所示：

*包括Oracle通信区域（ORACA）。EXEC SQL包含ORACA

ORACA必须在声明部分之外声明。

预编译程序时， INCLUDE ORACA语句将被多个程序变量声明替换。这些声明允许Oracle与您的程序进行通信。

8.5.2启用ORACA

要启用ORACA，必须在命令行上指定ORACA选项

ORACA = YES

或者内联

EXEC ORACLE选项（ORACA = YES）;

然后，您必须通过在ORACA中设置标志来选择适当的运行时选项。

8.5.3 ORACA中有哪些内容？

ORACA包含选项设置，系统统计信息和扩展诊断，例如

  • SQL语句文本（可以指定保存文本的时间）

  • 发生错误的文件的名称

  • 错误在文件中的位置

  • 游标缓存错误和统计信息

  • 0 - 禁用堆一致性检查（默认值）。

  • 1 - 启用堆一致性检查。

8.5.22 ORANEX
