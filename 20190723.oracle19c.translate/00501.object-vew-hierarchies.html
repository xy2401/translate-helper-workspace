6.14对象视图层次结构

对象视图层次结构是一组对象视图，每个对象视图基于类型层次结构中的不同类型。视图层次结构中的子视图是在超级视图下创建的，类似于在超类型下创建类型层次结构中的子类型的方式。

视图层次结构中的每个对象视图都填充有单个类型的对象，但是给定视图上的查询也隐式地查询其子视图。因此，对象视图层次结构为您提供了一种简单的方法来构建查询，这些查询可以返回给定特定级别或更高级别的多态对象集。

如果已基于此类型层次结构创建了对象视图层次结构，并且在每种类型上构建了对象视图，则可以查询与您感兴趣的特化级别相对应的对象视图。例如，您可以查询student_typ的视图以获取仅包含学生的结果集，包括兼职学生。

您可以将对象视图层次结构的根视图基于类型层次结构中的任何类型：您不需要以根类型启动对象视图层次结构。您也不需要将对象视图层次结构扩展到类型层次结构的每个叶子或覆盖每个分支。但是，您不能跳过下降线中的干预子类型。任何子视图必须基于其直接超级视图类型的直接子类型。

就像一个类型可以有多个兄弟子类型一样，对象视图可以有多个兄弟子视图。但是，基于给定类型的子视图只能参与一个对象视图层次结构：两个不同的对象视图层次结构不能各自具有基于相同子类型的子视图。

子视图从其超级视图继承对象标识符（OID）。无法在任何子视图中明确指定OID。

根视图可以使用WITH OBJECT ID子句显式指定对象标识符。如果OID是系统生成的，或者根视图中未指定子句，则只有在根视图基于同样使用系统生成的OID的表或视图时，才能创建子视图。

视图下的查询确定视图是否可更新。要使视图可更新，其查询必须不包含连接，集合运算符，聚合函数， GROUP BY子句， DISTINCT子句，伪列或表达式。这同样适用于子视图。

如果视图不可更新，则可以定义INSTEAD OF触发器以执行适当的DML操作。请注意，子视图不会继承INSTEAD OF触发器。

视图层次结构中的所有视图必须位于同一模式中。

注意：

您可以创建不可实例化的类型的视图。不可实例化的类型不能具有实例，因此通常在创建这种类型的对象视图时没有意义。但是，不可实例化的类型可以具有可实例化的子类型。通过创建不可实例化类型的对象视图，您可以将对象视图层次结构基于包含不可实例化类型的类型层次结构。

6.14.1创建对象视图层次结构

您可以通过在根视图下创建子视图来构建对象视图层次结构。您可以通过在CREATE VIEW语句中使用UNDER关键字来执行此操作，如示例6-17中所示 。

相同的对象视图层次结构可以基于不同的底层存储模型。换句话说，底层表的各种布局或设计可以产生相同的对象视图层次结构。底层存储模型的设计会影响对象视图层次结构的性能和可更新性。

描述了三种可能的存储模型。在第一个是平面模型，对象视图层次结构中的所有视图都基于同一个表。在第二个水平模型中，每个视图与不同的表一一对应。在第三个是垂直模型，视图是使用连接构建的。

要执行任何这些存储模型，请首先创建示例6-16中所示的类型。

示例6-16为存储模型示例创建类型

CREATE TYPE person_typ AS OBJECT（ssn NUMBER，名称VARCHAR2（30），地址VARCHAR2（100））NOT FINAL; / CREATE TYPE student_typ UNDER person_typ（deptid NUMBER，major VARCHAR2（30））NOT FINAL; / CREATE TYPE employee_typ UNDER person_typ（ empid NUMBER，mgr VARCHAR2（30））; /

话题：

  • 平面模型

  • 水平模型

  • 垂直模型

6.14.1.1平面模型

在平面模型中，层次结构中的所有视图都基于同一个表。

在以下示例中，单个表AllPersons包含person_typ ， student_typ和employee_typ所有属性的列。

- 需要Ex。6-16 CREATE TABLE AllPersons（typeid NUMBER（1），ssn NUMBER，name VARCHAR2（30），address VARCHAR2（100），deptid NUMBER，major VARCHAR2（30），empid NUMBER，mgr VARCHAR2（30））;创建视图Person_v OF person_typ WITH OBJECT OID（ssn）AS SELECT ssn，name，address FROM AllPersons WHERE typeid = 1;创建视图Student_v OF student_typ UNDER Person_v AS SELECT ssn，name，address，deptid，major FROM AllPersons WHERE typeid = 2; CREATE VIEW Employee_v OF employee_typ UNDER Person_v AS SELECT ssn，name，address，empid，mgr FROM AllPersons WHERE typeid = 3;

扁平模型具有简单的优点，并且不支持索引和约束。它的缺点是：

  • 单个表不能包含超过1000列，因此平面模型对对象视图层次结构可包含的列总数施加1000列限制。

  • 对于不属于其类型的所有属性，表的每一行都将具有NULL。此类非尾随NULL可能会对性能产生负面影响。

垂直模型可以有效地处理SELECT * FROM root_view类型的查询，并且可以索引单个属性并对它们施加唯一约束。但是，要重新创建类型的实例，必须对从层次结构中的根中删除类型的每个级别执行对象标识符（OID）的连接。

6.14.2关于在层次结构中查询视图

您可以查询对象视图层次结构中的任何视图或子视图。

该查询返回您查询的视图的声明类型以及该类型的任何子类型的行。

因此，例如，在基于person_typ类型层次结构的对象视图层次结构中，您可以查询person_typ的视图以获取包含所有人员（包括学生和员工）的结果集;或者您可以查询student_typ的视图以获得仅包含学生的结果集，包括兼职学生。

在查询的SELECT列表中，您可以包含返回对象实例的REF()和VALUE()函数，也可以指定视图的声明类型的对象属性，例如name和ssn属性。 person_typ 。

如果指定函数，则返回对象实例，查询将返回多态结果集：即，它返回视图的声明类型和该类型的任何子类型的实例。

6.14.3视图层次结构的操作权限

通常，对具有子视图的视图的查询仅需要对所引用视图的SELECT特权，并且不需要对子视图具有任何显式特权。

例如，以下查询仅需要Person_v上的SELECT权限，但不需要任何子视图上的SELECT权限。 

SELECT * FROM Person_v;

但是，为子类型中添加但未由根类型使用的任何属性选择的查询也需要对所有子视图具有SELECT权限。此类子类型属性可能包含应合理需要其他访问权限的敏感信息。

例如，以下查询需要对Person_v以及Student_v ， Employee_v （以及Person_v任何其他子视图）上的SELECT权限，因为查询选择对象实例，从而获取子类型的所有属性。 

SELECT VALUE（p）FROM Person_v p;

要简化在整个视图层次结构上授予SELECT权限的过程，可以使用HIERARCHY选项。在授予用户对视图的SELECT权限时指定HIERARCHY选项也会隐式授予对视图的所有当前和未来子视图的SELECT权限。例如：

GRANT选择Person_v给用户使用HIERARCHY选项;
