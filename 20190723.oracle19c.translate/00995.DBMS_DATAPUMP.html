46.1 DBMS_DATAPUMP概述

DBMS_DATAPUMP提供以下支持和功能。

  • 源数据库和目标数据库可以具有不同的硬件，操作系统，字符集，时区和版本。

  • 支持Oracle Database 11g及更高版本中存在的所有对象类型和数据类型。

  • 无需使用任何中间文件即可在数据库之间传输数据和元数据。

  • 可以基于对象类型和对象的名称来移动数据库的子集。

  • 可以在导入时转换模式名称，数据文件名，表空间名称和数据。

  • 以前中止的导出和导入作业可以重新启动，而不会复制或省略原始作业中的任何数据或元数据。

  • 可以修改应用于导出或导入作业的资源。

  • 可以卸载和加载Oracle专有格式的数据。

46.2 DBMS_DATAPUMP安全模型

DBMS_DATAPUMP包的安全性通过角色实现。

DATAPUMP_EXP_FULL_DATABASE和DATAPUMP_IMP_FULL_DATABASE角色允许特权用户充分利用API。 Data Pump API将使用这些角色来确定是否应将特权应用程序角色分配给组成该作业的进程。

  • DATAPUMP_EXP_FULL_DATABASE

    DATAPUMP_EXP_FULL_DATABASE角色仅影响导出操作。它允许运行这些操作的用户执行以下操作：

    • 在其架构范围之外执行操作

    • 监视由其他用户启动的作业

    • 导出无特权用户无法引用的对象（例如，TABLESPACE定义）

    尽管SYS架构没有DATAPUMP_EXP_FULL_DATABASE分配DATAPUMP_EXP_FULL_DATABASE角色，但Data Pump执行的所有需要DATAPUMP_EXP_FULL_DATABASE角色的安全检查也将授予对SYS架构的访问权限。

  • DATAPUMP_IMP_FULL_DATABASE

    DATAPUMP_IMP_FULL_DATABASE角色仅影响Import和SQL_FILE操作。它允许运行这些操作的用户执行以下操作：

    • 在其架构范围之外执行操作

    • 监视由其他用户启动的作业

    • 导入非特权用户无法创建的对象（例如，DIRECTORY定义）

    虽然SYS模式没有DATAPUMP_IMP_FULL_DATABASE分配DATAPUMP_IMP_FULL_DATABASE角色，但Data Pump执行的所有需要DATAPUMP_IMP_FULL_DATABASE角色的安全检查也将授予对SYS模式的访问权限。

46.4 DBMS_DATAPUMP数据结构

DBMS_DATAPUMP包定义了OBJECT类型。本节中描述的类型在SYS模式中定义，供GET_STATUS函数使用。定义和使用这些类型的方式可能与您习惯的方式不同。请务必仔细阅读本节。

定义用于GET_STATUS过程的类型集合是特定于版本的，并且包括类型名称中的版本信息。一旦引入，这些类型将始终在Oracle数据库的未来版本中提供和支持，并且不会更改。但是，在Oracle数据库的未来版本中，可能会创建这些类型的新版本，以提供新的或不同的信息。这些类型的新版本将在类型名称中嵌入不同的版本信息。

例如，在Oracle数据库12c中，第1版（12.1），有一个sys.ku$_Status1210类型，并在接下来的Oracle数据库释放，有可能是一个sys.ku$_Status1310定义的类型。这两种类型都可以与GET_STATUS过程一起使用。

已为与GET_STATUS过程一起使用的GET_STATUS类型定义了公共同义词。这使得更容易使用类型和方法，您不必关心它们所在的实际类型名称或模式的更改。Oracle建议您尽可能使用这些同义词。

对于每种类型，都有特定于版本的同义词和通用同义词。例如，为sys.ku$_Status1210类型定义了特定于版本的同义词ku$_Status1210 。

通用同义词始终描述该类型的最新版本。例如，在Oracle数据库12c中，第1版（12.1），通用同义词ku$_Status被定义为ku$_Status1210 。在将来的版本中， sys.ku$Status1310可能有一个ku$_Status1310同义词。因为ku$_Status泛型同义词总是指向最新的定义，所以它将指向ku$_Status1310而不是ku$_Status1210 。

是否使用特定于版本的同义词或通用同义词的选择会对您的工作方式产生重大影响。使用特定于版本的名称可以保护您的代码免受Oracle数据库未来版本中的更改，因为这些类型将继续存在并得到支持。但是，访问新信息需要更改代码才能为每种类型使用新的同义词名称。使用通用名称意味着您始终需要最新的类型定义，并准备好处理不同版本的Oracle数据库中的更改。

当您使用的Oracle数据库版本发生更改时，需要重新编译通过通用同义词名称访问类型的任何C代码。

注意：

PL / SQL以外的语言必须确保其类型定义与特定于版本的定义正确对齐。

使用说明

GET_STATUS过程用于监视正在进行的作业的进度并接收错误通知。您可以使用mask参数请求各种类型的信息。KU$_STATUS_JOB_DESC和KU$_STATUS_JOB_STATUS值被分类为同步信息，因为信息驻留在主表中。KU$_STATUS_WIP和KU$_STATUS_JOB_ERROR值被归类为异步，因为可以通过数据泵体系结构中的各个层随时生成体现这些类型信息的消息。

  • 如果only请求同步信息，则接口将忽略超时参数并简单地返回所请求的信息。

  • 如果请求异步信息，则接口将在返回客户端之前等待maximum超时秒数。如果收到请求的异步信息类型的消息，则呼叫将在超时秒之前完成。如果还请求了同步信息，则只要过程返回，就会返回该信息。

  • 如果job_state返回的GET_STATUS未指示终止作业，则该作业可能仍会在下次调用GET_STATUS之前终止。这将导致INVALID_HANDLE异常。或者，作业可能在调用GET_STATUS期间终止，这将导致NO_SUCH_JOB异常。呼叫者应该准备好处理这些情况。

错误处理

需要使用GET_STATUS过程处理两种类型的错误方案：

  • 其他过程调用导致的错误：例如， SET_PARAMETER过程可能会产生INCONSISTENT_ARGS异常。客户端应立即使用mask=8 （errors）和timeout=0调用GET_STATUS 。返回的ku $ _Status.error将包含一个ku $ _LogEntry，它更详细地描述了不一致性。

  • 从异步客户端（或多个）事件而导致的错误：一个例子可能是Table already exists试图创建一个表时。ku$_Status.error将包含一个ku$_LogEntry其中包含正确排序的所有错误行（来自所有处理层，其中添加了有关错误的上下文）。

作业开始后，客户端的主处理循环通常包括对GET_STATUS的调用，其中包含无限超时（-1）“监听” KU$_STATUS_WIP和KU$_STATUS_JOB_ERROR消息。如果请求了状态，则JOB_STATUS信息也将在请求中。

解释ku $ _Status时，应使用以下准则：

  • ku$_Status.ku$_JobStatus.percent_done仅指在作业中处理的数据量。计算中不考虑元数据。使用以下公式确定：

    • EXPORT或网络IMPORT-- (bytes_processed/estimated_bytes) * 100

    • IMPORT-- (bytes_processed/total_expected_bytes) * 100

    • SQL_FILE或仅估计EXPORT--如果不完成则为0.00或100 。 00如果完成

    在计算percent_done不考虑QUERY和PARTITION_EXPR数据过滤器的影响。

    预计返回的状态将由调用者转换为更加用户友好的状态。例如，当完成百分比不为零时，可以使用以下公式生成完成时间的估计值：

    （（SYSDATE  - 开始时间）/ ku $ _Status.ku $ _JobStatus.percent_done）* 100

  • 调用者不应使用ku$_Status.ku$_JobStatus.percent_done来确定作业是否已完成。相反，调用者应该只依赖于job_state找到的作业状态。

如果非零，则在作业重新启动时导致在上次执行作业时“正在进行”的操作被跳过。跳过仅适用于导入作业。此机制允许用户跳过触发致命错误并导致作业提前终止的操作。重启时可以跳过多个操作。日志文件将标识跳过哪些操作。如果正在处理域索引，则即使仅在域索引的子组件中发生错误，也会跳过所有域索引。

将跳过的操作的描述输入到日志文件中。对于作业中的初始START_JOB ，将忽略skip_current 。

如果为零，则重新启动时不会丢失任何数据或元数据。

abort_step

值必须为0。在此参数中插入0以外的值会产生意想不到的后果。

cluster_ok

如果= 0，则在当前实例上启动所有工作程序。否则，工作人员将在作业可用的实例上启动。

服务名称

如果指定，则表示用于将作业约束到特定实例或特定资源组的服务名称。

例外

  • INVALID_HANDLE 。指定的句柄未附加到数据泵作业。

  • INVALID_STATE 。此异常的原因可能是以下任何一种情况：

    • 没有为Export，非网络导入或SQL_FILE作业定义任何文件

    • 尚未调用ADD_FILE过程来定义SQL_FILE作业的输出

    • 尚未为可传输导入作业定义TABLESPACE_DATAFILE参数

    • 尚未为可传输或表空间模式导出或网络作业定义TABLESPACE_EXPR元数据筛选器

    • 在Import或SQL_FILE作业上设置的转储文件不完整或缺少主表规范

  • INVALID_OPERATION 。无法从转储文件集还原主表。

  • INTERNAL_ERROR 。作业启动时检测到不一致。可通过GET_STATUS程序获得更多信息。

  • SUCCESS_WITH_INFO 。该过程成功，但可通过GET_STATUS程序获得更多信息。

  • NO_SUCH_JOB 。指定的作业不存在。

使用说明

  • 调用此过程以请求启动或重新启动相应的作业时，作业的状态将从“定义”或“空闲”状态更改为“执行”状态。

  • 如果在START_JOB过程之前未调用SET_PARALLEL过程，则作业中使用的初始并行级别将为1。如果SET_PARALLEL在之前的工作开始调用， degree由过去的指定SET_PARALLEL调用确定作业的并行性。重新启动时，并行性由作业的先前并行设置确定，除非它被另一个SET_PARALLEL调用覆盖。

  • 要重新启动已停止的作业，必须在执行START_JOB过程之前执行ATTACH功能。

46.5.16 STOP_JOB程序

此过程终止作业，但可选地，保留作业的状态。

句法

DBMS_DATAPUMP.STOP_JOB（处理IN NUMBER，立即IN NUMBER DEFAULT 0，keep_master IN NUMBER DEFAULT NULL，延迟IN NUMBER DEFAULT 60）;

参数

表46-27 STOP_JOB过程参数

参数 描述

处理

工作的把柄。当前会话必须先通过调用OPEN或ATTACH函数附加到句柄。在该过程结束时，用户与手柄分离。

即时

如果非零，则立即中止工作进程。这会快速停止作业，但如果作业重新启动，则必须重新运行部分作业。

如果为零，则允许工作进程在终止之前完成其当前工作项（元数据或表数据）。当工人完成当前工作时，工作处于停止待处理状态。

keep_master

如果非零，则在作业停止时保留主表。如果为零，则在作业停止时删除主表。如果删除了主表，则该作业将无法重新启动。如果在导出作业期间删除了主表，则会删除创建的转储文件。

延迟

等待强制分离其他连接会话的秒数。延迟允许通知作业的其他会话已执行停止。作业一直运行，直到所有客户端都已分离或延迟已满足。如果未指定延迟，则默认延迟为60秒。如果使用较短的延迟，客户端可能无法通过GET_STATUS过程检索作业的最终消息。

例外

  • INVALID_HANDLE 。指定的句柄未附加到数据泵作业。

  • INVALID STATE 。该工作已经在停止或完成。

  • SUCCESS_WITH_INFO 。该过程成功，但可通过GET_STATUS程序获得更多信息。

  • NO_SUCH_JOB 。指定的作业不存在。

使用说明

  • 此过程用于请求相应的作业停止执行。

  • 处于执行状态的作业的终止可能需要几分钟才能以有序的方式完成。

  • 对于“定义”，“空闲”或“完成”状态中的作业，此过程在功能上等同于DETACH过程。

  • 一旦作业停止，可以使用ATTACH函数和START_JOB过程重新启动它，前提是主表和转储文件集保持不变。

  • 如果未指定KEEP_MASTER参数，并且作业处于“定义”状态或具有“可传输”模式，则会删除主表。否则，保留主表。

此过程运行作业，直到它正常完成或因某些其他原因停止。

句法

参数

表46-28 WAIT_FOR_JOB过程参数

参数 描述

处理

工作的把柄。当前会话必须先通过调用OPEN或ATTACH函数附加到句柄。在该过程结束时，用户与手柄分离。

job_state

作业停止执行时的状态;要么STOPPED或COMPLETED 。
