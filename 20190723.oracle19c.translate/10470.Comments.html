评论

您可以创建两种类型的注释：

  • SQL语句中的注释存储为执行SQL语句的应用程序代码的一部分。

  • 与单个模式或非模式对象关联的注释与对象本身的元数据一起存储在数据字典中。

SQL语句中的注释

注释可以使您的应用程序更易于阅读和维护。例如，您可以在语句中包含注释，该语句描述应用程序中语句的用途。除了提示之外，SQL语句中的注释不会影响语句执行。请参阅使用此特定评论形式的提示 。

注释可以出现在语句中的任何关键字，参数或标点符号之间。您可以通过两种方式在语句中包含注释：

  • 用斜杠和星号（/ *）开始注释。继续评论的文本。此文本可以跨越多行。用星号和斜杠（* /）结束注释。打开和终止字符不需要通过空格或换行符与文本分开。

  • 用 - （两个连字符）开始评论。继续评论的文本。此文本无法扩展到新行。用换行符结束评论。

用于输入SQL的一些工具有其他限制。例如，如果您使用的是SQL * Plus，则默认情况下，多行注释中不能包含空行。有关更多信息，请参阅用作数据库接口的工具的文档。

SQL语句可以包含两种样式的多个注释。注释文本可以包含数据库字符集中的任何可打印字符。

您可以使用COMMENT命令与架构对象评论关联（表，视图，物化视图，操作，索引类型，挖掘模型）或使用一个nonschema对象（版） COMMENT命令。您还可以在列上创建注释，该列是表架构对象的一部分。与schema和nonschema对象关联的注释存储在数据字典中。请参阅COMMENT这种形式的评论的描述。

提示

提示是SQL语句中的注释，它将指令传递给Oracle数据库优化器。优化程序使用这些提示为语句选择执行计划，除非存在阻止优化程序执行此操作的某些条件。

Oracle7中引入了提示，如果优化程序生成次优计划，则用户几乎没有追索权。现在，Oracle提供了许多工具，包括SQL Tuning Advisor，SQL计划管理和SQL性能分析器，以帮助您解决优化器无法解决的性能问题。Oracle强烈建议您使用这些工具而不是提示。这些工具远远优于提示，因为在持续使用时，它们会随着数据和数据库环境的变化而提供新的解决方案。

应该谨慎使用提示，并且只有在收集了相关表的统计信息并使用EXPLAIN PLAN语句评估优化器计划而没有提示之后。更改数据库条件以及后续版本中的查询性能增强功能会对代码中的提示如何影响性能产生重大影响。

本节的其余部分提供了一些常用提示的信息。如果您决定使用提示而不是更高级的调优工具，请注意使用提示所产生的任何短期利益可能不会继续导致长期性能的提高。

Oracle数据库忽略提示，并且在以下情况下不会返回错误：

  • 提示包含拼写错误或语法错误。但是，数据库确实考虑了同一注释中的其他正确指定的提示。

  • 包含提示的注释不遵循DELETE ， INSERT ， MERGE ， SELECT或UPDATE关键字。

在提示中指定查询块

您可以在许多提示中指定可选的查询块名称，以指定提示适用的查询块。此语法允许您在外部查询中指定适用于内联视图的提示。

查询块参数的语法是@ queryblock形式，其中queryblock是指定查询中查询块的标识符。queryblock标识符可以是系统生成的，也可以是用户指定的。在提示适用的查询块本身中指定提示时，省略@queryblock语法。

  • 系统生成的标识符可以通过使用EXPLAIN PLAN获取查询来获得。可以通过使用NO_QUERY_TRANSFORMATION提示运行EXPLAIN PLAN来查询NO_QUERY_TRANSFORMATION查询块名称。请参见NO_QUERY_TRANSFORMATION提示 。

  • 可以使用QB_NAME提示设置用户指定的名称。请参阅QB_NAME提示 。

ALL_ROWS提示


all_rows_hint.eps插图的描述

ALL_ROWS提示指示优化器优化语句块，目标是最大吞吐量，这是最小的总资源消耗。例如，优化程序使用查询优化方法来优化此语句以获得最佳吞吐量：

SELECT / * + ALL_ROWS * / employee_id，last_name，salary，job_id FROM employees WHERE employee_id = 107;

如果在SQL语句中指定ALL_ROWS或FIRST_ROWS提示，并且数据字典没有关于语句访问的表的统计信息，则优化程序使用默认统计值（例如此类表的已分配存储）来估计缺少统计数据并随后选择执行计划。这些估计可能不如DBMS_STATS包收集的估计值那么准确，因此您应该使用DBMS_STATS包来收集统计信息。

如果为访问路径或连接操作指定提示以及ALL_ROWS或FIRST_ROWS提示，则优化程序优先考虑提示指定的访问路径和连接操作。

APPEND提示


插图描述append_hint.eps

APPEND提示指示优化器使用直接路径INSERT和INSERT语句的子查询语法。

  • 常规INSERT是串行模式的默认设置。在串行模式下，只有包含APPEND提示时才能使用直接路径。

  • 直接路径INSERT是并行模式的默认设置。在并行模式下，只有在指定NOAPPEND提示时才能使用常规插入。

INSERT是否并行的决定与APPEND提示无关。

在直接路径INSERT ，数据附加到表的末尾，而不是使用当前分配给表的现有空间。因此，直接路径INSERT可以比传统INSERT快得多。

只有INSERT语句的子查询语法支持APPEND提示，而不支持VALUES子句。如果使用VALUES子句指定APPEND提示，则会忽略该提示并使用常规插入。要将直接路径INSERT与VALUES子句一起使用，请参阅“ APPEND_VALUES提示 ” 。

APPEND_VALUES提示


附图说明append_values_hint.eps

APPEND_VALUES提示指示优化器使用带有VALUES子句的直接路径INSERT 。如果未指定此提示，则使用常规INSERT 。

在直接路径INSERT ，数据附加到表的末尾，而不是使用当前分配给表的现有空间。因此，直接路径INSERT可以比传统INSERT快得多。

APPEND_VALUES提示可用于大大提高性能。其用途的一些例子是：

  • 在Oracle调用接口（OCI）程序中，当使用大型数组绑定或数组绑定与行回调时

  • 在PL / SQL中，使用带有VALUES子句的INSERT语句的FORALL循环加载大量行时

只有INSERT语句的VALUES子句支持APPEND_VALUES提示。如果使用INSERT语句的子查询语法指定APPEND_VALUES提示，则会忽略它并使用常规插入。要将直接路径INSERT与子查询一起使用，请参阅“ APPEND提示 ” 。

集装箱提示


container_hint.eps插图的描述

CONTAINERS提示在多租户容器数据库（CDB）中很有用。您可以在包含CONTAINERS()子句的SELECT语句中指定此提示。通过这样的语句，您可以在CDB或应用程序容器中的所有容器中查询指定表或视图中的数据。

  • 要查询CDB中的数据，您必须是连接到CDB根目录的普通用户，并且表或视图必须存在于根和所有PDB中。该查询返回CDB根目录和所有打开的PDB中的表或视图中的所有行。

  • 要查询应用程序容器中的数据，您必须是连接到应用程序根目录的公共用户，并且表或视图必须存在于应用程序根目录和应用程序容器中的所有PDB中。该查询返回应用程序根目录中的表或视图以及应用程序容器中所有打开的PDB中的所有行。

包含CONTAINERS()子句的语句在每个查询的PDB中生成并执行递归SQL语句。您可以使用CONTAINERS提示将默认PDB提示传递给每个递归SQL语句。对于hint ，您可以指定任何适合SELECT语句的SQL提示。

INDEX提示


说明index_hint.eps的说明

（请参阅在提示中指定查询块 ， tablespec :: = ， indexspec :: = ）

INDEX提示指示优化器对指定的表使用索引扫描。您可以将INDEX提示用于基于函数，域，B树，位图和位图连接索引。

提示的行为取决于indexspec规范：

  • 如果INDEX提示指定单个可用索引，则数据库将对此索引执行扫描。优化器不考虑对表进行全表扫描或扫描另一个索引。

  • 有关多个索引组合的提示，Oracle建议使用INDEX_COMBINE而不是INDEX ，因为它是一个更通用的提示。如果INDEX提示指定了可用索引的列表，则优化程序会考虑列表中每个索引的扫描成本，然后以最低成本执行索引扫描。如果此类访问路径的成本最低，则数据库还可以选择从此列表中扫描多个索引并合并结果。数据库不考虑对提示中未列出的索引进行全表扫描或扫描。

  • 如果INDEX提示未指定任何索引，则优化程序会考虑表上每个可用索引的扫描成本，然后以最低成本执行索引扫描。如果此类访问路径的成本最低，则数据库还可以选择扫描多个索引并合并结果。优化器不考虑全表扫描。

例如：

SELECT / * + INDEX（employees emp_department_ix）* / employee_id，department_id FROM employees WHERE department_id> 50;

PARALLEL提示指示优化器使用指定数量的并发服务器进行并行操作。此提示会覆盖PARALLEL_DEGREE_POLICY初始化参数的值。它适用于SELECT ， INSERT ， MERGE ， UPDATE和DELETE部分，以及表扫描部分。如果违反任何并行限制，则忽略提示。

注意：

如果还进行排序或分组操作，则可以使用的服务器数量是PARALLEL提示值的两倍。

对于语句级别的PARALLEL提示 ：

  • PARALLEL ：语句始终并行运行，数据库计算并行度，可以是2或更大。

  • PARALLEL （ DEFAULT ）：优化程序计算的并行度等于所有参与实例上可用的CPU数量乘以PARALLEL_THREADS_PER_CPU初始化参数的值。

  • PARALLEL （ AUTO ）：数据库计算并行度，可以是1或更大。如果计算的并行度为1，则语句将按顺序运行。

  • PARALLEL （ MANUAL ）：强制优化器被强制使用语句中对象的并行设置。

  • PARALLEL （ integer ）：优化程序使用integer指定的并行度。

在以下示例中，优化程序计算并行度。该语句始终并行运行。

SELECT / * + PARALLEL * / last_name FROM employees;

在以下示例中，优化程序计算并行度，但该度可能为1，在这种情况下，语句将以串行方式运行。

SELECT / * + PARALLEL（AUTO）* / last_name FROM employees;

在下面的示例中， PARALLEL提示建议优化器使用当前对表本身有效的并行度，即5：

CREATE TABLE parallel_table（col1 number，col2 VARCHAR2（10））PARALLEL 5; SELECT / * + PARALLEL（MANUAL）* / col2 FROM parallel_table;

对于对象级别的PARALLEL提示 ：

  • PARALLEL ：查询协调器应检查初始化参数的设置以确定默认的并行度。

  • PARALLEL （ integer ）：优化程序使用integer指定的并行度。

  • PARALLEL （ DEFAULT ）：优化程序计算的并行度等于所有参与实例上可用的CPU数量乘以PARALLEL_THREADS_PER_CPU初始化参数的值。

在以下示例中， PARALLEL提示将覆盖employees表定义中指定的并行度：

SELECT / * + FULL（hr_emp）PARALLEL（hr_emp，5）* / last_name FROM employees hr_emp;

表2-24负载分布值

分配 描述

没有

没有分发。即查询和加载操作被合并到每个查询服务器中。所有服务器都将加载所有分区。这种分布不足有助于避免在没有偏斜的情况下分配行的开销。由于空段或语句中的谓词过滤掉查询评估的所有行，可能会出现偏差。如果由于使用此方法而发生偏斜，则改为使用RANDOM或RANDOM_LOCAL分布。

随机

此方法以循环方式将生产者的行分配给消费者。当输入数据高度倾斜时，请使用此分配方法。

RANDOM_LOCAL

此方法将生成器中的行分配给一组服务器，这些服务器负责维护给定的一组分区。两个或多个服务器可以加载同一个分区，但没有服务器加载所有分区。当输入数据偏斜并且由于内存约束而无法组合查询和加载操作时，请使用此分发方法。

HASH ， HASH

注意 ：如果外部表未在分区键上进行分区或未等分，则优化程序将忽略此提示。

NONE ， NONE
