2错误处理和诊断

本章讨论适用于Pro * FORTRAN的错误报告和恢复。它补充了Oracle预编译程序员指南的第8章。

您将学习如何声明和使用SQLSTA状态变量和SQLCOD状态变量，以及如何包含SQL通信区域（SQLCA）。您还将学习如何声明和启用Oracle通信区域（ORACA）。

本章包含以下部分：

  • 错误处理替代方案

  • 关于MODE = {ANSI | ANSI14}时使用状态变量

  • 关于使用SQL通信区域

  • 关于使用Oracle通信区域

2.1错误处理替代方案

Pro * FORTRAN预编译器支持四个状态变量，用作错误处理机制：

  • SQLCOD

  • SQLSTA

  • SQLCA（使用WHENEVER声明）

  • ORACA

预编译器MODE选项控制ANSI / ISO合规性。SQLCOD，SQLSTA和SQLCA变量的可用性取决于MODE设置。无论MODE设置如何，您都可以声明和使用ORACA变量。有关更多信息，请参阅“ 关于使用Oracle通信区 ” 。

当MODE = {ORACLE | ANSI13}时，您必须声明SQLCA状态变量。接受SQLCOD和SQLSTA声明（不推荐），但不会将其识别为状态变量。有关更多信息，请参阅“ 关于使用SQL通信区 ” 。

当MODE = {ANSI | ANSI14}时，您可以使用SQLCOD，SQLSTA和SQLCA变量中的任何一个，两个或全部三个。要确定哪种变量（或变量组合）最适合您的应用，请参阅“ 关于在MODE = {ANSI | ANSI14}时使用状态变量 ” 。

2.1.1 SQLCOD和SQLSTA

SQLCOD存储错误代码和“未找到”条件。它仅保留与SQL-89的兼容性，并已在SQL：1999及该标准的后续版本中删除。

与SQLCOD不同，SQLSTA存储错误和警告代码并使用标准化编码方案。执行SQL语句后，Oracle服务器将状态代码返回到当前范围内的SQLSTA变量。状态代码指示SQL语句是成功执行还是引发异常（错误或警告条件）。为了提高互操作性 （系统轻松交换信息的能力），SQL标准预定义了所有常见的SQL异常。

2.1.2 SQLCA

SQLCA是一种类似记录的主机语言数据结构。Oracle在每个可执行的 SQL语句之后更新SQLCA。（在声明性陈述之后，SQLCA值未定义。）通过检查存储在SQLCA中的Oracle返回码，您的程序可以确定SQL语句的结果。这可以通过两种方式完成：

  • 使用WHENEVER语句进行隐式检查

  • 显式检查SQLCA变量

您可以使用WHENEVER语句，对SQLCA变量进行显式检查，或者同时执行这两种操作。通常，使用WHENEVER语句是首选，因为它更容易，更便携，并且符合ANSI标准。

2.1.3 ORACA

当需要有关运行时错误的信息比SQLCA提供的更多信息时，可以使用ORACA，其中包含游标统计信息，SQL语句数据，选项设置和系统统计信息。

ORACA是可选的，无论MODE设置如何，都可以声明。有关ORACA状态变量的更多信息，请参阅“ 关于使用Oracle通信区 ” 。

2.2关于MODE = {ANSI | ANSI14}时使用状态变量

当MODE = {ANSI | ANSI14}时，您必须至少声明一个 - 您可以声明两个或全部三个 - 以下状态变量：

  • SQLCOD

  • SQLSTA

  • SQLCA

在检查可执行的SQL和PL / SQL语句之后，通过使用您自己的代码显式检查SQLCOD或SQLSTA，您的程序可以检索最新的可执行SQL语句的结果。您的程序也可以隐式检查SQLCA（使用WHENEVER SQLERROR和WHENEVER SQLWARNING语句），也可以显式检查SQLCA变量。

注意：

当MODE = {ORACLE | ANSI13}时，您必须声明SQLCA状态变量。有关更多信息，请参阅使用SQL通信区域。

2.2.1一些历史信息

Oracle Pro * FORTRAN预编译器对状态变量和变量组合的处理从1.5版开始发展。

2.2.2版本1.5

Pro * FORTRAN版本1.5假定存在状态变量SQLCOD，无论它是否在声明部分中声明;事实上，预编译器从未注意到SQLCOD是否被声明 - 它只是假定它是。仅当存在SQLCA的INCLUDE时，SQLCA才会用作状态变量。

2.2.3版本1.6

从Pro * FORTRAN版本1.6开始，预编译器不再假定存在SQLCOD状态变量且不需要。预编译器要求声明SQLCA，SQLCOD或SQLSTA 中的至少一个。

当且仅当满足以下条件中的至少一个时，SQLCOD才被识别为状态变量：

  • 它在Declare Section中声明，具有完全正确的数据类型。

  • 预编译器找不到其他状态变量。

如果预编译器在声明段找到SQLSTA声明（ 完全正确类型的课程），或者找到一个包括SQLCA的，它不会擅自SQLCOD声明。

2.2.4版本1.7

由于Pro * FORTRAN版本1.5允许在声明部分之外声明SQLCOD变量，同时声明SQLCA，因此Pro * FORTRAN版本1.6及更高版本会出现兼容性问题。添加了一个新选项ASSUME_SQLCODE = {YES | NO}（默认为NO）以在版本1.6.7中修复此问题，并将其记录为版本1.7中的新功能。

当ASSUME_SQLCODE = YES，并且当SQLSTA和/或SQLCA被声明为状态变量时，预编译器假定声明SQLCOD是否在声明部分或正确类型中声明。这导致版本1.6.7及更高版本在这方面的行为类似于版本1.5。有关预编译器选项ASSUME_SQLCODE的信息，请参见ASSUME_SQLCODE 。

2.2.5关于声明状态变量

本节介绍如何声明SQLCOD和SQLSTA。有关声明SQLCA状态变量的信息，请参阅“ 关于声明SQLCA ” 。

2.2.6声明SQLCOD

SQLCOD必须在Declare Section 内部或外部声明为4字节整数变量。在下面的示例中，SQLCOD在Declare Section之外声明：

*声明主机和指标变量。EXEC SQL BEGIN DECLARE SECTION ...EXEC SQL END DECLARE SECTION *声明状态变量。INTEGER * 4 SQLCOD

如果在Declare Section之外声明，仅当ASSUME_SQLCODE = YES时，SQLCOD才被识别为状态变量。当MODE = {ORACLE | ANSI13 | ANSI14}时，将忽略SQLCOD变量的声明。

访问本地SQLCOD受其程序范围的限制。在每次SQL操作之后，Oracle都会将状态代码返回到当前范围内的SQLCOD。因此，您的程序可以通过显式检查SQLCOD或使用WHENEVER语句隐式检查来了解最新SQL操作的结果。

在特定编译单元中声明SQLCOD而不是SQLCA时，预编译器会为该单元分配内部SQLCA。您的主程序无法访问内部SQLCA。

2.2.7声明SQLSTA

SQLSTA 必须被声明为定义段内的5个字符数的字母数字字符串，如显示在下面的例子：

EXEC SQL BEGIN DECLARE SECTION ...CHARACTER * 5 SQLSTA ......EXEC SQL END DECLARE SECTION

当MODE = {ORACLE | ANSI13}时，将忽略SQLSTA声明。声明SQLCA是可选的。

2.3关于使用SQL通信区域

Oracle使用SQL通信区域（SQLCA）来存储在运行时传递给程序的状态信息。SQLCA是一种类似记录的FORTRAN数据结构，在每个可执行的SQL语句之后更新，因此它始终反映最新SQL操作的结果。要确定结果，可以使用自己的FORTRAN代码或使用WHENEVER语句隐式检查SQLCA中的变量。

当MODE = {ORACLE | ANSI13}时，需要SQLCA;如果未声明SQLCA，则会发生编译时错误。MODE = {ANSI | ANSI14}时，SQLCA是可选的，但如果没有SQLCA，则不能使用WHENEVER SQLWARNING语句。因此，如果要使用WHENEVER SQLWARNING语句，则必须声明SQLCA。

当MODE = {ANSI | ANSI14}时，您必须声明SQLSTA（请参阅“声明SQLSTA”）或SQLCOD（请参阅“声明SQLCOD”）或两者。SQLSTA状态变量支持SQL标准指定的SQLSTA状态变量。您可以在有或没有SQLCOD的情况下使用SQLSTA状态变量。有关更多信息，请参阅错误处理和诊断 。

2.3.1 SQLCA中有什么？

SQLCA包含有关SQL语句执行的运行时信息，例如Oracle错误代码，警告标志，事件信息，行处理计数和诊断。

2.3.2关于声明SQLCA

要声明SQLCA，只需在声明部分之外的Pro * FORTRAN源文件中包含它（使用EXEC SQL INCLUDE语句），如下所示：

*包括SQL通信区域（SQLCA）。EXEC SQL包含SQLCA

因为它是COMMON块，所以必须在Declare Section 之外声明SQLCA。此外，SQLCA必须位于CONNECT语句和第一个可执行FORTRAN语句之前。

您必须在包含SQL语句的每个子例程和函数中声明SQLCA。每次执行其中一个子例程或函数中的SQL语句时，Oracle都会更新COMMON块中保存的SQLCA。

通常，只有COMMON列表中变量的顺序和数据类型才有意义，而不是它们的名称。但是，您无法重命名SQLCA变量，因为预编译器会生成引用它们的代码。因此，SQLCA的所有声明必须相同。

预编译程序时，INCLUDE SQLCA语句将被多个允许Oracle与程序通信的变量声明替换。

2.3.3错误报告的关键组件

Pro * FORTRAN错误报告的关键组件取决于SQLCA中的几个字段。

2.3.4状态代码

每个可执行的SQL语句都返回SQLCA变量SQLCDE中的状态代码，您可以使用WHENEVER SQLERROR隐式检查或使用您自己的FORTRAN代码显式检查。

2.3.5警告标志

在SQLCA变量SQLWN0到SQLWN7中返回警告标志，您可以使用WHENEVER SQLWARNING或您自己的FORTRAN代码进行检查。这些警告标志对于检测Oracle未被视为错误的运行时条件非常有用。

2.3.6行处理计数

最近执行的SQL语句处理的行数记录在SQLCA变量SQLERD（3）中。对于OPEN游标上的重复FETCH，SQLERD（3）保持获取的行数的运行总和。

2.3.7解析错误偏移量

在执行SQL语句之前，Oracle必须解析它;检查它以确保它遵循语法规则并引用有效的数据库对象。如果Oracle发现错误，则会在SQLCA变量SQLERD（5）中存储偏移量，您可以显式检查该变量。偏移量指定SQL语句中解析错误开始时的字符位置。第一个角色占据零位。例如，如果偏移量为9，则解析错误从第十个字符开始。

如果您的SQL语句不会导致解析错误，Oracle会将SQLERD（5）设置为零。如果解析错误从第一个字符开始，Oracle也将SQLERD（5）设置为零，该字符占据零位。因此，仅当SQLCDE为负时才检查SQLERD（5），这意味着发生了错误。

2.3.8错误消息文本

SQLCA变量SQLEMC中提供了错误代码和Oracle错误消息。例如，您可以将以下语句放在错误处理例程中：

处理SQL执行错误。WRITE（*，10000）SQLEMC 10000 FORMAT（1X，70A1）EXEC SQL WHENEVER SQLERROR CONTINUE EXEC SQL ROLLBACK WORK RELEASE ...

最多存储消息文本的前70个字符。对于超过70个字符的消息，必须调用SQLGLM函数，这将在下面讨论。

2.3.9关于获取错误消息的全文

SQLCA可以容纳长度最多为70个字符的错误消息。要获取更长（或嵌套）错误消息的全文，您需要SQLGLM函数。如果连接到Oracle，则可以使用语法调用SQLGLM

CALL SQLGLM（MSGBUF，BUFLEN，MSGLEN）

哪里：

MSGBUF

是您希望Oracle存储错误消息的缓冲区。Oracle空白填充到此缓冲区的末尾。

BUFLEN

是一个整数变量，指定MSGBUF的最大长度（以字节为单位）。

MSGLEN

是一个整数变量，Oracle存储错误消息的实际长度。

Oracle错误消息的最大长度为512个字符，包括错误代码，嵌套消息和表和列名称等消息插入。SQLGLM返回的错误消息的最大长度取决于您为BUFLEN指定的值。在以下示例中，您使用SQLGLM来获取最多200个字符的错误消息：

*声明函数调用的变量。LOGICAL * 1 MSGBUF（200）INTEGER * 4 BUFLEN INTEGER * 4 MSGLEN DATA BUFLEN / 200 / EXEC SQL SQLERROR转到9000 ......*处理SQL执行错误。9000 WRITE（*，9100）9100 FORMAT（1X，'>>>检测到Oracle错误'，/）*获取并显示错误消息的全文。CALL SQLGLM（MSGBUF，BUFLEN，MSGLEN）WRITE（*，9200）（MSGBUF（J），J = 1，MSGLEN）9200 FORMAT（1X，200A1，/）......

在该示例中，仅在发生SQL错误时才调用SQLGLM。在调用SQLGLM 之前，始终确保SQLCOD为负数。如果在SQLCOD为零时调用SQLGLM，则会获得与先前SQL语句关联的消息文本。

2.3.10关于使用WHENEVER声明

默认情况下，Pro * FORTRAN预编译器会忽略Oracle错误和警告条件并继续处理（如果可能）。要执行自动条件检查和错误处理，您需要WHENEVER语句。

使用WHENEVER语句，您可以指定Oracle检测到错误，警告条件或“未找到”条件时要采取的操作。这些操作包括继续下一个语句，调用子例程，分支到带标签的语句或停止。

使用以下语法对WHENEVER语句进行编码：

EXEC SQL WHENEVER <condition> <action>

您可以让Oracle自动检查SQLCA是否存在以下任何一种情况，这些情况在错误处理和诊断中进行了描述：

  • SQLWARNING

  • SQLERROR

  • 未找到

当Oracle检测到上述条件之一时，您可以让程序执行以下任何操作：

  • 继续

  • DO subroutine_call

  • GOTO statement_label

  • 停

使用WHENEVER时......DO声明，适用于进入和退出子程序的通常规则。然而，传递参数给子程序是不允许的。此外，子例程不能返回值。

在以下示例中，WHENEVER SQLERROR DO语句用于处理特定错误：

注意子程序如何检查SQLCA中的变量以确定操作过程。有关WHENEVER条件和操作的详细信息，请参阅错误处理和诊断 。

2.3.11范围

因为WHENEVER是一个声明性陈述，它的范围是位置的，而不是逻辑的。它测试源文件中跟随它的所有可执行SQL语句，而不是程序逻辑流程。因此，在要测试的第一个可执行SQL语句之前编写WHENEVER语句。

WHENEVER语句一直有效，直到被另一个检查相同条件的WHENEVER语句取代。

小费：

您可能希望将WHENEVER语句放在包含SQL语句的每个程序单元的开头。这样，一个程序单元中的SQL语句将不会在另一个程序单元中引用WHENEVER操作，从而在编译或运行时导致错误。

2.4关于使用Oracle通信区域

SQLCA处理标准SQL通信。Oracle通信区域（ORACA）是一种类似的结构，您可以将其包含在程序中以处理特定于Oracle的通信。当您需要比SQLCA提供的更多运行时信息时，请使用ORACA。

除了帮助您诊断问题外，ORACA还允许您监视程序对Oracle资源的使用，例如SQL语句执行程序和游标缓存 ，这是为游标管理保留的内存区域。

2.4.2关于声明ORACA

要声明ORACA，只需在声明部分之外的Pro * FORTRAN源文件中包含它（使用EXEC SQL INCLUDE语句），如下所示：

*包括Oracle通信区域（ORACA）。EXEC SQL包含ORACA

因为它是COMMON块，所以必须在Declare Section 之外声明ORACA。此外，ORACA必须位于CONNECT语句和第一个可执行FORTRAN语句之前。

您可以在包含SQL语句的任何子例程或函数中重新声明ORACA。每次执行子例程或函数中的SQL语句时，Oracle都会更新COMMON中保存的ORACA。

通常，只有COMMON列表中变量的顺序和数据类型才有意义，而不是它们的名称。但是，您无法重命名ORACA变量，因为预编译器会生成引用它们的代码。因此，ORACA的所有声明必须相同。

2.4.3关于启用ORACA

要启用ORACA，必须在命令行或配置文件中将ORACA预编译器选项设置为YES。

ORACA = YES

或者内联

*启用ORACA。 EXEC ORACLE选项（ORACA = YES）

然后，您必须通过在ORACA中设置标志来选择适当的运行时选项。启用ORACA是可选的，因为它会增加运行时开销。默认设置为ORACA = NO。
