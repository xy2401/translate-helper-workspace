33.4创建包含文本挖掘的模型

了解如何创建包含文本挖掘的模型。

Oracle Data Mining支持VARCHAR2 ， CHAR ， CLOB ， BLOB和BFILE列中的非结构化文本，如下表所述：

表33-2可能包含非结构化文本的列数据类型

数据类型 描述

BFILE和BLOB

仅当您在创建模型时将列标识为文本时，Oracle Data Mining 才会将BLOB和BFILE解释为文本。如果未将列标识为文本，则CREATE_MODEL将返回错误。

CLOB

Oracle Data Mining将CLOB解释为文本。

CHAR

默认情况下，Oracle Data Mining将CHAR解释为分类。创建模型时，可以将CHAR列标识为文本。

VARCHAR2

Oracle Data Mining将VARCHAR2解释为文本长度> 4000的VARCHAR2 。

默认情况下，Oracle Data Mining将数据长度<= 4000的VARCHAR2解释为分类。创建模型时，可以将这些列标识为文本。

注意：

不支持文本嵌套列或作为监督数据挖掘中的目标。

下表中描述的设置控制模型中文本属性的术语提取过程。有关指定模型设置的说明，请参阅“指定模型设置” 。

表33-3文本的模型设置

设置名称 数据类型 设定价值 描述

ODMS_TEXT_POLICY_NAME

VARCHAR2（4000）

使用CTX_DDL.CREATE_POLICY创建的Oracle Text策略对象的名称

影响从非结构化文本中提取单个标记的方式。请参阅“ 创建文本策略 ” 。

ODMS_TEXT_MAX_FEATURES

整数

1 <= 值 <= 100000

传递给CREATE_MODEL的文档集（在每个文本列的所有文档中）中使用的最大功能数。

默认值为3000。

模型可以包括一个或多个文本属性。具有文本属性的模型还可以包括分类和数字属性。

要创建包含文本属性的模型：

  1. 创建Oracle Text策略对象..

  2. 指定“ 表33-3 ”中描述的型号配置设置。

  3. 指定必须将哪些列视为文本，并可选择为单个属性提供文本转换说明。

  4. 将模型设置和文本转换指令传递给DBMS_DATA_MINING.CREATE_MODEL 。

    注意：

    除O-Cluster之外的所有算法都可以支持非结构化文本列。

    不建议对关联规则使用非结构化文本（Apriori）。
