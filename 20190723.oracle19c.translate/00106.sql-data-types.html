使用正确和最具体的数据类型

为数据库应用程序创建的每个数据库列使用正确且最具体的数据类型可提高数据完整性，降低存储要求并提高性能。

话题：

正确的数据类型如何提高数据完整性

正确的数据类型通过充当约束来增加数据完整性。例如，如果对日期列使用日期时间数据类型，则只能在该列中存储日期。但是，如果对列使用字符或数字数据类型，则最终会有人存储不表示日期的字符或数字值。您可以编写代码来防止此问题，但使用正确的数据类型会更有效。因此，在字符数据类型，数字数据类型中的数字以及日期时间数据类型中的日期和时间中存储字符。

最具体的数据类型如何降低存储要求

除了使用正确的数据类型外，还要使用最具体的长度或精度;例如：

  • 创建用于最多n个字符的字符串的VARCHAR2列时，请指定VARCHAR2( n ) 。

  • 创建用于整数的列时，请使用数据类型NUMBER(38)而不是NUMBER 。

除了充当约束因而增加数据完整性之外，长度和精度也会影响存储要求。

如果为每个列提供其数据类型的最大长度或精度，那么您的应用程序将不必要地分配许多兆字节的RAM。例如，假设查询选择10个VARCHAR2(4000)列，批量提取操作返回100行。您的应用程序必须分配的RAM是10 x 4,000 x 100 - 几乎4 MB。相反，如果列长度为80，则应用程序必须分配的RAM为10 x 80 x 100-about 78 KB。这种差异对于单个查询很重要，您的应用程序将同时处理许多查询。因此，您的应用程序必须为每个连接分配4 MB或78 KB的RAM。

因此，不要仅为某个列提供其数据类型的最大长度或精度，因为您可能需要稍后增加该属性。如果必须在创建后更改列，请使用ALTER TABLE语句。例如，要增加列的长度，请使用：

ALTER TABLE table_name MODIFY column_name VARCHAR2（ greater_number ）

注意：

如果MAX_STRING_SIZE初始化参数为EXTENDED ，则VARCHAR2 ， NVARCHAR2和RAW数据类型的最大长度为32,767字节。

注意：

不要使用VARCHAR数据类型。请改用VARCHAR2数据类型。尽管VARCHAR数据类型当前与VARCHAR2同义，但VARCHAR数据类型计划重新定义为与不同比较语义相比用于可变长度字符串的单独数据类型。

在CHAR和VARCHAR2之间进行选择时，请考虑：

  • 空间使用

    Oracle数据库空白填充存储在CHAR列中的值，但不存储存储在VARCHAR2列中的值。因此， VARCHAR2列比CHAR列更有效地使用空间。

  • 性能

    由于填充差异，在包含VARCHAR2列的大型表上进行全表扫描可能会读取比包含存储在CHAR列中的相同数据的表上的完整表扫描更少的数据块。如果您的应用程序经常对包含字符数据的大型表执行全表扫描，那么您可以通过将数据存储在VARCHAR2列而不是CHAR列中来提高性能。

  • 比较语义

    在比较语义中需要ANSI兼容性时，请使用CHAR数据类型。在字符串比较中尾随空白很重要时，请使用VARCHAR2数据类型。

对于客户端/服务器应用程序，如果客户端上的字符集与服务器端的字符集不同，则Oracle数据库会将数据库字符集（由NLS_LANGUAGE参数确定）中的CHAR ， VARCHAR2和LONG数据转换为为用户会话定义的字符集。

代表数字数据

存储数字数据的SQL数据类型是NUMBER ， BINARY_FLOAT和BINARY_DOUBLE 。

NUMBER数据类型以固定点或浮点格式存储实数。NUMBER提供最多38位十进制数字的精度。在NUMBER列中，您可以存储1 x 10 -130到9.99 x10 125和0的正负数。所有Oracle数据库平台都支持NUMBER值。

BINARY_FLOAT和BINARY_DOUBLE数据类型分别以单精度（32位）IEEE 754格式和双精度（64位）IEEE 754格式存储浮点数。存储为BINARY_FLOAT和BINARY_DOUBLE时，高精度值使用的空间比存储为NUMBER 。对于BINARY_FLOAT和BINARY_DOUBLE值，浮点数的算术运算通常比对NUMBER值更快。

在Oracle数据库支持的客户机接口中， BINARY_FLOAT和BINARY_DOUBLE值的算术运算由硬件供应商提供的本机指令集执行。术语本机浮点数据类型包括BINARY_FLOAT和BINARY_DOUBLE数据类型以及支持的客户机接口中这些类型的所有实现。

原生浮点数据类型基本上符合电气和电子工程师协会（IEEE）二进制浮点运算标准IEEE标准754-1985（IEEE754）。

注意：

Oracle建议使用BINARY_FLOAT和BINARY_DOUBLE而不是FLOAT ，即NUMBER的子类型。

浮点数格式

浮点数格式指定如何表示浮点数的分量，从而确定格式可以表示的值的范围和精度。范围是由最小值和最大值限定的区间， 精度是有效位数。范围和精度都是有限的。如果浮点数对于给定格式而言过于精确，则对该数字进行舍入。

数字的四舍五入取决于其格式的基数，可以是十进制或二进制。以十进制格式存储的数字四舍五入到最接近的小数位（例如，1000,10或0.01）。以二进制格式存储的数字四舍五入到最近的二进制位置（例如，1024,512或1/64）。

NUMBER值以十进制格式存储。对于需要十进制舍入的计算，请使用NUMBER数据类型。

本机浮点值以二进制格式存储。

表9-5中的每个值由除NaN之外的特定位模式表示。 NaN是任何未定义操作的结果，由许多位模式表示。这些位模式中的一些具有符号位设置而一些不具有，但符号位没有意义。

IEEE 754标准区分了安静的NaN （它们在大多数操作中传播时不会产生额外的异常）和信令NaN （这样做）。IEEE 754标准规定了何时启用异常以及何时禁用异常的操作。

在Oracle数据库中，无法启用例外。Oracle数据库充当IEEE 754标准，指定何时禁用异常。特别是，Oracle数据库不区分安静和信令NaN 。您可以使用Oracle调用接口（OCI）从Oracle数据库中检索NaN值，但检索到的NaN值是信号还是静音取决于客户端平台，并且超出了Oracle数据库的控制范围。

具有本机浮点数据类型的算术运算

IEEE 754不要求浮点运算具有精确的可重现性。因此，操作结果可以传递到使用大于操作操作数使用范围的范围的目标。

您可以在扩展的双精度目标上计算双精度乘法的结果，但结果必须舍入，就像目标是单精度或双精度一样。结果的范围（即，指数使用的位数）可以使用更宽（扩展双精度）目的地支持的范围;但是，这可能会导致双舍入错误，其中结果的最低有效位不正确。

这种情况只发生在实现IA-32和IA-64指令集架构的硬件上的双精度乘法和除法。因此，除了这种情况之外，这些数据类型的算法可以跨平台重现。当计算结果为NaN ，所有平台都会生成IS NAN为真的值。但是，所有平台都不必使用相同的位模式。

显示当前日期和时间

显示当前日期和时间的最简单方法是：

SELECT TO_CHAR（SYSDATE， format_model ）FROM DUAL

默认格式模型取决于初始化参数NLS_DATE_FORMAT 。

标准Oracle数据库默认日期格式为DD‐MON-RR 。 RR日期时间格式元素允许您通过仅指定年份的最后两位数来存储21世纪的20世纪日期。例如，在日期时间格式DD-MON-YY ， 13-NOV-54指的是1950年至2049年期间发出的查询中的1954年，但是在2050年至2149年之间发布的查询中指向2054年。

注意：

为了使程序正确并避免SQL注入和动态SQL出现问题，Oracle建议为每个日期时间值指定格式模型。

ARGn数据类型

在视图V$SQLFN_METADATA ，列DATATYPE是函数的数据类型（即，数据类型，该函数返回）。此数据类型可以是Oracle数据类型，数据类型系列或ARG n 。ARG n是函数的第n个参数的数据类型。例如：

  • MAX函数返回一个具有第一个参数的数据类型的值，因此MAX函数具有返回数据类型ARG1 。

  • DECODE函数返回一个具有第三个参数的数据类型的值，因此DECODE函数的数据类型为ARG3 。
