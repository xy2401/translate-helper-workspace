Oracle Database Vault如何处理职责分离

职责分离是限制每个用户的权限， 只给任务，他或她负责，并没有更多的 。

您应该为特定用户分配特定类别的权限，而不是向一个用户授予许多权限。简而言之，职责分离为您的组织所需的每项任务创建了责任。

在过去十年中，职责分离的重要性日益增加。对于许多组织而言，职责分离是一个不断发展的新概念。数据库整合，法规遵从和外包只是增加职责分离的几个驱动因素。Oracle Database Vault职责分离通过将安全相关的管理与日常的DBA操作分开来增强安全性。您可以定制Database Vault职责分离实施，以轻松适应当前和未来的业务需求。特别是小型组织在尝试用有限的资源增加其安全性时需要灵活性。

Oracle Database Vault环境中的任务分离

Oracle Database Vault定义了几个主要职责。

这些责任如下：

  • 帐户管理。 帐户管理需要创建，修改和删除用户帐户。DV_ACCTMGR角色提供这些权限。在Oracle Database Vault注册过程中，将创建主要的日常DV_ACCTMGR用户和备份DV_ACCTMGR用户。作为安全措施，您可以保留并维护备份帐户，以防主DV_ACCTMGR帐户所有者忘记密码或离开公司。

  • 安全管理。 安全管理涵盖基本的安全任务，例如创建领域和命令规则，为数据库用户访问设置安全策略，以及授权数据库用户执行允许的作业。安全管理员还运行安全审核报告。DV_OWNER和DV_ADMIN角色提供这些权限。在Oracle Database Vault注册过程中，将创建主要的日常DV_OWNER用户和备份DV_OWNER用户。

    重要：

    作为安全措施，您应该保留并维护备份用户帐户，以防主DV_OWNER帐户所有者忘记密码或离开公司。同样重要的是，您不要失去对已授予DV_OWNER角色的所有用户帐户的访问权限。有没有办法恢复DV_OWNER如果您无法访问（如使用密码丢失或员工离职），以具有任何帐户角色DV_OWNER作用。如果您无法访问DV_OWNER角色，则无法修改任何Database Vault控件或禁用Database Vault。要解决此问题，您可以将数据库恢复到数据库拥有Database Vault所有者帐户的最后一个已知点。

    （可选）您可以合并帐户管理和安全管理职责。

  • 数据库管理。 数据库管理是指管理数据库系统但不访问业务数据。它包括以下操作：

    • 备份操作需要预定义的时间来使用预定义工具执行备份。

    • 调整和监控操作需要持续的性能监控和分析。

    • 修补操作仅在修补发生期间需要临时访问

    Oracle强烈建议您在分离任务的情况下查看数据库管理帐户。不同的数据库管理员可能有不同的职责，需要不同的权限和角色同样，经验丰富的数据库管理员可能拥有更多角色和权限。不考虑向用户授予用户默认的DBA角色，而是考虑为组织中的特定职位和资历定制数据库管理角色。将命名帐户用于日常活动非常重要。SYS和使用SYSDBA管理权限的帐户等帐户应使用特权帐户管理（PAM）系统进行管理，并在使用时签出（并审核）。您还应该使用PAM系统管理备份Oracle Database Vault所有者和帐户管理帐户。在操作系统中，您应该只通过结帐系统使root和oracle帐户可用，因为这些帐户具有强大的权限。

您应该拥有数据库帐户管理，数据库安全管理和备份操作的其他命名帐户的单独帐户。审计员检查不同职责的单独数据库帐户，并能够跟踪每个帐户的操作。不太重要的是分配给特定任务的用户数。请记住，Oracle Database Vault审核事件受到保护，并且Database Vault报告显示所有尝试的违规。

根数据库和操作系统访问Oracle Database Vault

为了更好的安全性，您应该仔细监视对Oracle数据库的根和操作系统访问。库。

Oracle Database Vault可防止高权限数据库用户访问敏感数据。此外，如果您使用的是Oracle数据库本身，则可以使用透明数据加密来阻止最高权限的操作系统用户访问敏感数据。透明数据加密使您可以加密表空间和表列。这可以防止操作系统用户浏览操作系统数据库文件并查找敏感数据。作为最佳实践，请始终仔细检查并限制对操作系统的直接访问。

您应该有个性化帐户访问操作系统。在Linux或UNIX环境中，这些个性化帐户应在需要时使用sudo登录oracle软件所有者。使用sudo ，您可以控制每个个性化用户可以执行的特定命令。请确保为这些用户阻止使用可能会损害数据库的make ， relink ， gdb或其他命令。但是，如果管理用户必须安装修补程序或执行其他紧急操作，则可以在有限时间内启用make和relink命令，并在此期间审核其操作。

应限于受信任个人的帐户和角色

您应该仅将强大的帐户和角色限制为受信任的个人。

  • 管理具有操作系统根访问权限的用户
    拥有root用户访问权限的用户可以完全控制系统。
  • 管理Oracle软件所有者
    有权以Oracle软件所有者身份访问系统的用户可以控制Oracle软件。
  • 管理SYSDBA访问
    您应该避免将SYS帐户和SYSDBA权限用于正常的数据库维护任务。
  • 管理SYSOPER Access
    默认情况下，Oracle数据库限制对OSOPER组中的操作系统用户和用户SYS SYSOPER访问。

父主题： Oracle Database Vault安全指南

管理具有操作系统根访问权限的用户

拥有root用户访问权限的用户可以完全控制系统。

这些用户可以执行的活动包括：

  • 读取未加密的文件

  • 移动和删除任何文件

  • 启动或停止系统上的任何程序

  • 以任何用户身份登录，包括拥有Oracle数据库安装的用户

Oracle Database Vault不提供针对操作系统root访问的保护。管理特权帐户管理（PAM）系统中的root帐户和oracle帐户。仅在某些任务需要时检查这些帐户。在使用高权限操作系统帐户时，增强审计级别，包括击键捕获和视频捕获。

管理SYSDBA访问

您应该避免将SYS帐户和SYSDBA权限用于正常的数据库维护任务。

而是使用具有所需系统特权或特定管理特权（如SYSBACKUP ， SYSDG或SYSKM命名帐户。但是，在某些情况下，需要SYSDBA权限才能执行修补程序，升级数据库或解决问题（例如，连接到down数据库）。

由于具有SYSDBA权限的用户可以直接或间接访问敏感应用程序数据（例如，通过诊断，数据库升级和修补），因此必须严格限制使用SYSDBA权限和帐户。高权限帐户列表包括SYS和数据库中具有SYSDBA权限的用户帐户，以及操作系统中的root和oracle帐户。访问数据库和操作系统中的高权限帐户应该是例外，并要求用户通过一个过程来解锁对这些帐户和权限的访问权限。Oracle建议您使用特权帐户管理（PAM）系统管理这些帐户。仅在某些任务需要时检查这些帐户。在使用高权限操作系统帐户（ root和oracle ）和数据库帐户（ SYS帐户和SYSDBA管理权限）时，增强审计级别，包括击键捕获和视频捕获。当这些高权限帐户访问数据库时，请审核SYS帐户以监视其活动。

管理SYSOPER Access

默认情况下，Oracle数据库限制对OSOPER组中的操作系统用户和用户SYS SYSOPER访问。

这可以防止SYSOPER直接修改Oracle数据字典。SYSOPER权限在数据库中具有有限的权限，但具有此角色的个人可以启动和关闭Oracle数据库。仅授予受信任个人SYSOPER权限。

关于UTL_FILE和DBMS_FILE_TRANSFER包的安全注意事项

UTL_FILE包由SYS拥有并授予PUBLIC 。

但是，用户必须有权访问目录对象才能操作该操作系统目录中的文件。

创建外部作业权限安全注意事项

在Oracle数据库10g第 2版（10.2）中引入了CREATE EXTERNAL JOB特权。

想要执行在数据库外部的操作系统上运行的作业的数据库用户需要此权限。默认情况下， CREATE EXTERNAL JOB特权被授予已被授予CREATE JOB特权的所有用户。为了提高安全性，请从不需要它的用户撤消此权限，然后仅将其授予需要它的用户。

关于ALTER SYSTEM和ALTER SESSION权限安全注意事项

请注意，trace和debug命令有可能显示Oracle数据库内存信息。

Oracle Database Vault无法防御这些命令。为了帮助保护Oracle数据库内存信息，Oracle建议您严格控制对ALTER SYSTEM和ALTER SESSION权限的访问。当作为SYSDBA连接时，用户SYS可以授予这些权限，并且任何授予DBA角色的用户都可以授予这些权限。

Oracle还建议您将规则添加到ALTER SYSTEM语句的现有命令规则中。您可以使用Oracle Database Vault Administrator创建规则并将其添加到规则集。您应该仅向受信任的用户授予ALTER SESSION权限。（例如， ALTER SESSION语句可以启用跟踪。）
