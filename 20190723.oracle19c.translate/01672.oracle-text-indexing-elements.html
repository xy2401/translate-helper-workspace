2.2创建首选项

要创建数据存储区，词法分析器，过滤器，分类器，字列表或存储首选项，请使用CTX_DDL。 CREATE_PREFERENCE过程并指定本章中描述的类型之一。对于某些类型，您还可以使用CTX_DDL设置属性。 SET_ATTRIBUTE过程。

索引类型命名一类可用于创建索引首选项的索引对象。因此，类型是抽象ID，而首选项是对应于类型的实体。许多系统定义的首选项与类型具有相同的名称（例如， BASIC_LEXER ），但不保证完全对应。假设索引类型或系统首选项的存在或性质时要小心。

您可以使用CREATE INDEX和ALTER INDEX语句指定索引首选项。索引首选项确定索引的创建方式。例如，词法分析器首选项指示要编入索引的文本的语言。您可以创建和指定自己的用户定义首选项，也可以使用系统定义的首选项。

要创建停止列表，请使用CTX_DDL。 CREATE_STOPLIST过程。使用CTX_DDL.ADD_STOPWORD将停用词添加到停止列表。

要创建节组，请使用CTX_DDL。 CREATE_SECTION_GROUP并指定节组类型。使用CTX_DDL.ADD_ZONE_SECTION或CTX_DDL.ADD_FIELD_SECTION过程将节添加到节组。

2.3.1 DIRECT_DATASTORE

对于直接存储在文本列中的文本，使用DIRECT_DATASTORE类型，每行一个文档。DIRECT_DATASTORE类型没有属性。

如果您的列是BFILE ，则索引所有者必须对BFILEs使用的所有目录具有读取权限。

2.3.2.2索引和DML

要进行索引，必须创建一个虚拟列以在CREATE INDEX语句中指定。除非在columns属性中指定了其名称，否则此列的内容不会成为虚拟文档的一部分。

仅在更新虚拟列时才会同步索引。您可以创建触发器以在需要时传播更改。

2.3.2.3 MULTI_COLUMN_DATASTORE限制

您无法使用XMLType列创建多列数据存储。MULTI_COLUMN_DATA_STORE不支持XMLType 。您可以使用XMLType列创建CONTEXT索引，如Oracle Text SQL语句和操作符中所述 。

2.3.2.5 MULTI_COLUMN_DATASTORE过滤器示例

以下示例创建多列数据存储区首选项，并表示将使用AUTO_FILTER过滤bar列。

ctx_ddl.create_preference（ 'MY_MULTI'， 'MULTI_COLUMN_DATASTORE'）; ctx_ddl.set_attribute（'MY_MULTI'，'COLUMNS'，'foo，bar'）; ctx_ddl.set_attribute（ 'MY_MULTI'， '过滤'， 'N，Y'）;
多列数据存储区获取foo和bar列的内容，过滤bar ，然后将复合文档组合为：
<FOO> foo contents </ FOO> <BAR> bar过滤内容（可能最初为HTML）</ BAR>
不需要指定N个标志，并且不需要为每个列都标记。只能指定Y标志，用逗号表示它们适用于哪个列。例如：
ctx_ddl.create_preference（ 'MY_MULTI'， 'MULTI_COLUMN_DATASTORE'）; ctx_ddl.set_attribute（'MY_MULTI'，'COLUMNS'，'foo，bar，zoo，jar'）; ctx_ddl.set_attribute（ 'MY_MULTI'， '过滤'， ',, Y'）;
此示例仅过滤列zoo 。
                        

                     

                  
2.3.2.6标记行为

                     

                        
在索引编制期间，系统会为每行创建一个虚拟文档。虚拟文档由列表顺序连接的列的内容组成，并自动添加列名标签。

                        
例如：
create table mc（id number primary key，name varchar2（10），address varchar2（80））;插入mc值（1，'John Smith'，'123 Main Street'）; exec ctx_ddl.create_preference（'mymds'，'MULTI_COLUMN_DATASTORE'）; exec ctx_ddl.set_attibute（'mymds'，'columns'，'name，address'）;
这将生成以下用于索引的虚拟文本：
<NAME> John Smith </ NAME> <ADDRESS> 123 Main Street </ ADDRESS>

                  
2.3.2.7将列索引为节

                     

                        
要将标记索引为节，您可以选择使用BASIC_SECTION_GROUP创建字段节。

                        

                           
注意：

                           
使用MULTI_COLUMN_DATASTORE时，不会创建任何节组。要为这些标记创建部分，您必须创建部分组。
                           

                        

                        
使用表达式或函数时，除非使用列别名，否则标记由表达式的前30个字符组成。

                        
例如，如果表达式如下：
exec ctx_ddl.set_attibute（'mymds'，'columns'，'4 + 17'）;
然后它产生以下虚拟文本：
<4 + 17> 21 </ 4 + 17>
如果你的表达如下：
exec ctx_ddl.set_attibute（'mymds'，'columns'，'4 + 17 col1'）;
然后它产生以下虚拟文本：
<col1> 21 <col1>
除非列名称或列别名为小写且由双引号括起，否则标记为大写。例如：
exec ctx_ddl.set_attibute（'mymds'，'COLUMNS'，'foo'）;
这会生成以下虚拟文本：
<FOO> foo的内容</ FOO>
对于小写标签，请使用以下内容：
exec ctx_ddl.set_attibute（'mymds'，'COLUMNS'，'foo“foo”'）;
这个表达式产生：
<foo> foo的内容</ foo>

但是，对于具有数据库和操作系统的多字节字符集的应用程序而言，这可能是不够的，因为WE8ISO8859p1和WE8EBCDIC1047都不支持多字节字符。filename_charset属性可以解决此问题。如果指定，则数据存储区将从数据库字符集转换为指定的字符集，而不是ISO8859或EBCDIC。

2.3.4.2 FILE_DATASTORE和安全性

                     

                        
文件和URL数据存储可以访问实际数据库磁盘上的文件。当安全性成为问题时，这可能是不合需要的，因为任何用户都可以浏览Oracle用户可访问的文件系统。尝试使用FILE或URL数据存储创建索引的任何用户都必须直接授予用户TEXT DATASTORE ACCESS系统特权，否则索引创建将失败。通过将用户授予用户角色来间接授予用户TEXT DATASTORE ACCESS权限不起作用，并且索引创建仍将失败。因此，默认情况下，用户无法创建使用FILE或URL数据存储的索引。向PUBLIC授予TEXT DATASTORE ACCESS允许任何用户有权在FILE数据存储区中索引文件系统中的任意文件，在URL数据存储区中索引任意URL，不建议使用。
                        

                        
例如，以下语句将TEXT DATASTORE ACCESS授予用户SCOTT：
向SCOTT授予TEXT DATASTORE ACCESS;
当没有TEXT DATASTORE ACCESS权限的用户尝试在FILE或URL数据存储上创建索引时，CREATE INDEX操作将失败。例如：
CREATE INDEX myindex ON mydocument（TEXT）INDEXTYPE是ctxsys.context PARAMETERS（'DATASTORE ctxsys.file_datastore'）
在这种情况下，如果用户没有直接授予它的TEXT DATASTORE ACCESS权限，则索引创建将失败并返回错误。对于具有TEXT DATASTORE ACCESS权限的用户，索引创建将正常进行。
                        

                        
每次访问数据存储区时都会检查用户的权限。这包括索引创建，索引同步和对文档服务的调用，例如CTX_DOC.HIGHLIGHT 。

2.4.3.3关于邮件过滤器配置文件

文件的位置必须在ORACLE_HOME / ctx / config中。要使用的文件的名称存储在新的系统参数中 MAIL_FILTER_CONFIG_FILE 。在安装时，它设置为drmailfl.txt，它具有有用的默认内容。
                        

                        
Oracle建议您创建自己的邮件过滤器配置文件，以避免因安装新版本或补丁集而覆盖。邮件过滤器配置文件应位于数据库字符集中。

                        

                           
邮件文件配置文件结构

                        

                        
                        

                           
该文件有两个部分， BEHAVIOR和CHARSETS 。指示行为部分的开头，如下所示：
[行为]
以下每行以mime类型开头，然后是空格，然后是行为规范。MIME类型可以是完整的TYPE / SUBTYPE ，也可以只是TYPE ，它将应用于该类型的所有子类型。TYPE / SUBTYPE规范覆盖TYPE规范，该规范覆盖默认行为。行为可以是INCLUDE ， AUTO_FILTER或IGNORE （有关定义，请参阅“ MAIL_FILTER Behavior ” ）。例如：
application / zip IGNORE application / msword AUTO_FILTER model IGNORE
您无法指定“multipart”或“message”类型的行为。如果这样做，这些行将被忽略。类型的重复规范取代了早期的规范。

                           
通过使用＃符号开始行，可以在邮件配置文件中包含注释。

                           
字符集映射部分以。开头
[字符集]
行包含IANA名称，然后是空格，然后是Oracle全球化支持字符集名称，如：
US-ASCII US7ASCI ISO-8859-1 WE8ISO8859P1
此文件是邮件过滤器获取映射的唯一方法。没有默认值。

                           
更改配置文件时，更改仅影响在该点之后编入索引的文档。更改文件后，您必须刷新共享池。

                        

                        
                     

                  
2.4.3.4 Mail_Filter示例

                     

                        
假设有一封电子邮件，其中包含以下表格，其中包含不同主题行的其他电子邮件附在此电子邮件中：
To：somebody @ someplace主题：mainheader内容类型：multipart / mixed。 。 。Content-Type：text / plain X-Ref：some_value主题：subheader 1。 。 。Content-Type：text / plain X-Control：blah blah blah主题：subheader 2。 。 。

将INDEX_FIELDS设置为“主题”，最初将PART_FIELD_STYLE为IGNORE 。
CTX_DDL.CREATE_PREFERENCE（'my_mail_filt'，'mail_filter'）; CTX_DDL_SET_ATTRIBUTE（my_mail_filt'，'INDEX_FILES'，'subject'）; CTX_DDL.SET ATTRIBUTE（'my_mail_filt'，'PART_FIELD_STYLE'，'忽略'）;
现在，当创建索引时，文件将被索引如下：
<SUBJECT> mainheader </ SUBJECT>
如果将PART_FIELD_STYLE设置为TAG ，则变为：
<SUBJECT> mainheader </ SUBJECT> <SUBJECT> subheader1 </ SUBJECT> <SUBJECT> subheader2 </ SUBJECT>
如果PART_FIELD_STYLE设置为FIELD ，则结果如下：
<SUBJECT> mainheader <SUBJECT> SUBJECT：subheader1 SUBJECT：subheader2
最后，如果将PART_FIELD_STYLE设置为TEXT ，则结果为：
<SUBJECT> mainheader </ SUBJECT> subheader1 subheader2

2.4.5.2 PROCEDURE_FILTER参数顺序

                     

                        
ROWID_PARAMETER ， FORMAT_PARAMETER和CHARSET_PARAMETER都是独立的。顺序是rowid，格式，然后是charset。但是，过滤程序仅传递所需的最小参数。
                        

                        
例如，假设INPUT_TYPE是BLOB而OUTPUT_TYPE是CLOB 。如果您的过滤程序需要所有参数，那么过程签名必须是：
（id IN ROWID，格式为IN VARCHAR2，charset IN VARCHAR2，输入IN BLOB，输出IN OUT NOCOPY CLOB）
如果您的过程只需要ROWID ，那么过程签名必须是：
（id IN ROWID，输入IN BLOB，输出IN OUT NOCOPY CLOB）

2.5.2 BASIC_LEXER

注意：

    • 
                              
  •

    • 
                              
  •

    • 
                              
  • 
                                 
    在查询词的开头或结尾具有至少K-1个连续非通配符的通配符查询词可以使用K-gram索引。例如，如果K值为3，则“ab％”和“％ab”之类的查询可以使用k-gram索引。
    
                              
    •

注意：此组类型不支持不平衡括号，注释标记和属性等输入。对此类输入使用HTML_SECTION_GROUP 。
                                    

                                 		
                              

                              

                                 
                                    
HTML_SECTION_GROUP

                                 		
                                 
                                    
使用此组类型索引HTML文档和定义HTML文档中的部分。

                                 		
                              

                              

                                 
                                    
JSON_SECTION_GROUP

                                 		
                                 
                                    
使用此组可以创建启用JSON的上下文索引。JSON ENABLE属性不能与XML ENABLE一起使用。节组只能标记为JSON ENABLE 。如果它已标记为XML ENABLE ，则路径部分组不能用于JSON ENABLE ，反之亦然。
                                    

                                 		
                              

                              

                                 
                                    
XML_SECTION_GROUP

                                 		
                                 
                                    
使用此组类型索引XML文档和定义XML文档中的部分。必须为该组手动定义要编制索引的所有部分。

                                 		
                              

                              

                                 
                                    
AUTO_SECTION_GROUP

                                 		
                                 
                                    
使用此组类型可自动为XML文档中的每个开始标记/结束标记对创建区域部分。从XML标记派生的节名称区分大小写，如XML格式。

                                    
为具有属性的XML标记自动创建属性部分。属性部分在表单标记@ attribute中命名。

                                    
可以将特殊部分添加到AUTO_SECTION_GROUP以进行WITHIN SENTENCE和WITHIN PARAGRAPH搜索。将句子或段落部分添加到AUTO_SECTION_GROUP ，具有相应标记名称“ sentence ”或“ paragraph ”（不区分大小写）的部分将被视为停止部分。
                                    

                                    
停止节，空标签，处理指令和注释未编入索引。

                                    
以下限制适用于自动节组：

                                    
    
                                       
  • 
                                          
    您无法将区域，字段，sdata或特殊部分添加到自动部分组。
    
                                       
    • 
                                       
  • 
                                          
    您可以定义仅适用于一种特定类型的停止部分;也就是说，如果你有两个不同的XML DTD，它们都使用一个名为FOO的标签，你可以定义(TYPE1)FOO停止，但(TYPE2)FOO不要停止。
                                          
    
                                       
    • 
                                       
  • 
                                          
    索引标记的长度（包括前缀和名称空间）不能超过64个字节。超过此标签的标签未编入索引。
    
                                       
    • 
                                    

                                 		
                              

                              

                                 
                                    
PATH_SECTION_GROUP

                                 		
                                 
                                    
使用此组类型来索引XML文档。表现得像AUTO_SECTION_GROUP 。

                                    
不同之处在于，使用此节组，您可以使用INPATH和HASPATH运算符进行路径搜索。查询对标记和属性名称也区分大小写。不允许停止部分。
                                    

                                 		
                              

                              

                                 
                                    
NEWS_SECTION_GROUP

                                 		
                                 
                                    
根据RFC 1036，使用此组定义新闻组格式文档中的部分。

                                 		
                              

                           
                        

                     

                     
                     

                        

                           
注意：
从Oracle Database 18c开始，Oracle Text中不推荐使用NEWS_SECTION_GROUP 。请改用外部处理。
如果要索引USENET帖子，则预处理帖子以在Oracle Text中使用BASIC_SECTION_GROUP或HTML_SECTION_GROUP 。USENET很少在商业上使用。

2.14 Oracle文本索引的令牌限制

               

                  
从Oracle Database Release 18c开始，对于单字节字符集，索引标记最大大小增加到255个字符。

                  
在Oracle Database Release 18c之前，除SDATA部分之外的所有Oracle Text索引类型都在VARCHAR2 （64 BYTE）类型的表列中存储了令牌。从Oracle Database Release 18c开始，除CTXCAT和CTXRULE索引之外的所有Oracle Text索引类型都在VARCHAR2 （255 BYTE）表列类型中存储令牌。对于单字节字符集，此更改将索引标记的最大大小增加到255个字符。多字节或可变长度字符集的大小增加较少。超过255个字节的标记将被截断。截断的标记不会阻止对整个标记字符串的搜索。但是，系统无法区分具有相同前255个字节的两个令牌。
                  

                  

                     
注意：

                     
在Oracle Database Release 18c之前，大于64字节的令牌被截断为64字节。升级到Oracle Database Release 18c后，令牌表从64字节增加到255个字节。搜索令牌中超过64个字节的搜索（即搜索字符串中的任何单个单词）都找不到任何被截断为64个字节的令牌。要避免此问题，请重建索引。如果您从不使用长度超过64字节的搜索令牌，则无需重建索引。

                  

                  
SDATA部分在类型为VARCHAR2 （249 BYTE）的表列中存储标记。CTXCAT和CTXRULE索引在类型为VARCHAR2 （64 BYTE）的表列中存储标记。
