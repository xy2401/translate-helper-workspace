2.2.2关于选择字符集

为数据库选择正确的字符集非常重要。Oracle建议将AL32UTF8用作数据库字符集。

AL32UTF8是Oracle标准的UTF-8编码的名称。Unicode标准是通用字符集，支持世界上大多数当前使用的语言。Unicode标准的使用对于任何多语言技术都是必不可少的，包括数据库处理。

创建数据库并累积生产数据后，更改数据库字符集是一个耗时且复杂的项目。因此，在安装时选择正确的字符集非常重要。即使数据库当前没有存储多语言数据，但预计会在几年内存储多语言数据，但AL32UTF8对数据库字符集的选择通常是唯一的好决定。Unicode的普遍性和灵活性通常超过与其相关的一些额外成本，例如与单字节字符集相比文本处理稍慢，与非Unicode字符集相比，非ASCII文本的存储空间要求更高。

如果您不想使用AL32UTF8，并且您不受供应商要求的限制，那么Oracle建议您使用按照数据库建议列出的字符集之一。根据现代客户端操作系统的要求选择推荐的字符集。Oracle Universal Installer（OUI）仅提供推荐列表，并且必须单独使用Database Configuration Assistant（DBCA）来选择不推荐的字符集。此外，DBCA中的默认数据库创建配置仅允许选择推荐的字符集。必须使用DBCA的高级配置模式或CREATE DATABASE语句来选择不推荐的字符集。

如果在OUI或DBCA安装模式下未显示任何字符集选项，则AL32UTF8将用作数据库字符集，除非已选择具有其他字符集的自定义数据库模板。

注意：

  • Oracle建议使用AL32UTF8作为数据库字符集。AL32UTF8是Unicode编码UTF-8的正确实现。从Oracle Database 12c第2版开始，AL32UTF8在使用Oracle Universal Installer（OUI）和Oracle Database Configuration Assistant（DBCA）创建数据库时用作默认数据库字符集。

  • 您只能在基于ASCII的平台上为数据库选择基于ASCII的字符集。

警告：

除非与Oracle8 i Release 1（8.1.7）及更早版本中的Oracle数据库客户端和服务器兼容，或者除非应用程序供应商明确要求，否则不要将UTF8用作数据库字符集。尽管名称非常相似，但UTF8并不是Unicode编码UTF-8的正确实现。如果在预期使用UTF-8处理的情况下使用UTF8字符集，则可能会发生数据丢失和安全问题。对于Web相关数据（例如XML和URL地址）尤其如此。

AL32UTF8和UTF8字符集彼此不兼容，因为它们具有不同的最大字符宽度。AL32UTF8的最大字符宽度为4个字节，而UTF8的最大字符宽度为3个字节。

2.2.3关于以只读模式配置Oracle Home

从Oracle Database 18c开始，您可以通过将只读Oracle主目录部署为软件映像来简化跨多个数据库服务器的软件修补和批量部署。

只读Oracle主目录（ ORACLE_HOME ）可防止在Oracle主目录中创建和修改文件。要以只读模式配置Oracle主目录，首先使用纯软件部署安装Oracle数据库软件，然后在创建侦听器和数据库之前将其配置为只读Oracle主目录。

传统的读写Oracle主目录包含特定于实例的文件，因此如果您对其进行修补，则必须单独修补每个文件。但是，当Oracle主目录为只读时，特定于实例的文件将单独存储在Oracle基本目录（ ORACLE_BASE ）中，而不是Oracle主目录中。由于此配置，您可以使用只读Oracle主目录作为可以跨多个数据库服务器共享的软件映像，因为它只存储静态文件。此选项简化了修补和批量部署，因为当您要将修补程序分发到多个数据库服务器时，只需更新一个Oracle主映像。

除了传统的ORACLE_BASE和ORACLE_HOME目录之外，只读Oracle主目录中还存在以下附加目录：

  • ORACLE_BASE_HOME ：这是ORACLE_BASE目录中的子目录。此目录包含特定于用户的文件，特定于实例的文件和日志文件。

  • ORACLE_BASE_CONFIG ：该目录与ORACLE_BASE目录相同。此目录包含特定于实例的动态文件，例如配置文件。

2.4使用CREATE DATABASE语句创建数据库

与使用Oracle数据库配置助手（DBCA）相比，使用CREATE DATABASE SQL语句是一种更加手动的方法来创建数据库。使用此语句而不是使用DBCA的一个优点是您可以在脚本中创建数据库。

  • 关于使用CREATE DATABASE语句创建数据库
    使用CREATE DATABASE语句时，必须先完成其他操作，然后才能拥有可操作的数据库。这些操作包括构建数据字典表的视图和安装标准PL / SQL包。您可以通过运行提供的脚本来执行这些操作。
  • 第1步：指定实例标识符（SID）
    ORACLE_SID环境变量用于将此实例与您稍后可能创建并在同一主机上并发运行的其他Oracle数据库实例区分开来。
  • 第2步：确保设置了所需的环境变量
    根据您的平台，在启动SQL * Plus之前（根据后续步骤的要求），您可能必须设置环境变量，或者至少验证它们是否已正确设置。
  • 第3步：选择数据库管理员身份验证方法
    必须进行身份验证并授予适当的系统权限才能创建数据库。
  • 第4步：创建初始化参数文件
    当Oracle实例启动时，它会读取初始化参数文件。此文件可以是文本文件，可以使用文本编辑器或二进制文件创建和修改，文件由数据库创建和动态修改。首选的二进制文件称为服务器参数文件 。在此步骤中，您将创建文本初始化参数文件。在稍后的步骤中，您将从文本文件创建服务器参数文件。
  • 步骤5 :(仅限Windows）创建实例
    在Windows平台上，在连接到实例之前，必须手动创建它（如果它尚不存在）。ORADIM命令通过创建新的Windows服务来创建Oracle数据库实例。
  • 第6步：连接到实例
    启动SQL * Plus并使用SYSDBA管理权限连接到Oracle数据库实例。
  • 第7步：创建服务器参数文件
    服务器参数文件使您可以使用ALTER SYSTEM命令更改初始化参数，并在数据库关闭和启动期间保留更改。您可以从已编辑的文本初始化文件创建服务器参数文件。
  • 第8步：启动实例
    在不安装数据库的情况下启动实例。
  • 第9步：发出CREATE DATABASE语句
    要创建新数据库，请使用CREATE DATABASE语句。
  • 步骤10：创建其他表空间
    要使数据库正常运行，必须为应用程序数据创建其他表空间。
  • 步骤11：运行脚本以构建数据字典视图
    运行构建数据字典视图，同义词和PL / SQL包所需的脚本，并支持SQL * Plus的正常运行。
  • 步骤12 :(可选）运行脚本以安装其他选项
    您可能想要运行其他脚本。您运行的脚本由您选择使用或安装的功能和选项决定。
  • 步骤13：备份数据库
    对数据库进行完整备份，以确保您有一整套文件，以便在发生介质故障时从中恢复。
  • 步骤14 :(可选）启用自动实例启动
    您可能希望将Oracle实例配置为在其主机重新启动时自动启动。

2.7.1什么是服务器参数文件？

2.7.6.2.1 ALTER SYSTEM SET语句中的SCOPE子句

ALTER SYSTEM SET语句中的可选SCOPE子句指定初始化参数更改的范围。

范围条款 描述

范围= SPFILE

此更改仅应用于服务器参数文件。效果如下：

  • 没有对当前实例进行任何更改。

  • 对于动态和静态参数，更改在下次启动时有效并且是持久的。

这是静态参数允许的唯一SCOPE规范。

范围=记忆

此更改仅应用于内存。效果如下：

  • 对当前实例进行更改并立即生效。

  • 对于动态参数，效果是立即的，但它不是持久的，因为服务器参数文件未更新。

对于静态参数，不允许使用此规范。

范围=两个

更改将应用于服务器参数文件和内存中。效果如下：

  • 对当前实例进行更改并立即生效。

  • 对于动态参数，效果是持久的，因为服务器参数文件已更新。

对于静态参数，不允许使用此规范。

如果实例未使用服务器参数文件启动，则指定SCOPE=SPFILE或SCOPE=BOTH是错误的。如果使用服务器参数文件启动实例，则默认值为SCOPE=BOTH如果使用文本初始化参数文件启动实例，则默认值为MEMORY 。

对于动态参数，您还可以指定DEFERRED关键字。指定后，更改仅对将来的会话有效。

将SCOPE指定为SPFILE或BOTH ，可选的COMMENT子句允许您将文本字符串与参数更新相关联。注释将写入服务器参数文件。

2.7.7清除初始化参数值

您可以使用ALTER SYSTEM RESET语句清除初始化参数值。执行此操作时，初始化参数值将更改为其默认值或其启动值。

ALTER SYSTEM RESET语句包含SCOPE子句。在非CDB或多租户容器数据库（CDB）根目录中执行时， ALTER SYSTEM RESET语句和SCOPE子句的行为与在可插拔数据库（PDB），应用程序根目录或应用程序PDB中执行语句的行为不同。

参数的启动值是实例启动或PDB打开完成后内存中参数的值。启动后立即在V$SYSTEM_PARAMETER视图的VALUE和DISPLAY_VALUE列中可以看到此值。启动值可以与spfile中的值或默认值不同（如果未在spfile中设置该参数），因为可以在启动时在内部调整参数的值。

ALTER SYSTEM RESET语句的SCOPE值在非CDB和CDB$ROOT中的行为如下：

  • SCOPE=SPFILE ：如果实例正在使用spfile，则从spfile中删除该参数;默认值在下次实例启动时生效。

  • SCOPE=MEMORY ：启动值立即生效。但是，更改不会存储在实例的spfile中，并且会在实例重新启动时丢失。

  • SCOPE=BOTH ：如果实例正在使用spfile，则从spfile中删除该参数;默认值立即生效，并且更改在实例重新启动时可用。

注意：

SCOPE=BOTH改变了SCOPE=MEMORY行为方式。发出SCOPE=BOTH后， SCOPE=MEMORY始终将参数重置为默认值。

ALTER SYSTEM RESET语句的SCOPE值在PDB，应用程序根或应用程序PDB中的行为如下：

  • SCOPE=SPFILE ：从容器的spfile中删除参数;容器将在下一个PDB打开时从其根继承参数值。

  • SCOPE=MEMORY ：有两种情况：

    • 打开容器时，该参数存在于容器的spfile中。参数值更新为参数的启动值。此更改不存储在容器的spfile中，并且在下一个容器打开时将丢失。

    • 打开容器时，该参数不存在于容器的spfile中。容器开始从其根继承参数值。

  • SCOPE=BOTH ：从容器的spfile中删除参数;容器将从其根继承参数值。

注意：

  • SCOPE=BOTH改变了SCOPE=MEMORY行为方式。发出SCOPE=BOTH后，当发出SCOPE=MEMORY时，容器始终从其根继承参数值。

  • 在容器从其根继承参数值的情况下，PDB从CDB$ROOT继承该值。在应用程序容器中，应用程序PDB从其应用程序根继承参数值，应用程序根从CDB$ROOT继承参数值。

2.8.1.1关于数据库服务

数据库服务将单个数据库的工作负载划分为相互不相交的分组。

每个数据库服务代表具有公共属性，服务级别阈值和优先级的工作负载。分组基于可能包括要使用的应用程序功能，应用程序功能的执行优先级，要管理的作业类或应用程序功能或作业类中使用的数据范围的工作属性。例如，Oracle电子商务套件为每个职责定义数据库服务，例如总帐，应收帐款，订单输入等。配置数据库服务时，为每个服务提供唯一的名称，关联的性能目标和相关的重要性。数据库服务与Oracle数据库紧密集成，并在数据字典中维护。

连接请求可以包括数据库服务名称。因此，中间层应用程序和客户端/服务器应用程序通过将数据库服务指定为TNS连接数据中的连接的一部分来使用服务。如果未包含数据库服务名称且Net Services文件listener.ora指定默认数据库服务，则连接将使用默认数据库服务。

通过数据库服务，您可以为单个数据库配置工作负载，对其进行管理，启用和禁用它，并将工作负载作为单个实体进行度量。您可以使用标准工具执行此操作，例如Database Configuration Assistant（DBCA），Oracle Net Configuration Assistant和Oracle Enterprise Manager Cloud Control（云控制）。Cloud Control支持整体查看和运营服务，并在需要时向下钻取到实例级别。

在Oracle Real Application Clusters（Oracle RAC）环境中，数据库服务可以跨越一个或多个实例，并根据事务性能促进工作负载平衡。此功能提供端到端的无人值守恢复，按工作负载进行的滚动更改以及完整的位置透明性。Oracle RAC还使您能够使用Cloud Control，DBCA和服务器控制实用程序（ SRVCTL ）管理多个数据库服务功能。

数据库服务将应用程序，应用程序功能和数据范围描述为功能服务或数据相关服务。功能服务是最常见的工作负载映射。使用特定功能的会话组合在一起。相反，数据相关路由基于数据密钥将会话路由到数据库服务。工作请求到数据库服务的映射发生在应用程序服务器和TP监视器的对象关系映射层中。例如，在Oracle RAC中，这些范围可以是完全动态的并且基于需求，因为数据库是共享的。

除了应用程序使用的数据库服务之外，Oracle数据库还支持两种内部数据库服务： SYS$BACKGROUND仅由后台进程使用， SYS$USERS是与服务无关的用户会话的缺省数据库服务。

使用数据库服务不需要更改应用程序代码。客户端工作可以连接到指定的数据库服务。服务器端工作（如Oracle Scheduler，并行执行和Oracle数据库高级队列）将数据库服务名称设置为工作负载定义的一部分。在数据库服务下执行的工作请求会继承服务的性能阈值，并作为服务的一部分进行度量。

2.8.1.2数据库服务和性能

数据库服务为性能调优提供了额外的维度。

通过“服务和SQL”进行调优可以取代所有会话都是匿名和共享的大多数系统中的“会话和SQL”调优。使用数据库服务，工作负载是可见且可测量的。资源消耗和等待可归因于应用程序。此外，当负载增加或减少时，可以增加分配给数据库服务的资源。这种动态资源分配可以实现经济高效的解决方案，以满足需求。例如，自动测量数据库服务，并将性能与服务级别阈值进行比较。性能违规将报告给Cloud Control，从而可以执行自动或计划的解决方案。

2.8.1.3使用数据库服务的Oracle数据库功能

多个Oracle数据库功能支持数据库服务。

自动工作负载存储库（AWR）管理服务的性能。AWR记录数据库服务性能，包括执行时间，等待类和服务消耗的资源。当超过数据库服务响应时间阈值时，AWR警报会发出警告。动态视图使用一小时的历史记录报告当前服务性能指标。每个数据库服务都具有响应时间和CPU消耗的服务质量阈值。

此外，数据库资源管理器可以将数据库服务映射到使用者组。因此，您可以自动管理一个数据库服务相对于其他服务的优先级。您可以使用使用者组来根据比率或资源消耗来定义相对优先级。

您还可以为数据库服务指定版本属性。版本可以在数据库中拥有两个或更多版本的相同对象。为数据库服务指定版本属性时，指定数据库服务的所有后续连接都将此版本用作初始会话版本。

将版本指定为数据库服务属性可以更轻松地管理资源使用情况。例如，与版本关联的数据库服务可以放置在Oracle RAC环境中的单独实例上，数据库资源管理器可以通过将资源计划与相应的数据库服务相关联来管理不同版本使用的资源。

对于Oracle Scheduler，您可以选择在创建作业类时分配数据库服务。在执行期间，作业被分配给作业类，作业类可以在数据库服务中运行。将数据库服务与作业类一起使用可确保确定作业调度程序执行的工作以进行工作负载管理和性能调整。

对于并行查询和并行DML，查询协调器就像任何其他客户端一样连接到数据库服务。并行查询进程在执行期间继承数据库服务。在查询执行结束时，并行执行进程将恢复为默认数据库服务。

2.9.4交易保护和应用程序连续性

Transaction Guard使用逻辑事务ID来防止客户端应用程序在发生可恢复错误后提交重复事务的可能性。应用程序连续性使得在数据库会话不可用的可恢复错误之后，以无中断和快速方式重放对数据库的请求。

Transaction Guard是一种可靠的协议和API，应用程序开发人员可以使用它为数据库会话上的最后一个打开事务提供已知结果，该事务变得不可用。中断后，从数据库发送到客户端的提交消息不是持久的。如果应用程序（客户端）和Oracle数据库（服务器）之间的连接中断，则客户端会收到一条错误消息，指示通信失败。此错误消息不会通知客户端提交操作或过程调用的成功或失败。

Transaction Guard使用称为逻辑事务标识符（LTXID）的概念，这是一个全局唯一标识符，从应用程序的角度标识事务。当发生可恢复的中断时，应用程序使用LTXID来确定事务的结果。可以将此结果返回给客户端，而不是模糊的通信错误。用户可以决定是否重新提交事务。如果状态正确，也可以对应用程序进行编码以重新提交事务。

应用程序连续性通过在可计划和计划中断后的可恢复中断期间恢复正在进行的数据库会话来屏蔽最终用户和应用程序的中断。成功重播后，应用程序可以继续使用原始数据库会话停止的新会话。应用程序连续性执行此恢复，以便中断在应用程序中显示为延迟执行。

应用程序连续性在服务级别启用，并针对可恢复的中断进行调用。这些中断通常与底层软件，前台，硬件，通信，网络或存储层有关。Application Continuity支持查询， ALTER SESSION语句，Java和OCI API，PL / SQL，DDL以及失败前的最后一个未提交的事务。Application Continuity使用Transaction Guard确定是否提交了最后一次正在进行的事务，以及最后一次用户调用是否已完成。

2.10.1.2使用CloneDB克隆数据库

您可以使用CloneDB克隆数据库。

在克隆数据库之前，必须满足以下先决条件：

  • 每个CloneDB数据库必须使用Direct NFS Client，并且生产数据库的备份必须位于NFS卷上。

    Direct NFS Client使Oracle数据库可以直接访问网络连接存储（NAS）设备，而不是使用操作系统内核NFS客户端。此CloneDB数据库功能在支持Direct NFS Client的平台上可用。

    有关Direct NFS Client的信息，请参阅适用于您的操作系统的Oracle Grid Infrastructure安装指南 。

  • 至少需要2 MB的额外系统全局区域（SGA）内存来跟踪CloneDB数据库中的已修改块。

    请参阅管理内存 。

  • 需要存储数据库备份和每个CloneDB数据库中已更改的块。

    数据库备份所需的存储取决于用于执行备份的方法。单个完整RMAN备份需要最多存储空间。使用存储设备功能执行的存储快照符合存储设备的要求。单个备份可以支持多个CloneDB数据库。

    每个CloneDB数据库所需的存储量取决于该数据库中的写入活动。每个被修改的块都需要一个可用的存储块。因此，总存储要求取决于CloneDB数据库中随时间修改的块数。

本节介绍创建一个CloneDB数据库并使用这些示例数据库和目录所需的步骤：

  • 生产数据库PROD1的Oracle主PROD1是/ u01 / PROD1 / oracle。

  • 数据库备份的文件位于/ u02 / oracle / backup / prod1中。

  • CloneDB数据库CLONE1的Oracle主CLONE1是/ u03 / clone1 / oracle。

使用CloneDB克隆数据库：

  1. 创建生产数据库的备份。您有以下备份选项：

    • 在线备份

      如果执行联机备份，请确保生产数据库处于ARCHIVELOG模式，并且所有必需的归档重做日志文件都已保存并可供CloneDB数据库环境访问。

    • 完整的脱机备份

      如果执行完全脱机备份，请确保CloneDB数据库环境可以访问备份文件。

    • 复制数据库文件的备份

      如果在RMAN中指定BACKUP AS COPY ，则RMAN将每个文件复制为映像副本，这是在磁盘上创建的数据库文件的逐位副本。映像副本与使用操作系统命令（例如Linux上的cp或Windows上的COPY创建的副本相同，但是记录在RMAN存储库中，因此可供RMAN使用。在数据库打开时，您可以使用RMAN进行映像复制。确保CloneDB数据库环境可以访问复制的数据库文件。

    有关备份数据库的信息，请参见“ Oracle数据库备份和恢复用户指南” 。

  2. 如果不存在，则创建文本初始化参数文件（PFILE）。

    如果使用服务器参数文件（SPFILE），则在生产数据库上运行以下语句以创建PFILE：

    从SPFILE创建PFILE;

  3. 创建用于克隆生产数据库的SQL脚本。

    您将使用一个或多个SQL脚本在稍后的步骤中创建CloneDB数据库。要创建SQL脚本，可以使用Oracle提供的名为clonedb.pl的Perl脚本，也可以手动创建SQL脚本。

    要使用clonedb.pl Perl脚本，请完成以下步骤：

    1. 在操作系统提示符下设置以下环境变量：

      MASTER_COPY_DIR - 指定包含在步骤1中创建的备份的目录。确保此目录仅包含生产数据库的数据文件的备份。

      CLONE_FILE_CREATE_DEST - 指定将创建CloneDB数据库文件的目录，包括数据文件，日志文件，控制文件。

      CLONEDB_NAME - 指定CloneDB数据库的名称。

      S7000_TARGET - 如果为备份提供文件系统的NFS主机和CloneDB数据库是Sun Storage 7000，则指定主机的名称。否则，请勿设置此环境变量。仅在必须使用存储快照完成克隆时才设置此环境变量。您可以将S7000存储阵列用于Direct NFS Client而无需设置此变量。

    2. 运行clonedb.pl Perl脚本。

      该脚本位于$ ORACLE_HOME / rdbms / install目录中，并具有以下语法：

      $ ORACLE_HOME / perl / bin / perl $ ORACLE_HOME / rdbms / install / clonedb.pl prod_db_pfile [ sql_script1 ] [ sql_script2 ]

2.10.1.3使用CloneDB克隆数据库之后

创建CloneDB数据库后，您几乎可以使用生产数据库的任何方式使用它。最初，CloneDB数据库为每个数据文件使用最少量的存储。对CloneDB数据库中的行的更改会导致按需分配存储空间。

您可以使用相同的备份文件来创建多个CloneDB数据库。此备份可以由RMAN或存储级别快照执行。如果在数据文件上创建了存储级别快照，则可以使用存储快照名称替换RMAN备份文件名。

您可以使用V$CLONEDFILE视图显示有关CloneDB数据库中每个数据文件的信息。此信息包括备份中的数据文件名，CloneDB数据库中的相应数据文件名，从备份文件读取的块数以及针对备份文件发出的请求数。

由于CloneDB数据库使用备份文件作为其后端存储，因此每个CloneDB数据库必须可以使用备份文件才能运行。如果备份文件不可用，则CloneDB数据库将返回错误。

完成CloneDB数据库的使用后，可以销毁CloneDB数据库环境。您可以删除CloneDB数据库环境中的所有文件，而不会影响生产数据库环境或备份环境。

2.13.3使用Oracle Wallet在DBCA命令中进行数据库用户验证

您可以将Oracle wallet用作安全外部密码存储，以便在DBCA静默模式命令中对数据库用户进行身份验证。

Oracle钱包是Oracle数据库外部的安全软件容器，可用于存储Oracle数据库用户的身份验证凭据。您可以使用以下DBCA silent mode命令参数来使用Oracle wallet验证数据库用户：

  • useWalletForDBCredentials ：指定true以使用Oracle wallet进行数据库用户身份验证，否则指定false 。默认值为false 。

    如果指定了true ，则提供以下附加参数：

    • dbCredentialsWalletLocation ：存储Oracle钱包文件的目录。

    • （可选） dbCredentialsWalletPassword ：Oracle钱包帐户用户的密码。如果Oracle钱包已启用自动登录 ，则无需指定此密码。

您可以在Oracle钱包中存储以下密钥和相关密码，DBCA可以在静默模式下使用这些密钥和相关密码来验证用户身份：

  • oracle.dbsecurity.sysPassword ： SYS用户密码

  • oracle.dbsecurity.systemPassword ： SYSTEM用户密码

  • oracle.dbsecurity.pdbAdminPassword ：可插拔数据库（PDB）管理员密码

  • oracle.dbsecurity.dbsnmpPassword ： DBSNMP用户密码

  • oracle.dbsecurity.asmsnmpPassword ： ASMSNMP用户密码

  • oracle.dbsecurity.lbacsysPassword ： LBACSYS用户密码

  • oracle.dbsecurity.sysdbaUserPassword ：您正在创建或配置的数据库的SYSDBA角色用户密码

  • oracle.dbsecurity.oracleHomeUserPassword ：Oracle主用户密码

  • oracle.dbsecurity.dvUserPassword ：Oracle Data Vault用户密码

  • oracle.dbsecurity.dvAccountManagerPassword ：Oracle Data Vault帐户管理员密码

  • oracle.dbsecurity.emPassword ：企业管理器管理员密码

  • oracle.dbsecurity.asmPassword ： ASM用户密码

  • oracle.dbsecurity.asmsysPassword ： ASMSYS用户密码

  • oracle.dbsecurity.walletPassword ：用于通过目录服务进行身份验证的Oracle钱包帐户用户密码

  • oracle.dbsecurity.userDNPassword ：目录服务用户密码

  • oracle.dbsecurity.srcDBsysdbaUserPassword ：您用作执行某些操作的源的数据库的SYSDBA角色用户密码，例如复制数据库

先决条件
