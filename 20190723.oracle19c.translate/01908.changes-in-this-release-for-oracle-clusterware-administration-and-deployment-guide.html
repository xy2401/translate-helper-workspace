“Oracle Clusterware管理和部署指南”中的此发行版中的更改

本章列出了Oracle Clusterware for Oracle Database 19c和18c中的新功能。

Oracle Clusterware Release 19c中的更改

以下是Oracle Clusterware管理和部署指南（适用于Oracle Clusterware 19c）中的新功能列表。

Oracle Clusterware 19c的SRVCTL更改

Oracle Clusterware 19c包括对服务器控制实用程序（SRVCTL）的更改，包括对现有命令的语法更改以及管理Oracle自动存储管理（Oracle ASM）的命令。

SRVCTL是用于管理Oracle Real Application Clusters（Oracle RAC）和Oracle Clusterware的工具之一。

零停机Oracle Grid Infrastructure补丁

零停机时间Oracle Grid Infrastructure修补功能可以在不中断数据库操作的情况下修补Oracle Grid Infrastructure。修补程序以非滚动方式应用，一次修补一个节点，而此节点上的数据库实例仍可运行。零停机Oracle Grid Infrastructure修补支持具有两个或更多节点的集群上的Oracle Real Application Clusters（Oracle RAC）数据库。

零停机时间Oracle Grid Infrastructure修补通过使您能够执行滚动的Oracle Grid Infrastructure修补程序而不中断正在修补的节点上的数据库操作，并且不会影响这些数据库实例的容量或性能，从而提高了数据库可用性。

快速家庭供应名称更改

在此版本中，以前称为Rapid Home Provisioning的功能将重命名为Fleet Patching and Provisioning。功能没有变化，RHPCTL实用程序仍然是用于管理Fleet Patching和Provisioning操作的工具。

细粒度单实例PDB修补

您可以在统一的Oracle Multitenant环境中修补单个可插拔数据库，从而使错误修复仅在特定可插拔数据库上修补，而不是在整个constainer数据库中修补。细粒度的单实例可插拔数据库修补可以减少广泛采用更改（例如错误修复）所带来的风险，并减少仅在必要时进行更改的影响。

使用车队补丁和配置实现零停机Oracle Grid Infrastructure补丁

零停机时间Oracle Grid Infrastructure修补程序可实现一次性Oracle Grid Infrastructure修补程序的应用，而不会影响Oracle Real Application Clusters（Oracle RAC）数据库实例。使用Fleet Patching和Provisioning将修补程序一次一个地应用于群集中的每个节点。此功能适用于具有两个或更多节点的所有Oracle RAC群集，但目前仅适用于一次性修补程序（不是发布更新或发行版更新修订版）。

使用Fleet Patching和Provisioning应用一次性Oracle Grid Infrastructure补丁，零数据库实例停机时间可以减少对用户的影响以及从Oracle RAC数据库实例到nil的服务中断。使用以前的数据库版本，您必须在应用Oracle Grid Infrastructure修补程序之前关闭数据库实例，这显然会影响企业操作。

Oracle Grid Infrastructure升级的自动化事务处理耗尽

根据数据库服务配置，Oracle Grid Infrastructure升级的自动事务处理耗尽可根据数据库实例自动排空事务，一次一个节点。事务处理功能是数据库服务设计不可或缺的一部分，现在可以自动集成到滚动Oracle Grid Infrastructure修补程序的应用程序中。

使用Fleet Patching和Provisioning在滚动修补期间自动和协调地排空数据库事务可以减少修补操作的影响。一旦用户事务耗尽，就可以完成对集群上特定节点的修补操作，之后重新启动实例和服务，在本地重新建立连接。在修补操作之前，连接将滚动到群集中的下一个节点。

Oracle重新启动修补和升级

使用Fleet Patching和Provisioning修补和升级Oracle Restart。在以前的版本中，Oracle Restart环境需要用户完成修补和升级操作，通常需要手动干预。Fleet Patching和Provisioning自动执行这些过程。

使用Fleet Patching和Provisioning修补和升级Oracle Restart可自动化并标准化Oracle RAC数据库安装中实现的流程。这还可以降低运营需求和风险，尤其是对于大量Oracle Restart部署。

支持OCR和表决磁盘的直接文件放置

从Oracle Grid Infrastructure 19c开始，Oracle Standalone Clusters将取消对共享文件系统上直接OCR和Voting磁盘文件放置的支持。对于Oracle域服务集群，要求将OCR和表决文件放在Oracle自动存储管理（Oracle ASM）中，托管在共享文件系统上并用作ASM磁盘的文件上。

在Oracle Grid Infrastructure 12c第2版（12.2）中，Oracle宣布它将不再支持直接在共享文件系统上放置Oracle Grid Infrastructure Oracle Cluster Registry（OCR）和投票文件。这种支持现在已经废除。从Oracle Grid Infrastructure 19c（19.3）开始，使用Oracle Standalone Clusters，您可以再次将OCR和Voting磁盘文件直接放在共享文件系统上。

Grid Infrastructure Management存储库的可选安装

从Oracle Grid Infrastructure 19c开始，Grid Infrastructure Management Repository（GIMR）对于Oracle Standalone Cluster的新安装是可选的。Oracle Domain Services Clusters仍然需要安装GIMR作为服务组件。

GIMR中包含的数据是基于应用机器学习的预防性诊断的基础，可以帮助提高Oracle Real Application Clusters（RAC）数据库的可用性。对GIMR进行可选安装可以实现更灵活的存储空间管理和更快的部署，尤其是在安装测试和开发系统期间。

Oracle Clusterware 19c中不推荐使用的功能

Oracle Clusterware 19c中不推荐使用以下功能，在将来的版本中可能不支持这些功能：

Addnode脚本的弃用

Oracle Grid Infrastructure 19c中不推荐使用addnode脚本。安装程序向导中提供了向群集添加节点的功能。

可以在将来的版本中删除addnode脚本。不使用addnode脚本（ addnode.sh或addnode.bat ），而是使用安装程序向导添加节点。安装程序向导相对于addnode脚本提供了许多增强功能。使用安装程序向导可将所有软件生命周期操作合并到一个工具中，从而简化管理。

clone.pl脚本的弃用

可以在将来的发行版中删除clone.pl脚本。Oracle建议您使用安装程序向导将提取的黄金映像作为主目录安装，而不是使用clone.pl脚本。

Oracle Clusterware 19c中不支持的功能

这些是Oracle Clusterware 19c不支持的功能：

Flex集群体系结构中叶节点的支持

Oracle Grid Infrastructure 19c中的Oracle Flex Cluster Architecture不再支持叶节点。

在Oracle Grid Infrastructure 19c（19.1）和更高版本中，Oracle Flex Cluster中的所有节点都充当中心节点。Leaf节点在Oracle Flex Cluster架构的原始实现中提供的功能可以很容易地由中心节点提供。因此，不再支持叶节点。

支持标准版2（SE2）数据库版的Oracle Real Application Clusters

从Oracle Database 19c开始，Oracle数据库标准版2（SE2）不支持Oracle Real Application Clusters（Oracle RAC）。

无法将从早期版本使用Oracle Real Application Clusters（Oracle RAC）功能的Oracle Database Standard Edition数据库升级到Oracle Database 19c。要将这些数据库升级到Oracle Database 19c，请在开始升级之前删除Oracle RAC功能，或者从Oracle Database Standard Edition升级到Oracle Database Enterprise Edition。有关每个步骤的更多信息，包括升级后如何重新配置系统，请参阅My Oracle Support说明2504078.1：“Oracle数据库标准版19c中对Oracle Real Application Clusters（RAC）的支持”。

Oracle Clusterware Release 18c中的更改

以下是Oracle Clusterware管理和部署指南（适用于Oracle Clusterware 18c）中的新功能列表。

跨群集依赖性代理

跨群集依赖关系代理提供从一个群集到另一个群集的资源状态更改通知，并使一个群集中的资源能够代表另一个群集中的资源的依赖关系。例如，您可以使用跨群集依赖关系代理，以确保Oracle应用程序成员群集中的应用程序仅在Oracle数据库成员群集中托管的关联数据库可用时才启动。同样，您可以使用跨群集依赖关系代理来确保仅当域服务群集上至少有一个Oracle自动存储管理（Oracle ASM）实例可用时，才会启动Oracle数据库成员群集中的数据库。

也可以看看：

跨群集依赖性代理

共享单客户端访问名称

共享的单客户端访问名称（SCAN）允许在数据中心的专用群集上与其他群集共享一组SCAN虚拟IP（VIP）和侦听器（称为SCAN设置），以避免部署一个每个群集的SCAN设置。此功能不仅减少了与SCAN相关的DNS条目的数量，还减少了必须为群集配置部署的VIP数量。

共享SCAN通过提供可由多个系统同时使用的共享SCAN设置，简化了数据中心中群集组的部署和管理。

节点VIP可选

从此版本开始，节点虚拟IP（VIP）地址的使用在群集环境中是可选的。此增强功能可减少没有节点VIP的部署所需的IP地址数。另一个好处是此更改简化了Oracle Clusterware部署。

此功能仅适用于测试和开发环境。

工程系统支持

使用Rapid Home Provisioning修补Oracle Exadata基础架构。除了修补Oracle数据库和Oracle Grid Infrastructure软件之家之外，您现在还可以在Oracle Exadata环境中修补数据库节点，存储单元和InfiniBand交换机的软件。将Oracle Exadata组件支持集成到Rapid Home Provisioning中，可通过Rapid Home Provisioning服务的集中清单管理和跟踪这些组件的维护。

Dry-Run命令验证

Rapid Home Provisioning命令中包含的工作流程由多个较小的步骤组成，其中一些步骤可能会失败。此版本的Oracle Clusterware包含一个用于多个RHPCTL命令的干运行命令模式，使您可以在不进行任何永久更改的情况下评估这些命令的implact。
