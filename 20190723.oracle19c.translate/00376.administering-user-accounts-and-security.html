7.1关于用户帐户

要使用户访问您的数据库，您必须创建用户帐户并为这些帐户授予适当的数据库访问权限。 用户帐户由用户名标识，并定义用户的属性，包括以下内容：

  • 身份验证方法

  • 数据库验证的密码

  • 用于永久和临时数据存储的默认表空间

  • 表空间配额

  • 帐户状态（已锁定或已解锁）

  • 密码状态（已过期或未过期）

创建用户帐户时，您不仅必须为该帐户分配用户名，密码和默认表空间，还必须执行以下操作：

  • 为帐户授予适当的系统权限，对象权限和角色。

  • 如果用户将创建数据库对象，则在每个将在其中创建对象的表空间上为用户帐户提供空间使用配额。

Oracle建议您为每个用户授予足够的权限来执行其工作，而不是更多。例如，数据库应用程序开发人员需要创建和修改表，索引，视图和存储过程的权限，但不需要（也不应该被授予）删除（删除）表空间或恢复数据库的权限。您可以为数据库管理创建用户帐户，并仅为这些帐户授予一部分管理权限。

此外，您可能希望创建仅由应用程序使用的用户帐户。也就是说，没有人使用这些帐户登录;相反，应用程序使用这些帐户连接到数据库，用户登录到应用程序。这种类型的用户帐户可以避免让应用程序用户直接登录数据库，从而无意中造成损坏。有关详细信息，请参阅“ 关于用户权限和角色 ” 。

创建用户帐户时，您还隐式为该用户创建架构。模式是用户创建的数据库对象（例如表，视图，触发器等）的逻辑容器。模式名称与用户名相同，可用于明确引用用户拥有的对象。例如， hr.employees引用hr模式中名为employees的表。（ employees表由hr拥有。）术语数据库对象和模式对象可互换使用。

删除用户时，必须同时删除该用户的所有模式对象，或者必须先在单独的操作中删除模式对象。

预定义的用户帐户

除了您创建的用户帐户之外，数据库还包括在安装时自动创建的多个用户帐户。

所有数据库都包括管理帐户SYS ， SYSTEM和DBSNMP 。 管理帐户是高权限帐户，只有被授权执行管理任务的人员才需要，例如启动和停止数据库，管理数据库内存和存储，创建和管理数据库用户等。使用SYS或SYSTEM登录Oracle Enterprise Manager Database Express（EM Express）。使用Oracle Database Configuration Assistant（DBCA）创建数据库时，可以为这些帐户分配密码。您不得删除这些帐户。

所有数据库还包括内部帐户 ，这些帐户是自动创建的，以便单个Oracle数据库功能或组件（如Oracle Application Express）可以拥有自己的模式。为了保护这些帐户免受未经授权的访问，它们最初被锁定且密码已过期。（ 锁定的帐户是禁用登录的帐户。）您不得删除内部帐户，也不得使用它们登录数据库。

如果在安装数据库时选择了在数据库中创建示例模式的选项，则数据库也可能包含示例模式 。示例模式是一组相互关联的模式，它们使Oracle文档和Oracle教学材料能够说明常见的数据库任务。这些模式还为您提供了一种在不危及生产数据的情况下进行实验的方法。

每个示例模式都有一个与之关联的用户帐户。例如， hr用户帐户拥有hr模式，该模式包含一组用于人力资源应用程序的简单表。示例模式帐户最初也是锁定的，并且密码已过期。作为数据库管理员，您负责解锁这些帐户并为这些帐户分配密码。

7.1.1关于CDB的共性

在多租户容器数据库（CDB）中，共性的基本原则是每个现有和未来容器中的普遍现象是相同的 。在CDB中，“共同”意味着“所有容器共有”。相反，局部现象仅限于一个现有容器。

共同性原则的必然结果是只有普通用户才能改变普通现象的存在 。更确切地说，只有连接到根的普通用户才能创建，销毁或修改普通用户或角色的CDB范围属性。

7.1.1.1 CDB中的普通用户

普通用户是在根目录以及每个现有和未来可插拔数据库（PDB）中具有相同标识的数据库用户。每个普通用户都可以在根目录内以及普通用户拥有权限的任何PDB中连接和执行操作。

每个普通用户都是Oracle提供的或用户创建的。Oracle提供的普通用户的示例是SYS和SYSTEM 。

普通用户具有以下特征：

  • 普通用户可以登录到具有CREATE SESSION权限的任何容器（包括CDB$ROOT ）。

    普通用户不需要在每个容器中具有相同的权限。例如， c##dba用户可能有权在根和一个PDB中创建会话，但不能在不同的PDB中创建会话。由于具有适当权限的普通用户可以在容器之间切换，因此根目录中的普通用户可以管理PDB。

  • 每个用户创建的普通用户的名称必须以字符c##或C##开头。（Oracle提供的通用用户名没有此限制。）

    没有本地用户名可以以字符c##或C##开头。

  • 普通用户的名称必须仅包含ASCII或EBCDIC字符。

  • 每个普通用户都在所有容器中唯一命名。

    普通用户驻留在根目录中，但必须能够连接到具有相同标识的每个PDB。

  • 普通用户的模式在每个容器中可以不同。

    例如，如果c##dba是对多个容器具有特权的普通用户，则每个容器中的c##dba架构可能包含不同的对象。

7.1.1.2 CDB中的本地用户
本地用户是不常见且只能在单个可插拔数据库（PDB）内操作的用户。 本地用户具有以下特征：

  • 本地用户特定于特定PDB并在此PDB中拥有架构。

  • 无法在根目录中创建本地用户。

  • 一个PDB上的本地用户无法登录到另一个PDB或根目录。

  • 本地用户的名称不能以字符c##或C##开头。

  • 本地用户的名称在其PDB中必须是唯一的。

  • 用户名和包含该用户架构的PDB确定唯一的本地用户。例如，名为rep的本地用户和模式可以存在于名为hrpdb的PDB上。名为salespdb的PDB上可以存在名为rep完全独立的本地用户和模式。

  • 本地用户是否可以访问公共模式中的对象取决于其用户权限。

    例如， c##dba普通用户可以在hrpdb PDB上的c##dba架构中创建一个表。除非c ## dba向此表上的本地hr用户授予必要的权限，否则hr无法访问它。

7.2关于用户权限和角色

用户权限提供基本级别的数据库安全性。它们旨在控制用户对数据的访问，并限制用户可以执行的SQL语句的种类。 创建用户时，您授予权限以使用户能够连接到数据库，运行查询和进行更新，创建模式对象等。

用户权限的主要类型如下 ：

  • 系统特权 - 系统特权使用户能够执行特定操作，或对特定类型的任何模式对象执行操作。例如，系统特权CREATE TABLE允许用户在与该用户关联的模式中创建表，系统特权CREATE USER允许用户创建数据库用户。

  • 对象权限 - 对象权限使用户能够对特定模式对象执行特定操作。不同的对象权限可用于不同类型的架构对象。从EMPLOYEES表中选择行或从DEPARTMENTS表中删除行的权限是对象权限的示例。

使用角色可以更轻松地管理Mana ging权限， 角色是相关权限的命名组。您可以创建角色，为角色授予系统和对象权限，然后向用户授予角色。您还可以将角色授予其他角色。与架构对象不同，角色不包含在任何架构中。

表7-1列出了Oracle数据库中预定义的三种广泛使用的角色。您可以在创建用户时或之后的任何时间授予这些角色。

允许用户在与该用户关联的模式中创建，修改和删除某些类型的模式对象。仅将此角色授予开发人员和必须创建架构对象的其他用户。此角色授予创建对象系统特权的子集。例如，它授予CREATE TABLE系统特权，但不授予CREATE VIEW系统特权。它仅授予以下权限： CREATE CLUSTER ， CREATE INDEXTYPE ， CREATE OPERATOR ， CREATE PROCEDURE ， CREATE SEQUENCE ， CREATE TABLE ， CREATE TRIGGER ， CREATE TYPE 。

7.2.1.1 CDB中的常见角色

常见角色是存在于根以及每个现有和未来可插拔数据库（PDB）中的数据库角色。通用角色对跨容器操作很有用，可确保普通用户在每个容器中都有一个角色。

每个常见角色都是用户创建的或Oracle提供的。所有Oracle提供的角色都很常见，例如DBA和PUBLIC 。用户创建的公共角色必须具有以C##或c##开头的名称，并且必须仅包含ASCII或EBCDIC字符。例如，多租户容器数据库（CDB）管理员可能会创建公共用户c##dba ，然后通常将DBA角色授予此用户，以便c##dba在任何现有和将来的PDB中具有DBA角色。

用户只能对公共角色执行常见操作，例如，在满足以下条件时，通常为角色授予权限：

  • 用户是当前容器为root的普通用户。

  • 用户具有通常授予的SET CONTAINER权限，这意味着该权限适用于所有容器。

  • 用户具有控制执行指定操作的权限的权限，并且此权限已被普遍授予。

例如，要创建公共角色，普通用户必须具有通常授予的CREATE ROLE和SET CONTAINER权限。必须在根目录中创建使用Oracle Enterprise Manager Database Express（EM Express）创建的公共角色。

7.2.1.2 PDB中的本地角色

本地角色仅存在于单个可插拔数据库（PDB）中，就像非CDB中的角色仅存在于非CDB中一样。本地角色只能包含在角色所在的容器中应用的角色和权限。

同一多租户容器数据库（CDB）中的PDB可能包含具有相同名称的本地角色。例如，用户创建的角色pdbadmin可能同时存在于hrpdb和salespdb PDB中。这些角色完全相互独立，就像它们在不同的非CDB中一样。

必须在将使用它的PDB中创建使用Oracle Enterprise Manager Database Express（EM Express）创建的本地角色。

7.2.2关于CDB中的权限和角色授予

就像在非CDB中一样，多租户容器数据库（CDB）中的用户可以授予角色和特权。CDB的主要区别在于本地授予和通常授予的角色和权限之间的区别。本地授予的权限或角色只能在授予它的容器中执行。通常授予的权限或角色可在每个现有和未来容器中执行。

用户和角色可以是普通的或本地的。但是，特权本身既不是普通的，也不是地方的。如果用户使用CONTAINER=CURRENT子句在本地授予权限， CONTAINER=CURRENT授权者只能在当前容器中执行权限。如果用户通常使用CONTAINER=ALL子句授予权限，则被授权者可以在任何现有和将来的容器中执行权限。

注意：

使用Oracle Enterprise Manager Database Express（EM Express）在CDB中授予权限或角色时，授予权限的容器将确定它是通常授予的权限还是本地授予的权限或角色。

例如，当您使用EM Express在根中授予权限时，该权限是被授权者可以在任何现有和未来容器中执行的通常授予的权限。当您使用EM Express在可插拔数据库（PDB）中授予权限时，该权限是被授权者只能在该PDB中执行的本地授予权限。

在国家开发银行中，每一项授权行为，无论是本地授权还是普通授权，都发生在特定的容器内。授予的基本原则如下：

  • 普通和地方现象都可以在当地授予和授予。

  • 只有普通现象可以一般授予或授予。

根据定义，本地用户，角色和特权仅限于特定容器。因此，本地用户可能不会共同授予角色和特权，并且可能不会普遍授予本地角色和特权。

7.3.1 SYS和SYSTEM用户

安装Oracle数据库时会自动创建SYS和SYSTEM管理用户帐户。它们都是使用您在安装时提供的密码创建的，并且它们都自动被授予DBA角色。

  • SYS

    此帐户可以执行所有管理功能。数据库数据字典的所有基础（基础）表和视图都存储在SYS模式中。这些基表和视图对Oracle数据库的操作至关重要。为了维护数据字典的完整性， SYS模式中的表仅由数据库操作。任何用户或数据库管理员都不应修改它们。您不能在SYS模式中创建任何表。

    SYS用户被授予SYSDBA权限，该权限使用户能够执行高级管理任务，如备份和恢复。

  • 系统

    此帐户可以执行除以下操作以外的所有管理功能：

    • 备份和恢复

    • 数据库升级

    虽然您可以使用此帐户执行日常管理任务，但Oracle强烈建议创建命名用户帐户以管理Oracle数据库以启用对数据库活动的监视。

注意：

SYSBACKUP是另一个自动创建的帐户，用于执行备份和恢复。有关详细信息，请参阅“ 配置用户以执行备份和恢复 ” 。

7.3.2 SYSDBA和SYSOPER系统特权

SYSDBA和SYSOPER是执行高级管理操作（如创建，启动，关闭，备份或恢复数据库）所需的管理权限。 SYSDBA系统特权适用于完全授权的数据库管理员， SYSOPER系统特权允许用户执行基本操作任务，但无法查看用户数据。

即使数据库未打开， SYSDBA和SYSOPER系统特权也允许访问数据库实例。因此，控制这些特权完全在数据库本身之外。此控件使被授予这些权限之一的管理员能够连接到数据库实例以启动数据库。

您还可以将SYSDBA和SYSOPER权限视为允许您执行某些数据库操作的连接类型，这些操作不能以任何其他方式授予特权。例如，如果您具有SYSDBA权限，则可以使用AS SYSDBA连接到数据库。

SYS用户在安装时自动被授予SYSDBA权限。以SYS用户身份登录时，必须以SYSDBA或SYSOPER身份连接到数据库。以SYSDBA用户身份连接将调用SYSDBA权限;连接为SYSOPER调用SYSOPER权限。EM Express允许您以SYS用户身份登录，并以SYSDBA或SYSOPER身份连接。

使用SYSDBA或SYSOPER权限进行连接时，将使用默认架构进行连接，而不是使用通常与用户名关联的架构。对于SYSDBA此架构是SYS ;对于SYSOPER ，架构是PUBLIC 。

注意：

以SYS用户身份连接时，您对数据字典表具有无限权限。确保您不修改任何数据字典表。

7.5.8使用户密码失效

您可以使用Oracle Enterprise Manager Database Express（EM Express）使用户密码到期。 当您使用户密码过期时，系统会提示用户在下次用户登录时更改其密码。使密码失效的原因包括：

  • 用户密码被破坏。

  • 您有一个安全策略，要求用户定期更改其密码。

    注意：

    您可以在一定时间间隔后自动使用户密码自动过期。请参阅“ 设置数据库密码策略 ” 。

  • 用户忘记了他或她的密码。

    在第三种情况下，您修改用户帐户，分配新的临时密码并使密码到期。然后，用户使用临时密码登录，并提示您选择新密码。

要使用户密码过期：

  1. 在EM Express中，转到“用户”页面，如“ 查看用户帐户 ”中所述 。

  2. 单击所需的用户帐户。

  3. 从“ 操作”菜单中，选择“ 更改帐户” 。

    将出现“更改帐户”页面。

  4. 启用密码已过期 ，然后单击“ 确定” 。

7.6.1关于密码策略

创建用户帐户时，会为该用户帐户分配默认密码策略。 新安装的数据库的默认密码策略包括以下指令：

  • 用户帐户的密码将在180天后自动过期。

  • 密码到期后7天锁定用户帐户。

  • 登录尝试失败10次后，用户帐户将被锁定1天。

默认密码策略通过名为profile的数据库对象分配给用户帐户。每个用户帐户都分配了一个配置文件，该配置文件有几个描述密码策略的属性。数据库附带默认配置文件（名为DEFAULT ），除非您在创建用户帐户时另行指定，否则默认配置文件将分配给用户帐户。

为了更好的数据库安全性，您可能需要实施更严格的密码策略。例如，您可能希望密码每70天过期一次，并且您可能希望在三次登录尝试失败后锁定用户帐户。（当用户输入错误的帐户密码时，会发生用户帐户登录尝试失败。）您可能还希望要求密码足够复杂，以便为试图通过猜测密码进入系统的入侵者提供合理保护。例如，您可以指定密码必须包含至少一个数字和一个标点符号。

您可以通过修改DEFAULT配置文件的密码相关属性来更改数据库中每个用户帐户的密码策略。

注意：

可以为不同的用户帐户设置不同的密码策略。您可以通过创建多个配置文件，为每个配置文件设置不同的密码相关属性，并为不同的用户帐户分配不同的配置文件来实现此目本节未涉及此方案。

7.6.2修改默认密码策略

您可以通过修改名为DEFAULT的配置文件的密码相关属性来修改每个数据库用户帐户的默认密码策略。您可以使用Oracle Enteprise Manager Database Express（EM Express）修改默认密码策略。

要修改默认密码策略：

  1. 使用具有管理默认密码策略权限的用户帐户登录EM Express。这种用户帐户的一个例子是SYSTEM 。

  2. 在Security菜单中，选择Profiles 。

    将显示“配置文件”页面。

    注意：

    当您使用EM Express管理多租户容器数据库（CDB）及其可插拔数据库（PDB）时，“配置文件”选项仅在PDB级别可用，因为配置文件位于CDB中的PDB级别。

  3. 选择名为DEFAULT的配置文件，然后从“ 操作”菜单中选择“ 更改配置文件” 。

    将出现“更改配置文件”向导，并显示“常规”页面。

  4. 单击向右箭头按钮。

    将出现“密码”页面。


    说明exp_alter_profile_pwd.gif的描述

  5. 根据需要更改字段值。单击每个字段旁边的向下箭头以查看选项列表。从列表中选择一个值，或输入一个值。

  6. 单击“ 确定”保存更改。

    出现确认消息。
