一个 DBMS_JAVA包

本章提供DBMS_JAVA包的说明。此包中的函数和过程提供了从Java访问RDBMS功能的入口点。

A.1长名

FUNCTION longname（短名称VARCHAR2）RETURN VARCHAR2

返回指定Java架构对象的完全限定名称。由于Java类和方法的名称可能超过最大SQL标识符长度，因此Oracle JVM在内部使用缩写名称进行SQL访问。此函数返回任何截断名称的原始Java名称。此函数的一个示例是显示无效的类的完全限定名称：

SELECT dbms_java.longname（object_name）FROM user_objects WHERE object_type ='JAVA CLASS'AND status ='INVALID';

A.6解析器

FUNCTION解析器（名称VARCHAR2，所有者VARCHAR2，类型VARCHAR2）RETURN VARCHAR2

返回在name和在owner指定的模式中指定的对象的解析器规范，其中对象具有type中指定的type 。调用者必须具有EXECUTE特权并且可以访问给定对象以使用此功能。

name参数是对象的简称。

type的值可以是SOURCE或CLASS 。

如果有错误，则返回NULL 。如果底层对象已更改，则抛出ObjectTypeChangedException 。

您可以按如下方式调用此函数：

SELECT dbms_java.resolver（'tst'，'HR'，'CLASS'）FROM DUAL;

这将返回：

DBMS_JAVA.RESOLVER（'TST'，'HR'，'CLASS'）---------------------------------- -------（（* HR）（* PUBLIC））

A.7衍生自

FUNCTION derivedFrom（名称VARCHAR2，所有者VARCHAR2，类型VARCHAR2）RETURN VARCHAR2

返回在type和在owner指定的模式中指定的类型name中指定的对象的源名称。调用者必须具有EXECUTE特权并且可以访问给定对象以使用此功能。

name参数以及返回的源名称是对象的简称。

type的值可以是SOURCE或CLASS 。

如果有错误，则返回NULL 。如果底层对象已更改，则抛出ObjectTypeChangedException 。

如果未在Oracle JVM中编译对象，则返回的值将为NULL 。

您可以按如下方式调用此函数：

SELECT dbms_java.derivedFrom（'tst'，'HR'，'CLASS'）FROM DUAL;

这将返回：

DBMS_JAVA.DERIVEDFROM（'TST'，'HR'，'CLASS'）---------------------------------- ------- tst

A.8 fixed_in_instance

FUNCTION fixed_in_instance（名称VARCHAR2，所有者VARCHAR2，类型VARCHAR2）返回编号

返回在type和在owner指定的模式中指定的类型name中指定的对象的永久保留状态。调用者必须具有EXECUTE特权并且可以访问给定对象以使用此功能。

name参数是对象的简称。

type的值可以是RESOURCE ， SOURCE ， CLASS或SHARED_DATA 。

返回的数字为0 ，表示状态未保留，或1 ，表示保持状态。

您可以按如下方式调用此函数：

SELECT dbms_java.fixed_in_instance（'tst'，'HR'，'CLASS'）FROM DUAL;

这将返回：

DBMS_JAVA.FIXED_IN_INSTANCE（'TST'，'HR'，'CLASS'）---------------------------------- ------- 0

请考虑以下声明：

SELECT dbms_java.fixed_in_instance（'java / lang / String'，'SYS'，'CLASS'）FROM DUAL;

这将返回：

DBMS_JAVA.FIXED_IN_INSTANCE（'JAVA / LANG / STRING'，'SYS'，'CLASS'）------------------------------ ------------------------------- 1

A.10 export_source

PROCEDURE export_source（名称VARCHAR2，模式VARCHAR2，src BLOB）PROCEDURE export_source（名称VARCHAR2，src BLOB）PROCEDURE export_source（名称VARCHAR2，src CLOB）PROCEDURE export_source（名称varchar2，模式varchar2，src CLOB）

用于将Java源代码作为Java源架构对象导出到Oracle数据库。源是通过name参数指定的。源可以导出到BLOB或CLOB对象中。源的内部表示使用UTF8格式，因此该格式也用于将源存储在BLOB中。源架构对象是在指定的架构中创建的。如果未指定架构，则使用当前架构。

A.12 export_resource

PROCEDURE export_resource（名称VARCHAR2，模式VARCHAR2，src BLOB）PROCEDURE export_resource（名称VARCHAR2，src BLOB）PROCEDURE export_resource（名称VARCHAR2，模式VARCHAR2，src CLOB）PROCEDURE export_resource（名称VARCHAR2，src CLOB）

A.13 loadjava

PROCEDURE loadjava（options VARCHAR2）PROCEDURE loadjava（options VARCHAR2，resolver VARCHAR2）PROCEDURE loadjava（options VARCHAR2，resolver VARCHAR2，status NUMBER）

允许您使用调用而不是通过loadjava命令行工具将类加载到数据库中。您可以在Java应用程序中调用此过程，如下所示：

CALL dbms_java.loadjava（'...选项......“）;

选项与命令行中指定的选项相同。每个选项应以空格分隔。不要用逗号分隔选项。唯一的例外是loadjava -resolver选项，它包含空格。对于-resolver ，首先指定所有其他选项，用逗号分隔这些选项，然后指定-resolver选项，如下所示：

CALL dbms_java.loadjava（'... options ...'，'resolver_options'）;

不要指定-thin ， -oci ， -user和-password选项，因为它们与数据库连接相关。输出定向到System.err 。输出通常转到跟踪文件，但可以重定向。

A.15 grant_permission

PROCEDURE grant_permission（grantee VARCHAR2，permission_type VARCHAR2，permission_name VARCHAR2，permission_action VARCHAR2）

调用grant_permission的结果是策略表中的活动行，该权限由permission_type ， permission_name和permission_action指定的permission_type grantee 。如果已存在与这些参数匹配的已启用行，则表未经修改。如果该行存在但已禁用，则启用该行。如果不存在匹配的行，则插入一行。参数说明：

  • grantee是架构或角色的名称

  • permission_type是扩展java.lang.security.的类的完全限定名称java.lang.security.允许

  • permission_name是Permission的名称

  • permission_action是permission_action的操作

A.25 set_preference

PROCEDURE set_preference（用户VARCHAR2，类型VARCHAR2，abspath VARCHAR2，密钥VARCHAR2，值VARCHAR2）

在SYS:java$prefs$表中插入或更新一行，如下所示：

user参数指定首选项应附加到的模式的名称。如果登录的架构不是SYS ，则user必须指定当前登录的架构，否则INSERT将失败。type参数可以取值U （表示用户首选项）或S （表示系统首选项）。abspath参数指定首选项的绝对路径。 key是用于查找的首选项键， value是首选项键的值。

A.59 set_runtime_exec_credentials

PROCEDURE set_runtime_exec_credentials（dbuser VARCHAR2，osuser VARCHAR2，ospass VARCHAR2）

其中， dbuser是数据库用户的名称或模式名称和osuser ， ospass是OS帐户凭据。

将数据库用户/模式dbuser与osuser/ospass操作系统（OS）凭据对osuser/ospass 。此关联已加密并存储在SYS用户拥有的表中。一旦建立了新的有效关联，每个新的OS进程都由java.lang.分叉java.lang.由dbuser调用以运行OS命令的Runtime.exec方法或每个ProceessBuilder作为UID osuser运行，而不是作为Oracle进程的OS ID运行。也就是说，分叉进程的UID位设置为UID osuser 。

注意：

DBA和安全管理员可以使用此过程来加强部署到Oracle数据库的Java应用程序的安全性。通过指定权限较低的帐户，DBA可以根据需要限制生成的进程的权限和访问权限。您必须是SYS用户才能使用set_runtime_exec_credentials过程，否则会引发ORA-01031: insufficient privileges错误。创建新进程时，使用无效的帐户凭据会导致IOException 。

以下示例显示了如何使用此过程：

例1

以下命令将用户/架构DBUSER到凭证osuser / ospass ：

dbms_java.set_runtime_exec_credentials（'DBUSER'，'osuser'，'ospass'）;

例2

以下任一命令都DBUSER绑定DBUSER和凭证osuser / ospass ：

dbms_java.set_runtime_exec_credentials（'DBUSER'，''，''）;
dbms_java.set_runtime_exec_credentials（'DBUSER'，null，null）;

注意：

要使用set_runtime_exec_credentials过程，必须在oracle产品安装期间将Oracle jssu工具配置为setuid root ，否则process spawn via jssu failed...的process spawn via jssu failed... 在创建进程时可能会引发IOException。
