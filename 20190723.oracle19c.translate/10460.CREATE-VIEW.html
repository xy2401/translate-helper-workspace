创建视图

目的

使用CREATE VIEW语句定义视图 ，该视图是基于一个或多个表或视图的逻辑表。视图本身不包含任何数据。视图所基于的表称为基表 。

您还可以在现有视图机制之上创建支持LOB，对象类型， REF数据类型，嵌套表或varray类型的对象视图或关系视图。对象视图是用户定义类型的视图，其中每行包含对象，每个对象具有唯一的对象标识符。

您还可以创建XMLType视图，这些视图类似于对象视图，但显示来自XMLType的基于XMLSchema的表的数据。

也可以看看：

  • “Oracle数据库概念” ，“ Oracle数据库开发指南”和“ Oracle数据库管理员指南” ，了解有关各种类型的视图及其用途的信息

  • 有关XMLType视图的信息，请参阅Oracle XML DB Developer's Guide

  • 有关修改视图和从数据库中删除视图的信息，请参阅ALTER VIEW和DROP VIEW

先决条件

要在自己的架构中创建视图，您必须具有CREATE VIEW系统特权。要在另一个用户的模式中创建视图，您必须具有CREATE ANY VIEW系统特权。

要创建子视图，您必须具有UNDER ANY VIEW系统特权或UNDER上的UNDER对象特权。

包含视图的模式的所有者必须具有从视图所基于的所有表或视图中选择（ READ或SELECT特权），插入，更新或删除行所需的特权。必须直接授予所有者这些权限，而不是通过角色。

要在创建对象视图时使用对象类型的基本构造方法，必须满足以下条件之一：

  • 对象类型必须与要创建的视图属于同一模式。

  • 您必须具有EXECUTE ANY TYPE系统权限。

  • 您必须具有该对象类型的EXECUTE对象特权。

语义

或替换

如果视图已存在，请指定OR REPLACE以重新创建视图。您可以使用此子句更改现有视图的定义，而无需删除，重新创建和重新授予先前在其上授予的对象权限。

重新创建视图时，将删除在传统视图上定义的INSTEAD OF触发器。重新创建编辑视图时，将保留在编辑视图上定义的DML触发器。但是，如果已更改版本视图以使其无法再编译，则此类触发器可永久无效 - 例如，如果已删除触发器定义中引用的版本控制视图列。

如果任何物化视图依赖于view ，那么这些物化视图将被标记为UNUSABLE并且需要完全刷新才能将它们恢复到可用状态。查询重写无法使用无效的实例化视图，并且在重新编译之前无法刷新。

您无法使用常规视图将版本视图或版本视图替换为传统视图。有关版本化视图的更多信息，请参见Oracle数据库开发指南 。

也可以看看：

  • ALTER MATERIALIZED VIEW以获取有关刷新无效实体化视图的信息

  • 有关物化视图的信息，请参阅Oracle数据库概念

  • 有关INSTEAD OF子句的更多信息，请创建TRIGGER

力

如果要创建视图，请指定FORCE ，无论视图的基表或引用的对象类型是否存在，或者包含该视图的模式的所有者是否具有对它们的特权。在对视图发出任何SELECT ， INSERT ， UPDATE或DELETE语句之前，这些条件必须为true。

如果视图定义包含任何约束，则CREATE VIEW ...如果基表不存在或引用的对象类型不存在， FORCE失败。CREATE VIEW ......如果视图定义命名不存在的约束， FORCE也会失败。

没有力量

如果要仅在基表存在且包含视图的模式的所有者对其具有特权NOFORCE创建视图，请指定NOFORCE 。这是默认值。

EDITIONING

使用此子句可以创建编辑视图 。编辑视图是单表视图，它从基表中选择所有行并显示基表列的子集。您可以使用版本控制视图在升级等管理操作期间将应用程序与DDL更改隔离到基表。您可以通过查询USER_ ， ALL_和DBA_EDITIONING_VIEW数据字典视图来获取有关现有编辑视图与其基表的关系的信息。

版本视图的所有者必须启用版本。有关更多信息，请参阅ENABLE EDITIONS 。

编辑视图的注释

编辑视图在几个重要方面与传统视图不同：

  • 编辑视图仅用于为表中的列子集选择和提供别名。因此，用于创建编辑视图的语法比用于创建传统视图的语法更受限制。即使您指定FORCE ，任何违反后续限制的行为都会导致视图创建失败。

  • 您可以在编辑视图上创建DML触发器。在这种情况下，数据库将编辑视图视为触发器的基础对象。当DML操作以编辑视图本身为目标时，会触发此类触发器。如果DML操作以基表为目标，则不会触发它们。

  • 您无法在编辑视图上创建INSTEAD OF触发器。

版本化视图的限制

编辑视图受以下限制：

  • 在任何版本中，您只能为任何单个表创建一个版本视图。

  • 您不能指定object_view_clause ， XMLType_view_clause或BEQUEATH子句。

  • 您无法在版本控制视图上定义WITH CHECK OPTION约束。

  • 在定义子查询的选择列表中，您只能指定对基表列的简单引用，并且您只能在选择列表中指定基表的每一列一次。星号通配符*和t_alias 。 *支持指定基表的所有列。

  • 视图的定义子查询的FROM子句只能引用单个现有数据库表。不允许加入。基表必须与正在创建的视图位于同一模式中。您不能使用同义词来标识表，但可以指定表别名。

  • 定义子查询的以下子句对于编辑视图无效： subquery_factoring_clause ， DISTINCT或UNIQUE ， where_clause ， hierarchical_query_clause ， group_by_clause ， HAVING condition， model_clause或set运算符（ UNION ， INTERSECT或MINUS ）

共享

仅当在应用程序根目录中创建视图时，此子句才适用。这种类型的视图称为应用程序公共对象，其数据可以与属于应用程序根目录的应用程序PDB共享。要确定如何共享视图数据，请指定以下共享属性之一：

  • METADATA - 元数据链接共享视图的元数据，但其数据对每个容器都是唯一的。这种类型的视图称为元数据链接的应用程序公共对象 。

  • DATA - 数据链接共享视图，其数据对于应用程序容器中的所有容器都是相同的。其数据仅存储在应用程序根目录中。这种类型的视图称为数据链接的应用程序公共对象 。

  • EXTENDED DATA - 扩展数据链接共享视图，应用程序根目录中的数据对应用程序容器中的所有容器都是相同的。但是，应用程序容器中的每个应用程序PDB都可以存储应用程序PDB唯一的数据。对于这种类型的视图，数据存储在应用程序根目录中，并且可选地存储在每个应用程序PDB中。这种类型的视图称为扩展数据链接应用程序公共对象 。

  • NONE - 视图未共享。

如果省略此子句，则数据库使用DEFAULT_SHARING初始化参数的值来确定视图的共享属性。如果DEFAULT_SHARING初始化参数没有值，则默认值为METADATA 。

创建传统视图时，可以指定METADATA ， DATA ， EXTENDED DATA或NONE 。

创建对象视图或XMLTYPE视图时，只能指定METADATA或NONE 。

创建后，您无法更改视图的共享属性。

别号

指定由视图的定义查询选择的表达式的名称。别名数必须与视图选择的表达式数相匹配。别名必须遵循命名Oracle数据库架构对象的规则。别名在视图中必须是唯一的。

如果省略别名，则数据库将从查询中的列或列别名派生它们。因此，如果查询包含表达式而不是列名，则必须使用别名。此外，如果视图定义包含约束，则必须指定别名。

对视图别名的限制

创建对象视图时，无法指定别名。

使用此子句指定视图列是VISIBLE还是INVISIBLE 。默认情况下，视图列VISIBLE不论其基表中的知名度，除非你指定INVISIBLE 。这适用于传统视图和编辑视图。有关这些子句的完整信息，请参阅CREATE TABLE文档中的“ VISIBLE | INVISIBLE ” 。

Oracle数据库不强制实施视图约束。有关视图约束（包括限制）的完整讨论，请参阅“ 视图约束 ” 。

OF type_name 子句

使用此子句显式创建type_name类型的对象视图 。对象视图的列对应于type_name类型的顶级属性。每行将包含一个对象实例，每个实例将与WITH OBJECT IDENTIFIER子句中指定的对象标识符相关联。如果省略schema ，则数据库将在您自己的模式中创建对象视图。

对象表以及XMLType表，对象视图和XMLType视图没有为它们指定任何列名。因此，Oracle数据库定义了系统生成的伪列OBJECT_ID 。您可以在查询中使用此列名，并WITH OBJECT IDENTIFIER子句创建对象视图。

WITH OBJECT IDENTIFIER子句

使用WITH OBJECT IDENTIFIER子句指定顶级（根）对象视图。此子句允许您指定将用作标识对象视图中每一行的键的对象类型的属性。在大多数情况下，这些属性对应于基表的主键列。您必须确保属性列表是唯一的，并确保视图中只有一行。WITH OBJECT IDENTIFIER和WITH OBJECT ID子句可以互换使用，并提供语义清晰度。

对象视图的限制

对象视图受以下限制：

  • 如果您尝试取消引用或固定在对象视图中解析为多个实例的主键REF ，则数据库将返回错误。

  • 如果要创建子视图，则不能指定此子句，因为子视图从超级视图继承对象标识符。

如果在对象表或对象视图上定义了对象视图，则可以省略此子句或指定DEFAULT 。

默认

如果希望数据库使用基础对象表或对象视图的内部对象标识符来唯一标识每一行，请指定DEFAULT 。

属性

对于attribute ，指定对象类型的属性，数据库应从该属性为对象视图创建对象标识符。

根据条款

使用UNDER子句根据对象超级视图指定子视图。

子视图的限制

子视图受以下限制：

  • 您必须在与superview相同的架构中创建子视图。

  • 对象类型type_name必须立即亚型superview 。

  • 您只能在同一个超级视图下创建特定类型的一个子视图。

默认收集

使用此子句指定视图的默认排序规则。默认排序规则用作视图定义查询中包含的所有字符文字的派生排序规则。视图列不使用默认排序规则;视图列的排序规则是从视图的定义子查询派生的。如果任何字符列基于定义子查询中没有派生排序规则的表达式，则CREATE VIEW语句将失败并显示错误。

对于collation_name ，请指定有效的命名排序规则或伪排序规则。

如果省略此子句，则视图的默认排序规则将设置为包含视图的模式的有效模式默认排序规则。有关有效架构默认排序规则的详细信息，请参阅ALTER SESSION的DEFAULT_COLLATION子句。

仅当COMPATIBLE初始化参数设置为12.2或更高，并且MAX_STRING_SIZE初始化参数设置为EXTENDED ，才能指定DEFAULT COLLATION子句。

要更改视图的默认排序规则，必须重新创建视图。

对视图的默认排序规则的限制

如果视图的定义查询包含WITH plsql_declarations子句，则视图的默认排序规则必须为USING_NLS_COMP 。

遗赠

使用BEQUEATH子句指定视图中引用的函数是使用视图调用者的权限还是视图定义者的权限执行的。

当前用户

如果指定BEQUEATH CURRENT_USER ，则只要满足下列条件之一，就会使用视图调用者的权限执行视图引用的函数：

  • 视图所有者对调用用户具有INHERIT PRIVILEGES对象特权。

  • 视图所有者具有INHERIT ANY PRIVILEGES系统特权。

如果视图的查询调用对身份或权限敏感的SQL函数或调用者的权限PL / SQL或Java函数，则操作执行中的当前模式，当前用户和当前启用的角色将继承自查询用户的环境，而不是来自视图的所有者。

该子句不会将视图本身变为调用者的权限对象。视图中的名称解析仍然使用视图所有者的架构进行处理，并且使用视图所有者的权限完成对视图的权限检查。

DEFINER

如果指定BEQUEATH DEFINER ，则视图引用的函数将使用视图定义者的权限执行。如果对视图的查询调用对身份或特权敏感的SQL函数或调用者权限PL / SQL或Java函数，则操作执行中的当前模式，当前用户和当前启用的角色将继承自风景。

视图中的名称解析使用视图所有者的架构进行处理，并且使用视图所有者的权限完成对视图的权限检查。

这是默认值。

对BEQUEATH条款的限制

您不能使用EDITIONING子句指定此子句。

定义视图查询的限制

视图查询受以下限制：

  • 子查询无法选择CURRVAL或NEXTVAL伪列。

  • 如果子查询选择ROWID ， ROWNUM或LEVEL伪列，则这些列必须在视图子查询中具有别名。

  • 如果子查询使用星号（*）来选择表的所有列，并且稍后向表中添加新列，则在通过发出CREATE OR REPLACE VIEW语句重新创建视图之前，视图将不包含这些列。

  • 对于对象视图，子查询选择列表中的元素数必须与对象类型的顶级属性数相同。每个选择元素的数据类型必须与相应的顶级属性相同。

  • 您不能指定SAMPLE子句。

前面的限制也适用于物化视图。

有关可更新视图的说明

以下说明适用于可更新视图：

可更新视图是可用于插入，更新或删除基表行的视图。您可以创建一个本质上可更新的视图，或者您可以在任何视图上创建一个INSTEAD OF触发器以使其可更新。

要了解是否可以修改固有可更新视图的列以及以何种方式修改，请查询USER_UPDATABLE_COLUMNS数据字典视图。此视图显示的信息仅对固有可更新视图有意义。要使视图本身可更新，必须满足以下条件：

  • 视图中的每个列都必须映射到单个表的列。例如，如果视图列映射到TABLE子句（未引入的集合）的输出，则视图本身不可更新。

  • 视图不得包含以下任何构造：

    • 一组运算符
    • DISTINCT运算符
    • 聚合或分析函数
    • GROUP BY ， ORDER BY ， MODEL ， CONNECT BY或START WITH子句
    • SELECT列表中的集合表达式
    • SELECT列表中的子查询
    • 指定WITH READ ONLY子查询
    • 如Oracle数据库管理员指南中所述 ，加入了一些例外情况

  • 此外，如果固有可更新视图包含伪列或表达式，则无法使用引用任何这些伪列或表达式的UPDATE语句更新基表行。

  • 如果希望连接视图可更新，则必须满足以下所有条件：

    • DML语句必须仅影响连接下的一个表。

    • 对于INSERT语句，不能WITH CHECK OPTION创建视图，并且插入值的所有列必须来自密钥保留表 。密钥保留表是基表中的每个主键或唯一键值在连接视图中也是唯一的。

    • 对于UPDATE语句，不能WITH CHECK OPTION创建视图，并且必须从保留密钥的表中提取更新的所有列。

    • 对于DELETE语句，如果连接导致多个密钥保留表，则Oracle数据库将从FROM子句中指定的第一个表中删除，无论该视图是否是WITH CHECK OPTION创建的。

subquery_restriction_clause

使用subquery_restriction_clause以下列方式之一限制视图的定义查询：

只读

指定WITH READ ONLY以指示无法更新表或视图。

与检查选项

指定WITH CHECK OPTION以指示Oracle数据库禁止对将生成子查询中未包含的行的表或视图进行任何更改。在DML语句的子查询中使用时，可以在FROM子句的子查询中指定此子句，但不能在WHERE子句的子查询中指定。

CONSTRAINT 约束

指定READ ONLY或CHECK OPTION约束的名称。如果省略此标识符，则Oracle会自动为约束指定SYS_C n形式的名称，其中n是一个整数，使数据库中的约束名称唯一。

注意：

对于表， WITH CHECK OPTION保证插入和更新产生定义表子查询可以选择的表。对于视图，如果出现以下情况， WITH CHECK OPTION无法保证：

  • 此视图的定义查询或此视图所基于的任何视图中都有子查询

  • 使用INSTEAD OF触发器执行INSERT ， UPDATE或DELETE操作。

对 subquery_restriction_clause的 限制

如果指定ORDER BY子句，则不能指定此子句。

也可以看看：

CONTAINER_MAP

指定CONTAINER_MAP子句以使用容器映射查看视图。

CONTAINERS_DEFAULT

指定CONTAINERS_DEFAULT子句来启用视图CONTAINERS条款。

例子

创建视图：示例

以下语句创建名为emp_view的示例表employees emp_view 。该视图显示了部门20中的员工及其年薪：

CREATE VIEW emp_view AS SELECT last_name，salary * 12 annual_salary FROM employees WHERE department_id = 20;

视图声明无需基于表达式salary*12定义列的名称，因为子查询对此表达式使用列别名（ annual_salary ）。

创建版本视图：示例

以下语句创建orders表的版本视图：

CREATE EDITIONING VIEW ed_orders_view（o_id，o_date，o_status）AS SELECT order_id，order_date，order_status FROM FROM WITH READ ONLY;

在管理操作（如升级）期间，您可以使用此视图将应用程序与DDL更改隔离到orders表。您可以在此视图上创建DML触发器，以便在DML操作以视图本身为目标时触发，但如果DML操作以orders表为目标，则不会触发。

创建具有约束的视图：示例

以下语句创建示例表hr.employees的受限视图，并在email视图列上定义唯一约束，并在emp_id视图列上定义视图的主键约束：

CREATE VIEW emp_sal（emp_id，last_name，email UNIQUE RELY DISABLE NOVALIDATE，CONSTRAINT id_pk PRIMARY KEY（emp_id）RELY DISABLE NOVALIDATE）AS SELECT employee_id，last_name，email FROM employees;

创建可更新视图：示例

以下语句创建一个名为clerk的可更新视图，该视图位于employees表中的所有职员。在此视图中只显示员工的ID，姓氏，部门编号和作业，并且这些列只能在员工是一种职员的行中更新：

CREATE VIEW职员AS SELECT employee_id，last_name，department_id，job_id FROM employees WHERE job_id ='PU_CLERK'或job_id ='SH_CLERK'或job_id ='ST_CLERK';

此视图允许您将采购员的job_id更改为采购经理（ PU_MAN ）：

UPDATE职员SET job_id ='PU_MAN'WHERER employee_id = 118;

下一个示例WITH CHECK OPTION创建相同的视图。如果新员工不是职员，您不能随后向clerk插入新行。您可以将员工的job_id从一种类型的职员更新为另一种类型的职员，但前面语句中的更新将失败，因为该视图无法访问具有非职员job_id员工。 

CREATE VIEW职员AS SELECT employee_id，last_name，department_id，job_id FROM employees WHERE job_id ='PU_CLERK'或job_id ='SH_CLERK'或job_id ='ST_CLERK'WITH CHECK OPTION;

创建加入视图：示例

连接视图是其视图子查询包含连接的视图。如果连接中至少有一列具有唯一索引，则可以在连接视图中修改一个基表。您可以查询USER_UPDATABLE_COLUMNS以查看连接视图中的列是否可更新。例如：

CREATE VIEW locations_view AS SELECT d.department_id，d.department_name，l.location_id，l.city FROM departments d，locations l WHERE d.location_id = l.location_id; SELECT column_name，updatable FROM user_updatable_columns WHERE table_name ='LOCATIONS_VIEW'ORDER BY column_name，updatable; COLUMN_NAME UPD ------------------------------ --- DEPARTMENT_ID YES DEPARTMENT_NAME YES LOCATION_ID NO NO NO

在前面的示例中， locations表的location_id列上的主键索引在locations_view视图中不是唯一的。因此， locations不是密钥保留的表，并且该基表中的列不可更新。 

INSERT INTO locations_view VALUES（999，'Entertainment'，87，'Roma'）; INSERT INTO locations_view第1行的VALUES * ERROR：ORA-01776：无法通过连接视图修改多个基表

您可以从departments基表中插入，更新或删除行，因为映射到departments表的视图中的所有列都标记为可更新，因为departments的主键保留在视图中。 

INSERT INTO locations_view（department_id，department_name）VALUES（999，'Entertainment'）;已创建1行。

注意：

要使用视图插入表，视图必须包含连接中所有表的所有NOT NULL列，除非您为NOT NULL列指定了DEFAULT值。
