31分布式数据库概念

与分布式数据库相关的概念包括分布式数据库架构，数据库链接，事务处理，应用程序开发和字符集支持。

  • 分布式数据库架构
    分布式数据库系统允许应用程序从本地和远程数据库访问数据。在同构的分布式数据库系统中 ，每个数据库都是Oracle数据库。在异构分布式数据库系统中 ，至少有一个数据库不是Oracle数据库。分布式数据库使用客户端/服务器架构来处理信息请求。
  • 数据库链接
    分布式数据库系统的核心概念是数据库链接 。数据库链接是两个物理数据库服务器之间的连接，允许客户端将它们作为一个逻辑数据库进行访问。
  • 分布式数据库管理
    分布式数据库管理包括与站点自治，安全性，审计数据库链接和管理工具相关的主题。
  • 分布式系统中的事务处理
    事务是由单个用户执行的一个或多个SQL语句构成的逻辑工作单元。事务以用户的第一个可执行SQL语句开始，并在该用户提交或回滚时结束。远程事务仅包含访问单个远程节点的语句。分布式事务包含访问多个节点的语句。
  • 分布式数据库应用开发
    分布式系统中的应用程序开发引发了在非分布式系统中不适用的问题。
  • 分布式环境的字符集支持
    不同的数据库和客户端可以在分布式环境中使用不同的字符集。

31.1.1.2分布式数据库与分布式处理

术语分布式数据库和分布式处理密切相关，但具有不同的含义。

定义如下：

  • 分布数据库

    分布式系统中的一组数据库，可以将应用程序显示为单个数据源。

  • 分布式处理

    应用程序在网络中的不同计算机之间分配其任务时发生的操作。例如，数据库应用程序通常将前端表示任务分发给客户端计算机，并允许后端数据库服务器管理对数据库的共享访问。因此，分布式数据库应用程序处理系统更通常被称为客户端/服务器数据库应用程序系统。

分布式数据库系统采用分布式处理架构。例如，Oracle数据库服务器在请求另一个Oracle数据库服务器管理的数据时充当客户端。

31.1.1.3分布式数据库与复制数据库

术语分布式数据库系统和数据库复制是相关的，但却截然不同。

在纯 （即未复制）分布式数据库中，系统管理所有数据和支持数据库对象的单个副本。通常，分布式数据库应用程序使用分布式事务来访问本地和远程数据，并实时修改全局数据库。

注意：

本书仅讨论纯分布式数据库。

术语复制是指在属于分布式系统的多个数据库中复制和维护数据库对象的操作。虽然复制依赖于分布式数据库技术，但数据库复制提供了纯粹的分布式数据库环境中无法实现的应用程序优势。

最常见的是，复制用于提高本地数据库性能并保护应用程序的可用性，因为存在备用数据访问选项。例如，应用程序通常可以访问本地数据库而不是远程服务器，以最小化网络流量并实现最大性能。此外，如果本地服务器出现故障，应用程序可以继续运行，但其他具有复制数据的服务器仍可访问。

    数据库链接的巨大优势在于它们允许用户访问远程数据库中的另一个用户的对象，以便它们受到对象所有者的权限集的限制。换句话说，本地用户可以访问远程数据库的链接，而不必是远程数据库上的用户。

31.2.1什么是数据库链接？

数据库链接是一个指针，用于定义从Oracle数据库服务器到另一个数据库服务器的单向通信路径。

对于公共和私有数据库链接，链接指针实际上被定义为数据字典表中的条目。要访问该链接，您必须连接到包含数据字典条目的本地数据库。对于全局数据库链接，链接指针在目录服务中定义。不同类型的数据库链接中有更详细的“描述的数据库链接类型 ”。

31.2.3为什么使用数据库链接？

数据库链接的巨大优势在于它们允许用户访问远程数据库中的另一个用户的对象，以便它们受到对象所有者的权限集的限制。换句话说，本地用户可以访问远程数据库的链接，而不必是远程数据库上的用户。

例如，假设员工将费用报表提交到应付帐款（A / P），并进一步假设使用A / P应用程序的用户必须从hq数据库中检索有关员工的信息。A / P用户应该能够连接到hq数据库并在远程hq数据库中执行存储过程，以检索所需的信息。A / P用户不应该是hq数据库用户来完成他们的工作;它们应该只能以受控方式访问hq信息，受程序限制。

31.2.8.2连接的用户数据库链接

连接的用户链接没有与之关联的连接字符串。连接用户链接的优点是引用该链接的用户作为同一用户连接到远程数据库，并且凭证不必存储在数据字典中的链接定义中。

连接的用户链接有一些缺点。由于这些链接要求用户在尝试连接的远程数据库上拥有帐户和权限，因此需要为管理员提供更多权限管理。此外，为用户提供超出他们需要的权限会违反最低权限的基本安全概念：用户应该只获得执行工作所需的权限。

使用连接的用户数据库链接的能力取决于几个因素，其中主要是用户是使用密码通过数据库进行身份验证，还是通过操作系统或网络身份验证服务进行外部身份验证。如果用户经过外部身份验证，则使用连接的用户链接的能力还取决于远程数据库是否接受用户的远程身份验证，这是由REMOTE_OS_AUTHENT初始化参数设置的。

REMOTE_OS_AUTHENT参数的操作如下：

REMOTE_OS_AUTHENT值 后果

TRUE为远程数据库

经过外部验证的用户可以使用连接的用户数据库链接连接到远程数据库。

远程数据库为FALSE

除非使用安全协议或网络身份验证服务选项，否则外部身份验证的用户无法使用连接的用户数据库链接连接到远程数据库。

如果从定义者的权限函数，过程或包中访问连接的用户数据库链接，则使用定义者的授权标识作为远程用户进行连接。例如，如果用户jane调用过程scott.p （由scott创建的定义者权限过程），并且该过程出现在过程scott.p ，则scott是连接用户。要运行包含连接的用户数据库链接的定义者权限函数，过程或包，必须授予调用函数，过程或包的用户INHERIT REMOTE PRIVILEGES权限。

31.2.8.4当前用户数据库链接

当前用户数据库链接使用全局用户。全局用户必须通过X.509证书或密码进行身份验证，并且是链接中涉及的两个数据库的用户。

调用CURRENT_USER链接的用户不必是全局用户。例如，如果通过密码对Accounts Payable数据库对jane进行身份验证（而不是全局用户），则可以访问存储过程以从hq数据库中检索数据。该过程使用当前用户数据库链接，该链接将她作为全局用户scott连接到hq 。用户scott是一个全局用户，通过SSL证书进行身份验证，但jane不是。

请注意，当前用户数据库链接具有以下后果：

  • 如果未从存储对象内访问当前用户数据库链接，则当前用户与访问该链接的连接用户相同。例如，如果scott通过当前用户链接发出SELECT语句，则当前用户为scott 。

  • 当执行存储对象（例如访问数据库链接的过程，视图或触发器）时，当前用户是拥有存储对象的用户，而不是调用该对象的用户。例如，如果jane调用过程scott.p （由scott创建），并且当前用户链接出现在被调用过程中，则scott是该链接的当前用户。

  • 如果存储的对象是调用者的权限函数，过程或包，那么调用者的授权ID用于作为远程用户进行连接。例如，如果用户jane调用过程scott.p （由scott创建的调用者权限过程），并且该过程出现在过程scott.p ，则jane是当前用户。

  • 您无法以企业用户身份连接到数据库，然后在共享的全局架构中存在的存储过程中使用当前用户链接。例如，如果用户jane访问数据库hq上的共享模式guest的存储过程，则她无法使用此模式中的当前用户链接登录到远程数据库。

SELECT * FROM emp @ foo;

  • 使用数据库链接命名模式对象

  • 访问远程模式对象的授权
    要访问远程模式对象，必须授予您对远程数据库中远程对象的访问权限。
  • Schema Objects的同义词

  • 架构对象名称解析

31.2.10.1使用数据库链接命名模式对象

Oracle数据库使用全局数据库名称来全局命名模式对象。

全局数据库名称采用以下形式：

schema.schema_object @ global_database_name

哪里：

  • schema是数据或模式对象的逻辑结构的集合。模式由数据库用户拥有，并且与该用户具有相同的名称。每个用户拥有一个架构。

  • schema_object是一个逻辑数据结构，如表，索引，视图，同义词，过程，包或数据库链接。

  • global_database_name是唯一标识远程数据库的名称。此名称必须与远程数据库初始化参数DB_NAME和DB_DOMAIN的串联相同，除非参数GLOBAL_NAMES设置为FALSE ，在这种情况下，任何名称都是可接受的。

例如，使用数据库链接到数据库sales.division3.example.com ，用户或应用程序可以引用远程数据，如下所示：

SELECT * FROM scott.emp@sales.division3.example.com; scott模式中的#emp表SELECT loc来自scott.dept@sales.division3.example.com;

如果GLOBAL_NAMES设置为FALSE ，则可以使用sales.division3.example.com链接的任何名称。例如，您可以调用链接foo 。然后，您可以按如下方式访问远程数据库：

SELECT name FROM scott.emp@foo; #link name与全局名称不同

31.2.10.2访问远程模式对象的授权

要解析对架构对象的应用程序引用（称为名称解析的过程），数据库将按层次结构形成对象名称。

例如，数据库保证数据库中的每个模式都具有唯一的名称，并且在模式中每个对象都具有唯一的名称。因此，架构对象名称在数据库中始终是唯一的。此外，数据库解析对该对象的本地名称的应用程序引用。

在分布式数据库中，系统中的所有应用程序都可以访问诸如表之类的模式对象。数据库使用全局数据库名称扩展分层命名模型，以有效地创建全局对象名称并解析对分布式数据库系统中的模式对象的引用。例如，查询可以通过指定其完全限定名称（包括它所在的数据库）来引用远程表。

例如，假设您以用户SYSTEM身份连接到本地数据库：

CONNECT SYSTEM @ sales1

然后，使用数据库链接hq.example.com发出以下语句，以访问远程数据库hq上的scott和jane模式中的对象：

SELECT * FROM scott.emp@hq.example.com; INSERT INTO jane.accounts@hq.example.com（acc_no，acc_name，balance）VALUES（5001，'BOWER'，2000）;更新jane.accounts@hq.example.com设置余额=余额+ 500; DELETE FROM jane.accounts@hq.example.com WHERE acc_name ='BOWER';

注意：

您可以审核全局用户的全局用户名。

您无法在远程对象上设置本地审核选项。因此，您无法审核数据库链接的使用，但可以在远程数据库上审核对远程对象的访问。

31.4分布式系统中的事务处理

事务是由单个用户执行的一个或多个SQL语句构成的逻辑工作单元。事务以用户的第一个可执行SQL语句开始，并在该用户提交或回滚时结束。远程事务仅包含访问单个远程节点的语句。分布式事务包含访问多个节点的语句。

  • 远程SQL语句
    远程SQL语句查询或修改一个或多个远程表，所有远程表都驻留在同一远程节点上。
  • 分布式SQL语句
    分布式SQL语句在两个或多个节点上查询或修改数据。

  • 远程交易
    远程事务包含一个或多个远程语句，所有远程语句都引用单个远程节点。
  • 分布式事务

31.4.1远程SQL语句

远程SQL语句查询或修改一个或多个远程表，所有远程表都驻留在同一远程节点上。

远程查询语句是从一个或多个远程表中选择信息的查询，所有远程表都驻留在同一远程节点上。例如，以下查询从远程sales数据库的scott模式中的dept表访问数据：

SELECT * FROM scott.dept@sales.us.americas.example_auto.com;

远程更新语句是修改一个或多个表中的数据的更新，所有表都位于同一远程节点。例如，以下查询更新远程sales数据库的scott模式中的dept表：

更新scott.dept@mktng.us.americas.example_auto.com SET loc ='NEW YORK'THERE deptno = 10;

注意：

远程更新可以包括从一个或多个远程节点检索数据的子查询，但由于更新仅在单个远程节点上发生，因此该语句被归类为远程更新。

父主题： 分布式系统中的事务处理

31.4.2分布式SQL语句

分布式SQL语句在两个或多个节点上查询或修改数据。

分布式查询语句从两个或多个节点检索信息。例如，以下查询访问本地数据库以及远程sales数据库中的数据：

SELECT ename，dname FROM scott.emp e，scott.dept@sales.us.americas.example_auto.com d WHERE e.deptno = d.deptno;

分布式更新语句修改两个或多个节点上的数据。使用PL / SQL子程序单元（例如包括两个或多个访问不同节点上的数据的远程更新的过程或触发器）可以进行分布式更新。例如，以下PL / SQL程序单元更新本地数据库和远程sales数据库上的表：

BEGIN UPDATE scott.dept@sales.us.americas.example_auto.com SET loc ='NEW YORK'THERE deptno = 10; UPDATE scott.emp SET deptno = 11 WHERE deptno = 10;结束;承诺;

数据库将程序中的语句发送到远程节点，并且它们作为一个单元执行成功或失败。

31.4.6两阶段提交机制

数据库两阶段提交机制保证参与分布式事务的所有数据库服务器都提交或全部回滚事务中的语句。

数据库必须保证事务中的所有语句（分布式或非分布式）作为一个单元提交或回滚。正在进行的事务的影响应该对所有节点上的所有其他事务都是不可见的;对于包含任何类型的操作（包括查询，更新或远程过程调用）的事务，此透明度应该为true。

Oracle数据库概念中讨论了非分布式数据库中事务控制的一般机制。在分布式数据库中，数据库必须通过网络协调具有相同特征的事务控制，并保持数据一致性，即使发生网络或系统故障。

两阶段提交机制还保护由完整性约束，远程过程调用和触发器执行的隐式DML操作。

31.5.1.1位置透明度

Oracle数据库分布式数据库系统具有允许应用程序开发人员和管理员从应用程序和用户隐藏数据库对象的物理位置的功能。

当用户可以普遍引用数据库对象（如表）时，无论应用程序连接到哪个节点，都存在位置透明性 。位置透明度有几个好处，包括：

  • 访问远程数据很简单，因为数据库用户不需要知道数据库对象的物理位置。

  • 管理员可以移动数据库对象，而不会影响最终用户或现有数据库应用程序。

通常，管理员和开发人员使用同义词来为表和支持应用程序模式中的对象建立位置透明性。例如，以下语句在数据库中为另一个远程数据库中的表创建同义词。

CREATE PUBLIC SYNONYM emp FOR scott.emp@sales.us.americas.example_auto.com;为scott.dept@sales.us.americas.example_auto.com创建公共SYNONYM部门;

现在，而不是使用如下查询访问远程表：

SELECT ename，dname FROM scott.emp@sales.us.americas.example_auto.com e，scott.dept@sales.us.americas.example_auto.com d WHERE e.deptno = d.deptno;

应用程序可以发出更简单的查询，而不必考虑远程表的位置。

SELECT ename，dname FROM emp e，dept d WHERE e.deptno = d.deptno;

除了同义词之外，开发人员还可以使用视图和存储过程为在分布式数据库系统中工作的应用程序建立位置透明性。

31.5.1.2 SQL和COMMIT透明度

Oracle数据库分布式数据库体系结构提供查询，更新和事务透明性。

例如，标准SQL语句（如SELECT ， INSERT ， UPDATE和DELETE工作方式与在非分布式数据库环境中的工作方式相同。此外，应用程序使用标准SQL语句COMMIT ， SAVEPOINT和ROLLBACK控制事务。不需要复杂的编程或其他特殊操作来提供分布式事务控制。

  • 单个事务中的语句可以引用任意数量的本地或远程表。

  • 数据库保证分布式事务中涉及的所有节点采取相同的操作：它们全部提交或全部回滚事务。

  • 如果在提交分布式事务期间发生网络或系统故障，则事务将自动且透明地全局解析。具体而言，当网络或系统恢复时，节点全部提交或全部回滚事务。

在数据库内部，每个已提交的事务都有一个关联的系统更改号 （ SCN ），用于唯一标识该事务中语句所做的更改。在分布式数据库中，通信节点的SCN在以下情况下进行协调：

  • 使用由一个或多个数据库链接描述的路径建立连接。

  • 执行分布式SQL语句。

  • 提交分布式事务。

除了其他好处之外，SCN在分布式数据库系统的节点之间的协调允许在语句和事务级别上的全局分布式读取一致性。如有必要，还可以完成全局分布式基于时间的恢复。

31.5.2 PL / SQL和远程过程调用（RPC）

开发人员可以编写PL / SQL包和过程来支持使用分布式数据库的应用程序。应用程序可以使本地过程调用在本地数据库和远程过程调用 （ RPC ）上执行工作，以在远程数据库上执行工作。

当程序调用远程过程时，本地服务器将所有过程参数传递给调用中的远程服务器。例如，以下PL / SQL程序单元调用位于远程sales数据库的打包过程del_emp ，并将参数1257传递给它：

BEGIN emp_mgmt.del_emp@sales.us.americas.example_auto.com（1257）;结束;

为了使RPC成功，被调用的过程必须存在于远程站点，并且连接的用户必须具有执行该过程的适当权限。

在为分布式数据库系统开发包和程序时，开发人员必须编写代码，了解远程位置应该执行哪些程序单元，以及如何将结果返回给调用应用程序。

31.5.3分布式查询优化

分布式查询优化是Oracle数据库功能，当事务从分布式SQL语句中引用的远程表检索数据时，可以减少站点之间所需的数据传输量。

分布式查询优化使用基于成本的优化来查找或生成SQL表达式，这些表达式仅从远程表中提取必要的数据，在远程站点或有时在本地站点处理该数据，并将结果发送到本地站点进行最终处理。与将所有表数据传输到本地站点进行处理所花费的时间相比，此操作减少了所需的数据传输量。

使用各种基于成本的优化器提示（如DRIVING_SITE ， NO_MERGE和INDEX ，您可以控制Oracle数据库处理数据的位置以及数据访问方式。
