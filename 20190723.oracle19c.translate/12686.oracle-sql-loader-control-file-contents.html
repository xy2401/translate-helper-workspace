9.1控制文件内容

SQL * Loader控制文件是一个包含数据定义语言（DDL）指令的文本文件。

DDL用于控制SQL * Loader会话的以下方面：

  • SQL * Loader将在哪里找到要加载的数据

  • SQL * Loader如何期望格式化数据

  • 如何在加载数据时配置SQL * Loader（内存管理，拒绝记录，中断负载处理等）

  • SQL * Loader如何操作正在加载的数据

有关SQL * Loader DDL的语法图，请参阅SQL * Loader语法图 。

要创建SQL * Loader控制文件，请使用文本编辑器（如vi或xemacs）。

通常，控制文件有三个主要部分，顺序如下：

  • 会话范围的信息

  • 表和字段列表信息

  • 输入数据（可选部分）

9.3.3.5有时不允许逃脱角色

在您的操作系统上运行的Oracle数据库的发布可能无法实现非可移植字符串的转义字符。

当不允许转义字符时，反斜杠被视为普通字符，而不是转义字符（尽管它仍可用于所有其他字符串）。然后可以正常指定以下路径名：

INFILE'topdir \ mydir \ myfile'

不需要双反斜杠。

由于反斜杠不会被识别为转义字符，因此单引号内的字符串不能嵌入由单引号分隔的另一个字符串中。此规则也适用于双引号。双引号内的字符串不能嵌入由双引号分隔的另一个字符串中。

9.5指定字段顺序

您可以使用SQL * Loader控制文件中的FIELD NAMES子句指定字段顺序。

语法如下：

字段名称{FIRST FILE | FIRST FILE IGNORE | ALL FILES | ALL FILES IGNORE | NONE}

FIELD NAMES选项包括：

  • FIRST FILE ：表示第一个数据文件包含第一个记录中数据的字段名称列表。此列表使用与数据文件中的数据相同的分隔符。读取记录以设置数据文件中的字段与目标表中的列之间的映射。处理数据时会跳过该记录。如果数据文件中的字段顺序与表中列的顺序不同，或者数据文件中的字段数与目标表中的列数不同，则此选项非常有用

  • FIRST FILE IGNORE ：表示第一个数据文件包含第一个记录中数据的字段名称列表，但应忽略该信息。处理数据时将跳过该记录，但不会将其用于设置字段。

  • ALL FILES ：表示所有数据文件都包含第一条记录中数据的字段名称列表。处理数据时，每个数据文件中都会跳过第一条记录。每个数据文件中的字段可以采用不同的顺序。SQL * Loader根据每个数据文件中字段的顺序设置负载。

  • ALL FILES IGNORE ：表示所有数据文件都包含第一条记录中数据的字段名称列表，但应忽略该信息。在每个数据文件中处理数据时会跳过该记录，但不会将其用于设置字段。

  • NONE ：表示数据文件包含第一条记录中的普通数据。这是默认值。

FIELD NAMES子句不能用于复杂的列类型，例如列对象，嵌套表或VARRAY。

9.7指定CSV格式文件

要指示SQL * Loader以逗号分隔值格式文件的形式访问数据文件，请使用CSV子句。

FIELDS CSV [嵌入式|无嵌入式] [字段终止于'，'] [选项封装'''']

以下是有关FIELDS CSV子句的要点：

  • SQL * Loader的默认设置是不使用FIELDS CSV子句。

  • WITH EMBEDDED和WITHOUT EMBEDDED选项指定是否在数据的任何字段中包含（嵌入）记录终止符。

9.10.3拒绝记录的标准

可以拒绝记录，原因如下：

  1. 插入后，记录会导致Oracle错误（例如给定数据类型的无效数据）。

  2. 记录格式不正确，因此SQL * Loader无法找到字段边界。

  3. 记录违反约束或尝试使唯一索引非唯一。

如果可以根据WHEN子句标准（即使使用不平衡分隔符）评估数据，则可以插入或拒绝数据。

如果任何一个表违反了上一个列表中的原因编号1或3，则传统路径加载不会向任何表写入行。该表被拒绝并写入拒绝文件。

在传统的路径加载中，如果数据文件具有正被加载到多个表中的记录并且该记录被从至少一个表中拒绝，则该记录不被加载到任何表中。

    UTF-16 Unicode编码是固定宽度的多字节编码，其中字符代码0x0000到0x007F与单字节ASCII代码0x00到0x7F具有相同的含义。

    UTF-8 Unicode编码是可变宽度多字节编码，其中字符代码0x00到0x7F具有与ASCII相同的含义。 UTF-8中的字符可以是1字节，2字节或3字节长。

  • Oracle建议使用AL32UTF8作为数据库字符集。AL32UTF8是Unicode编码UTF-8的正确实现。从Oracle Database 12 c第2版开始，使用Oracle Universal Installer（OUI）和Oracle Database Configuration Assistant（DBCA）创建数据库时，AL32UTF8用作默认数据库字符集。

  • 不要使用UTF8作为数据库字符集，因为它不是Unicode编码UTF-8的正确实现。如果在预期使用UTF-8处理的情况下使用UTF8字符集，则可能会发生数据丢失和安全问题。对于Web相关数据（例如XML和URL地址）尤其如此。

  • AL32UTF8和UTF8字符集彼此不兼容，因为它们具有不同的最大字符宽度（每个字符四个对三个字节）。

9.14.5.3控制文件字符集

假定SQL * Loader控制文件本身位于由NLS_LANG参数为会话指定的字符集中。

9.15.2.1因空间错误而中断负载

  • 将数据加载到多个子分区时（即，加载到分区表，复合分区表或复合分区表的一个分区）时出现空间错误：

    如果在加载到多个子分区时发生空间错误，则停止加载并且除非已指定ROWS否则不会保存任何数据（在这种情况下，将保存先前提交的所有数据）。此行为的原因是可能无法按顺序加载行。这是因为每个行都被分配（不一定按顺序）到分区，并且每个分区都是单独加载的。如果在加载分配给分区的所有行之前中断加载，则可能已加载记录“n”的行，但不加载记录“n-1”的行。因此，仅通过使用SKIP=N就不能继续负载。

  • 将数据加载到未分区表，分区表的一个分区或复合分区表的一个子分区时出现空间错误：

    如果控制文件中有一个INTO TABLE语句，则SQL * Loader将提交与错误发生之前加载的行数相同的行。

    如果控制文件中有多个INTO TABLE语句，则SQL * Loader将已从数据文件读取的数据加载到其他表中，然后提交数据。

    在任何一种情况下，此行为都与是否指定了ROWS参数无关。继续加载时，可以使用SKIP参数跳过已加载的行。对于多个INTO TABLE语句，可能已将不同数量的行加载到每个表中，因此要继续加载，您需要为每个表指定SKIP参数的不同值。SQL * Loader仅报告SKIP参数的值，如果它对所有表都相同。

SINGLEROW选项适用于在内存有限的系统上使用APPEND进行直接路径加载时，或者在将大量记录加载到大表中时使用。此选项将每个索引条目直接插入索引，一次一条记录。

使用SINGLEROW选项，新索引条目或新索引不需要存储空间。结果索引可能不如新分类的索引那么理想，但生成的空间较小。它还需要更多时间，因为为每个索引插入生成了额外的UNDO信息。建议在存在以下任一情况时使用此选项：

  • 可用存储空间有限。

  • 与表格的大小相比，要加载的记录数量较少（建议使用1:20或更低的比率）。

9.20.3指定行数与绑定数组的大小

使用命令行参数BINDSIZE或控制文件中的OPTIONS子句指定绑定数组大小时，将对绑定数组施加上限。

绑定数组永远不会超过该最大值。

作为其初始化的一部分，SQL * Loader确定加载单行所需的大小（以字节为单位）。如果该大小太大而不能达到指定的最大值，则加载将以错误终止。

然后，SQL * Loader将该大小乘以加载的行数，无论该值是使用命令行参数ROWS还是控制文件中的OPTIONS子句指定的。

如果该大小适合绑定数组最大值，则加载继续 - SQL * Loader不会尝试扩展行数以达到最大绑定数组大小。如果同时指定了行数和最大绑定数组大小，则SQL * Loader始终使用较小的值作为绑定数组。

如果最大绑定数组大小太小而无法容纳初始行数，则SQL * Loader使用的最小行数最大。

9.20.4确定绑定数组大小的计算

绑定数组的大小等于它包含的行数乘以每行的最大长度。

行的最大长度等于最大字段长度加上开销的总和，如下所示：

绑定数组大小=（行数）*（SUM（固定字段长度）+ SUM（最大变化字段长度）+（（变长字段数）*（长度指示符的大小）））

许多字段的大小不同。这些固定长度字段对于每个加载的行是相同的。对于这些字段，字段的最大长度是字段大小（以字节为单位），如SQL * Loader数据类型中所述 。这些字段没有开销。

可以从行到行大小不同的字段有：

  • CHAR

  • 日期

  • INTERVAL DAY TO SECOND

  • INTERVAL DAY TO YEAR

  • LONG VARRAW

  • 数字EXTERNAL

  • 时间

  • TIMESTAMP

  • TIME WITH TIME ZONE

  • TIMESTAMP WITH TIME ZONE

  • VARCHAR

  • VARCHARC

  • VARGRAPHIC

  • VARRAW

  • VARRAWC

SQL * Loader数据类型中描述了这些数据类型的最大长度。最大长度描述了字段在输入数据记录中可以占用的字节数。该长度还描述了每个字段在绑定数组中占用的存储量，但绑定数组包含可以在大小上变化的字段的额外开销。

使用分隔符指定字符数据类型（ CHAR ， DATE和数字EXTERNAL ）时，为这些字段指定的任何长度都是最大长度。如果指定时没有分隔符，则记录中的大小是固定的，但由于空白修剪，插入字段的大小可能仍会有所不同。因此在内部，这些数据类型始终被视为变长字段 - 即使它们是固定长度字段。

使用字节长度语义，此示例在绑定数组中使用（10 + 2）* 64 = 768个字节，假设长度指示符长度为2个字节，并且一次加载64个行。

对于字符长度语义，同一示例在绑定数组中使用（（10 * s）+ 2）* 64字节，其中“s”是数据文件字符集中字符的最大字节数。

现在考虑以下示例：

无论是使用字节长度语义还是字符长度语义，此示例使用（255 + 2）* 64 = 16,448字节，因为分隔字段的默认最大大小为255个字节。这可以使适合绑定数组的行数有很大差异。

9.20.6计算多个INTO表子句的绑定数组大小

计算具有多个INTO TABLE子句的控制文件的绑定数组大小时，计算好像INTO TABLE子句不存在一样。

想象一下，控制文件中列出的所有字段都是一个长数据结构 - 即绑定数组中单行的格式。

如果在多个INTO TABLE子句中提到了数据记录中的相同字段，则每次提到时都需要绑定数组中的额外空间。最小化这些字段的缓冲区分配尤为重要。

注意：

生成的数据由SQL * Loader函数CONSTANT ， EXPRESSION ， RECNUM ， SYSDATE和SEQUENCE 。这样生成的数据不需要绑定数组中的任何空间。
