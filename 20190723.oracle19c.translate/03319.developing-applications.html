应用程序的网关外观

一个为访问DRDA数据库中的信息而编写的应用程序 Oracle数据库。开发应用程序时，请记住以下信息：

  • 必须使用Oracle数据库中定义的数据库链接将DRDA数据库定义到应用程序。您的应用程序应使用数据库链接中定义的名称指定DRDA数据库中存在的表。例如，假设定义了数据库链接，以便为DRDA数据库链接DRDA命名，并假设应用程序需要从Oracle数据库和DRDA数据库中检索数据。使用以下SQL语句将两个表连接在一起：

    选择EMPNO，来自EMP L的SALAL，EMPS @ DRDA R WHERE L.EMPNO = R.EMPNO

    在这个例子中， EMP 是Oracle数据库上的表， EMPS是DRDA服务器上的表。您还可以在DRDA服务器表上定义同义词或视图，并访问不带数据库链接后缀的信息。

  • 您可以读取和写入定义的DRDA数据库的数据。SELECT ， INSERT ， UPDATE和DELETE都是有效的操作。

  • 单个事务可以写入一个DRDA数据库和多个Oracle数据库。

  • 单使用SQL语句 JOIN ，可以引用多个Oracle数据库中的表，多个DRDA数据库中的表，或两者中的表。

获取重新阻止

Oracle数据库支持使用HS_RPC_FETCH_REBLOCKING参数获取重新HS_RPC_FETCH_REBLOCKING 。

当此参数的值设置为ON （默认值）时， 数组大小 SELECT语句由HS_RPC_FETCH_SIZE值确定 。该 HS_RPC_FETCH_SIZE参数定义每个缓冲区从网关发送到Oracle数据库的字节数。缓冲区可能包含DRDA服务器中的一个或多个限定行。此功能可以提供重要意义性能增强，具体取决于您的应用程序设计，安装类型和工作负载。

客户端和Oracle数据库之间的数组大小由Oracle应用程序确定。请参阅POWER Systems（64位）上的IBM AIX上的Oracle Database Gateway安装和配置指南，Linux x86-64，SPARC上的Oracle Solaris（64位），x86-64（64位）上的Oracle Solaris和HP- 适用于Microsoft Windows的 UX Itanium或Oracle Database Gateway安装和配置指南 ，具体取决于您的平台，以获取更多信息。

将Oracle存储过程与网关一起使用

网关存储过程支持是Oracle存储过程的扩展。一个 Oracle存储过程是一个模式对象，它在逻辑上将一组SQL和其他PL / SQL编程语言语句组合在一起，以执行特定任务。Oracle存储过程存储在数据库中以供继续使用。应用程序使用标准Oracle PL / SQL到呼叫存储过程。

Oracle存储过程可以位于本地 Oracle数据库的实例和远程实例。图4-1说明了两个存储过程： oraproc1是存储在的一个过程 ORA1 Oracle实例， oraproc2是存储在ORA2 Oracle实例中的过程。

为oraproc2 @ ora2创建SYNONYM oraproc2;

创建此同义词后，应用程序不再需要使用数据库链接规范在中调用存储过程远程Oracle实例。

在图4-1中 ， oraproc1的第二个语句用于访问ORA2实例中的表。同样，Oracle存储过程可用于通过网关访问DB2表。

将DRDA服务器存储过程与网关一起使用

网关的过程功能允许调用本机DRDA服务器存储过程 。将存储过程定义到DRDA服务器后，网关可以使用现有DRDA服务器定义来运行该过程。网关不需要特殊定义调用DB2存储过程。标准Oracle Oracle应用程序使用PL / SQL来运行存储过程。

在图4-2中 ，一个Oracle应用程序调用为DRDA服务器定义的empproc存储过程（例如，DB2 UDB for z / OS）。

Oracle应用程序和DRDA服务器存储过程完成

对于调用DB2存储过程的Oracle应用程序，首先必须使用IBM SQL参考文档中记录的过程在DB2系统上创建DB2存储过程。

在DB2中定义存储过程后，网关可以使用标准PL / SQL访问数据呼叫 。例如，员工姓名John Smythe将传递给DB2 存储过程REVISE_SALARY 。 DB2存储过程从DB2数据库中检索工资值，以便为John Smythe计算新的年薪。修改后的薪水作为结果返回用于更新Oracle数据库的EMP表：

DECLARE INPUT VARCHAR2（15）;结果编号（8,2）; BEGIN INPUT：='JOHN SMYTHE'; REVISE_SALARY @ DB2（INPUT，RESULT）;更新EMP设置SAL =结果ENAME = INPUT;结束;

当网关收到一个调用在DRDA服务器上运行存储过程，它首先查找服务器目录中的过程名称。定义存储过程的信息以不同的形式存储在每个DRDA服务器上。例如，DB2 UDB for iSeries使用表QSYS2.SYSPROCS和QSYS2.SYSPARMS 。网关具有要搜索的已知目录列表，具体取决于正在访问的DRDA服务器。

该目录的搜索顺序是依赖于是否在目录支撑位置指示符（如LUNAME在SYSIBM.SYSPROCEDURES ），和授权或拥有者的ID（如AUTHID在SYSIBM.SYSPROCEDURES或OWNER在SYSIBM.SYSROUTINES ）。

某些DRDA服务器允许空白或公共授权限定符。如果当前连接的DRDA服务器支持这种形式的限定，则网关将在搜索目录中的过程名称时应用这些命名规则。

匹配规则将首先搜索公共定义，然后搜索所有者限定的过程名称。有关更多详细信息，请参阅DB2 SQL的IBM参考文档。

DB2的过程特性注意事项

以下是在网关中使用过程功能的特殊注意事项：

  • PL / SQL 在调用DB2存储过程时，不能将记录作为参数传递。

  • 网关支持DB2存储过程的GENERAL和DB2SQL链接转换。两种链接约定都要求传递到DB2存储过程和从DB2存储过程传递的参数不能为空。

结果集和存储过程

用于DRDA的Oracle数据库网关为返回结果集的存储过程提供支持。默认情况下，所有存储过程和函数都不会将结果集返回给用户。要启用结果集，请在初始化参数文件中设置HS_FDS_RESULTSET_SUPPORT参数。

也可以看看：

初始化有关编辑初始化参数文件和HS_FDS_RESULTSET_SUPPORT参数的HS_FDS_RESULTSET_SUPPORT参数。有关Oracle对非Oracle数据库中的结果集的支持的详细信息，请参阅“ Oracle数据库异构连接用户指南” 。

注意：

如果设置HS_FDS_RESULTSET_SUPPORT网关初始化参数，则必须更改所有现有存储过程的过程execute语句的语法，否则将发生错误。

通过Oracle Database Gateway for DRDA访问带有结果集的存储过程时，您将处于异构服务的顺序模式。在过程描述期间，网关将以下信息返回到异构服务：

  • 远程存储过程的所有输入参数

  • 没有输出参数

  • 类型引用游标的一个out参数（对应于存储过程返回的第一个结果集）

客户端程序必须使用虚拟包函数DBMS_HS_RESULT_SET.GET_NEXT_RESULT_SET来获取后续结果集的引用游标。返回的最后一个结果集是过程中的out参数。

访问结果集的限制如下：

  • 必须按照将它们放置在线路上的顺序检索远程存储过程返回的结果集。

  • 在执行存储过程时，将关闭先前执行的存储过程返回的所有结果集，而不管数据是否已被完全检索。

数据库链接行为

首次在Oracle数据库会话中使用时，通过数据库链接建立与网关的连接。在此上下文中，连接指的是Oracle数据库与网关之间的连接以及网关与目标DRDA数据库之间的DRDA网络连接。在Oracle数据库会话结束之前，连接将保持建立状态 。另一个会话或用户可以访问相同的数据库链接，并获得与网关和DRDA数据库的不同连接。

Oracle数据库SQL构造处理

Oracle Database Gateways产品最重要的功能之一是为用户和应用程序员提供SQL透明性。外部SQL构造可以分为四个区域：

  • 兼容

  • 翻译

  • 补偿

  • 本机语义

兼容的SQL函数

Oracle数据库自动将兼容的SQL函数转发到DRDA数据库，其中具有相同语法和含义的SQL构造在Oracle数据库和DRDA数据库上。这些SQL结构未经修改即转发。所有兼容的功能都是列功能 。不兼容的函数要么转换为等效的DRDA SQL函数，要么在从DRDA数据库返回数据后由Oracle数据库进行补偿（后处理）。

翻译的SQL函数

翻译的函数具有相同的含义，但Oracle数据库和DRDA数据库之间的名称不同。但是所有应用程序都必须使用Oracle函数名称。DRDA数据库使用不同语法（不同的函数名称）支持的这些SQL构造由Oracle数据库自动转换，然后转发到DRDA数据库。Oracle数据库在将函数名称发送到DRDA数据库之前更改函数名称，其方式对应用程序是透明的。

补偿的SQL函数

DRDA数据库可能不支持Oracle数据库支持的某些高级SQL结构。补偿函数是那些DRDA服务器无法识别或DRDA服务器识别的SQL函数，但在比较DRDA服务器和Oracle数据库时，函数的语义会有不同的解释。如果包含这些函数之一的SELECT语句从Oracle数据库传递到网关，则网关会在将SQL语句传递给DRDA服务器之前删除该函数。网关将选定的DRDA数据库行传递给Oracle数据库。Oracle数据库应用该功能。

后期处理

Oracle数据库可以通过从转发到DRDA数据库的SQL请求中自动排除不兼容的SQL构造来补偿缺少或不兼容的功能。然后，Oracle数据库从DRDA数据库中检索必要的数据并应用该功能。此过程称为后处理。

网关尝试将所有SQL功能传递给DRDA数据库。但是，当DRDA数据库不支持计算中表示的功能时，网关会更改该功能。例如，如果程序针对DB2 UDB for z / OS数据库运行以下查询：

从TABLE_X中选择COS（X_COOR）;

由于数据库不支持许多COS功能，因此网关会将查询更改为以下内容：

从TABLE_X中选择X_COOR;

在所有的数据X_COOR列TABLE_X从DB2 UDB通过了z / OS数据库向Oracle数据库。将数据移动到Oracle数据库之后执行COS功能。

如果要对存储在DRDA数据库中的大量数据执行操作，请记住某些功能需要进行后处理。

本机语义SQL函数

一些通常被补偿的SQL函数也可以通过Native Semantics工具覆盖。如果已为Native Semantics启用了SQL函数，则可以将该函数传递给DRDA数据库进行处理，而不是进行补偿。然后在DRDA数据库中本机处理SQL函数。有关更多信息，请参阅“本机语义 ” 。

原生语义

DRDA数据库可能不支持Oracle数据库支持的某些高级SQL构造。在这种情况下，Oracle数据库通过补偿缺少或不兼容的功能使用Oracle数据库功能对DRDA数据库数据进行后处理

也可以看看：

“ Oracle数据库SQL构造处理 ”以获取更多信息

此功能提供最大的透明度，但可能会影响性能。此外，特定DRDA数据库的新版本可能实现以前不受支持的版本功能或功能，或者他们可能会更改支持语义，使它们与Oracle数据库函数更兼容。

一些DRDA服务器还支持用户定义的功能。用户可以选择在DRDA数据库中本机实现Oracle数据库功能。这使DRDA服务器能够将该功能传递给底层数据库实现（例如，DB2）。Native Semantics提供了一种方法，可以使DRDA服务器本机处理特定功能。

本机语义考虑因素

在启用特定函数的本机语义特性时必须考虑各种因素，因为本机语义具有优点和缺点，通常是之间的权衡透明度和性能 。

  • 其中一个考虑因素是透明度数据强制。Oracle数据库提供强制（ 隐含数据转换）对很多人而言 SQL函数。这意味着如果特定函数的提供值不正确，则Oracle数据库将在处理之前强制该值（将其更改为正确的值类型）。但是，启用本机语义功能后，将完全按照提供的值传递给DRDA服务器进行处理。在许多情况下，DRDA服务器将无法将值强制转换为正确的类型，并将生成一个错误。

  • 另一个考虑因素涉及参数与特定SQL函数的兼容性。例如，Oracle数据库的实现 SUBSTR允许负值字符串索引，而SUBSTR大多数DRDA服务器实现不允许字符串索引的负值。但是，如果实现应用程序以与DRDA服务器兼容的方式调用SUBSTR ，则该函数在Oracle数据库或DRDA服务器中的行为相同。

  • 另一个考虑因素是由于该环境中的资源限制，DRDA服务器处的功能的处理可能是不期望的。

有关启用或禁用这些功能的详细信息，请参阅“ HS_FDS_CAPABILITY ” 。有关以下功能的Oracle数据库格式，请参阅Oracle数据库SQL语言参考 。

执行字符串操作

网关使用引用列的数据类型执行所有字符串比较，连接和排序，并确定应用程序使用网关传递的字符串值的有效性。网关自动转换字符串数据类型到另一个，并在需要时在字符串和日期之间转换。

DRDA数据库经常被设计用于保存字符列中的非字符二进制数据。在DRDA系统上执行的应用程序通常可以存储和检索数据，就好像它包含字符数据一样。但是，当访问此数据的应用程序在使用不同字符集的环境中运行时，可能会返回不准确的数据。

随着网关运行主机，从DB2 UDB for iSeries或DB2 UDB for z / OS主机检索的字符数据是从中转换而来的 EBCDIC到 ASCII。当字符数据从主机发送到DB2 UDB for iSeries或DB2 UDB for z / OS时，ASCII数据将转换为EBCDIC。当字符是字符列中的二进制数据时，此转换会导致应用程序接收不正确的信息或错误。要解决这些错误，必须使用以下方法创建DB2 UDB for iSeries或DB2 UDB for z / OS上包含非字符数据的字符列。 FOR BIT DATA选项。在应用程序中，应使用Oracle处理包含非字符数据的字符列数据类型RAW和LONG RAW 。该在主机上使用FOR BIT DATA定义的字符列的DESCRIBE信息始终表示RAW或LONG RAW 。

转换字符串数据类型

网关将来自主变量的字符串数据值绑定为固定长度的字符串。绑定长度是字符串数据值的长度。网关在每个绑定上执行此转换。

如果Oracle数据库配置为使用32767最大VARCHAR2大小，则DRDA VARCHAR数据类型的长度可介于1到32767字符之间。否则，限制为4000 。如果DRDA VARCHAR数据类型大于Oracle配置的VARCHAR2限制大小，则会将其转换为Oracle LONG数据类型。

DB2 VARCHAR数据类型不能超过32767字节，这比Oracle LONG数据类型的最大大小要短得多。如果定义长度大于32767字节的Oracle LONG数据类型，则会收到一个映射到DB2 VARCHAR数据类型时的错误消息。

执行图形字符串操作

DB2 GRAPHIC数据类型仅存储双字节字符串数据。DB2 GRAPHIC数据类型的大小通常具有的最大大小是其字符对应项的一半。例如， CHAR的最大大小可以是255个字符，而GRAPHIC的最大大小可以是127个字符。

Oracle数据库没有直接匹配的数据类型，因此网关在Oracle字符数据类型之间转换为DB2 Graphic数据类型。Oracle数据库字符数据类型可能包含单字节，混合字符或双字节字符数据。网关将字符串数据转换为适当的双字节格式，具体取决于目标DB2列是否为Graphic类型，以及是否设置了Gateway Initialization参数以执行此转换。有关更多配置信息，请参阅初始化参数 。

执行日期和时间操作

IBM DRDA数据库和Oracle数据库中日期和时间数据的实现差别很大。Oracle数据库只有一个日期数据类型DATE ，可以包含日历日期和时间信息。

IBM DRDA数据库支持以下三个不同的日期和时间数据类型：

DATE仅为日历日期。

TIME是一天中的时间 。

TIMESTAMP是一个在IBM DRDA数据库的内部格式中将日历日期和时间与微秒分辨率组合在一起的数值。

处理TIME和TIMESTAMP数据

没有内置机制将IBM TIME和TIMESTAMP数据转换为Oracle DATE数据 。应用程序必须处理 TIME数据类型为Oracle CHAR格式，长度为8个字节。应用程序必须处理 TIMESTAMP数据类型为Oracle CHAR格式，长度为26个字节。

应用程序将TIME和TIMESTAMP函数作为字符串读取，并转换或子集字符串的部分以执行数值运算。TIME和TIMESTAMP值可以作为字符发送到DRDA服务器文字或绑定适当长度和格式的变量。

处理日期数据

Oracle和IBM DATE 数据类型相互映射。如果查询IBM DATE ，那么它就是转换为Oracle DATE ，时间为零（午夜）。如果针对IBM DATE列处理Oracle DATE ，则日期值将转换为IBM DATE格式，并且将丢弃任何时间值。

Oracle格式和IBM DRDA格式的日期字符表示形式不同。当Oracle应用程序SQL语句包含日期时文字，或使用字符绑定变量传送日期，网关必须将日期转换为IBM DRDA兼容格式。

网关不会自动识别何时针对IBM DATE列处理字符值。应用程序需要通过将它们与Oracle封装来区分字符日期值 TO_DATE函数表示法。例如，如果EMP是访问IBM DRDA数据库上的数据的同义词或视图，则不应使用以下SQL语句：

SELECT * FROM EMP WHERE HIREDATE = '03 -MAR-81'

你应该使用以下内容：

SELECT * FROM EMP WHERE HIREDATE = TO_DATE（'03 -MAR-81'）

在使用字符绑定变量作为限定日期值的编程接口程序中，必须使用以下SQL语句：

SELECT * FROM EMP WHERE HIREDATE = TO_DATE（：1）

当对Oracle数据库表执行语句时，上述SQL表示法不会影响SQL语句语义。该声明在Oracle和IBM DRDA访问的数据存储中仍可移植。

在发送到DB2进行处理之前，将检查除插入值之外的任何日期文字以匹配Oracle NLS_DATE_FORMAT 。TG4DB2 v10.2不检查是否与NLS_DATE_FORMAT格式匹配。如果需要这种兼容性，则需要将NODATECHK/ON值指定为HS_FDS_CAPABILITY参数的一部分。然后，您可以使用任何DB2可接受的日期格式：

  • YYYY-MM-DD （ISO / JIS）

  • DD.MM.YYYY （欧洲）

  • MM/DD/YYYY （美国）

例如：

SELECT * FROM EMP WHERE HIREDATE ='1981-03-03'

TO_DATE要求也不属于输入绑定变量，采用Oracle日期7字节二进制格式。网关将这些值识别为日期。对于DB2 UDB for z / OS，如果安装了提供DATE EXIT的网关，那么您还可以使用另外两种Oracle日期格式：DD-MON-RR和DD-MON-YYYY

执行日期算术

以下形式的SQL表达式通常无法与网关一起正常运行：

日期+号码 +日期 日期 - 号码 date1  -  date2

日期和号码的加法和减法(date + number,number + date,date - number)表格将被发送到DRDA服务器，在那里它们被拒绝。支持的服务器不允许使用日期添加或减少数字。

由于支持的服务器中日期减法的解释不同，减去两个日期(date1 - date2)会产生因服务器而异的结果。

注意：

避免在所有网关SQL中使用日期算术表达式，直到解决日期算术问题。

日期

日期处理有两类：

  • 两位数年份日期，视为在2000年之前或50年之后发生的。

  • 四位数的年份日期，对于2000年来说并不含糊。

使用以下方法之一输入二十一世纪日期：

  • TO_DATE函数

    使用任何日期格式，包括四个字符的年份字段。有关可用的日期格式字符串选项，请参阅Oracle数据库SQL语言参考 。

    例如， TO_DATE （' 2008-07-23', 'YYYY-MM-DD '）可用于任何SELECT ， INSERT ， UPDATE或DELETE语句。

  • NLS_DATE_FORMAT参数

    ALTER SESSION SET NLS_DATE_FORMAT应该用于设置SQL中使用的日期格式。

Oracle TO_DATE函数

Oracle TO_DATE函数在SQL INSERT ， UPDATE ， DELETE和SELECT WHERE子句中进行预处理。SELECT结果列表中的TO_DATE函数未经过预处理。

通常需要TO_DATE函数来提供要更新或比较的值日期列。因此，在将SQL语句发送到DB2之前，网关会使用可接受的值替换TO_DATE子句中包含的信息。

除SELECT结果列表外，所有TO_DATE函数都经过预处理，并转换为TO_DATE函数结果的值。仅允许TO_DATE （ literal ）或TO_DATE (:bind_variable) 。除SELECT结果列表外，不支持TO_DATE (column_name)函数格式。

将Oracle TO_DATE函数预处理为简单值在INSERT VALUES子句中很有用，因为DB2不允许在 VALUES条款 。在这种情况下，DB2在VALUES列表中接收一个简单值。支持所有形式的TO_DATE函数（带有一个，两个或三个操作数）。

执行数值数据类型操作

IBM版本的DRDA服务器自动执行转换为目标列的数值数据类型（例如整数，双精度浮点或十进制）。用户无法控制数据类型转换，此转换可以独立于数据库中目标列的数据类型。

例如，如果PRICE是IBM DRDA数据库中PRODUCT表的整数列，则以下示例中显示的更新不准确地将冰淇淋的价格设置为1.00美元，因为IBM DRDA服务器会自动生成将浮点数转换为整数：

更新产品设定价格= 1.50 PRODUCT_NAME ='ICE CREAM CONE';

由于PRICE是整数，因此IBM DRDA服务器会自动将1.50的十进制数据值转换为1。

映射COUNT函数

Oracle数据库支持以下四个操作数 COUNT 功能：

  • 计数（*）

  • COUNT（DISTINCT colname）

  • COUNT（所有colname）

  • COUNT（colname的）

某些DRDA服务器不支持所有形式的COUNT ，特别是COUNT(colname)和COUNT(ALL colname) 。在这些情况下， COUNT函数及其参数被转换为COUNT （*）。这可能无法产生所需的结果，尤其是在计数的列包含NULL的情况下。

对于那些不支持上述表单的DRDA服务器，可以通过添加WHERE子句来实现等效功能。 例如，

SELECT COUNT （colname） FROM table @ dblink WHERE colname IS NOT NULL

要么

SELECT COUNT（ALL colname ）FROM table @ dblink WHERE colname IS NOT NULL

您还可以使用本机语义在网关初始化文件中使用以下参数指示对所有COUNT函数的支持：

HS_FDS_CAPABILITY =（COUNTCOL = YES）

有关不支持所有形式的COUNT已知DRDA服务器，请参阅SQL限制 。

执行分区十进制操作

分区十进制字段描述为 Oracle数据库上的压缩十进制数但是，诸如Pro * C程序之类的Oracle应用程序可以使用任何受支持的Oracle插入分区十进制列数字数据类型。网关将此数字转换为最合适的数据类型。可以将数据从DRDA数据库提取到任何Oracle数据类型，前提是它不会导致信息丢失。

通过网关传递本机SQL语句

直通SQL功能使应用程序开发人员能够将SQL语句直接发送到DRDA服务器，而不会由Oracle数据库解释该语句。 DBMS_HS_PASSTHROUGH 。网关支持的EXECUTE_IMMEDIATE SQL直通语句仅限于非查询（ INSERT ， UPDATE ， DELETE ，和 DDL语句）并不能包含捆绑变量。网关可以使用DBMS_HS_PASSTHROUGH运行本机SQL语句。 EXECUTE_IMMEDIATE 。

DBMS_HS_PASSTHROUGH 。 EXECUTE_IMMEDIATE是内置网关功能。此函数接收一个输入参数，并返回受SQL语句影响的行数。对于数据定义语言（DDL）语句，该函数返回零。

DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE是网关的保留名称，专门用于运行本机SQL。

Oracle Database Gateway for DRDA的12 c版本2（12.2）允许从通过passthrough发出的查询中检索结果集。语法与DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE函数不同。有关详细信息，请参阅“ 通过直通检索结果集 ” 。

通过Passthrough处理DDL语句

如上所述，Oracle数据库不会解释通过DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE函数处理的SQL语句。因此，Oracle数据库将不知道此类语句是否对DRDA服务器进行了任何修改。这意味着除非您在更改DRDA服务器后保持Oracle数据库的缓存信息是最新的，否则数据库可能会继续依赖于同一会话中后续查询中的不准确或过时信息。

当您通过添加或删除列来更改表的结构时，会出现这种情况。当应用程序通过网关引用表时（例如，对其执行查询时），Oracle数据库会缓存表定义。现在，假设在同一会话中，应用程序随后通过使用DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE添加列来更改表的形式。然后，应用程序对表的下一次引用将返回表的旧列定义，并将忽略表的新列。这是因为Oracle数据库没有处理语句，因此不知道更改。因为数据库不知道更改，所以它没有理由重新查询表格形式，因此，它将使用已经缓存的表单来处理任何新的查询。

为了使Oracle数据库获取表的新形式，必须关闭与网关的现有会话，并且必须打开新会话。这可以通过以下两种方式之一完成：

  • 通过结束与Oracle数据库的应用程序会话并在对DRDA服务器进行修改后启动新会话;要么

  • 通过在对DRDA服务器进行任何修改后运行ALTER SESSION CLOSE DATABASE LINK命令。

上述任一操作都将使缓存的表定义无效，并将强制Oracle数据库在下一个引用上获取新定义。

使用DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE

要使用DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE运行直通SQL语句，请使用以下语法： 

number_of_rows = DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE@dblink（'native_DRDA_sql'）;

哪里：

number_of_rows是一个变量，它被赋予了直通SQL完成所影响的行数。对于DDL语句，对于受影响的行数返回零 。

dblink是用于访问网关的数据库链接的名称。

native_DRDA_sql是一个有效的非查询SQL语句（ CONNECT ， COMMIT和ROLLBACK除外）。该语句不能包含绑定变量。DRDA服务器拒绝无法动态准备的本机SQL语句。 DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE函数传递的SQL语句必须是字符串。有关有效SQL语句的详细信息，请参阅特定DRDA服务器的SQL参考。

例子
  1. 使用DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE将行插入DB2表：
    DECLARE num_rows整数; BEGIN num_rows：= DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE@dblink（'INSERT INTO SCOTT.DEPT VALUES（10，''PURCHASING''，''PHOENIX''）'）;结束; /
  2. 使用DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE在DB2中创建表：
    DECLARE num_rows整数; BEGIN num_rows：= DBMS_HS_PASSTHROUGH.EXECUTE_IMMEDIATE@dblink（'CREATE TABLE MYTABLE（COL1 INTEGER，COL2 INTEGER，COL3 CHAR（14），COL4 VARCHAR（13））'）;结束; /
