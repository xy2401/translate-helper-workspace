13.53 RESTRICT_REFERENCES Pragma

RESTRICT_REFERENCES pragma声明用户定义的子程序不读取或写入数据库表或包变量。

注意：

不推荐使用RESTRICT_REFERENCES编译指示。Oracle建议使用DETERMINISTIC和PARALLEL_ENABLE而不是RESTRICT_REFERENCES 。

读取或写入数据库表或包变量的子程序很难优化，因为子程序的任何调用都可能产生不同的结果或遇到错误。如果用户定义的子程序中的语句违反了RESTRICT_REFERENCES所做的断言，则PL / SQL编译器在解析该语句时会发出错误消息，除非您指定TRUST 。

通常，此pragma是为函数指定的。如果函数调用过程，则也为这些过程指定此pragma。

RESTRICT_REFERENCES的限制Pragma

  • 该pragma只能出现在包规范或ADT规范中。

  • 只有一个RESTRICT_REFERENCES pragma可以引用给定的子程序。

语义

子程序

用户定义的子程序的名称，通常是函数。如果subprogram过载，则pragma仅适用于最新的子程序声明。

方法

MEMBER子程序的名称。有关更多信息，请参见“ CREATE TYPE语句 ” 。

默认

将pragma应用于程序包规范或ADT规范中的所有子程序（包括ADT的系统定义构造函数）。

如果还为单个子程序声明了pragma，它将覆盖该子程序的DEFAULT pragma。

RNDS

断言子程序不读取数据库状态（不查询数据库表）。

WNDS

断言子程序没有写入数据库状态（不修改表）。

RNPS

断言子程序没有读取包状态（不引用包变量的值）

对RNPS的限制

如果子程序调用SQLCODE或SQLERRM函数，则不能指定RNPS 。

WNPS

断言子程序没有写包状态（不改变包变量的值）。

对WNPS的限制

如果子程序调用SQLCODE或SQLERRM函数，则无法指定WNPS 。

信任

断言可以信任子程序不会违反其他指定的断言，并阻止PL / SQL编译器检查子程序体是否存在违规。跳过这些检查可以提高性能。

如果您的PL / SQL子程序调用C或Java子程序，则必须为PL / SQL子程序或C或Java子程序指定TRUST ，因为PL / SQL编译器无法在运行时检查C或Java子程序是否存在违规。
