自动更新索引

在讨论分区表和索引的各个维护操作之前，讨论可以在ALTER TABLE语句中指定的UPDATE INDEXES子句的影响很重要。

默认情况下，分区表上的许多表维护操作使相应的索引或索引分区无效（标记为UNUSABLE ）。然后，您必须重建整个索引，或者对于全局索引，重建每个分区。如果在ALTER TABLE语句中为维护操作指定UPDATE INDEXES ，则数据库允许您覆盖此默认行为。指定此子句会告诉数据库在执行维护操作DDL语句时更新索引。这提供了以下好处：

  • 使用基表操作更新索引。您不需要稍后更新并独立重建索引。

  • 全局索引的可用性更高，因为它们没有标记为UNUSABLE 。即使分区DDL正在执行，这些索引仍然可用，并且可以访问表中未受影响的分区。

  • 您无需查找所有无效索引的名称即可重建它们。

通过本地索引的可选子句，您可以为更新的本地索引及其分区指定物理和存储特征。

  • 您可以为每个本地索引的每个分区指定物理属性，表空间存储和日志记录。或者，您只能指定PARTITION关键字，并让数据库更新分区属性，如下所示：

    • 对于单个表分区（例如MOVE PARTITION和SPLIT PARTITION ）上的操作，相应的索引分区将继承受影响的表分区的属性。数据库不会为新索引分区生成名称，因此此操作产生的任何新索引分区都会从相应的新表分区继承其名称。

    • 对于MERGE PARTITION操作，生成的本地索引分区从结果表分区继承其名称，并从本地索引继承其属性。

  • 对于复合分区索引，您可以为每个子分区指定表空间存储。

以下操作支持UPDATE INDEXES子句：

  • ADD PARTITION | SUBPARTITION

  • COALESCE PARTITION | SUBPARTITION

  • DROP PARTITION | SUBPARTITION

  • EXCHANGE PARTITION | SUBPARTITION

  • MERGE PARTITION | SUBPARTITION

  • MOVE PARTITION | SUBPARTITION

  • SPLIT PARTITION | SUBPARTITION

  • TRUNCATE PARTITION | SUBPARTITION

SKIP_UNUSABLE_INDEXES初始化参数

SKIP_UNUSABLE_INDEXES是一个初始化参数，默认值为TRUE 。此设置禁用标记为UNUSABLE的索引和索引分区的错误报告。如果您不希望数据库选择备用执行计划以避免不可用元素，则应将此参数设置为FALSE 。

自动更新索引时的注意事项

指定UPDATE INDEXES时，以下含义值得注意：

  • 分区DDL语句可能需要更长时间才能执行，因为先前标记为UNUSABLE索引会更新。但是，您必须将此增加与执行DDL而不更新索引所花费的时间进行比较，然后重建所有索引。根据经验，如果分区的大小小于表大小的5％，则更新索引会更快。

  • EXCHANGE操作不再是快速操作。同样，您必须比较执行DDL然后重建所有索引所花费的时间。

  • 使用全局索引更新表时：

    • 索引已就地更新。将记录对索引的更新，并生成重做和撤消记录。相反，如果重建整个全局索引，则可以在NOLOGGING模式下执行此操作。

    • 手动重建整个索引会创建更高效的索引，因为它更紧凑，空间利用率更高。

  • 索引组织表不支持UPDATE INDEXES子句。但是， UPDATE GLOBAL INDEXES子句可以与DROP PARTITION ， TRUNCATE PARTITION和EXCHANGE PARTITION操作一起使用，以保持索引组织表上的全局索引可用。对于上面列表中的其余操作，索引组织表上的全局索引仍然可用。此外，在MOVE PARTITION操作之后，索引组织表上的本地索引分区仍然可用。

用于删除和截断分区的异步全局索引维护

通过仅对元数据进行索引维护来优化分区维护操作DROP PARTITION和TRUNCATE PARTITION 。

默认情况下执行DROP和TRUNCATE异步全局索引维护;但是，为了向后兼容，仍然需要UPDATE INDEXES子句。

以下列表总结了异步全局索引维护的限制：

  • 仅在堆表上执行

  • 不支持具有对象类型的表

  • 不支持具有域索引的表

  • 不为用户SYS执行

可以使用自动调度程序作业SYS.PMO_DEFERRED_GIDX_MAINT_JOB执行索引的维护操作，以清除所有全局索引。默认情况下，此作业计划定期运行。如果要主动清理索引，可以使用DBMS_SCHEDULER.RUN_JOB随时运行此作业。您还可以根据具体要求修改作业以使用计划运行。Oracle建议您不要放弃该作业。

您还可以使用以下选项之一强制清理需要维护的索引：

  • DBMS_PART.CLEANUP_GIDX - 此PL / SQL过程收集系统中可能需要清理的全局索引列表，并运行将索引还原到干净状态所需的操作。

  • ALTER INDEX REBUILD [ PARTITION ] - 此SQL语句重建整个索引或索引分区，就像Oracle Database 12 c第1版（12.1）之前的版本中所做的那样。生成的索引（分区）不包含任何陈旧条目。

  • ALTER INDEX [ PARTITION ] COALESCE CLEANUP - 此SQL语句清除索引块中的所有孤立条目。

修改子分区模板

您可以通过将其替换为新的子分区模板来修改复合分区表的子分区模板。

使用子分区模板的任何后续操作（例如ADD PARTITION或MERGE PARTITIONS ）现在都使用新的子分区模板。现有的子分区保持不变。

如果修改interval- *复合分区表的子分区模板，则尚未创建的间隔分区将使用新的子分区模板。

使用ALTER TABLE SET SUBPARTITION TEMPLATE语句指定新的子分区模板。例如：

ALTER TABLE employees_sub_template SET SUBPARTITION TEMPLATE（SUBPARTITION e TABLESPACE ts1，SUBPARTITION f TABLESPACE ts2，SUBPARTITION g TABLESPACE ts3，SUBPARTITION h TABLESPACE ts4）;

您可以通过指定空列表来删除子分区模板：

ALTER TABLE employees_sub_template SET SUBPARTITION TEMPLATE（）;

过滤维护操作

分区维护操作支持添加数据过滤，实现分区和数据维护的组合。

过滤的分区维护操作仅保留满足数据过滤的数据，作为分区维护的一部分。数据过滤的功能适用于MOVE PARTITION ， MERGE PARTITION和SPLIT PARTITION 。

示例4-27显示了使用ALTER TABLE语句移动分区，同时删除所有未打开的订单（已关闭的订单）。

过滤谓词必须位于分区表上。可以在线执行的所有分区维护操作（ MOVE和SPLIT ）也可以作为过滤的分区维护操作执行。如果指定了ONLINE ，则允许对正在维护的分区执行DML操作。

在联机模式下执行的筛选分区维护操作不会对并发正在进行的DML操作强制执行筛选器谓词。过滤条件仅在分区维护操作开始时应用一次。因此，任何后续DML都会成功，但会从过滤角度忽略。无论DML操作如何，都不会保留与分区维护开始时的过滤条件不匹配的记录。如果新插入的记录与分区键标准匹配，则插入新记录，无论它们是否满足分区维护操作的过滤条件。筛选条件仅限于分区表本身，并且不允许对其他表（例如连接或子查询表达式）进行任何引用。

在示例4-27的SQL语句中指定关键字ONLINE时，请考虑以下方案。

  • 分区维护操作开始后更新为status='open'分区q1_2016中的现有订单记录不会保留在分区中。

  • 在分区维护操作开始后以及分区维护操作正在进行时，可以在分区q1_2016插入status='closed'的新订单记录。

示例4-27执行维护操作时使用过滤子句

ALTER TABLE orders_move_part MOVE PARTITION q1_2016 TABLESPACE open_orders COMPRESS ONLINE包括行在哪里order_state ='open';
