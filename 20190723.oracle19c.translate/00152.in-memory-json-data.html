27.1内存中JSON数据概述

您可以将包含JSON数据列的表移动到内存中列存储（IM列存储），以通过缓存表达式结果来提高共享代价高昂的表达式的查询的性能。这对于扫描大量小型JSON文档的分析查询尤其有用。

仅对小于32,767字节的JSON文档支持IM列存储。如果混合使用文档大小，则会在没有内存优化的情况下处理大于32,767字节的文档。为了获得更好的性能，请考虑将大于32,767字节的文档拆分为较小的文档。

IM列存储是一个可选的SGA池，它以特殊的列式格式存储表和分区的副本，这些格式针对快速扫描进行了优化。IM列存储库补充了数据库缓冲区高速缓存中的基于行的存储。您不需要将同一对象加载到IM列存储和缓冲区缓存中。这两个缓存在事务上保持一致。数据库透明地将在线事务处理（OLTP）查询（例如主键查找）发送到缓冲区高速缓存，并将分析和报告查询发送到IM列存储。

您可以考虑在内存中使用JSON数据，以提高SQL / JSON路径访问的性能。SQL / JSON函数json_table ， json_query和json_value以及SQL条件json_exists都接受SQL / JSON路径参数，并且它们都可以从将JSON数据加载到IM列存储中获益。（使用SQL / JSON函数json_textcontains的全文搜索不会从IM列存储中受益。）一旦将JSON文档加载到内存中，对它们的任何后续基于路径的操作都使用内存中表示，这避免了与读取和解析磁盘格式相关的开销。

如果查询的JSON数据填充到IM列存储中，并且如果存在可应用于该数据的基于函数的索引，则优化程序会选择是使用索引还是扫描内存中的数据。通常，如果索引探测导致很少的文档，那么优化器可以优选功能索引。实际上，这意味着优化器可能更喜欢非常有选择性的查询或DML语句的功能索引。

另一方面，如果索引探测导致许多文档，那么优化器可能会选择扫描内存中的数据，方法是将基于函数的索引表达式扫描为虚拟列表达式。

即席查询（即，不经常用于定位给定SQL / JSON路径表达式的查询）通过快速扫描数据以一般方式从将JSON数据填充到IM列存储中受益。但是，如果您有一些经常使用的查询，那么您通常可以通过以下方式进一步提高其性能：

  • 创建从JSON数据列投影标量值（不在数组下）并将这些虚拟列加载到IM列存储中的虚拟列。

  • 在经常查询的json_table表达式上创建物化视图 ，并将视图加载到IM列存储中。

但是，如果您有一个基于函数的索引，它使用函数json_value标量值，那么您无需显式创建虚拟列来投影它。如上所述，在这种情况下，基于函数的索引表达式作为虚拟列自动加载到IM列存储中。优化器可以根据估计的成本选择是以通常方式扫描基于函数的索引，还是将索引表达式扫描为虚拟列表达式。

注意：

  • 虚拟列相对于物化视图的优点是您可以在其上构建索引，并且可以为优化程序获取有关它的统计信息。

  • 虚拟列（如一般列）受给定表的1000列限制。

在内存中使用JSON数据的先决条件

为了能够采取IM列存储的JSON数据的优势，下面都必须是真实的：

  • 数据库兼容性为12.2.0.0或更高版本。

  • 必须'extended' Oracle实例启动配置文件中为max_string_size设置的值。

  • 必须为IM列存储配置足够的SGA内存。

  • DBA已指定包含JSON列的表空间，表或INMEMORY在CREATE或ALTER语句中使用关键字INMEMORY填充到IM列存储中。

  • 初始化参数设置如下：

    • IMMEMORY_EXPRESSIONS_USAGE是STATIC_ONLY或ENABLE 。

      如果与PL / SQL过程DBMS_INMEMORY.ime_capture_expressions一起使用， ENABLE允许动态表达式的内存中实现。

    • IMMEMORY_VIRTUAL_COLUMNS是ENABLE ，表示IM列存储填充所有虚拟列。（默认值为MANUAL 。）

  • 存储JSON数据的列必须各自具有is json检查约束。（也就是说，必须知道数据是JSON数据。）

您可以使用命令SHOW PARAMETER检查每个初始化参数的值。 （您必须以数据库用户SYS或同等身份登录。）例如：

27.2将 JSON数据填充到内存中的列存储中

您可以使用ALTER TABLE INMEMORY将包含JSON数据列的表填充到内存中列存储（IM列存储）中，以通过缓存评估这些表达式的结果来提高共享代价高昂的表达式的查询的性能。

IM列存储是一个可选的SGA池，它以特殊的列式格式存储表和分区的副本，这些格式针对快速扫描进行了优化。IM列存储库补充了数据库缓冲区高速缓存中的基于行的存储。（它不会替换缓冲区缓存，但您不需要将同一对象加载到IM列存储和缓冲区缓存中。这两个缓存在事务上保持一致。）

您通过将表标记为INMEMORY指定具有给定JSON列的表（即具有is json检查约束的列）将填充到IM列存储中。 例27-1说明了这一点。

IM列存储用于查询小于32,767字节的文档。对大于该文档的文档的查询不会从IM列存储中受益。

注意：

如果要填充到IM列存储中的表中的JSON列是使用不具有至少12.2的兼容性设置的数据库创建的， 或者没有将max_string_size设置为extended （这是Oracle数据库之前的情况）例如，12c Release 2（12.2.0.1）然后你必须首先运行脚本rdbms/admin/utlimcjson.sql 。它准备所有具有JSON列的现有表，以利用版本12.2.0.1中添加的内存中JSON处理。请参阅使用JSON数据升级表以与内存中列存储一起使用 。

在将具有JSON列的表标记为INMEMORY ，会为每个JSON列添加一个In-Memory虚拟列。相应的虚拟列用于查询给定的JSON列。虚拟列包含与相应JSON列相同的JSON数据，但采用Oracle二进制格式OSON 。

示例27-1将JSON数据填充到IM列存储中

SELECT COUNT（1）FROM j_purchaseorder WHERE json_exists（po_document，'$。ShippingInstructions？（@。Address.zipCode == 99236）'）; - 执行计划显示：TABLE ACCESS FULL  - 将表指定为INMEMORY，默认PRIORITY设置为NONE， - 因此只有在触发完整扫描时才会填充它。ALTER TABLE j_purchaseorder INMEMORY ; - 再次查询表，将其填充到IM列存储中。SELECT COUNT（1）FROM j_purchaseorder WHERE json_exists（po_document，'$。ShippingInstructions？（@。Address.zipCode == 99236）'）; - 查询的执行计划现在显示：TABLE ACCESS INMEMORY FULL

27.3使用JSON数据升级表以与内存中列存储一起使用

在可以将其填充到内存中列存储（IM列存储）之前，必须先升级使用未具有至少12.2的兼容性设置或未具有max_string_size = extended的数据库创建的JSON列的表。为此，请运行脚本rdbms/admin/utlimcjson.sql 。

脚本rdbms/admin/utlimcjson.sql升级所有具有JSON列的现有表，以便可以将它们填充到IM列存储中。要使用它，必须满足以下所有条件 ：

  • 数据库参数compatible必须设置为12.2.0.0或更高。

  • 数据库参数max_string_size必须设置为extended 。

  • 要升级的JSON列必须在其上定义一个is json检查约束。
