14.1 procob命令

Pro * COBOL的位置因系统而异。通常，您的系统管理员或DBA定义环境变量或别名，或使用其他特定于操作系统的方法来使Pro * COBOL可执行文件可访问。

要运行Oracle Pro * COBOL预编译器，请发出命令

procob [ option_name = value ] [ option_name = value ] ...

选项值始终与选项名称分隔等号（=），等号周围没有空格。

例如，INAME选项指定要预编译的源文件。命令：

procob INAME = test

预编译当前目录中的test.pco文件，因为Pro * COBOL假定文件扩展名为.pco 。

除非扩展名是非标准的，否则在指定INAME时无需使用文件扩展名。

输入和输出文件名不需要附带各自的选项名称INAME和ONAME。如果未指定选项名称，Pro * COBOL会假定命令行中指定的第一个文件名是输入文件名，第二个文件名是输出文件名。

这样，命令

procob MODE = ANSI myfile myfile.cob

相当于

procob MODE = ANSI INAME = myfile.pco ONAME = myfile.cob

14.1.1区分大小写

通常，您可以使用大写或小写作为命令行选项名称和值。但是，如果您的操作系统区分大小写（例如在UNIX中），则必须使用正确的大写和小写字母组合指定文件名值，包括Pro * COBOL可执行文件的名称。

注意：未命名特定操作系统对象（如文件名）的选项名称和选项值不区分大小写。在本指南的示例中，选项名称以大写或小写形式编写，选项值通常为小写。文件名（包括Pro * COBOL可执行文件本身的名称）始终遵循执行它的操作系统使用的大小写约定。

对于某些操作系统和用户炮弹，如UNIX C shell中的? 可能需要以“转义”字符开头，例如反斜杠（\）。例如，而不是procob ? ，你可能需要使用procob \? 列出Pro * COBOL选项设置。

请参阅特定于平台的文档。

14.2预编译期间的操作

在预编译期间，Pro * COBOL生成COBOL代码，用于替换宿主程序中嵌入的SQL语句。生成的代码包括数据结构，其中包含每个主机变量的数据类型，长度和地址，以及Oracle运行时库SQLLIB所需的其他信息。生成的代码还包含对执行嵌入式SQL操作的SQLLIB例程的调用。

Pro * COBOL可以发出警告和错误消息。Oracle数据库错误消息中描述了这些消息 。

14.3关于选项

预编译时可以使用许多有用的选项。它们允许您控制资源的使用方式，报告错误的方式，输入和输出的格式，以及如何管理游标。

选项的值是文字，表示文本或数字值。例如，对于选项

...INAME = my_test

该值是一个字符串文字，用于指定文件名。

对于选项

...PREFETCH = 100

值是数字。

某些选项采用布尔值，您可以使用字符串YES或NO，TRUE或FALSE或整数文字1或0来表示布尔值。例如，选项

...SELECT_ERROR = YES

相当于

...SELECT_ERROR = TRUE

要么

...SELECT_ERROR = 1

在equals（=）符号周围不留空白区域。这是因为空格分隔了各个选项。例如，您可以在命令行上指定选项AUTO_CONNECT，如下所示：

...AUTO_CONNECT = YES

您可以缩写选项的名称，除非缩写不明确。例如，您不能使用缩写MAX，因为它可能代表MAXLITERAL或MAXOPENCURSORS。

在线提供了对Pro * COBOL选项的方便参考。要查看在线显示，请在操作系统提示符下输入不带参数的Pro * COBOL命令：

procob

显示屏提供每个选项的名称，语法，默认值和用途。标有星号（*）的选项可以在内联和命令行中指定。

14.3.1期权价值的优先顺序

选项值由以下优先级顺序确定：

  • Pro * COBOL内置的默认值

  • 系统中设置的值 配置文件

  • 用户设置的值 配置文件

  • 在命令行中输入的值

  • 内联规范中设置的值

例如，选项MAXOPENCURSORS指定缓存的打开游标的最大数量。此选项的内置Pro * COBOL默认值为10。但是，如果在系统配置文件中指定了MAXOPENCURSORS = 32，则该值将变为32。用户配置文件可以将其设置为另一个值，然后覆盖系统配置值。

如果在命令行上设置了MAXOPNCURSORS选项，则新的命令行值优先。最后，内联规范优先于所有先前的默认值。有关更多信息，请参阅“输入预编译器选项”。

14.3.2宏观和微观选项

选项模式称为宏选项。一些较新的选项（如END_OF_FETCH）仅控制一个函数，称为微选项。设置宏和微选项时，必须记住宏选项优先于微选项。当且仅当宏选项处于比微选项更高的优先级时才是这种情况。（如“选项值的优先顺序”一节中所述。）此行为是从8.0之前的Pro * COBOL版本更改。

例如，MODE的默认值为ORACLE，END_OF_FETCH的默认值为1403。如果在用户配置文件中指定MODE = ANSI，则Pro * COBOL将在获取结束时返回值100，从而覆盖默认的END_OF_FETCH值1403。如果在配置文件中同时指定MODE = ANSI和END_OF_FETCH = 1403，则将返回1403。如果在配置文件中指定END_OF_FETCH = 1403，在命令行中指定MODE = ANSI，则将返回100。

14.3.3确定当前值

您可以通过在命令行上使用问号以交互方式确定一个或多个选项的当前值。例如，如果您发出命令

procob？

终端上会显示完整的选项集以及当前值。在这种情况下，值是Pro * COBOL中内置的值，由系统配置文件中的任何值覆盖。但是，如果您发出以下命令

procob CONFIG = my_config_file.cfg？

并且当前目录中有一个名为my_config_file.cfg的文件， my_config_file.cfg文件中的选项与其他默认值一起列出。用户配置文件中的值提供缺失值，它们取代了Pro * COBOL中内置的值和系统配置文件中指定的值。

您还可以通过简单地指定选项名称后跟=?来确定单个选项的当前值=? 如在

procob MAXOPENCURSORS =？

14.4输入预编译器选项

可以在命令行或配置文件中输入所有Pro * COBOL选项（CONFIG除外）。许多选项也可以内联输入。在给定的运行期间，Pro * COBOL可以接受来自所有三个来源的选项。

14.4.1在命令行上

您可以使用命令行在命令行中输入预编译器选项... [option_name=value] [option_name=value] ... 

用一个或多个空格分隔每个选项。例如，您可以输入以下选项：
...错误=没有LTYPE =短

               
14.4.2内联

                  

                     
使用以下语法通过编写EXEC ORACLE OPTION语句来输入内联选项：
EXEC ORACLE OPTION（option_name = value）END-EXEC。
例如，您可以编写以下语句：
EXEC ORACLE选项（RELEASE_CURSOR = YES）END-EXEC。
输入内联的选项将覆盖在命令行中输入的相同选项。

                  
14.4.2.1优点

                     

                        
EXEC ORACLE功能对于在预编译期间更改选项值特别有用。例如，您可能希望逐个语句地更改HOLD_CURSOR和RELEASE_CURSOR值。“性能调优”向您展示了如何使用内联选项来优化运行时性能。

                        
如果操作系统限制了可以在命令行中输入的字符数，则可以在内容中指定内联选项，并且可以在配置文件中存储内联选项。这些将在下一节中讨论。

14.4.3配置文件

                  

                     
配置文件是包含预编译器选项的文本文件。文件中的每个记录（行）都包含一个选项及其关联的值。例如，配置文件可能包含这些行
FIPS = YES MODE = ANSI
设置FIPS和MODE选项的值。

                     
每个系统都有一个系统配置文件。系统配置文件的名称是
pcbcfg.cfg
该文件的位置是特定于操作系统的。在大多数UNIX系统上，Pro * COBOL配置文件通常位于$ ORACLE_HOME / precomp / admin目录中，其中$ ORACLE_HOME是数据库软件的环境变量。
                     

                     
请注意，在Pro * COBOL 8.0版之前，配置文件名为pccob.cfg 。
                     

                     
Pro * COBOL用户可以拥有一个或多个用户配置文件。必须使用CONFIG命令行选项指定配置文件的名称。有关更多信息，请参阅“确定当前值”。

                     
注意：您无法嵌套配置文件。这意味着CONFIG不是配置文件中的有效选项。

14.5预编译器选项的范围

               

                  
预编译单元是包含COBOL代码和一个或多个嵌入式SQL语句的文件。为给定的预编译单元指定的选项仅影响该单元;它们对其他单位没有影响。

                  
例如，如果为单元A而不是单元B指定HOLD_CURSOR = YES和RELEASE_CURSOR = YES，则单元A中的SQL语句将使用这些HOLD_CURSOR和RELEASE_CURSOR值运行，但单元B中的SQL语句将使用默认值运行。但是，连接到Oracle时生效的MAXOPENCURSORS设置在该连接的生命周期内保持有效。

                  
内联选项的范围是位置，而不是逻辑。也就是说，内联选项会影响源文件中跟随它的SQL语句，而不会影响程序逻辑流。除非您重新指定选项，否则选项设置将一直有效，直到文件结束。

14.7.20 HOLD_CURSOR

                  

                     

                        
目的

                     

                     
                     

                        
指定如何在游标缓存中处理SQL语句和PL / SQL块的游标。

                     

                     
                     

                        
句法

                     

                     
                     

                        
HOLD_CURSOR = {YES |没有}

                     

                     
                     

                        
默认

                     

                     
                     

                        
没有

                     

                     
                     

                        
使用说明

                     

                     
                     

                        
您可以使用HOLD_CURSOR来提高程序的性能。有关更多信息，请参阅“性能调整”。

                        
执行SQL数据操作语句时，其关联的游标将链接到游标缓存中的条目。游标缓存条目又链接到Oracle私有SQL区域，该区域存储处理语句所需的信息。HOLD_CURSOR控制光标和光标缓存之间的链接发生了什么。

                        
当HOLD_CURSOR = NO时，在Oracle执行SQL语句并关闭游标后，Pro * COBOL将链接标记为可重用。只要另一个SQL语句需要它指向的游标缓存条目，就会重用该链接。这将释放分配给私有SQL区域的内存并释放解析锁。

                        
当HOLD_CURSOR = YES时，保持链接; Pro * COBOL不会重复使用它。这对于经常执行的SQL语句非常有用，因为它可以加速后续执行，并且无需重新解析语句或为Oracle私有SQL区域分配内存。

                        
对于使用隐式游标的内联使用，请在执行SQL语句之前设置HOLD_CURSOR。对于使用显式游标的内联使用，请在打开游标之前设置HOLD_CURSOR。

                        
有关HOLD_CURSOR和RELEASE_CURSOR选项如何交互的信息，请参阅“性能调整”。

目的

句法

10

14.7.51 RELEASE_CURSOR

                  

                     

                        
目的

                     

                     
                     

                        
指定如何在游标缓存中处理SQL语句和PL / SQL块的游标。

                     

                     
                     

                        
句法

                     

                     
                     

                        
RELEASE_CURSOR = {YES |没有}

                     

                     
                     

                        
默认

                     

                     
                     

                        
没有

                     

                     
                     

                        
使用说明

                     

                     
                     

                        
您可以使用RELEASE_CURSOR来提高程序的性能。有关更多信息，请参阅。

                        
执行SQL数据操作语句时，其关联的游标将链接到游标缓存中的条目。游标缓存条目又链接到Oracle私有SQL区域，该区域存储处理语句所需的信息。RELEASE_CURSOR控制游标缓存和私有SQL区域之间的链接发生了什么。

                        
当RELEASE_CURSOR = YES时，在Oracle执行SQL语句并关闭游标后，Pro * COBOL会立即删除该链接。这将释放分配给私有SQL区域的内存并释放解析锁。要确保在关闭游标时释放关联的资源，必须指定RELEASE_CURSOR = YES。

对于使用隐式游标的内联使用，请在执行SQL语句之前设置RELEASE_CURSOR。对于使用显式游标的内联使用，请在打开游标之前设置RELEASE_CURSOR。

                        
请注意，RELEASE_CURSOR = YES会覆盖HOLD_CURSOR = YES。有关这两个选项如何交互的信息，请参阅“性能调整”。

  •

    • 
                           
  • 
                              
    无法使用定义表。在CSF模式下，当declare table语句传递给plsql时，plsql失败。
                              
    
                           
    • 
                           
  • 
                              
    如果使用sqlcheck=syntax和common_parser=yes则以下示例将失败：
    选择ename into：empname from emp @ dblink;
    在早期版本中，仅在使用sqlcheck =语法时才进行本地解析。但是现在当sqlcheck=syntax和common_parser=yes ，语句只是捆绑并发送到pcisyn（）。如果使用dblinks时没有连接，PLSQL将失败。
                              
    
                              
    上述语句可以预编译：
    sqlcheck = full userid = <userid> common_parser = yes
    • 
                           
  • 
                              
    插入的错误处理不遵循通常的顺序。请考虑以下语句，其中XYZ是一个组项。
    插入emp（empno，ename）值（：XYZ）
    PLSQL在对insert语句进行语法检查时不允许组项或结构（来自PLSQL的限制）。它期望将组项目分解为插入的各个元素。所以预编译器应该扩展上面的语句，以在语法阶段本身insert into emp (empno, ename) values (:b1,:b2) 。
                              
    
                              
    在语法阶段无法进行此扩展。因此， pcisyn()不会在预编译器的语法阶段中以其他语句的方式调用。该语句仅延迟到预编译器的语义阶段，仅用于插入语句。最终结果是错误消息可能不遵循通常的顺序。例如：
    EXEC SQL插入emp（empno，ename）alues（：XYZ）END-EXEC。 'alues'上的^^^^^语法错误EXEC SQL选择ename into：empname来取消END-EXEC。'fro'的^^^语法错误

    SELECT中'fro'的INSERT语法错误中'alues'的语法错误
    由于前面讨论的限制，错误序列将是：
    INSERT中“alues”的SELECT语法错误中的'fro'语法错误

14.7.57 UNSAFE_NULL

                  

                     

                        
目的

                     

                     
                     

                        
指定UNSAFE_NULL = YES可防止在不使用指示符变量的情况下获取NULL时生成ORA-01405消息。

                     

                     
                     

                        
句法

                     

                     
                     

                        
UNSAFE_NULL = {是|没有}

                     

                     
                     

                        
默认

                     

                     
                     

                        
没有

                     

                     
                     

                        
使用说明

                     

                     
                     

                        
无法输入内联。

                        
仅当MODE = ORACLE时才允许UNSAFE_NULL = YES。

                        
UNSAFE_NULL选项对嵌入式PL / SQL块中的主机变量没有影响。您必须使用指示符变量来避免ORA-01405错误。
                        

                        
当UNSAFE_NULL = YES时，如果SELECT或FETCH语句选择NULL，则不会返回错误，并且没有与输出主机变量关联的指示符变量。当UNSAFE_NULL = NO时，选择或将NULL列或表达式提取到没有关联指示符变量的主变量会导致错误（SQLSTATE为22002 ; SQLCODE为ORA-01405）。
