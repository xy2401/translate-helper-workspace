关于Oracle ODBC驱动程序

什么是ODBC？

开放式数据库连接（ODBC）提供了一个标准接口，允许一个应用程序访问许多不同的数据源。无需为每个数据源重新编译应用程序的源代码。数据库驱动程序将应用程序链接到特定数据源。数据库驱动程序是一个动态链接库，应用程序可以根据需要调用该库以获取对特定数据源的访问权限。因此，应用程序可以访问存在数据库驱动程序的任何数据源。

ODBC接口定义以下内容：

  • ODBC函数调用库，允许应用程序连接到数据源，执行结构化查询语言（SQL）语句和检索结果。

  • SQL语法基于SQL-99规范。

  • 一组标准的错误代码。

  • 连接和登录数据源的标准方法。

  • 数据类型的标准表示。

什么是Oracle ODBC驱动程序

Oracle ODBC驱动程序支持Microsoft Windows上的ODBC应用程序，以及Linux平台，Solaris和IBM Advanced Interactive eXecutive（AIX）等UNIX平台，使用Oracle Net Services软件通过ODBC接口对Oracle®数据库进行读写访问。

Oracle ODBC驱动程序使用Oracle调用接口（OCI）客户端和服务器软件向数据源提交请求并从数据源接收响应。Oracle Net Services通信协议用于OCI客户端和Oracle服务器之间的通信。

Oracle ODBC驱动程序将ODBC SQL语法转换为可用于访问数据源的语法。从数据源返回结果时，Oracle ODBC驱动程序会将它们转换回ODBC SQL语法。

Oracle ODBC驱动程序版本11.2.0.1.0的新增功能

适用于Microsoft Windows XP，Microsoft Windows 2003 Server，Microsoft Windows Vista，Linux X86-32（RHEL AS 4,5），Linux X86-64（RHEL AS 4,5）的Oracle ODBC驱动程序版本11.2.0.1.0软件的功能）（32/64位），Sun Solaris SPARC64（9,10）（32,64位），IBM AIX 5L 5.2（32,64位），Linux IA64（64位），Linux on Power（32,64位） ，Sun Solaris X64（64位），Hewlett Packard Itanium（32,64位）操作系统描述如下：

  • 预取LONG和LONG RAW数据

    Oracle ODBC驱动程序已得到增强，可预取LONG或LONG RAW数据，以提高ODBC应用程序的性能。为此，必须在Windows上的注册表中设置LONG数据的最大大小（ MaxLargeData ）（还必须在数据源名称（DSN）中添加注册表项MaxLargeData ），并在odbc.ini文件中手动设置在UNIX平台上。此增强功能可将Oracle ODBC驱动程序的性能提高10倍，具体取决于用户设置的MaxLargeData大小。MaxLargeData的默认值为0。您可以设置的MaxLargeData的最大值是64 KB（65536字节）。

    如果MaxLargeData的值大于65536，则提取的数据仅为65536字节。如果您的数据库具有大于65536字节的LONG或LONG RAW数据，则将MaxLargeData设置为0（默认值），这将导致单行提取并获取完整的LONG数据。如果在非收集模式下传递的缓冲区大小小于MaxLargeData大小，则如果数据库中的LONG数据大小大于缓冲区大小，则会发生数据截断错误。

  • 使用OCIDescribeAny()获取元数据的选项

    当应用程序对返回REF CURSORS小型打包过程进行大量调用时，可以通过强制驱动程序使用OCIDescribeAny()来提高性能。要启用此选项，请在UNIX平台上以及通过Windows上的注册表将odbc.ini的UseOCIDescribeAny值设置为T （True），默认值为F （False）。

Oracle ODBC驱动程序版本11.1.0.1.0的新增功能

用于Windows XP，Linux，Solaris和IBM AIX操作系统的Oracle ODBC驱动程序版本11.1.0.1.0软件的功能描述如下：

  • 禁用规则提示（DRH连接字符串）

    添加了新的连接选项Disable RULE Hint，允许用户指定选项以选择是否在目录API中使用RULE Hint。已完成更改以提高目录API的ODBC驱动程序的性能。该选项的默认值为TRUE ，这意味着默认情况下，目录API中不使用RULE Hint。

  • 将数字绑定为浮点数（BNF连接字符串）

    添加了新的连接选项，Bind Number As Float。当列包含float数据时，通过将NUMBER列的列绑定作为FLOAT引入，可以加快使用绑定变量作为FLOAT的查询执行速度。

  • 声明缓存

    添加了对OCI语句缓存功能的支持，该功能为每个会话提供和管理语句缓存。通过实现对OCI语句缓存选项的支持，当用户在同一连接中多次解析同一语句时，Oracle ODBC驱动程序性能得到改善。语句缓存标志的默认值为FALSE 。

配置数据源

注意：

以下配置步骤适用于Windows用户。Unix用户必须使用odbc_update_ini.sh文件来创建DSN。

安装Oracle ODBC驱动程序和配置Oracle Net Services之后 ，在使用Oracle ODBC驱动程序之前，必须配置数据源。

在应用程序可以与数据源通信之前，您必须提供配置信息。配置信息通知Oracle ODBC驱动程序您要访问哪些信息。

数据源包括您要访问的数据，其关联的操作系统，数据库管理系统以及用于访问数据库管理系统的网络平台。Oracle ODBC驱动程序提交的请求的数据源是Oracle数据库，并支持Oracle Net Services下可用的传输。

要配置或添加Oracle数据源：

DATE和TIMESTAMP

Oracle DATE和TIMESTAMP数据类型的语义与具有相同名称的ODBC数据类型不完全对应。Oracle DATE数据类型包含日期和时间信息，而SQL_DATE数据类型仅包含日期信息。Oracle TIMESTAMP数据类型还包含日期和时间信息，但它在小数秒内具有更高的精度。ODBC驱动程序将Oracle DATE和TIMESTAMP列的数据类型报告为SQL_TIMESTAMP以防止信息丢失。类似地，ODBC驱动程序将SQL_TIMESTAMP参数绑定为Oracle TIMESTAMP值。

浮点数据类型

连接到10.1或更高版本的Oracle服务器时，ODBC驱动程序将Oracle浮点数据类型BINARY_FLOAT和BINARY_DOUBLE分别映射到ODBC数据类型SQL_REAL和SQL_DOUBLE 。在以前的版本中， SQL_REAL和SQL_DOUBLE映射到通用Oracle数值数据类型。

Oracle对列类型为LONG SQL_LONGVARCHAR数据的限制为2,147,483,647字节。Oracle对列类型为CLOB的SQL_LONGVARCHAR数据的限制为4千兆字节。限制因素是客户端工作站内存。

SQL_LONGVARCHAR和SQL_LONGVARBINARY

Oracle数据库每个表只允许一个长数据列。长数据类型是SQL_LONGVARCHAR （ LONG ）和SQL_LONGVARBINARY （ LONG RAW ）。Oracle建议您改用CLOB和BLOB列。表中的CLOB和BLOB列数没有限制。

错误消息

发生错误时，Oracle ODBC驱动程序返回本机错误号， SQLSTATE （ODBC错误代码）和错误消息。驱动程序从驱动程序检测到的错误和Oracle服务器返回的错误中获取此信息。

本机错误

对于数据源中发生的错误，Oracle ODBC驱动程序返回Oracle服务器返回给它的本机错误。当Oracle ODBC驱动程序或驱动程序管理器检测到错误时，Oracle ODBC驱动程序返回零的本机错误。

SQLSTATE

对于数据源中发生的错误，Oracle ODBC驱动程序将返回的本机错误映射到适当的SQLSTATE 。当Oracle ODBC驱动程序检测到错误时，它会生成相应的SQLSTATE 。当驱动程序管理器检测到错误时，它会生成相应的SQLSTATE 。

在将SQL Server数据库迁移到Oracle期间，每个SQL Server过程（或函数）的定义将转换为其等效的Oracle语法，并在Oracle的模式中定义。迁移过程通常以下列方式之一进行重组（并在模式中创建）：

  • 所有过程都迁移到一个模式（默认选项）。

  • 在一个SQL Server数据库中定义的所有过程都将迁移到使用该数据库名称命名的模式。

  • 一个用户拥有的所有过程都将迁移到使用该用户名命名的模式。

要支持这三种组织迁移过程的方法，可以指定其中一个模式名称选项来转换过程名称。已转换的Oracle过程调用中的对象名称不区分大小写。

ODBC环境中的Unicode支持

Microsoft或unixODBC ODBC驱动程序管理器（驱动程序管理器）使所有ODBC驱动程序（无论它们是否支持Unicode）都显示为符合Unicode。这允许独立于底层ODBC驱动程序的Unicode功能编写ODBC应用程序。

驱动程序管理器可以模拟对ANSI ODBC驱动程序的Unicode支持的程度受限于Unicode数据和本地代码页之间可能的转换。当驱动程序管理器从Unicode转换为本地代码页时，可能会丢失数据。除非底层ODBC驱动程序支持Unicode，否则无法使用完全Unicode支持。Oracle ODBC驱动程序提供完整的Unicode支持。

ODBC API中的Unicode支持

ODBC API使用“W”和“A”后缀约定支持Unicode和ANSI入口点。ODBC应用程序开发人员无需显式调用带后缀的入口点。使用UNICODE和_UNICODE预处理程序定义编译的ODBC应用程序会生成相应的调用。例如，对SQLPrepare的调用被编译为SQLPrepareW 。

C数据类型SQL_C_WCHAR已添加到ODBC接口，以允许应用程序指定输入参数编码为Unicode或请求以Unicode返回的列数据。宏SQL_C_TCHAR对于必须同时构建为Unicode和ANSI的应用程序非常有用。 SQL_C_TCHAR宏编译为Unicode应用程序的SQL_C_WCHAR和ANSI应用程序的SQL_C_CHAR 。

SQL数据类型SQL_WCHAR ， SQL_WVARCHAR和SQL_WLONGVARCHAR已添加到ODBC接口，以将表中定义的列表示为Unicode。潜在地，这些值是从对SQLDescribeCol ， SQLColAttribute ， SQLColumns和SQLProcedureColumns调用返回的。

SQL列类型NCHAR ， NVARCHAR2和NCLOB支持Unicode编码。此外，如果在列定义中指定了字符语义，则SQL列类型CHAR和VARCHAR2也支持Unicode编码。

驱动程序管理器在检测到基础ODBC驱动程序不支持Unicode时执行以下功能：

  • 在调用ANSI ODBC驱动程序之前，将Unicode函数调用转换为ANSI函数调用。字符串参数从Unicode转换为本地代码页。例如，对SQLPrepareW的调用将转换为调用SQLPrepare 。SQL语句参数的文本从Unicode转换为本地代码页。

  • 将作为字符数据的返回参数从本地代码页转换为Unicode。例如，通过SQLColAttribute返回列名。

  • 对于绑定为SQL_C_WCHAR列，将数据从本地代码页转换为Unicode。

  • 将数据从Unicode转换为本地代码页，以获取绑定为SQL_C_WCHAR输入参数。

SQLGetData性能

SQLGetData函数允许ODBC应用程序指定数据类型，以便在获取数据后接收列。OCI要求Oracle ODBC驱动程序在获取之前指定数据类型。在这种情况下，Oracle ODBC驱动程序使用它拥有的有关数据库中定义的列的数据类型的知识来确定如何最好地默认通过OCI获取列。

如果包含字符数据的列未受SQLBindCol绑定，则Oracle ODBC驱动程序必须确定它是否必须将该列作为Unicode或本地代码页获取。驱动程序可能默认以Unicode格式接收列，但是，这可能导致多达两次不必要的转换。例如，如果数据在数据库中以ANSI编码，则会进行ANSI到Unicode转换以将数据提取到Oracle ODBC驱动程序中。如果ODBC应用程序随后将数据作为SQL_C_CHAR请求，则会进行额外的转换以将数据恢复为其原始编码。

获取数据时使用Oracle客户端的默认编码。但是，ODBC应用程序可以覆盖此缺省值，并通过将列或参数绑定为WCHAR数据类型将数据作为Unicode获取。

一般ODBC编程技巧

本节介绍一些用于提高ODBC应用程序性能的一般编程技巧。

  • 如果应用程序经常连接和断开数据源，则启用连接池。与重新建立连接相比，重用池化连接非常有效。

  • 最小化必须准备语句的次数。在可能的情况下，使用绑定参数使语句可以重用于不同的参数值。准备一次语句并多次执行它比为每个SQLExecute准备语句要有效得多。

  • 如果您知道应用程序不会检索它们，请不要在SELECT语句中包含列;特别是LONG专栏。由于数据库服务器协议的性质，ODBC驱动程序必须获取LONG列的全部内容（如果它包含在SELECT语句中），无论应用程序绑定列还是执行SQLGetData 。

  • 如果要执行不更新数据源的事务，请将ODBC SQLSetConnectAttr函数的SQL_ATTR_ACCESS_MODE属性设置为SQL_MODE_READ_ONLY 。

  • 如果未使用ODBC转义子句，请将ODBC SQLSetConnectAttr函数或ODBC SQLSetStmtAttr函数的SQL_ATTR_NOSCAN属性设置为true。

  • 使用ODBC SQLFetchScroll函数而不是ODBC SQLFetch函数从具有大量行的表中检索数据。

  • 多次使用相同的SQL语句时，启用OCI语句缓存（ StatementCache=T ）。

  • 将NUMBER列绑定为FLOAT加快查询执行速度（ BindAsFLOAT=T ）。

  • 在获取LONG或LONG RAW设置MaxLargeData=<value>以获得最佳性能。

  • 为UseOCIDescribeAny=T调用返回Ref Cursor小型打包过程的应用程序设置UseOCIDescribeAny=T可以提高性能。

数据源配置选项

本主题讨论以下ODBC数据源配置选项的性能影响：

话题：

  • 启用LOB

  • 将TIMESTAMP绑定为DATE

  • 启用关闭游标

  • 启用线程安全

启用结果集

此选项支持从过程调用返回结果集（例如， RefCursor ）。默认设置是启用结果集的返回。

ODBC驱动程序必须查询数据库服务器以确定过程及其数据类型的参数集，以确定是否存在任何RefCursor参数。在第一次准备和执行任何过程时，该查询引起额外的网络往返。

启用LOB

此选项支持插入和更新LOB。默认设置已启用。

ODBC驱动程序必须查询数据库服务器以确定INSERT或UPDATE语句中每个参数的数据类型，以确定是否存在任何LOB参数。第一次准备和执行INSERT或UPDATE时，此查询会产生额外的网络往返。

将TIMESTAMP绑定为DATE

将SQL_TIMESTAMP参数绑定为适当的Oracle数据类型。如果此选项为TRUE ，则SQL_TIMESTAMP将绑定为Oracle DATE数据类型。如果此选项为FALSE ，则SQL_TIMESTAMP将绑定为Oracle TIMESTAMP数据类型（这是默认值）。

启用关闭游标

ODBC函数的SQL_CLOSE选项SQLFreeStmt应该使用语句关闭关联的游标并丢弃所有挂起的结果。应用程序可以通过再次执行语句重新打开游标，而无需再次执行SQLPrepare 。典型的情况是一个空闲一段时间但重用相同SQL语句的应用程序。当应用程序空闲时，它可能释放相关的服务器资源。

Oracle ODBC驱动程序所在的Oracle调用接口（OCI）不支持关闭游标的功能。因此，默认情况下， SQL_CLOSE选项在Oracle ODBC驱动程序中无效。游标和相关资源在数据库服务器上保持打开状态。

启用此选项会导致关联的游标在数据库服务器上关闭。但是，这会导致SQL语句的解析上下文丢失。ODBC应用程序可以再次执行该语句，而无需调用SQLPrepare 。但是，ODBC驱动程序内部必须准备并执行该语句。启用此选项会严重影响一次准备语句并重复执行的应用程序的性能。

仅在绝对需要释放服务器上的资源时才启用此选项。

启用线程安全

如果应用程序是单线程的，则可以禁用此选项。默认情况下，ODBC驱动程序确保对所有内部结构（环境，连接，语句）的访问是线程安全的。单线程应用程序可以通过禁用此选项来消除一些线程安全开销。禁用此选项通常会显示性能略有改善。

获取缓冲区大小

将Oracle ODBC驱动程序配置对话框的Oracle选项选项卡中的获取缓冲区大小设置为以字节为单位指定的值。此值确定ODBC驱动程序一次从Oracle数据库预取到客户端缓存的数据行数，而不管应用程序在单个查询中请求的行数，从而提高性能。

通常一次获取少于20行数据的应用程序可以改善响应时间，尤其是在慢速网络连接或负载较重的服务器上。将此设置得太高会使响应时间变差或消耗大量内存。默认值为64,000个字节。选择最适合您应用的值。

注意：

当存在LONG和LOB数据类型时，ODBC驱动程序预取的行数不是由获取缓冲区大小决定的。包含LONG和LOB数据类型可以最大限度地降低性能，并可能导致过多的内存使用。ODBC驱动程序忽略了获取缓冲区大小，并在存在LONG和LOB数据类型时预取一定数量的行。

DATE和TIMESTAMP数据类型

如果在WHERE子句中使用数据库中的DATE列并且该列具有索引，则可能会对性能产生影响。例如：

SELECT * FROM EMP WHERE HIREDATE =？

在此示例中， HIREDATE列上的索引可用于使查询快速执行。但是，因为HIREDATE实际上是DATE值而ODBC驱动程序正在将参数值作为TIMESTAMP ，所以Oracle服务器的查询优化器必须应用转换函数。为了防止不正确的结果（如果参数值具有非零小数秒可能会发生），优化程序将转换应用于HIREDATE列，从而产生以下语句：

SELECT * FROM EMP WHERE TO_TIMESTAMP（HIREDATE）=？

不幸的是，这会导致禁止在HIREDATE列上使用索引，而是服务器执行表的顺序扫描。如果表有很多行，这可能需要很长时间。作为这种情况的解决方法，ODBC驱动程序具有将Bind TIMESTAMP as DATE的连接选项。启用此选项后，ODBC驱动程序将SQL_TIMESTAMP参数绑定为Oracle DATE数据类型而不是Oracle TIMESTAMP数据类型。这允许查询优化器使用DATE列上的任何索引。

注意：

此选项仅适用于Microsoft Access或将DATE列绑定为TIMESTAMP列的其他类似程序。当存在实际的TIMESTAMP列或可能发生数据丢失时，请勿使用此选项。Microsoft Access使用选择的任何列作为主键来执行此类查询。
