实际应用程序安全管理（RASADM）的安装和配置

执行先决条件，安装和配置任务，然后运行RASADM应用程序。

先决条件

在安装RASADM之前，请确保Oracle Database实例和Application Express都已启动并正在运行。

需要满足以下先决条件：

  • Oracle数据库12.1.0.1版及更高版本

  • Application Express 5.0及更高版本

安装

您可以从OTN下载RASADM，安装它以创建RASADM架构和RASADM APEX工作区，它们可以设置APEX安全功能。安装将创建名为RASADM的模式用户。如果提供了LDAP主机，它会配置外部主体存储（LDAP）。

执行以下步骤以安装RASADM：

  1. 从OTN下载RASADM（ Oracle Real Application Security（RAS）下载 ）。

  2. 将下载的zip文件解压缩到您选择的目录中。

  3. 在解压缩的目录中，将目录更改为installer目录。

  4. 将AS SYSDBA连接到SQL * Plus并运行以下PL / SQL脚本来安装RASADM：

SQL> @ rasadm_ins.sql

运行安装脚本时，将显示以下输出：

。。Oracle RASADM安装.............................................。。创建具有特权的模式用户。RASADM提供数据领域预览功能。如果启用，管理员将能够查看数据库中任何用户表中的数据。启用数据预览功能？[否]输入RASADM管理员用户（admin）： 密码的密码 。创建APEX工作区和管理员用户。。。导入APEX应用程序。如果使用外部用户和组，请输入LDAP主机名（如果没有，则输入ENTER）： hostname 。发布配置。。完成。如有任何错误，请查看rasadm_ins.log。.........................................。

安装脚本执行以下操作：

  1. 使用锁定的帐户和过期的密码创建名为RASADM的模式用户，并授予用户有限的权限，这些权限是编译某些API所必需的。

  2. 数据域预览功能将授予RAS角色RASADM_POLICY_ADMIN ，后者又被授予管理员。

  3. 使用Application Express管理员的UI创建RASADM工作区并将RASADM应用程序代码导入RASADM工作区，然后设置一些Application Express安全功能。

  4. 配置外部主体存储（LDAP）。如果提供了LDAP主机，则会创建网络ACL以授予给定LDAP主机和端口上的RASADM用户CONNECT和RESOLVE权限。在安装之后，RASADM用户还可以选择从RASADM UI配置外部主体存储（LDAP）。在这种情况下，RASADM管理员有责任手动向网络ACL授予数据集。

注意：

要以系统管理员身份卸载RASADM，请运行rasadm_unins.sql 。

用户管理

这是可以登录RASADM的admin用户。 RASADM运行时用户是具有密码的Real Application Security用户，可以直接登录到数据库。用户必须具有授予的Real Application Security角色（ RASADM_POLICY_ADMIN或RASADM_USER_ADMIN或两个角色）才能访问RASADM。

RAS提供两个管理角色 - RASADM_POLICY_ADMIN和RASADM_USER_ADMIN 。

RASADM_POLICY_ADMIN是Real Application Security角色，具有执行策略管理的权限;例如，具有此角色的Real Application Security用户可以创建策略，特权，特权类，命名空间以及常规和动态角色。用户无法创建Real Application Security用户。

RASADM_USER_ADMIN是Real Application Security角色，具有执行用户管理的权限;例如，具有此角色的Real Application Security用户可以创建用户以及常规和动态角色。用户无法使用RASADM提供的其他功能。

但是，这两个角色都具有在所有页面上查看报告的权限。

可以使用数据库中的Real Application Security Admin API（在PL / SQL中）创建RASADM用户。

RASADM期待两种用户。只能执行策略管理任务的人员和可以执行策略和用户管理的其他人员。前者需要授予RASADM_POLICY_ADMIN Real Application Security角色，后者需要同时授予RASADM_POLICY_ADMIN和RASADM_USER_ADMIN Real Application Security角色。

关于使用Oracle数据库Real Application Security的数据安全性

有效的安全性要求定义哪些应用程序用户，应用程序或功能可以访问哪些数据，以执行哪种操作。

因此，有效的安全性有以下三个方面：

  1. 哪个应用程序用户（ 校长 ）

  2. 可以执行哪些操作（ 应用程序权限 ）

  3. 关于哪些数据（ 数据领域 ）

您可以分别定义（1）主体，（2）应用程序权限和（3）与这三个维度相关的对象（数据域）。主体是用户和角色，其中用户是应用程序用户，角色可以表示应用程序用户，系统状态或一段代码的属性。

委托人在ACL中被授予应用程序权限。然后，通过定义保护表数据的行和列的数据安全策略，这些ACL与数据相关。

  3. 使用RASADM以开发流程中描述的逐步方式创建数据安全策略。

实际应用程序安全中使用的术语

为您定义术语，以便更好地理解RASADM应用程序中使用的Real Application Security概念。

应用程序用户是数据库已知的应用程序最终用户标识。这些用户是无模式的，不拥有任何数据库对象或资源。应用程序用户通过中间层创建到数据库的应用程序会话，并可以通过直接登录应用程序用户帐户直接创建到数据库的应用程序会话。使用RASADM，您可以创建应用程序用户。

应用程序特权是控制应用程序级操作执行的命名特权。操作可以是数据或应用程序工件，例如表示工作流任务的UI工件或Web应用程序中的按钮和页面。应用程序权限可以暗示其他应用程序权限，包括Real Application Security DML权限类的SELECT ， INSERT和DELETE 。

应用程序角色表示已被授予该角色的一组应用程序用户。在访问控制列表（ACL）中，可以为角色授予权限，这与向已授予该角色的所有用户授予这些权限具有相同的效果。应用程序角色只能授予应用程序用户或应用程序角色。

动态应用程序角色是仅在某些情况下启用的应用程序角色，例如，当用户使用SSL登录时，或在特定时间段内，等等。您无法将动态应用程序角色授予其他应用程序角色或用户，但您可以将其他应用程序角色授予动态应用程序角色。

权限类是一组应用程序权限的范围。权限类定义可以在数据安全策略的访问控制列表（ACL）中授予的一组权限。

访问控制条目（ACE）授予或拒绝特定主体（应用程序用户或应用程序角色）的应用程序特权。

访问控制列表（ACL）是一个命名的权限授予列表。Oracle Real Application Security ACL允许对权限授予进行各种约束，例如有序的否定授权。如果您创建的ACL依赖于您在自己的权限类中定义的一组自定义应用程序权限，那么您只能授予该权限类的权限。

数据域是与ACL关联的一组相关应用程序表中的逻辑行集的集合。数据领域表示应用程序级资源或业务对象，并使用SQL谓词定义。授权在ACL中定义，ACL指定授予数据集上哪些原则的特权。RASADM支持以下数据领域类型：

  • 常规 - 在SQL谓词中指定要保护的数据集的一部分以及用于执行授权的ACL时保护数据集。

  • 参数化 - 当您在SQL谓词中指定要使用的参数时，以及在ACL中指定要授予该参数的值的权限时，保护数据集。例如，考虑department=&DEPT_ID ，其中参数名称DEPT_ID在符号& DEPT_ID 。在这种情况下，参数名称DEPT_ID与不同的部门ID相关联。这使您可以基于部门ID创建授权，而无需为每个DEPT_ID值创建许多特定于部门ID的数据域。

  • 继承 - 称为主 - 详细信息策略使用主对象的策略通过指向主对象来保护详细信息对象。您将父模式和父对象（列）指定为主，然后在父列与子列（详细信息）对象或常量值之间构建1：1关系。您可以使用多个父对子对构建关系。对于多个parent = child对，常量值必须与其他parent = child对一起作为连接条件才能形成1：1关系。如果要使用父对象策略进一步保护子列，可以选择输入when条件。

数据安全策略通过将数据域与ACL相关联来保护数据域。可以使用细粒度访问控制来保护行和列级别的数据库记录。数据安全策略执行以下两个功能：

  • 数据领域授权。

    数据领域授权指定要在一行或多行的数据域中保护的数据，并将每个数据域与一个访问控制列表（ACL）相关联。ACL指定通过应用程序权限向特定主体授予或拒绝哪种访问权限。它控制对数据领域行的访问，以形成所谓的数据领域授权。给定的ACL保护给定的数据领域，并控制对特定应用程序用户或应用程序角色（称为主体）的访问。

  • 列授权。

    列授权可选地应用其他应用程序权限来保护特定列。当您需要为敏感列数据添加列授权安全性以及其他自定义应用程序权限时，这非常有用。

总之，只允许登录的应用程序用户根据其关联ACL中的应用程序权限，对数据域内的记录执行包括DML在内的操作，包括单独的数据行。因此，数据安全策略由数据领域授权和列授权组成，其通过仅允许访问在关联的ACL中具有应用程序特权的应用程序用户来保护数据领域。

授权服务检查是否授予用户特权。

会话命名空间属性是应用程序命名空间下具有关联访问控制策略的属性 - 值对的集合。通常，应用程序需要在不同的应用程序会话中使用相同的命名空间。您可以创建命名空间模板来定义和初始化命名空间。命名空间模板定义命名空间及其属性。它用于在应用程序会话中初始化命名空间。

应用程序会话是应用程序用户的会话，对应于数据库中应用程序用户的安全上下文，角色和命名空间属性。

设计阶段

设计阶段涉及识别应用程序执行的所有需要应用程序权限来控制数据访问的任务。

例如，在设计阶段，应用程序策略设计者必须确定：

  1. 需要访问控制的应用程序级操作集。

  2. 表和视图的行和列，可以作为应用程序级操作的一部分进行访问。

  3. 可以执行这些操作的一组actor或principal（用户和角色）。

  4. 运行时应用程序会话属性标识一个或多个表的行。这些属性名称在选择要授权的行的谓词中使用，并且在应用程序执行期间设置它们的值。

发展趋势

在开发阶段，作为RASADM管理员，您使用RASADM来开发数据安全策略。

请按照以下步骤开发数据安全策略：

  1. 创建相应的应用程序用户和角色。如果使用外部目录服务器，请在目录服务器中创建应用程序用户和角色或用户组。请按照此过程在数据库中本机创建这些主体：

    1. 如果需要，创建应用程序角色并将应用程序角色授予应用程序角色。请参阅创建应用程序角色 。

    2. 创建应用程序用户并向应用程序用户授予应用程序角色。请参阅创建应用程序用

    3. 使用来自外部存储的主体时，配置目录服务器以获取用户和角色。请参阅配置 。

    4. 对于外部Directory Server中的用户和角色，请管理将RASADM与Directory Server一起使用的参数设置。请参阅管理设置 。

  2. 创建计划用于为应用程序开发安全策略的每个权限类。每个权限类由一个或多个适当的权限组成，您可以在ACL中引用这些权限，并将其授予应用程序用户和应用程序角色。每个权限类都通过ACL授权数据安全策略所需的应用程序级操作。请参阅创建应用程序权限类 。

  3. 创建一个或多个可在不同应用程序会话中使用的会话命名空间。这包括为会话命名空间定义其属性集（应用程序属性）及其关联的访问控制策略或ACL，您可以从列表或创建中进行选择。请参见创建命名空间 。

  4. 通过将每个数据域与ACL相关联来创建数据安全策略，以便根据需要创建数据域授权和列授权。这个过程包括四个部分：

    1. 策略信息 - 您选择要保护的对象和保护它的权限类，以及指定策略名称并选择策略所有者。请参阅创建数据安全策略中的步骤3。

    2. 列级别授权 - 选择要保护的列的名称，并选择要授予访问列的权限，该列与您在步骤3a中选择的权限类相关联。请参阅创建数据安全策略中的步骤4。

    3. 数据领域授权 - 您创建一个SQL谓词来表示要保护的数据领域，并将每个谓词添加到数据领域授权列表中。然后选择或创建ACL以保护数据领域。接下来，通过选择适当的权限，创建要添加到由每个主体组成的权限授予列表以及是否允许授权或拒绝授权的权限授予。请参阅创建数据安全策略中的步骤5。

    4. 应用策略 - 您可以应用，删除，启用或禁用您正在创建的数据安全策略，并选择指定某些应用选项，允许表或视图的所有者绕过此数据安全策略，以及是否强制执行语句类型这个政策。请参阅创建数据安全策略中的步骤6。

    • 默认架构 - 单击（ ^ ）以选择此应用程序用户要访问的架构。

    • 结束日期 - 单击日历图标以选择此应用程序用户的有效结束日期，或将该字段留空。无结束日期表示此应用程序用户始终有效。如果指定开始日期，则必须指定结束日期。

  3. 在“ 管理设置”页面上，进行更新。

    注意：

    请仔细更改这些设置，因为“ 参数”部分中的错误设置可能会对与目录服务器的连接产生负面影响。

    LDAP组的名称显示在“ 设置”部分中。您可以在“ 设置”和“ 参数”部分中对以下字段进行更新：

    • 描述 - 用户名的描述。

    • 主机 - 运行目录服务器的主机的名称。

    • 端口 - 目录服务器端口号。

    • 用户 - 目录信息树（DIT）中的用户实体位置

    • Base - 用于在Directory服务器中存储用户条目的命名上下文。

    • 名称属性 - 名称属性和值对。

    • ID属性 - ID属性和值对。ID属性应该是LDAP条目的属性，由base指定。它应该全局识别用户。通常，使用GUID。对于Oracle Internet Directory（OID），它应该是orclguid ;对于Microsoft Active Directory（AD），它应该是objectGUID 。

    • 描述属性 - 描述属性和值对。

    • SSL Wallet Location - 文件系统上钱包的路径，例如file: /home/mywallet 。 word file:必须用作前缀。

    • SSL身份验证模式 - 模式值为：1表示不需要身份验证，2表示需要单向身份验证，或3表示需要双向身份验证。
