更多关于个人资料

SQLJ概要文件包含有关嵌入式SQL操作的信息，并为应用程序使用的每个连接上下文类创建单独的概要文件。配置文件在SQLJ转换程序代码生成阶段创建，并在自定义阶段进行自定义。自定义使您的应用程序能够使用供应商特定的数据库功能。将这些特定于供应商的操作分离到您的配置文件中，可以使生成的其余代码保持通用。

每个概要文件都包含一系列使用相关连接上下文类的SQLJ语句条目，其中每个条目对应于应用程序中的一个SQL操作。

配置文件作为序列化对象存在于存储在随应用程序打包的资源文件中。因此，可以随时加载，读取和修改（添加或重新定制）配置文件。自定义配置文件时，仅添加信息，从不删除信息。可以在不丢失先前自定义的情况下进行多个自定义，以便您的应用程序保持在多个环境中运行的能力。这称为二进制可移植性 。

对于具有二进制可移植性的配置文件，Oracle SQLJ实现中已满足SQLJ行业标准要求。

代码生成期间创建配置文件

在代码生成期间，翻译器创建每个配置文件，如下所示：

  1. 它创建一个配置文件对象作为sqlj.runtime.profile.一个实例sqlj.runtime.profile.Profile类。

  2. 它将有关嵌入式SQL操作的信息插入到配置文件对象中，以用于使用相关连接上下文类的SQLJ语句。

  3. 它将配置文件对象序列化为Java资源文件，称为配置文件 ，文件扩展名为.ser 。

注意：

Oracle SQLJ实现提供了一个选项，让翻译器自动将这些.ser文件转换为.class文件。某些浏览器不支持.ser文件，并且在将已翻译的应用程序加载到服务器时可能很麻烦。但是，这会阻止对配置文件进行任何进一步的自定义。

正如“ 代码生成 ”中所讨论的，应用程序Foo概要文件名具有以下形式：

Foo_SJProfile n .ser

SQLJ根据需要生成Foo_SJProfile0.ser ， Foo_SJProfile1.ser等，具体取决于您在代码中使用的连接上下文类的数量。或者，如果启用了-ser2class选项，则SQLJ将生成Foo_SJProfile0.class ， Foo_SJProfile1.class ，依此类推。

每个概要文件都有一个在SQLJ运行时期间调用的getConnectedProfile()方法。此方法返回与JDBC Connection对象等效的内容，但具有附加功能。这在“ 运行时定制配置文件的功能 ”中进一步讨论。

注意：

引用“配置文件对象”表示配置文件处于其原始的非序列化状态。引用“配置文件”表示配置文件在.ser文件中处于其序列化状态。

更多关于配置文件定制

使用ISO SQLJ代码时，在SQLJ源文件上运行sqlj脚本包括自动定制过程，其中在代码生成阶段创建的每个配置文件都是自定义的，以便与特定数据库一起使用。默认的自定义程序是Oracle自定义程序oracle.sqlj.runtime.OraCustomizer ，可优化您的配置文件，以使用特定于Oracle Database 12 c第2版（12.2）的类型扩展和性能增强。

您还可以运行sqlj脚本来自定义先前创建的配置文件。在SQLJ命令行上，您可以单独指定.ser文件，包含.ser文件的JAR文件或两者。

注意：

  • 无论何时在转换期间使用默认的Oracle自定义程序，即使您未在代码中使用Oracle扩展，您的应用程序在运行时也需要Oracle SQLJ运行时和Oracle JDBC驱动程序。

  • 如果应用程序没有自定义，或者没有适合连接的应用程序，则使用通用SQLJ运行时。

  • 您可以运行SQLJ来处理.sqlj和.java文件（用于转换，编译和自定义）或处理.ser和.jar文件（仅用于自定义），但不能同时处理这两个类别。

定制器线束和定制器概述

不管你是否使用Oracle定制或替代定制，SQLJ使用一种被称为在实现您的自定义的定制线束前端的定制工具。

运行SQLJ时，可以为自定义程序线束指定自定义选项（适用于您使用的任何自定义程序的常规自定义设置）和自定义程序（对于特定自定义程序使用的设置）。在任何一种情况下，您都可以在命令行或属性文件中指定这些选项。这在“ 自定义选项和选择自定义程序 ”中讨论 。

定制程序必须是遵循标准JavaBeans API的JavaBeans组件才能公开其属性，并且必须实现sqlj.runtime.profile.util.ProfileCustomizer接口，它指定了customize()方法。对于要自定义的每个配置文件，定制程序线束调用定制程序对象的customize()方法。

Oracle定制程序满足上述要求，并在oracle.sqlj.runtime.定义oracle.sqlj.runtime.OraCustomizer类。

定制过程中的步骤

翻译期间的SQLJ自定义过程包括以下步骤（如果适用），在端到端SQLJ运行的自定义阶段，或者运行SQLJ以仅自定义现有配置文件时：

  1. SQLJ实例化并调用自定义程序工具并将其传递给您指定的任何常规自定义选项。
  2. 自定义程序工具实例化您正在使用的自定义程序，并将您指定的任何自定义程序特定选项传递给它。
  3. 当您仅运行SQLJ进行自定义，在命令行上指定一个或多个JAR文件时，自定义程序工具会发现并提取这些JAR文件中的配置文件。
  4. 自定义程序.ser将每个配置文件反序列化为配置文件对象（在端到端SQLJ运行期间自动创建的.ser文件，在命令行上指定的仅用于自定义的.ser文件，或从在.ser指定的JAR文件中提取的.ser文件）。命令行仅用于自定义）。
  5. 如果您使用的自定义程序需要数据库连接，则自定义程序线束会建立该连接。
  6. 对于每个配置文件，线束调用在步骤2中实例化的定制器对象的customize()方法（与SQLJ一起使用的customize()必须具有customize()方法）。
  7. 对于每个配置文件， customize()方法通常在配置文件中创建和注册配置文件自定义。但是，这取决于定制器的预期功能。有些可能具有专门的目的，不需要以这种方式创建和注册自定义。
  8. 自定义程序线束将每个配置文件重新.ser并将其放回.ser文件中。
  9. 当您仅运行SQLJ进行自定义，在命令行上指定一个或多个JAR文件时，自定义程序工具会重新创建JAR内容，插入每个自定义.ser文件以替换原始相应的未自定义.ser文件。

注意：

  • 如果在自定义配置文件期间发生错误，则不会替换原始.ser文件。

  • 如果在自定义JAR文件中的任何配置文件期间发生错误，则不会替换原始JAR文件。

  • SQLJ一次只能运行一个自定义程序。如果要在单个配置文件上完成多个自定义，则必须多次运行SQLJ。对于其他自定义，请直接在SQLJ命令行上输入配置文件名称。

创建和注册配置文件自定义

当线束调用customize()方法来自定义配置文件时，它会传递配置文件对象，SQLJ连接上下文对象（如果您使用的是需要连接的自定义程序），以及错误日志对象（用于日志记录）自定义过程中的错误消息）。

在单次运行SQLJ时，相同的错误日志对象用于所有自定义，但其使用是透明的。定制器线束读取写入错误日志对象的消息，并将它们实时报告给标准输出设备（无论SQLJ使用什么，通常是您的屏幕）。

回想一下，每个配置文件都有一组条目，其中每个条目对应一个SQL操作。（这些是您的应用程序中使用与此配置文件关联的连接上下文类的实例的SQL操作。）

customize()方法对这些条目实现特殊处理。它可以像检查每个条目以验证其语法一样简单，也可以更复杂，例如创建与原始条目等效但被修改为使用特定数据库功能的新条目。

注意：

  • 任何配置文件条目的customize()处理都不会改变原始条目。

  • 自定义配置文件以在特定环境中使用不会阻止您的应用程序在不同的环境中运行。您可以多次自定义配置文件以在多个环境中使用，并且这些自定义不会相互干扰。

自定义错误和状态消息

定制器线束以与SQLJ转换器大致相同的方式输出错误和状态消息，并将它们输出到同一输出设备。但是，没有关于定制的警告是可以抑制的。

自定义程序工具栏报告的错误消息分为四类：

  • 未被承认或非法的选择

  • 连接实例化错误

  • 配置文件实例化错误

  • 定制程序实例化错误

自定义程序线束在自定义过程中报告的状态消息使您可以确定是否已成功自定义配置文件。它们分为三类：

  • 档案修改状态

  • JAR文件修改状态

  • 创建的备份文件的名称（如果启用了自定义程序线束backup选项）

特定定制程序的customize()方法可能会报告其他特定于定制程序的错误和警告。

在自定义过程中，配置文件自定义程序会将消息写入其错误日志，并且定制程序工具会实时读取日志内容，并将这些消息与任何其他线束输出一起输出到SQLJ输出设备。您永远不必直接访问错误日志内容。

运行时自定义配置文件的功能

自定义配置文件是与其关联的连接上下文类的静态成员。对于应用程序中的每个SQLJ语句，SQLJ运行时确定与该语句关联的连接上下文类和实例，然后使用连接上下文类的自定义配置文件以及特定连接上下文实例的基础JDBC连接来创建连接的个人资料 。此连接的配置文件是SQLJ运行时用于将特定于供应商的功能应用于SQLJ应用程序的执行的工具。

定制器线束选项概述

随Oracle SQLJ实现提供的定制程序工具提供了许多不特定于特定定制程序的选项。线束在自定义过程的前端协调中使用这些选项。

定制器线束选项的语法

SQLJ命令行上的定制程序线束选项设置具有以下语法：

-P- 选项 = 值

或者，在SQLJ属性文件中：

轮廓。 option = value

使用以下内容启用布尔选项（标志）：

-P- 选项

要么：

-P- 选项 = true

默认情况下禁用布尔选项，但您可以使用以下命令显式禁用它们：

-P- option = false
定制器线束支持的选项

定制器线束支持以下常规选项：

  • backup ：在自定义配置文件之前保存该配置文件的备份副本。

  • context ：将自定义限制为与列出的连接上下文类关联的配置文件。

  • customizer ：指定要使用的自定义程序。

  • digests ：为JAR文件清单指定摘要（仅在指定要自定义的JAR文件时才相关）。

  • help ：显示自定义程序选项（仅在SQLJ命令行中指定）。

  • verbose ：在自定义过程中显示状态消息。

自定义程序线束支持以下用于自定义程序数据库连接的选项。目前，如果为列定义启用其optcols选项（用于性能优化），Oracle自定义程序将使用这些选项。此外，如果您使用此专用定制程序对配置文件执行联机语义检查，则SQLCheckerCustomizer将使用它们。

  • user ：指定此自定义中使用的连接的用户名。

  • password ：指定此自定义中使用的连接的密码。

  • url ：指定此自定义中使用的连接的URL。

  • driver ：为此自定义中使用的连接指定JDBC驱动程序。

有关Oracle customizer optcols标志的信息，请参阅“ Oracle Customizer列定义选项（optcols） ” 。有关SQLCheckerCustomizer信息，请参阅“ 用于配置文件语义检查的SQLCheckerCustomizer ” 。

以下命令用作定制程序线束选项，但是通过Oracle SQLJ实现提供的专用定制程序实现。

  • debug ：将调试信息插入指定的配置文件，以便在运行时输出。这是调用Oracle SQLJ AuditorInstaller的快捷方式， “ AuditorInstaller Customizer for Debugging ”中对此进行了描述。

  • print ：以文本格式输出指定配置文件的内容。

  • verify ：对先前执行SQLJ转换器期间生成的配置文件执行语义检查（相当于在转换期间对源代码执行的语义检查）。这是调用Oracle SQLJ SQLCheckerCustomizer的快捷方式， “ SQLCheckerCustomizer for Profile Semantics-Checking ”中对此进行了描述。

常规定制器线束选项

配置文件备份选项（备份）

在替换原始文件之前，使用backup标志指示线束保存每个.jar文件和独立.ser文件的备份副本。（ .ser在.jar文件中单独备份.ser文件。）

备份文件名的扩展名为.bak n ，其中n表示在具有类似命名文件的情况下根据需要使用的数字。对于创建的每个备份文件，将发出信息性消息。

如果在自定义独立.ser文件期间发生错误，则不会替换原始.ser文件，也不会创建备份。同样，如果在自定义JAR文件中的任何.ser文件期间发生错误，则不会替换原始JAR文件，也不会创建备份。

此选项的命令行语法是：

-P-备份<=真|假>

命令行示例是：

P-备份

属性文件语法是：

profile.backup <=真|假>

默认值为：

假
自定义连接上下文选项（上下文）

使用context选项将自定义限制为与指定的连接上下文类对应的概要文件。完全限定类名并使用逗号分隔列表指定多个类。例如：

-P-上下文= sqlj.runtime.ref。的DefaultContext，foo.bar。MyCtxtClass

逗号两边必须没有空格。

如果未指定此选项，则会自定义所有配置文件，而不管其关联的连接上下文类。

命令行语法是：

-P-上下文= ctx_class1 <，ctx_class2，...>

命令行示例是：

-P-上下文= foo.bar。MyCtxtClass

属性文件语法是：

profile.context = ctx_class1 <， ctx_class2 ，...>

属性文件示例是：

profile.context = foo.bar。MyCtxtClass
定制器选项（定制器）

使用customizer选项指定要使用的自定义程序。完全限定类名，例如以下示例：

-P-定制= oracle.sqlj.runtime.util。OraCustomizer

如果未设置此选项，则SQLJ将使用SQLJ -default-customizer选项中指定-default-customizer 。除非另有设定，否则以下内容如下：

oracle.sqlj.runtime.util。OraCustomizer

命令行语法是：

-P-customizer = customizer_class

命令行示例是：

-P-定制= ABCMyCustomizer

属性文件语法是：

profile.customizer = customizer_class

属性文件示例是：

profile.customizer = ABCMyCustomizer

默认值为：

没有
自定义JAR文件摘要选项（摘要）

在生成JAR文件时，JAR实用程序可以根据一个或多个指定的算法为每个条目选择性地包含一个或多个摘要 ，以便稍后验证JAR文件条目的完整性。对于熟悉这些摘要的读者来说，摘要在概念上类似于校验和。

如果要在JAR文件中自定义配置文件并希望JAR实用程序在更新JAR文件时添加新摘要（或更新现有摘要），请使用digests选项指定一个或多个算法的逗号分隔列表。这些是JAR实用程序在为每个条目创建摘要时将使用的算法。JAR实用程序为JAR清单文件中的每个JAR文件条目为每个算法生成一个摘要。指定算法如下：

-P-消化= SHA，MD5

逗号两边必须没有空格。

在此示例中，JAR清单文件中的每个条目将有两个摘要： SHA摘要和MD5摘要。

注意：

访问Sun站点有关JAR清单文件的更多信息。

有关JAR文件和JAR实用程序的信息，请参阅以下Web站点之一：

http://www.javasoft.com/products/jdk/1.2/docs/guide/jar/index.html

命令行语法是：

-P-digests = algo1 <， algo2 ，...>

命令行示例是：

-P-消化= SHA，MD5

属性文件语法是：

profile.digests = algo1 <， algo2 ，...>

属性文件示例是：

profile.digests = SHA，MD5

默认值为：

SHA，MD5

配置文件的JAR文件

               

                  
如前所述，您可以在SQLJ命令行上指定JAR文件，以便自定义JAR文件包含的任何概要文件。

                  

                     
注意：

                     

                     
    
                        
  • 
                           
    请记住，您可以在SQLJ命令行上指定.sqlj或.java文件或两者以进行标准SQLJ处理，或者您可以在命令行上指定.ser或.jar文件或两者，仅用于自定义，但不能同时指定两个类别。
                           
    
                        
    • 
                        
  • 
                           
    .jar文件允许包含非配置文件的文件。在自定义过程中，将忽略其清单条目指示文件不是配置文件的任何文件。
                           
    
                        
    • 
                        
  • 
                           
    .jar文件用作其包含的每个配置文件的类加载上下文。如果概要文件包含对.jar文件中包含的类的引用，则该类将从.jar文件加载。如果概要文件包含对不在.jar文件中的类的引用，则系统类加载器将像往常一样根据类路径查找并加载类。
                           
    
                        
    • 
                     

                  

               
JAR文件要求

                  

                     
每个配置文件的清单条目都有要求。

                     
为将包含在JAR文件中的每个配置文件创建一个包含两行的纯文本文件。一行以“Name：”开头，后跟路径或包和名称。另一行如下：
SQLJProfile：TRUE
这两行必须是连续的（中间没有空行），并且在行对之前必须有一个空行用于其他配置文件。

                     
使用JAR实用程序-m选项输入此文件。
                     

                     
例如，假设您的MyApp应用程序（在目录foo/bar ）有三个配置文件，您将创建一个包含这些配置文件的JAR文件。完成以下步骤：

                     
    
                        
  1. 
                           
    创建一个包含以下八行的文本文件（包括用作分隔符的空行）：
    名称：foo / bar / MyApp_SJProfile0.ser SQLJProfile：TRUE名称：foo / bar / MyApp_SJProfile1.ser SQLJProfile：TRUE名称：foo / bar / MyApp_SJProfile2.ser SQLJProfile：TRUE
    假设您将此文件MyAppJarEntries.txt 。
                           
    
                        
    2. 
                        
  2. 
                           
    当您运行jar来创建JAR文件时，使用-m选项输入您的文本文件，如下所示（假设您要调用JAR文件myjarfile.jar ）：
    
                        
    3. 
                     
％jar -cvfm myjarfile.jar MyAppJarEntries.txt foo / bar / MyApp_SJProfile * .ser foo / bar / * .class
当JAR实用程序在创建JAR文件期间构造清单时，它会读取您的文本文件并将SQLJProfile: TRUE行插入每个配置文件的清单条目中。它通过将清单中的名称与您在文本文件中指定的名称进行匹配来实现此目的。
                     

                  

               
JAR文件物流

                  

                     
在SQLJ命令行上指定JAR文件时，JAR文件中的每个配置文件都将反序列化并自定义。

                     
仅当成功自定义其包含的所有配置文件时，才能成功自定义JAR文件。一个成功的定制完成后，每个资料已被reserialized成.ser文件，JAR文件已被修改，以取代原来的.ser使用自定义文件.ser文件和JAR文件清单已经被更新，以表明新的条目。
                     

                     
如果在JAR文件中的任何配置文件的自定义中遇到任何错误，则JAR文件自定义失败，并且原始JAR文件保持完全不变。

                     

                        
注意：

                        
如果使用签名文件进行身份验证，则原始JAR文件中出现的签名文件将在更新的JAR文件中保持不变。如果配置文件需要签名，您有责任重新签名新的JAR文件。

用于调试的AuditorInstaller定制器

               

                  
对于ISO代码生成，SQLJ提供了一个特殊的自定义程序AuditorInstaller 。此定制程序将在SQLJ命令行中指定的概要文件中插入称为审计程序的调试语句集。这些配置文件必须已存在于以前的自定义中。
                  

                  
调试语句将在SQLJ运行时（当有人运行您的应用程序时）执行，显示一系列方法调用和返回的值。

                  
使用自定义程序线束debug选项（前缀为-P-与任何常规自定义选项一样）来插入调试语句。（ “ 使用Customizer线束调试选项调用AuditorInstaller ”中讨论了此选项的语法。）
                  

               
审计师和代码层概述

                  

                     
自定义应用程序时，Oracle自定义程序会针对不同级别的运行时功能以代码层 （通常少于五个）实现配置文件。最深层使用直接的Oracle JDBC调用，并实现可以通过JDBC功能执行的任何SQLJ语句。每个更高层是JDBC不支持的某些类SQLJ功能的专用层，因此必须由SQLJ运行时特别处理。例如，迭代器转换语句（ CAST ）层用于将JDBC结果集转换为SQLJ迭代器。另一层用于赋值语句（ SET ）。
                     

                     
在运行时，每个SQLJ可执行语句首先传递到最浅层，然后逐层传递，直到它到达可以处理它的层（通常是最深层，执行所有JDBC调用）。

                     
在AuditorInstaller的单次执行期间，您只能在一个层上安装调试语句。安装在特定代码层的每组调试语句称为单独的审计员 。在运行期间，只要将呼叫传递到安装审核员的层，就会激活审核员。
                     

                     
在调试期间，JDBC层上方的任何一个专用代码层通常都没有特别的意义，因此通常在最深层或最浅层安装审计员。如果在最浅层安装审计员，则其运行时调试输出将是由所有SQLJ可执行语句产生的方法调用的跟踪。如果在最深层安装审计员，则其运行时输出将是来自导致JDBC调用的所有SQLJ可执行语句的方法调用的跟踪。

                     
使用AuditorInstaller多个执行来安装不同级别的审计员。例如，您可能希望在最深层和最浅层安装审计员。
                     

                     
有关如何指定安装审核员的层的信息，请参阅“ AuditorInstaller深度选项（深度） ” 。

AuditorInstaller深度选项（深度）

                     

                        
如“ 审计员和代码层概述 ”中所述 ， AuditorInstaller可以在任何一次执行期间仅在单个代码层安装一组调试语句（称为审计员）。AuditorInstaller depth选项使您可以指定哪个图层。使用AuditorInstaller多个执行来安装不同级别的审计员。
                        

                        
图层以整数编号。最浅的深度是0层;最大深度为2或3是典型的。通常使用的唯一深度设置是最浅层的0或最深层的-1。实际上，很难在任何其他特定层安装审计员，因为用于各种SQLJ可执行语句的层编号不会被公开。

                        
depth选项有时与prefix选项一起使用。通过不止一次运行AuditorInstaller ，使用不同层的不同前缀，您可以在运行时看到哪些信息来自哪些层。
                        

                        
如果未设置depth选项，或者规范超出给定配置文件中的层数，则将在最深层安装审核员。
                        

                        
命令行语法是：
-P-Cdepth = n

命令行示例是：
-P-Cdepth = 0
属性文件语法是：
轮廓。Cdepth = n

属性文件示例是：
轮廓。Cdepth = 0
默认值为：
-1（最深层）

                  
AuditorInstaller日志文件选项（日志）

-P-Clog = log_file

空（用于标准输出）

完整的命令行示例

                  

                     
以下是一些完整的SQLJ命令行示例，显示了AuditorInstaller选项的规范。
                     

                     
将一组调试语句或审计器插入最深层（这是默认层），运行时输出到标准输出：
％sqlj -P-debug MyApp_SJProfile * .ser
将审计员插入最深层，运行时输出到log.txt ：
％sqlj -P-debug -P-Clog = foo / bar / log.txt MyApp_SJProfile * .ser
将审计员插入最深层，运行时输出到标准输出，显示线程名称但不返回参数：
％sqlj -P-debug -P-CshowThreads = true -P-CshowReturns = false MyApp_SJProfile * .ser
将审核员插入第0层（最浅层）。将运行时输出发送到log.txt ;使用“ Layer 0: ”为每行运行时输出添加前缀（以下命令是单个环绕行）：
％sqlj -P-debug -P-Clog = foo / bar / log.txt -P-Cdepth = 0 -P-Cprefix =“Layer 0：”MyApp_SJProfile * .ser
卸载审核员（这将卸载最近安装的审核员;重复执行以卸载所有审核员）：
％sqlj -P-debug -P-Cuninstall MyApp_SJProfile * .ser
