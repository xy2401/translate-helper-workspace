264 UTL_RECOMP

UTL_RECOMP包按顺序或并行重新编译数据库中的无效PL / SQL模块，无效视图，Java类，索引类型和运算符。

本章包含以下主题：

  • 概观

  • 操作说明

  • 例子

  • UTL_RECOMP子程序摘要

264.1 UTL_RECOMP概述

在主要版本升级（通常使所有PL / SQL和Java对象无效）之后，此脚本特别有用。尽管在使用时会自动重新编译无效对象，但在操作之前运行此脚本非常有用，因为这将消除或最小化由于运行时按需自动重新编译而导致的后续延迟。

并行重新编译可以利用多个CPU来减少重新编译无效对象所需的时间。并行度由RECOMP_PARALLEL过程的第一个参数指定。

通常，每个可用CPU的一个线程的并行设置提供了良好的初始设置。但请注意，重新编译无效对象的过程会将大量数据写入系统表，并且相当I / O密集。慢速磁盘系统可能是一个重要的瓶颈，并且可以从更高程度的并行性限制加速。

264.2 UTL_RECOMP操作说明

UTL_RECOMP有几个操作说明。

  • 此程序包使用作业队列进行并行重新编译。

  • 必须使用SQL*PLUS运行此程序包。

  • 您必须连接AS SYSDBA才能运行此脚本。

  • 此程序包需要使用VALID状态创建以下程序包：

    • STANDARD （ standard sql ）

    • DBMS_STANDARD （ dbmsstdx 。 sql ）

    • DBMS_JOB （ dbmsjob 。 sql ）

    • DBMS_RANDOM（dbmsrand.sql）

  • 在此程序包中运行条目时，数据库上不应有其他DDL。不遵循此建议可能会导致死锁。

264.3 UTL_RECOMP示例

这些示例显示了UTL_RECOMP可以重新编译对象的各种方法。

  • 按顺序重新编译所有对象：

    执行UTL_RECOMP.RECOMP_SERIAL（）;

  • 按顺序重新编译模式SCOTT对象：

    执行UTL_RECOMP.RECOMP_SERIAL（'SCOTT'）;

  • 使用4个并行线程重新编译所有对象：

    EXECUTE UTL_RECOMP.RECOMP_PARALLEL（4）;

  • 使用参数JOB_QUEUE_PROCESSES指定的线程数重新编译模式JOE中的对象：

    EXECUTE UTL_RECOMP.RECOMP_PARALLEL（NULL，'JOE'）;

264.4.1 RECOMP_PARALLEL过程

此过程使用DBA_Dependencies视图中公开的信息来DBA_Dependencies地重新编译数据库或给定模式中的无效对象。

句法

UTL_RECOMP.RECOMP_PARALLEL（线程IN PLS_INTEGER DEFAULT NULL，架构IN VARCHAR2 DEFAULT NULL，标志IN PLS_INTEGER DEFAULT 0）;

参数

表264-2 RECOMP_PARALLEL过程参数

参数 描述

线程

并行运行的重新编译线程数。如果为NULL ，则使用' job_queue_processes '的值。

模式

要重新编译无效对象的模式。如果为NULL ，则重新编译数据库中的所有无效对象。

旗

标志值仅用于内部测试和诊断。

使用说明

并行重新编译利用多个CPU来减少重新编译无效对象所花费的时间。但请注意，重新编译会将大量数据写入系统表，因此磁盘系统可能成为瓶颈并阻止显着的加速。
