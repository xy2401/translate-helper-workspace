用户帐户

每个Oracle数据库都有一个有效数据库用户列表。

该数据库包含多个默认帐户，包括默认管理帐户SYSTEM 。您可以根据需要创建用户帐户。您还可以配置应用程序用户以访问Oracle数据库。

要访问数据库，用户必须提供有效的用户名和身份验证凭据。凭证可以是密码，Kerberos票证或公钥基础结构（PKI）证书。您可以配置数据库安全性以根据失败的登录尝试锁定帐户。

通常， 数据库访问控制涉及限制数据访问和数据库活动。例如，您可以限制用户查询指定的表或执行指定的数据库语句。

特权

用户权限是运行特定SQL语句的权限 。

权限分为以下几类：

  • 系统特权

    这是在数据库中执行特定操作或对特定类型的任何对象执行操作的权限。例如， CREATE USER和CREATE SESSION是系统特权。

  • 对象权限

    这是对对象执行特定操作的权限，例如，查询employees表。权限类型由数据库定义。

特权由其他用户自行决定授予用户。管理员应该为用户授予权限，以便他们可以完成工作所需的任务。良好的安全性实践涉及仅向需要该特权才能完成必要工作的用户授予特权。

特权分析

特权分析机制根据指定的条件捕获数据库的特权使用。

通过这种方式，您可以捕获运行应用程序模块或执行特定SQL语句所需的权限。例如，您可以找到用户在特定数据库会话期间执行的权限。

在生产数据库中，权限和角色，角色和角色以及角色和用户之间的关系可能很复杂。通过权限分析，您可以识别在复杂系统中不必要地授予的权限。根据对捕获结果的分析，您可以删除不必要的授权或重新配置权限授予，以使数据库更安全。

用户个人资料

在系统资源的上下文中， 用户配置文件是一组命名的资源限制和密码参数，用于限制用户的数据库使用和数据库实例资源。

配置文件可以限制用户的并发会话数，每个会话可用的CPU处理时间以及可用的逻辑I / O量。例如， clerk简档可以将用户限制为文书任务所需的系统资源。

注意：

最好使用数据库资源管理器来限制资源并使用配置文件来管理密码。

配置文件为共享一组属性的用户提供单一参考点。您可以将配置文件分配给一组用户，将默认配置文件分配给所有其他用户。每个用户在任何时间点最多只能分配一个配置文件。

数据库认证

在Oracle数据库中， 数据库身份验证是用户向数据库提供凭据的过程，该数据库验证凭据并允许访问数据库。

验证身份为进一步的交互建立了信任关系。身份验证还可以通过将访问和操作链接到特定身份来实现问责制。

Oracle数据库提供了不同的身份验证方法，包括：

  • 由数据库验证

    Oracle数据库可以使用密码，Kerberos票证或PKI证书对用户进行身份验证。Oracle还支持符合RADIUS标准的设备，用于其他形式的身份验证，包括生物识别。在Oracle数据库中创建用户时，必须指定身份验证类型。

  • 由操作系统进行身份验证

    某些操作系统允许Oracle数据库使用他们维护的信息来对用户进行身份验证。在通过操作系统进行身份验证后，用户可以在不指定用户名或密码的情况下连接到数据库。

非管理数据库用户帐户不得执行数据库操作，如关闭或启动数据库。这些操作需要SYSDBA ， SYSOPER ， SYSBACKUP或SYSDG权限。

加密

Oracle数据库加密是使用密钥和加密算法将数据转换为不可读格式的过程。

加密通常用于满足法规遵从性要求，例如与支付卡行业数据安全标准（PCI-DSS）或违规通知法律相关的要求。例如，必须加密信用卡号，社会安全号或患者健康信息。

网络加密

在客户端和服务器之间通过网络传输时加密数据称为网络加密 。

入侵者可以使用网络数据包嗅探器捕获网络上传输的信息，然后将其假脱机到文件以供恶意使用。加密网络上的数据可防止此类活动。

透明数据加密

Oracle Advanced Security 透明数据加密使您可以加密单个表列或表空间。

当用户将数据插入加密列时，数据库会自动加密数据。当用户选择列时，数据将被解密。这种加密形式是透明的，提供高性能，并且易于实现。

透明数据加密包括行业标准加密算法，如高级加密标准（AES）和内置密钥管理。

Oracle数据编辑

Oracle Data Redaction是Oracle Advanced Security的一部分，使您可以屏蔽（编辑）低权限用户或应用程序查询的数据。当用户查询数据时，编辑实时发生。

数据编辑支持以下编校功能类型：

  • 完整的数据编辑

    在这种情况下，数据库会在表或视图中编辑指定列的全部内容。例如，姓氏的VARCHAR2列显示单个空格。

  • 部分数据编辑

    在这种情况下，数据库会编辑显示输出的部分内容。例如，应用程序可以将以1234结尾的信用卡号码呈现为xxxx-xxxx-xxxx-1234 。您可以将正则表达式用于完整和部分编校。正则表达式可以基于搜索模式编辑数据。例如，您可以使用正则表达式来编辑特定的电话号码或电子邮件地址。

  • 随机数据编辑

    在这种情况下，数据库将数据显示为随机生成的值，具体取决于列的数据类型。例如，数字1234567可以显示为83933895 。

数据编辑不是一个全面的安全解决方案。例如，它不会阻止直接连接的特权用户对编辑数据执行推理攻击。此类攻击识别编辑的列，并通过消除过程，尝试通过重复猜测存储值的SQL查询来返回实际数据。为了检测和防止特权用户的推理和其他攻击，Oracle建议将Oracle Data Redaction与相关的数据库安全产品（如Oracle Audit Vault和Database Firewall以及Oracle Database Vault）配对。

数据编辑的工作原理如下：

  • 使用DBMS_REDACT包为指定的表创建编校策略。

  • 在策略中，指定预定义的编校功能。

  • 数据库是否显示列的实际值或编辑值取决于策略。如果数据被编辑，则在显示给用户之前立即在顶级选择列表中进行编校。

以下示例添加完整数据编校策略以编辑hr.employees表的员工ID（ employee_id ）列：

BEGIN DBMS_REDACT.ADD_POLICY（object_schema =>'hr'，object_name =>'employees'，column_name =>'employee_id'，policy_name =>'mask_emp_ids'，function_type => DBMS_REDACT.FULL，expression =>'1 = 1'）;结束; /

在前面的示例中，表达式设置（计算结果为true ）将编校应用于未授予EXEMPT REDACTION POLICY权限的用户。

取向

Oracle数据库提供了许多控制数据访问的技术。本节总结了其中一些技术。

Oracle Database Vault

Oracle Database Vault限制特权用户对应用程序数据的访问。

从Oracle Database 12c开始 ，Oracle Database Vault扩展了标准数据库审计数据结构。此外，如果迁移到统一审计，则数据库会将审计记录写入Oracle Secure Files中的统一审计跟踪，从而集中Oracle数据库的审计记录。

您可以使用Oracle Database Vault来控制访问数据库，数据和应用程序的时间，地点和方式。因此，您可以解决常见的安全问题，例如防范内部威胁，遵守法规要求以及强制执行职责分离。

要使Oracle Database Vault管理员负责，数据库必须审核对Oracle Database Vault元数据所做的配置更改。这些更改包括创建，修改和删除任何与Oracle Database Vault相关的强制实施，授予和撤销受保护角色，以及对Oracle Data Pump和Job Scheduler等组件的授权。

虚拟专用数据库（VPD）

Oracle虚拟专用数据库（VPD）使您可以在行和列级别强制实施安全性。

安全策略建立了用于保护数据库免于意外或恶意破坏数据或损坏数据库基础结构的方法。

当特权和角色等安全保护不够细致时，VPD非常有用。例如，您可以允许所有用户访问employees表，但可以创建安全策略以限制与用户在同一部门中对员工的访问。

实质上，数据库将动态WHERE子句添加到针对应用Oracle VPD安全策略的表， 视图或同义词发出的SQL语句。WHERE子句仅允许其凭据通过安全策略的用户访问受保护的数据。

Oracle Label Security（OLS）

Oracle Label Security（OLS）使您可以使用安全标签分配数据分类和控制访问权限。您可以为数据或用户分配标签。

分配给数据时，标签可以作为隐藏列附加到表，从而为SQL提供透明性。例如，您可以将包含高度敏感数据的行标记为HIGHLY SENSITIVE并将不敏感的行标记为SENSITIVE 。当用户尝试访问数据时，OLS会将用户标签与数据标签进行比较，并确定是否授予访问权限。与VPD不同，OLS提供了开箱即用的安全策略和用于定义和存储标签的元数据存储库。

如果启用了统一审核，则数据库会提供基于策略的框架来配置和管理审核选项。您可以为不同类型的操作（包括OLS操作）分组审核选项，并将其另存为审核策略。然后，您可以启用或禁用该策略以强制执行基础审核选项。

无论何时创建OLS策略，数据库都会将策略的标签列添加到数据库审计跟踪表中。OLS审计可以将审计记录（包括OLS管理员操作的记录）写入统一审计跟踪。

数据访问监控

Oracle数据库提供了多种用于监控用户活动的工具和技术。审计是监视数据访问的主要机制。

数据库审计

数据库审计是对所选用户数据库操作的监视和记录。

您可以配置统一审核策略以审核以下内容：

  • SQL语句，系统特权，模式对象和角色（作为直接授予它们的一组系统特权）

  • 管理和非管理用户

  • 应用上下文值

    应用程序上下文是指定命名空间中的属性名称 - 值对。应用程序在对数据库执行操作之前设置各种上下文。例如，应用程序存储指示应用程序事件状态的模块名称和客户机ID等信息。应用程序可以配置上下文，以便将有关它们的信息附加到审计记录中。

  • Real Application Security，Oracle Database Vault，Oracle Label Security，Oracle Data Pump和Oracle SQL * Loader直接路径事件的策略创建

统一审计跟踪可以捕获Recovery Manager事件，您可以在UNIFIED_AUDIT_TRAIL数据字典视图中查询这些事件。您不为Recovery Manager事件创建统一的审核策略。

您还可以使用细粒度审计来审计特定的表列，并在策略创建期间关联事件处理程序。对于统一和细粒度审计，您可以创建策略来测试捕获表上特定数据库操作或活动发生时间的条件。例如，您可以审核在晚上9点之后访问的表

审核的原因包括：

  • 为将来的行动启用问责制

  • 根据问责制阻止用户（或其他人，如入侵者）采取不当行为

  • 调查，监控和记录可疑活动

  • 满足合规性的审计要求

从Oracle Database 12c开始 ，当您使用统一审计时，默认情况下会启用数据库审计。您可以通过启用审核策略来控制数据库审核。但是，在使用统一审计之前，必须将数据库迁移到该审计。

审计政策

您可以使用单个SQL语句创建指定一组审计选项的命名统一审计策略。这些选项可以指定要在数据库内进行审计的系统特权，操作或角色。

在审核策略中，您可以选择设置可以针对每个语句评估的条件，针对会话设置一次，或针对数据库实例设置一次。对事件的审核取决于对适用审核策略的条件进行评估的结果。如果条件计算结果为true ，则数据库将生成审核记录。

以下示例创建一个策略，用于审核hr.employees表上的活动，除非用户从可信终端term1和term2登录：

创建审计策略EmployeesTableAudit ACTIONS更新ON hr.employees，删除ON hr.employees WHEN SYS_CONTEXT（“userenv”，“hostname”）NOT IN（“term1”，“term2”）评估每个会话;

以下语句为用户hr和hrvp启用策略：

审计政策EmployeesTableAudit BY hr，hrvp;

您可以将统一审核策略应用于任何数据库用户，包括管理用户，如SYSDBA ， SYSOPER等。但是，只能在使用ALTER DATABASE OPEN语句打开数据库后读取审核策略。因此，在数据库打开之前，始终会审核管理用户的顶级操作。数据库打开后，审核策略配置生效。

启用统一审核后，数据库会自动审核审核设置的更改。数据库还审核数据库实例启动和关闭。

审核管理员角色

要执行审核，必须为您授予适当的系统权限。

Oracle数据库提供以下系统提供的审计管理员角色：

  • AUDIT_ADMIN

    AUDIT_ADMIN角色管理数据库的审核设置。具有此角色的用户有权执行以下操作：

    • 创建，更改和删除审核策略，包括细粒度的审核策略

    • 为每个业务需求启用或禁用审核策略

    • 查看审核记录

    • 管理和清理审计跟踪

  • AUDIT_VIEWER

    AUDIT_VIEWER角色适用于只需要查看和分析数据的用户。具有此角色的用户仅有权查看审计跟踪内容。

统一审计跟踪

审计记录对于检测和识别未经授权的数据访问至关重要。

Oracle数据库可以为指定的事件配置审核。如果在用户会话期间发生事件，则数据库将生成审核记录。

审计跟踪是存储审计记录的位置。统一审计跟踪是Oracle Database 12c中的新增功能，它为来自所有类型的审计的审计记录提供统一存储。您必须手动从先前版本的传统审计跟踪迁移到统一审计。

审计包括标准和细粒度审计，还包括审计以下事件，包括从管理用户执行这些事件：

  • Oracle Data Pump

  • SQL * Loader直接路径加载

  • Oracle Database Vault

  • Oracle Label Security

  • 恢复管理器

  • 真实应用安全

统一审计跟踪是只读的，存储在AUDSYS模式中。默认情况下， SYSAUX表空间存储来自所有源的审核记录。您可以使用DBMS_AUDIT_MGMT包提供新的表空间。

UNIFIED_AUDIT_TRAIL视图从审计跟踪中检索审计记录，并以表格形式显示它们。APPLICATION_CONTEXTS列存储已配置的应用程序上下文属性的值。您可以使用AUDIT语句在审计记录中包含上下文属性的值。例如，以下语句从userenv命名空间捕获MODULE和CLIENT_INFO属性：

审核上下文NAMESPACE userenv ATTRIBUTES MODULE，CLIENT_INFO by hr;

根据审计的组件（例如Oracle Database Vault），可以使用其他与统一审计跟踪相关的视图。

高可用性概述

可用性是指按需提供应用程序，服务或功能的程度。

例如，在线书商使用的OLTP数据库可供购买的客户访问。可靠性，可恢复性，及时错误检测和连续操作是高可用性的主要特征。

数据库环境中高可用性的重要性与停机成本相关，停机成本是资源不可用的时间。停机时间可分为计划内或计划外。设计高可用性环境时的主要挑战是检查停机时间的所有可能原因并制定处理它们的计划。

网站失败

当事件导致应用程序的全部或大部分停止处理或减慢到无法使用的服务级别时，会发生站点故障 。

站点故障可能会影响数据中心的所有处理或数据中心支持的应用程序子集。示例包括扩展的站点范围的电源或网络故障，使数据中心无法运行的自然灾害，或对操作或站点的恶意攻击。

防止站点故障的最简单形式是使用RMAN创建数据库备份并将其存储在异地。您可以将数据库还原到其他主机。但是，这种技术可能很耗时，而备份可能不是最新的。通过在Data Guard环境中维护一个或多个备用数据库，可以在生产站点出现故障时提供持续的数据库服务。

数据腐败

当的硬件，软件，或网络组件导致要读取或写入的损坏的数据时，会发生数据损坏 。

数据损坏的一个示例是卷管理器错误导致磁盘读取或写入错误。数据损坏很少见，但可能对数据库产生灾难性影响，因此也会对业务造成灾难性影响。

除Data Guard和Recovery Manager之外，Oracle数据库还支持以下形式的数据损坏防护：

  • 丢失写入问题的解决方案

    当I / O子系统在未发生写入时确认数据块完成时，会发生丢失写入 。在随后的块读取中，I / O子系统返回数据块的陈旧版本，该版本可能用于更新数据库的其他块，从而破坏它。Oracle数据库解决方案如下：

    • 使用备用数据库丢失写保护

      在Oracle Database 11g中引入的标准丢失写保护中，您可以在主数据库和备用数据库上启用DB_LOST_WRITE_PROTECT初始化参数。每个数据库在联机重做日志中记录缓冲区缓存块读取。当备用数据库在托管恢复期间应用重做时，它会读取相应的块并将SCN与重做日志中的SCN进行比较，从而检测出差异。

    • 使用影子表空间丢失写保护

      对于阴影丢失写保护 ，您可以创建一个阴影表空间 。影子表空间包含跟踪数据文件中每个数据块的简短描述记录，包括SCN。

      如果启用了阴影丢失写保护（ ALTER DATABASE ENABLE LOST WRITE TRACKING ），并且数据库更新了跟踪的数据块，则数据库会将SCN写入相应的影子表空间。读取跟踪数据块时，数据库会查找阴影条目，然后将其与跟踪块进行比较。如果影子条目的SCN大于跟踪块，则发生丢失写入。

      阴影丢失写保护具有在不使用备用数据库的情况下检测丢失写入的优点。此外，由于待机丢失写保护中固有的延迟，当检测到丢失写入时，该块可能已经损坏了数据库的其他部分。为防止数据损坏，影子丢失写保护会在消耗之前检测到丢失的写入。

  • 数据块损坏检测

    块损坏是一种数据块，它不是可识别的Oracle格式，或者其内容不是内部一致的。几个数据库组件和实用程序（包括RMAN）可以检测损坏的块并将其记录在V$DATABASE_BLOCK_CORRUPTION 。如果环境使用Active Data Guard备用数据库，则可以自动修复损坏。

  • 数据恢复顾问

    Data Recovery Advisor是一个Oracle工具，可自动诊断数据故障，确定并提供适当的修复选项，并根据用户的请求执行修复。

  • 交易保护和应用程序连续性

    数据库会话中断，无论是计划内还是计划外中断，都会使最终用户不确定其工作状态。在某些情况下，用户可以重新提交已提交的事务，从而导致逻辑数据损坏。Transaction Guard提供事务idempotence ，使数据库能够保留指示事务是否已提交和完成的保证提交结果。Application Continuity（包括Transaction Guard）使应用程序能够在发生可恢复的错误后重放数据库中的事务，并在事务停止的地方继续。

人为错误

当提交无意或恶意操作导致数据库中的数据逻辑损坏或无法使用时，会发生人为错误中断 。根据受影响数据的数量和关键性质，人为错误中断的服务级别影响可能会有很大差异。

许多研究都将人为错误列为停机的最大原因。Oracle数据库提供了强大的工具，可帮助管理员快速诊断并从这些错误中恢复。它还包括使最终用户无需管理员参与即可从问题中恢复的功能。

Oracle数据库建议采用以下形式的防范人为错误：

  • 限制用户访问

    防止错误的最佳方法是限制用户对数据和服务的访问。Oracle数据库提供了广泛的安全工具，通过对用户进行身份验证来控制用户对应用程序数据的访问，然后允许管理员仅向用户授予执行其职责所需的权限。

  • Oracle闪回技术

    Oracle闪回技术是Oracle数据库中的一系列人为错误纠正功能。Oracle Flashback提供了一个SQL接口，可以快速分析和修复人为错误。例如，您可以执行：

    • 细粒度手术分析和修复局部损伤

    • 快速纠正更广泛的伤害

    • 在行，事务，表，表空间和数据库级别进行恢复

  • Oracle LogMiner

    Oracle LogMiner是一个关系工具，使您可以使用SQL来读取，分析和解释在线文件。

高可用性和计划停机时间

计划停机可能会对运营造成破坏，特别是在支持多个时区用户的全球企业中。在这种情况下，设计一个系统以最大限度地减少计划中断（例如日常操作，定期维护和新部署）非常重要。

计划停机时间可按其原因分类：

  • 系统和数据库更改

  • 数据变更

  • 申请变更

系统和数据库更改

执行例行和定期维护操作和新部署时会发生计划的系统更改，包括在数据库中组织数据结构之外发生的对操作环境的计划更改。

示例包括添加或删除CPU和群集节点（ 节点是数据库实例所在的计算机），升级系统硬件或软件以及迁移系统平台。

Oracle数据库提供动态资源配置，作为计划系统和数据库更改的解决方案：

  • 动态重新配置数据库

    Oracle数据库动态适应硬件和数据库配置的各种更改，包括从SMP服务器添加和删除处理器以及使用Oracle ASM添加和删除存储阵列。例如，Oracle数据库监视操作系统以检测CPU数量的变化。如果将CPU_COUNT初始化参数设置为缺省值，则数据库工作负载可以动态地利用新添加的处理器。

  • 自动调整内存管理

    Oracle数据库使用非中心化策略来释放和获取SGA和PGA的每个子组件中的内存。 Oracle数据库通过提示操作系统将内存颗粒传输到需要它的组件来自动调整内存。

  • 自动分发数据文件，控制文件和联机重做日志文件

    Oracle ASM通过在所有可用磁盘上自动分发数据文件，控制文件和日志文件来自动化和简化它们的布局。

可扩展性

在数据库服务器网格中，Oracle RAC允许您随着容量需求的增加将节点添加到群集。

容错

在高可用性体系结构中， 容错是针对体系结构中组件故障提供的保护。

Oracle RAC架构的一个关键优势是多个节点提供的固有容错能力。由于物理节点独立运行，因此一个或多个节点的故障不会影响群集中的其他节点。

故障转移可能发生在网格上的任何节点上。在极端情况下，Oracle RAC系统即使在除一个节点之外的所有节点都关闭时也提供数据库访问。此体系结构允许一组节点透明地联机或脱机，以进行维护，而群集的其余部分继续提供数据库访问。

Oracle RAC提供与Oracle客户端和连接池的内置集成。使用此功能，可以立即通过终止连接的池通知应用程序任何故障。应用程序避免等待TCP超时，并可立即采取适当的恢复操作。Oracle RAC将侦听器与Oracle客户端和连接池集成在一起，以创建最佳的应用程序吞吐量。Oracle RAC可以根据事务处理时的负载来平衡群集工作负载。

数据仓库和商业智能概述

数据仓库是一个关系数据库，专为查询和分析而设计，而不是用于事务处理。

例如，数据仓库可以跟踪历史股票价格或所得税记录。仓库通常包含从历史交易数据派生的数据，但它可以包括来自其他来源的数据。

除关系数据库外，数据仓库环境还包括多个工具。典型环境包括ETL解决方案， OLAP引擎，客户端分析工具以及收集数据并将其交付给用户的其他应用程序。

提取，转换和加载概述（ETL）

从源系统提取数据并将其带入仓库的过程通常称为ETL：提取，转换和加载。ETL指的是一个广泛的过程，而不是三个明确定义的步骤。

在典型的场景中，提取来自一个或多个操作系统的数据，然后物理地传输到目标系统或中间系统以进行处理。根据运输方法，在此过程中可能会发生一些转换。例如，通过网关直接访问远程目标的SQL语句可以将两列作为SELECT语句的一部分进行连接。

Oracle数据库本身不是ETL工具。但是，Oracle数据库提供了ETL工具和自定义ETL解决方案可用的丰富功能集。Oracle数据库提供的ETL功能包括：

  • 可传输的表空间

    您可以在不同的计算机体系结构和操作系统之间传输表空间。可传输表空间是在两个Oracle数据库之间移动大量数据的最快方法。

  • 表功能

    表函数是用户定义的PL / SQL函数，它返回行的集合（嵌套表或varray）。表函数可以生成一组行作为输出，并可以接受一组行作为输入。表函数支持在PL / SQL，C或Java中实现的转换的流水线和并行执行，而无需中间登台表。

  • 外部表格

    外部表使外部数据能够直接并行地连接，而不需要首先将其加载到数据库中。因此，外部表使得能够通过转换阶段对加载阶段进行流水线操作。

  • 表压缩

    要减少磁盘使用和内存使用，可以以压缩格式存储表和分区表。使用表压缩通常可以实现更好的只读操作扩展和更快的查询执行。

OLAP

Oracle在线分析处理（ OLAP ）在跨多个维度分析数据时提供本机多维存储和快速响应时间。OLAP使分析人员能够在交互式会话期间快速获取复杂迭代查询的答案。

  • Oracle OLAP集成在数据库中，因此您可以使用标准的SQL管理，查询和报告工具。

Oracle OLAP提供简单的功能：一个数据库，标准管理和安全性，以及标准接口和开发工具。

Oracle Information Integration概述

随着组织的发展，能够在多个数据库和应用程序之间共享信息变得越来越重要。

共享信息的基本方法如下：

  • 合并

    您可以将信息合并到一个数据库中，从而无需进一步集成。Oracle RAC，网格计算，多租户架构和Oracle VPD使您可以将信息整合到单个数据库中。

  • 联邦

    您可以分发信息，并提供联合此信息的工具，使其看起来位于单个虚拟数据库中。

  • 分享

    您可以共享信息，从而可以在多个数据存储和应用程序中维护信息。

本节重点介绍用于联合和共享信息的Oracle解决方案。

联合访问

联合访问的基础是分布式环境 ，它是一个由彼此无缝通信的不同系统组成的网络。

环境中的每个系统都称为节点 。用户直接连接的系统称为本地系统 。该用户访问的其他系统是远程系统 。

分布式环境使应用程序能够从本地和远程系统访问和交换数据。可以同时访问和修改所有数据。

分布式SQL

分布式SQL同步访问和更新分布在多个数据库中的数据。Oracle分布式数据库系统对用户是透明的，使其显示为单个Oracle数据库。

分布式SQL包括分布式查询和分布式事务。Oracle分布式数据库体系结构提供查询和事务透明性。例如，标准DML语句就像在非分布式数据库环境中一样工作。此外，应用程序使用标准SQL语句COMMIT ， SAVEPOINT和ROLLBACK控制事务。

Oracle GoldenGate

Oracle GoldenGate是一种异步的，基于日志的实时数据复制产品。

Oracle GoldenGate在异构数据库，硬件和操作系统环境中实时移动大量事务数据，影响最小。它优化了实时信息访问和可用性，因为它：

  • 支持涉及Oracle数据库和非Oracle数据库的异构混合的复制

  • 保持关键任务系统的持续可用性，从而最大限度地减少计划维护期间的停机时间

  • 实现整个企业的实时数据集成

  • 在分片表中自动配置分片之间的双向复制

典型的环境包括捕获，泵和交付过程。每个进程都可以在大多数流行的操作系统和数据库上运行，包括Oracle数据库和非Oracle数据库。可以复制部分或全部数据。可以针对异构环境和不同的数据库模式操纵任何这些过程中的数据。

Oracle GoldenGate支持多主机复制，中心辐射部署，数据整合和数据转换。因此，Oracle GoldenGate使您能够确保关键系统全天候运行，并且相关数据在整个企业中分布，以优化决策制定。

Oracle数据库高级队列（AQ）

高级队列（AQ）是一个与Oracle数据库集成的功能强大且功能丰富的消息队列系统。

当组织具有必须彼此通信的不同系统时，消息传递环境可以提供在这些系统之间传输关键信息的标准，可靠方式。

示例用例是在总部的Oracle数据库中输入订单的业务。输入订单时，业务使用AQ将订单ID和订单日期发送到仓库中的数据库。这些消息提醒仓库员工有关订单的信息，以便他们可以填写并发货。

消息队列和出队

高级队列将用户消息存储在称为队列的抽象存储单元中。

排队是生产者将消息放入队列的过程。出列是消费者从队列中检索消息的过程。

支持显式出列允许开发人员使用XStream和Oracle GoldenGate可靠地交换消息。他们还可以通过利用Oracle GoldenGate的更改捕获和传播功能来通知应用程序更改。

Oracle数据库高级队列功能

Oracle数据库高级队列（AQ）支持消息队列系统的所有标准功能。

功能包括：

  • 异步应用程序集成

    Oracle Database AQ提供了几种排队消息的方法。捕获过程或同步捕获可以隐式捕获消息，或者应用程序和用户可以显式捕获消息。

  • 可扩展的集成架构

    许多应用程序与分布式hub-and-spoke模型集成，Oracle Database作为集线器。Oracle数据库上的分布式应用程序与同一集线器中的队列进行通信。多个应用程序共享同一队列，无需添加队列来支持其他应用程序。

  • 异构应用程序集成

    Oracle Database AQ为应用程序提供了Oracle类型系统的全部功能。它包括对标量数据类型的支持，具有继承的Oracle数据库对象类型，带有XML数据的附加运算符的XMLType以及ANYDATA 。

  • 遗留应用程序集成

    Oracle Messaging Gateway将Oracle数据库应用程序与其他消息队列系统（如Websphere MQ和Tibco）集成在一起。

  • 基于标准的API支持

    Oracle Database AQ支持行业标准API：SQL，JMS和SOAP。使用SQL进行的更改将作为消息自动捕获。
