13.13编译条款

compile子句显式重新编译已变为无效的存储单元，从而消除了隐式运行时重新编译的需要，并防止了相关的运行时编译错误和性能开销。

COMPILE子句可以出现在以下SQL语句中：

  • 更改功能声明

  • ALTER PACKAGE声明

  • ALTER PROCEDURE声明

  • ALTER LIBRARY声明

  • ALTER TYPE语句

  • ALTER TRIGGER声明

语义

compile_clause

COMPILE

重新编译PL / SQL单元，无论它是有效还是无效。PL / SQL单元可以是库，包，包规范，包体，触发器，过程，函数，类型，类型规范或类型主体。

首先，如果单元所依赖的任何对象无效，则数据库会重新编译它们。

数据库还会使依赖于单元的任何本地对象无效。

如果数据库成功重新编译单元，则该单元变为有效。否则，数据库将返回错误，并且单元仍然无效。您可以使用SQL * Plus命令SHOW ERRORS查看关联的编译器错误消息。

在重新编译期间，数据库将删除所有持久编译器开关设置，从会话中再次检索它们，并在编译后存储它们。要避免此过程，请指定REUSE SETTINGS子句。

DEBUG

与PLSQL_OPTIMIZE_LEVEL=1具有相同的效果 - PLSQL_OPTIMIZE_LEVEL=1 PL / SQL编译器生成并存储供PL / SQL调试器使用的代码。Oracle建议使用PLSQL_OPTIMIZE_LEVEL=1而不是DEBUG 。

包

（默认）重新编译包规范和（如果存在）包体，无论它们是有效还是无效。如所描述的包装说明书和体内导致失效和相关对象的重新编译的重新编译SPECIFICATION和BODY 。

PACKAGE限制

PACKAGE只有在编译包时才会出现。

规格

仅重新编译包或类型规范，无论它是有效还是无效。您可能需要重新编译包或类型规范，以在修改规范后检查编译错误。

重新编译规范时，数据库会使依赖于规范的任何本地对象失效，例如调用程序包中的过程或函数的过程。包装的主体也取决于其规格。如果您随后引用其中一个依赖对象而未先显式重新编译它，则数据库会在运行时隐式重新编译它。

对SPECIFICATION限制

SPECIFICATION如果编译包或类型规范可以仅出现。

身体

仅重新编译包或类型主体，无论它是有效还是无效。您可能希望在修改包后重新编译包或键入主体。重新编译主体不会使依赖于其规范的对象无效。

重新编译包或键入主体时，数据库首先重新编译主体所依赖的对象，如果这些对象中的任何一个无效。如果数据库成功重新编译主体，则主体变为有效。

对BODY限制

只有在编译包或类型主体时才会出现BODY 。

重新设定

防止Oracle数据库丢弃并重新获取编译器开关设置。使用此子句，Oracle会保留现有设置并将其用于重新编译此语句中未指定值的任何参数。

另请参见DEFAULT COLLATION子句编译语义。

compiler_parameters_clause

指定表1-2中 PL / SQL编译参数的值。每个参数的编译时值与正在编译的PL / SQL单元的元数据一起存储。

您可以在每个语句中仅指定一个参数一次。每个设置仅对正在编译的PL / SQL单元有效，不会影响此会话或系统中的其他编译。要影响整个会话或系统，必须使用ALTER SESSION或ALTER SYSTEM语句为参数设置值。

如果省略此子句中的任何参数并指定REUSE SETTINGS ，那么如果在此PL / SQL单元的早期编译中为参数指定了值，则数据库将使用该早期值。如果省略任何参数，并且未指定REUSE SETTINGS或未在早期编译中为参数指定任何值，则数据库将从会话环境中获取该参数的值。
