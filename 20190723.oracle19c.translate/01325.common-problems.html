10.1常见问题

初始配置后，Oracle数据库服务质量管理是一个自动化系统。因此，您可能遇到的大多数问题都与配置Oracle数据库QoS管理有关。

以下部分说明了最常见的问题以及解决方法：

  • 无法启用Oracle数据库服务质量管理
  • 无法为数据库启用Oracle数据库QoS管理
    对于由Oracle Database QoS Management管理的数据库，数据库必须符合，启用，并且Oracle Database QoS Management必须能够访问APPQOSSYS数据库用户：
  • 未启用Oracle数据库资源管理器和资源计划错误
    Oracle数据库服务质量（QoS）管理要求数据库资源管理器使用特定的资源计划。
  • 无权访问服务器池
    通过配置特殊操作系统组，可以与数据库分开管理服务器池。
  • 服务器池标记为无法管理
    如果Oracle数据库QoS管理无法正确测量或预测该服务器池中部署的性能类的性能，则会将服务器池标记为无法管理。
  • 性能等级缺少度量标准
    Oracle Database QoS Management通过查询集群中的托管数据库实例来捕获度量标准。
  • Oracle数据库QoS管理不是生成建议
    登录Oracle数据库QoS管理仪表板（仪表板）时，每分钟生成一次建议。
  • 最近添加的服务器被放置在错误的服务器池中
    服务器由Oracle Clusterware在群集内移动，或者在管理员的指导下移动。
  • 检测到RMI端口冲突
    为群集安装Oracle Grid Infrastructure时， qosmserver资源的端口可能设置为23792。

10.1.2无法为数据库启用Oracle数据库QoS管理

对于由Oracle Database QoS Management管理的数据库，数据库必须符合，启用，并且Oracle Database QoS Management必须能够访问APPQOSSYS数据库用户：

  • 兼容数据库是运行Oracle Database 11.2.0.2或更高版本的Oracle RAC数据库。

  • 要启用数据库，Oracle数据库QoS管理必须能够连接到数据库。在Oracle Enterprise Manager云控制中选择“ 启用服务质量管理”链接时，将配置连接。选择此链接时，系统将提示您输入数据库中的APPQOSSYS帐户的群集凭据和密码。默认情况下， APPQOSSYS帐户已过期。提交密码时，帐户将被解锁。

  • 如果通过其他方法更改了APPQOSSYS用户的密码或帐户被锁定，则对此数据库禁用Oracle数据库QoS管理，直到通过再次选择“ 启用服务质量管理”链接更正此条件。

10.1.3 Oracle数据库资源管理器未启用和资源计划错误

Oracle数据库服务质量（QoS）管理要求数据库资源管理器使用特定的资源计划。

只要为Oracle数据库QoS管理启用数据库管理，Oracle数据库QoS管理就会安装特殊的Oracle数据库资源管理器计划APPQOS_PLAN 。Oracle数据库QoS管理要求此资源计划将性能类移动到不同级别的CPU调度。在此数据库上启用Oracle数据库QoS管理时，无法激活其他计划。如果未为Oracle RAC数据库启用资源计划，则在尝试启用数据库以供Oracle数据库QoS管理进行管理时会出现错误。启用资源计划后，启用数据库以便Oracle Database QoS Management进行管理成功。

要检查目标数据库上所需的APPQOS_PLAN是否处于活动状态，请使用SQL * Plus进行连接并发出以下语句：

SQL> show parameter resource_manager_plan

您应该看到类似于以下内容的输出：

NAME TYPE VALUE ------------------------------------ ----------- ------------------------------ resource_manager_plan string FORCE：APPQOS_PLAN

如果要查询Multitenant数据库，则输出类似于以下内容：

NAME TYPE VALUE ------------------------------------ ----------- ------------------------------ resource_manager_plan string FORCE：ORA $ QOS_CDB_PLAN

10.1.4无权访问服务器池

通过配置特殊操作系统组，可以与数据库分开管理服务器池。

默认情况下，为群集安装Oracle Grid Infrastructure的用户可以在服务器池上执行操作。如果为Oracle数据库安装使用单独的操作系统，则必须先将数据库软件所有者的执行权限授予数据库软件所有者，然后才能在该服务器池中部署数据库。要授予此权限，必须使用CRSCTL实用程序修改服务器池ACL属性。数据库管理员还可以使用服务器控制（SRVCTL）或Oracle Enterprise Manager创建服务器池。

10.1.5服务器池标记为无法管理

如果Oracle数据库QoS管理无法正确测量或预测该服务器池中部署的性能类的性能，则会将服务器池标记为无法管理。

在以下情况下，服务器池可能被视为无法管理：

  1. 服务器池中的服务器具有不同的物理CPU计数。

  2. 无法检索每个数据库实例的CPU计数。

  3. 服务器上所有数据库实例的已配置CPU计数总和大于其物理CPU计数。

  4. Singleton服务部署在最大大小为1的服务器池中。

  5. Oracle数据库QoS管理无法收集所有性能类的度量标准，或者收集的度量标准不包含有效数据。

  6. 服务器池具有未启用Oracle数据库QoS管理的数据库。

如果将服务器添加到服务器池，则Oracle会在新服务器上为服务器池中的策略管理数据库启动实例和服务。在新服务器上启动实例时，Oracle会检查现有实例的SPFILE以获取CPU_COUNT设置，并将此值用于新实例。默认情况下，如果SPFILE中没有CPU_COUNT设置， CPU_COUNT启动新服务器上的实例，并将CPU_COUNT设置为服务器的物理CPU数。这不是受支持的配置，将导致服务器池标记为无法管理。此外，如果修改CPU_COUNT参数但未将更改存储在SPFILE中，则CPU_COUNT参数可能会在新启动的实例上设置为错误的值，从而导致配置违规。

确保要由Oracle Database QoS Management管理的数据库和数据库实例符合“支持的数据库配置”中记录的要求。

10.1.6性能等级缺少度量标准

Oracle Database QoS Management通过查询集群中的托管数据库实例来捕获度量标准。

在以下情况下不显示度量标准：

  1. 数据库请求（正在完成的工作）与为特定性能类定义的分类器不匹配，并且显示“无需”。

  2. 首先评估性能类的更通用的分类器。因此，具有更具体分类器的Performance Class显示为具有“No demand”。例如，如果sales_pc Performance Class使用分类器service=Sales且sales_search Performance Class使用分类器（ s ervice=Sales and action=Search) ，那么，如果sales_pc Performance Class在Edit Policy Set Performance Class中首先列出订单屏幕，然后任何使用sales服务的工作请求都将放在sales_pc Performance Class中，包括执行Search操作的那些。sales_search性能类不生成任何度量标准，并且该性能等级显示“无需求”。

  3. 数据库实例被过度使用，度量标准查询达到超时限制，并显示“不完整数据”。

  4. 数据库不会为Performance Class生成相互一致的数据，并显示“Nonconforming data”。例如，对超过一秒的数据库请求进行分类的性能类将导致此响应。

  5. 通过查询收集的度量标准不会通过Oracle数据库QoS管理执行的完整性验证检查，并显示“不合格数据”。

10.1.7 Oracle数据库QoS管理不生成建议

登录Oracle数据库QoS管理仪表板（仪表板）时，每分钟生成一次建议。

在以下情况下，建议书不会出现在仪表板上：

  1. Oracle数据库QoS管理已禁用。

  2. 正在实施一项建议的行动正在进行中。

  3. 在启用Oracle数据库QoS管理的第一分钟内，或在提交策略集或激活策略集时。

10.1.8最近添加的服务器放置在错误的服务器池中

服务器由Oracle Clusterware在群集内移动，或者在管理员的指导下移动。

当新服务器加入群集时，Oracle Clusterware会根据放置算法以及Min，Max和Importance的服务器池属性的状态将服务器放置在服务器池中。有关此放置过程的完整说明，请参见“ Oracle Clusterware管理和部署指南” 。

10.1.9检测到RMI端口冲突

为群集安装Oracle Grid Infrastructure时， qosmserver资源的端口可能设置为23792。

这可能导致与Java远程方法调用（RMI）的端口冲突。

您应该将qosmserver资源设置为使用可用端口，然后使用以下命令重新启动qosmserver资源：

srvctl修改qosmserver -rmiport port srvctl stop qosmserver srvctl start qosmserver
