ALTER DATABASE

目的

使用ALTER DATABASE语句可以修改，维护或恢复现有数据库。

也可以看看：

  • 有关执行介质恢复的示例，请参见“Oracle数据库备份和恢复用户指南”

  • 有关使用ALTER DATABASE语句维护备用数据库的其他信息，请ALTER Oracle Data Guard概念和管理

  • 有关创建数据库的信息，请创建数据库

先决条件

您必须具有ALTER DATABASE系统特权。

要指定startup_clauses ，还必须连接AS SYSDBA ， AS SYSOPER ， AS SYSBACKUP或AS SYSDG 。

要指定general_recovery子句，还必须具有SYSDBA或SYSBACKUP系统特权。

要指定DEFAULT EDITION子句，还必须在指定版本上具有USE对象特权WITH GRANT OPTION 。

如果您连接到多租户容器数据库（CDB）：

  • 要修改整个CDB，当前容器必须是root，并且必须具有通常授予的ALTER DATABASE权限。

  • 要修改容器，它必须是当前容器，并且您必须具有ALTER DATABASE权限，可以是通常授予的权限，也可以是容器中的本地权限。

关于在CDB中使用ALTER DATABASE的注意事项

在连接到CDB时发出ALTER DATABASE语句时，语句的行为取决于当前容器和您指定的子句。

如果当前容器是根，则具有以下子句的ALTER DATABASE语句将修改整个CDB。要指定这些子句，您必须具有通常授予的ALTER DATABASE权限：

  • startup_clauses

  • recovery_clauses

    注意：支持recovery_clauses一个子集来备份和恢复单个可插拔数据库（PDB）。要指定这些子句，您必须具有ALTER DATABASE权限，可以在PDB中共同授予或在本地授予。有关更多信息，请参阅“ 有关在CDB中使用recovery_clauses的说明 ” 。

  • logfile_clauses

  • controlfile_clauses

  • standby_database_clauses

  • instance_clauses

  • security_clause

  • RENAME GLOBAL_NAME TO

  • ENABLE BLOCK CHANGE TRACKING

  • DISABLE BLOCK CHANGE TRACKING

  • undo_mode_clause

如果当前容器是根，则具有以下子句的ALTER DATABASE语句仅修改根。要指定这些子句，您必须具有ALTER DATABASE权限，通常可以在根目录中授予或在本地授予：

  • database_file_clauses

  • DEFAULT EDITION

  • DEFAULT TABLESPACE

如果当前容器是根，则具有以下子句的ALTER DATABASE语句将修改根并为PDB设置默认值。要指定这些子句，您必须具有通常授予的ALTER DATABASE权限：

  • DEFAULT [ LOCAL ] TEMPORARY TABLESPACE

  • flashback_mode_clause

  • SET DEFAULT { BIGFILE | SMALLFILE } TABLESPACE

  • set_time_zone_clause

如果当前容器是PDB，则ALTER DATABASE语句将修改该PDB。在这种情况下，您只能发出ALTER PLUGGABLE DATABASE语句也支持的ALTER DATABASE子句。提供此功能是为了保持已迁移到CDB环境的应用程序的向后兼容性。唯一的例外是修改PDB存储限制，为此，必须使用pdb_storage_clause的ALTER PLUGGABLE DATABASE 。有关这些子句的完整信息，请参阅ALTER PLUGGABLE DATABASE文档。

语义

database_clause

为非容器数据库指定DATABASE选项。

DB_NAME

指定要更改的数据库的名称。如果省略db_name ，则Oracle数据库会更改由初始化参数DB_NAME的值标识的数据库。您只能更改控制文件由初始化参数CONTROL_FILES指定的数据库。数据库标识符与Oracle Net数据库规范无关。

startup_clauses

startup_clauses允许您挂载和打开数据库，以便用户可以访问它。

MOUNT条款

使用MOUNT子句装入数据库。已装入数据库时，请勿使用此子句。

MOUNT STANDBY DATABASE

您可以指定MOUNT STANDBY DATABASE来安装物理备用数据库。关键字STANDBY DATABASE是可选的，因为Oracle数据库会自动确定要装入的数据库是主数据库还是备用数据库。一旦执行此语句，备用实例就可以从主实例接收重做数据。

也可以看看：

有关备用数据库的详细信息，请参阅Oracle Data Guard概念和管理

安装克隆数据库

指定MOUNT CLONE DATABASE以装入克隆数据库。

OPEN条款

使用OPEN子句使数据库可供正常使用。必须先装入数据库，然后才能打开它。

如果仅指定OPEN而不指定任何其他关键字，则缺省值为主数据库，逻辑备用数据库或快照备用数据库上的OPEN READ WRITE NORESETLOGS ，以及物理备用数据库上的OPEN READ ONLY 。

打开读写

指定OPEN READ WRITE以读/写模式打开数据库，允许用户生成重做日志。如果要打开主数据库，这是默认设置。您不能为物理备用数据库指定此子句。

也可以看看：

“ READ ONLY / READ WRITE：示例 ”

RESETLOGS | NORESETLOGS

此子句确定Oracle数据库是否将当前日志序列号重置为1，归档所有未归档日志（包括当前日志），并丢弃在恢复期间未应用的任何重做信息，以确保永远不会应用它。Oracle数据库自动使用NORESETLOGS但在以下特定情况下除外，这些情况需要此子句的设置：

  • 您必须指定RESETLOGS ：

    • 使用备份控制文件执行不完整的介质恢复或介质恢复后

    • 在之前的OPEN RESETLOGS操作之后没有完成

    • 在FLASHBACK DATABASE操作之后

  • 如果安装了创建的控制文件，则必须在联机日志丢失时指定RESETLOGS ，或者如果它们没有丢失则必须指定NORESETLOGS 。

升级| DOWNGRADE

仅在升级或降级数据库时才使用这些OPEN子句参数。该子句指示Oracle数据库分别根据升级和降级的需要动态修改系统参数。您可以使用SQL * Plus STARTUP UPGRADE或STARTUP DOWNGRADE命令获得相同的结果。

对CDB使用UPGRADE或DOWNGRADE参数时，将以指定模式打开根容器，但在READ WRITE模式下打开所有其他容器。

也可以看看：

  • “Oracle数据库升级指南” ，了解将数据库从一个发行版升级或降级到另一个发行版所需的步骤

  • SQL * Plus用户指南和参考有关SQL * Plus STARTUP命令的信息

打开阅读

指定OPEN READ ONLY以将用户限制为只读事务，从而阻止它们生成重做日志。打开物理备用数据库时，此设置是默认设置，因此即使从主数据库站点复制存档日志，物理备用数据库也可用于查询。

打开数据库的限制

以下限制适用于打开数据库：

  • 如果数据库当前由另一个实例以READ WRITE模式打开，则无法在READ ONLY模式下打开数据库。

  • 如果需要恢复，则无法在READ ONLY模式下打开数据库。

  • 在READ ONLY模式下打开数据库时，不能使表空间脱机。但是，您可以脱机和联机获取数据文件，并且可以在READ ONLY模式下打开数据库时恢复脱机数据文件和表空间。

recovery_clauses

recovery_clauses包括备份后操作。对于所有这些子句，Oracle数据库使用当前控制文件已知的任何数据文件和日志文件来恢复数据库。

也可以看看：

Oracle数据库备份和恢复用户指南 ，了解有关备份数据库和“ 数据库恢复：示例 ”的信息

关于 在CDB中 使用 recovery_clauses的 注意事项

当前容器是根时，您可以指定要备份和恢复整个CDB的所有recovery_clauses 。

当前容器是PDB时，您可以指定recovery_clauses的以下子条款来备份和恢复PDB：

  • BEGIN BACKUP

  • END BACKUP

  • full_database_recovery ：您只能指定DATABASE关键字

  • partial_database_recovery

  • general_recovery的LOGFILE和CONTINUE子句

您还可以使用ALTER PLUGGABLE DATABASE的pdb_recovery_clauses指定前面的子条款。请参阅ALTER PLUGGABLE DATABASE的语法图pdb_recovery_clauses 。

general_recovery

使用general_recovery子句可以控制数据库或备用数据库或指定表空间或文件的介质恢复。当您的实例安装，打开或关闭数据库并且未使用所涉及的文件时，可以使用此子句。

注意：

在完全或部分数据库恢复和日志文件恢复期间，默认情况下会启用并行性。数据库计算并行度。您可以通过指定NOPARALLEL来禁用这些操作的并行性，或者使用PARALLEL integer指定并行度，如相应的语法图所示。

一般数据库恢复的限制

一般恢复受以下限制：

  • 只有在数据库关闭时才能恢复整个数据库。

  • 您的实例必须以独占模式安装数据库。

  • 如果要恢复的表空间或数据文件处于脱机状态，则可以在打开或关闭数据库时恢复表空间或数据文件。

  • 如果通过共享服务器体系结构连接到Oracle数据库，则无法执行介质恢复。

自动

如果希望Oracle数据库自动生成继续恢复操作所需的下一个归档重做日志文件的名称，请指定AUTOMATIC 。如果定义了LOG_ARCHIVE_DEST_ n参数，则Oracle数据库将扫描那些对第一个本地目标有效并启用的参数。它将该目标与LOG_ARCHIVE_FORMAT结合使用，以生成目标重做日志文件名。如果未定义LOG_ARCHIVE_DEST_ n参数，则Oracle数据库将使用LOG_ARCHIVE_DEST参数的值。

如果找到结果文件，则Oracle数据库将应用该文件中包含的重做。如果找不到该文件，则Oracle数据库会提示您输入文件名，并将生成的文件名显示为建议。

如果既未指定AUTOMATIC也未指定LOGFILE ，则Oracle数据库会提示您输入文件名，并将生成的文件名显示为建议。然后，您可以接受生成的文件名或将其替换为完全限定的文件名。如果您知道归档文件名与Oracle数据库生成的文件名不同，那么您可以使用LOGFILE子句节省时间。

从 '位置'

指定FROM 'location'以指示从中读取归档重做日志文件组的位置。location的值必须是遵循操作系统约定的完全指定的文件位置。如果省略此参数，则Oracle数据库假定归档重做日志文件组位于初始化参数LOG_ARCHIVE_DEST或LOG_ARCHIVE_DEST_1指定的位置。

full_database_recovery

full_database_recovery子句允许您恢复整个数据库。

数据库

指定DATABASE子句以恢复整个数据库。这是默认值。只有在数据库关闭时才能使用此子句。

备用数据库

指定STANDBY DATABASE子句以使用从主数据库复制的控制文件和归档重做日志文件手动恢复物理备用数据库。必须装入备用数据库但不能打开。

该条款仅恢复在线数据文件。

  • 使用UNTIL子句指定恢复操作的持续时间。

    • CANCEL表示基于取消的恢复。在您使用RECOVER CANCEL子句发出ALTER DATABASE语句之前，此子句将恢复数据库。

    • TIME表示基于时间的恢复。此参数将数据库恢复到日期指定的时间。日期必须是格式为'YYYY-MM-DD:HH24:MI:SS'的字符文字。

    • CHANGE表示基于变更的恢复。此参数将数据库恢复到integer指定的系统更改编号之前的事务一致状态。

    • CONSISTENT恢复数据库，直到所有在线文件都进入一致的SCN点，以便数据库可以在只读模式下打开。此子句要求控制文件是备份控制文件。

  • 如果要使用备份控制文件而不是当前控制文件，请指定USING BACKUP CONTROLFILE 。

  • 使用存储快照优化指定SNAPSHOT TIME子句以使用存储快照恢复数据库。如果在创建存储快照时数据库未处于备份模式，则可以使用此子句。

    • date必须是格式为'YYYY-MM-DD:HH24:MI:SS'的字符文字。它必须表示快照完成后的一个时间。如果指定UNTIL TIME子句，则SNAPSHOT TIME date必须早于UNTIL TIME date 。

partial_database_recovery

partial_database_recovery子句允许您恢复单个表空间和数据文件。

TABLESPACE

指定TABLESPACE子句以仅恢复指定的表空间。如果要恢复的表空间处于脱机状态，则可以在打开或关闭数据库时使用此子句。

也可以看看：

“ 使用并行恢复过程：示例 ”

DATAFILE

指定DATAFILE子句以恢复指定的数据文件。如果要恢复的数据文件处于脱机状态，则可以在打开或关闭数据库时使用此子句。

您可以按名称或编号标识数据文件。如果通过号标识，然后filenumber是代表找到的数量的整数FILE#中的列V$DATAFILE动态性能视图或FILE_ID中的列DBA_DATA_FILES数据字典视图。

STANDBY {TABLESPACE | DATAFILE}

在早期版本中，您可以指定STANDBY TABLESPACE或STANDBY DATAFILE来恢复特定表空间的旧备份或备用数据库上的特定数据文件，以与备用数据库的其余部分保持一致。现在不支持这两个条款。相反，要将备用数据库恢复到一致点，但不能再使用语句ALTER DATABASE RECOVER MANAGED STANDBY DATABASE UNTIL CONSISTENT 。

LOGFILE

指定LOGFILE “ filename ”通过应用指定的重做日志文件继续介质恢复。

测试

使用TEST子句进行试验恢复。如果正常的恢复过程遇到一些问题，则试用恢复很有用。它可以让您展望重做流，以检测可能的其他问题。试用恢复以类似于正常恢复的方式应用重做，但它不会将更改写入磁盘，并在试用恢复结束时回滚其更改。

仅当您已恢复自上次RESETLOGS操作以来执行的备份时，才能使用此子句。否则，Oracle数据库会返回错误。

允许......腐败

ALLOW integer CORRUPTION子句允许您在日志文件损坏的情况下指定允许恢复继续时可以容忍的损坏块数。

继续

指定CONTINUE以在中断后继续多实例恢复以禁用线程。

如果未指定其他日志文件，请使用Oracle数据库自动生成的重做日志文件指定CONTINUE DEFAULT以继续恢复。此子句等效于指定AUTOMATIC ，但Oracle数据库不提示输入文件名。

取消

指定CANCEL以终止基于取消的恢复。

managed_standby_recovery

使用managed_standby_recovery子句在物理备用数据库上启动和停止Redo Apply。通过不断应用从主数据库接收的重做，重做应用使备用数据库在事务上与主数据库保持一致。

SET STANDBY NOLOGGING

备用nologging指示数据库不记录有资格在没有记录的情况下完成的操作。数据库将操作创建的数据块发送到Data Guard配置中的每个合格备用数据库，以防止备用数据库中丢失数据并使其与主数据库保持同步。

使用此子句可确定如何处理未记录的任务。在创建数据库时，可以为数据库选择两种日志记录模式之一，并且可以将数据库的日志记录模式从一种模式更改为另一种模式。

  • 为备份SET STANDBY NOLOGGING FOR LOAD PERFORMANCE ，使数据库进入备用nologging负载性能模式。在此模式下，作为未记录任务的一部分加载的数据将通过专用网络连接发送到符合条件的备用数据库，前提是这样做不会减慢加载过程的速度。如果加载过程变慢，则不会发送数据，但会在每个备用数据库遇到无效重做时自动从主数据中获取数据，并将重试直到最终收到数据块。

  • 指定SET STANDBY NOLOGGING FOR DATA AVAILABILITY将数据库置于数据可用性模式的备用nologging。在此模式下，作为未记录任务的一部分加载的数据将通过网络连接或重做中的块映像发送到符合条件的备用数据库，以防网络连接失败。也就是说，在这种模式下，如果网络连接或相关进程阻止通过专用网络连接发送数据，则负载将切换为以记录方式完成。

对于备用nologging模式，合格备用数据库是一个打开以进行读取，运行管理恢复和接收重做到备用重做日志的备用数据库。

最大限度地保护

此设置可建立最大保护模式，并提供最高级别的数据保护。在将恢复该事务所需的所有数据写入至少一个配置为使用SYNC日志传输模式的物理备用数据库之前，事务不会提交。如果主数据库无法将重做记录写入至少一个此类备用数据库，则主数据库将关闭。此模式可确保零数据丢失，但它对主数据库的性能和可用性具有最大的潜在影响。

建立最大保护模式的限制

仅当当前数据保护模式为MAXIMUM AVAILABILITY并且至少有一个同步备用数据库时，才能在打开的数据库上指定TO MAXIMIZE PROTECTION 。

最大限度地提高可用性

此设置可确定最高可用性模式，并提供下一个最高级别的数据保护。在将恢复该事务所需的所有数据写入至少一个配置为使用SYNC日志传输模式的物理或逻辑备用数据库之前，事务不会提交。与最大保护模式不同，如果主数据库无法将重做记录写入至少一个此类备用数据库，则不会关闭主数据库。相反，保护会降低到最高性能模式，直到故障得到纠正并且备用数据库已赶上主数据库。除非主数据库在最高性能模式下出现故障，否则此模式可确保零数据丢失。最高可用性模式提供最高级别的数据保护，而不会影响主数据库的可用性。

最大限度地提高性能

此设置可确定最高性能模式，并且是默认设置。事务在恢复该事务所需的数据已写入备用数据库之前提交。因此，如果主数据库失败并且您无法从主数据库恢复重做记录，则某些事务可能会丢失。此模式提供最高级别的数据保护，而不会影响主数据库的性能。

例子

READ ONLY / READ WRITE：示例

以下语句以只读模式打开数据库：

ALTER DATABASE OPEN READ ONLY;

以下语句以读/写模式打开数据库并清除联机重做日志：

ALTER DATABASE OPEN READ WRITE RESETLOGS;

使用并行恢复过程：示例

以下语句使用并行恢复过程执行表空间恢复：

ALTER DATABASE RECOVER TABLESPACE tbs_03 PARALLEL;

添加重做日志文件组：示例

以下语句添加具有两个成员的重做日志文件组，并使用GROUP参数值3标识它：

ALTER DATABASE添加LOGFILE GROUP 3（'diska：log3.log'，'diskb：log3.log'）SIZE 50K;

以下语句将包含两个成员的重做日志文件组添加到线程5（在Real Application Clusters环境中）并为其指定GROUP参数值4：

ALTER DATABASE添加LOGFILE THREAD 5 GROUP 4（'diska：log4.log'，'diskb：log4：log'）;

添加重做日志文件组成员：示例

以下语句将成员添加到上一示例中添加的重做日志文件组：

ALTER DATABASE添加LOGFILE成员'diskc：log3.log'到GROUP 3;

删除日志文件成员：示例

以下语句删除了上一个示例中添加的一个重做日志文件成员：

ALTER DATABASE DROP LOGFILE MEMBER'obcobb：log3.log';

以下语句将删除重做日志文件组3的所有成员：

ALTER DATABASE DROP LOGFILE GROUP 3;

重命名日志文件成员：示例

以下语句重命名重做日志文件成员：

ALTER DATABASE RENAME FILE'diskc：log3.log'TO'diskb：log3.log';

上述语句仅将重做日志组的成员从一个文件更改为另一个文件。该语句实际上并未将文件diskc:log3.log的名称更改为diskb:log3.log 。在发出此声明之前，您必须通过操作系统更改文件的名称。

设置表空间的默认类型：示例

以下语句指定默认情况下将后续创建的表空间创建为bigfile表空间：

ALTER DATABASE SET DEFAULT BIGFILE TABLESPACE;

更改默认临时表空间：示例

以下语句使tbs_05表空间（在“ 创建临时表空间：示例 ”中创建 ）成为数据库的默认临时表空间。如果在创建时未指定任何临时表空间，则此语句建立默认临时表空间，或者使用tbs_05替换现有的默认临时表空间：

ALTER DATABASE DEFAULT TEMPORARY TABLESPACE tbs_05;

或者，可以使用表空间组将一组表空间定义为默认临时表空间。以下语句使表空间组tbs_group_01中的表空间（在“ 将临时表空间添加到表空间组：示例 ”中创建 ）使用数据库的默认临时表空间：

ALTER DATABASE DEFAULT TEMPORARY TABLESPACE tbs_grp_01;

创建新数据文件：示例

以下语句基于文件tbs_f04.dbf创建新数据文件tbs_f03.dbf 。在创建新数据文件之前，必须使现有数据文件（或其所在的表空间）脱机。 

ALTER DATABASE CREATE DATAFILE'tbs_f03.dbf'AS'tbs_f04.dbf';

操作临时文件：示例

以下内容使在添加和删除数据文件和临时文件：示例中创建的临时文件temp02.dbf脱机，然后重命名临时文件：

ALTER DATABASE TEMPFILE'temp02.dbf'OFFLINE; ALTER DATABASE RENAME FILE'timp02.dbf'TO'temp03.dbf';

重命名临时文件的语句要求您首先在操作系统上创建文件temp03.dbf 。

更改全局数据库名称：示例

以下语句更改数据库的全局名称，并包括数据库名称和域：

ALTER DATABASE RENAME GLOBAL_NAME转发至demo.world.example.com;

启用和禁用块更改跟踪：示例

以下语句启用块更改跟踪，并使Oracle数据库创建名为tracking_file的块更改跟踪文件，并覆盖该文件（如果该文件已存在）：

ALTER DATABASE使用FILE'tracking_file'REUSE启用块更改跟踪;

以下语句禁用块更改跟踪并删除现有块更改跟踪文件：

ALTER DATABASE DISABLE BLOCK CHANGE TRACKING;

调整数据文件大小：示例

以下语句尝试更改数据文件diskb:tbs_f5.dbf的大小diskb:tbs_f5.dbf ：

ALTER DATABASE DATAFILE'obcobb：tbs_f5.dbf'RESIZE 10 M;

清除日志文件：示例

以下语句清除日志文件：

ALTER DATABASE CLEAR LOGFILE'diskc：log3.log';

数据库恢复：示例

以下语句执行整个数据库的完全恢复，让Oracle数据库生成所需的下一个归档重做日志文件的名称：

ALTER DATABASE RECOVER AUTOMATIC DATABASE;

以下语句明确指定要应用的Oracle数据库的重做日志文件：

ALTER DATABASE RECOVER LOGFILE'diskc：log3.log';

以下语句执行数据库的基于时间的恢复：

ALTER DATABASE RECOVER AUTOMATIC AUTOMATIC FOR TIME'2001-10-27：14：00：00';

Oracle数据库将恢复数据库，直到2001年10月27日下午2:00。

有关恢复表空间的示例，请参阅“ 使用并行恢复进程：示例 ” 。
