系统管理分片的多表系列支持

Oracle Database 18c的Oracle Sharding功能仅支持每个分片数据库的一个表系列（一组共享相同分片键的相关表）。在Oracle Database 19c中，Oracle Sharding包括对多个表系列的支持，其中来自不同表系列的所有数据都位于相同的块中。此功能仅适用于系统管理的分片数据库。访问不同表系列的不同应用程序现在可以托管在一个分片数据库中。

有一个新的GDSCTL命令， CONFIG TABLE FAMILY ，以及其他几个命令被扩展为支持此功能： ADD SERVICE ， MODIFY SERVICE ， CONFIG SERVICE ， CONFIG CHUNKS ， STATUS ROUTING和验证VALIDATE CATALOG 。

此功能没有引入新的SQL关键字或语句;但是，使用CREATE SHARDED TABLE和TABLESPACE SET会改变一些限制。

支持同一CDB中的多个PDB-Shards

在Oracle Database 18c中，Oracle Sharding引入了将CDB中的单个PDB用作分片或分片目录数据库的功能。在Oracle数据库19c中，Oracle Sharding允许您在CDB中使用多个PDB用于分片或分片目录数据库，但有一些限制。例如，此功能允许CDB包含来自不同分片数据库（SDB）的分片PDB，每个分片数据库都有自己独立的目录数据库。

跨越碎片生成唯一序列号

在Oracle Database 19c之前，如果您需要跨分片的唯一编号，则必须自己管理它。在Oracle Database 19c中，Oracle Sharding允许您在每个分片上独立生成序列号，这些序列号在所有分片中都是唯一的。

为了支持此功能，新的SEQUENCE对象的条款， SHARD和NOSHARD ，包括在SEQUENCE对象DDL语法。

支持分片目录备用数据库上的多分片查询协调器

在Oracle Database 19c之前，只有主分片目录数据库可以用作多分片查询协调器。在Oracle Database 19c中，您还可以在分片目录数据库的Oracle Active Data Guard备用数据库上启用多分片查询协调器。这提高了多分片查询工作负载的可伸缩性和可用性。

跨越Shards的参数设置的传播

在Oracle Database 19c之前，数据库管理员必须在分片数据库中的每个分片上配置ALTER SYSTEM参数设置。此功能允许管理员集中管理参数设置并将参数设置从分片目录传播到所有数据库分片，从而提供易管理性。在分片目录中配置设置后，它们会自动传播到分片数据库的所有分片。

请参阅碎片参数设置的传播

弃用和支持

父主题： Oracle Database 19c中的更改

支持在GDSCTL命令行中设置密码

为了增强安全性，从Oracle Database 19c开始，不再支持从操作系统提示符调用时从全局数据服务控制实用程序（GDSCTL）命令行指定密码的功能。

此desupport仅适用于从用户命令行提示符调用GDSCTL的密码更改。例如，不支持以下命令：

$ gdsctl add database -connect inst1 -pwd gsm_password

从GDSCTL实用程序本身指定密码仍然有效。例如，以下命令有效：

GDSCTL> add database -connect inst1 -pwd gsm_password

在从操作系统提示符调用的GDSCTL命令中指定密码时，此弃用解决了安全漏洞。

用户定义的分片方法

用户定义的分片允许您明确指定数据到各个分片的映射。由于性能，监管或其他原因，某些数据需要存储在特定分片上，并且您必须完全控制在分片之间移动数据时使用它。此方法允许您定义基于LIST或RANGE的分片。

支持PDB作为碎片

在CDB中使用PDB作为分片或分片目录数据库。在此版本中，Oracle Sharding支持将分片或分片目录作为CDB中的单个PDB。扩展了GDSCTL命令ADD SHARD，并实现了新命令ADD CDB，MODIFY CDB，CONFIG CDB和REMOVE CDB，以便Oracle Sharding可以支持多租户架构。

Oracle GoldenGate用于细粒度主动 - 主动复制，其中所有分片都是可写的，并且每个分片可以部分复制到分片组内的其他分片。Oracle 12.2中添加的自动冲突检测和解决功能用于处理由主动 - 主动复制引起的任何冲突

对JSON，LOB和空间对象的分片支持

此版本使JSON运算符能够生成临时LOB，大型JSON文档（需要LOB存储的文档），空间对象，索引和运算符以及持久性LOB，以便在分片环境中使用。

以下界面是新增功能或已更改为此功能的一部分。

  • 查询和DML语句

    支持涉及LOB的交叉分片查询。

    不支持涉及多个分片的DML。此行为类似于标量列。

    协调器支持涉及单个分片的DML。

    从分片中选择的定位器可以作为绑定值传递给同一分片。

  • OCILob

    支持分片环境中的所有非BFILE相关OCILob API。有一些限制。

    在协调器上，如果LOB是从分片表中获取的，则LOB描述符的OCI_ATTR_LOB_REMOTE属性返回TRUE。

    限制：对于以两个定位符作为输入的API，例如OCILobAppend，OCILobCompare，两个定位符都应该从同一个分片中获取。如果定位器来自不同的分片，则会给出错误。

  • DBMS_LOB

    支持分片环境中的所有非BFILE相关DBMS_LOB API，但有一些限制。在协调器上，如果从分片表获取LOB，则DBMS_LOB.isremote返回TRUE。

    限制：对于以两个定位符作为输入的API，例如DBMS_LOB.append和DBMS_LOB.compare，两个定位符都应该从同一个分片中获取。如果定位器来自不同的分片，则给出错误。

有关在分片数据库部署中使用LOB的示例，请参阅为系统管理的分片数据库 创建模式，为用户定义的SDB 创建模式以及为复合SDB创建模式 。

请参阅更换碎片

Oracle RAC Sharding

Oracle RAC Sharding为表分区创建与特定Oracle RAC实例的关联，并将指定分区键的数据库请求路由到逻辑上保存相应分区的实例。这提供了更好的缓存利用率，并显着减少了跨实例的块ping。分区键只能添加到性能最关键的请求中。未指定密钥的请求仍然可以透明地工作，并且可以路由到任何实例。无需更改数据库架构即可启用此功能。

请参阅“ Oracle Real Application Clusters管理和部署指南”
