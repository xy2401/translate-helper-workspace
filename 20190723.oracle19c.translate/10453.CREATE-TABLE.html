创建表

目的

我们使用CREATE TABLE语句创建以下类型的表之一：

  • 关系表 ，是保存用户数据的基本结构。

  • 一个对象表 ，它是一个使用对象类型进行列定义的表。显式定义对象表以保存特定类型的对象实例。

您还可以创建对象类型，然后在创建关系表时在列中使用它。

除非指定了子查询，否则将创建没有数据的表。您可以使用INSERT语句向表中添加行。创建表后，可以使用ALTER TABLE语句的ADD子句定义其他列，分区和完整性约束。您可以使用ALTER TABLE语句的MODIFY子句更改现有列或分区的定义。

先决条件

要在自己的模式中创建关系表 ，您必须具有CREATE TABLE系统特权。要在另一个用户的模式中创建表，您必须具有CREATE ANY TABLE系统特权。此外，包含表的模式的所有者必须在表空间上包含空间配额以包含表或UNLIMITED TABLESPACE系统特权。

除了这些表特权之外，要创建对象表或具有对象类型列的关系表，表的所有者必须具有EXECUTE对象特权才能访问表引用的所有类型，或者您必须具有EXECUTE ANY TYPE系统特权。必须明确授予这些权限，而不是通过角色获取。

此外，如果表所有者打算将该表的访问权限授予其他用户，则必须已为所有者授予对所引用类型WITH GRANT OPTION的EXECUTE对象特权，或者具有EXECUTE ANY TYPE系统特权WITH ADMIN OPTION 。如果没有这些权限，表所有者没有足够的权限将表的访问权限授予其他用户。

要启用唯一或主键约束，您必须具有在表上创建索引所需的权限。您需要这些权限，因为Oracle数据库会在包含该表的架构中的唯一键或主键的列上创建索引。

要在evaluation_edition_clause或unusable_editions_clause指定版本，您必须具有该版本的USE特权。

要指定zonemap_clause ，您必须具有创建区域映射所需的权限。请参阅CREATE MATERIALIZED ZONEMAP文档中的“ 先决条件 ”部分。

要创建外部表 ，您必须在相应的操作系统目录上具有所需的读写操作系统权限。您必须对与外部数据所在的操作系统目录对应的数据库目录对象具有READ对象特权。如果在opaque_format_spec指定日志文件或错误文件，或者通过指定AS subquery子句将数据从数据库表卸载到外部表中，则还必须对文件所在的数据库目录具有WRITE对象特权。

要在与您自己不同的数据库模式中创建XMLType表 ，您不仅必须具有CREATE ANY TABLE权限，还必须具有CREATE ANY INDEX权限。这是因为在创建表时，会在列OBJECT_ID上创建唯一索引。列OBJECT_ID存储系统生成的对象标识符。

语义

全球临时

指定GLOBAL TEMPORARY以创建临时表，其定义对具有适当权限的所有会话可见。临时表中的数据仅对将数据插入表中的会话可见。

首次创建临时表时，其元数据存储在数据字典中，但没有为表数据分配空间。在表上进行第一次DML操作时为表段分配空间。临时表定义的持续方式与常规表的定义相同，但表段和表中包含的任何数据都是特定于会话的数据或特定于 事务的数据。您可以使用ON COMMIT子句指定表段和数据是特定于会话还是特定于事务。

只有在没有绑定任何会话时，才能在临时表上执行DDL操作（例如ALTER TABLE ， DROP TABLE ， CREATE INDEX ）。会话绑定到临时表，并在表上INSERT操作。会话变为未绑定到具有TRUNCATE语句的临时表或在会话终止时，或者对于特定于事务的临时表，通过发出COMMIT或ROLLBACK语句。

私人临时

指定PRIVATE TEMPORARY以创建专用临时表。

私有临时表与临时表的不同之处在于，其定义和数据 仅在创建它的会话中可见。使用ON COMMIT子句定义私有临时表的范围： 事务或会话 。与关键字DROP DEFINITION一起使用的ON COMMIT子句创建一个特定于事务的表，其事务提交时将删除其数据和定义。这是默认行为。与关键字PRESERVE DEFINITION一起使用的ON COMMIT子句创建一个特定于会话的表，其定义在事务提交时保留。有关ON COMMIT子句的用法详细信息，请参见此处。

私有临时表支持三个DDL语句： CREATE ， DROP和TRUNCATE 。

您必须是SYS以外的用户才能创建专用临时表。

临时表的限制

临时表受以下限制：

  • 无法对临时表进行分区，聚簇或索引组织。

  • 您不能在临时表上指定任何外键约束。

  • 临时表不能包含嵌套表的列。

  • 您不能指定LOB_storage_clause的以下子句： TABLESPACE ， storage_clause或logging_clause 。

  • 临时表不支持并行UPDATE ， DELETE和MERGE 。

  • 您可以为临时表指定的segment_attributes_clause的唯一部分是TABLESPACE ，它允许您指定单个临时表空间。

  • 临时表不支持分布式事务。

  • 临时表不能包含INVISIBLE列。

私人临时表的限制

除了临时表的一般限制之外，私有临时表还受到以下限制：

  • 私有临时表的名称必须始终以init.ora参数PRIVATE_TEMP_TABLE_PREFIX定义的内容为前缀。默认值为ORA$PTT_ 。

  • 您无法在专用临时表上创建索引，物化视图或区域映射。

  • 您无法使用默认值定义列。

  • 您不能在任何永久对象中引用私有临时表，例如视图或触发器。

  • 通过数据库链接无法看到私有临时表。

分片

指定SHARDED以创建分片表。

仅当您使用Oracle Sharding时，此子句才有效，Oracle Sharding是一种数据层体系结构，其中数据在独立数据库中水平分区。这种配置中的每个数据库都称为分片。所有分片一起组成一个逻辑数据库，称为分片数据库（SDB）。水平分区涉及跨分片拆分表，以便每个分片包含具有相同列但行的不同子集的表。以这种方式拆分的表称为分片表。

创建分片表时，必须指定用于创建表的表空间集。没有为分片表设置默认表空间。有关更多信息，请参见CREATE TABLESPACE SET 。

Oracle Sharding基于Oracle分区功能。因此，分片表必须是分区或复合分区表。创建分片表时，必须指定table_partitioning_clauses 。有关这些子句的完整语义，请参阅table_partitioning_clauses 。

对Sharded Tables的限制

以下限制适用于分片表：

  • 在系统管理的分ORA-02530 ，当CREATE SHARDED TABLE语句不包含PARTITION BY REFERENCE或PARENT子句并且已存在根表时，您可以创建多个根表（因此表系列）而不抛出ORA-02530 。

  • 分片表不能是临时表或索引组织表。

  • 分片表不能包含嵌套表列或标识列。

  • 您不能使用TABLESPACE子句为分片系统或复合分片表指定表空间，因为用户定义的分片表需要分片表空间而不需要表空间集。

  • 您无法在用户定义的分片环境中创建表空间集。

  • 分片表需要分片表空间。不支持普通表空间。

  • 您不能为分片表的多个分区指定相同的表空间。此规则也适用于子分区。无法为属于分片表的不同分区的子分区指定相同的表空间。

  • 您必须为非参考分区分片表的每个分区指定一个表空间。

  • 对于用户定义的分片，分区方法必须是范围或列表。不支持自动列表和间隔分区。

  • 列表分区方法只能有一个分区列。

  • 列表分区表不支持默认分区。

  • 列表分区表中不支持NULL分区。

  • 在分片表上定义的主键约束必须包含分片列。不支持引用重复表列的分片表的列上的外键约束。

  • 分片表不支持系统分区和间隔范围分区。

  • 您不能在PARTITION BY或PARTITIONSET BY子句中的分片表中指定虚拟列。

虚拟列的注释

  • 如果column_expression引用实现了列级安全性的列，则虚拟列不会继承基列的安全性规则。在这种情况下，您必须通过复制虚拟列上的列级安全策略或应用隐式屏蔽数据的函数来确保虚拟列中的数据受到保护。例如，信用卡号码通常受到列级安全策略的保护，同时仍允许呼叫中心员工查看信用卡号码的最后四位数字以进行验证。在这种情况下，您可以定义虚拟列以获取信用卡号的最后四位数的子字符串。

  • 在虚拟列上定义的表索引等同于表上基于函数的索引。

  • 您无法直接更新虚拟列。因此，您无法在UPDATE语句的SET子句中指定虚拟列。但是，您可以在UPDATE语句的WHERE子句中指定虚拟列。同样，您可以在DELETE语句的WHERE子句中指定虚拟列，以根据虚拟列的派生值从表中删除行。

  • 如果函数在创建期间被明确指定为DETERMINISTIC ，则column_expression可以引用PL / SQL函数。但是，如果随后替换该函数，则依赖于虚拟列的定义不会失效。在这种情况下，如果表包含数据，则如果在约束，索引或物化视图的定义或结果缓存中使用虚拟列，则引用虚拟列的查询可能返回不正确的结果。因此，为了替换虚拟列的确定性PL / SQL函数。

    • 禁用并重新启用虚拟列上的任何约束。

    • 重建虚拟列上的任何索引。

    • 完全刷新访问虚拟列的物化视图。

    • 如果缓存的查询已访问虚拟列，则刷新结果缓存。

    • 重新统计表格。

  • 虚拟列可以是INVISIBLE列。column_expression可以包含INVISIBLE列。

您可以选择指定以下子句：

  • 使用start_time_column指定开始时间列的名称，该列包含开始日期或时间戳。

  • 使用end_time_column指定结束时间列的名称，该列包含结束日期或时间戳。

您为start_time_column和end_time_column指定的名称必须满足“ 数据库对象命名规则 ”中列出的要求。

您可以像在任何列中一样在开始时间列和结束时间列中插入和更新值，但需要考虑以下事项：

  • 如果开始时间列的值为NULL，则该行被视为对之前发生的所有时间值有效，但不包括结束时间列的值。

  • 如果结束时间列的值为NULL，则该行被视为对于在开始时间列的值之上或之后发生的所有时间值有效。

  • 如果两列的值均为NULL，则开始时间列的值必须早于结束时间列的值。该行被认为对于在开始时间列的值之上或之后发生的所有时间值有效，并且最多但不包括结束时间列的值。

  • 如果两列的值均为NULL，则该行被视为对所有时间值都有效。

使用此子句可以创建命名日志组。

  • 如果指定ALL COLUMNS ，则数据库在重做日志中包括该行的所有固定长度最大大小列。这样的重做日志是系统生成的无条件日志组。

  • 如果指定PRIMARY KEY COLUMNS ，则对于具有主键的所有表，只要执行更新，数据库就会将主键的所有列放入重做日志中。Oracle数据库评估哪些列以补充方式记录，如下所示：

    • 首先，如果约束被验证或标记为RELY并且未标记为DISABLED或INITIALLY DEFERRED ，则数据库选择主键约束的列。

    • 如果不存在此类索引，则数据库将补充记录表的所有标量列。

  • 如果指定UNIQUE COLUMNS ，则对于具有唯一键或位图索引的所有表，如果修改了任何唯一键或位图索引列，则数据库会将属于唯一键或位图索引的所有其他列放入重做日志。这样的日志组是系统生成的条件日志组。

  • 如果指定FOREIGN KEY COLUMNS ，则对于具有外键的所有表，如果修改了任何外键列，则数据库会将属于该外键的所有其他列放入重做日志中。这样的重做日志是系统生成的条件日志组。

通过指定ROW STORE COMPRESS ADVANCED启用表压缩时，启用高级行压缩 。Oracle数据库在表上的所有DML操作期间压缩数据。建议在OLTP环境中使用这种压缩形式。

使用ROW STORE COMPRESS ADVANCED或NOCOMPRESS使用PCTFREE默认值10，以最大化压缩，同时仍允许对数据进行一些未来的DML更改，除非您明确覆盖此默认值。

在早期版本中，Advanced Row Compression称为OLTP表压缩，并使用COMPRESS FOR OLTP启用。为了向后兼容，仍支持此语法。

COLUMN STORE COMPRESS {QUERY |存档}

当您指定COLUMN STORE COMPRESS FOR QUERY或COLUMN STORE COMPRESS FOR ARCHIVE ，您启用混合列压缩 。使用混合列压缩，可以在直接路径插入，传统插入和阵列插入期间压缩数据。在加载过程中，数据将转换为面向列的格式，然后进行压缩。Oracle数据库使用适合您指定级别的压缩算法。通常，级别越高，压缩比越大。混合列压缩可以在更高的CPU成本下实现更高的压缩比。因此，对于不经常更新的数据，建议使用这种形式的压缩。

COLUMN STORE COMPRESS FOR QUERY在数据仓库环境中非常有用。有效值为LOW和HIGH ， HIGH提供更高的压缩比。默认值为HIGH 。

COLUMN STORE COMPRESS FOR ARCHIVE使用比COLUMN STORE COMPRESS FOR QUERY更高的压缩率，可用于压缩将长期存储的数据。有效值为LOW和HIGH ， HIGH为最高压缩比。默认值为LOW 。

指定COLUMN STORE COMPRESS相当于指定COLUMN STORE COMPRESS FOR QUERY HIGH 。

具有COLUMN STORE COMPRESS FOR QUERY或COLUMN STORE COMPRESS FOR ARCHIVE使用PCTFREE值0来最大化压缩，除非您在physical_attributes_clause明确设置PCTFREE的值。对于这些表， PCTFREE对使用直接路径INSERT加载的块没有影响。 PCTFREE适用于使用常规INSERT加载的块，以及由于使用直接路径INSERT最初加载的块上的DML操作而创建的块。

[NO] ROW LEVEL LOCKING

如果指定ROW LEVEL LOCKING ，则Oracle数据库在DML操作期间使用行级锁定。这样可以在访问混合列压缩数据时提高这些操作的性能。如果指定NO ROW LEVEL LOCKING ，则不使用行级锁定。默认为NO ROW LEVEL LOCKING 。

在早期版本中，使用COMPRESS FOR QUERY和COMPRESS FOR ARCHIVE启用了混合列压缩。为了向后兼容，仍支持此语法。

表压缩的限制

表压缩受以下限制：

  • BasicFiles LOB的数据段未压缩。有关压缩SecureFiles LOB的信息，请参阅LOB_compression_clause 。

  • 您不能从使用COMPRESS BASIC的表中删除列，但您可以COMPRESS列设置为unused。ALTER TABLE ... drop_column_clause所有操作对使用ROW STORE COMPRESS ADVANCED ， COLUMN STORE COMPRESS FOR QUERY和COLUMN STORE COMPRESS FOR ARCHIVE 。

  • 您不能为索引组织表，溢出段的任何溢出段或分区或索引组织表的任何映射表段指定任何类型的表压缩。

  • 您不能为外部表或属于群集的表指定任何类型的表压缩。

  • 您不能为具有LONG或LONG RAW列的表，由SYS模式拥有并驻留在SYSTEM表空间中的表或启用了ROWDEPENDENCIES的表指定任何类型的表压缩。

  • 您无法在以下对象关系功能上指定混合列压缩：对象表， XMLType表，具有抽象数据类型的列，存储为表的集合或OPAQUE类型，包括存储为对象的XMLType列。

  • 当您更新使用混合列压缩进行压缩的表中的行时，该行的ROWID可能会更改。

  • 在使用混合列压缩进行压缩的表中，对单行的更新可能会导致多行锁定。因此，写入事务的并发性可能会受到影响。

  • 如果使用混合列压缩进行压缩的表具有外键约束，并且使用带有APPEND提示的INSERT插入数据，则数据将被压缩到比混合列压缩的典型值更低的水平。要使用混合列压缩来压缩数据，请禁用外键约束，使用带有APPEND提示的INSERT插入数据，然后重新启用外键约束。

inmemory_table_clause

使用此子句可启用或禁用内存中列存储（IM列存储）的表。IM列存储是一个可选的静态SGA池，它以特殊的列式格式存储表和分区的副本，这些格式针对快速扫描进行了优化。IM列存储不会替换缓冲区缓存，而是作为补充，以便两个内存区域可以以不同的格式存储相同的数据。

  • 指定INMEMORY以启用IM列存储的表。

    您可以选择使用inmemory_attributes子句指定表数据在IM列存储中的存储方式。此子句使您可以指定数据压缩方法和数据填充优先级。在Oracle RAC环境中，它还允许您指定数据在Oracle RAC实例之间的分布和复制方式。有关更多信息，请参阅inmemory_attributes子句。

  • 指定NO INMEMORY以禁用IM列存储的表。

  • 指定inmemory_column_clause以启用或禁用IM列存储的特定表列，并指定特定列的数据压缩方法。有关更多信息，请参阅inmemory_column_clause 。

在Oracle Active Data Guard环境中，如果为主数据库上的表指定此子句，则会为Oracle Active Data Guard实例中的IM列存储启用或禁用该表。

inmemory_attributes

使用inmemory_memcompress ， inmemory_priority ， inmemory_distribute和inmemory_duplicate子句指定表数据在IM列存储中的存储方式。

inmemory_memcompress

使用此子句为IM列存储中存储的表数据指定压缩方法。该数据称为内存数据。要指示数据库不压缩内存数据，请指定NO MEMCOMPRESS 。要指示数据库压缩内存数据，请指定MEMCOMPRESS FOR然后使用以下方法之一：

  • DML - 此方法针对DML操作进行了优化，并且几乎不执行数据压缩。

  • QUERY - 指定QUERY等同于指定QUERY LOW 。

  • QUERY LOW - 此方法压缩In-Memory数据的次数最少（ DML除外），从而获得最佳查询性能。这是默认值。

  • QUERY HIGH - 此方法压缩内存数据超过QUERY LOW ，但低于CAPACITY LOW 。

  • CAPACITY - 指定CAPACITY相当于指定CAPACITY LOW 。

  • CAPACITY LOW - 此方法压缩内存数据超过QUERY HIGH ，但低于CAPACITY HIGH ，从而获得出色的查询性能。

  • CAPACITY HIGH - 此方法最能压缩内存数据，从而获得良好的查询性能。

inmemory_priority

使用PRIORITY子句为IM列存储中的表数据指定数据填充优先级。该条款控制人口的优先权，但不控制人口的速度。

  • 为按需人口指定NONE 。在这种情况下，当通过全表扫描访问表时，数据库会在IM列存储中填充表数据。如果从不访问该表，或者仅通过索引扫描或通过rowid获取来访问该表，则永远不会发生填充。这是默认值。

  • 为基于优先级的填充指定以下优先级之一： LOW ， MEDIUM ， HIGH或CRITICAL 。在这种情况下，数据库使用内部管理的优先级队列自动填充IM列存储中的表数据;全扫描不是人口的必要条件。数据库根据指定的优先级对表数据的填充进行排队。例如，设置为INMEMORY PRIORITY CRITICAL的表PRIORITY于设置INMEMORY PRIORITY HIGH的表，而该表PRIORITY于设置INMEMORY PRIORITY LOW的表，依此类推。如果IM列存储空间不足，则在空间可用之前，数据库不会填充其他表数据。

inmemory_distribute

只有在使用Oracle Real Application Clusters（Oracle RAC）或Oracle Active Data Guard时， DISTRIBUTE子句才适用。它允许您指定IM列存储中的表数据如何跨Oracle RAC实例分布，并允许您指定数据库中可以填充数据的实例。

AUTO和BY

使用AUTO和BY子句指定IM列存储中的表数据如何跨Oracle RAC实例分布。您可以指定以下选项：

  • AUTO - Oracle数据库控制数据在Oracle RAC实例之间的分布方式。大型表根据其访问模式分布在Oracle RAC实例中。较小的表可以在实例之间分配。这是默认值。

  • BY ROWID RANGE - 某些rowid范围内的数据被分发到不同的Oracle RAC实例。

  • BY PARTITION - 分区中的数据分发到不同的Oracle RAC实例。

  • BY SUBPARTITION - 子分区中的数据分发到不同的Oracle RAC实例。

您只能使用AUTO和BY在单个Oracle RAC数据库中的实例之间分配对象的内存中压缩单元（IMCU），而不是在Active Data Guard中的主实例和备用实例之间。

为了服务

使用FOR SERVICE子句指定可以在其中填充对象的Oracle RAC或Oracle Active Data Guard实例。您可以指定以下选项：

  • DEFAULT - 该对象适用于使用PARALLEL_INSTANCE_GROUP初始化参数指定的所有实例的填充。如果未设置此参数，则会在所有实例上填充该对象。这是默认值。

  • ALL - 无论PARALLEL_INSTANCE_GROUP初始化参数的值如何，该对象都适用于所有实例的填充。

  • service_name - 该对象仅适用于属于指定服务的实例的填充，并且仅在服务处于活动状态且未在实例上阻止时才有资格。

  • NONE - 该对象不适用于任何实例的填充。此选项允许您禁用IM列存储填充，同时保留表的其他In-Memory属性。如果随后通过在ALTER TABLE语句的inmemory_distribute子句中指定FOR SERVICE DEFAULT ， FOR SERVICE ALL或FOR SERVICE service_name ，随后为表启用IM列存储填充，则这些属性将生效。

在Oracle RAC中， FOR SERVICE子句指定Oracle RAC数据库中的实例。在Active Data Guard中，主数据库和备用数据库可以使用单实例或Oracle RAC配置。在Active Data Guard中， FOR SERVICE子句指定主数据库中的实例，备用数据库中的实例或主实例和备用实例的混合。

inmemory_duplicate

只有在工程系统上使用Oracle Real Application Clusters（Oracle RAC）时， DUPLICATE子句才适用。它控制IM列存储中的表数据如何跨Oracle RAC实例复制。您可以指定以下选项：

对PCTTHRESHOLD的限制

您不能为索引组织表的各个分区指定PCTTHRESHOLD 。

mapping_table_clauses

指定MAPPING TABLE以指示数据库创建本地ROWID到物理ROWID的映射，并将它们存储在堆组织表中。需要此映射才能在索引组织表上创建位图索引。如果索引组织表已分区，则映射表也会进行分区，其分区与基表分区具有相同的名称和物理属性。

Oracle数据库在与其父索引组织的表或分区相同的表空间中创建映射表或映射表分区。您无法查询，执行DML操作或修改映射表或其分区的存储特性。

使用prefix_compression子句可以为索引组织表启用或禁用前缀压缩。

  • 指定COMPRESS以为索引组织表启用前缀 压缩 （也称为密钥压缩），从而消除索引组织表中重复出现的主键列值。使用integer指定前缀长度，前缀长度是要压缩的前缀列数。

    前缀长度值的有效范围是从1到主键列数减1。默认前缀长度是主键列数减1。

  • 指定NOCOMPRESS以在索引组织表中禁用前缀压缩。这是默认值。

对索引组织表的前缀压缩的限制

在分区级别，您可以指定COMPRESS ，但不能使用integer指定前缀长度。

index_org_overflow_clause

index_org_overflow_clause允许您指示数据库将超过指定阈值的索引组织表数据行放在此子句中指定的数据段中。

  • 创建索引组织表时，Oracle数据库会评估每列的最大大小以估计可能的最大行。如果需要溢出段但尚未指定OVERFLOW ，则数据库会引发错误并且不执行CREATE TABLE语句。此检查功能可确保对索引组织表的后续DML操作不会失败，因为缺少溢出段。

  • 在OVERFLOW关键字之后在此子句中指定的所有物理属性和存储特性仅适用于表的溢出段。必须在此关键字之前指定索引组织表本身的物理属性和存储特性，其所有分区的默认值以及各个分区的值。

  • 如果索引组织表包含一个或多个LOB列，那么除非您指定OVERFLOW ，否则LOB将被存储在线外，即使它们足够小也可以存储为内联。

  • 如果table被分区，则数据库将溢出数据段与主键索引段分开。

包含 column_name

指定一个列，将索引组织的表行划分为索引和溢出部分。主键列始终存储在索引中。 column_name可以是最后一个主键列，也可以是任何非主键列。跟随column_name所有非主键列都存储在溢出数据段中。

外部表的限制

外部表受以下限制：

  • 外部表不能是临时表。

  • 您只能在外部表上指定以下类型的约束： NOT NULL约束，唯一约束，主键约束和外键约束。指定唯一约束，主键约束或外键约束时，还必须指定RELY DISABLE 。这些约束是声明性的，不会强制执行。它们可以提高查询性能并减少资源消耗，因为可以考虑更多优化器转换。为了使优化器能够利用这些RELY DISABLE约束，必须将QUERY_REWRITE_INTEGRITY初始化参数设置为trusted或stale_tolerated 。

  • 外部表不能包含INVISIBLE列。

CHUNK 整数

指定要为LOB操作分配的字节数。如果integer不是数据库块大小的倍数，则数据库将以字节为单位向下舍入到下一个倍数。例如，如果数据库块大小为2048且integer为2050，则数据库将分配4096个字节（2个块）。最大值为32768（32K），这是允许的最大Oracle数据库块大小。默认的CHUNK大小是一个Oracle数据库块。

CHUNK的值必须小于或等于NEXT的值，默认值或storage_clause中指定的值。如果CHUNK超过NEXT的值，则数据库返回错误。设置后，您无法更改CHUNK的值。

您可以指定PCTVERSION参数，无论数据库是以手动还是自动撤消模式运行。PCTVERSION是手动撤消模式的默认设置。RETENTION是自动撤消模式的默认设置。您不能同时指定PCTVERSION和RETENTION 。

如果已指定SECUREFILE则此子句无效。如果同时指定SECUREFILE和PCTVERSION ，则数据库将以静默方式忽略PCTVERSION参数。

使用subpartition_by_range子句指示数据库应按table每个分区的范围进行子分区。子分区列列表与分区键无关，但受到相同的限制（请参阅列 ）。

您可以使用subpartition_template指定默认子分区特征值。请参阅subpartition_template 。数据库将此值用于此分区中未明确指定特征的任何子分区。

使用subpartition_by_list子句指示数据库应该从column列表中的文字值列表中对表中的每个分区进行子分区。您最多可以指定16个列表子分区键列。

您可以使用subpartition_template指定默认子分区特征值。请参阅subpartition_template 。数据库将此值用于此分区中未明确指定特征的任何子分区。

您还可以使用range_partition_desc或list_partition_desc的list_subpartition_desc为每个分区单独定义列表子分区。如果省略subpartition_template和list_subpartition_desc ，则数据库将创建单个DEFAULT子分区。

如果指定SUBPARTITIONS integer ，则确定table每个分区中的默认子分区数，以及可选择存储它们的一个或多个表空间。默认值为1。如果省略this子句和subpartition_template ，则数据库将使用一个哈希子分区创建每个分区。

关于复合分区的注释

以下说明适用于复合分区：

  • 对于所有子分区，您可以使用range_subpartition_desc ， list_subpartition_desc ， individual_hash_subparts或hash_subparts_by_quantity按名称指定单个子分区，还可以选择其他一些特征。

  • 或者，对于散列和列表子分区：

    • 您可以指定子分区的数量，也可以指定要存储它们的一个或多个表空间。在这种情况下，Oracle数据库会分配SYS_SUBPn形式的子分区名称。

    • 如果省略子分区描述，并且已创建子分区模板，则数据库将使用该模板创建子分区。如果尚未创建子分区模板，则数据库将创建一个哈希子分区或一个DEFAULT列表子分区。

  • 对于所有类型的子分区，如果完全省略子分区描述，则数据库按如下方式分配子分区名称：

    • 如果已指定子分区模板并且已指定分区名称，则数据库将生成partition_name underscore（_） subpartition_name形式的子分区名称（例如， P1_SUB1 ）。

    • 如果尚未指定子分区模板， 或者您已指定子分区模板但未指定分区名称，则数据库将生成SYS_SUBP n形式的子分区名称。

reference_partitioning

使用此子句按引用对表进行分区。通过引用进行分区是一种通过引用约束将正在创建的表（ 子表 ）均分为现有分区表（ 父表 ）的方法。通过引用对表进行分区时，随后在父表上执行的分区维护操作会自动级联到子表。因此，您无法直接在引用分区表上执行分区维护操作。

如果父表是间隔分区表，则在插入子表期间将创建与父表中的间隔分区对应的引用分区子表中的分区。创建子表中的间隔分区时，将从关联的父表分区继承分区名称。如果子表具有表级缺省表空间，则它将用作新间隔分区的表空间。否则，表空间将从父表分区继承。有关引用间隔分区表的更多信息，请参阅“ Oracle数据库VLDB和分区指南” 。

约束

  • 约束中引用的所有外键列必须为NOT NULL 。

  • 指定约束时，不能指定references_clause的ON DELETE SET NULL子句。

  • 约束中引用的父表必须是现有的分区表。它可以通过任何方法进行分区。

  • 外键和父键不能包含任何引用PL / SQL函数或LOB列的虚拟列。

如果在创建父级为区间分区表的引用分区子表时指定此子句，则分区描述符将用于子表的非间隔分区。无法为间隔分区指定分区描述符。
