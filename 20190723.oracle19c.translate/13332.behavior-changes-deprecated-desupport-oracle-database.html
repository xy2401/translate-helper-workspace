关于弃用和不支持的状态

除了新功能之外，Oracle数据库发行版还可以修改，弃用或取消支持功能，并为数据库引入升级行为更改

请注意已弃用和不支持的含义：

  • 不推荐使用的功能是不再增强的功能，但在此版本的Oracle数据库的整个生命周期内仍然受支持。

  • 不再支持的功能是由固定与该功能相关的错误不再支持的功能。Oracle可以选择删除使用该功能所需的代码。如果需要，可以在未来的主要版本中支持不推荐使用的功能。

Grid Infrastructure Management存储库的可选安装

从Oracle Grid Infrastructure 19c开始，Grid Infrastructure Management Repository（GIMR）对于Oracle Standalone Cluster的新安装是可选的。Oracle Domain Services Clusters仍然需要安装GIMR作为服务组件。

Oracle独立群集在Oracle ASM磁盘组或共享文件系统上本地托管GIMR;这个GIMR是一个带有单个可插拔数据库（PDB）的多租户数据库。全局GIMR在Oracle域服务集群中运行。Oracle Domain Services Cluster在单独的Oracle ASM磁盘组中本地托管GIMR。客户端群集（例如Oracle Member Cluster for Database）使用位于Oracle域服务群集上的远程GIMR。对于双节点或四节点群集，为远程群集上的群集托管GIMR可减少在群集上运行额外基础架构存储库的开销。Oracle域服务集群的GIMR是一个多租户数据库，其中包含一个PDB，并为添加的每个成员集群添加了额外的PDB。

Oracle Database 19c中不推荐使用以下Oracle Data Guard代理属性：

  • ArchiveLagTarget

  • DataGuardSyncLatency

  • LogArchiveMaxProcesses

  • LogArchiveMinSucceedDest

  • LogArchiveTrace

  • StandbyFileManagement

  • DbFileNameConvert

  • LogArchiveFormat

  • LogFileNameConvert

使用当前EDIT ...具有这些属性的SET PROPERTY命令继续有效。但是，使用new命令自动进行更新，现在参数数据不再存储在代理元数据文件中。

SQLNET.ENCRYPTION_WALLET_LOCATION参数的弃用

Oracle Database 19c中不推荐使用SQLNET.ENCRYPTION_WALLET_LOCATION sqlnet.ora参数。

SQLNET.ENCRYPTION_WALLET_LOCATION参数定义透明数据加密（TDE）的软件密钥库的位置。要配置软件密钥库位置，Oracle建议您设置WALLET_ROOT初始化参数和TDE_CONFIGURATION动态初始化参数，而不是设置SQLNET.ENCRYPTION_WALLET_LOCATION 。

Oracle建议您尽快使用WALLET_ROOT实例初始化参数，因为该值在实例启动时读取一次，因此所有会话和服务器后台进程在启动后共享相同的路径。如果使用SQLNET.ENCRYPTION_WALLET_LOCATION参数，则可能导致混淆错误配置，因为不同的会话可以具有不同的SQLNET参数值。使用WALLET_ROOT另一个原因是，您可以在其中找到其他功能的钱包，例如Oracle Enterprise User Security和Transport Layer Security。此位置可以成为所有服务器端钱包的主要位置。

查看可能影响您工作的更改。

ALL / USER / DBA_ARGUMENTS用户视图更改

ARGUMENTS视图包含的行数较少。特别是，只有顶级（ DATA_LEVEL=0 ）项存储在ARGUMENTS视图中。

在早期的Oracle数据库版本中，PL / SQL编译器收集了PL / SQL数据类型中所有嵌套类型的元数据。DATA_LEVEL表示类型的嵌套级别。从Oracle Database 18c开始，只有顶级类型元数据（ DATA_LEVEL=0 ）存储在ARGUMENTS视图中。

例如：请注意create-or-replace包中的NestedTypesExample ：

类型Level2Record是RECORD（Field1 NUMBER）;类型Level1Collection是binary_integer的Level2Record索引的表;类型Level0Record是RECORD（Field1 Level1Collection）;过程NestedTypesProc（Param1 Level0Record）;

在以前的Oracle数据库的版本中，顶级类型的的NestedTypeProc程序， parameter Param1, Level0Record ，则返回，同时内的所有嵌套类型的扩展描述Level0Record 。例如：

SQL> select user_name，type_subname，position，sequence，data_level from user_arguments where object_name ='NESTEDTYPESPROC'; ARGUMENT_NAME TYPE_SUBNAME POSITION SEQUENCE DATA_LEVEL --------------- ----------------- ---------- --- ------- --------- PARAM1 LEVEL0RECORD 1 1 0 FIELD1 LEVEL1COLLECTION 1 2 1 LEVEL2RECORD 1 3 2 FIELD1 1 4 3

相反，18.1数据库中的相同查询返回以下内容：

ARGUMENT_NAME TYPE_SUBNAME POSITION SEQUENCE DATA_LEVEL --------------- ----------------- ---------- --- ------- --------- PARAM1 LEVEL0RECORD 1 1 0

在早于Oracle Database 12c（12.1）的版本中，无法以顶级对象类型可访问元数据的方式访问PL / SQL包类型描述性元数据。使用顶级对象类型和集合，您可以查询ALL_TYPES和关联的用户视图ALL_TYPE_ATTRS和ALL_COLL_TYPES ，以获取类型元数据。但是，在Oracle Database 12.1之前，无法获取PL / SQL包类型的类型元数据，例如记录和打包的集合。引用这些PL / SQL包类型的函数或过程参数导致在ARGUMENTS视图中发布有关这些类型的所有元数据，包括任何嵌套类型。

这种方法的问题在于深层嵌套类型会占用SYS表空间中的大量内存。此外，由于无法在ARGUMENTS视图中共享类型元数据，因此具有深层嵌套类型的每个参数都需要其自己的类型元数据的冗余副本。ARGUMENTS视图和SYS表空间中的元数据量可能会导致各种问题，包括PL / SQL编译器性能下降。导致降级的原因是PL / SQL更新基础字典表中的行所需的时间。

在Oracle Database 12.1发行版中，PL / SQL引入了对包类型的增强支持，包括新用户视图ALL_PLSQL_TYPES ， ALL_PLSQL_TYPE_ATTRS和ALL_PLSQL_COLL_TYPES 。顾名思义，这些视图类似于ALL_TYPES视图系列。但是，您可以使用增强的PL / SQL类型视图来查询有关PL / SQL包类型的元数据，而不是顶级对象和集合类型。

由于Oracle Database 12.1中添加了包类型，因此不再需要在ARGUMENTS视图中插入大量描述性元数据。包含类型名称的单行元数据是ARGUMENTS视图中每种参数类型所需的全部元数据。您可以根据PL / SQL类型视图和任何嵌套类型获取查询中类型名称的完整描述。

OCIDescribeAny()基于ARGUMENTS视图使用的相同元数据。OCIDescribeAny()还为每个参数类型返回一行，而不是Oracle Database 12.1中更改之前通常返回的多行。

ALL/DBA/USER_ARGUMENTS包含新的列类型TYPE_OBJECT_TYPE 。要确定TYPE_OWNER ， TYPE_NAME和TYPE_SUBNAME描述的类型类型，请使用TYPE_OBJECT_TYPE列。可能的值包括TABLE ， VIEW ， PACKAGE和TYPE 。

如果您希望继续在ARGUMENTS视图中收集ALL_TYPES和关联的用户视图ALL_TYPE_ATTRS和ALL_COLL_TYPES，则可以将事件设置为events='10946, level 65536' 。设置此事件会将ARGUMENTS视图恢复为早于12.1的Oracle数据库版本中的行为，其中DATA_LEVEL可以大于0 ，并且视图中包含类型和任何嵌套类型的描述性元数据。如果进行此更改，则必须在设置事件后重新编译受影响的包。重新编译受影响的包时，编译器将重新收集其他元数据。此事件还将OCIDescribeAny()恢复为早于12.1的Oracle数据库版本中的行为。

从Oracle Database 12 c第 1版（12.1.0.2）开始，如果输入不带参数的过程，则ARGUMENTS视图不包含任何行。此更改是与ARGUMENTS视图的行减少更改分开的其他更改。在Oracle Database 12.1.0.2之前，在ARGUMENTS视图中将没有参数的过程显示为单行。

USER / ALL / DBA_IDENTIFIERS用户视图更改

从Oracle Database 18c开始，增强了PL / Scope以捕获有关PL / SQL代码中用户标识符的其他信息。附加信息包括对标识符的约束，以及指示函数何时是PL / SQL中内置SQL的指示符。

以下列是Oracle Database 18c中USER / ALL / DBA_IDENTIFIERS视图中的新增内容：

  • CHARACTER_SET ：当列用于变量标识符声明时，此列包含字符集子句的值。当字符集派生自另一个变量标识符时，可能的值为CHAR_CS ， NCHAR_CS和IDENTIFIER 。

  • ATTRIBUTE ：此列包含在变量声明中使用% attribute时的属性值。可能的值为ROWTYPE ， TYPE和CHARSET 。

  • CHAR_USED ：此列包含在字符串长度约束声明中使用约束时的长度约束的类型。可能的值是CHAR和BYTE 。

  • LENGTH ：此列包含字符串长度约束声明的数字长度约束值。
