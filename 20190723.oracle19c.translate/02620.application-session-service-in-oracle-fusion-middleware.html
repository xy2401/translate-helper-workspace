8 Oracle融合中间件中的应用程序会话服务

Real Application Security在Oracle融合中间件中提供应用程序会话服务，以透明和安全地设置应用程序会话，以支持现有应用程序用户，角色和安全上下文。此应用程序会话服务是一个servlet过滤器，负责应用程序会话设置和应用程序可以与应用程序会话一起使用的一组API。此应用程序会话服务支持Oracle融合中间件在外部管理的用户和角色。

从Oracle数据库12开始c解除1（12.1.0.2），这个应用会话服务支持使用Oracle平台安全服务（OPSS）为应用安全提供的Java EE Web应用程序。此应用程序会话服务可以与应用程序一起部署到OPSS可以支持的Java EE容器中。

关于Real Application Security Concepts

作为Oracle数据库授权系统，Real Application Security通过强制谁（应用程序用户）可以对哪些数据库资源（我的报告下的员工的采购订单记录，我的SSN）执行哪些应用程序级操作（ApprovePurchaseOrder，ViewSSN）来支持应用程序安全性。应用程序会话用于强制实施应用程序安全性。通常，在外部配置用户和角色，即，在身份存储中配置企业用户，并在策略存储中管理应用程序角色，例如Oracle身份管理和Oracle授权服务器（OES）。

应用程序用户和角色在外部管理

Real Application Security支持由外部方配置的用户和角色，例如Oracle Entitlement Server，用于管理应用程序用户和角色配置，而OPSS则提供运行时安全框架，用于强制执行应用程序角色的安全性。这些被称为外部应用程序用户和应用程序角色（有关详细信息，请参阅Oracle融合中间件与Real Application Security的集成 。）

Real Application Security还具有在数据库中本机管理的应用程序的用户和角色，这些用户和角色称为Real Application Security应用程序用户和应用程序角色（有关详细信息，请参阅配置应用程序用户和应用程序角色 ）。

对于外部应用程序用户和应用程序角色，Real Application Security不管理用户配置，包括用户的角色分配。但是，对于数据库中的本机应用程序用户和应用程序角色，在数据库中管理应用程序用户的应用程序角色，应用程序角色的数据库角色以及应用程序角色的应用程序角色。应用程序会话支持Real Application Security应用程序用户和应用程序角色以及外部应用程序用户和应用程序角色，并且可以在数据安全策略中使用。可以向在外部或在数据库中本地管理的用户授予应用程序特权。

Oracle融合中间件中的应用程序会话

应用程序会话表示应用程序用户的运行时安全上下文，其中包括用户标识，数据库和应用程序角色以及名称空间属性值。Oracle融合中间件中的应用程序会话使用外部管理的用户和角色。有关配置应用程序会话的详细信息，请参阅配置应用程序会话。

Oracle融合中间件中的会话管理器

在Real Application Security中，会话管理器授权应用程序会话操作，并具有创建或修改应用程序会话所需的权限。在应用程序代码或应用程序的数据库连接不应该有这些特权。对于数据库，会话管理器是Real Application Security直接登录用户（请参阅“ 关于创建直接登录应用程序用户帐户 ” ）。它在应用程序会话服务初始化开始时与数据库通信，以基于授权凭证与数据库服务器建立信任关系。随后使用此机制来代表应用程序进一步授权应用程序会话操作。

Oracle融合中间件中的动态角色

除应用程序角色外，应用程序会话还支持动态角色。这是一种必须在数据库中本机定义的Real Application Security角色（请参阅“ 动态应用程序角色 ” ）。此角色未授予用户或其他角色。必须在运行时在应用程序会话中以编程方式启用它。这可以通过Real Application Security过滤器自动完成，也可以由受信任的应用程序代码明确完成。

动态角色可以定义为请求范围或会话范围。会话范围表示在下次附加时仍启用启用的动态角色，除非您明确指定在下次附加时禁用它。请求范围表示在从连接分离应用程序会话后禁用该角色。

动态角色有两个一般用途：

  • 对象权限

    应用程序用户不是数据库用户。在外部标识存储中配置应用程序用户和角色时，可以将这些对象特权授予Real Application Security动态角色。当Real Application Security过滤器为应用程序用户设置应用程序会话时，它会在访问当前应用程序的每个应用程序会话中启用动态角色。动态角色仅适用于当前应用程序。

  • 应用程序会话权限提升

    某些受信任的应用程序代码必须暂时具有更高的权限才能执行某些数据库操作通过在使用基于Java代码的策略声明的受信任代码附加应用程序会话期间启用Real Application Security动态角色来支持此功能。分离时应该禁用该角色。

    一个用例是应用程序命名空间设置，其中会话命名空间属性以细粒度方式在Real Application Security中得到保护。必须在数据库中将命名空间预定义为命名空间模板。根据定义，可以在关联的命名空间ACL中指定授权策略，即谁（用户/角色）可以对命名空间执行什么操作（modify_namespace，modify_attribute）。要确保只有受信任的应用程序代码可以修改名称空间属性，才会将权限授予动态角色。此外，动态角色只能由Java代码权限标识的某些受信任的应用程序代码以编程方式启用。这支持只有受信任的代码才能设置某些名称空间的用例。

关于应用程序会话筛选器

Real Application Security应用程序会话筛选器是实现javax.servlet.的标准Java EE servlet筛选器javax.servlet.Filter界面。此过滤器的基本功能是根据经过身份验证的用户的安全上下文（OPSS主题）透明地设置应用程序会话。

此应用程序会话筛选器允许应用程序之间不断共享应用程序会话。它不能为每个请求创建，但必须绑定到有状态上下文并重新用于同一用户，直到注销。对于Web应用程序，http会话就是这样的上下文。它由容器维护，用于同一用户从登录到注销的连续访问，跨多个单点登录应用程序或容器。

始终可以从ServletFilter访问http会话对象，但可能无法从通用应用程序代码访问它。

关于应用程序会话筛选器操作

应用程序会话筛选器按以下方式设置应用程序会话：

  • 它在用户首次访问时创建应用程序会话。

    如果用户已登录，则会在身份验证上下文（OPSS主题）中以用户身份创建应用程序会话。

    如果用户尚未登录，则会以匿名用户身份创建应用程序会话。

  • 它重用现有的应用程序会话实例，以便用户随后访问同一个应用程序。

  • 当多个应用程序访问同一个Real Application Security数据库时，它在多个应用程序之间共享相同的应用程序会

  • 它在每个http请求开始时同步应用程序会话，以确保当前应用程序会话中的用户和角色始终与身份验证上下文（OPSS主题）同步，并且仅为每个应用程序会话启用已配置的动态角色。

    通过将OPSS主题值推送到服务器并获取当前应用程序会话的服务器计算值来完成同步。

在应用程序代码执行之前触发过滤器后，应用程序会话中的用户和角色将得到修复。过滤器负责同步用户和角色，而不是应用程序代码。

应用程序代码负责命名空间设置。过滤器只能帮助恢复以前的命名空间。有关命名空间设置的详细信息，请参阅“ 关于命名空间API ” 。

应用程序会话基于http会话ID在本地缓存。 http会话由容器管理。Real Application Security具有应用程序会话侦听器，用于侦听容器的应用程序会话事件。当容器使http会话无效时，Real Application Security监听器将从本地缓存中删除应用程序会话。

关于部署

Real Application Security应用程序会话服务在一个jar文件xsee.jar 。Oracle建议您将xsee.jar jar文件部署到公共目录，而不是与Web应用程序（ web-inf/lib WAR文件）一起部署。通过这种方式，您可以将jar与应用程序代码分开，并将一些特殊的基于代码的权限仅xsee.jar jar文件，而不是应用程序代码。

要使xsee.jar jar文件从CSF存储获取会话管理器的凭据，必须将基于代码的权限CredentialAccessPermission授予xsee.jar jar文件。过滤器内部使用Real Application Security会话管理器来授权会话操作。

在例8-1中 ， xsee.jar jar文件被部署到WLS的domain /lib目录。假设会话管理器的键/映射使用默认值，java策略文件（ system-jazn-data.xml ）具有CredentialAccessPermission授权。

有关部署说明，请参阅了解Oracle WebLogic Server中有关标准Java EE部署的部分。

有关部署应用程序以进行测试或评估的简单快速方法，请使用自动部署。这是通过将所有内容（类， web.xml ）打包到一个WAR文件并将其复制到Weblogic autodeploy目录来部署应用程序会话服务的更简单方法。请参阅“将应用程序部署到Oracle WebLogic Server”中有关在开发域中自动部署应用程序的部分。

要创建会话管理器的凭据，请参阅“ 手动配置 ”中的步骤2以获取更多信息。

关于应用程序会话筛选器的应用程序配置

过滤器配置在应用程序的web.xml配置文件以标准方式。它可以配置为仅应用于特定URL。这避免了不需要数据库访问的某些页面的不必要的应用程序会话设置。过滤器假定已完成用户身份验证并已建立身份验证上下文。在OPSS中，用户的应用程序上下文是在OPSS过滤器上计算的，因此必须在过滤器链中的应用程序会话过滤器之前部署OPSS过滤器。应用程序会话筛选器使用以下web.xml参数：

  • application.datasource

    该应用程序使用此application.datasource参数。应用程序会话筛选器需要此参数用于初始化，应用程序会话设置和命名空间操

  • dynamic.roles

    要使用的Real Application Security动态角色列表由逗号（，）分隔。必须已在数据库中创建动态角色作为会话范围;否则，抛出以下异常： ORA-46055: invalid role specified 。

    为当前应用程序中的每个应用程序会话启用角色，并在其他应用程序中自动禁用。请注意，为匿名会话启用了这些动态角色。如果每个应用程序会话不需要动态角色，则不应过度授予任何权限。通常，只应为动态角色授予对象权限。

    对于不受Real Application Security保护的任何表，应用程序仍可以灵活地使用数据库连接池用户进行访问，而不是应用程序用户。在这种情况下，不需要附加应用程序会话API调用，也不会为动态角色授予对象权限。

  • session.manager.pwd.key和session.manager.pwd.map

    session.manager.pwd.key参数和session.manager.pwd.map参数（固定为oracle.rdbms.ras ）指向凭证存储中的凭证（用户ID和密码）。session.manager.pwd.key参数用于检索会话管理器的凭据。目前，OPSS CSF凭证存储用于存储凭证，CSF API用于在运行时检索凭证。此外，可以从商店检索会话管理器的用户ID和密码。

    session.manager.pwd.key参数的默认值为default 。如果应用程序使用默认凭据，则可以省略此参数。

    如果应用程序想要使用特定的会话管理器而不是默认凭据，则应用程序的管理员必须使用不同的密钥名称创建凭据，并使用此参数对其进行配置。有关详细信息，请参阅“ Oracle Application Server Containers for J2EE安全指南”中的配置OPSS安全性存储。

  • session.manager.pool.min和session.manager.pool.max

    会话管理器的连接还用于查询中间层的数据安全策略（ACL）。此连接作为池进行管理。session.manager.pool.min参数确定池的最小大小。此参数是可选的。默认值为1。

    session.manager.pool.max参数确定池的最大大小。此参数是可选的。默认值为3。

    如果不需要权限检查，则session.manager.pool.min和session.manager.pool.max值的池大小应设置为1。

先决条件

使用必须在Oracle融合中间件的Java EE容器中部署和配置Real Application Security，即应用程序会话服务和OPSS。

对于WebLogic服务器，先决条件包括：

  • 基于JRF的WLS域（OPSS内置）通过Oracle数据库12c JDBC驱动程序认证。所需的JDBC jar可能很多，而不仅仅是一个驱动程序jar，具体取决于您需要的功能（UCP，I18N，SQLXML等）。

  • Oracle Database 12 c第1版（12.1）及更高版本

对于WebLogic Server 10.3.6和12.1.2 JRF版本（Oracle Fusion Middleware的一部分），所提供的JDBC驱动程序不兼容Oracle Database 12 c 。您必须获取Oracle Database 12 c JDBC jar（ ojdbc6.jar或ojdbc7.jar以及其他匹配的jar，具体取决于您需要的功能），并将这些jar添加到WebLogic Server的类路径的前面。有关详细说明，请参阅管理Oracle WebLogic Server的JDBC数据源 ，第B节。

如果JDBC驱动程序和数据库之间存在版本不匹配，则Real Application Security筛选器初始化将失败并显示错误消息。例如，

  • 如果Oracle Database 11g JDBC驱动程序与Oracle Database 12 c一起使用 ，则服务器日志中将显示以下错误消息： Fail to initialize RAS session manager due to method missing 。

  • 如果Oracle数据库12c的 JDBC驱动程序正在使用Oracle数据库11克使用，出现以下错误消息在服务器日志： ORA-00439: feature not enabled: Fusion Security 。

手动配置

请遵循以下手动配置步骤，以便应用程序使用应用程序会话服务。这些步骤应适用于WebLogic 10.3.6和12.1.2，JRF版本。

  1. 安装Real Application Security jar。

    将xsee.jar和xs.jar （ ORACLE_HOME/jlib/ ）复制到应用程序可以使用的公共目录。对于WebLogic，一个好的位置是DOMAIN_HOME/lib 。这允许部署在同一域中的许多应用程序共享Real Application Security jar。

  2. 创建Real Application Security会话管理器凭据。

    如“ 关于应用程序会话筛选器的应用程序配置”中所述 ，必须在OPSS的凭证存储中创建会话管理器的凭据。这可以使用OPSS脚本完成。有关如何使用OPSS脚本的详细信息，请参阅“ Oracle Application Server Containers for J2EE安全指南”中有关OPSS脚本的部分。 

    createCred（map ='oracle.rdbms.ras'，key ='default'，user ='myUsr'，password ='myPassword'）

    会话管理器的凭证存储在为域配置的默认凭证存储中。map名称必须是oracle.rdbms.ras ，它是为Real Application Security应用程序会话服务预定义的。这是固定的，无法更改。

  3. 授予Real Application Security jar文件的代码权限。

    如“ 关于部署 ”中所述 ，必须将CSF权限授予xsee.jar文件。这也是使用OPSS脚本完成的。 

    grantPermission（codeBaseURL ='file：$ {domain.home} /lib/xsee.jar'，permClass ='oracle.security.jps.service.credstore。CredentialAccessPermission'，permTarget ='context = SYSTEM，mapName = oracle.rdbms.ras，keyName = *'，permActions ='read'）

    请注意，上面的keyName (*)适用于所有键。如果为特定应用程序创建非默认密钥，则不需要进一步授予。

  4. 配置web.xml ，调用Real Application Security API（附加/分离），以及构建/部署应用程序。请参见示例8-2以了解如何配置web.xml 。

    这些是标准的Java EE开发过程。

    如果在应用程序代码中调用attachSessionPrivileged API，则必须按照“ 关于权限提升API ”中的讨论，将SessionCodePermission授予应用程序代码。这与第3步类似。这是一个例子：

    grantPermission（codeBaseURL ='file：$ {domain.home} /servers/DefaultServer/tmp/_WL_user/MyWar/pi47ig/war/WEB-INF/lib/trusted.jar',permClass ='oracle.security.xs.ee。会话。SessionCodePermission'，permTarget ='MY_NS_DROLE，permActions ='attach'）grantPermission（codeBaseURL ='file：$ {domain.home} /lib/xsee.jar',permClass ='oracle.security.xs.ee.session。SessionCodePermission'，permTarget ='MY_NS_DROLE，permActions ='attach'）

    OPSS脚本要求WLS管理服务器正在运行。此手动方法仅支持在线配置。第4步始终由应用程序管理员负责，而步骤1到3可以自动执行，如“ 关于自动配置 ”中所述 。

关于使用应用程序命名空间的重要知识

以下用法信息总结了有关使用应用程序命名空间的重要信息。

  • Real Application Security过滤器将所有应用程序命名空间缓存到当前应用程序会话。

    • 对于首次访问，必须在数据库中创建新的应用程序会话。目前尚未设置应用程序命名空间。

    • 对于用户的后续访问，过滤器始终会带来为应用程序会话创建的所有名称空间，并将它们缓存在JVM中的当前应用程序会话中。

  • 应用程序代码始终从当前应用程序会话访问命名空间。每个更新操作都是到服务器的往返，以更改表和当前应用程序会话（JVM）中的值。这就是为什么每个更新API都有一个数据库连接参数。但是，read属性是一个本地操作，用于在不访问数据库的情况下从JVM中的当前应用程序会话中读取。

  • 只要成功完成命名空间更改，更改就会传播到已连接的应用程序会话以及新连接的应用程序会话，因为所有这些连接的应用程序会话都引用单个源 - 当前应用程序会话。

  • 当前应用程序会话中的命名空间在Web应用程序的http请求范围内是一致的。甚至命名空间也可以随时由其他应用程序更改。只有在Real Application Security过滤器的当前http请求开始时才会获取一次更改。在同一http请求中发生的所有附件都引用当前应用程序会话中的相同名称空间。

  • 应用程序代码可以完全控制更改命名空间值。它可以随时读取当前应用程序会话的命名空间，并决定是否通过调用命名空间API来更新命名空间。
