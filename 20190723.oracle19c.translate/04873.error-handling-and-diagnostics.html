8错误处理和诊断

应用程序必须预测运行时错误并尝试从中恢复。本章提供了有关错误报告和恢复的深入讨论。您将学习如何使用ANSI状态变量SQLCODE和SQLSTATE或Oracle SQLCA（SQL通信区域）结构来处理警告和错误。您还将学习如何使用WHENEVER语句以及如何使用Oracle ORACA（Oracle通信区域）结构诊断问题。

8.1为什么需要进行错误处理

每个应用程序的重要部分都必须用于错误处理。错误处理的主要好处是它使您的程序能够在出现错误的情况下继续运行。错误源于设计错误，编码错误，硬件故障，无效的用户输入和许多其他来源

您无法预料到所有可能的错误，但您可以计划处理对您的程序有意义的某些类型的错误。对于Pro * COBOL，错误处理意味着检测并从SQL语句执行错误中恢复。您必须捕获错误，因为除非您停止处理，否则预编译器将继续运行而不管遇到的错误。

您还可以准备处理警告，例如“值截断”和状态更改，例如“数据结束”。在每个数据操作语句之后检查错误和警告条件尤为重要，因为在处理表中的所有符合条件的行之前，INSERT，UPDATE或DELETE语句可能会失败。

8.2.1 SQLCA

SQLCA是一种类似记录的主机语言数据结构，包括Oracle警告，错误号和错误文本。Oracle在每个可执行的 SQL或PL / SQL语句之后更新SQLCA。（在声明性陈述之后，SQLCA值未定义。）通过检查存储在SQLCA中的返回码，您的程序可以确定SQL语句的结果。这可以通过两种方式完成：

  • 使用WHENEVER语句进行隐式检查

  • 显式检查SQLCA变量

当您使用WHENEVER语句隐式检查SQL语句的状态时，Pro * COBOL会在每个可执行语句后自动插入错误检查代码。或者，您可以显式编写自己的代码来测试SQLCA结构的SQLCODE成员的值。使用嵌入式SQL INCLUDE语句包含SQLCA：

EXEC SQL包含SQLCA END-EXEC。

8.3使用SQL通信区域

Oracle使用SQL通信区域（SQLCA）来存储在运行时传递给程序的状态信息。SQLCA是一个类似记录的COBOL数据结构，它在每个可执行的SQL语句之后更新，因此它始终反映最新SQL操作的结果。其字段包含Oracle在执行SQL语句时更新的错误，警告和状态信息。要确定结果，可以使用自己的COBOL代码或使用WHENEVER语句隐式检查SQLCA中的变量。

当MODE = {ORACLE | ANSI13}，SQLCA是必需的;如果未声明SQLCA，则会发生编译时错误。当MODE = {ANSI |时，SQLCA是可选的ANSI14}，但是如果要使用WHENEVER SQLWARNING语句，则必须声明SQLCA。使用多字节NCHAR主机变量时，还必须包含SQLCA。

注意：

当您的应用程序使用Oracle Net Services同时访问本地和远程数据库的组合时，所有数据库都会写入一个SQLCA。每个数据库没有不同的SQLCA。有关更多信息，请参阅“ 并发登录 ” 。

8.3.3错误报告的关键组件

Pro * COBOL错误报告的关键组件取决于SQLCA中的几个字段。

8.3.3.1状态代码

每个可执行的SQL语句都返回SQLCA变量SQLCODE中的状态代码，您可以使用WHENEVER SQLERROR隐式检查或使用您自己的COBOL代码显式检查。

8.3.3.2警告标志

在SQLCA变量SQLWARN0到SQLWARN7中返回警告标志，您可以使用WHENEVER SQLWARNING或您自己的COBOL代码进行检查。这些警告标志对于检测不被视为错误的运行时条件很有用。

8.3.3.3行处理计数

最近执行的SQL语句处理的行数在SQLCA变量SQLERRD（3）中返回。对于OPEN游标上的重复FETCH，SQLERRD（3）保持获取的行数的总计。

8.3.3.4解析错误偏移量

在执行SQL语句之前，Oracle必须解析它;检查它以确保它遵循语法规则并引用有效的数据库对象。如果Oracle发现错误，则会在SQLCA变量SQLERRD（5）中存储偏移量，您可以显式检查该变量。偏移量指定SQL语句中解析错误开始时的字符位置。第一个角色占据零位。例如，如果偏移量为9，则解析错误从第十个字符开始。

如果您的SQL语句不会导致解析错误，Oracle会将SQLERRD（5）设置为零。如果解析错误从第一个字符（占据零位置）开始，Oracle也会将SQLERRD（5）设置为零。因此，仅当SQLCODE为负时才检查SQLERRD（5），这意味着发生了错误。

8.3.3.5错误消息文本

SQLCA变量SQLERRMC中提供了错误代码和错误消息。例如，您可以将以下语句放在错误处理例程中：

*处理SQL执行错误。将SQLERRMC移动到错误消息。显示错误消息。

最多存储消息文本的前70个字符。对于超过70个字符的消息，必须调用SQLGLM子例程，这将在“ 获取错误消息的全文 ”中讨论。

8.3.5 PL / SQL注意事项

当Pro * COBOL程序执行嵌入式PL / SQL块时，并未设置SQLCA中的所有字段。例如，如果块提取多行，则行处理的计数SQLERRD（3）将设置为1， 而不是实际提取的行数。因此，在执行PL / SQL块之后，您应该只依赖SQLCA中的SQLCODE和SQLERRM字段。

8.3.6获取错误消息的全文

无论MODE的设置如何，如果已显式声明SQLCODE且未包含SQLCA，则可以使用SQLGLM获取错误消息的全文。 SQLCA可以容纳长达70个字符的错误消息。要获取更长（或嵌套）错误消息的全文，需要SQLGLM子例程。

如果连接到数据库，则可以使用语法调用SQLGLM

8.3.8 WHENEVER指令

默认情况下，Pro * COBOL会忽略错误和警告条件并继续处理（如果可能）。要执行自动条件检查和错误处理，您需要WHENEVER语句。

使用WHENEVER语句，您可以指定Oracle检测到错误，警告条件或“未找到”条件时要采取的操作。这些操作包括继续下一个语句，执行段落，分支到段落或停止。

8.3.8.1条件

您可以让Oracle自动检查SQLCA是否满足以下任何条件。

8.3.8.2 SQLWARNING

设置SQLWARN（0）是因为Oracle返回了警告（警告标志之一，SQLWARN（1）到SQLWARN（7），也设置了）或SQLCODE具有+1403以外的正值。例如，当Oracle将截断的列值分配给输出主机变量时，会设置SQLWARN（1）。

当MODE = {ANSI |时，声明SQLCA是可选的ANSI14}。但是，要使用WHENEVER SQLWARNING， 必须声明SQLCA。

注意：

您必须包含SQLCA才能使用此功能。

8.3.8.3 SQLERROR

如果Oracle返回错误，则SQLCODE具有负值。

8.3.8.4找不到或找不到

当达到提取结束时，SQLCODE的值为+1403（当MODE = {ANSI | ANSI14 | ANSI13}时或当END_OF_FETCH = 100时为+100）。当已获取满足搜索条件的所有行或没有符合该条件的行时，可能会发生这种情况。

您可以使用END_OF_FETCH选项覆盖MODE宏选项使用的值。

END_OF_FETCH = 100 | 1403（默认1403）

8.3.8.5行动

您可以使用WHENEVER语句指定以下操作。

8.3.8.6继续

如果可能，您的程序将继续运行下一个语句。这是默认操作，相当于不使用WHENEVER语句。您可以使用它来“关闭”条件检查。

8.3.8.7 DO CALL

您的程序调用嵌套的子程序。到达子程序结束时，控制转移到失败的SQL语句后面的语句。

8.3.8.8 DO PERFORM

您的程序将控制转移到COBOL部分或段落。到达该部分的末尾时，控制转移到失败的SQL语句之后的语句。

EXEC SQL WHENEVER <condition> DO PERFORM <section_name> END-EXEC。
8.3.8.9 GOTO或转到

您的程序分支到指定的段落或部分。

8.3.8.10停止

您的程序停止运行，并且回滚未提交的工作。

小心。注销前，STOP操作不显示任何消息。

注意：

虽然在生成的代码中，EXEC SQL WHENEVER SQLERROR STOP转换为IF SQLCODE IN SQLCA等于1403然后停止RUN END-IF，Oracle服务器将负责回滚未提交的数据。

8.3.9编写WHENEVER声明

使用以下语法对WHENEVER语句进行编码：

EXEC SQL WHENEVER <condition> <action> END-EXEC。
8.3.9.1 DO PERFORM

使用WHENEVER时......DO PERFORM声明，适用于执行段落或部分的通常规则。但是，您不能使用THRU，TIMES，UNTIL或VARYING子句。

例如，以下WHENEVER ......DO声明无效 ：

程序部门。 *无效语句EXEC SQL WHENEVER SQLERROR DO PERFORM DISPLAY-ERROR THRU LOG-OFF END-EXEC。 ...DISPLAY-ERROR。 ...注销。 ...

在以下示例中，WHENEVER SQLERROR DO PERFORM语句用于处理特定错误：

程序部门。主要部分。 MSTART。 ...执行SQL WHENEVER SQLERROR DO执行INS-ERROR END-EXEC。 EXEC SQL INSERT INTO EMP（EMPNO，ENAME，DEPTNO）值（：EMP-NUMBER，：EMP-NAME，：DEPT-NUMBER）END-EXEC。执行SQL WHENEVER SQLERROR DO PERFORM DEL-ERROR END-EXEC。 EXEC SQL DELETE FROM DEPT WHERE DEPTNO =：DEPT-NUMBER END-EXEC。 ...MEXIT。停止运行。 INS-ERROR部分。 INSSTART。 *检查“重复键值”Oracle错误如果SQLCA.SQLCODE = -1 ...*检查“值太大”Oracle错误ELSE如果SQLCA.SQLCODE = -1401 ...ELSE ......万一。 ...INSEXIT。出口。 * DEL-ERROR部分。 DSTART。 *检查处理的行数。如果SQLCA.SQLERRD（3）= 0 ...ELSE ......万一。 ...DEXIT。出口。

请注意段落如何检查SQLCA中的变量以确定操作过程。

8.3.9.2 DO CALL

该子句调用动作子程序。以下是此子句的语法：

EXEC SQL WHENEVER <condition> DO CALL <subprogram_name> [USING <param1> ...]END-EXEC。

以下限制或规则适用：

  • 您不能在USING子句中使用RETURNING，ON_EXCEPTION或OVER_FLOW短语。

  • 您可能必须在COBOL源代码的PROGRAM-ID语句中输入子程序名称，后跟关键字COMMON。

  • 您必须在动作子程序中使用WHENEVER CONTINUE语句。

  • 动作子程序名称可能必须在WHENEVER指令的DO CALL子句中用双引号括起来。

8.3.9.3范围

因为WHENEVER是一个声明性陈述，它的范围是位置的，而不是逻辑的。它测试源文件中跟随它的所有可执行SQL语句，而不是程序逻辑流程。因此，在要测试的第一个可执行SQL语句之前编写WHENEVER语句。

WHENEVER语句一直有效，直到被另一个检查相同条件的WHENEVER语句取代。

建议 ：您可以将WHENEVER语句放在包含SQL语句的每个程序单元的开头。这样，一个程序单元中的SQL语句将不会在另一个程序单元中引用WHENEVER操作，从而在编译或运行时导致错误。
