支持Oracle Clusterware升级的干运行验证

从Oracle Grid Infrastructure 19c开始，Oracle Grid Infrastructure安装向导（ gridSetup.sh ）使您能够执行干运行模式升级以检查系统的升级准备情况。

在干运行升级模式下，安装向导会执行在实际升级中执行的所有系统准备情况检查，并使您能够在开始升级之前验证系统是否已准备好进行升级。此模式不执行实际升级。它有助于预测系统设置的潜在问题并避免升级失败。

多个ASMB

鉴于+ ASM1已安装dg1但未安装dg2，并且+ ASM2已安装dg2但未安装dg1，则多个ASMB项目允许数据库通过同时连接到两个ASM实例来同时使用dg1和dg2。我们现在可以拥有ASMBn，而不仅仅是ASMB。

此功能允许DB使用多个磁盘组，即使给定的ASM实例没有安装所有磁盘组，也可以提高Real Application Clusters（RAC）堆栈的可用性。

奇偶校验保护文件

REDUNDANCY文件类型属性指定文件组的冗余。PARITY值指定冗余的单奇偶校验。奇偶校验设置适用于一次写入文件，例如存档日志和备份集。

当两个或三个ASM镜像用于与数据库备份操作相关联的文件时，会消耗大量空间。备份文件是一次写入文件，此功能允许保护奇偶校验而不是传统镜像。结果可节省大量空间。

零停机Oracle Grid Infrastructure补丁

零停机Oracle Grid Infrastructure Patching可以在不中断数据库操作的情况下修补Oracle Grid Infrastructure。修补程序以非滚动方式应用，一次修补一个节点，同时节点上的数据库实例保持运行。零停机Oracle Grid Infrastructure Patching支持具有两个或更多节点的集群上的Oracle Real Application Clusters（Oracle RAC）数据库。

零停机时间网格基础结构修补通过允许客户执行Oracle Grid Infrastructure的滚动补丁而不中断正在修补的节点上的数据库操作而不影响这些数据库实例的容量或性能，显着提高了数据库可用性。

使用Oracle Fleet补丁和配置实现零停机Oracle Grid Infrastructure补丁

零停机时间Oracle Grid Infrastructure Patching可以在不影响Oracle RAC数据库实例的情况下应用一次性Oracle Grid Infrastructure修补程序。使用Oracle Fleet Patching and Provisioning（Oracle FPP）在整个群集中以滚动方式应用修补程序。此功能适用于具有两个或更多节点的所有Oracle Real Application Cluster，但目前仅适用于一次性补丁（不是RU和RUR）。

使用Oracle Fleet Patching和Provisioning应用一次性Oracle Grid Infrastructure补丁，数据库实例停机时间为零，可以减少对Oracle RAC数据库实例的用户和服务中断的影响，从而减少对nil的影响。使用以前的数据库版本，您必须在应用Oracle Grid Infrastructure修补程序之前关闭数据库实例，这显然会影响企业操作。

支持OCR和表决磁盘的直接文件放置

从Oracle Grid Infrastructure 19c开始，Oracle Standalone Clusters将取消对共享文件系统上直接OCR和Voting磁盘文件放置的支持。对于Oracle域服务集群，要求将OCR和表决文件置于Oracle自动存储管理（Oracle ASM）中，托管在共享文件系统上并用作ASM磁盘的文件之上。

在Oracle Grid Infrastructure 12c第2版（12.2）中，Oracle宣布它将不再支持直接在共享文件系统上放置Oracle Grid Infrastructure Oracle Cluster Registry（OCR）和投票文件。这种支持现在已经废除。从Oracle Grid Infrastructure 19c（19.3）开始，使用Oracle Standalone Clusters，您可以再次将OCR和Voting磁盘文件直接放在共享文件系统上。

Grid Infrastructure Management存储库的可选安装

从Oracle Grid Infrastructure 19c开始，Grid Infrastructure Management Repository（GIMR）对于Oracle Standalone Cluster的新安装是可选的。Oracle Domain Services Clusters仍然需要安装GIMR作为服务组件。

GIMR中包含的数据是基于应用机器学习的预防性诊断的基础，可以帮助提高Oracle Real Application Clusters（Oracle RAC）数据库的可用性。对GIMR进行可选安装可以实现更灵活的存储空间管理和更快的部署，尤其是在安装测试和开发系统期间。

不推荐的功能

查看从Oracle Grid Infrastructure 19c开始不推荐使用的功能。

有关不推荐使用的功能，参数和视图的更多信息，请参阅“ Oracle数据库升级指南”

  • Addnode脚本的弃用

    Oracle Grid Infrastructure 19c中不推荐使用addnode脚本。安装程序向导中提供了向群集添加节点的功能。

    可以在将来的版本中删除addnode脚本。不使用addnode脚本（ addnode.sh或addnode.bat ），而是使用安装程序向导添加节点。安装程序向导相对于addnode脚本提供了许多增强功能。使用安装程序向导可将所有软件生命周期操作合并到一个工具中，从而简化管理。

  • Clone.pl脚本的弃用

    在Oracle Database 19c中不推荐使用clone.pl脚本。安装程序向导中提供了使用黄金映像执行仅软件安装的功能。

    可以在将来的发行版中删除clone.pl脚本。Oracle建议您使用安装程序向导将提取的黄金映像作为主目录安装，而不是使用clone.pl脚本。

不受支持的功能

查看Grid Infrastructure 19c不支持的功能。

有关不支持的功能，参数和视图的更多信息，请参阅“ Oracle数据库升级指南”

  • Flex集群体系结构中叶节点的支持

    Oracle Grid Infrastructure 19c中的Oracle Flex Cluster Architecture不再支持叶节点。

    在Oracle Grid Infrastructure 19c（19.1）和更高版本中，Oracle Flex Cluster中的所有节点都充当中心节点。Leaf节点在Oracle Flex Cluster架构的原始实现中提供的功能可以很容易地由中心节点提供。因此，不再支持叶节点。

  • 支持标准版2（SE2）数据库版的Oracle Real Application Clusters

    从Oracle Database 19c开始，Oracle数据库标准版2（SE2）不支持Oracle Real Application Clusters（Oracle RAC）。

    无法将从早期版本使用Oracle Real Application Clusters（Oracle RAC）功能的Oracle Database Standard Edition数据库升级到Oracle Database 19c。要将这些数据库升级到Oracle Database 19c，请在开始升级之前删除Oracle RAC功能，或者从Oracle Database Standard Edition升级到Oracle Database Enterprise Edition。有关每个步骤的更多信息，包括升级后如何重新配置系统，请参阅My Oracle Support说明2504078.1：“Oracle数据库标准版19c中对Oracle Real Application Clusters（RAC）的支持”。
