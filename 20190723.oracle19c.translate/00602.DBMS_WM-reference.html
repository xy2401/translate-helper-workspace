4 DBMS_WM包：参考

Workspace Manager在名为DBMS_WM的包中包含PL / SQL子程序（过程和函数），用于执行产品的可用功能。本章提供有关每个子程序的参考信息。

注意：

大多数Workspace Manager子程序都是过程，但有一些是函数。（函数返回一个值;过程不返回值。）

大多数函数的名称都以Get开头（例如GetConflictWorkspace和GetWorkspace ）。

子程序按字母顺序显示。有关子程序根据其逻辑分组的简要说明，请参阅DBMS_WM子程序类别 。

Workspace管理器子程序中可能出现的错误（例外）记录在Workspace Manager错误消息中 ，包括每个错误的原因和建议的用户操作。

语法说明：

  • Workspace Manager PL / SQL包的DBMS_WM公共同义词必须与子程序名一起使用。DBMS_WM公共同义词包含在格式和任何示例中。

  • 子程序调用不区分大小写，除了任何引用的文字值。例如，以下代码行摘录在有效和语义上相同：

    EXECUTE DBMS_WM.CreateWorkspace（'NEWWORKSPACE'）; EXECUTE dbms_wm.createworkspace（'NEWWORKSPACE'）; EXECUTE dBms_Wm.cReatEwoRksPace（'NEWWORKSPACE'）;

注意：

从另一个过程执行DBMS_WM过程时，权限检查会考虑该过程是否具有定义者权限或其权限当前处于活动状态的数据库用户的权限。

4.2 AddAsParentWorkspace

将工作空间作为父工作空间添加到多工作空间环境中的子工作空间。

句法

DBMS_WM.AddAsParentWorkspace（工作空间IN VARCHAR2，parent_workspace IN VARCHAR2，auto_commit IN BOOLEAN DEFAULT TRUE）;

参数

表4-2 AddAsParentWorkspace过程参数

参数 描述 
工作区

要添加父工作空间的工作空间的名称。该名称区分大小写。

 
parent_workspace

要添加为工作workspace的父工作空间的workspace 。该名称区分大小写。 

AUTO_COMMIT

布尔值（ TRUE或FALSE ）。

TRUE （默认值）导致操作作为将在完成时提交的自治数据库事务执行。

FALSE导致操作作为调用者的打开数据库事务（如果存在）的一部分执行。如果没有打开的数据库事务，则在新的数据库事务中执行该操作。在任何一种情况下，调用者都负责提交事务。有关更多信息，请参阅自动提交工作区管理器操作 。

使用说明

此过程是用于multiparent工作空间功能，这是在所描述的支持的一部分Multiparent工作区 。如果workspace只有一个父工作空间，则此过程使workspace成为多个工作空间。如果workspace已是多工作空间，则此过程会将另一个父工作workspace添加到workspace 。

如果以下一项或多项适用，则会引发异常：

  • Workspace Manager系统参数ALLOW_MULTI_PARENT_WORKSPACES值为OFF 。

  • Workspace Manager系统参数CR_WORKSPACE_MODE或NONR_WORKSPACE_MODE （取决于workspace是否为持续刷新的工作workspace中的任何一个）都是OPTIMISTIC_LOCKING 。

  • workspace或parent_workspace不存在。

  • parent_workspace已经位于workspace的祖先层次结构中。

  • auto_commit为TRUE并且在需要修改的任何表的父或子工作空间中存在打开的事务。

  • 在workspace启用版本的表中的数据视图中存在违反主键约束，参照完整性约束或唯一约束的情况。

例子

以下示例将Workspace4添加为Workspace4的父工作Workspace3 。（请参阅“ 多面工作空间”中的层次结构图示。） 

- 允许多个工作空间。（对于AddAsParentWorkspace是必需的）EXECUTE DBMS_WM.SetSystemParameter（'ALLOW_MULTI_PARENT_WORKSPACES'，'ON'）; - 通过将Workspace4添加为父级，使Workspace3成为多项式。EXECUTE DBMS_WM.AddAsParentWorkspace（'Workspace3'，'Workspace4'）;

表4-3 AddUserDefinedHint过程参数

参数 描述 
hint_id

唯一标识用户定义提示的数字ID。必须与Workspace Manager用于一个或多个SQL语句的现有提示ID匹配。

 
的table_id

要应用提示的表的名称。该名称不区分大小写。如果此值为null，则提示将与所有启用了版本的表一起用于指定提示的任何SQL语句。

仅当您对任何DBMS_WM程序包操作的性能不满意时，以及您知道如何使用应用程序跟踪和SQL优化程序提示时，才使用此过程。有关跟踪的信息，请参阅“ Oracle数据库SQL调优指南”中有关应用程序跟踪工具的章节。

/ * WM $ SQL (hint_id) (table_id) * /

要删除用户定义的提示（即，要使用与提示ID关联的默认提示），请使用RemoveUserDefinedHint过程。

4.5 AlterVersionedTable

更改启用版本的表以添加有效时间支持，重命名约束或重命名索引。

句法

DBMS_WM.AlterVersionedTable（table_name IN VARCHAR2，alter_option IN VARCHAR2，parameter_options IN VARCHAR2 DEFAULT NULL，ignore_last_error IN BOOLEAN DEFAULT FALSE）;

参数

表4-5 AlterVersionedTable过程参数

参数 描述 
TABLE_NAME

要添加有效时间支持的启用版本的表的名称。该名称不区分大小写。

 
alter_option

以下值之一： ADD_VALID_TIME用于添加有效时间支持， DDL用于进行DDL更改， RENAME_CONSTRAINT用于重命名约束， REBUILD_INDEX用于重建索引， RENAME_INDEX用于重命名索引，或者USE_SCALAR_TYPES_FOR_VALIDTIME或USE_WM_PERIOD_FOR_VALIDTIME用于指定是否对现有版本进行查看-enabled表应该使用两个标量列作为有效时间范围。

有关这些选项的信息，请参阅用法说明，包括何时必须且可以使用此过程重命名索引或约束。

 
parameter_options

带引号的字符串（通常格式为'keyword = value，keyword2 = value2，...'），包含对指定的alter_option参数值有效的关键字。有关对每个alter_option参数值有效的关键字，请参阅使用说明。 

ignore_last_error

布尔值（ TRUE或FALSE ）。

TRUE忽略在上一次调用AlterVersionedTable过程期间发生的最后一个错误（如果有）。有关上一个错误的信息存储在USER_WM_VT_ERRORS和ALL_WM_VT_ERRORS静态数据字典视图中，这些视图在Workspace Manager静态数据字典视图中进行了描述。有关更多信息，请参阅使用说明。

FALSE (默认值）不会忽略在上一次调用AlterVersionedTable过程期间发生的最后一个错误（如果有）。

使用说明

使用此过程添加有效时间支持，重命名约束或重命名现有版本启用表的索引。有关添加有效时间支持的更多信息，请参阅向现有表添加有效时间支持 。

如果alter_option值为ADD_VALID_TIME ，则可以指定以下parameter_options关键字中的无，一个或多个：

  • validFrom ：要在所有现有行的WM_VALID列中设置的开始时间段。默认值是当前时间戳。

  • validTill ：在所有现有行的WM_VALID列中设置的结束时间段。默认值为UNTIL_CHANGED 。

  • fmt ：日期格式。默认值为'mmddyyyyhh24miss' 。这些选项与TO_TIMESTAMP_TZ函数相同，后者在Oracle数据库SQL语言参考中进行了描述。

  • nlsparam ：全球化支持选项。选项和缺省值与用于日期转换的TO_CHAR函数的nlsparam参数相同，这在Oracle数据库SQL语言参考中有所描述。

如果alter_option值为DDL ，则此过程当前支持的操作是添加，合并和拆分表分区。您必须具有SYSDBA权限，并且必须指定以下parameter_options关键字：

  • ddl ：要执行的DDL（数据定义语言）语句。DDL语句必须引用完全限定的基表（例如，如果SCOTT.EMP是启用版本的表，则为SCOTT.EMP_LT）。

  • force ：值为true会导致Workspace Manager尝试执行DDL语句，无论此过程是否正式支持该操作;值false （默认值）会导致Workspace Manager不尝试执行DDL语句。因此，要执行DDL语句，必须通过显式指定'force=true'来覆盖默认值;但是，除非你知道自己在做什么，否则不要指定'force=true' 。

如果alter_option值为RENAME_CONSTRAINT ，则必须指定以下两个parameter_options关键字：

  • constraint_name ：要重命名的约束的当前名称。该名称不区分大小写。

  • new_constraint_name ：约束的新名称。该名称不区分大小写。

如果alter_option值为RENAME_INDEX ，则必须指定以下所有parameter_options关键字：

  • index_owner ：拥有要重命名的索引的架构的名称。架构名称不区分大小写。

  • index_name ：要重命名的索引的当前名称。该名称不区分大小写。

  • new_index_name ：索引的新名称。该名称不区分大小写。

如果启用版本的表上的约束或索引的名称超过26个字符，则必须使用AlterVersionedTable过程（如果要重命名约束或索引）;您不能使用带有RENAME子句的ALTER TABLE（对于约束）或ALTER INDEX（对于索引）语句。如果使用AlterVersionedTable过程，则不需要在对BeginDDL和CommitDDL过程的调用之间包含它。

如果启用版本的表上的约束或索引的名称长度不超过26个字符，则可以执行以下任一操作以重命名约束或索引：使用AlterVersionedTable过程，或使用ALTER TABLE（用于约束）或者在对BeginDDL和CommitDDL过程的调用之间使用RENAME子句的ALTER INDEX（对于索引）语句（如与启用版本的表相关的DDL操作中所述 ）。

如果alter_option值为REBUILD_INDEX ，则必须指定index_owner和index_name关键字以标识拥有索引的数据库用户和索引的名称;并且您可以指定reverse或noreverse关键字，以指定是否以相反的顺序存储索引块的字节，不包括rowid。

该alter_option值USE_SCALAR_TYPES_FOR_VALIDTIME和USE_WM_PERIOD_FOR_VALIDTIME只能用来改变现有的支持版本表的看法是与工作区管理系统参数的当前设置一致USE_SCALAR_TYPES_FOR_VALIDTIME （描述系统参数工作区管理器 ）。例如，如果将Workspace Manager系统参数USE_SCALAR_TYPES_FOR_VALIDTIME为ON ，但是名为MYTABLE的现有版本启用表具有使用名为WM_VALID（类型为WM_PERIOD ）的单个列来指示有效时间范围的视图，则可以更改视图在MY_TABLE上，通过调用AlterVersionedTable过程并指定alter_option值USE_SCALAR_TYPES_FOR_VALIDTIME来使用TIMESTAMP WITH TIME ZONE类型的两列。

alter_option参数不能用于覆盖Workspace Manager系统参数USE_SCALAR_TYPES_FOR_VALIDTIME的当前值。如果系统参数值为ON ，则alter_option参数值必须为USE_SCALAR_TYPES_FOR_VALIDTIME ;如果系统参数值为OFF ，则alter_option参数值必须为USE_WM_PERIOD_FOR_VALIDTIME 。

您可以在parameter_options字符串中为参数值使用双引号。例如，以下两个规范在语义上是相同的：

'index_owner = scott，index_name = my_index，new_index_name = my_new_index''index_owner =“scott”，index_name =“my_index”，new_index_name =“my_new_index”'

如果对AlterVersionedTable过程的调用失败，您应该尝试修复错误原因。检查USER_WM_VT_ERRORS和ALL_WM_VT_ERRORS静态数据字典视图以查看SQL语句和错误消息。修复错误原因，然后使用默认的ignore_last_error参数值FALSE再次调用AlterVersionedTable过程。但是，如果调用仍然失败并且您无法修复错误原因，并且您确定忽略此错误是安全且适当的，则可以通过使用ignore_last_error参数值调用AlterVersionedTable过程来忽略该错误。 TRUE 。请注意，您有责任确保忽略错误是安全和适当的。

如果以下一项或多项适用，则会引发异常：

  • table_name不存在。

  • alterOptions不是ADD_VALID_TIME 。

句法

DBMS_WM.BeginBulkLoading（table_name IN VARCHAR2，工作空间IN VARCHAR2，版本IN INTEGER DEFAULT NULL，check_for_duplicates IN BOOLEAN DEFAULT TRUE，ignore_last_error IN BOOLEAN DEFAULT FALSE，single_transaction IN BOOLEAN DEFAULT FALSE，savepoint_name IN DEFAULT LATEST）;

参数

表4-7 BeginBulkLoading过程参数

参数 描述 
TABLE_NAME

将批量加载数据的启用版本的表的名称。该名称不区分大小写。

 
工作区

将执行批量加载的工作空间的名称。该名称区分大小写。

 
版

（忽略当前版本。在版本12.1之前，这是GetBulkLoadVersion函数返回的版本号。） 

check_for_duplicates

布尔值（ TRUE或FALSE ）。

TRUE （默认值）检查要批量加载的数据中的行，这些行在主键列中具有相同的值。对于任何重复记录，只有具有最低ROWID值的记录保留在表中，其余记录将移动到对CommitBulkLoading过程的调用中指定的discards表。有关此参数的详细信息，请参阅使用说明。

ignore_last_error

布尔值（ TRUE或FALSE ）。

TRUE忽略在上一次调用BeginBulkLoading过程期间发生的最后一个错误（如果有）。有关上一个错误的信息存储在USER_WM_VT_ERRORS和ALL_WM_VT_ERRORS静态数据字典视图中，这些视图在Workspace Manager静态数据字典视图中进行了描述。有关更多信息，请参阅使用说明。

FALSE (默认值）不会忽略在上一次调用BeginBulkLoading过程期间发生的最后一个错误（如果有）。 

single_transaction

布尔值（ TRUE或FALSE ）。

TRUE导致Workspace Manager在调用CommitBulkLoading过程后将执行的每个步骤之后不执行内部提交操作，而是仅在执行了所有必要步骤后才执行提交。TRUE还允许在启用版本的表上进行查询。

FALSE （默认值）导致Workspace Manager在调用CommitBulkLoading过程后将执行的每个步骤之后执行内部提交操作，并且它还禁止在表上进行查询，直到完成CommitBulkLoading或RollbackBulkLoading操作。

有关此参数的详细信息，请参阅使用说明。

 
savepoint_name

工作空间中将批量加载数据的版本。如果指定，则必须是以下之一： LATEST或ROOT_VERSION 。

LATEST （默认值）是工作区中的当前版本。

ROOT_VERSION是根版本（版本号0，位于LIVE工作区中）。根版本是所有其他版本的祖先，因此根版本中的数据在所有其他工作空间中都是可见的（除非非LIVE工作空间已更新数据）。仅当workspace为LIVE才能指定ROOT_VERSION 。

使用说明

在开始将数据批量加载到启用版本的表之前，必须调用BeginBulkLoading过程。您必须通过调用CommitBulkLoading过程（以提交加载数据时所做的更改）或RollbackBulkLoading过程（以回滚加载数据时所做的更改）来结束批量加载会话。有关使用Workspace Manager批量加载的详细信息，请参阅批量加载到启用版本的表中 。

如果single_transaction为FALSE （默认值），则BeginBulkLoading过程会删除表上的一些内部Workspace Manager视图，以防止表上的DML操作和某些Workspace Manager操作;但是，这也可以防止使用指定的启用版本的表进行任何查询。无论single_transaction参数值如何，特别是如果它为FALSE ，您应该尽快完成批量加载，并且应用程序和用户不需要访问该表。对于具有指定表的批量加载会话，对于BeginBulkLoading和CommitBulkLoading过程， single_transaction参数的值必须相同。

check_for_duplicates参数的TRUE值不会导致启用版本的表中的任何现有数据被检查。如果批量加载数据的版本中的现有行（可能是工作空间的最新版本或根版本）具有与要批量加载的数据中的行相同的主键值，则行为取决于表的历史选项设置：如果设置了VIEW_WO_OVERWRITE ，则新加载的行将链接到具有相同主键值的现有行;如果未设置VIEW_WO_OVERWRITE ，则新数据不会批量加载，而是移动到丢弃表。

如果对BeginBulkLoading过程的调用失败，您应该尝试修复错误原因。检查USER_WM_VT_ERRORS和ALL_WM_VT_ERRORS静态数据字典视图以查看SQL语句和错误消息。修复错误原因，然后使用默认的ignore_last_error参数值FALSE再次调用BeginBulkLoading过程。然而，如果呼叫仍然失败，你无法修复错误的原因，如果你确信它是安全的，适当的，忽略这个错误，那么你必须通过调用BeginBulkLoading程序与忽略错误的选项ignore_last_error参数值为TRUE 。请注意，您有责任确保忽略错误是安全和适当的。

如果性能是一个问题，请仔细考虑是否需要检查重复记录，因为check_for_duplicates值为TRUE （默认值）会导致Workspace Manager执行其他内部处理。

如果以下一项或多项适用，则会引发异常：

  • table_name不存在。

  • table_name未启用版本。

  • 用户不拥有该表或没有WM_ADMIN系统特权。

例子

以下示例将批量装入操作启动到W1工作空间中的EMP表中。 

EXECUTE DBMS_WM.BeginBulkLoading（'EMP'，'W1'）;

4.8 BeginDDL

为指定的表启动DDL（数据定义语言）会话。

句法

DBMS_WM.BeginDDL（table_name IN VARCHAR2）;

参数

表4-8 BeginDDL过程参数

参数 描述 
TABLE_NAME

已启用版本的表的名称。该名称不区分大小写。

使用说明

此过程启动DDL会话，并创建一个特殊的表，其名称与table_name相同，但将_LTS添加到表名中。调用此过程后，您可以对表或基于表的任何索引或触发器执行一个或多个DDL操作，然后调用CommitDDL或RollbackDDL过程。

  • 在<表名> _LTS表具有相同的触发器，列和索引作为<表名>表。

  • 对于<table-name>表与其具有参照完整性约束的每个父表，将为<table-name> _LTS表定义相同的约束。

  • 触发器列，并在<表名> _LTS表参照完整性约束具有相同的名称为对<表名>表中的对应的一个。

  • 对于<table-name>表上的每个索引， <table-name> _LTS表上的相应索引的名称格式为<index-name> _LTS 。

  • <table-name> _LTS表上的主键约束的名称格式为<primary-key> _LTS 。

  • <table-name> _LTS表上的所有唯一约束都具有<unique-constraint-name> _LTS形式的名称 。

有关执行与启用版本的表相关的DDL操作的详细信息，请参阅与启用版本的表相关的DDL操作 。

如果以下一项或多项适用，则会引发异常：

  • table_name不存在或未启用版本。

  • table_name上定义了域索引，并且未直接授予用户CREATE TABLE和CREATE SEQUENCE权限。

  • table_name存在一个打开的DDL会话。（也就是说，已经调用了BeginDDL过程来指定此表，并且尚未调用指定此表的CommitDDL或RollbackDDL过程。）

例子

以下示例开始DDL会话，使用名为COLA_MARKETING_BUDGET_LTS的特殊表将名为COMMENTS的列添加到COLA_MARKETING_BUDGET表，并通过提交更改来结束DDL会话。 

EXECUTE DBMS_WM.BeginDDL（'COLA_MARKETING_BUDGET'）; ALTER TABLE cola_marketing_budget_lts ADD（comments VARCHAR2（100））; EXECUTE DBMS_WM.CommitDDL（'COLA_MARKETING_BUDGET'）;

历史记录选项使您可以记录和审核修改。

有关Oracle Spatial和Graph拓扑中的表的Workspace Manager支持的信息，请参阅空间和图形拓扑支持 。

参数 描述

使用说明

您将在指定的日期和时间或附近显示当前工作空间的只读视图。确切的时间点取决于跟踪版本启用表中数据更改的历史记录选项，由EnableVersioning过程设置或由SetWoOverwriteOFF或SetWoOverwriteON过程修改：

  • NONE ：只读视图反映了in_date之后的第一个保存点。

  • VIEW_W_OVERWRITE ：只读视图反映了in_date中生效的数据值，除非in_date位于两个保存点之间且两个保存点之间的数据已更改。在这种情况下，保存点之间已更改的数据可能被视为空或具有先前值。要确保最完整和准确的数据视图，请在启用表的版本时指定VIEW_WO_OVERWRITE历史记录选项。

  • VIEW_WO_OVERWRITE ：只读视图反映in_date中生效的数据值。

有关历史记录选项的说明，请参阅EnableVersioning过程的hist参数说明。

以下示例方案显示了VIEW_WO_OVERWRITE设置的效果。假设以下事件序列：

  1. 连接的MANAGER_NAME值是Adams 。

  2. 保存点SP1已创建。

  3. MANAGER_NAME值更改为Baxter 。

  4. 将指定为in_date （在步骤7中）的时间点发生。

  5. MANAGER_NAME值已更改为Chang 。（因此，自第一个保存点和第二个保存点之前， in_date之前和之后的值都已更改。）

  6. Savepoint SP2已创建。

  7. 执行GotoDate操作，将步骤4中的时间点指定为in_date 。

在上面的场景中：

  • 如果生效的历史记录选项为VIEW_WO_OVERWRITE ，则步骤7后的MANAGER_NAME值为Baxter 。在第5步之后，版本化表有三行，每行都有不同的MANAGER_NAME值（ Adams ， Baxter ， Chang ），因为每次更改都是在行的新副本中进行的。

  • 如果有效的历史记录选项是VIEW_W_OVERWRITE ，则在步骤7之后看不到任何值。步骤3和5中的更新在行的相同副本中进行，并且步骤5中的更新将覆盖步骤3中的更新。因此，在第5步之后，版本化表格有两行，其中MANAGER_NAME值为Adams和Chang 。由于在指定时刻生效的MANAGER_NAME值（ Baxter ）已被覆盖，因此不会显示任何行。

  • 如果生效的历史记录选项为NONE ，则步骤7之后的MANAGER_NAME值为Chang ，因为指定时刻之后的第一个保存点是SP2 。在第5步之后，版本化表格有两行，其中MANAGER_NAME值为Adams和Chang 。

当用户存在于工作区中时，应执行GotoDate过程。没有与此过程关联的显式权限。

resolve_bnase_ne

FALSE （默认值）不允许解决冲突，而不支持不存在的基行。如果在解析其他冲突时遇到不存在的基行，则会有效地忽略冲突，并且不会将其视为已解决。

使用说明

此过程检查table_name和where_clause参数标识的条件，并查找workspace行值与其父工作空间之间的任何冲突。此过程通过使用父参数或父工作空间中的行值来解决冲突，如keep参数中所指定;但是，在提交事务（标准数据库提交操作）并调用CommitResolve过程以结束冲突解决会话之前，冲突解决方案实际上并未合并。（有关冲突解决的更多信息，包括流程的整体视图，请参阅在合并或刷新操作之前解决冲突 。）

例如，假设对于Department 20（ DEPARTMENT_ID = 20 ）， LIVE和Workspace1工作空间中的MANAGER_NAME是Tom 。然后，发生以下操作：

  1. 部门20的manager_name在LIVE数据库工作区中从Tom更改为Mary 。

  2. 提交更改（标准数据库提交操作）。

  3. 部门20的manager_name在Workspace1从Tom更改为Franco 。

  4. 调用MergeWorkspace过程以将Workspace1更改合并到LIVE工作区。

    但是，此时， Workspace1 Department 20的MANAGER_NAME存在冲突（ Franco ，它与LIVE工作区中的Mary冲突），因此对MergeWorkspace的调用不成功。

  5. 使用以下参数调用ResolveConflicts过程：（ 'Workspace1' ， 'department' ， 'department_id = 20' ， 'child' ）。

    在步骤7中的MergeWorkspace操作之后， MANAGER_NAME值将是Workspace1和LIVE工作区中的Franco 。

  6. 提交更改（标准数据库提交操作）。

  7. 调用MergeWorkspace过程以将Workspace1更改合并到LIVE工作区。

在冲突解决会话期间，以下注意事项适用：

  • ResolveConflicts操作会阻止目标工作空间或表上的其他工作空间操作（如合并，刷新或删除），直到执行CommitResolve或RollbackResolve过程为止。

  • 多个会话可以执行ResolveConflicts操作，并在同一个表上执行插入，更新和删除操作。但是，在此类操作期间，目标行将被锁定。如果多个会话尝试在同一行上执行插入，更新或删除操作或解决影响同一行的冲突，则允许第一个会话继续;并且在该会话执行CommitResolve或RollbackResolve过程之后，允许另一个会话继续。

有关冲突解决的详细信息，请参阅在合并或刷新操作之前解决冲突 。

例子

以下示例解决了涉及DEPARTMENT_ID为20的Workspace1中DEPARTMENT表中的行的冲突，并使用子工作空间中的值来解决所有此类冲突。然后通过首先提交事务（标准提交）然后调用MergeWorkspace过程来合并冲突解决的结果。 

EXECUTE DBMS_WM.BeginResolve（'Workspace1'）; EXECUTE DBMS_WM.ResolveConflicts（'Workspace1'，'department'，'department_id = 20'，'child'）;承诺; EXECUTE DBMS_WM.CommitResolve（'Workspace1'）;

  • 会话执行SetLockingOFF过程。

4.99 SetWorkspaceLockModeON

为指定的工作区启用Workspace Manager锁定。

句法

DBMS_WM.SetWorkspaceLockModeON（工作空间IN VARCHAR2，lockmode IN VARCHAR2，覆盖IN BOOLEAN DEFAULT FALSE，auto_commit IN BOOLEAN DEFAULT TRUE）;

参数

表4-83 SetWorkspaceLockModeON过程参数

参数 描述 
工作区

要为其启用Workspace Manager锁定的工作空间的名称。该名称区分大小写。

 
lockmode

行级锁定的默认锁定模式。必须是E ， WE ， VE ， S ， C或D

E （独占）模式锁定父工作空间中的行和当前工作空间中的相应行;任一工作空间中的其他用户都无法更改任何值。

WE （工作空间独占）模式锁定先前版本中的行和当前版本中的相应行，以便只有设置锁定的用户才能更改当前工作空间中的值。但是，其他工作区中的其他用户可以更改值。

VE （版本独占）模式锁定先前版本中的行和当前版本中的相应行，以便只有设置锁定的用户才能更改值;没有其他用户（在任何工作区中）可以更改值。

S （共享）模式锁定父工作空间中的行和当前工作空间中的相应行;但是，当前工作空间中的其他用户（但父工作空间中没有用户）可以更改这些行中的值。

C （前转）模式使用与父工作空间中相应行相同的锁定模式锁定当前工作空间中的行。（如果未在父工作空间中锁定行，则不会锁定子工作空间中的相应行。）

D （忽视）模式忽略所有锁定。任何行都可以在工作空间中更新，即使它被另一个工作空间锁定或工作空间处于悲观设置中。使用此模式的后果是：（1）设置此模式的工作空间不能在修改行之后关闭（使用SetLockingOFF ） 锁定模式或更改（使用SetLockingON ）。（2）工作区无法合并。（3）工作区无法刷新。（4） override选项必须设置为false 。（5）当从此工作空间创建任何子工作空间时，新工作空间将继承此相同的锁模及其限制。 

覆盖

布尔值（ TRUE或FALSE ）

TRUE允许工作空间中的会话通过使用SetLockingON和SetLockingOFF过程更改lockmode值。

FALSE （默认值）可防止工作空间中的会话更改lockmode值。 

AUTO_COMMIT

布尔值（ TRUE或FALSE ）。

TRUE （默认值）导致操作作为将在完成时提交的自治数据库事务执行。

FALSE导致操作作为调用者的打开数据库事务（如果存在）的一部分执行。如果没有打开的数据库事务，则在新的数据库事务中执行该操作。在任何一种情况下，调用者都负责提交事务。有关更多信息，请参阅自动提交工作区管理器操作 。

使用说明

此过程会影响Workspace Manager锁定，除了任何标准Oracle数据库锁定之外，还会发生此锁定。Workspace Manager锁可用于防止冲突。当用户锁定行时，父工作空间中的相应行也会被锁定。因此，当此工作空间在合并时与父项合并时，可确保此行不会发生冲突。

有关Workspace Manager锁管理的信息，请参阅使用Workspace Manager进行锁管理 。

“ lockmode ”锁定模式（ D lockmode值）的主要用途是使工作空间可以与系统中的其余工作空间完全隔离，并可以自由更新它想要的任何行。它将工作空间转换为测试（“沙箱”）工作空间，可以测试任何内容，但由于无法合并或刷新，工作空间无法将其更改传播到其他工作空间。它仅用于测试，之后可以删除工作空间。

独占锁定（ E锁定lockmode值）可防止使用假设情景，其中测试一列或多列的不同值。因此，在独占锁定无效时，计划任何场景测试。

如果override参数值为TRUE ，则还可以分别使用SetLockingON和SetLockingOFF过程在用户会话级别启用和禁用锁定。

除非会话通过执行SetLockingOFF过程关闭锁定，否则此会话或后续会话的所有新更改都将被锁定。

如果发生以下任何情况，则会引发异常：

  • 用户没有WM_ADMIN系统特权或不是workspace的所有者。

  • auto_commit为TRUE且存在打开的事务。

  • lockmode是D ，工作区要么不断刷新，要么是LIVE工作区。

DBMS_WM.UnlockRows（工作空间IN VARCHAR2，table_name IN VARCHAR2，where_clause IN VARCHAR2 DEFAULT''，all_or_user IN VARCHAR2 DEFAULT'USER'，lock_mode IN VARCHAR2 DEFAULT'ES'，Xmin IN NUMBER DEFAULT NULL，Ymin IN NUMBER DEFAULT NULL，Xmax IN NUMBER DEFAULT NULL，Ymax IN NUMBER DEFAULT NULL）;

参数

表4-85 UnlockRows过程参数

参数 描述 
工作区

工作空间的名称：此工作空间中的锁定行和父工作空间中的相应行将被解锁，如其余参数中所指定。该名称区分大小写。

如果lock_mode设置为VE （版本独占），则可以使用值NONE 。这会导致任何工作空间锁定的行被解锁。 

TABLE_NAME

表的名称或（如果指定了Xmin ， Ymin ， Xmax和Ymax ）要解锁行的空间和图形拓扑。该名称不区分大小写。 

where_clause

标识要解锁的行的WHERE子句（不包括WHERE关键字）。示例： 'department_id = 20'

除子查询外，只能在WHERE子句中指定主键列。子查询可以引用不是主键的列，但不能引用启用版本的表。

如果未指定where_clause参数，则可以访问table_name中的所有行。

如果table_name指定空间和图形拓扑名称，请不要指定where_clause参数。 

all_or_user

请求范围： ALL或USER 。

ALL ：考虑用户在指定工作空间中可访问的所有锁定。

USER （默认值）：仅考虑用户在指定工作空间中拥有的锁。 

LOCK_MODE

锁定模式： E ， WE ， VE ， S或ES （默认）。

E （独占）：仅考虑独占模式锁。

WE （工作空间独占）：仅考虑工作空间独占模式锁。

VE （版本独占）：仅考虑版本独占模式锁。

S （共享）：仅考虑共享模式锁。

ES （独占和共享：默认值）：考虑独占模式和共享模式锁。 

Xmin，Ymin

仅适用于Oracle Spatial and Graph拓扑（请参阅使用拓扑锁定注意事项 ），分别包含要锁定的行的窗口左下角的X和Y坐标值。如果为table_name指定了拓扑名称，则必须指定这些参数;否则，请勿指定这些参数。 

Xmax，Ymax

仅适用于Oracle Spatial and Graph拓扑（请参阅使用拓扑锁定注意事项 ），分别包含要锁定的行的窗口右上角的X和Y坐标值。如果为table_name指定了拓扑名称，则必须指定这些参数;否则，请勿指定这些参数。

使用说明

此过程会影响Workspace Manager锁定，除了任何标准Oracle数据库锁定之外，还会发生此锁定。有关Workspace Manager锁定的说明，请参阅使用Workspace Manager锁定管理 。

此过程将解锁先前锁定的行（请参阅LockRows过程）。它不会影响是否设置Workspace Manager锁定（由SetLockingON和SetLockingOFF过程确定）。

有关Oracle Spatial和Graph拓扑中的表的Workspace Manager锁定的信息，请参阅使用拓扑锁定注意事项 。

例子

以下示例解锁NEWWORKSPACE工作空间中last_name = 'Smith'的EMPLOYEES表。 

EXECUTE DBMS_WM.UnlockRows（'employees'，'NEWWORKSPACE'，'last_name =''Smith'''）;

4.102 UseDefaultValuesForNulls

当用户指定空值或未在启用版本的表的插入操作中为列指定任何值时，确定当前会话的Workspace Manager是否使用列的默认值。

句法

DBMS_WM.UseDefaultValuesForNulls（mode_var IN VARCHAR2）;

参数

表4-86 UseDefaultValuesForNulls过程参数

参数 描述 
mode_var

处理空值插入的模式： OFF或ON 。

OFF ：将空值插入列中。（这是正常的Oracle行为。）

ON ：列的默认值将插入列中。

使用说明

只有当已将列中的INSERT语句显式指定为列时，如果已将列定义为具有默认值或未指定列，则此过程会影响Workspace Manager执行的操作。例如，假设以下表定义：

CREATE TABLE播放器（名称为VARCHAR2（20）主键，评级为NUMBER DEFAULT 10）;

如果PLAYERS表是启用了版本的，并且如果您已使用mode_var参数值OFF执行此过程，则以下任一语句都会为Smith插入一行，其中RATING值为空：

INSERT INTO玩家VALUES（'Smith'，NULL）; INSERT INTO球员（姓名）VALUES（'史密斯'）;

但是，如果已执行UseDefaultValuesForNulls过程且mode_var参数值为ON ，则两个语句都会为Smith插入一行，其RATING值为10。如果此过程未在会话中执行，则默认行为与将mode_var指定为ON 。

例子

以下示例导致在当前会话的其余部分期间使用列缺省值，只要启用版本的表中的INSERT语句为具有默认值的列指定空值或者未指定列。

EXECUTE DBMS_WM.UseDefaultValuesForNulls（'ON'）;
