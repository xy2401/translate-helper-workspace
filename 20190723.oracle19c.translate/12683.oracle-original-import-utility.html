25.1.1表对象：导入顺序

从导出转储文件中读取表对象时将导入它们。

转储文件按以下顺序包含对象：

  1. 类型定义

  2. 表定义

  3. 表数据

  4. 表索引

  5. 完整性约束，视图，过程和触发器

  6. 位图，基于功能和域索引

导入顺序如下：创建新表，导入数据并构建索引，导入触发器，在新表上启用完整性约束，并构建任何位图，基于函数和/或域索引。此序列可防止数据因导入表的顺序而被拒绝。此序列还可以防止冗余触发器在同一数据上触发两次（一次最初插入时，再次导入时）。

25.3.1导入数据前手动创建表

您可以在导入数据之前手动创建表。

如果在从导出文件导入数据之前选择手动创建表，则应使用先前使用的相同表定义或兼容格式。例如，虽然您可以增加列的宽度并更改其顺序，但您无法执行以下操作：

    在导入数据之前手动创建TABLE时，导出转储文件中的CREATE TABLE语句将失败，因为该表已存在。要避免此故障并继续将数据加载到表中，请设置Import参数IGNORE = y 。否则，由于表创建错误，不会将任何数据加载到表中。

25.3.2禁用参照约束

描述如何禁用引用约束。

在正常导入顺序中，仅在导入所有表后才导入参照约束。此序列可防止在尚未导入的数据存在参照完整性约束时可能发生的错误。

将数据加载到现有表时，仍可能发生这些错误。例如，如果表emp在mgr列上具有参照完整性约束，用于验证emp,中是否存在管理器编号emp,那么如果尚未导入管理器行emp,则合法员工行可能会使参照完整性约束失败。

发生此类错误时，Import会生成错误消息，绕过失败的行，并继续导入表中的其他行。您可以手动禁用约束以避免这种情况。

表之间的引用约束也可能导致问题。例如，如果emp表出现在导出转储文件中的dept表之前，但是从emp表到dept表存在引用检查，那么由于引用约束，可能无法导入emp表中的某些行违反。

要防止出现这些错误，应在将数据导入现有表时禁用参照完整性约束。

25.3.3手动排序导入

描述手动排序导入。

在导入后重新启用约束时，将检查整个表，这可能需要很长时间才能生成大表。如果该检查所需的时间太长，那么手动订购导入可能是有益的。

为此，请从导出文件而不是导出文件执行多个导入。首先，导入作为参考检查目标的表。然后，导入引用它们的表。如果表不以循环方式相互引用，并且表不引用自身，则此选项有效。

25.4模式和数据库触发器对导入操作的影响

定义为触发特定模式的DDL事件或数据库的DDL相关事件的触发器是系统触发器。

这些触发器可能对某些进口操作产生不利影响。例如，它们可以阻止成功重新创建数据库对象，例如表。这会导致返回错误，而不会指示触发器导致问题。

数据库管理员和任何创建系统触发器的人都应该验证此类触发器不会阻止用户执行授权的数据库操作。要测试系统触发器，请执行以下步骤：

  1. 定义触发器。

  2. 创建一些数据库对象。

  3. 以表或用户模式导出对象。

  4. 删除对象。

  5. 导入对象。

  6. 验证是否已成功重新创建对象。

    注意：

    完全导出不会导出模式SYS拥有的触发器。您必须在完全导入之前或之后手动重新创建SYS触发器。Oracle建议您在导入后重新创建它们，以防它们定义会阻碍导入进度的操作。

25.5.2参数文件

您可以在参数文件中指定所有有效参数及其值。

将参数存储在文件中可以轻松修改或重用它们。如果对不同的数据库使用不同的参数，则可以有多个参数文件。

使用任何平面文件文本编辑器创建参数文件。命令行选项PARFILE= filename告诉Import从指定文件而不是从命令行读取参数。例如：

参数文件规范的语法可以是以下任何一种：

PARAMETER = value PARAMETER =（ value ）PARAMETER =（ value1，value2，... ）

在这种情况下，因为INDEXES=n出现在 PARFILE=params.dat ， INDEXES=n会覆盖参数文件中INDEXES参数的值。

25.6导入模式

导入实用程序支持四种操作模式。

特别：

  • 完整：导入完整的数据库。只有具有IMP_FULL_DATABASE角色的用户才能使用此模式。使用FULL参数指定此模式。

  • 表空间：允许特权用户将一组表空间从一个Oracle数据库移动到另一个Oracle数据库。使用TRANSPORT_TABLESPACE参数指定此模式。

  • 用户：允许您导入属于您的所有对象（例如表，授权，索引和过程）。以用户模式导入的特权用户可以导入指定用户集的模式中的所有对象。使用FROMUSER参数指定此模式。

  • 表：允许您导入特定的表和分区。特权用户可以通过指定包含它们的架构来限定表。使用TABLES参数指定此模式。

    注意：

    当您使用表模式导入具有ANYDATA类型列的表时，您可能会收到以下错误：

buffer_size = rows_in_array * maximum_row_size

对于包含LOB， LONG, BFILE, REF, ROWID, UROWID或TIMESTAMP列的表，将单独插入行。缓冲区的大小必须足够大，以包含整行，LOB和LONG列除外。如果缓冲区无法保存表中最长的行，则Import会尝试分配更大的缓冲区。

25.7.2 COMMIT

默认值： n

指定在每个数组插入后是否应提交Import。默认情况下，Import仅在加载每个表后提交，Import在发生错误时执行回滚，然后继续下一个对象。

如果COMMIT=n并且表已分区，则导出文件中的每个分区和子分区都将在单独的事务中导入。

对于包含LOB， LONG, BFILE, REF, ROWID, UROWID或TIMESTAMP列的表，不会执行数组插入。如果COMMIT=y ，则Import会在每行之后提交这些表。

25.7.12.1完整数据库导出和导入要考虑的要点

完整数据库导出和导入是复制或清理数据库的好方法。

但是，为避免出现问题，请务必牢记以下几点：

  • 完全导出不会导出模式SYS拥有的触发器。您必须在完全导入之前或之后手动重新创建SYS触发器。Oracle建议您在导入后重新创建它们，以防它们定义会阻碍导入进度的操作。

  • 完整导出也不会导出默认配置文件。如果已修改源数据库中的默认配置文件（例如，通过添加模式SYS拥有的密码验证功能），则必须手动预创建该功能，并在导入完成后修改目标数据库中的默认配置文件。

  • 如果可能，在开始之前，请创建导出数据库的物理副本以及要导入的数据库。这可以确保任何错误都是可逆的。

  • 在开始导出之前，建议生成包含以下信息的报告：

    • 表空间和数据文件的列表

    • 回滚段列表

    • 按用户计算每个对象类型（如表，索引等）

    通过此信息，您可以确保已创建表空间并且导入成功。

  • 如果您是从出口创造一个完全新的数据库，然后记得要创建一个额外的回滚段SYSTEM ，并使它在你的初始化参数文件（可init 。 ora)继续进行导入操作前。

  • 执行导入时，请确保指向正确的实例。这非常重要，因为在某些UNIX系统上，只需输入子shell即可更改执行导入操作的数据库。

  • 除非您确定已创建所有表空间，否则请勿对具有多个数据库的系统执行完全导入。完全导入使用与导出的数据库相同的数据文件名创建任何未定义的表空间。这可能会导致以下情况出现问题：

    • 如果数据文件属于任何其他数据库，则它们将被破坏。如果导出的数据库位于同一系统上，则尤其如此，因为其数据文件将由要导入的数据库重用。

    • 如果数据文件的名称与现有操作系统文件冲突。

指定是否导入对象授予。

默认情况下，导入实用程序会导入导出的所有对象授予。如果导出是用户模式导出，则导出文件仅包含第一级对象授予（由所有者授予的授权）。

25.7.15 IGNORE

IGNORE Import实用程序参数指定应如何处理对象创建错误。

默认值： n

指定应如何处理对象创建错误。如果接受默认值IGNORE=n ，则在继续之前导入日志或显示对象创建错误。

如果指定IGNORE=y ，则导入在尝试创建数据库对象时忽略对象创建错误，并继续而不报告错误。

请注意，仅忽略对象创建错误 ;其他错误（如操作系统，数据库和SQL错误） 不会被忽略，并可能导致处理停止。

在使用IGNORE=y完成单个导出文件的多次刷新的情况下，可以多次创建某些对象（尽管它们将具有唯一的系统定义名称）。您可以通过使用CONSTRAINTS=n导入来阻止某些对象（例如，约束）。如果使用CONSTRAINTS=n进行完全导入，则不会导入任何表的约束。

如果表已存在并且IGNORE=y ，则将行导入现有表中，而不会给出任何错误或消息。您可能希望将数据导入已存在的表中以便使用新的存储参数，或者因为您已在集群中创建了表。

如果表已存在且IGNORE=n,则报告错误并跳过表，不插入任何行。此外，将不会创建依赖于表的对象，例如索引，授权和约束。

注意：

导入现有表时，如果表中的列没有唯一索引，则可以复制行。

25.7.26 SKIP_UNUSABLE_INDEXES

您可以将INDEXFILE参数与INDEXES=n结合使用，以提供用于重新创建索引的SQL脚本。如果未指定SKIP_UNUSABLE_INDEXES参数，则尝试更新不可用索引的行插入将失败。

25.7.30表

默认值：无

指定导入是表模式导入，并列出要导入的表名，分区和子分区名称。表模式导入允许您导入整个分区或非分区表。TABLES参数限制导入到指定表及其关联对象，如导入模式中所列。您可以为TABLES参数指定以下值：

  • tablename指定要导入的一个或多个表的名称。如果列表中的表已分区，并且您未指定分区名称，则会导入其所有分区和子分区。要导入所有导出的表，请指定星号（*）作为唯一的表名参数。

    tablename可以包含任意数量的'％'模式匹配字符，每个匹配字符可以匹配导出文件中表名中的零个或多个字符。将选择名称与列表中特定表名称的所有指定模式匹配的所有表以进行导入。列表中包含所有模式匹配字符且没有分区名称的表名导致导入所有导出的表。

  • partition_name和subpartition_name允许您将导入限制为分区表中的一个或多个指定分区或子分区。

用于指定前述内容的语法格式如下：

tablename：partition_name tablename：subpartition_name

如果使用tablename : partition_name ，则必须对指定的表进行分区，并且partition_name必须是其分区或子分区之一的名称。如果未对指定的表进行分区，则忽略partition_name并导入整个表。

可以同时指定的表的数量取决于命令行限制。

处理导出文件时，导出文件中的每个表名称将与列表中的每个表名称进行比较，按参数中指定表名称的顺序进行比较。为避免歧义和过多的处理时间，特定的表名应出现在列表的开头，更常用的表名（带有模式的表名）应出现在列表的末尾。

imp TABLES =（jones.accts，scott.emp，scott.dept）

导入这些表的有效规范如下：

imp FROMUSER = jones TABLES =（accts）imp FROMUSER = scott TABLES =（emp，dept）

25.7.30.1表名限制

这是对导入实用程序的表名限制的说明。

以下限制适用于表名：

  • 默认情况下，数据库中的表名称存储为大写。如果您具有大小写或小写的表名，并且希望保留表名的区分大小写，则必须将名称括在引号中。名称必须与存储在数据库中的表名完全匹配。

    某些操作系统要求命令行上的引号前面有转义字符。以下是在不同导入模式下如何保持区分大小写的示例。

    • 在命令行模式下：

      表= '\ “的Emp \”'

    • 在交互模式下：

      要导出的表（T）：“Exp”

    • 在参数文件模式中：

      表= ' “的Emp”'

  • 除非表名用引号括起，否则在命令行中指定的表名不能包含井号（＃）。类似地，在参数文件中，如果表名包含井号（＃），则导入实用程序会将该行的其余部分解释为注释，除非表名用引号括起。

    例如，如果参数文件包含以下行，则在emp#作为注释后，Import会解释该行上的所有内容，并且不会导入表dept和mydata:

    TABLES =（emp＃，dept，mydata）

    但是，给定以下行，Import实用程序会导入所有三个表，因为emp#用引号括起来：

    TABLES =（“emp＃”，dept，mydata）

    注意：

    有些操作系统需要单引号而不是双引号，反之亦然;请参阅Oracle操作系统特定的文档。不同的操作系统对表命名也有其他限制。

    例如，UNIX C shell为美元符号（$）或井号（＃）（或某些其他特殊字符）附加了特殊含义。必须使用转义字符才能在名称中通过shell获取此类字符并进入“导入”。

25.7.32 TOID_NOVALIDATE

使用TOID_NOVALIDATE参数指定要从TOID比较中排除的类型。

默认值：无

当您导入引用类型的表但数据库中已存在该类型的类型时，Import会尝试验证预先存在的类型实际上是表所使用的类型（而不是刚刚发生的其他类型）有相同的名字）。

为此，Import将类型的唯一标识符（TOID）与存储在导出文件中的标识符进行比较。如果TOID不匹配，导入将不会导入表行。

在某些情况下，您可能不希望在指定类型上进行此验证（例如，如果类型是由盒式磁带安装创建的）。您可以使用TOID_NOVALIDATE参数指定要从TOID比较中排除的类型。

语法如下：

TOID_NOVALIDATE =（ [schemaname.]typename [, ...] ）

例如：

imp scott TABLES = jobs TOID_NOVALIDATE = typ1 imp scott TABLES =工资TOID_NOVALIDATE =（fred.typ0，sally.typ2，typ3）

如果未指定类型的模式名称，则默认为导入用户的模式。例如，在前面的第一个示例中，类型typ1默认为scott.typ1 ，在第二个示例中，类型typ3默认为scott.typ3 。

请注意， TOID_NOVALIDATE仅处理表列类型。它对表类型没有影响。

具有排除类型的典型导入的输出将包含类似于以下内容的条目：

[...]。将IMP3的对象导入IMP3。 。跳过IMP2类型的TOID验证。TOIDTYP0。 。导入表“TOIDTAB3”[...]

注意：

禁止验证类型标识符时，您有责任确保导入类型的属性列表与现有类型的属性列表匹配。如果这些属性列表不匹配，则结果是不可预测的。

25.10.2.1对象已经存在

如果数据库中已存在要导入的数据库对象，则会发生对象创建错误。

接下来会发生什么取决于IGNORE参数的设置。

如果IGNORE=n （默认值），则报告错误，并继续导入下一个数据库对象。不替换当前数据库对象。对于表，此行为意味着不导入导出文件中包含的行。

如果IGNORE=y ，则不报告对象创建错误。数据库对象未替换。如果对象是表，则将行导入其中。请注意，仅忽略对象创建错误 ;所有其他错误（如操作系统，数据库和SQL错误） 报告和处理可能停止。

注意：

指定IGNORE=y可以导致将重复的行输入到表中，除非使用UNIQUE完整性约束指定了表的一个或多个列。例如，如果导入运行两次，则可能发生这种情况。

25.10.2.2序列

如果需要将序列号重置为导出文件中的值作为导入的一部分，则应删除序列。

如果在导入之前未删除序列，则不会将其设置为导出文件中捕获的值，因为Import不会删除并重新创建已存在的序列。如果序列已存在，则导出文件的CREATE SEQUENCE语句将失败，并且不会导入序列。

如果在导入行时发生资源错误，则导入将停止处理当前表并跳至下一个表。如果已指定COMMIT=y ，则Import将提交当前表的部分导入。如果不是，则在继续导入之前发生当前表的回滚。请参阅COMMIT的说明。

25.11表级和分区级导入

您可以导入表，分区和子分区。

特别：

  • 表级导入：导入导出文件中指定表的所有数据。

  • 分区级别导入：仅导入指定源分区或子分区中的数据。

  • 使用表级导入的准则
    对于每个指定的表，表级导入会导入表的所有行。
  • 使用分区级导入的准则
    分区级导入只能在表模式下指定。它允许您有选择地从导出文件中的指定分区或子分区加载数据。
  • 跨分区和表迁移数据
    如果为复合分区指定分区名称，则复合分区中的所有子分区都将用作源。

父主题： 原始导入

25.11.1使用表级导入的指南

对于每个指定的表，表级导入会导入表的所有行。

使用表级导入：

  • 可以导入使用任何导出模式导出的所有表（ TRANSPORT_TABLESPACES除外）。

  • 用户可以将表级导出文件中的整个（分区或非分区）表，分区或子分区导入到具有相同名称的（分区或非分区）目标表中。

如果该表不存在，并且导出的表已分区，则表级导入将创建分区表。如果表创建成功，则表级导入会将导出文件中的所有源数据读入目标表。导入后，目标表包含与导出文件中的源表关联的所有分区和子分区的分区定义。此操作可确保在导入时维护源分区的物理和逻辑属性（包括分区边界）。

父主题： 表级和分区级导入

25.11.2使用分区级导入的指南

分区级导入只能在表模式下指定。它允许您有选择地从导出文件中的指定分区或子分区加载数据。

使用分区级导入时，请牢记以下准则。

  • Import始终根据目标表的分区方案存储行。

  • 分区级别导入仅插入指定源分区或子分区中的行数据。

  • 如果目标表已分区，则分区级别导入将拒绝任何落在目标表的最高分区之上的行。

  • 分区级导入无法导入未分区的导出表。但是，可以使用表级导入从非分区导出表导入分区表。

  • 仅当源表（即导出时名为tablename的表）已分区并存在于导出文件中时，分区级导入才是合法的。

  • 如果分区或子分区名称不是导出文件中的有效分区，则Import会生成警告。

  • 参数中的分区或子分区名称仅引用导出文件中的分区或子分区，该分区或子分区可能不包含导出源系统上表的所有数据。

  • 如果ROWS=y （缺省值），并且导入目标系统中不存在该表，则创建该表，并将源分区或子分区中的所有行插入到目标表的分区或子分区中。

  • 如果ROWS=y （默认值）和IGNORE=y ，但表在导入之前已经存在，那么表中指定分区或子分区的所有行都将插入到表中。根据目标表的现有分区方案存储行。

  • 如果ROWS=n ，则Import不会将数据插入目标表，并继续处理与指定表以及文件中的分区或子分区关联的其他对象。

  • 如果目标表是非分区的，则分区和子分区将导入到整个表中。导入需要IGNORE=y将导出文件中的一个或多个分区或子分区导入导入目标系统上的非分区表。

父主题： 表级和分区级导入

25.11.3跨分区和表迁移数据

如果为复合分区指定分区名称，则复合分区中的所有子分区都将用作源。

在以下示例中，分区名称指定的分区是复合分区。将导入所有子分区：

imp SYSTEM FILE = expdat.dmp FROMUSER = scott TABLES = b：py

以下示例导致将表scott.e的分区qc和qd的行数据导入到表scott.e ：

imp scott FILE = expdat.dmp TABLES =（e：qc，e：qd）IGNORE = y

如果导入目标数据库中不存在表e ，则会创建表e并将数据插入到相同的分区中。如果在导入之前目标系统上存在表e ，则将行数据插入其范围允许插入的分区中。行数据最终可以在除qc和qd之外的名称的分区中。

注意：

使用分区级导入到现有表时， 必须正确设置目标分区或子分区并使用IGNORE=y.

25.12.1延迟索引创建

Import为您提供了在完成导入和插入导出数据之前延迟索引创建和维护服务的功能。

导入完成后执行索引创建，重新创建或维护通常比更新Import导入的每一行的索引更快。

索引创建可能非常耗时，因此可以在导入所有其他对象完成后更有效地完成索引。您可以通过指定INDEXES=n来推迟创建索引，直到导入完成为止。（ INDEXES=y是默认值。）然后，您可以通过在使用INDEXFILE参数时运行Import来INDEXFILE缺少的索引定义存储在SQL脚本中。否则由Import发出的索引创建语句将存储在指定的文件中。

导入完成后，必须创建索引，通常在指定connect语句的密码后，使用文件的内容（使用INDEXFILE指定）作为SQL脚本。

注意：

  • 对于对象类型，如果IGNORE = y ，对象类型已存在，并且对象标识符，哈希码和类型描述符匹配，则不报告错误。如果对象标识符或哈希码不匹配且参数TOID_NOVALIDATE尚未设置为忽略对象类型，则会报告错误，并且不会导入使用该对象类型的任何表。

25.16.3导入嵌套表

内部嵌套表与外部表分开导出。因此，可能会出现内部嵌套表中的数据可能无法正确导入的情况：

  • 假设导出具有内部嵌套表的表，然后导入该表而不删除表或从表中删除行。如果使用IGNORE=y参数，则在外部表中插入每一行时将存在约束违规。但是，可以成功导入内部嵌套表中的数据，从而导致内部表中出现重复的行。

  • 如果在外部表中插入数据时发生不可恢复的错误，则会跳过外部表中的其余数据，但不会跳过相应的内部表行。这可能导致内部表行未被外部表中的任何行引用。

  • 如果在可恢复错误之后内部表的插入失败，则其外部表行已经插入到外部表中，并且数据将继续插入到其中以及包含表的任何其他内部表中。这种情况导致部分逻辑行。

  • 如果在内部表中插入数据时发生不可恢复的错误，则Import会跳过该内部表的其余数据，但不会跳过外部表或其他嵌套表。

您应该始终仔细检查日志文件以查找外部表和内部表中的错误。为了保持一致，可能需要修改或删除表数据。

由于内部嵌套表是与外部表分开导入的，因此尝试在导入时从中访问数据可能会产生意外结果。例如，如果在导入其内部行之前访问外部行，则可能会向用户返回不完整的行。

25.16.12当触发器存在时导入LOB列

在存在触发器时导入LOB列是导入数据库对象时的注意事项之一。导入实用程序会在导入时自动将导出时为空的所有LOB更改为NULL 。

从Oracle数据库10g开始，LOB处理已得到改进，以确保触发器正常工作，并且在加载LOB时性能仍然很高。为了实现这些改进，导入实用程序会在导入时自动将导出时为空的所有LOB更改为NULL 。

如果您的应用程序期望LOB为空而不是NULL ，则在导入之后，您可以为每个LOB列发出SQL UPDATE语句。根据LOB列类型是BLOB还是CLOB ，语法将是以下之一：

UPDATE <tablename> SET <lob column> = EMPTY_BLOB（）WHERE <lob column> = IS NULL; UPDATE <tablename> SET <lob column> = EMPTY_CLOB（）WHERE <lob column> = IS NULL;

重要的是要注意，一旦执行导入，就无法区分NULL为空的LOB列和NULL LOB列。因此，如果该信息对数据的完整性很重要，那么在执行导入之前，请确保知道哪些LOB列为NULL以及哪些列为空。

25.16.13导入视图

  3. 从表空间A删除joe的表。

25.24导入统计数据

在大多数情况下，Export还会将表，索引和列的预先计算的优化程序统计信息写入转储文件。请参阅导入参数STATISTICS的说明。

由于执行ANALYZE语句所需的时间，导入通常最好使用表（及其索引和列）的预先计算的优化程序统计信息，而不是执行Export保存的ANALYZE语句。默认情况下，Import将始终使用导出转储文件中找到的预先计算的统计信息。

导出实用程序将某些预先计算的统计信息标记为可疑。在以下情况下，导入程序可能只想导入无可置疑的统计信息，而不是预先计算的统计信息：

  • 转储文件与导入客户端和导入数据库之间的字符集转换可能会更改预先计算的统计信息中隐含的整理顺序。

  • 导入表时发生行错误。

  • 执行分区级别导入（列统计信息将不再准确）。

    注意：

    指定ROWS=n不会阻止使用预先计算的统计信息。此功能允许使用生产数据库中的统计信息在非生产数据库中调整查询的计划生成。在这些情况下，导入应指定STATISTICS=SAFE.

在某些情况下，导入程序可能希望始终使用ANALYZE语句而不是预先计算的统计信息。例如，当以压缩形式导入数据时，从碎片数据库收集的统计信息可能不相关。在这些情况下，导入器应指定STATISTICS=RECALCULATE以强制重新计算统计信息。

如果您不希望通过Import建立任何统计信息，则应指定STATISTICS=NONE.

  • 将数据库置于NOARCHIVELOG模式，直到导入完成。这将减少创建和管理存档日志的开销。

  • 创建几个大型重做文件，并使任何小的重做日志文件脱机。这将导致更少的日志切换。

  • 如果可能，请将回滚段，表数据和重做日志文件全部放在不同的磁盘上。这将减少I / O争用并提高吞吐量。

  • 如果可能，请不要同时运行可能与系统资源的导入操作竞争的任何其他作业。

  • 确保字典表没有统计信息。

  • 设置TRACE_LEVEL_CLIENT = OFF在sqlnet 。 ora文件。

  • 如果可能，请在重新创建数据库时增加DB_BLOCK_SIZE的值。块大小越大，所需的I / O周期数越少。这种变化是永久性的，因此在制作之前一定要仔细考虑它的所有效果。

25.26.2更改初始化参数

有关初始化参数文件中的设置的这些建议可能有助于提高导入操作的性能。

  • 将LOG_CHECKPOINT_INTERVAL设置为大于重做日志文件大小的数字。此编号位于操作系统块中（大多数UNIX系统上为512）。这将检查点减少到最小（在日志切换时）。

  • 增加SORT_AREA_SIZE的值。增加的数量取决于系统上发生的其他活动以及可用的可用内存量。（如果系统开始交换和分页，则该值可能设置得太高。）

  • 增加DB_BLOCK_BUFFERS和SHARED_POOL_SIZE的值。

25.26.3更改导入选项

有关导入选项使用的这些建议可能有助于提高性能。

请务必阅读“ 导入参数”中所有可用选项的各个描述。

  • 设置COMMIT = N这会导致Import在每个对象（表）之后提交，而不是在每个缓冲区之后提交。这就是需要一个大型回滚段的原因。（因为在将来的版本中不推荐使用回滚段，Oracle建议您使用自动撤消管理。）

  • 为BUFFER或RECORDLENGTH指定一个较大的值，具体取决于系统活动，数据库大小等。较大的大小减少了必须为数据访问导出文件的次数。几兆字节通常就足够了。请务必检查系统是否存在过多的分页和交换活动，这可能表示缓冲区大小过大。

  • 考虑设置INDEXES = N因为在导入之后的某个时刻可以创建索引，而时间不是一个因素。如果选择执行此操作，则需要使用INDEXFILE参数来提取用于创建索引的DLL，或者使用INDEXES = Y和ROWS = N重新运行导入。

25.27使用不同版本的导出和导入

这些部分描述了与使用不同版本的Export和Oracle数据库相关的兼容性问题。

无论何时在Oracle数据库的不同版本之间移动数据，都适用以下基本规则：

  • 导入实用程序和要导入数据的数据库（目标数据库）必须是同一版本。例如，如果您尝试使用导入实用程序9.2.0.7导入9.2.0.8数据库，则可能会遇到错误。

  • Export实用程序的版本必须等于源数据库或目标数据库的版本，以较早者为准。

25.27.1使用不同版本的导出和导入时的限制

使用不同版本的导出和导入时，将适用限制。

特别：

  • 导出转储文件只能由导入实用程序读取，因为它们以特殊的二进制格式存储。

  • 可以将任何导出转储文件导入到Oracle数据库的更高版本中。

  • 导入实用程序无法读取由以后的维护版本或版本的导出实用程序创建的导出转储文件。例如，版本9.0.1导入实用程序无法导入版本9.2导出转储文件。

  • 每当较低版本的导出实用程序与更高版本的Oracle数据库一起运行时，早期版本中不存在的数据库对象类别将从导出中排除。

  • Oracle9 i Export生成的导出文件（直接路径或常规路径）与早期版本的Import不兼容，只能通过Oracle9 i Import导入。如果向后兼容性存在问题，请对Oracle9 i数据库使用早期版本的Export实用程序或版本。
