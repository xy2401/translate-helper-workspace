28.1.1会话池

Oracle XML DB协议服务器维护一个共享的会话池。每个协议连接都与此池中的一个会话相关联。关闭连接后，会话将重新放回共享池中，并可用于以后的连接。

会话池通过避免重新创建会话状态的成本来提高HTTP（S）的性能，尤其是在使用HTTP 1.0时，会为每个请求创建新的连接。例如，在创建数据库会话所需的时间内，现有的HTTP / 1.1连接可以检索几个小文件。您可以通过在Oracle XML DB配置文件xdbconfig.xml设置session-pool-size来调整池中的会话xdbconfig.xml ，或者通过将池大小设置为零来禁用它。

会话池可以影响编写Java servlet的用户，因为其他用户可以看到由另一个用户的另一个请求初始化的会话状态。因此，servlet编写器应该只使用会话内存（例如Java静态变量）来保存整个应用程序而不是特定用户的数据。每个用户的状态必须存储在数据库或查找表中，而不是假设仅为单个用户存在会话。

28.2 Oracle XML DB协议服务器配置管理

Oracle XML DB协议服务器使用存储在文件xdbconfig.xml配置参数来初始化其启动状态并管理会话级配置。无法动态更改会话池大小和超时参数，也就是说，必须重新启动数据库才能使这些更改生效。

  • 协议服务器配置参数
    描述了Oracle XML DB协议配置参数。它们包括所有协议通用的协议，特定于FTP的协议，以及特定于HTTP（S）/ WebDAV的协议。
  • 配置安全HTTP（HTTPS）
    要使存储库能够使用安全 HTTP连接（HTTPS），数据库管理员（DBA）必须相应地配置数据库：配置参数http2-port和http2-protocol ，启用HTTP侦听器以使用SSL，并启用TCPS Dispatcher的启动。然后，DBA必须停止并重新启动数据库和侦听器。
  • 使用侦听器状态检查端口配置
    您可以使用TNS监听器命令lsnrctl status来验证是否已启用HTTP（S）和FTP支持。一个例子说明了这一点
  • 数据库合并后配置协议端口参数
    在多租户容器数据库（CDB）中，协议服务器端口号区分插入的可插拔数据库（PDB）：每个此类数据库必须具有唯一的端口号。数据库管理员（DBA）必须确保PDB使用的每个端口号都是唯一的。
  • HTTP身份验证机制的配置和管理
    您可以通过在配置文件xdbconfig.xml设置元素authentication （元素httpconfig的子级）来配置身份验证机制，以允许对Oracle XML DB存储库进行HTTP访问。
  • Oracle XML DB存储库和文件系统资源
    IETF协议规范，RFC 959（FTP），RFC 2616（HTTP）和RFC 2518（WebDAV）隐含地假设服务器端的抽象分层文件系统。这将映射到Oracle XML DB Repository。存储库提供名称解析，基于ACL的安全性以及存储和检索任何内容的功能。
  • 协议服务器处理基于XML模式或基于非模式的XML文档
    Oracle XML DB协议服务器始终检查正在插入的文档是否基于在Oracle XML DB Repository中注册的XML架构。如果是， XMLType使用的XMLType存储模型由该XML模式确定。如果不是，则将文档存储为BLOB 。
  • 基于事件的日志记录
    您可以通过将事件编号31098设置为级别2来记录接收的请求和协议服务器发送的响应。

    3. 将具有服务器的可分辨名称（DN）的有效证书导入服务器。

    4. 以模糊的形式保存新钱包，以便可以在没有密码的情况下打开它。

28.2.5 HTTP认证机制的配置和管理

您可以通过在配置文件xdbconfig.xml设置元素authentication （元素httpconfig的子级）来配置身份验证机制，以允许对Oracle XML DB存储库进行HTTP访问。

从12c版本1（12.1.0.1）开始，Oracle数据库不仅支持基本身份验证，还支持摘要访问身份验证。

用户凭据区分大小写。特别是，要进行身份验证，用户名必须与创建时的名称完全匹配（默认情况下全部为大写）。

摘要访问身份验证 （也称为摘要式身份验证）提供用户凭据（名称，密码等）的加密没有完整数据加密的开销。

注意：

默认情况下，创建用户时， 不会为新用户启用摘要式身份验证。要在创建用户时启用摘要式身份验证，请使用子句DIGEST ENABLE和SQL语句CREATE USER ，指定CREATE USER的密码。

您还可以为现有用户启用摘要式身份验证。为此， DIGEST ENABLE与SQL语句ALTER USER 。这会初始化摘要式身份验证的用户密码，但不会直接启用摘要式身份验证。当用户下次使用密码登录时，将为用户启用摘要式身份验证。

您可以通过在配置文件xdbconfig.xml设置元素authentication （元素httpconfig的子httpconfig ）来配置要使用的身份验证机制。元素authentication是可选的。如果不存在则仅使用基本认证。

元素authentication有两个可能的子元素：

  • 元素allow-mechanism指定允许的机制： basic ， digest或custom 。使用单独的allow-mechanism元素指定要允许的每个机制。

  • Element digest-auth是可选的。它指定摘要机制的信息。其子元素nonce-timeout指定给定nonce保持有效的秒数。默认值为300秒。

    如果存在指定digest的allow-mechanism但没有digest-auth元素，则使用默认值。如果没有指定digest allow-mechanism ，则忽略digest-auth元素。

对于指定的每个allow-mechanism都接受HTTP请求。身份验证挑战按指定的allow-mechanism类型的顺序显示。例如，如果依次存在digest和basic ，则在基本挑战之前呈现摘要挑战。Oracle建议您始终在较弱的身份验证之前进行更强大的身份验证。（摘要式身份验证比基本身份验证更强。）

28.2.5.1摘要式身份验证的随机数

使用摘要式身份验证，服务器会在发出未经授权的响应时生成随机数。客户端在服务器的请求中包含nonce。服务器检查从客户端收到的nonce，以查看是否需要拒绝客户端身份验证。客户端可以以相同的方式验证服务器。

随机数是服务器每次发出HTTP 401（未授权）响应时生成的唯一字符串。客户端在后续请求中包含nonce，它们将发送给服务器。服务器检查它从客户端收到的nonce。如果不正确或者nonce-timeout周期已过期，服务器可以立即拒绝进行身份验证。

（客户端可以使用相同的机制来验证服务器：它可以生成自己的随机数。客户端和服务器都可以使用此客户端随机数来帮助防止特定的纯文本攻击。）

每次服务器向客户端发送摘要质询时，都会创建一个新的随机数。nonce基于nonce密钥 。安装或升级数据库时，会随机生成初始随机数密钥。

如果使用摘要式身份验证，则Oracle还建议您定期创建新的nonce密钥，以确保密钥的完整性。您使用PL / SQL过程DBMS_XDB_ADMIN.createNonceKey来执行此操作。

28.3.1.10被动模式下的FTP客户端故障

不要在被动模式下使用FTP远程连接到在listener.ora配置HOSTNAME为localhost或127.0.0.1 。

28.4.1.11.1 HTTP（S）请求字符集

使用简单但有些复杂的算法确定HTTP（S）请求主体的字符集。

  1. 将评估Content-Type标头。如果Content-Type标头指定了charset值，则使用指定的charset。

  2. 文档的MIME类型计算如下：

    1. 如果MIME类型为“ */xml ”，则字符集确定如下：

      1. 如果既不存在BOM也不存在编码声明，则使用UTF-8 。

      2. 如果存在BOM，则使用UTF-16 。

      3. 如果存在编码声明，则使用指定的编码。

    2. 如果MIME类型是text则使用ISO8859-1 。

    3. 如果MIME类型既不是“ */xml ”也不是text则使用数据库字符集。
