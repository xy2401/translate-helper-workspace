关于特权和角色

授权仅允许某些用户访问，处理或更改数据;它还会对用户访问或操作造成限制。

对用户施加的限制（或从中删除）可以应用于诸如模式，整个表或表行之类的对象。

用户权限是运行特定类型的SQL语句的权限，或者是访问属于另一个用户的对象，运行PL / SQL包等的权限。权限类型由Oracle数据库定义。

角色由用户（通常是管理员）创建，以将权限或其他角色组合在一起。它们是一种促进向用户授予多个权限或角色的方法。

权限可以分为以下一般类别：

  • 系统权限。 这些权限允许被授权者在数据库中执行标准管理员任务。仅限受信任的用户使用它们。请参阅以下部分描述权限：

    • 管理管理权限

    • 管理系统权限

    • 管理通用和本地授予的权限

  • 角色。 角色将多个特权和角色组合在一起，以便可以同时向用户授予和撤消这些特权和角色。您必须先为用户启用该角色，然后才能使用该角色。有关更多信息，请参阅以下部分：

    • 管理常见角色和本地角色

    • 管理用户角色

  • 对象权限。 每种类型的对象都具有与之关联的权限。管理对象权限描述了如何管理不同类型对象的权限。

  • 表权限。 这些特权支持DML（数据操作语言）或DDL（数据定义语言）级别的安全性。表权限描述了如何管理表权限。

  • 查看权限。 您可以将DML对象权限应用于视图，类似于表。有关更多信息，请参阅查看权限 。

  • 程序权限。 程序（包括独立过程和函数）可以被授予EXECUTE特权。有关更多信息，请参阅过程权限 。

  • 输入权限。 您可以为命名类型（对象类型， VARRAY和嵌套表）授予系统特权。有关详细信息，请参阅类型权限 。

谁应该被授予特权？

您向用户授予权限，以便他们可以完成其作业所需的任务。

您应该仅向需要该权限才能完成必要工作的用户授予权限。过度授予不必要的权限可能会危及安全性。例如，您永远不应该向不执行管理任务的用户授予SYSDBA或SYSOPER管理权限。

您可以通过两种方式向用户授予权限：

  • 您可以明确地向用户授予权限。 例如，您可以向用户psmith明确授予将记录插入employees表的权限。

  • 您可以为角色（命名的一组权限）授予权限，然后将该角色授予一个或多个用户。 例如，您可以将从employees表中选择，插入，更新和删除记录的权限授予名为clerk的角色，然后您可以向用户授予psmith和robert 。

由于角色允许更轻松，更好地管理权限，因此通常应该为角色授予权限，而不是特定用户。

Oracle Multitenant选项如何影响权限

在多租户环境中，所有用户（包括普通用户）只能在当前容器中行使其权限。

但是，连接到根的用户可以执行影响其他可插拔数据库（PDB）的某些操作。这些操作包括ALTER PLUGGABLE DATABASE ， CREATE USER ， CREATE ROLE和ALTER USER 。普通用户必须拥有启用这些操作的通常授予的权限。连接到根目录的公共用户可以通过根目录中的容器数据对象（例如，多租户容器数据库（CDB）视图和V$视图）查看与PDB相关的元数据，前提是公共用户已被授予所需权限要访问这些视图，他的CONTAINER_DATA属性已设置为允许查看有关各种PDB的数据。普通用户无法查询PDB中的表或视图。

普通用户无法在其他PDB中行使其权限。他们必须首先切换到他们想要的PDB，然后从那里行使他们的特权。要切换到其他容器，普通用户必须具有SET CONTAINER权限。必须通常或在用户尝试切换的容器中授予SET CONTAINER权限。或者，普通用户可以启动一个新的数据库会话，其初始当前容器是该用户想要的容器，依赖于该PDB中的CREATE SESSION特权。

请注意，通常授予的权限可能会干扰为各个PDB配置的安全性。例如，假设应用程序PDB数据库管理员想要阻止PDB中的任何用户修改特定的应用程序公共对象。通常授予PUBLIC或公共用户或对象上的公共角色的权限（例如UPDATE ）将绕过PDB数据库管理员的意图。

角色的功能

角色对于快速轻松地向用户授予权限非常有用。

虽然您可以使用Oracle数据库定义的角色，但如果您创建自己的角色（仅包含与您的要求相关的权限），则可以获得更多控制和连续性。Oracle可能会更改或删除Oracle数据库定义的角色中的权限。

角色具有以下功能：

  • 可以为角色授予系统或对象权限。

  • 任何角色都可以授予任何数据库用户。

  • 授予用户的每个角色在给定时间启用或禁用。用户的安全域包括当前为用户启用的所有角色的权限，并排除当前为用户禁用的任何角色的权限。Oracle数据库允许数据库应用程序和用户启用和禁用角色，以提供特权的选择性可用性。

  • 可以将角色授予其他角色。但是，角色不能授予自己，也不能循环授予。例如，如果角色role2先前已被授予角色role1则无法将角色role2授予角色role1 。

  • 如果角色未通过密码身份验证或安全应用程序角色，则可以间接向该用户授予该角色。间接授予的角色是通过已授予此用户的另一个角色授予用户的角色。例如，假设您向用户psmith授予role1角色。然后将role2和role3角色授予role1角色。角色role2和role3现在位于role1下。这意味着psmith已经间接授予的角色role2和role3 ，除了直接授予role1 。为psmith启用direct role1 psmith可为此用户启用间接角色role2和role3 。

  • （可选）您可以将直接授予的角色设为默认角色。您可以使用ALTER USER语句的DEFAULT ROLE子句启用或禁用直接授予的角色的默认角色状态。确保DEFAULT ROLE子句仅引用已直接授予用户的角色。要查找用户的直接授予角色，请查询DBA_ROLE_PRIVS数据字典视图。此视图不包括用户间接授予的角色。要查找授予其他角色的角色，请查询ROLE_ROLE_PRIVS视图。

  • 如果角色经过密码身份验证或安全应用程序角色，则您无法间接向用户授予该角色，也无法将其设置为默认角色。您只能将此类角色直接授予用户。通常，您可以使用SET ROLE语句启用密码身份验证或安全应用程序角色。

角色如何帮助或限制DDL使用

用户需要一个或多个权限才能成功执行DDL语句，具体取决于语句。

例如，要创建表，用户必须具有CREATE TABLE或CREATE ANY TABLE系统特权。

要创建属于另一个用户的表的视图，创建者必须具有CREATE VIEW或CREATE ANY VIEW系统特权以及表的SELECT object特权或SELECT ANY TABLE系统特权。

Oracle数据库通过限制在某些DDL语句中使用特定权限来避免对通过角色接收的特权的依赖性。以下规则描述了有关DDL语句的这些权限限制：

  • 允许用户执行DDL操作的所有系统特权和对象特权在通过角色接收时都可用。例如：

    • 系统特权： CREATE TABLE ， CREATE VIEW,和CREATE PROCEDURE特权

    • 对象权限：表的ALTER和INDEX权限

      如果通过角色接收到特权，则不能使用表的REFERENCES对象特权来定义表的外键。

  • 允许用户执行发出DDL语句所需的DML操作的所有系统特权和对象特权在通过角色接收时都不可用。使用CREATE VIEW语句时，安全域不包含角色。例如，通过角色授予SELECT ANY TABLE系统特权或TABLE的SELECT object特权的用户不能使用这些特权中的任何一个在属于另一个用户的表上创建视图。这是因为视图是定义者的权限对象，因此在创建它们时，您不能使用通过角色授予您的任何权限（系统权限或对象权限）。如果直接授予您权限，则可以使用该权限。但是，如果稍后撤销该特权，则视图定义将变为无效（“包含错误”），并且必须重新编译才能再次使用。

以下示例进一步阐明了通过角色接收的权限的允许和限制使用。

假设用户是：

  • 授予具有CREATE VIEW系统特权的角色

  • 直接授予具有employees表的SELECT object特权的角色

  • 直接授予departments表的SELECT object权限

鉴于这些直接和间接授予的权限：

  • 用户可以在employees和departments表上发出SELECT语句。

  • 虽然用户同时具有CREATE VIEW和SELECT权限的employees通过角色表，用户不能创建在一个视图中employees表，因为SELECT object的特权employees表是通过一个角色授予。

  • 用户可以在departments表上创建视图，因为用户通过角色具有CREATE VIEW权限，而直接具有departments表的SELECT权限。

Oracle数据库安装中的预定义角色

Oracle数据库提供了一组预定义角色，以帮助进行数据库管理。

当您运行作为数据库创建一部分的标准脚本（例如catalog.sql和catproc.sql ）时，将自动为Oracle数据库定义表4-3中列出的这些预定义角色，并将它们视为常见角色。如果您安装其他选项或产品，则可能会创建其他预定义角色。您可以通过查询DBA_ROLES数据字典视图的ROLE和ORACLE_MAINTAINED列来查找由Oracle创建和维护的角色。如果ORACLE_MAINTAINED的输出为Y ，则除非通过运行用于创建角色的脚本，否则不得修改角色。

创建角色

您可以创建使用或不使用密码进行身份验证的角色。您还可以创建外部或全局角色。

  • 关于角色的创造
    您可以使用CREATE ROLE语句创建角色。
  • 创建使用密码进行身份验证的角色
    您可以使用IDENTIFIED BY子句创建密码身份验证角色。
  • 创建没有密码身份验证的角色
    您可以通过省略IDENTIFIED BY子句来创建不需要密码的角色。
  • 创建外部或全局角色
    外部或全局角色允许数据库外部的服务将数据库角色与经过身份验证的用户相关联。
  • 改变角色
    ALTER ROLE语句可以修改角色的授权方法。

关于角色的创造

您可以使用CREATE ROLE语句创建角色。

要创建角色，您必须具有CREATE ROLE系统特权。通常，只有安全管理员具有此系统特权。创建角色后，该角色没有与之关联的权限。下一步是为新角色授予特权或其他角色。

您必须为每个角色指定在数据库的现有用户名和角色名称中创建唯一名称。角色不包含在任何用户的架构中。在使用多字节字符集的数据库中，Oracle建议每个角色名称至少包含一个单字节字符。如果角色名称仅包含多字节字符，则加密的角色名称和密码组合的安全性会大大降低。有关密码准则，请参阅“ 密码保护准则”中的准则1 。

您可以使用IDENTIFIED BY子句通过密码授予角色。此子句指定在启用角色以供授予其使用的特定用户之前，必须如何授权用户。如果未指定此子句，或者指定NOT IDENTIFIED ，则在启用角色时不需要授权。可以通过以下方式指定角色：

  • 数据库使用密码

  • 使用指定包的应用程序

  • 外部由操作系统，网络或其他外部源

  • 全局由企业目录服务

作为创建受密码保护的角色的替代方法，Oracle建议您使用安全的应用程序角色。

请注意以下有关角色创建的限制：

  • 角色和用户不能具有相同的名称。

  • 角色名称不能以COMMON_USER_PREFIX参数（默认为C## ）的值开头，除非此角色是CDB常用角色。

创建使用密码进行身份验证的角色

您可以使用IDENTIFIED BY子句创建密码身份验证角色。

  • 要创建经过密码验证的角色，请使用带有IDENTIFIED BY子句的CREATE ROLE语句。

例如：

CREATE ROLE职员IDENTIFIED BY 密码 ;

注意：

  • 您可以在代理会话中启用受密码保护的角色。安全应用程序角色和受密码保护的角色都提供了在会话中启用角色的安全方法。Oracle建议使用安全密码角色而不是密码保护角色，其中密码必须通过不安全的渠道维护和传输，或者如果有多个人需要知道密码。代理会话中受密码保护的角色适用于使用自动化设置角色的情况。
  • 如果将SQLNET.ALLOWED_LOGON_VERSION_SERVER参数设置为11或更高，则必须重新创建使用IDENTIFIED BY子句创建的角色。

角色权限和安全应用程序角色

只能通过授权的PL / SQL包或过程启用安全应用程序角色。

PL / SQL包本身反映了控制对应用程序的访问所必需的安全策略。

这种角色创建方法限制了这种类型的角色在调用应用程序中的启用。例如，应用程序可以执行身份验证和自定义授权，例如检查用户是否已通过代理连接。

这种类型的角色增强了安全性，因为密码未嵌入应用程序源代码中或存储在表中。这样，数据库执行的操作基于安全策略的实现，并且这些定义存储在一个位置，即数据库，而不是存储在应用程序中。如果您需要修改策略，则可以在一个地方执行此操作，而无需修改您的应用程序。无论用户如何连接到数据库，结果始终都是相同的，因为策略绑定到角色。

要启用安全应用程序角色，必须在用户登录之前直接从应用程序调用它，然后在用户执行安全应用程序角色授予的权限之前执行其基础包。您不能使用登录触发器来启用安全应用程序角色，也不能将此类角色作为默认角色。

启用安全应用程序角色时，Oracle数据库会验证授权的PL / SQL程序包是否在调用堆栈上，即，它验证授权的PL / SQL程序包是否正在发出启用该角色的命令。

您可以使用安全应用程序角色来确保存在数据库连接。由于安全应用程序角色是由程序包实现的角色，因此程序包可以验证用户是否可以通过中间层或特定IP地址连接到数据库。通过这种方式，安全应用程序角色可防止用户访问应用程序外部的数据。他们被迫在他们被授予的应用程序权限的框架内工作。

关于PDB锁定配置文件

PDB锁定配置文件是一组控制一组操作的命名功能。

在某些情况下，您可以单独启用或禁用操作。例如，PDB锁定配置文件可以包含禁用ALTER SYSTEM语句随附的特定子句的设置。

PDB锁定配置文件限制用户访问所提供功能的功能，类似于为用户定义的资源限制。顾名思义，您可以在CDB，应用程序容器或PDB或应用程序PDB中使用PDB锁定配置文件。您可以创建自定义配置文件以满足您网站的要求。PDB配置文件使您可以为应用程序定义自定义安全策略。此外，您还可以创建基于另一个配置文件的锁定配置文件，称为基本配置文件 。您可以将此配置文件配置为在修改基本配置文件时动态更新，或者在更新基本配置文件时将其配置为静态（不变）。锁定配置文件专为Oracle云和本地环境而设计。

在PDB之间共享身份时，可能存在提升的权限。您可以使用锁定配置文件来防止此特权提升。在以下情况下可以共享身份：

  • 在操作系统级别，当数据库与操作系统资源（如文件或进程）交互时

  • 在网络级别，当数据库与其他系统通信时，网络身份很重要

  • 在数据库内部，PDB访问或创建公共对象，或者使用数据库链接等功能跨容器边界进行通信

使用共享标识并从PDB锁定配置文件中受益的功能分为以下几类：

  • 网络访问功能。 这些是使用网络在PDB外部进行通信的操作。例如，PL / SQL包UTL_TCP ， UTL_HTTP ， UTL_MAIL ， UTL_SNMP ， UTL_INADDR和DBMS_DEBUG_JDWP执行这些操作。目前，ACL用于控制这种访问以共享网络身份。

  • 普通用户或对象访问。 这些操作是PDB中的本地用户可以通过公共用户帐户代理或访问公共模式中的对象的操作。这些类型的操作包括在公共模式中添加或替换对象，为公共对象授予权限，访问公共目录对象，向普通用户授予INHERIT PRIVILEGES角色，以及操纵用户代理到普通用户。

  • 操作系统访问。 例如，您可以限制对UTL_FILE或DBMS_FILE_TRANSFER PL / SQL包的访问。

  • 连接。 例如，您可以限制普通用户连接到PDB，或者可以限制具有SYSOPER管理权限的本地用户连接到以受限模式打开的PDB。

PDB锁定配置文件继承

PDB锁定配置文件在CDB根，应用程序根目录及其关联的PDB之间具有继承行为。

  • PDB与其各自根之间的继承路径如下：

    • 所述PDB_LOCKDOWN在CDB PDB参数设定优先于PDB_LOCKDOWN在CDB根参数设置。类似地， PDB_LOCKDOWN在应用PDB设置优先一个PDB_LOCKDOWN在应用程序根设置。

    • 如果CDB PDB（或应用程序PDB）未设置PDB_LOCKDOWN参数，则PDB将继承CDB根（或应用程序根目录）中PDB_LOCKDOWN参数的设置。

    • 如果应用程序根目录未设置PDB_LOCKDOWN参数，则应用程序根目录将继承CDB根目录中PDB_LOCKDOWN参数的设置。

  • 如果CDB PDB或应用程序PDB中的PDB_LOCKDOWN参数设置为CDB锁定配置文件，则PDB将忽略由CDB根目录或应用程序根目录中的PDB_LOCKDOWN参数设置的任何锁定配置文件。

  • PDB锁定参数可以继承应用程序锁定配置文件中规定的规则，包括来自在其最近的祖先（即应用程序根目录或CDB根目录）中设置的CDB锁定配置文件的禁用规则。这适用于应用程序PDB中的PDB_LOCKDOWN参数设置为应用程序锁定配置文件而应用程序根目录或CDB根目录中的PDB_LOCKDOWN参数设置为CDB锁定配置文件的情况。

  • 有时，组成CDB锁定配置文件的规则与应用程序锁定配置文件之间会发生冲突。在这种情况下，CDB锁定配置文件中的规则优先。例如，对于一个设定OPTION_VALUE在CDB锁定轮廓子句优先于设置为OPTION_VALUE在应用程序中的锁定轮廓子句。

默认PDB锁定配置文件

Oracle数据库提供了一组默认的PDB锁定配置文件，您可以根据站点要求进行自定义。

默认情况下，大多数这些配置文件都是空的。它们旨在作为占位符或模板供您配置，具体取决于您的部署要求。

有关这些配置文件的详细信息如下：

  • PRIVATE_DBAAS包含适用于私有云数据库即服务（DBaaS）部署的限制。这些限制是：

    • 每个PDB必须具有相同的数据库管理员

    • 允许不同的用户连接到数据库

    • 允许不同的应用

    PRIVATE_DBAAS允许用户连接到PDB，但阻止他们使用Oracle数据库管理功能。

  • SAAS包含适用于软件即服务（SaaS）部署的限制。这些限制是：

    • 每个PDB必须具有相同的数据库管理员

    • 允许不同的用户连接到数据库

    • 必须使用相同的应用程序

    SAAS锁定配置文件比PRIVATE_DBAAS配置文件更具限制性。用户可以不同，但应用程序代码是相同的;用户无法直接连接，只能通过应用程序连接;并且不授予用户执行任何管理功能的权限。

  • PUBLIC_DBAAS包含适用于公共云数据库即服务（DBaaS）部署的限制。限制如下：

    • 每个PDB中有不同的DBA

    • 不同的用户

    • 不同的应用

    PUBLIC_DBAAS锁定配置文件是锁定配置文件中限制最严格的配置文件。

创建PDB锁定配置文件

要创建PDB锁定配置文件，您必须具有CREATE LOCKDOWN PROFILE系统特权。

创建锁定配置文件后，您可以在启用它之前添加限制。
  1. 以具有CREATE LOCKDOWN PROFILE系统特权的用户身份连接到CDB根目录或应用程序根目录。
    例如，要连接到CDB根目录：
    CONNECT c ## sec_admin输入密码： 密码
  2. 运行CREATE LOCKDOWN PROFILE语句以使用以下语法创建概要文件：
    CREATE LOCKDOWN PROFILE profile_name [FROM static_base_profile |包含dynamic_base_profile ];

    在本规范中：

    • profile_name是您为锁定配置文件指定的名称。您可以通过查询DBA_LOCKDOWN_PROFILES数据字典视图的PROFILE_NAMES列来查找现有名称。

    • FROM static_base_profile使用现有概要文件中的值创建新的锁定概要文件。对基本配置文件的任何后续更改都不会影响新配置文件。

    • INCLUDING dynamic_base_profile还使用现有基本配置文件中的值创建新的锁定配置文件，但新的锁定配置文件将继承构成基本配置文件的DISABLE STATEMENT规则，以及对基本配置文件的任何后续更改。如果显式添加到新配置文件的规则与基本配置文件中的规则冲突，则基本配置文件中的规则优先。例如， OPTION_VALUE在基本轮廓子句优先于OPTION_VALUE在新的配置文件子句。

    以下两个PDB锁定配置文件语句演示了继承的工作原理：
    CREATE LOCKDOWN PROFILE hr_prof包括PRIVATE_DBAAS; CREATE LOCKDOWN PROFILE hr_prof2 FROM hr_prof;

    在第一个语句中， hr_prof继承对PRIVATE_DBAAS基本概要文件所做的任何更改。如果为PRIVATE_DBAAS启用了新语句，则会为hr_prof启用该hr_prof 。在第二份声明中，相反，当hr_prof更改，然后hr_prof2 不会改变，因为它是独立于基础轮廓。

  3. 运行ALTER LOCKDOWN PROFILE语句以提供对配置文件的限制。
    例如：
    ALTER LOCKDOWN PROFILE hr_prof DISABLE STATEMENT =（'ALTER SYSTEM'）; ALTER LOCKDOWN PROFILE hr_prof ENABLE STATEMENT =（'ALTER SYSTEM'）clause =（'flush shared_pool'）; ALTER LOCKDOWN PROFILE hr_prof DISABLE FEATURE =（'XDB_PROTOCOLS'）;

    在前面的示例中：

    • DISABLE STATEMENT = ('ALTER SYSTEM')禁用PDB的所有ALTER SYSTEM语句。

    • ENABLE STATEMENT = ('ALTER SYSTEM') clause = ('flush shared_pool')只允许对ALTER SYSTEM使用FLUSH_SHARED_POOL子句。

    • DISABLE FEATURE = ('XDB_PROTOCOLS')禁止此PDB使用XDB协议（FTP，HTTP，HTTPS）

    创建PDB锁定配置文件后，您可以使用ALTER SYSTEM SET PDB_LOCKDOWN SQL语句启用它。

使用视图提高表安全性

数据库视图可以通过限制用户可以看到的数据来提高表安全性。

要使用视图，用户必须具有适当的权限，但仅限于视图本身，而不是其基础对象。但是，如果删除了视图的基础对象的访问权限，则用户将不再具有访问权限。

出现此问题的原因是用户查询视图时使用的安全域是视图的定义者。如果从视图的定义器中撤消对基础对象的特权，则视图将变为无效，并且没有人可以使用该视图。因此，即使已授予用户访问视图的权限，如果已从视图的基础对象撤消了定义者的权限，则用户可能无法使用该视图。

例如，假设用户A创建了一个视图。用户A具有对视图的基础对象的定义者权限。然后，用户A将该视图的SELECT权限授予用户B，以便用户B可以查询该视图。但是，如果用户A不再能够访问该视图的基础对象，则用户B也不再具有访问权限。

视图为表，列级安全性和基于值的安全性添加了两个更高级别的安全性，如下所示：

  • 视图可以提供对所选基表列的访问。 例如，您可以在employees表上定义一个视图，以仅显示employee_id ， last_name和manager_id列：

    CREATE VIEW employees_manager AS SELECT last_name，employee_id，manager_id FROM employees;

  • 视图可以为表中的信息提供基于值的安全性。 视图定义中的WHERE子句仅显示选定的基表行。考虑以下两个例子：

    CREATE VIEW lowsal AS SELECT * FROM employees WHERE salary <10000;

    lowsal视图允许访问employees表的所有行，其工资值小于10000。请注意， employees表的所有列都可以在lowsal视图中访问。 

    CREATE VIEW own_salary AS SELECT last_name，salary FROM employees WHERE last_name = USER;

    在own_salary视图中，只能访问具有与视图的当前用户匹配的last_name的行。own_salary视图使用user伪列，其值始终引用当前用户。此视图结合了列级安全性和基于值的安全性。

键入依赖项

与存储的对象（如过程和表）一样，其他对象引用的类型称为依赖项。

表所依赖的类型存在一些特殊问题。由于表包含依赖于访问类型定义的数据，因此对该类型的任何更改都会导致所有存储的数据无法访问。可能导致此问题的更改是在必要时撤消使用该类型所需的权限，或者删除类型或依赖类型。如果发生这些操作，则表变为无效且无法访问。

如果再次授予所需的权限，则由于缺少权限而无效的表可以自动变为有效且可访问。由于依赖类型被删除而无效的表永远不能再次访问，唯一允许的操作是删除表。

由于撤消对类型的特权或丢弃类型可能导致的严重影响，默认情况下，SQL语句REVOKE和DROP TYPE实现受限制的语义。这意味着如果任一语句中的命名类型具有表或类型依赖项，则会收到错误并且语句将取消。但是，如果使用任一语句的FORCE子句，则该语句始终成功。如果存在依赖表，则它们将失效。

WITH GRANT OPTION条款的工作原理

WITH GRANT OPTION子句和GRANT语句可以使被授权者向其他用户授予对象权限。

其架构包含对象的用户将自动WITH GRANT OPTION子句授予所有关联的对象权限。此特权允许受让人获得多项扩展权限：

  • 被授予者可以将对象特权授予数据库中的任何用户，无论是否有GRANT OPTION ，以及数据库中的任何角色。

  • 如果满足以下两个条件，则被授权者可以在表上创建视图，并将视图的相应权限授予数据库中的任何用户或角色：

    • 被授予者使用GRANT OPTION接收表的对象权限。

    • 被授予者具有CREATE VIEW或CREATE ANY VIEW系统特权。

注意：

如果尝试将对象权限授予角色，则WITH GRANT OPTION子句无效。Oracle数据库阻止通过角色传播对象权限，以便角色的被授权者不能传播通过角色接收的对象权限。

代表对象所有者授予对象权限

GRANT ANY OBJECT PRIVILEGE系统特权使用户能够代表对象所有者授予和撤消任何对象特权。

此权限为数据库和应用程序管理员提供了一种方便的方法，可以授予对任何模式中对象的访问权限，而无需将它们连接到模式。不需要为具有此权限的架构所有者维护登录凭据，这样可以减少配置期间所需的连接数。

此系统特权是Oracle数据库提供的DBA角色的一部分，因此（使用ADMIN option ）授予连接AS SYSDBA （用户SYS ）的任何用户。与其他系统特权一样， GRANT ANY OBJECT PRIVILEGE系统特权只能由拥有ADMIN option的用户授予。

记录的对象访问权限的授予者是对象所有者或行使GRANT ANY OBJECT PRIVILEGE系统特权的人。如果具有GRANT ANY OBJECT PRIVILEGE的授权者没有 GRANT OPTION的对象特权，则对象所有者将显示为授予者。否则，当该授予者具有GRANT OPTION的对象特权时，该授予者将被记录为授予的授予者。

注意：

GRANT语句生成的审计记录始终显示执行授权的实际用户。

此功能是安全管理员的替代方案，显式必须使用GRANT和REVOKE语句向用户授予和撤消数据库角色。

使用操作系统识别用户的数据库角色的优点是可以外部化Oracle数据库的权限管理。操作系统提供的安全设施控制用户权限。此选项可以为多个系统活动提供集中安全性的优势，例如以下情况：

  • MVS Oracle管理员希望RACF组识别数据库用户角色。

  • UNIX Oracle管理员希望UNIX组识别数据库用户角色。

  • VMS Oracle管理员希望使用权限标识符来标识数据库用户角色。

使用此功能的第二个缺点是，默认情况下，如果操作系统管理角色，则用户无法通过共享服务器或任何其他网络连接连接到数据库。但是，您可以按照使用操作系统角色管理的网络连接中的说明更改此默认值。

操作系统角色识别

OS_ROLES初始化参数可用于控制操作系统识别角色的方式。

要使数据库使用操作系统在创建会话时标识每个用户的数据库角色，可以将初始化参数OS_ROLES设置为TRUE 。

如果实例当前正在运行，则必须重新启动实例。当用户尝试使用数据库创建会话时，Oracle数据库使用操作系统标识的数据库角色初始化用户安全域。

要识别用户的数据库角色，每个Oracle数据库用户的操作系统帐户必须具有操作系统标识符（这些标识符可以称为组，权限标识符或其他类似名称），以指示哪些数据库角色可供用户使用。角色规范还可以指示哪些角色是用户的默认角色以及ADMIN选项可用的角色。无论使用哪种操作系统，操作系统级别的角色规范都遵循以下格式：

ora_ ID _ ROLE [[_d] [_ a] [_ da]]

在本规范中：

  • ID的定义因不同的操作系统而异。例如，在VMS上， ID是数据库的实例标识符;在VMS上，它是计算机类型;在UNIX上，它是系统ID 。

    ID区分大小写以匹配您的ORACLE_SID 。 ROLE不区分大小写。

  • ROLE是数据库角色的名称。

  • d是一个可选字符，表示此角色是数据库用户的默认角色。

  • a是一个可选字符，表示使用ADMIN选项将该角色授予用户。这允许用户仅将角色授予其他角色。如果使用操作系统管理角色，则无法向用户授予角色。

    如果指定了d或a字符，则在该字符前面加下划线（_）。

例如，假设操作系统帐户在其配置文件中标识了以下角色：

ora_PAYROLL_ROLE1 ora_PAYROLL_ROLE2_a ora_PAYROLL_ROLE3_d ora_PAYROLL_ROLE4_da

当相应的用户连接到Oracle数据库的payroll实例时， role3和role4是默认值，而role2和role4可以使用ADMIN选项。

默认情况下，如果操作系统管理角色，则用户无法通过共享服务器连接到数据库。

此限制是默认设置，因为远程用户可以通过不安全的连接模拟其他操作系统用户。

如果您不关心此安全风险并希望将操作系统角色管理与共享服务器或任何其他网络连接一起使用，请将初始化参数REMOTE_OS_ROLES设置为TRUE 。更改将在下次启动实例并装入数据库时生效。此参数的默认设置为FALSE 。

当授予和撤销生效

根据授予或撤销的权限，授予或撤消在不同时间生效。

拨款和撤销生效如下：

  • 对任何事物（用户，角色和PUBLIC ）的系统和对象权限的所有授予和撤销都会立即生效。

  • 所有对任何事物（用户，其他角色， PUBLIC ）的角色授予和撤销仅在当前用户会话发出SET ROLE语句以在授予和撤销后重新启用角色时，或者在授予后创建新用户会话时生效或撤销。

指定用户的默认角色

当用户登录时，Oracle数据库将启用明确授予用户的所有权限以及用户默认角色中的所有权限。

  1. 确保要为其设置默认角色的用户直接授予具有GRANT语句的角色，或者该角色是由具有CREATE ROLE权限的用户CREATE ROLE 。
  2. 使用带有DEFAULT ROLE子句的ALTER USER语句指定用户的默认角色。

例如，要为用户jane设置默认角色payclerk和pettycash ：

ALTER USER jane DEFAULT ROLE payclerk，pettycash;

有关ALTER USER语句的DEFAULT ROLE子句限制的信息，请参阅Oracle数据库SQL语言参考 。

您无法在CREATE USER语句中为用户设置默认角色。首次创建用户时，默认用户角色设置为ALL ，这会导致随后授予用户的所有角色成为默认角色。使用ALTER USER语句限制默认用户角色。

注意：

创建角色（全局角色或应用程序角色除外）时，会向您隐式授予该角色，并更新您的默认角色集以包含新角色。请注意，只能为用户会话启用148个角色。将聚合角色（例如DBA角色）授予用户时，授予该角色的角色将包含在用户拥有的角色数中。例如，如果角色有20个角色，并且您将该角色授予用户，则该用户现在有21个其他角色。因此，在向用户授予新角色时，请使用ALTER USER语句的DEFAULT ROLE子句，以确保没有将太多角色指定为该用户的默认角色。

用户可以启用的最大角色数

您可以根据需要为用户授予多个角色，但在任何给定时间，可以为登录用户启用不超过148个角色。

因此，在用户会话期间，并非所有权限都可供此用户使用。作为最佳实践，将授予用户的角色数限制为用户所需的最小角色。
