250 UTL_CALL_STACK

UTL_CALL_STACK包提供了一个接口，用于提供有关当前正在执行的子程序的信息。

函数返回给定动态深度的子程序名称，单元名称，所有者名称，版本名称和行号。其他函数返回错误堆栈信息。

250.1 UTL_CALL_STACK概述

UTL_CALL_STACK包为PL / SQL程序员提供了一个接口，以获取有关当前正在执行的程序的信息，包括来自动态和词汇堆栈的子程序名称以及这些堆栈的深度。

各个函数返回给定动态深度的子程序名称，单元名称，所有者名称，版本名称和行号。更多函数返回错误堆栈信息。此类信息可用于创建更具启发性的错误日志和应用程序执行跟踪。

动态深度

PL / SQL子程序的执行实例的动态深度是递归定义的。

  • 当前正在执行的子程序实例的动态深度为1。

  • 否则，子程序实例的动态深度比它调用的子程序的动态深度多一个。

  • 如果存在由执行子程序调用或调用的SQL，Java或其他非PL / SQL上下文，则它占用调用堆栈上的级别，就像它是子程序一样。

在一个调用堆栈的情况下， A调用B ，调用C ，调用D ，调用E ，调用F ，调用E ，这个堆栈可以写成一个下面有动态深度的行：

ABCDEFE 7 6 5 4 3 2 1

词汇深度

PL / SQL子程序的词法深度是递归定义的。

  • 单位的词法深度，匿名块，触发器或ADT是一（1）。

  • 在另一个对象中定义的子程序的词法深度是一个加上该对象的词法深度。

块不会影响词汇深度。

错误深度

错误深度是错误堆栈上的错误数。

例如，请考虑以下匿名块。

开始开始......（1）提高zero_divide;其他人然后提出no_data_found时的异常;结束;当别人那么例外......（2）结束;

（1）处的误差深度为零，（2）处的误差深度为2。

回溯

回溯是从抛出异常到检查回溯的位置的跟踪。

考虑一个调用堆栈，其中A调用B调用C和C引发异常。如果在C中检查了回溯，则回溯将具有一个单位C，并且回溯深度将为1。如果在A中检查，它将有三个单位，A，B和C，回溯深度为3。

在没有异常的情况下，回溯的深度为零。

250.2 UTL_CALL_STACK安全模型

UTL_CALL_STACK上的EXECUTE被授予PUBLIC 。

UTL_CALL_STACK包不显示包装的程序单元。例如，考虑一个调用堆栈，其中程序单元A调用B ，调用C ，然后调用UTL_CALL_STACK来确定子程序列表。如果程序单元B被包装，则子程序列表仅显示程序单元C

250.3 UTL_CALL_STACK操作说明

某些操作说明适用于UTL_CALL_STACK 。

  • 编译器优化可以改变词法，动态和回溯深度。

  • 超过RPC边界不支持UTL_CALL_STACK 。例如，如果A调用远程过程B，则B将无法使用UTL_CALL_STACK获取有关A的信息。

  • 词法单元信息可通过PL / SQL条件编译功能获得，因此不会通过UTL_CALL_STACK 。

250.5 UTL_CALL_STACK数据结构

UTL_CALL_STACK包定义了VARRAY类型UNIT_QUALIFIED_NAME 。

VARRAY类型

  • UNIT_QUALIFIED_NAME

250.5.1 UNIT_QUALIFIED_NAME

此数据结构是一个varray，其各个元素按顺序是单元名称，子程序的任何词法父项和子程序名称。

TYPE UNIT_QUALIFIED_NAME是VARCHAR2的VARRAY（256）（32767）;

例

考虑以下设计的PL / SQL过程：

程序topLevel IS FUNCTION localFunction（...）RETURNS VARCHAR2是函数innerFunction（...）RETURNS VARCHAR2是BEGIN DECLARE localVar PLS_INTEGER;开始 ...（1）结束;结束;开始 ...结束;

（1）中的单位限定名称为

[“topLevel”，“localFunction”，“innerFunction”]

如果单位是匿名块，则单位名称为"__anonymous_block"
