6.4 Oracle数据库QoS管理管理什么？

Oracle数据库QoS管理与Oracle Real Application Clusters（Oracle RAC）和Oracle Clusterware配合使用。Oracle数据库QoS管理在整个Oracle RAC集群上运行，该集群可支持各种应用程序。

注意：

Oracle数据库QoS管理仅支持OLTP工作负载。不支持以下类型的工作负载（或数据库请求）：

  • 批量工作负载

  • 需要一秒以上才能完成的工作量

  • 使用并行数据操作语言（DML）的工作负载

  • 在意义利用率级别查询GV $视图的工作负载

  • 管理数据库资源以满足服务级别
    Oracle数据库QoS管理管理群集的CPU资源。Oracle数据库QoS管理不管理I / O资源，因此Oracle数据库QoS管理无法有效管理I / O密集型应用程序。
  • 高可用性管理和Oracle数据库QoS管理
    Oracle数据库QoS管理可帮助您实现应用程序工作负载的最佳性能级别。

6.4.1管理数据库资源以满足服务级别

Oracle数据库QoS管理管理群集的CPU资源。Oracle数据库QoS管理不管理I / O资源，因此Oracle数据库QoS管理无法有效管理I / O密集型应用程序。

Oracle数据库QoS管理通过以下技术与Oracle RAC数据库集成，以管理集群中的资源：

  • 数据库服务

  • Oracle数据库资源管理器

  • Oracle Clusterware

  • 运行时连接负载平衡

Oracle数据库QoS管理定期评估所有已使用资源的资源等待时间。如果Performance Class中工作请求的平均响应时间大于其Performance Performance中指定的值，则Oracle Database QoS Management使用收集的指标查找瓶颈资源。如果可能，Oracle数据库QoS管理提供了有关调整服务器池大小或更改Oracle数据库资源管理器使用的资源计划中的使用者组映射的建议。

  • 数据库服务
    数据库服务提供了一种机制，可用于将相关工作请求组合在一起。
  • Oracle数据库资源管理器
    Oracle数据库QoS管理使用Oracle数据库资源管理器来管理为性能类分配资源。
  • Oracle Clusterware
    Oracle Database QoS Management管理和监视Oracle Clusterware配置的服务器池。
  • 运行时连接负载平衡
    使用Oracle数据库QoS管理所管理资源的应用程序也可以从连接负载平衡和透明应用程序故障转移（TAF）中受益。

6.4.1.1数据库服务

数据库服务提供了一种机制，可用于将相关工作请求组合在一起。

应用程序使用数据库服务连接到群集数据库。用户发起的针对数据库的查询可以使用与基于Web的应用程序不同的服务。不同的服务可以代表不同类型的工作请求。对Oracle RAC数据库发出的每个调用或请求都是一个工作请求 。

您还可以使用数据库服务来管理和测量策略管理，管理员管理和多租户数据库的数据库工作负载。要管理服务使用的资源，某些服务可能同时部署在多个Oracle RAC实例上，而其他服务可能仅部署在单个实例上以隔离使用该服务的工作负载。

在Oracle RAC集群中，Oracle数据库QoS管理监视提供数据库服务的服务器池及其节点。服务由数据库管理员为数据库创建。对于策略管理的数据库，该服务在指定的服务器池中的所有服务器上运行。如果由于应用程序无法水平扩展而需要单独服务，则可以将服务限制为在最小和最大大小为1的服务器池中运行。大于1的池中的策略管理单一服务支持仅限于测量和监视。

要使用Oracle数据库QoS管理来管理性能，请创建一个或多个在服务器池中运行的策略管理数据库。如果您具有管理员管理的数据库，则数据库实例将放置在通用服务器池中，Oracle数据库QoS管理只能监视这些数据库。

首次配置Oracle数据库QoS管理时，将为在受监视的服务器池上发现的每个服务创建默认性能策略。这些默认性能类的名称是service_name _pc 。要监视和管理资源的工作负载必须使用数据库服务连接到数据库。

6.4.1.2 Oracle数据库资源管理器

Oracle数据库QoS管理使用Oracle数据库资源管理器来管理为性能类分配资源。

Oracle数据库资源管理器（资源管理器）是资源分配方法的示例;资源管理器可以根据管理员指定的资源计划在资源使用者组的集合中分配CPU份额。资源计划分配在CPU上运行的机会百分比。

Oracle数据库QoS管理不会调整现有的资源管理器计划; Oracle数据库QoS管理激活名为APPQOS_PLAN的资源计划，该计划是一个复杂的多级资源计划。Oracle数据库QoS管理还会创建表示每个使用者组的性能类和资源计划指令的使用者组。

实施Oracle数据库QoS管理建议以升级或降级性能类的使用者组时，Oracle数据库QoS管理会对性能类映射到资源计划中指定的CPU份额进行建议的更改。通过更改使用者组，当前未满足其性能目标的性能类将获得对CPU资源的更多访问权限。

对于多租户数据库，Oracle数据库QoS管理不是直接为PDB管理CPU份额，而是通过将PDB共享分配给CDB中的每个PDB来管理多租户数据库的CPU份额。最初，多租户数据库中的每个PDB都分配了50个PDB共享。如果性能等级未达到其性能目标，则PDB共享将从捐赠者PDB重新分配并分配给目标PDB。 Oracle数据库QoS管理管理资源计划中PDB共享的分配。Oracle数据库QoS管理用于多租户数据库的两个资源计划是：

  • ORA$QOS_CDB_PLAN ：管理每个PDB的CPU份额

  • ORA$QOS_PLAN ：管理PDB份额

注意：

除非在“编辑 Oracle数据库QoS管理的资源计划”中指定，否则不要编辑Oracle数据库QoS管理资源计划。

6.4.1.3 Oracle Clusterware

Oracle Database QoS Management管理和监视Oracle Clusterware配置的服务器池。

必须先安装和配置Oracle Clusterware，然后才能使用Oracle数据库QoS管理。集群管理员应创建用于部署策略管理的Oracle RAC数据库的服务器池。管理员管理的Oracle RAC数据库仅使用通用服务器池。

首次配置Oracle数据库QoS管理并创建初始策略集时，您可以指定应由Oracle数据库QoS管理管理哪些服务器池，哪些服务器池应仅受监控。如果选择要由Oracle数据库QoS管理进行管理的服务器池，则Oracle数据库QoS管理将监视在该服务器池中运行的所有性能类使用的资源。如果性能等级不满足其性能目标，则Oracle数据库QoS管理可以建议在服务器池之间移动服务器，以便在需要时提供其他资源。

6.4.1.4运行时连接负载平衡

使用Oracle数据库QoS管理所管理资源的应用程序也可以从连接负载平衡和透明应用程序故障转移（TAF）中受益。

运行时连接负载平衡使Oracle客户端能够在应用程序请求连接完成某些工作时在连接池中提供连接的智能分配;决定将新连接路由到哪个实例是基于数据库实例提供的当前性能级别。

通过连接负载平衡，您可以在支持服务的所有实例之间分布用户连接。对于每个服务，您可以使用带有-clbgoal选项的相应SRVCTL命令，通过设置连接负载平衡目标，定义希望侦听器用于负载平衡的方法。您还可以使用带有-failovermethod ， -failovertype选项的SRVCTL为服务的所有用户指定单个TAF策略。

6.4.2高可用性管理和Oracle数据库QoS管理

Oracle数据库QoS管理可帮助您实现应用程序工作负载的最佳性能级别。

高可用性的性能管理和管理系统密切相关。用户通常认为系统只有在其性能可接受时才能启动或可用。您可以使用Oracle数据库QoS管理和性能目标来指定和维护可接受的性能级别。

Oracle数据库QoS管理是一种运行时性能管理产品，可优化资源分配，以帮助您的系统在动态工作负载条件下满足服务级别协议。Oracle数据库QoS管理提供了一些建议，可帮助您的业务中最重要的工作获得必要的资源。Oracle数据库QoS管理有助于根据当前需求和资源可用性重新平衡资源分配。禁止非必要工作以确保对您的业务至关重要的工作成功完成。

Oracle数据库QoS管理不是用于提高性能的功能; Oracle数据库QoS管理的目标是保持最佳性能水平。Oracle数据库QoS管理假定已经适当地设置了影响性能和可用性的系统参数，并且它们是不变的。例如， FAST_START_MTTR_TARGET数据库参数控制数据库写程序检查点阻塞数据文件的频率，以最大限度地缩短实例恢复时间。对此参数使用较低的值可减少恢复数据库所需的时间，但更频繁地写入重做日志数据的开销会对数据库的性能产生负面影响。Oracle数据库QoS管理不会就为此类参数指定的值提出建议。

高可用性管理包含许多与工作负载无关且不受管理工作负载影响的问题。例如，系统可用性主要取决于软件升级事件的频率和持续时间。系统可用性还直接取决于硬件故障的频率。管理工作负载不能改变软件升级的频率或硬件故障的频率。
