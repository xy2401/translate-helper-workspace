12调整SQL语句

SQL语句指定您希望Oracle数据库检索的数据。例如，SQL语句可以检索部门中员工的姓名。当Oracle数据库执行SQL语句时， 查询优化器 （也称为优化器 ）首先确定检索结果的最佳和最有效的方法。

优化器确定读取表中的所有数据（称为全表扫描）或使用索引是否更有效。它比较了所有可能方法的成本，并以最低成本选择方法。物理执行SQL语句的访问方法称为执行计划 ，优化器负责生成。确定执行计划是处理任何SQL语句的重要步骤，并且可以极大地影响执行时间。

查询优化器还可以帮助您调整SQL语句。通过使用SQL Tuning Advisor 和SQL Access Advisor ，您可以在顾问模式下运行查询优化器来检查SQL语句或一组语句，并确定如何提高其效率。SQL Tuning Advisor和SQL Access Advisor可以提出各种建议，例如：

  • 创建SQL配置文件

  • 重构SQL语句

  • 创建其他索引或物化视图

  • 刷新优化程序统计信息

此外，Oracle企业管理器云控制（云控制）使您可以轻松接受和实施其中的许多建议。

SQL Access Advisor主要负责提出架构修改建议，例如添加或删除索引和物化视图。SQL Tuning Advisor提出其他类型的建议，例如创建SQL配置文件和重构SQL语句。如果通过创建新索引可以获得显着的性能改进，那么SQL Tuning Advisor可能会推荐它。但是，应使用包含一组代表性SQL语句的SQL工作负载运行SQL Access Advisor来验证此类建议。

12.1使用SQL Tuning Advisor调整SQL语句

您可以使用SQL Tuning Advisor调整一个或多个SQL语句。调整多个语句时，SQL Tuning Advisor无法识别SQL语句之间的相互依赖性。相反，SQL Tuning Advisor提供了一种方便的方法来获取大量SQL语句的调优建议。

Oracle数据库可以自动生成SQL调优报告。自动SQL调整在系统维护窗口期间作为自动维护任务运行，搜索改进高负载SQL语句执行计划的方法。维护窗口是连续的时间间隔，在此期间运行自动维护任务。

12.1.2查看自动SQL调整结果

通过分析自动工作负载存储库（AWR）中的数据，数据库可以识别日常维护任务。自动维护任务基础结构（称为AutoTask ）将这些任务安排在维护窗口中运行。

维护窗口是属于名为MAINTENANCE_WINDOW_GROUP的窗口组的Oracle Scheduler时间间隔。默认情况下，为一周中的每一天安排一个窗口。您可以自定义这些维护窗口的属性，包括开始和结束时间，频率和一周中的几天。

默认情况下，AutoTask在所有维护窗口中运行以下自动维护任务：

  • 优化器统计信息收集

  • 细分顾问

12.2管理SQL调整集

SQL调优集 是一个数据库对象，包含一个或多个SQL语句及其执行统计信息和上下文。您可以将该集用作顾问程序的输入，例如SQL Tuning Advisor，SQL Access Advisor和SQL Performance Analyzer。您可以将SQL语句加载到来自不同SQL源的SQL调优集中，例如AWR，游标缓存或您标识的高负载SQL语句。

SQL调优集包括以下内容：

  • 一组SQL语句

  • 关联的执行上下文，例如：

    • 用户架构

    • 应用程序模块名称和操作

    • 绑定值列表

    • 游标编译环境

  • 关联的基本执行统计信息，例如：

    • 经过的时间和CPU时间

    • 缓冲区得到

    • 磁盘读取

    • 行已处理

    • 光标抓取

    • 执行次数和完成执行次数

    • 优化成本

    • 命令类型

  • 每个SQL语句的关联执行计划和行源统计信息（可选）

可以使用应用程序模块名称和操作或任何执行统计信息来过滤SQL语句。此外，可以根据执行统计信息的任意组合对SQL语句进行排名。

SQL调优集是可传输的，可以在数据库之间传输SQL工作负载，以进行远程性能诊断和调优。在生产系统上标识高负载SQL语句时，可能不希望直接在此系统上执行调查和调优活动。此功能使您可以将高负载SQL语句传输到测试系统，从而可以安全地分析和调整它们。

12.2.1创建SQL调优集

本节介绍如何使用Cloud Control创建SQL调优集。

要创建SQL调优集：

  1. 指定SQL调优集的初始选项，如“ 创建SQL调优集：选项 ”中所述 。

  2. 选择用于收集SQL语句并将其加载到SQL调优集中的加载方法 ，如“ 创建SQL调整集：加载方法 ”中所述 。

  3. 指定SQL调优集的过滤器选项，如“ 创建SQL调整集：过滤器选项 ”中所述 。

  4. 安排并提交作业以收集SQL语句并将其加载到SQL调优集中，如“ 创建SQL调优集：计划 ”中所述 。

12.2.1.1创建SQL调优集：选项

创建SQL调优集的第一步是指定集的选项，例如名称，所有者和描述。

要指定用于创建SQL调优集的选项：

  1. 访问数据库主页。

    有关详细信息，请参阅“ 访问数据库主页 ” 。

  2. 从“ 性能”菜单中，选择“ SQL” ，然后选择“ SQL调整集” 。

    如果出现“数据库登录”页面，则以具有管理员权限的用户身份登录。将出现“SQL调整集”页面。

  3. 单击“ 创建”

    将出现“创建SQL调整集：选项”页面。

  4. 输入以下详细信息：

    • 在“ SQL调整集名称”中 ，输入SQL调整集的名称 。

    • 在“ 所有者”中 ，输入SQL调整集的所有者。

    • 在“ 描述”中 ，输入SQL调优集的描述。

  5. （可选）要创建空SQL调优集并在以后向其添加SQL语句，请完成以下步骤：

    1. 选择“ 创建空SQL调优集” 。

    2. 单击下一步 。

      将出现“创建SQL调整集：查看”页面。

    3. 查看SQL调优集选项，然后单击“ 提交” 。

      创建空SQL调优集。您可以稍后向其添加SQL语句。

  6. 单击下一步 。

    将出现“创建SQL调整集：加载方法”页面。


    插图描述sts_load_methods.gif

12.2.1.2创建SQL调优集：加载方法

为SQL调整集指定选项后，选择加载方法用于收集SQL语句并将其加载到SQL调优集中，如以下部分所述：

  • 从游标缓存中递增地加载活动SQL语句

  • 从游标缓存加载SQL语句

  • 从AWR快照加载SQL语句

  • 从AWR基准加载SQL语句

  • 从用户定义的工作负载加载SQL语句

12.2.1.2.1从游标缓存中递增地加载活动SQL语句

您可以在指定的时间段内以增量方式将活动SQL语句从游标缓存加载到SQL调优集中。此技术使您不仅可以收集存储在SQL缓存中的当前和最新SQL语句，还可以收集将来在指定时间段内运行的SQL语句。

从游标缓存中逐步加载活动SQL语句：

  1. 访问“创建SQL调优集：加载方法”页面，如“ 创建SQL调整集：选项 ”中所述 。

  2. 选择从游标缓存中一段时间内逐步捕获活动SQL语句 。

  3. 在“ 持续时间”字段中，指定捕获活动SQL语句的时间。

  4. 在“ 频率”字段中，指定在指定持续时间内捕获活动SQL语句的频率 。

  5. 单击下一步 。

    将出现“创建SQL调整集：过滤器选项”页面。

  6. 继续执行下一步，如“ 创建SQL调整集：过滤器选项 ”中所述 。

12.2.1.2.2从游标缓存加载SQL语句

您可以将游标缓存中的SQL语句加载到SQL调优集中。但是，因为只有当前和最近的SQL语句在SQL缓存中，所以只收集这些语句一次可能会导致SQL调优集，这不代表整个数据库工作负载。

要从游标缓存加载SQL语句：

  1. 访问“创建SQL调优集：加载方法”页面，如“ 创建SQL调整集：选项 ”中所述 。

  2. 选择仅加载一次SQL语句 。

  3. 从“数据源”列表中，选择“ 光标缓存” 。

  4. 单击下一步 。

    将显示“创建SQL调整集：过滤器选项”页面。

  5. 继续执行下一步，如“ 创建SQL调整集：过滤器选项 ”中所述 。

12.2.1.2.3从AWR快照加载SQL语句

您可以加载在AWR快照中捕获的SQL语句。当您希望收集可用于以后比较或调整目的的特定快照时段的SQL语句时，这非常有用。

从AWR快照加载SQL语句：

  1. 访问“创建SQL调优集：加载方法”页面，如“ 创建SQL调整集：选项 ”中所述 。

  2. 选择仅一次加载语句 。

  3. 在“ 数据源”列表中，选择“ AWR快照” 。

  4. 在AWR快照字段中，选择要包括的快照。执行以下操作之一：

    • 选择ALL或一段时间，例如过去24小时 ，然后转到步骤6 。

      仅包括在指定时间内捕获并存储在AWR中的快照。

    • 选择“ 自定义” ，然后转到步骤5 。

      仅包括在您指定的自定义时间段内捕获并存储在AWR中的快照。

  5. 要选择快照的自定义时间段，请完成以下步骤：

    1. 选择Customize ，然后单击Go 。

      将打开“选择时间段”窗口。

    2. 对于启动快照，请选择“ 期间开始时间” ，然后单击“活动会话”图表下方与所需开始时间对应的快照图标。

    3. 对于结束快照，请选择“ 期末结束时间” ，然后单击“活动会话”图表下方与所需结束时间对应的快照图标。

    4. 单击选择 。

    在该示例中，选择在2011年12月27日上午9:00拍摄的快照作为开始时间，并且选择在2011年12月27日上午11:00拍摄的快照作为结束时间。


    插图描述sts_load_time_period.gif

  6. 单击下一步 。

    将显示“创建SQL调整集：过滤器选项”页面。

  7. 继续执行下一步，如“ 创建SQL调整集：过滤器选项 ”中所述 。

12.3管理SQL配置文件

SQL配置文件是在自动调整SQL语句期间构建的一组辅助信息。SQL配置文件是SQL语句对表的统计信息。

在运行具有有限范围的SQL Tuning Advisor任务时，优化程序会对基数，选择性和成本进行有时显着偏离的估计，从而导致执行计划不佳。要解决此问题，请考虑运行具有全面范围的SQL Tuning Advisor任务，以使用采样和部分执行技术将其他信息收集到SQL配置文件中。数据库可以使用配置文件来验证，并在必要时调整优化程序估计值。

在SQL分析期间，优化程序使用SQL语句的执行历史记录为优化程序参数创建适当的设置。SQL分析完成后，优化程序使用SQL配置文件和常规数据库统计信息中的信息来生成执行计划。附加信息使数据库能够为相应的SQL语句生成经过良好调整的计划。

运行具有全面范围的SQL Tuning Advisor任务后，可能会建议使用SQL配置文件。如果接受建议，则数据库将创建SQL配置文件并为SQL语句启用它。

在某些情况下，您可能希望禁用SQL配置文件。例如，您可能希望在不使用SQL配置文件的情况下测试SQL语句的性能，以确定SQL配置文件是否真正有用。如果在禁用SQL配置文件后SQL语句执行不佳，则应再次启用它以避免性能下降。如果在禁用SQL配置文件后SQL语句执行得最佳，那么您可能希望从数据库中删除SQL配置文件。

要启用，禁用或删除SQL配置文件：

  1. 访问数据库主页。

    有关详细信息，请参阅“ 访问数据库主页 ” 。

  2. 从Performance菜单中，选择Top Activity 。

    如果出现“数据库登录”页面，则以具有管理员权限的用户身份登录。将显示“热门活动”页面。

  3. 在Top SQL下，单击使用SQL配置文件的SQL语句的SQL ID链接。

    将出现“SQL详细信息”页面

  4. 单击“ 计划控制”选项卡。

    SQL配置文件和轮廓下显示SQL配置文件列表。

    • 要禁用已启用的SQL配置文件，请单击“ 禁用/启用” 。

    • 要删除SQL配置文件，请单击“ 删除” 。

    出现确认页面。

12.4管理SQL计划基准

SQL计划管理是一种预防机制，使优化程序能够自动管理执行计划，确保数据库仅使用已知或已验证的计划。在此上下文中，计划包括优化程序重现执行计划所需的所有与计划相关的信息。

SQL计划管理使用SQL计划基准。SQL计划基准是一组允许优化程序用于SQL语句的已接受计划。在典型用例中，只有在数据库验证计划执行良好后，才会在SQL计划基准中接受计划。

SQL计划管理避免了计划更改导致的SQL性能回归。新优化程序统计信息，初始化参数更改，数据库升级等事件可能会导致执行计划发生更改。这些更改可能导致手动修复的SQL性能回归很难且耗时。无论数据库中的更改如何，SQL计划基准都会保留SQL语句的性能。SQL计划管理通过仅验证和接受可提高性能的计划更改来适应此类更改。

自动捕获SQL计划或手动加载计划使SQL计划管理意识到计划。计划的演变是优化程序验证新计划并将其添加到现有SQL计划基准的过程。本节包含以下主题：

  • 自动捕获SQL计划基准

  • 手动加载SQL计划基准

  • 不断发展的SQL计划

12.4.3不断发展的SQL计划

SQL语句的SQL计划基准通常以单个接受的计划开始。但是，某些SQL语句在不同条件下使用不同计划执行时表现良好。例如，具有绑定变量的SQL语句可能有几个好的计划，这些绑定变量的值导致不同的选择性。创建物化视图或索引或重新分区表可能会使当前计划比其他计划更昂贵。

如果从未将新计划添加到SQL计划基准中，则某些SQL语句的性能可能会降低。因此，有时需要发展新发现的计划，以确定是否应将它们添加到SQL计划基准中。计划演变通过在将新计划包含在SQL计划基准之前验证新计划的性能来防止性能回归。

计划进化包括以下不同步骤：

  1. 验证未接受的计划的执行情况至少与SQL计划基准中的已接受计划一样。

  2. 将计划历史记录中的未接受计划添加到计划基准中作为已被证明执行的已接受计划以及先前接受的计划。

您可以手动生成计划，也可以使用SQL计划管理（SPM）Evolve Advisor。

默认情况下，SPM Evolve Advisor每天在计划的维护时段中运行。它对所有未接受的计划进行排名，然后在窗口期间执行尽可能多的计划的测试执行。evolve顾问程序在SQL计划基准中选择成本最低的已接受计划，以与每个未接受的计划进行比较。如果计划的执行情况比现有的已接受计划好得多，那么数据库会自动接受它。

  2. 选择Performance ，然后选择SQL ，然后选择SQL Plan Control 。

    如果出现“数据库登录”页面，则以具有管理员权限的用户身份登录。将出现“SQL计划控制”页面的“SQL配置文件”子页面。

  3. 单击“ SQL计划基准” 。

    将出现SQL Plan Baseline子页面。

  4. 在表中，选择一个或多个在“接受”列中为“否”的SQL计划，然后单击“ 演变” 。

    将显示“Evolve SQL计划基准”页面。

  5. 指定要执行的选项。

    • 对于“验证性能”，请选择以下选项之一：

      • 选择“ 是”以使数据库验证计划的性能是否与当前基准计划一样好或更好。

      • 选择“ 否”可自动演变计划，无论其执行方式如何。

    • 对于时间限制，请选择以下选项之一：

      • 自动让数据库确定花费多长时间来验证未接受计划的性能。

      • 无论需要多长时间， 无限制地运行验证。

      • 指定以指定验证过程的时间限制。在关联字段中输入以分钟为单位的值。

    • 对于Action，请选择以下选项之一：

      • 报告并接受让数据库接受计划并创建报告。

      • 报告仅用于让数据库创建报告但不接受该计划。

    单击“ 确定”以实施选项。

    出现一份报告。查看报告后，单击“ 返回”以返回“SQL计划基准”子页面。
