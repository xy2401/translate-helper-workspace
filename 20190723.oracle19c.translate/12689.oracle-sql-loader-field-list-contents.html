10.2.1将POSITION用于包含选项卡的数据

确定字段位置时，请警告数据文件中的选项卡。

假设您使用SQL * Loader高级SQL字符串功能从格式化报表中加载数据。您可能首先查看报告的打印副本，仔细测量所有字符位置，然后创建控制文件。在这种情况下，当您尝试加载数据时，很可能会因多个“无效数字”和“缺少字段”错误而导致加载失败。

当数据包含选项卡时会发生这类错误。打印时，每个选项卡都会展开以在纸张上消耗多个列。但是，在数据文件中，每个选项卡仍然只有一个字符。因此，当SQL * Loader读取数据文件时， POSITION规范是错误的。

要解决此问题，请检查数据文件中的选项卡并调整POSITION规范，否则使用分隔字段。

10.2.2使用具有多个表载荷的POSITION

本节介绍如何使用具有多个表加载的POSITION。

在多表加载中，指定多个INTO TABLE子句。为第一个表的第一列指定POSITION(*) ，将相对于逻辑记录的开头计算位置。为后续表的第一列指定POSITION(*) ，将相对于上次加载的表的最后一列计算位置。

因此，当后续的INTO TABLE子句开始时，位置不会自动设置为逻辑记录的开头。这允许多个INTO TABLE子句处理同一物理记录的不同部分。有关示例，请参阅提取多个逻辑记录 。

逻辑记录可能包含两个表之一的数据，但不包含两个表。在这种情况下，您将重置POSITION 。不使用POSITION(*+或使用POSITION(*+ n )作为INTO TABLE子句中的第一个字段，而是使用POSITION(1)或POSITION( n ) 。

10.4 SQL * Loader数据类型

SQL * Loader数据类型可以分为便携式和非便携式数据类型。

在这两组中的每一组中，数据类型被子分组为值数据类型和长度值数据类型。

便携式与非便携式是指数据类型是否与平台相关。平台依赖性可能存在有几个原因，包括不同平台的字节排序方案（big-endian与little-endian）的差异，平台中位数的差异（16位，32位，64位） ，有符号数表示方案的差异（2的补码与1的补码），依此类推。在某些情况下，例如字节排序方案和平台字长，SQL * Loader提供了帮助克服平台依赖性的机制。在适当的数据类型的描述中讨论了这些机制。

便携式和非便携式数据类型都可以是值或长度值。值数据类型假定数据字段具有单个部分。长度值数据类型要求数据字段由两个子字段组成，其中长度子字段指定值子字段可以有多长。

注意：

从Oracle Database 12c第1版（12.1）开始，当COMPATIBLE初始化参数设置为12.0或更高版本且MAX_STRING_SIZE初始化参数为0时，Oracle Database VARCHAR2 ， NVARCHAR2和RAW数据类型的最大大小已增加到32 KB。设置为EXTENDED 。 SQL * Loader支持这个新的最大大小。

10.4.1.1 INTEGER（ n ）

数据是全字二进制整数，其中n是可选地提供的长度1,2,4或8。如果未给出长度规范，则长度（以字节为单位）基于特定平台上C编程语言中LONG INT的大小。

INTEGER不可移植，因为它们的字节大小，字节顺序以及有符号值的表示在系统之间可能不同。但是，如果系统之间签名值的表示相同，则SQL * Loader可能能够以正确的结果访问INTEGER数据。如果使用长度规范（ n ）指定INTEGER ，并且使用适当的技术（如果需要）来指示数据的字节顺序，则SQL * Loader可以在系统之间以正确的结果访问数据。如果指定的INTEGER没有长度规范，那么只有当C编程语言中LONG INT的大小在两个系统上的字节长度相同时，SQL * Loader才能以正确的结果访问数据。在这种情况下，仍然必须使用适当的技术（如果需要）来指示数据的字节顺序。

在输入数据是在字长不同于运行SQL * Loader的平台上创建的情况下，为二进制整数指定显式长度非常有用。例如，可以在64位平台上创建包含二进制整数的输入数据，并使用32位平台上的SQL * Loader将其加载到数据库中。在这种情况下，使用INTEGER(8)指示SQL * Loader将整数处理为8字节数量，而不是4字节数量。

默认情况下， INTEGER被视为SIGNED数量。如果希望SQL * Loader将其视为无符号数量，请指定UNSIGNED 。要返回默认行为，请指定SIGNED 。

10.4.1.2 SMALLINT

数据是半字二进制整数。字段的长度是系统上半字整数的长度。默认情况下，它被视为SIGNED数量。如果希望SQL * Loader将其视为无符号数量，请指定UNSIGNED 。要返回默认行为，请指定SIGNED 。

只有在SHORT INT具有相同的字节长度的系统之间才能加载SMALLINT正确的结果。如果系统之间的字节顺序不同，则使用适当的技术指示数据的字节顺序。见字节顺序 。

注意：

这是C编程语言中的SHORT INT数据类型。确定其长度的一种方法是制作一个没有数据的小型控制文件，并查看生成的日志文件。无法在控制文件中覆盖此长度。

10.4.2.9冲突本机数据类型字段长度

有几种方法可以指定字段的长度。如果指定了多个长度且它们发生冲突，则其中一个长度优先。存在冲突时会发出警告。以下规则确定使用的字段长度：

  1. 无论POSITION子句中指定的字节数是多少， SMALLINT ， FLOAT和DOUBLE数据的大小都是固定的。

  2. 如果为DECIMAL ， INTEGER ， ZONED ， GRAPHIC ， GRAPHIC EXTERNAL或RAW字段指定的长度（或精度）与根据POSITION (start:end)规范计算的大小冲突，则使用指定的长度（或精度）。

  3. 如果为字符或VARGRAPHIC字段指定的最大大小与从POSITION (start:end)规范计算的大小冲突，则使用指定的最大值。

10.4.3数据类型转换

本节介绍数据类型转换。

控制文件中的数据类型规范告诉SQL * Loader如何解释数据文件中的信息。服务器定义数据库中列的数据类型。这两者之间的链接是控制文件中指定的列名 。

SQL * Loader从控制文件中的数据类型规范引导，从输入文件中的字段中提取数据。SQL * Loader然后将字段发送到服务器以存储在适当的列中（作为行插入数组的一部分）。

SQL * Loader或服务器进行任何必要的数据转换，以适当的内部格式存储数据。这包括将数据从数据文件字符集转换为数据库字符集时的数据。

注意：

当您使用SQL * Loader常规路径将数据文件中的字符数据加载到LONG RAW列时，字符数据被解释为具有HEX字符串。SQL将HEX字符串转换为其二进制表示形式。请注意，任何超过4000字节的字符串都超出了SQL HEXTORAW转换运算符的字节限制。将返回错误。SQL * Loader将拒绝该行并继续加载。

文件中数据的数据类型不需要与Oracle表中列的数据类型相同。Oracle数据库会自动执行转换，但您需要确保转换有意义且不会产生错误。例如，当数据类型为CHAR的数据文件字段加载到数据类型为NUMBER的数据库列中时，必须确保字符字段的内容表示有效数字。

注意：

SQL *装载机不包含用于Oracle内部数据类型，例如数据类型规格NUMBER或VARCHAR2 。SQL * Loader数据类型描述了可以使用文本编辑器（ 字符数据类型）和标准编程语言（ 本机数据类型）生成的数据。但是，尽管SQL * Loader无法识别NUMBER和VARCHAR2等数据类型，但Oracle数据库可以转换的任何数据都可以加载到这些或其他数据库列中。

10.4.6.1仅使用TERMINATED BY的字段

10.4.6.2使用不带TERMINATED BY的ENCLOSED BY的字段

本节介绍使用不带TERMINATED BY ENCLOSED BY字段。

  5. 如果在找到第二个ENCLOSED BY分隔符之前找到记录的结尾，则拒绝该行。

10.4.6.3使用带有终止方式的ENCLOSED BY的字段

  1. 跳过字段开头的任何空格。

  2. 找到的第一个非空白字符必须是与第一个ENCLOSED BY分隔符匹配的字符串的开头。如果不是，则拒绝该行。

  3. 如果找到第一个ENCLOSED BY分隔符，则开始搜索第二个ENCLOSED BY分隔符。

  6. 如果找到第二个ENCLOSED BY分隔符，则解析器将查找TERMINATED BY分隔符。如果TERMINATED BY分隔符不是WHITESPACE ，则跳过在第二个ENCLOSED BY分隔符的末尾和TERMINATED BY分隔符之间找到的空格。

    在第二个ENCLOSED BY分隔符和TERMINATED BY分隔符之间只允许WHITESPACE 。任何其他字符都会导致错误。

10.4.6.4使用带终止的BY的OPTIONALLY ENCLOSED的字段

本节介绍使用OPTIONALLY ENCLOSED BY和TERMINATED BY字段。

当字段使用OPTIONALLY ENCLOSED BY子句和TERMINATED BY子句时，会发生以下步骤。

  1. 跳过字段开头的任何空格。

  2. 解析器检查找到的第一个非空白字符是否是与第一个OPTIONALLY ENCLOSED BY分隔符匹配的字符串的开头。如果不是，并且数据中不存在OPTIONALLY ENCLOSED BY分隔符，则从字段的当前位置读取字段的数据，直到但不包括第一次出现的TERMINATED BY分隔符。如果在第一列位置找到TERMINATED BY分隔符，则该字段为空。如果在TERMINATED BY分隔符之前找到记录的结尾，则直到记录末尾的所有数据都被视为该字段的一部分。

  3. 如果找到第一个OPTIONALLY ENCLOSED BY分隔符，则开始搜索第二个OPTIONALLY ENCLOSED BY分隔符。

  4. 如果第二个OPTIONALLY ENCLOSED BY分隔符中的两个被发现彼此相邻，则它们被解释为分隔符的单个出现并且被包括作为该字段的数据的一部分。然后继续搜索第二个OPTIONALLY ENCLOSED BY分隔符。

  5. 如果在找到第二个OPTIONALLY ENCLOSED BY分隔符之前找到记录的结尾，则拒绝该行。

  6. 如果OPTIONALLY ENCLOSED BY分隔符是存在于数据，那么解析器寻找TERMINATED BY分隔符。如果TERMINATED BY分隔符不是WHITESPACE ，那么在第二个OPTIONALLY ENCLOSED BY结束之间发现空格 跳过分隔符和TERMINATED BY分隔符。

  7. 如果在找到TERMINATED BY分隔符之前找到记录结尾，则不会拒绝该行。

警告：

将空格字符指定为TERMINATED BY分隔符并使用OPTIONALLY ENCLOSED BY时要小心。在查找OPTIONALLY ENCLOSED BY分隔符时，SQL * Loader会删除前导空格。如果数据在记录中间包含两个相邻的TERMINATED BY分隔符（通常用于将记录中的字段设置为NULL），则第一个TERMINATED BY分隔符的空格将用于终止字段，但剩余的空格将被视为下一个字段的前导空格而不是下一个字段的TERMINATED BY分隔符。要加载NULL值，必须在数据中包含ENCLOSED BY分隔符。

10.6使用WHEN，NULLIF和DEFAULTIF子句

本节介绍如何使用WHEN ， NULLIF和DEFAULTIF子句。

以下信息适用于标量字段。对于非标量字段（列对象，LOB和集合）， WHEN ， NULLIF和DEFAULTIF子句的处理方式不同，因为非标量字段更复杂。

WHEN ， NULLIF或DEFAULTIF子句的结果可以不同，具体取决于子句是指定字段名称还是位置。

  • 如果WHEN ， NULLIF或DEFAULTIF子句指定字段名称，则SQL * Loader将该子句与该字段的计算值进行比较。评估值考虑了修剪的空白。有关修剪空白和制表符的信息，请参阅修剪空白 。

  • 如果WHEN ， NULLIF或DEFAULTIF子句指定位置，则SQL * Loader将该子句与数据文件中的原始逻辑记录进行比较。在这种情况下，逻辑记录上没有进行空白修剪。

如果字段具有修剪的空白，或者WHEN ， NULLIF或DEFAULTIF子句包含空格或制表符或使用BLANKS参数， NULLIF更有可能出现不同的结果。如果对name指定的字段和position指定的相同字段要求相同的结果，请使用PRESERVE BLANKS选项。PRESERVE BLANKS选项指示SQL * Loader在评估字段值时不要修剪空格。

WHEN ， NULLIF或DEFAULTIF子句的结果也受SQL * Loader操作顺序的影响，如以下步骤所述。SQL * Loader按顺序执行这些步骤，但并不总是执行所有这些步骤。设置字段后，将忽略该过程中的所有剩余步骤。例如，如果在步骤5中设置了该字段，则SQL * Loader不会继续执行步骤6。

  1. SQL * Loader评估输入记录的每个字段的值，并修剪任何应修剪的空白（根据修剪空白和制表符的现有指南）。

  2. 对于每个记录，SQL * Loader评估表的任何WHEN子句。

  3. 如果记录满足表的WHEN子句，或者没有指定WHEN子句，则SQL * Loader会检查每个字段是否有NULLIF子句。

  4. 如果存在NULLIF子句，则SQL * Loader将对其进行求值。

  5. 如果满足NULLIF子句，则SQL * Loader将字段设置为NULL 。

  6. 如果不满足NULLIF子句，或者没有NULLIF子句，则SQL * Loader将从字段评估中检查字段的长度。如果字段在字段评估中的长度为0（例如，它是空字段，或者空白修剪导致空字段），则SQL * Loader将字段设置为NULL 。在这种情况下，不会计算为该字段指定的任何DEFAULTIF子句。

  7. 如果任何指定的NULLIF子句为false或没有NULLIF子句，并且字段评估中字段的长度不为0，则SQL * Loader将在该字段中检查DEFAULTIF子句。

  8. 如果存在DEFAULTIF子句，则SQL * Loader将对其进行求值。

  9. 如果满足DEFAULTIF子句，则如果数据文件中的字段是数字字段，则该字段设置为0。如果字段不是数字字段，则设置为NULL 。以下字段是数字字段，如果它们满足DEFAULTIF子句，则将设置为0：

    • BYTEINT

    • SMALLINT

    • 整数

    • 浮动

    • 双

    • ZONED

    • （打包） DECIMAL

    • 数字EXTERNAL （ INTEGER ， FLOAT ， DECIMAL和ZONED ）

  10. 如果不满足DEFAULTIF子句，或者没有DEFAULTIF子句，则SQL * Loader将使用步骤1中的计算值设置该字段。

SQL * Loader运行的顺序可能会导致您不期望的结果。例如， DEFAULTIF子句可能看起来像是将数字字段设置为NULL而不是0。

注意：

如这些步骤所示， NULLIF和DEFAULTIF子句的存在导致SQL * Loader必须执行的额外处理。这可能会影响性能。请注意，在步骤1期间，如果SQL * Loader的计算长度为零，则将字段设置为NULL。要提高性能，请考虑是否可以更改数据以利用此功能。与处理NULLIF或DEFAULTIF子句相比，作为步骤1的一部分检测到NULL的速度要快得多。

例如，如果CHAR(5)从逻辑记录的末尾掉落，或者如果它包含所有空白并且空白修剪生效，则它将具有零长度。如果字段的开头和终结符之间没有字符，则分隔字段的长度为零。

此外，对于字符字段， NULLIF通常比DEFAULTIF更快（字符字段的默认值为NULL）。

10.8跨不同平台加载数据

当在一个平台上创建的数据文件要加载到不同的平台上时，必须以目标系统可以读取的形式写入数据。

例如，如果源系统具有使用16个字节的本机浮点表示，并且目标系统的浮点数为12个字节，则目标系统无法直接读取源系统上生成的数据。

最佳解决方案是通过Oracle Net数据库链接加载数据，利用数据类型的自动转换。这是推荐的方法，只要可行，并且意味着必须在源系统上运行SQL * Loader。

跨平台负载的问题通常发生在本机数据类型中。在某些情况下，可以通过用零填充来延长字段来避免问题，或者只读取字段的一部分来缩短字段（例如，当要在使用的系统上读取8字节整数时） 4字节整数，或反向）。但请注意，不兼容的数据类型实现可能会阻止这种情况。

如果您不能使用Oracle Net数据库链接并且必须由目标系统上运行的SQL * Loader访问数据文件，则建议仅使用可移植SQL * Loader数据类型（例如， CHAR ， DATE ， VARCHARC ，和数字EXTERNAL ）。使用这些数据类型编写的数据文件可能比使用本机数据类型编写的数据文件长。它们可能需要更多时间来加载，但它们更容易跨平台传输。

10.9字节顺序

SQL * Loader可以从在系统上创建的数据文件加载数据，该系统的字节顺序与运行SQL * Loader的系统上的字节顺序不同，即使数据文件包含某些不可移植的数据类型也是如此。

注意：

仅当您计划在系统上创建输入数据时，本节中的信息才适用，该系统具有与将运行SQL * Loader的系统不同的字节排序方案。否则，您可以跳过此部分。

默认情况下，SQL * Loader使用运行它的系统的字节顺序作为所有数据文件的字节顺序。例如，在Sun Solaris系统上，SQL * Loader使用big-endian字节顺序。在Intel或兼容Intel的PC上，SQL * Loader使用little-endian字节顺序。

字节顺序会影响写入数据时的结果，并且一次读取偶数个字节（通常为2个字节，4个字节或8个字节）。以下是一些例子：

  • 2字节整数值1在big-endian系统上写为0x0001，在little-endian系统上写为0x0100。

  • 4字节整数66051在big-endian系统上写为0x00010203，在little-endian系统上写为0x03020100。

如果字节数据被写入并读取为2字节实体，则字节顺序也会影响UTF16字符集中的字符数据。例如，字符'a'（ASCII中的0x61）在大端系统上以UTF16写为0x0061，但在小端系统上写为0x6100。

除UTF16外，所有Oracle支持的字符集一次写入一个字节。因此，即使对于UTF8等多字节字符集，也可以在所有系统上以相同的方式写入和读取字符，而不管系统的字节顺序如何。因此，UTF16字符集中的数据是不可移植的，因为它依赖于字节顺序。所有其他Oracle支持的字符集中的数据都是可移植的。

如果包含依赖于字节顺序的数据的数据文件是在与运行SQL * Loader的系统具有不同字节顺序的系统上创建的，则数据文件中的字节顺序只是一个问题。如果SQL * Loader知道数据的字节顺序，那么它会根据需要交换字节，以确保数据在目标数据库中正确加载。字节交换意味着将big-endian格式的数据转换为little-endian格式，反之亦然。

要指示数据到SQL * Loader的字节顺序，可以使用BYTEORDER参数，也可以在文件中放置字节顺序标记（BOM）。如果您不使用这些技术之一，则SQL * Loader将无法将数据正确加载到数据文件中。

  • 主数据文件的BYTEORDER规范也用作LOBFILE和SDF的默认值。要覆盖此缺省值， BYTEORDER使用LOBFILE或SDF规范指定BYTEORDER 。

  • BYTEORDER参数不适用于控制文件本身中包含的数据。

  • BYTEORDER参数适用于以下内容：

    • 二进制INTEGER和SMALLINT数据

    • 变长字段中的二进制长度（即VARCHAR ， VARGRAPHIC ， VARRAW和LONG VARRAW数据类型）

    • UTF16字符集中数据文件的字符数据

    • FLOAT和DOUBLE数据类型，如果编写数据的系统具有与SQL * Loader运行的系统上的兼容浮点表示

  • BYTEORDER参数不适用于以下任何内容：

    • 原始数据类型（ RAW ， VARRAW或VARRAWC ）

    • 图形数据类型（ GRAPHIC ， VARGRAPHIC或GRAPHIC EXTERNAL ）

    • UTF16以外的任何字符集中的数据文件的字符数据

    • ZONED或（打包的） DECIMAL数据类型

10.9.2使用字节顺序标记（BOM）

本节介绍使用字节顺序标记。

使用Unicode编码（UTF-16或UTF-8）的数据文件可能在文件的前几个字节中包含字节顺序标记（BOM）。对于使用字符集UTF16的数据文件，文件前两个字节中的值{0xFE，0xFF}是指示文件包含big-endian数据的BOM。值{0xFF，0xFE}是指示文件包含little-endian数据的BOM。

如果第一个主数据文件使用UTF16字符集并且它也以BOM开头，则读取并解释该标记以确定所有主数据文件的字节顺序。SQL * Loader读取并解释BOM，跳过它，并在BOM之后立即开始处理数据。 BOM设置会覆盖第一个主数据文件的任何BYTEORDER规范。读取除第一个主数据文件以外的数据文件中的BOM，并仅用于检查字节顺序冲突。它们不会更改SQL * Loader在处理数据文件时使用的字节顺序设置。

总之，第一个主数据文件的字节顺序指示符的优先级如下：

  • 第一个主数据文件中的BOM，如果数据文件使用依赖于字节顺序的Unicode字符集（UTF16）并且存在BOM

  • BYTEORDER参数值，如果在INFILE参数之前指定

  • SQL * Loader运行的系统的字节顺序

对于使用UTF8字符集的数据文件，前3个字节中的{0xEF，0xBB，0xBF}的BOM表示该文件包含UTF8数据。它不指示数据的字节顺序，因为UTF8中的数据不依赖于字节顺序。如果SQL * Loader检测到UTF8 BOM，则它会跳过它，但不会更改用于处理数据文件的任何字节顺序设置。

SQL * Loader首先使用刚刚定义的优先顺序为第一个主数据文件建立字节顺序设置。此字节顺序设置用于所有主数据文件。如果另一个主数据文件使用字符集UTF16并且还包含BOM，则将BOM值与为第一个主数据文件建立的字节顺序设置进行比较。如果BOM值与第一个主数据文件的字节顺序设置匹配，则SQL * Loader会跳过BOM，并使用该字节顺序设置开始使用BOM后面的字节处理数据。如果BOM值与为第一个主数据文件建立的字节顺序设置不匹配，则SQL * Loader会发出错误消息并停止处理。

如果在控制文件中指定了任何LOBFILE或辅助数据文件，则SQL * Loader在准备好处理文件时为每个LOBFILE和辅助数据文件（SDF）建立字节顺序设置。LOBFILE和SDF的默认字节顺序设置是为第一个主数据文件建立的字节顺序设置。如果使用LOBFILE或SDF指定BYTEORDER参数，则会覆盖此值。在任何一种情况下，如果LOBFILE或SDF使用UTF16字符集并包含BOM，则将BOM值与文件的字节顺序设置进行比较。如果BOM值与文件的字节顺序设置匹配，则SQL * Loader会跳过BOM，并使用该字节顺序设置开始使用BOM后面的字节处理数据。如果BOM值不匹配，则SQL * Loader会发出错误消息并停止处理。

总之，LOBFILE和SDF的字节顺序指示符的优先级如下：

  • 使用LOBFILE或SDF指定的BYTEORDER参数值

  • 为第一个主数据文件建立的字节顺序设置

    注意：

    如果数据文件的字符集是unicode字符集，并且文件的前几个字节中有字节顺序标记，则不要使用SKIP参数。如果这样做，则不会读取字节顺序标记并将其解释为字节顺序标记。

可以使用SQL字符串将各种SQL运算符应用于字段数据。此字符串可以包含Oracle数据库可识别的任何SQL表达式组合，这些表达式对INSERT语句的VALUES子句有效。通常，可以使用任何返回与目标列的数据类型兼容的单个值的SQL函数。SQL字符串可以应用于简单的标量列类型，也可以应用于用户定义的复杂类型，例如列对象和集合。

SQL字符串绑定变量中的列名称和列名称必须与SQL标识符规则的解释对应于同一列。但是这两个名称不一定必须以完全相同的方式编写，如下例所示：

LOAD DATA INFILE * APPEND IN TO TABLE XXX（“Last”position（1：7）char“UPPER（：\”Last \“）”first position（8:15）char“UPPER（：first ||：FIRST ||： \“FIRST \”）“）BEGINDATA格兰特菲尔泰勒杰森

请注意以下有关前面示例的内容：

  • 如果在表创建期间，使用双引号声明列标识符，因为它包含小写和/或特殊大小写字母（如上面名为"Last"的列），那么绑定变量中的列名必须与CREATE TABLE语句中使用的列名。

  • 如果一列标识符不表创建过程中双引号（如列名宣布first以上），则因为first ， FIRST ，和"FIRST"所有决心FIRST上盖完成后，所有的这些书面形式在SQL字符串绑定变量是可以接受的。

使用SQL字符串时请注意以下事项：

  • SQL字符串的执行不被视为字段设置的一部分。而是，当执行SQL字符串时，它使用任何字段设置和NULLIF或DEFAULTIF子句的结果。因此，评估顺序如下（步骤1和2是使用WHEN_ NULLIF_和DEFAULTIF子句中描述的步骤的摘要）：

    1. 现场设置完成。

    2. 应用任何NULLIF或DEFAULTIF子句（这可能会更改具有此类子句的字段的字段设置结果）。当NULLIF和DEFAULTIF子句与SQL表达式一起使用时，它们会影响字段设置结果，而不会影响最终的列结果。

    3. 使用在完成步骤1和2之后获得的字段结果来评估任何SQL表达式。结果将分配给具有SQL表达式的相应列。（如果不存在SQL表达式，则将从步骤1和2获得的结果分配给该列。）

  • 如果控制文件指定具有关联SQL字符串的字符输入，则SQL * Loader不会尝试修改数据。这是因为SQL * Loader假定使用SQL运算符修改的字符输入数据将产生适合数据库插入的结果。

  • SQL字符串必须出现在给定列的任何其他规范之后。

  • SQL字符串必须用双引号括起来。

  • 若要将列名称括在SQL字符串中的引号中，必须使用转义字符。

    在前面的示例中， Last用双引号括起来以保留混合大小写，而双引号则需要使用反斜杠（转义）字符。

  • 如果SQL字符串包含引用列对象属性的列名，则必须在绑定变量中使用完整的对象属性名称。全名中的每个属性名称都是单个标识符。每个标识符都受SQL标识符引用规则的约束，与全名中的其他标识符无关。例如，假设您有一个名为CHILD的列对象，其属性名称为"HEIGHT_%TILE" 。（请注意，属性名称是双引号。）要在绑定变量中使用完整对象属性名称，以下任何一种格式都可以使用：

    • ：儿童\ “HEIGHT_％TILE \”

    • ：孩子\ “HEIGHT_％TILE \”

    包含全名（ :\"CHILD.HEIGHT_%TILE\" ）会生成一条警告消息，指出绑定变量中使用的对象属性名称的引用规则已更改。警告只是建议正确写入绑定变量;它不会导致负载中止。引用规则已更改，因为将全名括在引号中会导致SQL将名称解释为一个标识符，而不是由多个标识符组成的完整列对象属性名称。

  • SQL字符串在任何NULLIF或DEFAULTIF子句之后但在日期掩码之前计算。

  • 如果Oracle数据库无法识别该字符串，则加载将错误终止。如果识别出字符串，但导致数据库错误，则拒绝导致该错误的行。

  • 在字段规范中使用EXPRESSION参数时，需要SQL字符串。

  • SQL字符串不能引用使用OID ， SID ， REF或BFILE加载的字段。此外，它不能引用填充字段或使用SQL字符串的其他字段。

  • 在直接路径模式下，SQL字符串不能引用VARRAY ，嵌套表或LOB列。这还包括VARRAY ，嵌套表或LOB列，它是列对象的属性。

  • SQL字符串不能用于RECNUM ， SEQUENCE ， CONSTANT或SYSDATE字段。

  • SQL字符串不能用于LOB， BFILE ， XML列或作为集合元素的文件。

  • 在直接路径模式下，在评估SQL字符串中的表达式后返回的最终结果必须是标量数据类型。也就是说，在执行直接路径加载时，表达式可能不返回对象或集合数据类型。

  • 引用字段
    要引用记录中的字段，请在字段名称前面加冒号（:)。

10.14.1引用字段

要引用记录中的字段，请在字段名称前面加冒号（:)。

来自当前记录的字段值被替换。SQL字符串中以冒号（:)开头的字段名称也称为绑定变量。请注意，用单引号括起来的绑定变量被视为文本文字， 而不是绑定变量。

以下示例说明了如何对当前字段和控制文件中的其他字段进行引用。它还说明了如何将单个引号中的绑定变量括起来将它们视为文本文字。请务必阅读本示例后面的注释，以帮助您完全理解其说明的概念。

LOAD DATA INFILE * APPEND INTO YYY（field1 POSITION（1：6）CHAR“LOWER（：field1）”field2 CHAR TERMINATED BY'，'NULLIF（（1）='a'）DEFAULTIF（（1）='b' ）“RTRIM（：field2）”，field3 CHAR（7）“TRANSLATE（：field3，'：field1'，'：1'）”，field4 COLUMN OBJECT（attr1 CHAR（3）NULLIF field4.attr2 ='ZZ'“ UPPER（：field4.attr3）“，attr2 CHAR（2），attr3 CHAR（3）”：field4.attr1 + 1“），field5 EXPRESSION”MYFUNC（：FIELD4，SYSDATE）“）BEGINDATA ABCDEF1234511，：field1500YYabc abcDEF67890，： field2600ZZghl

关于此示例的说明：

  • 在以下行中， :field1 未包含在单引号中，因此被解释为绑定变量：

    field1 POSITION（1：6）CHAR“LOWER（：field1）”

  • 在下面的行， ':field1'和':1' 括在单引号，并且因此视为文本文字和传递不变的TRANSLATE功能：

    field3 CHAR（7）“TRANSLATE（：field3，'：field1'，'：1'）”

    有关在带引号的字符串中使用引号的详细信息，请参阅指定文件名和对象名 。

  • 对于读取的每个输入记录，绑定变量引用的字段的值将替换绑定变量。例如，第一个记录中的值ABCDEF映射到第一个字段:field1 。然后将该值作为参数传递给LOWER函数。

  • SQL字符串中的绑定变量不需要引用当前字段。在前面的示例中， field4.attr1字段的SQL字符串中的bind变量引用field4.attr3字段。field4.attr1字段仍然映射到输入记录中的值500和NULL（因为NULLIF field4.attr2='ZZ'子句对于第二个记录为TRUE ），但存储在其相应列中的最终值是ABC和GHL。

    field4.attr3字段映射到输入记录中的值ABC和GHL，但存储在其对应列中的最终值为500 + 1 = 501和NULL，因为SQL表达式引用field4.attr1 。（将1添加到NULL字段仍会导致NULL字段。）

  • field5字段未映射到输入记录中的任何字段。存储在目标列中的值是执行MYFUNC PL / SQL函数的结果，该函数有两个参数。使用EXPRESSION参数需要使用SQL字符串来计算列的最终值，因为没有输入数据映射到该字段。

10.14.4将SQL字符串与日期掩码一起使用

当SQL字符串与日期掩码一起使用时，将在SQL字符串之后计算日期掩码。

考虑如下指定的字段：

field1 DATE“dd-mon-yy”“RTRIM（：field1）”

SQL * Loader在内部生成并插入以下内容：

TO_DATE（RTRIM（<field1_value>），'dd-mon-yyyy'）

请注意，将DATE字段数据类型与SQL字符串一起使用时，需要使用日期掩码。这是因为SQL * Loader假定它在DATE参数之后找到的第一个引用字符串是日期掩码。例如，以下字段规范将导致错误（ORA-01821：无法识别日期格式）：

field1 DATE“RTRIM（TO_DATE（：field1，'dd-mon-yyyy'））”

在这种情况下，一个简单的解决方法是使用CHAR数据类型。

10.15.7生成多个表的序列号

因为为每个逻辑输入记录生成唯一的序列号，而不是为每个表插入生成，所以在将数据插入多个表时可以使用相同的序列号。

这通常很有用。

但是，有时您可能希望为每个INTO TABLE子句生成不同的序列号。例如，您的数据格式可能在每个输入记录中定义三个逻辑记录。在这种情况下，您可以使用三个INTO TABLE子句，每个子句将记录的不同部分插入到同一个表中。当您使用SEQUENCE(MAX) ，SQL * Loader将使用每个表中的最大值，这可能导致序列号不一致。

要为这些记录生成序列号，必须为三个插入中的每一个生成唯一编号。使用每个记录的表插入数作为序列增量，并使用连续的数字开始每个插入的序列号。

10.15.7.1示例：为每个插入生成不同的序列号

假设您要将以下部门名称加载到dept表中。每个输入记录包含三个部门名称，您希望自动生成部门编号。 

会计人员制造航运采购维护...

您可以使用以下控制文件条目生成唯一的部门编号：

INTO TABLE dept（deptno SEQUENCE（1,3），dname POSITION（1:14）CHAR）INTO TABLE dept（deptno SEQUENCE（2,3），dname POSITION（16:29）CHAR）INTO TABLE dept（deptno SEQUENCE（3） ，3），dname POSITION（31:44）CHAR）

第一个INTO TABLE子句生成部门编号1，第二个编号2和第三个编号3。它们都使用3作为序列增量（每个记录中的部门名称数）。此控制文件将Accounting作为部门号1，Personnel作为2，Manufacturing作为3加载。

然后，序列号会针对下一条记录递增，因此装运为4，采购为5，依此类推。
