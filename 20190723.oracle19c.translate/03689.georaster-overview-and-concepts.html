1 GeoRaster概述和概念

GeoRaster是Oracle Spatial and Graph的一项功能，可用于存储，索引，查询，分析和交付栅格图像和网格化数据及其关联的元数据。

GeoRaster提供Oracle空间数据类型和对象关系模式。您可以使用这些数据类型和模式对象来存储多维网格图层和数字图像，这些图像可以参考地球表面或本地坐标系中的位置。如果数据是地理参考的，您可以在地球上找到图像中单元格的位置;或者在地球上给出一个位置，您可以在与该位置相关联的图像中找到该单元格。

GeoRaster可以与任何捕获或生成图像的技术（如遥感，摄影测量和专题制图）一起使用。它可用于各种应用领域，包括基于位置的服务，地理图像存档，环境监测和评估，地质工程和勘探，自然资源管理，国防，应急响应，电信，交通，城市规划和国土安全。

注意：

要使用GeoRaster，您必须了解Oracle Spatial and Graph的主要概念，数据类型，技术，操作符，过程和函数，这些内容已在Oracle Spatial and Graph Developer's Guide中进行了介绍 。

您还应熟悉栅格和图像概念和术语，捕获或创建栅格数据的技术以及处理栅格数据的技术。

GeoRaster使用并依赖于Oracle数据库中包含的几个组件，包括Java虚拟机（JVM）和Oracle XML DB。

注意：

默认情况下最初安装Oracle Spatial and Graph后，将禁用GeoRaster功能，并且必须为将使用GeoRaster的每个架构启用GeoRaster功能。有关信息和说明，请参阅在架构级别启用GeoRaster 。

您还必须确保正确安装了Oracle XML DB存储库，并且COMPATIBILITY数据库初始化参数的值为10.0或更高。有关更多信息，请参阅Oracle Spatial and Graph Developer's Guide中有关安装，兼容性和升级问题的附录。

1.1矢量和栅格数据

地理要素可以用矢量或栅格格式表示，也可以两者都表示。

对于矢量数据，点由它们的显式x，y，z坐标表示，线是点的串，区域表示为边界为线的多边形。这种矢量格式可用于精确记录空间物体的位置和形状。使用栅格数据，您可以通过为覆盖对象的单元格指定值来表示空间对象，并且可以将单元格表示为数组。这种栅格格式的精度低于矢量格式，但它适用于多种类型的空间分析。

在栅格地理信息系统（GIS）世界中，通常会调用这种栅格数据网格数据。在图像处理系统中，栅格数据表示通常称为图像而不是网格。尽管网格和图像之间存在任何差异，但两种形式的空间信息通常表示为矩阵结构（即，单元阵列），并且每个单元通常在空间中定期对齐。

1.2.1遥感

遥感通过未物理连接到区域或物体的设备获得关于区域或物体的信息。例如，传感器可能位于卫星，气球，飞机，船或地面站。传感器设备可以是各种设备中的任何一种，包括框架摄像机，推扫式（扫描）成像仪，合成孔径雷达（SAR），水文声纳或纸或胶片扫描仪。遥感应用包括环境评估和监测，全球变化检测和监测以及自然资源调查。

通过遥感收集的数据通常被称为地理成像 。波长，波段数和其他因素决定了地理图像的辐射特性。地理影像可以是单波段，多波段或高光谱，所有这些都可以由GeoRaster管理。这些地理图像可以覆盖地球的任何区域（特别是对于卫星感知的图像）。时间分辨率可能很高，例如气象卫星，可以更容易地检测到变化。对于遥感应用，各种类型的分辨率（时间，空间，光谱和辐射）通常很重要。

父主题： 栅格数据源

1.2.2摄影测量

摄影测量从照片上的测量得出度量信息。大多数摄影测量应用使用通过卫星遥感收集的空中照片或高分辨率图像。在传统的摄影测量中，主要数据包括诸如黑白照片，彩色照片和立体照片对之类的图像。

摄影测量法严格确定图像与物体之间的几何关系，因为它在成像事件时存在，并使您能够从图像中获取有关物体的信息。图像和对象之间的关系可以通过几种方式建立，可以分为两类：模拟（使用光学，机械和电子组件）或分析（其中建模是数学的，处理是数字的）。模拟解决方案越来越多地被分析/数字解决方案所取代，这些解决方案也被称为软拷贝摄影测量 。

来自软拷贝摄影测量系统的主要产品可以包括数字高程模型（DEM）和正射影像。GeoRaster可以管理所有这些栅格数据及其地理配准信息。

父主题： 栅格数据源

1.2.3地理信息系统

地理信息系统（GIS）捕获，存储和处理地理参考信息。传统上，GIS软件基于矢量或基于光栅;但是，通过GeoRaster功能，Oracle Spatial and Graph可处理栅格和矢量数据。

基于栅格的GIS系统通常处理经过地理处理的网格化数据。网格数据可以是离散的或连续的。离散数据，例如政治细分，土地使用和覆盖，公交线路和油井，通常存储为整数网格。连续数据，例如高程，纵横比，污染浓度，环境噪声水平和风速，通常存储为浮点网格。GeoRaster可以存储所有这些数据。

离散网格图层的属性存储在称为值属性表（VAT）的关系表中。VAT包含GIS供应商指定的列，也可能包含用户定义的列。VAT可以作为普通表存储在Oracle数据库中。VAT名称可以在相应的GeoRaster对象中注册，以便栅格GIS应用程序可以使用该表。

父主题： 栅格数据源

1.2.4制图

制图是创建地图的科学，地图是三维地球（或使用局部坐标系的非地球空间）的二维表示。今天，地图被数字化或扫描成数字形式，地图制作基本上是自动化的。可以快速查询，分析和更新存储在计算机上的地图。

有许多类型的地图，对应于各种用途或目的。地图类型的示例包括基础（背景），主题，浮雕（三维），方面，地籍（土地使用）和插图。地图通常包含多个注释元素以帮助解释地图，例如比例尺，图例，符号（例如北箭头）和标签（城市，河流等的名称）。

地图可以以栅格格式存储（因此可以由GeoRaster管理），矢量格式或混合格式。

1.2.5数字图像处理

数字图像处理用于处理标准图像格式的光栅数据，例如TIFF，GIF，JFIF（JPEG），以及许多地理图像格式，例如NITF，GeoTIFF，ERDAS IMG和PCI PIX。图像处理技术广泛用于遥感和摄影测量应用。根据需要使用这些技术来增强，校正和恢复图像以便于解释;纠正可能发生的任何模糊，扭曲或其他退化;并自动分类地理对象并识别目标。GeoRaster可以加载和管理源图像，中间图像和结果图像。

父主题： 栅格数据源

1.2.6地质学，地球物理学和地球化学

地质学，地球物理学和地球化学都使用数字数据并生成一些可由GeoRaster管理的数字栅格地图。

  • 在地质学中，数据包括区域地质图，地层图和岩石幻灯片图。在地质勘探和石油地质学中，广泛使用计算机化地质层模拟，合成矿物预测和三维油田表征，所有这些都涉及栅格数据。

  • 在地球物理学中，保存有关重力，磁场，地震波传输和其他主题的数据以及地理参考信息。

  • 在地球化学中，可以分析和测量多种化学元素的含量。三角网不规则网络（TIN）技术通常用于生成栅格地图以供进一步分析，并且图像处理被广泛使用。

1.3 GeoRaster数据模型

栅格数据可以包含以下部分或全部元素。

  • 细胞或像素

  • 空间域（足迹）

  • 空间，时间和频段参考信息

  • 细胞属性

  • 元数据

  • 处理数据和地图支持数据

GeoRaster定义了一个基于组件，逻辑分层和多维的通用栅格数据模型。栅格中的核心数据是多维数组或栅格单元矩阵。每个单元格是矩阵的一个元素，其值称为单元格值，在单元格的中心进行采样。如果GeoRaster对象表示图像，则单元格也可以称为像素，该像素只有一个值。（在GeoRaster中，术语单元格和像素可以互换。）矩阵具有多个维度，单元深度和每个维度的大小。单元格深度是每个单元格的值的数据大小。单元格深度定义了所有单元格值的范围，它适用于每个单个单元格，而不是单元格数组。可以阻止此核心栅格数据集以实现最佳存储和检索。

数据模型具有逻辑分层结构。核心数据由一个或多个逻辑层组成。例如，对于多通道遥感图像，这些图层用于模拟图像的通道。（ Bands_ Layers_和Metadata中解释了波段和层。）在当前版本中，每个图层都是一个二维矩阵单元格，由行维度和列维度组成。

GeoRaster数据具有元数据和属性，GeoRaster数据的每一层都可以拥有自己的元数据和属性。在GeoRaster数据模型中，除核心单元矩阵之外的所有数据都是GeoRaster元数据。GeoRaster元数据进一步划分为不同的组件（因此称为基于组件），其中包含以下类型的信息：

  • 对象信息

  • 栅格信息

  • 空间参考系统信息

  • 日期和时间（时间参考系统）信息

  • 频带参考系统信息

  • 每层的图层信息

基于此数据模型，GeoRaster对象由GeoRaster元数据XML模式（在GeoRaster元数据XML模式中描述）描述，该模式用于组织元数据。某些架构组件和子组件是必需的，而其他组件和子组件是可选的。如果您开发GeoRaster加载器，导出器或其他应用程序，则必须了解此XML架构。当前版本存在对元数据的一些限制，这些在SDO_GEOR.validateGeoRaster函数的使用说明（在SDO_GEOR包参考中记录 ）中进行了描述，该函数检查GeoRaster对象的元数据的有效性。

GeoRaster数据类型和相关结构中描述的GeoRaster对象数据类型基于GeoRaster数据模型。

在此数据模型中，需要考虑两种不同类型的坐标：栅格矩阵中每个像素（单元格）的坐标以及它们所代表的地球上的坐标。因此，定义了两种类型的坐标系或空间：单元坐标系和模型坐标系。

单元坐标系 （也称为栅格空间 ）用于描述栅格矩阵中的单元格及其间距，其尺寸（按此顺序）为行，列和带。模型坐标系 （也称为地面坐标系或模型空间 ）用于描述地球上的点或与Oracle SRID值关联的任何其他坐标系。模型坐标系的空间维度（按此顺序）X和Y，分别对应于单元坐标系中的列和行维度。逻辑层对应于单元空间中的带维度。

  • 在基于左上的单元格空间中，只要I <= x <I + 1.0且J <= y <J + 1.0，坐标（x，y）就映射到单元格（I，J）。

例如，子单元坐标（0.3,0.3）在两个坐标系中具有相同的整数单元坐标（0,0），而（0.3,0.6）在基于中心的单元空间中表示（0,1）但是表示（0） ，0）在基于左上的单元格空间中。这两种类型的单元坐标系由spatialReferenceInfo元数据中的modelCoordinateLocation元素定义;否则，默认类型是基于中心的。GeoRaster支持两种单元坐标系统，并且对Oracle Database 11 g有效，GeoRaster PL / SQL API支持子单元地址。 （在以前的版本中内部支持子单元地址。）

在GeoRaster中，虽然单元格空间的原点始终为（0,0），但栅格数据本身的左上角单元格在其单元格空间中可以与单元格空间原点的坐标具有不同的坐标。换句话说，左上角单元的整数（行，列）坐标不一定是（0,0）。左上角称为ULTCoordinate ，其值在元数据中注册。它基本上定义了单元格空间中数据的相对位置。如果存在波段维度，则ULTCOordinate值始终为（行，列，0）。每个单元格的坐标相对于单元格空间的原点，而不是ULTCOordinate值。单元坐标系的原点可能不完全符合ULTCOordinate值。

模型坐标系由空间维度和其他维度（如果有）组成。空间维度称为x ， y和z维度，这些维度中的值可以与大地坐标系，投影坐标系或本地坐标系相关联。其他维度包括光谱和时间维度（分别称为s维度和t维度）。GeoRaster SRS当前支持模型坐标系中的两个空间维度（X，Y）和三个空间维度（X，Y，Z）。（有关坐标系的信息，包括不同类型的坐标系，请参阅Oracle Spatial and Graph Developer's Guide 。）

GeoRaster模型坐标系由Oracle Spatial and Graph SRID定义。模型坐标与指定的SRID具有相同的单位，并且应该在模型坐标系定义的值范围内。例如，如果GeoRaster对象被地理参考到大地坐标系（例如4326（EPSG WGS84）），则从空间参考系统（SRS）派生的模型坐标的单位必须是十进制度，并且值应该在以下范围内：经度为-180.0至+180.0，纬度为-90.0至+90.0。

细胞坐标和模型坐标之间的关系由GeoRaster参考系统（映射方案）建模。定义了以下GeoRaster参考系统：

  • 空间参考系统 ，也称为GeoRaster SRS ，它将单元坐标（行，列，垂直）映射到模型坐标（X，Y，Z）。将空间参考系统与GeoRaster数据一起使用称为地理数据引用。（地理配准中讨论了地理配准 。）

  • 时间参考系统 ，也称为GeoRaster TRS ，它将细胞坐标（时间）映射到模型坐标（T）。

  • 带参考系统 ，也称为GeoRaster BRS ，它将细胞坐标（带）映射到模型坐标（S，用于光谱）。

目前，这些参考系统中的每一个至少部分地在GeoRaster XML模式中定义。但是，对于当前版本，仅支持空间参照系。这意味着只能映射（行，列）和（X，Y）或（X，Y，Z）坐标之间的关系。如果模型坐标系是大地测量，则（X，Y）表示（经度，纬度）。然而，可以使用时间和频带参考系统来存储有用的时间和光谱信息，例如光谱分辨率和何时收集光栅数据。

其他元数据存储在GeoRaster XML元数据中的<layerInfo>元素中，如Bands_ Layers_和Metadata中所述 。

1.4 GeoRaster物理存储

GeoRaster优化元数据和数据的物理存储。

如GeoRaster数据模型中所述 ，GeoRaster数据由多维单元矩阵和GeoRaster元数据组成。大多数元数据使用Oracle XMLType数据类型存储为XML文档。元数据被根据元数据的GeoRaster XML模式，这是在描述中定义的GeoRaster元数据的XML Schema 。GeoRaster对象的空间范围（覆盖区域）是元数据的一部分，但它作为GeoRaster对象的属性单独存储。此方法允许GeoRaster利用空间几何类型和相关功能，例如在GeoRaster对象上使用R树索引。空间范围在spatialExtent属性中描述。

使用CLOB存储选项或二进制XML存储选项存储GeoRaster元数据。GeoRaster元数据的二进制XML存储选项是默认选项，可节省磁盘空间并提高性能。您可以在创建GeoRaster表时指定或更改存储选项。

多维矩阵的单元被阻塞成小的子集，用于大规模的GeoRaster对象存储和最佳的检索和处理。每个块作为二进制大对象（BLOB）存储在表中，并且几何对象（类型为SDO_GEOMETRY）用于定义块的精确范围。该表的每一行仅存储一个块以及与该块相关的阻塞信息。（这种阻止方案也适用于金字塔。）

尺寸大小（沿行，列和带尺寸）可能无法按其各自的块大小平均分配。GeoRaster将填充添加到边界块，这些边界块没有足够的原始单元格来完全填充。边界块是沿每个维度的正方向的端块。填充单元具有与其他单元相同的单元深度，并且具有等于零的值。填充使每个块具有相同的BLOB大小。填充主要适用于行和列块，但对于多波段和高光谱图像，填充也可应用于波段维度。例如，假设以下规范：按行交织的频带，阻塞为（64,64,3）和8个频带，每个频带有64行和64列。在这种情况下：

  1. 频带0,1和2在第一个块中逐行存储。

  2. 频带3,4和5在第二块中逐行交错存储。

  3. 第三个块按以下顺序保存：带6的第1行，带7的第1行，填充的64列值，带6的第2行，带7的第2行，填充的64列值，等等打开，直到存储所有64行。

但是，如果金字塔等级的行和列尺寸分别小于或等于行块大小和列块大小的一半，则不填充顶层金字塔。有关金字塔物理存储的信息，请参阅金字塔。

每个GeoRaster块具有相同的大小。块的尺寸大小不需要是2的幂。它们可以是随机整数值。可以根据GeoRaster对象的尺寸大小自动优化块大小，以便每个GeoRaster对象仅使用最小填充空间。有关更多信息，请参阅存储参数中的 表1-1 。

栅格块（BLOB）包含栅格单元值的二进制表示。具体而言，浮点单元值在支持的平台上以IEEE 754标准格式表示。如果单元深度大于8位，则GeoRaster单元数据以栅格块中的big-endian格式存储。如果单元深度小于8位，则光栅块中的每个字节包含两个或更多个单元，因此一个字节的位完全用单元数据填充。单元格始终从左到右填充到字节中。例如，如果单元深度为4位，则一个字节包含两个单元：字节的前四位包含单元的值，后四位包含其后续单元的值，这由交错确定类型。

基于此物理存储模型，提供了两种对象类型：用于栅格数据集和相关元数据的SDO_GEORASTER，以及用于栅格图像中每个块的SDO_RASTER。

  • SDO_GEORASTER对象包含空间范围几何（覆盖范围或覆盖范围）和相关元数据。包含此对象类型的一个或多个列的表称为GeoRaster表 。

  • SDO_RASTER对象包含有关GeoRaster对象的块（tile）的信息，它使用BLOB对象存储块的栅格单元数据。此对象类型的对象表或包含与此对象类型的属性相同的列的关系表称为栅格数据表 （ RDT ）。

SDO_GEORASTER对象存储并引用图像或栅格数据集。SDO_RASTER对象是GeoRaster的内部对象。SDO_GEORASTER对象完全封装了栅格数据集的元数据和栅格单元数据，即SDO_RASTER对象的集合。SDO_GEORASTER对象与其SDO_RASTER对象之间的关系由GeoRaster自动维护。GeoRaster函数和过程的所有接口仅处理SDO_GEORASTER对象; SDO_GEORASTER对象的SDO_RASTER对象在内部自动处理。SDO_GEORASTER对象是用户构建和管理GeoRaster数据库的主要界面;您只需要使用SDO_RASTER对象来创建栅格数据表（RDT）。

每个SDO_GEORASTER对象都有一对属性（ rasterDataTable ， rasterID ），用于唯一标识RDT以及RDT中用于存储GeoRaster对象的栅格单元数据的行。

1.4.2栅格数据表

栅格数据表必须是SDO_RASTER类型的对象表，或具有以下列定义的关系表：

rasterID NUMBER，pyramidLevel NUMBER，bandBlockNumber NUMBER，rowBlockNumber NUMBER，columnBlockNumber NUMBER，blockMBR SDO_GEOMETRY，rasterBlock BLOB

栅格数据表（无论是对象表还是关系表）必须在列上定义主键(rasterID, pyramidLevel, bandBlockNumber, rowBlockNumber, columnBlockNumber) 。

每个栅格数据表名称必须或等效于有效的非加引号标识符，并且它将以全部大写字符存储在GeoRaster元数据视图和SDO_GEORASTER对象中，而不包含任何模式前缀。（每个GeoRaster列名必须或等效于有效的非引号标识符，并以全部大写字符存储在GeoRaster元数据视图中。）每个栅格数据表名称在数据库中也必须是唯一的。要解决栅格数据表名称中的任何重复，可以使用SDO_GEOR_ADMIN.maintainSysdataEntries函数。

通过创建栅格数据表，您可以控制RDT的放置和存储特性（例如，如果要对表进行分区以获得更好的性能）。对于大型GeoRaster对象，请考虑将其栅格数据放在单独的栅格数据表中，并按照金字塔等级或块编号或两者划分栅格数据表;但是，始终考虑为一定数量的较小GeoRaster对象共享RDT，以避免创建过多的RDT。不要使用SYSTEM表空间来存储GeoRaster表和栅格数据表。而是为GeoRaster表创建单独的本地管理（默认）表空间。

切勿直接在栅格数据表中插入或删除任何行。使用栅格数据创建GeoRaster对象或从GeoRaster表中删除时，将自动插入或删除相应RDT中的行。

在为栅格数据选择块大小时，请考虑以下事项：

  • 栅格块的最大长度为4 GB;因此，请勿指定大于4 GB的块大小。

  • 在计算栅格块的所需大小时，请考虑GeoRaster对象的cellDepth值。

  • 选择合适的块大小是栅格块大小和GeoRaster对象所需块数之间的权衡。对于大尺寸的栅格数据，Oracle建议行尺寸和列尺寸至少为512 x 512。阻塞大小值导致栅格块小于或接近4 KB（例如64乘64）通常是一个糟糕的选择，因为4 KB是存储Oracle BLOB外线的阈值。

1.4.4清空栅格块

GeoRaster支持空光栅块，以节省大型马赛克对象的存储空间，并提高光栅处理速度。当没有可用于大型GeoRaster对象的特定栅格块的栅格数据时，将使用空栅格块。这种GeoRaster数据是一种特殊的稀疏数据类型。栅格数据表中仍有一个条目用于每个空栅格块，但BLOB的长度为零（表示空）。

当GeoRaster操作（例如， SDO_GEOR.changeCellValue ， SDO_GEOR.changeFormatCopy ， SDO_GEOR.generatePyramid ， SDO_GEOR.getRasterData ， SDO_GEOR.getRasterSubset ， SDO_GEOR.mergeLayers ， SDO_GEOR.mosaic ， SDO_GEOR.scaleCopy ， SDO_GEOR.subset或SDO_GEOR.updateRaster ）时应用于具有空栅格块的源GeoRaster对象，可能会导致结果栅格块为空或部分为空。

如果结果中的所有单元格都是从空源栅格块派生的，则生成的栅格块为空。如果只有一些单元格是从空源栅格块派生的，则生成的栅格块部分为空。从空源栅格块派生的部分空结果栅格块中的任何单元格要么设置为某些背景值（如bgValues参数中所指定），要么设置为0（如果未指定bgValues参数）。完成此操作后，部分空的栅格块就像普通的非空栅格块一样;并且在操作完成后，生成的GeoRaster对象中的每个栅格块都为空或非空。

由于填充部分空的栅格块会永久更改栅格数据，因此在操作GeoRaster对象时应仔细选择一致的背景值。存储在GeoRaster元数据中的NODATA值（如果存在）是背景值的不错选择，但您也可以选择其他背景值，只要它们一致使用即可。

如果GeoRaster对象具有空栅格块，则其金字塔数据可能根本不包含任何空栅格块，因为在SDO_GEOR.generatePyramid操作期间，部分空的栅格块将填充背景值或0。当您调用此函数生成金字塔时，请谨慎选择一致的背景值，如本节所述。

位图掩码（请参见位图掩码 ）也可以具有空栅格块，缺少的单元格值指示0。如果需要填充，则缺少的单元格始终填充值0。

1.5波段，图层和元数据

在GeoRaster中， band和layer是不同的概念。

波段是多维栅格数据集的物理维度;也就是说，它是细胞空间中的一个纵坐标。例如，单元格空间可能具有纵坐标行，列和带。波段编号从0到n -1，其中n是最高层编号。Layer是GeoRaster数据模型中的逻辑概念。图层映射到波段。通常，一个层对应于一个带，并且它由大小为rowDimensionSize和columnDimensionSize的二维矩阵columnDimensionSize 。层数从1到n ;也就是说， layerNumber = bandNumber + 1 。

GeoRaster对象可以包含多个波段，也可以称为多个波段。例如，来自遥感设备的电磁波数据被分组为一定数量的信道，其中可能信道的数量取决于传感设备的能力。多光谱图像包含多个通道，而高光谱图像包含非常大量（例如，50个或更多）通道。通道全部映射到与层相关联的GeoRaster波段。

在栅格GIS应用程序中，数据集可以包含多个栅格图层，每个图层都称为a 主题 。例如，栅格可以具有人口密度层，其中不同的单元格值用于描绘邻域或县，这取决于他们每平方英里或公里的平均居民数量。其他主题的例子可能是平均收入水平，土地使用（农业，住宅，工业等）和海拔高度。栅格GIS主题可以存储在不同的GeoRaster对象或一个GeoRaster对象中，并且每个主题都被建模为一个层。栅格主题和多光谱图像通道也可以作为不同的图层一起存储在一个GeoRaster对象中，只要它们具有相同的尺寸即可。

1.6地理配准

GeoRaster空间参考系统（SRS）是GeoRaster对象的元数据组件，包括与地理配准相关的信息。地理配准建立了GeoRaster数据的单元坐标与真实世界地面坐标（或某些本地坐标）之间的关系。地理配准将地面坐标分配给单元格坐标，将单元格坐标分配给地面坐标。

在GeoRaster中，地理参考不同于地理修正，修正或正射校正。在后三个过程中，通常对栅格数据执行单元重采样，并且生成的GeoRaster数据可能具有不同的模型坐标系和尺寸大小。地理配准建立了单元坐标和真实世界坐标或某些局部坐标之间的关系。地理参考可以通过提供适当的数学公式，足够的地面控制点（GCP）坐标或来自遥感系统的严格模型数据来完成。地理配准不会更改GeoRaster单元格数据或其他元数据，除非是为了便于单元格坐标系和模型坐标系之间的坐标转换。

GeoRaster支持功能拟合模型（在功能拟合地理配准模型中解释）和存储的功能模型（在地面控制点（GCP）地理参考模型中解释 ）用于地理配准。不支持严格的模型。当使用函数拟合模型对GeoRaster对象进行地理配isReferenced ，SRS元数据中的isReferenced值将为TRUE ;否则，它应该是FALSE 。

可以使用水平坐标进行校正 ，以便可以将GeoRaster数据集的单元映射到投影图坐标系。在整流之后，每个单元在地图单元中被规则地定尺寸并且与模型坐标系对齐，即，具有东西方尺寸和南北方向尺寸。如果在整流中使用高程数据（DEM），则调用它 正射校正 ，一种纠正地形位移的特殊形式的整治。如果使用函数拟合模型对GeoRaster对象进行纠正和地理配isRectified ，则其元数据中的isRectified值将为TRUE ;否则，它应该是FALSE 。如果对GeoRaster对象进行了正射校正并使用函数拟合模型进行地理配isOrthoRectified ，则其元数据中的isOrthoRectified值将为TRUE ;否则，它应该是FALSE 。

  • 这些点不应该在一条线上，因此结果可以是稳定的。

要使用GCP进行地理配准，还必须选择几何模型，即GeoRaster对象的单元格空间与参考坐标系之间的关系应如何进行数学建模。在GeoRaster中，GCP地理配准支持以下几何模型：仿射（默认模型），二次多项式，立方多项式，DLT，二次有理和RPC。仿射，二次多项式和三次多项式是分别具有多项式阶数1,2和3的二维多项式模型; DLT，Quadratic Rational和RPC是分别具有多项式阶数1,2和3的三维有理多项式模型。所有多项式都具有多项式类型pType=1 。（有关地理配准模型类型的更多信息，请参阅功能配件地理配准模型。）

在使用GCP的地理配准中，在多项式或有理多项式模型的公式中使用GCP的单元和模型坐标，然后形成线性方程系统。公式中没有使用重量，也就是说，所有点的重量都相等1.0。线性方程系统通过最小二乘法求解，该方法生成最适合给定控制点的模型的系数。解决方案计算中只涉及控制点类型的GCP;具有Check Point类型的GCP用于检查求解模型的定位精度。基于解决方案中涉及的那些控制点的单元坐标的残差来评估求解精度。

不同的几何模型需要不同的模型坐标尺寸和不同的最小GCP数量。对于二维几何模型，模型坐标必须是2D（X，Y）;对于三维几何模型，模型坐标必须是3D（X，Y，Z）。几何模型所需的最小GCP数量如下：仿射：3，二次多项式：6，立方多项式：10，DLT：7，二次有理：19，和RPC：39。但是，您通常应使用超过最小数量的GCP进行地理配准。

1.6.3小区坐标和模型坐标转换

通过功能拟合地理配准模型，GeoRaster将地面坐标分配给单元坐标，将单元坐标分配给地面坐标。作为特殊情况，单元格的整数坐标（单元矩阵中单元格的数组索引）可以转换为模型坐标，该坐标标识模型空间中点的精确位置。该点或模型坐标可以是模型空间中的单元格所代表的区域的左上角或中心。

类似地，可以通过地理配准将模型坐标转换为单元坐标。然而，来自功能拟合地理配准模型的直接解的结果单元坐标大多数是浮点数。由modelCoordinateLocation元素决定的单元格空间坐标系的类型确定浮动坐标引用哪个单元格，如GeoRaster数据模型中所述 。GeoRaster支持其API的所有部分中的浮动（子单元）单元坐标和整数单元坐标。

单元坐标和模型坐标变换基于GeoRaster空间参考系统（SRS）的功能拟合模型。在使用GeoRaster SRS进行变换之前和之后，单元格的（行，列）坐标值都相对于GeoRaster单元格空间，不一定相对于栅格数据本身的左上角。ULTCoordinate可以具有与单元空间原点的坐标不同的坐标（行和列值）。也就是说，左上角的（行，列）坐标不一定是（0,0）。

任何将栅格数据的左上角定义为其自己的单元格空间的原点（0,0）的应用程序（如许多图像文件格式）都必须将从GeoRaster SRS派生的（行，列）转换为相对的如果GeoRaster ULTCoordinate（row0，column0）的值不是（0,0），则到该原点。此转换必须考虑GeoRaster ULTCoordinate，如以下公式所示：

row = row0 + m column = column0 + n

在这些公式中：

  • row =单元格相对于GeoRaster单元格空间原点的行索引。

  • column =单元格相对于GeoRaster单元格空间原点的列索引。

  • row0 = ULTCoordinate相对于GeoRaster单元格空间原点的行索引。

  • column0 = ULTCoordinate相对于GeoRaster单元格空间原点的列索引。

  • m =单元格相对于ULTCOordinate的行索引（即第m行，从第一行开始，第0行）。

  • n =单元格相对于ULTCOordinate的列索引（即第n列，第一列从0开始）。

在大多数应用中，ULTCoordinate和单元空间的原点是相同的（即，row0 = 0和column0 = 0），在这种情况下m = row且n = column。

1.8金字塔

金字塔是GeoRaster对象的子对象，表示不同大小和分辨率的栅格图像或栅格数据。

大小通常与应用程序检索和显示图像所需的时间量有关，尤其是在Web上。也就是说，图像尺寸越小，显示得越快;并且只要不需要详细的分辨率（例如，如果用户已经“大大缩小”），则较小图像的显示质量就足够了。

金字塔等级分别表示需要更少或更多存储空间的分辨率图像的减少或增加。（GeoRaster仅支持降低分辨率的金字塔。）金字塔等级为0表示原始栅格数据;也就是说，图像分辨率没有降低，所需的存储空间也没有变化。大于0（零）的值表示图像分辨率水平越来越低，存储空间要求也越来越低。

金字塔类型表示金字塔的类型 ，可以是以下值之一：

  • DECREASE意味着随着金字塔等级的增加，金字塔的大小会减小。

  • NONE表示没有与GeoRaster对象关联的金字塔。

  • 如果原始GeoRaster对象被阻止（即使被阻止为一个块），每个金字塔等级的单元格数据将以与原始级别数据相同的方式被阻止，并且每个块都存储在不同的BLOB对象中，只要实际金字塔等级图像的最大尺寸大小大于块尺寸。但是，如果生成较低分辨率的金字塔（即，如果金字塔等级的行和列维度大小分别小于或等于行块大小和列块大小的一半），则单元格数据为每个这样的金字塔等级存储在一个BLOB对象中，并且其大小与实际金字塔等级图像的大小相同。

1.9位图掩码

位图掩码是一个特殊的一位深矩形栅格网格，每个像素的值为0或1。它用于在另一个图像内定义不规则形状的区域。1位定义区域的内部，0位定义区域的外部。

位图蒙版可以附加到非空白GeoRaster对象或从中移除。非空GeoRaster对象的每个带或层也可以具有与其关联的单独位图掩码。因此，最多可以存在与非空GeoRaster对象相关联的n + 1个位图掩码，其中n是GeoRaster对象的子层的总数。也可以单独编辑或更新位图掩码。

如果位图蒙版与对象图层关联，它也将成为所有子图层的默认位图蒙版。与子图层关联的位图蒙版会覆盖与对象图层关联的默认位图蒙版。

附加到栅格图层的位图蒙版必须与图像中的任何其他栅格图层具有相同的行数和列数，并且必须精确覆盖相同的区域。它使用与GeoRaster对象本身相同的ULTCOordinate和SRS。从逻辑上讲，它不是光栅图像本身的组成部分，而是辅助信息;但是，在物理上，它存储在GeoRaster对象中。

位图蒙版的物理存储类似于GeoRaster对象的栅格数据。位图蒙版存储在关联的GeoRaster对象的栅格数据表中，具有完全相同的阻塞属性。但是，位图掩码条目的bandBlockNumber始终设置为与位图掩码关联的层编号。有关波段和图层之间关系的信息，请参阅波段，图层和元数据 。

pyramidLevel值以-99999而不是0开头，并且每个上金字塔等级增加1。金字塔是在位图掩码上与常规栅格数据上的金字塔一起构建的，位图掩码可以与关联的GeoRaster对象一起使用SDO_GEOR.scaleCopy过程进行缩放，但用于位图掩码的重采样方法始终是NN（最近邻）。当压缩或解压缩其关联的GeoRaster对象时，压缩或解压缩位图掩码，并且始终使用DEFLATE方法压缩位图掩码（无损）。位图掩码也可以是稀疏的，因此可以包含空块，缺少的单元格值指示0。

位图掩码通常由应用程序以下列方式之一或两者使用：

  • 当用作透明度蒙版时，显示应用程序可以使用位图蒙版来确定要显示的图像的哪个部分。例如，对应于位图掩模中的1位的主图像像素被成像到屏幕或打印机，但是不显示或打印对应于掩模中的0位的主图像像素。它也可以用作图像的alpha通道，因此0和1值可以映射到不同的透明度值以供显示。

  • 当在GIS应用程序中用作NODATA掩码时，位图掩码告诉应用程序将与掩码的外部（0位）对应的像素视为NODATA。为此，可以在GeoRaster元数据中将其注册为特殊类型的NODATA，如NODATA值和值范围中所述 。

多个PL / SQL子程序对位图蒙版执行操作，例如将位图蒙版附加到GeoRaster对象，替换现有位图蒙版，删除位图蒙版，检查GeoRaster对象是否具有某个位图蒙版，以及提取整个位图蒙版，它的一个子集，或它的单个单元格值。您还可以使用SDO_GEOR.mask过程在数据库中应用屏蔽操作。有关图像蒙版的详细信息，请参阅图像蒙版 。

1.10 NODATA值和值范围

NODATA值用于其值未知或无意义的单元格。

每个单独的栅格图层可以具有多个与其关联的NODATA值或NODATA值范围或两者。GeoRaster元数据模式存储每个栅格图层的NODATA信息。具体而言，与对象层关联的NODATA值和值范围适用于任何其他子层。子层的NODATA值和值范围是对象层和子层中存在的任何NODATA元数据的并集。从子图层中删除NODATA值或值范围时，无法删除对象图层中存在的任何值或值范围。

在重新采样期间可以考虑NODATA值和值范围，例如，生成金字塔时或通过缩放生成图像时。默认情况下，NODATA单元在这些过程中被视为常规单元，以避免膨胀或侵蚀。然而，当被选择NODATA值或值范围，以考虑和再采样方法是BILINEAR ， BIQUADRATIC ， CUBIC ， AVERAGE4 ，或AVERAGE16 ，然后每当涉及重采样计算一个单元格的值是NODATA值，则重新采样的结果被也是NODATA值。生成的NODATA值是每个重采样窗口内的第一个NODATA值，其中单元格值从左上角到右下角逐行排序。

如果在版本11g之前有GeoRaster对象，并且栅格描述中存储了NODATA元数据，则该元数据对于向后兼容性仍然有效。旧的NODATA值被视为对象范围，当您在对象层上调用SDO_GEOR.addNODATA过程或在对象层上调用SDO_GEOR.deleteNODATA过程而不删除旧的NODATA时，它将移动到对象层值。

使用SDO_RANGE_ARRAY类型描述NODATA值或值范围，SDO_RANGE_ARRAY类型定义为VARRAY(1048576) OF SDO_RANGE ; SDO_RANGE类型指定下限和上限，并定义为（ LB NUMBER, UB NUMBER) 。

  • 要在SDO_RANGE定义中指定单个数字，请将LB指定为数字，将UB指定为null。以下示例将2指定为NODATA值： SDO_RANGE_ARRAY(SDO_RANGE(2,NULL))

  • 其中LB = UB的 SDO_RANGE（ LB，UB ）被认为与SDO_RANGE（ LB ，NULL）相同。

  • 实际的NODATA值范围（其中UB不是NULL且LB小于UB）包含在下限并且在上限处是独占的。

  • 您可以指定多个NODATA值范围和单个NODATA值。以下示例指定一个NODATA值（5）和两个NODATA值范围（1,3）和（7,8）： SDO_RANGE_ARRAY(SDO_RANGE(1,3), SDO_RANGE(5,NULL), SDO_RANGE(7,8))

多个PL / SQL子程序对NODATA值和与GeoRaster层关联的值范围执行操作（如添加，删除和查询）。

在GeoRaster中，位图掩码可以被视为特殊类型的NODATA，即NODATA掩码，将一个或多个不规则区域指定为NODATA区域。在这种情况下，位图掩模不仅在所识别bitmapMask所述的元件layerInfo的元数据，但也与注册NODATA所述的元件layerInfo元数据。但是，在任何重采样处理和统计分析期间都不会考虑位图掩码NODATA值。

1.11压缩和解压缩

GeoRaster提供以下类型的本机压缩，以减少GeoRaster对象的存储空间要求：JPEG（JPEG-F），JPEG 2000和DEFLATE。

  • 使用JPEG（JPEG-F）和DEFLATE压缩，GeoRaster对象的每个块都是单独压缩的，作为一种独特的栅格表示;当解压缩压缩的GeoRaster对象时，每个块都被单独解压缩

  • 使用JPEG 2000压缩，每个GeoRaster对象作为JP2文件存储在单个BLOB中，其中可以在内部阻止栅格。

对于JPEG（JPEG-F）和DEFLATE压缩，可以在压缩的（未压缩的）GeoRaster对象上执行的任何GeoRaster操作也可以在压缩的GeoRaster对象上执行。当GeoRaster执行操作时，如果源GeoRaster对象被压缩，GeoRaster会根据需要在内部解压缩源对象的块，执行指定的操作，然后以compression关键字指定的格式压缩生成的对象，或者，如果compression关键字未在源对象的压缩格式中指定。因此，在执行某些操作之前，您不需要解压缩压缩的GeoRaster对象，但如果在执行其他操作之前解压缩对象，则可能会获得一些整体性能优势。

对于JPEG 2000压缩，大多数GeoRaster操作可以在执行操作时在内部解压缩JP2压缩的GeoRaster对象。

在数据库用户压缩或解压缩GeoRaster对象之前，请确保已使用默认临时表空间创建数据库，或者已为用户分配了临时表空间或表空间组。否则，默认情况下， SYSTEM表空间用于临时表空间，并且在GeoRaster操作期间生成的大型临时LOB数据将放入SYSTEM表空间，这可能会影响整体数据库性能。有关管理临时表空间的信息，请参见“ Oracle数据库管理员指南” 。

要指定GeoRaster对象的压缩或解compression ，请在storageParam参数中使用compression关键字，该参数在“ 存储参数”中进行了描述。您可以将storageParam参数中的compression关键字与所有GeoRaster过程一起使用。（对于JPEG（JPEG-F）和DEFLATE压缩，没有单独的程序来压缩和解压缩GeoRaster对象。）

如果源GeoRaster对象为空，则忽略compression关键字，但SDO_GEOR.getRasterSubset和SDO_GEOR.getRasterData函数除外。也就是说，永远不会压缩空白的GeoRaster对象，并且元数据中的压缩类型始终为NONE 。 （空白和空GeoRaster对象中解释了空白GeoRaster对象 。）

1.11.1 JPEG（JPEG-F）压缩GeoRaster对象

JPEG（JPEG-F）压缩仅用于与GeoRaster对象支持cellDepth的值8BIT_U和每块不超过4个频带，并且每个块必须具有1个谱带，3个条带，或4个频带。 （JPEG压缩不支持每个块2个波段。）您可以通过使用带块大小为1,3或4个波段重新锁定GeoRaster对象来JPEG压缩超过4个波段的GeoRaster对象。带有色彩映射的GeoRaster对象不支持JPEG压缩。

虽然任何大小的GeoRaster对象都支持JPEG压缩，但GeoRaster对象的每个块的总大小（ columnsPerBlock * rowsPerBlock * bandsPerBlock * cellDepth / 8 ）不得超过50兆字节（MB）。对于大型GeoRaster对象，可以调用SDO_GEOR.changeFormatCopy过程将GeoRaster对象阻塞为小于50 MB的块，然后压缩GeoRaster对象;或者您可以在对SDO_GEOR.changeFormatCopy过程的同一调用中执行阻塞和压缩。

GeoRaster支持JPEG-F压缩模式，该模式压缩全格式基线JPEG格式的对象。

JPEG-F压缩在CCITT Rec。中描述。 T.81 JPEG规范（或ICO / IEC IS 10918-1）。GeoRaster使用CCITT Rec表K.2中的量化表。 T.81 JPEG规范和（对于Huffman表）该规范的表K.4和K.6中的标准色度表。在将表应用于压缩过程期间的数据之前，通过压缩质量来缩放量化表。

JPEG-F是一种有损压缩格式。您可以使用quality关键字控制storageParam参数的损失程度。quality关键字采用0到100之间的整数值。值0（零）提供最大压缩，但会导致大量数据丢失。值75（GeoRaster默认值）提供的图像大多数人认为没有质量损失，但提供了显着的压缩。值100表示压缩程度最小，但质量最佳。

  • JPEG-B支持已弃用

1.11.2 JPEG 2000压缩GeoRaster对象

GeoRaster支持按照标准ISO / IEC 15444-1对单元深度8BIT_U和16BIT_U光栅图像进行JPEG 2000（JP2）压缩。JPEG 2000压缩的GeoRaster对象存储在一个栅格块中。该光栅块中的数据采用JP2文件格式，如标准ISO / IEC 15444-1附录I中所述。JPEG 2000压缩GeoRaster对象中包含的图像可以在内部平铺。

使用JPEG 2000压缩，金字塔隐式嵌入JP2压缩数据中，因此JP2压缩的GeoRaster对象中没有单独的显式金字塔存储。可以从JP2压缩的GeoRaster对象检索的最大金字塔等级是log2(min(tile_width, tile_height)) ，其中tile_width和tile_height分别是内部切片的宽度和高度。支持有损和无损压缩。

SDO_GEOR.compressJP2过程用于将GeoRaster对象压缩为JP2压缩的GeoRaster对象。SDO_GEOR.decompressJP2过程可用于将JP2压缩的GeoRaster对象显式解压缩为另一个GeoRaster对象。其他GeoRaster操作（例如整流，镶嵌和栅格代数 - 但不是SDO_GEOR.changeCellValue ， SDO_GEOR.reproject ， SDO_GEOR.scaleScopy和SDO_GEOR.mosaic ）可以在执行操作时在内部解压缩JP2压缩的GeoRaster对象。

可以压缩大图像，但大小受内存和最大图块数量的限制（max_mem_size / 20 * 65535）。要提高可伸缩性和性能，请始终应用内部平铺。 可以使用SDO_GEOR.compressJP2过程的compressParam参数中的tileSize关键字指定切片大小。支持的最大切片数为65535。

1.11.3对GeoRaster对象进行DEFLATE压缩

DEFLATE压缩根据Deflate压缩数据格式规范（网络工作组RFC 1951）压缩对象，并以ZLIB格式存储压缩数据，如ZLIB压缩数据格式规范（网络工作组RFC 1950）中所述。ZLIB头和校验和字段包含在压缩的GeoRaster对象中。

虽然任何大小的GeoRaster对象都支持DEFLATE压缩，但GeoRaster对象的每个块的总大小（ columnsPerBlock * rowsPerBlock * bandsPerBlock * cellDepth / 8 ）不得超过1千兆字节（GB）。对于大型GeoRaster对象，可以调用SDO_GEOR.changeFormatCopy过程将GeoRaster对象阻塞为小于1 GB的块，然后压缩GeoRaster对象;或者您可以在对SDO_GEOR.changeFormatCopy过程的同一调用中执行阻塞和压缩。

由于DEFLATE压缩是无损的，因此压缩质量不适用，如果指定则会被忽略。

1.11.4 GeoRaster对象的解压缩

您可以通过在storageParam参数中指定compression=NONE来解压缩数据库中的压缩GeoRaster对象。对于JPEG-F压缩，不应将压缩质量指定为存储参数。

您可以使用现有的应用程序编程接口（API）（如PL / SQL或Oracle调用接口（OCI））解压缩数据库外部（即客户端）的压缩GeoRaster对象，以检索相应的BLOB对象到GeoRaster对象的块，并根据相关压缩格式的规范单独解码每个压缩块。例如，如果在JPEG-F模式下压缩GeoRaster对象，则解码过程应首先解析JPEG标头以检索表和块尺寸，然后将霍夫曼解码和反量化应用于图像数据。

完全靠您自己实现JPEG解压缩是一个复杂的，注重细节的过程。根据应用程序，使用现有实现可能更好。诸如C中的jpeglib和Java中的几个映像API（例如，Oracle J2SE和JAI）之类的库已经实现了JPEG解压缩，您可以调整它们以对JPEG压缩的GeoRaster对象执行解码过程。您可以使用ZLIB C库或Java API为DEFLATE压缩应用基本相同的方法。

1.11.6高级LOB压缩

您可以使用Oracle数据库高级LOB压缩（在Oracle Database SecureFiles和大对象开发人员指南中简要介绍）来实现GeoRaster栅格数据表（RDT）的无损压缩，从而压缩GeoRaster对象。如果在创建表（例如RDT的rasterBlock列）时为LOB存储指定高级LOB压缩，则使用高级LOB压缩来压缩该表的所有行中的SecureFiles LOB。压缩对GeoRaster是透明的，因此不需要更改应用程序。但是，如果您还在这些块上使用任何特定于GeoRaster的压缩类型（例如JPEG，DEFLATE或第三方插件），则应避免在RDT栅格块上使用高级LOB压缩。

1.13 GeoRaster中的并行处理

GeoRaster有两种类型的并行处理。

  • 并行执行SQL语句

  • 并行化GeoRaster程序

SQL语句的并行执行允许大多数SQL语句（查询和DML）并行运行。执行SQL语句时，会将其分解为单个步骤或行源，这些步骤或行源在执行计划中被标识为单独的行。

所有GeoRaster只读函数（例如，与元数据相关的查询操作（即，所有GeoRaster元数据获取函数和SDO_GEOR.validateGeoRaster ））以及所有单栅格单元查询（ SDO_GEOR.getCellValue和SDO_GEOR.evaluateDouble ）都启用并行查询。这意味着在多CPU环境中，如果这些函数用于查询一个或多个GeoRaster表中的许多GeoRaster对象，并且如果使SQL语句并行运行，则GeoRaster行会自动划分为多个子集，并且多个Oracle服务器进程将同时处理每个子集以减少总体响应时间。通过将工作划分为在多个进程之间运行GeoRaster SQL语句，您可以更快速地维护空间索引并根据其位置，各种元数据和属性查找GeoRaster对象。您还可以使用流水线和并行表函数来实现更复杂的过程，包括在单个GeoRaster对象上并行化某些操作。

通过并行化GeoRaster过程，您可以指定多个子过程以同时处理GeoRaster对象。专门实现了一些单独的光栅和图像处理过程以支持这种类型的并行性。使用这些过程，您只需将并行度（DOP）的整数指定为输入参数，即可将操作拆分为多个子进程，以同时处理单个GeoRaster对象的子集。每个子流程都独立运行。完成所有子流程后，整个流程结束。以下过程直接支持这种并行处理：

  • SDO_GEOR.generatePyramid

  • SDO_GEOR_RA.classify

  • SDO_GEOR_RA.findCells

  • SDO_GEOR_RA.rasterMathOp

  • SDO_GEOR_RA.rasterUpdate

  • SDO_GEOR_AGGR.mosaicSubset

通过SDO_GEOR_AGGR.mosaicSubset过程，支持其他类型的并行操作。这些包括并行压缩和解压缩，并行复制或更改格式复制，并行子集化，并行重投影和并行校正。有关详细信息，请参阅并行压缩，复制和子集 。

图像和栅格数据通常非常大，因此前面的操作可能非常耗时。因此，在使用多CPU或多核服务器时，请始终考虑使用并行处理来提高性能。

1.19.1基于JAI的查看器，加载器和导出器

基于GeoRaster JAI的工具包括查看器，加载器和导出器。这些工具适用于DBA和应用程序开发人员。查看器对于检查所有类型的GeoRaster对象及其元数据特别有用。它还可以显示定义为一个或一组GeoRaster表或视图的虚拟马赛克。加载器和导出器是轻量级工具，可以方便地一次加载和导出有限数量的图像和光栅文件。 $ORACLE_HOME/md/demo/georaster/tool/README.txt文件包含有用的使用信息和使用以下工具的说明：

  • GeoRaster查看器显示GeoRaster对象和元数据，以及虚拟马赛克。您可以同时连接到多个数据库，并查看左窗格中列出的每个数据库中的GeoRaster对象。您可以快速切换各种分辨率的视图，从原始图像（金字塔等级0）到概览（最高金字塔等级）。您可以执行图像增强，例如线性拉伸（自动，手动或分段），标准化，均衡以及亮度，对比度和阈值控制。（有关查看GeoRaster对象的更多信息，请参阅查看GeoRaster对象 。）

    在查看器中，您可以调用GeoRaster加载程序和导出程序工具，从而使您可以使用单个工具作为所有GeoRaster工具功能的接口。本节和$ORACLE_HOME/md/demo/georaster/tool/JAI_based_tools_user_guide.txt文件中描述了加载程序和导出程序工具。

  • GeoRaster加载程序，它将栅格数据加载到GeoRaster对象中。它可以加载以下图像格式：TIFF，GeoTIFF，JPEG，BMP，GIF，PNG和JP2。可以从ESRI世界文件，GeoTIFF文件和Digital Globe RPC文本文件加载地理配准信息。

    您可以使用GeoRaster加载程序替代SDO_GEOR.importFrom过程，该过程在SDO_GEOR程序包参考中有记录 。但是，在非Windows系统上，此加载程序工具不支持BMP或GIF图像格式，因此必须在非Windows系统上使用具有这些格式的SDO_GEOR.importFrom过程。此工具不支持单元深度值为2BIT栅格数据，也不支持具有BIL或BSQ交错类型的源多波段栅格数据。导入的GeoRaster对象具有BIP交错类型。无法回滚此工具的加载操作。

    加载JPEG文件格式的图像时，操作所需的内存量取决于未压缩图像的大小，可以使用-Xmx选项指定为命令行参数（例如， java -Xmx256M oracle.spatial.georaster.tools.GeoRasterLoader ... ）。

  • GeoRaster导出器，将GeoRaster对象导出到图像文件。GeoRaster导出器工具支持以下目标图像文件格式：TIFF，GeoTIFF，JPEG，BMP，GIF，PNG和JP2。地理配准信息可以导出到ESRI世界文件，GeoTIFF文件和Digital Globe RPC文本文件。

    您可以使用它作为SDO_GEOR.exportTo过程的替代方法，该过程在SDO_GEOR包参考中有记录 。但请注意，GeoRaster导出器工具不支持GIF作为目标文件格式; SDO_GEOR.exportTo过程不支持GIF，JPEG或JP2作为目标文件格式。GeoRaster导出器工具不支持cellDepth值为2BIT GeoRaster对象。具有BSQ或BIL交织的具有8比特或更大的小区深度的GeoRaster对象以BIP交织格式导出。

    有关可以在单个操作中导出的GeoRaster数据量限制的信息，请参阅SDO_GEOR.exportTo过程的使用说明。

关于图像大小和类型，可能会对加载和导出操作施加一些限制;请参阅GeoRaster工具的$ORACLE_HOME/md/demo/georaster/tool/JAI_based_tools_user_guide.txt文件。

这些工具是用Java开发的，因此只要您与Oracle数据库建立网络连接，就可以通过Intranet或Internet在任何地方运行它们。

要使用GeoRaster客户端工具加载或导出GeoTIFF图像，请将以下库添加到CLASSPATH定义中：

  • xtiff-jai.jar （可从SourceForge Extensible-TIFF-JAI组获得）

  • geotiff-jai.jar （可从SourceForge GeoTIFF-JAI组获得）

要加载或导出JP2映像，请将以下库添加到CLASSPATH定义中： jai-imageio.jar （可从Oracle Java Advanced Imaging Image I / O Tools下载页面获得）。

将栅格或图像文件加载到GeoRaster对象后，数据将完全存储在本机GeoRaster对象数据类型中，并且与任何特定文件格式无关。

如果要创建自己的GeoRaster加载程序和导出程序工具，可以使用OCI，Oracle C ++调用接口（OCCI）或Java开发它们，并且可以将它们实现为客户端命令或服务器端SQL过程或函数。

1.19.2基于GDAL的ETL向导，用于并发批量加载和导出

GeoRaster包含一个ETL向导工具，可以使用GDAL自动执行并启用并发批量加载和导出各种图像和光栅文件。这个功能强大的工具可以批量加载和同时导出大量的光栅和图像文件。

它定义了XML模式，并提供了一个图形用户界面，用于以XML格式创建加载和导出描述文件。每个描述文件都描述了如何批量加载或从GeoRaster中导出一系列光栅文件。创建XML描述文件后，您可以使用相同的向导工具调用多个描述文件，以批量同时加载和导出光栅文件。捕获并记录任何运行时故障，但它们不会停止批处理加载或导出过程。此工具支持GDAL支持的所有栅格格式。

要使用此向导，必须从Oracle数据库示例介质安装演示文件（请参阅“ Oracle数据库示例安装指南” ）。安装完成后，此向导即可进入以下.jar文件（假设$ORACLE_HOME /md的默认空间和图形安装目录）：

$ ORACLE_HOME / MD /演示/的GeoRaster /工具/ georastertool.jar

$ORACLE_HOME/md/demo/georaster/tool/README.txt文件描述了如何设置GDAL并启动向导。

$ORACLE_HOME/md/demo/georaster/tool/GDAL_based_etl_user_guide.pdf文件详细描述了用法。

1.19.3从空间和图形安装中使用GDAL

地理空间数据抽象库（GDAL）是一个开源软件库，支持许多数据格式和服务。GDAL库，命令行工具和编程接口支持Oracle Spatial几何和GeoRaster对象。

GDAL随Oracle Spatial and Graph一起分发，它安装在Linux系统上的$ORACLE_HOME/md/gdal和Windows系统上的%ORACLE_HOME% \md\gdal 。（GDAL不与其他平台上的Oracle Spatial and Graph一起分发。）

要为命令行使用准备GDAL，必须将GDAL bin ， data ， lib和plugins文件夹添加到系统环境变量中。

以下示例在Linux上设置GDAL：

setenv GDAL_HOME $ {ORACLE_HOME} / md / gdal setenv GDAL_DATA $ {GDAL_HOME} / data setenv GDAL_DRIVER_PATH $ {GDAL_HOME} / lib / gdalplugins setenv PATH $ {GDAL_HOME} / bin：$ {PATH} setenv LD_LIBRARY_PATH $ {GDAL_HOME} / lib： $ {LD_LIBRARY_PATH}

set GDAL_HOME =％ORACLE_HOME％\ md \ gdal set GDAL_DATA =％GDAL_HOME％\ data set GDAL_DRIVER_PATH =％GDAL_HOME％\ bin \ gdalplugins set PATH =％GDAL_HOME％\ bin;％PATH％

前面的示例假定已在系统中配置Oracle OCI共享库。可以在Oracle数据库或Instant Client安装中找到Oracle OCI共享库。

1.19.4使用SDO_GEOR_GDAL包

SDO_GEOR_GDAL PL / SQL包通过外部过程将开源软件GDAL与Oracle Database Server集成在一起，并提供PL / SQL API来执行一组GDAL功能。

SDO_GEOR_GDAL包中的函数和过程将在Oracle数据库服务器系统上执行，并且可以与任何其他GeoRaster PL / SQL API一起使用。

目前，SDO_GEOR_GDAL包仅适用于Windows和Linux操作系统。
