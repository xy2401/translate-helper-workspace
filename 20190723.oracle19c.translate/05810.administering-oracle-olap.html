7.2存储管理

除非所有者另行指定，否则分析工作区将存储在所有者的默认表空间中。OLAP使用的所有表空间都应指定EXTENT MANAGEMENT LOCAL 。使用默认参数创建的表空间可能无效地使用资源。您应该创建适合存储分析工作区的撤消，永久和临时表空间。

7.2.1创建撤消表空间

使用EXTENT MANAGEMENT LOCAL子句创建一个undo表空间，如下例所示：

创建UNDO TABLESPACE olapundo DATAFILE'$ ORACLE_BASE / oradata / undo.dbf'SIZE 64M REUSE AUTOEXTEND on next 8M MAXSIZE UNLIMITED EXTENT MANAGEMENT LOCAL;

创建撤消表空间后，更改系统参数文件以包括以下设置，然后按照“ 设置数据库初始化参数 ”中的说明重新启动数据库。 

UNDO_TABLESPACE = 表空间 UNDO_MANAGEMENT = AUTO

7.2.2为OLAP使用创建永久表空间

每个维对象占据至少一个范围。固定范围大小可能会浪费大部分分配的空间。例如，如果对象为64K且范围设置为1M的统一大小（默认值），则仅使用范围的一小部分。

使用EXTENT MANAGEMENT LOCAL和SEGMENT SPACE MANAGEMENT AUTO子句创建永久表空间，如下例所示：

CREATE TABLESPACE glo DATAFILE'$ ORACLE_BASE / oradata / glo.dbf'SIZE 64M REUSE AUTOEXTEND on next 8M MAXSIZE UNLIMITED EXTENT MANAGEMENT LOCAL SATERMENT SPACE AUTO;

7.2.3为OLAP使用创建临时表空间

无论更改是数据加载还是数据分析的结果，Oracle OLAP都使用临时表空间来存储多维数据集中数据的所有更改。保存多维数据集会将更改移动到永久表空间并清除临时表空间。

此用法在表空间中创建了许多范围。适合Oracle OLAP使用的临时表空间应指定EXTENT MANAGEMENT LOCAL子句和小尺寸的UNIFORM SIZE子句，如下例所示：

CREATE TEMPORARY TABLESPACE glotmp TEMPFILE'$ ORACLE_BASE / oradata / glotmp.tmp'SIZE 50M REUSE AUTOEXTEND on next 5M MAXSIZE UNLIMITED EXTENT MANAGEMENT LOCAL UNIFORM SIZE 256K;

7.2.4跨存储资源传播数据

Oracle数据库提供了出色的存储管理工具来简化日常任务。自动存储管理（ASM）提供了一个简单的存储管理界面，可将数据库存储虚拟化为磁盘组。您可以管理一小组磁盘组，ASM可以自动在这些磁盘组中放置数据库文件。

ASM在所有可用存储资源上均匀分布数据，以优化性能和利用率。添加或删除磁盘后，ASM会自动重新平衡磁盘组中的文件。

由于OLAP是Oracle数据库的一部分，因此您可以使用ASM来管理关系数据和维度数据。

强烈建议ASM用于分析工作区。使用ASM管理的系统比文件系统更快，并且比原始设备更易于管理。ASM在使用Oracle RAC的系统和没有Oracle RAC的系统上优化分析工作区的性能。

但是，您不需要ASM来使用Oracle OLAP。您仍然可以跨多个磁盘传播数据，只需定义表空间，如下例所示：

CREATE TABLESPACE glo DATAFILE'disk1 / oradata / glo1.dbf'SIZE 64M REUSE AUTOEXTEND ON NEXT 8M MAXSIZE 1024M EXTENT MANAGEMENT LOCAL SATERMENT SPACE AUTO AUTO; ALTER TABLESPACE glo ADD DATAFILE'disk2 / oradata / glo2.dbf'SIZE 64M REUSE AUTOEXTEND ON NEXT 8M MAXSIZE 1024M，'disk3 / oradata / glo3.dbf'SIZE 64M REUSE AUTOEXTEND ON NEXT 8M MAXSIZE UNLIMITED;

7.3.3分析工作空间表

分析工作区存储在Oracle数据库的表中。这些表的名称始终以AW$开头。

例如，如果GLOBAL用户创建了两个分析工作区，一个名为FINANCIALS ，另一个名为MARKETING ，那么这些表是在GLOBAL模式中创建的：

AW $ FINANCIALS AW $ MARKETING

这些表存储了所有对象定义和数据。

7.3.4维护日志

第一次使用Analytic Workspace Manager将数据加载到多维数据集或维度时，它会创建多个日志。这些日志存储在与分析工作区相同的模式中的表中：

  • 多维数据集生成日志 ：包含有关生成期间发生的事件的信息。使用此日志来确定构建是否产生了您期望的结果，如果不是，为什么不产生。无论是通过Analytic Workspace Manager，数据库物化视图刷新子系统还是PL / SQL过程，每当刷新多维数据集或维度时，日志都会不断更新。您可以随时查询日志以评估构建进度并估计完成时间。默认表名称为CUBE_BUILD_LOG 。

  • 多维数据集维度编译日志 ：包含OLAP聚合多维数据集时在维度层次结构验证期间发生的错误。默认表名是CUBE_DIMENSION_COMPILE 。

  • 多维数据集操作日志 ：包含所有OLAP引擎事件的消息和调试信息。默认表名称为CUBE_OPERATIONS_LOG 。

  • 多维数据集拒绝记录日志 ：标识因未满足预期格式而被拒绝的任何记录。默认表名为CUBE_REJECTED_RECORDS 。

通过这些日志，您可以跟踪长时间运行的进程的进度，然后使用结果来分析性能特征。它们提供的信息可帮助您诊断和解决在开发和维护多维数据集期间可能发生的问题。它们还有助于诊断查询多维数据集时的性能问题。

您还可以运行$ ORACLE_HOME / olap / admin / utlolaplog.sql脚本来创建包含一些有用视图的构建日志。

Analytic Workspace Manager中的维护向导在维护日志页面上的每次构建期间显示这些表中的相关行。您可以在任何SQL界面中直接查询表。

7.4分区立方体和并行

多维数据集通常被分区以改善构建和维护时间。有关创建分区的多维数据集的信息，请参阅“ 分区多维数据集 ”。以下主题讨论了分区和并行性：

  • 查询多维数据集分区的元数据

  • 创建和删除分区

7.4.1查询多维数据集分区的元数据

要发现当前分区，请查询ALL_CUBES数据字典视图。PARTITION_DIMENSION_NAME ， PARTITION_HIERARCHY_NAME和PARTITION_LEVEL_NAME列显示分区信息。例如，以下查询显示“单位多维数据集”在“时间”维度，“日历”层次结构和“日历年”级别上进行了分区。 

SELECT partition_dimension_name，partition_hierarchy_name，partition_level_name FROM all_cubes WHERE owner ='GLOBAL'AND cube_name ='UNITS_CUBE'; PARTITION_DIMENSION_NAME PARTITION_HIERARCHY_NAME PARTITION_LEVEL_NAME ------------------------- ---------------------- --- -------------------- TIME CALENDAR CALENDAR_YEAR

7.4.2创建和删除分区

作为数据维护的一部分，OLAP引擎会自动创建和删除分区，因为会在分区维度中添加和删除成员。

例如，假设在示例“全局分析”工作空间中，“单位”多维数据集在“时间”维度上使用“日历”层次结构和“日历季度”级别进行分区。OLAP引擎为每个Calendar Quarter及其子项创建一个分区。默认顶部分区包含Calendar Years和Fiscal层次结构的所有成员。如果Global有三年的数据，那么Units多维数据集有13个分区：每个Calendar Year的四个底部分区，加上顶部分区。

数据刷新通常会创建新的时间段并删除旧的时间段。只要将“日历季度”值加载到“时间”维度中，就会将相应的分区添加到多维数据集中。每当从“时间”维度中删除“日历季度”值时，都会从多维数据集中删除相应的空分区。

7.4.3并行性

您可以通过启用并行处理来提高数据维护的性能。有两个级别的并行性：

  • 并行作业执行：使用多个进程加载和聚合数据。

  • 并行更新：使用多个进程将数据从临时表空间移动到永久表空间。

这些并行进程受以下因素控制：

  • 可以并行聚合的对象数。每个多维数据集和每个分区（包括顶部分区）都可以使用单独的进程。

    您可以在Analytic Workspace Manager的多维数据集属性表的“分区”选项卡上控制多维数据集中的分区数。

  • 用户被授权运行的并发数据库进程数。

    这个数字是由 JOB_QUEUE_PROCESSES参数。如果您具有SYS权限，则可以使用以下SQL命令获取当前参数设置：

    SHOW PARAMETER JOB_QUEUE_PROCESSES

  • 对于并行更新，您分配给作业的进程数。在指定任务处理选项时，或在多维数据集的“物化视图”选项卡上，可以在Analytic Workspace Manager的维护向导中指定进程数。

  • 分配给SQL以从关系源表中获取行的进程数。当PARALLEL_DEGREE_POLICY设置为AUTO或LIMITED ，数据库可以分配其他进程来执行SQL语句。

假设多维数据集在“季度”级别上进行分区，并且多维数据集包含三年的数据。多维数据集具有3 * 4 = 12个底部分区， JOB_QUEUE_PROCESSES设置为8，并且您将并行性选项设置为4用于构建。当PARALLEL_DEGREE_POLICY设置为其默认值MANUAL时，Oracle数据库以这种方式处理多维数据集：

  1. 使用单个过程连续加载和构建立方体的尺寸。

  2. 使用4个进程并行加载和构建12个底部分区。一旦一个过程完成，另一个过程开始直到所有12个过程完成。

    此多维数据集可以使用JOB_QUEUE_PROCESSES允许的8个进程，但构建设置限制为4。

  3. 加载并构建顶部分区。

当PARALLEL_DEGREE_POLICY设置为AUTO或LIMITED ，Oracle数据库可能会分配多于指定的进程。

7.5分析立方体和尺寸

如果应用程序直接针对单个多维数据集执行查询，则无需为多维数据集生成优化程序统计信息。这些查询在分析工作区内自动优化。

优化程序统计信息用于为连接两个多维数据集视图或将多维数据集视图连接到表或表视图的查询创建执行计划。它们还用于基于成本的重写以实现多维数据集物化视图。您必须仅为这些类型的查询生成统计信息。

生成优化器统计，使用 DBMS_AW_STATS PL / SQL包。您可以在Analytic Workspace Manager中将此包作为多维数据集脚本，SQL * Plus或任何其他SQL界面的一部分运行。生成统计信息不会产生显着的性能成本。

DBMS_AW_STATS具有以下语法：

DBMS_AW_STATS.ANALYZE（对象IN VARCHAR2）;

7.7关于备份和恢复分析工作区

您可以使用与数据库其余部分相同的工具和过程来备份和恢复分析工作区。

Oracle Recovery Manager（RMAN）是一款功能强大的工具，可简化，自动化和改进备份和恢复操作的性能。RMAN支持基于用户指定的恢复窗口，可重新启动的备份和还原以及测试还原/恢复的一次性备份配置，备份自动管理和归档日志。

RMAN实现了一个恢复窗口来控制备份何时到期。这使您可以通过执行数据库或表空间时间点恢复来建立可以发现逻辑错误并修复受影响对象的时间段。RMAN还会自动将不再需要的备份过期，以将数据库还原到恢复窗口中的某个时间点。即使RMAN存储库不可用，控制文件自动备份也允许还原或恢复数据库。

7.8关于复制分析工作区

您可以通过多种不同方式复制分析工作区，以便在另一台计算机上复制它们或备份它们。

  • 数据泵 。分析工作区与模式或数据库导出中的其他对象一起复制。使用expdp / impdp数据库实用程序。

    小费：

    验证导入的目标模式是否具有OLAP_XS_ADMIN权限。否则，将不会使用必要的权限创建分析工作区。

  • 可传输的表空间 。分析工作区将与其他对象一起复制到 ransportable表空间。但是，您只能将表空间传输到同一平台（例如，从Linux到Linux，从Solaris到Solaris，或从Windows到Windows），因为OLAP DECIMAL数据类型与硬件有关。使用expdp / impdp数据库实用程序。可传输表空间比转储文件快得多。

分析工作区的所有者可以将架构导出到转储文件。只有用户才有 EXP_FULL_DATABASE权限或特权用户（例如SYS或具有DBA角色的用户）可以导出完整数据库或创建可传输表空间。

7.9关于保存维对象定义

您可以将对象定义保存在外部文件中，以便将它们传输到另一个数据库或保存备份副本。您还可以将对象定义保存到表中，以使它们在Oracle数据库中可用。您可以将定义保存在XML模板或EIF文件中。两个文件都是平台无关的。

7.9.1关于XML模板

模板是描述维对象的XML文档。您可以将所有对象的XML描述保存在分析工作区或仅选定对象中，稍后在同一数据库或另一台计算机或平台上的数据库中重新创建它们。您可以使用模板备份工作，同时开发数据的维度模型或将设计分发给其他用户。

您可以保存以下类型对象的XML定义：

  • 分析工作区 ：保存所有维对象和所有用户定义的OLAP DML程序和对象。

  • 维度 ：保存维度及其级别，层次结构，属性和映射。

  • 多维数据集：保存多维数据集及其度量，计算度量，维度，映射以及与多维数据集关联的所有用户定义的OLAP DML程序和对象。

  • 度量文件夹 ：在度量文件夹中保存度量列表。它不保存对象。

模板存储元数据，而不是数据。您可以将模板存储在小型文本文件或数据库表中。从模板重新创建对象时，您必须有权访问源数据。

7.9.2关于EIF文件

您可以将分析工作区中的对象导出到EIF文件。EIF文件是用于复制分析工作区的特殊格式的文件。您可以使用EIF文件：

EIF文件不保存对象安全规则。

7.10.2启动数据刷新

您可以使用Analytic Workspace Manager或PL / SQL包以几种不同的方式启动多维数据集物化视图的数据刷新：

  • 自动刷新 ：在多维数据集的“物化视图”选项卡上，您可以为物化视图刷新子系统创建常规计划，如“ 向多维数据集添加物化视图功能 ”中所述 。

  • 维护向导 ：维护向导可用于刷新所有多维数据集和维度，包括多维数据集物化视图。

  • DBMS_CUBE ： DBMS_CUBE PL / SQL包可用于刷新所有多维数据集，多维数据集维度和多维数据集物化视图。

  • DBMS_MVIEW ： DBMS_MVIEW PL / SQL包中包含几个用于多维数据集物化视图的过程。

7.10.2.1使用DBMS_CUBE

您可以使用DBMS_CUBE创建和填充分析工作区或维护任何多维数据集，包括多维数据集物化视图。

以下命令启动UNITS_CUBE的完整刷新，该过程作为多维数据集物化视图启用。在刷新多维数据集之前，它会自动刷新任何陈旧的维度。 

EXECUTE dbms_cube.build（'GLOBAL.UNITS_CUBE'）;

您可以从USER_MVIEWS确定刷新方法，如“ 识别多维数据集物化视图 ”中所示 。

7.10.2.2使用DBMS_MVIEW

您可以使用DBMS_MVIEW刷新所有类型的物化视图。这些刷新过程可以与多维数据集物化视图一起使用：

  • REFRESH刷新一个或多个物化视图的列表。

  • REFRESH_ALL_MVIEWS刷新符合特定条件的所有物化视图。

  • REFRESH_DEPENDENT刷新依赖于特定主表并满足特定条件的所有物化视图。

必须在多维数据集之前刷新维度。如果任何关联的尺寸物化视图过时，则在刷新多维数据集物化视图期间会引发错误。DBMS_MVIEW的过程可以在一次调用中刷新多个物化视图，但它们不保证刷新顺序。要控制刷新顺序， DBMS_MVIEW.REFRESH多维数据集物化视图的DBMS_MVIEW.REFRESH化视图分开调用。

以下命令将启动CHANNEL_PRIMARY层次结构的CHANNEL_PRIMARY化视图的刷新。 

EXECUTE dbms_mview.refresh（'CB $ CHANNEL_PRIMARY'，'C'）;

7.10.3刷新方法

在Analytic Workspace Manager中，您可以指定用于刷新多维数据集的COMPLETE ， FAST或FORCE方法。物化视图子系统调用另外两种方法FAST_PCT和FAST_SOLVE 。它们不是单独的选择。

7.10.3.1刷新方法描述

表7-7说明了多维数据集物化视图支持的刷新方法。

表7-7多维数据集物化视图的刷新方法

刷新方法 描述

完成

删除并重新创建多维数据集。

此选项支持从源表到多维数据集的任意复杂映射。

快速

根据物化视图日志加载和重新聚合更改的值，或者在直接路径加载后，在ALL_SUMDELTA数据字典视图上加载和重新聚合。

刷新的来源是物化视图日志中捕获的增量差异，而不是原始映射源。这些差异用于逐步重建多维数据集。仅重新聚合受更改值影响的单元格。

此选项仅支持多维数据集物化视图的简单映射，即，在定义映射的查询中不会出现表达式（ table.column除外 ），视图或聚合。

物化视图子系统确定是执行FAST还是FAST_PCT刷新。有关该方法的信息，请参阅“ Oracle数据库数据仓库指南” 。

FAST_PCT

仅从已更改的分区加载和重新聚合数据。当源表和多维数据集在同一维度上分区时，此方法最有效。

FAST_PCT不使用更改日志。物化视图子系统确定是执行FAST还是FAST_PCT刷新。有关该方法的信息，请参阅。

FAST_SOLVE

仅根据原始映射数据源加载和重新聚合更改的值。

FAST_SOLVE是一种仅适用于多维数据集物化视图的刷新类型。即使刷新源是原始映射源而不是物化视图日志，它也会逐渐重新聚合多维数据集。聚合子系统识别差异，然后逐步重新聚合多维数据集。

从源表到多维数据集的任意复杂映射支持此选项。要发现是否已发生FAST_SOLVE刷新，请查看CUBE_BUILD_LOG表，如“ 快速解决刷新 ”中所示 。或者查看LAST_REFRESH_TYPE列ALL_MVIEWS ; FAST_SOLVE刷新显示为FAST_CS 。

力

使用尽可能最佳的方法加载和重新聚合值。

当不需要COMPLETE刷新时，物化视图系统首先尝试FAST刷新。如果无法FAST刷新多维数据集物化视图，则会执行FAST_SOLVE刷新。

7.10.4使用查询重写

查询重写更改查询以从实例化视图中选择数据，而不是从主表中计算结果集。转换对客户端完全透明，并且不需要在SQL语句中提及实体化视图。对于多维数据集物化视图，查询是针对星型或雪花式架构的表或视图编写的，并将其转换为针对多维数据集物化视图的查询。这种转换可以显着改善运行时性能。

查询重写需要有关多维数据集和维度的优化程序统计信息。您可以通过生成和检查其执行计划来发现是否重写了查询。

Oracle数据库使用两个初始化参数来控制查询重写：

  • QUERY_REWRITE_ENABLED ：为数据库启用或禁用全局查询重写。

  • QUERY_REWRITE_INTEGRITY ：确定查询重写监视具有源数据的QUERY_REWRITE_INTEGRITY化视图的一致性的程度。使用重写到多维数据集物化视图时，建议使用trusted或stale tolerated设置。

多维数据集物化视图的管理与任何其他物化视图相同，只是多维数据集物化视图必须与分析工作区位于同一模式中。用户需要 GLOBAL QUERY REWRITE特权，可以重写除了自己的模式之外的物化视图。但是，所有者可以从任何架构访问实例化视图，而无需其他权限。

也可以看看：

  • “ 分析多维数据集和维度 ”以获取有关优化程序统计信息的信息

  • 查看执行计划以获取有关执行计划的信息

  • Oracle Database Reference，用于初始化参数的完整描述

7.10.5获取有关多维数据集物化视图的其他信息

Oracle数据库有许多用于管理物化视图的PL / SQL包。多维数据集物化视图已经过优化，可提供最佳性能，因此您无需使用大多数这些程序包。很少有设计决定可供您使用。因此，对于多维数据集物化视图，将禁用DBMS_ADVISOR的TUNE_MVIEW过程。
