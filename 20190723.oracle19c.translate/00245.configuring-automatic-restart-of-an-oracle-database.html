4.1.2关于启动依赖关系

Oracle Restart可确保按照组件依赖性以正确的顺序启动Oracle组件。

例如，如果数据库文件存储在Oracle ASM磁盘组中，则在启动数据库实例之前，Oracle Restart会确保启动Oracle ASM实例并装入所需的磁盘组。同样，如果必须关闭组件，Oracle Restart会确保首先干净地关闭相关组件。

Oracle Restart还管理数据库实例与Oracle Net侦听器（侦听器）之间的弱依赖关系：启动数据库实例时，Oracle Restart会尝试启动侦听器。如果侦听器启动失败，则仍会启动数据库。如果侦听器稍后失败，则Oracle Restart不会关闭并重新启动任何数据库实例。

4.1.3关于使用Oracle Restart启动和停止组件

Oracle Restart会在需要时自动重新启动各种Oracle组件，并在您手动关闭系统时自动停止Oracle组件。

但是，有时您可能希望手动启动或停止单个Oracle组件。Oracle Restart包括用于手动启动和停止Oracle Restart管理的组件的服务器控制（SRVCTL）实用程序。使用Oracle Restart时，Oracle强烈建议您使用SRVCTL手动启动和停止组件。

使用SRVCTL停止组件后，如果发生故障，Oracle Restart不会自动重新启动该组件。如果随后使用SRVCTL启动组件，则该组件可再次用于自动重新启动。

SQL实用程序，监听器控制实用程序（ LSNRCTL ）和ASMCMD等Oracle实用程序与Oracle Restart集成在一起。如果使用SQL * Plus关闭数据库，则Oracle Restart不会将此解释为数据库故障，也不会尝试重新启动数据库。同样，如果使用SQL * Plus或ASMCMD关闭Oracle ASM实例，则Oracle Restart不会尝试重新启动它。

使用SR * CTL启动组件并使用SQL * Plus（或其他实用程序）启动组件之间的一个重要区别如下：

  • 使用SRVCTL启动组件时，将首先按正确的顺序自动启动此组件所依赖的任何组件。

  • 使用SQL * Plus（或其他实用程序）启动组件时，依赖关系链中的其他组件不会自动启动;您必须确保启动此组件所依赖的任何组件。

此外，Oracle Restart还允许您使用单个命令在指定的Oracle主目录中启动和停止Oracle Restart管理的所有组件。Oracle主目录可以是Oracle数据库主目录或Oracle Grid Infrastructure主目录。安装修补程序时，此功能非常有用。

4.1.5 Oracle重启配置

Oracle Restart维护其管理的所有Oracle组件的列表，并维护每个组件的配置信息。

所有这些信息统称为Oracle Restart配置 。当Oracle Restart启动组件时，它会根据该组件的配置信息启动组件。例如，Oracle Restart配置包括数据库的服务器参数文件（SPFILE）的位置，以及侦听侦听器的TCP端口。

如果安装Oracle Restart然后使用Database Configuration Assistant（DBCA）创建数据库，DBCA会自动将数据库添加到Oracle Restart配置中。当DBCA随后启动数据库时，将建立数据库与其他组件（例如数据库存储数据的磁盘组）之间所需的依赖关系，Oracle Restart开始管理数据库。

您可以使用SRVCTL命令从Oracle Restart配置中手动添加和删除组件。例如，如果将Oracle Restart安装到已运行数据库的主机上，则可以使用SRVCTL将该数据库添加到Oracle Restart配置中。手动将组件添加到Oracle Restart配置，然后使用SRVCTL启动它时，Oracle Restart开始管理组件，并在需要时重新启动它。

注意：

将组件添加到Oracle Restart配置也称为“使用Oracle Restart注册组件”。

使用其他SRVCTL命令可以查看Oracle Restart管理的组件的状态和配置，暂时禁用然后重新启用组件管理等。

4.1.6 Oracle重启与Oracle Data Guard的集成

Oracle Restart与Oracle Data Guard（Data Guard）和Oracle Data Guard Broker（代理）集成在一起。

当响应角色更改请求需要数据库关闭和重新启动时，Oracle Restart会按顺序关闭并重新启动数据库（考虑到依赖关系），并根据Oracle Restart配置中的设置。Oracle Restart还确保在Data Guard角色转换后，配置为在新数据库角色中运行的所有数据库服务都处于活动状态，并且所有未配置为在新角色中运行的服务都将停止。

此外，Oracle Restart配置支持以下组件的Data Guard相关配置选项：

  • 数据库 - 将数据库添加到Oracle Restart配置时，可以为数据库指定当前Data Guard角色： PRIMARY ， PHYSICAL_STANDBY ， LOGICAL_STANDBY或SNAPSHOT_STANDBY 。如果稍后使用代理更改角色，Oracle Restart将自动使用新角色更新数据库配置。如果在不使用代理的情况下更改数据库角色，则必须在Oracle Restart配置中手动修改数据库的角色以反映新角色。

  • 数据库服务 - 将数据库服务添加到Oracle Restart配置时，可以为服务指定一个或多个Data Guard角色。如果存在此配置选项，则在数据库打开时，仅当其中一个服务角色与当前数据库角色匹配时，Oracle Restart才会启动该服务。

4.1.7.1快速应用程序通知概述

FAN是一种高可用性通知机制，Oracle Restart可以使用该机制向其他进程通知包括服务状态更改的配置更改，例如UP或DOWN事件。

FAN提供在实例或服务器发生故障时立即终止飞行事务的能力。集成的Oracle客户端接收事件并进行响应。应用程序可以通过将错误传播给用户或通过重新提交事务并从应用程序用户屏蔽错误来响应。发生DOWN事件时，集成客户端会立即清除与已终止数据库的连接。发生UP事件时，客户端会创建与新主数据库实例的新连接。

每当托管实例或服务上升或下降时，Oracle Restart都会发布FAN事件。故障转移后，Oracle Data Guard Broker（代理）发布FAN事件。这些FAN事件可以通过以下方式使用：

  • 如果他们使用这些Oracle集成数据库客户端之一，应用程序可以将FAN与Oracle Restart一起使用而无需进行编程更改：Oracle数据库JDBC，Java通用连接池，Oracle调用接口和Oracle数据库ODP.NET。可以将这些客户端配置为快速连接故障转移（FCF），以便在故障转移后自动连接到新的主数据库。

  • 可以在数据库层上配置FAN服务器端标注。

对于DOWN事件（例如发生故障的主数据库），FAN会立即向客户端发出通知，以便它们可以尽快故障转移到新的主数据库。客户端不等待超时。将立即通知客户端，并且必须将它们配置为在收到通知时进行故障转移。

对于UP事件，当启动服务和实例时，可以创建新连接，以便应用程序可以立即利用额外资源。

通过服务器端标注，您还可以使用FAN：

  • 记录状态信息

  • 资源无法启动时，页面DBA或打开支持票证

  • 自动启动必须与服务位于同一位置的相关外部应用程序

使用ONS和Oracle数据库高级队列队列发布FAN事件。配置服务时会自动配置队列。您必须使用SRVCTL命令手动配置ONS。

Connection Manager（CMAN）和Oracle Net Services侦听器与FAN事件集成在一起，使CMAN和侦听器能够立即取消注册失败实例提供的服务，并避免错误地向失败的数据库发送连接请求。

FAN事件由Oracle Restart和Oracle Data Guard Broker通过ONS和高级队列发布。您还可以使用FAN标注执行通知。

注意：

Oracle Restart不会在保证订购的情况下运行标注。标注是异步运行的，它们受调度可变性的影响。

使用Oracle Restart，重启和恢复是自动的，包括重新启动子系统，例如监听器和Oracle自动存储管理（Oracle ASM）进程，而不仅仅是数据库。您可以使用FAN标注向故障管理系统报告故障并启动修复作业。

4.1.7.2.2管理计划中断

对于需要您关闭主数据库的修复，升级和更改，Oracle Restart提供了禁用和启用服务的接口，以最大限度地减少对应用程序用户的服务中断。

服务的管理策略控制在重新启动数据库时是否自动启动服务。如果服务的管理策略设置为AUTOMATIC ，则它会自动重新启动。如果服务的管理策略设置为MANUAL ，则必须手动启动。

    在使用Oracle Restart和Oracle Data Guard主数据库和备用数据库的配置中，数据库服务在切换或故障转移期间会自动从主数据库故障切换到备用数据库。

    快速连接故障切换为快速应用程序通知（FAN）集成客户端（例如使用JDBC，OCI或ODP.NET的客户端）提供高可用性。如果将客户端配置为使用快速连接故障转移，则客户端会自动订阅FAN事件，并可以对数据库UP和DOWN事件做出响应。作为响应，Oracle数据库为客户端提供与提供所请求的数据库服务的活动实例的连接。

4.2.14.2为JDBC客户端启用快速连接故障转移

    使用以下系统属性启用FAN而不进行数据源更改-D oracle.jdbc.FastConnectionFailover=true 。

4.2.14.3为Oracle Call Interface Client启用快速连接故障转移

Oracle Call Interface客户端可以通过注册接收有关Oracle Restart高可用性FAN事件的通知并在事件发生时进行响应来启用快速连接故障转移（FCF）。

这样可以改善Oracle Call Interface中的会话故障转移响应时间，并从连接和会话池中删除已终止的连接。此功能适用于Oracle Call Interface应用程序，包括使用透明应用程序故障转移（TAF），连接池或会话池的应用程序。

  • 终止客户端上受影响的连接并返回错误

  • 使用SRVCTL启动组件时，Oracle Restart可以首先启动此组件所依赖的任何组件。使用SRVCTL停止组件时，Oracle Restart可以先停止任何相关组件。

  • SRVCTL始终根据其Oracle Restart配置启动组件。通过其他方式启动组件可能不会。

    例如，如果在将数据库添加到Oracle Restart配置时指定了服务器参数文件（SPFILE）位置，并且该位置不是SPFILE的默认位置，则使用SQL * Plus启动数据库时，指定的SPFILE可能不会使用该配置。

  • 使用SRVCTL启动组件时，将设置存储在组件的Oracle Restart配置中的环境变量。

    有关详细信息，请参阅“ 在Oracle Restart Configuration中管理环境变量 ” 。

您可以使用SRVCTL启动和停止Oracle Restart管理的任何组件。

要使用SRVCTL启动或停止Oracle Restart管理的组件：

  1. 准备运行SRVCTL，如“ 准备运行SRVCTL ”中所述 。

4.4停止并重新启动Oracle Restart以进行维护操作

当Oracle Restart管理Oracle主目录中的多个组件时，您可以在Oracle主目录中停止Oracle Restart以及Oracle Restart管理的组件。

srvctl修改服务-db db_unique_name -service service_name [ db_unique_name [PRIMARY] [，PHYSICAL_STANDBY] [，LOGICAL_STANDBY] [，SNAPSHOT_STANDBY]] [-policy {AUTOMATIC | MANUAL}] [-failovertype {NONE |会话|选择| TRANSACTION}] [-failovermethod {NONE | BASIC}] [-failoverdelay integer ] [-failoverretry integer ] [-clbgoal {SHORT | LONG}] [-rlbgoal {SERVICE_TIME | THROUGHPUT | NONE}] [ - 通知{TRUE | FALSE}] [-edition edition_name ] [-pdb pluggable_database] [-sql_translation_profile sql_translation_profile] [-commit_outcome {TRUE | FALSE}] [-retention retention ] [-replay_init_time replay_init_time ] [-drain_timeout timeout ] [-stopoption stop_option ] [-session_state {STATIC | DYNAMIC}] [-global_override] [-verbose]

表4-35 srvctl修改服务选项

选项 描述
-db db_unique_name

数据库的唯一名称

该名称必须与DB_UNIQUE_NAME初始化参数设置匹配。如果未指定DB_UNIQUE_NAME ，则此选项必须与DB_NAME初始化参数设置匹配。DB_UNIQUE_NAME的默认设置使用DB_NAME的设置。
-service service_name

服务名称
-role [主要] [，PHYSICAL_STANDBY] [，LOGICAL_STANDBY] [，SNAPSHOT_STANDBY]

服务角色列表

此选项仅适用于Oracle Data Guard环境。如果存在此选项，则在数据库启动时，仅当其服务角色之一与当前数据库角色匹配时，才会启动该服务。

另请参阅： Oracle Data Guard概念和管理 ，以获取有关数据库角色的更多信息
-policy {AUTOMATIC |手册}

服务的管理政策

如果是AUTOMATIC （默认值），则在重新启动数据库时会自动启动该服务，无论是通过计划的重新启动（使用SRVCTL）还是在失败之后。但是，自动重启也受服务角色的限制（ -role选项）。

如果是MANUAL ，则计划重新启动数据库（使用SRVCTL）时，服务永远不会自动重新启动。MANUAL设置不会阻止Oracle Restart在运行时监视服务，如果发生故障则重新启动它。

-failovertype {NONE | SESSION |选择|交易}

要为OCI和Java启用应用程序连续性，请使用TRANSACTION 。

如果故障转移类型是TRANSACTION ，则OCI和Java会在收到可恢复错误时尝试恢复飞行事务。当故障转移类型为TRANSACTION ， commit_outcome选项必须设置为TRUE 。

要为OCI启用透明应用程序故障转移（TAF），请使用SELECT或SESSION 。
-failovermethod {NONE | BASIC}

仅用于向后兼容的TAF故障转移方法

如果故障转移类型（ -failovertype ）设置为NONE以外的值，则使用BASIC作为此选项。

-failoverdelay 整数

对于应用程序连续性和TAF，故障转移时每个事件的重新连接尝试之间的时间延迟（以秒为单位）
-failoverretry 整数

对于应用程序连续性和TAF，事件后尝试连接的次数
-clbgoal {简短|长}

连接负载平衡目标

使用SHORT进行运行时负载平衡。

使用LONG进行长时间运行的连接，例如批处理作业。

-rlbgoal {SERVICE_TIME | THROUGHPUT |没有}

运行时负载平衡目标

使用SERVICE_TIME按响应时间平衡连接。

使用THROUGHPUT通过吞吐量平衡连接。

- 通知{TRUE |假}

为OCI连接启用快速应用程序通知（FAN）
-edition EDITION_NAME

该服务的初始会话版本

如果未指定此选项，则不会为服务修改版本。

如果指定了此选项但edition_name为空，则将版本设置为NULL 。 NULL版无效。

为服务指定版本时，指定服务的所有后续连接都将此版本用作初始会话版本。但是，如果会话连接指定了不同的版本，则会话连接中指定的版本将用于初始会话版本。

SRVCTL不验证指定的版本名称。在连接期间，连接用户必须具有指定版本的USE权限。如果版本不存在或者连接用户对指定版本没有USE特权，则会引发错误。

-pdb pluggable_database

在CDB中，与服务关联的PDB的名称

如果此选项设置为空字符串，则该服务与root关联。
-sql_translation_profile sql_translation_profile

将应用程序从非Oracle数据库迁移到Oracle数据库后要添加的服务的SQL转换概要文件

注意：在使用SQL转换框架之前，必须将所有服务器端应用程序对象和数据迁移到Oracle数据库。

另请参阅： Oracle数据库SQL转换和迁移指南 ，以获取有关使用SQL转换配置文件的更多信息
-commit_outcome {TRUE |假}

对于Transaction Guard，当为TRUE时，由于可恢复的中断，在事务会话失败后，可以访问事务的提交结果。

如果为默认值FALSE ，则不保留事务的提交结果。

当此选项设置为TRUE ，事务提交的结果是持久的，并且应用程序可以在中断后确定事务的提交状态。您可以为用户定义的服务将commit_outcome设置为TRUE 。

commit_outcome设置对Oracle Active Data Guard和只读数据库没有影响。

另请参阅：有关详细信息，请参阅Oracle数据库开发指南 。

- 保留保留

如果commit_outcome设置为TRUE ，则此选项确定保留提交结果的时间量（以秒为单位）。默认值为24小时（86400）。

如果commit_outcome设置为FALSE ，则无法设置此选项。

-replay_init_time replay_init_time

对于Application Continuity，此选项指定请求的第一个操作的原始执行时间（以秒为单位）与成功重新连接后重放准备好开始的时间之间的差异。在指定的时间过后，应用程序连续性将不会重播。此选项旨在避免在长时间恢复系统后无意中执行事务。默认值为5分钟（300）。最大值为24小时（86400）。

如果failovertype未设置为TRANSACTION ，则不使用此选项。

-drain_timeout 超时

此选项指定在几秒钟内完成资源耗尽的时间。允许的值为NULL ， 0或任何正整数。

排水期用于计划的维护操作。在排水期间，处理所有当前客户端请求，但不接受新请求。排水的工作方式取决于-stopoption选项的设置。

默认值为NULL ，表示未设置此选项。如果未设置该选项，并且已在服务上设置了-drain_timeout ，则使用此值。

如果设置为0 ，则不会发生排水。

-stopoption stop_option

此选项指定服务停止的模式。允许以下值：

  • IMMEDIATE指定在服务停止之前允许会话耗尽。

  • TRANSACTIONAL指定允许会话在—drain_timeout选项中指定的时间内—drain_timeout 。达到时间限制时停止服务，并终止任何剩余的会话。

  • NONE是默认值。

-session_state {STATIC |动态}

对于Application Continuity，此参数指定应用程序是否更改非事务性的会话状态。Oracle建议为大多数应用程序设置DYNAMIC 。

注意：仅当-failovertype设置为TRANSACTION for Application Continuity时，才会考虑此参数。它描述了在请求期间如何更改非事务性。会话状态的示例包括NLS设置，优化程序首选项，事件设置，PL / SQL全局变量，临时表，高级队列，LOB和结果缓存。如果请求开始后非事务性值发生更改，则使用默认值DYNAMIC 。大多数应用程序应使用DYNAMIC模式。如果您不确定，请使用DYNAMIC模式。

力。如果Oracle Restart管理的任何资源仍在运行，请尝试正常停止这些资源。如果无法正常停止资源，请尝试强制停止资源。
